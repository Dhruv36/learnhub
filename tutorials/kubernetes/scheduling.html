Scheduling: Affinity & Taints Advanced

The scheduler decides which node each Pod runs on. By default it just finds a node with enough resources — but production often needs control: keep replicas spread across zones, pin GPU workloads to GPU nodes, isolate tenants, keep a Pod near its data. The tools are nodeSelector/affinity, taints & tolerations, and topology spread. This is where you shape placement for availability and performance.

The Scheduling Vocabulary

MechanismDirectionEffect
nodeSelectorPod → nodeSimple: run only on nodes with matching labels
Node affinityPod → nodeRicher node matching (required or preferred)
Pod affinity / anti-affinityPod → PodCo-locate with / avoid other Pods (by label + topology)
Taints & tolerationsNode repels PodsNode says "keep off unless you tolerate me"
Topology spread constraintsPod distributionEven spread across zones/nodes
🧲 Affinity ATTRACTS (Pods choosing nodes); taints REPEL (nodes rejecting Pods). They're complementary: affinity/nodeSelector are Pod-side rules pulling Pods toward nodes, while taints are node-side rules pushing Pods away unless the Pod has a matching toleration. Use taints to RESERVE nodes (e.g., GPU or dedicated nodes) and affinity to STEER Pods to where they belong.

Node Affinity — Required vs Preferred

affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:      # HARD rule: must match or stay Pending
      nodeSelectorTerms:
        - matchExpressions:
            - { key: disktype, operator: In, values: [ssd] }
    preferredDuringSchedulingIgnoredDuringExecution:     # SOFT rule: try, but schedule anyway
      - weight: 100
        preference: { matchExpressions: [{ key: zone, operator: In, values: [us-east-1a] }] }
// "required" = must; "preferred" = best-effort. "IgnoredDuringExecution" = not re-evaluated if the
// node's labels change after the Pod is already running.

Pod Anti-Affinity — Spread for High Availability

// Keep replicas of the same app OFF the same node/zone so one failure doesn't take them all:
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector: { matchLabels: { app: web } }
        topologyKey: kubernetes.io/hostname          # "not two web Pods on the same node"
// topologyKey defines the domain: hostname (per node), topology.kubernetes.io/zone (per AZ), etc.
// This is how you achieve real HA — spread across failure domains, not stacked on one machine.

Topology Spread Constraints (the modern spread tool)

topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule       # keep zones balanced within a skew of 1
    labelSelector: { matchLabels: { app: web } }
// More expressive than anti-affinity for "spread evenly across zones/nodes" — the preferred approach
// for balanced, multi-zone availability.

Taints & Tolerations — Reserving Nodes

// Taint a node so ordinary Pods won't land on it:
kubectl taint nodes gpu-node-1 gpu=true:NoSchedule
// Only Pods with a matching TOLERATION may schedule there:
tolerations:
  - { key: "gpu", operator: "Equal", value: "true", effect: "NoSchedule" }
// Effects: NoSchedule (block new Pods), PreferNoSchedule (avoid if possible),
//          NoExecute (also EVICT already-running Pods that don't tolerate).
// Kubernetes itself taints unhealthy nodes (e.g., not-ready) with NoExecute to drain them.

Interview Questions

Q1: What's the difference between node affinity and taints/tolerations?

They pull in opposite directions. Node affinity (and the simpler nodeSelector) is a Pod-side attraction rule: the Pod expresses which nodes it wants to run on based on node labels, either as a hard requirement (requiredDuringScheduling — the Pod stays Pending if unmet) or a soft preference (preferredDuringScheduling — the scheduler tries but will place it elsewhere if needed). Taints are a node-side repulsion mechanism: a tainted node repels all Pods that don't have a matching toleration, so it's used to reserve or isolate nodes (GPU nodes, dedicated tenants, control-plane nodes). A Pod needs an explicit toleration to be allowed onto a tainted node, but a toleration only permits scheduling — it doesn't attract. In practice you combine them: taint special nodes so ordinary workloads stay off, add tolerations to the workloads that belong there, and use node affinity to actively steer those workloads onto them.

Q2: How do you ensure replicas are highly available across failure domains?

Prevent them from stacking on a single failure domain using Pod anti-affinity or, preferably, topology spread constraints. Pod anti-affinity with a labelSelector matching the app and a topologyKey (e.g., kubernetes.io/hostname for nodes, or topology.kubernetes.io/zone for availability zones) tells the scheduler not to place two matching replicas in the same domain, so a node or zone failure can't take out all replicas at once. Topology spread constraints are the more expressive, modern approach: you set maxSkew, a topologyKey (like zone), and whenUnsatisfiable (DoNotSchedule or ScheduleAnyway) to keep replicas evenly balanced across zones/nodes within a tolerance. Combined with running enough replicas, a multi-zone cluster, and PodDisruptionBudgets to limit voluntary disruptions, this achieves real high availability — the key idea being to spread across failure domains rather than relying on a single machine or zone.

Q3: What do the taint effects (NoSchedule, PreferNoSchedule, NoExecute) mean?

They control how strongly a taint repels Pods that lack a matching toleration. NoSchedule prevents new Pods without the toleration from being scheduled onto the node, but leaves any already-running Pods untouched. PreferNoSchedule is a soft version: the scheduler tries to avoid placing non-tolerating Pods there but will do so if there's no better option. NoExecute is the strongest: it blocks scheduling AND evicts already-running Pods that don't tolerate the taint (a toleration can specify tolerationSeconds to allow a grace period before eviction). Kubernetes uses NoExecute taints itself — for example, tainting a node that becomes NotReady or unreachable so its Pods are evicted and rescheduled elsewhere. Understanding these lets you both reserve nodes (NoSchedule for GPU/ dedicated pools) and drain them safely (NoExecute), while tolerations on the right workloads let them stay or land where intended.

🏋️ Practice: label a node and use nodeSelector/node affinity (required vs preferred) to target it; observe a Pod stuck Pending when a required rule can't be met; add podAntiAffinity so replicas spread across nodes and confirm placement; use topologySpreadConstraints across zones; taint a node NoSchedule and watch Pods avoid it, then add a toleration; taint NoExecute and watch non-tolerating Pods get evicted.

🎯 Key Takeaways

  • Affinity/nodeSelector ATTRACT Pods to nodes (required = hard, preferred = soft); taints REPEL Pods unless they have a matching toleration.
  • Pod anti-affinity and topology spread constraints spread replicas across nodes/zones for real HA (don't stack on one failure domain).
  • topologyKey defines the failure domain (hostname per node, zone per AZ); topology spread is the modern balanced-spread tool.
  • Taint effects: NoSchedule (block new), PreferNoSchedule (avoid), NoExecute (block AND evict); K8s taints unhealthy nodes to drain them.
  • "IgnoredDuringExecution" means these rules affect scheduling only, not already-running Pods; over-constraining causes Pending Pods.
