GitOps (ArgoCD/Flux) Advanced

Kubernetes is declarative — so why apply changes by running kubectl from laptops and CI scripts? GitOps makes a Git repository the single source of truth for the desired cluster state, and an in-cluster agent (Argo CD or Flux) continuously reconciles the live cluster to match Git. Deploys become git commits; rollbacks become git reverts; drift is detected and corrected automatically. It's the production-grade way to operate clusters at scale.

The Core Idea

// Traditional ("push"): CI runs `kubectl apply` / `helm upgrade` against the cluster.
//   Problems: credentials to the cluster live in CI, no single source of truth, drift goes unnoticed,
//   who-changed-what is unclear, rollback is ad hoc.
// GitOps ("pull"): the desired state (manifests/Helm/Kustomize) lives in GIT. An in-cluster AGENT
//   watches Git and applies changes ITSELF, continuously reconciling live state → Git state.
//   Deploy = merge a PR. Rollback = git revert. Audit = git history. Drift = auto-detected/corrected.
🔄 Git is the source of truth; the cluster is a reconciled copy of it. This extends Kubernetes' own reconciliation loop out to your whole delivery process: instead of humans and pipelines pushing state INTO the cluster, an agent continuously pulls the desired state FROM Git and makes the cluster match. Every change is a reviewed, audited, revertible Git commit — the operational superpower of GitOps.

The Four GitOps Principles

PrincipleMeaning

Versioned & immutableDesired state is stored in Git — versioned, auditable, revertible

Continuously reconciledAgents detect drift and converge live state to the declared state

# Argo CD Application — "keep this cluster path in sync with this Git path":
apiVersion: argoproj.io/v1alpha1
kind: Application
spec:
  source: { repoURL: https://git/org/apps, path: prod/web, targetRevision: main }
  destination: { server: https://kubernetes.default.svc, namespace: web }
  syncPolicy:
    automated: { prune: true, selfHeal: true }   # auto-apply, delete removed resources, fix drift
// prune = remove resources deleted from Git; selfHeal = revert manual cluster changes back to Git.

CI vs CD in GitOps (a key separation)

// CI (build/test) still runs in your pipeline: build image → test → push image → update the manifest/
//   values in the GIT repo (e.g., bump image tag) via a commit/PR.
// CD (deploy) is done by the GitOps AGENT: it sees the Git change and reconciles the cluster.
// Benefit: CI never needs cluster credentials — the agent pulls from inside the cluster. Smaller
// attack surface and a clean separation between "produce artifacts" and "apply desired state".

Interview Questions

Q1: What is GitOps and how does it differ from traditional CD?

GitOps is an operating model where a Git repository is the single source of truth for the desired state of your system (Kubernetes manifests, Helm charts, Kustomize), and an in-cluster agent continuously reconciles the live cluster to match Git. It rests on four principles: the system is described declaratively; that description is versioned and immutable in Git; agents automatically PULL the approved state; and they continuously reconcile, detecting and correcting drift. The difference from traditional "push" CD is the direction and source of authority: traditionally a CI pipeline runs kubectl apply or helm upgrade against the cluster using stored cluster credentials, with no single source of truth and drift going unnoticed. In GitOps, changes are made by committing to Git (deploy = merge a PR, rollback = git revert), and the agent inside the cluster applies them — so every change is reviewed, audited, and revertible through Git history, drift is auto-corrected, and CI no longer needs cluster credentials.

Q2: How do CI and CD split in a GitOps workflow, and why does that improve security?

CI (continuous integration) keeps doing what it does best: build the image, run tests, push the image to a registry, and then update the desired state in the Git repository — typically bumping an image tag or values in the manifests/Helm values via a commit or pull request. CD (deployment) is handled separately by the GitOps agent (Argo CD or Flux) running inside the cluster, which notices the Git change and reconciles the cluster to it. This separation improves security because the CI system no longer needs credentials to the cluster — it only needs to write to Git and the registry — while the agent pulls changes from within the cluster boundary. That shrinks the attack surface (no powerful kube credentials sitting in external pipelines), makes access control cleaner, and cleanly divides "produce artifacts and declare desired state" (CI) from "apply desired state" (CD).

Q3: What are drift detection and self-healing, and how do Argo CD/Flux provide them?

Drift is any divergence between the live cluster state and the desired state declared in Git — for example, someone runs kubectl edit or scales a Deployment by hand, or a resource is deleted. GitOps agents continuously compare live state to the Git-declared state; drift detection surfaces these differences (Argo CD shows an out-of-sync/diff view, Flux reports it), and self-healing automatically re-applies the Git state to correct the drift (Argo CD's syncPolicy.automated.selfHeal, and Flux's continuous reconciliation), while pruning removes resources that were deleted from Git. Because Git is authoritative, manual changes don't stick — the cluster converges back to the reviewed state — so the environment stays consistent and reproducible, unauthorized or accidental changes are reverted, and you can even rebuild a cluster from Git for disaster recovery. Both Argo CD and Flux implement this reconciliation loop; Argo CD emphasizes a rich UI and application-centric model, while Flux is a composable set of controllers, but both watch Git, reconcile, and correct drift.

🏋️ Practice: install Argo CD (or Flux) and point an Application at a Git repo path; deploy an app by committing manifests and watch the agent sync it; manually kubectl scale the Deployment and watch selfHeal revert it (drift correction); roll back by git reverting the commit; enable prune and delete a manifest to see the resource removed; structure an app-of-apps; store a secret with Sealed Secrets so it's safe in Git.

🎯 Key Takeaways

  • GitOps makes Git the single source of truth; an in-cluster agent (Argo CD/Flux) continuously reconciles the cluster to Git.
  • Deploy = merge a PR; rollback = git revert; audit = git history; drift is auto-detected and self-healed.
  • CI builds/tests and updates the Git repo (image tag/values); the agent does CD — so CI needs no cluster credentials.
  • Argo CD (rich UI, app-centric) vs Flux (composable controllers); both pull from Git, reconcile, support Helm/Kustomize, and correct drift.
  • Never kubectl-apply by hand (selfHeal reverts it); keep secrets Git-safe with Sealed Secrets/SOPS/External Secrets.
