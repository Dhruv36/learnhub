Helm Intermediate

Real applications are dozens of YAML files (Deployments, Services, ConfigMaps, Ingress…) that differ per environment. Copy-pasting and hand-editing them doesn't scale. Helm is the package manager for Kubernetes: it templates your manifests, parameterizes them with values, and manages versioned releases you can upgrade and roll back — like apt/npm for the cluster.

The Problem Helm Solves

// Without Helm: dev/staging/prod each need their own copies of every manifest, differing only in
// replicas, image tag, resource sizes, hostnames… → duplication, drift, and error-prone edits.
// With Helm: ONE templated chart + a values file PER environment. Change values, not YAML.

Chart Anatomy

mychart/
  Chart.yaml            # chart metadata (name, version, appVersion)
  values.yaml          # DEFAULT configuration values
  templates/           # templated Kubernetes manifests
    deployment.yaml
    service.yaml
    _helpers.tpl       # reusable template snippets
  charts/              # dependency (sub)charts
# templates/deployment.yaml — Go templating pulls from values:
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          resources: {{- toYaml .Values.resources | nindent 12 }}
# values.yaml — defaults, overridable per environment:
replicaCount: 3
image: { repository: myapp, tag: "1.4.2" }
resources: { requests: { cpu: 250m, memory: 256Mi } }

Releases: Install, Upgrade, Rollback

helm install web ./mychart -f values-prod.yaml     # create a RELEASE named "web"
helm upgrade web ./mychart -f values-prod.yaml     # apply changes (new revision)
helm rollback web 3                                # roll back to revision 3 (instant)
helm history web                                   # list revisions
helm uninstall web                                 # remove the release
helm template ./mychart -f values.yaml             # render to raw YAML WITHOUT applying (dry-run/debug)
// A "release" is a tracked, versioned installation of a chart into a namespace — Helm remembers each
// revision, which is what makes upgrade/rollback safe and auditable.
📦 Chart = the package/template; Values = the configuration; Release = an installed, versioned instance. One chart + different values files = the same app deployed consistently across environments. Helm records each revision so you can roll back like Deployments do — but for your whole application stack, not just one workload. This templating-plus-release-management is Helm's core value.

Using Community Charts

helm repo add bitnami https://charts.bitnami.com/bitnami
helm install pg bitnami/postgresql -f my-values.yaml
// Thousands of ready-made charts (databases, ingress controllers, monitoring stacks) — install
// production-grade software with one command and override only what you need. (Vet charts you trust.)

Interview Questions

Q1: What problem does Helm solve and what are charts, values, and releases?

Helm is the package manager for Kubernetes. Real applications comprise many manifests (Deployments, Services, ConfigMaps, Ingress, etc.) that differ across environments only in parameters like replica count, image tag, resource sizes, and hostnames; managing separate copies leads to duplication and drift. Helm solves this by templating the manifests and parameterizing them. A CHART is the packaged set of templated manifests plus metadata and default values; VALUES are the configuration that fills the templates (defaults in values.yaml, overridable per environment with -f or --set); a RELEASE is a specific, versioned installation of a chart into a namespace. Because Helm records each release revision, you get consistent deployment across environments from one chart plus different values files, and safe helm upgrade/rollback for your whole application stack — essentially apt/npm-style packaging, configuration, and lifecycle management for Kubernetes.

Q2: How do upgrade and rollback work in Helm, and why is that valuable?

When you helm install a chart, Helm creates a release and records revision 1 (the rendered manifests and values). Each helm upgrade renders the chart with the current values, applies the changes, and records a new revision, keeping the history. Because every revision is stored, helm rollback <release> <revision> re-applies a previous revision's manifests to revert quickly and predictably, and helm history shows the full timeline. This is valuable because it brings versioned, auditable, reversible lifecycle management to an entire application stack — not just a single Deployment's rollout — so a bad release across many objects can be undone with one command, and environments stay reproducible. Combined with helm template (rendering manifests locally without applying) for dry-run debugging, it makes multi-resource deployments far safer than hand-applying YAML.

Q3: Helm vs Kustomize — when would you choose each?

They take different approaches. Helm uses Go templating plus a package-manager model with values, dependencies, and versioned releases (upgrade/rollback), which makes it strong for packaging and distributing applications, consuming third-party software (thousands of community charts), and managing release lifecycle. Kustomize takes a template-free approach: you write plain, valid YAML base manifests and apply overlays/patches per environment, and it's built into kubectl (apply -k). Kustomize shines for straightforward environment customization of your own manifests without the cognitive overhead of templating, keeping YAML readable and tooling-friendly, but it's weaker for packaging/sharing and has no built-in release/rollback concept. In practice: use Helm when you need packaging, parameterization, dependencies, or to install third-party charts; use Kustomize when you mainly need clean environment overlays of manifests you control. Some teams even combine them (render a chart, then Kustomize-patch it).

🏋️ Practice: create a chart with helm create, template a Deployment/Service from values, and helm template to render it; install it as a release with a prod values file; change values and helm upgrade, then helm rollback to the prior revision and inspect helm history; add a community chart (bitnami/postgresql) with overrides; compare doing the same environment override in Kustomize with kubectl apply -k.

🎯 Key Takeaways

  • Helm is Kubernetes' package manager: it templates manifests and manages versioned releases (apt/npm for the cluster).
  • Chart = templated package; Values = configuration (defaults + per-env overrides); Release = a versioned installed instance.
  • helm install/upgrade/rollback/history give safe, auditable, reversible lifecycle management for the whole app stack.
  • Reuse thousands of community charts to install production software in one command (but vet and pin them).
  • Kustomize is the template-free overlay alternative (built into kubectl -k); choose Helm for packaging/distribution, Kustomize for simple env overlays. Keep secrets out of values.yaml.
