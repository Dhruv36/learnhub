ConfigMaps & Secrets Basics

Configuration and credentials must live OUTSIDE your container image so one image runs across every environment (the Docker env-secrets principle, now in Kubernetes). ConfigMaps hold non-sensitive config; Secrets hold sensitive values. Both can be injected into Pods as environment variables or mounted files — decoupling config from code.

ConfigMap — Non-sensitive Configuration

apiVersion: v1
kind: ConfigMap
metadata: { name: app-config }
data:
  LOG_LEVEL: "info"
  FEATURE_FLAG: "true"
  app.properties: |          # a whole config file as a value
    timeout=30
    retries=3
// Inject into a Pod as env vars, or mount as files:
spec:
  containers:
    - name: app
      image: myapp
      envFrom: [{ configMapRef: { name: app-config } }]   # all keys → env vars
      volumeMounts: [{ name: cfg, mountPath: /etc/config }]
  volumes: [{ name: cfg, configMap: { name: app-config } }] # keys → files under /etc/config

Secret — Sensitive Values

apiVersion: v1
kind: Secret
metadata: { name: db-secret }
type: Opaque
data:
  password: c3VwZXJzZWNyZXQ=      # base64-encoded (NOT encrypted!)
// Inject exactly like a ConfigMap — as env vars or mounted files:
spec:
  containers:
    - name: app
      env:
        - name: DB_PASSWORD
          valueFrom: { secretKeyRef: { name: db-secret, key: password } }
// Mounting as a FILE is generally preferred for sensitive values (less leak-prone than env).
⚠️ Kubernetes Secrets are base64-ENCODED, not ENCRYPTED. Anyone who can read the Secret object (or etcd) sees the value trivially. To actually protect them: enable encryption-at-rest for etcd, lock down access with RBAC (advanced lesson), and for real-world use pull from an external manager (Vault, AWS/GCP Secret Manager) via the External Secrets Operator or CSI driver. Treating base64 as security is a classic, dangerous mistake.

A Subtlety: Updates Don't Auto-Restart Pods

// Changing a ConfigMap/Secret does NOT automatically restart Pods using it as ENV VARS —
// env is read once at container start. Options:
//   - Mounted-file config updates in the Pod over time (with some delay), but the app must RELOAD it.
//   - For env-based config, trigger a rollout: kubectl rollout restart deploy/web
//   - Tools (Reloader) or a config-hash annotation can auto-roll Pods when config changes.
// Interview gotcha: "I updated the ConfigMap but nothing changed" → the Pods need to restart.

Interview Questions

Q1: How do ConfigMaps and Secrets decouple config from images, and how are they consumed?

They externalize configuration and credentials so the same immutable container image runs unchanged across environments — the environment supplies the differences at runtime (the 12-factor principle from the Docker track). A ConfigMap holds non-sensitive key/value config (and whole config files); a Secret holds sensitive values. Both are consumed by Pods in two ways: as environment variables (env with configMapRef/secretKeyRef, or envFrom to import all keys) or as mounted files via a volume, where each key becomes a file. Mounted files are generally preferred for secrets because environment variables are more leak-prone (visible in describe/inspect, inherited by child processes, and captured in crash dumps/logs). This keeps configuration and credentials out of the image and version-controlled separately, so you promote one artifact through dev/staging/prod with different config attached.

Q2: Are Kubernetes Secrets secure? How do you actually protect them?

Not by default. A Secret's values are only base64-ENCODED, which is trivially reversible — it's not encryption. By default they're also stored in etcd essentially in the clear, so anyone who can read the Secret object or access etcd can read the values. To genuinely protect them you layer several controls: enable encryption-at-rest so etcd stores Secret data encrypted (ideally via a KMS); restrict who and what can read Secrets with RBAC (least privilege) and limit access from Pods; avoid committing raw Secrets to git (use Sealed Secrets, SOPS, or the External Secrets Operator); and for production, source secrets from a dedicated manager like HashiCorp Vault or a cloud Secret Manager via the External Secrets Operator or the Secrets Store CSI driver, favoring short-lived, rotated, scoped credentials. The key interview point is to never treat base64 as a security boundary.

Q3: You updated a ConfigMap but the app didn't change. Why?

Because how the config is consumed determines whether the change takes effect. If the ConfigMap (or Secret) is injected as environment variables, those are read once when the container starts and are never updated for a running Pod — so the existing Pods keep the old values until they're recreated. If it's mounted as a volume, the files in the Pod are eventually updated (with some propagation delay), but the application still has to detect and reload the changed files; many apps read config only at startup. The fix is to restart the Pods so they pick up the new config, e.g., kubectl rollout restart deployment/web, or use tooling that watches config objects and triggers a rollout automatically (like Reloader) or annotate the Pod template with a hash of the config so any change rolls the Deployment. In short: config changes generally require a Pod restart unless the app actively hot-reloads mounted files.

🏋️ Practice: create a ConfigMap and inject it both as env vars (envFrom) and as mounted files; create a Secret and consume it via secretKeyRef and as a mounted file; base64-decode a Secret to prove it isn't encrypted; update a ConfigMap and confirm running Pods don't change until kubectl rollout restart; enable/inspect encryption-at-rest conceptually; try the External Secrets Operator or Sealed Secrets for git-safe secrets.

🎯 Key Takeaways

  • ConfigMaps (non-sensitive) and Secrets (sensitive) externalize config so one image runs everywhere — config injected at runtime.
  • Consume both as env vars (env/envFrom) or mounted files; prefer files for secrets (env leaks more easily).
  • Secrets are base64-ENCODED, not encrypted — protect with encryption-at-rest, RBAC, and external managers (Vault/cloud).
  • Don't commit raw Secrets to git; use Sealed Secrets / SOPS / External Secrets Operator.
  • Config changes don't auto-restart Pods (env is read at startup) — kubectl rollout restart to apply.
