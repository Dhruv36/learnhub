Autoscaling (HPA) Intermediate

One of Kubernetes' biggest wins is scaling with load automatically. There are three distinct autoscalers — the Horizontal Pod Autoscaler (more Pods), the Vertical Pod Autoscaler (bigger Pods), and the Cluster Autoscaler (more nodes). Knowing which does what — and how the HPA's control loop actually decides — is a common interview theme.

The Three Autoscalers

AutoscalerScalesHow
HPA (Horizontal Pod Autoscaler)Number of Pod replicasAdds/removes Pods based on metrics (CPU/memory/custom)
VPA (Vertical Pod Autoscaler)Per-Pod CPU/memory requestsRight-sizes requests/limits (often restarts Pods)
Cluster AutoscalerNumber of nodesAdds nodes when Pods can't be scheduled; removes idle nodes
🔀 HPA and Cluster Autoscaler work together; HPA and VPA usually don't (on the same metric). HPA adds Pods → if there's no room, the Cluster Autoscaler adds nodes. But HPA (scaling on CPU) and VPA (changing CPU requests) fighting over the same signal causes thrashing — pick one per resource dimension. This division of labor is a favorite interview question.

The HPA in Practice

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef: { kind: Deployment, name: web }
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target: { type: Utilization, averageUtilization: 70 }   # keep avg CPU near 70% of REQUEST
// The HPA control loop (every ~15s):
//   desiredReplicas = ceil( currentReplicas × currentMetric / targetMetric )
// e.g., 4 pods at 90% CPU, target 70% → ceil(4 × 90/70) = ceil(5.14) = 6 replicas.
kubectl autoscale deploy web --cpu-percent=70 --min=3 --max=20   # quick imperative form
kubectl get hpa                                                  # watch current/target + replicas

Critical Prerequisites (why HPA "does nothing")

// 1. The METRICS SERVER must be installed — HPA reads CPU/memory from it. No metrics-server → no scaling.
// 2. Pods must set resource REQUESTS — utilization is measured as a % of the request. No request → HPA
//    can't compute CPU utilization for that Pod.
// Two of the most common "my HPA isn't scaling" causes, both about missing prerequisites.

Scaling on the Right Signal

// CPU/memory are easy but often the WRONG signal. A queue worker's load is queue DEPTH; an API's is
// REQUESTS-PER-SECOND or p95 LATENCY, not CPU. Scale on what actually reflects load:
//   - Custom metrics (via Prometheus Adapter): requests/sec, queue length
//   - External metrics: cloud queue depth (SQS/PubSub), Kafka lag
// KEDA is popular for event-driven autoscaling (scale on Kafka/SQS/cron, even to ZERO).

Stabilization & Flapping

behavior:
  scaleDown:
    stabilizationWindowSeconds: 300   # wait 5 min of low load before scaling down (avoid flapping)
// The HPA scales UP quickly but DOWN slowly by default, to avoid thrashing on spiky traffic.
// Tune scaleUp/scaleDown policies for your workload's tolerance.

Interview Questions

Q1: Explain HPA, VPA, and Cluster Autoscaler and how they relate.

They scale different dimensions. The Horizontal Pod Autoscaler changes the NUMBER of Pod replicas of a workload based on observed metrics (CPU/memory or custom/external signals) to match load. The Vertical Pod Autoscaler changes the SIZE of Pods by adjusting their CPU/memory requests (and limits) to right-size them, typically recreating Pods to apply changes. The Cluster Autoscaler changes the number of NODES: when Pods are Pending because no node has room, it adds nodes, and it removes underutilized nodes when their Pods can be placed elsewhere. They compose: HPA adds Pods and, if the cluster lacks capacity, the Cluster Autoscaler provisions nodes to host them — a very common pairing. HPA and VPA, however, generally shouldn't act on the same resource simultaneously (e.g., both keying on CPU) because one changing replica count while the other changes CPU requests causes thrashing; you pick one per resource dimension.

Q2: How does the HPA decide how many replicas to run?

The HPA runs a periodic control loop (roughly every 15 seconds) that reads the target metric for the workload's Pods and computes desiredReplicas = ceil(currentReplicas × currentMetricValue / targetMetricValue). For example, with 4 replicas averaging 90% CPU and a 70% target, it computes ceil(4 × 90/70) = 6 replicas and scales up; if utilization falls below target it scales down. For resource metrics like CPU, utilization is measured as a percentage of the Pod's resource REQUEST, which is why requests must be set. It clamps the result to minReplicas/maxReplicas, and by default scales up promptly but scales down conservatively using a stabilization window to avoid flapping on spiky load. It requires a metrics source (metrics-server for CPU/memory, or an adapter for custom/external metrics); without it the HPA can't read metrics and won't scale.

Q3: Why is CPU often the wrong metric, and what do you do instead?

Because for many workloads CPU doesn't reflect actual load or the thing you care about. A queue worker that's mostly waiting on I/O may sit at low CPU while its backlog explodes; an API's user experience is governed by request rate and tail latency, not CPU. Scaling on CPU there reacts too late or not at all. The fix is to autoscale on a signal that truly represents load: custom metrics such as requests-per-second or p95 latency exposed via the Prometheus Adapter, or external metrics like a cloud queue's depth (SQS/PubSub) or Kafka consumer lag. For event-driven and queue-based systems, KEDA is widely used — it scales deployments based on event sources (Kafka, SQS, RabbitMQ, cron, etc.) and can even scale to zero when idle. The principle is to pick the metric that is the real driver of load for that workload rather than defaulting to CPU.

🏋️ Practice: install metrics-server, create an HPA on CPU (min 3, max 20, target 70%), and load-test to watch replicas grow then shrink; observe <unknown> if requests are missing; compute the desiredReplicas formula by hand and compare; add a stabilization window and watch scale-down slow; try a custom metric (RPS via Prometheus Adapter) or KEDA on a queue; combine with the Cluster Autoscaler so new Pods trigger new nodes.

🎯 Key Takeaways

  • HPA scales Pod COUNT, VPA scales Pod SIZE (requests), Cluster Autoscaler scales NODE count.
  • HPA + Cluster Autoscaler compose (more Pods → more nodes); avoid HPA + VPA on the same resource (thrashing).
  • HPA formula: desired = ceil(current × currentMetric / targetMetric), clamped to min/max; loops ~every 15s.
  • Prerequisites: metrics-server installed AND Pods have resource requests — the top "HPA does nothing" causes.
  • CPU is often the wrong signal — scale on RPS/latency/queue depth via custom/external metrics or KEDA (scale to zero).
