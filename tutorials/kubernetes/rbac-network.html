RBAC & Network Policies Advanced

Two independent security controls every production cluster needs. RBAC governs WHO can do WHAT to the Kubernetes API (identity and permissions). Network Policies govern which Pods can TALK to which Pods on the network (segmentation). Together they enforce least privilege at the control-plane and data-plane layers — core to securing multi-tenant, real-world clusters.

RBAC — Authorization for the API

// Four objects, two pairs. A ROLE lists permissions; a BINDING grants a Role to a subject.
//   Role / RoleBinding            → scoped to ONE namespace
//   ClusterRole / ClusterRoleBinding → cluster-wide (all namespaces, or cluster resources)
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: { namespace: payments, name: pod-reader }
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]     # allowed actions (no create/delete)
---
kind: RoleBinding
metadata: { namespace: payments, name: read-pods }
subjects: [{ kind: ServiceAccount, name: api, namespace: payments }]
roleRef: { kind: Role, name: pod-reader, apiGroup: rbac.authorization.k8s.io }
🔑 Role = a set of permissions; Binding = who gets them. RBAC is additive and default-deny: subjects (users, groups, and especially ServiceAccounts that Pods run as) have NO permissions until a binding grants a Role's verbs on resources. There are no "deny" rules — you grant the minimum needed. Scope with Role (one namespace) vs ClusterRole (cluster-wide) to keep blast radius small.

ServiceAccounts — Pod Identity

// Every Pod runs as a ServiceAccount (the "default" SA if unspecified). That SA is the identity RBAC
// authorizes when the Pod calls the Kubernetes API. Best practices:
//   - Give each workload its OWN ServiceAccount with least-privilege RBAC (not the default SA).
//   - Don't grant broad ClusterRoles (like cluster-admin) to app ServiceAccounts.
//   - automountServiceAccountToken: false if the Pod doesn't need API access at all.
// A compromised Pod inherits its SA's permissions — over-permissioned SAs are a top attack path.

Network Policies — Pod-to-Pod Segmentation

// By DEFAULT, all Pods can talk to all Pods (flat network). A NetworkPolicy restricts this. Once a
// Pod is SELECTED by any policy, it becomes default-DENY for that direction (ingress/egress) and only
// the listed traffic is allowed.
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { namespace: payments, name: db-allow-api }
spec:
  podSelector: { matchLabels: { app: db } }     # applies to db Pods
  policyTypes: [Ingress]
  ingress:
    - from:
        - podSelector: { matchLabels: { app: api } }   # ONLY api Pods may reach db
      ports: [{ port: 5432 }]
// A common baseline: a "default deny all ingress" policy per namespace, then explicitly allow needed flows.
🌐 Network Policies are default-ALLOW until you write one — then default-DENY for selected Pods. An empty cluster lets every Pod reach every other Pod. Segmentation is opt-in: you apply a policy (often "deny all", then allow specific flows). Crucially, policies require a CNI that ENFORCES them (Calico, Cilium, etc.) — apply a NetworkPolicy under a non-enforcing CNI and it silently does nothing, a classic gotcha.

Interview Questions

Q1: Explain Kubernetes RBAC — Roles, ClusterRoles, and bindings.

RBAC controls authorization to the Kubernetes API: who (subjects) can perform which actions (verbs like get/list/create/delete) on which resources. It uses four objects in two pairs. A Role defines a set of permissions within a single namespace; a ClusterRole defines permissions cluster-wide or over cluster-scoped resources. A RoleBinding grants a Role (or a ClusterRole, scoped to that namespace) to subjects within a namespace; a ClusterRoleBinding grants a ClusterRole across the whole cluster. Subjects are users, groups, or ServiceAccounts (the identity Pods run as). RBAC is additive and default-deny with no deny rules — subjects have no access until a binding grants it — so you follow least privilege, granting only the minimum verbs on the minimum resources and preferring namespace-scoped Roles over cluster-wide ones to limit blast radius. This is the primary defense against both malicious and accidental misuse of the API.

Q2: How do Network Policies work, and what's the default behavior?

Network Policies segment Pod-to-Pod (and Pod-to-external) traffic at the network layer. The important default is that, without any policy, Kubernetes networking is flat — every Pod can reach every other Pod. Network Policies are opt-in: as soon as a Pod is SELECTED by at least one policy for a given direction (ingress or egress), that direction becomes default-deny for that Pod and only the explicitly allowed traffic (by pod selector, namespace selector, or IP block, on specified ports) is permitted. A common production baseline is a per-namespace "default deny all ingress" policy, followed by explicit allow rules for the flows that should exist (e.g., only api Pods may reach db Pods on 5432). A critical caveat: Network Policies are enforced by the CNI plugin, so they only take effect if you run an enforcing CNI like Calico or Cilium — applying policies under a CNI that doesn't enforce them silently does nothing. Together with restricting egress, they prevent lateral movement and data exfiltration if a Pod is compromised.

Q3: What's a ServiceAccount and why does it matter for security?

A ServiceAccount is the identity that a Pod uses when it authenticates to the Kubernetes API; if you don't specify one, the Pod uses the namespace's default ServiceAccount, and a token for it may be mounted into the Pod. RBAC authorizes ServiceAccounts just like users, so the permissions bound to a Pod's ServiceAccount are exactly what that Pod (and anything that compromises it) can do to the cluster API. This makes ServiceAccounts a major security boundary: best practice is to give each workload its own dedicated ServiceAccount with least-privilege RBAC rather than reusing the default, never grant broad roles like cluster-admin to application ServiceAccounts, and set automountServiceAccountToken: false for Pods that don't call the API so no token is exposed. Over-permissioned or unnecessarily-mounted ServiceAccount tokens are one of the most common paths from a single compromised Pod to broader cluster compromise.

🏋️ Practice: create a namespaced Role (pods get/list) and bind it to a ServiceAccount, then verify allowed/denied actions with kubectl auth can-i --as=system:serviceaccount:...; give a Pod that SA and confirm its API access matches; apply a default-deny ingress NetworkPolicy and watch traffic drop; add an allow rule so only api can reach db; verify a policy is ignored under a non-enforcing CNI vs enforced under Calico/Cilium; set automountServiceAccountToken: false.

🎯 Key Takeaways

  • RBAC = who can do what to the API: Role/ClusterRole (permissions) + RoleBinding/ClusterRoleBinding (grants); default-deny, additive, no deny rules.
  • Pods run as ServiceAccounts — give each workload its own least-privilege SA; never cluster-admin for apps; disable token automount if unused.
  • Network Policies segment Pod traffic; the network is default-ALLOW until a policy selects a Pod, then default-DENY for that direction.
  • A common baseline: default-deny ingress per namespace, then explicit allow rules (and restrict egress too).
  • Network Policies need an enforcing CNI (Calico/Cilium) — otherwise they're silently ignored.
