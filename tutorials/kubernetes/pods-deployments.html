Pods, ReplicaSets & Deployments Basics

These three objects form a hierarchy you'll use constantly: a Pod is the unit that runs your containers, a ReplicaSet keeps N identical Pods alive, and a Deployment manages ReplicaSets to give you rolling updates and rollbacks. You almost always create Deployments — but understanding the layers underneath is what lets you debug them.

The Pod — the Atom of Kubernetes

// A Pod is one or more containers that share a network namespace (same IP/localhost) and can share
// storage volumes. Usually ONE app container per Pod; extra "sidecar" containers assist it.
apiVersion: v1
kind: Pod
metadata: { name: web, labels: { app: web } }
spec:
  containers:
    - name: web
      image: nginx:1.27
      ports: [{ containerPort: 80 }]
// Pods are EPHEMERAL and DISPOSABLE: they get a new IP each time, are never "healed" in place —
// they're replaced. You rarely create bare Pods; a controller manages them for you.
📦 Pods are cattle, not pets. A Pod has no durable identity or IP — when it dies it's replaced by a brand-new Pod with a new IP. That's why you never point clients at a Pod IP (you use a Service), and why apps must be stateless or externalize state (volumes/DBs). This disposability is exactly what enables self-healing and scaling.

Multi-container Pods (sidecars)

// Containers in one Pod share localhost and volumes — used for tightly-coupled helpers:
//  - a log shipper reading the app's log volume
//  - a service-mesh proxy (Istio/Linkerd sidecar) intercepting traffic
//  - an init container that runs to completion BEFORE the app starts (migrations, waiting on deps)
spec:
  initContainers:
    - { name: migrate, image: myapp, command: ["./migrate.sh"] }   // runs first, must succeed
  containers:
    - { name: app, image: myapp }

ReplicaSet — Keep N Pods Running

// A ReplicaSet ensures a specified number of identical Pods exist. It uses a LABEL SELECTOR to
// know which Pods it owns. If one dies, actual < desired → it creates a replacement.
spec:
  replicas: 3
  selector: { matchLabels: { app: web } }   // manages Pods with label app=web
  template: { ... pod spec ... }            // the blueprint for new Pods
// You rarely create ReplicaSets directly — a Deployment creates and manages them for you.

Deployment — Rolling Updates & Rollbacks (what you actually use)

apiVersion: apps/v1
kind: Deployment
metadata: { name: web }
spec:
  replicas: 3
  selector: { matchLabels: { app: web } }
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }   // add 1 new before removing old → zero downtime
  template:
    metadata: { labels: { app: web } }
    spec:
      containers: [{ name: web, image: myapp:1.4.2 }]
// Change the image and apply → the Deployment creates a NEW ReplicaSet and shifts Pods over gradually:
kubectl set image deploy/web web=myapp:1.5.0     // triggers a rolling update
kubectl rollout status deploy/web                // watch it progress
kubectl rollout undo deploy/web                  // roll back to the previous ReplicaSet instantly
🚀 A Deployment = versioned ReplicaSets. Each image change creates a new ReplicaSet; the Deployment scales the new one up and the old one down per your strategy (maxSurge/maxUnavailable). Because the old ReplicaSet is kept (scaled to 0), rollback is instant — it just scales the previous one back up. This is how you get zero-downtime deploys and safe rollbacks declaratively.

Labels & Selectors (the glue)

// Labels are key/value tags on objects; selectors match them. This is how ReplicaSets find their
// Pods and Services find their targets — a loose, flexible coupling instead of hardcoded references.
labels:   { app: web, tier: frontend, version: v2 }
selector: { matchLabels: { app: web } }
kubectl get pods -l app=web        // query by label

Interview Questions

Q1: What is a Pod and why are Pods considered disposable?

A Pod is the smallest deployable unit in Kubernetes: one or more containers that share a network namespace (same IP and localhost) and can share storage volumes, scheduled together onto one node. Usually it's a single application container, sometimes with sidecars (a proxy, log shipper) or init containers that run to completion first. Pods are disposable because they have no durable identity — Kubernetes never repairs a Pod in place; if it dies or its node fails, the Pod is replaced by a brand-new Pod with a new IP. This is deliberate: it's what enables self-healing, horizontal scaling, and rescheduling. The practical consequences are that you should treat Pods as interchangeable ("cattle, not pets"), never hardcode a Pod IP (use a Service for a stable endpoint), and keep Pods stateless or externalize state to volumes/databases (or use StatefulSets when stable identity is truly required).

Q2: How do Deployment, ReplicaSet, and Pod relate?

They form a management hierarchy. A Pod runs the containers. A ReplicaSet sits above Pods and guarantees a desired number of identical Pods exist, using a label selector to identify the Pods it owns and recreating any that are lost. A Deployment sits above ReplicaSets and adds versioned rollout management: when you change the Pod template (e.g., a new image), the Deployment creates a NEW ReplicaSet and gradually scales it up while scaling the old one down according to the rollout strategy, giving zero-downtime updates; it keeps previous ReplicaSets (scaled to zero) so rollbacks are instant. In practice you create and edit Deployments for stateless apps and let them manage ReplicaSets and Pods for you — you rarely touch ReplicaSets or bare Pods directly.

Q3: How does a rolling update work and how do you roll back?

When you update a Deployment's Pod template (commonly the container image), the Deployment controller creates a new ReplicaSet for the new version and shifts capacity from the old ReplicaSet to the new one incrementally, governed by the strategy's maxSurge (how many extra Pods it may create above desired) and maxUnavailable (how many may be unavailable during the update). A common zero-downtime setting is maxSurge: 1, maxUnavailable: 0, which brings up a new Pod before terminating an old one so total capacity never drops. Combined with readiness probes, traffic only goes to Pods that are actually ready. You monitor with kubectl rollout status, and because the previous ReplicaSet is retained (scaled to zero), kubectl rollout undo rolls back instantly by scaling the old ReplicaSet back up — no rebuild needed. This makes deployments safe and reversible by design.

🏋️ Practice: create a Deployment with 3 replicas and watch the ReplicaSet + Pods appear; delete a Pod and see it recreated; kubectl set image to a new tag and watch the rolling update create a new ReplicaSet; kubectl rollout undo and confirm instant rollback; break the image tag and observe a stuck rollout; add an initContainer that must succeed first; query Pods with label selectors.

🎯 Key Takeaways

  • A Pod = one or more containers sharing network/storage; it's ephemeral and replaced (never healed in place) — cattle, not pets.
  • ReplicaSet keeps N identical Pods alive via a label selector; you rarely create it directly.
  • Deployment manages versioned ReplicaSets for rolling updates + instant rollbacks — the object you actually create for stateless apps.
  • Rolling update: new ReplicaSet scaled up while old scaled down per maxSurge/maxUnavailable (0/1 = zero downtime).
  • Labels + selectors loosely couple controllers to Pods and Services to targets; never hardcode Pod IPs (use a Service).
