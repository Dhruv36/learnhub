Probes, Resources & Limits Intermediate

Kubernetes can only keep your app healthy and schedule it sanely if you tell it two things: how to tell whether a container is alive and ready (probes), and how much CPU/memory it needs and may use (requests and limits). Getting these wrong causes most real-world "it's flapping / getting OOM-killed / not receiving traffic" incidents, so this is heavily tested.

The Three Probes

ProbeQuestion it answersAction on failure
livenessProbeIs the process alive (or wedged/deadlocked)?Restart the container
readinessProbeCan it serve traffic right now?Remove the Pod from Service endpoints (no restart)
startupProbeHas a slow-starting app finished booting?Hold off liveness/readiness until it passes
livenessProbe:
  httpGet: { path: /healthz, port: 8080 }
  initialDelaySeconds: 10
  periodSeconds: 10
readinessProbe:
  httpGet: { path: /ready, port: 8080 }   # e.g., checks DB connectivity
startupProbe:
  httpGet: { path: /healthz, port: 8080 }
  failureThreshold: 30                     # allow up to 30×periodSeconds to start
🩺 Liveness restarts; readiness gates traffic — don't confuse them. A readiness failure takes the Pod out of the load balancer until it recovers (great for a temporary dependency blip). A liveness failure KILLS and restarts the container. A common outage: pointing liveness at a check that also depends on a downstream service — when that service hiccups, liveness fails and Kubernetes restart-loops healthy Pods, amplifying the outage. Keep liveness about "is THIS process wedged," and put dependency checks in readiness.

Requests vs Limits (the scheduling + throttling contract)

resources:
  requests: { cpu: "250m", memory: "256Mi" }   # GUARANTEED reservation — used for SCHEDULING
  limits:   { cpu: "500m", memory: "512Mi" }   # CEILING — enforced at RUNTIME
// requests: the scheduler only places the Pod on a node with this much free; it's your guaranteed share.
// limits: the hard cap. Exceed the MEMORY limit → the container is OOM-KILLED (exit 137, Docker lesson).
//         Exceed the CPU limit → the container is THROTTLED (slowed), not killed.
// 1000m = 1 CPU core; Mi = mebibytes.

Quality of Service (QoS) Classes — Who Gets Evicted First

Guaranteedrequests == limits for CPU & memoryEvicted last (most protected)
Burstablerequests < limits (or only some set)Evicted in the middle
BestEffortno requests/limits setEvicted first (least protected)
// Under memory pressure the kubelet EVICTS Pods to protect the node, starting with BestEffort, then
// Burstable exceeding requests. Setting requests/limits isn't just tidiness — it decides who survives.

Why Requests/Limits Matter So Much

// No requests → the scheduler can't reason about capacity → nodes get overcommitted → OOM/evictions.
// No memory limit → one leaky Pod can consume the node and take down its neighbors.
// CPU limit too low → your app is silently throttled and slow (latency spikes) with no crash.
// Memory limit too low → periodic OOM kills (exit 137) and restart loops.
// Right-sizing (via real usage metrics) is an ongoing production discipline, not a one-time guess.

Interview Questions

Q1: Explain liveness vs readiness vs startup probes.

All three are health checks Kubernetes runs against a container, but they answer different questions and trigger different actions. The livenessProbe asks "is this process still healthy, or is it wedged/deadlocked?" — on failure Kubernetes RESTARTS the container, so it's for recovering a stuck process. The readinessProbe asks "can this Pod serve traffic right now?" — on failure the Pod is removed from its Service's endpoints (no restart) so no traffic is routed to it until it recovers, which is ideal for transient conditions like a warming cache or a briefly unavailable dependency, and it also gates rolling updates (new Pods only receive traffic once Ready). The startupProbe handles slow-starting apps: while it's still failing, liveness and readiness are suspended, giving the app time to boot without being prematurely killed. The classic mistake is putting downstream-dependency checks in liveness, which causes healthy Pods to restart-loop when a dependency hiccups — dependency checks belong in readiness.

Q2: What's the difference between resource requests and limits?

A request is the amount of CPU/memory a container is guaranteed and is what the SCHEDULER uses for placement — it only schedules a Pod onto a node that has at least the requested resources free, and the request represents the Pod's reserved share. A limit is the hard ceiling enforced at RUNTIME. The two resources behave differently at the limit: exceeding the MEMORY limit gets the container OOM-killed (exit 137) and restarted, while exceeding the CPU limit results in THROTTLING (the container is slowed, not killed). Setting these correctly is critical: without requests the scheduler can't reason about capacity and nodes become overcommitted, leading to OOMs and evictions; without a memory limit a leaky Pod can consume the whole node and harm neighbors; a too-low CPU limit causes silent latency from throttling; a too-low memory limit causes recurring OOM restart loops. Right-sizing from real usage metrics is an ongoing production discipline.

Q3: What are QoS classes and how do they affect eviction?

Kubernetes assigns each Pod a Quality of Service class based on its resource settings, and uses it to decide eviction order when a node is under resource pressure. Guaranteed Pods (every container has memory and CPU limits equal to its requests) are the most protected and evicted last. Burstable Pods (requests set but less than limits, or only partially specified) are in the middle. BestEffort Pods (no requests or limits at all) are the least protected and evicted first. When a node runs low on memory, the kubelet reclaims resources by evicting Pods starting with BestEffort, then Burstable Pods exceeding their requests, sparing Guaranteed Pods as long as possible. The practical implication is that specifying requests/limits isn't just hygiene — it directly determines which of your workloads survive under pressure, so critical services should be Guaranteed (or at least have solid requests).

🏋️ Practice: add liveness/readiness/startup probes to a Deployment and watch a readiness failure remove a Pod from endpoints without restarting it; make liveness fail and observe restarts; set requests/limits and check the Pod's QoS class (kubectl get pod -o yaml | grep qosClass); set a low memory limit and trigger an OOM kill (137); set a low CPU limit and observe throttling; deploy a BestEffort vs Guaranteed Pod and compare.

🎯 Key Takeaways

  • Liveness → restart a wedged container; readiness → gate traffic (remove from endpoints, no restart); startup → protect slow boots.
  • Never put downstream-dependency checks in liveness (causes restart loops) — those belong in readiness.
  • requests = guaranteed reservation used for scheduling; limits = runtime ceiling (memory over-limit = OOM kill 137, CPU over-limit = throttle).
  • QoS: Guaranteed (requests==limits) evicted last, Burstable middle, BestEffort (none set) evicted first.
  • Always set requests; right-size limits from real metrics — misconfiguration causes most flapping/OOM/latency incidents.
