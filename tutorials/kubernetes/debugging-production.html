Debugging Production Incidents Advanced

This is the lesson that separates senior engineers from certificate holders: when Pods won't start, traffic 503s, or nodes go sideways, you need a systematic method and knowledge of the specific failure signatures. Here are the commands, the common Pod statuses decoded, and the mental model for triaging Kubernetes incidents fast.

The Triage Toolkit

kubectl get pods -o wide            // status, restarts, node — the overview
kubectl describe pod <p>            // EVENTS at the bottom — scheduling failures, probe fails, pulls
kubectl logs <p>  [-p] [-c <ctr>]  // app logs; -p = PREVIOUS container (after a crash); -c = container
kubectl get events --sort-by=.lastTimestamp   // cluster-wide recent events
kubectl top pod / kubectl top node  // live resource usage (needs metrics-server)
kubectl exec -it <p> -- sh          // shell into a running container
kubectl debug <p> -it --image=busybox --target=<ctr>   // ephemeral debug container (distroless Pods)
kubectl get pod <p> -o yaml         // full spec/status (conditions, container states)
🔬 describe first, then logs. kubectl describe pod shows the Events — the single richest source for WHY a Pod won't run (can't schedule, image pull failed, probe failing, OOMKilled). If the container started but crashes, kubectl logs -p (previous instance) shows what it printed before dying. This two-step resolves the majority of Pod incidents.

"The Service Returns 503 / No Endpoints"

kubectl get endpoints <svc>      // EMPTY? → the Service matches no Ready Pods
// Checklist:
//   - Service selector matches the Pod labels? (typo = no endpoints)
//   - Are the Pods READY? (readiness probe failing → removed from endpoints)
//   - Right targetPort (the container's actual port)?
//   - NetworkPolicy blocking the traffic? (rbac-network lesson)
//   - Ingress: is the Ingress Controller running and routing to this Service?
// "503 with empty endpoints" is almost always a selector mismatch or failing readiness probe.

A Systematic Incident Method

1. SCOPE it: one Pod? one Service? one node? whole cluster? (kubectl get pods -A -o wide, get nodes)
2. WHAT CHANGED: recent deploy/config? (kubectl rollout history; Git/GitOps log) → consider rollback.
3. READ THE SIGNALS: describe (events) → logs (-p) → metrics (top) → events (cluster).
4. FORM A HYPOTHESIS from the status signature (Pending vs CrashLoop vs OOM vs 503).
5. MITIGATE FAST: rollback the bad release / scale / cordon+drain a bad node BEFORE deep root-causing.
6. ROOT-CAUSE after stability is restored; write the postmortem.

Node & Cluster-Level Issues

kubectl get nodes                        // NotReady? → kubelet/network/disk issue on that node
kubectl describe node <n>                // conditions: MemoryPressure, DiskPressure, PIDPressure
kubectl cordon <n>                       // stop scheduling new Pods there
kubectl drain <n> --ignore-daemonsets    // evict Pods to safely take a node out (maintenance)
// A NotReady node's NoExecute taint evicts Pods (scheduling lesson) → they reschedule elsewhere.

Interview Questions

Q1: A Pod is stuck in CrashLoopBackOff. How do you debug it?

CrashLoopBackOff means the container starts and then exits/crashes repeatedly, so Kubernetes backs off between restart attempts. I first run kubectl describe pod to read the Events and the container's last state (including the exit code and reason — e.g., OOMKilled/137 points to a memory limit). Then kubectl logs -p (the previous, crashed instance) to see what the app printed before dying — the most common causes are an application error, a missing or wrong config/env value or referenced ConfigMap/Secret, an unreachable dependency at startup, or a failing liveness probe restarting a process that's actually fine. If it's an init container, I check its logs with -c. I correlate with what recently changed (a new image or config) and, if a deploy caused it, roll back to restore service while investigating. The fix follows the signature: raise memory or fix a leak for OOM, correct config/secrets for config errors, fix the dependency or move its check into readiness for probe-induced loops.

Q2: A Service returns 503 / has no endpoints. Walk through it.

An empty endpoints list (kubectl get endpoints <svc>) means the Service isn't backed by any Ready Pods, and that's almost always one of two things. First, a selector mismatch: the Service's selector labels don't match the Pods' labels (a typo or a changed label), so it selects nothing — I compare the Service selector to the Pod labels. Second, the Pods aren't Ready: a failing readiness probe removes Pods from the endpoints, so I check the Pods' readiness with describe/logs and fix the probe or the underlying condition it's detecting. I also verify the Service's targetPort matches the container's actual listening port, that the app binds to 0.0.0.0, and that no NetworkPolicy is blocking the traffic. If the 503 comes through an Ingress, I confirm the Ingress Controller is running and its rules point to the right Service and port. Systematically: endpoints → selector → readiness → port → network policy → ingress.

Q3: Describe your systematic approach to a Kubernetes production incident.

I start by scoping the blast radius — is it one Pod, one Service, one node, or cluster-wide? — using kubectl get pods -A -o wide and get nodes, because that shapes everything. Next I ask what changed: recent deploys or config changes are the most likely trigger, so I check rollout history and the Git/GitOps log; if a recent release correlates, I'm ready to roll back. Then I read the signals in order: kubectl describe (Events) for scheduling/pull/probe problems, kubectl logs -p for crash output, kubectl top and cluster events for resource pressure. From the Pod status signature (Pending, ImagePullBackOff, CrashLoopBackOff, OOMKilled, 503/empty-endpoints) I form a hypothesis. Crucially, I mitigate before deep root-causing — roll back the bad release, scale out, or cordon and drain a bad node to restore service quickly — and only after stability is restored do I root-cause thoroughly and write a postmortem. The discipline is: scope, identify the change, read the signals, mitigate fast, then investigate.

🏋️ Practice: deliberately break things and diagnose each: a bad image tag (ImagePullBackOff), a crashing command (CrashLoopBackOff — read logs -p), a too-low memory limit (OOMKilled/137), a missing ConfigMap (CreateContainerConfigError), a mismatched Service selector (empty endpoints/503), a failing readiness probe (removed from endpoints), and a Pod that can't schedule (Pending — describe events). Practice cordon/drain on a node and a rollout undo as a mitigation.

🎯 Key Takeaways

  • describe (Events) first, then logs -p (previous instance for crashes); top/events/exec/debug round out the toolkit.
  • Learn the status signatures: Pending (scheduling), ImagePullBackOff (image/auth), CrashLoopBackOff (app/config/probe), OOMKilled/137 (memory).
  • 503 with empty endpoints ≈ selector mismatch or failing readiness probe — check kubectl get endpoints, then port/policy/ingress.
  • Method: scope → what changed → read signals → hypothesize from the signature → MITIGATE fast (rollback/scale/cordon) → root-cause after.
  • Node NotReady: check conditions (Memory/Disk/PIDPressure); cordon + drain for maintenance; NoExecute taints reschedule Pods.
