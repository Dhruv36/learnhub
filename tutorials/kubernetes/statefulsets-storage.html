StatefulSets & Storage Intermediate

Deployments treat Pods as interchangeable — perfect for stateless apps, wrong for databases that need stable identity and durable per-Pod storage. StatefulSets give each Pod a sticky name, stable DNS, and its own persistent volume. Underneath sits the Kubernetes storage model: PersistentVolumes, PersistentVolumeClaims, and StorageClasses. This is the stateful workload story.

The Storage Model: PV, PVC, StorageClass

ObjectRole
PersistentVolume (PV)A piece of actual storage in the cluster (a cloud disk, NFS share) — the supply
PersistentVolumeClaim (PVC)A Pod's REQUEST for storage ("I need 10Gi, ReadWriteOnce") — the demand
StorageClassA template for DYNAMICALLY provisioning PVs on demand (e.g., gp3 SSD)
// A Pod claims storage via a PVC; a StorageClass dynamically creates the PV (cloud disk) to satisfy it:
apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: data }
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: gp3
  resources: { requests: { storage: 10Gi } }
// The Pod mounts the PVC; the volume OUTLIVES the Pod (survives reschedule/restart).
💾 PVC = demand, PV = supply, StorageClass = the factory. You almost always use dynamic provisioning: the Pod declares a PVC referencing a StorageClass, and Kubernetes auto-creates a matching PV (a real cloud disk) — no manual disk management. The volume's lifecycle is independent of the Pod, which is exactly what stateful workloads need.

Access Modes (a common trap)

ModeMeaning
ReadWriteOnce (RWO)Mounted read-write by a single NODE (most block/cloud disks) — NOT shared across nodes
ReadOnlyMany (ROX)Read-only by many nodes
ReadWriteMany (RWX)Read-write by many nodes (needs NFS/CephFS/EFS — not typical block storage)
// Trap: expecting a cloud block disk (RWO) to be shared by Pods on different nodes. It can't be —
// RWO attaches to one node. For shared read-write across nodes you need an RWX filesystem (EFS/NFS).

StatefulSet — Stable Identity + Per-Pod Storage

apiVersion: apps/v1
kind: StatefulSet
metadata: { name: pg }
spec:
  serviceName: pg           # a HEADLESS service for stable per-Pod DNS
  replicas: 3
  template: { ... }
  volumeClaimTemplates:     # each Pod gets its OWN PVC/PV (not shared)
    - metadata: { name: data }
      spec: { accessModes: [ReadWriteOnce], resources: { requests: { storage: 10Gi } } }
// What a StatefulSet guarantees that a Deployment does NOT:
//   - STABLE names: pg-0, pg-1, pg-2 (not random) — a Pod keeps its identity across reschedules
//   - STABLE DNS: pg-0.pg.namespace.svc.cluster.local (via the headless Service)
//   - STICKY STORAGE: pg-0 always re-attaches to ITS volume (data follows the identity)
//   - ORDERED, controlled rollout/scaling (0,1,2 up; 2,1,0 down)

Deployment vs StatefulSet

DeploymentStatefulSet

DNSVia a Service (load-balanced)Stable per-Pod DNS (headless Service)
Rollout/scaleParallelOrdered
Use forWeb/API/stateless workersDatabases, Kafka, ZooKeeper, quorum systems
🗄️ Consider NOT running databases in Kubernetes at all. StatefulSets make it possible, but managed databases (RDS, Cloud SQL) handle backups, failover, and upgrades for you. Many teams run stateless workloads on K8s and use managed data services — a legitimate and often wiser architecture. Run stateful systems on K8s when you have the operational expertise or a good operator.

Reclaim Policy (don't lose data)

// A PV's persistentVolumeReclaimPolicy decides what happens to the disk when its PVC is deleted:
//   Retain  → keep the disk (manual cleanup) — safe for important data
//   Delete  → delete the underlying cloud disk (data gone!) — common default for dynamic PVs
// Deleting a StatefulSet does NOT delete its PVCs by default (data preserved) — but check your policy.

Interview Questions

Q1: Explain PV, PVC, and StorageClass.

They separate the demand for storage from its supply. A PersistentVolume (PV) represents a concrete piece of storage in the cluster — a cloud disk, an NFS share — with a capacity, access mode, and reclaim policy. A PersistentVolumeClaim (PVC) is a Pod's request for storage ("I need 10Gi, ReadWriteOnce"), which Kubernetes binds to a suitable PV; the Pod mounts the PVC, not the PV directly. A StorageClass defines how to DYNAMICALLY provision PVs on demand from a provisioner (e.g., an SSD tier like gp3), so that when a PVC references a StorageClass, Kubernetes automatically creates a matching PV (a real disk) to satisfy it — no manual disk management. The key property is that the volume's lifecycle is independent of any Pod: the data persists across Pod restarts and reschedules, which is what stateful workloads require. In practice you use dynamic provisioning (PVC + StorageClass) almost everywhere.

Q2: When do you use a StatefulSet instead of a Deployment?

When workloads need stable identity and durable per-Pod storage rather than being interchangeable. Deployments treat Pods as fungible with random names, shared or no storage, parallel rollouts, and load-balanced access — ideal for stateless web/API/worker apps. StatefulSets add guarantees that stateful systems depend on: stable, ordinal Pod names (app-0, app-1, app-2) that persist across reschedules; stable per-Pod DNS via a headless Service (app-0.svc...); sticky storage, where each Pod gets its own PersistentVolume through volumeClaimTemplates and always re-attaches to ITS volume; and ordered, controlled scaling and rollout. You use them for databases, message brokers, and quorum/consensus systems (PostgreSQL, Kafka, ZooKeeper, etcd) where a Pod's identity and its data must stay bound together. A caveat: StatefulSets make running stateful systems possible but don't understand your database's replication/failover — for that you use an operator, or often better, a managed data service.

Q3: What are access modes and reclaim policies, and why do they matter?

Access modes describe how a volume can be mounted: ReadWriteOnce (RWO) allows read-write by a single node (typical of cloud block disks), ReadOnlyMany (ROX) allows read-only mounting by many nodes, and ReadWriteMany (RWX) allows read-write by many nodes but requires a shared filesystem like NFS, EFS, or CephFS. A frequent mistake is expecting a cloud block disk (RWO) to be shared by Pods spread across different nodes — it can't, because it attaches to one node at a time, which causes multi-attach errors; shared read-write across nodes needs an RWX backend. The reclaim policy governs what happens to the underlying storage when its PVC is deleted: Retain keeps the disk for manual handling (safe for important data), while Delete removes the underlying cloud disk (and its data), which is a common default for dynamically provisioned PVs. Getting these wrong leads either to mounting failures or to irreversible data loss, so critical volumes should use Retain and workloads must match their access-mode needs to a compatible storage backend.

🏋️ Practice: create a PVC with a StorageClass and watch a PV be dynamically provisioned; mount it in a Pod, write data, delete and recreate the Pod, and confirm the data persists; deploy a 3-replica StatefulSet with volumeClaimTemplates and observe pg-0/1/2 each get their own PVC; resolve a Pod by its stable DNS name; delete a Pod and confirm it re-attaches to its volume; compare Retain vs Delete reclaim behavior.

🎯 Key Takeaways

  • Storage model: PVC (demand) binds to PV (supply); StorageClass dynamically provisions PVs — the volume outlives the Pod.
  • Access modes: RWO = single node (most cloud disks), RWX = many nodes (needs NFS/EFS/CephFS); don't expect RWO to be shared.
  • StatefulSets give stable ordinal names, stable per-Pod DNS (headless Service), sticky per-Pod volumes, and ordered rollout.
  • Use Deployments for stateless apps; StatefulSets for databases/brokers/quorum systems (Postgres, Kafka, ZooKeeper).
  • Reclaim policy Retain protects critical data (Delete destroys the disk); consider managed DBs over self-running stateful systems.
