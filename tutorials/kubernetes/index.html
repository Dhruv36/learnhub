Kubernetes Architecture Basics

Kubernetes (K8s) is a container orchestrator: you declare the desired state of your application, and Kubernetes continuously works to make reality match it — scheduling containers across a fleet of machines, restarting failed ones, scaling, and rolling out updates. Understanding its architecture — the control plane, the nodes, and the reconciliation loop — makes everything else make sense. Prerequisite: the Docker track (K8s runs containers).

The Big Idea: Declarative + Reconciliation

// You don't tell Kubernetes HOW to do things step by step. You declare the DESIRED STATE:
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3          // "I want 3 copies running"
// Kubernetes' controllers constantly compare desired vs ACTUAL state and act to close the gap.
// A pod dies? Actual (2) ≠ desired (3) → it starts a new one. This CONTROL LOOP is the heart of K8s.
🔁 Reconciliation is the single most important concept. Every Kubernetes object is managed by a controller running a loop: observe actual state → compare to desired state → take action to converge. This is why K8s self-heals, and why you manage infrastructure by editing declarative YAML (the desired state) rather than running imperative commands. "Desired state, reconciled continuously" explains nearly all K8s behavior.

The Control Plane (the brain)

ComponentRole
kube-apiserverThe front door — the REST API everything talks to; validates and persists all objects. The ONLY thing that writes to etcd.
etcdThe cluster's database — a consistent key-value store holding all cluster state (the source of truth).
kube-schedulerDecides WHICH node an unscheduled Pod runs on (based on resources, affinity, taints).
kube-controller-managerRuns the control loops (Deployment, ReplicaSet, Node controllers, etc.) that reconcile state.
cloud-controller-managerIntegrates with the cloud provider (load balancers, volumes, nodes).

The Nodes (the muscle)

ComponentRole
kubeletThe agent on every node — talks to the API server, ensures the Pods assigned to its node are running and healthy (via the container runtime).
Container runtimeActually runs containers (containerd/CRI-O) via the CRI — Docker is not required (internals lesson from the Docker track).
kube-proxyPrograms node networking rules so Service virtual IPs route to the right Pods.

How a Deployment Actually Happens (the flow)

1. You run:  kubectl apply -f deployment.yaml
2. kubectl sends the object to the API SERVER, which validates it and writes it to ETCD.
3. The DEPLOYMENT controller sees a new Deployment → creates a ReplicaSet.
4. The REPLICASET controller sees "want 3 pods, have 0" → creates 3 Pod objects.
5. The SCHEDULER sees unscheduled Pods → assigns each to a node.
6. Each node's KUBELET sees Pods assigned to it → tells the runtime to pull images and start containers.
7. kube-proxy + the network make the Pods reachable.
// Every step is a controller reconciling desired vs actual state. No single component does it all.

Key Objects You'll Meet (the vocabulary)

ObjectWhat it is

ReplicaSetKeeps N identical Pods running
DeploymentManages ReplicaSets → rolling updates & rollbacks (what you usually create)

NodeA worker machine (VM or physical) running Pods

Why Kubernetes (and when NOT to)

// Kubernetes gives you: self-healing, horizontal scaling, rolling zero-downtime deploys, service
// discovery + load balancing, declarative config, and portability across clouds.
// The cost: real operational complexity. For a single small app, Compose or a PaaS (managed
// containers, serverless) is often simpler. Reach for K8s when you have many services, need
// scale/HA across machines, or want a consistent platform across environments/teams.

Common Mistakes

MisconceptionReality
"I run commands to change the cluster"You declare desired state; controllers reconcile continuously
"The API server does the work"It only stores/serves state; controllers + kubelet do the work
"etcd is just a cache"It's the single source of truth — losing it loses the cluster state
"Kubernetes needs Docker"It uses the CRI (containerd/CRI-O); Docker not required
"A Pod = a container"A Pod can hold multiple containers sharing network/storage
"Always use Kubernetes"It adds real complexity — justify it with scale/HA/many services

Interview Questions

Q1: Explain the reconciliation loop and why it's central to Kubernetes.

Kubernetes is declarative: you submit objects describing the DESIRED state (e.g., "3 replicas of this Deployment"), and for each object a controller runs a continuous control loop that observes the ACTUAL state, compares it to the desired state, and takes corrective action to converge them. If a Pod crashes and actual replicas drop below desired, the ReplicaSet controller creates a replacement; if you change the desired image, the Deployment controller rolls out new Pods. This loop is why Kubernetes self-heals, scales, and performs rollouts without you scripting the steps — you change the desired state (the YAML) and the system does the rest. Almost every Kubernetes behavior is an instance of this "desired state, continuously reconciled" pattern, which is why it's the first thing to understand.

Q2: Walk through what happens when you kubectl apply a Deployment.

kubectl sends the Deployment object to the kube-apiserver, which authenticates, authorizes, validates it, and persists it to etcd (the cluster's source of truth; the API server is the only component that writes to etcd). The Deployment controller (in the controller-manager) notices the new Deployment and creates a ReplicaSet; the ReplicaSet controller sees it needs N Pods and none exist, so it creates N Pod objects. The scheduler watches for unscheduled Pods and assigns each to a suitable node based on resource requests, affinity/anti-affinity, taints/tolerations, etc. On each chosen node, the kubelet notices Pods assigned to it and instructs the container runtime (via the CRI) to pull images and start the containers, then reports status back. kube-proxy and the CNI network make the Pods reachable, and a Service provides a stable endpoint. Every step is an independent controller reconciling desired versus actual state — no single component performs the whole deployment.

Q3: What are the control plane components and what does each do?

The control plane is the cluster's brain. The kube-apiserver is the front door: a REST API that all components and users talk to, which validates and serves every object and is the sole writer to etcd. etcd is a consistent, distributed key-value store holding all cluster state — the source of truth, so it's backed up and run with quorum for HA. The kube-scheduler assigns unscheduled Pods to nodes based on resource availability and placement constraints. The kube-controller-manager runs the built-in control loops (Deployment, ReplicaSet, Node, Job controllers, etc.) that reconcile desired and actual state. The cloud-controller-manager integrates with the cloud provider for load balancers, storage, and node lifecycle. On the worker side, each node runs a kubelet (ensures its assigned Pods are running/healthy via the container runtime) and kube-proxy (programs networking so Service IPs route to Pods). Separating a declarative API + state store from many focused controllers is what makes Kubernetes extensible and resilient.

🏋️ Practice: spin up a local cluster (kind/minikube); run kubectl get nodes and kubectl get pods -A to see the control-plane + system pods; kubectl apply a simple Deployment and watch kubectl get pods; delete a pod and watch the ReplicaSet recreate it (reconciliation in action); use kubectl describe on a Pod to see its node assignment and events; explore kubectl get deploy,rs,pod to see the Deployment→ReplicaSet→Pod hierarchy.

🎯 Key Takeaways

  • Kubernetes is declarative: you specify desired state and controllers reconcile it continuously (self-healing, scaling, rollouts).
  • Control plane = apiserver (front door), etcd (source of truth), scheduler (placement), controller-manager (control loops).
  • Nodes run kubelet (keeps assigned Pods healthy), a CRI runtime (containerd/CRI-O — no Docker needed), and kube-proxy (Service networking).
  • apply → apiserver → etcd → controllers create ReplicaSet/Pods → scheduler places → kubelet runs them.
  • K8s buys self-healing/scale/HA at the cost of complexity — justify it; smaller apps may prefer Compose/PaaS.
