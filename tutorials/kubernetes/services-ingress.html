Services & Ingress Basics

Pods are ephemeral and get new IPs constantly — so how does anything reach them reliably? A Service gives a stable virtual IP and DNS name that load-balances across a set of Pods. An Ingress then exposes HTTP(S) services to the outside world with routing and TLS. Together they solve service discovery and external access — core to every real cluster.

The Problem Services Solve

// Pods come and go, each with a new IP. You can't hardcode Pod IPs. A Service provides:
//   - a STABLE virtual IP (ClusterIP) that never changes for the Service's lifetime
//   - a DNS name (my-svc.my-namespace.svc.cluster.local)
//   - automatic LOAD BALANCING across all Pods matching its label selector
apiVersion: v1
kind: Service
metadata: { name: web }
spec:
  selector: { app: web }        // targets Pods labeled app=web (same loose coupling as ReplicaSets)
  ports: [{ port: 80, targetPort: 8080 }]   // Service:80 → Pod:8080
// Now other Pods just call http://web — the Service tracks which Pods are healthy and routes to them.
🎯 A Service is a stable front for a shifting set of Pods. It watches which Pods match its selector and are Ready (the Endpoints), and load-balances to them. Pods can be added, removed, or rescheduled with new IPs and clients never notice — they only ever talk to the Service's stable name. This is Kubernetes service discovery.

Ingress — HTTP Routing & TLS at the Edge

// A LoadBalancer per service is expensive and gives no HTTP routing. An Ingress is ONE entry point
// that routes HTTP(S) by host/path to many backend Services, and terminates TLS:
apiVersion: networking.k8s.io/v1
kind: Ingress
spec:
  tls: [{ hosts: [shop.example.com], secretName: shop-tls }]   // HTTPS termination
  rules:
    - host: shop.example.com
      http:
        paths:
          - { path: /api,  pathType: Prefix, backend: { service: { name: api, port: { number: 80 }}}}
          - { path: /,     pathType: Prefix, backend: { service: { name: web, port: { number: 80 }}}}
// Requires an INGRESS CONTROLLER (nginx-ingress, Traefik, cloud ALB) actually running in the cluster —
// the Ingress object is just rules; the controller implements them.
🚪 Ingress = rules; Ingress Controller = the thing that enforces them. Creating an Ingress object does nothing unless an Ingress Controller (nginx, Traefik, a cloud LB integration) is deployed to watch Ingress objects and configure a real reverse proxy/load balancer. This trips up beginners: "my Ingress isn't working" is often "there's no controller installed."

How Traffic Flows In (cloud web app)

Internet
  → Cloud Load Balancer (from the Ingress Controller's LoadBalancer Service)
    → Ingress Controller (routes by host/path, terminates TLS)
      → ClusterIP Service (load-balances to healthy Pods)
        → Pod (your container)
// kube-proxy programs the node rules that make ClusterIP virtual IPs route to Pod IPs.

DNS & Service Discovery

// CoreDNS gives every Service a DNS name. From a Pod, you reach a Service by name:
curl http://web                          // same namespace
curl http://web.payments.svc.cluster.local   // fully qualified (cross-namespace)
// This is how microservices find each other — no service registry code needed.

Interview Questions

Q1: Why do you need a Service if Pods already have IPs?

Because Pod IPs are ephemeral and unreliable as endpoints: Pods are disposable, get new IPs when recreated or rescheduled, and scale up and down, so clients can't hardcode them. A Service provides a stable abstraction in front of a dynamic set of Pods — a fixed virtual IP (ClusterIP) and a stable DNS name for its lifetime, plus automatic load balancing. It continuously tracks which Pods match its label selector and are Ready (its Endpoints) and distributes traffic only to those, so Pods can be added, removed, or replaced with new IPs and clients are unaffected because they only ever talk to the Service name. That's Kubernetes' built-in service discovery and load balancing, and it's why you target Services, never Pods.

Q2: Explain the Service types and when to use each.

ClusterIP (the default) exposes the Service on an internal virtual IP reachable only inside the cluster — the right choice for internal microservice-to-microservice traffic. NodePort opens a static port on every node that forwards to the Service, giving basic external reachability (useful in dev or as a building block) but it's low-level. LoadBalancer provisions an external cloud load balancer with a public IP that fronts the Service — the typical way to expose something externally in a cloud, but one per Service is costly and offers no HTTP-level routing. ExternalName maps the Service to an external DNS name via CNAME, useful for aliasing an out-of-cluster dependency. A headless Service (clusterIP: None) skips the virtual IP and returns individual Pod IPs via DNS, used with StatefulSets or client-side load balancing where clients need per-Pod addressing. In practice: ClusterIP internally, and an Ingress (below) in front of ClusterIP Services for public HTTP rather than many LoadBalancers.

Q3: What's the difference between a Service and an Ingress, and what's an Ingress Controller?

A Service provides L4 (TCP/UDP) stable networking and load balancing to a set of Pods, primarily for in-cluster discovery (ClusterIP) or basic exposure (NodePort/LoadBalancer). An Ingress operates at L7 (HTTP/HTTPS): it's a single external entry point that routes requests to different backend Services based on host and URL path and can terminate TLS, so you can serve many services (and TLS certs) behind one load balancer instead of provisioning a LoadBalancer per service. Crucially, an Ingress object is just a set of routing rules — it does nothing on its own. An Ingress Controller (nginx ingress, Traefik, or a cloud provider's ALB/Gateway integration) must be running in the cluster to watch Ingress objects and configure an actual reverse proxy/load balancer that enforces those rules. A very common beginner issue is creating an Ingress with no controller installed, so nothing routes. (Newer clusters increasingly use the Gateway API as a more expressive successor to Ingress.)

🏋️ Practice: create a Deployment + ClusterIP Service and curl it by name from another Pod; scale the Deployment and watch the Service load-balance; check kubectl get endpoints to see which Pods back the Service; break the selector and observe an empty endpoints list (503); install an ingress controller (e.g., nginx) and create an Ingress routing two paths to two Services with TLS; compare NodePort vs LoadBalancer vs ClusterIP exposure.

🎯 Key Takeaways

  • A Service is a stable virtual IP + DNS name that load-balances to the Ready Pods matching its selector — Kubernetes service discovery.
  • Types: ClusterIP (internal, default), NodePort (node port), LoadBalancer (cloud LB), ExternalName (CNAME), headless (Pod IPs via DNS).
  • Ingress = L7 HTTP(S) routing by host/path + TLS termination, one entry point for many Services (cheaper than many LoadBalancers).
  • An Ingress needs an Ingress Controller (nginx/Traefik/cloud) actually running — the object alone does nothing.
  • Flow: LB → Ingress Controller → ClusterIP Service → Pod; CoreDNS resolves Service names; kube-proxy programs routing.
