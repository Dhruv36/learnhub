GitHub Actions: First Pipeline Basics

GitHub Actions is the most widely used CI/CD platform today: pipelines are YAML files in your repo under .github/workflows/, triggered by repo events. The concepts here — workflows, jobs, steps, runners, triggers, and reusable actions — transfer directly to GitLab CI, CircleCI, Jenkins, and the rest, so learning one teaches you all of them.

The Object Model

ConceptWhat it is
WorkflowA YAML file in .github/workflows/; triggered by events (on:)
JobA set of steps that run on one runner; jobs run in parallel unless they needs: each other
StepA single action or shell command within a job
ActionA reusable unit (e.g., actions/checkout) invoked with uses:
RunnerThe machine executing a job (GitHub-hosted or self-hosted)

A First Workflow

name: ci
on:
  push:
    branches: [main]
  pull_request:            # run on PRs too

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4          # reusable action: get the code
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci                         # install (clean, from lockfile)
      - run: npm run lint
      - run: npm test
      - run: npm run build
🎯 Triggers (on:) decide when it runs. Run tests on pull_request to gate merges, on push to main to validate/deploy the trunk, on tags for releases, and on schedule (cron) for nightly jobs. Matching the right trigger to each workflow is half of designing a good pipeline.

Jobs, Dependencies & Parallelism

jobs:
  test:
    runs-on: ubuntu-latest
    steps: [ ... ]
  build:
    needs: test               # runs ONLY after test succeeds
    runs-on: ubuntu-latest
    steps: [ ... ]
  deploy:
    needs: build
    if: github.ref == 'refs/heads/main'   # only deploy from main
    steps: [ ... ]
// Jobs without `needs` run in PARALLEL (faster). `needs` creates a dependency graph.

Matrix Builds

jobs:
  test:
    strategy:
      matrix:
        node: [18, 20, 22]        # run the job 3× in parallel, once per version
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with: { node-version: ${{ matrix.node }} }
      - run: npm ci && npm test
// A matrix fans one job across combinations (versions, OSes) in parallel — great for libraries.
🧩 Reuse over reinvention. The ecosystem has thousands of prebuilt actions (checkout, setup-language, cache, upload-artifact, cloud deploys). Prefer them over hand-rolled scripts — but pin them (to a version or, for security, a commit SHA) and vet third-party actions, because an action runs with access to your workflow (see the supply-chain lesson).

Interview Questions

Q1: Explain the GitHub Actions object model (workflow/job/step/runner).

A workflow is a YAML file in .github/workflows/ that is triggered by events declared under on: (push, pull_request, tag, schedule, manual dispatch). A workflow contains one or more jobs; each job runs on a runner — a machine, either GitHub-hosted or self-hosted — and by default jobs run in parallel, unless you declare dependencies with needs:, which creates an ordered graph (e.g., build needs test, deploy needs build). Within a job are steps, executed sequentially on that runner; a step is either a shell command (run:) or a reusable action invoked with uses: (like actions/checkout to get the code). This model — event-triggered workflows composed of parallelizable jobs made of sequential steps on runners, using reusable actions — is essentially the same across GitLab CI, CircleCI, and others, so the concepts transfer directly.

Q2: How do you make a GitHub Actions pipeline fast and correct with jobs?

I split the work into separate jobs so independent work runs in parallel — for example, lint, unit tests, and a type-check job can all run simultaneously rather than serially in one job — and use needs: to express real ordering, so build only runs after tests pass and deploy only after build. For correctness I gate deploy jobs with conditions like if: github.ref == 'refs/heads/main' so they don't run on feature branches, and use environments with protection rules (required reviewers, wait timers) for production. I pass build outputs between jobs with upload/download-artifact and speed up runs with actions/cache for dependencies. For libraries I use a matrix strategy to fan a job across language versions or OSes in parallel. I also use reproducible installs (npm ci from the lockfile) and pin action versions. The result is a parallelized, correctly-ordered pipeline that gives fast feedback while only deploying the right code from the right branch.

Q3: What is a matrix build and when would you use one?

A matrix build uses a strategy.matrix to run the same job multiple times in parallel, once for each combination of the values you specify — for instance, testing across Node 18, 20, and 22, or across Ubuntu, macOS, and Windows, or a cross-product of both. GitHub expands the matrix into separate parallel job runs, each parameterized by the matrix variables (referenced like ${{ matrix.node }}). You use it whenever you need to verify your code works across multiple environments or dependency versions, which is especially important for libraries, SDKs, CLIs, and cross-platform tools that must support a range of runtimes and operating systems. It gives broad compatibility coverage cheaply because the runs happen in parallel, and you can fine-tune it with includes/excludes for specific combinations and fail-fast behavior to control whether one failing combination cancels the rest.

🏋️ Practice: Add .github/workflows/ci.yml to a repo that runs on push and pull_request: checkout, set up your language, install with a reproducible command, then lint, test, and build. Split it into separate test and build jobs where build needs test. Add a matrix across two language versions. Add a deploy job guarded to only run on main, using an environment with a required reviewer. Pin every action to a version.

🎯 Key Takeaways

  • GitHub Actions workflows are YAML in .github/workflows/, triggered by events (on: push/PR/tag/schedule/dispatch).
  • Model: workflow → jobs (parallel by default; order with needs:) → steps (sequential: run: commands or uses: reusable actions) on runners.
  • Use matrix builds to fan a job across versions/OSes in parallel; guard deploy jobs with if: conditions and protected environments.
  • Pass data with artifacts, speed up with caching, inject secrets via ${{ secrets.* }} — never hardcode credentials.
  • Prefer reusable actions but pin them (tag/SHA) and vet third-party ones; use reproducible installs (npm ci). Concepts transfer to GitLab CI/CircleCI/Jenkins.
