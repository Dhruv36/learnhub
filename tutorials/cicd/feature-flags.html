Feature Flags & Progressive Delivery Advanced

Deployment strategies control which version users get; feature flags decouple that further by controlling which features are active — at runtime, without a deploy. This separation of deploy from release is the foundation of progressive delivery: ship code dark, turn it on for 1% of users, and kill it instantly if it misbehaves — no rollback deploy needed.

Deploy ≠ Release (the core idea)

if (flags.isEnabled("new-checkout", user)) {
    renderNewCheckout();     // only for users the flag targets
} else {
    renderOldCheckout();     // everyone else, unchanged
}
// The new code is DEPLOYED to production but not RELEASED until the flag turns it on.
// You control exposure at RUNTIME (a config change), not by deploying — instant on/off, no rebuild.
🚩 Feature flags decouple deploy from release. Code can sit in production, dormant, behind an off flag. Turning it on is a runtime decision independent of the pipeline. This unlocks trunk-based development (merge incomplete features safely behind a flag), instant kill-switches (disable a broken feature in seconds without a rollback deploy), and targeted rollouts (enable for internal users, then 1%, then everyone).

Progressive Delivery in Practice

// Progressive delivery = deployment strategies + feature flags + metrics, done gradually & measurably:
//   1. Deploy code dark (flag OFF).
//   2. Enable for internal/beta users → watch metrics.
//   3. Ramp: 1% → 5% → 25% → 100% of users (targeted by attributes: region, plan, cohort).
//   4. Abort instantly by flipping the flag if error rate / latency / conversion regresses.
// Tools: LaunchDarkly, Unleash, Flagsmith, Split; often paired with canary tooling (Flagger/Argo Rollouts).

The Discipline: Manage Flag Debt

// Every flag is a branch in your code → complexity. Rules that keep it sane:
//  - Give flags an OWNER and an EXPIRY; remove short-lived flags after full rollout (flag cleanup).
//  - Default to OFF/safe; make the fallback path work if the flag service is unreachable.
//  - Don't nest many flags (combinatorial testing explosion).
//  - Treat flag config as auditable, access-controlled change (who flipped what, when).
// Stale, forgotten flags are a real source of incidents ("someone flipped a 2-year-old flag").

Interview Questions

Q1: What problem do feature flags solve, and what does "decouple deploy from release" mean?

Feature flags let you control which features are active at runtime, independently of deploying code. "Decouple deploy from release" means the act of putting code into production (deploy) is separated from the act of exposing a feature to users (release): you can deploy new code that stays dormant behind an off flag, then release it later — and to a chosen subset — by flipping the flag as a runtime config change, with no rebuild or redeploy. This solves several problems at once: it enables trunk-based development because incomplete features can be merged safely behind an off flag instead of living on long branches; it provides instant kill-switches so a misbehaving feature can be disabled in seconds without a rollback deploy; and it allows targeted, gradual rollouts (internal users, then 1%, then everyone, or by region/plan). The result is smaller, safer, more frequent deploys and much faster recovery from feature-level problems.

Q2: How do feature flags relate to canary deployments and progressive delivery?

They operate at different layers and compose. A canary is an infrastructure-level technique: it routes a percentage of traffic to a new VERSION of the service and ramps up based on metrics, but changing that split is an infra action and targeting is usually a random percentage. Feature flags are application-level: they toggle a specific FEATURE, can target precisely by user attributes (region, plan, user id, cohort), and change instantly via runtime config without any deploy. Progressive delivery is the umbrella practice that combines deployment strategies, feature flags, and metrics to release changes gradually and measurably — deploy dark, enable for beta users, ramp 1%→5%→25%→100%, and abort instantly on metric regressions. In practice teams use canaries to validate that a whole new build is healthy and feature flags to control and experiment with individual features and provide kill-switches, often together (e.g., Argo Rollouts/Flagger for canary plus LaunchDarkly/Unleash for flags).

Q3: What is "flag debt" and how do you manage feature flags responsibly?

Flag debt is the accumulating complexity and risk from feature flags that are never removed: each flag adds a branch in the code (two code paths to reason about and test), and stale, forgotten flags are a real incident source — someone flips a long-dormant flag, or dead paths rot untested. Managing flags responsibly means treating them with lifecycle discipline: give every flag an owner and an expiry, and actively remove short-lived release toggles after a feature is fully rolled out (flag cleanup); default flags to a safe/off state and ensure the application still behaves correctly (via sensible defaults or a cached last-known value) if the flag service is unreachable, so the flag system doesn't become a single point of failure; test both states of critical flags and avoid deeply nested flag combinations that explode testing; and treat flag configuration as auditable, access-controlled change so you know who flipped what and when. Long-lived operational and entitlement flags are legitimate, but release toggles should be temporary by design.

🏋️ Practice: wrap a new feature in a flag and deploy it OFF, then enable it for just your own user; ramp a rollout by user attribute (e.g., a region) with a tool like Unleash/Flagsmith; wire a kill-switch and disable a feature in seconds without redeploying; simulate the flag service being down and confirm your safe default holds; add an owner+expiry to a flag and practice removing it after rollout; combine a flag with a canary rollout.

🎯 Key Takeaways

  • Feature flags decouple DEPLOY (code to prod) from RELEASE (feature to users) — exposure becomes a runtime config change.
  • They enable trunk-based development (merge behind off flags), instant kill-switches, and targeted/gradual rollouts.
  • Progressive delivery = deployment strategies + flags + metrics: deploy dark → beta → 1%→100%, abort instantly on regressions.
  • Flags (app-level, per-feature, attribute-targeted, no deploy) compose with canary (infra-level, per-version, % traffic).
  • Manage flag debt: owner + expiry + cleanup, safe defaults if the flag service is down, audit/permission flag changes.
