Docker in CI Intermediate

Containers and CI/CD are a natural pair: a container image is the perfect immutable artifact, and Docker gives you reproducible build and test environments. This lesson covers building images in the pipeline, using containers as consistent CI environments and test dependencies, and the layer-caching and registry practices that keep it fast. (For Docker itself, see the Docker track.)

📦 A container image is the ideal "build once, promote many" artifact. It bundles the app and its runtime into one immutable, versioned unit that runs identically everywhere, eliminating "works on my machine." That's why containerized pipelines dominate: the thing you test in staging is byte-for-byte the thing you run in prod.

Building & Pushing an Image in the Pipeline

jobs:
  build-image:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3        # BuildKit builder (better caching)
      - uses: docker/login-action@v3               # auth to the registry (ideally via OIDC)
        with: { registry: ghcr.io, username: ..., password: ${{ secrets.GHCR_TOKEN }} }
      - uses: docker/build-push-action@v6
        with:
          tags: ghcr.io/org/app:${{ github.sha }}  # immutable, SHA-tagged
          push: true
          cache-from: type=gha                     # reuse layers across runs
          cache-to: type=gha,mode=max

Layer Caching = Fast Image Builds

# Dockerfile order matters: put things that change RARELY first so cache is reused.
FROM node:20-slim
WORKDIR /app
COPY package*.json ./          # deps layer — changes only when deps change
RUN npm ci                     # cached across builds until package*.json changes
COPY . .                       # source layer — changes every commit (below deps on purpose)
RUN npm run build
# In CI, also enable a build cache backend (cache-from/cache-to) so layers persist between runs.
⚡ Order layers stable→volatile. Docker caches layers and invalidates everything after the first changed line. Copying package.json and installing deps before copying source means a code change doesn't bust the (expensive) dependency layer. Combined with a CI cache backend (BuildKit cache-from/to), image builds drop from minutes to seconds. More in the performance lesson.

Containers as Test Dependencies

# Run integration tests against REAL services, not mocks:
services:                     # GitHub Actions "service containers"
  postgres:
    image: postgres:16
    env: { POSTGRES_PASSWORD: test }
    ports: [5432:5432]
# Your test job connects to localhost:5432 → tests hit a real Postgres, then it's torn down.
# (Testcontainers is a popular library that does this programmatically inside the tests.)

Interview Questions

Q1: Why are containers a good fit for CI/CD?

Because a container image is simultaneously the ideal artifact and the ideal environment. As an artifact, it bundles the application and its entire runtime into one immutable, versioned unit that runs identically on a developer laptop, the CI runner, staging, and production — which is exactly the "build once, promote many" property you want, eliminating "works on my machine" drift so the image you tested in staging is byte-for-byte what runs in prod. As an environment, containers give reproducible builds and tests (the same base image means the same toolchain everywhere) and let you spin up real dependencies — a genuine Postgres or Redis container — for integration tests instead of mocking them, then tear them down cleanly. Add fast layer caching for builds and image vulnerability scanning as a gate, and containers give you reproducibility, environment parity, realistic testing, and a deploy-anywhere artifact all at once, which is why containerized pipelines are the norm.

Q2: How do you make Docker image builds fast in CI?

The two big levers are Dockerfile layer ordering and a persistent build cache. Docker caches each layer and invalidates every layer after the first one that changes, so I order instructions from least- to most-frequently changing: copy the dependency manifest (package.json, requirements.txt) and install dependencies before copying the application source, so an ordinary code change doesn't bust the expensive dependency-install layer. Then, because CI runners are often ephemeral and start with a cold cache, I enable a build cache backend — using BuildKit/buildx with cache-from and cache-to (e.g., the GitHub Actions cache or a registry cache) — so layers built in previous runs are reused across runs. I also use multi-stage builds to keep the final image small (build in a heavy stage, copy only the artifacts into a slim runtime stage), slim/distroless base images for faster pulls, and I pin base image versions for reproducibility. Together these turn multi-minute image builds into seconds when only code changed.

Q3: How do you run integration tests against real dependencies in CI?

Rather than mocking a database or cache, I run the real thing as a container alongside the test job. In GitHub Actions this is done with service containers — you declare, say, a postgres:16 service with its port exposed, and the test job connects to it on localhost; the runner starts the container before the job and tears it down after, so tests exercise a genuine Postgres with real SQL behavior, then leave no residue. A popular alternative is the Testcontainers library, which starts and stops dependency containers programmatically from within the test code itself, giving each test suite a fresh, isolated, ephemeral instance and working the same on a laptop and in CI. Either way, using real containerized dependencies makes integration tests far more trustworthy than mocks — they catch schema, query, and driver issues that mocks hide — while remaining reproducible and self-cleaning because the dependency lives only for the duration of the test run.

🏋️ Practice: Write a multi-stage Dockerfile ordered deps-before-source, and a pipeline job that builds and pushes it tagged with the commit SHA, using BuildKit with cache-from/cache-to. Measure the build time on a code-only change vs a dependency change. Add a service container (Postgres) and run integration tests against it. Add a Trivy image scan as a required gate and fix or acknowledge any findings.

🎯 Key Takeaways

  • A container image is the ideal immutable "build once, promote many" artifact and gives reproducible build/test environments and staging↔prod parity.
  • Build and push SHA-tagged images in the pipeline (via BuildKit/buildx); authenticate to the registry (ideally via OIDC).
  • Order Dockerfile layers stable→volatile (deps before source) and use a CI build cache backend (cache-from/cache-to) to make builds fast.
  • Use service containers / Testcontainers to run real dependencies (Postgres/Redis) for trustworthy integration tests.
  • Scan images for CVEs as a gate, use slim/distroless multi-stage builds, pin base versions, and never bake secrets into images.
