Deployment Strategies Advanced

Shipping new code is the moment of maximum risk. Deployment strategies exist to make releases safe and reversible — limiting who sees a bad version and how fast you can undo it. The main families are recreate, rolling, blue/green, and canary, each trading off risk, cost, and complexity. Knowing which to use when — and how rollback works for each — is a core senior skill.

The Strategies at a Glance

StrategyHowDowntime / risk
RecreateStop all old, start all newDowntime; simplest; fine for dev/non-critical
RollingReplace instances gradually (default in K8s)No downtime; a bad version reaches some users before you notice
Blue/GreenRun two full envs; switch traffic all at onceInstant rollback (switch back); double the infra briefly
CanarySend a small % of traffic to the new version, then rampLowest blast radius; needs metrics + automation
🎯 Every strategy is really about two questions: how many users does a bad release hit, and how fast can you undo it? Rolling limits exposure over time; blue/green makes rollback instant by keeping the old version fully running; canary minimizes the blast radius by exposing only a fraction of traffic and watching metrics before proceeding. Pick based on how costly a bad release is for that service.

Blue/Green — Instant Switch & Rollback

// Two identical production environments: BLUE (current, live) and GREEN (new version).
// 1. Deploy the new version to GREEN while BLUE still serves 100% of traffic.
// 2. Smoke-test GREEN in isolation (real prod infra, no user traffic yet).
// 3. Flip the router/load balancer/Service selector from BLUE → GREEN (atomic cutover).
// 4. Problem? Flip back to BLUE instantly — it never went away. That's the rollback.
// Cost: you run two full environments during the switch. Watch out for DB SCHEMA compatibility —
// both versions may hit the same database, so migrations must be backward-compatible.

Canary — Gradual, Metrics-Driven Rollout

// Release to a SMALL subset first, watch, then widen:
//   5% of traffic → new version.  Monitor error rate, latency, business metrics for N minutes.
//   Healthy? → 25% → 50% → 100%.   Unhealthy? → route back to 0% (automatic rollback).
// Automated canary analysis (Argo Rollouts, Flagger, Spinnaker) compares canary vs baseline metrics
// and promotes or aborts WITHOUT a human. This is progressive delivery (next lesson).
// Needs: good observability (metrics lesson) and traffic-splitting (ingress/mesh/LB).

Rolling Update (the common default)

// Replace instances a few at a time (Kubernetes Deployment default): maxSurge/maxUnavailable control
// pace (K8s track). No downtime, cheap, but a bad version is live for SOME users until you catch it,
// and rollback means rolling BACK (not instant). Great default for many services; combine with good
// health checks + fast rollback. Canary/blue-green add safety when a bad release is very costly.

The Rollback Question (per strategy)

StrategyRollback mechanismSpeed
RecreateRedeploy the old version (downtime again)Slow
RollingRoll back to previous ReplicaSet/versionMedium (another rollout)
Blue/GreenFlip traffic back to the old envInstant
CanaryRoute canary traffic to 0%Instant (only a % was affected)

The Database Trap (applies to all)

// If old and new code both run against ONE database (blue/green, canary, rolling all do this),
// schema changes MUST be backward-compatible. The safe pattern is EXPAND / CONTRACT:
//   EXPAND:  add the new column/table (both old & new code work) → deploy code that uses it
//   CONTRACT: only AFTER all old code is gone, remove the old column
// Never ship a migration that breaks the currently-running version. This is the #1 deployment-strategy
// gotcha in interviews and in production.

Interview Questions

Q1: Compare blue/green and canary deployments.

Both reduce release risk but differently. Blue/green runs two complete production environments — blue (current) and green (new) — deploys and smoke-tests the new version on green while blue still serves all traffic, then switches all traffic at once from blue to green; rollback is instant because blue is still running, so you just flip back. Its costs are running double the infrastructure during the switch and an all-at-once cutover (everyone moves together). Canary instead exposes the new version to a small percentage of real traffic (say 5%), watches error rate, latency, and business metrics, and gradually ramps to 100% only if healthy, automatically routing back to 0% if not. Canary has the smallest blast radius and is metrics-driven and gradual, but it requires solid observability and traffic-splitting (ingress/service mesh/LB) plus ideally automated canary analysis. Rule of thumb: blue/green for fast, clean cutovers with instant rollback; canary when you want to limit how many users a bad release can reach and validate against live metrics before full rollout.

Q2: Why can database migrations break deployments, and how do you avoid it?

Because during almost every zero-downtime strategy (rolling, canary, and blue/green) the old and new versions of the application run simultaneously against the SAME database, at least briefly. If a migration is backward-incompatible — for example renaming or dropping a column the still-running old code depends on — the old version breaks the moment the migration applies, causing errors mid-deploy. The fix is the expand/contract (a.k.a. parallel-change) pattern: first EXPAND the schema additively (add the new column/table) so both old and new code work; then deploy the new code that uses it; then, only after all old code is retired, CONTRACT by removing the obsolete column. Migrations are decoupled from code deploys and always kept backward-compatible with the currently-running version, and destructive changes happen a full release later. This lets you deploy and roll back freely without the database being a landmine.

Q3: How does rollback differ across strategies, and why does that matter?

Rollback speed is often the deciding factor because it caps how long a bad release hurts users. With recreate, rolling back means redeploying the old version, incurring downtime again — slow. With rolling updates, you roll back to the previous version via another gradual rollout — medium speed, and some users saw the bad version in the meantime. Blue/green gives instant rollback: the old environment is still fully running, so you just switch traffic back. Canary is also effectively instant and lowest-impact: since only a small percentage of traffic hit the new version, aborting means routing that percentage back to the stable version, and most users were never affected. This matters because mean-time-to-recovery is as important as preventing failures; choosing a strategy with fast, ideally automated rollback (blue/green or canary) for high-stakes services dramatically limits the damage of an inevitable bad deploy.

🏋️ Practice: deploy an app with a rolling update and roll it back; set up blue/green with two environments behind a switchable router and practice an instant flip-back; run a canary (5%→25%→100%) with a tool like Argo Rollouts/Flagger and trigger an automatic abort on bad metrics; write an expand/contract migration and prove both old and new code work during the expand phase; add a smoke test gate before cutover.

🎯 Key Takeaways

  • Strategies answer two questions: how many users a bad release hits, and how fast you can undo it.
  • Recreate (downtime, simple) → rolling (gradual, no downtime, default) → blue/green (instant rollback via a second env) → canary (smallest blast radius, metrics-driven).
  • Blue/green = flip traffic between two full envs; rollback is flipping back. Canary = ramp a % with automated analysis; rollback is routing to 0%.
  • Rollback speed matters as much as prevention — prefer blue/green or canary for high-stakes services.
  • The universal trap: old + new code share one DB, so keep migrations backward-compatible via expand/contract.
