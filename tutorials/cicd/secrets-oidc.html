Secrets & OIDC Auth Intermediate

Pipelines need credentials — to pull private packages, push images, and deploy to the cloud — and leaked CI credentials are one of the most common breach vectors. This lesson covers how to handle secrets safely and, crucially, how OIDC lets your pipeline authenticate to cloud providers with no long-lived secrets at all. That shift is the modern best practice.

Handling Secrets: the Basics

RuleWhy
Never commit secrets to the repoGit history is forever; anyone with repo access sees them
Use the CI secret storeEncrypted, injected as masked env vars at runtime
Least privilege & scopeA leaked secret should unlock as little as possible
Rotate regularly / on exposureLimits the window of a compromised secret
Prefer a secrets managerCentral storage, rotation, audit (Vault, AWS Secrets Manager)
steps:
  - run: ./deploy.sh
    env:
      API_TOKEN: ${{ secrets.API_TOKEN }}   # encrypted, masked in logs
// Secrets are masked in logs, but a malicious step/dependency could still exfiltrate them —
// which is exactly why removing long-lived secrets (OIDC) is better than guarding them.

The Problem with Long-Lived Cloud Keys

// Old way: store an AWS access key + secret as CI secrets to deploy.
//   Risks: they're long-lived (leak = standing access), rarely rotated, sit in the CI provider,
//   and a compromised action/dependency can steal them and use them from anywhere.
🔓 Static cloud keys in CI are the classic leak vector. They never expire, they're copied into a third-party system (your CI provider), and any step in the pipeline (including a compromised third-party action) can read and exfiltrate them. The fix isn't to guard them better — it's to not have them at all.

OIDC: Keyless Cloud Authentication

// OIDC (OpenID Connect) lets the CI provider ISSUE a short-lived, signed identity token per run,
// which the cloud verifies and exchanges for TEMPORARY credentials — no stored secret.
GitHub Actions run ──(signed OIDC token: "repo X, branch main, workflow deploy")──►
   AWS IAM role  (trust policy: "trust GitHub's OIDC issuer, only for repo X on main")
   └─► returns TEMPORARY STS credentials (minutes) → deploy
// Nothing long-lived is stored anywhere. The trust is scoped to your specific repo/branch/env.
🔑 OIDC replaces stored keys with short-lived, scoped, verifiable identity. The cloud provider trusts the CI provider's OIDC issuer and, via conditions in the role's trust policy, restricts it to a specific repository, branch, or environment. Each run gets fresh temporary credentials that expire in minutes. There's no secret to leak, rotate, or steal — the single biggest CI security upgrade you can make. (This is the same AssumeRole/STS mechanism from the AWS IAM lesson.)

Interview Questions

Q1: How should secrets be handled in a CI/CD pipeline?

The baseline rules are: never commit secrets to the repository (Git history is permanent and visible to anyone with access), store them in the CI provider's encrypted secret store or a dedicated secrets manager (Vault, AWS Secrets Manager) that offers central storage, rotation, and audit, and inject them at runtime as masked environment variables rather than hardcoding or passing them on command lines where they'd appear in logs or process listings. Beyond that, apply least privilege so each secret grants the minimum access needed, scope secrets to the jobs/environments that need them (e.g., production secrets only exposed to approved deploy jobs via environment protection rules), rotate them regularly and immediately on any suspected exposure, and enable secret scanning with push protection to catch accidental commits. Most importantly, for cloud authentication I eliminate long-lived secrets entirely using OIDC, because the safest secret is the one that doesn't exist to be leaked.

Q2: What is OIDC in CI/CD and why is it better than stored cloud keys?

OIDC (OpenID Connect) lets the CI provider act as an identity provider that issues a short-lived, signed token describing the current run — for example "this is repository X, branch main, workflow deploy." The cloud provider is configured to trust that OIDC issuer and, through conditions in an IAM role's trust policy, to accept the token only from your specific repository, branch, or environment; it then exchanges the token (via AssumeRole/STS on AWS) for temporary credentials that expire in minutes. This is better than storing a long-lived cloud access key in CI because there is no standing secret anywhere: nothing to leak, rotate, or exfiltrate. Stored keys never expire, live inside a third-party CI system, and can be stolen by any step or compromised dependency and reused from anywhere; OIDC credentials are ephemeral, verifiable, and cryptographically scoped to your exact CI context, so even an intercepted token is far less useful. It's the single biggest security upgrade for a deployment pipeline.

Q3: A third-party action in your pipeline is compromised — what's the exposure and how do you limit it?

An action runs inside your job with access to whatever that job can see — the secrets/environment variables exposed to it, the OIDC token if the job can request one, the source code, and network egress — so a compromised action could read and exfiltrate secrets or abuse deploy credentials. To limit the blast radius: pin actions to a specific commit SHA (not a moving tag like @main) and vet third-party actions before adoption so you're not silently pulling malicious updates; minimize what each job can access by scoping secrets to only the jobs/environments that truly need them and using environment protection rules so production secrets aren't exposed to ordinary CI jobs; prefer OIDC with least-privilege, tightly-scoped roles so even stolen credentials are short-lived and can do little; and restrict token permissions (e.g., least-privilege GITHUB_TOKEN). Defense in depth means that even if one action is compromised, it can't reach powerful long-lived credentials or deploy beyond a narrowly-scoped role.

🏋️ Practice: Configure OIDC between GitHub Actions and a cloud account: create an IAM role whose trust policy accepts GitHub's OIDC issuer only for your repo on the main branch, give it a least-privilege deploy policy, and have the workflow assume it to deploy — with zero stored cloud keys. Move any remaining secrets into the CI secret store, mask them, and enable secret scanning/push protection. Pin every third-party action to a commit SHA.

🎯 Key Takeaways

  • Never commit secrets; store them in an encrypted CI secret store or secrets manager, inject as masked env vars, scope tightly, rotate, and least-privilege everything.
  • Long-lived cloud keys in CI are a top leak vector — never-expiring, stored in a third party, stealable by any step or compromised dependency.
  • OIDC issues a short-lived, signed identity token per run that the cloud exchanges for temporary credentials — no stored secret to leak or rotate.
  • Scope OIDC trust to your exact repo/branch/environment and use least-privilege deploy roles so even a token is barely useful elsewhere.
  • Defense in depth: pin/vet third-party actions (they can read job secrets), use environment protection rules, and enable secret scanning.
