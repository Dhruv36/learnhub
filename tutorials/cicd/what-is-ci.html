What CI Actually Does Basics

"CI/CD" gets thrown around loosely, so let's pin down exactly what each term means and what a pipeline actually does when you push code. Getting the vocabulary right — Continuous Integration, Continuous Delivery, and Continuous Deployment — and understanding the stages of a pipeline is the conceptual backbone of this whole track.

The Three C's (say them precisely)

TermMeaning
Continuous Integration (CI)Automatically build and test every change as it's merged to the shared trunk — catch breakage immediately.
Continuous Delivery (CD)Every change that passes CI is automatically prepared and ready to deploy; releasing to prod is a manual button press.
Continuous Deployment (CD)Goes one step further: every change that passes the pipeline is deployed to production automatically, no human gate.
🔑 The two "CD"s differ by one thing: the manual gate. Continuous Delivery keeps prod deploys as a deliberate button press (the artifact is always deployable); Continuous Deployment removes even that gate so passing the pipeline ships to prod automatically. Both require a rock-solid automated pipeline you trust — Continuous Deployment just demands even more confidence (great tests, monitoring, and safe rollout strategies).

What a Pipeline Does on Every Push

push / open PR
   │
   ▼
[ 1. Checkout ]  get the code
[ 2. Build   ]  compile / bundle / install deps
[ 3. Test    ]  unit → integration tests (fail fast)
[ 4. Analyze ]  lint, type-check, security & dependency scans, coverage gate
[ 5. Package ]  produce a versioned ARTIFACT (jar, container image, bundle)
[ 6. Deliver ]  push artifact to a registry; deploy to staging
[ 7. Deploy  ]  (Delivery = manual approval → prod;  Deployment = automatic → prod)
// Fast feedback is the goal: a red pipeline blocks the merge/release.
⚡ Fail fast, cheap checks first. Order stages so the quickest, most likely to fail checks (lint, unit tests) run before slow expensive ones (integration, e2e, deploy). This gives developers feedback in minutes and avoids wasting compute on a change that a linter would've rejected. A slow pipeline is a pipeline people route around.

Core Concepts & Vocabulary

TermMeaning
Pipeline / WorkflowThe automated sequence triggered by an event (push, PR, tag, schedule)
JobA unit of the pipeline that runs on a runner (can run in parallel with other jobs)
StepA single command/action within a job
Runner / AgentThe machine (hosted or self-hosted) that executes jobs
ArtifactThe build output passed between stages / stored for deploy
Stage / GateA checkpoint that must pass (tests green, approval) to proceed

Interview Questions

Q1: Define Continuous Integration, Delivery, and Deployment.

Continuous Integration is the practice of automatically building and testing every change as it's merged into a shared trunk, so integration problems and regressions are caught immediately rather than accumulating. Continuous Delivery extends CI so that every change which passes the pipeline is automatically built into a deployable artifact and kept in a state where it could be released to production at any time — but the actual production release is a deliberate manual approval (a button press). Continuous Deployment goes one step further and removes that manual gate entirely: any change that passes the full automated pipeline is deployed to production automatically, with no human intervention. The key distinction between the two CDs is that single manual approval before prod: Delivery keeps it, Deployment automates it away. Both depend on a trustworthy automated pipeline, and Continuous Deployment additionally requires strong testing, monitoring, and safe rollout strategies because there's no human checkpoint.

Q2: Walk through the stages of a typical CI pipeline.

When a developer pushes or opens a PR, the pipeline triggers and runs stages in order of increasing cost. First it checks out the code, then builds it (installs dependencies, compiles or bundles). Next it runs tests, usually unit tests first (fast) then integration tests, failing fast on the first problem. Then static analysis and quality gates run: linting, type-checking, security and dependency scanning, and a code-coverage threshold. If everything passes, it packages a versioned artifact — a jar, a container image, or a bundle — which is the single build that will be promoted onward. That artifact is pushed to a registry and deployed to a staging environment for further verification. Finally comes production: under Continuous Delivery a human approves the release, while under Continuous Deployment it happens automatically. The guiding principles are fail-fast ordering (cheap checks first for quick feedback) and build-once-promote-many, so the exact artifact tested in staging is the one that reaches production.

Q3: Why is a fast, trustworthy pipeline so important?

Because the entire value of CI/CD comes from developers actually relying on the pipeline, and they only do that if it's fast and trustworthy. If the pipeline is slow, feedback loops stretch to tens of minutes or hours, so developers batch changes, context-switch, and effectively route around it — reintroducing the late-integration problems CI was meant to solve. If it's flaky (tests that fail intermittently), people start ignoring red builds and merging anyway, which destroys the signal and lets real breakage through. A fast pipeline (achieved with caching, parallelism, and fail-fast ordering) gives feedback in minutes so issues are caught while the change is fresh; a trustworthy pipeline (deterministic tests, quarantined flakiness, meaningful gates) means a green build genuinely implies the change is safe. That combination is what makes frequent, low-risk deployment possible — the whole point of continuous delivery.

🏋️ Practice: Sketch (or build) a pipeline for a small app with the stages checkout → install → lint → unit test → build → package → deploy-to-staging, ordered fail-fast. Identify which stages could run in parallel. Decide whether you want Continuous Delivery (manual prod gate) or Continuous Deployment (automatic) and justify it based on your test/monitoring maturity. Then deliberately break a lint rule and confirm the pipeline fails before the expensive stages run.

🎯 Key Takeaways

  • CI = auto build+test every change on the trunk; Continuous Delivery = always deployable with a manual prod gate; Continuous Deployment = auto to prod (no gate).
  • A pipeline runs stages on each push: checkout → build → test → analyze (lint/security/coverage) → package a versioned artifact → deliver/deploy.
  • Order stages fail-fast (cheap checks first) and build the artifact once, then promote the same one through environments.
  • Vocabulary: pipeline/workflow, job, step, runner/agent, artifact, stage/gate.
  • Speed and trustworthiness are essential — slow or flaky pipelines get bypassed or ignored, destroying CI's value.
