Supply-chain Security Advanced

Your pipeline builds the artifacts that run in production — which makes it a prime target. Attacks like SolarWinds and Codecov compromised the BUILD, not the running app. Supply-chain security is about trusting what you ship: knowing exactly what's in it (SBOM), proving how it was built (provenance), proving it hasn't been tampered with (signing), and hardening the pipeline itself. A senior/staff expectation and increasingly a compliance one.

In the Pipeline (what "good" looks like)

// A hardened build/release flow:
1. Pin dependencies (lockfiles) + verify integrity hashes; use Dependabot/Renovate to update safely.
2. Scan: SAST (code), SCA (dependencies/CVEs), secret scanning, IaC scanning — FAIL on high/critical.
3. Build reproducibly; generate an SBOM for the artifact.
4. Generate PROVENANCE (SLSA attestation): which commit, which workflow, which runner produced it.
5. SIGN the artifact + attestations (cosign/Sigstore — keyless via OIDC identity).
6. Push to the registry; at DEPLOY time, VERIFY the signature + provenance (admission policy) — reject
   anything unsigned or from an unexpected builder.

SLSA — Levels of Build Integrity

// SLSA (Supply-chain Levels for Software Artifacts) is a framework of increasing guarantees:
//   L1: provenance exists (you record how it was built)
//   L2: signed provenance from a hosted build service
//   L3: hardened, isolated, non-falsifiable builds (no tampering by the build itself)
// Higher levels = stronger proof that the artifact came from your source via your pipeline, unaltered.

Interview Questions

Q1: What is software supply-chain security and why does the pipeline matter?

Supply-chain security is about being able to trust the artifacts you ship — knowing what's in them, how and where they were built, and that they haven't been tampered with — across the whole path from source code and dependencies through the build system to the deployed artifact. The pipeline matters because it's privileged production infrastructure that assembles those artifacts, and real-world attacks (SolarWinds, Codecov) compromised the BUILD or a build-time tool rather than the running application, meaning code can pass every test and still be malicious if a dependency, a CI action, or the build environment was subverted. So the discipline shifts trust from "we tested the code" to provable guarantees: an SBOM for contents, provenance/attestation for how it was built, cryptographic signing for authenticity and integrity, and scanning for known vulnerabilities — plus hardening the pipeline itself with pinned actions, least-privilege short-lived credentials, protected branches, and isolated runners.

Q2: Explain SBOM, provenance, and signing and how they work together.

They answer three different trust questions. An SBOM (Software Bill of Materials) is a complete inventory of everything in the artifact — packages and versions — in a standard format (SPDX, CycloneDX), so you can answer "what's inside?" and rapidly determine whether you're affected when a new CVE drops. Provenance (an attestation, per frameworks like SLSA/in-toto) is a cryptographically recorded statement of HOW and WHERE the artifact was built — which source commit, which workflow, which runner produced it — so you can verify it truly came from your source through your pipeline. Signing (via cosign/Sigstore, often keyless using the pipeline's OIDC identity) attaches a signature to the artifact and its attestations so consumers can verify authenticity and that nothing was altered after build. Together they form a verifiable chain: the SBOM says what it contains, provenance says how it was made, and signatures prove both are genuine and untampered — and at deploy time an admission policy verifies the signature and expected provenance, rejecting anything unsigned or built by an unexpected builder.

Q3: How do you harden the CI/CD pipeline against attacks?

Treat the pipeline as a high-value production system and apply defense in depth. Control the inputs: pin dependencies with lockfiles and integrity hashes and update them via reviewed automation (Dependabot/Renovate), and critically pin third-party CI actions/plugins to a full commit SHA rather than a mutable tag, since a moved tag can silently swap in malicious code that runs with your secrets (the Codecov pattern). Minimize credentials: use short-lived, least-privilege tokens via OIDC federation instead of long-lived cloud keys, so there's little to steal and a small blast radius. Control the process: require protected branches and code review so no unreviewed code reaches the release path, and restrict who can modify pipelines or approve production releases. Isolate the build: use ephemeral, isolated runners to prevent cross-build contamination and tampering (aligning with higher SLSA levels). Add scanning gates (SAST, SCA/CVE, secret and IaC scanning) that fail on critical findings, and generate SBOMs, provenance, and signatures so downstream deploys can verify artifacts. The theme is least privilege, immutability/pinning, review, isolation, and verification at every step.

🏋️ Practice: generate an SBOM (syft) for a build artifact and scan it (trivy); produce SLSA provenance / a GitHub artifact attestation for an image; sign an image with cosign (keyless via OIDC) and verify it, then tamper and watch verification fail; pin a third-party GitHub Action from a tag to a commit SHA; add SAST/SCA/secret-scanning gates that fail on critical; add a deploy-time admission policy that rejects unsigned images.

🎯 Key Takeaways

  • The pipeline is a top attack target — real breaches (SolarWinds, Codecov) compromised the build, not the app; "tests passed" isn't security.
  • Four pillars: SBOM (what's inside), provenance/attestation (how/where built, SLSA), signing/verification (authentic & untampered, cosign/Sigstore), scanning (CVEs, SAST/SCA/secrets).
  • Good flow: pin+scan deps → build + SBOM → SLSA provenance → sign → push → verify signature+provenance at deploy (admission policy).
  • Harden the pipeline: pin actions to a SHA, OIDC short-lived least-privilege tokens, protected branches/reviews, isolated ephemeral runners.
  • SLSA levels escalate build-integrity guarantees (provenance → signed provenance → hardened non-falsifiable builds).
