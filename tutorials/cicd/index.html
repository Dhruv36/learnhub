Git Workflows: Trunk vs GitFlow Basics

CI/CD starts with how your team uses Git, because the branching model dictates how often you integrate, how fast you can ship, and how much merge pain you endure. The industry has largely converged on trunk-based development with short-lived branches — and understanding why it beats heavier models like GitFlow is the foundation for everything else in this track.

The Main Branching Models

ModelHow it works
Trunk-basedEveryone commits to main (trunk) frequently via tiny, short-lived branches merged within hours/a day. The mainstream model for CI/CD.
GitHub FlowA lightweight variant: branch off main → PR → merge → deploy. Simple, common for web apps/SaaS.
GitFlowLong-lived develop, release, feature, hotfix branches. Heavy; suited to versioned/desktop software with scheduled releases.
🌳 Continuous Integration means integrating continuously. The whole point of CI is to merge everyone's work into a shared trunk often — ideally many times a day — so conflicts and integration bugs surface small and early. Long-lived branches (GitFlow's develop/release) delay integration, causing painful "big bang" merges and drift. That's why high-performing teams favor trunk-based development.

Keeping Trunk Releasable

TechniquePurpose
Branch protection + required PR reviewsNo direct pushes to main; every change reviewed
Required status checks (CI must pass)Can't merge if tests/lint/build fail
Feature flagsMerge incomplete work safely — hidden behind a flag (see later lesson)
Small PRsFaster review, smaller blast radius, easier revert
🚩 Feature flags decouple "merge" from "release." The objection to trunk-based development is "but my feature isn't done." The answer is to merge it behind a feature flag — the code is integrated and tested continuously, but the feature stays off until you flip the flag. This is what lets teams commit to trunk many times a day without shipping half-built features.

Merge vs Rebase, and PR Hygiene

git merge   → preserves history as-is, adds a merge commit (keeps context, can be noisy)
git rebase  → replays your commits on top of main (linear history, cleaner, don't rebase shared branches)
// Common team convention: rebase your short-lived branch onto main to stay current, then
// "squash and merge" the PR so main gets one clean commit per change.
// Golden rule: never rebase/force-push a branch others are working on.

Interview Questions

Q1: What is trunk-based development and why is it preferred for CI/CD?

Trunk-based development is a branching model where all developers integrate their work into a single shared branch (the trunk, usually main) very frequently — typically through tiny, short-lived branches that are reviewed and merged within hours or at most a day. It's preferred for CI/CD because continuous integration literally means integrating continuously: merging small changes often makes conflicts and integration bugs surface early and small, keeps the trunk always healthy and releasable, and enables a fast, low-risk deployment cadence. The alternative — long-lived feature branches (as in GitFlow) — delays integration, so branches drift apart and you get painful "big bang" merges and "works on my branch" surprises. Trunk-based development pairs with branch protection, required CI status checks, small PRs, and feature flags (to merge unfinished work safely), which together let a team commit to trunk many times a day while keeping it deployable.

Q2: When might GitFlow still make sense?

GitFlow, with its long-lived develop, release, feature, and hotfix branches, makes sense when you have genuinely scheduled, versioned releases and need to support multiple versions in parallel — classic examples are packaged desktop software, mobile apps with store release cycles, libraries/SDKs with maintained release lines, or embedded/firmware where you can't just "deploy main." In those contexts the explicit release and hotfix branches give you a place to stabilize a version, backport fixes, and manage several supported versions at once. For modern web apps and SaaS that deploy continuously from a single production line, GitFlow's ceremony is overhead that slows integration and delivery, so trunk-based development or the lighter GitHub Flow is the better fit. The deciding factor is your release model: continuous single-version delivery favors trunk-based; scheduled multi-version releases can justify GitFlow.

Q3: How do you keep the trunk always releasable while people merge frequently?

Through a combination of automation and discipline. Branch protection prevents direct pushes to main and requires pull-request review, and required status checks mean a PR can't merge unless CI (tests, lint, build, security scans) passes — so broken code never lands on trunk. Developers keep PRs small and short-lived so they're quick to review, low-risk, and easy to revert if something slips through. For work that spans multiple merges or isn't ready to be seen by users, feature flags let the code be merged and continuously integrated while the feature stays turned off, decoupling "merged" from "released." Teams also often rebase their short-lived branch onto main and squash-merge for a clean history. The net effect is that every commit on trunk has passed CI and been reviewed, incomplete work is hidden behind flags, and the trunk stays in a deployable state at all times — which is the prerequisite for continuous delivery.

🏋️ Practice: On a repo, enable branch protection on main: require a PR, at least one review, and a passing status check. Make a change on a short-lived branch, open a PR, and merge with "squash and merge." Practice rebasing your branch onto an updated main to resolve a conflict early. Then simulate the alternative: let a branch diverge for many commits and feel the merge pain — contrasting it with the small-merge trunk-based flow.

🎯 Key Takeaways

  • Trunk-based development (frequent small merges to main via short-lived branches) is the mainstream model for CI/CD; GitHub Flow is its lightweight PR-based variant.
  • CI means integrating continuously — long-lived branches (GitFlow) delay integration and cause painful big-bang merges and drift.
  • Keep trunk releasable with branch protection, required PR reviews, required passing CI checks, and small PRs.
  • Feature flags decouple merge from release, letting you integrate unfinished work safely (off until flipped).
  • GitFlow still fits scheduled, versioned, multi-version software (desktop/mobile/libraries); rebase your own short-lived branch but never force-push shared branches.
