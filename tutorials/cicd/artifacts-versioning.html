Artifacts & Versioning Intermediate

A pipeline produces artifacts — the built, deployable outputs — and how you version and promote them determines whether "what you tested" is truly "what you ship." The two principles that matter most are build once, promote many and immutable, uniquely-versioned artifacts. Get these right and you eliminate a whole category of "works in staging, breaks in prod" incidents.

What Counts as an Artifact

TypeExample
Container imageA Docker image in a registry (ECR, GHCR, Docker Hub)
Compiled packageA jar/war, a Go binary, a .NET assembly
Language packageAn npm/PyPI/NuGet package in a registry
Frontend bundleA built static bundle (uploaded to S3/CDN)

Build Once, Promote Many (the golden rule)

BAD:  build for staging → test → build AGAIN for prod → deploy
      // prod runs a DIFFERENT binary than the one you tested. Bugs slip through the gap.

GOOD: build ONE artifact  ──►  deploy to staging (test it)  ──►  promote the SAME artifact to prod
      // The exact bytes you validated in staging are what runs in production.
// Config (env vars/secrets) changes per environment — the ARTIFACT does not.
📦 Separate the artifact from its configuration. Build a single immutable artifact and inject environment-specific config (DB URLs, secrets, feature flags) at deploy/run time. This is the classic "one build, config per environment" principle (a Twelve-Factor idea). Rebuilding per environment means prod runs code that was never tested exactly as-is — the source of subtle, environment-specific bugs.

Immutability & Unique Versioning

// Every artifact gets a UNIQUE, IMMUTABLE version — never overwrite an existing one.
myapp:1.4.2                 // semantic version tag (releases)
myapp:git-9f3a1c            // commit SHA (traceable to exact source)
myapp:2026.07.11-build.87   // date + build number
// AVOID `latest` for deploys: it's a MOVING tag → you can't tell what's actually running,
// and rollbacks become guesswork. Pin to an immutable version.
🏷️ Tag artifacts with the commit SHA for traceability. Being able to map a running artifact back to the exact source commit (and pipeline run) is invaluable for debugging incidents and audits. Immutable versions also make rollback trivial and reliable: "deploy 1.4.1" is deterministic, whereas re-deploying a mutable latest may pull something different than before.

Semantic Versioning

MAJOR . MINOR . PATCH   (e.g., 2.5.3)
  MAJOR ↑  breaking change (incompatible API)
  MINOR ↑  new feature, backward-compatible
  PATCH ↑  backward-compatible bug fix
// Communicates compatibility to consumers. Great for libraries/APIs; apps often use SHA/date
// schemes since "compatibility" is less meaningful for a deployed service.

Interview Questions

Q1: What does "build once, promote many" mean and why does it matter?

It means the pipeline produces a single artifact — one container image or package — and that same artifact is promoted through each environment: deployed to staging, tested there, and then the identical bytes are deployed to production, rather than rebuilding separately for each environment. It matters because if you rebuild for prod, you're shipping a binary that was never tested exactly as-is — even with the same source, differences in dependencies, build environment, or timing can introduce discrepancies, so a bug can slip through the gap between "what you tested" and "what you ran." Promoting one immutable artifact guarantees that the thing validated in staging is exactly the thing running in production, eliminating that class of environment-specific surprises. The corollary is that only configuration (environment variables, secrets, connection strings, feature-flag state) varies per environment and is injected at deploy or run time; the artifact itself stays constant. This gives reproducibility, reliable rollbacks, and clean traceability.

Q2: Why avoid deploying the latest tag, and how should you version artifacts?

latest is a mutable, moving pointer: it can point to different images over time, so you can't reliably tell what's actually running in production, and rollbacks become guesswork because re-pulling latest may fetch something different than before. Instead, every artifact should get a unique, immutable version that's never overwritten — for example a semantic version tag for releases, the commit SHA for exact source traceability, or a date-plus-build-number scheme. Tagging with the commit SHA is especially valuable because it maps a running artifact straight back to the source commit and pipeline run, which is essential during incident debugging and audits. Immutable versions make deployments deterministic ("deploy 1.4.1" always means the same bytes) and rollbacks trivial and reliable. So the rule is: pin deployments to immutable, uniquely-identified versions, keep a clear mapping to source, and never reuse or overwrite a version tag.

Q3: Explain semantic versioning and when it's the right scheme.

Semantic versioning uses MAJOR.MINOR.PATCH to communicate compatibility: you bump MAJOR for a breaking, backward-incompatible change, MINOR for new backward-compatible functionality, and PATCH for backward-compatible bug fixes. Its purpose is to let consumers understand the risk of upgrading at a glance — a patch or minor bump should be safe, a major bump signals they may need to change their code. That makes SemVer ideal for libraries, SDKs, and public APIs, where other people depend on your interface and need clear compatibility signals (and it underpins dependency version ranges in package managers). For a deployed application or service, the notion of "compatibility" is less meaningful — users don't "upgrade" your service the way they upgrade a library — so apps often prefer versioning schemes based on the commit SHA or a date/build-number for traceability and immutability instead. The key is to match the scheme to the consumer: SemVer to communicate compatibility for things others depend on, immutable SHA/date tags for deployable app artifacts.

🏋️ Practice: Change your pipeline to build a container image once, tagged with both a semantic version and the commit SHA, push it to a registry, deploy it to staging, then promote the same image tag to prod (no rebuild). Move all environment-specific config to env vars/secrets injected at deploy time. Practice a rollback by redeploying a previous immutable tag. Confirm you can map the running image back to its source commit.

🎯 Key Takeaways

  • An artifact is the built, deployable output (container image, package, binary, bundle) stored in a registry.
  • Build once, promote many: deploy and test one immutable artifact, then promote the exact same bytes to prod — only config varies per environment.
  • Give every artifact a unique, immutable version; never overwrite one and avoid deploying the mutable latest tag.
  • Tag with the commit SHA for traceability (map running code → source); immutable versions make rollbacks deterministic.
  • Use SemVer (MAJOR.MINOR.PATCH) to signal compatibility for libraries/APIs; apps often use SHA/date schemes.
