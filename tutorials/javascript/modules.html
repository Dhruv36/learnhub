Modules (ESM & CommonJS) Modules & Tooling

What you'll master: ES Modules (the standard) — named vs default exports, import syntax, how module scope and singletons work, dynamic import() for code splitting — plus CommonJS (Node's legacy system) and the interop pain between them. Prerequisites: Functions & Scope (module scope) and Variables (modules are strict-mode). Modules are how real codebases are organised; understanding them is essential for any non-trivial project and for reading the errors tooling throws at you.

Part 1 — Why Modules Exist

// Before modules: everything shared ONE global scope. Scripts leaked variables, collided, and
// depended on load ORDER in the HTML. A jQuery plugin could clobber your `$`.

// Modules give each file its OWN scope and EXPLICIT dependencies:
// - variables are private to the file unless exported
// - dependencies are declared with import, not assumed to be globally present
// - load order is resolved automatically from the import graph

Before modules, every script shared the global scope: files leaked variables into window, names collided, and everything depended on the order of <script> tags. Modules fix this by giving each file its own scope — top-level declarations are private unless explicitly exported — and making dependencies explicit via import. The bundler/runtime builds a dependency graph and loads things in the right order automatically. This is the foundation of maintainable code organisation, and it's why every modern project is module-based.

Part 2 — Named Exports & Imports

// math.js — export multiple named things:
export const PI = 3.14159;
export function add(a, b) { return a + b; }
export class Vector { /* ... */ }
// or export a list at the bottom:
const sub = (a, b) => a - b;
export { sub };

// app.js — import by NAME (names must match; curly braces):
import { PI, add, Vector } from "./math.js";
import { add as sum } from "./math.js";        // rename on import
import * as math from "./math.js";              // namespace import → math.PI, math.add
add(2, 3);   // 5

Named exports let a module expose multiple values, each by name. You import them by their exact names inside curly braces (renaming with as if needed), or import everything as a namespace object with import * as name. Named exports are the default choice for utility modules exposing several functions/constants — they're explicit (you see exactly what's imported), support tree-shaking (bundlers can drop unused exports — Tooling lesson), and give better editor autocomplete. Note the .js extension is required in standard ESM import paths in the browser and Node.

Part 3 — Default Exports

// Button.js — ONE default export per file:
export default function Button(props) { /* ... */ }
// or: export default class Button {}   or   const x = {}; export default x;

// app.js — import a default with ANY name you choose (no braces):
import Button from "./Button.js";
import MyButton from "./Button.js";       // same thing, different local name — no error

// A file can have BOTH:
import Button, { ButtonGroup, SIZES } from "./Button.js";   // default + named

A default export is the single "main" thing a module provides (one per file). You import it without braces and can name it anything — the import name isn't tied to the export name, which is both convenient and a mild footgun (two files can import the same default under different names, hurting greppability). Convention: use a default export when a module is fundamentally one thing (a React component, a class, a config object); use named exports for collections of utilities. A file can mix both. Many style guides actually prefer named exports even for single things, precisely because the enforced consistent naming aids refactoring and search.

Part 4 — Module Scope, Singletons & Live Bindings

// counter.js — module-level state is a SINGLETON, shared by every importer:
let count = 0;
export function increment() { return ++count; }
export function getCount() { return count; }

// a.js and b.js both import the SAME instance:
import { increment, getCount } from "./counter.js";
increment();  // in a.js → count is 1
// in b.js: getCount() → 1  — SAME module, SAME state (modules are evaluated ONCE and cached)

// Imports are LIVE, read-only bindings (not copies):
import { count } from "./counter.js";   // if counter.js reassigns count, you SEE the new value
// count = 5;   // ❌ TypeError — imported bindings are read-only

Two important semantics. First, a module is evaluated exactly once and its exports are cached — so module-level state is effectively a singleton shared across everything that imports it. This is how you make a shared config, a single database connection, or a global store (import it anywhere, get the same instance). Second, imports are live, read-only bindings, not copies: if the exporting module changes an exported variable, importers see the updated value (a "live binding") — but importers cannot reassign it (TypeError). This differs from CommonJS's value-copy semantics (Part 7) and matters for understanding circular dependencies and shared state.

Part 5 — Using ESM in the Browser and Node

<!-- Browser: mark the script as a module -->
<script type="module" src="app.js"></script>
<!-- module scripts are deferred, strict-mode, and same-origin/CORS-restricted -->

// Node: opt into ESM via package.json OR the .mjs extension
// package.json: { "type": "module" }   → .js files are treated as ESM
// otherwise .mjs = ESM, .cjs = CommonJS

In the browser, <script type="module"> enables ESM — such scripts are automatically deferred (run after HTML parsing), run in strict mode (Variables lesson), and have their own scope (top-level this is undefined, not window). In Node, you opt into ESM by setting "type": "module" in package.json (making .js files ESM) or using the .mjs extension; .cjs forces CommonJS. This dual-system reality is the source of much configuration pain (Part 7), so knowing which mode a file is in matters.

Part 6 — Dynamic import() for Code Splitting

// Static imports are resolved at load time (top of the file). Dynamic import() loads ON DEMAND:
button.addEventListener("click", async () => {
  const { renderChart } = await import("./chart.js");   // loads chart.js ONLY when clicked
  renderChart(data);
});

// import() returns a PROMISE of the module namespace — great for:
// - code splitting: don't ship the whole app upfront; load routes/features lazily
// - conditional loading: only load a heavy polyfill/library if actually needed
if (!window.IntersectionObserver) await import("./io-polyfill.js");

Dynamic import() loads a module at runtime and returns a promise of its namespace — unlike static import (hoisted, resolved upfront). This enables code splitting: instead of bundling your entire app into one big file the user downloads before anything works, you lazy-load features/routes only when needed — the heavy chart library loads when the user opens the chart, not on initial page load. It's the single most impactful performance technique for large apps (smaller initial bundle = faster first load), and it's how framework routers (React.lazy, Vue async components) split code per route. Also useful for conditional loading (a polyfill only when the feature is missing).

Part 7 — CommonJS & the Interop Pain

// CommonJS (Node's original system) — SYNCHRONOUS require, value COPIES:
const fs = require("fs");                  // import
const { readFile } = require("fs");        // destructure
module.exports = { add, sub };             // export (assign to module.exports)
exports.helper = () => {};                 // shorthand for a single addition

// The interop reality:
// - ESM can import CommonJS: import cjs from "./legacy.cjs"  (default = module.exports)
// - CommonJS CANNOT require() ESM synchronously (ESM is async) → must use dynamic import()
// - Named imports from CJS may not work (CJS has no static named exports to analyze)

CommonJS (CJS) is Node's original module system: require() to import, module.exports to export. Key differences from ESM: require is synchronous and returns a copy of the exports (not live bindings), and exports are resolved at runtime (not statically analysable, so no tree-shaking). The interop between the two is a genuine, ongoing pain point: ESM can import CJS (the whole module.exports becomes the default import), but CJS cannot synchronously require() an ESM module (because ESM loads asynchronously) — it must use dynamic import(). This mismatch causes most of the "Cannot use import statement outside a module" and "require is not defined" errors you'll hit. The industry is migrating to ESM, but CJS is everywhere in existing npm packages, so you'll navigate both.

Part 8 — Build It: a Well-Structured Feature Module

Let's structure a small feature the way real code is organised — a public API module re-exporting from internal files (the "barrel" pattern), with a lazy-loaded heavy part — showing named exports, module singletons, and dynamic import together.

// ── store.js — a module singleton (shared state, imported anywhere) ──
let state = { items: [] };
const listeners = new Set();
export function getState() { return state; }
export function setState(next) { state = next; listeners.forEach(fn => fn(state)); }
export function subscribe(fn) { listeners.add(fn); return () => listeners.delete(fn); }

// ── api.js — named exports for the data layer ──
export async function fetchItems() {
  const res = await fetch("/api/items");
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

// ── index.js — the BARREL: one public entry point re-exporting the feature's API ──
export { getState, setState, subscribe } from "./store.js";
export { fetchItems } from "./api.js";
export async function loadCharts() {
  const { renderChart } = await import("./chart.js");   // heavy dep, lazy-loaded on demand
  return renderChart;
}

// ── consumer: imports the clean public surface, unaware of internal file layout ──
import { fetchItems, subscribe, setState, loadCharts } from "./feature/index.js";
subscribe(s => console.log("state changed", s));
setState({ items: await fetchItems() });
const renderChart = await loadCharts();   // chart.js downloads only now

This is how features are actually organised. store.js is a module singleton — everything importing it shares one state object and subscriber set (Part 4). api.js uses named exports for the data functions. index.js is a barrel — a single public entry point re-exporting the internals, so consumers import from one clean path and the internal file structure can change freely without breaking them. And loadCharts uses dynamic import() to defer the heavy charting code until it's needed. This encapsulation-through-modules is the backbone of a maintainable codebase, and every technique here is from this lesson.

Interview Questions

Q1 (beginner): What's the difference between a named export and a default export?

A module can have many named exports (each exposed by a specific name) and at most one default export. You import named exports by their exact names inside curly braces (import { add } from "./math.js", renamable with as), and the default export without braces under any local name you choose (import anything from "./Button.js"). Named exports are explicit, tree-shakeable, and enforce consistent naming; default exports suit a module whose purpose is one main thing (a component, a class). A file can have both.

Q2 (beginner): What is module scope?

Each ES module has its own top-level scope: variables, functions, and classes declared at the top level of a module file are private to that file and not added to the global scope, unless explicitly exported. This is the core benefit of modules over old global scripts — no accidental global leaks or name collisions between files. Modules also run in strict mode automatically, and in the browser a module's top-level this is undefined rather than window. To share something between files you export it and import it, making dependencies explicit.

Q3 (mid): Explain module singletons and live bindings.

A module is evaluated only once; the result is cached, so every file that imports it gets the same module instance and shares its top-level state — effectively a singleton. This is how a shared store, config, or single DB connection works: import it anywhere and you touch the same object. Live bindings mean imported names are read-only references to the exporting module's bindings, not copies: if the exporting module reassigns an exported let, importers see the new value (they observe changes live), though they can't reassign the imported binding themselves (TypeError). This contrasts with CommonJS, where require returns a copy of the exports at require-time, so later reassignments in the exporter aren't reflected. Live bindings matter for understanding shared mutable state and circular-dependency behaviour.

Q4 (mid): What is dynamic import() and when do you use it?

import() is a function-like form that loads a module at runtime and returns a promise resolving to the module's namespace object — unlike static import, which is hoisted and resolved before the module runs. You use it for code splitting: rather than bundling the whole app into one file loaded upfront, you lazy-load features, routes, or heavy libraries only when needed, so the initial bundle is smaller and the page loads faster (e.g. load a charting library when the user opens the chart, not on first paint). It's also used for conditional loading — pulling in a polyfill only if a feature is missing, or loading locale data on demand. Framework routers use it to split code per route (React.lazy, Vue async components). The trade-off is the loaded code arrives asynchronously, so you handle a loading state.

Q5 (senior): Explain the ESM/CommonJS interop problems and how to navigate them.

The systems differ fundamentally: ESM is asynchronous and statically analysed with live read-only bindings; CommonJS is synchronous with runtime require returning value copies. The interop asymmetry: ESM can import CommonJS — Node wraps the CJS module.exports as the ESM default export (and attempts to expose named exports via static analysis, which doesn't always work because CJS exports are dynamic, so named imports from a CJS package sometimes fail and you must import the default then destructure). But CommonJS cannot synchronously require() an ES module, because ESM loads asynchronously and require is synchronous — so CJS code must use dynamic import() (which returns a promise) to load ESM, meaning the consuming function becomes async. This produces the common errors: "Cannot use import statement outside a module" (running ESM syntax in a CJS context), "require is not defined" (using CJS in an ESM context), and "does not provide an export named X" (bad named import from CJS). Navigation strategies: (1) set the module system explicitly — "type":"module" for ESM or use .mjs/.cjs extensions to be unambiguous; (2) for dual-published packages, ship both builds and use package.json "exports" conditional paths so ESM consumers get the ESM build and CJS consumers the CJS build; (3) when importing a CJS package from ESM and named imports fail, import pkg from "x"; const { thing } = pkg;; (4) prefer authoring new code as ESM (the standard direction), transpiling/bundling for compatibility; (5) avoid mixing in the same file. The senior reality is that this friction is a transitional cost of the ecosystem migrating from CJS to ESM — tooling (bundlers, Node's conditional exports) largely papers over it, but understanding the underlying sync-vs-async and copy-vs-live-binding distinction is what lets you diagnose the errors rather than guessing at config.

Q6 (senior): How does the choice between static and dynamic imports affect bundling and performance, and how do you decide what to split?

Static imports are resolved at build time, so a bundler can follow the whole graph, tree-shake unused exports, and pack everything into the initial bundle(s); dynamic import() creates a split point — the bundler emits the dynamically-imported module (and its unique dependencies) as a separate chunk loaded on demand at runtime. Performance impact: everything statically imported is downloaded, parsed, and executed before the app is interactive, so a large static graph means a big initial bundle and slow first load/Time-to-Interactive; moving code behind dynamic imports shrinks the initial bundle and defers that cost until the feature is actually used. Deciding what to split: (1) split by route — each page/route is a natural chunk most users won't visit all of, the highest-value split. (2) Split heavy, rarely-used features — a rich text editor, a charting library, a PDF viewer, an admin panel — anything large that only some users or some interactions need. (3) Split below-the-fold or interaction-triggered UI (modals, wizards) that isn't needed on first paint. (4) Conditionally load polyfills only for browsers that lack a feature. Cautions: don't over-split — each dynamic chunk is an extra network request with latency, so splitting tiny modules hurts more than helps (bundlers can be told to merge small chunks); pre-fetch likely-next chunks (e.g. on hover/idle) to hide the load latency; and ensure shared dependencies are factored into a common chunk rather than duplicated across split points. The decision framework: measure the initial bundle, identify the largest modules not needed for first paint, split those at route or feature boundaries, and prefetch predictable ones — balancing smaller initial load against the request overhead of many chunks. This is exactly what framework routers and bundler configs (Webpack/Rollup/Vite) automate, but the judgment of what to split by usage patterns is the engineering call.

🏋️ Exercises

1 (easy): Export and import. Write a math.js exporting add and PI (named) and a circle.js that imports both and computes a circle's area.
Solution 
// math.js
export const PI = 3.14159;
export const add = (a, b) => a + b;

// circle.js
import { PI } from "./math.js";
export const area = (r) => PI * r * r;

Named exports are imported by matching name in braces; each file has its own scope, sharing only what's exported/imported.

2 (easy): Default export. Write a logger.js with a default-exported logging function, and import it under the name log.
Solution 
// logger.js
export default function (msg) { console.log(`[LOG] ${msg}`); }

// app.js
import log from "./logger.js";   // any name — no braces for default
log("hello");

A default export imports without braces and can be named anything at the import site — here log.

3 (medium): Module singleton. Write a counter.js module whose count is shared across all importers, and demonstrate two importers seeing the same value.
Solution 
// counter.js
let count = 0;
export const increment = () => ++count;
export const get = () => count;

// a.js
import { increment } from "./counter.js";
increment(); increment();

// b.js
import { get } from "./counter.js";
console.log(get());   // 2 — same module instance, shared state

The module is evaluated once and cached, so a.js and b.js share the same count — a singleton by virtue of module semantics.

4 (medium): Lazy load. Load a heavy ./editor.js module only when a button is clicked, then call its exported initEditor.
Solution 
button.addEventListener("click", async () => {
  const { initEditor } = await import("./editor.js");   // downloads only now
  initEditor(document.querySelector("#editor"));
});

Dynamic import() defers downloading editor.js until the click, keeping it out of the initial bundle — code splitting on a user interaction.

5 (hard): Barrel module. Create a feature folder with store.js, api.js, and an index.js barrel that re-exports a clean public API, and show a consumer importing only from the barrel.
Solution 
// feature/store.js
export const store = { items: [] };
// feature/api.js
export const fetchItems = async () => (await fetch("/api/items")).json();
// feature/index.js  (the barrel)
export { store } from "./store.js";
export { fetchItems } from "./api.js";

// consumer.js — imports the public surface, not internal files
import { store, fetchItems } from "./feature/index.js";

The barrel (index.js) re-exports the feature's public API from one path, so consumers don't depend on the internal file layout — you can reorganise store.js/api.js freely without breaking them. (Note: overly large barrels can hurt tree-shaking — a real trade-off.)

6 (hard): Diagnose interop errors. A Node project gets "Cannot use import statement outside a module" and, after fixing that, "require is not defined." Explain both and how to resolve them consistently.
Solution

"Cannot use import statement outside a module" means a file using ESM import syntax is being interpreted as CommonJS — Node defaults .js to CJS unless told otherwise. Fix: add "type": "module" to package.json (making .js ESM) or rename the file to .mjs. But once the project is ESM, "require is not defined" appears in any file still using CommonJS require — because require doesn't exist in ESM scope. Resolve consistently by committing to one system: for an ESM project, convert require(x) to import ... from "x" (and module.exports to export); where you must load a CJS-only dependency that doesn't play well, use import pkg from "cjs-pkg" (default) or dynamic import(); and for the rare file that must stay CJS, give it a .cjs extension. The root cause both times is a file's actual module mode not matching its syntax — the fix is making the mode explicit and consistent across the project.

🎯 Key Takeaways

  • Modules give each file private scope and explicit dependencies, replacing the global-scope, load-order chaos of old scripts.
  • Named exports (braces, exact names, tree-shakeable) for collections; default exports (no braces, any name) for a module's one main thing.
  • Modules evaluate once and cache — module-level state is a shared singleton; imports are live, read-only bindings, not copies.
  • Enable ESM via <script type="module"> (browser) or "type":"module"/.mjs (Node); module scripts are deferred and strict.
  • Dynamic import() returns a promise and enables code splitting — lazy-load routes/heavy features to shrink the initial bundle.
  • CommonJS (require/module.exports, sync, value copies) coexists with ESM; CJS can't synchronously require ESM — the interop mismatch causes most module errors.
