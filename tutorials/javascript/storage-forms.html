Forms, Storage & Web APIs Browser & the Web Platform

What you'll master: reading and submitting forms with FormData, client-side validation (and why it's never enough), the browser storage options (localStorage/sessionStorage/cookies/IndexedDB) and when each fits, and a tour of the modern Web APIs worth knowing (observers, clipboard, geolocation). Prerequisites: The DOM, Events, and Fetch (submitting forms). This lesson connects JavaScript to the browser platform's built-in capabilities.

Part 1 — Reading Forms with FormData

form.addEventListener("submit", (e) => {
  e.preventDefault();                              // stop the default page reload (Events lesson)
  const data = new FormData(form);                 // collects all named fields

  data.get("email");                                // one field's value
  data.getAll("tags");                              // multiple values (checkboxes, multi-select)
  Object.fromEntries(data);                         // → plain object { email, name, ... } (single values)

  // Send it (Fetch lesson):
  fetch("/api/submit", { method: "POST", body: data });   // multipart — handles file inputs too!
});

FormData is the clean way to read a form: new FormData(form) collects every field with a name attribute (which is why name, not just id, matters on inputs). Object.fromEntries(data) turns it into a plain object for JSON APIs, or you can pass the FormData directly as a fetch body for multipart uploads (it handles <input type="file"> automatically — and remember from the Fetch lesson: don't set Content-Type manually for FormData). Use getAll for fields with multiple values (checkbox groups, multi-selects).

Part 2 — Reading Individual Inputs

input.value                    // text/number/etc. (ALWAYS a string — Number() it for math)
checkbox.checked               // boolean
select.value                   // selected option's value
radio group: form.querySelector('input[name="plan"]:checked')?.value
fileInput.files                // a FileList — files[0] is a File object

input.addEventListener("input", (e) => console.log(e.target.value));  // fires on every keystroke
input.addEventListener("change", (e) => console.log(e.target.value)); // fires on blur/commit

Individual inputs expose their state via properties: value (always a string — coerce with Number() for arithmetic, per the Types lesson), checked for checkboxes/radios, files for file inputs. Two events to distinguish: input fires on every change (each keystroke — good for live validation/search, often debounced), while change fires when the value is committed (on blur, or selecting an option). Pick based on whether you want continuous or settled updates.

Part 3 — Validation: Client-Side Is UX, Not Security

// Built-in HTML validation (HTML track) — check it from JS:
form.checkValidity();          // true if all constraints pass
input.validity.valueMissing;   // required but empty?
input.validity.typeMismatch;   // e.g. invalid email format
input.setCustomValidity("Passwords must match");  // custom error message
input.reportValidity();        // show the browser's validation UI

// Custom validation on submit:
form.addEventListener("submit", (e) => {
  if (password.value !== confirm.value) {
    e.preventDefault();
    confirm.setCustomValidity("Passwords don't match");
    confirm.reportValidity();
  }
});

The browser gives you free validation via HTML constraints (required, type, pattern, min/max — the HTML track), readable from JS via checkValidity(), the validity object, and setCustomValidity/reportValidity for custom rules and messages. But the non-negotiable principle: client-side validation is for user experience, never for security. It gives instant feedback and reduces bad requests, but anyone can bypass it (disable JS, use curl, edit the DOM), so the server MUST re-validate everything. Client validation improves the experience; server validation enforces correctness — you always need both, and the server one is the real one.

Part 4 — Browser Storage: Four Options

// localStorage — synchronous, STRINGS ONLY (serialise objects yourself):
localStorage.setItem("theme", "dark");
localStorage.getItem("theme");                       // "dark" (or null if absent)
localStorage.setItem("user", JSON.stringify(user));  // objects → JSON string
JSON.parse(localStorage.getItem("user") ?? "null");  // parse back (guard null)
localStorage.removeItem("theme");  localStorage.clear();

Four storage mechanisms for four jobs. localStorage: simple key-value, persists until cleared, per-origin — great for preferences and non-sensitive UI state, but it's synchronous (can block on large data) and strings only (JSON-serialise objects). sessionStorage: same API but cleared when the tab closes — per-tab temporary state. Cookies: small, and automatically sent to the server with every request — the right place for auth/session tokens (as HttpOnly so JS can't read them, protecting against XSS token theft — Security lesson). IndexedDB: an asynchronous, transactional database for large or structured data and offline apps. Rule: never store sensitive data (tokens, PII) in localStorage — it's readable by any script, so an XSS steals it; use HttpOnly cookies for auth.

⚠️ localStorage is synchronous and XSS-readable. Reading/ writing large values blocks the main thread, and any injected script can read everything in it — so it's for non-sensitive, small-ish data only. Auth tokens belong in HttpOnly cookies; large data belongs in IndexedDB (async).

Part 5 — Useful Observer APIs

// IntersectionObserver — efficiently detect when elements enter the viewport (lazy-load, infinite scroll):
const io = new IntersectionObserver((entries) => {
  for (const entry of entries) {
    if (entry.isIntersecting) { loadImage(entry.target); io.unobserve(entry.target); }
  }
}, { rootMargin: "200px" });          // start loading 200px before it's visible
document.querySelectorAll("img[data-src]").forEach(img => io.observe(img));

// ResizeObserver — react to an element's size changing (not just window resize):
new ResizeObserver((entries) => entries.forEach(e => adjust(e.contentRect))).observe(el);

// MutationObserver — watch for DOM changes (nodes added/removed, attributes changed):
new MutationObserver((mutations) => { /* react */ }).observe(el, { childList: true, subtree: true });

The observer APIs replace inefficient polling and scroll-listener hacks with the browser telling you when something happens. IntersectionObserver reports when elements enter/leave the viewport — the correct, performant way to do lazy-loading images and infinite scroll (versus a scroll listener computing positions on every frame). ResizeObserver fires when an element's size changes (even without a window resize). MutationObserver watches for DOM changes. All three are asynchronous, batched, and far more efficient than the manual alternatives they replaced — reach for them instead of scroll/resize/polling loops.

Part 6 — A Few More Platform APIs

// Clipboard (async, requires a user gesture + permission):
await navigator.clipboard.writeText("copied!");
const text = await navigator.clipboard.readText();

// Geolocation (permission-gated):
navigator.geolocation.getCurrentPosition(
  (pos) => console.log(pos.coords.latitude, pos.coords.longitude),
  (err) => console.error(err.message)
);

// URL & query params — parse and build safely:
const url = new URL(location.href);
url.searchParams.get("page");                 // read ?page=
url.searchParams.set("page", "2");            // update — auto-encodes
history.pushState({}, "", url);               // update the address bar without reload (SPA routing)

// Structured cloning, crypto, timers:
structuredClone(obj);                          // deep clone (Objects lesson)
crypto.randomUUID();                           // secure unique id
crypto.getRandomValues(new Uint8Array(16));    // secure random bytes (NOT Math.random — Numbers lesson)

A grab-bag of platform capabilities you'll reach for. navigator.clipboard (async, gesture-gated) for copy/paste; geolocation (permission-prompted) for location. The URL and URLSearchParams APIs are the correct way to parse/build URLs and query strings — they handle encoding so you don't concatenate strings and create injection/encoding bugs. history.pushState powers client-side routing (change the URL without reloading). And for anything security-sensitive, crypto.randomUUID()/ getRandomValues give cryptographically-secure randomness — never Math.random for tokens (Numbers & Security lessons).

Part 7 — Build It: a Persisted, Validated Settings Form

Let's tie it together: a settings form that validates on submit, persists to localStorage, restores on load, and syncs across tabs via the storage event — a complete, real client-side feature.

function initSettings(form) {
  const KEY = "app:settings";

  // 1. RESTORE saved settings on load (guard against corrupt/missing data):
  try {
    const saved = JSON.parse(localStorage.getItem(KEY) ?? "{}");
    for (const [name, value] of Object.entries(saved)) {
      const field = form.elements[name];
      if (field) field.type === "checkbox" ? (field.checked = value) : (field.value = value);
    }
  } catch { /* ignore corrupt storage */ }

  // 2. VALIDATE + SAVE on submit:
  form.addEventListener("submit", (e) => {
    e.preventDefault();
    if (!form.checkValidity()) { form.reportValidity(); return; }   // built-in validation UX
    const data = Object.fromEntries(new FormData(form));
    // checkboxes: FormData omits unchecked ones — normalise explicitly:
    data.notifications = form.elements.notifications.checked;
    localStorage.setItem(KEY, JSON.stringify(data));                // persist (strings only → JSON)
    showToast("Settings saved");
  });

  // 3. SYNC across tabs — the storage event fires in OTHER tabs when this origin's storage changes:
  window.addEventListener("storage", (e) => {
    if (e.key === KEY) initSettings(form);   // re-restore when another tab updates settings
  });
}
initSettings(document.querySelector("#settings"));

Every piece of the lesson is here. Restore reads and JSON-parses saved settings (guarding corrupt storage). Validate uses the browser's built-in checkValidity/reportValidity for free UX. Save collects via FormData→fromEntries, normalises the checkbox (FormData omits unchecked boxes — a real gotcha), and JSON-serialises to localStorage. And the storage event — which fires in other tabs of the same origin when storage changes — gives free cross-tab sync, so changing a setting in one tab updates the others. This is a genuinely production-shaped feature built entirely from platform APIs, no framework.

Interview Questions

Q1 (beginner): How do you read all fields from a form in JavaScript?

Use new FormData(formElement), which collects every field that has a name attribute. From it you can read individual fields with data.get("field") (or getAll for multi-value fields), convert to a plain object with Object.fromEntries(data) for a JSON API, or pass the FormData directly as a fetch body for multipart uploads (which handles file inputs). Remember to event.preventDefault() on submit to stop the default page reload, and that unchecked checkboxes are omitted from FormData, so read their .checked explicitly if you need the boolean.

Q2 (beginner): Why is client-side validation not enough?

Because it runs on the user's machine, where it can be trivially bypassed — disabling JavaScript, editing the DOM, or sending requests directly with curl/Postman that never touch your form. So client-side validation is a UX feature: it gives instant feedback and reduces invalid requests, but it provides no security guarantee. The server must independently re-validate everything it receives, since the server is the only place you actually control. You want both: client validation for responsiveness, server validation for correctness and security — and the server's is the one that truly enforces the rules.

Q3 (mid): Compare localStorage, sessionStorage, cookies, and IndexedDB.

localStorage: ~5–10MB, persists until cleared, synchronous, string-only, not sent to the server — for preferences and non-sensitive UI state. sessionStorage: same API but cleared when the tab closes — per-tab temporary state. Cookies: small (~4KB), sent automatically with every request to the origin, can have expiry — the right place for auth/session tokens, ideally HttpOnly (so JS can't read them, mitigating XSS token theft) and Secure/ SameSite. IndexedDB: an asynchronous, transactional database holding large amounts of structured data — for offline apps, caching big datasets, and complex client-side data. The key decision axes are size (cookies tiny, IndexedDB huge), lifetime (session vs persistent), whether it's sent to the server (only cookies), sync vs async (IndexedDB async), and sensitivity — never put tokens/PII in localStorage since any script can read it.

Q4 (mid): What is IntersectionObserver and why is it better than a scroll listener?

IntersectionObserver asynchronously notifies you when observed elements enter or leave the viewport (or a specified root), reporting their intersection ratio. It's used for lazy-loading images, infinite scroll, triggering animations on scroll-into-view, and impression tracking. It's better than a scroll listener because a scroll handler fires very frequently (potentially every frame) and forces you to synchronously read each element's getBoundingClientRect to compute visibility — causing layout thrashing and jank. IntersectionObserver offloads that computation to the browser, runs asynchronously off the main thread's critical path, batches callbacks, and only notifies you on actual threshold crossings — so it's dramatically more efficient and doesn't degrade scroll performance. It's the correct modern tool for anything "do X when this element is visible."

Q5 (senior): Where should authentication tokens be stored, and what are the trade-offs of each option?

The two realistic options are HttpOnly cookies and localStorage (or in-memory JS), and the trade-off is XSS vs CSRF exposure. localStorage/JS-accessible storage: convenient (JS attaches the token to Authorization headers, works cleanly with SPAs and cross-domain APIs, no CSRF risk because the browser doesn't auto-send it), but any XSS vulnerability can read the token straight out of localStorage and exfiltrate it — a single injected script fully compromises the session, which is a severe and common risk. HttpOnly cookies: the token is inaccessible to JavaScript (the HttpOnly flag), so XSS cannot read it — the token isn't exposed to script at all, which is the stronger default against the most common web vuln. The cost is that cookies are sent automatically with requests, opening CSRF risk, which you mitigate with SameSite=Lax/Strict (blocks cross-site sending), Secure (HTTPS only), and CSRF tokens for state-changing requests; cookies are also less convenient for cross-origin APIs and mobile. The senior recommendation: prefer HttpOnly, Secure, SameSite cookies for session tokens, because XSS is more prevalent and damaging than CSRF and HttpOnly removes the token from script reach entirely, while CSRF is well- mitigated by SameSite + CSRF tokens. A refined pattern for SPAs is a short-lived access token held only in memory (never persisted, so gone on reload and not in localStorage) plus a long-lived refresh token in an HttpOnly cookie — combining XSS resistance (nothing valuable in JS-readable storage) with usability. The anti-pattern to reject firmly is long-lived JWTs in localStorage, which is convenient but turns any XSS into full account takeover. Underlying principle: minimise what an injected script can steal, and layer defences (CSP, HttpOnly, SameSite, short token lifetimes) rather than relying on any single storage choice.

Q6 (senior): Design an offline-capable data layer for a web app. Which storage and APIs, and what are the sync challenges?

Architecture: use IndexedDB as the local durable store (it's async, transactional, and holds large structured data — localStorage is too small/synchronous), wrapped in a friendlier library (idb, Dexie) for ergonomics. Add a Service Worker to intercept network requests and serve from a cache/IndexedDB when offline (cache-first or network-first strategies via the Cache API for assets, IndexedDB for data). Use the online/offline events and navigator.onLine to detect connectivity, and the Background Sync API to defer writes until connectivity returns. The data flow: reads serve from IndexedDB immediately (instant UI), fetching fresh data in the background and updating the store; writes go to IndexedDB first (optimistic UI) and queue an outbound sync operation. The hard part is synchronisation and conflict resolution: (1) queue offline mutations durably (in IndexedDB) with enough info to replay them when back online, in order, with idempotency keys so retries don't duplicate. (2) Decide a conflict strategy when local and server state diverge — last-write-wins (simple but loses data), server-wins, client-wins, or field-level/CRDT merging (robust but complex); the right choice depends on the data (a shared document needs merging/CRDTs; a personal setting can be last-write-wins). (3) Handle partial failures and retries with backoff, and surface sync status to the user (pending/synced/conflict). (4) Version the local schema and handle migrations, since IndexedDB schemas evolve. (5) Manage storage quota and eviction. (6) Security — don't store sensitive data unencrypted locally, and re-authenticate on reconnect. The senior framing: offline-first is fundamentally a distributed-systems problem — you now have two sources of truth (client and server) that must reconcile, so the difficulty isn't the storage APIs (IndexedDB + Service Worker + Background Sync are well-defined) but the sync semantics: ordering, idempotency, conflict resolution, and eventual consistency. That's why frameworks like PouchDB/CouchDB, Replicache, and CRDT libraries exist — they encode battle-tested answers to exactly these reconciliation challenges, and rolling your own means solving them yourself.

🏋️ Exercises

1 (easy): Form to object. On submit, collect a form's fields into a plain object and log it (without reloading the page).
Solution 
form.addEventListener("submit", (e) => {
  e.preventDefault();
  console.log(Object.fromEntries(new FormData(form)));
});

preventDefault stops the reload; FormData+fromEntries collects named fields into an object (single-value fields).

2 (easy): Persist a preference. Save a theme choice to localStorage and restore it on the next page load.
Solution 
// Restore on load:
document.body.dataset.theme = localStorage.getItem("theme") ?? "light";
// Save on change:
themeSelect.addEventListener("change", (e) => {
  localStorage.setItem("theme", e.target.value);
  document.body.dataset.theme = e.target.value;
});

localStorage persists across reloads; reading it on load (with a ?? default) restores the saved theme.

3 (medium): Store and retrieve an object. Save a { name, age } object to localStorage and read it back safely (handling the missing/corrupt case).
Solution 
function saveUser(user) { localStorage.setItem("user", JSON.stringify(user)); }
function loadUser() {
  try { return JSON.parse(localStorage.getItem("user") ?? "null"); }
  catch { return null; }   // corrupt JSON → null instead of throwing
}

localStorage stores strings, so objects must be JSON-serialised; reading back needs a try/catch (or null guard) because stored data could be missing or corrupted.

4 (medium): Password match validation. On submit, block the form and show a browser validation message if two password fields don't match.
Solution 
form.addEventListener("submit", (e) => {
  confirm.setCustomValidity(
    password.value !== confirm.value ? "Passwords don't match" : ""   // "" clears the error
  );
  if (!form.checkValidity()) { e.preventDefault(); form.reportValidity(); }
});

setCustomValidity with a non-empty string marks the field invalid (and clears it with ""); reportValidity shows the browser's native message — reusing built-in validation UI for a custom rule.

5 (hard): Lazy-load images. Use IntersectionObserver to load data-src images only when they're about to enter the viewport.
Solution 
const io = new IntersectionObserver((entries, observer) => {
  for (const entry of entries) {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;       // swap in the real image
      observer.unobserve(img);          // stop watching — loaded once
    }
  }
}, { rootMargin: "200px" });            // pre-load 200px early
document.querySelectorAll("img[data-src]").forEach(img => io.observe(img));

The observer loads each image just before it scrolls into view and unobserves it afterward — far more efficient than a scroll listener recomputing positions, and it pre-loads with rootMargin so images appear ready.

6 (hard): Cross-tab counter. Build a click counter stored in localStorage that stays in sync across multiple open tabs using the storage event.
Solution 
const KEY = "clicks";
const render = () => { display.textContent = localStorage.getItem(KEY) ?? "0"; };

btn.addEventListener("click", () => {
  const next = Number(localStorage.getItem(KEY) ?? "0") + 1;
  localStorage.setItem(KEY, String(next));
  render();
});
// The storage event fires in OTHER tabs when this origin's localStorage changes:
window.addEventListener("storage", (e) => { if (e.key === KEY) render(); });
render();

Clicking updates localStorage and re-renders locally; the storage event (which fires only in other tabs, not the one that made the change) keeps every other open tab in sync — a simple cross-tab communication channel built on storage.

🎯 Key Takeaways

  • FormData reads forms cleanly (fields need name); Object.fromEntries for JSON, or pass it directly for multipart uploads.
  • Client-side validation is UX only — the server must always re-validate; input.value is always a string.
  • Four storage options: localStorage (small, sync, persistent, non-sensitive), sessionStorage (per-tab), cookies (auto-sent, HttpOnly for auth), IndexedDB (large, async, structured).
  • Never store tokens/PII in localStorage (XSS-readable) — use HttpOnly cookies; serialise objects with JSON for storage.
  • Observer APIs (IntersectionObserver, ResizeObserver, MutationObserver) replace inefficient scroll/resize/polling loops.
  • Use URL/URLSearchParams for safe URL building and crypto for secure randomness; the storage event syncs state across tabs.
