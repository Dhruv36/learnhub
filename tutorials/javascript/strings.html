Strings & Template Literals Language Core

What you'll master: string immutability and what it means for performance, the method toolkit you'll actually use (search, slice, transform, split/join), template literals and tagged templates, the UTF-16 reality behind .length and why emoji break naive character counting, and the security line every developer must hold — interpolation does not escape HTML. Prerequisites: Types & Coercion (strings are primitives with wrapper methods). Regular expressions get their own full lesson; here we use only simple patterns.

Part 1 — Strings Are Immutable

let s = "hello";
s[0] = "H";           // silently does NOTHING (throws in strict mode? no — just no-op)
console.log(s);       // "hello" — unchanged
s = s.toUpperCase();  // "HELLO" — every "mutation" returns a NEW string; reassign to keep it
s.length;             // 5 — read-only

A JavaScript string can never be modified in place. Every method that "changes" a string (toUpperCase, replace, slice, trim…) returns a brand-new string and leaves the original untouched; indexed assignment is a silent no-op. This is why you always reassign (s = s.replace(...)) — forgetting to is a classic "why didn't my replace work?" bug. Immutability also has a performance consequence: building a big string by repeated += in a loop creates a new string each iteration. For heavy concatenation, push pieces into an array and join("") once (Part 7).

Part 2 — Quotes and Template Literals

'single'   "double"   `backtick`   // three quote styles; single vs double is pure convention

const name = "Ada", count = 3;
const msg = `Hi ${name}, you have ${count} new ${count === 1 ? "message" : "messages"}.`;
//            ↑ interpolation      ↑ any expression, including a ternary, works inside ${ }

const multi = `Line one
Line two`;                    // real newlines — no \n needed
const html = `<a href="${url}">link</a>`;   // embed quotes freely, no escaping juggling

Template literals (backticks) are the modern default for any string that interpolates a value or spans lines. Inside ${ } you can put any expression — a variable, a function call, arithmetic, a ternary — and it's coerced to string (Types lesson). They give you real multi-line strings and let you mix quote characters without escaping. The one caution, spelled out in Part 8: interpolation performs no HTML escaping, so building HTML from user input this way and assigning to innerHTML is an XSS hole.

Part 3 — Searching & Testing

const s = "the quick brown fox";
s.includes("quick")      // true   — substring present? (the modern default)
s.startsWith("the")      // true
s.endsWith("fox")        // true
s.indexOf("o")           // 12     — first index, or -1 if absent
s.lastIndexOf("o")       // 17
s.at(-1)                 // "x"    — negative index from the end (also s[s.length-1])
s.charAt(0)              // "t"
s.match(/o/g)            // ["o","o"] — regex (Regex lesson)

Reach for includes/startsWith/endsWith for readable existence checks — they say what you mean far better than indexOf(x) !== -1. Use indexOf/lastIndexOf when you actually need the position. at() is the clean way to index from the end (s.at(-1) for the last character), mirroring the array method.

Part 4 — Extracting: slice / substring / substr

const s = "javascript";
s.slice(0, 4)     // "java"   — (start, end-exclusive); NEGATIVE indices allowed
s.slice(-6)       // "script" — from 6th-from-end to end
s.slice(4)        // "script" — start to end
s.substring(0, 4) // "java"   — like slice but NO negatives (clamps to 0) and swaps args if start>end
s.substr(4, 6)    // "script" — (start, LENGTH) — DEPRECATED, avoid

Use slice — it's the most capable (supports negative indices, which are constantly useful) and the most predictable. substring differs in two annoying ways (it clamps negatives to 0 and silently swaps arguments when start > end), and substr (start + length) is deprecated. One rule: reach for slice and forget the other two exist, exactly as with arrays.

Part 5 — Transforming

"  hi  ".trim()            // "hi"      — also trimStart() / trimEnd()
"hi".toUpperCase()         // "HI"       — toLowerCase() too
"5".padStart(3, "0")       // "005"      — zero-pad; padEnd() for the right
"ab".repeat(3)             // "ababab"
"a,b,c".split(",")         // ["a","b","c"] — string → array (split("") → chars, with a caveat, Part 6)
["a","b"].join("-")        // "a-b"       — array → string
"a-b-c".replace("-", "+")  // "a+b-c"     — replaces FIRST match only (string arg)
"a-b-c".replaceAll("-","+")// "a+b+c"     — all matches (or /-/g with replace)
"café".normalize("NFC")    // canonical unicode form — vital before comparing user text

Two footguns to internalise here. First, replace with a string argument replaces only the first occurrence — use replaceAll (or a global regex /x/g) for all of them; "my replace only changed one" is a daily bug. Second, split/join are the canonical string↔array bridge, and combining them is a common transform idiom (s.split("").reverse().join("") — though that breaks on emoji, Part 6). normalize matters more than people think: the same visible accented character can have two different code-point sequences, so user-entered text should be normalised before comparison or storage.

Part 6 — UTF-16: Why .length Lies About Emoji

"café".length        // 4  — fine
"👨‍👩‍👧".length      // 8! — one visible "character" made of multiple code points + joiners
"😀".length          // 2  — a single emoji is TWO UTF-16 code units (a surrogate pair)
"😀"[0]              // "\uD83D" — half a character! indexing splits the surrogate pair
[..."😀"].length     // 1  — spread iterates by CODE POINT, not code unit
"😀".charCodeAt(0)   // 55357 — the high surrogate
"😀".codePointAt(0)  // 128512 — the real code point

JavaScript strings are sequences of UTF-16 code units, and .length counts code units, not human-perceived characters. Characters above U+FFFF (most emoji, some CJK) are stored as two code units (a surrogate pair), so .length over-counts and naive indexing (s[0], s.split(""), reverse) can slice a character in half. The fix for iterating actual characters is the iterator-based approach ([...s] or for...of), which yields whole code points. For truly correct "grapheme" counting (emoji with skin-tone/ZWJ sequences like 👨‍👩‍👧, still multiple code points), you need Intl.Segmenter:

const seg = new Intl.Segmenter("en", { granularity: "grapheme" });
[...seg.segment("👨‍👩‍👧")].length;   // 1 — correct human-visible character count
🧠 Rule of thumb: for slicing/reversing/counting user-facing text with possible emoji, don't use s[i]/.length/split(""). Use [...s] (code points) for most needs, and Intl.Segmenter when you need true grapheme clusters. For ASCII/simple text, plain indexing is fine and fast.

Part 7 — Tagged Templates & Efficient Building

// A tagged template: a function receives the string PARTS and the interpolated VALUES separately.
function safe(strings, ...values) {
  return strings.reduce((out, str, i) =>
    out + str + (i < values.length ? escapeHtml(values[i]) : ""), "");
}
const html = safe`<p>${userInput}</p>`;   // values escaped, static parts left alone

A tagged template is a function invoked with the literal's static string segments as the first argument (an array) and the interpolated values as the rest. This separation — knowing exactly which parts are trusted template text and which are dynamic values — is what powers libraries like styled-components (CSS), graphql-tag, and safe SQL/HTML escapers: they escape only the values, never the static template. It's the sanctioned way to build safe interpolated strings.

// Efficient concatenation — build in an array, join once:
const parts = [];
for (const row of rows) parts.push(`<li>${escapeHtml(row.name)}</li>`);
const listHtml = parts.join("");   // one allocation, not one-per-iteration

Because strings are immutable (Part 1), result += piece in a hot loop allocates a new string every iteration. Pushing pieces into an array and calling join("") once is the idiomatic, faster pattern for building large strings (modern engines optimise simple += well, but join is the safe default for big or variable loops).

Part 8 — The Security Line: Interpolation ≠ Escaping

// ❌ XSS: template interpolation does NOT escape HTML
element.innerHTML = `<div>${userInput}</div>`;
// If userInput is `<img src=x onerror=alert(document.cookie)>`, that script RUNS.

// ✅ Safe options:
element.textContent = userInput;                 // textContent NEVER interprets HTML — safest
element.innerHTML = `<div>${escapeHtml(userInput)}</div>`;  // escape if you MUST build HTML
// or a sanitizer (DOMPurify) for rich HTML input

This is the single most important safety fact about strings: template literals interpolate, they do not escape. Any user-controlled value flowed into innerHTML (or document.write, or a framework's dangerous-HTML escape hatch) via string building is a cross-site scripting vulnerability. The default-safe answer is textContent (which treats everything as literal text); when you genuinely must build HTML, escape values through a real escaper or sanitise with DOMPurify. The Security lesson goes deep — but the habit starts here, at every template literal touching user input.

Part 9 — Build It: a Slug + Truncate Utility

Let's combine transformation, Unicode-awareness, and search into a small, real utility: turning a title into a URL slug and safely truncating a preview string on a whole-character boundary.

function slugify(title) {
  return title
    .normalize("NFKD")                    // decompose accented chars: "é" → "e" + combining accent
    .replace(/[̀-ͯ]/g, "")      // strip the combining marks → plain ASCII letters
    .toLowerCase()
    .trim()
    .replace(/[^a-z0-9]+/g, "-")          // non-alphanumerics → hyphen
    .replace(/^-+|-+$/g, "");             // trim leading/trailing hyphens
}
slugify("Café — Ñandú & Co.!");           // "cafe-nandu-co"

function truncate(str, max) {
  const chars = [...str];                 // code-point aware — won't split an emoji
  return chars.length <= max ? str : chars.slice(0, max - 1).join("") + "…";
}
truncate("Hello 😀 world", 8);            // "Hello 😀…"  (not a broken half-emoji)

slugify uses NFKD normalisation to peel accents off letters before stripping — far more robust than a hand-maintained "é→e" table — and truncate spreads to code points so it never slices an emoji in half (Part 6's lesson applied). Both are the kind of utility every content site needs, and both quietly depend on getting the Unicode story right.

Interview Questions

Q1 (beginner): Are JavaScript strings mutable? What does that imply for methods like toUpperCase?

No — strings are immutable primitives. You cannot change a character in place; s[0] = "X" is a silent no-op. Every "mutating" method (toUpperCase, replace, trim, slice) returns a brand-new string and leaves the original untouched. The practical implication is you must capture the return value — s = s.toUpperCase() — or the change is lost, which is a very common beginner bug.

Q2 (beginner): When would you use a template literal over string concatenation?

Whenever you interpolate values or need multiple lines. Template literals (backticks) embed any expression inside ${ }, support real multi-line content without \n, and let you mix quote characters without escaping — so `Hi ${name}, ${n} items` is clearer and less error-prone than "Hi " + name + ", " + n + " items". The one caveat is that interpolation does not escape HTML, so template-built HTML from user input into innerHTML is an XSS risk.

Q3 (mid): Why is "😀".length equal to 2, and how do you correctly count or iterate characters?

Strings are UTF-16, a sequence of 16-bit code units. Characters above U+FFFF — most emoji, some CJK — don't fit in one code unit, so they're encoded as a surrogate pair: two code units. .length counts code units, so a single emoji reports 2, and indexing (s[0]) returns just the high surrogate — half a character. To iterate or count actual characters use the string iterator, [...s] or for...of, which yields whole code points. For truly human-perceived characters (emoji ZWJ sequences, flags), even code points aren't enough — you need Intl.Segmenter with grapheme granularity.

Q4 (mid): What is a tagged template literal and what problem does it solve?

A tagged template is a function called with a template literal, receiving the static string segments as an array and the interpolated values as separate arguments: tag`Hello ${name}` calls tag(["Hello ", ""], name). Because it cleanly separates trusted static template text from dynamic values, it solves the safe-interpolation problem: the tag can escape or transform only the values while leaving the template intact. This is how styled-components processes CSS, how GraphQL gql tags parse queries, and how safe HTML/SQL builders escape injected values — the tag function is where the escaping or parsing logic lives.

Q5 (senior): A colleague reverses strings with s.split("").reverse().join(""). When does this break, and what's the correct implementation?

It breaks on any string containing characters outside the Basic Multilingual Plane — emoji, some CJK, mathematical symbols — because split("") splits by UTF-16 code units, so a surrogate pair (e.g. "😀" = two units) gets split into its two halves, and reversing then juxtaposes them in the wrong order, producing invalid/garbled output (a lone or swapped surrogate renders as "�"). It also mangles combining marks and ZWJ emoji sequences. A code-point correct version spreads the string: [...s].reverse().join(""), since the string iterator yields whole code points. For truly correct reversal of grapheme clusters (accented letters using combining marks, family emoji), you must segment with Intl.Segmenter (grapheme granularity), reverse the segments, and join — because even a single perceived character can span several code points that must stay together and in order. The senior point is that "a character" is ambiguous in Unicode (code unit vs code point vs grapheme), and the right tool depends on which one your requirement means.

Q6 (senior): Explain the XSS risk in string-built HTML and design a defense-in-depth approach for a comment-rendering feature.

String interpolation (template literals, +) does not escape HTML metacharacters, so any user-controlled value placed into innerHTML can inject active markup — e.g. <img src=x onerror=...> executes script in the victim's session, stealing cookies or acting as them. Defense in depth for rendering comments: (1) Prefer non-HTML sinks — if comments are plain text, render with textContent, which treats everything as literal and is immune by construction; this alone closes the hole for the common case. (2) If rich formatting is required, never build HTML by hand — run the user's HTML through a well-maintained sanitizer (DOMPurify) with a strict allow-list of tags/attributes, stripping event handlers, javascript: URLs, and unknown elements; do this at render time, not just on input, since sanitizer rules improve over time. (3) Escape at every interpolation boundary where you do assemble markup, using a tagged-template escaper so values are HTML- encoded automatically and the static template can't be bypassed. (4) Add a Content Security Policy (script-src without unsafe-inline) as a backstop so that even if an injection slips through, inline scripts won't execute. (5) Set cookies HttpOnly so a successful XSS can't read the session cookie via document.cookie. The layered principle: safe sink first, sanitisation second, escaping third, CSP and HttpOnly as backstops — no single layer is trusted alone, and the string-building step is never where you rely on manual correctness.

🏋️ Exercises

1 (easy): Reassignment bug. This logs "hello world", not the trimmed/capitalised result. Fix it: 
let s = "  hello world  ";
s.trim();
s.toUpperCase();
console.log(s);
Solution 
let s = "  hello world  ";
s = s.trim().toUpperCase();
console.log(s);   // "HELLO WORLD"

Strings are immutable, so trim() and toUpperCase() return new strings that were discarded. Chain and reassign to capture the result.

2 (easy): Replace all. Redact every digit in "call 555-123-4567 or 555-987" with #. Show the string-arg replace bug first.
Solution 
"call 555-123-4567".replace("5", "#");    // "call #55-123-4567" — only FIRST 5
"call 555-123-4567".replaceAll(/\d/g, "#"); // TypeError? no — replaceAll needs global flag or string
"call 555-123-4567".replace(/\d/g, "#");    // "call ###-###-####" — global regex, all digits

String-argument replace hits only the first match. Use a global regex /\d/g (or replaceAll with a string/global-regex) to replace every occurrence. Note replaceAll throws if given a non-global regex — a deliberate guard.

3 (medium): Character count that respects emoji. Write charCount(str) returning the code-point count, and show it differs from .length for "a😀b".
Solution 
const charCount = str => [...str].length;
"a😀b".length      // 4 — the emoji counts as 2 code units
charCount("a😀b")  // 3 — code points

Spreading uses the string iterator, which walks whole code points, so the surrogate-pair emoji counts once. For grapheme-perfect counting of ZWJ sequences, upgrade to Intl.Segmenter.

4 (medium): Safe HTML escape. Write escapeHtml(str) that escapes & < > " ', and demonstrate it neutralising <img src=x onerror=alert(1)>.
Solution 
const escapeHtml = str => str.replace(/[&<>"']/g, ch => ({
  "&": "&amp;", "<": "&lt;", ">": "&gt;", '"': "&quot;", "'": "&#39;"
}[ch]));
escapeHtml('<img src=x onerror=alert(1)>');
// "&lt;img src=x onerror=alert(1)&gt;"  — renders as literal text, no execution

Escaping & first (before </>) matters so you don't double-escape. The result, placed in innerHTML, displays the tag as text instead of creating a live <img>. For rich HTML input, prefer DOMPurify over a hand-rolled escaper.

5 (hard): Tagged-template SQL-ish escaper. Write a tag sql that quotes and escapes each interpolated value (wrapping strings in single quotes, doubling internal quotes) while leaving the static SQL text untouched. (Illustrative only — use real parameterised queries in production.)
Solution 
function sql(strings, ...values) {
  return strings.reduce((out, str, i) => {
    if (i >= values.length) return out + str;
    const v = values[i];
    const escaped = typeof v === "number" ? String(v)
      : `'${String(v).replace(/'/g, "''")}'`;   // double single-quotes; wrap in quotes
    return out + str + escaped;
  }, "");
}
sql`SELECT * FROM users WHERE name = ${"O'Brien"} AND age > ${30}`;
// SELECT * FROM users WHERE name = 'O''Brien' AND age > 30

The tag sees the static SQL (strings) and the values separately, so it escapes only the values — the exact separation that makes tagged templates safe for interpolation. (This is a teaching illustration; real code must use parameterised/prepared statements, which the database driver escapes correctly — the Security lesson covers SQL injection properly.)

6 (hard): Unicode-safe slug. Extend the slugify from Part 9 to also collapse multiple spaces, and prove it produces the same slug for "Café" whether the é is a single precomposed code point or an "e" + combining accent.
Solution 
const slugify = title => title
  .normalize("NFKD").replace(/[̀-ͯ]/g, "")
  .toLowerCase().trim()
  .replace(/\s+/g, " ")            // collapse whitespace runs first
  .replace(/[^a-z0-9]+/g, "-").replace(/^-+|-+$/g, "");

const precomposed = "Café";      // "Café" with single é
const decomposed  = "Café";     // "Café" with e + combining acute
slugify(precomposed) === slugify(decomposed);   // true — both → "cafe"

NFKD normalisation decomposes the precomposed é into e + combining mark, putting both inputs into the same form; stripping the combining-mark range then yields identical ASCII. This is why normalising before comparing or slugging user text is essential — the same visible string can arrive in multiple byte sequences.

🎯 Key Takeaways

  • Strings are immutable — every method returns a new string; always reassign to keep the result.
  • Template literals interpolate any expression and span lines; slice is the extraction method to standardise on; replaceAll/global regex for all matches (string replace hits only the first).
  • Strings are UTF-16: .length counts code units, so emoji/CJK over-count and naive indexing splits characters — iterate with [...s], count graphemes with Intl.Segmenter.
  • Normalise user text (normalize()) before comparing — the same character can have multiple code-point encodings.
  • Tagged templates separate static text from values — the mechanism behind safe escapers and DSLs like styled-components and gql.
  • Interpolation does NOT escape HTML — user input into innerHTML is XSS; default to textContent, escape or sanitise when building HTML.
