Events & Delegation Browser & the Web Platform

What you'll master: the event model — listeners, the event object, bubbling/capturing, and how to stop propagation correctly — event delegation (one listener for many elements) and why it scales, preventing default browser behaviour, custom events for component communication, and cleanup to avoid leaks. Prerequisites: The DOM (closest, selection), Closures (handlers are closures), and this (handler this). Events are how every interactive page works, and delegation is a favourite interview topic.

Part 1 — Listeners and the Event Object

const btn = document.querySelector("button");
btn.addEventListener("click", (event) => {    // register a handler for the "click" event
  event.target;        // the element that triggered the event (may be a descendant)
  event.currentTarget; // the element the listener is ATTACHED to (== `this` in a normal-fn handler)
  event.type;          // "click"
  event.timeStamp;     // when it happened
});

btn.removeEventListener("click", handler);     // remove — must be the SAME function reference

addEventListener(type, handler, options) registers a callback that runs when the event fires. The handler receives an event object describing what happened. Two properties you must not confuse: event.target is the element that originated the event (e.g. the exact icon clicked), while event.currentTarget is the element the listener is attached to — they differ when the event bubbles up from a descendant (Part 2), and that distinction is the whole basis of delegation. To remove a listener you must pass the same function reference, which is why anonymous inline handlers can't be removed (Part 6).

Part 2 — Bubbling, Capturing & Propagation

// Click a button inside a div inside the document. The event travels in THREE phases:
// 1. CAPTURE  — document → ... → div → button   (top-down; listeners with { capture: true })
// 2. TARGET   — the button itself
// 3. BUBBLE   — button → div → ... → document    (bottom-up; the DEFAULT phase)

parent.addEventListener("click", () => console.log("parent"));
child.addEventListener("click", () => console.log("child"));
// Clicking child logs: "child", then "parent" — the event BUBBLES up.

el.addEventListener("click", handler, { capture: true });  // listen in the capture phase instead

When an event fires, it doesn't just hit the target — it propagates through the tree in three phases: capturing (top-down from the document to the target), at target, then bubbling (bottom-up back to the document). By default listeners run in the bubble phase, so a click on a child triggers the child's handler, then its parent's, then up the chain. This bubbling is what makes event delegation possible (Part 4) — a parent can "hear" events from all its descendants. You can opt into the capture phase with { capture: true }, but bubbling is what you'll use 99% of the time.

Part 3 — Controlling Propagation & Default Behaviour

event.stopPropagation();       // stop the event from bubbling further up (other elements won't hear it)
event.stopImmediatePropagation(); // also stop OTHER listeners on the SAME element from running
event.preventDefault();        // cancel the browser's DEFAULT action (form submit, link navigation)

// Common: prevent a form from reloading the page, handle it in JS instead:
form.addEventListener("submit", (e) => {
  e.preventDefault();          // stop the full-page navigation
  submitViaFetch(new FormData(form));
});

// Common: intercept a link:
link.addEventListener("click", (e) => { e.preventDefault(); router.navigate(link.href); });

Two independent controls, often confused. preventDefault() cancels the browser's built-in reaction to the event — stopping a form from reloading the page, a link from navigating, a checkbox from toggling — without affecting propagation. stopPropagation() stops the event from traveling further up (or down) the tree, without affecting the default action. They do different jobs: use preventDefault to take over browser behaviour (SPA routing, AJAX forms), and stopPropagation sparingly to contain an event — overusing it breaks delegation and creates hard-to-debug "why didn't my parent handler fire?" issues.

⚠️ return false in a plain handler is not preventDefault. In inline HTML handlers return false does both prevent-default and stop-propagation, but in an addEventListener callback it does nothing. Call e.preventDefault() explicitly.

Part 4 — Event Delegation: One Listener for Many Elements

// ❌ A listener PER item — expensive to attach, breaks for dynamically-added items:
document.querySelectorAll(".item").forEach(item =>
  item.addEventListener("click", handleItem));   // 1000 items → 1000 listeners

// ✅ DELEGATION — one listener on the parent, use event.target + closest:
list.addEventListener("click", (e) => {
  const item = e.target.closest(".item");        // find which .item was clicked (bubbling + closest)
  if (!item || !list.contains(item)) return;     // click wasn't on an item
  handleItem(item);                               // handle it — works for items added LATER too
});

Event delegation puts a single listener on a common ancestor and uses bubbling + event.target.closest(selector) to figure out which descendant was actually interacted with. Its two big wins: performance (one listener instead of thousands — less memory, faster setup) and — crucially — it works for dynamically added elements automatically, because the listener is on the parent, not on each item. Add a new .item later and it just works, no re-binding. This is the single most important event pattern for lists, tables, menus, and any collection of interactive elements, and it's exactly why closest exists.

Part 5 — Custom Events for Component Communication

// Dispatch a custom event with data — decoupled component communication:
const event = new CustomEvent("cart:add", {
  detail: { productId: 42, qty: 2 },   // your payload lives in .detail
  bubbles: true,                        // let it bubble so ancestors can listen
});
cartButton.dispatchEvent(event);

// Listen anywhere up the tree:
document.addEventListener("cart:add", (e) => {
  console.log("added", e.detail.productId, "×", e.detail.qty);
});

CustomEvent lets you create and dispatch your own named events, carrying arbitrary data in the detail property. This is a clean, framework-free way for parts of a page to communicate without direct references: a component dispatches "cart:add" and any interested code listens, without the components knowing about each other (the observer pattern, built into the DOM). Namespacing event names ("cart:add") avoids collisions. It's the basis of how Web Components communicate outward and a lightweight alternative to a full event-bus library for intra-page messaging.

Part 6 — Listener Options & Cleanup (Avoiding Leaks)

el.addEventListener("scroll", onScroll, { passive: true });   // promise not to preventDefault → smoother scroll
el.addEventListener("click", onceHandler, { once: true });    // auto-removes after firing once

// Cleanup — a listener keeps its handler (a closure) and the element alive → leak if not removed:
function setup() {
  const handler = () => { /* closes over component state */ };
  window.addEventListener("resize", handler);
  return () => window.removeEventListener("resize", handler);  // teardown function
}

// Modern cleanup — AbortController removes ALL associated listeners at once:
const controller = new AbortController();
el.addEventListener("click", h1, { signal: controller.signal });
el.addEventListener("keydown", h2, { signal: controller.signal });
controller.abort();   // removes BOTH listeners in one call — great for component teardown

The options object matters. { passive: true } promises the handler won't call preventDefault, letting the browser scroll without waiting for your handler — important for smooth scrolling/touch performance. { once: true } auto-removes the listener after it fires once. And cleanup is essential: a registered listener holds its handler (a closure capturing whatever it references) and keeps the associated element reachable, so forgetting to remove listeners on long-lived targets (window, document) is a classic memory leak (Closures and Memory lessons). The modern one-liner: pass an AbortController's signal to every listener and call controller.abort() to remove them all at once — perfect for component unmount.

Part 7 — Debouncing & Throttling High-Frequency Events

// Events like input, scroll, resize, mousemove fire RAPIDLY — running heavy work each time janks:
const debounce = (fn, ms) => {
  let t;
  return (...args) => { clearTimeout(t); t = setTimeout(() => fn(...args), ms); };
};
// Debounce: run only AFTER the user stops firing events for `ms` (search-as-you-type):
searchInput.addEventListener("input", debounce((e) => searchAPI(e.target.value), 300));

const throttle = (fn, ms) => {
  let last = 0;
  return (...args) => { const now = Date.now(); if (now - last >= ms) { last = now; fn(...args); } };
};
// Throttle: run at most once per `ms` (scroll position, resize):
window.addEventListener("scroll", throttle(updateScrollIndicator, 100));

High-frequency events (input, scroll, resize, mousemove) can fire dozens of times per second — running expensive work (a network request, a layout calculation) on every one destroys performance. Debounce waits until the events stop for a quiet period, then runs once — ideal for search-as-you-type (wait until they stop typing) and resize-then-recalculate. Throttle runs at most once per interval regardless of how many events fire — ideal for scroll-position updates where you want regular but bounded updates. Both are closures holding a timer/timestamp (the Closures lesson's build); the Performance lesson covers the robust versions (leading/trailing edges).

Part 8 — Build It: a Delegated, Cleanable Todo List

Let's combine delegation, custom events, and clean teardown into a small interactive component — one listener handling clicks on any todo (including ones added later), emitting a custom event, with a single-call cleanup.

function createTodoList(container) {
  const controller = new AbortController();      // one signal for all listeners
  const signal = controller.signal;

  // ONE delegated listener handles toggle + delete for ALL todos, present and future:
  container.addEventListener("click", (e) => {
    const li = e.target.closest("li.todo");
    if (!li) return;
    if (e.target.matches(".delete")) {
      li.remove();
      container.dispatchEvent(new CustomEvent("todo:deleted",
        { detail: { id: li.dataset.id }, bubbles: true }));
    } else if (e.target.matches(".toggle")) {
      const done = li.classList.toggle("done");
      container.dispatchEvent(new CustomEvent("todo:toggled",
        { detail: { id: li.dataset.id, done }, bubbles: true }));
    }
  }, { signal });

  return {
    add(text, id) {
      const li = document.createElement("li");
      li.className = "todo"; li.dataset.id = id;
      li.innerHTML = "";                          // build safely below (no user data in innerHTML)
      const span = document.createElement("span"); span.textContent = text;
      const toggle = Object.assign(document.createElement("button"), { className: "toggle", textContent: "✓" });
      const del = Object.assign(document.createElement("button"), { className: "delete", textContent: "✕" });
      li.append(toggle, span, del);
      container.append(li);                        // the delegated listener already covers it
    },
    destroy() { controller.abort(); }              // removes the listener in one call
  };
}

const todos = createTodoList(document.querySelector("#todos"));
document.addEventListener("todo:deleted", (e) => console.log("deleted", e.detail.id));
todos.add("Learn events", "1");
todos.add("Ship feature", "2");
// todos.destroy();  // full cleanup, no leaks

This is a complete, leak-free interactive component built from the lesson's pieces. One delegated listener handles toggle and delete for every todo — including ones added after setup — using closest/matches to route the click. It emits custom events so parent code can react without coupling. Content is built with textContent/createElement (no XSS from user text). And a single AbortController means destroy() removes the listener with one call — no dangling handler, no leak. This delegation + custom-events + signal-cleanup trio is exactly how you'd structure vanilla interactive components in production.

Interview Questions

Q1 (beginner): What's the difference between event.target and event.currentTarget?

event.target is the element that originated the event — the deepest element actually interacted with (e.g. the specific icon inside a button that was clicked). event.currentTarget is the element the currently-running listener is attached to. They're the same when you click directly on the listener's element, but differ when the event bubbles up from a descendant: target stays the origin while currentTarget is whichever ancestor's listener is now handling it. In a normal-function handler, this equals currentTarget. This distinction is the basis of event delegation, where you attach to a parent (currentTarget) and inspect target to see what was clicked.

Q2 (beginner): What is event bubbling?

When an event fires on an element, after reaching the target it travels back up through the target's ancestors, triggering any matching listeners along the way — from the target up to the document. So clicking a button inside a div fires the button's click listener, then the div's, then further ancestors'. This default bottom-up propagation is called bubbling (preceded by a top-down capturing phase that's rarely used). Bubbling is what makes event delegation possible — a single listener on a common ancestor "hears" events from all its descendants.

Q3 (mid): What is event delegation and why use it?

Event delegation is attaching one listener to a common ancestor instead of one to each child, and using the event's bubbling plus event.target.closest(selector) to determine which descendant was interacted with. Two main benefits: (1) performance and memory — one listener instead of potentially thousands, cheaper to set up and lighter to hold; (2) it automatically handles dynamically added elements — because the listener lives on the parent, children added after setup are covered with no re-binding, whereas per-child listeners would miss them. It's the standard approach for lists, tables, menus, and any collection of interactive items. The one caveat is that events must bubble (most do; some like focus/blur don't, needing focusin/focusout or capture).

Q4 (mid): Explain preventDefault vs stopPropagation.

They control two independent things. preventDefault() cancels the browser's built-in default action for the event — stopping a form submission from reloading the page, a link from navigating, a checkbox from toggling — without affecting how the event travels through the tree. stopPropagation() halts the event's propagation to other elements (stops it bubbling further up or capturing further down), without affecting the default action. stopImmediatePropagation() additionally prevents other listeners on the same element from running. Use preventDefault when you're taking over browser behaviour (SPA routing, AJAX form submit); use stopPropagation sparingly to contain an event, since overusing it silently breaks parent handlers and event delegation.

Q5 (senior): How do event listeners cause memory leaks, and what are the modern mitigation strategies?

A registered listener holds a reference to its handler function, which — being a closure — holds references to everything in its enclosing scope (component state, DOM nodes, large data). As long as the listener remains registered, none of that can be garbage-collected, and the target element itself stays reachable through the listener registry. So the leak pattern is: attach a listener (especially to a long-lived target like window, document, or a shared element) whose handler captures a component's data or DOM, then destroy/replace the component without removing the listener — the handler, its captured data, and any DOM it references all leak, and the "zombie" handler may even run and error. Mitigations: (1) always pair addEventListener with removeEventListener in teardown, passing the same named function reference; (2) the modern approach — pass an AbortController's signal to every addEventListener for a component and call controller.abort() on unmount to remove them all at once, which is far less error-prone than tracking each removal; (3) { once: true } for one-shot listeners so they auto-remove; (4) delegate to a parent so you have one listener to manage instead of many; (5) avoid capturing more than needed in the handler closure. Detection: Chrome DevTools' memory heap snapshots reveal detached DOM nodes and retained closures, and the "Event Listeners" pane shows what's still attached. Frameworks automate this (React removes listeners on unmount), but the underlying discipline — every listener has a defined lifetime and removal path — is the same, and AbortController signals are the current best practice for making that removal reliable.

Q6 (senior): Design an event architecture for a complex widget with nested interactive parts. Discuss delegation boundaries, custom events, and propagation control.

The goal is loose coupling, predictable propagation, and clean teardown. Structure: (1) Delegate at meaningful boundaries — put one delegated listener at the widget's root (or at each self-contained sub-region), routing by target.closest(selector) to the right handler, rather than one listener per interactive element. This keeps listener count low and supports dynamic content. Choose the delegation root at the boundary of ownership — a sub-component delegates within itself, the container delegates for cross-cutting concerns. (2) Communicate upward with namespaced custom events — inner parts dispatch bubbling CustomEvents ("widget:itemSelected", payload in detail) that ancestors/consumers listen for, so components don't hold direct references to each other (observer pattern); this is how Web Components expose behaviour and how you decouple a widget from its host. Communicate downward via method calls/props, not events. (3) Control propagation deliberately, sparingly — avoid stopPropagation as a default; it breaks delegation and outside listeners in surprising ways. When a nested interactive control must not trigger an outer handler (e.g. a delete button inside a selectable row), prefer routing logic in the delegated handler (check target.closest(".delete") first and return) over stopping propagation, so the event model stays transparent. Reserve stopPropagation for genuine encapsulation boundaries (a modal that must not leak clicks to the page behind it), and document why. Use preventDefault for browser-default overrides (form submit, drag). (4) Own the lifecycle — register every listener with a shared AbortController signal so destroy() removes them all at once; this prevents leaks and zombie handlers when the widget is torn down or re-rendered. (5) Performance — debounce/throttle high-frequency inputs, and use { passive: true } for scroll/touch handlers that don't preventDefault. The overall philosophy: delegate for efficiency and dynamism, emit namespaced custom events for decoupled upward communication, keep propagation transparent by routing in handlers rather than stopping events, and tie all listeners to a single abortable lifecycle. This yields a widget whose parts are loosely coupled, whose event flow is predictable, and which cleans up completely — the same architecture principles frameworks encode, applied to vanilla DOM.

🏋️ Exercises

1 (easy): Prevent a form reload. Stop a form from navigating on submit and log its FormData as an object instead.
Solution 
form.addEventListener("submit", (e) => {
  e.preventDefault();                             // stop the full-page reload
  console.log(Object.fromEntries(new FormData(form)));
});

preventDefault cancels the browser's default submit navigation; FormData+fromEntries collects the fields into a plain object (Forms lesson).

2 (easy): Bubbling order. Predict the console output when the inner element is clicked: 
outer.addEventListener("click", () => console.log("outer"));
inner.addEventListener("click", () => console.log("inner"));
Answer

inner, then outer. Listeners run in the bubble phase by default, so the target's handler fires first, then the event bubbles up to the ancestor's handler.

3 (medium): Delegate a list. Instead of a listener per .item, attach one to the <ul> that logs the clicked item's text, working for items added later.
Solution 
list.addEventListener("click", (e) => {
  const item = e.target.closest(".item");
  if (item && list.contains(item)) console.log(item.textContent);
});

One listener on the parent uses bubbling + closest to identify the clicked item; because it's on the parent, items added after setup are handled automatically.

4 (medium): Debounced search. Wire an input so it calls search(value) only after the user stops typing for 300ms.
Solution 
function debounce(fn, ms) {
  let t;
  return (...args) => { clearTimeout(t); t = setTimeout(() => fn(...args), ms); };
}
input.addEventListener("input", debounce((e) => search(e.target.value), 300));

Each keystroke resets the timer, so search runs only 300ms after the last input — one request instead of one per keystroke.

5 (hard): Custom event bus. Build a tiny event system using a DOM node: emit(name, detail) and on(name, handler) that decouples publishers from subscribers.
Solution 
function createBus() {
  const target = new EventTarget();               // a standalone event target (no DOM needed)
  return {
    emit: (name, detail) => target.dispatchEvent(new CustomEvent(name, { detail })),
    on: (name, handler) => {
      const wrapped = (e) => handler(e.detail);
      target.addEventListener(name, wrapped);
      return () => target.removeEventListener(name, wrapped);   // unsubscribe
    }
  };
}
const bus = createBus();
const off = bus.on("login", (user) => console.log("welcome", user.name));
bus.emit("login", { name: "Ada" });   // "welcome Ada"
off();                                  // unsubscribe

Using EventTarget (or any DOM node) plus CustomEvent gives a decoupled pub/sub bus for free — publishers emit, subscribers on, and the returned function unsubscribes. This is the observer pattern built on native events, a lightweight alternative to an event-emitter library.

6 (hard): Leak-free component. Write a component that adds resize and keydown listeners on window and can be fully torn down with one call, removing both.
Solution 
function createComponent() {
  const controller = new AbortController();
  const { signal } = controller;
  window.addEventListener("resize", () => layout(), { signal });
  window.addEventListener("keydown", (e) => { if (e.key === "Escape") close(); }, { signal });
  return { destroy: () => controller.abort() };   // removes BOTH window listeners at once
}
const c = createComponent();
// ...later...
c.destroy();   // no leaked listeners on window

Passing the same signal to every listener means a single controller.abort() removes all of them — the modern, error-proof teardown that avoids leaking handlers on the long-lived window object (no need to keep individual function references for removeEventListener).

🎯 Key Takeaways

  • addEventListener(type, handler, options) registers handlers; target is the event's origin, currentTarget the listener's element.
  • Events propagate capture → target → bubble (bubble is default); bubbling is what makes delegation work.
  • preventDefault cancels the browser's default action; stopPropagation halts travel through the tree — different jobs, use the latter sparingly.
  • Event delegation (one parent listener + closest) scales to many elements and handles dynamically-added ones automatically — the key pattern for lists/tables.
  • CustomEvent with a detail payload enables decoupled, framework-free component communication (observer pattern).
  • Clean up listeners (same reference / AbortController signal) to avoid leaks; debounce/throttle high-frequency events (input, scroll).
