Async: Promises & await Intermediate

JavaScript is single-threaded: it cannot stop and wait. Anything slow — network, disk, timers — is started, the thread moves on, and a callback fires when the result lands. Promises are the protocol that tamed that chaos; async/await makes it read like sync code. This is the lesson with the highest bug-prevention-per-minute ratio in the track.

A Promise Is a One-Shot Future

States: PENDING → FULFILLED (value) | REJECTED (reason).  Settles ONCE, permanently.

fetch("/api/user")                      // returns a pending Promise IMMEDIATELY
  .then(res => res.json())              // .then returns a NEW promise → chains flatten
  .then(user => render(user))
  .catch(err => showError(err))         // ANY rejection above lands here
  .finally(() => spinner.hide());       // always runs — the cleanup slot

// Chaining vs nesting — the whole point:
// ❌ .then(a => { getB(a).then(b => { getC(b).then(…) }) })   ← callback pyramid rebuilt
// ✅ .then(a => getB(a)).then(b => getC(b))                    ← return the promise, chain flat

async/await — Same Machinery, Readable

async function loadUser(id) {                 // async fn ALWAYS returns a Promise
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);   // ← the most-forgotten line in JS
    return await res.json();
  } catch (err) {                              // network errors + the throw above
    logger.warn("loadUser failed", { id, err });
    throw err;                                 // handle OR rethrow — never swallow
  }
}
🚨 fetch's rejection contract: it rejects ONLY on network failure (DNS, offline, CORS block, abort). HTTP 404/500 fulfill — with res.ok === false. Every codebase eventually centralizes this check in an api() wrapper; every outage post-mortem that says "errors were silently ignored" is this line missing.

Sequential vs Parallel — the Performance Question

// ❌ THE WATERFALL — independent calls, serial awaits (300ms × 3 = 900ms):
const user  = await getUser();
const posts = await getPosts();
const ads   = await getAds();

// ✅ Parallel (total = slowest ≈ 300ms):
const [user, posts, ads] = await Promise.all([getUser(), getPosts(), getAds()]);

// Loops:
for (const id of ids) await process(id);              // sequential — order/rate-limits demand it
await Promise.all(ids.map(id => process(id)));        // parallel — independent work
// ⚠ ids.forEach(async id => …) fires all, awaits NONE — forEach ignores returned promises.
// Bounded concurrency (real-world: 500 items, API allows 10 at once) — chunk or a p-limit-style pool.

The Combinators — Know All Four Cold

CombinatorSettles when…Canonical use
Promise.allAll fulfill — or REJECTS FAST on the first rejectionIndependent required data; destructure results (order preserved)
Promise.allSettledAll settle, never rejects → [{status, value|reason}]Batch jobs where partial success is fine (send 100 emails)
Promise.raceFIRST settles (either way)Timeouts: race(fetch, delay(5000).then(throw))
Promise.anyFirst FULFILLS (ignores rejections; AggregateError if all reject)Mirrors/fallback sources — fastest success wins

Cancellation — the Production Requirement Everyone Forgets

// The stale-search bug: response for "ja" arrives AFTER "java" → old results overwrite new.
let ctl;
input.addEventListener("input", async (e) => {
  ctl?.abort();                                  // kill the previous request
  ctl = new AbortController();
  try {
    const res = await fetch(`/search?q=${e.target.value}`, { signal: ctl.signal });
    render(await res.json());
  } catch (err) {
    if (err.name !== "AbortError") throw err;    // aborts are EXPECTED — don't log as failures
  }
});
// Simpler timeout: fetch(url, { signal: AbortSignal.timeout(5000) })
// Combine causes: AbortSignal.any([userCtl.signal, AbortSignal.timeout(5000)])

Retry with Backoff — the Other Production Pattern

async function withRetry(fn, { tries = 3, base = 200 } = {}) {
  for (let i = 0; i < tries; i++) {
    try { return await fn(); }
    catch (err) {
      if (i === tries - 1 || !isTransient(err)) throw err;   // 4xx: don't retry client bugs
      await delay(base * 2 ** i + Math.random() * 100);      // exponential + JITTER
    }
  }
}
// Jitter prevents thundering herds when a recovering server meets synchronized retries —
// the same reasoning as the System Design track's backoff lesson.

Async Error Flows — the Traps

// 1. The un-awaited escape:
try { savePrefs(); } catch (e) { }       // ❌ rejection happens AFTER try exits → unhandled
try { await savePrefs(); } catch (e) { } // ✅ (or: savePrefs().catch(report) for fire-and-forget)

// 2. return vs return await inside try:
try { return doWork(); }        // ❌ rejection NOT caught here — promise escapes first
try { return await doWork(); }  // ✅ caught by THIS catch — 'return await' is meaningful in try

// 3. Building promises — executor anti-pattern:
new Promise(async (resolve) => resolve(await getX()))   // ❌ pointless wrapper
// new Promise() is ONLY for wrapping callback APIs:
const sleep = (ms) => new Promise(r => setTimeout(r, ms));
const load  = (img) => new Promise((res, rej) => { img.onload = res; img.onerror = rej; });

// 4. Global last resort — telemetry, not control flow:
window.addEventListener("unhandledrejection", (e) => report(e.reason));

Interview Questions

Q1: Implement Promise.all.

Return new Promise; results array + settled counter; for each input (Promise.resolve(p) to accept raw values) .then(v => { results[i] = v; if (++count === arr.length) resolve(results) }, reject). Full marks for: index-stable results (not push — completion order ≠ input order), first-rejection short-circuit, empty-array resolving immediately.

Q2: What does await actually do — precisely?

It suspends THE ASYNC FUNCTION (not the thread): the rest of the function is packaged as a continuation scheduled on the microtask queue when the awaited promise settles; control returns to the caller immediately. Rejected promise → the continuation throws at the await line (catchable by try/catch). Hence "async/await is generators + a promise-driving scheduler" — the event-loop lesson formalizes the ordering.

Q3: Design the frontend fetch layer for a real app — what's in it?

One api() wrapper owning: base URL + auth header injection, res.ok → typed errors (ApiError with status/code/body), JSON parsing with content-type check, AbortSignal pass-through + default timeout, retry-with-jitter for idempotent GETs, 401-refresh flow (single-flight so ten parallel 401s trigger ONE refresh), and telemetry hooks. Every screen then imports api, not fetch — policy lives in one file.

Q4: When is return await NOT redundant?

Inside try/catch (and try/finally): plain return hands the pending promise out before it settles, so its rejection bypasses the local catch; return await keeps settlement inside the frame. Outside try blocks it's semantically redundant (just an extra tick) — lint rules allow it in try for exactly this reason. Bonus: return await also keeps the function in async stack traces.

🎯 Key Takeaways

  • Promises settle once: pending → fulfilled/rejected; chains flatten by RETURNING promises; finally cleans up.
  • fetch rejects only on network failure — res.ok is your job; centralize it in an api() wrapper.
  • Independent work runs in Promise.all (waterfalls are the #1 async perf bug); know all/allSettled/race/any cold.
  • for…of awaits serially; all(map) parallelizes; forEach+async is always wrong; bound concurrency for big batches.
  • AbortController cancellation (stale-search!) and retry with exponential backoff + jitter are production requirements.
  • Un-awaited promises escape try/catch; return await matters inside try; new Promise only wraps callback APIs.
