Async: Promises & await Intermediate

JavaScript is single-threaded — it cannot wait. Anything slow (network, disk, timers) is started, the thread moves on, and a callback fires later. Promises organize that chaos; async/await makes it read like normal code.

A Promise Is a Value's Future

fetch("/api/user")                      // returns immediately: a pending Promise
  .then(res => res.json())              // runs when fulfilled
  .then(user => render(user))
  .catch(err => showError(err))         // any failure above lands here
  .finally(() => spinner.hide());       // always runs

// States: pending → fulfilled (value) | rejected (reason). One-way, once.

async/await — Same Machinery, Readable

async function loadUser(id) {                 // async fn ALWAYS returns a Promise
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);   // fetch does NOT reject on 404/500!
    return await res.json();
  } catch (err) {                              // network errors + the throw above
    logger.warn("loadUser failed", err);
    throw err;                                 // re-throw — don't swallow (Error Handling lesson)
  }
}

The !res.ok check is the most-forgotten line in JavaScript: fetch only rejects on network failure — HTTP errors resolve "successfully".

Sequential vs Parallel — the Performance Question

// ❌ 3 round trips, one after another (900ms total):
const user  = await getUser();
const posts = await getPosts();
const ads   = await getAds();

// ✅ all at once (300ms — the slowest one):
const [user, posts, ads] = await Promise.all([getUser(), getPosts(), getAds()]);

// Promise.all rejects fast if ANY rejects. When partial success is fine:
const results = await Promise.allSettled([...]);   // [{status:"fulfilled", value}, {status:"rejected", reason}]
Promise.race([fetchData(), timeout(5000)]);          // first to settle wins — timeout pattern
Promise.any([mirror1(), mirror2()]);                 // first to FULFILL wins

Loops + await

for (const id of ids) await process(id);                       // sequential — when order matters
await Promise.all(ids.map(id => process(id)));                 // parallel — when it doesn't
// ⚠ ids.forEach(async id => …) fires everything and waits for NOTHING — classic bug.

Cancellation (production requirement, always forgotten)

const ctl = new AbortController();
fetch(url, { signal: ctl.signal });
ctl.abort();                            // user navigated away / typed a new search query
// Without this, stale search results race back and overwrite fresh ones.
