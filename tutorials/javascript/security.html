JavaScript Security Ultra-Advanced

The HTML track covered XSS and CSP from the markup side. This lesson is the JavaScript-specific attack surface: the language features and ecosystem habits that create vulnerabilities.

Injection Sinks — Know Your innerHTML Cousins

el.innerHTML = userInput;                  // the famous one
el.outerHTML / el.insertAdjacentHTML(...)  // same risk
eval(userInput)                            // executes arbitrary code — never with any external input
new Function(userInput)                    // eval in a trench coat
setTimeout(userString, 100)                // string form = eval; always pass a FUNCTION
element.setAttribute("href", userInput)    // "javascript:alert(1)" URLs — validate scheme

Rule: treat every value that ever touched a user, a URL, or an API as radioactive until it goes through textContent, a sanitizer, or a validator.

Prototype Pollution — the JS-Specific Classic

// Vulnerable deep-merge (the bug behind dozens of CVEs in npm packages):
function merge(target, src) {
  for (const k in src) {
    if (typeof src[k] === "object") merge(target[k] ??= {}, src[k]);
    else target[k] = src[k];
  }
}
merge({}, JSON.parse('{ "__proto__": { "isAdmin": true } }'));
({}).isAdmin   // true — EVERY object in the app now "has" isAdmin 💥

// Defenses:
merge(Object.create(null), input)          // no prototype to pollute
if (["__proto__","constructor","prototype"].includes(k)) continue;   // skip magic keys
structuredClone / well-audited libs        // don't hand-roll deep merge

Supply Chain — Your Real Attack Surface Is node_modules

npm audit / npm audit fix        — known CVEs in your tree
npm ci                           — install EXACTLY the lockfile (CI must use this)
"overrides": { "vuln-pkg": "^2.1.1" }   — force-patch transitive deps
minimum-release-age policies / provenance (npm publish --provenance)

Real incidents to know: event-stream (maintainer handoff → bitcoin stealer), ua-parser-js (hijacked publish). Median app: your code is 2% of shipped JS — dependencies are the other 98%.

Secrets & Storage — Two Iron Laws

// 1. Anything in frontend JS is PUBLIC. Bundled "secret" keys are not secret.
//    API keys with power stay server-side; the browser gets short-lived, scoped tokens.

// 2. localStorage is readable by ANY script that runs on your page (i.e., by XSS).
//    Session/auth tokens belong in HttpOnly cookies — JS can't read them at all:
Set-Cookie: session=…; HttpOnly; Secure; SameSite=Lax

postMessage — Validate Both Directions

// Receiving: check origin, validate shape
window.addEventListener("message", (e) => {
  if (e.origin !== "https://trusted.app.com") return;
  handle(MessageSchema.parse(e.data));
});
// Sending: never postMessage(secret, "*") — name the target origin.
💡 Staff-level insight: the mature stack is layered so no single mistake is fatal — framework escaping by default → sanitizer for rich text → CSP + Trusted Types as the net → HttpOnly cookies so successful XSS still can't exfiltrate sessions → lockfile + audit gates in CI for the supply chain. Security reviews score the layers, not the absence of bugs.
