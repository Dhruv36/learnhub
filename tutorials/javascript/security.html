Security Production JavaScript

What you'll master: the vulnerability classes that recur throughout real JavaScript applications — XSS in all its forms, CSRF, prototype pollution, ReDoS, and dependency risk — tied together as one coherent defense-in-depth mindset, plus Content Security Policy as a systemic backstop. Prerequisites: Strings (XSS), Prototypes (pollution), Storage (token storage), and Regex (ReDoS) — this lesson is the capstone that assembles every security warning seeded across the track into one systematic model.

Part 1 — XSS: the Central Vulnerability, in Three Forms

// 1. REFLECTED XSS — malicious input from the URL/request is echoed back into the page:
// yoursite.com/search?q=<script>stealCookies()</script>
document.getElementById("results").innerHTML = `Results for: ${new URLSearchParams(location.search).get("q")}`;
// If the server (or client router) reflects `q` unescaped, the script tag EXECUTES.

// 2. STORED XSS — malicious input is SAVED (a comment, a profile bio) and served to OTHER users later:
// An attacker posts a comment containing <script>...</script>; EVERY user who views it runs the script.
// Worse than reflected — no need to trick a specific victim into clicking a crafted link.

// 3. DOM-BASED XSS — the vulnerability is entirely CLIENT-SIDE, no server involvement at all:
const redirect = new URLSearchParams(location.search).get("redirect");
document.location = redirect;   // "javascript:stealCookies()" as a URL executes arbitrary code
element.innerHTML = location.hash.slice(1);   // reading FROM the URL, writing unescaped INTO the DOM

Cross-Site Scripting is not one bug but a class of them, unified by one root cause: untrusted data flowing into a context where it's interpreted as code instead of as inert data. The Strings and DOM lessons already established the core defense (textContent over innerHTML) — this lesson adds the taxonomy: reflected (input echoed back immediately, requires tricking a victim into a crafted link), stored (input persisted and served to other users later — often the most damaging, since it needs no social engineering per victim), and DOM-based (the entire vulnerability lives in client-side JavaScript, reading from one DOM/URL source and writing unescaped into a dangerous sink, with no server involvement in the flaw itself). All three share the same fix: never let untrusted data reach an execution context unescaped — the specific sink (innerHTML, document.write, eval, a javascript: URL, even some CSS/style contexts) varies, but the discipline is identical.

Part 2 — Content Security Policy: a Systemic Backstop

<!-- Server sends this response header (preferred) or a <meta> tag: -->
Content-Security-Policy: script-src 'self' https://trusted-cdn.com; object-src 'none'; base-uri 'self';

// With a strict CSP in place, even if an attacker manages to INJECT a <script> tag via some
// overlooked hole, the BROWSER ITSELF refuses to execute it — because it didn't come from an
// allowed source. CSP is a defense-in-depth layer, not a substitute for fixing the actual XSS hole.

// The critical, commonly-violated rule: NEVER include 'unsafe-inline' in script-src if you can avoid it —
// it defeats almost the entire purpose of CSP, since it re-allows exactly the inline <script> injection
// XSS relies on.

Content Security Policy is a browser-enforced allowlist, delivered via an HTTP header, declaring exactly which sources scripts, styles, images, and other resources may load from. Its power: even if application code has an overlooked XSS hole and an attacker successfully injects a <script> tag, a properly configured CSP means the browser itself refuses to execute it, because inline scripts (or scripts from unlisted origins) aren't on the allowlist. This is defense in depth — a second, independent layer that catches what the first layer (careful escaping) might have missed — never a replacement for actually preventing the injection in the first place. The single most common CSP mistake is adding 'unsafe-inline' to script-src "to make things work easier," which re-permits exactly the inline-script execution that makes CSP valuable against XSS in the first place — largely defeating the protection.

Part 3 — CSRF: Exploiting Automatic Credential Sending

// The user is logged into bank.com (has a valid session COOKIE).
// They then visit evil.com, which contains:
// <img src="https://bank.com/transfer?to=attacker&amount=1000">
// The BROWSER automatically attaches bank.com's cookies to that request — it doesn't know
// or care that the request originated from evil.com's page. If bank.com's server only checks
// "is there a valid session cookie?", the transfer SUCCEEDS — the user never clicked anything.

// DEFENSES (server-side, but you must understand WHY as a frontend/full-stack dev):
// 1. SameSite cookies — SameSite=Lax/Strict tells the BROWSER ITSELF not to send this cookie
//    on cross-site requests, closing the hole at the browser level (the modern default/best fix).
// 2. CSRF tokens — a per-session random token embedded in forms; evil.com can't read/guess it,
//    so it can't include it in its forged request, and the server rejects requests missing it.

Cross-Site Request Forgery exploits a specific browser behavior: cookies are attached to a request automatically based on the target domain, regardless of what page/origin initiated the request. If your session is just "a valid cookie present," any page anywhere on the internet can trigger authenticated actions on your behalf simply by causing your browser to make a request to your logged-in site — you never see or click anything malicious yourself. This is distinct from XSS (which runs code on the vulnerable site); CSRF runs a request against the vulnerable site from elsewhere. The modern, browser-level fix — SameSite cookies — is now the default in modern browsers for cookies that don't explicitly opt out, which has meaningfully reduced this class of vulnerability platform-wide; CSRF tokens remain a defense-in-depth layer for sensitive actions regardless.

Part 4 — Prototype Pollution: Attacking the Shared Chain

// A "helpful" recursive merge function, naively written:
function merge(target, source) {
  for (const key in source) {
    if (typeof source[key] === "object") { merge(target[key] ??= {}, source[key]); }
    else target[key] = source[key];
  }
}
// An attacker controls `source` (e.g. it's parsed from a user-supplied JSON request body):
merge({}, JSON.parse('{"__proto__": {"isAdmin": true}}'));
// `merge` walks into target.__proto__ ({}.__proto__ IS Object.prototype) and sets isAdmin = true
// on Object.prototype ITSELF — now EVERY plain object in the ENTIRE application inherits isAdmin: true!
({}).isAdmin;   // true — even a totally unrelated, freshly-created object, anywhere in the app

// FIX — reject dangerous keys explicitly, or avoid the shared prototype entirely:
function safeMerge(target, source) {
  for (const key in source) {
    if (key === "__proto__" || key === "constructor" || key === "prototype") continue;   // reject
    if (typeof source[key] === "object") { safeMerge(target[key] ??= {}, source[key]); }
    else target[key] = source[key];
  }
}
// or: use Map/Object.create(null) for the target, which have no __proto__ special meaning at all.

This is the Prototypes lesson's warning made concrete and dangerous. A naive recursive merge/clone/set function that walks into __proto__ as if it were an ordinary key doesn't create a normal property — it reaches into and mutates Object.prototype itself, the shared prototype every plain object in the entire application inherits from. The consequence is severe precisely because it's shared: a single injection point pollutes global behavior, potentially bypassing authorization checks (if (user.isAdmin) now reads true on objects that never explicitly set it) or, in the worst documented real-world cases, escalating to remote code execution via sinks that trust a polluted property. This has been a real, high-severity vulnerability class in numerous popular npm packages (lodash's older merge, among others) — the fix is either explicitly rejecting the dangerous key names in any function that walks arbitrary user-controlled object keys, or avoiding plain-object targets for such data entirely (a Map, per the Collections lesson, has no __proto__ special-casing at all).

Part 5 — ReDoS: When a Regex Becomes a Denial-of-Service Vector

// A validation regex with NESTED quantifiers over the same characters — looks harmless:
const emailLike = /^([a-zA-Z0-9]+)+@/;
// For a carefully crafted non-matching input, backtracking becomes EXPONENTIAL:
emailLike.test("aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa!");
// This single .test() call can hang the thread for seconds, minutes, or effectively forever as
// the input grows by just a FEW more characters — a real, documented production DoS vector
// (the Regex lesson covers the mechanism; this lesson frames it as a SECURITY concern):
// an attacker submits ONE crafted string to a form field and hangs your server's event loop
// (Event Loop lesson — a blocked thread serves NO other requests while this runs).

The Regex lesson covered why catastrophic backtracking happens (ambiguous nested quantifiers force exponential-time matching). Here's the security framing: any regex applied to untrusted input — a form field, a URL parameter, an API payload — is a potential attack surface, not just a correctness concern. Because JavaScript is single-threaded (Event Loop lesson), a server-side regex that hangs for even a few seconds on one malicious request blocks that thread from serving any other request during that time — a single crafted input from one attacker can degrade or deny service to every other user. This has caused real, documented production outages. The defense: audit every regex applied to user input for nested/overlapping quantifiers, use a regex complexity linter in CI, and consider execution timeouts or RE2-style linear-time regex engines for anything processing untrusted patterns at scale.

Part 6 — Dependency & Supply Chain Risk

// Your app's REAL attack surface isn't just YOUR code — it's every line of every dependency,
// AND every line of every dependency's dependencies, transitively, often hundreds of packages deep.

// A compromised or malicious package can: steal environment variables/secrets at INSTALL time
// (a postinstall script runs arbitrary code the moment `npm install` completes), exfiltrate data
// at RUNTIME, or inject a backdoor that activates only under specific conditions to evade detection.
// This has happened repeatedly in real, widely-used npm packages (event-stream, ua-parser-js, and others).

// Practical mitigations:
// - `npm audit` / `npm audit fix` — checks installed packages against known-vulnerability databases
// - Lockfiles (Tooling lesson) pin EXACT versions — preventing a compromised NEW version from
//   being silently pulled in on your next `npm install`
// - Minimize dependencies — every package added is attack surface you don't control or fully review
// - Review what a new dependency actually DOES before adding it, especially install scripts

Modern JavaScript applications routinely depend on hundreds of transitive packages — and every one of them runs with the same privileges as your own code. A malicious or compromised package isn't a theoretical risk: it's happened repeatedly to real, widely-trusted npm packages, sometimes via a maintainer's account being compromised, sometimes via a subtly malicious update slipped past review. The Tooling lesson's lockfile discussion has a direct security dimension here: pinning exact versions means a compromised new release of an existing dependency doesn't silently get pulled into your build on the next install — you'd have to explicitly update and (ideally) review the change first. npm audit checks your dependency tree against known-vulnerability databases as a baseline, automatable check, though it can't catch a genuinely novel supply-chain attack — the deeper mitigation is deliberately minimizing dependency count and scrutinizing what a new package (and its install scripts specifically) actually does before adding it.

Part 7 — Secure Randomness & Sensitive Data

// Math.random() is NOT cryptographically secure — its internal algorithm is predictable/reproducible
// given enough samples, meaning an attacker can potentially PREDICT future "random" values:
const weakToken = Math.random().toString(36);   // ❌ NEVER for tokens, session ids, passwords, keys

// crypto.getRandomValues() uses the OS's cryptographically secure random number generator:
const strongToken = crypto.randomUUID();                          // secure, standard UUID
const bytes = crypto.getRandomValues(new Uint8Array(32));         // secure raw random bytes

// Sensitive data handling (tying together the Storage lesson's guidance):
// - Auth tokens: HttpOnly cookies (JS can't read them → immune to XSS token theft), NOT localStorage
// - Passwords: NEVER handled/hashed client-side as the actual security boundary — always over HTTPS,
//   hashed with a proper algorithm (bcrypt/argon2) SERVER-SIDE; client-side hashing is not a substitute
// - Never log sensitive data (tokens, passwords, PII) to the console/error trackers in production

Two focused reminders that tie earlier lessons' security notes together. First, from the Numbers lesson: Math.random() is never acceptable for anything security-sensitive — it's a fast, statistically-uniform generator, not a cryptographically unpredictable one, and its output can potentially be predicted by an attacker who observes enough samples. crypto.getRandomValues() (and the convenient crypto.randomUUID()) use the operating system's cryptographically secure random source and are the only correct choice for tokens, session identifiers, or anything else where unpredictability is a security property. Second, the Storage lesson's token-storage guidance (HttpOnly cookies over localStorage for auth) is worth restating here as part of the unified picture: it's specifically the XSS defense (Part 1) applied to credential storage — if JS can't read the token, an XSS hole can't steal it.

Part 8 — Build It: a Defense-in-Depth Comment System

Let's assemble every technique from this lesson into one coherent feature — a user comment system, which touches nearly every vulnerability class covered — demonstrating layered defense rather than any single fix.

// 1. RENDER safely — textContent by default; DOMPurify only if rich HTML is genuinely required (Part 1):
function renderComment(comment) {
  const el = document.createElement("div");
  const author = document.createElement("strong");
  author.textContent = comment.author;             // safe — never innerHTML with user data
  const body = document.createElement("p");
  body.textContent = comment.body;                  // safe — plain text comments
  el.append(author, body);
  return el;
}

// 2. SUBMIT safely — CSRF protection via SameSite cookies (server-configured) + explicit token (Part 3):
async function submitComment(text, csrfToken) {
  const res = await fetch("/api/comments", {
    method: "POST",
    headers: { "Content-Type": "application/json", "X-CSRF-Token": csrfToken },
    credentials: "same-origin",                       // don't send credentials cross-origin unnecessarily
    body: JSON.stringify({ text }),
  });
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

// 3. MERGE user-supplied metadata safely — reject dangerous keys (Part 4):
function safeMergeMetadata(target, userSuppliedFields) {
  const DANGEROUS = new Set(["__proto__", "constructor", "prototype"]);
  for (const key of Object.keys(userSuppliedFields)) {
    if (DANGEROUS.has(key)) continue;                 // reject silently — or log/reject the whole request
    target[key] = userSuppliedFields[key];
  }
  return target;
}

// 4. SEARCH comments safely — escape user query before embedding in a regex (Part 5 / Regex lesson):
function escapeRegex(str) { return str.replace(/[.*+?^${}()|[\]\\]/g, "\\$&"); }
function searchComments(comments, query) {
  const pattern = new RegExp(escapeRegex(query), "i");   // no catastrophic backtracking risk — literal match
  return comments.filter(c => pattern.test(c.body));
}

No single technique here is new — every piece traces back to a specific part of this lesson (or an earlier one). What's worth internalizing is the shape of the whole: a real feature touches rendering (XSS), network requests (CSRF), data merging (prototype pollution), and search (ReDoS) — security isn't one lesson's topic applied once, it's a lens applied to every piece of a feature that touches untrusted input. This is precisely why security is the capstone of this track rather than a standalone module: it's the discipline of asking "where does untrusted data enter, and is this specific sink safe for it?" at every boundary, using the specific tools each lesson already gave you.

Interview Questions

Q1 (beginner): What's the root cause common to all forms of XSS?

Untrusted data flowing into a context where the browser interprets it as executable code rather than inert text — most commonly assigning it to innerHTML (or similar dangerous sinks like document.write, a javascript: URL, or certain attribute contexts), where the browser parses it as HTML/script and runs it. The three named variants (reflected, stored, DOM-based) differ only in where the untrusted data originates and how it reaches the dangerous sink — the underlying flaw and the fix (never place untrusted data into an execution context without escaping or using a safe sink like textContent) are the same across all of them.

Q2 (beginner): What is Content Security Policy and what does it protect against?

CSP is a browser-enforced allowlist, set via an HTTP response header, specifying which sources are permitted to provide scripts, styles, images, and other resources for a page. Its main security value is as a backstop against XSS: even if an attacker manages to inject a script tag into the page despite your application's own defenses, a properly configured CSP means the browser itself will refuse to execute that script unless it comes from an explicitly allowed source. It's defense in depth — a second, independent layer — not a replacement for actually preventing the injection through proper escaping. The most common way to accidentally defeat it is including 'unsafe-inline' in script-src, which re-permits exactly the inline-script execution CSP exists to block.

Q3 (mid): Explain CSRF and how SameSite cookies defend against it.

CSRF exploits the fact that browsers automatically attach a site's cookies to any request targeting that site, regardless of which page or origin actually initiated the request. If a user is logged into a site (has a valid session cookie) and then visits an unrelated malicious page, that page can cause the user's browser to send an authenticated request to the logged-in site — triggering an action (a transfer, a password change) using the victim's own valid session, without the victim clicking anything malicious themselves or the attacker ever seeing the victim's credentials. SameSite cookies (set to Lax or Strict) close this at the browser level: they instruct the browser not to send the cookie on cross-site requests at all, so when the malicious page's forged request reaches the target site, the authenticating cookie simply isn't attached, and the request is treated as unauthenticated. This is now the default behavior for cookies that don't explicitly opt out in modern browsers, meaningfully reducing CSRF risk platform-wide, though CSRF tokens remain a valuable additional layer for sensitive operations.

Q4 (mid): How does prototype pollution happen, and why is it dangerous even though it "just" sets a property?

It happens when code that recursively merges, clones, or sets nested properties on an object treats attacker-controlled keys like __proto__, constructor, or prototype as ordinary property names without filtering them. Because obj.__proto__ is a live accessor to the object's actual prototype (not a normal data property), naively walking into it and assigning a value there doesn't create a property on obj — it mutates the shared prototype object itself, typically Object.prototype, which every plain object in the entire JavaScript runtime inherits from. It's dangerous precisely because of that sharing: a single successful injection can silently add or override a property that then appears to exist on every object application-wide, including objects that had nothing to do with the original attack — which can bypass authorization checks that trust an inherited property, corrupt logic that assumes certain properties are absent by default, or in severe cases chain into remote code execution if a polluted property is later trusted by a sensitive sink (like a template engine or a spawned command's arguments). It's not "just a property" — it's a global, application-wide mutation from a single input.

Q5 (senior): Design a layered security review checklist for a feature that accepts and displays user-generated content, covering each vulnerability class from this lesson.

A systematic review walks every place untrusted data enters and every sink it could reach, per vulnerability class: (1) XSS — for every place user content is rendered, confirm it uses textContent/safe DOM construction by default; if rich HTML is genuinely required, confirm it passes through a maintained sanitizer (DOMPurify) with an explicit allowlist, not a hand-rolled escaper; check attribute contexts too (a user-controlled href could be a javascript: URL). (2) CSP — verify a CSP header is actually present and doesn't include 'unsafe-inline'/'unsafe-eval' in script-src without a specific justified exception (nonce/hash-based inline scripts if truly needed). (3) CSRF — confirm state-changing endpoints (POST/PUT/DELETE) that this feature calls are protected by SameSite cookies and, for sensitive actions, an explicit CSRF token; verify credentials mode on fetch calls is scoped appropriately, not blanket cross-origin. (4) Prototype pollution — audit any code path that merges/clones user-supplied JSON into an object (settings, metadata, nested form data) for __proto__/constructor/prototype key filtering, or confirm it uses a Map/null-prototype object instead of a plain object as the merge target. (5) ReDoS — identify every regex applied to this feature's user input (search, validation, formatting) and check for nested/overlapping quantifiers; run a regex-safety linter if available. (6) Storage/secrets — confirm no sensitive data (auth tokens, PII) from this feature ends up in localStorage, client-side logs, or error-tracker payloads sent from the browser. (7) Dependencies — if the feature introduced any new package (a rich-text editor, a markdown renderer), review what it does, check its install scripts, and run npm audit. (8) Server-side re-validation — confirm every client-side check (validation, sanitization) is independently re-enforced server-side, since client code is never a trust boundary. The unifying method across all eight: trace every path untrusted data can take through the feature, and at each sink ask "is this specific sink safe for unescaped/unvalidated data, and if not, where exactly does sanitization/validation/escaping happen on this path" — a checklist derived directly from the vulnerability classes this lesson covers, applied systematically rather than relying on catching issues ad hoc during code review.

Q6 (senior): Compare sanitization, escaping, and CSP as three different layers of XSS defense. Why use all three rather than relying on just one?

Escaping transforms specific characters (< > & " ') into their inert HTML-entity equivalents at the point where untrusted data is inserted into a specific context (HTML body, an attribute, a URL, JavaScript string literal, CSS) — it's context-sensitive (the correct escaping differs by context) and it's the first, most fundamental layer, applied at every individual injection point. Its weakness: it must be applied correctly and consistently at every single sink across a potentially large codebase, and a single missed spot anywhere reintroduces the vulnerability — it doesn't scale as a sole defense across a large team/codebase over time. Sanitization (DOMPurify and similar) is different in kind: rather than escaping everything, it parses untrusted HTML and rebuilds it against an explicit allowlist of safe tags/attributes, deliberately permitting genuine rich content (bold, links, images from a trusted source) while stripping dangerous constructs (script tags, event handler attributes, javascript: URLs) — necessary specifically when the product requirement is "render some user-authored HTML," where pure escaping would defeat the purpose by displaying raw tags as visible text instead of rendering them. Its weakness: sanitizer correctness is itself a hard, evolving problem (new bypass techniques are found periodically), so it depends on using a well-maintained, actively-patched library, never a custom implementation. CSP operates at a completely different layer: browser-enforced, independent of application code correctness entirely — it doesn't care whether your escaping or sanitization was implemented correctly; it simply refuses to execute inline scripts or scripts from unlisted origins, full stop. Its weakness: it's coarse-grained (an allowlist of origins, not aware of your specific data flows) and can be weakened by careless configuration ('unsafe-inline'). The reason to use all three together is that they fail independently — escaping fails if one developer forgets it at one call site; sanitization fails if the library has an undiscovered bypass; CSP fails only if misconfigured with an overly permissive policy. Layering means a single mistake in any one layer doesn't result in a successful attack, because the other layers are still standing — this is the literal definition of defense in depth, and it's why mature security practice never relies on "we escape everything correctly" as the sole guarantee, however careful the team.

🏋️ Exercises

1 (easy): Identify the XSS type. Classify each as reflected, stored, or DOM-based: (a) a search results page echoes an unescaped query parameter into the page; (b) a forum post containing a script tag is saved and later viewed by other users; (c) client-side JS reads location.hash and writes it into innerHTML with no server round-trip.
Answers

(a) Reflected — the malicious input comes from the current request and is immediately echoed back. (b) Stored — the payload is persisted and served to other users later, without needing a crafted link per victim. (c) DOM-based — the entire vulnerable flow (source: URL fragment; sink: innerHTML) happens purely in client-side JavaScript with no server involvement in the flaw.

2 (easy): Fix an unsafe render. Rewrite this to safely display a user's display name: 
el.innerHTML = `Welcome, ${user.displayName}!`;
Solution 
el.textContent = `Welcome, ${user.displayName}!`;

textContent inserts the entire string as literal text — any HTML/script in displayName displays as visible text instead of executing, closing the XSS hole with no other change needed.

3 (medium): Reject dangerous merge keys. Given the lesson's naive merge function, write a test input that would pollute Object.prototype, then show the fixed version rejecting it.
Solution 
const malicious = JSON.parse('{"__proto__":{"polluted":true}}');

function safeMerge(target, source) {
  for (const key of Object.keys(source)) {
    if (["__proto__", "constructor", "prototype"].includes(key)) continue;
    target[key] = source[key];
  }
  return target;
}
safeMerge({}, malicious);
({}).polluted;   // undefined — NOT polluted, because __proto__ was rejected as a key

Using Object.keys(source) (which never includes __proto__ as an enumerable own key from JSON.parse's plain object) plus an explicit denylist check stops the dangerous key from ever being used as an assignment target.

4 (medium): Generate a secure token. Write a function generating a cryptographically secure random session token, and explain why Math.random() would be wrong here.
Solution 
function generateSessionToken() {
  return crypto.randomUUID();   // or: Array.from(crypto.getRandomValues(new Uint8Array(32))).map(b => b.toString(16).padStart(2,"0")).join("")
}

Math.random() is generated by a fast, non-cryptographic algorithm whose output can potentially be predicted by an attacker who observes enough samples (its internal state is reproducible) — using it for a session token means an attacker could potentially guess or derive valid tokens. crypto.randomUUID()/getRandomValues() use the OS's cryptographically secure random source, specifically designed to be unpredictable even to an attacker with significant computational resources.

5 (hard): Write a minimal CSP. For a page that loads scripts only from its own origin and a single trusted CDN, and allows no plugins/objects, write an appropriate CSP header value.
Solution 
Content-Security-Policy: default-src 'self'; script-src 'self' https://trusted-cdn.com; object-src 'none'; base-uri 'self';

default-src 'self' sets a safe baseline for unlisted resource types; script-src explicitly allows only the page's own origin and the named CDN (no 'unsafe-inline', so any injected inline script is blocked by the browser regardless of any application-level XSS hole); object-src 'none' blocks Flash/plugin-based vectors; base-uri 'self' prevents a <base> tag injection from redirecting relative URLs to an attacker's origin.

6 (hard): Full defense-in-depth review. A feature lets users submit a bio with basic formatting (bold, links) that other users see. List the specific defense for each of these risks: XSS via the rich text, CSRF on the submit endpoint, and a malicious link (javascript: URL) in the bio.
Solution

XSS via rich text: since actual HTML rendering is required (bold, links), plain textContent isn't an option — sanitize the submitted HTML through a maintained library (DOMPurify) with an explicit allowlist restricted to exactly the needed tags (<b>, <a>), stripping everything else including event handler attributes and script tags. CSRF on submit: ensure the bio-update endpoint requires SameSite cookie protection (default in modern browsers) plus, since this is a state-changing write, an explicit CSRF token validated server-side, so a forged cross-site request can't update another user's bio. Malicious javascript: link: a naive sanitizer allowlist might permit <a href="...">but a DOMPurify-class sanitizer specifically strips javascript: (and other dangerous) URL schemes from href/src attributes as part of its allowlist logic — confirm the sanitizer configuration does this (it's a common, well-known bypass class if a sanitizer only checks tag names and not attribute value schemes), and additionally consider adding rel="noopener noreferrer" and validating the href's scheme is http(s): explicitly before rendering as an extra defensive layer.

🎯 Key Takeaways

  • XSS (reflected, stored, DOM-based) shares one root cause — untrusted data reaching an execution context — and one fix: safe sinks (textContent) or proper escaping/sanitization, always.
  • CSP is a browser-enforced backstop against XSS, independent of application code — but 'unsafe-inline' defeats its core protection.
  • CSRF exploits automatic cookie-sending across origins; SameSite cookies (now default) close this at the browser level, with CSRF tokens as an additional layer.
  • Prototype pollution turns a naive recursive merge into a global, application-wide corruption — always reject __proto__/constructor/prototype keys or use Map.
  • Any regex on untrusted input is a potential ReDoS attack surface — audit for nested/overlapping quantifiers, since a single-threaded server can be blocked by one crafted request.
  • Security is a defense-in-depth discipline applied at every boundary where untrusted data enters — escaping, sanitization, CSP, SameSite, key filtering, and secure randomness are independent layers, not alternatives to each other.
