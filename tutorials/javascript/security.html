JavaScript Security Ultra-Advanced

The HTML track covered XSS/CSP from the markup side; this lesson is the JavaScript-specific attack surface — the language features and ecosystem habits that create holes, and the layered defenses that mean no single mistake is fatal. Frontend security is now mostly a supply-chain problem: your own code is often 2% of what ships.

Injection Sinks — Know Every Door

// HTML sinks (parse markup → execute):
el.innerHTML = x;   el.outerHTML = x;   el.insertAdjacentHTML(pos, x);   document.write(x);
// Code sinks (string → executable code):
eval(x);   new Function(x);   setTimeout(x, 0)/setInterval(x, 0) with a STRING first arg;
// URL/attribute sinks:
el.setAttribute("href", x)  // "javascript:…" executes on click
location = x;  el.src = x;   // navigation / script-src injection
// Style sink:
el.style.cssText = x         // CSS can exfiltrate via url() in some contexts

Rule: treat every value that touched a user, a URL, an API, or storage as radioactive until it passes through textContent, a sanitizer, a validator, or a scheme allow-list. Blocklists ("just strip <script>") always lose — onerror, javascript:, and mutation-XSS defeat them.

XSS Defense in JS Terms — the Layers

// 1. Escape by default: text as text
el.textContent = comment;                         // frameworks do this for {expr}

// 2. Sanitize when rich HTML is required — allow-list, audited lib:
import DOMPurify from "dompurify";
el.innerHTML = DOMPurify.sanitize(userHtml, { ALLOWED_TAGS: ["b","i","a","p","ul","li"] });
el.setHTML(userHtml);                             // native Sanitizer API where available

// 3. Validate URLs before assigning:
const safe = (u) => { try { const p = new URL(u, location.origin);
  return ["http:","https:","mailto:"].includes(p.protocol) ? u : "#"; } catch { return "#"; } };

// 4. Trusted Types (the endgame — browser-enforced):
// CSP: require-trusted-types-for 'script'  → innerHTML = string THROWS unless via a policy:
const policy = trustedTypes.createPolicy("app", { createHTML: s => DOMPurify.sanitize(s) });
el.innerHTML = policy.createHTML(userHtml);       // the ONLY door, and it sanitizes

Prototype Pollution — the JS-Specific Classic

// A naive deep-merge (the bug behind dozens of npm CVEs):
function merge(target, src) {
  for (const k in src) {
    if (typeof src[k] === "object" && src[k]) merge(target[k] ??= {}, src[k]);
    else target[k] = src[k];
  }
}
merge({}, JSON.parse('{ "__proto__": { "isAdmin": true } }'));
({}).isAdmin;     // true — EVERY object now "has" isAdmin → auth bypasses, DoS, RCE-adjacent

// Defenses (layer them):
if (["__proto__","constructor","prototype"].includes(k)) continue;   // skip magic keys
const target = Object.create(null);                                   // no prototype to pollute
Object.freeze(Object.prototype);                                      // belt-and-suspenders
// …or just don't hand-roll deep merge — use structuredClone / a hardened library.

Secrets & Storage — Two Iron Laws

// LAW 1: anything in frontend JS is PUBLIC. Minification ≠ encryption.
const STRIPE_SECRET = "sk_live_…";   // ❌ readable in the bundle by anyone with DevTools
// Powerful keys stay server-side; the browser gets short-lived, narrowly-scoped tokens.
// Publishable/public keys (pk_…) are fine — they're designed to be exposed.

// LAW 2: localStorage is readable by ANY script on your page (i.e., by any XSS).
localStorage.token = jwt;            // ❌ one XSS = stolen session
// Auth tokens belong in HttpOnly cookies — JS literally cannot read them:
// Set-Cookie: session=…; HttpOnly; Secure; SameSite=Lax
//   HttpOnly (no JS access) · Secure (HTTPS only) · SameSite (CSRF mitigation)

Cross-Context Messaging & CSRF

// postMessage — validate BOTH directions:
window.addEventListener("message", (e) => {
  if (e.origin !== "https://trusted.example.com") return;   // ALWAYS check origin
  const data = MessageSchema.parse(e.data);                  // validate shape (never trust structure)
  handle(data);
});
iframe.contentWindow.postMessage(payload, "https://trusted.example.com");  // never "*" with secrets

// CSRF (different from XSS): the attacker's site makes the BROWSER send a request with your
// cookies. Defenses: SameSite=Lax/Strict cookies (primary), anti-CSRF tokens, and checking
// Origin/Referer on state-changing requests. CORS does NOT prevent CSRF — common confusion.

Interview Questions

Q1: A comment feature executes attacker JS. Give the layered fix (not just one line).

① Change the sink: render with textContent (or sanitize with an allow-list if formatting is required) — never blocklist. ② Add CSP (script-src without unsafe-inline, ideally nonce + strict-dynamic) so even an injected payload can't execute or exfiltrate. ③ HttpOnly cookies so a successful XSS still can't read the session. ④ Consider Trusted Types to make DOM-XSS a browser-enforced impossibility. Defense in depth: no single failure is catastrophic — that's the answer they want.

Q2: Explain prototype pollution and why Object.create(null) helps.

Attacker-controlled keys like __proto__/constructor.prototype in a recursive merge/set land on Object.prototype, so ALL objects inherit attacker properties — flipping auth flags, poisoning option defaults, DoS. Object.create(null) makes a target with NO prototype chain, so writing __proto__ creates a harmless own property instead of touching the shared prototype. Layer with magic-key guards and not hand-rolling merges.

Q3: localStorage vs HttpOnly cookies for auth tokens — trade-offs?

localStorage: easy JS access, works with bearer-header APIs, immune to CSRF — but READABLE BY XSS (any injected script steals it). HttpOnly cookies: invisible to JS (XSS can't exfiltrate), auto-sent — but bring CSRF risk (mitigated by SameSite + tokens) and need CORS/credentials care. Consensus for sessions: HttpOnly cookies, because XSS is the more common and more catastrophic vector, and SameSite neutralizes most CSRF. (Bearer tokens in memory + refresh cookie is a common hybrid.)

Q4: Why is "your own code is 2% of shipped JS" a security statement?

Because the other 98% — transitive npm dependencies — runs with the same privileges in your users' browsers, and you didn't write or review it. A single compromised maintainer, hijacked publish, or typosquat (event-stream, ua-parser-js) executes in every session. Hence the emphasis on lockfiles, audit gates, provenance, min-release-age, and SRI: the attack surface is the tree, not your diff.

🏋️ Practice (Playground — safely, on your own page):
1. Build a comment box that's XSS-vulnerable (innerHTML), fire the <img onerror> payload against yourself, then fix it three ways (textContent, DOMPurify, + a CSP meta tag) and confirm each stops it.
2. Write the naive merge, pollute Object.prototype through it, then harden with all three defenses.
3. Implement the safe-URL validator; test javascript:, data:, http:, and a relative URL.
4. Run npm audit on a real project; pick one advisory and trace whether it's a direct or transitive dep.
5. Build an origin-checked, schema-validated postMessage receiver.

🎯 Key Takeaways

  • Know every sink (innerHTML family, eval/Function, string-timers, href/src, cssText); user data is radioactive until escaped/sanitized/validated.
  • Layer XSS defense: textContent → allow-list sanitizer → CSP → Trusted Types → HttpOnly cookies (so even a hit can't steal sessions).
  • Prototype pollution is JS-specific: guard __proto__/constructor, use null-proto targets, don't hand-roll merges.
  • Two iron laws: frontend JS is public (no secret keys), localStorage is XSS-readable (HttpOnly cookies for auth).
  • postMessage: check origin + validate shape, both directions; CORS ≠ CSRF defense (SameSite + tokens do that).
  • Supply chain is the surface: lockfile + npm ci + audit gates + provenance + min-age + SRI — your code is 2%.
