Promise Combinators & Concurrency Asynchronous JavaScript

What you'll master: the four promise combinators (all, allSettled, race, any) and exactly when each is the right tool, real concurrency patterns (bounded parallelism, timeouts, cancellation with AbortController), and the failure-handling decisions that separate robust async code from fragile code. Prerequisites: Promises and async/await (the sequential-vs-parallel trap especially). This lesson turns "I can await one thing" into "I can orchestrate many things correctly."

Part 1 — The Four Combinators at a Glance

CombinatorResolves whenRejects whenUse for
Promise.allALL fulfil (→ array of values)ANY rejects (fail-fast)All must succeed; you need every result
Promise.allSettledALL settle (→ array of {status, value/reason})Never rejectsWant every outcome regardless of failures
Promise.raceFIRST settles (fulfil OR reject)First settles as a rejectionTimeouts; first-to-respond
Promise.anyFirst to FULFILALL reject (→ AggregateError)First success from redundant sources

All four take an iterable of promises and return a single promise. The difference is entirely in their settling logic — whether they wait for all or the first, and whether a rejection is fatal. Picking the right one is a design decision about how failures should behave, which the rest of this lesson makes concrete.

Part 2 — Promise.all: All-or-Nothing

const [user, posts, stats] = await Promise.all([
  fetchUser(), fetchPosts(), fetchStats()   // all start immediately, run concurrently
]);
// Resolves to [userValue, postsValue, statsValue] — in INPUT order (not completion order)
// If ANY rejects, the whole thing rejects immediately with THAT error (fail-fast).

Promise.all runs promises concurrently and resolves to an array of their values in input order (not the order they finished) — the go-to for "I need all of these and they're independent" (the parallel fix from the async/await lesson). Its defining trait is fail-fast: the first rejection immediately rejects the whole combinator with that error, and you lose the results of the ones that did succeed. That's correct when a partial result is useless (you need all three or none), but wrong when you want to salvage what succeeded — for which you use allSettled.

⚠️ Promise.all doesn't cancel the others on rejection. When one rejects, the combinator rejects, but the other in-flight promises keep running to completion in the background (their results are just discarded). To actually stop them, you need AbortController (Part 6). Also: unhandled rejections from the "losing" promises can warn — attach handling if needed.

Part 3 — Promise.allSettled: Every Outcome

const results = await Promise.allSettled([
  fetchUser(1), fetchUser(2), fetchUser(3)
]);
// NEVER rejects. results is:
// [ { status: "fulfilled", value: ... },
//   { status: "rejected",  reason: ... },
//   { status: "fulfilled", value: ... } ]

const succeeded = results.filter(r => r.status === "fulfilled").map(r => r.value);
const failed    = results.filter(r => r.status === "rejected").map(r => r.reason);

Promise.allSettled waits for every promise to settle and never rejects — it resolves to an array describing each outcome ({status:"fulfilled", value} or {status:"rejected", reason}). This is the right tool when you want partial success: loading a dashboard where a broken widget shouldn't blank the whole page, sending N notifications where some may fail, or any batch where you need to know which items worked and which didn't. It trades all's fail-fast for completeness — you always get every result and decide per-item how to handle failures.

Part 4 — Promise.race: First to Settle

// Timeout pattern — race the real work against a timer:
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`timeout after ${ms}ms`)), ms));
  return Promise.race([promise, timeout]);   // whichever settles FIRST wins — value OR error
}
await withTimeout(fetch("/slow"), 3000);

Promise.race settles as soon as the first promise settles — whether it fulfils or rejects. Its classic use is timeouts: race a slow operation against a timer that rejects, and if the timer wins first, the whole thing rejects with a timeout error. It's also used for "first responder" scenarios. The subtlety: race reacts to the first settlement of either kind, so a fast rejection wins over a slower fulfilment — which is exactly what you want for timeouts, but means it's not "first success" (that's any).

Part 5 — Promise.any: First Success

// Fetch from redundant mirrors; use whichever responds successfully FIRST:
const data = await Promise.any([
  fetch("https://mirror1.example/data"),
  fetch("https://mirror2.example/data"),
  fetch("https://mirror3.example/data"),
]).then(r => r.json());
// Resolves with the first FULFILMENT, ignoring rejections.
// Rejects only if ALL reject — with an AggregateError (err.errors is the array of reasons).

Promise.any resolves with the first promise to fulfil, ignoring rejections along the way — the mirror image of race (which reacts to the first settlement of either kind). It only rejects if all promises reject, and does so with an AggregateError whose .errors holds every reason. Use it for redundancy: querying multiple mirrors/replicas and taking the first that succeeds, tolerating individual failures. race vs any: race = first to finish (success or failure); any = first to succeed.

Part 6 — Cancellation with AbortController

const controller = new AbortController();
const { signal } = controller;

const promise = fetch("/api/data", { signal });   // pass the signal to a cancelable API
// Later — cancel it:
controller.abort();                                 // fetch rejects with an AbortError

// Timeout via abort (cancels the actual request, unlike race which leaves it running):
const c = new AbortController();
const t = setTimeout(() => c.abort(), 3000);
try {
  const res = await fetch("/slow", { signal: c.signal });
  clearTimeout(t);
} catch (err) {
  if (err.name === "AbortError") console.log("cancelled");
}
// Shortcut: AbortSignal.timeout(3000) creates a signal that auto-aborts.

AbortController is how you actually cancel async work (a fetch, a stream, any API that accepts a signal). You create a controller, pass its signal to the operation, and call controller.abort() to cancel — the operation rejects with an AbortError. This is strictly better than a race-based timeout for network requests, because it stops the request rather than just ignoring its result (saving bandwidth and server load). It's essential for cleaning up on component unmount (cancel in-flight fetches), debounced search (abort the previous request when a new keystroke arrives), and any user-cancellable operation. AbortSignal.timeout(ms) is a built-in shortcut for the timeout case.

Part 7 — Build It: a Resilient Multi-Source Loader

Let's combine several combinators into one realistic loader: fetch primary data with a timeout and cancellation, fall back to mirrors on failure, and load optional widgets with partial-success tolerance — the shape of a real dashboard's data layer.

async function loadDashboard() {
  const controller = new AbortController();
  const signal = controller.signal;

  // 1. PRIMARY data — required. Try mirrors, take the first success, with a timeout that cancels.
  const timer = setTimeout(() => controller.abort(), 5000);
  let core;
  try {
    core = await Promise.any([
      fetch("/api/primary", { signal }),
      fetch("/api/mirror", { signal }),
    ]).then(r => { if (!r.ok) throw new Error(`HTTP ${r.status}`); return r.json(); });
  } finally {
    clearTimeout(timer);
  }

  // 2. OPTIONAL widgets — a failure of one must NOT blank the dashboard. allSettled.
  const widgetResults = await Promise.allSettled([
    fetch("/api/weather").then(r => r.json()),
    fetch("/api/news").then(r => r.json()),
    fetch("/api/stocks").then(r => r.json()),
  ]);
  const widgets = widgetResults
    .filter(r => r.status === "fulfilled")
    .map(r => r.value);                            // keep only what succeeded

  return { core, widgets, degraded: widgets.length < widgetResults.length };
}

Each combinator earns its place by matching the failure policy to the requirement. Primary data is required, so Promise.any over redundant sources takes the first that works (resilience), wrapped in an AbortController timeout that actually cancels the requests. Widgets are optional, so Promise.allSettled guarantees one broken endpoint can't fail the whole load — we keep the successes and flag the page as degraded. This is exactly the judgment this lesson teaches: the combinator you choose encodes how failures should behave, and real systems mix them per-requirement rather than defaulting everything to Promise.all.

Interview Questions

Q1 (beginner): What's the difference between Promise.all and Promise.allSettled?

Promise.all is fail-fast: it resolves to an array of all values (in input order) only if every promise fulfils, and rejects immediately with the first rejection — discarding the other results. Promise.allSettled waits for all to settle and never rejects; it resolves to an array of outcome objects ({status:"fulfilled", value} or {status:"rejected", reason}), so you get every result and can handle failures per-item. Use all when you need all of them and a partial result is useless; use allSettled when partial success is acceptable and you want to know each outcome.

Q2 (beginner): What does Promise.race do?

It returns a promise that settles as soon as the first of the input promises settles — adopting that promise's outcome, whether it's a fulfilment or a rejection. Its most common use is implementing timeouts: race the real operation against a timer promise that rejects after N ms, so whichever finishes first wins. Note it reacts to the first settlement of either kind, so a fast rejection beats a slower fulfilment — if you specifically want the first success, use Promise.any instead.

Q3 (mid): When would you use Promise.any vs Promise.race?

Promise.race settles on the first promise to settle (success or failure), so it's for "first to respond, and I care about that first response whatever it is" — chiefly timeouts, where a timer rejecting first must win. Promise.any resolves on the first promise to fulfil, ignoring rejections, and only rejects (with an AggregateError) if all fail — so it's for redundancy: querying multiple mirrors/replicas and taking whichever succeeds first, tolerating individual failures. Mnemonic: race = first to finish; any = first to succeed. Use race when a failure should be able to "win" (timeouts); use any when you want to keep trying others past a failure and only care about the first success.

Q4 (mid): How do you cancel an in-flight fetch, and why is that better than a race-timeout?

Use an AbortController: create one, pass its signal to fetch(url, { signal }), and call controller.abort() to cancel — the fetch rejects with an AbortError. It's better than a Promise.race against a timer because race only ignores the slow request's result — the request keeps running to completion in the background, still consuming bandwidth, a connection, and server resources. Abort actually stops the request. This matters for cleanup on unmount (cancel pending fetches so they don't resolve into a dead component), for debounced search (abort the previous request when a new keystroke fires), and for user-cancellable operations. AbortSignal.timeout(ms) is a built-in that produces a signal auto-aborting after a delay, combining timeout and real cancellation.

Q5 (senior): Design the failure-handling strategy for a page that loads one critical resource and several optional widgets. Which combinators and why?

Match each resource's combinator to its failure policy. Critical resource: it must succeed or the page is broken, so treat its failure as fatal — but add resilience and bounds. If there are redundant sources (primary + mirror/CDN), use Promise.any to take the first success and tolerate a dead mirror; wrap it in an AbortController timeout so a hung request fails fast and is actually cancelled (not left running). If there's a single source, just await it with a timeout. On total failure, show a full-page error/retry. Optional widgets: a broken widget must not blank the page, so use Promise.allSettled — render each that fulfilled, show a per-widget error/placeholder for each that rejected, and mark the page "degraded." Concurrency: if the widgets are many, bound the parallelism (worker pool) so you don't exhaust connections or hit rate limits. Cancellation: thread one AbortController (or a signal per resource) so navigating away cancels everything in flight. The overarching principle: the combinator is the failure policy — all for "all-required," allSettled for "partial-ok," any for "first-success redundancy," race/abort for "bounded by time" — and a real page composes several, choosing per-resource based on whether its failure is fatal, tolerable, or recoverable via redundancy. Defaulting everything to Promise.all is the common anti-pattern that makes one flaky optional endpoint able to take down the whole page.

Q6 (senior): Implement a robust fetchWithRetryAndTimeout combining timeout, cancellation, and bounded retries. Discuss the edge cases.

A robust version wraps each attempt in its own timeout+abort and retries with backoff:

async function fetchJSON(url, { retries = 3, timeout = 5000, backoff = 300 } = {}) {
  for (let attempt = 0; ; attempt++) {
    const ctrl = new AbortController();
    const timer = setTimeout(() => ctrl.abort(), timeout);
    try {
      const res = await fetch(url, { signal: ctrl.signal });
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      return await res.json();
    } catch (err) {
      if (attempt >= retries) throw err;                   // give up
      if (err.name === "AbortError" && /* external cancel */ false) throw err;
      await new Promise(r => setTimeout(r, backoff * 2 ** attempt)); // exp backoff
    } finally {
      clearTimeout(timer);                                  // always clear the per-attempt timer
    }
  }
}

Edge cases the design must handle: (1) distinguishing timeout-abort from user-abort — an AbortError from your own timeout should be retried, but one from an external/user cancellation (a shared signal) should propagate immediately, not be retried; you separate these by using a fresh internal controller per attempt and checking whether an external signal was the cause. (2) Only retry idempotent/safe requests — blindly retrying a non-idempotent POST can double-charge or duplicate; retries should be gated to GETs or requests with idempotency keys. (3) Retry only transient failures — a 4xx (client error) won't succeed on retry, so retry on network errors, timeouts, and 5xx/429, but fail fast on 4xx. (4) Respect Retry-After headers on 429/503 rather than blind backoff. (5) Always clear the timer in finally to avoid a dangling abort firing on the next attempt, and add jitter to backoff to avoid thundering-herd retries. (6) Cap total time, not just attempt count, so retries don't exceed the user's patience. The senior point is that "add retries and a timeout" is deceptively subtle: the correctness lives in which errors are retryable, distinguishing your timeout from external cancellation, idempotency safety, and honouring server backpressure signals — combining AbortController, error classification, and backoff-with-jitter into one policy. In production you'd typically use a battle-tested library (or a platform's retry policy) that encodes these, but knowing the edge cases is what lets you configure or review it correctly.

🏋️ Exercises

1 (easy): Pick the combinator. For each, name the right combinator: (a) load 3 required resources, fail if any fails; (b) send 5 emails, report which sent; (c) timeout a request after 2s; (d) fetch from 3 mirrors, use the first success.
Answers

(a) Promise.all (all required, fail-fast). (b) Promise.allSettled (want each outcome). (c) Promise.race against a timeout (or AbortSignal.timeout). (d) Promise.any (first fulfilment, tolerate failures).

2 (easy): Salvage successes. Given Promise.allSettled results, extract just the fulfilled values into an array.
Solution 
const values = results
  .filter(r => r.status === "fulfilled")
  .map(r => r.value);

Filter by status, then map to .value — the standard way to keep the successes from an allSettled batch and ignore (or separately handle) the rejections.

3 (medium): Timeout that cancels. Write fetchWithAbortTimeout(url, ms) that cancels the actual request (not just ignores it) after ms.
Solution 
async function fetchWithAbortTimeout(url, ms) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), ms);
  try {
    return await fetch(url, { signal: controller.signal });
  } finally {
    clearTimeout(timer);
  }
}
// or simply: fetch(url, { signal: AbortSignal.timeout(ms) })

Aborting via the signal stops the request itself, unlike a race timeout which leaves it running. Clearing the timer in finally avoids a stray abort. The AbortSignal.timeout one-liner does the same.

4 (medium): First responder. Query three replica URLs and return the first successful JSON, ignoring failures, but throw a clear error if all fail.
Solution 
async function fromReplicas(urls) {
  try {
    const res = await Promise.any(urls.map(u =>
      fetch(u).then(r => { if (!r.ok) throw new Error(`HTTP ${r.status}`); return r; })));
    return res.json();
  } catch (err) {
    // err is an AggregateError when ALL replicas failed
    throw new Error(`all replicas failed: ${err.errors.map(e => e.message).join(", ")}`);
  }
}

Promise.any returns the first fulfilment; if every replica rejects it throws an AggregateError whose .errors array we summarise into one clear message.

5 (hard): allSettled with a summary. Process a batch of async tasks and return { succeeded: [...], failed: [{index, reason}], successRate }.
Solution 
async function runBatch(tasks) {
  const results = await Promise.allSettled(tasks.map(t => t()));
  const succeeded = [], failed = [];
  results.forEach((r, index) => {
    if (r.status === "fulfilled") succeeded.push(r.value);
    else failed.push({ index, reason: r.reason });
  });
  return { succeeded, failed, successRate: succeeded.length / results.length };
}

allSettled never rejects, so the batch always completes; we partition outcomes into successes and indexed failures and compute a success rate — a reporting shape real batch jobs (imports, bulk sends) need.

6 (hard): Cancellable debounced search. Sketch a search function that, on each new query, aborts the previous in-flight request before starting a new one, so only the latest result is used.
Solution 
function createSearch(endpoint) {
  let controller = null;
  return async function search(query) {
    controller?.abort();                       // cancel the previous request, if any
    controller = new AbortController();
    try {
      const res = await fetch(`${endpoint}?q=${encodeURIComponent(query)}`,
        { signal: controller.signal });
      return await res.json();
    } catch (err) {
      if (err.name === "AbortError") return null;   // superseded by a newer query — ignore
      throw err;
    }
  };
}
const search = createSearch("/api/search");
search("a"); search("ab"); search("abc");   // first two aborted; only "abc" resolves

Each call aborts the prior request's controller before creating a fresh one, so stale responses are cancelled at the network level and their AbortError is swallowed as expected. This prevents the classic race where an earlier, slower response overwrites a later query's result — a real bug in naive autocomplete. (Pair with a debounce timer from the Performance lesson to also limit request frequency.)

🎯 Key Takeaways

  • Four combinators, four failure policies: all (all-or-nothing, fail-fast), allSettled (every outcome, never rejects), race (first to settle), any (first to succeed).
  • Promise.all preserves input order and is the parallel default — but a single rejection discards all successes; use allSettled for partial success.
  • race = first settlement (for timeouts, where a rejection can win); any = first fulfilment (for redundancy across mirrors).
  • AbortController truly cancels async work (fetch, streams) — better than a race-timeout, which leaves the request running; essential for cleanup and debounced requests.
  • The combinator you choose is the failure policy — real systems mix them per-resource (required vs optional vs redundant), not default everything to Promise.all.
  • Bound concurrency and classify retryable errors for robust orchestration; handle AbortError and AggregateError explicitly.
