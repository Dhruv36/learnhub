Functions & Scope Basics

Three Ways to Write a Function

function calcTax(amount) { return amount * 0.18; }        // declaration — hoisted
const calcTax = function (amount) { … };                   // expression
const calcTax = (amount) => amount * 0.18;                 // arrow — implicit return

// Arrow subtleties that matter later:
const double = x => x * 2;              // one param: parens optional
const make = () => ({ id: 1 });         // returning an object literal needs (parens)
// Arrows have NO own `this` — the crucial difference, explored in Closures & this.

Parameters, Production Style

function createUser(name, { role = "viewer", active = true } = {}) {
  // destructured options object with defaults — the signature real APIs use
}
createUser("Dhruv", { role: "admin" });

const sum = (...nums) => nums.reduce((a, b) => a + b, 0);   // rest params
sum(1, 2, 3);   // 6

Once a function takes 3+ arguments, switch to an options object — call sites become self-documenting and adding parameters stops breaking callers.

Scope — Where Names Live

const app = "global";               // module/global scope
function outer() {
  const a = 1;                       // function scope
  if (a) {
    const b = 2;                     // BLOCK scope — let/const live inside { }
  }
  // b is not defined here
}
// Inner scopes see outer names; never the reverse. This chain is what makes closures work.

Hoisting — Why Order Sometimes Doesn't Matter

greet();                            // ✅ works — declarations hoist entirely
function greet() { console.log("hi"); }

sayHi();                            // ❌ ReferenceError — let/const hoist but stay
const sayHi = () => console.log("hi");   //    uninitialized in the "temporal dead zone"

Practical rule: declare before use, prefer const functions for consistency — then hoisting trivia never bites you (but interviews still ask it).

Pure Functions — the Testability Foundation

// Impure: depends on and mutates outside state — hard to test, hard to reason about
let total = 0;
function addToTotal(x) { total += x; }

// Pure: same inputs → same output, no side effects
const add = (a, b) => a + b;

Real systems need side effects (network, DOM, logs) — the architecture skill is pushing them to the edges and keeping the core logic pure. React, Redux, and most testing philosophy build on this.

💡 Production tip: name functions after what they return or do (getUserById, isEligible, formatPrice) — verb-first, specific. A file of well-named small functions reads like documentation; one 200-line handleData() reads like a crime scene.
