Functions & Scope Language Core

What you'll master: the four ways to define a function and when each fits, how parameters really work (defaults, rest, destructuring, the arguments object), first-class functions and higher-order functions as the backbone of JS, arrow functions' two genuine differences from normal functions, and the scope chain that makes it all resolve. Prerequisite: Variables & Declarations (hoisting, TDZ, block scope). This lesson sets up the two that follow — Closures and this & binding — so we deliberately defer this and closure mechanics to them and focus here on functions as values and scope as structure.

Part 1 — Four Ways to Define a Function

// 1. Function DECLARATION — hoisted entirely (callable before its line — Variables lesson)
function add(a, b) { return a + b; }

// 2. Function EXPRESSION — a function value assigned to a binding; NOT hoisted as callable
const sub = function (a, b) { return a - b; };

// 3. ARROW function — concise expression; lexical `this` (Part 6)
const mul = (a, b) => a * b;

// 4. METHOD shorthand (in objects/classes)
const calc = { div(a, b) { return a / b; } };
FormHoisted?Own this?Best for
DeclarationYes (name + body)YesTop-level named helpers, recursion, readability
ExpressionNo (binding rules only)YesAssigning conditionally, passing inline
ArrowNoNo — inherits from enclosing scopeCallbacks, short transforms, preserving outer this
Method shorthandn/aYesObject/class methods

The two you'll write most are declarations (for named, reusable helpers — they read top-down and hoist, so order in a file doesn't matter) and arrows (for callbacks and short transforms). The choice is rarely about hoisting; it's mostly about this (Part 6 and the next-next lesson).

Part 2 — Parameters vs Arguments, and Modern Parameter Features

// Default parameters — evaluated at call time, only when the arg is undefined:
function greet(name = "friend", greeting = `Hello, ${name}`) {   // later defaults can use earlier params
  return `${greeting}!`;
}
greet();               // "Hello, friend!"
greet("Ada");          // "Hello, Ada!"
greet(undefined, "Hi") // "Hi!"  — passing undefined TRIGGERS the default; null does NOT

// Rest parameters — collect the remaining args into a REAL array:
function sum(...nums) { return nums.reduce((a, b) => a + b, 0); }
sum(1, 2, 3, 4);       // 10

// Destructured parameters — pull fields out of an object/array argument:
function createUser({ name, age = 0, role = "user" } = {}) {   // = {} lets you call with no arg
  return { name, age, role };
}
createUser({ name: "Ada", age: 30 });   // { name:"Ada", age:30, role:"user" }

Three modern features that replace old boilerplate. Default parameters fire only when an argument is undefined (note: passing null does not trigger them — null is a real value), and later defaults can reference earlier parameters. Rest parameters (...nums) gather trailing arguments into a genuine array — unlike the legacy arguments object (Part 3). Destructured parameters with a = {} fallback are the idiomatic "named options object" pattern, giving callers order-independent, self-documenting arguments with per-field defaults — the Destructuring lesson goes deeper.

Part 3 — The arguments Object (and Why Rest Replaced It)

function old() {
  console.log(arguments);        // array-LIKE: has length + indices, but NOT a real array
  console.log(arguments.length); // works
  // arguments.map(...)          // ❌ TypeError — no array methods
  return Array.from(arguments).map(x => x * 2);   // must convert first
}
function modern(...args) {       // args IS a real array — .map/.filter/.reduce all work directly
  return args.map(x => x * 2);
}

arguments is a legacy array-like object available in normal (non-arrow) functions, holding all passed arguments. It's not a real array (no map/filter), it doesn't exist in arrow functions, and it's a frequent source of confusion. Rest parameters supersede it entirely — ...args is a genuine array with all the methods, is explicit in the signature, and works everywhere. Recognise arguments in old code; write rest parameters in new code.

Part 4 — First-Class Functions & Higher-Order Functions

// Functions are VALUES — store them, pass them, return them:
const ops = { add: (a, b) => a + b, sub: (a, b) => a - b };   // stored in an object
[1, 2, 3].map(n => n * 2);                                    // passed as an argument
function once(fn) {                                            // returned from a function (HOF)
  let called = false, result;
  return (...args) => {
    if (!called) { called = true; result = fn(...args); }
    return result;
  };
}
const init = once(() => console.log("runs once"));
init(); init();   // logs "runs once" only the first time

Functions are first-class values: you can assign them to variables, store them in arrays/objects, pass them as arguments, and return them. A higher-order function (HOF) is one that takes and/or returns a function — map, filter, reduce, addEventListener, setTimeout, and the once wrapper above are all HOFs. This is the single most important structural idea in JavaScript: the language's expressiveness (array methods, callbacks, promises, middleware, decorators) is all built on functions being ordinary values you can compose. The once example also previews closures — called/result persist between calls (next lesson).

Part 5 — Function Composition & Currying (a Taste)

// Composition: chain single-argument functions right-to-left
const compose = (...fns) => (x) => fns.reduceRight((acc, fn) => fn(acc), x);
const shout = compose(s => s + "!", s => s.toUpperCase(), s => s.trim());
shout("  hi  ");   // "HI!"

// Currying: turn f(a, b) into f(a)(b) — enables partial application
const curry = (fn) => (a) => (b) => fn(a, b);
const add = curry((a, b) => a + b);
const add10 = add(10);   // partially applied — a reusable specialised function
add10(5);                // 15

Because functions are values, you can combine them. Composition builds a pipeline where each function's output feeds the next; currying transforms a multi-argument function into a chain of single-argument functions, enabling partial application (fixing some arguments now, supplying the rest later — add10 above). You don't need to write these from scratch daily (libraries like Ramda/lodash-fp provide them), but recognising the patterns explains a lot of functional-style code and is a common interview topic.

Part 6 — Arrow Functions: the Two Real Differences

const nums = [1, 2, 3];
nums.map(n => n * 2);            // concise: implicit return, no braces needed
nums.map((n, i) => ({ n, i }));  // returning an object literal needs parentheses (else {} = block)

// Difference 1: NO own `this` — inherits from the enclosing lexical scope (this-binding lesson)
const timer = {
  seconds: 0,
  start() { setInterval(() => this.seconds++, 1000); }   // arrow keeps `this` = timer ✅
  // a normal function callback here would have its own `this` (undefined/window) ✗
};

// Difference 2: NO `arguments` object (use rest instead)
const f = (...args) => args;

Arrow functions are not just "shorter functions." They have two behavioural differences that matter: (1) they have no own this — they inherit it from the surrounding lexical scope, which is exactly why they're the right choice for callbacks that need to keep the outer this (the setInterval case), and the wrong choice for object methods that need their own this; and (2) they have no arguments object. They also can't be used as constructors (new throws) and have no prototype. The this difference is the whole reason arrows exist and is covered fully in this, call, apply & bind — for now, just hold: arrows inherit this; normal functions get their own.

🧠 Implicit vs explicit return. x => x * 2 returns implicitly. Add braces and you have a body that needs an explicit return: x => { return x * 2; }. And to implicitly return an object literal, wrap it in parens — x => ({ x }) — otherwise { } is parsed as a function body.

Part 7 — The Scope Chain

const g = "global";
function outer() {
  const o = "outer";
  function inner() {
    const i = "inner";
    return `${g} ${o} ${i}`;   // inner sees ALL THREE — looks outward up the chain
  }
  return inner();
}
outer();   // "global outer inner"

When JavaScript resolves a name, it searches the current scope, then the enclosing function/block scope, outward through each level, up to the global scope — the scope chain. Two critical properties: it's lexical (determined by where code is written, not who calls it — Part 8), and it only goes outward — an outer scope cannot see an inner scope's variables. This one-directional, write-time-fixed structure is what makes closures possible (a returned inner function keeps access to its birth scope even after the outer function has returned) — the entire subject of the next lesson.

Part 8 — Lexical Scope vs Dynamic Scope (Why JS Is Predictable)

const x = "module";
function printX() { return x; }         // `x` resolves where printX was WRITTEN — to "module"
function caller() {
  const x = "local";
  return printX();                       // still "module"! NOT "local"
}
caller();   // "module" — lexical scope: the call site doesn't change name resolution

JavaScript uses lexical (static) scoping: a function's free variables are resolved against the scope where the function was defined, not where it's called. If JS used dynamic scoping (which some older languages do), printX would see caller's local x — but it doesn't. This is why you can read a function and know exactly what its variables refer to just from the surrounding code, without tracing every call site — a huge win for reasoning, and the foundation closures build on. (Note: this is the one thing that is dynamic in normal functions — bound by the call site — which is precisely why it's confusing and gets its own lesson.)

Part 9 — Build It: a Configurable Rate-Limited Logger

Let's combine HOFs, default/rest parameters, and scope into one small, genuinely useful tool: a function that wraps any logging function and drops calls that come in faster than a given interval.

function rateLimit(fn, everyMs = 1000) {         // HOF: takes a fn, returns a wrapped fn
  let lastRun = 0;                              // lives in the closure (next lesson) — persists across calls
  return (...args) => {                          // rest params: forward any arguments
    const now = Date.now();
    if (now - lastRun >= everyMs) {
      lastRun = now;
      return fn(...args);                        // spread the collected args back through
    }
    // else: silently dropped (too soon)
  };
}

const log = rateLimit((...msgs) => console.log("[LOG]", ...msgs), 2000);
log("first");     // prints
log("dropped");   // ignored — within 2s
// ...2s later...
log("prints again");

Every concept from this lesson is here: rateLimit is a higher-order function returning a new function; the default parameter gives a sensible interval; rest parameters forward arbitrary arguments transparently; and lastRun lives in the enclosing scope, surviving between calls — the closure mechanic we formalise next. This is the skeleton of real throttling utilities (the Performance lesson builds the full debounce/throttle pair).

Interview Questions

Q1 (beginner): What's the difference between a function declaration and a function expression?

A declaration (function foo(){}) is hoisted in full — both its name and body are available throughout the enclosing scope, so you can call it before its line in the source. An expression (const foo = function(){}) only follows its binding's hoisting rules: the name foo is in the Temporal Dead Zone (for const/let) until the assignment runs, so calling it earlier throws. Functionally they produce the same kind of function value; the difference is availability/hoisting and that expressions can be anonymous and assigned conditionally.

Q2 (beginner): What's a higher-order function? Give two you use regularly.

A higher-order function is one that takes a function as an argument and/or returns a function. Everyday examples: Array.prototype.map (takes a callback function), addEventListener (takes a handler), setTimeout (takes a callback), and factory-style functions like a debounce that returns a wrapped function. HOFs are possible because functions are first-class values — assignable, passable, and returnable — which is the structural backbone of JavaScript's callback- and composition-heavy style.

Q3 (mid): Name the ways arrow functions differ from normal functions, and give a case where using an arrow is a bug.

Arrows: (1) have no own this — they inherit it lexically from the enclosing scope; (2) have no arguments object; (3) cannot be used with new (they're not constructors) and have no prototype; (4) can't be generators. The classic bug is using an arrow as an object method that relies on this: const obj = { count: 0, inc: () => { this.count++; } } — here this is not obj but the surrounding scope (module/undefined), so this.count is wrong. Object/class methods that need their own this must use method shorthand or a normal function; arrows are for callbacks that should keep the outer this.

Q4 (mid): Explain lexical scoping and how it differs from dynamic scoping.

Lexical (static) scoping resolves a function's free variables against the scope in which the function was written, fixed at author time. Dynamic scoping would resolve them against the scope from which the function is called, varying per call site. JavaScript is lexically scoped (except for this in normal functions, which is dynamic): a function reading a variable x always refers to the x visible where it was defined, regardless of who calls it or what local x the caller has. This predictability — you can determine a name's meaning by reading the surrounding code, not by tracing all call sites — is what makes closures reliable and reasoning about code tractable.

Q5 (senior): Implement curry that supports both f(a)(b)(c) and f(a, b)(c) and f(a, b, c). Explain how it knows when to execute.

A general curry collects arguments across calls until it has at least as many as the original function's declared arity (fn.length), then invokes:

function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) return fn(...args);
    return (...more) => curried(...args, ...more);
  };
}
const f = curry((a, b, c) => a + b + c);
f(1)(2)(3); f(1, 2)(3); f(1, 2, 3);   // all 6

The mechanism: each call compares accumulated argument count to fn.length (the number of declared parameters). If enough have arrived, it calls the real function; otherwise it returns a new function that remembers the args so far (via closure) and waits for more, recursing until the threshold is met. Caveats: fn.length ignores rest/default parameters (they don't count toward arity), so variadic functions can't be curried this way; and it assumes a fixed arity, which is why real libraries let you specify arity explicitly. It's a clean demonstration of closures, recursion, and functions-as-values combined.

Q6 (senior): Compare passing options as positional parameters vs a destructured options object for a function that has grown to 6 parameters. Discuss API evolution.

Positional parameters — createServer(port, host, timeout, retries, tls, logLevel) — degrade badly as they grow: call sites become unreadable (createServer(8080, "localhost", 5000, 3, false, "info") — which boolean was which?), you can't omit a middle argument without passing undefined placeholders, reordering is a breaking change, and adding a new option means appending yet another positional slot. A destructured options object — createServer({ port, host, timeout = 5000, retries = 3, tls = false, logLevel = "info" } = {}) — fixes all of these: arguments are named and self-documenting at the call site, order- independent, individually omittable (each with its own default), and additively evolvable — a new option is just a new destructured key with a default, requiring no change to existing call sites, which is a backward-compatible API change rather than a breaking one. The trade-off is a tiny bit more ceremony for functions with one or two obvious arguments (where positional is clearer), so the rule of thumb is: 1–2 obvious, order-evident parameters → positional; 3+ or any with booleans/optionals → a destructured options object. The options-object pattern is also friendlier to TypeScript (a single named interface documents the shape) and to future refactors, which is why most mature library APIs converge on it for anything configurable.

🏋️ Exercises

1 (easy): Default vs null. Predict the output: 
function f(x = 10) { return x; }
console.log(f(), f(undefined), f(null), f(0));
Answers

10 10 null 0. Defaults fire only for undefined (including the no-arg call and an explicit undefined). null is a real value, so it's passed through unchanged; 0 is also a real value, unaffected by the default.

2 (easy): Sum any count of numbers. Write sum(...) accepting any number of arguments, using rest parameters, and returning their total.
Solution 
const sum = (...nums) => nums.reduce((a, b) => a + b, 0);
sum(1, 2, 3, 4);   // 10
sum();             // 0

Rest gathers all arguments into a real array, so reduce works directly — no arguments conversion needed. The 0 seed handles the empty call.

3 (medium): once(). Write once(fn) that returns a function running fn at most once and returning its cached result thereafter. Prove a second call doesn't re-run fn.
Solution 
function once(fn) {
  let called = false, result;
  return (...args) => {
    if (!called) { called = true; result = fn(...args); }
    return result;
  };
}
let runs = 0;
const init = once(() => ++runs);
init(); init(); init();
console.log(runs);   // 1

called and result live in the closure and persist between invocations, so after the first call the guarded block is skipped and the cached result is returned — a preview of the next lesson's closure state.

4 (medium): Compose. Write pipe(...fns) that runs functions left-to-right (opposite of compose). Use it to build trim → lowercase → append "!".
Solution 
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);
const process = pipe(s => s.trim(), s => s.toLowerCase(), s => s + "!");
process("  HELLO  ");   // "hello!"

pipe uses reduce (left-to-right) where compose uses reduceRight. Each function's output becomes the next's input — the readable, top-to-bottom data-flow order most people prefer.

5 (hard): Arrow-this bug. This counter never increments. Explain why and fix it two different ways: 
const counter = {
  count: 0,
  start() { setInterval(function () { this.count++; }, 1000); }
};
Solution

The setInterval callback is a normal function, so its this is determined by how the timer calls it — not the counter object — so this.count is undefined++ (NaN) or a global write, never counter.count. Fix 1, arrow (inherits the enclosing this):

start() { setInterval(() => { this.count++; }, 1000); }   // arrow keeps this = counter

Fix 2, capture this or bind:

start() { const self = this; setInterval(function () { self.count++; }, 1000); }
// or: setInterval(function () { this.count++; }.bind(this), 1000);

This is exactly the scenario arrow functions were designed for, and the full this rules are the next-next lesson.

6 (hard): Memoize. Write memoize(fn) that caches results by argument, so repeated calls with the same argument skip recomputation. Handle a single-argument function; use the argument as the cache key.
Solution 
function memoize(fn) {
  const cache = new Map();                 // lives in the closure
  return (arg) => {
    if (cache.has(arg)) return cache.get(arg);
    const result = fn(arg);
    cache.set(arg, result);
    return result;
  };
}
let calls = 0;
const slow = memoize(n => { calls++; return n * n; });
slow(4); slow(4); slow(4);
console.log(calls);   // 1 — computed once, then served from cache

The Map in the closure persists across calls, keyed by the argument. A Map (not a plain object) is used so non-string keys work by value/identity correctly (Collections lesson). For multi-argument functions you'd serialise the args into a key (e.g. JSON.stringify) with the usual caveats, or use a nested-map/WeakMap strategy — the Performance lesson expands on memoisation.

🎯 Key Takeaways

  • Four function forms: declaration (hoisted), expression, arrow (lexical this, no arguments), method shorthand — choose mostly by this needs.
  • Modern parameters: defaults (fire only on undefined), rest (...args — a real array, replacing arguments), and destructured options objects for scalable APIs.
  • Functions are first-class values; higher-order functions (take/return functions) are the backbone of callbacks, array methods, composition, and currying.
  • Arrows differ in exactly two behavioural ways that matter — no own this, no arguments — making them ideal for callbacks and wrong for methods.
  • The scope chain resolves names outward from the current scope to global; it's lexical (fixed by where code is written), never dynamic.
  • Lexical scope is what makes JS predictable and closures possible — a function's variables mean the same thing regardless of call site.
