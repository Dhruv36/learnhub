Functions & Scope Basics

Functions are JavaScript's core abstraction — they're values (assignable, passable, returnable), they create scope, and their creation-site scope is what closures (next lessons) capture. Master the mechanics here and closures, this, and hooks all become corollaries instead of mysteries.

Three Forms — and Their Real Differences

function calcTax(amount) { return amount * 0.18; }        // 1. DECLARATION
const calcTax = function (amount) { … };                   // 2. EXPRESSION
const calcTax = (amount) => amount * 0.18;                 // 3. ARROW
DeclarationExpressionArrow
Hoisted callable-before-definition✅ entire body❌ (TDZ on the const)❌
Own this / arguments✅✅❌ inherits from enclosing scope — THE difference
Constructable with new✅✅❌
Best forTop-level named utilitiesRare — arrows wonCallbacks, one-liners, methods needing lexical this
// Arrow syntax fluency:
const double = x => x * 2;                 // one param: parens optional
const rand   = () => Math.random();        // zero params: parens required
const make   = (id) => ({ id, ok: true }); // returning an object literal: WRAP IN PARENS
const area   = (w, h) => {                 // multi-statement: braces + explicit return
  const a = w * h;
  return a;
};

Parameters, Production Style

// Options-object pattern — the signature real APIs converge on:
function createUser(name, { role = "viewer", active = true, tags = [] } = {}) { … }
createUser("Dhruv", { role: "admin" });     // self-documenting call sites
createUser("Asha");                          // ={} default makes options optional entirely

const sum = (...nums) => nums.reduce((a, b) => a + b, 0);   // rest: real Array
fn(...args);                                                  // spread: the inverse

// Default gotcha: defaults fire on undefined ONLY —
f(null)      // null passes through!
f(undefined) // default applies
// Defaults evaluate LEFT→RIGHT and can reference earlier params:
function slice(arr, start = 0, end = arr.length) { … }

Rule of thumb: 3+ positional arguments (or any boolean flag argument) → options object. send("a@b.c", true, false, 3) is unreviewable; named options aren't.

Scope — the Chain That Powers Everything

const app = "global";                 // module/global scope
function outer() {                     // function scope
  const a = 1;
  if (a) {
    const b = 2;                       // block scope (let/const)
    console.log(app, a, b);            // inner sees ALL outer scopes ✅
  }
  // console.log(b);                   // ❌ ReferenceError — never inward/downward
}
// Lookup is LEXICAL: determined by WHERE CODE IS WRITTEN, at parse time —
// not by who calls whom at runtime. (Dynamic-by-call is `this`, a different lesson.)
// Shadowing: an inner `const a` hides the outer one for that block — legal, lint-flagged.

Hoisting — the Two-Pass Model

// The engine parses a scope FIRST (registering declarations), THEN executes:
greet();                                   // ✅ "hi" — declaration fully hoisted
function greet() { console.log("hi"); }

sayHi();                                   // ❌ ReferenceError (TDZ)
const sayHi = () => console.log("hi");

console.log(x);                            // undefined (var hoists initialized)
var x = 5;

Practical policy: declare before use, prefer const functions for uniformity — then hoisting is trivia you answer in interviews, not a behavior you rely on.

Pure Functions & Side Effects — the Architecture Seed

// PURE: same inputs → same output, touches nothing outside
const applyDiscount = (price, pct) => price * (1 - pct / 100);
// IMPURE: reads/writes external state, does I/O
let total = 0;
function addToTotal(x) { total += x; }            // hidden dependency + hidden output
async function saveOrder(o) { await db.insert(o); } // necessary! but architecturally distinct
Why purity paysConcretely
Testableassert(applyDiscount(100, 10) === 90) — no setup, no mocks
Cacheable/memoizableSame inputs → same output makes results reusable (performance lesson)
Parallelizable & reorderableNo shared state = no race conditions
The design rulePure core, effects at the edges — React render purity, Redux reducers, and functional-core/imperative-shell are all this one idea

Higher-Order Functions — Functions as Values

// Taking functions:
[3, 1, 2].sort((a, b) => a - b);
button.addEventListener("click", handleClick);      // pass the REFERENCE — handleClick, not handleClick()

// Returning functions (closure preview):
const multiplier = (k) => (x) => x * k;
const treble = multiplier(3);  treble(7);            // 21

// Composing:
const pipe = (...fns) => (x) => fns.reduce((v, f) => f(v), x);
const slugify = pipe(s => s.trim(), s => s.toLowerCase(), s => s.replaceAll(" ", "-"));
🐛 The classic beginner bug: onClick={save()} CALLS save during render and passes its result; onClick={save} (or () => save(id) when args are needed) passes the function. "My handler fires immediately on load" is always this.

Recursion — When and How Much

function countFiles(node) {
  if (node.type === "file") return 1;                     // base case FIRST
  return node.children.reduce((n, c) => n + countFiles(c), 0);
}
// The call stack is finite (~10⁴ frames): very deep data → RangeError.
// No reliable tail-call optimization across engines — deep traversals convert to a loop + explicit stack:
function countIter(root) {
  let n = 0; const stack = [root];
  while (stack.length) { const x = stack.pop();
    if (x.type === "file") n++; else stack.push(...x.children); }
  return n;
}

Interview Questions

Q1: Function declaration vs expression vs arrow — give the three differences that matter.

(1) Hoisting: declarations are callable before their line; expressions/arrows live in the TDZ. (2) this/arguments: arrows have neither — they inherit lexically, which is why they're right for callbacks and wrong for methods. (3) new: arrows can't construct. Bonus nuance: named function expressions give better stack traces than anonymous ones.

Q2: What is lexical scoping — and what's the alternative?

Name resolution follows where code is WRITTEN: each function carries a chain to its birth scopes, fixed at parse time. The alternative (dynamic scoping — resolve through the CALLER's scopes) exists in some languages; JS's only dynamic-by-call mechanism is this. Lexical scope is the foundation closures stand on: the chain survives as long as functions referencing it do.

Q3: Implement pipe (left-to-right composition). What does it assume?

const pipe = (...fns) => x => fns.reduce((v, f) => f(v), x). Assumes unary functions (adapt multi-arg heads by partial application) and purity for predictability — composing effects makes order load-bearing and testing hard.

Q4: Why do JS engines not guarantee tail-call optimization, and what do you do instead?

TCO is in the ES2015 spec but engines (except JavaScriptCore) declined — it erases stack frames, hurting debugging/stack traces, and shipped-web compatibility made it risky. Practice: iterate with an explicit stack/queue for deep structures; recursion stays for shallow, naturally-recursive data (ASTs, directories) where clarity wins.

🎯 Key Takeaways

  • Functions are values — passed, returned, composed; higher-order functions are the idiom, not the exception.
  • Declarations hoist whole; const/arrow live in the TDZ; arrows have no this/arguments/new — inherit lexically.
  • Scope is lexical: written-location decides, chains outward only, fixed at parse time — closures are this chain surviving.
  • Options objects over positional/boolean args; defaults trigger on undefined only.
  • Pure core, effects at the edges — the one-line architecture that React/Redux formalize.
  • Stack is finite and TCO isn't real: deep traversals go iterative with an explicit stack.
