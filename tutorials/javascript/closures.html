Closures Language Core

What you'll master: what a closure actually is (a function plus the live variables it captured), the fact that it captures variables not values — the root of the classic loop bug and its fix — and the patterns closures power: private state, factory functions, memoisation, the module pattern, and event handlers. Plus the flip side: closures that unintentionally retain memory. Prerequisites: Functions & Scope (lexical scope, the scope chain, HOFs). This is one of the two or three most-tested JavaScript topics — closures are asked in virtually every interview.

Part 1 — What a Closure Is

function makeGreeter(greeting) {     // `greeting` is a local variable of makeGreeter
  return function (name) {           // this inner function uses `greeting`...
    return `${greeting}, ${name}!`;  // ...which belongs to the OUTER function's scope
  };
}
const hello = makeGreeter("Hello"); // makeGreeter has RETURNED — its scope should be gone...
const hi    = makeGreeter("Hi");    // ...but it isn't:
hello("Ada");   // "Hello, Ada!"  — the inner fn still remembers greeting === "Hello"
hi("Bob");      // "Hi, Bob!"     — and this one remembers greeting === "Hi" — separate captures

A closure is a function bundled together with the lexical environment (the variables) it was created in. When makeGreeter returns, normally its local greeting would be discarded — but because the returned inner function still references greeting, that variable is kept alive, attached to the function. Each call to makeGreeter creates a fresh scope, so hello and hi close over independent greeting variables. That's the whole idea: a function remembers where it was born, and keeps its birth scope's variables alive as long as it lives. It's a direct, inevitable consequence of lexical scope (previous lesson) plus functions being values you can return.

Part 2 — Closures Capture Variables, Not Values (the Critical Distinction)

function counter() {
  let count = 0;
  return () => ++count;      // captures the VARIABLE count, not a snapshot of its value
}
const next = counter();
next();  // 1
next();  // 2   — same variable, mutated across calls
next();  // 3

This is the single most important fact about closures: a closure captures the variable itself (a live reference), not a copy of its value at capture time. Each call to next sees and mutates the same count. If closures captured values, count would be frozen at 0 forever. This "live reference" behaviour is exactly what makes closures useful for state — and exactly what causes the loop bug in Part 3.

Part 3 — The Classic Loop Bug (and Three Fixes)

// THE BUG — all three log 3:
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);   // captures the VARIABLE i (one shared binding)
}
// Output: 3, 3, 3
// By the time the timeouts fire, the loop is done and the single `i` === 3.

With var, there is exactly one i for the whole loop, and all three closures capture that same variable. When they finally run, they read its current value: 3. This is the "captures variables, not values" rule biting. Three fixes, in order of preference:

// FIX 1 (best): let — a fresh binding PER iteration (Variables lesson):
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);   // each closure captures its OWN i → 0, 1, 2
}

// FIX 2: an IIFE to manufacture a new scope per iteration (the pre-ES6 way):
for (var i = 0; i < 3; i++) {
  ((j) => setTimeout(() => console.log(j), 100))(i);   // j is a fresh param each call → 0, 1, 2
}

// FIX 3: pass i as an argument that's copied (works for some APIs):
for (var i = 0; i < 3; i++) {
  setTimeout(console.log, 100, i);   // i is copied into the timeout's argument list → 0, 1, 2
}

let is the modern fix — each iteration gets a fresh block-scoped binding, so each closure captures a distinct variable holding that iteration's value. The IIFE version (Fix 2) is what everyone wrote before let existed: an immediately-invoked function creates a new scope, and the parameter j is a fresh variable copied per iteration. Understanding all three cements the "variable vs value" distinction — and this is one of the most-asked interview questions in the language.

Part 4 — Pattern: Private State & the Module Pattern

function createAccount(initial) {
  let balance = initial;                 // PRIVATE — no external code can touch it directly
  return {
    deposit(n) { balance += n; return balance; },
    withdraw(n) {
      if (n > balance) throw new Error("Insufficient funds");
      balance -= n; return balance;
    },
    getBalance() { return balance; }
  };
}
const acc = createAccount(100);
acc.deposit(50);    // 150
acc.getBalance();   // 150
acc.balance;        // undefined — genuinely inaccessible; only the closures can reach it

Because balance lives in createAccount's scope and is reachable only through the returned methods' closures, it's genuinely private — there's no property to poke, no convention to violate. This is the module pattern: expose a public API of methods that all close over shared private state. It predates #private class fields (Classes lesson) and is still the idiomatic way to build encapsulated units without classes — every jQuery plugin, many npm modules, and countless singletons are built this way.

Part 5 — Pattern: Factory Functions & Memoisation

// Factory: each call bakes in captured configuration
function multiplier(factor) {
  return (n) => n * factor;              // factor captured, fixed per factory call
}
const double = multiplier(2), triple = multiplier(3);
double(5);  // 10    triple(5);  // 15

// Memoisation: a private cache lives in the closure
function memoize(fn) {
  const cache = new Map();               // survives across calls, invisible outside
  return (arg) => {
    if (cache.has(arg)) return cache.get(arg);
    const result = fn(arg);
    cache.set(arg, result);
    return result;
  };
}
const fib = memoize(n => n < 2 ? n : fib(n - 1) + fib(n - 2));
fib(40);   // fast — each subproblem computed once, cached in the closure

Both patterns lean on the same mechanic: a value (a factor, a cache) captured in a closure and kept alive between calls. Factory functions produce specialised functions with configuration baked in; memoisation keeps a private result cache. Neither needs a class or external state — the closure is the storage.

Part 6 — Closures in Event Handlers & Async

function setupButton(label) {
  let clicks = 0;                        // per-button private state
  const btn = document.querySelector(`#${label}`);
  btn.addEventListener("click", () => {  // the handler closes over clicks AND label
    clicks++;
    console.log(`${label} clicked ${clicks} times`);
  });
}

Every event handler and async callback is a closure — it captures the variables in scope where it was defined, and those stay alive as long as the handler is registered. This is why an inline addEventListener callback can "remember" a clicks counter or the label that identifies which element it's for, with no extra plumbing. It's also the mechanism behind React hooks (useState's setter closes over the component's state slot), every setTimeout callback, and every promise .then handler — closures are everywhere in async code, mostly invisibly.

Part 7 — The Flip Side: Closures and Memory Leaks

function attach() {
  const hugeData = new Array(1_000_000).fill("x");   // big
  const handler = () => console.log(hugeData.length); // closes over hugeData
  document.querySelector("#btn").addEventListener("click", handler);
  // As long as the listener is registered, hugeData CANNOT be garbage-collected.
  // Forget to removeEventListener when the element is gone → a leak.
}

The same "keep the birth scope alive" behaviour that makes closures powerful can retain memory you didn't intend. A closure holds a reference to every variable in its enclosing scope that it (or the engine conservatively) might use, so a long-lived closure (an event listener, an interval callback, a subscription) pins that memory until the closure itself is released. The discipline: tear down what you set up — removeEventListener, clearInterval, unsubscribe — so the closure (and everything it captured) becomes collectable. The Memory Management lesson covers detection and the full leak taxonomy; know here that closures are a leading cause.

Part 8 — Build It: a Stateful Rate Limiter with a Reset

Let's assemble private state, a factory, and a returned API into a small throttle-with-status utility — the kind of thing closures make trivial and classes would over-formalise.

function createLimiter(maxPerWindow, windowMs) {
  let count = 0;                          // private counters, shared by the methods below
  let windowStart = Date.now();

  function tryConsume() {
    const now = Date.now();
    if (now - windowStart >= windowMs) {  // window elapsed → reset
      count = 0;
      windowStart = now;
    }
    if (count < maxPerWindow) { count++; return true; }
    return false;                          // over the limit for this window
  }

  return {
    tryConsume,
    remaining: () => Math.max(0, maxPerWindow - count),
    reset: () => { count = 0; windowStart = Date.now(); }
  };
}

const limiter = createLimiter(3, 1000);
limiter.tryConsume();  // true
limiter.tryConsume();  // true
limiter.tryConsume();  // true
limiter.tryConsume();  // false — 3 used this window
limiter.remaining();   // 0

count and windowStart are private, shared by three methods that all close over them — a self-contained stateful object with no class, no this, and no way for outside code to corrupt its internals. This is closures doing exactly what they're best at: encapsulating mutable state behind a clean API. Real rate limiters (server-side, Redis-backed) scale this idea up, but the core is this closure.

Interview Questions

Q1 (beginner): What is a closure?

A closure is a function together with the lexical environment (the set of variables) it was created in. Concretely: when a function is defined inside another function and references the outer function's variables, it "closes over" them — those variables stay alive and accessible to the inner function even after the outer function has returned. It's a direct consequence of lexical scoping plus functions being first-class values you can return or store. Closures are how JavaScript gives functions persistent private state.

Q2 (beginner): Do closures capture the value of a variable or the variable itself?

The variable itself — a live reference, not a snapshot. So if the captured variable changes after the closure is created, the closure sees the new value when it runs. That's why a counter closure (() => ++count) increments across calls: every call reads and mutates the same count. It's also why the classic var loop bug happens — all closures share the one loop variable and read its final value.

Q3 (mid): Explain the var loop-closure bug and give two fixes.

for (var i...) { setTimeout(() => console.log(i)) } logs the final value N times because var i is a single function-scoped variable shared by every iteration; all the deferred closures capture that one variable, and by the time they run the loop has finished with i at its terminal value. Fix 1: use let i, which creates a fresh block-scoped binding per iteration, so each closure captures its own i with that iteration's value. Fix 2 (pre-ES6): wrap the body in an IIFE that takes i as a parameter — ((j) => setTimeout(() => console.log(j)))(i) — manufacturing a fresh scope with a copied variable per iteration. Both work by ensuring each closure captures a distinct variable rather than sharing one.

Q4 (mid): Show how closures implement private state, and why that's useful.

A factory function declares variables in its scope and returns an object of methods that reference them: function makeCounter(){ let c = 0; return { inc: () => ++c, get: () => c }; }. The variable c is reachable only through the returned methods' closures — there's no property on the returned object exposing it, so it's genuinely private and can't be read or corrupted by outside code, only mutated through the controlled API. This is useful for encapsulation: enforcing invariants (a balance that can't go negative), hiding implementation details, and preventing accidental external mutation — all without classes. It's the "module pattern," and it predates and complements private class fields.

Q5 (senior): How can closures cause memory leaks, and how do you prevent them?

A closure retains a reference to the variables of its enclosing scope for as long as the closure itself is reachable. So a long-lived closure — an event listener, an interval/timeout callback, a subscription handler, a cache of functions — keeps everything it captured alive, including large objects or DOM nodes, even after they're logically no longer needed. Classic leak: attaching a listener whose closure references a big data structure (or a DOM element) and never removing the listener, so neither the data nor the element can be garbage-collected. Prevention: (1) tear down what you set up — removeEventListener, clearInterval, unsubscribe — so the closure becomes unreachable and collectable; (2) avoid capturing more than you need (don't close over a huge object to use one field — extract the field first); (3) null out large captured references when done if the closure must outlive them; (4) for caches keyed by objects, use WeakMap/WeakRef so entries don't pin their keys. Tooling: the Chrome DevTools memory profiler's heap snapshots and "detached DOM node" detection surface exactly these closure-retention leaks — the Memory lesson covers the workflow.

Q6 (senior): Compare closure-based private state with class private fields (#x) in terms of memory, performance, and encapsulation guarantees.

Encapsulation: both are genuinely private — closure variables are unreachable outside the closure, and #fields throw a SyntaxError if accessed from outside the class body (true hard privacy, unlike the old _convention). Closures hide even the existence of the state; #fields are private but the field names are discoverable in the class source. Memory/performance: the key difference. With the closure/module pattern, each instance created by the factory gets its own copy of every method (each method is a fresh closure over that instance's variables), so N instances means N×M method function objects — cheap for a handful of instances, wasteful for thousands. With classes, methods live once on the shared prototype and every instance references the same function objects, so memory scales with instance count only for data, not methods — much better for many instances (the Prototypes lesson explains the sharing). Modern engines optimise both well, but the prototype approach has a structural memory edge at scale and slightly better shape-based optimisation. Practical guidance: for a small number of long-lived encapsulated units (a config manager, a singleton service, a handful of stateful widgets), the closure/module pattern is clean and idiomatic. For value types instantiated in large numbers (thousands of model objects, geometry/points, DOM-backed components), prefer a class with #private fields so methods are shared via the prototype. They're complementary tools, and the deciding factor is usually instance count and whether you want this-based method sharing or fully hidden state.

🏋️ Exercises

1 (easy): Counter factory. Write makeCounter() returning an object with inc(), dec(), and value(), sharing a private count. Prove two counters are independent.
Solution 
function makeCounter() {
  let count = 0;
  return { inc: () => ++count, dec: () => --count, value: () => count };
}
const a = makeCounter(), b = makeCounter();
a.inc(); a.inc(); b.inc();
a.value();  // 2   b.value();  // 1  — separate closures, separate count

Each makeCounter() call creates a fresh scope with its own count, so a and b close over independent variables.

2 (easy): Predict the loop output. What does each log, and why? 
for (var i = 0; i < 3; i++) setTimeout(() => console.log("var", i));
for (let j = 0; j < 3; j++) setTimeout(() => console.log("let", j));
Answers

The var loop logs var 3, var 3, var 3 — one shared i, read as 3 after the loop. The let loop logs let 0, let 1, let 2 — a fresh j per iteration, each closure capturing its own.

3 (medium): once() via closure. Implement once(fn) (runs fn at most once, caches result) and explain which variable in the closure enforces the "once."
Solution 
function once(fn) {
  let done = false, cached;
  return (...args) => {
    if (!done) { done = true; cached = fn(...args); }
    return cached;
  };
}

The done flag, captured in the closure and persisting across calls, is what enforces "once" — after the first call it stays true, so the guarded block never re-runs and the cached result is returned.

4 (medium): Private toggle. Build createToggle(initial) returning { toggle, isOn } where the boolean state is private and can only be flipped via toggle().
Solution 
function createToggle(initial = false) {
  let on = initial;
  return { toggle: () => (on = !on), isOn: () => on };
}
const t = createToggle();
t.isOn();    // false
t.toggle();  // true
t.isOn();    // true

on is inaccessible except through the two methods; there's no way to set it to an arbitrary value from outside, only to flip it — encapsulation enforcing the invariant that it's always a proper toggle.

5 (hard): Debounce. Implement debounce(fn, delay) using a closure to hold the pending timer id, so fn only runs after delayms of no further calls. Explain what the closure retains.
Solution 
function debounce(fn, delay) {
  let timer;                              // retained in the closure across calls
  return (...args) => {
    clearTimeout(timer);                  // cancel the previous pending run
    timer = setTimeout(() => fn(...args), delay);
  };
}
const search = debounce(q => console.log("search", q), 300);
search("a"); search("ab"); search("abc");   // only "abc" runs, 300ms after the last call

The closure retains timer, the id of the currently-pending timeout, so each new call can cancel the previous one and reschedule. Without the closure there'd be nowhere to keep that id between invocations. This is a core Performance-lesson utility, built entirely on closure state.

6 (hard): Leak and fix. The following leaks. Identify what's retained and rewrite it so the large buffer can be collected once processing finishes. 
function process() {
  const buffer = new Array(1e6).fill(0);
  const size = buffer.length;
  document.body.addEventListener("click", () => console.log(buffer.length));
  return () => size;
}
Solution

The click listener closes over buffer and is never removed, so the million-element array is pinned for the life of the page. The returned function only needs size, a number — it doesn't need buffer at all. Fix: don't capture the big array in a long-lived closure; capture only the primitive you need, and remove the listener when done:

function process() {
  const buffer = new Array(1e6).fill(0);
  const size = buffer.length;             // extract the primitive we actually keep
  const handler = () => console.log(size); // closes over `size`, NOT buffer
  document.body.addEventListener("click", handler);
  // buffer is now unreferenced after this function returns → collectable
  return {
    getSize: () => size,
    dispose: () => document.body.removeEventListener("click", handler)
  };
}

By closing over size (a number) instead of buffer, and exposing a dispose to unregister the listener, nothing long-lived references the big array, so the garbage collector reclaims it. "Capture only what you need, and tear down what you set up" is the whole discipline.

🎯 Key Takeaways

  • A closure is a function plus the live variables it captured from its birth scope — kept alive as long as the function lives.
  • Closures capture variables, not value snapshots — they read the current value at call time, which powers stateful counters and causes the var loop bug.
  • Fix the loop bug with let (fresh binding per iteration); the IIFE trick is the pre-ES6 equivalent.
  • Closures give private state (module pattern), factory functions with baked-in config, and memoisation caches — encapsulation without classes.
  • Every event handler, timer, and promise callback is a closure — it's the invisible mechanism behind most async and React hooks.
  • Long-lived closures retain everything they capture — a leading cause of memory leaks; capture only what you need and tear down listeners/intervals.
