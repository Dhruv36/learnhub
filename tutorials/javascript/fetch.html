Fetch, HTTP & AbortController Browser & the Web Platform

What you'll master: the fetch API for HTTP, the critical gotcha that it doesn't reject on error status, request/response anatomy (methods, headers, bodies, JSON), cancellation and timeouts with AbortController, CORS at a working level, and a production-grade request wrapper. Prerequisites: Promises, async/await, and Promise Combinators (AbortController). This is how JavaScript talks to servers — the backbone of every real app.

Part 1 — The One Gotcha That Trips Everyone

const res = await fetch("/api/users/999");   // returns a Response, resolving on any HTTP RESPONSE
// ⚠ fetch only REJECTS on a NETWORK failure (offline, DNS, CORS block) — NOT on HTTP error status!
// A 404 or 500 still FULFILS the promise. You MUST check res.ok yourself:
if (!res.ok) throw new Error(`HTTP ${res.status}`);   // res.ok is true only for 200–299
const data = await res.json();

Learn this first because it causes the most bugs: fetch rejects only when the request can't complete at all (no network, DNS failure, CORS block) — an HTTP 404, 500, or any error status fulfils the promise with a Response object. So a try/catch around await fetch() will not catch a 404; the "success" path runs with an error response. You must explicitly check response.ok (true only for 2xx) or response.status and throw yourself. Every robust fetch call has this check; forgetting it means error responses silently flow downstream as if successful.

Part 2 — Reading the Response Body

const res = await fetch("/api/data");
await res.json();        // parse body as JSON → object (the common case)
await res.text();        // body as a string (HTML, plain text, CSV)
await res.blob();        // binary (images, files) → Blob
await res.arrayBuffer(); // raw bytes
await res.formData();    // multipart form data

// The body is a STREAM — you can read it ONCE:
const data = await res.json();
await res.json();        // ❌ TypeError: body already consumed — clone first if you need it twice:
const res2 = res.clone();

The response body arrives as a stream and you pick how to read it: .json() (the usual choice — parses JSON to an object), .text(), .blob() (binary/files), .arrayBuffer(), or .formData(). Each returns a promise. The key constraint: the body can only be consumed once — calling .json() then .text() on the same response throws, because the stream is exhausted. If you need to read it twice (e.g. try JSON, fall back to text on error), res.clone() the response first.

Part 3 — Making Requests: Methods, Headers, Body

// GET is the default. For anything else, pass an options object:
const res = await fetch("/api/users", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",       // tell the server what you're SENDING
    "Authorization": `Bearer ${token}`,       // auth
    "Accept": "application/json",             // what you want BACK
  },
  body: JSON.stringify({ name: "Ada", age: 30 }),  // stringify objects; fetch won't do it for you
});

// Other body types (don't set Content-Type manually for FormData — the browser sets the boundary):
fetch(url, { method: "POST", body: new FormData(form) });   // file uploads / multipart
fetch(url, { method: "POST", body: new URLSearchParams({ q: "x" }) });  // form-urlencoded

For non-GET requests, pass an options object with method, headers, and body. Two things people forget: you must JSON.stringify a JSON body yourself (fetch sends the object as-is otherwise) and set Content-Type: application/json so the server parses it correctly. Conversely, for FormData (file uploads), do NOT set Content-Type — the browser sets it including the required multipart boundary; setting it manually breaks the upload. The HTTP method conveys intent: GET (read), POST (create), PUT/PATCH (update), DELETE (remove) — matching REST conventions the server expects.

Part 4 — Cancellation & Timeouts with AbortController

// fetch has NO built-in timeout — you add one with AbortController (Combinators lesson):
const controller = new AbortController();
const timer = setTimeout(() => controller.abort(), 5000);   // abort after 5s
try {
  const res = await fetch(url, { signal: controller.signal });
  clearTimeout(timer);
  return await res.json();
} catch (err) {
  if (err.name === "AbortError") throw new Error("request timed out");
  throw err;
}

// Shortcut for pure timeout:
fetch(url, { signal: AbortSignal.timeout(5000) });   // auto-aborts after 5s

// Cancel on demand (e.g. user navigates away, new search supersedes old):
controller.abort();   // the in-flight fetch rejects with AbortError — and the request is actually stopped

fetch has no built-in timeout — a hung request waits forever unless you cancel it. AbortController is the tool: pass its signal to fetch, and calling abort() (via a timer or on demand) rejects the fetch with an AbortError and actually cancels the network request (saving bandwidth). This is essential for timeouts, for cancelling superseded requests (debounced search — abort the previous query), and for cleanup on component unmount. AbortSignal.timeout(ms) is the built-in shortcut for the timeout case. Always distinguish AbortError from real errors in your catch, since a cancellation isn't a failure to report.

Part 5 — CORS: Why Your Cross-Origin Request Failed

// Same-origin: same scheme + host + port. Anything else is CROSS-origin.
fetch("https://api.other.com/data")   // may be blocked by CORS — the SERVER must opt in

// The browser enforces CORS. The SERVER must send:
//   Access-Control-Allow-Origin: https://yoursite.com  (or *)
// ...or the browser blocks your JS from reading the response (a security feature, not a bug in your code).

// Credentials (cookies) on cross-origin requests need explicit opt-in on BOTH sides:
fetch(url, { credentials: "include" });   // send cookies; server must also allow credentials

CORS (Cross-Origin Resource Sharing) is a browser security mechanism: by default, JS can't read responses from a different origin (scheme+host+port) unless the server explicitly allows it via Access-Control-Allow-Origin headers. A CORS failure is the server's configuration, not your code — you can't fix it purely client-side (a proxy or the server adding the header is the fix). For "unsafe" methods/headers the browser first sends a preflight OPTIONS request to check permission. Sending cookies cross-origin requires credentials: "include" on the request and matching server headers. Knowing CORS is browser-enforced and server-configured saves hours of debugging "why does it work in Postman but not the browser?" (Postman isn't a browser, so no CORS).

Part 6 — Handling Errors Properly

async function getJSON(url) {
  let res;
  try {
    res = await fetch(url);                     // rejects only on NETWORK failure
  } catch (networkErr) {
    throw new Error(`Network error: ${networkErr.message}`);   // offline, DNS, CORS
  }
  if (!res.ok) {                                 // HTTP error status — fetch didn't reject
    const body = await res.text().catch(() => "");
    throw new Error(`HTTP ${res.status} ${res.statusText}: ${body.slice(0, 200)}`);
  }
  try {
    return await res.json();                     // may throw if body isn't valid JSON
  } catch {
    throw new Error("Invalid JSON in response");
  }
}

Robust fetch error handling has three distinct failure points, each needing its own branch: (1) the fetch itself rejecting (network/DNS/CORS — caught by try/catch around fetch); (2) an HTTP error status (checked via !res.ok — fetch doesn't throw here); and (3) body parsing failing (invalid JSON). Naive code that just does await fetch().then(r => r.json()) conflates or misses these — a 500 with an HTML error page will throw a confusing JSON parse error instead of a clear "HTTP 500." Handle each explicitly and your errors become actionable. This three-way split is the skeleton of the production wrapper next.

Part 7 — Build It: a Production apiClient

Let's assemble everything — the res.ok check, JSON handling, auth headers, timeout via abort, and typed errors — into one reusable client, the kind every real app has.

class HttpError extends Error {                    // typed error carrying status + body (Error Handling lesson)
  constructor(status, statusText, body) {
    super(`HTTP ${status} ${statusText}`);
    this.name = "HttpError"; this.status = status; this.body = body;
  }
}

async function apiClient(path, { method = "GET", body, headers = {}, timeout = 10000, token } = {}) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), timeout);
  try {
    const res = await fetch(`/api${path}`, {
      method,
      signal: controller.signal,
      headers: {
        "Accept": "application/json",
        ...(body ? { "Content-Type": "application/json" } : {}),   // only when sending a body
        ...(token ? { "Authorization": `Bearer ${token}` } : {}),
        ...headers,
      },
      body: body ? JSON.stringify(body) : undefined,
    });

    if (!res.ok) {
      const errBody = await res.json().catch(() => res.statusText);  // best-effort error detail
      throw new HttpError(res.status, res.statusText, errBody);
    }
    if (res.status === 204) return null;          // 204 No Content — no body to parse
    return await res.json();
  } catch (err) {
    if (err.name === "AbortError") throw new Error(`Request timed out after ${timeout}ms`);
    throw err;                                     // network error or HttpError — propagate
  } finally {
    clearTimeout(timer);                           // always clear the abort timer
  }
}

// Clean call sites:
const user = await apiClient("/users/1", { token });
const created = await apiClient("/users", { method: "POST", body: { name: "Ada" }, token });

This one function encapsulates every lesson concept: it checks res.ok and throws a typed HttpError carrying the status and parsed body (so callers can branch on err.status === 401); it conditionally sets Content-Type/auth headers; it adds a timeout that actually cancels via AbortController, cleaned up in finally; it handles the 204 No Content edge case; and it distinguishes timeouts from other errors. Call sites become trivial and consistent. Real apps wrap this further (retries, refresh-token on 401, response typing with TypeScript), but this is the correct core — and notice how much of "using fetch well" is error handling, not the happy path.

Interview Questions

Q1 (beginner): Does fetch reject on a 404 or 500?

No. fetch rejects its promise only when the request fails at the network level — no connection, DNS failure, or a CORS block preventing a response. Any actual HTTP response, including error statuses like 404 and 500, fulfils the promise with a Response object. So you must check response.ok (true only for 2xx) or response.status yourself and throw if it's an error status — a try/catch around the fetch alone won't catch HTTP errors. This is the most common fetch mistake.

Q2 (beginner): How do you send JSON in a POST request with fetch?

Pass an options object with method: "POST", a Content-Type: application/json header, and a body that you manually stringify: body: JSON.stringify(data). Fetch does not serialise objects for you, so passing the object directly sends the wrong thing; and the server needs the Content-Type header to know to parse it as JSON. On the response, call await res.json() to parse the reply (after checking res.ok). Note that for FormData bodies you do the opposite — don't set Content-Type, since the browser must set it with the multipart boundary.

Q3 (mid): How do you add a timeout to a fetch, and why doesn't fetch have one built in?

Use AbortController: create one, pass its signal to fetch(url, { signal }), and start a setTimeout that calls controller.abort() after the timeout; abort rejects the fetch with an AbortError and cancels the actual request. Clear the timer when the request completes. The shortcut is fetch(url, { signal: AbortSignal.timeout(ms) }). Fetch has no built-in timeout largely for historical/spec reasons — it predates a standard cancellation mechanism, and AbortController was later designed as the general-purpose way to cancel not just fetch but any abortable async operation. So timeouts are composed from the general cancellation primitive rather than being a fetch-specific option. In the catch, distinguish AbortError (your timeout/cancel) from genuine network errors.

Q4 (mid): What is CORS and why can't you fix a CORS error in client code?

CORS (Cross-Origin Resource Sharing) is a browser-enforced security policy: JavaScript on one origin (scheme+host+port) cannot read responses from a different origin unless the server explicitly permits it by sending Access-Control-Allow-Origin (and related) headers. For non-simple requests the browser first sends a preflight OPTIONS request to check permission. Because the browser enforces the block based on server headers, you can't fix a CORS error from client JavaScript — no fetch option grants access the server hasn't allowed. The real fixes are server-side: configure the API to send the appropriate CORS headers for your origin, or route through a same-origin proxy/backend that makes the request server-to-server (where CORS doesn't apply). This is why a request works in Postman/curl (not browsers, so no CORS enforcement) but fails in the browser.

Q5 (senior): Design a resilient fetch wrapper with retries, timeout, and auth-token refresh. What are the correctness concerns?

Core structure: a function that, per attempt, sets up an AbortController timeout, performs the fetch, checks res.ok, parses the body, and on failure decides whether to retry. Layered concerns: (1) Retry only safe, transient failures — retry on network errors, timeouts (your AbortError), and 5xx/429; do not retry 4xx (a 400/401/404 won't change on retry) except 401 which triggers token refresh (below); and only retry idempotent methods (GET/PUT/DELETE) or requests carrying an idempotency key, since retrying a non-idempotent POST can duplicate side effects. (2) Backoff with jitter — exponential delays with random jitter to avoid thundering-herd retries, and honour a Retry-After header on 429/503. (3) Timeout per attempt, cap total — each attempt gets its own abort timeout, and cap overall elapsed time/attempt count. (4) Token refresh on 401 — on a 401, call a single-flight refresh (dedupe concurrent refreshes so ten parallel 401s trigger one refresh, not ten — a shared in-progress promise), then retry the original request once with the new token; if refresh fails, propagate an auth error and redirect to login. (5) Distinguish your abort from user abort — a caller-supplied signal that aborts should propagate immediately, not be retried, whereas your internal timeout abort should be retried. (6) Typed errors — throw an HttpError carrying status/body so callers branch precisely. (7) Clear timers in finally. The subtle correctness lives in: which errors are retryable, idempotency safety, single-flight token refresh (avoiding a refresh storm), separating internal vs external cancellation, and honouring server backpressure. In production you'd often use a library (or the platform's built-in retry policy) that encodes these, but knowing the concerns is what lets you configure/review it correctly — "add retries and a timeout" is deceptively subtle precisely because a naive implementation duplicates POSTs, hammers a struggling server, or triggers a refresh stampede.

Q6 (senior): Compare fetch with XMLHttpRequest and discuss when you'd reach for streaming responses.

fetch is the modern, promise-based HTTP API; XMLHttpRequest (XHR) is the older event/callback-based one. Fetch advantages: promise-based (composes with async/await and combinators), cleaner API, request/response stream support, and a proper Request/ Response/Headers object model. XHR still has a couple of things fetch historically lacked: native upload/download progress events (fetch can track download progress via the response body stream but upload progress remains awkward), and it predates AbortController (though fetch's cancellation via AbortController is now cleaner). For most work fetch is the default; XHR survives mainly for upload progress bars and legacy code. Streaming: fetch exposes the response body as a ReadableStream (res.body), so instead of awaiting the whole payload with .json()/.text(), you can read it chunk by chunk via a reader (res.body.getReader()) or for await...of over it. Reach for streaming when: the response is large and you want to process/render it incrementally without buffering it all in memory (a huge JSON array, a log tail); you're consuming a server-sent stream or AI token stream and want to show output as it arrives (the classic use — rendering LLM tokens progressively); you want to compute download progress (bytes received vs Content-Length); or you're piping the response into another stream (decompression, transformation) without materialising it. Streaming trades the simplicity of "await the whole thing" for bounded memory and progressive processing — worth it specifically for large or time-extended responses, unnecessary for typical small JSON API calls where .json() is simpler and fine. The senior point is matching the consumption model to the payload: buffer small responses, stream large or incremental ones, and use the streaming API (or XHR) when you need progress feedback.

🏋️ Exercises

1 (easy): Safe GET. Write getJSON(url) that fetches, checks the status, and returns parsed JSON — throwing a clear error on a non-2xx response.
Solution 
async function getJSON(url) {
  const res = await fetch(url);
  if (!res.ok) throw new Error(`HTTP ${res.status} ${res.statusText}`);
  return res.json();
}

The !res.ok check is the essential line — without it, a 404/500 would fall through to res.json() and either succeed with an error payload or throw a confusing parse error.

2 (easy): POST JSON. Send { name: "Ada" } to /api/users as JSON and return the created object.
Solution 
async function createUser(user) {
  const res = await fetch("/api/users", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(user),
  });
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

Stringify the body, set the JSON Content-Type, check res.ok, parse the response — the four essentials of a JSON POST.

3 (medium): Timeout. Add a 3-second timeout to a fetch using AbortController, throwing a distinct "timed out" error.
Solution 
async function fetchWithTimeout(url, ms = 3000) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), ms);
  try {
    const res = await fetch(url, { signal: controller.signal });
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.json();
  } catch (err) {
    if (err.name === "AbortError") throw new Error(`Timed out after ${ms}ms`);
    throw err;
  } finally {
    clearTimeout(timer);
  }
}

The abort timer cancels the request after ms; catching AbortError converts it into a clear timeout message, and finally clears the timer regardless of outcome.

4 (medium): Handle 204. Write a del(url) for DELETE requests where a successful response is 204 No Content (no body to parse).
Solution 
async function del(url) {
  const res = await fetch(url, { method: "DELETE" });
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.status === 204 ? null : res.json();   // 204 has no body — don't parse it
}

Calling res.json() on a 204 (empty body) throws, so you must special-case it and return null — a common real-world gotcha with DELETE/PUT endpoints.

5 (hard): Parallel with partial failure. Fetch three endpoints concurrently and return an object mapping each to its data or an error, without one failure aborting the others.
Solution 
async function fetchAll(urls) {
  const results = await Promise.allSettled(urls.map(async (url) => {
    const res = await fetch(url);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return res.json();
  }));
  return urls.reduce((acc, url, i) => {
    const r = results[i];
    acc[url] = r.status === "fulfilled" ? { data: r.value } : { error: r.reason.message };
    return acc;
  }, {});
}

Promise.allSettled (Combinators lesson) lets every request complete regardless of individual failures; we then map each URL to its data or error — the partial-success pattern for dashboards where one dead endpoint shouldn't blank everything.

6 (hard): Stream a response. Read a large text response incrementally via the response body stream, logging progress as bytes arrive.
Solution 
async function streamText(url, onChunk) {
  const res = await fetch(url);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  const reader = res.body.getReader();
  const decoder = new TextDecoder();
  let received = 0, text = "";
  while (true) {
    const { done, value } = await reader.read();   // pull one chunk
    if (done) break;
    received += value.length;
    const chunk = decoder.decode(value, { stream: true });
    text += chunk;
    onChunk(chunk, received);                        // progressive processing
  }
  return text;
}

Instead of buffering the whole body with .text(), we read the ReadableStream chunk by chunk, decoding and processing each as it arrives — bounded memory and progressive output (the pattern behind streaming AI token rendering and progress bars).

🎯 Key Takeaways

  • fetch rejects only on network failure — 404/500 still fulfil; always check response.ok and throw yourself.
  • Read the body once with .json()/.text()/.blob() (clone if you need it twice); stringify JSON bodies and set Content-Type (but not for FormData).
  • HTTP status ranges guide handling: 2xx success, 4xx client (don't blind-retry), 5xx server (retryable with backoff).
  • fetch has no built-in timeout — use AbortController/AbortSignal.timeout to time out and cancel, and treat AbortError separately.
  • CORS is browser-enforced and server-configured — you can't fix it in client code; the server must send the headers or you proxy the request.
  • Robust fetch handles three failure points (network reject, HTTP status, body parse); a typed-error client wrapper makes call sites clean and errors actionable.
