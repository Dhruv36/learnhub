Testing JavaScript Production JavaScript

What you'll master: the testing skill set every JS job assumes — writing clear unit tests with Vitest/Jest, the assertion vocabulary, testing async code correctly, test doubles (spies, stubs, mocks) and the discipline of mocking only at boundaries, fake timers, the testing pyramid/trophy and what each layer buys, and how seniors judge test quality (behavior over implementation, coverage as a lie detector — not a target). Prerequisites: Functions, async/await, Modules, and Error Handling — pure functions from Functional Patterns are the easiest things you'll ever test, and that's not a coincidence.

Part 1 — Anatomy of a Test: AAA and the Runner

// price.test.js — Vitest (Jest is API-identical for everything in this lesson)
import { describe, it, expect } from "vitest";
import { priceWithTax } from "./price.js";

describe("priceWithTax", () => {                    // a suite: groups related tests
  it("applies the rate to the base price", () => {  // one test = one behavior, named as a SENTENCE
    // Arrange — set up inputs
    const base = 100, rate = 0.2;
    // Act — do the one thing under test
    const total = priceWithTax(base, rate);
    // Assert — check the outcome
    expect(total).toBe(120);
  });

  it("throws on negative prices", () => {
    expect(() => priceWithTax(-5, 0.2)).toThrow(RangeError);   // pass a FUNCTION, not the call
  });
});
// run: npx vitest        (watch mode: re-runs affected tests on save)

Every good test has three visible phases — Arrange, Act, Assert — and tests one behavior, named so a failure reads like a bug report: "priceWithTax › throws on negative prices" tells you what broke without opening the file. The runner (Vitest or Jest — Vitest is the modern default: native ESM, fast, Jest-compatible API) discovers *.test.js files, runs each test in isolation, and reports failures with diffs. Note the throw-assertion form: you pass a function for expect to call — expect(priceWithTax(-5, 0.2)) would throw before expect ever ran.

Part 2 — The Assertion Vocabulary (and the One Distinction That Matters)

expect(2 + 2).toBe(4);                          // toBe = Object.is → IDENTITY (===-like)
expect({ a: 1 }).toEqual({ a: 1 });             // toEqual = deep STRUCTURAL equality
expect({ a: 1 }).not.toBe({ a: 1 });            // different objects — toBe would FAIL

expect(0.1 + 0.2).toBeCloseTo(0.3);             // floats: NEVER toBe (0.30000000000000004)
expect(user.name).toMatch(/^ada/i);             // regex on strings
expect(list).toContain("apple");                // membership
expect(fn).toThrow("out of range");             // message substring or Error class
expect(value).toBeNull();  toBeUndefined();  toBeDefined();

// Partial matching — assert what MATTERS, ignore the rest:
expect(response).toEqual(expect.objectContaining({
  id: expect.any(Number),                        // "some number" — don't pin generated IDs
  status: "active",
}));

The distinction juniors trip on: toBe is identity, toEqual is structure. Two separately-built objects with the same shape are toEqual but never toBe. Use toBe for primitives and "is literally the same object" checks; toEqual for data. Just as important: assert precisely what the behavior promises, and no more. Pinning an entire response object byte-for-byte makes the test fail when an unrelated field is added — objectContaining/expect.any keep tests focused on the contract, which is what makes them survive refactors.

Part 3 — Testing Async Code

// The golden rule: the test function must AWAIT (or return) the promise.
it("loads the user", async () => {
  const user = await loadUser(42);
  expect(user.name).toBe("Ada");
});

// Rejections — resolves/rejects matchers, awaited:
it("rejects for unknown ids", async () => {
  await expect(loadUser(-1)).rejects.toThrow("not found");
});
it("resolves with a role", async () => {
  await expect(loadUser(42)).resolves.toHaveProperty("role");
});

// ❌ THE classic false positive — forgetting await:
it("looks like it tests something", () => {
  expect(loadUser(-1)).rejects.toThrow("not found");   // promise floats; test ends first
});                                                    // → PASSES even if loadUser is broken
// ❌ try/catch variant that passes when nothing throws:
it("also lies", async () => {
  try { await loadUser(-1); }               // if it wrongly RESOLVES, catch never runs,
  catch (e) { expect(e.message).toMatch("not found"); }   // zero assertions → green ✓
});                                          // fix: expect.assertions(1) — or use .rejects

Async tests fail quietly when written wrong — they don't error, they pass without testing anything, which is worse. The runner needs the promise: async test + await every async assertion (including await expect(...).rejects... — that matcher returns a promise too). For the try/catch style, expect.assertions(n) makes the test fail if fewer than n assertions actually ran — turning the silent nothing-thrown path into a loud failure. When a test intermittently times out instead, you have a hanging promise — usually a mocked dependency that never resolves.

Part 4 — Test Doubles: Spies, Stubs, Mocks — at Boundaries Only

import { vi } from "vitest";   // Jest: jest.fn / jest.spyOn / jest.mock — same shapes

// SPY/STUB in one: vi.fn() records calls AND can fake a return:
const sendEmail = vi.fn().mockResolvedValue({ ok: true });
await registerUser({ email: "a@b.com" }, { sendEmail });   // dependency INJECTED → easy to swap
expect(sendEmail).toHaveBeenCalledTimes(1);
expect(sendEmail).toHaveBeenCalledWith(expect.objectContaining({ to: "a@b.com" }));

// SPY on an existing object — wraps the real method, records calls:
const spy = vi.spyOn(console, "error").mockImplementation(() => {});  // also silences it
expect(spy).toHaveBeenCalled();
spy.mockRestore();                                   // ALWAYS restore globals

// MODULE mock — replace an entire import for every consumer:
vi.mock("./api.js", () => ({
  fetchUser: vi.fn().mockResolvedValue({ id: 1, name: "Ada" }),
}));

// Sequencing per-call behavior:
fetchUser.mockResolvedValueOnce({ id: 1 }).mockRejectedValueOnce(new Error("down"));

afterEach(() => vi.restoreAllMocks());               // hygiene: no bleed between tests

Vocabulary: a spy records how it was called; a stub returns canned answers; a mock commonly means both (and vi.fn() is all of them). The craft is not the API — it's where you mock: at process boundaries only. Network, database, clock, randomness, filesystem, third-party SDKs: mock these, because they're slow, nondeterministic, or expensive. Your own modules: don't, or your test freezes the internal call graph — every refactor breaks tests while real bugs (in the now-mocked collaborator) slip through. A test suite where everything is mocked tests nothing but your mocks. Notice also how dependency injection (passing sendEmail in) made the first example trivially testable with no module-mock machinery — testability is a design property first, a tooling problem second.

Part 5 — Fake Timers, Setup Hooks & the DOM

// FAKE TIMERS — test time without waiting through it:
vi.useFakeTimers();
it("debounce fires once after the pause", () => {
  const fn = vi.fn();
  const debounced = debounce(fn, 300);
  debounced(); debounced(); debounced();
  vi.advanceTimersByTime(299);  expect(fn).not.toHaveBeenCalled();
  vi.advanceTimersByTime(1);    expect(fn).toHaveBeenCalledTimes(1);   // instant, deterministic
});
vi.useRealTimers();
// also: vi.setSystemTime(new Date("2026-01-01")) — freeze "now" for date logic

// SETUP/TEARDOWN hooks — shared arrange, per-test isolation:
beforeEach(() => { db = makeFreshFakeDb(); });    // every test gets CLEAN state
afterEach(() => vi.restoreAllMocks());
// beforeAll/afterAll: once per suite — only for expensive, READ-ONLY setup

// DOM component testing (Testing Library, any framework flavor):
render();
await user.click(screen.getByRole("button", { name: /increment/i }));
expect(screen.getByText("Count: 1")).toBeInTheDocument();
// query by ROLE/LABEL/TEXT — what users perceive — never by CSS class or test-id-first

Anything time-based — debounce, retries with backoff, cache TTLs, token expiry — is untestable with real clocks (slow at best, flaky at worst). Fake timers make time a controllable input: advance it precisely, assert between ticks. Hooks keep tests independent — every test must pass alone and in any order, which means fresh state in beforeEach, never accumulated state a previous test happened to leave behind (the root cause of "passes alone, fails in CI"). For UI, Testing Library's philosophy is the behavior-testing principle applied to the DOM: query by role and accessible name (as a user would find things), so tests survive markup refactors and simultaneously enforce accessibility.

Part 6 — Deep Dive: Strategy — the Pyramid, Coverage & What Seniors Actually Review

           /        \           Highest confidence, slowest, flakiest — reserve for

The layers trade confidence per test against speed and stability. Unit tests give instant, pinpoint feedback on logic; E2E tests prove the whole assembled system works but are slow and flake-prone (real network, real rendering, real timing); integration tests are the sweet spot where wiring bugs — the most common kind — get caught at near-unit speed. The classic pyramid says "mostly unit"; the modern trophy (Kent C. Dodds) fattens the integration layer because unit tests over-isolated with mocks were missing real failures. Both agree on the extremes: lots of fast tests, few E2E. Wherever you land, the budget question a senior asks is: "when this suite is green, what am I actually confident about?"

Senior framing — test behavior, not implementation: the tests worth having assert the contract (given X, the observable result is Y) and would keep passing if you rewrote the internals from scratch. Tests that assert implementation — "it called _helper once", "state went A→B→C internally", snapshot-everything — fail on every refactor and pass on real bugs, training the team to update tests reflexively instead of investigating. That's how suites rot into a tax nobody trusts. In review, seniors read tests first (they're the spec), ask "what breaks if this assertion is wrong?", check that failure output would point at the bug, and treat a flaky test as a sev-2: a suite people re-run until green provides negative value — worse than no suite, because it launders real failures as noise. And the highest-leverage move of all happens before any test is written: design code as pure core + injected boundaries, and the tests become short, fast, and mock-free almost by accident.

Interview Questions

Q1 (Beginner): What's the difference between toBe and toEqual?

toBe checks identity (Object.is, essentially ===): the same primitive value or the very same object reference. toEqual checks deep structural equality: two different objects with the same shape and values pass. expect({a:1}).toEqual({a:1}) passes; .toBe fails. Use toBe for primitives/identity, toEqual for data structures — and toBeCloseTo for floats.

Q2 (Beginner): What is the AAA pattern, and why should a test verify one behavior?

Arrange (set up inputs/state) → Act (invoke the one thing under test) → Assert (verify the outcome). One behavior per test means a failure points at exactly one broken promise — the test name is the bug report. Multi-behavior tests stop at the first failed assertion, hiding the rest, and their names go vague ("works correctly"), so failures require reading the test body to interpret.

Q3 (Mid): This test passes but the code is broken: it("rejects", () => { expect(load(-1)).rejects.toThrow(); }). Why — and give two fixes.

The rejects matcher returns a promise that is never awaited or returned, so the test completes synchronously before the assertion resolves — a floating promise, and a green test regardless of behavior. Fixes: make the test async and await expect(load(-1)).rejects.toThrow() (or return the expectation). For try/catch-style tests, add expect.assertions(1) so the no-throw path fails loudly. Prevention: enable an unhandled-rejection-strict runner mode or the no-floating-promises lint rule.

Q4 (Mid): Spy vs stub vs mock — and where should mocking happen in a well-designed suite?

A spy records calls (count/arguments) while optionally passing through; a stub returns canned responses to control the path under test; "mock" colloquially covers both (vi.fn()/jest.fn() does both). Mock at process boundaries: network, database, filesystem, clock, randomness, third-party SDKs — things that are slow, nondeterministic, or external. Don't mock your own internal modules: that couples tests to the current call graph, so refactors break tests while genuine bugs in the mocked-out collaborator sail through. Prefer dependency injection so boundaries can be swapped without module-mocking machinery.

Q5 (Senior): Your team hit their 90% coverage mandate, yet production bugs haven't dropped. Diagnose and fix.

Goodhart's law: coverage measures lines executed under test, not behavior verified — a mandate produces execute-only tests (few/weak assertions, over-mocked internals, blindly-approved snapshots) that raise the number without adding checking power. Diagnose by reading failing-power: mutation testing (e.g. Stryker) shows how many injected bugs the suite actually catches; review whether tests assert contracts or implementation details. Fix: drop coverage as a target (keep it as a map of untested areas), review tests for behavioral assertions like production code, shift budget toward integration tests on critical paths, and track escaped-defect rate — the metric that was supposed to move.

Q6 (Senior): A test fails ~1 run in 20 in CI. Why is that worse than a consistently failing test, and how do you attack it?

A consistent failure gets fixed; a flake trains the team to hit re-run — once green-by-retry is culture, the suite can no longer tell you anything, because real regressions are indistinguishable from noise (negative value). Attack: quarantine it immediately (skip + ticket) so trust survives; reproduce by running it in a loop and in random order (--sequence.shuffle) to expose order-dependence; hunt the usual suspects — unawaited promises, real timers/dates, shared state bleeding between tests, network reliance, race between assertion and async render. Fix by making the nondeterminism controllable: fake timers, injected clock/random, fresh state per test, and await-ing what the code actually does. Institutionally: track flake rate and treat quarantined tests as debt with an owner, not a graveyard.

🏋️ Exercises

1 (Easy): Write tests for const clamp = (n, lo, hi) => Math.min(hi, Math.max(lo, n)): inside range, below, above, and the n === lo edge.
Solution
import { describe, it, expect } from "vitest";
describe("clamp", () => {
  it("returns n when inside the range", () => expect(clamp(5, 0, 10)).toBe(5));
  it("clamps up to the lower bound",    () => expect(clamp(-3, 0, 10)).toBe(0));
  it("clamps down to the upper bound",  () => expect(clamp(99, 0, 10)).toBe(10));
  it("treats the bounds as inclusive",  () => expect(clamp(0, 0, 10)).toBe(0));
});
2 (Easy): parsePrice("12.50") returns cents (1250) or throws TypeError on bad input. Test both paths — including why the throw test needs a wrapper function.
Solution
it("parses a decimal string to integer cents", () => {
  expect(parsePrice("12.50")).toBe(1250);
});
it("throws TypeError on non-numeric input", () => {
  expect(() => parsePrice("abc")).toThrow(TypeError);
  // The wrapper matters: expect(parsePrice("abc")) would throw DURING argument
  // evaluation — before expect() ever runs — so the runner reports an error, not
  // a failed assertion. Passing a function lets expect call it inside try/catch.
});
3 (Medium): getUsername(id, api) calls api.fetchUser(id) and returns user.name uppercased; it rethrows a "user not found" error for null. Test both with a stubbed api.
Solution
it("uppercases the fetched user's name", async () => {
  const api = { fetchUser: vi.fn().mockResolvedValue({ name: "ada" }) };
  await expect(getUsername(1, api)).resolves.toBe("ADA");
  expect(api.fetchUser).toHaveBeenCalledWith(1);        // spy: right collaborator call
});
it("throws when the user does not exist", async () => {
  const api = { fetchUser: vi.fn().mockResolvedValue(null) };   // stub: force the path
  await expect(getUsername(9, api)).rejects.toThrow("user not found");
});
// No module mocks needed — `api` is injected. Testability came from the design.
4 (Medium): Using fake timers, test that throttle(fn, 1000) invokes immediately, swallows calls inside the window, and allows the next call after 1000ms.
Solution
it("throttles to one call per window", () => {
  vi.useFakeTimers();
  const fn = vi.fn();
  const t = throttle(fn, 1000);

  t();                                 // leading call goes through
  t(); t();                            // inside the window → swallowed
  expect(fn).toHaveBeenCalledTimes(1);

  vi.advanceTimersByTime(999);
  t();
  expect(fn).toHaveBeenCalledTimes(1); // still inside

  vi.advanceTimersByTime(1);
  t();
  expect(fn).toHaveBeenCalledTimes(2); // window elapsed → next call passes
  vi.useRealTimers();
});
5 (Hard): Test retry(fn, 3) (from the Functional Patterns lesson): succeeds first try; succeeds on the 3rd attempt; exhausts retries and rethrows the last error. Use per-call mock sequencing.
Solution
it("returns immediately on first success", async () => {
  const fn = vi.fn().mockResolvedValue("ok");
  await expect(retry(fn, 3)).resolves.toBe("ok");
  expect(fn).toHaveBeenCalledTimes(1);               // no wasted retries
});
it("retries until a success within the budget", async () => {
  const fn = vi.fn()
    .mockRejectedValueOnce(new Error("down"))
    .mockRejectedValueOnce(new Error("down"))
    .mockResolvedValueOnce("ok");                    // 3rd attempt wins
  await expect(retry(fn, 3)).resolves.toBe("ok");
  expect(fn).toHaveBeenCalledTimes(3);
});
it("rethrows the final error when retries are exhausted", async () => {
  const fn = vi.fn().mockRejectedValue(new Error("still down"));
  await expect(retry(fn, 3)).rejects.toThrow("still down");
  expect(fn).toHaveBeenCalledTimes(3);               // tried exactly the budget
});
6 (Hard): This suite passes in isolation but fails when run with others. Find all three isolation bugs and fix them: 
const cart = [];
vi.spyOn(console, "warn").mockImplementation(() => {});
it("adds", () => { cart.push("x"); expect(cart).toHaveLength(1); });
it("warns when full", () => { fillCart(cart); expect(console.warn).toHaveBeenCalled(); });
Solution
// Bug 1: shared mutable `cart` at module scope — test 2 inherits test 1's item,
//         and any OTHER file's test order changes what's in it.
// Bug 2: the console.warn spy is created once at module load and never restored —
//         it leaks into every subsequent test file (silencing real warnings there).
// Bug 3: expect(console.warn) asserts against a global whose call history ACCUMULATES —
//         a warn from a previous test makes this pass vacuously.
let cart, warnSpy;
beforeEach(() => {
  cart = [];                                                        // fresh state per test
  warnSpy = vi.spyOn(console, "warn").mockImplementation(() => {}); // fresh spy per test
});
afterEach(() => vi.restoreAllMocks());                              // no leakage outward

it("adds", () => { cart.push("x"); expect(cart).toHaveLength(1); });
it("warns when full", () => {
  fillCart(cart);
  expect(warnSpy).toHaveBeenCalledTimes(1);   // assert on THIS test's spy, precisely
});

🎯 Key Takeaways

  • AAA structure, one behavior per test, names that read as bug reports. toBe = identity, toEqual = structure, toBeCloseTo = floats; assert the contract, not the whole payload.
  • Async tests must await everything — including expect(...).rejects/resolves — or they pass without testing; expect.assertions(n) guards try/catch styles.
  • Mock at process boundaries only (network, db, clock, random); mocking your own modules freezes the call graph and lets real bugs through. Dependency injection beats module-mock machinery.
  • Fake timers make time an input: debounce/retry/TTL logic becomes instant and deterministic. Fresh state per test in beforeEach; suites must pass in any order.
  • Strategy: many fast unit tests, a strong integration middle (the trophy), few E2E on critical journeys. Coverage locates untested code — as a target it manufactures assertion-free tests.
  • Test behavior, not implementation — tests are the spec and should survive a rewrite of the internals. Flaky tests are urgent: a suite people re-run until green has negative value.
