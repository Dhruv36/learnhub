Closures & this Intermediate

The two most-asked JavaScript interview topics, and the machinery behind React hooks, event handlers, and every "why is this undefined?!" bug ever filed.

Closures — Functions Remember Where They Were Born

function makeCounter() {
  let count = 0;                      // lives in makeCounter's scope
  return () => ++count;               // this function CLOSES OVER count
}
const next = makeCounter();
next(); next(); next();               // 1, 2, 3 — count survives between calls
const other = makeCounter();          // separate closure, separate count
other();                              // 1

A closure = function + the scope it was created in. The scope lives as long as the function does. That's it — everything else is application:

// Private state (no class needed):
const createWallet = (balance) => ({
  deposit: (n) => balance += n,
  getBalance: () => balance,          // balance is unreachable except through these
});

// Debounce — THE practical closure interview question:
function debounce(fn, ms) {
  let timer;                          // closed-over between calls
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), ms);
  };
}
searchInput.addEventListener("input", debounce(e => search(e.target.value), 300));

The Loop Trap (asked in every screening round)

for (var i = 0; i < 3; i++) setTimeout(() => console.log(i));   // 3 3 3 — one shared i
for (let i = 0; i < 3; i++) setTimeout(() => console.log(i));   // 0 1 2 — let: fresh i per iteration

this — Decided by the CALL, Not the Definition

const user = {
  name: "Asha",
  greet() { return `Hi, ${this.name}`; },
};
user.greet();                         // "Hi, Asha"     — called ON user → this = user
const f = user.greet;
f();                                  // "Hi, undefined" — called bare → this = undefined (strict)
setTimeout(user.greet, 100);          // same bug: the method is passed, the receiver is lost
Call stylethis is…
obj.method()obj (whatever is left of the dot)
fn() bareundefined in modules/strict mode
new Fn()the newly created object
fn.call(x) / fn.bind(x)x — set explicitly
Arrow functionNo own this — inherits from where it was written

Why Arrows Fixed Everything

class Timer {
  seconds = 0;
  start() {
    // ❌ setInterval(function () { this.seconds++; }, 1000);  — this = undefined
    setInterval(() => this.seconds++, 1000);   // ✅ arrow inherits the method's this
  }
}

Modern rule of thumb: methods on objects/classes = regular functions (they need dynamic this); callbacks = arrows (they should inherit it). That covers 99% of code.

💡 Interview follow-ups to expect: implement throttle (cousin of debounce) • what does bind return? (a new function with this fixed — original untouched) • why do React hooks "capture stale state"? (each render's closure captures that render's variables — the same makeCounter mechanics).
