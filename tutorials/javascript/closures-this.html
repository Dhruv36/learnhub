Closures & this Intermediate

The two most-asked JavaScript interview topics — and not for hazing: closures are the machinery behind event handlers, debounce, module privacy, and React hooks; this is behind every "why is this undefined?!" production bug. Both reduce to one-sentence rules. This lesson drills the rules until the puzzle questions become boring.

Closures — the One-Sentence Rule

A function permanently keeps access to the scope where it was CREATED. Not a copy of the values — the living variables themselves.

function makeCounter() {
  let count = 0;                       // lives in makeCounter's scope
  return () => ++count;                // this arrow CLOSES OVER count
}
const next = makeCounter();
next(); next(); next();                // 1, 2, 3 — count SURVIVED makeCounter returning
const other = makeCounter();
other();                               // 1 — each call = fresh scope = independent state

// "Live variables, not copies":
function pair() {
  let n = 0;
  return { inc: () => ++n, get: () => n };   // TWO closures over the SAME n
}
const p = pair(); p.inc(); p.inc(); p.get(); // 2 — they share it

Mechanically: functions carry a reference to their birth environment; as long as the function is reachable, that environment can't be garbage-collected. (Which is also why closures appear in the memory-leaks lesson: a forgotten timer's closure pins everything it captured.)

The Loop Question (asked in ~every screening round)

for (var i = 0; i < 3; i++) setTimeout(() => console.log(i));   // 3 3 3
for (let i = 0; i < 3; i++) setTimeout(() => console.log(i));   // 0 1 2

Why: var is function-scoped — all three arrows close over ONE shared i, which is 3 by the time the timers fire. let in a for-header creates a fresh binding per iteration (the spec special-cases this), so each arrow captures its own i. Pre-let fixes you should recognize: an IIFE per iteration, or setTimeout's extra args.

Closure Applications You'll Ship This Month

// 1. DEBOUNCE — the practical closure interview question:
function debounce(fn, ms) {
  let timer;                                   // closed-over between calls
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), ms);
  };
}
search.addEventListener("input", debounce(e => runSearch(e.target.value), 300));

// 2. THROTTLE — its cousin (regular execution DURING activity):
function throttle(fn, ms) {
  let last = 0;
  return (...args) => {
    const now = Date.now();
    if (now - last >= ms) { last = now; fn(...args); }
  };
}   // scroll/resize → throttle; typing → debounce

// 3. PRIVATE STATE (pre-#fields, still idiomatic for factories):
const createWallet = (balance = 0) => ({
  deposit: (n) => balance += n,
  getBalance: () => balance,       // balance is UNREACHABLE except through these
});

// 4. MEMOIZE:
function memoize(fn) {
  const cache = new Map();
  return (x) => cache.has(x) ? cache.get(x) : (cache.set(x, fn(x)), cache.get(x));
}

// 5. ONCE:
const init = (() => { let done = false;
  return () => { if (!done) { done = true; setup(); } } })();

this — the One-Sentence Rule

this is decided by HOW the function is CALLED — nothing else. (Except arrows, which don't have one and inherit lexically.)

const user = {
  name: "Asha",
  greet() { return `Hi, ${this.name}`; },
};
user.greet();               // "Hi, Asha"        — called ON user (left of the dot)
const f = user.greet;
f();                        // "Hi, undefined"   — bare call: this = undefined (strict/modules)
setTimeout(user.greet, 100) // same bug — the METHOD was passed; the receiver was lost

Losing this — the Three Classic Crime Scenes

// 1. Passing methods as callbacks:
setTimeout(user.greet, 100);            // ❌ lost
setTimeout(() => user.greet(), 100);    // ✅ arrow preserves the call form
setTimeout(user.greet.bind(user), 100); // ✅ bind fixes the receiver permanently

// 2. Old-style function callbacks inside methods:
class Timer {
  seconds = 0;
  start() {
    setInterval(function () { this.seconds++; }, 1000);  // ❌ this = undefined
    setInterval(() => this.seconds++, 1000);              // ✅ arrow inherits the METHOD's this
  }
}

// 3. Class methods handed to event listeners:
class Menu {
  open = () => this.render();     // class-FIELD arrow: auto-bound per instance —
}                                  // the standard fix for handler methods
btn.addEventListener("click", menu.open);   // works
📏 The modern rule that ends 99% of this-bugs: methods on objects/classes = regular functions (they NEED dynamic this); callbacks = arrows (they should inherit it). The remaining 1%: know bind, and know that DOM listeners set this = currentTarget for regular functions (one reason arrows are the safer habit there too).

call / apply / bind — the Explicit Tools

fn.call(ctx, a, b);        // invoke now, this = ctx, args listed
fn.apply(ctx, [a, b]);     // same, args as array (mostly replaced by spread)
const g = fn.bind(ctx, a); // NEW function: this fixed forever + 'a' pre-filled (partial application)
// bind's this cannot be overridden later — not even by call, only by new.

The React Hooks Connection (why this lesson matters in 2026)

function Chat() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    const id = setInterval(() => console.log(count), 1000);  // closes over THIS render's count
    return () => clearInterval(id);
  }, []);                       // ⚠ empty deps: the interval logs 0 forever — a STALE CLOSURE
}
// It's makeCounter mechanics wearing a React costume: each render creates new functions closing
// over that render's variables. Fixes: correct deps, functional updates (setCount(c => c+1)),
// or a ref for the latest value. Interviewers LOVE candidates who name the mechanism.

Interview Questions

Q1: Implement debounce with immediate-first-call support.

Keep timer in the closure; if immediate && !timer, call fn now; always (re)set the timer, whose callback clears timer and, when !immediate, calls fn. Points: (...args) forwarding, clearTimeout before reset, and explaining WHERE timer lives and why it survives calls (the closure).

Q2: Predict and explain: obj = {x: 1, getX() { return this.x }, getX2: () => this.x} — obj.getX() vs obj.getX2() vs [obj.getX][0]().

obj.getX() → 1 (dot rule). obj.getX2() → undefined — the arrow has no this; it inherited the module's (undefined.x → TypeError in strict, undefined property on globalThis otherwise — say "the enclosing scope's this, so NOT obj"). [obj.getX][0]() → the array is left of the dot-ish call… actually the function is called as an element access: this = the ARRAY, so undefined.x → undefined. The pattern: find the call site, look left.

Q3: How do closures cause memory leaks, and how would you confirm one?

A live function pins its entire captured scope: forgotten intervals/listeners keep closed-over data (and DOM refs) reachable forever. Confirm: heap snapshot diff across repeated mount/unmount — growing objects retained via "context" links from a listener/timer. Fix: clear timers, abort listeners, capture minimally.

Q4: Why can't bind change an arrow's this? What DOES bind return?

Arrows never had a this binding to override — this inside resolves lexically at creation, like any outer variable; call/apply/bind's context argument is simply ignored. bind on a regular function returns a new "bound function" whose this (and any partial args) are locked; only new can override a bound this.

🎯 Key Takeaways

  • Closure = function + birth scope, holding LIVE variables — the basis of debounce, privacy, memoize, hooks.
  • var loops share one binding (3 3 3); let mints one per iteration (0 1 2).
  • this = how it's CALLED: new > bind/call > dot > bare(undefined); arrows opt out and inherit lexically, unbindably.
  • Methods = regular functions; callbacks = arrows; handler methods = class-field arrows or bind.
  • React stale closures ARE this lesson: each render's functions capture that render's values.
  • Long-lived closures pin what they capture — clear timers, abort listeners, capture small.
