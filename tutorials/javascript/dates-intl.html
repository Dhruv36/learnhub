Dates, Time & Intl Objects & Data

What you'll master: the mental model that makes dates stop being scary — an instant on a universal timeline vs. its many local representations — plus the Date object's real API and its famous traps, ISO 8601 and how to move dates across the network safely, timezone handling that survives DST, the Intl APIs that replace hand-rolled formatting, and where Temporal fits. Prerequisites: Numbers & Precision (timestamps are just numbers) and Objects Deep Dive. Date bugs are a production-incident genre of their own — this lesson is how you stop writing them.

Part 1 — The Mental Model: Instant vs. Wall-Clock

Date.now()            // 1752835200000 — ms since 1970-01-01T00:00:00 UTC (the "epoch")
const d = new Date(); // a Date "object" is a THIN WRAPPER around exactly one such number

d.getTime()           // → that number. This is what a Date IS. Everything else is a VIEW:
d.toISOString()       // "2026-07-18T10:40:00.000Z"  — the instant, rendered in UTC
d.toString()          // "Sat Jul 18 2026 16:10:00 GMT+0530" — SAME instant, rendered in the
                      //   machine's LOCAL timezone. Different text, identical moment.

The single idea that untangles every date bug: separate the instant (one absolute point on the universal timeline — a number of milliseconds since the epoch, no timezone attached) from a wall-clock representation ("18 July, 4:10 PM" — which is meaningless without saying where). A JS Date stores only the instant; every getter (getHours(), getDate()…) is a view of that instant through the machine's local timezone, and every getUTC* twin is the view through UTC. Two users on different continents holding the "same" Date object see different hours — same instant, different views. The professional rule that follows: store and transmit instants (UTC/epoch); convert to local wall-clock only at the display edge.

Part 2 — Constructing Dates & the Classic Traps

new Date(2026, 6, 18)        // ⚠️ July 18 — the MONTH is 0-indexed (0=Jan … 11=Dec); days are 1-indexed
new Date("2026-07-18T10:40:00Z")      // ISO string with Z → parsed as UTC. Reliable.
new Date("2026-07-18T10:40:00+05:30") // ISO with offset → reliable.
new Date("2026-07-18")       // date-ONLY ISO → parsed as UTC midnight ⚠️ (may display as the 17th locally!)
new Date("07/18/2026")       // non-ISO string → implementation-defined parsing. NEVER do this.

const d = new Date(2026, 6, 18);
d.getDay()    // 6 = Saturday — day of WEEK (0=Sun). getDate() is day of MONTH. Endless confusion.
d.setDate(d.getDate() + 14); // add 14 days — setters MUTATE and auto-rollover across months ✓

// The invalid-date footgun — no throw, just a poisoned object:
const bad = new Date("garbage");
bad.getTime()                 // NaN
Number.isNaN(bad.getTime())   // ✅ the standard validity check

Four traps account for most date bugs. (1) Months are 0-indexed in the numeric constructor and getMonth() — new Date(2026, 12, 1) is January 2027 (rollover, silently). (2) Date-only ISO strings parse as UTC midnight, so new Date("2026-07-18") shown to a user west of Greenwich displays as July 17 — the classic "birthday is off by one day" bug. If you mean local midnight, use new Date(2026, 6, 18). (3) Non-ISO string parsing is not standardized — feed Date only ISO 8601 strings or numbers. (4) Invalid dates don't throw; they return a Date whose time is NaN, and NaN silently propagates through any arithmetic. Validate with Number.isNaN(d.getTime()) at trust boundaries.

Part 3 — Date Arithmetic & Durations

// Differences: subtracting Dates coerces to their numbers → milliseconds:
const ms = new Date("2026-07-18T00:00Z") - new Date("2026-07-01T00:00Z");
const days = ms / 86_400_000;                     // 17 — fine for INSTANT math

// Adding calendar units: use the setters' auto-rollover (never add ms for "a day"):
const d = new Date(2026, 0, 31);
d.setMonth(d.getMonth() + 1);    // ⚠️ Jan 31 + 1 month → Mar 3! (Feb 31 rolls over)
// Rollover is also the FIX for day math:
const due = new Date(2026, 6, 25);
due.setDate(due.getDate() + 10); // Aug 4 — crosses the month boundary correctly

// Why "1 day = 86,400,000 ms" is a LIE for wall-clock math:
// On a DST-change day, a local day is 23 or 25 hours. Adding 24h of ms to "9:00 AM"
// can land on 8:00 or 10:00 AM the next day. setDate(+1) keeps the wall-clock time.

// Measuring ELAPSED TIME in code — don't use Date at all:
const t0 = performance.now();       // monotonic, sub-ms, immune to clock changes
doWork();
console.log(performance.now() - t0, "ms");   // Date.now() can jump (NTP sync, manual change)

Split arithmetic the same way you split the model: instant math (differences, timeouts, "is X before Y") works directly on milliseconds and is safe; calendar math ("next month", "+1 day", "same time tomorrow") must go through the calendar-aware setters, because months have different lengths and DST makes some local days 23 or 25 hours long. The setMonth rollover surprise (Jan 31 → Mar 3) is worth knowing cold: rollover is usually your friend (it handles month/year boundaries in day arithmetic for free) but wrong for "same day next month" logic, which needs a clamp-to-month-end rule — a genuine business decision, and a big reason date libraries exist. And for benchmarking or timeouts inside a program, use performance.now() — it's monotonic (never goes backwards), while Date.now() follows the system clock wherever it jumps.

Part 4 — Timezones Done Right

// The server/API contract: always UTC, always ISO 8601:
JSON.stringify({ createdAt: new Date() })   // {"createdAt":"2026-07-18T10:40:00.000Z"}
// (Date has a toJSON() that emits toISOString() — automatic and correct. JSON.parse does
//  NOT revive it — it stays a string; convert back with new Date(str) at the edge.)

// Displaying an instant in a SPECIFIC timezone (not the machine's):
const d = new Date("2026-07-18T10:40:00Z");
d.toLocaleString("en-US", { timeZone: "America/New_York" })  // "7/18/2026, 6:40:00 AM"
d.toLocaleString("en-IN", { timeZone: "Asia/Kolkata" })      // "18/7/2026, 4:10:00 pm"

// IANA names ("America/New_York"), NOT fixed offsets ("-05:00") — the IANA zone
// encodes DST rules and history; a fixed offset is wrong half the year.

// What timezone is the user in?
Intl.DateTimeFormat().resolvedOptions().timeZone   // "Asia/Kolkata"

The architecture that survives contact with reality: store UTC instants; render per-viewer at the edge. A Date can't "hold" a foreign timezone — but you don't need it to: toLocaleString with a timeZone option renders any instant in any IANA zone, DST handled for you. Use IANA zone names, never fixed offsets — "New York time" is UTC-5 in winter and UTC-4 in summer, and only the IANA database knows which applies to a given instant. The one case where UTC-everything is wrong: future-scheduled local events ("standup at 9:00 AM in Berlin"). If Berlin's DST rules change between now and then, a pre-computed UTC instant lands at the wrong wall-clock time — for those, store the wall-clock time plus the IANA zone, and resolve to an instant as late as possible.

Part 5 — Intl: Stop Hand-Formatting

const d = new Date("2026-07-18T10:40:00Z");

// Dates — one formatter, any locale, reused (construction is the expensive part):
const fmt = new Intl.DateTimeFormat("de-DE", { dateStyle: "full", timeStyle: "short",
                                               timeZone: "Europe/Berlin" });
fmt.format(d)    // "Samstag, 18. Juli 2026 um 12:40"

// Numbers & currency — never template-string money:
new Intl.NumberFormat("en-US", { style: "currency", currency: "EUR" }).format(123456.7)
// "€123,456.70"
new Intl.NumberFormat("de-DE", { style: "currency", currency: "EUR" }).format(123456.7)
// "123.456,70 €"   ← same value: different separators, symbol position. This is why.

new Intl.NumberFormat("en", { notation: "compact" }).format(1_234_567)   // "1.2M"

// Relative time — "3 days ago" without a library:
new Intl.RelativeTimeFormat("en", { numeric: "auto" }).format(-3, "day")  // "3 days ago"
new Intl.RelativeTimeFormat("es").format(2, "hour")                       // "dentro de 2 horas"

// Also in the family: Intl.ListFormat ("a, b, and c"), Intl.PluralRules (1 item / 2 items),
// Intl.Collator (locale-correct sorting: "ä" sorts differently in German vs Swedish)

Every hand-rolled formatter — `$${amount.toFixed(2)}`, month-name arrays, "3 days ago" if-chains — is a localization bug waiting for its first non-US user. Intl is the built-in, locale-aware answer, and it ships with the platform (no date-library download needed for formatting). Two practical notes: formatter construction is expensive and format calls are cheap, so create formatters once (module scope) and reuse them in loops and render paths; and pass undefined as the locale to use the user's own locale while still controlling the options.

Sorting user-visible strings? names.sort() compares UTF-16 code units — uppercase sorts before lowercase and accented letters land at the end. Use names.sort(new Intl.Collator("de").compare) (or localeCompare) for human-correct ordering, plus { numeric: true } to get "file2" < "file10".

Part 6 — Deep Dive: Temporal, and How Pros Structure Date Code

// Temporal (TC39 Stage 3; shipping progressively, polyfill available today) fixes Date's design:
Temporal.Instant.from("2026-07-18T10:40:00Z")       // an instant — what Date should have been
Temporal.PlainDate.from("2026-07-18")               // a DATE with no time/zone — birthdays, holidays
Temporal.ZonedDateTime.from("2026-07-18T16:10[Asia/Kolkata]")   // instant + zone TOGETHER
Temporal.Duration.from({ months: 1 })               // a real duration type

// Immutable arithmetic that answers the hard questions explicitly:
const jan31 = Temporal.PlainDate.from("2026-01-31");
jan31.add({ months: 1 })                            // 2026-02-28 — CLAMPS (documented!), not Mar 3
// distinct types make "date-only vs instant" bugs unrepresentable

// Until Temporal is everywhere, the same discipline by hand:
// 1. API boundary: parse ISO strings → Date ONCE, validate getTime() !== NaN
// 2. Core logic: pass Date/epoch-ms around; compare/diff freely; calendar math via setters
// 3. Display edge: Intl formatters with explicit timeZone
// 4. date-fns/Day.js only if you need heavy calendar math (business days, recurrence)

Temporal exists because Date conflates concepts the domain keeps separate: it has one type where the problem has several (instant, zoned datetime, date-only, time-only, duration), it's mutable, and its parsing is quirky. Temporal gives each concept its own immutable type, makes every ambiguous operation (month-end clamping, DST-gap resolution) an explicit documented choice, and bundles the IANA zone into ZonedDateTime so "9 AM in Berlin, whatever UTC that turns out to be" is finally representable. It's the direction the language is going — learn its type distinctions now, because they're the correct mental model even when you're writing plain Date code.

Senior framing: date-handling maturity is an architecture question, not an API question. The systems that don't have date bugs all converge on the same shape: UTC instants in storage and transit (ISO 8601 with Z) → validation at parse boundaries → instant math in the core → locale/zone rendering only at the display edge, via Intl, with IANA zone names. Every date incident you'll ever debug — the off-by-one birthday, the meeting that moved an hour after DST, the report that double-counts the 25-hour day — is a violation of one of those layers. When you review code, don't review the formatting calls; review where in the flow timezone conversion happens. If it happens more than once per value, or before storage, that's the bug.

Interview Questions

Q1 (Beginner): What does a JavaScript Date actually store, and why do toISOString() and toString() show different times?

A single number: milliseconds since the Unix epoch (1970-01-01T00:00:00 UTC) — an absolute instant with no timezone inside it. toISOString() renders that instant in UTC; toString() renders the same instant in the machine's local timezone. Different views, identical moment — getTime() is equal in both.

Q2 (Beginner): A user reports their birthday "July 18" displays as July 17. The code does new Date("2026-07-18").toLocaleDateString(). Explain.

Date-only ISO strings are parsed as UTC midnight. For any user in a timezone west of UTC, UTC midnight is still the previous local day, so the local rendering shows July 17. Fix: build a local date (new Date(2026, 6, 18)), or better, treat date-only values (birthdays) as calendar dates — a string or Temporal.PlainDate — never as instants at all.

Q3 (Mid): Why is time + 24 * 60 * 60 * 1000 the wrong way to compute "same time tomorrow"? When is it fine?

Because a local calendar day isn't always 24 hours: on DST transition days it's 23 or 25. Adding exactly 24h of milliseconds lands an hour early or late on the wall clock. Use setDate(getDate() + 1), which does calendar-aware rollover. Millisecond arithmetic is fine for instant math — durations, diffs, timeouts — where "24 hours of elapsed time" is literally what you mean.

Q4 (Mid): Design the date handling for an API + web app used across timezones. What format goes over the wire, and where do conversions happen?

Wire and storage: UTC instants as ISO 8601 with Z (what Date.prototype.toJSON already emits). Parse to Date once at the API boundary and validate (getTime() not NaN). All comparisons and diffs happen on instants in the core. Convert to wall-clock only at the display edge with toLocaleString/Intl.DateTimeFormat using IANA timeZone names. Exception: future local-time events (a 9 AM meeting) store wall-clock + IANA zone and resolve to an instant late, so DST-rule changes can't shift them.

Q5 (Senior): const d = new Date(2026, 0, 31); d.setMonth(1); — what's the result, why, and what does it teach about "add one month"?

March 3, 2026 (Feb 31 doesn't exist → rolls over 3 days into March), and note setMonth mutated d in place. It teaches that "add one month" is not a well-defined operation — the Jan 31 case needs a policy: clamp to Feb 28 (what billing systems want, and what Temporal.PlainDate.add does by default) or roll over (what Date does). Senior answer: name the policy explicitly in code (a addMonthsClamped helper or Temporal), never leave it to Date's accidental behavior.

Q6 (Senior): What problems with Date does Temporal solve? Name the key types.

Date is one mutable type for several distinct concepts, with 0-indexed months, unspecified non-ISO parsing, and no way to represent "a date without a time" or "a time in a specific zone". Temporal is immutable, and splits the domain into explicit types: Instant (point on the timeline), PlainDate/PlainTime/PlainDateTime (calendar values, no zone), ZonedDateTime (instant + IANA zone together, DST-aware arithmetic), and Duration. Ambiguities (month-end clamping, DST gaps) become explicit options instead of silent rollovers — making whole bug classes unrepresentable.

🏋️ Exercises

1 (Easy): Given const d = new Date("2026-07-18T10:40:00Z"), print the epoch milliseconds, the UTC ISO string, and the date rendered for a user in Tokyo.
Solution
const d = new Date("2026-07-18T10:40:00Z");
console.log(d.getTime());          // 1784544000000-ish — the instant as a number
console.log(d.toISOString());      // "2026-07-18T10:40:00.000Z"
console.log(d.toLocaleString("ja-JP", { timeZone: "Asia/Tokyo" }));  // "2026/7/18 19:40:00"
2 (Easy): Write isValidDate(input) that accepts anything, tries to build a Date, and returns true only for a real date.
Solution
const isValidDate = (input) => {
  const d = new Date(input);
  return !Number.isNaN(d.getTime());
};
isValidDate("2026-07-18")   // true
isValidDate("garbage")      // false — invalid Dates don't throw, their time is NaN
isValidDate(0)              // true — the epoch itself is valid
3 (Medium): Write addDays(date, n) that returns a NEW Date n days later (original untouched), correct across month boundaries and DST.
Solution
const addDays = (date, n) => {
  const copy = new Date(date.getTime());   // Date setters mutate — copy first
  copy.setDate(copy.getDate() + n);        // calendar-aware rollover handles month/DST
  return copy;
};
const d = new Date(2026, 6, 25);
const later = addDays(d, 10);
console.log(later.getMonth(), later.getDate());  // 7 4 → Aug 4
console.log(d.getDate());                        // 25 — original untouched
4 (Medium): Format 1234567.891 as Indian-locale INR currency and as compact notation, using Intl.NumberFormat.
Solution
const inr = new Intl.NumberFormat("en-IN", { style: "currency", currency: "INR" });
console.log(inr.format(1234567.891));   // "₹12,34,567.89" — note the Indian digit grouping

const compact = new Intl.NumberFormat("en-IN", { notation: "compact" });
console.log(compact.format(1234567.891));   // "12L" (lakh) — locale-aware compaction
5 (Hard): Write timeAgo(date) returning "3 days ago" / "in 2 hours" style strings via Intl.RelativeTimeFormat, choosing the largest sensible unit.
Solution
const timeAgo = (date, locale = "en") => {
  const rtf = new Intl.RelativeTimeFormat(locale, { numeric: "auto" });
  const diffSec = Math.round((date.getTime() - Date.now()) / 1000);
  const units = [["year", 31536000], ["month", 2592000], ["day", 86400],
                 ["hour", 3600], ["minute", 60], ["second", 1]];
  for (const [unit, sec] of units) {
    if (Math.abs(diffSec) >= sec || unit === "second")
      return rtf.format(Math.trunc(diffSec / sec), unit);
  }
};
timeAgo(new Date(Date.now() - 3 * 86400_000));  // "3 days ago"
timeAgo(new Date(Date.now() + 7200_000));       // "in 2 hours"
6 (Hard): A meeting is stored as UTC instant "2026-03-07T14:00:00Z" (meant to be 9:00 AM New York). Show what happens to the displayed New York time one week later (after US DST starts Mar 8), and explain the storage fix.
Solution
const show = (iso) => new Date(iso).toLocaleTimeString("en-US",
  { timeZone: "America/New_York", hour: "numeric", minute: "2-digit" });

show("2026-03-07T14:00:00Z");   // "9:00 AM" — before DST, NY is UTC-5 ✓
show("2026-03-14T14:00:00Z");   // "10:00 AM" — after DST starts, NY is UTC-4 ✗ meeting drifted!

// Fix: a RECURRING/future local event isn't an instant — it's wall-clock + zone.
// Store { time: "09:00", zone: "America/New_York", date: ... } and resolve to a UTC
// instant per-occurrence, as late as possible (Temporal.ZonedDateTime models this directly).

🎯 Key Takeaways

  • A Date is one number — an instant (ms since epoch, UTC) — and every getter is a view of it through the machine's local zone (or UTC via getUTC*).
  • Traps to know cold: 0-indexed months, date-only ISO strings parse as UTC midnight, non-ISO parsing is unreliable, invalid dates give NaN silently, setters mutate.
  • Instant math = milliseconds (safe); calendar math = setters with rollover (DST-proof); elapsed-time measurement = performance.now() (monotonic).
  • Architecture: store/transmit UTC ISO instants → validate at parse boundaries → convert to wall-clock only at the display edge, with IANA zone names (never fixed offsets). Future local events store wall-clock + zone instead.
  • Intl (DateTimeFormat, NumberFormat, RelativeTimeFormat, Collator, PluralRules) replaces every hand-rolled formatter; construct formatters once, reuse everywhere.
  • Temporal fixes Date with immutable, purpose-specific types (Instant, PlainDate, ZonedDateTime, Duration) — its type distinctions are the right mental model today.
