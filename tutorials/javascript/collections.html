Map, Set, Symbol & WeakRefs Objects & Data

What you'll master: the keyed collections the language added to fix plain objects' limitations — Map (any-type keys, real ordering), Set (unique values), their Weak variants for leak-free caches, and Symbol for unique keys and the protocols that power iteration and coercion. Prerequisites: Objects (the object-as-map limitations these fix) and Prototypes. These are the right tools for a huge class of problems where people reach for plain objects and arrays and get subtle bugs.

Part 1 — Map: a Real Dictionary

const m = new Map();
m.set("name", "Ada");
m.set(42, "number key");           // keys can be ANY type — not just strings
const objKey = { id: 1 };
m.set(objKey, "object key");       // even objects as keys (by reference identity)

m.get("name");        // "Ada"
m.has(42);            // true
m.size;               // 3          — a property, not Object.keys(o).length
m.delete("name");     // true
for (const [k, v] of m) { }        // directly iterable, in INSERTION order
new Map([["a", 1], ["b", 2]]);     // construct from [key, value] pairs (e.g. Object.entries)

Map is the proper key-value collection, fixing every plain-object limitation from the Objects lesson: keys can be any type (objects, numbers, functions — compared by SameValueZero, so identity for objects), iteration is in true insertion order (no integer-key reordering), it has a direct .size, it's directly iterable, and it has no prototype keys (so no __proto__ pollution risk with user-controlled keys). Reach for Map whenever you have a genuine dictionary — dynamic keys, frequent add/delete, non-string keys, or untrusted keys.

Part 2 — Set: Unique Values

const s = new Set([1, 2, 2, 3, 3, 3]);   // duplicates dropped
s.size;               // 3
s.has(2);             // true — O(1) membership (vs array.includes O(n))
s.add(4).add(4);      // chainable; second 4 ignored
s.delete(1);          // true
[...s];               // [2, 3, 4] — spread to array (insertion order)

// The classic one-liner — dedupe an array:
const unique = [...new Set([1, 1, 2, 3, 3])];   // [1, 2, 3]

Set stores unique values (SameValueZero equality, so one NaN, one of each object reference) with fast O(1) has/add/ delete. Its two everyday uses: deduplication ([...new Set(arr)]) and fast membership testing — converting an array to a Set turns repeated arr.includes(x) (O(n) each) into O(1) lookups, a real performance win inside loops. Note equality is by identity for objects, so new Set([{}, {}]).size is 2 (two distinct object references).

Part 3 — WeakMap & WeakSet: Leak-Free Association

const cache = new WeakMap();       // keys MUST be objects; held WEAKLY
function getMeta(obj) {
  if (!cache.has(obj)) cache.set(obj, computeExpensiveMeta(obj));
  return cache.get(obj);
}
// When `obj` is no longer referenced ANYWHERE ELSE, it (and its cache entry) can be GC'd automatically.
// No manual cleanup, no leak — even though the cache "holds" it.

// WeakMap has NO .size, NO iteration, NO .keys() — because entries can vanish at any time.

WeakMap and WeakSet hold their keys/values weakly: a weak reference does not prevent garbage collection. So if the only thing referencing an object is a WeakMap key, the object (and its entry) is collected automatically once nothing else points to it. This makes them the correct tool for associating metadata with objects you don't own (DOM nodes, third-party objects) and for caches keyed by object identity — without the memory leak a regular Map would cause (a Map holds its keys strongly, pinning them forever). The trade-off, which is inherent: because entries can disappear unpredictably, WeakMap/WeakSet are not iterable and have no .size — you can only get/ set/has/delete by a key you already hold.

🧠 Map vs WeakMap decision: need to iterate, count, or use primitive keys → Map. Associating data with object lifetimes where you want automatic cleanup and never need to enumerate → WeakMap. The classic WeakMap use is private per-instance state and object-keyed caches (the Memory lesson revisits leaks).

Part 4 — Symbol: Unique, Unforgeable Values

const id = Symbol("id");           // the string is just a DESCRIPTION (for debugging), not an identity
const id2 = Symbol("id");
id === id2;                        // false — EVERY Symbol is unique, even with the same description

const user = { name: "Ada", [id]: 12345 };   // Symbol as an object key
user[id];                          // 12345
Object.keys(user);                 // ["name"] — Symbol keys are HIDDEN from normal enumeration
Object.getOwnPropertySymbols(user);// [Symbol(id)] — the explicit way to read them

Symbol.for("shared");              // GLOBAL registry — Symbol.for("shared") === Symbol.for("shared")

A Symbol is a guaranteed-unique primitive. Its main uses: collision-proof object keys (a Symbol key can never clash with a string key or another Symbol, so libraries use them for metadata that won't interfere with user properties), and hidden-ish properties (Symbol keys don't appear in Object.keys, for...in, or JSON.stringify — though they're not truly private, since Object.getOwnPropertySymbols reveals them). Symbol.for(key) uses a global registry so the same key returns the same Symbol across your whole program (unlike plain Symbol(), which is always fresh).

Part 5 — Well-Known Symbols: the Language's Extension Points

// Symbol.iterator — makes ANY object iterable with for...of and spread:
const range = {
  from: 1, to: 3,
  [Symbol.iterator]() {
    let current = this.from, last = this.to;
    return { next: () => current <= last ? { value: current++, done: false } : { value: undefined, done: true } };
  }
};
[...range];               // [1, 2, 3] — now spreadable/iterable
for (const n of range) { } // works

// Other well-known symbols you've already met:
// Symbol.toPrimitive  — controls object→primitive coercion (Types lesson)
// Symbol.asyncIterator — for await...of (Generators lesson)

Well-known symbols are built-in Symbols the language uses as hooks to let your objects plug into core behaviour. Symbol.iterator is the big one: defining a [Symbol.iterator]() method that returns an iterator makes an object work with for...of, spread, destructuring, and Array.from — this is the iteration protocol (the Generators lesson builds on it). Symbol.toPrimitive (Types lesson) hooks coercion. These symbols are why arrays, strings, Maps, and Sets are all iterable by the same mechanism — and how you make your own types first-class participants in it.

Part 6 — Converting Between Collections and Objects/JSON

// Object ↔ Map:
const obj = { a: 1, b: 2 };
const map = new Map(Object.entries(obj));      // object → Map
const back = Object.fromEntries(map);           // Map → object

// Map/Set → JSON (they DON'T serialise directly):
JSON.stringify(new Map([["a", 1]]));            // "{}" — LOSES everything! ⚠
JSON.stringify([...map]);                        // '[["a",1]]' — serialise the entries array
JSON.stringify([...new Set([1, 2])]);            // "[1,2]"

// A reviver/replacer for round-tripping Maps if you need to (advanced):
JSON.stringify(map, (k, v) => v instanceof Map ? { __map: [...v] } : v);

A crucial gotcha: Map and Set do not serialise to JSON — JSON.stringify(map) silently produces "{}", dropping all your data (because they have no enumerable own properties). To persist them you must convert first: spread to an array of entries ([...map]) or use Object.fromEntries for a plain object. This bites people who store a Map in state and try to serialise it — always convert at the JSON boundary. The Object.entries/fromEntries pair is the clean bridge between plain objects and Maps.

Part 7 — Build It: an LRU-ish Cache and a Tag Index

Let's use Map's insertion-order guarantee and Set's membership speed in two realistic mini-structures.

// 1. A bounded cache that evicts the OLDEST entry — relies on Map's insertion order:
class LRUCache {
  #max; #map = new Map();
  constructor(max = 100) { this.#max = max; }
  get(key) {
    if (!this.#map.has(key)) return undefined;
    const val = this.#map.get(key);
    this.#map.delete(key); this.#map.set(key, val);   // re-insert → now the NEWEST
    return val;
  }
  set(key, val) {
    this.#map.delete(key);                              // ensure re-insert moves it to newest
    this.#map.set(key, val);
    if (this.#map.size > this.#max) {
      this.#map.delete(this.#map.keys().next().value);  // evict the FIRST (oldest) key
    }
  }
}

// 2. A tag index using Sets for O(1) membership:
const tagsByPost = new Map();               // postId → Set<tag>
function addTag(postId, tag) {
  if (!tagsByPost.has(postId)) tagsByPost.set(postId, new Set());
  tagsByPost.get(postId).add(tag);          // duplicates auto-ignored
}
function hasTag(postId, tag) { return tagsByPost.get(postId)?.has(tag) ?? false; }

The LRU cache leans entirely on Map preserving insertion order: the oldest key is always map.keys().next().value (the first inserted), and re-inserting a key on access moves it to newest — a real eviction policy in a dozen lines, impossible to write this cleanly with a plain object (whose key order isn't reliable). The tag index pairs a Map (postId → tags) with Set values so each post's tags are automatically unique and membership checks are O(1). Both are textbook cases where the right collection makes the code short and correct.

Interview Questions

Q1 (beginner): When should you use a Map instead of a plain object?

When keys aren't strings/Symbols (objects, numbers, functions as keys); when you need guaranteed insertion order regardless of key type (objects reorder integer-like keys numerically); when you add and remove entries frequently and want a direct .size and easy iteration; or when keys come from user input and you want to avoid prototype-pollution footguns (a Map has no inherited keys). Use a plain object for fixed-shape records and anything you serialise to JSON, since Maps don't stringify directly.

Q2 (beginner): What are the two most common uses of Set?

Deduplication and fast membership testing. Deduplication: [...new Set(array)] removes duplicate values in one expression. Membership: a Set gives O(1) has checks, so converting an array to a Set before repeatedly testing membership turns O(n)-per-check includes calls into O(1) lookups — a real speedup inside loops. Note Set uses identity for objects, so it dedupes primitives by value but keeps distinct object references.

Q3 (mid): What's the difference between Map and WeakMap, and when is WeakMap the right choice?

Map holds its keys strongly — an entry keeps its key alive, so a Map used as a cache pins every key object forever (a leak unless you manually delete). WeakMap holds keys weakly and requires object keys — if nothing else references a key object, it (and its entry) is garbage-collected automatically. Because entries can vanish unpredictably, WeakMap is not iterable and has no .size. WeakMap is right for associating data with objects you don't control and whose lifetime you don't manage — object-keyed caches, private per-instance state keyed by the instance, metadata on DOM nodes — anywhere you want the association to disappear automatically when the object does, without manual cleanup.

Q4 (mid): What is a Symbol and why would you use one as an object key?

A Symbol is a guaranteed-unique primitive; every Symbol() call produces a distinct value even with the same description. As an object key it's collision-proof — a Symbol key can never clash with a string key or another Symbol — so libraries and frameworks use Symbol keys to attach metadata or internal markers to objects without any risk of overwriting or being overwritten by the user's own properties. Symbol keys are also excluded from Object.keys, for...in, and JSON.stringify (though discoverable via Object.getOwnPropertySymbols), so they stay out of normal enumeration. And well-known symbols like Symbol.iterator let objects hook into language protocols.

Q5 (senior): Explain the iteration protocol via Symbol.iterator. How would you make a custom object work with for...of and spread?

An object is iterable if it has a method keyed by Symbol.iterator that returns an iterator — an object with a next() method returning { value, done } on each call (done: true signals the end). Constructs like for...of, spread ([...x]), array destructuring, Array.from, and Promise.all all consume iterables by calling their [Symbol.iterator]() and pulling values via next() until done. To make a custom object work with them, implement [Symbol.iterator](): e.g. a range object returns an iterator whose next yields successive numbers then { done: true }. In practice you rarely hand-write the next state machine — a generator function (Generators lesson) is an iterator, so *[Symbol.iterator]() { yield* ... } implements the protocol far more concisely. The elegance is uniformity: arrays, strings, Maps, Sets, and your custom types are all iterable by the same contract, so every iterable-consuming construct works on all of them without special-casing — which is exactly why the protocol exists as a Symbol-keyed extension point rather than a hard-coded check.

Q6 (senior): Design a memoisation cache that won't leak memory when the cached keys are objects with limited lifetimes. Discuss WeakMap's guarantees and limits.

Use a WeakMap keyed by the object argument: const cache = new WeakMap(); function memo(obj) { if (!cache.has(obj)) cache.set(obj, compute(obj)); return cache.get(obj); }. The guarantee: because WeakMap holds keys weakly, once an object key is no longer referenced anywhere else in the program, the garbage collector can reclaim both the key and its cached value automatically — so the cache never keeps objects alive past their natural lifetime, eliminating the classic memoisation leak where a strong Map pins every key forever. Limits and caveats: (1) keys must be objects — you can't memoise by primitive arguments (numbers, strings) with a WeakMap; for those you need a Map plus an eviction policy (LRU) to bound growth. (2) You can't enumerate, count, or clear a WeakMap, so you can't implement size limits, TTL, or "clear all" on it — GC timing is non-deterministic and outside your control, so you can't predict when entries disappear. (3) The value is held as long as the key is, so a large cached value stays alive while its key does — fine, but be aware the value's lifetime is coupled to the key's. Design consequence: WeakMap is ideal when the cache key is a long-lived-ish object whose lifetime you want to track rather than control (DOM nodes, model instances, request objects), giving leak-free caching for free. When you need bounded size, primitive keys, TTL, or explicit invalidation, use a Map with an eviction strategy (LRU as in this lesson's build) — possibly combining both: a WeakMap for object-keyed entries and a bounded Map for primitive-keyed ones. The senior point is matching the cache's memory-management model to who controls the key's lifetime: WeakMap when the runtime does, Map+policy when you do.

🏋️ Exercises

1 (easy): Dedupe and count. Given ["a","b","a","c","b","a"], produce the unique values and a count of how many uniques there are.
Solution 
const arr = ["a", "b", "a", "c", "b", "a"];
const set = new Set(arr);
[...set];      // ["a", "b", "c"]
set.size;      // 3

A Set drops duplicates; .size gives the count directly, and spreading recovers the unique array in insertion order.

2 (easy): Object to Map and back. Convert { x: 1, y: 2 } to a Map, add a key 3 → "three", then convert the string-keyed entries back to an object.
Solution 
const map = new Map(Object.entries({ x: 1, y: 2 }));
map.set(3, "three");                       // numeric key — impossible to keep distinct in a plain object
const obj = Object.fromEntries(map);       // { x:1, y:2, "3":"three" } — note 3 stringifies as a key here
map.get(3);                                 // "three" — but in the Map it stays a real number key

The Map keeps 3 as a genuine number key; converting back to an object stringifies it (objects can't have number keys), illustrating exactly why Map exists.

3 (medium): Fast membership. Given a large allowedIds array and a stream of ids to check, rewrite a naive allowedIds.includes(id) loop to be O(1) per check.
Solution 
const allowed = new Set(allowedIds);          // build once — O(n)
for (const id of incoming) {
  if (allowed.has(id)) process(id);           // O(1) per check instead of O(n)
}

Converting the array to a Set once turns each membership test from a linear scan into a constant- time lookup — an O(n·m) loop becomes O(n+m).

4 (medium): Symbol-keyed metadata. Attach a hidden version metadata value to an object using a Symbol so it doesn't appear in Object.keys or JSON.stringify, and show how to read it back.
Solution 
const VERSION = Symbol("version");
const config = { name: "app", [VERSION]: 3 };
Object.keys(config);                      // ["name"] — Symbol key hidden
JSON.stringify(config);                    // '{"name":"app"}' — Symbol key excluded
config[VERSION];                           // 3 — read via the Symbol you hold
Object.getOwnPropertySymbols(config);      // [Symbol(version)] — the way to enumerate them

The Symbol key stays out of normal enumeration and serialisation, so it won't collide with real data or leak into JSON — ideal for library metadata — while remaining readable by anyone holding the Symbol.

5 (hard): Make it iterable. Give a LinkedList object (nodes with value/next) a Symbol.iterator so it works with for...of and spread.
Solution 
class LinkedList {
  #head = null;
  add(value) { this.#head = { value, next: this.#head }; return this; }
  *[Symbol.iterator]() {                    // a generator IS an iterator (Generators lesson)
    let node = this.#head;
    while (node) { yield node.value; node = node.next; }
  }
}
const list = new LinkedList().add(1).add(2).add(3);
[...list];                 // [3, 2, 1]
for (const v of list) { }  // works

Implementing [Symbol.iterator] makes the list a first-class iterable — usable anywhere an iterable is expected. Using a generator method (*[Symbol.iterator]()) avoids hand-writing the next() state machine.

6 (hard): Leak-free private state. Use a WeakMap to store private state for instances of a class (an alternative to #fields) such that the state is inaccessible externally and is collected when the instance is. Explain the GC behaviour.
Solution 
const _private = new WeakMap();            // module-scoped, not exported
class Widget {
  constructor(id) { _private.set(this, { id, clicks: 0 }); }
  click() { const s = _private.get(this); s.clicks++; return s.clicks; }
  get id() { return _private.get(this).id; }
}
let w = new Widget("a");
w.click(); w.click();     // 2
// No external access to _private → state is private
w = null;                 // the instance is now unreferenced...
// ...so its WeakMap entry becomes collectable too — no leak, no manual cleanup.

The WeakMap keyed by this stores each instance's private state; since the WeakMap is module-private and not exported, outside code can't reach it. Crucially, because WeakMap holds the instance key weakly, once the instance is unreferenced (w = null) both the instance and its state entry can be garbage-collected — no manual deletion, no leak. This was the standard private-state pattern before #fields, and it's still useful for adding private state to objects you can't modify with class fields.

🎯 Key Takeaways

  • Map is the real dictionary: any-type keys, true insertion order, .size, direct iteration, no prototype-key hazards — use it for dynamic/untrusted/non-string keys.
  • Set stores unique values with O(1) membership — the go-to for deduping ([...new Set(arr)]) and fast lookups.
  • WeakMap/WeakSet hold keys weakly (object keys only, no iteration/size) — leak-free object-keyed caches and metadata that GC cleans up automatically.
  • Symbol is a unique primitive: collision-proof object keys and enumeration-hidden metadata; Symbol.for shares via a global registry.
  • Well-known symbols (Symbol.iterator, Symbol.toPrimitive) are the language's extension points — implement [Symbol.iterator] to make anything for...of/spread-able.
  • Map/Set don't serialise to JSON — convert with [...map]/Object.fromEntries at the boundary.
