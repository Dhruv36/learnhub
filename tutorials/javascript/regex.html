Regular Expressions Production JavaScript

What you'll master: regex syntax fluency (character classes, quantifiers, anchors, groups), the string/RegExp methods that actually use them, flags and their subtle stateful gotcha, named capture groups for readable extraction, and — critically — knowing when not to reach for a regex. Prerequisites: Strings (methods, UTF-16). Regex is a genuinely different, terser language embedded in JS — this lesson builds real fluency, not just pattern memorization.

Part 1 — What a Regex Is, and the Two Ways to Write One

const re1 = /hello/i;              // LITERAL syntax — preferred, the pattern is fixed at write time
const re2 = new RegExp("hello", "i");   // CONSTRUCTOR — needed when the pattern is built from a variable
const dynamic = new RegExp(`^${escapeRegex(userInput)}$`, "i");   // interpolating user input — MUST escape it (Part 8)

A regex is a pattern for matching text. Use the literal syntax (/pattern/flags) whenever the pattern is known at write time — it's checked at parse time and is the idiomatic default. Use the RegExp constructor only when you must build the pattern dynamically from a string (e.g. incorporating user input) — and if that input isn't itself meant to be a regex, you must escape its special characters first (Part 8), or a user typing .* hijacks your pattern's meaning.

Part 2 — Character Classes & Quantifiers

\d   digit [0-9]        \D  NOT a digit
\w   word char [A-Za-z0-9_]   \W  NOT a word char
\s   whitespace          \S  NOT whitespace
.    ANY character except newline (unless the /s flag is set)
[abc]     any ONE of a, b, c        [^abc]   any char EXCEPT a, b, c
[a-z]     a range                    [a-zA-Z0-9]  combined ranges

*    0 or more    +    1 or more    ?    0 or 1 (optional)
{3}       exactly 3       {2,4}     between 2 and 4      {2,}    2 or more
*?  +?  ??  {2,4}?   — LAZY (non-greedy) versions: match as FEW as possible, not as many

Character classes describe what to match; quantifiers describe how many. Quantifiers are greedy by default — .* tries to match as much as possible, then backtracks if needed. This is a frequent source of surprising matches: /<.*>/ against "<a>text</a>" greedily matches the whole string from the first < to the last >, not just "<a>". Adding ? after a quantifier (.*?) makes it lazy — match as few characters as possible — which is usually what you actually want for "match up to the next delimiter" patterns.

Part 3 — Anchors, Boundaries & Alternation

^     start of string (or line, with /m flag)
$     end of string (or line, with /m flag)
\b    word BOUNDARY — the edge between a word char and a non-word char (zero-width, matches nothing itself)
\B    NOT a word boundary

/^\d+$/          matches ONLY if the ENTIRE string is digits — ^ and $ anchor the whole match
/\bcat\b/        matches "cat" as a WHOLE word — won't match "category" or "concatenate"
/cat/            matches "cat" ANYWHERE, including inside "category" — often NOT what you want

a|b|c            alternation — matches "a" OR "b" OR "c"
/^(yes|no)$/     the entire string must be exactly "yes" or "no"

^ and $ anchor the match to the string's boundaries (or line boundaries with the /m flag) — without them, a pattern matches anywhere in the string, which is the #1 validation bug: /\d+/.test("abc123xyz") is true because it finds digits somewhere, even though the string isn't "just a number." For full- string validation you almost always want ^...$. \b (word boundary) is different from anchors — it's a zero-width assertion matching the transition between a word character and a non-word character, letting you match whole words without anchoring to the whole string.

Part 4 — Groups: Capturing, Non-Capturing & Named

const m = "2024-03-15".match(/(\d{4})-(\d{2})-(\d{2})/);
m[0];   // "2024-03-15" — the FULL match
m[1];   // "2024"        — first capturing group
m[2];   // "03"
m[3];   // "15"

// NAMED groups — far more readable than numbered indices:
const named = "2024-03-15".match(/(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/);
named.groups.year;    // "2024" — self-documenting, order-independent

// NON-capturing group — group for alternation/quantifying WITHOUT creating a numbered capture:
/(?:https?|ftp):\/\/(?<host>[^/]+)/    // (?:...) groups "https?|ftp" but doesn't add to m[1]

Capturing groups ((...)) extract sub-matches, available as match[1], match[2], etc. — but numbered groups get unreadable and fragile fast (insert a new group and every later index shifts). Named groups ((?<name>...)) solve this: access by match.groups.name, self-documenting and immune to reordering — prefer named groups for anything with more than one capture. Non-capturing groups ((?:...)) let you group for alternation or apply a quantifier to a sub-pattern without cluttering the capture-index list — use them whenever you don't need to extract that particular group's content.

Part 5 — The Methods: String vs RegExp, and Which to Use

const re = /\d+/g;
"a1b22c333".match(re);        // ["1","22","333"] — WITH /g: all matches as an array
"a1b".match(/\d/);             // [ "1", index:1, ... ]  — WITHOUT /g: rich info on the FIRST match

re.test("abc");                // true/false — fastest, just "does it match?"
re.exec("a1b22c");             // like match without /g, but STATEFUL with /g (Part 6!)

"a-b-c".split(/-/);             // ["a","b","c"] — split on a pattern, not just a literal string
"cat dog cat".replace(/cat/g, "X");        // "X dog X" — replaceAll needs /g on the regex form
"a1b22".replace(/\d+/g, (match) => `[${match}]`);   // replacer FUNCTION — computed replacement
[..."a1b22c333".matchAll(/\d+/g)];   // richest: full match objects (with groups) for EVERY match
NeedUse
Just yes/no — does it match?regex.test(str) (fastest)
All matches, plain strings onlystr.match(/x/g)
All matches WITH groups/index for each[...str.matchAll(/x/g)]
Replace matchesstr.replace(/x/g, replacement or fn)
Split on a patternstr.split(/x/)

test() is the cheapest existence check. For extracting text, matchAll (which requires the /g flag) is the modern, richest choice — it gives you a full match object (including named groups) for every match, not just strings. replace accepts a function as the replacement, receiving the match (and captured groups) as arguments — essential for computed replacements, not just fixed text substitution.

Part 6 — Flags, and the /g Statefulness Trap

g   global — find ALL matches, not just the first
i   case-insensitive
m   multiline — ^ and $ match line boundaries, not just string boundaries
s   dotAll — . also matches newlines
u   unicode — treat the pattern as a sequence of CODE POINTS, not UTF-16 code units (Strings lesson!)

// ⚠ THE TRAP: a /g regex used with exec() or test() is STATEFUL — it remembers lastIndex between calls:
const re = /\d/g;
re.test("a1b2");   // true  — matched at index 1, re.lastIndex is now 2
re.test("a1b2");   // true  — resumes searching FROM index 2, matches the "2"
re.test("a1b2");   // false — no more digits after index 4; lastIndex RESETS to 0
// Reusing the SAME /g regex object across unrelated calls gives WRONG, inconsistent results!

Flags modify matching behavior — /g (all matches), /i (case-insensitive), and critically /u, which makes the regex operate on Unicode code points instead of raw UTF-16 code units, fixing the same surrogate-pair issues from the Strings lesson (without it, a pattern can split an emoji in half). The statefulness trap is genuinely dangerous: a regex with /g has a lastIndex property that test()/exec() advance and remember between calls on the same regex object — so reusing one /g regex across multiple, unrelated test() calls silently gives wrong results depending on call order. Fix: create a fresh regex literal per use, or don't use /g with test() at all (only needed for match/matchAll/replace).

Part 7 — Lookahead & Lookbehind: Matching Context Without Consuming It

(?=...)     positive lookahead — "followed by," doesn't consume/capture the lookahead text
(?!...)     negative lookahead — "NOT followed by"
(?<=...)    positive lookbehind — "preceded by"
(?<!...)    negative lookbehind — "NOT preceded by"

/\d+(?= dollars)/.exec("50 dollars");    // "50" — matched, but " dollars" wasn't consumed/included
/(?<=\$)\d+/.exec("$50");                // "50" — the "$" was required but not part of the match

// Password strength — combine multiple lookaheads (each an independent condition on the SAME position):
const strong = /^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[^A-Za-z0-9]).{8,}$/;
strong.test("Weak1");        // false — no special character
strong.test("Str0ng!Pass");  // true  — lowercase, uppercase, digit, special char, 8+ length, all satisfied

Lookaround assertions check that a pattern is (or isn't) adjacent, without including that adjacent text in the actual match. This is exactly what you need for "match a number that's followed by a specific word, but only extract the number" — or the classic password strength check: stacking several lookaheads at the same starting position lets you require "contains a lowercase letter AND contains an uppercase letter AND contains a digit AND is 8+ characters" all as independent conditions, without them needing to appear in any particular order.

Part 8 — When NOT to Use Regex

// ❌ Parsing HTML/JSON/nested structures with regex — regex can't count/balance nesting:
/<div>.*<\/div>/    // breaks on nested <div>s; famously "you can't parse HTML with regex"
// ✅ Use a real parser: DOMParser, JSON.parse, a proper HTML/XML library

// ❌ A hand-rolled email regex trying to cover the full spec — always wrong on some edge case
// ✅ Use the platform: <input type="email"> validation, or a maintained library

// ❌ Building a regex from unescaped user input — a security/correctness bug:
new RegExp(userInput);    // if userInput is "(a+)+b" → catastrophic backtracking (ReDoS, Part 9)
// ✅ Escape special characters first:
function escapeRegex(str) { return str.replace(/[.*+?^${}()|[\]\\]/g, "\\$&"); }

Regex is a powerful but limited tool — it matches patterns, not structure, so it fundamentally cannot correctly handle nested/recursive constructs like balanced HTML tags or JSON (a real, well-known engineering maxim: "you can't parse HTML with regex," because regular languages can't count nesting depth). Use a real parser for structured formats. Similarly, resist the urge to hand-roll a "complete" email/URL validator regex — the specs have deep edge cases, and the platform (<input type="email">) or a maintained library already got it right. Finally, never build a regex directly from unescaped user input — beyond it not doing what you expect, it can also be a ReDoS (Regular Expression Denial of Service) vector (Part 9).

Part 9 — Build It: a Robust Input Parser with Named Groups

Let's combine named groups, anchoring, and lookahead into a realistic, readable parser — extracting structured data from a log line — the kind of task regex genuinely excels at (flat, line-oriented text).

const LOG_PATTERN = /^\[(?<timestamp>[\d-]+T[\d:]+)\]\s+(?<level>ERROR|WARN|INFO)\s+(?<service>\w+):\s+(?<message>.+)$/;

function parseLogLine(line) {
  const m = LOG_PATTERN.exec(line);
  if (!m) return null;                              // doesn't match the expected shape — caller decides
  const { timestamp, level, service, message } = m.groups;   // destructure NAMED groups (Destructuring lesson)
  return { timestamp: new Date(timestamp), level, service, message };
}

parseLogLine("[2024-03-15T10:30:00] ERROR auth: invalid token");
// { timestamp: Date, level: "ERROR", service: "auth", message: "invalid token" }
parseLogLine("garbage input");
// null — anchored pattern correctly rejects non-matching lines

// A safe user-input matcher — escape before embedding, avoid catastrophic backtracking:
function escapeRegex(str) { return str.replace(/[.*+?^${}()|[\]\\]/g, "\\$&"); }
function findWholeWord(text, word) {
  return new RegExp(`\\b${escapeRegex(word)}\\b`, "i").test(text);   // safe: no user-controlled repetition
}

The log parser uses named groups for self-documenting extraction (destructured directly, connecting back to the Destructuring lesson), is fully anchored (^...$) so it correctly rejects malformed lines instead of partially matching garbage, and returns null on no match rather than throwing — a clean, checkable contract. findWholeWord demonstrates the safe pattern for embedding user input: escape special characters first, and keep the surrounding pattern free of user-controlled repetition/nesting (avoiding ReDoS — a regex built naively from unescaped, attacker- controlled input with nested quantifiers can take exponential time on certain inputs, a real, documented denial-of-service vector in production systems).

Interview Questions

Q1 (beginner): Why does /\d+/.test("abc123xyz") return true, and how would you require the whole string to be digits?

Without anchors, a regex matches if the pattern is found anywhere in the string — /\d+/ just needs one or more digits somewhere, and "123" inside "abc123xyz" satisfies that. To require the entire string to consist of digits, anchor the pattern to both ends: /^\d+$/ — ^ anchors to the start, $ to the end, so the digits must span the whole string with nothing else before or after. Forgetting anchors on a validation regex is one of the most common regex bugs.

Q2 (beginner): What's the difference between a greedy and a lazy quantifier?

Quantifiers (*, +, {n,m}) are greedy by default — they match as many characters as possible, backtracking only if needed to let the rest of the pattern match. Appending ? makes a quantifier lazy — it matches as few characters as possible, expanding only if needed. The difference is visible with repeated delimiters: greedy /<.*>/ against "<a><b>" matches the whole string (first < to last >), while lazy /<.*?>/ matches just "<a>" (stopping at the first >). Lazy quantifiers are usually what you want for "up to the next delimiter" patterns.

Q3 (mid): What are named capture groups and why prefer them over numbered ones?

Named groups ((?<name>pattern)) let you extract a sub-match by a descriptive name via match.groups.name, instead of a numeric index like match[1]. They're preferable for anything beyond a single capture because numbered groups are fragile and unreadable: inserting a new group anywhere earlier in the pattern shifts every subsequent index, silently breaking code that references them by number, and match[3] tells a reader nothing about what it contains. Named groups are self-documenting, immune to reordering, and destructure cleanly (const { year, month, day } = match.groups) — the regex equivalent of preferring named function parameters over positional ones for anything non-trivial.

Q4 (mid): Explain the statefulness bug with a global (/g) regex and test()/exec().

A regex literal with the /g flag carries a lastIndex property that exec() and test() update after each call — they resume searching from lastIndex rather than from the start of the string, and reset it to 0 only once a call finds no further match. This is by design for iterating all matches with repeated exec() calls, but it becomes a bug when the same regex object is reused for unrelated calls: the second test() call doesn't start fresh — it continues from wherever the previous call left off, producing wrong, order-dependent true/false results, especially insidious because it often works correctly in quick manual testing (first call) and only misbehaves on the second/later call in a loop. Fix: create a new regex literal for each independent use (regex literals in a loop are cheap and get a fresh, unshared object each iteration in modern engines), or simply don't add /g to a regex you're only using with test() — it's not needed for a single existence check.

Q5 (senior): What is catastrophic backtracking (ReDoS), and how do you write patterns that avoid it?

Catastrophic backtracking happens when a regex pattern has ambiguous nested/adjacent quantifiers — e.g. (a+)+b or (a|a)*b — where the engine's backtracking search has to try an exponential number of ways to partition the matched characters among the nested repetitions before concluding there's no match. For a pathological but plausible input (many repeated 'a's with no trailing 'b'), the match attempt can take exponential time relative to input length — seconds, then minutes, then effectively forever as the input grows by just a few characters. This becomes a genuine Regular Expression Denial of Service (ReDoS) vulnerability when the pattern is applied to untrusted input (a validation regex on a form field, a URL parser) — an attacker submits a crafted string and hangs the server thread processing it (this has caused real production outages, including a well-known Cloudflare incident). Avoidance: (1) avoid nested quantifiers over the same characters — (a+)+ is the classic red flag; if you need "one or more groups of one-or-more a's," simplify to just a+ if the nesting adds no real distinguishing power; (2) avoid overlapping alternation inside a repeated group ((a|ab)* — both branches can match "a", creating ambiguity); (3) prefer possessive/atomic patterns or explicit character classes over broad, overlapping quantified alternatives; (4) use a regex linter/analyzer (safe-regex, recheck) in CI to catch dangerous patterns before they ship; (5) for untrusted input specifically, consider a timeout/worker-based execution boundary, or use engines with linear-time guarantees (RE2-based libraries) that refuse pathological patterns by construction rather than backtracking. The senior instinct: treat any regex applied to untrusted input as a potential attack surface, not just a correctness concern, and specifically scrutinize nested or overlapping quantifiers.

Q6 (senior): Design a validation strategy for structured user input (e.g. usernames, slugs, phone numbers) that balances regex use with its limitations.

The strategy layers regex for what it's good at with other tools for what it isn't. (1) Use regex for flat, character-level format validation — a username matching /^[a-zA-Z0-9_]{3,20}$/, a slug matching /^[a-z0-9]+(-[a-z0-9]+)*$/ — these are exactly regex's strength: constrained character sets, fixed structure, no nesting. Keep these patterns simple, anchored, and free of nested/overlapping quantifiers to avoid ReDoS (previous question). (2) Don't over-engineer regex for genuinely complex formats — full email validation and phone number formats have deep, non-regular edge cases (internationalization, RFC corner cases); prefer the platform (<input type="email">, type="tel" with a pattern hint) combined with a light regex sanity check, or a maintained library (libphonenumber for phone numbers) rather than a hand-rolled "complete" regex that's both wrong and a maintenance burden. (3) Escape any user input embedded into a dynamically-built regex (e.g. a "does this text contain the search term" feature) to prevent both incorrect matching and ReDoS from attacker-supplied pattern fragments. (4) Validate at the boundary, both client and server — client-side regex for instant UX feedback (Forms lesson), but the server independently re-validates with the same or a canonical rule, since client validation is bypassable. (5) Prefer semantic validation over pure format matching where it matters — a slug regex confirms character-set correctness but not uniqueness or reserved-word conflicts, which need a database check, not a bigger regex. (6) Test with a regex analyzer/linter in CI for catastrophic-backtracking risk on any pattern applied to user input, and add unit tests for known edge cases (empty string, max length, Unicode input with /u where relevant). The overall principle: regex owns the "is this character sequence shaped correctly" layer; parsers, platform validators, libraries, and database checks own everything regex structurally cannot express (nesting, semantic uniqueness, full-spec correctness) — and every regex touching user input gets escaped/anchored/reviewed for backtracking risk as a matter of course, not as an afterthought.

🏋️ Exercises

1 (easy): Anchor a validator. Write a regex that matches a string consisting ENTIRELY of 5 digits (a US zip code), rejecting "1234" and "abc12345".
Solution 
const zip = /^\d{5}$/;
zip.test("12345");    // true
zip.test("1234");     // false — only 4 digits
zip.test("abc12345"); // false — anchors reject the extra leading text

^ and $ require the digits to span the entire string; {5} requires exactly five.

2 (easy): Extract with named groups. Given "John:25", extract the name and age using named capture groups.
Solution 
const m = "John:25".match(/^(?<name>\w+):(?<age>\d+)$/);
const { name, age } = m.groups;   // "John", "25"

Named groups let you destructure the match result directly by meaningful names instead of numeric indices.

3 (medium): Lazy vs greedy. Given "<b>bold</b> and <i>italic</i>", write a regex extracting each individual tag pair's content, and explain why greedy would fail here.
Solution 
const html = "<b>bold</b> and <i>italic</i>";
[...html.matchAll(/<(\w+)>(.*?)<\/\1>/g)].map(m => m[2]);   // ["bold", "italic"]

The lazy .*? stops at the next closing tag; a greedy .* would match from the first <b> all the way to the last closing tag, wrongly capturing "bold</b> and <i>italic" as one blob. The \1 backreference matches whatever tag name group 1 captured, correctly pairing open and close tags.

4 (medium): Password strength. Write a regex requiring at least one lowercase, one uppercase, one digit, and a minimum length of 8, using lookaheads.
Solution 
const strong = /^(?=.*[a-z])(?=.*[A-Z])(?=.*\d).{8,}$/;
strong.test("weak");        // false — too short, no uppercase/digit
strong.test("Str0ngPass");  // true

Each lookahead independently asserts a condition exists somewhere in the string without consuming characters, so all three can apply starting from position 0, followed by the actual length requirement.

5 (hard): Safe search. Write escapeRegex(str) and a highlightMatches(text, userQuery) function that safely wraps every occurrence of a user-supplied search term in <mark>, even if the query contains regex special characters.
Solution 
function escapeRegex(str) { return str.replace(/[.*+?^${}()|[\]\\]/g, "\\$&"); }
function highlightMatches(text, userQuery) {
  const pattern = new RegExp(escapeRegex(userQuery), "gi");
  return text.replace(pattern, (match) => `<mark>${match}</mark>`);
}
highlightMatches("Price: $5.00 (was $10)", "$5.00");
// "Price: <mark>$5.00</mark> (was $10)" — the literal "$5.00", not a broken regex

Without escaping, a query like "$5.00" would be interpreted as regex syntax ($ as an anchor, . as any-char) and match incorrectly or throw. escapeRegex neutralizes every special character so the query is matched literally.

6 (hard): Parse structured log lines. Extend the lesson's log parser to also capture an optional trailing JSON payload (e.g. "... message {"userId":42}"), parsing it into an object when present, or null when absent.
Solution 
const LOG_PATTERN = /^\[(?<timestamp>[\d-]+T[\d:]+)\]\s+(?<level>ERROR|WARN|INFO)\s+(?<service>\w+):\s+(?<message>[^{]+?)\s*(?<payload>\{.*\})?$/;

function parseLogLine(line) {
  const m = LOG_PATTERN.exec(line);
  if (!m) return null;
  const { timestamp, level, service, message, payload } = m.groups;
  return {
    timestamp: new Date(timestamp), level, service,
    message: message.trim(),
    payload: payload ? JSON.parse(payload) : null,   // optional group is undefined if absent
  };
}
parseLogLine('[2024-03-15T10:30:00] INFO auth: login {"userId":42}');
// { ..., message: "login", payload: { userId: 42 } }
parseLogLine("[2024-03-15T10:30:00] INFO auth: login");
// { ..., message: "login", payload: null }

Making the payload group itself optional (via the trailing ? on the whole capturing group) means it's undefined in match.groups when absent, which the ternary converts cleanly to null; the message group is changed to [^{]+? (lazy, excluding {) so it doesn't greedily swallow the JSON part.

🎯 Key Takeaways

  • Prefer regex literals; anchor (^...$) full-string validation — without anchors, a pattern matches anywhere.
  • Quantifiers are greedy by default; use *?/+? for lazy "up to the next delimiter" matching.
  • Named groups ((?<name>...)) beat numbered ones for readability and reorder-safety; non-capturing groups ((?:...)) group without adding an index.
  • A /g regex is stateful across test()/exec() calls (lastIndex) — reusing one across unrelated calls gives wrong results; use fresh literals.
  • Lookahead/lookbehind assert adjacent context without consuming it — the tool for password-strength-style multi-condition checks.
  • Regex can't handle nested structures (HTML/JSON — use a real parser) and must never embed unescaped user input (wrong matches, or ReDoS via catastrophic backtracking).
