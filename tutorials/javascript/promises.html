Promises Asynchronous JavaScript

What you'll master: what a promise is (a value that will exist later), its three states, chaining with .then/.catch/.finally and why chaining beats nesting, how errors propagate down a chain, how to create promises, and the classic mistakes (forgotten returns, swallowed errors, the "callback hell" this all replaced). Prerequisites: The Event Loop (promise callbacks are microtasks) and Functions. Promises are the foundation async/await is built on — this lesson is where async fluency starts.

Part 1 — What a Promise Is: the Three States

const p = fetch("/api/user");    // returns a Promise immediately — the value isn't here YET
// A promise is in ONE of three states:
//   pending   — not settled yet (the operation is in flight)
//   fulfilled — settled successfully, with a value
//   rejected  — settled with an error (reason)
// Once settled (fulfilled OR rejected), it's IMMUTABLE — it never changes state again.

A promise is an object representing the eventual result of an asynchronous operation — a placeholder for a value that will exist later (or an error that will explain why it won't). It starts pending and eventually settles either as fulfilled (with a value) or rejected (with a reason). The critical property: settling is one-way and permanent — a promise transitions from pending to settled exactly once and then is frozen, so its handlers run at most once. This immutability is what makes promises composable and predictable, unlike raw callbacks that a buggy API might invoke twice or never.

Part 2 — Consuming: .then, .catch, .finally

fetch("/api/user")
  .then(response => response.json())     // runs on fulfilment; returns a NEW promise
  .then(user => console.log(user.name))  // receives the resolved value of the previous .then
  .catch(error => console.error(error))  // runs if ANY step above rejects/throws
  .finally(() => hideSpinner());          // runs regardless of outcome — cleanup

.then(onFulfilled) registers a callback for success and — crucially — returns a new promise, which is what makes chaining work. .catch(onError) handles rejection (it's shorthand for .then(null, onError)), and .finally(fn) runs on both success and failure for cleanup (hiding a spinner, closing a connection) without altering the value. All three callbacks run as microtasks (Event Loop lesson), so they execute after the current synchronous code, in order.

Part 3 — Chaining: Each .then Returns a New Promise

Promise.resolve(1)
  .then(n => n + 1)          // returns 2 → next .then receives 2
  .then(n => n * 10)         // returns 20
  .then(n => Promise.resolve(n + 5))  // returning a PROMISE → chain WAITS for it, unwraps to 25
  .then(n => console.log(n));// 25

The magic of chaining: whatever a .then callback returns becomes the resolved value passed to the next .then. If it returns a plain value, that value flows on. If it returns another promise, the chain waits for that promise to settle and unwraps it — this "flattening" is why you can sequence async steps linearly instead of nesting them. This single rule (return a value → next step gets it; return a promise → chain awaits it) is the entire mental model of promise chains, and forgetting to return is the #1 promise bug (Part 7).

Part 4 — Chaining Beats Nesting (Escaping Callback Hell)

// ❌ Callback hell — the pyramid of doom promises were invented to kill:
getUser(id, (user) => {
  getPosts(user, (posts) => {
    getComments(posts[0], (comments) => {
      // 3 levels deep, error handling duplicated at every level, hard to read
    });
  });
});

// ✅ Flat promise chain — sequential, single error handler:
getUser(id)
  .then(user => getPosts(user))
  .then(posts => getComments(posts[0]))
  .then(comments => render(comments))
  .catch(handleError);        // ONE handler covers every step

Before promises, sequential async steps meant deeply nested callbacks — the "pyramid of doom" — where each level added indentation and had to handle its own errors. Promise chaining flattens this into a linear sequence: each step returns the promise for the next, so they read top-to- bottom, and a single .catch at the end handles a failure in any step. Returning the inner promise (return getPosts(user)) is what keeps the chain flat — that's the flattening rule from Part 3. This readability and unified error handling is the entire reason promises exist.

Part 5 — Error Propagation Down the Chain

Promise.resolve()
  .then(() => { throw new Error("boom"); })   // a throw (or a rejected promise) skips...
  .then(() => console.log("skipped"))          // ...all subsequent .then callbacks...
  .then(() => console.log("also skipped"))
  .catch(err => console.log("caught:", err.message))  // ...until the first .catch — "caught: boom"
  .then(() => console.log("chain continues"));  // after a handled catch, the chain RESUMES

Errors propagate like exceptions: a thrown error or rejected promise skips all following .then success-callbacks and jumps to the next .catch (or rejection handler). This is why one .catch at the end can cover a whole chain — any step's failure lands there. And after a .catch handles an error, the chain recovers: the promise the .catch returns is fulfilled (with whatever the catch returns, or undefined), so subsequent .thens run again. This lets you catch, recover, and continue — or re-throw from the catch to keep the chain rejected. It's structured, exception-like error handling for async code.

Part 6 — Creating Promises

// Wrap a callback-based API in a promise (the executor runs synchronously):
function delay(ms) {
  return new Promise((resolve, reject) => {   // resolve = fulfil, reject = fail
    if (ms < 0) reject(new Error("negative delay"));
    setTimeout(() => resolve(`waited ${ms}ms`), ms);
  });
}
delay(100).then(msg => console.log(msg));

// Shorthand creators:
Promise.resolve(42);                  // an already-fulfilled promise
Promise.reject(new Error("nope"));    // an already-rejected promise

// "Promisify" a Node-style callback API:
const readFileP = (path) => new Promise((res, rej) =>
  fs.readFile(path, (err, data) => err ? rej(err) : res(data)));  // or util.promisify

You mostly consume promises returned by APIs (fetch, DB clients), but sometimes you must create one — usually to wrap an older callback-based API. new Promise((resolve, reject) => {...}) takes an executor that runs immediately; call resolve(value) to fulfil or reject(error) to fail. The rules: resolve/ reject can each be called once (later calls are ignored — the settle-once guarantee), and an exception thrown in the executor automatically rejects. Promise.resolve/reject are shortcuts for already-settled promises. Node's util.promisify automates the callback- wrapping pattern.

🧠 The new Promise anti-pattern: don't wrap something that's already a promise. return new Promise(res => fetch(url).then(res)) is redundant and drops errors — just return fetch(url). Only use new Promise to bridge a genuinely non-promise (callback/event) API.

Part 7 — The Classic Promise Mistakes

// 1. FORGOTTEN RETURN — breaks the chain's sequencing:
getUser(id)
  .then(user => { getPosts(user); })   // ❌ no return → next .then gets undefined, doesn't wait
  .then(posts => posts.length);         // 💥 posts is undefined

// 2. SWALLOWED ERRORS — no .catch anywhere:
doAsyncThing();                          // ❌ if it rejects, "unhandled promise rejection" (crash in Node)

// 3. NESTING instead of chaining (recreating the pyramid):
getUser(id).then(user => {
  return getPosts(user).then(posts => { /* ❌ nested — flatten by returning */ });
});

// 4. Not returning in a .catch when you mean to recover vs propagate.

Four mistakes to burn into memory. (1) Forgetting return inside a .then means the chain doesn't wait for the inner async operation and the next step gets undefined — the single most common promise bug. (2) No .catch leaves rejections unhandled (a warning in browsers, a process crash in modern Node) — every chain needs error handling. (3) Nesting instead of returning the inner promise rebuilds the callback pyramid promises were meant to eliminate. Getting these right is most of what "using promises well" means.

Part 8 — Build It: a Retry-with-Backoff Wrapper

Let's compose creation, chaining, and error propagation into a genuinely useful utility: retry a promise-returning function a few times with exponential backoff before giving up — the backbone of resilient network code.

function retry(fn, { retries = 3, baseDelay = 200 } = {}) {
  return new Promise((resolve, reject) => {
    function attempt(remaining, delayMs) {
      fn()
        .then(resolve)                              // success → resolve the outer promise, done
        .catch(err => {
          if (remaining <= 0) return reject(err);    // out of tries → propagate the last error
          setTimeout(() => attempt(remaining - 1, delayMs * 2), delayMs);  // wait, then retry (backoff)
        });
    }
    attempt(retries, baseDelay);
  });
}

// Usage: retry a flaky fetch up to 3 times, 200ms → 400ms → 800ms:
retry(() => fetch("/api/data").then(r => {
  if (!r.ok) throw new Error(`HTTP ${r.status}`);   // fetch only rejects on network errors — check r.ok!
  return r.json();
}))
  .then(data => console.log("got", data))
  .catch(err => console.error("gave up:", err.message));

Every concept is here: new Promise wraps the retry loop into a single promise the caller chains on; each attempt uses .then/.catch to branch on success/failure; error propagation carries the last error to reject when retries are exhausted; and the exponential backoff (delayMs * 2) spaces out attempts. Note the crucial fetch detail flagged in the usage: fetch only rejects on network failure, not on HTTP error status — a 404 or 500 still fulfils, so you must check response.ok and throw yourself (the Fetch lesson elaborates). This retry wrapper is something you'll genuinely reuse.

Interview Questions

Q1 (beginner): What is a promise and what are its states?

A promise is an object representing the eventual result of an asynchronous operation — a placeholder for a value (or error) that will be available later. It has three states: pending (not yet settled), fulfilled (settled successfully with a value), and rejected (settled with an error). It starts pending and transitions to fulfilled or rejected exactly once; once settled it's immutable and never changes again, so its handlers run at most once. You consume it with .then (success), .catch (failure), and .finally (cleanup).

Q2 (beginner): How does promise chaining work?

Each .then returns a new promise, and whatever its callback returns becomes the resolved value passed to the next .then. If the callback returns a plain value, that value flows on; if it returns another promise, the chain waits for that promise to settle and passes its unwrapped value along. This lets you sequence asynchronous steps linearly — each .then handling one stage — instead of nesting callbacks, with a single .catch at the end catching a failure in any step.

Q3 (mid): Explain error propagation in a promise chain and how recovery works.

A thrown error or a returned/awaited rejected promise causes the chain to skip all subsequent .then success callbacks and jump to the nearest rejection handler (.catch or the second argument of .then) — behaving like exceptions bubbling to a catch block. This is why one .catch at the end handles failures from any earlier step. After a .catch handles the error, the promise it returns is fulfilled (with whatever the catch returns, or undefined), so the chain recovers and following .thens run again. To keep the chain rejected instead of recovering, re-throw inside the .catch (or return a rejected promise). So a .catch can either recover-and-continue or handle-and-propagate depending on whether it returns normally or throws.

Q4 (mid): What's the most common promise bug and how do you avoid it?

Forgetting to return inside a .then callback. If you write .then(user => { getPosts(user); }) without return, the chain doesn't wait for getPosts and the next .then receives undefined instead of the posts — a race/undefined bug that's easy to miss because the code "looks" sequential. Avoid it by always returning the value or promise from every .then (arrow functions with implicit return help: .then(user => getPosts(user))). A close second is omitting .catch, leaving rejections unhandled. Both are caught by linters (e.g. the promise/always-return rule) and by using async/await with try/catch, where the sequencing and error handling are more visually obvious.

Q5 (senior): Explain the difference between Promise.resolve(p) when p is a promise vs a value, and why "thenable" assimilation matters.

Promise.resolve(value) for a non-promise wraps it in an already-fulfilled promise. But if you pass it something that's already a promise, it returns that same promise unchanged (no double-wrapping). More generally, promises perform thenable assimilation: whenever a promise would resolve to an object that has a .then method (a "thenable"), it doesn't fulfil with that object — it calls the object's .then and adopts its eventual state, effectively flattening it. This is why returning a promise from a .then makes the chain wait for it (Part 3's flattening), and why you can never have a promise that fulfils with another promise as its value — it always unwraps. It matters for interoperability (different promise libraries and native promises compose because they all follow the thenable contract) and for a subtle gotcha: if you accidentally resolve with an object that happens to have a then method that isn't a real promise, the runtime will still try to call it, which can hang or misbehave — so "thenable" is a structural, duck-typed contract, and objects with an incidental then property can surprise you. Assimilation is the mechanism that makes the whole promise ecosystem composable and the flattening behaviour consistent.

Q6 (senior): Design error handling for a multi-step promise pipeline where some failures are recoverable and others fatal. How do you distinguish and route them?

The key is that a .catch can either recover (return a normal value → chain continues fulfilled) or propagate (re-throw → chain stays rejected), so you route by placing catches strategically and by typing errors. Strategy: (1) Define distinct error types (e.g. classes RecoverableError vs FatalError, or errors with a .retryable/.code field — the Error Handling lesson covers custom errors) so a handler can decide by inspecting the error. (2) Put local .catches right after steps that can recover, handling only their recoverable cases and re-throwing anything else: .then(fetchOptional).catch(err => { if (err.code === "NOT_FOUND") return defaultValue; throw err; }) — this recovers a missing- optional-resource case inline while letting real failures bubble. (3) Keep a terminal .catch at the end for fatal errors, which logs, alerts, and surfaces a user-facing failure. (4) For steps that must not fail the whole pipeline (best-effort logging, analytics), catch and swallow them explicitly with a comment, so the swallow is intentional, not accidental. (5) Use .finally for cleanup that must run regardless (release locks, hide spinners). The result is a pipeline where each step declares its own recovery policy via a local catch, unhandled/ unexpected errors propagate to a single terminal handler, and error type (not just presence) drives routing — the async equivalent of catching specific exception types and re-throwing the rest. In practice this is often clearer with async/await and typed try/catch blocks, but the underlying model — recover by returning, propagate by throwing, route by error type — is identical.

🏋️ Exercises

1 (easy): delay(). Write delay(ms) returning a promise that resolves after ms milliseconds, then chain it to log "done" after 500ms.
Solution 
const delay = ms => new Promise(resolve => setTimeout(resolve, ms));
delay(500).then(() => console.log("done"));

The executor calls resolve after the timer; the returned promise settles then, and .then runs the continuation.

2 (easy): Fix the missing return. This logs undefined. Fix it: 
Promise.resolve(2)
  .then(n => { Promise.resolve(n * 2); })
  .then(n => console.log(n));
Solution 
Promise.resolve(2)
  .then(n => Promise.resolve(n * 2))   // return it (arrow implicit return)
  .then(n => console.log(n));           // 4

The original didn't return the inner promise, so the second .then received undefined. Returning it (here via an arrow's implicit return) lets the chain wait for and unwrap the value.

3 (medium): Sequential chain. Given getUser(id) and getOrders(user) (both promise-returning), fetch a user then their orders and log the order count, with a single error handler.
Solution 
getUser(id)
  .then(user => getOrders(user))     // return the inner promise → chain waits
  .then(orders => console.log(orders.length))
  .catch(err => console.error("failed:", err));

Each step returns the next promise, flattening the sequence; one .catch handles a failure in either fetch.

4 (medium): Recover in catch. Fetch a config, but if it fails, recover with a default config and continue the chain to log the theme. Show that the chain continues after the catch.
Solution 
fetchConfig()
  .catch(() => ({ theme: "light" }))   // recover: return a default → chain continues fulfilled
  .then(config => console.log("theme:", config.theme));  // runs whether fetch succeeded or was recovered

Because the .catch returns a value (rather than re-throwing), the promise it produces is fulfilled, so the following .then runs with the recovered config — demonstrating post-catch recovery.

5 (hard): Promisify. Convert a Node-style callback function fn(arg, (err, result) => ...) into one returning a promise, generically.
Solution 
function promisify(fn) {
  return (...args) => new Promise((resolve, reject) => {
    fn(...args, (err, result) => err ? reject(err) : resolve(result));
  });
}
const readFileP = promisify(fs.readFile);
readFileP("./data.txt").then(data => console.log(data.toString())).catch(console.error);

The returned function forwards its arguments plus an appended Node-style callback that rejects on err and resolves on result — the exact contract Node's util.promisify implements.

6 (hard): Timeout wrapper. Write withTimeout(promise, ms) that rejects if promise doesn't settle within ms, otherwise resolves/rejects as it does. (Hint: race it against a timeout promise.)
Solution 
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`timed out after ${ms}ms`)), ms));
  return Promise.race([promise, timeout]);   // whichever settles first wins
}
withTimeout(fetch("/slow"), 3000)
  .then(r => console.log("in time"))
  .catch(err => console.error(err.message));  // "timed out after 3000ms" if too slow

Promise.race (Promise Combinators lesson) settles as soon as the first of its promises settles — so if the timeout rejects before promise settles, the whole thing rejects with the timeout error; otherwise promise's result wins. A clean, composable timeout with no changes to the wrapped operation. (Note the original promise keeps running in the background — an AbortController is needed to actually cancel it, covered in the Fetch lesson.)

🎯 Key Takeaways

  • A promise is a placeholder for a future value with three states — pending, fulfilled, rejected — and settling is one-way and permanent (handlers run at most once).
  • .then (success) returns a new promise enabling chaining; .catch (failure) and .finally (cleanup) complete the trio — all run as microtasks.
  • Chaining rule: return a value → next .then gets it; return a promise → the chain waits and unwraps it (the flattening that escapes callback hell).
  • Errors propagate like exceptions to the next .catch; after a catch the chain recovers unless you re-throw.
  • Create promises with new Promise((resolve, reject) => ...) only to wrap non-promise APIs; never wrap an existing promise.
  • Top bugs: forgotten return, missing .catch, nesting instead of chaining, and assuming fetch rejects on HTTP error status.
