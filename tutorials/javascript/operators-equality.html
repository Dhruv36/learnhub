Operators, Equality & Truthiness Language Core

What you'll master: the exact difference between == and === (and the one sanctioned use of ==), the complete list of falsy values and how truthiness drives conditions, the modern operator trio ?? / ?. / || and the real bug class that separates them, short-circuit evaluation as control flow, and the comparison/ logical/assignment operators you'll use every day. Prerequisite: The Type System & Coercion — equality and truthiness are coercion applied, so that lesson's model is the toolkit here.

Part 1 — === Strict Equality: the Default

1 === 1          // true
1 === "1"        // false — different types, no coercion; === compares TYPE then value
null === null    // true
NaN === NaN      // false — NaN's defining weirdness (Types lesson)
{a:1} === {a:1}  // false — objects compare by REFERENCE, not shape

=== (strict equality) returns true only when both operands have the same type and the same value — no coercion, ever. It has exactly two edge cases: NaN === NaN is false, and -0 === 0 is true (occasionally you want them distinguished). For objects, "same value" means same reference — two distinct objects with identical contents are never ===, which is why deep equality needs a helper the language doesn't provide. === is your default in all professional code, enforced by the ESLint eqeqeq rule.

Part 2 — == Loose Equality: Why It's Banned (and Its One Good Use)

1 == "1"           // true  — coerces the string to a number
0 == ""            // true  — both coerce to 0
0 == false         // true  — false → 0
"" == false        // true  — both → 0
null == undefined  // true  — SPECIAL-CASED: they equal each other and nothing else
null == 0          // false — null does NOT coerce to 0 for ==  (but DOES for <, a real inconsistency)
[] == ![]          // true  — the infamous one: ![] is false, [] == false → "" == 0 → 0 == 0

== (loose equality) coerces operands to a common type before comparing, following a set of rules convoluted enough that no one reliably keeps them in their head — which is precisely why it's banned by default. The chains it produces (0 == "" == false, [] == ![]) aren't puzzles worth solving; they're a warning. The single sanctioned use of == is x == null, which — because null == undefined is the one useful coercion — checks "is this null OR undefined?" in one expression:

if (x == null) { }    // true for null AND undefined, false for everything else (even 0, "", false)
// equivalent strict form: if (x === null || x === undefined)

That one idiom is the exception the eqeqeq rule explicitly allows ("eqeqeq": ["error", "smart"]). Everywhere else: ===.

Part 3 — Object Equality Is Reference Equality

const a = { x: 1 };
const b = { x: 1 };
const c = a;
a === b   // false — different objects, same shape
a === c   // true  — same reference (c is an alias, from the Variables lesson)

There is no built-in value/deep equality for objects. Comparing by contents requires either a manual recursive walk, a library (lodash.isEqual), or the pragmatic-but-flawed JSON.stringify(a) === JSON.stringify(b) — flawed because it depends on key order, drops undefined/functions, throws on BigInt and cyclic objects, and mangles special numbers. Know the trade-offs; reach for a tested library when correctness matters.

Part 4 — Object.is: === With Its Edge Cases Fixed

Object.is(NaN, NaN)   // true   — fixes ===' first edge case
Object.is(-0, 0)      // false  — fixes ===' second edge case
Object.is(1, 1)       // true   — otherwise identical to ===
Object.is({}, {})     // false  — still reference equality for objects

Object.is is the "SameValue" algorithm: exactly === except it treats NaN as equal to itself and -0 as distinct from +0. You reach for it rarely — mainly when you specifically need NaN-safe comparison or to detect negative zero. The related "SameValueZero" (used by includes, Map/Set keys) is the same but treats -0 and +0 as equal — which is why [NaN].includes(NaN) is true.

Part 5 — Truthiness and the Eight Falsy Values

// The complete falsy list — memorise it; EVERYTHING else is truthy:
false      0      -0      0n      ""      null      undefined      NaN

// The traps — all TRUTHY:
"0"   "false"   " "   []   {}   function(){}

In any boolean context (if, while, ? :, &&, ||, !), a value is coerced to boolean by this rule. There are exactly eight falsy values; commit them to memory, because "everything not in that list is truthy" is then a complete, reliable model. The traps that catch people: the strings "0" and "false" are truthy (non-empty strings), and empty arrays and empty objects are truthy — if ([]) runs its block. That last one is a classic bug: to test an array for emptiness you check arr.length === 0, never if (!arr).

⚠️ if (arr) does not check emptiness. Every array and object is truthy regardless of contents. Empty array → arr.length === 0. Empty object → Object.keys(obj).length === 0. A truthiness check on a reference only tells you it's not null/undefined.

Part 6 — The Modern Trio: || vs ?? vs ?.

const a = userValue || "default";   // || : fall back for ANY falsy value
const b = userValue ?? "default";   // ?? : fall back ONLY for null/undefined
const c = user?.address?.city;      // ?. : short-circuit to undefined if a link is null/undefined
user.save?.();                       // optional CALL — invokes only if save exists

This trio is where a genuine, common bug lives:

const volume = settings.volume || 50;   // ❌ user set volume to 0 → becomes 50!
const volume = settings.volume ?? 50;   // ✅ 0 is kept; only null/undefined → 50

|| returns the right side for any falsy left side — so a legitimate 0, "", or false gets clobbered by the default. ?? (nullish coalescing) exists precisely to fix this: it falls back only for null/undefined, preserving falsy-but-valid values. Rule: use ?? for defaults; reserve || for genuine boolean logic where you actually mean "any falsy."

?. optional chaining — and when NOT to use it

const city = user?.address?.city;    // undefined if user OR address is nullish — no TypeError
const first = arr?.[0];               // optional index access
const result = obj.method?.();        // optional call — skips if method absent

?. short-circuits the entire chain to undefined the moment any link is null/undefined, replacing defensive user && user.address && ... ladders. But don't scatter it defensively: if user.address is supposed to exist, an early crash is better than silently computing with undefined and failing far away. Use ?. for genuine optionality (data that legitimately may be absent), not to mute errors you'd rather see loudly (the Error Handling lesson develops this).

🧠 Combining ?? and || requires parentheses. a || b ?? c is a SyntaxError — the language forces you to disambiguate: (a || b) ?? c or a || (b ?? c). A deliberate guard against the ambiguity itself.

Part 7 — Short-Circuit Evaluation as Control Flow

a && b    // evaluates a; if a is falsy, returns a WITHOUT evaluating b; else returns b
a || b    // evaluates a; if a is truthy, returns a WITHOUT evaluating b; else returns b

&& and || don't return booleans — they return one of their operands, and they stop early ("short-circuit") as soon as the result is determined. That gives three idioms you'll read constantly:

isLoggedIn && renderDashboard();          // guard: run the right side only if left is truthy
const name = input || "Anonymous";          // default (any-falsy variant)
config.retries || (config.retries = 3);     // lazy init (though ??= is cleaner — Part 8)
// In JSX/templates: {items.length && <List/>}  ← trap: length 0 renders "0"! use items.length > 0

The right-side of && only runs when the left is truthy — that's not just an optimisation, it's how you write conditional side-effects and guards inline. The JSX trap is worth flagging: {count && <X/>} renders the number 0 when count is 0 (React renders 0 but not false), so use an explicit count > 0 && ... — a direct, real consequence of "&& returns the operand, not a boolean."

Part 8 — Logical Assignment & the Operators You'll Use Daily

x ??= 3;    // assign 3 only if x is null/undefined     (x = x ?? 3, but short-circuits)
x ||= 3;    // assign 3 only if x is falsy
x &&= 3;    // assign 3 only if x is truthy
x += 1; x -= 1; x *= 2; x /= 2; x %= 3; x **= 2;   // arithmetic assignment
a < b, a <= b, a > b, a >= b   // comparison — coerce to number (except two strings → lexicographic)
a ** b       // exponentiation (2 ** 10 === 1024) — right-associative

The logical assignment operators (??=, ||=, &&=) combine a logical op with assignment and — crucially — short-circuit: x ??= expensive() only calls expensive() when x is nullish, unlike x = x ?? expensive() which always evaluates it. Comparison operators coerce to number, with one special case: two strings compare lexicographically by UTF-16 code unit ("b" > "a" is true; "10" < "9" is true — string order, not numeric!), which is a frequent sorting bug covered in Arrays.

Part 9 — Build It: a Safe Config Reader

Let's compose the trio and truthiness rules into a small utility that reads nested config with correct default semantics — the kind of thing every app has, usually with subtle || bugs baked in.

function readConfig(config) {
  return {
    // ?? preserves 0 and false (a volume of 0, a flag explicitly set false):
    volume:   config?.audio?.volume ?? 100,
    muted:    config?.audio?.muted  ?? false,
    // || is correct HERE — an empty theme string is not a valid choice, so fall back:
    theme:    config?.ui?.theme || "light",
    // optional call with a safe default:
    onReady:  config?.onReady ?? (() => {}),
  };
}

readConfig({ audio: { volume: 0, muted: false } });
// { volume: 0, muted: false, theme: "light", onReady: fn }  ← 0 and false SURVIVED

Notice the deliberate mix: ?? where 0/false are legitimate values that must survive, but || for theme where an empty string should fall back because it's not a real choice. Choosing per-field between ?? and || based on "is a falsy value legitimate here?" is exactly the judgment this lesson trains — the operators are trivial; knowing which one each situation wants is the skill.

Interview Questions

Q1 (beginner): What's the difference between == and ===? When is == acceptable?

=== (strict) compares type and value with no coercion; == (loose) coerces operands to a common type first, following complicated rules that produce surprising results like 0 == "" and [] == ![] both being true. Default to === everywhere. The one accepted use of == is x == null, which leverages the single useful coercion (null == undefined is true) to check "null or undefined" in one expression — allowed explicitly by the eqeqeq "smart" lint setting.

Q2 (beginner): List every falsy value. Is an empty array falsy?

Exactly eight: false, 0, -0, 0n, "", null, undefined, NaN. Everything else is truthy — including "0", "false", [], and {}. So no, an empty array is truthy; if ([]) runs its block. Testing an array for emptiness requires arr.length === 0, not a truthiness check.

Q3 (mid): Explain the bug in const timeout = opts.timeout || 3000 and how ?? fixes it.

|| falls back to the right operand for any falsy left value. If a caller legitimately passes opts.timeout = 0 (meaning "no timeout / immediate"), || treats 0 as falsy and overrides it with 3000 — silently ignoring the caller's explicit choice. ?? (nullish coalescing) falls back only when the left side is null or undefined, so opts.timeout ?? 3000 preserves a deliberate 0 while still defaulting when the option is genuinely absent. The general rule: ?? for "default when not provided," || only when you truly mean "any falsy."

Q4 (mid): What do && and || actually return, and how does short-circuiting enable common idioms?

They return one of their operands, not a boolean. a && b evaluates a; if falsy it returns a without touching b, else it evaluates and returns b. a || b returns a if truthy, else b. This short-circuiting powers: guards (isReady && run() — right side runs only if left is truthy), defaults (x || fallback), and lazy evaluation (the right side isn't evaluated when the result is already decided — important when it has side effects or is expensive). The gotcha is that the returned value is the operand itself, which is why {0 && <X/>} in JSX renders 0.

Q5 (senior): Explain why [] == ![] evaluates to true, step by step, and what this teaches about ==.

Precedence first: ![] is evaluated before ==. [] is truthy (all objects are), so ![] is false. Now it's [] == false. Loose equality with a boolean coerces the boolean to a number: false → 0, giving [] == 0. Comparing an object to a number, == converts the object to a primitive via ToPrimitive: [] → toString → "". Now "" == 0; a string-vs-number comparison coerces the string to a number: "" → 0. Finally 0 == 0 → true. The lesson: == chains several independent coercion rules, and the compound result is nearly impossible to predict by intuition — which is the entire justification for banning it and using ===, where [] === false is trivially false (different types, done).

Q6 (senior): A codebase overuses ?. defensively — user?.profile?.name?.trim?.() everywhere, even where data is guaranteed. Critique this and give a principle for when optional chaining belongs.

Defensive ?. everywhere is an anti-pattern because it converts bugs into silent wrong behaviour. If user.profile is a documented invariant that should always exist, and it's suddenly undefined, that's a real defect upstream — a broken API contract, a bad migration, a logic error. Writing user?.profile swallows that: instead of a loud TypeError pointing at the exact broken assumption, the whole chain silently yields undefined, which then flows onward and fails somewhere unrelated, far harder to diagnose. It also lies to readers about the data model — every ?. signals "this might legitimately be absent," so scattering it everywhere destroys that signal. The principle: use ?. only where absence is a genuine, expected state (optional API fields, DOM lookups that may miss, config that may be unset). Where a value is an invariant, access it directly and let a violation throw — a fail-fast error at the source is a feature, not a gap. Pair this with validation at boundaries (Types lesson) so that once data is inside your domain, its shape is guaranteed and defensive chaining is unnecessary by construction. Optional chaining is a tool for modelling real optionality, not a blanket crash-suppressant.

🏋️ Exercises

1 (easy): === vs == predictions. Predict each: 1 === "1", 0 == "", null == undefined, null === undefined, NaN === NaN, "" == false.
Answers

false, true, true, false, false, true. The only true === here is none of them among these — note null === undefined is false (different types) while null == undefined is the special-cased true. This is exactly why only x == null is allowed among == uses.

2 (easy): Falsy filter. Write a one-liner that removes all falsy values from an array: [0, 1, "", "a", null, false, NaN, "0"] → [1, "a", "0"].
Solution 
arr.filter(Boolean);   // [1, "a", "0"]

Boolean as the predicate coerces each element to its truthiness; falsy ones are dropped. Note "0" survives (non-empty string = truthy) — a good check that you remember the traps.

3 (medium): The 0 bug. A slider component does const value = props.value || props.defaultValue. A user drags to 0 and it snaps to the default. Fix it and explain.
Solution 
const value = props.value ?? props.defaultValue;

|| treats the user's legitimate 0 as falsy and overrides it with the default. ?? falls back only for null/undefined, so a value of 0 is preserved and only a genuinely-unset value uses the default. This is the single most common real-world || bug.

4 (medium): Deep equality, honestly. Explain why JSON.stringify(a) === JSON.stringify(b) is a flawed deep-equality check, and name three concrete inputs where it gives a wrong or throwing answer.
Solution

It depends on serialisation, which is lossy and order-sensitive. Three failures: (1) {a:1,b:2} vs {b:2,a:1} — equal in meaning but different string (key order differs) → reports not equal. (2) {a:undefined} vs {} — JSON.stringify drops undefined properties, so both stringify to "{}" → reports equal when arguably they aren't. (3) A cyclic object, or one containing a BigInt — JSON.stringify throws, so the comparison crashes. Also functions/Symbols vanish. Use a real deep-equal (lodash isEqual or a correct recursive implementation) when it matters.

5 (hard): Implement deepEqual. Write a recursive deepEqual(a, b) handling primitives (including NaN), arrays, and plain objects, with correct handling for differing key counts. Ignore cycles for this exercise.
Solution 
function deepEqual(a, b) {
  if (Object.is(a, b)) return true;                 // handles primitives incl. NaN and -0
  if (typeof a !== "object" || a === null ||
      typeof b !== "object" || b === null) return false;
  if (Array.isArray(a) !== Array.isArray(b)) return false;
  const ka = Object.keys(a), kb = Object.keys(b);
  if (ka.length !== kb.length) return false;         // differing key counts → not equal
  return ka.every(k => Object.prototype.hasOwnProperty.call(b, k) && deepEqual(a[k], b[k]));
}
deepEqual({a:1, b:[2, NaN]}, {b:[2, NaN], a:1});   // true — order-independent, NaN-safe

Object.is for the base case gives correct NaN handling for free; the key-count check catches the "b has extra keys" asymmetry that a naive one-directional walk would miss; hasOwnProperty avoids inherited-property false positives (Prototypes lesson).

6 (hard): Correct defaults everywhere. Given this options merge, identify every place the default semantics are wrong and rewrite it so that legitimate falsy values survive but genuinely-empty ones fall back: 
function makeRequest(opts) {
  const url     = opts.url || "/api";
  const retries = opts.retries || 3;
  const cache   = opts.cache || true;
  const timeout = opts.timeout || 5000;
  const label   = opts.label || "";
}
Solution 
const url     = opts.url || "/api";       // ✅ || correct — empty url is not valid, fall back
const retries = opts.retries ?? 3;        // ❌→✅ retries:0 is legit (no retries) — was clobbered
const cache   = opts.cache ?? true;       // ❌→✅ cache:false was IMPOSSIBLE to set (|| true always true!)
const timeout = opts.timeout ?? 5000;     // ❌→✅ timeout:0 is legit
const label   = opts.label ?? "";         // ❌→✅ though harmless here, ?? is semantically right

The worst bug is cache: opts.cache || true is always true — passing false can never disable the cache, because false || true is true. This is a silent, unfixable-from-the-outside defect that ?? resolves. The rule applied per-field: use || only where a falsy value genuinely isn't a valid choice (an empty url), and ?? everywhere a falsy value (0, false, "") is a legitimate setting.

🎯 Key Takeaways

  • === is the default: same type and value, no coercion; objects compare by reference, so distinct-but-identical objects are never equal.
  • == is banned except x == null (checks null OR undefined) — its coercion chains ([] == ![]) are a warning, not a puzzle.
  • Eight falsy values (false 0 -0 0n "" null undefined NaN); everything else is truthy — including [], {}, "0". Test emptiness with .length, not truthiness.
  • ?? for defaults (preserves 0/false/""), || only for genuine "any falsy" logic — mixing them wrong is a top real-world bug.
  • ?. for genuine optionality, not defensive crash-suppression — an early throw beats a silent undefined flowing onward.
  • &&/|| return an operand and short-circuit; logical-assignment (??= ||= &&=) short-circuits too, deferring expensive right-sides.
