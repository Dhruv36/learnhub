Generators & Iterators Advanced

The iteration protocol is the invisible contract behind for…of, spread, and destructuring; generators are pausable functions that implement it for free. Their production niche is precise and valuable: lazy sequences that don't fit (or shouldn't sit) in memory — paginated APIs, streams, logs — plus the conceptual key to understanding async/await itself.

The Protocol — What for…of Actually Calls

// An ITERABLE has [Symbol.iterator]() returning an ITERATOR: { next() → {value, done} }
const it = [10, 20][Symbol.iterator]();
it.next();   // { value: 10, done: false }
it.next();   // { value: 20, done: false }
it.next();   // { value: undefined, done: true }

// for…of, [...spread], destructuring, Array.from, Promise.all — ALL just call this protocol.
// Iterables you already use: Array, String, Map, Set, NodeList, FormData, URLSearchParams.
// NOT iterable: plain objects — hence Object.keys/values/entries as the bridge.

Generators — Pausable Functions

function* idGenerator() {              // the * makes it a generator
  let id = 1;
  while (true) yield id++;             // yield = pause HERE, emit a value, keep ALL state
}
const ids = idGenerator();             // calling it runs NOTHING — returns an iterator
ids.next().value;   // 1               // runs until the first yield
ids.next().value;   // 2               // resumes exactly where it paused — id survived

// Two-way channel — next(arg) becomes yield's return value:
function* dialogue() {
  const name = yield "What's your name?";
  yield `Hello, ${name}!`;
}
const d = dialogue();
d.next().value;          // "What's your name?"
d.next("Asha").value;    // "Hello, Asha!"

Mentally: a generator is a resumable stack frame — locals persist across yields. Infinite generators are safe because they're lazy: values compute only when pulled. That laziness is the entire value proposition.

Composition — Small Generators, Big Pipelines

function* take(n, iterable) {
  for (const x of iterable) { if (n-- <= 0) return; yield x; }
}
function* map(fn, iterable)    { for (const x of iterable) yield fn(x); }
function* filter(fn, iterable) { for (const x of iterable) if (fn(x)) yield x; }

[...take(3, filter(x => x % 2, map(x => x * x, idGenerator())))]   // [1, 9, 25]
// LAZY pipeline: each value flows through the whole chain one at a time —
// no intermediate arrays, works on infinite sources. (Array chains .map().filter()
// materialize a full array per step.) Iterator helpers are landing natively:
idGenerator().map(x => x * x).filter(x => x % 2).take(3).toArray()   // check support

function* all() { yield* gen1(); yield* gen2(); }   // yield* = delegate to another iterable

The Production Star: Async Generators

async function* fetchAllUsers() {              // async function* — await AND yield
  let url = "/api/users?page=1";
  while (url) {
    const res = await fetch(url);
    const { items, nextPage } = await res.json();
    yield* items;                               // emit each user as it's available
    url = nextPage;
  }
}

for await (const user of fetchAllUsers()) {     // consumer sees ONE clean stream
  if (user.flagged) await review(user);
  if (reviewedEnough) break;                    // ← remaining pages are NEVER fetched
}
// The consumer neither knows nor cares about pagination — and early exit saves real
// bandwidth. Same shape consumes: Node streams (async-iterable!), fetch body readers
// (ReadableStream), event streams, file readers, tailing logs.

Cleanup — return() / throw() and the finally Guarantee

function* readLines(file) {
  const handle = open(file);
  try {
    while (hasMore(handle)) yield nextLine(handle);
  } finally {
    close(handle);                     // runs on natural end, break, AND .return()
  }
}
for (const line of readLines("app.log")) {
  if (line.includes("FATAL")) break;   // break calls the iterator's .return() →
}                                       // generator resumes INTO finally → handle closed ✅
// gen.throw(err) similarly injects an exception at the paused yield — consumers can
// signal failure INTO the producer. This protocol is why for…of/for-await are safe
// with resources and manual .next() loops without cleanup are not.

The Conceptual Payoff: async/await IS Generators

// Pre-2017 production code literally ran on this (the 'co' library):
co(function* () {
  const user  = yield fetch("/user");     // yield a promise to the runner
  const posts = yield fetch(`/posts/${user.id}`);
  return posts;
});
// The runner: call next(), get a promise, .then(v => next(v)) — resume with the value.
// async/await standardized exactly this: await ≈ yield to the engine's promise-driver.
// Redux-Saga kept the explicit form because yielding EFFECT DESCRIPTIONS makes logic
// unit-testable without mocks — feed values in with next(), assert what's yielded out.

Interview Questions

Q1: Make a plain object work with for…of and [...spread].

Implement *[Symbol.iterator]() — e.g., class Range { constructor(a,b){…} *[Symbol.iterator](){ for (let i=this.a;i<=this.b;i++) yield i; } } → [...new Range(1,4)] = [1,2,3,4]. Full credit for knowing spread/destructuring/for…of all consume the same protocol, and that returning a fresh generator each call makes the object RE-iterable (unlike a raw generator).

Q2: How does break inside for await stop a paginated fetcher from downloading more pages?

break makes the loop call the async iterator's .return(); the async generator resumes as if a return happened at the paused yield — running finally blocks and never reaching the next while-iteration's fetch. The loop protocol, not magic, provides the resource safety — which is why manual .next() pulls skip it.

Q3: Sketch how you'd implement async/await using generators (the co pattern).

run(genFn): start the iterator; a step(value) function calls it.next(value); if done, resolve with value; else Promise.resolve(yielded).then(step, err => it.throw(err)). That's the whole engine — await = yield + this driver + microtask scheduling. Explaining throw() routing rejections back INTO the generator (so try/catch works around yields) is the senior detail.

Q4: Lazy pipeline vs array-method chain — when does laziness actually win?

When the source is huge/infinite/expensive-per-item (paginated APIs, files, streams), when you'll early-exit (take/find semantics — only computes what's consumed), and when intermediate arrays would pressure memory. For a 50-item list, array chains win on clarity. Numbers: .map on 10M items materializes 10M intermediates per step; the lazy chain holds one value at a time.

🎯 Key Takeaways

  • The protocol ({next()→{value,done}} via [Symbol.iterator]) is all that for…of/spread/destructuring call — implement it, join the language.
  • Generators = resumable stack frames: lazy, stateful, one-shot iterators; nothing runs until pulled.
  • Compose small generators into lazy pipelines — no intermediate arrays, early exit computes nothing extra.
  • Async generators + for await turn pagination/streams into one clean, break-able sequence — the production use.
  • for…of/for-await call .return() on early exit → finally cleanup runs; manual next() loops don't.
  • await ≈ yield + a promise-driving runner — knowing the co pattern is knowing what async/await is.
