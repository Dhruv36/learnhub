Iterators & Generators Asynchronous JavaScript

What you'll master: the iteration protocol that makes for...of and spread work on anything, generator functions that produce values lazily and pause mid-execution, infinite and lazy sequences, and async generators for streaming data. Prerequisites: Symbols (Symbol.iterator) and async/await (for async iteration). These are advanced but genuinely useful: they power lazy evaluation, custom iterables, and streaming — and they're a favourite senior interview topic.

Part 1 — The Iteration Protocol

// An ITERATOR is an object with a next() method returning { value, done }:
const iterator = {
  current: 1, last: 3,
  next() {
    return this.current <= this.last
      ? { value: this.current++, done: false }
      : { value: undefined, done: true };   // done:true signals the end
  }
};
iterator.next();   // { value: 1, done: false }
iterator.next();   // { value: 2, done: false }
iterator.next();   // { value: 3, done: false }
iterator.next();   // { value: undefined, done: true }

// An ITERABLE is an object with a [Symbol.iterator]() method returning an iterator.
// for...of, spread, destructuring, Array.from all consume ITERABLES.

Two related contracts. An iterator is any object with a next() method that returns { value, done } — call it repeatedly to pull values until done: true. An iterable is any object with a [Symbol.iterator]() method that returns an iterator. Every construct that "loops over" things — for...of, spread ([...x]), destructuring, Array.from, Promise.all — works by calling the iterable's [Symbol.iterator]() and pulling via next(). Arrays, strings, Maps, and Sets are all iterable by this same protocol (Collections lesson), which is why one for...of works on all of them.

Part 2 — Generators: Iterators You Don't Hand-Write

function* count() {         // function* declares a GENERATOR
  yield 1;                  // yield produces a value AND pauses execution here
  yield 2;
  yield 3;
}                           // (implicit return → done:true)
const gen = count();        // calling it returns a generator object (an iterator) — body hasn't run yet!
gen.next();                 // { value: 1, done: false } — runs until the first yield, then PAUSES
gen.next();                 // { value: 2, done: false } — RESUMES after yield 1, runs to yield 2
gen.next();                 // { value: 3, done: false }
gen.next();                 // { value: undefined, done: true }

[...count()];               // [1, 2, 3] — a generator is iterable, so spread/for...of just work

A generator function (function*) is the easy way to create an iterator — no hand-written next() state machine. The magic is yield: it produces a value and pauses the function's execution, freezing all local state, until the next next() call resumes right where it left off. This ability to pause and resume mid-function is unique — normal functions run to completion. A generator object is both an iterator (has next()) and iterable (has [Symbol.iterator] returning itself), so it plugs straight into for...of, spread, and destructuring.

Part 3 — Lazy & Infinite Sequences

// An INFINITE generator — impossible with an array, fine here because values are produced ON DEMAND:
function* naturals() {
  let n = 1;
  while (true) yield n++;   // never terminates, but only computes what you pull
}
const nums = naturals();
nums.next().value;  // 1
nums.next().value;  // 2   — only two values computed; the rest never exist unless requested

// Take the first N from any (even infinite) iterable — lazy composition:
function* take(iterable, n) {
  let i = 0;
  for (const x of iterable) {
    if (i++ >= n) return;
    yield x;
  }
}
[...take(naturals(), 5)];   // [1, 2, 3, 4, 5] — pulled lazily, generator paused after the 5th

Generators are lazy: they compute values only when pulled, so you can represent infinite sequences (all natural numbers, an endless ID stream) that would be impossible as arrays. Combined with lazy operators like take, filter, and map written as generators, you can process potentially-huge or infinite streams while only materialising what you consume — the memory-efficiency win of lazy evaluation. This is how you'd handle "the first 10 primes" or "paginate through a million records" without loading everything at once.

Part 4 — yield* Delegation & Two-Way Communication

// yield* delegates to another iterable/generator — flattening composition:
function* inner() { yield "a"; yield "b"; }
function* outer() {
  yield 1;
  yield* inner();     // yields each value from inner: "a", "b"
  yield 2;
}
[...outer()];         // [1, "a", "b", 2]

// next(value) sends a value BACK INTO the generator — two-way communication:
function* echo() {
  const received = yield "ready";   // the value passed to the NEXT next() becomes `received`
  yield `you said: ${received}`;
}
const g = echo();
g.next();            // { value: "ready", done: false }  — runs to first yield
g.next("hello");     // { value: "you said: hello", done: false } — "hello" becomes `received`

Two advanced capabilities. yield* delegates to another iterable, yielding all of its values in place — useful for composing generators and for recursive traversal (walking a tree). And generators are two-way: gen.next(value) passes value back into the generator as the result of the yield expression that paused it. This bidirectional flow (pull values out with yield, push values in with next(v)) is what makes generators powerful enough to implement coroutines — and it's exactly the mechanism early async libraries (co, redux-saga) used to build async/await-like control flow before it was native.

Part 5 — Custom Iterables with Symbol.iterator

// Make ANY object work with for...of / spread by implementing [Symbol.iterator] as a generator:
class Range {
  constructor(start, end, step = 1) { Object.assign(this, { start, end, step }); }
  *[Symbol.iterator]() {                    // generator method — IS the iterator
    for (let i = this.start; i <= this.end; i += this.step) yield i;
  }
}
const r = new Range(0, 10, 2);
[...r];                    // [0, 2, 4, 6, 8, 10]
for (const n of r) { }     // works
Math.max(...r);            // 10 — spread into arguments
const [first, ...rest] = r; // destructuring — all from one generator method

Implementing [Symbol.iterator]() makes a custom object a first-class iterable — usable everywhere the language accepts one. Writing it as a generator method (*[Symbol.iterator]()) is far cleaner than a hand-rolled iterator object, because yield handles the state machine for you. This is the recommended way to make your data structures (trees, linked lists, ranges, paginated collections) iterable — one small method and they gain for...of, spread, destructuring, and every iterable-consuming API for free.

Part 6 — Async Generators & for await...of

// An async generator yields promises resolved one at a time — perfect for streaming/pagination:
async function* fetchPages(url) {
  let next = url;
  while (next) {
    const res = await fetch(next);          // await inside the generator
    const page = await res.json();
    yield page.items;                        // yield each page's items as they arrive
    next = page.nextUrl;                      // follow pagination until there's no next page
  }
}

// Consume with for await...of — awaits each yielded value:
for await (const items of fetchPages("/api/items?page=1")) {
  render(items);                             // process each page as it loads, not all at once
}

Async generators (async function*) combine generators with await: they yield values that are awaited, and you consume them with for await...of, which awaits each value in turn. This is the ideal tool for streaming and pagination — processing a paginated API page-by-page as each loads (rather than fetching everything then processing), reading a large file or network stream in chunks, or consuming a WebSocket. It gives you backpressure for free (the next page isn't fetched until you're ready for it) and keeps memory bounded regardless of total data size — a genuinely elegant solution to "iterate over data that arrives over time."

Part 7 — Build It: a Lazy Pipeline over an Infinite Source

Let's compose generator-based operators into a lazy pipeline — map and filter that work on infinite sequences, materialising only what's taken. This is a miniature of what libraries like iter-tools and transducers provide.

function* map(iterable, fn) { for (const x of iterable) yield fn(x); }
function* filter(iterable, pred) { for (const x of iterable) if (pred(x)) yield x; }
function* take(iterable, n) {
  let i = 0;
  for (const x of iterable) { if (i++ >= n) return; yield x; }
}
function* naturals() { let n = 1; while (true) yield n++; }

// Compose lazily: squares of even numbers, first 5 — over an INFINITE source:
const pipeline = take(
  map(
    filter(naturals(), n => n % 2 === 0),   // evens: 2, 4, 6, ...
    n => n * n                               // squared: 4, 16, 36, ...
  ),
  5
);
[...pipeline];   // [4, 16, 36, 64, 100] — only 10 naturals ever generated, nothing wasted

The whole pipeline is lazy: no array of "all evens" is ever built (impossible — it's infinite). Instead, pulling one value from take pulls one from map, which pulls from filter, which pulls from naturals — values flow one at a time, on demand, and generation stops the instant take has its 5. Compare this to the eager array-method version (Arrays lesson): naturals.filter(...).map(...) could never work on an infinite source and would build full intermediate arrays on a finite one. Generators enable a fundamentally different, memory-bounded style of data processing — the payoff of understanding the protocol.

Interview Questions

Q1 (beginner): What's the difference between an iterator and an iterable?

An iterator is an object with a next() method that returns { value, done } — you call it repeatedly to pull values until done is true. An iterable is an object with a [Symbol.iterator]() method that returns an iterator. Iterables are what for...of, spread, destructuring, and Array.from consume — they call [Symbol.iterator]() to get an iterator and then pull values. Arrays, strings, Maps, and Sets are all iterable. A generator object is conveniently both: it's an iterator and it's iterable (its [Symbol.iterator] returns itself).

Q2 (beginner): What does a generator function do that a normal function can't?

It can pause and resume mid-execution. A function* with yield produces a value and suspends, freezing all its local state, until the next next() call resumes it from exactly that point. Normal functions run to completion each call and can't be paused. Calling a generator doesn't run its body — it returns a generator object (an iterator); the body executes incrementally as you pull values. This pause/resume ability is what enables lazy sequences, infinite sequences, and streaming.

Q3 (mid): How do generators enable lazy and infinite sequences, and why is that useful?

Because a generator computes each value only when it's pulled (via next() or iteration) and pauses in between, it never needs to produce or store values you don't consume. That makes infinite sequences possible — while(true) yield n++ represents all natural numbers but only materialises the ones you take — which is impossible with an array. And it makes processing large data memory-efficient: a lazy pipeline of generator-based map/filter/take flows values one at a time and stops as soon as enough are collected, never building intermediate arrays. This is useful for streaming large files/APIs, representing unbounded sequences, and building composable lazy iteration that bounds memory regardless of total data size.

Q4 (mid): What are async generators and when would you use one?

An async generator (async function*) is a generator whose yielded values are awaited and which can use await internally; you consume it with for await...of, which awaits each value in turn. Use it for data that arrives over time and should be processed incrementally: paginating an API (yield each page as it loads, following the next-page link), reading a large file or network response in chunks, or consuming a stream/WebSocket. It gives natural backpressure — the next chunk isn't fetched until the consumer is ready for it — and keeps memory bounded no matter how much total data flows through, which is far cleaner than manually chaining promises for each page or buffering everything first.

Q5 (senior): Explain two-way generator communication (next(value)) and how it relates to async/await's implementation.

Generators are bidirectional: yield expr both produces expr to the consumer and becomes an expression whose value is whatever is passed to the next next(value) call. So const x = yield "ask" yields "ask", pauses, and when resumed with gen.next(42), x becomes 42. This lets a driver pull a value out, compute something, and push a result back in — the essence of a coroutine. Its connection to async/await: before async/await was native, libraries (co, Bluebird's coroutines, redux-saga) implemented it by combining generators with promises. The pattern: a generator yields a promise; an external "runner" awaits that promise and, when it settles, resumes the generator by pushing the resolved value back in via next(resolvedValue) (or throws the rejection into it via gen.throw()). Iterating that loop drives the generator forward one await at a time. In effect, async/await is syntactic/engine-level sugar for exactly this generator-plus-promise-runner pattern — await corresponds to yielding a promise, and the engine is the runner that resumes the state machine when it settles. Understanding this demystifies both: an async function is a generator-like pausable state machine whose scheduler is built into the runtime, using the same two-way communication (push resolved values in, throw rejections in) that next(value)/ throw() provide manually.

Q6 (senior): Design a memory-bounded processor for a multi-gigabyte data source using generators. What are the key considerations?

The core idea is to never hold the whole dataset in memory — stream it through a lazy generator pipeline that processes one chunk/record at a time. Design: an async generator at the source yields records incrementally (reading a file in chunks via a stream, or paginating an API following next-links, awaiting each read), then compose pure generator transforms (map, filter, batching) that pull from it lazily, and a consumer (for await...of) that processes and discards each item before pulling the next — so peak memory is bounded by one chunk plus any batch buffer, regardless of total size. Key considerations: (1) Backpressure — the pull-based model gives it naturally (nothing is read until the consumer asks), but if you write to a slower sink, honour its drain signals so you don't buffer unboundedly. (2) Batching — for efficiency (bulk DB inserts, batched API calls) a generator that groups records into fixed-size batches amortises per-item overhead while still bounding memory to one batch. (3) Cleanup on early exit — if the consumer breaks early, the generator's return() is called, so put try/finally around resource acquisition (close file handles, DB cursors, abort fetches) to release them even on early termination. (4) Error handling — decide per-record whether an error aborts the stream or is collected and skipped (an errors sink), since one bad record shouldn't necessarily kill a multi-hour job. (5) Concurrency — pure lazy generators are sequential; if per-item work is I/O-bound you may want bounded parallelism (a pool pulling from the generator), trading strict ordering for throughput. (6) Progress/resumability — track offset/cursor so a failed job can resume rather than restart. The senior framing: generators (especially async generators) are the right abstraction for bounded-memory stream processing because they invert control to pull-based, lazy, one-at-a-time flow with built-in cleanup hooks — but production robustness requires layering batching, backpressure, cleanup, per-record error policy, and optionally bounded concurrency on top, which is why tools like Node streams and libraries formalise these patterns.

🏋️ Exercises

1 (easy): Range generator. Write function* range(start, end, step) yielding numbers from start to end (inclusive) by step, and spread it into an array.
Solution 
function* range(start, end, step = 1) {
  for (let i = start; i <= end; i += step) yield i;
}
[...range(0, 10, 2)];   // [0, 2, 4, 6, 8, 10]

yield in a loop produces each value lazily; because the generator is iterable, spread collects them.

2 (easy): Fibonacci. Write an infinite fibonacci() generator and take the first 8 values.
Solution 
function* fibonacci() {
  let [a, b] = [0, 1];
  while (true) { yield a; [a, b] = [b, a + b]; }
}
const fib = fibonacci();
Array.from({ length: 8 }, () => fib.next().value);   // [0, 1, 1, 2, 3, 5, 8, 13]

The infinite while(true) is safe because values are only produced on demand; pulling 8 via next() materialises exactly 8.

3 (medium): Make a tree iterable. Given a tree of { value, children: [...] } nodes, add a generator that yields all values depth-first using yield* for recursion.
Solution 
function* walk(node) {
  yield node.value;
  for (const child of node.children ?? []) {
    yield* walk(child);          // delegate to the child's generator — recursive flattening
  }
}
const tree = { value: 1, children: [{ value: 2, children: [] }, { value: 3, children: [{ value: 4, children: [] }] }] };
[...walk(tree)];   // [1, 2, 3, 4]

yield* delegates to each recursive walk(child), yielding all its values in place — a clean depth-first traversal without an explicit stack.

4 (medium): Lazy take-while. Write takeWhile(iterable, pred) that yields values until the predicate first fails, then stops.
Solution 
function* takeWhile(iterable, pred) {
  for (const x of iterable) {
    if (!pred(x)) return;         // stop at the first failure — generator ends
    yield x;
  }
}
[...takeWhile(range(1, 100), n => n < 5)];   // [1, 2, 3, 4]

It pulls lazily and returns (ending the generator) the moment the predicate fails, so it works even on an infinite source and never over-consumes.

5 (hard): Async page iterator. Write an async generator paginate(fetchPage) that yields items from successive pages until a page reports no next cursor, and consume it with for await...of.
Solution 
async function* paginate(fetchPage) {
  let cursor = null;
  do {
    const { items, next } = await fetchPage(cursor);   // fetch one page
    yield* items;                                       // yield each item of this page
    cursor = next;
  } while (cursor);                                      // stop when no next cursor
}

for await (const item of paginate(cursor => api.getItems(cursor))) {
  process(item);      // one item at a time, pages fetched on demand
}

The async generator awaits each page, yields its items (via yield*), and follows the cursor until exhausted. for await...of pulls lazily, so a page is only fetched when the consumer is ready — bounded memory and natural backpressure over an arbitrarily large dataset.

6 (hard): Cleanup on early break. Write a generator that acquires a resource, yields values, and guarantees the resource is released even if the consumer breaks early. Demonstrate the cleanup firing.
Solution 
function* withResource(makeResource) {
  const resource = makeResource();
  try {
    let i = 0;
    while (true) yield resource.read(i++);
  } finally {
    resource.close();               // runs on normal end OR early break (generator.return())
    console.log("resource closed");
  }
}
for (const chunk of withResource(openFile)) {
  if (chunk === "STOP") break;       // early exit → the generator's return() triggers finally
}
// logs "resource closed" — cleanup happened despite the break

When a for...of loop breaks (or throws), it calls the generator's return(), which runs any finally block — so wrapping resource use in try/finally guarantees cleanup even on early termination. This is the generator equivalent of RAII/using-blocks and is essential for streaming code that holds file handles or connections.

🎯 Key Takeaways

  • The iteration protocol: an iterator has next() → {value, done}; an iterable has [Symbol.iterator]() — this is what for...of/spread consume.
  • Generators (function* + yield) are pausable functions that create iterators without a hand-written state machine; calling one is lazy (body runs on pull).
  • Laziness enables infinite and memory-bounded sequences; compose generator map/filter/take for lazy pipelines that only compute what's consumed.
  • yield* delegates to another iterable (great for recursion); next(value) pushes values back in — two-way communication underpinning coroutines and async/await's design.
  • Implement *[Symbol.iterator]() to make custom objects iterable everywhere; generators are one-shot (re-call the function for a fresh run).
  • Async generators (async function* + for await...of) stream data over time with backpressure — ideal for pagination and chunked I/O; use try/finally for cleanup on early exit.
