Destructuring, Spread & Rest Objects & Data

What you'll master: object and array destructuring with defaults, renaming, and nesting; the spread operator for copying and merging; rest for collecting; and the crucial distinction between spread (in a value position) and rest (in a binding position) that share the same ... syntax. Prerequisites: Objects, Arrays, and Functions (parameters). These features are everywhere in modern JS and React — fluency makes code dramatically more concise and readable.

Part 1 — Object Destructuring

const user = { name: "Ada", age: 30, role: "admin" };

const { name, age } = user;          // pull properties into variables by matching key name
const { role: userRole } = user;     // RENAME: role → userRole (property : newName)
const { country = "US" } = user;     // DEFAULT: used only if the property is undefined/missing
const { name: n = "?" } = user;      // rename AND default together
const { address: { city } = {} } = user;  // NESTED with a fallback (avoids crash if address missing)

Object destructuring pulls properties into variables, matching by key name (order doesn't matter, unlike arrays). You can rename (key: newName), supply defaults (used only when the value is undefined — same rule as function defaults), and combine both. Nested destructuring reaches into sub-objects, and adding = {} on a nested target guards against a missing parent throwing. This is why function signatures like function f({ name, age = 0 } = {}) are so common — they're object destructuring applied to parameters.

Part 2 — Array Destructuring

const rgb = [255, 128, 0];
const [r, g, b] = rgb;               // r=255, g=128, b=0 — matches by POSITION
const [first, , third] = rgb;        // skip the middle with an empty slot → first=255, third=0
const [x, y, z = 0] = [1, 2];        // default → z=0 (position 2 missing)
const [head, ...tail] = [1, 2, 3, 4];// head=1, tail=[2,3,4] — REST collects the rest
const [a, b] = [b, a];               // (with let) SWAP without a temp variable
[state.x, state.y] = [state.y, state.x];  // swap object props too

Array destructuring matches by position, not name. You can skip elements with empty slots, supply defaults, and use a rest element (...tail) to collect the remaining items into an array (it must be last). The one-line swap ([a, b] = [b, a]) is a neat idiom. Because it works on any iterable, array destructuring also unpacks things like Map entries and function returns:

for (const [key, value] of Object.entries(obj)) { }   // destructure each [k,v] pair (Objects lesson)
const [data, error] = await tryCatch(fetchUser());    // returning a tuple, Go-style

Part 3 — Spread: Expanding a Value

// Arrays — copy, merge, insert:
const copy = [...arr];                       // shallow copy
const merged = [...a, ...b];                 // concatenate
const withExtra = [0, ...arr, 99];           // insert at any position
Math.max(...[3, 1, 4]);                       // spread into individual arguments → Math.max(3,1,4)

// Objects — copy, merge, override:
const objCopy = { ...obj };                  // shallow copy (Objects lesson)
const updated = { ...obj, age: 31 };         // copy + override (later wins)
const combined = { ...defaults, ...overrides };  // merge

// Strings/iterables → array:
[..."abc"];                                   // ["a","b","c"] (code-point aware — Strings lesson)
[...new Set([1, 1, 2])];                      // [1, 2] — dedupe

Spread (...) in a value position expands an iterable (or object) into its elements/properties. It's the idiomatic tool for shallow-copying, merging, inserting, and turning an array into individual function arguments (replacing the old fn.apply(null, arr)). For objects, spread copies own enumerable properties, with later spreads overriding earlier keys — the basis of immutable updates. Remember it's shallow (Objects lesson): nested objects are shared, so structuredClone is still needed for deep copies.

Part 4 — Rest: Collecting Values

// Rest in a FUNCTION parameter — gather trailing args into an array (Functions lesson):
function sum(...nums) { return nums.reduce((a, b) => a + b, 0); }

// Rest in ARRAY destructuring — collect remaining elements:
const [first, ...others] = [1, 2, 3];        // others = [2, 3]

// Rest in OBJECT destructuring — collect remaining properties (great for "omit"):
const { password, ...safeUser } = user;      // safeUser = user without password
// safeUser is a NEW object; user is untouched — a clean way to strip a field

Rest (...) in a binding position collects multiple values into a single array or object. It's the mirror image of spread. In object destructuring, the rest pattern is the cleanest way to omit a property (const { secret, ...rest } = obj gives you everything except secret, as a new object) — a pattern you'll use constantly to strip sensitive or unwanted fields before passing data on.

Part 5 — Spread vs Rest: Same Syntax, Opposite Jobs

SpreadRest
PositionValue position (RHS, argument list, literal)Binding position (LHS pattern, parameter list)
DirectionExpands one thing into manyCollects many into one
Examplefn(...arr), [...a, ...b]function fn(...args), const [x, ...rest] = arr
ResultIndividual elements/propsAn array (or object) holding the collected items

They look identical (...) but do opposite things, distinguished entirely by where they appear. Spread is on the value side — you're producing a list of elements (into a call, a literal, an argument list). Rest is on the binding side — you're capturing a variable number of items into one name (a parameter, a destructuring target). One memory hook: rest must be last (there's nothing left to collect after it), while spread can appear anywhere in a value list. If you can place the ... mentally as "unpacking into here" (spread) vs "gathering from here" (rest), you'll never confuse them.

Part 6 — Destructuring in Function Parameters

// Named options object — the scalable-API pattern (Functions lesson):
function createChart({ type = "bar", color = "blue", data = [], animate = true } = {}) {
  return { type, color, data, animate };
}
createChart({ type: "line", color: "red" });   // other options use defaults
createChart();                                   // = {} lets you call with NO argument

// Destructuring an array parameter (e.g. a coordinate pair):
const distance = ([x1, y1], [x2, y2]) => Math.hypot(x2 - x1, y2 - y1);
distance([0, 0], [3, 4]);   // 5

// React props — destructuring in the parameter is idiomatic:
function Button({ label, onClick, disabled = false }) { /* ... */ }

Destructuring shines in function signatures. The named-options-object pattern (destructure the parameter with per-field defaults and a = {} fallback) gives callers order-independent, self-documenting, individually-optional arguments — the standard for any function with more than two options, and the reason React components destructure props right in the signature. This is the same idea as Part 1, applied where it's most useful.

Part 7 — Gotchas Worth Knowing

// 1. Destructuring null/undefined THROWS:
const { a } = null;              // TypeError: Cannot destructure property 'a' of null
const { a } = undefined ?? {};   // guard with ?? {} or a default parameter

// 2. Default only fires on undefined, NOT null (same rule as everywhere):
const { x = 5 } = { x: null };   // x is null, NOT 5

// 3. Destructuring into existing variables needs parens (statement vs block ambiguity):
let a, b;
({ a, b } = obj);                // MUST wrap in () — bare { } starts a block

// 4. Swapping without let/const reassignment:
[obj.x, obj.y] = [obj.y, obj.x]; // fine — assignment targets can be properties

Four traps: (1) destructuring null/undefined throws — guard with ?? {} or a parameter default; (2) defaults fire only on undefined, so a null value passes through (consistent with function defaults and the Types lesson); (3) assigning (not declaring) via object destructuring requires wrapping the whole thing in parentheses, because a leading { is parsed as a block statement; and (4) destructuring assignment targets can be object properties or array elements, not just fresh variables — enabling neat swaps and in-place updates.

Part 8 — Build It: a Response Normaliser

Let's combine destructuring, defaults, rename, and rest into one realistic function: safely unpacking a messy API response into a clean, defaulted shape while stripping fields you don't want to forward.

function normalizeResponse(res = {}) {
  const {
    data: { user = {}, items = [] } = {},     // nested destructure with fallbacks (won't crash if data missing)
    meta: { page = 1, total = 0 } = {},
    status = "unknown",
  } = res;

  const {
    id,
    name = "Anonymous",
    internalToken,          // pull out the sensitive field...
    ...publicUser           // ...and collect everything else via rest (omit internalToken)
  } = user;

  return {
    user: { id, name, ...publicUser },   // spread the safe fields back; internalToken excluded
    items,
    page,
    total,
    status,
  };
}

normalizeResponse({
  data: { user: { id: 1, name: "Ada", internalToken: "secret", email: "a@x.com" } },
  meta: { page: 2 },
});
// { user: { id:1, name:"Ada", email:"a@x.com" }, items:[], page:2, total:0, status:"unknown" }

Every feature earns its place: nested destructuring with = {} fallbacks means a missing data or meta yields sensible defaults instead of a crash (Part 7's guard); defaults fill in absent scalars; the rest pattern (...publicUser) strips internalToken cleanly by collecting everything except it; and spread reassembles the safe object. The result is a robust, declarative normaliser — the boundary-normalisation idea from the Types lesson, expressed with destructuring.

Interview Questions

Q1 (beginner): What's the difference between object and array destructuring?

Object destructuring matches by property name, so order is irrelevant: const { b, a } = { a: 1, b: 2 } binds correctly regardless of order, and you rename with key: newName. Array destructuring matches by position: const [a, b] = [1, 2] binds by index, you skip with empty slots, and there's no renaming (position is the identity). Both support defaults (fired only on undefined) and a trailing rest element/property.

Q2 (beginner): What does the spread operator do, and give two common uses.

Spread (...) in a value position expands an iterable or object into its individual elements/properties. Common uses: shallow-copying ([...arr], { ...obj }); merging ([...a, ...b], { ...defaults, ...overrides }); immutable updates ({ ...state, count: 1 }); and passing an array as individual function arguments (Math.max(...nums)). It's shallow, so nested objects are shared, not cloned.

Q3 (mid): Spread and rest use the same ... syntax — how do you tell them apart?

By position. In a value position — inside a function call, an array/object literal, an argument list — ... is spread: it expands one collection into many elements. In a binding position — a function parameter list, or the left-hand side of a destructuring pattern — ... is rest: it collects many values into a single array or object. They're inverse operations: spread unpacks, rest gathers. A useful tell: rest must always be the last element (nothing can follow what "collects the remainder"), while spread can appear anywhere in a value list.

Q4 (mid): How would you remove a property from an object immutably in one line?

Use rest in object destructuring: const { unwanted, ...rest } = obj; — this pulls unwanted out into its own variable and collects every other property into a brand-new rest object, leaving the original obj untouched. So rest is obj minus unwanted, created immutably. It's the idiomatic way to strip a field (e.g. removing a password or an internal token before forwarding data), far cleaner than copying and delete-ing, and it doesn't mutate the source. For a dynamic key name you'd fall back to building the object with Object.fromEntries(Object.entries(obj).filter(...)).

Q5 (senior): Explain the evaluation order and semantics of nested destructuring with defaults, using const { a: { b = 1 } = {} } = obj.

Read it inside-out and right-to-left through the = defaults. First, obj.a is looked up. If obj.a is undefined, the default = {} supplies an empty object to destructure instead — this is the guard that prevents "cannot destructure of undefined" when a is absent. Then, from whichever object resulted (the real obj.a or the {} fallback), property b is read; if that is undefined, the inner default = 1 applies. So b ends up as: obj.a.b if present; 1 if obj.a exists but b is missing/undefined; and 1 if obj.a itself is missing (because the = {} gives an object with no b). The two defaults serve different roles — = {} guards the nesting level against a missing parent, and = 1 provides the leaf value's fallback. Crucially, both only trigger on undefined, not null: if obj.a is null, the = {} does not fire and destructuring null throws — a real bug source when APIs send null for empty objects. That's why boundary code sometimes normalises null→undefined before destructuring, or uses explicit null-checks instead of relying on defaults.

Q6 (senior): You're reviewing code that heavily uses spread for immutable updates in a hot render path. What performance and correctness concerns would you raise?

Correctness first: spread is shallow, so { ...state, nested: state.nested } shares the nested reference — if later code mutates nested, both the "new" and "old" state see it, breaking the immutability guarantee and any ===-based change detection (React memo, PureComponent, useMemo deps). Every level you intend to change must be spread; forgetting one silently reintroduces shared mutable state. Performance: each spread allocates a new object/array and copies all top-level properties, so spreading a large object or a large array on every render/iteration is O(n) per update and generates garbage-collector pressure in hot paths — spreading a 10k-element array inside a render loop, for instance, is a real cost. Concerns/remedies I'd raise: (1) confirm the spread depth actually matches the mutation depth (correctness); (2) for large or deeply-nested state, prefer structural-sharing tools (Immer, or a setIn helper) that copy only the changed path and reuse untouched branches, giving both correctness and less copying; (3) avoid spreading inside tight loops — build the final array once rather than spreading to append per iteration ([...acc, item] in a reduce is accidentally O(n²)); (4) for very large immutable collections, consider persistent data structures (Immutable.js) whose updates are sub-linear. The judgment: spread is perfect for small, shallow immutable updates, but on a hot path with large or deep data its shallow semantics (correctness traps) and full-copy cost (performance) both argue for structural-sharing alternatives — verified by profiling, not assumed.

🏋️ Exercises

1 (easy): Swap and rename. Given const p = { lat: 51.5, lng: -0.1 }, destructure into latitude and longitude, and separately swap two variables a=1, b=2 without a temp.
Solution 
const { lat: latitude, lng: longitude } = p;   // 51.5, -0.1
let a = 1, b = 2;
[a, b] = [b, a];                                // a=2, b=1

Renaming uses key: newName; the swap uses array destructuring to assign both at once from a temporary array literal.

2 (easy): Head and tail. Split [1, 2, 3, 4, 5] into the first element and an array of the rest, in one line.
Solution 
const [head, ...tail] = [1, 2, 3, 4, 5];   // head=1, tail=[2,3,4,5]

The rest element collects everything after head into a new array. It must be last.

3 (medium): Omit a field. Write withoutPassword(user) returning a copy of user without its password property, immutably, in one expression.
Solution 
const withoutPassword = ({ password, ...rest }) => rest;
withoutPassword({ id: 1, name: "Ada", password: "x" });   // { id: 1, name: "Ada" }

Destructuring the parameter pulls password out and collects the remainder into rest, which is returned — a new object without the field, source untouched.

4 (medium): Merge with defaults. Write configure(options) that merges caller options over defaults { retries: 3, timeout: 5000, verbose: false }, working even when called with no argument.
Solution 
const DEFAULTS = { retries: 3, timeout: 5000, verbose: false };
const configure = (options = {}) => ({ ...DEFAULTS, ...options });
configure({ timeout: 1000 });   // { retries:3, timeout:1000, verbose:false }
configure();                     // all defaults

The default parameter = {} handles the no-argument call; spread merges options over defaults with the caller winning. (Note this is shallow — nested option objects would need the deep-merge from the Objects lesson.)

5 (hard): Safe nested extract. Given a possibly-incomplete res, extract res.data.user.name defaulting to "Guest" and res.data.items defaulting to [], without any intermediate crash.
Solution 
function extract(res = {}) {
  const { data: { user: { name = "Guest" } = {}, items = [] } = {} } = res;
  return { name, items };
}
extract({});                                        // { name:"Guest", items:[] }
extract({ data: { user: { name: "Ada" } } });       // { name:"Ada", items:[] }

Each nesting level has a = {} fallback so a missing parent yields an empty object to destructure rather than throwing, and the leaves have value defaults. (An optional-chaining approach — res?.data?.user?.name ?? "Guest" — is often more readable for deep single-value reads; both are valid.)

6 (hard): Reorder function to options object. Refactor function connect(host, port, useTls, timeout, retries) {} into a destructured options-object signature with sensible defaults, and show two call sites that are now clearer.
Solution 
function connect({ host, port = 443, useTls = true, timeout = 5000, retries = 3 } = {}) {
  return { host, port, useTls, timeout, retries };
}
connect({ host: "api.app.com" });                       // everything else defaulted
connect({ host: "db.local", port: 5432, useTls: false }); // named, order-free, self-documenting
// vs the old: connect("db.local", 5432, false, 5000, 3) — which boolean was which?

The options object makes each argument named and order-independent, lets callers omit any option (each has a default), and evolves additively — a new option is a new key, not a breaking positional change. This is the Functions-lesson API-scalability point realised with destructuring.

🎯 Key Takeaways

  • Object destructuring matches by name (rename with key: new); array destructuring by position (skip with empty slots); both take defaults (fired only on undefined).
  • Spread (value position) expands: shallow-copy, merge, insert, and pass arrays as arguments — but it's shallow.
  • Rest (binding position) collects: variadic parameters, remaining array elements, and remaining object properties (the clean immutable "omit").
  • Spread and rest share ... but do opposite jobs, told apart by position — spread unpacks, rest gathers (and rest is always last).
  • The destructured options-object parameter is the scalable-API pattern; nested destructuring with = {} fallbacks safely unpacks messy data.
  • Watch the traps: destructuring null/undefined throws, defaults skip null, and destructuring-assignment needs wrapping parens.
