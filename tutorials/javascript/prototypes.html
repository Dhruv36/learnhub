Prototypes & the Prototype Chain Objects & Data

What you'll master: the mechanism underneath all of JavaScript's object system — the prototype chain — how property lookup walks it, how new and constructor functions wire it up, the difference between __proto__ and prototype (the most confusing pair of names in the language), and why understanding this makes classes (next lesson) obvious rather than magical. Prerequisites: Objects and this. This is the conceptual bedrock: classes, inheritance, and every built-in method you call (arr.map, str.trim) are prototype chain lookups.

Part 1 — Every Object Has a Prototype

const obj = { a: 1 };
Object.getPrototypeOf(obj);          // Object.prototype — obj's prototype (its "parent")
obj.toString();                       // "[object Object]" — toString isn't on obj; it's INHERITED
obj.hasOwnProperty("a");              // true — hasOwnProperty also lives on Object.prototype

Object.getPrototypeOf(Object.prototype);  // null — the top of the chain, the end of the line

Every JavaScript object has an internal link to another object called its prototype. When you read a property that the object doesn't have directly, the engine follows this link to the prototype and looks there — and if not found, to that object's prototype, and so on. This is why obj.toString() works even though you never defined toString: it's found on Object.prototype, the prototype of all plain objects. The chain ends at null (the prototype of Object.prototype), and if a property isn't found anywhere along the way, the result is undefined.

Part 2 — Property Lookup Walks the Chain

const animal = { eats: true, walk() { return "walking"; } };
const dog = Object.create(animal);   // dog's prototype IS animal
dog.barks = true;

dog.barks;    // true    — own property
dog.eats;     // true    — NOT own; found on animal (one hop up the chain)
dog.walk();   // "walking"— found on animal too
dog.flies;    // undefined— not found anywhere in the chain

// The lookup order for dog.eats:
// 1. Does dog have own "eats"?      No.
// 2. Does animal (dog's proto)?     Yes → return true. Stop.

Object.create(proto) makes a new object with proto as its prototype — the most direct way to see the chain. Reading a property searches own properties first, then up the prototype chain, returning the first match. Crucially, writing is different: assigning dog.eats = false creates an own property on dog that shadows the inherited one — it never modifies animal. So reads climb the chain, but writes always land on the object itself (with one exception: inherited setters). This asymmetry is the heart of prototypal inheritance.

Part 3 — prototype vs __proto__ (the Naming Nightmare)

function Dog(name) { this.name = name; }
Dog.prototype.bark = function () { return `${this.name} barks`; };  // methods go on .prototype

const d = new Dog("Rex");
d.bark();                             // "Rex barks"

// The two confusingly-named things:
Dog.prototype                         // the object that will BECOME the prototype of instances
Object.getPrototypeOf(d) === Dog.prototype   // true — d's prototype IS Dog.prototype
d.__proto__ === Dog.prototype         // true — __proto__ is the (legacy) accessor for an object's prototype
Dog.prototype.constructor === Dog     // true — .prototype has a back-pointer to the function

This is the single most confusing naming in JavaScript, so pin it down precisely:

  • SomeFunction.prototype is a property on constructor functions. It's the object that will be used as the prototype of instances created with new SomeFunction(). It is not the function's own prototype.
  • object.__proto__ (or Object.getPrototypeOf(object), the preferred form) is any object's actual prototype link — the thing lookups follow.

The relationship: new Dog() creates an object whose __proto__ is set to Dog.prototype. So methods you put on Dog.prototype become available to every instance via the chain. Use Object.getPrototypeOf/Object.setPrototypeOf in real code; __proto__ is a legacy accessor kept for compatibility (and a prototype- pollution vector — Security lesson).

Part 4 — How new Works (Four Steps)

function Person(name) {
  // 1. a new empty object is created
  // 2. its __proto__ is set to Person.prototype
  // 3. `this` is bound to that new object (this-binding lesson: the `new` rule)
  this.name = name;        // 4a. the body runs, usually assigning properties to `this`
  // 4b. if the function returns no object, the new object is returned automatically
}
const p = new Person("Ada");

// Equivalent, by hand:
function myNew(Ctor, ...args) {
  const obj = Object.create(Ctor.prototype);   // steps 1 + 2
  const result = Ctor.apply(obj, args);        // step 3 + 4a: run body with this = obj
  return (result && typeof result === "object") ? result : obj;   // step 4b
}

new does exactly four things: creates a fresh object, links its prototype to the constructor's .prototype, calls the constructor with this bound to the new object, and returns that object (unless the constructor explicitly returns a different object). The hand-written myNew above is new — understanding it demystifies both constructor functions and classes (which are constructor functions with nicer syntax). Forgetting new is a classic bug: without it, this is undefined/global and properties leak or throw (classes prevent this by throwing helpfully).

Part 5 — Why Methods Go on the Prototype (Memory)

// ❌ Methods defined in the constructor — one COPY PER INSTANCE:
function BadDog(name) {
  this.name = name;
  this.bark = function () { return `${this.name} barks`; };  // new function object every instance
}
// 10,000 BadDogs → 10,000 bark functions

// ✅ Methods on the prototype — ONE shared copy for all instances:
function GoodDog(name) { this.name = name; }
GoodDog.prototype.bark = function () { return `${this.name} barks`; };
// 10,000 GoodDogs → 1 bark function, shared via the chain

This is the core practical reason the prototype exists: methods on the prototype are shared by all instances, defined once, whereas methods created inside the constructor are duplicated per instance. For a handful of objects it doesn't matter; for thousands it's a real memory difference (the exact trade-off from the Closures lesson's private-state Q6). This is also why classes put methods on the prototype automatically — you get the sharing for free. Data (like name) belongs on the instance; behaviour (methods) belongs on the prototype.

Part 6 — Prototypal Inheritance Chains

function Animal(name) { this.name = name; }
Animal.prototype.eat = function () { return `${this.name} eats`; };

function Dog(name) {
  Animal.call(this, name);              // call the parent constructor with this instance (super)
}
Dog.prototype = Object.create(Animal.prototype);  // Dog.prototype's proto IS Animal.prototype
Dog.prototype.constructor = Dog;         // restore the constructor back-pointer
Dog.prototype.bark = function () { return `${this.name} barks`; };

const d = new Dog("Rex");
d.bark();   // "Rex barks" — own prototype (Dog.prototype)
d.eat();    // "Rex eats"  — inherited (Animal.prototype, two hops up)
d instanceof Dog;     // true
d instanceof Animal;  // true — instanceof walks the chain

Multi-level inheritance is just a longer chain: d → Dog.prototype → Animal.prototype → Object.prototype → null. Setting it up by hand (the pattern above) requires three careful steps — call the parent constructor (Animal.call(this, ...), the manual "super"), link the prototypes with Object.create, and restore constructor. This boilerplate is exactly what class ... extends automates (next lesson) — classes are syntactic sugar over this chain-wiring, which is why understanding the mechanism makes classes trivial. instanceof returns true if a constructor's .prototype appears anywhere in the object's chain.

Part 7 — Inspecting & Extending (Carefully)

obj.hasOwnProperty("x");              // own property? (but see the safer form below)
Object.hasOwn(obj, "x");             // modern, safe even if obj overrode hasOwnProperty
"x" in obj;                          // own OR inherited?
Object.getPrototypeOf(obj);          // read the prototype
Object.create(null);                 // an object with NO prototype — no inherited keys at all

// ❌ NEVER extend built-in prototypes in shared code:
Array.prototype.last = function () { return this[this.length - 1]; };  // DON'T

Use Object.hasOwn(obj, key) to test own properties (safer than obj.hasOwnProperty, which breaks if the object has an own property named hasOwnProperty or a null prototype). Object.create(null) makes a "dictionary" object with no prototype — no inherited toString/__proto__, which is both faster for pure key-value use and safe from prototype-pollution. And the firm rule: never extend built-in prototypes (Array.prototype, Object.prototype) in shared/library code — it risks colliding with future language features or other libraries and makes for...in loops leak your additions. (This is how early libraries like MooTools broke when the language later added methods with the same names.)

Part 8 — Build It: a Prototype-Based Store Without Classes

To make the mechanism concrete, let's build a small reactive store using raw prototypes — the same thing a class would express, so you can see exactly what the class sugar hides.

function Store(initial) {
  this.state = initial;             // per-instance DATA
  this.listeners = [];              // per-instance data
}
// Behaviour on the prototype — shared by every Store instance:
Store.prototype.get = function () { return this.state; };
Store.prototype.set = function (next) {
  this.state = next;
  this.listeners.forEach(fn => fn(this.state));   // notify (this = the instance, via method call)
};
Store.prototype.subscribe = function (fn) {
  this.listeners.push(fn);
  return () => { this.listeners = this.listeners.filter(l => l !== fn); };  // unsubscribe
};

const store = new Store({ count: 0 });
const off = store.subscribe(s => console.log("changed:", s));
store.set({ count: 1 });            // logs "changed: { count: 1 }"
off();                               // unsubscribe
Object.getPrototypeOf(store) === Store.prototype;  // true — the wiring new set up

Every instance gets its own state and listeners (data on the instance), but all three methods live once on Store.prototype and are shared via the chain (behaviour on the prototype) — the memory-efficient split from Part 5. this inside each method is the instance, because they're called as store.method() (the implicit-binding rule from the this lesson). Rewriting this as class Store { get() {...} } produces identical runtime structure — which is precisely the point of the next lesson.

Interview Questions

Q1 (beginner): What is the prototype chain?

Every object has an internal link to another object, its prototype. When you access a property the object doesn't have as an own property, the engine looks up the chain — to the prototype, then that object's prototype, and so on — returning the first match, or undefined if the chain ends (at null) without finding it. This is how inheritance works in JavaScript: arr.map is found on Array.prototype, obj.toString on Object.prototype. Reads climb the chain; writes create own properties on the object itself (shadowing inherited ones).

Q2 (beginner): What's the difference between __proto__ and prototype?

prototype is a property that exists on constructor functions — it's the object that becomes the prototype of instances created with new. __proto__ (or better, Object.getPrototypeOf(obj)) is the actual prototype link on any object — the reference the engine follows during lookup. The connection: new Foo() creates an object whose __proto__ is Foo.prototype. So you put shared methods on Foo.prototype, and every instance reaches them because its __proto__ points there. Use Object.getPrototypeOf/setPrototypeOf in real code; __proto__ is a legacy accessor.

Q3 (mid): Explain what new does step by step.

Four steps: (1) create a new empty object; (2) set that object's prototype (__proto__) to the constructor's .prototype; (3) call the constructor with this bound to the new object (the new binding rule); (4) return the new object automatically — unless the constructor explicitly returns a different object, in which case that is returned instead (a returned primitive is ignored). You can replicate it: const obj = Object.create(Ctor.prototype); const r = Ctor.apply(obj, args); return (r && typeof r === "object") ? r : obj;. Forgetting new means step 3's this isn't the new object, so properties are assigned to the global object or throw in strict mode.

Q4 (mid): Why put methods on the prototype instead of in the constructor?

Memory and sharing. A method defined inside the constructor (this.m = function(){}) creates a brand-new function object for every instance, so N instances hold N copies of the same logic. A method on the prototype (Ctor.prototype.m = function(){}) exists once and is shared by all instances through the prototype chain — one function object regardless of instance count. For a few objects the difference is negligible, but for thousands it's a meaningful memory saving, and it also lets engines optimise the shared method. The rule: instance-specific data goes on this in the constructor; shared behaviour (methods) goes on the prototype. Classes do this automatically — class methods land on the prototype.

Q5 (senior): How does class ... extends map onto the prototype mechanism? What does super compile to?

A class is essentially a constructor function plus automatic prototype wiring. class Dog extends Animal sets up two links: instance methods on Dog.prototype whose own prototype is Animal.prototype (so instances inherit Animal's methods through the chain), and Dog's own prototype set to Animal (so static methods are inherited too — an extra link the manual pattern often omits). super(...) in the constructor corresponds to calling the parent constructor with the current instance — conceptually Animal.call(this, ...), though classes enforce it more strictly: in a derived class this doesn't exist until super() runs (accessing it before throws), because the parent constructor is what actually creates/initialises the instance in the class model. super.method() in a regular method looks up method starting from the parent's prototype (using the home-object's prototype, roughly Object.getPrototypeOf(Dog.prototype).method.call(this)), so it invokes the overridden parent version with the current this. The manual ES5 pattern — Dog.prototype = Object.create(Animal.prototype); Dog.prototype.constructor = Dog; Animal.call(this) — is exactly what classes generate, minus the static-inheritance link and the stricter this/super semantics. Knowing this is why "classes are sugar over prototypes" is accurate but slightly incomplete — they add real guarantees (static inheritance, super resolution, must-call-super) on top of the sugar.

Q6 (senior): What is prototype pollution, and how does the prototype chain make it dangerous?

Prototype pollution is an attack where untrusted input writes to Object.prototype (or another shared prototype), typically via a key like __proto__, constructor, or prototype in a recursive merge/clone/set-path operation. Because the prototype chain means every plain object inherits from Object.prototype, polluting it injects a property that suddenly appears on all objects application-wide. The danger is twofold: (1) logic subversion — code that checks if (obj.isAdmin) or reads a config flag may now see an attacker-controlled inherited value on objects that never had it set, enabling privilege escalation or auth bypass; (2) in some sinks it escalates to remote code execution — e.g. polluting a property that a template engine or child-process spawn later reads as code/arguments. It happens because unsafe deep-merge/clone functions treat these special keys as ordinary and walk into obj.__proto__, writing onto the shared prototype instead of the object. Defenses: reject/skip __proto__/constructor/prototype keys in merge/set operations; use Object.create(null) or Map for user-controlled key-value data (no prototype to pollute); Object.freeze(Object.prototype) as a hardening measure; validate input shapes with a schema; and use well-audited libraries for deep merge. This is a real, high-severity vulnerability class (many CVEs in popular npm packages), and it exists precisely because the prototype chain shares one object across all instances — the same mechanism that makes inheritance efficient makes pollution global. The Security lesson covers it in depth.

🏋️ Exercises

1 (easy): Trace the lookup. For const a = { x: 1 }; const b = Object.create(a); b.y = 2;, state where each is found: b.y, b.x, b.toString, b.z.
Answers

b.y → own property of b. b.x → inherited from a (one hop). b.toString → inherited from Object.prototype (two hops, since a's prototype is Object.prototype). b.z → undefined (not found anywhere in the chain).

2 (easy): Constructor with shared method. Write a Circle(radius) constructor storing radius on the instance and an area() method shared on the prototype. Prove two circles share the same method function.
Solution 
function Circle(radius) { this.radius = radius; }
Circle.prototype.area = function () { return Math.PI * this.radius ** 2; };

const a = new Circle(1), b = new Circle(2);
a.area();   // ~3.14
b.area();   // ~12.57
a.area === b.area;   // true — one shared function on the prototype

Data (radius) is per-instance; area lives once on the prototype and is shared, which the === check confirms.

3 (medium): Implement Object.create. Write myCreate(proto) that returns a new object whose prototype is proto, without using Object.create.
Solution 
function myCreate(proto) {
  function F() {}          // an empty constructor
  F.prototype = proto;      // its instances will have `proto` as their prototype
  return new F();           // new links the instance's __proto__ to F.prototype (= proto)
}
const parent = { greet() { return "hi"; } };
const child = myCreate(parent);
child.greet();                            // "hi" — inherited
Object.getPrototypeOf(child) === parent;  // true

This is literally how Object.create was polyfilled before it was native: set an empty function's .prototype to the desired proto, then new it so the instance's prototype link points there.

4 (medium): instanceof by hand. Write myInstanceof(obj, Ctor) that returns true iff Ctor.prototype appears in obj's prototype chain, without the instanceof operator.
Solution 
function myInstanceof(obj, Ctor) {
  let proto = Object.getPrototypeOf(obj);
  while (proto !== null) {
    if (proto === Ctor.prototype) return true;
    proto = Object.getPrototypeOf(proto);   // climb one link
  }
  return false;
}
myInstanceof([], Array);    // true
myInstanceof([], Object);   // true (Array.prototype's proto is Object.prototype)
myInstanceof([], String);   // false

It walks the chain link by link, returning true if it ever hits Ctor.prototype, and stopping at null — exactly what the real instanceof does.

5 (hard): Manual inheritance. Set up Cat inheriting from Animal using ES5 prototype wiring (no class), with Cat calling the parent constructor and adding a meow(). Verify instanceof for both.
Solution 
function Animal(name) { this.name = name; }
Animal.prototype.describe = function () { return `${this.name} the animal`; };

function Cat(name) { Animal.call(this, name); }         // "super" — run parent ctor on this instance
Cat.prototype = Object.create(Animal.prototype);         // link prototypes
Cat.prototype.constructor = Cat;                          // restore constructor pointer
Cat.prototype.meow = function () { return `${this.name} says meow`; };

const c = new Cat("Felix");
c.meow();               // "Felix says meow"
c.describe();           // "Felix the animal" — inherited
c instanceof Cat;       // true
c instanceof Animal;    // true

The three-step pattern — parent-constructor call, Object.create link, restore constructor — is precisely what class Cat extends Animal automates. Doing it by hand once makes the class version transparent.

6 (hard): Null-prototype dictionary. Explain why Object.create(null) is safer than {} for storing user-supplied keys, and show a case where a plain object misbehaves that a null-prototype object doesn't.
Solution 
const plain = {};
"toString" in plain;             // true — inherited from Object.prototype!
plain["__proto__"] = { x: 1 };   // doesn't create a normal key; reassigns the prototype (footgun)

const dict = Object.create(null);
"toString" in dict;              // false — no prototype, no inherited keys
dict["__proto__"] = { x: 1 };    // just a normal key now — safe
dict.__proto__;                  // { x: 1 } — an ordinary property, no special meaning

A plain object inherits from Object.prototype, so keys like toString, constructor, and especially __proto__ collide with inherited members — "toString" in userKeys is wrongly true, and assigning __proto__ can reassign the prototype instead of storing a value (a prototype-pollution vector). Object.create(null) has no prototype, so every key is an ordinary own property with no inherited baggage — the correct, safe choice for a pure dictionary of untrusted keys (a Map is the other correct choice).

🎯 Key Takeaways

  • Every object links to a prototype; property reads climb the chain to the first match (or undefined), while writes create shadowing own properties.
  • Constructor.prototype (on functions) becomes instances' prototype; Object.getPrototypeOf(obj)/__proto__ is an object's actual prototype link — don't conflate them.
  • new = create object, link its prototype to Ctor.prototype, run the constructor with this bound, return the object.
  • Put shared behaviour (methods) on the prototype (one copy for all instances); put per-instance data on this in the constructor.
  • Inheritance is a longer chain; the ES5 wiring (parent-call + Object.create + restore constructor) is exactly what class extends automates.
  • Object.hasOwn/Object.create(null) for safe key handling; never patch built-in prototypes; the shared chain is why prototype pollution is dangerous.
