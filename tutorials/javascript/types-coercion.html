The Type System & Coercion Language Core

What you'll master: JavaScript's eight types and the primitive-vs-reference divide that governs copying and comparison, why typeof lies about two things, how primitives borrow methods from wrapper objects, and — the part that dominates interviews — the coercion rules: which ones are genuinely worth memorising and which you should simply avoid triggering by converting explicitly. Prerequisite: Variables & Declarations (the name→value / primitive-vs-reference picture). The equality operators and truthiness get their own full treatment next in Operators & Equality; here we build the type foundation they rest on.

Part 1 — The Eight Types

// SEVEN primitives — immutable, compared BY VALUE:
"text"          // string
42              // number  (one numeric type for ints AND floats — Numbers lesson)
true            // boolean
undefined       // "no value assigned yet" — the language's absence
null            // "deliberately empty" — the programmer's absence
10n             // bigint  — integers beyond 2^53 (Numbers lesson)
Symbol("id")    // symbol  — unique, unforgeable keys (Collections lesson)

// ONE reference type — mutable, compared BY REFERENCE:
{}  []  function(){}  new Date()  new Map()   // all "object"

Seven primitives plus objects — that's the whole type system. The single most consequential line in it is the divide: primitives are copied and compared by value; objects by reference. That one fact, established in the previous lesson, explains aliasing bugs, why {a:1} !== {a:1}, and why you need spread/structuredClone to copy — we'll see all three land throughout this track.

Part 2 — typeof, and Its Two Lies

typeof "hi"        // "string"       typeof 42          // "number"
typeof true        // "boolean"      typeof undefined   // "undefined"
typeof 10n         // "bigint"       typeof Symbol()    // "symbol"
typeof {}          // "object"       typeof function(){}// "function"  ← convenient inconsistency
typeof []          // "object"       ← LIE #1: arrays report as object
typeof null        // "object"       ← LIE #2: the 30-year-old bug

typeof is your first-line type check, but know its two famous falsehoods. Arrays report "object" — to distinguish, use Array.isArray(x), the only reliable test (it even works across iframes/realms where x instanceof Array can fail). typeof null is "object" — a bug from the first implementation that backward-compatibility froze forever; so the correct "is this a non-null object?" check is:

function isObject(x) {
  return x !== null && typeof x === "object";   // the null guard is mandatory
}
🧠 Why typeof function is "function" but array is "object"? Functions are callable objects; the spec gives them their own typeof result because "can I call this?" is a common, useful question. Arrays got no such privilege — hence Array.isArray. There's no deep principle; it's history you memorise.

Part 3 — Primitives Have Methods? The Wrapper-Object Illusion

"hello".toUpperCase();   // "HELLO" — but strings are primitives with no methods... how?
(42).toFixed(2);         // "42.00"
true.toString();         // "true"

Primitives are not objects and have no methods. When you call a method on one, the engine temporarily wraps it in a corresponding wrapper object (String, Number, Boolean), calls the method on that throwaway object, and discards the wrapper. This "autoboxing" is why primitives appear to have methods. The consequence you must not miss: you cannot store state on a primitive, because the wrapper is thrown away instantly:

let s = "hi";
s.custom = 5;        // creates a wrapper, sets .custom on it, discards the wrapper
console.log(s.custom); // undefined — the property vanished with the wrapper

Corollary: never use the wrapper constructors explicitly. new String("hi") makes a real object (typeof "object", always truthy, !== the primitive "hi") — a well-known footgun. Use String(x)/Number(x) without new for conversion; those return primitives.

Part 4 — Coercion: the Model, Not the Table

Coercion is JavaScript automatically converting a value from one type to another. It happens in three situations, and understanding which conversion each triggers beats memorising a giant table:

  1. To string — when a value meets + with a string, or lands in a template literal / String().
  2. To number — with every arithmetic operator except +-with-a-string, and with comparison operators < > <= >=.
  3. To boolean — in if, while, &&, ||, !, ternary conditions (truthiness — next lesson).
// String coercion (the + operator, when EITHER side is a string):
"5" + 3      // "53"   — 3 becomes "3", concatenation
"5" + true   // "5true"
[] + []      // ""     — both arrays stringify to "" (empty join)
[] + {}      // "[object Object]"  — the infamous one; array→"", object→"[object Object]"

// Number coercion (every other arithmetic operator):
"5" - 3      // 2      — "5" becomes 5
"5" * "2"    // 10     — both become numbers
true + 1     // 2      — true→1, false→0
null + 1     // 1      — null→0
undefined+1  // NaN    — undefined→NaN
"abc" - 1    // NaN    — non-numeric string→NaN

The + operator is the whole problem child: it's both numeric addition and string concatenation, and if either operand is (or coerces to) a string, it concatenates; otherwise it adds. Every other math operator (- * / % **) is unambiguously numeric and coerces both sides to number. That single asymmetry explains the bulk of "why is my number a string?" bugs — typically a value read from an input or JSON arriving as a string and hitting +.

How objects coerce: toString / valueOf / Symbol.toPrimitive

const money = {
  amount: 5,
  valueOf() { return this.amount; },          // used in numeric contexts
  toString() { return `$${this.amount}`; }    // used in string contexts
};
money + 1        // 6      — numeric context → valueOf
`${money}`       // "$5"   — string context → toString
money * 2        // 10     — valueOf again

When an object must become a primitive, the engine asks it — via Symbol.toPrimitive if defined, else valueOf/toString in an order set by the "hint" (number vs string context). Plain objects have no meaningful valueOf, so they fall back to toString → "[object Object]", which is why {} + 1 tends to be "[object Object]1". You rarely define these, but knowing the protocol demystifies every weird object-in-arithmetic result and is a genuine senior interview topic.

Part 5 — Explicit Conversion: Say What You Mean

The professional stance on coercion is: don't rely on implicit coercion; convert explicitly so intent is on the page. The tools:

// To number:
Number("42")       // 42
Number("42px")     // NaN   — the WHOLE string must be numeric
Number("")         // 0     — empty string → 0 (a trap)
Number("  42  ")   // 42    — trims whitespace
parseInt("42px",10)// 42    — reads LEADING digits, stops at non-digit; ALWAYS pass the radix
parseFloat("3.14x")// 3.14
+"42"              // 42    — unary plus, terse; same rules as Number()

// To string:
String(42)         // "42"
String(null)       // "null"       — String() handles null/undefined; (null).toString() THROWS
(42).toString()    // "42"
(255).toString(16) // "ff"         — radix for number→string too

// To boolean:
Boolean(0)         // false
!!"hi"             // true          — double-NOT idiom; identical to Boolean()

Part 6 — NaN: the Value That Isn't Equal to Itself

typeof NaN          // "number"  — yes, NaN is a number (it means "failed numeric operation")
NaN === NaN         // false     — the ONLY value not equal to itself
[NaN].includes(NaN) // true      — includes uses SameValueZero, which DOES match NaN
[NaN].indexOf(NaN)  // -1        — indexOf uses ===, which does NOT
Number.isNaN(NaN)   // true      — the correct test
isNaN("abc")        // true      — WRONG TOOL: global isNaN coerces first ("abc"→NaN)
Number.isNaN("abc") // false     — "abc" is not the NaN value; it's a string

NaN ("not a number," but confusingly still of type number) is produced by failed numeric operations (0/0, Number("x"), Math.sqrt(-1)). Its defining weirdness — NaN !== NaN — means you cannot test for it with equality. Use Number.isNaN(x) (no coercion, checks it's actually the NaN value), never the global isNaN(x) which coerces its argument first and so reports non-numbers as "NaN" too. This ===-vs-SameValueZero split also explains why includes finds NaN but indexOf doesn't.

Part 7 — Build It: A Robust toNumber Coercion Helper

Real code constantly receives "numbers" as strings (form inputs, query params, CSV cells). Let's build one honest converter that treats empty/whitespace and non-numeric input as failures rather than silently producing 0 or NaN downstream.

function toNumber(input) {
  if (typeof input === "number") return Number.isFinite(input) ? input : null;
  if (typeof input === "bigint") return Number(input);
  if (typeof input !== "string") return null;      // reject objects/booleans/null explicitly
  const trimmed = input.trim();
  if (trimmed === "") return null;                 // Number("") is 0 — we refuse that trap
  const n = Number(trimmed);                        // strict: whole string must be numeric
  return Number.isNaN(n) ? null : n;
}

toNumber("42")      // 42
toNumber("  3.14 ") // 3.14
toNumber("42px")    // null   — not silently 42
toNumber("")        // null   — not silently 0
toNumber(null)      // null
toNumber(NaN)       // null   — non-finite rejected
toNumber(1e309)     // null   — Infinity rejected (isFinite)

Every decision here is a coercion rule made explicit and safe: the empty-string→0 trap closed, the partial-parse laxity of parseInt avoided, non-finite results rejected, and a single sentinel (null) for "couldn't convert" so callers branch once. This is what "convert explicitly" looks like in production — the implicit rules from Part 4 are exactly what this helper exists to not depend on.

Interview Questions

Q1 (beginner): List the JavaScript types. Which is the odd one out and why?

Seven primitives — string, number, boolean, undefined, null, bigint, symbol — plus one reference type, object (which includes arrays, functions, dates, maps, etc.). The odd one out is object: it's the only reference type, meaning it's mutable and compared/ copied by reference, whereas all seven primitives are immutable and compared/copied by value. That primitive-vs-reference split is the single most important structural fact of the type system.

Q2 (beginner): What does typeof return for null, [], and a function, and how do you correctly detect each?

typeof null is "object" (a historic bug) — detect null with x === null. typeof [] is "object" — detect arrays with Array.isArray(x). typeof function(){} is "function" — that one's honest, and you can rely on it. The general "plain non-null object" check is x !== null && typeof x === "object" && !Array.isArray(x).

Q3 (mid): Explain why "5" + 3 is "53" but "5" - 3 is 2.

The + operator is overloaded: it's both numeric addition and string concatenation, and if either operand is a string, it concatenates — so 3 is coerced to "3" and you get "53". Every other arithmetic operator (- * / % **) is only numeric, so - coerces "5" to the number 5 and subtracts, giving 2. This asymmetry is the source of most "my number became a string" bugs, which is why you convert explicitly with Number() before arithmetic on values that might be strings.

Q4 (mid): How do primitives appear to have methods, and what does that imply you can't do?

Autoboxing: when you call a method on a primitive, the engine temporarily wraps it in the corresponding wrapper object (String/Number/Boolean), invokes the method on that wrapper, and discards it. So the methods live on the wrapper's prototype, not on the primitive. The implication: you can't persist state on a primitive — let s="hi"; s.x = 1; creates a wrapper, sets x on it, then throws the wrapper away, so s.x reads back undefined. It also means never use new String() etc., which create real, always-truthy objects distinct from the primitive.

Q5 (senior): Walk through exactly how [] + {} and an object like {valueOf(){return 5}} * 2 get evaluated.

When an operator needs a primitive from an object, it runs the ToPrimitive algorithm with a "hint." For [] + {}: + uses hint "default," which tries valueOf first then toString. Arrays' and plain objects' valueOf return the object itself (not a primitive), so both fall through to toString: [] → "" (empty join), {} → "[object Object]"; then + sees two strings and concatenates → "[object Object]". For {valueOf(){return 5}} * 2: * uses hint "number," which tries valueOf first; it returns the primitive 5, so the object becomes 5 and the result is 10. If Symbol.toPrimitive is defined it takes precedence over both and receives the hint directly. Knowing this protocol is what turns "weird object arithmetic" from mystery into a mechanical trace.

Q6 (senior): Your API returns JSON where numeric IDs sometimes arrive as strings ("42") and sometimes as numbers (42), and downstream code does if (id === selectedId). Diagnose the class of bug and propose a robust boundary strategy.

The bug class is type inconsistency at a trust boundary producing failing strict-equality checks: "42" === 42 is false, so the comparison silently mismatches for exactly the records whose ID arrived in the "wrong" representation — an intermittent, data-dependent bug that's painful to reproduce. The wrong "fix" is to switch to == (loose equality), which papers over the inconsistency and invites worse coercion bugs elsewhere. The robust strategy is normalisation at the boundary: the moment data crosses from the network into your domain, run it through a parsing/validation layer (hand-written, or a schema library like Zod) that coerces every ID to one canonical type — and importantly, decide whether that type is number or string based on the domain. For 64-bit IDs the correct choice is string, because such IDs exceed Number.MAX_SAFE_INTEGER and coercing them to number would corrupt them (Numbers lesson) — so you normalise everything to string and compare as strings. The principle: coerce once, explicitly, at the edge, into a documented canonical type; never let mixed-type values flow into equality checks deep in the code, and never reach for == to hide the inconsistency.

🏋️ Exercises

1 (easy): typeof drill. Predict typeof for each: null, [], NaN, 10n, function(){}, undefined, Symbol(). Then write the one-liner that correctly reports "array" for arrays and "null" for null while otherwise returning typeof.
Answers

"object", "object", "number", "bigint", "function", "undefined", "symbol". Helper:

const kind = x => x === null ? "null" : Array.isArray(x) ? "array" : typeof x;
2 (easy): Coercion predictions. Evaluate by hand, then verify: "5" + 3, "5" - 3, true + true, null + 1, undefined + 1, [] + [], "" == 0 (just predict; equality is next lesson).
Answers

"53", 2, 2, 1, NaN, "", and "" == 0 is true (both coerce to 0 — the reason == is avoided). Rule of thumb: + with any string concatenates; everything else coerces to number, where true→1, null→0, undefined→NaN.

3 (medium): NaN detection. Write containsNaN(arr) returning true iff the array holds a NaN. Show why arr.includes(NaN) works but arr.indexOf(NaN) !== -1 doesn't.
Solution 
const containsNaN = arr => arr.some(Number.isNaN);   // or arr.includes(NaN)

includes uses the SameValueZero algorithm, which treats NaN as equal to NaN, so it finds it. indexOf uses ===, and NaN === NaN is false, so it never matches — returning -1. Number.isNaN in some is the most explicit form.

4 (medium): Wrapper footgun. Explain why the following logs what it does, and fix it to log true: 
const a = new String("hi");
const b = "hi";
console.log(a === b, typeof a, Boolean(new Boolean(false)));
Solution

It logs false "object" true. new String("hi") is a wrapper object, so a === b compares object-vs-primitive (false) and typeof a is "object". Worse, new Boolean(false) is a truthy object (all objects are truthy), so Boolean(...) of it is true despite wrapping false. Fix: drop new — use the primitive "hi" and false directly (or String(x)/Boolean(x) for conversion), and a === b becomes true.

5 (hard): Implement toPrimitive awareness. Create a Temperature object that behaves as its Celsius number in arithmetic but as "22°C" in string contexts, using Symbol.toPrimitive. Prove all three hints.
Solution 
const temp = {
  c: 22,
  [Symbol.toPrimitive](hint) {
    if (hint === "number") return this.c;
    if (hint === "string") return `${this.c}°C`;
    return this.c;                    // "default" (e.g. +): behave numerically here
  }
};
temp + 1        // 23    (default hint)
temp * 2        // 44    (number hint)
`${temp}`       // "22°C"(string hint)
String(temp)    // "22°C"

Symbol.toPrimitive, when present, wins over valueOf/toString and receives the context hint directly, giving you full control over each conversion — the cleanest way to make a value dual-natured.

6 (hard): Boundary normaliser. Write normalizeUser(raw) that takes an untrusted object from JSON and returns { id: string, age: number|null, active: boolean }, coercing each field explicitly and rejecting nonsense (e.g. age: "old" → null, active: "true" → true, active: 0 → false). Reuse the toNumber idea from Part 7.
Solution 
function normalizeUser(raw) {
  const toNumber = v => {
    const n = Number(typeof v === "string" ? v.trim() : v);
    return typeof v !== "boolean" && v !== "" && v != null && Number.isFinite(n) ? n : null;
  };
  return {
    id: String(raw.id),                    // 64-bit-safe: always string (see Q6)
    age: toNumber(raw.age),
    active: raw.active === true || raw.active === "true" || raw.active === 1
  };
}
normalizeUser({ id: 42, age: "30", active: "true" });  // {id:"42", age:30, active:true}
normalizeUser({ id: "7", age: "old", active: 0 });     // {id:"7",  age:null, active:false}

Every field is coerced once, explicitly, at the boundary into a documented canonical type — IDs to string (64-bit-safe), age to a validated number-or-null, active to a real boolean with an explicit allow-list rather than relying on truthiness. This is the "coerce at the edge" strategy from the senior interview question, in code.

🎯 Key Takeaways

  • Seven primitives (value-copied, immutable) + one reference type object (reference-copied, mutable) — that divide governs copying, comparison, and mutation bugs.
  • typeof lies twice: null and arrays both report "object" — use x===null and Array.isArray; functions honestly report "function".
  • Primitives borrow methods via throwaway wrapper objects (autoboxing) — so you can't store state on them, and you must never use new String/Number/Boolean.
  • Coercion has three targets: to-string (+ with a string, templates), to-number (all other math + comparisons), to-boolean (conditions). + is the overloaded troublemaker.
  • Objects coerce via Symbol.toPrimitive → valueOf/toString by hint — the mechanism behind every weird object-in-arithmetic result.
  • Convert explicitly (Number/String/Boolean, Number.isNaN) at boundaries; never depend on implicit coercion or the global isNaN.
