Design Patterns in JavaScript Production JavaScript

What you'll master: the patterns that actually recur in JavaScript codebases — module, singleton, factory, observer/pub-sub, strategy, decorator, adapter/facade, and proxy — each in its idiomatic JS form (which is often a closure or a plain function, not the class diagram from the Java book), plus the judgment layer: recognizing patterns you already use unnamed, and knowing when naming one adds clarity versus when it adds ceremony. This is a capstone — it assembles Closures, Functional Patterns, Classes, Events, and Proxy & Reflect into the shared vocabulary senior engineers use in design discussions and code review.

Part 1 — Why Patterns, and Why JS Patterns Look Different

// The GoF book (1994) targeted C++/Smalltalk: classes were the only tool, so every
// pattern is drawn as a class diagram. JavaScript has cheaper primitives:

// "Strategy pattern" in classic OO:          // …and in JavaScript:
// interface SortStrategy { sort(list) }      const strategies = {
// class QuickSort implements SortStrategy    {   price:  (a, b) => a.price - b.price,
// class Sorter { constructor(strategy) {…} }     name:   (a, b) => a.name.localeCompare(b.name),
//                                                newest: (a, b) => b.createdAt - a.createdAt,
//                                              };
//                                              items.toSorted(strategies[sortKey]);

// First-class functions + closures + object literals dissolve half the GoF catalog.
// The IDEAS survive; the ceremony doesn't.

A design pattern is a named, reusable solution-shape for a recurring design problem — the value is as much the shared vocabulary ("make it an observer", "wrap it in an adapter") as the structure itself. But patterns are shaped by their language: half the classic catalog existed to work around missing first-class functions. In JS, a strategy is a function in a lookup object; a command is a closure; an iterator is built into the language (generators). Knowing the JS-idiomatic form matters in both directions: you'll implement patterns with a tenth of the code, and you'll recognize that the "plain code" you already write — callbacks, config objects, event emitters — is patterns, and can be discussed as such in interviews.

Part 2 — Module & Singleton: Encapsulation and "Exactly One"

// MODULE pattern — private state + public API. Modern form: an ES module IS one.
// cartStore.js
let items = [];                                  // module-scoped = private (not exported)
export const addItem  = (item) => { items = [...items, item]; notify(); };
export const getItems = () => [...items];        // expose a COPY, not the live array

// SINGLETON — "exactly one instance, shared by all importers".
// In JS you rarely write the class version, because MODULES ARE SINGLETONS:
// every importer of cartStore.js shares the same module instance (evaluated once, cached).

// When you DO need lazy/controlled construction:
let instance = null;
export function getDb() {
  if (!instance) instance = createConnection(process.env.DB_URL);   // lazy init, once
  return instance;
}

The module pattern is JS's native encapsulation: anything not exported is invisible outside the file, no #private fields or closures-over-IIFEs needed (the IIFE version of this pattern is what everyone wrote before ES modules — recognize it in legacy code). And because the module registry caches each module after first evaluation, every ES module is already a singleton — importing it twice yields the same instance. So the singleton pattern in JS is usually just "put it in a module"; write the lazy getInstance form only when construction must be deferred or configured at runtime. Singleton's bad reputation is worth knowing too: it's shared global state in a trenchcoat — it couples everything that touches it and makes tests order-dependent. The testable variant is creating one instance at the top and passing it down (dependency injection) rather than importing it everywhere.

Part 3 — Factory & Builder: Constructing Things Flexibly

// FACTORY FUNCTION — a function that returns objects; JS's default way to make things:
function createLogger({ level = "info", prefix = "" } = {}) {
  const should = (l) => ["debug","info","warn","error"].indexOf(l) >=
                        ["debug","info","warn","error"].indexOf(level);
  return {
    debug: (m) => should("debug") && console.debug(prefix + m),
    error: (m) => console.error(prefix + m),
  };
}
const log = createLogger({ level: "debug", prefix: "[auth] " });
// vs `new Logger(...)`: no `this`, no `new` to forget, real privacy via closure,
// and it can return ANY shape — a cached instance, a subtype, a proxy.

// ABSTRACT-factory flavor — pick the implementation behind one interface:
const makeStore = (env) =>
  env === "test" ? createMemoryStore() : createRedisStore(process.env.REDIS_URL);

// BUILDER — when construction has many optional steps (chainable, immutable-ish):
const query = new QueryBuilder("users").where("age", ">", 18)
                                       .orderBy("name").limit(10).build();
// In JS, a single OPTIONS OBJECT usually replaces Builder: fn({ where, orderBy, limit })
// — reach for a real builder only when steps are order-sensitive or incrementally composed.

A factory is just a function that makes objects — and in JS that's often better than new: callers are decoupled from the concrete "class" (the factory can swap implementations, cache, or wrap without callers changing — that's the makeStore example, and it's exactly how test doubles get injected), and closures give true privacy. This is also the "closure factory" from Functional Patterns viewed through a patterns lens. Builder solves "constructor with nine optional params" — but JS named-options objects with defaults already solve most of that, so the honest rule is: options object first, builder only when construction is genuinely stepwise (query builders, test-data builders).

Part 4 — Observer / Pub-Sub: Decoupling Who-Knows-Whom

// OBSERVER — subjects you subscribe to directly. Tiny, complete implementation:
function createEmitter() {
  const handlers = {};                            // event → Set of callbacks
  return {
    on(event, fn) {
      (handlers[event] ??= new Set()).add(fn);
      return () => handlers[event].delete(fn);    // return UNSUBSCRIBE — leak prevention
    },
    emit(event, data) {
      handlers[event]?.forEach(fn => fn(data));
    },
  };
}
const cart = createEmitter();
const off = cart.on("item-added", (item) => updateBadge(item));
cart.emit("item-added", { sku: "A1" });
off();                                            // component unmounts → MUST unsubscribe

// You already use this pattern daily:
// DOM: el.addEventListener / removeEventListener       (observer on elements)
// Node: EventEmitter                                    (same shape, built in)
// PUB-SUB variant: a central BROKER between emitters and listeners —
// publishers and subscribers don't know each other AT ALL (looser, harder to trace).

Observer inverts dependencies: instead of the cart knowing about the badge, the analytics module, and the inventory syncer (three imports, growing forever), interested parties subscribe to the cart's events — the subject stays closed while listeners come and go. This is the pattern behind DOM events, Node's EventEmitter, RxJS, and every state library's subscribe(). The two production lessons hiding in the small print: always capture and call the unsubscribe function (a subscribed handler is a reference — the #1 memory-leak source in long-lived UIs, see Memory Management), and know the observer/pub-sub distinction: pub-sub adds a central broker so publisher and subscriber are fully decoupled — maximum flexibility, but "who reacts to this event?" is no longer answerable by reading the code, which is why overusing global event buses turns codebases into murder mysteries.

Part 5 — Strategy, Decorator, Adapter & Facade: Wrapping and Swapping

// STRATEGY — swap the algorithm, keep the skeleton (Part 1's lookup object is the idiom):
const pricingStrategies = {
  regular:  (total) => total,
  member:   (total) => total * 0.9,
  campaign: (total) => Math.max(0, total - 15),
};
const checkout = (cart, kind) => pricingStrategies[kind](sum(cart));
// replaces: if (kind === "member") … else if (kind === "campaign") … — OPEN for extension

// DECORATOR — same interface, added behavior, stackable:
const withRetry  = (fn, n = 3)   => async (...a) => { /* try n times */ return fn(...a); };
const withTiming = (fn, label)   => async (...a) => {
  const t0 = performance.now();
  try { return await fn(...a); }
  finally { console.log(label, performance.now() - t0, "ms"); }
};
const getUser = withTiming(withRetry(fetchUser, 3), "getUser");   // compose the cross-cutting bits
// getUser has fetchUser's EXACT signature — callers can't tell it's wrapped.

// ADAPTER — convert an interface you HAVE into the one you NEED:
const paymentAdapter = (stripe) => ({
  charge: (cents, token) => stripe.paymentIntents.create({ amount: cents, source: token }),
});      // app code depends on OUR `charge()` shape — vendor swap touches ONE file

// FACADE — one simple front door over a messy subsystem:
async function placeOrder(cart, user) {         // hides: validate → reserve stock →
  …                                             // charge → email → analytics
}                                               // callers get ONE call with a clear name

Four wrapping-shaped patterns, distinguished by intent: strategy swaps the varying algorithm behind a stable skeleton (kills else-if ladders; adding a case = adding an entry, not editing tested code). Decorator layers cross-cutting behavior — retry, timing, caching, auth — onto a function without touching it, preserving its interface so decorators stack in any order; this is memoize/throttle/debounce's family, and React's HOCs. Adapter reshapes a third-party interface into the one your app defines, confining vendor coupling to one file. Facade doesn't reshape — it simplifies, giving a subsystem one well-named front door. In review you'll tell them apart by asking: swapping behavior (strategy), adding behavior (decorator), translating an interface (adapter), or hiding complexity (facade)?

Part 6 — Deep Dive: Proxy-Powered Patterns & the Judgment Layer

// JS Proxy makes some patterns nearly free (see metaprogramming.html for the full API):
// VIRTUAL PROXY / lazy loading — stand in for an expensive object until first use:
const lazyReport = new Proxy({}, {
  get(_, prop) {
    realReport ??= buildExpensiveReport();       // built on FIRST property access
    return realReport[prop];
  }
});

// OBSERVER via Proxy — react to mutations without explicit emit() calls:
const state = new Proxy({ count: 0 }, {
  set(obj, prop, value) {
    obj[prop] = value;
    render(obj);                                 // Vue's reactivity = exactly this idea
    return true;
  }
});
state.count++;                                   // assignment triggers re-render

// The judgment layer — pattern-picking heuristics that survive code review:
// 1. Patterns are DISCOVERED in refactoring, not installed up front.
//    Third else-if on the same axis → strategy. Third boolean flag param → factory/options.
//    Module imported by everyone and mutated → event/observer or DI.
// 2. The pattern must pay for its indirection with a REAL axis of change.
// 3. If the idiomatic JS version is a plain function/object — write that. The NAME
//    belongs in the conversation, not necessarily in class names.

Proxy gives JS something the GoF authors had to hand-build: transparent interception. Lazy initialization, change-tracking, validation walls, and access logging become a handler object instead of a wrapper class per pattern — it's how Vue's reactivity and many ORM/mocking libraries work under the hood. But the deeper senior skill is the judgment layer. Pattern-itis — AbstractLoggerFactorySingleton for a 40-line script — is a real disease, and the cure is treating patterns as refactoring targets: write the direct code first; when the second or third variation appears along the same axis, then extract the strategy/factory/observer, and the pattern earns its indirection with demonstrated (not speculated) change.

Senior framing — patterns as vocabulary, architecture as consequence: in system-design and code-review conversations, patterns are compression: "adapter around the payment SDK, events for checkout side-effects, strategies for pricing tiers" communicates an architecture in one sentence, including what can change cheaply (swap vendors, add listeners, add tiers) — that's what the interviewer is probing when they ask "how would you structure this?". The anti-pattern vocabulary matters equally: god object (one module knows everything), event-bus soup (nothing traceable), singleton-as-global (hidden coupling everywhere), premature abstraction (the wrong-guess interface a pattern froze in place too early). Strong engineers name both the pattern and its failure mode in the same breath — that's the difference between having read the catalog and having lived with the consequences.

Interview Questions

Q1 (Beginner): What is a design pattern, and what's the singleton pattern's simplest form in modern JavaScript?

A named, reusable solution-shape for a recurring design problem — as much shared vocabulary as structure. The simplest JS singleton is an ES module: modules are evaluated once and cached, so every importer shares the same instance of whatever the module creates. The classic getInstance() lazy form is only needed when construction must be deferred or parameterized at runtime.

Q2 (Beginner): You have if (type === "csv") … else if (type === "json") … else if (type === "xml") … repeated in three functions. Which pattern, and what does it look like in JS?

Strategy: put the varying behavior in a lookup object — const exporters = { csv: toCsv, json: toJson, xml: toXml } — and call exporters[type](data) (with a guard for unknown types). Adding a format becomes adding one entry instead of editing three if-chains; each strategy is independently testable; and in JS no interface/class machinery is needed because functions are first-class.

Q3 (Mid): Observer vs pub-sub — difference, one production risk of each, and a codebase example of observer you use daily.

Observer: subscribers register directly with the subject (cart.on("change", fn)) — they know the subject. Pub-sub adds a central broker; publishers and subscribers know only the broker and topic names. Observer's risk: forgotten unsubscribes leak memory by pinning listeners (and their closures) alive. Pub-sub's risk: total decoupling makes flows untraceable — "who reacts to ORDER_PLACED?" requires a global search, and event-chains can trigger each other invisibly. Daily example: DOM addEventListener/removeEventListener (or Node's EventEmitter, RxJS, any store's subscribe).

Q4 (Mid): Show how to add retry and timing to an existing fetchUser without modifying it, and name the pattern and its one hard rule.

Decorator — higher-order functions that wrap and preserve the interface: const getUser = withTiming(withRetry(fetchUser, 3), "getUser"), where each wrapper takes a function and returns one with the identical signature that adds its behavior around the call. The hard rule: the interface must be preserved exactly (parameters, return shape, thrown types) — that's what makes decorators stackable in any order and invisible to callers. Same family as memoize, throttle, debounce, and middleware.

Q5 (Senior): Your app integrates Stripe directly in 30 files, and leadership wants the option to switch providers. Which patterns fix this, and how do they interact?

Adapter + facade + factory working together. Define the interface your domain needs (charge, refund, customer) and write a Stripe adapter implementing it — vendor types and quirks now live in one file. Expose it through a payments facade that owns cross-cutting policy (idempotency keys, error normalization, logging), which is what the 30 call sites migrate to. A factory (makePaymentProvider(config)) selects the adapter per environment — which also makes a fake adapter injectable for tests. Honest senior caveat: provider-agnostic abstractions leak (webhooks, auth flows, and error taxonomies differ), so design the interface around your actual use-cases, not the union of every provider's features.

Q6 (Senior): A teammate proposes factories, strategies, and an event bus in the design for a brand-new small feature. How do you evaluate that?

Ask what demonstrated axis of change each indirection serves. Patterns pay rent with flexibility along an axis that actually varies; installed speculatively, they usually guess the wrong axis and then resist the change that really comes, while making every reader traverse indirection meanwhile. Counter-proposal: write the direct version; extract a strategy at the second algorithm variant, a factory when construction genuinely branches, events when a third decoupled consumer appears — patterns as refactoring targets, not upfront scaffolding (YAGNI applied to structure). Exception worth conceding: boundaries that are known volatile on day one (a payment vendor already scheduled to change) justify their adapter immediately. The evaluation is evidence-based flexibility, not pattern-count.

🏋️ Exercises

1 (Easy): Convert this to a strategy lookup: function ship(kind, w) { if (kind === "air") return w * 9; else if (kind === "sea") return w * 2 + 50; else if (kind === "road") return w * 4; } — with a helpful error for unknown kinds.
Solution
const shippingRates = {
  air:  (w) => w * 9,
  sea:  (w) => w * 2 + 50,
  road: (w) => w * 4,
};
function ship(kind, weight) {
  const rate = shippingRates[kind];
  if (!rate) throw new Error(`Unknown shipping kind "${kind}". Valid: ${Object.keys(shippingRates).join(", ")}`);
  return rate(weight);
}
// Adding "rail" = one line in the object; ship() and its tests never change.
2 (Easy): Write a factory createCounter(start) returning { increment, decrement, value } where the count is truly private, and show two independent counters.
Solution
function createCounter(start = 0) {
  let count = start;                       // closure = real privacy, no `this`, no `new`
  return {
    increment: () => ++count,
    decrement: () => --count,
    value:     () => count,
  };
}
const a = createCounter();
const b = createCounter(100);
a.increment(); a.increment();
console.log(a.value(), b.value());   // 2 100 — independent instances, no shared state
console.log(a.count);                // undefined — genuinely inaccessible
3 (Medium): Build createEmitter() supporting on (returning unsubscribe), once, and emit. Demonstrate that once fires a single time.
Solution
function createEmitter() {
  const handlers = {};
  const on = (event, fn) => {
    (handlers[event] ??= new Set()).add(fn);
    return () => handlers[event].delete(fn);
  };
  return {
    on,
    once(event, fn) {
      const off = on(event, (data) => { off(); fn(data); });  // self-unsubscribe before firing
      return off;
    },
    emit(event, data) { handlers[event]?.forEach(fn => fn(data)); },
  };
}
const em = createEmitter();
let calls = 0;
em.once("ping", () => calls++);
em.emit("ping"); em.emit("ping"); em.emit("ping");
console.log(calls);   // 1
4 (Medium): Write a decorator withCache(fn, ttlMs) for async single-argument functions: cache results per argument, expire after ttlMs, preserve the interface.
Solution
function withCache(fn, ttlMs) {
  const cache = new Map();                       // arg → { value, expires }
  return async (arg) => {                        // same signature as fn — callers can't tell
    const hit = cache.get(arg);
    if (hit && hit.expires > Date.now()) return hit.value;
    const value = await fn(arg);                 // misses/expired: delegate to the real fn
    cache.set(arg, { value, expires: Date.now() + ttlMs });
    return value;
  };
}
const getUserCached = withCache(fetchUser, 60_000);
// Decorator rules honored: adds behavior around the call, changes nothing about the
// interface, stacks with others: withTiming(withCache(withRetry(fetchUser, 3), 60_000), "u")
5 (Hard): Two weather vendors return different shapes: { temp_f, wind: { mph } } and { celsius, windKmh }. Define your app's interface, write both adapters, and a factory that picks by config.
Solution
// The interface OUR app defines (SI units, flat):  { tempC, windKmh }
const adaptVendorA = (api) => ({
  async current(city) {
    const d = await api.getWeather(city);                 // { temp_f, wind: { mph } }
    return { tempC: (d.temp_f - 32) * 5 / 9, windKmh: d.wind.mph * 1.609 };
  },
});
const adaptVendorB = (api) => ({
  async current(city) {
    const d = await api.fetchCurrent(city);               // { celsius, windKmh }
    return { tempC: d.celsius, windKmh: d.windKmh };      // near-passthrough is FINE —
  },                                                      // the adapter still isolates the vendor
});
const makeWeather = (cfg) =>
  cfg.vendor === "A" ? adaptVendorA(new VendorA(cfg.key))
                     : adaptVendorB(new VendorB(cfg.key));
// App code: const weather = makeWeather(config); await weather.current("Pune");
// Vendor differences (and a future vendor C, or a test fake) touch only this file.
6 (Hard): Using Proxy, build observable(target, onChange) that invokes onChange(prop, oldVal, newVal) on every property set — skipping no-op writes — and use it to drive a render function.
Solution
function observable(target, onChange) {
  return new Proxy(target, {
    set(obj, prop, value) {
      const old = obj[prop];
      if (Object.is(old, value)) return true;   // no-op write → no notification
      obj[prop] = value;
      onChange(prop, old, value);
      return true;                               // set traps must return true on success
    },
  });
}
const state = observable({ count: 0, user: "ada" },
  (prop, oldVal, newVal) => {
    console.log(`${prop}: ${oldVal} → ${newVal}`);
    render(state);                               // observer pattern with ZERO emit() calls —
  });                                            // mutation itself is the event (Vue's core idea)
state.count = 1;      // "count: 0 → 1" + render
state.count = 1;      // (silent — no-op skipped)
state.user = "grace"; // "user: ada → grace" + render

🎯 Key Takeaways

  • Patterns are shared vocabulary for solution-shapes — and JS idiom compresses them: strategy = function lookup, module/singleton = ES module (evaluated once, cached), command = closure, iterator = built-in.
  • Factories (functions returning objects) beat new for decoupling, closure-privacy, and swappable implementations; options objects usually replace Builder.
  • Observer inverts who-knows-whom (DOM events, EventEmitter, store.subscribe) — always keep the unsubscribe; broker-based pub-sub decouples further at the cost of traceability.
  • Wrapping patterns by intent: strategy swaps, decorator adds (interface preserved, stackable), adapter translates a vendor interface, facade hides a subsystem behind one door.
  • Proxy makes interception-based patterns (lazy init, reactivity, validation) nearly free — it's the engine of Vue-style reactivity.
  • Patterns are discovered in refactoring, not installed speculatively: extract at the second/third real variation, and let each indirection prove its axis of change — vocabulary in the design conversation, minimum ceremony in the code.
