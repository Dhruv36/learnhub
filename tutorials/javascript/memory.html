Memory Leaks & Profiling Ultra-Advanced

"JavaScript has garbage collection" ≠ "JavaScript can't leak". The GC frees what's unreachable — leaks are objects that stay reachable by accident. In an SPA that lives for hours (dashboards, editors, chat), small leaks compound into tab-killing memory and jank.

How GC Decides (mark & sweep)

Roots: globals, the current stack, active closures
  → everything reachable FROM roots is kept (marked)
  → everything else is freed (swept)
A leak = an accidental path from a root to something you thought was gone.

The Four Classic Leaks

// 1. Forgotten listeners/timers keep their closures (and captured DOM) alive:
setInterval(() => render(bigData), 1000);          // never cleared → bigData immortal
window.addEventListener("resize", onResize);        // component unmounted? listener remains
//    Fix: clearInterval, AbortController, framework cleanup hooks (useEffect return).

// 2. Detached DOM — removed from the page but referenced from JS:
const rows = [];
rows.push(document.querySelector("#table"));        // later: table.remove()
// The whole subtree stays in memory because the array points at it.

// 3. Ever-growing caches:
const cache = new Map();                             // unbounded — every entry forever
//    Fix: LRU with a max size, TTLs — or WeakMap (below).

// 4. Accidental globals & module-level accumulators (arrays that only ever .push).

Weak References — Caches That Don't Own Their Keys

const meta = new WeakMap();                 // key must be an object
meta.set(domNode, { rendered: Date.now() });
// When domNode becomes otherwise unreachable, the entry evaporates with it.
// WeakMap/WeakSet = metadata about objects you don't own. Not iterable — by design.
// FinalizationRegistry / WeakRef exist for exotic cases; needing them is usually a smell.

Finding Leaks with DevTools (the workflow)

Memory tab → Heap snapshot:
1. Take snapshot A
2. Do the suspicious action 5× (open/close the modal…)
3. Take snapshot B → compare, filter "Detached" to catch detached DOM
4. Any object with growing count: its "Retainers" panel shows the exact
   path from a GC root that keeps it alive — that path is your bug.

Performance tab → record with Memory checked: sawtooth = healthy churn;
a staircase that never comes down = leak.
💡 Staff-level insight: in interviews, connect leaks to closures — "the timer's closure captures the component, so the whole tree stays reachable" is the answer that shows you understand both lessons. In production, memory regressions are caught by CI budgets (Lighthouse/Playwright memory assertions on key flows), not by heroes with DevTools after the incident.
