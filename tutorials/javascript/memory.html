Memory Leaks & Profiling Ultra-Advanced

"JavaScript has garbage collection" ≠ "JavaScript can't leak". The GC frees what's unreachable; a leak is memory that stays reachable by accident. In pages that live minutes, nobody notices. In SPAs that live all day — dashboards, editors, chat, trading screens — small leaks compound into 2GB tabs, GC pauses (jank!), and crashes. This lesson: how GC actually decides, the four leak archetypes, and the DevTools workflow that finds them in minutes.

How GC Decides — Reachability, Not Cleverness

ROOTS: globals (window/globalThis) · the running stack · active closures · live timers/listeners
MARK:  everything reachable from roots, transitively
SWEEP: everything else is freed

⇒ A leak = an unintended PATH from a root to something you thought was gone.
  The Retainers panel (below) literally shows you that path.

Engine reality worth knowing (V8): generational GC — most objects die young and are
collected cheaply in the "nursery"; survivors get promoted and become expensive to collect.
Long-lived churn (caches that grow, arrays that only push) is what hurts.

The Four Leak Archetypes (with the fix pattern for each)

// 1. FORGOTTEN TIMERS & LISTENERS — the #1 by incident count
setInterval(() => render(bigData), 1000);        // never cleared → closure pins bigData FOREVER
window.addEventListener("resize", onResize);      // component unmounted? listener remains, and it
                                                  // captures the component → whole tree retained
// FIX PATTERN — symmetric lifecycle:
const ctl = new AbortController();
window.addEventListener("resize", onResize, { signal: ctl.signal });
const id = setInterval(tick, 1000);
function destroy() { ctl.abort(); clearInterval(id); }   // one teardown, always called
// (React's useEffect cleanup-return, Vue's onUnmounted — same contract, framework-flavored.)

// 2. DETACHED DOM — removed from the page, referenced from JS
const cache = [];
cache.push(document.querySelector("#bigTable"));  // later: bigTable.remove()
// The 5,000-row subtree is invisible but fully retained — the array points at it.
// FIX: null out references on teardown; or key caches by WeakRef-free design (see WeakMap).

// 3. UNBOUNDED CACHES & ACCUMULATORS
const responseCache = new Map();                  // every key forever; logs.push(entry) forever
// FIX: bound it — LRU with max size, TTL eviction, or scope the cache to a lifecycle.

// 4. ACCIDENTAL GLOBALS & MODULE-LEVEL STATE
leaked = computeHuge();                           // no declaration → global (modules/strict throw — one more reason for them)
export const registry = [];                       // module singleton that only grows

WeakMap / WeakSet — Caches That Don't Own Their Keys

const meta = new WeakMap();                 // keys must be objects
meta.set(domNode, { renderedAt: Date.now() });
// When domNode becomes otherwise unreachable, the entry EVAPORATES with it —
// impossible to leak through this cache. THE tool for "data about objects I don't own".
// Not iterable, no .size — by design (contents can vanish between any two lines).

// WeakRef + FinalizationRegistry exist for exotic interop — but if CORRECTNESS depends
// on WHEN GC runs (non-deterministic!), the design is broken. Prefer explicit lifecycles.

The DevTools Workflow — Finding a Leak in 10 Minutes

MEMORY PANEL → Heap snapshot workflow:
1. Get to a steady state, force GC (🗑 icon), take SNAPSHOT A
2. Perform the suspicious action 5–7× (open/close the modal, navigate & return…)
3. Force GC again, SNAPSHOT B → switch to "Comparison" view
4. Sort by #Delta: what grew by ~5–7? That's your leak (the count matching your
   repetitions is the smoking gun)
5. Select an instance → RETAINERS panel = the exact chain of references from a GC
   root keeping it alive. Read it bottom-up: "…held by listener in EventTarget map,
   held by window" → there's your uncleaned listener.
6. Filter "Detached" in the class list to catch detached-DOM leaks specifically.

PERFORMANCE PANEL → record with Memory ✓:
sawtooth (grow → GC drop → grow) = healthy churn
staircase that never comes down   = leak
Also: performance.memory / the Allocation instrumentation timeline for WHERE allocations happen.

Interview Questions

Q1: Users report the dashboard tab at 1.8GB by afternoon. Walk your investigation end-to-end.

Reproduce with the snapshot-diff workflow on the highest-frequency flows (panel open/close, data refresh, route switches); comparison view sorted by delta identifies the growing class; Retainers names the pinning path (typically: uncleared interval closure, listener on window, or an unbounded response cache). Fix, then lock it in with a CI memory assertion on that flow. Bonus signal: check the Performance memory graph first — staircase confirms leak vs "just heavy".

Q2: Why does a forgotten setInterval leak MORE than the interval callback itself?

The timer is a GC root holding the callback; the callback's closure holds everything it captured — data, DOM references, often the whole component (this). One line pins an object graph. It also keeps RUNNING — CPU + potential re-renders. The mechanics come straight from the closures lesson: closures hold live environments, and roots make them immortal.

Q3: WeakMap vs Map for caching — when is each correct?

Map when YOU own the lifecycle (bounded LRU of API responses keyed by string — strings can't be weak anyway). WeakMap when keys are objects owned elsewhere (DOM nodes, class instances): the association must die with the key. Litmus: "should this cache keep its keys alive?" If no → WeakMap. If keys are primitives → Map + explicit bounds, no choice.

Q4: What's a detached DOM node, why is it invisible-but-costly, and how do you hunt it specifically?

A node removed from the document but still referenced from JS (arrays, closures, framework caches): it renders nothing yet retains its whole subtree (and event listeners, and often images). Hunt: heap snapshot → filter "Detached" → Retainers shows who's pointing at it. Classic sources: node caches surviving route changes, "previous page" references, tooltip/portal managers.

🏋️ Practice (Playground + DevTools):
1. Build a leaky modal: open/close creates an un-removed window listener capturing a 5MB array. Prove the leak with the ×5 snapshot diff; read the Retainers path aloud; fix with AbortController; re-prove.
2. Create a detached-DOM leak (cache a removed table) and find it via the Detached filter.
3. Write a 50-line LRUCache (Map + max size) and swap it into an unbounded cache.
4. Record the Performance memory graph during a leaky loop vs the fixed version — screenshot sawtooth vs staircase.
5. Shrink a closure capture: interval pinning a whole object → destructured primitive only; verify with snapshots.

🎯 Key Takeaways

  • GC frees the unreachable; leaks are accidental reachability — Retainers shows the exact guilty path.
  • Four archetypes: timers/listeners, detached DOM, unbounded caches, accidental globals/module state.
  • Symmetric lifecycle (create↔destroy, AbortController) and bounded-everything are the architectural fixes.
  • WeakMap for metadata about objects you don't own — associations that can't outlive their subject.
  • Workflow: GC → snapshot → action ×5 → snapshot → comparison delta → Retainers; sawtooth healthy, staircase leak.
  • Leaks are caught by CI memory budgets on key flows — not by heroes at 3am.
