Memory Management & Leaks Production JavaScript

What you'll master: how the garbage collector actually decides what to free (reachability, not reference counting), the four classic leak patterns that recur in every real codebase, how WeakMap/WeakRef solve them structurally, and the DevTools workflow for finding a real leak instead of guessing. Prerequisites: Closures (what gets retained), Events (listener leaks), and Collections (WeakMap). This lesson connects ideas seeded across the whole track into one coherent mental model of memory.

Part 1 — How Garbage Collection Actually Works: Reachability

let obj = { data: "large payload" };   // obj is REACHABLE from a root (a global/local variable)
let obj2 = obj;                          // now TWO references point to the same object
obj = null;                              // one reference gone — but obj2 still points to it: NOT collected
obj2 = null;                              // NOW nothing references it — it becomes UNREACHABLE, eligible for GC

// GC does NOT use simple reference counting (which breaks on cycles) — it does MARK-AND-SWEEP:
// starting from "roots" (global object, currently-executing call stack), it walks every reachable
// reference, marking each object found. Anything NOT marked afterward is garbage — swept and freed.
let a = {}, b = {};
a.ref = b; b.ref = a;      // a CYCLE — each references the other
a = null; b = null;         // no external reference to EITHER anymore...
// ...but a mark-and-sweep GC still collects them: neither is reachable from any ROOT, cycle or not.

JavaScript's garbage collector frees memory based on reachability, not on how many references point to an object. Starting from a set of "roots" (global variables, the current call stack, active closures), the GC walks every reference it can reach and marks each object found as alive; anything left unmarked afterward — unreachable from any root — is garbage and gets freed. This is mark-and-sweep, and it's specifically designed to correctly collect reference cycles (two objects referencing each other) as long as nothing reachable from a root points into that cycle — a real advantage over naive reference-counting garbage collectors, which leak on cycles unless specially handled. The critical consequence: an object leaks not because you "forgot to free it" (there's no manual free in JS) but because something — anything — still holds a reference to it that you didn't intend. Every leak in this lesson is a variation of "an unintended reference kept something reachable."

Part 2 — Leak Pattern 1: Forgotten Timers and Intervals

function startPolling(element) {
  const id = setInterval(() => {
    element.textContent = new Date().toString();   // the CALLBACK closes over `element`
  }, 1000);
  // If the caller never calls clearInterval(id), this interval — and its closure holding
  // `element` — runs FOREVER, even after `element` is removed from the DOM and should be dead.
}
startPolling(document.querySelector("#clock"));
// The #clock element is now UNREACHABLE from the DOM tree once removed, but the running
// interval's closure still references it → it can NEVER be garbage collected.

A running setInterval (or an unresolved setTimeout chain) is itself a GC root-reachable reference — the browser/Node runtime keeps the timer, and therefore its callback closure, alive as long as the timer is active. If that closure references a DOM element or large data (per the Closures lesson's capture rules), the timer keeps it reachable forever, even after the element is removed from the page — a classic, extremely common leak. The fix is always the same: every setInterval/recurring setTimeout needs a matching clearInterval/clearTimeout tied to the lifecycle of whatever it's associated with (a component unmounting, a page navigating away).

Part 3 — Leak Pattern 2: Unremoved Event Listeners

function attachHandler(bigData) {
  const handler = () => console.log(bigData.length);   // closes over bigData (Closures lesson)
  window.addEventListener("resize", handler);
  // No corresponding removeEventListener anywhere → this listener, and bigData with it,
  // lives for the ENTIRE page lifetime — even if the component that created it is long gone.
}

This is the Events lesson's memory-leak warning, generalized: registering a listener on a long-lived target (window, document, a shared event bus) without ever removing it keeps the handler function — and everything its closure captured — reachable for the rest of the page's life. It's especially dangerous in single-page apps where "components" come and go without a full page reload: each mount that forgets to clean up its listeners adds another permanent leak, compounding over the session. The fix, as covered in Events: pair every addEventListener with removal on teardown, ideally via a shared AbortController signal so cleanup is a single, hard-to-forget call.

Part 4 — Leak Pattern 3: Detached DOM Nodes

const cache = {};                          // a plain object living for the app's whole lifetime
function cacheRow(id, rowElement) {
  cache[id] = rowElement;                   // keeps a reference to the DOM node
}
const row = document.querySelector("#row-5");
cacheRow(5, row);
row.remove();                                // removed from the VISIBLE DOM tree...
// ...but cache[5] STILL references it → the node is "detached" but NOT collectable:
// it's gone from the page visually, yet still fully alive in memory, along with its
// entire subtree (children, attached listeners, everything).

A detached DOM node is an element that's been removed from the visible document but is still referenced from JavaScript (a cache, a closure, a data structure) — Chrome DevTools literally labels these "Detached" in memory snapshots, and they're one of the most common real-world leaks in long-running single-page apps. The node itself, plus its entire subtree and any listeners still attached to it, stays fully in memory, invisible to the user but consuming real resources. This happens innocently: caching a reference "for later" and never clearing the cache entry when the element is removed. Fix: clear cache/data-structure entries when their associated DOM node is removed, or — better — use a WeakMap keyed by the element (Part 6), so the cache entry disappears automatically the moment nothing else references the node.

Part 5 — Leak Pattern 4: Global Variables and Accidental Accumulation

// Accidental global (sloppy mode — Variables lesson):
function leaky() { accidentalGlobal = "never declared with let/const/var"; }   // becomes a property of the GLOBAL object
// It's now a GC root forever, for the rest of the page's life — strict mode/modules throw on this instead.

// Unbounded caches/logs — growing collections nobody ever trims:
const requestLog = [];
function logRequest(req) { requestLog.push(req); }   // called on every request, NEVER removed, NEVER capped
// After enough calls, requestLog alone can consume gigabytes — a slow, "my app gets sluggish after
// hours of use" leak that's easy to miss in short manual testing.

Two related patterns. Accidental globals (assigning to an undeclared name in sloppy mode) become properties of the global object — permanently reachable roots, from the Variables lesson's strict-mode discussion; modules and strict mode convert this silent leak into a loud ReferenceError, one more reason to prefer them. Unbounded accumulation — an array or Map that only ever grows (a log, an event history, a cache with no size limit or expiry) — isn't a "leak" in the reachability sense (you genuinely still reference it), but it has the same symptom: memory grows without bound over the application's lifetime. This is why every cache needs an eviction policy (the Collections lesson's LRU cache) and every log/history needs a cap.

Part 6 — WeakMap, WeakSet & WeakRef: Structural Fixes

// WeakMap-keyed cache — the CORRECT fix for Part 4's detached-node problem:
const rowCache = new WeakMap();               // keys held WEAKLY — don't prevent GC
function cacheRow(rowElement, data) { rowCache.set(rowElement, data); }
const row = document.querySelector("#row-5");
cacheRow(row, { sorted: true });
row.remove();
// row = null (nothing else references it) → the row AND its rowCache entry become collectable TOGETHER
// — automatically, with no manual cache-clearing code required.

// WeakRef — a direct weak reference to a single object (rarer, more advanced; use WeakMap first):
let ref = new WeakRef(someLargeObject);
ref.deref();   // returns the object if still alive, or undefined if it's been collected
// FinalizationRegistry — run cleanup code AFTER an object is collected (advanced, non-deterministic timing):
const registry = new FinalizationRegistry((heldValue) => console.log(`${heldValue} was collected`));
registry.register(someObject, "someObject's label");

This is the Collections lesson's payoff, applied to leaks directly. WeakMap/WeakSet hold their keys weakly — a weak reference doesn't count toward reachability, so if nothing else references the key, it (and its associated value/entry) becomes collectable, automatically, with zero manual cleanup code. This is the structural fix for the detached-node and object-keyed-cache leak patterns: instead of remembering to manually clear a cache entry when its associated object goes away (easy to forget), using a WeakMap makes the cleanup automatic by construction. WeakRef and FinalizationRegistry are lower-level, rarer tools for direct weak references and post-collection cleanup callbacks — powerful but with non-deterministic timing (you can't know when collection happens), so reach for WeakMap/WeakSet first for the vast majority of cases.

Part 7 — Finding Leaks: the DevTools Workflow

// The reliable, systematic workflow (Chrome DevTools → Memory tab):
// 1. Load the page / reset to a known state.
// 2. Take a HEAP SNAPSHOT (baseline).
// 3. Perform the suspected-leaky action several times (open/close a modal, navigate a route, N times).
// 4. Force a GC (the trash-can icon) to collect anything that legitimately SHOULD be gone.
// 5. Take a SECOND heap snapshot.
// 6. Use "Comparison" view — sort by "# New" / size delta — objects that GREW by roughly your
//    repetition count (e.g. you clicked 10 times, and there are 10 extra of something) are your leak.
// 7. Inspect a suspect object's "Retainers" panel — it shows EXACTLY what's still referencing it,
//    walking back to a root — this is how you find the actual leaking reference, not just guess.

// The Performance tab's memory timeline is the quick SIGNAL (a sawtooth that never returns to
// baseline after GC = a leak); heap snapshots + comparison + retainers is how you DIAGNOSE it.

Don't guess at leaks — measure them. The Performance tab's memory graph gives you the first signal: normal usage shows a sawtooth pattern (memory grows, GC runs, drops back down); a leak shows the floor of that sawtooth rising over time even after GC runs — memory that should have been reclaimed wasn't. To find the specific cause: take a baseline heap snapshot, repeat the suspected action a known number of times (say, 10), force GC, take a second snapshot, and use the comparison view — an object type with roughly 10 extra instances is very likely your leak. Then the Retainers panel on a specific leaked instance shows you the exact chain of references keeping it alive, back to a root — turning "something is leaking" into "this specific addEventListener call on line 42 is the retainer." This snapshot-diff-retainer workflow is the professional, repeatable process — not staring at a growing number and guessing.

Part 8 — Build It: an Auto-Cleaning Component Registry

Let's build a small "component" system that demonstrates the structural fix directly: associating private data with DOM elements via WeakMap so cleanup happens automatically when elements are removed, with an AbortController for listener cleanup — combining Parts 3 and 6 into one leak-proof pattern.

const componentData = new WeakMap();     // element → { state, controller } — auto-collected with the element

function mountComponent(element, initialState) {
  const controller = new AbortController();          // one signal for ALL this component's listeners
  const state = { ...initialState };

  element.addEventListener("click", () => {
    state.clicks = (state.clicks ?? 0) + 1;
    console.log("clicks:", state.clicks);
  }, { signal: controller.signal });                  // tied to the controller — removable in one call

  const timer = setInterval(() => render(element, state), 5000);

  componentData.set(element, { state, controller, timer });   // WEAKLY keyed by the element itself
}

function unmountComponent(element) {
  const data = componentData.get(element);
  if (!data) return;
  data.controller.abort();          // removes ALL listeners registered with this signal, in one call
  clearInterval(data.timer);        // stop the recurring timer explicitly — WeakMap does NOT do this part!
  element.remove();                  // now nothing references element (once componentData's entry is gone too)
  // componentData's entry for `element` becomes collectable automatically, since the key is weak.
}

This combines every leak-prevention technique in the lesson. The WeakMap means if unmountComponent is somehow never called and the element is simply discarded some other way, the state object still doesn't leak — nothing strongly references it once the element itself is unreachable (Part 6's structural guarantee). But note the crucial limit that explains why unmountComponent still does explicit work: a WeakMap only breaks the "cache holds the element" reference direction — it does not know about or cancel the still-running setInterval (Part 2), which is a completely separate GC root keeping element reachable through its own closure regardless of the WeakMap. This is an important, easy-to-miss distinction: WeakMap solves one specific reference direction, not every leak automatically — you still need explicit teardown (controller.abort(), clearInterval) for resources that themselves hold strong references, and the WeakMap is a safety net for the remaining "did I forget to clear the cache" case, not a replacement for disciplined cleanup.

Interview Questions

Q1 (beginner): How does JavaScript's garbage collector decide what to free?

It's based on reachability, not reference counting. Starting from "roots" — global variables and the currently-executing call stack (including active closures) — the collector walks every reference it can follow and marks each object it reaches as alive. After this walk, anything left unmarked is unreachable from any root and is freed — this is "mark-and-sweep." The key implication: an object is eligible for collection the moment nothing reachable from a root points to it anymore, regardless of how many references existed before. There's no manual "free()" in JavaScript — you cause an object to become collectable by removing every reference to it (setting variables to null, clearing it from arrays/objects, removing listeners/timers that reference it).

Q2 (beginner): Why does an uncleared setInterval cause a memory leak?

A running interval is itself reachable from a root (the runtime keeps active timers alive), so its callback function — and everything that callback's closure captured — stays reachable for as long as the interval keeps running. If you never call clearInterval, the interval runs forever (or until the page unloads), which means anything its closure references (a DOM element, application state, large data) can never be garbage collected either, even after that element is removed from the page and logically should be gone. The fix is to always pair a setInterval with a corresponding clearInterval tied to whatever the interval is logically associated with — a component unmounting, a feature being turned off.

Q3 (mid): What is a detached DOM node, and how does it happen?

A detached DOM node is an element that's been removed from the visible document tree (via .remove(), replacing its parent's content, etc.) but is still referenced somewhere in JavaScript — a cache object, a closure, an array — so it remains fully alive in memory even though it no longer appears on the page. It happens innocently: caching a reference to an element "for later use" (e.g. keying a lookup table by element) and then removing the element from the DOM without also clearing that cache entry. The node, its entire subtree, and any listeners still attached to it all stay in memory. Chrome DevTools' heap snapshots explicitly flag these as "Detached" nodes, making them one of the easier leak types to spot once you know to look — the structural fix is caching such references in a WeakMap keyed by the element, so the cache entry disappears automatically once nothing else references the node.

Q4 (mid): How does WeakMap prevent the leaks a regular Map would cause?

A regular Map holds its keys strongly — as long as the Map itself is reachable, every key it contains is also kept reachable, even if nothing else in the program references that key anymore. So using a Map to cache data associated with DOM elements (or any objects) means every element you've ever cached stays alive for as long as the Map exists, regardless of whether the element was removed from the page — a slow, guaranteed leak that grows with usage. WeakMap holds its keys weakly: a weak reference doesn't count toward reachability, so if nothing else references a key, both the key and its associated value become eligible for garbage collection together, automatically. This makes WeakMap the correct choice whenever you're associating data with objects whose lifetime you don't want to manage manually — the cache cleans itself up as a structural property of how it's built, not because you remembered to write cleanup code.

Q5 (senior): Walk through the DevTools process for diagnosing a suspected memory leak in a single-page app, from initial signal to root cause.

Start with the signal: open the Performance (or Memory) tab's timeline and record while performing normal usage — a healthy app shows a sawtooth memory graph (usage rises, GC runs, drops back near baseline repeatedly); a leak shows the post-GC floor of that sawtooth climbing over time — memory that should have returned to baseline doesn't. Once you suspect a leak in a specific flow (e.g. opening and closing a modal repeatedly), move to diagnosis: (1) reset to a clean state and take a baseline heap snapshot; (2) perform the suspected action a known, fixed number of times (e.g. exactly 10 open/close cycles) — the specific count matters for the next step; (3) force a manual GC via the trash-can icon to ensure you're not counting objects that would have been legitimately collected anyway; (4) take a second heap snapshot; (5) switch to the "Comparison" view between the two snapshots, sorted by the "# New" or size-delta column — look for an object type/constructor with a count that roughly matches your repetition count (10 extra instances of some class after 10 modal cycles is a strong, specific signal, not a coincidence). (6) Select a suspect instance and open its Retainers panel, which shows the actual chain of references keeping it alive, walking back to a root — this converts "something related to the modal is leaking" into "the modal's close handler was never removed from the document listener, and that handler's closure captures the modal's entire state object," pointing at the exact line to fix. The discipline throughout is: never guess or "eyeball" a fix based on a hunch — the snapshot- diff-retainer workflow gives you the actual retaining reference, which is the only reliable way to fix a leak rather than changing code speculatively and hoping.

Q6 (senior): Design a leak-resistant architecture for a long-lived single-page application with many mounted/unmounted UI components over a session.

The core principle: every resource a component acquires must have an explicit, enforced release path tied to that component's lifecycle, and the architecture should make forgetting that release path hard rather than relying on developer discipline alone. Concrete measures: (1) Standardize a single teardown mechanism — every component gets one AbortController whose signal is passed to every listener it registers, and a documented convention that all timers/ intervals/subscriptions created during mount are recorded in a per-component cleanup list, invoked as a single destroy()/unmount() call. This turns "did I remember to remove everything" into "did I call one function," dramatically reducing missed cleanup. (2) Prefer weak references for cross-cutting caches — any cache or registry that associates metadata with DOM elements or component instances (not the component's own primary state, but auxiliary/derived data) should default to WeakMap/WeakSet, so even a missed explicit cleanup doesn't permanently leak that specific association. (3) Bound everything that accumulates — logs, history stacks, undo buffers, caches — with an explicit max size and eviction policy (LRU), since long sessions will otherwise slowly grow these without bound regardless of whether individual components leak. (4) Centralize subscriptions to global/shared resources (a global store, a WebSocket, a shared event bus) behind a subscription manager that ties each subscription to a component's lifecycle automatically, rather than components directly subscribing/unsubscribing by hand each time — reducing the surface area for a missed unsubscribe. (5) Automate detection in CI/testing — an automated test that mounts and unmounts a component N times and asserts heap growth stays bounded (or a manual, periodic heap-snapshot-comparison audit of key user flows) catches regressions before they reach production, rather than relying on users eventually reporting "the app gets slow after a while." (6) If using a framework (React, Vue), lean on its lifecycle hooks (useEffect cleanup functions, onUnmounted) which already encode "tie this resource's release to this component's lifetime" as a first-class pattern — much of this architecture is what such frameworks provide automatically, which is a strong argument for using one at scale rather than hand-rolling component lifecycle management. The unifying theme: leaks happen when resource acquisition and resource release are not structurally coupled — the architecture's job is to make that coupling automatic or, at minimum, impossible to forget through convention and tooling, rather than trusting every future contributor to remember every cleanup call by hand.

🏋️ Exercises

1 (easy): Spot the leak. Identify what's wrong and fix it: 
function startClock(el) {
  setInterval(() => { el.textContent = Date.now(); }, 1000);
}
Solution 
function startClock(el) {
  const id = setInterval(() => { el.textContent = Date.now(); }, 1000);
  return () => clearInterval(id);   // caller gets a way to stop it
}
const stop = startClock(document.querySelector("#clock"));
// later: stop();

The original never returns a way to clear the interval, so it (and its closure over el) runs forever. Returning a stop function ties the interval's lifetime to something the caller can control.

2 (easy): Fix the listener leak. This component never removes its resize listener. Fix it with a teardown function. 
function watchResize(callback) { window.addEventListener("resize", callback); }
Solution 
function watchResize(callback) {
  window.addEventListener("resize", callback);
  return () => window.removeEventListener("resize", callback);
}
const stopWatching = watchResize(() => console.log("resized"));
// later: stopWatching();

Returning an unsubscribe function is the standard pattern for making cleanup explicit and easy for the caller to trigger at the right time.

3 (medium): WeakMap cache. Rewrite a plain-object element cache to use WeakMap, and explain what changes about its collection behavior. 
const cache = {};
function cacheData(el, data) { cache[el.id] = data; }
Solution 
const cache = new WeakMap();
function cacheData(el, data) { cache.set(el, data); }   // key is the ELEMENT itself, not a string id

The original keys by el.id (a string) in a plain object, which never releases entries even after the element is removed. Keying the WeakMap directly by the element means once the element is unreferenced elsewhere, the cache entry is collected automatically — no manual removal needed, and no risk of an id collision either.

4 (medium): Bounded log. Rewrite an unbounded request log so it never holds more than 100 entries, dropping the oldest first. 
const log = [];
function logRequest(req) { log.push(req); }
Solution 
const log = [];
const MAX = 100;
function logRequest(req) {
  log.push(req);
  if (log.length > MAX) log.shift();   // drop the oldest — bounds memory regardless of app lifetime
}

Capping the array's size means memory usage plateaus instead of growing without bound over a long session — the simplest possible eviction policy, appropriate when you only need "the last N," not a true LRU.

5 (hard): Diagnose from a description. A single-page app's memory grows by roughly 2MB every time a user opens and closes a particular settings modal, and never drops back down even after forcing GC. Propose two concrete, plausible root causes tied to specific mechanisms from this lesson, and how you'd confirm which one it is.
Solution

Plausible cause 1: the modal's close handler removes it from the DOM (modal.remove()) but a listener was registered on document or window (for outside-click detection, keyboard shortcuts) that was never removed — that listener's closure keeps the modal's DOM subtree and any large state it captured reachable, even though the modal is visually gone (a detached-node leak driven by an unremoved listener). Plausible cause 2: the modal starts a setInterval (e.g. polling for validation, an autosave timer) that's never cleared on close, keeping its closure (and whatever data it references) alive and running indefinitely in the background. To confirm which: take a heap snapshot before and after opening+closing the modal several times (a fixed count, say 5), force GC, compare snapshots — if the growing object type is tied to DOM nodes/detached elements, suspect the listener; if it's an interval-related closure still executing (check the Performance timeline for continued activity tied to the modal after closing, or inspect active timers), suspect the unceared interval. The Retainers panel on a leaked instance will show definitively whether the retaining reference is an event listener registration or a still-running timer's closure — that's the deciding evidence, not a guess.

6 (hard): Auto-cleanup component. Extend the lesson's mountComponent/unmountComponent pair so that forgetting to call unmountComponent still doesn't leak the interval — using FinalizationRegistry as a last-resort safety net (acknowledging its non-deterministic timing).
Solution 
const componentData = new WeakMap();
const registry = new FinalizationRegistry((timer) => {
  clearInterval(timer);           // best-effort cleanup, runs at SOME POINT after the element is collected
  console.warn("Component was garbage collected without explicit unmount — interval cleared as a fallback.");
});

function mountComponent(element, initialState) {
  const controller = new AbortController();
  const state = { ...initialState };
  element.addEventListener("click", () => { state.clicks = (state.clicks ?? 0) + 1; }, { signal: controller.signal });
  const timer = setInterval(() => render(element, state), 5000);
  componentData.set(element, { state, controller, timer });
  registry.register(element, timer);   // fallback: if `element` is ever collected, clear this timer too
}

function unmountComponent(element) {
  const data = componentData.get(element);
  if (!data) return;
  data.controller.abort();
  clearInterval(data.timer);
  registry.unregister(element);        // proper cleanup happened — cancel the fallback, it's no longer needed
}

FinalizationRegistry lets you register a cleanup callback that runs at some indeterminate future point after the registered object is actually garbage collected — a genuine safety net for exactly the case this exercise asks about (a caller who forgets to unmount). But its timing is explicitly non-deterministic (it might run seconds or minutes later, or not before the page closes) and it must never be relied on as your primary cleanup mechanism — it's a last line of defense against bugs, not a substitute for calling unmountComponent properly. registry.unregister(element) on proper unmount avoids the fallback firing redundantly once real cleanup has already happened.

🎯 Key Takeaways

  • GC is reachability-based (mark-and-sweep from roots), not reference counting — an object leaks because something unintended still references it, not because you "forgot to free" it.
  • Uncleared timers and unremoved event listeners are the two most common leaks — both keep their closures (and captured data) reachable indefinitely; always pair setup with teardown.
  • Detached DOM nodes (removed from the page but still referenced in JS) stay fully alive — cache element references in a WeakMap, not a plain object, to fix this structurally.
  • Accidental globals (sloppy mode) and unbounded caches/logs are related but distinct issues — strict mode prevents the former; an eviction policy or size cap fixes the latter.
  • WeakMap/WeakSet hold keys weakly, enabling automatic cleanup — but they only break one reference direction; timers/listeners with their own strong references still need explicit teardown.
  • Diagnose leaks systematically: Performance timeline for the signal (rising post-GC floor), heap snapshot comparison for the culprit object, Retainers panel for the exact retaining reference.
