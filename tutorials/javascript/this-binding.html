this, call, apply & bind Language Core

What you'll master: the one rule that determines this in any normal function — it's set by how the function is called, not where it's defined — the four call-patterns and their precedence, why arrow functions ignore all of it, the "lost this" bug that bites everyone, and call/apply/bind for controlling it explicitly. Prerequisites: Functions & Scope and Closures (lexical scope — the thing this notably does not follow). this is the single most misunderstood keyword in JavaScript; this lesson makes it mechanical.

Part 1 — The Core Rule: this Is Set at the Call Site

function whoAmI() { return this; }

whoAmI();              // global/undefined — a plain call
const obj = { whoAmI };
obj.whoAmI();          // obj — called as a method OF obj

// SAME function, DIFFERENT this — because it was CALLED differently.

Here is the entire mental model, and it contradicts most people's first intuition: in a normal function, this is not fixed when the function is written — it's determined fresh on every call, by how the function is invoked. The exact same function gives different this values depending on the call syntax. This is the one place JavaScript is dynamically scoped (Functions lesson): lexical scope fixes variable names at author time, but this is bound at the call site. Master the four call patterns below and this stops being mysterious.

Part 2 — The Four Binding Rules (in Priority Order)

// 1. `new` binding — `this` is the brand-new object being constructed:
function Person(name) { this.name = name; }
const p = new Person("Ada");     // this === the new object → p.name === "Ada"

// 2. Explicit binding — call/apply/bind set `this` directly (Part 5):
whoAmI.call(obj);                // this === obj

// 3. Implicit binding — called as a method; `this` is the object left of the dot:
obj.whoAmI();                    // this === obj

// 4. Default binding — a plain call; `this` is undefined (strict) or globalThis (sloppy):
whoAmI();                        // this === undefined (in a module / strict mode)

When a normal function runs, this is resolved by asking these questions in order:

  1. Was it called with new? → this is the freshly created object.
  2. Was it called with call/apply/bind? → this is the explicitly provided object.
  3. Was it called as obj.method()? → this is obj (the thing left of the last dot).
  4. None of the above (a plain call)? → this is undefined in strict mode/modules, or globalThis in sloppy mode.

Higher rules win. This precedence list is the answer to virtually every "what is this here?" question — walk the call site through it and you have the answer mechanically, no guessing.

Part 3 — Arrow Functions Ignore All Four Rules

const obj = {
  name: "Ada",
  normal() { return this.name; },        // this = obj (implicit binding)
  arrow: () => this.name,                // this = enclosing scope's this (NOT obj!) → undefined
  delayed() {
    setTimeout(function () { console.log(this.name); }, 0);  // this = undefined/global (default) ✗
    setTimeout(() => console.log(this.name), 0);             // this = obj (inherited from delayed) ✓
  }
};
obj.normal();   // "Ada"
obj.arrow();    // undefined — arrow's this came from where it was DEFINED, not obj

Arrow functions have no this of their own. They don't participate in the four rules at all — instead, this inside an arrow is lexically inherited from the enclosing scope where the arrow was written (and it can never be changed, not even by call/bind). This is why arrows are perfect for callbacks nested inside a method (the setTimeout arrow keeps the method's this) and wrong as top-level object methods (there's no meaningful enclosing this, so it's the module scope). The whole reason arrow functions exist is to make this lexical and predictable in nested callbacks — the exact pain point the four dynamic rules create.

Part 4 — The "Lost this" Bug

const user = {
  name: "Ada",
  greet() { return `Hi, ${this.name}`; }
};
user.greet();                   // "Hi, Ada" — implicit binding, this = user

const fn = user.greet;          // extracted — the method is now just a function value
fn();                           // "Hi, undefined" — plain call, this = undefined! 💥

setTimeout(user.greet, 100);    // same bug — the timer calls it as a plain function
[1].forEach(user.greet);        // same — forEach calls the callback with its own this rules
btn.addEventListener("click", user.greet);  // this = the button, not user

This is the most common this bug in real code: the moment you detach a method from its object — assign it to a variable, pass it as a callback, hand it to setTimeout/addEventListener — you lose the implicit binding, because this depends on how it's called, and now it's called as a plain function (or by some API with its own this). The value this.name becomes undefined (or the function throws in strict mode). Three fixes, next.

Part 5 — call, apply, bind: Explicit Control

function intro(greeting, punct) { return `${greeting}, ${this.name}${punct}`; }
const user = { name: "Ada" };

// call — invoke NOW, this + args listed individually:
intro.call(user, "Hi", "!");        // "Hi, Ada!"

// apply — invoke NOW, this + args as an ARRAY:
intro.apply(user, ["Hi", "!"]);     // "Hi, Ada!"   (mnemonic: Apply = Array)

// bind — return a NEW function with this (and optionally args) permanently fixed; call it later:
const greetAda = intro.bind(user);
greetAda("Hey", ".");               // "Hey, Ada."
const sayHiAda = intro.bind(user, "Hi");   // partial application: fix this AND the first arg
sayHiAda("!!");                     // "Hi, Ada!!"
MethodInvokes?ArgsUse for
callImmediatelyListed individuallyOne-off invocation with a chosen this
applyImmediatelyAs an arrayWhen args are already in an array; spreading into variadic fns
bindLater (returns a fn)Optionally pre-fill leading argsFixing this for a callback; partial application

call and apply invoke immediately, differing only in how arguments are passed (individually vs as an array — "Apply = Array"). bind is the important one for the lost- this bug: it returns a new function with this permanently locked, which you can then pass around safely — setTimeout(user.greet.bind(user), 100) fixes the Part 4 bug. bind also does partial application (pre-filling leading arguments). Note a bound function's this can't be re-bound — the first bind wins.

Part 6 — Fixing the Lost-this Bug, Four Ways

// Given the detach problem: btn.addEventListener("click", user.greet)

// FIX A — bind at the point of passing:
btn.addEventListener("click", user.greet.bind(user));

// FIX B — wrap in an arrow (calls it as a method, preserving the dot):
btn.addEventListener("click", () => user.greet());

// FIX C — class field arrow (auto-bound per instance, common in React class components):
class C { name = "Ada"; greet = () => `Hi ${this.name}`; }   // greet's this is always the instance

// FIX D — bind in the constructor (older React pattern):
class D { constructor(){ this.greet = this.greet.bind(this); } greet(){ /*...*/ } }

All four ensure the method runs with the right this. The arrow wrapper (B) and class-field arrow (C) are the modern defaults — they're the clearest and hardest to get wrong. bind (A/D) is explicit and fine, just more verbose. Which you reach for depends on context, but the underlying fix is always the same: guarantee the function is called in a way that sets this correctly, rather than hoping the call site does.

Part 7 — this in Classes and Common APIs

class Counter {
  count = 0;
  increment() { this.count++; }          // this = the instance, WHEN called as instance.increment()
  incrementArrow = () => { this.count++; } // auto-bound: this is ALWAYS the instance
}
const c = new Counter();
const inc = c.increment;
// inc();               // 💥 lost this — undefined.count++
const incA = c.incrementArrow;
incA();                 // ✅ works — the class-field arrow captured the instance

Class methods follow the normal four rules — so a detached c.increment loses its this exactly like any method (a constant source of bugs when passing methods as event handlers). Class-field arrow functions (= () => {}) sidestep this by capturing the instance's this at construction, at the cost of one function per instance rather than one shared on the prototype (Closures Q6's trade-off). The Classes lesson goes deeper; the key point here is that classes don't exempt you from the this rules — they run on top of them.

Part 8 — Build It: a Manual bind Polyfill

Implementing bind yourself is a classic interview task and the best proof you understand both this and closures. It must fix this, support partial application, and forward later arguments.

Function.prototype.myBind = function (thisArg, ...boundArgs) {
  const fn = this;                              // the function myBind was called on
  return function (...callArgs) {               // the new, bound function
    return fn.apply(thisArg, [...boundArgs, ...callArgs]);  // fixed this + merged args
  };
};

function intro(greeting, punct) { return `${greeting}, ${this.name}${punct}`; }
const boundIntro = intro.myBind({ name: "Ada" }, "Hi");
boundIntro("!");   // "Hi, Ada!"  — this fixed to {name:"Ada"}, "Hi" pre-filled, "!" passed at call

Every piece is a concept from these three lessons: this inside myBind is the original function (implicit binding — it's called as fn.myBind(...)); the returned inner function is a closure capturing fn, thisArg, and boundArgs; and it uses apply to invoke fn with the fixed this and the merged argument list. (A fully spec-compliant polyfill also handles being called with new, but this captures the essence.) If you can write this, you understand this.

Interview Questions

Q1 (beginner): What determines the value of this in a normal function?

How the function is called — the call site — not where it's defined. There are four patterns, in priority order: called with new (this is the new object); called with call/apply/bind (this is the explicitly given object); called as obj.method() (this is obj); or a plain call (this is undefined in strict mode/modules, globalThis in sloppy mode). The same function can have different this values on different calls because it's resolved dynamically each time.

Q2 (beginner): How is this different in an arrow function?

An arrow function has no this of its own — it doesn't participate in the four call-site rules. Instead this inside an arrow is inherited lexically from the scope where the arrow was written, and it's fixed there permanently (you can't change it with call/apply/bind). This makes arrows ideal for callbacks nested inside a method — they keep the method's this — and unsuitable as standalone object methods, where there's no meaningful enclosing this.

Q3 (mid): Explain the "lost this" bug and two ways to fix it.

When you detach a method from its object — const f = obj.method, or pass obj.method as a callback to setTimeout/addEventListener/ forEach — the implicit obj. binding is lost, because this depends on how the function is called and now it's called as a plain function (or by an API with its own this). So this becomes undefined/global and this.someProp breaks. Fixes: (1) obj.method.bind(obj) — returns a new function with this permanently fixed to obj, safe to pass anywhere; (2) wrap in an arrow that calls it as a method: () => obj.method(), preserving the dot; (3) for classes, use a class-field arrow so the method is auto-bound to the instance.

Q4 (mid): What's the difference between call, apply, and bind?

All three set this explicitly. call and apply invoke the function immediately; they differ only in argument form — call(thisArg, a, b) takes arguments individually, apply(thisArg, [a, b]) takes them as an array (mnemonic: Apply = Array). bind(thisArg, ...args) does not invoke — it returns a new function with this (and any provided leading arguments) permanently fixed, to be called later. So call/apply are for one-off invocation with a chosen this (and apply is handy for spreading an array into a variadic function), while bind is for creating a reusable pre-bound function — the standard fix for passing a method as a callback, and a way to do partial application.

Q5 (senior): Walk through the precedence of the binding rules with a concrete example where multiple could apply.

The order is new > explicit (bind/call/apply) > implicit (method call) > default (plain call). Explicit beats implicit: obj.method.call(other) — even though it looks like a method call on obj, call(other) wins, so this is other. new beats explicit, with a subtlety: if you const Bound = Fn.bind(obj) and then new Bound(), the new wins — this is the freshly constructed object, not obj; the bound this is ignored (though bound arguments still apply). Arrows sit outside this hierarchy entirely: their lexical this can't be overridden by any of the four, so arrowFn.call(x) ignores x. The practical value of knowing the precedence is debugging surprising this values: you trace the call site top-down through the rules and the first matching one is definitive.

Q6 (senior): Implement Function.prototype.bind from scratch, including partial application. What subtlety involves new?

A basic version captures the target function and bound state in a closure and uses apply:

Function.prototype.myBind = function (thisArg, ...bound) {
  const fn = this;
  return function (...args) { return fn.apply(thisArg, [...bound, ...args]); };
};

The pieces: this inside myBind is the function it's called on (implicit binding); the returned closure captures fn, thisArg, and the pre-filled bound args, merging them with call-time args via apply. The new subtlety: per spec, a bound function used as a constructor (new boundFn()) must ignore the bound this and instead use the newly-created object, while still applying the bound arguments — so a fully correct polyfill detects construction (e.g. via new.target, or by checking whether this instanceof the inner function) and, in that case, invokes the original with the real new object as this instead of thisArg, and preserves the prototype chain so instanceof works. That construction-aware branch is the part naive implementations miss and interviewers probe for — it demonstrates you understand that new outranks explicit binding in the precedence hierarchy.

🏋️ Exercises

1 (easy): Predict this. For const o = { x: 1, get(){ return this.x; } }, what does each return: o.get(), const g = o.get; g(), o.get.call({x: 99})?
Answers

o.get() → 1 (implicit binding, this = o). g() → throws / undefined.x (plain call, this undefined in strict mode — the lost-this bug). o.get.call({x:99}) → 99 (explicit binding overrides).

2 (easy): Fix the timer. const t = { s: 0, tick(){ setTimeout(function(){ this.s++; }, 100); } } doesn't increment. Fix with an arrow and explain.
Solution 
const t = { s: 0, tick() { setTimeout(() => { this.s++; }, 100); } };

The original callback is a normal function called by the timer as a plain function, so its this isn't t. An arrow has no own this and inherits it from tick, where this === t — so this.s++ hits the right object.

3 (medium): bind for a callback. Given const logger = { prefix: "[APP]", log(msg){ console.log(this.prefix, msg); } }, pass log to ["a","b"].forEach(...) so it prints with the prefix.
Solution 
["a", "b"].forEach(logger.log.bind(logger));
// [APP] a
// [APP] b

forEach calls its callback as a plain function, losing this. bind(logger) locks this to logger so this.prefix resolves. An arrow wrapper (m) => logger.log(m) would also work.

4 (medium): call for borrowing. Use Array.prototype methods via call to turn the array-like arguments (or a NodeList) into a mapped real array, without Array.from.
Solution 
function double() {
  return Array.prototype.map.call(arguments, x => x * 2);
}
double(1, 2, 3);   // [2, 4, 6]
// Method borrowing: map is invoked with `this` = the array-like arguments object.

call lets you invoke Array.prototype.map with this set to an array-like object that doesn't have map itself — "method borrowing." (Modern code prefers [...arguments].map(...) or rest parameters, but this demonstrates call's power.)

5 (hard): Implement call. Write Function.prototype.myCall(thisArg, ...args) that invokes the function with the given this without using the built-in call/apply/bind.
Solution 
Function.prototype.myCall = function (thisArg, ...args) {
  const ctx = thisArg ?? globalThis;            // null/undefined → global (sloppy semantics)
  const key = Symbol("fn");                      // unique key to avoid clobbering a real property
  ctx[key] = this;                               // put the function ON the context object...
  const result = ctx[key](...args);              // ...so calling it makes `this` === ctx (implicit binding!)
  delete ctx[key];                               // clean up
  return result;
};
function intro() { return `I am ${this.name}`; }
intro.myCall({ name: "Ada" });   // "I am Ada"

The trick uses implicit binding: by temporarily assigning the function as a property of thisArg and calling it via ctx[key](), the method-call rule sets this to ctx. A Symbol key avoids overwriting an existing property, and we delete it afterward. This demonstrates that call is "just" a controlled application of the implicit-binding rule.

6 (hard): Precedence puzzle. Predict and explain: 
function F() { this.tag = "F"; }
const bound = F.bind({ tag: "bound" });
const obj = new bound();
console.log(obj.tag);
Solution

It logs "F", not "bound". Even though F was bound to {tag:"bound"}, calling new bound() invokes the construction path, and new outranks explicit binding in the precedence hierarchy — so this inside F is the freshly created object, not the bound one. this.tag = "F" sets the property on the new object, giving obj.tag === "F". The bound this is ignored under new (bound arguments, had there been any, would still apply). This is the exact subtlety a correct bind polyfill must handle.

🎯 Key Takeaways

  • In a normal function, this is set by the call site, resolved fresh each call — not by where the function is defined.
  • Four rules in priority order: new > explicit (call/apply/bind) > implicit (obj.method()) > default (undefined/global).
  • Arrow functions have no own this — they inherit it lexically and can't be rebound; ideal for nested callbacks, wrong as object methods.
  • Detaching a method (variable, callback, timer) loses its this — the most common this bug; fix with bind, an arrow wrapper, or a class-field arrow.
  • call/apply invoke now (args listed vs as an array); bind returns a permanently-bound new function and does partial application.
  • Classes run on the same rules — methods can lose this too; class-field arrows auto-bind at the cost of one function per instance.
