Functional Patterns Language Core

What you'll master: the functional toolkit that JavaScript teams actually use daily — pure functions and why they're the unit of testability, immutability as a discipline (not a library), higher-order functions, currying and partial application, function composition, and the judgment call of when FP style helps versus when it becomes ceremony. Prerequisites: Functions & Scope and Closures — every pattern here is a closure wearing a different hat.

Part 1 — Pure Functions: the Unit of Predictability

// PURE: output depends ONLY on inputs; touches nothing outside itself
const priceWithTax = (price, rate) => price * (1 + rate);

// IMPURE — three different ways to smuggle in surprise:
let taxRate = 0.2;
const withTax   = (price) => price * (1 + taxRate);       // reads mutable outer state
const logTotal  = (price) => { console.log(price); return price; };  // side effect (I/O)
const addItem   = (cart, item) => { cart.push(item); return cart; }; // MUTATES its argument

// The pure version of addItem — returns a NEW array, input untouched:
const addItemPure = (cart, item) => [...cart, item];

A pure function (1) returns the same output for the same inputs, and (2) has no side effects — no mutation of arguments or outer state, no I/O, no reading clocks or randomness. Why care? Pure functions are trivially testable (call it, assert the return — no mocks, no setup), safe to cache (memoization is only correct for pure functions), safe to reorder and parallelize, and readable in isolation — you never need to ask "but what else does this touch?" Real programs need side effects (rendering, network, storage) — the goal isn't zero impurity, it's pushing impurity to the edges: a thin impure shell that does I/O, around a pure core that does the logic. React is built on exactly this idea — render functions are expected to be pure; effects are quarantined into useEffect.

Part 2 — Immutability as a Discipline

const user = { name: "Ada", roles: ["admin"], address: { city: "London" } };

// UPDATE without mutating — spread copies one level:
const renamed = { ...user, name: "Grace" };                 // user untouched
const promoted = { ...user, roles: [...user.roles, "owner"] };   // nested array: copy IT too
const moved = { ...user, address: { ...user.address, city: "Paris" } }; // nested object: same

// ⚠️ Spread is SHALLOW: renamed.roles === user.roles (same array!)
// Forget the inner copy and you've mutated shared state through the "copy".

// Deep clone when you truly need one:
const snapshot = structuredClone(user);    // handles nesting, Dates, Maps — the modern answer

// Arrays: prefer the non-mutating twins (ES2023):
[3,1,2].toSorted()     // sorted COPY  — .sort() mutates in place
[1,2,3].toReversed()   // reversed COPY — .reverse() mutates
arr.with(2, "new")     // copy with index 2 replaced — arr[2] = x mutates
// map / filter / slice / concat / spread — all non-mutating already

Immutability means treating data as read-only: to "change" it, build a new version. The payoff is that change becomes detectable by identity — if newState !== oldState, something changed; if they're ===, nothing did. That single cheap comparison is what powers React's re-render skipping, Redux's time-travel debugging, and undo stacks. The tax is copying cost and the shallow-copy trap shown above — spread only copies the top level, so every nested level you're changing needs its own spread. For deeply nested updates done frequently, that spread-pyramid gets error-prone; that's the problem Immer solves (write "mutating" code against a draft, get an immutable update out). Object.freeze() exists but is shallow and silent outside strict mode — it's a development guard, not an immutability strategy.

Interview trap: const does NOT make data immutable — it only prevents rebinding the variable. const arr = []; arr.push(1) is perfectly legal. Immutability is about the value; const is about the binding.

Part 3 — Higher-Order Functions: Functions as Values

// A higher-order function TAKES a function, RETURNS a function, or both.
// You already use them: map, filter, reduce, addEventListener, setTimeout.

// RETURNING functions = manufacturing behavior (a closure factory):
const makeValidator = (regex, message) => (value) =>
  regex.test(value) ? null : message;

const validateEmail = makeValidator(/^[^@\s]+@[^@\s]+$/, "Invalid email");
const validateZip   = makeValidator(/^\d{5}$/, "Invalid ZIP");
validateEmail("a@b.com");   // null (valid)
validateZip("abc");         // "Invalid ZIP"

// TAKING functions = injecting the varying part of an algorithm:
const retry = async (fn, times = 3) => {
  for (let i = 0; i < times; i++) {
    try { return await fn(); }
    catch (e) { if (i === times - 1) throw e; }
  }
};
await retry(() => fetch("/api/flaky").then(r => r.json()));

Because functions are first-class values, the varying part of any algorithm can be passed in as an argument instead of hard-coded. That's the whole idea. makeValidator is a factory: each call closes over its own regex and message and stamps out a specialized function. retry is the inverse: the retry skeleton is fixed, the operation is injected. Compare with the OO equivalent — a RetryPolicy class implementing a Retryable interface — and you see why JS codebases lean functional: the closure version is the same power with a tenth of the ceremony.

Part 4 — Currying & Partial Application

// Normal:            f(a, b, c)
// Curried:           f(a)(b)(c)     — each call takes ONE arg, returns a fn awaiting the next
// Partially applied: f(a, b) → g(c) — pre-fill SOME args now, rest later

const log = (level) => (tag) => (message) =>
  console.log(`[${level}] [${tag}] ${message}`);

const warn      = log("WARN");           // level locked in
const warnAuth  = warn("auth");          // tag locked in
warnAuth("token expired");               // [WARN] [auth] token expired
warnAuth("bad password");                // reuse — no repetition of level/tag

// Partial application without currying — just bind or wrap:
const fetchJson = (baseUrl, path) => fetch(baseUrl + path).then(r => r.json());
const fromApi = (path) => fetchJson("https://api.example.com", path);  // wrap
fromApi("/users");   fromApi("/orders");

// Where currying quietly powers real code — callbacks that need config:
const handleClick = (id) => (event) => selectRow(id, event.shiftKey);
rows.forEach(row => row.addEventListener("click", handleClick(row.id)));

Currying transforms a multi-argument function into a chain of single-argument functions; partial application just pre-fills some arguments. The practical win is the same for both: specialize once, reuse many times — configuration flows in early, data flows in late. That "config first, data last" argument order is a deliberate FP convention (you'll see it all over Ramda and fp-ts): it makes the curried functions drop straight into map/filter pipelines. The event-listener example is the version you'll write weekly whether you call it currying or not — a function that takes setup data and returns the actual handler is a curried function.

Part 5 — Composition: Building Pipelines

// compose: right-to-left (math order).  pipe: left-to-right (reading order).
const pipe = (...fns) => (input) => fns.reduce((acc, fn) => fn(acc), input);

const trim        = (s) => s.trim();
const lower       = (s) => s.toLowerCase();
const dedupeWs    = (s) => s.replace(/\s+/g, " ");
const slugify     = (s) => s.replace(/\s/g, "-");

const toSlug = pipe(trim, lower, dedupeWs, slugify);
toSlug("  Hello   Functional WORLD  ");   // "hello-functional-world"

// Each step: one job, pure, individually testable.
// Adding a step = adding a name to the list — no surgery on existing code.

// The same idea inline, via chaining (composition you already write):
orders
  .filter(o => o.status === "paid")
  .map(o => o.total)
  .reduce((sum, t) => sum + t, 0);

Composition means building a big transformation out of small named ones. pipe feeds each function's output into the next — the data flows through a pipeline. The design pressure it creates is healthy: each stage must be a small pure function with one job, because that's the only shape that composes. Method chaining on arrays is the same mental model with different syntax, and it's why map/filter beat a single mega-loop for readability: the pipeline names its stages. (The |> pipeline operator that would make x |> trim |> lower native syntax has been a TC39 proposal for years — until it lands, a 3-line pipe helper is the idiom.)

Part 6 — Deep Dive: Reduce, Memoization & Knowing When to Stop

// reduce is the universal fold — map and filter are just reduces:
const map    = (arr, fn) => arr.reduce((acc, x) => [...acc, fn(x)], []);
const filter = (arr, fn) => arr.reduce((acc, x) => fn(x) ? [...acc, x] : acc, []);
// (Educational! [...acc, x] re-copies every step → O(n²). Real code: acc.push(x); return acc;
//  a LOCAL accumulator you created is fine to mutate — purity is about the boundary, not the inside.)

// Memoization — caching that is ONLY correct for pure functions:
const memo = (fn) => {
  const cache = new Map();
  return (arg) => {
    if (!cache.has(arg)) cache.set(arg, fn(arg));
    return cache.get(arg);
  };
};
const slowScore = memo(expensiveScoringFn);   // repeat calls with same arg → instant

Two senior-level points. First, purity is a boundary property: a function that creates a local array and mutates it internally is still pure to its callers — dogmatically avoiding all mutation inside a function trades real performance (that O(n²) spread-in-reduce is a genuine production footgun) for no benefit. Second, memoization is the direct cash value of purity: if a function is pure, an arg→result cache is always correct; if it's impure, the cache silently serves stale answers. This pairing — purity enables caching enables performance — is the argument that convinces performance-minded skeptics that FP discipline isn't just aesthetics.

Senior framing — when NOT to go functional: FP style loses when it fights the problem or the platform. Signs you've overdone it: point-free one-liners nobody can debug (you can't set a breakpoint mid-composition as easily as mid-loop); spread-copying large arrays in hot loops (immutability has a real allocation cost — measure before adopting it in tight loops); currying helpers where a plain object argument ({retries, timeout}) would be clearer; and reimplementing Ramda badly instead of using plain methods. The professional default in JS: pure core + immutable updates at state boundaries + map/filter/reduce pipelines — and plain loops, local mutation, and classes wherever they're simply clearer. FP is a toolkit, not a religion; the codebase that mixes paradigms judiciously beats the one that's ideologically consistent.

Interview Questions

Q1 (Beginner): What makes a function "pure", and give two concrete benefits.

Pure = same inputs always give the same output, and no side effects (no mutation of external state or arguments, no I/O). Benefits: trivially testable (no mocks or setup — call and assert), and safely cacheable/memoizable, since the result depends only on the arguments. Also safe to reorder, retry, and reason about in isolation.

Q2 (Beginner): const cart = ["apple"]; cart.push("banana"); — does this throw? What does that tell you about const?

No throw — cart is now ["apple","banana"]. const prevents reassigning the binding (cart = [] would throw), not mutating the value it points to. Immutability of data is a separate discipline: use non-mutating operations or freeze/clone.

Q3 (Mid): const copy = {...state}; copy.user.name = "X"; — why might this be a bug?

Spread is shallow: copy.user is the same object as state.user, so the assignment mutates the original state through the "copy". Every consumer holding state sees the change, and identity-based change detection (React/Redux) won't notice it because state.user's reference didn't change. Fix: {...state, user: {...state.user, name: "X"}} or structuredClone.

Q4 (Mid): Write pipe and explain how it differs from compose.

const pipe = (...fns) => x => fns.reduce((acc, fn) => fn(acc), x); — applies functions left-to-right (reading order): pipe(a,b,c)(x) === c(b(a(x))). compose is the same with reduceRight — right-to-left, matching math notation (c∘b∘a)(x). Same power; pipe is usually preferred in JS because the execution order matches the reading order.

Q5 (Senior): Your team memoized an "expensive" function and now gets stale results intermittently. Diagnose.

The function is impure — it reads something not in its argument list (a global config, the current time, a mutable object's property, fetched data), so the cache key doesn't capture all real inputs. Same-args calls hit the cache and replay an answer computed under different hidden state. Fixes: make every real input an explicit argument (then the cache key is complete), or key the cache on the hidden dependency too, or drop memoization. Also check the object-identity variant: memoizing by reference while callers pass fresh-but-equal objects each time — the opposite failure, a cache that never hits.

Q6 (Senior): When is immutable-update style the wrong choice, and what's the escape hatch?

In hot paths over large data: copying a 100k-element array per update (or spreading an accumulator per iteration — O(n²)) costs real allocations and GC pressure where an in-place mutation is O(1). Escape hatches: mutate locally owned data (purity is a boundary property — a function may mutate structures it created), batch changes then publish one new reference at the state boundary, or use Immer, which gives immutable semantics with draft-mutation ergonomics and structural sharing. The discipline that matters is immutability of shared state, not of every temporary.

🏋️ Exercises

1 (Easy): Write a pure function increment(counter) that takes {count: n} and returns a new object with count+1, without mutating the input.
Solution
const increment = (counter) => ({ ...counter, count: counter.count + 1 });
const a = { count: 1 };
const b = increment(a);
console.log(a.count, b.count, a === b);   // 1 2 false — input untouched, new identity
2 (Easy): Using non-mutating methods only, produce a sorted-descending copy of [3,1,2] and prove the original is unchanged.
Solution
const nums = [3,1,2];
const sorted = nums.toSorted((a,b) => b - a);   // or [...nums].sort((a,b)=>b-a)
console.log(sorted);   // [3,2,1]
console.log(nums);     // [3,1,2] — untouched (plain .sort() would have mutated it)
3 (Medium): Build makeRateLabel(currency) that returns a formatter: makeRateLabel("USD")(42.5) → "USD 42.50". Then explain which pattern this is.
Solution
const makeRateLabel = (currency) => (amount) => `${currency} ${amount.toFixed(2)}`;
const usd = makeRateLabel("USD");
usd(42.5);    // "USD 42.50"
usd(7);       // "USD 7.00"
// Pattern: currying / a closure factory — config (currency) captured early, data (amount) supplied late.
4 (Medium): Update state = {user: {name: "Ada", prefs: {theme: "light"}}} to theme "dark" immutably. Verify state.user.prefs !== next.user.prefs but untouched branches keep identity where possible.
Solution
const next = {
  ...state,
  user: { ...state.user, prefs: { ...state.user.prefs, theme: "dark" } }
};
console.log(state.user.prefs.theme);          // "light" — original intact
console.log(next.user.prefs !== state.user.prefs);   // true — changed path has new identity
// Note: every object ALONG the changed path gets a new identity (next, next.user, next.user.prefs);
// siblings NOT on the path (e.g. state.user.name's string, other branches) are shared — structural sharing.
5 (Hard): Implement pipe, then use it with three small pure functions to turn " The QUICK fox " into "the-quick-fox".
Solution
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);
const trim  = s => s.trim();
const lower = s => s.toLowerCase();
const slug  = s => s.replace(/\s+/g, "-");
const toSlug = pipe(trim, lower, slug);
console.log(toSlug("  The QUICK fox  "));   // "the-quick-fox"
6 (Hard): Write memo(fn) for single-argument functions, then demonstrate both its win (repeat call is cached) and its danger (memoizing an impure function).
Solution
const memo = (fn) => {
  const cache = new Map();
  return (arg) => {
    if (!cache.has(arg)) cache.set(arg, fn(arg));
    return cache.get(arg);
  };
};
// Win:
let calls = 0;
const square = memo(n => { calls++; return n * n; });
square(4); square(4); square(4);
console.log(calls);   // 1 — computed once

// Danger:
let taxRate = 0.2;
const withTax = memo(p => p * (1 + taxRate));   // IMPURE: reads taxRate
withTax(100);        // 120 — cached
taxRate = 0.5;
console.log(withTax(100));   // 120 — STALE! Cache replays the old answer.

🎯 Key Takeaways

  • Pure functions (same input → same output, no side effects) are the unit of testability and the precondition for memoization — push impurity to the edges.
  • const locks the binding, not the value; immutability is a discipline built on non-mutating updates (... spread, toSorted/toReversed/with, structuredClone).
  • Spread is shallow — every nested level you change needs its own copy, and immutable updates give each object on the changed path a new identity (what React/Redux change detection relies on).
  • Higher-order functions inject the varying part of an algorithm; currying/partial application specialize once (config early) and reuse (data late).
  • pipe/composition builds big transformations from small named pure stages — same mental model as array method chaining.
  • Purity is a boundary property: mutating locally-owned data inside a function is fine (and avoids O(n²) spread-in-reduce). FP is a toolkit, not a religion — mix paradigms where clearer.
