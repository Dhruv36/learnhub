The DOM Browser & the Web Platform

What you'll master: what the DOM actually is (a live tree, not your HTML source), selecting and traversing elements, creating and modifying nodes safely, the textContent vs innerHTML security line, and — the part that matters at scale — why DOM operations are expensive and how to batch them to avoid layout thrashing. Prerequisites: Objects, Arrays, and Strings (the XSS discussion). Events get their own lesson next; here we focus on reading and manipulating structure.

Part 1 — The DOM Is a Live Tree of Objects

// The browser parses your HTML into a tree of NODE objects — the DOM.
// It is NOT your HTML text; it's a live, in-memory object model you manipulate with JS.
document                      // the root Document node
document.documentElement      // <html>
document.head, document.body  // <head>, <body>
element.nodeType              // 1 = element, 3 = text node, 8 = comment
element.tagName               // "DIV"

When a page loads, the browser parses the HTML source into the Document Object Model — a tree of node objects representing every element, text, and comment. Crucially, the DOM is live and separate from your original HTML text: JavaScript changes to it update what the user sees, and reading it reflects the current state (including changes made by scripts), not the source you wrote. Most nodes you care about are element nodes (nodeType 1), but text between tags becomes text nodes (nodeType 3) — which is why traversal sometimes surprises you with whitespace text nodes (Part 3).

Part 2 — Selecting Elements

document.querySelector(".card")          // FIRST match of a CSS selector, or null
document.querySelectorAll("li.active")   // ALL matches — a static NodeList (snapshot)
document.getElementById("header")        // by id (fastest, but querySelector is fine)

// Scope a search to within an element (not the whole document):
const card = document.querySelector(".card");
card.querySelector("button");            // searches only inside .card

// NodeList → array to use array methods:
[...document.querySelectorAll("li")].filter(li => li.textContent.includes("x"));

querySelector/querySelectorAll are the modern, universal selectors — they take any CSS selector, so you already know the syntax from the CSS track. querySelector returns the first match (or null — always guard against that); querySelectorAll returns a static NodeList (a snapshot that doesn't update if the DOM changes later). A NodeList isn't a real array — spread it ([...nodeList]) or use Array.from to get map/filter. You can call querySelector on any element to scope the search to its subtree, which is cleaner and faster than searching the whole document.

🧠 Static vs live collections. querySelectorAll returns a static NodeList (frozen at query time). Older APIs like getElementsByClassName/getElementsByTagName return live HTMLCollections that auto-update as the DOM changes — which can cause infinite loops if you add matching elements while iterating. Prefer querySelectorAll for predictability.

Part 3 — Traversing the Tree

const el = document.querySelector(".item");
el.parentElement            // the parent element
el.children                 // ELEMENT children (an HTMLCollection — no text nodes)
el.childNodes               // ALL child nodes INCLUDING text/whitespace nodes (often surprising)
el.firstElementChild        // first child element (skips text nodes)
el.nextElementSibling       // next sibling element
el.closest(".card")         // nearest ANCESTOR (or self) matching a selector — very useful
el.matches(".active")       // does this element match the selector? → boolean

Two families: the element-only properties (children, firstElementChild, nextElementSibling, parentElement) skip text and comment nodes and are what you almost always want. The all-nodes properties (childNodes, firstChild) include whitespace text nodes — a classic gotcha where el.firstChild is a newline text node, not the element you expected. Two standouts: closest(selector) walks up to the nearest matching ancestor (essential for event delegation, next lesson), and matches(selector) tests an element against a selector.

Part 4 — Creating & Inserting Nodes

const li = document.createElement("li");        // create a detached element
li.textContent = "New item";                     // safe text (Part 6)
li.className = "item";                            // set attributes/properties
li.dataset.id = "42";                            // data-id="42" via the dataset API

const list = document.querySelector("ul");
list.append(li);                                 // add as LAST child (modern; accepts text + multiple nodes)
list.prepend(li);                                // add as FIRST child
el.before(newNode); el.after(newNode);           // insert as siblings
el.replaceWith(newNode);                         // swap
el.remove();                                     // delete the element

// Clone a template subtree:
const clone = template.content.cloneNode(true);  // deep clone (true) — the <template> pattern

The modern insertion methods — append, prepend, before, after, replaceWith, remove — are far cleaner than the legacy appendChild/insertBefore/removeChild (which are verbose and require a parent reference). append/prepend even accept plain strings (auto- creating text nodes) and multiple arguments. Build detached elements with createElement, set their content/attributes, then insert once. The <template> element + cloneNode(true) is the efficient pattern for repeating a fixed structure (list rows, cards).

Part 5 — Modifying Attributes, Classes & Styles

// Classes — use classList, not string manipulation:
el.classList.add("active");
el.classList.remove("hidden");
el.classList.toggle("open");           // add if absent, remove if present
el.classList.contains("active");       // boolean

// Attributes vs properties:
el.setAttribute("aria-label", "Close");  el.getAttribute("href");
el.id = "main";  el.href = "/x";         // many attributes are also direct properties
el.dataset.userId = "7";                 // data-user-id (camelCase ↔ kebab-case)

// Inline styles (prefer toggling classes over per-property styles for maintainability):
el.style.color = "red";
el.style.setProperty("--theme", "dark"); // CSS custom properties (CSS track)

classList (add/remove/toggle/ contains) is the correct way to manipulate classes — never build className strings by hand. Prefer toggling classes over setting inline styles: it keeps styling in CSS (where it belongs), is more performant, and separates concerns — set a class and let the stylesheet decide the appearance. The dataset API reads/writes data-* attributes with automatic camelCase↔kebab-case conversion (the HTML track's data attributes, now from JS). Note the attribute-vs- property subtlety: they usually mirror each other, but not always (e.g. value on inputs diverges — Forms lesson).

Part 6 — textContent vs innerHTML: the Security Line

el.textContent = userInput;              // ✅ SAFE — sets text; HTML is NOT interpreted
el.innerHTML = userInput;                // ❌ DANGEROUS — parses userInput AS HTML → XSS
// If userInput = "<img src=x onerror=alert(document.cookie)>", innerHTML EXECUTES it.

el.innerHTML = `<b>${escapeHtml(userInput)}</b>`;  // if you MUST build HTML, escape first
// or build with createElement + textContent (safest), or sanitise with DOMPurify for rich input.

This is the single most important DOM safety rule, first met in the Strings lesson and repeated because it matters: innerHTML parses its input as HTML and will execute injected scripts/handlers, so assigning user-controlled data to it is a cross-site scripting vulnerability. textContent treats everything as literal text — it's immune, and it's your default for inserting dynamic text. Only reach for innerHTML with trusted, static markup or after escaping/sanitising every dynamic value (DOMPurify for rich HTML). The Security lesson goes deep; the habit — textContent by default — starts here.

Part 7 — Performance: Reflow, Repaint & Batching

// ❌ LAYOUT THRASHING — interleaved reads and writes force repeated reflows:
for (const el of items) {
  el.style.height = el.offsetHeight + 10 + "px";  // READ offsetHeight, then WRITE — forces sync layout
}                                                  // ...every iteration recomputes layout: O(n) reflows

// ✅ BATCH — read all first, then write all:
const heights = items.map(el => el.offsetHeight); // all reads (one layout)
items.forEach((el, i) => el.style.height = heights[i] + 10 + "px");  // all writes

// ✅ DocumentFragment — build off-DOM, insert once (one reflow instead of N):
const frag = document.createDocumentFragment();
for (const item of data) {
  const li = document.createElement("li");
  li.textContent = item;
  frag.append(li);                                 // appended to the fragment (not the live DOM)
}
list.append(frag);                                 // ONE DOM insertion → one reflow

DOM operations are expensive because they can trigger reflow (the browser recomputing layout — positions and sizes) and repaint (redrawing pixels). Two performance rules fall out. First, don't interleave reads and writes: reading a layout property (offsetHeight, getBoundingClientRect) after a write forces the browser to synchronously recompute layout to give you an accurate value — do that in a loop and you get "layout thrashing," O(n) reflows. Batch all reads, then all writes. Second, build off-DOM and insert once: append many nodes to a DocumentFragment (an in-memory container) and add it to the live DOM in a single operation, causing one reflow instead of N. These two techniques are the core of DOM performance and the reason frameworks use virtual DOMs / batched updates.

Part 8 — Build It: an Efficient Filterable List

Let's combine selection, safe creation, fragments, and batching into a small render-and-filter widget — the shape of countless real UI components, done performantly.

function renderList(container, items) {
  const frag = document.createDocumentFragment();   // build off-DOM
  for (const item of items) {
    const li = document.createElement("li");
    li.className = "row";
    li.dataset.id = item.id;
    li.textContent = item.name;                      // SAFE — no innerHTML with user data
    frag.append(li);
  }
  container.replaceChildren(frag);                   // clear + insert in ONE operation, one reflow
}

function filterList(container, query) {
  const q = query.toLowerCase();
  // Batch: read all matches first, then toggle classes (writes) — no interleaved layout reads
  for (const row of container.querySelectorAll(".row")) {
    const match = row.textContent.toLowerCase().includes(q);
    row.classList.toggle("hidden", !match);          // toggle a class; CSS handles display
  }
}

const list = document.querySelector("#list");
renderList(list, data);
searchInput.addEventListener("input", (e) => filterList(list, e.target.value));

Every performance and safety idea is applied: rows are created with textContent (no XSS), built in a DocumentFragment and inserted with a single replaceChildren (one reflow, not one per row), and filtering toggles a hidden class rather than reading layout or rebuilding the DOM — so the CSS does the hiding and there's no layout thrashing. This is a genuinely production-shaped component; the same principles scale to virtualised lists and framework renderers (which automate exactly this batching).

Interview Questions

Q1 (beginner): What is the DOM?

The Document Object Model is the browser's in-memory representation of a page as a tree of node objects — one node per element, text run, and comment. The browser builds it by parsing the HTML, and JavaScript reads and modifies this tree to change what the user sees. Key point: the DOM is live and distinct from the original HTML source — it reflects the current state including any changes scripts have made, and manipulating it updates the rendered page. You interact with it via document and element objects and their methods/properties.

Q2 (beginner): What's the difference between textContent and innerHTML?

textContent gets/sets an element's text as plain text — assigning to it inserts the string literally, without interpreting any HTML, so it's safe against XSS. innerHTML gets/sets the element's HTML as markup — assigning parses the string as HTML and builds nodes from it, which executes injected scripts/event handlers if the string contains them, making it dangerous with user-controlled data. Default to textContent for dynamic text; only use innerHTML with trusted static markup or after escaping/sanitising every dynamic value.

Q3 (mid): What is layout thrashing and how do you avoid it?

Layout thrashing is repeatedly forcing the browser to recompute layout (reflow) by interleaving DOM writes with reads of layout-dependent properties (offsetHeight, offsetTop, getBoundingClientRect, scrollTop). After a write, the layout is "dirty"; reading a layout property forces a synchronous recalculation to return an accurate value. Do that in a loop and you trigger O(n) reflows, causing jank. Avoid it by batching: read all the layout values you need first (into variables), then perform all writes — so layout is recomputed at most once. For visual updates, schedule writes in requestAnimationFrame. Tools like FastDOM formalise the read-then-write batching.

Q4 (mid): Why use a DocumentFragment, and how does it help performance?

A DocumentFragment is a lightweight, off-screen container node. When you append many elements to it, none of those appends touch the live DOM, so they cause no reflow/repaint. Then you insert the whole fragment into the document in a single operation, and when a fragment is inserted, its children are moved in (the fragment itself disappears) — so N elements enter the live DOM with one reflow instead of N. This is much faster than appending each element directly to a live parent in a loop, which can trigger layout work per insertion. It's the standard pattern for rendering lists/tables efficiently, and it's essentially what frameworks' batched DOM updates automate.

Q5 (senior): Compare direct DOM manipulation with a virtual-DOM/framework approach. When is each appropriate?

Direct DOM manipulation gives full control and zero abstraction overhead — you touch exactly the nodes you mean to, which is ideal for small, isolated, high-performance interactions (a widget, a canvas overlay, a library with no framework). Its cost is that keeping the DOM in sync with application state by hand is error-prone and doesn't scale: as UI complexity grows, you accumulate imperative update code scattered across handlers, easy to get out of sync, and manual batching (fragments, read/write separation) becomes your responsibility. A virtual DOM (React et al.) inverts this to declarative: you describe the UI as a function of state, and the framework diffs the new description against the previous one and applies the minimal set of real DOM mutations, batching them automatically. Benefits: state→UI consistency by construction, composability, and automatic batching/optimisation; costs: the diffing has its own CPU and memory overhead, a bundle-size and learning cost, and for truly performance-critical or simple cases the abstraction can be slower than hand-tuned direct manipulation. When to use each: direct DOM for small/isolated interactions, libraries, or extreme performance paths (and modern signal-based frameworks like Solid narrow the gap by skipping the VDOM entirely); a framework for non-trivial applications where the maintainability of declarative state→UI mapping outweighs the overhead — which is most product UIs. The senior nuance is that the two aren't mutually exclusive: even in a framework you sometimes drop to direct DOM (refs) for measurement, focus management, or integrating imperative libraries, applying the same batching principles. The underlying performance rules (minimise reflows, batch reads/writes, insert once) are what the framework automates and what you apply manually when you go direct.

Q6 (senior): You need to render and keep updated a list of 10,000 rows. Walk through the techniques to keep it fast.

Rendering and updating 10,000 rows naively (one node per row, direct appends, full rebuilds on change) will thrash layout and use large memory. Techniques, layered: (1) Batch initial render — build all rows in a DocumentFragment and insert once, so 10k nodes enter with one reflow. (2) Virtualise (windowing) — the decisive technique: only render the rows currently visible in the viewport (plus a small buffer), not all 10k. As the user scrolls, recycle/reposition a fixed pool of ~30 row elements to show the current window, using a spacer to preserve scrollbar size. This caps DOM node count at what's visible regardless of dataset size — libraries like react-window/TanStack Virtual do this. (3) Avoid per-row layout reads — measure row height once (or use fixed heights) rather than reading offsetHeight per row (thrashing). (4) Batch updates — when data changes, compute the diff and mutate only changed rows' textContent/classes; don't rebuild the list. Debounce/throttle scroll and filter handlers, and do filtering by toggling a class or by recomputing the virtual window, not by removing/re-adding nodes. (5) Use requestAnimationFrame for scroll-driven DOM writes so they align with the paint cycle. (6) Delegate events — one listener on the container (next lesson) instead of 10k listeners, saving memory and setup. (7) For pure display, consider content-visibility: auto (CSS) to let the browser skip rendering off-screen content. The dominant win is virtualisation — the others (fragments, batching, delegation, avoiding layout reads) matter but the fundamental principle is "don't put 10k nodes in the DOM at all; render only what's visible." That's why every serious data-grid is virtualised.

🏋️ Exercises

1 (easy): Safe text insert. Given an input's value, display it in #output safely so that <script> tags show as text, not execute.
Solution 
document.querySelector("#output").textContent = input.value;

textContent inserts the value literally, so any HTML/script in it is displayed as text and never parsed or executed — the safe default over innerHTML.

2 (easy): Toggle a class. Toggle a collapsed class on #panel each time a button is clicked, and report whether it's now collapsed.
Solution 
btn.addEventListener("click", () => {
  const isCollapsed = panel.classList.toggle("collapsed");
  console.log(isCollapsed ? "collapsed" : "expanded");
});

classList.toggle adds/removes the class and returns whether it's now present — cleaner than checking and branching manually, and it keeps the styling in CSS.

3 (medium): Efficient list render. Render an array of 1,000 strings as <li> items into a <ul> with a single reflow.
Solution 
const frag = document.createDocumentFragment();
for (const text of items) {
  const li = document.createElement("li");
  li.textContent = text;
  frag.append(li);
}
document.querySelector("ul").append(frag);   // one insertion → one reflow

Building in a fragment keeps all 1,000 appends off the live DOM; the single append(frag) moves them in together, triggering just one reflow instead of a thousand.

4 (medium): closest() navigation. Given a click on a button inside a .card, find and log the card's data-id.
Solution 
button.addEventListener("click", (e) => {
  const card = e.target.closest(".card");    // walk up to the nearest ancestor .card
  if (card) console.log(card.dataset.id);
});

closest walks up the ancestor chain from the clicked element to the nearest matching .card, returning it (or null). This is the foundation of event delegation (next lesson).

5 (hard): Fix the thrashing. This resizes each box to its content but thrashes layout. Rewrite it to batch reads and writes. 
boxes.forEach(box => {
  box.style.width = box.scrollWidth + 20 + "px";
});
Solution 
const widths = boxes.map(box => box.scrollWidth);         // ALL reads first (one layout)
boxes.forEach((box, i) => box.style.width = widths[i] + 20 + "px"); // then ALL writes

The original reads scrollWidth (a layout property) right after a write on the previous iteration, forcing a synchronous reflow every loop — O(n) layouts. Separating all reads from all writes means layout is recomputed once, not per box.

6 (hard): Build a safe HTML card. Create a card element for { title, body } where both fields may contain user text, ensuring no XSS while still producing structured markup (heading + paragraph).
Solution 
function createCard({ title, body }) {
  const card = document.createElement("div");
  card.className = "card";
  const h = document.createElement("h3");
  h.textContent = title;                     // safe — text, not HTML
  const p = document.createElement("p");
  p.textContent = body;                      // safe
  card.append(h, p);                          // structured markup, no innerHTML with user data
  return card;
}
container.append(createCard({ title: userTitle, body: userBody }));

By building the structure with createElement and setting each field via textContent, you get proper markup (heading + paragraph) with zero XSS risk — the safe alternative to innerHTML = `<h3>${title}</h3>...`. For rich HTML input specifically, you'd sanitise with DOMPurify instead.

🎯 Key Takeaways

  • The DOM is a live in-memory tree of node objects, distinct from your HTML source — JS reads and mutates it to change the page.
  • querySelector/querySelectorAll (CSS selectors) are the universal selectors; NodeLists are static and not arrays (spread them); guard against null.
  • Traverse with element-only properties (children, *ElementSibling) to skip whitespace text nodes; closest walks up, matches tests.
  • Create with createElement and modern insertion (append/prepend/before/replaceWith); manage classes with classList, prefer toggling classes over inline styles.
  • textContent is the safe default; innerHTML with user data is XSS — escape or sanitise.
  • DOM ops cause reflow/repaint — batch reads-then-writes to avoid layout thrashing, and build off-DOM with DocumentFragment to insert with one reflow; virtualise huge lists.
