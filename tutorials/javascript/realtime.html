WebSockets & Real-Time Browser & Web Platform

What you'll master: the full real-time toolbox — polling, long-polling, Server-Sent Events, and WebSockets — with the decision framework for choosing between them, the WebSocket lifecycle and its production hardening (heartbeats, reconnection with backoff, missed-message recovery), and the scaling story a senior interviewer will push you on. Prerequisites: Fetch, HTTP & AbortController and The Event Loop. "The UI updates by itself" is a product requirement in chat, dashboards, collaboration, and notifications — this is how it's actually built.

Part 1 — The Problem: HTTP Only Speaks When Spoken To

Plain HTTP: client asks → server answers → connection's job is done.
The server has NO WAY to say "hey, something changed" on its own.

Four escalating workarounds:
1. POLLING        client asks every N seconds ("anything new?")           — simplest, laggy, wasteful
2. LONG-POLLING   client asks; server HOLDS the request open until there  — near-realtime over plain HTTP
                  IS news (or timeout), answers, client immediately re-asks
3. SSE            one long-lived HTTP response the server keeps writing   — true push, SERVER→CLIENT only
                  events into (text/event-stream)
4. WEBSOCKET      HTTP handshake UPGRADES the TCP connection to a         — full-duplex: both sides send
                  persistent two-way message channel                        anytime, low latency

Everything in real-time engineering is a workaround for one fact: in the request/response model the server cannot initiate. Polling brute-forces it with repeated questions — fine for slow-changing data, but latency averages half the poll interval and most requests return "nothing new". Long-polling inverts the waiting: the request parks on the server until there's news, so updates arrive almost instantly without a new protocol. SSE formalizes "one response that never ends" into a real API with auto-reconnect. WebSocket abandons request/response entirely: after an HTTP Upgrade handshake, the underlying TCP connection becomes a raw two-way message pipe. Cost rises with capability: each step down the list means more server-side statefulness and more infrastructure awareness (proxies, load balancers, connection limits).

Part 2 — WebSocket: the API and the Lifecycle

const ws = new WebSocket("wss://api.example.com/live");   // wss:// = TLS. Never ws:// in prod.

ws.addEventListener("open",    () => ws.send(JSON.stringify({ type: "subscribe", room: 42 })));
ws.addEventListener("message", (e) => {
  const msg = JSON.parse(e.data);          // e.data: string (or Blob/ArrayBuffer for binary)
  render(msg);
});
ws.addEventListener("error",   () => { /* fires right before close; details go to console only */ });
ws.addEventListener("close",   (e) => console.log(e.code, e.reason, e.wasClean));
// 1000 = normal, 1006 = dropped WITHOUT close frame (network died) — the reconnect trigger

ws.readyState   // 0 CONNECTING → 1 OPEN → 2 CLOSING → 3 CLOSED
// ⚠️ send() while CONNECTING throws — queue until "open" fires, or check readyState.

ws.close(1000, "done");   // polite shutdown: close frame exchanged, wasClean: true

The API is small; the discipline is in the lifecycle. A socket is only sendable in OPEN state — real apps keep an outbound queue and flush it on open. Messages are ordered and complete (WebSocket is message-oriented over TCP: you never receive half a message or messages out of order on one connection) — but delivery stops the instant the connection dies, and nothing is buffered for you while disconnected. The close event's code tells you why: 1000 means an orderly goodbye; 1006 means the transport vanished mid-flight (wifi drop, proxy timeout, server crash) — that's the case all your resilience code exists for. One more production note: the browser WebSocket API can't set custom headers, so auth happens via cookies, a token in the URL query (careful — URLs get logged), or a first-message auth handshake after open.

Part 3 — Server-Sent Events: the Underrated Middle Option

// Client — three lines, auto-reconnect included:
const es = new EventSource("/api/notifications");        // plain HTTP GET, works through proxies
es.onmessage = (e) => notify(JSON.parse(e.data));        // default "message" events
es.addEventListener("deploy", (e) => banner(e.data));    // named event types
es.onerror = () => { /* EventSource RECONNECTS BY ITSELF — you handle UI state only */ };

// Server just keeps writing this text format (Content-Type: text/event-stream):
//   event: deploy
//   id: 1042            ← client remembers; sent back as Last-Event-ID header on reconnect
//   retry: 5000         ← server can tune the client's reconnect delay
//   data: {"version":"2.4.1"}
//                       ← blank line = event dispatched

If data only flows server → client — notifications, live scores, price tickers, progress bars, activity feeds, AI-token streaming — SSE does the job with dramatically less machinery than WebSocket. It's plain HTTP: normal auth headers and cookies work, proxies and load balancers understand it, HTTP/2 multiplexes many streams over one connection. And the two hardest resilience problems come built in: the browser auto-reconnects on drop, and the Last-Event-ID mechanism lets the server replay what was missed — features you must hand-build on WebSocket. Limits: text-only (no binary), one direction (client talks via ordinary fetch POSTs, which is usually fine), and on legacy HTTP/1.1 the ~6-connections-per-origin cap bites if you open many streams. The senior take: teams reach for WebSocket by default when SSE + fetch would've shipped sooner and run more reliably — reserve WebSocket for genuinely bidirectional, high-frequency traffic (chat typing indicators, multiplayer, collaborative editing).

Part 4 — Production Hardening: Heartbeats & Reconnection

// Problem 1: DEAD connections that look alive. If the network drops without a FIN/RST
// (laptop lid, cell handoff, killed router), neither side gets "close" — maybe for minutes,
// until TCP gives up. Idle proxies also silently kill connections (~60s is common).
// Fix: HEARTBEAT — smaller interval than every proxy timeout in the path:
let alive = true;
setInterval(() => {
  if (!alive) return ws.close();        // no pong since last ping → declare it dead, trigger reconnect
  alive = false;
  ws.send(JSON.stringify({ type: "ping" }));   // app-level ping (browser API exposes no protocol ping)
}, 30_000);
// on message: if (msg.type === "pong") alive = true;

// Problem 2: RECONNECTING without DDoSing your own server.
// If 50k clients drop when a server restarts and all retry instantly-and-forever in sync,
// the reconnect wave keeps knocking it back down (thundering herd).
// Fix: EXPONENTIAL BACKOFF + JITTER:
let attempt = 0;
function reconnect() {
  const base = Math.min(30_000, 1000 * 2 ** attempt);   // 1s, 2s, 4s … capped at 30s
  const delay = base / 2 + Math.random() * base / 2;    // jitter: spread the herd out
  setTimeout(() => { attempt++; connect(); }, delay);
}
// on "open": attempt = 0.  on "close": if not intentional, reconnect().

// Problem 3: MESSAGES MISSED WHILE DISCONNECTED. WebSocket buffers nothing for you.
// Fix: sequence numbers — client tracks lastSeq, sends {type:"resume", from: lastSeq}
// on reconnect; server replays from a short history buffer, or signals "too old → full resync".

These three problems — dead connections, reconnect storms, missed messages — are what separate a demo from a product, and every serious real-time system solves all three. Heartbeats bound how long you can be wrong about a connection's health and keep idle-timeout proxies from culling it. Backoff-with-jitter turns a synchronized reconnect stampede into a gentle trickle — the jitter matters as much as the exponent, because 50k clients backing off on identical schedules still arrive in synchronized waves. And sequence-numbered resume acknowledges the core truth that the socket is a transport, not a source of truth — on reconnect you must reconcile state, whether by replay, by re-fetching a snapshot over plain HTTP, or both. (Libraries like Socket.IO bundle heartbeat + backoff + fallbacks for you — fine to use, but you now know what they're doing and can debug them when they misbehave.)

Also on the client checklist: pause or downgrade activity when document.visibilityState === "hidden" (background tabs get throttled timers anyway); listen to online/offline events to reconnect eagerly when the network returns; and always distinguish intentional close (user logout — don't reconnect) from unexpected close (1006 — reconnect) with a flag.

Interviewers love this question because the wrong answer is a default ("always WebSockets" ≈ résumé-driven engineering; "just poll" ≈ hasn't scaled anything). The right answer is a decision tree: How fresh must data be? Minutes-fresh → poll and go home. Which directions flow fast? Only server→client → SSE. Truly bidirectional and latency-sensitive? → WebSocket, and budget for the hardening in Part 4 plus the infrastructure below. And hybrid is normal: an app might use SSE for notifications, plain fetch for commands, and a WebSocket only inside the live-collaboration view.

Part 6 — Deep Dive: What Scaling Real-Time Actually Involves

The moment you run MORE THAN ONE server, WebSocket state bites:

   [client A] ──ws── [server 1]          A sends a chat message to room #42…
   [client B] ──ws── [server 2]          …but B's socket lives on server 2.
                                          Server 1 can't reach it directly!

Standard fix — a PUB/SUB BACKPLANE all servers share:
   server 1 ── publish("room:42", msg) ──▶ Redis/Kafka/NATS ──▶ every subscribed server
   each server forwards to ITS locally-connected sockets for that room.

The rest of the checklist:
- LOAD BALANCER must speak the upgrade: proxy_http_version 1.1 + Upgrade/Connection headers
  (nginx); ALB/managed proxies: enable & RAISE idle timeout above your heartbeat interval
- CONNECTION MEMORY: each socket is held state — file descriptor + buffers + app session;
  plan capacity per-node (tens of thousands per node is realistic, not millions)
- DEPLOYS: restarting a node drops every socket it holds → mass reconnect. Drain gracefully:
  stop accepting, tell clients to reconnect (they'll land on other nodes), then stop.
- AUTH: sockets outlive tokens. Re-validate on sensitive actions or push a re-auth message
  when the token expires — don't trust a connection authenticated an hour ago.

Stateless HTTP scales by adding servers behind a load balancer — any server can answer any request. Persistent connections break that: a WebSocket pins a client to one process, so cross-client delivery needs a backplane (Redis pub/sub is the classic; Kafka/NATS at larger scale), room membership becomes distributed state, and deploys/restarts turn into reconnect events you must choreograph. None of this is exotic — it's the standard checklist — but it's why "add real-time" is an architecture decision, not a feature ticket.

Senior framing: the strongest system-design answers treat the real-time channel as a notification layer, not the source of truth. The durable state lives behind normal HTTP APIs and a database; the socket/SSE stream just tells clients "something changed" (optionally carrying the delta as an optimization). Then every hard problem gets easier: a missed message costs a refetch, not data loss; reconnect recovery is "GET the current state"; ordering conflicts resolve against the DB, not in socket buffers; and you can even fall back to polling and the product still works — just slower. Design the system to degrade from push to pull, and real-time becomes an enhancement instead of a single point of failure.

Interview Questions

Q1 (Beginner): What problem do WebSockets solve that plain HTTP can't?

In HTTP's request/response model the server can never initiate — it only answers. WebSockets upgrade the connection (via an HTTP Upgrade handshake) into a persistent, full-duplex message channel where either side can send at any time with low latency — enabling server push and rapid bidirectional exchange (chat, live games, collaborative editing) without repeated polling.

Q2 (Beginner): Compare polling and long-polling. Why is long-polling "near real-time"?

Polling asks on a fixed interval; average latency is half the interval and most requests return empty. Long-polling sends a request that the server holds open until it actually has news (or a timeout), then answers; the client immediately re-requests. Because the response fires the moment an event occurs rather than at the next tick, latency is near-zero — at the cost of the server holding many parked requests.

Q3 (Mid): You're adding a live notifications feed (server→client only). WebSocket or SSE — and defend the choice.

SSE. The flow is one-directional, which is exactly SSE's shape: it runs over plain HTTP (normal auth, proxies, load balancers, HTTP/2 multiplexing), and auto-reconnect plus Last-Event-ID-based replay are built into the browser — the two hardest resilience features you'd otherwise hand-build on WebSocket. The client can still send the occasional action via ordinary fetch. WebSocket would add stateful-infra burden (upgrade-aware LBs, hand-rolled heartbeat/ reconnect/replay) and buy nothing the feature needs.

Q4 (Mid): What is close code 1006, and what should a robust client do differently for 1000 vs 1006?

1000 is a clean, intentional close — a close frame was exchanged; typically don't reconnect. 1006 means the connection died without a close frame — network drop, proxy kill, server crash; the browser synthesizes it locally. A robust client treats 1006 (and any unexpected close) as "reconnect with exponential backoff + jitter", then resumes/resyncs missed state — while an intentional-close flag set before calling close() suppresses reconnection for deliberate shutdowns like logout.

Q5 (Senior): Why do WebSocket apps need heartbeats when TCP already has keepalive?

Three reasons. TCP keepalive defaults are useless here (often 2 hours, OS-controlled, not configurable from browser JS). Half-open connections — where the far side vanished without RST — look perfectly healthy until you try to send. And intermediate proxies/LBs kill idle connections on their own timers (~60s is common) regardless of TCP settings. An application-level ping/pong on a ~25–30s timer solves all three: it generates traffic that keeps proxies happy, and a missing pong gives you a bounded-time, app-controlled dead-connection detector that triggers reconnection. (The browser API exposes no protocol-level ping, so it must be an app message.)

Q6 (Senior): Chat app, clients connected across 10 server instances. How does a message from a user on server 1 reach a room member on server 7 — and what else does multi-node real-time force you to handle?

Via a shared pub/sub backplane: server 1 publishes the message to a channel (e.g. Redis room:42); every instance subscribes and forwards to the sockets it holds for that room. Persistence happens before/alongside publish so the stream is a notification layer, not the source of truth. Multi-node also forces: upgrade-aware load balancing with idle timeouts above the heartbeat interval; per-node connection capacity planning (sockets are held memory/FDs); graceful drain on deploys (stop accepting, nudge clients to reconnect elsewhere, then restart) to avoid thundering-herd reconnects; distributed room-membership/presence state; and re-validating auth on long-lived connections whose tokens expire mid-session.

  const res = await fetch("/api/status");
  render(await res.json());

es.onmessage = (e) => addToFeed(JSON.parse(e.data));            // "message" (default) events

4 (Medium): Add a send-queue to a WebSocket wrapper so that safeSend(msg) works in any readyState: queue while connecting, flush on open.
Solution
function makeSocket(url) {
  const queue = [];
  const ws = new WebSocket(url);
  ws.addEventListener("open", () => {
    while (queue.length) ws.send(queue.shift());   // flush in order — WS preserves ordering
  });
  return {
    ws,
    safeSend(obj) {
      const data = JSON.stringify(obj);
      if (ws.readyState === WebSocket.OPEN) ws.send(data);
      else if (ws.readyState === WebSocket.CONNECTING) queue.push(data);
      else throw new Error("socket closed");        // caller decides: reconnect or surface error
    }
  };
}
5 (Hard): Implement connectResilient(url): reconnect on unexpected close with exponential backoff + jitter (cap 30s), reset attempts on success, and never reconnect after an intentional shutdown().
Solution
function connectResilient(url, onMessage) {
  let attempt = 0, intentional = false, ws;
  function connect() {
    ws = new WebSocket(url);
    ws.addEventListener("open", () => { attempt = 0; });
    ws.addEventListener("message", onMessage);
    ws.addEventListener("close", () => {
      if (intentional) return;                          // logout/leave: stay closed
      const base = Math.min(30_000, 1000 * 2 ** attempt++);
      const delay = base / 2 + Math.random() * (base / 2);   // jitter spreads the herd
      setTimeout(connect, delay);
    });
  }
  connect();
  return { shutdown() { intentional = true; ws.close(1000, "bye"); } };
}
// Delays: ~1s, ~2s, ~4s … capped ~30s, each randomized — a restart's reconnect wave arrives
// as a trickle instead of synchronized spikes.
6 (Hard): Add missed-message recovery: messages carry seq; on reconnect send {type:"resume", from: lastSeq}; detect gaps and fall back to a full resync. Sketch both sides' logic.
Solution
// CLIENT
let lastSeq = 0;
function handle(e) {
  const msg = JSON.parse(e.data);
  if (msg.seq !== lastSeq + 1) return fullResync();   // gap detected → refetch snapshot via HTTP
  lastSeq = msg.seq;
  apply(msg);
}
function onReconnectOpen(ws) {
  ws.send(JSON.stringify({ type: "resume", from: lastSeq }));
}
async function fullResync() {
  const snap = await fetch("/api/state").then(r => r.json());  // HTTP API = source of truth
  lastSeq = snap.seq;
  replaceState(snap);
}

//  on resume(from): if from >= oldest buffered seq → replay (from, latest]
//                   else → send {type:"resync"} — history evicted, client must snapshot.

// notification layer over a durable store, never the source of truth itself.

🎯 Key Takeaways

  • All real-time techniques work around one fact: HTTP servers can't initiate. Polling → long-polling → SSE → WebSocket trade rising infrastructure cost for falling latency and rising capability.
  • SSE is the underrated default for server→client-only streams: plain HTTP, auto-reconnect and Last-Event-ID replay built in. Reserve WebSocket for genuinely bidirectional, high-frequency traffic.
  • WebSocket messages are ordered and complete per connection — but nothing is buffered across disconnects, send only works in OPEN state, and close code 1006 = died without goodbye.
  • Production hardening is non-optional: app-level heartbeats (~30s, below proxy timeouts), reconnect with exponential backoff plus jitter, intentional-close flags, and sequence-based resume or snapshot resync.
  • Scaling past one server means a pub/sub backplane (Redis/Kafka/NATS), upgrade-aware load balancing, per-node connection capacity, and graceful drain on deploys.
  • Architect the stream as a notification layer over durable HTTP+DB state — then missed messages cost a refetch, reconnect is a resync, and the app can degrade from push to pull and keep working.
