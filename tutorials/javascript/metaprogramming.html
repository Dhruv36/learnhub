Metaprogramming: Proxy & Reflect Production JavaScript

What you'll master: intercepting the language's own fundamental operations — property reads, writes, deletions, function calls, new — with Proxy; performing those same operations correctly from inside your interceptors with Reflect; and the real systems built on this pair: Vue 3's reactivity, validation layers, and mocking libraries. Prerequisites: Objects Deep Dive (property descriptors, internal methods) and Prototypes — a Proxy intercepts exactly the internal operations those lessons described.

Part 1 — The Mental Model: Every Object Operation Is a Method Call

The spec defines objects by a set of internal methods: [[Get]], [[Set]], [[Delete]], [[Has]], [[OwnPropertyKeys]], and so on. When you write obj.x, the engine calls obj.[[Get]]("x"). Normally you can't touch these. A Proxy is an object whose internal methods are your functions — the traps.

const target = { greeting: "hello" };

const handler = {
  get(target, prop, receiver) {                // trap for [[Get]] — fires on EVERY property read
    console.log(`read: ${String(prop)}`);
    return Reflect.get(target, prop, receiver); // then do the default behavior
  }
};

const proxy = new Proxy(target, handler);
proxy.greeting;        // logs "read: greeting", returns "hello"
proxy.missing;         // logs "read: missing", returns undefined — traps fire even for absent keys

// The target is untouched — reads through `target` itself never hit the trap:
target.greeting;       // no log. Proxies wrap; they don't modify.
The one-sentence model: a Proxy is a transparent wrapper where each of the ~13 internal object operations can be replaced by a function of yours; any trap you don't define falls through to the target's default behavior. Metaprogramming = your code running when the language, not the programmer, invokes an operation.

// A validating "set" trap — the classic first real use:
function typed(target, types) {
  return new Proxy(target, {
    set(t, prop, value, receiver) {
      if (prop in types && typeof value !== types[prop]) {
        throw new TypeError(`${String(prop)} must be a ${types[prop]}, got ${typeof value}`);
      }
      return Reflect.set(t, prop, value, receiver);  // MUST return true on success
    }
  });
}

const user = typed({ name: "Ada", age: 36 }, { name: "string", age: "number" });
user.age = 37;        // fine
user.age = "old";     // TypeError: age must be a number, got string
Traps have contracts. set and deleteProperty must return a boolean (falsy = a TypeError in strict mode). ownKeys must return only strings/symbols. Violate an invariant — e.g. report a non-configurable property as absent — and the engine itself throws. You can lie to callers, but not about the target's non-configurable/non-extensible facts.

Part 3 — Reflect: the Default Behaviors as Functions

Reflect has one static method per internal operation, with the same signature as the corresponding trap. That symmetry is the whole point: inside a trap, Reflect.get(...arguments) is exactly "do what would have happened without me".

Reflect.get(obj, "x");             // like obj.x
Reflect.set(obj, "x", 1);          // like obj.x = 1, but returns false instead of throwing (strict) on failure
Reflect.has(obj, "x");             // like "x" in obj
Reflect.deleteProperty(obj, "x");  // like delete obj.x, returns boolean
Reflect.ownKeys(obj);              // ALL own keys: string + symbol, enumerable or not
Reflect.construct(Fn, args);       // like new Fn(...args)
Reflect.defineProperty(obj, k, d); // like Object.defineProperty, but returns false instead of throwing

Why not just target[prop] inside a get trap? The receiver. If a proxy sits in a prototype chain, a getter on the target should run with this = the original object the read started on. Only Reflect.get(target, prop, receiver) forwards that correctly:

const base = {
  _name: "base",
  get name() { return this._name; }        // `this` should be whoever the read started on
};
const proxy = new Proxy(base, {
  get(t, prop, receiver) {
    return Reflect.get(t, prop, receiver); // receiver-aware: correct
    // return t[prop];                     // WRONG: getter would always see this === base
  }
});
const child = Object.create(proxy);
child._name = "child";
child.name;   // "child" with Reflect.get — would be "base" with t[prop]

Part 4 — Build It: a Reactivity System (How Vue 3 Works)

Let's build the real thing, incrementally: a reactive() that tracks which function read which property, and re-runs that function when the property changes.

// Step 1 — track the currently running effect
let activeEffect = null;
function effect(fn) { activeEffect = fn; fn(); activeEffect = null; }

// Step 2 — a dependency map: target → prop → Set of effects
const deps = new WeakMap();
function track(target, prop) {
  if (!activeEffect) return;
  let propMap = deps.get(target) ?? deps.set(target, new Map()).get(target);
  let set = propMap.get(prop) ?? propMap.set(prop, new Set()).get(prop);
  set.add(activeEffect);
}
function trigger(target, prop) {
  deps.get(target)?.get(prop)?.forEach(fn => fn());
}

// Step 3 — the proxy: reads track, writes trigger
function reactive(target) {
  return new Proxy(target, {
    get(t, prop, receiver) {
      track(t, prop);                                   // "this effect depends on this prop"
      return Reflect.get(t, prop, receiver);
    },
    set(t, prop, value, receiver) {
      const ok = Reflect.set(t, prop, value, receiver);
      trigger(t, prop);                                 // "re-run everything that depended on it"
      return ok;
    }
  });
}

// Step 4 — use it
const state = reactive({ count: 0 });
effect(() => console.log("count is", state.count));  // logs "count is 0" and REGISTERS the dependency
state.count++;                                        // logs "count is 1" — automatically

Every read inside effect registers a dependency; every write replays the dependents. That's the entire core of Vue 3's reactive() — the production version adds nested-object wrapping, array/collection handling, and batching, but no new ideas. Note the WeakMap keyed by target: when the object is garbage-collected, its whole dependency tree goes with it (see Memory Management).

Part 5 — Other Production Patterns

// 1. Negative array indices (à la Python):
const arr = new Proxy([10, 20, 30], {
  get(t, prop, r) {
    const i = Number(prop);
    if (Number.isInteger(i) && i < 0) return t[t.length + i];
    return Reflect.get(t, prop, r);
  }
});
arr[-1];   // 30

// 2. Throw on missing properties — kill silent undefined bugs in config objects:
function strict(obj) {
  return new Proxy(obj, {
    get(t, prop, r) {
      if (!(prop in t)) throw new ReferenceError(`No such config key: ${String(prop)}`);
      return Reflect.get(t, prop, r);
    }
  });
}

// 3. Revocable proxies — hand out access you can cancel later:
const { proxy, revoke } = Proxy.revocable({ secret: 42 }, {});
proxy.secret;   // 42
revoke();
proxy.secret;   // TypeError: revoked — the target is now unreachable through it
// 4. Auto-mocking / method-missing (how test spies work):
const api = new Proxy({}, {
  get(t, prop) {
    return (...args) => console.log(`called ${String(prop)}(${args.join(", ")})`);
  }
});
api.anything(1, 2);      // "called anything(1, 2)" — no such method ever existed
When NOT to reach for Proxy: a getter/setter pair (Object.defineProperty) covers one known property with zero overhead and full engine optimization. Proxies defeat inline caches — every trapped operation is a function call, roughly 2–10× slower than a plain property access. Use a Proxy when the property names are unknown ahead of time or you must intercept structural operations (delete, enumeration, in). That's also why Vue 3 moved to Proxy: defineProperty (Vue 2) couldn't see added/deleted keys or array index writes.

Part 6 — Edge Cases & Gotchas (the Senior Layer)

// 1. Built-ins with internal slots break through naive proxies:
const p = new Proxy(new Map([["a", 1]]), {});
p.get("a");   // TypeError! Map.prototype.get reads the [[MapData]] internal slot,
              // and `this` is the proxy, which has no such slot.
// Fix: in the get trap, bind functions to the target:
const fixed = new Proxy(new Map([["a", 1]]), {
  get(t, prop, r) {
    const v = Reflect.get(t, prop, r);
    return typeof v === "function" ? v.bind(t) : v;
  }
});
fixed.get("a");   // 1  — same trick needed for Date, Set, and DOM nodes

// 2. Identity: proxy !== target. Using objects as Map/Set keys or in === checks
//    breaks if one side holds the proxy and the other the raw target.
//    (Vue exposes toRaw() precisely for this.)

// 3. this inside target methods is the proxy — method calls route reads/writes
//    through your traps again. Usually what you want (nested reactivity), but it
//    can cause infinite recursion if a trap itself touches the proxy.

// 4. JSON.stringify(proxy) works — it triggers ownKeys + get traps — but
//    structuredClone(proxy) and postMessage throw: proxies can't be cloned.

Interview Questions

Beginner: What is a Proxy, and what happens for operations you don't write a trap for?

A Proxy wraps a target object and lets you intercept fundamental operations (get, set, delete, enumeration, calls) with handler functions called traps. Any operation without a trap forwards transparently to the target — an empty handler {} makes a proxy that behaves identically to its target (except identity).

Beginner: What is Reflect and why does it mirror the Proxy traps?

Reflect exposes each internal object operation as a static function with the same signature as the corresponding trap. Inside a trap, calling the matching Reflect method with the trap's arguments performs the default behavior — so traps can observe or adjust an operation and then delegate, instead of reimplementing the language's semantics by hand.

Mid: Why must you pass receiver to Reflect.get/Reflect.set?

receiver is the object the operation originally started on. If the proxy sits in a prototype chain and the target property is an accessor, the getter/setter's this must be the receiver, not the target. target[prop] hard-codes this = target, silently breaking inherited getters; Reflect.get(target, prop, receiver) preserves the correct binding.

Mid: Sketch how Vue-3-style reactivity uses Proxy.

State is wrapped in a Proxy. The get trap records "currently running effect depends on this (target, key)" in a WeakMap of dependency sets; the set trap looks up that key's dependents and re-runs them. Rendering is just an effect, so reads during render auto-subscribe the component, and writes auto-rerender it. Vue 2's defineProperty approach couldn't detect added/deleted keys or index writes — Proxy traps those structurally.

Senior: Why does new Proxy(new Map(), {}).get("k") throw, and how do you fix it?

Map.prototype.get reads the [[MapData]] internal slot of its this. Called through the proxy, this is the proxy, which has no internal slots — internal-slot access isn't a trappable operation, so it fails with a TypeError. Fix: in the get trap, when the resolved value is a function, return it bound to the raw target. The same applies to Date, Set, WeakMap, and DOM objects.

Senior: What are proxy invariants, and why do they exist?

Invariants are spec-enforced consistency rules: e.g. a trap can't report a non-configurable own property as nonexistent or return a different value for it, and can't claim a non-extensible target has extra keys. The engine verifies trap results against the target's actual descriptors and throws TypeError on violation. They exist so code holding a "frozen" or non-configurable object can rely on those guarantees even through a hostile or buggy proxy — otherwise Object.freeze would be meaningless.

🏋️ Graded Exercises

1 (easy): Create a proxy that logs every property read on an object, then returns the real value.
Solution 
const logged = new Proxy(obj, {
  get(t, prop, r) { console.log("read", prop); return Reflect.get(t, prop, r); }
});
2 (easy): Make a proxy where reading any missing property returns the string "N/A" instead of undefined.
Solution 
const withDefault = new Proxy(obj, {
  get(t, prop, r) { return prop in t ? Reflect.get(t, prop, r) : "N/A"; }
});
// `prop in t` (not truthiness of the value!) so stored falsy values still come through.
3 (medium): Build readonly(obj): reads work, but any write or delete throws a descriptive TypeError.
Solution 
function readonly(obj) {
  return new Proxy(obj, {
    set(t, prop)            { throw new TypeError(`Cannot set "${String(prop)}": object is read-only`); },
    deleteProperty(t, prop) { throw new TypeError(`Cannot delete "${String(prop)}": object is read-only`); },
    defineProperty(t, prop) { throw new TypeError(`Cannot define "${String(prop)}": object is read-only`); }
  });
}
// Unlike Object.freeze, the original stays mutable for whoever holds the raw reference.
4 (medium): Create a case-insensitive object: o.NAME, o.Name, and o.name all reach the same property, for get, set, and in.
Solution 
function caseInsensitive(obj) {
  const norm = p => typeof p === "string" ? p.toLowerCase() : p;
  return new Proxy(obj, {
    get: (t, p, r)    => Reflect.get(t, norm(p), r),
    set: (t, p, v, r) => Reflect.set(t, norm(p), v, r),
    has: (t, p)       => Reflect.has(t, norm(p))
  });
}
5 (medium): Implement hideUnderscored(obj): properties starting with _ are invisible — absent from Object.keys, in, and reads return undefined.
Solution 
function hideUnderscored(obj) {
  const hidden = p => typeof p === "string" && p.startsWith("_");
  return new Proxy(obj, {
    get: (t, p, r) => hidden(p) ? undefined : Reflect.get(t, p, r),
    has: (t, p)    => hidden(p) ? false : Reflect.has(t, p),
    ownKeys: t     => Reflect.ownKeys(t).filter(p => !hidden(p))
  });
}
// Works because _props are configurable — hiding non-configurable keys would violate invariants.
6 (hard): Build tracked(fn) using the apply trap: the returned function works normally but records { args, result, ms } of every call in a .calls array.
Solution 
function tracked(fn) {
  const calls = [];
  const proxy = new Proxy(fn, {
    apply(t, thisArg, args) {
      const start = performance.now();
      const result = Reflect.apply(t, thisArg, args);
      calls.push({ args, result, ms: performance.now() - start });
      return result;
    },
    get(t, p, r) { return p === "calls" ? calls : Reflect.get(t, p, r); }
  });
  return proxy;
}
const add = tracked((a, b) => a + b);
add(2, 3); add(10, 5);
add.calls.length;   // 2 — this is the skeleton of every test-spy library.
7 (hard): Extend Part 4's reactive() so nested objects are reactive too: state.user.name = "x" must trigger effects that read state.user.name.
Solution 
const proxyCache = new WeakMap();
function reactive(target) {
  if (proxyCache.has(target)) return proxyCache.get(target);  // stable identity per target
  const proxy = new Proxy(target, {
    get(t, prop, receiver) {
      track(t, prop);
      const value = Reflect.get(t, prop, receiver);
      return typeof value === "object" && value !== null
        ? reactive(value)          // wrap lazily, on read — not eagerly on creation
        : value;
    },
    set(t, prop, value, receiver) {
      const ok = Reflect.set(t, prop, value, receiver);
      trigger(t, prop);
      return ok;
    }
  });
  proxyCache.set(target, proxy);
  return proxy;
}
// The cache is essential: without it, state.user !== state.user (new proxy per read),
// breaking === checks and dependency identity. This is exactly Vue 3's approach.
8 (hard): Why does this loop forever, and how do you fix it while keeping the logging? 
const p = new Proxy({ n: 1 }, {
  get(t, prop, r) { console.log("read", prop, "on", p.n); return Reflect.get(t, prop, r); }
});
Solution 
// p.n inside the trap goes THROUGH the proxy → fires the get trap again → reads p.n → …
// infinite recursion, stack overflow on the first read.
// Fix: read from the raw target inside the trap — t.n, not p.n:
const p = new Proxy({ n: 1 }, {
  get(t, prop, r) { console.log("read", prop, "on", t.n); return Reflect.get(t, prop, r); }
});
// General rule: trap bodies must only touch the target; touching the proxy re-enters the traps.

🎯 Key Takeaways

  • A Proxy replaces an object's internal methods with your trap functions; untrapped operations fall through to the target unchanged.
  • Inside every trap, delegate to the mirror-image Reflect method with all the trap's arguments — especially receiver, which keeps inherited getters' this correct.
  • set/deleteProperty/defineProperty traps must return booleans, and no trap may contradict the target's non-configurable/non-extensible facts (invariants — the engine enforces them).
  • Reactivity systems are two traps and two maps: get tracks (effect ← prop), set triggers; a WeakMap keys dependencies so GC cleans them up.
  • Built-ins with internal slots (Map, Set, Date) need method-binding in the get trap; proxy ≠ target identity bites in Maps and ===; trap bodies must touch the target, never the proxy.
  • Proxies cost real performance per operation — wrap boundaries (stores, config, spies), not hot loops; for one known property, a plain getter/setter wins.
