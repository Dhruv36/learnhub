Variables, Types & Operators Basics

JavaScript runs in every browser, on servers (Node), phones, and most interview screens. It's also a language whose 1995 quirks were frozen forever by backward compatibility — so professional JS is a disciplined subset: this track teaches that subset and names every trap you'll be tested on. This first lesson covers how values work — which sounds basic and is actually where half of all JS interview questions live.

Declarations — the 2-Rule System (and why var died)

const apiUrl = "https://api.app.com";   // DEFAULT: block-scoped, must init, no reassignment
let retries = 0;                         // only when reassignment is genuinely needed

🧠 const ≠ immutable. const freezes the binding (the name→value arrow), not the value: const arr = []; arr.push(1) is fine; arr = [] throws. Shallow value-freezing is Object.freeze(obj); deep freezing requires recursion. This distinction is a guaranteed interview checkpoint.

Equality — the Full Story

1 == "1"          // true  — == COERCES types by arcane rules
1 === "1"         // false — === compares type AND value: THE default
null == undefined // true  — the ONE useful ==: x == null checks both nullish states
NaN === NaN       // false — use Number.isNaN
{a:1} === {a:1}   // false — objects compare by REFERENCE, not shape
Object.is(NaN, NaN)  // true; Object.is(-0, 0) // false — fixes ===' s two edge cases

Codebases enforce eqeqeq (ESLint): always ===, with x == null as the single sanctioned exception. Deep equality of objects needs a utility — the language doesn't provide one.

Truthiness & the Modern Operator Trio

// The EIGHT falsy values — memorize; everything else is truthy:
false, 0, -0, 0n, "", null, undefined, NaN
// Traps: "0" is truthy. [] is truthy. {} is truthy. "false" is truthy.

const name  = user.nickname ?? "Anonymous";   // ?? : default ONLY for null/undefined
const limit = config.limit || 10;             // || : default for ANY falsy — eats legit 0/""!
const city  = user?.address?.city;            // ?. : safe navigation — undefined, no TypeError
user.save?.();                                 // optional CALL — runs only if save exists
const label = count === 1 ? "item" : "items"; // ternary: value selection (one level max)
⚠️ || vs ?? is a real bug class: volume || 50 turns a user's chosen volume of 0 into 50. ?? exists precisely for this. Related discipline: don't chain ?. everywhere defensively — if user.address MUST exist, crashing early beats silently computing with undefined (errors lesson).

Coercion — the Rules Worth Knowing (skip the rest)

"5" + 3    // "53"  — + with a string CONCATENATES
"5" - 3    // 2     — every other math operator coerces to number
+"42"      // 42    — unary plus: terse string→number
1 < 2 < 3  // true, but by accident: (1<2)→true→1, 1<3 ✓
3 > 2 > 1  // false! (3>2)→true→1, 1>1 ✗ — chained comparisons don't exist

// Explicit conversion — say what you mean:
Number("12px")   // NaN (whole string must be numeric)
parseInt("12px", 10)  // 12 (reads leading digits; always pass radix in old codebases)

Interview Questions

Q1: Explain the Temporal Dead Zone precisely.

let/const ARE hoisted — the binding is created at scope entry — but left uninitialized until the declaration line executes. Accessing it in between throws ReferenceError ("Cannot access before initialization" — note: a DIFFERENT error than an undeclared variable). var instead hoists pre-initialized to undefined, which silently hides use-before-define bugs — the TDZ is a feature, converting that bug class into loud errors.

Q2: Why does 0.1 + 0.2 !== 0.3, and how do you handle it in a payments app?

IEEE-754 doubles store binary fractions; 0.1 and 0.2 are repeating in binary, so both are stored as nearest representable values — their sum lands on 0.30000000000000004. Payments: never floats — integer minor units (cents), BigInt for huge sums, or a decimal library; format only at display time with Intl.NumberFormat. Comparisons elsewhere: Math.abs(a-b) < Number.EPSILON * scale.

Q3: What exactly does const guarantee — and how would you get real immutability?

Only binding immutability: the name can't be re-pointed. The value, if an object, stays mutable. Real immutability: Object.freeze (shallow — nested objects still mutable), recursive deep-freeze, or immutable-by-convention with types (readonly in TS) and immutable update patterns (spread) — which is what React-style codebases actually do.

Q4: When is BigInt necessary, and what are its sharp edges?

Integers beyond 2⁵³−1: 64-bit database IDs, high-precision counters, crypto math. Edges: can't mix with Number in arithmetic (TypeError — convert explicitly), JSON.stringify throws on it, Math.* doesn't accept it, and equality 10n == 10 is true but 10n === 10 is false. Many APIs just use strings for big IDs instead.

🏋️ Practice (Playground console — F12):
1. Predict, then verify: typeof null, [] + [], [] + {}, "5" - - "2", 0.1+0.2 === 0.3.
2. Write getConfig(userValue) that defaults to 10 — prove || breaks for 0 and ?? doesn't.
3. Demonstrate the TDZ: log a let before declaring it inside a block; compare to var.
4. Build a price formatter storing cents as integers, displaying via Intl.NumberFormat("en-US", {style:"currency", currency:"USD"}).

🎯 Key Takeaways

  • const by default, let when needed, var never — block scope + TDZ turn silent bugs into loud errors.
  • Seven value-copied primitives + reference-copied objects — the distinction behind equality, copying, and mutation bugs.
  • === always (x == null is the one exception); objects compare by reference; NaN needs Number.isNaN.
  • Eight falsy values; ?? for defaults (|| eats 0/""); ?. for genuine optionality, not defensive soup.
  • One float number type: money in integer cents; 2⁵³ is the integer cliff; BigInt beyond it.
  • + concatenates with strings, everything else coerces to number — convert explicitly instead of memorizing tables.
