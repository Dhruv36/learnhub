Variables, Types & Declarations Language Core

What you'll master: the three ways to declare a binding and why exactly two of them survive in professional code, what "hoisting" really means (a runtime mechanic, not folklore), the Temporal Dead Zone and why it's a feature, how scope actually resolves names, and the binding-vs-value distinction that underlies half the mutation bugs you'll ever debug. This is lesson one — no prerequisites beyond having seen a line of code before. The type system itself (the eight types, coercion, equality) gets full lessons of its own next; here we build the ground every later lesson stands on: how names come to hold values, and where those names are visible.

Part 1 — A Variable Is a Name Bound to a Value

let score = 10;      // "score" is a NAME; 10 is a VALUE; = BINDS them
score = 20;          // rebinding: the name now points at a different value
console.log(score);  // 20

Say this precisely, because the precise version pays off later: a variable is a binding — an arrow from a name to a value. Assignment (=) points the arrow. The name and the value are separate things: the name lives in a scope (Part 5), the value lives in memory. Almost every "wait, why did that change?" bug comes from confusing the arrow with the thing it points at — which is exactly what Part 6 is about. Keep the picture: name → value.

Part 2 — The Three Declarations, and the Two-Rule System

const apiUrl = "https://api.app.com";   // DEFAULT: block-scoped, must initialise, cannot rebind
let retries = 0;                        // only when you genuinely reassign the binding later
retries += 1;

var legacy = 1;                          // the pre-2015 keyword — recognise it, essentially never write it

Modern JavaScript is a disciplined subset of everything the language technically allows, and the very first rule of that subset is a two-line decision procedure: reach for const by default; downgrade to let only when you will actually rebind the name; never write var. That's not style preference — each clause prevents a concrete bug class, which the rest of this lesson demonstrates.

🧠 Why const first, not let first? A binding that never rebinds is a binding you never have to track through a function to know its current value — you read the one line that created it and you're done. const turns "does this change anywhere below?" from a question you must answer by scanning into one the compiler answers for you. That's the entire ergonomic argument: less to hold in your head, per name.

Part 3 — Hoisting Is Not Magic: It's Two-Phase Execution

Before any line runs, the engine does a quick pass over each scope and registers every declaration it finds — that pass is "hoisting." It does not move your code; it means the names exist from the moment the scope is entered. What differs by keyword is the state each name is in during the gap between scope-entry and the declaration line:

console.log(a);   // undefined  — var: name exists AND is pre-set to undefined
var a = 1;

console.log(b);   // ReferenceError: Cannot access 'b' before initialization
let b = 1;        // let/const: name exists but is UNINITIALISED — the Temporal Dead Zone

console.log(c);   // ReferenceError: c is not defined  — no declaration anywhere: name doesn't exist at all

Three genuinely different outcomes, three different meanings — and the error messages tell you which is which. var's "helpfully" pre-initialising to undefined is exactly what makes use-before-declare bugs silent: the code runs, produces undefined, and fails somewhere later far from the cause. Note the last case too: "is not defined" (no such name anywhere) is a different error than "Cannot access before initialization" (name exists, in its TDZ) — reading which one you got is a real debugging shortcut.

Function declarations hoist differently — the whole function

greet();                       // "hi" — WORKS: function declarations hoist name AND body
function greet() { return "hi"; }

sayBye();                      // TypeError: sayBye is not a function
var sayBye = function () {};   // only the var NAME hoists (as undefined); the function value doesn't

A function declaration is fully available throughout its scope, body and all — this is why you can call helpers defined lower in a file. A function expression assigned to a variable hoists only by the variable's rules (so a var holding a function is undefined until the assignment runs). One more reason the const/let + arrow-function combination is predictable: it obeys the TDZ, so calling it too early throws loudly instead of the confusing "undefined is not a function".

Part 4 — The Temporal Dead Zone, Precisely

{
  //  ↑ TDZ for `x` STARTS here (top of the block)
  console.log(typeof x);   // ReferenceError — even typeof throws inside the TDZ (unusual!)
  const y = 5;
  let x = 10;              // ← TDZ for `x` ENDS here, at the declaration
  console.log(x);          // 10 — fine from here on
}

The TDZ is the region from the top of the scope down to the declaration line, during which the name exists (so it shadows any outer name of the same name) but touching it throws. It's not a bug or a quirk to route around — it exists so that use-before-initialisation becomes a loud, immediate error at the exact spot instead of a silent undefined that propagates. Note the genuinely surprising detail worth remembering for interviews: typeof, which is normally the one operator safe to use on any undeclared name, also throws inside a TDZ — because the name is declared, just not yet initialised.

Part 5 — Scope: How a Name Resolves to a Value

When the engine hits a name, it looks for it in the current scope; if not found, the enclosing scope; outward, up the chain, to the global scope; and if still not found, ReferenceError. That chain is fixed by where code is written (lexical scope), not by who calls it — a fact the Closures lesson turns into a superpower.

const app = "global";                 // module/global scope

function outer() {
  const layer = "outer";              // outer's function scope
  {
    const inner = "block";            // block scope — const/let respect { }
    console.log(app, layer, inner);   // "global outer block" — all visible looking outward
  }
  // console.log(inner);              // ReferenceError — inner doesn't leak out of its block
}

Block scope is why let fixed the classic loop bug

// The interview classic — with var, all three log 3:
for (var i = 0; i < 3; i++) { setTimeout(() => console.log(i)); }   // 3, 3, 3
// One `i` shared by every callback; by the time they run, the loop finished and i === 3.

// With let, each iteration gets a FRESH binding:
for (let j = 0; j < 3; j++) { setTimeout(() => console.log(j)); }   // 0, 1, 2

This is the single most-asked scope question in interviews, and the explanation is entirely in this lesson's vocabulary: var i is one function-scoped binding the whole loop mutates, so all deferred callbacks close over the same, final i. let j creates a new block-scoped binding per iteration, so each callback closes over its own snapshot. No tricks, no IIFE workaround needed anymore — block scope did it. (The "closes over" mechanic itself is the Closures lesson.)

Part 6 — The Binding Is Not the Value: const, Copying, and Mutation

const arr = [1, 2, 3];
arr.push(4);        // ✅ FINE — we mutated the VALUE the binding points at
console.log(arr);   // [1, 2, 3, 4]
arr = [];           // ❌ TypeError: Assignment to constant variable — rebinding the NAME

const freezes the arrow, not the thing it points at. The binding can't be re-pointed; the object it points at is as mutable as ever. This trips up nearly everyone once — and it's the crisp version of the name→value picture from Part 1. If you need the value itself to be unchangeable, that's a different tool:

const config = Object.freeze({ retries: 3 });
config.retries = 5;          // silently ignored in sloppy mode; throws in strict mode
console.log(config.retries); // 3 — but ONLY one level deep:
const deep = Object.freeze({ nested: { x: 1 } });
deep.nested.x = 99;          // ✅ succeeds — freeze is SHALLOW; nested object is still mutable

Two more facts you'll code around constantly, both consequences of name→value:

  • Primitives copy by value; objects copy by reference. let b = a for a number copies the number; for an object it copies the arrow, so a and b now point at the same object and a mutation through one is visible through the other. This is the root of the whole "I changed one and the other changed too" bug family — the Objects and Arrays lessons cover copying properly (spread, structuredClone).
  • Object.freeze is one level deep. Real deep immutability needs recursion or a library, and in practice most teams reach for immutable update patterns (never mutate; produce a new object) instead — the approach React-style code is built on.

Part 7 — Build It: A Small Scope-and-Binding Demo

Let's assemble the ideas into one runnable snippet that makes each mechanic visible. Paste it into the console (F12) and predict each line before it runs.

"use strict";                       // opt into strict mode — makes silent failures throw (Part 8)

function makeCounter(start) {
  let count = start;                // block/function-scoped, rebindable — correct use of let
  const step = 1;                   // never rebinds — const
  return {
    inc() { count += step; return count; },   // closes over count & step (Closures lesson)
    value() { return count; }
  };
}

const c = makeCounter(10);
console.log(c.inc());   // 11
console.log(c.inc());   // 12
console.log(c.value()); // 12
// `count` is invisible out here — it lives in makeCounter's scope, reachable ONLY through the
// returned methods. That's genuine private state, built entirely from scope + const/let. No class,
// no #private field, no convention — just this lesson's rules, composed.

Everything in that snippet is from Parts 2–6: const for the constant, let for the one rebindable name, block scope hiding count from the outside, and the returned methods keeping the binding alive. That's the payoff of getting the fundamentals exactly right — the advanced patterns are just these rules combined.

Part 8 — Strict Mode, Briefly (and Why Modules Give It Free)

"use strict";     // at the top of a script or function
undeclared = 5;   // ReferenceError (in sloppy mode this SILENTLY creates a global — a classic leak)

Strict mode turns a set of silent, dangerous behaviours into errors: assigning to an undeclared name, writing to a frozen/read-only property, duplicate parameter names, and more. You rarely type "use strict" yourself anymore, because ES modules and class bodies are strict automatically — which is one more quiet reason to write modern module-based code (the Modules lesson). Know it exists, know it's the safer default, and know that the loose, forgiving behaviour it disables is exactly the legacy baggage this track keeps steering you away from.

Interview Questions

Q1 (beginner): What's the difference between let and const, and which should you default to?

Both are block-scoped and both obey the Temporal Dead Zone (hoisted but uninitialised). The only difference is rebinding: let allows reassigning the binding to a new value; const forbids it (a TypeError). Default to const — it communicates and enforces "this name never re-points," which is one less thing a reader has to verify by scanning the scope. Downgrade to let only when you genuinely reassign the name (a loop counter, an accumulator). Neither makes an object's value immutable — that's a separate concern.

Q2 (beginner): Explain what "hoisting" means. Does let hoist?

Hoisting is the engine registering every declaration in a scope during a pass made before the scope's code runs, so the names exist from scope entry. var, let, const, and function declarations all hoist — the myth is that only var does. What differs is the state the name is in before its declaration line: var is pre-initialised to undefined (so reads are silently allowed); let/const are uninitialised and in the Temporal Dead Zone (reads throw); function declarations hoist their entire body (so they're callable before their line).

Q3 (mid): Precisely explain the Temporal Dead Zone, and why typeof behaves surprisingly in it.

The TDZ is the span from the top of a scope down to a let/const declaration line, during which the name is declared (it shadows outer names) but uninitialised, so any access throws ReferenceError: Cannot access before initialization. It's a deliberate feature: it converts use-before-initialisation from a silent undefined (the var behaviour) into a loud error at the exact site. The surprise: typeof someUndeclaredName normally returns "undefined" safely, but typeof x inside x's TDZ throws — because x IS declared, just not yet initialised, so it isn't the "totally unknown name" case typeof is safe for.

Q4 (mid): Why do var and let behave differently in a for loop with a deferred callback (e.g. setTimeout)?

var i creates one function-scoped binding that the entire loop mutates in place. Every deferred callback closes over that same single binding, so when they finally run — after the loop has completed — they all read the final value. let j creates a fresh block-scoped binding for each iteration, and each callback closes over its own iteration's binding, capturing that iteration's value. So var logs the final value N times; let logs 0,1,2,…. Before let, the fix was an IIFE per iteration to manufacture a fresh scope; block scope makes that unnecessary.

Q5 (senior): A teammate argues "we should use let everywhere for flexibility, since const is restrictive." Make the counter-case rigorously.

const isn't about restricting the author; it's about informing the reader. A const binding is a local invariant: anyone reading the code knows this name's value is established once and never re-points, so reasoning about it requires reading exactly one line, not scanning the whole scope for reassignments. That compounds: in a 40-line function, the difference between "3 of these names might change and I must track which" and "all are fixed except these 2 explicit lets" is a real cognitive load reduction and a real bug-surface reduction (accidental reassignment becomes a compile-time-ish error). "Flexibility" here means "the freedom to introduce mutation the reader now has to account for" — which is a cost, not a benefit. The disciplined default is const, with let as a deliberate, visible signal that "yes, this one really does change." Note it still says nothing about value immutability — a const object is mutable — so it's an honest, cheap guarantee, not an over-claim.

Q6 (senior): Explain the difference between binding immutability and value immutability, and how you'd achieve genuine deep immutability in production.

Binding immutability (const) locks the name→value arrow: the name can't be re-pointed. Value immutability locks the pointed-at object's contents. const gives only the first; a const object's properties still mutate freely. Object.freeze gives value immutability but only one level deep — nested objects remain mutable, so a "frozen" object with object-valued properties isn't truly immutable. Genuine deep immutability options, in increasing order of practicality: (1) a recursive deep-freeze that walks and freezes every nested object (guarding against cycles) — correct but has runtime cost and mutates nothing at the type level; (2) TypeScript's readonly/Readonly<T> and as const — compile-time enforcement, zero runtime cost, but no runtime guarantee; (3) the approach most large codebases actually use: immutable update patterns — never mutate, always produce a new object/array via spread or a library (Immer, Immutable.js) — which sidesteps the freezing question entirely and is what state-management libraries (Redux, React state) are built around. In practice the winning combination is TS readonly for authoring guarantees plus immutable update discipline at runtime, reserving deep-freeze for a debug-mode guard when you want mutations to throw loudly during development.

🏋️ Exercises

1 (easy): Predict the errors. Without running it, say what each line logs or throws, and the exact error name where it throws: 
console.log(a); var a = 1;
console.log(b); let b = 2;
console.log(c);
foo(); function foo(){ return 1; }
Answers

Line 1: undefined — var a hoists pre-initialised. Line 2: ReferenceError: Cannot access 'b' before initialization — let in its TDZ. Line 3: ReferenceError: c is not defined — no declaration anywhere (a different error than line 2's). Line 4: works, returns 1 — function declarations hoist body and all.

2 (easy): const or let? For each, pick the correct keyword and justify: (a) the value of π; (b) a running total inside a loop; (c) a DOM element you look up once; (d) the current retry attempt number that increments.
Solution

(a) const — never changes. (b) let — reassigned each iteration. (c) const — looked up once, the binding never re-points (the element may mutate, but the binding doesn't). (d) let — it increments, i.e. rebinds. The rule in action: default const, downgrade to let only for (b) and (d) where you truly reassign.

3 (medium): Fix the loop. This logs 5 5 5 5 5; change exactly one keyword to make it log 0 1 2 3 4, and explain why the change works. 
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 0);
}
Solution

Change var i to let i. With var there's one shared function-scoped binding the loop mutates to its final value 5 before any timeout fires, so all five callbacks read 5. With let, each iteration gets a fresh block-scoped binding and each callback closes over its own, capturing 0,1,2,3,4 respectively.

4 (medium): The aliasing trap. Predict the output, then fix copy so that mutating it does not affect original: 
const original = { count: 1 };
const copy = original;
copy.count = 99;
console.log(original.count);
Solution

It logs 99 — const copy = original copies the reference (the arrow), so both names point at the same object; mutating through copy is visible through original. Fix by copying the value, not the reference: const copy = { ...original } (shallow) or const copy = structuredClone(original) (deep). Now original.count stays 1. Note the aliasing is a consequence of objects copying by reference, not a const issue — let would behave identically.

5 (hard): Deep-freeze. Object.freeze is shallow. Write deepFreeze(obj) that recursively freezes an object and all nested objects/arrays, and prove it stops a nested mutation. Handle the case where a property is already frozen.
Solution 
function deepFreeze(obj) {
  for (const key of Object.keys(obj)) {
    const val = obj[key];
    if (val !== null && typeof val === "object" && !Object.isFrozen(val)) {
      deepFreeze(val);                 // recurse BEFORE freezing the parent
    }
  }
  return Object.freeze(obj);
}

const s = deepFreeze({ user: { name: "Ada", roles: ["admin"] } });
"use strict";
try { s.user.name = "x"; } catch (e) { console.log(e.name); } // TypeError in strict mode
console.log(s.user.name);            // "Ada" — nested mutation blocked

The Object.isFrozen check both avoids redundant work and prevents infinite recursion on cyclic references (a frozen node won't be re-entered). In sloppy mode the assignment fails silently rather than throwing — another reason to run under strict mode / modules, where the violation is loud.

6 (hard): Private state without a class. Using only const/let and scope (no classes, no # fields), write createAccount(initial) returning { deposit, withdraw, balance } where the balance can't be read or mutated except through those methods, and withdraw refuses to overdraw.
Solution 
function createAccount(initial) {
  let balance = initial;             // truly private: only the closures below can see it
  return {
    deposit(n) { balance += n; return balance; },
    withdraw(n) {
      if (n > balance) throw new Error("Insufficient funds");
      balance -= n; return balance;
    },
    balance() { return balance; }
  };
}

const a = createAccount(100);
a.deposit(50);          // 150
a.withdraw(30);         // 120
// a.balance is a function; there is NO way to reach the `balance` variable directly.

The balance binding lives in createAccount's scope and is reachable only through the returned methods' closures — genuine encapsulation from Parts 5–7's rules alone. This is the classic "module pattern," and it's the conceptual seed of both the Closures lesson and private class fields in the Classes lesson.

🎯 Key Takeaways

  • A variable is a binding: a name→value arrow. Assignment points the arrow; the name lives in a scope, the value in memory.
  • const by default, let only when you truly rebind, var never — each clause kills a concrete bug class.
  • Hoisting registers all declarations before code runs; var reads undefined, let/const throw in the TDZ, function declarations are fully callable early.
  • The TDZ is a feature: use-before-init becomes a loud error at the site instead of a silent undefined — and even typeof throws inside it.
  • Scope is lexical (set by where code is written); block scope is why let fixed the classic loop-closure bug with no IIFE.
  • const freezes the binding, not the value; objects still mutate and copy by reference; Object.freeze is shallow — deep immutability needs recursion or immutable patterns.
