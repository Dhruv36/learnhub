Performance Patterns at Scale Ultra-Advanced

Two budgets govern frontend performance: bytes shipped (how fast it loads) and main-thread milliseconds (how responsive it feels — Core Web Vital INP). Every technique here serves one of those, and every one is preceded by the same discipline: measure, fix the top item, re-measure. Optimization without measurement is superstition with extra steps.

Budget 1 — Ship Less JavaScript (the biggest lever, always)

// Route-level code splitting — each page loads only its code:
const Settings = lazy(() => import("./pages/Settings"));       // React
// Feature-level — heavy/rare functionality on demand:
exportBtn.onclick = async () => (await import("./pdf.js")).generate(report);   // 800KB deferred

// Audit before cutting (measure first!):
npx vite-bundle-visualizer            // who is 40% of the bundle?
// Classic wins:
//   moment (300KB) → date-fns / native Intl        lodash → lodash-es named imports (tree-shakes)
//   drop polyfills for dead browsers (browserslist)  charts/editors/PDF → dynamic import
//   check duplicate versions: npm dedupe / npm ls

Why bytes dominate: JS is the most expensive resource byte-for-byte — it's downloaded, parsed, compiled, AND executed (and hydration cost scales with it). A 200KB JS bundle hurts more than a 200KB image. Set a CI budget (bundle ≤ N KB) so it can't regress silently.

Budget 2 — Main-Thread Patterns

// DEBOUNCE user-driven storms (search-as-you-type) — act after silence:
input.oninput = debounce(runSearch, 250);
// THROTTLE continuous streams (scroll/resize/mousemove) — act at a steady rate:
window.onscroll = throttle(updateHeader, 100);
// (Both are closures — see the closures lesson. Create ONCE, not per render/event.)

// Defer non-urgent work to idle time:
requestIdleCallback(() => prefetchLikelyNextPage());
// Sync visual work to frames:
requestAnimationFrame(step);                     // animations, batched DOM reads
// Break long tasks (the #1 INP killer) — yield so input/paint interleave:
for (const chunk of chunks) { process(chunk); await scheduler?.yield?.() ?? nextTask(); }
// Prioritize with the Scheduler API:
scheduler.postTask(work, { priority: "background" });   // user-blocking | user-visible | background

Don't Thrash the DOM (the runtime villain from the CSS lesson, JS side)

// ❌ read-write-read-write → forced synchronous layout EVERY iteration:
items.forEach(el => { el.style.height = el.offsetHeight * 2 + "px"; });
// ✅ batch: read all, then write all → ONE layout:
const hs = items.map(el => el.offsetHeight);
items.forEach((el, i) => el.style.height = hs[i] * 2 + "px");

// Huge lists: render only the VISIBLE window (virtualization) — 100k rows, ~30 in the DOM:
//   @tanstack/virtual, react-window; the idea = absolute-positioned window + scroll math
// Observe visibility with IntersectionObserver, never scroll + getBoundingClientRect per frame.
// content-visibility: auto (CSS) skips rendering off-screen sections — pairs with this.

Caching & Memoization — the Same Idea at Every Layer

// In-memory memoize (pure functions only!):
const memoize = (fn) => { const c = new Map();
  return (k) => c.has(k) ? c.get(k) : (c.set(k, fn(k)), c.get(k)); };
// ⚠ unbounded Map = leak (memory lesson) — bound it (LRU) for hot caches.

// The layers, all the same pattern:
//   React.memo / useMemo / useCallback   → skip recompute/re-render on equal inputs (reference equality!)
//   HTTP Cache-Control / ETags           → skip the network
//   Service Worker cache                 → offline + instant repeat loads
//   CDN edge cache                        → skip the origin  (System Design track's caching lesson)

📈 The order that actually works: measure (RUM tells you WHICH metric and page) → attack the dominant bucket (bytes → parse/execute → thrash) → fix ONE thing → re-measure → lock it with a budget. Teams with fast sites (Google, Shopify, Vercel customers) win by watching p75 RUM dashboards and gating CI — so the 1000th PR is as fast as the first. Heroic one-off optimizations without budgets regress within a quarter.

Interview Questions

Q1: A page scores well in Lighthouse but users say it's sluggish. Reconcile that.

Lighthouse is a lab test on a simulated device/network at load time; "sluggish" is usually INTERACTION latency (INP) on real mid-range hardware after load — long tasks from heavy handlers, un-virtualized lists, layout thrashing, or a bloated third-party. Reconcile with FIELD data: web-vitals RUM at p75, then a Performance recording of the specific slow interaction. Lab tells you about load; the field tells you about use.

Q2: Debounce vs throttle — implementation difference and when each is correct.

Debounce: reset a timer on every call, fire only after ms of SILENCE — search-as-you-type, resize-settled, autosave. Throttle: fire at most once per ms DURING activity (timestamp or leading/trailing timer) — scroll position, mousemove, rate-limited sends. Both hold state in a closure and must be created once. Naming which UX each fits is the real signal.

Q3: 100,000-row table must scroll smoothly. Design it.

Virtualize: only render the ~30 visible rows (+overscan) into an absolutely-positioned window inside a spacer sized to total height; update on scroll (throttled/rAF). Data: paginate/stream from the server, index by id (Map) for O(1) row lookup, memoize row rendering. Avoid per-row listeners (delegate). Optionally content-visibility for off-screen chunks. The phrase: "constant DOM size independent of data size."

Q4: Where does memoization appear across the whole stack, and what's the shared risk?

Function memoize (Map cache) → React useMemo/memo (reference-equality skip) → HTTP caching (Cache-Control/ETag) → Service Worker cache → CDN edge → DB query cache. Shared risks: unbounded growth (leak — bound with LRU/TTL) and staleness/invalidation ("one of the two hard problems"). Same trade-offs the System Design track's caching lesson covers at datacenter scale — it's one idea repeated at every layer.

🏋️ Practice:
1. Run Coverage + bundle-visualizer on a real app; identify the largest dependency and propose a lighter path.
2. Implement debounce and throttle; wire debounce to a search input and throttle to scroll; log to see the behavioral difference.
3. Build a 10k-row list naively (freezes) then virtualize it (or use react-window); compare with the Performance panel.
4. Reproduce layout thrashing on 500 nodes, record it, fix with read/write batching, re-record.
5. Add web-vitals to a page, log LCP/INP/CLS to the console, and interact to watch INP change.

🎯 Key Takeaways

  • Two budgets: bytes (loading) and main-thread ms (INP). JS bytes are the priciest resource — split routes/features, audit, budget in CI.
  • Debounce (act on silence) / throttle (steady rate) / rIC (idle) / rAF (frames) / scheduler.yield (break long tasks) — created once.
  • Batch DOM reads then writes (no thrash); virtualize long lists; index with Maps not find-in-loop.
  • Memoization/caching is one pattern at every layer (fn → React → HTTP → SW → CDN) — always bound it and mind staleness.
  • Measure → fix the top item → re-measure → lock with a budget; p75 field RUM is the truth, the lab is a proxy.
  • Third-party scripts are often your biggest main-thread cost — audit them like your own code.
