Modules & Tooling Intermediate

Modules give code explicit boundaries and dependencies; tooling turns thousands of module files into what browsers efficiently download. This lesson is the bridge between "writing JavaScript" and "working on a JavaScript team" — including the why behind every tool, so the stack stops being cargo cult.

ES Modules — Syntax & Semantics

// utils/format.js
export const formatPrice = (cents) => `$${(cents / 100).toFixed(2)}`;
export const CURRENCY = "USD";
export default class Formatter { … }            // one default max (prefer named — see table)

// checkout.js
import Formatter, { formatPrice, CURRENCY as CUR } from "./utils/format.js";
import * as fmt from "./utils/format.js";        // namespace object
import "./analytics.js";                          // side-effect only (runs, imports nothing)
const { Chart } = await import("./charts.js");    // DYNAMIC — a promise, and a code-split point
SemanticConsequence
Static structure (imports at top, analyzable without running)Bundlers can tree-shake (drop unused exports), verify names at build time, and parallelize loading — the property CommonJS's dynamic require() lacks
Live bindings, not copiesimport { count } sees the exporter's CURRENT value if it changes — exports are views, not snapshots
Singleton per moduleFirst import runs the module once; everyone shares that instance — module state is shared state (deliberate for caches, accidental for bugs)
Always strict; own scope; top-level this = undefinedSloppy-mode footguns gone; globals must be explicit (globalThis)
Circular imports tolerated via hoistingBut a cycle where A uses B's value at load time gets undefined — restructure shared deps downward
Named vs defaultNamed exports: better autocomplete, refactor-safe renames, no name drift (import { Button } is always Button) — most style guides prefer them
// Browser:    (defers automatically, CORS rules apply)
// Node:    "type": "module" in package.json  (or .mjs)
// Legacy CommonJS — recognize, interoperate, don't write new:
const fmt = require("./format");  module.exports = { formatPrice };

Code Splitting — Where Bundles Get Cut

// Route-level (the big win — each page loads only its code):
const Settings = lazy(() => import("./pages/Settings"));      // React
const routes = [{ path: "/settings", component: () => import("./Settings.vue") }];

// Feature-level (heavy, rarely-used):
exportBtn.onclick = async () => {
  const { generatePdf } = await import("./pdf.js");            // 800KB loads on demand
  generatePdf(report);
};

Interview Questions

Q1: Why can bundlers tree-shake ESM but not CommonJS?

ESM imports/exports are static syntax — resolvable without executing code, so the bundler builds an exact usage graph and drops unexported/unimported code. require() is a runtime function call: conditional, dynamic, reassignable — the bundler can't prove what's unused. Live bindings vs value copies is the same static-vs-dynamic story.

Q2: npm install vs npm ci — and why does CI care?

install resolves ranges against the registry (may update the lockfile, may differ over time); ci installs EXACTLY the committed lockfile, fails if package.json disagrees, and starts clean. CI/deploys need reproducibility — the artifact you tested must be the artifact you ship. Bonus: provenance/audit gates and minimum-release-age policies for supply-chain safety.

Q3: Your initial bundle is 2.8MB. Walk your reduction plan.

Measure first (bundle visualizer): usually 2-3 libraries dominate. Then: route-level dynamic imports (biggest structural win), swap heavy deps (moment→date-fns/Intl, lodash→lodash-es named imports), lazy-load heavy features (charts, editors, PDF), check duplicate versions (npm dedupe), drop dead polyfills via browserslist, verify tree-shaking isn't blocked by barrels/ side-effect flags. Finish with a CI size budget so it never regresses.

Q4: What actually happens when two modules import the same module?

The module executes once; both importers receive live bindings into the same module instance (per graph/realm). Consequences: module-level caches work; module-level mutable state is shared (and hazardous under SSR where one process serves many requests); circular imports resolve via hoisting but expose half-initialized bindings if used at load time.

1. Scaffold a Vite vanilla-ts app; add a math.ts with named exports; verify HMR by editing while dev server runs.
2. Add a heavy fake module (big array) and load it only on button click via dynamic import; watch the Network panel chunk appear.
3. Break tree-shaking on purpose: import an entire library namespace vs one named function; compare build output sizes.
4. Set up ESLint+Prettier+husky+lint-staged; commit a badly formatted file and watch the hook fix it.
5. Create a deliberate circular import (a↔b) that reads a value at load time; observe, then fix by extracting shared code to c.

🎯 Key Takeaways

  • ESM: static, live-bound, singleton, strict — the static-ness IS why tree-shaking, build-time checks, and code splitting work.
  • Named exports over default; side-effect imports and dynamic import() are the special forms to know.
  • package.json is the contract; the lockfile is the truth — commit it, npm ci in automation.
  • Each tool has one job: Vite bundles, esbuild transpiles, ESLint catches bugs, Prettier ends debates, hooks enforce pre-commit.
  • Dynamic import = the code-split point; route-level splitting + a visualizer + CI budgets is the bundle-size playbook.
  • Frontend env vars are public; module-level mutable state is shared state — treat both accordingly.
