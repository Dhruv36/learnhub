Modules & Tooling Intermediate

Modules split code into files with explicit dependencies; tooling turns those files into what browsers download. This lesson is the bridge between "writing JS" and "working on a JS team".

ES Modules — the Syntax

// utils/format.js
export const formatPrice = (cents) => `$${(cents / 100).toFixed(2)}`;
export default class Formatter { … }          // one default per file (use sparingly)

// checkout.js
import Formatter, { formatPrice } from "./utils/format.js";
import * as fmt from "./utils/format.js";      // namespace import
const { heavy } = await import("./charts.js"); // DYNAMIC import — lazy-load on demand
FactConsequence
Imports are live bindings, hoisted, staticBundlers can analyze the graph → tree-shaking (dead code dropped)
Modules are singletonsImport the same file twice → same instance (easy shared state)
Module scope is privateNothing leaks to global unless exported
Always strict mode, this = undefinedOld sloppy-mode bugs can't happen

In the browser: <script type="module" src="app.js"> (defers automatically). In Node: "type": "module" in package.json. The old CommonJS (require/module.exports) still dominates older Node code — recognize it, write ESM.

npm & package.json — the Contract

{
  "name": "myapp",
  "type": "module",
  "scripts": { "dev": "vite", "test": "vitest", "build": "vite build" },
  "dependencies":    { "zod": "^3.23.0" },      // ships to production
  "devDependencies": { "vitest": "^2.0.0" }     // build/test only
}
// ^3.23.0 = any 3.x ≥ 3.23 — package-lock.json pins EXACT versions: commit it always.
// npm ci (not npm install) in CI: reproducible installs from the lockfile.

The Toolchain, Demystified

ToolJobStandard picks
Bundler/dev serverModule graph → optimized files + instant reloadVite (esbuild/Rollup inside)
TranspilerNew syntax/TS → target JSesbuild, SWC, Babel (legacy)
LinterCatch bugs & enforce rulesESLint
FormatterEnd style debates mechanicallyPrettier
Test runnerRun tests fastVitest / Jest

Why bundle at all if browsers support modules? A big app = thousands of modules; naive loading = thousands of requests, no minification, no tree-shaking. Bundlers produce a few cache-friendly, code-split chunks — dynamic import() is where they split.

💡 Production tip: the highest-leverage tooling habit is pre-commit automation — Prettier + ESLint on staged files (husky + lint-staged), and the same checks in CI. Style and lint issues then simply never reach review. New-project default in 2026: Vite + TypeScript + ESLint + Prettier + Vitest.
