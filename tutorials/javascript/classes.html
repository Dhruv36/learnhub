Classes Objects & Data

What you'll master: class syntax as sugar over prototypes, constructors and fields, instance vs static members, real private fields (#), getters/setters, inheritance with extends/super, and the judgment call of when classes actually earn their place versus plain objects and closures. Prerequisites: Prototypes (classes ARE prototype wiring) and this (methods run on the same this rules). Because you already understand the prototype chain, classes will feel like convenient notation rather than a new system.

Part 1 — Class Syntax Is Prototype Sugar

class Dog {
  constructor(name) {          // runs on `new Dog(...)`; sets up instance data
    this.name = name;
  }
  bark() {                     // methods go on Dog.prototype AUTOMATICALLY (Prototypes lesson)
    return `${this.name} barks`;
  }
}
const d = new Dog("Rex");
d.bark();                                          // "Rex barks"
Object.getPrototypeOf(d) === Dog.prototype;         // true — same wiring as constructor functions
typeof Dog;                                         // "function" — a class IS a function under the hood

A class is a constructor function with cleaner syntax. The constructor is the body that runs on new; methods declared in the class body are placed on Dog.prototype — so they're shared by all instances, exactly the memory-efficient arrangement from the Prototypes lesson, but without the manual Dog.prototype.bark = ... boilerplate. Two behavioural differences from plain constructor functions worth noting: class declarations are not hoisted in a usable way (they're in the TDZ until evaluated), and class bodies run in strict mode automatically. And calling a class without new throws — a helpful guard against the forgotten-new bug.

Part 2 — Fields, Methods, and Static Members

class Counter {
  count = 0;                   // INSTANCE FIELD — set on each instance before the constructor body
  step;                        // declared, initialised to undefined
  static instances = 0;        // STATIC FIELD — lives on the CLASS, shared, not on instances

  constructor(step = 1) {
    this.step = step;
    Counter.instances++;       // static accessed via the class name
  }
  increment() { this.count += this.step; return this.count; }   // instance method (on prototype)
  static reset() { Counter.instances = 0; }                     // STATIC method — called on the class
}

const c = new Counter(5);
c.increment();          // 5
Counter.instances;      // 1 — static, on the class
Counter.reset();        // static method, called on the class not an instance
// c.reset();           // ❌ TypeError — statics are NOT on instances

Instance fields (count = 0) are a concise way to declare per-instance data; they're assigned to each new object before the constructor body runs. Static members (fields and methods, marked static) belong to the class itself, not to instances — use them for constants, factory methods, counters, and utilities related to the type but not to any single object (Array.from, Math.max, and Object.keys are all static methods). A common mistake is trying to call a static method on an instance (or vice versa) — they live in different places.

Part 3 — Private Fields (#): Real Encapsulation

class BankAccount {
  #balance = 0;                // TRULY private — inaccessible outside the class body
  constructor(initial) { this.#balance = initial; }

  deposit(n) { this.#balance += n; return this.#balance; }
  withdraw(n) {
    if (n > this.#balance) throw new Error("Insufficient funds");
    this.#balance -= n; return this.#balance;
  }
  get balance() { return this.#balance; }         // read-only public view

  #audit() { /* private METHOD */ }               // methods can be private too
  static #count = 0;                              // static privates exist as well
}

const acc = new BankAccount(100);
acc.deposit(50);        // 150
acc.balance;            // 150 (getter)
acc.#balance;           // ❌ SyntaxError — genuinely private, not just convention
"#balance" in acc;      // false

#private fields are true hard privacy — accessing #balance from outside the class body is a SyntaxError, not a runtime workaround-able error. This replaces the old _balance naming convention (which was privacy by politeness — anyone could still read it). Private fields, methods, and static privates all use the # prefix, and the # is part of the name (there's no dynamic this["#x"] access). Use them to enforce invariants and hide implementation — the class-based equivalent of the closure/module pattern's private state (Closures lesson), with the advantage that methods are shared on the prototype rather than duplicated per instance.

Part 4 — Getters, Setters & Computed Access

class Temperature {
  #celsius = 0;
  get celsius() { return this.#celsius; }
  set celsius(v) {
    if (v < -273.15) throw new RangeError("below absolute zero");
    this.#celsius = v;
  }
  get fahrenheit() { return this.#celsius * 9/5 + 32; }         // computed, read like a property
  set fahrenheit(f) { this.celsius = (f - 32) * 5/9; }          // delegates to the celsius setter (validation)
}
const t = new Temperature();
t.celsius = 25;         // setter runs (validates)
t.fahrenheit;           // 77   — computed getter
t.fahrenheit = 212;     // setter → celsius = 100

Getters and setters (Objects lesson, now in class form) let a property be backed by logic while accessed like a plain field — ideal for computed/derived values (fahrenheit) and validation on assignment. Note how the fahrenheit setter delegates to the celsius setter so validation lives in one place — a small but real design win. Getters/setters are defined on the prototype like methods.

Part 5 — Inheritance: extends and super

class Animal {
  constructor(name) { this.name = name; }
  speak() { return `${this.name} makes a sound`; }
  describe() { return `I am ${this.name}`; }
}
class Dog extends Animal {
  constructor(name, breed) {
    super(name);              // MUST call super() before using `this` in a derived constructor
    this.breed = breed;
  }
  speak() {                   // OVERRIDE
    return `${super.speak()} — specifically, a bark`;   // super.method() calls the parent version
  }
}
const d = new Dog("Rex", "Lab");
d.speak();     // "Rex makes a sound — specifically, a bark"
d.describe();  // "I am Rex" — inherited, not overridden
d instanceof Dog;     // true
d instanceof Animal;  // true

extends sets up the whole prototype chain automatically (the three-step ES5 wiring from the Prototypes lesson, plus static inheritance). Two super rules: in a derived constructor, you must call super(...) (which runs the parent constructor) before accessing this — reading this first throws, because in the class model the parent constructor is what brings this into existence. In a method, super.method() invokes the parent's version of that method with the current this — the clean way to extend rather than fully replace behaviour. This is real subtype polymorphism: d.speak() dispatches to Dog's override, while d.describe() falls through to Animal's.

Part 6 — The this-Loss Problem in Classes

class Timer {
  seconds = 0;
  tick() { this.seconds++; }                 // normal method — on the prototype
  tickBound = () => { this.seconds++; };      // class-field arrow — auto-bound to the instance
}
const t = new Timer();
const loose = t.tick;
// loose();                 // 💥 lost `this` (this-binding lesson) — undefined.seconds++
setInterval(t.tick, 1000);  // 💥 same bug — called as a plain function
setInterval(t.tickBound, 1000);  // ✅ works — arrow field captured the instance
setInterval(() => t.tick(), 1000);  // ✅ also works — called as a method via the wrapper

Classes don't exempt methods from the this rules: a method detached from its instance (passed as a callback to setInterval, addEventListener, map) loses its this exactly like any function. Two fixes: a class-field arrow (tickBound = () => {}), which auto-binds to the instance at construction (at the cost of one function per instance, not shared on the prototype), or wrapping the call in an arrow at the point of passing. This is one of the most common real-world class bugs — especially in event handlers — so know both the cause and the fixes cold.

Part 7 — When to Use a Class (and When Not To)

Reach for a class when…Prefer a plain object / function when…
You'll create many instances of the same shapeYou need one config/record/singleton
You need inheritance / polymorphism (a type hierarchy)Composition (mixing behaviours) fits better than an is-a hierarchy
Encapsulated state + behaviour that belong togetherPure data with no behaviour — use a plain object
An external API/framework expects a class (custom elements, some libs)Pure functions do the transformation cleanly

Classes are a tool, not a default. They shine for many instances of one shape (shared prototype methods = memory-efficient) and for genuine type hierarchies with polymorphism. But JavaScript is happy with plain objects for data, functions for behaviour, and closures for encapsulation — and the community broadly favours composition over deep inheritance (a well-known trap: rigid, brittle class hierarchies where a change high up ripples everywhere). Rule of thumb: model with data + functions first; introduce a class when you have repeated instances with shared behaviour, or a real is-a relationship — not merely to "organise code." Deep inheritance chains (3+ levels) are usually a smell; prefer shallow hierarchies or composition/mixins.

Part 8 — Build It: an Observable Store as a Class

Let's rebuild the Prototypes lesson's Store as a class with private state — same runtime structure, dramatically cleaner source, plus true encapsulation of the internals.

class Store {
  #state;
  #listeners = new Set();      // private — outside code can't tamper with the subscriber list

  constructor(initial) { this.#state = initial; }

  get state() { return this.#state; }               // read-only public view

  set(next) {
    this.#state = typeof next === "function" ? next(this.#state) : next;  // value or updater fn
    for (const fn of this.#listeners) fn(this.#state);
  }
  subscribe(fn) {
    this.#listeners.add(fn);
    return () => this.#listeners.delete(fn);          // unsubscribe closure
  }
  static from(obj) { return new Store(obj); }        // static factory — nicer than `new`
}

const store = Store.from({ count: 0 });
const off = store.subscribe(s => console.log("changed:", s));
store.set(s => ({ count: s.count + 1 }));   // logs "changed: { count: 1 }"
store.state;    // { count: 1 }
off();          // unsubscribe
// store.#state;  // SyntaxError — internals genuinely hidden

Everything from this lesson is here: private fields (#state, #listeners) give real encapsulation the prototype version couldn't; a getter exposes a read-only view; a static factory (Store.from) reads better than new at call sites; and set accepts either a value or an updater function (the React setState pattern) using this-based dispatch. The runtime shape is identical to the prototype version — methods on the prototype, data on the instance — but the source is clearer and the internals are truly private. That's the value classes add: not new capability, but better ergonomics and hard privacy over the prototype mechanics you already understand.

Interview Questions

Q1 (beginner): Are JavaScript classes "real" classes or something else?

They're syntactic sugar over the existing prototype-based system, not a separate class model like Java's. A class is fundamentally a constructor function: typeof MyClass is "function", methods declared in the body are placed on MyClass.prototype (shared by instances via the prototype chain), and extends wires up the prototype chain. Classes do add real features on top — automatic strict mode, non-hoisting, a required new, true #private fields, and stricter super/this semantics — but underneath it's the same prototypes you'd set up by hand.

Q2 (beginner): What's the difference between an instance method and a static method?

An instance method lives on the class's prototype and is called on instances (obj.method()), operating on that instance's data via this. A static method (marked static) lives on the class itself and is called on the class (ClassName.method()), not on instances — it's for functionality related to the type but not tied to a single instance: factories, utilities, constants, counters. Built-in examples of statics: Array.from, Object.keys, Math.max. Calling a static on an instance (or an instance method on the class) fails because they live in different places.

Q3 (mid): What are #private fields and how do they differ from the _ convention?

#private fields (and methods, and static privates) are genuinely private — they're accessible only inside the class body, and referencing obj.#field from outside is a SyntaxError at parse time, not a runtime error you can work around. The name including the # is part of the field's identity, and there's no dynamic/computed access to it. The old _balance convention was merely a social signal ("please don't touch this"); the property was fully public and readable/writable by anyone. So # gives real encapsulation and invariant enforcement, whereas _ gave none — it's the class-based equivalent of closure-private state, but with methods shared on the prototype.

Q4 (mid): Explain the rules around super in a derived class.

Two contexts. In a derived constructor, you must call super(...args) to run the parent constructor, and you must do so before accessing this — reading this before super() throws a ReferenceError, because in the class model the parent constructor is responsible for creating and initialising the instance that this refers to. If a derived class has no constructor, a default one calling super(...args) is supplied. In a method, super.method() calls the parent class's version of that method with the current this — used to extend (rather than fully replace) inherited behaviour, e.g. an override that does extra work and then delegates the base case up. super resolves via the method's home object's prototype, so it correctly targets the parent even through multiple inheritance levels.

Q5 (senior): When would you choose composition over class inheritance? Give a concrete refactor.

Prefer composition when behaviours are orthogonal and would otherwise force an artificial or combinatorial hierarchy — the classic "gorilla/banana" problem where inheriting one behaviour drags in a whole base class you don't want, and where multiple independent capabilities can't be modelled by a single is-a chain. Concrete example: a game has entities that may be movable, drawable, and damageable in any combination. An inheritance approach forces an unnatural order (Entity → MovableEntity → DrawableMovableEntity → ...) that explodes combinatorially and can't express "drawable but not movable" cleanly. The composition refactor models each capability as a small standalone piece and assembles entities from them — e.g. factory functions returning objects that mix in the behaviours they need ({ ...movable(state), ...drawable(state) }), or capability objects held as fields (this.movement = new Movement()) and delegated to. This gives per-entity flexibility (a tree is drawable+damageable but not movable), avoids fragile base classes (a change to Entity can't ripple through unrelated subclasses), and keeps each behaviour independently testable. The senior heuristic: use inheritance only for genuine, stable is-a relationships with shared identity and shallow depth; reach for composition/mixins/dependency injection when you're combining independent capabilities, when the hierarchy would exceed ~2 levels, or when you find yourself inheriting to reuse code rather than to model a real subtype. "Favour composition over inheritance" is a default precisely because inheritance couples subclasses tightly to their ancestors, and that coupling is the source of most large-hierarchy maintenance pain.

Q6 (senior): A class method is used as an event handler and loses this. Compare the fixes (class-field arrow, bind in constructor, arrow wrapper) on correctness, memory, and testability.

All three fix the lost-this by ensuring the handler runs with the instance as this. Class-field arrow (handle = () => {...}): auto-binds at construction, so the reference is stable and can be passed directly to addEventListener and removed later by the same reference. Cost: the method is an instance property (one function object per instance) rather than a shared prototype method, so it uses more memory with many instances and isn't on the prototype (can't be overridden via super, doesn't show as a prototype method — a minor testability/introspection quirk). Bind in constructor (this.handle = this.handle.bind(this)): also produces one stable bound function per instance (same memory cost), keeps the original method on the prototype (so it's overridable and unit-testable via the prototype), but is more boilerplate and easy to forget for new methods. Arrow wrapper at the call site (el.addEventListener("click", () => this.handle())): keeps handle a clean shared prototype method (best for memory and testability), but creates a new wrapper function each time you write it — which is a real problem for removeEventListener, since you no longer hold the same reference to remove, risking a listener leak; you'd have to store the wrapper. Recommendation: for handlers you must add and remove by reference, a class-field arrow (or a constructor-bound method stored on the instance) is safest because the reference is stable; for fire-and-forget handlers or where you keep the wrapper reference, the arrow wrapper preserves the cleanest prototype method. The deciding axes are: do you need a stable reference for removal (favour arrow field/bind), and do you have enough instances that per-instance function copies matter (favour keeping the prototype method + wrapper). Testability slightly favours keeping methods on the prototype (bind/wrapper) over instance-field arrows.

🏋️ Exercises

1 (easy): Basic class. Write a Rectangle class with width/height, an area() method, and a static square(size) factory returning a square rectangle.
Solution 
class Rectangle {
  constructor(width, height) { this.width = width; this.height = height; }
  area() { return this.width * this.height; }
  static square(size) { return new Rectangle(size, size); }
}
new Rectangle(3, 4).area();      // 12
Rectangle.square(5).area();       // 25

area is an instance method (on the prototype); square is a static factory on the class, a common cleaner alternative to a multi-purpose constructor.

2 (easy): Private counter. Write a Counter class with a private #count, an increment() method, and a read-only value getter. Prove #count can't be read from outside.
Solution 
class Counter {
  #count = 0;
  increment() { return ++this.#count; }
  get value() { return this.#count; }
}
const c = new Counter();
c.increment(); c.increment();
c.value;            // 2
// c.#count;        // SyntaxError — genuinely private

The #count field is inaccessible outside the class body; the only way in is through the method and getter, enforcing controlled access.

3 (medium): Inheritance + super. Create Shape with a describe() returning its name, and Circle extends Shape that overrides describe() to add the radius, calling super.describe().
Solution 
class Shape {
  constructor(name) { this.name = name; }
  describe() { return `A ${this.name}`; }
}
class Circle extends Shape {
  constructor(radius) { super("circle"); this.radius = radius; }
  describe() { return `${super.describe()} of radius ${this.radius}`; }
}
new Circle(5).describe();   // "A circle of radius 5"

super("circle") runs the parent constructor before this is used; super.describe() calls the parent method and the override extends its result.

4 (medium): Fix the handler. This class loses this when its method is used as a click handler. Fix it with a class-field arrow and explain the trade-off. 
class Button {
  clicks = 0;
  handleClick() { this.clicks++; }
}
someEl.addEventListener("click", new Button().handleClick);
Solution 
class Button {
  clicks = 0;
  handleClick = () => { this.clicks++; };   // arrow field: auto-bound to the instance
}
const b = new Button();
someEl.addEventListener("click", b.handleClick);   // stable reference, correct this

The arrow class-field captures the instance's this at construction, so it works even when detached, and its reference is stable (so you can removeEventListener it later). Trade-off: it's an instance property (one function per instance), not a shared prototype method — fine for components, costlier if you have thousands of instances.

5 (hard): Validated value class. Build an Email value object whose constructor validates the format (throwing on invalid), stores it privately, exposes it read-only, and implements equals(other) and toString().
Solution 
class Email {
  #value;
  constructor(value) {
    if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value)) throw new TypeError(`Invalid email: ${value}`);
    this.#value = value.toLowerCase();
  }
  get value() { return this.#value; }
  equals(other) { return other instanceof Email && other.value === this.#value; }
  toString() { return this.#value; }
}
const a = new Email("Ada@X.com"), b = new Email("ada@x.com");
a.equals(b);        // true — normalised to lowercase
`${a}`;             // "ada@x.com" — toString used in template
// new Email("bad") // throws TypeError

A "value object": validated at construction (invalid values can't exist), immutable (private field, read-only getter), with value-based equals and a toString for string contexts. Normalising to lowercase in the constructor makes equality reliable. This is a clean use of classes — encapsulated, invariant-enforcing data + behaviour.

6 (hard): Mixin composition. Instead of inheritance, compose two capabilities — serializable (a toJSON) and timestamped (a createdAt) — into a class using a mixin function. Show why this is more flexible than a base class.
Solution 
const Serializable = (Base) => class extends Base {
  toJSON() { return JSON.stringify({ ...this }); }
};
const Timestamped = (Base) => class extends Base {
  createdAt = Date.now();
};

class Model {}                                  // plain base
class User extends Serializable(Timestamped(Model)) {
  constructor(name) { super(); this.name = name; }
}
const u = new User("Ada");
typeof u.createdAt;   // "number" — from Timestamped
u.toJSON();           // '{"createdAt":...,"name":"Ada"}' — from Serializable

Mixins are functions that take a base class and return an extended one, so you compose capabilities in any combination (Serializable(Timestamped(Model))) instead of forcing a single inheritance line. A different class can pick just Serializable, or a different order — flexibility a fixed base-class hierarchy can't offer. This is the composition- over-inheritance principle realised: independent behaviours combined à la carte rather than baked into a rigid ancestry.

🎯 Key Takeaways

  • Classes are prototype sugar: the constructor sets instance data, methods go on the prototype (shared), plus strict mode, non-hoisting, and required new.
  • Instance members live on objects; static members (fields/methods) live on the class — for factories, constants, and type-level utilities.
  • #private fields/methods are true hard privacy (SyntaxError from outside) — replacing the toothless _ convention.
  • extends wires the chain; in a derived constructor call super() before this; super.method() invokes the parent's version.
  • Class methods still obey the this rules — detaching one loses this; fix with a class-field arrow or an arrow wrapper.
  • Use classes for many-instances-of-one-shape and real type hierarchies; favour composition/mixins over deep inheritance, and plain objects/functions for data and transforms.
