Event Loop & Microtasks Advanced

The single most reliable senior-JS interview question is an ordering puzzle — and every one of them is answerable from one model: one thread, two queues, a loop that never sleeps, and rendering that squeezes in between. This lesson builds that model, drills it on puzzles, then applies it to the production skill: keeping the main thread responsive.

The Complete Model

┌─────────────┐  registers callbacks   ┌──────────────────────────────┐
│  Web APIs   │ ─────────────────────▶ │  MACROTASK (task) queue      │
│ setTimeout  │                        │  timers · UI events · I/O ·  │
│ fetch, click│                        │  postMessage · script itself │
└─────────────┘                        └──────────────────────────────┘
                                       ┌──────────────────────────────┐
                                       │  MICROTASK queue             │
                                       │  promise .then / await ·     │
                                       │  queueMicrotask · Observers  │
                                       └──────────────────────────────┘
THE LOOP, forever:
  1. Take ONE macrotask, run it to completion (nothing preempts a running frame)
  2. Drain the ENTIRE microtask queue — including microtasks queued WHILE draining
  3. If it's time (~60fps) → rendering: rAF callbacks → style → layout → paint
  4. Go to 1.

Three sentences that answer every puzzle:
· Running code finishes before ANY callback runs.
· Microtasks always beat waiting macrotasks — and drain completely.
· Rendering happens between macrotasks, never between microtasks.

The Puzzles, Graded

// LEVEL 1 — the classic:
console.log("1");
setTimeout(() => console.log("2"));               // macrotask
Promise.resolve().then(() => console.log("3"));   // microtask
console.log("4");
// → 1 4 3 2   (script runs; drain micro: 3; next macro: 2)

// LEVEL 2 — await is a suspension point:
async function f() { console.log("a"); await null; console.log("b"); }
f(); console.log("c");
// → a c b     (await ALWAYS yields, even on resolved values — 'b' is a microtask)

// LEVEL 3 — interleaved:
setTimeout(() => console.log("t1"));
Promise.resolve().then(() => { console.log("p1");
  Promise.resolve().then(() => console.log("p2")); });   // queued DURING draining — still drains
setTimeout(() => console.log("t2"));
// → p1 p2 t1 t2   (the whole microtask cascade beats the FIRST timer)

// LEVEL 4 — the Node curveball:
process.nextTick(() => console.log("tick"));       // Node's nextTick queue runs
Promise.resolve().then(() => console.log("micro")); //   even BEFORE promise microtasks
// → tick micro   (browser has no nextTick; know the difference exists)

Keeping the Main Thread Responsive — the Three Tools

// 1. CHUNK long work — cooperative yielding between batches:
async function processAll(items) {
  const CHUNK = 500;
  for (let i = 0; i < items.length; i += CHUNK) {
    items.slice(i, i + CHUNK).forEach(process);
    await scheduler?.yield?.() ?? new Promise(r => setTimeout(r));   // let input/paint through
  }
}
// scheduler.yield()/postTask: the modern Scheduler API — prioritized, resumes ahead of
// other work (where supported); the setTimeout fallback works everywhere.

// 2. SHOW-THEN-WORK — paint the spinner before the crunch:
spinner.hidden = false;
requestAnimationFrame(() => requestAnimationFrame(() => crunch()));
// double-rAF = "after the NEXT paint" — the reliable paint-first idiom

// 3. MOVE it — CPU-bound work belongs off-thread entirely (Web Workers lesson):
// chunking = responsive but still slow overall; worker = responsive AND main thread free.

Debugging the Loop

DevTools → Performance → record the frozen interaction:
· Long yellow "Task" blocks = your macrotask hogging the thread (click one → bottom-up = the culprit fn)
· "Long tasks" get red corner flags at >50ms
· Console trick: monitor responsiveness cheaply —
  let last = performance.now();
  setInterval(() => { const d = performance.now() - last - 100;
    if (d > 50) console.warn(`main thread blocked ~${d|0}ms`); last = performance.now(); }, 100);

Interview Questions

Q1: Solve and EXPLAIN: async function a(){ console.log(1); await b(); console.log(4) } ; async function b(){ console.log(2) } ; a(); console.log(3);

1 2 3 4. a runs sync to the await: logs 1; b() runs synchronously to completion (logs 2) returning a resolved promise; await suspends a — the rest becomes a microtask; control returns to the caller, logs 3; script ends → drain microtasks → 4. Key sentence: an async function runs synchronously until its first await.

Q2: Why do microtasks exist as a separate queue at all?

Promise semantics need "as soon as possible, but never reentrantly, and before anything else observes intermediate state" — .then callbacks must run after the current stack unwinds (consistent async timing whether the promise was already settled or not) yet before unrelated events/rendering interleave. A same-tick, drain-completely queue delivers exactly that; macrotasks alone would let events interleave between chained .thens.

Q3: A dashboard freezes for 3s parsing a 30MB JSON. Give the escalation ladder.

① Don't parse it all: paginate/stream from the server (best fix is upstream). ② If it must land: parse in a Worker (JSON.parse is CPU-bound; postMessage the result — or transfer an ArrayBuffer). ③ If stuck on main: chunk custom parsing with yields — responsive but slower overall. Mentioning that response.json() itself is off-main in some engines but the resulting object cloning isn't free = bonus depth.

Q4: How does the browser decide when to render, and how can JS starve it?

Rendering is scheduled between macrotasks at the display's refresh cadence (~16.7ms at 60Hz): rAF callbacks → style recalc → layout → paint → composite. It CANNOT interrupt a running task — so one long task drops frames, and a self-refilling microtask queue (step 2 never ends) blocks rendering indefinitely. That's the mechanical reason "break up long tasks" is THE INP fix.

🎯 Key Takeaways

  • One macrotask → drain ALL microtasks → maybe render → repeat. Three sentences answer every puzzle.
  • An async function runs synchronously until its first await; everything after is a microtask.
  • Microtasks beat timers and drain to exhaustion — self-queuing chains starve paint forever.
  • setTimeout 0 = "next task, maybe": clamped, throttled, unordered under load — never a correctness tool.
  • >50ms tasks hurt INP: chunk + scheduler.yield, double-rAF to paint first, Workers for real compute.
  • Node adds phases + process.nextTick (ahead of microtasks) — the standard follow-up.
