The Event Loop Asynchronous JavaScript

What you'll master: how single-threaded JavaScript does non-blocking I/O — the call stack, Web APIs, the task and microtask queues, and the loop that ties them together — plus the precise ordering rules that let you predict the output of any mixed setTimeout/promise/ synchronous snippet (a top interview exercise). Prerequisites: Functions and Closures (callbacks are closures). This model is the foundation for Promises, async/await, and every "why did this log in that order?" question.

Part 1 — One Thread, One Call Stack

function a() { b(); }
function b() { c(); }
function c() { console.log("done"); }
a();
// Call stack grows:  a → a,b → a,b,c → logs → unwinds c,b,a → empty

JavaScript runs on a single thread with one call stack. When you call a function it's pushed onto the stack; when it returns it's popped. Only one thing executes at a time — there is no parallel JS execution in a single thread. This raises the obvious question: if it's single-threaded and synchronous, how does it handle a network request or a timer without freezing the whole page for seconds? The answer is that the async work doesn't happen in JavaScript — it happens in the host environment (the browser or Node), and the event loop feeds the results back to the single thread when it's free. That's the whole trick.

⚠️ Blocking the thread freezes everything. A long synchronous loop, a huge JSON.parse, or while(true){} holds the single stack, so no clicks render, no timers fire, nothing repaints until it finishes. "Don't block the event loop" is the golden rule — offload heavy work to Web Workers or chunk it.

Part 2 — The Players: Stack, Web APIs, Queues, Loop

console.log("1");
setTimeout(() => console.log("2"), 0);   // handed to a Web API (timer), callback queued LATER
console.log("3");
// Output: 1, 3, 2  — even with 0ms delay!

Four components work together:

  • Call stack — where synchronous code runs, one frame at a time.
  • Web APIs / host APIs — timers, fetch, DOM events, etc. These run outside the JS thread (in browser/Node C++). setTimeout hands its callback to the timer API and returns immediately.
  • Task queue (macrotask queue) — when a Web API finishes (timer elapses, response arrives), it puts the callback here to wait its turn.
  • Event loop — the coordinator: "when the call stack is empty, take the next task from the queue and push it onto the stack."

So setTimeout(fn, 0) logs after the synchronous code: fn is queued, and the loop only runs it once the stack is empty (after "1" and "3"). The 0 is a minimum delay before queuing, not a guarantee of when it runs — it runs when the stack clears and its turn comes.

Part 3 — Microtasks vs Macrotasks (the Critical Distinction)

console.log("start");
setTimeout(() => console.log("timeout"), 0);        // MACROtask
Promise.resolve().then(() => console.log("promise")); // MICROtask
console.log("end");
// Output: start, end, promise, timeout   ← promise BEFORE timeout, always

There are two queues, and their priority is the single most important event-loop fact:

  • Macrotask queue: setTimeout, setInterval, I/O, UI events, setImmediate (Node).
  • Microtask queue: promise callbacks (.then/.catch/ .finally), await continuations, queueMicrotask, MutationObserver.

After each macrotask (and after the initial synchronous script), the loop drains the ENTIRE microtask queue before taking the next macrotask. So every pending promise callback runs before the next setTimeout, no matter how the delays compare. That's why "promise" prints before "timeout" above even though both were "immediate." Microtasks have priority; macrotasks wait. This rule alone explains most tricky ordering questions.

Part 4 — The Full Loop, Step by Step

1. Run all synchronous code (the current macrotask) to completion — stack empties.
2. Drain the microtask queue COMPLETELY (running microtasks can queue MORE microtasks — all run now).
3. Render (browser may repaint here, between macrotasks).
4. Take ONE macrotask from the task queue, push it to the stack, run it (step 1 for it).
5. Go to step 2. Repeat forever.

The loop is: one macrotask, then all microtasks, (maybe render), one macrotask, all microtasks… Two consequences worth internalising. First, a flood of microtasks can starve rendering and macrotasks: if a microtask keeps queuing new microtasks, step 2 never finishes, and the page freezes (recursive Promise.resolve().then chains can do this). Second, rendering happens between macrotasks, after microtasks — which is why a synchronous DOM update loop shows only the final state (no intermediate paints), and why you use requestAnimationFrame to schedule work right before a paint.

Part 5 — Predicting Output: the Classic Puzzle

console.log("A");
setTimeout(() => console.log("B"), 0);
Promise.resolve().then(() => {
  console.log("C");
  setTimeout(() => console.log("D"), 0);
});
Promise.resolve().then(() => console.log("E"));
console.log("F");
// Output: A, F, C, E, B, D

Trace it with the rules: (1) synchronous first — A, then F (the setTimeout and .thens are only scheduled, not run). (2) Stack empty → drain microtasks: C runs (and schedules a new macrotask D), then E runs. Microtask queue now empty. (3) Take one macrotask: B. (4) Drain microtasks (none). (5) Take next macrotask: D. Result: A, F, C, E, B, D. The method never changes: run sync, drain all microtasks, then one macrotask at a time, draining microtasks after each. Practice a few of these and the "predict the order" interview question becomes mechanical.

Part 6 — await Is Microtask Scheduling in Disguise

async function demo() {
  console.log("1");
  await null;                 // everything AFTER await becomes a MICROTASK continuation
  console.log("3");
}
console.log("0");
demo();
console.log("2");
// Output: 0, 1, 2, 3   — "3" runs as a microtask, after the synchronous "2"

An async function runs synchronously up to the first await; at that point it suspends and schedules the rest of the function as a microtask, returning control to the caller. So the code after await runs after the current synchronous code finishes, via the microtask queue — which is why "2" (synchronous, after demo() returned at the await) prints before "3" (the awaited continuation). This is the whole connection between the async/await lesson and the event loop: await doesn't block the thread, it yields it and resumes later as a microtask. Understanding this makes async ordering predictable.

Part 7 — Practical Consequences

// 1. queueMicrotask — defer work to after the current sync code but before any macrotask:
queueMicrotask(() => console.log("runs after sync, before timers"));

// 2. Yield to the event loop so the UI can paint / not block:
async function processHugeArray(items) {
  for (let i = 0; i < items.length; i++) {
    doWork(items[i]);
    if (i % 1000 === 0) await new Promise(r => setTimeout(r, 0)); // yield: let a macrotask/paint happen
  }
}

// 3. requestAnimationFrame — run right before the next paint (for animations):
requestAnimationFrame(() => updateAnimation());   // NOT a normal macro/microtask — paint-synced

// 4. Node's process.nextTick jumps ahead of even promise microtasks (Node-specific priority).

These are the everyday tools that fall out of the model. queueMicrotask defers work to just after the current synchronous run (before timers) — useful for batching. Awaiting a setTimeout(0) inside a heavy loop yields the thread so the browser can process input and paint, keeping the UI responsive during big computations (a poor-man's chunking; Workers are better for truly heavy work). requestAnimationFrame schedules a callback right before the next repaint — the correct place for visual updates, since it's synced to the display refresh rather than an arbitrary timer.

Part 8 — Build It: a Cooperative Task Scheduler

Let's use the event loop deliberately to run many tasks without blocking, yielding between batches so the UI stays responsive — a real pattern for processing large workloads in the browser.

async function runCooperatively(tasks, { batchSize = 50 } = {}) {
  const results = [];
  for (let i = 0; i < tasks.length; i++) {
    results.push(tasks[i]());                 // run one task synchronously
    if ((i + 1) % batchSize === 0) {
      // yield to the event loop: let queued macrotasks (input, paint) run before continuing
      await new Promise(resolve => setTimeout(resolve, 0));
    }
  }
  return results;
}

// 10,000 small tasks, but the page stays responsive because we yield every 50:
const tasks = Array.from({ length: 10000 }, (_, i) => () => i * i);
runCooperatively(tasks).then(r => console.log("done", r.length));

The scheduler processes tasks in synchronous batches, then awaits a setTimeout(0) to yield the thread back to the event loop — giving the browser a chance to handle clicks, run other timers, and repaint between batches, so a heavy workload doesn't freeze the page (Part 1's golden rule). The await here is pure event-loop scheduling (Part 6): it suspends the loop as a microtask/macrotask boundary and resumes after the browser has had its turn. This is exactly how libraries keep large operations non-blocking without Workers — and it only makes sense once you understand the loop.

Interview Questions

Q1 (beginner): JavaScript is single-threaded — how does it do async work without blocking?

The asynchronous work itself doesn't run in JavaScript — it runs in the host environment (browser Web APIs or Node's libuv), outside the single JS thread. When you call setTimeout or fetch, JS hands the operation to the host and returns immediately, keeping the thread free. When the operation completes, the host places its callback in a queue. The event loop, when the call stack is empty, takes callbacks from the queue and runs them on the single thread. So JS stays single-threaded and non-blocking by delegating the waiting to the environment and processing results one at a time when free.

Q2 (beginner): Why does setTimeout(fn, 0) not run immediately?

Because it schedules fn as a macrotask rather than calling it. The 0 is a minimum delay before the callback is placed in the task queue, not a promise to run now. The event loop only runs queued tasks when the call stack is empty — i.e. after all currently-executing synchronous code finishes (and after all microtasks). So fn runs at the earliest opportunity after the current synchronous execution completes, which is why console.logs around it print first.

Q3 (mid): Explain the difference between microtasks and macrotasks and their ordering.

Macrotasks include setTimeout/setInterval callbacks, I/O, and UI events; microtasks include promise reactions (.then/.catch/.finally), await continuations, and queueMicrotask. The key rule: after the current synchronous execution (or the current macrotask) finishes, the event loop drains the entire microtask queue before taking the next macrotask — and microtasks queued during that draining also run in the same drain. So all pending promise callbacks run before the next timer, regardless of delays. That's why Promise.resolve().then(...) always logs before a sibling setTimeout(..., 0). Practically, microtasks have priority and macrotasks wait their turn one at a time.

Q4 (mid): What does await do to the event loop?

An async function executes synchronously until it hits an await. At that point it evaluates the awaited expression, then suspends the function and schedules the remainder of its body as a microtask (the continuation), returning control to the caller. When the awaited promise settles, the continuation is queued as a microtask and runs when the microtask queue is next drained. So await does not block the thread — it yields it and resumes later — and code after an await runs after the current synchronous code and in microtask order, which is why it interleaves with other promise callbacks but always before pending timers.

Q5 (senior): Trace the exact output of a mixed snippet and explain how microtasks queued inside microtasks are handled.

Consider: console.log(1); setTimeout(()=>console.log(2)); Promise.resolve().then(()=>{ console.log(3); Promise.resolve().then(()=>console.log(4)); }); console.log(5);. Trace: synchronous run logs 1, schedules the timer (macrotask) and the .then (microtask), logs 5. Stack empties → drain microtasks: the first microtask logs 3 and, critically, queues a new microtask (log 4) — and because draining continues until the microtask queue is completely empty, that newly-queued microtask runs in the same drain, logging 4, before any macrotask. Microtask queue now empty → take the macrotask: log 2. Output: 1, 5, 3, 4, 2. The senior point is step 2's "drain completely" semantics: microtasks scheduled while draining are appended and also run before control returns to macrotasks, which is both powerful (guaranteed to finish all promise work before the next timer) and dangerous — an unbounded chain of self-queuing microtasks never yields, starving rendering and macrotasks and freezing the page. That starvation risk is why breaking long async chains with a macrotask (setTimeout) is sometimes necessary.

Q6 (senior): You have a CPU-heavy computation freezing the UI. Walk through the options, from yielding to Workers, with trade-offs.

The problem is that heavy synchronous work holds the single thread, so no input, timers, or paints occur until it finishes. Options, in increasing capability: (1) Chunk + yield on the main thread — split the work into batches and await a setTimeout(0) (or scheduler.yield() where available) between batches, letting the event loop process input and repaint. Simple, no new files, keeps the UI responsive, but the total work still competes for the main thread so it's slower overall and can still jank if batches are too big; good for moderate work. (2) requestIdleCallback — run chunks during idle periods so you don't steal time from higher-priority work; better cooperative scheduling, but idle time may be scarce under load, so throughput is unpredictable. (3) Web Worker — move the computation to a separate thread entirely (Workers lesson), so the main thread stays fully free for UI; the computation runs at full speed in parallel. Trade-offs: data crosses the boundary via structured-clone (copying cost) or transferable objects (zero-copy for ArrayBuffers), workers can't touch the DOM, and there's setup/messaging complexity — but for genuinely heavy, long-running CPU work this is the correct answer. (4) WASM in a Worker or offloading to the server for extreme workloads. The decision: for short bursts, chunk-and-yield; for sustained heavy computation that must not affect the UI, a Web Worker; combine with transferable objects if large data is involved. The unifying principle from the event-loop model is "never hold the main thread" — either give it back periodically (yielding) or don't take it at all (workers).

🏋️ Exercises

1 (easy): Predict the order. What does this log? 
console.log("a");
setTimeout(() => console.log("b"), 0);
console.log("c");
Answer

a, c, b. The synchronous console.logs run first (a, c); the setTimeout callback is a macrotask that runs only after the stack is empty, so b is last despite the 0ms delay.

2 (easy): Promise vs timeout. Predict: 
setTimeout(() => console.log("timeout"), 0);
Promise.resolve().then(() => console.log("promise"));
console.log("sync");
Answer

sync, promise, timeout. Synchronous first (sync); then the microtask queue drains (promise) before any macrotask; then the timer macrotask (timeout).

3 (medium): Full trace. Predict the output and justify each step: 
console.log(1);
setTimeout(() => console.log(2), 0);
Promise.resolve().then(() => console.log(3)).then(() => console.log(4));
console.log(5);
Answer

1, 5, 3, 4, 2. Sync: 1, then 5 (timer + then chain merely scheduled). Drain microtasks: 3 runs and its .then queues another microtask, which runs in the same drain → 4. Microtasks empty → macrotask → 2. Chained .thens each schedule a microtask, so they still complete before the timer.

4 (medium): async ordering. Predict: 
async function f() { console.log("A"); await null; console.log("B"); }
console.log("start"); f(); console.log("end");
Answer

start, A, end, B. f() runs synchronously to the await, logging A, then suspends and schedules the rest as a microtask, returning control. end logs (synchronous). Then the microtask continuation runs → B.

5 (hard): Non-blocking sum. Sum an array of 1,000,000 numbers without freezing the UI, yielding to the event loop periodically. Show the yielding mechanism.
Solution 
async function sumNonBlocking(nums, chunk = 50000) {
  let total = 0;
  for (let i = 0; i < nums.length; i++) {
    total += nums[i];
    if ((i + 1) % chunk === 0) await new Promise(r => setTimeout(r)); // yield each chunk
  }
  return total;
}
sumNonBlocking(new Array(1_000_000).fill(1)).then(t => console.log(t)); // 1000000

Processing in chunks and awaiting a setTimeout between them hands the thread back to the event loop, so input and paints happen during the computation. For truly heavy math, a Web Worker is better (no main-thread cost at all), but this keeps the UI alive with no new file.

6 (hard): Microtask starvation. Write a snippet that starves the macrotask queue (a setTimeout that never runs) using recursive microtasks, then explain how to fix it.
Solution 
setTimeout(() => console.log("I never run (for a long time)"), 0);
function flood() { Promise.resolve().then(flood); }  // each microtask queues another microtask
flood();
// The microtask queue never empties, so step 4 (take a macrotask) is never reached → timeout starved.

// Fix: break the chain with a MACROtask so the loop can process the timer:
function polite() { setTimeout(polite, 0); }  // now each iteration is a macrotask — timers interleave

Because the loop fully drains microtasks before any macrotask, an endlessly self-queuing microtask chain prevents the timer (a macrotask) from ever running and freezes rendering. Switching the recursion to a macrotask (setTimeout) yields between iterations, letting other macrotasks and paints occur. This is the concrete danger behind "don't build unbounded microtask chains."

🎯 Key Takeaways

  • JavaScript is single-threaded with one call stack; async work runs in host Web APIs, and the event loop feeds results back when the stack is empty.
  • setTimeout(fn, 0) queues a macrotask — it runs after all synchronous code, not immediately; the delay is a minimum.
  • Two queues: microtasks (promises, await, queueMicrotask) drain completely after each macrotask and before the next one — microtasks always beat timers.
  • The loop: run sync → drain all microtasks → (render) → one macrotask → drain microtasks → repeat; predict any snippet by applying this order.
  • await suspends and schedules its continuation as a microtask — it yields the thread, it doesn't block it.
  • Never block the main thread: chunk-and-yield for moderate work, Web Workers for heavy CPU; unbounded microtask chains starve rendering.
