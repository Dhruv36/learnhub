Event Loop & Microtasks Advanced

The single most reliable senior-JS interview question: "what does this log?" It's answerable from one model — one thread, two queues, and a loop that never sleeps.

The Model

┌──────────────┐   push callbacks    ┌───────────────────────────┐
│ Web APIs     │ ──────────────────▶ │ MACROTASK queue           │
│ setTimeout   │                     │ (timers, clicks, I/O)     │
│ fetch, click │                     └───────────────────────────┘
└──────────────┘                     ┌───────────────────────────┐
                                     │ MICROTASK queue           │
                                     │ (promise .then, await,    │
                                     │  queueMicrotask)          │
                                     └───────────────────────────┘
EVENT LOOP, forever:
  1. Run ONE macrotask to completion (the call stack empties)
  2. Drain the ENTIRE microtask queue (including ones added while draining!)
  3. Maybe render (browser paints between macrotasks, ~60fps)
  4. Repeat

The Interview Question, Solved Slowly

console.log("1");
setTimeout(() => console.log("2"));               // macrotask
Promise.resolve().then(() => console.log("3"));   // microtask
console.log("4");

// Output: 1 4 3 2
// Script itself is the first macrotask → logs 1, 4
// Script ends → drain microtasks → 3
// Next macrotask → 2. Microtasks ALWAYS beat waiting macrotasks.
async function f() {
  console.log("a");
  await null;               // everything below = a .then callback (microtask)
  console.log("b");
}
f(); console.log("c");       // a c b — await yields the thread at that exact line

Consequences That Matter in Production

FactConsequence
One thread runs everythingA 200ms sync loop freezes clicks, scrolling, rendering — everything
Microtasks drain completelyA promise that queues itself recursively starves the page forever (worse than setTimeout loops, which let rendering breathe)
setTimeout(fn, 0) ≠ 0msIt means "next macrotask at the earliest" (~4ms clamped, later if busy) — never use timers for correctness ordering
Rendering happens between macrotasksTo paint before heavy work: requestAnimationFrame, or split work into chunks

Keeping the Main Thread Responsive

// Chunk long work so rendering interleaves:
async function processAll(items) {
  for (const [i, item] of items.entries()) {
    process(item);
    if (i % 500 === 0) await new Promise(r => setTimeout(r));   // yield to the loop
  }
}
// Better for CPU-heavy work: move it off-thread entirely → Web Workers lesson.
💡 Staff-level insight: Node's loop adds phases (timers → I/O → check/setImmediate) and process.nextTick, which jumps even ahead of microtasks — the classic Node interview follow-up. The browser rule that settles most questions: microtasks run to exhaustion after every callback, before anything else happens.
