Error Handling Patterns Production JavaScript

What you'll master: the Error object and its subtypes, custom error classes that carry structured context, the operational-vs-programmer-error distinction that should drive every catch block, global error handlers as a safety net, and the Result/Either pattern as an alternative to exceptions. Prerequisites: Classes (custom error subclassing) and Promises/async-await (async error flow). Good error handling is what separates hobby code from production code — this lesson is about treating failure as a first-class part of your design, not an afterthought.

Part 1 — The Error Object and Built-In Subtypes

const err = new Error("Something failed");
err.message;    // "Something failed"
err.name;       // "Error"
err.stack;      // multi-line string: message + call stack at CREATION time — invaluable for debugging

// Built-in subtypes the ENGINE throws for specific situations:
new TypeError("x is not a function");      // wrong type used
new RangeError("invalid array length");    // value outside allowed range
new ReferenceError("x is not defined");    // accessing an undeclared name
new SyntaxError("Unexpected token");       // (usually only at parse time, can't catch your own code's)

try { null.foo; } catch (e) { e instanceof TypeError; }   // true — the engine picked the right subtype

Every error is (or should be) an instance of Error, which gives you message, name, and critically stack — a trace of the call stack at the moment the error was constructed (not thrown), which is why you should always new Error(...) right where the problem is detected. The engine itself throws specific built-in subtypes (TypeError, RangeError, ReferenceError) so you can instanceof-check what kind of failure occurred. Always throw Error objects, never plain strings or values (throw "failed") — a thrown string has no stack trace, no instanceof hierarchy, and makes catch blocks that expect err.message break.

Part 2 — Custom Error Classes: Carrying Structured Context

class AppError extends Error {
  constructor(message, { code, statusCode = 500, cause } = {}) {
    super(message, { cause });          // native `cause` chains the original error (Part 6)
    this.name = this.constructor.name;   // "AppError", or the actual subclass name
    this.code = code;                    // machine-readable code, e.g. "USER_NOT_FOUND"
    this.statusCode = statusCode;
  }
}
class ValidationError extends AppError {
  constructor(message, fields) { super(message, { code: "VALIDATION_ERROR", statusCode: 400 }); this.fields = fields; }
}
class NotFoundError extends AppError {
  constructor(resource) { super(`${resource} not found`, { code: "NOT_FOUND", statusCode: 404 }); }
}

throw new ValidationError("Invalid signup data", { email: "must be a valid email" });
// catch: if (err instanceof ValidationError) render field errors; else generic failure UI

Plain Error only gives you a message string — real applications need structured, machine-readable context: an error code for programmatic branching (not string-matching the message, which is fragile and breaks on wording changes), a status code for HTTP APIs, and domain-specific fields (which validation failed, which resource was missing). Subclassing Error gives you this: each error type carries exactly the data its handlers need, and callers can instanceof-check to route different failures to different UI/logic — precisely the typed-error routing referenced in the Promises lesson's senior question, made concrete here.

Part 3 — Operational vs Programmer Errors: the Decision That Drives Everything

Operational errorsProgrammer errors
CauseExpected failure modes of a working systemActual bugs — code that's wrong
ExamplesNetwork timeout, invalid user input, file not found, rate limit hitCalling a function with the wrong type, a null reference, a typo
ResponseHandle — catch, show a message, retry, recoverDon't handle — let it crash loudly, fix the code
Recoverable at runtime?Yes — it's part of normal operationNo — the program is in an unknown/wrong state

This distinction, articulated clearly by Node.js's error-handling philosophy, should govern every catch block you write. Operational errors are expected, recoverable parts of a working system — a user submits invalid data, a network request times out, a file doesn't exist — and you should catch and handle them gracefully (show an error message, retry, fall back). Programmer errors are actual bugs — calling undefined.foo, passing the wrong type, a logic error — and catching them to "keep the app running" is dangerous: the program is now in a state the developer never intended, and silently continuing can corrupt data or mask the real problem. The professional stance: catch operational errors specifically; let programmer errors crash loudly (or be caught only by a top-level handler that logs and fails safely — Part 5). A blanket try { everything } catch (e) { /* ignore */ } conflates the two and is a serious anti-pattern.

Part 4 — Async Error Handling: Sync and Async Unified

async function loadUser(id) {
  try {
    if (!id) throw new ValidationError("id is required", { id: "missing" });  // SYNC throw
    const res = await fetch(`/api/users/${id}`);                              // ASYNC rejection
    if (!res.ok) throw new NotFoundError("User");
    return await res.json();
  } catch (err) {
    if (err instanceof ValidationError) return { error: "bad input", fields: err.fields };
    if (err instanceof NotFoundError) return { error: "not found" };
    throw err;    // an UNEXPECTED error (programmer error) — re-throw, don't swallow (Part 3)
  }
}

One of async/await's biggest wins (from the async/await lesson) is that a single try/catch uniformly catches both synchronous throws and asynchronous rejections — you don't need separate handling paths. The pattern above demonstrates the operational-vs-programmer split in code: known error types (ValidationError, NotFoundError) are caught and converted into a handled response; anything else is re-thrown rather than swallowed, because an unrecognized error might be a real bug that a blanket catch would otherwise hide. This "catch what you expect, re-throw what you don't" discipline is the concrete application of Part 3's principle.

Part 5 — Global Handlers: the Safety Net, Not the Strategy

// Browser — catches errors that escaped every local handler:
window.addEventListener("error", (event) => {
  logToService({ message: event.message, stack: event.error?.stack, url: event.filename });
});
window.addEventListener("unhandledrejection", (event) => {   // a promise rejected with NO .catch
  logToService({ message: event.reason?.message ?? String(event.reason) });
  event.preventDefault();   // suppress the default browser console warning, since we've logged it
});

// Node — process-level (crash-and-restart is often the SAFEST response to an unknown state):
process.on("uncaughtException", (err) => { logToService(err); process.exit(1); });   // then let a supervisor restart it
process.on("unhandledRejection", (reason) => { throw reason; });   // promote to uncaughtException, then exit

Global handlers are your last line of defense — they catch what fell through every local try/catch, log it so you know it happened, and in Node's case, often deliberately crash the process. This isn't giving up: per Part 3, an uncaught exception means the program hit a state nobody anticipated, and continuing to run in that unknown state risks corrupting data or serving wrong results silently — a clean crash-and-restart (under a process supervisor like PM2/Kubernetes) is the safer failure mode. In the browser you can't "restart," so you log and often show a fallback UI. The key mental model: global handlers are a monitoring/safety net, never your primary error-handling strategy — if errors are routinely reaching them, that's a signal your local handling has gaps, not a reason to rely on the global handler more.

Part 6 — Error Cause Chains: Preserving the Root Cause

async function getUserProfile(id) {
  try {
    return await fetchFromDb(id);
  } catch (dbErr) {
    // WRAP with context, but PRESERVE the original via `cause` — don't just lose it:
    throw new AppError(`Failed to load profile for user ${id}`, { code: "PROFILE_LOAD_FAILED", cause: dbErr });
  }
}

try { await getUserProfile(42); }
catch (err) {
  console.log(err.message);        // "Failed to load profile for user 42" — the HIGH-LEVEL context
  console.log(err.cause?.message); // the ORIGINAL low-level DB error — still available for debugging
}

When you catch a low-level error and re-throw a higher-level, more meaningful one (turning "connection ETIMEDOUT" into "Failed to load profile"), you gain context for the caller but risk losing the original diagnostic detail. The native cause option (new Error(msg, { cause: originalErr })) solves this: it chains the original error onto the new one, so logging/debugging tools can walk the chain to the root cause while callers still get a clean, contextual message. This replaces older hacky patterns (concatenating messages, custom .originalError properties) with a standard, tool-supported mechanism — Node and browser consoles automatically print the full cause chain for you.

Part 7 — An Alternative to Exceptions: the Result Pattern

// Exceptions make errors INVISIBLE in a function's signature — you must read the implementation
// (or docs) to know it can throw. The Result pattern makes failure an EXPLICIT return value:
async function tryFetchUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) return { ok: false, error: new NotFoundError("User") };
    return { ok: true, value: await res.json() };
  } catch (err) {
    return { ok: false, error: err };
  }
}

const result = await tryFetchUser(42);
if (!result.ok) { showError(result.error); return; }   // the caller is FORCED to handle both paths
console.log(result.value.name);   // TS narrows `value` to be defined only after the ok check

Exceptions have a real downside: a function's signature doesn't say it can throw, or what — you only find out by reading the implementation or being surprised in production. The Result (or "Either") pattern, common in Rust/Go and increasingly in TypeScript codebases, makes failure an explicit part of the return value: the function returns { ok: true, value } or { ok: false, error }, and the caller must check ok before accessing value (which TypeScript can enforce via type narrowing). This trades exceptions' convenience (automatic propagation up the call stack) for explicitness (you can't forget to handle the error case; it's a visible branch). It's not a wholesale replacement — genuinely exceptional, unexpected failures (programmer errors) still belong to throw/global handlers — but for expected operational failures (a lookup that might not find something, a parse that might fail), many teams increasingly prefer Result over exceptions for exactly this forcing-function property.

Part 8 — Build It: a Layered Error-Handling Strategy

Let's assemble custom errors, the operational/programmer split, cause chains, and a global safety net into one coherent strategy for a small API client layer — the shape of real production error handling.

// 1. Typed errors with context (Part 2):
class ApiError extends Error {
  constructor(message, { status, code, cause } = {}) {
    super(message, { cause });
    this.name = "ApiError"; this.status = status; this.code = code;
  }
}

// 2. A client that catches low-level failures and re-throws typed, contextual ones (Parts 4 & 6):
async function apiRequest(path) {
  let res;
  try {
    res = await fetch(`/api${path}`, { signal: AbortSignal.timeout(8000) });
  } catch (networkErr) {
    throw new ApiError(`Network failure calling ${path}`, { code: "NETWORK", cause: networkErr });
  }
  if (!res.ok) {
    throw new ApiError(`Request to ${path} failed`, { status: res.status, code: "HTTP_ERROR" });
  }
  return res.json();
}

// 3. Callers handle EXPECTED operational failures specifically; let anything else propagate (Part 3):
async function loadDashboard() {
  try {
    return await apiRequest("/dashboard");
  } catch (err) {
    if (err instanceof ApiError && err.status === 401) return redirectToLogin();
    if (err instanceof ApiError && err.code === "NETWORK") return showOfflineBanner();
    throw err;   // unrecognized — NOT swallowed; let it reach the global handler
  }
}

// 4. A global safety net catches anything that slipped through (Part 5):
window.addEventListener("unhandledrejection", (e) => {
  reportToSentry(e.reason);            // log with the full cause chain intact
  showGenericErrorToast();
  e.preventDefault();
});

This is a complete, layered strategy, not a single trick. Layer 1 defines typed errors carrying the context handlers need. Layer 2 is the boundary where raw failures (network exceptions, HTTP status) get converted into typed errors with cause chains preserved. Layer 3 is where business logic handles the operational failures it specifically expects (401 → redirect, network → offline banner) and re-throws everything else rather than guessing how to handle an unknown failure. Layer 4 is the safety net that ensures nothing crashes silently or vanishes — every unrecognized error is still logged and shown to the user in some form. This four-layer shape — typed errors, boundary conversion, selective local handling, global net — is what "good error handling" looks like in a real codebase.

Interview Questions

Q1 (beginner): Why should you always throw new Error(...) instead of throw "some string"?

An Error object carries a .stack trace captured at construction time (invaluable for debugging — it tells you exactly where and how the call chain got there), a .message, and participates in the Error/subclass instanceof hierarchy so catch blocks can distinguish error types. A thrown string has none of this — no stack trace, no type information — and any code expecting err.message or err instanceof SomeError breaks. Throwing non-Error values is a common source of painful, context-free bug reports; always throw Error or a subclass of it.

Q2 (beginner): What's the difference between an operational error and a programmer error?

An operational error is an expected failure mode of a working system — a network timeout, invalid user input, a file that doesn't exist, a rate limit — things that happen during normal operation and should be caught and handled gracefully (show a message, retry, fall back). A programmer error is an actual bug — calling a function with the wrong type, a null reference, faulty logic — indicating the program is in a state nobody intended. The distinction matters because it tells you how to respond: catch and handle operational errors specifically; let programmer errors surface loudly (crash, get logged, get fixed) rather than being silently caught, since continuing to run with a real bug can corrupt data or produce wrong results without anyone noticing.

Q3 (mid): How do you create a custom error type, and why would you?

Extend the built-in Error class: class ValidationError extends Error { constructor(msg, fields) { super(msg); this.name = "ValidationError"; this.fields = fields; } }. You'd do this to carry structured, machine-readable context beyond a message string — an error code for programmatic branching, an HTTP status for API errors, domain-specific data (which fields failed validation) — and so callers can distinguish error types with instanceof rather than fragile string-matching on the message. Custom error classes let different parts of the catch logic route different failure types to different handling (a validation error shows field-level messages; a not-found error shows a 404 page) instead of one generic error path for everything.

Q4 (mid): What does the cause option on Error do, and what problem does it solve?

new Error(message, { cause: originalError }) chains the original, lower-level error onto the new, higher-level one via an .cause property. It solves the problem of wrapping errors for context while preserving the root cause: when you catch a low-level failure (a DB connection error) and re-throw a more meaningful one ("Failed to load user profile"), naively you'd lose the original error's detail. With cause, the caller/logger gets the clean, high- level message and can walk err.cause (potentially a chain of several) to find the actual root cause for debugging. It's a standard, tool-supported alternative to ad-hoc patterns like concatenating messages or inventing custom .originalError properties — modern console/logging tools print the full cause chain automatically.

Q5 (senior): Why is a blanket try { ... } catch (e) { /* log and continue */ } around large sections of code often an anti-pattern?

It conflates operational and programmer errors into one undifferentiated response. A large try block can throw for many different reasons — a network timeout (operational, expected, safe to handle) or a null-reference bug three function calls deep (a programmer error indicating actual broken logic) — and catching everything the same way means the code "handles" a real bug by silently swallowing it and continuing, which is dangerous: the program is now in a state the author never designed for and never tested, potentially operating on corrupted or partial data, and the underlying bug goes unnoticed until it causes a much more confusing failure downstream (or a silent data-correctness issue nobody catches until much later). It also erases the specific error information that would let you respond correctly — a network error and a bug both just become "logged and ignored." The fix is narrower, more deliberate error handling: wrap only the specific operations whose failure modes you actually understand and can meaningfully recover from, check the error's type/code before deciding how to respond, and let unrecognized errors propagate (to a caller with more context, or ultimately to a global handler that logs and fails safely) rather than being caught by a catch-all that treats every failure identically. The size and specificity of a try block should roughly match your actual understanding of what can go wrong inside it.

Q6 (senior): Compare the exception-based error model with the Result/Either pattern. When would you choose each for a library's public API?

Exceptions propagate automatically up the call stack until caught, which is convenient — you don't have to thread error handling through every intermediate function — but the cost is invisibility: a function's type signature gives no indication it can throw or what, so callers discover failure modes only by reading implementation/docs or being surprised at runtime, and it's easy to forget a catch block entirely, silently letting failures propagate further than intended. The Result/Either pattern makes failure an explicit part of the return type ({ok: true, value} | {ok: false, error}, or a proper tagged union in TypeScript), which forces callers to handle both branches — you cannot access the success value without first checking/narrowing past the failure case, so the type system (in a typed language) makes forgetting error handling a compile error rather than a runtime surprise. The trade-off is verbosity: every call site needs an explicit check, and composing/chaining multiple Result-returning operations needs helper combinators (map/andThen, similar to how promises compose) to avoid deeply nested if-checks. For a library's public API: use Result for operations with expected, common failure modes the caller should be nudged to handle explicitly — a parse function that might receive malformed input, a lookup that might not find a match, a validation function — where forcing the caller to acknowledge failure improves correctness and API clarity, and TypeScript's exhaustiveness checking can enforce it. Use exceptions for genuinely exceptional conditions that most call sites shouldn't need to handle individually and that are fine propagating up to a general handler — infrastructure failures, invariant violations, "this should never happen" assertions — where forcing every caller to handle it locally would just add boilerplate for a rare case better handled centrally. Many well-designed modern libraries use both: Result for the expected, narrow failure modes of a specific operation, and exceptions (or promise rejection) for the broader, less-recoverable failures — the judgment is whether forcing the caller to look at the error case, right there at the call site, genuinely improves correctness for that specific operation.

🏋️ Exercises

1 (easy): Custom error. Write a NotFoundError class extending Error that stores a resource name and sets a sensible message and name.
Solution 
class NotFoundError extends Error {
  constructor(resource) {
    super(`${resource} not found`);
    this.name = "NotFoundError";
    this.resource = resource;
  }
}
try { throw new NotFoundError("User"); }
catch (e) { console.log(e.message, e.name, e.resource); }   // "User not found" "NotFoundError" "User"

Calling super(message) sets up the base Error correctly (including the stack trace); setting .name makes it identifiable in logs/DevTools; the extra resource field carries structured context.

2 (easy): instanceof routing. Given a mix of ValidationError and generic Error thrown from different calls, write a catch block that handles validation errors specially and re-throws everything else.
Solution 
try {
  doSomething();
} catch (err) {
  if (err instanceof ValidationError) {
    showFieldErrors(err.fields);
  } else {
    throw err;   // not a validation error — don't swallow it, let it propagate
  }
}

The instanceof check routes the known, expected error type to specific handling; anything else is re-thrown rather than silently absorbed, per the operational/programmer split.

3 (medium): Wrap with cause. Catch a low-level parse error and re-throw a higher-level ConfigError that preserves the original via cause.
Solution 
class ConfigError extends Error {
  constructor(message, options) { super(message, options); this.name = "ConfigError"; }
}
function loadConfig(text) {
  try {
    return JSON.parse(text);
  } catch (parseErr) {
    throw new ConfigError("Failed to load config: invalid JSON", { cause: parseErr });
  }
}
try { loadConfig("{bad json"); }
catch (e) { console.log(e.message, "\ncaused by:", e.cause.message); }

The high-level message is clear for users/logs, while .cause retains the original SyntaxError's detail for debugging — nothing is lost in translation.

4 (medium): Async try/catch. Write an async function that fetches a user, throwing a typed NotFoundError on a 404 and letting network failures propagate unchanged.
Solution 
async function getUser(id) {
  const res = await fetch(`/api/users/${id}`);   // network failure propagates as-is (unrecognized → not caught here)
  if (res.status === 404) throw new NotFoundError("User");
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

No try/catch is needed here specifically because we're not recovering from anything at this layer — we convert the 404 case into a typed error and let a genuine network rejection propagate naturally to the caller, who decides how to handle it.

5 (hard): Result-pattern wrapper. Write toResult(promise) that converts any promise into a {ok, value}/{ok, error} result, and use it to handle a fetch without try/catch at the call site.
Solution 
async function toResult(promise) {
  try {
    return { ok: true, value: await promise };
  } catch (error) {
    return { ok: false, error };
  }
}

const result = await toResult(fetch("/api/data").then(r => r.json()));
if (!result.ok) { console.error("failed:", result.error.message); }
else { console.log(result.value); }

The wrapper contains the only try/catch; every call site gets an explicit, checkable ok flag instead of needing its own try/catch — the Result pattern applied generically to any promise.

6 (hard): Layered strategy. Build a small safeDivide(a, b) API-boundary function that throws a typed DivisionError for division by zero (operational, expected) but lets a TypeError from non-numeric input propagate unmodified (programmer error — caller misused the API). Demonstrate both paths.
Solution 
class DivisionError extends Error {
  constructor() { super("Cannot divide by zero"); this.name = "DivisionError"; this.code = "DIV_ZERO"; }
}
function safeDivide(a, b) {
  if (typeof a !== "number" || typeof b !== "number") {
    throw new TypeError("safeDivide requires two numbers");   // programmer error — caller's bug, not caught here
  }
  if (b === 0) throw new DivisionError();                       // operational — an EXPECTED edge case
  return a / b;
}

try { safeDivide(10, 0); } catch (e) { if (e instanceof DivisionError) console.log("handled:", e.message); }
try { safeDivide(10, "x"); } catch (e) { console.log("propagated (a real bug):", e instanceof TypeError); }

Division by zero is a foreseeable, valid input the function's contract explicitly handles with a typed error meant to be caught. Passing a non-number violates the function's basic contract — that's the caller's bug, so it throws the generic TypeError (a programmer error) rather than being wrapped into something that looks "handleable," making it clear at the call site that this path represents a code defect to fix, not a recoverable runtime condition.

🎯 Key Takeaways

  • Always throw Error objects (never strings) — they carry a stack trace, message, and participate in the instanceof hierarchy.
  • Custom error subclasses carry structured, machine-readable context (codes, status, fields) so handlers route by instanceof, not fragile message-matching.
  • Operational errors (expected: network, input, not-found) should be caught and handled; programmer errors (bugs) should surface loudly, not be silently swallowed.
  • try/catch unifies sync throws and async rejections; catch what you specifically expect and re-throw the rest.
  • Global handlers (unhandledrejection, uncaughtException) are a safety net for logging/failing safely — never the primary error-handling strategy.
  • The native cause option chains errors when wrapping, preserving root-cause detail; the Result/Either pattern makes failure an explicit, forced-to-handle return value as an alternative to exceptions.
