Error Handling Patterns Advanced

Junior code handles the happy path; production code is mostly about the unhappy paths. The difference between "we lost the request" and "we lost the customer" is usually error handling.

Throw Errors, Not Strings

throw new Error("Payment failed");             // ✅ has .message, .stack, .cause
throw "payment failed";                        // ❌ no stack trace — undebuggable in prod

// Chain causes instead of losing them:
try { await chargeCard(order); }
catch (err) {
  throw new Error(`Order ${order.id} payment failed`, { cause: err });
}   // logging the top error shows the WHOLE story: business context + root cause

Custom Error Taxonomy (how services organize failure)

class AppError extends Error {
  constructor(message, { code, status = 500, cause } = {}) {
    super(message, { cause });
    this.name = new.target.name;      // "ValidationError", not "Error"
    this.code = code;                  // machine-readable: "USER_NOT_FOUND"
    this.status = status;
  }
}
class ValidationError extends AppError { constructor(msg, cause) { super(msg, { code: "VALIDATION", status: 400, cause }); } }
class NotFoundError   extends AppError { constructor(msg)        { super(msg, { code: "NOT_FOUND",  status: 404 }); } }

// One boundary maps error type → response/UI:
if (err instanceof ValidationError) showFieldErrors(err);
else if (err instanceof NotFoundError) show404();
else { logger.error(err); showGenericFailure(); }    // unknown = bug: log loudly, tell softly

The Two Cardinal Sins

// SIN 1 — the silent catch (bugs vanish into the void):
try { await save(); } catch (e) {}                     // ❌ NEVER

// SIN 2 — catching what you can't handle:
try { … } catch (err) {
  if (err instanceof QuotaError) return retryLater();  // I can handle THIS
  throw err;                                            // everything else: not mine — rethrow
}

Rule: catch at the level that can do something — retry, fall back, show UI. Everything else bubbles to a boundary that logs + reports.

Async Traps

// A rejected promise you forgot to await escapes try/catch entirely:
try { savePrefs(); } catch (e) { }        // ❌ fire-and-forgot — rejection is UNHANDLED
try { await savePrefs(); } catch (e) { }  // ✅

// Global last-resort handlers (report to Sentry/telemetry, don't "handle"):
window.addEventListener("error", report);
window.addEventListener("unhandledrejection", report);

Errors as Values (the alternative style)

// For EXPECTED failures, many teams prefer results over throws:
function parseConfig(text) {
  try { return { ok: true, value: JSON.parse(text) }; }
  catch (e) { return { ok: false, error: e }; }
}
const r = parseConfig(raw);
if (!r.ok) return useDefaults();        // failure is visible in the type/flow, not invisible control flow
// Throws = bugs & unexpected states. Results = failures you PLAN for. Mixing them deliberately is the senior move.
💡 Production tip: an error message should contain what failed, for which entity, and what to do: "Failed to send invoice #4211 to billing-service (timeout after 3 retries)" beats "request failed". Your 3am on-call self is the audience.
