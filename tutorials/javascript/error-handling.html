Error Handling Patterns Advanced

Junior code handles the happy path; production code is mostly about the unhappy paths — networks fail, inputs lie, disks fill. The difference between "we lost a request" and "we lost the customer (and can't reproduce why)" is the discipline in this lesson: real Error objects, a typed taxonomy, catch-where-you-can-act, and honest telemetry.

Rule 1 — Throw Errors, Chain Causes

throw new Error("Payment failed");           // ✅ .message + .stack (+ .cause)
throw "payment failed";                       // ❌ no stack — undebuggable at 3am

// The cause chain — context WITHOUT losing the root:
try { await chargeCard(order); }
catch (err) {
  throw new Error(`Order ${order.id}: payment failed`, { cause: err });
}
// Logging the top error walks the chain: business context AND the socket timeout underneath.
// Re-throwing a NEW error without { cause } is destroying evidence.

// The built-in family — throw the precise one:
TypeError (wrong kind of value) · RangeError (out of bounds) · SyntaxError ·
AggregateError (Promise.any) · DOMException (AbortError lives here)

Rule 2 — a Typed Taxonomy (how services organize failure)

class AppError extends Error {
  constructor(message, { code, status = 500, cause, meta } = {}) {
    super(message, { cause });
    this.name = new.target.name;        // "ValidationError", not "Error" — in every log line
    this.code = code;                    // machine-readable: "USER_NOT_FOUND"
    this.status = status;                // HTTP mapping for API boundaries
    this.meta = meta;                    // structured context: { userId, orderId }
  }
}
class ValidationError extends AppError { constructor(msg, meta) { super(msg, { code: "VALIDATION", status: 400, meta }); } }
class NotFoundError   extends AppError { constructor(msg, meta) { super(msg, { code: "NOT_FOUND",  status: 404, meta }); } }
class RateLimitError  extends AppError { constructor(retryAfter) { super("Rate limited", { code: "RATE_LIMIT", status: 429, meta: { retryAfter } }); } }

// ONE boundary maps type → response/UI:
try { await handleRequest(req); }
catch (err) {
  if (err instanceof ValidationError)  return res.status(400).json({ code: err.code, errors: err.meta });
  if (err instanceof NotFoundError)    return res.status(404).json({ code: err.code });
  if (err instanceof RateLimitError)   return res.status(429).set("Retry-After", err.meta.retryAfter).end();
  logger.error(err);                   // UNKNOWN = a bug: log loudly…
  return res.status(500).json({ code: "INTERNAL" });   // …tell softly (no stack to users)
}

Rule 3 — Catch Where You Can ACT

// The anti-pattern: catch-log-rethrow at every layer → 5 duplicate logs per incident
// The pattern: intermediate layers catch ONLY what they can handle; else let it fly:
try { return await primaryStore.get(key); }
catch (err) {
  if (err instanceof TimeoutError) return fallbackStore.get(key);  // I can handle THIS
  throw err;                                                        // not mine — pass it up
}
// Handlers exist at exactly two-ish places: local recovery points (retry/fallback/UI state)
// and the top-level boundary (log + map + respond). Everything between stays clean.

The Async Traps (again, because they cause most incidents)

try { savePrefs(); } catch {}            // ❌ un-awaited — rejection escapes AFTER try exits
try { await savePrefs(); } catch {}      // ✅
savePrefs().catch(report);                // ✅ deliberate fire-and-forget, still observed

try { return doWork(); }                  // ❌ pending promise escapes before settling
try { return await doWork(); }            // ✅ 'return await' is MEANINGFUL inside try

// Global last resorts — telemetry, not control flow:
window.addEventListener("error", report);
window.addEventListener("unhandledrejection", report);
process.on("unhandledRejection", …)  // Node: log + crash-restart policy (state may be corrupt)

Errors as Values — the Second Style (use both, deliberately)

// EXPECTED failures read better as data than as control flow:
function parseConfig(text) {
  try { return { ok: true,  value: JSON.parse(text) }; }
  catch (e) { return { ok: false, error: `Invalid config: ${e.message}` }; }
}
const r = parseConfig(raw);
if (!r.ok) return useDefaults(r.error);      // failure is VISIBLE in the flow & the types
use(r.value);

// The division that scales:
//   Results/{ok} → failures you PLAN for: validation, parsing, not-found, user input
//   throw       → bugs & broken invariants: can't-happen states, programmer errors
// TS discriminated unions make this airtight ({ok:true;value:T}|{ok:false;error:E}) —
// the compiler forces callers to check. (Same idea as Rust's Result / Go's err returns.)

Telemetry — Errors Nobody Sees Don't Exist

logger.error("invoice send failed", {
  code: err.code, message: err.message, stack: err.stack,
  cause: err.cause?.message,
  invoiceId, userId, attempt,           // STRUCTURED context — filterable, aggregatable
});
// Production reality: Sentry-style aggregation (grouping by stack), release tagging,
// alert on NEW error types & rate spikes — not on every occurrence.
// Messages answer: WHAT failed, for WHICH entity, WHY-ish, what NEXT:
// "Invoice #4211 send to billing-svc failed (timeout after 3 retries) — queued for retry at 14:32"

Interview Questions

Q1: Design error handling for a checkout flow, end to end.

Taxonomy: ValidationError(400)/PaymentDeclinedError(402, user-facing message)/ InventoryError(409)/upstream TimeoutError(retryable). Client: api() wrapper maps status→typed errors; declined→inline UI with recovery action; timeout→auto-retry with backoff then "try later"; unknown→generic toast + Sentry with order context. Server: boundary maps types→status, logs structured with cause chains, alerts on new types. The grader listens for: WHERE things are caught (act-or-pass), user-message vs log-detail separation, retryability by type.

Q2: Exceptions vs result objects — argue BOTH, then your policy.

Exceptions: zero-cost happy path, non-local exit to boundaries, uncatchable-by-accident (loud). Results: failures visible in signatures/types, forces handling, composes in pipelines, no try/catch noise for common cases. Policy that survives review: results for EXPECTED domain failures (validation/parse/not-found), exceptions for invariant violations and infrastructure surprises — and the boundary converts between them exactly once.

Q3: What's wrong with: async function f() { try { return fetchData() } catch (e) { log(e); return null } }?

Three things: fetchData() isn't awaited so its rejection ESCAPES the try (the catch is dead code for async failures) — needs return await; returning null converts a loud failure into a silent one callers must null-check (consider rethrowing or a result); log(e) without context/structure is telemetry theater. Bonus: name that a lint rule (no-return-await confusion aside, @typescript-eslint/no-floating-promises) catches the first.

Q4: When SHOULD a Node process crash on error?

On truly unknown exceptions/unhandled rejections in stateful contexts: after logging, crash and let the supervisor restart — continuing with possibly-corrupt state risks worse (data corruption, zombie behavior). Known/expected errors are handled per-request without crashing. The pattern: per-request boundaries absorb known failure; process-level handlers log + graceful-shutdown + exit(1). "Let it crash" is a strategy, not negligence — when paired with supervision.

🎯 Key Takeaways

  • Real Errors only (stack!), chained with { cause } — context on top, root preserved underneath.
  • A small typed taxonomy (code/status/meta) lets ONE boundary map failure→response/UI coherently.
  • Catch where you can act — local recovery or top boundary; everything between passes errors through.
  • Async: await inside try, return await matters, fire-and-forget gets .catch, globals are telemetry only.
  • Results for expected failures, throws for bugs — written down as team policy, enforced by types.
  • Structured, aggregated, alerted telemetry — an error message is written for the 3am responder.
