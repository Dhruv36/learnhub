Tooling, Bundling & Packages Modules & Tooling

What you'll master: npm and package.json (dependencies, scripts, semantic versioning), the lockfile's purpose, what bundlers and transpilers actually do and why you need them, tree-shaking, and reading/configuring the modern toolchain (Vite/esbuild-class tools) well enough to diagnose real build problems. Prerequisites: Modules (what bundlers consume). This is the "how does my code actually get from source to a running app" lesson — essential for working in any real project.

Part 1 — npm, package.json, and Dependencies

{
  "name": "my-app",
  "version": "1.2.3",
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "test": "vitest"
  },
  "dependencies": {                    // needed at RUNTIME (shipped to production)
    "react": "^18.2.0"
  },
  "devDependencies": {                 // needed only for DEVELOPMENT/BUILD (not shipped)
    "vite": "^5.0.0",
    "eslint": "^8.50.0"
  }
}

package.json is a project's manifest: identity, scripts (run via npm run dev), and dependencies. The dependencies vs devDependencies split matters: dependencies are needed at runtime (a UI library your app imports and ships), devDependencies are only needed to build/test/lint (the bundler itself, the test runner) and aren't included when your package is installed by others. npm install reads this file and downloads packages (and their dependencies, recursively) into node_modules.

Part 2 — Semantic Versioning & the Lockfile

"react": "^18.2.0"    // ^ = compatible with 18.2.0: allows 18.x.x, NOT 19.0.0 (MAJOR pinned)
"lodash": "~4.17.21"  // ~ = allows 4.17.x only (PATCH-level updates)
"exact-pkg": "4.0.0"  // no prefix = exact version only

// Semantic Versioning: MAJOR.MINOR.PATCH
// MAJOR — breaking changes    MINOR — new features, backward-compatible    PATCH — bug fixes only

Semantic Versioning (SemVer) is the convention MAJOR.MINOR.PATCH promises: a MAJOR bump may break your code, MINOR adds features without breaking anything, PATCH is bug fixes only. The caret (^, npm's default) allows any MINOR/PATCH update but never MAJOR — the idea being those should be safe. In practice packages sometimes violate SemVer, which is exactly why the lockfile (package-lock.json, or yarn.lock/ pnpm-lock.yaml) exists: it records the exact resolved version of every dependency (including nested ones) actually installed, so npm install on a teammate's machine or in CI reproduces exactly the same tree — not just "a version matching the range." Always commit the lockfile; it's what makes builds reproducible.

Part 3 — Why You Need a Bundler

// Your source: dozens of ES modules importing each other.
// import { Button } from "./Button.js";  import { formatDate } from "./utils/date.js"; ...

// A BUNDLER (Webpack, Rollup, esbuild, Vite's build) walks the import graph starting
// from an entry point and combines everything into one or few optimized output files.

// Why not just ship the raw modules?
// - Hundreds of tiny files = hundreds of HTTP requests (even with HTTP/2, overhead adds up)
// - No minification (smaller = faster download)
// - No tree-shaking (unused code still shipped)
// - Some syntax/features aren't supported in all target browsers (needs transpiling too)

A bundler starts at your entry file, follows every import, and produces optimized output bundles. Why not just serve your ES module files directly (browsers do support ESM natively now)? For production you still want: fewer, larger files instead of hundreds of small requests; minification (stripping whitespace/renaming variables to shrink file size); and tree-shaking (Part 5). During development, though, modern tools (Vite) actually serve native ES modules unbundled for instant startup, and only bundle for the production build — the best of both worlds.

Part 4 — Transpiling: Writing Tomorrow's JS Today

// You write modern syntax:
const greet = (name) => `Hi, ${name ?? "friend"}`;

// Babel/esbuild/SWC TRANSPILE it to older syntax for browsers that don't support it:
"use strict";
var greet = function (name) {
  return "Hi, " + (name !== null && name !== void 0 ? name : "friend");
};

// Polyfills go further — they ADD missing runtime features (not just syntax):
// e.g. Array.prototype.flat didn't exist in old engines — a polyfill implements it in JS.

Transpiling (Babel, or the faster Rust/Go-based esbuild/SWC) converts modern syntax (optional chaining, arrow functions, classes) into older syntax that runs on browsers/environments that don't support the new syntax natively. This is different from polyfilling: a polyfill provides a missing runtime feature (like Array.prototype.flat or fetch on very old browsers) by implementing it in JS, since you can't transpile a missing method into existence — syntax transforms and runtime polyfills solve different problems and are often used together. Modern tools use a browserslist config (or a "target") to decide how far back to transpile, balancing bundle size against browser support.

Part 5 — Tree-Shaking: Shipping Only What You Use

// utils.js — a library with many exports:
export function usedFn() { return "used"; }
export function unusedFn() { return "dead code"; }

// app.js:
import { usedFn } from "./utils.js";   // only usedFn is imported

// A tree-shaking bundler can STATICALLY see that unusedFn is never imported anywhere,
// and OMITS it from the final bundle entirely — smaller output, for free.

// Tree-shaking needs ESM's STATIC import/export structure — it can't see through
// CommonJS's dynamic require()/module.exports (Modules lesson) or side-effectful imports.

Tree-shaking is dead-code elimination based on statically analysing the import graph: if a named export is never imported anywhere, the bundler omits it from the output. This is why ES Modules' static import/export syntax matters beyond style — it's what makes tree-shaking possible, since the bundler can determine usage without running the code. CommonJS's dynamic require() can't be analysed the same way, which is one more reason the ecosystem is moving to ESM (Modules lesson). Import a whole library's default export or use a barrel that re-exports everything, and tree-shaking can be defeated — importing only what you need (import { specific } from "lib") keeps it effective.

Part 6 — The Modern Dev Server: Vite/esbuild-Class Tools

// vite.config.js — a minimal modern config:
export default {
  plugins: [/* framework plugins, e.g. React/Vue */],
  build: {
    outDir: "dist",
    sourcemap: true,        // map minified output back to original source for debugging
  },
  server: { port: 3000 },
};

// The dev-server model (Vite, and similar esbuild-based tools):
// - serves your SOURCE FILES as native ES modules directly to the browser — no bundling in dev
// - the browser requests each imported module on demand as you navigate — near-instant startup
// - Hot Module Replacement (HMR): edit a file, only THAT module reloads in the browser, state preserved
// - for PRODUCTION, it switches to a real bundler (Rollup, under Vite) for the optimized build

Older bundlers (classic Webpack) re-bundled the entire app on every save, which got progressively slower as a project grew. Modern tools like Vite flipped the model for development: they serve your source files as native ES modules directly (leveraging the browser's own module loader), so the dev server does almost no work upfront and startup is near-instant regardless of project size — only the module you actually navigate to gets compiled on request. Hot Module Replacement (HMR) swaps just the changed module in a running page, preserving app state instead of a full reload. For the production build, these tools still bundle and optimize (tree-shake, minify) for the reasons in Part 3 — dev and prod use different strategies because they optimize for different things (iteration speed vs. shipped-file efficiency).

Part 7 — Source Maps: Debugging Minified Code

// Your minified production file: one unreadable line of renamed variables.
// A SOURCE MAP is a separate file mapping each position in the minified output
// back to the original source file, line, and column.

// Browser DevTools READ the source map automatically:
// - set breakpoints in your ORIGINAL source, even though the browser runs the minified code
// - stack traces show original file names/line numbers, not "bundle.js:1:48213"

// Enable in build config: build: { sourcemap: true }
// ⚠ Don't ship source maps PUBLICLY for proprietary code — they reveal your original source.

Minified production code is unreadable and untraceable on its own. A source map (typically a .map file alongside the bundle) records the mapping from every position in the compiled output back to the original source — so browser DevTools can show you your actual code in the debugger and in stack traces/error reports, even though what's actually running is the minified bundle. Enabling source maps is essential for debugging production issues (paired with an error-tracking service like Sentry, which uses them to symbolicate stack traces from real user errors) — but be aware they expose your original source, so treat them as sensitive for closed-source projects (many teams upload them privately to their error tracker rather than serving them publicly).

Part 8 — Build It: Reading and Diagnosing a Real Build

Rather than a code snippet, let's walk the diagnostic process for the build issues you'll actually hit — this is the practical skill tooling knowledge buys you.

// Scenario 1: "Module not found: Can't resolve './Button'"
// → Check the exact filename/extension/case (case-sensitive on Linux CI, case-insensitive on your Mac/Windows —
//   a classic "works on my machine" bug). Check it's actually exported.

// Scenario 2: Production bundle is huge; a chart library shows up despite being used on one rarely-visited page.
// → Run a bundle analyzer (rollup-plugin-visualizer / webpack-bundle-analyzer) to SEE what's inside the bundle.
// → Likely fix: dynamic import() the chart library (Modules lesson) so it's a separate, lazy-loaded chunk.

// Scenario 3: "process is not defined" in browser code.
// → `process.env` is a NODE global; browser code needs the bundler to replace it at BUILD time
//   (Vite: import.meta.env; Webpack: DefinePlugin) — it's not actually available at runtime in the browser.

// Scenario 4: works with `npm run dev`, breaks in the PRODUCTION build only.
// → Dev often skips minification/tree-shaking — a bug that only appears when code is minified (e.g. relying
//   on a function's .name, which minification changes) or tree-shaken (an import with a hidden side effect
//   that got dropped) surfaces only in the prod build. Reproduce locally with `npm run build && npm run preview`.

Each scenario is a real, common failure mode, and each maps directly to a concept from this lesson: resolution errors are about the module graph the bundler is walking (Modules lesson); bundle bloat is diagnosed by looking inside the bundle and fixed with the dynamic-import code-splitting from Part 6/Modules; environment-variable confusion is about build-time replacement vs. runtime availability; and dev-vs-prod discrepancies come from the different optimization passes (minify, tree-shake) that only run in the production build. Knowing what a bundler does — not just how to run npm run build — is what turns a cryptic build error into a two-minute fix.

Interview Questions

Q1 (beginner): What's the difference between dependencies and devDependencies?

dependencies are packages your code needs at runtime — libraries it imports and that must be present when the app actually runs (and are included when your package is installed by others). devDependencies are needed only during development or building — the bundler, test runner, linter, type checker — and are not required at runtime, so they're excluded from what gets shipped or installed as a dependency of your package. The practical check: if removing it would break the running app in production, it's a dependency; if it would only break your build/test/lint tooling, it's a devDependency.

Q2 (beginner): What is a lockfile and why must you commit it?

A lockfile (package-lock.json, yarn.lock, pnpm-lock.yaml) records the exact, fully-resolved version of every package in your dependency tree — including nested dependencies — as actually installed. package.json only specifies version ranges (like ^18.2.0), which could resolve to different actual versions at different times or on different machines. The lockfile pins those ranges to specific versions so that npm install produces an identical node_modules tree everywhere — your machine, a teammate's, and CI. Without committing it, builds aren't reproducible: a dependency's new patch release could introduce a bug that appears on one machine but not another, purely due to version drift.

Q3 (mid): What does a bundler actually do, and why is it still needed even though browsers support ES modules natively?

A bundler starts from an entry file, follows the import graph, and combines the modules into one or a few optimized output files, typically also minifying and tree-shaking. Even though browsers can load native ES modules directly, production bundling still matters for: fewer, larger files instead of many small requests (reducing per-request overhead); minification (smaller downloads, faster parse); tree-shaking (omitting code that's never imported); and transpiling modern syntax for browsers that don't support it. Modern dev servers (Vite) actually skip bundling in development and serve native ES modules directly for fast startup, but still switch to a real bundler for the production build — because dev prioritizes iteration speed and prod prioritizes the smallest, fastest-loading shipped output, which are different goals needing different strategies.

Q4 (mid): What is tree-shaking and what prevents it from working?

Tree-shaking is dead-code elimination based on statically analysing which exports are actually imported and used — a bundler that sees a named export is never imported anywhere can omit it entirely from the output bundle. It depends on ES Modules' static import/export structure, which the bundler can analyse without executing code. It's prevented or weakened by: (1) CommonJS modules (require/module.exports), whose dynamic nature the bundler can't statically analyse the same way; (2) importing a library's default export or a whole namespace when you only use one piece, which can pull in everything; (3) modules with side effects at the top level (code that runs just from being imported, like a polyfill or a CSS import) — the bundler must keep those since removing them could change behaviour, unless the package explicitly marks itself "sideEffects": false in package.json to opt into more aggressive shaking. Importing only the specific named exports you use, from ESM-published packages, keeps tree-shaking effective.

Q5 (senior): Explain the trade-offs between a native-ESM dev server (Vite) and a traditional bundle-everything dev server (classic Webpack dev mode).

Classic Webpack dev mode bundles the entire application graph before serving anything (or incrementally re-bundles a large graph on change), which means startup time and rebuild time scale with project size — a large app can take many seconds to start or hot-reload, actively hurting iteration speed as the codebase grows. It does have the advantage of dev and prod running through essentially the same bundling pipeline, so dev-only discrepancies are rarer. A native-ESM dev server (Vite, and similar tools) instead serves source files directly as ES modules, transforming each file on demand only when the browser actually requests it (as you navigate/import), so server startup is near-instant and independent of total project size, and HMR updates are equally fast since only the changed module needs re-processing. The trade-off: dev and production now go through different pipelines (native ESM serving in dev, a real bundler like Rollup for the prod build), which opens a narrow but real class of dev-vs-prod discrepancy bugs — code that behaves correctly when each module loads independently in dev but breaks once bundled/tree-shaken/minified for prod (e.g. import-order- dependent side effects, or code relying on something the production optimizer changes). Also, very large numbers of individual module requests in dev (before HTTP/2 multiplexing, or with deeply nested dependency graphs) can add network overhead, though HTTP/2 mostly absorbs this. The senior takeaway: the native-ESM approach is almost universally faster for development at any non-trivial scale, which is why it's become the standard (Vite, and Webpack's own dev-server evolution moving this direction) — but it means you cannot fully trust "it works in dev" and must verify the actual production build (build + preview) before shipping, since dev is no longer a perfect proxy for prod's code path.

Q6 (senior): A production bundle has grown to 2MB and page load is slow. Walk through your diagnostic and remediation process.

Diagnostic: (1) Run a bundle analyzer (rollup-plugin-visualizer, webpack-bundle- analyzer, or Vite's built-in equivalent) to get a visual/tabular breakdown of what's actually inside the bundle and how much each dependency contributes — you cannot optimize what you haven't measured. (2) Identify the biggest contributors: often a handful of large libraries (a full charting/date/UI library where only a fraction is used, moment.js instead of a lighter date library, an icon library importing all icons instead of the ones used) or duplicate versions of the same package (mismatched version ranges pulling in two copies of a dependency — the lockfile/dedupe tooling can reveal this). (3) Check whether tree-shaking is actually working for each — CommonJS-only packages or ones that import "the whole library" defeat it. Remediation, roughly in order of impact: (a) Code-split by route/feature with dynamic import() (Modules lesson) so large, not-always-needed code (a rich editor, charts, an admin panel) loads only when visited, not in the initial bundle. (b) Replace or trim heavy dependencies — swap a large library for a lighter alternative, or import only the specific functions needed (import debounce from "lodash/debounce" vs the whole lodash) where the library supports it. (c) Deduplicate — align version ranges so only one copy of a shared dependency ships. (d) Verify minification and compression are actually enabled in the prod build, and that the server serves gzip/brotli-compressed assets (a huge, often-overlooked win — text assets compress 60-80%). (e) Lazy-load below-the-fold or interaction-gated UI (modals, secondary tabs). (f) Set a bundle size budget in CI (many bundlers support failing the build if a chunk exceeds a threshold) so regressions are caught before merge, not discovered later at 2MB. After remediation, re-measure with the analyzer and with real load-time metrics (Lighthouse, actual field data) to confirm the fix moved the number that matters — time-to-interactive for real users, not just raw byte count.

🏋️ Exercises

1 (easy): Classify dependencies. For a React app using Vite and ESLint, decide whether each belongs in dependencies or devDependencies: react, vite, eslint, react-router-dom.
Solution

dependencies: react, react-router-dom — needed at runtime for the app to function. devDependencies: vite, eslint — only needed to build and lint the project, never imported by the shipped app itself.

2 (easy): Read a version range. For "lib": "^2.4.1", state whether each of these installs would be allowed by npm install: 2.4.5, 2.9.0, 3.0.0.
Solution

2.4.5 — allowed (patch bump). 2.9.0 — allowed (minor bump, still MAJOR 2). 3.0.0 — NOT allowed (^ never crosses a MAJOR version boundary). The caret keeps you within the same MAJOR version, accepting any MINOR/PATCH update.

3 (medium): Diagnose a resolution error. A build fails with "Module not found: Can't resolve './utils/Formatter'" but the file formatter.js exists in that folder. What's likely wrong, and why might it pass locally but fail in CI?
Solution

The import's casing (Formatter) doesn't match the actual filename (formatter.js). It can pass locally on macOS/Windows because their filesystems are case-insensitive by default, so the mismatched case still resolves — but Linux (most CI runners) has a case-sensitive filesystem, where the mismatch is a hard failure. Fix: match the import's case to the actual filename exactly.

4 (medium): Shrink a bundle. A page imports an entire icon library (import * as Icons from "icon-lib") but uses only two icons. Propose a fix and explain the mechanism that makes it work.
Solution 
// Before — imports the ENTIRE namespace, defeating tree-shaking in many bundlers:
import * as Icons from "icon-lib";
<Icons.Home />

// After — import only the specific named exports actually used:
import { Home, Settings } from "icon-lib";

Namespace imports (import * as) can prevent a bundler from statically determining which specific exports are used, so it may conservatively keep the whole module. Importing only the named exports you use gives the bundler's tree-shaking a precise, analysable usage signal, letting it omit every unused icon from the final bundle.

5 (hard): Investigate a dev/prod discrepancy. An app works with npm run dev but a feature silently breaks in the production build. List three plausible root causes tied to what a production build does differently, and how you'd narrow down which one it is.
Solution

Three plausible causes: (1) Tree-shaking removed something with a needed side effect — an import relied on for its side effect (a polyfill, a CSS import, a global registration) that the bundler determined was "unused" and dropped. (2) Minification changed observable behaviour — code relying on function.name, or on specific property key order, or on unminified variable names via some reflection trick, breaks once names are mangled. (3) Environment variable handling differs — dev might have a fallback or different injected value for import.meta.env.MODE/process.env.NODE_ENV-gated code paths, so a branch that's dev-only (or prod-only) never got exercised in dev testing. Narrowing down: reproduce locally with the actual production build (npm run build && npm run preview, not dev) so you're debugging the real artifact; use a bundle analyzer to check if the suspected module/code is actually present in the output; temporarily disable minification in the prod build config to see if the bug disappears (isolates minification vs. tree-shaking); and check the build config for any NODE_ENV-gated code that might silently take a different path in production.

6 (hard): Design a CI safeguard. Propose a CI check that catches both "someone forgot to commit the lockfile update" and "the bundle size regressed significantly" before merge.
Solution 
# Lockfile check — fail if package.json changed but the lockfile wasn't updated to match:
npm ci   # (not `npm install`) — installs STRICTLY from the lockfile, FAILS if it's out of sync
         # with package.json, catching exactly this class of mistake in CI.

# Bundle size check — build, then compare the output size against a budget:
npm run build
node -e "
  const size = require('fs').statSync('dist/assets/main.js').size;
  const budget = 500 * 1024;  // 500KB budget, tune per project
  if (size > budget) { console.error('Bundle exceeds budget:', size, '>', budget); process.exit(1); }
"

npm ci is the key insight for the lockfile check — unlike npm install, it refuses to proceed if package.json and the lockfile disagree, which is exactly the "forgot to commit the updated lockfile" mistake, caught automatically rather than relying on reviewer vigilance. The bundle-size check is a simple threshold gate; production setups typically use a dedicated tool (bundlesize, size-limit) that tracks size per-chunk over time and fails on a percentage regression rather than a single hard-coded number, but the underlying principle — measure the build output in CI and gate on it — is the same.

🎯 Key Takeaways

  • package.json declares dependencies (runtime) vs devDependencies (build/dev-only) and scripts; the lockfile pins exact versions for reproducible installs — always commit it.
  • SemVer (MAJOR.MINOR.PATCH) and range operators (^/~) express compatible-update intent, but the lockfile is what guarantees identical installs everywhere.
  • A bundler combines the import graph into optimized output; transpiling converts modern syntax for older targets, polyfilling adds missing runtime features — different problems.
  • Tree-shaking statically removes unused exports — it needs ESM's static structure and breaks down with CommonJS, namespace imports, or unmarked side effects.
  • Modern dev servers (Vite) serve native ES modules unbundled for fast dev startup/HMR, then switch to real bundling for production — meaning dev isn't a perfect proxy for prod behavior.
  • Source maps let DevTools debug minified code against original source — enable them, but treat them as sensitive for closed-source projects; diagnose build issues by understanding what each tool actually does to your code.
