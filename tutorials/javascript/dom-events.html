DOM & Events Basics

The DOM is the browser's live object tree built from your HTML; events are how it talks back. Frameworks wrap this layer — but they're debugged at this layer, their performance problems live here, and interviews test it raw. Everything in this lesson is what React/Vue do under the hood.

Selecting — and the Live-vs-Static Trap

const form   = document.querySelector("#signup");        // first match, ANY CSS selector
const inputs = document.querySelectorAll("input");         // STATIC NodeList — a snapshot
const rows   = table.getElementsByTagName("tr");           // LIVE HTMLCollection — updates itself!

// The trap: iterate a LIVE collection while adding matching elements → infinite loop / skipped items.
// Modern default: querySelector(All). Convert for array methods: [...inputs].map(…)
el.matches(".card")      // does el match this selector?
el.closest("[data-id]")  // walk UP to the nearest matching ancestor (incl. self) — delegation's engine

Creating & Inserting — Batching Is the Skill

const li = document.createElement("li");
li.textContent = task.title;
list.append(li);            // append/prepend/before/after/replaceWith/remove — the modern set

// 500 rows the WRONG ways:
rows.forEach(r => list.append(makeRow(r)));      // 500 DOM touches — style/layout churn
rows.forEach(r => list.innerHTML += rowHtml(r)); // re-PARSES everything each time: O(n²), listeners destroyed

// The RIGHT way — build detached, attach once:
const frag = document.createDocumentFragment();
rows.forEach(r => frag.append(makeRow(r)));
list.append(frag);                               // ONE DOM touch
// (This is the idea virtual DOM industrializes: compute changes off-tree, commit minimally.)

Events — Registration Done Right

button.addEventListener("click", onSave);                    // reference, not onSave()
button.addEventListener("click", onSave, { once: true });     // self-removing
window.addEventListener("scroll", onScroll, { passive: true }); // "I won't preventDefault" → browser scrolls without waiting

// Cleanup — the leak-prevention contract (memory lesson):
const ctl = new AbortController();
window.addEventListener("resize", onResize, { signal: ctl.signal });
input.addEventListener("input", onType,   { signal: ctl.signal });
ctl.abort();                                                   // removes EVERYTHING tied to the signal — component unmount in one line

The Event Object — Your Five Tools

form.addEventListener("submit", (e) => {
  e.preventDefault();                    // stop the BROWSER action (nav, submit, context menu)
  const data = Object.fromEntries(new FormData(form));   // whole form → object, files included
});

e.target          // where it actually HAPPENED (deepest element)
e.currentTarget   // where THIS listener is attached — the difference is delegation
e.stopPropagation()  // halt bubbling — deliberate tool, not a reflex (breaks outside-click/analytics)
e.key             // "Enter" "Escape" "ArrowDown" — keyboard UX

Bubbling, Capture & Delegation — the Senior Pattern

Phases: CAPTURE (document → target) → TARGET → BUBBLE (target → document)
Listeners default to bubble; {capture: true} for the rare down-phase needs.
Some events don't bubble: focus/blur (use focusin/focusout), mouseenter/leave (use over/out).

// ONE listener runs a 10,000-row table:
table.addEventListener("click", (e) => {
  const btn = e.target.closest("[data-action]");
  if (!btn || !table.contains(btn)) return;         // clicked outside any action
  const row = btn.closest("tr");
  actions[btn.dataset.action]?.(row.dataset.id);     // {delete, edit, star}[action](id)
});
Why delegation winsDetail
Memory & setup1 listener vs 10,000; nothing to rebind
Dynamic rows just workRows added later are covered automatically — the killer feature
It's how frameworks workReact attaches at the root and routes synthetic events — you're learning the real mechanism

Observers — Reacting Without Polling

// Visibility (lazy-load, infinite scroll, analytics impressions, active-nav):
new IntersectionObserver((entries) => {
  entries.forEach(en => en.isIntersecting && load(en.target));
}, { rootMargin: "200px" }).observe(img);          // preload 200px early

// Element size (charts, virtualized lists) — never window.resize math:
new ResizeObserver(([en]) => chart.resize(en.contentRect)).observe(panel);

// Third-party DOM changes you don't control:
new MutationObserver(cb).observe(widget, { childList: true, subtree: true });
// All three: disconnect() on teardown — observers hold references (memory lesson).

Interview Questions

Q1: Implement event delegation for a dynamic todo list with complete/delete buttons.

One listener on the UL; e.target.closest("[data-action]") to find the button, guard with list.contains, closest("li").dataset.id for the item, dispatch on data-action. Points scored for: why closest (clicks land on icons inside buttons), the containment guard, dynamic items working free, and one cleanup point instead of n.

Q2: target vs currentTarget — and when are they the same?

target = the deepest element where the event originated; currentTarget = the element this listener is bound to (equals `this` in a regular-function handler). They're identical only when the event originates on the listening element itself. Delegation is literally "attach at currentTarget, route by target".

Q3: What does {passive: true} actually change, mechanically?

Scroll/touch handlers CAN call preventDefault to block scrolling, so by default the browser must WAIT for your handler before scrolling — JS jank becomes scroll jank. passive:true is a promise you won't preventDefault, letting the compositor scroll immediately and run your handler in parallel. Browsers already default it on window/body touch listeners because the win is so large.

Q4: Why did the platform add IntersectionObserver when scroll events existed?

Scroll handlers fire constantly on the main thread, and visibility math (getBoundingClientRect) forces layout — per frame, per element: the classic jank recipe. IO is asynchronous, batched, and computed by the engine (which already knows geometry), with rootMargin/thresholds expressing intent declaratively. Same shift as CSS container queries vs resize listeners: move the question into the engine.

🎯 Key Takeaways

  • querySelector(All) + matches/closest; know live vs static collections.
  • textContent for user data (XSS); innerHTML only for trusted markup; fragments batch inserts — one DOM touch.
  • Attributes (initial, string) vs properties (live, typed) — .value vs getAttribute("value").
  • Events: reference not call, once/passive options, AbortController = one-line teardown, submit not click, focusin not focus for delegation.
  • Capture → target → bubble; delegation (closest + containment guard) scales to any list size and future rows.
  • Observers (Intersection/Resize/Mutation) replace polling and scroll math — and must be disconnected.
