Prototypes & Classes Intermediate

JavaScript's object model is delegation through a chain of live links — not class-copies-blueprint like Java. class syntax is ergonomic paint over that chain. Interviews probe under the paint; production code mostly stays on top of it. This lesson gives you both layers and the design judgment (composition vs inheritance) that seniors are graded on.

The Prototype Chain — Delegation, Not Copying

const animal = { eats: true, describe() { return `eats: ${this.eats}`; } };
const dog = Object.create(animal);        // dog's [[Prototype]] = animal
dog.barks = true;

dog.barks      // true       — OWN property
dog.eats       // true       — not own → walk UP the chain → found on animal
dog.describe() // "eats: true" — method found on animal, but this = dog (call-site rule!)
dog.flies      // undefined  — chain exhausted: animal → Object.prototype → null

animal.eats = false;
dog.describe() // "eats: false" — LIVE link: change the prototype, every delegator sees it

// This chain is why [1,2].map exists: the array's chain is
// arr → Array.prototype (map, filter…) → Object.prototype (toString…) → null
🔍 Reading vs writing asymmetry: property READS walk the chain; WRITES create an own property on the receiver (dog.eats = true shadows animal's, doesn't change it). That asymmetry is why "modifying an inherited object property" sometimes seems to fork state — and why shared MUTABLE state on prototypes is a bug factory.

Classes — the Modern Syntax, Annotated

class Account {
  #balance = 0;                                 // truly PRIVATE (language-enforced)
  static #count = 0;                            // private static
  owner;                                        // public field declaration

  constructor(owner) { this.owner = owner; Account.#count++; }

  deposit(amount) {                             // → Account.prototype.deposit (SHARED)
    if (amount <= 0) throw new RangeError("amount must be positive");
    this.#balance += amount;
    return this;                                // chaining: acc.deposit(5).deposit(10)
  }
  get balance() { return this.#balance; }       // computed property, read like a field
  static from(json) { return new Account(json.owner); }   // factory on the CLASS
  static get count() { return Account.#count; }
}

const acc = new Account("Asha");
acc.deposit(100).deposit(50).balance;   // 150
// acc.#balance                          // SyntaxError — private is private (vs the old _convention)
Object.getPrototypeOf(acc) === Account.prototype   // true — the paint over the chain

What new does (the 4 steps interviewers want): ① create a fresh object linked to Fn.prototype → ② bind this to it → ③ run the constructor body → ④ return the object (unless the body explicitly returns a different object). Methods live ONCE on the prototype — a thousand instances share one deposit. Class bodies are always strict mode; classes are not hoisted callable (TDZ like const).

Inheritance — Mechanics, Then the Warning Label

class SavingsAccount extends Account {
  constructor(owner, rate) {
    super(owner);                 // MUST run before touching `this` (parent creates the object)
    this.rate = rate;
  }
  addInterest() { this.deposit(this.balance * this.rate); }
  deposit(amount) {               // override + extend:
    console.log("audited");
    return super.deposit(amount); // call the parent version
  }
}
// instanceof walks the chain: s instanceof SavingsAccount, Account, Object — all true
Inheritance is right for…Composition is right for…
True is-a with stable base contracts: Error subclasses (ValidationError extends AppError), framework base classes you're told to extendAlmost everything else: capabilities assembled from parts
Shallow (1 level) hierarchiesBehavior that varies independently (the "flying-swimming-robot-duck" explosion)
// Composition: build from capabilities, not ancestry
const canNotify = (s) => ({ notify: (m) => send(s.email, m) });
const canAudit  = (s) => ({ log: (a) => audit.push([s.id, a]) });
const makeAdmin = (id, email) => { const s = { id, email };
  return { ...s, ...canNotify(s), ...canAudit(s) }; };
// Deep class trees age badly: base changes ripple through every subclass; the diamond looms.
// React's classes→hooks migration is the era's loudest vote for composition.

Chain Literacy — Inspection & Edge APIs

Object.getPrototypeOf(dog)              // animal   (never use __proto__ in code)
Object.hasOwn(dog, "eats")              // false — own vs inherited (modern hasOwnProperty)
Object.keys(dog) / Object.entries(dog)  // OWN enumerable only — usually what you want
for (const k in dog) { }                // own + INHERITED enumerables — legacy, avoid
Object.create(null)                     // no prototype at all: pure dictionary,
                                        // immune to prototype pollution (security lesson)
JSON.parse(...)                          // plain objects — methods/prototypes don't survive JSON

Never Mutate Built-in Prototypes (the law + the story)

Array.prototype.myHelper = function () { … };   // ❌ global, forever, for every library on the page

Why it's law: collisions with future language features. The canonical incident — a popular library defined Array.prototype.flatten; when TC39 tried to ship flatten, it broke enough of the web that the committee renamed the standard method to flat ("SmooshGate"). Extend via utility functions or subclasses; polyfills implementing the standard behavior are the one sanctioned exception.

Interview Questions

Q1: Implement `new` yourself (myNew(Fn, ...args)).

const obj = Object.create(Fn.prototype); const ret = Fn.apply(obj, args); return (typeof ret === "object" && ret !== null) ? ret : obj. Naming the four steps and the object-return override is the full-credit answer; bonus for typeof function returns also overriding.

Q2: Where do methods live and why does that design matter?

On ClassName.prototype — one function object shared by all instances via the chain (memory: n instances, 1 method). Per-instance alternatives (constructor-assigned functions, class-field arrows) cost a copy per object — acceptable for bound handlers, wasteful as a default. Follow-up they want: this still resolves per CALL, so shared methods work on each receiver.

Q3: prototype vs __proto__ vs [[Prototype]] — untangle them.

[[Prototype]] is the internal link every object has (inspect via Object.getPrototypeOf). __proto__ is a legacy accessor for it (avoid in code). Fn.prototype is a property only functions have: the object that will BECOME [[Prototype]] of instances created by new Fn(). So acc.__proto__ === Account.prototype, but Account.prototype is not Account's own [[Prototype]] (that's Function.prototype).

Q4: Composition over inheritance — argue it with a concrete JS-ecosystem example.

React: class components (inheritance from React.Component, reuse via fragile HOC/mixin patterns) → hooks (behavior as composable functions: useFetch + useDebounce + useAuth combine freely, test in isolation, no diamond). Same story in Node middleware and functional utility stacks. Rule: inherit stable contracts (Error), compose capabilities.

🏋️ Practice (Playground):
1. Build the animal/dog delegation chain with Object.create; demonstrate the read-walk, the write-shadow, and the live-link update.
2. Implement myNew and verify it against new on the Account class.
3. Write Account with #balance, a getter, a static factory, and chaining; prove #balance is inaccessible.
4. Refactor a 3-level class hierarchy (User→Admin→SuperAdmin) into composition (canModerate, canConfigure); compare test setup for both.
5. Reproduce the shared-mutable-prototype bug (an array on the prototype) and fix it.

🎯 Key Takeaways

  • Objects delegate up a live chain; reads walk it, writes shadow on the receiver.
  • new: create-link → bind this → run → return; methods live once on the prototype; class syntax = this machinery, strict, TDZ'd.
  • #fields are real privacy; getters/setters guard access; statics belong to the type.
  • super() before this; instanceof walks the chain; JSON strips it all — rehydrate via factories.
  • Inherit stable contracts (Errors); compose capabilities — the ecosystem's verdict (hooks) is in.
  • Built-in prototypes are read-only by law (SmooshGate); Object.create(null) for pollution-proof dictionaries.
