Prototypes & Classes Intermediate

JavaScript's object model is a chain of live links, not copied blueprints. class syntax is a friendly coat of paint over it — interviews check whether you know what's under the paint.

The Prototype Chain

const animal = { eats: true };
const dog = Object.create(animal);     // dog's prototype = animal
dog.barks = true;

dog.barks   // true  — own property
dog.eats    // true  — not on dog → JS walks UP the chain and finds it on animal
dog.flies   // undefined — chain exhausted (…→ Object.prototype → null)

// This is DELEGATION, not copying: change animal.eats and every dog sees it.
// It's also why [1,2].map exists: arrays delegate to Array.prototype.

Classes — the Modern Syntax

class Account {
  #balance = 0;                        // # = truly private (enforced by the language)
  static from(json) { return new Account(json.owner); }   // factory on the class itself

  constructor(owner) { this.owner = owner; }

  deposit(amount) {
    if (amount <= 0) throw new RangeError("amount must be positive");
    this.#balance += amount;
    return this;                       // enable chaining: acc.deposit(5).deposit(10)
  }
  get balance() { return this.#balance; }   // computed property, read like a field
}

const acc = new Account("Asha");
acc.deposit(100).deposit(50);
acc.balance;          // 150 (getter)
// acc.#balance       // SyntaxError — actually private, unlike the old _convention

Under the hood: methods land on Account.prototype, shared by all instances; new creates an object linked to that prototype and runs the constructor with this bound to it.

Inheritance — and When to Avoid It

class SavingsAccount extends Account {
  constructor(owner, rate) {
    super(owner);                      // MUST call before touching this
    this.rate = rate;
  }
  addInterest() { this.deposit(this.balance * this.rate); }
}

The senior take: deep class hierarchies age badly — a change in the base ripples through every subclass. Production JS strongly favors composition: small objects and functions assembled together. React itself famously moved from classes to functions + hooks. Use classes where identity + state + methods genuinely belong together (services, models, errors).

Chain Literacy (debugging & interviews)

Object.getPrototypeOf(dog) === animal        // true
dog instanceof Object                         // true — instanceof walks the chain
Object.hasOwn(dog, "eats")                    // false — own vs inherited
for (const k in dog) { }                      // ⚠ includes inherited enumerables
Object.keys(dog)                              // own keys only — usually what you meant
💡 Production tip: never mutate built-in prototypes (Array.prototype.myHelper = …). It's global, collides with future language features, and once broke the web badly enough that TC39 had to rename a proposed method (the "SmooshGate" incident). Extend via plain utility functions instead.
