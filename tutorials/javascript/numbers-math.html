Numbers, BigInt & Precision Language Core

What you'll master: why JavaScript has exactly one number type and what that costs you (floating-point rounding, the 2⁵³ integer cliff), how to do money and large-integer math without corruption, when BigInt is mandatory and its sharp edges, and the Math / Number / Intl.NumberFormat tools that turn raw numbers into correct results and correct displays. Prerequisite: The Type System & Coercion (where number and NaN were introduced). This is the lesson that keeps you from shipping a payments bug.

Part 1 — One Number Type: IEEE-754 Double

42          // integer? No — it's a 64-bit floating-point double, same as:
42.0        // identical value
3.14
1e3         // 1000
0xff        // 255  (hex literal)
0b1010      // 10   (binary literal)
0o17        // 15   (octal literal)
1_000_000   // 1000000 — underscores are just readability separators, ignored by the engine

Every JavaScript number — whether it looks like an integer or a decimal — is stored as a 64-bit IEEE-754 double-precision float. There is no separate integer type (that's what BigInt is for, Part 5). A double splits its 64 bits into 1 sign bit, 11 exponent bits, and 52 fraction (mantissa) bits — and those finite bits are the source of both famous problems: decimals that can't be represented exactly (Part 2) and integers too big to be exact (Part 4).

Part 2 — Floating-Point: Why 0.1 + 0.2 !== 0.3

0.1 + 0.2                 // 0.30000000000000004
0.1 + 0.2 === 0.3         // false
(0.1 + 0.2).toFixed(2)    // "0.30"  — rounds for DISPLAY, returns a STRING
0.3 - 0.1                 // 0.19999999999999998

Doubles store numbers in binary fractions. Just as 1/3 is a non-terminating decimal (0.333…), 0.1 and 0.2 are non-terminating in binary, so the engine stores the nearest representable value to each. Their stored approximations sum to a value whose nearest representation is 0.30000000000000004, not 0.3. This is not a JavaScript bug — it's IEEE-754, identical in Java, C, Python floats, and every other language using doubles. Two consequences you must internalise:

// 1. NEVER compare floats with === — compare within a tolerance (epsilon):
function almostEqual(a, b, epsilon = Number.EPSILON) {
  return Math.abs(a - b) < epsilon;
}
almostEqual(0.1 + 0.2, 0.3);   // true

// 2. NEVER store money as a float:
const price = 0.1 + 0.2;        // 0.30000000000000004 — a cent could vanish or appear
🧠 Number.EPSILON is the smallest gap between 1 and the next representable double (~2.2e-16) — a reasonable tolerance for values near 1. For larger magnitudes you must scale it (epsilon * Math.max(|a|, |b|)) because the gap between representable doubles grows with magnitude. There is no single universal epsilon.

Part 3 — Money: Integer Minor Units

The professional rule: store money as an integer number of the smallest currency unit (cents), never as a decimal. $19.99 is stored as 1999 cents — an exact integer (well within the safe range) — and you only convert to a decimal at the very last moment, for display, using Intl.NumberFormat.

// Store & compute in integer cents — always exact:
const priceCents = 1999;            // $19.99
const taxCents   = Math.round(priceCents * 0.08);   // round at each boundary you leave integer-land
const totalCents = priceCents + taxCents;           // 2159 — exact integer addition

// Format for display ONLY at the edge:
const usd = new Intl.NumberFormat("en-US", { style: "currency", currency: "USD" });
usd.format(totalCents / 100);       // "$21.59"

Every arithmetic step stays in exact integer space; the only division-by-100 happens at display, where Intl.NumberFormat handles rounding, currency symbol, thousands separators, and locale conventions correctly (it even does €, ¥ with no decimals, etc.). For very large sums or high-precision financial math beyond safe-integer range, use BigInt cents or a decimal library (decimal.js, dinero.js). The one thing you never do is price + tax on floating dollars.

Part 4 — The 2⁵³ Integer Cliff

Number.MAX_SAFE_INTEGER      // 9007199254740991  (2^53 − 1)
Number.MAX_SAFE_INTEGER + 1  // 9007199254740992   — still fine
Number.MAX_SAFE_INTEGER + 2  // 9007199254740992   — WRONG! same as +1; precision lost
9007199254740993 === 9007199254740992   // true 😱 — two different integers, one representation
Number.isSafeInteger(2 ** 53)           // false

With only 52 mantissa bits, doubles can represent every integer exactly only up to 2⁵³ − 1 (Number.MAX_SAFE_INTEGER). Beyond that, consecutive integers start sharing representations — you literally cannot store 2⁵³ + 1 distinctly from 2⁵³. This is why 64-bit database IDs, Twitter/Discord snowflake IDs, and large counters must be strings or BigInt, never plain numbers: parse a 64-bit ID into a JS number and it silently corrupts. JSON.parse does exactly this by default, which is a real, shipped-in-production bug class — many APIs send big IDs as strings specifically to avoid it.

Part 5 — BigInt: Arbitrary-Precision Integers

const big = 9007199254740993n;        // the `n` suffix makes a BigInt — exact, unbounded
const also = BigInt("9007199254740993");
big + 1n                               // 9007199254740994n — exact, unlike the number version
2n ** 64n                              // 18446744073709551616n — arbitrary precision

// The sharp edges:
1n + 1        // ❌ TypeError: Cannot mix BigInt and other types — convert explicitly
1n + BigInt(1)// 2n
Number(9007199254740993n)  // 9007199254740992 — converting BACK to number re-loses precision
typeof 1n     // "bigint"
1n == 1       // true  (loose ==  coerces)
1n === 1      // false (different types)
JSON.stringify({x: 1n})    // ❌ TypeError: Do not know how to serialize a BigInt
Math.max(1n, 2n)           // ❌ TypeError — Math.* rejects BigInt

BigInt represents integers of arbitrary size exactly. Use it for: 64-bit IDs you must do math on, cryptographic/precise integer computation, and large counters. But respect its edges: you cannot mix BigInt and number in arithmetic (explicit conversion required), JSON.stringify throws on it (serialise as a string yourself), Math.* rejects it, and it's integers only — no fractions. Because of the friction, many teams handle large IDs as opaque strings and only reach for BigInt when they must actually compute.

Part 6 — The Number Static Toolkit

Number.isInteger(5.0)        // true       — is it a whole number?
Number.isSafeInteger(2**53)  // false      — whole AND within ±(2^53−1)?
Number.isFinite(x)           // true unless Infinity/-Infinity/NaN (no coercion, unlike global isFinite)
Number.isNaN(x)              // NaN test without coercion (Types lesson)
Number.parseFloat / parseInt // same as the globals; namespaced versions
Number.MAX_SAFE_INTEGER      // 2^53 − 1
Number.EPSILON               // smallest representable gap near 1
Number.MAX_VALUE             // ~1.8e308; beyond it → Infinity
(1234.5678).toFixed(2)       // "1234.57"  — fixed decimals, returns STRING, rounds
(0.00001234).toExponential(2)// "1.23e-5"
(255).toString(16)           // "ff"       — radix conversion
(3.14159).toPrecision(3)     // "3.14"     — significant figures

Prefer the Number.* namespaced predicates (Number.isNaN, Number.isFinite) over the global isNaN/isFinite, because the globals coerce their argument first (isNaN("abc") is true) while the namespaced ones don't — matching the "no implicit coercion" discipline. Note every toFixed/toPrecision/toExponential returns a string, for display; don't feed their output back into arithmetic with + (concatenation trap).

Part 7 — Math and the Rounding Family

Math.round(2.5)    // 3    — .5 rounds toward +Infinity (so -2.5 → -2, not -3!)
Math.round(-2.5)   // -2   — the asymmetry that surprises people
Math.floor(2.9)    // 2    — toward -Infinity
Math.ceil(2.1)     // 3    — toward +Infinity
Math.trunc(-2.9)   // -2   — just drops the fraction (toward zero)
Math.floor(-2.9)   // -3   — DIFFERENT from trunc for negatives!
Math.sign(-5)      // -1
Math.abs(-5)       // 5
Math.max(1,2,3)    // 3    (Math.max() with no args is -Infinity; Math.max(...arr) to spread)
Math.hypot(3,4)    // 5
Math.random()      // [0, 1) — NOT cryptographically secure (use crypto.getRandomValues for that)

The rounding subtleties bite in real code. Math.round rounds .5 toward positive infinity, so Math.round(-2.5) is -2, not -3 — asymmetric. For negatives, Math.floor (toward −∞) and Math.trunc (toward 0) diverge: Math.floor(-2.9) is -3 but Math.trunc(-2.9) is -2. Pick deliberately based on whether you want "largest integer ≤ x" (floor) or "drop the decimal part" (trunc). And Math.random() is fine for shuffles and jitter but never for tokens, passwords, or anything security-sensitive — use crypto.getRandomValues() (Security lesson).

Part 8 — Build It: a Correct Currency Type

Let's assemble Parts 3, 6, and 7 into a tiny Money helper that stores integer cents, does exact arithmetic, rounds correctly at the tax boundary, and formats per-locale — the shape of a real (if minimal) money value.

const Money = {
  fromDollars: (d) => Math.round(d * 100),            // "19.99" → 1999 cents, rounding the fp noise
  add: (a, b) => a + b,                               // exact: integer addition
  applyRate: (cents, rate) => Math.round(cents * rate), // round at each fractional boundary
  format: (cents, locale = "en-US", currency = "USD") =>
    new Intl.NumberFormat(locale, { style: "currency", currency }).format(cents / 100),
};

const subtotal = Money.add(Money.fromDollars(19.99), Money.fromDollars(5.01)); // 2500
const tax      = Money.applyRate(subtotal, 0.0875);   // Math.round(218.75) = 219
const total    = Money.add(subtotal, tax);            // 2719
Money.format(total);                 // "$27.19"
Money.format(total, "de-DE", "EUR"); // "27,19 €"  — locale-correct symbol, separator, placement

Every value in flight is an exact integer; the only two places we touch floats are the deliberate, rounded boundaries (fromDollars parsing user input, applyRate computing tax) — and each is immediately snapped back to an integer with Math.round. Display goes through Intl.NumberFormat, which is locale-correct for free. This is the entire recipe for money in JS: integers in the middle, controlled rounding at the edges, Intl at display. For anything beyond a toy, use a maintained library (dinero.js) that does exactly this, hardened.

Interview Questions

Q1 (beginner): Why does 0.1 + 0.2 not equal 0.3?

JavaScript numbers are IEEE-754 doubles, which store values as binary fractions. 0.1 and 0.2 are non-terminating in binary, so each is stored as the nearest representable approximation. The sum of those approximations is closest to 0.30000000000000004, not exactly 0.3. It's inherent to binary floating-point, not a JS-specific bug — it happens identically in Java, C, and Python floats. The practical takeaways: compare floats with a tolerance, and never store money as a float.

Q2 (beginner): How should you store and compute monetary amounts in JavaScript?

As integers in the smallest currency unit — cents for USD — so all arithmetic is exact integer math within the safe range. $19.99 is 1999. Add, subtract, and (with a rounding step) apply rates entirely in integer cents. Convert to a decimal string only at display time using Intl.NumberFormat with { style: "currency", currency }, which handles rounding, symbol placement, and locale separators. For amounts beyond safe-integer range or requiring high precision, use BigInt cents or a decimal library. Never do arithmetic on floating-point dollar values.

Q3 (mid): What is Number.MAX_SAFE_INTEGER and why does it matter for API design?

It's 2⁵³−1 (9007199254740991) — the largest integer a double can represent exactly. Beyond it, consecutive integers start colliding onto the same representation, so 2⁵³+1 can't be stored distinctly from 2⁵³. It matters because 64-bit identifiers (database bigints, snowflake IDs) exceed it, and if such an ID is delivered as a JSON number, JSON.parse silently rounds it, corrupting the ID. The fix in API design is to serialise large IDs as strings (or use BigInt with a custom serialiser), so the client never parses them into a lossy double. Always check Number.isSafeInteger when accepting integer IDs from untrusted numeric sources.

Q4 (mid): When would you use BigInt, and what are its three biggest gotchas?

Use it when you need exact integer math on values beyond 2⁵³−1: doing arithmetic on 64-bit IDs, large counters, cryptographic integer computation. Gotchas: (1) you can't mix BigInt with number in an operation — 1n + 1 throws TypeError; you must convert one side explicitly, and converting a BigInt back to Number re-introduces the precision loss you used BigInt to avoid. (2) JSON.stringify throws on BigInt, so you must serialise it as a string manually. (3) It's integers only (no fractions) and Math.* rejects it. Because of the friction, large IDs are often kept as opaque strings unless you genuinely need to compute with them.

Q5 (senior): Explain the difference between Math.floor, Math.trunc, and Math.round for negative numbers, and give a case where choosing wrong causes a bug.

Math.floor rounds toward −∞: floor(-2.1) = -3. Math.trunc drops the fractional part (toward 0): trunc(-2.1) = -2. Math.round rounds to nearest with ties going toward +∞: round(-2.5) = -2, which is asymmetric with round(2.5) = 3. A concrete bug: computing a zero-based grid cell index from a negative coordinate with Math.trunc(y / cellSize) when you meant Math.floor — for y = -0.5, trunc gives cell 0 but the point is actually in cell -1, so trunc mis-buckets everything just below an axis. Pagination, histogram bucketing, and coordinate-to-tile mapping all want floor (consistent stepping across zero), while "discard the decimals" display logic wants trunc. Another: financial rounding that assumes symmetric round-half-up will systematically bias negatives with Math.round — banker's rounding or an explicit sign-aware round is needed there.

Q6 (senior): A service receives JSON like {"id": 9007199254740993, "balance": 10.10} and reports occasional wrong IDs and off-by-a-cent balances. Diagnose both and give a robust ingestion design.

Two independent floating-point defects. The id exceeds Number.MAX_SAFE_INTEGER, so JSON.parse rounds it to 9007199254740992 — the client can never see the true ID, producing "wrong ID" reports for exactly the records with large IDs. The balance arrives as a float, so 10.10 is stored as its nearest double and any subsequent arithmetic accumulates the representation error into off-by-a-cent totals. Robust ingestion: (1) For IDs, don't let JSON.parse coerce them — either have the API emit IDs as JSON strings and keep them as strings/BigInt end to end, or parse with a reviver / a JSON parser that preserves large integers as BigInt; validate with Number.isSafeInteger and reject/flag anything unsafe at the boundary. (2) For money, treat the incoming decimal as untrusted display data: parse it into integer minor units immediately (Math.round(value * 100) at the boundary, or better, have the API send integer cents directly), do all arithmetic in integer cents, and format back with Intl.NumberFormat only at output. The unifying principle: floating-point corruption must be prevented at the ingestion boundary, because once a large ID or a money value has been parsed into a lossy double, the information is already gone and no downstream code can recover it.

🏋️ Exercises

1 (easy): Epsilon compare. Write almostEqual(a, b) that returns true for 0.1 + 0.2 vs 0.3 but false for 0.3 vs 0.31. Scale the tolerance to magnitude.
Solution 
const almostEqual = (a, b, eps = Number.EPSILON) =>
  Math.abs(a - b) <= eps * Math.max(1, Math.abs(a), Math.abs(b));
almostEqual(0.1 + 0.2, 0.3);  // true
almostEqual(0.3, 0.31);       // false

Scaling by Math.max(1, |a|, |b|) keeps the tolerance meaningful for large magnitudes, where the gap between representable doubles is much bigger than Number.EPSILON.

2 (easy): Safe integer guard. Write isSafeId(n) that returns true only for integers that a JS number can represent exactly. Test it on 2**53 and 2**53 - 1.
Solution 
const isSafeId = n => Number.isSafeInteger(n);
isSafeId(2**53 - 1);  // true
isSafeId(2**53);      // false — first unsafe integer

Number.isSafeInteger checks both "is a whole number" and "within ±(2⁵³−1)" without coercion — exactly the guard for accepting numeric IDs.

3 (medium): Cents arithmetic. A cart has items priced (in dollars) [19.99, 5.00, 3.33] with 8.5% tax. Compute the total correctly (to the cent) using integer cents, and format it as USD.
Solution 
const cents = [19.99, 5.00, 3.33].map(d => Math.round(d * 100)); // [1999, 500, 333]
const subtotal = cents.reduce((a, b) => a + b, 0);              // 2832
const tax = Math.round(subtotal * 0.085);                        // Math.round(240.72) = 241
const total = subtotal + tax;                                    // 3073
new Intl.NumberFormat("en-US", {style:"currency", currency:"USD"}).format(total/100); // "$30.73"

Each dollar amount is snapped to integer cents at the boundary; the subtotal and tax stay integer; only the final /100 for display touches a float, immediately consumed by Intl.NumberFormat.

4 (medium): BigInt round-trip. Show that a 64-bit ID "18014398509481985" survives as a BigInt but is corrupted if routed through Number. Then serialise a BigInt to JSON without throwing.
Solution 
const id = BigInt("18014398509481985");
id.toString();              // "18014398509481985" — exact
Number(id).toString();      // "18014398509481984" — corrupted (lost the last digit)

// JSON-safe: serialise BigInt as a string
JSON.stringify({ id }, (k, v) => typeof v === "bigint" ? v.toString() : v);
// '{"id":"18014398509481985"}'

The Number(id) conversion drops precision because the value exceeds 2⁵³. A JSON.stringify replacer that converts BigInts to strings avoids the default TypeError and keeps the value intact on the wire.

5 (hard): Rounding-mode bug. A billing system uses Math.round to round half-cents. Show that for the set of amounts [0.5, 1.5, 2.5, -0.5, -1.5] cents, Math.round is asymmetric, and implement round-half-away-from-zero so positives and negatives are treated symmetrically.
Solution 
[0.5, 1.5, 2.5, -0.5, -1.5].map(Math.round);   // [1, 2, 3, 0, -1]  — note -0.5→0, -1.5→-1
// Math.round ties always go toward +∞, so negatives round "up" toward zero — asymmetric.

const roundHalfAway = x => Math.sign(x) * Math.round(Math.abs(x));
[0.5, 1.5, 2.5, -0.5, -1.5].map(roundHalfAway); // [1, 2, 3, -1, -2]  — symmetric

By rounding the absolute value and reapplying the sign, ties always move away from zero regardless of sign — the symmetric behaviour billing usually wants. (For strict accounting, banker's rounding / round-half-to-even reduces cumulative bias further.)

6 (hard): Parse big IDs from JSON safely. Given a JSON string '{"id": 9007199254740993, "name": "x"}', parse it so that id is preserved exactly (not rounded), returning it as a string. Hint: you can't fix it after JSON.parse — intercept during parsing.
Solution 
// The value is already lost after a plain JSON.parse, so operate on the raw text.
// Simplest robust approach: quote large integers before parsing, via a reviver-friendly transform.
function parsePreservingBigIds(json) {
  // wrap integer literals that are too long to be safe in quotes so they parse as strings
  const safe = json.replace(/:\s*(-?\d{16,})(?=[,}\]])/g, ': "$1"');
  return JSON.parse(safe);
}
parsePreservingBigIds('{"id": 9007199254740993, "name": "x"}');
// { id: "9007199254740993", name: "x" }  — exact

The key insight from Part 4: once JSON.parse has coerced the literal into a double, the precision is already gone — so you must intervene on the text before parsing (or use a BigInt-aware JSON parser). The regex quotes any 16+ digit integer so it parses as a string. In production, prefer a library like json-bigint or have the API emit IDs as strings in the first place.

🎯 Key Takeaways

  • One number type: 64-bit IEEE-754 double — no separate integer type; that's why both float rounding and the 2⁵³ cliff exist.
  • 0.1 + 0.2 !== 0.3 is inherent binary floating-point — compare floats within a magnitude-scaled epsilon, never with ===.
  • Money = integer minor units (cents) for exact arithmetic; Intl.NumberFormat for locale-correct display only at the edge.
  • Integers are exact only to Number.MAX_SAFE_INTEGER (2⁵³−1) — 64-bit IDs must be strings or BigInt, and JSON.parse corrupts them silently.
  • BigInt gives exact arbitrary-precision integers but can't mix with number, throws in JSON.stringify, and is rejected by Math.
  • Know your rounding: floor (→−∞), trunc (→0), round (nearest, ties→+∞, asymmetric for negatives); use Number.isNaN/isFinite and crypto for secure randomness.
