TypeScript Essentials Production JavaScript

What you'll master: TypeScript as a structural, compile-time-only type layer over JavaScript — basic and object types, interfaces vs type aliases, generics, union/narrowing (the feature that makes TS feel different from other typed languages), utility types, and the honest trade-offs of adopting it. Prerequisites: everything so far — TypeScript adds types to JavaScript's existing semantics, it doesn't change them. This lesson gives you enough fluency to read, write, and reason about typed codebases, which is most professional JS work today.

Part 1 — TypeScript Is JavaScript, Plus Types, Erased at Compile Time

// TypeScript (.ts) — add type ANNOTATIONS to ordinary JS:
function add(a: number, b: number): number {
  return a + b;
}
add(1, "2");   // ❌ compile-time ERROR: Argument of type 'string' is not assignable to 'number'

// The compiler STRIPS all types, producing plain runnable JS — types have ZERO runtime cost/presence:
// compiles to:
function add(a, b) { return a + b; }
// typeof at runtime, instanceof, etc. know NOTHING about TS types — they never existed at runtime

TypeScript is a superset of JavaScript: every valid JS file is (almost) valid TS, and TS adds optional type annotations that the compiler checks before your code runs, then erases entirely — the compiled output is plain JavaScript with zero trace of types. This is the single most important mental model: TypeScript catches a whole class of bugs (wrong argument types, typos in property names, forgetting a null check) at compile time, for free, but it provides zero runtime safety — a value from JSON.parse or an untyped third-party call can violate your types at runtime with no error, which is why runtime validation at trust boundaries (the Types lesson's normalization idea) is still necessary even in a fully-typed codebase.

Part 2 — Basic Types & Type Inference

let name: string = "Ada";
let age: number = 30;
let active: boolean = true;
let tags: string[] = ["a", "b"];        // array of strings
let tuple: [string, number] = ["x", 1]; // FIXED-length, fixed-type-per-position array
let anything: any = "danger";            // opts OUT of type checking — avoid (Part 7)
let safer: unknown = fetchSomething();   // like any, but FORCES a check before use (the safe "any")

// TypeScript INFERS types — you rarely annotate everything explicitly:
let inferred = "hello";        // TS infers: string — annotating `: string` here is redundant
function double(n: number) {   // return type INFERRED as number — no need to write it
  return n * 2;
}

Basic types map directly to JS's runtime types, plus TS-specific additions: tuples (fixed-length arrays with a specific type per position — great for functions returning multiple values), and the crucial pair any vs unknown. any completely disables type checking for that value — a checked-out escape hatch you should minimize. unknown is the type-safe alternative: it also accepts anything, but the compiler forces you to narrow/check it before you can use it for anything specific — the correct type for "I don't know this value's shape yet" (like a JSON response). TypeScript's inference is powerful enough that you rarely need to annotate every variable — let the compiler infer from the initializer and only annotate function parameters (which it can't infer) and public API boundaries.

Part 3 — Object Types & Interfaces

interface User {
  id: number;
  name: string;
  email?: string;          // OPTIONAL property (may be undefined)
  readonly createdAt: Date; // cannot be reassigned after creation (compile-time check, not runtime freeze)
}

function greet(user: User): string {
  return `Hi, ${user.name}`;
}
greet({ id: 1, name: "Ada", createdAt: new Date() });   // email omitted — fine, it's optional
greet({ id: 1, name: "Ada" });   // ❌ error: missing createdAt

// type aliases can describe the SAME shape:
type UserAlias = { id: number; name: string };
// interfaces can be REOPENED/extended; type aliases (mostly) cannot — the main practical difference:
interface User { role?: string; }   // ✅ merges into the earlier `interface User` declaration
// type UserAlias = { role?: string };  // ❌ error: duplicate identifier

An interface describes the shape of an object — its properties and their types. ? marks a property optional; readonly prevents reassignment (a compile-time-only guarantee, unlike Object.freeze's runtime enforcement — the Objects lesson's distinction, now at the type level). TypeScript's type system is structural (also called "duck typing"): a value satisfies an interface if it has the right shape, regardless of how it was created or what it's explicitly declared as — unlike nominally-typed languages (Java/C#) where you must explicitly implement an interface. This is a deep and consequential difference covered fully in Part 8. type aliases can describe the same object shapes as interfaces; the practical difference is that interfaces support declaration merging (reopening to add more members) while type aliases don't — a real but often minor distinction (most style guides default to one or the other by convention).

Part 4 — Union Types & Narrowing (What Makes TS Feel Different)

type Status = "loading" | "success" | "error";   // a UNION of literal string types — an enum-like set
function render(status: Status) {
  if (status === "loading") return "Spinner";
  if (status === "success") return "Data";
  return "Error message";                          // TS knows this branch is only reachable for "error"
}

// NARROWING — TS tracks which branch you're in and refines the type accordingly:
function process(value: string | number) {
  if (typeof value === "string") {
    value.toUpperCase();   // TS KNOWS value is `string` HERE — full string method autocomplete
  } else {
    value.toFixed(2);      // TS KNOWS value is `number` HERE — no need to re-check
  }
}

function isUser(x: unknown): x is User {    // a TYPE GUARD — a custom narrowing function
  return typeof x === "object" && x !== null && "id" in x && "name" in x;
}
if (isUser(data)) { data.name; }   // TS trusts the guard and narrows `data` to `User` here

This is the feature that makes TypeScript feel qualitatively different from a Java/C#-style type system. A union type (A | B) says a value could be one of several types. Narrowing is TypeScript's static analysis of your control flow: after a typeof/instanceof/equality check, the compiler knows which branch you're in and refines the type accordingly — inside the if (typeof value === "string") branch, value is fully and correctly typed as string, with no manual casting. A type guard (a function returning x is SomeType) lets you teach the compiler your own custom narrowing logic — essential for validating unknown data (like parsed JSON) into a trusted, specific type. This flow-sensitive typing is what makes modeling real, branch-heavy application logic in TS feel natural rather than fighting the type system.

Part 5 — Generics: Types Parameterized by Types

function firstElement<T>(arr: T[]): T | undefined {   // T is a TYPE PARAMETER — filled in at call time
  return arr[0];
}
firstElement([1, 2, 3]);       // T inferred as number  → return type: number | undefined
firstElement(["a", "b"]);      // T inferred as string  → return type: string | undefined

interface Box<T> { value: T; }
const numberBox: Box<number> = { value: 42 };
const stringBox: Box<string> = { value: "hi" };

// Constraining a generic — T must have a `.length`:
function logLength<T extends { length: number }>(item: T): void {
  console.log(item.length);
}
logLength("hello");   // ✅ strings have .length
logLength([1,2,3]);   // ✅ arrays have .length
logLength(42);        // ❌ numbers don't have .length

Generics let a function or type be written once and work correctly for many concrete types, without losing type information — this is the crucial difference from using any, which also "works for anything" but throws away all type safety. firstElement<T> preserves the relationship "whatever type went in, that (or undefined) comes out" — call it with number[] and TS knows the result is number | undefined, with full type checking downstream. This is exactly how the built-in generic types you already use work: Array<T>, Promise<T>, Map<K, V> are all generics. You can constrain a type parameter (T extends {...}) to require it has certain properties, letting you write reusable code that still safely accesses what it needs.

Part 6 — Useful Utility Types

interface User { id: number; name: string; email: string; }

Partial<User>         // { id?: number; name?: string; email?: string; } — all optional (patch/update payloads)
Required<User>        // opposite — all REQUIRED, even if the source had optionals
Readonly<User>        // all properties readonly
Pick<User, "id"|"name">  // { id: number; name: string; } — a SUBSET of properties
Omit<User, "email">      // { id: number; name: string; } — everything EXCEPT the listed keys
Record<string, number>   // { [key: string]: number } — a dictionary type

function updateUser(id: number, patch: Partial<User>) { /* only changed fields required */ }
type PublicUser = Omit<User, "email">;   // reuse User's shape, minus a sensitive field

Utility types transform existing types instead of making you redeclare similar shapes by hand — a direct type-level parallel to the Objects lesson's pick/omit runtime functions. Partial<T> is exactly what an "update"/"patch" function's argument type usually needs (any subset of fields). Pick/ Omit derive a narrower type from an existing one — reusing User's definition for a "public-safe" version rather than duplicating and risking drift between two hand-maintained shapes. Reaching for these instead of redefining similar interfaces from scratch is a mark of idiomatic TypeScript — types, like code, should follow DRY.

Part 7 — Type Assertions, any, and Honest Escape Hatches

const el = document.getElementById("app") as HTMLDivElement;   // "trust me" — NOT a runtime check/conversion
// If it's actually a <span>, TS won't catch it; using it as a div-only method will FAIL AT RUNTIME

const data: any = JSON.parse(text);   // ❌ silences ALL checking on everything `data` touches downstream
const data2: unknown = JSON.parse(text);   // ✅ forces validation before use (Part 4's type guards)

// The NON-NULL assertion (!) — "I know this isn't null/undefined, trust me":
function getUser(id: number): User | undefined { /* ... */ }
const user = getUser(1)!;   // asserts non-undefined — WRONG if it actually is, causes a runtime crash later

Type assertions (as Type) tell the compiler "trust me, I know this is actually this type" — but unlike a real type check, an assertion does zero runtime verification; if you're wrong, TS won't catch it and you get a runtime error somewhere downstream instead of a compile error where the mistake actually is. The non-null assertion (!) is the same idea applied to nullability. Both are necessary escape hatches for cases TS genuinely can't infer (a DOM query you know succeeds, a value you've validated some other way) — but each one is a place you've told the compiler to stop checking, so minimize them and prefer real narrowing/validation (Part 4) wherever possible. any is the broadest escape hatch and should be rare in a well-typed codebase — reach for unknown instead whenever you can.

Part 8 — Structural Typing: the Deep Difference from Java/C#

interface Point { x: number; y: number; }

function logPoint(p: Point) { console.log(p.x, p.y); }

const literal = { x: 1, y: 2, z: 3 };   // has an EXTRA property, z — not declared as a Point anywhere
logPoint(literal);                        // ✅ WORKS — it structurally HAS x and y; that's all that matters

class Vector3 { x = 0; y = 0; z = 0; }    // never declared to "implement Point"
logPoint(new Vector3());                   // ✅ also works — structural match, no explicit relationship needed

// BUT a fresh object LITERAL is checked more strictly ("excess property checking"):
logPoint({ x: 1, y: 2, z: 3 });   // ❌ error — literal has an unexpected extra property `z`
// (assigning through a variable first, as `literal` above, bypasses this extra check — a known quirk)

TypeScript's type system is structural, not nominal. In a nominally- typed language, a value only satisfies an interface if it was explicitly declared to implement it. In TypeScript, any value with the right shape satisfies the type, regardless of its declared origin — a Vector3 instance that happens to have x and y numeric properties is a valid Point, with no implements keyword needed anywhere. This is a deliberate design choice matching JavaScript's own duck-typed runtime behavior, and it's what makes TS feel natural for JS developers rather than bolted-on. The one quirk to know: TypeScript applies stricter "excess property checking" specifically to fresh object literals passed directly (catching likely typos, like naem: "Ada", that structural typing alone would silently allow) — but this extra check doesn't apply once the value has been assigned to a variable first, which surprises people expecting it to be a hard rule about structural typing itself.

Part 9 — Build It: a Typed API Client with Runtime Validation

Types alone don't protect you at runtime (Part 1) — let's build a small client that combines TypeScript's compile-time guarantees with a runtime type guard at the actual trust boundary, the correct pattern for consuming external data safely.

interface User {
  id: number;
  name: string;
  email: string;
}

// A runtime TYPE GUARD — the ONLY thing that can actually verify unknown, external data:
function isUser(x: unknown): x is User {
  return (
    typeof x === "object" && x !== null &&
    typeof (x as any).id === "number" &&
    typeof (x as any).name === "string" &&
    typeof (x as any).email === "string"
  );
}

async function fetchUser(id: number): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  const data: unknown = await res.json();          // json() returns `any` by default — widen to unknown
  if (!isUser(data)) throw new TypeError("API response did not match the User shape");
  return data;                                       // TS now trusts `data` is User, verified at runtime
}

// A PATCH function using Partial<T> — only changed fields required:
async function updateUser(id: number, patch: Partial<Omit<User, "id">>): Promise<User> {
  const res = await fetch(`/api/users/${id}`, {
    method: "PATCH",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(patch),
  });
  const data: unknown = await res.json();
  if (!isUser(data)) throw new TypeError("Invalid response");
  return data;
}
updateUser(1, { name: "Ada Lovelace" });   // ✅ only name changes — id excluded by Omit, email untouched

This encapsulates the honest relationship between TypeScript and runtime safety: the compiler protects call sites (you can't call fetchUser("bad") or forget to handle the returned User shape correctly downstream), but the network response itself is untyped external data until isUser actually verifies it at runtime — treating fetch().json()'s result as unknown rather than trusting a type annotation forces that verification. updateUser's signature (Partial<Omit<User, "id">>) precisely expresses "any subset of User's fields except id" using two utility types composed together — exactly the kind of precise, DRY API contract Part 6 introduced. This is what "properly typed" production code looks like: strong compile-time contracts plus real validation at the one place types can't help you — the trust boundary.

Interview Questions

Q1 (beginner): Does TypeScript provide any runtime type checking?

No. TypeScript's types exist only at compile time — the compiler checks your code against them, then erases all type annotations, producing plain JavaScript with zero trace of types at runtime. This means typeof, instanceof, and every runtime mechanism know nothing about TS types, and any value from outside your typed code (an API response, JSON.parse, user input, an untyped library) can violate your declared types with no error — the type system will have "believed" it was correctly shaped based only on your annotation, not any actual verification. This is why runtime validation at trust boundaries is still necessary even in a fully-typed codebase — types catch bugs in code you control; they cannot verify data you don't.

Q2 (beginner): What's the difference between any and unknown?

Both can hold a value of any type, but any completely disables type checking — once a value is any, you can call any method or access any property on it with no compiler complaint, silently propagating "no safety" to everything downstream that touches it. unknown also accepts any value, but the compiler forces you to narrow it (with typeof, instanceof, or a type guard) before you can use it for anything specific — you can't call a method or access a property on an unknown value until you've proven what it actually is. unknown is the type-safe way to represent "I don't know this value's shape yet" (like a JSON response), and should be preferred over any almost everywhere.

Q3 (mid): Explain type narrowing with an example.

Narrowing is TypeScript's static analysis of your control flow: after a runtime check like typeof, instanceof, an equality comparison, or a custom type guard, the compiler refines a value's type within that branch to reflect what the check proved. For example, given function f(x: string | number), inside if (typeof x === "string") { ... } TypeScript treats x as fully and precisely typed string in that block — string methods autocomplete and type-check correctly, with no manual casting — while the else branch narrows x to number. This flow-sensitive tracking is what lets you write natural, branch-heavy conditional logic on union types and have the compiler understand exactly what's true in each branch, rather than needing to manually assert types everywhere.

Q4 (mid): What are generics and how do they differ from using any?

Generics let a function or type be parameterized by a type variable (T), so it works correctly for many concrete types while preserving the relationship between input and output types. function first<T>(arr: T[]): T | undefined called with number[] is known by the compiler to return number | undefined; called with string[], it returns string | undefined — full type safety is maintained through the generic. any also "works for anything," but by discarding all type information — a function typed to take and return any gives you no compiler help at all, and callers get no useful type back regardless of what they passed in. Generics give you both flexibility (works for many types) and safety (the specific type used is tracked and checked); any gives you flexibility at the total cost of safety. Built-ins like Array<T>, Promise<T>, and Map<K,V> are all generics.

Q5 (senior): Explain structural typing and a concrete situation where it causes a surprising result compared to nominal typing.

TypeScript's type system is structural: a value satisfies a type if it has the required shape (properties and their types), regardless of how it was created, what class it's an instance of, or whether it was ever explicitly declared to implement that type — this mirrors JavaScript's own duck-typed runtime semantics. A concrete surprise: given interface Auth { token: string } and interface Coordinates { token: string } where token happens to mean completely different things in each domain (an auth token vs., implausibly, a map coordinate label), a value intended as one can be silently accepted where the other is expected, because structurally they're identical — TypeScript has no concept of these being semantically distinct "kinds" of string, only that both have a token: string property. A more common real version: two unrelated interfaces that happen to share a shape (e.g. both have { id: number; name: string }) are freely interchangeable even though domain logic treats them as fundamentally different entities (a Product vs a Category). Nominal typing would prevent this by requiring explicit conformance declarations, catching the mix-up at the type level. TypeScript's mitigation for cases where you specifically want nominal-like distinctness is the "branded type" pattern — adding a unique, unused marker property (type UserId = number & { readonly __brand: "UserId" }) so structurally-identical-but-semantically-different types can no longer be assigned to each other by accident, at the cost of needing an explicit cast to construct one. This is a deliberate, occasionally necessary opt-in to nominal-like safety within an otherwise structural system.

Q6 (senior): You're advocating for adopting TypeScript on an existing large JavaScript codebase. What's the honest cost/benefit case, and how would you roll it out?

Benefits: compile-time catching of a real, common bug class (wrong argument types, typos in property/method names, forgetting to handle a case in a union, null/undefined mistakes) — empirically a large fraction of production JS bugs are exactly these "type errors" that TS eliminates before runtime; dramatically better editor tooling (accurate autocomplete, safe rename-refactoring, jump-to-definition that actually works across a large codebase, inline documentation via types); and self-documenting function signatures that reduce the need for separate API docs and make onboarding faster. Costs, honestly stated: a real learning curve for the team (generics, narrowing, utility types take time to use well, not just tolerate); build tooling complexity (a compile step, config, potential friction with existing JS-only tooling); the migration itself is nontrivial for a large codebase — a big-bang rewrite is usually impractical and risky, and even incremental adoption has friction around untyped third-party dependencies (needing @types packages, which vary in quality/completeness, or writing your own declarations); and TypeScript gives zero runtime safety — it doesn't reduce the need for runtime validation at trust boundaries, so teams sometimes over-trust their types and under-validate external data, actually introducing a false sense of security if adopted carelessly. Rollout strategy: (1) TypeScript is designed for incremental adoption — enable it with allowJs so existing .js files keep working unmodified while new files can be .ts; convert file-by-file rather than all at once, prioritizing files with high bug rates or complex data shapes (API clients, state management) where types add the most value first. (2) Start with a lenient compiler config (strict: false) to avoid an avalanche of errors on day one, then incrementally tighten (strictNullChecks, then full strict) as the codebase converts, rather than demanding full strictness immediately. (3) Type the most-imported, most "central" modules first (shared utilities, API layer, core data models) since typing them gives downstream consumers useful types even before those consumers are themselves converted. (4) Keep runtime validation (the type-guard/schema-validation pattern from this lesson) at every external trust boundary regardless of TS adoption — that discipline doesn't go away. (5) Budget real time for the learning curve and expect initial velocity to dip before it recovers and then exceeds the pre-TypeScript baseline, and communicate that honestly rather than overselling an immediate win. The overall case: strongly positive for any codebase expected to be maintained and grown by more than one person for more than a few months, with the caveat that the migration itself has a real, non-trivial cost that should be planned and communicated rather than understated.

🏋️ Exercises

1 (easy): Type a function. Add proper parameter and return types to a function computing the area of a rectangle.
Solution 
function rectangleArea(width: number, height: number): number {
  return width * height;
}

Explicit parameter types (TS can't infer these), and the return type is technically inferable but often annotated on public functions for clarity/documentation.

2 (easy): Interface with optional field. Define a Product interface with required name/price and an optional discount, and write a function computing the final price.
Solution 
interface Product { name: string; price: number; discount?: number; }
function finalPrice(p: Product): number {
  return p.price * (1 - (p.discount ?? 0));
}

discount? makes it optional; ?? supplies 0 when absent — TS requires this kind of handling since p.discount could genuinely be undefined.

3 (medium): Narrowing. Write a function accepting string | string[] and always returning a joined string, using narrowing (no assertions).
Solution 
function toDisplayString(value: string | string[]): string {
  if (Array.isArray(value)) {
    return value.join(", ");   // TS knows value is string[] here
  }
  return value;                 // TS knows value is string here
}

Array.isArray is itself a built-in type guard TS understands, narrowing the union in each branch without any manual casting.

4 (medium): Generic function. Write a generic last<T>(arr: T[]): T | undefined returning the last element, preserving the element type.
Solution 
function last<T>(arr: T[]): T | undefined {
  return arr[arr.length - 1];
}
last([1, 2, 3]);      // number | undefined
last(["a", "b"]);     // string | undefined

The generic parameter T is inferred from the argument's element type and carried through to the return type — full type safety without hardcoding a specific type or falling back to any.

5 (hard): Runtime-validated fetch. Write a type guard isPost(x: unknown): x is { id: number; title: string } and a fetchPost(id: number) that validates the response before returning it typed.
Solution 
interface Post { id: number; title: string; }
function isPost(x: unknown): x is Post {
  return typeof x === "object" && x !== null &&
    typeof (x as any).id === "number" && typeof (x as any).title === "string";
}
async function fetchPost(id: number): Promise<Post> {
  const res = await fetch(`/api/posts/${id}`);
  const data: unknown = await res.json();
  if (!isPost(data)) throw new TypeError("Invalid post shape");
  return data;
}

Treating the JSON result as unknown and running it through a real type guard is the only way to have a genuine runtime guarantee that the compile-time Post type actually holds for this specific network response.

6 (hard): Derive types with utilities. Given a full Order interface with id, customerId, items, total, internalNotes, derive (a) a type for creating a new order (no id, everything else required) and (b) a type safe to send to the customer-facing API (no internalNotes).
Solution 
interface Order {
  id: number; customerId: number; items: string[]; total: number; internalNotes: string;
}
type NewOrder = Omit<Order, "id">;                       // everything but id — id is server-assigned
type CustomerFacingOrder = Omit<Order, "internalNotes">; // hide internal-only field

function createOrder(order: NewOrder): Order { /* assign an id, save, return full Order */ return { id: 1, ...order }; }
function toCustomerView(order: Order): CustomerFacingOrder {
  const { internalNotes, ...rest } = order;   // strip at runtime too — types alone don't remove data!
  return rest;
}

Both derived types reuse Order's definition via Omit rather than being hand-duplicated (avoiding drift). Note the important reminder in toCustomerView: the type CustomerFacingOrder only affects what TypeScript checks at compile time — you must still actually strip internalNotes from the real object at runtime (via destructuring/rest, as shown), or the field would still be present in the actual JSON sent to the customer despite the type claiming otherwise. This is Part 1's "types are erased, zero runtime effect" lesson applied concretely.

🎯 Key Takeaways

  • TypeScript's types are compile-time-only and fully erased — zero runtime cost, but also zero runtime safety; always validate untrusted data at boundaries.
  • Prefer unknown over any — it forces narrowing before use instead of silently disabling all checking.
  • Narrowing is TS's flow-sensitive type refinement after runtime checks (typeof, instanceof, custom type guards) — what makes union types feel natural to use.
  • Generics preserve type relationships across inputs/outputs, unlike any which discards them; utility types (Partial, Pick, Omit) derive new types from existing ones, avoiding drift.
  • TypeScript is structurally typed (shape determines compatibility, like JS's own duck typing) — not nominally typed like Java/C#.
  • Type assertions (as, !) perform zero runtime verification — prefer real narrowing/type guards, and reserve assertions for cases the compiler genuinely can't infer.
