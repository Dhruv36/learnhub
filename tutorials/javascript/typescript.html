TypeScript Essentials Advanced

TypeScript = JavaScript + a type layer that vanishes at build time. It converts a whole category of production incidents ("undefined is not a function") into red squiggles before commit. New serious JS projects in 2026 are TS by default — this lesson is the 20% that covers 90% of daily use.

Types Where They Matter

// Inference does most of the work — annotate BOUNDARIES, let locals infer:
const count = 3;                                  // inferred: number. Don't annotate.
function applyDiscount(price: number, code?: string): number {   // params + return = the contract
  return code ? price * 0.9 : price;
}

interface User {                    // shape of your data
  id: number;
  name: string;
  role: "admin" | "viewer";         // union of LITERALS — typo-proof enums
  lastLogin?: Date;                 // optional
}

type ApiResult =                 // generics + discriminated union — the API pattern
  | { status: "ok"; data: T }
  | { status: "error"; message: string };

Narrowing — How TS Follows Your Logic

function handle(r: ApiResult) {
  if (r.status === "ok") {
    r.data.length;                  // ✅ TS KNOWS data exists in this branch
  } else {
    r.message;                      // ✅ and message here — autocomplete proves it
  }
}
// Narrowing tools: typeof x === "string" · Array.isArray(x) · x instanceof Error
//                  "key" in obj · x != null

unknown > any (the discipline line)

const data: unknown = await res.json();     // honest: we DON'T know what the API sent
// data.name                                 ❌ compiler blocks until you prove it
const user = UserSchema.parse(data);         // runtime validation (zod) → typed AND checked

// `any` switches the compiler off for that value — and it spreads through everything
// it touches. Every `any` is a small hole in the ship. Lint rule: no-explicit-any.

Key mental model: types are erased at runtime. They verify your code's internal consistency — anything crossing the boundary (API responses, JSON.parse, form input) needs runtime validation. zod/valibot give you both from one schema.

tsconfig — One Setting to Die For

{ "compilerOptions": { "strict": true } }
// Non-negotiable. Most valuable inside: strictNullChecks — null/undefined become
// visible in types ("string | undefined"), and the billion-dollar mistake gets
// caught at compile time instead of 3am.
💡 Staff-level insight: TypeScript's ROI is organizational, not personal — types are enforced documentation that survives refactors and lets a stranger (or you, in six months) change code confidently. That's why big companies mandate it: it lowers the cost of every future engineer touching the file. Resist type-golf; a readable interface beats a clever conditional type in application code.
