TypeScript Essentials Advanced

TypeScript = JavaScript + a static type layer that vanishes at build time. It converts a whole category of production incidents ("undefined is not a function", typo'd fields, wrong-shape API assumptions) into red squiggles before commit. New serious projects default to TS in 2026; this lesson is the 20% that runs 90% of daily work — plus the boundary discipline (runtime validation) that separates real safety from theater.

The Golden Rule: Annotate Boundaries, Infer Locals

const count = 3;                    // inferred number — annotating is noise
const users = await getUsers();      // inferred from getUsers' return type

// The CONTRACTS get explicit types:
function applyDiscount(price: number, code?: string): number {
  return code ? price * 0.9 : price;
}

interface User {                     // your data shapes
  id: number;
  name: string;
  role: "admin" | "viewer";          // literal union — typo-proof enum, zero runtime cost
  lastLogin?: Date;                  // optional (absent OR undefined)
  readonly createdAt: Date;          // no reassignment through this type
}
type UserId = User["id"];            // indexed access — derive, don't duplicate

interface vs type: interfaces for object shapes (extendable, mergeable, better errors); type aliases for unions, intersections, functions, and derived types. Team convention matters more than the choice.

Discriminated Unions — TypeScript's Best Feature

type ApiResult =
  | { status: "ok";    data: T }
  | { status: "error"; message: string; code: number };

function handle(r: ApiResult) {
  switch (r.status) {
    case "ok":    return render(r.data);       // TS KNOWS data exists here
    case "error": return toast(r.message);     // and message here — autocomplete proves it
    default:      return assertNever(r);        // exhaustiveness: adding a variant breaks
  }                                              // compilation everywhere it's unhandled ✅
}
function assertNever(x: never): never { throw new Error(`Unexpected: ${JSON.stringify(x)}`); }
// This pattern — a literal 'discriminant' field + switch — models loading states,
// events, reducer actions, form states… it's the type-level version of the errors lesson's taxonomy.

Narrowing — the Compiler Follows Your Logic

function fmt(x: string | number | Date | null) {
  if (x === null) return "-";                 // null gone
  if (typeof x === "string") return x.trim();  // string branch
  if (typeof x === "number") return x.toFixed(2);
  return x.toISOString();                      // only Date remains — TS knows
}
// Tools: typeof · instanceof · Array.isArray · "key" in obj · x != null · literal ===
// Custom type guards when logic is reusable:
function isUser(x: unknown): x is User {
  return !!x && typeof x === "object" && "id" in x && "role" in x;
}

unknown vs any — the Discipline Line

const data: unknown = await res.json();   // HONEST: we don't know what arrived
// data.name                               ❌ compiler blocks until you prove the shape
if (isUser(data)) data.name;               // ✅ narrowed

const data2: any = await res.json();       // any = checker OFF for this value…
data2.naem.frist;                           // …compiles fine, explodes at runtime 💥
// and any is CONTAGIOUS: everything it touches degrades. Lint: no-explicit-any.
🧱 The boundary rule (this is the whole game): types are ERASED at runtime — they verify internal consistency, not reality. Anything crossing the boundary (API responses, JSON.parse, form input, URL params, localStorage) must be runtime-validated. Schema libraries give you both from one source:

import { z } from "zod";
const UserSchema = z.object({ id: z.number(), name: z.string(),
                              role: z.enum(["admin", "viewer"]) });
type User = z.infer;            // static type FROM the schema
const user = UserSchema.parse(await res.json());   // runtime check → typed value
// Schema drift now throws AT THE BOUNDARY with a precise message — not three
// components later as `undefined is not a function`.

Generics — Types With Parameters

function first(arr: T[]): T | undefined { return arr[0]; }
first(["a", "b"]);            // T inferred as string — callers rarely write 

async function fetchJson(url: string, schema: z.ZodType): Promise {
  const res = await fetch(url);
  if (!res.ok) throw new ApiError(res.status);
  return schema.parse(await res.json());          // generic + runtime validation = the pattern
}
const users = await fetchJson("/api/users", z.array(UserSchema));  // users: User[]

// Constraints — "any T that has…":
function longest(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

The theme: derive types from one source of truth (an interface, a schema, typeof a const object) — duplicated shapes drift; derived shapes can't. satisfies completes the kit: const config = {...} satisfies Config checks the shape WITHOUT widening — literal values stay literal for downstream inference (vs as Config, which silences instead of checks).

tsconfig — the Settings That Matter

{
  "compilerOptions": {
    "strict": true,                          // NON-NEGOTIABLE — the umbrella
    // inside it, the crown jewel: strictNullChecks —
    // string | undefined must be handled BEFORE use: the billion-dollar mistake, caught
    "noUncheckedIndexedAccess": true,        // arr[i] is T | undefined — honest about bounds
    "noImplicitOverride": true,
    "verbatimModuleSyntax": true             // explicit `import type` — clean erasure
  }
}
// Migrating legacy JS? checkJs + // @ts-check per file, allowJs, then rename to .ts
// module by module — strict from day one on NEW code, ratcheting on old.

Interview Questions

Q1: "TypeScript makes code safe" — attack and defend that claim precisely.

Attack: types are erased — they can't validate runtime data; any/as create holes; external inputs arrive untyped. Defend: within the typed region, whole bug classes (typos, wrong shapes, missing null checks, non-exhaustive switches) are compile-errors; combined with boundary validation (schemas) the typed region's assumptions become RUNTIME-guaranteed at entry. Safety = strict config + boundary parsing + no-any discipline; TS alone is necessary, not sufficient.

Q2: Model a data-fetching hook's state so invalid combinations can't exist.

Discriminated union: {status:"idle"} | {status:"loading"} | {status:"success"; data:T} | {status:"error"; error:AppError}. Now data exists ONLY in success — the classic bug state (isLoading && data && error all truthy) is unrepresentable. Consumers switch on status; assertNever future-proofs. "Make illegal states unrepresentable" is the phrase graders wait for.

Q3: unknown vs any vs never — one sentence + one use each.

any: opt OUT of checking (contagious; avoid) — last-resort interop. unknown: opt INTO checking — the type of external data before narrowing/validation. never: the impossible type — exhaustiveness checks (assertNever) and functions that only throw. Bonus: unknown is the top type (everything assignable TO it), never the bottom (assignable to everything).

Q4: Why do teams derive types from zod schemas rather than writing interfaces first?

One artifact, two guarantees: z.infer produces the static type FROM the runtime validator, so the type can never drift from what's actually checked. Interface-first + manual validators = two sources of truth that diverge silently. Same principle as OpenAPI/Prisma codegen: derive types from the source that's enforced.

🏋️ Practice (use the TS Playground at typescriptlang.org/play):
1. Model the fetch-state discriminated union + assertNever; add a fourth state and watch every switch light up.
2. Write fetchJson with a zod schema; feed it a deliberately wrong shape and read the boundary error.
3. Derive UpdateUserDto (Partial+Omit) and PublicUser (Omit password) from one User interface.
4. Turn on noUncheckedIndexedAccess and fix the resulting arr[0] errors properly (no !).
5. Replace an enum with an as const object + literal union type; compare the emitted JS (spoiler: none).

🎯 Key Takeaways

  • Annotate boundaries, infer locals; interfaces for shapes, unions for alternatives, literal unions over enums.
  • Discriminated unions + assertNever = illegal states unrepresentable and future variants un-forgettable.
  • Types are erased: unknown + schema-parse at every boundary (z.infer = type and validator from one source).
  • Narrowing follows real runtime checks; custom guards (x is T) package them reusably.
  • Derive with Pick/Omit/Partial/ReturnType/satisfies — one source of truth, no drift; `as` is a smell.
  • strict: true non-negotiable (strictNullChecks is the payoff); TS's ROI is organizational — enforced documentation that survives refactors.
