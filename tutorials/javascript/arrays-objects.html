Arrays & Objects Basics

90% of application code is one shape: arrays of objects in (API responses), transformed arrays of objects out (view models). Fluency here is fluency in the job. The two deep ideas underneath: transformation over loops, and reference semantics — the source of both the mutation-bug family and React's entire change-detection model.

The Transform Trio (with the details that matter)

const users = [
  { name: "Asha",  role: "admin",  score: 92 },
  { name: "Bruno", role: "viewer", score: 71 },
  { name: "Chen",  role: "admin",  score: 88 },
];

users.map(u => u.name)                         // transform 1:1 → new array
users.filter(u => u.score > 80)                // keep passers → new array
users.reduce((sum, u) => sum + u.score, 0)     // fold to ONE value — ALWAYS pass the initial!
// (no initial + empty array = TypeError; no initial also makes acc start as element[0])

// Chained, as production code reads:
const topAdminNames = users
  .filter(u => u.role === "admin")
  .map(u => u.name)
  .toSorted();                                  // immutable sort (more below)

Mutation vs Copy — the Bug Family, Named

// MUTATING methods (change the array in place):
push pop shift unshift splice sort reverse fill copyWithin
// NON-mutating (return new):
map filter slice concat flat toSorted toReversed toSpliced with

// The two incidents these cause:
// ① The shared-reference surprise:
const a = [3, 1, 2];
const b = a;             // NOT a copy — same array, two names
b.sort();                // a is now sorted too 😱

// ② The framework invisibility:
items.push(newItem);     // same reference → React/Vue see "no change" → no re-render
setItems([...items, newItem]);   // new reference → change detected ✅

// Default sort is STRING-based — the third classic:
[10, 9, 1].sort()                 // [1, 10, 9] 😱
[10, 9, 1].toSorted((a,b) => a-b) // [1, 9, 10] — copy + numeric comparator

Copying — Shallow vs Deep, Precisely

const copy = { ...user, score: 95 };        // SHALLOW: top level new, nested still SHARED
copy.address.city = "Pune";                  // 😱 original's address changed too

const deep = structuredClone(state);         // real deep copy: Dates, Maps, Sets, cycles all survive
// The old JSON.parse(JSON.stringify(x)) hack: drops undefined & functions, Dates→strings,
// Maps/Sets→{}, throws on cycles/BigInt. Recognize it; write structuredClone.
// (Neither copies functions/class instances meaningfully — state should be plain data anyway.)

Destructuring & Object Syntax — Full Fluency

const { name, score: points = 0, ...rest } = users[0];   // extract + rename + default + rest
const [first, , third = "n/a"] = list;                     // arrays: skip + default
function greet({ name, role = "viewer" }) { … }            // in parameters (options pattern)
const { a: { b } = {} } = obj;                              // nested with safety default

const key = "role";
const query = { [key]: "admin", active: true };             // computed keys
const point = { x, y, move() { … } };                       // shorthand props + methods
// Swap without temp: [a, b] = [b, a]

Objects as Dictionaries — and When to Use Map/Set Instead

// Plain-object keys are STRINGS or SYMBOLS — everything else coerces:
const o = {}; o[1] = "a";  o["1"]        // "a" — same key
o[{id:1}] = "x";                          // key becomes "[object Object]" for EVERY object 😱

const cache = new Map();                  // Map: ANY key type, by identity
cache.set(domNode, data);                 // insertion order, .size, iterable
const seen = new Set([1, 2, 2, 3]);       // {1,2,3};  [...new Set(arr)] = dedupe idiom

// keyBy — the lookup-table pattern you'll write weekly:
const byId = new Map(users.map(u => [u.id, u]));
byId.get(42);                             // O(1) vs find's O(n) in a loop → O(n²)
ChooseWhen
Plain objectFixed, known string keys (records, configs) — destructuring, JSON, spread all natural
MapDynamic keys, non-string keys, frequent add/delete, need .size/ordered iteration
SetMembership & dedupe: has() is O(1) vs includes() O(n)
WeakMap/WeakSetMetadata about objects you don't own — entries die with the key (memory lesson)

Iteration & Entries-Based Transforms

for (const u of users) { }                       // values (any iterable)
for (const [i, u] of users.entries()) { }        // index + value
for (const [k, v] of Object.entries(obj)) { }    // object pairs
// for…in: string keys INCLUDING inherited enumerables — wrong tool for arrays, legacy for objects

// The entries round-trip — object transforms via array methods:
const clean = Object.fromEntries(
  Object.entries(form).filter(([, v]) => v != null && v !== "")
);   // strip empty fields in one expression

JSON — What Survives the Round Trip

JSON.stringify({ a: undefined, f: () => {}, d: new Date(), s: new Set([1]) })
// → '{"d":"2026-07-06T…","s":{}}'   undefined & functions DROPPED, Date→string, Set→{}
JSON.parse(json, (k, v) => k.endsWith("At") ? new Date(v) : v);   // reviver to revive dates
JSON.stringify(obj, null, 2)                                       // pretty-print for logs/files

Interview Questions

Q1: Why does immutability matter to React specifically — mechanics, not vibes?

Reconciliation compares by REFERENCE (Object.is) — props/state that mutate in place keep their reference, so React concludes "unchanged" and skips re-render; memoized components and effect dependency arrays fail the same way. New-reference updates ([...arr, x], {...obj, k:v}) make change detectable in O(1) without deep comparison. Immutability here isn't ideology — it's the contract the diffing model is built on.

Q2: Implement groupBy(arr, keyFn) without Object.groupBy.

reduce into an object: (acc, x) => { const k = keyFn(x); (acc[k] ??= []).push(x); return acc; } with {} initial. Follow-ups they'll ask: Map version for non-string keys; why ??= beats acc[k] = acc[k] || [] (doesn't clobber legit falsy); complexity O(n).

Q3: ["1","7","11"].map(parseInt) — output and why?

[1, NaN, 3]. map passes (value, index): parseInt("1",0)→1 (radix 0 = auto), parseInt("7",1)→NaN (base-1 invalid), parseInt("11",2)→3 (binary 11). The lesson: know callback arity; pass Number, or x => parseInt(x, 10).

Q4: Design the data structure for "5,000 products, frequent lookup by id, iterate by insertion, dedupe tags".

Map<id, product> for O(1) lookup + insertion-order iteration (keyBy on load); a Set per product (or one global Set) for tags. Mention: plain object would coerce numeric ids to strings (usually fine but subtle), find-in-array per lookup would be O(n) → O(n²) across a render pass.

🎯 Key Takeaways

  • map/filter/reduce transform, never mutate; know callback arity (the parseInt trap) and always seed reduce.
  • Mutators (push/sort/splice…) vs copiers — reference semantics make mutation invisible to frameworks; toSorted/toReversed/with are the modern defaults.
  • Spread is shallow; structuredClone is the real deep copy; the JSON hack corrupts Dates/Maps/undefined.
  • Destructuring everywhere: parameters, renaming, defaults, rest-omission.
  • Object = record; Map = dictionary (any keys, O(1), ordered); Set = membership/dedupe; index big lists with keyBy Maps.
  • for…of (+ .entries()) over for…in; Object.entries/fromEntries for object transforms.
