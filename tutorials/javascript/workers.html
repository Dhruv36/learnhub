Web Workers & Concurrency Ultra-Advanced

The event loop lesson proved the main thread must never block — but real products parse huge CSVs, diff documents, process images. Workers are JavaScript's answer: real OS threads with no shared state, talking through messages.

The Basic Contract

// main.js
const worker = new Worker("parse-worker.js", { type: "module" });
worker.postMessage({ csvText });                        // structured-clone copy, not shared
worker.onmessage = (e) => renderTable(e.data.rows);     // result comes back async
worker.onerror   = (e) => showError(e.message);

// parse-worker.js — no DOM, no window; fetch/timers/IndexedDB available
self.onmessage = (e) => {
  const rows = heavyParse(e.data.csvText);              // blocks THIS thread — UI unaffected
  self.postMessage({ rows });
};

Messages are copied (structured clone). Big payloads have a fast path — transfer ownership instead of copying:

const buf = new ArrayBuffer(64 * 1024 * 1024);
worker.postMessage(buf, [buf]);          // moved in ~0ms; main thread's buf is now empty

When a Worker Pays For Itself

Worker-worthy (CPU-bound)Not worker-worthy
Parsing/serializing MBs of JSON, CSV, logsfetch calls (already async — no thread is blocked)
Image/video/audio processing, compressionDOM updates (workers can't touch the DOM)
Search indexing, diffing, crypto, ML inferenceWork under ~30ms — postMessage overhead eats the win

Heuristic: measurable main-thread task > 50ms, done repeatedly → worker. And wrap the message-passing in a Promise API (or use the tiny comlink library) so callers just await parseInWorker(csv).

True Shared Memory (the deep end)

const sab = new SharedArrayBuffer(1024);          // SAME memory in both threads
const arr = new Int32Array(sab);
Atomics.add(arr, 0, 1);                           // race-free increment
Atomics.wait(arr, 0, expected); Atomics.notify(arr, 0);   // thread coordination
// Requires cross-origin isolation headers (COOP/COEP). This is how WebAssembly
// multithreading (Photoshop web, games) works. App code rarely needs it — know it exists.

The Worker Family

TypePurpose
Dedicated WorkerThis lesson — one page, one worker
Shared WorkerOne instance serving multiple tabs (shared websocket, cross-tab state)
Service WorkerNetwork proxy: offline caching, push notifications, PWAs — lifecycle, not compute
WorkletsTiny hooks into the render pipeline (audio, paint)
💡 Staff-level insight: the message-passing model isn't a limitation — it's the same "share memory by communicating" philosophy as Go channels and the actor model, chosen because shared-state threading is where bugs live. In Node, the equivalent is worker_threads (same postMessage API) — and the same rule: I/O scales on the event loop, CPU scales on workers.
