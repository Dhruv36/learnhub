Web Workers & Concurrency Production JavaScript

What you'll master: real parallel execution via Web Workers (as opposed to the cooperative concurrency of the event loop), the message-passing communication model and why it exists, transferable objects for zero-copy performance, SharedArrayBuffer for true shared memory, and when a worker is actually the right tool versus over-engineering. Prerequisites: The Event Loop (the single-threaded model this escapes) and Objects (structured cloning). This is JavaScript's actual answer to "but I need real parallelism," not just cooperative yielding.

Part 1 — Workers Are Real Threads, Not Just Async

// EVERYTHING up to this lesson has been ONE thread, cooperatively multitasking (Event Loop lesson):
// async/await, promises, setTimeout — all still execute on the SAME single thread, taking turns.
// Heavy synchronous work (a big loop, a large JSON.parse, image processing) BLOCKS that one thread —
// no amount of async/await avoids this, because there's only ONE thread to block.

// A Web Worker is a SEPARATE thread, with its OWN event loop, running in TRUE parallel:
const worker = new Worker("worker.js");
// worker.js now runs concurrently with your main script — actual parallelism,
// not the cooperative time-slicing of async/await on one thread.

This is the critical distinction the whole track has been building toward: async/await and promises do not give you parallelism — they give you cooperative concurrency on a single thread (the Event Loop lesson), where operations take turns and a genuinely CPU-heavy synchronous computation still blocks everything, no matter how many awaits surround it. A Web Worker is a real, separate OS-level thread with its own JavaScript engine instance, own event loop, and own memory — it runs your code in true parallel alongside the main thread. This is the actual answer to "how do I do heavy computation without freezing the UI," where the Event Loop lesson's chunk-and-yield technique was a workaround, not a real fix.

Part 2 — No Shared Memory: the Message-Passing Model

// main.js
const worker = new Worker("worker.js");
worker.postMessage({ command: "process", data: [1, 2, 3, 4, 5] });   // send a COPY, not a reference
worker.onmessage = (event) => console.log("Result:", event.data);
worker.onerror = (event) => console.error("Worker error:", event.message);

// worker.js — runs in a completely SEPARATE global scope (no `window`, no DOM access):
self.onmessage = (event) => {
  const { command, data } = event.data;
  if (command === "process") {
    const result = data.map(n => n * n);        // heavy work happens HERE, off the main thread
    self.postMessage(result);
  }
};

Because true parallelism means two threads could touch the same memory at the exact same instant — the classic race-condition/data-corruption problem — workers cannot directly access the main thread's variables, and vice versa. Communication happens exclusively through postMessage: you send data, and by default it's copied (via the structured clone algorithm — the Objects lesson's structuredClone, the exact same mechanism) rather than shared, so each side has its own independent copy with no risk of one thread mutating data the other is using. A worker also runs in a completely separate global scope — no window, no direct DOM access (a worker cannot touch the page at all) — which is why it's safe to run heavy synchronous work there without blocking the UI: the UI thread and the worker thread are genuinely independent.

Part 3 — Structured Cloning: the Cost of Message Passing

const bigArray = new Array(10_000_000).fill(0);
worker.postMessage(bigArray);
// Under the hood: the ENTIRE array is serialized (structured-cloned) and copied to the worker's memory.
// For large data, this copy itself takes real, measurable time — a genuine cost of the safety guarantee.

// You can pass complex data (not just JSON) — Dates, Maps, Sets, ArrayBuffers, even other objects:
worker.postMessage({ when: new Date(), tags: new Set(["a","b"]), buffer: new ArrayBuffer(1024) });
// (though NOT functions, DOM nodes, or class instances — same limits as structuredClone)

The default copy-on-send safety has a real cost: for large payloads, structured cloning itself takes measurable time and memory (doubling the data momentarily — original plus copy), which can eat into the performance win you moved to a worker to get in the first place. Structured cloning supports far more than JSON (Dates, Maps, Sets, RegExp, ArrayBuffers, and more — see the Objects lesson's deep-clone discussion), but it still can't clone functions, DOM nodes, or arbitrary class instances (they lose their prototype and become plain objects). For truly large binary data, Part 4's transferable objects avoid this copy cost entirely.

Part 4 — Transferable Objects: Zero-Copy Handoff

const buffer = new ArrayBuffer(100 * 1024 * 1024);   // 100MB of raw binary data

// Regular postMessage: COPIES all 100MB — slow, doubles memory momentarily.
worker.postMessage(buffer);

// TRANSFER instead: ownership MOVES to the worker, ZERO copy — near-instant regardless of size:
worker.postMessage(buffer, [buffer]);   // the second argument lists TRANSFERABLE objects
buffer.byteLength;   // 0 — the main thread's buffer is now NEUTERED/detached; it no longer owns the data!
// The WORKER now exclusively owns those bytes — this is a genuine ownership TRANSFER, not a copy.

For large binary data (ArrayBuffer, and by extension typed arrays, ImageData, MessagePort), the platform offers transferable objects: instead of copying the data, ownership itself moves from sender to receiver — a zero-copy handoff that's essentially instant regardless of data size, because no bytes are actually duplicated. The catch, which is the whole point: the sending side's reference becomes unusable (its byteLength becomes 0 — it's "neutered") the instant you transfer it, because ownership is now exclusively the receiver's — this is a real trade-off (you give up access), not a free lunch, but it's exactly the right trade-off when you're handing a large buffer to a worker specifically to have it process that data and don't need the main thread to touch it simultaneously anyway.

Part 5 — SharedArrayBuffer: True Shared Memory (Advanced, Guarded)

// Regular ArrayBuffer: transferred (moved) OR copied — never simultaneously accessible from both sides.
// SharedArrayBuffer: BOTH threads can read/write the SAME underlying memory AT THE SAME TIME:
const sab = new SharedArrayBuffer(1024);
const view = new Int32Array(sab);
worker.postMessage(sab);   // NOT copied, NOT transferred — the WORKER gets a VIEW onto the SAME bytes

// Because it's genuinely shared, you need explicit synchronization to avoid races —
// Atomics provides indivisible, race-free operations on shared memory:
Atomics.add(view, 0, 1);           // atomic increment — safe even if another thread does this simultaneously
Atomics.wait(view, 0, 0);           // block THIS thread until view[0] changes from 0 (worker-thread only)
Atomics.notify(view, 0, 1);         // wake up 1 thread waiting on that index

SharedArrayBuffer is the one true exception to "no shared memory": both the main thread and a worker can hold views onto the exact same underlying bytes and read/write them concurrently — real shared memory, with all the classic concurrency hazards that implies (race conditions, torn reads/writes). The Atomics object provides indivisible read-modify-write operations (increment, compare-and-swap, wait/notify) that are guaranteed race-free even under true concurrent access — the low-level primitives real concurrent-programming languages provide. This is genuinely advanced, rarely-needed territory (high-performance computation like audio/video processing, physics simulations, or WASM-heavy code), requires specific cross-origin isolation headers to even enable (a security mitigation after Spectre), and most applications never need it — reach for regular postMessage first, always.

Part 6 — When a Worker Is (and Isn't) the Right Tool

Good fit for a WorkerNot worth a Worker
Heavy, sustained CPU computation (parsing huge files, image/video processing, complex calculations)Occasional light work — the messaging overhead can exceed the work itself
Work that genuinely blocks the main thread noticeably (100ms+)Anything needing direct, frequent DOM access
Background tasks that don't need results immediately (indexing, precomputation)I/O-bound work (network requests) — these DON'T block the thread anyway; async/await already handles them
Independent, parallelizable chunks of a big job (split N items across M workers)Simple, quick synchronous work — chunk-and-yield (Event Loop lesson) is simpler and sufficient

Workers have real overhead: spawning one, and every message you send, has a cost (structured cloning, thread context-switching) — using one for trivial work is slower than just doing the work on the main thread. Crucially, network requests and other I/O don't need a worker at all — fetch is already non-blocking on the main thread via the event loop (the whole point of async/await); a worker is specifically for CPU-bound, blocking, synchronous computation. The decision rule: if the work is a tight synchronous loop that would visibly freeze the UI for 100ms or more, a worker is likely worth it; if it's async I/O or quick, a worker adds complexity and overhead for no benefit — profile first (Performance lesson) rather than reaching for a worker preemptively.

Part 7 — A Practical Pattern: Promise-Wrapped Worker Calls

// Raw postMessage/onmessage is callback-based and awkward for request/response patterns.
// Wrap it in a promise-based API — familiar, awaitable, composable with everything else you know:
class WorkerPool {
  #worker;
  constructor(scriptUrl) { this.#worker = new Worker(scriptUrl); }

  run(payload) {
    return new Promise((resolve, reject) => {
      const id = crypto.randomUUID();                       // correlate THIS call's response
      const handleMessage = (e) => {
        if (e.data.id !== id) return;                        // not our response — ignore (other calls in flight)
        this.#worker.removeEventListener("message", handleMessage);
        e.data.error ? reject(new Error(e.data.error)) : resolve(e.data.result);
      };
      this.#worker.addEventListener("message", handleMessage);
      this.#worker.postMessage({ id, payload });
    });
  }
}

const pool = new WorkerPool("compute-worker.js");
const result = await pool.run({ numbers: hugeArray });   // reads like any other async call

Raw postMessage/onmessage is a callback-style API that doesn't compose with async/await or Promise.all out of the box, and doesn't handle multiple concurrent requests to the same worker cleanly. Wrapping it in a promise-based request/response layer — tagging each call with a unique id so responses can be correctly matched to their originating call — turns worker communication into something that reads and composes exactly like any other async operation you've learned in this track. This pattern (sometimes called "Comlink- style" after the popular library that formalizes it) is how production code actually uses workers day to day, rather than hand-writing raw message handlers everywhere.

Part 8 — Build It: an Off-Thread Image Processor

Let's build a realistic worker use case end-to-end: processing a large image (a genuinely CPU-heavy, synchronous operation) off the main thread, using a transferable object for the pixel data to avoid the copy cost.

// main.js
async function processImageOffThread(imageData) {
  const worker = new Worker("image-worker.js");
  return new Promise((resolve, reject) => {
    worker.onmessage = (e) => { resolve(e.data); worker.terminate(); };   // clean up when done
    worker.onerror = (e) => { reject(new Error(e.message)); worker.terminate(); };
    // Transfer the pixel buffer — zero-copy, since the main thread doesn't need it during processing:
    worker.postMessage({ data: imageData.data.buffer, width: imageData.width, height: imageData.height },
                        [imageData.data.buffer]);
  });
}

// image-worker.js
self.onmessage = (e) => {
  const { data, width, height } = e.data;
  const pixels = new Uint8ClampedArray(data);    // reconstruct a typed array view over the transferred bytes
  for (let i = 0; i < pixels.length; i += 4) {    // heavy per-pixel work — grayscale conversion
    const gray = pixels[i] * 0.299 + pixels[i+1] * 0.587 + pixels[i+2] * 0.114;
    pixels[i] = pixels[i+1] = pixels[i+2] = gray;
  }
  self.postMessage({ data: pixels.buffer, width, height }, [pixels.buffer]);   // transfer BACK, zero-copy
};

// Usage — UI never freezes even on a huge image:
const grayscale = await processImageOffThread(originalImageData);

Every technique from the lesson lands here. The main thread stays fully responsive during a genuinely heavy per-pixel loop because it's running on a real, separate thread (Part 1) — this is not chunk-and-yield, it's actual parallelism. The pixel buffer is transferred, not copied, in both directions (Part 4) — for a large image this avoids doubling potentially tens of megabytes of memory and the associated copy latency, since neither side needs concurrent access to it during processing. And the whole thing is wrapped as a clean, await-able promise (Part 7), so calling code looks exactly like any other async operation in this track, hiding the worker/message-passing machinery behind a familiar interface. worker.terminate() on completion releases the thread rather than leaving it idle forever — worker lifecycle management is its own small but real resource-cleanup concern, echoing the Memory lesson's theme.

Interview Questions

Q1 (beginner): Does async/await give you real parallelism?

No. async/await and promises run entirely on JavaScript's single main thread — they enable cooperative concurrency (operations take turns via the event loop's microtask/macrotask scheduling) but never actual simultaneous execution. A CPU-heavy synchronous computation still blocks that one thread completely, no matter how it's wrapped in async syntax, because there's fundamentally only one thread available to run it on. Real parallelism — genuinely simultaneous execution on separate CPU cores — requires a Web Worker, which is a separate thread with its own JavaScript engine and event loop, running truly concurrently with the main thread.

Q2 (beginner): Why can't a worker directly access the main thread's variables or the DOM?

Because workers run in true parallel on a separate thread, allowing direct shared access to the same variables would create classic race conditions — two threads reading and writing the same memory at the same instant, causing corrupted or unpredictable results. To make parallelism safe by default, workers run in a completely separate global scope with no access to window, the DOM, or the main thread's variables; the only way to communicate is explicit message-passing via postMessage, where data is copied (or transferred) rather than shared. This is also why a worker can safely run heavy, blocking synchronous code without freezing the page — the two threads are genuinely independent, with communication happening only at well-defined message boundaries.

Q3 (mid): What is structured cloning, and what's its cost when passing data to a worker?

Structured cloning is the algorithm postMessage uses by default to send data to/from a worker: it serializes the data (supporting far more than JSON — objects, arrays, Dates, Maps, Sets, ArrayBuffers, RegExp, but not functions, DOM nodes, or arbitrary class instances) and creates a genuinely independent copy on the receiving side, so both threads have their own data with no shared references and no race-condition risk. The cost is that for large payloads, this copying takes real, measurable time and momentarily doubles memory usage (both the original and the copy exist simultaneously) — for a large array or buffer, this copy overhead can meaningfully eat into the performance benefit you moved to a worker to get. This is exactly the problem transferable objects (Part 4) solve for large binary data, by moving ownership instead of copying.

Q4 (mid): What's the difference between transferring an object and cloning it when using postMessage?

Cloning (the default) creates an independent copy of the data for the receiver, leaving the sender's original intact — safe, but costs time and memory proportional to data size, especially for large payloads. Transferring (via the second postMessage argument, a list of transferable objects like ArrayBuffer) moves ownership of the underlying memory to the receiver with zero copying — essentially instant regardless of size, because no bytes are actually duplicated. The trade-off: after transferring, the sender's reference becomes unusable (its byteLength becomes 0) since it no longer owns that memory. Transfer is the right choice for large binary data (big buffers, image data) that the sender doesn't need to keep using simultaneously; clone is the right default for smaller data or when both sides genuinely need their own independent copy.

Q5 (senior): Design a decision framework for when to offload work to a Web Worker versus keeping it on the main thread.

The decision hinges on the nature of the work, not just its presence. First, distinguish CPU-bound from I/O-bound: I/O (network requests, timers) never blocks the main thread regardless — it's already handled by the event loop's non-blocking model, so a worker adds no benefit there and only overhead. Workers matter only for CPU-bound, synchronous computation. Second, estimate the actual blocking duration: work under roughly 50ms is often not worth the overhead of spawning a worker and message-passing (structured cloning/transfer cost, thread setup) — chunk-and-yield on the main thread (the Event Loop lesson's technique) is simpler and sufficient for moderate work, since it keeps the UI responsive by periodically ceding control without the complexity of a separate thread and its communication protocol. For work that would visibly block for 100ms+ — large-scale data processing, complex calculations, image/video/audio processing, parsing huge files — a worker is clearly worth it, since chunking such heavy work would either take far longer (constant yielding overhead) or still cause perceptible jank between yields. Third, consider data volume and transfer cost: if the work requires passing large amounts of data back and forth repeatedly (not once), the round-trip message-passing overhead may erode the parallelism benefit — this favors either transferable objects/SharedArrayBuffer to minimize copy cost, or restructuring so more work happens per message (batch rather than chatty). Fourth, consider parallelizability: if a big job splits cleanly into independent chunks (e.g. processing 10,000 independent records), a pool of several workers each handling a slice can give genuine multi-core speedup, not just UI responsiveness. Finally, always measure rather than assume — profile the actual blocking duration on representative hardware/data before committing to the added architectural complexity of workers, since the "is this actually slow enough to matter" question is empirical, not something to guess at. Rule of thumb ranking from simplest to most complex, escalating only as needed: do it directly → chunk-and-yield → single worker → worker pool with transferables → SharedArrayBuffer (rare, last resort).

Q6 (senior): Explain SharedArrayBuffer's security history and why it requires special HTTP headers to use.

SharedArrayBuffer provides genuine shared memory between threads, which is exactly the primitive needed for the Spectre class of speculative-execution side-channel attacks: Spectre allows malicious code to use extremely precise, high-resolution timing measurements to infer data it shouldn't have access to, by observing subtle timing differences in how the CPU executes speculatively. SharedArrayBuffer, combined with Atomics' high-resolution timing capabilities, gave attackers exactly the precise clock and shared-memory primitives needed to mount practical Spectre attacks from within a malicious webpage — reading data from other origins/processes that should have been isolated by the browser's same-origin security model. In response, browsers disabled SharedArrayBuffer entirely for a period after Spectre's disclosure (2018), then reintroduced it gated behind a strict opt-in security requirement: a page must be cross-origin isolated, which requires serving it with two specific HTTP response headers — Cross-Origin-Opener-Policy: same-origin (isolating the page's browsing context group so other origins can't hold a reference to its window) and Cross-Origin-Embedder-Policy: require-corp (requiring every cross-origin resource the page loads to explicitly opt in via CORP/CORS, preventing arbitrary cross-origin data from being pulled into the isolated context). Only when both are set does the browser guarantee the page's process is sufficiently isolated from other origins that the timing-based side-channel risk is mitigated, and only then does it re-enable SharedArrayBuffer/high-resolution timers. This is a strong, concrete example of a language/platform feature being restricted specifically because of a hardware-level security vulnerability class, and it's why enabling SharedArrayBuffer in a real deployment isn't just a JavaScript code change — it requires server/CDN configuration changes and can break loading of any cross-origin resource (images, fonts, iframes) that doesn't itself opt in via CORP headers, a real deployment cost that reinforces why this feature should be reserved for cases that genuinely need true shared memory.

🏋️ Exercises

1 (easy): Basic worker round-trip. Write a worker that squares an array of numbers, and the main-thread code that sends the array and logs the result.
Solution 
// worker.js
self.onmessage = (e) => {
  self.postMessage(e.data.map(n => n * n));
};

// main.js
const worker = new Worker("worker.js");
worker.onmessage = (e) => console.log(e.data);   // [1, 4, 9, 16]
worker.postMessage([1, 2, 3, 4]);

The simplest possible message round-trip: send data in, the worker processes it and sends a result back via its own postMessage, received on the main thread's onmessage.

2 (easy): Handle errors. Add error handling so a worker exception doesn't silently vanish, logging it clearly on the main thread.
Solution 
worker.onerror = (event) => {
  console.error(`Worker error: ${event.message} (${event.filename}:${event.lineno})`);
};

Uncaught exceptions inside a worker fire an error event on the worker object rather than propagating like a normal exception — without an onerror handler, a worker failure is easy to miss entirely.

3 (medium): Transfer vs clone. Given a 50MB ArrayBuffer, write the postMessage call using transfer, and explain what you'd observe if you then tried to read from the original buffer.
Solution 
const buffer = new ArrayBuffer(50 * 1024 * 1024);
worker.postMessage(buffer, [buffer]);
console.log(buffer.byteLength);   // 0 — the buffer is now detached/neutered
new Uint8Array(buffer)[0];         // still "works" but reads/writes nothing meaningful — 0 bytes

Because transfer moves ownership rather than copying, the main thread's buffer reference is immediately detached — its byteLength becomes 0, signaling it no longer owns any data, since the worker now exclusively holds those bytes.

4 (medium): Decide worker vs no worker. For each scenario, decide whether a Web Worker is appropriate and justify briefly: (a) fetching a list of products from an API; (b) parsing a 200MB CSV file client-side; (c) validating a form field on every keystroke; (d) running a physics simulation for a game.
Solution

(a) No — fetch is I/O-bound and already non-blocking via the event loop; a worker adds no benefit. (b) Yes — parsing 200MB is heavy, sustained synchronous CPU work that would visibly freeze the UI; a clear worker candidate. (c) No — per-keystroke validation is typically trivial, fast synchronous work; worker overhead (message-passing) would likely exceed the validation cost itself. (d) Yes, likely — a physics simulation running every frame is sustained CPU work that benefits from real parallelism, especially if it shouldn't compete with rendering on the main thread (though for very simple simulations, main-thread + requestAnimationFrame chunking might suffice — profile first).

5 (hard): Promise-wrapped worker call. Using the lesson's WorkerPool pattern, implement a sortLargeArray(arr) function that offloads sorting to a worker and returns a promise resolving to the sorted array.
Solution 
// sort-worker.js
self.onmessage = (e) => {
  const { id, payload } = e.data;
  const sorted = [...payload].sort((a, b) => a - b);
  self.postMessage({ id, result: sorted });
};

// main.js
class WorkerPool {
  #worker;
  constructor(url) { this.#worker = new Worker(url); }
  run(payload) {
    return new Promise((resolve) => {
      const id = crypto.randomUUID();
      const handler = (e) => {
        if (e.data.id !== id) return;
        this.#worker.removeEventListener("message", handler);
        resolve(e.data.result);
      };
      this.#worker.addEventListener("message", handler);
      this.#worker.postMessage({ id, payload });
    });
  }
}
const pool = new WorkerPool("sort-worker.js");
async function sortLargeArray(arr) { return pool.run(arr); }
const sorted = await sortLargeArray([5, 3, 8, 1, 9]);   // [1, 3, 5, 8, 9]

Wrapping the raw message protocol in a promise, correlated by a unique id, gives a clean await-able API — sorting genuinely happens off the main thread, but the call site looks identical to any other async operation.

6 (hard): Worker pool for parallel chunks. Sketch how you'd split a 1,000,000-item array-processing job across 4 workers running in true parallel, and combine their results.
Solution 
async function processInParallel(items, workerCount = 4) {
  const chunkSize = Math.ceil(items.length / workerCount);
  const chunks = Array.from({ length: workerCount }, (_, i) =>
    items.slice(i * chunkSize, (i + 1) * chunkSize));

  const pools = chunks.map(() => new WorkerPool("process-worker.js"));   // one worker PER chunk
  const results = await Promise.all(                                     // all run TRULY concurrently
    chunks.map((chunk, i) => pools[i].run(chunk))
  );
  return results.flat();   // recombine each worker's partial result into one array
}

Splitting into workerCount independent chunks and processing them with separate workers via Promise.all gets genuine multi-core parallelism — unlike a single worker (which is still just one thread), four workers can use up to four CPU cores simultaneously, potentially giving a near-4x speedup for embarrassingly parallel work. The results are combined with flat() once all chunks complete — the classic map-reduce-style pattern applied to real parallel threads instead of just async scheduling.

🎯 Key Takeaways

  • async/await is cooperative concurrency on ONE thread — Web Workers are real, separate threads giving true parallelism.
  • No shared memory by default — workers communicate exclusively via postMessage, with data copied via structured cloning (safe, but costs time/memory for large payloads).
  • Transferable objects (ArrayBuffer etc.) move ownership with zero copy, at the cost of detaching the sender's reference — ideal for large binary data.
  • SharedArrayBuffer + Atomics give true shared memory with race-free primitives — powerful, rare, and gated behind cross-origin-isolation headers for Spectre-related security reasons.
  • Workers are for sustained CPU-bound blocking work, not I/O (fetch is already non-blocking) — the overhead of spawning/messaging makes them wrong for trivial or quick work.
  • Wrap raw message-passing in a promise-based, id-correlated API to make worker calls compose naturally with the rest of your async code.
