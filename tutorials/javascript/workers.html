Web Workers & Concurrency Ultra-Advanced

The event-loop lesson's hard limit: async reorders waiting, it doesn't move compute. Parsing a 50MB CSV is CPU-bound — await doesn't help; chunking keeps the UI alive but still burns the only UI thread. Workers are JavaScript's real answer: OS threads with no shared state, communicating by messages — a design that deletes the data-race bug class entire languages struggle with.

The Basic Contract

// main.js
const worker = new Worker(new URL("./parse-worker.js", import.meta.url), { type: "module" });
worker.postMessage({ csvText });                        // structured-clone COPY crosses over
worker.onmessage = (e) => renderTable(e.data.rows);     // result returns async
worker.onerror   = (e) => showError(e.message);
worker.terminate();                                      // hard-stop when done (or on unmount!)

// parse-worker.js — separate global (self), NO DOM/window; HAS fetch, timers, IndexedDB, crypto
self.onmessage = (e) => {
  const rows = heavyParse(e.data.csvText);              // blocks THIS thread — UI unaffected
  self.postMessage({ rows });
};
// The bundler-friendly `new URL(..., import.meta.url)` form is how Vite/webpack split worker code.

What Crosses the Boundary — Three Modes

ModeMechanicsWhen
Structured clone (default)Deep copy — objects, arrays, Maps, Sets, Dates, ArrayBuffers, Files. NOT functions, DOM nodes, class methods (data arrives as plain objects!)Normal payloads; cost scales with size
TransferpostMessage(buf, [buf]) — ownership MOVES in ~0ms; sender's copy is detached (length 0)Big binary: ArrayBuffer, ImageBitmap, OffscreenCanvas, streams — 100MB without a copy
SharedArrayBufferActual shared memory + Atomics for coordination; requires cross-origin isolation (COOP/COEP headers — Spectre fallout)WASM multithreading (Photoshop web, games, ffmpeg.wasm); app code rarely
// Transfer in action — the difference between 800ms and ~0ms for 100MB:
const buf = new Float64Array(12_500_000).buffer;
worker.postMessage({ buf }, [buf]);        // moved
buf.byteLength;                             // 0 — you gave it away

// Atomics teaser (the deep end):
const shared = new Int32Array(new SharedArrayBuffer(4));
Atomics.add(shared, 0, 1);                 // race-free increment
Atomics.wait(shared, 0, 0);  Atomics.notify(shared, 0);   // thread parking/waking

Ergonomics — From postMessage Soup to Awaitable Calls

// Hand-rolled RPC (know the pattern):
function callWorker(worker, payload) {
  return new Promise((resolve, reject) => {
    const id = crypto.randomUUID();
    const onMsg = (e) => { if (e.data.id !== id) return;
      worker.removeEventListener("message", onMsg);
      e.data.error ? reject(new Error(e.data.error)) : resolve(e.data.result); };
    worker.addEventListener("message", onMsg);
    worker.postMessage({ id, ...payload });
  });
}
const rows = await callWorker(worker, { csvText });     // reads like a local call

// Or the 1.1KB library that does this + proxies: Comlink
import * as Comlink from "comlink";
const api = Comlink.wrap(new Worker("./worker.js", { type: "module" }));
const rows = await api.parse(csvText);                   // worker methods look local

When a Worker Pays — the Decision Table

Worker-worthy (CPU-bound)NOT worker-worthy
Parsing/serializing MBs (CSV, JSON, logs)fetch/IO waiting — already async, no thread blocked
Image/video/audio processing, compression, resizingDOM updates — workers can't touch the DOM (OffscreenCanvas is the exception for canvas)
Search indexing, diffing, spreadsheet-style recalcTasks under ~30–50ms — postMessage overhead eats the win
Crypto, compression, ML inference (often WASM inside)Anything needing main-thread APIs constantly (chatty = clone-cost dominated)

Heuristic: the Performance panel shows a recurring main-thread task > 50ms → worker candidate. Design the boundary coarse: one "parse this file" message, not ten thousand "parse this row" messages. Worker startup is ~ms + script load — reuse via a persistent worker or a small pool (navigator.hardwareConcurrency − 1 threads) for repeated jobs.

The Worker Family — Four Siblings, Four Jobs

Interview Questions

Q1: "We made it async but the page still freezes" — diagnose and fix.

async/await moves WAITING off the timeline, not COMPUTE — a CPU-bound parse inside an async function still owns the only thread while it runs. Confirm with a Performance recording (one long yellow task). Fix ladder: move the parse to a Worker (right fix for >50ms recurring work); interim: chunk with scheduler.yield. This question separates people who know the words from people who know the model.

Q2: Why did the platform choose message passing over shared memory as the default?

Shared mutable state + threads = data races, deadlocks, heisenbugs — the hardest bug class in computing. Copy-on-message makes races structurally impossible: each thread owns its data. It's the same "share memory by communicating" philosophy as Go channels and the actor model (Erlang). SharedArrayBuffer exists as the explicit, gated opt-in for the niche (WASM threads) that truly needs it — with Atomics because once memory is shared, you inherit all the classic problems.

Q3: Design the worker architecture for a browser spreadsheet (10k×100 cells, formula recalc).

Persistent compute worker owning the dependency graph + values; main thread owns DOM/editing. Edits post deltas {cell, formula}; worker recalcs the dirty subgraph and posts back changed cells only (coarse batches). Big imports transfer ArrayBuffers. Recalc >1 core? Pool with the graph partitioned, or WASM+SAB if profiling justifies. Key phrases: coarse boundary, dirty-set diffs, transfer for bulk, UI never waits on recalc.

Q4: Service Worker vs Web Worker — a junior mixed them up in a design doc. Correct it.

Web Worker = compute thread, page-owned, lives/dies with it — for CPU work. Service Worker = installable network proxy shared per origin, event-driven, terminated when idle — for offline caching, push, background sync; it must NOT be used for long computation (it gets killed) and a Web Worker can't intercept requests. Names similar, roles disjoint.

🏋️ Practice:
1. Generate a 5M-row array; sum it on the main thread (feel the freeze), then in a worker — spinner keeps spinning.
2. Implement the RPC callWorker pattern with ids + error propagation; then redo it with Comlink and compare.
3. Benchmark clone vs transfer for a 100MB Float64Array (console.time around postMessage).
4. Build a 4-worker pool that maps over chunks of an array and merges results; compare wall-time vs single worker.
5. Break it on purpose: post an object with a method, observe the plain-data clone in the worker.

🎯 Key Takeaways

  • Async ≠ parallel: awaiting reorders waiting; CPU work needs another THREAD — that's workers.
  • No shared state by design: structured clone (copy, data-only), transfer (move, ~0ms binary), SAB+Atomics (gated deep end).
  • Worker-worthy = recurring >50ms CPU tasks; coarse message boundaries; persistent workers/pools over spawn-per-task.
  • Wrap postMessage in promise RPC (or Comlink) — worker calls should read like local awaits, with errors propagated.
  • Four siblings: dedicated (compute), shared (cross-tab), service (network proxy/PWA), worklets (render/audio hooks) — plus Node's worker_threads, same philosophy.
  • Message passing is the industry's answer to data races — Go/Erlang arrived at the same design.
