async / await Asynchronous JavaScript

What you'll master: async/await as readable syntax over promises, error handling with try/catch, the sequential-vs-parallel trap that silently makes code slow, looping over async work correctly, and the subtle behaviours (an async function always returns a promise; await yields the thread). Prerequisites: Promises (async/await IS promises underneath) and The Event Loop (await schedules a microtask). This is how you'll actually write async code day to day — but only correct because you understand the promises beneath it.

Part 1 — async/await Is Promise Syntax

// Promise chain:
function getUser(id) {
  return fetch(`/api/users/${id}`)
    .then(res => res.json())
    .then(user => user.name);
}

// EXACTLY equivalent with async/await — same behaviour, flatter reading:
async function getUser(id) {
  const res = await fetch(`/api/users/${id}`);   // await "unwraps" the promise
  const user = await res.json();
  return user.name;                               // an async fn's return value → the resolved promise value
}

async/await is syntactic sugar over promises — it doesn't add capability, it makes promise-based code read like synchronous code. await pauses the function until the awaited promise settles, then evaluates to its resolved value (or throws its rejection). Two rules define it: an async function always returns a promise (its return value becomes the resolved value; a thrown error becomes a rejection), and await can only be used inside an async function (or at the top level of a module — "top-level await"). Everything from the Promises lesson still applies; this is just a nicer way to express it.

Part 2 — Error Handling with try/catch

async function loadUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);   // fetch doesn't reject on 404/500 (Fetch lesson)
    return await res.json();
  } catch (err) {
    console.error("loadUser failed:", err);
    return null;                              // recover, or re-throw to propagate
  } finally {
    hideSpinner();                            // runs regardless — like promise .finally
  }
}

The big readability win: async errors are caught with ordinary try/catch, the same construct as synchronous errors. A rejected await throws into the catch block, unifying sync and async error handling in one familiar structure (versus chained .catch). finally works too, for cleanup. This is a genuine advantage over raw promise chains — the error flow is visually obvious and colocated with the code that can fail. As with promises, you either recover (return from catch) or re-throw to let the caller handle it.

Part 3 — The Sequential-vs-Parallel Trap (the Big One)

// ❌ SLOW — sequential: each await WAITS for the previous, even though they're independent:
async function loadDashboard() {
  const user = await fetchUser();       // 300ms
  const posts = await fetchPosts();     // 300ms — starts only AFTER user finishes
  const stats = await fetchStats();     // 300ms — starts only AFTER posts finishes
  return { user, posts, stats };        // total: ~900ms
}

// ✅ FAST — parallel: start all three, THEN await them together:
async function loadDashboard() {
  const [user, posts, stats] = await Promise.all([
    fetchUser(), fetchPosts(), fetchStats()   // all three start immediately, in flight together
  ]);
  return { user, posts, stats };              // total: ~300ms (the slowest one)
}

This is the most important and most common async mistake. Sequential awaits run one after another — if the operations are independent, you've made three 300ms requests take 900ms for no reason, because each await blocks the next from even starting. The fix: start all independent operations first (don't await them individually), then await them together with Promise.all — they run concurrently and the total time is the slowest one, not the sum. The rule: await sequentially only when a step genuinely depends on the previous step's result; otherwise fire them off in parallel. Recognising when awaits are needlessly sequential is a top-tier code-review and interview skill (Promise Combinators lesson goes deeper).

Part 4 — Looping Over Async Work

const ids = [1, 2, 3];

// SEQUENTIAL — for...of with await: each iteration waits (use when order/backpressure matters):
async function sequential() {
  const results = [];
  for (const id of ids) results.push(await fetchUser(id));   // one at a time
  return results;
}

// PARALLEL — map to promises, then Promise.all (use when independent & you want speed):
async function parallel() {
  return Promise.all(ids.map(id => fetchUser(id)));          // all at once
}

// ⚠ TRAP — forEach does NOT await; this returns before anything finishes:
ids.forEach(async (id) => { await fetchUser(id); });         // ❌ fire-and-forget, uncontrolled

Two correct patterns depending on intent. For sequential processing (order matters, or you must not overwhelm a server), use for...of with await inside — each iteration waits. For parallel (independent items, want speed), map to an array of promises and Promise.all them. The trap: array.forEach with an async callback does not wait — forEach ignores the returned promises, so the surrounding function proceeds before any iteration completes, and errors go unhandled. Never use forEach for awaited work; use for...of or map+Promise.all.

Part 5 — await Yields the Thread (Event-Loop Recap)

async function demo() {
  console.log("1");
  await Promise.resolve();     // suspend here; schedule the rest as a microtask
  console.log("3");            // runs after the current synchronous code
}
console.log("0");
demo();
console.log("2");
// Output: 0, 1, 2, 3

From the Event Loop lesson: await does not block the thread — the async function runs synchronously up to the first await, then suspends and schedules its continuation as a microtask, returning control to the caller. So other code ("2") runs before the post-await code ("3"). This is why async functions don't freeze the UI: awaiting a slow request frees the thread to handle clicks and paints, and the function resumes when the promise settles. await is cooperative yielding, not blocking — a crucial distinction from threads in other languages.

Part 6 — Common Patterns & Gotchas

// 1. An async function ALWAYS returns a promise — even a plain return:
async function f() { return 42; }
f();               // Promise<42>, not 42 — you must await/then it

// 2. await a non-promise just returns it (wrapped-and-unwrapped) — harmless:
await 5;           // 5

// 3. Top-level await (in modules) — no wrapper function needed:
const data = await fetch("/api").then(r => r.json());   // at module top level

// 4. Don't await in a loop when you meant parallel (Part 3/4 trap).

// 5. Parallel with error isolation — Promise.allSettled (combinators lesson):
const results = await Promise.allSettled(ids.map(fetchUser));  // never rejects; per-item status

Key behaviours to remember: an async function always returns a promise, so callers must await or .then it (a frequent surprise — const x = f() gives you a promise, not the value). awaiting a non-promise simply returns the value. Modern modules support top-level await, letting you await at a module's top scope (useful for config/setup). And for parallel work where you don't want one failure to reject everything, Promise.allSettled gives per-item results — the Combinators lesson covers the full family.

Part 7 — Build It: a Concurrency-Limited Batch Fetcher

Real APIs rate-limit, so "fire all requests in parallel" (Promise.all over 1000 items) will get you throttled or crash the server. The professional pattern is bounded concurrency — run at most N at a time. Let's build it with async/await.

async function mapLimit(items, limit, asyncFn) {
  const results = new Array(items.length);
  let nextIndex = 0;

  async function worker() {
    while (nextIndex < items.length) {
      const i = nextIndex++;                 // claim the next item (single-threaded → no race)
      results[i] = await asyncFn(items[i], i);   // process it, await before taking another
    }
  }
  // Launch `limit` workers that pull from the shared queue until it's empty:
  await Promise.all(Array.from({ length: Math.min(limit, items.length) }, worker));
  return results;
}

// Fetch 100 users, but never more than 5 requests in flight at once:
const ids = Array.from({ length: 100 }, (_, i) => i + 1);
const users = await mapLimit(ids, 5, (id) => fetchUser(id));
console.log(users.length);   // 100 — fetched with controlled concurrency

The pattern: spawn limit "workers," each a loop that claims the next item from a shared index and awaits its processing before grabbing another. Because JavaScript is single-threaded (Event Loop lesson), nextIndex++ can't race — no locks needed. The result is exactly limit operations in flight at any moment, balancing speed against server limits — far better than the all-or-nothing Promise.all. This is essentially what libraries like p-limit provide, and it combines every idea in this lesson: async functions, await for backpressure, and Promise.all to run the workers concurrently.

Interview Questions

Q1 (beginner): What does an async function return, and what does await do?

An async function always returns a promise: its return value becomes the promise's resolved value, and a thrown error becomes a rejection — even async function f() { return 42; } gives a Promise<42>. await pauses the async function until the awaited promise settles, then evaluates to its resolved value (or throws its rejection, catchable with try/catch). await can only be used inside an async function or at a module's top level. It's syntactic sugar over promises — same behaviour, more readable.

Q2 (beginner): How do you handle errors with async/await?

With ordinary try/catch: wrap the awaited calls in a try block, and a rejected promise throws into the catch, where you handle or recover. A finally block runs regardless for cleanup. This unifies synchronous and asynchronous error handling under one familiar construct, which is a major readability advantage over chained .catch. As with promises, you either recover (return from the catch) or re-throw to let the caller handle it. Note some rejections (like fetch on HTTP 404/500) require you to throw manually, since fetch only rejects on network failure.

Q3 (mid): Explain the difference between sequential and parallel awaits, with a performance example.

Sequential awaits run one at a time — each await blocks the next line, so const a = await x(); const b = await y(); starts y() only after x() resolves. If x and y each take 300ms and are independent, that's 600ms wasted serialisation. Parallel means starting both before awaiting: const [a, b] = await Promise.all([x(), y()]) fires both immediately and awaits them together, finishing in ~300ms (the slower one). The rule: await sequentially only when a step genuinely needs the previous step's result; otherwise start independent operations concurrently and await them with Promise.all. Accidental sequential awaits on independent work are one of the most common real-world performance bugs.

Q4 (mid): Why doesn't array.forEach(async ...) work for awaited work?

forEach calls its callback for each element but ignores the callback's return value — so when the callback is async and returns a promise, forEach discards those promises and returns undefined immediately, without waiting for any of them. The surrounding code proceeds before the async work finishes (a fire-and-forget), and any rejections become unhandled. To process items sequentially, use a for...of loop with await inside (which does pause each iteration); to process them in parallel and wait for all, use array.map(async ...) to build a promise array and await Promise.all(...). The general lesson: only constructs that actually consume the returned promise (for...of+await, Promise.all) provide awaiting; forEach does not.

Q5 (senior): When is sequential processing actually correct, and how do you implement bounded concurrency?

Sequential is correct when: each step depends on the previous step's output (a true data dependency); ordering must be preserved and can't be reordered; or you must apply backpressure — not overwhelming a rate-limited API, a database connection pool, or memory by launching thousands of simultaneous operations. Fully parallel (Promise.all over a huge list) can trigger rate limits, exhaust connections, or spike memory. The middle ground — bounded concurrency — runs at most N operations at once: implement it by spawning N "worker" loops that each pull the next item from a shared index/queue and await processing it before claiming another, then Promise.all the workers. Because JS is single-threaded, the shared-index increment can't race, so no locks are needed. This gives controlled throughput — much faster than sequential, much safer than unbounded parallel. Libraries like p-limit/p-map provide it, but knowing how to write it (the worker-pool pattern) demonstrates real async fluency. Choosing the concurrency limit is a tuning decision based on the downstream resource's capacity (API rate limit, DB pool size). The senior framing: async concurrency isn't binary (sequential vs all-at-once) — the right answer is usually a bounded pool sized to the constraining resource.

Q6 (senior): Trace how async/await maps onto promises and microtasks, and why an unhandled rejection in an async function behaves as it does.

An async function is desugared into a state machine that returns a promise. It executes synchronously until the first await; there it takes the awaited value, wraps it via Promise.resolve (so awaiting a non-promise or a thenable is normalised), and registers a continuation as a microtask to resume when that promise settles — then returns the function's own pending promise to the caller. Each subsequent await similarly suspends and resumes via a microtask. So the code between awaits runs in microtask order relative to the rest of the program (Event Loop lesson): synchronous code and already-queued microtasks run in their order, and post-await continuations interleave as microtasks — always before pending macrotasks (timers). When the function returns, its promise fulfils with the return value; when it throws (or an await rejects and isn't caught), its promise rejects. An unhandled rejection — the async function's returned promise rejects and no .catch/try-catch anywhere consumes it — triggers the environment's unhandled-rejection handling: a console warning and unhandledrejection event in browsers, and in modern Node a process crash by default. This is why every async entry point needs error handling: because the rejection propagates through the returned promise, if nothing awaits or catches that promise, it surfaces as an unhandled rejection. Understanding the desugaring also explains subtle ordering — e.g. why two async functions' post-await code interleaves as microtasks, and why await of an already-resolved promise still defers the continuation to the next microtask tick rather than running it synchronously.

🏋️ Exercises

1 (easy): Rewrite a chain. Convert this promise chain to async/await with try/catch: 
function load() {
  return fetch("/api").then(r => r.json()).then(d => d.value).catch(e => null);
}
Solution 
async function load() {
  try {
    const r = await fetch("/api");
    const d = await r.json();
    return d.value;
  } catch (e) {
    return null;
  }
}

Each .then becomes an await; the .catch becomes a try/catch. Same behaviour, flatter reading.

2 (easy): Spot the promise. Why does this log Promise{...} instead of 10, and how do you fix it? 
async function double(n) { return n * 2; }
console.log(double(5));
Solution

An async function always returns a promise, so double(5) is a Promise, not 10. Fix by awaiting or chaining: console.log(await double(5)) (inside an async function) or double(5).then(console.log).

3 (medium): Fix the slow loader. This takes ~900ms for three independent 300ms calls. Make it ~300ms. 
async function load() {
  const a = await fetchA();
  const b = await fetchB();
  const c = await fetchC();
  return [a, b, c];
}
Solution 
async function load() {
  return Promise.all([fetchA(), fetchB(), fetchC()]);  // all start immediately, run concurrently
}

The calls are independent, so start them all at once and await together with Promise.all — total time becomes the slowest single call (~300ms) instead of the sum.

4 (medium): Sequential vs parallel loop. Given ids = [1,2,3] and fetchUser(id), write both a sequential version (order preserved, one at a time) and a parallel version.
Solution 
// Sequential:
async function seq(ids) {
  const out = [];
  for (const id of ids) out.push(await fetchUser(id));
  return out;
}
// Parallel:
async function par(ids) {
  return Promise.all(ids.map(id => fetchUser(id)));
}

for...of+await runs one at a time (use when order/backpressure matters); map+Promise.all runs all concurrently (use for independent items). Both preserve result order.

5 (hard): Timeout an async operation. Write an async fetchWithTimeout(url, ms) that throws if the fetch takes longer than ms, using Promise.race.
Solution 
async function fetchWithTimeout(url, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error("timeout")), ms));
  const res = await Promise.race([fetch(url), timeout]);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

Promise.race settles with whichever finishes first; if the timeout rejects before the fetch resolves, the await throws the timeout error into the caller's try/catch. (For true cancellation of the underlying request, combine with AbortController — Fetch lesson.)

6 (hard): Bounded concurrency. Implement mapLimit(items, limit, fn) that runs at most limit async operations at once and preserves result order. Prove no more than limit run concurrently.
Solution 
async function mapLimit(items, limit, fn) {
  const results = new Array(items.length);
  let i = 0, active = 0, maxActive = 0;
  async function worker() {
    while (i < items.length) {
      const idx = i++;
      active++; maxActive = Math.max(maxActive, active);   // track peak concurrency
      results[idx] = await fn(items[idx], idx);
      active--;
    }
  }
  await Promise.all(Array.from({ length: Math.min(limit, items.length) }, worker));
  console.assert(maxActive <= limit, "exceeded limit!");
  return results;
}

Exactly limit workers pull from a shared index; each awaits its task before claiming another, so at most limit are ever in flight (the maxActive assertion confirms it). Results are stored by original index, preserving order. This is the production pattern behind p-limit, built from this lesson's primitives.

🎯 Key Takeaways

  • async/await is readable syntax over promises — an async function always returns a promise; await unwraps one (or throws its rejection).
  • Handle async errors with ordinary try/catch/finally — unifying sync and async error flow.
  • Independent awaits should run in parallel with Promise.all, not sequentially — needless serialisation is the top async performance bug.
  • Loop async work with for...of+await (sequential) or map+Promise.all (parallel); never forEach (it doesn't wait).
  • await yields the thread (schedules a microtask continuation) — it doesn't block, so the UI stays responsive.
  • For large workloads use bounded concurrency (a worker pool), not unbounded Promise.all, to respect rate limits and memory.
