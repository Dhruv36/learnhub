Objects Deep Dive Objects & Data

What you'll master: how objects really store data (properties with hidden attributes), the full toolkit for creating, reading, iterating, and copying them, the shallow-vs-deep copy problem that causes so many mutation bugs, property descriptors and getters/setters, and the key-ordering and iteration rules people constantly get wrong. Prerequisites: Types & Coercion (objects are the one reference type) and Variables (reference semantics). Prototypes get their own lesson; here we focus on objects as data containers.

Part 1 — Creating Objects & Property Shorthand

const name = "Ada", age = 30;
const user = {
  name,                          // shorthand for name: name
  age,
  "full name": "Ada Lovelace",   // keys with spaces/special chars need quotes
  [`dynamic_${age}`]: true,      // COMPUTED key — evaluated at creation
  greet() { return `Hi ${this.name}`; },   // method shorthand
  get initials() { return this.name[0]; }, // getter (Part 6)
};
user.name;             // "Ada"           — dot access
user["full name"];     // "Ada Lovelace"  — bracket access (required for non-identifier keys)
user[`dynamic_30`];    // true

Objects are collections of key→value properties. Keys are always strings or Symbols (a numeric key like obj[1] is coerced to the string "1"). Modern syntax gives you property shorthand ({ name } when the variable and key share a name), method shorthand, and computed keys ([expr]:) for dynamic property names. Use dot access by default; bracket access when the key is in a variable, is computed, or isn't a valid identifier.

Part 2 — Reading, Checking Existence, and Deleting

const o = { a: 1, b: undefined };
o.a;                       // 1
o.c;                       // undefined — missing property AND a property set to undefined both read undefined!
"a" in o;                  // true  — the `in` operator: does the key EXIST (incl. inherited)?
"c" in o;                  // false
o.hasOwnProperty("b");     // true  — OWN property (not inherited); b exists though its value is undefined
Object.hasOwn(o, "b");     // true  — modern, safer replacement for hasOwnProperty (Part 8)
o.b === undefined;         // true  — can't distinguish "missing" from "set to undefined" by value alone
delete o.a;                // removes the property entirely; now "a" in o === false

A crucial subtlety: reading a missing property and reading a property explicitly set to undefined both yield undefined — so value comparison can't tell you whether a key exists. Use the in operator (checks own and inherited keys) or Object.hasOwn(obj, key) (own keys only, the modern replacement for hasOwnProperty) when existence matters. delete genuinely removes a property (unlike setting it to undefined), though it can deoptimise objects in hot paths — prefer reconstructing without the key, or a Map, when you delete frequently.

Part 3 — Iterating Objects (Four Ways, One Right Default)

const o = { a: 1, b: 2, c: 3 };
Object.keys(o);      // ["a","b","c"]           — own enumerable keys
Object.values(o);    // [1, 2, 3]               — own enumerable values
Object.entries(o);   // [["a",1],["b",2],["c",3]] — own enumerable [key,value] pairs
for (const [k, v] of Object.entries(o)) { }     // the idiomatic object loop

for (const k in o) { }   // ⚠ includes INHERITED enumerable keys — needs a hasOwn guard

Object.entries + for...of is the default way to loop an object — it gives you keys and values together, only own enumerable properties, and works with destructuring. Object.keys/values when you need just one side. Avoid bare for...in: it also walks inherited enumerable properties (from the prototype chain), so it needs an Object.hasOwn(o, k) guard to be safe — an easy footgun that Object.keys sidesteps entirely.

🧠 Key order is defined but quirky. Integer-like keys come first in ascending numeric order, then string keys in insertion order, then Symbols. So { 2: "a", 1: "b", x: "c" } iterates as 1, 2, x — the numeric keys are reordered! If order matters and keys might be numeric strings, use a Map (Collections lesson), which preserves true insertion order for all key types.

Part 4 — Copying: Shallow vs Deep (the Big One)

const original = { name: "Ada", address: { city: "London" } };

// SHALLOW copy — top level copied, nested objects still SHARED:
const shallow = { ...original };              // spread (idiomatic)
const shallow2 = Object.assign({}, original); // older equivalent
shallow.name = "Bob";                          // ✅ independent — top-level primitive
shallow.address.city = "Paris";                // 💥 also changes original.address.city! (shared ref)

// DEEP copy — fully independent, nested included:
const deep = structuredClone(original);        // built-in, modern — the right default
deep.address.city = "Paris";                    // original untouched

This is the source of countless bugs. { ...obj } and Object.assign make a shallow copy: primitive top-level values are copied, but nested objects/arrays are still shared references (Types lesson) — mutating a nested object through the copy mutates the original too. For genuine independence, structuredClone(obj) is the modern built-in deep copy (handles nested objects, arrays, Dates, Maps, Sets, and cycles). The old JSON.parse(JSON.stringify(obj)) trick works only for plain JSON-safe data — it drops functions, undefined, and Symbols, mangles Dates into strings, and throws on BigInt and cycles. Reach for structuredClone.

Part 5 — Merging & Immutable Updates

const defaults = { theme: "light", fontSize: 14 };
const overrides = { fontSize: 16 };
const merged = { ...defaults, ...overrides };   // { theme:"light", fontSize:16 } — later wins

// Immutable update — produce a NEW object, don't mutate (the React/Redux pattern):
const state = { user: { name: "Ada", age: 30 }, loggedIn: true };
const next = { ...state, user: { ...state.user, age: 31 } };   // deep-ish: spread each level you touch
// state is UNCHANGED; next has the updated age

Spread merges objects left-to-right — later sources overwrite earlier keys — which is the clean way to layer overrides on defaults. For immutable updates (the foundation of React state and Redux reducers), you spread at every level you change to produce a new object while leaving the original untouched — note you must spread each nested level you modify ({ ...state.user, age: 31 }), because a single top-level spread would share the nested user reference. This "never mutate, always replace" discipline is why change detection (React's === comparison of state) works, and why deep nesting makes manual immutable updates verbose — the reason libraries like Immer exist.

Part 6 — Getters, Setters & Property Descriptors

const temp = {
  _celsius: 20,
  get celsius() { return this._celsius; },
  set celsius(v) { if (v < -273) throw new RangeError("below absolute zero"); this._celsius = v; },
  get fahrenheit() { return this._celsius * 9/5 + 32; },   // COMPUTED, read like a property
};
temp.celsius;         // 20   — invokes the getter, no ()
temp.fahrenheit;      // 68   — computed on read
temp.celsius = 25;    // invokes the setter (validation runs)

Getters and setters let a property be backed by a function while still being accessed like a plain property (temp.celsius, not temp.getCelsius()) — useful for computed/derived values and for validation on assignment. Under the hood, every property has a descriptor controlling its behaviour:

Object.getOwnPropertyDescriptor({ a: 1 }, "a");
// { value: 1, writable: true, enumerable: true, configurable: true }

// Define a non-enumerable, read-only property:
const obj = {};
Object.defineProperty(obj, "id", { value: 42, writable: false, enumerable: false });
obj.id = 99;              // silently ignored (throws in strict mode) — writable:false
Object.keys(obj);         // [] — enumerable:false hides it from iteration

A property's descriptor has (for data properties) value, writable (can it be reassigned?), enumerable (does it show in Object.keys/for...in/ spread?), and configurable (can it be deleted/redefined?). Plain literal properties default all three flags to true; Object.defineProperty lets you create hidden or locked properties (how many built-in and framework properties are defined). You rarely need this directly, but understanding it demystifies why some properties don't appear in loops or can't be overwritten.

Part 7 — Freezing & Sealing

const config = Object.freeze({ api: "/v1", retries: 3 });
config.retries = 5;        // ignored (throws in strict mode) — frozen: no writes, no adds, no deletes
config.newKey = 1;         // ignored
Object.isFrozen(config);   // true
// ⚠ SHALLOW: Object.freeze({ nested: {} }).nested is still mutable (Variables lesson's deepFreeze)

Object.seal(obj);          // can't ADD or DELETE keys, but CAN modify existing writable ones
Object.preventExtensions(obj);  // can't ADD keys; existing ones fully mutable

Three levels of lockdown: preventExtensions (no new keys), seal (no add/delete, but existing values mutable), and freeze (fully read-only). All are shallow — nested objects remain mutable — so deep immutability still needs recursion (the Variables lesson's deepFreeze). freeze is genuinely useful for constants you want to protect against accidental mutation (config objects, enums), and as a development-time guard.

Part 8 — Objects vs Maps: Choosing the Right Container

Use a plain object for fixed-shape records (a user, a config, anything you'd serialise to JSON) and a Map for a genuine dictionary — dynamic keys, non-string keys, frequent insertion/deletion, or when you need reliable ordering and a .size. A subtle security reason to prefer Map for user-controlled keys: plain objects inherit from Object.prototype, so a key like "__proto__" or "constructor" can trigger prototype-pollution bugs (Security lesson) — Map has no such keys. For a null-prototype plain object with no inherited keys, use Object.create(null).

Part 9 — Build It: a Safe Deep-Merge for Config

Real apps merge nested configuration (defaults ← user config ← environment overrides). A naive spread is shallow, clobbering nested objects entirely. Let's build a correct recursive merge that respects nested structure.

function deepMerge(target, source) {
  const out = { ...target };                    // shallow copy of this level
  for (const key of Object.keys(source)) {
    const s = source[key], t = target[key];
    if (isPlainObject(s) && isPlainObject(t)) {
      out[key] = deepMerge(t, s);               // recurse into matching nested objects
    } else {
      out[key] = s;                             // primitive, array, or new key → source wins
    }
  }
  return out;
}
function isPlainObject(v) {
  return v !== null && typeof v === "object" && !Array.isArray(v);
}

const defaults = { ui: { theme: "light", density: "cozy" }, retries: 3 };
const user     = { ui: { theme: "dark" }, retries: 5 };
deepMerge(defaults, user);
// { ui: { theme: "dark", density: "cozy" }, retries: 5 }  — density PRESERVED, theme overridden

The key correctness point: when both sides have a plain object at the same key, we recurse and merge them rather than replacing — so density survives even though user.ui only specified theme. A shallow { ...defaults, ...user } would have replaced the entire ui object, losing density. Arrays and primitives take the source value wholesale (a deliberate policy choice — some mergers concatenate arrays instead). This is the shape of what config libraries do; every concept — spread copies, recursion, type-checking with the null guard — comes from this lesson.

Interview Questions

Q1 (beginner): How do you iterate an object's key/value pairs, and why prefer it over for...in?

Use for (const [key, value] of Object.entries(obj)), or Object.keys/Object.values for one side. These return only the object's own enumerable properties. for...in also iterates inherited enumerable properties from the prototype chain, so it can surprise you with keys you didn't define, and needs an Object.hasOwn(obj, key) guard to be safe. Object.entries avoids the issue entirely and pairs cleanly with destructuring, so it's the idiomatic default.

Q2 (beginner): What's the difference between a shallow and a deep copy?

A shallow copy ({ ...obj }, Object.assign) duplicates only the top-level properties; nested objects and arrays are copied by reference, so they're still shared — mutating a nested value through the copy also changes the original. A deep copy duplicates every level, producing a fully independent structure. The modern built-in deep copy is structuredClone(obj), which handles nested objects, Dates, Maps/Sets, and cyclic references. The classic bug is shallow-copying and then mutating nested data, expecting the original to be unaffected.

Q3 (mid): Explain object property descriptors and give a use for enumerable: false.

Every property has a descriptor: for a data property, value, writable (can it be reassigned), enumerable (does it appear in Object.keys/ for...in/spread), and configurable (can it be deleted or redefined). Literal properties default all flags to true; Object.defineProperty lets you set them explicitly. A use for enumerable: false: attaching metadata or internal bookkeeping to an object (an id, a cache) that you don't want showing up when the object is iterated or serialised — many built-in prototype methods are non-enumerable for exactly this reason, so they don't pollute for...in loops.

Q4 (mid): When would you use a Map instead of a plain object?

Use a Map when: keys aren't strings/Symbols (e.g. objects or numbers as keys); you need guaranteed insertion order regardless of key type (plain objects reorder integer-like keys numerically); you add and remove entries frequently (Maps are optimised for it and give a .size); or the keys come from user input and you want to avoid prototype-pollution footguns (a plain object inherits Object.prototype, so a "__proto__" key is dangerous, whereas a Map has no prototype keys). Use a plain object for fixed-shape records, config, and anything you serialise to JSON, since Maps don't stringify to JSON directly.

Q5 (senior): Why does JSON.parse(JSON.stringify(obj)) fail as a general deep-clone, and what does structuredClone do better?

The JSON round-trip is lossy because JSON can only represent a subset of JS values. It: drops undefined properties and functions entirely; converts Date objects to strings (losing the Date type); loses Map, Set, RegExp, and typed arrays (turning them into plain objects/empty objects); throws on BigInt; throws on cyclic references; and mishandles special numbers (NaN/Infinity become null). It also can't preserve object identity when the same nested object appears twice (it duplicates it). structuredClone uses the structured-clone algorithm, which handles nested objects and arrays, preserves Date/Map/Set/ RegExp/typed arrays/ArrayBuffer, supports cyclic references correctly, and maintains shared-reference identity within the graph. Its limits: it can't clone functions, DOM nodes, or class prototype chains (the clone is a plain object, losing the class), and it throws on those. So structuredClone is the right default for cloning data, but for class instances or functions you need a custom clone or a library.

Q6 (senior): Design an immutable state-update utility for deeply nested state. What are the trade-offs of manual spread vs a library like Immer?

Manual immutable updates require spreading every level along the path you change: { ...state, a: { ...state.a, b: { ...state.a.b, c: newVal } } }. This is explicit, has zero dependencies, and makes the changed path obvious — but it's verbose, error-prone (forget one spread and you mutate shared state or lose sibling keys), and scales badly with depth. A helper can reduce boilerplate: a path-based setter setIn(state, ["a","b","c"], val) that recursively rebuilds only the objects along the path (structurally sharing untouched branches for performance and cheap === change detection). Immer takes a different approach: you write apparently-mutative code against a Proxy "draft" (draft.a.b.c = val), and it produces a new immutable state, structurally sharing everything you didn't touch. Trade-offs: Immer is far more ergonomic and less bug-prone for deep updates, preserves structural sharing automatically (good for memoisation and React re-render minimisation), but adds a dependency, a small runtime cost from the Proxy, and a layer of "magic" that can obscure what's happening. Manual/helper approaches are dependency-free and transparent but tedious and easy to get subtly wrong at depth. The senior call: for shallow state, plain spread; for moderately nested state, a small tested setIn helper; for deeply nested, frequently-updated state (typical Redux), Immer's ergonomics and correctness usually justify the dependency — which is why Redux Toolkit bundles it by default.

🏋️ Exercises

1 (easy): Existence check. Given const o = { a: 1, b: undefined }, write expressions that distinguish "key b exists but is undefined" from "key c doesn't exist."
Solution 
Object.hasOwn(o, "b");   // true  — exists
Object.hasOwn(o, "c");   // false — missing
o.b === undefined;       // true for BOTH b and c — value can't distinguish them

Only an existence operator (Object.hasOwn or in) can tell "present but undefined" apart from "absent," because both read as undefined by value.

2 (easy): Invert an object. Write invert(obj) that swaps keys and values: { a: "x", b: "y" } → { x: "a", y: "b" }.
Solution 
const invert = obj =>
  Object.fromEntries(Object.entries(obj).map(([k, v]) => [v, k]));
invert({ a: "x", b: "y" });   // { x: "a", y: "b" }

Object.entries → map each pair swapped → Object.fromEntries rebuilds the object. Note: values must be valid (string) keys, and duplicate values collide — last wins.

3 (medium): Shallow vs deep proof. Demonstrate that { ...obj } shares nested objects but structuredClone doesn't, using a two-level object.
Solution 
const obj = { a: 1, nested: { b: 2 } };
const shallow = { ...obj };
shallow.nested.b = 99;
obj.nested.b;              // 99 — shared! shallow copy

const deep = structuredClone(obj);
deep.nested.b = 7;
obj.nested.b;              // still 99 — independent

The spread copies a by value but nested by reference, so mutating shallow.nested hits the original. structuredClone duplicates nested too, fully decoupling the copy.

4 (medium): pick / omit. Write pick(obj, keys) returning a new object with only the listed keys, and omit(obj, keys) returning one without them.
Solution 
const pick = (obj, keys) =>
  Object.fromEntries(keys.filter(k => k in obj).map(k => [k, obj[k]]));
const omit = (obj, keys) =>
  Object.fromEntries(Object.entries(obj).filter(([k]) => !keys.includes(k)));

pick({ a: 1, b: 2, c: 3 }, ["a", "c"]);   // { a: 1, c: 3 }
omit({ a: 1, b: 2, c: 3 }, ["b"]);        // { a: 1, c: 3 }

Both build from entries/fromEntries. pick filters the requested keys (guarding existence with in); omit filters out the excluded keys. These are staples of lodash — worth being able to write from scratch.

5 (hard): Validated property. Create an object with a percent property that can only be set to 0–100 (throwing otherwise) and a read-only id that doesn't appear in Object.keys.
Solution 
const widget = {
  _percent: 0,
  get percent() { return this._percent; },
  set percent(v) {
    if (v < 0 || v > 100) throw new RangeError("percent must be 0–100");
    this._percent = v;
  }
};
Object.defineProperty(widget, "id", { value: crypto.randomUUID(), writable: false, enumerable: false });

widget.percent = 50;      // ok
// widget.percent = 150;  // RangeError
widget.id = "x";          // ignored (writable:false)
Object.keys(widget);      // ["_percent"] — id hidden (enumerable:false)

The setter enforces the range invariant on every assignment; defineProperty creates id as read-only and non-enumerable so it's protected and hidden from iteration/ serialisation.

6 (hard): Deep merge with array policy. Extend the Part 9 deepMerge so that when both sides have an array at a key, they concatenate (rather than the source replacing). Prove nested objects still merge recursively.
Solution 
function deepMerge(target, source) {
  const out = { ...target };
  for (const key of Object.keys(source)) {
    const s = source[key], t = target[key];
    if (Array.isArray(s) && Array.isArray(t)) {
      out[key] = [...t, ...s];                 // concat policy
    } else if (isPlainObject(s) && isPlainObject(t)) {
      out[key] = deepMerge(t, s);
    } else {
      out[key] = s;
    }
  }
  return out;
}
const isPlainObject = v => v !== null && typeof v === "object" && !Array.isArray(v);

deepMerge(
  { tags: ["a"], ui: { theme: "light", size: "m" } },
  { tags: ["b"], ui: { theme: "dark" } }
);
// { tags: ["a","b"], ui: { theme:"dark", size:"m" } }

The array branch concatenates both sides; the object branch recurses (preserving size while overriding theme); everything else takes the source. Making the array policy explicit is important — merge semantics for arrays (concat vs replace vs union) is a genuine design decision, not a default anyone should assume.

🎯 Key Takeaways

  • Objects are key→value stores (string/Symbol keys); use shorthand, computed keys, and bracket access for dynamic/non-identifier keys.
  • Existence needs Object.hasOwn/in — value comparison can't distinguish missing from undefined.
  • Object.entries + for...of is the safe iteration default; bare for...in walks inherited keys; integer keys reorder numerically.
  • Spread/Object.assign are shallow — nested objects stay shared; structuredClone is the modern deep copy (JSON round-trip is lossy).
  • Immutable updates spread every level you change; descriptors control writable/enumerable/configurable; freeze/seal lock objects (shallowly).
  • Plain objects for fixed-shape records/JSON; Map for dynamic dictionaries, non-string keys, reliable order, and user-controlled keys (prototype-pollution safety).
