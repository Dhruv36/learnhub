Template-Driven Forms Forms

What you'll master: Angular's simpler forms flavor — ngModel, ngForm, validation attributes, control state classes (touched/dirty/valid), and submit handling — plus an honest framework for choosing template-driven vs reactive. Prerequisites: Data Binding (two-way binding) and Directives (ngModel is one). Template-driven forms are underrated for what they're good at — and knowing their ceiling is what makes the next lesson's machinery feel justified rather than bureaucratic.

Part 1 — The Model: Directives Build the Form For You

<!-- Import FormsModule in the component; then a form materializes from the template: -->
<form #f="ngForm" (ngSubmit)="save(f)">
  <input name="email" [(ngModel)]="model.email" required email>
  <input name="age"   [(ngModel)]="model.age" type="number" min="18">
  <button [disabled]="f.invalid">Save</button>
</form>

What the directives conjure behind the scenes:
  <form>            → an NgForm instance (root FormGroup)   — exported via #f="ngForm"
  [(ngModel)] + name → an NgModel per input (a FormControl), REGISTERED into the form
  required, email…   → validators attached to that control

So: the TEMPLATE is the single source of the form's structure. Angular derives the
model object (f.value → { email: '…', age: … }), validity (f.valid), and state
tracking — you never construct anything in the class.

Template-driven means the template declares, directives assemble: every ngModel with a name registers a control into the enclosing ngForm, validators arrive as HTML-looking attributes, and the class holds only a plain data object. It's the two-way binding story from Data Binding upgraded with registration and validation. The trade is visibility: the form's structure exists only as a side effect of template directives — great while forms are simple, limiting when you need to inspect, test, or reshape the form in code (the reactive lesson's opening argument).

Part 2 — Control State: the Six Flags and Their CSS Classes

Every control (and the form itself) tracks three state pairs:

  touched / untouched     has the user BLURRED it yet?           .ng-touched / .ng-untouched
  dirty   / pristine      has the VALUE ever changed?            .ng-dirty   / .ng-pristine
  valid   / invalid       do all validators pass?                .ng-valid   / .ng-invalid

<!-- The UX-correct error display — wrong-timing errors are the #1 form annoyance: -->
<input name="email" [(ngModel)]="model.email" #email="ngModel" required email>
@if (email.invalid && email.touched) {         <!-- only nag AFTER they've left the field -->
  <p class="error">
    @if (email.hasError('required')) { Email is required. }
    @else if (email.hasError('email')) { That doesn't look like an email. }
  </p>
}

/* Or style purely with the auto-applied classes: */
input.ng-invalid.ng-touched { border-color: #e53935; }

The state flags encode form UX best practice directly: invalid alone says "there's a problem", but invalid && touched says "there's a problem the user has had a chance to cause" — which is when showing red is fair. (Strict teams use invalid && (touched || submitted) so submit reveals everything.) Exporting the directive with #email="ngModel" gives the template a handle on the control's state and errors — the same template-reference-variable mechanics from Data Binding, pointed at a directive instead of a DOM element. The auto-applied classes make pure-CSS validation styling possible with zero template logic.

Part 3 — Validation Attributes & Error Objects

required                    → hasError('required')      value is empty
email                       → hasError('email')         not email-shaped
minlength="3"               → hasError('minlength')     {requiredLength: 3, actualLength: 1}
maxlength="20"              → (mostly prevented by the browser; error if set programmatically)
min="18" / max="99"         → hasError('min') / ('max') numeric bounds
pattern="[A-Z]{2}\d{4}"     → hasError('pattern')       regex mismatch

// Each failing validator writes a key into control.errors:
email.errors    // { required: true }  or  { email: true }  or null when valid
// hasError('key') is the template-friendly accessor.

<!-- Cross-field feel without machinery — compare via template refs: -->
<input name="password" [(ngModel)]="model.password" #pw="ngModel" required minlength="8">
<input name="confirm"  [(ngModel)]="model.confirm"  #cf="ngModel" required>
@if (cf.touched && cf.value !== pw.value) {
  <p class="error">Passwords don't match.</p>
}
<!-- honest caveat: this shows the MESSAGE but doesn't mark the FORM invalid —
     real cross-field validation wants a form-level validator → form-validation.html -->

Validators are the standard HTML validation vocabulary, intercepted by Angular directives (Angular disables native browser validation via novalidate and runs its own — consistent styling and messages across browsers). The errors object convention — one key per failed rule, with detail payloads for parametrized rules like minlength — carries through everything: the same shape appears in reactive forms and your custom validators. Note the password-confirm example's ceiling: template refs can display cross-field messages, but the form's valid flag doesn't know — the first crack where template-driven starts asking for the next lesson.

Part 4 — Submit, Reset & the Model Object

@Component({ imports: [FormsModule], template: `
  <form #f="ngForm" (ngSubmit)="save(f)">
    <input name="name"  [(ngModel)]="model.name" required>
    <input name="email" [(ngModel)]="model.email" required email>
    <select name="plan" [(ngModel)]="model.plan">
      <option value="free">Free</option><option value="pro">Pro</option>
    </select>
    <label><input type="checkbox" name="tos" [(ngModel)]="model.tos" required> I agree</label>
    <button [disabled]="f.invalid || saving()">{{ saving() ? 'Saving…' : 'Sign up' }}</button>
  </form>
`})
export class SignupComponent {
  model = { name: '', email: '', plan: 'free', tos: false };   // plain object = the form's data
  saving = signal(false);

  save(f: NgForm) {
    if (f.invalid) {                          // belt-and-braces (button disabling can be bypassed)
      Object.values(f.controls).forEach(c => c.markAsTouched());  // reveal ALL errors on submit
      return;
    }
    this.saving.set(true);
    this.api.signup(this.model).subscribe({
      next: () => { f.resetForm({ plan: 'free' }); this.saving.set(false); },
      error: () => this.saving.set(false),
    });
  }
}

(ngSubmit) fires on Enter and button clicks with preventDefault handled — always use it over (click) on the button (keyboard submission is accessibility, not garnish). The submit handler shows three production habits: re-check validity in code (disabled buttons are a UX hint, not security — and note the server must validate regardless, the trust-boundary rule); mark-all-touched on failed submit so every error becomes visible at once; and resetForm(defaults) — which resets values and the touched/dirty state (plain reset() on values alone leaves stale red borders behind).

Part 5 — ngModelGroup & Structuring Larger Templates

<form #f="ngForm" (ngSubmit)="save(f)">
  <fieldset ngModelGroup="address" #addr="ngModelGroup">
    <legend>Shipping address</legend>
    <input name="street" [(ngModel)]="model.address.street" required>
    <input name="city"   [(ngModel)]="model.address.city" required>
    <input name="zip"    [(ngModel)]="model.address.zip" required pattern="\d{5}">
  </fieldset>
  @if (addr.invalid && addr.touched) { <p class="error">Please complete the address.</p> }
</form>

f.value  →  { street-level flat?  NO:
  { address: { street: '…', city: '…', zip: '…' } }     ← groups nest the value object
}
// ngModelGroup gives you: nested value shape, group-level valid/touched state,
// and a place to hang group-level messages. It mirrors FormGroup nesting in reactive forms.

ngModelGroup is the structuring tool template forms offer: it nests the value object (matching your API payload shape), aggregates state (addr.invalid = any child invalid), and pairs naturally with <fieldset>/<legend> for accessible grouping. What template-driven forms conspicuously lack is a clean dynamic-list story — "add another phone number" rows require looping with index-suffixed name attributes and fighting the registration model; that use case is FormArray's home turf in the reactive lesson, and it's the second crack in the ceiling.

Part 6 — Deep Dive: Choosing a Forms Flavor Honestly

TEMPLATE-DRIVEN shines:                      REACTIVE earns its ceremony:
  login, signup, contact, feedback             multi-step wizards
  search/filter bars                           dynamic rows (FormArray)
  settings toggles                             cross-field validation as form logic
  admin CRUD with flat models                  value-stream reactions (valueChanges pipelines)
  — few fields, static shape,                  programmatic get/set/enable/disable
    validation = per-field attributes          unit-testing form logic WITHOUT the DOM
                                               typed form contracts shared with services

The real decision inputs:
  · SHAPE: static → template; dynamic/conditional structure → reactive
  · LOGIC: per-field checks → template; cross-field/async/derived → reactive
  · TESTING: template forms are tested THROUGH the DOM; reactive logic tests as plain code
  · TEAM: mixed codebases are fine — flavor per form, not per app

The community pendulum ("real apps use reactive forms") overshoots: a login form built with FormBuilder, typed controls, and subscription plumbing is more code for zero more capability — ceremony without payoff. The honest heuristic is the table above: template-driven for static-shape, per-field-validated forms (which is most forms in a typical app); reactive when structure is dynamic, validation is relational, or form logic deserves DOM-free unit tests. Knowing both — and switching flavors per form without ideology — is the actual senior position.

Senior framing: forms are where accessibility review concentrates, whatever the flavor: every input needs a real <label for> (placeholder text is not a label — it vanishes on input and fails contrast); errors should be programmatically associated (aria-describedby pointing at the error element, aria-invalid on the control) so screen readers announce them; groups want fieldset/legend; and the submit button must be a real <button> in the form for Enter-key submission. Also name the signals-era horizon honestly: signal-based forms are on Angular's roadmap and will eventually unify the flavors around signal state — the state flags, validator vocabulary, and UX timing rules you learned here transfer wholesale.

Interview Questions

Q1 (Beginner): How does a template-driven form come into existence — what do ngForm and ngModel actually do?

Importing FormsModule activates directives that assemble a form model from the template: the <form> element gets an NgForm (a root FormGroup); each [(ngModel)] input with a name creates an NgModel (wrapping a FormControl) and registers it into that group under its name; validation attributes attach validators to the control. The class holds only a plain data object — structure, value aggregation (f.value), validity, and state tracking are all derived by the directives from the template.

Q2 (Beginner): Explain touched vs dirty, and why error display conventionally requires invalid && touched.

touched = the user has blurred the control at least once (focus left it); dirty = the value has been changed from its initial state — independent flags (you can be touched-but-pristine by tabbing through, or dirty-but-untouched while still typing in the field). Errors show on invalid && touched because flagging a field the user hasn't finished (or even visited) punishes them for the form's initial emptiness; blur signals "I'm done here — now feedback is fair". Submit then marks all touched so remaining errors reveal at once.

Q3 (Mid): What does #email="ngModel" do, and how does it differ from #email alone?

#email alone captures the DOM element (HTMLInputElement) — good for .value or .focus(). #email="ngModel" uses the directive's exportAs name to capture the NgModel directive instance instead — exposing Angular's control state: email.invalid, email.touched, email.errors, email.hasError('required'). That's the mechanism enabling per-field error UI in template forms; the same exportAs pattern appears across Angular (#f="ngForm", routerLinkActive, Material directives).

Q4 (Mid): Where's the trust boundary in client-side form validation? What is it actually for?

Client validation is UX, not security: it gives immediate feedback and reduces pointless server round-trips, but everything in the browser — disabled buttons, validators, the JS itself — is user-controllable (devtools, curl). The server must re-validate every input as the actual trust boundary (injection, type, range, authorization checks). The practical division: client validates for helpfulness (format hints, required prompts, instant feedback), server validates for correctness and safety, and the two rule sets should derive from one source where possible to avoid drift.

Q5 (Senior): Make the honest case for template-driven forms in a modern codebase — when is reactive the wrong default?

For static-shape forms with per-field validation — login, contact, settings, most CRUD — template forms deliver identical UX with a fraction of the code: no FormBuilder assembly, no manual subscription plumbing, validators as readable attributes, and the model object already matching the API payload. Reactive forms spend their complexity budget buying capabilities those forms never use (dynamic structure, relational validation, DOM-free logic tests, valueChanges pipelines). Defaulting to reactive everywhere is ceremony-as-policy — the senior move is flavor-per-form: template until a form exhibits a reactive-shaped need (FormArray rows, cross-field rules, wizard state), then switch that form. Both flavors share the state model and validator concepts, so the switching cost is low.

Q6 (Senior): A template-driven signup form needs: password-confirm matching that blocks submission, a username-availability check against the API, and an "add another email" repeater. Assess each requirement.

These three requirements are precisely the template-driven ceiling. Password-confirm as a blocking rule needs a cross-field validator on the group — template refs can render a message but don't affect f.valid; doable via a custom directive-based validator on ngModelGroup, but that's more machinery than the reactive equivalent. Username availability is an async validator — again directive-wrapped in template forms, first-class in reactive. The email repeater is dynamic structure — FormArray territory, which template forms handle only with index-named hacks. Verdict: this form has outgrown the flavor; rebuild it reactive (one form, not the whole app), where all three are native features — next lesson and the one after build exactly these.

🎯 Key Takeaways

  • Template-driven = directives assemble the form from the template: ngForm root, ngModel+name registers controls, validation attributes attach validators — the class keeps a plain model object.
  • State flags drive UX: show errors on invalid && touched, mark-all-touched on submit, resetForm() to clear values and state; .ng-* classes enable pure-CSS styling.
  • Errors follow one convention everywhere: errors object keyed by failed rule, read via hasError() — identical in reactive forms and custom validators.
  • (ngSubmit) + real buttons + real labels + aria-wired errors: form accessibility is non-negotiable review material.
  • Client validation is UX; the server is the trust boundary — always revalidate server-side.
  • Choose flavor per form: template for static shape + per-field rules (most forms); switch to reactive when you hit dynamic structure (FormArray), cross-field/async validation, or DOM-free testing needs.
