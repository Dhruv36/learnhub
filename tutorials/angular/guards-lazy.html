Guards, Resolvers & Lazy Loading Routing & HTTP

What you'll master: functional route guards (CanActivate, CanDeactivate, CanMatch), redirect-vs-block decisions, resolvers and the data-loading trade-off, and lazy loading — loadComponent/loadChildren, route-level providers, preloading strategies, and how route config becomes your bundle map. Prerequisites: Routing & Navigation and DI (guards are injection contexts). This is where the route table becomes the app's access-control layer and its performance architecture.

Part 1 — Functional Guards: Navigation Checkpoints

// A guard is a FUNCTION the router calls before completing a navigation:
export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);              // guards run in an injection context
  const router = inject(Router);

  if (auth.isLoggedIn()) return true;            // allow
  return router.createUrlTree(['/login'], {      // REDIRECT (don't just block!)
    queryParams: { returnUrl: state.url },       // so login can send them back
  });
};

// Wire it — guards are arrays (all must pass):
{ path: 'account', component: AccountComponent, canActivate: [authGuard] },
{ path: 'admin', canActivate: [authGuard, roleGuard('admin')],   // composition
  loadChildren: () => import('./admin/admin.routes') },

// Parameterized guards = factories returning guards (the closure-factory pattern again):
export const roleGuard = (role: string): CanActivateFn => () =>
  inject(AuthService).roles().includes(role)
    ? true
    : inject(Router).createUrlTree(['/forbidden']);

// Return types: boolean | UrlTree | Promise/Observable of either — async guards
// (e.g. check a session with the server) just return the observable.

Guards centralize "may this navigation complete?" into the route table — the declarative alternative to sprinkling if (!loggedIn) router.navigate(...) through components. The modern form is a plain function (CanActivateFn) using inject() — class-based guards are deprecated-in-spirit legacy you'll still read. Two decisions embedded in the example matter: return a UrlTree to redirect rather than false to silently kill the navigation (a dead click is hostile UX; a redirect with a returnUrl is a flow), and guards being composable arrays means each stays single-purpose. And the standing caveat from the permission-directive lesson applies doubly: guards are UX routing, not security — the API must enforce authorization on every request regardless.

Part 2 — CanDeactivate & CanMatch: the Other Checkpoints

// CANDEACTIVATE — "may the user LEAVE?" (unsaved changes — template-forms.html teased it):
export interface HasUnsavedChanges { hasUnsavedChanges(): boolean; }

export const unsavedChangesGuard: CanDeactivateFn<HasUnsavedChanges> = (component) =>
  !component.hasUnsavedChanges()
    || confirm('You have unsaved changes. Leave anyway?');

{ path: 'editor/:id', component: EditorComponent, canDeactivate: [unsavedChangesGuard] }
// EditorComponent implements hasUnsavedChanges() { return this.form.dirty; }

// CANMATCH — "does this route DEFINITION even apply?" — runs during matching:
{ path: 'dashboard', canMatch: [() => inject(AuthService).isAdmin()],
  loadComponent: () => import('./admin-dashboard.component') },
{ path: 'dashboard',                                    // same path, next candidate!
  loadComponent: () => import('./user-dashboard.component') },
// admin → first route wins; non-admin → matching FALLS THROUGH to the second.
// canActivate can't do this: it blocks, it doesn't unmatch.

// CanMatch also gates the CHUNK: a lazy route that fails canMatch never downloads its code.

Each guard type answers a different question at a different moment. CanDeactivate protects egress — the unsaved-changes dialog every editor needs (pair it with the browser's beforeunload for tab-closing; the router can't intercept that). CanMatch is subtler and more powerful than CanActivate: failing it makes the router pretend the route definition doesn't exist and continue matching — enabling same-URL-different-component patterns (role-based dashboards, A/B tests, feature-flagged rewrites) and keeping unauthorized users from even downloading the admin bundle. Rule of thumb: gate access with canActivate, gate existence with canMatch.

Part 3 — Resolvers: Pre-Loading Route Data (and the Honest Trade-off)

// A resolver fetches data BEFORE the navigation completes:
export const productResolver: ResolveFn<Product> = (route) =>
  inject(ProductApi).getProduct(route.paramMap.get('id')!);

{ path: 'products/:id', component: ProductDetailComponent,
  resolve: { product: productResolver } },

// Component receives it — with input binding, as a plain input:
export class ProductDetailComponent {
  product = input.required<Product>();     // resolved data binds by KEY name
}

THE TRADE-OFF (this is the interview question):
  RESOLVER: navigation WAITS for data → new page appears complete (no spinners,
            no layout shift)… but the click feels DEAD during the fetch — the old
            page just sits there. Errors must cancel navigation (handle them!).
  IN-COMPONENT (resource/switchMap): navigation is INSTANT → skeleton/spinner
            renders immediately, data streams in. Perceived speed usually WINS.

Modern default: navigate instantly + in-component resource loading + skeletons.
Resolvers earn their keep when the page is USELESS without the data (tiny fast
payloads), or when multiple sibling components need the same pre-fetched data.

Resolvers front-load data so components mount complete — appealing until you feel the frozen click on a slow connection. The industry (and Angular's own drift toward resource) has largely settled on navigate-then-load with skeleton states as the better default: users perceive instant response, and loading UI is honest about what's happening. When you do use resolvers, handle their failure path explicitly (an erroring resolver cancels navigation silently — pipe a catchError that redirects or returns a fallback) and keep them fast. Their remaining sweet spots are genuinely blocking data (auth context, feature config) and shared pre-fetches.

Part 4 — Lazy Loading: the Route Table as Bundle Map

// EAGER (default): everything in the main bundle — users download admin code to see login.

// LAZY — a single component:
{ path: 'profile', loadComponent: () =>
    import('./profile/profile.component').then(m => m.ProfileComponent) },

// LAZY — a whole feature (its own routes file):
{ path: 'admin', canActivate: [authGuard],
  loadChildren: () => import('./admin/admin.routes').then(m => m.ADMIN_ROUTES) },

// admin/admin.routes.ts — the feature owns its internal routing:
export const ADMIN_ROUTES: Routes = [
  { path: '', component: AdminLayoutComponent,
    providers: [AdminStateService, provideAdminApi()],   // ← ROUTE-LEVEL PROVIDERS:
    children: [                                          //   feature-scoped services,
      { path: '', component: AdminDashboardComponent },  //   created when feature loads
      { path: 'users', component: UserListComponent },
    ]},
];

// What the build does: each import() = a SPLIT POINT → separate chunk file.
// First navigation to /admin downloads admin-*.js; before that, it costs nothing.
// The dynamic import is cached — subsequent navigations are instant.

Lazy loading turns the route table into the app's code-splitting map: each loadComponent/loadChildren boundary becomes a separate bundle the browser fetches on first navigation. For any app with distinct areas (admin, settings, reports), this is the single highest-leverage startup-performance decision — the main bundle shrinks to the shell plus the landing route. Route-level providers complete the picture from Advanced DI: feature-scoped services instantiate when the feature loads and scope to its router subtree — the feature is a self-contained unit of code, routes, and dependencies. (Template-level @defer from Directives is the same idea one level finer.)

Part 5 — Preloading: Lazy Without the Wait

// The lazy trade-off: first navigation to a feature pays the chunk download.
// Preloading downloads chunks AFTER startup, in the background:

provideRouter(routes, withPreloading(PreloadAllModules))   // everything, once idle-ish

// Selective — flag the likely-next features:
{ path: 'products', loadChildren: …, data: { preload: true } },

export class FlaggedPreloadStrategy implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<unknown>) {
    return route.data?.['preload'] ? load() : of(null);    // load() triggers the fetch
  }
}
provideRouter(routes, withPreloading(FlaggedPreloadStrategy))

// The spectrum:
//   no preloading      smallest startup, first-visit lag per feature
//   PreloadAllModules  startup small, everything warm soon after — GOOD DEFAULT for
//                      modest app sizes on decent networks
//   custom (flags, on-hover, network-aware via navigator.connection) — large apps
// Guards note: canMatch-failing routes aren't preloaded for that user. canActivate
// ones ARE (access ≠ existence) — another reason canMatch guards the admin bundle.

Preloading resolves lazy loading's UX cost: ship the minimal startup bundle, then warm the rest in the background so first navigations feel eager. PreloadAllModules is the sensible default until measurements argue otherwise; custom strategies scale it — flag-based, hover-triggered ("intent preloading"), or network-aware (skip preloading on saveData/2G). This layered posture — lazy boundaries + background preload + @defer inside pages — is the complete Angular loading-performance answer you should be able to narrate end-to-end in an interview, finishing with bundle analysis to verify the split actually happened.

Part 6 — Deep Dive: Feature Architecture & the Complete Route Table

// The shape mature Angular apps converge on — app.routes.ts as a THIN composition root:
export const routes: Routes = [
  { path: '', component: HomeComponent, title: 'Shop' },
  { path: 'products',
    loadChildren: () => import('./features/products/products.routes') },   // default export
  { path: 'checkout', canActivate: [authGuard],
    loadChildren: () => import('./features/checkout/checkout.routes') },
  { path: 'admin', canMatch: [adminMatchGuard],
    loadChildren: () => import('./features/admin/admin.routes') },
  { path: 'login', component: LoginComponent, title: 'Sign in' },
  { path: '**', component: NotFoundComponent, title: 'Not found' },
];

// Each feature: OWNS its routes, components, feature-scoped providers, guards.
// The app table: WIRES features and applies cross-cutting policy (auth, titles).
// Dependency rule (services-di.html) at the routing level:
//   features never import each other's routes/components — navigation BY URL is
//   the only cross-feature contract. (Enforced with lint boundaries in big repos.)

This composition shape is the payoff of everything in this lesson: the app-level table reads as a site map with policy annotations; features are self-contained (deletable, extractable, owned by a team); bundle boundaries fall along team boundaries; and cross-feature coupling is reduced to URLs — the loosest possible contract. In monorepos this graduates into enforced module boundaries (Enterprise lesson), but the discipline starts here, in how routes compose.

Senior framing: the guard/resolver/lazy cluster is where interviews test judgment through mechanisms: Do you redirect or block (UrlTree vs false)? Do you know canMatch vs canActivate (existence vs access — and the bundle-download implication)? Can you argue resolver vs in-component loading as a UX trade-off rather than reciting one as "best practice"? Do you say unprompted that guards are not security? And the architecture question — "how do you keep a 40-developer Angular app's startup fast?" — expects the layered answer: route-level code splitting along feature boundaries, preloading strategy, @defer for heavy widgets, and bundle-size CI budgets so the split doesn't silently erode. Mechanisms are table stakes; the trade-off narration is the senior signal.

Interview Questions

Q1 (Beginner): What is a route guard, and what can a CanActivateFn return?

A function the router runs at a navigation checkpoint to decide whether it completes. A CanActivateFn receives the target route and router state and returns boolean (allow/block), a UrlTree (redirect — the polite form of blocking), or a Promise/Observable of either for async decisions (e.g., validating a session server-side). Guards run in an injection context, so inject(AuthService) works; multiple guards compose as an array where all must pass.

Q2 (Beginner): What does lazy loading change about what the browser downloads, and where do you declare it?

Eagerly, the whole app compiles into the startup bundles — users download the admin area to view the login page. Declaring loadComponent/loadChildren with a dynamic import() in the route config makes the build emit that subtree as a separate chunk, downloaded only on first navigation to it (then cached). The route table thus doubles as the application's bundle map — startup payload shrinks to the shell plus the landing route.

Q3 (Mid): canActivate vs canMatch — semantics, and two things only canMatch can do.

canActivate runs after a route has matched and decides whether the navigation may complete — failing it stops (or redirects) the navigation. canMatch runs during URL matching and decides whether the route definition counts as existing — failing it makes the router continue to the next candidate route. Only canMatch can: (1) enable same-path alternatives (two dashboard routes — admin variant matched first, user variant as fall-through — role-based UIs and feature-flag switches); (2) prevent the lazy chunk from being downloaded at all for unauthorized users, since the route never matches (canActivate blocks after the loader is engaged).

Q4 (Mid): Argue resolver vs in-component loading for a product detail page — and give the case where a resolver is still right.

Resolver: navigation waits on the fetch — the arriving page is complete (no spinner, no shift), but the user stares at the frozen old page; slow API = dead-feeling click, and resolver errors cancel navigation unless handled. In-component (resource/switchMap on the param): navigation is instant, skeleton renders immediately, data streams in — perceived performance wins, loading and error states are explicit UI. Modern default: in-component. Resolver still right when the page is meaningless without fast, small, blocking data — auth/tenant context, feature config — or when several sibling components need one shared pre-fetch. The senior tell is presenting it as a UX trade-off, not dogma.

Q5 (Senior): Route-level providers: what do they give a lazy feature, and what's the classic mistake with them?

A lazy feature's routes can declare providers — creating an injector scoped to that router subtree, instantiated when the feature first loads. This gives features their own service instances (feature stores, configured API clients via provide*() factories) that don't exist until needed and are invisible outside — the DI hierarchy (Advanced DI) extended to routing. Classic mistake: providing a service meant to be an app-wide singleton inside a feature's providers — each feature (or the feature vs the shell) gets its own instance, and "shared" state mysteriously isn't; global singletons stay providedIn: 'root', route providers are for deliberately feature-scoped state.

Q6 (Senior): Your app's main bundle is 2.8MB despite "lazy loading everything". Walk the investigation.

(1) Measure, don't guess: ng build + esbuild/source-map bundle analyzer — see what's actually in main. Usual suspects, in frequency order: (2) static imports of lazy code — one import { AdminThing } from '../admin/…' in a shared file pulls the whole feature into main (the split exists only if the ONLY path to the code is the dynamic import); fix and add a lint boundary rule. (3) Heavyweight libraries in the shell — charting, editors, moment-with-locales — move them behind @defer or the lazy features using them, check for tree-shakable alternatives. (4) An eager "shared" barrel exporting everything (barrel files defeat tree-shaking when they re-export side-effectful modules). (5) Styles/i18n: global CSS bloat and eager locale data. Then protect the win: CI bundle budgets (angular.json budgets fail builds on regression), and preloading so the newly-split chunks don't degrade UX. The narrative arc — measure, find the eager path, fix, enforce — matters more than any single cause.

3 (Medium): Implement the unsaved-changes guard against an interface, apply it to an editor whose dirtiness comes from a reactive form, and handle the tab-close case the router can't.
Solution
export interface HasUnsavedChanges { hasUnsavedChanges(): boolean; }

export const unsavedChangesGuard: CanDeactivateFn<HasUnsavedChanges> = (c) =>
  !c.hasUnsavedChanges() || confirm('Discard unsaved changes?');

{ path: 'editor/:id', component: EditorComponent, canDeactivate: [unsavedChangesGuard] }

export class EditorComponent implements HasUnsavedChanges {
  form = inject(FormBuilder).nonNullable.group({ /* … */ });
  hasUnsavedChanges() { return this.form.dirty; }        // pristine after save/reset

  // Router can't see tab close/reload — the browser API covers that half:
  @HostListener('window:beforeunload', ['$event'])
  onBeforeUnload(e: BeforeUnloadEvent) {
    if (this.hasUnsavedChanges()) e.preventDefault();    // triggers the native prompt
  }
}
// Interface-typed guard = reusable across every editor; form.dirty (not touched!) is
// the right dirtiness signal (template-forms.html exercise 5's logic, formalized).
4 (Medium): Serve /dashboard as AdminDashboard for admins and UserDashboard for everyone else — without redirects — and explain why canActivate can't express this.
Solution
const isAdmin: CanMatchFn = () => inject(AuthService).roles().includes('admin');

export const routes: Routes = [
  { path: 'dashboard', canMatch: [isAdmin],
    loadComponent: () => import('./admin-dashboard.component') },
  { path: 'dashboard',
    loadComponent: () => import('./user-dashboard.component') },   // fall-through
];
// Matching walks candidates in order: admins match route 1; for others, canMatch fails
// and matching CONTINUES to route 2 — same URL, different component, no redirect dance.
// canActivate can't do it: by the time it runs, matching is OVER — it can only block or
// redirect, not "try the next definition". Bonus: non-admins never download admin-*.js.
5 (Hard): Write a resolver for tenant config (genuinely blocking data), including the error path that redirects to an error page instead of silently killing navigation, then wire preloading with a flag-based strategy.
Solution
export const tenantResolver: ResolveFn<TenantConfig> = () => {
  const router = inject(Router);
  return inject(TenantApi).config().pipe(
    timeout(3000),                                    // blocking data must be FAST
    catchError(() => {
      router.navigateByUrl('/service-unavailable');   // explicit failure UX —
      return EMPTY;                                   // EMPTY cancels this navigation
    }),
  );
};

{ path: 'app', resolve: { tenant: tenantResolver }, children: [ /* whole tenant area */ ] }

// Flag-based preloading:
export class FlaggedPreload implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<unknown>) {
    return route.data?.['preload'] ? load() : of(null);
  }
}
{ path: 'products', loadChildren: …, data: { preload: true } },   // likely next
{ path: 'admin',    canMatch: [isAdmin], loadChildren: … },       // NOT flagged — and
                                                                  // canMatch-gated anyway
provideRouter(routes, withPreloading(FlaggedPreload))
// Resolver justified here: EVERY page under /app is meaningless without tenant config,
// it's one small fast call, and children read it from parent route data.
6 (Hard): Design the complete route architecture for a shop (public catalog, authed checkout, admin area, login, 404): lazy boundaries, guard placement (right type per job), preloading, and the rule that keeps features decoupled.
Solution
export const routes: Routes = [
  // Public, eager — it's the landing experience:
  { path: '', component: HomeComponent, title: 'Shop' },

  // Catalog: lazy + preloaded (everyone goes here next):
  { path: 'products', data: { preload: true },
    loadChildren: () => import('./features/catalog/catalog.routes') },

  // Checkout: lazy, auth-GATED (canActivate — users may BECOME authorized; redirect):
  { path: 'checkout', canActivate: [authGuard],
    loadChildren: () => import('./features/checkout/checkout.routes') },
  //   inside checkout.routes: canDeactivate on the payment step (abandon-warning),
  //   route providers: [CheckoutStateService] (wizard state, feature-scoped)

  // Admin: canMatch (existence + chunk gating — non-admins never see or download it):
  { path: 'admin', canMatch: [adminMatch],
    loadChildren: () => import('./features/admin/admin.routes') },

  { path: 'login', component: LoginComponent, title: 'Sign in' },
  { path: '**', component: NotFoundComponent, title: 'Not found' },
];
provideRouter(routes, withComponentInputBinding(),
  withPreloading(FlaggedPreload), withInMemoryScrolling({ scrollPositionRestoration: 'enabled' }));

// Decoupling rule: features never statically import each other — cross-feature
// navigation is by URL only (routerLink="/checkout"), enforced by lint boundaries.
// Guard-type reasoning: authGuard = access you can GAIN (redirect to login, come back);
// adminMatch = existence you either have or don't (fall through to 404, chunk untouched).
// Server-side: every checkout/admin API re-authorizes — guards are choreography, not locks.

🎯 Key Takeaways

  • Guards are functions at navigation checkpoints: redirect with UrlTree (+returnUrl) instead of dead-click false; compose single-purpose guards in arrays; they're UX — the server authorizes for real.
  • canActivate gates access; canMatch gates existence — enabling same-path alternatives and keeping unauthorized users from downloading the chunk. canDeactivate + beforeunload cover unsaved changes.
  • Resolvers trade instant navigation for complete-on-arrival pages — modern default is navigate-then-load with skeletons; resolvers for fast, genuinely blocking data, always with an error path.
  • loadComponent/loadChildren make the route table the bundle map; route-level providers scope services to lazy features — code + routes + dependencies as one deletable unit.
  • Preloading (all, flagged, or network-aware) buys back lazy's first-visit lag; one stray static import silently un-splits a bundle — verify with the analyzer, enforce with budgets and lint boundaries.
  • App routes = thin composition root wiring self-contained features; cross-feature contract = URLs only.
