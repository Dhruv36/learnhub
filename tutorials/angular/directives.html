Directives & Control Flow Foundations

What you'll master: the built-in control flow (@if, @for, @switch) that replaced *ngIf/*ngFor, why track is mandatory and what it protects you from, the legacy structural-directive syntax you'll still meet in every existing codebase, attribute directives, and writing your own directives — the tool for reusable behavior the way components are for reusable UI. Prerequisites: Templates & Data Binding.

Part 1 — Built-in Control Flow: @if, @for, @switch

@if (user(); as u) {                       <!-- condition + capture in one -->
  <p>Welcome back, {{ u.name }}</p>
} @else if (loading()) {
  <app-spinner />
} @else {
  <a routerLink="/login">Sign in</a>
}

@for (item of items(); track item.id) {    <!-- track is REQUIRED — see Part 2 -->
  <app-card [item]="item" />
} @empty {
  <p>No results.</p>                       <!-- built-in empty state! -->
}

@switch (status()) {
  @case ('active')   { <span class="ok">Active</span> }
  @case ('past_due') { <span class="warn">Past due</span> }
  @default           { <span>Unknown</span> }
}

Since Angular 17, control flow is template syntax, not directives — block keywords the compiler understands directly. Compared to the old *ngIf/*ngFor: no imports needed (it's part of the language), real @else if chains instead of nested workarounds, a built-in @empty block (previously an awkward *ngIf on length), better type narrowing inside blocks, and faster generated code. The as u capture solves the async-value pattern elegantly: test once, use the non-null captured value throughout the block. Inside @for you also get implicit variables: $index, $first, $last, $even, $odd, $count.

Part 2 — track: Identity, Reuse & the Bug It Prevents

@for (user of users(); track user.id) { … }     // ✓ stable business key
@for (tag of tags(); track tag) { … }           // ✓ primitives track themselves
@for (row of rows(); track $index) { … }        // ⚠️ ONLY for never-reordered static lists

Why it matters — re-fetch the same 100 users (new array, new objects):
  track user.id  → Angular MATCHES old & new items by id: DOM reused, only changed
                   bindings patched. Fast, and component STATE inside rows survives.
  track $index   → item #3 is "whatever is at position 3" — insert one item at the top
                   and EVERY row's content shifts: full re-bind of the list, and any
                   state in row components (open menus, input text, focus) attaches to
                   the WRONG row.

track tells Angular what makes a row the same row across updates — the exact job React's key does. With a stable business key, list updates become minimal DOM patches and row-level state stays glued to the right data. With $index (or the old default — object identity — combined with fresh objects from an API), you get the classic bug family: rows visually shuffle their internal state, animations replay on unchanged rows, inputs lose focus mid-typing, and performance quietly degrades to rebuild-the-list. The new syntax making track mandatory (unlike optional trackBy, which almost nobody wrote) is the framework turning a senior code-review comment into a compile-time requirement.

Part 3 — The Legacy Syntax You'll Still Meet: *ngIf, *ngFor & the Asterisk

<!-- Same UI, pre-17 style (needs CommonModule / NgIf, NgFor imports): -->
<p *ngIf="user; else anon">Welcome {{ user.name }}</p>
<ng-template #anon><a routerLink="/login">Sign in</a></ng-template>

<li *ngFor="let item of items; trackBy: trackById">{{ item.name }}</li>
// class: trackById(index: number, item: Item) { return item.id; }

<!-- What the * actually MEANS — sugar for an ng-template: -->
<p *ngIf="cond">Hi</p>
<!-- desugars to: -->
<ng-template [ngIf]="cond"><p>Hi</p></ng-template>
<!-- A STRUCTURAL directive receives a template it can stamp out 0..N times.
     That's the whole trick: *ngIf stamps 0 or 1; *ngFor stamps N. -->

<!-- ng-container: grouping without a wrapper element (works with both syntaxes): -->
<ng-container *ngIf="ready">
  <dt>{{ term }}</dt><dd>{{ def }}</dd>      <!-- no illegal div inside <dl> -->
</ng-container>

Millions of lines of Angular use the asterisk syntax, so fluency is non-optional. The key insight — worth internalizing even in the @if era — is that the * is sugar: a structural directive is a directive handed an ng-template (a chunk of template that renders nothing by itself), which the directive instantiates zero or more times. That model explains everything odd about the old syntax: why only one structural directive per element (one template, one owner — the reason for ng-container nesting), why else points at a template reference, and how third-party structural directives (*appHasPermission) work. Migration is mechanical (ng generate @angular/core:control-flow converts a whole app), which tells you the two syntaxes are semantically equivalent — the new one is ergonomics + speed, not new power.

Part 4 — Attribute Directives: Behavior Without Templates

// A directive = a component minus the template: it attaches BEHAVIOR to a host element.
import { Directive, ElementRef, HostListener, input, inject } from '@angular/core';

@Directive({ selector: '[appAutofocus]' })          // attribute selector — [brackets]
export class AutofocusDirective {
  private el = inject(ElementRef<HTMLElement>);
  ngOnInit() { this.el.nativeElement.focus(); }
}
// usage: <input appAutofocus>

@Directive({ selector: '[appConfirmClick]' })
export class ConfirmClickDirective {
  message = input('Are you sure?', { alias: 'appConfirmClick' });
  confirmed = output<void>();

  @HostListener('click', ['$event'])
  onClick(e: MouseEvent) {
    if (confirm(this.message())) this.confirmed.emit();
  }
}
// usage: <button [appConfirmClick]="'Delete order?'" (confirmed)="del()">Delete</button>

// Built-ins you already know are attribute directives: ngClass, ngStyle, ngModel, routerLink.

When several components need the same behavior — autofocus, click-outside detection, permission-based hiding, tooltips, drag handles — a directive packages it as an attribute you sprinkle onto any element. The toolkit: inject(ElementRef) for the host element, @HostListener to react to host events, host bindings for host properties/classes, plus normal input()/output(). Directives compose where component wrappers can't: three behaviors on one native <button> beats three nested wrapper components. Rule of thumb: reusable UI → component; reusable behavior on existing elements → directive.

Part 5 — Writing a Structural Directive (and When You Actually Should)

// The permission-gate — the classic justified custom structural directive:
import { Directive, TemplateRef, ViewContainerRef, inject, input, effect } from '@angular/core';

@Directive({ selector: '[appIfRole]' })
export class IfRoleDirective {
  private tpl = inject(TemplateRef<unknown>);       // the template the * wrapped
  private vcr = inject(ViewContainerRef);           // where to stamp it
  private auth = inject(AuthService);

  appIfRole = input.required<string>();             // *appIfRole="'admin'"

  constructor() {
    effect(() => {                                  // re-evaluate when role or input changes
      this.vcr.clear();
      if (this.auth.roles().includes(this.appIfRole())) {
        this.vcr.createEmbeddedView(this.tpl);      // render the guarded content
      }
    });
  }
}
// usage: <button *appIfRole="'admin'">Delete user</button>
// The button doesn't just hide — it's NEVER IN THE DOM for non-admins.

A structural directive injects the TemplateRef (the content the asterisk wrapped) and a ViewContainerRef (the spot in the DOM), then decides when to createEmbeddedView. The permission gate shows why this beats @if scattered everywhere: the policy (how roles are checked) lives in one tested place, and templates read declaratively (*appIfRole="'admin'"). Honest guidance: with @if/@for covering the generic cases, custom structural directives are rare — permission gates, feature flags, A/B testing wrappers, "loading/error/data" state templates. If you're not stamping templates conditionally, you wanted an attribute directive or a component.

Part 6 — Deep Dive: @defer — Lazy Loading at the Template Level

@defer (on viewport) {                       <!-- code loads when scrolled into view -->
  <app-comments [postId]="id()" />           <!-- its JS is SPLIT OUT of the main bundle -->
} @placeholder {
  <div class="skeleton">…</div>              <!-- shown until loading starts -->
} @loading (minimum 300ms) {
  <app-spinner />                            <!-- shown while fetching the chunk -->
} @error {
  <p>Comments failed to load.</p>
}

Triggers: on idle (default) | on viewport | on interaction | on hover |
          on timer(3s) | when condition() | prefetch on idle   (combinable)

@defer brings route-level lazy loading (covered later) down to template granularity: everything inside the block — components, their dependencies — is compiled into a separate chunk that isn't downloaded until the trigger fires. Heavy below-the-fold widgets (comment threads, charts, maps, rich editors) stop taxing initial load. The sub-blocks handle the full lifecycle honestly (placeholder → loading → content/error), and prefetch triggers let you download early but render late. This is a genuinely compiler-powered feature: the dependency graph analysis that decides what goes in the chunk happens at build time — a good interview example of what AOT compilation buys beyond error checking.

Senior framing: control flow choices are identity choices. @if destroys and recreates its content — state inside (form inputs, child component fields, scroll positions) dies with it; CSS [hidden]/display:none merely hides, keeping state and staying in the accessibility tree unless managed. Toggling something expensive frequently? Hiding may beat recreating. Sensitive content (admin buttons, PII)? It must be absent, not hidden — and truly secured server-side, since anything client-side is inspectable. The same identity lens applies to @for track: interviewers who ask "why did the input lose focus when the list refreshed?" are testing whether you know that DOM reuse — not rendering — is the thing being controlled.

Interview Questions

Q1 (Beginner): What are the three kinds of directives in Angular?

Components (a directive with a template — the special case you use most), attribute directives (attach behavior/appearance to an existing element — ngClass, ngModel, custom ones like an autofocus), and structural directives (receive a template and stamp it into the DOM zero or more times — the legacy *ngIf/*ngFor; since v17 the built-in cases are covered by @if/@for/@switch block syntax instead).

Q2 (Beginner): What does @empty do in a @for block, and what did this require before?

@empty renders its content when the iterated collection has no items — the empty-state UI in the same construct as the loop. Pre-17 you needed a separate *ngIf="items.length === 0" (with its own imports and a wrapper or ng-container), keeping the loop and its empty state apart and easy to let drift.

Q3 (Mid): After a data refresh, users typing in a row's inline-edit input find their text jumps to a different row. Diagnose.

The list is tracked by index (track $index — or, legacy, no trackBy with fresh objects each fetch, so identity tracking also fails). When the refresh reorders or inserts items, Angular reuses DOM/component instances by position, so row-local state — the input's text, focus — stays at position N while the data at position N changes. Fix: track item.id (a stable business key), making Angular match rows across updates by identity, moving DOM with the item instead of re-binding positions.

Q4 (Mid): What does the asterisk in *ngIf desugar to, and what does that reveal about how structural directives work?

<p *ngIf="cond">Hi</p> desugars to <ng-template [ngIf]="cond"><p>Hi</p></ng-template>. It reveals the mechanism: a structural directive is an ordinary directive that injects the TemplateRef it wraps plus a ViewContainerRef, and calls createEmbeddedView to stamp the template 0..N times. It also explains the one-structural-directive-per-element rule (one template, one owner) and how custom ones (*appIfRole) plug into the same syntax.

Q5 (Senior): @if vs [hidden] for toggling a section — walk through the trade-offs.

@if adds/removes from the DOM: state inside (child component fields, form inputs, scroll) is destroyed on hide and rebuilt on show; lifecycle hooks re-run; memory is freed; content is genuinely absent (right for permission-gated UI — though real security is server-side). [hidden]/CSS keeps the subtree alive: instant toggling, state survives, but it keeps consuming memory, its bindings still get change-checked, and it's present in the DOM (inspectable; mind accessibility — hidden via display:none leaves the a11y tree, mere visibility/opacity tricks may not). Heuristics: frequently-toggled + expensive or state-carrying → hide; rarely shown, heavy, or sensitive → @if; huge below-fold widgets → @defer.

Q6 (Senior): Your app gates admin UI with @if (user.isAdmin) in forty templates. Critique and improve.

Two problems: policy duplication (the check's shape is copy-pasted forty times — changing to permission-based roles means forty edits) and semantic noise (templates express a mechanism, not the intent). Improve with a structural permission directive — *appIfRole="'admin'" or *appCan="'user.delete'" — injecting an AuthService: policy is defined and tested once, templates read as intent, and reactive role changes (via signals/effect re-stamping the view) update everywhere automatically. Complete the answer with the security caveat: client-side gating is UX, not security — every gated action must be authorized server-side too, since the JS bundle and API are fully inspectable.

🏋️ Exercises

1 (Easy): Render a products() list as cards with @for (tracked by id), an @empty message, and a header shown only when the list is non-empty.
Solution
@if (products().length > 0) {
  <h2>{{ products().length }} products</h2>
}
@for (p of products(); track p.id) {
  <app-product-card [product]="p" />
} @empty {
  <p>No products match your filters.</p>
}
2 (Easy): Show an order's status with @switch: "shipped" → green text, "pending" → amber, "cancelled" → struck-through, anything else → "Unknown status".
Solution
@switch (order().status) {
  @case ('shipped')   { <span class="ok">Shipped</span> }
  @case ('pending')   { <span class="warn">Pending</span> }
  @case ('cancelled') { <s>Cancelled</s> }
  @default            { <span>Unknown status</span> }
}
<!-- Exhaustive, flat, and — unlike a chain of @if — obviously about ONE discriminator. -->
3 (Medium): Convert this legacy template to modern control flow and explain each change: 
<div *ngIf="user; else guest">Hi {{ user.name }}</div>
<ng-template #guest><p>Please log in</p></ng-template>
<li *ngFor="let t of tasks; trackBy: byId; let i = index">{{ i }}: {{ t.title }}</li>
Solution
@if (user; as u) {
  <div>Hi {{ u.name }}</div>
} @else {
  <p>Please log in</p>
}
@for (t of tasks; track t.id) {
  <li>{{ $index }}: {{ t.title }}</li>
}
<!-- Changes: @else replaces the detached #guest ng-template (locality!);
     `as u` captures the narrowed value; trackBy method → inline `track t.id`;
     `let i = index` → implicit $index; and no CommonModule import is needed at all. -->
4 (Medium): Write an attribute directive appClickOutside that emits an output when a click lands outside its host element (the classic dropdown-closer).
Solution
import { Directive, ElementRef, output, inject, HostListener } from '@angular/core';

@Directive({ selector: '[appClickOutside]' })
export class ClickOutsideDirective {
  private el = inject(ElementRef<HTMLElement>);
  clickOutside = output<void>();

  @HostListener('document:click', ['$event'])       // listen on document, not the host
  onDocClick(e: MouseEvent) {
    if (!this.el.nativeElement.contains(e.target as Node)) {
      this.clickOutside.emit();
    }
  }
}
// usage:
// <div class="dropdown" (appClickOutside)="open.set(false)" appClickOutside>…</div>
// Behavior packaged once; any element can opt in — the directive sweet spot.
5 (Hard): Implement the *appIfRole structural directive from Part 5, then extend it with an else-template input (*appIfRole="'admin'; else: denied" style using a second input).
Solution
@Directive({ selector: '[appIfRole]' })
export class IfRoleDirective {
  private tpl = inject(TemplateRef<unknown>);
  private vcr = inject(ViewContainerRef);
  private auth = inject(AuthService);

  appIfRole = input.required<string>();
  appIfRoleElse = input<TemplateRef<unknown> | null>(null);  // maps from "…; else: tpl"

  constructor() {
    effect(() => {
      const allowed = this.auth.roles().includes(this.appIfRole());
      this.vcr.clear();                                  // idempotent re-stamp
      if (allowed) this.vcr.createEmbeddedView(this.tpl);
      else if (this.appIfRoleElse()) this.vcr.createEmbeddedView(this.appIfRoleElse()!);
    });
  }
}
// usage:
// <button *appIfRole="'admin'; else: denied">Delete</button>
// <ng-template #denied><span>Admins only</span></ng-template>
// The microsyntax maps "else: x" onto the appIfRoleElse input automatically
// (inputName = directive selector + capitalized key).
6 (Hard): A product page has a heavy <app-reviews> component (charts + comments) below the fold. Wire it with @defer so it loads on viewport, prefetches on idle, shows a skeleton placeholder and a spinner while loading (min 200ms), and has an error state.
Solution
@defer (on viewport; prefetch on idle) {
  <app-reviews [productId]="id()" />
} @placeholder {
  <div class="skeleton" style="height: 320px"></div>   <!-- reserves space: no CLS -->
} @loading (minimum 200ms) {
  <app-spinner />
} @error {
  <p>Couldn't load reviews. <button (click)="reload()">Retry</button></p>
}
<!-- app-reviews' code is compiled into a separate chunk: downloaded early (idle prefetch)
     but instantiated only when scrolled near. minimum 200ms stops a spinner flash when
     the prefetch already finished. The placeholder's fixed height prevents layout shift. -->

🎯 Key Takeaways

  • @if/@for/@switch are built-in template syntax (no imports): as captures, @else if chains, @empty states, and implicit $index/$first/… variables.
  • track defines row identity across updates — use a stable business key; index-tracking glues row state to positions and causes the focus-jumping/state-shuffling bug family.
  • The legacy * is sugar for wrapping content in an ng-template handed to the directive — hence one structural directive per element, and ng-container for element-free grouping.
  • Attribute directives package reusable behavior (ElementRef + HostListener + inputs/outputs); custom structural directives are for stamping templates by policy — permission gates, feature flags.
  • @defer lazy-loads at template granularity with real lifecycle blocks (placeholder/loading/error) and triggers (viewport, interaction, prefetch) — compiler-powered code splitting.
  • Choose by identity: @if destroys state, CSS hiding preserves it (and its costs); sensitive UI must be structurally absent client-side and enforced server-side.
