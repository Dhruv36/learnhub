Signals Deep Dive Components & Reactivity

What you'll master: Angular's reactive core — signal, computed, effect — the dependency-graph mental model that explains their behavior, immutable-update discipline, linkedSignal and resource for the awkward cases, RxJS interop (toSignal/toObservable), and the design rules (derive, don't sync; effects for the outside world only). Prerequisites: signal inputs and Lifecycle. Signals are the biggest shift in Angular since its rewrite — every modern lesson downstream assumes them.

Part 1 — The Primitive: Reactive Values, Read as Functions

import { signal } from '@angular/core';

const count = signal(0);          // a WRITABLE SIGNAL wrapping a value

count()                           // READ by calling — returns 0, and (inside reactive
                                  // contexts) REGISTERS "you depend on me"
count.set(5)                      // replace the value
count.update(c => c + 1)          // update from previous

// Object values: REPLACE, don't mutate —
const user = signal({ name: 'Ada', role: 'admin' });
user.update(u => ({ ...u, role: 'owner' }));   // ✓ new reference → dependents notified
user().role = 'owner';                          // ✗ same reference → NOBODY notified

// Equality: notifications fire only when the value CHANGES (Object.is by default)
count.set(5); count.set(5);       // second set is a no-op for dependents
const list = signal<number[]>([], { equal: (a, b) => a.length === b.length });  // custom

A signal is a value with a subscription list. The genius is in the read syntax: because reading is a function call, the framework can record who is reading — a template binding, a computed, an effect — and notify exactly those readers when the value changes. That's what makes Angular's rendering surgical: a template that reads count() depends on count and nothing else, so only that view is marked for update. The discipline this buys you is the same one from pure pipes and Functional Patterns: change means new value (or new reference) — in-place mutation is invisible to the graph.

Part 2 — computed: Derived State That Maintains Itself

const items = signal<CartItem[]>([]);
const coupon = signal<number>(0);          // 0..1

const subtotal = computed(() => items().reduce((s, i) => s + i.price * i.qty, 0));
const total    = computed(() => subtotal() * (1 - coupon()));
const summary  = computed(() => `${items().length} items — ${total().toFixed(2)} €`);

// Properties:
// LAZY      — nothing computes until someone reads total()
// MEMOIZED  — repeated reads return the cached value; recompute only after a dependency changed
// DYNAMIC   — dependencies are whatever the LAST run actually read:
const label = computed(() => loggedIn() ? `Hi ${name()}` : 'Guest');
// while loggedIn() is false, changing name() does NOT recompute label —
// name() wasn't read on the last run, so it isn't currently a dependency.

// computed is read-only: no .set/.update — its value IS its formula.

computed is the workhorse: state that is defined by a formula stays correct by construction — there is no code path that can forget to update total, because total isn't stored, it's derived. The three properties matter practically: laziness means unused derivations cost nothing; memoization means templates can read total() in ten places for one computation; dynamic dependency tracking means branches prune the graph automatically. This replaces whole categories of pre-signals code: manually recomputing in ngOnChanges, combineLatest chains for view models, and the stale-flag bookkeeping every imperative cache needs.

Part 3 — effect: the Bridge to the Non-Reactive World

constructor() {
  effect(() => {
    localStorage.setItem('theme', this.theme());     // reads theme → tracks it
  });    // runs once immediately, then after every theme() change
}

// Rules of effects:
// 1. They're for the OUTSIDE WORLD: storage, logging, analytics, title bars,
//    imperative DOM/widgets, focus. NOT for updating other signals.
// 2. Created in an injection context (constructor/field) — auto-destroyed with it.
// 3. Writing signals inside is blocked by default — deliberately.

// ❌ The anti-pattern (state synced by effect — two sources of truth, ordering bugs):
count = signal(0);
double = signal(0);
constructor() { effect(() => this.double.set(this.count() * 2)); }  // throws by default!
// ✓ The fix is ALWAYS derivation:
double = computed(() => this.count() * 2);

// Cleanup between runs (and on destroy) — for timers/listeners the effect creates:
effect((onCleanup) => {
  const id = setInterval(() => poll(this.url()), this.interval());
  onCleanup(() => clearInterval(id));    // old interval cleared before re-run with new url
});

Effects are for consequences the signal graph can't express — synchronizing reactive state out to systems that aren't signals. The default prohibition on writing signals inside an effect is Angular codifying a design lesson other ecosystems learned painfully: effect-writes-signal chains recreate the untraceable update cascades signals were meant to end (dev answer: "if you think you need it, you want computed or linkedSignal"). The onCleanup callback completes the story — an effect that acquires (interval, listener, socket) releases before each re-run and at destroy, the same acquire/release pairing as DestroyRef.

Part 4 — linkedSignal & resource: the Awkward Cases, Solved

// linkedSignal — writable state that RESETS when a source changes.
// Classic case: selected item must reset when the list reloads:
options = signal<string[]>(['S', 'M', 'L']);
selected = linkedSignal(() => this.options()[0]);   // derives default from options
// user picks:      selected.set('L')        → writable like any signal
// options reload:  options.set(['32','34']) → selected RESETS to '32' automatically
// (previous/others accessible via the {source, computation} long form)

// resource — async data as a first-class reactive citizen (v19+):
userId = signal(1);
user = resource({
  params: () => ({ id: this.userId() }),                    // reactive request params
  loader: ({ params, abortSignal }) =>                       // fetch with cancellation
    fetch(`/api/users/${params.id}`, { signal: abortSignal }).then(r => r.json()),
});
// In templates:  @if (user.isLoading()) … @else if (user.error()) … @else { user.value() }
// userId changes → previous in-flight fetch ABORTED → reload. The switchMap pattern, built in.
// (httpResource(() => url) is the HttpClient flavor — http.html.)

Two purpose-built primitives close the gaps where plain signal/computed felt wrong. linkedSignal legitimizes the one pattern that tempted people toward effect-writes-signal: writable-but-derived-default state (selection following a list, form working copy following a record). resource tackles async: it packages the request-params → loader → {value, status, error} cycle with automatic cancellation of stale requests — the bug class where a slow response for user 1 overwrites the answer for user 2 (you'll meet the manual RxJS version, switchMap, in RxJS Patterns — and recognize resource as its declarative packaging).

Part 5 — RxJS Interop: toSignal / toObservable

import { toSignal, toObservable } from '@angular/core/rxjs-interop';

// Observable → signal (subscribe + auto-unsubscribe handled for you):
prices$ = inject(PriceService).stream();                  // Observable<number>
price = toSignal(this.prices$, { initialValue: 0 });      // Signal<number>
// template just reads {{ price() }} — no async pipe, no subscription management

// Signal → observable (when you need operators):
query = signal('');
results = toSignal(
  toObservable(this.query).pipe(
    debounceTime(300),                                    // TIME-based logic: RxJS's home turf
    distinctUntilChanged(),
    switchMap(q => this.api.search(q)),                   // cancel stale searches
  ),
  { initialValue: [] as Result[] },
);

// The division of labor that's emerging as best practice:
//   SIGNALS: state — current values, derivations, template consumption
//   RxJS:    events over time — debounce, retry, streams, cancellation pipelines
// Convert at the boundary; don't force either to do the other's job.

Angular didn't replace RxJS with signals — it repositioned it. Signals answer "what is the current value and what derives from it"; observables answer "how do events compose over time". The interop functions are the seam: toSignal turns any stream into template-friendly state (managing the subscription against the component's lifetime — one more manual-unsubscribe class eliminated), and toObservable lets a signal feed a pipeline when you genuinely need time operators. The search box above is the canonical hybrid: signal state at both ends, RxJS in the middle for debounce + cancellation.

Part 6 — Deep Dive: How the Graph Updates, and Signal Architecture

The propagation model (why glitches don't happen):

  a = signal(1)
  b = computed(() => a() * 10)          a ──▶ b ──▶ template
  c = computed(() => a() + b())         a ──▶ c ◀── b

a.set(2):
  1. a marks its dependents DIRTY (b, c) — nothing recomputes yet   (push: invalidation)
  2. when the template/effect next READS c(), c pulls fresh b(),
     which pulls fresh a() — each recomputes ONCE, in dependency order (pull: evaluation)
  → c can never see "new a, old b" (a GLITCH). Push-pull + memoization = consistent,
    minimal recomputation. No subscription bookkeeping, no unsubscribe, no zones needed.

Architecture rules that follow:
  1. Store the MINIMUM; derive the rest (computed).            "derive, don't sync"
  2. Effects face OUTWARD only (storage/DOM/analytics).
  3. Immutable updates always — the graph sees references.
  4. Services expose: private writable signals + public readonly/computed + methods.
     (the pattern from component-communication.html, formalized in state-management.html)

Understanding the two-phase update — push invalidation, pull evaluation — lets you answer every "when does this recompute?" question from first principles: writes are cheap (they only mark dirty), reads settle the graph in dependency order, memoization prevents redundant work, and consumers can never observe a half-updated state. This is also the foundation for the next lesson: change detection can become signal-driven — the framework knows exactly which views read which signals, so it can skip everything else, including Zone.js itself.

Senior framing: the migration question ("we have a big RxJS/zone codebase — how do we adopt signals?") has a standard good answer: adopt at the edges first — new components signal-first; templates consume signals (via toSignal over existing streams) instead of async-pipe chains; state services get signal read-models while keeping their internals; RxJS remains for genuinely event-driven pipelines (websockets, typeahead, retry logic). The design smell catalog for reviews: stored state that could be computed ("derive, don't sync"); effects writing signals (hidden data flow — use computed/linkedSignal); signal reads outside reactive contexts cached into locals then reused (stale snapshots); and mutation of signal-held objects (the graph never hears about it). Signals reward exactly the discipline this track has been building: minimal state, immutable updates, explicit derivation.

Interview Questions

Q1 (Beginner): What is a signal, and why is reading one a function call?

A signal is a wrapper around a value that tracks its readers and notifies them on change — created with signal(initial), written with set/update. Reading via call syntax (count()) is what enables automatic dependency tracking: when a template, computed, or effect calls the signal, the framework records the dependency, so later writes notify exactly the affected readers — no manual subscriptions, and rendering can update precisely the views that depend on the changed value.

Q2 (Beginner): signal vs computed — and why should cartTotal be a computed rather than a signal you update?

signal is writable source state; computed is read-only state defined by a formula over other signals, recalculated (lazily, memoized) when dependencies change. A stored cartTotal must be updated by every code path that touches the cart — forget one and the UI lies. As computed(() => items().reduce(...)) it cannot be wrong: no update code exists to forget. Rule: store the minimum, derive the rest.

Q3 (Mid): Why does Angular (by default) throw when an effect writes a signal, and what are the sanctioned alternatives?

Because effect→signal writes recreate hidden, imperative data flow: state becomes the product of effect execution order rather than declared derivation — cascading updates, glitches, and untraceable "who set this?" debugging. Alternatives by intent: value derived from other state → computed; writable state whose default follows a source (selection resetting on list reload) → linkedSignal; async data per reactive params → resource (or RxJS interop with switchMap). Effects should face outward — storage, DOM, analytics — not inward at the graph.

Q4 (Mid): A colleague's list UI stopped updating after they switched an update to this.items().push(newItem). Explain, and connect it to the wider Angular discipline.

items() returns the array held by the signal; push mutates it in place, so the signal's value (the reference) is unchanged — equality says "no change", nobody is notified, the template keeps its last render. Fix: items.update(l => [...l, newItem]). It's the same reference-equality contract as pure pipes and OnPush inputs: throughout modern Angular, new value = new reference; immutable updates are what make change observable.

Q5 (Senior): Explain computed's dynamic dependency tracking with a branch example, and a subtle behavior it produces.

A computed's dependencies are whatever signals its most recent execution actually read. For computed(() => loggedIn() ? name() : 'Guest'): while logged out, only loggedIn is a dependency — name.set(...) triggers nothing. When loggedIn flips true, the recompute reads name(), adding it to the graph; changing name now recomputes the label. The subtlety: dependency sets change at runtime, so "why didn't X recompute?" is answered by "did the last run read that signal?" — branches prune the graph automatically, which is a performance feature and occasionally a debugging surprise (e.g. a signal read hidden behind an early return is only sometimes tracked).

Q6 (Senior): Signals vs RxJS — is RxJS obsolete in modern Angular? Give the division of labor and a concrete hybrid example.

Not obsolete — repositioned. Signals excel at state: current values, synchronous derivation, glitch-free consistency, template consumption without subscription management. RxJS excels at events over time: debouncing, throttling, retries with backoff, cancellation (switchMap), merging streams, websockets. Boundary rule: model state as signals; run time-based event pipelines in RxJS; convert at the seam (toObservable in, toSignal out). Canonical hybrid — typeahead: query signal → toObservable → debounceTime(300), distinctUntilChanged(), switchMap(search) → toSignal(results). Newer resource APIs absorb common async-fetch cases, but composition-over-time remains RxJS's home turf.

🏋️ Exercises

1 (Easy): Model a like button: likes signal, liked boolean signal, clicking toggles liked and adjusts the count. Show template + class.
Solution
likes = signal(41);
liked = signal(false);
toggle() {
  this.liked.update(v => !v);
  this.likes.update(n => n + (this.liked() ? 1 : -1));
}
// template:
// <button (click)="toggle()" [class.on]="liked()">❤ {{ likes() }}</button>
2 (Easy): Given items = signal<Item[]>, derive activeItems, activeCount, and a headline like "3 of 7 active" — no stored duplicates.
Solution
items = signal<Item[]>([]);
activeItems = computed(() => this.items().filter(i => i.active));
activeCount = computed(() => this.activeItems().length);
headline = computed(() => `${this.activeCount()} of ${this.items().length} active`);
// Chained computeds are fine — each memoizes; headline recomputes only when items changes,
// and reads cached activeCount rather than re-filtering.
3 (Medium): Persist a theme signal ('light'|'dark') to localStorage, initialize from storage, and reflect it onto document.body's class — using the right primitive for each part.
Solution
theme = signal<'light' | 'dark'>(
  (localStorage.getItem('theme') as 'light' | 'dark') ?? 'light'   // init: plain read
);
constructor() {
  effect(() => {                                   // OUTWARD sync → effect is correct here
    const t = this.theme();
    localStorage.setItem('theme', t);
    document.body.classList.toggle('dark', t === 'dark');
  });
}
toggleTheme() { this.theme.update(t => t === 'dark' ? 'light' : 'dark'); }
// Storage + DOM are outside the graph — exactly what effects are for. No signal writes inside.
4 (Medium): A size picker must reset to the first available size whenever the product changes. Implement with linkedSignal, and show why computed alone can't do it.
Solution
product = input.required<Product>();
sizes = computed(() => this.product().availableSizes);
selectedSize = linkedSignal(() => this.sizes()[0]);

pick(size: string) { this.selectedSize.set(size); }     // user choice: WRITABLE ✓
// product changes → sizes recomputes → selectedSize RESETS to new first size ✓

// computed can't do it: it's read-only — the user couldn't pick.
// A plain signal can't do it: nothing would reset it on product change (the tempting
// fix — an effect writing it — is the exact anti-pattern linkedSignal exists to replace).
5 (Hard): Build the debounced search: query signal from an input, 300ms debounce, cancel stale requests, expose results as a signal. Annotate which world each piece lives in.
Solution
query = signal('');                                        // SIGNAL: state
results = toSignal(                                        // SIGNAL out: template reads results()
  toObservable(this.query).pipe(                           // → RxJS in: time-based pipeline
    debounceTime(300),                                     //   wait for typing pause
    distinctUntilChanged(),                                //   skip repeats
    switchMap(q => q.length > 1                            //   CANCEL stale request when
      ? this.api.search(q)                                 //   a new query supersedes it
      : of([] as Result[])),
  ),
  { initialValue: [] as Result[] },                        // value before first emission
);
// template:
// <input [value]="query()" (input)="query.set($any($event.target).value)">
// @for (r of results(); track r.id) { … }
// Signals at the edges (state), RxJS in the middle (events over time) — the canonical split.
6 (Hard): Code-review this service: find every signal-design defect and rewrite it. 
@Injectable({ providedIn: 'root' })
export class CartService {
  items = signal<CartItem[]>([]);
  total = signal(0);
  count = signal(0);
  constructor() {
    effect(() => { this.total.set(this.items().reduce((s,i) => s + i.price, 0)); });
    effect(() => { this.count.set(this.items().length); });
  }
  add(item: CartItem) { this.items().push(item); }
}
Solution
// Defects:
// 1. add() MUTATES the array inside the signal — no notification, everything stale.
// 2. total/count are STORED derivables, synced by effects — two sources of truth,
//    and effect-writes-signal throws by default (the code "works" only if someone
//    disabled the guard).
// 3. items is a PUBLIC WRITABLE signal — any component can .set() the cart arbitrarily,
//    bypassing business rules.

@Injectable({ providedIn: 'root' })
export class CartService {
  private _items = signal<CartItem[]>([]);                 // 3: private writable
  readonly items = this._items.asReadonly();               //    public read-only view
  readonly total = computed(() => this._items().reduce((s, i) => s + i.price, 0));  // 2
  readonly count = computed(() => this._items().length);                            // 2

  add(item: CartItem) {
    this._items.update(l => [...l, item]);                 // 1: immutable update
  }
  remove(id: number) {
    this._items.update(l => l.filter(i => i.id !== id));
  }
}
// Private writable + public computed/readonly + mutation methods — the standard
// signal-service shape (state-management.html builds on exactly this).

🎯 Key Takeaways

  • Signals are values with tracked readers: call to read (registering dependency), set/update to write, notifications only on actual change — and only with new references; mutation is invisible.
  • computed = lazy, memoized, dynamically-tracked derivation — store the minimum, derive the rest; it can't go stale because it isn't stored.
  • effect faces outward (storage, DOM, analytics) with onCleanup for acquired resources; effect-writes-signal is the anti-pattern — use computed or linkedSignal.
  • linkedSignal = writable with derived reset; resource = reactive async with built-in stale-request cancellation.
  • RxJS is repositioned, not replaced: signals for state, observables for events-over-time (debounce, retry, switchMap), converting at the boundary with toSignal/toObservable.
  • Updates are push-invalidate then pull-evaluate: writes mark dirty, reads settle in dependency order, memoized — glitch-free by construction, and the basis for zoneless change detection.
