Signals & Change Detection Basics

Signals are Angular's modern reactive primitive: a wrapper around a value that tracks who reads it and notifies them precisely when it changes. They're replacing Zone.js-based change detection as the way Angular knows when to re-render — a fundamental shift that makes apps faster and more predictable, and heavily tested in current interviews.

The Old World: Zone.js & "Check Everything"

// Classic Angular patches (monkey-patches) async APIs — setTimeout, click handlers, HTTP — via Zone.js.
// After ANY of these fires, Angular runs change detection across the ENTIRE component tree, checking
// every binding to see if it changed ("dirty checking"). It works, but it's coarse: one click can
// trigger checks in components that have nothing to do with what changed.

Signals: Fine-Grained, Precise Reactivity

import { signal, computed, effect } from '@angular/core';

const count = signal(0);              // a reactive container holding a value
count();                              // READ by calling it — Angular tracks who read it
count.set(5);                         // WRITE — replaces the value
count.update(n => n + 1);             // WRITE — derive from the current value

const doubled = computed(() => count() * 2);   // a DERIVED signal — recomputes only when count changes,
                                                // and is cached (memoized) between reads

effect(() => {                        // a SIDE EFFECT that reruns automatically when its signals change
  console.log('count is now', count());
});
// When count changes, Angular knows EXACTLY which components/bindings read it and updates only those —
// no tree-wide dirty checking needed.
🎯 Signals make dependencies explicit and trackable. Reading a signal inside a template, computed, or effect automatically registers that consumer as a dependent. When the signal changes, only its actual dependents are notified and re-rendered — a fundamentally more precise (and faster) model than "check everything after every event," and it's what makes zoneless Angular possible (enterprise lesson).

Signals in Components (the common pattern)

@Component({ selector: 'app-cart', standalone: true, template: `
  <p>Items: {{ items().length }}</p>
  <p>Total: {{ total() }}</p>
  <button (click)="addItem(newItem)">Add</button>
`})
export class CartComponent {
  items = signal<Item[]>([]);
  total = computed(() => this.items().reduce((sum, i) => sum + i.price, 0));

  addItem(item: Item) {
    this.items.update(list => [...list, item]);   // immutable update — new array reference
  }
}
// Reading items()/total() in the template auto-subscribes the view. No manual subscription/unsubscription,
// no async pipe needed — this is a big ergonomic win over the old Observable-in-template pattern for state.

🔗 They're complementary, not competitors. Use signals for state that templates read directly; use RxJS (next: RxJS Deep Dive lesson) for async event streams and complex operator pipelines — then bridge an Observable into a signal with toSignal() when a template needs to read a stream's latest value.

Interview Questions

Q1: How do Signals change Angular's change detection model?

Classic Angular relies on Zone.js, which monkey-patches asynchronous browser APIs (timers, event listeners, XHR/fetch) so that after any of them fires, Angular triggers change detection across the ENTIRE component tree and dirty-checks every binding to see if it changed. This is correct but coarse — one unrelated event can trigger checks in components that have nothing to do with what actually changed. Signals replace this with fine-grained, dependency-tracked reactivity: a signal is a reactive container, and reading it (by calling it as a function) inside a template, a computed(), or an effect() automatically registers that consumer as a dependent. When the signal's value changes, Angular knows EXACTLY which components and bindings depend on it and updates only those, without walking the whole tree. This precision is both a performance win and the technical foundation that makes zoneless Angular (running without Zone.js at all) possible.

Q2: Explain signal(), computed(), and effect() and when to use each.

signal() creates a writable reactive container for a piece of state — you read its current value by calling it as a function (count()) and write with set() (replace) or update() (derive from the current value); it's the primitive for component or application state. computed() derives a READ-ONLY value from other signals via a pure function; it recomputes lazily — only when one of its dependencies actually changes — and its result is memoized between reads, so it's the right tool for derived data like a filtered list or a running total, and it should have no side effects. effect() runs a function for its SIDE EFFECTS (logging, syncing to localStorage, imperative DOM work) whenever any signal it reads changes; it doesn't produce a value and shouldn't be used for state you could instead express as a computed(). The rule of thumb: state → signal, derived data → computed, side effects → effect — reaching for effect() to compute derived data is a common anti-pattern because it adds indirection and risks manual synchronization bugs that computed() avoids automatically.

Q3: How do Signals relate to RxJS Observables — do they replace them?

They're complementary, addressing different problems, not a strict replacement. Signals are a synchronous, pull-based container for STATE: you read the current value directly, updates are simple set/update calls, and cleanup is automatic since a signal doesn't need explicit subscription management. RxJS Observables model asynchronous EVENT STREAMS over time — HTTP responses, WebSocket messages, keystroke streams — with a rich operator pipeline (map, filter, debounceTime, switchMap, etc.) for composing and transforming those streams, but they require manual subscribe/unsubscribe (or the async pipe) to avoid leaks. In practice you use signals for state that templates read directly (component data, UI state) and RxJS for complex asynchronous event processing (typeahead search, WebSocket feeds, combining multiple async sources). Angular provides interop utilities — toSignal() to expose an Observable's latest value as a signal for easy template consumption, and toObservable() to go the other way — so the two models bridge cleanly rather than competing.

🏋️ Practice: convert a component's plain class field to a signal and observe reactivity in the template; add a computed() that derives a total/count from a signal and confirm it only recomputes when its dependency changes; write an effect() that logs on every change; deliberately mutate an array in place inside a signal and observe the missed update, then fix it with an immutable update; use toSignal() to read an HTTP Observable's latest value in a template.

🎯 Key Takeaways

  • Signals replace Zone.js's tree-wide dirty checking with fine-grained, dependency-tracked reactivity — read = subscribe, write = notify only real dependents.
  • signal() = writable state, computed() = pure derived + memoized value, effect() = side effects that rerun on signal change.
  • Always call a signal as a function to read it; always produce new object/array references on write (no in-place mutation).
  • Use computed() for anything that's just derived data — don't reach for effect() to compute state.
  • Signals (sync state) and RxJS (async event streams) are complementary; toSignal()/toObservable() bridge them.
