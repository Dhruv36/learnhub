Lifecycle Hooks Components & Reactivity

What you'll master: the birth-to-death sequence of a component, the four hooks that matter in practice (ngOnInit, ngOnChanges, ngAfterViewInit, ngOnDestroy), constructor-vs-ngOnInit — the interview classic — cleanup discipline with DestroyRef, and how signals (effect, afterNextRender) are absorbing the hooks' old jobs. Prerequisites: Inputs & Outputs. Lifecycle bugs — reading inputs too early, touching DOM too early, cleaning up never — are a distinct genre; this lesson inoculates you.

Part 1 — The Timeline: What Runs When

new Component()             constructor        — DI wiring only; INPUTS NOT SET YET
   ↓  inputs bound
ngOnChanges(changes)                           — before init + on every later input change
ngOnInit()                                     — inputs ready; the setup hook (runs ONCE)
ngDoCheck()                                    — every CD pass (rarely used — see Part 6)
   ↓  content projected
ngAfterContentInit / ngAfterContentChecked     — projected content (ng-content) ready
   ↓  own template rendered
ngAfterViewInit / ngAfterViewChecked           — MY template's DOM + viewChildren ready
   ↓  … component lives, CD passes run …
ngOnDestroy()                                  — about to be removed: CLEAN UP HERE

The order encodes a dependency chain: the class is constructed (DI only), then inputs are bound, then init runs, then caller-projected content settles, then the component's own view exists. Each hook marks the earliest moment its resources are safe to touch — most lifecycle bugs are simply using a resource one stage too early: inputs in the constructor (undefined), viewChild results in ngOnInit (undefined), DOM measurements before ngAfterViewInit (zero heights). You implement a hook by declaring the method (interfaces like OnInit are optional but good documentation).

Part 2 — Constructor vs ngOnInit: the Interview Classic, Settled

export class OrderDetailComponent {
  private api = inject(OrderApi);          // ✓ constructor territory: acquiring dependencies
  id = input.required<number>();

  constructor() {
    // this.id()  → ❌ THROWS (required input not yet set) — inputs bind AFTER construction
    // this.load()→ ❌ work in constructors makes the class hard to test & instantiate
  }

  ngOnInit() {
    // ✓ inputs are bound; do the component's REAL setup once:
    this.load(this.id());
  }
}

Rule: constructor = wiring (inject dependencies, initialize fields).
      ngOnInit    = work    (fetch, subscribe, compute from inputs).

The constructor belongs to JavaScript — it runs when the class is instantiated, before Angular has done anything but DI. ngOnInit belongs to Angular — it runs after the first input binding, which is why "why is my input undefined?" is almost always constructor code that should be init code. The deeper reason to keep constructors trivial: a class whose constructor does work (fetches, subscribes) can't be instantiated in tests without triggering that work — thin constructors are a testability decision, the same one you met in other ecosystems. With signal inputs + effect()/computed(), much "react to inputs" code no longer needs ngOnInit at all — but the hook remains the home for one-time imperative setup.

Part 3 — ngOnChanges vs Signal Reactions

// OLD WORLD — decorator inputs notify via ngOnChanges + a SimpleChanges bag:
@Input() userId!: number;
ngOnChanges(changes: SimpleChanges) {
  if (changes['userId'] && !changes['userId'].firstChange) {
    this.reload(changes['userId'].currentValue);      // stringly-typed, easy to typo
  }
}

// NEW WORLD — signal inputs make change-reaction declarative:
userId = input.required<number>();
user = computed(() => this.users().find(u => u.id === this.userId()));  // derive: no hook
constructor() {
  effect(() => {                       // side-effect per change (runs once immediately too)
    console.log('userId is now', this.userId());
  });
}
// Async per-input-change work (fetch on id change) — rxResource/toObservable+switchMap:
// covered properly in rxjs-patterns.html; the point here: the HOOK is no longer the tool.

ngOnChanges is the legacy mechanism for "do something when an input changes": one method for all inputs, changes addressed by string key, with firstChange flag juggling. You must read it fluently (it's everywhere in existing code), but signal inputs dissolve most uses: derivations become computed() (no hook, no diffing, memoized), and side effects become effect() (tracks exactly the signals it reads). What remains genuinely hook-shaped is less than people think — and the Signals lesson formalizes the replacement patterns.

Part 4 — ngAfterViewInit: When the DOM Actually Exists

export class ChartComponent {
  private canvas = viewChild.required<ElementRef<HTMLCanvasElement>>('cv');
  private chart?: Chart;

  ngAfterViewInit() {
    // ✓ my template is rendered: canvas exists, has real dimensions
    this.chart = new Chart(this.canvas().nativeElement, this.config);
  }
  ngOnDestroy() { this.chart?.destroy(); }
}

// Modern alternative — render callbacks (also SSR-safe: they run ONLY in the browser):
constructor() {
  afterNextRender(() => {
    this.chart = new Chart(this.canvas().nativeElement, this.config);
  });
}

ngAfterViewInit is the earliest moment the component's own DOM — and therefore viewChild results, element sizes, third-party widget mount points — exists. It's the hook for integrating imperative libraries (charts, maps, editors), focusing elements, and measuring. Two production notes: change detection has just finished when it runs, so synchronously mutating bound state inside it triggers the dev-mode ExpressionChangedAfterItHasBeenChecked error (defer with an effect/microtask, or restructure); and on the server (SSR) there's no real layout — the newer afterNextRender()/afterEveryRender() callbacks run only in the browser, making them the SSR-safe home for DOM work (Enterprise lesson).

Part 5 — ngOnDestroy & the Cleanup Discipline

export class LiveTickerComponent {
  private es?: EventSource;
  private timer?: number;

  ngOnInit() {
    this.es = new EventSource('/api/prices');            // acquired → must be released
    this.es.onmessage = (e) => this.apply(JSON.parse(e.data));
    this.timer = window.setInterval(() => this.ping(), 30_000);
  }
  ngOnDestroy() {
    this.es?.close();                                    // symmetric teardown
    clearInterval(this.timer);
  }
}

// The modern, forget-proof version — register cleanup WHERE you acquire:
export class LiveTickerComponent {
  private destroyRef = inject(DestroyRef);
  ngOnInit() {
    const es = new EventSource('/api/prices');
    es.onmessage = (e) => this.apply(JSON.parse(e.data));
    this.destroyRef.onDestroy(() => es.close());         // colocated: acquire + release together
  }
}
// RxJS subscriptions get takeUntilDestroyed(...) — same idea, rxjs-patterns.html.

Every component that acquires — subscriptions, intervals, sockets, observers (IntersectionObserver…), manual event listeners, third-party widgets — must release on destroy, or the resource (and everything its callback closes over, usually the whole component) outlives the view: the SPA memory-leak pattern from the JS track, plus ghost callbacks acting on dead views. DestroyRef improves on the hook by colocation: the release registers next to the acquire, so reviewers verify pairing at a glance and refactors can't separate them. Note what does not need manual cleanup: HttpClient one-shot requests (they complete), the async pipe (unsubscribes itself), signal effect()s (auto-disposed with their context), and DOM listeners bound in templates — Angular removes those with the view.

Part 6 — Deep Dive: The Checked Hooks, DoCheck & Why Signals Shrink This Lesson

ngDoCheck / ngAfterContentChecked / ngAfterViewChecked run on EVERY change-detection
pass — for every component instance. That's their danger:

ngAfterViewChecked() {
  this.recomputeLayout();     // ❌ runs dozens of times per second under interaction
}

Legitimate-but-rare uses:
  ngDoCheck            custom dirty-checking of mutated structures Angular can't see
                       (with KeyValueDiffers/IterableDiffers) — mostly pre-signals era
  AfterViewChecked     last-resort sync with stubborn third-party DOM widgets

The direction of travel: hooks are IMPERATIVE lifecycle ("run my code at moment X");
signals are DECLARATIVE dependency ("this derives from that"). computed/effect/
afterNextRender/DestroyRef replace most hook use cases with code that states WHY it
runs, not WHEN. What genuinely remains hook-shaped: one-time imperative setup
(ngOnInit), browser-only DOM integration (afterNextRender/ngAfterViewInit), teardown
(DestroyRef/ngOnDestroy).

The *Checked hooks exist because pre-signals Angular couldn't know what changed — only that a pass ran; they gave code a chance to react after each check. Anything non-trivial inside them is a performance bug multiplied by pass frequency, and mutating bound state there loops change detection. Treat them as read-only observation points, prefer restructuring so you don't need them, and recognize that their existence is the strongest argument for the signals model you'll meet next lesson: fine-grained dependency tracking makes "react to any change, somehow" hooks obsolete.

Senior framing: reviewers read lifecycle code as a competence signal. Green flags: trivial constructors; ngOnInit that reads as a setup table of contents; every acquire visibly paired with a release (ideally via DestroyRef/ takeUntilDestroyed at the acquisition site); DOM work confined to afterNextRender/ngAfterViewInit. Red flags: work in constructors, input reads before init, business logic in *Checked hooks, empty ngOnDestroy in a component that clearly subscribes, and setTimeout sprinkled to dodge ExpressionChangedAfterItHasBeenChecked — each timeout is an unfixed ordering bug wearing a band-aid. The question a senior asks of any hook usage: "could this be a computed, an effect, or a render callback instead?" — increasingly, yes.

Interview Questions

Q1 (Beginner): Constructor vs ngOnInit — what belongs where, and why does the split exist?

Constructor: dependency acquisition (inject(...)) and trivial field init — it runs at class instantiation, before Angular binds inputs. ngOnInit: the component's real setup — fetching, subscribing, computing from inputs — because it runs once after the first input binding, so inputs are safely readable. The split exists because construction is a JavaScript concern and initialization is an Angular-lifecycle concern; merging them causes the two classic bugs: undefined inputs and untestable work-performing constructors.

Q2 (Beginner): Name the lifecycle order for: constructor, ngOnInit, ngAfterViewInit, ngOnChanges, ngOnDestroy — and what each marks.

constructor (class created, DI done) → ngOnChanges (first input binding — and again on every later input change) → ngOnInit (inputs ready; one-time setup) → ngAfterViewInit (own template rendered; view queries and DOM ready) → …lifetime… → ngOnDestroy (about to be removed; release everything acquired). Between init and destroy, content/view checked hooks fire per change-detection pass.

Q3 (Mid): A component starts a setInterval and subscribes to a WebSocket service in ngOnInit, with no ngOnDestroy. What happens as users navigate, and what's the modern fix?

Each visit creates a new interval and subscription; navigation destroys the view but neither is released, so callbacks keep firing against destroyed components — the component (and everything it references) is pinned in memory by the callback closures. Symptoms: rising memory, duplicated handling after re-visits, "ghost" updates/errors. Fix: release on destroy — modernly by colocating cleanup with acquisition: inject(DestroyRef).onDestroy(() => clearInterval(id)) and ws.messages$.pipe(takeUntilDestroyed()), so the pairing is visible where the resource is created rather than trusted to a distant hook.

Q4 (Mid): How do signal inputs change the role of ngOnChanges?

ngOnChanges bundled all input reactions into one method keyed by strings, with SimpleChanges/firstChange bookkeeping. Signal inputs make each input an observable-by-the-graph value: derivations become computed() (memoized, no hook), side effects become effect() (tracking exactly what they read), and async reactions (refetch on id change) use interop like toObservable(id).pipe(switchMap(...)) or resource APIs. The hook remains necessary only for legacy decorator inputs — in signal-first components it typically disappears entirely.

Q5 (Senior): Explain ExpressionChangedAfterItHasBeenCheckedError — what sequence triggers it, why only in dev mode, and how to fix it properly.

Change detection evaluates bindings top-down; afterwards, dev mode runs a second verification pass asserting nothing changed mid-flight. If code that runs during/after the first pass (child constructor/init mutating a parent's bound value, ngAfterViewInit setting bound state, a shared service mutated during render) alters an already-checked binding, the verification pass detects the drift and throws. Prod skips verification for speed — the inconsistency just silently persists until the next pass. Proper fixes: restructure so the value is correct before rendering (lift state, derive with computed), or make the mutation explicitly asynchronous (effect/microtask) so it starts a new CD cycle. setTimeout sprinkling "fixes" the symptom by accident and should be treated as a flagged TODO, not a solution.

Q6 (Senior): Your app is moving to SSR and several components initialize charts in ngAfterViewInit. What breaks and how do you modernize?

On the server, lifecycle hooks run during rendering but there's no real browser: window and layout APIs are missing, dimensions are meaningless, and chart libraries touching canvas/DOM globals crash the render or produce markup that mismatches on hydration. Modernize with afterNextRender(() => …), which is contractually browser-only — the chart initializes after client-side render with real dimensions; SSR emits the placeholder markup cleanly. Pair with @defer for heavy widgets (server renders the placeholder block), keep any remaining platform checks explicit (isPlatformBrowser in legacy spots), and destroy widget instances via DestroyRef so client navigation doesn't leak them.

  name = input.required<string>();

}

// Constructor runs at instantiation; Angular binds inputs AFTER construction.
// Even better, if the goal is "log whenever name changes":
// constructor() { effect(() => console.log('Hello ' + this.name())); }
2 (Easy): Write a component that autofocuses its search input when it appears — using the right hook — and note the SSR-safe alternative.
Solution
@Component({ selector: 'app-search', template: `<input #box placeholder="Search">` })
export class SearchComponent implements AfterViewInit {
  private box = viewChild.required<ElementRef<HTMLInputElement>>('box');
  ngAfterViewInit() { this.box().nativeElement.focus(); }   // DOM exists here, not in ngOnInit
}
// SSR-safe modern form:
// constructor() { afterNextRender(() => this.box().nativeElement.focus()); }
// afterNextRender never runs on the server — no window/DOM assumptions leak into SSR.
3 (Medium): Build a CountdownComponent (input: seconds) that ticks once per second, emits finished, and cannot leak its interval — using DestroyRef.
Solution
@Component({ selector: 'app-countdown', template: `{{ remaining() }}s` })
export class CountdownComponent implements OnInit {
  seconds = input.required<number>();
  finished = output<void>();
  remaining = signal(0);
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    this.remaining.set(this.seconds());
    const id = window.setInterval(() => {
      this.remaining.update(r => r - 1);
      if (this.remaining() <= 0) { clearInterval(id); this.finished.emit(); }
    }, 1000);
    this.destroyRef.onDestroy(() => clearInterval(id));   // colocated with the acquire —
  }                                                        // navigation mid-countdown can't leak
}
4 (Medium): A legacy component reloads data via ngOnChanges when its userId input changes. Show the legacy version, then the signal-input modernization.
Solution
// LEGACY:
@Input() userId!: number;
ngOnChanges(ch: SimpleChanges) {
  if (ch['userId']) this.load(ch['userId'].currentValue);   // string keys, manual diffing
}

// MODERN — the reaction is declared, not diffed:
userId = input.required<number>();
user = signal<User | null>(null);
constructor(private api: UserApi) {
  effect(() => {
    const id = this.userId();                 // effect tracks this read
    this.user.set(null);
    this.api.get(id).subscribe(u => this.user.set(u));
    // production version: toObservable(this.userId).pipe(switchMap(id => api.get(id)))
    // to cancel stale requests — see rxjs-patterns.html
  });
}
// No hook, no SimpleChanges, runs on init AND every change — one code path.

6 (Hard): Code-review this component: list every lifecycle defect and rewrite it cleanly. 
export class StockWidgetComponent {
  symbol = input.required<string>();
  price = signal(0);
  constructor(private ws: WsService) {
    this.ws.connect('/stocks/' + this.symbol()).subscribe(p => this.price.set(p));
  }
  ngAfterViewChecked() { this.resizeChart(); }
}
Solution
// Defects:
// 1. this.symbol() read in the CONSTRUCTOR — required input not bound yet: throws.
// 2. Work (connect+subscribe) in the constructor — untestable, and wrong timing anyway.
// 3. Subscription never released — leaks socket + component on navigation.
// 4. symbol changes are ignored — connected once to the initial (broken) value.
// 5. resizeChart() in ngAfterViewChecked — runs EVERY CD pass; layout work per pass.

export class StockWidgetComponent {
  symbol = input.required<string>();
  private ws = inject(WsService);
  // 1+2+4: declarative, re-connects per symbol, cancels the old stream (switchMap):
  price = toSignal(
    toObservable(this.symbol).pipe(switchMap(s => this.ws.connect('/stocks/' + s))),
    { initialValue: 0 },
  );  // 3: toSignal ties the subscription to the component's lifetime — auto-cleanup.

  constructor() {
    afterNextRender(() => this.resizeChart());          // 5: once, in the browser,
    // + a ResizeObserver (registered with DestroyRef cleanup) if it must track size.
  }
}

🎯 Key Takeaways

  • The lifecycle is a dependency chain — constructor (DI only) → inputs bound → ngOnInit (setup) → content → view (ngAfterViewInit: DOM/queries ready) → destroy. Most bugs are "one stage too early".
  • Constructor wires, ngOnInit works: thin constructors are a testability decision, and inputs simply don't exist before init.
  • Signal inputs retire most ngOnChanges: derive with computed(), side-effect with effect() — reactions declared by dependency, not diffed by hook.
  • Every acquire needs a release: DestroyRef.onDestroy colocated at the acquisition site (or takeUntilDestroyed) beats a distant ngOnDestroy; async pipe/effects/template listeners clean themselves.
  • *Checked/ngDoCheck run every CD pass — near-empty observation points at most; heavy work there is a frame-rate tax.
  • DOM integration goes in afterNextRender (browser-only, SSR-safe) or ngAfterViewInit; ExpressionChangedAfterItHasBeenChecked means state mutated mid-render — restructure, don't setTimeout.
