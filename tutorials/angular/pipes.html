Pipes Foundations

What you'll master: the built-in pipes and their locale story, writing custom pipes, and — the part interviews probe — why pipes exist at all: pure pipes are memoized template-level transforms, the sanctioned answer to "never call work-performing methods in bindings". You'll also learn the purity contract, its interaction with mutation, when impure pipes are justified, and the async pipe (fully explored later with RxJS). Prerequisites: Templates & Data Binding.

Part 1 — What a Pipe Is: Presentation Transforms, in the Template

{{ order.createdAt | date:'mediumDate' }}        →  Jul 18, 2026
{{ product.price | currency:'EUR' }}             →  €1,499.00
{{ user.name | titlecase }}                      →  Ada Lovelace
{{ score | percent:'1.0-1' }}                    →  87.5%
{{ description | slice:0:80 }}…                  →  first 80 chars
{{ config | json }}                              →  debug-print any object

<!-- Chaining — output of one feeds the next: -->
{{ event.start | date:'shortTime' | uppercase }} →  4:10 PM

<!-- Parameters after colons: -->
{{ price | currency:'INR':'symbol':'1.0-0' }}    →  ₹1,499

A pipe is a pure display transform: the model keeps clean data (a Date, a number in cents), and the template declares how to present it. That separation is the point — formatting is a view concern, so it lives in the view, reusable across every template with zero imports of formatting code into components. Pipes take arguments after colons and chain left-to-right. The formatting pipes (date, currency, number, percent) sit on top of the same i18n machinery you met in the JS track's Intl lesson — locale-aware by design, driven by Angular's LOCALE_ID.

Two habits worth forming immediately. First, formatting pipes are standalone imports now — imports: [DatePipe, CurrencyPipe] per component (or CommonModule for the lot). Second, the digits-info mini-language ('1.2-2' = at least 1 integer digit, exactly 2 fraction digits) appears in number, currency, and percent — learn it once, use it everywhere.

Part 3 — Why Pipes Beat Methods: the Purity Contract

<!-- ❌ Runs on EVERY change-detection pass — see data-binding.html Part 6: -->
<li>{{ formatDistance(row.meters) }}</li>

<!-- ✓ A PURE pipe runs only when its INPUT (by reference) or its args change: -->
<li>{{ row.meters | distance }}</li>

How Angular decides whether to re-run a pure pipe on a CD pass:
  prevInput === newInput  &&  same args   →  SKIP, reuse last result   (memoized!)
  reference/primitive changed             →  re-run transform once

The fine print: the check is REFERENCE equality. Mutate an array in place
(items.push(x)) and a pure pipe filtering it will NOT re-run — same reference.
Immutable updates (items = [...items, x]) are what make pure pipes correct.

This is the lesson's core idea. A pure pipe is a memoized function at the template level: Angular caches the last input/args and result, and skips the transform entirely when nothing changed — turning "formatting in a hot list" from an every-pass cost into an almost-never cost. The contract cuts both ways: purity is assumed, so a pipe reading global state, or fed in-place-mutated data, silently shows stale output. This dovetails with the immutability discipline from Functional Patterns — Angular's pure pipes, like OnPush change detection later, are designed around new reference = new data.

Part 4 — Writing Custom Pipes

// distance.pipe.ts
import { Pipe, PipeTransform } from '@angular/core';

@Pipe({ name: 'distance' })                     // pure: true is the DEFAULT
export class DistancePipe implements PipeTransform {
  transform(meters: number, unit: 'auto' | 'km' | 'm' = 'auto'): string {
    if (meters == null || Number.isNaN(meters)) return '—';     // be null-tolerant
    if (unit === 'm' || (unit === 'auto' && meters < 1000)) {
      return `${Math.round(meters)} m`;
    }
    return `${(meters / 1000).toFixed(1)} km`;
  }
}

// usage (after adding DistancePipe to the component's imports):
// {{ run.meters | distance }}          → "850 m" or "12.3 km"
// {{ run.meters | distance:'km' }}     → forced km

// A filter pipe — fine IF fed immutable data:
@Pipe({ name: 'byStatus' })
export class ByStatusPipe implements PipeTransform {
  transform(orders: Order[], status: string): Order[] {
    return orders.filter(o => o.status === status);
  }
}

A custom pipe is a class with transform(value, ...args) — keep it a pure function: no service calls, no state, deterministic output. Design notes that separate production pipes from tutorial pipes: tolerate null/undefined gracefully (templates often render before data arrives — returning "—" beats a crash), type the signature honestly (it participates in template type-checking), and name it as a domain word (distance, timeAgo, fileSize) so templates read as specifications. For filtering/sorting pipes specifically, the classic advice ("don't — do it in the component") is really about the mutation trap and hidden cost on big lists; with immutable updates and signal-derived data, a computed() in the class is usually still the clearer home.

Part 5 — Impure Pipes: the Escape Hatch and Its Price

@Pipe({ name: 'filterLive', pure: false })      // ← re-runs EVERY change-detection pass
export class FilterLivePipe implements PipeTransform {
  transform(items: Item[], q: string): Item[] {
    return items.filter(i => i.name.includes(q));   // now runs constantly — even on
  }                                                  // mutated-in-place arrays
}

Built-in impure pipes — and why they must be:
  async     — has to notice when the Observable EMITS (no input reference changes)
  json      — object may be mutated in place; it's a debug tool
  keyvalue  — (impure-ish behavior for the same reason)

The price: whatever an impure pipe does, it does on every CD pass for every usage.
An impure filter over 10k rows = a full filter per pass. That's why 'pure: false'
should trigger the same reviewer reflex as a method call in a binding.

Marking a pipe pure: false opts out of memoization: it runs on every pass, so it notices in-place mutation and external changes — and costs accordingly. The built-in async pipe is the legitimate archetype: its input (the Observable reference) never changes, but its output must update per emission, which is impossible under pure semantics. Your own use of pure: false should be rare and deliberate: tiny transforms, or migration stopgaps for legacy mutation-heavy code. If you're tempted to write an impure filter pipe, the real fix is upstream — immutable updates or a computed() — not disabling the cache.

Part 6 — Deep Dive: Locale, i18n & Where Pipes Fit in Architecture

// Formatting pipes read LOCALE_ID — default 'en-US' unless you register more:
// app.config.ts
import { registerLocaleData } from '@angular/common';
import localeDe from '@angular/common/locales/de';
registerLocaleData(localeDe);

providers: [{ provide: LOCALE_ID, useValue: 'de-DE' }]
// Now: {{ 1234.5 | number }}  → "1.234,5"     {{ d | date }} → "18.07.2026"
// Per-usage override: {{ amount | currency:'EUR':'symbol':'1.2-2':'de' }}

// The architecture question — where should formatting live?
UI formatting (dates, money, casing)      → PIPES (view concern, locale-aware, memoized)
Derived DATA (filtered lists, totals)     → computed() signals in the class (testable,
                                            reusable outside templates, signal-reactive)
Reusable app-wide value logic             → plain functions in shared/ — pipes can WRAP them
                                            (pipe = template adapter over a tested function)

Angular's formatting pipes ship with real CLDR locale data — registered per locale to keep bundles lean — which is why you should never hand-roll date or currency strings in templates: you'd be reimplementing, badly, what LOCALE_ID switches for free. Architecturally, the mature pattern is pipe as a thin template adapter over a plain, unit-tested function: the function lives in shared/, usable in services and components; the pipe just exposes it to templates with memoization. And the modern boundary with signals is worth stating crisply: pipes format values; computed() derives data. When a transformation's result is "data you might also use in logic" (a filtered list, a total), derive it in the class; when it's "how this value looks to a human", pipe it.

Senior framing: pipes are one of Angular's best examples of making the fast thing the easy thing — the syntax developers reach for naturally (| currency) is memoized, locale-correct, and testable, while the naive alternative (method calls in bindings) is the slow path. When you review templates, the pipe usage tells you about data hygiene too: a team whose filter pipes "stopped working" has an in-place mutation problem, not a pipe problem — the same reference-equality discipline that OnPush and signals demand. One caution from the other direction: don't over-pipe business logic into templates — {{ cart | applyDiscounts | withTax | grandTotal }} hides a pricing engine in view syntax; that chain belongs in a tested service/computed, with at most a final formatting pipe.

Interview Questions

Q1 (Beginner): What is a pipe, and why use {{ price | currency }} over formatting in the component class?

A pipe is a declarative display transform applied in the template. Keeping formatting in a pipe (a) keeps the model clean — the class holds a number, not a display string; (b) is locale-aware for free via LOCALE_ID; (c) is reusable in every template without importing formatting logic into components; and (d) — the performance point — pure pipes are memoized, re-running only when the input changes, unlike a formatting method which would run every change-detection pass.

Q2 (Beginner): What does "pure pipe" mean, and when does Angular re-run one?

Pure (the default) means Angular assumes same input → same output, and therefore caches: on each change-detection pass the pipe re-runs only if the input value (by reference for objects, by value for primitives) or any argument changed — otherwise the previous result is reused. That makes pure pipes effectively free in steady state, and is why they're the sanctioned way to transform in bindings.

Q3 (Mid): A filterByStatus pipe stops updating after the team switched to orders.push(newOrder). Explain and give two fixes, with the trade-off.

The pipe is pure, and push mutates in place — the array reference never changes, so the memoization check says "same input" and skips the transform: the new order never appears. Fix A (correct one): immutable updates — orders = [...orders, newOrder] (or signals with update) — new reference, pipe re-runs once. Fix B: mark the pipe pure: false — it now re-runs every CD pass, noticing the mutation but paying a full filter per pass for every usage; acceptable as a stopgap, wrong as a policy. The underlying issue is data-flow discipline, not the pipe.

Q4 (Mid): Why must the async pipe be impure? What does it do besides unwrap values?

Its input — the Observable/Promise reference — typically never changes, but its output must update every time the source emits. Pure semantics ("re-run only when the input reference changes") would freeze it at the first value, so it's impure: checked every pass, returning the latest emitted value. Beyond unwrapping, it subscribes on first use, triggers change detection on emissions (it works even under OnPush), and — critically — unsubscribes automatically when the view is destroyed, eliminating a whole leak class handled manually otherwise (RxJS Patterns).

Q5 (Senior): Where do you draw the line between pipes, computed() signals, and service methods for transformations?

By what the result is. Human-facing formatting of a single value (money, dates, casing, sizes) → pipes: view-layer, locale-aware, memoized, reusable across templates. Derived data — filtered/sorted lists, aggregates, view models — → computed() in the class: it's testable, reactive to signal changes, usable by logic (not just templates), and computed once rather than per-template-usage. Cross-cutting business transforms (pricing, permissions) → services/plain functions, unit-tested, with pipes at most as thin template adapters over them. Red flag in either direction: business rules hidden in pipe chains, or components hand-assembling display strings that a pipe + locale would own.

Q6 (Senior): Your dashboard renders 5,000 rows, each using three custom pipes, and profiling shows pipe transforms are cheap but the page is still slow on every interaction. A colleague proposes making the pipes impure "so they cache better". Respond.

The proposal is backwards: impure pipes disable caching — they'd run all 15,000 transforms per change-detection pass, making things strictly worse. If pure-pipe transforms are already cheap and skipped in steady state, the cost is elsewhere: 5,000 rows × dozens of bindings each is the real per-pass burden. The fixes are structural: narrow change detection (OnPush/signal components so unaffected rows aren't checked), track-keyed @for for minimal DOM churn, virtualization (CDK virtual scroll) so only visible rows exist, and precomputed view models so each row binds plain fields. Pipes are the one part already behaving optimally — the interview point is recognizing that memoization ≠ impurity, and that binding count, not transform cost, dominates at this scale.

3 (Medium): Write a null-tolerant fileSize pipe: 532 → "532 B", 2048 → "2.0 KB", 5,242,880 → "5.0 MB", null → "—". Include a decimals argument (default 1).
Solution
@Pipe({ name: 'fileSize' })
export class FileSizePipe implements PipeTransform {
  transform(bytes: number | null | undefined, decimals = 1): string {
    if (bytes == null || Number.isNaN(bytes)) return '—';
    if (bytes < 1024) return `${bytes} B`;
    const units = ['KB', 'MB', 'GB', 'TB'];
    let value = bytes, i = -1;
    do { value /= 1024; i++; } while (value >= 1024 && i < units.length - 1);
    return `${value.toFixed(decimals)} ${units[i]}`;
  }
}
// {{ doc.bytes | fileSize }}      → "2.0 KB"
// {{ doc.bytes | fileSize:0 }}    → "2 KB"
// Null-tolerance matters: templates render during loading, before doc arrives.
4 (Medium): Write a timeAgo pipe returning "3 days ago"/"in 2 hours" using Intl.RelativeTimeFormat, and explain the purity caveat it carries.
Solution
@Pipe({ name: 'timeAgo' })
export class TimeAgoPipe implements PipeTransform {
  private rtf = new Intl.RelativeTimeFormat('en', { numeric: 'auto' });
  transform(date: Date | string | null): string {
    if (!date) return '';
    const diffSec = Math.round((new Date(date).getTime() - Date.now()) / 1000);
    const units: [Intl.RelativeTimeFormatUnit, number][] =
      [['year', 31536000], ['month', 2592000], ['day', 86400], ['hour', 3600], ['minute', 60]];
    for (const [unit, sec] of units)
      if (Math.abs(diffSec) >= sec) return this.rtf.format(Math.trunc(diffSec / sec), unit);
    return this.rtf.format(diffSec, 'second');
  }
}
// CAVEAT: the pipe reads Date.now() — a hidden input. As a PURE pipe it won't re-run
// until the bound date changes, so "3 minutes ago" goes stale on screen. Options:
// accept it (fine for feeds that re-render), tick a signal timer that recreates the
// bound value periodically, or (last resort) pure: false — knowing it then runs per pass.
5 (Hard): Demonstrate the pure-pipe mutation trap end to end: a component with an add-item button whose filter pipe fails under push, then the corrected version. Annotate the key lines.
Solution
@Pipe({ name: 'active' })
export class ActivePipe implements PipeTransform {
  transform(items: Task[]): Task[] { return items.filter(t => !t.done); }  // pure (default)
}

@Component({
  imports: [ActivePipe],
  template: `
    <button (click)="addBroken()">Add (broken)</button>
    <button (click)="addFixed()">Add (fixed)</button>
    @for (t of tasks() | active; track t.id) { <li>{{ t.title }}</li> }
  `,
})
export class TaskListComponent {
  tasks = signal<Task[]>([{ id: 1, title: 'First', done: false }]);

  addBroken() {
    this.tasks().push({ id: Date.now(), title: 'New', done: false });
    // Mutates the array INSIDE the signal: same reference → neither the signal
    // nor the pure pipe registers a change → UI never shows the task.
  }
  addFixed() {
    this.tasks.update(list => [...list, { id: Date.now(), title: 'New', done: false }]);
    // New array reference → signal notifies → pipe input changed → re-runs ONCE. ✓
  }
}
6 (Hard): Your app must support en-US and de-DE. Show the locale registration, and a price line that renders "€1,234.50" for en-US users but "1.234,50 €" for de-DE — from the same template.

// <p>{{ price | currency:'EUR' }}</p>
// en-US → "€1,234.50"        de-DE → "1.234,50 €"

// exactly the class of detail hand-formatting always gets wrong.

🎯 Key Takeaways

  • Pipes are declarative display transforms: model stays clean data, template declares presentation — date/currency/number/percent are locale-aware via LOCALE_ID.
  • Pure pipes (the default) are memoized: re-run only when input reference or args change — the sanctioned fix for "no method calls in bindings".
  • The purity contract assumes immutability: in-place mutation leaves the reference unchanged → stale output. Fix the data flow, don't mark pipes impure.
  • pure: false runs every CD pass — legitimate for async (must react to emissions), a performance smell almost everywhere else.
  • Custom pipes: pure, null-tolerant, honestly typed, domain-named — ideally thin template adapters over plain tested functions.
  • Boundary rule: pipes format values; computed() derives data; services own business transforms.
