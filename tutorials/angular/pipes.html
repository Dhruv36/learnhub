Pipes Basics

Pipes transform a value for DISPLAY, right in the template, without changing the underlying data or cluttering the component class with formatting logic. Angular ships common ones (dates, currency, JSON); you can write your own. The most important interview topic here is pure vs impure pipes and how that interacts with performance.

Built-in Pipes

{{ user.createdAt | date:'mediumDate' }}         <!-- Jan 5, 2026 -->
{{ price | currency:'USD' }}                      <!-- $19.99 -->
{{ ratio | percent:'1.0-2' }}                      <!-- 42.5% -->
{{ name | uppercase }}  {{ name | lowercase }}     <!-- text case -->
{{ longText | slice:0:100 }}                       <!-- substring -->
{{ obj | json }}                                   <!-- pretty-printed, great for debugging -->
{{ items | async }}                                <!-- unwraps a Promise/Observable, auto-unsubscribes! -->
// Pipes chain left-to-right:  {{ price | currency:'USD' | uppercase }}
🔄 The async pipe is the RxJS/template bridge you'll use constantly. It subscribes to an Observable (or resolves a Promise), returns its latest value to the template, AND automatically unsubscribes when the component is destroyed — eliminating a whole category of manual subscription-management bugs. Combined with signals (previous lesson) and toSignal(), it's one of several tools for getting async data into a template safely.

Writing a Custom Pipe

import { Pipe, PipeTransform } from '@angular/core';

@Pipe({ name: 'truncate', standalone: true })
export class TruncatePipe implements PipeTransform {
  transform(value: string, limit = 20): string {
    return value.length > limit ? value.slice(0, limit) + '…' : value;
  }
}
// Usage:  {{ description | truncate:50 }}
// A pipe is just a class with a transform() method — pure input → output, ideal for display formatting.

Interview Questions

Q1: What's the difference between a pure and an impure pipe?

A pure pipe (the default) only re-executes its transform() when Angular detects that its input REFERENCE has changed — so passing a new array/object triggers it, but mutating the existing array/object in place (push, splice, direct property assignment) does not, because the reference is identical and Angular's pure-pipe check is a cheap reference comparison. An impure pipe (declared with pure: false) re-executes on EVERY change detection cycle regardless of whether its input changed, so it does catch in-place mutations, but at a real performance cost, especially on large lists or expensive transforms, since it reruns far more often than necessary. The practical guidance is to keep pipes pure by default and fix the actual problem — usually treat data immutably (replace the array/object rather than mutating it) — rather than reaching for pure: false as a blanket fix, since that reintroduces the exact "check everything constantly" cost that fine-grained reactivity (signals) is designed to avoid.

Q2: What does the async pipe do and why is it preferred over manual subscription?

The async pipe takes an Observable (or Promise) directly in the template — {{ data$ | async }} — and does three things: it subscribes to it, returns the latest emitted value for rendering, and automatically unsubscribes when the host component is destroyed. This is preferred over manually calling .subscribe() in the component class because manual subscriptions require you to remember to unsubscribe (commonly in ngOnDestroy, or via a takeUntil pattern) — forgetting this is one of the most common sources of memory leaks and stale-callback bugs in Angular apps, especially for long-lived services or components that are created and destroyed repeatedly (like items in a routed view). The async pipe removes that entire class of bug by tying subscription lifecycle to the component's lifecycle automatically, while also keeping the template declarative — you bind directly to the stream rather than manufacturing a plain field the class has to keep in sync.

Q3: When would you use a pipe versus a computed signal for derived display data?

Both transform data for display, but they suit different situations. A pipe is the better choice when the transformation is generic and reusable across many components — formatting a date, currency, or percentage, truncating text — because it can be dropped into any template with | pipeName:args and is parameterized declaratively; it's also the natural tool when you need the async pipe's automatic subscription/unsubscription handling for an Observable or Promise. A computed() signal is preferable for derived state that's specific to a component's own logic — a filtered list, a running total, a validity flag built from several other signals — because it's colocated with the rest of the component's typed state, composes naturally with other signals, and (being lazily evaluated and memoized) is efficient without needing the pure/impure distinction pipes require. In modern Angular codebases, the general trend is to prefer computed() for component-local derived state and reserve pipes for reusable, template-level formatting concerns and Observable interop via async.

🏋️ Practice: use built-in date/currency/percent pipes and chain two together; write a custom truncate pipe with a parameter; bind an HTTP Observable directly in a template with the async pipe and confirm no manual unsubscribe is needed; break a pure pipe by mutating its input array in place, observe the stale UI, then fix it with an immutable update; deliberately mark a filter pipe impure and compare change-detection frequency; rewrite a pipe-based derivation as a computed() signal.

🎯 Key Takeaways

  • Pipes transform values for display in the template ({{ value | pipeName:args }}) without touching the underlying data.
  • Built-ins: date, currency, percent, uppercase/lowercase, slice, json, and especially async (auto-subscribes AND auto-unsubscribes).
  • Pure pipes (default) rerun only when the input reference changes — cheap, but miss in-place mutations; impure pipes rerun every cycle — catch mutations but cost performance.
  • Fix pure-pipe "stale UI" bugs with immutable data updates, not by defaulting every pipe to impure.
  • Prefer computed() for component-local derived state; reach for a pipe when the transform is generic/reusable or needs async's subscription handling.
