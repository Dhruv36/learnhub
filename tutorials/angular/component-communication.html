Inputs, Outputs & Communication Components & Reactivity

What you'll master: every sanctioned way components talk — signal inputs (required, transformed, aliased), outputs, two-way model(), content projection with ng-content, template queries (viewChild/contentChild), and the decision framework for parent–child vs distant-component communication (spoiler: distant means a shared service). Prerequisites: Components and Data Binding. Component API design is the skill that separates "can write a component" from "can design a component library".

Part 1 — Inputs: the Component's Parameters

import { Component, input, computed } from '@angular/core';

@Component({ selector: 'app-price-tag', template: `…` })
export class PriceTagComponent {
  // The modern signal-input API (v17.1+):
  amount   = input.required<number>();            // no default — parent MUST bind it (compile-checked)
  currency = input('EUR');                        // optional, defaulted, type inferred
  discount = input(0, {
    transform: (v: number | string) => Number(v), // normalize what parents pass
  });
  size     = input<'sm' | 'lg'>('sm', { alias: 'tagSize' });  // public name ≠ field name

  // Inputs are SIGNALS — derive state instead of recomputing in the template:
  final = computed(() => this.amount() * (1 - this.discount()));
}

<!-- parent: -->
<app-price-tag [amount]="p.price" [discount]="p.promo" tagSize="lg" />
<!-- static string for a non-changing value needs no brackets: tagSize="lg" -->

Inputs are the component's parameters — its data contract with any parent. input.required moves "you forgot to pass X" from runtime undefined-bugs to compile errors. Because signal inputs are signals, reacting to changes stops being ceremony: derive with computed(), side-effect with effect() — no ngOnChanges diffing (the Lifecycle lesson covers the old hook and when you still see it). The transform option handles the boundary annoyance of attribute-style values: booleanAttribute and numberAttribute (from @angular/core) are prebuilt transforms so <app-x collapsed /> arrives as a real true.

Legacy form you'll still meet everywhere: @Input() amount!: number; — a decorated plain property. It works, but changes are only observable via ngOnChanges or setters, not reactively. New code: signal inputs. Reading old code: @Input() ≈ input() without the signal superpowers.

Part 2 — Outputs: Typed Events Up

import { Component, output } from '@angular/core';

@Component({
  selector: 'app-search-box',
  template: `
    <input #q (keydown.enter)="submit(q.value)" placeholder="Search…">
    <button (click)="submit(q.value)">Go</button>
  `,
})
export class SearchBoxComponent {
  search = output<string>();                     // typed event stream OUT
  cleared = output<void>();                      // payload-less events use void

  submit(value: string) {
    const q = value.trim();
    q ? this.search.emit(q) : this.cleared.emit();
  }
}

<!-- parent hears it like a DOM event; $event IS the payload: -->
<app-search-box (search)="runSearch($event)" (cleared)="reset()" />

Outputs are the mirror of inputs: the child announces, the parent decides. The child stays policy-free — it doesn't know what a search does, only that one was requested — which is exactly what makes it reusable. Design guidance that scales: name outputs as events (search, itemSelected, dismissed), not as commands aimed at a known parent (updateParentList); emit domain data (the item, the id), not internal state or DOM events; and keep payloads minimal — a parent handed the child's whole internal state is a coupling smell. (Legacy: @Output() search = new EventEmitter<string>() — same usage, RxJS-based; output() is the modern, leaner equivalent.)

Part 3 — model(): Two-Way State Shared With the Parent

// Child owns the interaction; parent owns the value — both stay in sync:
@Component({
  selector: 'app-visibility-toggle',
  template: `<button (click)="visible.update(v => !v)">
               {{ visible() ? 'Hide' : 'Show' }}
             </button>`,
})
export class VisibilityToggleComponent {
  visible = model(false);        // = input `visible` + output `visibleChange`, as ONE signal
}

<!-- parent — two-way binds, and can ALSO set it programmatically: -->
<app-visibility-toggle [(visible)]="showDetails" />
@if (showDetails) { <app-details /> }

When to use which:
  input()   parent → child data, child never writes it            (most props)
  output()  child → parent events, parent owns all state          (actions)
  model()   genuinely SHARED state both sides read AND write      (toggles, selected id,
                                                                   form-control-like values)

model() is the honest API for the "controlled component" pattern: the child manipulates the value through its UI, the parent binds it with [( )] (or reads/writes the halves separately), and both see one source of truth. Reserve it for state that is truly shared — overusing model() where a plain input would do reintroduces the two-way-everything data flow that made pre-Angular-2 apps untraceable. The classic use cases: toggles, tab indexes, selected items, custom form controls (though full form integration is better done via ControlValueAccessor).

Part 4 — Content Projection: Components With Holes

<!-- card.component.html — a component that WRAPS caller-provided content: -->
<section class="card">
  <header><ng-content select="[card-title]" /></header>   <!-- named slot -->
  <div class="body"><ng-content /></div>                    <!-- default slot -->
  <footer><ng-content select="[card-actions]" /></footer>  <!-- named slot -->
</section>

<!-- caller — projects arbitrary markup into the slots: -->
<app-card>
  <h3 card-title>Invoice #4211</h3>
  <p>Total: {{ total | currency:'EUR' }}</p>              <!-- default slot content -->
  <ng-container card-actions>
    <button (click)="pay()">Pay</button>
    <button (click)="dispute()">Dispute</button>
  </ng-container>
</app-card>

Inputs pass data; projection passes markup. When a component's job is structure — cards, dialogs, panels, layouts — accepting content beats accepting a dozen inputs trying to parameterize every possible interior. select matches CSS selectors (attributes as shown, or element/class selectors) to route content to named slots; unmatched content lands in the default <ng-content />. Two facts worth knowing before they bite: projected content's bindings evaluate in the caller's context (that pay() is the caller's method — which is exactly what makes projection composable), and ng-content doesn't create content — it repositions it; you can't loop or conditionally re-instantiate it (that requires template-passing with ng-template + NgTemplateOutlet, the advanced form used by libraries like tabs and tables).

Part 5 — Queries: viewChild & contentChild

import { Component, viewChild, contentChild, ElementRef, AfterViewInit } from '@angular/core';

@Component({
  selector: 'app-player',
  template: `
    <video #vid [src]="src()"></video>
    <button (click)="toggle()">⏯</button>
  `,
})
export class PlayerComponent {
  private vid = viewChild.required<ElementRef<HTMLVideoElement>>('vid');
  toggle() {
    const v = this.vid().nativeElement;
    v.paused ? v.play() : v.pause();
  }
}

// viewChild    → finds things in MY OWN template          (elements, child components)
// contentChild → finds things PROJECTED INTO me by caller (the tabs/accordion pattern)
// plural: viewChildren / contentChildren → a reactive list

// Classic contentChildren use — a tab GROUP reading its projected tabs:
@Component({ selector: 'app-tabs', template: `…` })
export class TabsComponent {
  tabs = contentChildren(TabComponent);     // all <app-tab>s the caller nested inside
  active = signal(0);
}

Queries give a component a handle on its own rendered pieces. viewChild targets the component's own template — DOM elements for imperative needs (focus, media, canvas, measuring) or child component instances to call their methods. contentChild(ren) targets what the caller projected in — the machinery behind every tabs/accordion/stepper API where a parent coordinates children it didn't create. The signal versions (viewChild() etc., v17.2+) return signals that populate when the view initializes and update as queries change — replacing the old @ViewChild-plus-ngAfterViewInit-timing dance. Discipline: queries are for orchestration and imperative DOM needs; if you're reaching into a child to read or write its state routinely, the missing design is an input, output, or shared service.

Part 6 — Deep Dive: Beyond Parent–Child, and Designing Component APIs

Sibling A ──?── Sibling B          Deeply nested grandchild ──?── distant ancestor

The wrong answers seniors reject on sight:
  ✗ output-chains: bubbling an event up 4 levels of pass-through outputs
  ✗ input-drilling: threading data down 4 levels of pass-through inputs
  ✗ viewChild-reaching: parent grabbing children to shove state into them

The right answer, 90% of the time — a SHARED SERVICE holding signal state:
@Injectable({ providedIn: 'root' })
export class CartService {
  private items = signal<CartItem[]>([]);
  readonly count = computed(() => this.items().length);
  readonly total = computed(() => this.items().reduce((s, i) => s + i.price, 0));
  add(item: CartItem) { this.items.update(list => [...list, item]); }
}
// Any component, any depth: private cart = inject(CartService);
// Header shows cart.count(), product page calls cart.add() — no chain between them.
// (Full pattern: services-di.html and state-management.html)

Inputs/outputs are a parent–child protocol. The moment two components aren't directly related — siblings, header vs deep page, dialog vs opener — threading state through the tree turns every intermediate component into a coupling point that must know about data it doesn't use. A shared injectable holding signals inverts that: state lives in one place, components at any depth read the signals they care about and call mutation methods. This is the first step of the state management story that culminates in State Patterns and NgRx.

Senior framing — component API design: treat a component's inputs/outputs/slots as a published API, because that's what they are: the compiler enforces them at every call site, and changing them is a breaking change for every consumer. The heuristics of good component APIs: minimal inputs (each one is a promise to support a configuration forever — "one more boolean input" is how god-components are born); events out, never callbacks in (passing functions down as inputs hides the control flow the template syntax exists to make visible); projection over configuration for structural variability (slots age better than showHeader/headerText/headerIcon... input piles); and smart/presentational separation — container components inject services and orchestrate; leaf components take inputs, emit outputs, and stay trivially testable. When an interviewer asks "how would you design a reusable X", this vocabulary — contract, slots, controlled state via model(), orchestration via queries — is the expected register.

Interview Questions

Q1 (Beginner): How do parent and child components communicate in Angular?

Downward: the parent binds data to the child's inputs ([product]="p" → product = input.required<Product>()). Upward: the child emits typed events through outputs (buy = output<number>(); this.buy.emit(id)) which the parent handles with (buy)="addToCart($event)". This keeps flow one-way and explicit: data down, events up, with the child reusable because it never knows who its parent is. For shared read-write state there's model() (two-way), and for unrelated components, a shared service.

Q2 (Beginner): What's the difference between input() and input.required(), and why prefer required?

input(default) is optional — parents may omit it and the default applies. input.required<T>() has no default and makes omitting the binding a compile error at every call site. Prefer required for data the component can't render meaningfully without (the product in a product card): it converts a whole class of "why is it undefined" runtime bugs into red squiggles, and documents the contract honestly.

Q3 (Mid): What does model() declare, when is it the right choice over input+output, and what's the risk of overuse?

model(initial) declares a writable signal that is simultaneously an input (x) and an output (xChange) — enabling [(x)]="value", with child-side set/update calls propagating to the parent. Right when both sides genuinely read and write one value: toggles, selection, form-control-like components. Overuse risk: it reintroduces pervasive two-way binding — many writers to shared state with no visible flow direction — which makes change origins untraceable; one-way input+output should remain the default, with model() as the deliberate exception.

Q4 (Mid): Explain multi-slot content projection and whose context projected bindings evaluate in — and why that matters.

A component places <ng-content select="[slot-name]" /> markers; caller content matching each selector is projected into that slot, unmatched content into the bare <ng-content />. Projected bindings evaluate in the caller's context — a projected button's (click)="pay()" calls the caller's method, not the card's. That's what makes wrapper components composable (the card needs zero knowledge of what it wraps), and it's also the gotcha: the wrapped component cannot directly hand data to projected content — for that you pass an ng-template and instantiate it with a context object (NgTemplateOutlet).

Q5 (Senior): viewChild vs contentChild — and describe a real component pattern that requires contentChildren.

viewChild queries the component's own template; contentChild(ren) queries what the caller projected into it. The canonical pattern: a tabs component — <app-tabs><app-tab title="A">…</app-tab><app-tab title="B">…</app-tab></app-tabs>. The group does tabs = contentChildren(TabComponent) to discover the tabs the caller declared, renders their titles as buttons, and toggles which tab body is visible. The caller gets a declarative API; the coordination logic lives in one place. Same architecture as accordions, steppers, and column-driven tables (where columns arrive as templates).

Q6 (Senior): A grandchild component needs to notify a distant ancestor, and a sibling needs the same data. A teammate proposes chaining outputs up and inputs across. Critique and propose the architecture.

Chaining makes every intermediate component a pass-through: each must declare, forward, and maintain bindings for data it doesn't use — N components coupled to a contract that belongs to two. It also scales quadratically as more distant consumers appear. Architecture: lift the state into a shared injectable — private writable signals, public computed read models, and mutation methods (cart.add(item)). Producer calls the method; ancestor and sibling read the computed signals; intermediates know nothing. Reserve inputs/outputs for genuine parent–child contracts. If state grows complex (many interacting updates, devtools/undo needs), the same shape graduates to a store (State Patterns / NgRx) without changing consumers.

1 (Easy): Build a UserAvatarComponent: required input user ({name, imageUrl}), optional size ('sm'|'md'|'lg', default 'md') mapped to a CSS class, and alt text from the name.
Solution
@Component({
  selector: 'app-user-avatar',
  template: `<img [src]="user().imageUrl" [alt]="user().name"
                  class="avatar" [class]="'avatar-' + size()">`,
  styles: `.avatar-sm { width: 24px } .avatar-md { width: 40px } .avatar-lg { width: 64px }`,
})
export class UserAvatarComponent {
  user = input.required<{ name: string; imageUrl: string }>();
  size = input<'sm' | 'md' | 'lg'>('md');
}
// <app-user-avatar [user]="author" size="lg" /> — omitting [user] fails the build.
2 (Easy): Add a removed output to a TagChipComponent that emits the tag string when its × button is clicked — and show why the chip itself must not delete anything.
Solution
@Component({
  selector: 'app-tag-chip',
  template: `<span class="chip">{{ tag() }}
               <button (click)="removed.emit(tag())" aria-label="Remove">×</button>
             </span>`,
})
export class TagChipComponent {
  tag = input.required<string>();
  removed = output<string>();
}
// Parent owns the list, so the parent deletes:
// <app-tag-chip [tag]="t" (removed)="tags.update(l => l.filter(x => x !== $event))" />
// The chip announcing-not-acting is what lets ANY parent reuse it — filters, editors,
// read-only views that ignore `removed` entirely.
3 (Medium): Build a StarRatingComponent usable as [(rating)]="score" with a max input (default 5), clicking star N sets the rating to N.
Solution
@Component({
  selector: 'app-star-rating',
  template: `
    @for (s of stars(); track s) {
      <button class="star" [class.filled]="s <= rating()"
              (click)="rating.set(s)" [attr.aria-label]="'Rate ' + s">★</button>
    }
  `,
})
export class StarRatingComponent {
  max = input(5);
  rating = model(0);                                        // two-way capable
  stars = computed(() => Array.from({ length: this.max() }, (_, i) => i + 1));
}
// <app-star-rating [(rating)]="score" [max]="10" />
// model() is right here: the child writes (clicks), the parent also reads/writes (reset,
// load saved score) — genuinely shared state.

5 (Hard): Build the tabs pattern: TabComponent (title input, projected body) + TabsComponent using contentChildren to render title buttons and show only the active tab.
Solution
@Component({ selector: 'app-tab', template: `
  <div [hidden]="!active()"><ng-content /></div>` })
export class TabComponent {
  title = input.required<string>();
  active = signal(false);            // set by the parent group, not by callers
}

@Component({
  selector: 'app-tabs',
  imports: [],
  template: `
    <nav>
      @for (tab of tabs(); track tab.title(); let i = $index) {
        <button [class.on]="i === index()" (click)="select(i)">{{ tab.title() }}</button>
      }
    </nav>
    <ng-content />   <!-- the projected app-tab elements live here -->
  `,
})
export class TabsComponent {
  tabs = contentChildren(TabComponent);       // discovers caller-projected tabs
  index = signal(0);
  constructor() {
    effect(() => this.tabs().forEach((t, i) => t.active.set(i === this.index())));
  }
  select(i: number) { this.index.set(i); }
}

<!-- caller — fully declarative: -->
<app-tabs>
  <app-tab title="Details"><app-details /></app-tab>
  <app-tab title="Reviews"><app-reviews /></app-tab>
</app-tabs>
<!-- [hidden] (not @if) keeps tab state alive when switching — deliberate choice
     from the directives lesson's identity discussion. -->
6 (Hard): A product page's "Add to cart" (deep in the tree) must update a header cart badge (different branch). Implement the shared-service solution with signals, and state why outputs can't reasonably do this.
Solution
@Injectable({ providedIn: 'root' })
export class CartService {
  private items = signal<CartItem[]>([]);
  readonly count = computed(() => this.items().length);
  readonly total = computed(() => this.items().reduce((s, i) => s + i.price, 0));
  add(item: CartItem)    { this.items.update(l => [...l, item]); }
  remove(id: number)     { this.items.update(l => l.filter(i => i.id !== id)); }
}

// Deep product component — no knowledge of the header:
export class ProductDetailComponent {
  private cart = inject(CartService);
  addToCart() { this.cart.add(this.toCartItem()); }
}

// Header badge — no knowledge of product pages:
@Component({ selector: 'app-cart-badge',
  template: `🛒 @if (cart.count() > 0) { <span class="badge">{{ cart.count() }}</span> }` })
export class CartBadgeComponent {
  cart = inject(CartService);
}

// Outputs can't do this reasonably: the event would have to bubble through every
// component up to a common ancestor, then flow back down as inputs — every component
// on both paths becomes a pass-through coupled to cart concerns. The service gives
// one source of truth at depth-independent cost, and signals make the badge update
// automatically, precisely, with no subscriptions to manage.

🎯 Key Takeaways

  • Inputs are compile-checked parameters — input.required for must-haves, transform for normalization; as signals they compose with computed()/effect().
  • Outputs are typed events named as announcements (itemSelected), keeping children policy-free and reusable; model() = input + output as one writable signal, reserved for genuinely shared state.
  • Content projection passes markup: multi-slot via select, bindings run in the caller's context, and ng-content repositions — it can't loop or re-instantiate.
  • Queries (viewChild/contentChildren) are for orchestration and imperative DOM — the tabs/accordion pattern — not a side-channel for state.
  • Non-parent-child communication = shared service with private writable signals, public computed reads, and mutation methods. Never drill, never chain.
  • Component APIs are published contracts: minimal inputs, events out (not callbacks in), projection over configuration, smart/presentational split.
