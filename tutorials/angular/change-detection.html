Change Detection: Zones to Zoneless Components & Reactivity

What you'll master: the question every framework must answer — when and where to re-render — and Angular's three eras of answering it: Zone.js check-everything, OnPush check-less, and signal-driven zoneless. You'll learn what Zone.js actually does, the OnPush contract and its classic bugs, how signals change the game, and the production migration path. Prerequisites: Signals Deep Dive and Data Binding (Part 6 — bindings compile to guarded update instructions). This is a top-three Angular interview topic.

Part 1 — The Problem, and Zone.js's Blunt Answer

Rendering = keeping DOM in sync with state. The hard part isn't HOW (bindings
compile to update instructions) — it's WHEN. What tells Angular "state may have changed"?

Zone.js's answer: PATCH EVERY ASYNC API in the browser.
  setTimeout / setInterval / Promise.then / addEventListener / XHR / fetch …
  → each becomes a wrapped version that runs the original, then tells Angular:
    "an async task just finished — something MIGHT have changed"

Then Angular runs CHANGE DETECTION: walk the ENTIRE component tree, top-down,
re-evaluating every binding and patching the DOM where values differ.

  click on ONE button
    → zone notifies
    → EVERY component's bindings re-checked  (header, sidebar, all 500 rows…)
    → the one changed text node is patched

Zone.js is a monkey-patching library: at app start it wraps the browser's async entry points so that any callback completing signals Angular to check. The genius is what it made unnecessary — you write this.count++ in a click handler and the UI just updates, no setState, no subscriptions — Angular literally could not miss a change. The cost is the bluntness: "something might have changed somewhere" forces "check everything". Dirty-checking a binding is fast (compare a value, usually skip), so small apps never notice; big apps — thousands of components × dozens of bindings × every keystroke/timer/HTTP tick — absolutely do. Everything else in this lesson is progressively sharpening that answer.

Part 2 — The Mechanics: What a CD Pass Actually Does

For each component, top-down (parents before children):
  1. run its template's UPDATE INSTRUCTIONS (data-binding.html Part 6):
     each binding: evaluate expression → compare to previous value → patch DOM if different
  2. descend into children, repeat.

Key properties:
  UNIDIRECTIONAL — one top-down sweep per pass; parents settle before children.
     (mutating a parent's bound state from a child mid-pass ⇒
      ExpressionChangedAfterItHasBeenChecked — lifecycle.html)
  VALUE-DIFFING — Angular compares expression VALUES, not DOM or virtual trees.
     The DOM is written only for actual changes.
  PER-VIEW — each component's compiled template is checked independently;
     "skip this component" is therefore a natural optimization unit → Part 3.

Demystified, change detection is a tree walk running tiny compiled functions — no virtual DOM, no diffing of markup, just "evaluate, compare, maybe patch" per binding. That's why binding hygiene (cheap expressions, memoized pipes/computeds) is Angular performance tuning, and why the optimization story is about skipping subtrees: if Angular could know a component's inputs and internal state didn't change, it could skip its whole subtree's check. That knowledge is exactly what OnPush declares and signals prove.

Part 3 — OnPush: the Contract That Lets Angular Skip

@Component({
  selector: 'app-product-card',
  changeDetection: ChangeDetectionStrategy.OnPush,   // "I promise to be predictable"
  template: `…`,
})

An OnPush component is CHECKED only when:
  1. an INPUT changes — by REFERENCE (Object.is on the bound value)
  2. an event handler fires INSIDE it (or its subtree)
  3. the async pipe in its template receives an emission
  4. a signal it reads in its template changes            (the modern trigger)
  5. you explicitly ask: ChangeDetectorRef.markForCheck()

Otherwise: the component AND ITS ENTIRE SUBTREE are skipped in the pass.

The contract's price — the classic OnPush bug:
  user.name = 'New';          // mutation: same reference
  → input didn't "change" (reference-equal) → card never re-checks → stale UI
  fix: this.user = { ...this.user, name: 'New' };   // new reference = visible change

OnPush converts the reference-equality discipline you've now met three times (pure pipes, signal updates, immutable state) into a rendering contract: "my output depends only on my inputs (by reference), my own events, and my subscribed streams — if none changed, skip me." On a wide tree, skipping whole branches turns the per-pass cost from "everything" to "the active path". The bugs are always contract violations: someone mutated a shared object (UI stale — the fix is immutable updates, or as a band-aid markForCheck()), or state arrived via a side-channel the triggers can't see (a plain service field written by a timer — the fix is making it a signal or observable). Historically OnPush was "advanced"; in signal-era code it's simply the correct default, because signals satisfy the contract automatically.

Part 4 — Signals Change the Question

Zone.js asks:  "did SOME async task finish?"          → check everything
OnPush asks:   "did THIS component's declared triggers fire?" → skip clean subtrees
Signals ask:   "WHICH signals changed, and which VIEWS read them?" → check exactly those

@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<h2>{{ title() }}</h2>
             <p>{{ cart.total() }} €</p>`,
})
export class HeaderComponent {
  title = signal('Shop');
  cart = inject(CartService);
}
// The template READ title() and cart.total() → those views are registered as
// dependents in the signal graph (signals.html Part 6).
// cart.total changes → EXACTLY this view is marked dirty → next pass checks the
// marked path only. No guessing, no tree-wide sweep, no zone notification needed.

Signals give change detection the missing information: because templates read signals, the framework knows precisely which views depend on which state — the same dependency graph that drives computed memoization drives rendering. A signal write marks its dependent views dirty (push), the next pass checks just the marked paths (pull) — the two-phase model from the Signals lesson, now applied to the DOM. This is what makes the last era possible: if signal reads tell Angular what to update and events/signals tell it when, the zone's "something somewhere finished" broadcast is redundant.

Part 5 — Zoneless: Removing Zone.js

// app.config.ts (stable API in v20.2+; provideExperimentalZonelessChangeDetection in 18/19):
export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    // + remove 'zone.js' from angular.json "polyfills"
  ],
};

Without the zone, Angular schedules CD from FIRST-PARTY notifications only:
  ✓ signal writes (set/update) read by templates
  ✓ template event handlers (click)="…"
  ✓ async pipe emissions / markForCheck()
  ✓ input changes, host listeners

What STOPS updating the UI (the migration checklist):
  ✗ setTimeout(() => this.plainField = x)        — no signal, no notification
  ✗ third-party callback mutating component state — same
  ✗ code that leaned on "zone catches everything"
  → fix: make the state a SIGNAL (or emit via observable + async pipe/toSignal)

Wins: no zone patching (startup + less magic), no CD storms from stray timers/
      analytics scripts, cleaner stack traces, smaller bundle, SSR-friendlier.

Zoneless flips the default from "assume anything may have changed" to "update when told through a reactive channel". For signal-first codebases the flip is nearly free — templates already read signals, events already notify — while legacy code that mutates plain fields from async callbacks surfaces immediately as "UI stopped updating" (which is the migration working: every such spot was an undeclared dependency). The practical path: OnPush + signals throughout → run the app zoneless in dev → fix the silent spots by converting state to signals → ship, and delete the zone.js polyfill. New Angular projects (v20+) offer zoneless from the start.

Part 6 — Deep Dive: Debugging CD & the Performance Toolbox

Diagnosing "my UI doesn't update" — by era:
  Zone app:      state changed outside the zone? (rare: custom schedulers, workers,
                 runOutsideAngular) → NgZone.run() or restructure
  OnPush app:    mutation instead of new reference? side-channel state? →
                 immutable update / signal / markForCheck
  Zoneless app:  plain-field mutation from async code → make it a signal

Diagnosing "my app is slow" (CD flavor):
  Angular DevTools profiler → which components check, how often, how long
  Typical villains and fixes:
    checks everywhere per keystroke   → OnPush/signals to prune the tree
    expensive bound expressions       → computed()/pure pipes (memoize)
    giant lists fully checked         → @for track + virtualization (performance.html)
    zone storms from timers/analytics → runOutsideAngular, or go zoneless

Escape hatches (know them, use them rarely):
  NgZone.runOutsideAngular(fn)  run noisy work (rAF loops, scroll math) without CD
  ChangeDetectorRef.detach()    take a subtree fully manual (+ detectChanges() on demand)
  markForCheck()                schedule this OnPush view for the next pass

Change-detection debugging is contract-checking: identify which era's contract the code assumed, find the state change that bypassed it. The performance toolbox is likewise layered: prune with OnPush/signals, memoize with computed/pipes, bound the DOM with track/virtualization, and silence noisy non-UI work with runOutsideAngular (a zone-era tool that zoneless makes unnecessary). detach()/detectChanges() exist for extreme cases (high-frequency dashboards hand-controlling their refresh) — treat manual CD as the assembly language of Angular rendering: powerful, rarely justified.

Senior framing: the interview arc worth being able to narrate: Angular started with maximum convenience (zone catches everything, developers think about nothing) and has been trading toward maximum precision (signals declare everything, framework checks only what changed) — with OnPush as the decade-long bridge that made apps declare their dependencies by contract before the primitives existed to declare them by construction. The honest trade-off: zoneless-signal Angular asks slightly more of developers (state must live in reactive containers) and repays it with predictability — updates you can trace ("who wrote this signal?") instead of passes you can only profile. When asked "React vs Angular rendering", the crisp answer: React re-runs component functions and diffs a virtual DOM; Angular re-runs compiled binding checks against real DOM — and with signals, skips even the checking for anything that provably didn't change.

Interview Questions

Q1 (Beginner): What is change detection, and what role does Zone.js play?

Change detection is Angular's process of syncing component state to the DOM: walking the component tree and re-running each template's compiled binding checks, patching where values changed. Zone.js answers the when: it monkey-patches browser async APIs (timers, events, promises, XHR) so that completion of any async task notifies Angular to run a pass. That's why plain this.count++ in a click handler updates the UI with no further ceremony — and why, by default, one event triggers checks across the whole tree.

Q2 (Beginner): What does ChangeDetectionStrategy.OnPush change, and when does an OnPush component get checked?

Default strategy checks every component on every pass. OnPush skips the component (and its subtree) unless one of its triggers fired: an input changed by reference, an event fired inside it, its template's async pipe emitted, a signal it reads changed, or markForCheck() was called. It's a promise of predictability in exchange for being skipped — and it's why OnPush code must use immutable updates: mutation keeps the old reference and is invisible to the trigger list.

Q3 (Mid): An OnPush child bound to [user]="user" doesn't update after this.user.name = 'New' in the parent. Explain and give the two fixes with their trade-off.

OnPush input-change detection compares references; mutating user leaves the reference identical, so the child is skipped. Proper fix: immutable update — this.user = { ...this.user, name: 'New' } — the new reference trips the trigger, and the discipline pays off everywhere else that relies on reference equality (pure pipes, signals, memoized view models). Band-aid: inject ChangeDetectorRef in the child (or push through an observable/signal) and markForCheck() — works, but treats a symptom; each such call marks an undeclared data flow that will bite again elsewhere.

Q4 (Mid): How do signals interact with change detection, and why do they make OnPush "automatic"?

Template signal reads register the view as a dependent in the signal graph; a write marks exactly the dependent views dirty and schedules a pass that checks only marked paths. OnPush's contract — "check me only when my declared triggers fire" — required developers to route all state through inputs/streams; signals satisfy it by construction, since any state the template uses is a tracked read: the framework never needs to guess or broadly sweep, and dependency declaration is a side effect of simply reading state. That's also the foundation of zoneless: with signals + events as notification, the zone's blanket "something finished" becomes unnecessary.

Q5 (Senior): Walk through migrating a large zone-based app to zoneless: prerequisites, process, and what breaks.

Prerequisites: OnPush + signal adoption for state the templates read (async pipe/toSignal for streams), removal of "zone catches it" habits. Process: enable provideZonelessChangeDetection() in dev builds; exercise the app; every UI area that freezes marks a spot where state changes via plain fields from async callbacks (timers, sockets, third-party SDKs) — convert each to a signal write or observable consumed reactively; audit NgZone uses (runOutsideAngular becomes moot; zone.run spots are undeclared-notification smells); then drop the zone.js polyfill. What breaks by design: libraries relying on zone patching for their updates — wrap their callbacks to write signals. Benefits realized: no global patching, no CD storms from stray async, cleaner traces, better SSR; risk is bounded because breakage is loud (frozen UI in dev), not corrupting.

Q6 (Senior): A dashboard re-renders visibly on every websocket tick (5/sec) even though only one widget's number changes. Diagnose across all three eras and give the modern fix.

Zone era: each tick completes an async task → tree-wide CD; with default strategy every widget's bindings re-check 5×/sec — jank scales with dashboard size. OnPush era: if ticks mutate a shared object or bypass inputs, either nothing updates (violation) or someone added detectChanges() per tick — still re-checking a subtree wholesale. Modern fix: the socket feeds a signal (toSignal on the stream or set() in the handler); the one widget's template reads it; signal-driven CD marks exactly that view — one binding check and one DOM patch per tick, zoneless removing even the global scheduling noise. If the widget renders a large list per tick, add track and virtualization so the patch stays proportional to what changed. The narrative — blunt broadcast → declared contract → precise graph — is the whole lesson in one bug.

🏋️ Exercises

1 (Easy): A zone-era component does setTimeout(() => this.message = 'Done!', 1000) with {{ message }} in the template — and it works. Explain precisely why, naming each step.
Solution
1. Zone.js patched setTimeout at startup: the callback is wrapped.
2. After 1000ms the wrapped callback runs your code (message = 'Done!'),
   then notifies Angular: "an async task completed".
3. Angular runs a change-detection pass over the tree.
4. This component's compiled update instruction re-evaluates `message`,
   sees 'Done!' ≠ previous value, patches the text node.
// Remove zone.js (zoneless) and the same code silently does nothing —
// a plain field write notifies nobody. signal + message.set('Done!') fixes it.
2 (Easy): Convert this component to OnPush and fix the update that breaks under it: 
@Component({ template: `<app-list [items]="items" /><button (click)="add()">+</button>` })
export class PageComponent {
  items = [{ id: 1 }];
  add() { this.items.push({ id: Date.now() }); }
}
Solution
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<app-list [items]="items" /><button (click)="add()">+</button>`,
})
export class PageComponent {
  items = [{ id: 1 }];
  add() { this.items = [...this.items, { id: Date.now() }]; }   // NEW reference
}
// push() mutates: if app-list is OnPush, its [items] reference never changes → skipped →
// the new row never appears. The spread creates a new array, tripping the input trigger.
// (Signal version: items = signal([...]); add() { this.items.update(l => [...l, …]) })
3 (Medium): Using ChangeDetectorRef, make a legacy OnPush component correctly display messages that a service pushes via a plain callback API (onMessage(cb)) — then show the modern signal version that needs no CDR.
Solution
// Legacy bridge — a side-channel (callback) must explicitly mark the view:
export class InboxComponent implements OnInit {
  messages: string[] = [];
  private cdr = inject(ChangeDetectorRef);
  private svc = inject(MessageService);
  ngOnInit() {
    this.svc.onMessage(msg => {
      this.messages = [...this.messages, msg];
      this.cdr.markForCheck();          // schedule THIS OnPush view for the next pass
    });
  }
}

// Modern — route the side channel into the graph once, in the service:
@Injectable({ providedIn: 'root' })
export class MessageService {
  private _messages = signal<string[]>([]);
  readonly messages = this._messages.asReadonly();
  constructor() { legacyApi.onMessage(m => this._messages.update(l => [...l, m])); }
}
// Component: messages = inject(MessageService).messages;  template: @for (m of messages()…)
// No CDR anywhere — the signal read IS the dependency declaration.
4 (Medium): A mousemove-driven "cursor spotlight" component causes app-wide CD 60×/sec in a zone app. Fix it with runOutsideAngular while still updating its own DOM.
Solution
export class SpotlightComponent {
  private el = inject(ElementRef<HTMLElement>);
  private zone = inject(NgZone);

  constructor() {
    this.zone.runOutsideAngular(() => {          // listener registered OUTSIDE the zone:
      document.addEventListener('mousemove', (e) => {   // its firing triggers NO CD
        // direct DOM write — no bindings involved, no pass needed:
        this.el.nativeElement.style.setProperty('--x', e.clientX + 'px');
        this.el.nativeElement.style.setProperty('--y', e.clientY + 'px');
      });
    });
  }
}
// CSS does the rest: background: radial-gradient(at var(--x) var(--y), …)
// Pattern: high-frequency, render-only work runs outside the zone and touches DOM/CSS
// directly; re-enter with zone.run() ONLY if a real state change must reach bindings.
// (In a zoneless app this problem disappears — listeners don't trigger passes at all.)
5 (Hard): Prepare this component for a zoneless app — find every update that would silently stop working and modernize it: 
export class StatusComponent implements OnInit {
  status = 'connecting'; lastPing = 0; latency = 0;
  ngOnInit() {
    this.sock.onopen = () => this.status = 'online';
    this.sock.onmessage = (e) => { this.lastPing = Date.now(); this.latency = e.data.ms; };
    setInterval(() => { if (Date.now() - this.lastPing > 10_000) this.status = 'stale'; }, 5000);
  }
}
Solution
// All three writes are plain-field mutations from async callbacks — in a zoneless app
// none of them notify anyone: the UI freezes at 'connecting'.
export class StatusComponent {
  status = signal<'connecting' | 'online' | 'stale'>('connecting');
  lastPing = signal(0);
  latency = signal(0);
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    this.sock.onopen = () => this.status.set('online');            // signal write → notifies
    this.sock.onmessage = (e) => {
      this.lastPing.set(Date.now());
      this.latency.set(e.data.ms);
    };
    const id = setInterval(() => {
      if (Date.now() - this.lastPing() > 10_000) this.status.set('stale');
    }, 5000);
    this.destroyRef.onDestroy(() => clearInterval(id));            // and no leak (lifecycle.html)
  }
}
// Template reads status()/latency() — signal-driven CD updates exactly this view per event.
// Note the interval still RUNS fine zoneless; it just must communicate via signals.
6 (Hard): Your team inherited an app where 14 components call detectChanges() in various callbacks "because updates were missing". Write the code-review verdict: what each call signifies, the risk of the pattern, and the refactor plan.
Solution
// Verdict:
// Each detectChanges() marks a place where state reaches the template through a channel
// change detection can't see (plain fields written by callbacks/SDKs) — 14 undeclared
// data flows. detectChanges() also runs the subtree SYNCHRONOUSLY, mid-callback:
//   - re-entrancy hazards (CD during another CD → ExpressionChanged errors)
//   - performance: full subtree checks at callback frequency
//   - fragility: remove one call and some unrelated screen goes stale — nobody knows why.
//
// Refactor plan (incremental, low-risk):
// 1. Inventory the 14 sites; for each, identify the STATE being changed (not the call).
// 2. Move that state into signals at its source — usually a service wrapping the SDK:
//      sdk.on(evt => this._state.update(...))   → components just read state()
//    (streams → toSignal; keep RxJS where it's genuinely event-pipeline logic).
// 3. Delete the detectChanges() call in the same PR that converts each site — the test
//    is "UI still updates with the call gone".
// 4. When all 14 are gone: components to OnPush, then trial zoneless in dev — any
//    remaining freeze is a site the inventory missed.
// End state: updates traceable by "who writes this signal", checks proportional to
// actual change — and zero manual CD calls to maintain.

🎯 Key Takeaways

  • Change detection = when + where to re-run compiled binding checks. Zone.js answers "when" by patching all async APIs — convenient, but one event checks the whole tree.
  • Passes are top-down, value-diffing, per-view — no virtual DOM; binding hygiene and subtree-skipping are the performance levers.
  • OnPush is a skip-contract: re-check only on reference-changed inputs, internal events, async-pipe emissions, signal reads, or markForCheck() — and it's why immutable updates are non-negotiable.
  • Signals make dependencies explicit per-view: writes mark exactly the dependent views dirty — OnPush satisfied by construction, precision instead of guessing.
  • Zoneless (provideZonelessChangeDetection()) drops the zone: UI updates flow only from signals/events/streams — migration surfaces every undeclared dependency as a loud, fixable freeze.
  • Debugging is contract-checking (which era's contract did the code assume?); markForCheck/detectChanges sprinkles are symptoms — route state through signals instead.
