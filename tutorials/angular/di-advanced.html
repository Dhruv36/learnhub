Advanced DI: Providers & Tokens Services & DI

What you'll master: the full provider vocabulary (useClass, useValue, useFactory, useExisting), InjectionToken for non-class dependencies, multi-providers, the injector hierarchy in detail (with resolution modifiers), and the patterns these unlock — config injection, plugin systems, per-feature scoping. Prerequisites: Services & DI. This is the machinery underneath every provide*() function you call in app.config.ts — after this lesson, none of it is magic.

Part 1 — Providers: a Token and a Recipe

// EVERY provider is: { provide: TOKEN, use___: RECIPE } — "when asked for TOKEN, make it THIS way"

{ provide: OrderApi,  useClass: OrderApi }          // the default recipe (shorthand: OrderApi)
{ provide: OrderApi,  useClass: MockOrderApi }      // swap the concrete — consumers unchanged
{ provide: API_URL,   useValue: 'https://api.example.com' }        // a ready-made value
{ provide: Logger,    useExisting: ConsoleLogger }  // ALIAS: same instance, second token
{ provide: OrderApi,  useFactory: (auth: AuthService) =>           // computed at injection time
    auth.isAdmin() ? new AdminOrderApi() : new OrderApi(),
  deps: [AuthService] }
// modern factory style — inject() inside, no deps array:
{ provide: OrderApi, useFactory: () => inject(AuthService).isAdmin() ? … : … }

// @Injectable({providedIn:'root'}) is just sugar for registering
// { provide: TheClass, useClass: TheClass } in the root injector — tree-shakably.

Demystified, the injector is a Map from tokens to recipes, consulted on each inject(TOKEN): found → run recipe once, cache instance, return it. Everything else in DI is variations on this. useClass substitutes implementations; useValue injects data; useFactory makes construction conditional (environment, platform, feature flags); useExisting aliases one token to another's instance (useful for exposing a narrow interface to an existing service, or migrating token names without duplicating state). The provide*() functions you've been calling — provideRouter, provideHttpClient — are just functions returning arrays of these objects.

Part 2 — InjectionToken: Injecting Things That Aren't Classes

// Problem: tokens must exist at RUNTIME. Interfaces and primitives don't.
// inject(string) ✗   inject(AppConfig-interface) ✗

import { InjectionToken, inject } from '@angular/core';

export interface AppConfig { apiUrl: string; retries: number; featureFlags: string[]; }

export const APP_CONFIG = new InjectionToken<AppConfig>('app.config');   // typed token

// provide it (app.config.ts):
providers: [
  { provide: APP_CONFIG, useValue: { apiUrl: '/api', retries: 3, featureFlags: ['v2'] } },
]

// consume it — fully typed:
export class ApiService {
  private config = inject(APP_CONFIG);        // AppConfig — TS knows the shape
  private base = this.config.apiUrl;
}

// Token with a DEFAULT (no provider needed unless overriding):
export const PAGE_SIZE = new InjectionToken<number>('page.size', {
  providedIn: 'root',
  factory: () => 25,
});

InjectionToken<T> creates a unique runtime token carrying a compile-time type — the bridge between TypeScript's erased types and DI's runtime resolution. Use it for configuration objects, primitives (URLs, flags, sizes), and function-shaped dependencies. The description string is only for error messages; uniqueness comes from object identity — which is why tokens live in shared files both provider and consumer import (two new InjectionToken('same-name') calls are different tokens, a classic multi-bundle bug). The factory-default form makes tokens self-providing with override capability — the cleanest shape for tunable constants.

Part 3 — Multi-Providers: the Plugin Pattern

export const VALIDATION_RULE = new InjectionToken<Rule>('validation.rule');

providers: [
  { provide: VALIDATION_RULE, useClass: RequiredFieldsRule, multi: true },
  { provide: VALIDATION_RULE, useClass: PriceRangeRule,     multi: true },
  { provide: VALIDATION_RULE, useClass: SkuFormatRule,      multi: true },
]

export class ValidationService {
  private rules = inject(VALIDATION_RULE) as Rule[];    // ALL of them, as an array
  validate(order: Order) { return this.rules.flatMap(r => r.check(order)); }
}

// Angular's own APIs are built on this:
//   HTTP_INTERCEPTORS (http.html)  — every interceptor is a multi-provider entry
//   NG_VALIDATORS (form-validation.html), APP_INITIALIZER, route guards' data
// The point: CONTRIBUTORS register themselves; the CONSUMER iterates whatever showed up.
// Adding a rule = adding a provider line — the consumer never changes. (Open/closed, via DI.)

multi: true flips a token from "one recipe wins" to "collect all recipes into an array". That single flag is Angular's plugin architecture: features contribute interceptors/validators/initializers from wherever they're defined, and the consuming machinery iterates the array with no registry code, no imports of contributors, no central list to maintain. When you design an extensible subsystem (validation pipelines, export formats, dashboard widgets), reaching for a multi-token is the framework-idiomatic move — it's the strategy pattern where the lookup table assembles itself.

Part 4 — The Injector Hierarchy, Properly

                 ROOT injector            providedIn:'root', app.config providers
                    ↑
          ROUTE injectors (lazy features)  route-level `providers: []` (guards-lazy.html)
                    ↑
        COMPONENT/DIRECTIVE injectors      @Component providers / viewProviders

Resolution: start at the requesting component's injector, walk UP until found;
not found at root → NullInjectorError: No provider for X!

RESOLUTION MODIFIERS — tune the walk:
  inject(X, { optional: true })    null instead of throwing        (soft dependencies)
  inject(X, { self: true })        ONLY my own injector            (require local provider)
  inject(X, { skipSelf: true })    start from my PARENT            (get the outer instance)
  inject(X, { host: true })        stop at my host component       (directive boundaries)

// skipSelf's classic use — nesting-aware components (menus, form groups):
constructor() {
  const parentMenu = inject(MenuComponent, { optional: true, skipSelf: true });
  this.depth = parentMenu ? parentMenu.depth + 1 : 0;    // "am I nested inside another me?"
}

The hierarchy makes provider placement a scoping tool: root = app singleton; route providers = one instance per lazy feature (alive as long as the feature is loaded); component providers = one per component instance (the wizard-state case from last lesson). Two subtleties worth owning: lazy route injectors mean a "singleton" provided in a lazy feature is singleton within that feature — root-provide anything meant to be truly global; and viewProviders (vs providers) hides a component's provider from projected content — the tool for keeping internal services out of reach of ng-content children. The modifiers turn resolution into a precision instrument — skipSelf for parent-instance lookups, self + optional for "use local if configured", host for directive/host contracts.

Part 5 — Real Patterns Built From These Pieces

// 1. ENVIRONMENT/PLATFORM WIRING (SSR-safe storage from services-di.html, completed):
{ provide: StorageService,
  useFactory: () => isPlatformBrowser(inject(PLATFORM_ID))
      ? new LocalStorageService() : new MemoryStorageService() }

// 2. RUNTIME CONFIG loaded before startup (config from /assets/config.json, not build-time):
export const appConfig: ApplicationConfig = {
  providers: [
    provideAppInitializer(() => {
      const cfg = inject(ConfigService);
      return cfg.load();                       // Promise — app WAITS for it before bootstrap
    }),
  ],
};
// then APP_CONFIG's factory reads ConfigService's loaded data.

// 3. NARROW INTERFACES over fat services (useExisting — interface segregation):
abstract class ReadonlyCart { abstract items: Signal<CartItem[]>; abstract total: Signal<number>; }
{ provide: ReadonlyCart, useExisting: CartService }
// display components inject ReadonlyCart — they CANNOT call add/remove; same instance.

// 4. TESTING is just providers (testing.html):
TestBed.configureTestingModule({
  providers: [
    { provide: APP_CONFIG, useValue: testConfig },
    { provide: OrderApi,  useValue: fakeApi },
  ],
});

These four patterns cover most advanced-DI needs in real apps: platform/environment selection moved entirely into wiring; startup-time async configuration via app initializers (the standard answer to "our API URL comes from the server, not the build"); interface segregation with useExisting so read-only consumers get a capability-limited token onto the same state; and the recognition that TestBed configuration is nothing but another provider list — the same substitution mechanism, aimed at tests. Notice the theme: decisions move out of consuming code and into composition — classes stay simple, wiring gets expressive.

Part 6 — Deep Dive: How provide* Functions Compose, and DI Discipline

// What provideHttpClient(withInterceptors([auth])) RETURNS is... providers:
export function provideMyFeature(config: FeatureConfig): EnvironmentProviders {
  return makeEnvironmentProviders([
    { provide: FEATURE_CONFIG, useValue: config },
    FeatureService,
    { provide: VALIDATION_RULE, useClass: FeatureRule, multi: true },
  ]);
}
// app.config.ts:  providers: [provideMyFeature({ mode: 'pro' })]
// Your libraries should ship the same shape: one function, options as arguments,
// providers as the return value. EnvironmentProviders (vs Provider[]) marks providers
// that belong ONLY at app/route level — the compiler stops component-level misuse.

DI discipline checklist (the senior review lens):
  ✓ token per CONCEPT, not per class you happen to have    (contracts first)
  ✓ providers placed at the NARROWEST scope that's correct  (root by default, narrower deliberately)
  ✓ factories THIN — construction logic only, never business logic
  ✗ service locator style: injecting Injector and .get()-ing things dynamically
    (hides dependencies from the type system — the anti-pattern DI exists to prevent)
  ✗ provider soup: overriding the same token at many levels — resolution becomes
    position-dependent and unreviewable

The provide*() convention is the standalone era's packaging story: features export one composable function returning providers, options flow in as typed arguments (replacing the old forRoot/forChild module dance). Writing your own for shared libraries makes consumers' app configs read declaratively. The discipline list is what separates powerful DI use from DI abuse — the summary heuristic: if you can't tell what a class depends on by reading its inject() lines, DI is being used to hide structure instead of reveal it.

Senior framing: advanced DI questions in interviews usually probe whether you can place mechanisms, not recite them. Strong signals: choosing InjectionToken-with-factory-default over a config service for constants; multi-providers when asked to design "pluggable" anything; useExisting for interface narrowing (most candidates only know useClass/useValue); knowing lazy-route injectors create per-feature scope; and — most telling — knowing when to stop: a config object passed as an input beats a token when only one component tree needs it, and plain providedIn: 'root' classes cover 90% of real apps. The engineer who deploys the whole arsenal everywhere is as suspect as the one who knows none of it.

Common Mistakes

MistakeWhy it bitesFix

Forgetting multi: true on one contributorThat provider replaces the array — all other contributions vanish silentlyEvery entry for a multi token carries multi: true; lint/review for it
Business logic inside useFactoryUntestable, hidden in wiring, runs once at construction — surprises when "logic" doesn't re-runFactories construct; logic lives in the service (re-runnable, testable)

Interview Questions

Q1 (Beginner): Name the four provider recipes and when each is right.

useClass: instantiate a class — the default; swap implementations per environment. useValue: hand over a ready value — config objects, constants, test doubles. useFactory: compute the instance at injection time — conditional construction on platform/flags/other services. useExisting: alias — resolve this token to another token's (same) instance — interface narrowing and token migrations. All are entries of the same shape: when TOKEN is requested, produce the dependency THIS way.

Q2 (Beginner): Why does injecting an interface or a string require InjectionToken?

The injector resolves by runtime token identity. TypeScript interfaces are erased at compile time — nothing exists at runtime to be the key — and primitives like a URL string have no unique identity to key on. new InjectionToken<T>('desc') creates a unique runtime object carrying the type T for compile-time safety: provide with { provide: TOKEN, useValue }, consume with inject(TOKEN), fully typed at both ends.

Q3 (Mid): How do multi-providers work, and which Angular features are built on them? Design one use of your own.

multi: true makes a token accumulate: every provider entry contributes to an array, and inject(TOKEN) returns all contributions. Angular builds interceptors (HTTP_INTERCEPTORS/functional equivalents), form validators (NG_VALIDATORS), and app initializers on it. Own design example: an export subsystem — EXPORT_FORMAT token; PDF/CSV/XLSX features each register { provide: EXPORT_FORMAT, useClass: …, multi: true }; the export menu injects the array and renders whatever registered. Adding a format touches zero existing code — open/closed achieved through wiring.

Q4 (Mid): Walk through injector resolution for a component, including what optional, self, skipSelf change — and give skipSelf's classic use.

Resolution starts at the requesting component/directive's own injector and walks parent component injectors → route injectors (for lazy features) → root; first provider found wins; nothing found → NullInjectorError. optional: true returns null instead of throwing (soft dependencies); self: true checks only the local injector (assert a local provider exists); skipSelf: true starts from the parent (skip your own provider). Classic skipSelf: recursive/nesting-aware components — a menu that injects MenuComponent with { skipSelf: true, optional: true } to detect its parent menu and compute nesting depth, or a form group finding its containing group.

Q5 (Senior): Your app needs config (API URL, flags) fetched from the server before anything renders. Design it.

App-initializer pattern: a root ConfigService with a load() returning a Promise that fetches /assets/config.json (or an endpoint) and stores the result in a signal; register provideAppInitializer(() => inject(ConfigService).load()) — bootstrap waits for the promise, so by first render every consumer sees loaded config. Expose values via typed InjectionTokens whose factories read the service (API_URL = new InjectionToken('api.url', { factory: () => inject(ConfigService).value().apiUrl })) so consumers depend on narrow tokens, not the loader. Trade-offs to name: startup now blocks on that request (add a timeout/fallback config; keep the payload tiny), and build-time environment.ts remains right for truly-static values — runtime config is for what ops must change without a rebuild.

Q6 (Senior): A teammate "fixed" a bug by injecting Injector and calling injector.get(ServiceX) inside a method. Review it.

That's the service-locator anti-pattern: the dependency on ServiceX vanishes from the class's declared surface — types, tests, and readers no longer see it; resolution errors move from construction (loud, immediate) to whenever the method happens to run; and doubles in tests require knowing internals. Legitimate uses of dynamic resolution are narrow infrastructure: deferred/lazy resolution to break a genuine construction-order knot, generic component-outlet machinery, or runInInjectionContext bridges — all rare and worth a comment. The usual real cause is an injection-context error (they called inject() too late and reached for the locator to dodge it) or a circular dependency — both have proper fixes: capture dependencies at construction, or break the cycle by extracting the shared concern (previous lesson). Verdict: revert, name the actual problem, fix that.

🏋️ Exercises

1 (Easy): Create a typed FEATURE_FLAGS token (string array) with a factory default of [], override it in app config, and consume it in a component that checks for 'beta-ui'.
Solution
export const FEATURE_FLAGS = new InjectionToken<string[]>('feature.flags', {
  providedIn: 'root',
  factory: () => [],                       // default: no flags — no provider required
});

// app.config.ts (override):
providers: [{ provide: FEATURE_FLAGS, useValue: ['beta-ui', 'new-checkout'] }]

// consumer:
export class ShellComponent {
  private flags = inject(FEATURE_FLAGS);
  betaUi = this.flags.includes('beta-ui');
}
// One token object, exported once, imported by both sides — identity is the token.
2 (Easy): Wire a PaymentGateway abstract class to StripeGateway in prod and FakeGateway otherwise, using useClass — and show the test override.
Solution
export abstract class PaymentGateway {
  abstract charge(cents: number): Promise<Receipt>;
}
@Injectable() export class StripeGateway extends PaymentGateway { /* real */ }
@Injectable() export class FakeGateway extends PaymentGateway {
  async charge(cents: number) { return { id: 'fake', cents } as Receipt; }
}

// app.config.ts:
providers: [
  { provide: PaymentGateway,
    useClass: environment.production ? StripeGateway : FakeGateway },
]

// test:
TestBed.configureTestingModule({
  providers: [{ provide: PaymentGateway, useValue: { charge: vi.fn().mockResolvedValue({ id: 't' }) } }],
});
// Consumers inject PaymentGateway everywhere — three wirings, zero consumer changes.
3 (Medium): Build a pluggable "health check" system: a multi-token, two checks (API reachable, storage quota), and a service that runs them all and reports failures.
Solution
export interface HealthCheck { name: string; check(): Promise<boolean>; }
export const HEALTH_CHECK = new InjectionToken<HealthCheck>('health.check');

@Injectable() export class ApiHealthCheck implements HealthCheck {
  name = 'api';
  private http = inject(HttpClient);
  check() { return firstValueFrom(this.http.get('/api/ping')).then(() => true, () => false); }
}
@Injectable() export class StorageHealthCheck implements HealthCheck {
  name = 'storage';
  async check() {
    const { quota = 0, usage = 0 } = await navigator.storage.estimate();
    return usage / quota < 0.9;
  }
}

providers: [
  { provide: HEALTH_CHECK, useClass: ApiHealthCheck,     multi: true },
  { provide: HEALTH_CHECK, useClass: StorageHealthCheck, multi: true },
]

@Injectable({ providedIn: 'root' })
export class HealthService {
  private checks = inject(HEALTH_CHECK) as HealthCheck[];
  async failing(): Promise<string[]> {
    const results = await Promise.all(this.checks.map(async c => ({ c, ok: await c.check() })));
    return results.filter(r => !r.ok).map(r => r.c.name);
  }
}
// New check = new class + one provider line. HealthService never changes.
4 (Medium): Using useExisting, give read-only components a ReadonlyCart view of CartService (from the previous lesson) and prove both tokens share one instance.
Solution
export abstract class ReadonlyCart {
  abstract readonly items: Signal<readonly CartItem[]>;
  abstract readonly total: Signal<number>;
}

// CartService already matches the shape (items/total readonly signals) — just alias it:
providers: [{ provide: ReadonlyCart, useExisting: CartService }]

// Display component — CANNOT call add/remove (not on the type):
export class CartSummaryComponent {
  cart = inject(ReadonlyCart);          // {{ cart.total() }}
}

// Proof of shared instance (e.g. in a test):
const svc = TestBed.inject(CartService);
const ro  = TestBed.inject(ReadonlyCart);
svc.add({ id: 1, price: 10, qty: 1 });
expect(ro.total()).toBe(10);            // same signals — useExisting aliases, never duplicates
// (useClass here would have created a SECOND cart — the bug this exercise vaccinates against.)
5 (Hard): Implement the runtime-config pattern: ConfigService.load() fetching JSON, an app initializer awaiting it, and a typed API_URL token derived from the loaded config.
Solution
@Injectable({ providedIn: 'root' })
export class ConfigService {
  private _config = signal<AppConfig | null>(null);
  readonly config = computed(() => {
    const c = this._config();
    if (!c) throw new Error('Config read before load completed');   // loud, not silent
    return c;
  });
  async load(): Promise<void> {
    const res = await fetch('/assets/config.json');
    this._config.set(await res.json());
  }
}

export const API_URL = new InjectionToken<string>('api.url', {
  providedIn: 'root',
  factory: () => inject(ConfigService).config().apiUrl,   // safe: bootstrap awaited load
});

// app.config.ts:
export const appConfig: ApplicationConfig = {
  providers: [
    provideAppInitializer(() => inject(ConfigService).load()),   // bootstrap WAITS
  ],
};

// consumers: private base = inject(API_URL);
// Ops can change /assets/config.json per environment — no rebuild; consumers depend on
// a narrow typed token, not on the loading machinery.
6 (Hard): Package a shareable "toast" library the modern way: a provideToasts(options) function (position + max toasts as options, service, and a multi-token for custom toast renderers). Show consumer wiring.
Solution
// toast.tokens.ts
export interface ToastOptions { position: 'top' | 'bottom'; max: number; }
export const TOAST_OPTIONS = new InjectionToken<ToastOptions>('toast.options');
export interface ToastRenderer { type: string; render(t: Toast): TemplateResult; }
export const TOAST_RENDERER = new InjectionToken<ToastRenderer>('toast.renderer');

// provide-toasts.ts — the library's single entry point:
export function provideToasts(options: Partial<ToastOptions> = {}): EnvironmentProviders {
  return makeEnvironmentProviders([
    { provide: TOAST_OPTIONS,
      useValue: { position: 'bottom', max: 3, ...options } },     // defaults + overrides
    ToastService,                                                 // the engine
  ]);
}

// toast.service.ts (sketch):
@Injectable() export class ToastService {
  private opts = inject(TOAST_OPTIONS);
  private renderers = inject(TOAST_RENDERER, { optional: true }) as ToastRenderer[] | null;
  private _toasts = signal<Toast[]>([]);
  readonly toasts = this._toasts.asReadonly();
  show(t: Toast) { this._toasts.update(l => [...l, t].slice(-this.opts.max)); }
}

// CONSUMER app.config.ts:
providers: [
  provideToasts({ position: 'top', max: 5 }),
  { provide: TOAST_RENDERER, useClass: OrderToastRenderer, multi: true },  // their plugin
]
// EnvironmentProviders stops component-level misuse; options are typed arguments;
// extension is a multi-token — the same composition grammar as Angular's own provide*().

🎯 Key Takeaways

  • Every provider is token + recipe: useClass (swap), useValue (data), useFactory (conditional construction), useExisting (alias/narrow) — and provide*() functions just return arrays of them.
  • InjectionToken<T> bridges erased types to runtime DI — one exported token object per concept; factory defaults make constants overridable without mandatory providers.
  • multi: true is the plugin architecture: contributors self-register, consumers iterate — interceptors, validators, initializers, and your own extensible subsystems.
  • Hierarchy = scoping: root (app), route (lazy feature), component (per instance); modifiers (optional/self/skipSelf/host) tune resolution — skipSelf for parent-instance lookups.
  • The power patterns: platform/env factories, app-initializer runtime config, useExisting interface segregation, tests as provider lists.
  • Discipline: thin factories, narrowest-correct scope, no service-locator Injector.get, no token overriding at many levels — DI should reveal structure, never hide it.
