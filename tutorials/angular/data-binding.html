Templates & Data Binding Foundations

What you'll master: the four binding forms and the mental model that unifies them (data down, events up), property-vs-attribute — the distinction that explains most "why doesn't this bind?" bugs — class/style bindings, two-way binding and what the banana-in-a-box actually desugars to, template reference variables, and what template expressions may and may not do. Prerequisites: Components & Standalone Angular. Bindings are the alphabet of Angular — every lesson after this one is written in them.

Part 1 — The Model: a Template Is a Contract, Not a String

The four binding forms — direction is encoded in the SYNTAX:

{{ expression }}          INTERPOLATION      class → DOM (text)
[property]="expression"   PROPERTY BINDING   class → DOM (data down)
(event)="handler($event)" EVENT BINDING      DOM → class (events up)
[(ngModel)]="prop"        TWO-WAY            both — sugar for a [pair] + (event)

<img [src]="user.avatarUrl" [alt]="user.name">      <!-- data flows down -->
<button (click)="save()" [disabled]="form.invalid"> <!-- events flow up, state flows down -->
  Save
</button>

Read the punctuation as arrows: square brackets push data into the DOM, parentheses listen for events coming out. That's the whole system — one-way data flow in each direction, meeting in the component class. Because templates are compiled (previous lesson), every binding is type-checked: bind [disabled]="user.name" and strict templates flags a string flowing into a boolean property at build time. Interpolation is just property binding into a text node — {{ user.name }} and [textContent]-style updates are the same machinery, and both HTML-escape their output (no XSS via interpolation; Angular sanitizes, unlike raw innerHTML).

Part 2 — Property vs Attribute: the Bug-Explaining Distinction

<!-- HTML ATTRIBUTES initialize; DOM PROPERTIES are the live state. Angular binds PROPERTIES. -->

<input value="static">             <!-- attribute: the INITIAL value, then inert -->
<input [value]="user.name">        <!-- property: LIVE — updates whenever user.name does -->

<!-- Some attributes have NO property twin — bind the attribute explicitly: -->
<td [attr.colspan]="span">         <!-- [colspan] fails: no colspan DOM property -->
<button [attr.aria-label]="closeLabel">  <!-- ARIA: attr. binding -->
<div [attr.data-testid]="rowId">

<!-- Boolean properties want booleans, not strings: -->
<button [disabled]="isBusy">       <!-- ✓ boolean expression -->
<button disabled="false">          <!-- ✗ attribute PRESENT ⇒ disabled; "false" is truthy here! -->

HTML attributes and DOM properties are different worlds that happen to share names: the attribute is markup that initializes the element; the property is the object field JavaScript (and Angular) reads and writes at runtime. [value], [disabled], [src] bind properties — live and typed. When no property exists (colspan, ARIA attributes, data-*, SVG attributes), use [attr.name], which calls setAttribute (removing the attribute when the value is null). This single distinction decodes a whole genre of confusion — including why static disabled="false" still disables a button: for boolean attributes, presence is what the browser reads.

Part 3 — Class, Style & Multi-Value Bindings

<!-- Single class toggled by a boolean — the workhorse: -->
<li [class.selected]="item.id === selectedId()">
<div [class.error]="control.invalid && control.touched">

<!-- Several at once (object syntax): -->
<div [class]="{ selected: isSel, disabled: !enabled }">

<!-- Style bindings — with units built in: -->
<div [style.width.px]="progress">
<div [style.--accent]="brandColor">   <!-- CSS custom properties too -->

<!-- Events: $event is the native event; type it in the handler -->
<input (input)="onSearch($event)">
// onSearch(e: Event) { this.q.set((e.target as HTMLInputElement).value); }
<input (keydown.enter)="submit()">    <!-- key modifiers: keydown.escape, keydown.control.s -->

[class.x]="bool" is the idiom you'll write most — declarative state-to-class mapping, replacing imperative classList juggling. Style bindings accept a unit suffix ([style.width.px]) so the class stores a number, not a string with units glued on. For events, $event is the native DOM event (Angular adds no synthetic-event layer — what you know from the JS track applies directly), and key modifiers like (keydown.enter) compile the filtering you'd otherwise write by hand. One senior habit: handlers should delegate to a well-named class method rather than inlining logic in the template — templates state what happens, methods say how.

Part 4 — Two-Way Binding: the Banana in a Box, Desugared

<!-- [( )] = "banana in a box" — visual mnemonic for [prop] + (propChange) -->
<input [(ngModel)]="username">          <!-- needs FormsModule in imports -->

<!-- It is EXACTLY sugar for: -->
<input [ngModel]="username" (ngModelChange)="username = $event">

<!-- The convention generalizes to YOUR components: an input `value`
     plus an output named `valueChange` makes [(value)] work: -->
export class RatingComponent {
  value = model(0);          // model() = input + output pair in one (signals API)
  setTo(n: number) { this.value.set(n); }   // updating it notifies the parent
}
<app-rating [(value)]="score" />

Two-way binding is not a special data channel — it's mechanical sugar for a property binding plus an event binding whose name is the input's name + Change. Understanding the desugared form matters practically: you can bind the halves separately when you need to intercept ([ngModel]="v" (ngModelChange)="onChange($event)" to validate or transform before accepting), and you can make any component two-way-bindable by following the naming convention — or, in modern Angular, by declaring model(), which bundles the pair and gives you a writable signal. [(ngModel)] itself belongs to template-driven forms; for anything beyond simple cases, the Reactive Forms lesson is where robust form state lives.

Part 5 — Template Reference Variables & Expression Rules

<!-- #var captures the ELEMENT (or component/directive instance): -->
<input #search (keydown.enter)="go(search.value)">
<button (click)="search.focus()">Focus the box</button>

<video #player src="…"></video>
<button (click)="player.paused ? player.play() : player.pause()">⏯</button>

<!-- Template EXPRESSION rules — what's allowed in {{ }} and [ ]: -->
{{ user.name }}  {{ price * qty }}  {{ isVip ? 'VIP' : 'Standard' }}  {{ items.length }}
{{ user?.address?.city }}          <!-- optional chaining works -->
<!-- ✗ NOT allowed: assignments (x = y), new, ++/--, chained expressions with ; -->
<!-- ✗ AVOID: function calls doing real work — they run on EVERY change detection pass -->
{{ computeTotal() }}               <!-- runs constantly; make it a computed() signal or a pipe -->

A #ref variable gives the template a handle on a DOM element or child component without any class code — perfect for small element-to-element interactions (focus, media control, reading a value on enter). Scope is the whole template (with nuances inside structural blocks). Template expressions are a deliberately restricted JS subset: no assignments or side-effect operators, because templates must be declarative descriptions that Angular can safely re-evaluate any number of times. That re-evaluation is also why method calls in bindings are a performance smell: {{ computeTotal() }} reruns on every change-detection cycle, computing from scratch — the fixes are memoized derivation (computed() from Signals) or a pure pipe, both of which cache until inputs change.

Part 6 — Deep Dive: How Bindings Update, and Reading Templates Like a Senior

What the compiler makes of:   <p [class.vip]="user.vip">{{ user.name }}</p>

→ creation instructions: make <p>, make a text node       (run ONCE)
→ update instructions:   if (user.vip changed)  toggle class  (run on EVERY CD pass)
                         if (user.name changed) set text

Each binding becomes a tiny guarded instruction. Change detection (full story in
change-detection.html) executes these update instructions; each compares the
expression's CURRENT value to the LAST one and touches the DOM only on change.

Consequences you can reason from:
1. DOM updates are minimal & precise — Angular never re-renders "the whole component",
   it re-runs binding checks and patches what differs.        (no virtual DOM needed)
2. Every binding has a small per-check COST — thousands of bindings in a hot list,
   or expensive expressions, make each pass slower.           (→ performance.html)
3. Expressions must be SIDE-EFFECT-FREE — they may run at any time, any number of times.

Under the hood, each template compiles to creation code (build the DOM once) and update code (per-binding dirty checks). This is why Angular has no virtual DOM and doesn't need one: bindings are known statically at compile time, so the framework diffs values, not DOM trees. It also explains the two template performance rules you'll apply forever: keep bound expressions cheap (they run per check), and prefer derived state that caches (computed/pipes) over method calls that recompute.

Senior framing: in code review, templates are read as the component's public behavior spec: every [input] is a dependency, every (event) an exit point. Two smells reviewers flag on sight: logic-heavy templates (nested ternaries, chained expressions — push it into the class as named computeds: {{ statusLabel() }} reads like documentation) and stringly-typed escape hatches ([attr.*] where a property exists, innerHTML where interpolation would do — the latter is also an XSS review item: Angular sanitizes [innerHTML], but bypassSecurityTrustHtml calls deserve hostile scrutiny). The binding syntax being visually loud — brackets, parens, bananas — is a feature: the direction of every data flow in the view is greppable.

Interview Questions

Q1 (Beginner): Name the four binding forms and the direction data flows in each.

Interpolation {{ expr }} — class → DOM as escaped text. Property binding [prop]="expr" — class → DOM (live values into element/component properties). Event binding (event)="handler($event)" — DOM → class. Two-way [(ngModel)]="prop" — both directions, and it's pure sugar for [ngModel] plus (ngModelChange). Mnemonic: square brackets push data down, parentheses catch events coming up.

Q2 (Beginner): Why does <button disabled="false"> stay disabled, and what's the Angular fix?

Because disabled is a boolean attribute: the browser only checks whether it's present — the string "false" is still presence. Angular's property binding writes the DOM property instead, which is a real boolean: [disabled]="isBusy" enables and disables live as the expression changes. The attribute-vs-property distinction is the root of this whole bug family.

Q3 (Mid): When do you need [attr.something] instead of [something], and what does binding null to an attr. do?

When the attribute has no corresponding DOM property — colspan, ARIA attributes (aria-label, role), data-*, many SVG attributes. Property binding fails ("can't bind to X since it isn't a known property"); [attr.name]="expr" calls setAttribute under the hood. Binding null removes the attribute entirely — useful for conditionally omitting ARIA attributes rather than writing empty strings.

Q4 (Mid): Desugar <app-rating [(value)]="score" />. What must RatingComponent declare for this to compile?

It desugars to <app-rating [value]="score" (valueChange)="score = $event" />. The component needs an input named value and an output named exactly valueChange — the xChange naming convention is what the compiler looks for. Modern shortcut: value = model(0) declares the pair as one writable signal; this.value.set(n) updates locally and emits to the parent.

Q5 (Senior): Why are template expressions forbidden from having side effects, and how does that connect to how Angular updates the DOM?

Compiled templates consist of per-binding update instructions that change detection may execute on any pass, any number of times, in template order. Correctness therefore requires expressions to be pure value descriptions: if evaluating a binding mutated state, each check could change what later checks see — update loops, order dependence, and the dev-mode "ExpressionChangedAfterItHasBeenChecked" error. The same design is why Angular needs no virtual DOM: bindings are statically known, so the framework diffs each expression's value against its previous value and patches only real changes — cheap precisely because re-evaluation is assumed safe.

Q6 (Senior): A list page shows visible jank. Profiling shows template expressions dominating. What are the likely template-level causes and the fixes?

Likely causes: method calls in bindings ({{ format(row) }}, [class.x]="isX(row)") re-running for every row on every change-detection pass; expensive getters hiding computation; object/array literals created inline in bindings (new identity each check, defeating downstream OnPush/input caching); and thousands of bindings from rendering the full list. Fixes: precompute per-row view models once (or computed() signals / pure pipes, which cache by input identity); hoist literals to class fields; narrow change detection with OnPush/signals; add track to @for so rows keep identity; and virtualize long lists — the full toolkit lives in Performance.

🏋️ Exercises

1 (Easy): Bind an image's src and alt from a user object, and disable a "Save" button while saving is true.
Solution
<img [src]="user.avatarUrl" [alt]="user.name">
<button [disabled]="saving" (click)="save()">
  {{ saving ? 'Saving…' : 'Save' }}
</button>
<!-- Properties, not attributes: both update live as the class fields change. -->
2 (Easy): Give a list item the class active when its id matches selectedId, and set its aria-current attribute to "true" only in that case (absent otherwise).
Solution
<li [class.active]="item.id === selectedId"
    [attr.aria-current]="item.id === selectedId ? 'true' : null">
  {{ item.label }}
</li>
<!-- class.x for the styling toggle; attr. for ARIA (no DOM property), and binding
     null REMOVES the attribute — cleaner for screen readers than aria-current="false". -->
3 (Medium): Build a search box that updates a query field on every keystroke and triggers search() on Enter — using a typed handler for the input event.
Solution
<input [value]="query()" (input)="onInput($event)" (keydown.enter)="search()">
<p>Searching for: {{ query() }}</p>

// class:
query = signal('');
onInput(e: Event) {
  this.query.set((e.target as HTMLInputElement).value);   // $event is the NATIVE event
}
search() { console.log('searching', this.query()); }
// (keydown.enter) — the key-modifier syntax replaces manual e.key === 'Enter' checks.
4 (Medium): A progress bar: bind its width in percent from progress (0–100), turn it red via a class when below 20, and expose the number in a data-progress attribute for tests.
Solution
<div class="bar"
     [style.width.%]="progress"
     [class.low]="progress < 20"
     [attr.data-progress]="progress">
</div>

/* styles */
.bar { height: 8px; background: #4caf50; transition: width .3s; }
.bar.low { background: #e53935; }
<!-- [style.width.%] — unit suffix keeps the class holding a clean number. -->
5 (Hard): Create a TemperatureInputComponent usable as [(celsius)]="temp" — with model(), plus a button inside that bumps the value by 0.5 (proving child-side writes reach the parent).
Solution
import { Component, model } from '@angular/core';

@Component({
  selector: 'app-temperature-input',
  template: `
    <input type="number" [value]="celsius()" (input)="onInput($event)">
    <button (click)="celsius.update(c => c + 0.5)">+0.5°</button>
  `,
})
export class TemperatureInputComponent {
  celsius = model(0);        // = input `celsius` + output `celsiusChange` in one signal
  onInput(e: Event) {
    this.celsius.set(Number((e.target as HTMLInputElement).value));
  }
}

// Parent:  <app-temperature-input [(celsius)]="temp" />  — both the typing and the
// +0.5 button update `temp` in the parent, because model() emits on every set/update.
6 (Hard): Refactor this jank-prone template so no work-performing method runs per change-detection pass: 
<li *ngFor="let o of orders">{{ formatTotal(o) }} — {{ daysOverdue(o) }} days</li>
Solution
// Precompute a VIEW MODEL once per data change (computed caches until `orders` changes):
orders = signal<Order[]>([]);
rows = computed(() => this.orders().map(o => ({
  id: o.id,
  total: formatCurrency(o.total),                        // formatting done ONCE per change
  overdue: Math.floor((Date.now() - o.due) / 86_400_000),
})));

// Template — bindings are now plain property reads (cheap per check):
// @for (row of rows(); track row.id) {
//   <li>{{ row.total }} — {{ row.overdue }} days</li>
// }
// Alternative when you can't restructure: pure pipes ({{ o.total | currency }}) — cached
// by input identity. The rule: bindings READ; derivation happens once, elsewhere.

🎯 Key Takeaways

  • Four forms, direction in the syntax: {{ }} and [prop] push data down, (event) sends events up, [( )] is sugar for a [prop] + (propChange) pair.
  • Angular binds DOM properties (live, typed); use [attr.x] when no property exists (colspan, ARIA, data-*) — and binding null removes the attribute.
  • [class.x]="bool" and [style.prop.unit]="num" are the declarative styling idioms; $event is the native DOM event.
  • model() makes your own components two-way-bindable; the xChange naming convention is what the banana-in-a-box compiles to.
  • Template expressions are a pure, side-effect-free JS subset because compiled bindings re-run on every change-detection pass — keep them cheap, derive with computed()/pipes, never call work-performing methods.
  • Templates compile to creation + guarded update instructions — Angular diffs values, not DOM trees; that's why there's no virtual DOM and why binding hygiene is the performance story.
