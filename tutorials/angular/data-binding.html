Data Binding & Directives Basics

Beyond the four binding forms from the previous lesson, Angular has a whole vocabulary of directives — instructions attached to DOM elements that change their appearance or behavior. Understanding the difference between component, structural, and attribute directives (and when Angular got rid of most structural ones) rounds out the template model.

Three Kinds of Directives

KindWhat it doesExample
ComponentA directive WITH a template (everything from lesson 1)<app-item>
Attribute directiveChanges appearance/behavior of an existing element[ngClass], [ngStyle], custom
Structural directiveAdds/removes elements from the DOM*ngIf, *ngFor (legacy) → @if/@for (modern)
🎛️ A component IS a directive — specifically one with a template. Attribute directives decorate what's already there; structural directives control WHETHER and HOW MANY times something exists in the DOM. Angular 17+'s built-in control flow (@if/@for/@switch) replaces the old *ngIf/*ngFor for structural logic — smaller bundles, better type-checking, better performance — but you'll still meet *ngIf in existing codebases and it's fair game in interviews.

Attribute Bindings You'll Use Constantly

<!-- ngClass: toggle CSS classes based on an expression -->
<div [ngClass]="{ active: isActive(), disabled: isDisabled() }">...</div>
<div [class.active]="isActive()">...</div>      <!-- single-class shorthand, often preferred -->

<!-- ngStyle: dynamic inline styles -->
<div [ngStyle]="{ color: theme().color }">...</div>
<div [style.color]="theme().color">...</div>    <!-- single-style shorthand -->

<!-- Attribute (not property) binding, for things with no DOM property -->
<td [attr.colspan]="span()">...</td>

Writing a Custom Attribute Directive

import { Directive, ElementRef, inject, input, effect } from '@angular/core';

@Directive({ selector: '[appHighlight]', standalone: true })
export class HighlightDirective {
  private el = inject(ElementRef);
  color = input('yellow');            // <p [appHighlight] [color]="'lightblue'">

  constructor() {
    effect(() => { this.el.nativeElement.style.backgroundColor = this.color(); });
  }
}
// Usage: <p appHighlight [color]="'pink'">Highlighted text</p>
// A directive with NO template — it attaches behavior to an existing element via its selector.

Legacy Structural Directives (still common in real codebases)

<div *ngIf="loaded(); else loading">Content</div>
<ng-template #loading><p>Loading…</p></ng-template>

<li *ngFor="let item of items(); trackBy: trackById">{{ item.label }}</li>
// The `*` is sugar for <ng-template> — *ngIf="x" desugars to an <ng-template [ngIf]="x"> wrapper.
// trackBy (a function) served the same role @for's `track` now serves: stable identity for diffing.

Directive vs Component — the Interview Distinction

// Component:  @Component decorator, HAS a template, renders its own view.
// Directive:  @Directive decorator, NO template, attaches behavior/appearance to an EXISTING element.
// Both are classes Angular instantiates and manages; a component is really "a directive + a view."

Common Mistakes

MistakeSymptomFix
Using ngClass/ngStyle for a single class/styleUnnecessary overhead[class.x] / [style.prop] shorthand is lighter
Forgetting trackBy in *ngForFull list re-render on every changeAlways provide trackBy (legacy) or track (modern)
Binding to a non-existent DOM propertySilent no-op or errorUse [attr.x] for plain HTML attributes without a JS property
Mixing *ngIf and @if inconsistentlyConfusing codebaseStandardize on modern control flow for new code
Manipulating the DOM directly instead of via bindingsBypasses change detection, fragileUse bindings/directives, not direct DOM access
Overusing custom directives for one-off logicIndirection for no reuse benefitReach for a directive when behavior is reused across elements

Interview Questions

Q1: What's the difference between a component, an attribute directive, and a structural directive?

All three are Angular directives — classes that attach behavior to the DOM — but they differ in what they do. A component is a directive that HAS a template: it renders its own view and is used as a custom element (like <app-item>). An attribute directive has no template; it changes the appearance or behavior of an EXISTING element it's attached to, like [ngClass] toggling CSS classes, [ngStyle] setting inline styles, or a custom directive like [appHighlight] that changes background color. A structural directive controls whether and how many times an element exists in the DOM at all — it adds or removes elements, like the legacy *ngIf/*ngFor or, in modern Angular, the built-in @if/@for/@switch control-flow blocks. So the mental model is: components render views, attribute directives decorate elements that are already there, and structural directives decide if/how many elements are there in the first place.

Q2: Why does @for require a track expression (or *ngFor a trackBy)?

Because without a stable identity for each item, Angular's default change-detection strategy for lists is to compare by array position/reference, so when the underlying array changes (items reordered, one added/removed, or the array reference replaced), Angular can't tell which DOM nodes correspond to which logical items and may destroy and recreate DOM elements unnecessarily. This is expensive and also loses element state — like input focus, CSS transition state, or a form field's cursor position — because Angular is literally throwing away and rebuilding the DOM node. A track expression (in @for) or a trackBy function (in *ngFor) tells Angular how to identify each item (typically a stable id), so on a re-render Angular can match old and new items by that identity, reuse existing DOM nodes for unchanged items, and only create/destroy nodes for items that were actually added or removed — a much cheaper and more correct update. This is why @for makes track a required part of the syntax rather than optional.

Q3: When would you write a custom directive instead of a component?

You reach for a custom directive when you want to add reusable behavior or appearance to elements that already exist and already have their own template — you're not trying to render new markup, just attach a capability to arbitrary host elements. A classic example is a highlight or auto-focus directive: [appHighlight] can be applied to a <p>, a <div>, or any element, injecting the host element (via ElementRef, typically combined with Renderer2 for safe DOM manipulation, or an effect reacting to a signal input) and modifying its style or behavior without owning any view of its own. If instead you need to render your own markup/UI — a card, a list item, a form field with its own layout — that's a component, because components own a template and view. The dividing line is ownership of a view: components render UI, directives augment existing UI. In practice, cross-cutting concerns like access-control visibility, tooltips, drag handles, or lazy-image-loading behaviors are idiomatic candidates for custom attribute directives.

🏋️ Practice: replace an [ngClass] with the [class.x] shorthand and compare; write a custom highlight directive that reacts to an input signal via effect(); convert a legacy *ngFor + trackBy list to @for + track and compare re-render behavior when reordering items; build an *ngIf/else with an <ng-template>, then rewrite it as @if/@else; bind a table's colspan with [attr.colspan].

🎯 Key Takeaways

  • A component is a directive with a template; attribute directives decorate existing elements; structural directives control whether/how many elements exist.
  • [class.x] / [style.prop] shorthands are lighter than [ngClass] / [ngStyle] for single toggles.
  • Modern Angular replaces *ngIf/*ngFor with built-in @if/@for/@switch — smaller, faster, better type-checked — but legacy syntax is still common in real codebases.
  • track (or trackBy) gives Angular stable item identity for list diffing — always provide it to avoid unnecessary DOM recreation and lost element state.
  • Write a custom directive for reusable behavior on existing elements; write a component when you need to render your own view.
