Performance Production

What you'll master: the two distinct performance problems (startup vs runtime) and the toolbox for each — bundle discipline, lazy/defer/preload as one loading strategy, change-detection pruning, list virtualization, image/font loading — plus the measurement-first workflow that separates engineering from cargo cult. Prerequisites: this lesson deliberately assembles earlier ones — Change Detection, Lazy Loading, @defer/track, binding hygiene — into a single coherent practice.

Part 1 — Two Problems, Two Budgets

STARTUP performance                        RUNTIME performance
"how fast does it become usable?"          "how smooth is it once running?"
  bundle size (download + parse)             change-detection cost per interaction
  render-blocking resources                  DOM size and list rendering
  first data fetch                           memory leaks degrading over hours
  Metrics: LCP, TTI, FCP                     Metrics: INP, frame times, memory curve

MEASURE FIRST — the workflow that keeps you honest:
  1. Lighthouse / PageSpeed        startup lab numbers + diagnostics
  2. ng build + bundle analyzer    what's actually in the payload (source-map-explorer /
                                   esbuild-analyzer — find the 800KB chart lib in main)
  3. Angular DevTools profiler     which components check, how often, how long
  4. Chrome Performance panel      long tasks, layout thrash, real frame budget
  5. Real-user metrics (web-vitals → analytics)   because your M3 MacBook lies about
                                   your users' mid-range Android on hotel wifi
Optimize the measured bottleneck. Re-measure. Keep the receipt (before/after numbers).

Conflating the two problems produces absurd advice ("we added OnPush but the bundle is 4MB"). Startup is dominated by bytes and requests — the fix toolbox is loading strategy; runtime is dominated by work per interaction — the fix toolbox is change-detection pruning and DOM discipline. The measurement workflow is non-negotiable senior practice: every tool in this lesson can be applied uselessly to a non-bottleneck, and the difference between "we improved LCP 2.1s→1.2s on mid-tier mobile" and "we added performance best practices" is the difference between engineering and ritual.

Part 2 — Startup: Bundle Discipline & the Loading Strategy

// The loading strategy — one coherent system you already know the pieces of:
ROUTE SPLITTING    loadChildren/loadComponent per feature      (guards-lazy.html)
  + PRELOADING     background-warm likely-next chunks
  + @defer         heavy widgets below the fold load on viewport/interaction (directives.html)
  + canMatch       unauthorized users never download gated features

// BUNDLE DISCIPLINE — keeping main.js lean:
1. No static imports of lazy code (one import un-splits the bundle — lint boundaries)
2. Audit heavy deps: the analyzer names them.
     moment+locales → date-fns/Intl (../javascript/dates-intl.html)
     lodash → lodash-es cherry-picks or native (../javascript/arrays.html)
     chart/editor/pdf libs → inside @defer blocks only
3. Budgets in angular.json — regressions FAIL CI:
   "budgets": [{ "type": "initial", "maximumWarning": "500kB", "maximumError": "1MB" }]

// IMAGES — usually a bigger LCP lever than all JS work combined:
<img [ngSrc]="hero.url" width="1200" height="600" priority />   // NgOptimizedImage:
// enforces dimensions (no CLS), lazy-loads below-fold by default, priority hint +
// preconnect for the LCP image, srcset generation with a loader (CDN resizing).

// FONTS: font-display: swap, preload the one critical woff2, subset. inlineCritical CSS
// is on by default in the builder. And: prod build ships NO Angular compiler, AOT, minified.

Startup optimization is mostly not doing work: don't ship code the first paint doesn't need (the lazy/defer/preload triad), don't ship bytes a dependency audit could delete, don't let images without dimensions shift layout or load eagerly below the fold. NgOptimizedImage deserves special note — it packages the whole image-best-practice checklist (dimensions, lazy, priority, srcset) as a directive, and images are the actual LCP bottleneck for most content pages. Budgets close the loop: performance decays by a thousand small merges, and a CI gate is the only durable defense.

Part 3 — Runtime: Pruning Change Detection

// The layered defense you built across the track, now as one checklist:

1. OnPush / signal components EVERYWHERE          (change-detection.html)
   — unaffected subtrees skip checking entirely; signals mark precisely.
2. Cheap bindings                                  (data-binding.html Part 6)
   — no work-performing method calls in templates; computed()/pure pipes memoize.
   — no inline object/array literals in bindings ({} per check = new reference
     per check = downstream OnPush inputs "change" every pass).
3. track on every @for                             (directives.html)
   — DOM reuse instead of rebuild; row state survives.
4. runOutsideAngular for noisy non-UI work         (zone apps; zoneless makes it moot)
5. Defer/virtualize what isn't visible             (Part 4)

// Finding the offender when interaction is janky:
// DevTools profiler → sort by self time → it's almost always one of:
//   a method call in a hot list's binding   → precompute a view model (computed)
//   a huge @for without track/virtualization
//   a component tree checking on every keystroke → OnPush/signals missing at the top

Runtime jank in Angular is nearly always too much checking or too much DOM, and the fixes are the disciplines this track taught as correctness habits — which is the deeper point: in signal-era Angular, the correct architecture is the fast architecture. Signal reads narrow change detection; immutable updates make memoization trustworthy; derive-don't-sync eliminates redundant recomputation; track preserves DOM. Performance work becomes verifying the disciplines held, then profiling for the exceptions, rather than a separate optimization phase.

Part 4 — Big Lists: Virtualization

// 10,000 rows = 10,000× (DOM nodes × bindings × checks). The DOM is the cost —
// render only what's VISIBLE:

// CDK Virtual Scroll (@angular/cdk/scrolling):
<cdk-virtual-scroll-viewport itemSize="48" class="list-viewport">
  <div *cdkVirtualFor="let order of orders(); trackBy: trackById" class="row">
    {{ order.id }} — {{ order.customer }} — {{ order.total | currency }}
  </div>
</cdk-virtual-scroll-viewport>
/* .list-viewport { height: 600px; }  — the viewport needs a fixed height */

// ~12 rows rendered + buffer, recycled as you scroll. 10k rows ≈ cost of 15.
// itemSize: fixed row height (fastest path); variable heights need autosize strategies.

// The decision ladder for "the list is slow":
  < ~100 rows      @for + track is fine — fix the BINDINGS if slow (Part 3)
  100–1000         pagination or virtualization — often pagination is better UX anyway
  1000+            virtualize (or infinite-scroll + windowing)
  and ALWAYS       cheap rows: precomputed view models, OnPush row components,
                   no per-row subscriptions

Virtualization attacks the only cost that scales unboundedly with data — live DOM. The CDK viewport renders the visible window plus a buffer and recycles nodes on scroll, making list cost a function of viewport height, not data size. Trade-offs to name in review: in-page Ctrl+F stops finding unrendered rows, accessibility needs care (row count announcements), and variable-height rows complicate the math — which is why the ladder puts pagination first where product allows: it's simpler, shareable (page in the URL), and SEO-friendly. Often the honest answer to "render 10,000 rows fast" is "don't".

Part 5 — Memory: Performance Over Hours

// The leak inventory (each one is an earlier lesson's discipline):
subscriptions without lifetime binding      → takeUntilDestroyed / toSignal   (rxjs-patterns)
intervals/listeners/observers not released  → DestroyRef.onDestroy            (lifecycle)
event-emitter/store references to dead components → unsubscribe on destroy
detached DOM kept in closures               → null refs; beware caching nodes
"caches" that only grow                     → bound them (LRU / TTL / WeakMap for
                                              object-keyed metadata — ../javascript/memory.html)

// Detecting degradation (the "fine at 9am, janky at 4pm" bug):
// Chrome Memory panel → heap snapshot → use the app → snapshot → COMPARE:
//   growing counts of YOUR component classes = views outliving destruction.
//   Search the retainer tree for what's pinning them — it's almost always a callback.
// Performance monitor (live): JS heap + DOM node count while navigating back and forth
// between two routes — the sawtooth should return to baseline after GC.

Memory is runtime performance on a longer clock: leaked views keep their signals subscribed and their DOM referenced, so change detection and GC pressure grow through the day — SPAs are long-running processes and must be engineered like it. The detection technique (snapshot, navigate, snapshot, diff by your component class names) turns "it feels slower after a while" into a named culprit with a retainer chain. The fix list is deliberately a rerun of earlier lessons — leaks are lifecycle discipline failures, and the review-time question is always the same: where is the release for this acquire?

Part 6 — Deep Dive: A Performance Investigation, End to End

CASE: "The orders dashboard is slow." — the walkthrough that interview answers are made of:

1. DEFINE: slow HOW? First load (startup) or while using it (runtime)? Get a number:
   LCP 4.8s on mid-tier mobile; typing in the filter box janks (INP ~600ms).

2. STARTUP path: analyzer shows main.js 2.4MB — a charting lib + xlsx export are eager.
   → charts behind @defer (on viewport), export behind dynamic import on click,
   dashboard feature already lazy but preloaded. LCP 4.8 → 2.1s. Budget added to CI.

3. RUNTIME path: DevTools profiler during typing — 1,800 components checked per keystroke;
   OrderRowComponent self-time dominated by formatCurrency(row) calls.
   → filter input is a signal, rows are OnPush signal components reading computed
     view models (formatting done ONCE per data change, not per check);
     @for track order.id verified; 5,000 rows → cdk-virtual-scroll (viewport renders ~14).
   INP 600ms → 90ms.

4. VERIFY + GUARD: re-measure both numbers, add web-vitals reporting for real users,
   document the before/after in the PR. Done means MEASURED done.

The case study is the lesson's shape in miniature: classify the complaint, measure to a number, apply the matching toolbox (loading strategy for startup, checking/DOM discipline for runtime), and re-measure. Note what didn't happen: no speculative micro-optimizations, no rewriting working code the profiler never named, no "performance sprint" divorced from metrics. This narrative — problem → measurement → targeted fix → verified number → regression guard — is precisely what senior interviews are listening for when they ask "tell me about a performance problem you solved".

Senior framing: the anti-patterns worth calling by name in review: cargo-cult OnPush (added everywhere while bindings still call methods — the strategy can't skip what its own bindings make dirty); premature virtualization (a 30-row list behind a viewport, breaking find-in-page for nothing); trackBy theater (track $index on churning data — worse than nothing, per the directives lesson); bundle-blind lazy loading (routes split beautifully while one shared barrel import hauls every feature into main — only the analyzer tells the truth); and benchmark solipsism (dev-machine Lighthouse 100, no field data). The virtue that beats them all is boring: budgets and vitals wired into CI/analytics, so performance is a monitored invariant rather than a quarterly rescue mission. The JS-track performance lesson covers the platform-level substrate (long tasks, layout thrash, rendering pipeline) — this lesson is the Angular-specific layer on top of it.

Interview Questions

Q1 (Beginner): Startup vs runtime performance — what dominates each, and one key tool per problem?

Startup = bytes and requests: bundle size, render-blocking resources, first fetch — measured by LCP/TTI; key tools are the bundle analyzer (what's in the payload) and the loading strategy (lazy routes, @defer, image optimization). Runtime = work per interaction: change-detection breadth, binding cost, DOM size — measured by INP/frame times; key tool is the Angular DevTools profiler (which components check, how often, how long). Different bottlenecks, different toolboxes — diagnosing which problem you have precedes any fix.

Q2 (Beginner): What does NgOptimizedImage actually do, and why are images often the real LCP story?

It enforces explicit width/height (reserving layout space — no CLS), lazy-loads below-fold images by default, adds priority fetch hints + preconnect for the marked LCP image, and generates responsive srcsets with a CDN loader. Images matter because on content pages the largest contentful paint usually is an image — hundreds of KB that no amount of JS optimization touches; correct dimensions, priority, and sizing routinely move LCP more than all bundle work combined.

Q3 (Mid): A teammate added OnPush everywhere but typing in a filter still checks 1,800 components and janks. What did they miss?

OnPush prunes unaffected subtrees — but several things defeat it: the filter input's keystrokes are events inside the tree that's re-rendering (its ancestors and the list it feeds legitimately re-check); bindings that call methods re-do work every check regardless of strategy; inline object literals create new references per check, making child inputs "change" constantly; and without signals, the whole list re-checks for one row's change. Fixes: make the filter a signal feeding a computed filtered list (signal-driven CD marks precisely), precompute row view models so bindings are property reads, verify track, and virtualize if the row count is large. OnPush is necessary, not sufficient — binding hygiene and signals do the actual narrowing.

Q4 (Mid): Walk the complete loading strategy for a large Angular app, naming the mechanism for each layer.

Route level: loadChildren/loadComponent split every feature into its own chunk; canMatch guards keep gated features' chunks from unauthorized users entirely. Warmth: a preloading strategy (all, flagged, or network-aware) downloads likely-next chunks in the background so first navigations don't stall. Template level: @defer (on viewport / interaction; prefetch on idle) for heavy below-fold widgets — charts, editors, comment threads — with placeholder/loading/error blocks. Asset level: NgOptimizedImage (dimensions/priority/srcset), font preloading + swap, critical CSS inlined. Enforcement: angular.json budgets failing CI, analyzer checks that no static import un-split a chunk. Each layer removes bytes from the critical path; together they're one strategy, not five tricks.

Q5 (Senior): Users report the app "gets slow by afternoon". Design the investigation and name the three most likely causes.

This signature — degradation over session time — is memory/leak territory, not startup. Investigate: Chrome Performance Monitor live (JS heap + DOM nodes while navigating between two routes repeatedly — a healthy app's sawtooth returns to baseline post-GC; a leaking one stair-steps up); then heap snapshots before/after the navigation cycle, diffed by constructor name — growing counts of your component classes prove views outlive destruction; the retainer tree names the pinner. Likely causes, in order: (1) subscriptions to long-lived streams without takeUntilDestroyed/async-pipe — the callback pins the component graph; (2) intervals, sockets, or observers (Intersection/Resize/Mutation) acquired without a DestroyRef release; (3) unbounded growth structures — caches without TTL/LRU, event logs in stores, detached DOM held by closures. Fix at the acquire site (colocated release), verify by re-running the snapshot diff, and add the navigation-cycle memory check to the E2E suite so the class of bug stays fixed.

Q6 (Senior): Leadership wants a "performance sprint". Argue for what actually keeps an Angular app fast long-term.

A sprint fixes today's numbers and decays immediately — performance regresses by a thousand unreviewed merges, not by one dramatic failure. The durable system: (1) budgets as CI gates — initial-bundle and component-style budgets in angular.json fail builds on regression; (2) field data — web-vitals (LCP/INP/CLS) reported from real users into analytics with alerting, because lab numbers on dev hardware systematically lie; (3) architecture defaults that are fast by construction — signal components, OnPush, track-required @for, lazy feature boundaries with lint-enforced module isolation — so the easy path is the fast path; (4) review vocabulary — binding hygiene, acquire/release pairing, analyzer checks on dependency PRs — the checklists from this track; (5) a measurement culture: perf PRs carry before/after numbers. Then the sprint becomes what it should be: an occasional targeted investigation (Part 6's walkthrough) rather than a recurring rescue of an unmonitored system.

3 (Medium): Fix this janky row template so no work happens per change-detection pass, and state where each computation moved: 
@for (o of orders; track o.id) {
  <tr [class.overdue]="isOverdue(o)" [style]="{ opacity: o.draft ? 0.5 : 1 }">
    <td>{{ formatId(o) }}</td><td>{{ o.total | currency }}</td>
    <td>{{ daysLeft(o) }} days</td>
  </tr>
}
Solution
// Class: one computed view model — ALL derivation happens once per data change:
orders = signal<Order[]>([]);
rows = computed(() => this.orders().map(o => ({
  id: o.id,
  label: formatId(o),                                    // was: per-check method call
  overdue: o.due < Date.now(),                           // was: isOverdue() per check
  daysLeft: Math.ceil((o.due - Date.now()) / 86_400_000),// was: daysLeft() per check
  dim: o.draft,
})));

// Template — bindings are now property reads (near-free per check):
@for (row of rows(); track row.id) {
  <tr [class.overdue]="row.overdue" [class.dim]="row.dim">
    <td>{{ row.label }}</td><td>{{ row.total | currency }}</td>
    <td>{{ row.daysLeft }} days</td>
  </tr>
}
/* .dim { opacity: 0.5 } */
// Moves: methods → computed view model (memoized until orders changes); inline style
// object literal (new reference per check!) → a CSS class; currency pipe stays — pure
// pipes are already memoized. This is data-binding.html Part 6 as a reflex.

6 (Hard): Write the memory-leak hunt runbook for "the app slows down after heavy use", as you'd hand it to a mid-level engineer — steps, what to look for, and the three fixes they'll most likely apply.
Solution
RUNBOOK: session-degradation investigation
1. REPRODUCE CHEAPLY: pick two routes (list ↔ detail). Chrome DevTools → Performance
   monitor (live JS heap + DOM nodes). Navigate back and forth 20×, force GC (bin icon).
   HEALTHY: heap/nodes sawtooth back to ~baseline. LEAK: stair-step growth.
2. NAME THE LEAK: Memory tab → heap snapshot → do 10 more cycles → second snapshot →
   "Comparison" view → filter by our prefix (e.g. OrderDetailComponent).
   Growing instance count = views outliving destruction.
3. FIND THE PINNER: click an instance → Retainers tree → walk up until you hit
   something recognizable: a subscription callback, a setInterval closure, an
   observer, a Map in a root service holding the component.
4. FIX at the ACQUIRE SITE (not with a random ngOnDestroy elsewhere):
   a. Long-lived stream subscribe → add takeUntilDestroyed() (or convert to
      toSignal/async pipe) — the most common by far.
   b. interval/listener/Intersection-Resize-Mutation observer → register release:
      inject(DestroyRef).onDestroy(() => clearInterval(id) / observer.disconnect()).
   c. Unbounded structure → bound it: LRU/TTL for caches, WeakMap for
      object-keyed metadata, cap event logs in stores.
5. VERIFY: repeat step 1 — sawtooth must return to baseline. Attach before/after
   screenshots to the PR.
6. GUARD: add the 20-cycle navigation to the E2E suite with a heap-growth assertion
   (playwright + CDP), so the bug CLASS stays fixed, not just this instance.

🎯 Key Takeaways

  • Two problems, two toolboxes: startup (bytes — loading strategy: lazy/preload/@defer/canMatch, image+font discipline, budgets) vs runtime (work — CD pruning, binding hygiene, DOM bounds). Diagnose before fixing.
  • Measure first, re-measure after: analyzer for bundles, DevTools profiler for checking, Performance panel for frames, heap snapshots for leaks, web-vitals for the truth about real users.
  • The correct architecture IS the fast one: signals + OnPush narrow checking, computed view models make bindings property-reads, immutable updates make memoization trustworthy, track preserves DOM.
  • Big lists: pagination first where UX allows; CDK virtual scroll makes cost proportional to viewport, with find-in-page and a11y caveats to manage.
  • Memory is performance over hours: every acquire needs a colocated release; snapshot-diff by component class name turns "feels slower" into a named retainer chain.
  • Durable speed is institutional: CI budgets, lint module boundaries, field vitals with alerts, and before/after numbers on every perf PR — not sprints.
