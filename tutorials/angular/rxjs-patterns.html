RxJS Patterns & Pitfalls RxJS

What you'll master: the four flattening operators (switchMap, mergeMap, concatMap, exhaustMap) and the decision framework that is the RxJS interview question, subscription lifecycle discipline (takeUntilDestroyed), error-handling placement that keeps streams alive, combination operators, and the production patterns — typeahead, polling, retry with backoff. Prerequisites: RxJS Fundamentals. This is where stream fluency becomes production competence.

Part 1 — The Problem: Streams That Launch Streams

// A query stream where each value should TRIGGER AN HTTP CALL (a stream of streams):
query$.pipe(
  map(q => this.http.get(`/api/search?q=${q}`)),   // Observable<Observable<Result>> 😵
).subscribe(innerObs => innerObs.subscribe(render)); // nested subscribe — the anti-pattern

// FLATTENING operators subscribe to the inner stream FOR you — the question is:
// what happens when a NEW outer value arrives while the PREVIOUS inner is still running?

switchMap   CANCEL the previous inner, switch to the new one     "only the latest matters"
mergeMap    run them ALL concurrently                            "every one matters, order doesn't"
concatMap   QUEUE — one at a time, in order                      "every one matters, order matters"
exhaustMap  IGNORE new values while one is running               "first wins until it finishes"

Higher-order streams appear the moment a stream's values each start async work — searches per keystroke, requests per route change, saves per click. The four flatteners differ only in their concurrency policy for overlapping inners, and choosing by that policy — not by habit — is the skill. The mnemonic that sticks: switch = latest wins (cancel stale), merge = all at once, concat = all in order, exhaust = busy, go away. Every one of them exists because some overlap policy is catastrophically wrong somewhere: the rest of this Part-by-Part makes each concrete.

Part 2 — Choosing the Flattener: the Canonical Cases

// SWITCHMAP — typeahead/search/route-driven loads. Stale responses must DIE:
results$ = query$.pipe(switchMap(q => this.api.search(q)));
// without it: response for "ja" can arrive AFTER "java"'s → stale results overwrite fresh
// (also: unsubscribing an HttpClient inner ABORTS the actual request — real cancellation)

// EXHAUSTMAP — submit/save buttons. Double-click must NOT double-charge:
save$ = saveClicks$.pipe(exhaustMap(() => this.api.saveOrder(this.order())));
// clicks while the save is in flight are swallowed — first one wins

// CONCATMAP — ordered writes. Audit events must arrive in sequence:
logs$ = auditEvents$.pipe(concatMap(e => this.api.log(e)));
// each POST waits for the previous — order preserved, at the cost of queuing latency

// MERGEMAP — independent parallel work. Enrich each row, order irrelevant:
details$ = ids$.pipe(mergeMap(id => this.api.detail(id), 4));  // ← concurrency cap!

The wrong-choice failure modes (interview gold):
  switchMap on SAVES   → in-flight save CANCELLED by the next click: silent data loss
  mergeMap on SEARCH   → all responses land, LAST-ARRIVED wins — race condition UI
  exhaustMap on SEARCH → typing while a slow search runs is IGNORED — "search feels broken"
  concatMap on SEARCH  → every stale query still runs, queued — results lag far behind typing

Map the policy to the domain: reads want switchMap (only the latest query/route matters; cancelling stale requests is free correctness), writes want anything but (switchMap cancelling a write is data loss — the single most damaging RxJS bug in production code). Buttons that trigger a mutation want exhaustMap (idempotent UX for free); ordered side effects want concatMap; fan-out enrichment wants mergeMap with a concurrency argument (unbounded mergeMap over a big list is a self-inflicted DDoS — the same lesson as Promise.all over 500 fetches in the JS track).

Part 3 — Subscription Lifecycle: takeUntilDestroyed & Friends

// The leak (rxjs.html warned): manual subscribe to a long-lived stream, no teardown.
// The modern, ordered checklist:

// 1. DON'T subscribe — let the template do it:
vm$ = this.store.vm$;                          // {{ (vm$ | async)?.name }} — auto-unsub

// 2. DON'T subscribe — settle into a signal:
vm = toSignal(this.store.vm$, { initialValue: null });   // lifetime-bound automatically

// 3. MUST subscribe (imperative side effects)? Bind to the component lifetime:
constructor() {
  this.ws.messages$
    .pipe(takeUntilDestroyed())                // completes the stream on component destroy
    .subscribe(m => this.handle(m));
}
// outside constructor: takeUntilDestroyed(this.destroyRef)

// 4. One-shots complete themselves — no ceremony needed:
this.http.post('/api/save', data).subscribe();          // completes after one emission
this.route.data.pipe(first()).subscribe(...);           // first() completes it

// LEGACY patterns you'll still read: private destroy$ = new Subject<void>() +
// takeUntil(this.destroy$) + ngOnDestroy(){ this.destroy$.next(); } — same idea, manual.

The hierarchy is deliberate: each level down adds ceremony and risk, so justify descending. Template consumption (async pipe) and toSignal tie stream lifetime to view lifetime structurally — there is no cleanup to forget. takeUntilDestroyed() is the sanctioned form of the old destroy$ dance, completing your pipeline when the component goes — one operator, injected context, done. And knowing what doesn't need management (completing streams: HTTP one-shots, first()/take(n) pipelines) matters equally: teams that wrap every HTTP subscribe in lifecycle machinery are cargo-culting — the stream already completes; the machinery guards against nothing.

Part 4 — Error Handling: Placement Decides Survival

// Errors are TERMINAL — where you catch decides WHAT dies:

// ❌ catchError on the OUTER stream — one bad search kills typeahead FOREVER:
query$.pipe(
  switchMap(q => this.api.search(q)),
  catchError(() => of([])),          // handles the error... by COMPLETING the outer stream
).subscribe(render);                  // → no further keystrokes are processed. Dead UI.

// ✓ catchError on the INNER stream — the failed request is replaced, the pipeline lives:
query$.pipe(
  switchMap(q => this.api.search(q).pipe(
    catchError(() => of([] as Result[])),   // this inner emits [] and completes
  )),
).subscribe(render);                  // next keystroke → next search. UI survives.

// RETRY — transient failures, with backoff (the production shape):
this.api.critical().pipe(
  retry({ count: 3, delay: (err, n) => timer(1000 * 2 ** (n - 1)) }),  // 1s, 2s, 4s
  catchError(err => { this.toast.error('Still failing'); return EMPTY; }),
)
// EMPTY completes silently (no value); of(fallback) emits a default; throwError rethrows
// after logging. Choose by what downstream should SEE.

Because error terminates a stream, catch placement is architecture: catching inside the flattener sacrifices only the failed inner request; catching outside sacrifices the whole long-lived pipeline — the "search stopped working after one timeout" bug that teams chase for days. retry belongs upstream of catchError (retry the source, then admit defeat), with delay/backoff for anything network-flavored — the same exponential-backoff reasoning as reconnection in the JS track. The recovery-value choice (of(fallback) vs EMPTY vs rethrow) is a UX decision wearing operator clothes: show empty state, skip a tick silently, or propagate to a global handler.

Part 5 — Combining Streams

combineLatest([filters$, page$, sort$]).pipe(   // LATEST of each, re-emit when ANY changes
  switchMap(([f, p, s]) => this.api.query(f, p, s)),
)   // ⚠️ emits nothing until EVERY source has emitted once — startWith defaults fix stalls

forkJoin({ user: api.user(), orders: api.orders() })   // wait for ALL to COMPLETE; emit once
  // the Promise.all of RxJS — only for one-shot streams (HTTP); never-completing → never emits

merge(clicks$, keyEnter$)            // interleave: either source triggers
race(primary$, fallbackTimer$)       // first to emit wins, others unsubscribed
withLatestFrom(auth$)                // main stream drives; snapshot secondary at emit time
zip(a$, b$)                          // pairwise by index — rare; queues if speeds differ

// The signals-era note: combineLatest-for-VIEW-MODELS is better as computed():
//   vm = computed(() => ({ user: user(), filter: filter() }))
// combineLatest survives where TIME matters: triggering re-queries, gating on multiple
// async sources, coordinating streams that stay streams.

Combination operators answer "when do multiple sources produce one value": combineLatest for live recombination (the filters-page-sort re-query engine), forkJoin for parallel one-shots (page-load data), merge for "any of these events", withLatestFrom for drive-plus-snapshot (submit click reads current auth). Two classic stalls to know: combineLatest waits for every source's first emission (seed with startWith), and forkJoin waits for completion — feed it a BehaviorSubject and it never fires. In signal-era code, static view-model assembly migrates to computed(); the operators keep the jobs where emission timing is the point.

Part 6 — Deep Dive: Production Recipes & the Review Checklist

// TYPEAHEAD (the complete, correct shape — every piece earns its place):
results = toSignal(
  toObservable(this.query).pipe(
    debounceTime(300),                        // wait for a pause in typing
    distinctUntilChanged(),                   // "abc" → "abcd" → backspace → "abc": skip repeat
    switchMap(q => q.length < 2
      ? of([] as Result[])                    // short queries: no request at all
      : this.api.search(q).pipe(
          catchError(() => of([] as Result[])),   // inner catch: pipeline survives
        )),
  ),
  { initialValue: [] as Result[] },
);

// POLLING with pause-when-hidden (visibility from the JS realtime lesson):
poll$ = visible$.pipe(                        // BehaviorSubject<boolean> of tab visibility
  switchMap(vis => vis ? timer(0, 15_000) : EMPTY),   // hidden → no timer at all
  switchMap(() => this.api.status().pipe(catchError(() => EMPTY))),
  shareReplay({ bufferSize: 1, refCount: true }),
);

// SAVE BUTTON, fully armored:
save = (clicks$: Observable<void>) => clicks$.pipe(
  exhaustMap(() => this.api.save(this.draft()).pipe(
    retry({ count: 2, delay: () => timer(1000) }),
    tap(() => this.toast.success('Saved')),
    catchError(err => { this.toast.error('Save failed'); return EMPTY; }),
  )),
);

These three recipes cover a startling fraction of real UI async work, and each line answers a failure mode you can now name: debounce (request storms), distinctUntilChanged (duplicate work), switchMap (stale-response races), inner catchError (pipeline death), exhaustMap (double submits), retry-with-delay (transient blips), EMPTY vs of([]) (what downstream sees). When you read team code, compare against these shapes — most production RxJS bugs are one missing line from a known recipe.

Senior framing — the RxJS review checklist: (1) every subscribe justified — could the template/toSignal consume it? — and lifetime-bound if long-lived; (2) flattener choice defensible in one sentence naming the overlap policy ("switchMap because only the latest route's data matters"); (3) catchError placed inside flatteners on long-lived pipelines; (4) writes never under switchMap; (5) unbounded mergeMap has a concurrency cap; (6) no nested subscribes (a flattener is missing); (7) shared expensive sources (shareReplay with refCount) instead of N executions. Teams that encode this list in PR templates stop shipping the whole genre of async-race bugs — it's the highest-leverage seven lines of review guidance in Angular.

Interview Questions

Q1 (Beginner): What problem do switchMap/mergeMap/concatMap/exhaustMap all solve, and what single question distinguishes them?

They flatten higher-order streams: when each outer value starts an inner async operation (HTTP call per keystroke), they subscribe to the inners for you, replacing nested subscribes. The distinguishing question: what happens when a new outer value arrives while a previous inner is still running? switchMap cancels the old; mergeMap runs all concurrently; concatMap queues in order; exhaustMap ignores newcomers until the current finishes. Same inputs, four different concurrency policies.

Q2 (Beginner): Why does a typeahead need both debounceTime and switchMap? What does each prevent?

debounceTime(300) prevents the request storm: it waits for a typing pause, so "angular" fires one request, not seven. switchMap prevents the race: even debounced, an earlier slow response ("ang") can arrive after a later fast one ("angular"); switchMap cancels the previous request when a new query arrives, so only the latest query's response can ever render. Different bugs — volume vs ordering — needing both fixes; add distinctUntilChanged for the third (duplicate queries after backspacing).

Q3 (Mid): Users double-click "Place order" and get charged twice. The code is clicks$.pipe(switchMap(() => api.placeOrder())). Explain the bug — and why switchMap makes it worse, not better.

Two bugs stacked. Double charge: each click starts a POST; the server processes both — flattener choice can't fully fix non-idempotent APIs (backend needs an idempotency key), but the client should not initiate duplicates. switchMap makes it worse: the second click cancels the client's view of the first request — but the HTTP request may already be committed server-side, so the user gets charged by a request the app abandoned (no success handler ran: no receipt, no redirect — charged and confused). Fix: exhaustMap — clicks during flight are ignored, one order at a time — plus disabled-button UX and a server idempotency key (the backend half).

Q4 (Mid): When do you still need takeUntilDestroyed in a signals-era component, and what replaced the old destroy$ Subject pattern?

You need it exactly when you manually subscribe to a long-lived stream for imperative side effects — websocket messages driving non-state actions, router events triggering analytics. Template consumption (async pipe) and toSignal handle their own lifetimes, and one-shot streams (HTTP, first()) complete themselves. takeUntilDestroyed() — injection-context aware, or explicit takeUntilDestroyed(destroyRef) — replaces the manual pattern of a private destroy$ = new Subject(), takeUntil(destroy$) in every pipe, and ngOnDestroy(){ destroy$.next() }: same semantics, zero boilerplate to forget.

Q5 (Senior): "Our dashboard search worked for weeks, then permanently broke for a user session after one 500 — reload fixed it." Diagnose from the symptom alone.

The symptom signature — works, then permanently dead within a session, healed by reload — is a terminated long-lived stream. The 500 errored the inner HTTP observable; with no inner catchError, the error propagated to the outer keystroke pipeline, which terminated (errors are terminal). Every subsequent keystroke emits into a dead stream — no requests, no renders — until reload rebuilds the subscription. Fix: catchError inside the switchMap, returning of([]) (empty results for that query) so the outer pipeline survives; optionally retry with backoff before conceding. The generalizable lesson: on long-lived pipelines, error handling placement is a survival property, and this exact bug is why the review checklist pins catchError placement.

Q6 (Senior): Design the data loading for a route: on param id change, load the record; show loading; handle errors without killing navigation; support a manual refresh button. Name every operator choice.

merge(paramId$, refresh$.pipe(withLatestFrom(paramId$), map(([, id]) => id))) — both navigation and refresh feed one id stream (merge: either triggers). switchMap per id (reads: only latest matters; navigating cancels the stale load). Inner pipe: map(data => ({ status: 'loaded', data })), catchError(err => of({ status: 'error' as const, err })) — inner catch so future navigations still work; startWith({ status: 'loading' }) inside the switchMap so every id change resets to loading. Consume via toSignal; template switches on status. This is the state-machine-as-stream shape — and the honest senior note: the newer resource() API (Signals) packages exactly this pattern; hand-rolling it remains necessary knowledge for the codebases that predate it.

🏋️ Exercises

1 (Easy): For each scenario pick the flattener and justify in one sentence: (a) load user profile on route change; (b) "Export PDF" button; (c) sync queued offline edits in order; (d) prefetch details for 50 visible rows.
Solution
(a) switchMap  — navigation means the old profile no longer matters; cancel the stale load.
(b) exhaustMap — repeated clicks while exporting must be ignored, not queued or parallel.
(c) concatMap  — writes whose ORDER matters; each edit waits for the previous to commit.
(d) mergeMap(fn, 4–6) — independent reads, order irrelevant, but CAP concurrency
    so 50 rows don't become 50 simultaneous requests.
2 (Easy): Fix this leak two ways (modern + template-consumption): 
export class TickerComponent implements OnInit {
  price = 0;
  ngOnInit() { this.feed.prices$.subscribe(p => this.price = p); }
}
Solution
// Fix A — don't subscribe at all; settle into a signal:
export class TickerComponent {
  price = toSignal(inject(FeedService).prices$, { initialValue: 0 });
}   // template: {{ price() }} — lifetime handled, and zoneless-safe

// Fix B — must stay imperative? Bind the subscription to the component:
export class TickerComponent {
  price = signal(0);
  constructor(feed: FeedService) {
    feed.prices$.pipe(takeUntilDestroyed()).subscribe(p => this.price.set(p));
  }
}
// The original: prices$ never completes → subscription (and the component it captures)
// outlives every navigation. Classic leak from the fundamentals lesson, now with its cures.
3 (Medium): Build a status poller: every 10s call api.status(), skip failed ticks silently, share one execution among all subscribers, and stop entirely while the tab is hidden.
Solution
visible$ = new BehaviorSubject(document.visibilityState === 'visible');
// (wire: document.addEventListener('visibilitychange', () => visible$.next(...)))

status$ = this.visible$.pipe(
  switchMap(vis => vis ? timer(0, 10_000) : EMPTY),       // hidden → NO timer running at all
  switchMap(() => this.api.status().pipe(
    catchError(() => EMPTY),                              // failed tick: emit nothing, live on
  )),
  shareReplay({ bufferSize: 1, refCount: true }),          // one poller, N consumers,
);                                                         // stops when last consumer leaves
// Each choice: outer switchMap swaps timer/EMPTY on visibility; inner catchError placement
// keeps the poller alive; EMPTY (not of(null)) means consumers never see a fake value.
4 (Medium): Implement retry with exponential backoff + jitter for a critical GET: 4 attempts (1s/2s/4s base), then surface a user-facing error while keeping the calling pipeline alive.
Solution
load$ = trigger$.pipe(
  switchMap(() => this.api.critical().pipe(
    retry({
      count: 3,                                            // 1 original + 3 retries = 4 attempts
      delay: (_err, retryIndex) => {
        const base = 1000 * 2 ** (retryIndex - 1);         // 1s, 2s, 4s
        return timer(base / 2 + Math.random() * (base / 2)); // + jitter (JS realtime lesson:
      },                                                     //   don't stampede in sync)
    }),
    catchError(err => {
      this.toast.error('Service unavailable — try again later');
      return EMPTY;                                        // no value; OUTER stream survives
    }),
  )),
);
// retry sits UPSTREAM of catchError: exhaust recovery first, then admit defeat gracefully.
5 (Hard): A form's "check username availability" must: debounce 400ms, skip identical values, cancel stale checks, map to 'free'|'taken'|'error' without killing the stream, and expose a signal. Write it.
Solution
username = signal('');
availability = toSignal(
  toObservable(this.username).pipe(
    debounceTime(400),
    distinctUntilChanged(),
    switchMap(name => {
      if (name.length < 3) return of('idle' as const);     // no request for short input
      return this.api.checkUsername(name).pipe(
        map(r => (r.available ? 'free' : 'taken') as const),
        startWith('checking' as const),                    // loading state PER check
        catchError(() => of('error' as const)),            // inner catch: next input still works
      );
    }),
  ),
  { initialValue: 'idle' as const },
);
// Template: @switch (availability()) { @case ('checking') …spinner… @case ('taken') …red… }
// Note startWith INSIDE the switchMap — each new check resets to 'checking' automatically.
// (This is also a hand-rolled async validator — form-validation.html shows the Forms API way.)
6 (Hard): Code-review this checkout flow — identify every pattern violation and rewrite: 
onCheckout() {
  this.cartService.items$.subscribe(items => {
    this.api.validateStock(items).subscribe(ok => {
      if (ok) this.api.placeOrder(items).subscribe(order => {
        this.router.navigate(['/confirm', order.id]);
      });
    });
  });
}
Solution
// Violations:
// 1. Triple-nested subscribes — no cancellation, no unified error path.
// 2. Subscribing to items$ (a BehaviorSubject-style LIVE stream) inside a click handler:
//    it NEVER completes → every checkout click adds a PERMANENT subscription that re-runs
//    the whole flow on every future cart change (ghost orders!). The worst bug here.
// 3. No error handling anywhere — one failure silently strands the user.
// 4. No double-click protection on a WRITE.

checkout$ = this.checkoutClicks$.pipe(
  exhaustMap(() =>                                        // 4: ignore clicks while running
    this.cartService.items$.pipe(
      first(),                                            // 2: SNAPSHOT the cart — completes
      switchMap(items => this.api.validateStock(items).pipe(
        switchMap(ok => ok
          ? this.api.placeOrder(items)                    // 1: flatten, don't nest
          : throwError(() => new Error('OUT_OF_STOCK'))),
      )),
      tap(order => this.router.navigate(['/confirm', order.id])),
      catchError(err => {                                 // 3: one error path for the flow
        this.toast.error(err.message === 'OUT_OF_STOCK'
          ? 'Some items are out of stock' : 'Checkout failed — you were not charged');
        return EMPTY;                                     // outer stream survives for retry
      }),
    ),
  ),
);
constructor() { this.checkout$.pipe(takeUntilDestroyed()).subscribe(); }
// first() on live streams inside event flows is the unsung hero: snapshot semantics,
// self-completing, no ghost re-runs.

🎯 Key Takeaways

  • Flatteners are concurrency policies for overlapping inner streams: switch = cancel stale (reads), exhaust = ignore while busy (buttons), concat = ordered queue (writes in order), merge+cap = parallel fan-out. switchMap on writes is data loss.
  • Subscription hierarchy: template/async pipe → toSignal → manual subscribe with takeUntilDestroyed(); one-shots and first()/take(n) pipelines clean themselves.
  • Errors are terminal, so placement is survival: catchError inside the flattener sacrifices one request; outside, it kills the pipeline ("search broke until reload").
  • retry({count, delay}) with exponential backoff + jitter before conceding; recover with of(fallback), EMPTY, or rethrow — a UX decision per stream.
  • Combining: combineLatest (live, seed with startWith), forkJoin (parallel one-shots only), merge (any event), withLatestFrom (drive + snapshot); static view-models now belong to computed().
  • The recipes — typeahead, polling, armored save — are assembled answers to nameable failure modes; review code by diffing against them, and first() live streams before using them in event flows.
