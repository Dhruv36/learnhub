NgRx State Management

What you'll master: the Redux architecture as NgRx implements it — actions, reducers, selectors, effects — the unidirectional loop that justifies the ceremony, entity patterns, and the modern middle path (@ngrx/signals SignalStore). Prerequisites: State Patterns (the escalation criteria) and RxJS Patterns (effects are operator pipelines). You'll meet NgRx in most large Angular codebases — fluency is required even where you'd choose lighter machinery.

Part 1 — The Architecture: One Loop, Strictly One-Way

          dispatch(action)                    "SOMETHING HAPPENED"
component ────────────────▶ ACTION ─────┬─▶ REDUCER: (state, action) => newState
   ▲                                    │      pure. THE ONLY WAY STATE CHANGES.
   │                                    │
   │  selectors (memoized queries)      └─▶ EFFECT: action$ pipeline → side effects
   └────────────── STORE ◀──────────────────  (HTTP, storage) → dispatches RESULT actions
                (single state tree)

Everything is one of four roles:
  ACTION    a described event:        { type: '[Cart] Add Item', product }
  REDUCER   pure state transition:    how each action changes the tree
  SELECTOR  memoized read:            derive views of the tree
  EFFECT    the impure world:         listen for actions, do I/O, dispatch results

The payoff of forcing EVERYTHING through this loop: every change is a logged,
serializable event. DevTools shows the action history; time-travel replays it;
any state is reproducible as "initial + these 47 actions".

NgRx is Redux (from the React track) fused with RxJS and DI. The discipline — state changes only via dispatched actions through pure reducers — is exactly what the ceremony buys: an event log. When a bug report says "the cart emptied itself", DevTools shows precisely which action did it and what dispatched it — debugging becomes reading history instead of reproducing mysteries. The previous lesson's escalation criteria are really asking: is that log worth the indirection for this app? This lesson teaches the machinery so you can answer — and maintain the codebases that answered yes.

Part 2 — Actions & Reducers

// actions — createActionGroup, events named as FACTS in past/imperative domain language:
export const CartActions = createActionGroup({
  source: 'Cart',
  events: {
    'Item Added':      props<{ product: Product; qty: number }>(),
    'Item Removed':    props<{ productId: number }>(),
    'Checkout Requested': emptyProps(),
    'Checkout Succeeded': props<{ orderId: string }>(),
    'Checkout Failed':    props<{ error: string }>(),
  },
});

// reducer — pure transitions, immutable updates (the discipline you already have):
export interface CartState { items: CartItem[]; status: 'idle' | 'placing' | 'placed' | 'error'; }
const initialState: CartState = { items: [], status: 'idle' };

export const cartReducer = createReducer(
  initialState,
  on(CartActions.itemAdded, (state, { product, qty }) => ({
    ...state,
    items: upsertItem(state.items, product, qty),        // pure helper — testable alone
  })),
  on(CartActions.itemRemoved, (state, { productId }) => ({
    ...state, items: state.items.filter(i => i.productId !== productId),
  })),
  on(CartActions.checkoutRequested, s => ({ ...s, status: 'placing' })),
  on(CartActions.checkoutSucceeded, s => ({ ...s, items: [], status: 'placed' })),
  on(CartActions.checkoutFailed,    s => ({ ...s, status: 'error' })),
);

// registration:  provideStore({ cart: cartReducer })  → state tree: { cart: CartState }

Actions are events, not commands — "Item Added", not "setItems" — a naming discipline with teeth: event-named actions describe what happened (many reducers/effects may react), while command-named actions smuggle imperative thinking back in and couple dispatchers to handlers. Reducers are pure functions built on the immutability you've practiced all track — and being pure, they test as trivially as any pure function: expect(cartReducer(state, CartActions.itemAdded({...}))).toEqual(...). Note the async status woven into state as a union — the Async<T> shape reappearing in reducer form.

Part 3 — Selectors: the Memoized Read Layer

// feature selector → slice; createSelector → derived, MEMOIZED views:
export const selectCart = createFeatureSelector<CartState>('cart');

export const selectItems = createSelector(selectCart, s => s.items);
export const selectCount = createSelector(selectItems,
  items => items.reduce((n, i) => n + i.qty, 0));
export const selectTotal = createSelector(selectItems,
  items => items.reduce((sum, i) => sum + i.price * i.qty, 0));
export const selectCheckoutVm = createSelector(       // compose selectors into view models
  selectCount, selectTotal, selectCart,
  (count, total, cart) => ({ count, total, placing: cart.status === 'placing' }));

// consumption — modern: signals
export class CartBadgeComponent {
  private store = inject(Store);
  count = this.store.selectSignal(selectCount);       // Signal<number> — template: count()
}
// classic: observables — vm$ = this.store.select(selectCheckoutVm) + async pipe

// Memoization: selectTotal recomputes ONLY when selectItems' result changes reference —
// unrelated dispatches (theme toggled) don't touch it. computed()'s equivalent, Redux-side.

Selectors are NgRx's computed(): memoized derivations composed into exactly the view each component needs — components never know the state tree's shape, which is what lets reducers refactor freely. selectSignal is the modern consumption bridge: the store exposes signals, so templates read count() and signal-driven change detection applies — NgRx apps get the same precise rendering as signal stores. The architecture rule mirrors the store lesson: keep state minimal, derive in selectors — a state tree caching derivable values has the same desync disease as a store with stored totals.

Part 4 — Effects: Side Effects as Action Pipelines

// An effect: listen for actions → do impure work → dispatch result actions:
export const checkoutEffect = createEffect(
  (actions$ = inject(Actions), api = inject(OrderApi)) =>
    actions$.pipe(
      ofType(CartActions.checkoutRequested),
      concatLatestFrom(() => inject(Store).select(selectItems)),   // current cart snapshot
      exhaustMap(([, items]) =>                                    // ← rxjs-patterns choice!
        api.placeOrder(items).pipe(
          map(order => CartActions.checkoutSucceeded({ orderId: order.id })),
          catchError(err => of(CartActions.checkoutFailed({ error: err.message }))),
        )),
    ),
  { functional: true },
);
// registration: provideEffects({ checkoutEffect })

// EVERY rxjs-patterns rule applies verbatim:
//   exhaustMap on the checkout (double-submit protection)
//   switchMap on search effects (stale cancellation)
//   catchError INSIDE the flattener — or ONE error kills the effect stream FOREVER
//   (the "search stopped working" bug, NgRx edition — the #1 production NgRx incident)

// And effects can react to effects: checkoutSucceeded → navigation effect → router.navigate.
// That fan-out (one event, N reactions, none knowing the others) is THE effects payoff.

Effects quarantine the impure world: components dispatch facts, effects translate facts into I/O and translate outcomes back into facts, reducers fold facts into state. Every flattener-choice and error-placement rule transfers unchanged — an effect is just a long-lived pipeline, and the un-caught inner error that silently kills it is the most common NgRx production bug. The architectural win over service methods appears at fan-out: "order placed" triggering navigation + analytics + cart-clearing + toast as four independent effects, each added without touching the others — the observer pattern at application scale.

Part 5 — Entities & the Feature Bundle

// @ngrx/entity — normalized collections without the boilerplate:
export const productAdapter = createEntityAdapter<Product>();
// state shape: { ids: number[], entities: { [id]: Product } }   ← O(1) lookup, no dupes

const productReducer = createReducer(
  productAdapter.getInitialState({ loaded: false }),
  on(ProductActions.loadSucceeded, (s, { products }) =>
    productAdapter.setAll(products, { ...s, loaded: true })),
  on(ProductActions.updated, (s, { update }) => productAdapter.updateOne(update, s)),
  on(ProductActions.deleted, (s, { id }) => productAdapter.removeOne(id, s)),
);
const { selectAll, selectEntities } = productAdapter.getSelectors(selectProductState);

// The modern feature bundle — createFeature kills the selector boilerplate:
export const cartFeature = createFeature({
  name: 'cart',
  reducer: cartReducer,
});   // auto-generates selectCartState, selectItems, selectStatus …
provideState(cartFeature);            // lazy-feature registration (guards-lazy providers!)

@ngrx/entity standardizes the normalized-collection shape every CRUD feature needs (id-keyed map + ids array — the same normalization argument as database design: one copy per entity, O(1) updates, no list-vs-detail duplication), with adapter methods replacing hand-written spread gymnastics. createFeature plus provideState/provideEffects in route-level providers completes the modern shape: lazy features register their state slice when they load — the state tree grows with navigation, mirroring the bundle map.

Part 6 — Deep Dive: SignalStore, the Middle Path — and Reading NgRx Codebases

// @ngrx/signals — the store lesson's pattern with library structure (no actions/reducers):
export const BooksStore = signalStore(
  { providedIn: 'root' },
  withState({ books: [] as Book[], filter: '' }),
  withComputed(({ books, filter }) => ({
    filtered: computed(() => books().filter(b => b.title.includes(filter()))),
    count: computed(() => books().length),
  })),
  withMethods((store, api = inject(BookApi)) => ({
    setFilter(filter: string) { patchState(store, { filter }); },
    async load() {
      const books = await firstValueFrom(api.list());
      patchState(store, { books });
    },
  })),
);
// + withEntities() for adapter-style collections, rxMethod() for stream-driven methods.
// Same mental model as your hand-rolled stores; adds conventions, entity helpers, devtools.

// READING a full-NgRx codebase (the skill you'll actually use first):
//   start from the ACTION — grep its name → the reducer `on()` (state change) and the
//   effects ofType() (side effects) are the complete behavior, by construction.
//   That greppability IS the architecture: nothing happens off the books.

@ngrx/signals is the official acknowledgment that most features don't need the action log: it packages the signal-store pattern with structure (withState/withComputed/withMethods), entity helpers, and ecosystem integration — the escalation ladder's step 2, from the NgRx team itself. For full NgRx, the reading protocol in the code block is the day-one skill: because all behavior routes through actions, grep the action type and you've found every consequence — the indirection that costs writing speed pays it back as navigability, which is precisely the org-scale trade.

Senior framing: NgRx interviews sort candidates by whether they understand the why behind each role: actions as events (can you explain why command-naming is an anti-pattern?), reducers pure (what does purity buy? — time-travel, replayability, trivial tests), selectors memoized (what's the performance model?), effects isolated (which flattener for checkout vs search — and what happens to an effect whose inner error escapes?). The architecture question — "would you use NgRx here?" — wants the state-management lesson's escalation framework, then per-domain nuance: full Store for the collaborative-audit-heavy document module, SignalStore for the cart, router for the filters. And the maintenance reality: most NgRx work is reading — action-grep navigation, DevTools replay of bug reports, and resisting the entropy of "just this once" state mutated outside the loop (which silently breaks time-travel and is why strictStateImmutability runtime checks stay on in dev).

Interview Questions

Q1 (Beginner): Walk one user click through the full NgRx loop.

User clicks "Add to cart" → component dispatches CartActions.itemAdded({product, qty}) — a described event, no logic. The store runs every registered reducer; the cart reducer's matching on() returns a new state object with the item upserted (pure, immutable). Selectors reading affected slices recompute (memoized); components consuming them via selectSignal/async pipe re-render precisely. If effects listen for that action (e.g. persist-cart), they run side effects and dispatch result actions, which loop through reducers again. One direction, every step inspectable in DevTools.

Q2 (Beginner): Why must reducers be pure, concretely?

Purity (same state+action → same result, no side effects, no mutation) is what makes the action log meaningful: any past state can be reproduced by replaying actions (time-travel debugging), tests are input/output assertions with no mocks, and memoized selectors can trust reference equality to detect change. A reducer that mutates state or calls an API breaks all three — DevTools shows states that replay differently, and selector caching serves stale views. NgRx's dev-mode runtime checks (strictStateImmutability) exist because these breakages are silent otherwise.

Q3 (Mid): The products feature's search effect stopped responding after one failed request — everything else works. Diagnose from the architecture.

The NgRx edition of the terminated-stream bug: the effect's inner HTTP observable errored, and catchError sat on the outer action pipeline (or was missing) — the error completed the effect's subscription, so subsequent search actions dispatch into a dead stream. Fix: catch inside the flattener and emit a failure action — switchMap(q => api.search(q).pipe(map(ok), catchError(e => of(searchFailed(e))))) — so each request's failure is one dispatched fact and the pipeline survives. Same root cause and cure as RxJS Patterns Q5; NgRx just relocates where the bug hides. Bonus check: the failure action should exist and be reduced (status: 'error') so the UI says so.

Q4 (Mid): What do selectors add over reading state directly, and how does their memoization interact with immutable updates?

Selectors add: decoupling (components ask questions — selectCheckoutVm — without knowing tree shape, so reducers refactor freely), derivation (minimal state, computed views — the derive-don't-store rule), composition (view models from smaller selectors), and memoization (recompute only when inputs change). Memoization works because reducers update immutably: unchanged branches keep reference identity, so a selector whose input slice is === to last time returns its cached result without recomputation — and components skip re-rendering. Mutate state and both guarantees silently die — the two disciplines are one system.

Q5 (Senior): Compare full NgRx, @ngrx/signals SignalStore, and plain signal services for a mid-size app — and give a concrete allocation across one app's features.

Plain signal services: minimal ceremony, logic readable in-place, ideal for a few self-contained domains — no event log, conventions are yours to enforce. SignalStore: same signal mental model with library structure (withState/withComputed/withMethods, entities, devtools) — consistency across teams without action indirection. Full NgRx: the action log — time-travel, fan-out effects, enforced update discipline — at the cost of per-feature boilerplate and onboarding. Allocation for a shop app: filters/pagination → router (no store at all); product catalog → httpResource or a SignalStore with entities; cart/session → SignalStore (or plain services); order-processing with audit requirements and multi-system side effects → full NgRx feature (the fan-out and log genuinely pay). The senior signal is allocating per-domain rather than picking one answer — and noting the public surface (signals + methods/dispatch) stays similar enough to migrate between levels.

Q6 (Senior): Design the NgRx flow for checkout: double-submit-proof, current cart included, success navigates + clears + tracks analytics, failure shows a retryable error. Name every piece.

Actions: Checkout Requested (from the button), Checkout Succeeded {orderId}, Checkout Failed {error}. Reducer: requested → status: 'placing' (button disables off a selector), succeeded → clear items + status: 'placed', failed → status: 'error' (banner + retry button re-dispatches requested). Checkout effect: ofType(checkoutRequested) → concatLatestFrom(selectItems) (cart snapshot from the store, not the component) → exhaustMap (double-submits ignored while placing — the flattener is the double-charge protection, plus an Idempotency-Key header at the API layer) → map/catchError inside to result actions. Fan-out on success as three independent effects: navigation (ofType(succeeded) → router.navigate(['/confirm', orderId]), non-dispatching), analytics (tap(track), non-dispatching), and nothing else touches them. Tests: reducer transitions as pure assertions; effect with a marble/mock Actions stream verifying exhaust semantics and both outcome actions. Every requirement maps to exactly one role — which is the argument for NgRx on flows this cross-cutting.

🏋️ Exercises

1 (Easy): Define the actions and reducer for a counter feature (increment, decrement by amount, reset) and write two reducer unit tests.
Solution
export const CounterActions = createActionGroup({
  source: 'Counter',
  events: {
    'Incremented': emptyProps(),
    'Decremented By': props<{ amount: number }>(),
    'Reset': emptyProps(),
  },
});

export const counterReducer = createReducer(
  { value: 0 },
  on(CounterActions.incremented,  s => ({ value: s.value + 1 })),
  on(CounterActions.decrementedBy, (s, { amount }) => ({ value: s.value - amount })),
  on(CounterActions.reset, () => ({ value: 0 })),
);

// tests — pure function, no TestBed:
it('decrements by the given amount', () => {
  expect(counterReducer({ value: 10 }, CounterActions.decrementedBy({ amount: 3 })))
    .toEqual({ value: 7 });
});
it('ignores unknown actions', () => {
  const state = { value: 5 };
  expect(counterReducer(state, { type: 'Noise' } as any)).toBe(state);  // same reference!
});
2 (Easy): Write selectors for a todos slice: all, remaining count, percent done (guarding empty), and a footer view model — then consume the vm as a signal.
Solution
export const selectTodosState = createFeatureSelector<TodosState>('todos');
export const selectTodos = createSelector(selectTodosState, s => s.todos);
export const selectRemaining = createSelector(selectTodos,
  t => t.filter(x => !x.done).length);

  t => t.length === 0 ? 0 : Math.round(100 * t.filter(x => x.done).length / t.length));

export class TodoFooterComponent {

// Composition means percent/remaining each memoize off selectTodos — a dispatch that
// doesn't change todos recomputes nothing.
3 (Medium): Write the load-products effect with switchMap, inner error handling, and a loading status in the reducer — then explain why switchMap (not exhaustMap) fits loads.
Solution
export const loadProducts = createEffect(
  (actions$ = inject(Actions), api = inject(ProductApi)) =>
    actions$.pipe(
      ofType(ProductActions.loadRequested),
      switchMap(() => api.list().pipe(
        map(products => ProductActions.loadSucceeded({ products })),
        catchError(err => of(ProductActions.loadFailed({ error: err.message }))),  // INSIDE
      )),
    ),
  { functional: true },
);

const productReducer = createReducer(
  initialState,
  on(ProductActions.loadRequested, s => ({ ...s, status: 'loading' as const })),
  on(ProductActions.loadSucceeded, (s, { products }) =>
    ({ ...s, status: 'loaded' as const, products })),
  on(ProductActions.loadFailed, (s, { error }) => ({ ...s, status: 'error' as const, error })),
);
// switchMap: loads are READS — if a refresh is requested while one is in flight, only
// the LATEST matters; cancelling the stale request is free correctness. exhaustMap
// would IGNORE the refresh (stale wins) — right for submits, wrong for loads.

Solution

const reducer = createReducer(

5 (Hard): Rebuild the BooksStore from Part 6 as an @ngrx/signals SignalStore with entities and a stream-driven debounced search via rxMethod. Note where each rxjs-patterns idea reappears.
Solution
export const BooksStore = signalStore(
  { providedIn: 'root' },
  withEntities<Book>(),
  withState({ query: '', loading: false }),
  withComputed(({ entities, query }) => ({
    filteredCount: computed(() => entities().length),
    header: computed(() => query() ? `Results for "${query()}"` : 'All books'),
  })),
  withMethods((store, api = inject(BookApi)) => ({
    search: rxMethod<string>(pipe(                    // a stream-driven method:
      debounceTime(300),                              // ← typeahead debounce
      distinctUntilChanged(),                         // ← duplicate suppression
      tap(query => patchState(store, { query, loading: true })),
      switchMap(query => api.search(query).pipe(      // ← cancel stale searches
        tapResponse({
          next: books => patchState(store, setAllEntities(books), { loading: false }),
          error: () => patchState(store, { loading: false }),   // ← inner error handling:
        }),                                                     //   stream survives
      )),
    )),
  })),
);
// Component: store = inject(BooksStore);
//   <input (input)="store.search($any($event.target).value)">
//   @for (b of store.entities(); track b.id) { … }
// Every rxjs-patterns piece — debounce, distinct, switchMap, survive-the-error — appears
// verbatim; SignalStore just gives it a structured home with signal outputs.

Solution
// Investigation with Redux DevTools:
// 1. Reproduce (or load the user's exported action log) and READ THE HISTORY around
//    checkout: expect Checkout Requested → Checkout Succeeded → (navigation).
// 2. Inspect state AFTER Checkout Succeeded: does cart.items === []?

// Likely defect A — the reducer: Checkout Succeeded's on() sets status but FORGOT to
//    clear items (or a later action re-hydrates them from persistence). DevTools shows
//    items intact in the post-action state diff → fix the transition, add the reducer
//    test that was missing:
//    expect(cartReducer(placing, checkoutSucceeded({orderId})).items).toEqual([]).
//
// Likely defect B — a race between effects: navigation happens on Checkout REQUESTED
//    (someone wired the nav effect to the wrong action, or the component navigates
//    imperatively before the result), so success lands after render; or a persistence
//    effect (localStorage rehydrate on route change) dispatches Cart Restored with the
//    OLD snapshot after the clear. DevTools' timeline makes either visible as an
//    action ordered after Checkout Succeeded that re-populates items.
//
// The meta-lesson: because every change is a logged action, the bug is FINDABLE by
// reading history — the exact capability the NgRx ceremony was purchased for. If the
// badge desynced with NO action in the log, that's mutation outside the loop —
// re-enable strictStateImmutability/strictActionImmutability and hunt the offender.

🎯 Key Takeaways

  • NgRx = Redux + RxJS + DI: dispatch event-named actions → pure reducers transition a single tree → memoized selectors project views → effects quarantine I/O and dispatch results. The purchased capability is the action log (DevTools, time-travel, greppability).
  • Actions are events, not commands; reducers stay pure and immutable (time-travel and selector memoization depend on it); state stays minimal with derivation in selectors; selectSignal bridges to signal-driven rendering.
  • Effects are rxjs-patterns pipelines: right flattener per flow (exhaust=checkout, switch=search), catchError inside the flattener dispatching failure actions — or the effect dies silently.
  • @ngrx/entity normalizes collections ({ids, entities} — O(1) updates, no duplicates); createFeature + route-level provideState/provideEffects = lazy state slices.
  • @ngrx/signals SignalStore is the official middle path — the signal-store pattern with structure, entities, and rxMethod, no action ceremony.
  • Read NgRx by grepping actions (reducer + effects = complete behavior); adopt it per-domain on the escalation criteria — the log must earn the boilerplate.
