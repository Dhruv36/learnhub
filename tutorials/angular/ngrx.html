NgRx State Management Advanced

As an app grows, state shared across many unrelated components (the current user, a shopping cart, a notification list) becomes hard to reason about if every component/service mutates it ad hoc. NgRx brings the Redux pattern to Angular: a single, predictable store updated only through explicit, traceable actions. It's a significant complexity investment — knowing WHEN to reach for it matters as much as knowing HOW.

The Core Redux Loop

Component dispatches an ACTION  →  a REDUCER computes a NEW state from (old state, action)  →
Store updates  →  SELECTORS read slices of state  →  Components re-render
// One-way data flow, always. State is NEVER mutated directly — only replaced via pure reducer functions.
// This is what makes NgRx state fully traceable: every change is an explicit, named, logged action.

Actions, Reducers, Selectors

// Actions — plain, typed descriptions of "what happened":
export const addItem = createAction('[Cart] Add Item', props<{ item: Item }>());
export const removeItem = createAction('[Cart] Remove Item', props<{ id: string }>());

// Reducer — a PURE function: (state, action) => newState. Never mutate `state` directly.
export const cartReducer = createReducer(
  initialState,
  on(addItem, (state, { item }) => ({ ...state, items: [...state.items, item] })),
  on(removeItem, (state, { id }) => ({ ...state, items: state.items.filter(i => i.id !== id) })),
);

// Selectors — derive/read data from the store, memoized like computed():
export const selectCartItems = createSelector(selectCartState, state => state.items);
export const selectCartTotal = createSelector(selectCartItems, items => items.reduce((s, i) => s + i.price, 0));

// In a component:
private store = inject(Store);
items = this.store.selectSignal(selectCartItems);       // read as a signal
add(item: Item) { this.store.dispatch(addItem({ item })); }   // the ONLY way to change state
🔒 Immutability + pure reducers = time-travel debugging and predictability. Because reducers are pure functions and state is never mutated, every state transition is a reproducible snapshot. Redux DevTools can replay the exact sequence of actions that led to any bug — "click undo on this action and see the previous state" — something impossible with scattered, mutable, ad hoc state. This traceability is NgRx's core value proposition.

Effects — Side Effects Live Outside Reducers

// Reducers MUST be pure (no HTTP calls, no side effects) — Effects handle async work:
export const loadCart$ = createEffect((actions$ = inject(Actions), api = inject(CartApi)) =>
  actions$.pipe(
    ofType(loadCart),
    switchMap(() => api.getCart().pipe(
      map(items => loadCartSuccess({ items })),
      catchError(err => of(loadCartFailure({ error: err.message })))
    ))
  ), { functional: true }
);
// An Effect listens for an action, performs a side effect (HTTP call), and dispatches a NEW action
// with the result. This keeps the reducer pure while still handling real-world async work.

Do You Need NgRx? (the most important judgment call)

SignalLean toward
State is local to one component/featureComponent signals — no global store needed
A few services share simple stateA plain injectable service with signals (providedIn: 'root')
Many unrelated features read/write the SAME stateNgRx — centralized, traceable state
You need time-travel debugging / strict audit of state changesNgRx (Redux DevTools)
Team needs enforced consistency across many contributorsNgRx's structure pays off at scale
⚖️ NgRx is a scaling tool, not a default. Signals + a few well-scoped services (services-DI lesson) handle the majority of real apps perfectly well. Reaching for NgRx on a small app adds boilerplate (actions, reducers, selectors, effects for every feature) without a matching payoff. The interview-savvy answer is knowing the trade-off, not reciting "always use NgRx."

Interview Questions

Q1: Explain the core NgRx data flow — actions, reducers, and selectors.

NgRx implements the Redux pattern: application state lives in a single, centralized store, and it can ONLY be changed through a strict one-way flow. A component (or effect) dispatches an ACTION — a plain, typed object describing "what happened" (e.g., [Cart] Add Item with a payload). The store routes that action to REDUCERS, which are pure functions taking the current state and the action and returning a brand-new state object — reducers never mutate the existing state, they always construct a new one (typically via object/array spreads), which is essential for both change detection and NgRx's tooling. Components then read data out of the store via SELECTORS, which are memoized functions that derive a specific slice or computed value from state (analogous to computed() signals) — selectors are composed from smaller selectors and only recompute when their inputs actually change. This unidirectional loop (dispatch → reduce → select → render) makes every state change explicit, named, and traceable, which is the entire point of adopting Redux-style state management over ad hoc mutation.

Q2: Why must reducers be pure, and how do you handle asynchronous side effects like HTTP calls?

A reducer's whole contract is (state, action) => newState with no side effects — no HTTP calls, no random values, no direct DOM/timer access — because NgRx's tooling and guarantees depend on that purity. Redux DevTools' time-travel debugging works by replaying a recorded sequence of actions through the reducers to reconstruct any past state; if a reducer performed an HTTP call or had other side effects, replaying it wouldn't be deterministic or safe, breaking that guarantee. Purity also makes reducers trivially unit-testable (call the function, assert on the output) and keeps state transitions predictable and free of race conditions caused by network timing. Since real applications obviously need to make HTTP calls and other async work, NgRx separates that into EFFECTS: an Effect listens for a specific action (via ofType), performs the actual side effect (typically an HTTP request via the injected API service, flattened with a suitable RxJS operator like switchMap), and then dispatches a NEW action carrying the result (a success or failure action) back into the store, which a normal pure reducer then handles. This keeps the reducer pure while cleanly isolating all async/impure work in one well-defined layer.

Q3: When would you introduce NgRx into an Angular app, versus just using signals and services?

NgRx is a genuine complexity investment — actions, reducers, selectors, and effects for every feature add real boilerplate — so it should be adopted deliberately, not by default. For state that's local to one component or feature, component-level signals are simplest and sufficient. For state shared across a modest number of related services/components (say, a cart or a current-user profile), a plain @Injectable({ providedIn: 'root' }) service holding signals is usually enough, and it's far less ceremony than NgRx while still centralizing the logic. NgRx starts paying off once an app has MANY largely-unrelated features reading and writing the SAME shared state, where ad hoc mutation from many different places becomes hard to trace and reason about, or when a team specifically needs the traceability/audit trail NgRx provides — every state change as a named, logged action, replayable via Redux DevTools for debugging, which is invaluable at scale with many contributors. The interview-savvy framing isn't "always/never use NgRx" but recognizing it as a scaling tool for state complexity and team coordination, reached for once the simpler signals-based approach genuinely starts straining — not reflexively at project start.

🏋️ Practice: build a small cart feature with createAction/createReducer/createSelector and dispatch actions from a component; add an Effect that loads cart data via HTTP and dispatches success/failure actions; open Redux DevTools and time-travel through a sequence of dispatched actions; compose two selectors (items → total) and confirm memoization by logging recomputation; refactor the same feature using just a signals-based service and compare the code volume/tradeoffs.

🎯 Key Takeaways

  • NgRx = Redux for Angular: dispatch an action → a pure reducer computes new state → selectors read slices → components re-render (one-way flow).
  • Reducers must be pure (no side effects, no mutation) — this is what enables Redux DevTools' time-travel debugging.
  • Effects handle async work (HTTP) outside reducers: listen for an action, perform the side effect, dispatch a result action.
  • Selectors are memoized and composable, analogous to computed() signals.
  • NgRx is a scaling tool for widely-shared, complex state — not a default; component signals and simple injectable services handle most apps well.
