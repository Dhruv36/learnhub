HttpClient & Interceptors Intermediate

Angular's HttpClient is the standard way to talk to a backend — every method returns an Observable, which is why understanding RxJS (next lesson) matters so much for real-world Angular apps. Interceptors let you hook into every request/response centrally — auth headers, error handling, logging — without touching every call site.

Making Requests — Everything Is an Observable

import { HttpClient } from '@angular/common/http';

export class ProductService {
  private http = inject(HttpClient);

  getProducts(): Observable<Product[]> {
    return this.http.get<Product[]>('/api/products');   // returns COLD Observable — does nothing until subscribed
  }

  createProduct(p: Product) {
    return this.http.post<Product>('/api/products', p);
  }
}
// Nothing happens until something SUBSCRIBES — the async pipe in a template, or .subscribe() in code,
// or toSignal() to bridge it into a signal. An unsubscribed HTTP Observable never fires the request.
📡 HttpClient requests are lazy and Observable-based, not Promise-based. Calling http.get() builds a request description but fires NOTHING until subscribed — this is different from fetch()/axios, which fire immediately. That laziness is what lets you retry, cancel (unsubscribe), race, or combine requests with RxJS operators before they ever hit the network — powerful, but a classic gotcha if you expect it to behave like a Promise.

Interceptors — Central Request/Response Hooks

import { HttpInterceptorFn } from '@angular/common/http';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const auth = inject(AuthService);
  const cloned = req.clone({ setHeaders: { Authorization: `Bearer ${auth.token()}` } });
  return next(cloned);         // pass the (possibly modified) request along the chain
};

export const errorInterceptor: HttpInterceptorFn = (req, next) =>
  next(req).pipe(
    catchError(err => {
      if (err.status === 401) inject(Router).navigate(['/login']);
      return throwError(() => err);
    })
  );

// Registered once, applied to EVERY request:
provideHttpClient(withInterceptors([authInterceptor, errorInterceptor]));
🔗 Interceptors form a chain — think Express/Node middleware for HTTP calls. Each interceptor can inspect/modify the outgoing request, call next() to continue the chain, and inspect/modify the response (or error) on the way back. This is THE place to add auth tokens, centralize error handling (redirect on 401, toast on 500), add logging, or handle retries — once, for every request, instead of repeating that logic at every call site.

Requests Are Immutable — req.clone()

// HttpRequest objects are IMMUTABLE. You can't do `req.headers = ...` — you must clone with changes:
const cloned = req.clone({ setHeaders: { 'X-Trace-Id': traceId } });
// This mirrors the immutable-update discipline from signals — predictable, and safe to reuse/retry
// the original request object without side effects leaking between interceptors.

Common HTTP Patterns with RxJS

// switchMap: cancel the previous request when a new one starts (e.g., typeahead search)
searchTerm$.pipe(
  debounceTime(300),
  switchMap(term => this.http.get<Result[]>(`/api/search?q=${term}`))
).subscribe(results => ...);

// Bridge to a signal for template use:
results = toSignal(this.searchResults$, { initialValue: [] });

Interview Questions

Q1: Why does HttpClient return Observables instead of Promises, and what's the practical implication?

HttpClient methods return Observables because HTTP requests, modeled this way, are COLD and LAZY: an Observable returned by http.get() is just a description of a request that fires nothing on its own — the actual network call only happens once something subscribes to it, whether that's an explicit .subscribe() call, the async pipe in a template, or bridging it to a signal with toSignal(). This is fundamentally different from a Promise (or fetch/axios), which begins executing the moment it's created. The practical payoff is that Observables compose with the full RxJS operator toolkit before the request ever touches the network: you can retry a failed request (retry()), cancel an in-flight one simply by unsubscribing, race multiple sources, or — most commonly — use switchMap to automatically cancel a stale request when a newer one supersedes it (as in a search-as-you-type box). The gotcha this creates is that developers coming from Promise-based HTTP clients sometimes call http.get() and expect it to "just run" — if nothing subscribes (say, you forget the async pipe or a .subscribe()), the request silently never fires.

Q2: What are HTTP interceptors and what are they used for?

An interceptor is a function (HttpInterceptorFn) registered globally via provideHttpClient(withInterceptors([...])) that sits in the middle of every outgoing HTTP request and incoming response, forming a chain much like server-side middleware. Each interceptor receives the request and a next function representing the rest of the chain; it can inspect or modify the outgoing request (commonly cloning it to add an Authorization header, a trace/correlation ID, or a base URL) before calling next(req) to continue, and it can also operate on the returned Observable to inspect or transform the response or error on the way back (for example, using catchError to detect a 401 and redirect to a login page, or to show a global error toast on 5xx responses). The value is centralization: instead of repeating "attach the auth token" or "handle a 401" logic at every single call site across the app — which is easy to forget on new endpoints — you write that logic once as an interceptor and it automatically applies to every request the app makes. Common uses include authentication, centralized error handling, logging/telemetry, caching, and retry logic.

Q3: In a typeahead search feature, why use switchMap instead of mergeMap for the HTTP call?

Both operators map each emission from a source stream (the search-term input, typically after debounceTime) to an inner Observable (the HTTP request) and flatten the result, but they differ in how they handle overlapping inner Observables. mergeMap runs all inner Observables concurrently and emits results as each one resolves, with no cancellation — so if a user types "a", then quickly types "ab", TWO requests are in flight, and if the "a" request happens to resolve AFTER the "ab" request (which is common with network latency variance), the UI ends up displaying results for the stale "a" query, overwriting the correct "ab" results — a classic race condition. switchMap instead CANCELS the previous inner Observable (unsubscribes from the in-flight request) the moment a new value arrives from the source, so only the most recent search term's request is allowed to complete and update the UI; anything from an outdated request is discarded. This makes switchMap the correct default for "only the latest request matters" scenarios like typeahead search, autocomplete, or any request-per-keystroke pattern, whereas mergeMap suits cases where you genuinely want all concurrent requests to complete and be processed (like firing off several independent uploads at once).

🏋️ Practice: build a ProductService with HttpClient methods and consume one via the async pipe; write an authInterceptor that clones requests to add a bearer token, and an errorInterceptor that redirects on 401; register both with withInterceptors; build a typeahead search using debounceTime + switchMap and compare behavior against mergeMap by simulating variable response latency; bridge a search-results Observable into a signal with toSignal() for template use.

🎯 Key Takeaways

  • HttpClient methods return cold, lazy Observables — nothing fires until subscribed (async pipe, .subscribe(), or toSignal()).
  • Interceptors (HttpInterceptorFn, registered via withInterceptors) form a middleware-like chain for every request/response — the place for auth, error handling, logging.
  • HttpRequest objects are immutable — modify via req.clone({...}), never by direct mutation.
  • Use switchMap for "only the latest request matters" (typeahead) to cancel stale in-flight requests and avoid race conditions; mergeMap runs all concurrently with no cancellation.
  • HTTP + RxJS composition (debounceTime, switchMap, catchError) is central to real-world Angular data fetching, bridging to signals with toSignal() for templates.
