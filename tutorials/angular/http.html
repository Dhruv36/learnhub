HttpClient & Interceptors Routing & HTTP

What you'll master: HttpClient's observable-based API and why its requests behave the way they do (lazy, one-shot, cold), typed requests and error handling as an architecture, functional interceptors — the middleware pipeline for auth/retry/logging — httpResource for signal-era data fetching, and the API-service layer pattern. Prerequisites: RxJS Patterns and Advanced DI (interceptors are DI-registered). HTTP is where your app meets reality; this lesson is how Angular apps industrialize it.

Part 1 — HttpClient: the API and Its Semantics

// app.config.ts:  providers: [provideHttpClient(withInterceptors([...]))]

export class ProductApi {
  private http = inject(HttpClient);

  list(page: number)        { return this.http.get<Product[]>('/api/products',
                                { params: { page } }); }          // typed + query params
  byId(id: number)          { return this.http.get<Product>(`/api/products/${id}`); }
  create(dto: NewProduct)   { return this.http.post<Product>('/api/products', dto); }
  update(p: Product)        { return this.http.put<Product>(`/api/products/${p.id}`, p); }
  remove(id: number)        { return this.http.delete<void>(`/api/products/${id}`); }
}

// SEMANTICS you must know cold (all consequences of rxjs.html):
this.http.get<User>('/api/me');            // ← NOTHING happens. No subscribe, no request.
const req$ = this.http.get<User>('/api/me');
req$.subscribe(...); req$.subscribe(...);   // TWO requests (cold — producer per subscriber)
// Emits ONCE then completes → no unsubscribe ceremony needed for one-shots.
// Unsubscribing an in-flight request ABORTS it (XHR/fetch cancellation) — what
// switchMap exploits for stale-request cancellation.

HttpClient returns cold, one-shot observables, and all four practical behaviors follow from that: requests are recipes until subscribed (forgetting .subscribe() — or better, the async pipe/toSignal — is the classic "my POST never fires" bug); each subscription is its own request (the duplicate-fetch trap); completion after one emission means no leak management; and unsubscription is genuine cancellation. Note the generic parameter (get<Product[]>) is a compile-time assertion, not validation — the server can still send anything; runtime validation at trust boundaries (zod-style, per the TS lesson) is a separate, deliberate choice for critical payloads.

Part 2 — Error Handling: Status Codes Meet RxJS

// Failures arrive as HttpErrorResponse through the ERROR channel:
this.api.byId(42).pipe(
  catchError((err: HttpErrorResponse) => {
    if (err.status === 404) return of(null);              // expected: absence is a value
    if (err.status === 0)   this.toast.offline();         // network/CORS — no response at all
    return throwError(() => err);                          // unexpected: propagate
  }),
).subscribe(product => this.product.set(product));

// RETRY — transient-only, with backoff (the rxjs-patterns recipe, HTTP-tuned):
const isTransient = (e: HttpErrorResponse) => e.status === 0 || e.status >= 500;
this.http.get<Data>('/api/data').pipe(
  retry({ count: 2, delay: (e, n) => isTransient(e) ? timer(500 * 2 ** n) : throwError(() => e) }),
  catchError(...)
)
// NEVER blanket-retry: retrying a 400 re-sends garbage; retrying a POST can DUPLICATE
// the action (only retry non-idempotent requests with an Idempotency-Key — see the
// Spring Boot api-design lesson for the server half).

Error handling splits by who can act: expected domain outcomes (404 → show not-found, 409 → merge conflict UI) are handled where the feature knows what they mean; transport-level noise (0, 5xx) gets retried-with-backoff then surfaced generically; and cross-cutting reactions (401 → re-login, 403 → forbidden page) belong in an interceptor (Part 3) so they're handled once. The transient-only retry predicate and the idempotency caveat are the two production details that separate this from tutorial retry code — the full client-server contract (status code taxonomy, RFC 7807 error bodies you can map into forms per the validation lesson) is a shared design artifact, not a client afterthought.

Part 3 — Functional Interceptors: the Middleware Pipeline

// An interceptor sees EVERY request out and EVERY response in:
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(TokenStore).token();
  if (!token || req.url.startsWith('/public')) return next(req);
  return next(req.clone({ setHeaders: { Authorization: `Bearer ${token}` } }));
  //          ^^^^^^^^^ requests are IMMUTABLE — clone to modify
};

export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  const router = inject(Router);
  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      if (err.status === 401) router.navigateByUrl('/login');      // once, for the whole app
      if (err.status >= 500) inject(TelemetryService).report(err);
      return throwError(() => err);                                // features still see it
    }),
  );
};

export const timingInterceptor: HttpInterceptorFn = (req, next) => {
  const started = performance.now();
  return next(req).pipe(
    finalize(() => console.debug(`${req.method} ${req.urlWithParams}:
                                  ${Math.round(performance.now() - started)}ms`)),
  );
};

// Registration = ORDER: requests flow down the list, responses bubble back up:
provideHttpClient(withInterceptors([authInterceptor, timingInterceptor, errorInterceptor]))
//  request:  auth → timing → error → SERVER
//  response: SERVER → error → timing → auth

Interceptors are Angular's HTTP middleware — the same onion model as Express middleware or Spring's filters: each wraps next(req), transforming the request going down and the response/error stream coming back up. They solve the "every request needs X" class once: auth headers, correlation IDs, global error policy, timing/telemetry, loading-bar counters, caching. Two mechanics to internalize: requests are immutable (clone() or your mutation silently vanishes), and ordering is meaningful (auth before logging if logs should show the header; error-mapping innermost if others should see mapped errors). Functional interceptors + withInterceptors are the modern form of the old class-based HTTP_INTERCEPTORS multi-provider — which you now recognize as a multi-token plugin system.

Part 4 — httpResource & Signal-Era Data Fetching

// httpResource (v19.2+): reactive GET-as-signal — url derives from signals, refetches on change:
export class ProductDetailComponent {
  id = input.required<string>();                              // route param (routing.html)
  product = httpResource<Product>(() => `/api/products/${this.id()}`);
  // id changes → in-flight request ABORTED → new request. switchMap semantics, declarative.
}
// template:
// @if (product.isLoading()) { <app-skeleton /> }
// @else if (product.error()) { <app-error [error]="product.error()" /> }
// @else { <h1>{{ product.value()?.name }}</h1> }
// plus: product.reload()   — manual refresh for free

// With params object form:
users = httpResource<User[]>(() => ({
  url: '/api/users',
  params: { q: this.query(), page: this.page() },
}));

// The decision matrix:
//   GET driven by reactive state (route/query/filters)  → httpResource
//   mutations (POST/PUT/DELETE)                          → HttpClient + subscribe/exhaustMap
//   streams needing operators (retry, polling, combine)  → HttpClient + RxJS → toSignal
// Interceptors apply to ALL of them — httpResource rides the same pipeline.

httpResource packages the read-side pattern this track has been assembling piece by piece — params-as-signals, switchMap-style stale-request cancellation, loading/error/value states, manual reload — into one declarative primitive (it's resource from the Signals lesson specialized for HTTP). It deliberately covers reads: mutations stay imperative (they're events, not derived state — often through the exhaustMap-armored save), and anything needing stream composition drops down to HttpClient + operators. Since interceptors sit below all of it, your auth/error/telemetry policy applies uniformly regardless of which consumption style a feature picked.

Part 5 — The API Service Layer: Where HTTP Lives

// The layering (services-di.html Part 6, now concrete):
//   components → stores/facades → API services → HttpClient → interceptors → network

@Injectable({ providedIn: 'root' })
export class OrderApi {                        // API service: HTTP in, domain types out
  private http = inject(HttpClient);
  private base = inject(API_URL);              // di-advanced.html token

  search(criteria: OrderCriteria): Observable<Page<Order>> {
    return this.http.get<Page<OrderDto>>(`${this.base}/orders`,
        { params: toHttpParams(criteria) })
      .pipe(map(page => ({ ...page, items: page.items.map(fromDto) })));  // DTO → domain
  }
  place(order: NewOrder): Observable<Order> {
    return this.http.post<OrderDto>(`${this.base}/orders`, toDto(order),
        { headers: { 'Idempotency-Key': crypto.randomUUID() } })
      .pipe(map(fromDto));
  }
}

// Why the layer exists:
//  - components/stores never see URLs, HttpParams, DTO shapes — swap/refactor freely
//  - DTO ↔ domain mapping in ONE place (fromDto/toDto) — API drift contained
//  - the mock seam for tests: { provide: OrderApi, useValue: fakeApi }
//  - request idioms (idempotency keys, pagination params) standardized per resource

Raw HttpClient calls scattered through components couple every view to URL structure, wire formats, and header conventions — the API service layer is the boundary that contains all of it. Each backend resource gets one injectable speaking domain vocabulary outward and HTTP inward, with DTO mapping localized so backend renames touch one file. This is also where client and server contracts meet: pagination shapes, idempotency keys, error formats — if your backend is the Spring Boot API-design lesson, this layer is its mirror image. Testing follows the seams: HttpTestingController verifies the API service's requests (Testing lesson); everything above it doubles the service.

Part 6 — Deep Dive: Progress, Caching, Security & the Full Pipeline

// UPLOAD PROGRESS — reportProgress + event stream:
upload(file: File) {
  const form = new FormData();
  form.append('file', file);
  return this.http.post('/api/files', form, {
    reportProgress: true, observe: 'events',
  }).pipe(map(event => {
    if (event.type === HttpEventType.UploadProgress)
      return { kind: 'progress', pct: Math.round(100 * event.loaded / (event.total ?? 1)) };
    if (event.type === HttpEventType.Response) return { kind: 'done', body: event.body };
    return { kind: 'other' };
  }));
}
// (FormData: do NOT set Content-Type yourself — the browser sets the multipart boundary)

// CACHING interceptor — GET memoization with TTL (shareReplay's HTTP cousin):
export const cacheInterceptor: HttpInterceptorFn = (req, next) => {
  if (req.method !== 'GET' || req.headers.has('x-no-cache')) return next(req);
  const cache = inject(HttpCacheService);
  const hit = cache.get(req.urlWithParams);
  if (hit) return of(hit.clone());
  return next(req).pipe(tap(evt => {
    if (evt instanceof HttpResponse) cache.set(req.urlWithParams, evt, 30_000);
  }));
};

// SECURITY notes that belong to HTTP:
//  - XSRF: provideHttpClient(withXsrfConfiguration(...)) — cookie→header echo for
//    cookie-authenticated APIs (token-in-header APIs don't need it)
//  - tokens: HttpOnly cookies beat localStorage (XSS can't read them) — js track security
//  - never build URLs by concatenating user input — HttpParams encodes safely

The long tail of HTTP work — progress UIs, client-side caching, CSRF, credential storage — all slots into the same two extension points (request options and interceptors), which is the architectural point of this lesson: the pipeline is programmable, so cross-cutting concerns become composable units instead of per-call boilerplate. The caching interceptor deserves its caveats: it's for read-mostly reference data (droplists, configs), it must be bypassable, and server cache headers + ETags are the more honest mechanism when you control the backend.

Senior framing: the HTTP review checklist that catches real incidents: every mutation triggered exactly once (no switchMap-on-writes, no double-subscribe — and exhaustMap or disabled buttons on user-triggered writes); retry limited to transient statuses on idempotent requests; 401/403 policy in one interceptor, not twelve components; API services as the only HttpClient consumers (lint-able); DTO mapping present (grep for components touching .data.attributes-style raw shapes); errors mapped to user-meaningful states, with a global handler as backstop, not primary UX; and interceptor order documented — auth/correlation outermost, error-mapping innermost. Interview-wise, the differentiator question is usually interceptors ("how would you add auth + global 401 handling?") — the answer that names immutable-clone, ordering, and the keep-propagating-after-handling detail (throwError re-throw) reads senior.

Interview Questions

Q1 (Beginner): Why does an HttpClient request not fire until subscribed, and why don't HTTP subscriptions need unsubscribe management?

HttpClient returns cold observables — the request is the producer, created per subscription (laziness): no subscriber, no XHR. This enables composition (retry re-subscribes, switchMap cancels) but means an unconsumed request is a no-op. No unsubscribe ceremony is needed because HTTP observables emit once and complete — completion runs teardown automatically, so there's nothing left to leak. (Unsubscribing early is still meaningful: it aborts the in-flight request.)

Q2 (Beginner): What is an interceptor, and what are three canonical uses?

A function in HttpClient's middleware pipeline — it receives every outgoing request and a next handler, and can transform the request (immutably, via clone) and the response/ error stream. Canonical uses: attaching auth headers (Bearer tokens) in one place; global error policy (401 → login redirect, 5xx → telemetry) handled once; and observability — timing, correlation IDs, logging, loading-bar counters. Registered via provideHttpClient(withInterceptors([...])), running in registration order outward, reverse order inward.

Q3 (Mid): Your error interceptor logs 500s and returns of(null). Feature teams complain their error handling "stopped working". Explain and fix.

Returning of(null) from catchError replaces the error with a successful null emission — downstream, every feature's catchError sees a normal value, so their 404 handling, form error mapping, and retry logic never trigger; worse, code expecting a body gets null and fails weirdly later. Interceptors should handle their cross-cutting slice and re-propagate: catchError(err => { telemetry.report(err); return throwError(() => err); }). Swallowing is only correct when the interceptor genuinely owns the outcome (e.g. returning a cached response) — global logging never does.

Q4 (Mid): When is httpResource the right tool vs HttpClient + RxJS, and what problem does it solve automatically?

httpResource fits reads driven by reactive state: the URL/params derive from signals (route param, filters), and it re-fetches on change while aborting the stale in-flight request — the switchMap race-condition cure, plus loading/error/value signals and reload(), all declaratively. Drop to HttpClient + RxJS for mutations (imperative events — exhaustMap-guarded saves), polling/retry pipelines, response-event streams (upload progress), or anything needing operator composition; convert to signals at the boundary with toSignal. Both ride the same interceptor pipeline, so cross-cutting policy is unaffected by the choice.

Q5 (Senior): Design token-refresh: access token expires mid-session; concurrent requests start failing with 401. What must the interceptor handle?

The refresh interceptor catches 401s (excluding the auth endpoints themselves), calls the refresh endpoint, then replays the original request with the new token (switchMap(() => next(retryReq))). The hard requirements: (1) single-flight — N concurrent 401s must trigger ONE refresh, with the other requests waiting on the same refresh observable (a shared in-progress subject/shareReplay), else you get a refresh stampede and token-rotation races; (2) failure path — refresh fails → clear session, redirect to login, propagate; (3) no infinite loops — a 401 from the retried request must not re-trigger refresh (flag the retry); (4) queued requests use the new token, not the captured old one. Alternative worth naming: proactive refresh on a timer before expiry, which simplifies the interceptor to the reactive fallback. This question is a de-facto Angular senior filter because it composes interceptors, RxJS sharing, and failure design in one.

Q6 (Senior): Where does client-side HTTP caching belong, and when is it the wrong answer?

Options, in order of preference: (1) HTTP-native — server cache headers (Cache-Control, ETag/304) let the browser and proxies cache with correct invalidation; prefer when you control the backend. (2) State-layer — the store/facade caches domain data with explicit staleness rules (refresh on focus, TTL) — right when "freshness" is a product decision. (3) A GET-caching interceptor — transparent, app-wide, good for reference data (configs, droplists) with short TTLs and a bypass header. Wrong answers: caching mutating requests' effects, caching per-user data in shared caches, interceptor caching as a substitute for fixing duplicate-subscription bugs (mask, not cure), and long-TTL caches without invalidation stories ("why is the UI a day old?"). The senior habit is naming the invalidation plan before the caching mechanism.

1 (Easy): Write a typed TodoApi (list with completed-filter param, add, toggle, remove) and consume list via httpResource in a component.
Solution
@Injectable({ providedIn: 'root' })
export class TodoApi {
  private http = inject(HttpClient);
  list(completed?: boolean) {
    return this.http.get<Todo[]>('/api/todos',
      { params: completed === undefined ? {} : { completed } });
  }
  add(title: string)   { return this.http.post<Todo>('/api/todos', { title }); }
  toggle(t: Todo)      { return this.http.put<Todo>(`/api/todos/${t.id}`,
                           { ...t, completed: !t.completed }); }
  remove(id: number)   { return this.http.delete<void>(`/api/todos/${id}`); }
}

export class TodoListComponent {
  filter = signal<boolean | undefined>(undefined);
  todos = httpResource<Todo[]>(() => ({
    url: '/api/todos',
    params: this.filter() === undefined ? {} : { completed: String(this.filter()) },
  }));
  // filter changes → stale request aborted → refetch; todos.reload() after mutations.
}

3 (Medium): Implement the global error interceptor: 401 → login redirect with returnUrl, 403 → /forbidden, ≥500 → telemetry + toast — while features keep full error visibility.
Solution
export const errorPolicyInterceptor: HttpInterceptorFn = (req, next) => {
  const router = inject(Router);
  const toast = inject(ToastService);
  const telemetry = inject(TelemetryService);
  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      switch (true) {
        case err.status === 401 && !req.url.includes('/auth/'):
          router.navigate(['/login'], { queryParams: { returnUrl: router.url } });
          break;
        case err.status === 403:
          router.navigateByUrl('/forbidden');
          break;
        case err.status >= 500 || err.status === 0:
          telemetry.report(err);
          toast.error('Something went wrong. We are on it.');
          break;
      }
      return throwError(() => err);        // ← ALWAYS re-throw: features still catch
    }),
  );
};
// The /auth/ exclusion prevents the login request's own 401 from redirect-looping.
// Features layer their specific handling (404 → empty state) on top, untouched.
4 (Medium): Build a loading-bar interceptor pair: a service counting in-flight requests (signal), an interceptor incrementing/decrementing via finalize, and the header bar reading it.
Solution
@Injectable({ providedIn: 'root' })
export class PendingRequestsService {
  private count = signal(0);
  readonly loading = computed(() => this.count() > 0);
  start() { this.count.update(n => n + 1); }
  end()   { this.count.update(n => Math.max(0, n - 1)); }
}

export const loadingInterceptor: HttpInterceptorFn = (req, next) => {
  if (req.headers.has('x-background')) return next(req);   // opt-out for polls
  const pending = inject(PendingRequestsService);
  pending.start();
  return next(req).pipe(finalize(() => pending.end()));    // success, error, OR abort
};

// header component:
// pending = inject(PendingRequestsService);
// @if (pending.loading()) { <div class="loading-bar"></div> }
// finalize is the crux: it runs on completion, error, AND unsubscribe (aborted
// requests) — a counter using tap(success)/error branches leaks "up" states.
5 (Hard): Implement an upload component with progress bar and cancel button using reportProgress — cancellation must actually abort the transfer.
Solution
export class UploaderComponent {
  private http = inject(HttpClient);
  progress = signal<number | null>(null);
  private sub?: Subscription;

  upload(file: File) {
    const form = new FormData();
    form.append('file', file);                       // no manual Content-Type!
    this.progress.set(0);
    this.sub = this.http.post('/api/files', form,
        { reportProgress: true, observe: 'events' })
      .subscribe({
        next: (event) => {
          if (event.type === HttpEventType.UploadProgress)
            this.progress.set(Math.round(100 * event.loaded / (event.total ?? event.loaded)));
          if (event.type === HttpEventType.Response) this.progress.set(null);
        },
        error: () => this.progress.set(null),
      });
  }
  cancel() {
    this.sub?.unsubscribe();          // unsubscribe = genuine abort of the transfer
    this.progress.set(null);
  }
}

6 (Hard): Sketch the single-flight token-refresh interceptor from Q5 — concurrent 401s share one refresh, retried requests are flagged against loops, refresh failure logs out.
Solution
@Injectable({ providedIn: 'root' })
export class TokenRefreshCoordinator {
  private auth = inject(AuthService);
  private inFlight$: Observable<string> | null = null;
  refresh(): Observable<string> {                      // SINGLE-FLIGHT:
    this.inFlight$ ??= this.auth.refreshToken().pipe(  // first caller starts it,
      map(r => r.accessToken),
      shareReplay(1),                                  // concurrent callers share it,
      finalize(() => (this.inFlight$ = null)),         // next expiry starts fresh
    );
    return this.inFlight$;
  }
}

const RETRIED = new HttpContextToken(() => false);

export const refreshInterceptor: HttpInterceptorFn = (req, next) => {
  const coordinator = inject(TokenRefreshCoordinator);
  const auth = inject(AuthService);
  return next(req).pipe(
    catchError((err: HttpErrorResponse) => {
      const isAuthCall = req.url.includes('/auth/');
      if (err.status !== 401 || isAuthCall || req.context.get(RETRIED))
        return throwError(() => err);                  // not ours / already retried
      return coordinator.refresh().pipe(
        switchMap(token => next(req.clone({            // replay with FRESH token,
          setHeaders: { Authorization: `Bearer ${token}` },
          context: req.context.set(RETRIED, true),     // flagged: a second 401 propagates
        }))),
        catchError(refreshErr => {                     // refresh itself failed:
          auth.logout();                               // clear session, go to login
          return throwError(() => refreshErr);
        }),
      );
    }),
  );
};
// The three hard parts, addressed: shareReplay single-flight (no refresh stampede),
// HttpContextToken retry flag (no 401 loops), explicit logout path (no zombie sessions).

🎯 Key Takeaways

  • HttpClient observables are cold one-shots: no subscribe = no request, N subscribes = N requests, completion = no cleanup, unsubscribe = real abort.
  • Error handling layers by who can act: domain outcomes in features, transient retry (predicate-gated, idempotent-only) in pipes, cross-cutting policy (401/403/5xx) in one interceptor that always re-throws.
  • Interceptors are the immutable-clone middleware onion — auth, correlation, timing, caching, loading-bars — registered in meaningful order via withInterceptors.
  • httpResource = declarative reactive reads (signal-derived URL, stale-abort, loading/error/reload); mutations and operator-heavy flows stay on HttpClient + RxJS.
  • API services are the only HttpClient consumers: domain types out, DTO mapping contained, idempotency keys on writes, the mock seam for everything above.
  • get<T> asserts, it doesn't validate — schema-check critical payloads; FormData sets its own Content-Type; tokens prefer HttpOnly cookies; single-flight token refresh is the senior interview set-piece.
