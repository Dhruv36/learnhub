State Patterns: Signals & Services State Management

What you'll master: the state taxonomy that prevents over-engineering (server state vs client state vs URL state vs local state), the signal-store service pattern in full, async state done right (loading/error/data as one shape), optimistic updates, where derived state lives — and the escalation criteria for when a store library (NgRx, next lesson) actually earns its complexity. Prerequisites: Signals, Services & DI, HttpClient. Most Angular apps need exactly what this lesson teaches and nothing more.

Part 1 — The Taxonomy: Not All State Is the Same Problem

Classify BEFORE choosing machinery:

LOCAL/UI STATE      is this dropdown open, wizard step, hover     → component signals.
                    Dies with the view. NEVER globalize it.
URL STATE           filters, page, sort, selected id, active tab  → router (routing.html).
                    Shareable/bookmarkable = it belongs in the URL, not a store.
SERVER STATE        products, orders, user profile — data the     → API + caching layer
                    BACKEND owns; your copy is a CACHE with         (httpResource/stores).
                    staleness, loading, and errors.
CLIENT/APP STATE    cart contents, theme, session, draft edits —  → signal-store services.
                    state the FRONTEND genuinely owns.

The #1 architecture error: dumping everything into one global store.
Server state pretends to be client state (no staleness handling → stale UIs);
URL state duplicates into memory (back button desyncs); local state goes global
(every dropdown toggle rebuilds half the app's subscriptions).

The taxonomy is the lesson. Each kind of state has a different lifecycle, ownership, and failure mode — server state can be stale, client state can't; URL state must survive reload, local state must not outlive its view — so each gets different machinery. Classify first and most "state management" debates dissolve: the filter panel is URL state (already solved by the router), the product list is server state (a caching problem, not a store problem), and what's left — genuinely client-owned, cross-component state — is usually small enough for a few signal services. This framing is also the strongest answer to the interview opener "how do you manage state in Angular?"

Part 2 — The Signal Store Service, Formalized

// The pattern assembled through this track, now as the reference shape:
@Injectable({ providedIn: 'root' })
export class CartStore {
  // 1. PRIVATE writable state — the single source of truth, minimal:
  private _items = signal<CartItem[]>([], { equal: (a, b) => a === b });
  private _couponCode = signal<string | null>(null);

  // 2. PUBLIC read model — readonly + computed projections (derive, don't store):
  readonly items = this._items.asReadonly();
  readonly count = computed(() => this._items().reduce((n, i) => n + i.qty, 0));
  readonly subtotal = computed(() => this._items().reduce((s, i) => s + i.price * i.qty, 0));
  readonly discount = computed(() => this._couponCode() ? this.subtotal() * 0.1 : 0);
  readonly total = computed(() => this.subtotal() - this.discount());
  readonly isEmpty = computed(() => this._items().length === 0);

  // 3. MUTATION METHODS — named after DOMAIN OPERATIONS, enforcing invariants:
  add(product: Product, qty = 1) {
    this._items.update(items => {
      const existing = items.find(i => i.productId === product.id);
      return existing
        ? items.map(i => i === existing ? { ...i, qty: i.qty + qty } : i)
        : [...items, { productId: product.id, name: product.name, price: product.price, qty }];
    });
  }
  setQty(productId: number, qty: number) {
    if (qty <= 0) return this.remove(productId);          // invariant lives HERE, once
    this._items.update(l => l.map(i => i.productId === productId ? { ...i, qty } : i));
  }
  remove(productId: number) {
    this._items.update(l => l.filter(i => i.productId !== productId));
  }
  applyCoupon(code: string) { this._couponCode.set(code); }
  clear() { this._items.set([]); this._couponCode.set(null); }
}

Three layers, each doing one job: private writable signals hold the minimum facts; public computeds derive every projection (totals can never desync from items — the derive-don't-sync rule as architecture); and mutation methods named as domain operations (add, applyCoupon — not setItems) centralize invariants so no component can construct an illegal state. Consumers inject(CartStore) at any depth, read signals in templates (precise re-rendering via signal-driven CD), and call methods. This shape is testable as a plain class, needs zero libraries, and scales further than most teams expect.

Part 3 — Server State: the Async Shape

// Server state's irreducible shape: loading | error | data (+ staleness). Model it ONCE:
export type Async<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'error'; error: string }
  | { status: 'loaded'; data: T; fetchedAt: number };

@Injectable({ providedIn: 'root' })
export class ProductsStore {
  private api = inject(ProductApi);
  private _state = signal<Async<Product[]>>({ status: 'idle' });
  readonly state = this._state.asReadonly();
  readonly products = computed(() =>
    this._state().status === 'loaded' ? (this._state() as any).data as Product[] : []);

  load(force = false) {
    const s = this._state();
    if (!force && s.status === 'loaded' && Date.now() - s.fetchedAt < 60_000) return;
    this._state.set({ status: 'loading' });                 // ← STALE-WHILE: or keep old data
    this.api.list().subscribe({
      next: data => this._state.set({ status: 'loaded', data, fetchedAt: Date.now() }),
      error: () => this._state.set({ status: 'error', error: 'Could not load products' }),
    });
  }
}

// For PER-PARAM server state, skip the hand-rolled store — the platform now has it:
//   product = httpResource<Product>(() => `/api/products/${this.id()}`);
// httpResource/resource IS a per-request async store: loading/error/value/reload built in.
// Hand-rolled stores earn their place for SHARED, CACHED, MUTATED-BY-MANY server data.

Server state's essence is that your copy is a cache: it can be missing, in flight, failed, or stale — and modeling those as a discriminated union makes illegal combinations (data and error, loading forever) unrepresentable, while templates @switch over status exhaustively. The TTL check shows staleness as a product decision encoded in one place. Honest guidance on tooling: for route-driven single fetches, resource/httpResource already is this pattern — hand-roll stores only where data is shared across features, mutated from multiple places, or cached with custom policies. (This "server cache ≠ app state" insight is the same one React's ecosystem reached via TanStack Query.)

Part 4 — Mutations & Optimistic Updates

// PESSIMISTIC (default — correct first): show pending, apply on server confirm:
rename(id: number, name: string) {
  this._saving.set(true);
  this.api.rename(id, name).subscribe({
    next: updated => { this.applyUpdate(updated); this._saving.set(false); },
    error: () => { this._saving.set(false); this.toast.error('Rename failed'); },
  });
}

// OPTIMISTIC (snappy — for high-success, low-stakes ops): apply now, roll back on failure:
toggleDone(id: number) {
  const before = this._todos();                            // snapshot for rollback
  this._todos.update(l => l.map(t => t.id === id ? { ...t, done: !t.done } : t));  // instant UI
  this.api.toggleDone(id).subscribe({
    error: () => {
      this._todos.set(before);                             // roll back
      this.toast.error("Couldn't save — change undone");   // and SAY so
    },
  });
}

// Choosing:
//   optimistic — likes, toggles, reorders, renames: high success rate, easy rollback,
//                user benefit from instant feedback is high
//   pessimistic — payments, deletions, anything hard to undo or likely to fail validation
// Concurrency note: rapid toggles need the rxjs-patterns treatment (concatMap queue or
// debounced last-write-wins) — optimistic UI doesn't excuse racing writes.

Mutations complete the store: write to the server through the store method, reconcile local state on the response. Optimistic updates are a UX trade — instant feedback against the complexity of snapshot-and-rollback — and the decision criteria (success probability, undo difficulty, stakes) matter more than the technique. Whichever mode, the store owns the choreography; components just call store.toggleDone(id) and render signals, which keeps the subtle code in one tested place.

Part 5 — Scoping Stores & Composing Them

// Store lifetime = provider placement (di-advanced.html, applied):
providedIn: 'root'          app-lifetime singletons: cart, session, theme
route providers             feature-lifetime: CheckoutWizardStore lives while /checkout
                            is mounted — fresh per entry, GC'd on leave
component providers         widget-lifetime: one FilterPanelStore per filter panel instance

// COMPOSITION — stores may read other stores' PUBLIC signals:
@Injectable({ providedIn: 'root' })
export class CheckoutStore {
  private cart = inject(CartStore);
  private session = inject(SessionStore);
  readonly canCheckout = computed(() =>
    !this.cart.isEmpty() && this.session.loggedIn());
  readonly summary = computed(() => ({
    items: this.cart.count(), total: this.cart.total(),
    email: this.session.user()?.email ?? '',
  }));
}
// Rules: depend DOWNWARD or sideways-via-public-API only; never reach into another
// store's privates; cycles between stores = extract the shared source (services-di.html).

Because stores are just injectables, every DI scoping tool applies: a checkout wizard whose state must reset per visit is a route-provided store — no manual reset methods, the injector lifecycle does it. Composition via public computed signals keeps stores small and single-domain (cart ≠ session ≠ checkout orchestration) while cross-domain views assemble reactively — the same facade idea one level up. The dependency rules prevent the tangle that gives "lots of services" a bad name.

Part 6 — Deep Dive: When Do You Need More? (The Honest Escalation Criteria)

Signal stores comfortably handle:      Signs you're outgrowing them:
  a handful of domain stores             many stores updating in COORDINATED transactions
  CRUD + optimistic updates              audit/undo/time-travel requirements
  derived view models                    the same action triggering effects in N places
  per-feature scoping                    (email + analytics + cache-bust + navigation)
                                         large teams needing ENFORCED update discipline
                                         complex event sourcing / replay needs

The escalation path is GRADUAL, not a rewrite:
  1. plain signal stores            (this lesson — the default)
  2. @ngrx/signals SignalStore      same mental model, structured: withState/withComputed/
                                    withMethods, entity helpers, devtools — library rigor
                                    without actions/reducers ceremony
  3. full NgRx (Store/Effects)      global action log, devtools time-travel, effect
                                    isolation — earns its cost at org scale (ngrx.html)

And regardless of level: server state might belong in resource/httpResource,
URL state in the router, local state in components. The store is for what REMAINS.

The escalation table is the senior judgment this lesson builds toward: signal stores are not "beginner NgRx" — they're the correct default, and each escalation step buys specific capabilities (structure and entity helpers at step 2; a global action log with devtools time-travel and enforced indirection at step 3) at real cost (ceremony, onboarding, dilution of "just read the method"). Choosing machinery your app's actual coordination complexity doesn't require is the front-end equivalent of microservices-for-a-todo-app. The next lesson teaches NgRx properly — partly because you'll inherit codebases built on it, partly because at some scale it genuinely wins — but the burden of proof sits with the heavier tool.

Senior framing: state reviews probe invariants and ownership, not library choice: Can illegal states be represented (two booleans instead of one status union)? Can any component corrupt state (public writable signals — the cardinal sin)? Is derived data stored (count fields drifting from arrays)? Is server data cached without a staleness story (the eternally-stale dashboard)? Does URL-worthy state hide in memory (share a link, get a different view)? Is optimistic rollback actually tested? A crisp store shape — minimal private facts, computed projections, domain-named mutations, explicit async unions — answers all six by construction, which is why interviewers who ask "design the state for X" are really asking for this lesson's Part 2 + Part 3 shapes with your domain's nouns.

Interview Questions

Q1 (Beginner): Describe the signal-store service pattern and why each layer exists.

Three layers: private writable signals (the minimal source of truth — private so invariants can't be bypassed), public read model (asReadonly() views plus computed() projections — consumers get everything derived, guaranteed consistent, precisely reactive), and mutation methods named as domain operations (add, applyCoupon — the only write path, where validation and invariants live once). Components inject the store at any depth, read signals in templates, call methods on events — no subscriptions to manage, no libraries required.

Q2 (Beginner): Why should cartTotal and itemCount never be stored fields in a cart store?

They're pure functions of the items list — storing them creates two sources of truth that every mutation must remember to update in lockstep; miss one path and the badge shows 3 items with an empty cart. As computed(() => items().reduce(...)) they cannot desync (no update code exists to forget), they're lazy and memoized (recompute only when items actually changes), and consumers get the same reactive read API. Store the minimum facts; derive everything else.

Q3 (Mid): Explain the four-way state taxonomy and give the failure mode of misclassifying each.

Local/UI state (component signals — globalizing it couples unrelated views to dropdown toggles); URL state (router query/params — hiding it in memory breaks share/reload/back); server state (a cache of backend-owned data with loading/error/staleness — treating it as plain client state yields stale UIs with no refresh semantics); client/app state (frontend-owned, cross-component — the only category that belongs in stores; scattering it across components yields prop-drilling and desync). The classification determines the machinery, which is why it precedes any library discussion.

Q4 (Mid): Implement optimistic toggle-favorite conceptually: what are the steps, and when would you refuse optimistic UI for an operation?

Steps: snapshot current state → apply the change to the signal immediately (instant UI) → fire the API call → on success, optionally reconcile with the server's canonical response → on error, restore the snapshot and inform the user (silent rollback reads as a glitch). Refuse optimistic treatment when failure is likely (server-side validation), rollback is complex (cascading effects), or stakes are high (payments, destructive deletes) — there, pessimistic pending-state UX is honest. Also handle rapid repeats: queue (concatMap) or debounce, since optimistic UI doesn't serialize the underlying writes.

Q5 (Senior): A checkout wizard's store must reset every time the user re-enters checkout, but the cart must persist. Where does each store live and why?

CartStore: providedIn: 'root' — app-lifetime, shared across features (product pages add to it, header reads it). CheckoutWizardStore: provided in the checkout feature's route providers — the injector for that route subtree creates it on first navigation into /checkout and destroys it when the user leaves, so re-entry gets a pristine instance with zero manual reset code; its lifetime is the feature's lifetime. The wizard store composes the cart via injection (reading its public signals for the summary). This is DI hierarchy (di-advanced) applied as state architecture: provider placement is the state-lifetime API.

Q6 (Senior): Your team of 25 debates adopting NgRx app-wide. Lay out the decision framework you'd bring to that meeting.

First, reclassify the state inventory: how much is URL state (router's job), server state (resource/httpResource or thin cached stores), or local (components)? The store-worthy remainder is often 3–5 domains. Then assess coordination complexity: are there multi-store transactional updates, one-action-many-effects fan-outs, undo/audit requirements, or discipline problems (states mutated from everywhere) that an enforced action log would fix? If yes, weigh full NgRx's real wins (devtools time-travel, effect isolation, org-scale conventions) against its costs (ceremony, onboarding, indirection) — and consider @ngrx/signals SignalStore as the middle step giving structure and entity helpers on the same signal mental model. Decide per-domain, not app-wide: a cart doesn't need actions; a collaborative editor's document state might. And whatever wins, the public surface stays the same — signals in, methods/actions out — so the choice is revisable. The framework matters less than showing the escalation is evidence-driven, reversible, and scoped.

1 (Easy): Build a ThemeStore (light/dark/system, persisted, applied to document.body) using the store shape — identify which part is state, which derived, which effect.
Solution
@Injectable({ providedIn: 'root' })
export class ThemeStore {
  private _preference = signal<'light' | 'dark' | 'system'>(
    (localStorage.getItem('theme') as any) ?? 'system');            // STATE (minimal fact)
  readonly preference = this._preference.asReadonly();
  readonly resolved = computed(() =>                                 // DERIVED
    this._preference() === 'system'
      ? (matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light')
      : this._preference());

  constructor() {
    effect(() => {                                                   // EFFECT (outward sync)
      localStorage.setItem('theme', this._preference());
      document.body.classList.toggle('dark', this.resolved() === 'dark');
    });
  }
  setPreference(p: 'light' | 'dark' | 'system') { this._preference.set(p); }  // MUTATION
}
// One stored fact (the preference); the applied theme is derived; storage/DOM are
// effects — the signals.html division of labor, in store form.

3 (Medium): Implement the Async<T> union and a UsersStore with TTL caching and a force-refresh — plus the template that switches over status exhaustively.
Solution
export type Async<T> =
  | { status: 'idle' } | { status: 'loading' }
  | { status: 'error'; error: string }
  | { status: 'loaded'; data: T; fetchedAt: number };

@Injectable({ providedIn: 'root' })
export class UsersStore {
  private api = inject(UserApi);
  private _state = signal<Async<User[]>>({ status: 'idle' });
  readonly state = this._state.asReadonly();

  load(force = false) {
    const s = this._state();
    if (s.status === 'loading') return;                              // no duplicate loads
    if (!force && s.status === 'loaded' && Date.now() - s.fetchedAt < 120_000) return;
    this._state.set({ status: 'loading' });
    this.api.list().subscribe({
      next: data => this._state.set({ status: 'loaded', data, fetchedAt: Date.now() }),
      error: () => this._state.set({ status: 'error', error: 'Failed to load users' }),
    });
  }
  refresh() { this.load(true); }
}

4 (Medium): Add optimistic rename(id, name) to a notes store with rollback + user notification, and a pessimistic delete(id) with confirm + pending state. Justify the split.
Solution
rename(id: number, name: string) {                     // OPTIMISTIC — safe + reversible:
  const before = this._notes();
  this._notes.update(l => l.map(n => n.id === id ? { ...n, name } : n));   // instant
  this.api.rename(id, name).subscribe({
    error: () => {
      this._notes.set(before);                          // rollback
      this.toast.error('Rename failed — reverted');     // honesty
    },
  });
}

deleting = signal<number | null>(null);
delete(id: number) {                                    // PESSIMISTIC — destructive:
  if (!confirm('Delete this note permanently?')) return;
  this.deleting.set(id);                                // pending UI on that row
  this.api.delete(id).subscribe({
    next: () => { this._notes.update(l => l.filter(n => n.id !== id));
                  this.deleting.set(null); },
    error: () => { this.deleting.set(null); this.toast.error('Delete failed'); },
  });
}
// Rename: high success odds, trivially reversible, instant feedback valuable → optimistic.
// Delete: irreversible, worth a beat of friction — apply only on server confirmation.
5 (Hard): Build the route-scoped CheckoutWizardStore (steps, per-step data, canProceed) composed over the root CartStore, wired into checkout's route providers — and show why re-entering checkout is automatically fresh.
Solution
@Injectable()                                       // NO providedIn — route-provided
export class CheckoutWizardStore {
  private cart = inject(CartStore);                 // root store, composed via public API
  private _step = signal(0);
  private _shipping = signal<ShippingInfo | null>(null);
  private _payment = signal<PaymentInfo | null>(null);

  readonly step = this._step.asReadonly();
  readonly canProceed = computed(() => {
    switch (this._step()) {
      case 0: return !this.cart.isEmpty();          // review step: needs a cart
      case 1: return this._shipping() !== null;
      case 2: return this._payment() !== null;
      default: return false;
    }
  });
  readonly summary = computed(() => ({
    total: this.cart.total(), items: this.cart.count(),
    shipping: this._shipping(), payment: this._payment() && '•••• saved',
  }));

  next() { if (this.canProceed()) this._step.update(s => s + 1); }
  back() { this._step.update(s => Math.max(0, s - 1)); }
  setShipping(s: ShippingInfo) { this._shipping.set(s); }
  setPayment(p: PaymentInfo)   { this._payment.set(p); }
}

export default [
  { path: '', component: CheckoutLayoutComponent,
    providers: [CheckoutWizardStore],               // ← injector-scoped to this subtree
    children: [ /* step routes bind to the store */ ]},
] satisfies Routes;

🎯 Key Takeaways

  • Classify before building: local (component signals), URL (router), server (a cache — resource/httpResource or async stores), client (signal stores). Misclassification is the root state error.
  • The store shape: private writable signals (minimal facts) → public readonly/computed (derived, consistent) → domain-named mutation methods (invariants once). No libraries required.
  • Server state = discriminated union (idle/loading/error/loaded+fetchedAt) — illegal combinations unrepresentable, staleness an explicit policy.
  • Optimistic updates = snapshot → instant apply → rollback + toast on failure; reserve for high-success, reversible, low-stakes ops — and serialize racing writes.
  • Provider placement is state-lifetime: root (app), route (feature — auto-fresh per entry), component (per widget). Stores compose via public signals, downward/sideways only.
  • Escalate on evidence: signal stores → @ngrx/signals (structure) → full NgRx (action log, time-travel, org discipline) — the burden of proof is on the heavier tool.
