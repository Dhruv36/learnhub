RxJS Fundamentals RxJS

What you'll master: the Observable mental model (lazy push streams), creating and subscribing, the core operator vocabulary (map, filter, tap, take…), Subjects and their variants (BehaviorSubject, ReplaySubject), hot vs cold — and where RxJS sits in a signals-era Angular app. Prerequisites: Promises and Signals. RxJS remains load-bearing in Angular — HttpClient, the Router, and forms all speak Observables — so fluency is not optional, even as signals take over state.

Part 1 — The Model: a Stream You Subscribe To

            ONE value          MANY values over time
PULL        function call      iterator / generator
PUSH        Promise            OBSERVABLE  ← RxJS

const clicks$ = fromEvent(document, 'click');       // a STREAM of future events
const sub = clicks$.subscribe(e => console.log(e)); // nothing ran until NOW (lazy)
sub.unsubscribe();                                   // stop listening, release resources

// The full observer — three channels:
obs$.subscribe({
  next:     v   => render(v),         // 0..N emissions
  error:    err => showError(err),    // terminal: stream is DEAD after this
  complete: ()  => done(),            // terminal: no more values, clean end
});

// vs Promise: eager, single value, no cancellation.
// Observable: LAZY (runs per subscribe), MULTI-value, CANCELLABLE (unsubscribe).

An Observable is a lazy push stream: a recipe for producing values over time that does nothing until subscribed, may emit any number of values, and can be cancelled. Those three properties are exactly what Promises lack, and each maps to a real Angular need: laziness means an http.get() observable doesn't fire until something subscribes (and fires again for each new subscriber — Part 5); multiplicity models event streams (router events, form value changes, websockets); cancellation is what makes "abandon the stale search" possible (next lesson). The $-suffix naming convention (clicks$) marks stream variables — adopt it.

Part 2 — Creating Streams

of(1, 2, 3)                          // emit the arguments, then complete
from([1, 2, 3])                      // from array/promise/iterable
from(fetch('/api/x'))                // promise → observable
fromEvent(input, 'input')            // DOM events as a stream
interval(1000)                       // 0,1,2,… every second (never completes!)
timer(5000)                          // one 0 after 5s, then complete
timer(0, 30_000)                     // immediately, then every 30s — polling skeleton

// Where streams actually COME FROM in Angular (you rarely hand-create):
this.http.get<User[]>('/api/users')     // HttpClient: emits once, completes (http.html)
this.route.paramMap                      // router params over time (routing.html)
this.form.valueChanges                   // form edits as a stream (forms.html)

// Custom source — bridging a callback API:
const messages$ = new Observable<string>(subscriber => {
  const ws = new WebSocket(url);
  ws.onmessage = (e) => subscriber.next(e.data);
  ws.onerror   = (e) => subscriber.error(e);
  return () => ws.close();            // TEARDOWN — runs on unsubscribe/error/complete
});

Creation functions wrap every shape of source into the one uniform interface. The custom-Observable form shows the contract from the inside: the function you pass runs once per subscriber, pushes via next/error/complete, and returns teardown — which is what unsubscribe() invokes, and why unsubscribing a websocket stream closes the socket. In day-to-day Angular you mostly receive observables from the framework; knowing how they're built is what makes their behavior (laziness, per-subscriber execution, teardown) predictable rather than magical.

Part 3 — Operators: the Pipeline Vocabulary

searchBox$.pipe(
  map(e => (e.target as HTMLInputElement).value),   // transform each value
  filter(q => q.length > 1),                        // let some through
  tap(q => console.log('searching', q)),            // side effect, value untouched (debug!)
).subscribe(q => this.search(q));

// The core set — learn these cold:
map(fn)          transform each emission                  ([1,2,3] stream → [2,4,6])
filter(pred)     drop non-matching emissions
tap(fn)          side effects (logging, metrics) without touching the stream
take(n)          first n, then COMPLETE (auto-unsubscribes upstream!)
first() / find(pred)   one value then complete
takeUntil(sig$)  live until another stream emits — the lifecycle pattern (rxjs-patterns)
distinctUntilChanged()  skip consecutive duplicates
startWith(v)     seed an initial value (loading states, defaults)
scan(acc, seed)  running fold — reduce that emits every step
catchError(fn)   handle errors → return a fallback stream (http.html)

// Operators are PURE: each returns a NEW observable; the source is never modified.
// pipe() composes them left→right — same mental model as array chains / pipe() from
// the JS track's functional lesson, applied to time.

Operators are function composition over streams: pipe chains small pure transformations, each wrapping the previous observable in a new one. Fluency in the core set covers most Angular code you'll read. Two with outsized practical weight: take(n)/first() complete the stream — after them, upstream is unsubscribed automatically, making route.data.pipe(first()) a self-cleaning one-shot read; and tap is the honest home for side effects — hiding state mutations inside map is the stream equivalent of side effects in array map, and reviewers read it the same way.

Part 4 — Subjects: Streams You Push Into

// A Subject is BOTH observer (you can .next() it) and observable (others subscribe):
const refresh$ = new Subject<void>();
refresh$.subscribe(() => this.reload());       // consumer A
refresh$.subscribe(() => this.track('refresh'));// consumer B — MULTICAST: both fire
refresh$.next();                                // push an event to all subscribers

// Variants — what does a LATE subscriber see?
new Subject<T>()          nothing until the next emission        (pure event bus)
new BehaviorSubject(init) the CURRENT value immediately, then updates   (state-like)
new ReplaySubject(3)      the last 3 emissions, then live              (catch-up buffers)

// The classic pre-signals state service — you WILL meet this in every codebase:
@Injectable({ providedIn: 'root' })
export class CartService {
  private items$ = new BehaviorSubject<CartItem[]>([]);
  readonly cart$ = this.items$.asObservable();        // read-only view for consumers
  readonly total$ = this.cart$.pipe(map(list => list.reduce((s, i) => s + i.price, 0)));
  add(item: CartItem) { this.items$.next([...this.items$.value, item]); }
}
// Recognize the shape? private-writable + public-readonly + methods —
// the SAME architecture as the signal service; BehaviorSubject was the signal before signals.

Plain Observables are pull-started push pipes — the producer is defined inside. Subjects invert that: you push values in imperatively, and all current subscribers receive them (multicast). The variants differ only in late-subscriber behavior, and choosing correctly is a favorite interview probe: Subject for pure events (a subscriber who missed it, missed it), BehaviorSubject for state (there's always a current value — new subscribers need it immediately), ReplaySubject for catch-up semantics. The BehaviorSubject state service is the single most important legacy pattern to read fluently — and to recognize as exactly what signal services replace, with the same public shape and less machinery.

Part 5 — Hot vs Cold, and the Multiple-Subscription Trap

// COLD: the producer runs PER SUBSCRIBER (each gets its own execution):
const users$ = this.http.get<User[]>('/api/users');
users$.subscribe(render);        // HTTP request #1
users$.subscribe(count);         // HTTP request #2  ← the classic accidental double-fetch!

// In templates, the same trap wears async-pipe clothing:
//   {{ (users$ | async)?.length }} … @for (u of users$ | async …)   ← 2 subscriptions, 2 requests

// HOT: the producer runs INDEPENDENTLY; subscribers tap into it (events, subjects, sockets)
fromEvent(document, 'click')     // clicks happen whether or not anyone listens

// COLD → HOT (share one execution among subscribers):
const users$ = this.http.get<User[]>('/api/users').pipe(
  shareReplay({ bufferSize: 1, refCount: true }),   // one request; late subscribers get the value
);

Cold observables restart their producer for every subscriber — correct for "each subscriber wants the work done" (retrying a request), surprising when the work is an HTTP call and three parts of a template subscribe. Diagnosing duplicate network requests almost always ends here. The fixes, in order of preference: subscribe once and share the result (assign toSignal(users$) or one async-pipe'd variable via @if (users$ | async; as users)), or make the stream itself shared with shareReplay (mind refCount — without it the source stays alive after the last unsubscribe). Hot sources — subjects, DOM events, websockets — don't have this problem but have the mirror one: emissions while nobody listens are simply gone (hence ReplaySubject/buffering when that matters).

Part 6 — Deep Dive: RxJS's Place in Signals-Era Angular

What each is FOR (the boundary from signals.html, now from the RxJS side):

SIGNALS                                RXJS
current state, derivations             events over time, async pipelines
template consumption                   composition: debounce/retry/cancel/merge
glitch-free sync graph                 backpressure-ish control of WHEN things run

Where RxJS remains the right tool in a modern app:
  · HttpClient composition: retry with backoff, request cancellation   (http.html)
  · typeahead/debounce pipelines                                       (rxjs-patterns)
  · router events, form valueChanges — framework streams you consume
  · websockets/SSE event handling before it lands in a signal
  · anything where TIME is the domain: throttling, buffering, sequencing

Consumption rules of thumb (modern):
  component state from a stream   → toSignal(stream$, {initialValue})
  template-only, no logic needed  → stream$ | async (with `as` capture)
  manual .subscribe() in a component → LAST RESORT; requires teardown discipline
                                       (takeUntilDestroyed — next lesson)

The framing that keeps teams sane: RxJS is the events-and-time layer; signals are the state layer. Streams flow through RxJS operators and settle into signals for templates. What shrank in the signals era is not RxJS's event role but its state role — BehaviorSubject services, combineLatest view-models, and manual subscription choreography convert to signal/computed equivalents that are simpler and auto-tracked. What remains is what RxJS was always best at: expressing "wait 300ms, drop stale, retry twice, then give up" as a declarative pipeline instead of a state machine of flags and timers.

Senior framing: RxJS interviews sort candidates into three tiers. Tier 1 knows operator names. Tier 2 can explain laziness, per-subscriber execution, and teardown — the model that predicts behavior. Tier 3 can choose: Subject vs BehaviorSubject by late-subscriber semantics; cold-sharing via one subscription vs shareReplay (and the refCount leak nuance); signals vs streams by state-vs-events; and knows the failure taxonomy — double-fetch (cold, multi-subscribed), missed emissions (hot, late subscriber), leaks (subscribe without teardown), dead streams (error is terminal — un-caught errors kill the pipeline permanently, the #1 "my stream stopped working" cause, handled properly in Patterns). Aim for tier 3: every one of those is a one-line question with a mental-model answer.

Interview Questions

Q1 (Beginner): Observable vs Promise — the three differences and why each matters in Angular.

(1) Lazy vs eager: an Observable runs its producer per subscribe; a Promise runs on creation — laziness lets Angular hand you request recipes (http.get) that only fire when consumed. (2) Many vs one: Observables model streams (router events, form changes, sockets); Promises settle once. (3) Cancellable vs not: unsubscribe() runs teardown (abort the request, close the socket) — the basis of stale-request cancellation. Bonus: Observables compose via operators (debounce, retry) that Promises have no equivalent for.

Q2 (Beginner): What are the three observer callbacks, and what's special about error and complete?

next (0..N values), error (failure with a reason), complete (clean end). Both error and complete are terminal: the stream is finished, teardown runs, and no further emissions ever arrive — a stream errors or completes, never both. The practical consequence: an unhandled error permanently kills a pipeline (a polling stream that hit one bad response stops polling forever) — which is why error handling placement matters (Patterns lesson).

Q3 (Mid): Subject vs BehaviorSubject vs ReplaySubject — semantics and a correct use for each.

They differ in what late subscribers receive. Subject: nothing — events are fire-and-forget (correct for "refresh clicked", notifications). BehaviorSubject(initial): the current value immediately — it is state (correct for cart contents, auth status; also exposes .value). ReplaySubject(n): the last n emissions — catch-up semantics (correct for "recent activity" feeds or late-mounting consumers that need short history). Choosing Subject where BehaviorSubject was needed is the classic bug: components mounted after the emission render empty.

Q4 (Mid): A page fires the same GET three times. Walk the diagnosis and the fixes.

Diagnosis: cold observable, multiple subscriptions — http.get runs its producer per subscriber; three | async pipes (or async pipes plus a manual subscribe) = three requests. Fixes in preference order: (1) subscribe once — capture with @if (users$ | async; as users) and bind users below, or convert with toSignal(users$) and read the signal everywhere; (2) share the stream — shareReplay({ bufferSize: 1, refCount: true }) multicasts one execution and replays to late subscribers (name the nuance: without refCount the source can stay subscribed forever — a leak). The model answer notes this is correct cold behavior, not a bug in RxJS.

Q5 (Senior): Explain the Observable contract from the producer side — what does the subscribe function receive, when does it run, and what is teardown?

new Observable(subscriberFn): the function receives a Subscriber with next/error/complete and runs once per subscribe call — that's laziness and per-subscriber execution in one sentence (cold by default; hot sources move the producer outside). Its return value is the teardown function, invoked on unsubscribe or terminal events — the place to close sockets, clear timers, abort fetches; this is what makes cancellation real rather than advisory. The contract also guarantees ordering (no emissions after error/complete). From this one contract you can derive most RxJS behavior: why unsubscribing an HTTP stream aborts the request, why take(1) cleans up upstream, why sharing operators exist (to sit between one producer execution and many subscribers).

Q6 (Senior): Your team debates "signals have replaced RxJS — remove it". Give the senior verdict with concrete boundaries.

Verdict: replace RxJS-for-state, keep RxJS-for-events-over-time. Convert: BehaviorSubject state services → signal services; combineLatest view-models → computed; template consumption → signals (via toSignal at boundaries). Keep: time-domain pipelines — debounced typeahead, retry/backoff, stale-request cancellation (switchMap), websocket/event composition — and framework streams (router, forms, HttpClient) which you consume and convert at the seam. Removing RxJS entirely means hand-rolling timing/cancellation state machines (flags, timers, race guards) that operators express in a line — trading a learning curve for a bug farm. The architecture that wins: signals at the edges, RxJS in the middle where time is the domain, interop (toSignal/toObservable) as the explicit boundary.

🏋️ Exercises

1 (Easy): Create a stream from a button's clicks that logs only the first 5 clicks with their index, then completes — prove completion with a log.
Solution
fromEvent(button, 'click').pipe(
  scan(count => count + 1, 0),      // running index: 1,2,3,…
  take(5),                          // completes after the 5th — upstream unsubscribed
).subscribe({
  next: n => console.log(`click #${n}`),
  complete: () => console.log('done listening'),   // fires after click #5
});
// take(n) both limits AND completes — the listener detaches itself; no manual cleanup.
2 (Easy): Wire a "status" pipeline: from interval(1000), emit "tick N" only for even N, log each raw N via the proper side-effect operator, and stop after 10 seconds total.
Solution
interval(1000).pipe(
  takeUntil(timer(10_000)),          // hard stop at 10s — completes the pipeline
  tap(n => console.log('raw', n)),   // side effect: tap, NOT map
  filter(n => n % 2 === 0),
  map(n => `tick ${n}`),
).subscribe(msg => status.set(msg));
// interval never completes on its own — takeUntil bounds it. Order matters:
// takeUntil first ensures even tap stops at 10s.
3 (Medium): Build the legacy-style ThemeService with a BehaviorSubject ('light'|'dark'), read-only stream, toggle method — then show a late-subscribing component proving why Subject would fail here.
Solution
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private theme$$ = new BehaviorSubject<'light' | 'dark'>('light');
  readonly theme$ = this.theme$$.asObservable();          // consumers can't .next()
  toggle() { this.theme$$.next(this.theme$$.value === 'light' ? 'dark' : 'light'); }
}

// A component mounted LATER (e.g. lazy route):
export class SettingsComponent {
  theme = toSignal(inject(ThemeService).theme$);   // receives 'dark' IMMEDIATELY if
}                                                   // toggled before mount — BehaviorSubject
                                                    // replays the current value.
// With a plain Subject: theme() stays undefined until the NEXT toggle — the settings
// screen renders wrong until the user happens to act. State needs current-value semantics.
4 (Medium): A template uses users$ | async in three places and fires three requests. Show both fixes (as-capture and shareReplay) and say when each is right.
Solution
<!-- Fix A — ONE subscription, capture the value (right when one template consumes): -->
@if (users$ | async; as users) {
  <h2>{{ users.length }} users</h2>
  @for (u of users; track u.id) { <li>{{ u.name }}</li> }
  <p>Active: {{ activeCount(users) }}</p>
}

// Fix B — share the STREAM (right when multiple components/templates consume):
users$ = this.http.get<User[]>('/api/users').pipe(
  shareReplay({ bufferSize: 1, refCount: true }),
);
// refCount: true → when the last subscriber leaves, the source unsubscribes (no leak);
// late subscribers within the window get the cached value instead of a new request.

// Fix C (modern): users = toSignal(this.http.get<User[]>('/api/users')) — one
// subscription at declaration; templates read users() freely. Prefer this in signal-era code.
5 (Hard): Wrap a WebSocket in a custom Observable with correct teardown; then expose it through a ReplaySubject(5) so late subscribers see recent messages. Note what's hot/cold at each stage.
Solution
// COLD wrapper: each subscriber would open ITS OWN socket (usually not what you want):
const socket$ = new Observable<string>(sub => {
  const ws = new WebSocket('wss://api.example.com/feed');
  ws.onmessage = e => sub.next(e.data);
  ws.onerror   = e => sub.error(e);
  ws.onclose   = () => sub.complete();
  return () => ws.close();                         // teardown: unsubscribe closes the socket
});

// Make it HOT + buffered: ONE socket, recent history for late arrivals:
@Injectable({ providedIn: 'root' })
export class FeedService {
  private buffer$$ = new ReplaySubject<string>(5);
  readonly feed$ = this.buffer$$.asObservable();
  private sub = socket$.subscribe(this.buffer$$);  // single subscription drives the subject
  // (production: reconnect/backoff from the JS realtime lesson wraps socket$ here)
}
// Stage 1 cold (producer per subscriber) → single subscribe makes it effectively one
// execution → ReplaySubject multicasts it (hot) and replays last 5 to late subscribers.
6 (Hard): Code-review this service — find every RxJS defect and rewrite: 
@Injectable({ providedIn: 'root' })
export class PriceService {
  prices = new Subject<number>();
  latest = 0;
  constructor(http: HttpClient) {
    interval(10_000).subscribe(() => {
      http.get<{p:number}>('/api/price').subscribe(r => {
        this.latest = r.p;
        this.prices.next(r.p);
      });
    });
  }
}
Solution
// Defects:
// 1. PUBLIC Subject — any consumer can .next() fake prices.
// 2. Subject for STATE — late subscribers see nothing for up to 10s (want current value).
// 3. nested subscribe — composition done imperatively; also NO cancellation: a slow
//    response can land AFTER a newer one (out-of-order prices).
// 4. `latest` duplicates the stream's state manually (two sources of truth).
// 5. interval subscription never torn down (root service ~ok, but unbounded by design)
//    and one HTTP error would kill that inner emission silently with no recovery.

@Injectable({ providedIn: 'root' })
export class PriceService {
  private http = inject(HttpClient);
  // one declarative pipeline: poll → switch to request (cancels stale) → share as state:
  readonly price$ = timer(0, 10_000).pipe(
    switchMap(() => this.http.get<{ p: number }>('/api/price').pipe(
      map(r => r.p),
      catchError(() => EMPTY),               // one failure skips a tick, stream survives
    )),
    shareReplay({ bufferSize: 1, refCount: true }),   // current value for late subscribers
  );
  // modern consumption: price = toSignal(this.price$, { initialValue: null });
}
// switchMap/catchError are next lesson's stars — this exercise is the bridge.

🎯 Key Takeaways

  • An Observable is a lazy, cancellable push stream: producer runs per subscriber, values flow via next, and error/complete are terminal (teardown runs).
  • Angular hands you streams everywhere — HttpClient (emit once, complete), router params, form valueChanges — you compose them with pure operators in pipe().
  • Core vocabulary: map/filter/tap/take/takeUntil/distinctUntilChanged/startWith/scan/catchError — take/first complete and self-clean; tap is the only honest home for side effects.
  • Subjects are push-in multicast streams; choose by late-subscriber semantics: Subject (events), BehaviorSubject (state, current value), ReplaySubject (catch-up). Expose asObservable(), never the raw subject.
  • Cold = producer per subscriber (the duplicate-HTTP trap — fix with as-capture, toSignal, or shareReplay({refCount:true})); hot = producer independent (missed emissions are gone).
  • Signals own state, RxJS owns time: streams compose through operators and settle into signals via toSignal; manual subscribes are a last resort with teardown discipline.
