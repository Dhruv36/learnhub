RxJS Deep Dive Advanced

RxJS underlies HttpClient, the Router, and reactive forms — you can't avoid it in real Angular work. This lesson goes past "Observables are like Promises but multiple values" into the operators and subject types that come up constantly in production code and interviews: the flattening operators, hot vs cold streams, and combining multiple sources.

Observable Fundamentals, Precisely

const obs$ = new Observable<number>(subscriber => {
  subscriber.next(1);
  subscriber.next(2);
  subscriber.complete();
  return () => console.log('cleanup');    // teardown, runs on unsubscribe/complete
});
const sub = obs$.subscribe(val => console.log(val));
sub.unsubscribe();     // stops future emissions, runs teardown
// An Observable is a FUNCTION that, when subscribed, produces a sequence of values over time, and can
// be torn down. Nothing happens until subscribe() — this laziness is the core RxJS/HttpClient behavior.

The Flattening Operators — the Most-Asked RxJS Topic

OperatorBehavior on a new source emissionUse for
switchMapCANCELS the previous inner ObservableTypeahead search — only the latest matters
mergeMapRuns all inner Observables CONCURRENTLYIndependent parallel requests (e.g., batch uploads)
concatMapQUEUES inner Observables, runs one at a time in orderSequential writes that must not interleave
exhaustMapIGNORES new emissions while one is in-flightSubmit button — ignore extra clicks until the first completes
// The same "click a button" scenario, four different (very different!) behaviors:
save$.pipe(switchMap(() => this.http.post(...)))    // cancels an in-flight save if clicked again
save$.pipe(mergeMap(() => this.http.post(...)))     // fires a NEW parallel save every click (usually wrong!)
save$.pipe(concatMap(() => this.http.post(...)))    // queues saves, runs them one after another in order
save$.pipe(exhaustMap(() => this.http.post(...)))   // ignores extra clicks until the current save finishes
🎯 Picking the wrong flattening operator is THE classic RxJS bug. They all "flatten" an Observable-of-Observables into one stream, but with completely different concurrency semantics. Ask: do I want to cancel-and-replace (switchMap), run-in-parallel (mergeMap), queue-in-order (concatMap), or ignore-while-busy (exhaustMap)? Guessing wrong causes race conditions, duplicate submits, or dropped requests that are notoriously hard to reproduce in testing.

Subjects — Observables You Can Push Into

TypeBehaviorUse for
SubjectMulticasts to current subscribers only; no memoryEvent buses, one-off notifications

ReplaySubjectReplays the last N values to new subscribersCaching a limited history of emissions

const state$ = new BehaviorSubject<User | null>(null);
state$.subscribe(u => console.log('sub A sees', u));    // gets `null` immediately
state$.next({ name: 'Alex' });                            // both current + future subscribers get this
state$.subscribe(u => console.log('sub B sees', u));    // gets { name: 'Alex' } immediately (not null!)
// BehaviorSubject is why many Angular services (before Signals) hold state as a BehaviorSubject
// exposed as .asObservable() — new subscribers never miss "what's the value right now."

Combining Multiple Streams

combineLatest([user$, settings$]).pipe(
  map(([user, settings]) => ({ ...user, theme: settings.theme }))
)   // emits whenever ANY source emits, using the LATEST value from each

forkJoin([http.get('/a'), http.get('/b')])   // waits for ALL to complete, emits once with all results
  .subscribe(([a, b]) => ...)                 // like Promise.all, but for Observables

merge(click$, keydown$)   // a single stream of events from multiple sources, interleaved as they occur

Interview Questions

Q1: Explain switchMap, mergeMap, concatMap, and exhaustMap with a concrete scenario.

All four "flatten" a stream of Observables-of-Observables into a single stream, but they differ in how they handle a new outer emission while a previous inner Observable is still active. Take a save button clicked multiple times quickly, each click triggering an HTTP POST. switchMap CANCELS the previous in-flight request the moment a new click arrives and switches to the new one — appropriate for something like a search box where only the latest query's result should matter, but WRONG for a save action since it could cancel a save mid-flight. mergeMap lets ALL requests run concurrently and emits results as each resolves in whatever order they complete — useful for genuinely independent parallel work (like uploading five separate files), but dangerous for sequential-dependent actions since responses can arrive out of order. concatMap QUEUES each new request and runs them strictly one at a time in the order they were triggered, waiting for each to complete before starting the next — the right choice when order and non-overlap matter, like a sequence of writes that must not interleave. exhaustMap IGNORES new emissions entirely while an inner Observable is still active — the correct choice for a submit button, since it silently drops extra clicks until the current save finishes rather than cancelling or queuing them. Choosing the wrong one is the single most common source of subtle race-condition bugs in real Angular/RxJS code.

Q2: What's the difference between Subject and BehaviorSubject, and why does it matter?

A plain Subject is a simple multicast Observable you can push values into with .next() — it has no memory of past emissions, so a subscriber only receives values emitted AFTER it subscribes; if it subscribes late, it misses everything that already happened. A BehaviorSubject requires an initial value at construction and always remembers the CURRENT (most recent) value — critically, any new subscriber immediately receives that current value upon subscribing, even if they subscribed long after the last .next() call, and only then receives subsequent emissions. This distinction matters a lot for modeling STATE: if a service holds, say, the current logged-in user as a Subject, a component that subscribes late (after the user was already set) sees nothing until the next change — a bug. Using a BehaviorSubject instead guarantees that late subscribers always get "the value right now," not just future changes, which is exactly the semantics you want for state and is why many pre-Signals Angular services expose state as a BehaviorSubject (via .asObservable() to prevent external mutation) rather than a plain Subject.

Q3: How do you combine multiple asynchronous sources, and how do combineLatest and forkJoin differ?

RxJS provides several combination operators for different needs. combineLatest takes multiple source Observables and re-emits a combined result (usually via an array or object) EVERY time ANY of the sources emits, always using the most recently emitted value from each other source — it's ideal for reactive derived state, like combining a user$ stream and a settings$ stream so that a component reacts whenever either changes, always reflecting the latest of both. forkJoin instead waits for ALL the provided Observables to COMPLETE, then emits exactly once with the final value from each — it's the RxJS analog of Promise.all, and it's the right tool when you're firing several independent HTTP requests and want to proceed only once all of them have finished (e.g., loading a dashboard that needs data from three separate endpoints before rendering). merge is a simpler operator that just interleaves emissions from multiple sources into one stream as they occur, useful for something like combining click and keydown events into a single "user interacted" stream. Choosing between them comes down to whether you want continuous reactive recombination on every change (combineLatest), a single result once everything finishes (forkJoin), or simple interleaving of independent events (merge).

🏋️ Practice: build a typeahead search with debounceTime + distinctUntilChanged + switchMap and verify stale results are dropped; convert a save-button handler from mergeMap to exhaustMap and confirm rapid double-clicks only fire one request; replace a state Subject with a BehaviorSubject and observe a late subscriber now getting the current value; use combineLatest to derive a value from two sources; use forkJoin to wait for three parallel HTTP calls before rendering a dashboard.

🎯 Key Takeaways

  • An Observable is lazy — a function that produces values over time only once subscribed, with teardown on unsubscribe/complete.
  • Flattening operators differ in concurrency: switchMap (cancel-and-replace), mergeMap (parallel), concatMap (ordered queue), exhaustMap (ignore while busy) — picking the wrong one is the classic RxJS bug.
  • BehaviorSubject (has a current value, late subscribers get it immediately) is the right tool for modeling state; plain Subject has no memory.
  • combineLatest recombines on every source emission using the latest of each; forkJoin waits for all to complete (like Promise.all); merge just interleaves events.
  • Prefer async pipe / toSignal() / takeUntilDestroyed() over manual subscribe()/unsubscribe() to avoid leaks.
