Validation & Custom Controls Forms

What you'll master: custom synchronous validators, cross-field (group-level) validation done right, async validators with debouncing, runtime validator manipulation, centralized error-message strategy — and ControlValueAccessor, the interface that lets your own components plug into both forms systems as first-class controls. Prerequisites: Reactive Forms. This is the lesson that turns forms from "fields with attributes" into a validation architecture.

Part 1 — Custom Validators: Just Functions

import { AbstractControl, ValidationErrors, ValidatorFn } from '@angular/forms';

// A validator: control in → error object (or null) out. Pure function, nothing more:
export function forbiddenWords(words: string[]): ValidatorFn {
  return (control: AbstractControl<string>): ValidationErrors | null => {
    const hit = words.find(w => control.value?.toLowerCase().includes(w));
    return hit ? { forbiddenWord: { word: hit } } : null;   // error KEY + detail payload
  };
}

export const strongPassword: ValidatorFn = (control) => {
  const v: string = control.value ?? '';
  const failures: string[] = [];
  if (!/[A-Z]/.test(v)) failures.push('an uppercase letter');
  if (!/[0-9]/.test(v)) failures.push('a digit');
  if (!/[^A-Za-z0-9]/.test(v)) failures.push('a symbol');
  return failures.length ? { strongPassword: { missing: failures } } : null;
};

// Usage — exactly like built-ins:
password: ['', [Validators.required, Validators.minLength(10), strongPassword]],
username: ['', [Validators.required, forbiddenWords(['admin', 'root'])]],

// Template reads the payload for rich messages:
@if (pw.hasError('strongPassword')) {
  <p class="error">Password needs {{ pw.getError('strongPassword').missing.join(', ') }}.</p>
}

A validator is a pure function from control to error-or-null — the same contract behind Validators.required. Two conventions make yours production-grade: the factory form (forbiddenWords(words) returning the validator — a closure factory) for parameterized rules, and detail payloads in the error object so messages can say what's wrong, not just that something is. Being pure functions, validators unit-test in one line (expect(strongPassword(new FormControl('abc'))).toEqual({...})) — keep them in a shared validators/ module, tested once, reused everywhere.

Part 2 — Cross-Field Validation: the Group Owns the Rule

// The rule "passwords must match" belongs to NEITHER field — it's a relationship.
// Attach it to the GROUP:
export const passwordsMatch: ValidatorFn = (group: AbstractControl) => {
  const pw = group.get('password')?.value;
  const confirm = group.get('confirm')?.value;
  return pw === confirm ? null : { passwordsMatch: true };
};

form = this.fb.nonNullable.group({
  password: ['', [Validators.required, strongPassword]],
  confirm:  [''],
}, { validators: passwordsMatch });          // ← group-level options object

// Where does the error live? On the GROUP — not on either control:
form.hasError('passwordsMatch')      // true
form.controls.confirm.errors          // null! (its own validators all pass)

<!-- Template: read from the group, time it off the RELEVANT field's state: -->
@if (form.hasError('passwordsMatch') && form.controls.confirm.touched) {
  <p class="error">Passwords don't match.</p>
}
<!-- Optional pattern: the validator can also setErrors on the confirm control to
     drive its .ng-invalid styling — do it deliberately, and merge, don't clobber. -->

Cross-field rules are the reactive flavor's home turf (recall the template-forms ceiling): a group validator sees all children, runs whenever any child changes, and — the part everyone gets wrong once — puts its error on the group. The affected field still shows valid, so template display logic must read the group's error while gating on the field's touched state for correct timing. Other classic group rules: date ranges (start ≤ end), at-least-one-of (phone or email), sum constraints (percentages totaling 100), conditional requirements (reason required when rating < 3 — though enable/disable from last lesson often models that better).

Part 3 — Async Validators: Talking to the Server

// Async validator: control → Observable/Promise of error-or-null. THIRD constructor slot:
export function usernameAvailable(api: UserApi): AsyncValidatorFn {
  return (control) =>
    timer(400).pipe(                                   // debounce INSIDE the validator
      switchMap(() => api.checkUsername(control.value)),
      map(res => (res.available ? null : { usernameTaken: true })),
      catchError(() => of(null)),                      // server hiccup ≠ invalid input
      first(),                                         // async validators MUST complete
    );
}

username: ['',
  [Validators.required, Validators.minLength(3)],      // sync validators (slot 2)
  [usernameAvailable(inject(UserApi))],                // async validators (slot 3)
],

// While the observable is in flight the control's status is 'PENDING':
<input formControlName="username">
@if (form.controls.username.pending) { <span class="spinner">checking…</span> }
@if (form.controls.username.hasError('usernameTaken')) { <p class="error">Taken.</p> }
<button [disabled]="form.invalid || form.pending">Sign up</button>   // ← don't forget pending!

Async validators run after sync validators pass (no server call for an empty field — automatic short-circuiting), set the control to PENDING while in flight, and must complete (hence first()). Three production details are in the code because each is a real bug when missed: debounce inside via timer + switchMap (each keystroke re-runs the validator, cancelling the previous — the switchMap policy applied to validation), catchError → null (an availability-check outage shouldn't block signups — fail open for UX checks, and let the server's authoritative validation have the final word on submit), and gating submit on form.pending (or a fast submitter beats the check).

Part 4 — Runtime Validators & Centralized Messages

// Rules that change at runtime — add/remove validators, then RE-RUN them:
setRequiredIf(role: string) {
  const vat = this.form.controls.vatNumber;
  if (role === 'business') vat.addValidators(Validators.required);
  else vat.removeValidators(Validators.required);
  vat.updateValueAndValidity();               // ← forgotten 90% of the time: re-evaluate NOW
}

// CENTRALIZED error messages — stop copy-pasting @if chains per field:
const MESSAGES: Record<string, (e: any) => string> = {
  required:        () => 'This field is required.',
  email:           () => 'Please enter a valid email.',
  minlength:       e => `At least ${e.requiredLength} characters.`,
  strongPassword:  e => `Password needs ${e.missing.join(', ')}.`,
  usernameTaken:   () => 'That username is taken.',
};

export function firstErrorMessage(control: AbstractControl): string | null {
  if (!control.errors) return null;
  const [key, detail] = Object.entries(control.errors)[0];
  return (MESSAGES[key] ?? (() => 'Invalid value.'))(detail);
}
// One dumb component renders it everywhere:
// <app-field-error [control]="form.controls.email" />
// → @if (control().invalid && control().touched) { <p class="error">{{ msg() }}</p> }

The validator-manipulation API (addValidators/removeValidators/setValidators) handles rules that depend on runtime state — with updateValueAndValidity() as the non-optional final step, because validators are only consulted when a control revalidates. The message map is an architecture decision disguised as a utility: error keys become a vocabulary (every validator anywhere maps to one message definition), fields stop accumulating bespoke @if chains, i18n has a single integration point, and a ten-line FieldErrorComponent standardizes timing (invalid && touched) across the whole app. Most mature Angular codebases converge on exactly this shape.

Part 5 — ControlValueAccessor: Your Component as a Form Control

// The goal: <app-star-rating formControlName="rating" /> — YOUR component, native citizen.
// The bridge is the ControlValueAccessor interface:

@Component({
  selector: 'app-star-rating',
  template: `
    @for (s of stars(); track s) {
      <button type="button" class="star" [class.filled]="s <= value()"
              [disabled]="disabled()" (click)="pick(s)" (blur)="onTouched()">★</button>
    }
  `,
  providers: [{
    provide: NG_VALUE_ACCESSOR,               // the multi-token forms machinery looks up
    useExisting: forwardRef(() => StarRatingComponent),
    multi: true,
  }],
})
export class StarRatingComponent implements ControlValueAccessor {
  max = input(5);
  stars = computed(() => Array.from({ length: this.max() }, (_, i) => i + 1));
  value = signal(0);
  disabled = signal(false);

  private onChange: (v: number) => void = () => {};
  onTouched: () => void = () => {};

  // FORMS → COMPONENT: the form writes a value into you (setValue/patchValue/reset):
  writeValue(v: number | null) { this.value.set(v ?? 0); }
  // COMPONENT → FORMS: the form hands you callbacks to report through:
  registerOnChange(fn: (v: number) => void) { this.onChange = fn; }
  registerOnTouched(fn: () => void) { this.onTouched = fn; }
  // form.disable()/enable() reach you here:
  setDisabledState(isDisabled: boolean) { this.disabled.set(isDisabled); }

  pick(n: number) {
    this.value.set(n);
    this.onChange(n);                          // ← this is what makes the form see the value
  }
}

// Now ALL of it just works:
// rating: [0, Validators.min(1)]   +   <app-star-rating formControlName="rating" />
// touched on blur, disabled on form.disable(), value in form.value, validators apply.

ControlValueAccessor is the adapter contract (literally the adapter pattern) between the forms API and any widget: four methods define both directions — writeValue (form→you) and the registered callbacks (onChange/onTouched, you→form) plus disabled-state propagation. The provider registration uses the multi-token machinery: ngModel/formControlName directives look up NG_VALUE_ACCESSOR on the element to find their bridge (that's also how Angular wires native inputs — DefaultValueAccessor, CheckboxControlValueAccessor etc. are CVAs shipped in the box). Every design system you'll work in — Material included — is CVAs all the way down; writing one fluently is a distinguishing interview skill.

Part 6 — Deep Dive: Validation Architecture & the Trust Boundary, Completed

Where validation logic should live (the layered answer):

  VALIDATOR FUNCTIONS  (shared/validators/)   pure rules: format, ranges, relationships
       ↓ composed into
  FORM DEFINITIONS     (feature form builders)  which rules apply to which fields HERE
       ↓ displayed via
  MESSAGE MAP + FieldErrorComponent            one vocabulary, one timing policy
       ↓ mirrored by
  SERVER VALIDATION    (the real boundary)      re-checks EVERYTHING; client is UX only

Handling the server's verdict — mapping 422 field errors back onto the form:
  this.api.signup(payload).subscribe({
    error: (err: HttpErrorResponse) => {
      if (err.status === 422) {
        for (const [field, message] of Object.entries(err.error.fieldErrors)) {
          this.form.get(field)?.setErrors({ server: { message } });   // 'server' key in the
        }                                                              // message map renders it
      }
    },
  });
// The 'server' error clears on the next value change (validators re-run and overwrite) —
// exactly the UX you want: fix the field, error disappears.

The architecture stack keeps each concern where it's testable and reusable — and completes the trust-boundary story from template forms: the client validates for helpfulness, the server for truth, and the 422 → setErrors({server}) pattern is the seam where server truth flows back into client UX. Design your API error format (RFC 7807-style field errors) and this mapping together — teams that don't end up with toast-only server errors ("something failed") while the form sits there looking valid.

Senior framing: validation review heuristics: rules as pure, tested, shared functions (not inline lambdas per form); group rules on groups (a field validator reaching into control.parent is a smell — it runs at the wrong times and couples upward); async validators debounced, cancelling, failing open for UX checks; pending handled at submit; one message vocabulary with an i18n path; CVAs for every custom input (a "form control" that syncs via inputs/outputs plus manual glue code re-implements the CVA contract badly — no touched, no disabled propagation, no validator integration); and server errors mapped to fields, not toasts. The interview differentiator question is usually CVA ("how would you make your date-picker work with formControlName?") — the four-method contract answer, with the multi-token provider and the pick-vs-writeValue direction distinction, is a senior tell.

Interview Questions

Q1 (Beginner): What is a validator function's contract, and how do custom ones integrate with the errors system?

A sync validator is a pure function (control) => ValidationErrors | null: null means valid; an object means failed, keyed by rule name with optional detail payload ({ minlength: { requiredLength: 3 } }). Custom validators return the same shape, so they compose with built-ins in the validators array, surface through control.errors/hasError('key'), drive the .ng-invalid classes, and slot into any centralized message map — the error key is the integration contract.

Q2 (Beginner): Why must "passwords match" be a group validator, and where does its error appear?

Because the rule is a relationship between two controls: attached to one field it only re-runs when that field changes — edit the other field and the verdict goes stale. A group validator re-runs when any child changes and sees all values. Its error lands on the group (form.hasError('passwordsMatch')); the individual controls stay valid by their own rules — so templates read the group error while timing display off the relevant field's touched state (or the validator deliberately mirrors the error onto a control via setErrors).

Q3 (Mid): Walk through a production-grade async username-availability validator — name each operator and the failure mode it prevents.

timer(400): debounce inside the validator — prevents a request per keystroke. switchMap(() => api.check(value)): cancellation — a newer keystroke's validation cancels the stale request, preventing out-of-order verdicts. map to null | { usernameTaken: true }: the contract. catchError(() => of(null)): fail open — an availability-service outage shouldn't block signups (the server still enforces uniqueness authoritatively on submit). first(): async validators must complete or the control stays PENDING forever. Plus consumer-side: show pending state, and gate submit on form.pending as well as form.invalid.

Q4 (Mid): How do you surface server-side validation errors (a 422 with field errors) in the form, and why is this better than a toast?

Map them onto controls: iterate the response's field-error entries and form.get(field)?.setErrors({ server: { message } }), with a server key in the centralized message map rendering the text. The error clears naturally when the user edits (validators re-run and overwrite errors). Better than a toast because the feedback lands where the fix goes — field-anchored, screen-reader-associated, persistent until addressed — while a toast disappears, names no field, and leaves the form looking valid. Design the API's error shape and this mapping as one contract.

Q5 (Senior): Explain the ControlValueAccessor contract — the four methods, both data directions, and how the forms system finds your implementation.

CVA is the adapter between forms machinery and a widget. Form→component: writeValue(v) (programmatic writes: setValue/patchValue/reset) and setDisabledState(b) (form.disable/enable). Component→form: registerOnChange(fn) and registerOnTouched(fn) hand you callbacks — you call onChange(newValue) when the user changes the value (this is how the form learns) and onTouched() on blur (drives touched state). Discovery: you provide yourself under the NG_VALUE_ACCESSOR multi-token (with forwardRef); formControlName/ngModel directives inject the accessors present on their host element and wire the bridge. Native inputs work identically via built-in CVAs — your component becomes indistinguishable from them: validators, touched, disabled, and value flow all apply.

Q6 (Senior): Design the validation architecture for a large app: where do rules, messages, async checks, and server errors each live — and what does the PR checklist enforce?

Rules: pure validator functions (and factories) in a shared module, unit-tested in isolation — the single source for format/range/relationship logic. Composition: feature form builders attach rules to fields/groups; group rules on groups only. Messages: one key→message map (i18n-ready) + a FieldErrorComponent enforcing app-wide timing (invalid && touched, mark-all-touched on submit). Async checks: debounced, cancelling, fail-open validators for UX; the server remains authoritative. Server verdicts: a standardized 422 field-error shape mapped via setErrors({server}) — API contract and mapping designed together. PR checklist: no inline anonymous validators for reusable rules; no control.parent reaching; pending handled at submit; every custom input is a CVA; messages via the map (no bespoke @if chains); and a server-side mirror exists for every client rule that matters. The theme: validation is a layered subsystem with contracts, not per-form decoration.

1 (Easy): Write and test a noWhitespaceEdges validator (rejects values with leading/trailing whitespace, error payload shows the trimmed suggestion).
Solution
export const noWhitespaceEdges: ValidatorFn = (control) => {
  const v: string = control.value ?? '';
  return v !== v.trim() ? { whitespaceEdges: { suggestion: v.trim() } } : null;
};

// tests — validators are pure functions:
it('rejects padded values with a suggestion', () => {
  expect(noWhitespaceEdges(new FormControl('  ada '))).toEqual(
    { whitespaceEdges: { suggestion: 'ada' } });
});
it('accepts clean values and empties', () => {
  expect(noWhitespaceEdges(new FormControl('ada'))).toBeNull();
  expect(noWhitespaceEdges(new FormControl(''))).toBeNull();
});
2 (Easy): Add a date-range rule to a booking form (checkIn before checkOut, both required) with the error displayed once, under the second field, at the right time.
Solution
const dateRange: ValidatorFn = (group) => {
  const inV = group.get('checkIn')?.value, outV = group.get('checkOut')?.value;
  if (!inV || !outV) return null;              // let required handle empties — one error at a time
  return new Date(inV) < new Date(outV) ? null : { dateRange: true };
};

form = this.fb.nonNullable.group({
  checkIn:  ['', Validators.required],
  checkOut: ['', Validators.required],
}, { validators: dateRange });

}

3 (Medium): Build the async email-uniqueness validator with all five production properties (debounce, cancel, fail-open, completion, pending UX) and wire it into a signup form.
Solution
export function emailUnique(api: AuthApi): AsyncValidatorFn {
  return (control) => timer(400).pipe(                     // 1: debounce
    switchMap(() => api.emailExists(control.value)),       // 2: cancel stale checks
    map(exists => (exists ? { emailTaken: true } : null)),
    catchError(() => of(null)),                            // 3: fail open (server re-checks)
    first(),                                               // 4: must complete
  );
}

form = this.fb.nonNullable.group({
  email: ['', [Validators.required, Validators.email], [emailUnique(inject(AuthApi))]],
});

@if (form.controls.email.pending) { <span>Checking…</span> }          <!-- 5: pending UX -->
@if (form.controls.email.hasError('emailTaken')) { <p class="error">Already registered.</p> }

4 (Medium): Build the message map + FieldErrorComponent and refactor a field to use it — including a signal-based implementation.
Solution
// error-messages.ts — the single vocabulary:
export const MESSAGES: Record<string, (e: any) => string> = {
  required:  () => 'This field is required.',
  email:     () => 'Please enter a valid email.',
  minlength: e => `At least ${e.requiredLength} characters (you have ${e.actualLength}).`,
  emailTaken:() => 'This email is already registered.',
  server:    e => e.message,
};

@Component({
  selector: 'app-field-error',
  template: `@if (show()) { <p class="error">{{ message() }}</p> }`,
})
export class FieldErrorComponent {
  control = input.required<AbstractControl>();
  // poll control state into signals via events → statusChanges/valueChanges interop:
  private status = toSignal(this.control().statusChanges, { initialValue: 'VALID' });
  show = computed(() => { this.status();                       // re-eval on status change
    const c = this.control(); return c.invalid && c.touched; });
  message = computed(() => { this.status();
    const errors = this.control().errors;
    if (!errors) return '';
    const [key, detail] = Object.entries(errors)[0];
    return (MESSAGES[key] ?? (() => 'Invalid value.'))(detail);
  });
}

5 (Hard): Implement a TagInputComponent (chips + text entry, Enter adds, ✕ removes) as a full ControlValueAccessor whose value is string[], then use it with a max-5-tags validator.
Solution
@Component({
  selector: 'app-tag-input',
  template: `
    @for (tag of tags(); track tag) {
      <span class="chip">{{ tag }}
        <button type="button" [disabled]="disabled()" (click)="remove(tag)">✕</button>
      </span>
    }
    <input #box [disabled]="disabled()" (keydown.enter)="add(box); $event.preventDefault()"
           (blur)="onTouched()" placeholder="Add tag…">
  `,
  providers: [{ provide: NG_VALUE_ACCESSOR,
                useExisting: forwardRef(() => TagInputComponent), multi: true }],
})
export class TagInputComponent implements ControlValueAccessor {
  tags = signal<string[]>([]);
  disabled = signal(false);
  private onChange: (v: string[]) => void = () => {};
  onTouched: () => void = () => {};

  writeValue(v: string[] | null) { this.tags.set(v ?? []); }         // form → me
  registerOnChange(fn: any) { this.onChange = fn; }
  registerOnTouched(fn: any) { this.onTouched = fn; }
  setDisabledState(d: boolean) { this.disabled.set(d); }

  add(box: HTMLInputElement) {
    const t = box.value.trim();
    if (t && !this.tags().includes(t)) {
      this.tags.update(l => [...l, t]);
      this.onChange(this.tags());                                    // me → form
    }
    box.value = '';
  }
  remove(tag: string) {
    this.tags.update(l => l.filter(x => x !== tag));
    this.onChange(this.tags());
  }
}

// usage — validators apply to the ARRAY value like any control:
const maxTags = (n: number): ValidatorFn => c =>
  (c.value?.length ?? 0) > n ? { maxTags: { max: n } } : null;
tags: [[] as string[], [maxTags(5)]],
<app-tag-input formControlName="tags" />
<app-field-error [control]="form.controls.tags" />
6 (Hard): Code-review this validator setup — find all five defects and fix them: 
confirm: ['', [(c: AbstractControl) => c.value === c.parent?.get('password')?.value
                 ? null : { mismatch: true }]],
email: ['', [], [(c) => this.api.checkEmail(c.value).pipe(map(r => r.ok ? null : { taken: true }))]],
// … save(): if (this.form.invalid) return; this.api.submit(this.form.value);
Solution
// Defects:
// 1. Cross-field rule as a FIELD validator via control.parent — doesn't re-run when
//    password changes: stale mismatch verdicts.
// 2. That inline lambda is untested/unreusable — rules belong in shared functions.
// 3. Async email check: NO debounce (request per keystroke), NO switchMap cancellation
//    (implicit — new run cancels, but the http observable itself should be the inner),
//    NO catchError (API blip marks the field invalid → fails closed), NO first() needed
//    only if source completes — http does, but timer-debounced versions must add it.
// 4. save() ignores form.pending — fast submits beat the in-flight check.
// 5. Submitting this.form.value raw — if email is fine but disabled states/DTO drift
//    exist, payload is wrong; also no server-error mapping back.

form = this.fb.nonNullable.group({
  password: ['', [Validators.required, strongPassword]],
  confirm:  [''],
  email:    ['', [Validators.required, Validators.email],
                 [emailUnique(this.api)]],                 // the Part-3 armored validator
}, { validators: passwordsMatch });                        // 1+2: group rule, shared fn

save() {
  if (this.form.invalid || this.form.pending) {            // 4
    this.form.markAllAsTouched(); return;
  }
  this.api.submit(this.toPayload()).subscribe({            // 5: explicit mapping
    error: (err) => this.applyServerErrors(err),           //    + 422 → setErrors({server})
  });
}

🎯 Key Takeaways

  • Validators are pure functions (control → errors | null) — factories for parameters, detail payloads for rich messages, shared modules + unit tests for reuse.
  • Relationship rules go on the group (options object); the error lives on the group while display timing reads the relevant field's touched state.
  • Async validators (third slot): debounce + switchMap inside, fail open on errors, must complete, run only after sync rules pass — and submit gates on pending too.
  • Runtime rule changes need updateValueAndValidity(); error keys form one vocabulary rendered by a message map + FieldErrorComponent.
  • ControlValueAccessor (4 methods + NG_VALUE_ACCESSOR multi-provider) makes your components native form controls — value, touched, disabled, validators all integrate; every design system is CVAs underneath.
  • The server is the boundary: mirror critical rules server-side and map its 422 field errors back onto controls via setErrors({server}) — feedback lands where the fix goes.
