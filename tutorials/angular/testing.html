Testing Advanced

Angular's DI system (services-DI lesson) makes testing genuinely pleasant — you can substitute any dependency with a fake. Historically the toolchain was Karma + Jasmine; increasingly teams use Jest or the newer built-in Vitest-based builder. The concepts transfer regardless of runner: TestBed for components, direct instantiation for services, and harnesses for maintainable DOM queries.

Testing a Service (the easy case)

describe('CartService', () => {
  it('adds an item', () => {
    const service = new CartService();      // plain class — just instantiate it directly
    service.add({ id: '1', price: 10 });
    expect(service.items$().length).toBe(1);
  });
});
// If a service has no injected dependencies, you don't even need Angular's testing utilities —
// it's a plain class. This is a direct payoff of keeping business logic out of components (DI lesson).

Testing a Component with TestBed

import { TestBed } from '@angular/core/testing';

describe('CartBadgeComponent', () => {
  let fixture: ComponentFixture<CartBadgeComponent>;

  beforeEach(() => {
    TestBed.configureTestingModule({
      imports: [CartBadgeComponent],                          // standalone components self-import
      providers: [{ provide: CartService, useValue: fakeCartService }],  // swap the real service for a fake
    });
    fixture = TestBed.createComponent(CartBadgeComponent);
  });

  it('shows the item count', () => {
    fixture.detectChanges();                                   // triggers change detection manually
    const el: HTMLElement = fixture.nativeElement;
    expect(el.textContent).toContain('2');
  });
});
// TestBed builds a mini Angular testing module/injector. Overriding a provider with a FAKE is the whole
// reason DI makes testing easy — the component doesn't know or care it's talking to a fake.
🧪 Change detection doesn't run automatically in tests. You must call fixture.detectChanges() to apply bindings after setup (and again after any state change you expect to affect the view) — a very common "why isn't my assertion seeing the update" gotcha for people new to Angular testing.

Testing HTTP with HttpTestingController

import { HttpTestingController, provideHttpClientTesting } from '@angular/common/http/testing';

TestBed.configureTestingModule({ providers: [provideHttpClient(), provideHttpClientTesting()] });
const httpMock = TestBed.inject(HttpTestingController);

it('fetches products', () => {
  service.getProducts().subscribe(products => expect(products.length).toBe(2));

  const req = httpMock.expectOne('/api/products');   // assert the exact request was made
  req.flush([{ id: 1 }, { id: 2 }]);                  // manually provide the "response" — no real network call
  httpMock.verify();                                  // fail the test if any unexpected requests were made
});
// No real HTTP calls happen in tests — you control exactly what the "server" returns and assert on
// what was requested, making HTTP-dependent code fully deterministic and fast to test.

Testing Guards and Observables (async testing basics)

// Signals: read synchronously, easy — no special async handling needed.
expect(component.total()).toBe(30);

// Observables: use fakeAsync/tick for time-based operators (debounceTime, etc.):
it('debounces search', fakeAsync(() => {
  component.search('a');
  tick(300);                    // advance virtual time past the debounce window
  expect(searchSpy).toHaveBeenCalled();
}));
// fakeAsync + tick() let you deterministically test debounce/delay/interval-based code without
// actually waiting in real time — essential for testing RxJS-heavy components quickly and reliably.

Component Test Harnesses (maintainable DOM queries)

// Instead of fragile fixture.nativeElement.querySelector('.some-class'), Angular CDK/Material provide
// HARNESSES — a stable API for interacting with a component in tests, decoupled from its internal DOM:
const loader = TestbedHarnessEnvironment.loader(fixture);
const button = await loader.getHarness(MatButtonHarness.with({ text: 'Submit' }));
await button.click();
// If the component's internal markup changes, tests using a harness keep passing — only the harness
// implementation needs updating, not every test that touches that component.

Interview Questions

Q1: How does Angular's DI system make testing easier?

Because components and services declare their dependencies (via inject() or constructor parameters) rather than constructing them internally, a test can substitute ANY dependency with a fake or mock simply by providing a different implementation for the same injection token — the class under test has no idea it's talking to a fake, since it only ever interacted with an abstraction (the injected type), never with concrete construction details. In practice, TestBed.configureTestingModule's providers array lets you write { provide: CartService, useValue: fakeCartService } to swap out a real service for a lightweight test double that returns canned data instantly with no network calls, no timing dependencies, and full control over edge cases (errors, empty states, slow responses). This isolates the unit under test from its collaborators' real behavior, making tests fast, deterministic, and focused specifically on the logic being tested rather than accidentally also testing everything it depends on. It's a direct, practical payoff of the dependency-injection architecture covered in the services-DI lesson.

Q2: Why doesn't change detection run automatically in a component test, and what problems does forgetting it cause?

In a real running application, Angular's change detection is triggered automatically by the framework (traditionally via Zone.js patching async APIs, or via signal-driven fine-grained updates) whenever something might have changed. In the TestBed testing environment, that automatic triggering is deliberately NOT wired up by default — creating a component fixture with TestBed.createComponent() instantiates the component and its template but does not render/bind anything to the DOM until you explicitly call fixture.detectChanges(). This gives tests precise control: you can set up input properties or mock service state BEFORE the first render, call detectChanges() once to apply initial bindings, then make further changes and call it again to see their effect — rather than the framework re-rendering unpredictably mid-test. The classic mistake is writing assertions against fixture.nativeElement right after configuring the test but before calling detectChanges() at all (or after a state change without calling it again), which leads to confusing failures where the DOM appears empty or stale even though the component's underlying state is actually correct — the template simply hasn't been applied yet.

Q3: How do you test a component that makes an HTTP call, without hitting a real network?

Angular provides HttpTestingController specifically for this. You configure the testing module with provideHttpClient() and provideHttpClientTesting() (which swaps the real HTTP backend for a controllable mock), then inject HttpTestingController in the test. When the code under test calls a service method that internally uses HttpClient, no real network request is made — instead the request is captured and held pending. The test then asserts on that pending request with httpMock.expectOne('/api/products') (which also fails the test if the expected request wasn't made, or if the URL/method doesn't match), and manually resolves it by calling req.flush(mockData) to supply exactly the response data the test wants — letting you deterministically test success paths, error paths (via req.flush with an error status), and edge cases (empty arrays, malformed data) without depending on a real backend being available or fast. Finally, httpMock.verify() is called (often in an afterEach) to assert that no OTHER unexpected HTTP requests were made during the test, catching bugs where code fires extra or unintended requests. This makes HTTP-dependent logic fully testable, fast, and deterministic.

🏋️ Practice: unit test a plain service with no dependencies (direct instantiation); test a component with TestBed, overriding an injected service with a fake and calling fixture.detectChanges(); mock an HTTP call with HttpTestingController, flush a response, and verify no stray requests with httpMock.verify(); use fakeAsync/tick to test a debounced search; try a component harness for a UI library component and refactor the component's markup to confirm the harness-based test still passes.

🎯 Key Takeaways

  • DI makes testing easy — override any provider with a fake/mock; the class under test never knows the difference.
  • TestBed builds a test module/injector for components; call fixture.detectChanges() to apply bindings (it doesn't run automatically).
  • HttpTestingController mocks every HTTP call — assert with expectOne(), resolve with flush(), verify no strays with verify().
  • fakeAsync + tick() deterministically test time-based RxJS operators (debounceTime, delay) without real waiting.
  • Prefer testing observable behavior/output over implementation details; component harnesses give stable, DOM-decoupled test APIs.
