Routing & Navigation Routing & HTTP

What you'll master: the router's mental model (URL ⇄ component tree), route configuration with params and wildcards, routerLink vs programmatic navigation, reading route data reactively (including withComponentInputBinding — params as inputs), child routes and layouts, and query-param state. Prerequisites: Components, Signals, RxJS. The router is what turns a pile of components into an application — URLs that deep-link, back buttons that work, state you can share.

Part 1 — The Model: URL as the Source of Truth

// app.routes.ts — the URL → component-tree mapping:
export const routes: Routes = [
  { path: '',            component: HomeComponent },
  { path: 'products',    component: ProductListComponent },
  { path: 'products/:id', component: ProductDetailComponent },   // :id = route PARAM
  { path: 'about',       redirectTo: '', pathMatch: 'full' },
  { path: '**',          component: NotFoundComponent },         // wildcard LAST
];

// app.config.ts:  providers: [provideRouter(routes)]
// AppComponent template:
//   <a routerLink="/products">Products</a>
//   <router-outlet />        ← matched component renders HERE

The SPA deal: the browser never reloads. The router intercepts navigation,
updates the URL (History API), destroys the outlet's old component, creates
the new one. The URL is STATE — shareable, bookmarkable, back-button-able.

Client-side routing keeps the single page while restoring everything users expect from URLs. The mental model that prevents whole bug categories: the URL is the application's primary state, and the component tree is a projection of it. Navigation = URL change → router matches top-to-bottom (first match wins — order matters, wildcard last) → outlet swaps components (destroy/init lifecycles run). Design URLs like an API: /products/42/reviews should be meaningful, because users share it, tests target it, and analytics group by it.

Part 2 — Links, Active State & Programmatic Navigation

<!-- Declarative — for anything user-clickable: -->
<a routerLink="/products">Products</a>
<a [routerLink]="['/products', product.id]">{{ product.name }}</a>   <!-- segments array -->
<a routerLink="/search" [queryParams]="{ q: 'angular', page: 1 }">Search</a>

<!-- Highlight the current section (nav menus): -->
<a routerLink="/products" routerLinkActive="active">Products</a>
<a routerLink="/" routerLinkActive="active"
   [routerLinkActiveOptions]="{ exact: true }">Home</a>   <!-- else / matches EVERYTHING -->

// Programmatic — for navigation AFTER logic (submit, login, timeout):
private router = inject(Router);
this.router.navigate(['/products', id]);
this.router.navigate([], { queryParams: { page: 2 }, queryParamsHandling: 'merge' });
this.router.navigateByUrl('/login');

// WHY routerLink beats (click)="router.navigate(...)" on links:
// it renders a REAL <a href> — ctrl+click new tab, copy link, middle-click,
// SEO crawling, and screen readers all work. (click) handlers give you none of that.

The declarative/programmatic split is principled, not stylistic: routerLink produces genuine anchor semantics (accessibility + browser affordances), so it's for everything a user clicks; Router.navigate is for navigation that concludes logic — after a successful save, a login, a guard redirect. routerLinkActive handles nav highlighting (with exact for the root link, or it matches every URL), and queryParamsHandling: 'merge' is the detail that keeps existing query state (filters) when updating one param (page) — its absence is why "changing page resets my filters" bugs exist.

Part 3 — Reading Route Data: Params as Inputs

// THE MODERN WAY — withComponentInputBinding: route params bind DIRECTLY to inputs:
provideRouter(routes, withComponentInputBinding())

// /products/:id → ProductDetailComponent:
export class ProductDetailComponent {
  id = input.required<string>();               // ← the :id param, as a signal input!
  product = resource({
    params: () => ({ id: this.id() }),
    loader: ({ params }) => this.api.getProduct(params.id),
  });   // id changes → old fetch aborted → reload (signals.html)
}
// Query params and route data bind the same way: page = input<string>(); etc.

// THE CLASSIC WAY — ActivatedRoute (all existing code; still needed for complex cases):
private route = inject(ActivatedRoute);
id$ = this.route.paramMap.pipe(map(p => p.get('id')!));      // reactive stream
idOnce = this.route.snapshot.paramMap.get('id');             // one-time snapshot ⚠️

// THE CLASSIC BUG: snapshot in a component that stays alive across param changes.
// /products/1 → /products/2 REUSES the component instance (no destroy/recreate!) —
// ngOnInit doesn't re-run, the snapshot stays '1', the page shows stale data.
// Fix: consume params REACTIVELY (paramMap stream, or input binding — which is reactive).

The router's component-reuse optimization — same route, new params, same instance — is the source of the most classic Angular routing bug, and the reason params must be consumed as streams of values rather than one-time reads. withComponentInputBinding is the modern resolution: params/query-params/data arrive as ordinary signal inputs, reactive by construction, and the component stops knowing about ActivatedRoute at all (which also makes it trivially testable — set the input, done). Reach for ActivatedRoute when you need the fuller picture: matrix params, parent route data, or the URL segments themselves.

Part 4 — Child Routes & Layouts

export const routes: Routes = [
  { path: 'admin', component: AdminLayoutComponent,      // layout shell: sidebar + header
    children: [
      { path: '',        component: AdminDashboardComponent },
      { path: 'users',   component: UserManagementComponent },
      { path: 'users/:id', component: UserEditComponent },
      { path: 'settings', component: AdminSettingsComponent },
    ]},
];

// AdminLayoutComponent template — its OWN outlet for its children:
// <app-admin-sidebar />
// <main><router-outlet /></main>      ← /admin/users renders UserManagement HERE

URL structure maps to component NESTING:
  /admin/users/42 → AppComponent outlet → AdminLayout → its outlet → UserEdit
  Navigating users → settings: the LAYOUT persists (no re-render), only the
  inner outlet swaps. Layout state (scroll, sidebar collapse) survives.

// Relative navigation inside a feature:
this.router.navigate(['../', 'settings'], { relativeTo: this.route });
<a [routerLink]="['./', user.id]">Edit</a>      <!-- relative to current route -->

Child routes are the layout system: each level of URL nesting can contribute a component wrapping an outlet, so shells (admin chrome, settings tabs, wizard frames) render once and persist while their children swap — state preservation for free, and the route config documents the UI hierarchy. Relative links keep features portable (a feature mounted under a different parent path keeps working). This structure is also the skeleton that lazy loading hangs off: a child-route tree with its own providers, loaded on first navigation.

Part 5 — Query Params: Shareable UI State

// Filters, page, sort — UI state that SHOULD survive reload and sharing → query params:
// /products?q=laptop&page=2&sort=price

// Write (merge to preserve others):
setPage(page: number) {
  this.router.navigate([], {
    relativeTo: this.route,
    queryParams: { page },
    queryParamsHandling: 'merge',
  });
}

// Read reactively (input binding — or the stream):
q    = input<string>('');      // ?q=…      (withComponentInputBinding)
page = input('1');             // ?page=…   — NOTE: arrives as string!

results = resource({
  params: () => ({ q: this.q(), page: Number(this.page()) }),
  loader: ({ params }) => this.api.search(params),
});

The litmus test for "signal state vs URL state":
  Would a user want to SHARE or BOOKMARK this view, or lose it on refresh?
    filters/search/page/sort/selected-tab  → URL (query params)
    hover states, open dropdowns, drafts   → component/service state

Query params are the router's answer to shareable UI state — the search results page a colleague sends you renders their filters because the filters live in the URL. Making the URL the single source of truth (write via navigation, read reactively, derive everything downstream) avoids the two-sources bug where component state and URL disagree after back-navigation. Mind the string typing — everything in a URL is a string; parse numbers/booleans at the read boundary. Combined with resource/switchMap loading, this yields the standard senior-approved list-page architecture: URL → params → fetch → render, every arrow reactive.

Part 6 — Deep Dive: Router Events, Scroll, Reuse Strategy & SPA Realities

// Router EVENTS — the navigation lifecycle as a stream (progress bars, analytics):
inject(Router).events.pipe(
  filter(e => e instanceof NavigationStart || e instanceof NavigationEnd
           || e instanceof NavigationError || e instanceof NavigationCancel),
  takeUntilDestroyed(),
).subscribe(e => this.loading.set(e instanceof NavigationStart));

// Useful provideRouter features (composable withX():
withComponentInputBinding()          params → inputs (Part 3)
withInMemoryScrolling({ scrollPositionRestoration: 'enabled' })   // back-button scroll!
withViewTransitions()                View Transitions API page animations
withRouterConfig({ paramsInheritanceStrategy: 'always' })         // child sees parent params

// SPA realities checklist:
//  - the SERVER must serve index.html for ALL routes (deep links 404 otherwise) —
//    hosting config, not Angular config
//  - titles per route: { path: 'products', title: 'Products — Shop' } (a11y + tabs + SEO)
//  - focus management on navigation (screen readers don't announce SPA "pages" by default)
//  - RouteReuseStrategy: custom caching of route components (keep a heavy list page
//    alive behind a detail page) — powerful, rarely needed, know it exists

Router events expose the navigation machine for cross-cutting UI: global loading bars, analytics page-views, error toasts on failed navigations. The with* feature functions are the standalone-era configuration grammar (providers under the hood) — scroll restoration and per-route title are the two everyone should enable on day one. The SPA-realities list is what separates demo apps from products: deep links need server cooperation (the classic "works locally, 404s deployed" issue is hosting rewrites, not code), and navigation that a screen reader never announces is an accessibility failure the default setup doesn't fix for you.

Senior framing: routing questions probe architecture instincts. "Where does this state live?" — the URL-vs-store litmus test above is the expected framework. "How do features stay decoupled?" — feature-owned route files composed into the app table (next lesson), relative links, no cross-feature routerLink string literals scattered everywhere (centralize route-building helpers when paths churn). "What happens on param change?" — the component-reuse behavior, stated precisely, plus reactive consumption as the cure. And the trade-off framing for interviews: the router is Angular's biggest convention-over-configuration win — URL structure, code splitting boundaries (lazy routes), data loading (resolvers/resource), and access control (guards) all hang off one declarative table, which is why senior Angular architecture reviews usually start at app.routes.ts.

Interview Questions

Q1 (Beginner): Walk through what happens when a user clicks a routerLink in an SPA.

The router intercepts the click (no browser reload), matches the target URL against the route table top-to-bottom (first match wins), runs any guards/resolvers, updates the address bar via the History API, then updates each affected router-outlet: the old component is destroyed (ngOnDestroy), the new one instantiated and rendered. Unchanged parent layouts/outlets persist. The URL is now shareable, bookmarkable state — the back button replays the History entries through the same machinery.

Q2 (Beginner): Route params vs query params — when does each belong in the URL?

Route params (/products/:id) identify which resource — they're part of the resource's address, usually required, and changing them means a different entity. Query params (?q=laptop&page=2) parameterize the view of a resource — optional, order-free state like filters, search, pagination, sort, active tab. Heuristic: params that name the thing → path; params that tune the presentation → query string. Both should be consumed reactively, and both arrive as strings.

Q3 (Mid): Navigating from /products/1 to /products/2, the page shows product 1's data. Explain the bug and both fixes.

Same route, different param → the router reuses the component instance: no destroy, no re-init. Code that read the id once (snapshot.paramMap.get('id') in ngOnInit) never sees the change, so the stale product remains. Fix 1 (modern): withComponentInputBinding() — the param binds to a signal input; feed it into a resource/computed and loading re-runs per change (with stale-request cancellation built in). Fix 2 (classic): consume route.paramMap as a stream — paramMap.pipe(map(p => p.get('id')), switchMap(id => api.get(id))). Both are the same idea: params are streams, not constants.

Q4 (Mid): Design the URL and data flow for a searchable, filterable, paginated product list where sharing the link reproduces the exact view.

All view state in query params: /products?q=laptop&brand=apple&page=2&sort=price. Writes go through navigation only — filter controls call router.navigate([], { queryParams: { brand }, queryParamsHandling: 'merge' }) (merge preserves siblings; reset page to 1 when filters change). Reads are reactive: query params as inputs (or queryParamMap), parsed at the boundary (strings!), feeding a resource/switchMap fetch. The URL is the single source of truth — the form controls reflect it rather than owning state — so reload, share, back/forward, and bookmark all reproduce the view with zero extra code. This one-way loop (URL → params → fetch → render, writes via navigate) is the canonical list-page architecture.

Q5 (Senior): Your deployed Angular app 404s on every deep link but works when navigating from the home page. Diagnose, fix, and explain why dev mode hid it.

Client-side routes exist only in the browser: when a user opens /products/42 directly, the server receives that path, finds no such file, and 404s — client navigation from / works because the router handles it after index.html loaded. Fix: configure the host to rewrite all (non-asset) paths to index.html — nginx try_files $uri /index.html, or the SPA-fallback setting in Netlify/Vercel/S3+CloudFront/IIS — after which the router boots and resolves the deep path. Dev mode hid it because the CLI dev server includes exactly this HTML5-fallback behavior by default. Follow-ups worth naming: keep real 404 semantics via the ** route (and proper status codes need SSR), and don't rewrite asset/API paths.

Q6 (Senior): What accessibility work does SPA navigation require that server-rendered sites get for free, and how do you implement it in Angular?

Full page loads give screen readers a hard signal: new document, announce title, focus resets to top. SPA navigation swaps DOM silently — a blind user clicks a link and hears… nothing. Required work: (1) per-route titles — the route title property (or a TitleStrategy for dynamic ones) so the tab and assistive tech name the page; (2) focus management — on NavigationEnd, move focus to the new view's main heading or a skip-target (tabindex="-1" on it) so reading resumes at the content; (3) optionally an aria-live region announcing "navigated to Products"; (4) scroll restoration (withInMemoryScrolling) so keyboard/low-vision users aren't dumped mid-page; (5) real links via routerLink (href semantics). Teams bundle 1–3 into one router-events service — a dozen lines that decide whether the app is usable non-visually.

export class ProductListComponent {
  private router = inject(Router);
  private route = inject(ActivatedRoute);
  private api = inject(ProductApi);

  // READ: URL → signals (via input binding; strings parsed at the boundary):
  q    = input('');
  page = input('1');

  results = resource({
    params: () => ({ q: this.q(), page: Number(this.page()) || 1 }),
    loader: ({ params }) => this.api.search(params.q, params.page),
  });

  // WRITE: user actions → navigation (never directly into local state):
  search(q: string) {
    this.router.navigate([], { relativeTo: this.route,
      queryParams: { q: q || null, page: 1 },        // null REMOVES the param when empty;
      queryParamsHandling: 'merge' });               // new search resets pagination
  }
  goToPage(page: number) {
    this.router.navigate([], { relativeTo: this.route,
      queryParams: { page }, queryParamsHandling: 'merge' });
  }
}

// 2 + 3. One root-level service wires events once:
@Injectable({ providedIn: 'root' })
export class NavigationUxService {
  readonly navigating = signal(false);
  constructor(router: Router) {
    router.events.pipe(takeUntilDestroyed()).subscribe(e => {
      if (e instanceof NavigationStart) this.navigating.set(true);
      if (e instanceof NavigationEnd || e instanceof NavigationCancel
          || e instanceof NavigationError) {
        this.navigating.set(false);
        if (e instanceof NavigationEnd) {
          // focus the new view's heading so screen readers resume at the content:
          queueMicrotask(() => {
            const h = document.querySelector<HTMLElement>('main h1');
            h?.setAttribute('tabindex', '-1');
            h?.focus();
          });
        }
      }
    });
  }
}
// AppComponent: ux = inject(NavigationUxService);
// template: @if (ux.navigating()) { <div class="loading-bar"></div> }  <main><router-outlet /></main>
// A dozen lines that make SPA navigation announce itself — title + focus + progress.

🎯 Key Takeaways

  • The URL is the app's primary state; the component tree is its projection. Routes match top-to-bottom (specific → general, ** last), rendering into nested outlets.
  • routerLink for user-clickable navigation (real anchors: a11y + browser affordances); Router.navigate for logic-concluding navigation; routerLinkActive with exact on home.
  • Params are streams: same-route navigation reuses component instances, so consume via withComponentInputBinding (params as signal inputs → resource) or paramMap — never snapshots in reusable routes.
  • Child routes = persistent layouts: shells render once, inner outlets swap, state survives; relative links keep features portable.
  • Shareable view state (filters/page/sort) lives in query params — write via navigate with merge, read reactively, parse strings at the boundary.
  • Production routing = router events (loading/analytics), scroll restoration, per-route titles, focus management, and server rewrites to index.html for deep links.
