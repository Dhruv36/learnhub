Routing & Guards Intermediate

The Angular Router turns a URL into a rendered component tree, giving a single-page app real navigation: back/forward buttons, bookmarkable/shareable URLs, and lazy-loaded chunks so users only download the code for the page they're on. Guards then let you control access and unsaved-changes protection declaratively at the route level.

Defining Routes

// app.routes.ts
export const routes: Routes = [
  { path: '', component: HomeComponent },
  { path: 'products', component: ProductListComponent },
  { path: 'products/:id', component: ProductDetailComponent },   // route PARAMETER
  { path: 'admin', loadChildren: () => import('./admin/admin.routes').then(m => m.ADMIN_ROUTES) },
  { path: '**', component: NotFoundComponent },                  // wildcard — always LAST
];
// bootstrapApplication(AppComponent, { providers: [provideRouter(routes)] });
// <router-outlet /> in a template marks WHERE the matched route's component renders.

Reading Route Data & Navigating

import { ActivatedRoute, Router, RouterLink } from '@angular/router';
import { toSignal } from '@angular/core/rxjs-interop';

export class ProductDetailComponent {
  private route = inject(ActivatedRoute);
  id = toSignal(this.route.paramMap.pipe(map(p => p.get('id'))));   // reactive route param as a signal

  private router = inject(Router);
  goBack() { this.router.navigate(['/products']); }        // imperative navigation
}
// <a [routerLink]="['/products', product.id]">View</a>   <!-- declarative navigation -->
// <a routerLink="/products" routerLinkActive="active">Products</a>   <!-- highlight the active link -->

Lazy Loading — Why Routing Matters for Performance

// loadChildren / loadComponent split your app into separate JS chunks, downloaded ONLY when that
// route is visited:
{ path: 'admin', loadComponent: () => import('./admin/admin.component').then(m => m.AdminComponent) }
// A user who never visits /admin never downloads the admin bundle. This is THE main lever for keeping
// initial bundle size (and load time) small in a large Angular app — standalone components made
// per-route/per-component lazy loading much simpler than the old NgModule-based approach.
📦 Route-based code splitting is a production non-negotiable at scale. Without lazy loading, every feature — including ones most users never touch — ships in the initial bundle. Structuring routes so admin panels, rarely-used settings pages, and heavy third-party-library features load on demand is one of the highest-leverage performance techniques available, and it comes almost for free with the modern loadComponent/loadChildren syntax.

Guards — Controlling Navigation

canActivateBefore entering a routeAuth checks — redirect if not logged in
canActivateChildBefore entering a CHILD routeProtect a whole subtree at once

// Modern guards are plain FUNCTIONS (not classes) — functional guards:
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  const router = inject(Router);
  return auth.isLoggedIn() ? true : router.parseUrl('/login');   // redirect if not authenticated
};

{ path: 'admin', canActivate: [authGuard], loadComponent: () => import('./admin.component')... }

Interview Questions

Q1: How does the Angular Router work, and why is lazy loading important?

The Router maps URL paths to components via a routes configuration array, and a <router-outlet /> placeholder in a template marks where the currently matched route's component is rendered. Navigating (via routerLink declaratively, or Router.navigate() imperatively) updates the browser URL and swaps the component rendered in the outlet, giving a single-page app real navigation semantics — working back/forward buttons, deep-linkable and bookmarkable/shareable URLs — without full page reloads. Lazy loading, via loadComponent or loadChildren, tells the build to split a route's code into a separate JavaScript chunk that's fetched only when a user actually navigates to that route, rather than bundling everything into the initial download. This matters enormously for performance at scale: without it, every feature in the app — including admin panels, settings pages, and rarely-used flows most users never touch — ships in the initial bundle, inflating load time for everyone. Lazy loading is one of the highest-leverage techniques for keeping a large app's initial load fast, and standalone components made per-route lazy loading substantially simpler than the older NgModule-based approach.

Q2: Explain the different route guards and what each is for.

Guards are functions attached to route definitions that control navigation at specific points. canActivate runs before a route is entered and is the classic tool for authentication/authorization checks — return true to allow entry or a UrlTree (via router.parseUrl) to redirect elsewhere, such as to a login page. canActivateChild applies the same check to an entire subtree of child routes at once, useful for protecting a whole nested section (like everything under /admin) without repeating the guard on each child. canDeactivate runs before LEAVING the current route and is used to prevent accidental navigation away from unsaved work, typically prompting a confirmation dialog. canMatch runs even before route matching decides which configuration to use, letting you choose between alternate route definitions — for example serving a different implementation behind a feature flag. resolve prefetches data before the route activates, so the component doesn't render until its required data is ready, avoiding a flash of empty/loading state. Modern Angular guards are plain functions (like CanActivateFn) that use inject() to access services, rather than the older class-based guard pattern.

Q3: You navigate from /products/1 to /products/2 but the component doesn't update. Why, and how do you fix it?

Angular reuses the same component instance when navigating between routes that match the same route configuration but differ only in a parameter — it doesn't destroy and recreate ProductDetailComponent just because the :id changed from 1 to 2, since that would be wasteful. The bug happens when the component reads the route parameter only ONCE, typically via a snapshot (this.route.snapshot.paramMap.get('id')) in ngOnInit or a constructor: that snapshot captures the param's value at that one moment and never updates on subsequent navigations to the same reused component, so the displayed data goes stale. The fix is to read route parameters REACTIVELY instead of as a one-time snapshot — subscribe to this.route.paramMap (an Observable that emits on every param change) and react to new values, or, in modern signal-based code, wrap it with toSignal(...) so the component's template automatically reflects each new id as navigation occurs. The underlying lesson is that route data must be treated as a reactive stream, not a value read once, precisely because Angular's component reuse means the same instance can live across multiple navigations.

🏋️ Practice: set up routes with a parameterized detail page and a wildcard 404, wired to a router-outlet; add routerLink navigation with routerLinkActive highlighting; convert an eagerly-loaded feature route to loadComponent and confirm (via devtools network tab) it only downloads on visit; write a functional canActivate auth guard that redirects unauthenticated users; add a canDeactivate guard that warns on unsaved form changes; read a route param reactively with toSignal and navigate between two similar detail routes to confirm it updates.

🎯 Key Takeaways

  • Routes map URL paths to components; <router-outlet /> renders the match; routerLink/Router.navigate() navigate declaratively/imperatively.
  • Lazy loading (loadComponent/loadChildren) splits routes into on-demand chunks — the top lever for initial bundle size at scale.
  • Guards: canActivate (entry/auth), canActivateChild (subtree), canDeactivate (unsaved-changes confirm), canMatch (alt configs), resolve (prefetch data).
  • Modern guards are plain functions using inject(), not classes.
  • Read route params reactively (paramMap / toSignal), not as a one-time snapshot — Angular reuses component instances across param-only navigations.
