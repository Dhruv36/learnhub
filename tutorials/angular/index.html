Components & Standalone Angular Foundations

What you'll master: what Angular actually is (a compiler + runtime, not a script you include), the component-tree mental model every other lesson builds on, the anatomy of a standalone component, how an app boots from main.ts to pixels, style encapsulation, and the CLI workflow. Prerequisites: solid JavaScript (classes, modules, decorators are just functions) and TypeScript basics — Angular is TypeScript-first by design. Angular is the enterprise front-end framework: opinionated, batteries-included, and built to keep 500-developer codebases coherent.

Part 1 — What Angular Is: a Compiler and a Component Tree

React:  a LIBRARY — you assemble routing, forms, http, state from the ecosystem.
Angular: a FRAMEWORK — router, forms, HttpClient, DI, testing, i18n ship IN the box,
         with one blessed way to use each. Less choosing, more conventions.

Your app = a TREE of components:            <app-root>
                                            ├── <app-header user="…">
                                            ├── <app-product-list>
                                            │     ├── <app-product-card> ×N
                                            │     └── <app-pagination>
                                            └── <app-footer>
Data flows DOWN via inputs; events flow UP via outputs. One-way, predictable.

And crucially: Angular templates are COMPILED (AOT — ahead of time, at build).
Your HTML-like template becomes optimized JS instructions; errors surface at
BUILD time, not when a user hits the page.

Two ideas orient everything else. First, the component tree: every screen is composed of components — a class holding state and logic, plus a template describing its view — and the framework's whole job is keeping each component's DOM in sync with its state. Second, Angular is a compiler: templates aren't strings parsed in the browser; the Angular compiler turns them into efficient rendering instructions at build time (AOT), which is why a typo like {{ usre.name }} is a red build error instead of a silent blank in production, and why the framework itself can be tree-shaken down — the runtime only ships what your templates actually use.

Part 2 — Anatomy of a Standalone Component

// product-card.component.ts
import { Component, input, output } from '@angular/core';
import { CurrencyPipe } from '@angular/common';

@Component({
  selector: 'app-product-card',            // the custom HTML tag this component answers to
  imports: [CurrencyPipe],                 // what THIS template uses — explicit, tree-shakable
  template: `
    <article class="card">
      <h3>{{ product().name }}</h3>
      <p>{{ product().price | currency:'USD' }}</p>
      <button (click)="buy.emit(product().id)">Add to cart</button>
    </article>
  `,
  styles: `.card { border: 1px solid #ddd; padding: 1rem; }`,
})
export class ProductCardComponent {
  product = input.required<Product>();     // typed data IN (from the parent)
  buy = output<number>();                  // typed events OUT (to the parent)
}

// Used by a parent template simply as:
// <app-product-card [product]="p" (buy)="addToCart($event)" />

A component is a TypeScript class with a @Component decorator gluing three things together: a selector (its tag name — prefixed app- by convention to never collide with real HTML), a template (inline here; or templateUrl for a separate .html file — teams usually split once templates exceed a dozen lines), and styles. The imports array is the standalone era's key move: each component declares exactly which components, directives, and pipes its own template uses. Nothing is ambiently available; dependencies are explicit, navigable, and tree-shakable. The class body is deliberately thin — state and logic that belongs to the view; anything reusable or shared moves into services.

Standalone vs NgModules: before Angular 14–17, every component had to be declared in an @NgModule — a grouping class that managed declarations/imports/exports, and the #1 source of beginner confusion ("why can't the template see my pipe?"). Standalone components import what they need directly and are the default since v17 (in v19+, standalone: true is implicit). You'll still meet NgModules in older codebases: read them as "a bag of components that share template dependencies" — and migrate incrementally, since the two interoperate.

Part 3 — How an App Boots: main.ts → Rendered Tree

// main.ts — the real entry point:
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { appConfig } from './app/app.config';

bootstrapApplication(AppComponent, appConfig);   // "mount AppComponent onto <app-root>"

// app.config.ts — app-wide services & features, registered ONCE:
import { ApplicationConfig } from '@angular/core';
import { provideRouter } from '@angular/router';
import { provideHttpClient } from '@angular/common/http';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),        // routing (lesson: routing.html)
    provideHttpClient(),          // HttpClient (lesson: http.html)
  ],
};

// index.html contains just:  <body><app-root></app-root></body>
// The build injects your compiled bundles; bootstrap finds <app-root>,
// instantiates AppComponent, renders its template, then ITS children, recursively.

The boot sequence demystifies the "magic": the browser loads a nearly-empty index.html, your compiled JS runs bootstrapApplication, Angular instantiates the root component into <app-root>, and rendering recurses down the tree — each child selector in a template becomes a live component instance. app.config.ts is where application-wide capabilities are switched on via provide* functions; it replaces the old root NgModule and is the pattern you'll see in every modern lesson in this track. Everything from here is just the tree re-rendering in response to state changes — how Angular knows when is the subject of Change Detection and Signals.

Part 4 — Building a Component Step by Step

// 1. Generate it (CLI does naming, files, wiring):
//    ng generate component counter        → counter.component.ts|html|css|spec.ts

// 2. State + behavior in the class (signals are the modern default — full lesson later):
import { Component, signal, computed } from '@angular/core';

@Component({
  selector: 'app-counter',
  template: `
    <p>Count: {{ count() }} — double: {{ double() }}</p>
    <button (click)="increment()">+1</button>
    <button (click)="count.set(0)" [disabled]="count() === 0">Reset</button>
  `,
})
export class CounterComponent {
  count = signal(0);                        // reactive state
  double = computed(() => this.count() * 2);// derived state — recomputes automatically
  increment() { this.count.update(c => c + 1); }
}

// 3. Use it anywhere — the parent imports it:
@Component({
  selector: 'app-dashboard',
  imports: [CounterComponent],              // ← without this line: "app-counter is not a known element"
  template: `<h2>Dashboard</h2><app-counter />`,
})
export class DashboardComponent {}

This tiny component already shows the full loop: signal(0) holds reactive state, the template reads it with count(), the (click) binding calls a class method, the signal updates, and Angular re-renders precisely what depended on it. Note the error every Angular developer meets in week one: using a component's tag without adding it to the parent's imports gives "'app-counter' is not a known element" — the fix is always the imports array, and the error existing at build time is the compiler earning its keep.

Part 5 — Styles & View Encapsulation

@Component({
  selector: 'app-card',
  template: `<h3 class="title">Hi</h3>`,
  styles: `.title { color: rebeccapurple; }`,   // applies ONLY inside this component
})
// Rendered DOM (Emulated encapsulation — the default):
// <app-card _nghost-ng-c123><h3 class="title" _ngcontent-ng-c123>Hi</h3></app-card>
// and the CSS is rewritten to:  .title[_ngcontent-ng-c123] { … }
// → your .title can NEVER leak out and restyle someone else's .title.

// Options: ViewEncapsulation.Emulated (default) | ShadowDom (real shadow root) | None (global)
// Component styles also can't reach INTO children — pass state down and let the child
// style itself, or use CSS custom properties (--card-accent) which DO pierce boundaries.

Angular scopes each component's CSS by rewriting selectors with generated attributes — emulated encapsulation. The payoff is huge in large apps: components are styling islands, so no team's .title rule can break another's. The flip side surprises newcomers: you also can't style a child component's internals from the parent (the old ::ng-deep hack is deprecated). The sanctioned tools are CSS custom properties (which inherit through boundaries by design), component inputs that toggle classes, or global styles in styles.css for genuinely app-wide rules (resets, typography, utility classes).

Part 6 — Deep Dive: the CLI, Project Layout & What "Enterprise Framework" Buys

ng new shop --style=scss     # scaffold: strict TS, testing, build pipeline — zero config by you
ng serve                     # dev server, HMR, on :4200
ng generate component cart   # consistent naming/files — 'ng g c cart' for short
ng build                     # production: AOT compile, tree-shake, minify, hash filenames
ng test                      # unit tests (Karma/Jest per setup)
ng update @angular/core @angular/cli   # AUTOMATED migrations rewrite your code across versions

src/app/
  app.component.ts|html      # root shell
  app.config.ts              # app-wide providers
  app.routes.ts              # route table
  features/                  # one folder per feature (products/, cart/, checkout/)
  shared/                    # cross-feature components/pipes/directives
  core/                      # singleton services (auth, api, logging)

The CLI is not a convenience — it's the enforcement arm of Angular's opinions. Every Angular codebase has the same file naming, the same build setup, the same test wiring, which is precisely why large organizations pick it: an engineer moving between Angular teams is productive on day one, and ng update's automated migrations are why Angular's twice-yearly major versions are routine maintenance rather than rewrites. The features/shared/core layout is the community-standard scaling structure — features stay isolated (and lazy-loadable, see Lazy Loading), shared stays dumb-reusable, core stays singleton.

Senior framing: "React or Angular?" is an organizational question, not a technical scorecard. Angular's value proposition is convention over configuration at scale: DI for testable service boundaries, one router/forms/http story instead of five ecosystem options, compiler-enforced templates, and codemod-backed upgrades. The cost is ceremony on small apps and a steeper initial ramp (TS + DI + RxJS + templates all at once). Strong answer in an interview: name the trade-off, then note that modern Angular (standalone components, signals, functional guards/interceptors, zoneless) has been systematically shedding the ceremony while keeping the guarantees — the gap between "Angular in 2018" and "Angular now" is the size of a framework.

Interview Questions

Q1 (Beginner): What are the parts of an Angular component, and how do components form an application?

A component = a TypeScript class (state + methods) with a @Component decorator binding it to a selector (its custom tag), a template (its view), styles, and — in standalone Angular — an imports array declaring what its template uses. The app is a tree: bootstrap mounts the root component; each selector found in a template instantiates a child. Data flows down through inputs, events flow up through outputs, and Angular keeps each component's DOM in sync with its state.

Q2 (Beginner): Template shows "'app-user-badge' is not a known element". The component exists. What's wrong?

The component whose template uses <app-user-badge> hasn't imported it: in standalone Angular every template dependency must be listed in that component's imports array (in NgModule apps, declared/imported via the module). Add UserBadgeComponent to imports. The error appearing at build time is Angular's AOT template compilation working as designed.

Q3 (Mid): What did standalone components change versus NgModules, and why was it done?

NgModules made template-dependency management indirect: a component could use whatever its declaring module imported — so understanding one template meant reading module graphs, and refactoring meant module bookkeeping. Standalone components declare their own template dependencies in their imports, making them self-describing, easier to lazy-load (a route can point straight at a component), friendlier to tree-shaking, and massively less boilerplate. App-wide setup moved from the root module to app.config.ts with provide* functions. Since v17 it's the default; the two styles interoperate for incremental migration.

Q4 (Mid): Explain view encapsulation — how does Angular stop component styles from leaking, and what can't it do?

Default (Emulated) encapsulation rewrites each component's CSS selectors with a generated attribute (.title[_ngcontent-ng-c123]) and stamps that attribute on the component's DOM — so styles apply only inside the component without real Shadow DOM. What it can't do: parent styles can't reach into child components either (and ::ng-deep is deprecated because it escapes scoping globally). Cross-boundary theming uses CSS custom properties (which inherit through), inputs toggling classes, or intentional global styles. Alternatives: ShadowDom (real shadow root) or None (global).

Q5 (Senior): What does it mean that Angular is "AOT-compiled", and what do you gain from it?

Templates are compiled at build time into imperative rendering instructions (Ivy), not shipped as HTML parsed at runtime. Gains: template errors (bad bindings, wrong input types with strictTemplates) fail the build instead of users' sessions; no compiler ships to the browser (smaller bundles, faster startup, and no template-injection surface); instructions are tree-shakable so unused framework features drop out; and the compiler can optimize per-template. JIT compilation still exists for dev tooling edge cases, but production Angular is AOT-only in practice.

Q6 (Senior): Your company is choosing between Angular and React for a 60-developer, multi-year product. Argue Angular's case honestly — including its costs.

Angular's case at that scale: one blessed answer for routing, forms, HTTP, DI, testing and i18n means 60 developers converge on the same patterns instead of relitigating libraries per team; the CLI and style guide enforce uniform structure, so cross-team mobility is cheap; DI makes service boundaries mockable and testable by default; compiler-checked templates catch a bug class React finds at runtime; and ng update codemods make major upgrades routine — critical over a multi-year horizon. Honest costs: steeper onboarding (TS + DI + RxJS at once), more ceremony than React for small tools, a smaller hiring pool in some markets, and less ecosystem flexibility when the blessed way doesn't fit. The decision hinges on whether enforced consistency is worth more than flexibility — at 60 devs, it usually is.

3 (Medium): Build a BadgeComponent with a required input label and an optional input tone ("info" | "warn", default "info") that sets a CSS class. Show parent usage.
Solution
import { Component, input } from '@angular/core';

@Component({
  selector: 'app-badge',
  template: `<span class="badge" [class.warn]="tone() === 'warn'">{{ label() }}</span>`,
  styles: `
    .badge { padding: 2px 8px; border-radius: 8px; background: #e3f2fd; }
    .badge.warn { background: #fff3cd; }
  `,
})
export class BadgeComponent {
  label = input.required<string>();                 // parent MUST provide
  tone = input<'info' | 'warn'>('info');            // typed, defaulted
}

// Parent (after adding BadgeComponent to its imports):
// <app-badge label="New" />
// <app-badge label="Deprecated" tone="warn" />
// Forgetting label → compile-time error: required input has no value. The compiler again.
4 (Medium): A teammate's PR styles the child component's internals from the parent CSS with ::ng-deep .item-title { … }. Explain the problem and refactor it two sanctioned ways.
Solution
/* Problem: ::ng-deep disables encapsulation for that rule — once compiled it's GLOBAL,
   so it restyles every .item-title in the app, and silently breaks when the child renames
   its classes. It's deprecated. */

/* Fix A — CSS custom property (pierces boundaries BY DESIGN): */
/* parent css   */  app-item { --item-title-color: rebeccapurple; }
/* child css    */  .item-title { color: var(--item-title-color, inherit); }

/* Fix B — make it part of the child's API: */
// child:  emphasis = input(false);
//         <h3 class="item-title" [class.emphasis]="emphasis()">
// parent: <app-item [emphasis]="true" />
/* Both make the styling contract explicit instead of reaching into private markup. */
5 (Hard): Build the counter from Part 4, then add a step input (default 1) and a limitReached output that fires once when count reaches 10. Wire it in a parent.
Solution
@Component({
  selector: 'app-counter',
  template: `
    <p>Count: {{ count() }}</p>
    <button (click)="increment()" [disabled]="count() >= 10">+{{ step() }}</button>
  `,
})
export class CounterComponent {
  step = input(1);
  limitReached = output<void>();
  count = signal(0);

  increment() {
    this.count.update(c => Math.min(10, c + this.step()));
    if (this.count() === 10) this.limitReached.emit();   // guard by disabled → fires once
  }
}

// Parent template:
// <app-counter [step]="2" (limitReached)="onMaxed()" />
// One-way flow on display: state lives in the child, parent hears about it via the output.

Solution

🎯 Key Takeaways

  • Angular is a batteries-included, compiler-driven framework: the app is a component tree, and templates are AOT-compiled so binding errors fail the build, not the user.
  • A standalone component = class + @Component({selector, template, styles, imports}); the imports array makes every template's dependencies explicit and tree-shakable.
  • Boot flow: main.ts → bootstrapApplication(AppComponent, appConfig) → recursive render; app-wide features are enabled by provide* functions in app.config.ts.
  • Emulated view encapsulation makes components styling islands — style children via CSS custom properties or inputs, never ::ng-deep.
  • The CLI enforces the conventions that are Angular's actual product: ng generate, strict templates, and ng update codemods that make major upgrades routine.
  • Structure at scale: core/ singletons, shared/ dumb reusables, features/ isolated and lazy-loaded per route.
