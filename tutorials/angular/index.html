Components & Templates Basics

Angular is a batteries-included, TypeScript-first framework for building large front-end applications. Its fundamental building block is the component: a class with a template and styles that renders a piece of UI. Understanding components, templates, and how they compose into a tree is the foundation for everything else. Prerequisite: the JavaScript track plus TypeScript basics.

A Component = Class + Template + Metadata

import { Component, signal } from '@angular/core';

@Component({
  selector: 'app-counter',            // the custom HTML tag: <app-counter>
  standalone: true,                   // modern Angular: no NgModule needed
  template: `
    <button (click)="increment()">Clicked {{ count() }} times</button>
  `,
  styles: [`button { padding: 8px 16px; }`],
})
export class CounterComponent {
  count = signal(0);                  // reactive state (Signals lesson)
  increment() { this.count.update(n => n + 1); }
}
// The @Component DECORATOR attaches metadata to the class. The selector makes it usable as a tag;
// the template is the HTML; the class holds state and behavior. That's the whole model.
🧩 Everything in an Angular UI is a component tree. A root component (<app-root>) hosts child components, which host their own children. Each component owns a slice of the screen, its state, and its logic. Modern Angular uses standalone components (no NgModule boilerplate) — a component declares its own dependencies via imports. This composition of self-contained components is how you build and reason about large UIs.

Modern Control Flow (@if / @for / @switch)

<!-- Angular 17+ built-in control flow — replaces *ngIf / *ngFor -->
@if (user(); as u) {
  <p>Welcome {{ u.name }}</p>
} @else {
  <p>Please log in</p>
}

@for (item of items(); track item.id) {   <!-- track is REQUIRED — identity for efficient updates -->
  <li>{{ item.label }}</li>
} @empty {
  <li>No items</li>
}
// The new block syntax is faster and clearer than the old structural directives, and `track` prevents
// needless DOM re-creation by keying list items on a stable identity.

Component Inputs & Outputs (parent ↔ child)

import { Component, input, output } from '@angular/core';

@Component({ selector: 'app-item', standalone: true, template: `
  <div (click)="select.emit(id())">{{ label() }}</div>
`})
export class ItemComponent {
  id = input.required<number>();     // data flows DOWN from parent via [id]
  label = input('');                 // an optional input with a default
  select = output<number>();         // events flow UP to parent via (select)
}
// Usage in a parent:  <app-item [id]="1" [label]="'A'" (select)="onSelect($event)" />
// Data down via inputs, events up via outputs — the unidirectional data-flow model.

Interview Questions

Q1: What is an Angular component and what are standalone components?

A component is Angular's fundamental UI building block: a TypeScript class decorated with @Component that binds together a template (the HTML view), optional styles, and metadata like a selector (the custom tag it renders as). The class holds the component's state and behavior, and the template binds to that state, so each component owns a self-contained slice of the screen. Applications are built as a tree of components — a root component hosting children that host their own children. Standalone components (now the default in modern Angular) let a component declare its own dependencies directly via the imports array instead of being registered in an NgModule, eliminating the historical NgModule boilerplate. This makes components more self-contained and composable, simplifies lazy loading, and is the recommended way to structure new Angular apps.

Q2: Explain Angular's template binding syntax.

Angular templates connect the view to the component class through a few binding forms. Interpolation, {{ expression }}, renders a value into the DOM as text. Property binding, [property]="expression", sets a DOM or component property (for example [disabled]="isBusy()" or a child's [id]="1"). Event binding, (event)="handler($event)", responds to DOM or component events (like (click)="save()"). Two-way binding, [(ngModel)]="value" (the "banana in a box"), combines a property binding and an event binding for form fields. Together these express the data-flow model: [] pushes data INTO an element/child, () sends events OUT. Modern Angular also adds built-in control-flow blocks — @if, @for, @switch — in templates, replacing the older *ngIf/*ngFor structural directives with clearer, faster syntax; @for requires a track expression to key list items for efficient DOM updates.

Q3: How do parent and child components communicate?

Through a unidirectional data-flow model: data flows DOWN from parent to child via inputs, and events flow UP from child to parent via outputs. A child declares inputs (with the input() function, or the @Input() decorator in older code) that the parent sets using property binding, e.g., <app-item [id]="1">; the child treats these as read-only view data. For communicating back, the child declares outputs (output(), or @Output() with an EventEmitter) and calls emit(); the parent listens with event binding, e.g., (select)="onSelect($event)". This keeps data flow predictable — parents own the state and pass it down, children notify parents of user actions by emitting events, and children don't mutate their inputs or reach into parents. For cross-cutting or sibling communication that doesn't fit the parent/child tree, you use shared services (the DI lesson) or a state library rather than forcing it through inputs/outputs.

🏋️ Practice: scaffold an app (ng new) and create a standalone counter component with a signal; build a parent that renders a list of child components with @for and a track expression; pass data down with an input and bubble a click up with an output; add an @if/@else block driven by a signal; move a heavy template expression into the class and observe the difference; try two-way binding on a text input.

🎯 Key Takeaways

  • A component = a class + template + metadata (@Component); apps are a tree of composed components.
  • Modern Angular uses standalone components (no NgModule) that declare their own imports.
  • Bindings: {{ }} interpolation, [prop] data-in, (event) events-out, [(ngModel)] two-way.
  • Use built-in control flow @if/@for/@switch; @for REQUIRES track for efficient list updates.
  • Parent↔child: data down via inputs, events up via outputs; keep components small and inputs read-only.
