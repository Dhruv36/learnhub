Reactive Forms Intermediate

Angular has two form APIs: template-driven (simple, [(ngModel)]-based) and reactive forms (the form's structure and validation live explicitly in the component class as a tree of objects). Reactive forms are the production standard for anything beyond a trivial form — testable, predictable, and composable in a way template-driven forms aren't.

The Building Blocks

ClassRepresents
FormControlA single input's value + validation state
FormGroupA named collection of controls (an object shape)
FormArrayA dynamic, indexed list of controls (add/remove rows)
FormBuilderA convenience service for building the above with less boilerplate
import { FormBuilder, Validators } from '@angular/forms';

export class SignupComponent {
  private fb = inject(FormBuilder);

  form = this.fb.group({
    email: ['', [Validators.required, Validators.email]],
    password: ['', [Validators.required, Validators.minLength(8)]],
  });

  submit() {
    if (this.form.invalid) { this.form.markAllAsTouched(); return; }   // surface errors, then bail
    console.log(this.form.value);   // { email: '...', password: '...' }
  }
}
<form [formGroup]="form" (ngSubmit)="submit()">
  <input formControlName="email" />
  @if (form.get('email')?.invalid && form.get('email')?.touched) {
    <p class="error">A valid email is required</p>
  }
  <input type="password" formControlName="password" />
  <button type="submit">Sign up</button>
</form>
// The template BINDS to the form model ([formGroup], formControlName) rather than owning state itself —
// the class is the single source of truth for values AND validation.

Why Reactive > Template-driven at Scale

Template-driven ([(ngModel)])Reactive (FormGroup)
Source of truthThe template/DOMThe component class (explicit objects)
TestabilityNeeds a rendered DOM to testTest the form model directly, no DOM
Dynamic formsAwkwardNatural (FormArray, build controls programmatically)
Validation compositionDirective-based, harder to reusePlain functions, easy to compose/reuse
Best forVery simple formsAnything non-trivial (production default)

Custom & Cross-field Validators

function passwordsMatch(group: AbstractControl): ValidationErrors | null {
  const pass = group.get('password')?.value;
  const confirm = group.get('confirm')?.value;
  return pass === confirm ? null : { mismatch: true };   // validator returns an error object or null
}

form = this.fb.group({
  password: [''],
  confirm: [''],
}, { validators: passwordsMatch });    // GROUP-level validator — needed for cross-field rules
// A validator is just a pure function: control/group in, error object (or null if valid) out.
// Async validators (for things like "is this username taken?") return an Observable/Promise instead.

Reactive State via valueChanges

this.form.get('country')?.valueChanges.subscribe(country => {
  // e.g., dynamically enable/disable a "state" field based on country
});
// valueChanges is an Observable — you can pipe it (debounceTime, distinctUntilChanged) just like any
// other RxJS stream, which is how reactive forms interoperate naturally with the RxJS lesson.

Common Mistakes

MistakeSymptomFix
Showing errors before the user has touched a fieldErrors appear immediately on page load — bad UXGate error display on control.touched (or .dirty)
Cross-field validation on individual controlsCan't compare two fields' valuesPut cross-field validators on the FormGroup, not a single control
Not unsubscribing from valueChangesMemory leaks in long-lived componentstakeUntilDestroyed() / async pipe / auto-cleanup patterns
Rebuilding the whole form on every changeLost focus, poor performanceUse patchValue/setValue on the existing form model
Mixing template-driven and reactive forms in one formConfusing, conflicting sources of truthPick one approach per form
Validating only on submit for expensive async checksPoor UX for things like username availabilityAsync validators with debounce on valueChanges

Interview Questions

Q1: Compare reactive forms and template-driven forms in Angular.

They differ in where the form's state and validation logic live. Template-driven forms use [(ngModel)] directives in the template, and the DOM/template is effectively the source of truth — Angular creates the form model implicitly behind the scenes. This is quick for very simple forms but becomes awkward for anything complex: testing requires rendering the DOM, dynamic forms (adding/ removing fields at runtime) are unnatural, and validation logic tends to live scattered across directives. Reactive forms instead build the form model explicitly in the component class as a tree of FormControl/FormGroup/FormArray objects (often via FormBuilder), and the template simply BINDS to that model with [formGroup] and formControlName rather than owning any state itself. This makes the class the single, explicit source of truth for values and validation, which is far easier to unit test (you can assert on the form model directly with no DOM), naturally supports dynamic and cross-field validation, and composes well with RxJS via valueChanges. Reactive forms are the standard choice for any non-trivial production form; template-driven forms remain reasonable for very simple cases.

Q2: How do you implement cross-field validation, like confirming a password match?

A validator is a pure function that receives a control (or group) and returns either null (valid) or an error object describing what's wrong. For a single field's rule (like "required" or "email"), Validators.required/email etc. are attached to that individual FormControl. But a rule that compares TWO fields — like password and confirm-password matching — can't be expressed on either field individually, since a single control's validator only sees that control's own value. The correct approach is to write a custom validator function that takes the parent FormGroup (or FormArray), reads both fields via group.get('password')/.get('confirm'), and returns { mismatch: true } (or whatever error key) if they don't match, or null if they do. This validator is attached at the GROUP level, not on either child control, via the group's second constructor argument ({ validators: passwordsMatch }) when building it with FormBuilder. The resulting error then typically needs to be displayed near one of the fields (commonly the confirm field) by checking form.hasError('mismatch').

Q3: How do you handle async validation, like checking if a username is already taken?

Angular supports async validators alongside synchronous ones — a function with the same shape (a control in, an error result out) except it returns an Observable or Promise of the result instead of a synchronous value, since the check requires a network call. You attach it as the asyncValidators argument when building the control, and Angular marks the control's status as PENDING while the async validator is resolving, then updates to VALID/INVALID once it completes. For a good user experience you typically don't want to fire a network request on every keystroke, so you combine this with the control's valueChanges Observable piped through debounceTime (to wait for typing to pause) and distinctUntilChanged (to skip redundant identical checks) before triggering the actual API call, and you show a "checking availability…" state while pending. This pattern — RxJS operators on valueChanges driving async validation — is a common interview illustration of how reactive forms and RxJS compose naturally, since valueChanges is just another Observable stream.

🏋️ Practice: build a signup form with FormBuilder including required/email/minLength validators; gate error messages on control.touched; add a group-level passwordsMatch cross-field validator; subscribe to valueChanges to conditionally enable/disable a field; add an async validator (simulated) for username availability, debounced; add a FormArray for a dynamic list of phone numbers with add/remove buttons.

🎯 Key Takeaways

  • FormControl (single value), FormGroup (named collection), FormArray (dynamic list) — built with FormBuilder.
  • Reactive forms make the component class the explicit source of truth; the template only binds to it — far more testable than template-driven forms.
  • Cross-field validators go on the FormGroup (not individual controls) since they need to read multiple fields at once.
  • Async validators return an Observable/Promise; combine with valueChanges + debounceTime/distinctUntilChanged for things like username checks.
  • Gate error display on touched/dirty so errors don't appear before the user interacts with a field.
