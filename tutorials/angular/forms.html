Reactive Forms Forms

What you'll master: the code-first forms model — FormControl, FormGroup, FormArray, FormBuilder — strict typing, dynamic structure (the add-a-row pattern), valueChanges pipelines, programmatic control (setValue/patchValue/enable/disable), and DOM-free form testing. Prerequisites: Template-Driven Forms (shared state model) and RxJS Patterns (valueChanges are streams). This is the flavor for forms that are logic, not just fields.

Part 1 — The Inversion: Model First, Template Binds

// The form is built IN CODE — an object you own before any template exists:
import { FormBuilder, ReactiveFormsModule, Validators } from '@angular/forms';

export class SignupComponent {
  private fb = inject(FormBuilder);

  form = this.fb.nonNullable.group({
    name:  ['', [Validators.required, Validators.minLength(2)]],
    email: ['', [Validators.required, Validators.email]],
    plan:  ['free' as 'free' | 'pro'],
  });
}

<!-- The template just POINTS at the model (imports: [ReactiveFormsModule]): -->
<form [formGroup]="form" (ngSubmit)="save()">
  <input formControlName="name">
  <input formControlName="email">
  <select formControlName="plan">…</select>
  <button [disabled]="form.invalid">Sign up</button>
</form>

// Everything is now programmatic:
this.form.value            // { name?: string, email?: string, plan?: 'free'|'pro' }
this.form.getRawValue()    // full value incl. disabled controls — properly typed
this.form.controls.email.errors        // typed access to any control
this.form.valid / dirty / touched      // same state model as template forms

Reactive forms invert the ownership: the class builds a control tree (FormGroup of FormControls), and template directives ([formGroup], formControlName) merely connect DOM elements to it. Everything you could only do through the template before — read structure, attach validators, toggle controls, react to changes — is now ordinary code: inspectable, refactorable, and testable without rendering anything. The state model (touched/dirty/valid, the errors object, .ng-* classes) carries over unchanged from last lesson — you're changing who owns the structure, not the semantics.

Part 2 — Typed Forms & nonNullable

// Angular 14+ forms are STRICTLY TYPED — the compiler knows every control's type:
form.controls.email.setValue('x@y.z');   // ✓
form.controls.email.setValue(42);        // ✗ compile error
form.controls.emial                       // ✗ compile error — typo caught

// The null wrinkle: reset() returns controls to their INITIAL value…
const name = new FormControl('Ada');      // type: FormControl<string | null>
name.reset();                             // → null! (default initial value is null)

// nonNullable: reset returns to the ORIGINAL value instead — kills the null noise:
const name2 = new FormControl('Ada', { nonNullable: true });   // FormControl<string>
name2.reset();                            // → 'Ada'
// fb.nonNullable.group({...}) applies it to every control — the modern default for
// forms whose values feed APIs (no string|null unions leaking into payloads).

// Value types: form.value is Partial<…> (disabled controls drop out) —
// use getRawValue() when you need the complete, fully-typed object.

Typed forms turned a stringly-typed API (form.get('emial') returning any — the pre-14 reality) into compiler-checked contracts: control names, value types, and even the value-vs-getRawValue distinction are all in the type system. The nonNullable choice matters practically: without it every control's type unions null (because reset() defaults to null), which infects every consumer with null-handling for a state that never legitimately occurs in most forms. Rule of thumb: fb.nonNullable.group unless a control genuinely means "no answer yet" as distinct from empty.

Part 3 — Nesting & FormArray: Dynamic Structure

form = this.fb.nonNullable.group({
  recipient: ['', Validators.required],
  address: this.fb.nonNullable.group({              // nested group — payload-shaped
    street: ['', Validators.required],
    city:   ['', Validators.required],
    zip:    ['', [Validators.required, Validators.pattern(/^\d{5}$/)]],
  }),
  phones: this.fb.array([this.makePhone()]),        // FormArray — the dynamic list
});

makePhone() {
  return this.fb.nonNullable.group({
    label:  ['mobile' as 'mobile' | 'work' | 'home'],
    number: ['', [Validators.required, Validators.pattern(/^\+?[\d\s-]{7,}$/)]],
  });
}
get phones() { return this.form.controls.phones; }
addPhone()        { this.phones.push(this.makePhone()); }
removePhone(i: number) { this.phones.removeAt(i); }

<!-- template — arrays bind by INDEX: -->
<div formGroupName="address">
  <input formControlName="street"> <input formControlName="city"> <input formControlName="zip">
</div>
<div formArrayName="phones">
  @for (phone of phones.controls; track phone; let i = $index) {
    <div [formGroupName]="i">
      <select formControlName="label">…</select>
      <input formControlName="number">
      <button type="button" (click)="removePhone(i)">✕</button>
    </div>
  }
</div>
<button type="button" (click)="addPhone()">+ Add phone</button>

FormArray is the payoff feature: structure that grows and shrinks at runtime — phone lists, invoice line items, quiz questions, wizard-collected entries — with validation, state tracking, and value aggregation handled per element. The pattern above is the complete idiom: a factory method (one place defines what a row is), a typed getter, push/removeAt mutations, and index-bound template groups. Two details that bite: buttons inside the form need type="button" (default is submit — clicking "add row" would submit the form), and track phone (the control instance) keeps DOM rows glued to their controls when rows are removed from the middle — the track identity lesson applied to forms.

Part 4 — valueChanges: the Form as a Stream Source

// Every control/group/array exposes valueChanges (and statusChanges) — Observables:

// 1. Live search filter (the typeahead recipe, form-flavored):
results = toSignal(
  this.form.controls.query.valueChanges.pipe(
    debounceTime(300), distinctUntilChanged(),
    switchMap(q => this.api.search(q).pipe(catchError(() => of([])))),
  ), { initialValue: [] });

// 2. Dependent fields — country determines the city list:
constructor() {
  this.form.controls.country.valueChanges.pipe(takeUntilDestroyed()).subscribe(c => {
    this.form.controls.city.reset();                   // stale city cleared
    this.cities.set(CITIES_BY_COUNTRY[c] ?? []);
  });
}

// 3. Autosave — dirty-state draft persistence:
this.form.valueChanges.pipe(
  debounceTime(1000),
  takeUntilDestroyed(),
).subscribe(v => localStorage.setItem('draft', JSON.stringify(v)));

// Programmatic writes — and the events they trigger:
this.form.patchValue({ address: { city: 'Pune' } });   // PARTIAL update — the workhorse
this.form.setValue({ /* EVERY field required */ });    // full replacement — throws if shapes differ
control.setValue(v, { emitEvent: false });             // write WITHOUT triggering valueChanges
                                                       // (breaks autosave loops — see mistakes)

valueChanges is where reactive forms earn their name: the form becomes a stream source composable with every RxJS pattern — debounced search, dependent-field cascades, autosave, analytics. The programmatic write API completes the loop: patchValue for partial updates (loading a record into an edit form), setValue when you want shape-mismatch to fail loudly, and emitEvent: false for the write-without-reacting case — without it, a valueChanges subscriber that writes back into the form recurses into itself (the forms equivalent of the effect-writes-signal loop from Signals).

Part 5 — Programmatic Control: enable/disable, Conditional Fields

// Controls can be switched on and off — and disabled controls LEAVE form.value:
this.form.controls.company.disable();     // validators suspended, dropped from .value
this.form.controls.company.enable();

// The conditional-section idiom — "business account?" reveals company fields:
constructor() {
  const business = this.form.controls.isBusiness;
  const company  = this.form.controls.company;         // a nested group
  business.valueChanges.pipe(startWith(business.value), takeUntilDestroyed())
    .subscribe(is => is ? company.enable() : company.disable());
}
// Why disable() beats @if-ing the fields out of the template:
//   - validators stop running (an @if-hidden required field still blocks submission!)
//   - .value stays payload-correct (no company keys for personal accounts)
//   - state is inspectable in code and tests
// Use BOTH: disable() for the logic + @if (or [hidden]) for the visuals.

// Other levers: control.markAsTouched(), markAsPristine(), updateValueAndValidity(),
// setValidators()/addValidators() at runtime (form-validation.html)

The enable/disable API is the sanctioned way to model conditional form logic, and the note in the code block is a genuine trap: hiding a control's DOM with @if does nothing to the control's validators — a hidden required field keeps the form invalid forever, a bug that presents as "submit button mysteriously disabled". Disabling suspends validation and removes the value from form.value (it remains in getRawValue()), keeping the submitted payload honest. The runtime validator APIs preview the next lesson — validation as manipulable logic rather than static attributes.

Part 6 — Deep Dive: Testing Forms Without a DOM, and Form Architecture

// The reactive payoff nobody markets enough — form LOGIC tests as plain code:
describe('SignupForm', () => {
  it('requires a valid email', () => {
    const c = new SignupComponent();          // no TestBed, no DOM, no fixture
    c.form.controls.email.setValue('nope');
    expect(c.form.controls.email.hasError('email')).toBe(true);
  });
  it('disables company fields for personal accounts', () => {
    const c = new SignupComponent();
    c.form.controls.isBusiness.setValue(false);
    expect(c.form.controls.company.disabled).toBe(true);
    expect(c.form.value).not.toHaveProperty('company');   // payload stays clean
  });
});

// Architecture for big forms:
//  - FACTORY functions/services build recurring groups (makeAddressGroup()) — one
//    definition for shape + validators, reused across forms
//  - WIZARDS: one parent FormGroup, each step binds a sub-group; step validity =
//    group validity; the model outlives step components (state survives navigation)
//  - value → API DTO mapping in ONE place (a toPayload() method) — forms model the UI,
//    payloads model the API; they drift, so translate deliberately

Because the control tree is a plain object graph, form logic — validators, conditional structure, cross-field rules, value mapping — tests at unit speed with zero rendering: instantiate, poke, assert, exactly the plain-class testing ideal. This is the decisive argument for reactive forms on complex logic, and it shapes architecture: put the logic in the form model (where it's testable) rather than scattered across template conditions. The wizard pattern deserves its callout — a parent-owned FormGroup whose sub-groups bind per step is the difference between wizards that survive back-navigation and wizards that lose your data.

Senior framing: review heuristics for reactive forms: forms built in services/factories when shapes recur (DRY applies to control trees); every manual valueChanges.subscribe lifetime-bound (takeUntilDestroyed) and checked for the emitEvent recursion trap; nonNullable unless null is a real state; payload mapping isolated (never this.http.post(url, this.form.value) with a form whose shape drifted from the API's); and the flavor decision re-examined per form — a reactive login form is over-engineering exactly as a template-driven wizard is under-engineering (the honest table). The signals-era note: valueChanges pipelines increasingly convert to signal-consumption via toSignal, and signal-native forms are the roadmap — the control-tree concepts and testing story transfer.

Interview Questions

Q1 (Beginner): Reactive vs template-driven forms — what's the fundamental difference, and what stays the same?

Ownership of structure. Template-driven: directives assemble the control tree from the template (ngModel registers controls); the class holds a data object. Reactive: the class constructs the tree explicitly (FormGroup/FormControl/FormArray) and the template merely binds to it (formControlName). Shared: the entire state model — touched/dirty/valid flags, the errors object, .ng-* CSS classes, submit semantics — so the UX rules (errors on invalid&&touched, etc.) transfer verbatim.

Q2 (Beginner): What are FormControl, FormGroup, and FormArray? Give a use for each.

FormControl: one value + validators + state — a single input (email field). FormGroup: a keyed collection of controls whose value is an object — a form or section (address group with street/city/zip). FormArray: an indexed collection whose value is an array — dynamic lists (phone numbers, invoice line items) with push/removeAt for runtime structure. They nest arbitrarily: a group containing an array of groups is the standard "list of structured rows" shape.

Q3 (Mid): The submit button on a form with conditionally-shown sections stays disabled even when everything visible is valid. Diagnose.

Classic hidden-validator bug: the conditional section was removed from the template (@if) but its controls still exist in the FormGroup with active validators — a hidden required field keeps form.invalid true forever. Fix: drive the condition through the control state — section.disable() when hidden (suspends validators, drops it from form.value), enable() when shown — keeping the template condition purely visual. Diagnose in general by logging form.errors plus walking controls for invalid ones (a findInvalidControls helper is a common team utility).

Q4 (Mid): What do typed forms and nonNullable buy, and when is a nullable control actually right?

Typed forms make control names and value types compile-checked: controls.emial or setValue(42) on a string control are build errors, and form.value has a real type instead of any. nonNullable removes the | null union caused by reset()'s default-to-null behavior — reset returns to the initial value instead, keeping API-bound values clean. A nullable control is right when null is a meaningful domain state distinct from empty: "no rating given yet" vs zero stars, an optional date that can be explicitly cleared — cases where you'd store null in the database too.

Q5 (Senior): Design an invoice editor: header fields, a dynamic line-items table (product, qty, price, computed row total), and a grand total. Sketch the form model and where computations live.

Model: fb.nonNullable.group({ customer, issueDate, items: fb.array([]) }) with a makeItem() factory returning group({ productId, qty: [1, [required, min(1)]], price }). Row/grand totals are derived state — not form controls: storing them in controls invites drift (the derive-don't-sync rule). Compute via toSignal(form.valueChanges) feeding computed()s — grandTotal = computed(() => rows().reduce(...)) — or valueChanges + map. Add/remove via the array API with track-keyed rows; validators on qty/price per row; a form-level validator if business rules span rows (e.g. max invoice total — next lesson). Payload via explicit toPayload() translating UI shape to API DTO. Tests: instantiate the component class, push rows, assert totals and validity — no DOM.

Q6 (Senior): A five-step checkout wizard must preserve entered data across back/forward navigation, validate per step, and submit everything at the end. Architect it with reactive forms.

One parent FormGroup owned above the step components — in a wizard-scoped service (component-provided at the wizard shell, per-instance scoping) so it survives step component destroy/recreate. Each step is a sub-group (shipping, payment, …); the step component injects the service and binds [formGroup]="wizard.form.controls.shipping". Per-step gating: next-button disabled on that sub-group's invalidity, and a CanDeactivate-style guard or step-router check for direct navigation. Cross-step rules (shipping country restricts payment methods) live as parent-level logic — valueChanges cascades with enable/disable. Final submit: getRawValue() → toPayload() → API, with mark-all-touched on failure jumping to the first invalid step. The architecture's crux: the form model's lifetime is the wizard's, not any step's — which is precisely what class-owned control trees make natural and template-assembled forms make painful.

2 (Easy): Load a user record into an edit form with patchValue, and explain why setValue would be wrong here.
Solution
ngOnInit() {
  this.api.getUser(this.id()).subscribe(user => {
    this.form.patchValue({
      name: user.name,
      email: user.email,
      // API also returned createdAt, roles, … — patchValue IGNORES keys the form
      // doesn't have, and fills only what you pass.
    });
  });
}
// setValue would throw twice over: it requires EVERY form key to be present in the
// argument (partial → runtime error), and rejects extra keys. patchValue is the
// load-a-record workhorse; setValue is for deliberate full replacement where a
// shape mismatch SHOULD fail loudly.

4 (Medium): Implement the country→city dependent fields with one twist: preserve the city if it's still valid for the new country.
Solution
constructor() {
  const { country, city } = this.form.controls;
  country.valueChanges.pipe(startWith(country.value), takeUntilDestroyed())
    .subscribe(c => {
      const options = CITIES_BY_COUNTRY[c] ?? [];
      this.cityOptions.set(options);
      if (!options.includes(city.value)) {
        city.reset();                        // only clear when the choice became invalid
      }                                      // (reset with nonNullable → initial '', not null)
    });
}
// startWith seeds the initial population; the includes() check is the twist — blind
// reset() on every change throws away valid user input (a real UX complaint).
5 (Hard): Add autosave to a settings form: persist drafts to localStorage 1s after edits stop, restore on load, and — critically — don't let the restore trigger an immediate re-save. Then clear the draft on successful submit.
Solution
private static DRAFT_KEY = 'settings-draft';

constructor() {
  // RESTORE first — without emitting, so the save pipeline doesn't fire on load:
  const draft = localStorage.getItem(SettingsComponent.DRAFT_KEY);
  if (draft) this.form.patchValue(JSON.parse(draft), { emitEvent: false });

  // SAVE pipeline — debounced, lifetime-bound:
  this.form.valueChanges.pipe(
    debounceTime(1000),
    takeUntilDestroyed(),
  ).subscribe(v => localStorage.setItem(SettingsComponent.DRAFT_KEY, JSON.stringify(v)));
}

save() {
  if (this.form.invalid) return;
  this.api.saveSettings(this.form.getRawValue()).subscribe(() => {
    localStorage.removeItem(SettingsComponent.DRAFT_KEY);   // draft superseded
    this.form.markAsPristine();                             // unsaved-changes guards relax
  });
}
// emitEvent: false on restore is the exam point — without it the restored value
// round-trips straight back into storage (harmless here, an infinite loop in
// two-way sync scenarios). markAsPristine pairs with the dirty-guard pattern.
6 (Hard): Write three DOM-free unit tests for the business-account form from Part 5: personal accounts exclude company data from the payload; business accounts require a company name; toggling back to personal re-validates instantly.
Solution
describe('AccountFormComponent', () => {
  const make = () => new AccountFormComponent();   // plain instantiation — no TestBed

  it('excludes company data for personal accounts', () => {
    const c = make();
    c.form.controls.isBusiness.setValue(false);
    expect(c.form.controls.company.disabled).toBe(true);
    expect(c.form.value).not.toHaveProperty('company');    // disabled → out of .value
    expect(c.form.valid).toBe(true);                       // company validators suspended
  });

  it('requires company name for business accounts', () => {
    const c = make();
    c.form.controls.isBusiness.setValue(true);
    expect(c.form.controls.company.enabled).toBe(true);
    expect(c.form.valid).toBe(false);                      // company.name required + empty
    c.form.controls.company.controls.name.setValue('Acme GmbH');
    expect(c.form.valid).toBe(true);
  });

  it('re-validates instantly when switching back to personal', () => {
    const c = make();
    c.form.controls.isBusiness.setValue(true);             // invalid (empty company)
    expect(c.form.valid).toBe(false);
    c.form.controls.isBusiness.setValue(false);            // company disabled again
    expect(c.form.valid).toBe(true);                       // no stale invalidity
  });
});
// The whole conditional-validation feature — logic, payload shape, state transitions —
// verified in milliseconds without rendering a single element. This is the reactive
// forms testing dividend.

🎯 Key Takeaways

  • Reactive = class-owned control tree (FormGroup/FormControl/FormArray via FormBuilder); templates bind with formControlName. Same state model as template forms, different owner.
  • Typed forms + nonNullable: compile-checked names and values, reset-to-initial instead of null; form.value is partial (drops disabled), getRawValue() is complete.
  • FormArray + a row-factory + track-keyed template = dynamic structure done right; type="button" on row buttons.
  • valueChanges makes forms stream sources — debounced search, dependent fields, autosave — with emitEvent: false guarding write-back loops and takeUntilDestroyed guarding leaks.
  • Conditional logic = enable()/disable() (validators + payload), not template hiding — an @if-hidden required control blocks submission invisibly.
  • The reactive dividend: form logic unit-tests as plain code — and architecture follows (factory groups, wizard-owning services, explicit toPayload() mapping).
