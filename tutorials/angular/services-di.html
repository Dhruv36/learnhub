Services & Dependency Injection Services & DI

What you'll master: why DI is Angular's structural backbone — not boilerplate — services and the single-responsibility split with components, inject() vs constructor injection, providedIn: 'root' and tree-shakable singletons, injector hierarchy basics, and the testability payoff that justifies the whole system. Prerequisites: Component Communication (you've already seen the shared-service pattern). If Spring is your backend, this is the same idea in the browser — a comparison this lesson uses deliberately.

Part 1 — The Problem DI Solves

// WITHOUT DI — every consumer constructs its own dependencies:
class OrderListComponent {
  private api = new OrderApi(new HttpCli(), new AuthTokenStore());   // knows the whole graph
}
// Consequences at scale:
//   - OrderApi's constructor changes → EVERY consumer edits
//   - two components construct two OrderApis → duplicated caches/state
//   - testing OrderListComponent → you get the REAL OrderApi (and real HTTP) — unmockable
//   - cross-cutting swaps (mock server, logging wrapper) → touch every `new`

// WITH DI — consumers DECLARE needs; the injector supplies them:
class OrderListComponent {
  private api = inject(OrderApi);        // "I need an OrderApi" — from where? Not my problem.
}

Dependency injection inverts one decision: who constructs the object graph. Instead of each class new-ing what it needs (hard-coding the concrete class, its constructor signature, and its lifetime), classes declare dependencies and a container — Angular's injector — resolves, constructs (once), caches, and supplies them. Every benefit of the system falls out of that inversion: shared singletons for free, swap-ability (test doubles, platform variants), constructor-signature changes localized to one place, and dependency graphs you can read in a class's first lines. This is the exact architecture Spring's IoC container applies server-side; Angular ships it in the browser.

Part 2 — Services: Where Non-View Logic Lives

// A service is a plain class with @Injectable — state + logic, no view:
import { Injectable, signal, computed, inject } from '@angular/core';

@Injectable({ providedIn: 'root' })          // ONE instance, app-wide, tree-shakable
export class CartService {
  private http = inject(HttpClient);         // services inject other services

  private _items = signal<CartItem[]>([]);           // private writable state
  readonly items = this._items.asReadonly();          // public read-only view
  readonly total = computed(() => this._items().reduce((s, i) => s + i.price * i.qty, 0));
  readonly count = computed(() => this._items().length);

  add(item: CartItem) { this._items.update(l => [...l, item]); }
  remove(id: number)  { this._items.update(l => l.filter(i => i.id !== id)); }
  checkout()          { return this.http.post('/api/orders', { items: this._items() }); }
}

// The division of labor:
// COMPONENT — view concerns: bind state, handle events, delegate    (thin)
// SERVICE   — everything else: data access, business rules, shared state, caching  (fat-ish, tested)

Services are where the code that isn't about rendering lives. The signal-service shape shown here — private writable signals, public readonly/computed projections, mutation methods enforcing the rules — is the modern standard you saw emerging in Component Communication and will formalize in State Patterns: consumers can read everything and corrupt nothing. The component/service split is also a testability split: components need TestBed and DOM; services are plain classes testable with new CartService()-level simplicity (Part 6). When a component grows a method that doesn't touch the view, that method is a service escapee.

Part 3 — inject() vs Constructor Injection

// MODERN — inject() as field initializers:
export class CheckoutComponent {
  private cart = inject(CartService);
  private router = inject(Router);
  private analytics = inject(AnalyticsService, { optional: true });   // null if absent
}

// CLASSIC — constructor parameters (all older code; still fully supported):
export class CheckoutComponent {
  constructor(
    private cart: CartService,
    private router: Router,
  ) {}
}

// Why inject() won:
//  - works in FUNCTIONS too: guards, interceptors, factories (guards-lazy.html, http.html)
//  - no constructor ceremony; plays well with inheritance (no super(...deps) chains)
//  - field initializers can USE the dependency immediately:
//    user = toSignal(inject(UserApi).currentUser$);
// Rule: inject() only in an INJECTION CONTEXT — field initializers, constructor body,
// provider factories. Calling it later (event handler, ngOnInit) throws
// "inject() must be called from an injection context". Capture at construction time.

Both forms do the same thing — ask the current injector for an instance — but inject() is the direction of travel because the modern APIs are functional: a route guard is now a plain function, and functions have no constructors. The injection-context rule is the one gotcha: inject() works only while Angular is constructing things; you capture dependencies during construction and use them later (or wrap late needs in runInInjectionContext — rare). Read both styles fluently; write inject() in new code.

Part 4 — providedIn: 'root' & Service Lifetimes

@Injectable({ providedIn: 'root' })     // THE default choice:
// - ONE instance for the whole app (singleton), created LAZILY on first injection
// - TREE-SHAKABLE: if nothing injects it, the class is dropped from the bundle
//   (this is why root-provided beats listing in app.config providers)

// Other lifetimes exist by providing ELSEWHERE (full detail: di-advanced.html):
@Component({ selector: 'app-wizard', providers: [WizardStateService] })
// → a NEW instance PER COMPONENT INSTANCE, destroyed with it —
//   two wizards on screen = two isolated states. The injector is hierarchical:
//   children see their own providers first, then walk up to root.

// Lifetime rules of thumb:
//   app-wide state/API access            → providedIn: 'root'      (99% of services)
//   per-widget-instance state            → component providers
//   per-lazy-feature singleton           → route-level providers (guards-lazy.html)

providedIn: 'root' gives the semantics you almost always want — one lazily-created, tree-shakable instance — and its "shared by everyone" nature is exactly what makes the cart-badge-updates-from-anywhere pattern work: every injector in the app resolves CartService to the same object holding the same signals. The component-provider alternative exists precisely for when sharing is wrong: reusable stateful widgets (wizard, complex filter panel) where each instance needs its own isolated state service. The hierarchy — component injectors → route injectors → root — is the mechanism, explored properly in Advanced DI.

Part 5 — The Payoff: Testing With Doubles

// The service under test — depends on HttpClient? No: on ANOTHER service we can double.
describe('CartService', () => {
  it('totals items and enforces the add rules', () => {
    const cart = new CartService();               // plain class, plain test — no Angular at all
    cart.add({ id: 1, price: 10, qty: 2 });
    expect(cart.total()).toBe(20);
  });
});

// A component test swaps the REAL service for a double via the injector:
TestBed.configureTestingModule({
  providers: [
    { provide: CartService, useValue: {
        items: signal([]).asReadonly(),
        total: signal(99).asReadonly(),
        add: vi.fn(),
      } },
  ],
});
const fixture = TestBed.createComponent(CheckoutComponent);
// CheckoutComponent injects "CartService" and receives OUR object — it can't tell.
// No HTTP, no shared state, tests isolated and fast. (Full treatment: testing.html)

This is DI's payoff made concrete: because consumers depend on tokens resolved by the injector rather than on constructed concretes, tests substitute doubles at the provider level and the component under test is none the wiser — the same "mock at boundaries" discipline from the JS testing lesson, with the framework supplying the seam. Notice also what the signal-service shape does for testability: business rules live in plain methods on a plain class, so most logic is testable without TestBed at all — the injector matters only where wiring does.

Part 6 — Deep Dive: DI as Architecture

Facade pattern over a feature — components see ONE service:
  ProductsFacade
    ├─ inject(ProductsApi)        HTTP calls (http.html)
    ├─ inject(ProductsCache)      staleness rules
    └─ exposes: products, loading (signals) + load(), search()   ← components use ONLY this

Abstraction at boundaries — depend on a contract, choose the concrete at wiring time:
  export abstract class StorageService {
    abstract get(k: string): string | null;
    abstract set(k: string, v: string): void;
  }
  { provide: StorageService, useClass: isBrowser ? LocalStorage : MemoryStorage }
  // consumers inject StorageService, never know which — SSR-safe by wiring
  // (useClass/useFactory/useValue/InjectionToken mechanics: di-advanced.html)

The dependency DIRECTION rule (mirrors backend clean architecture):
  components → facades/services → api/infrastructure
  never sideways between features; never infrastructure → components.
  core/ = root singletons; features own their facades (index.html's folder layout, motivated)

At architecture scale, DI is what makes layering enforceable rather than aspirational: a feature's components inject its facade; the facade composes API, cache, and state services; swap the infrastructure (mock server, new backend, platform storage) by changing providers — zero component edits. The abstract-class-as-contract pattern is Angular's interface-injection idiom (TypeScript interfaces vanish at runtime, so abstract classes serve as injectable tokens — di-advanced covers InjectionToken for the rest). If you know Spring's @Service/@Autowired and profile-based bean selection, you have the complete mental model already — Angular is the same inversion at the other end of the wire.

Senior framing: DI code smells reviewers flag: components injecting many services (orchestration creeping into the view layer — introduce a facade); services injected but used for one property (pass the value as an input instead — DI is for behavior/state, inputs are for data); new-ing service classes anywhere outside tests (bypasses the graph — that instance shares nothing, gets no doubles); business logic in components "because it was quicker" (untestable without DOM); and circular service dependencies (A injects B injects A — the injector throws; the fix is extracting the shared piece both need into a third service, which is also the design fix). The interview one-liner: components are how the app looks; services are what the app is; DI is what lets you test the latter without the former.

Interview Questions

Q1 (Beginner): What is a service, and how does the component/service split improve an Angular app?

A service is an @Injectable class holding non-view logic and shared state — data access, business rules, caching. The split keeps components thin (bind state, handle events, delegate), which buys: reuse (many components, one service), single sources of truth (one cart, not per-component copies), and testability — service logic is plain-class-testable without DOM or TestBed, while components stay simple enough that their tests are about rendering and interaction.

Q2 (Beginner): What does providedIn: 'root' do, and why is it preferred over registering the service in app.config.ts providers?

It registers the class with the root injector as an app-wide singleton, created lazily on first injection. Preferred because it's tree-shakable: the service is only bundled if something actually injects it, whereas listing it in the app config's providers array references it unconditionally, forcing it into the bundle used or not. Same instance semantics, better dead-code elimination, and the registration lives with the class it describes.

Q3 (Mid): Compare inject() and constructor injection. Why did Angular add inject(), and what's its one hard constraint?

Both resolve from the current injector. inject() was added because Angular's modern APIs are functional — route guards, interceptors, and factory functions have no constructors to inject into — and because field-initializer style composes well (user = toSignal(inject(UserApi).stream$)) and avoids constructor/super ceremony in inheritance. Constraint: it must run in an injection context — field initializers, constructor bodies, provider/guard factories. Called later (event handler, lifecycle hook body) it throws; the pattern is capture-at-construction, use-later.

Q4 (Mid): A "shared" filter service works in one page but two widget instances on another page overwrite each other's filters — or, in a variant bug, each widget mysteriously has empty state. What's going on?

Both are lifetime/provider-scope bugs. Overwriting: the service is root-provided — one shared instance — while the widgets each expected their own; per-instance state needs the service in the component's providers array, giving each widget its own injector-scoped copy destroyed with it. Empty-state variant: someone put the service in component providers accidentally, so each widget got a fresh private instance instead of the shared one — remove the component provider to fall back up the hierarchy to root's singleton. Diagnosis question in both directions: "how many instances does this state want?" — then place the provider at exactly that level.

Q5 (Senior): How do you inject "an interface" in Angular, given TypeScript interfaces don't exist at runtime?

DI resolves by runtime token, and interfaces erase — so you either (a) use an abstract class as the contract (it exists at runtime, can be a token, and concretes implement/extend it): { provide: StorageService, useClass: LocalStorageService }, with consumers injecting the abstract type; or (b) create an InjectionToken<T> for values/configs/multi-implementations, providing with useValue/useFactory/useClass. Abstract-class contracts read most naturally for service-shaped dependencies (and enable per-platform/per-environment wiring like SSR-safe storage); tokens handle primitives and config objects that have no class at all — Advanced DI details the mechanics.

Q6 (Senior): Design the service layer for a products feature that fetches, caches, filters, and exposes loading state — and show where the seams for testing and backend swaps are.

Three layers behind one facade. ProductsApi (root): pure HTTP — methods returning observables, no state; this is the mock seam for tests and the swap seam for a backend change. ProductsStore (root or feature-route-provided): the signal-service — private writable signals for products/loading/filters, public computed projections (filtered lists, counts), mutation methods with the business rules. ProductsFacade (optional at small scale — the store can serve directly): composes api+store, exposing exactly what components need: products, loading, load(), setFilter(). Components inject only the facade — so component tests double one flat object; store tests are plain class tests; api tests use HttpTestingController. Backend swap = new api class + one provider line. This shape scales down (merge layers) and up (the same seams are where NgRx slots in if complexity demands it).

🏋️ Exercises

1 (Easy): Create a root-provided NotificationService with a read-only messages signal list, notify(msg), and dismiss(index) — and a component that shows the count.
Solution
@Injectable({ providedIn: 'root' })
export class NotificationService {
  private _messages = signal<string[]>([]);
  readonly messages = this._messages.asReadonly();
  notify(msg: string)   { this._messages.update(l => [...l, msg]); }
  dismiss(index: number){ this._messages.update(l => l.filter((_, i) => i !== index)); }
}

@Component({ selector: 'app-bell',
  template: `🔔 @if (notes.messages().length) { <b>{{ notes.messages().length }}</b> }` })
export class BellComponent {
  notes = inject(NotificationService);
}
// Any component calling notify() updates every reader — one instance, root-provided.
2 (Easy): This code throws "inject() must be called from an injection context". Fix it both ways (capture early / constructor style). 
export class ExportButtonComponent {
  download() { inject(ReportApi).fetch().subscribe(save); }
}
Solution
// Fix A — capture at construction (field initializer IS an injection context):
export class ExportButtonComponent {
  private reportApi = inject(ReportApi);
  download() { this.reportApi.fetch().subscribe(save); }
}
// Fix B — classic constructor injection:
export class ExportButtonComponent {
  constructor(private reportApi: ReportApi) {}
  download() { this.reportApi.fetch().subscribe(save); }
}
// inject() runs only while the injector is constructing the class — event handlers run
// long after. Capture dependencies during construction; use them whenever.
3 (Medium): A ProductFilterComponent is used twice on a compare page, and the instances share (and fight over) filter state stored in a root service. Fix with the correct provider scope, and note what changed about lifetime.
Solution
@Injectable()                                   // ← NO providedIn — provided by components
export class FilterStateService {
  private _query = signal('');
  readonly query = this._query.asReadonly();
  setQuery(q: string) { this._query.set(q); }
}

@Component({
  selector: 'app-product-filter',
  providers: [FilterStateService],              // ← NEW instance PER component instance
  template: `<input [value]="filters.query()" (input)="filters.setQuery($any($event.target).value)">`,
})
export class ProductFilterComponent {
  filters = inject(FilterStateService);         // resolves to MY OWN instance
}
// Each <app-product-filter /> now has isolated state, created with the component and
// destroyed with it (lifetime = component lifetime, not app lifetime). The hierarchy
// resolves to the nearest provider — the component's own — before ever reaching root.
4 (Medium): Define an abstract Logger contract with ConsoleLogger (dev) and SilentLogger (prod) implementations, wired by environment — consumers must not know which they get.
Solution
export abstract class Logger {
  abstract log(msg: string): void;
  abstract error(msg: string, err?: unknown): void;
}

@Injectable() export class ConsoleLogger extends Logger {
  log(msg: string) { console.log('[app]', msg); }
  error(msg: string, err?: unknown) { console.error('[app]', msg, err); }
}
@Injectable() export class SilentLogger extends Logger {
  log() {} error() {}                       // or: ship to a telemetry endpoint
}

// app.config.ts — the ONLY place that knows the concrete:
providers: [
  { provide: Logger, useClass: environment.production ? SilentLogger : ConsoleLogger },
]

// Consumers depend on the contract:
private log = inject(Logger);               // abstract class as runtime token
// Tests: { provide: Logger, useValue: { log: vi.fn(), error: vi.fn() } } — same seam.
5 (Hard): Refactor this component so its logic is testable without TestBed, then write the two plain unit tests that become possible. 
export class DiscountBannerComponent {
  code = ''; percent = 0; valid = false;
  private http = inject(HttpClient);
  apply() {
    if (!/^[A-Z]{4}\d{2}$/.test(this.code)) { this.valid = false; return; }
    this.http.post<{pct:number}>('/api/discount', { code: this.code })
      .subscribe(r => { this.percent = r.pct; this.valid = r.pct > 0; });
  }
}
Solution
// Extract: validation (pure) and application (service) — the component just delegates.
export const isValidCodeFormat = (code: string) => /^[A-Z]{4}\d{2}$/.test(code);

@Injectable({ providedIn: 'root' })
export class DiscountService {
  private http = inject(HttpClient);
  apply(code: string) {
    return this.http.post<{ pct: number }>('/api/discount', { code });
  }
}

@Component({ /* … */ })
export class DiscountBannerComponent {
  private discounts = inject(DiscountService);
  code = signal(''); percent = signal(0); valid = signal(false);
  apply() {
    if (!isValidCodeFormat(this.code())) { this.valid.set(false); return; }
    this.discounts.apply(this.code()).subscribe(r => {
      this.percent.set(r.pct); this.valid.set(r.pct > 0);
    });
  }
}

// Plain tests, no TestBed:
it('rejects malformed codes', () => {
  expect(isValidCodeFormat('SAVE20')).toBe(true);
  expect(isValidCodeFormat('save20')).toBe(false);
  expect(isValidCodeFormat('SAVE2')).toBe(false);
});
it('marks valid only for positive percentages', () => {
  // testing the component's decision via a hand-made double — DI seam, no HTTP:
  const comp = Object.create(DiscountBannerComponent.prototype);

     or better — move the r.pct > 0 decision into DiscountService and test THAT as a
     plain class with a mocked http. The refactor's point: each rule now has a home
     testable at its own layer. */
});
6 (Hard): Two services end up needing each other: AuthService wants to clear CartService on logout; CartService wants the auth token for checkout. Resolve the circular dependency with a clean design.
Solution
// Circular: Auth → Cart (clear on logout) and Cart → Auth (token) — injector error,
// and a symptom: both services know too much about each other.

// Design fix: invert one direction with a shared, dependency-free signal source.
@Injectable({ providedIn: 'root' })
export class SessionStore {                      // depends on NOBODY
  private _token = signal<string | null>(null);
  readonly token = this._token.asReadonly();
  readonly loggedIn = computed(() => this._token() !== null);
  setToken(t: string | null) { this._token.set(t); }
}

@Injectable({ providedIn: 'root' })
export class AuthService {
  private session = inject(SessionStore);        // Auth → Session
  logout() { this.session.setToken(null); }
}

@Injectable({ providedIn: 'root' })
export class CartService {
  private session = inject(SessionStore);        // Cart → Session (NOT → Auth)
  constructor() {
    effect(() => {                               // reacts to logout instead of being called
      if (!this.session.loggedIn()) this.clear();
    });
  }
  checkout() { const token = this.session.token(); /* … */ }
  clear() { /* … */ }
}
// Both depend downward on SessionStore; neither knows the other exists. The cycle is
// gone AND the design improved: logout is an observable fact, not a call-everyone chore.

🎯 Key Takeaways

  • DI inverts object construction: classes declare dependencies, the injector builds/caches/supplies — enabling sharing, swapping, and localized change.
  • Services own non-view logic in the signal-service shape: private writable signals, public readonly/computed, mutation methods. Components bind and delegate.
  • inject() is the modern form (works in functional guards/interceptors, field initializers) with one rule: only in injection contexts — capture at construction.
  • providedIn: 'root' = lazy, tree-shakable app singleton (the default); component providers = deliberate per-instance state; the injector hierarchy resolves nearest-first.
  • The payoff is seams: provider-level doubles in tests, abstract-class contracts for environment/platform swaps, facades so components depend on one surface.
  • Smells: many-service components (want a facade), new-ed services, public writable signals, circular injection (extract the shared piece both need).
