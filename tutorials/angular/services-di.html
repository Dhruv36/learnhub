Services & Dependency Injection Intermediate

Components should focus on presenting UI, not fetching data or holding business logic. A service is a plain class that holds that shared logic/state, and Angular's Dependency Injection (DI) system hands components the service instances they need automatically. Understanding DI — especially provider scope — is one of the most interview-tested Angular topics because it explains a huge share of "why is my state wrong" bugs.

A Service Is Just a Class

import { Injectable, signal } from '@angular/core';

@Injectable({ providedIn: 'root' })   // registers it as a SINGLETON for the whole app
export class CartService {
  private items = signal<Item[]>([]);
  readonly items$ = this.items.asReadonly();   // expose read-only to consumers

  add(item: Item) { this.items.update(list => [...list, item]); }
}
// @Injectable marks the class as available for injection. providedIn: 'root' is the default and
// common choice — one shared instance, tree-shakeable if never actually injected anywhere.

Injecting a Service into a Component

import { Component, inject } from '@angular/core';

@Component({ selector: 'app-cart-badge', standalone: true, template: `{{ cart.items$().length }}` })
export class CartBadgeComponent {
  cart = inject(CartService);          // modern function-based injection
  // Older constructor-injection style (still valid, still common):
  // constructor(private cart: CartService) {}
}
// Angular's INJECTOR resolves CartService and hands back the SAME instance every component asked for
// (because providedIn: 'root' makes it a singleton) — no `new CartService()` anywhere in your code.
💉 DI inverts control: components declare what they need, Angular supplies it. This decouples components from HOW a dependency is constructed (which HTTP client, which config), makes testing trivial (inject a fake/mock service instead), and centralizes shared state and logic outside the component tree. The inject() function (usable in field initializers, constructors, and factory functions) is the modern idiom, replacing constructor-parameter injection in many codebases.

Provider Scope — the Single Most Important DI Concept

RegistrationInstance scopeUse for
providedIn: 'root'ONE instance for the whole app (singleton)Shared app state, API clients — the default choice
providers: [X] on a componentA NEW instance per that component subtreePer-feature/per-instance state (e.g., a wizard's form state)
Route-level providers:One instance per active routeState scoped to a lazily-loaded feature
@Component({ selector: 'app-wizard', providers: [WizardStateService], ... })
export class WizardComponent { }
// Every WizardComponent instance gets its OWN WizardStateService — useful when you open multiple
// wizards, or need the service to reset when the component is destroyed and recreated. This is DIFFERENT
// from providedIn: 'root', where the ENTIRE APP shares one instance forever.
⚠️ Registering a service at the wrong level is the #1 DI bug. Expecting shared state but declaring the service in a component's own `providers:` array silently gives every instance of that component its own isolated copy — components think they're sharing data but aren't. Conversely, a service that should reset per-feature but is `providedIn: 'root'` leaks state across navigations. Always ask: "should this be ONE instance for the app, or a fresh one per X?"

Injection Tokens (for non-class dependencies)

import { InjectionToken } from '@angular/core';
export const API_URL = new InjectionToken<string>('API_URL');

// Registered in app config:
providers: [{ provide: API_URL, useValue: 'https://api.example.com' }]

// Consumed anywhere:
const apiUrl = inject(API_URL);
// InjectionTokens let you inject plain values/config (not just classes) through the same DI system —
// useful for environment config, feature flags, or swapping implementations (useClass/useFactory).

Interview Questions

Q1: What is dependency injection and why does Angular use it?

Dependency Injection is a design pattern where a class declares what it depends on (its "dependencies") rather than constructing them itself, and an external system — Angular's injector — supplies (injects) the actual instances at runtime. In Angular, you mark a class @Injectable, and components or other services request it via constructor parameters or the inject() function; the injector resolves and provides the instance, reusing an existing one if the registration scope calls for a singleton. This inverts control: components don't know or care HOW a service is constructed (which config, which underlying HTTP client), only that they need "a CartService." The benefits are substantial: it decouples classes from their dependencies' construction details, makes shared state and logic easy to centralize outside the component tree, and — critically for testing — lets you substitute a real service with a fake/mock in tests simply by providing a different implementation for the same token, with no changes to the consuming component's code.

Q2: Explain provider scope — providedIn: 'root' vs component-level providers.

Provider scope determines how many instances of a service exist and who shares them, and it's controlled by WHERE the service is registered in Angular's hierarchical injector tree. providedIn: 'root' (set on the @Injectable decorator, the common default) registers the service at the application root, so the ENTIRE app shares exactly one instance — appropriate for global state like a shopping cart, auth session, or a shared API client, and it's tree-shakeable (removed from the bundle if never injected). Registering the same service in a component's own providers: array instead creates a new injector scope rooted at that component, so EVERY instance of that component (and its children) gets its OWN separate instance of the service — appropriate when you want isolated state per component instance, such as a wizard's in-progress form state that should reset when the wizard is destroyed and recreated, or when multiple copies of a widget need independent state. The classic bug is registering at the wrong level: expecting app-wide shared state but scoping the service to a component (each instance silently gets its own isolated copy, so "sharing" appears broken), or the reverse, where state that should reset per feature instead persists app-wide and leaks across navigations.

Q3: What's an InjectionToken and when do you need one?

Angular's DI system is normally keyed by class type — you inject CartService by referring to the CartService class itself, which the injector uses as a lookup key. But not every dependency is a class: you might want to inject a plain string (an API base URL), a primitive config value, or an interface that has no runtime representation to key on. An InjectionToken is a unique, typed token you create explicitly (new InjectionToken<string>('API_URL')) to serve as that lookup key for non-class values. You register a value for it with a provider entry like { provide: API_URL, useValue: 'https://api.example.com' } (or useFactory/useClass for more dynamic providers), and any component or service can then inject(API_URL) to receive it — going through the same DI system as classes. This is the standard way to inject environment configuration, feature flags, or to let different environments/tests swap in different implementations of an abstraction without changing consumer code.

🏋️ Practice: create a providedIn:'root' CartService with a signal and inject it into two sibling components, confirming both see the same shared state; then move it to a component's own providers: array and observe each instance getting isolated state; write a service that fetches data and inject it into a component (no HTTP calls in the component); create an InjectionToken for a config value and provide/inject it; swap a real service for a fake in a test via a different provider.

🎯 Key Takeaways

  • A service is a plain @Injectable class holding shared logic/state; Angular's DI supplies instances so you never `new` a service manually.
  • inject() (modern) or constructor injection (classic) both work; DI decouples consumers from construction and makes testing/mocking easy.
  • Provider scope is the critical concept: providedIn: 'root' = one app-wide singleton; component-level providers: = a fresh instance per component subtree.
  • Registering a service at the wrong scope is the #1 DI bug — always ask "one instance for the app, or fresh per X?"
  • InjectionTokens let you inject non-class values (config, flags) through the same DI system via provider entries.
