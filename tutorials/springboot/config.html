Configuration & Profiles Web Layer

What you'll master in this lesson: how Spring Boot externalizes configuration so the same jar runs in every environment; the property-source precedence order (why an env var overrides the file); profiles for per-environment config; and — critically — how to keep secrets out of source control. This complements the Configuration & @Bean lesson (which covered binding into beans); here the focus is the property sources themselves and the operational discipline of environment-specific, secret-safe configuration. Getting this right is the difference between "works on my machine" and a clean, secure deploy across dev, staging, and production.

Part 1 — application.yml: The Config File

# src/main/resources/application.yml
server:
  port: 8080
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/shop
    username: shop
    # password comes from an ENV VAR, never hardcoded — see Part 5
  jpa:
    hibernate:
      ddl-auto: validate          # never `create-drop` in prod (Data lesson)
app:
  payment:
    currency: USD
    retry-attempts: 3
💡 YAML or .properties both work; YAML nests better and is the common choice. Anything Spring or a starter reads is just a property key — server.port, spring.datasource.url, spring.jpa.hibernate.ddl-auto — and you bind your own settings into typed @ConfigurationProperties objects (the Configuration & @Bean lesson). This file is the base layer of configuration; the rest of this lesson is about how other sources override it and how to keep it secret-free.

Part 2 — Property-Source Precedence (later wins)

# Spring reads configuration from MANY sources; LATER sources override earlier ones. Roughly:
#   application.yml  <  application-{profile}.yml  <  OS env vars  <  JVM system props  <  command-line args
# So the SAME jar picks up overrides per environment WITHOUT a rebuild:

DB_PASSWORD=s3cret java -jar app.jar                       # env var overrides the file
java -jar app.jar --server.port=9000                       # command-line arg wins over everything
java -Dserver.port=9000 -jar app.jar                       # JVM system property

# Reference an env var inside yml, with a default:
#   password: ${DB_PASSWORD}
#   port: ${PORT:8080}         # 8080 if PORT is unset
🎯 The precedence order is what makes one immutable artifact reconfigurable per environment. You build the jar once; the platform injects environment-specific values via environment variables or command-line arguments, which override the packaged defaults — no rebuild, no per-environment binaries. Env vars also map to property keys by relaxed binding: SPRING_DATASOURCE_URL (uppercase, underscores) binds to spring.datasource.url. This "build once, configure per environment" model is fundamental to reliable deployment and to twelve-factor apps.

Part 3 — Profiles: Per-Environment Configuration

# application.yml holds SHARED defaults. Profile-specific files override for each environment:
#   application-dev.yml    application-prod.yml    application-test.yml

# application-dev.yml
spring:
  jpa: { hibernate: { ddl-auto: create-drop } }   # convenient locally
  h2: { console: { enabled: true } }

# application-prod.yml
spring:
  jpa: { hibernate: { ddl-auto: validate } }      # safe in prod
  datasource: { url: "${DB_URL}" }

# Activate a profile (via env var or arg — NOT hardcoded in the jar):
java -jar app.jar --spring.profiles.active=prod
# or  SPRING_PROFILES_ACTIVE=prod  as an env var
// Beans can be profile-scoped too:
@Bean @Profile("dev")  DataSource h2()       { ... }   // only in dev
@Bean @Profile("prod") DataSource postgres() { ... }   // only in prod
@Profile("!prod")  // active when prod is NOT active
🔍 Profiles select per-environment configuration. The base application.yml holds shared defaults; application-{profile}.yml overrides them for a specific environment; and the active profile is chosen at runtime (via SPRING_PROFILES_ACTIVE or --spring.profiles.active), not baked into the jar. You can also scope beans to profiles with @Profile. The classic bug is forgetting to set the profile in production, so it runs with dev defaults — always set the active profile explicitly on the platform.

Part 4 — Where Config Comes From, Beyond Files

SourceTypical use
application.yml (packaged)Base defaults, non-secret structure
Profile files (application-prod.yml)Per-environment overrides
OS environment variablesSecrets and per-deployment values (12-factor)
Command-line argsOne-off overrides, container entrypoints
External config (Spring Cloud Config, Vault)Centralized/secure config across services
🔍 For a single service, files + env vars cover almost everything. In a microservices setup, centralized configuration (Spring Cloud Config server) and a secrets manager (HashiCorp Vault, AWS Secrets Manager) let many services share and securely source their config (the Microservices lesson). The property-source model extends naturally: these become additional, higher-precedence sources the same way env vars are.

Part 5 — Secrets: Keep Them Out of Git

# ❌ NEVER commit secrets to application.yml (they land in git history forever):
spring:
  datasource:
    password: "s3cr3t-do-not-do-this"

# ✅ Reference an env var; the value is injected by the platform at runtime:
spring:
  datasource:
    password: "${DB_PASSWORD}"
# The platform (container secret, CI/CD, Kubernetes Secret, Vault) supplies DB_PASSWORD.
# The config file holds STRUCTURE and non-secret defaults only.
🔐 Never commit secrets — passwords, API keys, tokens — to source control. Once in git history they're effectively permanent (and leaked if the repo is ever exposed). Keep only structure and non-secret defaults in application.yml, and inject secrets as environment variables (from a container secret, CI/CD variable, Kubernetes Secret, or a vault) referenced with ${VAR} placeholders. Profiles select which secret source/values, not the secret itself. This is a core secure-configuration practice and a frequent real-world breach cause when ignored.

Part 6 — Deep Dive: Configuration as an Operational Concern

  • One artifact, many environments. Externalized configuration plus precedence means you build a single immutable jar/image and reconfigure it per environment via env vars, args, and profiles — never a per-environment build. This is foundational to reliable, repeatable deployment.
  • Fail fast on bad config. Bind settings into typed, @Validated @ConfigurationProperties (previous lesson) so misconfiguration fails at startup with a clear message rather than causing a mysterious runtime error later.
  • Secrets are managed, not committed. Configuration discipline is a security boundary: secrets flow in from the environment/vault at runtime, never from the codebase, so the same artifact is safe to build and store anywhere.
  • Profiles keep environments honest. Dev conveniences (H2 console, create-drop) are isolated to the dev profile; prod gets safe settings — so a convenience can't accidentally ship to production.
🔍 Senior framing: configuration is an operational and security concern, not an afterthought, and Spring Boot's model is built around one principle — externalize everything environment-specific so a single immutable artifact runs everywhere, driven by configuration rather than by rebuilding. The property-source precedence (packaged file < profile file < env vars < system properties < command-line args, with relaxed binding of env-var names to property keys) is what makes that work: the base application.yml carries shared, non-secret defaults; profile files carry per-environment overrides selected at runtime via SPRING_PROFILES_ACTIVE (never hardcoded); and the platform injects deployment-specific values and, crucially, secrets as environment variables or from a vault, referenced by ${VAR} placeholders and never committed to git, because a secret in version history is a permanent leak. The mature engineer treats this deliberately: builds once and promotes the same image through environments, sets the active profile and injects secrets on the platform, binds configuration into typed, validated @ConfigurationProperties so bad config fails fast at startup with a clear message instead of a runtime surprise, isolates dev conveniences (H2 console, ddl-auto: create-drop) to the dev profile so they cannot reach production, and in a microservices context sources config from a central config server and secrets from a manager like Vault. The recurring real-world failures this prevents are exactly the ones interviewers and incident post-mortems focus on — committed credentials, a prod deploy accidentally running with dev defaults or a destructive create-drop, per-environment builds that drift, and mysterious runtime failures from unvalidated config — so a strong grasp of externalized configuration, precedence, profiles, and secret management signals someone who can ship and operate a service safely, not just write its code.

Interview Questions

Q1 (beginner): How does Spring Boot externalized configuration let one jar run in every environment?

Boot reads configuration from external sources rather than baking environment-specific values into the code, so you build a single immutable jar and reconfigure it per environment at runtime. The packaged application.yml holds shared, non-secret defaults; profile-specific files (application-prod.yml) hold per-environment overrides; and the platform supplies deployment-specific values and secrets via environment variables or command-line arguments, which override the file. Because later sources win, the same artifact picks up the right values in dev, staging, and production without a rebuild. This "build once, configure per environment" model is the basis of reliable, repeatable deployment.

Q2 (beginner): What are profiles and how do you activate one?

Profiles are named sets of configuration for different environments. The base application.yml holds shared defaults, and a profile-specific file like application-dev.yml or application-prod.yml overrides settings when that profile is active; you can also annotate beans with @Profile("dev") so they exist only in certain profiles. You activate a profile at runtime — not by hardcoding it in the jar — via the environment variable SPRING_PROFILES_ACTIVE=prod or the command-line argument --spring.profiles.active=prod. This lets the same artifact run with dev conveniences locally and safe settings in production, selected by the deployment environment. Forgetting to set the active profile in production (so it runs with dev defaults) is a classic bug.

Q3 (mid): Explain Spring Boot's property-source precedence.

Boot loads properties from many ordered sources and later sources override earlier ones. From lower to higher precedence, roughly: the packaged application.yml/.properties, then profile-specific files (application-{profile}.yml), then OS environment variables, then Java system properties (-D), then command-line arguments (--key=value), with a few additional sources like config-data imports and test properties. This ordering is what lets one immutable artifact be reconfigured per environment without rebuilding: the base file provides defaults and the platform injects env vars/args that win. Environment variables also bind to property keys by relaxed binding — SPRING_DATASOURCE_URL maps to spring.datasource.url — which is how you set nested properties via the environment. You can also reference an env var inside the file with a placeholder and default, e.g. ${PORT:8080}.

Q4 (mid): How should secrets be handled in a Spring Boot app?

Never in source control or the packaged config file — a secret committed to git is effectively permanent and leaked the moment the repository is exposed. Keep only structure and non-secret defaults in application.yml, and reference secrets via placeholders (e.g. password: ${DB_PASSWORD}) whose values are supplied at runtime by the environment: an OS environment variable injected by the platform, a container secret, a CI/CD secret variable, a Kubernetes Secret, or a secrets manager / vault (HashiCorp Vault, AWS Secrets Manager, Spring Cloud Vault). Profiles select which environment's values apply, not the secret itself. This keeps the same artifact deployable everywhere and keeps credentials out of git history, logs, and the image. For stronger setups, integrate a vault so secrets are fetched securely at startup and can be rotated without redeploying.

Q5 (senior): Why is "build once, configure per environment" important, and what failures does violating it cause?

Building a single immutable artifact and configuring it per environment at runtime is important because it guarantees that what you tested is exactly what you deploy: the same jar/image is promoted through dev, staging, and production, with only externalized configuration (env vars, args, active profile) differing, so there's no chance of behavioral drift from per-environment builds and the pipeline is simpler and reproducible. It's a core twelve-factor principle and the basis of trustworthy releases and easy rollbacks (you redeploy a known-good image). Violating it — producing a separate build per environment, or baking environment-specific values (including the active profile or secrets) into the artifact — causes a family of real failures: drift, where the prod binary differs subtly from what was tested so bugs appear only in production; the classic wrong-profile incident, where a deploy runs with dev defaults (an H2 database, a destructive ddl-auto: create-drop that wipes data on restart, verbose debug logging, or a disabled security setting) because the active profile wasn't set on the platform; secret leakage, when credentials committed to the image or repo end up in git history, image layers, or logs; non-reproducibility, where you can't reliably rebuild or roll back because the build depended on environment state; and configuration surprises at runtime when unvalidated config is wrong. The mitigations are precisely the externalized-configuration discipline: build once, inject per-environment values and secrets from the platform/vault, select the profile at runtime, bind config into typed and @Validated properties so bad config fails fast at startup, and keep dev conveniences isolated to the dev profile. The senior instinct is that the artifact should be inert with respect to environment — all environment-specific behavior comes from configuration supplied outside it — because that inertness is what makes deployments predictable, secure, and reversible.

Q6 (senior): How does the configuration model scale from a single service to microservices, and what does it add?

For a single service, the property-source model — packaged defaults, profile overrides, and env vars/args injecting per-deployment values and secrets — is sufficient and clean. As you move to many services, two new needs arise that the model extends to naturally, since additional sources simply plug into the same precedence chain. First, centralized configuration: instead of each service carrying and drifting its own copies of shared settings (database hosts, feature flags, common timeouts), a config server (Spring Cloud Config) serves configuration from a central, version-controlled source, so services fetch their config at startup (and can refresh it), giving one place to manage and audit config across the fleet and avoiding per-service duplication. Second, centralized secret management: rather than injecting each secret as an env var per deployment, services source secrets from a secrets manager or vault (HashiCorp Vault, AWS Secrets Manager, Spring Cloud Vault), which provides secure storage, fine-grained access control, auditing, and — importantly — rotation without redeploying, plus dynamic/short-lived credentials. These integrate as higher-precedence property sources (Spring's config-data / cloud config mechanisms), so the mental model is unchanged: later, more authoritative sources override earlier defaults, and services still bind everything into typed properties. The additions are operational and organizational: consistency and auditability of configuration across many services, secure and rotatable secrets, the ability to change config centrally (with refresh) rather than redeploying each service, and per-service profiles/environments layered on top. The senior point is that the same externalized-configuration principles scale up — you don't adopt a different model, you add central config and secret sources into the existing precedence chain — while the discipline (build once, don't commit secrets, validate config, select environment at runtime) becomes even more important because a misconfiguration or leaked secret now potentially affects an entire system of services rather than one.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Env var override. Set server.port in application.yml, then override it with an environment variable and a command-line arg, and confirm the precedence.
Solution

The file sets, say, 8080; SERVER_PORT=8090 java -jar app.jar (relaxed binding) runs on 8090; java -jar app.jar --server.port=9000 runs on 9000 (args are highest). This demonstrates the precedence chain and relaxed binding — the same jar, different ports per invocation, no rebuild.

2 (easy): Placeholder with default. Reference a ${PORT:8080} and a ${DB_PASSWORD} in the config, showing the default applies and the secret comes from the environment.
Solution

server.port: ${PORT:8080} uses 8080 when PORT is unset and the env value otherwise; spring.datasource.password: ${DB_PASSWORD} has no default and must be supplied by the environment. This shows how to keep secrets out of the file while giving non-secret settings sensible defaults.

3 (medium): Profiles. Create application-dev.yml and application-prod.yml with different ddl-auto and logging, boot with the prod profile, and confirm prod values load.
Solution

dev sets ddl-auto: create-drop and debug logging; prod sets ddl-auto: validate and info logging. java -jar app.jar --spring.profiles.active=prod loads the prod overrides on top of the base file. Note the profile is chosen at runtime, not baked into the jar — and forgetting it would leave you on dev defaults.

4 (medium): Profile-scoped beans. Define an H2 DataSource bean for @Profile("dev") and a Postgres one for @Profile("prod"); verify only one is created per active profile.
Solution

Two @Bean methods annotated @Profile("dev") and @Profile("prod"). With the dev profile active, only the H2 bean is created; with prod, only the Postgres one. This lets environment-specific infrastructure be selected declaratively, isolated from the other environment.

5 (hard): Secret-safe setup. Take a config that has a hardcoded database password and refactor it to source the secret from an environment variable. Explain what would go wrong if it stayed in git.
Solution

Replace password: "s3cret" with password: ${DB_PASSWORD} and inject DB_PASSWORD from the platform (container secret / K8s Secret / vault). If it stayed in git, the secret is in history forever — recoverable from any clone even after "removal", exposed if the repo leaks, and impossible to truly rotate without treating it as compromised. Secrets must be runtime-injected, never committed.

🎯 Key Takeaways

  • Externalize all environment-specific settings so one immutable artifact runs everywhere, driven by configuration.
  • Precedence (later wins): packaged file < profile file < env vars < system props < command-line args; env-var names map to keys via relaxed binding.
  • Profiles (application-{env}.yml, @Profile beans) hold per-environment overrides; activate at runtime via SPRING_PROFILES_ACTIVE — never hardcode it.
  • Never commit secrets — reference them with ${VAR} and inject from env vars, container secrets, or a vault; a secret in git history is a permanent leak.
  • Bind config into typed @Validated @ConfigurationProperties so bad config fails fast at startup; isolate dev conveniences to the dev profile.
  • The model scales to microservices via a central config server and a secrets manager — added as higher-precedence sources in the same chain.
