Configuration & Profiles Basics

Every real service needs different settings per environment — a local H2 database, a staging Postgres, production secrets from a vault. Spring Boot externalizes all of it so the same jar runs everywhere, driven by configuration rather than code. Getting this right is the difference between "works on my machine" and a clean deploy.

application.yml — The Config File

# src/main/resources/application.yml
server:
  port: 8080
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/shop
    username: shop
    # password comes from an ENV VAR, never hardcoded — see below
  jpa:
    hibernate:
      ddl-auto: validate          # never `create-drop` in prod!
app:
  payment:
    currency: USD
    retry-attempts: 3

YAML or .properties both work; YAML nests better. Anything Spring or a starter reads is just a property key — server.port, spring.datasource.url, etc.

Reading Config Into Code

// One value:
@Value("${app.payment.currency}") String currency;

// PREFERRED — a typed, validated group via @ConfigurationProperties:
@ConfigurationProperties(prefix = "app.payment")
@Validated
public record PaymentProps(
    @NotBlank String currency,
    @Min(1) int retryAttempts
) {}

@EnableConfigurationProperties(PaymentProps.class)   // register it (or @ConfigurationPropertiesScan)
@Configuration class ConfigSetup {}

@Service
class PaymentService {
    private final PaymentProps props;                // inject the whole typed group
    PaymentService(PaymentProps props) { this.props = props; }
}
🎯 Prefer @ConfigurationProperties over scattered @Value: you get type safety, IDE autocomplete, validation at startup, and a single cohesive object instead of a dozen string injections. @Value is fine for a one-off; groups belong in a properties class.

Property Precedence — Env Overrides File

# Spring reads config from many sources; LATER wins over earlier. Roughly:
#   application.yml  <  profile file  <  env vars  <  command-line args
# So the same jar picks up overrides without a rebuild:

DB_PASSWORD=s3cret java -jar app.jar                       # env var
java -jar app.jar --server.port=9000                       # command-line arg
# Reference an env var inside yml with a default:
#   password: ${DB_PASSWORD}
#   port: ${PORT:8080}       # 8080 if PORT unset
🔐 Never commit secrets. Keep passwords, API keys, and tokens out of application.yml and out of git. Inject them as environment variables (from the platform, a Docker secret, or a vault). The config file holds structure and non-secret defaults only.

Profiles — Per-Environment Config

# application.yml holds shared defaults. Profile-specific files override:
#   application-dev.yml    application-prod.yml

# application-dev.yml
spring:
  jpa: { hibernate: { ddl-auto: create-drop } }   # convenient locally
  h2: { console: { enabled: true } }

# application-prod.yml
spring:
  jpa: { hibernate: { ddl-auto: validate } }      # safe in prod

# Activate a profile:
java -jar app.jar --spring.profiles.active=prod
# or SPRING_PROFILES_ACTIVE=prod as an env var
// Beans can be profile-scoped too:
@Bean @Profile("dev")  DataSource h2()       { ... }   // only in dev
@Bean @Profile("prod") DataSource postgres() { ... }   // only in prod

Interview Questions

Q1: How does Spring Boot's externalized configuration precedence work?

Boot loads properties from many ordered sources and later sources override earlier ones. Roughly: packaged application.yml, then profile-specific files (application-{profile}.yml), then OS environment variables, then Java system properties, then command-line arguments (with a few more like config data imports and test properties). This lets one immutable jar be reconfigured per environment without rebuilding — the base file holds defaults, and the platform injects env vars/args that win. Env vars also map to property keys by relaxed binding (SPRING_DATASOURCE_URL → spring.datasource.url).

Q2: @Value vs @ConfigurationProperties?

@Value("${key}") injects a single property as a string/coerced type — fine for a one-off, but there's no grouping, no validation, and typos fail silently or at use. @ConfigurationProperties(prefix=…) binds a whole tree of related properties into a typed object (ideally a record), supports @Validated with Jakarta constraints so bad config fails fast at startup, gives IDE autocomplete via metadata, and keeps related settings cohesive. For anything more than a lone value, prefer configuration properties.

Q3: How should secrets be handled?

Never in source control or the packaged config file. Keep only structure and non-secret defaults in application.yml, and reference secrets via placeholders (password: ${DB_PASSWORD}) resolved from environment variables injected by the platform, container secrets, or a secrets manager/vault (AWS Secrets Manager, HashiCorp Vault, Spring Cloud Config with encryption). Profiles select which secret source, not the secret itself. This keeps the same artifact deployable everywhere and keeps credentials out of git history and logs.

🏋️ Practice: add an app.* config group bound with @ConfigurationProperties and validated; create application-dev.yml and application-prod.yml with different ddl-auto; reference a ${DB_PASSWORD} env var with a default; boot with --spring.profiles.active=prod and confirm the prod values load; override server.port from the command line.

🎯 Key Takeaways

  • Externalize all environment-specific settings; the same jar runs everywhere driven by config.
  • Prefer typed, validated @ConfigurationProperties groups over scattered @Value.
  • Precedence: file < profile file < env vars < command-line args — env/args override the file.
  • Keep secrets out of git — inject via env vars / vault with ${VAR} placeholders.
  • Profiles (application-{env}.yml, @Profile beans) select per-environment config.
