AOP & Proxies Cross-Cutting

What you'll master in this lesson: Aspect-Oriented Programming — the mechanism behind every "magic" cross-cutting feature in Spring (transactions, caching, async, security, retry); the vocabulary (aspect, advice, pointcut, join point); how Spring implements AOP with proxies, and why that single fact explains the self-invocation gotcha you met in the transactions lesson; and how to write your own aspect for logging, timing, or auditing. This lesson ties together the recurring proxy theme across the track — once you understand AOP, the framework's magic becomes predictable engineering.

Part 1 — The Problem AOP Solves: Cross-Cutting Concerns

// WITHOUT AOP — logging, timing, transactions, security tangled into every method:
public UserDto createUser(CreateUserRequest req) {
    log.info("createUser start");                    // logging concern
    long t0 = System.nanoTime();                     // timing concern
    checkPermission("user:create");                  // security concern
    tx.begin();                                       // transaction concern
    try {
        UserDto result = doCreateUser(req);           // ← the ONE line of actual business logic
        tx.commit();
        return result;
    } finally {
        log.info("createUser took " + (System.nanoTime()-t0));
    }
}
// The business logic drowns in cross-cutting boilerplate, repeated in EVERY method. AOP extracts it.

A cross-cutting concern is behavior needed across many methods but unrelated to any one's core logic — logging, timing, transactions, security, caching, retry. Scattering it through every method is repetitive and tangles concerns. AOP lets you define that behavior once (an aspect) and apply it declaratively to many methods.

Part 2 — The AOP Vocabulary

TermMeaning
AspectA module of cross-cutting behavior (e.g. "logging aspect")
AdviceThe action taken — before, after, around a method
Join pointA point where advice can apply (in Spring, a method execution)
PointcutAn expression selecting which join points to advise
WeavingApplying aspects to targets (in Spring, at runtime via proxies)
@Aspect
@Component
public class TimingAspect {
    @Around("execution(* com.example.service.*.*(..))")   // POINTCUT: all methods in the service package
    public Object time(ProceedingJoinPoint pjp) throws Throwable {   // AROUND advice
        long t0 = System.nanoTime();
        try {
            return pjp.proceed();                          // run the actual method
        } finally {
            log.info(pjp.getSignature() + " took " + (System.nanoTime() - t0) + "ns");
        }
    }
}
💡 An aspect (@Aspect) contains advice methods (@Before, @After, @AfterReturning, @AfterThrowing, @Around) each with a pointcut expression saying which methods (join points) they apply to. @Around is the most powerful — it wraps the method, so you control before and after and whether to call proceed() at all. The pointcut expression language (execution(...), plus matching by annotation, class, etc.) is how you target exactly the methods you want.

Part 3 — How Spring Implements AOP: Proxies

// Spring "weaves" aspects at RUNTIME by wrapping your bean in a PROXY (Bean Lifecycle lesson):
//
//   caller ──► [ PROXY ] ──► your real bean
//                 │
//                 └─ runs the advice (start tx, check cache, log, ...) around the real method
//
// Two proxy types:
//   - JDK dynamic proxy: if the bean implements an interface (proxies the interface)
//   - CGLIB proxy:       if not (creates a runtime SUBCLASS of the bean)   ← Boot's default now
🔑 This is the single most important fact about Spring AOP. When a bean needs cross-cutting behavior, a BeanPostProcessor replaces it with a proxy during initialization (Bean Lifecycle lesson). Other beans receive the proxy, not your object; the proxy runs the advice around each call and then delegates to your real bean. This is why @Transactional, @Cacheable, @Async, and @PreAuthorize work at all — and why they share the same limitation.

Part 4 — Why Self-Invocation Breaks (the unifying gotcha)

@Service
public class OrderService {
    public void process() {
        this.placeOrder();          // ❌ INTERNAL call — goes straight to the real method, NOT the proxy
    }
    @Transactional                  // (or @Cacheable, @Async, @PreAuthorize)
    public void placeOrder() { ... }  // advice is SKIPPED — no transaction/cache/async
}
🚨 The proxy only intercepts calls that come THROUGH it — i.e. from other beans. An internal this.method() call never leaves the object, so it bypasses the proxy and none of the advice runs. This is the same root cause behind the transactions lesson's self-invocation gotcha, the caching lesson's, and the async one's — all proxy-based annotations share it. Fixes: call the annotated method from another bean, self-inject the proxy and call through that reference, or (rarely) switch to compile-time AspectJ weaving, which weaves the advice into the class itself so even internal calls are advised. Understanding proxies makes this one concept, not four separate mysteries.

Part 5 — Writing Your Own Aspect

// A custom @Audited annotation + an aspect that records audit events — declarative cross-cutting:
@Retention(RUNTIME) @Target(METHOD)
public @interface Audited { String action(); }

@Aspect @Component
public class AuditAspect {
    @AfterReturning(pointcut = "@annotation(audited)", returning = "result")
    public void audit(JoinPoint jp, Audited audited, Object result) {
        auditLog.record(audited.action(), currentUser(), jp.getArgs());
    }
}
// Use it declaratively:
@Audited(action = "DELETE_USER")
public void deleteUser(long id) { ... }        // the aspect logs an audit event on success
🎯 Custom aspects are ideal for genuinely cross-cutting concerns you control — auditing, timing/metrics, structured logging, custom retry, feature flags. An @annotation(...) pointcut targets methods bearing a marker annotation, keeping the behavior declarative and opt-in. But use AOP judiciously: it's powerful but adds indirection that can obscure control flow. Reserve it for true cross-cutting concerns; don't hide business logic in aspects, where it's hard to find and reason about.

Part 6 — Deep Dive: AOP as the Framework's Foundation

  • AOP is why Spring feels declarative. You annotate a method (@Transactional, @Cacheable, @Async, @PreAuthorize, @Retryable) and behavior appears — all implemented as aspects woven via proxies. Knowing this, the "magic" is a single, understandable mechanism.
  • Proxies explain the shared limitations. Self-invocation, the need for public methods (JDK proxies can't advise private/final in the usual way), and the fact that the injected object is a proxy — all follow from runtime proxy-based weaving. One mental model, many consequences.
  • Separation of concerns at the method boundary. AOP keeps business methods focused on business logic while cross-cutting concerns live in aspects — the Single Responsibility Principle (Java SOLID lesson) applied across the codebase.
  • Use it sparingly and for the right things. Cross-cutting infrastructure (tx, cache, security, metrics, audit) is the sweet spot. Business logic in aspects is an anti-pattern — it scatters domain rules into invisible, hard-to-test places.
🔍 Senior framing: AOP is the foundational mechanism that makes Spring's declarative programming model possible, and understanding it converts a whole category of framework "magic" into one predictable concept. A cross-cutting concern — a transaction boundary, a cache lookup, an async dispatch, a security check, a metric, an audit record — is behavior that many methods need but that isn't any method's business logic, and AOP lets you define it once as an aspect (advice targeted by a pointcut) and apply it declaratively wherever it's needed, keeping business methods clean. Crucially, Spring implements this by weaving aspects at runtime through proxies: a BeanPostProcessor wraps the bean in a JDK dynamic proxy or CGLIB subclass during initialization, other beans receive that proxy, and the proxy runs the advice around each call before delegating to the real object — which is precisely why @Transactional, @Cacheable, @Async, and method-level security work, and why they all share the same limitations. The most important of these is self-invocation: because the proxy only intercepts calls that arrive through it from other beans, an internal this.method() call bypasses it and the advice silently does nothing — a single root cause that, once understood, explains the recurring gotcha across transactions, caching, and async rather than four separate surprises, and whose fixes (call through another bean, self-inject the proxy, or use compile-time AspectJ weaving) all follow from the mechanism. The mature engineer therefore uses AOP deliberately: leans on the framework's built-in aspects for standard cross-cutting infrastructure, writes custom aspects for genuinely cross-cutting concerns like auditing, timing, and structured logging (typically targeted by a marker annotation to stay declarative and opt-in), keeps the proxy model in mind to avoid and diagnose self-invocation and related issues, and resists putting business logic in aspects because that hides domain rules in invisible, hard-to-test indirection. Interviewers ask about AOP, proxies, and self-invocation because the answers reveal whether a candidate understands how Spring's declarative features are actually implemented — a proxy wrapping the bean and running advice around external calls — which is the difference between someone who can only apply the annotations and someone who can reason about, extend, and debug the framework's cross-cutting behavior.

Interview Questions

Q1 (beginner): What is AOP and what problem does it solve?

Aspect-Oriented Programming is a way to modularize cross-cutting concerns — behavior that many methods need but that isn't any one method's core logic, such as logging, timing, transactions, security, caching, and retry. Without AOP that behavior gets scattered and repeated through every method, tangling infrastructure with business logic. AOP lets you define the behavior once as an aspect and apply it declaratively to many methods (selected by a pointcut), so business methods stay focused on business logic while the cross-cutting behavior lives in one place. In Spring, this is how @Transactional, @Cacheable, @Async, and security annotations are implemented.

Q2 (beginner): Define aspect, advice, pointcut, and join point.

An aspect is a module of cross-cutting behavior (e.g. a logging or timing aspect). A join point is a point in program execution where behavior can be inserted — in Spring AOP, a method execution. A pointcut is an expression that selects which join points (which methods) an aspect applies to — for example "all methods in the service package" or "methods annotated with @Audited". Advice is the actual action taken at a matched join point, and it comes in flavors by timing: @Before, @AfterReturning, @AfterThrowing, @After (finally), and @Around (which wraps the method and can run code before and after and decide whether to proceed). So: an aspect bundles advice, a pointcut says where it applies, and each application site is a join point.

Q3 (mid): How does Spring implement AOP, and what are the two proxy types?

Spring uses runtime proxy-based weaving. During bean initialization, a BeanPostProcessor detects that a bean needs cross-cutting behavior and replaces it with a proxy that wraps the real bean; other beans are injected with the proxy, and when a method is called the proxy runs the relevant advice (start a transaction, check the cache, log, enforce security) and then delegates to the real object. There are two proxy mechanisms: a JDK dynamic proxy, used when the bean implements an interface — it creates a proxy implementing that interface; and a CGLIB proxy, used when the bean doesn't implement an interface (or when configured as the default, as in current Spring Boot) — it creates a runtime subclass of the bean and overrides its methods to add the advice. Because CGLIB subclasses the bean, it can't proxy final classes or final methods, and because both approaches wrap the object, only calls that go through the proxy (from other beans) are advised — which is the source of the self-invocation limitation.

Q4 (mid): Why does self-invocation bypass Spring AOP, and how do you fix it?

Because Spring AOP works by wrapping the bean in a proxy, and the proxy only intercepts calls that arrive through it — that is, calls made by other beans holding the proxy reference. When a method of the bean calls another method of the same bean directly via this.method(), the call is a plain in-object invocation that never passes through the proxy, so the advice (transaction, cache, async, security) is never applied and the annotation silently does nothing. This is one root cause behind the confusing "my @Transactional/@Cacheable/@Async method isn't working when called internally" reports. Fixes: (1) call the annotated method from a different bean, so the call crosses the proxy; (2) inject the bean's own proxy into itself (self-injection) and invoke the method through that reference; (3) restructure so the annotated method is a genuine external entry point; or (4) use full AspectJ compile-time (or load-time) weaving instead of Spring's proxy-based AOP, which weaves the advice into the class bytecode so even internal calls are advised. The right first step is usually to move the annotated method to a separate bean or rethink the call structure rather than reaching for AspectJ.

Q5 (senior): When should you write a custom aspect, and when is AOP the wrong tool?

Write a custom aspect when you have a genuine cross-cutting concern — behavior that applies uniformly across many methods and is orthogonal to their business logic — that isn't already covered by a framework annotation. Good candidates: auditing (record who did what), timing and metrics collection, structured or contextual logging, custom retry/circuit-breaking, feature-flag gating, and enforcing a policy across a layer. Targeting such an aspect with an @annotation pointcut (a marker like @Audited) keeps it declarative, opt-in, and visible at the call site, which mitigates AOP's main downside. AOP is the wrong tool when the behavior is actually business logic: putting domain rules or core workflow into aspects scatters them into invisible, hard-to-discover, hard-to-test indirection — a maintainer reading the method can't see what an aspect silently adds, control flow becomes non-local, and debugging is harder. It's also wrong when a simpler mechanism suffices (a plain method call, a decorator, a base class, or just calling a shared service is often clearer than an aspect), when the pointcut would be so broad or fragile that it advises unintended methods, or when the indirection cost outweighs the duplication it removes for a concern that only appears in a couple of places. The senior judgment is that AOP is powerful precisely because it's invisible and pervasive, so it should be reserved for infrastructure/cross-cutting concerns where that invisibility is a feature (you genuinely want every method transactional or audited without cluttering them), and avoided for anything a reader needs to see explicitly to understand the code's behavior. Overusing AOP produces systems where a method's real behavior is spread across aspects nobody can find, which is a maintainability trap as bad as the boilerplate AOP was meant to remove.

Q6 (senior): How does understanding the proxy mechanism help you reason about Spring's declarative features as a whole?

Because nearly all of Spring's declarative, annotation-driven behavior — @Transactional, @Cacheable/@CacheEvict, @Async, method-level security (@PreAuthorize), @Retryable, validation on beans, and custom aspects — is implemented by the same mechanism: at initialization a BeanPostProcessor wraps the bean in a proxy, and the proxy runs advice around external method calls before delegating to the target. Holding this single model lets you reason uniformly about a family of otherwise-disparate features and their shared consequences instead of memorizing each as a special case. Concretely, it explains: (1) self-invocation failures across all of them — an internal this.method() bypasses the proxy, so none of the advice applies, and the fix is always the same (cross a bean boundary or use real weaving); (2) why the object you inject is not literally your class but a proxy, which matters when debugging, reflecting, or comparing types, and why CGLIB proxies can't advise final classes/methods; (3) the ordering of multiple aspects on one method (multiple proxies/advisors compose, and their order can matter — e.g. transaction vs security vs cache), which you can control with @Order; (4) that advice runs at the method boundary, so behaviors like transactions and caching are per-call around a bean method, which informs where you place annotations (service methods, external entry points) and why placement on the wrong method or the wrong layer misbehaves; and (5) how to extend the framework — writing your own aspect plugs into the same machinery, so custom cross-cutting concerns behave consistently with the built-in ones. In short, once you know "declarative feature = aspect woven via a proxy that intercepts external calls," the framework stops being a collection of magic annotations and becomes a coherent, debuggable system: you can predict when a feature will and won't apply, diagnose why one silently didn't, decide where to place annotations, reason about aspect interactions, and add your own — which is exactly the depth that distinguishes an engineer who can operate and troubleshoot Spring in production from one who can only apply annotations and hope.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Timing aspect. Write an @Around aspect that logs how long every method in your service package takes. Confirm it applies without changing the service code.
Solution

An @Aspect @Component with @Around("execution(* com.example.service.*.*(..))") that records nanoTime around pjp.proceed() and logs the signature and duration. The service methods are unchanged — the timing is woven in via the proxy. That's cross-cutting behavior applied declaratively.

2 (easy): Advice types. For logging on success vs on exception, which advice types would you use? Sketch both.
Solution

@AfterReturning for success (access the returned value), @AfterThrowing for exceptions (access the thrown exception). @Before/@After for entry/finally, and @Around if you need both sides and control over proceeding. Matching advice type to the moment you care about is the skill.

3 (medium): Reproduce self-invocation. Add @Cacheable to a method, call it from another method of the same class, and show it isn't cached; then fix it.
Solution

Calling the @Cacheable method via this.method() bypasses the proxy, so the cache advice never runs — every call recomputes. Fix by calling it from another bean or self-injecting the proxy. This is the same proxy-bypass as the transactions gotcha — one mechanism, one fix pattern.

4 (medium): Annotation-targeted aspect. Create an @Audited annotation and an aspect that records an audit event after any @Audited method returns.
Solution

A runtime method annotation @Audited(action=...) and an @AfterReturning(pointcut="@annotation(audited)", ...) aspect that records the action, current user, and args. Annotate any method with @Audited and it's audited — declarative, opt-in, visible at the call site. This is the idiomatic way to keep custom cross-cutting behavior discoverable.

5 (hard): Explain the shared gotcha. Explain in proxy terms why @Transactional, @Cacheable, and @Async all fail on self-invocation, and list the fixes.
Solution

All three are implemented as aspects woven via a proxy that wraps the bean; the proxy runs their advice only around calls that come through it (from other beans). An internal this.method() call never crosses the proxy, so the advice — transaction, cache lookup, async dispatch — is skipped. Fixes: call through another bean, self-inject the proxy and call via it, or use AspectJ compile-time weaving (advises the class itself). One root cause, one set of fixes.

6 (hard): Explain-it test. Out loud: (a) what AOP is and its vocabulary; (b) how Spring implements it with proxies and the two proxy types; (c) why self-invocation breaks it and when to (and not to) use AOP.
Self-check rubric

Pass (a) if you defined cross-cutting concerns and aspect/advice/pointcut/join point. Pass (b) if you said a BeanPostProcessor wraps the bean in a JDK or CGLIB proxy that runs advice around calls. Pass (c) if you said internal calls bypass the proxy (so advice is skipped) and reserved AOP for cross-cutting concerns, not business logic. This unifies the transactions/caching/async gotchas.

🎯 Key Takeaways

  • AOP modularizes cross-cutting concerns (tx, cache, async, security, logging) as aspects applied declaratively — keeping business methods clean.
  • Vocabulary: aspect (module), advice (before/after/around action), pointcut (which methods), join point (a method execution).
  • Spring implements AOP by wrapping beans in proxies (JDK dynamic or CGLIB) that run advice around external calls — this is how @Transactional/@Cacheable/@Async work.
  • Self-invocation bypasses the proxy — the unifying root cause of the tx/cache/async gotchas; fix by crossing a bean boundary or self-injecting the proxy.
  • Write custom aspects for genuine cross-cutting concerns (auditing, timing, logging), ideally targeted by a marker annotation; never put business logic in aspects.
  • Understanding proxies turns Spring's "magic" into one predictable mechanism you can reason about, extend, and debug.
