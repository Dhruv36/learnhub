Request Mapping & Binding Web Layer

What you'll master in this lesson: exactly how Spring extracts data from an HTTP request and binds it to your method parameters — path variables, query parameters, request bodies, headers; how request mapping matches a request to a handler (path, method, content type); pagination and sorting via Pageable; content negotiation and JSON serialization details; and CORS for browser clients. The previous lesson established controller design; this one is the mechanics of getting data in and out correctly — the everyday details that cause real bugs when misunderstood.

Part 1 — Binding Request Data

        @PathVariable long id,                                    // from the path
        @RequestParam(defaultValue = "0") int page,               // from the query, with a default

@PostMapping("/users")

    return service.create(req);
}
💡 Spring resolves each parameter from the request based on its annotation and type, converting strings to the target type (a path "42" to a long, a query "2024-01-15" to a LocalDate). A type-conversion failure (e.g. "abc" for an int path variable) results in a 400 by default. required and defaultValue control optionality on @RequestParam.

Part 2 — Request Mapping: Matching a Request to a Handler

@RestController
@RequestMapping("/api/users")                 // class-level base path
public class UserController {

    @GetMapping                                // GET  /api/users
    @GetMapping("/{id}")                        // GET  /api/users/42
    @PostMapping                                // POST /api/users
    @PutMapping("/{id}")  @PatchMapping("/{id}")  @DeleteMapping("/{id}")

    // Narrow the match further:
    @PostMapping(consumes = "application/json", produces = "application/json")
    @GetMapping(params = "role")                // only if a 'role' query param is present
    @GetMapping(headers = "X-API-Version=2")    // only for a specific header value
}
🔍 A handler is selected by path + HTTP method, and can be narrowed by consumes (the request's Content-Type), produces (the client's Accept), params, and headers. If no handler matches, you get a 404 (no path/method match) or 405 (path matches but wrong method) or 415 (unsupported media type). The @GetMapping/@PostMapping shortcuts are just @RequestMapping(method = …) — use the shortcuts for readability.

Part 3 — Pagination & Sorting with Pageable

// Spring Data + Spring MVC bind paging/sorting query params to a Pageable automatically:
@GetMapping("/users")
public Page<UserDto> list(Pageable pageable) {          // ?page=0&size=20&sort=name,asc
    return service.list(pageable);                      // pass straight to a Spring Data repository
}
// Page<T> carries the content plus total count, total pages, and current page — ready for a client.

// Manual paging params if you prefer explicit control:
@RequestParam(defaultValue = "0") int page,
@RequestParam(defaultValue = "20") int size
🎯 Pageable is a huge convenience: declare it as a parameter and Spring binds page, size, and sort query params to it, then you pass it directly to a Spring Data repository method, which returns a Page<T> with content and paging metadata (the JPA lesson). This turns "implement pagination" into a one-line parameter. Cap the page size (spring.data.web.pageable.max-page-size) so a client can't request a million rows.

Part 4 — Content Negotiation & JSON Serialization

// Jackson (auto-configured by the web starter) serializes/deserializes JSON both ways automatically.
// Customize the mapping with Jackson annotations on your DTOs:
public record UserDto(
    long id,
    @JsonProperty("full_name") String name,        // rename the JSON field
    @JsonFormat(pattern = "yyyy-MM-dd") LocalDate joined,
    @JsonIgnore String internalNote                // never serialize this
) {}

// Global Jackson tuning via properties (no code):
//   spring.jackson.property-naming-strategy=SNAKE_CASE
//   spring.jackson.default-property-inclusion=non_null   // omit null fields
//   spring.jackson.serialization.write-dates-as-timestamps=false   // ISO dates, not epoch
🔍 Content negotiation lets the same endpoint serve different representations based on the client's Accept header (JSON by default; XML if configured). Jackson handles JSON both directions; you control the shape with annotations (@JsonProperty, @JsonIgnore, @JsonFormat) or global properties. This is the same "shape your output explicitly" discipline as DTOs (previous lesson) — the serialization layer is where the JSON contract is finalized.

Part 5 — CORS for Browser Clients

// A browser SPA on a DIFFERENT origin is blocked by the same-origin policy unless you enable CORS.
// Per-controller:
@CrossOrigin(origins = "https://app.example.com")
@RestController class UserController { }

// Global (preferred — explicit, one place):
@Configuration
class CorsConfig implements WebMvcConfigurer {
    public void addCorsMappings(CorsRegistry r) {
        r.addMapping("/api/**")
         .allowedOrigins("https://app.example.com")     // NEVER "*" with credentials
         .allowedMethods("GET", "POST", "PUT", "DELETE");
    }
}
⚠️ CORS is a browser security mechanism, and misconfiguring it is a real vulnerability. The browser blocks cross-origin requests unless the server explicitly allows the origin. Configure allowed origins explicitly — never "*", especially not with credentials (cookies), which the spec forbids and which would expose your API broadly. Prefer a global WebMvcConfigurer config over scattered @CrossOrigin annotations so the policy is in one auditable place. Note CORS is enforced by the browser, not a substitute for server-side authorization.

Part 6 — Deep Dive: The Binding Pipeline & Its Failure Modes

  • Argument resolvers do the binding. The DispatcherServlet uses a chain of HandlerMethodArgumentResolvers, each responsible for one kind of parameter (@PathVariable, @RequestParam, @RequestBody via message converters, etc.). Understanding this explains where binding can fail and how to customize it.
  • Type conversion is automatic but fallible. Spring converts request strings to your parameter types; a conversion failure yields a 400 (MethodArgumentTypeMismatchException). You can register custom converters/formatters for domain types.
  • Message converters handle bodies. @RequestBody/@ResponseBody go through HttpMessageConverters (Jackson for JSON); malformed JSON yields a 400 (HttpMessageNotReadableException) before your method runs.
  • Validation runs at binding. @Valid on a bound parameter triggers Bean Validation before the method executes; failures become a 400 (Validation lesson). So binding, conversion, and validation are the gate every request passes before your logic sees clean, typed input.
🔍 Senior framing: request mapping and binding are the boundary where the untyped, string-based HTTP world is translated into typed, validated Java, and understanding the pipeline turns a lot of "why is this a 400/404/415?" confusion into precise diagnosis. A request is matched to a handler by path, method, and optionally content type/params/headers (a mismatch producing a specific 404/ 405/415), then each method parameter is populated by an argument resolver — path/query/header values converted to their target types (conversion failures becoming 400s), and the body deserialized by a message converter like Jackson (malformed JSON becoming a 400) — with @Valid triggering validation at the same stage. The practical implications are large: you get clean, typed, validated input to your method for free, but you must design the binding surface deliberately — meaningful path/query structure, DTOs whose JSON contract you control via Jackson annotations or global properties, sensible defaults and optionality on parameters, capped and convenient pagination via Pageable, and correctly-scoped CORS for browser clients (explicit origins, never a wildcard with credentials). Getting this layer right means your controllers receive well-formed domain data and clients get predictable, correctly-negotiated responses; getting it wrong produces the subtle bugs — silent type coercion surprises, over-broad CORS holes, unbounded page sizes, leaked or mis-serialized fields — that interviewers and code reviewers look for. The framework does the mechanical work; the engineering value is in shaping the request/response contract precisely and knowing, from the pipeline, exactly where and why binding succeeds or fails.

Interview Questions

Q1 (beginner): What's the difference between @PathVariable, @RequestParam, and @RequestBody?

@PathVariable binds a segment of the URL path (from a template like /users/{id}) to a parameter — used for identifying a resource. @RequestParam binds a query-string (or form) parameter (like ?page=2) — used for filters, options, and pagination, and supports required and defaultValue. @RequestBody binds the entire request body, deserializing it (JSON via Jackson) into an object — used for the payload of POST/PUT/PATCH. In short: path variable identifies the resource in the URL, request param passes options in the query string, request body carries the structured payload.

Q2 (beginner): How does Spring know which handler method to call?

Through request mapping: each handler declares a path and HTTP method via @RequestMapping/@GetMapping/etc., and Spring's DispatcherServlet matches an incoming request's URL and method to the corresponding method. The match can be narrowed further by consumes (the request's Content-Type), produces (the client's Accept), and required params or headers. If no handler's path matches you get a 404; if the path matches but the method doesn't, a 405; if the content type isn't supported, a 415. So mapping is path + method plus optional content/param/header constraints.

Q3 (mid): How do you implement pagination and sorting in a Spring endpoint?

Declare a Pageable parameter on the handler method; Spring MVC binds the page, size, and sort query parameters to it automatically (e.g. ?page=0&size=20&sort=name,asc). You then pass that Pageable directly to a Spring Data repository method, which returns a Page<T> containing the page of content plus metadata — total elements, total pages, current page — ready to return to the client. This requires almost no code. Important production detail: cap the maximum page size (via spring.data.web.pageable.max-page-size or validation) so a client can't request an unbounded number of rows and exhaust memory or the database. You can also set default page size and provide manual @RequestParam paging if you want explicit control.

Q4 (mid): How do you control the JSON shape of a response?

Primarily by using DTOs/records whose fields are exactly what you want to expose, and then refining the serialization with Jackson. Per-field control comes from annotations on the DTO: @JsonProperty("name") renames a field, @JsonIgnore excludes it, @JsonFormat controls date/number formatting, and @JsonInclude can omit null or empty values. Global control comes from properties (no code): spring.jackson.property-naming-strategy (e.g. snake_case), spring.jackson.default-property-inclusion (e.g. non_null), spring.jackson.serialization.write-dates-as-timestamps=false for ISO dates, and so on, or by customizing the ObjectMapper bean. Combined with content negotiation (the same endpoint can serve JSON or, if configured, XML based on the Accept header), this lets you define a precise, stable JSON contract while keeping your DTOs clean.

Q5 (senior): Explain CORS, why it exists, and how to configure it safely in Spring.

CORS (Cross-Origin Resource Sharing) is a browser security mechanism enforcing the same-origin policy: by default a web page can't make requests to a different origin (scheme+host+port) than the one it was served from, which prevents a malicious site from silently calling APIs the user is authenticated to. CORS is the controlled relaxation of that policy — the server tells the browser, via response headers (Access-Control-Allow-Origin etc.), which foreign origins, methods, and headers are permitted, and the browser enforces it (including a preflight OPTIONS request for non-simple calls). It exists because legitimate architectures (a browser SPA hosted on one origin calling an API on another) need cross-origin access, but it must be granted deliberately. In Spring you configure it either per-controller with @CrossOrigin or, preferably, globally via a WebMvcConfigurer's addCorsMappings (or a CorsConfigurationSource bean when using Spring Security) so the policy lives in one auditable place. Safe configuration means listing the exact allowed origins explicitly — never a wildcard "*", and specifically never "*" together with allowCredentials(true), which the spec forbids and which would let any site make credentialed requests — restricting allowed methods and headers to what's needed, and setting a sensible max-age for preflight caching. Two crucial senior caveats: CORS is enforced only by browsers, so it is not a security control against non-browser clients (curl, servers) and must never be treated as a substitute for server-side authentication and authorization; and because it's easy to over-open, a permissive CORS policy is a common real vulnerability that reviewers look for. So the safe pattern is explicit origins, minimal methods/headers, credentials only with specific origins, centralized configuration, and the understanding that CORS relaxes the browser's protection but is layered on top of — not instead of — real authorization.

Q6 (senior): Walk through what happens when a client POSTs malformed JSON, and how the framework produces the response.

The request first matches a handler by path, HTTP method, and — for a body — the consumes/Content-Type constraint (a mismatched content type would itself produce a 415 before any parsing). Once matched, resolving the @RequestBody parameter invokes the appropriate HttpMessageConverter — Jackson's for application/json — to deserialize the body into the target DTO. Because the JSON is malformed (a syntax error, or a value that can't be mapped to the field's type), Jackson throws during parsing and Spring wraps it in an HttpMessageNotReadableException, which happens before your controller method is ever invoked — so your business logic never sees the bad input. Spring's exception handling then turns that into a response: by default a 400 Bad Request with a generic error body, but in a well-built application a @RestControllerAdvice with an @ExceptionHandler(HttpMessageNotReadableException.class) (and, for validation failures on well-formed-but-invalid JSON, MethodArgumentNotValidException) maps it to a consistent, typed error payload with the 400 status (the Error Handling and Validation lessons). If instead the JSON were well-formed but violated a constraint (e.g. a blank required field), it would parse successfully and then fail at the @Valid validation step, again yielding a 400 but via a different exception. Understanding this flow — content-type match → message-converter deserialization → (validation) → handler invocation, with failures at each stage producing a specific status and exception type before your code runs — is what lets you provide clean, consistent 400 responses for every category of bad input rather than leaking framework default error pages, and to distinguish "unparseable body" (400 from the converter) from "parseable but invalid" (400 from validation) from "wrong content type" (415) when diagnosing client errors.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Bind everything. Write one endpoint that reads a path variable, an optional query param with a default, a required header, and a JSON body — and echoes them back.
Solution

A handler with @PathVariable long id, @RequestParam(defaultValue="0") int page, @RequestHeader("X-Tenant") String tenant, and @RequestBody SomeDto body, returning a small object combining them. Each parameter is bound from its respective request part; a missing X-Tenant header would 400 (required), while page defaults to 0 if absent.

2 (easy): Pageable. Add a Pageable parameter to a list endpoint and pass it to a repository; call it with ?page=1&size=5&sort=name,desc and inspect the Page metadata.
Solution

Declare list(Pageable pageable) and return repo.findAll(pageable).map(this::toDto);. The query params bind to the Pageable; the returned Page exposes getContent(), getTotalElements(), getTotalPages(), and the current page. Almost no code implements full pagination — remember to cap the max page size.

3 (medium): Shape the JSON. Given a DTO, rename one field to snake_case, omit nulls, format a date as ISO, and hide an internal field — using Jackson annotations, then globally via properties.
Solution

Per-field: @JsonProperty("full_name"), @JsonInclude(NON_NULL) (or on the field), @JsonFormat(pattern="yyyy-MM-dd"), @JsonIgnore. Global equivalents: spring.jackson.property-naming-strategy=SNAKE_CASE, spring.jackson.default-property-inclusion=non_null, spring.jackson.serialization.write-dates-as-timestamps=false. Both approaches finalize the JSON contract; annotations for per-field control, properties for app-wide defaults.

4 (medium): Configure CORS. Enable CORS for an SPA at https://app.example.com on /api/** for GET/POST/PUT/DELETE, globally. Explain why not "*".
Solution

A WebMvcConfigurer with addCorsMappings mapping /api/**, allowedOrigins("https://app.example.com"), and the four methods. Not "*" because it would allow any site to call your API, and "*" with credentials is forbidden by the spec and a security hole — explicit origins scope access to your known front-end. Global config keeps the policy auditable in one place.

5 (hard): Diagnose the status codes. For each, state the status and why: (a) POST with Content-Type: text/plain to a JSON endpoint; (b) GET on a path that exists but with the wrong method; (c) a path variable abc where an int is expected; (d) malformed JSON body.
Solution

(a) 415 Unsupported Media Type — the consumes=application/json constraint doesn't match text/plain. (b) 405 Method Not Allowed — path matches, method doesn't. (c) 400 Bad Request — type-conversion failure (MethodArgumentTypeMismatchException). (d) 400 Bad Request — the message converter can't parse the body (HttpMessageNotReadableException), before your method runs. Mapping symptoms to the pipeline stage is the diagnostic skill.

6 (hard): Explain-it test. Out loud: (a) the binding annotations and what each reads; (b) how request mapping selects a handler and the 404/405/415 distinctions; (c) safe CORS configuration and why CORS isn't authorization.
Self-check rubric

Pass (a) if you mapped path/query/body/header to their annotations. Pass (b) if you said path+method (+consumes/produces/params/headers) and distinguished 404 (no path) / 405 (wrong method) / 415 (bad media type). Pass (c) if you specified explicit origins (never "*" with credentials), central config, and that CORS is browser-enforced and not a replacement for server-side authz. These set up validation and error handling.

🎯 Key Takeaways

  • Bind data with @PathVariable (path), @RequestParam (query, with required/defaultValue), @RequestBody (JSON body), @RequestHeader (headers).
  • Handlers match by path + HTTP method, narrowable by consumes/produces/params/headers — mismatches give 404/405/415.
  • Declare a Pageable for automatic pagination/sorting → a Page<T> with metadata; cap the max page size.
  • Control the JSON contract with Jackson annotations (@JsonProperty/@JsonIgnore/@JsonFormat) or global properties; content negotiation serves representations by Accept.
  • CORS: allow explicit origins (never "*" with credentials), configure globally, and remember it's browser-enforced — not a substitute for authorization.
  • Binding, type conversion, and validation form the gate that turns untyped HTTP into clean, typed input — failures produce specific 400/415 statuses before your method runs.
