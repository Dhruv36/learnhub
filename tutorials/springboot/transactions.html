Transactions Data Layer

What you'll master in this lesson: what @Transactional actually does and where to put it; the ACID guarantees and why multi-step writes must be atomic; the two gotchas that cause the most confusion — self-invocation (a proxy is bypassed) and the rollback rules (checked exceptions don't roll back by default); and propagation and isolation levels for advanced control. Transactions are how you keep your data consistent under concurrent writes and partial failures, and @Transactional's subtleties are among the most-tested Spring interview topics.

Part 1 — @Transactional: Atomic Multi-Step Writes

@Service
public class TransferService {
    private final AccountRepository accounts;
    TransferService(AccountRepository a) { this.accounts = a; }

    @Transactional                                   // both writes COMMIT together or ROLL BACK together
    public void transfer(long from, long to, BigDecimal amount) {
        Account src = accounts.findById(from).orElseThrow();
        Account dst = accounts.findById(to).orElseThrow();
        src.debit(amount);                           // if the SECOND step fails...
        dst.credit(amount);                          // ...the FIRST is rolled back — no money lost
        // no explicit save() needed: managed entities are flushed on commit (dirty checking — JPA lesson)
    }
}
🎯 @Transactional makes a group of operations atomic: they all succeed and commit, or any failure rolls everything back. Without it, a crash between the debit and the credit would lose money — the account is debited but never credited. Put @Transactional on the service method that represents one business operation, not on individual repository calls, so the whole operation is the unit of atomicity. And because managed entities are dirty-checked (JPA lesson), you often don't need an explicit save() — the changes flush on commit.

// readOnly lets Hibernate skip dirty-checking/flush and can hint the DB/driver for read optimizations.
🔍 Transactions deliver the ACID guarantees — the reason databases can be trusted under concurrency and failure. Mark read methods @Transactional(readOnly = true): it tells Hibernate not to bother with dirty-checking (no flush of changes it doesn't expect) and can hint the database for read optimizations. It's both a small performance win and documentation that the method doesn't write.

Part 3 — Gotcha 1: Self-Invocation Bypasses the Proxy

@Service
public class OrderService {
    public void process() {
        placeOrder();               // ❌ direct in-class call — @Transactional IGNORED!
    }
    @Transactional
    public void placeOrder() { ... }
}
// @Transactional works via an AOP PROXY that wraps the bean (Bean Lifecycle / AOP lessons). Calling an
// annotated method from ANOTHER method of the SAME class bypasses the proxy, so no transaction starts.
🚨 The #1 @Transactional gotcha: self-invocation. Spring applies transactions by wrapping your bean in a proxy at startup (Bean Lifecycle lesson); the proxy starts the transaction around incoming calls. When one method calls another method of the same class via this.method(), the call never leaves the object, bypasses the proxy, and the annotation does nothing — placeOrder() runs with no transaction. Fixes: call the annotated method through another bean, self-inject the proxy and call through that reference, or move the method to a separate bean. This same proxy-bypass affects @Cacheable and @Async (Caching lesson) — it's one root cause across all AOP-based annotations.

Part 4 — Gotcha 2: Rollback Rules

@Transactional                              // default: rolls back on UNCHECKED exceptions only
public void doWork() throws IOException {
    repo.save(entity);
    if (bad) throw new IOException("oops");  // ❌ CHECKED exception → does NOT roll back by default!
}
// The save COMMITS despite the exception — surprising and dangerous.

@Transactional(rollbackFor = Exception.class)   // ✅ roll back on ANY exception, including checked
public void doWork() throws IOException { ... }
⚠️ By default, @Transactional rolls back only on unchecked exceptions (RuntimeException and Error) — NOT checked exceptions. So a method that throws a checked IOException after saving will commit the save, which is almost never what you want. This trips up everyone. Fix with @Transactional(rollbackFor = Exception.class) (or a specific checked type) to roll back on checked exceptions too. This connects to the Java Exceptions lesson's "prefer unchecked exceptions" — with unchecked domain exceptions, the default rollback behavior does the right thing.

Part 5 — Propagation & Isolation

// PROPAGATION — how a transactional method behaves when called within an existing transaction:
@Transactional(propagation = Propagation.REQUIRED)     // DEFAULT: join the existing tx, or start one
@Transactional(propagation = Propagation.REQUIRES_NEW) // ALWAYS start a new, independent tx (suspend the outer)
@Transactional(propagation = Propagation.NESTED)       // a savepoint within the outer tx

// ISOLATION — how much concurrent transactions can affect each other:
@Transactional(isolation = Isolation.READ_COMMITTED)   // common default; no dirty reads
Isolation levelPrevents
READ_UNCOMMITTED(nothing — allows dirty reads)
READ_COMMITTEDDirty reads
REPEATABLE_READDirty + non-repeatable reads
SERIALIZABLEAll anomalies (dirty, non-repeatable, phantom) — strictest, slowest
🔍 Propagation controls what happens when a transactional method is called inside another transaction: REQUIRED (the default) joins the caller's transaction; REQUIRES_NEW suspends it and runs in an independent transaction (useful for audit logging that must commit even if the outer rolls back); NESTED uses a savepoint. Isolation trades consistency against concurrency: higher levels prevent more anomalies (dirty reads, non-repeatable reads, phantoms) but reduce throughput and can cause more lock contention/deadlocks. Most applications use the database default (often READ_COMMITTED) and only raise isolation where a specific correctness need justifies the cost.

Part 6 — Deep Dive: Where Transactions Belong & Why the Gotchas Exist

  • Transactions belong at the service layer. A service method is one business operation, so it's the natural unit of atomicity — all its repository writes commit or roll back together. Repository-level transactions can't span a multi-step operation, and controller-level ones mix HTTP with persistence concerns.
  • The gotchas are consequences of the proxy mechanism. Self-invocation fails because the transaction is applied by a proxy around the bean, and an internal call doesn't cross it. Understanding proxies (Bean Lifecycle / AOP lessons) makes both this and the caching/async versions predictable rather than mysterious.
  • Keep transactions short. An open transaction holds a database connection and locks; long transactions (especially with external calls or user think-time inside) cause connection-pool exhaustion and lock contention. Do slow, non-transactional work (calling another service) outside the transaction.
  • Rollback semantics must be deliberate. Know that checked exceptions don't roll back by default, decide rollback behavior explicitly for each service, and prefer unchecked domain exceptions so the default does the right thing.
🔍 Senior framing: transactions are how a service preserves data integrity under partial failure and concurrency, and using @Transactional well is mostly about understanding where it applies, how it's implemented, and the two behaviors that surprise people. Place it on the service method that represents a single business operation so that operation is the unit of atomicity — all its writes commit together or roll back together (the transfer that debits and credits, the order that saves and decrements stock) — because a crash mid-operation must not leave the data half-updated; repositories are too granular and controllers mix in HTTP concerns. Recognize that Spring implements it with an AOP proxy that wraps the bean and starts the transaction around incoming calls, which directly explains the self-invocation gotcha (an in-class this.method() call bypasses the proxy, so the annotation silently does nothing — the same proxy-bypass that affects @Cacheable and @Async, fixed by calling through another bean or a self-injected proxy) and why the framework can add this behavior transparently at all. Know the rollback default — only unchecked exceptions trigger rollback, so a checked exception after a write will surprisingly commit unless you set rollbackFor — which is a strong argument for unchecked domain exceptions so the default is correct. For advanced control, use propagation (REQUIRED to join, REQUIRES_NEW for independent work like audit logs that must persist regardless of the outer outcome) and isolation (raise it only where a specific concurrency anomaly must be prevented, accepting the throughput/locking cost), and keep transactions short — never hold one open across slow external calls or user think-time, since it pins a connection and locks and leads to pool exhaustion and contention. The mature engineer therefore treats the transaction boundary as a deliberate design decision per business operation, reasons about failure and concurrency explicitly, and understands the proxy mechanism well enough that the self-invocation and rollback behaviors are expected rather than bugs discovered in production. Interviewers press hard on @Transactional — placement, self-invocation, rollback rules, propagation, isolation — precisely because these reveal whether a candidate understands transactional integrity and the proxy-based mechanics beneath Spring's declarative convenience, which is exactly what separates data access that stays consistent under load and failure from data access that quietly corrupts state.

Interview Questions

Q1 (beginner): What does @Transactional do, and where should it go?

It makes the annotated method run within a database transaction, so all its operations are atomic — they all commit together or, on failure, all roll back together — giving the ACID guarantees. It should go on the service method that represents a single business operation (for example a transfer that debits one account and credits another), so the whole operation is the unit of atomicity: if any step fails, none of the writes persist, preventing half-completed, inconsistent state. It shouldn't go on individual repository methods (too granular to span a multi-step operation) or on controllers (which should stay focused on HTTP). Because JPA dirty-checks managed entities, you often don't even need an explicit save() — changes flush on commit.

Q2 (beginner): What are the ACID properties?

Atomicity — a transaction's operations are all-or-nothing; either every operation commits or, on any failure, all are rolled back. Consistency — the transaction moves the database from one valid state to another, preserving constraints and invariants. Isolation — concurrent transactions don't interfere in ways that corrupt each other; the isolation level controls how much they can affect one another. Durability — once committed, the data survives crashes (it's persisted, typically to a write-ahead log). These four guarantees are why transactional databases can be trusted under concurrency and failure, and @Transactional is how you get them for a unit of work in Spring.

Q3 (mid): Explain the self-invocation problem with @Transactional.

Spring implements @Transactional using an AOP proxy: at startup it wraps the bean in a proxy that starts (and commits/rolls back) the transaction around calls that come through the proxy — i.e. calls from other beans. When a method of the bean calls another method of the same bean directly via this.method(), the call never leaves the target object, so it doesn't pass through the proxy and the transactional behavior is never applied — the annotated method runs with no transaction, silently. This is a common, confusing bug because the annotation is present but ineffective. The fixes are to invoke the annotated method from a different bean (so the call crosses the proxy), to self-inject the bean's own proxy and call through that reference, or to move the transactional method into a separate bean. The same proxy-bypass affects other AOP-based annotations like @Cacheable and @Async, so it's really one root cause to understand: proxies only intercept external calls.

Q4 (mid): What are the default rollback rules, and why do they surprise people?

By default, @Transactional rolls back only on unchecked exceptions — RuntimeException and Error — and not on checked exceptions. So if a method performs a write and then throws a checked exception (like IOException), the transaction commits the write despite the failure, which is almost always the opposite of what you want and a genuinely dangerous surprise. The rationale is that checked exceptions were historically treated as recoverable conditions rather than fatal errors, but in practice it means you must be deliberate. The fix is to specify @Transactional(rollbackFor = Exception.class) (or a specific checked type) to roll back on checked exceptions too, or — better aligned with modern practice — to use unchecked domain exceptions so the default behavior rolls back correctly. This ties to the guidance to prefer unchecked exceptions: with unchecked domain exceptions, "throw to abort the transaction" just works.

Q5 (senior): Explain transaction propagation and give a real use for REQUIRES_NEW.

Propagation defines how a transactional method behaves with respect to any transaction already in progress when it's called. The default, REQUIRED, joins the existing transaction if there is one (so the inner method's work is part of the same atomic unit and rolls back with it) or starts a new one if there isn't — this is what you want most of the time, giving a single transaction spanning a business operation across service calls. REQUIRES_NEW always suspends any current transaction and runs in a brand-new, independent transaction that commits or rolls back on its own, resuming the outer transaction afterward. NESTED creates a savepoint within the current transaction so the inner work can roll back to that point without aborting the whole outer transaction (database support permitting), and there are others (SUPPORTS, MANDATORY, NOT_SUPPORTED, NEVER). A canonical use of REQUIRES_NEW is audit or event logging that must persist regardless of the outer operation's outcome: you want to record "an attempt to do X was made" even if X ultimately fails and its transaction rolls back, so you write the audit record in a REQUIRES_NEW transaction that commits independently — the outer rollback undoes the business changes but the audit entry survives. Another is isolating a sub-operation whose failure shouldn't doom the parent (or whose success shouldn't be undone by the parent's failure). The caveats are important: REQUIRES_NEW uses a second database connection while the outer transaction is suspended, so overusing it can exhaust the connection pool and it can interact subtly with locks (the two transactions can even contend), and it breaks the all-or-nothing intuition, so it should be used deliberately for cases that genuinely need independent commit semantics rather than as a workaround. Understanding propagation lets you compose service methods into the transaction structure the business actually needs — one atomic unit for the main flow, with specific sub-operations promoted to independent transactions where their durability must not depend on the parent.

Q6 (senior): How do isolation levels trade correctness against performance, and how do you choose one?

Isolation levels control which concurrency anomalies the database prevents, and each stronger level prevents more at the cost of throughput and increased locking/contention. The anomalies, from least to most subtle: a dirty read (reading another transaction's uncommitted changes, which may be rolled back), a non-repeatable read (re-reading a row within the same transaction and getting a different value because another transaction updated and committed it in between), and a phantom read (re-running a range query and finding new rows another transaction inserted). READ_UNCOMMITTED prevents nothing (allows dirty reads) and is almost never appropriate; READ_COMMITTED prevents dirty reads and is the common default (each statement sees only committed data); REPEATABLE_READ additionally guarantees rows read once stay stable within the transaction (preventing non-repeatable reads, and in some databases phantoms too); SERIALIZABLE prevents all anomalies by making transactions behave as if run one at a time, the strictest and slowest, typically via extensive locking or serialization checks that reduce concurrency and can cause more blocking and deadlocks/serialization failures requiring retries. Choosing: start with the database's default (usually READ_COMMITTED), because most application logic is correct under it and it offers good concurrency, and raise the level only for a specific operation where a concrete anomaly would cause a correctness bug — for example, a calculation that reads the same data multiple times and must see a consistent snapshot might need REPEATABLE_READ, and a critical invariant that must hold against concurrent inserts might need SERIALIZABLE — scoping the higher level narrowly (per transaction, not globally) so you pay the performance cost only where it's needed. Often a better alternative to a globally high isolation level is optimistic locking (a @Version column that detects concurrent modification and fails the second writer, who retries) or explicit pessimistic locks on the specific rows, which target the exact contention point without serializing everything. The senior approach is to identify the actual concurrency hazard for a given operation, choose the minimum mechanism (default isolation, optimistic versioning, a targeted lock, or a raised isolation level) that prevents it, keep transactions short to limit the window of contention, and be prepared to handle serialization/deadlock failures with retries — trading isolation strength for throughput deliberately and locally rather than reaching for SERIALIZABLE everywhere, which would cripple concurrency, or leaving everything at the default when a real anomaly exists.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Atomic transfer. Write a @Transactional service method that debits one account and credits another; make the second step throw and confirm the first is rolled back.
Solution

The method loads both accounts, debits src, credits dst; throwing a RuntimeException after the debit rolls back the whole transaction, so src's balance is unchanged — atomicity in action. Without @Transactional, the debit would persist and money would vanish.

2 (easy): read-only. Mark a list/query method @Transactional(readOnly = true) and explain the benefit.
Solution

readOnly=true tells Hibernate the method won't write, so it can skip dirty-checking and flushing and hint the database/driver for read optimizations — a small performance win and clear documentation that the method is a pure read. Use it on query methods.

3 (medium): Reproduce self-invocation. Make a public method call an @Transactional method in the same class and show (via logging or behavior) that no transaction runs; then fix it.
Solution

Calling this.placeOrder() from another method of the same bean bypasses the proxy, so @Transactional is ignored — a rollback that should happen doesn't, or transaction logging shows none. Fix: move placeOrder to another bean and call it, or self-inject the proxy and call through that reference. This is the proxy-bypass root cause.

4 (medium): Rollback on a checked exception. Have a @Transactional method save then throw a checked exception; observe the save commits, then fix it with rollbackFor.
Solution

By default, throwing a checked IOException after repo.save(...) commits the save — the default only rolls back on unchecked exceptions. Adding @Transactional(rollbackFor = Exception.class) makes it roll back on the checked exception too. (Or throw an unchecked domain exception, which rolls back by default.)

5 (hard): REQUIRES_NEW for audit. Design an audit log that records an attempt even when the main business transaction rolls back, using propagation. Explain the trade-off.
Solution

Write the audit record in a method annotated @Transactional(propagation = REQUIRES_NEW), called from the main service. It runs in an independent transaction that commits on its own, so even if the outer business transaction rolls back, the audit entry persists. Trade-off: it uses a second connection while the outer is suspended (pool pressure) and can contend on locks, so use it deliberately for cases that genuinely need independent durability.

6 (hard): Explain-it test. Out loud: (a) what @Transactional does and where it goes; (b) the self-invocation and rollback-default gotchas and their causes; (c) propagation/isolation and choosing them.
Self-check rubric

Pass (a) if you said atomic all-or-nothing on the service method (the business operation). Pass (b) if you explained self-invocation bypasses the proxy and only unchecked exceptions roll back by default, both consequences of the proxy mechanism. Pass (c) if you described REQUIRED vs REQUIRES_NEW and using the default isolation, raising it only for a specific anomaly. This completes the data layer.

🎯 Key Takeaways

  • @Transactional makes a group of writes atomic (ACID) — put it on the service method that is one business operation.
  • Mark read methods readOnly = true; managed entities flush on commit (dirty checking), so explicit save() is often unneeded.
  • Self-invocation bypasses the proxy — a @Transactional method called from the same class runs with no transaction (fix: another bean / self-injected proxy).
  • Default rollback is on unchecked exceptions only — checked exceptions commit unless you set rollbackFor (prefer unchecked domain exceptions).
  • Propagation (REQUIRED joins, REQUIRES_NEW is independent) and isolation (raise only for a specific anomaly) give advanced control.
  • Keep transactions short — never hold one across slow external calls; they pin a connection and hold locks.
