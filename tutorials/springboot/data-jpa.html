Spring Data JPA Intermediate

Spring Data JPA sits on top of Hibernate and removes almost all data-access boilerplate: you declare an interface, and Spring generates the implementation. The power is real, but so are the footguns — the N+1 problem, lazy loading, and transaction boundaries trip up developers who treat it as magic. This lesson makes the machinery visible.

Entity → Repository → Query

@Entity
@Table(name = "users")
public class User {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    @Column(nullable = false, unique = true)
    private String email;
    private String name;
    @Enumerated(EnumType.STRING) private Role role;   // store the NAME, not the ordinal
    // getters/setters (or use a builder)
}

public interface UserRepository extends JpaRepository<User, Long> {
    // Method name → generated query. No implementation needed:
    Optional<User> findByEmail(String email);
    List<User> findByRoleAndActiveTrue(Role role);
    List<User> findByNameContainingIgnoreCase(String q);
    long countByRole(Role role);
}

JpaRepository already gives you save, findById, findAll, delete, paging, and sorting for free. Derived query methods add custom lookups just by naming them.

Custom Queries with @Query

public interface OrderRepository extends JpaRepository<Order, Long> {
    @Query("SELECT o FROM Order o WHERE o.total > :min ORDER BY o.total DESC")
    List<Order> findExpensive(@Param("min") BigDecimal min);   // JPQL (entity-based)

    @Query(value = "SELECT * FROM orders WHERE created_at > now() - interval '7 days'",
           nativeQuery = true)                                  // raw SQL when JPQL can't express it
    List<Order> recentNative();

    @Modifying @Transactional
    @Query("UPDATE Order o SET o.status = 'CANCELLED' WHERE o.id = :id")
    int cancel(@Param("id") Long id);
}

The N+1 Problem — the Interview Classic

@Entity class Order {
    @ManyToOne(fetch = FetchType.LAZY) private User customer;   // lazy by default for *ToMany
    @OneToMany(mappedBy = "order")     private List<LineItem> items;
}
// This loop fires 1 query for the orders, then 1 MORE per order to load each customer → N+1:
for (Order o : orderRepo.findAll())
    System.out.println(o.getCustomer().getName());   // lazy load per iteration!

// FIX: fetch the association up front in ONE query.
@Query("SELECT o FROM Order o JOIN FETCH o.customer")
List<Order> findAllWithCustomer();
// ...or an @EntityGraph:
@EntityGraph(attributePaths = "customer")
List<Order> findAll();
🚨 N+1 is the most common Spring performance bug. Symptom: a list endpoint that fires hundreds of tiny queries. Cause: accessing a lazy association inside a loop. Fix: JOIN FETCH or @EntityGraph to load what you need in one query. Turn on spring.jpa.show-sql=true in dev to see the queries — you cannot fix what you can't see.

Transactions

@Service
public class TransferService {
    private final AccountRepository accounts;
    TransferService(AccountRepository a) { this.accounts = a; }

    @Transactional                                   // both writes commit together or roll back together
    public void transfer(long from, long to, BigDecimal amount) {
        Account src = accounts.findById(from).orElseThrow();
        Account dst = accounts.findById(to).orElseThrow();
        src.debit(amount);
        dst.credit(amount);
        // no explicit save() needed: managed entities are flushed on commit (dirty checking)
    }
}
// @Transactional on the SERVICE, not the repository — the whole business op is one unit.
// Default rollback is on unchecked (RuntimeException) only. Note: self-invocation of a
// @Transactional method from the same class BYPASSES the proxy — the annotation is ignored.

Interview Questions

Q1: What is the N+1 problem and how do you fix it?

N+1 happens when loading a list of N entities fires 1 query for the list plus 1 additional query per entity to lazily load an association — N+1 total. It surfaces as a list endpoint issuing hundreds of small queries. Fixes: eagerly fetch the needed association in a single query with JOIN FETCH in JPQL or an @EntityGraph; for read models, a projection/DTO query that selects only required columns; and batching (hibernate.default_batch_fetch_size) to turn N into N/batch. You diagnose it by logging SQL (show-sql / a query counter in tests). Avoid the opposite trap of making everything EAGER, which causes over-fetching.

Q2: Where should @Transactional go, and what are its gotchas?

Put it on the service method that represents one business operation, so all its repository calls commit or roll back atomically — not on individual repository methods. Gotchas: (1) it works via a Spring proxy, so self-invocation (calling one @Transactional method from another method in the same class) bypasses the proxy and the annotation is ignored; (2) by default rollback triggers on unchecked exceptions only — checked exceptions commit unless you set rollbackFor; (3) the transaction spans the method, so lazy associations are accessible inside it but not after it returns — map to DTOs before the boundary.

Q3: LAZY vs EAGER fetching — which and why?

@ManyToOne/@OneToOne default to EAGER, collections to LAZY. The production default should be LAZY everywhere, then fetch exactly what a given use case needs via JOIN FETCH/@EntityGraph. EAGER associations over-fetch on every load (you pay for data you may not use) and can cascade into large joins or their own N+1; they also can't be turned off per query. LAZY defers the load until access, but accessing it outside the transaction throws LazyInitializationException — which is why you resolve associations inside the service transaction and return DTOs.

🏋️ Practice: model Order and Customer with a lazy @ManyToOne; write a list endpoint that triggers N+1, confirm it with show-sql, then fix it with JOIN FETCH and re-count the queries; add a derived query (findByEmail) and a @Query JPQL method; make a two-step write @Transactional and prove it rolls back on an exception; page a large list.

🎯 Key Takeaways

  • Extend JpaRepository — CRUD, paging, and derived queries come from method names, no impl needed.
  • Use @Query (JPQL or native) for anything derivation can't express.
  • N+1 is the classic bug — access lazy associations in a loop → hundreds of queries; fix with JOIN FETCH/@EntityGraph.
  • @Transactional belongs on service methods; watch self-invocation and rollback rules.
  • Default fetching to LAZY, map to DTOs inside the transaction, and page large result sets.
