Error Handling Web Layer

What you'll master in this lesson: how to turn exceptions into clean, consistent HTTP responses with @RestControllerAdvice — one place that owns error translation; how to design domain exceptions and map them to the right status; the modern ProblemDetail (RFC 7807) standard error format; and — critically — how to never leak a stack trace, which is both a professional and a security requirement. This is the "never leak internals" half of robust APIs; together with the validation lesson it makes every error your API returns a well-formed, typed JSON object.

Part 1 — The Problem: Unhandled Exceptions Leak Internals

@GetMapping("/{id}")
public UserDto get(@PathVariable long id) {
    return service.find(id).orElseThrow(() -> new NotFoundException(id));
}
// Without a handler, an unhandled exception yields Spring's DEFAULT error response — which can
// include a stack trace, exception class, and internal paths. That's ugly for clients and a
// SECURITY risk (it reveals framework versions, file paths, SQL — reconnaissance for an attacker).

The goal of error handling is two-fold: give clients a consistent, actionable, typed error body with the correct status code, and never expose internal details (stack traces, class names, SQL, paths). Both are achieved by centralizing exception-to-response translation in one place.

Part 2 — Global Error Handling with @RestControllerAdvice

public record ApiError(String code, String message, Map<String,String> fields) {}

@RestControllerAdvice                       // ONE place translating exceptions → clean JSON, app-wide
class GlobalExceptionHandler {

    @ExceptionHandler(NotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)                       // 404
    ApiError notFound(NotFoundException e) {
        return new ApiError("NOT_FOUND", e.getMessage(), Map.of());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)    // @Valid body failures → 400
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    ApiError invalidBody(MethodArgumentNotValidException e) {
        var fields = e.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage, (a,b)->a));
        return new ApiError("VALIDATION_FAILED", "Invalid request", fields);
    }

    @ExceptionHandler(ConstraintViolationException.class)      // @Validated param failures → 400
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    ApiError invalidParams(ConstraintViolationException e) { /* map violations → fields */ }

    @ExceptionHandler(Exception.class)                          // CATCH-ALL → 500, NO stack leak
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    ApiError unexpected(Exception e) {
        log.error("Unhandled exception", e);                    // LOG the detail server-side
        return new ApiError("INTERNAL", "Something went wrong", Map.of());  // GENERIC to the client
    }
}
🎯 @RestControllerAdvice centralizes error translation across every controller. Each @ExceptionHandler method maps a specific exception type to a response with a consistent body and correct status. This is far better than try/catch in every controller: it's DRY, guarantees consistency, and — via the catch-all — ensures no exception ever escapes as a raw stack trace. Note you must handle both validation exceptions (MethodArgumentNotValidException for bodies and ConstraintViolationException for params — the Validation lesson) to give uniform 400s.

Part 3 — The Catch-All Is a Security Control

@ExceptionHandler(Exception.class)
@ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
ApiError unexpected(Exception e) {
    log.error("Unhandled exception on request", e);   // full detail → server logs (for you)
    return new ApiError("INTERNAL", "Something went wrong", Map.of());   // generic → client
}
🔒 Log the real exception server-side; return a generic message to the client. Leaking a stack trace exposes framework and library versions (which have known CVEs), file-system paths, internal class names, and sometimes SQL or data — all reconnaissance for an attacker, and unprofessional besides. The catch-all handler is the safety net that guarantees this never happens accidentally: every unexpected exception becomes a clean 500 with a generic body, while the diagnostic detail goes to your logs where only you can see it. This "log detail, return generic" pattern is a core secure-coding practice (Java Exceptions lesson).

Part 4 — Custom Domain Exceptions

// Domain exceptions carry meaning and structured data; usually UNCHECKED (Java Exceptions lesson):
public class NotFoundException extends RuntimeException {
    private final long id;
    public NotFoundException(long id) { super("Not found: " + id); this.id = id; }
    public long getId() { return id; }
}
public class InsufficientFundsException extends RuntimeException {
    private final long shortfallCents;
    public InsufficientFundsException(long shortfall) { super("short by " + shortfall); this.shortfallCents = shortfall; }
    public long getShortfallCents() { return shortfallCents; }
}
// Services throw these; the advice maps each to a status (404, 409/422). This keeps HTTP status
// codes OUT of the service layer — the service speaks domain, the advice speaks HTTP.
🔍 Throw meaningful domain exceptions in the service and map them to HTTP in the advice. This separates concerns: the service layer expresses what went wrong in domain terms (not found, insufficient funds, conflict) without knowing about HTTP; the advice owns the HTTP translation. Domain exceptions can carry structured data (an id, a shortfall) that the handler includes in the response. This is the same "business rules throw domain exceptions, mapped to status codes" idea from the Validation lesson.

Part 5 — ProblemDetail (RFC 7807), the Modern Standard

// Spring 6 / Boot 3 ships a STANDARD error format: application/problem+json (RFC 7807).
@ExceptionHandler(NotFoundException.class)
ProblemDetail handleNotFound(NotFoundException e) {
    var pd = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, e.getMessage());
    pd.setTitle("Resource not found");
    pd.setType(URI.create("https://api.example.com/errors/not-found"));
    pd.setProperty("resourceId", e.getId());        // custom fields
    return pd;                                       // → a documented, interoperable error shape
}
// You can also extend ResponseEntityExceptionHandler, which already returns ProblemDetail for
// Spring's built-in exceptions (validation, message-not-readable, method-not-supported, ...).
🎯 ProblemDetail is a standardized error format (type, title, status, detail, instance, plus custom properties) that clients and tools understand across APIs — preferable to inventing your own ad-hoc error JSON. In Boot 3 it's built in, and extending ResponseEntityExceptionHandler gives you ProblemDetail responses for the framework's own exceptions for free. Adopting a standard error contract makes your API more interoperable and your errors self-describing — a mark of a well-designed modern API.

Part 6 — Deep Dive: Consistent, Secure Errors as a Contract

  • Errors are part of the API contract. Clients need to reliably parse failures — a consistent error shape (code, message, fields) with correct status codes lets them handle errors programmatically. Inconsistent or leaky errors force clients into brittle string-matching.
  • Centralize, don't scatter. One @RestControllerAdvice owns error translation, so every endpoint's errors are consistent and the catch-all guarantees nothing leaks. Per-controller try/catch duplicates logic and inevitably misses cases.
  • Handle at the right layer. Services throw domain exceptions (what went wrong); the advice, at the top of the request, translates them to HTTP (how to respond). This mirrors the Java Exceptions lesson's "handle at the level that can act" — the HTTP boundary is where an error becomes a response.
  • Security by default. The catch-all's "log detail, return generic" ensures internal information never reaches clients, turning a common leak into an impossible one.
🔍 Senior framing: error handling is where an application demonstrates whether it treats its API as a contract and its internals as private. A well-built service centralizes exception-to-response translation in a single @RestControllerAdvice so that every error — validation failures (both body and parameter variants), missing resources, conflicts, and truly unexpected faults — is mapped to a consistent, typed JSON body with an accurate status code, and a catch-all handler guarantees that no exception ever escapes as a raw stack trace, logging the real detail server-side while returning a generic message to the client. This combination delivers three things at once: a stable, parseable error contract that clients can rely on (rather than brittle string-matching against ad-hoc messages); clean separation of concerns, with services throwing meaningful domain exceptions that carry structured data and know nothing about HTTP, and the advice owning the HTTP translation at the request boundary; and security, since leaking framework versions, paths, class names, or SQL is reconnaissance for attackers and the "log detail, return generic" catch-all makes that leak structurally impossible. Modern Spring pushes this further with ProblemDetail (RFC 7807), a standardized, interoperable error format that makes your errors self-describing across tools and APIs. The mature engineer therefore designs the error contract as deliberately as the success contract — consistent shapes, correct statuses, domain exceptions mapped centrally, no internal leakage, and a standard format — because how a system fails is as much a part of its quality as how it succeeds, and interviewers probe error handling precisely because a strong answer (centralized advice, the body-vs-param validation exceptions, domain exceptions mapped to status, and the catch-all as a security control) reveals someone who builds APIs that are robust and safe under failure, not just correct on the happy path.

Interview Questions

Q1 (beginner): How do you handle exceptions globally in Spring Boot?

With @RestControllerAdvice: a single class containing @ExceptionHandler methods, each mapping a specific exception type to a response with a consistent body and the correct status code (via @ResponseStatus or by returning a ResponseEntity/ ProblemDetail). Because the advice applies across all controllers, every endpoint's errors are handled uniformly in one place, and a catch-all handler for Exception ensures no exception escapes as a raw stack trace. This is far better than try/catch in each controller, which duplicates logic and inevitably misses cases.

Q2 (beginner): Why shouldn't you return stack traces to clients?

Two reasons. Professionally, a stack trace is not an actionable error for a client — it's noise that clients can't program against, so errors should be clean, typed, and consistent instead. More importantly, it's a security risk: a leaked stack trace exposes framework and library versions (which map to known vulnerabilities), file-system paths, internal class and package names, and sometimes SQL statements or data — all reconnaissance an attacker can use to target the system. The correct pattern is to log the full detail server-side (where only you see it) and return a generic message to the client, which a catch-all handler in the advice enforces for every unexpected exception.

Q3 (mid): How do you design domain exceptions and map them to HTTP statuses?

Define custom, usually unchecked, exception classes that express what went wrong in domain terms — NotFoundException, InsufficientFundsException, DuplicateEmailException — and let them carry structured data (an id, a shortfall amount) that a handler can include in the response. Services throw these when a business condition fails, without referencing HTTP status codes, keeping the service layer free of web concerns. Then in the @RestControllerAdvice, an @ExceptionHandler per domain exception maps it to the appropriate status (404 for not found, 409 or 422 for conflicts/unprocessable, 400 for bad input) and builds the error body, possibly using the exception's data. This separation means the service speaks domain and the advice speaks HTTP, so business logic is decoupled from the web layer and the HTTP mapping lives in one consistent place.

Q4 (mid): What is ProblemDetail and why use it?

ProblemDetail is Spring 6 / Boot 3's implementation of RFC 7807, a standardized error format (media type application/problem+json) with well-known fields — type (a URI identifying the error kind), title, status, detail, instance — plus custom properties you can add. Using it instead of a hand-rolled error JSON makes your API's errors self-describing and interoperable: clients, gateways, and tooling that understand RFC 7807 can parse and act on your errors uniformly, and you don't reinvent an error schema. In practice you build a ProblemDetail in your exception handlers, and extending ResponseEntityExceptionHandler gives you ProblemDetail responses for Spring's built-in exceptions (validation, unreadable message, method not supported, etc.) automatically. It's the modern, recommended way to represent errors in a Spring Boot API.

Q5 (senior): Contrast centralized exception handling with per-controller try/catch, and explain why the catch-all matters.

Centralized handling via @RestControllerAdvice puts all exception-to-response translation in one place: each @ExceptionHandler maps an exception type to a consistent body and status, applied across every controller. Per-controller try/catch, by contrast, duplicates the same mapping logic in every endpoint, is easy to forget or apply inconsistently (so some paths leak stack traces while others return different error shapes), tangles HTTP error concerns into business flow, and is a maintenance burden as the mapping evolves. The advice pattern is DRY, guarantees a uniform error contract, keeps controllers and services focused on their real work, and makes the error format a single, changeable thing. The catch-all handler for Exception is the load-bearing piece: without it, any exception type you didn't explicitly handle falls through to Spring's default error response, which can leak internals — so there's always a risk that a new, unanticipated exception exposes a stack trace. The catch-all closes that gap absolutely: every exception, anticipated or not, is caught, its detail logged server-side, and a generic 500 returned, so information disclosure becomes structurally impossible rather than a matter of remembering to handle each case. It also gives you a place to add cross-cutting behavior (correlation IDs in the response, alerting on unexpected errors). So centralized handling delivers consistency and separation of concerns, and the catch-all specifically delivers the security guarantee that nothing leaks — together they turn error handling from a scattered, leak-prone afterthought into a reliable part of the API contract.

Q6 (senior): How should error handling interact with logging, observability, and the different error categories, in production?

Error handling should distinguish error categories and treat each appropriately for both the client and operations. Client errors (4xx) — validation failures, not-found, conflicts, unauthorized — are the client's fault and expected, so they should return a clear, typed body with the right status and generally be logged at a low level (info/debug) or not at all beyond metrics, because they're normal traffic; logging every 400 at error level creates noise that hides real problems. Server errors (5xx) — unexpected exceptions caught by the catch-all — indicate a bug or outage and must be logged at error level with the full stack trace and enough context (the request path, a correlation/trace ID, sanitized parameters) to diagnose them, and typically should trigger alerts. Crucially, the client sees a generic message while the logs get the detail, so you never trade security for diagnosability. For observability, propagate a correlation/trace ID (from the tracing setup in the Actuator/observability lesson) and include it both in the logged error and in the error response body, so a user reporting an error id lets you find the exact failure in the logs — connecting the client-facing error to the server-side trace. Emit metrics on error rates by status/type so you can alert on spikes and track SLOs, and structure logs as JSON so they're queryable. Handlers should also avoid logging the same exception multiple times (log once, at the boundary where it's translated) and must be careful never to log sensitive data (passwords, tokens, PII) even server-side. Finally, the advice is a good place to enforce these policies uniformly: map each exception category to a status and a log level, attach the trace id, record a metric, and return the standardized (ideally ProblemDetail) body. The senior view is that error handling is not just about the response the client sees but about the entire failure-observability loop — correct statuses and safe bodies for clients, appropriately-leveled and correlated logs plus metrics and alerts for operators — so that failures are both handled gracefully and diagnosable, without leaking internals or drowning real incidents in noise.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Global 404. Write a NotFoundException and a @RestControllerAdvice handler mapping it to a 404 with a typed body. Trigger it and inspect the response.
Solution

An unchecked NotFoundException, and an @ExceptionHandler(NotFoundException.class) with @ResponseStatus(NOT_FOUND) returning an ApiError. A GET for a missing id returns a clean 404 with your error shape instead of a stack trace. The service just throws the domain exception; the advice owns the HTTP mapping.

2 (easy): No leak. Add a catch-all @ExceptionHandler(Exception.class) that logs the exception and returns a generic 500. Throw a RuntimeException and confirm no stack trace reaches the client.
Solution

The catch-all logs e server-side (log.error("Unhandled", e)) and returns new ApiError("INTERNAL", "Something went wrong", ...) with status 500. The client sees only the generic body; the stack trace is in your logs. This is the security-critical "log detail, return generic" pattern.

3 (medium): Consistent validation errors. Handle both MethodArgumentNotValidException (body) and ConstraintViolationException (params) so all validation failures return the same 400 shape.
Solution

Two @ExceptionHandler methods, both returning a 400 with a per-field map — one extracting getBindingResult().getFieldErrors(), the other extracting the ConstraintViolations. Now a bad body and a bad query param both produce the same error shape, instead of the param failure leaking the default format (the Validation lesson's two-exception point).

4 (medium): Domain exception → status. Add an InsufficientFundsException carrying a shortfall, throw it in a service, and map it to a 409 (or 422) including the shortfall in the body.
Solution

The service throws new InsufficientFundsException(shortfall) (no HTTP knowledge); an @ExceptionHandler maps it to 409/422 and puts e.getShortfallCents() in the response. The status and structured data communicate the failure precisely, and HTTP concerns stay out of the service.

5 (hard): ProblemDetail. Convert one handler to return a ProblemDetail with a type URI, title, and a custom property. Explain the interoperability benefit.
Solution

ProblemDetail.forStatusAndDetail(NOT_FOUND, msg) with setTitle, setType(URI...), and setProperty("resourceId", id), returned from the handler → application/problem+json. Benefit: it's the RFC 7807 standard, so clients, gateways, and tooling can parse it uniformly across APIs without learning your custom error schema — self-describing, interoperable errors.

6 (hard): Explain-it test. Out loud: (a) why centralize error handling and how @RestControllerAdvice works; (b) why the catch-all is a security control; (c) domain exceptions vs HTTP mapping and what ProblemDetail adds.
Self-check rubric

Pass (a) if you said one place maps exceptions to consistent typed responses, DRY and uniform. Pass (b) if you said it logs detail server-side and returns a generic body so stack traces never leak. Pass (c) if you said services throw domain exceptions (no HTTP), the advice maps to status, and ProblemDetail is the RFC 7807 standard error format. This completes the web-layer error story.

🎯 Key Takeaways

  • Centralize error translation in one @RestControllerAdvice — consistent, typed JSON errors with correct statuses across every controller.
  • Handle both validation exceptions (MethodArgumentNotValidException and ConstraintViolationException) for uniform 400s.
  • The catch-all handler is a security control: log detail server-side, return a generic message — stack traces never leak.
  • Throw meaningful domain exceptions in services (no HTTP knowledge); map them to statuses in the advice — separation of concerns.
  • Prefer ProblemDetail (RFC 7807) for a standardized, interoperable error format.
  • Errors are part of the API contract — design them as deliberately as success responses.
