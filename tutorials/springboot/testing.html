Testing Spring Boot Production

What you'll master in this lesson: how to test a Spring Boot app at every level of the pyramid — plain unit tests with mocks (no Spring at all), slice tests (@WebMvcTest, @DataJpaTest) that boot only the layer under test, and full @SpringBootTest integration tests — plus the two things that separate a trustworthy suite from a slow, flaky one: using the narrowest slice that proves what you need, and testing against the real database engine with Testcontainers instead of H2. Prerequisites: the whole track so far (you'll test controllers from the Web Layer, repositories from the Data Layer) and Java Testing (JUnit/Mockito).

Part 1 — The Testing Pyramid, Spring Edition

LevelAnnotationLoadsSpeedUse for
Unitnone (JUnit + Mockito)nothing Springms — thousandsService/business logic in isolation
Web slice@WebMvcTestControllers + MVC onlyfastRouting, binding, validation, JSON, status codes
Data slice@DataJpaTestJPA + a test DBmediumRepositories, queries, mappings
Integration@SpringBootTestWhole contextslow — fewEnd-to-end wiring, real HTTP
🎯 The pyramid says: many fast tests at the bottom, few slow ones at the top. Most of your tests should be plain unit tests that new a service with mocked collaborators and run in milliseconds — no Spring context to boot. Above them, thin slices boot only the layer they test. At the very top, a small number of @SpringBootTest integration tests verify the pieces wire together for real. Inverting the pyramid — @SpringBootTest for everything — gives a suite that takes minutes, is painful to run on every change, and tests too much at once to localize failures.

Part 2 — Unit Test: No Spring at All (Fastest)

class OrderServiceTest {
    @Test void appliesDiscount() {
        PaymentGateway gateway = mock(PaymentGateway.class);       // Mockito — a fake collaborator
        OrderRepository repo   = mock(OrderRepository.class);
        var service = new OrderService(gateway, repo);             // constructor injection pays off HERE

        when(repo.findById(1L)).thenReturn(Optional.of(new Order(...)));   // arrange
        var result = service.total(1L);                            // act

        assertThat(result).isEqualByComparingTo("90.00");          // assert (AssertJ)
        verify(gateway).charge(any());                             // and assert the interaction happened
    }
}
🎯 This is why constructor injection matters (Core Container lesson): because the service takes its dependencies as constructor parameters, you can new it directly with mocks — no Spring context, no @Autowired, no @SpringBootTest. These tests run in milliseconds, so you can have thousands of them, and they isolate a single unit's logic so a failure points straight at the bug. Field injection would make this awkward (you'd need reflection or a context). The pattern: arrange (create mocks, stub with when(...).thenReturn(...)), act (call the method), assert (check the return value and, where behavior matters, verify the interaction with verify(...)). Most of your suite should look like this.

Part 3 — Web Slice: @WebMvcTest + MockMvc

@WebMvcTest(UserController.class)          // loads ONLY the web layer for this one controller — no DB, no services
class UserControllerTest {
    @Autowired MockMvc mvc;                // simulates HTTP requests against the controller (no real server/port)
    @MockBean UserService service;         // the collaborator is a mock in the context, not the real bean

    @Test void returns404WhenMissing() throws Exception {
        when(service.find(99L)).thenReturn(Optional.empty());
        mvc.perform(get("/api/users/99"))
           .andExpect(status().isNotFound());              // proves the not-found mapping (Error Handling lesson)
    }

    @Test void rejectsInvalidBody() throws Exception {
        mvc.perform(post("/api/users").contentType(APPLICATION_JSON)
              .content("""{"name":"","email":"bad"}"""))
           .andExpect(status().isBadRequest())             // proves @Valid + the validation advice fire (Validation lesson)
           .andExpect(jsonPath("$.errors").exists());
    }
}
🔍 @WebMvcTest boots the MVC machinery and the named controller and nothing else — no database, no service beans (you mock those with @MockBean). It's the right level to test everything the web layer owns: URL routing, request binding, @Valid validation and the resulting 400s, your @ExceptionHandler/@ControllerAdvice mappings (404/409/etc.), status codes, and JSON serialization — all through MockMvc, which drives the controller without starting a real server or opening a port. Because the service is mocked, the test doesn't depend on business logic or a DB, so it's fast and focused on web correctness.

Part 4 — Data Slice: @DataJpaTest (and use the real engine)

@DataJpaTest                               // configures JPA + a test DB; each test runs in a transaction, ROLLED BACK
class UserRepositoryTest {
    @Autowired UserRepository repo;

    @Test void findByEmailWorks() {
        repo.save(new User("a@x.com", "Ann"));
        assertThat(repo.findByEmail("a@x.com")).isPresent();   // exercises the derived query (Data JPA lesson)
    }
}
🐳 Test repositories against the real database engine, not H2. An in-memory H2 approximates SQL but differs from Postgres/MySQL in types, functions, constraint behavior, and dialect quirks — a query, cascade, or Flyway migration that passes on H2 can fail in production, giving false confidence. Testcontainers launches the actual database in a Docker container for the test run, so your data-layer tests run against the same engine you deploy. It costs a little startup time and eliminates the entire "works on H2, breaks on Postgres" class of bug — it's the modern standard.
@SpringBootTest                            // FULL context — this is a true end-to-end integration test
@Testcontainers
class OrderFlowIT {
    @Container static PostgreSQLContainer<?> db = new PostgreSQLContainer<>("postgres:16");   // real Postgres in Docker

    @DynamicPropertySource static void props(DynamicPropertyRegistry r) {
        r.add("spring.datasource.url", db::getJdbcUrl);        // point Spring at the container at runtime
        r.add("spring.datasource.username", db::getUsername);
        r.add("spring.datasource.password", db::getPassword);
    }

    @Autowired TestRestTemplate rest;                          // makes REAL HTTP calls against the running app
    @Test void createThenFetch() { /* POST a user, then GET it, assert the round-trip through every layer */ }
}
🔍 @SpringBootTest loads the whole application context and, with webEnvironment = RANDOM_PORT, starts a real server so TestRestTemplate/ WebTestClient can make actual HTTP calls end-to-end — controller → service → repository → database. This is the top of the pyramid: use it sparingly for the handful of flows where the real wiring is what you need to verify. Combined with Testcontainers, it's the most realistic test you can write (the "IT" suffix conventionally marks integration tests, often run in a separate build phase).

Part 5 — @MockBean, Test Data & Isolation

@MockBean UserService service;   // REPLACES the real bean in the context with a Mockito mock
@SpyBean AuditService audit;     // wraps the REAL bean; stub some methods, let others run

// Isolation: @DataJpaTest and @Transactional tests roll back after each test, so tests don't leak state.
// Testcontainers gives a fresh DB per class/run. AVOID @DirtiesContext unless truly needed — it rebuilds the
// whole context (very slow); prefer resetting mocks and relying on rollback.
🔧 @MockBean swaps a bean in the Spring context for a mock — it's how a slice or integration test replaces a real collaborator (a payment gateway, an email sender) with a controllable fake, without changing production wiring. @SpyBean wraps the real bean so you can stub a method or two while the rest runs for real. For isolation, lean on transactional rollback (@DataJpaTest and transactional tests undo their writes) and fresh containers so tests don't depend on each other's state or execution order — order-dependent failures are a classic flaky-test cause. Avoid overusing @DirtiesContext: rebuilding the context per test is slow and usually a sign a test is mutating shared singletons it shouldn't.

Part 6 — Deep Dive: Choosing the Right Level & Keeping Tests Fast and Honest

  • Match the test to what can break. Business logic → unit test the service with mocks. Routing/binding/validation/JSON → @WebMvcTest. Query/mapping correctness → @DataJpaTest on the real engine. Wiring/config → a few @SpringBootTest. Testing logic through a full context is slow and blurs which layer failed.
  • Context caching keeps slice suites fast. Spring caches the application context between tests that share the same configuration, so many tests reuse one boot. Anything that varies the config (a different @MockBean set, @DirtiesContext, per-test properties) forces a fresh context — a common reason a suite balloons in time.
  • Test the error and edge paths, not just the happy path. The 400 for an invalid body, the 404 for a missing entity, the 409 for a conflict, the rollback on failure — these are exactly what breaks in production and what slice tests verify cheaply. A suite that only asserts success ships broken error handling.
  • Prefer real dependencies where they're cheap and honest. Testcontainers made "use the real DB" cheap; mock only what's genuinely external or slow (a third-party API, a payment provider). Over- mocking produces tests that pass while the integration is broken.
🔍 Senior framing: a good Spring test suite is fast, focused, and honest, and achieving that is entirely about choosing the right level for each thing that can break. The pyramid isn't dogma — it's a response to real forces: full-context tests are slow (seconds of boot each) and test so much at once that a failure doesn't localize, so you want the narrowest slice that exercises the code under test. Plain unit tests (enabled by constructor injection, so you new the service with mocks and skip Spring entirely) cover business logic in milliseconds and should dominate the suite. @WebMvcTest covers what the web layer owns — routing, binding, validation-driven 400s, exception-advice mappings, JSON, status codes — through MockMvc with collaborators mocked, so it's fast and pinpoints web bugs. @DataJpaTest covers queries and mappings, and here the crucial senior call is to run against the real database engine via Testcontainers rather than H2, because the whole point of a data test is to catch dialect, constraint, and migration issues that H2's approximation hides — testing Postgres behavior on H2 is a false positive generator. A thin top layer of @SpringBootTest (ideally with a real port and Testcontainers) verifies the wiring and true end-to-end flows for the few paths that warrant it. Around all of this sit the discipline points that keep a suite trustworthy: isolation via transactional rollback and fresh containers (so no order-dependent flakiness), leveraging Spring's context caching by not needlessly varying configuration (the #1 reason slice suites get slow), testing the error and edge paths that actually break in production, and mocking only what's truly external so tests don't pass while the real integration is broken. The mature engineer treats the test pyramid as an economic optimization — maximum confidence per second of test time — and can justify, for any given behavior, exactly which level proves it and why, which is precisely what an interviewer is checking when they ask "how would you test this controller / this repository / this whole flow?"

Interview Questions

Q1 (beginner): What is the testing pyramid and why does it matter for a Spring app?

The pyramid prescribes many fast, narrow tests at the bottom and few slow, broad tests at the top: lots of plain unit tests, a middle band of slice tests, and a thin layer of full integration tests. It matters because Spring integration tests boot the application context, which is slow, and they test everything at once, so a failure is hard to localize. Keeping most tests as plain unit tests (which run in milliseconds because you new the service with mocks) and slices (which boot only one layer) gives a suite that's fast to run on every change and precise about where a failure is, reserving @SpringBootTest for the handful of flows where real end-to-end wiring is what you're verifying.

Q2 (beginner): How do you unit-test a service without starting Spring?

You construct it directly: new OrderService(mock(PaymentGateway.class), mock(OrderRepository.class)), stub the mocks' behavior with Mockito (when(...).thenReturn(...)), call the method under test, and assert on the result and interactions (verify(...)). No Spring context is involved, so the test runs in milliseconds. This is only clean because the service uses constructor injection — its dependencies are constructor parameters you can pass mocks to. It's the fastest, most focused kind of test and should make up most of the suite; you only need Spring when you're testing Spring's own behavior (wiring, MVC, JPA).

Q3 (mid): @SpringBootTest vs the slice annotations — when do you use each?

@SpringBootTest loads the entire application context (optionally starting a real server) — use it sparingly for genuine end-to-end/integration tests where the full wiring is the point. Slice annotations load only a portion: @WebMvcTest boots the MVC infrastructure plus one controller (with collaborators mocked via @MockBean) to test routing, binding, validation, exception mapping, status codes, and JSON via MockMvc — no DB, no services; @DataJpaTest boots JPA and a test database to test repositories and queries, rolling back each test. Slices are far faster and more focused, so the suite should be mostly plain unit tests and slices with a thin layer of full-context integration tests. The rule: pick the narrowest slice that still exercises the thing that can break.

Q4 (mid): Why prefer Testcontainers over an in-memory H2 database?

H2 approximates SQL but differs from production engines (Postgres/MySQL) in data types, functions, constraint and locking behavior, and dialect quirks — so a query, cascade, or Flyway migration can pass on H2 and fail in production, which is worse than no test because it gives false confidence. Testcontainers launches the actual database engine in a Docker container for the test run, so repository tests, native queries, and migrations run against the same database you deploy. It costs a little container-startup time (mitigated by reusing containers across a class/run) but eliminates the entire "works on H2, breaks on Postgres" class of bug, which is why it's the modern standard for data-layer and integration tests. The same idea extends to other infrastructure — a real Redis or Kafka container instead of a fake.

Q5 (senior): What role does @MockBean play, and how does context caching affect suite speed?

@MockBean replaces a bean in the Spring context with a Mockito mock, letting a slice or integration test substitute a real collaborator (a payment gateway, an email sender, a downstream service) with a controllable fake without altering production wiring — you drive its behavior with when(...) and assert interactions with verify(...). Its interaction with speed is important: Spring caches the application context across tests that share the same configuration, so many tests reuse a single boot. But each distinct set of @MockBeans (or @TestConfiguration, or per-test properties, or @DirtiesContext) produces a different context that must be built and cached separately — so proliferating mock combinations or dirtying the context repeatedly defeats caching and makes the suite rebuild the context over and over, which is a leading cause of a slice suite that "should be fast" taking minutes. The senior practice is to keep the number of distinct test configurations small (share a common set of mocks/config where possible), avoid @DirtiesContext unless a test genuinely mutates shared singleton state, and reset mock behavior between tests instead of rebuilding the context — maximizing reuse of the cached context so the suite stays fast even as it grows.

Q6 (senior): How do you decide the right test level for a given piece of behavior, and how do you keep the suite honest?

Match the level to what can break. Pure business logic (a discount calculation, a state transition) → a plain unit test of the service with mocked collaborators, because that's the fastest and most focused way to cover branches and edge cases. Web-layer concerns — routing, request binding, @Valid validation and its 400s, exception-advice mappings (404/409), status codes, JSON shape → @WebMvcTest with MockMvc, since those are exactly what that slice owns. Persistence correctness — derived/ custom queries, entity mappings, cascades, migrations → @DataJpaTest on the real engine via Testcontainers, so dialect and constraint behavior is real. True end-to-end wiring and cross-layer flows → a few @SpringBootTest tests with a real port and Testcontainers. Keeping it honest means two disciplines: test the error and edge paths, not just the happy path (the invalid body, the missing entity, the conflict, the rollback), because those are what actually fail in production and are cheap to assert at the slice level; and mock only what's genuinely external or slow (a third-party API, a payment provider) — over-mocking produces green tests over a broken integration, while Testcontainers makes using the real database/queue cheap enough that you rarely need to fake infrastructure. Add isolation via transactional rollback and fresh containers so there's no order-dependent flakiness, and you get a suite that's fast, localizes failures, and actually reflects production behavior — the goal being maximum confidence per second of test time. Being able to justify, for any behavior, which level proves it and why is the mark of someone who has maintained a real test suite rather than just written tests.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Unit-test a service. Given a service with a repository and a gateway dependency, write a JUnit + Mockito test that stubs the repository and verifies the gateway is called.
Solution

new the service with mock(Repo.class) and mock(Gateway.class), stub the repo with when(repo.findById(1L)).thenReturn(...), call the method, assert the result, and verify(gateway).charge(...). No Spring context — it runs in milliseconds. This works because the service uses constructor injection.

2 (easy): Assert a 404. Write a @WebMvcTest that mocks the service to return empty and asserts the controller responds 404.
Solution

@WebMvcTest(Controller.class), @MockBean the service, stub it to return Optional.empty(), then mvc.perform(get("/api/users/99")).andExpect(status().isNotFound()). It boots only the web layer, so it's fast and proves the not-found mapping without a DB.

3 (medium): Prove validation rejects bad input. Write a @WebMvcTest that POSTs an invalid body and asserts a 400.
Solution

POST JSON that violates the constraints (empty name, malformed email) and .andExpect(status().isBadRequest()); optionally assert the error body shape with jsonPath. This verifies @Valid plus your validation advice fire — an error path that's cheap to test here and expensive to discover in production.

4 (medium): Repository test on the real engine. Write a @DataJpaTest (or @SpringBootTest) using Testcontainers Postgres and test a derived query.
Solution

Start a PostgreSQLContainer, wire it via @DynamicPropertySource, save an entity, and assert the derived query finds it. Running against real Postgres catches dialect/constraint issues H2 would hide. The test rolls back (or the container is fresh), so it's isolated.

5 (hard): End-to-end flow. Write a @SpringBootTest(webEnvironment=RANDOM_PORT) integration test that POSTs to create a resource then GETs it, all over real HTTP with a Testcontainers DB.
Solution

Full context + real port + TestRestTemplate/WebTestClient makes a real POST then GET through every layer to a real Postgres container, asserting the round-trip. Keep only a few of these at the top of the pyramid — they're slow but verify the actual wiring end-to-end.

6 (hard): Explain-it test. Out loud: (a) the pyramid and why slices beat full-context tests; (b) when to use @WebMvcTest vs @DataJpaTest vs @SpringBootTest; (c) why Testcontainers over H2.
Self-check rubric

Pass (a) if you said many fast narrow tests, few slow broad ones, because full-context tests are slow and don't localize. Pass (b) if you mapped web concerns→WebMvcTest, persistence→DataJpaTest, wiring→ SpringBootTest, using the narrowest slice. Pass (c) if you explained H2 differs from prod engines so it gives false positives, and Testcontainers runs the real engine. That's the production-testing story.

🎯 Key Takeaways

  • Follow the pyramid: mostly fast unit tests, a band of slices, a few full integration tests.
  • Constructor injection lets you unit-test services with mocks — no Spring context, milliseconds each.
  • @WebMvcTest tests the web layer (routing, validation, error mapping, JSON) via MockMvc; @MockBean the collaborators.
  • @DataJpaTest tests repositories — run it against the real engine with Testcontainers, not H2.
  • @SpringBootTest loads the full context for true end-to-end tests — use it sparingly (top of the pyramid).
  • Keep tests isolated (rollback / fresh containers) and test the error paths, not just the happy path.
