Testing Spring Boot Intermediate

Spring Boot's test support is a layered toolkit: fast plain unit tests where you need speed, and "slice" tests that spin up just the part of the context you're exercising. The mistake juniors make is loading the entire application for every test — slow and coarse. The skill is picking the narrowest test that proves what you care about. (Builds on the Java JUnit lesson.)

The Testing Pyramid, Spring Edition

Unit Test — No Spring at All (Fastest)

class OrderServiceTest {
    @Test void appliesDiscount() {
        PaymentGateway gateway = mock(PaymentGateway.class);       // Mockito
        OrderRepository repo   = mock(OrderRepository.class);
        var service = new OrderService(gateway, repo);             // constructor injection pays off here!

        when(repo.findById(1L)).thenReturn(Optional.of(new Order(...)));
        var result = service.total(1L);

        assertThat(result).isEqualByComparingTo("90.00");
        verify(gateway).charge(any());
    }
}
// This is why constructor injection matters: you `new` the service with mocks, no context to boot.
// Most of your tests should look like this — they run in milliseconds.

Web Slice — @WebMvcTest + MockMvc

@WebMvcTest(UserController.class)          // loads ONLY the web layer for this controller
class UserControllerTest {
    @Autowired MockMvc mvc;
    @MockBean UserService service;         // the service is mocked, not real

    @Test void returns404WhenMissing() throws Exception {
        when(service.find(99L)).thenReturn(Optional.empty());
        mvc.perform(get("/api/users/99"))
           .andExpect(status().isNotFound());
    }
    @Test void rejectsInvalidBody() throws Exception {
        mvc.perform(post("/api/users").contentType(APPLICATION_JSON)
              .content("""{"name":"","email":"bad"}"""))
           .andExpect(status().isBadRequest());           // proves @Valid + advice work
    }
}
// Tests routing, binding, validation, status codes, and JSON — without a database or real service.

Data Slice — @DataJpaTest

@DataJpaTest                               // configures JPA + an in-memory/Testcontainers DB, rolls back each test
class UserRepositoryTest {
    @Autowired UserRepository repo;

    @Test void findByEmailWorks() {
        repo.save(new User("a@x.com", "Ann"));
        assertThat(repo.findByEmail("a@x.com")).isPresent();
    }
}
🐳 Test against the real database engine. An in-memory H2 has subtle SQL differences from Postgres — a query that passes on H2 can fail in prod. Use Testcontainers to spin up a real Postgres in Docker for repository/integration tests. It's the standard for trustworthy data-layer tests.
@SpringBootTest
@Testcontainers
class OrderFlowIT {
    @Container static PostgreSQLContainer<?> db = new PostgreSQLContainer<>("postgres:16");
    @DynamicPropertySource static void props(DynamicPropertyRegistry r) {
        r.add("spring.datasource.url", db::getJdbcUrl);        // point Spring at the container
        r.add("spring.datasource.username", db::getUsername);
        r.add("spring.datasource.password", db::getPassword);
    }
    @Autowired TestRestTemplate rest;                          // real HTTP against the running app
    @Test void createThenFetch() { /* POST then GET, assert round-trip */ }
}

Interview Questions

Q1: @SpringBootTest vs the slice annotations — when each?

@SpringBootTest loads the entire application context — use it sparingly for genuine end-to-end/integration tests where the full wiring matters. Slice annotations load only a portion: @WebMvcTest boots the MVC infrastructure and one controller (with collaborators mocked via @MockBean) to test routing, binding, validation, and JSON; @DataJpaTest boots JPA and a test database to test repositories and queries, rolling back each test. Slices are far faster and more focused, so the pyramid should be mostly plain unit tests and slices, with a thin layer of full-context integration tests.

Q2: Why prefer Testcontainers over an in-memory H2 database?

H2 approximates SQL but differs from production engines (Postgres/MySQL) in types, functions, constraint behavior, and dialect quirks — so a query or migration can pass on H2 and fail in prod, giving false confidence. Testcontainers launches the actual database in a Docker container for the test run, so repository tests, native queries, and Flyway migrations run against the same engine you deploy. It costs a bit of startup time but eliminates the "works on H2, breaks on Postgres" class of bugs, which is why it's the modern standard for data-layer and integration tests.

Q3: What role does @MockBean play?

@MockBean replaces a bean in the Spring test context with a Mockito mock. In a @WebMvcTest you mock the service layer so the controller test exercises only web concerns (mapping, validation, serialization) without a real service or database. It's how you isolate a slice: the framework wires your mock wherever that type is injected, and you stub its behavior with when(...).thenReturn(...). (In Boot 3.4+ the successor is @MockitoBean.) It differs from a plain Mockito mock() in that it participates in the application context rather than being passed manually.

🏋️ Practice: write a plain-JUnit unit test for a service using Mockito mocks (constructor injection, no context); a @WebMvcTest asserting a 404 and a 400-on-invalid-body with MockMvc; a @DataJpaTest for a derived query; and a @SpringBootTest + Testcontainers Postgres integration test that POSTs then GETs a resource over real HTTP. Compare their run times.

🎯 Key Takeaways

  • Favor the narrowest test: plain unit tests (Mockito) > slices > full @SpringBootTest.
  • Constructor injection lets you unit-test services with mocks and no Spring context.
  • @WebMvcTest + MockMvc tests the web layer (routing, validation, JSON) with mocked services.
  • @DataJpaTest tests repositories; use Testcontainers to run the real DB engine.
  • Cover error/validation paths (400/404), and keep tests isolated via rollback or fresh containers.
