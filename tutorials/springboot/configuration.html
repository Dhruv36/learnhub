Configuration & @Bean Core Container

What you'll master in this lesson: the two ways beans enter the container — component scanning (@Component and friends) versus explicit @Bean methods in a @Configuration class; when to use each; how @Configuration classes are proxied so inter-bean method calls return singletons; conditional bean registration (@Conditional); and how @Value and @ConfigurationProperties pull external configuration into beans. This is the "how do I define and wire beans deliberately" lesson, and it's the foundation the auto-configuration lesson builds on.

Part 1 — Two Ways to Register a Bean

// WAY 1 — Component scanning: annotate YOUR class, the scanner finds it:
@Service                                   // (or @Component, @Repository, @RestController)
public class OrderService { ... }

// WAY 2 — @Bean factory method in a @Configuration class: for classes you DON'T own,
// or when construction needs logic:
@Configuration
public class AppConfig {
    @Bean
    public PaymentGateway paymentGateway(@Value("${stripe.key}") String key) {
        return new StripeGateway(key);     // YOU construct it; Spring manages it
    }
    @Bean
    public RestClient restClient() {
        return RestClient.builder().baseUrl("https://api.example.com").build();
    }
}
🎯 Use stereotype annotations for your own classes; use @Bean methods for third-party classes or construction that needs logic. You can't annotate a library's class with @Service, so you register it with a @Bean method that builds it (often using a builder, applying config, wiring in dependencies as method parameters). Both produce container-managed beans; the choice is about who owns the class and how complex construction is.

Part 2 — @Configuration and the Proxy Guarantee

@Configuration
public class AppConfig {
    @Bean public DataSource dataSource() { return buildDataSource(); }

    @Bean public JdbcTemplate jdbcTemplate() {
        return new JdbcTemplate(dataSource());   // calls dataSource() — but do you get a NEW DataSource?
    }
    // With @Configuration, NO — Spring PROXIES the config class so dataSource() returns the SAME
    // singleton bean, not a fresh object. jdbcTemplate() gets the one managed DataSource. ✅
}
🔍 The @Configuration proxy is what makes inter-bean references work. Spring subclasses (CGLIB-proxies) a @Configuration class so that calling one @Bean method from another returns the shared singleton instead of executing the method again and creating a duplicate. This "full mode" is why jdbcTemplate() and any other consumer of dataSource() all share one DataSource. If you used @Component instead of @Configuration ("lite mode"), that guarantee is lost and each call would create a new object — a subtle bug. So use @Configuration for classes with inter-dependent @Bean methods. (Alternatively, take the dependency as a method parameter, which Spring injects — often cleaner than calling the method.)

Part 3 — Conditional Bean Registration

@Configuration
public class CacheConfig {

    @Bean
    @ConditionalOnProperty(name = "cache.enabled", havingValue = "true")
    public CacheManager cacheManager() { ... }        // only if the property is set

    @Bean
    @ConditionalOnMissingBean(CacheManager.class)      // only if no other CacheManager exists
    public CacheManager noopCache() { return new NoOpCacheManager(); }

    @Bean
    @Profile("prod")                                   // only in the prod profile
    public MetricsExporter prodMetrics() { ... }
}
ConditionRegisters the bean only if…
@ConditionalOnPropertyA property has a given value
@ConditionalOnMissingBeanNo bean of that type is already defined
@ConditionalOnClassA class is on the classpath
@ProfileA given profile is active
🔍 The @Conditional family is the exact mechanism behind auto-configuration (next lesson): Spring Boot's auto-config classes register beans guarded by these conditions, and @ConditionalOnMissingBean is why defining your own bean overrides a default. Understanding conditionals here makes auto-configuration stop being magic — it's just conditional @Bean methods at framework scale.

Part 4 — Injecting External Configuration: @Value vs @ConfigurationProperties

// @Value — a single property, inline:
@Value("${app.timeout:30}") private int timeout;    // with a default of 30

// @ConfigurationProperties — a TYPED, validated GROUP of related properties (preferred):
@ConfigurationProperties(prefix = "app.payment")
@Validated
public record PaymentProps(
    @NotBlank String currency,
    @Min(1) int retryAttempts,
    Duration timeout
) {}
// application.yml:
//   app.payment.currency: USD
//   app.payment.retry-attempts: 3
//   app.payment.timeout: 5s

@EnableConfigurationProperties(PaymentProps.class)   // register it (or @ConfigurationPropertiesScan)
@Configuration class ConfigSetup {}
🎯 Prefer @ConfigurationProperties over scattered @Value for related settings. It gives type safety, IDE autocomplete, relaxed binding (retry-attempts → retryAttempts), rich type conversion (a Duration from "5s"), and — with @Validated — validation that fails fast at startup on bad config. @Value is fine for a one-off value; a cohesive group of settings belongs in a typed properties class (the Config & Profiles lesson goes deeper on the property sources themselves).

Part 5 — Import, Component Scan Boundaries & Organizing Config

@SpringBootApplication          // includes @ComponentScan of THIS package and below
public class App { }
// Beans must live under the main class's package to be scanned — a common "bean not found" cause.

@Configuration
@Import({SecurityConfig.class, DataConfig.class})   // pull in other config classes explicitly
public class AppConfig { }

// @ComponentScan(basePackages = "com.example.other")  // scan additional packages if needed
// Organize config by concern: SecurityConfig, WebConfig, DataConfig — each a focused @Configuration.
🔍 Component scanning is package-relative to the @SpringBootApplication class, so beans in sibling or parent packages aren't found by default — the root cause of many "no qualifying bean" / "consider defining a bean" errors. Keep your code under the main package, use @Import to compose configuration classes, and organize @Configuration classes by concern (security, web, data) rather than one giant config — the same single-responsibility discipline from the Java SOLID lesson.

Part 6 — Deep Dive: The Container as the Assembly Point

  • Explicit configuration is where you take control back from auto-config. Auto-configuration provides sensible defaults; your @Bean/@Configuration classes override or supplement them (thanks to @ConditionalOnMissingBean). Knowing how to define beans deliberately is how you customize the framework.
  • Configuration classes are the composition root. They're where you wire third-party objects, apply conditional logic, and bind external properties — the one place that knows how the application is assembled, keeping that concern out of your business beans (dependency inversion at the application level).
  • Conditionals make configuration adaptive. The same jar behaves differently based on properties, profiles, and classpath — the foundation of auto-configuration and of a single artifact that runs across environments.
  • Typed properties turn config into a first-class, validated part of the app. Bad configuration fails at startup with a clear message rather than causing a mysterious runtime error — the same fail-fast principle as validating inputs at a boundary.
🔍 Senior framing: the configuration model is the seam between "the framework's opinions" and "your application's needs." Auto-configuration (next lesson) supplies defaults via conditional @Bean methods; your @Configuration classes are where you accept, override, or extend those defaults, register beans for classes you don't own, adapt behavior via @Conditional/@Profile, and bind external settings into typed, validated properties. Getting this right — using @Configuration (not @Component) so inter-bean references share singletons, preferring @ConfigurationProperties for grouped settings, keeping config classes focused and under the scanned package — is what separates an application that's cleanly assembled and easy to customize from one that fights the framework. Interviewers ask about @Bean vs @Component, the @Configuration proxy, and @Value vs @ConfigurationProperties precisely because the answers reveal whether you understand the container as a deliberate assembly mechanism rather than a bag of annotations.

Interview Questions

Q1 (beginner): What's the difference between @Component and @Bean?

@Component (and its stereotypes @Service, @Repository, @RestController) is a class-level annotation that marks your own class for component scanning — the container finds it and instantiates it as a bean. @Bean is a method-level annotation inside a @Configuration class where you write the code that constructs and returns the object, and Spring manages the returned instance as a bean. Use @Component for classes you own and control; use @Bean for third-party classes you can't annotate, or when construction requires logic (builders, conditional setup, wiring). Both result in container-managed beans.

Q2 (beginner): When would you use a @Bean method over a stereotype annotation?

When you can't or shouldn't put a stereotype annotation on the class: most commonly a third-party library class (you don't own the source to add @Service), or when creating the object needs real construction logic — using a builder, reading configuration, applying conditional setup, or assembling it from other beans passed as method parameters. A @Bean method gives you full control over instantiation while still handing the result to the container. Stereotype annotations are simpler and preferred for your own straightforward classes.

Q3 (mid): Why must inter-dependent @Bean methods live in a @Configuration class rather than a @Component?

Because of how Spring handles calls between @Bean methods. In a @Configuration class ("full mode"), Spring creates a CGLIB proxy of the class so that when one @Bean method calls another (e.g. jdbcTemplate() calling dataSource()), the call is intercepted and returns the existing singleton bean rather than executing the method again. This guarantees that every consumer shares the same managed instance. If the class is a plain @Component ("lite mode"), that proxying doesn't happen, so an inter-bean method call runs the method normally and creates a new object each time — producing duplicate instances and subtle bugs (two different DataSources, for example). So use @Configuration whenever @Bean methods reference each other. (An alternative that works in either mode is to declare the dependency as a method parameter, which Spring injects from the container rather than by calling the method.)

Q4 (mid): What do the @Conditional* annotations do, and how do they relate to auto-configuration?

They register a bean only when a condition holds: @ConditionalOnProperty (a property has a value), @ConditionalOnMissingBean (no bean of that type is already defined), @ConditionalOnClass (a type is on the classpath), and @Profile (a profile is active), among others. They make bean registration adaptive to the environment. This is exactly the mechanism Spring Boot's auto-configuration uses: auto-config classes declare @Bean methods guarded by these conditions, so beans are contributed only when appropriate — and @ConditionalOnMissingBean is specifically why defining your own bean of a type causes the auto-configured default to back off. Understanding conditionals here demystifies auto-configuration as conditional bean registration applied at framework scale.

Q5 (senior): Compare @Value and @ConfigurationProperties and justify a default choice.

@Value("${key}") injects a single externalized property (with an optional default) directly into a field or parameter — quick and fine for a one-off value, but it offers no grouping, no type safety beyond simple coercion, no validation, and typos or missing keys tend to fail late or silently. @ConfigurationProperties(prefix=…) binds a whole tree of related properties into a typed object (ideally an immutable record or a class with getters), giving several advantages: type safety and rich conversion (e.g. binding "5s" to a Duration, lists and maps, nested objects), relaxed binding (kebab-case retry-attempts maps to retryAttempts), IDE autocomplete and metadata, and — with @Validated plus Jakarta constraints — validation that fails fast at startup with a clear message when configuration is wrong. It also keeps related settings cohesive in one object rather than scattered across many @Value fields, and makes configuration a first-class, testable part of the design. The justified default is therefore @ConfigurationProperties for any group of related settings (which is most real configuration), reserving @Value for a genuinely isolated single value. The deeper reason is the fail-fast principle: binding and validating configuration into typed objects at startup turns a whole class of "misconfigured in production" runtime surprises into immediate, well-located startup errors.

Q6 (senior): How do explicit configuration and auto-configuration interact, and how do you override a default the framework provides?

Auto-configuration supplies sensible default beans through conditional @Bean methods, and the vast majority of those are guarded by @ConditionalOnMissingBean — meaning "provide this default only if the user hasn't defined their own bean of this type." So the interaction is cooperative and ordered so that your configuration wins: when you declare your own @Bean (or a @Component) of the same type in a @Configuration class, the auto-configured default's condition fails and it backs off, leaving your bean in place. That's the primary override mechanism — simply define the bean yourself. Beyond that, you can tune many auto-configured beans purely through application.yml properties without writing any bean; you can exclude an entire auto-configuration class with @SpringBootApplication(exclude = …) or the spring.autoconfigure.exclude property when you want none of its defaults; and you can use @Conditional/@Profile on your own beans to make your overrides themselves environment-dependent. Ordering matters (user configuration is processed such that auto-config, which runs with lower priority and missing-bean conditions, defers to it), which is why the pattern reliably lets application config take precedence. The mental model is that auto-configuration is a set of conditional defaults and your explicit configuration is the authoritative layer on top; to customize the framework you either adjust properties, replace a specific bean (letting the default back off), or exclude an auto-config class entirely — and you diagnose what actually happened with the condition-evaluation report (--debug or /actuator/conditions).

🏋️ Exercises (attempt before opening solutions)

1 (easy): Register a third-party bean. Register a RestClient (or any library object) as a @Bean in a @Configuration class, configuring its base URL, and inject it into a service.
Solution

A @Configuration class with @Bean public RestClient restClient() { return RestClient.builder().baseUrl(...).build(); }, then inject RestClient into the service via constructor. You used a @Bean method because you don't own RestClient and construction needs the builder — the canonical case for @Bean over a stereotype.

2 (easy): @Value with a default. Inject an app.timeout property with a fallback of 30 using @Value, and confirm the default applies when the property is absent.
Solution

@Value("${app.timeout:30}") int timeout;. With the property unset, timeout is 30 (the value after the colon is the default); set app.timeout: 60 and it becomes 60. This is fine for a lone value — a group of settings would use @ConfigurationProperties.

3 (medium): Typed properties. Bind an app.payment.* group (currency, retry-attempts, timeout) with @ConfigurationProperties and @Validated, and make startup fail on a blank currency.
Solution

A record annotated @ConfigurationProperties(prefix="app.payment") and @Validated with @NotBlank String currency, @Min(1) int retryAttempts, Duration timeout; register with @EnableConfigurationProperties. A blank currency fails at startup with a validation error — fail-fast configuration. Note relaxed binding maps retry-attempts to retryAttempts and "5s" to a Duration.

4 (medium): The @Configuration proxy. Write two @Bean methods where one calls the other, and verify both consumers get the same singleton. Then switch @Configuration to @Component and observe the difference conceptually.
Solution

In a @Configuration class, jdbcTemplate() calling dataSource() returns the shared singleton DataSource (proxy intercepts the call). Switched to @Component ("lite mode"), the call would execute dataSource() again and create a second DataSource — a duplicate-bean bug. This is why inter-dependent @Bean methods must live in a @Configuration class.

5 (hard): Conditional beans. Provide a real CacheManager when cache.enabled=true and a no-op one otherwise, using @ConditionalOnProperty and @ConditionalOnMissingBean. Explain how this mirrors auto-configuration.
Solution

A @Bean @ConditionalOnProperty(name="cache.enabled", havingValue="true") real CacheManager, plus a @Bean @ConditionalOnMissingBean(CacheManager.class) NoOpCacheManager fallback. When the property is true, the real one registers and the fallback backs off (a CacheManager already exists); otherwise only the no-op registers. This is exactly how auto-configuration works — conditional @Bean methods, with @ConditionalOnMissingBean letting user beans override defaults.

6 (hard): Explain-it test. Out loud: (a) @Component vs @Bean and when to use each; (b) the @Configuration proxy and why it matters; (c) @Value vs @ConfigurationProperties and how conditionals underpin auto-configuration.
Self-check rubric

Pass (a) if you said stereotype for your classes, @Bean for third-party/complex construction. Pass (b) if you said the proxy makes inter-bean method calls return shared singletons. Pass (c) if you preferred typed @ConfigurationProperties and tied @Conditional* to auto-config's conditional bean registration. This sets up the Auto-Configuration lesson.

🎯 Key Takeaways

  • Beans come from stereotype annotations (your classes) or @Bean methods in a @Configuration class (third-party/complex construction).
  • @Configuration is proxied so inter-bean method calls return the shared singleton — don't use @Component for inter-dependent @Bean methods.
  • @Conditional* (OnProperty/OnMissingBean/OnClass/@Profile) registers beans adaptively — the exact mechanism behind auto-configuration.
  • Prefer typed, validated @ConfigurationProperties over scattered @Value for related settings (fail fast on bad config).
  • Component scanning is package-relative to the main class; organize config by concern and compose with @Import.
  • Explicit configuration is where you override auto-config (via @ConditionalOnMissingBean) — the deliberate assembly layer.
