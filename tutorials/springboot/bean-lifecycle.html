Bean Scopes & Lifecycle Core Container

What you'll master in this lesson: the exact sequence a Spring bean goes through from instantiation to destruction; the scopes (singleton, prototype, and the web scopes) and the classic "singleton depends on a prototype" trap; the initialization/destruction callbacks and when each fires; and why understanding the lifecycle is what lets you reason about startup order, resource management, and a whole class of subtle bugs. This deepens the IoC container model from the previous lesson into the mechanics of how beans actually live and die.

Part 1 — The Bean Lifecycle, Step by Step

1. INSTANTIATION      The container calls the bean's constructor (constructor injection happens here).
2. POPULATE           Setter/field dependencies are injected.
3. AWARE callbacks    If the bean implements *Aware interfaces (BeanNameAware, ApplicationContextAware…),
                      those setters are called so it can learn about its environment.
4. @PostConstruct     Your init method runs — the bean is fully wired; do startup work here.
   (or InitializingBean.afterPropertiesSet, or a @Bean initMethod)
5. READY              The bean is in service, handling requests.
   ... application runs ...
6. @PreDestroy        On context shutdown, your cleanup method runs (release resources).
   (or DisposableBean.destroy, or a @Bean destroyMethod)
7. DESTROYED          The bean is discarded.
💡 The key insight: dependencies are guaranteed to be injected before @PostConstruct runs. That's why initialization logic that needs collaborators (open a connection, warm a cache, validate config) belongs in a @PostConstruct method, not the constructor — in the constructor, field-injected dependencies aren't set yet (constructor injection is the exception, since those arrive as constructor arguments). This ordering is a container guarantee you can rely on, and it's why the framework distinguishes construction from initialization.

Part 2 — Initialization & Destruction Callbacks

@Service
public class ConnectionManager {

    @PostConstruct                       // runs AFTER wiring, BEFORE the bean serves requests
    public void init() {
        pool = openConnectionPool();     // safe: all dependencies are injected by now
    }

    @PreDestroy                          // runs on graceful shutdown
    public void cleanup() {
        pool.close();                    // release resources deterministically
    }
}

// Three equivalent mechanisms (prefer the annotations):
//   @PostConstruct / @PreDestroy         — JSR-250 annotations (recommended)
//   InitializingBean / DisposableBean    — Spring interfaces (couples you to Spring)
//   @Bean(initMethod=..., destroyMethod=...) — for third-party classes you can't annotate
🎯 Prefer @PostConstruct/@PreDestroy — they're standard annotations, don't couple your class to Spring interfaces, and read clearly. Use the @Bean(initMethod=…, destroyMethod=…) form for third-party classes you can't modify. Important caveat: @PreDestroy only runs on a graceful shutdown (the context closing normally); it won't run if the JVM is killed (kill -9), so don't rely on it for critical persistence — that needs a more robust mechanism.

@Scope("prototype")            // a NEW instance every time it's requested

🔑 Singleton is the default, and Spring singletons are shared across every request thread — so keep them stateless (the concurrency lesson from Java: mutable state on a shared singleton is a race). A Spring "singleton" is one-per-container, not the JVM-wide Singleton pattern (there's one instance per ApplicationContext). Prototype scope gives a fresh instance per injection for genuinely stateful, short-lived objects — but note the trap in Part 4.

Part 4 — Deep Dive: The Singleton-Injecting-Prototype Trap

@Service                                  // a SINGLETON
public class CheckoutService {
    private final ShoppingCart cart;      // a PROTOTYPE dependency
    public CheckoutService(ShoppingCart cart) { this.cart = cart; }
    // ❌ The prototype is injected ONCE, when the singleton is created. Every call reuses the SAME
    //    cart instance — you do NOT get a new prototype per method call, defeating the point!
}

A prototype-scoped bean injected into a singleton is created only once — at the time the singleton is constructed — because the singleton itself is only constructed once. The prototype's "new instance every time" semantics don't magically apply per method call. This surprises everyone the first time.

// FIXES — get a fresh prototype when you actually need one:
// 1. Inject a provider and ask for a new one each time:
@Service
public class CheckoutService {
    private final ObjectProvider<ShoppingCart> cartProvider;
    public CheckoutService(ObjectProvider<ShoppingCart> p) { this.cartProvider = p; }
    public void checkout() {
        ShoppingCart cart = cartProvider.getObject();   // a NEW prototype each call
    }
}
// 2. @Lookup method injection, or a scoped-proxy — same idea: defer creation to use time.
🔍 Senior nugget: the root cause is scope mismatch — an object with a shorter lifecycle (prototype) injected into one with a longer lifecycle (singleton) can't refresh itself, because injection happens at the injector's creation. The same issue applies to injecting a request-scoped bean into a singleton, which Spring solves with a scoped proxy: it injects a proxy that, on each method call, resolves the real bean for the current request/scope. Recognizing "shorter-lived injected into longer-lived → need a provider/proxy" is the general principle; it's a favorite interview question because it tests whether you understand that DI wiring happens at construction time.

Part 5 — Aware Interfaces & BeanPostProcessors (how the framework hooks in)

// *Aware interfaces let a bean access framework infrastructure (use sparingly — couples to Spring):
@Component
class MyBean implements ApplicationContextAware {
    public void setApplicationContext(ApplicationContext ctx) { this.ctx = ctx; }
}

// BeanPostProcessor — the extension point the framework itself uses to modify beans during init.
// This is HOW @Autowired, @Transactional, AOP proxies, etc. are applied: a BeanPostProcessor wraps
// or configures your bean between instantiation and readiness. You rarely write one, but knowing it
// exists explains "how does Spring add transactional behavior to my plain @Service?" (answer: a
// BeanPostProcessor creates a proxy — see the AOP lesson).
🔍 BeanPostProcessors run for every bean during initialization and are the mechanism behind Spring's "magic": annotation processing, proxy creation for @Transactional/@Async/security, and property injection all happen here. You don't usually implement one, but understanding that a post-processor wraps your bean in a proxy demystifies a lot — and it's the direct link to the self-invocation gotcha (a proxied method called internally bypasses the proxy) that recurs across the transactions, caching, and AOP lessons.

Part 6 — Deep Dive: Why the Lifecycle Matters in Practice

  • Resource management. @PostConstruct/@PreDestroy give you deterministic init/cleanup tied to the container's lifecycle — open pools/connections on init, close them on shutdown — which, combined with graceful shutdown, prevents leaks and dropped work.
  • Startup ordering. Beans are created in dependency order (a bean's dependencies are instantiated first). Knowing this explains startup failures ("bean X requires Y which failed") and why circular dependencies are a problem (two beans each needing the other at construction).
  • Statelessness discipline. Because singletons are shared and long-lived, the lifecycle reinforces "keep beans stateless" — per-request state belongs in method parameters, request-scoped beans, or the call stack, not singleton fields.
  • The proxy mental model. Understanding that init is when post-processors wrap beans in proxies is the foundation for reasoning about every AOP-based feature (transactions, caching, async, security) and their shared self-invocation caveat.
🔍 Senior framing: the bean lifecycle is the container's contract, and internalizing it converts a lot of Spring "magic" into predictable mechanics. You know that dependencies are wired before @PostConstruct, that singletons are created once and shared (so keep them stateless), that a shorter-lived scope injected into a longer-lived one needs a provider or scoped proxy, that @PreDestroy runs only on graceful shutdown, and that the framework applies its cross-cutting behavior by wrapping beans in proxies during initialization via BeanPostProcessors. With that model you can diagnose startup-order failures, circular-dependency errors, resource leaks, prototype scope surprises, and the self-invocation gotcha — the recurring puzzles of real Spring applications — from first principles rather than by trial and error. Interviewers probe the lifecycle and scopes precisely because the answers reveal whether you understand the container or just memorized annotations.

Interview Questions

Q1 (beginner): What are the main phases of a Spring bean's lifecycle?

Instantiation (the constructor runs, with constructor-injected dependencies supplied), dependency population (setter/field injection), any *Aware callbacks, initialization (@PostConstruct / afterPropertiesSet / a configured init method) after the bean is fully wired, the ready/in-service phase, and finally destruction on context shutdown (@PreDestroy / destroy / a configured destroy method). The crucial ordering guarantee is that all dependencies are injected before the initialization callback runs, which is why startup work needing collaborators goes in @PostConstruct, not the constructor (for field injection).

Q2 (beginner): What's the default bean scope, and what does it imply?

Singleton — one instance per Spring ApplicationContext, shared everywhere it's injected. The implication is that a singleton bean is used concurrently by every request thread, so it must be stateless: any mutable instance field is shared state and a race-condition risk. Keep collaborators (other beans, immutable config) as fields, and put per-request data in method parameters and local variables. Note "singleton" here means one-per-container, not the JVM-wide Singleton design pattern — a different context yields a different instance.

Q3 (mid): Where should initialization logic go, and why not the constructor?

In a @PostConstruct method (or afterPropertiesSet / a configured init method), because that runs after all dependencies are injected, so it can safely use collaborators to open resources, warm caches, or validate configuration. The constructor is too early for field/setter injection — those dependencies aren't set yet, so using them there causes a NullPointerException. (Constructor injection is the exception, since those dependencies arrive as constructor arguments, but even then non-trivial startup side effects are cleaner in @PostConstruct to keep construction simple and to run after the full context is ready.) The container guarantees the wiring-before-init ordering, so it's the reliable place for setup.

Q4 (mid): What happens when you inject a prototype bean into a singleton?

The prototype is instantiated only once — at the moment the singleton is constructed — and that single instance is reused for the singleton's entire life, because the singleton is created just once and DI happens at construction time. So you do not get a fresh prototype per method call, which defeats the purpose of prototype scope. To get a new instance when you actually need one, don't inject the prototype directly; inject an ObjectProvider<T> (or Provider) and call getObject() at use time, use @Lookup method injection, or configure a scoped proxy. The general principle is that a shorter-lived scope injected into a longer-lived one can't refresh itself, so you defer resolution to use time via a provider or proxy — the same technique Spring uses for request-scoped beans injected into singletons.

Q5 (senior): How does Spring apply cross-cutting behavior like @Transactional during the bean lifecycle, and what limitation does that create?

Through BeanPostProcessors that run during initialization: after a bean is instantiated and wired, a post-processor inspects it and, if it needs cross-cutting behavior (transactions, caching, async, security, or any AOP aspect), wraps it in a proxy — a dynamic subclass (CGLIB) or a JDK dynamic proxy implementing its interfaces — that adds the behavior around the real method invocations before delegating to the target. So the object other beans receive is actually the proxy, and it's the proxy that starts/commits a transaction, checks the cache, or enforces security around each call. The key limitation this creates is self-invocation: when a method of the bean calls another method of the same bean directly (via this.method()), the call goes straight to the target and never passes through the proxy, so the cross-cutting behavior is silently skipped — a @Transactional or @Cacheable method invoked from within the same class is not intercepted, and the annotation appears to "not work." The fixes are to call through another bean (so the invocation crosses the proxy), self-inject the proxy and call through that reference, or restructure so the annotated method is invoked externally. Understanding that these features are proxy-based, applied during initialization by post-processors, explains both how the "magic" works and why this recurring gotcha exists across transactions, caching, async, and AOP.

Q6 (senior): Explain why @PreDestroy can't be relied on for critical cleanup, and how you'd handle resources robustly.

@PreDestroy (and any container destruction callback) runs only when the application context is closed gracefully — a normal shutdown where Spring gets the chance to walk its beans and invoke their destroy methods. It will not run if the process is killed abruptly (kill -9, an OOM kill, a power loss, a hard container termination), because the JVM dies before any cleanup code executes. So using @PreDestroy as the sole mechanism for critical, must-happen work — flushing unsaved data, releasing an external lock, finalizing a transaction — risks data loss or leaked external state on any non-graceful exit. Robust handling layers defenses: make operations transactional and durable so incomplete work is rolled back or recoverable rather than depending on shutdown cleanup; write data as you go rather than buffering it to flush at the end; use idempotent operations and external timeouts/leases (so an abandoned lock expires on its own) instead of relying on releasing it in @PreDestroy; and configure the platform's graceful shutdown (Spring Boot's server.shutdown: graceful plus a termination grace period in the orchestrator) so that in the common case the process is given time to drain in-flight requests and run destroy callbacks, while designing so that even a hard kill leaves the system in a recoverable state. In short, use @PreDestroy for best-effort, convenient cleanup of in-process resources (closing pools, stopping threads) on the normal path, but never make correctness depend on it — durability and recoverability must come from mechanisms that survive an abrupt death.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Lifecycle callbacks. Add @PostConstruct and @PreDestroy methods that log to a bean, run the app, and observe when they fire relative to startup and shutdown.
Solution

The @PostConstruct log appears during startup, after the bean's dependencies are wired and before it serves requests; the @PreDestroy log appears on graceful shutdown (e.g. Ctrl+C / context close). Try killing the process abruptly and note @PreDestroy does not run — proving it's for graceful shutdown only.

2 (easy): Prove singleton sharing. Inject the same @Service into two different components and compare identity (==). Then make one @Scope("prototype") and compare again.
Solution

The default singleton is the same instance in both places (== true) — one per container. With @Scope("prototype"), each injection point gets a distinct instance (== false). This demonstrates the scopes concretely and reinforces "singletons are shared, so keep them stateless."

3 (medium): Reproduce the prototype trap. Inject a prototype bean directly into a singleton, call a method twice, and show it's the same prototype instance both times. Then fix it with ObjectProvider.
Solution

Injected directly, the prototype is created once with the singleton, so both method calls see the same instance (identity equal) — the "new per use" semantics don't apply. Fix: inject ObjectProvider<Prototype> and call provider.getObject() inside the method; now each call yields a fresh prototype. This is the shorter-lived-into-longer-lived principle in action.

4 (medium): Init needs a dependency. Write a bean whose startup work requires an injected collaborator. Put it in the constructor (with field injection) and see it fail, then move it to @PostConstruct.
Solution

With field injection, using the collaborator in the constructor throws an NPE because the field isn't injected yet. Moving the logic to a @PostConstruct method fixes it — by then all dependencies are wired. (Alternatively, use constructor injection so the collaborator arrives as an argument.) This demonstrates the wiring-before-init ordering guarantee.

5 (hard): Trace the proxy. Add @Transactional to a @Service method, then call it from another method of the same class and observe the transaction doesn't apply. Explain via the lifecycle/proxy model and fix it.
Solution

Spring wraps the @Service in a proxy during initialization (via a BeanPostProcessor); the transaction is started by the proxy around external calls. Calling the annotated method from another method of the same class (this.method()) bypasses the proxy, so no transaction is opened — self-invocation. Fix: call the annotated method through another bean, self-inject the proxy and call via that reference, or move the method to a separate bean. This connects the lifecycle (proxy created at init) to the recurring self-invocation gotcha.

6 (hard): Explain-it test. Out loud: (a) the lifecycle phases and the wiring-before-init guarantee; (b) why singletons must be stateless and the prototype-in-singleton trap; (c) how the framework applies transactions via proxies at init and the self-invocation consequence.
Self-check rubric

Pass (a) if you ordered instantiate → populate → aware → init(@PostConstruct) → ready → destroy and noted deps are wired before init. Pass (b) if you said singletons are shared per container (so stateless) and a prototype injected into a singleton is created once. Pass (c) if you said a BeanPostProcessor wraps the bean in a proxy at init and self-invocation bypasses it. These set up the AOP, Transactions, and Caching lessons.

🎯 Key Takeaways

  • Lifecycle: instantiate → inject deps → *Aware → @PostConstruct → ready → @PreDestroy → destroy; deps are wired before init.
  • Put startup work needing collaborators in @PostConstruct (not the constructor for field injection); prefer @PostConstruct/@PreDestroy over Spring interfaces.
  • Singleton (default) = one per container, shared across threads — keep beans stateless; prototype = new per injection/lookup.
  • A prototype injected into a singleton is created once — use ObjectProvider/@Lookup/scoped proxy for a fresh instance per use.
  • The framework applies @Transactional/caching/async/security by wrapping beans in proxies at init (via BeanPostProcessors) — hence the self-invocation gotcha.
  • @PreDestroy runs only on graceful shutdown — never rely on it for critical persistence.
