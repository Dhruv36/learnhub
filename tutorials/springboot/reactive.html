Reactive Spring & WebFlux Event-Driven & Reactive

What you'll master in this lesson: the reactive paradigm and why it exists — the thread-per-request model's limit under massive I/O-bound concurrency — how WebFlux, Mono/Flux, and non-blocking I/O let a few threads serve tens of thousands of connections, the one rule that must never be broken (never block a reactive thread), and the honest senior verdict: most apps don't need reactive, and virtual threads (Loom) now deliver much of its benefit with none of its complexity. Prerequisites: Executors, Virtual Threads, REST Controllers, Concurrency Patterns.

Part 1 — The Problem: Thread-per-Request Under Heavy I/O

// Traditional Spring MVC: ONE thread per request, and that thread BLOCKS during I/O:
@GetMapping("/user/{id}")
public User get(@PathVariable Long id) {
    return webClient.get()...block();   // thread SITS IDLE waiting on the network — doing nothing, but occupied
}
// With a 200-thread pool, ~200 concurrent slow requests exhausts it — the 201st waits, even though the
// CPU is nearly idle (all threads are just WAITING on I/O). Threads are the bottleneck, not the CPU.
🎯 The classic server model dedicates one thread to each request and that thread blocks while waiting on I/O — so under massive I/O-bound concurrency, threads become the scarce resource. A thread waiting on a database or a remote API is occupied but doing no work, yet it still consumes memory (a MB of stack) and a pool slot. With a bounded pool (say 200 threads), 200 simultaneously-blocked requests exhaust it and further requests queue — even though the CPU is nearly idle because everyone is just waiting. This is the specific problem reactive programming targets: high-concurrency, I/O-bound workloads (API gateways, services fanning out to many downstreams, streaming) where you want to handle far more concurrent connections than you have threads.

Part 2 — The Reactive Model: Mono, Flux & Non-Blocking I/O

// WebFlux is non-blocking: a small number of event-loop threads handle MANY connections. Nothing blocks;
// when I/O completes, a callback resumes the work. You express results as reactive STREAMS:
//   Mono<T>  = 0 or 1 value (a single result)      Flux<T>  = 0..N values (a stream)

@GetMapping("/user/{id}")
public Mono<User> get(@PathVariable Long id) {
    return webClient.get().uri("/u/{id}", id)
            .retrieve().bodyToMono(User.class);   // returns IMMEDIATELY; no thread blocks waiting
}

@GetMapping("/users")
public Flux<User> all() {
    return repository.findAll()                    // a stream of users, backpressure-aware
            .filter(User::isActive)
            .map(this::toDto);                     // operators build a PIPELINE; nothing runs until subscribed
}
🔧 Reactive replaces "block a thread until the value arrives" with "describe a pipeline that produces values asynchronously." A Mono<T> is a promise of 0-or-1 value; a Flux<T> is an asynchronous stream of 0-to-N values. You build a processing pipeline with operators (map, filter, flatMap), and crucially nothing executes until something subscribes — the pipeline is a lazy description, and in a WebFlux app the framework subscribes when the HTTP response is written. Under the hood a few event-loop threads drive many connections: when a request awaits I/O, its thread isn't parked — it moves on to other work, and a callback resumes this request when the I/O completes. That's how a handful of threads serve tens of thousands of concurrent connections.

Part 3 — Backpressure: The Feature That Names the Paradigm

// BACKPRESSURE = a slow CONSUMER can signal a fast PRODUCER to slow down, so it isn't overwhelmed.
// Reactive Streams builds this in: the subscriber requests N items; the publisher sends at most N.
flux.onBackpressureBuffer()      // buffer overflow (bounded)
    .limitRate(100)              // pull 100 at a time from upstream
// vs a naive stream that pushes as fast as it can → the consumer OOMs or drops data under load.
🎯 Backpressure — a slow consumer throttling a fast producer — is the defining feature of Reactive Streams, and it's what "reactive" really refers to. In a naive push model, a fast producer (a firehose database cursor, a busy Kafka topic) overwhelms a slower consumer, forcing it to buffer unboundedly (OOM) or drop data. The Reactive Streams contract inverts this: the subscriber requests a number of items and the publisher sends no more than requested, so demand flows upstream and the system self-regulates end-to-end. This is the same back-pressure idea as a bounded BlockingQueue (Concurrency Patterns) but woven through the entire async pipeline — it's why reactive can stream large or unbounded data safely where a naive async approach can't.

Part 4 — The Cardinal Rule: Never Block a Reactive Thread

// The event loop has very FEW threads. Block ONE and you stall every connection it was serving.
@GetMapping("/bad")
public Mono<String> bad() {
    Thread.sleep(1000);                     // ❌ CATASTROPHE — blocks an event-loop thread; throughput collapses
    String r = jdbcTemplate.queryForObject(...);   // ❌ blocking JDBC on a reactive thread — same disaster
    return Mono.just(r);
}

// If you MUST call blocking code, push it to a separate scheduler so the event loop stays free:
Mono.fromCallable(() -> blockingCall())
    .subscribeOn(Schedulers.boundedElastic());   // runs the blocking work on a dedicated pool
🚨 The one unbreakable rule of reactive: never perform a blocking operation on an event-loop thread. Because so few threads drive all connections, blocking one — with Thread.sleep, blocking JDBC, a synchronous HTTP client, a .block() call, or any blocking library — stalls every connection that thread was multiplexing, and throughput collapses. This is the biggest source of reactive bugs and the reason reactive is all-or-nothing: your entire stack must be non-blocking (R2DBC instead of JDBC, WebClient instead of RestTemplate, reactive drivers for everything) — a single blocking call anywhere poisons the model. If you genuinely must call blocking code, isolate it on a separate scheduler (boundedElastic) so the event loop stays free, but needing that often is a sign reactive isn't the right fit.

@GetMapping("/user/{id}")

    return webClient.get()...block();    // "blocking" code — but on a VIRTUAL thread it doesn't tie up an OS thread
}
// spring.threads.virtual.enabled=true → Spring MVC serves each request on a virtual thread.
// You write simple, debuggable, thread-per-request code AND scale to huge I/O concurrency.
⚖️ Reactive's power comes at a real cost — and virtual threads have changed the calculus. Reactive code is a different paradigm with a steep learning curve, notoriously hard debugging (a stack trace doesn't reflect your logical flow), an all-or-nothing non-blocking requirement, and ecosystem friction. Virtual threads (Project Loom, Java 21) now deliver much of the same scalability benefit — a blocking call on a virtual thread unmounts from its carrier OS thread instead of tying it up, so you can have millions of cheap threads — while letting you keep the simple, debuggable, thread-per-request blocking style. For the common case (an I/O-bound service that needs high concurrency), virtual threads give you the scalability with a fraction of the complexity, which is why the industry increasingly reaches for Loom over WebFlux for that use case. Reactive still wins for genuine streaming and backpressure-heavy pipelines — but "we need to scale, so we should go reactive" is now often answered by "enable virtual threads instead."

Part 6 — Deep Dive: A Powerful Tool You Usually Don't Need

  • Reactive solves one specific problem: massive I/O-bound concurrency with few threads. It's not faster for a single request (often slightly slower); its win is handling far more concurrent connections per unit of memory/threads. If you don't have that problem, you pay the complexity for nothing.
  • Backpressure and streaming are its genuine differentiators. Where you truly need to stream unbounded data with a slow consumer safely throttling a fast producer, reactive is the right model and virtual threads don't replace it.
  • The all-or-nothing constraint is the practical killer. One blocking call anywhere defeats it, and much of the ecosystem is still blocking — so adopting reactive is a whole-stack commitment.
  • Virtual threads reset the default. For "scale an I/O-bound app," Loom now gives most of the benefit with ordinary blocking code — so reactive should be a deliberate choice for its unique strengths, not a reflexive answer to "we need concurrency."
🔍 Senior framing: reactive programming is a powerful, specialized tool, and the mark of judgment is knowing precisely what problem it solves and — increasingly — when not to reach for it. The problem is the thread-per-request model under massive I/O-bound concurrency: a blocking thread waiting on I/O is occupied but idle, consuming a stack and a pool slot, so a bounded pool caps how many concurrent slow requests you can serve even while the CPU sits nearly idle. WebFlux answers this with non-blocking I/O over a few event-loop threads that multiplex many connections, expressing results as lazy reactive streams — Mono (0..1) and Flux (0..N) — built from operators that don't execute until subscribed, so a handful of threads serve tens of thousands of connections. Its defining feature, and the source of the name, is backpressure: the Reactive Streams contract lets a slow consumer signal demand upstream so a fast producer can't overwhelm it, enabling safe streaming of large or unbounded data where a naive push model would OOM. But two hard truths shape the senior verdict. First, the cardinal rule — never block an event-loop thread — makes reactive all-or-nothing: because so few threads carry everything, one blocking call (blocking JDBC, Thread.sleep, a synchronous client, .block()) stalls every connection that thread serves and collapses throughput, so the entire stack must be non-blocking (R2DBC, WebClient, reactive drivers), and much of the ecosystem still isn't. Second, reactive carries a genuine cost in learning curve and, especially, debuggability — stack traces don't mirror the logical flow, and reasoning about async pipelines is genuinely hard. Crucially, virtual threads (Loom, Java 21) have changed the default: a blocking call on a virtual thread unmounts from its carrier rather than tying up an OS thread, so you get most of reactive's I/O-concurrency scalability while writing simple, debuggable, thread-per-request blocking code — which means the old syllogism "we need to scale ⇒ go reactive" is now usually better answered by "enable virtual threads." Reactive remains the right tool for true streaming and backpressure-heavy pipelines, but for the common high-concurrency I/O-bound service it's often unnecessary complexity. The mature engineer therefore treats reactive as a deliberate choice justified by streaming/ backpressure needs, understands it doesn't speed up a single request (its win is concurrency density), respects the non-blocking-everywhere constraint, and — post-Loom — reaches for virtual threads first for plain scalability, reserving WebFlux for where its unique strengths actually pay for its cost. Being able to say "here's exactly when I'd use reactive and, more often, why I wouldn't" is precisely the seniority signal, since it demonstrates matching a sophisticated tool to the specific problem it's for rather than adopting it as a fashion.

Interview Questions

Q1 (beginner): What problem does reactive programming solve?

It targets high-concurrency, I/O-bound workloads under the traditional thread-per-request model. In that model each request gets a dedicated thread that blocks while waiting on I/O (a database, a remote API), and a blocked thread is occupied but doing nothing while still consuming memory (a stack) and a pool slot. So with a bounded thread pool, a modest number of simultaneously-blocked slow requests exhausts the pool and further requests queue — even though the CPU is nearly idle because all those threads are just waiting. Reactive (WebFlux) uses non-blocking I/O so a small number of event-loop threads can handle tens of thousands of concurrent connections: a request awaiting I/O doesn't hold a thread; a callback resumes it when the I/O completes. So the problem solved is "serve far more concurrent I/O-bound connections than you have threads," typical of API gateways and services that fan out to many downstreams. Note it doesn't make a single request faster — the benefit is concurrency density, not per-call latency.

Q2 (beginner): What are Mono and Flux?

They're the two core reactive types in Project Reactor (which WebFlux uses), representing asynchronous results as streams. A Mono<T> is a publisher of 0 or 1 value — the reactive equivalent of a single (possibly-absent) result, like a lookup by id. A Flux<T> is a publisher of 0 to N values — an asynchronous stream, like a query returning many rows or a continuous feed. You build processing pipelines on them with operators (map, filter, flatMap), and a key property is laziness: the pipeline is just a description and nothing executes until something subscribes — in a WebFlux endpoint, the framework subscribes when it writes the HTTP response. So a controller returns a Mono<User> or Flux<User> rather than a User/List<User>, and the value(s) are produced asynchronously without blocking a thread. Forgetting the laziness (not returning/subscribing the publisher) is a common beginner mistake where "the pipeline does nothing."

Q3 (mid): What is backpressure and why does it matter?

Backpressure is the ability of a slow consumer to signal a fast producer to slow down, so the consumer isn't overwhelmed — and it's the defining feature of Reactive Streams (indeed what "reactive" really denotes). In a naive push model, a fast producer (a database cursor streaming rows, a busy message topic, a firehose) emits as fast as it can, and a slower consumer must either buffer the excess — growing unboundedly until it runs out of memory — or drop data. The Reactive Streams contract inverts control: the subscriber requests a specific number of items (demand), and the publisher emits no more than has been requested, so demand propagates upstream and the whole pipeline self-regulates to the consumer's sustainable rate. This matters because it lets you safely process large or even unbounded streams end-to-end without OOM or data loss, which a naive asynchronous approach can't guarantee. It's the same principle as a bounded BlockingQueue providing back-pressure between producer and consumer threads, but built into every stage of the async pipeline — and it's the capability that makes reactive genuinely differentiated for streaming workloads, even compared to virtual threads.

Q4 (mid): What's the one rule you must never break in reactive code, and why?

Never perform a blocking operation on an event-loop (reactive) thread. The reactive model runs on a very small number of event-loop threads, each multiplexing many connections by moving on to other work whenever a request awaits I/O. If you block one of those threads — with Thread.sleep, blocking JDBC, a synchronous HTTP client like RestTemplate, a .block() call, or any blocking library — that thread can no longer service the other connections it was juggling, so throughput collapses and latency spikes for all of them; blocking a few event-loop threads can effectively take the service down. This is why reactive is all-or-nothing: to get the benefit, your entire stack must be non-blocking — R2DBC instead of JDBC for the database, WebClient instead of RestTemplate for HTTP, reactive drivers for every external system — because a single blocking call anywhere in the request path poisons the whole model. If you genuinely must call blocking code (e.g. a library with no reactive alternative), you isolate it by scheduling it on a separate thread pool with subscribeOn(Schedulers.boundedElastic()) so the event loop stays free, but frequently needing that is a strong signal reactive isn't the right fit for the app. This blocking pitfall is both the most common source of reactive bugs and a key reason the paradigm is hard to adopt incrementally.

Q5 (senior): Given virtual threads, when would you still choose reactive/WebFlux?

Rarely for plain scalability, and deliberately for its unique strengths. Virtual threads (Loom, Java 21) changed the default because they deliver much of reactive's core benefit — handling massive I/O-bound concurrency without a thread-per-request bottleneck — while letting you keep the simple, blocking, thread-per-request programming model: a blocking call on a virtual thread unmounts from its carrier OS thread rather than tying it up, so millions of cheap virtual threads can be blocked on I/O concurrently, and you write ordinary, debuggable, sequential code. Since reactive's biggest costs are its steep learning curve and notoriously difficult debugging (stack traces don't reflect logical flow) plus the all-or-nothing non-blocking requirement, virtual threads eliminate most of the reason people previously adopted WebFlux ("we need to scale an I/O-bound service"). So for the common high-concurrency CRUD/API service, I'd choose Spring MVC with virtual threads. I'd still choose reactive/WebFlux when I genuinely need its differentiators: (1) streaming with real backpressure — processing large or unbounded data where a slow consumer must throttle a fast producer end-to-end, which virtual threads don't provide; (2) rich async composition — complex pipelines fanning out to many sources, merging, retrying, timing out, and transforming streams, where Reactor's operators are genuinely expressive; (3) an already fully-reactive ecosystem/stack where the model is consistent top-to-bottom; and (4) server-sent events / streaming responses. The senior stance is that reactive is a specialized tool for streaming and backpressure-heavy, compositional async workloads, not the default answer to "we need concurrency" — post-Loom, that default is virtual threads, and reaching for reactive should be justified by its specific strengths against its real complexity cost. Being able to articulate that shift — and to say "I would not go reactive here" for a typical service — is itself the seniority signal.

Q6 (senior): Why is reactive "all-or-nothing," and what does that imply for adoption?

It's all-or-nothing because the entire performance model rests on never blocking the few event-loop threads, and a single blocking call anywhere in the request path breaks that. Since those threads multiplex many connections, blocking one stalls all the connections it was serving, so if any layer — the database access, an HTTP call to another service, a logging or serialization library, a security check — performs a blocking operation, the benefit isn't just diminished, it's inverted: you get a system that's more complex than MVC and performs worse under load. Therefore going reactive means committing the whole stack to non-blocking: R2DBC (not JDBC) for the database, WebClient (not RestTemplate) for HTTP, reactive clients for caches/queues/everything, and being careful that every library in the path is non-blocking or properly offloaded to a boundedElastic scheduler. The implications for adoption are significant: you can't easily go "partially reactive," incremental migration is hard, and you're constrained to the subset of the ecosystem that has reactive support, with more involved testing, tracing, and debugging throughout. This whole-stack commitment, combined with the steep learning curve and difficult debugging, is why reactive adoption is a major architectural decision rather than a local optimization — and why, now that virtual threads deliver comparable I/O scalability with ordinary blocking code and no such constraint, the bar for choosing reactive has risen: you take on the all-or-nothing commitment only when you specifically need streaming/backpressure or rich async composition that justifies it. Recognizing that the constraint is structural (not a matter of being careful) and that it drives a whole-stack, hard-to-reverse commitment is exactly the architectural judgment a senior interview is probing.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Mono vs Flux. Which return type for "fetch one user by id" vs "stream all active users"?
Solution

Mono<User> for the single lookup (0..1), Flux<User> for the stream (0..N). Both are lazy publishers; the framework subscribes when writing the response.

2 (easy): Spot the disaster. Why is Thread.sleep(1000) in a WebFlux handler catastrophic?
Solution

It blocks an event-loop thread, which was multiplexing many connections — all of them stall. With few such threads, blocking a couple collapses throughput. Never block a reactive thread; offload unavoidable blocking to boundedElastic.

3 (medium): Fix a blocking call. A WebFlux service must call a legacy blocking library. How do you keep the event loop free?
Solution

Wrap it: Mono.fromCallable(() -> blockingCall()).subscribeOn(Schedulers.boundedElastic()), running the blocking work on a dedicated pool so event-loop threads aren't blocked. But needing this a lot means reactive may be the wrong fit.

4 (medium): Backpressure. Explain what breaks without backpressure when a fast producer feeds a slow consumer.
Solution

The consumer must buffer the excess (unbounded → OOM) or drop data. Backpressure lets the subscriber request N items and the publisher send at most N, so the pipeline self-throttles to the consumer's rate — enabling safe streaming of large/unbounded data.

5 (hard): Reactive or virtual threads? For a high-concurrency service that fans out to 3 REST APIs per request, argue the choice.
Solution

Both scale the I/O concurrency. Virtual threads let you write simple blocking fan-out code that's easy to debug — usually the better choice now. Reactive wins if you also need streaming/backpressure or rich async composition (merging, retry/timeout per call). Default to virtual threads unless those differentiators apply.

6 (hard): Explain-it test. Out loud: (a) the thread-per-request problem and how WebFlux solves it; (b) the never-block rule and all-or-nothing; (c) why virtual threads change the default.
Self-check rubric

Pass (a) if you said blocked threads exhaust the pool under I/O concurrency, and few event-loop threads multiplex many connections non-blockingly. Pass (b) if you said blocking a reactive thread stalls all its connections, so the whole stack must be non-blocking. Pass (c) if you said Loom gives the scalability with simple blocking code, so reactive is now for streaming/backpressure. That's the reactive verdict.

🎯 Key Takeaways

  • Reactive targets massive I/O-bound concurrency — few event-loop threads serve many connections non-blockingly (not faster per request).
  • Mono (0..1) and Flux (0..N) are lazy streams — nothing runs until subscribed.
  • Backpressure (slow consumer throttles fast producer) is the defining feature — safe streaming of unbounded data.
  • Never block an event-loop thread — reactive is all-or-nothing (R2DBC, WebClient, reactive everywhere).
  • It's complex and hard to debug — and virtual threads now give most of the scalability with simple blocking code, so choose reactive deliberately for streaming/backpressure.
