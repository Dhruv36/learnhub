REST Controllers Basics

The web layer is where Spring Boot earns its keep. A @RestController maps HTTP requests to Java methods, binds path/query/body data automatically, and serializes return values to JSON via Jackson. The skill here is drawing clean boundaries: thin controllers, DTOs at the edge, correct status codes.

A Full CRUD Controller

@RestController
@RequestMapping("/api/users")          // base path for every method here
public class UserController {
    private final UserService service;
    public UserController(UserService s) { this.service = s; }   // constructor injection

    @GetMapping                                          // GET /api/users?role=ADMIN
    public List<UserDto> list(@RequestParam(required=false) Role role) {
        return service.list(role);
    }
    @GetMapping("/{id}")                                 // GET /api/users/42
    public UserDto get(@PathVariable long id) {
        return service.find(id).orElseThrow(() -> new NotFoundException(id));
    }
    @PostMapping                                         // POST /api/users
    @ResponseStatus(HttpStatus.CREATED)                  // → 201, not the default 200
    public UserDto create(@Valid @RequestBody CreateUserRequest req) {
        return service.create(req);
    }
    @PutMapping("/{id}")                                 // PUT /api/users/42
    public UserDto update(@PathVariable long id, @Valid @RequestBody UpdateUserRequest req) {
        return service.update(id, req);
    }
    @DeleteMapping("/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)               // → 204
    public void delete(@PathVariable long id) { service.delete(id); }
}

🧱 Use DTOs/records at the boundary — never expose JPA entities. A record like record UserDto(long id, String name, String email) {} gives you exact control over the JSON contract, decouples your API from the database schema (a column rename stays internal), and avoids lazy-loading serialization errors. This is the same idea as FastAPI's response_model or a serializer — shape input and output explicitly.

ResponseEntity — Full Control

// Return type UserDto → 200 + JSON body. When you need headers or a computed status, use ResponseEntity:
@PostMapping
public ResponseEntity<UserDto> create(@Valid @RequestBody CreateUserRequest req) {
    UserDto created = service.create(req);
    URI location = URI.create("/api/users/" + created.id());
    return ResponseEntity.created(location).body(created);   // 201 + Location header
}
// Rule of thumb: return the DTO directly for the simple case; reach for ResponseEntity
// only when you need headers, a conditional status, or an empty body with a specific code.

@CrossOrigin(origins = "https://app.example.com")   // per-controller

Interview Questions

Q1: Difference between @Controller and @RestController?

@RestController is @Controller + @ResponseBody on every method. A plain @Controller method returns a view name that a template engine resolves (server-rendered HTML). @RestController instead serializes the return value directly into the response body (JSON via Jackson by default). For REST APIs you want @RestController; a forgotten @ResponseBody on a plain @Controller is why "my endpoint returns a 404 for a view template" happens.

Q2: Why map entities to DTOs instead of returning them directly?

Returning JPA entities couples your public API contract to your persistence schema — a column rename becomes a breaking API change. It can leak sensitive/internal fields, and serializing lazily-loaded associations outside a transaction throws LazyInitializationException or fires surprise N+1 queries. Dedicated DTOs/records let you expose exactly the fields you intend, evolve the API independently of the schema, and sidestep the lazy-loading traps. Input DTOs also let validation and API-shape differ from the entity (e.g. a create request without an id).

Q3: When do you use ResponseEntity vs returning the object?

Return the DTO directly for the common case — the return value becomes the body and the status is 200 (or whatever @ResponseStatus declares). Reach for ResponseEntity when a single endpoint must vary its status at runtime (200 vs 304), set response headers (a Location on create, caching headers), or return an empty body with a specific code. It's more verbose, so don't use it reflexively; use @ResponseStatus for a fixed non-200 and ResponseEntity only when the response metadata is dynamic.

🏋️ Practice: build a @RestController for a Book resource with full CRUD; use records as request/response DTOs; return 201 + Location on create and 204 on delete; add a @RequestParam filter to the list endpoint; wire it to a @Service so the controller holds zero business logic; verify each status code with curl or Postman.

🎯 Key Takeaways

  • @RestController + @RequestMapping map HTTP to methods; Jackson handles JSON both ways.
  • Bind data with @PathVariable, @RequestParam, @RequestBody, @RequestHeader.
  • Keep controllers thin — delegate all logic to a @Service.
  • Use DTOs/records at the boundary; never expose JPA entities.
  • Return correct status codes (201/204/400/404); use ResponseEntity only for dynamic headers/status.
