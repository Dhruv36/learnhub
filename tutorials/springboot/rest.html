REST Controllers Web Layer

What you'll master in this lesson: how a @RestController turns HTTP requests into Java method calls and Java return values into JSON; the layered architecture that keeps controllers thin (controller → service → repository); why you expose DTOs/records at the boundary and never JPA entities; how to choose the right HTTP status code; and how ResponseEntity gives you full control when you need it. This is the web layer's foundation — request/response mechanics come next; here we establish the shape of a well-designed REST API.

Part 1 — A Full CRUD Controller

@RestController                        // = @Controller + @ResponseBody (returns data, not view names)
@RequestMapping("/api/users")          // base path for every method here
public class UserController {
    private final UserService service;
    public UserController(UserService s) { this.service = s; }   // constructor injection (IoC lesson)

    @GetMapping                                          // GET /api/users?role=ADMIN
    public List<UserDto> list(@RequestParam(required=false) Role role) {
        return service.list(role);
    }
    @GetMapping("/{id}")                                 // GET /api/users/42
    public UserDto get(@PathVariable long id) {
        return service.find(id).orElseThrow(() -> new NotFoundException(id));
    }
    @PostMapping                                         // POST /api/users
    @ResponseStatus(HttpStatus.CREATED)                  // → 201, not the default 200
    public UserDto create(@Valid @RequestBody CreateUserRequest req) {
        return service.create(req);
    }
    @PutMapping("/{id}")                                 // PUT /api/users/42
    public UserDto update(@PathVariable long id, @Valid @RequestBody UpdateUserRequest req) {
        return service.update(id, req);
    }
    @DeleteMapping("/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)               // → 204
    public void delete(@PathVariable long id) { service.delete(id); }
}

Part 2 — How @RestController Works

@RestController is @Controller + @ResponseBody: every method's return value is serialized directly into the HTTP response body (JSON via Jackson by default), rather than being treated as a view name to render. Behind the scenes, Spring's DispatcherServlet (auto-configured by the web starter) routes an incoming request to the matching handler method, resolves its parameters from the request, invokes it, and serializes the return value.

🔑 The distinction that trips people up: a plain @Controller method returns a view name that a template engine resolves into server-rendered HTML. @RestController instead writes the return value as the response body. A forgotten @ResponseBody on a plain @Controller is why "my endpoint returns a 404 for a view template" happens. For REST APIs you always want @RestController.

Part 3 — The Layered Architecture (thin controllers)

@RestController                    // WEB layer — HTTP concerns only
class UserController {
    private final UserService service;
    @PostMapping UserDto create(@Valid @RequestBody CreateUserRequest r) { return service.create(r); }
    // thin: bind input, delegate, return output. NO business logic here.
}
@Service                           // BUSINESS layer — the actual logic
class UserService {
    private final UserRepository repo;
    UserDto create(CreateUserRequest r) { /* rules, validation, orchestration */ }
}
@Repository                        // DATA layer — persistence
interface UserRepository extends JpaRepository<User, Long> { }
🎯 Keep controllers thin — delegate all logic to a @Service. A controller's job is HTTP: bind the request, call the service, map the result to a response with the right status. Business rules, orchestration, and transactions belong in the service; persistence in the repository. This is the Single Responsibility Principle (Java SOLID lesson) applied as layers: each layer changes for one reason, each is independently testable (you can test the service with no HTTP, the controller with a mocked service — the Testing lesson), and the structure is the standard every Spring developer recognizes.

Part 4 — DTOs at the Boundary (never expose entities)

// The request/response shapes are DTOs (ideally records — Java Records lesson), NOT JPA entities:
public record CreateUserRequest(@NotBlank String name, @Email String email) {}
public record UserDto(long id, String name, String email) {}   // exactly what the API exposes

@PostMapping
public UserDto create(@Valid @RequestBody CreateUserRequest req) {
    User saved = service.create(req);          // User is the JPA ENTITY (internal)
    return new UserDto(saved.getId(), saved.getName(), saved.getEmail());  // map to the DTO
}
// ❌ Returning the User entity directly leaks internals and causes lazy-loading errors.
🧱 Use DTOs/records at the boundary — never expose JPA entities. Three reasons: (1) it decouples your API contract from your database schema, so a column rename stays internal instead of becoming a breaking API change; (2) it prevents leaking sensitive/internal fields; (3) it avoids serializing lazily-loaded JPA relationships outside a transaction, which throws LazyInitializationException or triggers surprise N+1 queries (the JPA lessons). Input DTOs also let you shape and validate the request independently of the entity (e.g. a create request with no id). This is the same "control your input/output shape" idea as a serializer or FastAPI's response model.

Part 5 — Status Codes & ResponseEntity

SituationStatus
GET / PUT success with body200 OK
Resource created201 Created (+ Location header)
Success, no body (DELETE)204 No Content
Invalid/malformed input400 Bad Request
Not authenticated / not allowed401 / 403
Resource missing404 Not Found
Conflict (duplicate, version)409 Conflict
// Return the DTO directly → 200 (or whatever @ResponseStatus declares). Use ResponseEntity when
// you need dynamic status or response headers:
@PostMapping
public ResponseEntity<UserDto> create(@Valid @RequestBody CreateUserRequest req) {
    UserDto created = service.create(req);
    URI location = URI.create("/api/users/" + created.id());
    return ResponseEntity.created(location).body(created);   // 201 + Location header
}
// Rule of thumb: return the DTO for the simple case; reach for ResponseEntity only when you need
// headers, a computed status, or an empty body with a specific code.

Part 6 — Deep Dive: What Makes a REST API Well-Designed

  • Resource-oriented URLs, correct HTTP verbs. Nouns for resources (/api/users/{id}), verbs conveyed by the HTTP method (GET read, POST create, PUT/PATCH update, DELETE remove) — not /api/getUser. This makes the API predictable and cacheable.
  • Meaningful status codes. The status is part of the contract: 201 vs 200 tells a client a resource was created, 404 vs 400 distinguishes "not found" from "bad request," 409 signals a conflict. Returning 200 for everything forces clients to parse bodies to detect failure.
  • A stable contract via DTOs. The API's shape is defined by DTOs you control, decoupled from the database, so persistence can evolve without breaking clients — and clients see exactly the fields you intend.
  • Thin controllers, layered design. HTTP concerns stay in the controller; logic in the service; persistence in the repository — testable, changeable, and the structure everyone expects.
🔍 Senior framing: a REST controller is a thin translation layer between the HTTP world and your domain, and getting its design right is mostly about discipline rather than framework tricks. The controller binds the request, delegates to a service that owns the business logic, and maps the result to a response with an accurate status code — no rules, no persistence, no cross-cutting logic living in the controller. The single most consequential habit is exposing DTOs/records at the boundary instead of JPA entities: it decouples the API contract from the schema (so you can refactor persistence freely), prevents leaking internal or sensitive fields, and sidesteps the lazy-loading and N+1 traps that plague entity serialization — a mistake so common it's almost a rite of passage, and one interviewers reliably probe. Combined with resource-oriented URLs, correct verbs, and honest status codes, this produces an API that is predictable, evolvable, and independently testable at each layer. The framework makes the mechanics trivial (annotations map HTTP to methods, Jackson handles JSON both ways); the value you add as an engineer is the architecture — thin controllers, a clean service layer, a stable DTO contract, and correct HTTP semantics — which is exactly what distinguishes a maintainable service from one where business logic, persistence, and HTTP concerns are tangled in the controller.

Interview Questions

Q1 (beginner): What's the difference between @Controller and @RestController?

@RestController is @Controller plus @ResponseBody on every method, meaning each method's return value is serialized directly into the HTTP response body (JSON via Jackson by default). A plain @Controller method returns a view name that a template engine resolves into server-rendered HTML. So for REST APIs you use @RestController; a forgotten @ResponseBody on a plain @Controller is why an endpoint mistakenly tries to resolve a view and 404s. In short: @RestController returns data, @Controller returns views.

Q2 (beginner): Why should controllers be thin?

Because a controller's responsibility is HTTP concerns only — binding the request, delegating, and mapping the result to a response with the right status. Putting business logic in the controller tangles layers, makes the logic hard to test (you'd need HTTP to exercise it), and violates the Single Responsibility Principle. Keeping controllers thin and delegating to a @Service means the business logic is independently testable (no HTTP needed), reusable across entry points, and cleanly separated from web details — the standard layered architecture (controller → service → repository).

Q3 (mid): Why map entities to DTOs instead of returning them directly?

Several reasons. Returning JPA entities couples your public API contract to your database schema, so a column rename becomes a breaking API change; it can leak sensitive or internal fields you didn't intend to expose; and serializing lazily-loaded associations outside a transaction throws LazyInitializationException or triggers surprise N+1 queries. Dedicated DTOs/records let you expose exactly the fields you intend, evolve the API independently of persistence, and sidestep the lazy-loading traps. Input DTOs also let the request shape and validation differ from the entity (for example a create request with no id). It's the same principle as controlling output with a serializer or response model.

Q4 (mid): When do you use ResponseEntity vs returning the object directly?

Return the DTO directly for the common case — the return value becomes the response body and the status is 200 (or whatever @ResponseStatus declares). Reach for ResponseEntity when a single endpoint must vary its status at runtime (e.g. 200 vs 304), set response headers (a Location header on create, caching headers), or return an empty body with a specific code. It's more verbose, so don't use it reflexively: use @ResponseStatus for a fixed non-200 status and ResponseEntity only when the response metadata (status or headers) is dynamic.

Q5 (senior): What are the principles of a well-designed REST API, and how do you enforce them in Spring?

A well-designed REST API is resource-oriented and honest about HTTP semantics. URLs name resources with nouns (/api/users/{id}) and actions are expressed by HTTP verbs — GET (safe, idempotent read), POST (create), PUT (full idempotent replace), PATCH (partial update), DELETE (idempotent removal) — rather than verbs in the path. Status codes are part of the contract and must be accurate: 201 with a Location header for creation, 204 for a successful no-body operation, 400 for bad input, 401/403 for auth, 404 for missing resources, 409 for conflicts — so clients can react to outcomes without parsing bodies. The API's data shape is defined by DTOs you control and version, decoupled from the persistence model, so the schema can evolve without breaking clients and internal fields never leak. In Spring you enforce these with the mapping annotations (@GetMapping/@PostMapping/etc. on noun-based @RequestMapping paths), @ResponseStatus/ResponseEntity for correct codes and headers, records as request/response DTOs mapped to/from entities in the service layer, and a layered structure that keeps HTTP concerns in thin controllers, business logic in services, and persistence in repositories. Cross-cutting concerns — validation (@Valid → 400), consistent error bodies (a @RestControllerAdvice), auth (Spring Security), pagination, and content negotiation — round it out, each handled at the appropriate layer rather than scattered through controllers. The senior point is that Spring makes the mechanics trivial, so the engineering value is in the design discipline: predictable resource URLs, correct verbs and status codes, a stable DTO contract, and clean layering, which together produce an API that clients can rely on and that the team can evolve safely.

Q6 (senior): How does a request flow from the network to your controller method and back, and why does understanding it help?

When the embedded server (auto-configured Tomcat) receives an HTTP request, it hands it to Spring's DispatcherServlet — the front controller the web auto-configuration registers. The DispatcherServlet consults handler mappings to find the controller method whose @RequestMapping/@GetMapping/etc. (path, HTTP method, and any consumes/produces constraints) matches the request. It then uses argument resolvers to build each method parameter from the request — resolving @PathVariable from the URL, @RequestParam from the query string, @RequestBody by deserializing the body via an HttpMessageConverter (Jackson for JSON), @RequestHeader from headers — running validation when @Valid is present, and invokes your method. The return value is then processed by return-value handlers: for a @ResponseBody/@RestController method the object is serialized to the response body by a message converter (again Jackson for JSON), with the status from @ResponseStatus/ResponseEntity, and if an exception is thrown it's routed to exception handlers (a @RestControllerAdvice / the default handler). Understanding this pipeline helps because it explains where things can be customized and where they go wrong: content-negotiation and serialization issues (registering or configuring message converters/ObjectMapper), why a 404 happens (no handler mapping matched — wrong path, verb, or missing @RestController), why binding fails or a 400 is returned (argument resolution/validation), how @RequestBody deserialization and @ResponseBody serialization work (message converters), how filters and the security chain wrap the whole thing, and where exception handling plugs in. So the DispatcherServlet-centric mental model — mapping → argument resolution → invoke → return-value handling / exception handling, all around message converters — turns the "annotations magically produce JSON" experience into a debuggable, extensible pipeline, which is exactly what lets a senior developer diagnose web-layer problems and customize serialization, binding, and error handling with confidence.

🏋️ Exercises (attempt before opening solutions)

1 (easy): CRUD controller. Build a @RestController for a Book resource with GET (all + by id), POST, PUT, DELETE, returning correct status codes and delegating to a service.
Solution

@RestController @RequestMapping("/api/books") with @GetMapping, @GetMapping("/{id}"), @PostMapping (@ResponseStatus(CREATED)), @PutMapping("/{id}"), @DeleteMapping("/{id}") (@ResponseStatus(NO_CONTENT)), each delegating to a BookService. The controller holds no logic — it binds, delegates, and returns with the right status.

2 (easy): Controller vs RestController. Take a method returning a String, annotate the class @Controller, and observe the "view not found" behavior; then switch to @RestController.
Solution

With @Controller, returning "hello" makes Spring look for a view named "hello" (and 404/error if none exists). With @RestController (or adding @ResponseBody), the String is written as the response body. This demonstrates the view-vs-body distinction concretely.

3 (medium): DTO mapping. Given a User JPA entity, define a UserDto record and a CreateUserRequest record, and map between them in the service/controller so the entity never leaves the app.
Solution

record CreateUserRequest(String name, String email) as input, record UserDto(long id, String name, String email) as output; the controller/service builds a User from the request, saves it, and maps the saved entity to a UserDto for the response. The entity stays internal — no schema coupling, no lazy-loading leaks, and the API exposes exactly the intended fields.

4 (medium): Correct status codes. Make create return 201 with a Location header (via ResponseEntity), delete return 204, and a missing resource return 404. Verify with curl.
Solution

Create: return ResponseEntity.created(URI.create("/api/books/" + saved.id())).body(dto); → 201 + Location. Delete: @ResponseStatus(HttpStatus.NO_CONTENT) → 204. Missing: throw a NotFoundException mapped to 404 (Error Handling lesson). Each status communicates the outcome so clients don't parse bodies to detect success/failure.

5 (hard): Layered refactor. Given a fat controller that validates, computes, persists, and formats, refactor it into thin controller + service + repository, and explain what each layer now owns and how you'd test each.
Solution

The controller keeps only HTTP concerns (bind @Valid @RequestBody, call the service, return with a status); the service owns business rules, orchestration, and (with @Transactional) transactions; the repository owns persistence. Testing: the service with plain unit tests using a mocked repository (no Spring), the controller with a slice test (@WebMvcTest + MockMvc) mocking the service, the repository with a data slice test against a real DB (Testing lesson). Each layer changes for one reason and is independently testable — the payoff of thin controllers.

6 (hard): Explain-it test. Out loud: (a) @RestController vs @Controller and the layered architecture; (b) why DTOs at the boundary; (c) status-code choices and when ResponseEntity is warranted.
Self-check rubric

Pass (a) if you said RestController returns data (not views) and controllers stay thin, delegating to services. Pass (b) if you named schema decoupling, no field leaks, and no lazy-loading/N+1. Pass (c) if you mapped 201/204/400/404/409 to situations and reserved ResponseEntity for dynamic headers/status. These set up request mapping, validation, and error handling.

🎯 Key Takeaways

  • @RestController (= @Controller + @ResponseBody) serializes return values to JSON via Jackson; @Controller returns view names.
  • Keep controllers thin — bind, delegate to a @Service, return; logic in the service, persistence in the repository (layered SRP).
  • Expose DTOs/records at the boundary, never JPA entities — decouples the API from the schema and avoids field leaks and lazy-loading/N+1 errors.
  • Return correct status codes (201 + Location, 204, 400, 404, 409) — the status is part of the contract.
  • Return the DTO directly for the simple case; use ResponseEntity only for dynamic status/headers.
  • A request flows through the DispatcherServlet (mapping → argument resolution → invoke → serialize) — a debuggable pipeline, not magic.
