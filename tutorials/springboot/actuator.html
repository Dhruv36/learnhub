Actuator & Observability Production

What you'll master in this lesson: how to make a Spring Boot service operable in production — Actuator endpoints (and how to secure them), health checks wired to load balancers and Kubernetes liveness/readiness probes, metrics via Micrometer (counters, gauges, timers) exported to Prometheus, and structured logging and distributed tracing — the three pillars (metrics, logs, traces) that let you answer "is it healthy, how is it behaving, and why is this one request slow?" Prerequisites: the track's app model; ties into the same production concerns as the Node.js and Docker/Kubernetes tracks.

Part 1 — Enabling & Securing Actuator

<!-- add the dependency: spring-boot-starter-actuator -->

# application.yml — endpoints are LOCKED DOWN by default; expose ONLY what you need:
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus     # explicit allow-list, not "*"
  endpoint:
    health:
      show-details: when-authorized                 # don't leak component internals to anonymous callers
  server:
    port: 9000                                      # bind actuator to a SEPARATE management port (not public)
EndpointGives you
/actuator/healthUP/DOWN + component health (db, disk, custom indicators)
/actuator/infoBuild / version / git info
/actuator/metricsJVM, HTTP, datasource metrics (browsable)
/actuator/prometheusAll metrics in Prometheus scrape format
/actuator/env, /loggers, /conditions, /heapdumpConfig, live log-level control, auto-config report, heap dump
🔒 Actuator is powerful and therefore dangerous — secure its surface. Endpoints like /env, /heapdump, and /loggers expose configuration (including possibly secrets), memory contents, and let callers change log levels at runtime. They're locked down by default; keep them that way by exposing the minimum explicit list (never include: "*" in production), putting them behind authentication (Spring Security), setting health.show-details: when-authorized so anonymous callers don't see internals, and ideally binding actuator to a separate management port that isn't reachable from the public internet. A leaked /heapdump or /env is a real security incident.

Part 2 — Health Checks for Load Balancers & Kubernetes

# Kubernetes probes hit dedicated liveness/readiness groups (Boot maps these automatically):
#   /actuator/health/liveness   — is the app ALIVE? (if DOWN, K8s RESTARTS the pod)
#   /actuator/health/readiness  — is it READY for traffic? (if DOWN, K8s removes it from the Service/LB)
management.endpoint.health.probes.enabled: true

// A custom health indicator for a downstream dependency:
@Component
class PaymentGatewayHealth implements HealthIndicator {
    private final PaymentGateway gateway;
    PaymentGatewayHealth(PaymentGateway g) { this.gateway = g; }
    public Health health() {
        return gateway.ping()
            ? Health.up().build()
            : Health.down().withDetail("gateway", "unreachable").build();   // shows in /health, flips it DOWN
    }
}
🎯 Liveness ≠ readiness, and confusing them causes outages. Liveness answers "is the process wedged and unrecoverable?" — if DOWN, the orchestrator restarts the pod. Readiness answers "can it serve requests right now?" — if DOWN, the orchestrator stops routing traffic to it but leaves it running. The dangerous mistake is putting a dependency check (e.g. the database) in the liveness probe: when the DB blips, every pod's liveness goes DOWN, Kubernetes restarts them all simultaneously, and a transient dependency outage becomes a full self-inflicted outage. Dependency health belongs in readiness (stop sending traffic, don't restart); liveness should reflect only the app's own internal wedged-ness. Custom HealthIndicators let you surface a downstream's status in /health.

Part 3 — Metrics with Micrometer

// Micrometer is "SLF4J for metrics" — one vendor-neutral API, many backends (Prometheus, Datadog, CloudWatch).
@Service
class OrderService {
    private final Counter placed;
    private final Timer   checkout;
    OrderService(MeterRegistry reg) {                       // inject the registry, create meters once
        this.placed   = reg.counter("orders.placed");
        this.checkout = reg.timer("orders.checkout.duration");
    }
    public void place(Order o) {
        checkout.record(() -> doCheckout(o));               // times the block (records duration + count)
        placed.increment();                                 // monotonic total
    }
}
// HTTP request metrics (http.server.requests: count, latency percentiles, status) come FREE — no code.
Meter typeMeasuresExample
CounterMonotonic totals (only goes up)requests, errors, orders placed
GaugePoint-in-time value (up and down)queue depth, cache size, active connections
TimerDuration + rate (latency)checkout time, external-call latency
Distribution summaryDistribution of sizes/amountsrequest payload bytes, order value
📊 Micrometer decouples your code from the monitoring vendor — you record to its API and switch backends (Prometheus, Datadog, CloudWatch) by changing a dependency, not your code. Boot auto-instruments a lot for free (HTTP request count/latency/status, JVM memory and GC, datasource pool, cache hit ratios), so /actuator/prometheus is immediately useful. Add domain meters for what the business cares about (orders placed, checkout latency, payment failures). Watch tag cardinality: a tag with unbounded values (a user id, a full URL with path variables) creates a separate time series per value and can explode your metrics store — tag by bounded dimensions (status, endpoint template, region) only.

Part 4 — Structured Logging & the Three Pillars

# Log as JSON in prod so aggregators (ELK, Loki) can index fields instead of grepping text:
logging.structured.format.console: ecs        # Boot 3.4+ built-in structured logging

// Distributed tracing: add Micrometer Tracing + an exporter (OTLP/Zipkin). Every request gets a
// traceId/spanId that PROPAGATES across service calls and into the logs, so you can follow ONE
// request through the whole system and see exactly which hop was slow or failed.
PillarAnswersSpring tooling → backend
MetricsIs it healthy? What are the trends/rates?Micrometer → Prometheus → Grafana + alerts
LogsWhat exactly happened for this event?Structured (JSON) logs → Loki / ELK
TracesWhy is this request slow / where did it fail?Micrometer Tracing → Tempo / Jaeger
📊 The three pillars answer different questions and you need all three. Metrics tell you that something is wrong and show trends cheaply (aggregate, low cardinality); logs give the detail of a specific event (structured JSON so fields are queryable, not grep-only); traces follow one request across services via a propagated traceId, revealing which hop caused the latency or error. Correlating them — a Grafana alert on a metric → drill into traces for the slow requests → read the structured logs for those exact trace ids — is how you go from "p99 latency spiked" to "the payment service's DB call is timing out" in minutes. Wire this once and every service is operable the same way, the same production discipline the Node.js and Kubernetes tracks describe.

Part 5 — /info, Live Log Levels & Graceful Signals

# /actuator/info — surface build + git metadata (add the git-commit-info + build-info plugins):
management.info.git.mode: full
management.info.env.enabled: true

# Change a log level at RUNTIME without redeploying (great for debugging prod):
#   POST /actuator/loggers/com.myapp  {"configuredLevel": "DEBUG"}

# Graceful shutdown so in-flight requests finish on SIGTERM (K8s rolling deploy):
server.shutdown: graceful
spring.lifecycle.timeout-per-shutdown-phase: 30s
🔧 Actuator isn't just monitoring — it's live operability. /actuator/loggers lets you raise a package to DEBUG at runtime to diagnose a production issue and drop it back, with no redeploy. /info exposes exactly which build/git commit is running (invaluable when a bug report says "it broke after the last deploy"). And server.shutdown: graceful makes the app, on receiving SIGTERM during a rolling deploy, stop accepting new requests but let in-flight ones finish before exiting — combined with the readiness probe flipping DOWN first, this gives zero-dropped-requests deployments.

Part 6 — Deep Dive: Operability Is a Feature

  • You can't operate what you can't see. A service without health, metrics, and traces is a black box — when it misbehaves at 3am you have nothing to go on. Observability is what turns "it's slow, I don't know why" into a five-minute diagnosis.
  • Health checks must distinguish liveness from readiness. Restart-on-liveness and remove-from-LB-on-readiness are different responses; conflating them (dependency checks in liveness) turns transient outages into cascading restarts.
  • Instrument the business, not just the JVM. Free JVM/HTTP metrics are table stakes; the meters that matter for a real incident are domain ones (orders/sec, payment failure rate, queue depth) — they're what your alerts and dashboards should center on.
  • The management surface is an attack surface. Expose the minimum, authenticate it, and isolate it on a management port — a public /heapdump or /env leaks secrets and memory.
🔍 Senior framing: observability is not an add-on you bolt onto a service once it's misbehaving — it's a feature you design in, because a production service's real job includes being diagnosable and operable, and Spring Boot Actuator plus Micrometer give you the whole stack cheaply if you configure it deliberately. The health model is where seniors are separated from juniors: liveness and readiness are different questions with different consequences — liveness failing restarts the pod, readiness failing just removes it from the load balancer — so dependency checks (database, downstream service) belong in readiness (stop sending traffic, don't kill the process), and putting them in liveness turns a transient DB blip into a fleet-wide restart storm, a classic self-inflicted outage. Metrics via Micrometer's vendor-neutral API give you trends and alerting; the free JVM/HTTP instrumentation is table stakes, but the meters that matter in an incident are the domain ones (orders placed, checkout latency, payment-failure rate, queue depth), and you must watch tag cardinality so an unbounded tag doesn't explode the metrics store. Structured logs make event detail queryable rather than grep-only, and distributed tracing ties a single request's journey across services together via a propagated trace id — and the real power is correlating all three: an alert on a metric leads you to the slow traces, which point at the exact log lines, collapsing "p99 spiked" into a root cause in minutes. Finally, the management surface is a genuine security concern — /env, /heapdump, and /loggers expose secrets, memory, and runtime control — so it must be minimized, authenticated, and ideally isolated on a non-public management port. The mature engineer builds every service so that from day one it answers "is it up, how is it behaving, and why is this request slow," treats liveness/readiness and the three pillars as deliberate design, and can explain in an interview exactly why a database check in a liveness probe is a production landmine — which is precisely the kind of operational judgment senior interviews probe.

Interview Questions

Q1 (beginner): What is Spring Boot Actuator and what does it give you?

Actuator adds production-ready operational endpoints to a Spring Boot app under /actuator: a health endpoint (UP/DOWN plus component and custom health), metrics (JVM, HTTP, datasource, and your own via Micrometer, including a Prometheus scrape format), build/git info, live log-level control, environment and auto-configuration reports, and more. It makes a service operable — you can check if it's healthy, see how it's behaving, and diagnose it — without writing that plumbing yourself. You add spring-boot-starter-actuator and then explicitly expose the endpoints you want (they're locked down by default), because the surface includes sensitive endpoints like /env and /heapdump that must be secured.

Q2 (beginner): What's the difference between a liveness and a readiness probe?

Both are health checks an orchestrator (Kubernetes) polls, but they trigger different actions. Liveness answers "is the application alive, or is it wedged in an unrecoverable state?" — if it reports DOWN, the orchestrator restarts the pod. Readiness answers "is the application ready to serve traffic right now?" — if it reports DOWN, the orchestrator stops routing traffic to it (removes it from the load balancer / Service) but leaves the process running so it can recover. Spring Boot exposes these as /actuator/health/liveness and /actuator/health/readiness. The key rule is that dependency checks (database, downstream services) belong in readiness, not liveness — so a transient dependency outage takes pods out of rotation rather than restarting them all.

Q3 (mid): Why must a database health check go in readiness, not liveness?

Because the two probes cause opposite, and in this case catastrophic, responses. If the database check is in the liveness probe and the database has a transient outage (a failover, a brief network blip), every application pod's liveness simultaneously reports DOWN, so Kubernetes restarts them all at once — turning a short, recoverable dependency hiccup into a full application outage, and often making recovery slower because all pods are cold-starting and hammering the recovering database. Putting the check in readiness instead means that during the DB outage the pods are simply removed from the load balancer (they stop receiving traffic) but keep running, and as soon as the database recovers their readiness flips back UP and they rejoin rotation — no restarts, no cascade. Liveness should reflect only the app's own internal health (e.g. a deadlocked thread pool it can't recover from), where a restart is actually the right fix. This liveness/readiness distinction is one of the most common real-world production landmines and a frequent interview question precisely because getting it wrong amplifies outages.

Q4 (mid): What is Micrometer and what meter types would you use?

Micrometer is a vendor-neutral metrics facade — "SLF4J for metrics" — that lets you record metrics through one API and export them to any backend (Prometheus, Datadog, CloudWatch) by swapping a dependency, without changing your code. Boot auto-instruments HTTP requests (count, latency percentiles, status), JVM (memory, GC, threads), datasource pools, and caches for free. The core meter types: a Counter for monotonic totals that only increase (requests served, errors, orders placed); a Gauge for a point-in-time value that goes up and down (queue depth, cache size, active connections); a Timer for durations plus rate (the latency of an operation, giving you percentiles); and a Distribution Summary for the distribution of sizes/amounts (payload bytes, order value). You inject the MeterRegistry, create meters once, and record to them. The main pitfall is tag cardinality — tagging by unbounded values like a user id creates a time series per value and can overwhelm the metrics store, so tag only by bounded dimensions.

Q5 (senior): Explain the three pillars of observability and how you'd use them together to diagnose a latency spike.

The three pillars are metrics, logs, and traces, and they answer different questions. Metrics are aggregated, low-cardinality time series (via Micrometer → Prometheus) that tell you that something is wrong and show trends and rates cheaply — they're what you alert on. Logs are detailed records of individual events; making them structured (JSON, e.g. ECS format) means fields are indexed and queryable in an aggregator (Loki/ELK) rather than grep-only. Traces (via Micrometer Tracing → Tempo/Jaeger) follow a single request across service boundaries by propagating a traceId/spanId, so you can see the full path and which hop consumed the time or failed. Diagnosing a p99 latency spike, you'd chain them: a Grafana alert on the http.server.requests latency metric tells you the spike exists and roughly where (which endpoint); you then drill into traces for the slow requests on that endpoint, which show, say, that a downstream call to the payment service is taking 2s instead of 50ms; you follow that trace id into the structured logs of the payment service and see it's a database timeout on a specific query. In minutes you've gone from "p99 spiked" to "the payment service's DB call to X is timing out under load," because the three pillars are correlated (ideally sharing the trace id across all of them). The senior point is that no single pillar is sufficient — metrics without traces tell you something's slow but not why; traces without metrics don't tell you it's happening at scale; logs without structure and correlation are a needle in a haystack — so a well-instrumented service wires all three and links them, making incident response a guided investigation rather than guesswork.

Q6 (senior): Actuator is powerful — how do you keep it from becoming a security liability, and what enables zero-downtime deploys?

Actuator's power is exactly its risk: endpoints like /env (configuration, possibly including secrets), /heapdump (a full memory dump — potentially containing tokens, PII, keys), and /loggers (runtime control of logging) are dangerous if exposed. Securing them: (1) never expose "*" — use an explicit minimal allow-list of only the endpoints you need (typically health, info, metrics/prometheus); (2) put the actuator surface behind authentication with Spring Security, and set health.show-details: when-authorized so anonymous callers don't see component internals; (3) bind actuator to a separate management port (management.server.port) that is firewalled off from the public internet and only reachable by your monitoring infrastructure; (4) treat a publicly reachable /heapdump or /env as a genuine incident. For zero- downtime deploys, the enablers are graceful shutdown plus readiness: set server.shutdown: graceful so that on SIGTERM (which Kubernetes sends during a rolling update) the app stops accepting new requests but lets in-flight ones finish within a timeout before exiting, and rely on the readiness probe flipping DOWN as the pod begins shutdown so the load balancer stops routing new traffic to it before it stops accepting — together these mean no request is dropped mid-deploy. Combined with liveness handling genuine wedged states and health indicators reflecting real dependency status, you get a service that deploys, scales, and recovers cleanly under an orchestrator — which is what "production-ready" actually means and what these Actuator features are for.

4 (medium): Right probe placement. Explain where a database health check should go and what happens if it's in the wrong probe.
Solution

In readiness. If it's in liveness, a transient DB outage makes every pod's liveness DOWN, so Kubernetes restarts them all at once — amplifying a brief dependency blip into a full outage. In readiness, the pods just leave the load balancer during the outage and rejoin when the DB recovers, no restarts.

🎯 Key Takeaways

  • Actuator gives production endpoints (health, metrics, info, loggers) — expose the minimum, authenticate, isolate on a management port.
  • Liveness (DOWN → restart) ≠ readiness (DOWN → leave the LB); dependency checks go in readiness or a DB blip restarts every pod.
  • Micrometer is vendor-neutral metrics — Counter / Gauge / Timer; instrument the business, and watch tag cardinality.
  • The three pillars (metrics, logs, traces) answer different questions — correlate them to diagnose fast.
  • Use structured logging + distributed tracing (propagated traceId) and server.shutdown: graceful for clean deploys.
