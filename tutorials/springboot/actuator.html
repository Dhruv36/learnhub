Actuator & Observability Advanced

A service you can't observe is a service you can't operate. Spring Boot Actuator adds production-grade endpoints for health, metrics, and diagnostics with one starter, and Micrometer wires your metrics to Prometheus, Datadog, and friends. Observability rests on three pillars — metrics, logs, and traces — and Spring gives you all three.

Enabling Actuator

<!-- add spring-boot-starter-actuator -->
# application.yml — expose ONLY what you need (endpoints are locked down by default):
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus
  endpoint:
    health:
      show-details: when-authorized      # don't leak internals to anonymous callers
EndpointGives you
/actuator/healthUP/DOWN + component health (db, disk, custom)
/actuator/infoBuild/version/git info
/actuator/metricsJVM, HTTP, datasource metrics
/actuator/prometheusMetrics in Prometheus scrape format
/actuator/env, /loggers, /conditionsConfig, live log levels, auto-config report
🔒 Secure the actuator surface. Endpoints like /env, /heapdump, and /loggers expose config and internals — expose the minimum, protect them behind auth, and ideally bind actuator to a separate management port not reachable from the public internet.

Health Checks — for Load Balancers & Kubernetes

# Kubernetes probes hit dedicated liveness/readiness groups:
#   /actuator/health/liveness   — is the app alive? (restart if DOWN)
#   /actuator/health/readiness  — ready for traffic? (remove from LB if DOWN)
management.endpoint.health.probes.enabled: true

// A custom health indicator (e.g. a downstream dependency):
@Component
class PaymentGatewayHealth implements HealthIndicator {
    public Health health() {
        return gateway.ping() ? Health.up().build()
                              : Health.down().withDetail("gateway", "unreachable").build();
    }
}

Metrics with Micrometer

// Micrometer is "SLF4J for metrics" — one API, many backends (Prometheus, Datadog, CloudWatch).
@Service
class OrderService {
    private final Counter placed;
    private final Timer   checkout;
    OrderService(MeterRegistry reg) {
        this.placed   = reg.counter("orders.placed");
        this.checkout = reg.timer("orders.checkout.duration");
    }
    public void place(Order o) {
        checkout.record(() -> { doCheckout(o); });    // times the block
        placed.increment();
    }
}
// HTTP request metrics (http.server.requests: count, latency, status) come FREE — no code.
Metric typeUse for
CounterMonotonic totals (requests, errors, orders)
GaugePoint-in-time values (queue depth, cache size)
TimerDuration + rate (latency of an operation)

Interview Questions

Q1: What are the three pillars of observability and how does Spring provide each?

Metrics (aggregate numbers over time), logs (discrete events), and traces (the path of one request across services). Spring Boot provides metrics via Actuator + Micrometer (JVM, HTTP, datasource metrics out of the box, plus custom counters/gauges/timers) exported to Prometheus/Datadog; logs via SLF4J/Logback with structured JSON output in Boot 3.4+; and distributed tracing via Micrometer Tracing with an OTLP/Zipkin exporter that propagates trace/span IDs across service calls and into log lines. Together they let you alert on a metric, drill into correlated logs, and follow a single request end-to-end.

Q2: Liveness vs readiness probes — why separate them?

Liveness answers "is the process healthy enough to keep running?" — if it fails, the orchestrator restarts the pod. Readiness answers "can it serve traffic right now?" — if it fails, the pod is removed from the load balancer but not restarted. Conflating them causes two failures: a pod that's merely warming up (cache load, DB migration) gets killed instead of temporarily pulled from rotation, or a wedged process keeps receiving traffic because the single check still passes. Spring exposes them as /actuator/health/liveness and /health/readiness groups so Kubernetes can act correctly on each.

Q3: Why is metric cardinality a concern?

Each unique combination of metric name + tag values creates a distinct time series that the monitoring backend must store and index. Tagging a metric with an unbounded value — a user ID, a request path with IDs, a session token — produces effectively infinite series, exploding memory and cost and potentially crashing Prometheus. Keep tags low-cardinality and bounded (status code, endpoint template, region), never per-entity identifiers. Actuator's HTTP metrics use the route template (/users/{id}) rather than the concrete URL precisely to bound cardinality.

🏋️ Practice: add Actuator, expose only health,info,metrics,prometheus; enable liveness/readiness probes; write a custom HealthIndicator for a downstream dependency; register a Counter and a Timer via MeterRegistry and scrape /actuator/prometheus; turn on structured JSON logging; confirm HTTP request metrics appear without extra code.

🎯 Key Takeaways

  • Actuator adds health/metrics/diagnostics endpoints with one starter — expose the minimum and secure them.
  • Separate liveness (restart) and readiness (traffic) probes for correct orchestration.
  • Micrometer is a vendor-neutral metrics facade; HTTP/JVM metrics are free, add custom counters/timers.
  • Keep metric tag cardinality bounded — never tag with user/entity IDs.
  • Observability = metrics + structured logs + distributed traces, correlated by trace IDs.
