Validation & Error Handling Intermediate

Two things separate a toy API from a production one: it rejects bad input cleanly, and it never leaks a stack trace. Spring gives you Jakarta Bean Validation for the first and @RestControllerAdvice for the second. Done right, every error your API returns is a consistent, typed JSON object with the correct status code.

Bean Validation on the Request DTO

// Add spring-boot-starter-validation. Annotate the request record:
public record CreateUserRequest(
    @NotBlank String name,
    @Email @NotBlank String email,
    @Min(18) @Max(120) int age,
    @Size(min = 8) String password
) {}

@PostMapping
@ResponseStatus(HttpStatus.CREATED)
public UserDto create(@Valid @RequestBody CreateUserRequest req) {   // @Valid triggers validation
    return service.create(req);
}
// A body that violates a constraint → MethodArgumentNotValidException BEFORE your method runs.
// Without a handler that becomes an ugly 400 with a huge default body — so we handle it (below).

🧩 @Valid on a @RequestBody validates the object graph (nested @Valid fields too). To validate @RequestParam/@PathVariable directly, put @Validated on the controller class and constraints on the params. Validate at the trust boundary — never assume the client sent good data.

Global Error Handling — @RestControllerAdvice

public record ApiError(String code, String message, Map<String,String> fields) {}

@RestControllerAdvice                       // ONE place to translate exceptions → clean JSON
class GlobalExceptionHandler {

    @ExceptionHandler(NotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)                       // 404
    ApiError notFound(NotFoundException e) {
        return new ApiError("NOT_FOUND", e.getMessage(), Map.of());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)    // @Valid failures → 400
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    ApiError invalid(MethodArgumentNotValidException e) {
        var fields = e.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage, (a,b)->a));
        return new ApiError("VALIDATION_FAILED", "Invalid request", fields);
    }

    @ExceptionHandler(Exception.class)                          // catch-all → 500, NO stack leak
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    ApiError unexpected(Exception e) {
        log.error("Unhandled", e);                              // LOG the detail server-side
        return new ApiError("INTERNAL", "Something went wrong", Map.of());  // GENERIC to the client
    }
}
🔒 The catch-all is a security control: log the real exception server-side, return a generic message to the client. Leaking stack traces exposes framework versions, file paths, and SQL — reconnaissance for an attacker.

The Modern Alternative — ProblemDetail (RFC 7807)

// Spring 6 / Boot 3 ships a standard error format. Enable it or build it directly:
@ExceptionHandler(NotFoundException.class)
ProblemDetail handle(NotFoundException e) {
    var pd = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, e.getMessage());
    pd.setTitle("Resource not found");
    pd.setProperty("resourceId", e.getId());
    return pd;   // → application/problem+json, a documented interoperable shape
}

Custom Domain Exceptions

public class NotFoundException extends RuntimeException {   // UNCHECKED → triggers rollback + advice
    private final long id;
    public NotFoundException(long id) { super("Not found: " + id); this.id = id; }
    public long getId() { return id; }
}
// Throw meaningful domain exceptions in the service; let the advice map them to HTTP.
// Keeps controllers and services free of HTTP status codes — separation of concerns.

Interview Questions

Q1: How does @Valid validation flow through Spring MVC?

When a handler parameter annotated @Valid (or @Validated) is bound, Spring runs the Jakarta Bean Validation provider (Hibernate Validator) against the object's constraint annotations before the controller method executes. On a @RequestBody, failures throw MethodArgumentNotValidException; on @RequestParam/@PathVariable with a @Validated class, they throw ConstraintViolationException. You then translate those in a @RestControllerAdvice into a 400 with per-field messages. The key point is that validation runs at the boundary, before business logic, and rejection is automatic.

Q2: How do you handle exceptions globally, and why not per-controller try/catch?

@RestControllerAdvice centralizes exception-to-response mapping: one class with @ExceptionHandler methods handles specific exception types across every controller, each returning a consistent typed error body and the right status. Per-controller try/catch duplicates the same mapping everywhere, is easy to forget (leading to leaked stack traces), and couples HTTP concerns into business flow. The advice pattern keeps controllers/services throwing plain domain exceptions while a single place owns the HTTP translation — DRY, consistent, and secure (a catch-all prevents any stack trace from escaping).

Q3: Why validate on the server when the client already does?

Client-side validation is a UX convenience, not a security control — anyone can call the API directly (curl, Postman, a malicious client) and bypass the browser entirely. The server is the only trust boundary you control, so it must independently enforce every constraint: required fields, formats, ranges, and business rules. Client validation improves responsiveness; server validation guarantees integrity. Treat all inbound data as untrusted until validated on the server.

🏋️ Practice: annotate a request record with @NotBlank, @Email, @Min; add @Valid and confirm a bad body returns 400; write a @RestControllerAdvice that maps validation errors to a per-field JSON map, a custom NotFoundException to 404, and a catch-all to a generic 500; verify no stack trace reaches the client; convert one handler to return a ProblemDetail.

🎯 Key Takeaways

  • @Valid + Jakarta constraints validate request DTOs at the boundary before your method runs.
  • Centralize exception handling in @RestControllerAdvice — consistent, typed JSON errors.
  • Map each exception to the correct status (400/404/409) via @ResponseStatus or ResponseEntity.
  • A catch-all handler logs detail server-side and returns a generic message — never leak stack traces.
  • Throw domain exceptions in services; keep HTTP status codes in the advice (separation of concerns).
