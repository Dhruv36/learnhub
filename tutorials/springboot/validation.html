Validation Web Layer

What you'll master in this lesson: how Bean Validation (Jakarta Validation) rejects bad input declaratively before your logic runs; the difference between validating a request body (@Valid @RequestBody) and validating path/query parameters (@Validated on the class); the standard constraint annotations and custom constraints; nested and grouped validation; and why server-side validation at the trust boundary is non-negotiable regardless of what the client does. This is the "reject bad input cleanly" half of robust APIs; the "never leak a stack trace" half is the next lesson.

Part 1 — Bean Validation on the Request Body

// Add spring-boot-starter-validation. Annotate the request record with constraints:
public record CreateUserRequest(
    @NotBlank String name,
    @Email @NotBlank String email,
    @Min(18) @Max(120) int age,
    @Size(min = 8) String password
) {}

@PostMapping
@ResponseStatus(HttpStatus.CREATED)
public UserDto create(@Valid @RequestBody CreateUserRequest req) {   // @Valid TRIGGERS validation
    return service.create(req);
}
// A body violating any constraint → MethodArgumentNotValidException BEFORE your method runs,
// which (with a handler — next lesson) becomes a 400 with per-field messages.
💡 @Valid is what triggers validation. The constraint annotations on the DTO declare the rules; @Valid on the @RequestBody parameter tells Spring to run them before invoking your method. Without @Valid, the constraints are inert. On a violation, Spring throws MethodArgumentNotValidException and never calls your method — so your business logic only ever sees valid input.

🔍 Know the difference between @NotNull, @NotEmpty, and @NotBlank: @NotNull only requires the value to be present (a String of "" passes); @NotEmpty additionally requires length/size > 0 (a String of " " passes); @NotBlank additionally requires non-whitespace content (only for strings). For a required text field, @NotBlank is almost always what you want. Each constraint has a default message you can override with message = "...".

Part 3 — Body Validation vs Parameter Validation

// BODY: @Valid on the @RequestBody validates the object graph. Fails → MethodArgumentNotValidException.
@PostMapping void create(@Valid @RequestBody CreateUserRequest req) { }

// PATH/QUERY PARAMETERS: put @Validated on the CLASS, and constraints directly on the params.
// Fails → ConstraintViolationException (a DIFFERENT exception, handled separately).
@RestController
@Validated                                                        // enables parameter validation
public class UserController {
    @GetMapping("/users")
    public List<UserDto> list(@RequestParam @Min(1) int page,
                              @RequestParam @Size(max=50) String q) { ... }
}
🎯 Two mechanisms for two locations: @Valid on a @RequestBody validates the bound object (and throws MethodArgumentNotValidException); @Validated on the controller class plus constraints on @RequestParam/ @PathVariable validates method parameters directly (and throws ConstraintViolationException). Both yield a 400 once handled, but they're different exceptions, so your error handler (next lesson) must handle both to give consistent responses. This distinction is a common interview and code-review gotcha.

Part 4 — Nested & Grouped Validation

// NESTED: put @Valid on a field to cascade validation into a nested object:
public record OrderRequest(
    @NotBlank String orderId,
    @Valid @NotNull AddressDto shippingAddress,        // @Valid cascades into AddressDto's constraints
    @Valid List<LineItem> items                        // validates each element
) {}

// GROUPS: apply different constraints in different contexts (e.g. create vs update):
public interface OnCreate {}
public interface OnUpdate {}
public record UserRequest(
    @Null(groups = OnCreate.class) @NotNull(groups = OnUpdate.class) Long id,  // no id on create, required on update
    @NotBlank String name
) {}
@PostMapping void create(@Validated(OnCreate.class) @RequestBody UserRequest r) { }
@PutMapping  void update(@Validated(OnUpdate.class) @RequestBody UserRequest r) { }
🔍 Nested validation requires @Valid on the field — without it, the nested object's own constraints are not checked (a common silent gap). Validation groups let one DTO carry context-specific rules — the classic case is an id that must be absent on create but present on update — activated by passing the group to @Validated(Group.class). These handle the real-world cases where a single flat set of constraints isn't enough.

Part 5 — Custom Constraints

// When the built-in constraints don't fit, write your own annotation + validator:
@Constraint(validatedBy = StrongPasswordValidator.class)
@Target(FIELD) @Retention(RUNTIME)
public @interface StrongPassword {
    String message() default "password too weak";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}
public class StrongPasswordValidator implements ConstraintValidator<StrongPassword, String> {
    public boolean isValid(String value, ConstraintValidatorContext ctx) {
        return value != null && value.length() >= 8 && value.matches(".*[0-9].*") && value.matches(".*[A-Z].*");
    }
}
// Use it like any constraint:  @StrongPassword String password
// (Cross-field validation — e.g. password == confirmPassword — uses a class-level custom constraint.)
🔍 Custom constraints keep validation declarative and reusable rather than scattering if-checks through your code. A single-field rule implements ConstraintValidator<Annotation, FieldType>; a cross-field rule (two fields must agree) is a class-level constraint validating the whole object. Prefer a custom constraint over ad-hoc validation code whenever a rule is reused or belongs to the input contract — it's testable, composable, and lives with the DTO.

Part 6 — Deep Dive: Why Server-Side Validation Is Non-Negotiable

  • The client is untrusted. Client-side (browser) validation is a UX convenience — anyone can bypass it with curl, Postman, a modified client, or a direct API call. The server is the only place you control, so it must independently enforce every rule.
  • Validate at the boundary, before logic. Because @Valid runs before your method, your business code operates on already-valid input — it doesn't need defensive checks for "what if name is blank?" The boundary is the single, declarative place those checks live.
  • Distinguish input validation from business rules. Bean Validation is for structural input constraints (present, formatted, in range). Rules that need the database or domain state ("email already taken", "insufficient balance") belong in the service, throwing domain exceptions — not in annotations.
  • Consistent 400s, not leaked internals. Validation failures should produce a clean, typed 400 with per-field messages (the Error Handling lesson maps the exceptions), giving clients actionable feedback without exposing implementation details.
🔍 Senior framing: validation is a security and correctness boundary, not a formality, and its core principle is that the server must independently and declaratively enforce the input contract because the client cannot be trusted — client-side validation improves responsiveness but is trivially bypassed, so any rule that matters must exist on the server. Spring makes this declarative and boundary-placed: constraint annotations on DTOs describe the contract, @Valid/@Validated run them before your method executes (throwing MethodArgumentNotValidException for bodies and ConstraintViolationException for parameters — two exceptions your error handling must both cover), and violations become clean 400s with per-field detail. The mature engineer designs this deliberately: choosing the right constraints (knowing @NotBlank vs @NotEmpty vs @NotNull), cascading into nested objects with @Valid and using groups for context-specific rules (create vs update), extracting reusable or cross-field rules into custom constraints rather than imperative checks, and — critically — keeping structural input validation at the annotation boundary while pushing business rules that require state ("already exists", "insufficient funds") into the service layer as domain exceptions. The payoff is that business logic always receives clean, typed input and never has to re-validate, the API returns consistent, actionable 400s, and the trust boundary is enforced in one declarative place. Interviewers probe validation to see whether you understand it as the server-side enforcement of an input contract at the boundary — including the two-exception distinction and the input- vs-business-rule separation — rather than as a set of annotations to sprinkle on fields, because that understanding is what produces robust, secure APIs.

Interview Questions

Q1 (beginner): How does @Valid validation work in Spring?

You annotate a request DTO's fields with Jakarta Bean Validation constraints (@NotBlank, @Email, @Min, etc.) that declare the rules, and put @Valid on the @RequestBody parameter to trigger them. When a request arrives, Spring binds the JSON to the DTO and then runs the validator before invoking your controller method; if any constraint is violated it throws MethodArgumentNotValidException and never calls your method, so your business logic only ever sees valid input. With an exception handler that failure becomes a 400 with per-field messages. The key point is that constraints are declarative on the DTO and @Valid is what activates them at the boundary.

Q2 (beginner): What's the difference between @NotNull, @NotEmpty, and @NotBlank?

They're progressively stricter for strings/collections. @NotNull only requires the value to be non-null — an empty string "" passes. @NotEmpty requires non-null and a length/size greater than zero — "" fails but a whitespace string " " passes. @NotBlank (strings only) requires non-null and at least one non-whitespace character — " " fails. So for a required text field you almost always want @NotBlank, for a required collection @NotEmpty, and @NotNull for a required non-string value like a number or object reference.

Q3 (mid): How do you validate path/query parameters, and how does that differ from validating a body?

Validating a request body uses @Valid on the @RequestBody parameter, which validates the bound object graph and, on failure, throws MethodArgumentNotValidException. Validating individual method parameters (@RequestParam/@PathVariable) is different: you put @Validated on the controller class to enable method-parameter validation and place constraints directly on the parameters (e.g. @RequestParam @Min(1) int page); a violation throws ConstraintViolationException instead. Both ultimately should produce a 400, but because they throw different exception types, your @RestControllerAdvice must handle both to give consistent error responses — a common gap where parameter-validation failures leak the default error format. So: @Valid for bodies, @Validated-on-class for params, two exceptions to handle.

Q4 (mid): How do you validate nested objects and apply context-specific rules?

Nested validation requires @Valid on the field that holds the nested object (or on a collection field to validate each element); without @Valid there, the nested object's own constraints are silently not checked — a subtle, common gap. Context-specific rules use validation groups: you define marker interfaces (e.g. OnCreate, OnUpdate), tag constraints with the groups they apply to (e.g. @Null(groups=OnCreate.class) @NotNull(groups=OnUpdate.class) Long id so an id is forbidden on create but required on update), and activate a group per endpoint with @Validated(OnCreate.class). Groups let a single DTO carry different validation rules for different operations instead of duplicating DTOs, and cascading with @Valid lets you validate a whole object tree in one declarative pass.

Q5 (senior): Where should validation live, and how do you separate input validation from business rules?

Structural input validation — is the field present, correctly formatted, within range, the right length — belongs at the boundary as declarative Bean Validation constraints on the request DTOs, run by @Valid/@Validated before the controller method executes, so invalid requests are rejected with a 400 before any business logic runs and your service code never has to re-check basic well-formedness. Business rules — constraints that depend on application or database state, such as "this email is already registered", "the account has insufficient balance", or "this transition isn't allowed from the current status" — cannot be expressed as static annotations because they require querying the domain, so they belong in the service layer, checked there and signalled by throwing meaningful domain exceptions (e.g. DuplicateEmailException, InsufficientFundsException) that a @RestControllerAdvice maps to appropriate statuses (409, 422, 400) with clear messages. Mixing these up causes problems: trying to encode stateful rules in annotations doesn't work (and pollutes the DTO with dependencies), while doing structural checks imperatively in the service scatters validation and duplicates it. The clean separation — declarative structural validation at the boundary, imperative business-rule checks in the service raising domain exceptions — keeps each concern in its proper place, makes both testable, and produces consistent, well-typed error responses. The senior instinct is to ask of every rule "does this depend on state?": if no, it's an input constraint at the boundary; if yes, it's a business rule in the service. This layered validation is also why the trust boundary discussion matters: the boundary enforces the input contract independently of the client, and the service enforces the domain invariants independently of the input.

Q6 (senior): Why is server-side validation mandatory even when the client validates, and what's the security dimension?

Because client-side validation is not a security or correctness control — it's purely a user-experience enhancement that gives immediate feedback in the browser, and it can be bypassed trivially by anyone who doesn't use your client: a direct call with curl or Postman, a script, a modified or malicious front-end, an intercepting proxy, or simply a bug in the client. The server is the only component you control end to end, so it is the sole place where the input contract can actually be enforced; any rule that must hold — required fields, formats, ranges, lengths, allowed values — has to be checked on the server or it effectively doesn't exist. The security dimension is significant: unvalidated input is the root of entire vulnerability classes. Without server-side validation and proper handling, attackers can submit oversized or malformed payloads (denial of service, buffer/precision issues), values that flow into injection sinks (SQL/NoSQL/command/LDAP injection, XSS if reflected), out-of-range or type-confused data that corrupts state, mass-assignment of fields the client shouldn't set, and inputs that violate business invariants leading to fraud (e.g. negative amounts, quantities beyond stock). Server-side validation at the trust boundary — combined with parameterized queries, output encoding, DTOs that expose only intended fields, and authorization checks — is a primary defense: it constrains what data can even enter your logic. Treating all inbound data as untrusted until validated on the server is a foundational security principle, and Spring's Bean Validation makes enforcing it declarative and consistent. So the rule is absolute: validate on the server regardless of the client, because the client's validation protects nothing and the server's protects everything.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Constrain a DTO. Annotate a request record with @NotBlank name, @Email email, and @Min(18) age, add @Valid to the handler, and confirm a bad body is rejected before your method runs.
Solution

The record carries the constraints; @Valid @RequestBody triggers them. A body with a blank name, bad email, or age < 18 throws MethodArgumentNotValidException and your method is never called (add a log to confirm). With an error handler it returns a 400. Without @Valid, the constraints are inert.

2 (easy): NotBlank vs NotEmpty vs NotNull. For a required text field, test each with null, "", and " " and record which pass.
Solution

@NotNull: null fails, "" and " " pass. @NotEmpty: null and "" fail, " " passes. @NotBlank: all three fail (requires non-whitespace content). For a required text field, @NotBlank is the right choice.

3 (medium): Validate a query param. Add validation to a @RequestParam @Min(1) int page and observe the exception type differs from body validation. Explain what your error handler must do.
Solution

Put @Validated on the controller class and @Min(1) on the @RequestParam. A page=0 request throws ConstraintViolationException (not MethodArgumentNotValidException). Your @RestControllerAdvice must handle both exceptions to return consistent 400s — otherwise param-validation failures leak the default error format.

4 (medium): Nested validation. Add a nested AddressDto with its own constraints to a request; show that without @Valid on the field they're skipped, then add it.
Solution

With just AddressDto shippingAddress, the address's @NotBlank street etc. are not checked — the nested constraints are silently skipped. Adding @Valid to the field (@Valid AddressDto shippingAddress) cascades validation into the nested object. This silent gap is a common bug.

5 (hard): Custom constraint + business rule. Write a @StrongPassword custom constraint for structural password rules, and separately implement an "email already exists" check. Explain why one is a constraint and the other isn't.
Solution

@StrongPassword is a custom ConstraintValidator checking length/character classes — purely structural, no state, so it belongs at the boundary as a declarative constraint. "Email already exists" requires a database query, so it can't be a static annotation; it belongs in the service, which checks the repository and throws a DuplicateEmailException mapped to 409/400. The distinction — structural input rule (constraint) vs stateful business rule (service + domain exception) — is the key design decision.

6 (hard): Explain-it test. Out loud: (a) how @Valid validation flows and the body-vs-param exception distinction; (b) nested/grouped validation; (c) why server-side validation is mandatory and where business rules go.
Self-check rubric

Pass (a) if you said constraints on the DTO + @Valid run before the method, throwing MethodArgumentNotValidException (body) vs ConstraintViolationException (params via @Validated). Pass (b) if you named @Valid cascading for nested and groups for create-vs-update. Pass (c) if you said the client is untrusted so the server must enforce the contract, with structural rules at the boundary and stateful business rules in the service. This sets up error handling.

🎯 Key Takeaways

  • Constraint annotations on the DTO declare rules; @Valid on the @RequestBody triggers them before your method runs.
  • Know @NotNull vs @NotEmpty vs @NotBlank; use @NotBlank for required text.
  • Body validation (@Valid → MethodArgumentNotValidException) differs from parameter validation (@Validated on class → ConstraintViolationException) — handle both.
  • Cascade into nested objects with @Valid on the field; use groups for context-specific rules (create vs update); extract reusable/cross-field rules into custom constraints.
  • Keep structural input validation at the boundary (annotations) and stateful business rules in the service (domain exceptions).
  • Server-side validation is mandatory — the client is untrusted, and unvalidated input is a primary security risk.
