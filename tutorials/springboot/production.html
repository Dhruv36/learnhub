Production & Deployment Ultra-Advanced

This lesson ties the whole track together into what it takes to ship a Spring Boot service and keep it healthy: packaging, schema migrations, containerization, graceful shutdown, and a production readiness checklist. Everything below is the difference between "it runs on my laptop" and "it survives a Tuesday afternoon traffic spike".

Packaging — The Executable Jar

# Boot builds a single self-contained "fat jar" with an embedded Tomcat — no external server:
./mvnw clean package        # → target/app.jar
java -jar target/app.jar    # runs on port 8080

# Layered jars (for better Docker caching — dependencies change less often than your code):
# the Boot Maven plugin produces layers: dependencies / spring-boot-loader / snapshot-deps / application

Database Migrations — Flyway (NOT ddl-auto)

# NEVER let Hibernate manage the prod schema (ddl-auto: create/update). Use versioned migrations:
# src/main/resources/db/migration/
#   V1__create_users.sql
#   V2__add_email_index.sql
spring:
  flyway: { enabled: true }
  jpa: { hibernate: { ddl-auto: validate } }   # validate the entity ⟷ schema match, don't mutate

# Flyway runs pending migrations in order at startup, records them in a history table, and
# refuses to run if a checksum changed — reproducible, auditable schema evolution across environments.
🗃️ ddl-auto: update feels convenient but silently mutates schemas, can't be reviewed, can't drop columns, and diverges across environments. Versioned migrations (Flyway or Liquibase) are code-reviewed SQL in git — the only safe way to evolve a production schema.

Containerizing — Multi-Stage Dockerfile

# Stage 1: build (fat JDK, only in the builder)
FROM eclipse-temurin:21-jdk AS build
WORKDIR /app
COPY . .
RUN ./mvnw -q clean package -DskipTests

# Stage 2: runtime (slim JRE, small attack surface, non-root)
FROM eclipse-temurin:21-jre
RUN useradd -r app
USER app                                    # never run as root
COPY --from=build /app/target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java","-jar","/app.jar"]

See the Docker and Kubernetes tracks for the deployment side. Boot's bootBuildImage / Cloud Native Buildpacks can also build an optimized image without a Dockerfile.

Graceful Shutdown & Resilience

server:
  shutdown: graceful                          # finish in-flight requests before exiting
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
# On SIGTERM (a K8s rolling deploy), the app stops accepting new requests, drains active ones,
# then exits — no dropped connections mid-request.

Interview Questions

Q1: Why use Flyway/Liquibase instead of Hibernate ddl-auto in production?

ddl-auto: update lets Hibernate mutate the schema at startup by diffing entities against the database. That's unreviewable, non-deterministic across environments, additive-only (it won't drop or rename safely), and offers no rollback or audit trail — a recipe for prod drift and data loss. Versioned migration tools store ordered, checksummed SQL scripts in version control that run exactly once, in order, recorded in a history table, and fail fast if a previously-applied script changed. That gives reproducible, code-reviewed, auditable schema evolution. In prod you set ddl-auto: validate so Hibernate only verifies the entity/schema match without altering it.

Q2: How do you deploy Spring Boot safely under a rolling deployment?

Package a self-contained jar in a slim, non-root, multi-stage Docker image. Enable graceful shutdown (server.shutdown: graceful) so on SIGTERM the app stops accepting new requests and drains in-flight ones before exiting — no dropped connections as pods are replaced. Wire Kubernetes liveness/readiness probes to Actuator so traffic is only routed to ready pods and wedged pods are restarted. Run schema migrations via Flyway (ideally as an init step) so the new version's schema is in place. Externalize config/secrets so the same image is promoted across environments, and size connection/thread pools to the instance. Together these make a rollout invisible to clients.

Q3: What resilience patterns protect a service from downstream failures?

Set aggressive timeouts on every outbound call so a slow dependency can't tie up request threads indefinitely. Wrap unreliable calls in a circuit breaker (Resilience4j) that trips after a failure threshold and fails fast (with a fallback) instead of hammering a dead service, then half-opens to probe recovery. Add bounded retries with exponential backoff and jitter for transient errors — but only for idempotent operations, and never retry into a cascading overload. Use bulkheads / bounded thread pools and rate limiting to isolate failures so one struggling dependency can't exhaust resources shared with healthy paths. Size the HikariCP connection pool deliberately. The theme is failing fast and containing blast radius rather than blocking and cascading.

🏋️ Practice: take the app you've built across this track and make it production-ready: add Flyway migrations and switch to ddl-auto: validate; write a multi-stage Dockerfile running as non-root on a slim JRE; enable graceful shutdown; add a Resilience4j circuit breaker + timeout on a downstream call; wire Actuator liveness/readiness for Kubernetes; run it containerized and confirm a rolling restart drops zero requests.

🎯 Key Takeaways

  • Ship a self-contained fat jar; containerize with a multi-stage, non-root, slim-JRE Docker image.
  • Evolve schemas with Flyway/Liquibase migrations and ddl-auto: validate — never update in prod.
  • Enable graceful shutdown and wire liveness/readiness probes for zero-downtime rolling deploys.
  • Add resilience — timeouts, circuit breakers, bounded retries and pools — to contain downstream failures.
  • The production checklist (layering, validation, errors, auth, config, observability, testing, schema, packaging) is the whole track applied at once.
