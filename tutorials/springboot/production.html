Production & Deployment Production

What you'll master in this lesson — the track capstone: how a Spring Boot app becomes a deployable, operable artifact — the self-contained executable jar, safe schema migrations with Flyway (never ddl-auto in prod), a multi-stage Dockerfile (slim, non-root, layer-cached), graceful shutdown for zero-downtime deploys, JVM tuning in containers, and a production-readiness checklist that synthesizes every lesson in the track. Prerequisites: everything before it — this lesson assembles the whole track into a shippable service. Deployment specifics live in the Docker, Kubernetes, and CI/CD tracks.

Part 1 — Packaging: The Executable Jar

# Boot builds a single self-contained "fat jar" with an EMBEDDED server — no external Tomcat to install:
./mvnw clean package        # → target/app.jar  (your code + all dependencies + embedded Tomcat/Netty)
java -jar target/app.jar    # just run it — starts on port 8080

# Layered jars for better Docker caching (dependencies change far less often than your code):
# the Boot Maven/Gradle plugin splits the jar into layers — dependencies / spring-boot-loader /
# snapshot-dependencies / application — so a code change only invalidates the small top layer.
🎯 Spring Boot's executable jar is the whole app in one file — embedded server included. There's no separate application server to install and configure; java -jar app.jar starts a fully working service. That's what makes Boot apps so container- and cloud-friendly: the deployable unit is one self-contained artifact. The layered jar refinement matters for Docker: because your dependencies change rarely and your code changes constantly, splitting them into separate image layers means a rebuild after a code change reuses the cached (large) dependency layer and only rewrites the tiny application layer — dramatically faster builds and smaller pushes.

Part 2 — Schema Migrations: Flyway, Never ddl-auto

# NEVER let Hibernate manage the prod schema (ddl-auto: create/update). Use VERSIONED migrations:
# src/main/resources/db/migration/
#   V1__create_users.sql
#   V2__add_email_index.sql
spring:
  flyway: { enabled: true }
  jpa: { hibernate: { ddl-auto: validate } }   # only VALIDATE entity ⟷ schema match — never mutate

# Flyway runs pending migrations in order at startup, records them in a flyway_schema_history table,
# and REFUSES to start if an already-applied migration's checksum changed — reproducible, auditable
# schema evolution that's identical across dev, staging, and prod.
🗃️ ddl-auto: update feels convenient in dev and is a disaster in production. It silently mutates the live schema based on your entities: it can't be code-reviewed, won't drop or rename columns safely, applies changes no one approved, and drifts differently across environments. Versioned migrations (Flyway or Liquibase) make schema changes reviewed SQL files in git, applied in order, checksum-verified, and recorded — the only safe, auditable way to evolve a production schema. Set ddl-auto: validate so Hibernate confirms your entities match the migration-built schema (catching drift at startup) without ever mutating it. This is a non-negotiable production practice and a common interview checkpoint.

Part 3 — Containerizing: A Proper Multi-Stage Dockerfile

# Stage 1: BUILD — the full JDK + build tools live only in the throwaway builder image
FROM eclipse-temurin:21-jdk AS build
WORKDIR /app
COPY . .
RUN ./mvnw -q clean package -DskipTests

# Stage 2: RUNTIME — slim JRE only (smaller image, smaller attack surface), running as NON-ROOT
FROM eclipse-temurin:21-jre
RUN useradd -r app
USER app                                    # NEVER run the container as root
COPY --from=build /app/target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java","-jar","/app.jar"]
🐳 A production image is small, minimal, and non-root. The multi-stage build keeps the heavy JDK and build tooling in a throwaway builder stage, so the final image ships only a slim JRE plus your jar — smaller to pull and a smaller attack surface. Running as a non-root user limits the blast radius if the app is compromised. Boot's bootBuildImage / Cloud Native Buildpacks can produce an optimized, layered image with no Dockerfile at all. The deployment side — resource limits, probes (which map to the Actuator liveness/readiness endpoints), rolling updates — lives in the Docker and Kubernetes tracks.

Part 4 — Graceful Shutdown & Runtime Resilience

server:
  shutdown: graceful                          # finish in-flight requests before exiting
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
# On SIGTERM (what K8s sends during a rolling deploy), the app STOPS accepting new requests,
# DRAINS the ones already in flight, then exits — no connection dropped mid-request.
Runtime concernApproach (see linked lessons)
Slow/failing downstreamTimeouts + circuit breaker — Resilience4j
Transient failuresBounded retries with backoff + jitter (idempotent ops only)
OverloadBulkheads / rate limiting; bounded thread & connection pools
Connection stormsHikariCP pool sizing (Boot's default connection pool)
Clean deploysGraceful shutdown + readiness probe flips DOWN first (Actuator)
🔄 Zero-downtime deploys need graceful shutdown, not just a fast restart. During a rolling update Kubernetes sends SIGTERM to the old pod; with server.shutdown: graceful the app stops accepting new connections but lets in-flight requests complete within the timeout before exiting, so no user gets a dropped connection mid-request. Paired with the readiness probe flipping DOWN as shutdown begins (so the load balancer stops routing new traffic before the app stops accepting it), you get seamless deploys. The rest of runtime resilience — timeouts, circuit breakers, bounded pools — is covered in the Resilience and Observability lessons; production is where they all come together.

Part 5 — JVM & Performance in Containers

# Modern JVMs are container-aware (respect cgroup CPU/memory limits) — but set heap explicitly for predictability:
java -XX:MaxRAMPercentage=75 -jar app.jar     # use up to 75% of the container's memory limit for the heap

# Startup speed options (matter for serverless / scale-to-zero / fast autoscaling):
#   Class Data Sharing (CDS)      — lighter-weight startup win, no code changes
#   Spring AOT + GraalVM native   — millisecond startup, low memory, at the cost of build complexity
#                                   and needing reflection/resource config (AOT hints)
⚙️ Tell the JVM about its container. Recent JVMs read cgroup limits so they don't assume the whole host's memory, but you should still set heap sizing explicitly (-XX:MaxRAMPercentage) so memory use is predictable and the container isn't OOM-killed by the orchestrator. For workloads where startup time matters (serverless, scale-to-zero, rapid autoscaling), Spring AOT + GraalVM native image gives millisecond startup and low memory — trading longer, more complex builds and the need to declare reflection/resource usage; CDS is a lighter win with no code changes. For most long-running services the standard JVM with sane heap settings is the right default.

Interview Questions

Q1 (beginner): What is a Spring Boot executable (fat) jar and why is it convenient?

It's a single jar that contains your application code, all its dependencies, and an embedded web server (Tomcat/Jetty/Netty), so you can run the whole application with just java -jar app.jar — there's no separate application server to install, configure, and deploy a WAR into. This self-contained artifact is what makes Spring Boot apps so easy to containerize and run in the cloud: the deployable unit is one file that starts a fully working service on a port. The layered-jar variant additionally splits the jar into layers (dependencies, loader, snapshots, application) so Docker image builds cache the rarely-changing dependency layers and only rebuild the small application layer when your code changes, making builds faster and images cheaper to push.

Q2 (beginner): Why should you use Flyway migrations instead of ddl-auto: update in production?

Because ddl-auto: update lets Hibernate silently alter the live database schema to match your entity classes, which is unsafe and unreviewable in production: the changes it makes aren't code-reviewed or version-controlled, it won't drop or rename columns cleanly (so the schema accretes cruft), and different environments drift apart because each mutates independently. Versioned migrations (Flyway or Liquibase) instead make every schema change an explicit, reviewed SQL file checked into git (e.g. V2__add_email_index.sql); Flyway applies pending migrations in order at startup, records them in a history table, and refuses to start if an already-applied migration's checksum changed — giving reproducible, auditable schema evolution that's identical across dev, staging, and prod. You pair this with ddl-auto: validate, which makes Hibernate merely check that your entities match the migration-built schema (catching mismatches at startup) without mutating anything. Managing a production schema through reviewed migrations is a non-negotiable practice.

Q3 (mid): Walk through a good production Dockerfile for a Spring Boot app.

Use a multi-stage build. The first stage starts from a full JDK image, copies the source, and runs the build (./mvnw clean package) to produce the jar — all the heavy build tooling lives only in this throwaway builder stage. The second stage starts from a slim JRE image (you don't need the JDK or build tools at runtime, and a smaller image means fewer bytes to pull and a smaller attack surface), creates a non-root user and switches to it (so a compromise of the app doesn't run as root), copies only the built jar from the builder stage, exposes the port, and sets the entrypoint to run the jar. Optionally you use the layered-jar feature to copy dependency layers separately for better caching, or skip the Dockerfile entirely with Boot's bootBuildImage / Cloud Native Buildpacks, which produce an optimized layered image. The principles being demonstrated — multi-stage to keep the runtime image minimal, slim base image, non-root user, copy only what's needed — are general container best practices, and getting them right is what separates a bloated, insecure image from a production-grade one.

Q4 (mid): How do you achieve zero-downtime deployments with Spring Boot on Kubernetes?

Two mechanisms working together. First, graceful shutdown: set server.shutdown: graceful (with a timeout-per-shutdown-phase) so that when Kubernetes sends SIGTERM to a pod during a rolling update, the app stops accepting new requests but lets the requests already in flight finish before the process exits — no connection is dropped mid-request. Second, the readiness probe (from Actuator's /actuator/health/readiness): as the pod begins shutting down it reports NOT ready, so Kubernetes stops routing new traffic to it before it stops accepting connections, and when a new pod starts it only receives traffic once its readiness probe passes (after the context is fully up and dependencies are reachable). Combined with a rolling update strategy (bring up new pods, wait for readiness, then drain and terminate old ones) this means at every moment there are enough ready pods to serve traffic and no in-flight request is cut off — a seamless deploy. The distinction from liveness matters here too: readiness controls traffic routing during the transition, while liveness (which should not depend on external dependencies) handles genuinely wedged processes.

Q5 (senior): Walk me through your production-readiness checklist for a Spring Boot service.

I'd go concern by concern, because production-readiness is the assembly of all of them. Architecture: clean layering — thin controllers, business logic in services, data access in repositories, DTOs at the API edge so the entity model isn't exposed. Input handling: @Valid with Jakarta constraints returning 400s, and a @RestControllerAdvice that maps exceptions to typed JSON error responses without leaking stack traces. Data: Spring Data JPA with N+1 avoided (fetch joins / entity graphs), @Transactional on the service methods that represent business operations, and optimistic locking where concurrent writes need it. Security: Spring Security with authentication (JWT/OAuth2) and server-enforced authorization (@PreAuthorize), never trusting the client. Config & secrets: externalized configuration via application.yml plus environment variables / a secret store, with profiles per environment and no secrets in the image or git. Cross-cutting: caching that's bounded with a TTL and invalidated on writes, async work on bounded pools, scheduled jobs made cluster-safe. Resilience: timeouts on every remote call, circuit breakers and fallbacks on hot paths, retries only for idempotent operations, and sagas/idempotency instead of cross-service transactions. Observability: Actuator with Micrometer metrics exported to Prometheus, structured JSON logs, distributed tracing, and correctly separated liveness/readiness probes. Testing: a real pyramid — mostly fast unit tests, slices for web and data layers, a few full integration tests, all against real dependencies via Testcontainers. Schema: versioned Flyway/Liquibase migrations with ddl-auto: validate. Packaging & deploy: a self-contained jar in a multi-stage, non-root, slim image with graceful shutdown and container-aware JVM heap sizing, deployed with rolling updates and resource limits. The senior signal isn't reciting the list — it's understanding why each item exists and being able to prioritize (security and data safety are non-negotiable; some observability depth can follow) and to recognize which are missing in a given codebase. It's the difference between a service that demos and one that survives real traffic, deploys, and failures.

Q6 (senior): How does running in a container change how you configure and tune the JVM, and when would you go native?

In a container the JVM must respect the cgroup CPU and memory limits, not the host's. Modern JVMs are container-aware and read those limits automatically, but you should still configure heap sizing explicitly — -XX:MaxRAMPercentage to use a sensible fraction (e.g. 75%) of the container's memory limit for the heap — so memory usage is predictable and the container isn't OOM-killed by the orchestrator when off-heap usage plus heap exceeds the limit (a common surprise, since the heap isn't the JVM's only memory: metaspace, thread stacks, direct buffers, and code cache all count). You also right-size the container's CPU and memory to the app's real profile and set the connection/thread pools accordingly. On startup and footprint: a standard JVM has slow-ish startup and a JIT warm-up period, which is fine for long-running services but costly for serverless / scale-to-zero / rapid-autoscaling workloads where cold starts hit users. There, Spring AOT + GraalVM native image compiles the app ahead of time to a native executable with millisecond startup and much lower memory, at the cost of longer, more complex builds and needing to declare reflection/resource/proxy usage via AOT hints (since native closed-world compilation can't do arbitrary runtime reflection). A lighter middle option is Class Data Sharing (CDS), which speeds startup without code changes or going fully native. So the decision: stick with the tuned JVM for typical long-running services (it has the best peak throughput and simplest build), and reach for native when startup latency and memory density genuinely matter (serverless functions, very high pod counts, scale-to-zero) and you can absorb the build complexity — a deliberate trade-off rather than a default.

🎯 Key Takeaways

  • Boot ships a self-contained executable jar (embedded server) — java -jar just runs it; layer it for Docker caching.
  • Evolve the schema with versioned migrations (Flyway/Liquibase) + ddl-auto: validate — never update in prod.
  • Package in a multi-stage, non-root, slim-JRE image — small and hardened.
  • Graceful shutdown + readiness probe = zero-downtime rolling deploys (drain on SIGTERM).
  • Make the JVM container-aware (heap sizing); go native only when cold-start/memory density justify the build cost.
  • "Production-ready" = the whole checklist assembled: layering, validation, data, security, config, resilience, observability, testing, schema, packaging.
