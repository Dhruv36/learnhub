Auto-Config & Starters Basics

"Spring Boot" is Spring plus two conveniences that removed the framework's old reputation for XML hell: starter dependencies and auto-configuration. Together they let a single annotated class boot a fully-wired web server. Understanding how they work — not just that they do — is what separates someone who can debug a broken startup from someone who googles stack traces.

The Entry Point

@SpringBootApplication                 // = @Configuration + @ComponentScan + @EnableAutoConfiguration
public class ShopApplication {
    public static void main(String[] args) {
        SpringApplication.run(ShopApplication.class, args);   // boots the container + embedded server
    }
}
// @ComponentScan scans THIS package and below for @Component/@Service/@Controller beans.
// That's why your beans must live under this class's package — a common "bean not found" cause.

@SpringBootApplication unpacks into three annotations. The interesting one is @EnableAutoConfiguration.

Starters — Curated Dependency Bundles

<!-- pom.xml: ONE line pulls in Spring MVC, Jackson, validation, and embedded Tomcat -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
StarterPulls in
spring-boot-starter-webSpring MVC, Jackson (JSON), embedded Tomcat
spring-boot-starter-data-jpaHibernate, Spring Data JPA, JDBC
spring-boot-starter-securitySpring Security auto-configured
spring-boot-starter-validationJakarta Bean Validation (Hibernate Validator)
spring-boot-starter-testJUnit 5, Mockito, AssertJ, MockMvc
spring-boot-starter-actuatorHealth/metrics endpoints
📦 The Boot parent POM / BOM pins compatible versions of every dependency, so you don't specify versions on starters. This "dependency management" is the quiet superpower — it kills the version-conflict matrix that plagued classic Spring.

Auto-Configuration — Beans Based on the Classpath

// Auto-config classes ship inside the starters and run conditionally. Simplified:
@Configuration
@ConditionalOnClass(DataSource.class)          // only if a JDBC driver is on the classpath
@ConditionalOnMissingBean(DataSource.class)    // only if YOU didn't define one yourself
class DataSourceAutoConfiguration {
    @Bean DataSource dataSource(DataSourceProperties props) { ... }   // builds one from your config
}
// So: add the JPA starter + an H2/Postgres driver → Boot auto-creates a DataSource,
// an EntityManagerFactory, a TransactionManager. Add the web starter → it configures
// DispatcherServlet, Jackson, an embedded Tomcat on port 8080. You wrote none of it.

The key is the @Conditional* family: auto-config only kicks in when its conditions hold, and backs off the moment you define your own bean of that type (@ConditionalOnMissingBean). That's why overriding a default is as simple as declaring your own @Bean.

Seeing & Controlling What Happened

# Ask Boot to print the auto-config report (what matched, what didn't, and WHY):
java -jar app.jar --debug

# Exclude an auto-config you don't want:
@SpringBootApplication(exclude = { SecurityAutoConfiguration.class })

# Or via properties:
spring.autoconfigure.exclude=org.springframework...SecurityAutoConfiguration
🔎 With Actuator on, GET /actuator/conditions shows the same condition-evaluation report over HTTP. When startup does something you don't expect ("why is there a Tomcat?", "why no DataSource?"), this report is the answer — read it instead of guessing.

Common Mistakes

Pinning starter versions manuallyVersion conflicts, odd errorsLet the Boot BOM manage versions
Treating auto-config as a black boxCan't debug or override startupRead --debug / /actuator/conditions
Adding a driver but no configDataSource fails to buildSet spring.datasource.* (or use H2)
Fat pile of unused startersSlow startup, bloat, attack surfaceOnly add starters you use

Interview Questions

Q1: How does auto-configuration actually work?

Starters place auto-configuration classes on the classpath, registered via META-INF/spring/…AutoConfiguration.imports. At startup @EnableAutoConfiguration loads them, but each is guarded by @Conditional annotations — @ConditionalOnClass (a type is present), @ConditionalOnMissingBean (you haven't defined your own), @ConditionalOnProperty, etc. Only configurations whose conditions pass contribute beans. So Boot inspects what's on the classpath and what you've already defined, then fills in sensible defaults. It's not reflection magic — it's ordinary conditional bean registration, inspectable via the condition-evaluation report.

Q2: How do you override an auto-configured bean?

Just define your own bean of the same type. Most auto-config is guarded by @ConditionalOnMissingBean, so as soon as your @Bean (or a @Component) of that type exists, Boot backs off and yours wins. You can also exclude a whole auto-configuration class with @SpringBootApplication(exclude = …) or the spring.autoconfigure.exclude property, and tune many defaults purely through application.yml properties without writing any bean at all.

Q3: What does @SpringBootApplication combine?

It's a convenience meta-annotation for three: @Configuration (the class itself can declare @Bean methods), @ComponentScan (scan this package and sub-packages for stereotypes), and @EnableAutoConfiguration (turn on the conditional auto-config machinery). The @ComponentScan part is why bean-discovery is package-relative to the main class — a frequent source of "no qualifying bean" errors when code lives in a sibling or parent package.

🏋️ Practice: generate a project on start.spring.io with Web + JPA + H2; run with --debug and find where the DataSource and Tomcat were auto-configured; override the default Jackson ObjectMapper by declaring your own @Bean and confirm the auto-config backs off; exclude SecurityAutoConfiguration and observe the login page disappear.

🎯 Key Takeaways

  • Starters are curated, version-managed dependency bundles — add one line, get a working stack.
  • Auto-configuration registers beans conditionally on classpath contents and your own definitions.
  • The @Conditional* family (esp. @ConditionalOnMissingBean) makes overriding trivial — just define your own bean.
  • @SpringBootApplication = @Configuration + @ComponentScan + @EnableAutoConfiguration; scanning is package-relative.
  • Use --debug / /actuator/conditions to see exactly what auto-configured and why.
