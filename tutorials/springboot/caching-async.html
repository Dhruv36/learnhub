Caching, Async & Scheduling Advanced

Three cross-cutting capabilities Spring provides through annotations backed by AOP proxies: cache expensive results, run work off the request thread, and schedule recurring jobs. All three are one-annotation easy — and all three share the same subtle proxy gotcha you must understand to use them correctly.

Caching — @Cacheable

@EnableCaching                      // on a @Configuration class
// ...
@Service
public class ProductService {
    @Cacheable("products")                          // cache by the method arguments (id)
    public Product findById(long id) {
        return repo.findById(id).orElseThrow();     // runs ONLY on a cache miss
    }
    @CachePut(value = "products", key = "#p.id")    // update the cache after a write
    public Product update(Product p) { return repo.save(p); }

    @CacheEvict(value = "products", key = "#id")    // remove on delete
    public void delete(long id) { repo.deleteById(id); }
}
⚡ The default ConcurrentMapCache is fine for a single instance but has no eviction or expiry. For real workloads use Caffeine (in-process, size/TTL-bounded) or Redis (shared across instances — see the Redis track). Cache invalidation is the hard part: pick a TTL you can tolerate serving stale, and evict on writes.

Async — @Async

@EnableAsync
// ...
@Service
public class NotificationService {
    @Async                                          // runs on a separate thread pool; returns immediately
    public void sendWelcomeEmail(User u) { /* slow SMTP call */ }

    @Async
    public CompletableFuture<Report> buildReport(long id) {   // return a future to get the result later
        return CompletableFuture.completedFuture(generate(id));
    }
}
// Caller returns to the client without waiting. Define an explicit Executor bean — the default
// SimpleAsyncTaskExecutor makes a new thread per call (unbounded, dangerous under load):
@Bean Executor taskExecutor() {
    var ex = new ThreadPoolTaskExecutor();
    ex.setCorePoolSize(4); ex.setMaxPoolSize(16); ex.setQueueCapacity(100);
    ex.initialize(); return ex;
}

Scheduling — @Scheduled

@EnableScheduling
// ...
@Component
public class Jobs {
    @Scheduled(fixedRate = 60_000)                  // every 60s (from start to start)
    public void refreshRates() { ... }

    @Scheduled(fixedDelay = 30_000)                 // 30s AFTER the previous run finishes
    public void poll() { ... }

    @Scheduled(cron = "0 0 3 * * *")                // 03:00 every day (sec min hour dom mon dow)
    public void nightlyCleanup() { ... }
}
🕒 Scheduling in a multi-instance deployment: every instance runs the job, so a nightly cleanup fires N times. For run-once-across-the-cluster semantics use ShedLock, a distributed lock, or a dedicated scheduler (Quartz clustered / an external cron). Naive @Scheduled assumes a single instance.

The Shared Gotcha — Proxy Self-Invocation

@Service
public class ReportService {
    public void run() {
        buildReport();          // ❌ direct in-class call — @Async/@Cacheable IGNORED!
    }
    @Async public void buildReport() { ... }
}
// @Async, @Cacheable, @Transactional, @Scheduled all work via an AOP PROXY that wraps the bean.
// Calling an annotated method from ANOTHER method of the SAME class bypasses the proxy, so the
// annotation does nothing. Fix: call through another bean, self-inject the proxy, or restructure.
🎯 This is the single most-asked "why isn't my annotation working?" question across @Transactional (data lesson), @Async, and @Cacheable. They all share the same root cause: Spring's proxy only intercepts calls that come through the proxy, not internal this.method() calls.

Interview Questions

Q1: Why does an annotation like @Cacheable "not work" when called from the same class?

Spring implements these annotations with AOP proxies: at startup it wraps the bean in a proxy that intercepts incoming calls to add the caching/async/transaction behavior, then delegates to your method. Interception only happens for calls that go through the proxy — i.e. calls from other beans. When a method calls another method of the same class via this, the call never leaves the target object, bypasses the proxy, and the annotation is ignored. Fixes: move the annotated method to a separate bean, inject the bean into itself (self-injection) and call through that reference, or use AspectJ load-time weaving which weaves the behavior into the class itself.

Q2: What must you configure for @Async in production?

Define an explicit, bounded Executor bean (a ThreadPoolTaskExecutor with core/max pool sizes and a queue capacity) and a rejection policy — otherwise Boot's default can create an unbounded number of threads and exhaust resources under load. Return a CompletableFuture when you need the result or want to handle failures (a fire-and-forget void @Async swallows exceptions unless you set an AsyncUncaughtExceptionHandler). Also remember the security context and request attributes don't propagate to async threads by default — propagate them explicitly if needed.

Q3: What breaks with @Scheduled when you scale to many instances?

By default every application instance has its own scheduler, so a scheduled job runs once per instance. For a nightly report or cleanup that must run exactly once cluster-wide, that means N duplicate runs — double emails, double charges, race conditions. Solutions: a distributed lock (ShedLock backed by the DB/Redis so only one instance executes each fire), a clustered scheduler (Quartz in clustered mode), or moving the job to a dedicated single runner / external cron triggering an endpoint. Plain @Scheduled is single-instance semantics.

🏋️ Practice: add @Cacheable to an expensive lookup and prove the DB is hit only on a miss (log SQL); back it with Caffeine and set a TTL; make an email send @Async with a bounded ThreadPoolTaskExecutor and confirm the request returns immediately; add a @Scheduled(cron=…) job; then reproduce the self-invocation bug (annotation ignored) and fix it by moving the method to another bean.

🎯 Key Takeaways

  • @Cacheable/@CachePut/@CacheEvict cache method results — use Caffeine or Redis, not the default, in prod.
  • @Async offloads work to a thread pool — always define a bounded executor; return a future for results/errors.
  • @Scheduled runs fixedRate/fixedDelay/cron jobs — needs a distributed lock across multiple instances.
  • All are AOP-proxy based, so self-invocation bypasses them — the #1 "my annotation is ignored" cause.
  • Cache invalidation and thread-pool sizing are the real engineering; the annotations are the easy part.
