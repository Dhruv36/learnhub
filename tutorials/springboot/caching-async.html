Caching, Async & Scheduling Cross-Cutting

What you'll master in this lesson: three declarative Spring abstractions that all ride the same AOP-proxy mechanism you met in the AOP lesson — @Cacheable for caching, @Async for off-thread work, and @Scheduled for periodic jobs. You'll learn what each annotation actually does, how to configure a real cache (Caffeine / Redis) with TTLs, how to give @Async its own thread pool and stop it silently swallowing exceptions, and how the same self-invocation and proxy gotchas from @Transactional bite all three. Prerequisites: AOP & Proxies (the proxy model) and Transactions (self-invocation).

Part 1 — @Cacheable: Skip the Work on a Repeat Call

@Service
public class BookService {
    @Cacheable("books")                              // first call runs the method; result stored under key = id
    public Book findById(long id) {
        return slowDatabaseLookup(id);               // ...NOT run again for the same id until evicted
    }

    @CachePut(value = "books", key = "#book.id")     // ALWAYS runs, then updates the cache with the return value
    public Book update(Book book) { return repo.save(book); }

    @CacheEvict(value = "books", key = "#id")        // removes the entry so the next read reloads
    public void delete(long id) { repo.deleteById(id); }
}

// Turn the machinery on once, at the app level:
@SpringBootApplication
@EnableCaching
public class App { public static void main(String[] a){ SpringApplication.run(App.class,a); } }
🎯 @Cacheable means "if I've computed this before for these arguments, return the stored result instead of running the method again." Spring wraps the bean in a proxy (AOP lesson) that, before each call, builds a key from the method arguments and checks the named cache: a hit returns the cached value and the method body never runs; a miss runs the method and stores the return value. @CachePut always runs and refreshes the entry (use it on updates), and @CacheEvict removes entries (use it on deletes/writes) so stale data isn't served. You must add @EnableCaching once to activate the proxies.

Part 2 — A Real Cache: Keys, TTLs and Providers

// The DEFAULT cache is a plain ConcurrentHashMap that NEVER evicts and NEVER expires — a memory leak
// waiting to happen and useless across multiple app instances. In production pick a real provider:

// Caffeine (in-process, high-performance) — great for a single instance:
@Bean
public CacheManager cacheManager() {
    CaffeineCacheManager m = new CaffeineCacheManager("books");
    m.setCaffeine(Caffeine.newBuilder()
        .maximumSize(10_000)                 // bounded — evicts least-recently-used past the limit
        .expireAfterWrite(Duration.ofMinutes(10)));   // TTL — entries expire so data can't go stale forever
    return m;
}

// Custom key when the default (all args) isn't right, plus conditional caching:
@Cacheable(value = "books", key = "#isbn", unless = "#result == null")
public Book byIsbn(String isbn) { ... }   // don't cache nulls
ProviderScopeUse when
CaffeineIn-process (per instance)Single instance or per-node cache is fine; lowest latency
RedisShared across instancesMultiple app instances must see the same cache; survive restarts (Redis track)
Default (ConcurrentHashMap)In-process, unboundedNever in production — no eviction, no TTL
⚠️ The default cache manager is unbounded and never expires — a memory leak and a correctness bug (stale data forever). Always configure a real provider with a size bound and a TTL. Choose Caffeine for a fast in-process cache when a per-instance copy is acceptable, or Redis when several app instances must share one cache (so an update on one node is seen by all) — with multiple instances, a per-instance cache means one node can serve data another already invalidated. The right eviction/TTL is a data-freshness decision: how stale can this value be?

Part 3 — @Async: Do It Off the Request Thread

@Service
public class EmailService {
    @Async                                            // returns immediately; runs on a background thread
    public void sendWelcome(String to) {
        slowSmtpCall(to);                             // the caller isn't blocked waiting for this
    }

    @Async
    public CompletableFuture<Report> buildReport(long id) {   // return a Future to get the result/errors later
        return CompletableFuture.completedFuture(generate(id));
    }
}

@SpringBootApplication
@EnableAsync                                          // activate the @Async proxies (like @EnableCaching)
public class App { ... }
🎯 @Async makes a method run on a different thread so the caller returns immediately instead of blocking. Use it for fire-and-forget side work — sending an email, pushing a notification, warming a cache — so an HTTP request doesn't wait on slow I/O it doesn't need to. A void @Async method is truly fire-and-forget; return CompletableFuture<T> when the caller needs the eventual result or must observe failures. Activate with @EnableAsync. For CPU-bound parallelism prefer the JDK's own tools (executors / structured concurrency) — @Async shines for I/O-bound side effects.

Part 4 — @Async's Two Traps: The Default Pool and Swallowed Exceptions

// TRAP 1 — the DEFAULT executor. Older Boot used SimpleAsyncTaskExecutor, which starts a NEW THREAD per
// task (no pooling) — unbounded thread creation under load. ALWAYS define your own bounded pool:
@Bean
public Executor taskExecutor() {
    ThreadPoolTaskExecutor ex = new ThreadPoolTaskExecutor();
    ex.setCorePoolSize(8);
    ex.setMaxPoolSize(16);
    ex.setQueueCapacity(500);                 // bounded queue — back-pressure instead of OOM
    ex.setThreadNamePrefix("async-");
    ex.initialize();
    return ex;
}

// TRAP 2 — a void @Async method that throws: the exception goes NOWHERE by default (silently lost).
@Bean
public AsyncUncaughtExceptionHandler asyncExceptionHandler() {
    return (ex, method, params) -> log.error("Async failure in {}", method, ex);   // now it's logged
}
// (return CompletableFuture instead and the exception surfaces when the caller calls .get()/.join())
🚨 Two things bite everyone with @Async. First, the default executor historically created a new thread per task (no pool), so a burst of work spawns unbounded threads and can exhaust memory — always register your own bounded ThreadPoolTaskExecutor and pick pool size and queue capacity deliberately so overload turns into back-pressure, not an OOM. Second, an exception thrown from a void @Async method has nowhere to go — the caller already moved on — so it's silently swallowed; wire an AsyncUncaughtExceptionHandler to log it, or return a CompletableFuture so the failure surfaces when the result is awaited. Silent async failures are among the nastiest production bugs precisely because nothing tells you they happened.

Part 5 — @Scheduled: Run It on a Timer

@Component
public class ReportJobs {
    @Scheduled(fixedRate = 60_000)        // every 60s measured start-to-start (may overlap if a run is slow)
    public void poll() { ... }

    @Scheduled(fixedDelay = 60_000)       // 60s AFTER the previous run finishes (never overlaps itself)
    public void drain() { ... }

    @Scheduled(cron = "0 0 2 * * *")      // 02:00 every day (sec min hour day month weekday)
    public void nightlyCleanup() { ... }
}

@SpringBootApplication
@EnableScheduling                          // activate @Scheduled
public class App { ... }
🕐 @Scheduled runs a method periodically. fixedRate fires every N ms measured start-to-start (runs can overlap if one is slow); fixedDelay waits N ms after the previous run finishes (never overlaps itself — safer for jobs that mustn't run concurrently); cron gives calendar scheduling. By default a single thread runs all scheduled tasks, so one slow job delays the others — configure a scheduling pool (spring.task.scheduling.pool.size) if you have several. Critical caveat for production: with multiple app instances, a plain @Scheduled job runs on every instance simultaneously — for "run this exactly once" cluster-wide jobs you need a distributed lock (e.g. ShedLock) or an external scheduler.

Part 6 — Deep Dive: One Proxy Mechanism, One Family of Gotchas

  • All three are AOP proxies (AOP lesson). @Cacheable, @Async and @Transactional are added by wrapping the bean at startup, so they only intercept calls that come through the proxy from another bean.
  • Self-invocation breaks all of them. Calling a @Cacheable / @Async method from another method of the same class via this.method() bypasses the proxy — the cache is ignored, the async never leaves the calling thread. Same root cause, same fixes as the transactions self-invocation gotcha: call through another bean or a self-injected proxy.
  • Cache invalidation is the hard part. "There are only two hard things in CS: cache invalidation and naming." A cache is only correct if every write that changes the underlying data also updates or evicts the cached copy (@CachePut/@CacheEvict) — and with a shared Redis cache, all instances see the eviction; with per-instance Caffeine, they don't. TTLs are your safety net for the invalidations you miss.
  • @Async loses the request context. The background thread has no SecurityContext, request-scoped beans, or MDC log context by default — you must propagate them explicitly if the async work needs them (a common surprise when security or per-request logging silently disappears off-thread).
🔍 Senior framing: caching, async execution and scheduling look like three unrelated features but are the same mechanism — declarative cross-cutting behavior added by an AOP proxy around your bean — which is why understanding the proxy model (AOP lesson) lets you predict how all three behave and, crucially, how they fail. Each one trades a specific complexity for its convenience. Caching trades correctness-under-change for speed: it's trivial to add @Cacheable and easy to serve stale data forever, so the real engineering is invalidation (evict/put on every mutating path) and bounding (size + TTL), plus choosing in-process (Caffeine, per-node) vs shared (Redis, cluster-wide, so evictions are seen everywhere) based on whether instances must agree. @Async trades simple control flow for throughput: it frees the request thread from slow side work, but the default per-task-thread executor invites unbounded thread growth (register a bounded ThreadPoolTaskExecutor), void methods swallow exceptions silently (add an AsyncUncaughtExceptionHandler or return a CompletableFuture), and the background thread loses security/request/log context unless you propagate it. @Scheduled trades manual timing for a declarative timer, but runs on a single thread by default (size the scheduling pool) and — the production killer — runs on every instance in a cluster, so "once daily" jobs need a distributed lock. Across all three, self-invocation bypasses the proxy exactly as it does for transactions, so an in-class call silently gets none of the behavior. The mature engineer therefore reaches for these annotations knowingly: enabling the machinery (@EnableCaching/@EnableAsync/@EnableScheduling), configuring the real resource behind it (bounded cache with TTL, bounded thread pool, sized scheduler), handling the failure modes (invalidation, swallowed exceptions, cluster duplication, lost context), and remembering that all of it is proxy-based — which is precisely the layered understanding interviewers probe when they ask "how does @Cacheable actually work and how would it go wrong in production?"

Interview Questions

Q1 (beginner): What does @Cacheable do?

It caches the result of a method keyed by its arguments: the first call with given arguments runs the method body and stores the return value in the named cache; subsequent calls with the same arguments return the stored value without executing the method (a cache hit), until the entry is evicted or expires. Spring implements it with an AOP proxy that checks the cache before invoking the method. You enable the machinery once with @EnableCaching. It's for expensive, repeatable, read-heavy computations (a slow lookup, an external call) whose results don't change on every request — you trade some staleness/memory for speed.

Q2 (beginner): What's the difference between @Cacheable, @CachePut and @CacheEvict?

@Cacheable returns the cached value if present and only runs the method on a miss (for reads). @CachePut always runs the method and then stores/updates the cache with the return value (for updates — keep the cache fresh after a write). @CacheEvict removes one or all entries from a cache (for deletes or any write that makes a cached value invalid). Together they let you keep the cache consistent with the underlying data: cache reads, update the cache on writes, evict on deletes. Forgetting to put/evict on the mutating paths is the classic cause of a cache serving stale data.

Q3 (mid): Why is the default cache manager unsafe for production, and what do you use instead?

The default backing store is a ConcurrentHashMap that is unbounded (never evicts) and has no TTL (never expires). Unbounded means it grows without limit — a memory leak that can OOM the process — and no expiry means entries can serve stale data indefinitely if you ever miss an invalidation. It's also strictly per-instance, so with multiple app nodes each has its own copy and they diverge. In production you configure a real provider: Caffeine for a fast in-process cache with a maximumSize bound and expireAfterWrite TTL when a per-node cache is acceptable, or Redis when multiple instances must share one cache so an eviction on any node is seen by all and the cache survives restarts. The choice hinges on whether instances must agree on cache contents and how stale the data may be.

Q4 (mid): What are the two main pitfalls of @Async?

First, the executor: the historical default (SimpleAsyncTaskExecutor) creates a new thread per task rather than pooling, so a burst of async work spawns unbounded threads and can exhaust memory. You should register your own ThreadPoolTaskExecutor with a bounded core/max pool size and a bounded queue, so overload produces back-pressure instead of an OOM. Second, exception handling: an exception thrown from a void @Async method has nowhere to propagate — the caller already returned — so it's silently swallowed, hiding failures. Fix by registering an AsyncUncaughtExceptionHandler (to at least log it) or by returning a CompletableFuture<T> so the exception surfaces when the caller awaits the result. Also note the background thread loses the security/request/log context by default. All of this rests on @Async being a proxy, so self-invocation bypasses it entirely.

Q5 (senior): Explain fixedRate vs fixedDelay, and what breaks with @Scheduled in a clustered deployment.

fixedRate schedules the next execution a fixed interval after the start of the previous one, so if a run takes longer than the interval, executions can overlap (or queue up on the single scheduler thread). fixedDelay schedules the next execution a fixed interval after the previous run completes, guaranteeing no overlap — the right choice for jobs that must not run concurrently with themselves (draining a queue, a batch that isn't idempotent under overlap). cron gives calendar-based scheduling. The clustering problem: a plain @Scheduled method runs on every application instance independently, because each instance has its own scheduler — so a "nightly cleanup" or "send monthly invoices" job runs N times, once per node, causing duplicate work, duplicate emails, or data corruption. Solutions are a distributed lock so only one instance actually runs each fire (e.g. ShedLock, which records a lock row/key the winning node holds for the job's duration), leader election, or delegating to an external scheduler/queue that dispatches the job to exactly one worker. This is one of the most common real-world Spring scheduling bugs, and it only appears once you scale past a single instance — so it must be designed for up front. Also remember all scheduled tasks share one thread by default, so size the scheduling pool if you have several.

Q6 (senior): These three annotations share a mechanism and a family of gotchas — explain.

@Cacheable, @Async and @Transactional are all implemented the same way: Spring wraps the bean in an AOP proxy at startup that intercepts calls and adds the cross-cutting behavior (check the cache / dispatch to another thread / open a transaction) around the method invocation. Because the behavior lives in the proxy, it only applies to calls that arrive through the proxy from another bean — so self-invocation (an in-class this.method() call to an annotated method) bypasses the proxy and silently gets none of the behavior: no caching, no async, no transaction. That's a single root cause producing the same confusing "the annotation does nothing" symptom across all three, with the same fixes (call through another bean or a self-injected proxy). Beyond that shared gotcha, each has its own: caching needs correct invalidation and bounding (and shared vs per-instance semantics); async needs a bounded pool, exception handling, and context propagation; scheduling needs cluster-safety. The senior insight is to see them as one declarative-AOP family — understand the proxy and you understand how all of them work, compose, and fail, rather than memorizing three separate feature sets. This is also why they can be enabled independently (@EnableCaching/@EnableAsync/@EnableScheduling): each installs its own proxy advice.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Cache a slow lookup. Add @Cacheable("books") to a method with an artificial delay and @EnableCaching; call it twice with the same argument and confirm the second call is instant.
Solution

The first call runs the method (and the delay); the second with the same argument is a cache hit and returns immediately without entering the body — proving the proxy short-circuits repeat calls. Without @EnableCaching, nothing is cached because the caching proxy is never installed.

2 (easy): Keep the cache fresh. Add an update method and make a stale read appear; then fix it with @CachePut or @CacheEvict.
Solution

After caching a read, updating the entity in the DB without touching the cache makes subsequent reads serve the old value. @CachePut(key="#book.id") on the update refreshes the cached entry with the new value, or @CacheEvict removes it so the next read reloads. This is cache invalidation — the hard part of caching.

3 (medium): Bound the async pool. Add @Async plus @EnableAsync, then register a ThreadPoolTaskExecutor with small core/max sizes and a bounded queue; explain why the default is dangerous.
Solution

Defining a bounded ThreadPoolTaskExecutor (core, max, queue capacity, thread name prefix) caps concurrency and turns overload into back-pressure. The historical default created a new thread per task with no pooling, so a burst of async calls spawns unbounded threads and can OOM the process — hence always supply your own bounded executor.

4 (medium): Catch a swallowed async exception. Make a void @Async method throw, observe nothing is logged, then wire an AsyncUncaughtExceptionHandler.
Solution

A void @Async method that throws loses the exception — the caller already returned, so it goes nowhere and no stack trace appears. Providing an AsyncUncaughtExceptionHandler (via AsyncConfigurer) logs it; alternatively returning a CompletableFuture makes the exception surface when .get()/.join() is called. Silent async failures are a classic production trap.

5 (hard): Cluster-safe schedule. Describe what happens when a @Scheduled "send daily digest" job runs on 3 instances, and design a fix.
Solution

Each instance's scheduler fires the job independently, so the digest is sent three times — duplicate emails and wasted work. Fix with a distributed lock so only one instance actually runs each fire: e.g. ShedLock records a lock (a DB row / Redis key) that the winning node holds for the job's duration and the others skip; alternatives are leader election or dispatching the job to a single worker via a queue. The duplication only shows up once you scale past one instance, so design for it up front.

6 (hard): Explain-it test. Out loud: (a) how @Cacheable works and how a cache goes stale/leaks; (b) the two @Async traps; (c) why self-invocation breaks all three annotations.
Self-check rubric

Pass (a) if you said proxy checks a keyed cache — hit skips the method — and named invalidation (evict/put on writes) + bounding (size/TTL). Pass (b) if you named the unbounded default executor and swallowed void exceptions with their fixes. Pass (c) if you explained all three are AOP proxies so an in-class call bypasses the proxy and silently gets no behavior. That completes the cross-cutting section.

🎯 Key Takeaways

  • @Cacheable returns a stored result on repeat calls (miss runs the method, hit skips it); enable with @EnableCaching.
  • The default cache is unbounded with no TTL — use Caffeine (per-instance) or Redis (shared) with a size bound and TTL, and evict/put on every write.
  • @Async runs work off the caller's thread — but supply a bounded ThreadPoolTaskExecutor and handle swallowed exceptions on void methods.
  • @Scheduled gives fixedRate/fixedDelay/cron — but a plain job runs on every clustered instance (needs a distributed lock).
  • All three are AOP proxies — self-invocation bypasses them, exactly like @Transactional.
