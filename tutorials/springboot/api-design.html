REST API Design & Versioning Event-Driven & Reactive

What you'll master in this lesson: the design decisions that separate a professional HTTP API from a pile of endpoints — resource modeling and correct status codes, pagination, filtering and sorting that don't fall over at scale, versioning strategies and how to evolve an API without breaking clients, idempotency for safe retries, and the senior principle that ties it together: an API is a long-lived contract, and backward compatibility is the whole game. Prerequisites: REST Controllers, Request Mapping, Error Handling, Java Records (for DTOs).

Part 1 — Resource Modeling & Correct Status Codes

// Model NOUNS (resources), not verbs; let HTTP methods be the verbs:
GET    /orders           // list          POST   /orders        // create
GET    /orders/42        // read one      PUT    /orders/42     // full replace
DELETE /orders/42        // delete        PATCH  /orders/42     // partial update
GET    /orders/42/items  // sub-resource (items of an order)
// ❌ /getOrders, /createOrder, /orders/42/delete  — verbs in the path are an anti-pattern
StatusMeansUse for
200 / 201 / 204OK / Created / No ContentRead/update / create (with Location) / delete
400 / 422Bad Request / UnprocessableMalformed syntax / valid syntax but semantically invalid
401 / 403Unauthenticated / ForbiddenNot logged in / logged in but not allowed
404 / 409Not Found / ConflictMissing resource / version or uniqueness conflict
429 / 503Too Many Requests / UnavailableRate-limited / overloaded or down (with Retry-After)
🎯 REST models resources (nouns) and uses HTTP methods as the verbs — and the status code is part of your API's contract, not an afterthought. Clients (and caches, proxies, monitoring) rely on status codes to behave correctly, so returning 200 with an error body, or 500 for a validation failure, actively breaks them. Distinguish the pairs that matter: 401 (who are you?) vs 403 (you can't do this); 400 (malformed) vs 422 (well-formed but invalid); 404 vs 409. Return 201 Created with a Location header pointing at the new resource. Getting status codes right is the cheapest way to make an API feel professional and interoperable.

Part 2 — Pagination, Filtering & Sorting (Don't Return Everything)

// NEVER return an unbounded collection — it will eventually OOM the server or the client.
@GetMapping("/orders")
public Page<OrderDto> list(@RequestParam(defaultValue="0") int page,
                           @RequestParam(defaultValue="20") int size,
                           @RequestParam(required=false) OrderStatus status) {
    return service.find(status, PageRequest.of(page, size, Sort.by("createdAt").descending()));
}
// Spring Data returns a Page: content + totalElements + totalPages (offset pagination).

// At large scale / high offsets, prefer KEYSET (cursor) pagination — "items after this id/timestamp":
GET /orders?after=2026-07-18T10:00:00Z&size=20   // stable, no expensive OFFSET, no drift under inserts
⚠️ A list endpoint that returns everything is a latent outage — always paginate. Offset pagination (page/size, Spring's Pageable/Page) is simple and fine for moderate data, but it degrades at large offsets (the database still scans and discards all skipped rows) and can drift — rows inserted/deleted between page requests shift items across page boundaries so users see duplicates or gaps. For large or fast-changing datasets, keyset (cursor) pagination — "give me N items after this key" — is stable and cheap because it seeks by an indexed key instead of counting an offset. Expose filtering and sorting as query parameters, but only allow sorting on indexed columns (an unindexed sort on a huge table is a performance trap).

Part 3 — Versioning: Evolving Without Breaking Clients

StrategyLooks likeTrade-off
URI path/v1/orders, /v2/ordersSimple, visible, cache-friendly; "un-RESTful" (URI shouldn't encode version) — most common in practice
Header / media typeAccept: application/vnd.myapp.v2+jsonKeeps URIs clean; harder to test/discover
Query param/orders?version=2Easy but muddies caching/routing
// The REAL goal is to avoid breaking changes so you rarely need a new version:
// NON-breaking (safe, no version bump):  add a new optional field, add a new endpoint, add an enum value clients ignore.
// BREAKING (needs a new version):  remove/rename a field, change a type, make an optional field required,
//                                  change status-code semantics, tighten validation.
🔑 Versioning strategy matters less than a discipline of backward compatibility. Pick one scheme and apply it consistently — URI path versioning (/v1, /v2) is the pragmatic industry default (visible, easy to route and cache, trivial to test), even though purists note a URI ideally shouldn't encode a version; header/media-type versioning is "more RESTful" but harder to work with. But the deeper skill is minimizing breaking changes so you seldom need a new version at all: adding optional fields and new endpoints is safe (tolerant clients ignore unknown fields), whereas removing/renaming fields, changing types, or tightening validation breaks existing callers. When you must break, run versions in parallel and deprecate the old one on a published timeline rather than cutting clients off.

Part 4 — Idempotency & Safe Retries

// HTTP method semantics your clients (and the network) rely on:
//   GET, PUT, DELETE = idempotent (repeating has the same effect)     GET = also safe (no side effects)
//   POST            = NOT idempotent by default (repeating creates duplicates)

// Make POST safe to retry with an Idempotency-Key (clients WILL retry on timeouts):
@PostMapping("/payments")
public Receipt pay(@RequestHeader("Idempotency-Key") String key, @RequestBody PayRequest req) {
    return store.find(key)                       // seen this key before? return the SAME result
                .orElseGet(() -> store.save(key, process(req)));   // else process once and record it
}
🎯 Because the network makes clients retry, mutating endpoints must be safe to retry — and that means idempotency. A client whose POST /payments times out doesn't know whether the charge happened, so it retries; without protection, it charges twice. PUT and DELETE are idempotent by design (setting a resource to a value, or deleting it, twice has the same end state), but POST creates a new resource each time. The standard fix is an Idempotency-Key: the client sends a unique key, the server records the result against that key and returns the same result on a repeat — so a retry is a no-op. This is the same idempotency principle as cross-service calls in the Resilience lesson, applied at the API boundary.

Part 5 — DTOs, Errors & Documentation as Contract

// Expose DTOs, NEVER your JPA entities — decouple the API contract from the database schema:
record OrderDto(Long id, String status, BigDecimal total) {}   // (Java Records lesson)
// Entities leak internals (lazy proxies, relations), couple the API to the DB, and risk over-exposure.

// Consistent, machine-readable errors (RFC 7807 Problem Details, built into Spring 6):
{ "type": "https://api/errors/insufficient-funds", "title": "Insufficient funds",
  "status": 409, "detail": "Balance 20 < amount 50", "instance": "/payments/42" }

// Document with OpenAPI (springdoc) — the spec IS part of the contract, generated from the code.
🔧 Return DTOs, standardize your error shape, and treat the OpenAPI spec as part of the contract. Serializing JPA entities directly couples your public API to your database schema (a column rename becomes a breaking API change), leaks internals, and triggers lazy-loading surprises — a dedicated DTO (a Java record) is the clean boundary. Make errors consistent and machine-readable with RFC 7807 Problem Details (native in Spring 6 via ProblemDetail), so clients can handle failures programmatically instead of parsing ad-hoc messages (ties to the Error Handling lesson). And generate an OpenAPI document (springdoc) from your controllers — it's living documentation, enables client generation, and makes the contract explicit and testable.

Part 6 — Deep Dive: An API Is a Contract You Can't Easily Take Back

  • Publishing an API is a long-term commitment. Once clients depend on it, you can add but rarely remove, and every breaking change imposes coordinated work on everyone who integrated. Design as if you'll support it for years — because you will.
  • Design from the consumer's perspective. Model resources and errors around what clients need to accomplish, use conventional status codes and predictable shapes, and paginate everything — the goal is an API that behaves the way an experienced integrator expects.
  • Backward compatibility beats versioning. The best version is the one you never had to create; additive, tolerant evolution keeps a single version healthy far longer than a proliferation of /v2, /v3 you must all maintain.
  • Assume retries and misuse. The network retries, clients send bad input, and callers hold old assumptions — idempotency, validation, and clear errors are how a robust API copes.
🔍 Senior framing: the defining shift from writing endpoints to designing APIs is treating the API as a durable public contract rather than an implementation detail — because the moment external clients depend on it, you largely lose the freedom to change it, and every breaking change becomes a coordinated migration imposed on everyone who integrated. That reframing drives every concrete practice. You model resources as nouns with HTTP methods as verbs and treat status codes as part of the contract (the 401/403, 400/422, 404/409 distinctions, 201+Location on create), because clients, caches, and monitoring all key off them and a wrong code silently breaks them. You paginate every collection — an unbounded list is a latent outage — reaching for keyset/cursor pagination at scale where offset pagination degrades and drifts. You choose a versioning strategy (URI path being the pragmatic default) but understand it matters far less than a discipline of backward compatibility: additive changes (new optional fields, new endpoints) are safe for tolerant clients, while removals, renames, type changes, and tightened validation are breaking and demand a new version run in parallel with a deprecation timeline — so the real skill is evolving the API so you rarely need to break it. You design for the reality that the network retries and clients misbehave, which makes idempotency (Idempotency-Key on POST; PUT/DELETE idempotent by design) essential so a timed-out retry can't double-charge, and makes validation and clear, machine-readable errors (RFC 7807 Problem Details) part of robustness. And you keep the contract decoupled from internals by exposing DTOs rather than JPA entities (so a schema change isn't an API break) and by treating the OpenAPI spec as generated, living documentation. The throughline is consumer-centric, compatibility-first design: an API engineered to be predictable, safe under retries, and evolvable without breaking the people who depend on it — which is exactly the judgment interviewers probe when they ask you to design or version an endpoint, because it reveals whether you think of an API as code you wrote or as a promise you must keep.

Interview Questions

Q1 (beginner): What makes an API "RESTful" in its URLs and methods?

REST models resources — nouns — as URLs, and uses the HTTP methods as the verbs acting on them: GET /orders lists, GET /orders/42 reads one, POST /orders creates, PUT /orders/42 replaces, PATCH partially updates, DELETE /orders/42 removes, and sub-resources nest (/orders/42/items). So you should not put verbs in the path (/getOrders, /createOrder, /orders/42/delete) — the method already expresses the action. This convention makes the API predictable: an experienced integrator can guess the endpoints, and tools/caches/proxies that understand HTTP semantics (e.g. that GET is safe and cacheable, DELETE is idempotent) behave correctly. Consistent resource modeling plus correct method usage is the baseline of a professional API.

Q2 (beginner): Why are correct HTTP status codes important?

Because the status code is part of the API's contract that clients, caches, proxies, and monitoring all rely on to behave correctly — it's not cosmetic. If you return 200 OK with an error message in the body, a client's success/failure logic breaks (it thinks the call succeeded) and caches may cache an error; if you return 500 for a user validation error, monitoring pages an on-call engineer for what is actually a client mistake, and the client can't distinguish "I sent bad input" from "the server is broken." Using the right codes — 201 with a Location header on create, 204 for a body-less success, 400 vs 422 for malformed vs semantically-invalid input, 401 vs 403 for unauthenticated vs forbidden, 404 vs 409, 429 with Retry-After for rate limiting — lets every consumer respond appropriately without parsing bodies. It's the cheapest, highest-leverage way to make an API correct and interoperable.

Q3 (mid): How do you paginate a large collection, and when is offset pagination a problem?

Never return an unbounded collection — it risks OOM and huge, slow responses — so every list endpoint paginates. The simple approach is offset pagination: page and size parameters (Spring Data's Pageable/Page, which also returns total counts), which is easy and fine for moderate data. It has two problems at scale. First, performance: to return page N the database typically still scans and discards all the rows before the offset, so deep pages (large offsets) get progressively slower on big tables. Second, drift: because the offset is positional, rows inserted or deleted between requests shift items across page boundaries, so a user paging through can see duplicates or miss items. The fix for both is keyset (cursor) pagination — "give me N items after this key" (an indexed id or timestamp) — which seeks directly to the position via the index (fast regardless of depth) and is stable under concurrent inserts/deletes because it anchors on a value, not a count. The trade-off is that keyset pagination doesn't easily support jumping to an arbitrary page number, so offset is fine for small admin lists while keyset is preferred for large or fast-changing, infinitely-scrolled data. You'd also restrict sorting to indexed columns to avoid an expensive unindexed sort.

Q4 (mid): What changes are backward-compatible vs breaking, and why does it matter?

It matters because once clients depend on your API you can't change it freely — a breaking change forces coordinated updates on every integrator and, if deployed without a new version, breaks them in production. Backward-compatible (safe) changes are additive and don't invalidate existing client assumptions: adding a new optional field to a response (tolerant clients ignore unknown fields), adding a new endpoint, adding a new accepted optional request parameter, or adding a new enum value clients can ignore. Breaking changes invalidate what existing clients rely on: removing or renaming a field, changing a field's type or meaning, making a previously-optional field required, tightening validation so previously-accepted requests now fail, changing status-code semantics, or changing default behavior. The practical discipline is to prefer additive evolution so a single version stays healthy for a long time, and when a genuine breaking change is unavoidable, introduce a new version, run old and new in parallel, and deprecate the old on a published timeline rather than cutting clients off. Recognizing which bucket a change falls into — and defaulting to compatible changes — is core API-evolution skill and directly reduces how often you need to version at all.

Q5 (senior): Compare API versioning strategies and explain why versioning discipline matters more than the scheme.

The main strategies: URI path versioning (/v1/orders, /v2/orders) is simple, highly visible, trivial to route/cache/test, and by far the most common in practice — its theoretical downside is that purists argue a URI identifies a resource and shouldn't encode a representation version. Header / media-type versioning (Accept: application/vnd.myapp.v2+json) keeps URIs clean and is considered more "RESTful," but it's harder to explore, test, and cache, and easy for clients to get wrong. Query-parameter versioning (?version=2) is easy but muddies caching and routing. Each is defensible and the choice is mostly pragmatic — I'd usually pick URI path for its operational simplicity. But the senior point is that which scheme you choose matters far less than the discipline around it. Versioning is a heavy hammer: every live version is code and behavior you must maintain, test, and secure indefinitely, so a proliferation of versions is a real maintenance burden. The goal is to need new versions rarely by evolving the API in backward-compatible ways (additive fields, new endpoints, tolerant readers) so a single version serves for years; and when a breaking change is truly necessary, to manage it well — run the new version alongside the old, communicate a deprecation timeline, give clients time to migrate, and monitor usage of the old version before retiring it. An engineer who reflexively spins up /v2 for every change creates sprawl; one who treats a new version as a last resort, minimizes breaks, and manages deprecation deliberately keeps the API healthy and its consumers happy. So versioning is really a backward-compatibility and change-management discipline with a chosen syntax, not a syntax choice — which is exactly the maturity interviewers are checking.

Q6 (senior): Why do mutating endpoints need idempotency, and how do you implement it for POST?

Because the network is unreliable and clients therefore retry, and a mutating request that isn't safe to retry causes duplicate side effects. The concrete scenario: a client sends POST /payments, the request succeeds on the server but the response is lost (a timeout, a dropped connection), so the client — unable to tell whether the charge happened — retries, and without protection the server charges the customer twice. GET is safe (no side effects) and PUT/DELETE are idempotent by design (setting a resource to a value or deleting it repeatedly yields the same end state), but POST creates a new resource on each call and so is not idempotent. The standard implementation is an Idempotency-Key: the client generates a unique key (e.g. a UUID) per logical operation and sends it as a header; the server, before processing, checks whether it has already handled that key — if so, it returns the stored original result without re-executing the side effect; if not, it processes the request once and records the result against the key (atomically, to handle concurrent duplicates). The result is that any number of retries of the same logical request produce exactly one charge and return a consistent response. This is the same idempotency principle used for cross-service calls in a distributed system, applied at the public API boundary, and it's essential for any endpoint that moves money, creates orders, or has other non-repeatable effects. The senior framing is that you must design mutating endpoints assuming clients will retry — idempotency isn't an edge case, it's a precondition for correctness under real network conditions — and that combined with clear status codes and validation, it's what makes an API robust rather than merely functional in the happy path.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Fix the endpoints. Rewrite POST /createOrder, GET /getOrder/42, POST /orders/42/delete RESTfully with correct status codes.
Solution

POST /orders → 201 + Location; GET /orders/42 → 200 (or 404); DELETE /orders/42 → 204. Nouns as resources, HTTP methods as verbs, semantically correct codes.

2 (easy): Paginate. Turn an unbounded GET /orders into a paginated endpoint and say why the original is dangerous.
Solution

Accept page/size, return a Page via PageRequest.of(...). The unbounded version can return millions of rows, OOMing the server/client and producing huge slow responses — always cap with pagination.

3 (medium): Classify changes. For each — add optional field, rename a field, make an optional field required, add an endpoint — say breaking or not.
Solution

Add optional field: safe. Rename field: breaking. Make optional required: breaking. Add endpoint: safe. Additive changes are compatible; removals/renames/tightening are breaking and need a new version.

4 (medium): DTO vs entity. Explain two concrete problems with returning a JPA entity from a controller.
Solution

(1) It couples the API to the DB schema — a column rename becomes a breaking API change. (2) Lazy associations trigger serialization/lazy-loading errors and can over-expose internal fields/relations. A DTO record is the clean, decoupled contract.

5 (hard): Idempotent payments. Design POST /payments so a client retry after a timeout doesn't double-charge.
Solution

Require an Idempotency-Key header; store the result keyed by it; on a repeat key return the stored result without re-charging, recording the first result atomically to handle concurrent duplicates. Retries become no-ops — exactly-once effect under an at-least-once network.

6 (hard): Explain-it test. Out loud: (a) resources + status codes as contract; (b) offset vs keyset pagination; (c) why backward compatibility beats versioning and why POST needs idempotency.
Self-check rubric

Pass (a) if you said nouns+methods and codes clients rely on. Pass (b) if you said offset is simple but slow/drifts at scale, keyset seeks by indexed key and is stable. Pass (c) if you said additive evolution avoids version sprawl and POST retries double-charge without an idempotency key. That's API-design maturity.

🎯 Key Takeaways

  • Model resources (nouns) with HTTP methods as verbs; treat status codes as contract (401/403, 400/422, 404/409, 201+Location).
  • Always paginate — offset for moderate data, keyset/cursor for scale (stable, no expensive offset).
  • Pick a versioning scheme (URI path is pragmatic) but backward compatibility beats versioning — additive changes are safe, removals/renames break.
  • Mutating endpoints must be safe to retry — PUT/DELETE are idempotent; give POST an Idempotency-Key.
  • Expose DTOs, not entities; standardize errors (RFC 7807); treat the OpenAPI spec as living contract.
