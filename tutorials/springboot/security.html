Spring Security Cross-Cutting

What you'll master in this lesson: Spring Security's mental model — a filter chain that answers "who are you?" (authentication) and "are you allowed?" (authorization) on every request; how to configure it with the modern SecurityFilterChain bean; stateless JWT authentication for REST APIs; password hashing done right; method-level authorization; and the CSRF/session decisions that are real security choices, not boilerplate. Security is Spring's steepest learning curve and its most important feature — this lesson demystifies the chain and builds a correct token-based setup.

Part 1 — The Mental Model: A Filter Chain

// Every request passes through a chain of security filters BEFORE reaching your @RestController:
//
//  request ─► [CORS] ─► [CSRF] ─► [Authentication filter] ─► [Authorization check] ─► your controller
//                                        │                          │
//                             "who are you?" (build an           "are you allowed?"
//                              Authentication from a JWT/          (role/authority vs the
//                              session cookie)                      required permission)
//
// Fail authentication → 401 (we don't know who you are).
// Fail authorization  → 403 (we know who you are, but you're not permitted).
💡 Spring Security is fundamentally a chain of servlet filters sitting in front of your app. Two questions get answered on every request: authentication (who is the caller — verify credentials and build an Authentication) and authorization (may this caller access this resource — check their authorities against the rule for the matched URL/method). If either fails, the chain short-circuits before your controller runs. Internalizing "it's a filter chain answering authn then authz" makes the whole framework tractable instead of magical.

Part 2 — 401 vs 403

StatusMeaningCause
401 UnauthorizedNot authenticatedMissing/invalid credentials — "we don't know who you are"
403 ForbiddenAuthenticated but not permittedKnown identity lacks the required authority — "you can't do this"
🎯 401 = who are you? (authentication failed). 403 = you're known but not allowed (authorization failed). Mixing these up is a classic bug and a guaranteed interview question. The chain returns 401 when the authentication filter can't establish identity and 403 when an authenticated principal fails an authorization rule.

Part 3 — Configuring the SecurityFilterChain (modern DSL)

@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    SecurityFilterChain chain(HttpSecurity http, JwtAuthFilter jwtFilter) throws Exception {
        http
          .csrf(csrf -> csrf.disable())                         // OK for a stateless token API (Part 6)
          .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
          .authorizeHttpRequests(auth -> auth
              .requestMatchers("/api/auth/**", "/actuator/health").permitAll()
              .requestMatchers(HttpMethod.GET, "/api/products/**").permitAll()
              .requestMatchers("/api/admin/**").hasRole("ADMIN")
              .anyRequest().authenticated())
          .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }
    @Bean PasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(); }  // NEVER plaintext
}
🔍 Modern Spring Security is configured by declaring a SecurityFilterChain bean and building rules with the lambda DSL (the old WebSecurityConfigurerAdapter is deprecated). You declare URL authorization rules (permitAll, hasRole, authenticated), the session policy, CSRF handling, and add your own filters. Rules are evaluated top-to-bottom, so order matters — put specific rules before broad ones, and end with anyRequest().authenticated() as a deny-by-default catch-all.

Part 4 — Password Hashing & JWT Authentication

// PASSWORD STORAGE — hash with BCrypt (adaptive, salted); NEVER store or log plaintext:
String hash = passwordEncoder.encode(rawPassword);           // store `hash`
boolean ok = passwordEncoder.matches(rawPassword, storedHash); // verify (one-way, can't decrypt)

// JWT FLOW for a stateless REST API:
// 1. POST /api/auth/login {email, password} → verify with matches() → issue a SIGNED JWT.
// 2. Client sends it on every request:  Authorization: Bearer <token>
// 3. A once-per-request filter validates the signature + expiry and sets the SecurityContext:
public class JwtAuthFilter extends OncePerRequestFilter {
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain) {
        String token = extractBearer(req);
        if (token != null && jwt.isValid(token)) {
            var auth = jwt.toAuthentication(token);          // subject + roles from the claims
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        chain.doFilter(req, res);
    }
}
🔐 Store passwords hashed with BCrypt (or Argon2), never plaintext or reversible encryption — a breach of hashed passwords is far less catastrophic, and BCrypt is adaptive (tunable work factor) and salted. For a stateless REST API, authenticate with a signed JWT sent in the Authorization: Bearer header: the server verifies the signature to trust the token's claims without a session store, so it scales horizontally. Keep tokens short-lived and use a refresh token for renewal — and remember the JWT payload is signed, not encrypted, so it's readable: never put secrets in the claims.

Part 5 — Method-Level Authorization

@Configuration
@EnableMethodSecurity                              // turn on annotation-based method security
class MethodSecurityConfig {}

@Service
class UserService {
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteUser(long id) { ... }

    @PreAuthorize("#userId == authentication.name or hasRole('ADMIN')")   // owner OR admin
    public UserDto get(long userId) { ... }
}
// Defence in depth: URL rules in the filter chain + method rules on sensitive operations.
🎯 @PreAuthorize (with @EnableMethodSecurity) adds authorization checks at the method level, expressed with SpEL — including data-aware rules like "the caller is the owner or an admin." This is defence in depth: URL-level rules in the filter chain are the first gate, and method-level rules protect sensitive operations directly, so a mistake in one layer is caught by the other. (Method security is proxy-based — the AOP lesson — so it's subject to the self-invocation caveat.)

Part 6 — Deep Dive: CSRF, Sessions & Security as Server-Side Enforcement

  • CSRF: disable only for stateless token APIs. CSRF attacks rely on the browser auto-attaching an ambient credential (a session cookie) to a forged cross-site request. A stateless API that authenticates via a token in the Authorization header has no such ambient credential, so disabling CSRF is safe there. If you authenticate with session cookies, you MUST keep CSRF protection on — blanket-disabling it is a real vulnerability.
  • Stateless vs session. JWT/token auth is stateless (no server session — scales horizontally) but revocation is hard (a valid token is trusted until expiry). Session auth is stateful (server tracks sessions — easy revocation) but needs sticky sessions or a shared store to scale, and requires CSRF protection. Choose deliberately per system.
  • Authorization is always server-side. UI-side checks (hiding a button) are UX only — anyone can call the API directly. Every authorization decision must be enforced on the server (filter chain + method security), never trusted from the client.
  • Never store secrets in a JWT. The payload is signed for integrity, not encrypted for confidentiality — it's base64, readable by anyone with the token. Put only non-sensitive claims (subject, roles, expiry) in it.
🔍 Senior framing: Spring Security is best understood as a filter chain that enforces, on every request and entirely server-side, the two questions of authentication (who is the caller) and authorization (what they may do), and configuring it well is a series of deliberate security decisions rather than boilerplate. You define a SecurityFilterChain with deny-by-default URL rules (specific-before-broad, ending in anyRequest().authenticated()), hash passwords with an adaptive salted algorithm like BCrypt and never store or log them in the clear, and for a REST API typically choose stateless authentication with a signed, short-lived JWT carried in the Authorization header — verifying its signature to trust its claims without a session store, while remembering the token is readable (signed, not encrypted, so no secrets in the claims) and hard to revoke (hence short expiry plus refresh tokens). You layer method-level @PreAuthorize checks as defence in depth on sensitive operations, and you make the CSRF and session decisions consciously: disable CSRF only for stateless header-token auth that has no ambient cookie credential, and keep it on for cookie/session-based auth where a forged cross-site request would otherwise ride the user's session. The recurring, correctness-critical points interviewers probe are the 401-vs-403 distinction (unauthenticated vs authenticated-but-forbidden), why and when CSRF can be disabled, the stateless-vs- session trade-off (scalability vs revocation), password hashing, and the absolute rule that authorization is enforced on the server and never trusted from the UI — because these reveal whether a candidate understands security as a real, layered, server-side enforcement problem with genuine trade-offs, or merely as annotations to copy. The mature engineer treats every one of these as a decision with security consequences, understands the filter-chain mechanism well enough to reason about where a request is accepted or rejected and why, and defends in depth so that no single misconfiguration exposes the system — which is exactly the difference between a service that is secure by design and one that ships an exploitable default.

Interview Questions

Q1 (beginner): What's the difference between authentication and authorization?

Authentication establishes who the caller is — verifying credentials (a password, a token, a session) and producing an identity (an Authentication in Spring's SecurityContext). Authorization decides what that identity is allowed to do — checking its roles/authorities against the permission required for a URL or method. In HTTP terms, failing authentication yields 401 (we don't know who you are) and failing authorization yields 403 (we know who you are, but you're not permitted). Spring Security's filter chain does authentication first (build the identity from a JWT or session) and then authorization (compare that identity's authorities to the rule for the requested resource).

Q2 (beginner): What is 401 vs 403?

401 Unauthorized means authentication failed or is missing — the request lacks valid credentials, so the server doesn't know who the caller is; the client typically needs to log in or supply a valid token. 403 Forbidden means the caller is authenticated (identity established) but lacks permission for this specific resource — a known user without the required role/authority. So 401 is about identity ("who are you?") and 403 is about permission ("you can't do this"). In Spring's chain, the authentication filter produces a 401 when it can't establish identity, and an authorization failure on an authenticated principal produces a 403.

Q3 (mid): How does JWT authentication work for a stateless REST API?

The client authenticates once (e.g. POST credentials to a login endpoint), the server verifies them and issues a signed JWT containing claims like the subject and roles plus an expiry. The client then sends that token on every subsequent request in the Authorization: Bearer header. A once-per-request filter extracts the token, validates its signature (proving the server issued it and it wasn't tampered with) and its expiry, and if valid builds an Authentication from the claims and places it in the SecurityContext, so the request proceeds as that identity. Because the signature lets any server instance verify the token without shared session state, authentication is stateless and scales horizontally. The trade-offs: the token is signed but not encrypted, so its contents are readable — never put secrets in it; and revocation is difficult because a valid signature is trusted until expiry, so tokens should be short-lived with a refresh-token mechanism (or a denylist) for logout/rotation.

Q4 (mid): When is it safe to disable CSRF protection?

CSRF (Cross-Site Request Forgery) attacks work by tricking a victim's browser into making an authenticated request to your site using an ambient credential the browser attaches automatically — classically a session cookie. The defense (a synchronizer token, or SameSite cookies) ensures a forged cross-site request can't succeed. It's safe to disable CSRF only when there is no such ambient credential to exploit: a stateless API that authenticates via a token sent explicitly in the Authorization header, because a cross-site page cannot read or set that header on the victim's behalf, so there's nothing for CSRF to ride on. If your app authenticates with session cookies (or any cookie the browser sends automatically), you must keep CSRF protection enabled — disabling it there is a genuine vulnerability. So the rule is: header-token, cookie-less auth → CSRF can be off; cookie/session auth → CSRF must stay on. Blanket-disabling CSRF "because it's a REST API" without confirming the auth is truly stateless and cookie-free is a common mistake.

Q5 (senior): Compare stateless (JWT) and session-based authentication and their trade-offs.

Session-based authentication stores authentication state on the server: after login the server creates a session (with the user's identity) and gives the browser a session-id cookie, which is sent on each request and looked up server-side. It's simple, and revocation is trivial (invalidate the session), but it's stateful — to scale across multiple servers you need sticky sessions or a shared session store (e.g. Redis), and because it relies on a cookie the browser sends automatically, it requires CSRF protection. Token-based (JWT) authentication is stateless: after login the server issues a signed token containing the claims, the client sends it in the Authorization header, and any server instance can verify the signature and read the claims without shared state — so it scales horizontally trivially and fits a distributed/microservices architecture, and (as header-based auth) it avoids the CSRF concern. Its costs are the mirror image: revocation is hard because a valid signature is trusted until the token expires, so you can't easily "log someone out" server-side — you mitigate with short expiries plus refresh tokens (and optionally a denylist of revoked tokens, which reintroduces some state), which adds complexity; the token is readable (signed, not encrypted), so no secrets in claims and larger tokens travel on every request; and clock/expiry handling and key management (signing keys, rotation) must be done correctly. The choice depends on the system: sessions are fine and simple for a traditional server-rendered app or a monolith where revocation matters and scale is modest; stateless tokens suit APIs consumed by SPAs/mobile clients and distributed backends where horizontal scale and cross-service verification matter more than easy revocation. Many real systems combine ideas (short-lived access tokens plus server-side refresh tokens, or an opaque token validated against an auth server) to balance statelessness with revocability. The senior point is that this is a deliberate trade-off between scalability and control (revocation, state) with concrete security implications (CSRF exposure, token leakage, key management), not a default to pick without reasoning about the deployment and threat model.

Q6 (senior): Why must authorization be enforced server-side, and how does defence in depth apply?

Authorization must be enforced on the server because the server is the only component that controls access to the actual resources and is the only one the client cannot tamper with. Anything done in the client — hiding a button, disabling a menu, not rendering an admin page — is purely cosmetic: an attacker (or any user) can bypass the UI entirely and call the API directly with curl, Postman, a script, or a modified client, so client-side "authorization" enforces nothing and provides only UX convenience. Every access decision — can this identity read this record, invoke this operation, see this field — has to be checked on the server before the resource is touched, or it effectively doesn't exist. Defence in depth means enforcing authorization at multiple layers so that a gap in one is caught by another. In Spring that typically combines: coarse-grained URL-level rules in the SecurityFilterChain (deny by default, restrict admin paths to admins, etc.) as the first gate every request passes; method-level @PreAuthorize/@PostAuthorize checks on sensitive service operations, including data-aware rules ("the caller owns this resource or is an admin") that URL rules can't express; and often data-level checks (query scoping so a user can only fetch their own rows, preventing insecure direct object references). The layers reinforce each other — if a new endpoint is added without a URL rule, method security still guards the operation; if a method rule is too broad, data scoping still limits exposure — so a single misconfiguration doesn't open the system. Additional depth comes from validating input, using DTOs that expose only intended fields (preventing mass assignment and over-exposure), and never trusting client-supplied identifiers of "who" the caller is (derive identity from the authenticated principal, not a request parameter). The senior instinct is that authorization is a server-side invariant enforced redundantly at the boundaries where resources are accessed, precisely because clients are untrusted and any one check can be missed or bypassed — which is exactly the layered, fail-safe posture that separates a genuinely secured application from one that merely looks secured in the UI.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Filter chain rules. Write a SecurityFilterChain that permits /api/auth/**, requires auth everywhere else, and restricts /api/admin/** to ROLE_ADMIN. Explain rule ordering.
Solution

authorizeHttpRequests with .requestMatchers("/api/auth/**").permitAll(), .requestMatchers("/api/admin/**").hasRole("ADMIN"), then .anyRequest().authenticated(). Order matters — rules evaluate top-to-bottom, so specific matchers must precede the broad catch-all; ending with authenticated() gives deny-by-default.

2 (easy): Hash and verify. Use BCryptPasswordEncoder to hash a password on registration and verify it on login. Explain why you can't "decrypt" the stored hash.
Solution

encoder.encode(raw) to store, encoder.matches(raw, hash) to verify. BCrypt is a one-way, salted, adaptive hash — you can't reverse it to recover the password; verification re-hashes the input and compares. That's why a hashed-password breach is far less severe than a plaintext one.

3 (medium): JWT filter. Sketch a OncePerRequestFilter that reads the Bearer token, validates it, and sets the SecurityContext. Note where it goes in the chain.
Solution

Extract the token from the Authorization header, validate signature+expiry, build an Authentication from the claims, and SecurityContextHolder.getContext().setAuthentication(auth) if valid, then continue the chain. Add it with addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class) so it runs before the default auth filter. It runs once per request and establishes identity for the authorization step.

4 (medium): Method security. Protect a deleteUser with @PreAuthorize("hasRole('ADMIN')") and a get with an owner-or-admin rule. Verify 403 for a non-admin.
Solution

With @EnableMethodSecurity, @PreAuthorize("hasRole('ADMIN')") on delete and @PreAuthorize("#userId == authentication.name or hasRole('ADMIN')") on get. A non-admin calling delete gets 403 (authenticated but not permitted); the owner or an admin can call get. Defence in depth alongside URL rules. (Method security is proxy-based — watch self-invocation.)

5 (hard): CSRF decision. For (a) a stateless JWT API and (b) a session-cookie web app, decide whether to disable CSRF and justify each in terms of ambient credentials.
Solution

(a) JWT in the Authorization header: safe to disable CSRF — there's no ambient credential a cross-site page can cause the browser to attach (it can't set that header), so there's nothing for CSRF to exploit. (b) Session-cookie app: keep CSRF on — the browser auto-attaches the session cookie to forged cross-site requests, which is exactly the CSRF attack, so the protection (token/SameSite) is required. The deciding factor is whether an ambient, auto-sent credential exists.

6 (hard): Explain-it test. Out loud: (a) the filter-chain model and 401 vs 403; (b) stateless JWT auth and its revocation trade-off; (c) when CSRF can be disabled and why authz is server-side.
Self-check rubric

Pass (a) if you said a filter chain does authn then authz, 401=unauthenticated, 403=forbidden. Pass (b) if you said signed token in the header, stateless/scalable, hard to revoke → short expiry + refresh. Pass (c) if you tied CSRF-disable to stateless header-token (no ambient cookie) and said authorization must be enforced server-side (UI is bypassable). These are the core security interview answers.

🎯 Key Takeaways

  • Spring Security is a filter chain answering authentication (who) then authorization (allowed) on every request — configured via a SecurityFilterChain bean.
  • 401 = not authenticated; 403 = authenticated but not permitted.
  • Hash passwords with BCrypt/Argon2 (never plaintext); for stateless REST use a signed JWT in the Authorization header (short-lived + refresh; no secrets in claims).
  • Add method-level @PreAuthorize as defence in depth (proxy-based — watch self-invocation).
  • Disable CSRF only for stateless header-token auth; keep it on for cookie/session auth.
  • Authorization is always enforced server-side — UI checks are UX only; stateless vs session is a deliberate scalability-vs-revocation trade-off.
