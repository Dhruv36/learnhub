Spring Security Advanced

Spring Security is the framework's steepest learning curve and its most important feature. It's a filter chain sitting in front of your app that answers two questions on every request: who are you (authentication) and are you allowed (authorization). For a modern REST API the answer is usually a stateless JWT. This lesson demystifies the chain and builds a token-based setup.

The Mental Model — A Filter Chain

// A request passes through a chain of filters BEFORE reaching your controller:
//   [ CORS ] → [ CSRF ] → [ Authentication filter ] → [ Authorization check ] → your @RestController
// Authentication filter: extracts credentials (a JWT, a session cookie) and builds an Authentication.
// Authorization: checks that Authentication has the required authority/role for the matched URL/method.
// If either fails, the chain short-circuits with 401 (not authenticated) or 403 (not allowed).
🔑 401 vs 403. 401 Unauthorized = we don't know who you are (missing/bad credentials). 403 Forbidden = we know who you are, but you lack permission. Mixing these up is a classic bug and a common interview question.

The SecurityFilterChain Bean (Modern, Lambda DSL)

@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    SecurityFilterChain chain(HttpSecurity http, JwtAuthFilter jwtFilter) throws Exception {
        http
          .csrf(csrf -> csrf.disable())                         // OK for a stateless token API (see note)
          .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
          .authorizeHttpRequests(auth -> auth
              .requestMatchers("/api/auth/**", "/actuator/health").permitAll()
              .requestMatchers(HttpMethod.GET, "/api/products/**").permitAll()
              .requestMatchers("/api/admin/**").hasRole("ADMIN")
              .anyRequest().authenticated())
          .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }

    @Bean PasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(); }  // NEVER plaintext
}
🛡️ CSRF: disable it only for a stateless, token-in-header API (no cookies used for auth) — there's no ambient credential a browser would auto-attach. If you authenticate with session cookies, keep CSRF protection on. Blanket-disabling CSRF on a cookie-based app is a real vulnerability.

JWT Authentication Flow

// 1. POST /api/auth/login {email, password} → verify with passwordEncoder.matches → issue a signed JWT.
// 2. Client sends it on every request:  Authorization: Bearer <token>
// 3. A once-per-request filter validates the signature + expiry and sets the SecurityContext:
public class JwtAuthFilter extends OncePerRequestFilter {
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain) {
        String token = extractBearer(req);                          // from Authorization header
        if (token != null && jwt.isValid(token)) {
            var auth = jwt.toAuthentication(token);                 // subject + roles from claims
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        chain.doFilter(req, res);
    }
}
// The token is SIGNED (HMAC/RSA) so the server trusts its claims without a session store — stateless,
// horizontally scalable. Keep tokens short-lived; use a refresh token for renewal.

Method-Level Authorization

@EnableMethodSecurity        // turn on annotation-based checks
// ...
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(long id) { ... }

@PreAuthorize("#userId == authentication.name or hasRole('ADMIN')")   // owner OR admin
public UserDto get(long userId) { ... }
// Defence in depth: URL rules in the chain + method rules on sensitive operations.

Accessing the Current User

@GetMapping("/me")
public UserDto me(@AuthenticationPrincipal UserDetails principal) {   // injected from SecurityContext
    return service.byUsername(principal.getUsername());
}

Interview Questions

Q1: Authentication vs authorization, and 401 vs 403?

Authentication establishes who the caller is (verifying credentials → an Authentication in the SecurityContext). Authorization decides what that identity may do (roles/authorities vs the required permission for a URL or method). In HTTP, 401 Unauthorized means authentication failed or is missing — we don't know who you are; 403 Forbidden means you're authenticated but lack permission for this resource. Spring's filter chain returns 401 when the authentication filter can't establish identity and 403 when an authenticated principal fails an authorization rule.

Q2: Why are JWTs suited to a stateless REST API, and what are the caveats?

A JWT is a signed token carrying claims (subject, roles, expiry). Because the signature lets any server verify it without a shared session store, auth becomes stateless — you can scale horizontally and add nodes without sticky sessions. Caveats: the payload is signed, not encrypted, so it's readable — never put secrets in it; revocation is hard because a valid signature is trusted until expiry, so keep access tokens short-lived and use refresh tokens (or a denylist) for logout/rotation; and you must validate signature and expiry on every request. Cookie-based sessions, by contrast, are trivially revocable but stateful.

Q3: When is disabling CSRF protection acceptable?

CSRF attacks rely on the browser automatically attaching an ambient credential (a session cookie) to a forged cross-site request. If your API is stateless and authenticates via a token sent explicitly in the Authorization header — which a cross-site page cannot set on your behalf — there's no ambient credential to abuse, so disabling CSRF is safe and conventional. If you authenticate with session cookies, you must keep CSRF protection enabled (synchronizer token or SameSite cookies). The rule: disable CSRF only for header-token, cookie-less auth.

🏋️ Practice: add spring-boot-starter-security; write a SecurityFilterChain that permits /api/auth/**, requires auth elsewhere, and restricts /api/admin/** to ROLE_ADMIN; implement register/login hashing with BCryptPasswordEncoder and issuing a JWT; add a OncePerRequestFilter that validates the token; protect a method with @PreAuthorize; verify 401 (no token) and 403 (wrong role) responses.

🎯 Key Takeaways

  • Security is a filter chain answering authentication (who) then authorization (allowed) on every request.
  • Configure it with a SecurityFilterChain bean and the lambda DSL; hash passwords with BCrypt.
  • For stateless REST, use signed JWTs (Bearer header) + a once-per-request validation filter; disable CSRF only then.
  • 401 = not authenticated; 403 = authenticated but not permitted.
  • Defence in depth: URL rules + @PreAuthorize; enforce server-side; short-lived tokens; no secrets in JWT claims.
