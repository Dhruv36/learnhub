IoC, DI & Beans Core Container

What you'll master in this lesson: the single idea at the heart of Spring — the Inversion of Control (IoC) container — and why every other feature plugs into it; the difference between Inversion of Control and Dependency Injection; the three injection styles and why constructor injection wins decisively; what a "bean" is and how the container creates and wires them; and how to resolve ambiguity when several beans could satisfy a dependency. Master this and the rest of Spring stops being magic — REST controllers, data repositories, security, transactions are all just beans plugged into this container. (Assumes the Java Interfaces and Inheritance lessons — programming to interfaces and composition.)

Part 1 — The Problem IoC Solves

// WITHOUT dependency injection — the class creates its own collaborators, hard-wired:
public class OrderService {
    private final PaymentGateway gateway = new StripeGateway();     // hard-wired to Stripe!
    private final OrderRepository repo    = new JdbcOrderRepository();
}
// Problems: you can't swap StripeGateway for a test double, can't reuse OrderService with a
// different gateway, and every class that needs a gateway constructs its own. OrderService is
// COUPLED to concrete implementations it shouldn't care about — the opposite of "program to
// interfaces" and "favor composition" from the Java OOP lessons.

Inversion of Control flips the responsibility: your code no longer creates its dependencies — a container does, and hands ("injects") them in. Your class just declares what it needs. The control over object creation and wiring is inverted from your code to the framework.

Part 2 — IoC vs DI, and Dependency Injection in Practice

@Service                                      // marks this class as a Spring-managed BEAN
public class OrderService {
    private final PaymentGateway gateway;     // dependencies as final fields
    private final OrderRepository repo;

    public OrderService(PaymentGateway g, OrderRepository r) {   // Spring PROVIDES these
        this.gateway = g;
        this.repo = r;
    }
}
// The container sees the constructor, finds a PaymentGateway bean and an OrderRepository bean,
// and passes them in. You never write `new OrderService(...)` — Spring does, wiring the graph.
💡 IoC is the principle; DI is the technique. Inversion of Control is the general idea that the framework controls the flow (creating objects, calling your code). Dependency Injection is the specific way Spring implements it for wiring: the container instantiates beans and supplies each bean's dependencies (via its constructor, setters, or fields). So "Spring's IoC container" and "Spring does dependency injection" describe the same machinery at two levels of abstraction — a distinction interviewers like to probe.

Part 3 — The Three Injection Styles (and why constructor wins)

// 1. CONSTRUCTOR injection — PREFERRED:
@Service
public class OrderService {
    private final PaymentGateway gateway;                 // final → immutable, thread-safe
    public OrderService(PaymentGateway gateway) { this.gateway = gateway; }
    // (Since Spring 4.3, a single constructor needs no @Autowired.)
}

// 2. FIELD injection — DISCOURAGED:
@Service
public class OrderService {
    @Autowired private PaymentGateway gateway;            // hidden dependency, hard to test
}

// 3. SETTER injection — for genuinely OPTIONAL dependencies only:
@Autowired public void setGateway(PaymentGateway g) { this.gateway = g; }
🎯 Always prefer constructor injection. It gives you final fields (immutable, safely published across threads — the Java concurrency lessons); it makes required dependencies explicit and impossible to forget; it fails fast at startup if a dependency is missing rather than throwing an NPE later; and — crucially — it lets you instantiate the class in a plain unit test with new OrderService(mockGateway, mockRepo), no Spring context needed (the Java Testing lesson). Field injection hides the dependency list, requires reflection to test, and can mask circular dependencies. Setter injection is only for truly optional collaborators. Constructor injection is dependency inversion done right.

Part 4 — What Is a Bean?

// A "bean" = an object the container creates, wires, and manages the lifecycle of.
// You declare beans two ways (Configuration lesson goes deeper):

// 1. Stereotype annotations on YOUR classes (component scanning finds them):
@Service       class OrderService { ... }
@Repository    class JpaOrderRepo { ... }
@Component     class RetryPolicy { ... }
@RestController class OrderController { ... }

// 2. @Bean factory methods (for third-party classes you can't annotate):
@Configuration
class AppConfig {
    @Bean PaymentGateway paymentGateway(@Value("${stripe.key}") String key) {
        return new StripeGateway(key);
    }
}
StereotypeRole (all are @Component under the hood)
@ComponentGeneric Spring-managed bean
@ServiceBusiness-logic bean (semantic marker)
@RepositoryData-access bean (adds exception translation)
@RestController/@ControllerWeb-layer bean handling HTTP
🔑 Beans are singletons by default (one per container) and shared across every request thread, so keep them stateless — the Bean Lifecycle lesson covers scopes and why mutable state on a shared singleton is a race condition. The stereotypes are semantically identical to @Component but document intent and, in the case of @Repository, add behavior (translating persistence exceptions into Spring's DataAccessException hierarchy).

Part 5 — Resolving Ambiguity

public interface PaymentGateway {}
@Component class StripeGateway implements PaymentGateway {}
@Component class PaypalGateway implements PaymentGateway {}
// Injecting a PaymentGateway now FAILS: "expected single matching bean but found 2".

// Fix options:
@Component @Primary class StripeGateway implements PaymentGateway {}   // the default winner
// ...or name it explicitly at the injection point:
public OrderService(@Qualifier("paypalGateway") PaymentGateway g) { ... }
// ...or inject ALL of them:
public Router(List<PaymentGateway> gateways) { ... }   // gets both, e.g. to dispatch by type
🔍 When one interface has multiple implementations, Spring can't guess which to inject. Resolve it with @Primary (mark a default), @Qualifier("beanName") (name the one you want at the injection point), or by injecting a List/Map of all of them (useful for strategy dispatch — the Java Design Patterns lesson's Strategy at framework scale). The "expected single bean but found N" error is one of the most common Spring startup failures, and knowing these three fixes resolves it immediately.

Part 6 — Deep Dive: Why the Container Changes How You Design

  • You program to interfaces, and the container binds the implementation. Beans declare interface-typed dependencies; the container injects whichever implementation is registered. So the concrete choice becomes a late-bound, configuration-driven decision — the Dependency Inversion Principle (Java SOLID lesson) operationalized by a framework.
  • Testability comes for free. Because dependencies are injected (not new-ed), you swap real collaborators for mocks in tests by constructing the object with test doubles — no Spring needed for unit tests. This is why constructor injection and DI make code testable by design.
  • The container manages the object graph and lifecycle. You describe what depends on what; Spring instantiates everything in dependency order, wires it, manages singletons, and tears it down on shutdown. You stop writing wiring code entirely.
  • Everything else in Spring is beans in this container. A @RestController, a @Repository, a security filter, a transaction manager — all are beans the container creates and wires. Understanding IoC/DI is the foundation for reasoning about every other feature.
🔍 Senior framing: the IoC container is the one concept that unlocks all of Spring. Its value isn't just avoiding new — it's that it inverts the dependency direction so your business classes depend on abstractions and never on concrete wiring, which delivers loose coupling (swap implementations via configuration), testability (inject mocks), centralized lifecycle and configuration, and a uniform model where every part of the application is a managed bean. The single most important practical decision is to use constructor injection — final fields, explicit required dependencies, fail-fast startup, and framework-free unit tests — because it makes the benefits of DI real rather than theoretical (field injection quietly undermines all of them). Interviewers ask about IoC vs DI, injection styles, and bean ambiguity precisely because the answers reveal whether you understand Spring as a dependency-management container — the thing every other feature builds on — or just as a collection of annotations you sprinkle on classes. Get this lesson deeply and the rest of the track is applying the same container to the web, data, security, and production concerns.

Interview Questions

Q1 (beginner): What is Inversion of Control, and how does Dependency Injection relate to it?

Inversion of Control means the framework — not your code — controls object creation and the flow of wiring dependencies. Traditionally a class calls new on its collaborators; with IoC that control is inverted to a container. Dependency Injection is the concrete technique Spring uses to implement IoC for wiring: the container instantiates beans and supplies (injects) each bean's dependencies via its constructor (preferred), setters, or fields. So IoC is the general principle and DI is the specific mechanism — "Spring's IoC container" and "Spring does dependency injection" describe the same machinery. Benefits: loose coupling (depend on interfaces, swap implementations via config), testability (inject mocks), centralized lifecycle and configuration.

Q2 (beginner): What is a Spring bean, and how do you define one?

A bean is any object that the Spring IoC container instantiates, wires with its dependencies, and manages the lifecycle of. You define beans two ways: (1) put a stereotype annotation (@Component, or the semantic variants @Service, @Repository, @RestController) on your own class, so component scanning discovers and instantiates it; or (2) write a @Bean factory method inside a @Configuration class that constructs and returns the object, used for third-party classes you can't annotate or when construction needs logic. Beans are singletons by default (one instance per container), so they should be kept stateless since they're shared across request threads.

Q3 (mid): Why is constructor injection preferred over field injection?

Constructor injection yields final fields (immutable, safely published across threads); makes required dependencies explicit and impossible to instantiate without; fails fast at startup if a dependency is missing rather than throwing a NullPointerException later; and — crucially — lets you construct the object in a plain unit test (new Service(mock1, mock2)) with no Spring context or reflection, so it's testable by design. Field injection (@Autowired on a field) hides the dependency list, permits partially-built objects, forces reflection-based test setup, and makes circular dependencies easy to hide (they surface at startup with constructor injection). Setter injection is appropriate only for genuinely optional dependencies. So constructor injection is the way to get DI's benefits — loose coupling and testability — for real.

Q4 (mid): You have two implementations of an interface. How do you tell Spring which to inject?

By default, injecting the interface type when two beans implement it fails with a NoUniqueBeanDefinitionException ("expected single matching bean but found 2"). Resolve it three ways: (1) annotate one implementation with @Primary to make it the default when the type is injected without further qualification; (2) use @Qualifier("beanName") at the injection point to name exactly which bean you want; or (3) inject all of them as a List<PaymentGateway> or Map<String, PaymentGateway> (keyed by bean name), which is useful when you want to dispatch to different implementations at runtime — a Strategy-pattern setup. Choose @Primary for a sensible default, @Qualifier for explicit per-injection control, and the collection form when you genuinely need all implementations.

Q5 (senior): How does the IoC container relate to the Dependency Inversion Principle, and why does that matter for architecture?

The Dependency Inversion Principle (from SOLID) says high-level modules should depend on abstractions, not concrete low-level implementations, and that both should depend on abstractions. Spring's IoC container is that principle operationalized at the wiring level: your business classes declare dependencies by interface type and never construct or name concrete implementations, and the container supplies whichever implementation is registered — so the concrete choice becomes a late-bound, configuration-driven decision made outside the business code. This inverts the compile-time dependency direction: instead of OrderService depending on StripeGateway, both depend on the PaymentGateway abstraction, with the container binding them. Architecturally this matters because it produces loose coupling (you can swap Stripe for PayPal, or a real datasource for an in-memory one, by changing configuration rather than editing business logic), testability (inject a mock PaymentGateway in a unit test with no framework), and a clean separation between what the application does (business beans) and how it's assembled (configuration). It also enables cross-cutting concerns to be added transparently — because the container controls instantiation, it can wrap beans in proxies for transactions, security, and caching (the AOP lesson) — and it makes the whole system a uniform graph of interchangeable, independently-testable components. The container is therefore not just a convenience for avoiding new; it's the mechanism that lets you build systems whose modules are decoupled through abstractions, which is the entire point of dependency inversion and the reason Spring scales to large, maintainable, testable codebases.

Q6 (senior): What are the trade-offs of field injection beyond testability, and why can it hide circular dependencies?

Field injection carries several problems beyond being hard to test. First, it produces mutable, non-final fields, so the object can technically exist in a partially-constructed state and you lose the thread-safety and clarity guarantees of final fields. Second, it hides the dependency contract: the dependencies aren't visible in the constructor signature, so a class can accumulate many hidden collaborators (a code smell for a bloated class) without it being obvious, whereas a constructor with eight parameters loudly signals the class does too much. Third, it couples your code to the DI framework's reflection — the class can't be instantiated normally without a container, so unit tests need reflection utilities or a full context. The circular-dependency point is subtle: with constructor injection, if bean A requires B and B requires A, the container cannot construct either (each needs the other to exist first), so it fails fast at startup with a clear circular-dependency error — surfacing a design problem immediately. With field injection, Spring can construct both beans first (calling their no-arg constructors) and then set the fields afterward, so the cycle is silently satisfied and the application starts — the circular dependency, which is usually a sign of poor separation of concerns, is masked rather than exposed, and it can lead to subtle initialization-order bugs (a field being used before it's injected) or simply an unhealthy design that persists unnoticed. So constructor injection's "inability" to handle cycles is actually a feature: it forces you to break the cycle by redesigning (extracting a third collaborator, using an event, or reconsidering responsibilities). The senior takeaway is that field injection's convenience trades away compile-time-visible contracts, immutability, framework-free testability, and the fail-fast detection of a genuine design smell — which is why constructor injection is the strongly recommended default and field injection is discouraged in production code.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Constructor-injected service. Create a @Service that depends on an interface, injected via the constructor (no @Autowired needed), and inject that service into a @RestController.
Solution

A @Service with a single constructor taking the interface-typed dependency and assigning it to a final field; Spring injects it automatically (single constructor since 4.3). The controller likewise takes the service in its constructor. No new anywhere — the container wires the graph. This is the idiomatic, testable setup.

2 (easy): Prove the singleton. Inject the same @Service into two components and compare identity (==) to confirm it's one shared instance.
Solution

The two injected references are identical (== true) — the default singleton scope means one instance per container, shared everywhere. This is why singletons must be stateless: every request thread uses the same object. (Bean Lifecycle lesson covers changing scope.)

3 (medium): Resolve ambiguity. Add two implementations of a Notifier interface (email, SMS). Make the injection fail, then fix it with @Primary, then instead with @Qualifier.
Solution

With both @Components and a bare Notifier injection, startup fails with NoUniqueBeanDefinitionException. Fix 1: annotate one with @Primary so it's the default. Fix 2: remove @Primary and use @Qualifier("smsNotifier") at the injection point to pick the SMS one explicitly. Both resolve the ambiguity; @Primary sets a default, @Qualifier chooses per injection.

4 (medium): Inject all implementations. Inject a List<Notifier> and send through every notifier. Explain when this is useful.
Solution

Injecting List<Notifier> gives you every Notifier bean, so you can loop and dispatch to all (fan-out) or select one by type. Useful for Strategy-style dispatch, plugins, or fan-out notifications — the container hands you all implementations without you naming them. A Map<String, Notifier> gives them keyed by bean name for lookup.

5 (hard): Field-injection cycle. Create beans A and B that depend on each other. Wire them with constructor injection and observe the startup failure; then switch to field injection and observe it "works." Explain what that reveals.
Solution

With constructor injection, A needs B and B needs A, so neither can be constructed first — Spring fails fast with a circular-dependency error, surfacing the design problem. With field injection, Spring constructs both beans then sets the fields, so the app starts — the cycle is silently masked. This reveals that constructor injection's fail-fast behavior is a feature (it forces you to break the cycle by redesigning), while field injection hides a genuine design smell. The right fix is to remove the cycle, not to switch to field injection.

6 (hard): Explain-it test. Out loud: (a) IoC vs DI; (b) why constructor injection wins; (c) how the container relates to the Dependency Inversion Principle and what that buys architecturally.
Self-check rubric

Pass (a) if you said IoC is the principle (framework controls creation/flow) and DI is the wiring technique. Pass (b) if you named final fields, explicit/fail-fast deps, and framework-free testability. Pass (c) if you connected the container to depending on abstractions (DIP) → loose coupling, testability, and swappable implementations via config. This is the foundation for every later lesson.

🎯 Key Takeaways

  • Spring's core is the IoC container: it creates, wires, and manages the lifecycle of beans so your code declares what it needs instead of new-ing dependencies.
  • IoC is the principle; DI is the technique — the container injects each bean's dependencies via constructor (preferred), setter, or field.
  • Prefer constructor injection: final fields, explicit/fail-fast dependencies, framework-free unit testing; field injection hides deps and masks circular dependencies.
  • A bean is a container-managed object, defined by stereotype annotations or @Bean methods; singletons by default, so keep them stateless.
  • Resolve multiple candidates with @Primary, @Qualifier, or by injecting a List/Map of all of them.
  • The container operationalizes dependency inversion — business beans depend on abstractions, the container binds implementations — giving loose coupling and testability, and it's the foundation every other Spring feature plugs into.
