IoC, DI & Beans Basics

Spring Boot is the dominant Java backend framework — what most "Java developer" jobs actually mean. Strip away the starters and auto-config and the beating heart is a single idea: the Inversion of Control (IoC) container. Instead of your code creating and wiring its collaborators, a container does it for you. Everything else — REST, data, security — is beans plugged into that container. Master this and the rest of the framework stops being magic. (Assumes the Java OOP lesson — interfaces and composition.)

The Problem IoC Solves

// WITHOUT DI — the class creates its own dependencies:
public class OrderService {
    private final PaymentGateway gateway = new StripeGateway();   // hard-wired!
    private final OrderRepository repo = new JdbcOrderRepository();
}
// Problems: can't swap StripeGateway for a test double, can't reuse OrderService with
// a different gateway, and every class that needs a gateway constructs its own.
// The class is COUPLED to concrete implementations it shouldn't care about.

Inversion of Control flips the responsibility: you don't new your dependencies — the container creates them and hands ("injects") them in. Your class just declares what it needs.

Dependency Injection — Constructor Style

@Service                                      // marks this class as a Spring-managed BEAN
public class OrderService {
    private final PaymentGateway gateway;     // dependencies as final fields
    private final OrderRepository repo;

    public OrderService(PaymentGateway g, OrderRepository r) {  // Spring PROVIDES these
        this.gateway = g;
        this.repo = r;
    }
}
// The container sees the constructor, finds a PaymentGateway bean and an OrderRepository
// bean, and passes them in. You never write `new OrderService(...)` — Spring does.
☕ Always prefer constructor injection. It gives you final fields (immutable, thread-safe), makes required dependencies explicit and impossible to forget, fails fast at startup if a bean is missing, and lets you instantiate the class in a plain unit test with new OrderService(mockGateway, mockRepo) — no Spring needed. Field injection (@Autowired on a field) hides dependencies and can't be tested without reflection.

What Is a Bean?

// A "bean" = an object the container creates, wires, and manages the lifecycle of.
// You declare beans two ways:

// 1. Stereotype annotations on your own classes (component scanning finds them):
@Service       class OrderService { ... }
@Repository    class JpaOrderRepo { ... }
@Component     class RetryPolicy { ... }

// 2. @Bean factory methods in a @Configuration class (for third-party classes you can't annotate):
@Configuration
class AppConfig {
    @Bean
    PaymentGateway paymentGateway(@Value("${stripe.key}") String key) {
        return new StripeGateway(key);          // YOU build it; Spring manages it
    }
}
StereotypeMeaning (all are @Component under the hood)
@ComponentGeneric Spring-managed bean
@ServiceBusiness-logic bean (semantic marker)
@RepositoryData-access bean (adds exception translation)
@RestController/@ControllerWeb-layer bean handling HTTP
@ConfigurationHolds @Bean factory methods

Bean Scope & Lifecycle

@Service                       // default scope: SINGLETON — ONE instance for the whole app
public class Counter { ... }

@Component
@Scope("prototype")            // a NEW instance every time it's injected/requested
public class ShoppingCart { ... }

@Service
public class CacheWarmer {
    @PostConstruct void init()   { /* runs after wiring, before use */ }
    @PreDestroy   void cleanup() { /* runs on shutdown */ }
}
🔑 Singletons are shared across every thread — so keep beans stateless. Storing per-request data in a field on a singleton service is a classic concurrency bug: two requests stomp on each other. Put request state in method parameters/local variables, not bean fields.

Resolving Ambiguity

public interface PaymentGateway {}
@Component class StripeGateway  implements PaymentGateway {}
@Component class PaypalGateway  implements PaymentGateway {}
// Injecting a PaymentGateway now FAILS: "expected single bean but found 2".

// Fix options:
@Component @Primary class StripeGateway implements PaymentGateway {}   // the default winner
// ...or name it explicitly at the injection point:
public OrderService(@Qualifier("paypalGateway") PaymentGateway g) { ... }
// ...or inject ALL of them:
public Router(List<PaymentGateway> gateways) { ... }   // gets both

Interview Questions

Q1: What is Inversion of Control, and how does DI implement it?

IoC means the framework — not your code — controls object creation and the flow of wiring dependencies. Traditionally a class calls new on its collaborators; with IoC that control is inverted to a container. Dependency Injection is the concrete technique: the container instantiates beans and supplies (injects) each bean's dependencies via its constructor (preferred), setters, or fields. Benefits: loose coupling (depend on interfaces, swap implementations via config), testability (inject mocks), centralized lifecycle and configuration. It's "program to interfaces + favor composition" enforced by a container.

Q2: Why is constructor injection preferred over field injection?

Constructor injection yields final fields (immutable, safely published across threads); makes required dependencies explicit and impossible to instantiate without; fails fast at startup if a dependency is missing rather than throwing an NPE later; and — crucially — lets you construct the object in a plain unit test (new Service(mock1, mock2)) with no Spring context or reflection. Field injection hides the dependency list, permits partially-built objects, and forces reflection-based test setup. It also makes circular dependencies easy to hide, whereas constructor injection surfaces them at startup.

Q3: What's the default bean scope and why does it matter for correctness?

The default scope is singleton: one shared instance per application context, injected everywhere it's needed. Because that single instance is used concurrently by every request thread, any mutable instance field is shared state and a race-condition risk. The rule is to keep singleton beans stateless — hold only immutable collaborators (other beans, config) and put per-request data in method parameters and local variables. Use prototype (new instance per injection) or request/session scopes only when you genuinely need per-use state, and understand that a prototype injected into a singleton is created only once unless you use a provider/lookup.

🏋️ Practice: create a @Service with a constructor-injected interface dependency; add two implementations and resolve the ambiguity with @Primary, then with @Qualifier; register a third-party class as a @Bean in a @Configuration; add a @PostConstruct that logs on startup; prove the service is a singleton by injecting it twice and comparing identity.

🎯 Key Takeaways

  • Spring's core is the IoC container: it creates, wires, and manages the lifecycle of beans.
  • Prefer constructor injection — final fields, explicit deps, fail-fast, trivially testable.
  • Beans come from stereotype annotations (@Service…) or @Bean factory methods.
  • Default scope is singleton and shared across threads — keep beans stateless.
  • Resolve multiple candidates with @Primary, @Qualifier, or inject a List.
