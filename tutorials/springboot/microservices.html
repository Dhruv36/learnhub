Resilience & Microservices Production

What you'll master in this lesson: what changes when a Spring Boot service talks to other services over the network — the fallacies that make distributed calls dangerous, the resilience patterns (timeouts, retries, circuit breakers, bulkheads) that keep one slow dependency from taking down your service, how Spring implements them with Resilience4j, the calling tools (RestClient/WebClient/declarative HTTP interfaces), and the surrounding concerns — service discovery, config, gateways, and the idempotency and saga patterns that replace transactions across service boundaries. Prerequisites: REST Controllers, Transactions (why you can't just @Transactional across services), and Observability (tracing is how you debug distributed calls). Ties into the AWS and Kubernetes tracks.

Part 1 — Why Distributed Calls Are Different (The Fallacies)

Fallacy (people wrongly assume…)Reality — what actually bites you
The network is reliableCalls fail, packets drop, connections reset — every remote call can fail
Latency is zeroRemote calls are 1000× slower than local; a chain of them adds up fast
Bandwidth is infiniteLarge payloads and chatty calls saturate links and pools
The topology doesn't changeInstances come and go (autoscaling, deploys); hard-coded hosts break
A response always comes backA call can hang forever — the killer default: no timeout
🎯 An in-process method call always returns; a network call might hang, fail, or lie — and your code must assume it will. The "fallacies of distributed computing" name the wrong assumptions that cause outages when you split a monolith into services. The single most dangerous one is assuming a response always comes: an HTTP client with no timeout will wait forever on a hung dependency, and because it holds a thread while waiting, a few hundred hung calls exhaust your thread/connection pool and your service — which is perfectly healthy itself — stops responding. Every remote call needs a timeout, and beyond that, deliberate resilience patterns.

Part 2 — The Resilience Patterns

PatternProtects againstWhat it does
TimeoutHung dependencyGive up after N ms — never wait forever (the non-negotiable baseline)
RetryTransient failuresTry again (with backoff) — only for idempotent ops
Circuit breakerA dependency that's downAfter many failures, stop calling ("open") and fail fast; probe to recover
BulkheadOne dependency starving othersIsolate resources (separate pools) so one slow call can't consume them all
FallbackAny failureReturn a default/cached/degraded response instead of an error
Rate limiterOverwhelming a dependency (or being overwhelmed)Cap the call rate
🔍 The circuit breaker is the pattern to understand deeply. Like an electrical breaker, it has three states: CLOSED (calls flow normally, failures counted); OPEN (after the failure rate crosses a threshold, calls are rejected immediately without even trying — "fail fast" — so you stop hammering a dead dependency and stop tying up threads waiting on it); and HALF-OPEN (after a cool-down, a few trial calls are allowed — if they succeed the breaker closes and normal traffic resumes, if they fail it re-opens). It converts slow, thread-consuming failures into instant ones and gives the struggling dependency room to recover instead of being pounded while it's down. Timeouts and circuit breakers together are what stop a cascading failure — one service's outage rippling up through everyone who calls it.

Part 3 — Resilience4j in Spring Boot

// Add resilience4j-spring-boot3. Configure per-instance in application.yml:
resilience4j:
  circuitbreaker:
    instances:
      payments:
        failure-rate-threshold: 50        # open when 50% of calls in the window fail
        wait-duration-in-open-state: 10s  # stay open 10s, then try HALF-OPEN
        sliding-window-size: 20
  timelimiter:
    instances:
      payments: { timeout-duration: 2s }  # cap each call at 2s
  retry:
    instances:
      payments: { max-attempts: 3, wait-duration: 200ms }

@Service
class PaymentClient {
    @CircuitBreaker(name = "payments", fallbackMethod = "fallback")   // annotations = AOP proxies (AOP lesson)
    @Retry(name = "payments")
    @TimeLimiter(name = "payments")
    public CompletableFuture<Receipt> charge(Order o) { return remoteCharge(o); }

    private CompletableFuture<Receipt> fallback(Order o, Throwable t) {   // same signature + Throwable
        return CompletableFuture.completedFuture(Receipt.pending(o));      // degrade gracefully, don't 500
    }
}
🔧 Resilience4j provides the patterns as composable, annotation-driven decorators. Each is configured per named instance in application.yml and applied with an annotation (@CircuitBreaker, @Retry, @TimeLimiter, @Bulkhead, @RateLimiter) — which, like everything else in Spring, works via an AOP proxy, so the same self-invocation caveat applies. A fallbackMethod (matching signature plus a Throwable) gives you a graceful degraded response instead of an error. Order matters when composing: typically Retry wraps CircuitBreaker wraps TimeLimiter — you retry, each attempt is bounded by a timeout and counted by the breaker, and if the breaker is open you fail fast without retrying a dead service.

Part 4 — Calling Other Services

// RestClient — modern synchronous client (Boot 3.2+), replaces RestTemplate:
Receipt r = restClient.post().uri("/charge").body(order)
                .retrieve().body(Receipt.class);

// WebClient — reactive/non-blocking, for high-concurrency or streaming:
Mono<Receipt> r = webClient.post().uri("/charge").bodyValue(order)
                .retrieve().bodyToMono(Receipt.class);

// Declarative HTTP Interface — define an interface, Spring generates the client (like Feign):
interface PaymentApi {
    @PostExchange("/charge")
    Receipt charge(@RequestBody Order o);
}
🎯 Prefer RestClient (sync) or WebClient (reactive) over the legacy RestTemplate. For most services a synchronous RestClient is simplest; reach for reactive WebClient when you need high concurrency with few threads or streaming. The declarative HTTP interface (Spring's built-in equivalent of OpenFeign) lets you define a Java interface annotated with the endpoints and have Spring generate the implementation — clean, testable, and easy to wrap with Resilience4j. Whatever you choose, configure connect and read timeouts on the underlying client — this is the baseline that every other pattern builds on.

// Cross-service consistency: you CANNOT use one @Transactional across services (no shared DB/transaction).
// Two essential patterns replace it:
//   IDEMPOTENCY   — a retried "create payment" must not charge twice. Use an idempotency key the
//                   receiver dedupes on, so a retry (Part 2) is safe.
//   SAGA          — a multi-service workflow (order → payment → shipping) as a sequence of local
//                   transactions, each with a COMPENSATING action to undo prior steps on failure
//                   (refund the payment if shipping fails) — eventual consistency, not ACID.
⚠️ You lose the database transaction the moment a workflow spans services. There's no @Transactional across service boundaries — each service has its own database. Two patterns fill the gap. Idempotency makes retries safe: because the network makes you retry (Part 2), a "charge payment" call might arrive twice, so the receiver must dedupe on an idempotency key and apply the effect once. Sagas replace a distributed transaction with a chain of local transactions plus compensating actions that undo completed steps if a later one fails (refund if shipment can't be booked) — giving eventual consistency rather than atomic all-or-nothing. Recognizing that distributed workflows need idempotency and compensation instead of transactions is a hallmark of someone who has actually built microservices.

Part 6 — Deep Dive: Should You Even Use Microservices?

  • Microservices trade in-process simplicity for operational complexity. You gain independent deployability and scaling; you pay with network failure, distributed data, eventual consistency, and a much harder debugging story. That trade is worth it at organizational scale, and a net loss for a small team or a young product.
  • Start with a modular monolith. A well-structured monolith (clear module boundaries, one deploy, one database, real transactions) is faster to build and operate; extract a service only when a specific part has a real independent scaling or team-ownership need.
  • Resilience is not optional once you're distributed. Every remote call needs at minimum a timeout; hot paths need circuit breakers and fallbacks. Skipping this is how one slow dependency cascades into a full outage.
  • Observability is the price of admission. Distributed tracing (Actuator lesson) is what makes a request's path across services debuggable — without it, a microservice system is nearly impossible to diagnose.
🔍 Senior framing: the most valuable thing to say about microservices in an interview is often when not to use them — they are an organizational solution (independent teams deploying and scaling independently) bought at a steep technical price: the network is unreliable and slow, you lose the database transaction across boundaries, data becomes eventually consistent, and debugging spans many services — so the honest default for a small team or an unproven product is a well-structured modular monolith, extracting a service only where a concrete scaling or ownership need justifies the cost. Once you are distributed, the non-negotiable engineering is resilience: every remote call must have a timeout (a hung, thread-holding call is how a healthy service dies from a sick dependency), and hot paths need a circuit breaker (CLOSED → OPEN on sustained failure to fail fast and stop hammering a dead dependency → HALF-OPEN to probe recovery), retries with backoff for idempotent operations only, bulkheads to stop one dependency from starving all your threads, and fallbacks to degrade gracefully rather than error — Resilience4j gives all of these as composable annotations, ordered so retry wraps the breaker wraps the timeout. Because retries and network duplicates are unavoidable, mutating operations must be idempotent (dedupe on an idempotency key), and multi-service workflows that once would have been one transaction become sagas — sequences of local transactions with compensating actions for rollback, yielding eventual consistency. Around this sit discovery, externalized config, an API gateway, async messaging, and — critically — distributed tracing, without which a distributed system is undebuggable. The mature engineer therefore reaches for microservices deliberately and late, designs every cross-service call defensively, replaces transactions with idempotency and sagas, and treats observability as a precondition rather than an afterthought — which is exactly the judgment (cost awareness plus concrete resilience patterns) that senior distributed-systems interviews are testing.

Interview Questions

Q1 (beginner): Why does every remote call need a timeout?

Because unlike an in-process method call, a network call can hang indefinitely — the remote service may be overloaded, deadlocked, or unreachable, and without a timeout your client waits forever. Crucially, a thread (and often a connection) is held for the entire wait, so if a dependency hangs and you keep sending it requests, those threads pile up and exhaust your thread pool and connection pool — at which point your service, which is itself perfectly healthy, can no longer handle any requests and effectively goes down because of someone else's problem. A timeout bounds the wait so a hung dependency causes a fast, contained failure (which you can then retry or fall back from) instead of a slow, spreading one. It's the single most important and most commonly forgotten resilience measure.

Q2 (beginner): What is a circuit breaker and what are its states?

A circuit breaker protects a service from a failing dependency by monitoring call outcomes and, when failures cross a threshold, temporarily stopping calls so you fail fast instead of repeatedly waiting on something that's down. It has three states, like an electrical breaker: CLOSED — calls pass through normally and failures are counted; OPEN — once the failure rate exceeds the threshold, calls are rejected immediately without attempting the remote call (fast failure, and the struggling dependency gets a break from the load); and HALF-OPEN — after a wait duration, a limited number of trial calls are allowed through, and if they succeed the breaker closes (normal traffic resumes), if they fail it re-opens. This turns slow, thread-consuming timeouts into instant failures during an outage and lets the dependency recover, which is central to preventing cascading failures.

Q3 (mid): When is it safe to retry, and what's the danger otherwise?

Retrying is safe for idempotent operations — ones where doing them more than once has the same effect as doing them once (a GET, a PUT that sets a value, a delete). For those, a transient failure (a dropped packet, a brief blip) can be transparently recovered by trying again, ideally with backoff (increasing delays, often with jitter) so retries don't synchronize into a thundering herd that overwhelms a recovering dependency. The danger is retrying non-idempotent operations like "charge this card" or "place this order": the original request may actually have succeeded but its response was lost, so a retry performs the action again — a double charge or duplicate order. Because the network makes retries and duplicates unavoidable, the fix is to make such operations idempotent using an idempotency key: the client sends a unique key with the request and the receiver records processed keys, so a duplicate is recognized and the effect applied only once. So the rule is: retry idempotent operations with backoff; make mutating operations idempotent so they become safe to retry.

Q4 (mid): How does a hung dependency cause a cascading failure, and which patterns stop it?

A cascading failure is when one component's problem spreads to bring down others. The mechanism: service A calls a dependency B; B becomes slow or unresponsive; A's calls to B don't time out (or time out too late), so each waiting call holds a thread; under load, A's threads all end up blocked waiting on B, so A's pool is exhausted and A can no longer serve any requests — including ones that don't need B. Now everything calling A starts to hang the same way, and the failure ripples up the call graph until a whole system is down because of one slow dependency. The patterns that stop it: timeouts bound how long any call waits so threads are released quickly; circuit breakers detect the sustained failure and fail fast (rejecting calls instantly instead of tying up threads, and giving B room to recover); bulkheads isolate resources so calls to B use a separate limited pool that, even if fully consumed, leaves A's other work unaffected; and fallbacks return a degraded response so callers get something rather than hanging or erroring. Together these contain the blast radius of a single dependency's failure to just the feature that needs it, which is the whole point of resilience engineering.

Q5 (senior): You can't use a transaction across services — how do you keep a multi-service workflow consistent?

Because each service owns its own database, there's no shared transaction to make an operation like "place order → charge payment → reserve inventory → book shipment" atomic across them. Two complementary patterns provide consistency without distributed transactions. First, idempotency: since the network forces retries and can duplicate messages, every mutating cross-service call must be idempotent — the receiver dedupes on an idempotency key so a retried "charge" charges once. Second, the saga pattern: the workflow is modeled as a sequence of local transactions, one per service, coordinated either by choreography (each service emits an event that triggers the next) or orchestration (a central coordinator drives the steps). Crucially, each step has a compensating action that semantically undoes it, and if a later step fails, the saga runs the compensations for the already-completed steps in reverse (refund the payment, release the inventory) to return the system to a consistent state. This yields eventual consistency rather than ACID atomicity — there's a window where the system is partially updated, and compensation isn't a true rollback but a business-level reversal (a refund is not "un-charging"). The senior points are: recognize that "just use a transaction" doesn't work across services and reflexively reach for idempotency + sagas; understand the consistency you're actually getting (eventual, with compensations) and design the business to tolerate the intermediate states; and prefer avoiding cross-service transactions entirely (via better service boundaries that keep a transaction within one service) where possible, because sagas are complex and every compensating path is code you must build and test.

Q6 (senior): When should a team NOT adopt microservices?

Most of the time, for most teams — microservices are frequently the wrong default. They are fundamentally an organizational solution: they let independent teams own, deploy, and scale their services on separate cadences, which is valuable when you have many engineers whose coordination overhead on a single codebase has become the bottleneck. But you pay a heavy technical price for that organizational benefit: the network is unreliable and adds latency to every call; you lose database transactions across boundaries and must adopt eventual consistency, idempotency, and sagas; a single logical operation now spans multiple deploys, databases, and failure modes; and debugging requires distributed tracing and correlated logs just to follow one request. For a small team or a young, unproven product, that overhead dramatically slows delivery while providing benefits (independent scaling, team autonomy) you don't yet need — you spend your time on operational plumbing instead of the product. The far better default is a modular monolith: one deployable, one database (so real transactions), but with clean internal module boundaries and dependencies pointing the right way, so that if a specific part later develops a genuine independent scaling or ownership need, it can be extracted into a service along a boundary you've already defined. So the mature answer is to start monolithic, resist premature decomposition, and split out services surgically and late, driven by a concrete scaling or team-structure need rather than by architecture fashion — being able to articulate this cost/benefit judgment, rather than reflexively reaching for microservices, is itself a strong senior signal.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Add timeouts. Configure connect and read timeouts on a RestClient/WebClient and explain why waiting forever is dangerous.
Solution

Set connect and read timeouts on the underlying client factory. Without them, a hung dependency ties up a thread per call; under load the pool exhausts and your healthy service stops responding — a slow dependency becomes your outage. Timeouts bound the wait so failures are fast and contained.

2 (easy): Circuit breaker + fallback. Wrap a remote call with @CircuitBreaker and a fallbackMethod that returns a degraded response.
Solution

Annotate the call with @CircuitBreaker(name="x", fallbackMethod="fallback") and implement a fallback with the same signature plus a Throwable returning a default/cached value. When the dependency fails enough, the breaker opens and calls go straight to the fallback — fast failure, graceful degradation, no cascade.

3 (medium): Make retries safe. Explain why retrying "charge payment" is dangerous and design an idempotency-key fix.
Solution

A retry may re-run a charge whose original response was lost → double charge. Fix: the client sends a unique idempotency key; the payment service records processed keys and, on a duplicate key, returns the original result without charging again. Now the operation is idempotent and safe to retry.

4 (medium): Compose the patterns. Explain the correct nesting order of Retry, CircuitBreaker, and TimeLimiter and why.
Solution

Retry wraps CircuitBreaker wraps TimeLimiter: each attempt is bounded by a timeout and its outcome counted by the breaker; retries happen around that; and if the breaker is OPEN you fail fast without wasting retries on a known-dead dependency. Wrong ordering (e.g. retrying inside the timeout) undermines the guarantees.

5 (hard): Design a saga. For order → payment → shipping, sketch the local transactions and compensating actions, and state the consistency guarantee.
Solution

Each service does a local transaction (create order, charge payment, book shipment). If shipping fails, run compensations in reverse: refund the payment, cancel the order. Coordinated by choreography (events) or an orchestrator. Guarantee is eventual consistency — a partial-update window exists, and compensation is a business-level reversal (a refund), not a true DB rollback.

6 (hard): Explain-it test. Out loud: (a) the distributed fallacies and why timeouts are non-negotiable; (b) the circuit-breaker states and how it stops cascades; (c) when NOT to use microservices.
Self-check rubric

Pass (a) if you named "network unreliable / a response may never come" and explained hung threads exhaust the pool. Pass (b) if you gave CLOSED/OPEN/HALF-OPEN and fail-fast stopping the cascade. Pass (c) if you said microservices are organizational, costly technically, and a modular monolith is the right default for a small team. That completes resilience & microservices.

🎯 Key Takeaways

  • Distributed calls violate local assumptions (the fallacies) — the network fails, is slow, and a response may never come.
  • Every remote call needs a timeout; hot paths need a circuit breaker (CLOSED→OPEN→HALF-OPEN) + fallback to stop cascades.
  • Resilience4j supplies timeout / retry / circuit-breaker / bulkhead / fallback as composable annotations (Retry ⊃ CircuitBreaker ⊃ TimeLimiter).
  • Retry only idempotent ops; make mutating ops idempotent (idempotency keys) since retries are unavoidable.
  • No transaction across services — use sagas with compensating actions (eventual consistency).
  • Microservices are an organizational choice with a steep technical cost — default to a modular monolith.
