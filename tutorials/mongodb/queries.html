Query Operators & Projection Basics

MongoDB queries are documents themselves — you describe the shape you want to match. Mastering the operators (comparison, logical, array, element) and projection (choosing which fields return) covers the vast majority of day-to-day querying, and querying arrays correctly is a subtle interview favorite.

Comparison & Logical Operators

db.users.find({ score: { $gt: 80, $lte: 100 } })          // 80 < score <= 100
db.users.find({ role: { $in: ["admin", "editor"] } })     // membership
db.users.find({ status: { $ne: "banned" } })              // not equal
db.users.find({ $or: [{ plan: "pro" }, { score: { $gt: 90 } }] })
db.users.find({ $and: [{ active: true }, { score: { $gt: 50 } }] })   // AND is implicit for diff fields
db.users.find({ age: { $exists: true }, nickname: { $exists: false } })
db.users.find({ score: { $type: "number" } })
// $eq $ne $gt $gte $lt $lte $in $nin | $and $or $nor $not | $exists $type

Querying Arrays — the Subtle Part

// tags: ["admin", "editor", "vip"]
db.users.find({ tags: "admin" })                    // array CONTAINS "admin"
db.users.find({ tags: { $all: ["admin", "vip"] } }) // contains ALL of these
db.users.find({ tags: { $size: 3 } })               // exactly 3 elements

// Arrays of SUBDOCUMENTS — the $elemMatch trap:
// items: [{ product: "A", qty: 2 }, { product: "B", qty: 5 }]
db.orders.find({ "items.product": "A", "items.qty": { $gt: 3 } })   // ❌ matches! (A in one, qty>3 in ANOTHER)
db.orders.find({ items: { $elemMatch: { product: "A", qty: { $gt: 3 } } } })  // ✅ SAME element
// Without $elemMatch, conditions can match across DIFFERENT array elements — a classic bug.
🎯 $elemMatch is the answer whenever you need multiple conditions to hold for the same array element. Plain dot-notation conditions are evaluated independently across the whole array, so {"items.product":"A", "items.qty":{$gt:3}} is true if ANY element has product A and ANY (possibly different) element has qty > 3. Interviewers love this one.

Projection — Choosing Returned Fields

db.users.find({}, { name: 1, score: 1 })            // include name, score (+ _id by default)
db.users.find({}, { password: 0, _id: 0 })          // exclude (can't mix include/exclude, except _id)
db.orders.find({}, { items: { $slice: 5 } })        // first 5 array elements
db.orders.find({ "items.product": "A" },
               { "items.$": 1 })                     // positional: the FIRST matching array element
// Projection reduces payload (bandwidth/memory) and enables covered queries (indexes lesson) —
// like naming columns in SQL instead of SELECT *.

Pattern, Range & Existence Queries

db.users.find({ name: /^A/ })                        // regex (anchored — can use an index)
db.users.find({ name: { $regex: "asha", $options: "i" } })   // case-insensitive (no index)
db.events.find({ createdAt: { $gte: ISODate("2026-01-01"), $lt: ISODate("2026-02-01") } })
db.users.find({ "address.zip": { $exists: true } })
// For real text search, use a TEXT INDEX + $text (indexes lesson) or Atlas Search, not regex.

Cursor Methods

db.users.find(q).sort({ score: -1, _id: 1 }).skip(20).limit(10)   // paginate (add _id tiebreaker!)
db.users.find(q).count()          // countDocuments(q) is the accurate modern form
db.users.distinct("plan")          // unique values of a field
// Deep skip() is slow (like SQL OFFSET) — use range/keyset pagination on _id for feeds.

Interview Questions

Q1: Explain the $elemMatch trap with array-of-subdocuments.

When a document has an array of subdocuments, plain dot-notation conditions are evaluated INDEPENDENTLY across all elements: { "items.product": "A", "items.qty": { $gt: 3 } } matches if SOME element has product A and SOME (possibly different) element has qty > 3 — not necessarily the same one. To require all conditions on a SINGLE element, wrap them in $elemMatch: { items: { $elemMatch: { product: "A", qty: { $gt: 3 } } } }. Forgetting this returns false positives — a common, hard-to-spot correctness bug.

Q2: How does projection help, and what's the include/exclude rule?

Projection selects which fields the query returns, reducing network payload and memory, and can enable covered queries (answered entirely from an index without fetching the document). The rule: within one projection you use inclusions (field: 1) OR exclusions (field: 0), not both — mixing them errors. The single exception is _id, which you can explicitly exclude (_id: 0) alongside inclusions. It's the equivalent of naming columns instead of SELECT * in SQL, with the same performance benefits.

Q3: Why is regex often a performance problem, and what's the alternative?

A left-anchored, case-sensitive regex (/^A/) can use a B-tree index by prefix. But an unanchored (/asha/) or case-insensitive ($options: "i") regex must test every document — a full collection scan that's slow at scale. For substring/full-text needs, use a text index with $text (tokenized, stemmed matching) or, in production, Atlas Search (Lucene-based) which offers relevance, fuzzy matching, and autocomplete. Regex is fine for prefix matches and small collections, not for general text search.

🏋️ Practice: query with $gt/$in/$or/$exists; store documents with an items array of subdocuments and demonstrate the $elemMatch difference vs plain dot-notation; project only certain fields (and exclude _id); use $slice and the positional $ operator; run an anchored regex vs case-insensitive and note the difference; paginate with sort + _id tiebreaker.

🎯 Key Takeaways

  • Queries are documents; operators: comparison ($gt/$in/$ne), logical ($or/$and/$not), element ($exists/$type).
  • Array queries: field:"x" = contains; $all = contains all; $size = length; $elemMatch = same-element conditions.
  • Projection selects fields (all-include or all-exclude, except _id) — smaller payloads, covered queries.
  • Anchored regex can use an index; unanchored/case-insensitive scans — use text index / Atlas Search for real search.
  • Sort with an _id tiebreaker; avoid deep skip(); index your query fields.
