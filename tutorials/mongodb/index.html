Documents & Collections Basics

MongoDB is a document database: instead of rows in tables, it stores flexible JSON-like documents in collections. That flexibility is its superpower and its footgun — the make-or-break skill (covered throughout this track) is schema design, not syntax. If you know the SQL track, this lesson maps the mental models across.

Documents, Collections & BSON

// A document — JSON-like, stored as BSON (Binary JSON):
{
  _id: ObjectId("64f1a2..."),        // primary key, auto-generated if omitted
  name: "Asha",
  email: "asha@example.com",
  roles: ["admin", "editor"],         // arrays are first-class
  address: { city: "Pune", zip: "411001" },   // nested/embedded documents
  createdAt: ISODate("2026-01-15"),
  score: 92
}
// A COLLECTION holds documents (like a table holds rows) but is SCHEMA-FLEXIBLE:
// two documents in one collection can have different fields.

BSON Types & _id

// BSON adds types JSON lacks: ObjectId, ISODate, Decimal128, Binary, Int32/Int64, etc.
_id: ObjectId("...")   // 12 bytes: 4-byte timestamp + machine/process + counter
// → roughly time-ordered, globally unique WITHOUT a central sequence (great for sharding).
// You can use your own _id (e.g., a natural key) — it must be unique.
// Money: use Decimal128 (NumberDecimal("19.99")), NOT double — same float trap as everywhere.
📄 The document model's core idea: data that is accessed together should be stored together. A blog post with its tags and author snapshot in ONE document is a single fast read — no joins. This "locality" is MongoDB's biggest performance win over normalized SQL for the right access patterns — and its biggest trap when misapplied (schema-design lesson).

Schema Flexibility — Power and Responsibility

// Two documents, same collection, different shapes — LEGAL:
{ _id: 1, name: "Asha", plan: "pro" }
{ _id: 2, name: "Bo", plan: "free", trialEndsAt: ISODate(...), referredBy: 5 }
// Flexibility helps: evolving schemas, varied/sparse attributes, rapid iteration.
// It hurts: without discipline, a collection becomes an inconsistent mess your code
// must defensively handle. Production teams add SCHEMA VALIDATION rules:
db.createCollection("users", { validator: { $jsonSchema: {
  required: ["name", "email"], properties: { email: { bsonType: "string" } } } } });
// "Flexible" ≠ "no schema" — it means the schema is enforced where YOU choose, not blindly.

Interview Questions

Q1: How does MongoDB's data model differ from relational, and what's the core trade-off?

MongoDB stores flexible, possibly-nested documents in schema-flexible collections; relational stores rigid rows in normalized tables linked by keys. The core trade-off: MongoDB optimizes for LOCALITY — data accessed together is stored together in one document, so common reads are a single fast lookup with no joins, and the schema can evolve freely. The cost is that relationships spanning documents lack native joins (you embed, denormalize, or use $lookup) and you give up some of SQL's consistency/constraint guarantees unless you add them. It excels when access patterns are document-shaped; it struggles when data is highly relational and queried many ways.

Q2: What is an ObjectId and why is it useful?

The default 12-byte _id: a 4-byte creation timestamp + a random per-process value + an incrementing counter. Properties: globally unique without coordinating a central sequence (critical for distributed writes/sharding), roughly time-ordered (so it doubles as a creation time and sorts chronologically), and generatable client-side before insert. Contrast with a SQL auto-increment, which needs the database to assign it and doesn't distribute well. You can override _id with your own unique value when a natural key fits.

Q3: "MongoDB is schemaless" — is that accurate?

It's schema-FLEXIBLE, not schemaless. Documents in a collection aren't forced into an identical shape by the engine, but real applications always have an implicit schema (the code expects certain fields), and MongoDB supports explicit $jsonSchema validation to enforce required fields/types at write time. "Schemaless" invites inconsistent data and brittle defensive code; the professional framing is that YOU decide where and how strictly to enforce the schema — flexibility as a tool, not an excuse to skip modeling.

🏋️ Practice (mongosh or Atlas free tier): create a users collection and insert documents with different shapes; observe the auto-generated ObjectId and extract its timestamp; add a $jsonSchema validator requiring name+email and try to violate it; store a price as NumberDecimal and compare to double; embed an address subdocument and query by address.city.

🎯 Key Takeaways

  • Documents (BSON) in collections; map: table→collection, row→document, column→field, id→_id.
  • ObjectId = time-ordered, globally-unique, client-generatable primary key — no central sequence.
  • Core model idea: store data accessed together, together (locality) — single fast reads, no joins.
  • Schema-FLEXIBLE, not schemaless — enforce with $jsonSchema validation + app models where it matters.
  • Decimal128 for money; beware the 16MB document limit and unbounded arrays.
