Indexes Intermediate

Indexes are as decisive in MongoDB as in SQL — the difference between a query touching 10 documents and scanning 10 million. MongoDB uses B-tree indexes with the same fundamentals as the SQL track, plus document-specific types (multikey for arrays, text, geospatial) and its own version of the leftmost-prefix rule (the ESR rule).

Creating & the Automatic _id Index

db.users.createIndex({ email: 1 })            // ascending single-field index
db.users.createIndex({ email: 1 }, { unique: true })   // enforce uniqueness
db.users.getIndexes()                         // list; _id always has an index
db.users.dropIndex("email_1")
// Every collection has an automatic unique index on _id. Indexes speed reads but slow writes
// (each write updates every index) and use RAM — the same read-vs-write trade as SQL.

Compound Indexes & the ESR Rule

db.orders.createIndex({ userId: 1, status: 1, createdAt: -1 })
// Like SQL's leftmost-prefix. MongoDB's guideline for ORDERING the fields is ESR:
//   Equality  → fields matched by exact value ($eq, $in)   FIRST
//   Sort      → fields used in .sort()                       NEXT
//   Range     → fields matched by ranges ($gt/$lt)          LAST
// Query: find({ userId: 5, status: "paid", amount: { $gt: 10 } }).sort({ createdAt: -1 })
//   → index { userId:1 (E), createdAt:-1 (S), amount:1 (R) } serves match + sort + range with no in-memory sort.
// A prefix of the index is usable: { userId } and { userId, status } work; { status } alone does not.
📐 ESR (Equality, Sort, Range) is the MongoDB-specific twist on composite-index ordering. Putting sort fields before range fields lets a single index satisfy the filter AND the sort, avoiding a separate in-memory sort (which is capped and can fail on large result sets). It's the highest-value index-design rule to remember for interviews and production.

Multikey (Array) Indexes

db.users.createIndex({ tags: 1 })     // indexing an ARRAY field → a MULTIKEY index automatically
// Each array element gets an index entry, so find({ tags: "admin" }) is fast.
// Limit: a compound index can include AT MOST ONE array field (else the entry combinations explode).

Analyzing with explain

db.orders.find({ userId: 5 }).explain("executionStats")
// Read the winningPlan.stage:
IXSCAN                    // ✅ used an index
COLLSCAN                  // 💀 scanned the whole collection — add/fix an index
FETCH + IXSCAN            // index found candidates, then fetched full docs
PROJECTION_COVERED        // ✅✅ answered entirely from the index (covered query)
totalKeysExamined vs totalDocsExamined vs nReturned  // want examined ≈ returned (selective index)
// A COLLSCAN on a large collection, or docsExamined >> nReturned, means a missing/poor index.

Covered Queries & the Working Set

// COVERED: all queried + projected fields are in the index → no document fetch (fastest):
db.users.createIndex({ email: 1, name: 1 })
db.users.find({ email: "a@x.com" }, { name: 1, _id: 0 })   // served from the index alone
// Index performance depends on indexes fitting in RAM (the "working set"). If indexes spill to
// disk, lookups slow dramatically — a key capacity-planning consideration (performance lesson).

Interview Questions

Q1: Explain the ESR rule for compound indexes.

ESR orders the fields of a compound index as Equality, then Sort, then Range. Equality-matched fields ($eq/$in) go first so the index narrows to exact values; the sort field(s) come next so the index's ordering satisfies .sort() directly (no separate in-memory sort — which is memory-capped and fails on big results); range fields ($gt/$lt) go last because a range "opens up" the remaining index and later fields can't be used for further seeking. Following ESR lets one index serve a filter + sort + range together. It's MongoDB's refinement of the leftmost-prefix idea from SQL.

Q2: What is a multikey index and its main limitation?

When you index an array field, MongoDB creates a multikey index automatically — it stores a separate index entry for EACH array element, so queries matching any element (find({ tags: "x" })) are fast. The key limitation: a single compound index can include AT MOST ONE array field, because indexing two arrays would require the Cartesian product of their elements' entries, which explodes combinatorially. So you can compound an array field with scalar fields, but not two arrays. This constrains how you index documents with multiple arrays and sometimes informs schema choices.

Q3: How do you tell if a query is using an index effectively?

Run explain("executionStats") and inspect the winning plan: IXSCAN means an index was used, COLLSCAN means a full scan (bad on large collections). Then compare totalKeysExamined, totalDocsExamined, and nReturned — ideally examined counts are close to returned (a selective index), whereas docsExamined ≫ nReturned signals a poor/missing index. PROJECTION_COVERED indicates a covered query answered from the index alone (no document fetch — fastest). Also confirm no in-memory SORT stage appears when you have a sort (that means the index didn't cover ordering — apply ESR). These metrics turn index tuning from guessing into measuring.

🏋️ Practice: query an unindexed field and see COLLSCAN in explain, then add the index and re-check for IXSCAN; build a compound index using ESR for a find+sort+range query and confirm no in-memory SORT; create a multikey index on tags and try (and fail) to compound two array fields; make a covered query with a projection matching an index; add a TTL index to auto-expire session documents.

🎯 Key Takeaways

  • B-tree indexes (like SQL); _id is always indexed; indexes speed reads, cost writes + RAM.
  • Compound-index field order follows ESR: Equality → Sort → Range (avoids in-memory sorts).
  • Array fields → automatic multikey indexes; at most ONE array field per compound index.
  • Special types: text, geospatial, partial, TTL (auto-expire), hashed (sharding).
  • explain("executionStats"): want IXSCAN (not COLLSCAN), examined≈returned, PROJECTION_COVERED; keep the index working set in RAM.
