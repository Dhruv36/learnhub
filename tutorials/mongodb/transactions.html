Transactions & Consistency Advanced

MongoDB long lacked multi-document transactions — because good schema design (embedding) meant you rarely needed them. Since 4.0 they exist and are ACID, but the guidance is nuanced: single-document atomicity first, multi-document transactions as a considered exception. Plus write/read concerns tune the durability-vs-latency trade in a replicated system.

Single-Document Atomicity (the default tool)

// EVERY single-document write is atomic — including updates to embedded arrays/subdocuments:
db.accounts.updateOne(
  { _id: 1, balance: { $gte: 100 } },
  { $inc: { balance: -100 }, $push: { history: { type: "debit", amt: 100 } } }
)
// The balance change AND the history append apply together, atomically, under concurrency.
// This is WHY embedding is powerful: model related data in one document and most "transactions"
// become single atomic updates — faster and simpler than multi-document transactions.

Multi-Document Transactions (when you truly need them)

const session = client.startSession();
try {
  session.startTransaction();
  await accounts.updateOne({ _id: 1 }, { $inc: { balance: -100 } }, { session });
  await accounts.updateOne({ _id: 2 }, { $inc: { balance:  100 } }, { session });
  await session.commitTransaction();     // both apply, or neither
} catch (e) {
  await session.abortTransaction();      // rollback on any error
} finally { session.endSession(); }
// ACID across multiple documents/collections. Requires a REPLICA SET (or sharded cluster).
// Costs: locks, overhead, must complete within a time limit (default 60s), and can abort on
// write conflicts (retry). Use for genuine cross-document invariants (transfers between accounts) —
// not as a substitute for good schema design.
⚖️ The philosophy: MongoDB's answer to "how do I keep related data consistent?" is first "put it in one document" (single-doc atomicity), then "duplicate carefully" (schema patterns), and only then "use a transaction". Reaching for multi-document transactions on every write is a sign the schema is modeled like SQL. They're a real, correct tool — used sparingly.

Write Concern — Durability vs Latency

db.orders.insertOne(doc, { writeConcern: { w: "majority", j: true } })
// w: how many replica-set members must ACKNOWLEDGE the write before it returns:
//   w: 1          → the primary only (fast, but a primary crash before replication loses it)
//   w: "majority" → a majority of members (durable across failover — the safe default)
// j: true        → wait until written to the on-disk journal (survives a crash)
// Higher durability = higher latency. Money/critical writes: w:"majority", j:true.
// Fire-and-forget metrics: w:1 is fine. This is YOUR knob on the CAP/durability trade-off.

Read Concern & Read Preference

db.orders.find(q).readConcern("majority")     // read only majority-committed (won't be rolled back) data
// readConcern levels: "local" (latest, may roll back) → "majority" (durable) → "linearizable" (strictest)
// Read PREFERENCE (which member to read from) — the replication lesson:
//   primary (default, freshest) | secondaryPreferred (offload reads, may be STALE)
// Reading from a secondary can return data behind the primary (replication lag) — the
// read-your-writes concern from the SQL scaling lesson, in MongoDB terms.

Interview Questions

Q1: Does MongoDB support ACID transactions, and what's the guidance on using them?

Yes — since 4.0 MongoDB supports multi-document ACID transactions (across documents, collections, and, since 4.2, sharded clusters), requiring a replica set. But the guidance is that they're a considered exception, not the default: single-document writes are already atomic (including nested array/subdocument changes), so a schema that embeds related data turns most "transactions" into one atomic update — faster and lock-free. Multi-document transactions carry overhead (locks, a time limit, possible write-conflict aborts needing retry) and are reserved for genuine cross-document invariants like transferring balance between two accounts. Needing them everywhere usually means the schema is modeled relationally rather than for MongoDB.

Q2: Explain write concern and when you'd raise it.

Write concern controls how many replica-set members must acknowledge a write (and whether it's journaled to disk) before the operation returns. w:1 acknowledges from the primary only — fast, but if the primary crashes before replicating, the write can be lost on failover. w:"majority" waits for a majority of members, so the write survives failover (durable); j:true additionally waits for the on-disk journal. Higher durability costs latency. Raise it (w:"majority", j:true) for money and other critical writes where losing an acknowledged write is unacceptable; leave it low (w:1) for high-volume, non-critical data like metrics. It's your explicit control over the durability-vs-latency trade-off in a distributed system.

Q3: What are the consistency implications of reading from secondaries?

Secondaries replicate the primary asynchronously, so they can lag — a read from a secondary (readPreference secondary/secondaryPreferred) may return data older than the primary, including not yet reflecting a write you just made (a read-your-writes violation). This is the classic replication-lag trade-off: secondary reads scale read throughput and provide locality/failover, but sacrifice freshness. For flows that must see the latest data (a user viewing their own just-saved change), read from the primary; for analytics or tolerant reads, secondaries are fine. readConcern "majority" further ensures you only see data that won't be rolled back after a failover.

🏋️ Practice (replica set / Atlas): do an atomic debit+history-append in one updateOne and confirm atomicity; write a two-account transfer as a multi-document transaction with commit/abort; set writeConcern w:"majority", j:true on a critical insert and observe latency vs w:1; read from a secondary right after a write and observe possible staleness; add retry logic for a transaction that aborts on write conflict.

🎯 Key Takeaways

  • Single-document writes are atomic (incl. embedded arrays) — embedding turns most "transactions" into one atomic update.
  • Multi-document ACID transactions exist (replica set required) — use sparingly for true cross-document invariants; handle aborts/retries.
  • Write concern (w, j) trades durability vs latency — w:"majority", j:true for critical writes.
  • Read concern controls committed-data visibility; secondary reads scale but can be stale (replication lag).
  • Reach for schema (embed/duplicate) before transactions; keep transactions short.
