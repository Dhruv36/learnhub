Performance Tuning Ultra-Advanced

MongoDB performance is mostly about three things: indexes that match your queries, a working set that fits in RAM, and a schema shaped for your access patterns. This lesson is the diagnostic workflow and the levers — the same measure-first discipline as the SQL optimization lesson, applied to documents.

The Diagnostic Workflow

1. FIND slow operations:
   - The slow-query log / database profiler (db.setProfilingLevel(1, { slowms: 100 }))
   - Atlas Performance Advisor (suggests indexes), real-time metrics
2. EXPLAIN the offender: db.coll.find(q).explain("executionStats")
   → COLLSCAN? docsExamined ≫ nReturned? in-memory SORT? (indexes lesson)
3. Apply the matching fix (below); re-measure. One change at a time.
4. Watch server metrics: cache hit ratio, page faults, oplog window, connections.

The WiredTiger Cache & Working Set (the big one)

// MongoDB's storage engine (WiredTiger) caches data + indexes in RAM (~50% of RAM by default).
// Your WORKING SET = the indexes + frequently-accessed documents. Performance is great WHILE the
// working set fits in the cache; when it exceeds RAM, reads cause PAGE FAULTS to disk and latency
// spikes dramatically. Symptoms: rising page faults, falling cache hit ratio, slow reads under load.
// Fixes: fewer/smaller indexes, more RAM, shard to spread the working set, or archive cold data.
// This RAM-fit constraint is the #1 capacity-planning fact for MongoDB.
🧠 Why "indexes in RAM" dominates: an index only accelerates queries if traversing it doesn't hit disk. When indexes outgrow the cache, even indexed queries slow down because index pages must be read from disk. So index hygiene (drop unused ones — they consume cache and slow writes) is as much about RAM as about write speed.

Schema & Access-Pattern Levers

// Many MongoDB "performance problems" are schema problems:
// - Data read together but stored in separate collections → $lookup per request → embed it.
// - The common query can't be indexed selectively → reshape the document / add a computed field.
// - Precompute expensive aggregates (computed pattern) instead of recomputing per read.
// The schema-design lesson IS a performance lesson — access-pattern-driven modeling prevents most issues.

Write Performance

// Every index slows writes (each write updates all indexes) — don't over-index.
// Bulk operations reduce round trips:
db.coll.bulkWrite([ { insertOne: {...} }, { updateOne: {...} }, ... ], { ordered: false })
// { ordered: false } lets independent ops proceed in parallel and past errors.
// Write concern (transactions lesson) trades durability for latency — tune per workload.

Interview Questions

Q1: Why is "does the working set fit in RAM?" the central MongoDB performance question?

WiredTiger caches indexes and hot documents in memory; queries are fast as long as what they touch is in the cache. Once the working set (frequently-used indexes + documents) exceeds RAM, operations must fetch pages from disk (page faults), and latency — even for indexed queries — spikes sharply because index traversal itself hits disk. So RAM fit determines whether your indexes and schema actually deliver their theoretical speed. It drives capacity planning (size RAM to the working set), index hygiene (unused indexes waste cache and slow writes), and the decision to shard (spread the working set across machines). Rising page faults / falling cache-hit ratio are the warning signs.

Q2: Walk your process for diagnosing a slow MongoDB query.

Identify it first via the database profiler (or slow-query log / Atlas Performance Advisor) — don't guess. Run explain("executionStats") and read the plan: COLLSCAN means a missing index; an in-memory SORT means the index doesn't cover the sort (fix with ESR ordering); docsExamined ≫ nReturned means a non-selective index. Apply the matching fix — add/adjust an index, make the query a covered query with projection, or reshape the schema if it's really an access-pattern problem — then re-measure one change at a time. Also check server-level signals (cache hit ratio, page faults) in case the real issue is the working set exceeding RAM rather than any single query.

Q3: Why do many MongoDB performance issues turn out to be schema issues?

MongoDB's speed comes from locality — storing data accessed together in one document so common reads are a single indexed lookup. If the schema is modeled relationally (data spread across many collections), the app compensates with $lookup or per-document N+1 queries on every request, which no amount of index tuning fully fixes. Similarly, unbounded arrays bloat documents and updates, and missing precomputed fields force expensive per-read aggregation. Reshaping toward access patterns (embedding, subset/bucket/computed patterns) removes the work rather than optimizing it — which is why the schema-design lesson is effectively the most important performance lesson.

🏋️ Practice: enable the profiler with slowms and find a slow query; explain it and identify COLLSCAN vs in-memory SORT; add the ESR-ordered index and re-check; make a covered query with a projection; simulate working-set-exceeds-RAM by querying a large collection with no index and watch metrics; convert a loop of updateOne into a single bulkWrite and compare; find and drop an unused index via $indexStats.

🎯 Key Takeaways

  • Diagnose with the profiler + explain (COLLSCAN, in-memory SORT, docsExamined≫nReturned) — measure first.
  • The working set (indexes + hot docs) must fit in the WiredTiger RAM cache — the central perf constraint.
  • Index for real queries (ESR), keep them selective, drop unused ones (they cost cache + writes).
  • Many "perf problems" are schema problems — remodel for access patterns (embed, subset/bucket/computed).
  • bulkWrite for throughput; connection pooling; tune write concern per workload.
