Replica Sets & Read Preferences Advanced

A replica set is a group of MongoDB servers holding the same data — the foundation of high availability and read scaling. Understanding automatic failover, the oplog, and read preferences is essential production knowledge and a common interview topic. This maps directly onto the System Design track's replication lesson.

The Replica Set Topology

            writes + reads              replicate via oplog
   App ──────────────────▶ PRIMARY ──────────────────────▶ SECONDARY
                              │                            ▶ SECONDARY
                              └── heartbeats every 2s ──────┘
// One PRIMARY (takes all writes) + N SECONDARIES (copy the primary's data).
// Recommended: 3 members (or a primary + secondary + ARBITER, which votes but holds no data)
// so a majority can elect a new primary. Odd numbers avoid election ties.

The Oplog — How Replication Works

// The OPLOG (operations log) is a capped collection on the primary recording every write as an
// idempotent operation. Secondaries continuously TAIL the oplog and replay it to stay in sync.
// Properties:
//   - Idempotent entries → safe to re-apply (crash recovery, resync).
//   - Capped (fixed size) → if a secondary falls too far behind (oplog window exceeded), it must
//     do a full resync. Size the oplog for your write volume + expected downtime.
//   - Also powers Change Streams (next lesson) and point-in-time recovery.

Automatic Failover & Elections

// Members exchange heartbeats. If the primary becomes unreachable (past electionTimeout, ~10s),
// the remaining members hold an ELECTION: a MAJORITY must agree on a new primary, which is
// promoted automatically — typically within seconds. The app's driver detects the change and
// reconnects. This is why you need an odd number / a majority: a set that can't reach majority
// (e.g., 2 members, one down) can't elect and goes READ-ONLY.
// w:"majority" writes (transactions lesson) survive failover because a majority already has them.
🔄 Why majority matters everywhere: elections need a majority to avoid split-brain (two primaries). Durable writes (w:"majority") and safe reads (readConcern:"majority") also hinge on majority acknowledgment — that's why a 3-member set (tolerates 1 failure) is the practical minimum for HA, and even-numbered sets get an arbiter to break ties.

Interview Questions

Q1: How does a replica set provide high availability?

A replica set has one primary (accepting writes) and secondaries that continuously replicate its data by tailing the oplog. Members exchange heartbeats; if the primary becomes unreachable, the remaining members hold an election and a majority promotes a new primary automatically, usually within seconds, after which drivers reconnect and writes resume. Because a MAJORITY is required to elect (preventing split-brain), you need an odd number of voting members (three is the practical minimum, tolerating one failure); a set that can't reach majority goes read-only. Combined with w:"majority" writes, this gives automatic failover without losing acknowledged data.

Q2: What is the oplog and why does its size matter?

The oplog is a capped (fixed-size) collection on the primary that records every write as an idempotent operation; secondaries tail it and replay the operations to stay in sync, and it also drives change streams and point-in-time recovery. Idempotency makes re-application safe. Because it's capped, it holds only a rolling window of recent operations — if a secondary is down or lagging longer than that window ("falls off the oplog"), it can no longer catch up incrementally and must perform a full initial resync (expensive). So the oplog must be sized to cover your peak write volume times the longest expected secondary downtime/maintenance window.

Q3: When would you read from secondaries, and what's the risk?

Read from secondaries (via readPreference secondary/secondaryPreferred/nearest) to scale read throughput off the primary, offload analytics/reporting, or get lower latency from a geographically nearer member. The risk is staleness: replication is asynchronous, so secondaries can lag behind the primary and return data that's slightly old — including not yet showing a write the same user just made (a read-your-writes violation). The rule: route reads that must reflect the latest state (a user viewing their own change, correctness-critical logic) to the primary; send tolerant reads to secondaries. readConcern "majority" additionally guards against reading data that could be rolled back after a failover.

🏋️ Practice (Atlas or a local 3-node set): inspect rs.status() and identify primary/secondaries; kill the primary and watch an election promote a new one; observe the oplog (rs.printReplicationInfo()); set readPreference to secondary and read right after a write to observe possible lag; combine w:"majority" writes with a failover to confirm the write survived.

🎯 Key Takeaways

  • Replica set = 1 primary (writes) + N secondaries (copies); ≥3 members / odd count for majority elections.
  • The oplog (capped, idempotent) drives replication, change streams, and recovery — size it for volume + downtime.
  • Automatic failover: heartbeats + majority election promote a new primary in seconds; no-majority = read-only.
  • Read preferences route reads: primary (fresh, read-your-writes) vs secondaries (scale, but may be stale).
  • Majority underpins durable writes, safe reads, and elections — the recurring theme.
