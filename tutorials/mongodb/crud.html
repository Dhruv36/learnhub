CRUD Operations Basics

Create, Read, Update, Delete — the four verbs of any database. MongoDB's update operators ($set, $inc, $push) do partial, atomic modifications inside a document, and upserts plus single-document atomicity are the concepts that matter most for correctness.

Create

db.users.insertOne({ name: "Asha", score: 92, tags: ["a"] })
db.users.insertMany([{ name: "Bo" }, { name: "Cy" }])   // batch — one round trip
// _id is auto-generated (ObjectId) if you don't provide one. insertMany is ordered by default
// (stops on first error); { ordered: false } continues past errors (parallel-ish inserts).

Read — find & findOne

db.users.find({ score: { $gt: 80 } })                    // all matching (a cursor)
db.users.findOne({ email: "asha@x.com" })                // first match or null
db.users.find({ "address.city": "Pune" })                // dot notation into nested docs
db.users.find({ tags: "admin" })                         // array contains "admin"
db.users.find({}, { name: 1, score: 1, _id: 0 })         // PROJECTION: only these fields
    .sort({ score: -1 }).skip(20).limit(10)              // sort, paginate (queries lesson)
// find returns a CURSOR (lazy, batched) — iterate it; toArray() materializes all.

Update — Operators, Not Replacement

db.users.updateOne(
  { _id: 1 },
  { $set: { score: 95, "address.city": "Mumbai" },   // set fields (creates if missing)
    $inc: { loginCount: 1 },                           // atomic increment
    $push: { tags: "vip" },                            // append to array
    $addToSet: { roles: "admin" },                     // append only if not present
    $currentDate: { updatedAt: true } }
)
db.users.updateMany({ plan: "trial" }, { $set: { plan: "free" } })
// ⚠ WITHOUT operators you REPLACE the whole document:
db.users.updateOne({ _id: 1 }, { name: "X" })   // 💀 error in modern drivers, or wipes other fields
// Always use $set etc. for partial updates. replaceOne() is the explicit full-replace.
OperatorDoes
$set / $unsetSet / remove a field
$inc / $mulAtomic increment / multiply
$push / $pull / $addToSetArray append / remove-matching / append-if-absent
$min / $maxSet only if smaller / larger
$rename, $currentDateRename field, set timestamp

Upsert — Insert-or-Update

db.counters.updateOne(
  { _id: "visits" },
  { $inc: { n: 1 } },
  { upsert: true }          // create the doc if the filter matches nothing, then apply
)
// Atomic "create if missing, else update" — no SELECT-then-INSERT race (SQL upsert parallel).
// findOneAndUpdate({...}, {...}, { upsert: true, returnDocument: "after" }) returns the doc.

Delete

db.users.deleteOne({ _id: 1 })
db.users.deleteMany({ plan: "expired" })
db.users.deleteMany({})     // 💀 deletes EVERYTHING (like SQL DELETE without WHERE)

Atomicity — the Single-Document Guarantee

// A single-document write is ATOMIC: this whole update applies fully or not at all,
// even under concurrency — no other operation sees a half-updated document:
db.accounts.updateOne({ _id: 1, balance: { $gte: 100 } },
                      { $inc: { balance: -100 } })   // race-safe debit; check matchedCount
// This is why the schema-design lesson pushes EMBEDDING related data: a single-document
// update gives you atomicity WITHOUT multi-document transactions. Multi-document atomicity
// needs explicit transactions (transactions lesson) — slower, use when truly needed.

Interview Questions

Q1: What happens if you update a document without using operators like $set?

You attempt a full-document REPLACEMENT: the update document replaces the entire matched document except _id. In practice, updateOne with a raw (non-operator) document throws in modern drivers to prevent accidental data loss; the intentional full replace is replaceOne. The lesson: for partial updates always use update operators ($set, $inc, $push…), which modify specific fields atomically and leave the rest intact. Forgetting this and using an old driver silently wipes every field you didn't include.

Q2: How do you safely increment a counter or decrement a balance under concurrency?

Do it in a single atomic update with $inc, not read-modify-write in application code (which races and loses updates). db.accounts.updateOne({ _id, balance: { $gte: amt } }, { $inc: { balance: -amt } }) both checks sufficiency and applies the change atomically at the document level; then verify matchedCount === 1 to confirm it applied (0 = insufficient balance). Single-document writes are atomic in MongoDB, so this is race-safe without a transaction — the same principle as SQL's atomic UPDATE.

Q3: What does upsert do and why prefer it?

With { upsert: true }, an update whose filter matches no document INSERTS a new one (built from the filter + update operators) instead of doing nothing; if it matches, it updates normally. It's the atomic "create if missing, else update" primitive — avoiding the SELECT-then-INSERT race where two concurrent requests both find nothing and both insert (duplicates or errors). Ideal for counters, idempotent writes, and get-or-create semantics; findOneAndUpdate with upsert returns the resulting document.

🏋️ Practice: insertMany some users; find with $gt + projection + sort + limit; update one with $set + $inc + $push in a single call; try updating without operators to see the error; do an atomic balance decrement guarded by $gte and check matchedCount; implement a visit counter with upsert; run deleteMany with a specific filter (verify with find first).

🎯 Key Takeaways

  • insertOne/insertMany, find/findOne (cursor + projection + sort/skip/limit), updateOne/Many, deleteOne/Many.
  • Update with OPERATORS ($set/$inc/$push/$addToSet) — a raw document replaces the whole thing.
  • Single-document writes are atomic — use $inc for race-safe counters/balances; check matchedCount.
  • upsert: true = atomic create-or-update (no SELECT-then-insert race).
  • Guard deleteMany/updateMany filters (test with find); embedding + single-doc atomicity avoids many transactions.
