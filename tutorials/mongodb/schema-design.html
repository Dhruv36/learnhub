Schema Design: Embed vs Reference Intermediate

This is the MongoDB lesson. Unlike SQL where normalization is the default and design is largely mechanical, MongoDB schema design is an art driven by one question: how is the data accessed? Get embed-vs-reference right and MongoDB flies; get it wrong and you fight the database forever. It's the most important MongoDB interview topic by far.

The Fundamental Choice

// EMBED — nested inside the parent document (data accessed together):
{ _id: 1, name: "Asha",
  address: { city: "Pune", zip: "411001" },        // one-to-one
  orders: [ { total: 50, date: ... }, { total: 30, date: ... } ] }   // one-to-few

// REFERENCE — store an id, look up separately (like a SQL FK):
{ _id: 1, name: "Asha" }
{ _id: 100, userId: 1, total: 50 }                 // orders collection references the user
Embed when…Reference when…
Data is read together (one query, no join)Data is queried independently
One-to-one or one-to-FEW, boundedOne-to-MANY (unbounded) or many-to-many
The child doesn't change often / has no independent lifeThe child is large, shared, or updated frequently
You want single-document atomicityThe embedded array would grow without limit (→16MB)

The Golden Rule: Model for Access Patterns

// Don't ask "what's the data?" (SQL thinking) — ask "what QUERIES will run, how often?"
// Blog post shown with its comments on one page? → embed comments (one read).
// Comments also shown across posts / paginated / millions per post? → separate collection.
// The SAME entities can be modeled differently depending on how the app reads them.
// This is why there's no universal "correct" schema — it's access-pattern-driven.
🎯 The reframing that unlocks MongoDB: in SQL you normalize first and let joins assemble data at read time. In MongoDB you shape documents to match your READ patterns, so the common query is a single document fetch. You're trading write-time duplication and denormalization for read-time speed and locality — a deliberate inversion of the SQL default. Design around the queries your app actually makes.

The Patterns You Must Know

PatternWhat / when
One-to-few (embed array)A user's few addresses — embed directly
One-to-many (reference)A user's orders (grows unbounded) — orders collection with userId
One-to-squillions (parent-reference)Log lines per host — the child references the parent (host_id on each log)
Subset patternEmbed the 5 most recent reviews on the product; keep all in a reviews collection — fast common read + full data available
Bucket patternGroup time-series readings into hourly "bucket" documents — fewer, bigger docs; avoids per-reading overhead & unbounded arrays
Extended referenceDuplicate a few fields (author name) onto the order so the common view needs no join — denormalization for reads
Computed patternStore precomputed totals/counts, updated on write — avoid recomputing on every read

$lookup — Joins Exist, But…

db.orders.aggregate([
  { $lookup: { from: "users", localField: "userId", foreignField: "_id", as: "user" } }
])
// $lookup performs a LEFT JOIN in the aggregation pipeline. Use it for genuinely relational
// reporting/analytics. But if you find yourself $lookup-ing on every request, your schema is
// probably modeled like SQL — reconsider embedding or the extended-reference pattern. $lookup
// is a tool, not the default; it doesn't leverage the document model's locality.

Duplication & the Consistency Trade-off

// Extended reference duplicates author.name onto each order. If the author renames themselves,
// you must update the duplicates (or accept staleness). Rules:
// - Duplicate data that RARELY changes (names for display, price-at-purchase snapshots).
// - For a snapshot of a point in time (order's price), duplication is CORRECT, not a bug.
// - For frequently-changing shared data, reference it instead.
// The write-side cost of keeping duplicates in sync is the price of read-side speed.

Interview Questions

Q1: How do you decide between embedding and referencing?

Drive it from access patterns and cardinality. Embed when the data is read together, the relationship is one-to-one or one-to-few and bounded, and the child has no independent query life — you get a single fast read and single-document atomicity. Reference when the relationship is one-to-many/unbounded (would blow the 16MB doc limit or bloat updates), the child is large or shared, or it's queried on its own. The heuristic: "data accessed together, stored together"; cardinality (few vs many vs squillions) and update frequency break ties. Crucially, the same entities can be modeled either way depending on how the application reads them.

Q2: What's the subset (and bucket) pattern, and what problem does each solve?

Subset: embed only the frequently-accessed portion (e.g., the 5 latest reviews on a product document) while keeping the full set in a separate collection — the common page render is one read, and the complete data is still available on demand, avoiding an unbounded embedded array. Bucket: group many small related items (IoT/time-series readings) into fewer larger "bucket" documents (e.g., one doc per sensor per hour holding an array of readings) — this cuts per-document overhead, keeps arrays bounded, and makes range queries efficient. Both manage the tension between read-locality and the document-size/array-growth limits.

Q3: When is duplicating data across documents correct rather than a mistake?

When the duplicated value rarely changes or represents a point-in-time snapshot. Extended reference (copying an author's display name onto orders) trades a tiny, infrequent sync cost for eliminating a join on every read. Snapshotting the price at purchase time onto the order is not denormalization to fix later — it's correct temporal modeling (the historical price must not change when the product's price does). The rule: duplicate rarely-changing or historical data freely; reference data that changes often and must stay consistent everywhere. Duplication is a deliberate read- optimization with an accepted write-side maintenance cost — the same trade-off as SQL denormalization, just more central to MongoDB.

🏋️ Practice: model a blog: decide embed vs reference for author, tags, and comments given "post page shows post + author name + latest 10 comments" and "a moderation queue lists all comments across posts"; implement the subset pattern (latest N comments embedded + full comments collection); design a bucket schema for per-minute metrics; use $lookup to join orders→users and note when you'd rather have embedded/extended-referenced.

🎯 Key Takeaways

  • Schema design is MongoDB's core skill — driven by ACCESS PATTERNS, not by normalizing the data.
  • Embed for read-together / one-to-few / atomicity; reference for one-to-many-unbounded / shared / independently-queried.
  • Know the patterns: subset, bucket, extended reference, computed, parent-reference for one-to-squillions.
  • $lookup joins exist but shouldn't be your default — $lookup-on-every-request means SQL-style modeling.
  • Duplicate rarely-changing / snapshot data (correct, not a smell); reference volatile shared data.
