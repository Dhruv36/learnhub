Aggregation Pipeline Intermediate

The aggregation pipeline is MongoDB's data-processing engine — documents flow through a series of stages, each transforming the stream. It's how you do grouping, joins, reshaping, and analytics (MongoDB's GROUP BY, window functions, and more). Think of it as a Unix pipe of documents; the order of stages is everything for both correctness and performance.

The Pipeline Model

db.orders.aggregate([
  { $match:  { status: "paid" } },                       // filter (like WHERE) — do this FIRST
  { $group:  { _id: "$userId", total: { $sum: "$amount" }, n: { $sum: 1 } } },   // like GROUP BY
  { $sort:   { total: -1 } },
  { $limit:  10 }
])
// Documents stream through stages top-to-bottom. $match early SHRINKS the data so later stages
// (and any index) do less work — the #1 aggregation performance rule.

The Essential Stages

StageDoes (SQL analogy)
$matchFilter documents (WHERE) — use early; uses indexes if first
$groupGroup + aggregate (GROUP BY): $sum, $avg, $min, $max, $push, $addToSet, $first
$sort / $limit / $skipOrder and paginate
$project / $set (addFields)Reshape: include/exclude/compute fields (SELECT)
$lookupLeft join another collection
$unwindDeconstruct an array → one document per element
$count, $facetCount; run multiple sub-pipelines at once (dashboards)

$group — the Heart of Aggregation

{ $group: {
    _id: "$category",                       // group key (null = whole collection)
    revenue: { $sum: "$price" },
    avgPrice: { $avg: "$price" },
    count: { $sum: 1 },                      // count rows in the group
    products: { $push: "$name" }             // collect values into an array
} }
// _id is the grouping key; every other field uses an accumulator. Group by multiple fields:
_id: { cat: "$category", month: { $month: "$createdAt" } }

$unwind — Working With Arrays

// order: { items: [{ product: "A", qty: 2 }, { product: "B", qty: 5 }] }
db.orders.aggregate([
  { $unwind: "$items" },                    // → one doc per item (flattens the array)
  { $group: { _id: "$items.product", sold: { $sum: "$items.qty" } } }
])
// $unwind explodes an array so you can group/analyze individual elements. Essential for
// "top products", "revenue per line item", etc. Watch cardinality — it multiplies document count.

Worked Example — Monthly Revenue Report

db.orders.aggregate([
  { $match: { status: "paid", createdAt: { $gte: ISODate("2026-01-01") } } },
  { $group: {
      _id: { $dateToString: { format: "%Y-%m", date: "$createdAt" } },
      revenue: { $sum: "$amount" },
      orders:  { $sum: 1 },
      avgOrder:{ $avg: "$amount" } } },
  { $sort: { _id: 1 } },
  { $project: { month: "$_id", revenue: 1, orders: 1,
                avgOrder: { $round: ["$avgOrder", 2] }, _id: 0 } }
])
// $match → $group → $sort → $project: filter, aggregate, order, reshape. Also available:
// $bucket (histograms), $setWindowFields (window functions — running totals/rank), $merge (write results).

Performance Rules

1. $match and $sort EARLY — before $group/$unwind — so they use indexes and shrink data.
   (An index helps $match/$sort only while it's near the FRONT of the pipeline.)
2. $project to drop big unused fields before heavy stages.
3. Avoid $unwind on huge arrays without a preceding $match.
4. Use allowDiskUse for large $group/$sort that exceed the 100MB per-stage memory limit.
5. explain("executionStats") shows whether a stage used an index (IXSCAN) or scanned (COLLSCAN).

Interview Questions

Q1: How does the aggregation pipeline work, and why does stage order matter?

Documents stream through an ordered list of stages, each taking the previous stage's output and transforming it (filter, group, reshape, join, unwind…). Order matters for correctness (grouping before filtering gives different results) and, critically, performance: $match and $sort placed at the FRONT can use indexes and drastically reduce how many documents later, expensive stages ($group, $unwind, $lookup) must process. Put a late $match after a $group and you've grouped the entire collection unnecessarily. The rule is filter-and-shrink early.

Q2: What does $unwind do and when do you need it?

$unwind deconstructs an array field, emitting one output document per array element (with that element replacing the array). You need it to analyze or group by individual array members — e.g., computing units sold per product from orders whose items are an embedded array, or faceting on tags. Caveat: it multiplies document count (an order with 5 items becomes 5 documents), so filter first ($match) and be mindful of memory. It's the bridge between the document model's embedded arrays and element-level aggregation.

Q3: How do you make an aggregation fast?

Place $match (and $sort) at the very start so they leverage indexes and cut the working set before costly stages — an index only helps these stages while they're near the front. $project away large unneeded fields early to reduce data moved between stages. Avoid unfiltered $unwind on big arrays. For large $group/$sort that exceed the 100MB per-stage limit, enable allowDiskUse (or better, add indexes so sorts are index-backed). Always verify with explain("executionStats") that key stages show IXSCAN (index) not COLLSCAN. And if a pipeline leans on heavy $lookup every request, the schema — not the query — is usually the real fix.

🏋️ Practice: build the monthly-revenue pipeline ($match→$group→$sort→$project); compute top-5 products by units sold using $unwind + $group; group by two keys (category + month); use $facet to return both a count and a top-N list in one query; run explain and confirm your $match uses an index; force a big sort and add allowDiskUse.

🎯 Key Takeaways

  • Documents stream through ordered stages; it's MongoDB's GROUP BY + reshape + join + window engine.
  • Core stages: $match (filter), $group (aggregate), $project/$set (reshape), $sort/$limit, $lookup (join), $unwind (arrays).
  • $group needs _id (the key) + accumulators ($sum/$avg/$push); $unwind flattens arrays for element-level analysis.
  • Put $match/$sort FIRST to use indexes and shrink data — the top performance rule.
  • Mind the 100MB/stage limit (allowDiskUse); verify with explain (IXSCAN vs COLLSCAN).
