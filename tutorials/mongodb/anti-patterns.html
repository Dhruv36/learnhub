Anti-Patterns at Scale Ultra-Advanced

MongoDB's flexibility makes it easy to build things that work in development and collapse in production. This capstone catalogues the anti-patterns that senior engineers watch for — most trace back to ignoring access patterns or the document model's limits. Recognizing them is the difference between MongoDB being a superpower and a liability.

The Anti-Pattern Catalogue

The Two Root Causes

Almost every anti-pattern above reduces to one of two mistakes:
1. IGNORING ACCESS PATTERNS — designing around "what is the data?" (SQL habit) instead of
   "what queries run, how often?" → wrong embed/reference choices, $lookup sprawl, bad indexes.
2. FIGHTING THE DOCUMENT MODEL'S LIMITS — 16MB docs, bounded arrays, RAM-fit working set,
   single-array multikey, shard-key permanence. Respect these and design within them.
Get the schema right (for YOUR queries) and most scale problems never appear.
🎓 The senior mindset: MongoDB rewards teams that decide their access patterns first and model documents to serve them, and punishes teams that treat it as a schemaless dumping ground or a relational database with JSON columns. When evaluating a MongoDB design, the first question is always "what are the queries, and how often?" — the answer dictates embed vs reference, indexes, shard key, and everything else.

Scaling Escalation (recap)

Slow queries    → indexes (ESR) + schema for access patterns
Read volume     → secondaries (read preference) + application cache
Big/hot docs    → subset/bucket patterns, referencing
Write volume /  → sharding with a well-chosen key (last resort)
  data > 1 node
// The same ladder as SQL: exhaust cheaper rungs before adding distributed complexity.

MongoDB vs SQL — When to Choose Which

Lean MongoDB when…Lean relational when…
Data is document-shaped and read togetherData is highly relational, queried many ways
Schema evolves rapidly / variesStrong multi-row transactional integrity is central
Access patterns are known and stableAd-hoc querying across entities is common
Horizontal write scaling is a near-term needComplex joins/aggregations across normalized data

The mature answer is rarely dogmatic — many systems use both (polyglot persistence): the right store per workload. Knowing the trade-offs, not defending a favorite, is the senior signal.

Interview Questions

Q1: What's wrong with storing an unbounded array in a document, and what do you do instead?

Documents have a hard 16MB limit, and every update to an array rewrites relevant portions of the document, so an array that grows without bound (all comments on a post, all events for a device) eventually hits the limit, makes updates progressively slower, bloats the working set, and forces reads to pull huge documents. Instead: reference the children in a separate collection (one-to-many), use the subset pattern (embed only the latest N, keep the rest in a collection), or the bucket pattern for time-series (group readings into capped bucket documents). The principle is to keep embedded arrays BOUNDED and move unbounded growth out of the parent document.

Q2: Someone says "MongoDB doesn't scale / is slow." How do you evaluate the claim?

Usually the schema, not the database, is the problem. I'd check for the classic anti-patterns: data modeled relationally so $lookup runs on every request (should be embedded/extended-reference); unbounded arrays or bloated documents; missing or ESR-violating indexes causing COLLSCANs and in-memory sorts; the working set exceeding RAM (page faults); a monotonic shard key creating a hot shard; regex used for text search. Almost all of these trace to ignoring access patterns or fighting the document model's limits. Fix the schema and indexes to match the actual queries and MongoDB scales well; if the data is genuinely highly relational and queried every which way, that's a signal it may be the wrong store — a legitimate but different conclusion.

Q3: When would you NOT choose MongoDB?

When the data is highly relational and queried in many unpredictable ways (ad-hoc joins across normalized entities), where a relational engine's joins, constraints, and mature query planner shine and MongoDB would force you into $lookup sprawl or heavy duplication. When strong multi-row/multi- table transactional integrity is central and pervasive (complex financial ledgers with many invariants), relational ACID with foreign keys and constraints is a more natural fit — MongoDB has transactions but they're meant to be the exception. And when access patterns are unknown/volatile, normalization's flexibility to query any way beats MongoDB's need to model for specific patterns. The honest answer avoids dogma: pick per workload, and polyglot persistence (using both) is common and legitimate.

🏋️ Practice: audit a given schema for anti-patterns (unbounded arrays, SQL-style modeling, missing indexes, monotonic shard key) and propose fixes; take a "comments embedded in post" design that's hitting limits and redesign with subset + reference; add $jsonSchema validation to a loose collection; for three sample systems decide MongoDB vs relational and justify with access patterns; write the one-paragraph "what are the queries?" analysis you'd do before modeling any MongoDB schema.

🎯 Key Takeaways

  • Top anti-patterns: unbounded arrays, SQL-style modeling + $lookup sprawl, bloated docs, over-indexing, monotonic shard keys, transactions everywhere, "schemaless" chaos.
  • Two root causes: ignoring access patterns, and fighting the document model's limits (16MB, arrays, RAM fit, shard-key permanence).
  • Get the access-pattern-driven schema right and most scale problems never occur.
  • Escalation ladder mirrors SQL: indexes/schema → read replicas + cache → patterns → shard last.
  • Choose MongoDB vs relational by workload (document-shaped vs highly-relational); polyglot is fine — know the trade-offs, don't be dogmatic.
