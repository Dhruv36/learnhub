Change Streams Advanced

Change streams let your application subscribe to data changes in real time — inserts, updates, deletes — without polling. They power reactive features (live dashboards, notifications, cache invalidation) and event-driven architectures, turning the database into an event source. They're built on the oplog you met in the replication lesson.

The Basic Pattern

const stream = db.collection("orders").watch();     // subscribe to all changes
for await (const change of stream) {
  console.log(change.operationType, change.documentKey);
  // operationType: "insert" | "update" | "replace" | "delete" | "invalidate"
  // change.fullDocument (for inserts, or updates with the option below)
  // change.updateDescription.updatedFields (what changed on an update)
}
// Instead of polling "any new orders?" every second, the DB PUSHES changes as they happen —
// lower latency, less load. Watch a collection, a database, or the whole deployment.

Filtering & Options

// Filter with an aggregation pipeline so you only receive relevant changes:
db.orders.watch([
  { $match: { operationType: "insert", "fullDocument.total": { $gt: 1000 } } }
], { fullDocument: "updateLookup" });   // include the full doc on updates (else only the delta)
// The pipeline runs server-side — you don't ship irrelevant events over the network.

Resumability — the Production-Critical Feature

// Every change event has a RESUME TOKEN. Persist the last one; if your consumer crashes or
// disconnects, resume EXACTLY where you left off — no missed or duplicated events:
let resumeToken;
const stream = db.orders.watch([], { resumeAfter: savedToken });
for await (const change of stream) {
  process(change);
  resumeToken = change._id;         // persist this (e.g., to a DB) after handling
}
// This is what makes change streams reliable for event pipelines — unlike naive polling,
// you can't lose events across restarts (as long as the change is still in the oplog window).
🔑 Resume tokens + the oplog: change streams read from the oplog, so resumability works only while the events are still within the oplog window (replication lesson). If a consumer is down longer than the oplog retains, the token becomes invalid and you must do a full reconciliation. Size the oplog for your maximum consumer downtime — the same constraint as secondary resync.

Interview Questions

Q1: What are change streams and how do they differ from polling?

Change streams are a real-time subscription to data changes (insert/update/replace/delete) on a collection, database, or deployment, delivered as events as they happen. Compared to polling (repeatedly querying "what's new since last time?"), they push changes with lower latency, less database load, and no missed-window/duplicate issues from interval timing. They're built on the oplog and require a replica set. They turn the database into an event source for reactive features (live UIs, cache invalidation, search-index sync) and event-driven architectures without a separate CDC tool.

Q2: Why are resume tokens important, and what's their limitation?

Each change event carries a resume token; by persisting the last processed token and passing it as resumeAfter when reconnecting, a consumer picks up EXACTLY where it left off after a crash, deployment, or network blip — no lost or (mostly) duplicated events. This is what makes change streams reliable for building event pipelines. The limitation: change streams read from the capped oplog, so a token is only valid while its position is still within the oplog window. If a consumer is offline longer than the oplog retains, the token becomes invalid and you must fall back to a full reconciliation/resync. Hence you size the oplog to cover expected consumer downtime, and make handlers idempotent to tolerate rare duplicates.

Q3: Give a production use case and the pattern.

Syncing a search index (Atlas Search/Elasticsearch): watch the products collection with a $match for relevant operations and fullDocument: "updateLookup", and on each change upsert/delete the corresponding search document — keeping the index consistent with the source in near-real-time, resumable via tokens and idempotent so replays are safe. The same shape powers the outbox/CDC pattern (publish DB changes to Kafka for microservices), cache invalidation, and live dashboards (push events to clients over WebSocket). The key production concerns are persisting resume tokens, sizing the oplog for downtime, filtering server-side, and idempotent handlers.

🏋️ Practice (replica set/Atlas): open a watch() on a collection and insert/update documents in another shell to see events; add a $match filter for only large inserts and fullDocument:"updateLookup"; persist and use a resume token — kill the consumer, insert more, restart with resumeAfter, and confirm no missed events; make the handler idempotent and re-process a token to prove no double effects.

🎯 Key Takeaways

  • Change streams push real-time change events (no polling) — built on the oplog; require a replica set.
  • Filter server-side with a $match pipeline; use fullDocument:"updateLookup" to get the whole doc on updates.
  • Resume tokens make consumers crash-safe — persist the last token; resume exactly where you left off.
  • Validity is bounded by the oplog window; size it for downtime and make handlers idempotent.
  • Use cases: live UIs, cache invalidation, search-index sync, CDC/outbox to event systems.
