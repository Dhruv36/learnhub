Templates & Slots Production

What you'll master: the template element (inert, reusable markup fragments — the platform's own answer to "clone this HTML repeatedly without re-parsing it"), named slots for composing content INTO a Web Component from the outside, and how these two elements complete the Web Components picture from the previous lesson into something genuinely production-capable. Prerequisites: Web Components & Shadow DOM — this lesson is its direct continuation.

Part 1 — template: Inert, Parsed, But Not Rendered

<template id="card-template">
  <div class="card">
    <h3></h3>
    <p></p>
  </div>
</template>

Content inside <template> is genuinely unusual among HTML elements: it IS parsed by the browser (unlike a raw string in a JS variable, which the browser knows nothing about until you explicitly do something with it) but is never rendered and never executed — images inside don't load, scripts inside don't run, and it's entirely absent from the render tree (lesson "Critical Rendering Path"'s Stage 3, for the exact same reason display:none content is absent — template content simply isn't part of the "live" document at all until explicitly cloned out). It exists specifically to solve the "I need to stamp out the same chunk of markup many times" problem WITHOUT the cost of either re-parsing an HTML string repeatedly (slow, and — critically — a genuine XSS risk if that string ever contains unescaped user data, since innerHTML parses and EXECUTES) or hand-building the same DOM structure node-by-node in JavaScript every single time (verbose, error-prone).

Part 2 — Cloning: the Actual Usage Pattern

const template = document.getElementById("card-template");
const products = [
  { name: "Yirgacheffe", price: "$18" },
  { name: "Huila", price: "$16" },
];

products.forEach(product => {
  const clone = template.content.cloneNode(true);   // true = deep clone, including all descendants
  clone.querySelector("h3").textContent = product.name;   // textContent — SAFE, no HTML parsing
  clone.querySelector("p").textContent = product.price;
  document.getElementById("product-list").appendChild(clone);
});

Two details that matter, both directly connected to earlier lessons: template.content is a DocumentFragment (a lightweight container specifically designed to hold nodes before they're inserted into the real document — cloning it and appending is more efficient than repeatedly touching the live DOM one element at a time, since the live DOM only needs to update ONCE per clone rather than once per individual node created). And using .textContent (not .innerHTML) to fill in the cloned template's placeholders is a DELIBERATE security choice, not an arbitrary one — it's the exact same "escape/don't-parse untrusted data" discipline from the Global Attributes and Web Components lessons' XSS warnings, now applied specifically at the template-cloning step where it's genuinely easy to forget because the TEMPLATE itself already felt "safe."

Part 3 — slot: Composing Content INTO a Component From Outside

This is the piece that completes the previous lesson's Shadow DOM story. Shadow DOM hides a component's INTERNALS from the outside page — but real components almost always need the OUTSIDE page to supply SOME content (a button's label, a card's body text). <slot> is the explicit, deliberate hole you punch in that isolation boundary for exactly this purpose:

class InfoCard extends HTMLElement {
  connectedCallback() {
    const shadow = this.attachShadow({ mode: "open" });
    shadow.innerHTML = `
      <style>
        .card { border: 1px solid #ddd; border-radius: 8px; padding: 16px; }
        .title { font-weight: 700; }
      </style>
      <div class="card">
        <div class="title"><slot name="title">Untitled</slot></div>
        <div><slot>No content provided.</slot></div>    <!-- the DEFAULT (unnamed) slot -->
      </div>
    `;
  }
}
customElements.define("info-card", InfoCard);
<info-card>
  <span slot="title">Ethiopia Yirgacheffe</span>
  <p>Floral, tea-like, small-batch roasted.</p>   <!-- lands in the DEFAULT slot -->
</info-card>

The mechanics: light-DOM children (the actual <info-card>...</info-card> content the page author writes) get PROJECTED into the matching named <slot> inside the shadow root — slot="title" matches <slot name="title">; anything with no slot attribute at all falls into the single unnamed/default slot. Each slot can carry FALLBACK content (shown: "Untitled", "No content provided.") that renders only when the outside page supplies nothing for that slot — genuinely useful default-content behavior, for free. This is directly analogous to children/props.children in React or the default `<slot>` concept in Vue — but, as with everything in this Production section, it's the platform-native version, requiring no framework runtime to work.

Part 4 — Worked Example: template + slot + Custom Element, Combined

<template id="product-card-template">
  <style>
    .card { border: 1px solid #e5e7eb; border-radius: 8px; padding: 16px; max-width: 280px; }
    .price { color: #059669; font-weight: 700; }
  </style>
  <div class="card">
    <slot name="image"></slot>
    <h3><slot name="title">Product</slot></h3>
    <p class="price"><slot name="price">$0</slot></p>
    <slot></slot>   <!-- default slot: description -->
  </div>
</template>

<script>
class ProductCard extends HTMLElement {
  connectedCallback() {
    const template = document.getElementById("product-card-template");
    const shadow = this.attachShadow({ mode: "open" });
    shadow.appendChild(template.content.cloneNode(true));   // template feeds the shadow root's structure
  }
}
customElements.define("product-card", ProductCard);
</script>

<product-card>
  <img slot="image" src="yirg.avif" alt="Ethiopia Yirgacheffe beans" width="240" height="160">
  <span slot="title">Ethiopia Yirgacheffe</span>
  <span slot="price">$18</span>
  <p>Floral, tea-like, small-batch roasted.</p>
</product-card>

This is the complete, real pattern: <template> defines the component's internal STRUCTURE once (parsed, inert, reusable); the Custom Element clones it into a Shadow DOM for genuine style isolation (previous lesson); named <slot>s let the page author compose real content INTO specific positions inside that isolated structure, each with sensible fallback content, using zero framework runtime and zero build step. Every one of the earlier lessons' rules still applies unchanged INSIDE this pattern — the image still needs real alt text (Images Mastery), the markup still needs correct semantics (Choosing the Right Element) — Web Components change WHERE your HTML lives and how it's packaged, never what makes HTML correct in the first place.

Interview Questions

Q1 (beginner): What makes template element content different from a normal element's content?

It's parsed by the browser (unlike a raw JS string) but never rendered, never executed, and entirely absent from the render tree — images don't load, scripts don't run, and nothing is visible on screen until the content is explicitly cloned out via template.content.cloneNode() and inserted elsewhere in the live document. It exists specifically to hold reusable markup fragments efficiently and safely.

Q2 (beginner): Why use textContent rather than innerHTML when filling in a cloned template's placeholder values?

innerHTML parses its input as HTML and will execute any script/event-handler content embedded in it — a genuine XSS risk if the value being inserted ever comes from user input or any untrusted source. textContent inserts the value as plain, literal text with no parsing at all, which is the safe default whenever you're inserting DATA (a name, a price) rather than deliberately inserting markup you've already sanitized.

Q3 (mid): Explain what a named slot does, mechanically, and what happens when a page author supplies no content for it.

A named slot (<slot name="title">) inside a Shadow DOM structure acts as a projection point: any light-DOM child of the custom element carrying a matching slot="title" attribute gets rendered at that exact position inside the shadow root, while the component's internal isolation (previous lesson) otherwise remains fully intact. If no matching content is supplied, whatever markup is written directly INSIDE the <slot> tag itself renders as fallback content — genuinely useful default behavior requiring no additional JS logic.

Q4 (mid): A developer clones a template 500 times to render a large product grid and notices the page becomes sluggish. What's the likely cause, and how does it relate to the Critical Rendering Path lesson?

Most likely: each clone is being individually appendChild'd directly to the live DOM one at a time INSIDE the loop, forcing 500 separate layout/paint cycles (Critical Rendering Path lesson's Stage 4/5) — one per insertion — rather than being batched. The fix: accumulate all 500 clones into a single DocumentFragment first (or use a single parent fragment), then append that ONE fragment to the live DOM a single time at the end of the loop — this reduces the operation to one layout/paint pass instead of 500, directly applying the earlier lesson's "batch writes, don't interleave with the live DOM repeatedly" principle to this specific cloning pattern.

Q5 (senior): Your design system's Card component uses three named slots (header, media, footer) plus a default slot for body content. A consuming team reports that when they forget to include the media slot's content, there's an awkward empty gap in the rendered card rather than the layout gracefully collapsing. Diagnose and fix.

Diagnosis: an empty/unfilled named slot with NO fallback content still occupies its structural position in the shadow root's layout (a <slot> with nothing projected into it and no fallback content renders as an empty, zero-content — but still positionally present — element, and any CSS giving that slot's container fixed dimensions, padding, or margin will visibly reserve that space regardless of whether content exists). Two complementary fixes: (1) CSS-level — use :has() or a similar selector strategy to detect an empty slot and collapse its container's margin/padding when genuinely empty (a real, if occasionally verbose, CSS technique for this exact "collapse when empty" problem); (2) JS-level, more robust — in connectedCallback or via a slotchange event listener, check `slot.assignedNodes().length === 0` for the media slot specifically and toggle a class on its container (e.g., `.media-slot--empty { display: none }`) when nothing was actually projected into it, giving explicit, reliable control rather than relying purely on CSS heuristics. The senior framing: this is fundamentally the same "reserve space correctly" discipline as the Performance section's CLS lessons, applied here to a component's OWN internal layout rather than the page's overall layout — the same underlying principle (know what's actually going to be there before deciding how much space to reserve for it) recurring at a different scale.

Q6 (senior): Compare, in concrete engineering terms, using template+slot+Custom Elements versus React's JSX+props.children for building a shared "Modal" component that other teams will consume. What does each approach genuinely gain and lose?

template+slot approach gains: zero framework dependency for CONSUMERS (any team, on any stack, can drop in <my-modal>...</my-modal> and it works — the exact cross-framework distribution argument from the previous lesson's Q5); genuine style isolation via Shadow DOM with no CSS-in-JS/CSS-Modules tooling required; and long-term platform stability (a Custom Element defined today keeps working indefinitely, independent of any framework's major-version migrations). It genuinely loses: authoring ergonomics (JSX's inline conditional rendering, component composition, and TypeScript prop-typing integration are considerably more expressive and mature than manually wiring attributeChangedCallback/slots by hand — though Lit and similar Web-Component-authoring libraries meaningfully close this gap while still compiling down to real Custom Elements); and DEEP reactive integration — a modal whose open/closed state needs to participate in a larger app's centralized state management (Redux, a signal-based store) requires more manual bridging code than a React component that natively reads from the same store via hooks. React's approach gains: best-in-class DX for teams already fully invested in React, first-class TypeScript prop typing, and seamless integration with the rest of a React app's state and rendering model. It genuinely loses: zero portability to non-React consumers without a separate wrapper/port, and the modal's actual DOM structure/styling has no PLATFORM-level isolation guarantee (relying instead on CSS Modules/CSS-in-JS discipline for style scoping, exactly the discipline-based workaround the previous lesson noted Shadow DOM structurally obsoletes). The concrete decision criterion, matching the previous lesson's Q5: if "other teams" means other TEAMS ON DIFFERENT FRAMEWORKS or an unknown future framework, template+slot+Custom Elements is the stronger long-term choice; if "other teams" means other React teams within the same overall stack, React's native component model wins on pure authoring ergonomics with no portability cost actually being paid.

1 (easy): Your first template clone. Build a template containing a simple <li> with a placeholder <span>. Write a loop that clones it 5 times with different text content (via textContent) and appends each clone to a <ul>. Confirm the final rendered list in the Elements panel shows 5 genuinely separate li elements, and that the <template> itself remains empty/invisible in the actual page.
Solution 
<template id="item-tpl"><li><span></span></li></template>
<ul id="list"></ul>
<script>
["Apples", "Bread", "Coffee", "Dates", "Eggs"].forEach(item => {
  const clone = document.getElementById("item-tpl").content.cloneNode(true);
  clone.querySelector("span").textContent = item;
  document.getElementById("list").appendChild(clone);
});
</script>

Confirm the <template> tag itself never appears as rendered content anywhere on the page (inspect it in Elements — it has no visible box at all), while the <ul> genuinely accumulates 5 separate, independent li elements — direct confirmation that cloning creates real, independent DOM nodes, not references back to the original template.

2 (easy): The XSS trap, template edition. Repeat exercise 1, but this time use innerHTML instead of textContent to fill the clone, and set one item's "text" to `<img src=x onerror=alert(1)>`. Confirm the alert fires. Then fix it back to textContent and confirm the payload renders as harmless, literal text instead.
Expected contrast

innerHTML version: the alert fires — direct proof that "it came from a template" provides zero inherent safety, exactly matching this lesson's Common Mistakes warning. textContent version: the literal string `<img src=x onerror=alert(1)>` appears as visible TEXT on the page (angle brackets and all) rather than being parsed as an element at all — textContent never interprets its input as markup, which is precisely why it's safe by construction for this use case.

3 (medium): Named slots, built from scratch. Build a <user-card> Custom Element (no template needed for this one — direct shadow.innerHTML is fine) with two named slots (avatar, name) and one default slot (bio), each with fallback content. Use it three times: once supplying all three, once supplying only name, once supplying nothing at all. Confirm each instance renders correctly, including the appropriate fallbacks.
Solution sketch 
class UserCard extends HTMLElement {
  connectedCallback() {
    const shadow = this.attachShadow({ mode: "open" });
    shadow.innerHTML = `
      <div class="card">
        <slot name="avatar">👤</slot>
        <strong><slot name="name">Anonymous</slot></strong>
        <p><slot>No bio provided.</slot></p>
      </div>`;
  }
}
customElements.define("user-card", UserCard);

The "supplying nothing at all" instance should show ALL THREE fallback values simultaneously (👤, "Anonymous", "No bio provided.") — confirming each slot's fallback is genuinely independent and doesn't require the others to also be empty/filled together.

4 (medium): Batched cloning, measured. Build a page that clones a moderately complex template (a card with 4-5 child elements) 300 times. First implementation: append each clone directly to the live DOM inside the loop. Record in DevTools' Performance panel. Second implementation: accumulate all clones into one DocumentFragment first, append once at the end. Compare the number/duration of layout passes between the two recordings.
Expected contrast 
// version B — batched:
const fragment = document.createDocumentFragment();
items.forEach(item => {
  const clone = template.content.cloneNode(true);
  // ...fill clone...
  fragment.appendChild(clone);

container.appendChild(fragment);   // ONE live-DOM insertion for all 300 clones

Version A (per-iteration appendChild) should show many small, repeated layout bars in the Performance panel — one per insertion. Version B should show dramatically fewer, typically consolidating into one (or very few) larger layout pass at the single final append — direct, measured confirmation of Q4's interview answer, applied to your own recording rather than taken on faith.

5 (hard): The complete ProductCard, built and reused. Build Part 4's complete example exactly (template + slots + Custom Element combined). Use it to render a grid of 6 different coffee products with real, varied content (image, title, price, description) for each. Confirm each card renders its own independent content correctly and that changing one card's slotted content has zero effect on any other instance.
What "complete" verification looks like

All 6 <product-card> instances should render with fully independent shadow roots (verify in the Elements panel — each instance has its own separate #shadow-root, not a shared one) and independent projected content. Try deliberately editing one instance's slotted content live in DevTools and confirm no other card instance is affected — this independence is a direct, necessary consequence of each Custom Element instance getting its own attachShadow() call in connectedCallback, which fires separately per instance.

6 (hard): Empty-slot detection, implemented. Extend exercise 3's user-card component: use the slotchange event (or check assignedNodes() directly in connectedCallback) to detect when the avatar slot specifically has no projected content, and apply a CSS class that hides its container entirely in that case — implementing Q5's interview answer as real, working code rather than just describing it.
Reference implementation 
connectedCallback() {
  const shadow = this.attachShadow({ mode: "open" });
  shadow.innerHTML = `
    <style>.avatar-slot--empty { display: none; }</style>
    <div class="avatar-wrap"><slot name="avatar"></slot></div>
    ...`;
  const avatarSlot = shadow.querySelector('slot[name="avatar"]');
  const wrap = shadow.querySelector('.avatar-wrap');
  const checkEmpty = () =>
    wrap.classList.toggle('avatar-slot--empty', avatarSlot.assignedNodes().length === 0);
  avatarSlot.addEventListener('slotchange', checkEmpty);
  checkEmpty();   // run once immediately too — slotchange only fires on CHANGES, not initial state
}

The `checkEmpty()` call OUTSIDE the event listener (running once immediately) is the detail most first attempts miss — slotchange fires when the assigned nodes CHANGE, but the initial state (whether the slot starts out empty or filled) needs its own explicit check, since there's no "change" to fire an event for on first connection.

🎯 Key Takeaways

  • template content is parsed but genuinely inert — never rendered, never executed, absent from the render tree — until explicitly cloned via template.content.cloneNode() and inserted into the live document.
  • Fill cloned template placeholders with textContent, not innerHTML, when the data is dynamic/untrusted — the template's own inertness provides zero protection at the fill-in step.
  • slot elements are the deliberate, explicit holes in Shadow DOM's isolation boundary — named slots project matching light-DOM children into specific shadow-root positions, with fallback content when nothing is supplied.
  • Batch multiple clones into one DocumentFragment before a single live-DOM append, rather than appending inside a loop — directly reduces layout/paint passes per the Critical Rendering Path lesson's principles.
  • template + slot + Custom Elements together form the complete, production-capable Web Components pattern: reusable structure, real style isolation, and content composition, with zero framework runtime required.
  • Every earlier lesson's HTML-correctness rules (alt text, semantics, accessibility) still apply unchanged inside this pattern — Web Components change packaging and encapsulation, never what makes markup correct.
