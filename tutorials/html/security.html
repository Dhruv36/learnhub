Security: XSS, CSP, Sanitization Production

What you'll master: XSS as ONE unified vulnerability class (not a grab-bag of unrelated tricks — you've met roughly a dozen instances of it already across this track without the formal name), the three XSS variants and how they differ mechanically, Content Security Policy as a genuine second line of defense, sanitization done correctly (allow-list, not block-list), and a complete review of every XSS-shaped warning from lessons 1–30, now assembled into one coherent threat model. Prerequisites: the whole track — this lesson deliberately collects and formalizes security threads from Text/Headings, Images, Forms, Global Attributes, iframes, SVG/Canvas, and Web Components.

Part 1 — XSS: One Vulnerability, Recognized Everywhere

Cross-Site Scripting (XSS) is fundamentally simple, once named: an attacker gets THEIR JavaScript to execute in the context of YOUR page — with YOUR user's cookies, YOUR user's session, YOUR page's full access to their account. Every single "escape this," "sanitize that," "never use innerHTML with untrusted data" warning scattered across this entire track has been THIS ONE vulnerability, encountered in a dozen different disguises:

LessonThe specific XSS-shaped warning
Text & Headings< and & must be escaped as entities, even inside <pre> — pre changes whitespace rendering, never disables parsing
Inline SemanticsHighlighting search matches (<mark>) into user-derived content: escape FIRST, then wrap — wrapping before escaping lets attacker input become live HTML
Images MasteryUser-uploaded SVG is executable XML — never served from your main domain/origin

This is deliberately the lesson's opening move: recognizing that you have ALREADY internalized the pattern many times over — this lesson exists to give it ONE formal name, a precise mechanical model, and the two remaining, genuinely new defenses (CSP, and sanitization done rigorously) that tie it all together.

Part 2 — The Three Variants, Mechanically Distinguished

1. STORED XSS      Attacker's payload is SAVED (a database, a file) and served to OTHER users later.
                    Example: a malicious comment, saved, rendered to every future visitor of that page.
                    SEVERITY: highest — one injection point, potentially thousands of victims over time.

2. REFLECTED XSS   Attacker's payload travels in the REQUEST itself (a URL parameter, a form field)
                    and is immediately reflected back into the response, unescaped, in the SAME request.
                    Example: a search page echoing ?q=<script>... directly into "Results for: [q]".
                    Requires tricking a victim into clicking a malicious, crafted link.

3. DOM-BASED XSS   The vulnerability lives ENTIRELY in client-side JavaScript — no malicious data ever
                    touches the server at all. Example: JS reading location.hash and inserting it into
                    innerHTML directly. The server's response is completely clean; the browser's OWN
                    script creates the vulnerability, reading from a source it shouldn't trust.

The mechanical distinction matters because the FIX differs by variant: stored XSS is defended at the point of RENDERING saved content (sanitize on output, ideally also on input as defense in depth — the exact contenteditable rule from the Global Attributes lesson); reflected XSS is defended at the point of ECHOING request data back into a response (escape any user-supplied value before interpolating it into HTML, on the server); DOM-based XSS is defended entirely in CLIENT-SIDE JavaScript discipline — never trust location.hash, document.referrer, window.name, or URL parameters as safe to insert via innerHTML, regardless of what the server did or didn't do, because the server never even saw this data.

Part 3 — Content Security Policy: a Genuine Second Line of Defense

Everything so far has been about NOT LETTING malicious script get into your page in the first place (escaping, sanitizing). CSP is fundamentally different: it's a SECOND, independent layer that limits the DAMAGE even if a script somehow gets through — a real defense-in-depth mechanism, not a substitute for correct escaping.

Content-Security-Policy: default-src 'self';
                          script-src 'self' https://cdn.trusted-analytics.example;
                          style-src 'self' 'unsafe-inline';
                          img-src 'self' https: data:;
                          frame-ancestors 'self';

An HTTP response header (you met frame-ancestors already, in the iframes lesson, as the clickjacking defense — CSP is the same header system, with many other directives) declaring an explicit ALLOW-LIST of where scripts, styles, images, and frames are permitted to load from. script-src 'self' means: only scripts from your OWN origin may execute — an attacker who successfully injects <script src="https://evil.example/steal.js"> via some XSS hole finds the browser ITSELF refuses to load or execute it, because evil.example isn't on the allow-list, REGARDLESS of the fact that the malicious tag made it into your HTML. This is the precise meaning of "defense in depth" — your escaping/sanitization is meant to prevent the injection entirely (layer one), and CSP limits the blast radius on the rare occasion something still gets through anyway (layer two) — exactly the two-layer client+server validation model the Forms track lessons established for a completely different concern (data correctness vs. security), the same architectural pattern recurring here.

The notable, deliberate absence from that policy: 'unsafe-inline' is NOT included in script-src — meaning inline <script> tags and onclick="..."-style inline event handlers are BLOCKED by this policy entirely, even ones YOU legitimately wrote (a strict, but genuinely much safer, real-world posture — a growing majority of security-conscious production sites enforce exactly this). Sites needing legitimate inline scripts use a nonce (a random, per-request token added to both the CSP header and the specific script tag it authorizes: script-src 'nonce-r4nd0m' + <script nonce="r4nd0m">) rather than blanket-allowing all inline scripts, which would defeat much of CSP's protective value.

Part 4 — Sanitization: Allow-List, Never Block-List

This is the single most important operational principle in this lesson, and it directly explains why every sanitizer recommendation across this track has named a SPECIFIC library (DOMPurify) rather than "just write a regex to strip script tags":

❌ BLOCK-LIST (naive, always eventually broken):
   "Strip anything containing <script>" — misses <img src=x onerror=alert(1)>,
   <svg onload=alert(1)>, javascript: URLs, and dozens of other execution vectors that were
   never on the list because the list can never be exhaustive against a creative attacker.

✅ ALLOW-LIST (what DOMPurify and every serious sanitizer actually does):
   "Permit ONLY these specific tags (b, i, p, a...) and ONLY these specific attributes
   (href, but only http/https schemes — never javascript:) — reject/strip EVERYTHING else,
   by default, with no exceptions needing to be individually anticipated."

The reasoning: a block-list requires you to anticipate and enumerate EVERY possible attack vector in advance — a fundamentally losing, permanently incomplete game against attackers who only need to find ONE vector you didn't think of. An allow-list requires you to specify only what's LEGITIMATELY needed (a genuinely small, stable, enumerable set — bold text, links, paragraphs) and everything else is rejected by construction, with no ongoing arms race required. This is precisely why sandbox (iframes lesson) works the identical way — start from nothing, opt in individually — and why this same allow-list PHILOSOPHY has now appeared at three completely different layers across this track (iframe permissions, CSP's script-src allow-list, and HTML sanitization) — it's a general security principle, not a coincidence of API design.

<script src="https://cdn.jsdelivr.net/npm/dompurify@3/dist/purify.min.js"></script>
<script>
const dirty = userComment;   // untrusted, from anywhere: a form, a URL param, an API response
const clean = DOMPurify.sanitize(dirty, {
  ALLOWED_TAGS: ["b", "i", "em", "strong", "a", "p"],
  ALLOWED_ATTR: ["href"],
});
commentEl.innerHTML = clean;   // now genuinely safe to insert
</script>

Part 5 — Worked Example: a Comment System, Fully Defended

Assembling every layer from this lesson into one realistic, complete flow — this is the actual shape of a genuinely production-safe comment/UGC feature:

1. SERVER: on comment SUBMISSION, sanitize before storing (defense layer 1, per Global
   Attributes lesson's "sanitize on save" rule) — never trust that render-time sanitization
   alone is sufficient; a future code path might render this data without going through your
   sanitizer.

2. SERVER: on comment RENDERING, sanitize AGAIN before sending to the browser (defense layer 2 —
   redundant with step 1 by design; a sanitizer bug, library upgrade, or a second insertion path
   bypassing step 1 doesn't leave you fully exposed).

3. HTTP RESPONSE HEADER: Content-Security-Policy: script-src 'self' (defense layer 3) — even if
   an attacker somehow found a gap in steps 1-2, the browser itself refuses to execute any
   injected inline/external script that isn't from your own origin.

4. CLIENT: if comments are ALSO rendered/updated client-side (e.g., a live-updating comment
   feed via WebSocket), run DOMPurify.sanitize() there too (defense layer 4) — the exact
   same discipline the server applies, now also applied to the DOM-based-XSS-prone client
   rendering path from Part 2.

Four independent layers, each catching what the others might miss — this is what "defense in depth" concretely means in practice, not an abstract security-theory phrase. No single layer is assumed sufficient on its own.

Part 6 — Deep Dive: Why This Track Kept Returning to This Topic

A closing, honest observation: this lesson didn't teach you a new SKILL so much as give a name to a DISCIPLINE you've been practicing since the Text & Headings lesson's entity-escaping rule. The reason security warnings appeared so persistently across this entire track — rather than being confined to one chapter — is that XSS isn't a feature you can add; it's a property that emerges from how CONSISTENTLY every single point where untrusted data meets rendered HTML is handled. A team can get 49 of 50 injection points correct and remain fully exploitable through the 1 they missed — which is exactly why this track's approach (repeating the same escape/sanitize discipline at every relevant lesson, rather than teaching it once and assuming transfer) mirrors how security actually has to be practiced in a real codebase: as a habit applied EVERYWHERE untrusted data flows toward the DOM, not as a single feature implemented once and considered "done."

Interview Questions

Q1 (beginner): What is XSS, in one precise sentence?

An attacker gets their own JavaScript to execute within the context of your page, gaining whatever access your legitimate page's script would have — the victim's cookies, session, and full logged-in access to their own account on your site. Every "escape this" / "sanitize that" warning across HTML is ultimately about preventing exactly this one outcome.

Q2 (beginner): What's the fundamental difference between stored and reflected XSS?

Stored XSS: the malicious payload is saved (database, file) and served to OTHER users later — one injection point can compromise many victims over time, and it's the most severe variant for that reason. Reflected XSS: the payload travels in the request itself (a URL parameter, form field) and is echoed back unescaped in the SAME response — it requires tricking a specific victim into clicking a crafted malicious link, affecting one victim per exploit attempt rather than persisting for future visitors.

Q3 (mid): Why is an allow-list sanitization approach fundamentally more sound than a block-list approach, even a very thorough one?

A block-list requires exhaustively anticipating every possible attack vector in advance — an inherently losing, permanently incomplete strategy, since an attacker only needs to find ONE vector the list-writer didn't think of (onerror handlers, SVG event attributes, javascript: URL schemes, and dozens of other real historical vectors). An allow-list instead specifies only what's legitimately NEEDED (a small, stable, fully-enumerable set of safe tags/attributes) and rejects everything else by construction — no ongoing arms race, no need to anticipate future attack techniques individually.

Q4 (mid): Explain what Content-Security-Policy's script-src 'self' actually prevents, and why it's called "defense in depth" rather than a primary defense.

It instructs the browser to refuse to load/execute any script NOT originating from your own origin — so even if an attacker successfully injects a <script src="evil.example/x.js"> tag through some XSS hole your escaping/sanitization missed, the browser itself blocks that script from actually running. It's called defense in depth (a SECOND, independent layer) rather than a primary defense because it does nothing to prevent the malicious tag from being injected into your HTML in the first place — that's still the job of correct escaping/sanitization; CSP only limits the DAMAGE on the rare occasion an injection still gets through despite that primary layer.

Q5 (senior): Your team's comment system sanitizes on save (server-side, at submission time) but a code review reveals a NEW feature — a live-updating "recent comments" widget rendered entirely client-side via WebSocket messages — bypasses this sanitization path entirely. Assess the risk and design the fix.

Genuine, serious risk: the WebSocket-delivered comment data never passes through the original save-time sanitization code path at all (it's presumably reading from the same underlying stored data, but the NEW client-side rendering code is a separate insertion point that was never audited against the original defense) — if this new code renders comment text via innerHTML directly, it reintroduces a full stored-XSS vulnerability specifically through this one new feature, even though the "main" comment-rendering path remains correctly defended. This is a textbook instance of Part 6's closing point: 49 correctly-defended insertion points don't protect against a 50th that was missed. Fix: (1) immediate — run DOMPurify.sanitize() on the client, on every WebSocket-delivered comment, before any innerHTML insertion — the SAME allow-list configuration used server-side, ideally sharing the actual configuration/logic between server and client where the stack allows it, to avoid the two sanitization points silently drifting apart over time; (2) systemic — this incident argues for a project-wide policy/lint rule flagging any raw innerHTML usage anywhere in the codebase for mandatory security review before merge, specifically because new insertion points (like this WebSocket feature) will keep appearing as the codebase grows, and relying on developers to remember the original save-time defense applies "everywhere" is exactly the assumption that just failed here.

Q6 (senior): Design the complete, layered security architecture for a rich-text blog-post editor (contenteditable-based, per the Global Attributes lesson) where posts are publicly rendered to thousands of readers. Walk through every defense layer and what specifically could fail at each.

Layer 1 — INPUT sanitization: the moment contenteditable content is saved (author clicks "Publish"), run it through an allow-list sanitizer server-side before persisting — what could fail: a sanitizer version with a known bypass (a real, recurring category of vulnerability in sanitizer libraries historically — argues for keeping the dependency current and monitoring security advisories), or an overly permissive ALLOWED_TAGS/ALLOWED_ATTR configuration someone loosened to "fix a formatting bug" without appreciating the security implication. Layer 2 — OUTPUT sanitization: sanitize AGAIN at render time, independent of layer 1 — what could fail: skipped if a developer assumes "it's already sanitized in the database" and adds a new rendering code path (a new "embed this post in an RSS feed" feature, say) that reads the stored content and renders it WITHOUT going through the same output sanitizer, exactly Q5's failure mode recurring. Layer 3 — CSP: script-src 'self' (or stricter, with nonces for any genuinely needed inline scripts) on every page rendering post content — what could fail: a CSP configured with 'unsafe-inline' as an expedient fix for some OTHER inline-script need, silently reopening the exact hole this layer exists to close for the blog content specifically. Layer 4 — periodic AUDIT: because rich-text editors have historically been a disproportionate source of real-world XSS incidents (the format inherently permits a wide surface of markup/attributes, unlike simple plain-text comment fields), schedule recurring security review of the actual allow-list configuration and any new rendering code paths, rather than treating this as a "build once" feature — directly reflecting Part 6's thesis that XSS-safety is an ongoing discipline, not a completed task. The senior framing across all four layers: name SPECIFICALLY how each layer could independently fail, because assuming any single layer is bulletproof is itself the actual vulnerability pattern this whole lesson has been building toward recognizing.

2 (easy): Reflected XSS, built and fixed. Build a tiny local "search" page that reads a query parameter (via URLSearchParams) and displays "Results for: [query]" using innerHTML with NO escaping. Visit it with a URL containing ?q=<img src=x onerror=alert(1)> and confirm the alert fires. Then fix it using textContent (or proper escaping) and confirm the payload renders as harmless literal text.
Reference + fix 
// vulnerable:
const q = new URLSearchParams(location.search).get("q");
resultsHeading.innerHTML = `Results for: ${q}`;

// fixed:
resultsHeading.textContent = `Results for: ${q}`;

This is a genuine, self-contained, hands-on reproduction of reflected XSS's exact mechanism — the payload traveled entirely in the URL, was echoed back unescaped in the same page load, with no server or database involved at all, matching Part 2's definition precisely.

3 (medium): Allow-list sanitizer, configured and tested. Using DOMPurify (CDN script tag is fine for this exercise), sanitize a string containing a mix of legitimate formatting (<b>, <a href="https://example.com">) AND a malicious payload (<img src=x onerror=alert(1)>, a javascript: URL in an <a> tag). Configure ALLOWED_TAGS/ ALLOWED_ATTR narrowly. Confirm the legitimate formatting survives sanitization while both malicious pieces are stripped or neutralized.
Solution + expected result 
const dirty = `<b>Great product!</b> <a href="https://example.com">link</a>
  <img src=x onerror=alert(1)> <a href="javascript:alert(2)">click</a>`;
const clean = DOMPurify.sanitize(dirty, {
  ALLOWED_TAGS: ["b", "a"],
  ALLOWED_ATTR: ["href"],
});
console.log(clean);

Expected: the <b> tag and the legitimate https:// link survive intact; the <img> tag is stripped ENTIRELY (it's not in ALLOWED_TAGS at all); the javascript: URL link is neutralized specifically because DOMPurify's allow-list logic scheme-checks href values even when "a" and "href" are both allowed — a genuinely important detail proving allow-listing operates at both the tag level AND the attribute-VALUE level, not just "is this tag/attribute name permitted".

4 (medium): CSP, configured and observed blocking an injection. If you can configure response headers for a local test server (or use a <meta http-equiv="Content-Security-Policy"> tag as a page-level substitute for this exercise), set script-src 'self', then deliberately inject an inline <script>alert(1)</script> via innerHTML on the page. Confirm the Console shows a CSP violation error and the alert does NOT fire — even though the script tag genuinely made it into the DOM.
What to observe 
<meta http-equiv="Content-Security-Policy" content="script-src 'self'">
<script>
document.body.innerHTML += "<script>alert('blocked?')</script>";
</script>

The Console should show an explicit CSP violation message naming the blocked directive, and the alert should NOT fire — direct, observed confirmation of Part 3's core claim: the malicious script tag genuinely exists in the DOM (inspect it in Elements — it's there), but the BROWSER itself refuses to execute it because of the policy, independent of any application-level escaping. This is what "defense in depth, a second independent layer" concretely looks like when you watch it happen.

5 (hard): The four-layer comment system, built. Implement Part 5's complete worked example as real, working code: a simple in-memory "comment store" (an array is fine), a submit function that sanitizes before storing (layer 1), a render function that sanitizes AGAIN before displaying (layer 2), and a CSP meta tag (layer 3). Test by submitting a comment containing a malicious payload and confirming it's neutralized, then INSPECT your stored data directly to confirm layer 1 actually sanitized before storage (not just at render time).
What "verified" means here

The key verification step most people skip: checking the STORED data (console.log your comment array/store directly) to confirm the malicious payload was ALREADY neutralized at storage time, not merely hidden by render-time sanitization while remaining dangerous in storage. If you find the raw stored data still contains an unescaped <script> or onerror attribute despite your render function "working," that's a real, meaningful gap — layer 2 was compensating for a broken layer 1, exactly the single-point-of-failure risk Part 5 and the Common Mistakes table both warn against, now caught in your own implementation rather than only described abstractly.

6 (hard): DOM-based XSS audit. Search through any JavaScript you've written across this ENTIRE track (or any real open-source project's client-side code) for instances of innerHTML, outerHTML, or document.write being fed ANY value ultimately derived from location.hash, location.search, document.referrer, or window.name — even indirectly through several variable assignments. Report every instance found, and for each, state whether it's genuinely exploitable or a false positive (e.g., the value passes through a sanitizer or gets used only via textContent downstream).
What a real audit distinguishes

Genuinely exploitable: a direct, unescaped path from a URL-derived value to innerHTML with no sanitization step anywhere in between. False positive (common, and important to correctly identify — over-flagging erodes trust in your own audit findings): the value passes through textContent instead of innerHTML at the final insertion point (safe by construction, per Part 4 and Templates lesson's rule), or genuinely goes through DOMPurify.sanitize() first. If your audit of this track's OWN example code finds zero genuinely exploitable instances, that's actually the expected, correct result — every code sample across all 31 preceding lessons was deliberately written using textContent/proper escaping specifically because this exact audit exercise was anticipated; finding that discipline consistently applied, by your own independent check, is itself meaningful confirmation of the pattern this whole lesson has been building.

🎯 Key Takeaways

  • XSS is ONE unified vulnerability — attacker script executing with your page's full access — recognized across a dozen disguises throughout this track: uploaded SVG, contenteditable, textarea, template innerHTML, and more.
  • Three variants, three distinct fixes: stored (sanitize on save AND render), reflected (escape request data before echoing it back), DOM-based (never trust location.hash/referrer/window.name even when the server response is clean).
  • CSP (script-src, an HTTP header) is a genuine second, independent defense layer — it limits damage from injected scripts, it does NOT replace correct escaping/sanitization as the primary defense.
  • Allow-list sanitization (DOMPurify: permit only specific known-safe tags/attributes/schemes) is fundamentally sound where block-lists are permanently, structurally incomplete — the same allow-list philosophy recurs across iframe sandboxing, CSP, and HTML sanitization.
  • Defense in depth means multiple independent layers (input sanitization, output sanitization, CSP, client-side re-sanitization) — no single layer is assumed sufficient, because any one layer alone has a plausible, real failure mode.
  • XSS-safety is a consistently-applied discipline across EVERY untrusted-data-to-DOM boundary in a codebase, not a single feature implemented once — 49 correct insertion points don't protect against a 50th that was missed, a real, recurring incident pattern.
