Performance Advanced

Speed is a business metric: Amazon-scale studies attribute ~1% revenue per 100ms of latency; Google ranks on Core Web Vitals; half your mobile users abandon pages slower than ~3s. Most of a page's loading behavior is decided by HTML attributes — before any JS optimization exists.

Core Web Vitals — the Scoreboard

MetricMeasuresGood (p75)Your main HTML levers
LCP (Largest Contentful Paint)When the main content appears< 2.5spreload the hero image, fetchpriority="high", never lazy-load it, small render-blocking CSS
CLS (Cumulative Layout Shift)How much the page jumps around< 0.1width/height on images/video/iframes, reserved slots for ads/embeds, font-display strategy
INP (Interaction to Next Paint)Responsiveness to clicks/keys< 200msship less JS, defer everything non-critical, break long tasks

Lab vs field: Lighthouse simulates (lab); Google ranks on real-user Chrome data (field, via CrUX, at the 75th percentile). A fast dev machine proves nothing — check field data.

The Critical Rendering Path (what blocks first paint)

HTML ──parse──▶ DOM ─┐
CSS  ──parse──▶ CSSOM ─┴─▶ Render Tree ─▶ Layout ─▶ Paint
        ▲ CSS blocks RENDERING (not parsing)
        ▲ sync <script> blocks PARSING (and waits for CSSOM if it might read styles!)

Consequences: keep head CSS lean (inline the critical part on very perf-sensitive sites); load JS as modules/defer; split off non-critical CSS (media="print", or preload-then-apply patterns).

Resource Hints — Scheduling the Network

<link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="/img/hero.avif" as="image">           <!-- the LCP image -->
<link rel="preconnect" href="https://api.myapp.com">             <!-- DNS+TCP+TLS early -->
<link rel="dns-prefetch" href="https://cdn.thirdparty.com">      <!-- cheaper, DNS only -->
<link rel="prefetch" href="/next-page.html">                     <!-- idle-time, for NEXT nav -->
<script type="speculationrules">                                  <!-- modern prerendering -->
  { "prerender": [{ "where": { "href_matches": "/products/*" }, "eagerness": "moderate" }] }
</script>
HintMeaningTrap
preload"Needed for THIS page, fetch now at high priority"Preloading too much starves the critical path — budget 2–3; unused preloads log console warnings
preconnectOpen the connection to an origin you'll definitely hitEach one costs a socket — only 2–3 critical origins
prefetch / speculation rulesLikely NEXT navigation, lowest priority / full prerenderDon't prerender logout links or heavy pages on data-saver connections

Why fonts and LCP images specifically: both are discovered late by default — fonts hide inside CSS, background images behind CSS, and srcset decisions happen after layout. Preload jumps them to the front of the line.

Lazy Loading — Power Tool, One Famous Trap

<img src="/img/comments-avatar.jpg" loading="lazy" …>    ✅ below the fold
<iframe src="…/map-embed" loading="lazy"></iframe>        ✅ heavy embeds
<img src="/img/hero.avif" fetchpriority="high" …>         ✅ the LCP image — eager, boosted
<img src="/img/hero.avif" loading="lazy">                 ❌ THE classic self-inflicted LCP wound

Fonts Without the Flash

@font-face {
  font-family: Inter;
  src: url(/fonts/inter-var.woff2) format("woff2");   /* woff2 only — 30% smaller than woff */
  font-display: swap;      /* show fallback text instantly, swap when loaded */
}
/* Advanced: size-adjust/ascent-override on the fallback font to make the swap shift-free (CLS) */

Strategy stack: system font (zero cost) → one variable font, woff2, preloaded, swap → subset to your languages (cuts 60–90% of glyph weight).

Compression & Caching (the server half you must know)

Content-Encoding: br                      Brotli: ~15–20% smaller than gzip for text
Cache-Control: max-age=31536000, immutable   for fingerprinted assets (app.3f9c2.js)
Cache-Control: no-cache                      for the HTML itself (revalidate each time)
HTTP/2+ multiplexing                          many small files ≈ fine; inlining less needed

Interview Questions

Q1: Walk me through diagnosing a slow LCP.

Identify the LCP element (Lighthouse names it) → is it discovered late (CSS background / lazy-loaded / behind JS)? → move to an <img> in HTML, preload it, fetchpriority=high → shrink it (AVIF, srcset) → check render-blocking CSS and server TTFB. Order matters: discovery, priority, size, blocking, backend.

Q2: preload vs prefetch vs preconnect?

preload: this page, this resource, now (high priority). prefetch: probably the next page, when idle (lowest priority). preconnect: no specific resource — just warm the connection to an origin. Misusing preload as prefetch (or vice versa) either starves the critical path or downloads too late.

Q3: What causes CLS besides images, and the fixes?

Ad/embed slots (reserve fixed dimensions), web fonts swapping metrics (font-display + size-adjust fallback tuning), content injected above existing content (never — append below or overlay), animations of layout properties (use transform). Every fix is "tell the browser the size in advance".

🎯 Key Takeaways

  • LCP / CLS / INP are the scoreboard; field data at p75 is what counts.
  • CSS blocks rendering, sync JS blocks parsing — defer/module everything, keep head CSS lean.
  • Preload late-discovered critical assets (fonts, hero); prefetch/prerender likely next pages.
  • lazy-load below the fold only; the hero gets fetchpriority="high".
  • Fonts: woff2 + swap + preload + subset. Assets: fingerprint + immutable cache + Brotli.
