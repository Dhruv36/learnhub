Forms & Validation Intermediate

Forms are where money changes hands — signups, checkouts, bookings. They're also the most under-learned part of HTML: modern forms validate, format, autofill, and adapt keyboards before a single line of JavaScript runs. Checkout abandonment studies put form friction among the top reasons users quit — this is revenue engineering.

Input Types — Each One Is Free UX

<input type="email">     format check + @-keyboard on phones
<input type="tel">       numeric phone keyboard (no format validation — formats vary)
<input type="url">       URL check + url keyboard
<input type="number" min="1" max="99" step="1">   spinners + range check
<input type="range">     slider          <input type="date">  native date picker
<input type="password">  masked          <input type="search"> clear-button styling
<input type="checkbox">  many-of-many    <input type="radio">  one-of-many (same name!)
<input type="file" accept="image/*" multiple>     <input type="color">
<input type="hidden" name="csrf" value="…">       carried but invisible

Wrong type = wrong mobile keyboard = measurable conversion loss. type="number" trap: it's for quantities — never for credit cards, ZIP codes, or IDs (leading zeros vanish, scroll-wheel changes values). Use inputmode="numeric" pattern="[0-9]*" for digit codes.

Group related controls with <fieldset> + <legend> — essential for radio groups, where the question ("Plan") lives in the legend and each option has its own label.

Built-in Validation — the Full Arsenal

required                          non-empty / checked / selected
minlength="12" maxlength="64"     string length
min="2026-01-01" max="2026-12-31" numeric/date bounds (with step)
pattern="[A-Za-z0-9-]{4,}"        regex (anchored automatically; add title="…" to explain)
type="email|url"                  format checks
/* Style the states — :user-invalid waits until the user has interacted (kinder than :invalid): */
input:user-invalid { border-color: #cf222e; }
input:user-valid   { border-color: #1a7f37; }

Customize or extend via the Constraint Validation API:

pw2.addEventListener("input", () => {
  pw2.setCustomValidity(pw2.value === pw.value ? "" : "Passwords don't match");
});
form.checkValidity();       // boolean — all constraints pass?
form.reportValidity();      // …and show the native bubbles
<form novalidate>           // opt out of native UI to fully custom-render errors (keep the API!)

Autocomplete — the Attribute Worth Real Money

<input name="email"    autocomplete="email">
<input name="pw"       autocomplete="new-password">    ⟵ password managers GENERATE here
<input name="pw"       autocomplete="current-password"> ⟵ …and FILL here
<input name="name"     autocomplete="name">
<input name="cc"       autocomplete="cc-number">
<input name="otp"      autocomplete="one-time-code">    ⟵ SMS codes autofill on mobile

Correct tokens let browsers/password managers fill entire checkout forms in one tap. E-commerce teams A/B-test this; it wins every time. Disabling autocomplete on login fields to "improve security" does the opposite — it fights password managers, the best security tool users have.

Interview Questions

Q1: Why must the server re-validate if HTML validation passed?

Client validation runs in an environment the attacker controls — curl, DevTools, or a modified page bypasses it entirely. HTML validation is UX (fast feedback); server validation is security (the trust boundary). Always both.

Q2: GET vs POST for forms — full comparison?

GET: parameters in URL, idempotent, bookmarkable, cached, length-limited — searches/filters. POST: body payload, state-changing, not cached — everything else. Security note: GET data leaks via history/logs/Referer. Semantics note: browsers warn on re-POSTing (the "resubmit form?" dialog) — the Post/Redirect/Get pattern avoids double-submits.

Q3: Build an accessible radio group.

fieldset around the group, legend carrying the question, each input paired with its own label (for/id or wrapping), all sharing one name. Arrow keys move within the group natively — a div-based rebuild has to reimplement all of that.

Q4: Why did my form submit when I clicked "Show password"?

Buttons inside forms default to type="submit". Any helper button needs explicit type="button". One of the most common real-world form bugs.

🏋️ Practice:
1. Build the full signup form: email, password (minlength 12, new-password), role select, plan radio group in a fieldset, terms checkbox — all labeled, all validating natively.
2. Add the password-confirmation field wired through setCustomValidity.
3. Submit with method="get" once to SEE the data in the URL, then switch to post and explain (to yourself) each thing that changed.

🎯 Key Takeaways

  • name attributes decide what the server receives; method decides how (get=read, post=write).
  • Right input type + right autocomplete token = keyboards, autofill, and conversions handled.
  • Every control gets a real label; radio/checkbox groups get fieldset+legend.
  • HTML validation is UX; the Constraint Validation API extends it; the server re-checks everything.
  • Secondary buttons need type="button" or they submit the form.
