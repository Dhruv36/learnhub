Forms & Validation Intermediate

Forms are where users give you data — and where products lose the most users to friction and bugs. Modern HTML validates, formats, and assists before any JavaScript runs.

A Production Signup Form

<form action="/api/signup" method="post">
  <label for="email">Email</label>
  <input id="email" name="email" type="email"
         required autocomplete="email" placeholder="you@company.com">

  <label for="pw">Password</label>
  <input id="pw" name="password" type="password"
         required minlength="12" autocomplete="new-password">

  <label for="role">Role</label>
  <select id="role" name="role">
    <option value="dev">Developer</option>
    <option value="pm">Product Manager</option>
  </select>

  <fieldset>
    <legend>Plan</legend>
    <label><input type="radio" name="plan" value="free" checked> Free</label>
    <label><input type="radio" name="plan" value="pro"> Pro</label>
  </fieldset>

  <label><input type="checkbox" name="tos" required> I accept the terms</label>
  <button type="submit">Create account</button>
</form>

The Rules That Matter

RuleWhy
Every input gets a <label for>Clicking the label focuses the input; screen readers announce it. Placeholder is NOT a label — it vanishes on typing
name on every fieldIt's the key the server receives — no name, no data
Right type: email, tel, url, number, dateFree validation + the right mobile keyboard
autocomplete attributesPassword managers and browser autofill depend on them; forms without them measurably lose conversions
method="post" for mutationsGET puts data in the URL — fine for search, catastrophic for passwords

Built-in Validation

required                 — must be filled
minlength / maxlength    — string bounds        min / max / step — numeric bounds
pattern="[A-Za-z0-9-]+"  — regex check
type="email|url|number"  — format check

The browser blocks submission and shows a native error bubble. Style invalid fields with the :invalid / :user-invalid CSS pseudo-classes.

⚠️ Security rule you'll re-meet in the backend tracks: client-side validation is a UX feature, not a security boundary. Anyone can bypass it with DevTools or curl. The server must re-validate everything. Always both, never either.
