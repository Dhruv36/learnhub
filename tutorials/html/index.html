How the Web & HTML Work Foundations

What you'll master in this lesson: what actually happens between typing a URL and seeing a page, what HTML really is (and isn't), how the browser turns your text file into a living tree of objects, and how to write, save, and inspect your first complete pages. No prerequisites — this is lesson one, and everything else in this track builds on the mental model you form here.

Part 1 — What HTML Actually Is

HTML stands for HyperText Markup Language. Unpack the name and you understand the technology:

  • HyperText — text containing links to other text. This was the revolutionary idea in 1991: documents that reference each other, forming a web. Every <a href> you write continues that idea.
  • Markup — you take plain content and mark it up with tags that declare what each piece is: "this is a heading", "this is a paragraph", "this is a button". Compare a manuscript where an editor circles a line and writes "make this the title" — tags are those editor's notes, written in a form the browser can read.
  • Language — but not a programming language. HTML has no variables, no loops, no if-statements, no functions. It cannot compute anything. It describes structure and meaning. This isn't a limitation to apologize for; it's a design decision that makes HTML fault-tolerant, accessible, and learnable in a weekend.

The web platform is a three-layer cake, and keeping the layers straight will save you from years of muddled code:

LayerLanguageResponsibilityExample
Structure & meaningHTMLWHAT the content is"This is a navigation menu containing 4 links"
PresentationCSSHOW it looks"The menu is horizontal, dark blue, fixed to the top"
BehaviorJavaScriptWHAT it does when interacted with"On small screens, tapping ☰ opens the menu"
💡 The professional habit this table creates: when you're about to solve a problem, ask which layer owns it. Centering text? CSS, not    spaces in HTML. Hiding content until a click? Often HTML (<details>) before JavaScript. Marking a heading? HTML (<h2>), never a styled <div>. Juniors reach for the wrong layer constantly; seniors are defined partly by not doing that.

Part 2 — The Journey of a Page Load, Step by Step

Type https://example.com and press Enter. Here is the full journey — learn it now in outline, because the Performance and Production lessons at the end of this track zoom into almost every step:

1. URL PARSING     The browser splits the URL: scheme (https) + domain (example.com) + path (/)

2. DNS LOOKUP      "What is example.com's IP address?" → DNS answers e.g. 93.184.216.34
                   (DNS = the internet's phone book; cached aggressively at every level)

3. TCP + TLS       Browser opens a connection to that IP on port 443 and negotiates
                   encryption (the padlock). ~1-3 round trips before any content moves.

4. HTTP REQUEST    GET / HTTP/1.1
                   Host: example.com
                   Accept: text/html
                   ...a plain-text (well, HTTP/2 binary-framed) request for the page.

5. HTTP RESPONSE   HTTP/1.1 200 OK
                   Content-Type: text/html; charset=utf-8

                   <!DOCTYPE html><html>...        ← the HTML, as text

6. PARSING         The browser reads the HTML top-to-bottom and builds the DOM tree.
                   When it meets <link rel="stylesheet">, <script>, <img> — it issues
                   MORE requests (steps 2-5 again for each resource).

7. RENDER          DOM + CSS → layout (where does everything go?) → paint (pixels).
                   You see the page. Total elapsed: ideally under 2 seconds.

Two facts from this journey shape everything in this track:

  1. HTML is just text. The server sent characters. You can write a complete website in Notepad, and viewing any page's source (Ctrl+U / Cmd+Option+U) shows you exactly what the server sent. There is no compilation, no binary format, no magic.
  2. The browser builds a tree. Every tag becomes a node with a parent and possibly children. CSS selectors navigate this tree ("every <li> inside nav"), JavaScript manipulates it, screen readers walk it, and search engines index it. The tree — the DOM (Document Object Model) — is the real product of your HTML.

Seeing the tree yourself

Given this HTML:

<body>
  <h1>Recipes</h1>
  <ul>
    <li>Pasta</li>
    <li>Tacos</li>
  </ul>
</body>

the browser builds:

body
├── h1
│   └── "Recipes"        (text is a node too!)
└── ul
    ├── li ── "Pasta"
    └── li ── "Tacos"

Open DevTools (F12) → Elements panel — that collapsible tree you see is the DOM, live. It may differ from your source: the browser fixes your mistakes (auto-closing tags, moving misplaced elements), and JavaScript may have added nodes. View Source shows what the server sent; the Elements panel shows what the browser made of it. Knowing the difference is a real interview question and a daily debugging skill.

Part 3 — Anatomy of a Tag

    opening tag                       closing tag
   ┌────┴─────┐                        ┌───┴───┐
   <a href="/menu.html" class="link">Our menu</a>
      └──────┬───────┘  └────┬─────┘ └───┬──┘
        attribute        attribute    content
       (name="value")
  • Element = opening tag + content + closing tag, all together. People say "tag" loosely; strictly, tags are the bracketed parts and the element is the whole thing.
  • Attributes live in the opening tag only, as name="value" pairs separated by spaces. Quotes: technically optional for simple values, always write them anyway — unquoted values break the moment a space appears, and every style guide requires quotes.
  • Void elements have no content and no closing tag: <br>, <img>, <input>, <meta>, <link>, <hr>. Writing <img /> with a slash is legal and harmless (JSX requires it) but the slash does nothing in HTML.
  • Nesting must be properly contained: close inner elements before outer ones. <p><em>text</em></p> ✅ — <p><em>text</p></em> ❌ (the browser will "fix" this, but its fix may not match your intent, and CSS/JS then behave mysteriously).
  • Case: tag and attribute names are case-insensitive (<DIV> works) but the universal convention is lowercase. Attribute values can be case-sensitive (ids, URLs).

Part 4 — Build Your First Real Page (follow along)

Don't read this section — do it. Create a folder, open any editor (VS Code recommended, Notepad works), and build in four steps.

Step 1: the minimum viable page

<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8">
    <title>Maya's Coffee</title>
  </head>
  <body>
    <h1>Maya's Coffee</h1>
  </body>
</html>

Save as index.html (the filename servers look for by default) and double-click it — your browser opens it from disk. You should see a big black heading. Every line here earns its place, and the next lesson dissects each one; for now: <head> holds information about the page (nothing visible except the tab title), <body> holds everything visible.

Step 2: add real content

<body>
  <h1>Maya's Coffee</h1>
  <p>Small-batch roasting in Portland since 2019.</p>

  <h2>This Week's Beans</h2>
  <ul>
    <li>Ethiopia Yirgacheffe — floral, tea-like</li>
    <li>Colombia Huila — chocolate, caramel</li>
  </ul>

  <h2>Visit Us</h2>
  <p>Open daily 7am–3pm at 420 Oak Street.</p>
</body>

Refresh the browser (F5). Notice what you got for free: heading sizes, paragraph spacing, list bullets, a sensible font. That's the browser's built-in stylesheet responding to your semantic choices. You told it what things mean; it chose a reasonable look.

Step 3: add a link and an image

<p>Order online at <a href="https://example.com/shop">our shop</a>.</p>
<img src="beans.jpg" alt="Freshly roasted coffee beans in a burlap sack" width="400">

The link works immediately — hypertext! For the image, put any JPEG named beans.jpg in the same folder. Now deliberately break it: rename the file and refresh. You get a broken-image icon plus your alt text — your first demonstration that alt text is real functionality (for broken images, slow connections, and screen-reader users), not a formality. Links and images each get a full lesson later; this is the seed.

Step 4: inspect what you built

Right-click the heading → Inspect. In the Elements panel: click nodes and watch the page highlight; press Delete on a node and watch it vanish (refresh restores — you edited the DOM copy, not your file); double-click text and change it live. This edit-and-inspect loop is how professionals debug HTML every day, so wire it into your fingers now.

Part 5 — Deep Dive: Why HTML Is Forgiving (and why that cuts both ways)

Delete a </p> and the page still renders. Misspell a tag (<pargraph>) and the browser shrugs, treats it as an unknown inline element, and renders its contents. HTML parsers never throw errors at the user — the spec defines exact recovery behavior for every malformed input, a decision that let the messy early web survive. Contrast JavaScript, where one syntax error kills the whole script.

The trap: "it renders" ≠ "it's correct." The browser's repairs produce a DOM you didn't intend — a table cell hoisted out of its table, a <div> silently closed early — and then your CSS "randomly" doesn't apply and your JavaScript can't find elements. The professional countermeasures:

  • Validate at validator.w3.org while learning — it catches nesting mistakes your eyes skate over.
  • When CSS/JS misbehaves, check the Elements panel first — is the tree what you think it is?
  • Certain elements are notorious repair magnets: content directly inside <ul> (not in an <li>), block elements inside <p> (the browser closes the <p> early — a classic!), stray </div>s.
🔍 Senior-level nugget: because parsing rules are fully specified, every browser repairs identically — that's why "works in Chrome, broken in Firefox" is almost never a parsing issue nowadays. It's usually CSS or JS API differences. Blame parsing last.

Part 6 — Deep Dive: Where Pages Actually Live

You double-clicked a file (the address bar shows file:///C:/...). Real pages come from a web server — a program that listens for HTTP requests and responds with files or generated HTML. The gap matters even on day one:

file:// (double-click)http:// (server)
Good forFirst experimentsEverything real
Limitationsfetch()/modules/some APIs blocked, paths behave oddlyFull platform available
Get it—VS Code "Live Server" extension, or npx serve, or python -m http.server

Install Live Server now (it also auto-refreshes on save — a huge quality-of-life gain for the rest of this track). Later, "deploying a site" simply means copying your files to a computer that runs a server 24/7 — GitHub Pages, Netlify, and friends do this for free, and this very site is plain HTML files on GitHub Pages.

Part 7 — HTML's Evolution in Five Lines (interview context)

1991  HTML invented at CERN (Tim Berners-Lee) — 18 tags, documents for physicists
1995-99  HTML 2/3/4 — tables, forms, scripts; the browser wars add chaos
2000  XHTML — a strict XML detour (draconian error handling; the web ignored it)
2008+ HTML5 — pragmatic reboot: semantic elements, audio/video, canvas, form types
Today "HTML" = the WHATWG Living Standard — versionless, continuously updated

Two takeaways: there is no "HTML6" coming — features ship individually and you check support on caniuse.com or MDN; and when you meet <center> or <font> in old code, they're deprecated relics from before CSS took over the presentation layer.

Interview Questions

Q1 (beginner): Is HTML a programming language? Defend your answer.

No — it's a markup language. It has no logic: no variables, conditionals, loops, or functions; it cannot compute. It declares structure and meaning, which the browser renders. Strong answers add the layer model (HTML = structure, CSS = presentation, JS = behavior) and note that "not a programming language" ≠ "not deep" — accessibility, semantics, SEO, and parsing behavior are genuine expertise.

Q2 (beginner): What's the difference between an element, a tag, and an attribute?

A tag is the bracketed marker (<p> opening, </p> closing). An element is the complete unit: opening tag + content + closing tag. Attributes are name="value" pairs inside the opening tag providing extra information (href, src, class). Bonus: void elements (img, br, input) have no content or closing tag.

Q3 (mid): Walk me through what happens between typing a URL and seeing a page.

URL parsing → DNS resolution (domain → IP, heavily cached) → TCP connection + TLS handshake → HTTP GET request → server responds with HTML → browser parses top-down, building the DOM → discovers subresources (CSS, JS, images) and fetches them (CSS blocks rendering; classic scripts block parsing) → style + layout + paint. Interviewers probe follow-ups from whichever step you name confidently, so know one detail per step.

Q4 (mid): Why might the Elements panel show different markup than View Source?

View Source shows the bytes the server sent. The Elements panel shows the current DOM, which differs because (1) the parser repaired invalid markup — auto-closed tags, relocated misnested elements; (2) JavaScript added/removed/modified nodes after load; (3) with client-side frameworks the source may be nearly empty and the DOM entirely script-built. Debugging rule: CSS and JS operate on the DOM, so trust the Elements panel.

Q5 (senior): HTML parsers never throw errors. Why was it designed that way, and what are the engineering consequences?

Postel's-law pragmatism: early web authors were amateurs, and a browser that white-screened on malformed markup would have killed adoption (XHTML tried strict error handling and failed precisely because one bad ampersand nuked the page). HTML5 went further: it specifies exact recovery for every malformed input, so all engines repair identically — cross-browser parsing divergence essentially disappeared. Consequences: resilience (a half-downloaded page still renders — this enables streaming HTML), but silent divergence between author intent and the DOM, pushing correctness onto tooling (validators, linters, CI checks) rather than the runtime. Also a security dimension: sanitizers must parse exactly like browsers do, or mismatched repair behavior becomes an XSS vector (mutation XSS — covered in this track's Security lesson).

Q6 (senior): Your CSS selector matches in DevTools but the rule "doesn't work" on load. The markup has a div inside a p. Explain.

Block-level elements can't be children of <p>. When the parser hits <div> inside an open <p>, it closes the paragraph first, so the DOM is p then sibling div then an empty stray </p> is dropped. A selector like p div or p > div matches nothing because no such ancestry exists in the repaired tree. The fix is correcting the markup. The senior point: this class of bug is invisible in the source and obvious in the Elements panel — parser repair is the first suspect whenever the DOM contradicts the source.

What matters: writing the skeleton from memory (you'll type it thousands of times), and the validator run — most first attempts have at least one surprise.

2 (easy): Break it on purpose. Take exercise 1 and, one at a time: delete the doctype, delete </ul>, put a <div>hello</div> inside a paragraph. After each change, refresh and inspect the Elements panel. Write down what the browser did.
What you should observe

(1) No doctype → page still renders but the browser enters quirks mode (see the document-structure lesson) — check by typing document.compatMode in the Console: "BackCompat" = quirks. (2) Missing </ul> → the parser closes it for you at a sensible boundary; the Elements panel shows a complete list. (3) The div-in-p → the DOM shows the <p> closed before the div: parser repair in action. Lesson: breakage is silent — the tree just quietly differs from your source.

3 (medium): Trace the tree. Without a browser, draw the DOM tree (like Part 2's diagram) for: <body><h1>Menu</h1><p>Our <em>daily</em> specials:</p><ol><li>Soup</li><li>Salad</li></ol></body>. Include text nodes. Then verify in DevTools.
Solution 
body
├── h1 ── "Menu"
├── p
│   ├── "Our "          ← text node
│   ├── em ── "daily"
│   └── " specials:"    ← text node
└── ol
    ├── li ── "Soup"
    └── li ── "Salad"

The key insight most beginners miss: <p> has three children — text, element, text. Mixed content like this is why the DOM distinguishes text nodes from elements.

4 (medium): The coffee page, extended. Rebuild Part 4's page and extend it: a second <h2> section "Brewing Guide" with an ordered list of 4 steps, a link to an external site that you verify works, and an image with meaningful alt text. Then break the image path and confirm the alt text appears.
Solution sketch & checks 
<h2>Brewing Guide</h2>
<ol>
  <li>Grind 20g of beans, medium-coarse</li>
  <li>Heat water to 94°C</li>
  <li>Pour in slow circles for 3 minutes</li>
  <li>Rest 30 seconds, then serve</li>
</ol>
<p>Beans from <a href="https://en.wikipedia.org/wiki/Coffee">around the world</a>.</p>
<img src="pourover.jpg" alt="Barista pouring water over coffee grounds in a V60 dripper">

Checks: ordered list because brewing steps have a sequence (semantic choice!); alt text describes what the image shows, not "image of coffee".

5 (hard): Serve it properly. Install VS Code + Live Server (or run python -m http.server in your folder). Open the page at http://localhost:... instead of file://. Then open DevTools → Network tab and refresh: identify the request for your HTML, its status code, and every subresource request your page triggered. Screenshot-level goal: you can point at each Network row and say which line of your HTML caused it.
What you should find

Row 1: localhost (or index.html), status 200, type document — the page itself. Then one row per <img> (type img). If you added none, the only extra may be favicon.ico — the browser requests it automatically (status 404 if you have none; harmless). Every row maps to either the address bar or a tag with a src/ href. This request-per-resource model is the foundation of the Performance lesson later. Refresh with cache disabled (checkbox in Network tab) to see honest timings.

6 (hard): Explain-it test. The real mastery check: explain to another person (or a rubber duck, out loud) (a) the three-layer model with an example of a problem belonging to each layer, (b) the page-load journey, (c) why the Elements panel and View Source can differ. No notes. If you stumble anywhere, reread that part — these three explanations are the foundation the next 30 lessons build on.
Self-check rubric

You pass (a) if you named HTML/CSS/JS with responsibilities AND gave a concrete misuse (e.g., "centering with HTML instead of CSS"). You pass (b) if you hit DNS → connection → request → response → parse → render without prompting. You pass (c) if you used the words "parser repair" or "JavaScript modified the DOM". Teaching-back is the strongest known self-test — use it at the end of every lesson in this track.

🎯 Key Takeaways

  • HTML declares structure and meaning; CSS handles looks, JS handles behavior — route every problem to its layer.
  • A page load = DNS → connection → HTTP request → HTML response → parse into the DOM tree → fetch subresources → render.
  • Element = opening tag + content + closing tag; attributes are quoted name="value" pairs in the opening tag; void elements never close.
  • Browsers repair broken HTML silently and identically (spec-defined) — validate while learning, and debug against the Elements panel, not View Source.
  • file:// is for first experiments; real development uses a local server (Live Server / python -m http.server).
  • HTML is a versionless living standard — check feature support on MDN/caniuse, and treat <center>-era tags as museum pieces.
