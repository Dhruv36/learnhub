HTML Introduction Basics

HTML (HyperText Markup Language) is the language every web page is written in. "HyperText" means documents connected by links; "Markup" means you annotate content with tags that give it meaning. HTML is not a programming language — it has no logic, loops, or variables. It describes structure and meaning: this is a heading, this is a paragraph, this is a button. CSS then controls appearance, and JavaScript controls behavior.

How the Web Actually Works (60-second version)

1. You type https://example.com → browser asks DNS for the server's IP address
2. Browser sends an HTTP request:   GET / HTTP/1.1
3. Server responds with an HTML document (plain text!)
4. Browser PARSES the HTML into a tree of objects — the DOM
5. Browser fetches CSS/JS/images referenced by the HTML, then paints pixels

Two facts from this that shape everything you'll learn: HTML is just text (you can write a website in Notepad), and the browser builds a tree (every tag becomes a node with parents and children — CSS selectors and JavaScript both navigate this tree).

HTML collapses any run of spaces/tabs/newlines into a single space when rendering — indent freely for readability. Tag and attribute names are case-insensitive, but lowercase is the universal convention (and required in XML-based tooling).

Interview Questions

Q1: Is HTML a programming language? Defend your answer.

No — it's a markup language: it declares structure/semantics but has no control flow, variables, or computation. The follow-up interviewers want: that's by design — a declarative document format is parseable, indexable, accessible, and error-tolerant in ways executable code isn't.

Q2: What happens between typing a URL and seeing a page?

DNS resolution → TCP + TLS handshake → HTTP request → server returns HTML → browser parses HTML to DOM (pausing for sync scripts) → CSS parsed to CSSOM → render tree → layout → paint. Mentioning that parsing is incremental (the page can start rendering before the HTML finishes downloading) is a strong signal.

Q3: Why must <meta charset> appear early?

The parser must know the encoding before decoding bytes into characters; the spec requires it within the first 1024 bytes or the browser may re-parse the document from scratch with the newly discovered encoding.

Q4: What is quirks mode?

A compatibility mode browsers use when the doctype is missing/ancient: it emulates pre-standards behavior (e.g., the old IE box model where width includes padding+border). Triggering it accidentally is the classic cause of "CSS behaves differently on this one page".

🏋️ Practice (open the Playground):
1. Build a page with doctype, lang, charset, title, one h1 and two paragraphs.
2. Break it on purpose: remove the doctype, cross-nest two tags — observe that the browser still renders. Inspect with F12 → Elements to see how the parser "repaired" your tree.
3. Add a data-theme="dark" attribute to body and view it in the inspector.

🎯 Key Takeaways

  • HTML = structure & meaning; CSS = appearance; JS = behavior. Keep the roles separate.
  • The browser parses HTML into the DOM tree — everything (CSS, JS, a11y) operates on that tree.
  • Doctype, lang, charset, title: four lines, all non-negotiable.
  • Parsers forgive broken HTML silently — validate instead of trusting what renders.
  • Choose tags for meaning, never for their default styling.
