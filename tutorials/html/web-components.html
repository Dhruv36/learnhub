Web Components & Shadow DOM Ultra-Advanced

The browser's native component model: your own elements, with encapsulated markup, style, and behavior — no framework. YouTube's player, GitHub's relative timestamps, Adobe's design system, and Photoshop-on-web all ship on this. Their production superpower: one component that works identically in React, Angular, Vue, and plain HTML — the standard answer for design systems in multi-framework organizations.

The Three Pillars

TechnologyGives you
Custom ElementsYour own tags + lifecycle callbacks (connected/disconnected/attributeChanged)
Shadow DOMA private DOM subtree with scoped CSS — page styles can't get in, component styles can't leak out
Templates & Slots<template>: inert reusable markup; <slot>: holes where consumer content projects in

A Real Component, Annotated

class UserBadge extends HTMLElement {
  static observedAttributes = ["status"];             // watch these attributes

  connectedCallback() {                                // added to the DOM
    const shadow = this.attachShadow({ mode: "open" });
    shadow.innerHTML = `
      <style>
        :host { display: inline-block; }               /* the element itself */
        :host([status="online"]) .dot { background: #1a7f37; }
        .badge { display: inline-flex; gap: .5rem; align-items: center;
                 padding: .3rem .7rem; border-radius: 999px;
                 background: var(--badge-bg, #eaf5ee); }   /* themable knob */
        .dot { width: 8px; height: 8px; border-radius: 50%; background: #999; }
      </style>
      <span class="badge" part="badge">                <!-- external styling hook -->
        <span class="dot"></span>
        <slot>Anonymous</slot>                        <!-- consumer content lands here -->
      </span>`;
  }

  attributeChangedCallback(name, oldV, newV) { /* react to status changes */ }
  disconnectedCallback() { /* cleanup: listeners, timers — the memory-leak lesson applies */ }
}
customElements.define("user-badge", UserBadge);        // tag name MUST contain a hyphen
<user-badge status="online">Dhruv · Staff Engineer</user-badge>
<style> user-badge { --badge-bg: #fff8c5; } user-badge::part(badge) { font-weight: 600; } </style>

Shadow DOM Encapsulation — Exactly What Crosses the Boundary

ThingCrosses?Detail
Page CSS selectors❌.badge { } outside cannot touch the shadow .badge — the global-CSS-collision problem, solved
CSS custom properties✅The deliberate theming channel: expose --badge-bg-style knobs
::part()✅Style only what the component marked with part="" — a public styling API
Inherited properties✅color, font-family inherit through unless reset
Events✅*Bubble out but are retargeted — outside listeners see the host element as target, not shadow internals
document.querySelector❌Can't reach in (open mode allows el.shadowRoot.querySelector explicitly)

Slots — Composition Like a Framework

<template>  <!-- inside a card component's shadow -->
  <header><slot name="title">Untitled</slot></header>
  <slot></slot>                          <!-- default slot: everything unnamed -->
</template>

<my-card>
  <h3 slot="title">Q3 Report</h3>         <!-- projects into slot[name=title] -->
  <p>Revenue grew 40%…</p>                <!-- default slot -->
</my-card>

Slotted content stays in the light DOM (page CSS still styles it) — it's projected, not moved. Components style slotted content only via ::slotted(h3).

SSR Arrives: Declarative Shadow DOM

<user-badge>
  <template shadowrootmode="open">
    <style>…</style><span class="badge"><slot></slot></span>
  </template>
  Dhruv
</user-badge>

The parser builds the shadow root from HTML alone — no JS needed for first paint. This unlocked server-side rendering for web components and is supported across modern browsers.

When to Choose What (the architecture answer)

SituationChoice
Design-system primitives shared across teams/frameworksWeb components (usually via Lit — ~5KB, ergonomics without lock-in)
Application state, routing, data flowA framework (React/Angular/Vue) — WCs don't solve app architecture
Embeddable third-party widget (chat bubble, payments)Web components — style isolation on hostile host pages is exactly Shadow DOM's job
Small enhancement on a content siteA single custom element beats importing a framework

Interview Questions

Q1: Shadow DOM vs iframe for isolation?

Both isolate; iframe is a separate document (own context, expensive, awkward sizing/communication), Shadow DOM is same-document scoping (cheap, events bubble, shares JS context). Style isolation → Shadow DOM; security isolation of untrusted code → iframe + sandbox, because Shadow DOM is not a security boundary.

Q2: How do web components handle FOUC before definition?

Until customElements.define runs, the tag is an unknown inline element — content may flash unstyled. Fixes: :not(:defined) { visibility: hidden } or reserved dimensions; Declarative Shadow DOM eliminates the problem by shipping the shadow tree in HTML.

Q3: Why does the tag name need a hyphen?

The hyphen namespace-separates custom elements from all current and future standard HTML tags — the parser can treat anything hyphenated as potentially upgradeable without ambiguity.

🏋️ Practice:
1. Build <copy-button>: shadow DOM, a slot for the label, clipboard write on click, and a "Copied!" state via attribute + :host([copied]) styling.
2. Add part="button" and theme it from page CSS via ::part.
3. Then rewrite it with a Declarative Shadow DOM template and observe it render with JS disabled (DevTools → Cmd/Ctrl+Shift+P → "Disable JavaScript").

🎯 Key Takeaways

  • Custom elements + Shadow DOM + slots = the browser's own component model.
  • Encapsulation is selective: custom properties and ::part are the intended public styling API.
  • Slotted content stays in the light DOM; events retarget at the boundary.
  • Declarative Shadow DOM makes web components SSR-able.
  • Use WCs for cross-framework primitives and embeddable widgets; frameworks for app architecture. Not rivals — layers.
