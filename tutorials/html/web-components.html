Web Components & Shadow DOM Production

What you'll master: Custom Elements (defining genuinely new HTML tags), Shadow DOM (real, browser-enforced style/DOM encapsulation — not a convention, an actual boundary), the component lifecycle callbacks, and where Web Components genuinely fit versus React/Vue/Svelte in a real engineering decision. This opens the track's final, most advanced section. Prerequisites: Choosing the Right Element, Global Attributes (attributeChangedCallback mirrors the dataset pattern).

Part 1 — Custom Elements: Genuinely New Tags

class StarRating extends HTMLElement {
  connectedCallback() {
    this.innerHTML = `<span>★★★☆☆</span>`;
  }
}
customElements.define("star-rating", StarRating);
<star-rating value="3"></star-rating>   <!-- a REAL element, not a div with a class -->

This is meaningfully different from every JS UI framework's component model: <star- rating> becomes a genuine, first-class element — it appears in the DOM tree exactly like <video> or <details>, browser DevTools inspect it natively, and it works with ZERO build step or framework runtime, in any HTML page, indefinitely (this is the "custom" half of the platform's own extensibility mechanism — genuinely part of the HTML/DOM standard, not a library convention). The one hard rule: custom element names MUST contain a hyphen (star-rating, never starrating) — this is the platform's explicit reservation mechanism, guaranteeing your custom tag can never collide with a future native HTML element the spec might add later (imagine defining <dialog> yourself in 2015, only for the real spec version to ship a few years later and silently conflict).

Part 2 — The Lifecycle Callbacks

class StarRating extends HTMLElement {
  static observedAttributes = ["value"];         // MUST declare which attributes to watch

  connectedCallback() {                           // element inserted into the DOM
    this.render();
  }
  disconnectedCallback() {                         // element removed from the DOM
    this.cleanup?.();                               // unsubscribe listeners, clear timers — avoid leaks
  }
  attributeChangedCallback(name, oldVal, newVal) {  // a WATCHED attribute changed
    this.render();
  }
  render() {
    const value = Number(this.getAttribute("value")) || 0;
    this.innerHTML = "★".repeat(value) + "☆".repeat(5 - value);
  }
}

Four callbacks cover the entire component lifecycle, each with a genuinely distinct, necessary job: connectedCallback is where you do initial setup (analogous to React's mount/useEffect); disconnectedCallback is where you clean up — this is the exact discipline the JS track's event-listener and closure-memory-leak lessons will assume you already know, now with a concrete hook to do it correctly; attributeChangedCallback fires when a watched HTML attribute changes (note: it does NOT fire automatically for every attribute — you must explicitly opt in via static observedAttributes, a genuinely easy detail to forget on a first attempt). This attribute-reactivity model directly parallels the dataset/data-* pattern from the Global Attributes lesson: the element's HTML attributes ARE its public, declarative API surface, exactly like any native element's attributes.

Part 3 — Shadow DOM: Real Encapsulation, Not a Convention

This is the section's most important idea, and the one genuinely new capability nothing else in this entire track provides: Shadow DOM creates a separate DOM subtree whose internal styles and markup are invisible to and unreachable from the outside page — enforced by the browser itself, not by a naming convention your team has to remember to follow.

class InfoCard extends HTMLElement {
  connectedCallback() {
    const shadow = this.attachShadow({ mode: "open" });
    shadow.innerHTML = `
      <style>
        /* This CSS is SCOPED — it cannot leak out, and the page's CSS cannot leak in */
        p { color: blue; font-weight: bold; }
      </style>
      <p>Card content</p>
    `;
  }
}
customElements.define("info-card", InfoCard);
<style>p { color: red; }</style>   <!-- the PAGE's CSS -->
<info-card></info-card>              <!-- its internal <p> renders BLUE, not red — genuinely isolated -->
<p>Regular text</p>                  <!-- this DOES render red, from the page's rule -->

Recall every "class namespace collision" and "specificity war" problem from the CSS track this lesson anticipates (and the class-naming discipline from the Global Attributes lesson, js- vs content-class separation) — Shadow DOM makes that entire category of problem structurally impossible for anything built inside it, rather than merely disciplined against via convention. This is the actual platform-native mechanism that CSS-in-JS libraries, CSS Modules, and BEM's strict naming conventions (all CSS-track topics) exist specifically to WORK AROUND at the tooling/discipline level — Shadow DOM solves the identical problem one level down, at the platform itself, requiring no tooling or discipline at all.

mode: "open" (shown above) allows the outside page's JavaScript to reach in via element.shadowRoot if genuinely needed (debugging, specific integration cases); mode: "closed" refuses even that — a fully sealed boundary. Open is the overwhelmingly common, recommended default; closed is a rare, deliberate choice for cases needing the strongest possible isolation guarantee.

Part 4 — Worked Example: a Complete, Self-Contained Component

class ProductBadge extends HTMLElement {
  static observedAttributes = ["label", "variant"];

  connectedCallback() {
    this.shadow = this.attachShadow({ mode: "open" });
    this.render();
  }
  attributeChangedCallback() {
    if (this.shadow) this.render();   // guard: attribute can change before connectedCallback runs
  }
  render() {
    const label = this.getAttribute("label") || "New";
    const variant = this.getAttribute("variant") || "default";
    this.shadow.innerHTML = `
      <style>
        .badge { padding: 4px 10px; border-radius: 4px; font: 600 12px sans-serif; }
        .default { background: #e5e7eb; color: #374151; }
        .sale    { background: #fee2e2; color: #b91c1c; }
      </style>
      <span class="badge ${variant}">${label}</span>
    `;
  }
}
customElements.define("product-badge", ProductBadge);
<product-badge label="Sale!" variant="sale"></product-badge>
<product-badge label="Best Seller"></product-badge>

This single, dependency-free file ships a genuinely reusable, styled, attribute-driven component that works IDENTICALLY whether dropped into a plain HTML page, a React app, a Vue app, or a Node- rendered template — a real, practical payoff of the platform-native approach, since Web Components are, by design, framework-agnostic in a way no single framework's own component model can be for components built in a DIFFERENT framework.

Part 5 — Deep Dive: Where This Genuinely Fits (the honest tradeoff)

The question this whole track has trained you to ask by now, applied here: when does Web Components' platform-native approach actually win against React/Vue/Svelte's more feature-rich, ecosystem-backed component models?

Web Components genuinely win when…A framework genuinely wins when…
Building a component that must work across MULTIPLE frameworks/no framework (a design-system's shared primitives, an embeddable widget for third-party sites)Building a large, cohesive application where the framework's ecosystem (state management, routing, dev tooling) matters more than any single component's portability
Long-term stability matters more than developer ergonomics — a platform standard genuinely doesn't get deprecated the way a specific framework version canThe team's existing productivity/tooling investment is already deep in one framework
Zero-dependency, framework-agnostic distribution is a real requirement (an npm-published UI-kit meant for consumption by unknown downstream frameworks)Complex, deeply nested reactive state needs to flow through many components — frameworks' reactivity systems are considerably more ergonomic for this than raw Custom Elements' manual attribute/property wiring

The honest, current-industry reality: most production applications still reach for a framework for the application layer, while Web Components see genuine, growing adoption specifically at the SHARED-COMPONENT-LIBRARY layer — design systems (Adobe's Spectrum, Salesforce's Lightning, many others) increasingly ship their base primitives (buttons, date pickers, form controls) as actual Custom Elements precisely BECAUSE that guarantees the same button component works correctly whether a consuming team is on React, Vue, Angular, or nothing at all. This is not "Web Components lost" or "Web Components won" — it's two genuinely different layers of a real system, each best served by a different tool, and recognizing which layer you're actually building for is the real skill this lesson leaves you with.

Interview Questions

Q1 (beginner): Why must a custom element's name contain a hyphen?

It's the platform's explicit reservation mechanism, guaranteeing your custom element name can never collide with a native HTML element the spec adds in the future — every native element name is guaranteed hyphen-free, so any hyphenated name is permanently safe for custom use. customElements.define() actually throws an error and refuses to register a name without one, so this isn't just a style convention, it's enforced by the browser.

Q2 (beginner): What does Shadow DOM's style encapsulation actually guarantee, mechanically?

CSS defined inside a shadow root cannot leak out to affect the outside page, AND the outside page's CSS cannot leak in to affect the shadow root's internal elements — a real, browser-enforced boundary, not a naming-convention discipline (like BEM or CSS Modules) that a team must remember to follow correctly. A `p { color: red }` rule on the page genuinely has zero effect on a `p` element rendered inside a component's shadow root.

Q3 (mid): Walk through the four lifecycle callbacks and what each is responsible for.

connectedCallback: fires when the element is inserted into the DOM — initial setup/rendering. disconnectedCallback: fires on removal — cleanup (unsubscribe listeners, clear timers) to avoid memory leaks. attributeChangedCallback: fires when a WATCHED attribute changes value — but only for attributes explicitly listed in static observedAttributes; it does not fire automatically for every attribute change. Together these cover mount, attribute-reactivity, and unmount — the same three concerns any component framework's lifecycle model addresses, here at the platform level.

Q4 (mid): A team is building a date-picker component meant to be published as an npm package and used across their React app, their Vue admin panel, and a plain-HTML marketing site. Why might Web Components be the right choice here specifically?

This is precisely the cross-framework distribution case Web Components genuinely solve better than any single framework's component model — a React component can't be natively dropped into a Vue app or plain HTML without a framework-specific wrapper/adapter layer, while a Custom Element works identically in all three environments because it's a real, native DOM element, not a framework abstraction requiring a matching runtime. Building it once as a Web Component avoids maintaining three separate implementations (or three separate wrapper layers) of the same date-picker logic.

Q5 (senior): Your design system team is deciding whether to build the organization's shared component library (buttons, inputs, cards, modals) as Web Components or as a React library with framework-specific ports for the two other frameworks used internally (Vue, Angular). Walk through the real tradeoffs.

Web Components approach: build once, works natively everywhere (including any FUTURE framework the org might adopt, without needing a new port) — the strongest argument specifically for long-lived, foundational primitives that many teams across the org will depend on for years; real costs: the developer experience writing Custom Elements/Shadow DOM directly is measurably more verbose than a modern framework's component authoring (though wrapper libraries like Lit significantly narrow this gap while still compiling to real Custom Elements), and deep reactive- state integration (a modal's open state syncing with a complex app-wide store) requires more manual wiring than a framework's native reactivity. React-plus-ports approach: best-in-class authoring ergonomics for the PRIMARY framework, but each additional framework port is genuinely separate ongoing maintenance — bugs must be fixed and features added in multiple codebases, and behavioral drift between ports is a real, common failure mode in practice. The senior recommendation, and the actual industry pattern at this scale: build the CORE, foundational, highest-reuse primitives (button, input, icon, basic layout primitives) as Web Components — where the "build once, works everywhere including the future" argument is strongest and the components are simple enough that the authoring-ergonomics cost is minor — while allowing more complex, framework-specific, deeply-app-integrated components (a data-grid tightly coupled to one app's specific state architecture) to remain framework-native where the reactivity/ergonomics tradeoff genuinely favors it. This is a real, common hybrid strategy at organizations operating multiple frameworks, not a purely theoretical middle ground.

Q6 (senior): A junior engineer builds a Custom Element that renders user-submitted product review text directly via `this.shadow.innerHTML = review.text`. Assess the security implications, and explain whether Shadow DOM's encapsulation provides any protection here.

Zero protection, and the vulnerability is exactly as severe as anywhere else in the application: Shadow DOM's boundary is about STYLE and DOM-STRUCTURE isolation between the component and the outside page — it has NOTHING to do with sanitizing untrusted content, and a script tag or onerror-laden element injected via innerHTML inside a shadow root executes with full access to the PAGE's JavaScript context, cookies, and session — the shadow boundary doesn't sandbox script execution at all, only style/DOM leakage between component and page. This is the exact same stored-XSS pattern from the Global Attributes lesson's contenteditable warning and the Inline Semantics lesson's mark-highlighter exercise, recurring here inside a Web Component with no additional protection whatsoever — a genuinely easy, dangerous assumption for a team new to Web Components to make ("it's isolated, so it must be safer"), and a false one. Fix: identical discipline to every other untrusted-content rendering path in this track — sanitize with an allow-list library (DOMPurify) before interpolating into innerHTML, inside a shadow root exactly as anywhere else, with no exception granted by the encapsulation boundary.

🏋️ Exercises

1 (easy): Your first Custom Element. Build a <hello-badge> element that renders "Hello, [name]!" using a name attribute, with a sensible default if the attribute is omitted. Register it and use it three times on a page with different name values. Inspect it in DevTools' Elements panel and confirm it appears as a real, named element in the tree.
Solution 
class HelloBadge extends HTMLElement {
  connectedCallback() {
    const name = this.getAttribute("name") || "World";
    this.innerHTML = `<strong>Hello, ${name}!</strong>`;
  }
}
customElements.define("hello-badge", HelloBadge);

The Elements panel should show <hello-badge name="..."> as a genuine, distinctly-named node in the tree — not a div, confirming this is real platform-level element registration, not a JS abstraction layered on top of a generic container.

2 (easy): Lifecycle logging. Add console.log statements to connectedCallback and disconnectedCallback on any custom element. Insert it into the page, then remove it via el.remove() in the Console, then re-insert it via appendChild. Confirm the logs fire in the expected order and count.
Expected sequence

connectedCallback fires once on initial page insertion, disconnectedCallback fires once on el.remove(), and connectedCallback fires AGAIN if you re-append the same element reference — confirming these callbacks genuinely track DOM insertion/removal events, not just "component created once" — an element can connect and disconnect multiple times across its lifetime if moved around the DOM.

3 (medium): attributeChangedCallback, correctly wired. Build the Part 2 StarRating component completely, including static observedAttributes. Add a button outside the component that changes its value attribute via setAttribute() on click. Confirm the star display updates live. Then deliberately REMOVE the observedAttributes line and confirm updates silently stop working.
What to confirm

With observedAttributes correctly declared: clicking the button visibly updates the star display immediately, confirming attributeChangedCallback fired in response to the setAttribute call. After removing the declaration: the attribute value genuinely changes (verify via el.getAttribute("value") in the Console) but the visual display silently fails to update — a direct, hands-on demonstration of the "easy to forget" mistake this lesson's table warns about.

4 (medium): Shadow DOM isolation, proven. Build the Part 3 InfoCard example exactly as shown, with a page-level `p { color: red }` rule already in place before you add the component. Confirm the component's internal paragraph renders blue (its own scoped style) while regular page paragraphs render red. Then try to select the component's internal p via document.querySelector("info-card p") from the Console — confirm it returns null, proving genuine DOM isolation, not just CSS isolation.
Expected result

The color isolation should render exactly as described — visually confirming CSS encapsulation. The querySelector attempt returning null (rather than the paragraph element) demonstrates something even stronger: regular DOM traversal from OUTSIDE the component cannot reach INSIDE the shadow root at all — you'd need element.shadowRoot.querySelector(...) explicitly, proving this is genuine structural isolation, not merely a styling trick.

5 (hard): Complete, reusable component. Build Part 4's ProductBadge component completely, with both observedAttributes correctly wired and the attributeChangedCallback guard (checking this.shadow exists) explained in Part 4's comment. Use it 5 times on a page with different label/variant combinations, and add a button that changes one instance's variant dynamically. Confirm only that one instance's styling updates.
What "complete" means here

A genuinely complete version handles the subtle timing issue Part 4's guard exists for: attributeChangedCallback can theoretically fire before connectedCallback has run (if an attribute is set programmatically before the element is even connected to the DOM) — without the this.shadow existence check, this produces a real runtime error trying to call render() on a shadow root that doesn't exist yet. Deliberately triggering this specific scenario (set an attribute on a detached, not-yet-appended element) and confirming the guard prevents a crash is the exercise's real payoff — most first attempts at Custom Elements miss this exact edge case.

6 (hard): The XSS trap, demonstrated and fixed. Build a <review-card> Custom Element that renders review text via a `text` attribute directly into shadow innerHTML with no escaping. Set its text attribute to a string containing an <img src=x onerror=alert('shadow DOM does not save you')> payload and confirm the alert fires DESPITE the content being inside a shadow root. Then fix it using DOMPurify (or manual escaping) and confirm the payload is neutralized.
Fix 
render() {
  const raw = this.getAttribute("text") || "";
  const safe = DOMPurify.sanitize(raw);   // or manual escaping if DOMPurify unavailable
  this.shadow.innerHTML = `<p>${safe}</p>`;
}

The alert firing in the unfixed version is the whole point of this exercise — a hands-on, undeniable demonstration that Shadow DOM's real isolation guarantee (style/DOM leakage between component and page) provides absolutely no protection against script execution from unescaped content, directly confirming Q6's interview answer through your own reproduced exploit rather than only reading the claim.

🎯 Key Takeaways

  • Custom Elements register genuinely new, first-class HTML tags (hyphen required, browser-enforced) — no build step or framework runtime needed, working in any HTML page indefinitely.
  • Four lifecycle callbacks — connectedCallback (mount), disconnectedCallback (cleanup, avoid leaks), attributeChangedCallback (reacts only to explicitly observedAttributes) — cover the full component lifecycle at the platform level.
  • Shadow DOM provides REAL, browser-enforced style and DOM-structure isolation — solving the exact class-namespace-collision problem CSS-in-JS/CSS Modules/BEM work around via discipline, but at the platform level instead.
  • Shadow DOM's isolation is about style/DOM leakage ONLY — it provides zero protection against XSS; untrusted content rendered via innerHTML inside a shadow root needs identical sanitization discipline as anywhere else in the app.
  • The real industry pattern: Web Components win at the shared, cross-framework, long-lived component-library layer (design-system primitives); frameworks still win at the application layer (state, routing, complex reactivity) — recognizing which layer you're building for is the actual skill.
  • Attribute-driven components mirror the dataset/data-* pattern from earlier in the track — an element's HTML attributes ARE its public, declarative API, exactly like any native element.
