Constraint Validation API Forms Mastery

What you'll master: the JavaScript face of HTML validation — reading validity state, adding rules HTML attributes can't express (password confirmation, at-least-one-checkbox, async username checks), replacing ugly native bubbles with accessible inline errors, and the requestSubmit-vs-submit distinction. This is the lesson where Layers 1 and 2 of the validation model become one coherent system. Prerequisites: Built-in Validation; basic JavaScript reading ability (the JS track deepens everything used here).

Part 1 — The API Surface (small, powerful)

Every form control carries validation machinery; the HTML attributes from last lesson are its declarative face, and this API is its programmatic face — one system, two doors:

// On each CONTROL:
input.validity                 // ValidityState object — WHY it's invalid (see Part 2)
input.checkValidity()          // boolean; fires an 'invalid' event on the control if false
input.reportValidity()         // same + shows the native bubble
input.setCustomValidity(msg)   // add/clear a custom error ("" = valid again)
input.validationMessage        // the current human-readable error text
input.willValidate             // is this control even participating? (disabled/hidden inputs: no)

// On the FORM:
form.checkValidity()           // all controls valid?
form.reportValidity()          // check + bubble on first invalid
form.requestSubmit()           // submit WITH validation + submit event (use this)
form.submit()                  // submit, skipping validation AND the submit event (legacy trap)
form.elements                  // all controls, by index and by name — your iteration tool

Part 2 — ValidityState: Asking WHY

input.validity = {
  valueMissing:    true if required and empty
  typeMismatch:    email/url shape wrong
  patternMismatch: pattern regex failed
  tooShort / tooLong:        minlength/maxlength violated
  rangeUnderflow / rangeOverflow:  min/max violated
  stepMismatch:    step violated
  badInput:        browser can't even parse it (letters in a number field)
  customError:     setCustomValidity() is active
  valid:           the summary boolean
}

This is what turns one generic "invalid" into specific, kind error messages — you branch on the reason:

function messageFor(input) {
  const v = input.validity;
  if (v.valueMissing)   return "Please fill in your " + label(input) + ".";
  if (v.typeMismatch && input.type === "email")
                        return "That doesn't look like an email — check for typos.";
  if (v.tooShort)       return `At least ${input.minLength} characters (you have ${input.value.length}).`;
  if (v.patternMismatch)return input.title || "Unexpected format.";
  if (v.customError)    return input.validationMessage;   // whatever we set ourselves
  return input.validationMessage;                          // native fallback
}

Note the philosophy: the rules still live in HTML attributes (single source of truth, works without JS); JavaScript only upgrades the presentation. That's the architecture to carry into every framework.

Part 3 — setCustomValidity: Rules HTML Can't Express

The keystone method. A control with a non-empty custom validity message is invalid; setting "" clears it. The trap everyone hits once: forget to clear it and the field is invalid forever — always set or clear on every check.

Worked example 1: password confirmation

const pw = document.getElementById("pw");
const confirm = document.getElementById("pw2");

function checkMatch() {
  confirm.setCustomValidity(
    confirm.value && confirm.value !== pw.value ? "Passwords don't match." : ""
  );
}
pw.addEventListener("input", checkMatch);
confirm.addEventListener("input", checkMatch);

Both fields listen (editing the first password can invalidate the match), empty confirm stays neutral (required handles emptiness — don't double-report), and the empty string clears. Fifteen lines, fully integrated: the browser now treats mismatch exactly like any native constraint — blocking submission, showing in :user-invalid, everything.

Worked example 2: at-least-one-checkbox (the native gap)

const boxes = [...form.querySelectorAll('input[name="toppings"]')];
function checkGroup() {
  const anyChecked = boxes.some(b => b.checked);
  // park the message on the FIRST box; clear on all
  boxes[0].setCustomValidity(anyChecked ? "" : "Choose at least one topping.");
}
boxes.forEach(b => b.addEventListener("change", checkGroup));
checkGroup();

Worked example 3: async check (username taken?)

let timer;
username.addEventListener("input", () => {
  username.setCustomValidity("");            // optimistic while typing
  clearTimeout(timer);
  timer = setTimeout(async () => {           // debounced — remember the JS track's closure lesson
    if (!username.checkValidity()) return;   // don't waste a request on locally-invalid values
    const res = await fetch(`/api/check-username?u=${encodeURIComponent(username.value)}`);
    const { taken } = await res.json();
    username.setCustomValidity(taken ? "That username is taken." : "");
  }, 400);
});

Patterns to note: debounce (not one request per keystroke), local validity gate before the network, encodeURIComponent (lesson 5's URL encoding, now defensive), and the race-condition caveat — a slow early response can overwrite a fast later one; production code tags requests and ignores stale responses (an AbortController per keystroke is the clean fix — JS track).

Part 4 — Replacing Native Bubbles with Inline Errors (the full pattern)

Native bubbles: free and localized, but unstylable, one-at-a-time, and they vanish on blur. Most products want persistent inline errors. Here's the complete accessible pattern — this is the lesson's centerpiece; read it slowly:

<form id="signup" action="/signup" method="post" novalidate>  <!-- WE own presentation now -->
  <label for="em">Email</label>
  <input id="em" name="email" type="email" required
         aria-describedby="em-err">                <!-- ties the error text to the field -->
  <p class="field-error" id="em-err"></p>          <!-- error lives HERE, persistently -->
  ...
  <button type="submit">Create account</button>
</form>

<script>
const form = document.getElementById("signup");

form.addEventListener("submit", (e) => {
  if (!form.checkValidity()) {           // constraints still come from the HTML attributes!
    e.preventDefault();                  // stop the (novalidate) submission
    let first = null;
    for (const el of form.elements) {
      if (el.willValidate) showError(el) && (first ??= el);
    }
    first?.focus();                      // replicate the native focus-first behavior
  }
});

// live re-validation: clear an error the moment the user fixes it
form.addEventListener("input", (e) => {
  if (e.target.willValidate) showError(e.target);
});

function showError(el) {
  const box = document.getElementById(el.getAttribute("aria-describedby"));
  if (!box) return false;
  const bad = !el.checkValidity();
  box.textContent = bad ? messageFor(el) : "";     // Part 2's messageFor
  el.setAttribute("aria-invalid", bad ? "true" : "false");
  return bad;
}
</script>

Why each piece exists: novalidate hands presentation to us while the attributes keep defining the rules; aria-describedby makes screen readers read the error with the field; aria-invalid announces state; focus-first mirrors the native UX blind users rely on; input-event re-validation means errors disappear the moment they're fixed (validate late, re-validate eagerly — errors appear on submit/blur, vanish on input: the kindest timing). Add an error summary at the top ("3 fields need attention", links to each) for long forms — the Form UX lesson formalizes it.

Part 5 — Deep Dive: submit() vs requestSubmit(), and the invalid event

  • form.submit() skips everything — no validation, no submit event. By spec, for legacy reasons. If your JS submits forms, use requestSubmit(), which behaves like a real button click: validation runs, submit event fires (so your handler + preventDefault work). A whole class of "validation mysteriously bypassed" bugs is this one method choice.
  • The invalid event fires on each failing control when check/reportValidity runs — cancel it (e.preventDefault()) to suppress the native bubble per-control (the pre-novalidate technique; you'll meet it in older codebases).
  • willValidate is false for disabled controls, output, and controls inside a <datalist> — iterate with it (as Part 4 does) instead of guessing.
  • Controls tied to a form via the form="form-id" attribute (living outside the form element — grid layouts love this) participate fully in that form's validation and submission.

Interview Questions

Q1 (beginner): What does setCustomValidity do, and what's the classic mistake with it?

It attaches a custom error to a control, making it invalid with that message — integrating custom rules into native validation (blocking submission, styling hooks, bubbles). Clearing requires explicitly setting the empty string; the classic mistake is setting a message on failure but never clearing on success, leaving the control invalid forever.

Q2 (beginner): Where should validation rules live when you enhance a form with JS, and why?

In the HTML attributes (required, pattern, min…). JS reads them via checkValidity/validity rather than re-implementing. One source of truth (no drift), no-JS fallback keeps working, and native machinery (:user-invalid, bubbles if wanted) stays consistent. JS adds only what attributes can't express — cross-field, group, async rules — via setCustomValidity.

Q3 (mid): Implement password-confirmation validation. What details separate a correct implementation from a naive one?

setCustomValidity on the confirm field: message when non-empty and mismatched, "" otherwise. Correctness details: (1) BOTH fields listen — editing the original invalidates a previous match; (2) empty confirm stays neutral — required owns emptiness, avoid double-errors while the user hasn't gotten there; (3) timing — show on blur/submit, clear on input; (4) the error text is associated via aria-describedby if using custom UI. Bonus: mention this pairs with autocomplete="new-password" on both.

Q4 (mid): Why does form.submit() exist alongside requestSubmit(), and which bugs does confusing them cause?

submit() predates the validation API and is spec-frozen to skip constraint validation AND the submit event (legacy pages relied on it). requestSubmit() (2020+) emulates a submit-button click: validates, fires the event, honors preventDefault. Confusion bugs: JS-driven submissions that bypass all validation (data quality incidents), and submit handlers that "never run" when triggered programmatically — analytics and CSRF-token-injection handlers silently skipped. Rule: requestSubmit() unless you can articulate why not.

Q5 (senior): Sketch the architecture of a reusable inline-error system for all your product's forms. What are the invariants?

Invariants: (1) rules declared in markup (attributes + data-* for custom rule hooks), never in per-form JS; (2) every error programmatically associated — aria-describedby to a stable error node, aria-invalid state, focus management to first failure; (3) timing contract: errors appear at blur/submit, clear on input, error summary for 3+ failures; (4) all submissions route through requestSubmit; (5) async validators debounced + abortable with stale-response protection; (6) server errors re-enter the SAME rendering path (a 422 response maps field→message onto setCustomValidity/error nodes) so client and server failures look identical to users and to tests. Implementation: one delegated listener set at the form level (submit/input/blur), config via attributes — no per-field wiring. That last point — server errors flowing through the same pipeline — is what distinguishes a system from a snippet.

Q6 (senior): Your async username check causes a support ticket: "it said available, then signup failed." Explain the inherent limitation and the correct system design.

TOCTOU — time-of-check to time-of-use: availability was true at check time and false at submit time (another user registered in between, or the async race showed a stale response). Client checks can only ever be advisory. Correct design: (1) the check is UX ("likely available"), debounced/aborted properly; (2) the authoritative check is the INSERT with a unique constraint at the database — the only race-free arbiter; (3) the signup endpoint returns a structured 409/422 that the form renders through the standard error path (Q5's invariant 6), suggesting alternatives; (4) optionally, short-lived reservations if product wants stronger promises. Naming TOCTOU and putting the truth at the uniqueness constraint is the expected senior answer.

🏋️ Exercises

1 (easy): validity explorer. Put an email field with required+minlength on a page. In the Console, for each of: empty, "x", "xxxx", "x@y.com" — log input.validity and note which flags flip. Map each flag to the attribute that drives it.
Expected mapping

Empty → valueMissing (required). "x" → tooShort? No — typeMismatch first surprise: minlength isn't evaluated until… actually both can be true; check: typeMismatch=true (not email-shaped), tooShort=true if minlength>1. "xxxx" → typeMismatch only. "x@y.com" → all false, valid:true. The exercise's point: validity is a set of independent flags, not one boolean — your error UX can be as specific as the flags.

2 (easy): The eternal-invalid bug. Deliberately write a custom check that sets a message but never clears it. Try to make the form submittable again without reloading. Then fix the code properly.
Resolution

Once set, nothing clears it except setCustomValidity("") — no amount of correct typing helps, because YOUR code is the only party that knows the rule. (Console rescue: input.setCustomValidity("").) The proper fix restructures the check so every code path ends in either a message or "" — the set-or-clear-always discipline from Part 3.

3 (medium): Group validation. Implement at-least-2-checkboxes ("pick at least two toppings") with a custom message that updates live and blocks submission. Then make the error show in an inline error node (Part 4 pattern), not a bubble.
Solution core 
function checkGroup() {
  const n = boxes.filter(b => b.checked).length;
  boxes[0].setCustomValidity(n >= 2 ? "" : `Pick at least two (you have ${n}).`);
  showError(boxes[0]);      // reuse the Part 4 renderer
}

The count-in-the-message detail matters: specific errors ("you have 1") outperform generic ones. Parking the validity on box[0] but rendering into a group-level error node (aria- describedby on the fieldset's first box, or better: the fieldset legend area) is the standard compromise for group errors.

4 (medium): Full inline-error form. Implement Part 4's pattern completely for a 4-field signup (email, password, confirm, handle) — novalidate, per-field error nodes, messageFor branching on validity flags, focus-first, clear-on-input. Test the whole matrix: submit empty, fix one field, break it again.
Self-check matrix

Submit empty → all four errors visible simultaneously (native bubbles can't do this — your system can), focus on email. Type into email → its error clears live, others stay. Password mismatch → confirm shows the custom message. Fix everything → submission proceeds (watch Network). If the errors also announce with NVDA/VoiceOver focus, you've built production-quality form UX from primitives.

5 (hard): Async + abort. Implement the username checker with: debounce, a fake API (setTimeout resolving randomly), AbortController-style stale-response protection (or a request counter), and a "checking…" pending state that never lies. Prove the race protection: make the first response slower than the second (two timeouts) and verify the UI shows the latest truth.
Reference core (counter version) 
let seq = 0;
async function check(value) {
  const my = ++seq;
  status.textContent = "checking…";
  const taken = await fakeApi(value);        // resolves after random delay
  if (my !== seq) return;                    // a newer check superseded us — drop it
  username.setCustomValidity(taken ? "Taken." : "");
  status.textContent = taken ? "✗ taken" : "✓ available";
}

The sequence-counter guard is the whole exercise: without it, the slow first response lands last and lies. This exact bug ships in real typeahead/validation UIs constantly; the guard (or AbortController) is the difference. Remember Q6 though — even correct client checks are advisory; the DB constraint is the truth.

6 (hard): The 422 bridge. Simulate a server rejecting a submission: on submit, preventDefault, fake a fetch that returns {errors: {email: "Already registered", handle: "Reserved word"}}, and render those server errors through your EXACT exercise-4 pipeline (setCustomValidity + showError), including focus-first. One paragraph: why routing server errors through the client pipeline matters.
Why it matters

Implementation: iterate the errors object, setCustomValidity(msg) on each named control, showError each, focus the first. The paragraph: users experience ONE error system — same placement, styling, announcement, and clearing behavior whether the rule ran locally or on the server; tests assert one path; and new server rules need zero client releases. The alternative — a separate toast/banner for server errors — trains users that errors come in two dialects, and screen-reader users often never find the second one. One pipeline, two sources.

🎯 Key Takeaways

  • One validation system, two doors: attributes declare, the API reads (validity, checkValidity) and extends (setCustomValidity) — never duplicate rules in JS.
  • ValidityState's flags turn "invalid" into specific, kind messages — branch on the reason.
  • setCustomValidity handles cross-field, group, and async rules; every check must set OR clear — "" is the clear.
  • Custom error UI = novalidate + aria-describedby error nodes + aria-invalid + focus-first + clear-on-input; server errors re-enter the same pipeline.
  • requestSubmit() validates and fires events; submit() skips both — a one-word difference with incident-report consequences.
  • Async checks: debounce, guard against stale responses, and remember they're advisory — the database constraint is the only race-free truth (TOCTOU).
  • Timing kindness: errors appear on blur/submit, disappear on input.
