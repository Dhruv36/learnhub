Document Structure & the <head> Foundations

What you'll master: every line of the document skeleton — what it does, what breaks without it, and how the browser interprets it — plus the full production-grade <head> used by real products. Prerequisite: How the Web & HTML Work (you should know what the DOM is and how to use the Elements panel).

Part 1 — The Skeleton, Line by Line

<!DOCTYPE html>                <!-- 1: parser mode switch -->
<html lang="en">               <!-- 2: root element + language -->
  <head>                       <!-- 3: machine-facing metadata -->
    <meta charset="UTF-8">     <!-- 4: byte decoding -->
    <title>Page Title</title>  <!-- 5: the only required head element -->
  </head>
  <body>                       <!-- 6: everything human-facing -->
  </body>
</html>

Six lines, and every one has a real job. Let's take them apart the way an interviewer would.

Line 1: <!DOCTYPE html> — the mode switch

This is not a tag and not "declaring the HTML version." It's a parser instruction with exactly one modern job: switching the browser into standards mode. History: 1990s pages were written against buggy browser behavior (especially IE5's wrong box model). When browsers fixed their bugs, old pages would have shattered — so browsers kept the bugs behind a flag. No doctype (or an ancient one) = quirks mode: the browser deliberately emulates ~20-year-old bugs. Doctype present = standards mode: the specs apply.

// Prove it in the Console on any page:
document.compatMode
// "CSS1Compat"  → standards mode ✅
// "BackCompat"  → quirks mode ⚠ find the missing/malformed doctype

What quirks mode actually breaks (so you recognize the symptoms): the box model changes (width includes padding/border), height: 100% chains behave differently, inline images get mysterious baseline gaps, font inheritance into tables breaks. The bug report reads "CSS is slightly wrong everywhere" — check compatMode first. The doctype must be the very first thing in the file; even a stray character before it can trigger quirks in old engines (a comment before it is safe in modern ones, but why gamble).

Line 2: <html lang="en"> — root + language

<html> is the root of the tree; everything else is its descendant. The lang attribute is the one beginners skip and professionals never do, because four systems read it:

  • Screen readers choose their pronunciation engine from it — English text read with French pronunciation rules is genuinely unintelligible. This makes missing lang a WCAG accessibility failure, not a style nit.
  • Browsers use it for hyphenation, quotation-mark styling (q element), and offering translation.
  • Search engines use it as a language signal.
  • CSS can target it: :lang(de).

Use BCP-47 codes: en, en-GB, hi, zh-Hans. Mark inline language switches too: <p>The German word <span lang="de"> Schadenfreude</span>...</p>.

Line 4: <meta charset="UTF-8"> — how bytes become characters

The file on disk is bytes. The browser must guess or be told which encoding maps those bytes to characters. Get it wrong and "café" renders as "caf├⌐" — the classic mojibake. UTF-8 encodes every character of every language (and emoji) and is the encoding of ~98% of the web; there is no modern reason to use anything else. Two rules:

  • It must appear within the first 1024 bytes of the document — the browser commits to an encoding early. Convention: make it the first thing in <head>.
  • Your editor must actually save the file as UTF-8 (every modern editor's default). The meta tag declares; the editor delivers. Mojibake with a correct meta tag = the file was saved as something else (often Windows-1252 from a legacy pipeline).
🔍 Senior detail: servers can also declare encoding via the Content-Type: text/html; charset=utf-8 HTTP header — and the header wins over the meta tag. A misconfigured server can therefore mojibake a perfectly-authored file; check the Network panel's response headers when debugging encoding issues.

Line 5: <title> — small tag, five jobs

The only element HTML requires inside <head>. It's used by: the browser tab, bookmarks, the task switcher, screen readers (the first thing announced on page load), and — critically — search results, where it's the blue headline link. Craft rules that the SEO lesson expands: unique per page, front-load the specific part (Order #1042 · MyShop, not MyShop — the best shop — Order), roughly under 60 characters or Google truncates it.

Lines 3 & 6: <head> vs <body>

The cleanest mental model: <head> is machine-facing (metadata, for the browser/search engines/social networks — renders nothing), <body> is human-facing (everything visible). If you put visible content in the head, the parser actually ends the head and starts the body right there — another silent repair that produces a baffling DOM.

Part 2 — Building the Production Head, Step by Step

Now we grow the minimal head into what real products ship. Add each block to your page from last lesson and verify the described effect.

Step 1: the viewport meta — why mobile pages aren't tiny

<meta name="viewport" content="width=device-width, initial-scale=1.0">

History you need to know: the iPhone (2007) had a 320px-wide screen but the web was designed for ~980px desktop layouts. Apple's fix: mobile browsers render pages into a fake 980px-wide viewport, then zoom out to fit — that's why old sites on phones look like shrunken desktop sites. This meta tag says "don't do that: make the layout viewport equal to the device width and start at 1:1 zoom." Without it, all your responsive CSS silently does nothing (media queries evaluate against the fake 980px). Test: remove it, open DevTools device emulation, watch your page shrink. Never ship without it; never add user-scalable=no or maximum-scale=1 — disabling pinch-zoom is an accessibility failure for low-vision users.

Step 2: description + icons + canonical

<meta name="description" content="Track your team's metrics in real time.">
<link rel="icon" href="/favicon.svg" type="image/svg+xml">
<link rel="apple-touch-icon" href="/icon-180.png">
<link rel="canonical" href="https://myapp.com/dashboard">
  • description — doesn't affect ranking directly, but it's the gray text under your search-result headline: it's ad copy that decides whether people click. ~155 chars.
  • icon — the favicon. SVG favicons are modern and scale everywhere; keep a favicon.ico at the site root as fallback because some tools still request it blindly. apple-touch-icon is the home-screen icon on iOS.
  • canonical — declares the "official" URL when several URLs serve the same content (with/without ?utm_source=..., http/https, trailing slash), so search engines merge ranking signals instead of splitting them. The SEO lesson covers strategy.

Step 3: styles and scripts — order is performance

<link rel="stylesheet" href="/css/main.css">   <!-- CSS in the head: render-blocking on purpose -->
<script src="/js/app.js" defer></script>      <!-- JS in the head WITH defer -->

Two rules with reasons:

  • CSS goes in the head because browsers refuse to paint until the CSS arrives (rendering unstyled content then restyling it — FOUC, flash of unstyled content — is worse). This makes CSS render-blocking: the performance lesson is largely about keeping this small.
  • Scripts get defer. A bare <script src> makes the parser stop dead, download, and execute before continuing (the parser must assume the script might document.write). defer downloads in parallel and runs after parsing, in order. async runs the moment it arrives, order not guaranteed — right for analytics, wrong for interdependent app code. The old "scripts at the bottom of body" advice is the pre-defer workaround; defer in the head is its modern replacement.

Step 4: the assembled production head

<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Dashboard · MyApp</title>
  <meta name="description" content="Track your team's metrics in real time.">
  <link rel="canonical" href="https://myapp.com/dashboard">
  <link rel="icon" href="/favicon.svg" type="image/svg+xml">
  <link rel="stylesheet" href="/css/main.css">
  <script src="/js/app.js" defer></script>
</head>

Order rationale: charset first (1024-byte rule), viewport before anything renders, title early (screen readers announce it), then metadata, then the render-blocking CSS, then deferred JS. Social cards (og: tags) and structured data join this head in the SEO lessons; preload hints join it in Performance.

Part 3 — The Body Skeleton (a preview of semantics)

Production bodies share a recognizable top-level shape — regions, not soup:

<body>
  <header>   site logo, main navigation           </header>
  <main>     THE content of THIS page (one per page) </main>
  <footer>   copyright, secondary links            </footer>
</body>

These landmark elements let screen-reader users jump straight to content ("skip to main") and make code navigable for humans too. Section 3 of this track gives them a full lesson — for now, start every practice page with this shape instead of bare content, so the habit forms early.

Part 4 — Deep Dive: What the Parser Does With Your Skeleton

Facts that explain otherwise-mysterious behavior, and that separate senior answers from junior ones:

  • <html>, <head>, and <body> tags are all technically optional. The parser creates the elements anyway — a file containing only <title>Hi</title><p>Hello yields a complete, valid tree. That's why document.body always exists. Write the tags anyway: explicit structure is for humans, and lang needs somewhere to live.
  • The parser is streaming. It builds the DOM as bytes arrive over the network and can paint the top of the page before the bottom has even been sent. This is why HTML "just works" on slow connections, why servers can stream (flush the head early so CSS starts downloading while the server is still computing the page body), and part of why blocking scripts hurt so much — they stall this pipeline.
  • Whitespace mostly collapses. Any run of spaces/tabs/newlines between words renders as a single space, which is why your indentation is free. Exceptions: inside <pre>, and with CSS white-space. One classic gotcha: inline-block elements separated by a newline in source get a visible gap — a whitespace text node between them.
  • Comments — <!-- like this --> — become comment nodes in the DOM and are sent to every visitor. Never leave secrets, internal URLs, or "TODO: fix auth bypass" in them; attackers read HTML comments as reconnaissance, routinely.

Interview Questions

Q1 (beginner): What does <!DOCTYPE html> do?

It switches the parser into standards mode. Without it the browser uses quirks mode, deliberately emulating 1990s bugs (notably the old box model) for backward compatibility. It is not a tag and no longer declares a "version" — HTML5 chose the shortest string that triggers standards mode. Verify a page's mode with document.compatMode ("CSS1Compat" = standards).

Q2 (beginner): What belongs in <head> vs <body>?

Head = machine-facing metadata that renders nothing: charset, viewport, title, description, icons, canonical, stylesheets, (deferred) scripts. Body = all human-visible content. Bonus point: visible content placed in the head causes the parser to close the head and open the body early — a silent repair.

Q3 (mid): Compare script loading: bare src vs defer vs async vs module.

Bare: parser halts, downloads, executes, resumes — worst for performance, kept for legacy (document.write). defer: downloads in parallel, executes after parsing in document order, before DOMContentLoaded — the default for app code. async: executes the instant it arrives, any order, may interrupt parsing — for independent scripts like analytics. type="module": deferred semantics by default, plus module scope and imports. Strong answers mention why bare scripts block: the parser must assume document.write could rewrite the page.

Q4 (mid): A designer reports the mobile site "shows the desktop layout shrunken down." Diagnose.

Missing viewport meta tag. Without it, mobile browsers lay the page out in a legacy ~980px viewport and scale it down, so @media (max-width: ...) queries never match the device's real width. Fix: <meta name="viewport" content="width=device-width, initial-scale=1.0">. Follow-up worth volunteering: never add user-scalable=no while you're in there — zoom is an accessibility requirement.

Q5 (senior): Page renders "café" as "caf├⌐" in production but fine locally. The meta charset tag is present and correct. What's your diagnostic path?

The HTTP Content-Type header overrides the meta tag — check the Network panel: if production's server/CDN sends charset=iso-8859-1 (or the file was transcoded by a deploy pipeline), you get mojibake despite correct markup. Diagnostics: (1) response header charset, (2) actual bytes on the server (was the file re-saved/transcoded?), (3) meta within the first 1024 bytes, (4) database/API layer if the text is dynamic — mojibake often enters at a non-UTF-8 database connection, and then it's stored corrupted (fixing display won't fix the data). The "works locally" clue points at infrastructure (headers/pipeline), not markup.

Q6 (senior): Why is HTML parsing streaming, and what production techniques does that enable?

The parser builds and renders the DOM incrementally as bytes arrive — a design forced by 1990s modems and kept because it's a massive perceived-performance win. It enables: early paint of above-the-fold content; early flush (server sends the <head> immediately so the browser starts fetching CSS while the server still queries the database); modern streaming SSR (React 18's renderToPipeableStream sends shell first, suspended content later — same principle); and the preload scanner, which speculatively fetches resources found ahead of the parser even while a script blocks it. The cost: anything that stalls the stream (sync scripts, slow TTFB on a monolithic response) delays everything downstream — which is why the Performance lesson treats the head as the critical path.

🏋️ Exercises

1 (easy): Skeleton from memory. Type the full production head (Part 2, Step 4) from memory, then diff against the lesson. Repeat tomorrow — this exact block starts every project of your career.
Self-check

All eight lines present? Charset first? defer on the script? If you missed the canonical or description, fine on round one — those are per-page; the other six are non-negotiable boilerplate.

2 (easy): Quirks-mode safari. Take your coffee page, delete the doctype, and set a border + padding + fixed width on a paragraph via a style tag. Compare its rendered width in quirks vs standards mode. Confirm modes with document.compatMode.
What you should see

Standards mode: rendered box = width + padding + border (content-box). Quirks: the old IE model — padding and border squeeze inside the declared width, so the box is visibly narrower. document.compatMode reads "BackCompat" without the doctype. This one experiment makes the doctype unforgettable.

3 (medium): Viewport before/after. Serve a page with a paragraph of text and this CSS: p { font-size: 16px } @media (max-width: 600px) { p { color: red } }. View in DevTools device emulation (e.g. iPhone SE) with and without the viewport meta. Explain both observations in writing.
Explanation

Without the meta: layout viewport is ~980px, so the max-width:600px query doesn't match even on a 375px device — text stays black and the page appears zoomed out (16px text looks tiny). With it: viewport = 375px, query matches, text is red and readable at natural size. Moral: responsive design is dead on arrival without this tag.

4 (medium): Script-timing lab. Create three tiny scripts that console.log their name plus performance.now(). Load one bare in the head, one with defer, one with async, plus an inline log at the very end of the body. Predict the order before opening the console, then explain any surprise.
Expected order & why

Bare head script first (parser stops for it), then end-of-body inline (parser reached it), then defer (after parsing completes), while async lands anywhere after its download — run it several times (and with DevTools network throttling) to watch async move around. That movement is the whole argument for not using async for order-dependent code.

5 (hard): Mojibake forensics. In VS Code, save a page containing "café — naïve résumé" as Windows-1252 ("Save with encoding"), while keeping the UTF-8 meta tag. Open it. Then fix the meta to match, reload, then re-save as UTF-8 with the original meta. Document what rendered each time and why.
Explanation

Round 1: bytes are 1252, decoder is UTF-8 → é's single 1252 byte isn't valid UTF-8 → � or mangled pairs. Round 2: declared 1252, bytes 1252 → renders correctly (declaration matches reality — the tag is a claim, not magic). Round 3: UTF-8 bytes + UTF-8 claim → correct, and the only configuration you should ever ship. Deep lesson: encoding bugs are always a mismatch between claim and bytes; either side can be the liar.

6 (hard): Audit three real sites. View-source on any three major sites (a newspaper, a web app, a shop). For each, find: doctype, lang, charset, viewport, title, description, canonical, how scripts load (count bare vs defer vs async vs module). Note one thing each site does that this lesson didn't cover, and look it up on MDN.
What you'll likely find

All three: standard doctype, UTF-8, viewport. Likely finds you haven't met yet: rel="preconnect"/preload (Performance lesson), og: meta (Social Cards lesson), rel="alternate" hreflang (international SEO), nonce= on scripts (CSP, Security lesson), and possibly a huge inlined <style> block (critical CSS). Treat every unknown as a pointer to a later lesson — this audit is a preview of the whole track.

🎯 Key Takeaways

  • The doctype is a parser mode switch — omit it and you inherit 1990s layout bugs (quirks mode; check document.compatMode).
  • lang on <html> is an accessibility requirement (screen-reader pronunciation), not decoration.
  • Charset meta within 1024 bytes + file actually saved as UTF-8 + server header agreeing = no mojibake; the HTTP header outranks the meta tag.
  • The viewport meta is what makes responsive design possible on mobile; never disable user zoom.
  • CSS is render-blocking by design (in the head); scripts get defer by default, async only when independent.
  • The parser streams, repairs, and auto-creates html/head/body — write them anyway, for humans and for lang.
  • HTML comments ship to every visitor — never leave secrets in them.
