Document Structure Basics

The difference between a toy page and a production page is almost entirely in the <head> and the top-level skeleton of the <body>. This lesson dissects both, line by line, exactly as shipped by real products.

The Complete Production Head

<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Dashboard · MyApp</title>
  <meta name="description" content="Track your team's metrics in real time.">

  <link rel="icon" href="/favicon.ico" sizes="32x32">
  <link rel="icon" href="/icon.svg" type="image/svg+xml">
  <link rel="apple-touch-icon" href="/apple-touch-icon.png">

  <link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>
  <link rel="stylesheet" href="/css/app.css">
  <script src="/js/app.js" type="module"></script>
</head>
ElementDeep detail you're expected to know
viewportWithout it, mobile browsers assume a 980px page and zoom out; media queries never fire. width=device-width = use the real screen width; initial-scale=1 = no default zoom. Never add user-scalable=no — it blocks pinch-zoom (accessibility failure).
titlePattern: "Page · Product" keeps the unique part visible in narrow tabs. 50–60 chars — Google truncates beyond that. Unique per page — duplicate titles are an SEO red flag.
description~155 chars shown under the Google link. Doesn't affect ranking, strongly affects click-through.
IconsModern set: .ico fallback (32px), SVG (scales, supports dark-mode media queries inside!), apple-touch-icon (180px PNG for iOS home screens).
CSS <link>Render-blocking by design (prevents a flash of unstyled content). This is why CSS goes in head — and why it must be small (HTML Performance lesson).
type="module" scriptModules defer automatically: download in parallel, execute after parsing. Legacy pattern defer attribute achieves the same for non-modules. A bare <script src> in head halts parsing — the classic performance bug.

Script Loading — the Three Behaviors, Visualized

<script src>              parse ────■■■ STOP: download+run ■■■──── resume parse
<script defer / module>   parse ───────────────────────────────── end, THEN run (in order)
<script async>            parse ────────■■ run whenever it lands ■■────── (any order!)

Rules of thumb: app code → module/defer · independent trackers → async · body-end scripts → legacy habit, defer is better

The Body Skeleton — Landmarks

<body>
  <a class="skip-link" href="#main">Skip to content</a>
  <header>
    <nav aria-label="Main">…</nav>
  </header>
  <main id="main">
    …the content unique to THIS page…
  </main>
  <footer>…</footer>
</body>
RuleWhy
Exactly one <main>Screen-reader users jump to it with one keypress; multiple mains defeat that
header/footer can repeatPage-level once, but also valid inside each <article>
Label repeated landmarksTwo <nav>s? aria-label="Main" / aria-label="Breadcrumbs" distinguishes them when announced
Skip link first in bodyKeyboard users escape 40 nav links; visually hidden until focused

Where Metadata Lies to You — Base URLs & Favicons

<base href="https://cdn.example.com/v2/">   <!-- rewrites how EVERY relative URL resolves -->

<base> is powerful and dangerous: one tag silently changes every relative link, image, and script path on the page. Some SPA routers require it; most sites should never use it. If a page's relative links all mysteriously 404 — check for a <base> tag first.

Interview Questions

Q1: defer vs async — when does each run, and which preserves order?

Both download in parallel with parsing. defer executes after parsing completes, in document order — right for app code with dependencies. async executes the moment it arrives, in arrival order — only for scripts with no dependencies (analytics). Modules are deferred by default; async on a module makes it run ASAP after fetch.

Q2: Why is CSS render-blocking and JS parser-blocking?

CSS blocks rendering (not parsing) to avoid painting unstyled content then repainting. Classic JS blocks parsing because scripts can document.write or query the DOM built so far — the parser can't safely continue until the script runs. defer/async/module are the opt-outs that promise the parser it's safe.

Q3: What's the minimal valid HTML5 document?

Technically <!DOCTYPE html><title>x</title> parses to a full tree — html/head/body tags are optional per spec (the parser synthesizes them). Production answer: write them explicitly anyway; implicit structure confuses humans and some tools.

🏋️ Practice:
1. In the Playground, build a page with skip-link → header/nav → main → footer, and verify the skip link works with Tab + Enter.
2. Add <script>alert(1)</script> first in head, reload, and observe when the alert fires vs. when content appears. Move it to a defer-style module and compare.
3. View-source three real sites (GitHub, Wikipedia, a news site) and identify every head element you now recognize.

🎯 Key Takeaways

  • The head is a contract with browsers, search engines, and social networks — every line has a consumer.
  • Script loading: module/defer for app code, async only for independent scripts, bare src never.
  • Landmarks (header/nav/main/footer) + skip link = keyboard & screen-reader navigation for free.
  • One <main>, labeled repeated landmarks, viewport without zoom-blocking.
