Selects, Datalist, Files & More Forms Mastery

What you'll master: the non-input controls — select/option/optgroup and their real-world pitfalls, textarea's quirks, datalist (the free autocomplete nobody uses), output and progress/meter, and the select-vs-radio-vs-datalist decision framework. Prerequisites: the previous three forms lessons.

Part 1 — select: More Depth Than It Looks

<label for="roast">Roast level</label>
<select id="roast" name="roast">
  <option value="">— Choose a roast —</option>      <!-- placeholder option -->
  <option value="light">Light</option>
  <option value="medium" selected>Medium</option>
  <option value="dark">Dark</option>
</select>
  • value vs text: the user sees the option's text; the server receives its value. Omit value and the text submits — fine for simple cases, but explicit values decouple display copy from your data model (rename "Medium" to "Classic" without a backend change).
  • The placeholder-option pattern + required: give the placeholder value="" and mark the select required — empty value fails valueMissing, so "— Choose —" can't be submitted. Without value="", the placeholder text itself submits: a classic data-pollution bug ("why do 4% of orders have roast='— Choose a roast —'?"). Add disabled to it to prevent re-selecting the placeholder after a real choice.
  • optgroup adds non-selectable group headers: 
    <select name="bean">
  <optgroup label="Africa">
    <option value="yirg">Ethiopia Yirgacheffe</option>
    <option value="kii">Kenya Kii</option>
  </optgroup>
  <optgroup label="Americas">
    <option value="huila">Colombia Huila</option>
  </optgroup>
</select>
  • multiple + size: <select multiple size="6"> allows several selections (submits the name repeatedly: bean=yirg&bean=huila) — but requires Ctrl/Cmd-clicking, which most users don't know. UX verdict: a checkbox group beats multi-select almost always; multi-select survives mainly in dense admin UIs.
  • Styling reality: the closed select styles decently; the open dropdown is OS-rendered and barely stylable. This limitation birthed a thousand div-based fake dropdowns — which then need ARIA, keyboard handling, and virtual scrolling reimplemented (badly, usually). New CSS capabilities (appearance: base-select, customizable select) are landing to fix this natively — check current support before building a fake one.

Part 2 — The Decision Framework: radio vs select vs datalist vs checkbox

SituationUseWhy
2–5 options, one choice, space availableRadio groupAll options visible — zero interaction cost to scan; comparison is the task
6–15 options, one choiceselectCompact; scanning cost acceptable
Huge list but arbitrary input allowedtext + datalistSuggestions without restriction (Part 3)
Huge list, restricted (countries, currencies)select (or combobox library if search is essential)Restriction is the requirement; search is the UX rescue
Multiple choicesCheckbox groupVisible, familiar, no hidden modifier keys
Binary on/offSingle checkboxDon't build a Yes/No select — it doubles the clicks

The principle underneath: interaction cost vs. screen cost. Radios spend screen space to save interaction; selects spend interaction (open, scan, choose) to save space. Mobile tilts toward select/native pickers (the OS wheel is excellent); desktop with few options tilts toward radios. And the famous cautionary tale: a Yes/No select is three taps where a checkbox is one.

Part 3 — datalist: Autocomplete Without JavaScript

<label for="dest">Destination city</label>
<input id="dest" name="dest" list="cities">
<datalist id="cities">
  <option value="Portland"></option>
  <option value="Porto"></option>
  <option value="Port-au-Prince"></option>
</datalist>

The input stays free text; the datalist supplies suggestions that filter as the user types. That's the crucial semantic difference from select: datalist suggests, select restricts. The user can type "Munich" even though it's not listed — so the server must treat the value as free text (validate accordingly!). Great for: known-common-but-not-exhaustive values (cities, job titles, tags). Limitations to know: the dropdown UI is unstylable and varies across browsers, option label display differs annoyingly between them, and there's no "loading" state — for API-driven suggestions at scale, teams graduate to a combobox component (ARIA lesson covers what that must implement). Bonus: datalist also attaches to range inputs to render tick marks.

Part 4 — textarea: the Multi-line Field and Its Quirks

<label for="notes">Delivery notes</label>
<textarea id="notes" name="notes" rows="4" maxlength="500"
          placeholder="Gate code, safe place, dog's name…"></textarea>
  • Not a void element — its content is its initial value, and whitespace counts: <textarea> indented text</textarea> ships those spaces/newlines as the value. Write open-close tags flush together for empty textareas.
  • There is no value attribute — server-rendered values go between the tags (escaped! a textarea containing user content is an XSS sink like any other — remember lesson 3's pre discussion; textarea IS one of the raw-text parsing modes, and the escape ambush is </textarea> appearing in the content).
  • rows sets visible height (CSS can override); users can drag-resize by default — constrain direction with CSS resize: vertical (never none without reason; taking away resize on a long-text field is hostile).
  • Auto-growing textareas (height follows content) are a JS/CSS pattern — field-sizing: content is the arriving pure-CSS answer; check support.
  • minlength/maxlength/required all work; remember maxlength's silent paste-truncation from the validation lesson — pair long limits with a visible counter.

Part 5 — The Read-back Controls: output, progress, meter

<!-- output: a calculation result, semantically tied to its inputs -->
<form oninput="total.value = (qty.valueAsNumber * 14.50).toFixed(2)">
  <label for="qty">Bags</label>
  <input type="number" id="qty" name="qty" value="1" min="1">
  <p>Total: $<output name="total" for="qty">14.50</output></p>
</form>

<progress max="100" value="70">70%</progress>     <!-- task completion; omit value = indeterminate -->
<meter min="0" max="10" low="3" high="7" optimum="9" value="8">8/10</meter>  <!-- a gauge -->
  • output is the semantic home for computed values — screen readers treat it as a live region (changes get announced). The for attribute lists the ids of inputs it derives from. Use it for totals, sliders' current values, character counters.
  • progress = how done is a task (upload, wizard). Without a value it renders indeterminate ("something's happening"). meter = a measurement on a known scale (disk usage, password strength, rating) — its low/high/optimum drive automatic color-coding. Choosing progress for a gauge (or meter for a task) is a semantics smell.

Part 6 — Worked Build: the Order Form, Upgraded

Take lesson 9's order form and upgrade it with this lesson's controls — the diff that matters:

<label for="bean">Bean</label>
<select id="bean" name="bean" required>
  <option value="" disabled selected>— Choose your bean —</option>
  <optgroup label="Africa">…</optgroup>
  <optgroup label="Americas">…</optgroup>
</select>

<label for="ref">How did you hear about us?</label>
<input id="ref" name="ref" list="ref-list">          <!-- suggest, don't restrict -->
<datalist id="ref-list">
  <option value="Friend"></option><option value="Instagram"></option>
  <option value="Farmers market"></option>
</datalist>

<label for="notes">Delivery notes</label>
<textarea id="notes" name="notes" rows="3" maxlength="500"></textarea>
<p><output id="counter" for="notes">0</output>/500</p>

<label for="qty2">Bags</label>
<input type="number" id="qty2" name="qty" value="1" min="1" max="20">
<p>Subtotal: $<output name="subtotal" for="qty2">14.50</output></p>

Submit to httpbin and read the payload: the select's value (not its label), the datalist field as free text, the textarea's exact content newlines included. The wire format never changes — every control in this lesson still reduces to name=value pairs.

Interview Questions

Q1 (beginner): select vs datalist — the key difference?

select restricts: the user must pick a provided option, and the server can trust the value set (still validate!). datalist suggests: the input remains free text with autocomplete hints, so anything can arrive. Choose by whether "other" answers are acceptable. Bonus: datalist's UI is unstylable and inconsistent — fine for enhancement, wrong as a critical widget.

Q2 (beginner): Why does the classic placeholder option need value=""?

required validates against the value: with value="", selecting the placeholder counts as valueMissing and blocks submission. Without it, the option's text becomes its value and "— Choose a roast —" submits as legitimate data. Adding disabled prevents users returning to the placeholder after choosing; selected makes it the initial state.

Q3 (mid): When do radios beat a select, and what's the underlying principle?

Radios for 2–5 options where comparison matters and space exists: all options visible at zero interaction cost, one click to choose, group scanned at a glance. Select for longer lists where screen economy wins. Principle: interaction cost vs screen cost — radios pay pixels to save clicks; selects pay clicks to save pixels. Context modifiers: mobile native pickers are excellent (tilts select); binary choices are checkboxes, never either.

Q4 (mid): A form pre-fills a textarea with the user's saved bio. List everything the template layer must get right.

(1) The value goes between the tags (no value attribute exists); (2) HTML-escape it — textarea is raw-text parsed, and the specific breakout string is </textarea> in the content ending the element and turning the rest into live HTML (stored XSS); (3) no incidental indentation inside the tags (it becomes content); (4) maxlength consistent with the DB column so saved bios don't get paste-truncated on edit; (5) newlines survive round-trips (they're submitted as-is — normalize CRLF server-side).

Q5 (senior): Product wants a searchable, styled, 200-option "country + flags" dropdown. Walk through your build-vs-native decision and what a custom combobox must implement.

Interrogate the requirement first: native select gives search-by-typing, perfect mobile pickers, AT support, zero bytes — flags can render adjacent to it, and modern customizable-select CSS may cover the styling ask natively (check support). If a true combobox is genuinely required: use a battle-tested library (Downshift/React-Aria/etc.) or implement the FULL ARIA combobox pattern — role=combobox, aria-expanded/-activedescendant, listbox+option roles, complete keyboard set (arrows, Home/End, Esc, Enter, type-ahead), focus management, touch behavior, virtualized options for 200+ — plus a mobile fallback strategy (many teams swap in the native select on touch). The senior signal is cost honesty: a correct custom combobox is weeks, not days, and the common div-dropdown snippet is an accessibility incident, not a component.

Q6 (senior): Your analytics show the "How did you hear about us?" datalist field's top values include misspellings and synonyms ("insta", "IG", "instagram"). Discuss the data-design tradeoff you made and remediation.

The tradeoff was intrinsic: datalist chose expressiveness (capture unforeseen answers) over cleanliness (bounded categories). Remediations: (1) normalize server-side — mapping table folding known variants, applied at ingest AND retroactively (keep the raw value in a column; never destroy source data); (2) tighten the UX without losing expressiveness — case-insensitive suggestion matching, reorder suggestions by popularity; (3) if analysis needs categories more than discovery needs free text, graduate to select + "Other (explain)" — the standard two-tier design; (4) measure: fraction of free-text answers that map to existing options tells you which way to go. The meta-point: control choice IS schema design — the front-end widget determines your data's shape downstream.

🏋️ Exercises

1 (easy): The grouped select. Build the bean select with two optgroups, a correct disabled/selected/value="" placeholder, and required. Verify: (a) submitting untouched blocks with valueMissing, (b) the payload carries the value not the label, (c) the placeholder can't be re-selected after choosing.
Check

All three behaviors come from the placeholder trio: value="" (validation), disabled (unreselectable), selected (initial). If (a) doesn't block, your placeholder has a non-empty value — the exact bug from the mistakes table.

2 (easy): Widget-choice drill. Pick the control (and defend it in one line): (a) T-shirt size S/M/L/XL, (b) country of residence, (c) pizza toppings, (d) newsletter opt-in, (e) job title, (f) satisfaction 1–5 with labels.
Answers

(a) radios — 4 options, comparison visible. (b) select — long restricted list. (c) checkbox group — multiple. (d) single checkbox — binary. (e) text + datalist — suggest common titles, allow anything. (f) radios — 5 labeled points need visibility (a select hides the scale's shape). Defensible alternates exist for (a) and (f) on cramped mobile — if you argued interaction-vs-screen cost, full marks.

3 (medium): Live character counter. Wire the notes textarea to an output-based counter (0/500) that updates on input and turns the last 50 characters into a warning state (class swap). Bonus: announceable — confirm the output updates are polite, not per-keystroke chatty (hint: update the text only every 10 chars, or on threshold crossings).
Reference core 
notes.addEventListener("input", () => {
  const n = notes.value.length;
  counter.value = n;                       // output.value updates its text
  counter.parentElement.classList.toggle("warn", n >= 450);
});

The bonus matters: output is a live region, so screen readers announce changes — per-keystroke announcements ("47… 48… 49…") are torture. Throttling announcements to thresholds (450, 490, 500) while updating visually every keystroke is the refined pattern: two channels, two cadences.

4 (medium): datalist reality check. Build the referral field with a datalist. Type a value NOT in the list and submit — confirm it goes through. Then try to "fix" that with pattern restricting to listed values, and write two sentences on why that fix is wrong (and what the right change is if restriction is truly needed).
Answer

The pattern "fix" turns suggestions into secret restrictions: users see a free-text field but get rejected for free text — the worst of both widgets, with a baffling error. If values must be bounded, the honest control is a select (the UI then communicates the restriction); if discovery matters, keep datalist and normalize server-side. Widget semantics and validation must tell the same story.

5 (hard): progress vs meter, used correctly. Build a fake file upload: a button starts a setInterval that advances a progress element 0→100 (then completes), plus a "storage used" meter (with low/high/optimum showing color changes as you adjust its value via a range input). One paragraph: why swapping the two elements would be wrong even though both "show a bar".
Paragraph core

progress states "a task is this fraction done" — AT announces it as a progressbar, indeterminate mode exists, and completion is an expected terminal state. meter states "a measurement sits here on a known scale" — no task, no completion, and its low/high/optimum semantics drive good/bad coloring AT can convey. A storage gauge as progress claims storage is "finishing"; an upload as meter loses indeterminate + progressbar announcements. Same pixels, different claims — the whole track in one pair of elements.

6 (hard): The full upgraded order form. Assemble Part 6 completely (select+optgroups, datalist, counted textarea, output subtotal), submit to httpbin, and verify the payload byte by byte: select value, free-text referral, textarea newlines, number as string. Then break it deliberately: put whitespace inside the textarea tags and re-inspect the payload.
What to verify

Payload: bean=huila (value, not "Colombia Huila"), ref exactly as typed, notes with %0A newlines (URL-encoded), qty=3 (all form values are strings on the wire — servers cast). The whitespace break: your indentation arrives as leading spaces/newline in notes — invisible in the UI, present in the data; the kind of bug that survives for years until someone asks why every note starts with "\n ". Flush tags fix it.

🎯 Key Takeaways

  • select restricts, datalist suggests — the widget choice IS a data-schema decision, and validation must match the story the UI tells.
  • Placeholder options: value="" + disabled + selected + required on the select — or your placeholder text becomes production data.
  • Radios for ≤5 visible-comparison choices, checkboxes for multiples and binaries, select for long restricted lists; multiple-select is a desktop-admin relic.
  • textarea: value between the tags, whitespace-sensitive, no value attribute, raw-text XSS sink (escape; beware </textarea> breakout).
  • output for computed values (announced live region), progress for task completion, meter for measurements — three read-back elements, three claims.
  • Native selects over div-dropdowns until requirements demonstrably exceed them — then a proven combobox implementation, never a bare div.
