Social Cards & Structured Data SEO

What you'll master: Open Graph and Twitter Card tags (why a shared link shows a rich preview instead of a bare blue URL), JSON-LD structured data and the rich-results it unlocks (star ratings, recipe cards, FAQ dropdowns directly in search results), and how to validate both before shipping. This closes the SEO section. Prerequisites: SEO & Meta Tags.

Part 1 — Why Pasted Links Sometimes Show Rich Previews

Paste a link into Slack, iMessage, Twitter/X, or Facebook and — for a well-configured page — you get a rich card: a headline, description, and image, instead of a bare URL. This isn't magic or a built-in browser feature; it's the receiving platform fetching your page and reading a SPECIFIC set of meta tags you control, entirely separate from the SEO tags from the last lesson. Two competing- but-largely-compatible standards cover this: Open Graph (Facebook-originated, now the de facto universal standard most platforms read) and Twitter Cards (X's own variant, with some Twitter-specific refinements). Ship both — they're cheap and largely overlapping.

Part 2 — Open Graph: the Universal Layer

<meta property="og:title" content="Aria X2 Wireless Headphones">
<meta property="og:description" content="30-hour battery, adaptive noise cancellation, studio-tuned sound.">
<meta property="og:image" content="https://shop.example/img/aria-x2-social.jpg">
<meta property="og:url" content="https://shop.example/products/aria-x2">
<meta property="og:type" content="product">
<meta property="og:site_name" content="SoundCraft">

Note the attribute is property, not name — a real, easy-to-typo detail (Open Graph predates and deliberately diverges from HTML's native meta convention; browsers don't care about the distinction, but some strict parsers/validators do, so get it right). Field-by-field:

  • og:title / og:description — usually similar to your SEO title/description (last lesson) but genuinely allowed to differ: search-result copy and social-share copy serve different contexts and different reader intents, and treating them as identical is a missed opportunity, not a shortcut.
  • og:image — the highest-leverage field by far. A card with no image (or a broken one) gets dramatically less engagement than one with a properly sized image — this is measured, not folklore. Must be an absolute URL (relative paths don't resolve correctly when a remote platform's server fetches your page), and sized correctly: 1200×630px is the standard recommendation, matching the ~1.91:1 aspect ratio most platforms crop to. Undersized images get stretched and blurry; wildly-off-ratio images get cropped unpredictably by each platform's own algorithm.
  • og:url — the canonical URL for the shared content, working alongside (not instead of) rel="canonical" from the last lesson — same underlying idea (one authoritative URL), different consuming systems.
  • og:type — a semantic hint (article, product, video, website) that some platforms use to render type-specific card layouts.

Part 3 — Twitter Cards: the Refinement Layer

<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="Aria X2 Wireless Headphones">
<meta name="twitter:description" content="30-hour battery, adaptive noise cancellation.">
<meta name="twitter:image" content="https://shop.example/img/aria-x2-social.jpg">

Note these use name, not property — the standards genuinely diverge on this exact attribute (worth committing to memory, since it's a common source of validator warnings when copy-pasted carelessly between the two blocks). twitter:card's value controls the LAYOUT: summary (small square image, compact) versus summary_large_image (full-width hero image — almost always the better choice for engagement). In practice, X/Twitter FALLS BACK to your Open Graph tags for any twitter:* field you've omitted — so the minimum viable Twitter-specific addition is often just the single twitter:card line declaring your preferred layout, letting og:title/description/image fill the rest.

Part 4 — JSON-LD: Structured Data for Rich Results

A separate, more ambitious system: telling search engines not just "here's a title and description" but "here's the precise, machine-readable STRUCTURE of this content" — a recipe's ingredients and cook time, a product's price and star rating, an FAQ's actual question/answer pairs. This is what unlocks rich results: the star-rating snippets, recipe cards with photos, and expandable FAQ dropdowns you've seen directly in Google search results, standing out visually against plain blue-link competitors.

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Product",
  "name": "Aria X2 Wireless Headphones",
  "image": "https://shop.example/img/aria-x2.jpg",
  "description": "30-hour battery, adaptive noise cancellation, studio-tuned sound.",
  "brand": { "@type": "Brand", "name": "SoundCraft" },
  "offers": {
    "@type": "Offer",
    "price": "249.00",
    "priceCurrency": "USD",
    "availability": "https://schema.org/InStock"
  },
  "aggregateRating": {
    "@type": "AggregateRating",
    "ratingValue": "4.6",
    "reviewCount": "1284"
  }
}
</script>

Mechanically distinct from every other meta tag in this lesson: it's a <script> tag with type="application/ld+json", containing plain JSON, following the schema.org vocabulary — a shared, standardized set of type definitions (Product, Recipe, Article, FAQPage, Event, Organization, and dozens more) that search engines have agreed to interpret consistently. It doesn't render visually and doesn't affect page behavior at all — its sole purpose is machine consumption by search-engine parsers looking specifically for this block.

🔍 Why JSON-LD over the two alternative structured-data formats (Microdata, RDFa)? Both older alternatives require weaving structured-data attributes directly into your visible HTML elements (itemscope/itemprop scattered through your markup) — messy, fragile, and tightly coupling your content structure to your data-markup structure. JSON-LD is a single, separate, self-contained block — add it, remove it, or regenerate it entirely independent of your visible HTML. Google explicitly recommends JSON-LD as the preferred format for exactly this separation-of-concerns reason, and it's what the overwhelming majority of real sites use today.

A second worked example: FAQ structured data

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [{
    "@type": "Question",
    "name": "What's your return policy?",
    "acceptedAnswer": {
      "@type": "Answer",
      "text": "Full refund within 30 days, no questions asked."
    }
  }]
}
</script>

This is the machine-readable twin of an actual visible FAQ built with <details>/<summary> (lesson 15) — the JSON-LD doesn't replace real, visible, accessible content; it DESCRIBES content that must already exist, visibly, on the page. This is the section's most important rule, stated explicitly next.

Part 5 — The Non-Negotiable Rule: Structured Data Must Match Visible Content

Google's guidelines are explicit and actively enforced: structured data describing content that ISN'T actually visible on the page — a fabricated 5-star rating with no real reviews anywhere on the page, an FAQ block whose questions don't appear in the visible HTML at all — is a policy violation that can result in a manual action (a real, human-reviewed penalty) stripping rich-result eligibility site-wide, not just for the offending page. This directly connects back to the last lesson's Q6 (thin/manipulative content risk): structured data is powerful precisely because search engines trust it as an accurate machine-readable summary of real page content — abuse that trust and the entire mechanism gets revoked for your domain. The correct mental model: JSON-LD is a translation layer for content that genuinely exists and is genuinely visible, never a shortcut to claiming things that aren't true.

Part 6 — Validation: Never Ship Unvalidated

Open Graph / Twitter Cards:
  → Facebook Sharing Debugger (developers.facebook.com/tools/debug) — also FORCES a re-scrape,
    since platforms cache og: data aggressively; a stale cached preview after you fix a bug is
    a real, common confusion until you know this tool exists
  → Twitter Card Validator (for X-specific rendering)

Structured Data:
  → Google's Rich Results Test (search.google.com/test/rich-results) — shows exactly which
    rich-result types your page is eligible for, and flags missing REQUIRED fields per type
  → Schema.org's own validator for pure spec-compliance checking

Both validation categories matter for the same underlying reason: these are entirely invisible systems from a normal browser preview — your page can look completely correct in Chrome while silently shipping malformed JSON-LD (a single missing comma breaks the ENTIRE block, often silently) or an og:image path that 404s when Slack's server (not your browser, which has different caching/auth) tries to fetch it. Treat both validators as a required step before every deploy touching these tags, not an optional nice-to-have — the failure mode (a broken preview, or zero rich-result eligibility) is invisible until someone actually shares the link or searches for the page, which is exactly the wrong time to discover it.

Interview Questions

Q1 (beginner): What makes a pasted link show a rich preview card instead of a bare URL, mechanically?

The receiving platform (Slack, iMessage, Twitter/X, etc.) fetches the shared page and reads specific Open Graph (og:*) and/or Twitter Card (twitter:*) meta tags in its head — it's not a browser feature, it's each platform's own server-side fetch-and-parse process reading tags you explicitly control, entirely separate from the search-engine-facing SEO tags from the previous lesson.

Q2 (beginner): Why must og:image always be an absolute URL, never relative?

Social platforms' servers fetch your page independently of any browser context — a relative path like /img/photo.jpg has no base URL to resolve against from their fetching server's perspective, so it fails to load. Always https://full-domain.com/img/photo.jpg, unlike some other paths in your document that browsers happily resolve relative to the current page.

Q3 (mid): What is JSON-LD, mechanically, and why does Google recommend it over Microdata/RDFa?

A script tag with type="application/ld+json" containing plain JSON following the schema.org vocabulary — entirely separate from and non-interfering with your visible HTML, versus Microdata/RDFa which require scattering itemscope/itemprop-style attributes directly through your visible markup. JSON-LD's separation of concerns (structured data lives in one self-contained block, addable/removable/regeneratable independent of visible content structure) is why Google explicitly prefers it and why the majority of real-world sites use it today.

Q4 (mid): A team ships a Product page with JSON-LD claiming a 4.8-star aggregate rating, but the visible page has no reviews or ratings shown anywhere. What's the risk?

A direct policy violation of Google's structured-data guidelines — structured data must accurately describe content that's genuinely visible on the page, and a fabricated/unsupported rating is exactly the kind of misrepresentation that triggers a manual action, a real human-reviewed penalty that can strip rich-result eligibility across the ENTIRE domain, not just the offending page. The fix isn't a markup tweak — it's either adding genuine, visible reviews that support the claimed rating, or removing the aggregateRating block entirely until real review data exists.

Q5 (senior): Your team ships a content fix to a heavily-shared article page — updated title, updated og:image — but three days later, links shared in Slack still show the OLD title and image. Diagnose and explain to a confused teammate why "the code is definitely deployed" and "the preview is definitely wrong" can both be true simultaneously.

Both facts ARE simultaneously true because of aggressive platform-side caching: Slack (and most other platforms) cache the og: data they scraped on FIRST share of a given URL and don't automatically re-fetch on every subsequent share — they're showing their own stale cached copy, not re-reading your live, correctly-updated page. This is a genuinely common source of confusion precisely because "check the deployed HTML" (which is correct) provides no visibility into "what does Slack currently have cached" (which is the actual problem). Fix: use the Facebook Sharing Debugger (which most platforms' caches respect, being the most prominent public re-scrape tool) to force a re-scrape of the specific URL — this typically propagates the refresh to other platforms' caches as well, though not always instantly or universally. Teach the teammate: this cache-staleness behavior is now a standing item to check FIRST whenever "the preview looks wrong despite a confirmed-correct deploy" recurs in the future, rather than re-debugging the deployed code every time.

Q6 (senior): Design the structured-data strategy for a recipe website that wants recipe rich-results (photo, cook time, rating) in Google search. What are the real requirements beyond just adding a Recipe JSON-LD block, and what's the ongoing maintenance burden?

Real requirements beyond the JSON-LD block itself: (1) EVERY field claimed in the structured data (cook time, ingredients, rating, nutrition info) must correspond to genuinely visible content on the page — Google's Recipe rich-result guidelines are specifically strict and actively audited given how much recipe-SEO gaming has historically occurred in this exact vertical; (2) the aggregateRating field requires REAL user-submitted ratings, not a static number the team picks — this typically means building actual review/rating functionality into the site, a real product feature, not just a markup addition; (3) the recipe's ingredient list and instructions in the JSON-LD must match the visible HTML content precisely — automated consistency, ideally generated FROM the same underlying data source as the visible page rather than hand-maintained separately in two places that can drift apart; (4) image requirements are specific and enforced (minimum dimensions, recipe must be the clear subject of the image, not a lifestyle/ambiance shot). Ongoing maintenance burden: structured data isn't "set once" — if the visible recipe content changes (updated cook time, ingredient substitution) the JSON-LD must update in lockstep, which argues strongly for GENERATING the JSON-LD programmatically from the same CMS/database fields that render the visible page (a templating-layer decision), rather than maintaining it as separately hand-written markup that WILL drift out of sync with real content over time — exactly the kind of two-copies-of-the-truth problem the Global Attributes lesson's data-* discussion warned about, recurring here at the structured-data layer instead.

2 (easy): Validator run. Take any real page you control (or a public page you're just checking, not editing) and run it through Google's Rich Results Test AND the Facebook Sharing Debugger. Report what each tool found — even a "no structured data detected" or "no og: tags found" result is a valid, informative finding.
What to note

A surprisingly large fraction of real pages, even on established sites, have PARTIAL og: tag coverage (title and image but no description, or vice versa) — this is a genuinely common, low-severity real-world gap worth noticing rather than assuming every production site has this fully solved. If the Rich Results Test finds zero structured data on a content-heavy page (an article, product, or recipe site), that's a real missed-opportunity finding worth writing down.

3 (medium): Product JSON-LD, built and validated. Write a complete Product JSON-LD block for a coffee bag product (name, image, description, brand, offers with price/currency/availability — no aggregateRating, since you have no real reviews to back one). Deliberately omitting aggregateRating (per Part 5's non-negotiable rule) is itself the correct, tested decision here — validators will happily accept a Product block without it; adding a fabricated rating just to "complete" the fields would pass VALIDATION while violating the real policy, an important distinction the exercise is designed to surface.

4 (medium): FAQ, matched to visible content. Build a real, visible FAQ section using details/summary (lesson 15) with 3 questions, THEN write the matching FAQPage JSON-LD whose questions/answers are IDENTICAL, word-for-word, to the visible content. Explain in one sentence why word-for-word matching (not just "similar") is the safer standard to hold yourself to.
Answer

Word-for-word matching (rather than "close enough" paraphrasing) removes any judgment call about whether the structured data "accurately represents" the visible content — it simply, provably does, by construction, eliminating any risk of drift or subjective disagreement about accuracy, and making the two blocks trivially auditable against each other by anyone (including an automated script) checking for consistency.

5 (hard): The cache-staleness incident, reproduced conceptually. Without needing an actual public URL, write out the exact sequence of steps you'd take (in order) to diagnose and resolve Q5's scenario — a shared link showing stale og: data three days after a confirmed-correct deploy — as if walking a teammate through it live. Include the specific tool name and what "success" looks like at each step.
Model walkthrough

Step 1: confirm the deployed HTML is actually correct — View Source (not DevTools' rendered view) on the live URL, checking the og:title/og:image values directly; success = the raw HTML genuinely has the new values. Step 2: since step 1 confirms the code IS correct, the problem is necessarily cache-side, not code-side — open the Facebook Sharing Debugger, paste the exact URL, and note what IT currently shows (very likely still the stale version, confirming the cache-staleness theory). Step 3: use the Debugger's explicit "Scrape Again" / re-fetch action — this is the actual fix, forcing a fresh fetch; success = the Debugger's own preview updates to the new title/image immediately after this action. Step 4: re-test in Slack/iMessage with a FRESH share of the link (not scrolling back to a previously-shared instance, which itself may be locally cached client-side too) to confirm the fix propagated. This exact 4-step sequence, rehearsed once, becomes muscle memory for a genuinely recurring real-world scenario.

🎯 Key Takeaways

  • Rich link previews come from platforms fetching your page and reading Open Graph (property=) and/or Twitter Card (name=) meta tags — entirely separate from SEO's title/description tags.
  • og:image is the highest-leverage field — absolute URL required, 1200×630px standard, and undersized/relative images silently break the preview.
  • JSON-LD (script type="application/ld+json", schema.org vocabulary) unlocks rich results — star ratings, recipe cards, FAQ dropdowns — and is Google's recommended format over Microdata/RDFa for its clean separation from visible markup.
  • The non-negotiable rule: structured data must describe content genuinely visible on the page — fabricated claims risk a manual action stripping rich-result eligibility site-wide.
  • Both og:/twitter: tags and JSON-LD must be validated before shipping (Sharing Debugger, Rich Results Test) — failures are invisible in a normal browser and only surface when someone actually shares or searches.
  • Social platforms cache og: data aggressively — a correct deploy with a stale-looking preview needs a forced re-scrape, not a re-debug of the code.
