Links Mastery Foundations

What you'll master: every kind of URL and when to use it (the #1 source of broken links), fragment navigation, the full menu of link behaviors (target, download, mailto:/tel:), the rel attribute's security and SEO meanings, link accessibility, and the link-vs-button rule that separates clean codebases from messy ones. Prerequisites: lessons 1–4.

Part 1 — Anatomy and the Three Kinds of URL

<a href="target">link text</a>

Simple element, but href can hold three fundamentally different kinds of URL, and confusing them is the top cause of "works on my machine, 404 in production":

1. Absolute URLs — a complete address

<a href="https://developer.mozilla.org/en-US/docs/Web/HTML">MDN's HTML docs</a>

Scheme + domain + path. Works from anywhere. Use for: other people's sites. Avoid for your own pages — absolute self-links break when you move between localhost, staging, and production domains.

2. Root-relative URLs — absolute path, your own domain

<a href="/pricing/teams.html">Team pricing</a>    ← leading slash = start at domain root

Resolves against the domain root no matter which page contains the link. Use for: site-wide navigation on your own site (headers, footers) — the same markup works from every page depth. One trap: on GitHub Pages project sites (like this one), your site lives at /repo-name/, so /pricing.html escapes your site — root-relative links need the base path accounted for.

3. Document-relative URLs — from where this file sits

<a href="teams.html">             same folder
<a href="details/faq.html">      down into a subfolder
<a href="../index.html">         up one folder
<a href="../../css/style.css">   up two, then down

Resolves relative to the current document's location. Use for: files that travel together (this tutorial's lessons link to each other this way — the whole folder can move anywhere and links survive). Trap: move one file to a different depth and its relative links all break.

Part 2 — Fragments: Linking Within Pages

<!-- Give any element an id... -->
<h2 id="pricing">Pricing</h2>

<!-- ...then link to it: -->
<a href="#pricing">Jump to pricing</a>              (same page)
<a href="plans.html#pricing">See plan pricing</a>   (other page, then scroll)
<a href="#top"></a> or href="#"                      (top of page)

This is how tables of contents, "skip to content" accessibility links, and every docs-site.com/page#section-name URL work. Details that matter:

  • id values must be unique per page — duplicate ids make fragments (and JS and label/for) ambiguous.
  • The browser scrolls the target to the top — if you have a sticky header it will overlap the target; the CSS fix is scroll-margin-top: 80px on targets (file this away for the CSS track).
  • The :target CSS pseudo-class styles the element currently navigated to — free "highlight the section you jumped to" UX.
  • Fragment changes don't reload the page — which is why early single-page apps routed with #/inbox URLs.

Part 3 — Link Behaviors: target, download, and special schemes

New tabs — and the security bug hiding in them

<a href="https://partner.example" target="_blank" rel="noopener noreferrer">
  Partner site</a>

target="_blank" opens a new tab. Historically it shipped a real vulnerability — tabnabbing: the opened page received a window.opener reference to your page and could navigate your tab to a phishing clone while the user looked away. rel="noopener" severs that reference; noreferrer additionally hides the Referer header. Modern browsers imply noopener for target="_blank", but write it anyway: older browsers, and it documents intent. Also: don't force new tabs on ordinary links — it hijacks the user's choice (they can middle-click/Cmd-click if they want one) and breaks the Back button, the web's most-used feature. Legitimate uses: leaving a form the user is mid-way through, external references from documentation apps.

Downloads

<a href="/files/report-q2.pdf" download>Download the Q2 report</a>
<a href="/files/8f3a2c.pdf" download="quarterly-report.pdf">Download</a>  ← renames it

The download attribute makes the browser save instead of navigate; the optional value renames the file (great when your storage uses hash filenames). Same-origin only — cross-origin downloads ignore the attribute (a security measure: you can't disguise a foreign file as your own).

Beyond web pages

<a href="mailto:support@example.com?subject=Order%20%231042">Email support</a>
<a href="tel:+15035550100">Call us: (503) 555-0100</a>
<a href="sms:+15035550100">Text us</a>

Schemes trigger apps: mail client, phone dialer (mobile), SMS. Note the URL-encoding in the mailto subject (%20 = space, %23 = #). Caveat: mailto on desktops without a configured mail app does nothing — pair it with the visible address as the link text so users can copy it.

Part 4 — rel: the Link-Relationship Vocabulary

nofollow"Don't count this as my endorsement" (SEO)Paid/user-generated/untrusted links

Why nofollow exists (interview-worthy): search ranking historically flowed through links ("PageRank"), so comment spam exploded — spammers seeding links everywhere. nofollow (and its refinements sponsored/ugc) tells crawlers "this link is not my editorial endorsement." If your platform renders user-submitted links, adding rel="ugc nofollow" is a de-facto requirement — it removes the spam incentive.

Part 5 — Link Accessibility & Writing Good Link Text

Screen-reader users pull up a list of all links on a page (same idea as the heading list) and hear them out of context. Now imagine that list reads: "click here, click here, read more, here, more info." Useless. Rules:

  • Link text describes the destination: "Download the Q2 report (PDF, 2 MB)", not "click here". Bonus: descriptive link text is also an SEO ranking signal — accessibility and SEO usually pull in the same direction.
  • Front-load the distinctive part — screen readers let users jump by first letters.
  • Links must be visually identifiable without color alone (WCAG): underlines are the convention. Removing underlines and relying on a color difference fails users with color vision deficiency.
  • Image-only links take their accessible name from the image's alt: <a href="/"><img src="logo.svg" alt="LearnHub home"></a> — alt describes the destination, not the picture ("LearnHub home", not "blue logo").
  • Same visible text should mean same destination; different destinations need different text (or aria-label disambiguation — ARIA lesson).

Part 6 — Deep Dive: Link vs Button (the rule teams fight about)

NAVIGATION (URL changes, new content)  → <a href>
ACTION (something happens here)        → <button>   (Forms lessons)

"Delete account", "Add to cart", "Open menu" are actions — buttons. "View cart", "Settings", "Next page" are navigation — links. Why it matters beyond taste:

  • Built-in behavior: links support open-in-new-tab, copy address, bookmark, prefetch on hover, and announce as "link" to assistive tech; buttons respond to Space and Enter and announce as "button". Fake one with the other and those affordances silently break.
  • <a> without href is not a link: not focusable, not in the tab order, announced as plain text. The <a href="#" onclick="..."> pattern is a triple bug: wrong semantics, scrolls to top, and pollutes history. If JS does the work and no URL is involved, it was a button all along.
  • Styling is not an argument: CSS can make a link look like a button and vice versa (this site's .btn class does exactly that on the pager links below — legal, because they navigate!). The element states the behavior contract; CSS chooses the costume.
🔍 Senior detail — block links: HTML5 allows wrapping block content in an anchor: <a href="/post/42"><article>…whole card…</article></a>. Valid and common for card UIs, with tradeoffs: the entire card's text becomes the accessible link name (keep cards concise), text inside becomes unselectable-by-drag, and nested interactive elements (a button inside the card) are invalid — interactive content can't nest. The alternative pattern is a normal link on the card's heading plus CSS ::after stretching its hit area.

Interview Questions

Q1 (beginner): Explain absolute vs root-relative vs document-relative URLs, with a use case each.

Absolute (scheme+domain+path) — external sites. Root-relative (leading /) — resolves from your domain root; site-wide nav that works from any page. Document-relative — resolves from the current file's folder; groups of files that move together (docs, tutorials). Strong answers name the failure modes: absolute self-links break across environments; root-relative breaks on subpath hosting; document-relative breaks when a single file moves.

Q2 (beginner): Why is "click here" bad link text?

Screen-reader users navigate via an out-of-context list of links — "click here" ×10 conveys nothing. It also wastes an SEO signal (link text describes the target for ranking) and forces sighted scanners to read surrounding prose. Fix: make the destination the text ("View the Q2 report (PDF)").

Q3 (mid): What is tabnabbing and how do you prevent it?

A page opened via target="_blank" historically got a window.opener handle and could set opener.location, redirecting the original tab to a phishing lookalike while the user's attention was on the new tab — users "return" to what looks like your login page. rel="noopener" nulls the handle (noreferrer also drops the Referer). Modern browsers imply noopener on _blank, but explicit is still best practice for legacy coverage and intent. Deeper point: it's a same-origin-policy gap — cross-origin pages normally can't touch each other, but navigation of the opener was allowed.

Q4 (mid): Link or button? "Log out", "Next photo", "Print this page", "Checkout". Justify each.

Log out: action with server state change → button (bonus: it should be a POST — GET logout links get prefetched by browsers/extensions and log users out spontaneously!). Next photo: if the URL changes per photo (deep-linkable) → link; if it mutates a carousel in place → button. Print: pure action (window.print()) → button. Checkout: navigation to the checkout page → link styled as a big fat button. The rubric: URL/new content = link; do-something-here = button — and the logout prefetch anecdote is what makes the answer senior.

Q5 (senior): Your marketplace lets sellers write product descriptions with links. What link-related protections does the rendering pipeline need?

(1) Scheme allow-list — permit http(s)/mailto only; block javascript: URLs, the classic href-based XSS (sanitizers must catch encodings like jAvAsCrIpT&#58;). (2) rel="ugc nofollow noopener" on every rendered link — removes SEO spam incentive and opener access; add target rules per product decision. (3) Consider a redirect/interstitial for outbound links (lets you scan for malware domains and strip referrer). (4) Visual marking of external links so sellers can't disguise off-site links as platform UI (phishing). (5) HTML sanitization of the whole fragment (allow-list elements/attributes — never regex). Mentioning sanitizer + scheme allow-list + rel triad is the expected senior answer.

Q6 (senior): Design the linking strategy for docs deployed at docs.example.com locally, /docs/ in production, and also exported to offline HTML. Which URL forms survive all three?

Document-relative URLs are the only form that survives all three unchanged — they're location-independent, which is why static-site generators and offline docs (and this tutorial site) use them. Root-relative breaks under /docs/; absolute breaks offline and across environments. Mitigations if you need root-relative ergonomics: a build-time base-path variable rewriting links per target, or <base href> (mention its trap: it rewrites the resolution of every relative URL and fragment on the page — #anchors resolve against the base, not the current page — so most teams avoid it). Senior signal: naming the export/offline constraint before being prompted.

🏋️ Exercises

1 (easy): URL-form drill. From a file at /shop/products/chairs.html, write the href to reach: (a) /shop/cart.html, (b) /index.html, (c) products/tables.html in the same folder, (d) MDN's site — choosing the best URL form for each.
Answers

(a) ../cart.html (or root-relative /shop/cart.html for site nav), (b) ../../index.html (or /index.html), (c) tables.html — same folder needs no prefix, (d) absolute https://developer.mozilla.org/. If you wrote /products/tables.html for (c), trace it again: leading slash goes to domain root, which is wrong here.

2 (easy): Table of contents. Add a linked TOC to your sourdough article (lesson 3): a list at the top linking to each h2 via fragments, plus a "back to top" link after each section. Verify every jump.

  </ul>
</nav>

<h2 id="ingredients">Ingredients</h2>

ids on the headings themselves, lowercase-hyphenated, unique. Try the :target trick too: :target { background: #fffbcc } in a style tag — instant jumped-here highlight.

3 (medium): The contact block. Build a contact section with: an email link with pre-filled subject AND body, a phone link, and a vCard download renamed via the download attribute. Test on desktop and (if possible) a phone.
Solution 
<p><a href="mailto:hello@mayascoffee.com?subject=Wholesale%20inquiry&amp;body=Hi%20Maya%2C">
  hello@mayascoffee.com</a></p>
<p><a href="tel:+15035550100">(503) 555-0100</a></p>

Notes: & between mailto params must be written &amp; in HTML source (lesson 3's entities, in the wild!); %2C is an encoded comma; the visible text is the real address so it's copyable when no mail app exists.

4 (medium): Link-list audit. On any news site, open DevTools and run [...document.querySelectorAll('a')].map(a => a.textContent.trim() || '[EMPTY]'). Count how many entries would be meaningless out of context. Then check three target="_blank" links for rel hygiene.
What you'll find

Typically 10–40% meaningless ("Read more", "Here", empty strings from icon-only links missing accessible names). Empty entries are the worst finds — those links are unusable via screen reader. Most major sites do carry noopener (frameworks add it); finding one without it on a _blank link is a reportable bug. You just performed a real accessibility audit step.

5 (hard): Card component, both patterns. Build a product card (image, h3 title, description, price) twice: (a) whole card wrapped in one anchor, (b) link on the title only, stretched over the card with CSS position:relative on the card and ::after { position:absolute; inset:0 } on the link. Tab through both; select text in both; explain which you'd ship and why.
Comparison

(a) is simpler but: accessible name = entire card text (verbose), drag-selecting text is impossible, and you can never add a secondary button inside. (b) keeps a concise accessible name (the title), text remains selectable-ish, and secondary actions can sit above the ::after layer with z-index. Most design systems ship (b) — the "stretched link" pattern (Bootstrap has it built in). Shipping answer: (b), unless the card is trivially small.

6 (hard): Broken-link crawler. Preview of scripting: write a Node script (or console snippet) that collects every same-site href on a page of yours, fetches each, and reports non-200s. Run it against your tutorial folder served locally.
Reference (browser console version) 
const links = [...document.querySelectorAll('a[href]')]
  .map(a => a.href)
  .filter(h => h.startsWith(location.origin));
for (const url of [...new Set(links)]) {
  fetch(url, { method: 'HEAD' })
    .then(r => { if (!r.ok) console.warn(r.status, url); })
    .catch(() => console.warn('FAILED', url));
}
console.log(`checking ${new Set(links).size} internal links…`);

HEAD requests avoid downloading bodies; the Set dedupes. Real projects run exactly this in CI (tools: lychee, linkinator) because link rot is inevitable — automate the detection.

🎯 Key Takeaways

  • Three URL forms, three jobs: absolute = external, root-relative = your site-wide nav (beware subpath hosting), document-relative = files that travel together.
  • Fragments (#id) power TOCs, skip-links, and deep links — ids unique, scroll-margin-top fixes sticky-header overlap.
  • target="_blank" always brings rel="noopener noreferrer" — and usually shouldn't be forced at all.
  • rel is a vocabulary: noopener (security), nofollow/sponsored/ugc (SEO trust), and user-generated links get "ugc nofollow" plus a scheme allow-list (no javascript: URLs).
  • Link text describes the destination — screen-reader link lists and SEO both depend on it; image links inherit alt as their name.
  • Navigation = <a href>; action = <button>. CSS chooses the costume; the element states the contract. href-less anchors and onclick-divs are neither.
