SVG & Canvas in HTML Media & Embedding

What you'll master: inline SVG as real, stylable, scriptable DOM (versus the img-tag SVG from lesson 6), the SVG-vs-canvas decision that actually matters, canvas's accessibility gap and how to cover it, and when to reach for neither because CSS already does the job. This closes the Media & Embedding section. Prerequisites: Images Mastery (format basics), iframes & Embedding Security.

Part 1 — Two Graphics Models, One Decision

SVG     = a scene graph of shapes, described in XML — "draw a circle at (50,50) radius 20"
CANVAS  = a pixel bitmap you paint into with JS — "color these pixels" (the paint dries immediately)

That one-line contrast explains nearly every practical difference between them. SVG shapes remain individually addressable DOM nodes forever — you can select the circle, change its fill, animate it with CSS, or click it, at any point after drawing. Canvas has no memory of "the circle" once painted — it's only pixels; to change anything you must know what to erase and redraw. This single distinction should drive the decision far more than "which do I know better."

Part 2 — Inline SVG: Real DOM, Not an Image

<svg viewBox="0 0 100 100" width="48" height="48" role="img" aria-labelledby="star-title">
  <title id="star-title">Featured item</title>
  <polygon class="star" points="50,5 61,35 95,35 68,57 79,91 50,70 21,91 32,57 5,35 39,35"/>
</svg>
.star { fill: gold; stroke: #b8860b; stroke-width: 2; transition: fill .2s; }
.star:hover { fill: orange; }             /* real CSS, on a real DOM node */

Lesson 6 covered SVG referenced via <img src="icon.svg"> — opaque, uncustomizable from your page's CSS/JS, exactly like a JPEG. Inline SVG — the markup pasted directly into your HTML — is different in kind: every <polygon>, <path>, <circle> becomes a live DOM node your page's stylesheet can select and your JavaScript can query, exactly like any HTML element. This is why every serious icon system (icon fonts' modern replacement) and every data-visualization library (D3, most charting libraries) uses inline SVG — you need to recolor an icon on hover, or update a chart bar's height in response to new data, both of which require addressable individual shapes.

viewBox is the attribute that makes SVG effortlessly responsive: it defines an internal coordinate system (here, 100×100 units) independent of the rendered width/height — resize the element with CSS to any dimension and the artwork scales perfectly, no pixelation, ever (the core promise of vector graphics, and the reason logos/icons/illustrations belong in SVG per lesson 6). role="img" + a <title> element (SVG's own, distinct from HTML's global title attribute — do not confuse them) is SVG's accessible-naming pattern, parallel to img's alt.

Part 3 — canvas: Pixels You Paint

<canvas id="chart" width="400" height="200">
  A bar chart showing quarterly revenue: Q1 $40k, Q2 $55k, Q3 $48k, Q4 $70k.
</canvas>

<script>
const ctx = document.getElementById("chart").getContext("2d");
const data = [40, 55, 48, 70];
data.forEach((v, i) => {
  ctx.fillStyle = "#3b82f6";
  ctx.fillRect(i * 90 + 20, 200 - v * 2, 60, v * 2);   // just... colored rectangles. No shape objects.
});
</script>

Three facts that matter in practice: (1) width/height are attributes on the canvas element itself, defining its actual pixel grid — NOT the same as CSS width/height, which only stretches the existing bitmap (mismatching them causes blurry, stretched output, a very common canvas bug); (2) once fillRect runs, there is no "the second bar" object to later select — redrawing means clearing and repainting everything, which is why canvas-based charting libraries maintain their OWN internal data model and redraw from scratch on every update; (3) the fallback content between the tags (the sentence describing the chart) renders only in browsers without canvas support — functionally never today, but it hints at the real problem the next section covers properly.

Part 4 — The Decision Table

NeedChooseWhy
Icon, logo, illustrationSVG (inline if it needs styling/interaction, img if static)Infinite scaling, small file size for geometric art
Interactive chart (click a bar, hover a point)SVGEach shape is a real, clickable, styleable DOM node
Diagram/illustration users can select-and-copy text fromSVGText stays real, selectable text — not baked into pixels
Thousands of moving particles / real-time video effectsCanvas (or WebGL for GPU-heavy cases)DOM node overhead per shape becomes the bottleneck at high counts; raw pixel painting is faster at scale
Photo editing / pixel-level manipulationCanvasYou're manipulating pixels directly by definition
Game rendering loopCanvas (2D) or WebGL (3D/heavy 2D)Full per-frame repaint control, no DOM overhead
A shape/gradient that's really just decorationCSS (gradient, clip-path, border-radius)Neither SVG nor canvas needed — check the ladder first

The rule of thumb worth memorizing: SVG for "a moderate number of things that need to stay individually addressable"; canvas for "a very large number of things, or raw pixels, repainted every frame." A dashboard with 8 interactive chart types → SVG (D3/Chart.js's SVG mode). A particle-effect background with 5,000 dots → canvas (5,000 DOM nodes would visibly slow the page; 5,000 painted pixels are trivial). Most real projects need SVG far more often than canvas — canvas is reached for less frequently than its popularity in tutorials suggests.

Part 5 — Canvas's Real Cost: Accessibility Is Entirely Manual

This is the fact this lesson most wants you to remember: canvas content is invisible to screen readers, to text search (Ctrl+F), to translation tools, and to search engine crawlers — it's a bitmap, indistinguishable from a photograph to every one of those systems. SVG, being real DOM with real text nodes, gets all of that for free. Choosing canvas for a data visualization is choosing to manually rebuild everything SVG gives away:

<div role="img" aria-label="Quarterly revenue bar chart. Q1: $40,000. Q2: $55,000.
                              Q3: $48,000. Q4: $70,000. Revenue grew 75% over the year.">
  <canvas id="chart" width="400" height="200"></canvas>
</div>
<table class="sr-only">      <!-- a real, visually-hidden but AT-visible data table -->
  <caption>Quarterly revenue</caption>
  <tr><th>Quarter</th><th>Revenue</th></tr>
  <tr><td>Q1</td><td>$40,000</td></tr>
  ...
</table>

The wrapper's aria-label gives a summary announcement; the visually-hidden real <table> (lesson 8's full table semantics, reused) gives AT users the actual underlying data on request — the professional pattern for any canvas-based chart. This isn't optional polish; it's the direct, unavoidable cost of the "just pixels" model from Part 1, and it's precisely why data-heavy dashboards that need to be accessible often default to SVG-based charting libraries specifically to avoid building this parallel data table by hand for every chart.

Part 6 — Worked Comparison: the Same Chart, Both Ways

An interactive bar that highlights on hover and shows a tooltip on click, SVG version:

<svg viewBox="0 0 400 200">
  <rect class="bar" x="20" y="120" width="60" height="80" data-value="40" tabindex="0"
        role="img" aria-label="Q1: $40,000"/>
  <rect class="bar" x="110" y="90" width="60" height="110" data-value="55" tabindex="0"
        role="img" aria-label="Q2: $55,000"/>
</svg>
<style>.bar:hover, .bar:focus { fill: orange; outline: 2px solid; }</style>
<script>
document.querySelectorAll(".bar").forEach(bar =>
  bar.addEventListener("click", () => showTooltip(bar.dataset.value)));
</script>

Hover/focus styling: pure CSS, zero JS. Click handling: a normal addEventListener per real element, same as any button. Accessibility: each bar independently focusable and labeled, same as any interactive HTML. The canvas equivalent needs manual hit-testing (calculating which rectangle a click's x/y coordinates fall inside — geometry math you write yourself), no CSS :hover at all (you must track mouse position and redraw on every mousemove to simulate a hover state), and the full Part 5 accessibility scaffold. This comparison is the whole lesson distilled: SVG gives you the platform's interaction model for free; canvas asks you to rebuild it.

Interview Questions

Q1 (beginner): In one sentence each, what's fundamentally different about how SVG and canvas represent graphics?

SVG is a scene graph of individually addressable shape objects described declaratively (real DOM nodes you can select/style/script forever); canvas is an immediate-mode pixel bitmap you paint into imperatively, with no memory of what you drew once the paint call completes.

Q2 (beginner): Why does an inline SVG respond to a CSS :hover rule while an SVG referenced via img src does not?

Inline SVG markup becomes real DOM nodes in your document, exactly like any HTML element, fully reachable by your page's stylesheet and scripts. img src="icon.svg" treats the file as an opaque raster-equivalent resource — same as a JPEG — with no access to its internal structure at all. The choice between the two is exactly the choice between "needs styling/interaction" and "just needs to display."

Q3 (mid): A canvas element renders blurry on a retina display despite looking fine on a standard display. Diagnose.

Almost certainly the canvas's width/height HTML attributes (its actual pixel grid) weren't set to account for devicePixelRatio, while CSS is stretching a lower-resolution bitmap to fill a larger display area. Fix: set the canvas's real attribute dimensions to CSS-size × devicePixelRatio, then scale the drawing context down by the same factor (ctx.scale(dpr, dpr)) so drawing coordinates stay in CSS-pixel units while the underlying bitmap is genuinely higher-resolution — the canvas equivalent of lesson 6's retina-image density descriptors.

Q4 (mid): Why would a data-visualization library default to SVG rendering rather than canvas, even though canvas is often described as "faster"?

Canvas is faster specifically at raw painting throughput for very large numbers of shapes, but most charts have tens to low-hundreds of data points — nowhere near where that advantage matters — while SVG gives interactivity (hover/click/focus per data point), CSS-based styling and transitions, accessibility (real text/DOM for screen readers and Ctrl+F), and easier debugging (inspectable DOM in DevTools) essentially for free. The performance argument only tips toward canvas at shape counts in the thousands, which is a small minority of real chart use cases.

Q5 (senior): Your team built a canvas-based interactive org chart (draggable nodes, hover tooltips, click-to-expand) and it's now failing an accessibility audit badly. Assess the situation and the remediation options.

Assessment: canvas was the wrong default here — an org chart is a moderate number (dozens to low hundreds) of genuinely interactive, individually meaningful nodes, squarely SVG's sweet spot, and the team paid canvas's full manual-accessibility cost (Part 5) without apparently budgeting for it. Remediation options ranked by cost: (1) full rewrite to inline SVG — correct long-term answer, each node becomes a real focusable/labeled element with native hover/click, but a real engineering cost against a working feature; (2) accessible-overlay retrofit — keep the canvas rendering, but generate a parallel, visually-hidden but fully real and synchronized DOM structure (nested lists or a tree with role="tree"/role="treeitem") mirroring the chart's data, kept in sync with every state change (drag, expand) — significant ongoing maintenance burden keeping two representations consistent, but avoids the rewrite; (3) minimum-viable text alternative — a "view as text/table" toggle showing the full org structure as a real accessible list, satisfying WCAG without visual parity. The senior judgment: recommend (1) if the component has a long roadmap ahead of it (the sync burden of (2) compounds every feature added), (3) as a fast interim fix while (1) is scheduled — and flag clearly that this situation is a predictable consequence of the SVG-vs-canvas decision being made without weighing its accessibility cost up front, exactly the gap this lesson exists to close before the fact rather than after an audit.

Q6 (senior): Design the rendering strategy for a data-dashboard product with both (a) 12 standard business charts (bar/line/pie, tens of data points each) and (b) one real-time network-traffic visualization showing up to 50,000 live-updating connection dots.

Split by the decision table's core axis, not by consistency-for-its-own-sake: (a) SVG-based charting (D3, Chart.js's SVG mode, or similar) — interactivity, accessibility, and CSS theming all come standard at these data volumes, and a design system wants ONE consistent charting approach across 12 chart types, which SVG's tooling ecosystem serves well. (b) Canvas (likely with requestAnimationFrame-driven redraws, possibly WebGL if 50,000 points with physics/force-layout pushes past 2D canvas's comfortable range) — SVG would mean 50,000 live DOM nodes updating every frame, a guaranteed performance failure; this is exactly canvas's designed use case. Accessibility for (b): per Part 5, a summary aria-label plus a real, visually-hidden data table or list surfacing the aggregate/key connection data (not literally 50,000 individual entries — a reasonable summarized view: top talkers, anomalies, totals), since a literal per-dot table at that scale would itself be unusable. The senior shape: two different tools, chosen per-component by actual shape-count and interactivity needs, not by a single project-wide "we use library X" mandate that ignores the decision table entirely.

🏋️ Exercises

1 (easy): Inline vs referenced, felt. Take one simple SVG icon (a star or heart shape). Reference it once via img src="icon.svg" and once pasted inline. Try adding a CSS :hover rule changing fill color to both. Report which works and why, in your own words tying back to the DOM-vs-opaque-resource distinction.
Expected

The inline version responds to :hover (its shapes are real DOM); the img-referenced version does not (opaque resource, same as a PNG). If you got a surprising result, double check you're targeting the actual shape element's class inside the SVG, not the outer svg tag alone — inline SVG needs the same specificity care as any nested markup.

2 (easy): The decision drill. SVG or canvas: (a) a company logo in the header, (b) an interactive US map where each state is clickable and highlights, (c) a real-time audio waveform visualizer, (d) a line chart with 6 data points and a tooltip on hover, (e) a photo-filter preview tool, (f) an icon that needs to change color based on a CSS variable.
Answers

(a) SVG (img reference is fine if static; inline if the logo needs a hover effect). (b) SVG — 50 individually clickable, styleable regions is exactly SVG's sweet spot. (c) Canvas — real- time, high-frequency full-repaint, no need for individually addressable bars. (d) SVG — small data count, needs interactivity. (e) Canvas — literal pixel manipulation. (f) Inline SVG — CSS variable access requires being real, styled DOM.

3 (medium): Responsive icon system. Build 3 inline SVG icons (viewBox-based) styled entirely via one shared CSS class, each recoloring on hover and resizing cleanly from 16px to 128px via a CSS width change alone (no separate assets). Confirm zero pixelation at the largest size.
What to verify

Because viewBox decouples the internal coordinate system from rendered size, a single 24×24 viewBox icon should render perfectly crisp at 128px — this is the concrete proof of vector scaling's promise from lesson 6, now demonstrated rather than just stated. If you see pixelation, check you resized via CSS width/height, not by editing the SVG's own width/height attributes to a fixed large value with a mismatched viewBox.

4 (medium): Retina-correct canvas. Build a canvas drawing a simple pattern (a grid of circles). Fix it for retina displays using the devicePixelRatio technique from Q3's answer. Compare sharpness with and without the fix using your browser's zoom or a retina display if available.
Reference core 
const dpr = window.devicePixelRatio || 1;
const cssWidth = 400, cssHeight = 200;
canvas.width = cssWidth * dpr;
canvas.height = cssHeight * dpr;
canvas.style.width = cssWidth + "px";
canvas.style.height = cssHeight + "px";
ctx.scale(dpr, dpr);          // now draw using CSS-pixel coordinates as normal

Post-fix, circles should render crisp at any display density; your drawing code itself stays unchanged (still thinking in 400×200 CSS-pixel terms) because the ctx.scale call absorbs the density difference transparently.

5 (hard): Accessible canvas chart. Build Part 5's complete pattern: a canvas bar chart plus its wrapping aria-label summary plus a visually-hidden real data table with the same numbers. Verify with DevTools' Accessibility panel that the canvas region announces the summary, and that the hidden table is genuinely present in the accessibility tree (not display:none, which would hide it from AT too — use a proper sr-only clip technique).
Key detail 
.sr-only {                    /* visually hidden but AT-visible — NOT display:none */
  position: absolute; width: 1px; height: 1px;
  overflow: hidden; clip: rect(0,0,0,0); white-space: nowrap;
}

display:none or visibility:hidden remove content from the accessibility tree entirely — exactly the opposite of the goal here. The clip-based sr-only pattern keeps content programmatically present while visually invisible, which is the whole point: sighted users see the chart, AT users get the real table. Confirm this distinction in the Accessibility panel directly — it's the one detail most people get wrong on their first attempt at this pattern.

6 (hard): Build the same interactive chart twice. Implement Part 6's hover-highlight, click-for-tooltip bar chart TWICE — once in SVG (as shown), once in canvas (manual hit-testing on click, manual hover-tracking via mousemove + redraw). Expected finding: the canvas version needs manual hit-testing geometry, a full-canvas clearRect+redraw on every single mousemove event (no native :hover), and still has zero accessibility without Part 5's additions layered on top — noticeably more code and complexity for an identical visual/interactive result. This exercise is designed to make Part 6's "SVG gives it away free, canvas asks you to rebuild it" argument land experientially rather than just as a claim you read.

🎯 Key Takeaways

  • SVG = a scene graph of individually addressable shape DOM nodes; canvas = a pixel bitmap you paint into imperatively with no memory of what was drawn.
  • Inline SVG (pasted markup) is real, stylable, scriptable DOM; SVG referenced via img src is opaque, exactly like any raster image — choose based on whether you need styling/interaction.
  • viewBox decouples SVG's internal coordinate system from rendered size — the mechanism behind lossless scaling to any dimension.
  • Decision rule: SVG for a moderate number of things needing individual addressability/interactivity; canvas for very large shape counts, raw pixel work, or high-frequency full repaints.
  • Canvas accessibility is entirely manual — invisible to screen readers, search, and translation by default; budget an aria-label summary plus a visually-hidden real data table for any canvas visualization.
  • canvas width/height are pixel-grid attributes, not CSS sizing — mismatching them (or ignoring devicePixelRatio) is the most common canvas rendering bug.
