ARIA Deep Dive Accessibility

What you'll master: the Five Rules of ARIA (starting with "don't use ARIA"), the three attribute categories (roles, properties, states), live regions for dynamic content, the full combobox pattern as a worked case study in what "ARIA is hard" actually means, and — critically — when NOT to reach for it. Prerequisites: Accessibility Fundamentals (the accessibility tree concept is load- bearing here), Choosing the Right Element.

Part 1 — What ARIA Actually Is (and the First Rule)

ARIA (Accessible Rich Internet Applications) is a set of HTML attributes that directly edit the accessibility tree from lesson 22 — overriding or supplying the role, name, and state that plain HTML would otherwise compute. Critically: ARIA changes nothing about behavior, styling, or keyboard handling — it only changes what gets ANNOUNCED. This single fact is the source of nearly every ARIA bug in existence, and it's why the W3C's own First Rule of ARIA Use exists:

"If you can use a native HTML element or attribute with the semantics and behavior you require already built in, instead of re-purposing an element and adding an ARIA role, state or property to make it accessible, then do so."

You've been living this rule since lesson 16 without the citation: a real <button> gets role, keyboard handling, and focus management as one package deal; <div role="button"> gets ONLY the announced role — you must then hand-build the Tab focusability (tabindex="0"), the Enter/Space activation (a keydown handler, careful about Space's default scroll behavior), and the disabled-state handling entirely yourself. ARIA can make something announce as a button; it cannot make it behave like one. That gap is this whole lesson's central theme.

Part 2 — The Five Rules of ARIA (W3C's own guidance, condensed)

1. Use native HTML first (Part 1) — always the preferred solution when available.
2. Don't change native semantics unless you truly have to (no role="heading" on a <button>).
3. All interactive ARIA controls must be usable with the keyboard — YOU must implement this;
   ARIA grants no keyboard behavior automatically.
4. Don't use role="presentation"/aria-hidden="true" on a FOCUSABLE element — it creates a focus
   stop with nothing announced, a confusing dead zone for AT users.
5. All interactive elements must have an accessible name — a role with no name announces as
   "button" with nothing else, useless.

Rule 3 deserves its own emphasis because it's the most-violated rule in real codebases: adding role="button" to a div and stopping there produces something screen readers announce correctly but keyboard users cannot operate at all — arguably WORSE than doing nothing, because it actively lies about the element's capability. "No ARIA is better than bad ARIA" is a genuine, frequently-cited principle for exactly this reason.

Part 3 — The Three Categories

Roles: what kind of thing is this

<div role="alert">             a role NOT expressible in plain HTML — legitimate use
<div role="tablist">           a design-system tab component built from divs (also legitimate —
  <div role="tab">             no native HTML tab element exists)
<ul role="list">                overriding a role STRIPPED by CSS — lesson 15's list-style:none fix

Roles fall into two honest categories: ones with no native HTML equivalent (tablist, alert, tooltip, slider-as-a-composite-widget) where ARIA is the only tool available — the clearly legitimate case; and ones patching a semantic loss (role="list" restoring what list-style:none removed in some browsers, per lesson 15) — also legitimate, narrowly. What's NOT legitimate: role="button" on a div when <button> was sitting right there, unused, the whole time.

Properties: relationships and characteristics (mostly static)

aria-label="Close"                  accessible name when no visible text exists (icon buttons — lesson 16)
aria-labelledby="heading-id"        name FROM another element's text (reuse instead of duplicating)
aria-describedby="hint-id err-id"   supplementary description — the Forms UX lesson's core mechanism
aria-required="true"                (usually redundant with the native required attribute — skip it)
aria-haspopup="listbox"             warns AT that activating this opens a popup/menu/dialog

You've used aria-describedby and aria-label extensively already (Forms UX, lesson 16's icon buttons) — this section is naming the pattern formally: properties describe relationships and characteristics that don't typically change during the component's lifetime.

States: what's happening RIGHT NOW (dynamic — you must keep these in sync)

aria-expanded="false"    a disclosure/menu/accordion's open-ness (details/summary does this natively!)
aria-checked="true"      a custom checkbox/switch/radio's checked-ness
aria-selected="true"     which item in a list/tab set is currently selected
aria-current="page"      which nav link represents the CURRENT page (a real, useful, underused one)
aria-invalid="true"      a form field's validity (Forms lesson's exact mechanism)
aria-disabled="true"     disabled, but WITHOUT the native disabled attribute's side effects (rare, deliberate)
aria-hidden="true"       hide decorative content from AT (lesson 16's icon-button pattern)

States are the ones that bite: every state attribute must be updated by YOUR JavaScript every time the actual state changes — set aria-expanded="true" when you open something, set it back to "false" when you close it, on every single interaction, forever. A stale state attribute (menu visibly closed, aria-expanded still stuck at "true") is worse than no attribute at all — the exact "bad ARIA" failure mode from Part 2's Rule 3, now showing up in the states category specifically.

Part 4 — Live Regions: Announcing Change Without Moving Focus

The gap this section fills: when content changes WITHOUT any user-initiated focus change (a chat message arrives, a form saves in the background, a cart total updates), nothing tells a screen- reader user it happened — they're not "looking" at that part of the page. aria-live solves this, and you've already used its concrete embodiment: <output> (Forms lesson) is an implicit live region; role="alert" (Part 3) is an implicit assertive live region. Here's the explicit form:

<div aria-live="polite" id="save-status"></div>
<div aria-live="assertive" role="alert" id="error-banner"></div>

<script>
saveStatus.textContent = "Draft saved";        // announced when the AT user is next idle
errorBanner.textContent = "Connection lost";   // announced IMMEDIATELY, interrupting whatever's being read
</script>
ValueBehaviorUse for
politeAnnounced at the next natural pause — never interruptsSave confirmations, search-result counts, non-urgent updates
assertiveAnnounced immediately, interrupting current speechErrors, critical alerts only — overuse trains users to ignore it
off (default)No announcementContent that changes but isn't informationally important

The rule that prevents the most common live-region bug: the container must exist in the DOM BEFORE you populate it — creating a div with aria-live AND its content in the same operation often fails to announce, because AT needs to already be "watching" the region when the mutation happens. Render the empty container on page load; fill it later. And use assertive sparingly — Part 2's "worse than nothing" principle applies here too: an assertive region firing on every minor update trains users to tune it out entirely, defeating its purpose for the one moment it truly matters.

Part 5 — Worked Case Study: the Combobox Pattern (why "ARIA is hard" is a real claim)

Lesson 13 flagged this without detail: a genuinely custom searchable dropdown (autocomplete input + filtered listbox) is the canonical example of ARIA done thoroughly, and seeing the real requirement list is the fastest way to internalize why "just add role=combobox" undersells the work:

<label for="city-input" id="city-label">City</label>
<input id="city-input" role="combobox" type="text"
       aria-expanded="false" aria-controls="city-listbox"
       aria-autocomplete="list" aria-activedescendant="">
<ul id="city-listbox" role="listbox" aria-labelledby="city-label" hidden>
  <li role="option" id="opt-portland">Portland</li>
  <li role="option" id="opt-porto">Porto</li>
</ul>

The COMPLETE requirement list this markup only begins to satisfy: role + aria-expanded kept in sync with actual open/closed state on every keystroke and blur; aria-activedescendant updated to track which option is currently highlighted as arrow keys move (the input keeps DOM focus the whole time — this is intentional, and confuses people building it the first time); a full keyboard contract — Down/Up move highlight, Enter selects, Escape closes and returns focus correctly, Home/End jump to first/last, typing continues filtering; each option needs aria-selected as highlight moves; the listbox needs role + aria-labelledby; touch/mobile needs its own consideration since none of this keyboard model applies there; and it needs testing against multiple real screen readers because combobox support historically varies across AT/browser combinations more than almost any other pattern. This is a multi-day component, not an afternoon's ARIA sprinkle — which is precisely why lesson 13 recommended a battle-tested library (Downshift, React Aria, Radix) over a hand-rolled version for anything beyond a learning exercise: you're not paying for convenience, you're paying for hundreds of hours of exactly this kind of cross-AT testing you'd otherwise have to redo yourself.

Part 6 — Deep Dive: aria-hidden and the Inert Attribute

<span aria-hidden="true">★</span>              decorative icon — invisible to AT, visible to eyes (lesson 16)
<div aria-hidden="true">...background content...</div>   hide a whole subtree from AT

<div inert>...content behind an open modal...</div>      hides from AT AND disables
                                                                interaction/focus — the dialog
                                                                element does this automatically
                                                                (lesson 16) when you use showModal()

aria-hidden="true" removes a subtree from the accessibility tree ENTIRELY — screen readers skip it as if it doesn't exist, while it remains fully visible and (dangerously) fully focusable by default, violating Rule 4 from Part 2 if you're not careful: tabbing can land a keyboard user on a control the screen reader never announced, a genuinely disorienting trap. inert (the modern, correct tool for "everything behind my modal should be untouchable") solves both problems at once — hidden from AT AND removed from the tab order AND unclickable — which is exactly what dialog.showModal() applies automatically to the rest of the page (lesson 16), and exactly why that native element was recommended over a hand-rolled modal in the first place. If you ever reach for aria-hidden on a region containing focusable elements, reach for inert instead.

Interview Questions

Q1 (beginner): What is the First Rule of ARIA, and why does it exist?

Use a native HTML element/attribute with the required semantics and behavior built in, rather than repurposing a generic element and adding ARIA to fake it — because ARIA can only change what gets ANNOUNCED (the accessibility tree), never behavior, keyboard handling, or focus management. A native button gets all three as one package; a div with role="button" gets only the announcement, leaving keyboard/focus entirely to be hand-built.

Q2 (beginner): What's the practical difference between an ARIA property and an ARIA state?

Properties (aria-label, aria-describedby, aria-labelledby) describe relatively static relationships/characteristics that rarely change during a component's lifetime. States (aria-expanded, aria-checked, aria-selected, aria-invalid) describe current condition and MUST be kept in sync with real behavior by your JavaScript on every relevant interaction — a stale state attribute actively misinforms assistive technology.

Q3 (mid): Explain why "no ARIA is better than bad ARIA" is a real, defensible principle rather than just a scary-sounding warning.

Because ARIA overrides what's announced without granting any of the behavior implied by that announcement (Rule 3). role="button" on an unhandled div tells a screen-reader user "this is operable as a button" — a promise the element then fails to keep when they try Enter or Space, producing genuine confusion and wasted effort. A plain, unstyled div with no role at all is at least honestly announced as nothing special — it doesn't actively mislead about its capabilities. Bad ARIA is a false promise; no ARIA is (at worst) a missed opportunity.

Q4 (mid): Why must a live region's container exist in the DOM before you populate it with the announcement text?

Assistive technology needs to already be observing (subscribed to mutations on) the live-region element at the moment content changes — if you create the div AND set its text content in one operation, AT wasn't watching yet when the mutation "happened" from its perspective, and the announcement is commonly missed entirely. The reliable pattern: render an empty aria-live container on initial page load, then mutate its textContent later in response to events — the classic bug is building both in the same JS call and wondering why nothing announces.

Q5 (senior): Your team is building a custom multi-select "tag picker" component (type to filter, arrow keys to navigate suggestions, Enter to add a tag, backspace to remove the last tag). List the complete ARIA + keyboard contract this needs, and identify which parts are most likely to be forgotten in a first implementation.

Complete contract: role="combobox" on the input with aria-expanded tracking suggestion-list visibility; the suggestion list as role="listbox" with role="option" children; aria-activedescendant on the input tracking the currently arrow-key-highlighted suggestion (NOT moving actual DOM focus — the input keeps it); aria-selected on options as highlight moves; each ADDED tag needs to be both visually present AND properly exposed — commonly as a role="listbox"/role="option" set of removable tags, or at minimum each tag's remove control needs a real accessible name ("Remove tag: JavaScript", not just an X); a live region announcing "Tag added: JavaScript" / "Tag removed: JavaScript" since neither addition nor removal alone reliably conveys the state change to a screen-reader user who isn't visually watching the tag list; full keyboard contract including Backspace-removes-last-tag-when-input-is-empty (a specific, easy-to-miss interaction distinct from normal text editing). Most commonly forgotten in a first pass: the live-region announcements for add/remove (the component LOOKS complete without them, since sighted testing never surfaces the gap), and the aria-activedescendant-not-real-focus detail, which developers unfamiliar with the pattern often get backwards by moving real focus into the listbox instead, breaking the typing-while-navigating interaction model users expect from every OS-level autocomplete they've ever used.

Q6 (senior): A code reviewer flags "excessive ARIA" on a PR — role, aria-label, and 3 aria-* state attributes on what's ultimately a styled <button>. Assess whether this concern is valid and what the fix should be.

Valid concern, and a real, common anti-pattern: ARIA attributes added defensively/by habit onto an element that's ALREADY a real button already have correct native role, name (from its text content or existing label), and require no state attributes unless the button is genuinely stateful (a toggle button needs aria-pressed; an accordion trigger needs aria-expanded — those are legitimate). Redundant role="button" on an actual <button> element does nothing useful and adds maintenance surface (another thing that can drift out of sync); aria-label duplicating visible text content is redundant and can actually make VOICE CONTROL worse (voice-control tools match against the accessible name, and a mismatched aria-label can mean saying the VISIBLE text doesn't trigger the button — a real, somewhat under-known consequence). Fix: remove role (redundant on a real button), remove aria-label if it duplicates visible text (let the text content BE the name), and audit each remaining aria-* state attribute against whether the button genuinely has that state — keep only what's earning its place. The broader lesson for the reviewer's comment: ARIA is a scalpel for gaps native HTML can't cover, not a garnish applied by default to elements that already need none of it — over-application is a real code-smell exactly parallel to lesson 15's "section soup" and lesson 16's "semantic-element-itis," just in the ARIA layer instead of the element-choice layer.

🏋️ Exercises

1 (easy): The gap, demonstrated. Build a div with role="button" and a click handler, styled to look like a button. With keyboard only (no mouse), try to activate it via Tab + Enter, then Tab + Space. Report what happens at each step and why, tying back to Part 1's "announces but doesn't behave" claim.
Expected

Tab: the div is skipped entirely — no tabindex means it's not in the tab order at all, regardless of role. Even after adding tabindex="0": Enter does nothing (no keydown handler), Space scrolls the page (default browser behavior for Space, unprevented). Every one of these gaps is something a real <button> would have handled automatically — the exercise makes Part 1's abstract claim concrete and specific.

2 (easy): Live region, minimal. Build a "Save" button that, on click, updates an aria-live="polite" region's text to "Saved at [time]". Test with a screen reader if available; otherwise verify in DevTools' Accessibility panel that the live region's properties are set correctly, and inspect that the container existed in the DOM before the first update.
Solution shape 
<div aria-live="polite" id="status"></div>   <!-- empty, present from page load -->
<button type="button" id="save">Save</button>
<script>
save.addEventListener("click", () =>
  status.textContent = `Saved at ${new Date().toLocaleTimeString()}`);
</script>

If you accidentally created the status div dynamically ONLY on click (rather than rendering it empty upfront), that's the exact bug Part 4 warns about — rebuild it the correct way and note the difference for yourself.

3 (medium): Accordion, ARIA-correct. Build a custom accordion (NOT using details/summary — the point is practicing the ARIA pattern directly) with 3 panels: each trigger is a real button with aria-expanded, each panel has aria-labelledby pointing to its trigger, and clicking toggles both the visual state AND aria-expanded together, every time, in the same code path.
Solution core 
<h3><button type="button" aria-expanded="false" aria-controls="panel-1" id="trig-1">
  Shipping</button></h3>
<div id="panel-1" role="region" aria-labelledby="trig-1" hidden>...</div>

<script>
trigger.addEventListener("click", () => {
  const expanded = trigger.getAttribute("aria-expanded") === "true";
  trigger.setAttribute("aria-expanded", String(!expanded));   // ONE code path, always in sync
  panel.hidden = expanded;
});
</script>

The critical discipline: aria-expanded and the actual hidden state are set in the SAME function call, so they structurally cannot drift apart — this is the concrete fix for the "state set once, never updated" mistake from the table above.

4 (medium): aria-hidden vs inert, felt. Build a simple modal- like overlay (doesn't need to be a real dialog element for this exercise) with background content containing a link. First hide the background with aria-hidden="true" alone and Tab through the page — note the focus lands on the hidden link with no announcement. Then switch to inert and repeat.
Expected contrast

With aria-hidden alone: Tab still visits the background link (visible focus outline appears on it) but a screen reader announces nothing — a genuine, disorienting dead zone, exactly Rule 4's warning made concrete. With inert: Tab skips the background entirely, landing only on elements inside the overlay — this is the complete, correct behavior, and precisely what dialog. showModal() gives you automatically without writing any of this yourself.

5 (hard): Minimal combobox, attempted. Build the simplest possible version of Part 5's combobox pattern: a text input filtering a list of 5 cities, arrow keys moving a highlighted option via aria-activedescendant (NOT moving real focus), Enter selecting, Escape closing. After building it, list which parts of the FULL requirement list from Part 5 your version still doesn't cover.
Expected gap list

Even a careful attempt commonly still misses: aria-selected on options as highlight moves (easy to forget alongside activedescendant), Home/End key support, proper focus return to the input on Escape if it had moved anywhere, mobile/touch behavior (this pattern is keyboard-model-first by design), and — almost certainly — real cross-screen-reader testing, since that requires actual AT sessions this exercise likely didn't include. Producing this honest gap list yourself, having just built the "simple" version, is the exercise's real payoff: it's the fastest way to viscerally understand why Part 5 called this a multi-day component rather than an afternoon's work.

6 (hard): ARIA audit and reduction. Find (or recall) a component you've built anywhere in this track, or search a real open-source project's markup, for ARIA attributes. For each one found, categorize: (a) legitimate — no native equivalent exists, (b) legitimate — patching a semantic loss, (c) redundant on an already-correct native element, (d) genuinely missing required keyboard behavior to back up its claimed role. Report your findings with one fix per category-(c)/(d) finding.
What a good audit looks like

Real codebases very commonly contain category (c) findings — aria-label duplicating a button's own visible text, role="button" on an actual <button>, redundant aria-required alongside a native required attribute — these accumulate from defensive/habitual addition rather than deliberate design and are safe, easy wins to simplify. Category (d) findings (a role claiming interactivity with no matching keyboard handler) are the more serious ones worth flagging prominently, since they're Rule 3 violations actively misleading AT users rather than just harmless redundancy. If your audit found zero (d) issues, that's a genuinely good sign about the codebase — but re-check any custom widgets specifically, since that's disproportionately where they hide, consistent with this lesson's Part 5 case study and lesson 22's closing interview question.

🎯 Key Takeaways

  • ARIA edits the accessibility tree's announced role/name/state — it changes NOTHING about behavior, keyboard handling, or focus. The First Rule of ARIA: use native HTML first, always, when available.
  • "No ARIA is better than bad ARIA" — a role that promises capability the element doesn't deliver (Rule 3 violation) actively misleads users, worse than an honest, unstyled plain element.
  • Roles (what kind of thing), properties (mostly-static relationships: aria-label, aria-describedby), states (dynamic, must-stay-synced: aria-expanded, aria-checked, aria-invalid) — three categories, different maintenance obligations.
  • Live regions (aria-live="polite"/"assertive") announce changes with no user-initiated focus shift — container must exist in the DOM before you populate it; assertive is for genuine errors only.
  • The combobox pattern's full requirement list (activedescendant, synced states, complete keyboard contract, cross-AT testing) is why battle-tested libraries beat hand-rolled widgets for anything beyond a learning exercise.
  • inert (hides from AT AND removes from tab order AND blocks interaction) is the correct tool where aria-hidden alone leaves a focusable, unannounced dead zone — exactly what dialog.showModal() applies for you automatically.
