Audio & Video Media & Embedding

What you'll master: video/audio with correct codec fallback, the autoplay policy maze (and why your video "doesn't work" in production), captions/subtitles via track (an accessibility requirement, not a nicety), the JS media API surface, and the "silent movie" trick replacing GIFs. Prerequisites: Responsive Images (the source-fallback pattern repeats here).

Part 1 — video, Fully Specified

<video controls width="800" height="450" poster="thumbnail.jpg" preload="metadata">
  <source src="talk.av1.mp4" type="video/mp4; codecs=av01.0.05M.08">
  <source src="talk.webm" type="video/webm">
  <source src="talk.mp4" type="video/mp4">
  <track kind="captions" src="talk-en.vtt" srclang="en" label="English" default>
  <p>Your browser doesn't support video.
     <a href="talk.mp4">Download the talk</a> instead.</p>
</video>
  • controls — without it there's no UI at all (a silent, unusable black box) unless you're building fully custom controls via JS (Part 5). Always include it unless you truly have a custom control layer.
  • width/height — same CLS story as images (lesson 6): reserves the aspect ratio before the video loads.
  • poster — the still image shown before playback starts. Without one, the first frame (or black) shows, which loads slower and looks unintentional. Always set one — it's also your Largest Contentful Paint candidate if the video is above the fold, so treat it with the same rigor as a hero image (lesson 6 rules apply directly: sized, compressed, correct format).
  • preload — "metadata" (duration/dimensions only, cheap) is the sane default; "none" for below-the-fold video galleries (pure bandwidth savings); "auto" only when you're confident the user will actually play it.
  • The multiple <source> fallback works exactly like picture's format fallback (last lesson): browser tries each type top to bottom, uses the first it can play. Order best-compression-first (AV1 → WebM/VP9 → H.264 as the universal floor).
  • Fallback content (the paragraph) renders ONLY in browsers with zero video support — essentially unreachable today, but free insurance and a reminder that video is progressive enhancement over "here's a download link," not a hard requirement.

Part 2 — The Autoplay Maze (where most video bugs live)

<video autoplay muted loop playsinline>    <!-- the ONLY combination guaranteed to autoplay -->
  <source src="bg-loop.mp4" type="video/mp4">
</video>

Browsers aggressively block autoplay-with-sound — a deliberate, hard-won UX policy after years of surprise blaring ads. The rule every engineer eventually learns the hard way: autoplay reliably works only when combined with muted (Chrome, Safari, and Firefox all enforce this; autoplay alone silently fails and the video just... sits there, paused, with no error). Add loop for background-video effects, and playsinline — without it, iOS Safari forces fullscreen on any playback, muted-autoplay included, breaking the entire "video as background element" pattern.

Consequence for design: any autoplaying video is necessarily silent — if the content needs sound, it cannot autoplay; you present a poster and a play button instead, full stop. This is a real product constraint teams re-learn every time a marketing site launches with a "hero video" spec that assumed otherwise.

🔍 The GIF-replacement pattern: the once-standard "auto-playing silent looping clip" (a GIF) is now correctly a muted+loop+autoplay+playsinline video — same visual effect, files 5–20× smaller because video codecs compress motion far better than GIF's per-frame palette scheme. Any GIF still in production in 2026 is a performance bug waiting to be filed.

Part 3 — track: Captions Are Not Optional

<track kind="captions" src="talk-en.vtt" srclang="en" label="English" default>
<track kind="captions" src="talk-es.vtt" srclang="es" label="Español">
<track kind="descriptions" src="talk-desc.vtt" srclang="en" label="Audio descriptions">
<track kind="chapters" src="talk-chapters.vtt" srclang="en" label="Chapters">

Captions are a WCAG requirement (1.2.2, Level A — the lowest, non-negotiable conformance tier) for any prerecorded video with audio, and their value extends far past deaf/hard-of-hearing users: anyone watching muted on a train, in an office, with a sleeping baby nearby — a huge fraction of real-world video consumption is caption-driven regardless of ability. The .vtt (WebVTT) format is plain text and genuinely simple:

WEBVTT

00:00:00.000 --> 00:00:03.500
Welcome back to the show.

00:00:03.500 --> 00:00:07.200
Today we're talking about <i>streaming HTML</i>.

Four kind values, distinct purposes: captions (dialogue + non-speech sound cues like "[door slams]" — for deaf/hard-of-hearing viewers), subtitles (dialogue translation only, assumes you can hear), descriptions (narrated description of visual content, for blind users), chapters (navigation markers, surfaced in the native scrubber UI). default marks which track is on initially; multiple language tracks let users switch via the native controls menu — zero JS required for any of this.

Part 4 — audio: the Same Ideas, Simpler

<audio controls preload="metadata">
  <source src="episode-12.opus" type="audio/opus">
  <source src="episode-12.mp3" type="audio/mpeg">
</audio>

Same fallback/preload story, no width/height/poster/playsinline (nothing to visually reserve). Codec note: Opus is smaller than MP3 at equivalent quality for spoken word/podcasts — worth the fallback pair for a podcast platform. controlslist="nodownload" hints the browser to hide its native download button (a UX hint, not DRM — the file is still fully downloadable by anyone determined; never treat this as content protection).

Part 5 — The JS Media API (knowing it exists)

const v = document.querySelector("video");
v.play(); v.pause();                 // control playback
v.currentTime = 30;                  // seek
v.volume = 0.5; v.muted = true;
v.playbackRate = 1.5;                // speed control
v.addEventListener("ended", showRelatedVideos);
v.addEventListener("timeupdate", updateProgressBar);
v.requestFullscreen();               // enter fullscreen (Fullscreen API)

This is the layer custom video players (YouTube's own chrome, course platforms) build on — the native controls attribute is a complete solution for most cases, and hand-rolling a replacement means reimplementing keyboard shortcuts, scrubbing, volume, captions toggling, and fullscreen from scratch (echoes of lesson 16's dialog argument: check whether native controls truly fall short before reaching for a custom player library). When you DO need custom controls (branded player chrome), hide the native ones (controls removed) and rebuild the essential set — play/pause, seek, volume, captions toggle, fullscreen — as real buttons, not divs, per lesson 16's method.

Part 6 — Deep Dive: Why Multiple Formats Still Matter in 2026

H.264 (MP4) plays everywhere and is the safe universal floor; AV1 offers dramatically better compression (30–50% smaller at equal quality) but isn't universal on older devices/browsers; WebM/ VP9 sits between the two in both compression and support. The three-source fallback in Part 1 isn't legacy caution — it's the same "best format, graceful fallback" pattern as images (lesson 6), applied to video, and for high-traffic video (a landing page hero, a course platform) the bandwidth savings from AV1-first genuinely matter at scale. Pair this with type including exact codec strings (as shown) when you can — it lets the browser skip a source it can't decode without even requesting it, versus a bare video/mp4 which some browsers will request before discovering they can't play the specific codec inside.

Interview Questions

Q1 (beginner): Why does a video with just the autoplay attribute fail to play automatically in production?

Browsers block autoplay-with-sound by policy (an anti-annoyance measure) — autoplay alone requests audible playback, which gets silently blocked with no error thrown. The reliable combination is autoplay + muted (+ loop + playsinline for background-video use), which every major browser permits. This is one of the most common "why doesn't my video work" support tickets in real projects.

Q2 (beginner): Why does video replace GIF for short looping clips?

Video codecs (H.264/VP9/AV1) exploit temporal compression — encoding only what CHANGES between frames — while GIF's ancient per-frame palette format has no such optimization, making files 5-20x larger for the same visual result. The muted+loop+autoplay+playsinline combination reproduces GIF's "just plays, no controls, no sound" behavior exactly, at a fraction of the bytes.

Q3 (mid): What's the difference between captions, subtitles, and descriptions as track kinds, and which is a WCAG requirement?

captions: dialogue plus non-speech audio cues ("[phone rings]") for deaf/hard-of-hearing viewers — this is the WCAG 1.2.2 Level A requirement for any prerecorded video with audio. subtitles: dialogue translation only, assumes the viewer can hear (foreign-language content). descriptions: narrated description of visually-conveyed information, for blind viewers (a higher WCAG conformance level). Confusing captions with subtitles in an implementation often means deaf users get a translation feature they don't need instead of the sound-cue information they do.

Q4 (mid): Why does the video element accept multiple source children instead of one src with format negotiation via HTTP?

Same rationale as picture's format fallback: browser codec support varies (H.264 universal, WebM/ VP9 and AV1 progressively better-compressed but not universally decodable), and the browser is best positioned to pick the first format it can actually decode. Ordering sources best- compression-first with exact codec strings in type lets unsupported browsers skip a source without even requesting it — meaningfully better than server-side content negotiation, which would need to guess capability from User-Agent strings (an unreliable, constantly-drifting signal).

Q5 (senior): A course platform's video pages score poorly on LCP, and video-heavy pages have high bounce rates on mobile data plans. Diagnose and propose fixes.

LCP diagnosis: check whether the poster image is the LCP element and whether it's properly sized/ compressed/preloaded — a missing or oversized poster is a very common silent LCP killer on video pages (treat it exactly like a hero image per lesson 6/Performance-lesson rigor; consider rel="preload" for it if it's confirmed LCP). Bounce-rate/mobile-data diagnosis: check preload value — "auto" on every video on a listing page means the browser starts downloading megabytes for videos the user never plays; switch to preload="metadata" or "none" for any below-the-fold or gallery video, reserving "auto" for the single video the user has explicitly chosen to watch. Broader fix: lazy-load video elements below the fold (loading="lazy" works on img but NOT natively on video — use IntersectionObserver to swap in the source/poster only when nearing viewport), and confirm multi-format sources are actually AV1/WebM-first for the bandwidth win on supporting devices. The senior move is diagnosing LCP and data-usage as two separate problems with two separate root causes, not one "video is slow" ticket.

Q6 (senior): Product wants a fully custom-branded video player (custom scrubber, branded play button, no native chrome). Walk through your build, and what native features you must NOT lose in the process.

Remove the controls attribute, build the visual chrome as real elements per lesson 16's method (play/pause as button, scrubber as input type="range" bound to currentTime via timeupdate, volume similarly) — never as styled divs, since keyboard/AT parity is exactly what's at stake. Must-not- lose checklist: keyboard operability (space to play/pause, arrow keys to seek — native controls give this for free; a custom build must re-wire it), captions/track support (the browser's native track rendering can be kept even with custom controls — don't rebuild subtitle rendering from scratch unless there's a real design reason), fullscreen (Fullscreen API, with its own focus- management considerations), and — often forgotten — the video element itself needs an accessible name/context (aria-label or a visible heading) since removing native controls also removes the browser's default video landmark cues in some AT. Budget-wise: this is a multi-week component with an ongoing maintenance tail (browser media API quirks), not a weekend CSS skin — the same cost-honesty argument as lesson 16's combobox case, applied to video.

🏋️ Exercises

1 (easy): The reliable-autoplay checklist. Build a video with just autoplay (no muted) and confirm it does NOT play (check the Console — some browsers log a rejection). Add muted and confirm it plays. Add loop + playsinline and note (from the spec, since desktop won't show the difference) what playsinline fixes on iOS specifically.
Expected

Bare autoplay: paused, possibly a console warning about a blocked play() promise. With muted: plays immediately on load. playsinline's effect is invisible on desktop (it's iOS-only behavior) — the exercise's real lesson is that you must know this from the spec, since you can't observe every platform-specific media policy locally; note it in your README/component docs for the next developer who won't have an iPhone handy either.

2 (easy): GIF replacement. Find (or make) a short looping clip. Export it as an animated GIF and as an MP4 at equivalent visual quality (any converter/Squoosh- adjacent tool, or just note approximate sizes if you can't). Compare file sizes and build the video version with the full four-attribute combo.
Expected result

The MP4 should land at roughly 5-15% of the GIF's file size for the same clip — often the single biggest per-asset win available on a page using legacy GIFs. Confirm the video plays, loops, is silent, and never goes fullscreen unexpectedly (playsinline doing its job).

3 (medium): Captioned video, real file. Take any short video (record a 15-second clip on your phone if needed) and hand-write a 4-5 cue .vtt caption file matching the dialogue/narration timing. Wire it up with track kind="captions" default. Toggle captions on/off via the native controls menu (usually a "CC" icon) and confirm the text appears synced.
Solution shape 
WEBVTT

00:00:00.000 --> 00:00:03.000
Hi, this is a test caption file.

00:00:03.000 --> 00:00:06.500
Built for the LearnHub video lesson.

Getting the timestamps roughly right by watching the video and noting seconds is the whole exercise — you'll immediately understand why captioning services exist as a paid industry (accurate timing at scale is genuinely tedious manual work, which is exactly why WCAG treats it as a real requirement rather than an afterthought).

4 (medium): Multi-source fallback, verified. Build a video element with THREE source children (reuse one real file renamed/re-typed as a stand-in for the others if you can't actually transcode) in deliberately WRONG order (worst format first) versus correct order. Explain in one paragraph why order matters even though your test browser can't show a visible difference.
Answer

Since source-selection is "first supported type wins," listing a widely-supported format (H.264 MP4) FIRST means every modern browser stops there and never even considers your better- compressed AV1/WebM options below it — you'd ship the least-efficient format to 100% of visitors despite having better ones available. The fix is structural ordering (best compression first), which is invisible in casual testing on one browser but has real bandwidth cost at production scale — exactly the kind of bug that survives code review because "it plays fine for me."

5 (hard): Lazy video gallery. Build a page with 4 video elements below the fold, each with preload="none" and a poster, NOT autoplaying. Using IntersectionObserver, swap in the source (i.e., set the src or reveal source children) only when a video scrolls within ~200px of the viewport. Verify in the Network tab that videos outside the viewport generate zero requests until scrolled near.
Reference core 
const io = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const v = entry.target;
      v.querySelectorAll("source").forEach(s => s.src = s.dataset.src);
      v.load();                      // re-run source selection now that src is populated
      io.unobserve(v);
    }
  });
}, { rootMargin: "200px" });
document.querySelectorAll("video[data-lazy]").forEach(v => io.observe(v));

Author sources with data-src instead of src initially, so nothing downloads until the observer fires; v.load() is required to make the browser re-evaluate the now-populated sources. Confirm in Network: 0 video bytes at page load, requests appearing one by one as you scroll — this is the loading="lazy" equivalent that video doesn't get natively, built by hand.

6 (hard): Minimal custom player. Build a video WITHOUT the controls attribute, plus real button elements for play/pause and a real input type="range" for seeking, wired via the JS API from Part 5 (play/pause/currentTime, synced via timeupdate). Test with keyboard only (Tab to buttons, Enter/Space to activate, arrow keys on the range) and confirm full operability without a mouse.
Core wiring 
playBtn.addEventListener("click", () => v.paused ? v.play() : v.pause());
v.addEventListener("timeupdate", () => { seek.value = (v.currentTime / v.duration) * 100 || 0; });
seek.addEventListener("input", () => { v.currentTime = (seek.value / 100) * v.duration; });

Because both controls are REAL button/input elements (lesson 16's method, applied), keyboard operability is automatic — Tab reaches them in DOM order, Space/Enter activates the button, arrow keys move the range natively. If you'd built these as styled divs, you'd now be manually implementing everything this exercise got for free — the exact cost-comparison lesson 16 argued for, now felt firsthand on a real component.

🎯 Key Takeaways

  • autoplay reliably works only paired with muted (+ loop + playsinline for background-video effects) — browsers block audible autoplay by policy, silently.
  • Multi-source fallback mirrors picture's format-fallback pattern: best compression (AV1/WebM) first, universal H.264 as the floor, first-supported-type wins.
  • Captions (track kind="captions" + .vtt) are a WCAG 1.2.2 requirement for any video with audio — and serve a much larger silent-viewing audience regardless of ability.
  • poster is your video's hero image — size, compress, and treat it with full Performance-lesson rigor since it's often the LCP element.
  • Video replaces GIF for short loops: same muted+loop+autoplay+playsinline effect at 5-20x smaller file sizes via real temporal compression.
  • Native controls give keyboard operability, captions UI, and fullscreen for free — a custom player rebuilding these from real button/input elements (never divs) is a real, ongoing-maintenance commitment.
