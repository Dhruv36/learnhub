Keyboard Navigation & Focus Accessibility

What you'll master: the complete focus model (how it actually works under the hood), skip links done correctly, focus management for dynamic UI (the single most-botched accessibility pattern in modern JS apps), keyboard traps and how to avoid building one by accident, and roving tabindex for composite widgets. This closes the Accessibility section by consolidating everything Forms UX (lesson 14), lesson 16, and the ARIA lesson touched on keyboard behavior into one complete model. Prerequisites: ARIA Deep Dive.

Part 1 — The Focus Model, Precisely

At any moment, exactly one element in the document has focus — the target of keyboard input and the anchor point for AT navigation. Three facts govern it completely:

1. FOCUSABLE BY DEFAULT: a, button, input, select, textarea, details/summary, audio/video[controls]
2. NOT FOCUSABLE BY DEFAULT: div, span, p, li, img, and every other non-interactive element
3. TAB ORDER = DOM ORDER of focusable elements (lesson 14's rule, restated as the actual mechanism)

document.activeElement always tells you the current focus target (try it in the Console right now on this page). :focus matches it in CSS; :focus-visible (lesson 14) matches only when the browser's heuristic decides the focus indicator should show — keyboard nav, generally; not a mouse click on most elements. Two methods move focus programmatically: element.focus() and — critically — element.blur(), which returns focus to nothing (the body), a rarely-correct choice since it strands the user; moving focus TO the next sensible place is almost always what you actually want instead.

Part 2 — Skip Links: the Simplest, Highest-Value Pattern

<body>
  <a class="skip-link" href="#main">Skip to main content</a>
  <header>...30 nav links, a search box, a language switcher...</header>
  <main id="main">...</main>
</body>

<style>
.skip-link {
  position: absolute; top: -100px; left: 0;      /* off-screen until focused */
  background: #000; color: #fff; padding: 12px;
}
.skip-link:focus { top: 0; }                      /* visible ONLY when tabbed to */
</style>

Without a skip link, a keyboard user must Tab through EVERY link in your header — potentially dozens — on EVERY single page, before reaching the actual content, every single visit. This is lesson 15's "main is the payoff of skip-to-content links" claim, now with the actual implementation: a real, visible-on-focus link as the very first focusable element on the page, jumping straight to #main. The off-screen-until-focused CSS pattern matters specifically because a permanently visible skip link looks broken/unstyled to sighted mouse users who never need it — this is the correct compromise, not a corner cut. Multiple skip links (to nav, to search, to main) are legitimate on especially link-dense pages, but one to main covers the overwhelming majority of real benefit for the effort.

Part 3 — Focus Management for Dynamic UI (the big one)

This is where most real-world keyboard-accessibility bugs live, because it's invisible to sighted mouse-driven testing: whenever your JavaScript adds, removes, or reveals content, ask "where should focus go now?" — the browser never answers this question for you.

Case 1: opening a modal

function openModal() {
  dialog.showModal();          // if using <dialog> (lesson 16): focus moves automatically
  // if hand-building a modal instead, YOU must do this manually:
  const previouslyFocused = document.activeElement;
  modal.querySelector("h2, button, input")?.focus();   // move focus INTO the modal
  modal.addEventListener("close", () => previouslyFocused.focus(), { once: true });  // and back
}

Two moments, two required focus moves: INTO the modal on open (so keyboard/AT attention actually follows the visual change), and BACK to the triggering element on close (so the user isn't left stranded on a now-vanished element, or worse, silently reset to the top of the page). dialog.showModal() gives you both for free — the concrete payoff, once again, of lesson 16's platform-first argument.

Case 2: route changes in a single-page app

function onRouteChange(newPageTitle) {
  document.title = newPageTitle;
  const heading = document.querySelector("h1");
  heading.setAttribute("tabindex", "-1");     // make an otherwise-non-focusable h1 a focus TARGET
  heading.focus();                             // move focus there — screen readers announce it
  liveRegion.textContent = `Navigated to ${newPageTitle}`;   // and confirm via live region too
}

Traditional full-page navigation does this automatically — a fresh page load naturally resets focus to the top and the new <title> gets announced. Client-side routing (React Router, etc.) breaks this silently: the URL changes, the DOM changes, but focus stays exactly where it was — often on a nav link that's now visually in the wrong context — and nothing announces the navigation happened at all. This is the single most common accessibility bug in modern SPA codebases, precisely because it's entirely invisible to visual/mouse testing and only surfaces under actual keyboard/screen-reader use.

Case 3: removing the focused element

function deleteItem(id) {
  const item = document.getElementById(id);
  const next = item.nextElementSibling || item.previousElementSibling;
  item.remove();
  (next ?? list).focus();      // NEVER let focus silently fall back to <body>
}

If a focused element is removed from the DOM mid-interaction (a "delete" button removing its own row), focus reverts to <body> by default — from the user's perspective, focus simply vanishes with no landing point, and the next Tab press starts over from the top of the page. Always redirect focus deliberately to a logical neighbor.

Part 4 — Keyboard Traps: the Cardinal Sin

A keyboard trap is any state where a keyboard user CANNOT leave an element/region using standard keys — WCAG 2.1.2 (Level A, the baseline conformance tier from lesson 22) calls this out explicitly because it's so severe: it doesn't just inconvenience, it can strand a user on your page entirely, unable to proceed OR go back.

<!-- The classic accidental trap: a custom modal that intercepts Tab but never handles Escape,
     and whose Tab-cycling logic has an off-by-one bug that never releases focus. -->

// Correct: EVERY custom overlay must implement Escape-to-close, always, no exceptions:
modal.addEventListener("keydown", (e) => {
  if (e.key === "Escape") closeModal();
});

Traps happen almost exclusively in hand-rolled focus-trapping code for custom modals/menus (another entry on the growing list of reasons dialog.showModal() is the better default — its trap is correctly implemented and Escape-exits by construction). If you ever DO hand-build focus trapping (a genuine, rare need — e.g., a modal in an environment where dialog isn't viable), the non- negotiable rule: Escape must always work, tested explicitly, every time, with no code path that can skip binding it.

Part 5 — Roving Tabindex: One Tab Stop, Arrow-Key Navigation Within

For composite widgets (a toolbar, a tab list, a menu, a radio group) the correct keyboard model isn't "Tab visits every item" — it's one Tab stop for the whole widget, arrow keys move within it, exactly how native radio buttons already behave (lesson 9 — you experienced this pattern without its name). The technique:

<div role="toolbar" aria-label="Text formatting">
  <button tabindex="0">Bold</button>      <!-- the ONE tab stop, currently -->
  <button tabindex="-1">Italic</button>
  <button tabindex="-1">Underline</button>
</div>

<script>
toolbar.addEventListener("keydown", (e) => {
  if (!["ArrowRight", "ArrowLeft"].includes(e.key)) return;
  const buttons = [...toolbar.querySelectorAll("button")];
  const current = buttons.indexOf(document.activeElement);
  const next = buttons[(current + (e.key === "ArrowRight" ? 1 : -1) + buttons.length) % buttons.length];
  buttons[current].tabIndex = -1;    // move the "0" to the new position — ROVING
  next.tabIndex = 0;
  next.focus();
});
</script>

The name "roving tabindex" describes exactly this: tabindex="0" ROVES from element to element as arrow keys move the active selection, while every other member of the group sits at tabindex="-1" (lesson 14's "focusable by script only" value, repurposed here as "focusable by arrow-key logic only"). The payoff: Tab moves through the PAGE efficiently (one stop per widget, not one per button inside it — imagine Tabbing through 40 toolbar buttons individually to reach the next section), while arrow keys give fine-grained control within the widget, matching the exact mental model users already have from native radio groups, tab lists, and menus.

Part 6 — The Complete Keyboard Test Protocol

Formalizing lesson 14's "unplug the mouse" exercise into a repeatable protocol — run this on every new interactive component you ship, for the rest of your career:

1. Tab from the very top of the page. Is the skip link the first stop? Does it work?
2. Continue tabbing. Is every interactive element reachable? Is the ORDER logical (matches
   visual/reading order, not scattered by tabindex tricks)?
3. Is focus ALWAYS visible? (No outline:none without a :focus-visible replacement — lesson 14.)
4. For each composite widget (menu, tabs, toolbar): do arrow keys work as expected? Is Tab a
   single stop per widget, not one per item?
5. Open every modal/menu/dropdown you can. Does Escape close it EVERY time? Does focus land
   inside on open, and return to the trigger on close?
6. Trigger every dynamic update (delete a row, submit a form, navigate a route). Does focus
   land somewhere SENSIBLE afterward — never silently reverting to <body>?
7. Can you complete every core task on the page — genuinely, start to finish — without ever
   touching the mouse?

Interview Questions

Q1 (beginner): What determines tab order by default, and which elements are focusable without any extra attributes?

Tab order = DOM order of naturally focusable elements. Focusable by default: a[href], button, input, select, textarea, details/summary, and media elements with controls. Everything else (div, span, p, li, img, etc.) is not focusable unless given tabindex="0" — and per lesson 14, arbitrary tabindex ordering (1, 2, 3...) should never be used; correct DOM order is the only maintainable way to control tab sequence.

Q2 (beginner): Why is a skip link hidden until focused, rather than always visible or always hidden?

Always visible looks like unstyled/broken UI to sighted mouse users who have no need for it and will simply reach for the content directly; always hidden (display:none) removes it from the tab order entirely, defeating its purpose for the keyboard users who genuinely need it. Off-screen positioning that becomes visible on :focus serves both audiences correctly — invisible until the exact moment it's needed, then fully visible and usable.

Q3 (mid): Why do single-page app route changes routinely break accessibility even when the visual page updates correctly?

Traditional full navigation resets focus to the top of a fresh document and the browser announces the new <title> automatically — behavior that's free with real page loads. Client-side routing changes the URL and DOM but does neither: focus silently stays wherever it was (often a now-contextually-wrong nav link), and nothing announces that navigation happened at all. The fix must be built explicitly on every route change: move focus to a heading (via tabindex="-1" + .focus()), update document.title, and ideally announce via a live region — none of it automatic in an SPA architecture.

Q4 (mid): What is roving tabindex, and why is it the correct pattern for a toolbar or tab list instead of making every item individually tabbable?

Roving tabindex keeps exactly one item in a composite widget at tabindex="0" (the current Tab stop) while every other member sits at tabindex="-1"; arrow keys move which item holds the "0" and shift focus within the group, while Tab itself only ever stops once per widget. This matches the interaction model users already know from native radio groups (lesson 9) and lets Tab traverse the PAGE efficiently — imagine a 40-button toolbar where Tab had to visit all 40 individually before reaching the next page section; roving tabindex is what prevents that.

Q5 (senior): Your team's design system ships a custom Modal component. Code review finds it doesn't restore focus to the trigger element on close, and occasionally traps focus if closed via a background-overlay click rather than the close button. Diagnose the architecture problem and the fix.

Diagnosis: the component almost certainly hand-implements focus-trapping and doesn't route every close path (button click, Escape key, overlay click, and potentially a programmatic/API close) through a SINGLE close function — the trap and lost-focus-restoration bugs are symptoms of having multiple, inconsistent code paths that each partially handle focus instead of one централized path that always does. First recommendation: evaluate replacing the hand-rolled implementation with the native <dialog>.showModal() (lesson 16) entirely — it correctly restores focus to the invoking element automatically on close via ANY method (Escape, form submission, .close()) and can't trap by construction, eliminating this whole bug class at the platform level rather than patching application code. If a hand-rolled component must remain (a real constraint sometimes, e.g. needing animation control dialog doesn't offer), the fix is architectural: ALL close triggers (button, Escape, overlay click, programmatic API) must call one shared closeModal() function that captures the previously-focused element on open and restores it on every close path without exception — no per-trigger custom logic that can drift out of sync with the others.

Q6 (senior): Design the focus-management contract for a data table with inline-editable cells (click or Enter on a cell turns it into an input; Escape cancels and reverts; Enter/Tab commits and moves to the next cell). What needs deliberate handling at every transition?

Every state TRANSITION is a focus-management decision point, not just the obvious open/close cases: (1) entering edit mode — focus must move INTO the newly-rendered input immediately (a common bug: the input renders but focus stays on the now-replaced static cell content, stranding keyboard users); (2) committing via Enter — focus should move to the NEXT logical cell (typically down a row, matching spreadsheet conventions users expect) after the edit reverts to static display, not vanish or reset to the table's first cell; (3) committing via Tab — focus moves to the next cell in reading order and, per this table's likely composite-widget nature, may warrant roving tabindex across cells rather than every cell being independently Tab-stoppable, exactly like Part 5's toolbar case; (4) canceling via Escape — the cell must revert to its ORIGINAL pre-edit value (not just close the input, but genuinely restore state) and return focus to the static cell, which itself needs a sensible focusable/announceable state even when not editing; (5) validation failures on commit attempt — focus must STAY in the invalid input (per the Forms Constraint Validation lesson's pattern) rather than advancing, with the error announced via aria-describedby/live region, not silently blocking with no feedback. The senior framing: a component with N interaction states has roughly N focus-transition decisions to make deliberately — none of them are optional, and "we'll test it visually" catches none of them, which is precisely why this entire lesson's protocol (Part 6) exists as a keyboard-first, not visual-first, verification method.

🏋️ Exercises

1 (easy): Skip link, built and felt. Add a correctly-implemented skip link to your most link-dense page from earlier in this track (ideally one with a full nav menu). Load the page, press Tab ONCE, and confirm the skip link appears and works. Then temporarily remove it and count how many Tab presses it takes to reach main content without it.
What to notice

The tab-count-without-a-skip-link number (often 10-30+ on a real header with nav + search + utility links) is usually higher than people expect before actually counting — this concrete number is far more persuasive in a design/product conversation than an abstract accessibility argument, and worth remembering as a talking point.

2 (easy): document.activeElement, live. Open the Console on any page. Type document.activeElement, then Tab a few times and check it again each time. Click various elements (a heading, a paragraph, a button) and check after each — confirm non-interactive elements never become activeElement via click alone (they need tabindex first).
Expected

Clicking a button/link updates activeElement to that element; clicking a plain paragraph or heading leaves activeElement unchanged (still whatever was previously focused, often body) — direct, hands-on confirmation of Part 1's "not focusable by default" list, using the exact property that governs the entire focus model.

3 (medium): Modal focus, both directions. Build a simple dialog-based modal (lesson 16's showModal). Add a text input inside it that receives focus on open (verify: Tab from the trigger button, press Enter/click to open — does focus land on the input, or somewhere less useful like the dialog's outer container?). Close via Escape and confirm focus lands back exactly on the trigger button, not somewhere else on the page.
Notes

dialog.showModal() focuses the FIRST focusable element inside by default unless you set autofocus explicitly on a specific one — if your input isn't first in DOM order among focusable elements, focus may land elsewhere (a real, easy-to-miss detail); add autofocus to the input directly to guarantee it regardless of position. Confirming focus RETURNS to the trigger on close (not just that the modal visually closes) is the check most people skip — verify it explicitly by checking document.activeElement after Escape.

4 (medium): SPA route-change simulation. Without a real router, simulate the pattern: a page with two "views" (divs toggled via a button, faking navigation), each with its own h1. On "navigating," update document.title, move focus to the new view's h1 (via tabindex="-1"), and announce via a polite live region. Verify with the Accessibility panel that focus genuinely moved, not just that the visual content changed.
Reference core 
function navigateTo(view) {
  document.title = view.dataset.title;
  const h1 = view.querySelector("h1");
  h1.setAttribute("tabindex", "-1");
  h1.focus();
  liveRegion.textContent = `Navigated to ${view.dataset.title}`;
}

Check document.activeElement after calling this — it should be the NEW view's h1, not still the button that triggered navigation. This three-line function is the entire fix for what Q3's interview question calls "the #1 accessibility bug in modern SPA codebases" — worth keeping as a reference snippet for real projects.

5 (hard): Roving tabindex toolbar, from scratch. Build Part 5's toolbar completely (3+ buttons, arrow-key navigation, correct roving tabindex management). Test: Tab into it (one stop), arrow through all items, Tab OUT continues to the next page element (not back into the toolbar), then Shift+Tab backward through the page correctly re-enters at the toolbar's current position.
Verification checklist

The trickiest part to get right: after arrowing to item 3 and then Tabbing away, a subsequent Shift+Tab back into the toolbar should land on item 3 (wherever tabindex="0" currently sits), not reset to item 1 — this is the actual point of "roving" rather than a fixed tab stop, and the detail most first attempts get wrong. If Shift+Tab always re-enters at item 1 regardless of where you last arrowed to, your tabIndex updates aren't persisting correctly between interactions.

6 (hard): Full protocol audit, applied. Take the most complex interactive page you've built anywhere in this track (a form with validation, a component with a modal, anything multi-part). Run Part 6's complete 7-step protocol against it, writing down the result of each step. Fix any failures found, then re-run the full protocol to confirm.
What counts as success

A genuine pass means completing step 7 — an entire core task, start to finish, mouse fully unplugged — not just individually passing steps 1-6 in isolation. It's common to pass every individual check yet still fail step 7 because of some interaction BETWEEN components (e.g., a modal that individually manages focus correctly, but whose trigger button itself isn't reachable due to an earlier, unrelated tab-order issue) — which is exactly why the protocol ends on the holistic, end-to-end check rather than stopping at the component-level checks alone.

🎯 Key Takeaways

  • Focus is a single, well-defined pointer (document.activeElement); tab order is DOM order of naturally-focusable elements — never fight this with numeric tabindex.
  • Skip links (off-screen until :focus, jumping to #main) are the highest-value-per-effort accessibility pattern that exists — one link, dozens of saved Tab presses on every page visit.
  • Every dynamic UI change is a focus-management decision: modals move focus in and restore it on close; SPA route changes need explicit focus + title + live-region handling (the #1 real-world SPA a11y bug); removed elements need focus redirected to a logical neighbor.
  • Keyboard traps (WCAG 2.1.2, Level A) are a severe, stranding failure — every custom overlay needs Escape-to-close with no exceptions; dialog.showModal() avoids this class of bug by construction.
  • Roving tabindex (one item at tabindex="0", the rest at -1, arrow keys move the "0") is the correct model for composite widgets — one Tab stop per widget, matching native radio-group behavior.
  • The complete keyboard-only test protocol (Part 6) is the practical, repeatable synthesis of this entire section — run it on every new interactive component.
