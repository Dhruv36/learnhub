Text & Headings Basics

Text elements carry semantics — machine-readable meaning. Screen readers navigate by headings, Google extracts structure from them, reader modes rebuild your article from them. Choosing the right text element is the daily craft of HTML.

Headings — the Document Outline

<h1>Pricing</h1>                        ← the page's single topic
  <h2>Plans</h2>
    <h3>Free tier</h3>
    <h3>Pro tier</h3>
      <h4>Pro annual discount</h4>
  <h2>Frequently Asked Questions</h2>
RuleReason
Exactly one <h1>It's the page's identity for assistive tech & search. (HTML5 "outline algorithm" promised per-section h1s — browsers never implemented it; the plan was officially dropped.)
Never skip levels downward (h2→h4 ❌)Screen-reader users navigating by heading level hit a hole in the hierarchy
Don't pick by font sizeSizes are CSS's job: h2 { font-size: … }. A "small heading" need is a styling need.
Headings label the content below themIf nothing follows, it's not a heading — probably a <p> with bold styling

📊 Field data: screen-reader surveys (WebAIM) consistently report ~65–70% of users navigate new pages primarily by headings — heading discipline is the single highest-impact a11y habit.

Paragraphs & Line Breaks

<p>Each paragraph of prose gets its own p element. Browsers add default
   vertical margins; override with CSS, never with empty paragraphs.</p>

<p>221B Baker Street<br>London NW1 6XE</p>   <!-- br = real line break WITHIN content -->
<hr>   <!-- thematic shift between topics — semantic, not just a gray line -->

Character Entities — Escaping Reserved Characters

&lt; → <      &gt; → >      &amp; → &      &quot; → "      &#169; / &copy; → ©
&nbsp; → non-breaking space (keeps "10&nbsp;km" on one line — NOT a spacing tool)

Only & and < strictly require escaping in content (plus quotes inside attribute values). This rule graduates into a security boundary in the XSS lesson: unescaped user text is how script injection happens.

Interview Questions

Q1: strong vs b, em vs i — do they matter if they render the same?

Yes: rendering is identical but meaning isn't. strong/em convey importance/emphasis to assistive tech and parsers; b/i are purely stylistic ("stylistically offset" text like keywords or ship names). The senior answer notes: if the styling has no meaning, arguably CSS should do it, not b/i.

Q2: How many h1s should a page have and why?

One. Multiple h1s aren't invalid, but the sectioning-based outline that would have justified them was never implemented by browsers and was removed from the spec — so assistive tech and SEO treat h1 as the page topic. One h1, then nested h2–h6 without skips.

Q3: When is <br> legitimate?

When the line break is part of the content's meaning: postal addresses, poetry, song lyrics. Never for visual spacing between blocks — that's margin.

🏋️ Practice:
1. Mark up a recipe: title (h1), sections for Ingredients and Steps (h2), a warning using strong, timings using <time>.
2. Write a changelog entry using del/ins for a price change and abbr for two acronyms.
3. In DevTools, run document.querySelectorAll("h1,h2,h3,h4") on a news article and check: does the outline make sense with the styling removed?

🎯 Key Takeaways

  • Headings form the outline: one h1, no skipped levels, chosen by hierarchy not size.
  • strong/em carry meaning; b/i are stylistic; span is the last resort.
  • Prefer the specific element (time, abbr, kbd, mark) — each one is data for some consumer.
  • Escape & and <; the habit becomes your first XSS defense.
