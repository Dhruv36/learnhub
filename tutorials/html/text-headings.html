Text: Paragraphs & Headings Foundations

What you'll master: structuring any document with headings and paragraphs the way screen readers, search engines, and reader modes expect; the heading-hierarchy rules that accessibility audits actually check; block vs inline as a load-bearing concept; and the supporting cast (blockquote, pre, hr, br) used correctly. Prerequisites: lesson 1 and lesson 2.

Part 1 — Why Text Elements Matter More Than They Look

A paragraph in a <p> and the same words in a <div> look almost identical. So why care? Because three audiences besides sighted visitors consume your markup:

  • Screen-reader users navigate by structure. Surveys of screen-reader users (WebAIM, repeatedly) show the #1 way they explore a page is jumping between headings — they pull up a list of your h1–h6 and treat it as a table of contents. No real headings = no map = they must listen to the entire page linearly.
  • Search engines extract structure. Headings tell Google what a page and its sections are about; heading text is weighted in ranking and shapes snippets and "jump to" links.
  • Machines rebuild your content. Reader mode, RSS readers, "listen to this article" features, AI summarizers — all reconstruct your document from its semantic skeleton. Style-only markup gives them nothing.
💡 The test to internalize: strip the CSS and the document should still make sense — a hierarchy of headings, paragraphs under them, lists where things enumerate. Try it on this very page: DevTools → Elements → delete the stylesheet link. Still readable. That's the target for everything you build.

Part 2 — Headings: The Six Levels and the Three Rules

<h1>Building a REST API</h1>                    ← THE topic of the page
  <h2>Designing the endpoints</h2>               ← major section
    <h3>Naming conventions</h3>                  ← subsection
    <h3>Versioning</h3>
  <h2>Authentication</h2>
    <h3>API keys</h3>
      <h4>Key rotation</h4>                      ← sub-subsection (rare but fine)

Headings form an outline — exactly like a book's table of contents. The number is a structural depth, not a font size. Three rules cover every real situation:

  1. One <h1> per page, stating what the page is. (The spec technically allows more; accessibility tooling, SEO practice, and every style guide say one. Just use one.)
  2. Never skip levels going down. h1 → h2 → h3, not h1 → h3. Skips read as broken structure to screen-reader users ("did I miss a section?") and are flagged by audit tools like Lighthouse and axe. Going back up to any higher level is fine — that's just starting a new section.
  3. Choose the level by structure, never by size. "h4 looks about right here" is the classic beginner mistake. If the correct level looks wrong, restyle it in CSS: h2 { font-size: 1.2rem } is completely legitimate. Structure and appearance are separate axes — that's the whole layer model from lesson 1.

Worked example: marking up a real article

Take this content and decide its heading structure before peeking:

"Sourdough at Home" (the piece)
  intro paragraph
  "Ingredients" — flour... water...
  "Method" with three phases: "Day 1: The starter", "Day 2: Bulk ferment", "Day 3: Bake"
  "Troubleshooting" — "Dense crumb", "Too sour"
Solution + reasoning 
<h1>Sourdough at Home</h1>
<p>intro...</p>
<h2>Ingredients</h2>
<h2>Method</h2>
  <h3>Day 1: The starter</h3>
  <h3>Day 2: Bulk ferment</h3>
  <h3>Day 3: Bake</h3>
<h2>Troubleshooting</h2>
  <h3>Dense crumb</h3>
  <h3>Too sour</h3>

The days are h3 because they're phases of the Method, not top-level sections. Notice you went h3 → h2 at "Troubleshooting": moving back up is always legal. If you had made the days h2, nothing would look different — but the outline would claim baking day is a peer of "Ingredients", which is false. Outlines encode claims about your content.

Part 3 — Paragraphs, and Block vs Inline (a load-bearing concept)

<p> holds a paragraph: one self-contained point, usually several sentences. Browsers add vertical margins between paragraphs — never create "spacing paragraphs" (<p>&nbsp;</p>) or stack <br>s; spacing belongs to CSS margins.

Now the concept that explains half of HTML's rules. Every text-level element is historically either:

Block-levelInline
BehaviorStarts on a new line; fills available widthFlows within a line of text, like a word
Examplesp, h1-h6, ul, ol, li, blockquote, pre, div, tablea, em, strong, code, span, img, abbr
May containDepends on element (p: inline only!)Other inline elements + text (a is special-cased)

The rule that bites everyone once: <p> may only contain inline content. Put a block element (div, ul, another p) inside a paragraph and the parser closes the paragraph before it — you met this repair in lessons 1–2; now you know the rule behind it. Nested paragraphs are impossible in HTML for the same reason: <p>a<p>b parses as two sibling paragraphs.

🔍 Senior framing: modern specs replaced "block/inline" with content categories ("flow content", "phrasing content") — <p> accepts only phrasing content. And CSS display can restyle anything (span { display:block }) — but that changes rendering, not parsing: a div inside a p still explodes no matter what CSS says. Parsing rules are baked in at the HTML layer.

Part 4 — The Supporting Cast, Used Correctly

<br> — line break, almost always wrong

✅ Legitimate: line structure IS the content
<p>Maya Chen<br>420 Oak Street<br>Portland, OR</p>     (address)
<p>Roses are red<br>Violets are blue</p>                (verse)

❌ Wrong: <br><br> to fake paragraph spacing → use separate <p>s
❌ Wrong: <br> to force a heading to wrap "nicely" → that's CSS's job (max-width)

<hr> — thematic break

Semantically "the scene changes" — a topic shift within a section (common in fiction and essays). It is not "a horizontal line for decoration"; decorative lines are CSS borders. Screen readers announce <hr> as a separator, so decorative use adds noise.

<blockquote>, <cite> — quoting at block level

<figure>
  <blockquote>
    <p>Talk is cheap. Show me the code.</p>
  </blockquote>
  <figcaption>— Linus Torvalds, <cite>linux-kernel mailing list</cite></figcaption>
</figure>

Notes: the quoted text sits in <p>s inside the blockquote; the attribution lives outside the blockquote (the author didn't say their own name — a real spec point) — the figure/figcaption wrapper is the standard pattern; <cite> marks the title of a work, not a person's name. Inline quotes use <q>, which auto-inserts locale-correct quotation marks: <q>quoted phrase</q>.

<pre> — preformatted text

The one element where whitespace does not collapse: spaces, newlines, and indentation render exactly as typed, in monospace. Essential for code blocks (idiom: <pre><code>...</code></pre> — pre for formatting, code for semantics), ASCII diagrams, terminal output. Two gotchas: the indentation of your source becomes visible (align pre content to column 0 or trim in your templating), and < inside code samples must still be escaped as &lt; — pre does not turn off parsing.

Character entities — typing what HTML reserves

&lt;    <     (less-than — REQUIRED whenever you show a literal <)
&gt;    >     (greater-than, for symmetry)
&amp;   &     (ampersand — required, & starts an entity)
&quot;  "     (inside attribute values delimited by ")
&nbsp;  non-breaking space — keeps "10 km" or "Mr.&nbsp;Chen" on one line

Because your files are UTF-8 (lesson 2), you do not need entities for é, ™, →, emoji — type them directly. Entities are for HTML's reserved characters and for non-breaking behavior. &nbsp; abuse for spacing/indentation is a classic anti-pattern: spacing is CSS.

Part 5 — Step-by-Step Build: A Complete Article Page

Combine everything: build sourdough.html with the lesson-2 production head, then this body — typing it, not pasting:

<body>
  <header><a href="index.html">Maya's Coffee</a></header>
  <main>
    <h1>Sourdough at Home</h1>
    <p>You need flour, water, salt, and patience — mostly patience. This guide
       covers a three-day bake for a first loaf you'll actually be proud of.</p>

    <h2>Ingredients</h2>
    <p>500&nbsp;g bread flour, 375&nbsp;g water, 10&nbsp;g salt, 100&nbsp;g starter.</p>

    <h2>Method</h2>
    <h3>Day 1: The starter</h3>
    <p>Feed your starter and wait until it doubles — 4 to 8 hours.</p>
    <h3>Day 2: Bulk ferment</h3>
    <p>Mix, rest, then fold every 30 minutes for 3 hours.</p>
    <h3>Day 3: Bake</h3>
    <p>Preheat to 250&nbsp;°C. Bake 20 minutes covered, 25 uncovered.</p>

    <hr>
    <p>A note on failure: my first loaf could have served as a doorstop.</p>

    <h2>Words from the masters</h2>
    <figure>
      <blockquote><p>Bread is time made visible.</p></blockquote>
      <figcaption>— Chad Robertson, <cite>Tartine Bread</cite></figcaption>
    </figure>
  </main>
  <footer><p>© 2026 Maya's Coffee</p></footer>
</body>

Now verify like a professional: (1) DevTools → Elements — is the tree what you wrote? (2) Run Lighthouse (DevTools → Lighthouse → Accessibility) — heading order should pass. (3) Open the page in your browser's reader mode — a correct structure reconstructs beautifully; that's your machine audience confirming the semantics work.

Interview Questions

Q1 (beginner): Why not just pick heading levels by how big you want the text?

Because the level is structural, not visual: screen-reader users navigate by heading list, search engines infer document structure, and reader modes rebuild from it. Wrong levels lie to all three. Size is CSS's job — any heading can be restyled to any size without touching the outline.

Q2 (beginner): When is <br> appropriate, and what's the alternative when it isn't?

Only when line breaks are part of the content itself: postal addresses, poetry/lyrics. For spacing between blocks use separate paragraphs plus CSS margins; for wrapping control use CSS (max-width, text-wrap: balance). Two consecutive brs is nearly always a code smell.

Q3 (mid): What happens — exactly — when the parser meets <div> inside an open <p>, and why?

The spec says p contains only phrasing content, and its end tag is optional; a start tag for a block element implies </p>. So the parser closes the paragraph, then opens the div as its sibling; a later stray </p> is dropped (or becomes an empty p). Consequences: selectors like p > div match nothing, styles "stop applying", and the bug is invisible in View Source but obvious in the Elements panel.

Q4 (mid): A page uses styled <div class="heading-2"> everywhere instead of h2. List concretely what breaks.

(1) Screen-reader heading navigation — the #1 exploration method — finds nothing. (2) Lighthouse/ axe accessibility audits fail. (3) Search engines lose the section structure used for ranking and snippets. (4) Reader mode/RSS extraction degrades. (5) document.querySelectorAll('h2') tooling (test selectors, TOC generators) breaks. Fix is mechanical: swap to real h2 and keep the class for styling — semantics and appearance are independent.

Q5 (senior): Your CMS emits article bodies where every section heading is an h2, including subsections. An accessibility audit flagged the flat hierarchy. What's your remediation strategy?

First, quantify: crawl the content (or sample it) to see whether nesting is recoverable — often the source data has section depth the template flattened. Fix at the template layer if so. If depth truly isn't modeled, options: (a) extend the CMS content model with nesting (right long-term fix), (b) infer levels heuristically at render time (risky — false structure is worse than flat), or (c) accept flat h2s: a flat-but-honest outline passes WCAG; a fabricated hierarchy may not. Also enforce going forward: a lint rule in CI that checks rendered heading order (axe-core has this), so regressions can't ship. The senior signal: fix the pipeline, not the pages.

Q6 (senior): Why does <pre> not protect < from parsing, and what's the security-adjacent implication?

pre changes whitespace rendering, not tokenization — the parser still processes tags and entities inside it (only script/style/textarea/title have raw-text parsing modes). So user-supplied content dropped into a pre (log viewers, code-sharing tools do this constantly) is an XSS vector exactly like any other unescaped output: <pre>USER_INPUT</pre> with input <img src=x onerror=alert(1)> executes. Escape on output (&lt;, &amp;) regardless of the wrapping element — covered in depth in this track's Security lesson.

🏋️ Exercises

1 (easy): Outline extraction. Pick a Wikipedia article. Without looking at its markup, write down what you'd expect its h1/h2/h3 outline to be. Then inspect and compare (or run document.querySelectorAll('h1,h2,h3') in the Console).
What to notice

Wikipedia: exactly one h1 (the title), h2 for major sections, h3 nested under them, never skipping down. The console one-liner returns the outline in document order — a two-second audit trick worth keeping. Any surprise between your guess and reality shows you where visual hierarchy and structural hierarchy diverge.

2 (easy): Fix the broken outline. Correct this: <h2>My Blog</h2> <h4>About me</h4> <h1>Posts</h1> <h3>Why I love CSS</h3> <h3>Git tips</h3>
Solution 
<h1>My Blog</h1>
<h2>About me</h2>
<h2>Posts</h2>
  <h3>Why I love CSS</h3>
  <h3>Git tips</h3>

Site title = h1 (one per page, first). "About me" and "Posts" are peer sections = h2. Individual posts belong to "Posts" = h3. Original sins: no h1 at top, skipped levels, h1 mid-page.

3 (medium): The whitespace lab. Predict the rendering of: a paragraph containing 5 spaces between two words; the same in a <pre>; two inline-block spans on separate source lines; "10 km" vs "10&nbsp;km" in a container narrowed until it wraps. Then build and verify all four.
Expected results

(1) One space — runs collapse. (2) All 5 spaces — pre preserves. (3) A small gap between the spans: the newline collapsed to a space text node between them (the classic inline-block gap; removing the source whitespace removes it). (4) "10 km" can break across lines; "10&nbsp;km" moves as one unit. Collapsing is why indentation is free — and these are the four cases where it stops being invisible.

4 (medium): Quote three ways. Mark up: a full-paragraph quote with author and book title; a mid-sentence spoken quote; and a code snippet showing literal <div> syntax. Use the semantically correct element for each.
Solution 
<figure>
  <blockquote><p>Simplicity is a great virtue...</p></blockquote>
  <figcaption>— Edsger Dijkstra, <cite>On the nature of computing science</cite></figcaption>
</figure>

<p>She said <q>ship it Friday</q> and left the meeting.</p>

<pre><code>&lt;div class="card"&gt;...&lt;/div&gt;</code></pre>

Keys: attribution outside the blockquote, cite for the work's title, q for inline speech (it renders its own quotation marks), and &lt;/&gt; escaping inside pre — pre does not disable parsing.

5 (hard): Full article build. Do Part 5's sourdough build completely, then run all three verifications (Elements tree, Lighthouse accessibility, reader mode). Fix anything flagged. Add one more section — "FAQ" with two questions as h3s — placed correctly in the outline.
Verification notes

Lighthouse should report no heading-order issues and will also nudge you about things from other lessons (that's fine). Reader mode should show a clean, magazine-like article — if a section is missing or merged, your structure has a bug reader mode just found for you. The FAQ goes as an h2 after "Words from the masters", its two questions as h3 children.

6 (hard): Write the linter. In the Console (or a small script), write JS that walks document.querySelectorAll('h1,h2,h3,h4,h5,h6') and reports: more than one h1, and any place the level increases by more than 1. Run it on your article and on a real site. (This is a taste of the JS track; adapt freely.)
Reference implementation 
const hs = [...document.querySelectorAll('h1,h2,h3,h4,h5,h6')];
let prev = 0, h1s = 0;
hs.forEach(h => {
  const lvl = +h.tagName[1];
  if (lvl === 1 && ++h1s > 1) console.warn('Extra h1:', h.textContent.trim());
  if (prev && lvl > prev + 1)
    console.warn(`Skip h${prev}→h${lvl}:`, h.textContent.trim());
  prev = lvl;
});
console.log(`${hs.length} headings, ${h1s} h1(s) — done`);

This is essentially what axe-core's heading-order rule does. Running your own audit script once demystifies the tools you'll rely on later — they're all just walking the DOM you learned to read in lesson 1.

🎯 Key Takeaways

  • Headings are a navigable outline consumed by screen readers, search engines, and reader modes — one h1, descend one level at a time, choose by structure never by size.
  • Block vs inline (spec: flow vs phrasing content) governs what may nest where; <p> takes phrasing content only, and the parser enforces it by silently closing your paragraph.
  • CSS display changes rendering, never parsing rules.
  • <br> = content line breaks (addresses, verse) only; <hr> = topic shift, not decoration; spacing always = CSS.
  • Quotes: blockquote + figure/figcaption with attribution outside; <q> inline; <cite> for work titles.
  • Whitespace collapses everywhere except <pre> — and pre still parses tags, so escape &lt; always.
  • UTF-8 files need entities only for < > &, quotes in attributes, and &nbsp; gluing.
