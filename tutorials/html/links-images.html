Links & Images Basics

Links are what make the web a web; images are ~50% of the average page's bytes. Both look trivial and both have serious production depth: URL resolution, security attributes, responsive loading, and Core Web Vitals all live here.

Links — Every Form You'll Write

<a href="https://example.com">Absolute URL</a>
<a href="/pricing">Root-relative (same site, from the domain root)</a>
<a href="../docs/api.html">File-relative (from THIS file's location)</a>
<a href="#faq">Anchor — jumps to id="faq" on this page</a>
<a href="/report.pdf" download="q3-report.pdf">Download with suggested filename</a>
<a href="mailto:team@app.com?subject=Support">Email</a>
<a href="tel:+14155550132">Call us</a>

URL Resolution — the Part That Bites

Current page: https://site.com/docs/guide/intro.html
  href="setup.html"        → /docs/guide/setup.html     (sibling)
  href="../api/rest.html"  → /docs/api/rest.html        (up one, then down)
  href="/pricing"          → /pricing                   (root-relative: ignores current path)
  href="//cdn.x.com/a.js"  → protocol-relative (legacy; just use https:)

Rule of thumb: root-relative for app navigation (survives the page moving), file-relative only within self-contained folders (like this tutorial site, which must work from any hosting subpath).

target="_blank" — the Security Attribute Set

<a href="https://partner.com" target="_blank" rel="noopener noreferrer">Partner ↗</a>
PieceWhat it prevents
rel="noopener"New tab getting window.opener — a malicious page could redirect YOUR tab to a phishing clone (tabnabbing). Modern browsers imply it, but writing it is still the professional default.
rel="noreferrer"Leaking your full page URL in the Referer header
rel="nofollow" / "sponsored" / "ugc"SEO: tells crawlers not to pass ranking credit (paid links, user-generated content)

UX rule: don't force new tabs for same-site navigation — it breaks the back button, the user's most-used control. Reserve _blank for external references and unsaved-work contexts.

Images — the Production Checklist

<img src="/img/chart.png"
     alt="Revenue grew 40% quarter-over-quarter in 2025"
     width="800" height="450"
     loading="lazy" decoding="async">
AttributeDeep detail
altConveys the image's function, not its file description. Informative image → describe the information ("Line chart: revenue up 40%"). Decorative flourish → alt="" (empty but PRESENT — screen readers skip it; a missing alt makes them read the filename aloud). Image inside a link → describe the destination.
width/heightThe browser computes aspect ratio from these and reserves space before download → no layout shift (CLS). CSS max-width:100%; height:auto still makes it responsive.
loading="lazy"Defers offscreen images until scroll approaches. NEVER on the hero/LCP image — that delays the metric Google ranks you on. Above-the-fold hero gets fetchpriority="high" instead.
decoding="async"Decode off the main thread — tiny win, free

Responsive Images — Serve Small Screens Small Files

<!-- Same image, multiple SIZES (browser picks by viewport & DPR): -->
<img src="/img/hero-800.jpg"
     srcset="/img/hero-400.jpg 400w, /img/hero-800.jpg 800w, /img/hero-1600.jpg 1600w"
     sizes="(max-width: 600px) 100vw, 50vw"
     alt="…" width="800" height="450">

<!-- Different FORMATS (first supported wins) or different CROPS per breakpoint: -->
<picture>
  <source srcset="/img/hero.avif" type="image/avif">
  <source srcset="/img/hero.webp" type="image/webp">
  <source media="(max-width: 600px)" srcset="/img/hero-portrait.jpg">
  <img src="/img/hero.jpg" alt="…" width="1200" height="600">
</picture>

srcset+sizes = resolution switching (same picture, right size). <picture> = format switching and art direction (different crop on mobile). A phone downloading a 400px file instead of 1600px saves ~90% of the bytes — this is routinely the biggest single performance win on image-heavy sites.

Interview Questions

Q1: srcset/sizes vs <picture> — when each?

srcset+sizes: same image content, let the browser pick the optimal resolution (it factors viewport, layout width via sizes, device pixel ratio, even data-saver mode). picture: you dictate — different formats (AVIF→WebP→JPEG) or different crops per breakpoint (art direction). They compose: source elements can carry their own srcset.

Q2: Empty alt vs missing alt vs alt with text — differences?

alt="" = "decorative, skip me" (correct for flourishes). Missing alt = broken: screen readers announce the filename. Text alt = the image's information or function. Bonus: for a logo linking home, alt="Acme home" (function), not alt="Acme logo" (description).

Q3: Why do width/height attributes still matter with responsive CSS?

They give the browser the aspect ratio before the image downloads, so it reserves the right space — eliminating Cumulative Layout Shift. CSS overrides the final displayed size; the attributes exist for the pre-download layout calculation.

🏋️ Practice:
1. Build a mini photo card: figure + image with width/height + figcaption + a properly-secured external link below it.
2. Break it: remove width/height, put a paragraph of text under the image, and use DevTools network throttling (Slow 3G) to watch the text jump when the image loads. Restore them and compare.
3. Write srcset/sizes for a product image that renders full-width on phones and one-third width on desktop.

🎯 Key Takeaways

  • Root-relative URLs for app links; meaningful anchor text always; noopener with _blank.
  • Every img: alt (possibly empty), width+height, lazy unless above the fold.
  • Modern stack: AVIF/WebP via picture, srcset for resolutions, SVG for anything vector.
  • Images are the #1 CLS and LCP lever — these attributes ARE performance engineering.
