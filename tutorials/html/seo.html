SEO & Meta Tags Advanced

Organic search is most sites' largest traffic source, and HTML is the interface between your content and the crawler. This lesson covers on-page SEO end to end: the tags, the structured data behind rich results, and how crawling/indexing actually works.

How Search Engines Process Your Page

1. CRAWL    Googlebot fetches the URL (found via links/sitemap)
2. RENDER   runs your JS in headless Chrome — but possibly DAYS later (render queue)
3. INDEX    extracts content, headings, links, structured data → stores
4. RANK     per query: relevance + authority (links) + experience signals (Core Web Vitals, mobile)

The render queue is why client-side-rendered content is risky for SEO — anything critical should be in the initial HTML (SSR/SSG — the Production Patterns lesson closes this loop).

The Core Tags, With Their Real Rules

<title>Standing Desk DX-1 — Motorized, 10-yr Warranty | DeskCo</title>
<meta name="description" content="Dual-motor height-adjustable desk with memory
  presets. Free shipping, assembled in 10 minutes.">
<link rel="canonical" href="https://deskco.com/desks/dx-1">
<meta name="robots" content="index, follow">
TagRules that matter
<title>~50–60 chars before truncation; unique per page; primary keyword early; the single strongest on-page element. Google rewrites weak titles — write strong ones.
description~150–155 chars; a pitch, not a keyword dump — it changes CTR, not rank. Google substitutes page text if yours is bad/missing.
canonicalDeclares the "real" URL among duplicates (?utm=…, print views, http/https, m-dot). Without it, ranking signal splits across variants. Self-referencing canonicals on every page are standard practice.
robotsnoindex = stay out of results (staging, thin pages, internal search results). Crucial distinction: robots.txt blocks crawling, not indexing — a blocked-but-linked page can still appear in results; only noindex (which requires crawling to be seen!) removes it.

Structure Signals Crawlers Read

✔ One h1 with the page's topic; meaningful h2/h3 sections (heading lesson pays off here)
✔ REAL <a href> links — crawlers don't reliably follow onclick-div "links"
✔ Descriptive anchor text ("standing desk assembly guide", not "read more")
✔ alt text — the entirety of image SEO, plus Google Images traffic
✔ Clean URLs: /desks/dx-1 beats /p?id=8412&cat=7
✔ Internal linking: your most important pages should be few clicks from home

Social Cards — Open Graph & Twitter

<meta property="og:title"       content="Standing Desk DX-1">
<meta property="og:description" content="Dual motors, memory presets, 10-yr warranty.">
<meta property="og:image"       content="https://deskco.com/img/dx1-card.jpg"> <!-- 1200×630 -->
<meta property="og:url"         content="https://deskco.com/desks/dx-1">
<meta property="og:type"        content="product">
<meta name="twitter:card"       content="summary_large_image">

These control the preview card in Slack, WhatsApp, LinkedIn, iMessage, X. Absolute URLs required; og:image must be publicly fetchable (no auth). Preview debuggers exist for every platform — always test before a launch post.

Structured Data (JSON-LD) — Rich Results

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Product",
  "name": "Standing Desk DX-1",
  "image": "https://deskco.com/img/dx1.jpg",
  "offers": { "@type": "Offer", "price": "499.00", "priceCurrency": "USD",
              "availability": "https://schema.org/InStock" },
  "aggregateRating": { "@type": "AggregateRating", "ratingValue": "4.7", "reviewCount": "1284" }
}
</script>

This is what produces ⭐4.7, price, and stock status directly in the results page. High-value types: Product, Article, FAQPage, HowTo, JobPosting, Recipe, Event, BreadcrumbList. Rules: the data must match visible page content (mismatches draw penalties); validate with Google's Rich Results Test. JSON-LD in a script tag is Google's recommended format — no microdata attributes scattered through your markup.

Crawl Infrastructure — the Two Files

# robots.txt (at domain root) — crawler etiquette, NOT security:
User-agent: *
Disallow: /admin/
Sitemap: https://deskco.com/sitemap.xml

<!-- sitemap.xml — every canonical URL you want indexed, with lastmod: -->
<url><loc>https://deskco.com/desks/dx-1</loc><lastmod>2026-07-01</lastmod></url>

Also know hreflang links for multi-language sites (en-us/hi-in variants of a page point at each other) — without them, the wrong language ranks in the wrong country.

Interview Questions

Q1: robots.txt Disallow vs meta noindex?

Disallow stops crawling (save crawl budget, keep bots out of infinite calendars) but the URL can still be indexed from external links — title-only. noindex removes from the index but must be crawlable to be seen. The classic blunder: blocking a page in robots.txt AND adding noindex — the crawler can never see the noindex.

Q2: Why do SPAs have SEO problems and what are the fixes?

Content rendered client-side reaches Google late (render queue) or never (other crawlers, social scrapers don't run JS). Fixes: SSR or SSG for public pages, prerendering for small sites, and meta tags must be in the server response (social crawlers never execute JS).

Q3: Same product at 4 URLs (color variants + tracking params) — what do you ship?

One canonical URL declared via rel=canonical on all variants; parameter-stripped internal links; the canonical in the sitemap. If variants deserve their own indexing (different colors genuinely searched), separate canonical pages with distinct content instead.

🏋️ Practice:
1. Write the complete head (title, description, canonical, OG set, twitter:card) for an article "10 Docker Mistakes That Break Production".
2. Add FAQPage JSON-LD with two questions and validate mentally against the visible-content rule.
3. Run a Lighthouse SEO audit on any page you've built; explain each flag it raises.

🎯 Key Takeaways

  • Unique title + description per page; canonical everywhere; noindex ≠ robots.txt.
  • Semantic structure (h1, real links, alt, clean URLs) is half of on-page SEO already done.
  • OG/Twitter tags own your link previews; JSON-LD owns rich results — both must match visible content.
  • Critical content belongs in server-rendered HTML; JS-only content gambles on the render queue.
