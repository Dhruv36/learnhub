Semantic Layout Elements Semantics

What you'll master: the landmark elements (header, nav, main, article, section, aside, footer) as a precise vocabulary rather than vague "good practice" — what each claims, how screen readers turn them into a navigation system, the article-vs-section decision that confuses everyone, and details/summary + dialog, the interactive semantics that replace JS widgets. Prerequisites: Foundations section; lesson 3's outline concepts.

Part 1 — Landmarks Are a Navigation System

Recall lesson 3: screen-reader users navigate by headings. The second system they use is landmarks — the major regions of the page. AT builds a landmark list ("banner, navigation, main, complementary, contentinfo") and offers single-key jumps between them. A page of divs has no landmarks: the user must arrow through everything, every visit. The landmark elements exist to end that:

<body>
  <header>          → role "banner"        site-wide masthead: logo, search, main nav
    <nav>           → role "navigation"    a major block of links
  </header>
  <main>            → role "main"          THE content — exactly one per page
    ...
  </main>
  <aside>           → role "complementary" tangential: related links, ads, pull quotes
  <footer>          → role "contentinfo"   copyright, legal, site links
</body>

Three precision points that separate real understanding from cargo-cult usage:

  • main is singular and content-only: one per page, containing the content unique to this page — never the header/nav/footer chrome. "Skip to main content" links target it; reader modes extract it; AT users jump straight to it. If you're unsure where chrome ends and content begins, main forces the decision — that's its gift.
  • header and footer are contextual: at body level they're banner/contentinfo, but they're also legal inside article/section (an article's own header with title+byline, its own footer with tags) — where they lose the landmark role. Nesting them is a feature, not a hack.
  • Multiple navs need names: main menu, breadcrumbs, footer links, pagination — all legitimately nav. Distinguish them with aria-label: <nav aria-label="Breadcrumb">, or AT announces three anonymous "navigation"s. Not every link group is a nav — a paragraph with links is just a paragraph; nav claims major navigation blocks.

Part 2 — article vs section vs div (the eternal confusion, resolved)

The decision tree, in the order to ask:

1. Would this make sense on its own — in an RSS feed, search result, or share?
   YES → <article>     (blog post, product card, comment, tweet, news item, widget)

2. Is it a thematic grouping that deserves a heading — a chapter of this page?
   YES → <section>     (features section, "Reviews", steps of a guide, tab panels)

3. Is it only there for styling/scripting — no meaning claimed?
   YES → <div>         (wrappers, grids, flex containers — MOST of your elements, honestly)
  • article's test is syndication: could it stand alone elsewhere and still make sense? Comments pass (nested articles inside the post's article — legal and correct!). Product cards pass. A page's hero banner fails — it's chrome, not content.
  • section's contract is a heading: a section without a heading (or aria-label) is almost always wrong — the spec says sections should be "listed in the document's outline". No heading to list = you wanted a div. This single rule kills most section misuse.
  • div is not failure. The styling-wrapper div is honest markup — it claims nothing, and most layout containers should claim nothing. Semantic-element-itis (section soup: section for every visual box) is as real a disease as div soup, and harder to cure because it feels virtuous. The accessibility tree gets noisier with every false claim.

Worked example — a blog index page, fully decided:

<main>
  <h1>Engineering Blog</h1>
  <section aria-label="Featured post">        <!-- thematic group, labeled -->
    <article>                                  <!-- syndicatable unit -->
      <header>
        <h2><a href="/posts/streaming-html">Streaming HTML in 2026</a></h2>
        <p>By Maya Chen · <time datetime="2026-07-10">July 10</time></p>
      </header>
      <p>Why the oldest performance trick still beats your hydration strategy…</p>
      <footer><a href="/tags/performance">#performance</a></footer>
    </article>
  </section>
  <section>
    <h2>All posts</h2>
    <div class="post-grid">                    <!-- pure layout: div, proudly -->
      <article>…</article>
      <article>…</article>
    </div>
  </section>
</main>

Part 3 — details/summary: Disclosure Without JavaScript

<details>
  <summary>What's your refund policy?</summary>
  <p>Full refund within 30 days, no questions — email us your order number.</p>
</details>

<details open>…</details>              <!-- expanded initially -->
<details name="faq">…</details>         <!-- shared name = accordion: opening one closes others -->

Native expand/collapse: keyboard support, AT announcements ("button, collapsed"), find-in-page can search closed details (Chrome auto-expands matches), zero JS. The name attribute (2024+, all modern browsers) gives exclusive accordions natively — the last excuse for JS accordions is gone for content use-cases. The toggle event hooks JS in when you need side effects (lazy-loading the content, analytics). Interview-adjacent details: summary is the always-visible first child (browser provides a default "Details" label if you omit it — don't); everything else shows/hides; and this very page's exercise solutions are details elements — you've been using the pattern all track.

When NOT: details is content disclosure, not app UI — for menus, tooltips, and overlays you'll outgrow its styling/positioning fast (and its semantics are "disclosure", not "menu"). For those, meet:

Part 4 — dialog: Modals Done by the Platform

<dialog id="confirm">
  <h2>Delete this post?</h2>
  <p>This can't be undone.</p>
  <form method="dialog">                     <!-- method="dialog": submitting CLOSES the dialog -->
    <button value="cancel">Cancel</button>
    <button value="delete">Delete</button>   <!-- dialog.returnValue becomes "delete" -->
  </form>
</dialog>

<script>
  confirm.showModal();                        // MODAL: backdrop, focus trapped, Esc closes
  confirm.show();                             // non-modal variant
  confirm.addEventListener("close", () => {
    if (confirm.returnValue === "delete") reallyDelete();
  });
</script>

What showModal() provides that a div-modal must hand-build (and usually gets wrong): focus trapped inside, focus restored to the opener on close, Esc handling, a ::backdrop pseudo-element, the rest of the page made inert (unfocusable, invisible to AT), and top-layer rendering that no z-index war can bury. The years of modal-library bugs this element retires is the strongest recent argument for "check the platform first" — and its cousin, the popover attribute (popover + popovertarget), does the same for menus/tooltips/toasts with light-dismiss (click-outside closes). Know both exist; reach for libraries only when they demonstrably fall short.

Part 5 — Reading a Real Page's Structure (the audit skill)

The skill this lesson builds: given any page, you can sketch its landmark tree. Do it top-down:

body
├── header[banner] ── nav[aria-label="Main"]
├── nav[aria-label="Breadcrumb"]
├── main
│   ├── article        (the product / the post / the doc)
│   │   ├── header     (title, byline)
│   │   ├── section×N  (chapters with h2s)
│   │   └── footer     (tags, related)
│   └── aside          (related products / TOC)
├── aside[complementary]  (site-level: newsletter signup)
└── footer[contentinfo]

Verify with tooling: DevTools → Accessibility panel shows the computed role per node; the axe extension lists landmarks; VoiceOver's rotor / NVDA's D key jump through them. Common findings on real sites: multiple mains (invalid), anonymous nav armies, banner content outside header, and everything-is-a-section soup. You'll audit one in the exercises.

Interview Questions

Q1 (beginner): Why use header/main/footer when divs with classes render identically?

Rendering was never the point — the elements map to landmark roles (banner/main/contentinfo) that AT exposes as a jump-navigation system; reader modes and crawlers use them to find the content; and the vocabulary is self-documenting for teammates. Divs give pixels only; landmarks give the page an API.

Q2 (beginner): article vs section in one breath each, with an example that's commonly gotten wrong.

article: self-contained, makes sense syndicated alone — posts, comments, product cards. section: a thematic chapter OF something, deserving a heading in the outline. Commonly wrong: comments — they're articles (standalone units, legally nested inside the post's article), not sections; and "hero sections" — usually neither, just a div, because they claim no outline entry.

Q3 (mid): What does dialog.showModal() give you that a positioned div cannot get from CSS alone?

Top-layer rendering (above every z-index, unclippable by overflow ancestors), the ::backdrop pseudo-element, focus trapped within and restored to the opener on close, Esc-to-close, and — the big one — the rest of the document made inert: unfocusable and hidden from AT. A div modal must reimplement focus trapping, inert-ing (the inert attribute helps now), Esc, and restore — the exact features modal libraries historically got wrong. method="dialog" forms closing with a returnValue is the bonus detail.

Q4 (mid): Your page has four navs. Walk through making them distinguishable and how AT presents the result.

aria-label each by function: "Main", "Breadcrumb", "Footer", "Pagination" (don't include the word "navigation" — the role already announces it: "Main navigation" reads as "Main, navigation"). AT rotors/landmark lists then show four named entries users can jump between. Also audit whether all four ARE major navigation — inline related-links might not merit nav at all. Labels are for humans navigating blind: name by purpose, not position ("left sidebar nav" is meaningless reflowed on mobile).

Q5 (senior): A React codebase renders every component wrapped in section with a generated aria-label. Lighthouse passes. Critique.

Passing Lighthouse ≠ good semantics — automated tools check presence, not sense. Every component a labeled section means: the landmark/region list explodes into dozens of machine-named regions (worse than none — AT users wade through noise), the outline fills with entries no human organized, and genuine landmarks drown. Fix: components render divs by default; semantics are a page-level design decision — the page composes a handful of intentional landmarks and sections with human headings. Deeper point for the review: semantics can't be automated bottom-up from components; they're information architecture, owned top-down like the visual design is. (And: measure with a screen reader session, not just axe.)

Q6 (senior): When would you still write a JS modal/accordion in 2026 despite dialog and details? Draw the actual line.

Legitimate residual cases: (1) product requirements exceeding the primitive — nested/stacked modal flows with custom transition choreography (dialog stacks, but animation control is limited), virtualized accordion contents, disclosure state synced to router/URL; (2) design systems needing pixel-identical behavior across browsers where native rendering differs (details markers, dialog positioning quirks); (3) legacy support floors (rare now). The line: you may wrap/extend the native element (dialog inside your Modal component — best of both), but reimplementing focus trapping/inert/top-layer from scratch requires justifying each against the platform version in review. "Designer wants a different arrow icon" clears none of this — that's CSS on details. The senior discipline: platform first, deviation documented, never silently.

🏋️ Exercises

1 (easy): Landmark the coffee site. Take your running Maya's Coffee page and give it the full skeleton: header with labeled nav, main, an aside (newsletter pitch), footer. Verify roles in the DevTools Accessibility panel.
Check

The Accessibility panel should show banner → navigation → main → complementary → contentinfo. If your aside sits inside main: fine when tangential-to-this-page's-content; site-level asides sit outside. Both are defensible — what matters is you can now argue the choice.

2 (easy): The decision drill. article, section, or div? (a) one tweet in a timeline, (b) the "Similar products" strip, (c) a flexbox wrapper centering a card, (d) chapter 3 of a long doc page, (e) a weather widget on a dashboard, (f) the cookie-consent bar.
Answers

(a) article — syndication test passes. (b) section with its heading ("Similar products"). (c) div — pure layout. (d) section with h2. (e) article — the spec's own example: a self-contained widget. (f) div (arguably a labeled region/dialog depending on behavior — but never section/article). Score 5+ and the tree is yours.

3 (medium): FAQ, upgraded twice. Build a 5-question FAQ with details/summary. Upgrade 1: shared name so only one opens at a time; upgrade 2: make the third question deep-linkable — arriving at #q3 opens it (small JS on the hashchange/load, or research the :target trick + open attribute). Check keyboard and find-in-page behavior.
Solution keys 
<details name="faq" id="q3">
  <summary>Do you ship internationally?</summary>…</details>

// deep-link opener:
function openFromHash() {
  const el = document.querySelector(location.hash + "[name=faq]");
  if (el) el.open = true;
}
addEventListener("hashchange", openFromHash); openFromHash();

Keyboard: Tab reaches each summary, Enter/Space toggles — free. Find-in-page: Chrome searches closed details and auto-opens the match (hidden=until-found under the hood) — try it. Total JS for a deep-linkable exclusive accordion: six lines.

4 (medium): The confirm dialog. Implement Part 4's delete confirmation completely: showModal on a Delete button, method="dialog" form, returnValue handling, and verify with keyboard only — Tab cycles INSIDE the dialog, Esc cancels, focus returns to the Delete button after close. Then style ::backdrop.
Verification notes

All four keyboard behaviors come free — that's the demo's point; your only code was showModal, the form, and the close listener. ::backdrop styling: dialog::backdrop { background: rgb(0 0 0 / .5); backdrop-filter: blur(2px) }. Now delete the dialog element and imagine implementing those four behaviors on a div — that imagined afternoon is why this element exists.

5 (hard): Landmark audit of a major site. Pick a top-20 site. Using the Accessibility panel (or axe), map its actual landmark tree. Score it: one main? navs labeled? banner/contentinfo correct? section soup? Write five findings with severity, as if filing an audit report.
Typical findings

Real-world commonness ranking: anonymous multiple navs (near-universal), landmark-less mid-page regions (divs where aside/section belong), section soup from component frameworks, double banners (a page header inside body header), occasionally zero landmarks on legacy properties. Rate anonymous navs "moderate" (usable but noisy), missing main "serious" (skip-navigation broken). Writing severity forces the judgment this lesson teaches: which semantics actually gate tasks vs merely offend taste.

6 (hard): Semantic refactor. Here's a div-soup page shape: div.top > div.logo + div.links; div.crumbs; div.wrap > div.story > div.story-head + div.story-body + div.story-tags; div.also-read; div.bottom. Rewrite it with full semantics (landmarks, article structure, labeled navs, time elements where implied), then list what each change unlocked.
Solution 
<header>                                <!-- banner -->
  <a class="logo" href="/">…</a>
  <nav aria-label="Main">…</nav>
</header>
<nav aria-label="Breadcrumb"><ol>…</ol></nav>   <!-- ordered! path order matters -->
<main>
  <article>
    <header><h1>…</h1><p>By … <time datetime="…">…</time></p></header>
    …body…
    <footer>tag links</footer>
  </article>
  <aside aria-label="Also read">…</aside>
</main>
<footer>…</footer>                       <!-- contentinfo -->

Unlocked: landmark jump-nav (5 stops), reader-mode extraction of exactly the article, breadcrumb as ordered list announced with position, article syndicatable, date machine-readable, and — the quiet one — the next developer reads the page's information architecture from the tags alone. Same pixels, an API where there was soup.

🎯 Key Takeaways

  • Landmarks are AT's second navigation system: header=banner, nav, main (exactly one, content-only), aside=complementary, footer=contentinfo — plus aria-label when a role repeats.
  • article = passes the syndication test (comments and widgets included); section = a chapter that owes the outline a heading; div = honest "no claim" — and most wrappers should be divs.
  • header/footer nest contextually inside articles/sections, losing landmark status — structure all the way down.
  • details/summary (+ name accordions, + toggle event) replaces JS disclosure; dialog.showModal() replaces modal libraries (top layer, inert page, focus trap/restore, Esc, ::backdrop); popover covers menus/toasts.
  • Semantics are page-level information architecture — they can't be automated bottom-up from components, and passing Lighthouse doesn't mean they make sense.
  • Audit skill: sketch any page's landmark tree from the Accessibility panel — findings, severity, fix.
