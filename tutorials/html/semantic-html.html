Semantic HTML Intermediate

"Div soup" renders identically to semantic markup — to sighted users. To screen readers, search crawlers, reader modes, and future maintainers, semantic elements are the difference between a document and a pile of boxes.

Same Page, Two Ways

<!-- ❌ Div soup -->                      <!-- ✅ Semantic -->
<div class="header">…</div>              <header>…</header>
<div class="nav">…</div>                 <nav aria-label="Main">…</nav>
<div class="content">                    <main>
  <div class="post">…</div>                <article>…</article>
  <div class="side">…</div>                <aside>…</aside>
</div>                                   </main>
<div class="footer">…</div>              <footer>…</footer>

Choosing the Right Container

ElementUse when…
<article>Content that stands alone if extracted: blog post, product card, comment
<section>A thematic grouping with a heading — chapters of a page
<aside>Tangential content: sidebars, pull quotes, related links
<nav>Major navigation blocks (not every list of links)
<div>Pure styling/layout hook with no meaning — a valid choice once the above don't fit

These create the landmark map assistive tech navigates by: a screen-reader user hits one key to cycle header → nav → main → aside → footer, skipping everything in between.

Interactive Semantics Without JavaScript

<details>
  <summary>What's your refund policy?</summary>
  <p>Full refund within 30 days, no questions asked.</p>
</details>

<dialog id="confirm">…</dialog>   <!-- opened with .showModal(): focus trap for free -->

Accordion and modal behavior, keyboard support included, zero JS libraries. (This site's own solution toggles in the LeetCode track are <details>.)

💡 The button-vs-link law (most common frontend review comment in existence): navigating somewhere = <a href>; doing something = <button>. A clickable <div onclick> is invisible to keyboards and screen readers — recruiters' accessibility audits catch this in minutes.
