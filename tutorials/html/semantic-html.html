Semantic HTML Intermediate

Semantic HTML means the tag names what the content is. A sighted user can't tell <div class="nav"> from <nav> — but a screen reader, a search crawler, a reader mode, a browser extension, and the next developer all can. Semantics is writing for every consumer of your page, not just eyes.

Who Actually Consumes Semantics

ConsumerWhat it does with your tags
Screen readersLandmark navigation (jump to main/nav), element counts, role announcements
Search enginesWeigh headings/articles, extract snippets, understand page zones
Reader mode / PocketExtract <article>, drop nav/aside clutter
BrowsersFree keyboard behavior, form semantics, focus management
Your team<aside> tells the next dev what this block IS; <div class="rightBox2"> doesn't

The Landmark Elements

<body>
  <header>         site banner: logo, search, primary nav
    <nav aria-label="Main">…</nav>
  </header>
  <main>           THE content — exactly one per page
    <article>      self-contained: could be syndicated alone
      <header>…article title, byline…</header>
      <section>    thematic chapter WITH a heading
      <aside>      tangent: related links, pull quote
      <footer>…article tags, share buttons…</footer>
    </article>
  </main>
  <footer>         site footer: sitemap, legal, contact
</body>

The Decision Guide (with the hard cases)

ElementChoose when…Hard case resolved
<article>Standalone & complete: blog post, product card, comment, tweetCards in a grid? Each card = article if it makes sense alone (news teaser: yes; table row: no)
<section>A titled chapter of a larger wholeRule: no natural heading → probably not a section, use div
<nav>Major navigation blocksFooter link farm? One nav around it max — not every link list is nav
<aside>Removable without losing the plotSidebar with the page's filters? That's functionality, not aside — use section/form
<div>Pure styling/layout hookPerfectly legitimate — AFTER the above don't fit. Wrappers for flex/grid are honest div work

Built-in Interactive Semantics (skip the JS library)

<details>
  <summary>What's your refund policy?</summary>
  <p>Full refund within 30 days.</p>
</details>
<!-- Accordion: keyboard + a11y included. name="faq" on several = exclusive group. -->

<dialog id="confirm">
  <p>Delete this project?</p>
  <form method="dialog"><button value="yes">Delete</button><button value="no">Cancel</button></form>
</dialog>
<script>confirmBtn.onclick = () => document.querySelector("#confirm").showModal();</script>
<!-- showModal(): focus trap, Esc-to-close, ::backdrop, top-layer rendering — all free.
     A DIY div-modal needs 100+ lines of JS to match it. -->

The Cost of Div Soup — a Real Example

❌ <div class="btn" onclick="save()">
  Save
</div>
· invisible to Tab key
· no Enter/Space activation
· announced as plain text
· :disabled, forms: nothing
✅ <button type="button"
        onclick="save()">
  Save
</button>
· focusable, keyboard-activated
· announced "Save, button"
· disabled, form integration free
· restyle with CSS in 3 lines

The button-vs-link law (the most repeated code-review comment in frontend): navigating somewhere → <a href>; performing an action → <button>. Links can be opened in new tabs, copied, prefetched; buttons can be disabled and submit forms. A link styled as a button is fine; a div acting as either is a defect.

Interview Questions

Q1: Sell semantic HTML to a skeptic in 30 seconds.

Same rendering, four free wins: accessibility (landmarks/roles announced), SEO (structure extracted), robustness (native keyboard/focus behavior you'd otherwise rebuild), maintainability (tags document intent). Cost: zero — it's the same number of tags with better names.

Q2: section vs article vs div — your decision process?

Could it stand alone in an RSS feed? article. Is it a titled chapter of this page? section. Is it only there so CSS can position things? div. And section without a heading is usually a div in denial.

Q3: What do <dialog> and <details> replace, and what's left for JS?

They replace modal/accordion libraries: focus trapping, Esc handling, backdrop, open/close state all native. JS still owns: triggering showModal(), reacting to the close event's returnValue, and any animation beyond CSS.

🏋️ Practice:
1. Take this div-soup and rewrite it semantically: div.header > div.menu / div.body > div.post + div.right / div.bottom.
2. Build an FAQ with three exclusive <details name="faq"> items — verify only one opens at a time.
3. Build a delete-confirmation <dialog> with method="dialog" buttons and log the returnValue.

🎯 Key Takeaways

  • Semantics = machine-readable meaning; renderers, readers, robots, and colleagues all consume it.
  • Landmarks (header/nav/main/aside/footer) are the page's navigation API for assistive tech.
  • article = standalone; section = titled chapter; div = honest styling hook.
  • details/dialog give you accordions and modals with built-in a11y — use them before libraries.
  • Action → button. Navigation → a. Never a clickable div.
