Inline Text Semantics Foundations

What you'll master: the full toolkit of inline elements that add meaning within a sentence — emphasis, importance, code, keyboard input, dates, abbreviations, edits, and more — including the em-vs-i and strong-vs-b distinctions interviewers love, and when plain <span> is the honest choice. Prerequisite: Text: Paragraphs & Headings (block vs inline).

Part 1 — The Famous Four: em, i, strong, b

HTML has two italic-looking and two bold-looking elements, and the difference is the single most common inline-semantics interview question. The key: two are about meaning, two are about conventional typography.

ElementMeaningUse when…Example
<em>Stress emphasis — changes the sentence's meaningYou'd stress the word aloudI said we ship Friday (not Monday)
<i>Alternate voice/mood — conventionally italicForeign phrases, ship names, species, thoughts, technical terms on first useThe E. coli sample; the Nautilus sailed
<strong>Strong importance, seriousness, urgencyWarnings, key facts the reader must not missDo not unplug during the update.
<b>Drawing attention without extra importanceKeywords in a summary, product names in a review, lede sentencesChoose the Pro plan for teams

The self-test that resolves every case: read the sentence aloud. If your voice stresses the word — <em>. If the content is a warning or crucial fact — <strong>. If it's just "typographically set apart by convention" — <i>/<b>. If you only want it to look different with no semantic claim at all — a <span> with a class and CSS.

<!-- em placement changes MEANING — the classic demonstration: -->
<p><em>I</em> didn't take the money.</p>   (someone else did)
<p>I didn't take <em>the money</em>.</p>   (I took something else)

<!-- Nesting em increases stress; strong inside strong increases importance: -->
<p><strong>Warning: <strong>never</strong> mix these chemicals.</strong></p>
🔍 Honest senior context: most screen readers do not change voice for em/strong by default, so don't claim that in interviews — the real value is machine readability (translation quality, reader modes, AI extraction), maintainable styling hooks, and correctness-by-default typography (i self-italicizes appropriately in contexts like citations). Choosing correctly costs nothing; choosing wrongly bakes lies into the content layer.

Part 2 — The Developer's Toolkit: code, kbd, samp, var

Four monospace-rendering siblings with distinct meanings — technical docs (like this site) use all four:

<code>npm install</code>      — fragment of code / identifiers: the <code>map()</code> method
<kbd>Ctrl</kbd>+<kbd>C</kbd>  — user INPUT: keys pressed, commands typed
<samp>404 Not Found</samp>   — program OUTPUT: what the system printed back
<var>n</var>                 — a variable/placeholder: runs in O(<var>n</var>²) time

Block-level code uses the idiom from last lesson: <pre><code>. The convention for a class name carrying the language — <code class="language-js"> — is what syntax highlighters (Prism, highlight.js) key off.

Part 3 — Machine-Readable Data: time, abbr, data

<time> — dates a machine can parse

<time datetime="2026-07-12">July 12th</time>
<time datetime="2026-07-12T09:30-07:00">9:30 AM Pacific</time>
<time datetime="PT2H30M">2½ hours</time>   (a duration)

Humans read the friendly text; machines read the ISO-8601 datetime attribute. Payoffs: search engines show accurate dates in results, browsers/extensions can offer calendar actions, and your own JavaScript can reliably re-render dates ("3 days ago") from the attribute without parsing prose. Rule: any date/time your content states, wrap in <time>.

<abbr> — abbreviations that explain themselves

<abbr title="Cascading Style Sheets">CSS</abbr>

Hover shows the expansion; screen readers can announce it. Best practice for prose: spell out on first use — "Cascading Style Sheets (CSS)" — then use the abbr alone. Don't wrap every occurrence of common terms (HTML, USA) — the first-use pattern is enough; title-only affordances are invisible on touch devices, which is why first-use expansion in text matters more than the tooltip.

<data> — attach a machine value to any text

<data value="8471">Ergonomic Chair</data>   — e.g. product ID behind a display name

Rarely needed (usually you'd use data-* attributes — covered in the Global Attributes lesson), but it's the semantic option when the text itself has a machine equivalent.

Part 4 — Edits, Corrections, and Fine Print: ins, del, s, mark, small, sub, sup

<p>Price: <del>$79</del> <ins>$49</ins> — launch sale!</p>
<p><s>Sold out.</s> Back in stock!</p>
<p>Search results for <mark>hydration</mark>: ...</p>
<p>H<sub>2</sub>O boils at 100&nbsp;°C. E = mc<sup>2</sup>.</p>
<p><small>© 2026 Maya's Coffee. Prices include VAT.</small></p>
  • del/ins — document edits (removed/added content). Both accept cite (URL explaining the change) and datetime attributes — useful in changelogs and legal documents. Price-drop markup with del/ins is the canonical e-commerce use.
  • s — no longer accurate/relevant (not an edit — a fact that expired).
  • mark — highlighted relevance in the current context: search-term highlighting, the key line in a quoted passage. Not a general yellow highlighter for "important" — that's strong.
  • small — side comments and fine print (copyright, disclaimers, licensing). Semantic, not just "smaller font".
  • sub/sup — real subscripts/superscripts: chemical formulas, exponents, footnote markers1. Not for styling arbitrary text small-and-raised.

Part 5 — span, and the Discipline of Choosing

<span> is the inline element with no meaning at all — the inline twin of <div>. It exists purely as a hook for CSS/JS. Using it is not failure; using it when a semantic element exists is. The professional decision procedure for any inline markup need:

1. Does a specific element exist for this meaning?   (em/strong/code/time/abbr/...)  → use it
2. Is this pure styling with no meaning claim?        → span + class
3. Never: pick an element because of its default look ("b is how I bold things")

Worked example — a product-card line, marked up honestly:

<p class="deal">
  <b>Ergonomic Chair</b> —                      <!-- product name: attention, no urgency → b -->
  <del>$79</del> <ins>$49</ins>                 <!-- price edit → del/ins -->
  <strong>ends tonight</strong>                  <!-- urgency → strong -->
  <small>(excl. shipping)</small>                <!-- fine print → small -->
  <span class="badge">BESTSELLER</span>          <!-- pure UI decoration → span -->
</p>

Five different elements in one line, each earning its place. This is what "semantic HTML" means in daily practice — not memorizing tags, but making honest claims.

Part 6 — Deep Dive: Styling Is Not Guaranteed

Browsers style these elements by convention, not contract: em is italic and strong bold only because the default stylesheet says so. Consequences worth knowing:

  • CSS resets (common in real projects) may strip these defaults — your semantics remain, your italics vanish. Correct: restore desired styles in your stylesheet; incorrect: conclude "em is broken" and switch to i.
  • You can and sometimes should restyle: many design systems render strong with a highlight color instead of bold weight. Legal, because meaning and rendering are independent layers.
  • Nested same-element styling needs care: em em renders italic-inside-italic (visually identical); typographic convention is to flip back to roman — em em { font-style: normal }. A tiny detail that marks a craftsperson.

Interview Questions

Q1 (beginner): em vs i — explain the difference with examples.

em = stress emphasis that changes sentence meaning ("I didn't say that"). i = text conventionally italicized without emphasis: foreign phrases, ship names, species names, technical terms, inner thoughts. Test: read aloud — if you stress it, em; if it's just typographic convention, i. Same rendering by default, different claims to machines.

Q2 (beginner): When would you reach for span?

When you need an inline hook for CSS or JS and no semantic claim applies — a decorative badge, a character-counting wrapper, a styling target. It's the honest "no meaning" element. The mistake is using it (or b) where em/strong/code/time/abbr would make a true claim for free.

Q3 (mid): Mark up "Was $99, now $59, offer ends Sunday June 14th" for an e-commerce card, with reasoning.

<del>$99</del> <ins>$59</ins> <strong>offer ends <time datetime="2026-06-14">Sunday June 14th</time></strong>. Reasoning: the price change is an edit (del/ins — also gives screen readers and scrapers the old/new relationship), urgency is genuine importance (strong), the deadline is machine-readable (time). Bonus: aria/CSS can make del announce as "was" — some teams add visually-hidden "was/now" text for screen-reader clarity, since not all announce del/ins by default.

Q4 (mid): Why does <time datetime> exist when you could put the ISO date in a data-attribute?

Standardization: because time is in the spec, consumers you don't control understand it — search engines (article dates in results), reader modes, browser features, assistive tech. data-* attributes are private to your own scripts; nothing external reads them. Rule of thumb: spec elements/attributes broadcast to the world, data-* whispers to your own code. (Same argument as structured data — covered later in the SEO lessons.)

Q5 (senior): A CSS reset stripped italic/bold from em/strong and a junior concludes semantic elements "do nothing" and switches to spans with classes. Correct the reasoning.

The default styling was never the point — it's a convenience from the UA stylesheet, and a reset removing it proves the layers are independent, not that semantics are useless. What em/strong still deliver unstyled: machine-readable meaning for translation engines, reader modes, and content extraction; searchable/lintable markup; styling hooks that survive redesigns; and honest content when the CSS fails to load. The fix is one line restoring the desired rendering in the project stylesheet. Deeper point for the review: resets are supposed to force deliberate styling decisions — the junior found a missing decision, not a broken element.

Q6 (senior): You're building a search-results page. Where do mark, em, strong each legitimately appear, and what's the XSS consideration?

mark wraps the matched query terms in each snippet (its exact purpose: contextual relevance); em/strong only if the source content itself contained them (preserve, don't invent); strong might mark a "best answer" label — importance, not relevance. XSS: highlighting requires injecting markup into user-derived content (the query echoed back, the snippet text) — the classic reflected-XSS shape. Escape the content first, then wrap matches with mark (operate on text nodes, or escape-then-replace); never regex-insert <mark> into raw HTML, and never put the raw query into innerHTML. Search pages are among the most-attacked surfaces precisely because of this pattern.

🏋️ Exercises

1 (easy): Classification drill. For each, name the right element: (a) a Latin species name, (b) "never share your recovery codes", (c) the word you'd stress in "we deploy tomorrow", (d) keyboard shortcut Ctrl+K, (e) a © line, (f) "SQL" with its expansion, (g) chemical formula CO₂, (h) a decorative colored dot.
Answers

(a) i (+ lang attr) (b) strong (c) em (d) kbd (e) small (f) abbr title="Structured Query Language" (g) sub for the 2 (h) span — no meaning, pure decoration. If you missed ≤2 you're on track; revisit the decision table for the ones you missed.

2 (easy): The read-aloud test. Write one sentence and mark it up three times, moving <em> to a different word each time. Below each, write in a comment what the sentence now implies.
Example 
<p><em>She</em> reviewed my code.</p>  <!-- she, not the intern -->
<p>She <em>reviewed</em> my code.</p>  <!-- actually read it, not rubber-stamped -->
<p>She reviewed <em>my</em> code.</p>  <!-- mine, not yours -->

If moving the element changes what the sentence implies, you've understood em: it's meaning, not styling.

3 (medium): Terminal transcript. Mark up a doc snippet showing: a command the user types, the output the system prints, a filename placeholder the user substitutes, and a note that the command is destructive. Use code/kbd/samp/var/strong appropriately in a pre-based block.
Solution 
<p><strong>Warning: this permanently deletes the branch.</strong></p>
<pre><kbd>git branch -D <var>branch-name</var></kbd>
<samp>Deleted branch feature-x (was 3ac1f09).</samp></pre>
<p>Verify with <code>git branch --list</code>.</p>

kbd = what you type, samp = what it prints, var = the placeholder you substitute, code = referring to a command in prose. Four claims, four elements — this is how MDN's own docs are marked up.

4 (medium): Changelog page. Build a small "Release notes" page: version heading, release <time>, three bullet changes where one uses ins (added feature), one del (removed option), one s (a note that's no longer accurate) — plus fine print in small. Inspect with DevTools and confirm every element carries its intended claim.
Solution sketch 
<h2>v2.4 — <time datetime="2026-07-01">July 1, 2026</time></h2>
<ul>
  <li><ins datetime="2026-07-01">Dark mode for dashboards.</ins></li>
  <li><del datetime="2026-07-01" cite="/blog/why-we-removed-x">Legacy CSV export.</del></li>
  <li><s>Requires the beta flag.</s> Now enabled for everyone.</li>
</ul>
<p><small>Rollout completes within 48 hours. SLA unaffected.</small></p>

Note del's cite attribute pointing at an explanation URL, and the s-then-correction pattern — expired fact struck, current fact plain.

5 (hard): Audit and fix real markup. Find (or write) a page that uses b/i/span for everything. Rewrite its inline markup honestly, keeping the rendering identical via CSS where defaults differ. Deliverable: the diff plus one sentence per change naming the claim the new element makes.
Grading rubric

Strong solutions: every b→strong change is justified by actual urgency (not blanket replacement!); i→em only where stress applies, i kept (or added with lang) for foreign terms/titles; at least one span correctly kept as span, with the sentence "no semantic claim applies". Weak solutions swap every b→strong mechanically — that's the same mistake in mirror image. The exercise is judgment, not find-and-replace.

6 (hard): Highlighter, done safely. Preview of the Security lesson: write a function that takes a plain-text snippet and a query string and returns HTML with matches wrapped in <mark>, immune to injection even if snippet or query contains <script> or quotes. (Hint: escape first, then wrap.)
Reference implementation 
function esc(s) {
  return s.replace(/[&<>"']/g, c => ({
    "&":"&amp;", "<":"&lt;", ">":"&gt;", '"':"&quot;", "'":"&#39;"
  }[c]));
}
function highlight(snippet, query) {
  const safe = esc(snippet);
  const safeQ = esc(query);
  if (!safeQ) return safe;
  // escape regex metacharacters in the (already HTML-escaped) query
  const pattern = safeQ.replace(/[.*+?^${}()|[\]\\]/g, "\\$&");
  return safe.replace(new RegExp(pattern, "gi"), m => `<mark>${m}</mark>`);
}

Order is everything: escape both inputs first, so the only tags in the output are the marks you added. Escaping after wrapping would escape your own marks; wrapping before escaping lets attacker input become live HTML. This exact ordering bug ships to production constantly — you've now fixed it before ever making it.

🎯 Key Takeaways

  • em = stress that changes meaning; strong = importance/urgency; i/b = typographic convention without those claims; span = explicitly no claim.
  • Developer docs use four distinct monospace elements: code (code), kbd (input), samp (output), var (placeholder).
  • Wrap every date/time in <time datetime="ISO-8601"> — spec elements broadcast to consumers you don't control; data-* is private to your scripts.
  • del/ins = edits (support cite/datetime); s = no longer true; mark = contextual relevance, not importance.
  • Default styling is convention, not contract — resets can strip it; semantics survive. Restyle freely; the claims remain.
  • Decision procedure: specific element exists → use it; pure styling → span; never choose by default appearance.
  • Injecting highlights into user content = escape first, then wrap — the mark element's main production trap is XSS ordering.
