Lists Foundations

What you'll master: the three list types and the judgment calls between them, nesting done correctly, ordered-list attributes you probably don't know (start, reversed, value), description lists for real key-value content, why navigation menus are lists, and what screen readers actually do with list markup. Prerequisites: lessons 1–4.

Part 1 — Three Types, One Decision Rule

<ul>  unordered — order does NOT matter     (ingredients, features, nav links)
<ol>  ordered   — order IS the information   (steps, rankings, legal clauses)
<dl>  description — name → description pairs (glossaries, metadata, FAQ, specs)

The test for ul vs ol: shuffle the items — is anything lost? Shuffled ingredients: still a valid shopping list → ul. Shuffled brewing steps: broken recipe → ol. This is a semantic claim like every other in this track: an ol tells machines (and screen-reader users, who hear the numbers) that position carries meaning.

<ul>
  <li>500 g bread flour</li>
  <li>375 g water</li>
  <li>10 g salt</li>
</ul>

<ol>
  <li>Mix flour and water; rest 30 minutes.</li>
  <li>Add salt and starter.</li>
  <li>Fold every 30 minutes for 3 hours.</li>
</ol>

Content rules (the parser enforces these with silent repairs — lesson 1's theme): ul/ol may contain only <li> elements (plus script/template). Text or other elements directly inside a ul get hoisted out by the parser. An <li>, on the other hand, may contain anything: paragraphs, images, headings, entire nested structures. That asymmetry — strict parent, permissive child — is what makes lists the workhorse container for cards, menus, feeds, and comment threads.

Part 2 — Nesting, Done Right and Done Wrong

The #1 list mistake in the wild is misplaced nesting. A nested list goes inside the parent <li> — inside the item it belongs to — not between items:

❌ WRONG — the sublist is a direct child of ul (invalid; parser repairs unpredictably):
<ul>
  <li>Frontend</li>
  <ul>
    <li>HTML</li>
  </ul>
</ul>

✅ RIGHT — the sublist lives inside "Frontend"'s li, because it elaborates Frontend:
<ul>
  <li>Frontend
    <ul>
      <li>HTML</li>
      <li>CSS</li>
    </ul>
  </li>
  <li>Backend</li>
</ul>

Why it matters beyond validity: screen readers announce nesting depth ("list, 2 items… nested list, 2 items") — correct nesting is the outline a listening user perceives. Mixed nesting is fine and common: an ol of steps where one step contains a ul of options. Browsers automatically vary the bullet style per depth (disc → circle → square); numbering style per depth is yours to control in CSS (list-style-type: lower-alpha etc.).

Part 3 — Ordered-List Superpowers: start, reversed, value, type

<ol start="6">          <!-- continues a list split by a paragraph or ad: 6, 7, 8 -->
<ol reversed>           <!-- countdown: 10, 9, 8... (top-10 lists!) -->
<ol>
  <li>Alpha</li>
  <li value="10">Jumps to 10</li>   <!-- subsequent items continue: 11, 12 -->
</ol>
<ol type="i">           <!-- i, ii, iii — legal documents; prefer CSS list-style-type
                             EXCEPT where the numbering is content (contracts, citations) -->

The classic real-world case for start: an article interrupts a numbered list with a paragraph of commentary, then resumes. Two <ol>s, the second with start="4" — the numbering stays honest. Without it, every "continued" list on the page restarts at 1 and someone patches it with fake bold numbers in a ul (you will see this in real codebases; now you can fix it properly).

Part 4 — Description Lists: the Underused Third Type

<dl> holds name–description groups: <dt> (term) + <dd> (description). Far more of your real content is key-value shaped than you'd think:

<!-- Product specs — the canonical use: -->
<dl>
  <dt>Weight</dt>      <dd>1.2 kg</dd>
  <dt>Material</dt>    <dd>Anodized aluminum</dd>
  <dt>Warranty</dt>    <dd>3 years, parts and labor</dd>
</dl>

<!-- Multiple dd per dt (one term, several descriptions): -->
<dl>
  <dt>Serves</dt>
  <dd>2 adults</dd>
  <dd>or 1 hungry developer</dd>
</dl>

<!-- Multiple dt per dd (synonyms sharing a definition): -->
<dl>
  <dt>Mocha</dt><dt>Caffè mocha</dt>
  <dd>Espresso with chocolate and steamed milk.</dd>
</dl>

Metadata blocks (Author / Published / Reading time), FAQ pages, API parameter docs, glossaries — all dl-shaped. Layout note: dl/dt/dd are begging for CSS grid two-column layouts (CSS track); don't reject the element because its default styling is plain. One more spec detail: since HTML5.2 you may wrap each group in a <div> inside the dl — purely as a styling/grouping hook: <dl><div><dt>…</dt><dd>…</dd></div>…</dl>.

📏 dl vs table, the judgment call: one entity's properties (a product's specs) → dl. Many entities sharing the same properties (comparing 5 products) → table (next lesson). The moment you want column headers, you've outgrown dl.

Part 5 — Lists as Application Structure (why nav menus are ULs)

Open any production site's header: the links live in <nav> > <ul> > <li> > <a>. Why a list, when CSS strips the bullets anyway?

  • Screen readers announce the count up front: "navigation, list, 5 items" — the user instantly knows the menu's size, and can skip the whole thing as one unit. A pile of bare <a>s gives neither.
  • The structure IS a list — a collection of parallel items. Semantic honesty, same as everywhere else in this track.
  • Dropdown submenus map exactly onto nested lists — structure for free.
<nav aria-label="Main">
  <ul>
    <li><a href="/roasts">Roasts</a></li>
    <li><a href="/brewing">Brewing</a>
      <ul>
        <li><a href="/brewing/pourover">Pour over</a></li>
        <li><a href="/brewing/espresso">Espresso</a></li>
      </ul>
    </li>
    <li><a href="/visit">Visit</a></li>
  </ul>
</nav>

The same pattern powers card grids (ul of products), comment threads (nested lists), breadcrumbs (ol — order matters!), and step indicators. When a collection of parallel things needs markup, the answer is usually a list with rich <li> contents.

🔍 Senior gotcha: list-style: none causes Safari + VoiceOver to stop announcing the list as a list (a deliberate heuristic — too many decorative lists). If list semantics genuinely matter for a de-bulleted list, add role="list" to the ul. Knowing this exact quirk is a strong accessibility-interview signal.

Interview Questions

Q1 (beginner): How do you decide between ul, ol, and dl?

Shuffle test for ul vs ol: if reordering loses information (steps, rankings), it's ol; otherwise ul. dl is for name→description pairs: specs, glossaries, metadata. Bonus: mention that the choice is a machine-readable claim — screen readers read ol numbers aloud and announce item counts.

Q2 (beginner): What may legally sit directly inside a ul? Inside an li?

ul/ol: only li (plus script/template). li: anything — flow content, paragraphs, images, nested lists, headings. Strict parent, permissive child. The strictness is parser-enforced: stray content inside a ul gets relocated by the repair algorithm, producing a DOM that doesn't match the source.

Q3 (mid): An article's numbered list is interrupted by paragraphs of commentary, then continues. Mark it up honestly.

Two ordered lists, the second carrying start="n" (e.g., start="4" if three items preceded). Alternatives: value on the first li of the continuation. Anti-pattern to name: a ul with hand-typed "4." text — it decouples the claimed numbering from reality and breaks when items are inserted. If the numbering is deeply content-critical (legal docs), note that type="a"/"i" attributes are acceptable in HTML precisely because the numbering is content, not style.

Q4 (mid): Why is nav > ul > li > a the standard pattern rather than bare anchors in a nav?

Screen readers announce "list, N items" giving menu size up front, and allow list-level skipping; nesting maps to submenus; and it's semantically honest — a menu is a collection of parallel items. Also stable styling hooks. Follow-up worth volunteering: with list-style:none, Safari/ VoiceOver drops the list semantics — role="list" restores them; and the nav itself gets an aria-label when there are multiple navs (main, footer, breadcrumbs).

Q5 (senior): Your design system renders comment threads. Compare nested lists vs flat divs with data-depth for the tree, considering AT, virtualization, and maintenance.

Nested ul inside the parent comment's li is the semantically correct tree: AT announces depth and item counts natively; styling child threads is structural (ul ul) rather than attribute-driven. Costs: deep threads = deep DOM (styling/perf manageable), and virtualization (rendering only visible comments, needed at scale) fights real nesting — windowing libraries want flat lists. Real systems often compromise: flat rendering with aria-level/role="treeitem" or aria-setsize/posinset to preserve announced depth over a flat DOM. Senior answer shape: correct semantics by default, documented escape hatch under measured performance pressure, never silently losing the depth information AT users rely on.

Q6 (senior): The spec once suggested dl for dialogue and no longer does. What does this teach about semantic drift, and how do you keep a team's markup current?

Element meanings are versioned social contracts — dl was redefined from "definition list" to "description list" (name-value groups), explicitly excluding dialogue; similar drift: b/i/s/small were rehabilitated with new meanings, menu/dialog changed roles. Consuming tools follow the current contract, so stale idioms slowly rot in AT and extraction quality. Keeping current: lint with html-validate/axe in CI (rules track the living standard), prefer MDN over blog folklore, encode decisions in a team markup guide with examples, and review markup semantics in code review like any API usage. The meta-lesson: cite the living standard, not what HTML4 taught you.

🏋️ Exercises

1 (easy): Choose and build. Mark up: (a) five countries by population (largest first), (b) your grocery list, (c) three coffee terms with one-line definitions. One list each; justify each choice in a comment.
Answers

(a) ol — the ranking IS the information (bonus flair: this is also a defensible reversed candidate if you present 5th→1st). (b) ul — shuffle-safe. (c) dl with three dt/dd pairs. If you used ul for (a): shuffle it — "3rd most populous" just became wrong; order carries meaning → ol.

2 (easy): Fix the nesting. Repair this and verify your fix in the Elements panel: <ul><li>Coffee</li><ul><li>Espresso</li> <li>Filter</li></ul><li>Tea</li></ul>
Solution 
<ul>
  <li>Coffee
    <ul>
      <li>Espresso</li>
      <li>Filter</li>
    </ul>
  </li>
  <li>Tea</li>
</ul>

The sublist moves inside Coffee's li — it elaborates Coffee. In the broken version, check what the parser actually built (it varies!) — the repaired tree won't match anyone's intent, which is the whole argument for getting it right.

3 (medium): The interrupted top-10. Build a "Top 10 brewing mistakes" article that counts DOWN from 10, is interrupted after item 6 by a paragraph and an image, then resumes correctly at 5. Verify the numbering renders right.
Solution sketch 
<ol reversed start="10">
  <li>Boiling water straight onto the grounds</li>
  ... (items 10 → 6)
</ol>
<p>Before the top five, a word about grinders...</p>
<img src="grinder.avif" alt="..." width="800" height="500">
<ol reversed start="5">
  <li>Stale beans</li>
  ... (items 5 → 1)
</ol>

reversed + start together handle the countdown-with-interruption case with zero fake numbers. If your second list shows 5,4,3… you've got it.

4 (medium): Spec sheet + FAQ. Add to the coffee site: a product dl (6+ spec pairs, including one term with two descriptions) and an FAQ marked up as a dl. Then state in one sentence why the FAQ might be better served by details/summary instead (you met it in this track's exercises; it's formalized in the semantic lessons).
Solution notes

Spec dl is mechanical (Weight/Origin/Roast/…). FAQ as dl: dt = question, dd = answer — semantically fine (name→description). The one-sentence answer: details/summary adds native expand/collapse interaction for free, which most FAQ designs want; dl is purely structural. Either is defensible; interactive disclosure usually wins for long FAQs.

5 (hard): Real nav, keyboard-checked. Build the Part 5 nav with one dropdown submenu, style it horizontal with CSS (list-style:none, flex — borrow from any example), then: tab through it with the keyboard and listen with a screen reader if available. Report: item count announced? Submenu reachable? Then add role="list" and note any Safari/VoiceOver difference.
What you should find

Tab reaches every link in DOM order including the submenu (links are natively focusable — no JS needed for reachability; hover-only reveal hiding the submenu from keyboard users is the classic bug — if your CSS hides it except on :hover, keyboard users can focus invisible links: fix with :focus-within). NVDA/VoiceOver: "navigation … list … 3 items". After list-style:none, VoiceOver (Safari) may drop "list" — role="list" restores it. You've just done a genuine component accessibility review.

6 (hard): List linter. Console script: find (a) uls/ols with element children that aren't li, (b) lists nested as direct children of lists (the Part 2 bug), and (c) "fake lists" — 3+ consecutive paragraphs starting with •, -, or a number-dot. Run it on two real sites.
Reference implementation 
document.querySelectorAll('ul, ol').forEach(list => {
  [...list.children].forEach(child => {
    const tag = child.tagName;
    if (!['LI','SCRIPT','TEMPLATE'].includes(tag))
      console.warn('Non-li child', tag, 'in', list);
    if (tag === 'UL' || tag === 'OL')
      console.warn('List nested as sibling of li (should be inside one):', child);
  });
});
const paras = [...document.querySelectorAll('p')];
let run = [];
for (const p of [...paras, null]) {
  const fake = p && /^\s*([•\-*]|\d+[.)])\s/.test(p.textContent);
  if (fake) run.push(p);
  else { if (run.length >= 3) console.warn('Fake list of', run.length, 'paragraphs:', run[0]); run = []; }
}

Finding (c) hits on real CMS content constantly — authors typing bullets by hand. Each hit is a free semantic upgrade waiting to happen: convert to a real list, gain AT announcements and styling control at once.

🎯 Key Takeaways

  • Shuffle test: order meaningful → ol; not → ul; name→value pairs → dl. The choice is a machine-readable claim.
  • ul/ol accept only li children (parser-enforced); li accepts anything — which makes lists the container for menus, cards, feeds, and threads.
  • Nested lists live inside the li they elaborate — nesting depth is what screen readers announce.
  • start, reversed, and value make interrupted, countdown, and custom numbering honest — no fake numbers ever.
  • dl handles specs, glossaries, metadata, and FAQs; multiple dt/dd per group is legal, div wrappers are allowed for styling; many-entities comparisons graduate to tables.
  • Nav menus are lists for the count announcement and structural honesty; list-style:none can drop AT semantics in Safari — role="list" restores.
