Responsive Images Media & Embedding

What you'll master: srcset and sizes — the two attributes that let ONE <img> serve the right file to every device, the density-vs-width descriptor distinction that trips up everyone, and <picture> for art direction and format fallback. This is the payoff lesson for Images Mastery (lesson 6) and sets up Performance later. Prerequisites: Images Mastery.

Part 1 — The Problem This Solves

A single <img src="hero.jpg"> forces one impossible choice: ship a huge file (wasted bandwidth on phones) or a small one (blurry on retina desktops). Real traffic spans a 6× range of device pixel densities and a 10× range of viewport widths — one file cannot be correctly sized for all of them. srcset offers the browser several image candidates and lets IT choose, per-device, using information only the browser has: actual pixel density, actual viewport width, even the user's data-saver setting. This is the same "browser knows more than you" pattern behind <video> codec fallback and <source type> — you'll see it again in the next lesson.

Part 2 — Density Descriptors: the Simple Case (retina)

<img src="logo.png"
     srcset="logo.png 1x, logo@2x.png 2x, logo@3x.png 3x"
     alt="Maya's Coffee">

Use when the image displays at a fixed CSS size regardless of viewport (logos, icons, avatars). 1x/2x/3x tell the browser each file's pixel density relative to CSS pixels; it picks the best match for the device's devicePixelRatio (a retina MacBook picks 2x, a budget Android might pick 1x, an iPhone Pro might grab 3x). src remains the fallback for browsers that don't understand srcset (effectively none today, but it's also what renders if srcset is malformed — never omit it).

Part 3 — Width Descriptors: the Real Case (fluid images)

Most content images aren't fixed-size — they scale with the viewport (a blog's hero, a product photo in a fluid grid). For these, density is the wrong axis; width is:

<img src="hero-800.jpg"
     srcset="hero-400.jpg 400w,
             hero-800.jpg 800w,
             hero-1200.jpg 1200w,
             hero-2000.jpg 2000w"
     sizes="(max-width: 600px) 100vw, (max-width: 1000px) 50vw, 800px"
     alt="Roasted beans cooling on a tray"
     width="800" height="533" loading="lazy">

Two attributes, two different jobs — this pairing is the single most-confused part of responsive images, so slow down here:

  • srcset lists candidates with their intrinsic width in pixels (400w = "this file is actually 400px wide"). Purely a menu — it doesn't say anything about layout.
  • sizes tells the browser how wide the image will actually be displayed, as a CSS-length per media condition, evaluated before layout even runs (that's the whole point — see Part 4). It answers "at this viewport, how many CSS pixels will this image occupy?"
  • The browser combines them: reads sizes to learn the slot width at the current viewport (say, 50vw at a 1000px viewport = 500px), multiplies by the device's pixel density (2x retina = 1000px needed), then picks the smallest srcset candidate that covers that — hero-1200.jpg. Nobody hand-computes this; the browser's algorithm does, correctly, every time.
⚠ The single most common bug: writing width descriptors in srcset but forgetting sizes entirely. Without sizes, the browser assumes the image is 100vw (full viewport width) — so a hero image styled to fill a 500px-wide sidebar downloads the 2000px-wide file every time, on every device. srcset without sizes is nearly always wrong for fluid images.

Part 4 — Why sizes Can't Be Computed From CSS (the deep "why")

The natural question: "why don't browsers just read my CSS width and figure it out?" Because image selection happens during HTML parsing, before the CSSOM and layout exist — the preload scanner (lesson 2's streaming-parser callback) needs to start downloading images the instant it sees the <img> tag, for performance. If it had to wait for full CSS layout first, you'd lose the entire head-start that makes preloading valuable. So sizes exists as a parallel, parseable-without-layout description of the same information your CSS will eventually express — a genuine duplication, and the reason this API feels like "shouldn't the browser just know this?" It structurally can't, yet, at the moment it needs to.

Part 5 — picture: When Candidates Aren't Just Resizes

srcset assumes every candidate is the same image at different resolutions. Two real cases need genuinely different sources:

Case 1: format fallback (met conceptually in lesson 6, here's the syntax)

<picture>
  <source srcset="hero.avif" type="image/avif">
  <source srcset="hero.webp" type="image/webp">
  <img src="hero.jpg" alt="Roasted beans cooling on a tray" width="1200" height="600">
</picture>

The browser walks the <source>s top to bottom and uses the first one whose type it supports — AVIF-capable browsers stop at line 1; others fall through to WebP, then to the <img>'s plain JPEG, which is mandatory (it's both the universal fallback and the element that actually renders — alt/width/height live there, never on source). Order matters: best format first, since first-match wins.

Case 2: art direction (a genuinely different crop, not just a resize)

<picture>
  <source media="(max-width: 600px)" srcset="hero-tight-crop.jpg">
  <source media="(max-width: 1000px)" srcset="hero-medium-crop.jpg">
  <img src="hero-wide-crop.jpg" alt="Barista pouring latte art, café interior in background"
       width="1200" height="600">
</picture>

On mobile a wide establishing shot goes illegible — cropping tighter on the subject (the cup, not the whole café) is a genuine editorial decision, not a resize. media conditions (real CSS media queries) pick the crop; only ONE source can match at a time here, unlike the type-fallback case. This is "art direction": different pixels for different contexts, versus srcset's "same picture, different resolution."

Part 6 — Step-by-Step Build

1. Design your breakpoints from the LAYOUT, not guesswork:
   mobile: image is 100vw · tablet: 50vw in a 2-col grid · desktop: fixed 800px in a sidebar

2. Export 3-4 width variants covering the range with ~1.5-2x steps:
   400w, 800w, 1200w, 2000w  (each in AVIF, per lesson 6)

3. Write sizes to MATCH your actual CSS breakpoints exactly:
   sizes="(max-width: 600px) 100vw, (max-width: 1000px) 50vw, 800px"

4. Write srcset listing every exported width:
   srcset="hero-400.jpg 400w, hero-800.jpg 800w, hero-1200.jpg 1200w, hero-2000.jpg 2000w"

5. Verify in DevTools: Network panel, throttle to a phone viewport + reload —
   confirm the SMALL file downloaded, not the 2000w one. Resize the window and reload at
   desktop width — confirm a LARGER file was chosen. This is the one habit that catches
   90% of responsive-image bugs: always check the actual bytes on the wire.

Interview Questions

Q1 (beginner): What problem does srcset solve that a single img src can't?

One file can't be simultaneously small-enough-for-mobile-bandwidth and sharp-enough-for-retina- desktop — real traffic spans a wide range of device pixel densities and viewport widths. srcset offers multiple candidate files and lets the browser choose per-device using information (actual DPR, actual viewport, network conditions) that only it has at request time.

Q2 (beginner): Density descriptors vs width descriptors — when does each apply?

Density (1x/2x/3x) for images that render at a FIXED CSS size regardless of viewport — logos, icons, avatars — describing "this file is N times the CSS size" for retina handling. Width (400w/800w…) for fluid images that scale with viewport/layout — paired mandatorily with sizes, which tells the browser the actual rendered slot width per media condition.

Q3 (mid): A developer adds srcset with width descriptors but the Network tab still shows the largest file loading on mobile. Diagnose.

Missing sizes attribute — without it, the browser defaults to assuming the image renders at 100vw and picks accordingly (usually the largest candidate, especially at high DPR). The fix is adding sizes that accurately describes the rendered width per breakpoint, matching the real CSS. This is the single most common responsive-images bug, and the diagnostic habit (Network tab, throttled viewport) is as important as knowing the cause.

Q4 (mid): Why can't the browser just read your CSS to determine image size, instead of requiring the separate sizes attribute?

Because the preload scanner selects and starts downloading images while parsing HTML, before CSSOM/layout exist — waiting for full layout first would forfeit the performance benefit of early image requests. sizes is a parseable-without-layout duplicate of information your CSS will eventually express. It's a real architectural constraint, not an oversight — and it's why sizes must be kept honest against your actual breakpoints or the browser is choosing based on stale information.

Q5 (senior): Design the responsive-image pipeline for a CMS where editors upload arbitrary photos into a fluid 3-column grid that collapses to 1 column on mobile. What does the system need to automate?

On upload: generate a fixed set of width variants (e.g., 400/800/1200/2000) in AVIF+WebP+JPEG per image (format fallback via picture, resolutions via srcset within each source) — never trust editors to export variants manually. The sizes attribute should be GENERATED from the same breakpoint config the CSS grid uses (single source of truth — a build-time or CMS-config value feeding both the Sass/CSS breakpoints and the templated sizes string), because hand-syncing two independent strings drifts silently. Width/height (or aspect-ratio) must be stored as image metadata at upload time so every render includes them for CLS. A CDN with on-the-fly resizing (Cloudinary/Imgix/自建) is the usual implementation, parameterizing width+format per request rather than pre-generating every combination. Cache the transforms; regenerate on re-crop only. The senior shape: automate what an editor could get wrong, and keep sizes and layout breakpoints mechanically linked, not manually synced.

Q6 (senior): When would you reach for picture's art-direction mode over just accepting a single crop that scales down? Give a concrete product case and the cost you're accepting.

Genuine case: a wide establishing/landscape photo (café storefront, team photo, panoramic hero) where the subject becomes illegible or oddly composed when simply scaled into a narrow mobile viewport — the editorial intent (what the photo is "of") changes with aspect ratio, not just size. A portrait-crop close-up on mobile and a wide establishing shot on desktop are different photos serving the same emotional purpose. Cost being accepted: multiple crops to produce and maintain per image (each needing its own width-variant set for srcset within each source — art direction and resolution switching compose, multiplying asset count), more CMS/editor workflow complexity (someone must choose/approve each crop), and a genuinely harder-to-test matrix (crop × format × width × breakpoint). Most product images don't warrant this — a single well- chosen crop that scales via plain srcset/sizes is simpler and covers the vast majority of cases; reserving picture's art-direction mode for hero/editorial imagery where the cost is justified is the practiced judgment call.

🏋️ Exercises

1 (easy): Retina logo. Export your coffee site's logo at 1x/2x/ 3x (any image editor, or just resize the same file three ways for practice). Wire up density descriptors. In DevTools, use the device toolbar to simulate a DPR-1 and DPR-3 device and confirm different files load each time (Network panel, filter by Img).
Check

DPR-1 simulated device requests logo.png; DPR-3 requests logo@3x.png. If both request the same file, check the device toolbar's DPR setting took effect (a hard reload is sometimes needed — cached responses don't re-trigger selection).

2 (easy): Diagnose the bug. Given <img src="a.jpg" srcset="a-400.jpg 400w, a-1200.jpg 1200w"> with no sizes, styled to display at 300px wide in a sidebar — explain exactly which file loads and why, then fix it.
Answer

a-1200.jpg loads (or a-400.jpg only accidentally, on a slow/narrow test) — no sizes means the browser assumes 100vw, wildly overestimating a 300px sidebar slot at any real viewport width. Fix: sizes="300px" (fixed slot) — now the browser correctly picks a-400.jpg even at high DPR (300×2=600, still under 1200, and a-400 likely still wins over a-1200 depending on exact math — the point is it now COMPUTES instead of assuming full-width).

3 (medium): The full fluid hero. Build Part 3's complete example with real exported files (Squoosh.app can batch-resize) at 400/800/1200/2000w, sizes matching a 3-breakpoint layout, width/height set. Verify at three simulated viewports (375px, 800px, 1440px) that the Network tab shows a DIFFERENT, appropriately-sized file at each.
Expected mapping

375px (mobile, 100vw) → 400w file. 800px (tablet, 50vw = 400px slot, ×DPR) → likely 800w. 1440px (desktop, fixed 800px slot) → 1200w (covers 800px comfortably even at 1.5x DPR without over-fetching 2000w). Exact picks depend on your DPR simulation — the exercise's real skill is reading the Network tab to VERIFY rather than assume.

4 (medium): Format fallback picture. Wrap your hero in picture with AVIF and WebP sources plus a JPEG img fallback (reuse lesson 6's format exports). In DevTools, use the "Network conditions" or extension tooling to force-disable AVIF support (or just check which loads on your current browser) and explain the observed fallback chain.
What to observe

A modern browser loads the AVIF source (smallest bytes — confirm in the Size column vs the other formats' sizes for comparison). There's no easy in-browser AVIF-disable toggle, so the practical version of this exercise is comparing file sizes across the three exports and trusting the type-matching algorithm (documented in Part 5) rather than observing the fallback directly — note in your answer which real-world browsers/versions this fallback chain actually protects (older Safari/Firefox releases, primarily).

5 (hard): Art direction, built. Take (or fake) a wide landscape photo. Create a tight mobile crop (crop to the center 40% width) and a medium tablet crop using any image tool. Build the full art-direction picture from Part 5, and confirm via DevTools' device toolbar at 3 widths that visually DIFFERENT crops appear (not just different sizes of the same crop).
Verification

At ≤600px you should see the tight crop's composition (not a shrunk wide shot); at ≤1000px the medium crop; above that the full wide shot. If you see the same composition just scaled at every width, your source elements aren't matching correctly — check media attribute syntax and source ORDER (first-match-wins top to bottom is unforgiving of misordered conditions).

6 (hard): Audit a real site. Pick any image-heavy site (news, e-commerce). Inspect 3 of its images: do they use srcset? sizes? picture? Are the sizes values plausible given the layout? Using the Network tab at 3 simulated viewports, determine whether appropriately-sized files are actually being served, or whether the site is over-fetching. Write a 3-bullet findings report.
What real audits typically find

Most major sites use srcset+sizes correctly on hero/product images (it's standard practice at scale) but often skip it on secondary content images (article body photos, related-content thumbnails) — inconsistency across a codebase is the norm, not the exception. A good finding calls out a SPECIFIC over-fetch: "the related-articles thumbnail at 150px wide is served as a 1200px file, an 8x over-fetch" — with the Network tab numbers to back it. This is genuinely the kind of finding that gets filed as a real performance ticket.

🎯 Key Takeaways

  • srcset offers candidate files; the browser chooses using device information (pixel density, viewport, network) unavailable to you at authoring time.
  • Density descriptors (1x/2x/3x) for fixed-size images; width descriptors (400w…) + mandatory sizes for fluid images — mixing them or omitting sizes is the #1 bug (defaults to 100vw, over-fetches).
  • sizes exists as a separate, layout-independent string because image selection happens during HTML parsing, before CSSOM/layout — keep it mechanically synced to your real CSS breakpoints.
  • picture handles two cases srcset can't: format fallback (first supported type wins, ordered best-first) and true art direction (different crops via media queries, first match wins).
  • alt, width, and height always live on the actual img — never on source, which has no rendering role.
  • Always verify in the Network tab at multiple simulated viewports — the algorithm is correct, but your sizes string is a claim the browser trusts blindly.
