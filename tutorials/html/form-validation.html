Built-in Validation Forms Mastery

What you'll master: every constraint attribute (required, min/max, minlength/ maxlength, pattern, step), how the browser's validation flow actually behaves at submit time, the CSS pseudo-classes that style validity (and the :user-invalid fix for their biggest UX flaw), and the three-layer validation model every production app uses. The JS-powered half (Constraint Validation API) is the next lesson. Prerequisites: Forms Fundamentals, Every Input Type.

Part 1 — The Three-Layer Model (memorize this)

Layer 1: HTML attributes   instant, free, works without JS      → UX guardrail
Layer 2: JavaScript        custom rules, async checks, nice UI  → UX enhancement
Layer 3: SERVER            the only layer that provides security → THE gate

Layers 1–2 run on a machine the user controls — DevTools can delete every attribute, disable every script, or skip the browser entirely (curl). So client validation is a courtesy: it catches mistakes early, cheaply, kindly. Server validation is the law: it defends data integrity against everything the client layer cannot promise. Every "validation bypassed" security finding in history reduces to a team confusing the courtesy for the law. This lesson is Layer 1 done fully.

Part 2 — The Constraint Attributes

<input required>                          must not be empty (checkbox: must be checked)
<input type="number" min="1" max="20">    numeric/date bounds (also date/range/time)
<input minlength="8" maxlength="64">      string length bounds
<input pattern="[A-Za-z0-9-]+">           regex the WHOLE value must match
<input type="number" step="0.25">         value granularity (0.25 increments; "any" disables)

Detail notes that bite in practice:

  • required on radios: put it on any one radio of the group — the constraint applies to the group ("one must be selected"). On checkboxes it means this specific box must be checked (terms-of-service pattern). "At least one of these checkboxes" has NO native constraint — that's next lesson's custom-validity territory.
  • maxlength is a silent trap: it doesn't show errors — it prevents typing past the limit. Users pasting a too-long value get silent truncation. Decide whether that's the UX you want; often minlength+counter UI (JS) is kinder.
  • pattern is anchored: it must match the entire value (implicit ^…$) and doesn't run on empty values — pattern without required lets empty through (usually what you want: "optional, but if present, shaped like this").
  • Always pair pattern with a title attribute or visible hint — the browser's default error ("Please match the requested format") is useless without your explanation; title text gets appended to it.
  • step interacts with min: validity requires (value − min) divisible by step — min="0.5" step="1" accepts 1.5, not 2. And type="number" ignores minlength/maxlength (length is a string concept); use min/max.
<!-- A production-grade username field, all pieces cooperating: -->
<label for="user">Username</label>
<input id="user" name="user" type="text"
       required minlength="3" maxlength="20"
       pattern="[a-z0-9-]+"
       title="Lowercase letters, digits, and hyphens only"
       autocomplete="username">
<p class="hint" id="user-hint">3–20 characters: a–z, 0–9, hyphens.</p>

Note the visible hint paragraph: constraints users can't see are traps, not guidance. (Formally associating it via aria-describedby="user-hint" is the accessibility upgrade — Form UX lesson.)

Part 3 — What Happens at Submit (the flow, precisely)

User clicks submit
  → browser checks every control against its constraints, in DOM order
  → ALL VALID:   submission proceeds (Layer 1 passed; the server still re-checks)
  → ANY INVALID: submission is CANCELLED
       · focus jumps to the FIRST invalid control
       · a native error bubble appears on it (localized to the user's browser language, free!)
       · the form fires no submit event at all — your JS submit handler never runs

That last line is a real debugging classic: "my submit handler stopped firing" — because a hidden or forgotten required field is invalid, the event never dispatches. Also know novalidate/formnovalidate:

<form novalidate>                    turn native checking off for the whole form
<button formnovalidate>Save draft</button>   bypass for ONE button (drafts! back-buttons!)

novalidate is how apps that want fully custom error UI keep the constraint data (the attributes still describe validity — next lesson's API reads them) while suppressing native bubbles. formnovalidate solves "the Back button on step 2 refuses to work because step 2's fields are empty."

Part 4 — Styling Validity: the CSS Hooks and Their Big Flaw

input:required   { border-left: 3px solid var(--accent); }
input:valid      { border-color: green; }
input:invalid    { border-color: red; }       /* ⚠ read on before using */
input:out-of-range { background: #fff3f3; }   /* number/date outside min/max */

The flaw: :invalid matches from page load. A required empty field is technically invalid at first paint — so the user opens a pristine form already bleeding red. Terrible UX, and the reason many teams abandoned native validation styling. The modern fix:

/* Only flags fields the user has actually interacted with: */
input:user-invalid { border-color: red; }
input:user-valid   { border-color: green; }

:user-invalid/:user-valid (supported in all modern browsers since 2023) match only after the user has edited or attempted to submit — red appears when it's feedback, not accusation. Interim pattern you'll still meet in codebases: .was-submitted input:invalid with a class added on first submit attempt.

Part 5 — Worked Example: Full Signup Form, Layer 1 Complete

<form action="/signup" method="post">
  <label for="em">Email</label>
  <input id="em" name="email" type="email" required autocomplete="email">

  <label for="pw">Password</label>
  <input id="pw" name="password" type="password"
         required minlength="12" maxlength="128"
         autocomplete="new-password"
         title="At least 12 characters">
  <p class="hint">12+ characters. Length beats complexity — passphrases welcome.</p>

  <label for="hnd">Handle</label>
  <input id="hnd" name="handle" required pattern="[a-z0-9-]{3,20}"
         title="3–20 chars: lowercase letters, digits, hyphens">

  <label for="dob">Date of birth</label>
  <input id="dob" name="dob" type="date" required
         min="1900-01-01" max="2013-07-12">   <!-- 13+ policy, server re-checks -->

  <label><input type="checkbox" name="tos" value="agreed" required>
         I accept the <a href="/terms">terms</a></label>

  <button type="submit">Create account</button>
</form>

Notes: password composition rules (uppercase+digit+symbol) are deliberately absent — current guidance (NIST) favors length over composition theater, and pattern-based composition rules infuriate password managers' generated passwords. The max on dob implements an age gate as UX; the server enforces the actual policy (Layer 3, always). Every constraint has a visible explanation. This form works perfectly with JavaScript disabled.

Interview Questions

Q1 (beginner): Why do we validate on the client at all if the server must validate anyway?

Speed and kindness: instant feedback with no round trip, before the user leaves the field or the page; free localized error messages; less server load from doomed submissions. The framing: client validation is UX, server validation is security — they answer different questions and you always need both.

Q2 (beginner): What does required do on a checkbox vs on one radio button of a group?

Checkbox: this specific box must be checked (terms-of-service). Radio: applied to any member, it requires that the group (same name) have a selection. Gap worth naming: "at least one of several checkboxes" has no native attribute — it needs the Constraint Validation API's setCustomValidity.

Q3 (mid): A dev complains their form's submit event handler "randomly doesn't fire." Walk through your diagnosis.

Most likely: native validation cancels submission before the event dispatches — any invalid control (check for required fields that are display:none'd by a wizard step — the notorious case, which also logs "not focusable" console errors). Diagnose: form.checkValidity() in the Console lists false; iterate elements checking el.validity. Fixes: remove/disable constraints on hidden controls, or novalidate + API-driven validation. Also verify nobody put the handler on the button's click and swapped it to type="button". The senior tell: knowing validation runs pre-event.

Q4 (mid): :invalid vs :user-invalid — what problem does the newer one solve?

:invalid matches whenever constraints aren't met — including at first paint for a required empty field, so untouched forms open red ("accusatory red"). :user-invalid matches only after user interaction with the field (edit or attempted submit), turning the styling into feedback rather than a greeting. Pre-2023 workaround worth naming: a .submitted class toggled on submit attempt scoping the :invalid styles.

Q5 (senior): Your signup rejects "müller@büro.de" and a user with the password "correct horse battery staple" complains your rules forced them to weaken it. Audit the form's likely constraints.

Two distinct over-validation smells. The email: someone replaced native type="email" checking with an ASCII-only regex — internationalized addresses (IDN/EAI) exist; fix: loosen to native validation + verification email as the real check (deliverability is the only truth). The password: a composition pattern (must contain digit+symbol+case mix) rejected a high-entropy passphrase — composition rules anti-correlate with real strength and fight password managers; fix: minlength 12–16, maxlength generous (128+, never 16!), no composition pattern, check against breached-password lists server-side (that's Layer 3). Underlying principle: validation should reject impossible values, not unusual ones — false rejections are conversion and security bugs at once.

Q6 (senior): Design the validation strategy for a multi-step checkout wizard (3 steps, one form submission at the end). Where does native validation help and where does it fight you?

Fights you: one form spanning steps means hidden steps' required fields block submission with unfocusable-error chaos; Back buttons trip validation. Options: (a) per-step forms, each POSTing to the server (or intercepted) — native validation per step works cleanly, state accumulates server-side/in storage — the progressive-enhancement-friendly choice; (b) one form + novalidate, running checkValidity() manually on the current step's fieldset at each Next (the API reads constraints per-subtree — next lesson), formnovalidate on Back; (c) SPA-style full custom — most control, most work, native localization lost. Regardless: constraints as attributes stay as the single source of truth (JS reads them), errors summarized accessibly at step level, and the server validates the complete order at the end plus per-step if state is server-held. Senior shape: name the hidden-required trap unprompted.

🏋️ Exercises

1 (easy): Constraint bingo. Build one form using every constraint attribute at least once (required, min, max, minlength, maxlength, pattern, step). Trigger each one's native error and screenshot/note the message text your browser shows.
Check

Seven attributes, seven distinct bubble messages (your browser's language!). Note which message was useless without context — almost certainly pattern's — and confirm your title attribute text appears appended to it. That observation IS the lesson's pattern rule.

2 (easy): The pristine-form fix. Style a required email field with :invalid{border:red} and open the page — observe the accusatory red. Fix with :user-invalid and verify: clean at load, red after typing garbage and tabbing away, green (add :user-valid) after fixing it.
Expected sequence

Load: neutral. Type "x", blur: red (interaction happened, value invalid). Complete to "x@y.com": green. The three states in order are exactly what error styling should do: silent → feedback → confirmation. Keep this pattern; it's production-ready as-is.

3 (medium): Reproduce the hidden-required bug. Form with two "steps" (two fieldsets, second one display:none via a style attribute), a required field in each, and a submit button. Try to submit. Read the Console. Explain what users would experience and give two fixes.
Expected + fixes

Submission blocks; Chrome logs "An invalid form control with name='...' is not focusable" — the browser wants to focus/bubble the hidden invalid field and can't; the user sees a dead submit button with zero feedback. Fixes: (1) toggle disabled on hidden steps' controls when hiding (disabled controls skip validation AND submission — re-enable when shown); (2) novalidate + per-step manual checkValidity. This bug ships constantly in wizard UIs — you've now debugged it in advance.

4 (medium): Draft-save button. Extend exercise 3: add a "Save draft" button that submits (to httpbin) even with invalid/empty fields, while the main "Submit order" button still validates. No JavaScript allowed.
Solution 
<button type="submit" formnovalidate formaction="/draft">Save draft</button>
<button type="submit">Submit order</button>

formnovalidate bypasses checking for that button only; bonus formaction sends drafts to a different endpoint (the form* attributes — formmethod, formaction, formnovalidate — let one form serve multiple submit intents). All native, zero JS.

5 (hard): The bypass demo. Take exercise 1's form. Bypass every constraint three ways: (a) delete attributes in the Elements panel and submit garbage, (b) submit via Console with form.submit() (note: it skips validation entirely — look it up), (c) replicate the request with fetch/curl, no form involved. Then write the three-sentence moral.
Moral + detail

(b) is the subtle one: the submit() method skips constraint validation AND the submit event by spec (requestSubmit() is the one that validates — next lesson). All three bypasses take under a minute. Moral: the browser enforces constraints only for cooperative users; the network accepts anything; only the server can refuse. Anyone who's done this exercise never again says "but the field was required".

6 (hard): Validation-UX review. Find a real signup form (any major site). Audit against this lesson: constraints visible before errors? Errors on load? Where do errors appear (bubble/native vs inline custom)? Can you save/back without validating? Composition password rules? Write a 5-bullet review with one recommended fix each.
What good findings look like

Typical: custom inline errors (novalidate under the hood), some hint text only in placeholders (violation), password composition rules still common (flag with the NIST argument), errors usually post-interaction (good), Back buttons occasionally trapped (flag formnovalidate). The exercise's real skill: you can now READ a form implementation from its behavior — that's what a senior review of someone else's checkout looks like.

🎯 Key Takeaways

  • Three layers: HTML constraints = instant guardrail, JS = enhanced UX, server = the only security. Never confuse the courtesy for the law.
  • required / min / max / minlength / maxlength / pattern / step — with the traps: group-required radios, silently-truncating maxlength, whole-value anchored pattern (+ title!), step-relative-to-min.
  • Invalid controls cancel submission before the submit event — the "handler never fires" and "hidden required field" bugs both live here.
  • novalidate (form-wide) and formnovalidate (per-button) are the escape hatches — drafts and Back buttons need them.
  • Style with :user-invalid/:user-valid, not bare :invalid — feedback, not accusation.
  • Passwords: length over composition; generous maxlength; never fight password managers.
  • form.submit() skips validation; requestSubmit() honors it — remember for the next lesson.
