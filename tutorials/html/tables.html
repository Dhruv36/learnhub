Tables Foundations

What you'll master: the full table anatomy (most developers know a third of it), header/data-cell relationships that make tables navigable by screen reader, row/column spanning, when a table is the right tool (and the layout-table sin of the past), and the responsive-table problem every real project hits. Prerequisites: lessons 1–4 (lesson 7's dl-vs-table rule helps too).

Part 1 — When a Table Is Correct

A table is for tabular data: multiple entities sharing the same properties, where the row–column intersection is what carries meaning. "What's the caffeine content of the Colombia roast?" — that question has a row (Colombia) and a column (caffeine); the answer lives at their crossing. If your content has that shape, a table is the honest markup. If not — if you just want things side by side — that's layout, and layout is CSS (grid/flexbox).

One entity, many properties      → <dl>          (last lesson)
Many entities, same properties   → <table>
Things visually side by side     → CSS layout      (never a table)

The history you need for interviews: before CSS matured (~1997–2005), entire page layouts were nested tables — headers, sidebars, everything. It worked visually and was a disaster otherwise: screen readers announced navigation as spreadsheet cells, markup was unmaintainable, and semantics were gibberish. "Don't use tables for layout" became the rallying cry of the web-standards movement. The inverse mistake is now common: developers so afraid of tables that they build actual data grids out of divs — equally wrong, in the other direction. Data → table. Layout → CSS. Both rules, no fear.

Part 2 — Full Anatomy (build it up with me)

Step 1: minimum table

<table>
  <tr><th>Roast</th><th>Origin</th><th>Price/kg</th></tr>
  <tr><td>Yirgacheffe</td><td>Ethiopia</td><td>$38</td></tr>
  <tr><td>Huila</td><td>Colombia</td><td>$29</td></tr>
</table>

tr = table row; th = header cell; td = data cell. Columns are implicit — HTML tables are written row by row, and cell N of each row forms column N.

Step 2: give it structure — thead, tbody, tfoot, caption

<table>
  <caption>Current roast lineup and pricing (July 2026)</caption>
  <thead>
    <tr><th scope="col">Roast</th><th scope="col">Origin</th><th scope="col">Price/kg</th></tr>
  </thead>
  <tbody>
    <tr><th scope="row">Yirgacheffe</th><td>Ethiopia</td><td>$38</td></tr>
    <tr><th scope="row">Huila</th><td>Colombia</td><td>$29</td></tr>
  </tbody>
  <tfoot>
    <tr><th scope="row">Average</th><td>—</td><td>$33.50</td></tr>
  </tfoot>
</table>

What each part buys you:

  • caption — the table's accessible name, announced before anything else ("what am I about to hear?") and visible context for everyone. First child of table, always worth writing. Alternative when design forbids a visible caption: aria-label on the table — but visible captions help everyone.
  • thead/tbody/tfoot — semantic regions. Payoffs: sticky headers (thead th { position: sticky; top: 0 }) during scroll, repeated headers when printing multi-page tables, styling hooks, and — subtle — the browser creates tbody in the DOM even if you don't write it (remember lesson 1: another reason table > tr CSS selectors mysteriously fail).
  • scope — the accessibility keystone; next section.
  • First-column ths: row headers — the entity each row is about. Real tables usually have both column headers and row headers.

Step 3: spanning

<tr><th colspan="2">Africa</th><td>$38</td></tr>   ← one cell covering 2 columns
<tr><th rowspan="3">Q3</th>...</tr>                 ← one cell covering 3 rows
<!-- The spanned-over cells are simply NOT WRITTEN in the other rows —
     count carefully; a miscounted span shifts every later cell sideways. -->

Spans express grouped/merged data (quarter groupings, category bands). Keep them minimal: every span makes screen-reader navigation and your own maintenance harder. Heavily-merged spreadsheet imports often signal the data wants restructuring (or splitting into several tables).

Part 3 — Why scope Matters (the screen-reader walkthrough)

A sighted user reading a big table constantly glances at the headers — that's what tracks "which column am I in?" A screen-reader user navigating cell by cell gets the same service only if the markup encodes the relationships. With scope set, moving to a cell announces its headers: "Huila, Price per kilogram, $29". Without it, the user hears "$29" — naked, contextless. In a 10-column table that's unusable.

  • scope="col" on every thead th; scope="row" on every row-header th. That covers ~95% of real tables.
  • Complex tables (multi-level headers via spans) graduate to headers/id: each td lists the ids of its header cells: <td headers="q3 africa price">. Powerful, verbose, error-prone — treat it as the escape hatch, and treat needing it as a hint to simplify the table.
  • th vs td is itself semantic: bolding a td does not make it a header. th carries the relationship machinery.

Part 4 — The Responsive-Table Problem

Tables have a minimum honest width — 8 columns don't fit a 375px phone. Every production team hits this; know the standard options and their tradeoffs:

StrategyHowTradeoff
Horizontal scroll (default choice)Wrap the table: .table-wrap { overflow-x: auto }Simple, honest, keeps table semantics; needs a visible scroll affordance
Column priorityHide low-value columns at breakpoints (CSS)Data loss on mobile; choose hidden columns deliberately
Cards transformCSS restructures each row into a stacked card at small widthsPopular but dangerous: display overrides can strip table semantics from AT — test with a screen reader
Separate mobile viewRender different markup for small screensMost work, most control — for data-critical products

The scroll wrapper is the right default: it preserves semantics perfectly and costs one div. (This site uses exactly that approach for its wide tables.) If you do the cards transform, re-adding role="table"/"row"/"cell" or using visually-restructured real tables needs an actual AT test — CSS display values on table parts have a history of destroying the accessibility tree.

Part 5 — Deep Dive: Tables and Data Integrity

  • Numbers align right, text aligns left (CSS text-align) — column scanning of figures depends on decimal alignment; every finance UI does this. Use font-variant-numeric: tabular-nums so digits share a fixed width.
  • Empty cells stay in the markup (<td></td> or with an explicit "—") — omitting a cell shifts everything after it left by one column: the data silently lies. This is the single most damaging table bug because nothing looks broken.
  • Sortable tables: sorting is JS behavior, but HTML participates — aria-sort="ascending" on the active th, and sort buttons inside the th (th content is interactive-allowed). The JS track builds one.
  • Very large tables: browsers layout tables expensively (auto layout reads all rows to size columns). table-layout: fixed + explicit column widths makes rendering O(visible) instead of O(all rows) — the classic fix when a 5,000-row report "freezes the tab". Beyond that: pagination or virtualization.

Interview Questions

Q1 (beginner): When should you use a table, and when absolutely not?

Use for tabular data — multiple entities sharing properties, where row×column intersections carry meaning. Never for visual layout (CSS grid/flexbox's job) — layout tables read as data to AT and are unmaintainable. Complete answer includes the modern inverse mistake: div-built data grids are just as wrong; fear of tables shouldn't extend to actual data.

Q2 (beginner): What do thead, tbody, tfoot, and caption each contribute?

caption: the table's accessible name, announced first, visible context. thead/tbody/tfoot: semantic regions enabling sticky headers, printed header repetition per page, and clean styling. Bonus point: browsers insert tbody implicitly even when unwritten — table > tr selectors fail because rows are actually inside a generated tbody.

Q3 (mid): Explain scope, and what a screen-reader user experiences with and without it.

scope declares which cells a th governs: col (its column) or row (its row). With it, navigating to any cell announces its headers first — "Huila, Price, $29" — replicating the sighted glance at the headers. Without it, cells announce bare values, making wide tables unusable. Complex merged-header tables escalate to headers/id association; needing that is often a design smell.

Q4 (mid): An 8-column table must work on phones. Compare your options.

(1) overflow-x scroll wrapper — one div, semantics intact, the right default. (2) Column priority hiding — deliberate data loss, fine for low-value columns. (3) Row→card CSS transform — nice UX but display overrides can strip table semantics from the accessibility tree; requires AT testing or ARIA table roles re-added. (4) Distinct mobile markup — max control, max cost. Senior signal: naming the accessibility-tree hazard in option 3 unprompted.

Q5 (senior): A 5,000-row report table freezes the tab on render. Diagnose and fix, staying in the browser.

Cause: table auto-layout is O(all content) — the engine measures every row to compute column widths before painting anything. Fixes in order: (1) table-layout: fixed with explicit column widths — sizing no longer depends on content; render cost drops massively. (2) Paginate or virtualize rows (render the visible window; libraries or manual) — but note virtualization vs semantics tension: windowed rows break AT row counts; mitigate with aria-rowcount/aria-rowindex. (3) contain: strict / content-visibility: auto on row groups for off-screen skip. (4) If sorting/filtering runs in JS, move to the server for datasets this size. Measuring first (Performance panel: where's the time — layout vs script?) is the expected preamble.

Q6 (senior): Your CMS authors keep producing tables that fail audits (no caption/scope, merged chaos). Fix the system, not the tables.

Root cause: the authoring tool permits (or encourages) inaccessible output. Interventions: (1) constrain the editor — table insertion wizard that requires a caption and generates thead/th+scope automatically; disable arbitrary merging or gate it behind a warning. (2) Validate in the pipeline — CI/lint on rendered output (axe rules for tables) blocking publish, with author-readable errors. (3) Transform legacy content in one migration pass (scriptable: first row → thead th scope=col, first column heuristics need review). (4) Teach with defaults: templates and docs showing the golden pattern. The senior shape: make the correct thing the path of least resistance and the wrong thing impossible, rather than reviewing tables forever.

🏋️ Exercises

1 (easy): Full-anatomy table. Build the roast-lineup table from Part 2 with all parts: caption, thead with scoped column headers, tbody with row headers, tfoot average row. Validate it, then hover-check in the Elements panel that your structure survived parsing intact.
Self-check

Reuse Part 2's step-2 code as the target — but type it. Checks: caption is the FIRST child; every th has a scope; the tfoot th also carries scope="row". In the Elements panel confirm no surprise tbody wrapping or relocated rows.

2 (easy): dl or table? For each dataset choose dl, table, or CSS layout: (a) one product's specs, (b) 5 pricing plans × 8 features, (c) a hero image beside intro text, (d) office hours for one location, (e) office hours for 12 locations.
Answers

(a) dl (b) table — many entities × same properties (c) CSS layout — that's presentation, no data relationships (d) dl (day → hours pairs) or a small table; dl is lighter (e) table — 12 entities sharing the same 7 properties. The pivot between (d) and (e) — same data type, different entity count — is the entire decision rule in miniature.

3 (medium): Spanning practice. Build a quarterly sales table: columns Month / Region / Revenue; rows grouped so "Q3" is one rowspan="3" cell beside July, August, September. Add a colspan'd tfoot total. Deliberately miscount one span, observe the wreckage, then fix it.
Solution core 
<tbody>
  <tr><th rowspan="3" scope="rowgroup">Q3</th>
      <th scope="row">July</th><td>$41,200</td></tr>
  <tr><th scope="row">August</th><td>$44,800</td></tr>
  <tr><th scope="row">September</th><td>$47,100</td></tr>
</tbody>
<tfoot>
  <tr><th colspan="2" scope="row">Q3 total</th><td>$133,100</td></tr>
</tfoot>

Rows 2–3 write only TWO cells — the rowspan occupies their first slot. Note scope="rowgroup" on the Q3 cell: it heads the whole group. The miscount experiment (rowspan="2") shows September's cells sliding left — the silent-shift failure you now recognize on sight.

4 (medium): Responsive wrapper + sticky header. Give your exercise-1 table 8 columns of data, wrap it in <div class="table-wrap" style="overflow-x:auto">, and make the header sticky (thead th { position: sticky; top: 0; background: white }). Test at 375px width and with 50 rows. Note the one caveat you observe.
Expected observations

Horizontal scroll works at 375px with semantics untouched. Sticky header holds while vertically scrolling — but note: position:sticky on thead works against the page or wrapper scroll depending on which container scrolls; inside an overflow wrapper with its own vertical scrolling you may need the sticky on th (as given) and a set wrapper height. The background on th matters — without it, rows show through the "transparent" sticky header. Both caveats are real-project rites of passage.

5 (hard): Screen-reader run. With NVDA (Win) or VoiceOver (Mac), navigate exercise 3's table using table-navigation commands (NVDA: Ctrl+Alt+arrows). Document what's announced at three cells with scope in place; then strip all scopes/ths to tds and repeat. Two paragraphs on the difference.
What you should hear

With semantics: entering a cell announces column header, row header(s) including the rowgroup ("Q3, August, Revenue, $44,800"), plus row/col position. Without: bare values and coordinates — you have to memorize the grid to know what "$44,800" is. Your write-up should land on: the entire difference was th + scope; the visual rendering never changed. Semantics are invisible to sighted users and everything to AT users — the thesis of this whole track, experienced firsthand.

6 (hard): Table auditor. Console script: for every table on a page report — has caption? has thead? th count; any th missing scope; and any row whose effective cell count (accounting for colspans) differs from the header row's. Run it on a data-heavy site (sports standings, financial data) and summarize findings.
Reference implementation 
document.querySelectorAll('table').forEach((t, i) => {
  const width = r => [...r.cells].reduce((n, c) => n + c.colSpan, 0);
  const rows = [...t.rows];
  const head = rows[0] ? width(rows[0]) : 0;
  const report = {
    table: i,
    caption: !!t.caption,
    thead: !!t.tHead,
    ths: t.querySelectorAll('th').length,
    thsNoScope: [...t.querySelectorAll('th')].filter(h => !h.scope).length,
    raggedRows: rows.filter(r => width(r) !== head).length, // rowspans overcount; treat as leads
  };
  console.table([report]);
});

Note the honest limitation flagged in the comment: rowspan carry-over makes naive width checking report false positives — a correct checker must track spans across rows (good extension challenge). Even this simple version finds missing captions and scopeless ths everywhere, including on famous sites.

🎯 Key Takeaways

  • Tables are for data whose meaning lives at row×column intersections — never for layout, and divs are never for actual data grids.
  • Full anatomy: caption (accessible name, first child) + thead/tbody/tfoot + th with scope="col"/"row" — that's what makes cells announce their context.
  • Spanned-over cells are not written; miscounted spans and omitted empty cells shift data silently — the worst table bugs look fine.
  • Responsive default: an overflow-x wrapper (semantics intact); CSS display transforms on table parts risk stripping the accessibility tree.
  • Data-integrity kit: right-aligned tabular numerals, explicit empty cells, aria-sort for sortables, table-layout:fixed for big tables.
  • th is semantics, bold is styling — one creates relationships, the other pixels.
