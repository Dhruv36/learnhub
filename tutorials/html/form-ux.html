Form UX & Accessibility Forms Mastery

What you'll master: the capstone of the forms section — autocomplete tokens (the highest-ROI attribute in commerce), hint/error association with aria-describedby, error summaries, autofocus/tab order discipline, spellcheck/autocapitalize/enterkeyhint refinements, and a complete audited checkout form assembling all six lessons. Prerequisites: lessons 9–13.

Part 1 — autocomplete: the Money Attribute

Browsers and password managers can fill entire forms in one tap — if you tell them what each field is. The autocomplete attribute takes standardized tokens:

<input name="fname"  autocomplete="given-name">
<input name="lname"  autocomplete="family-name">
<input name="email"  autocomplete="email">
<input name="tel"    autocomplete="tel">
<input name="addr1"  autocomplete="address-line1">
<input name="city"   autocomplete="address-level2">
<input name="zip"    autocomplete="postal-code">
<input name="cc"     autocomplete="cc-number">
<input name="exp"    autocomplete="cc-exp">
<input name="pw"     autocomplete="new-password">      <!-- signup: offers generation -->
<input name="pw"     autocomplete="current-password">  <!-- login: fills the saved one -->
<input name="otp"    autocomplete="one-time-code">     <!-- SMS codes auto-fill on mobile -->

<!-- Scoping tokens compose: -->
<input autocomplete="shipping address-line1">
<input autocomplete="billing address-line1">

Why it's "the money attribute": address forms with correct tokens fill in one tap — measured checkout-completion gains are real and large (Google's studies showed ~30% faster completion). It's also an accessibility requirement: WCAG 1.3.5 (AA) requires identifying input purpose — motor- and cognitive-impaired users depend on autofill. Two rules: the tokens are a fixed vocabulary (look them up; don't invent), and autocomplete="off" mostly doesn't work (browsers ignore it for address/login data — they consider autofill user-serving; stop fighting it).

Part 2 — Hints and Errors That Screen Readers Actually Read

The single pattern that fixes most form accessibility, formalizing what the API lesson used:

<label for="pw">Password</label>
<input id="pw" name="pw" type="password" required minlength="12"
       autocomplete="new-password"
       aria-describedby="pw-hint pw-err">     <!-- SPACE-SEPARATED id list! -->
<p class="hint"  id="pw-hint">12+ characters. Passphrases welcome.</p>
<p class="error" id="pw-err"></p>

How AT consumes this: on focus, the screen reader announces label → role → state → described-by text: "Password, edit, required, 12+ characters, passphrases welcome." The hint travels with the field at the moment of need. When your JS writes an error into #pw-err, it's automatically part of the field's description on next focus; adding aria-invalid="true" announces the state change. Without describedby, hints are proximity-only — visible to eyes, invisible to AT.

Error summaries (long forms)

<div role="alert" tabindex="-1" id="summary" hidden>
  <h2>3 fields need attention</h2>
  <ul>
    <li><a href="#em">Email — doesn't look like an address</a></li>
    ...
  </ul>
</div>

On failed submit: fill it, unhide it, summary.focus(). role="alert" announces immediately; the links jump to each field (fragment navigation — lesson 5 — meets forms). This is the GOV.UK-standardized pattern, the most user-tested form pattern in existence; when in doubt about form UX, read the GOV.UK Design System — it's public and evidence-based.

Part 3 — Focus and Keyboard Discipline

  • autofocus — focuses a field at load. Use on single-purpose pages only (login, search). On content pages it scrolls to the form uninvited and screen-reader users skip everything before it. One per page; treat with suspicion.
  • Tab order = DOM order. Keep it that way. tabindex="1+" (jumping the queue) is a maintenance disaster and confuses everyone — the fix for wrong tab order is reordering the DOM (CSS can rearrange visuals independently… carefully — visual order diverging from focus order is itself a WCAG issue). The only tabindex values you should ship: 0 (make focusable, in order) and -1 (focusable by script only — the error summary above).
  • Never remove focus outlines without replacement. :focus-visible styles keyboard focus without flashing mouse users — the modern compromise. An invisible focus ring strands sighted keyboard users completely.
  • Enter submits (lesson 9) and Esc should close whatever your JS opened. Test every form with the mouse unplugged — five minutes, most revealing audit you can run.

Part 4 — The Refinement Attributes

<input spellcheck="false">          codes, usernames, emails — red squiggles are noise
<input autocapitalize="none">       usernames/emails on mobile (default capitalizes sentences)
<input enterkeyhint="search">       mobile Enter key label: go|done|next|search|send
<input inputmode="decimal">         keyboard choice (input-types lesson)
<form  autocomplete="off">          discouraged-but-known: fights the browser, mostly loses
<input readonly> vs <input disabled>  submits vs excluded (lesson 9 — keep them straight)

Small individually; together they're the difference between a form that feels engineered and one that fights the user's phone. A useful bundle for identifier fields: spellcheck="false" autocapitalize="none" autocorrect="off" (the last is Safari-specific but harmless elsewhere).

Part 5 — Structure and Copy Rules (evidence-based, briefly)

  • One column. Multi-column forms cause skipped fields and Z-pattern confusion in eye-tracking studies, every time. (Related short fields — City/ZIP — may share a row.)
  • Labels above fields, not beside: fastest completion in testing, and mobile agrees by necessity.
  • Mark the exceptions, minimize fields: if most fields are required, mark the optional ones ("(optional)") rather than starring everything; every removed field measurably raises completion. Ask for data when you need it, not at signup.
  • Buttons say what they do: "Create account", "Pay $43.50" — never "Submit". The final button is a promise; specificity builds trust at the scariest click.
  • Don't split what users think of as one thing: single name field beats first/last when you don't truly need the split (many names don't fit Western first/last); single-field phone beats three boxes; and never auto-advance between fields — it breaks backspace, paste, and expectations.

Part 6 — Capstone: the Audited Checkout Form

Everything from six lessons in one artifact — abbreviated here to the load-bearing lines; build the full version in exercise 6:

<form action="/checkout" method="post" novalidate>
  <div role="alert" tabindex="-1" id="summary" hidden></div>

  <fieldset>
    <legend>Contact</legend>
    <label for="em">Email</label>
    <input id="em" name="email" type="email" required
           autocomplete="email" autocapitalize="none" spellcheck="false"
           aria-describedby="em-hint em-err">
    <p class="hint" id="em-hint">Receipt and delivery updates go here.</p>
    <p class="error" id="em-err"></p>
  </fieldset>

  <fieldset>
    <legend>Shipping address</legend>
    <label for="name">Full name</label>
    <input id="name" name="name" required autocomplete="shipping name">
    <label for="addr">Address</label>
    <input id="addr" name="addr" required autocomplete="shipping address-line1">
    <label for="city">City</label>
    <input id="city" name="city" required autocomplete="shipping address-level2">
    <label for="zip">ZIP</label>
    <input id="zip" name="zip" required autocomplete="shipping postal-code"
           inputmode="numeric" pattern="[0-9]{5}(-[0-9]{4})?"
           title="12345 or 12345-6789">
  </fieldset>

  <fieldset>
    <legend>Payment</legend>
    <label for="cc">Card number</label>
    <input id="cc" name="cc" required autocomplete="cc-number"
           inputmode="numeric" pattern="[0-9 ]{12,23}" spellcheck="false">
    ...cc-exp, cc-csc following the same recipe...
  </fieldset>

  <label><input type="checkbox" name="save" value="1"> Save my details
         <span class="hint">(optional — you can delete them anytime)</span></label>

  <button type="submit">Pay $43.50</button>
</form>

Audit trail per line: novalidate + summary div (API lesson's custom UI), legends grouping announcements, every field labeled + autocompleted + hint-associated, digit strings as text/inputmode (input-types lesson), the pattern+title pair (validation lesson), optional marked not required-starred, button naming the exact promise.

Interview Questions

Q1 (beginner): What does autocomplete="email" do that name="email" doesn't?

name is your server's key — private vocabulary, browsers can only guess from it. autocomplete is the standardized public vocabulary browsers/password managers act on: guaranteed correct autofill mapping. Together: name for the wire, autocomplete for the fill. Bonus: it's also WCAG 1.3.5 — identifying input purpose is an accessibility requirement, not just convenience.

Q2 (beginner): How does a screen-reader user learn a field's requirements ("12+ characters") with your markup?

The requirement lives in a visible hint element whose id appears in the input's aria-describedby; on focus, AT announces label, role, state, then the description. Visible-but-unassociated hints are announced never. The same mechanism carries errors — write the error into an associated node and set aria-invalid.

Q3 (mid): Why is autocomplete="off" largely ignored by browsers, and what should teams do instead of fighting it?

Browsers concluded autofill of addresses/credentials serves users (speed, password-manager adoption — which improves security) and that sites disabled it mostly for anti-patterns, so they ignore off for those categories. Instead: if the concern is a shared-machine kiosk, that's a deployment/browser-profile problem; if it's "wrong suggestions", fix your tokens (mislabeled fields cause bad fills); for genuinely non-identity fields (search boxes with custom typeahead), off still mostly works; one-time codes get one-time-code. Fighting the browser with random name attributes/readonly hacks breaks WCAG 1.3.5 and loses anyway.

Q4 (mid): Describe the GOV.UK error-summary pattern and why each piece exists.

On failed submit: a container at the top with role="alert" (immediate announcement), tabindex="-1" + script focus (screen-reader and visual attention move there), a heading with the error count (orientation), and a link per error targeting the field's id (fragment navigation to each fix). Fields simultaneously show inline associated errors. Exists because long forms otherwise leave users hunting for scattered red text — and it's the most usability-tested form pattern published. Mentioning field-links doubling as a to-do list is the practitioner detail.

Q5 (senior): Design review: a fintech signup uses 6 separate OTP boxes that auto-advance, autofocus on load, and tabindex numbering. Critique and redesign.

Critique: six boxes break paste (the #1 OTP action!), backspace navigation, password-manager/SMS auto-fill, and AT announcement (six unlabeled edits); auto-advance steals control mid-correction; tabindex numbers guarantee maintenance breakage; autofocus is defensible ONLY because the page is single-purpose. Redesign: one input, autocomplete="one-time-code" (iOS/Android auto-fill from SMS), inputmode="numeric", pattern="[0-9]{6}", maxlength="6", spellcheck="false", autocapitalize="none"; label + associated hint ("6-digit code from your text message"); DOM-order tabbing; keep autofocus here. The single-field version is less code AND better on every axis — the design-system lesson that fancy interaction ≠ better interaction.

Q6 (senior): You're asked to raise checkout completion by 15%. Which HTML-layer changes have measured impact, and how would you verify?

Ranked by evidence: (1) correct autocomplete tokens end-to-end (address+payment autofill — largest single lever); (2) field-count reduction (each removed field helps; guest checkout, defer account creation); (3) input types/inputmode right (mobile keyboards); (4) inline validation with kind timing (blur-not-keystroke errors, clear-on-input); (5) one column, labels above, specific pay button with amount; (6) error summary preventing abandonment at failure. Verify: funnel analytics per field (focus→completion drop-off identifies killer fields), A/B where traffic allows, session replays for rage-signals, and Lighthouse/axe as regression gates. The senior frame: the HTML layer is a measurable conversion surface, not plumbing — and every improvement listed is also an accessibility improvement, which is not a coincidence.

🏋️ Exercises

1 (easy): Autofill test-drive. Build a name/email/address form twice: once with no autocomplete attributes, once fully tokened. Fill each with your browser's autofill (add a profile in browser settings if needed). Time both. Note which fields the tokenless version got wrong or skipped.
Expected

The tokened form fills completely in one interaction; the tokenless one partially fills based on name-attribute guessing — typically missing address-line2/level fields or confusing city/state. The delta you just timed is the conversion argument in miniature. (Chrome: chrome://settings/addresses to set up a test profile.)

2 (easy): describedby verification. Add hints to three fields — one properly associated, one merely adjacent, one associated to TWO ids (hint + error). Focus each with a screen reader (or inspect the Accessibility panel in DevTools → the computed Description). Record the differences.
What to observe

DevTools → Elements → Accessibility pane shows the computed accessible Description: populated for the associated fields (both ids' text concatenated for the double one), EMPTY for the adjacent-only field despite looking identical on screen. That empty pane is the invisible failure this lesson exists to prevent — and now you know where to check without a screen reader.

3 (medium): Keyboard-only gauntlet. Unplug the mouse (really). Complete your capstone-in-progress form start to finish: tab everywhere, operate radios with arrows, submit with Enter, reach and use the error summary. Log every point of friction and fix each with lesson techniques.
Common findings

Typical catches: focus invisible on custom-styled fields (add :focus-visible rules), radios trap confusion (arrows move within group, Tab exits — that's correct, learn it), error summary unreachable (missing tabindex="-1" + focus()), and a link styled as a button not responding to Space (it's a link — lesson 5's contract). A form that survives this gauntlet is accessible to keyboard users by construction, not hope.

4 (medium): The refinement pass. Take an email + username + amount form and apply every relevant refinement: spellcheck, autocapitalize, inputmode, enterkeyhint, correct autocomplete. Test on a real phone and document each attribute's visible effect.
Expected effects

Email: no auto-capital first letter (autocapitalize="none"), no red squiggles (spellcheck="false"), @ keyboard (type). Username: same trio. Amount: decimal keypad (inputmode="decimal"), Enter key reading "done"/"go" per enterkeyhint. None of these are visible in desktop DevTools — the exercise's meta-lesson repeats: mobile refinements need mobile devices.

5 (hard): Error summary, wired. Implement the full GOV.UK pattern on a 5-field form: on failed submit, populate the summary (count + linked errors), unhide, focus it; links jump to fields; fixing a field updates both its inline error AND removes its summary line; summary hides when empty. Screen-reader (or Accessibility-panel) verify the alert announces.
Implementation keys

Build summary items from the same showError pass (API lesson) — one validation sweep feeding two renderings (inline + summary) keeps them synchronized by construction. Focus order: summary first (role=alert announces), user clicks a link → field focuses (href="#field-id" works natively). The subtle bit: re-render the summary on input events too, so it never lists fixed fields — stale summaries teach users to ignore it.

6 (hard): The capstone, complete and audited. Build Part 6's checkout fully (all three fieldsets, every attribute), then run the complete audit battery: axe DevTools (0 violations), Lighthouse accessibility (100), keyboard gauntlet, autofill one-tap test, httpbin payload verification, and JS-disabled submission. Fix until all six pass. This artifact is your forms-mastery certificate — keep it as a reference for every form you build after.
Audit expectations

axe typically catches a missed describedby or contrast issue on hint text (fix: darker gray). Lighthouse 100 requires labels+names complete. The JS-disabled run must still validate natively — which means novalidate should be ADDED by your JS, not hardcoded (one line: form.setAttribute('novalidate','') on script load — the elegant progressive-enhancement trick that gives no-JS users native validation and JS users your custom UI). If you caught that yourself before reading this: you've mastered the section.

🎯 Key Takeaways

  • autocomplete tokens are conversion, accessibility (WCAG 1.3.5), and security at once — standard vocabulary on every identity/address/payment field; "off" mostly loses.
  • aria-describedby (space-separated ids) is how hints and errors actually reach AT; aria-invalid + role="alert" summaries complete the error system (GOV.UK pattern).
  • Tab order = DOM order; tabindex ships only as 0 or -1; focus stays visible via :focus-visible; test with the mouse unplugged.
  • Refinements matter in aggregate: spellcheck/autocapitalize off for identifiers, enterkeyhint labels, inputmode keyboards — testable only on devices.
  • Structure by evidence: one column, labels above, optional-marked, fewest fields, buttons that state the exact promise ("Pay $43.50").
  • Add novalidate via JS, not markup — no-JS users keep native validation; JS users get your upgraded UI. Progressive enhancement to the last line.
