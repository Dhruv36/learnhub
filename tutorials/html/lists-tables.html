Lists & Tables Basics

Lists structure everything from nav menus to checkout steps; tables are the only correct container for two-dimensional data. Both announce their shape to assistive tech ("list, 5 items"; "table, 4 columns") — structure users literally hear.

Unordered & Ordered Lists

<ul>
  <li>No inherent order — bullets</li>
  <li>Direct children MUST be <li> (divs inside ul = invalid)</li>
</ul>

<ol start="5" reversed type="a">
  <li value="10">Ordered — sequence matters</li>   <!-- value overrides numbering -->
</ol>
Attribute (ol)Effect
start="5"Begin numbering at 5 (continuing a split list)
reversedCount down — top-10 lists
type="1|a|A|i|I"Numbering style (CSS list-style-type is preferred)

Nesting — the One Legal Shape

❌ <ul>
  <li>Frontend</li>
  <ul>               ← ul as direct
    <li>React</li>    child of ul: invalid
  </ul>
</ul>
✅ <ul>
  <li>Frontend
    <ul>             ← sublist INSIDE
      <li>React</li>   the parent li
    </ul>
  </li>
</ul>

This exact pattern — nav > ul > li > a (+ nested ul for submenus) — is how virtually every production navigation menu is built, styled entirely by CSS.

Description Lists — Key/Value Pairs

<dl>
  <dt>QPS</dt>      <dd>Queries per second</dd>
  <dt>SLA</dt>      <dd>Service level agreement</dd>
  <dt>TTL</dt>      <dd>Time to live</dd><dd>Also: a Redis expiry setting</dd>
</dl>   <!-- one dt may have several dd; glossaries, FAQs, metadata panels, spec sheets -->

Tables — Data, Never Layout

<table>
  <caption>Q3 revenue by region (USD)</caption>
  <colgroup><col><col span="2" class="numeric"></colgroup>
  <thead>
    <tr><th scope="col">Region</th><th scope="col">Revenue</th><th scope="col">Growth</th></tr>
  </thead>
  <tbody>
    <tr><th scope="row">EMEA</th><td>$4.2M</td><td>+12%</td></tr>
    <tr><th scope="row">APAC</th><td>$3.1M</td><td>+28%</td></tr>
    <tr><th scope="row">Americas</th><td>$5.6M</td><td>+9%</td></tr>
  </tbody>
  <tfoot>
    <tr><th scope="row">Total</th><td>$12.9M</td><td>+15%</td></tr>
  </tfoot>
</table>
Element/attrDeep detail
<caption>The table's announced title — first thing screen-reader users hear; also visible context for everyone. First child of table.
scope="col|row"Binds headers to cells: the cell "$3.1M" is announced as "APAC, Revenue, $3.1M". Without scope, users navigating cell-by-cell hear naked numbers.
headers + idFor complex multi-level headers where scope can't express the relationship
thead/tbody/tfootSemantic bands; enable sticky headers (CSS) and repeated headers on printed pages
colspan/rowspanMerge cells: <th colspan="2">. Keep rare — every merge makes screen-reader navigation harder.
<colgroup>/<col>Style whole columns (width, background) without touching every cell

<div style="overflow-x:auto"><table>…</table></div>

Why Not Tables for Layout? (the history question)

Pre-CSS (1990s), entire sites were nested tables. The costs: screen readers announce layout tables as data ("table, 3 columns" for a sidebar), source order gets scrambled, mobile rendering breaks, and markup quadruples. CSS Grid does everything layout-tables did, correctly. If you inherit legacy layout tables: role="presentation" mutes their table semantics as a stopgap.

Interview Questions

Q1: How do you make a data table accessible?

caption for the title, th with scope=col/row for all headers, thead/tbody structure, avoid merged cells where possible (headers+id when unavoidable), and never use a table for layout. Test by navigating cell-by-cell with a screen reader — each cell should announce its headers.

Q2: ul vs ol vs dl — how do you choose?

Order meaningful → ol (steps, rankings). Order irrelevant → ul (features, nav). Name/value or term/definition pairs → dl (glossaries, spec sheets, FAQ). The test: would shuffling the items change the meaning?

Q3: A table has 10 columns and must work on a 360px phone. Options?

In order of preference: (1) rethink — does mobile need all 10 columns, or a summary + detail view? (2) horizontal scroll container with a sticky first column; (3) collapse to stacked cards via CSS + data-labels; (4) column-priority hiding. State the trade-off: scrolling preserves comparisons, stacking preserves readability.

🏋️ Practice:
1. Build a 3-level nested nav list (categories → subcategories → items) with valid nesting throughout.
2. Recreate the revenue table above from scratch, then verify every cell announces correctly by inspecting with the Accessibility panel in DevTools (Elements → Accessibility).
3. Wrap it in an overflow container and shrink the window — the page itself should never scroll horizontally.

🎯 Key Takeaways

  • Lists: only li as direct children; sublists nest INSIDE an li; nav = ul by convention.
  • dl is the forgotten gem for key/value content — stop building glossaries out of divs.
  • Tables need caption + scoped headers to be heard correctly, and an overflow wrapper to survive mobile.
  • Data → table. Layout → CSS Grid. No exceptions in new code.
