Accessibility (ARIA) Advanced

~1.3 billion people live with significant disability (WHO); accessibility lawsuits run in the thousands per year in the US alone; the EU Accessibility Act now applies to private-sector apps. And beyond compliance: accessible HTML is more robust, more testable, more SEO-friendly HTML. Interviews probe one thing above all — knowing when you don't need ARIA.

How Assistive Tech Sees Your Page

DOM  ──▶  Accessibility Tree  ──▶  screen reader / voice control / switch device
Each node exposes: ROLE (button), NAME ("Save draft"), STATE (disabled, expanded)
Semantic HTML fills all three automatically. Divs expose: nothing.

DevTools → Elements → Accessibility panel shows the tree per element — check it whenever unsure what a widget "sounds like".

Rule #1 of ARIA: Prefer Native HTML

❌ <div role="button" tabindex="0"
     aria-pressed="false"
     onclick="…" onkeydown="if(
       e.key===' '||e.key==='Enter')…">
  Mute
</div>
Rebuilds ~half of what button does,
and still misses form/disabled/focus
behaviors.
✅ <button type="button"
        aria-pressed="false">
  Mute
</button>
Role, name, keyboard, focus: free.
The ONLY aria needed is the state
that native HTML can't express
(toggle pressed-ness).

The five rules of ARIA, condensed: use native elements first; don't change native semantics; all interactive ARIA needs keyboard support you write yourself; don't hide focusable elements; interactive elements need accessible names. No ARIA is better than bad ARIA — a wrong role misleads users worse than no role.

Live regions deserve emphasis: content that appears without a page load (search results, toasts, cart updates) is invisible to screen readers unless a live region announces it. The region must exist in the DOM before the update lands in it.

Keyboard Navigation — Where Most Apps Fail

Tab / Shift+Tab   move between interactive elements (source order = tab order)
Enter             activate links & buttons          Space   buttons, checkboxes, scroll
Arrows            WITHIN widgets: radio groups, selects, tabs, menus
Escape            close the topmost modal/menu
tabindexMeaning
0Join the natural tab order (custom widgets)
-1Programmatically focusable only — focus targets like modal headings
≥ 1Override the order — always a bug; fix the source order instead

The three failures to hunt: focus lost after closing a modal (return it to the trigger), focus invisible (never outline:none without a :focus-visible replacement), focus trapped in an embedded widget.

Interview Questions

Q1: When should you reach for ARIA?

Only when HTML can't express the semantics: composite widgets (tabs, combobox, tree), state (expanded/pressed/selected), live announcements, and naming for icon-only controls. If a native element exists — button, details, dialog, input type — it wins, because ARIA adds only announcements, never behavior: every keyboard interaction still has to be hand-written.

Q2: A React SPA changes routes — screen reader says nothing. Fix?

Client-side routing doesn't reload the page, so nothing is announced. Fixes: move focus to the new page's h1 (tabindex="-1" + focus()), and/or announce the new title in an aria-live region. Router libraries have this built in — knowing WHY it's needed is the interview point.

Q3: Why is tabindex="3" a bug?

Positive values create a parallel tab order that runs before every tabindex-0 element in the document, in a maintenance-hostile way. The correct fix for wrong tab order is correcting DOM source order (which also fixes reading order).

1. Take yesterday's signup form and complete this checklist: labels, fieldset/legend, :focus-visible styling, error text tied via aria-describedby, role="alert" on the error summary.
2. Navigate this very page with only the keyboard — count anything you can't reach or see focused.
3. Run Lighthouse's a11y audit on your Playground creation and fix every flag.

🎯 Key Takeaways

  • Accessibility = role + name + state reaching the accessibility tree; native HTML supplies all three.
  • ARIA is a last resort that only changes announcements — the keyboard work is always yours.
  • Tab/Enter/Space/Arrows/Escape must all work; focus must be visible, returned, and never trapped.
  • Dynamic updates need live regions; SPAs must manage focus on route change.
  • Automated tools catch a third of issues — the keyboard pass and screen-reader spot-check are mandatory.
