Accessibility Fundamentals Accessibility

What you'll master: the accessibility tree (the actual thing screen readers read — not the DOM, not the visual page), WCAG's POUR framework as an organizing lens for everything you've learned so far, the disability categories and how each interacts with the web differently, the legal/business reality, and the testing toolkit you'll use for the rest of your career. This lesson names and organizes the accessibility knowledge distributed across lessons 1–21 into one coherent model, then opens two dedicated deep-dive lessons (ARIA, Keyboard & Focus). Prerequisites: the whole track so far — this is intentionally a synthesis lesson.

Part 1 — The Accessibility Tree (the missing piece)

You've built a mental model of the DOM (lesson 1) and of CSSOM+layout (implicitly, throughout). There's a third tree, and it's the one that actually matters for this entire lesson:

DOM tree           →  what your HTML parsed into (lesson 1)
Render tree/layout →  what gets painted on screen (CSS track)
ACCESSIBILITY TREE →  what screen readers, voice control, and switch devices actually perceive

The browser derives the accessibility tree from the DOM automatically, computing for every node: a role ("what kind of thing is this" — button, heading level 2, list with 5 items), a name ("what do I call it" — from a label, aria-label, alt, or text content, following a defined priority order), and a state ("what's its current condition" — checked, expanded, disabled, invalid). This is precisely the "role + name + state" triad you've seen surface repeatedly since lesson 5 (link + destination text), lesson 8 (th scope + cell content), lesson 14 (aria-invalid + described error) — every one of those was you shaping this tree without yet having its name. Everything in this lesson and the next two is about controlling this tree correctly — semantic HTML gets it right automatically; ARIA patches it when semantics genuinely can't reach; and layout tricks that only change the render tree (visual CSS) never touch it at all, which is exactly why "it looks right" and "it's accessible" are independent claims.

See it yourself: DevTools → Elements panel → the "Accessibility" side-tab (Chrome/ Edge) shows the computed role, name, and properties for any selected node — this is your primary tool for the rest of this section, more useful day-to-day than actually running a screen reader for most quick checks.

Part 2 — POUR: the Four Pillars (WCAG's organizing framework)

WCAG (Web Content Accessibility Guidelines) organizes every requirement under four principles — useful less as a checklist and more as a lens for categorizing any accessibility problem you encounter:

PrincipleMeansYou've already met it in…
PerceivableContent must be presentable in more than one sensealt text (lesson 6), captions (lesson 19), color-not-alone (lesson 5), contrast (this lesson)
OperableAll functionality works via keyboard, with enough time, without seizure triggersNative interactive elements (lesson 16), focus order (lesson 14), no keyboard traps (next lesson)
UnderstandableContent and operation are predictable and clearlang attribute (lesson 2), form labels/hints (lesson 14), consistent navigation (lesson 15)
RobustWorks across current and future assistive technologyValid markup (lesson 1), correct roles from real elements (lesson 16), tested ARIA (next lesson)

Notice the pattern: almost nothing here is new information — POUR is a filing system for accessibility knowledge you've been accumulating since lesson 1. When you hit a genuinely new accessibility problem in the future, ask "which pillar is this?" and you'll usually already know the relevant technique from the pillar's established patterns.

Conformance levels (A / AA / AAA) matter practically: A is baseline (missing it is a severe, often legally actionable gap — e.g., captions, keyboard operability); AA is the real-world target — what the EU's European Accessibility Act (2025+) and most organizations' policies require, and what this whole track has been building toward; AAA is aspirational/contextual (some AAA criteria are genuinely impractical for certain content types, and WCAG itself doesn't expect full AAA site-wide).

Part 3 — Who You're Actually Building For

"Accessibility" collapses a wide range of distinct interaction needs into one word — worth unpacking because each category breaks the web differently, and the fixes are different too:

  • Blind users — screen readers (NVDA, JAWS, VoiceOver) read the accessibility tree aloud, navigating by headings/landmarks/links-list (lessons 3, 5, 15) rather than visually scanning. Zero use of a mouse; everything must be reachable and operable via keyboard/AT commands.
  • Low-vision users — screen magnifiers (200-400%+ zoom), high-contrast modes, custom stylesheets. Breaks: fixed-pixel layouts that don't reflow, text that becomes unreadable at zoom, content that disappears behind fixed-position headers when magnified.
  • Deaf/hard-of-hearing users — need captions (lesson 19) and transcripts; audio- only cues (a "success" sound with no visual equivalent) are invisible to this group entirely.
  • Motor-impairment users — keyboard-only, switch devices (a single button scanning through options), voice control (Dragon, Voice Access — which "click X" by reading the accessible NAME aloud, another reason correct naming matters beyond screen readers). Breaks: tiny click targets, hover-only reveals (lesson 15's nav submenu trap), time-limited interactions, anything requiring precise mouse movement.
  • Cognitive/learning differences — ADHD, dyslexia, autism spectrum, memory conditions. Breaks: inconsistent navigation, unclear error messages (Forms UX lesson), auto- advancing carousels, walls of unstructured text, unexpected context changes on interaction.
  • Situational/temporary limitations — a broken arm (motor), bright sunlight (low-vision equivalent), a noisy commute (deaf equivalent), a crying baby preventing audio (deaf equivalent again). This category is why accessibility work benefits far more people than the "permanent disability" framing suggests — it's the same reason lesson 19 noted captions serve a broad silent-viewing audience, not just deaf users.
💡 The reframe worth internalizing: accessibility is rarely "extra work for a small group" — it's usually "doing the thing correctly," and the correct version serves more people than the checkbox framing suggests. Every accessible pattern in this track (real buttons, labeled forms, proper headings) also happens to be better-engineered, more testable, and more SEO-friendly code. That's not coincidence — it's the same underlying discipline.

Part 4 — The Legal and Business Reality

This stopped being optional-in-practice years ago, and the trend is accelerating: ADA (US) lawsuits over inaccessible websites number in the thousands annually and target businesses of every size, not just large enterprises; Section 508 mandates it for US government/federal-contractor sites; the European Accessibility Act (enforced from June 2025) requires WCAG AA conformance for most consumer-facing digital products/services sold in the EU, with real penalties. On the business side: ~15-20% of the population has some disability (WHO estimates), a substantial and often-overlooked market segment; accessible sites consistently show better SEO and usability metrics across the board (the same "correct code helps everyone" pattern from Part 3); and — the quieter but very real cost — remediating accessibility failures after launch is dramatically more expensive than building correctly from the start, the same "fix the root cause, not each symptom" principle that's run through this whole track's mistakes tables.

Part 5 — The Testing Toolkit

Four layers, each catching different problems — use all four, because none alone is sufficient (an idea that will recur explicitly in the ARIA lesson's "no ARIA is better than bad ARIA" rule):

  1. Automated scanners (axe DevTools extension, Lighthouse's Accessibility audit, WAVE) — fast, catch ~30-40% of real issues (missing alt, contrast failures, missing form labels, duplicate ids). Necessary but drastically insufficient — they cannot tell you whether your alt text is any GOOD, whether your tab order makes logical sense, or whether your custom widget's keyboard behavior matches user expectations. A 100% Lighthouse score is a floor, not a finish line.
  2. Manual keyboard testing — unplug the mouse (lesson 14's exercise, generalized to every page you build from now on). Can you reach everything? Is focus always visible? Does order make sense? Can you escape everything you can open?
  3. Screen reader testing — NVDA (free, Windows) or VoiceOver (free, built into macOS/iOS — Cmd+F5) on your real, finished pages. This is where you discover the gaps automated tools structurally cannot see: confusing announcements, missing context, redundant verbosity. Budget real time for this — it's the closest you'll get to experiencing the page as many of your actual users do.
  4. Real users — the gold standard, and the one most teams skip. If you can access any usability-testing budget, prioritize including disabled participants — patterns that seem fine in your own screen-reader testing sometimes fail for someone with years of real fluency in how AT is supposed to behave, catching gaps your own novice testing missed entirely.

Part 6 — The Audit Skill, Applied

Given any page, run this five-minute pass — it's the fastest way to find the highest-severity issues before reaching for tooling:

1. Zoom to 200% — does content reflow, or does it get cut off / require horizontal scroll?
2. Tab through the whole page — is every interactive thing reachable, in a sensible order,
   with visible focus at every step?
3. DevTools Accessibility panel — spot-check 5 interactive elements: correct role? Real name?
4. Turn off images (or check alt in the Elements panel) — does the page still make sense?
5. Run axe or Lighthouse — treat the report as a STARTING list, not a completion certificate.

This five-step pass is deliberately the same shape as several exercises across this track (lesson 1's parser-repair check, lesson 3's Lighthouse heading-order check, lesson 15's landmark audit) — accessibility auditing is a skill built from many small, specific checks compounding, not one grand "accessibility review" you do once at the end.

Interview Questions

Q1 (beginner): What is the accessibility tree, and how is it different from the DOM?

The DOM is your parsed HTML structure; the accessibility tree is a derived structure the browser computes FROM the DOM, exposing a role, name, and state for each relevant node — this is literally what screen readers, voice control, and switch devices read/interact with, not the DOM or the rendered pixels directly. Semantic HTML populates it correctly and automatically; poor markup (div soup) either populates it wrongly or leaves nodes out entirely.

Q2 (beginner): What does POUR stand for and why is it useful as a framework rather than just a checklist?

Perceivable, Operable, Understandable, Robust — WCAG's four organizing principles. It's useful as a lens: given a new accessibility problem, categorizing which pillar it falls under usually surfaces the right category of established fix (e.g., "this is an Operable problem" points straight at keyboard-access patterns) rather than needing to reason from scratch every time.

Q3 (mid): Why is a 100% automated accessibility score insufficient, specifically — what categories of problems can tools like axe/Lighthouse structurally not detect?

Automated tools check for the PRESENCE of certain patterns (an alt attribute exists, contrast ratio meets a threshold, form inputs have associated labels) but cannot judge QUALITY or experience: whether alt text is actually descriptive versus just present, whether tab order is logical, whether a custom widget's keyboard behavior matches user expectations, whether announcements are confusing or redundant when actually heard. Estimates put automated coverage around 30-40% of real-world issues — necessary as a fast first pass, insufficient as the only check.

Q4 (mid): A stakeholder says "we don't have any disabled users, so this isn't a priority." How do you respond?

Several angles, ideally combined: (1) you likely DO have disabled users and don't know it — disability is frequently invisible and self-reported analytics rarely capture it; (2) the situational/temporary category (Part 3) means every user experiences disability-equivalent conditions sometimes (bright sunlight, a broken arm, a noisy environment); (3) accessible code is measurably better-engineered code — real semantic elements, testable structure, better SEO — so the investment pays off independent of the accessibility framing; (4) legal exposure (ADA lawsuits, EAA compliance from 2025) is real regardless of a specific site's known user base; (5) retrofitting later costs substantially more than building correctly now. Lead with whichever resonates with the stakeholder's actual priorities — cost-avoidance framing often lands better than a pure ethics argument in a business context, even though both are true.

Q5 (senior): Your team ships fast and relies entirely on axe-core in CI as the accessibility gate. Describe the false confidence this creates and how you'd design a more complete process.

CI-only automated gating creates a specific false confidence: it blocks the ~30-40% of issues that are structurally detectable (missing labels, contrast, duplicate ids) while shipping 100% "passing" on everything automated tools cannot judge — illogical tab order, confusing screen- reader announcements, custom widgets with wrong keyboard behavior, poor-quality alt text that technically exists. A more complete process layers: automated CI gating (catch regressions cheaply, continuously — keep this), a manual keyboard-and-screen-reader pass as a REQUIRED step in PR review for any new interactive component (not just before major launches), periodic full audits by someone with genuine AT fluency (internal accessibility champion or external audit, depending on team size/budget) covering pages CI can't meaningfully assess, and — the highest- value addition most teams skip — actual usability sessions with disabled users at key product milestones, which surface the class of issue no amount of internal testing reliably catches. The senior framing: automated testing is the floor of a four-layer system (Part 5), not a substitute for the other three layers.

Q6 (senior): You're auditing a product with a 95 Lighthouse accessibility score that nonetheless generates frequent complaints from screen-reader users. Walk through your diagnostic approach.

The score-vs-complaints gap itself is the diagnostic signal: it points directly at the class of issue automated tools cannot detect (Q3), so the investigation should skip re-running more automated scans and go straight to manual/AT-based methods. Approach: (1) actually use the product with a screen reader end-to-end on the top 2-3 user flows the complaints reference, narrating the experience as you go — this alone usually surfaces the gap within minutes; (2) common culprits at this score level specifically: custom interactive widgets (lesson 16's territory — divs with ARIA roles bolted on that don't fully replicate native keyboard behavior), dynamic content updates with no live-region announcement (a cart total changing with no aria-live, so screen-reader users never learn it updated), illogical focus management after route changes/modal closes/form submissions (nothing in Lighthouse checks whether focus LANDS somewhere sensible, only whether focus indicators exist), or verbose/redundant announcements that are technically correct but practically unusable; (3) correlate specific complaints to specific flows rather than treating "accessibility" as one undifferentiated problem — the fix for a dynamic-content gap (aria-live) is entirely different from a focus-management gap (JS focus() calls at the right moments), and conflating them wastes remediation effort. The senior instinct: a high automated score with real complaints means the bug is in exactly the territory automated tools are known to miss — go straight there instead of re-auditing what's already passing.

🏋️ Exercises

1 (easy): See the third tree. Open any page you've built in this track. For 5 different elements (a heading, a link, a button, an image, a form input), compare the DOM node (Elements panel) against its accessibility tree entry (Accessibility side-panel). Note the computed role and name for each.
What to notice

Most elements' accessible name matches what you'd expect from their content/label — confirming the mapping is usually automatic and correct when you use real semantic elements. If any name is empty or wrong, that's a live bug in your own page, found using exactly the tool you'll use professionally from now on.

2 (easy): POUR categorization. For each real-world complaint, name the POUR pillar it falls under: (a) "the site is unusable with a screen reader," (b) "I can't tell which link I'm currently focused on while tabbing," (c) "the error message just says 'Error' with no explanation," (d) "a custom dropdown breaks entirely in an older screen reader version."
Answers

(a) Perceivable (content isn't reaching the user's available sense at all) — arguably also Robust if it's a structural/semantic failure. (b) Operable (focus indicator problem). (c) Understandable (unclear, unhelpful communication). (d) Robust (fails to work across the range of real assistive technology). If you assigned (a) differently with solid reasoning, that's fine — POUR categories genuinely overlap sometimes; the value is in the exercise of categorizing, not landing one "correct" single answer.

3 (medium): The five-minute audit, run for real. Pick any page from earlier in this track (or a real site). Run Part 6's five-step pass completely, writing down what you find at each step before moving to the next. Compare your manual findings (steps 1-4) against what axe/Lighthouse (step 5) actually catches.
Expected outcome

You should find at least one issue in steps 1-4 that the automated tool in step 5 either missed entirely or reported without conveying its real user impact — this gap, experienced firsthand rather than just read about, is the exercise's real point. If your manual pass and the automated tool agree on everything, try the exercise again on a page with more custom interactive components — the gap tends to widen exactly where automation is weakest.

4 (medium): Screen reader, cold start. If on Mac: enable VoiceOver (Cmd+F5) and navigate one of your built pages using ONLY VoiceOver gestures/keyboard for 10 minutes, no looking at the screen if you can manage it. If on Windows: install NVDA (free) and do the same. Write down three things that surprised you about the experience.
Common findings

Frequent first-timer surprises: how much FASTER experienced users navigate than the halting pace of a first attempt (screen reader fluency is a real, learned skill — don't judge your own slow first pass as representative); how disorienting a page with poor heading structure feels compared to a well-structured one you may have also tested; and how much information sighted users take for granted from layout/proximity that has zero equivalent when navigating linearly by structure alone. Any one of these three insights, genuinely felt rather than just read, is worth more than another chapter of theory.

5 (hard): Disability-category impact matrix. Pick a moderately complex page (a real e-commerce product page is ideal). For each of the six disability categories in Part 3, identify one SPECIFIC element or interaction on that page that would be a problem for that category if implemented carelessly, and state the correct fix (referencing earlier lessons where relevant).
Example matrix (product page)

Blind: an image-gallery carousel with no alt text on thumbnails → real alt per lesson 6. Low- vision: price/reviews in low-contrast gray text → sufficient contrast ratio, checked against WCAG AA's 4.5:1 minimum for body text. Deaf: an autoplay product demo VIDEO with spoken narration and no captions → track kind="captions" per lesson 19. Motor: a "quick view" only reachable via hover with no keyboard equivalent → lesson 15's hover-trap fix (focus-within or a real focusable trigger). Cognitive: a countdown "only 2 left, offer ends in 4:59" creating time pressure with no way to pause → WCAG's timing-adjustable requirement, genuinely worth raising with product. Situational: a checkout only reachable by precise small-target tapping, hard to use one-handed on a crowded train → adequate touch target sizing (a real WCAG 2.5.5/2.5.8 criterion). If your matrix names six DIFFERENT concrete elements rather than repeating one issue six ways, you've demonstrated the category actually shapes distinct fixes.

6 (hard): Write the audit report. Using this lesson's toolkit, conduct a full audit (all four testing layers, to whatever extent available to you) of ANY real production site of your choosing. Produce a report: 8-10 findings, each with a POUR category, a severity (Critical/Serious/Moderate/Minor — standard axe severity language), and a specific, actionable fix. Rank by severity.
What a strong report looks like

Strong reports specify EXACT elements/flows ("the 'Add to cart' button on the product detail page has no accessible name — it renders only a cart icon with aria-hidden and no aria-label") rather than generic statements ("buttons need labels"). Critical/Serious findings should concentrate on Operable failures (can't complete a core task) and severe Perceivable gaps (no alt on functionally important images); Moderate/Minor on Understandable/Robust refinements. If you found zero Critical issues on a real production site, look harder at custom interactive components specifically — that's disproportionately where they hide, per the pattern established across Parts 5-6 and the earlier interview questions in this lesson.

🎯 Key Takeaways

  • The accessibility tree — role, name, state, derived from the DOM — is what AT actually perceives; semantic HTML populates it automatically, ARIA patches it, visual CSS never touches it.
  • POUR (Perceivable, Operable, Understandable, Robust) is a filing system for accessibility knowledge, not a new checklist — nearly everything under it was already covered across lessons 1–21.
  • "Accessibility" spans blind, low-vision, deaf/HoH, motor, cognitive, and situational/temporary categories — each breaks the web differently and needs distinct fixes.
  • Legal exposure (ADA, EAA 2025+) and business cost (SEO, better-engineered code, expensive retrofits) both argue against treating this as optional or deferred.
  • Four testing layers — automated (fast, ~30-40% coverage), keyboard-only, screen reader, real users — each catch different problems; automated alone is a floor, not a finish line.
  • The five-minute audit pass (zoom, tab-through, Accessibility panel spot-check, images-off check, then automated scan) is a repeatable, fast first response to any new page.
