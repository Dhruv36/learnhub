Global Attributes & data-* Semantics

What you'll master: the attributes every element carries regardless of tag — id/class/style/title done rigorously, contenteditable and the editable-web, tabindex's full ruleset, the draggable API's HTML half, and data-* as the sanctioned bridge between markup and JavaScript (with its security boundary). This closes the Semantics & Structure section. Prerequisites: lessons 1–16.

Part 1 — What "Global" Actually Means

Every HTML element — <div>, <a>, even <html> itself — accepts a shared set of attributes, because they answer questions that apply to any node in a tree: "how do I identify this?", "can this be focused?", "can this be edited?". You've used several already (id, class, lang since lesson 2) — this lesson is the complete, rigorous tour.

Part 2 — id and class: Precision Matters

<section id="pricing" class="section section--highlighted">
  • id: unique per document (the browser doesn't enforce this — duplicate ids parse fine and silently break getElementById, label for, and fragment links, each grabbing the first match). Valid characters are looser than folklore suggests (spaces are the only hard no in modern HTML), but the universal convention is kebab-case-ascii — needed anyway for CSS selectors and URL fragments without escaping.
  • class: a space-separated list, intentionally many-per-element and reused across elements — the opposite of id's uniqueness. Two audiences use it: CSS (styling hooks) and JavaScript (behavior hooks) — mixing the two carelessly is the #1 source of "I can't rename this class, the tests will break" tech debt. Convention worth adopting: class="card js-card" — CSS never selects .js-*, JS never selects bare content classes. BEM, utility-CSS (Tailwind), and CSS Modules are three different disciplines for taming class soup — CSS track territory.

Part 3 — style and title: Present, But Suspicious

<div style="color: red;">              inline CSS — one element, one-off
<abbr title="Cascading Style Sheets">   tooltip text (met in lesson 4)

style is legal and occasionally right (a computed value from JS — el.style.width = pct + "%"; email HTML, which ignores stylesheets) but wrong as a habit: unstylable by your CSS cascade (inline wins nearly every specificity fight — the "!important of markup"), unmaintainable at scale, and a CSP red flag (Security lesson — inline styles need style-src 'unsafe-inline' or nonces, weakening your defense). Default to classes; reach for style only when the value is truly per-instance and computed.

title looks like free accessibility text and isn't: it's mouse-hover only — invisible on touch devices (no cursor), inconsistently exposed to screen readers (many skip it or require special reading modes), and impossible to see mid-interaction. Legitimate uses: supplementary hover context on already-labeled elements (abbr expansions, a truncated table cell's full value), validation constraint hints (the pattern lesson). Never your only accessible name for something — that's a job for visible text, aria-label, or alt, all covered in prior lessons.

Part 4 — tabindex: the Full Ruleset (three values, three meanings)

tabindex="0"     make a normally non-focusable element focusable, IN natural DOM order
tabindex="-1"    make it focusable ONLY via script (el.focus()) — skipped by Tab entirely
tabindex="1+"    ⚠ jump the queue to a fixed position — DON'T (revisit lesson 14's warning)

Concrete uses of the two you should ship: 0 turns a genuinely custom interactive widget into a tab stop (last resort — after confirming no native element fits, per lesson 16's method); -1 marks a programmatic focus target — error summaries (Form UX lesson), a dialog's heading, a route-changed announcement in an SPA. Natively interactive elements (a[href], button, input, select, textarea, details) never need tabindex — they're already in the tab order; adding tabindex="0" to a button is a tell that something upstream is already wrong.

Part 5 — contenteditable: HTML's Built-in Rich-Text Editor

<div contenteditable="true">
  <p>Click and just start typing — this is a live rich-text region.</p>
</div>
<div contenteditable="plaintext-only">No formatting sticks here.</div>

One attribute, an entire editing surface: cursor, selection, typing, Enter creating paragraphs, Ctrl+B/I/U, undo/redo — all native. Every "rich text" comment box, Notion-style block editor, and CMS WYSIWYG starts here (production ones layer heavy JS on top — execCommand is deprecated; modern editors intercept beforeinput events and manage the DOM directly — a rabbit hole for the JS track).

The security fact that must not be missed: contenteditable regions produce arbitrary HTML as users type/paste (paste especially — pasting from Word or a webpage brings full markup). Whatever you save and later render back is a raw user-HTML injection point exactly like the mark-highlighter and pre lessons warned about. The non-negotiable pipeline: element.innerHTML (or textContent, if you only need plain text) → sanitize with an allow-list library (DOMPurify is the industry default) → store → sanitize AGAIN on render (defense in depth — never trust that yesterday's sanitizer, or another writer's, was correct). Skipping sanitization on a contenteditable-backed feature is one of the most common stored-XSS vectors in real products — comment boxes, bios, CMS bodies.

Part 6 — draggable: the HTML Half of Drag-and-Drop

<li draggable="true" id="card-1">Design the API</li>
<ul id="todo" ondragover="event.preventDefault()"
             ondrop="handleDrop(event)">...</ul>

<script>
li.addEventListener("dragstart", e => e.dataTransfer.setData("text/plain", e.target.id));
function handleDrop(e) {
  e.preventDefault();
  const id = e.dataTransfer.getData("text/plain");
  document.getElementById("todo").appendChild(document.getElementById(id));
}
</script>

draggable="true" is the marker; the actual behavior (dragstart/dragover/drop events, the DataTransfer object carrying payload) is JavaScript's job — this lesson's scope is knowing the attribute exists and what it enables (Trello-style boards, file-drop zones, reorderable lists). Note images and links are draggable by default without the attribute (dragging a link to a new tab) — draggable="false" turns that off when it conflicts with custom drag logic. Accessibility caveat worth flagging now, detailed in the ARIA lesson: native drag-and-drop is mouse/touch-only — shipping it as the only way to reorder something excludes keyboard users; pair it with an accessible alternative (up/down buttons, or a full ARIA-grid reordering pattern).

Part 7 — data-*: the Sanctioned Bridge (and its boundary)

<button data-product-id="8471" data-action="add-to-cart" data-price="24.99">
  Add to cart
</button>

<script>
btn.addEventListener("click", () => {
  const { productId, action, price } = btn.dataset;   // camelCase, automatically!
  addToCart(productId, Number(price));
});
</script>

Any attribute starting with data- is guaranteed valid forever (the spec reserves the whole prefix for authors) and surfaces in JS as element.dataset, with automatic kebab-case → camelCase conversion (data-product-id → dataset.productId). This is the mechanism this whole track has been gesturing at since lesson 4's aside comparing it to <time datetime>: data-* is private to your own code; spec attributes/elements are read by everyone. Test frameworks lean on this distinction hard — data-testid is the standard convention precisely because it can never collide with a styling class or an ARIA attribute a designer/dev might change.

  • data-* is state storage, not a security boundary. It's plain markup — visible in View Source, editable in DevTools by anyone (parameter tampering, again — Forms lesson's hidden-input rule applies identically here: never trust a data-price attribute server-side for the real charge).
  • CSS can read it via attribute selectors and attr() — [data-state="loading"] .spinner { display: block }, content: attr(data-count) — a legitimate no-JS styling hook for state that a script toggles.
  • Don't let it become a shadow database. data-* holding one flag or id is idiomatic; a component tracking twelve interdependent data-* attributes has smuggled application state into the DOM where nothing enforces consistency — that's what JS variables/objects (or a framework's state) are for. The DOM should reflect state, not BE the sole copy of it, for anything nontrivial.

Part 8 — Worked Example: A Card Wiring Every Attribute Correctly

<li class="cart-item js-cart-item" id="item-8471" data-product-id="8471" data-qty="2">
  <img src="/img/chair.avif" alt="Ergonomic desk chair" width="80" height="80">
  <div>
    <h3 title="Ergonomic Desk Chair — Walnut Frame, Grey Mesh">
      Ergonomic Desk Chair</h3>                        <!-- title: supplementary, name is visible -->
    <p>$189.00 × <span class="js-qty" data-testid="qty-value">2</span></p>
  </div>
  <button type="button" class="js-remove" aria-label="Remove Ergonomic Desk Chair from cart"
          data-product-id="8471">✕</button>
</li>

Every attribute earns its place: id for the one DOM hook this item needs; class split into styling (cart-item) vs behavior (js-*) namespaces; data-product-id duplicated onto the remove button so its click handler doesn't need to walk up to the parent; title adds supplementary detail over an already-visible, already-adequate heading (never the only name); data-testid gives tests a stable hook independent of styling/copy changes; no style attribute, no misplaced tabindex (the button needs none — it's already focusable) — and the server, receiving a remove/quantity-change request, re-derives the real price from product-id, never trusting a data-price you'll notice was deliberately absent here.

Interview Questions

Q1 (beginner): Why does duplicate id "still work" in the browser, and what actually breaks?

The parser doesn't reject duplicate ids — it's a document validity rule, not a hard parsing rule, so the page renders fine. What breaks silently: getElementById, label[for], CSS #id selectors, and URL fragments all resolve to the FIRST matching element only — the second one becomes permanently unreachable by id, a bug that shows up as "my click handler affects the wrong item."

Q2 (beginner): Why is title a poor substitute for a visible label or aria-label?

title only appears on mouse hover after a delay — invisible to touch/mobile users (no hover concept), unreliable across screen readers (many skip it or need special reading modes), and impossible to reference mid-task. It's supplementary hover context on top of an already-adequately-named element, never the sole source of an accessible name.

Q3 (mid): Explain data-* to someone who thinks "why not just use a regular attribute or a class?"

Arbitrary custom attributes aren't spec-guaranteed — a browser update could introduce a real attribute with the same name and collide; data-* is the reserved-forever prefix, so data-anything can never collide with a future spec addition. dataset gives typed, camelCased JS access automatically. Classes are for CSS-facing groups/hooks, not key-value state — data-* is explicitly for attaching structured private state/values, which classes can't hold cleanly (you'd need to parse "qty-2" out of a class string).

Q4 (mid): A comment box uses contenteditable and saves innerHTML directly to the database, then renders it back with innerHTML on other users' pages. Find the bug.

Stored XSS: contenteditable produces real HTML from typing AND pasting (pasted content can carry <script>, onerror handlers, iframes — anything). Saving raw innerHTML persists an attacker's payload; rendering it back with innerHTML on every viewer's page executes it against every viewer's session. Fix: sanitize with an allow-list library (DOMPurify) before storing AND again before rendering (defense in depth against a sanitizer bypass or a second insertion path bypassing the first check).

Q5 (senior): Your team stores UI state entirely in data-* attributes (data-open, data-step, data-selected-ids — a comma list) and reads them everywhere via querySelector. Critique the architecture.

The DOM has become the single source of truth for application state with none of a real store's guarantees: no atomicity (two data-* writes can leave inconsistent intermediate states visible to a mid-update querySelector elsewhere), no change notifications (nothing re-renders automatically; manual sync code proliferates), string-only storage (comma-joined ids are a hand-rolled, fragile serialization format), and debuggability suffers (state is scattered across N elements' attribute maps instead of one inspectable object). Legitimate uses of data-* remain: styling hooks ([data-open] CSS), test ids, and small, genuinely per-element flags. The fix isn't zero data-*, it's moving relational/derived/multi-element state into an actual state layer (even a plain JS object with one render function) and letting data-* attributes be a projection of it for CSS, not the record of truth. This is exactly the "DOM reflects state, doesn't own it" line from the lesson, at production scale.

Q6 (senior): Design a drag-and-drop reorderable list that's also fully keyboard accessible. What does the native draggable attribute NOT give you, and how do you complete the picture?

draggable + dragstart/dragover/drop is mouse/touch-only by construction — there's no keyboard equivalent event pipeline, so a pure drag-and-drop implementation silently excludes keyboard and many switch-device users, a real WCAG 2.1.1 failure. Complete design: keep native drag-and-drop for pointer users (it's the expected, performant path — no need to reimplement it), and add a parallel keyboard path — each item focusable (already true if it's a real button/li with a control), with "Move up"/"Move down" buttons or an ARIA-grid-with-arrow-keys pattern, both paths calling the SAME reorder function so state stays single-sourced; announce moves via a live region ("Design the API moved to position 2 of 5") since neither drag nor button-click alone conveys the new order to a screen-reader user watching the list. Naming the WCAG gap and the shared-reorder- function detail unprompted is the senior signal.

🏋️ Exercises

1 (easy): The duplicate-id bug, reproduced. Build a page with two elements sharing id="box", give the second one distinguishing text, and run document.getElementById('box').textContent in the Console. Also try a #box CSS rule and a label for="box". Report which element "wins" each time.
Expected

All three — getElementById, the CSS selector, and label's association — resolve to the FIRST box in every case. The second is unreachable by id through any of these three mechanisms. This single experiment is worth more than memorizing the rule.

2 (easy): Class namespace audit. Take any component you've built earlier in this track and split its classes into content (styling) vs js- (behavior) prefixed ones. Update your CSS selectors and any JS querySelectors accordingly. Confirm nothing broke.
What to verify

CSS should now select zero .js-* classes; JS should select zero bare content classes. If your JS querySelector still targets a content class, that's the exact coupling this convention removes — a future designer renaming ".card" for a redesign would silently break your click handler. Fixing it now costs one find-and-replace; finding it in production costs a support ticket.

3 (medium): dataset round-trip. Build a product card with data-product-id, data-in-stock (string "true"/"false"), and data-tags="new,sale". Read all three via dataset in the Console, correctly converting in-stock to a real boolean and tags to a real array. Note what dataset gives you natively vs what you must parse yourself.
Solution 
const id = card.dataset.productId;                  // string, as-is
const inStock = card.dataset.inStock === "true";     // YOU convert — dataset is always strings
const tags = card.dataset.tags.split(",");           // YOU parse — no array support

dataset gives free camelCasing and nothing else — every value is a string, always. Booleans and lists are your responsibility to encode/decode consistently; this is exactly why nontrivial state (Q5) doesn't belong here.

4 (medium): Sanitized comment box. Build a contenteditable div, a "Post" button that reads innerHTML, and a rendered list of "saved" comments (in-memory array is fine — no real backend needed). Paste in something containing a script tag and an onerror-laden img tag from another page (or type them as literal text via a workaround) and confirm your naive version executes them. Then add DOMPurify (via CDN or download) and fix it.
Fix 
const clean = DOMPurify.sanitize(editableDiv.innerHTML);
comments.push(clean);
list.innerHTML += `<li>${clean}</li>`;   // sanitize on save AND would re-sanitize on render too

Confirm the script/onerror payloads are stripped or neutralized after sanitizing, while legitimate formatting (bold, links from paste) survives — that selective behavior is exactly what an allow-list sanitizer does that a naive regex or "just don't allow <script>" check cannot (attribute-based vectors like onerror slip past keyword blocklists routinely).

5 (hard): Keyboard-accessible reorder. Build a 5-item draggable list (native HTML5 drag-and-drop, mouse-only) then add parallel "▲"/"▼" buttons per item that call the identical reorder function your drop handler uses. Verify: dragging reorders, buttons reorder, and add an aria-live region announcing "X moved to position N of 5" for the button path. Tab through with the mouse unplugged and confirm full reordering is possible without it.
Architecture 
function moveItem(id, direction) {          // ONE function, both paths call it
  const list = [...items];
  const i = list.findIndex(x => x.id === id);
  const j = direction === "up" ? i - 1 : i + 1;
  if (j < 0 || j >= list.length) return;
  [list[i], list[j]] = [list[j], list[i]];
  render(list);
  announce(`${list[j].label} moved to position ${j + 1} of ${list.length}`);
}
// dragstart/drop handler calls moveItem(draggedId, computedDirection)
// button onclick calls moveItem(item.id, "up"/"down")

The shared function is the whole exercise — two input paths, one source of truth, one announcement point. This is Q6's answer, built.

6 (hard): The tampering demo, closed. Build a fake checkout row with data-price="24.99" and a JS-computed total reading it. Edit the attribute in DevTools to "0.01" and watch the total update. Then write the two-paragraph fix describing what a real checkout's client AND server must each do, tying together this lesson's data-* boundary with the Forms lesson's hidden-input rule and the earlier parameter-tampering exercise.
Fix write-up core

Client: data-price (like a hidden input's value) is a convenience for rendering/optimistic UI only — fine to display, never authoritative. Server: on checkout submission, re-look-up every product's real price by product-id from the database/price service, ignoring any client- supplied price entirely; recompute the total server-side; sign or server-generate any total the client is merely allowed to display back. The unifying principle across data-*, hidden inputs, and query parameters is identical: anything that travels through the client's DOM or request is editable by the client, so authority lives only where the client cannot reach — the server, re-deriving from source data every time.

🎯 Key Takeaways

  • id must be unique (unenforced — duplicates silently break getElementById/label-for/fragments); class serves two audiences — separate CSS-facing and js- behavior-facing namespaces.
  • style is for computed per-instance values only; title is supplementary hover-only text, never the sole accessible name.
  • tabindex: 0 makes a custom widget a natural-order tab stop, -1 makes a programmatic focus target — natively interactive elements need neither.
  • contenteditable is a full built-in rich-text editor — and its output is arbitrary user HTML: sanitize (allow-list, e.g. DOMPurify) on save AND on render, always.
  • draggable="true" plus JS events builds drag-and-drop — which is pointer-only; ship a keyboard-operable parallel path through the same reorder function.
  • data-* is the spec-safe bridge from markup to your own JS (typed via .dataset) and to CSS (attribute selectors/attr()) — but it's visible, user-editable state, never a security boundary, and not a substitute for real application state at scale.
