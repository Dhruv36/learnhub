Images Mastery Foundations

What you'll master: the img element done properly — alt text as a craft (the most common accessibility failure on the entire web), width/height and layout shift, choosing among JPEG/PNG/WebP/AVIF/SVG, figure/figcaption, and lazy loading. Responsive images (srcset/picture) are big enough to get their own lesson in the Media section — this lesson is the foundation it builds on. Prerequisites: lessons 1–5.

Part 1 — The Element, Fully Specified

<img src="team-retreat.jpg"
     alt="Six engineers around a whiteboard covered in sticky notes"
     width="800" height="533"
     loading="lazy">

Four attributes, and every one prevents a specific production problem. <img> is a void element (no closing tag) and — importantly — a replaced element: the browser replaces it with external content, which arrives later than the HTML. Most img subtleties flow from that time gap.

Part 2 — Alt Text: The Craft

WebAIM's annual survey of the top million homepages finds missing/useless alt text is among the top two accessibility failures every single year. It's not hard — it's just actually a writing skill nobody teaches. The core question: "What would I say instead of the image if I were reading this page aloud to someone?"

Image rolealt should beExample
Informative (shows content)The information it conveysalt="Line chart: signups doubled after the June launch"

Decorative (adds no information)Empty — but present: alt=""Background flourishes, spacer graphics
Complex (chart/diagram)Short alt + long description nearbyalt="Q2 revenue chart, described below" + a real text summary
Text in an image (avoid making these)The exact textScreenshot of an error → alt contains the error message

The rules behind the table:

  • alt="" ≠ missing alt. Empty alt says "decorative — skip me silently." A missing alt attribute makes many screen readers read the filename aloud: "IMG underscore 4-7-3-2 dot JPEG." Every img gets an alt attribute; some get an empty one. This distinction is a guaranteed interview check.
  • Don't write "image of / picture of" — the screen reader already announces "image." Straight to the content.
  • Context decides the text. The same photo of a barista needs alt="Latte art being poured" in a brewing guide, but alt="Maya Chen, founder" on the About page. Alt describes the image's job here, not its pixels.
  • Length: aim under ~125 characters; complex images get a short alt plus adjacent visible text (which helps everyone, not just screen-reader users).

Part 3 — width/height and the Layout-Shift Story

Here's the production problem: HTML arrives first, images seconds later. If the browser doesn't know an image's size, it renders the page with a 0-height gap, then shoves everything down when the image arrives. You've experienced it: you go to tap a link and the page jumps and you tap an ad instead. That jump is CLS — Cumulative Layout Shift — one of Google's Core Web Vitals, meaning it directly affects your search ranking.

<img src="hero.jpg" alt="..." width="1200" height="600">

/* with this CSS (standard in every modern project): */
img { max-width: 100%; height: auto; }

How the fix works: the width/height attributes give the browser the image's aspect ratio (2:1 here) before any pixels arrive, so it reserves correctly-shaped space; the CSS then lets the image scale responsively within that ratio. You are not fixing the display size in stone — you're declaring the shape. Always set both attributes, to the image file's intrinsic pixel dimensions.

The two decisions that cover 95% of cases: photo-like → AVIF (or WebP), geometric/line art → SVG. The responsive-images lesson shows <picture> serving AVIF with JPEG fallback. Two more production notes: strip EXIF metadata from uploads (photos carry GPS coordinates — a privacy leak your pipeline must handle), and remember SVG is XML text — it can contain scripts, so never serve user-uploaded SVG from your main domain (XSS vector; details in the Security lesson).

Part 5 — figure & figcaption: Images With Captions

<figure>
  <img src="signup-funnel.png" alt="Funnel chart, key numbers in caption"
       width="900" height="500">
  <figcaption>Signup funnel, June 2026: 14,200 visits → 3,100 trials → 640 paid (4.5%).</figcaption>
</figure>

figure = self-contained content referenced from the main text (image, chart, code listing, even a quote — you used it in lesson 3); figcaption = its visible caption. Semantics: the caption is programmatically associated with the image — screen readers announce them together. Notice the division of labor above: the caption carries the data, so the alt can be brief and point to it. Alt and caption should complement, never duplicate — a screen reader user would hear the same sentence twice.

Part 6 — Loading Behavior

<img src="comment-avatar.jpg" alt="" loading="lazy" width="48" height="48">
<img src="hero.jpg" alt="..." fetchpriority="high" width="1200" height="600">
  • loading="lazy" — the browser defers fetching until the image nears the viewport. Free bandwidth savings for long pages. The critical rule: never lazy-load above-the-fold images — you'd delay your own hero image and worsen LCP (Largest Contentful Paint, another Core Web Vital). Default: lazy for everything below the fold, eager (the default) above it.
  • fetchpriority="high" — the opposite lever: tells the browser this image (your LCP hero) should jump the queue.
  • decoding="async" — allows decode off the critical path; minor, but harmless to add on non-critical images.
  • Broken images: the alt text displays in place (you proved this in lesson 1) — one more reason every image has one. There's no native "fallback image" in HTML; that needs JS (onerror) or the picture element.

Part 7 — Step-by-Step: Ship a Correct Image

The full professional checklist, applied to adding one product photo:

1. Resize the source: largest displayed size × 2 (retina) — not the camera's 6000px original
2. Convert: AVIF or WebP for the photo (aim < 200 KB; hero images < 500 KB)
3. Name it: ergonomic-chair-side.avif (descriptive, hyphenated — filenames are an SEO signal)
4. Write the tag:
   <img src="/img/ergonomic-chair-side.avif"
        alt="Side view showing the adjustable lumbar support"
        width="800" height="800" loading="lazy">
5. Verify: DevTools Network tab (size on the wire), Lighthouse (CLS, properly-sized-images audits)

Interview Questions

Q1 (beginner): What's the difference between alt="" and no alt attribute?

alt="" is a deliberate declaration that the image is decorative — screen readers skip it silently. A missing attribute is an authoring error — many screen readers then announce the filename, which is noise. Rule: the attribute is universal; its emptiness is the decorative signal. Follow-up worth volunteering: purely decorative images are often better as CSS backgrounds, removing the question entirely.

Q2 (beginner): Write alt text for a logo image that links to the homepage.

Functional image → describe the function: alt="Acme home" (or just "Acme" — the link context implies navigation). Not "logo", not "Acme logo, blue hexagon with white lettering" — the visual details don't help someone trying to navigate. The image's job is "go home"; the alt says so.

Q3 (mid): Why do width and height attributes still matter now that CSS controls layout?

They give the browser the intrinsic aspect ratio before the image downloads, so it reserves correctly-shaped space and avoids layout shift (CLS — a Core Web Vital affecting ranking). Modern browsers literally derive a CSS aspect-ratio from the attributes. With max-width:100%/height:auto CSS, the attributes don't fix the display size — they fix the shape. Omitting them = every image load shoves the page around.

Q4 (mid): Walk through format choice for: a product photo, a screenshot of an error dialog, the company logo, an animated product demo.

Photo → AVIF (best compression, universal modern support), WebP or JPEG as pipeline fallback. Screenshot → PNG (lossless — JPEG artifacts smear text) or lossless WebP. Logo → SVG (vector: crisp at every size, tiny, recolorable via CSS). Animated demo → an actual video file (<video> muted autoplay loop) — 10× smaller than GIF; animated AVIF/WebP if it must be an image. Naming GIF for anything is the wrong answer in 2026.

Q5 (senior): Your image-heavy site scores poorly on LCP and CLS. Give a prioritized remediation plan.

Measure first: Lighthouse + field data (CrUX) to find the LCP element — usually the hero image. Then, in impact order: (1) hero: no lazy-loading, fetchpriority="high", possibly rel="preload"; (2) width/height (or CSS aspect-ratio) on every image → kills image CLS; (3) modern formats + right-sized variants (srcset — responsive images lesson) → cuts bytes, speeds LCP; (4) lazy-load below-fold images → frees bandwidth for critical ones; (5) CDN with on-the-fly transforms so the pipeline stays automated; (6) guard against regressions: Lighthouse CI budgets. Senior markers: measuring before touching anything, and knowing lazy-loading the hero makes things worse.

Q6 (senior): Why is user-uploaded SVG dangerous, and what are the mitigation options?

SVG is XML that can embed <script>, event handlers, foreignObject HTML, and external references — an uploaded avatar.svg can be a stored-XSS payload executing on your origin with your users' cookies. Note the subtlety: SVG in an <img> tag does NOT execute scripts, but the same file opened directly (its URL) or inlined into the DOM does — and attackers link directly. Mitigations, in preference order: (1) rasterize uploads to PNG/AVIF server-side (eliminates the class), (2) serve uploads from a separate sandboxed domain (userscontent.example) so any XSS lands on a cookie-less origin — the industry pattern (googleusercontent, etc.), (3) sanitize the XML with an allow-list library, defense-in-depth alongside CSP. Regex stripping is not on the list.

🏋️ Exercises

1 (easy): Alt-text drill. Write alt for: (a) a chart showing churn dropping from 8% to 3% after onboarding emails, (b) a decorative wavy divider, (c) a magnifying glass icon inside a search button, (d) a screenshot of a 500 error page in a bug report, (e) your About-page portrait.
Answers

(a) alt="Churn fell from 8% to 3% after onboarding emails launched" — the finding, not "a chart". (b) alt="" — decorative. (c) alt="Search" — the function (better: text in the button and alt="" or an aria-label — ARIA lesson). (d) alt="Error page: 500 Internal Server Error at /checkout, request ID abc123" — the text a teammate needs. (e) alt="Maya Chen" or with relevant context "Maya Chen roasting coffee" — not a physical description inventory.

2 (easy): CLS in the wild. Make a page with three paragraphs, an image between them WITHOUT width/height, and DevTools network throttling set to "Slow 3G". Reload and watch the text jump. Add the attributes; reload; confirm the reserved box. Then run Lighthouse and find the CLS number both times.
Expected result

Without attributes: visible shove when the image lands; Lighthouse CLS > 0 and an "image elements do not have explicit width and height" audit failure. With them: a correctly-shaped placeholder box from first paint, CLS ≈ 0. One experiment, permanent habit.

3 (medium): The format bake-off. Take one photo. Export it as JPEG (quality ~80), PNG, WebP, and AVIF (Squoosh.app does all four in-browser). Record the file sizes, view them side by side at 100%, and write two sentences on what you'd ship and why. Bonus: run the same test on a screenshot of text and watch the ranking invert.
Typical results

Photo: PNG enormous (often 10× JPEG), JPEG baseline, WebP ~30% under JPEG, AVIF ~half of JPEG — ship AVIF. Screenshot: JPEG smears the text with artifacts while PNG/lossless-WebP are crisp AND small (flat colors compress losslessly very well) — ship PNG or lossless WebP. Moral: "which format" always starts with "what kind of pixels".

4 (medium): Figure gallery. Extend the coffee site with a 3-figure "Our roastery" section: each figure has a properly-sized image (attributes included), alt that complements (not repeats) its figcaption, and lazy loading. Verify with a screen reader if you can (NVDA is free; VoiceOver is built into macOS) — hear how figure+alt+caption announce together.
Solution sketch + check 
<figure>
  <img src="/img/roaster-drum.avif" width="800" height="600" loading="lazy"
       alt="Beans mid-roast inside the drum">
  <figcaption>Our 1959 Probat drum roaster, restored in 2021.</figcaption>
</figure>

Check the complement rule: the alt says what's shown, the caption adds what's known. If reading alt+caption aloud sounds redundant, rewrite one of them.

5 (hard): Image audit script. Console script: list every img on a page with its rendered size vs natural size, missing-alt status, and loading attribute. Run it on your site and one major site; report the worst offender.
Reference implementation 
console.table([...document.images].map(img => ({
  src: img.currentSrc.split('/').pop().slice(0, 30),
  alt: img.hasAttribute('alt') ? (img.alt || '(decorative)') : '❌ MISSING',
  natural: `${img.naturalWidth}×${img.naturalHeight}`,
  rendered: `${img.clientWidth}×${img.clientHeight}`,
  oversized: img.naturalWidth > img.clientWidth * 2.2 ? '⚠ yes' : '',
  loading: img.loading || 'eager',
})));

The oversized column (natural > ~2× rendered, allowing for retina) finds wasted bytes — exactly what Lighthouse's "properly size images" audit checks. Missing-alt rows are instant accessibility bugs. You now own a one-paste image auditor.

6 (hard): The fallback problem. HTML has no native fallback-src. Implement one: an img that, on error, swaps to /img/placeholder.svg exactly once (guard against the placeholder itself failing → infinite loop). Then write two sentences on why the guard matters.
Reference implementation 
<img src="/img/product-9931.avif" alt="Walnut desk organizer"
     width="600" height="400"
     onerror="if (!this.dataset.failed) { this.dataset.failed = 1; this.src = '/img/placeholder.svg'; }">

Without the guard: if placeholder.svg 404s too, error → set src → error → set src… an infinite request loop hammering your server (a real incident pattern!). The dataset flag ensures one swap. In frameworks you'd do the same in an onError handler; the loop hazard is identical.

🎯 Key Takeaways

  • Alt text answers "what would I say instead of the image?" — informative images get their information, functional images their function, decorative images an empty but present alt="".
  • width + height attributes declare the aspect ratio before pixels arrive — with max-width:100%; height:auto, that's the CLS fix, and CLS is a ranking factor.
  • Formats: photos → AVIF/WebP, exact-pixel/text → PNG, vector art → SVG, animation → real video. GIF is retired.
  • loading="lazy" below the fold only; the hero gets eager + fetchpriority="high" — lazy-loading your LCP image is self-sabotage.
  • figure/figcaption associate captions programmatically; alt and caption complement, never duplicate.
  • Pipeline responsibilities: resize to ~2× display size, compress, strip EXIF (GPS!), and treat user-uploaded SVG as executable content.
