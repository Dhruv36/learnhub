Choosing the Right Element Semantics

What you'll master: the unified decision method that makes every "which element?" question mechanical — plus the high-stakes cases teams actually argue about: button-vs-link (revisited with the full contract table), clickable cards, icon buttons, headings-vs-bold, and the cost model of getting it wrong. This lesson turns six lessons of vocabulary into a reflex. Prerequisites: lessons 1–15 (this is a consolidation lesson).

Part 1 — The Method: Behavior → Meaning → Nothing

Ask in this order:
1. BEHAVIOR: does the user DO something with it?
   → the element with that behavior built in:
     navigate=a · act=button · choose=input/select · disclose=details · overlay=dialog
2. MEANING: does the content CLAIM something?
   → the element making that claim:
     heading=h1-h6 · step-list=ol · term/def=dl · data-grid=table · date=time
     emphasis=em · importance=strong · code=code · quote=blockquote/q · image w/ meaning=img+alt
3. NOTHING: pure styling hook?
   → div (block) / span (inline), with a class — honestly and without guilt.

Behavior outranks meaning because interactive elements carry contracts — keyboard handling, focus, AT announcements, form participation — that no amount of ARIA retrofitting fully recovers. Meaning outranks nothing because claims are free machine-readability. And "nothing" is a legitimate answer — the method's power is that it terminates: no more staring at MDN's element list.

Part 2 — The Contract Table (what native elements actually ship)

<button>Tab-focusable; Enter AND Space activate"button" + its labelform participation, disabled state, :focus-visible

That last row is the argument. A clickable div needs, to reach parity with button: tabindex="0", role="button", a keydown handler for Enter AND Space (Space must also not scroll the page — preventDefault), aria-disabled management, focus styles… and after all that it still doesn't participate in forms. The going rate is ~15 lines of subtle code per fake button, times every place someone copies it, times every bug in it. The native element is the cheap option. Restyling a real button to look like anything costs three lines of CSS.

Part 3 — The Cases Teams Argue About

Case 1: "It looks like a button but goes somewhere" (and vice versa)

Resolved in lesson 5, now with the full rule: the URL decides. Changes the URL / shows new content → <a href>, styled as button-y as design wants. Mutates state here → <button>, styled as link-y as design wants. The two costumes are trivially swappable in CSS; the contracts are not. Litmus: "would open-in-new-tab make sense?" — if yes, it must be a link.

Case 2: the clickable card

Whole product card navigates on click. Wrong: onclick on the card div (fake link). Wrong-ish: anchor wrapping the whole card (accessible name becomes the entire card text; nested buttons illegal). The design-system answer (lesson 5's exercise, now canonized): real link on the title, stretched over the card with CSS; secondary actions layer above with z-index. Bookmark this pattern; it recurs in every product UI.

Case 3: the icon button

<button type="button" aria-label="Close">
  <svg aria-hidden="true" …></svg>      <!-- icon is decorative; the LABEL carries meaning -->
</button>

Real button (behavior), aria-label naming the action (not "X icon"), icon hidden from AT. The same trio handles hamburger menus, kebabs, and every toolbar. If the button toggles, add aria-pressed/aria-expanded — state, not just name (ARIA lesson).

Case 4: "make it big and bold" ≠ heading (and the reverse)

Big bold marketing text that isn't a document section → styled p/div, not an h3 (false outline entry). A section's title that design wants small → real h2, styled small (true outline entry). The outline is a claim graph; size is a costume. You know this from lesson 3 — it remains the most-violated rule in production, which is why it's re-stated here in the consolidation.

Case 5: the "clickable row"

Table rows that navigate on click: keep a real link in the row's first (th) cell; optionally extend the click target with JS as enhancement (row onclick delegating to the link's click, with text-selection guards). The link remains the source of truth — keyboard, AT, new-tab all work. Row-onclick-only is the div-button mistake wearing a table costume.

Part 5 — Worked Refactor: One Component, Method Applied

A "notification item" as juniors ship it:

<div class="notif" onclick="openThread('t42')">
  <div class="notif-title"><b>New reply</b></div>
  <div class="notif-body">Asha replied to your comment</div>
  <div class="notif-time">2h ago</div>
  <div class="notif-x" onclick="dismiss(event,'t42')">✕</div>
</div>

Apply the method line by line: the item navigates (behavior → link); "New reply" is the item's title within a list of parallel items (meaning → heading? No — in a list context, b for keyword-attention is honest; a heading per notification would flood the outline); "2h ago" is a date (meaning → time); ✕ acts (behavior → button); the wrapper is one of many parallel self-contained items (meaning → li, in a ul, possibly article if syndication-shaped):

<li class="notif">
  <a class="notif-link" href="/threads/t42">
    <b>New reply</b> — Asha replied to your comment
  </a>
  <time datetime="2026-07-12T07:40" class="notif-time">2h ago</time>
  <button type="button" class="notif-x" aria-label="Dismiss notification">✕</button>
</li>

Same rendered pixels (CSS: stretched-link pattern, button layered above). Gained: keyboard operation, AT announcements ("link… button, Dismiss notification"), middle-click-to-open-thread, machine-readable time, and the dismiss/navigate collision resolved structurally instead of with event.stopPropagation() prayers.

Interview Questions

Q1 (beginner): Walk me through your process for choosing an element for any given UI piece.

Behavior → meaning → nothing. Does the user do something? Use the element with that contract (a/button/input/details/dialog). Does the content claim something? Use the claiming element (headings, lists, table, time, em/strong…). Neither? div/span honestly. Order matters: behavior contracts (keyboard/AT/forms) are the hardest to retrofit.

Q2 (beginner): List what a native button provides that a clickable div doesn't.

Tab focus, Enter AND Space activation (Space without page-scroll), "button" role + label announcement, disabled semantics, form participation (submit/reset), :focus-visible behavior. Reaching parity needs tabindex, role, two-key keydown handling with preventDefault, ARIA state management — ~15 fragile lines, and form participation never comes back.

Q3 (mid): Design the accessible clickable product card. What's wrong with the two obvious approaches?

Obvious #1 — onclick on the card: fake link, no keyboard/AT/new-tab. Obvious #2 — anchor around the card: accessible name becomes the whole card's text (verbose), text drag-selection dies, and a nested Add-to-Cart button is invalid (interactive-in-interactive). The pattern: real link on the title + CSS stretched-link (::after inset:0 over a relative card); secondary buttons positioned above it. Result: concise name, working buttons, full link contract.

Q4 (mid): A teammate says "we add role='button' and a click handler to divs — same thing, and our lint requires the role." Respond.

role="button" changes the announcement only — it adds no keyboard behavior (Tab/Enter/Space), no focus, no disabled semantics, no form participation; you must hand-build each, including Space's scroll-prevention subtlety, and maintain it everywhere the pattern is copied. The lint rule is a symptom: it's institutionalizing the patch instead of the fix. The native button is less code, zero maintenance, and legally safer. Offer the migration: it's usually a codemod (div→button + type + removing the ARIA scaffolding), and the CSS diff is button { all: unset }-style resets or a .btn class swap.

Q5 (senior): Your design system's Button component renders a button, but PMs keep asking for "buttons that link". Engineers pass onClick={() => router.push(...)}. Assess and fix at the system level.

Assessment: router.push in onClick is the fake link at scale — no new-tab/middle-click, no status-bar preview, broken cmd-click muscle memory, invisible to crawlers, and analytics loses referrer semantics. Every design system converges on the same fix: the Button component accepts href and renders an anchor with button styling when present (polymorphic "as" prop), keeping ONE visual API over two correct elements; lint bans router.push inside onClick handlers of non-form elements; docs state the URL rule ("if it changes the URL, it gets href"). The senior point: this is an API-design problem — make the correct markup the ergonomic default and the wrong one unrepresentable, rather than reviewing it forever.

Q6 (senior): You inherit 400 templates of div-soup. You can't fix everything. Prioritize semantic remediation by impact.

Triage by task-blocking severity: (1) interactive fakes first — div-buttons/fake links on core flows (checkout, auth, nav): these BLOCK keyboard/AT task completion and carry legal risk; mechanical codemod where patterns repeat. (2) Forms: labels, names, fieldsets — data integrity + blocked completion. (3) Page skeletons: one main + landmarks + labeled navs on the highest- traffic templates — cheap (wrapper swaps), unlocks AT navigation site-wide. (4) Headings on content templates (usually template-level fixes multiplying across pages). (5) Cosmetic semantics (em/strong/time) last — real but non-blocking. Instrument: axe-core in CI freezes regression while you dig; screen-reader session on the top task validates reality. The shape: template-level fixes over page-level, blocking over annoying, measured over exhaustive.

🏋️ Exercises

1 (easy): Speed drill. Element for each, five seconds apiece: (a) "Load more" appending items, (b) "View all" going to /products, (c) a togglable FAQ answer, (d) "2 days ago", (e) a keyboard shortcut shown in docs, (f) the modal asking "discard draft?", (g) a decorative divider swoosh, (h) "Step 1… Step 5" checkout indicator.
Answers

(a) button (acts here) (b) a href (URL) (c) details/summary (d) time datetime (e) kbd (f) dialog + showModal (g) div/CSS — or nothing at all (background-image) (h) ol — order IS the meaning (inside a nav aria-label="Progress" if it links). 7+ instant = the reflex is forming.

2 (easy): Costume swap. Style a real link to look exactly like your site's button, and a real button to look exactly like an inline link. Confirm both keep their contracts (new-tab menu on the link; Space activation on the button).
Point proven

~6 lines of CSS each (.btn class on the anchor; background:none;border:0;color:link-blue; text-decoration:underline;cursor:pointer on the button). Right-click the button-looking link: full link menu. Space on the link-looking button: activates. Appearance and contract are fully independent — you now have the demo to end the "but it needs to LOOK like" argument forever.

3 (medium): Build the canonical card. Product card with image, title (linked+stretched), price, and a working "Add to cart" button layered above the stretched link. Verify: Tab order sensible, both actions keyboard-operable, card click navigates, button click doesn't navigate — with zero stopPropagation.
CSS core 
.card { position: relative; }
.card h3 a::after { content: ""; position: absolute; inset: 0; }
.card .add-btn { position: relative; z-index: 1; }

The stacking does the collision resolution structurally: the button sits above the stretched pseudo-element, so its clicks never reach the link — no event gymnastics. Tab hits link then button, in DOM order. This is the exact pattern in Bootstrap ("stretched-link") and most design systems; you've now built it from first principles.

4 (medium): The notification refactor, extended. Take Part 5's refactored notification and put five of them in a proper list with: an "unread" state conveyed semantically (not color-only!), a "Mark all read" control, and a live count. Choose every element via the method and annotate your choices.
Key choices

ul (parallel items) > li; unread: a visually-hidden "Unread:" prefix span or aria-label addition PLUS the visual dot (never color alone — lesson 5's WCAG rule generalizes); "Mark all read": button (acts); the count: output or aria-live region (it changes dynamically — output is the honest form-adjacent choice; a span with aria-live="polite" is fine too). If your annotations cite behavior/meaning/nothing per element, the lesson stuck.

5 (hard): Fake-widget hunt. On any two major sites, find three interactive elements and inspect: are they native (button/a/details/dialog) or ARIA'd divs? For each fake, test what's broken (keyboard, Space key, new-tab) and write the one-line native fix. Score the sites.
What you'll find

Even excellent sites ship div-dropdowns and span-buttons in older corners; the commonest finding is role="button" divs missing Space handling (test: focus it, press Space — page scrolls instead of activating). Fake links (onclick divs) reveal themselves by an empty right-click menu. Your one-liners will mostly read "replace with button type=button" — the entire lesson in a repeated sentence.

6 (hard): Write the team guide. Distill this lesson into a one-page "choosing elements" doc for a hypothetical team: the method, the contract table (abbreviated), the five argued cases with verdicts, and three lint rules you'd adopt. Optimize for the engineer with 30 seconds and a PR open.
Grading rubric

Strong versions: method as three questions at top (scannable), the URL rule and card pattern as the two most-cited verdicts, lint rules like jsx-a11y/no-noninteractive-element-interactions, no-static-element-interactions, and anchor-is-valid (or html-validate equivalents), and a closing "when unsure: MDN the element's 'Technical summary' box". Weak versions re-explain philosophy — the 30-second constraint is the exercise. Writing the guide is how you find out whether you can operationalize what you know.

🎯 Key Takeaways

  • The method: behavior → meaning → nothing. Behavior first because contracts don't retrofit; "nothing" (div/span) is a legitimate terminal answer.
  • Native elements are the cheap option — the contract table's last row (div: nothing, nothing, nothing) is the whole argument, and the CSS to restyle is trivial.
  • The URL rule decides button-vs-link; the stretched-link pattern solves clickable cards; aria-label + hidden icon solves icon buttons; real link in first cell solves clickable rows.
  • Outline entries and text sizes are independent axes — still the most-violated rule in production.
  • Argue semantics as costs (blocked users, legal exposure, maintenance) not ideology — and fix at the system level (component APIs, lint) so it stays fixed.
  • ARIA patches announcements, not behavior — it's the fallback, never the plan.
