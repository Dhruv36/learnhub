Production Patterns: SSR & Hydration Ultra-Advanced

One question defines modern web architecture: where is the HTML generated? Build server, app server, edge, or browser — each answer trades freshness, speed, SEO, and infrastructure cost differently. This is a guaranteed senior-frontend interview topic and the capstone of this track.

The Four Rendering Strategies

SSG (static generation)Once, at build timeFastest possible (CDN-served files), cheap, unhackable-ishContent frozen until rebuild; build time grows with pagesDocs, blogs, marketing — this site
SSR (server rendering)Per requestAlways fresh, personalized, full SEOServer fleet + TTFB per request; caching complexityE-commerce, news, feeds
CSR (client rendering)In the browser via JSApp-like interactivity; server = static shell + APIsBlank-until-JS first load; weak SEO; big bundlesDashboards & tools behind login
ISR / revalidationBuild + regenerate per-page on a timer/on-demandStatic speed with periodic freshnessStaleness window; platform-specific semanticsLarge catalogs (millions of product pages)

Hydration — Making Server HTML Come Alive

1. Server sends complete HTML         → user SEES content fast (great LCP) …but it's inert
2. Browser downloads the JS bundle
3. Framework re-builds its component tree and ATTACHES listeners to existing DOM
4. Page becomes interactive           → the 1→4 gap is the "uncanny valley"

The classic failure: user clicks during the gap — nothing happens. On low-end phones
with big bundles this gap is seconds. Hydration cost ≈ proportional to JS shipped.

Hydration Mismatch — the Bug You WILL Meet

// Server rendered:  <p>Last updated 14:02:31</p>      (server clock)
// Client hydrates:  new Date() differs → React: "Hydration failed" warnings,
//                   content flicker, or full client re-render.
Causes: timestamps, locale formatting, Math.random IDs, browser-only APIs during render.
Fixes:  render deterministic content on both sides; defer non-deterministic bits to
        a post-hydration effect; or mark them client-only.

The Decision Framework (say this in interviews)

Same content for everyone?                → SSG (+ CDN, done)
Public + fresh/personalized, SEO matters? → SSR with streaming; cache aggressively
Behind login, tool-like, SEO irrelevant?  → CSR is legitimate — stop over-engineering it
Millions of mostly-static pages?          → ISR / on-demand revalidation
Real production answer: MIX PER ROUTE —
  /            marketing   → SSG
  /products/*  catalog     → ISR
  /feed        personalized→ streaming SSR
  /admin       dashboard   → CSR

The Cost Dimension (staff-level framing)

Rendering strategy is an infrastructure bill, not just UX: SSR at 300K QPS means a compute fleet rendering HTML on every request; the same traffic on SSG is CDN cache hits at near-zero marginal cost. Real systems put a CDN in front of SSR with Cache-Control: s-maxage=60, stale-while-revalidate — collapsing thousands of renders into one per minute. Connect this to the System Design track's caching and estimation lessons and you're answering at the level interviewers rarely hear.

Interview Questions

Q1: Why not SSR everything, always?

Costs: per-request compute, higher TTFB than CDN'd static, cache complexity for personalized content, and hydration still ships the full JS anyway. For logged-in tools SEO is irrelevant and CSR's simplicity wins; for static content SSG dominates. SSR earns its cost only where freshness + SEO + personalization coincide.

Q2: What EXACTLY happens during hydration, and why is it expensive?

The framework re-executes component code to rebuild its virtual tree, walks the existing DOM to match nodes, attaches event listeners, and wires state. CPU-bound and proportional to bundle size — which is why every mitigation (islands, RSC, resumability) is fundamentally "run less JS on the client".

Q3: How do Server Components differ from SSR?

SSR renders the first paint on the server, then ships all component JS for hydration. Server Components never ship their code — they render to a serialized format on every relevant request, composing with client components that alone hydrate. Result: bundle shrinks by whatever logic stays server-side (data fetching, markdown rendering, heavy libs).

🏋️ Practice:
1. Classify five sites you use daily (view-source: is content in the HTML or rendered by JS?) — bucket each into SSG/SSR/CSR/ISR and justify.
2. Design the per-route rendering map for an e-commerce startup (home, category, product, cart, account) with a one-line reason per route.
3. Explain (out loud, interview-style) why a timestamp component causes hydration mismatch and two ways to fix it.

🎯 Key Takeaways

  • SSG = build-time, SSR = request-time, CSR = browser, ISR = static-with-revalidation; mix per route.
  • Hydration attaches behavior to server HTML; its cost scales with shipped JS — the gap is the UX risk.
  • Streaming, islands, Server Components, resumability: different routes to "less client JS".
  • Deterministic render output or client-only guards prevent hydration mismatches.
  • Think in three axes at once: user experience, SEO, and infrastructure cost.
