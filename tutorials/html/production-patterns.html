Production Patterns: SSR & Hydration Ultra-Advanced

Where does the HTML actually come from? That one question defines modern web architecture — and it's a guaranteed senior frontend interview topic.

The Four Rendering Strategies

StrategyHTML generated…Trade-offUsed by
SSG (static)At build timeFastest possible; content frozen until rebuildDocs, blogs, this site
SSR (server)Per requestFresh + SEO-friendly; server cost, TTFB depends on backendE-commerce, news
CSR (client)In the browser by JSRich interactivity; blank first paint, weak SEODashboards, tools behind login
ISR (incremental)Build + revalidate per-pageStatic speed with periodic freshnessLarge catalogs (Vercel pattern)

Hydration — Making Server HTML Interactive

1. Server sends complete HTML     → user SEES the page fast (good LCP)
2. Browser downloads the JS bundle
3. Framework "hydrates": attaches listeners onto the existing DOM
4. Page becomes INTERACTIVE       → gap between 1 and 4 = "uncanny valley"

The classic failure: page looks ready, user clicks, nothing happens — hydration hasn't finished. Frameworks attack this gap differently:

TechniqueIdeaWhere
Streaming SSRSend HTML in chunks as it renders; show shell immediatelyReact 18 Suspense, Next.js
IslandsOnly interactive widgets get JS; static HTML stays staticAstro
Server ComponentsComponent code that never ships to the client at allReact/Next.js App Router
ResumabilitySerialize state into HTML; skip re-execution entirelyQwik

Picking a Strategy (the interview answer)

Is the content the same for everyone?        → SSG
Public + personalized/fresh (SEO matters)?   → SSR (+ streaming)
Behind a login, highly interactive?          → CSR is fine — SEO is irrelevant there
Huge catalog, mostly-static pages?           → ISR
Real answer at scale: MIX per route — marketing SSG, product SSR, dashboard CSR.
💡 Staff-level insight: rendering strategy is a cost decision as much as a UX one. SSR at 300K QPS means a compute fleet rendering HTML; SSG+CDN serves the same traffic from edge caches at ~zero marginal cost. That framing — performance, SEO, and infrastructure spend — is what distinguishes staff-level answers. (Ties directly into the System Design track's caching and estimation lessons.)

Interview Follow-ups to Expect

• Why can hydration mismatch errors happen? (server rendered different HTML than client expects — timestamps, locale, random IDs)
• How do you SSR personalized content on a CDN? (edge rendering / cache-key on user segment / SSG shell + client-fetch for the personal bits)
• What does "islands architecture" optimize? (shipping less JS — the metric that dominates INP)
