Big-O & Complexity Analysis Foundation

Concept: Big-O describes how an algorithm's time/space cost GROWS as input size n grows — not the exact runtime. It's the language interviewers use to evaluate your solution, and the tool YOU use to recognize when a brute-force approach won't pass (and which pattern will).

The Complexity Ladder (memorize the order)

🎯 The single most useful interview skill: read the CONSTRAINTS, infer the required complexity. n ≤ 10 → exponential is fine (try all subsets/permutations). n ≤ 1,000 → O(n²) is fine. n ≤ 10⁵–10⁶ → need O(n log n) or O(n). n ≤ 10⁹ or "find the answer" with huge bounds → need O(log n), usually binary search on the answer. This single habit tells you WHICH pattern to reach for before you write a line of code.

Reading Complexity Off Code

for (let i = 0; i < n; i++) { ... }                    // O(n)
for (let i = 0; i < n; i++)                             // O(n²) — nested, same n
  for (let j = 0; j < n; j++) { ... }
for (let i = 0; i < n; i *= 2) { ... }                  // O(log n) — i DOUBLES each step
for (let i = 0; i < n; i++)                              // O(n log n) — inner loop is log n
  for (let j = 1; j < n; j *= 2) { ... }
function fib(n) { return n <= 1 ? n : fib(n-1) + fib(n-2); }   // O(2ⁿ) — TWO recursive calls, no memo

Amortized Complexity — a Common Interview Trap

// A dynamic array's push() is usually O(1), but OCCASIONALLY O(n) when it must resize/copy.
// AMORTIZED over many pushes, it averages to O(1) — because resizes happen exponentially less often
// (double the capacity each time), so the total copying cost across n pushes is O(n), not O(n²).
arr.push(x);   // "O(1) amortized" — the correct, precise answer in an interview

Space Complexity Matters Too

// Recursion has HIDDEN space cost: the call stack.
function sumRec(arr, i) {
  if (i === arr.length) return 0;
  return arr[i] + sumRec(arr, i + 1);   // O(n) call stack depth = O(n) SPACE, even with O(1) locals
}
// A common interview follow-up: "can you do this iteratively to get O(1) space?"

Interview Questions

Q1: What's the time complexity of this code, and why? 
function hasDuplicate(arr) {
  for (let i = 0; i < arr.length; i++) {
    if (arr.indexOf(arr[i], i + 1) !== -1) return true;
  }
  return false;
}

O(n²). The outer loop runs n times, but arr.indexOf() inside it is ITSELF an O(n) linear scan — so the total work is n × n = O(n²), even though there's only one explicit `for` loop written in the code. This is the single most common way candidates misjudge complexity: they count loops, not the cost of what's called INSIDE them. The fix (O(n) time, O(n) space) is a hash set: track seen values as you scan once, turning each membership check into O(1).

Q2: Why is dynamic array push() considered O(1) "amortized" rather than just O(1)?

A dynamic array (like a JS array or Python list) is backed by a fixed-size buffer. Most push() calls are genuinely O(1) — there's room, so it just writes to the next slot. But when the buffer is full, push() must allocate a NEW, larger buffer (typically double the size) and copy every existing element into it — an O(n) operation for that one call. Doubling means this expensive resize happens at pushes 1, 2, 4, 8, 16... — exponentially less frequently. Summing the copying cost across n total pushes (1+2+4+...+n/2 ≈ n) gives O(n) total copying work spread across n pushes, which AMORTIZES to O(1) per push on average, even though any INDIVIDUAL push could be O(n) in the worst case. This distinction matters because a naive worst-case analysis ("the last push could be O(n)!") gives the wrong intuition about real performance.

Q3: Given constraints "1 ≤ n ≤ 10⁵", what complexity should you target, and why?

Competitive programming and interview judges typically allow roughly 10⁸ simple operations per second. With n = 10⁵, an O(n²) solution performs about 10¹⁰ operations — far too slow, will time out. An O(n log n) solution performs about 10⁵ × 17 ≈ 1.7×10⁶ operations — comfortably fast. So the constraint n ≤ 10⁵ is a strong signal the intended solution is O(n log n) (commonly: sort the input, or use a heap/balanced tree) or better (O(n) with a hash map or two pointers). This is exactly why reading constraints FIRST, before writing any code, is a core interview skill — it tells you which family of patterns (brute force / O(n²) two-loop / O(n log n) sort-based / O(n) linear-scan / O(log n) binary-search-on-answer) the problem is actually expecting, often before you've even fully worked out the algorithm.

🏋️ Practice: for five LeetCode "Easy" problems, write down your solution's complexity BEFORE looking at the editorial, then verify against the discussion; time an O(n²) vs O(n log n) solution on n=100,000 random integers and compare real elapsed time; identify the amortized cost of a hash map insertion (why isn't collision handling always O(1)?); rewrite a recursive Fibonacci as iterative and compare space complexity.

🎯 Key Takeaways

  • Big-O describes growth rate, not exact time — memorize the ladder: O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!).
  • Read the CONSTRAINTS first — they tell you which complexity (and therefore which pattern) is expected before you write code.
  • Count the cost of what's INSIDE a loop, not just the number of loops — a hidden linear operation turns O(n) into O(n²).
  • Amortized complexity (dynamic array push) averages occasional expensive operations across many cheap ones.
  • Recursion has hidden O(depth) space cost from the call stack, even with no explicit data structures.
