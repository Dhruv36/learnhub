Union-Find Trees & Graphs

Concept: Also called Disjoint Set Union (DSU). Tracks a collection of elements partitioned into disjoint groups, supporting two operations — "are these two in the same group?" and "merge these two groups" — in NEAR O(1) time each, with two specific optimizations that make the naive O(n) version fast.

Recognition Cues

🔍 "Number of connected components" • "are these two nodes connected" (asked REPEATEDLY, dynamically, as edges are added) • "redundant connection" (find the edge that creates a cycle) • "accounts merge" / "friend circles" • grid problems asking about connected regions where DFS/BFS would also work but Union-Find is often simpler for INCREMENTAL connectivity.

The Two Core Operations

class UnionFind {
  constructor(n) {
    this.parent = Array.from({length: n}, (_, i) => i);   // everyone starts as their OWN parent (own group)
    this.rank = new Array(n).fill(0);                      // tracks approximate tree height, for union by rank
  }

  find(x) {                                                 // returns the ROOT (representative) of x's group
    if (this.parent[x] !== x) this.parent[x] = this.find(this.parent[x]);   // PATH COMPRESSION
    return this.parent[x];
  }

  union(x, y) {
    const rootX = this.find(x), rootY = this.find(y);
    if (rootX === rootY) return false;                      // already in the same group
    if (this.rank[rootX] < this.rank[rootY]) [rootX, rootY] = [rootY, rootX];   // UNION BY RANK
    this.parent[rootY] = rootX;
    if (this.rank[rootX] === this.rank[rootY]) this.rank[rootX]++;
    return true;
  }
}

Path Compression — the First Optimization

// WITHOUT path compression, find() can degrade to O(n) — a long chain of parent pointers means
// walking all the way up for EVERY call, every time.
// WITH path compression: this.parent[x] = this.find(this.parent[x]) — every node visited during a
// find() gets REWIRED to point DIRECTLY at the root, flattening the tree for all FUTURE lookups.
// After enough calls, nearly every node points straight at its root — find() approaches O(1).
🎯 Path compression is a one-line change with an outsized effect. The recursive line `this.parent[x] = this.find(this.parent[x])` does two things at once: it computes the root (via recursion) AND permanently shortcuts x's parent pointer to point directly at that root — so the NEXT find(x) call is O(1) instead of retracing the same chain. This is why it's often called a "self-optimizing" data structure.

Union by Rank — the Second Optimization

// WITHOUT union by rank, you could accidentally attach a TALL tree under a SHORT one's root, making
// the tree even taller and slower for future find() calls.
// WITH union by rank: always attach the SHORTER tree under the TALLER tree's root — this keeps the
// combined tree's height growing as slowly as possible (roughly log n in the worst case, even before
// path compression kicks in).
// Together, path compression + union by rank give amortized time per operation of O(α(n)) — the
// INVERSE ACKERMANN function, which is ≤ 5 for any n you could ever practically encounter — "essentially
// O(1)" is the correct interview-level characterization.

Number of Connected Components — the Canonical Use

function countComponents(n, edges) {
  const uf = new UnionFind(n);
  let components = n;                              // start: everyone is their own component
  for (const [a, b] of edges) {
    if (uf.union(a, b)) components--;                // successful union means TWO groups became ONE
  }
  return components;
}
// Simpler and often faster in practice than DFS/BFS for problems that are fundamentally about
// counting/merging groups rather than needing full traversal paths through the graph.

Interview Questions

Q1: Explain path compression and why it makes find() operations nearly O(1) over time.

Without any optimization, the disjoint-set structure is essentially a forest of trees where union() can attach one tree under another arbitrarily, potentially creating a long, chain-like tree (in the worst case, a straight line of n nodes). find() on such a structure has to walk all the way from a node up through every ancestor to reach the root — an O(n) walk in the worst case, and this cost would be paid AGAIN on every future find() call for any node in that same long chain, since nothing about the structure changes just from looking it up. Path compression fixes this by making find() itself modify the structure as a side effect: the recursive step `this.parent[x] = this.find(this.parent[x])` computes x's root via recursion as normal, but then immediately REWIRES x's parent pointer to point DIRECTLY at that root, rather than leaving it pointing at its original (possibly distant) immediate parent. This happens for every node along the path traversed during that find() call, so a single find() operation flattens the ENTIRE chain it walks through — every node on that path now points straight at the root. The next time find() is called on any of those now-flattened nodes, it resolves in a single step instead of retracing the whole original chain. Over many operations, the tree structure becomes increasingly flat, and the amortized cost per find() call approaches constant time — this "self-flattening" behavior is what makes path compression such a high-leverage, low-code optimization.

Q2: What is union by rank, and why does combining it with path compression give such a strong complexity guarantee?

Union by rank addresses a different failure mode than path compression alone would fully solve: when merging two groups (two trees) via union(), you have a choice of which root becomes the new overall root — attaching tree A under tree B's root, or vice versa. If you always make this choice arbitrarily (say, always attaching the second argument under the first), you could repeatedly attach a tall tree underneath a short one's root, making the combined tree even taller than either was before — and taller trees mean longer find() walks before compression has a chance to flatten them. "Rank" here is a value tracking (roughly) the height of a tree rooted at each node (not updated perfectly precisely, but a reasonable upper-bound estimate). Union by rank always attaches the tree with the SMALLER rank underneath the tree with the LARGER rank's root, only incrementing the larger tree's rank if the two ranks were equal — this policy provably keeps the resulting tree's height growing logarithmically (O(log n)) in the worst case, even with NO path compression at all, simply because you can only double a tree's effective height by merging two trees of EQUAL height, and you'd need exponentially many merges to keep doing that repeatedly. When union by rank (bounding height growth to O(log n)) is combined WITH path compression (continuously flattening trees toward O(1) as they're accessed), the two optimizations compound: the formal result is that the AMORTIZED cost per operation (across a long sequence of unions and finds) is O(α(n)), where α is the inverse Ackermann function — a function that grows so absurdly slowly that its value is less than 5 for any n you could ever practically encounter (far beyond the number of atoms in the observable universe), which is why "effectively O(1)" is the standard, accurate characterization used in interviews and textbooks alike.

Q3: When would you choose Union-Find over BFS/DFS for a connectivity problem?

Both can answer "are these two nodes connected" and "how many connected components exist," but they have meaningfully different strengths depending on the problem's SHAPE. BFS/DFS is naturally suited to problems where you need to explore or find an actual PATH through the graph, or where the graph is given complete and STATIC upfront and you're doing a single full traversal to answer a one-time question (like counting islands in a fixed grid) — a single BFS/DFS pass costs O(V+E) and answers connectivity for the whole graph at once. Union-Find becomes the clearly better choice when connectivity needs to be queried or updated INCREMENTALLY — for example, edges are being added one at a time over a sequence of operations (as in "Redundant Connection," where you process edges one by one and need to detect the very first edge that creates a cycle, i.e., connects two nodes already in the same component), or when you need to repeatedly ask "are X and Y connected right now" interleaved with adding new edges, without wanting to re-run a full O(V+E) traversal from scratch after every single addition. Union-Find's union() and find() operations are each nearly O(1) (amortized O(α(n))) individually, so processing a whole sequence of m incremental edge additions and connectivity queries costs roughly O(m·α(n)) total — dramatically better than re-traversing the graph from scratch after every update, which would cost O(m·(V+E)). The trade-off is that Union-Find only tracks WHICH group each element belongs to — it doesn't retain or let you reconstruct the actual PATH between two connected nodes the way a BFS/DFS traversal naturally would, so if the problem needs the actual route (not just "are they connected"), BFS/DFS remains the right tool regardless of how the edges arrive.

🎯 Key Takeaways

  • Union-Find (Disjoint Set Union) tracks group membership with two operations — find() and union() — both nearly O(1) amortized with the right optimizations.
  • Path compression flattens the tree during find(), rewiring visited nodes to point directly at the root — makes future lookups faster.
  • Union by rank always attaches the shorter tree under the taller one's root, bounding height growth to O(log n) even without compression.
  • Combined, both optimizations give O(α(n)) amortized time — the inverse Ackermann function, "effectively O(1)" for any practical n.
  • Prefer Union-Find over BFS/DFS specifically for INCREMENTAL connectivity (edges added over time, repeated connectivity queries) — prefer BFS/DFS when you need the actual path.
