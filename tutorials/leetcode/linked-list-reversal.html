Linked List In-place Reversal Core Pattern

Concept: Reverse a linked list (or a sublist) by walking through it once and flipping each node's `next` pointer to point BACKWARD instead of forward — using three tracking pointers (previous, current, next) and O(1) extra space. No new nodes, no array conversion.

Recognition Cues

🔍 "Reverse a linked list" (whole or a sub-range) • "reverse in groups of K" • "palindrome linked list" (reverse the second half, compare) • "reorder list" (reverse + merge) • anywhere you need the LIST ORDER flipped without O(n) extra space for an array.

Template (reverse the whole list)

function reverseList(head) {
  let prev = null, curr = head;
  while (curr) {
    const next = curr.next;       // SAVE the next node before we overwrite curr.next
    curr.next = prev;              // flip the pointer to point BACKWARD
    prev = curr;                   // advance prev
    curr = next;                   // advance curr (using the saved reference)
  }
  return prev;                     // prev is the new head (curr is now null, past the old tail)
}
// The save-before-overwrite order is EVERYTHING: overwrite curr.next before saving it, and you've
// permanently lost the rest of the list — this is the #1 bug in every first attempt at this pattern.
⚠️ The three-pointer dance (prev, curr, next) is the mechanical skeleton for EVERY reversal variant in this pattern. Once this is muscle memory, "reverse between positions m and n," "reverse in groups of k," and "reverse alternating groups" are all just this same loop with different start/stop conditions and reconnection logic at the boundaries.

Reverse a Sublist (positions left..right)

function reverseBetween(head, left, right) {
  const dummy = new ListNode(0, head);              // DUMMY node avoids special-casing "reverse from head"
  let prevGroup = dummy;
  for (let i = 0; i < left - 1; i++) prevGroup = prevGroup.next;   // walk to just BEFORE the sublist

  let curr = prevGroup.next;
  for (let i = 0; i < right - left; i++) {           // classic "insert at front" reversal, (right-left) times
    const moved = curr.next;
    curr.next = moved.next;
    moved.next = prevGroup.next;
    prevGroup.next = moved;
  }
  return dummy.next;
}
// A DUMMY head node is the standard trick that eliminates "what if we're reversing starting at the
// very first node" as a special case — prevGroup.next always has a valid predecessor to update.

Reverse in Groups of K

function reverseKGroup(head, k) {
  let node = head, count = 0;
  while (node && count < k) { node = node.next; count++; }   // check there ARE k nodes left
  if (count < k) return head;                                 // fewer than k remain — leave as-is

  let prev = reverseKGroup(node, k);                          // recursively reverse the REST first
  let curr = head;
  for (let i = 0; i < k; i++) {                                // reverse THIS group of k
    const next = curr.next;
    curr.next = prev;
    prev = curr;
    curr = next;
  }
  return prev;                                                  // new head of THIS reversed group
}
// This recursive version reverses from the END backward: recurse to reverse everything after this
// group FIRST, then reverse this group and attach its tail to the already-reversed rest.

Common Mistakes

MistakeSymptomFix
Overwriting curr.next BEFORE saving itPermanently loses the rest of the list — silent data loss, not a crashAlways: save next FIRST, then reassign curr.next = prev
Not using a dummy head for sublist reversalSpecial-case bugs when the reversal starts at the very first nodePrepend a dummy node; return dummy.next at the end
Losing the connection to the node AFTER a reversed sublistThe reversed section becomes disconnected from the rest of the listTrack and reconnect both the "before" and "after" boundary nodes explicitly
Off-by-one in counting k nodes for group reversalReverses one too many/few nodes, or crashes on a short remaining listExplicitly count k nodes first; bail out cleanly if fewer than k remain
Forgetting reversal is O(1) SPACE — using an array as a shortcutTechnically correct but defeats the point of the pattern (and the interview ask)Solve with pointer manipulation only, no auxiliary array/list

Interview Questions

Q1: Walk through the three-pointer technique for reversing a linked list and explain why the ordering of operations matters.

The algorithm maintains three references as it walks the list once: `prev` (the node that should come BEFORE the current node in the reversed list — starts as null, since the original head will become the new tail with nothing after it), `curr` (the node currently being processed), and a temporary `next` (used only to remember where to continue). At each step, the critical sequencing is: first, save curr.next into the temporary `next` variable — this preserves the link to the rest of the still-unprocessed list BEFORE it's touched. Second, only now reassign curr.next = prev — this is the actual "flip," making the current node point backward instead of forward. Third, advance prev to curr (the just-reversed node is now the new "previous" for the next iteration). Fourth, advance curr to the saved `next` reference to continue with the next unprocessed node. The reason the SAVE must happen before the OVERWRITE is that curr.next is the ONLY reference the algorithm has to the rest of the original list — once you overwrite it with curr.next = prev, if you hadn't already saved the original value elsewhere, that entire remaining portion of the list becomes permanently unreachable (not a crash, just silently lost data, which makes this bug particularly easy to miss without careful testing). This ordering — save, then overwrite, then advance both trackers — is the mechanical skeleton underlying every variant of linked list reversal in this pattern family.

Q2: Why is a "dummy head" node commonly used when reversing a SUBLIST rather than the whole list?

When reversing an entire list, there's no ambiguity about what happens before the reversed section — there IS nothing before it, the reversed list's new head becomes the function's return value directly. But when reversing only a SUBLIST (say, positions left through right within a longer list), you need to correctly reconnect the node immediately BEFORE the sublist to the sublist's new (post-reversal) first node, and this requires having a valid, addressable "predecessor" node to update — except when `left` is 1 (reversing starting from the very first node of the list), there IS no real predecessor node; the sublist starts at the actual head, and there's nothing before it to hold a reference and update. Without special handling, this creates an annoying edge case: the code that walks to "the node before the sublist" and later updates "that node's next pointer" would need a completely separate code path for "the sublist starts at the head" versus "the sublist starts somewhere in the middle." A DUMMY node — an extra placeholder node created before the actual head, whose `next` initially points to the real head — sidesteps this entirely: now EVERY sublist, even one starting at the original head, has a valid predecessor to reference and update (the dummy node, or something reached by walking forward from it), eliminating the special case. At the end, the answer is dummy.next (which correctly reflects whatever the new head actually is, whether or not the original head itself was part of the reversed section) rather than needing to check which case applied.

Q3: Compare the iterative and recursive approaches to "Reverse Nodes in k-Group" — what are the trade-offs?

Both approaches must handle two core sub-problems: reversing exactly k nodes at a time using the standard three-pointer technique, and correctly stitching each reversed group of k nodes to the group before and after it, while leaving any final remaining group with FEWER than k nodes unreversed (per the problem's typical requirement). The RECURSIVE approach (shown above) handles the "process many groups" aspect through recursion: it first checks whether at least k nodes remain from the current position; if not, it returns the list as-is (base case). If there are enough nodes, it recursively calls itself on the node k positions ahead FIRST — meaning "reverse everything after this group" happens before "reverse this group" — and then reverses the current group of k nodes, attaching its tail to whatever the recursive call already returned as the head of the (already fully reversed) rest of the list. This is elegant and mirrors how you'd naturally describe the algorithm in words, but it uses O(n/k) stack depth (one recursive call per group), which for a very long list could risk a stack overflow in languages/environments with limited recursion depth, and interviewers sometimes explicitly ask for the O(1)-space iterative version as a follow-up specifically to test whether a candidate can convert the recursive idea into a loop using an explicit "previous group's tail" pointer that gets updated and reconnected after each group is reversed. The ITERATIVE version achieves the same O(n) time with true O(1) auxiliary space (no call stack growth), at the cost of slightly more intricate bookkeeping to manually track and reconnect each group boundary that the recursion handled implicitly through the call stack. In an interview, presenting the recursive version first (often faster to arrive at and easier to explain) and then discussing the iterative conversion as a space-optimization follow-up is a strong way to demonstrate depth on this pattern.

🏋️ Practice: implement reverseList and trace prev/curr/next through a 4-node example by hand before running it; implement reverseBetween with a dummy head and test reversing starting exactly at position 1; implement reverseKGroup both recursively and iteratively, comparing stack depth conceptually; solve "Palindrome Linked List" using find-the-middle (fast/slow pointers) + reverse the second half + compare; solve "Reorder List" (L0→Ln→L1→Ln-1→...) using reversal plus merging two halves.

🎯 Key Takeaways

  • The three-pointer skeleton (prev, curr, saved-next) reverses a linked list in O(n) time, O(1) space — save curr.next BEFORE overwriting it, or the rest of the list is lost.
  • This exact skeleton is reused, with different boundaries, for every variant: whole-list, sublist, and k-group reversal.
  • A dummy head node eliminates special-casing "the sublist starts at the very first node" when reversing a range.
  • k-group reversal can be done recursively (elegant, O(n/k) stack space) or iteratively (O(1) space, more manual bookkeeping) — know both.
  • This pattern combines with fast/slow pointers (find the middle) for problems like palindrome-check and list reordering.
