Prefix Sum Core Pattern

Concept: Precompute cumulative sums once in O(n), then answer ANY range-sum query in O(1). The pattern generalizes to a hash map of prefix sums for "subarray sums equal to K" problems — arguably the single highest ROI pattern for the "count subarrays matching X" problem family.

Recognition Cues

🔍 "Range sum query" (repeated) • "subarray sum equals K" • "count subarrays with sum divisible by K" • "equilibrium index" / "find a pivot where left sum = right sum" • any problem doing REPEATED sum-of-a-range queries on a STATIC array.

Template (basic prefix array)

const prefix = [0];                              // prefix[i] = sum of arr[0..i-1]
for (const x of arr) prefix.push(prefix[prefix.length - 1] + x);

function rangeSum(l, r) {                          // inclusive [l, r], O(1)
  return prefix[r + 1] - prefix[l];
}
// Why prefix has a leading 0: rangeSum(0, r) becomes prefix[r+1] - prefix[0] = prefix[r+1] - 0,
// avoiding a special case for ranges starting at index 0.

Template (hash map for "subarray sum = K")

function subarraySum(nums, k) {
  const seen = new Map([[0, 1]]);                  // prefix sum 0 has occurred once (the empty prefix)
  let sum = 0, count = 0;
  for (const x of nums) {
    sum += x;
    count += seen.get(sum - k) || 0;                // if (sum-k) occurred before, that gap sums to k
    seen.set(sum, (seen.get(sum) || 0) + 1);
  }
  return count;
}
// Insight: subarray(i+1, j) sums to k  ⟺  prefix[j] − prefix[i] = k  ⟺  prefix[i] = prefix[j] − k.
// So at each j, count how many EARLIER prefix sums equal (current sum − k) — that's O(1) via the map.
🎯 The `seen = new Map([[0, 1]])` initialization is the trap that eliminates most candidates. Without it, a subarray starting at INDEX 0 that itself sums to exactly k is never counted (there's no "prefix before it" to subtract from). Seeding the map with prefix-sum 0 occurring once represents the empty prefix before the array starts — get this wrong and every "subarray from the beginning" case silently under-counts.

2D Prefix Sum (a common follow-up)

// Range sum queries over a 2D grid — precompute once, answer in O(1):
function build2DPrefix(grid) {
  const m = grid.length, n = grid[0].length;
  const P = Array.from({length: m+1}, () => new Array(n+1).fill(0));
  for (let i = 1; i <= m; i++)
    for (let j = 1; j <= n; j++)
      P[i][j] = grid[i-1][j-1] + P[i-1][j] + P[i][j-1] - P[i-1][j-1];   // inclusion-exclusion
  return P;
}
function regionSum(P, r1, c1, r2, c2) {                // inclusive rectangle, O(1)
  return P[r2+1][c2+1] - P[r1][c2+1] - P[r2+1][c1] + P[r1][c1];
}

Interview Questions

Q1: Why must the prefix-sum hash map for "subarray sum equals K" be initialized with {0: 1}?

The algorithm's core insight is that a subarray ending at index j with sum k exists if and only if some earlier prefix sum equals (current prefix sum − k). But consider a subarray that starts at index 0 — there's no "earlier prefix" before it in the array at all; the relevant earlier prefix sum is 0, representing "the sum of nothing," i.e., the empty prefix before any elements. If the map doesn't contain an entry for prefix sum 0 with count 1, then when the running sum EXACTLY equals k at some index j (meaning the subarray from index 0 to j sums to k), the lookup for (sum − k) = 0 would find nothing in the map, and that valid subarray would never be counted. Seeding the map with {0: 1} before the loop begins represents exactly this "empty prefix" case, ensuring subarrays starting at the very beginning of the array are correctly counted just like any other subarray. This is one of the most commonly-missed details in this pattern precisely because it's invisible in test cases that don't happen to include a from-the-start matching subarray.

Q2: How does the prefix-sum hash map approach achieve O(n) time for "count subarrays summing to K," and why is this better than the brute force?

The brute-force approach checks every possible subarray (every pair of start and end indices), computing each subarray's sum — either by re-summing from scratch each time (O(n³) overall) or by precomputing a basic prefix array first so each subarray sum is an O(1) lookup, reducing to O(n²) overall (checking O(n²) start/end pairs). The hash-map approach does a SINGLE pass through the array, maintaining a running prefix sum, and at each position asks "how many EARLIER prefix sums, when subtracted from my current sum, equal k?" — which is exactly the count of subarrays ending at this position that sum to k. Because a hash map gives O(1) average lookup and insertion, each of the n positions does O(1) work (one lookup, one insert), for O(n) total time and O(n) space (the map can hold up to n distinct prefix sums). This is asymptotically faster than both brute-force approaches because it never explicitly enumerates all O(n²) possible subarrays — it counts matches implicitly via the map's aggregated frequency information, turning what looks like an inherently pairwise (quadratic) counting problem into a single linear pass.

Q3: When would you use a basic prefix-sum array versus the hash-map variant?

They solve related but different questions. The basic prefix-sum ARRAY (prefix[i] = sum of the first i elements) is the right tool when you need to answer MULTIPLE explicit range-sum queries — "what's the sum from index l to r?" — repeatedly, on a STATIC (unchanging) array; you precompute the array once in O(n), and then each individual range query costs O(1) via a simple subtraction (prefix[r+1] - prefix[l]), which is ideal when there are many queries (Q of them) since the total cost becomes O(n + Q) instead of O(n·Q) from re-summing each range from scratch. The hash-map variant is the right tool for a fundamentally different question: "how MANY subarrays (or which subarrays) satisfy some sum-based condition" — most commonly "sum equals exactly k" — where you're not given explicit (l, r) query ranges up front but instead need to discover/count all qualifying subarrays yourself in one pass, using the frequency of previously-seen prefix sums to detect matches as you go. In short: array form for "answer many explicit range queries fast," hash-map form for "count/find subarrays matching a sum condition in one pass" — both share the same underlying mathematical idea (sum(i,j) = prefix[j] − prefix[i]) but are applied to different problem shapes.

🏋️ Practice: implement basic prefix sums and answer 5 range-sum queries in O(1) each; implement "subarray sum equals K" with the hash-map approach and verify the {0:1} seed with a subarray starting at index 0; solve "find the equilibrium/pivot index" using prefix and suffix sums; implement 2D prefix sum and answer a few rectangle-sum queries; solve "subarray sums divisible by K" (hint: use prefix sum MODULO K as the hash key).

🎯 Key Takeaways

  • Precompute cumulative sums once (O(n)) to answer any range-sum query in O(1) — the core trade-off of the pattern.
  • For "count subarrays with sum = K," use a hash map of prefix-sum frequencies, seeded with {0: 1} for subarrays starting at index 0.
  • sum(i, j) = prefix[j] − prefix[i] is the key algebraic identity underlying both the array and hash-map forms.
  • 2D prefix sums extend the idea via inclusion-exclusion for O(1) rectangle-sum queries after an O(mn) precompute.
  • Prefix sum arrays are for STATIC data — frequent updates need a Fenwick Tree/Segment Tree instead.
