Trie Advanced Pattern

Concept: A tree where each path from the root spells out a string, and nodes SHARE common prefixes. Insert, search, and prefix-search all cost O(L) — the length of the word — completely INDEPENDENT of how many words are stored. This is the structural reason tries beat hash sets for prefix-based operations.

Recognition Cues

🔍 "Autocomplete" / "type-ahead suggestions" • "does any word START WITH this prefix" • "implement a dictionary with prefix search" • Word Search II (multiple words in a grid) • "longest word built from other words" • anywhere PREFIX relationships between many strings matter, not just exact-match lookup.

Template (Trie Node & Core Operations)

class TrieNode {
  constructor() {
    this.children = new Map();          // char → TrieNode
    this.isEndOfWord = false;             // marks "a complete word ends HERE" (not just any node)
  }
}

class Trie {
  constructor() { this.root = new TrieNode(); }

  insert(word) {
    let node = this.root;
    for (const ch of word) {
      if (!node.children.has(ch)) node.children.set(ch, new TrieNode());
      node = node.children.get(ch);
    }
    node.isEndOfWord = true;              // O(L) — L = word length
  }

  search(word) {                          // exact match
    const node = this._traverse(word);
    return node !== null && node.isEndOfWord;   // must be BOTH reachable AND marked as a complete word
  }

  startsWith(prefix) {                    // prefix match — the operation a hash set CAN'T do efficiently
    return this._traverse(prefix) !== null;
  }

  _traverse(str) {
    let node = this.root;
    for (const ch of str) {
      if (!node.children.has(ch)) return null;
      node = node.children.get(ch);
    }
    return node;
  }
}
🎯 The isEndOfWord flag is the detail that trips up most first attempts. Without it, search("app") would incorrectly return true just because "app" is a PREFIX of "apple" (the path exists in the tree) — even if "app" itself was never inserted as its own complete word. Reaching a node via a valid path only tells you the STRING is a prefix of something stored; isEndOfWord tells you it's an actual COMPLETE entry.

Why O(L), Not O(n) — the Core Efficiency Argument

// A hash SET of n words: checking "does any word start with 'app'?" requires SCANNING ALL n words —
// O(n × L) in the worst case, since you can't efficiently query by prefix, only by exact match.
//
// A TRIE with n words (total length N across all words): startsWith("app") walks EXACTLY 3 nodes
// (one per character), regardless of whether there are 10 words or 10 million words stored — O(L),
// where L is the QUERY's length, completely independent of n.
// This is the trie's entire value proposition: shared prefixes mean the STRUCTURE itself encodes
// "which words share this prefix," so you never need to check words individually.

Word Search II — Trie + Backtracking Combined

// Finding MULTIPLE target words in a grid (Word Search II) combines two patterns from this track:
// build a TRIE of all target words, then BACKTRACK through the grid, but PRUNE any path the moment
// it no longer matches ANY prefix in the trie — instead of independently searching for each word
// (which would re-scan the grid once per word), the shared trie structure prunes ALL non-matching
// words simultaneously as you explore.
function findWords(board, words) {
  const trie = new Trie();
  for (const w of words) trie.insert(w);
  const found = new Set();

  function dfs(r, c, node, path) {
    if (r < 0 || r >= board.length || c < 0 || c >= board[0].length) return;
    const ch = board[r][c];
    if (ch === "#" || !node.children.has(ch)) return;    // PRUNE — not a valid prefix continuation
    const next = node.children.get(ch);
    path += ch;
    if (next.isEndOfWord) found.add(path);
    board[r][c] = "#";                                     // mark visited
    for (const [dr, dc] of [[1,0],[-1,0],[0,1],[0,-1]]) dfs(r+dr, c+dc, next, path);
    board[r][c] = ch;                                       // un-mark (backtrack)
  }
  for (let r = 0; r < board.length; r++)
    for (let c = 0; c < board[0].length; c++) dfs(r, c, trie.root, "");
  return [...found];
}

Interview Questions

Q1: Why does a Trie answer "does any word start with this prefix" in O(L) time, while a hash set of the same words cannot do this efficiently?

A hash set stores each word as an independent, opaque key — hashing gives O(1) average lookup for an EXACT match, but the hash of a string reveals nothing about which OTHER strings share a prefix with it (two strings with a common prefix can hash to completely unrelated buckets). So to answer "does any stored word start with 'app'," a hash set fundamentally has no better option than checking every single stored word individually to see if IT happens to start with "app" — an O(n × L) operation in the worst case (n words, each comparison potentially costing up to L character checks). A trie structurally encodes prefix relationships directly in its shape: every word that shares the prefix "app" necessarily passes through the SAME sequence of three nodes (for 'a', then 'p', then 'p') on its way from the root, because trie construction explicitly reuses existing nodes for any prefix already present from a previously-inserted word. This means answering "does any word start with 'app'" reduces to simply walking three nodes deep from the root, following the specific characters 'a', 'p', 'p' — a fixed-cost walk whose length depends ONLY on the length of the query string itself (L), completely independent of how many total words (n) are stored in the trie, whether that's ten or ten million. This asymptotic independence from n — trading some memory overhead for shared prefix nodes — is the trie's entire reason for existing over a plain hash set whenever prefix-based queries (not just exact-match lookups) are actually needed.

Q2: Why is the isEndOfWord flag necessary — why can't you just check whether a path exists in the trie?

A trie's structure inherently represents PREFIXES, not just complete words — every node reachable from the root, at any depth, represents SOME valid prefix of at least one word that was inserted. Consider inserting only the single word "apple": this creates a path through nodes for 'a', 'p', 'p', 'l', 'e'. If search() merely checked "does a path exist for this exact string," then calling search("app") would successfully walk to the node for the third character, find that the path DOES exist (since "app" is a legitimate prefix along the route to "apple"), and incorrectly report that "app" was found as a complete word — even though "app" itself was never inserted as its own word, only "apple" was. The isEndOfWord boolean flag solves this by marking SPECIFICALLY which nodes represent the END of an actual, complete, inserted word — set to true only on the final node reached during an insert() call for that particular word, left false on every other node (including nodes that are merely intermediate prefixes on the way to a longer word). search() must then check BOTH conditions: that the path for the query string exists in the trie AND that the node reached at the end of that path has isEndOfWord set to true — reaching a valid node alone only proves the query string is a prefix of SOMETHING stored, not that it was itself ever inserted as a complete entry. This is precisely why search() and startsWith() have different implementations despite both starting with the same path-traversal logic: startsWith() only needs the path to exist (any prefix relationship counts), while search() additionally requires isEndOfWord to distinguish "this exact word was inserted" from "this happens to be a prefix of something longer that was inserted."

Q3: How does combining a Trie with backtracking in "Word Search II" avoid re-searching the grid separately for each target word?

A naive approach to finding multiple target words in a grid would run a SEPARATE backtracking search for each individual word — for m target words, this means m independent full traversals of the grid, each one checking at every step "does the current path still match THIS specific word" — with no sharing of work between searches for different words, even when those words happen to share a common prefix (like "cat" and "car," which are identical for their first two characters). The trie-based approach instead builds ONE trie containing ALL target words upfront (an O(total characters across all words) one-time cost), and then performs a SINGLE combined DFS/backtracking traversal of the grid, where instead of tracking progress against one specific target word, it tracks its CURRENT POSITION within the shared trie structure. At each grid cell during the DFS, the algorithm checks whether the current trie node has a CHILD matching the letter at that grid cell — if "cat" and "car" share the trie path for 'c' then 'a', a single step of the grid traversal simultaneously makes progress toward BOTH words at once, without needing to separately verify each one. Whenever the current trie node's isEndOfWord flag is true, that means a complete target word has just been found ending at the current grid position, and it's added to the results — the algorithm continues exploring further from that same point in case a LONGER target word also happens to continue along the same path (e.g., if both "cat" and "cats" were in the target word list). Crucially, PRUNING also benefits from this sharing: the moment a grid path no longer corresponds to any valid trie continuation (node.children.has(currentLetter) is false), that entire branch is abandoned immediately — and because the trie merges shared prefixes from ALL target words into one structure, this single pruning check simultaneously rules out every remaining word that would have required this same invalid continuation, rather than needing m separate pruning checks for m separate word-specific searches.

🎯 Key Takeaways

  • A trie stores strings as shared paths from the root — insert/search/startsWith all cost O(L), the QUERY length, independent of how many words (n) are stored.
  • This is the key advantage over a hash set, which can't answer prefix queries without scanning every stored word — O(n×L) versus the trie's O(L).
  • isEndOfWord distinguishes "this is a valid prefix of something stored" from "this exact string was itself inserted as a complete word" — the most commonly missed detail.
  • Use a Map for children when the alphabet is large/unknown; a fixed-size array only for known small alphabets (like lowercase English).
  • Combining a trie with backtracking (Word Search II) lets multiple target words share pruning during a SINGLE traversal, instead of independently re-searching per word.
