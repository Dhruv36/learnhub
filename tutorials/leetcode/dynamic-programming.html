Dynamic Programming Advanced Pattern

Concept: When a problem has OVERLAPPING SUBPROBLEMS (the same smaller computation needed repeatedly) and OPTIMAL SUBSTRUCTURE (the optimal answer is built from optimal answers to subproblems), cache each subproblem's result so it's computed once instead of exponentially many times. DP is "recursion with memory," not a separate way of thinking about problems.

Recognition Cues

🔍 "Count the number of ways to..." • "minimum/maximum cost to..." • "can you reach/achieve X" • Fibonacci-like recurrences • knapsack-style "include or exclude each item" • longest common/increasing subsequence • edit distance. The tell: a brute-force recursive solution exists but is exponential because it RECOMPUTES the same subproblem many times.

The Universal DP Recipe (four questions, in order)

// 1. What does dp[i] (or dp[i][j]) REPRESENT? Define this in a full sentence before writing code.
// 2. What's the BASE CASE — the smallest subproblem you can answer directly?
// 3. What's the RECURRENCE — how does dp[i] relate to smaller/earlier dp values?
// 4. What's the FINAL ANSWER — often dp[n], but sometimes max/sum over the whole dp array.
// Skipping step 1 (a precise definition of what dp[i] MEANS) is why most DP attempts fail — the
// recurrence in step 3 is impossible to get right if you don't know exactly what you're computing.

Top-Down (Memoization) vs Bottom-Up (Tabulation)

// TOP-DOWN: write the natural recursive solution, add a cache. Closer to how you'd first THINK
// about the problem — start here when designing a new DP solution.
function fib(n, memo = new Map()) {
  if (n <= 1) return n;
  if (memo.has(n)) return memo.get(n);                  // the ENTIRE optimization: don't recompute
  const result = fib(n - 1, memo) + fib(n - 2, memo);
  memo.set(n, result);
  return result;
}

// BOTTOM-UP: build the answer iteratively from the base case UP. No recursion/call-stack overhead,
// often easier to further optimize for space.
function fibBottomUp(n) {
  if (n <= 1) return n;
  const dp = new Array(n + 1);
  dp[0] = 0; dp[1] = 1;
  for (let i = 2; i <= n; i++) dp[i] = dp[i - 1] + dp[i - 2];
  return dp[n];
}
// Both are O(n) time, O(n) space — a MASSIVE improvement over the naive recursive fib's O(2ⁿ), which
// recomputes fib(k) an exponential number of times for every k.
🎯 Memoization is the entire idea — everything else is implementation detail. The naive recursive Fibonacci is exponential SOLELY because fib(5) calls fib(3) which ALSO gets called (again, redundantly) via fib(4)'s own call to fib(3) — the same subproblem, recomputed from scratch every time it's needed. A cache checking "have I already solved this exact subproblem?" before recomputing is the ENTIRE transformation from exponential to polynomial.

0/1 Knapsack — the Canonical 2D DP

function knapsack(weights, values, capacity) {
  const n = weights.length;
  const dp = Array.from({length: n + 1}, () => new Array(capacity + 1).fill(0));
  for (let i = 1; i <= n; i++) {
    for (let w = 0; w <= capacity; w++) {
      dp[i][w] = dp[i - 1][w];                            // OPTION 1: don't take item i
      if (weights[i - 1] <= w) {
        dp[i][w] = Math.max(dp[i][w], dp[i - 1][w - weights[i - 1]] + values[i - 1]);   // OPTION 2: take it
      }
    }
  }
  return dp[n][capacity];
}
// dp[i][w] = "the best value achievable using the first i items, with capacity w." The recurrence is
// exactly the include/exclude decision for item i — this SAME include/exclude skeleton solves an
// enormous family of DP problems (subset sum, partition equal subset, coin change variants).

Space Optimization — the Common Follow-Up

// Notice: dp[i][w] only ever depends on dp[i-1][...] — the PREVIOUS row, never anything older.
// So you don't need to keep ALL n rows — collapse to a single 1D array, updated in place:
function knapsackOptimized(weights, values, capacity) {
  const dp = new Array(capacity + 1).fill(0);
  for (let i = 0; i < weights.length; i++) {
    for (let w = capacity; w >= weights[i]; w--) {         // iterate BACKWARD — critical!
      dp[w] = Math.max(dp[w], dp[w - weights[i]] + values[i]);
    }
  }
  return dp[capacity];
}
// Iterating w BACKWARD when collapsing to 1D is essential: it ensures dp[w - weights[i]] still holds
// the PREVIOUS row's value (item i not yet applied) when it's read — forward iteration would read an
// ALREADY-UPDATED value from the CURRENT row, incorrectly reusing item i more than once.

Interview Questions

Q1: What are "overlapping subproblems" and "optimal substructure," and why does DP require both?

Optimal substructure means the optimal solution to the overall problem can be constructed directly from optimal solutions to its smaller subproblems — for example, the shortest path from A to C through some intermediate point B is the shortest path from A to B PLUS the shortest path from B to C; you don't need to consider non-optimal sub-paths to build the optimal overall path. This property alone justifies writing a RECURSIVE solution, but doesn't by itself make dynamic programming beneficial — a plain recursive divide-and-conquer algorithm (like merge sort) also has optimal substructure without needing memoization, because its subproblems never repeat. Overlapping subproblems is the SECOND, independently necessary property: it means the recursive solution ends up solving the EXACT SAME subproblem multiple times as it explores different branches — like how naive recursive Fibonacci computes fib(3) both directly and as part of computing fib(4), which is itself needed for fib(5). When BOTH properties hold together, memoization becomes powerful: optimal substructure guarantees a correct recursive formulation exists, and overlapping subproblems means caching each subproblem's result the first time it's computed saves real, repeated work on every subsequent occurrence — turning what would otherwise be exponential re-computation into work proportional to the NUMBER of distinct subproblems (often polynomial). A problem with optimal substructure but NO overlapping subproblems (like merge sort) doesn't benefit from memoization at all, since there's nothing to cache — each subproblem is only ever solved once regardless.

Q2: Compare top-down memoization and bottom-up tabulation — what are the practical trade-offs?

Top-down memoization starts from the NATURAL recursive formulation of the problem — you write the recursion exactly as you'd first think about it (base case, recursive calls expressing the recurrence relation), then add a cache (a map or array) that's checked at the start of each call and populated before returning, so any subproblem is computed at most once. This is usually the EASIER starting point when first designing a DP solution, because it mirrors how you naturally reason about the recursive structure of the problem, and it only computes subproblems that are ACTUALLY needed for the specific input (lazy evaluation) — if some subproblems in the theoretical full table are never reached by the recursion for a given input, they're simply never computed, which can save real work for certain problem shapes. The cost is recursive call overhead and stack depth, which can matter for very large inputs (risking a stack overflow) and typically has slightly more per-call overhead than a tight iterative loop. Bottom-up tabulation instead builds the answer ITERATIVELY, starting from the base case(s) and filling in a table (array) in an order that guarantees every value a given cell depends on has already been computed by the time it's needed. This avoids recursion entirely (no stack depth concerns, generally faster in practice due to lower per-iteration overhead) and makes SPACE OPTIMIZATION much more natural to reason about and apply (since you can often see directly that a row only depends on the previous row, and collapse the table's dimensionality accordingly, as shown in the knapsack space-optimization example). The trade-off is that bottom-up requires you to have already fully worked out the right ITERATION ORDER (which indices to fill in first) — sometimes genuinely trickier to get right than expressing the same idea recursively — and it computes every cell in the table regardless of whether the specific input actually needs all of them.

Q3: In the space-optimized 0/1 knapsack solution, why must the capacity loop iterate BACKWARD instead of forward?

The full 2D version's recurrence for a given item i reads dp[i][w] = max(dp[i-1][w], dp[i-1][w - weight] + value) — critically, BOTH terms on the right-hand side reference row i-1 (the state BEFORE item i was considered), never row i itself. When collapsing the 2D table down to a single 1D array to save space, that single array is being reused and updated in place to represent "the current row" as you process each item — but this creates a risk: if you update dp[w] while processing item i by iterating w from LOW to HIGH (forward), then by the time you reach some larger capacity w' and compute dp[w' - weight], that value might have ALREADY been updated earlier in this SAME pass through item i (if w' - weight was a smaller index already visited during this same forward iteration) — meaning you'd be reading a value that ALREADY includes item i's contribution, and adding item i's value to it a second time, effectively allowing item i to be used TWICE (or more) in the same solution. This is precisely wrong for 0/1 knapsack, where each item may be used AT MOST ONCE. Iterating the capacity dimension BACKWARD (from high to low) fixes this: when computing dp[w] for a given (large) w, all the SMALLER indices dp[w - weight] that this computation depends on haven't been touched YET during this item's pass (since we're going from high to low, we haven't reached them yet) — so they still hold their values from the PREVIOUS row (before item i), which is exactly what the correct 2D recurrence requires. Interestingly, this exact bug (reading an already-updated value within the same pass) is actually the DESIRED behavior for the UNBOUNDED knapsack variant, where items CAN be reused unlimited times — that variant's space-optimized solution iterates FORWARD precisely because allowing a just-updated value to be read again is what correctly models "this item is available for reuse."

🏋️ Practice: implement Fibonacci three ways (naive recursive, top-down memoized, bottom-up) and time the naive version's exponential blowup past n=35; solve 0/1 Knapsack with the full 2D table, then convert to the space-optimized 1D version and verify backward iteration is necessary by deliberately breaking it forward; solve "Climbing Stairs" (a Fibonacci-shaped DP) and "House Robber" (include/exclude DP); solve "Longest Common Subsequence" (2D DP over two strings) and "Coin Change" (minimum coins, an unbounded-knapsack variant using forward iteration).

🎯 Key Takeaways

  • DP applies when a problem has BOTH overlapping subproblems (repeated work) AND optimal substructure (optimal answer built from optimal sub-answers) — memoization only helps when subproblems actually repeat.
  • Follow the four-question recipe: define dp[i] precisely FIRST, then base case, recurrence, and final answer — skipping the definition is why most DP attempts fail.
  • Top-down (recursion + cache) mirrors natural problem-solving and only computes needed subproblems; bottom-up (iterative table-fill) avoids recursion overhead and space-optimizes more naturally.
  • The include/exclude skeleton (0/1 Knapsack) solves a huge family of problems: subset sum, partition, coin change variants.
  • When space-optimizing 2D→1D, iteration DIRECTION matters critically — backward for 0/1 (each item once), forward for unbounded (items reusable) — this single detail is a common, precise interview probe.
