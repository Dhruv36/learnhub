Binary Tree Patterns Trees & Graphs

Concept: Nearly every tree problem is solved with the SAME recursive skeleton: define what a function returns for a single node in terms of what it returns for that node's children. The skill isn't memorizing solutions — it's recognizing which of a few return-value shapes a given problem needs.

Recognition Cues

🔍 Any problem operating on a binary tree/BST: height, diameter, path sums, validating BST properties, serialization, lowest common ancestor, or converting between tree and array/list forms. The pattern is less about WHEN to use recursion (almost always) and more about WHAT shape of value each recursive call should return.

Template (the universal recursive skeleton)

function solve(node) {
  if (!node) return BASE_CASE;                      // define carefully — this is often where bugs hide
  const left = solve(node.left);
  const right = solve(node.right);
  return combine(node.val, left, right);             // combine children's results with THIS node
}
// Every tree problem in this pattern fits this shape. The three decisions that define the SPECIFIC
// problem are: (1) what does BASE_CASE mean for an empty subtree, (2) what does solve() RETURN for a
// node, (3) how does combine() merge a node with its children's results.

Height / Depth — the Simplest Instance

function maxDepth(node) {
  if (!node) return 0;                                // an empty tree has height 0
  return 1 + Math.max(maxDepth(node.left), maxDepth(node.right));
}

Diameter of a Binary Tree — Returning MORE Than the Answer

function diameterOfBinaryTree(root) {
  let diameter = 0;
  function height(node) {
    if (!node) return 0;
    const left = height(node.left);
    const right = height(node.right);
    diameter = Math.max(diameter, left + right);      // update the ANSWER as a side effect...
    return 1 + Math.max(left, right);                  // ...while RETURNING height for the parent's use
  }
  height(root);
  return diameter;
}
// Key insight: the diameter through any node = left height + right height. But the function must
// RETURN height (what the parent needs), while tracking the diameter as a SIDE EFFECT in a closure
// variable — the value you need for the FINAL ANSWER isn't always the same value the RECURSION needs.
🎯 "What does this function return?" is the single most important design question in tree recursion. Sometimes it's the answer itself (height). Sometimes it's auxiliary information the PARENT needs, while the real answer is tracked separately (diameter). Some problems need a function to return MULTIPLE pieces of information at once (e.g., "is this a valid BST" needs both a boolean AND the subtree's min/max values) — often via an array/object/tuple return.

Validate BST — Returning a Range, Not Just a Boolean

function isValidBST(node, min = -Infinity, max = Infinity) {
  if (!node) return true;
  if (node.val <= min || node.val >= max) return false;
  return isValidBST(node.left, min, node.val) &&        // left subtree: everything must be < node.val
         isValidBST(node.right, node.val, max);          // right subtree: everything must be > node.val
}
// A common WRONG approach only checks node.val against its immediate children — missing violations from
// grandchildren further down. Passing a (min, max) RANGE down through the recursion is what makes this
// correct: each node must respect bounds set by ALL its ancestors, not just its direct parent.

Lowest Common Ancestor

function lowestCommonAncestor(root, p, q) {
  if (!root || root === p || root === q) return root;
  const left = lowestCommonAncestor(root.left, p, q);
  const right = lowestCommonAncestor(root.right, p, q);
  if (left && right) return root;                      // p and q found in DIFFERENT subtrees → this is the LCA
  return left || right;                                  // both in the SAME subtree → bubble that answer up
}
// Elegant because it doesn't need to find PATHS to p and q separately — it discovers the LCA as the
// first node where p and q's "signals" converge from different children.

Interview Questions

Q1: Describe the general recursive pattern used to solve most binary tree problems.

Nearly every binary tree problem follows the same three-part skeleton: define a base case for what an empty subtree (null) should produce, recursively call the same function on the left and right children to get their results, and then COMBINE the current node's value with the two children's results to produce this node's own result. What makes different problems different is entirely captured by three design decisions within that skeleton: what the base case value should be (0 for height, true for BST validity with no constraints yet, an empty result for a collection-building problem), what shape of value the function should RETURN for a given node (sometimes a single number, sometimes a boolean, sometimes a tuple/object bundling multiple pieces of information), and how the combine step merges the node's own value with its children's returned results (max plus one for height, logical AND with range checks for BST validation, min/max for finding the tightest bounding range). Recognizing this shared skeleton is far more valuable than memorizing individual solutions, because it turns "how do I solve this new tree problem" into "what should this function return for a single node, given what I know about its children" — a much more tractable design question to answer from first principles.

Q2: In the "Diameter of Binary Tree" problem, why does the recursive helper function return HEIGHT rather than the diameter itself?

The diameter of a binary tree (the longest path between any two nodes, measured in edges) doesn't have a clean recursive definition on its own — the diameter of the WHOLE tree isn't simply some combination of the diameters of the left and right subtrees, because the longest path might actually pass THROUGH the root, combining a path down the left subtree with a path down the right subtree, which is not captured by either subtree's own internal diameter. However, there IS a clean recursive relationship for a DIFFERENT quantity: the diameter of the path passing through any given node equals that node's left subtree's height plus its right subtree's height (the longest path down each side, joined at the node). And height itself DOES have a clean, standard recursive definition (1 + max of children's heights). So the solution computes height recursively as usual — which is what each recursive call needs to return so its PARENT can correctly compute its own height — while, as a SIDE EFFECT during that same computation, it checks whether left-height + right-height at the CURRENT node beats the best diameter found so far, updating a variable that lives outside the recursive function (typically captured via a closure). This illustrates an important general lesson: what a recursive function needs to RETURN (to make the recursion itself well-defined and useful to the caller) is not always the same thing as what the ORIGINAL PROBLEM is actually asking for — sometimes you need to track the true answer separately, off to the side, while the recursion carries different information needed purely to make the recursive relationship work.

Q3: Why is checking only "is this node's value greater than its left child and less than its right child" insufficient to validate a BST?

The binary search tree property is a GLOBAL constraint, not merely a local one: every node in the LEFT subtree of a given node must be less than that node's value, and every node in the RIGHT subtree must be greater — not just the immediate children, but every descendant at every depth. A check that only compares a node against its direct children misses violations that occur further down the tree. Consider a node with value 10 whose left child is 5 (satisfying the local check, since 5 < 10) — but if that left child (5) itself has a RIGHT child with value 15, that grandchild sits in the left subtree of the root (10) yet has a value GREATER than 10, violating the true BST property, even though every individual parent-child comparison along the way might look locally fine in a naive check (5 < 10 is satisfied, and 15 > 5 might look "fine" if you're only checking that the right child of a node is greater than that node — without realizing 15 also needs to be less than 10). The correct approach threads a valid RANGE (a minimum and maximum bound) down through the recursion: initially unbounded (-infinity to +infinity) at the root, and as you recurse into a left child, you tighten the upper bound to the parent's value (everything in a left subtree must be less than that ancestor); recursing into a right child tightens the lower bound similarly. This way, each node is checked against the CUMULATIVE constraints imposed by ALL of its ancestors, not just its immediate parent, correctly catching violations no matter how deep in the tree they occur.

🏋️ Practice: implement maxDepth and trace it on a small unbalanced tree by hand; implement diameterOfBinaryTree and verify the diameter can pass through a non-root node; implement isValidBST both the WRONG way (local check only) and the correct way (range-based), constructing a test case that distinguishes them; implement lowestCommonAncestor and trace why it correctly handles p and q both being in the same subtree; solve "serialize and deserialize a binary tree" using pre-order traversal with null markers.

🎯 Key Takeaways

  • Almost every tree problem fits one recursive skeleton: base case for null, recurse on children, combine results — the design work is choosing base case, return shape, and combine logic.
  • What a function RETURNS (for its parent's benefit) isn't always the same as the FINAL ANSWER — diameter-style problems track the answer separately via a closure while returning height for the recursion to work.
  • BST validation requires threading a (min, max) range down through the recursion — checking only immediate parent/child misses deep violations.
  • Lowest Common Ancestor is found elegantly by letting p/q "signals" bubble up and converge at the first node where they come from different subtrees.
  • Avoid O(n²) by computing auxiliary values (like height) in the SAME traversal as the main property being checked, not in a separate nested call.
