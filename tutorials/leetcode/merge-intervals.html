Merge Intervals Core Pattern

Concept: Sort intervals by start time, then sweep once — if the current interval overlaps the previous one, merge them; otherwise, start a new group. Sorting turns an O(n²) pairwise-overlap check into an O(n log n) single sweep.

Recognition Cues

🔍 "Merge overlapping intervals" • "insert a new interval" • "meeting rooms" (can all meetings be attended? minimum rooms needed?) • "employee free time" • anything with (start, end) pairs and a question about overlap, coverage, or scheduling.

Template (merge overlapping intervals)

function merge(intervals) {
  intervals.sort((a, b) => a[0] - b[0]);           // SORT BY START — the critical first step
  const result = [intervals[0]];
  for (let i = 1; i < intervals.length; i++) {
    const last = result[result.length - 1];
    const current = intervals[i];
    if (current[0] <= last[1]) {                    // overlap: current starts before last ends
      last[1] = Math.max(last[1], current[1]);       // extend the merged interval
    } else {
      result.push(current);                          // no overlap: start a new group
    }
  }
  return result;
}
// Why sorting works: once sorted by start, you ONLY ever need to compare each interval against the
// LAST merged one — an earlier interval can never suddenly overlap something far ahead after sorting.

Meeting Rooms II — Minimum Rooms Needed

function minMeetingRooms(intervals) {
  const starts = intervals.map(i => i[0]).sort((a, b) => a - b);
  const ends = intervals.map(i => i[1]).sort((a, b) => a - b);
  let rooms = 0, maxRooms = 0, s = 0, e = 0;
  while (s < starts.length) {
    if (starts[s] < ends[e]) { rooms++; s++; }       // a meeting starts before the earliest one ends
    else { rooms--; e++; }                            // a meeting ends — free up a room
    maxRooms = Math.max(maxRooms, rooms);
  }
  return maxRooms;
}
// Separating starts and ends into their OWN sorted arrays and sweeping both simultaneously is a
// distinct sub-pattern: it answers "what's the max overlap at any point in time" in O(n log n).

Insert Interval — a Common Variant

function insert(intervals, newInterval) {
  const result = [];
  let i = 0;
  while (i < intervals.length && intervals[i][1] < newInterval[0])   // 1. before: no overlap yet
    result.push(intervals[i++]);
  while (i < intervals.length && intervals[i][0] <= newInterval[1]) {  // 2. overlapping: merge into newInterval
    newInterval = [Math.min(newInterval[0], intervals[i][0]), Math.max(newInterval[1], intervals[i][1])];
    i++;
  }
  result.push(newInterval);
  while (i < intervals.length) result.push(intervals[i++]);            // 3. after: no overlap
  return result;
}
// Already-sorted input means you DON'T need to re-sort — three linear phases (before/overlap/after)
// achieve O(n), better than the O(n log n) of appending then re-sorting from scratch.

Interview Questions

Q1: Why must intervals be sorted by start time before merging, and why does this make one pass sufficient?

Without sorting, an interval near the end of the input array could overlap with one near the beginning, forcing you to compare every pair of intervals to be sure nothing was missed — an O(n²) approach. Sorting by START TIME establishes an invariant that eliminates this: once the intervals are processed in increasing order of start time, if the CURRENT interval doesn't overlap with the LAST interval added to your result (i.e., current's start is after last's end), it's guaranteed that current also doesn't overlap with any EARLIER interval already in the result — because all earlier intervals in the result have start times ≤ last's start time, and since current comes after last in sorted order and doesn't even reach back to overlap last, it certainly doesn't reach back further to overlap something that started even earlier. This means at each step you only ever need to compare the current interval against the SINGLE most-recently-merged interval, not against the entire result so far — reducing the merging step to one O(n) linear pass after the O(n log n) sort, for O(n log n) total, dominated by the sort. This is a very common technique across interval problems: sorting converts an inherently pairwise comparison problem into a single linear sweep by establishing that "no overlap with the most recent" implies "no overlap with anything earlier."

Q2: Explain the two-pointer sweep technique used in "Meeting Rooms II" to find the minimum number of rooms needed.

Rather than tracking full (start, end) interval pairs together, this approach separates all the start times into one sorted array and all the end times into another sorted array, then walks through both with two pointers simultaneously, effectively simulating a timeline of "events." At each step, it compares the next unprocessed start time against the next unprocessed end time: if the next meeting STARTS before the earliest currently-tracked meeting ENDS (starts[s] < ends[e]), that represents a new meeting beginning while an existing one is still occupying a room, so you need an ADDITIONAL room — increment the room count and advance the start pointer. Otherwise, it means a meeting is ENDING before (or exactly as) the next one starts, freeing up a room — decrement the room count and advance the end pointer. Tracking the MAXIMUM value the room count ever reaches during this sweep gives the minimum number of rooms needed to handle every meeting without conflict, since that peak represents the maximum number of meetings that are simultaneously in progress at any single instant across the whole schedule. This achieves O(n log n) overall (dominated by sorting the two arrays), avoiding a naive approach that might check every pair of meetings for overlap (O(n²)) or simulate a room allocation with a more complex data structure like a min-heap of end times (also correct, also O(n log n), but more code).

Q3: Why does "Insert Interval" achieve O(n) time instead of the O(n log n) that merging usually requires?

The standard interval-merging algorithm needs O(n log n) specifically because it must SORT the input first — the intervals could arrive in any order. "Insert Interval" is a special case where the existing intervals array is GIVEN as already sorted and non-overlapping (a stated precondition of the problem), and you're only inserting ONE new interval into it. Because the existing data is already sorted, you don't need to re-sort anything — you can process the array in a single linear pass with three distinct phases: first, copy over every interval that ends entirely before the new interval starts (no overlap possible, since everything is sorted, these can never interact with the new interval); second, for every interval that DOES overlap the new interval (its start is before or at the new interval's current end), merge it into a growing "newInterval" by expanding its bounds to encompass both; third, once you've passed all overlapping intervals, copy over the rest of the original array unchanged (nothing there can overlap either, again due to the sorted precondition). Each phase does a single linear scan through disjoint portions of the array, giving O(n) total time — faster than the general "append then merge-with-sort" approach, which would cost O(n log n) purely because it redundantly re-sorts data that was already sorted before the insertion.

🏋️ Practice: implement merge intervals and test with overlapping, touching, and disjoint interval sets; implement Meeting Rooms II with the two-pointer sweep and verify against a brute-force overlap-counting approach; implement Insert Interval with the three-phase linear scan and confirm it never re-sorts; solve "Non-overlapping Intervals" (minimum removals to make the rest non-overlapping — a greedy variant); solve "Employee Free Time" by merging all employees' schedules together first.

🎯 Key Takeaways

  • Sort intervals by start time first — this is what reduces merging from O(n²) pairwise comparison to an O(n) single sweep after the sort.
  • Once sorted, only compare each interval against the LAST merged one — earlier overlaps are structurally impossible to miss.
  • The two-pointer start/end sweep (Meeting Rooms II) finds maximum concurrent overlap in O(n log n) without simulating a heap.
  • Already-sorted input (Insert Interval) allows a three-phase O(n) linear scan instead of re-sorting.
  • Distinguish "any overlap at all" (Meeting Rooms I) from "maximum concurrent overlap" (Meeting Rooms II) — different questions, different algorithms.
