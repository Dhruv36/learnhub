Bit Manipulation Advanced Pattern

Concept: Operate directly on a number's binary representation using AND, OR, XOR, NOT, and shifts. A handful of memorized bit tricks solve an entire class of "O(1) space" problems that would otherwise need a hash set or extra array — and XOR's self-canceling property is the single most interview-tested idea in this pattern.

Recognition Cues

🔍 "Find the single/unique number" among duplicates • "count set bits" • "is this a power of two" • "without using extra space" combined with array/number manipulation • subset generation via bitmasks • anywhere the problem explicitly forbids extra data structures but allows O(1) space.

The Core Operators, Precisely

a & b     // AND: 1 only where BOTH bits are 1        — used for masking/checking specific bits
a | b     // OR:  1 where EITHER bit is 1                — used for setting bits
a ^ b     // XOR: 1 where bits DIFFER                     — self-canceling: a ^ a = 0, a ^ 0 = a
~a        // NOT: flips every bit
a << n    // left shift: multiply by 2ⁿ (with overflow caveats)
a >> n    // right shift: divide by 2ⁿ (rounding toward -∞ for signed numbers)

XOR's Self-Canceling Property — Single Number

function singleNumber(nums) {                    // every number appears TWICE except one — find it
  return nums.reduce((acc, n) => acc ^ n, 0);
}
// XOR is commutative and associative, and a ^ a = 0, a ^ 0 = a. XOR-ing the WHOLE array together:
// every number appearing TWICE cancels itself out (n ^ n = 0), leaving ONLY the number that appears
// once. O(n) time, O(1) space — no hash set needed at all.
🎯 XOR-everything-together is the highest-leverage single trick in this pattern. It generalizes: "single number where others appear THREE times" needs bit-counting per position instead of plain XOR, but the core insight — duplicates cancel, leaving only the unique value — is the same idea driving an entire family of "find the odd one out with O(1) space" problems.

Checking / Setting / Clearing a Specific Bit

function getBit(num, i)   { return (num >> i) & 1; }        // isolate bit i: shift it to position 0, mask
function setBit(num, i)   { return num | (1 << i); }         // force bit i to 1
function clearBit(num, i) { return num & ~(1 << i); }         // force bit i to 0
function toggleBit(num, i){ return num ^ (1 << i); }          // flip bit i

Two Classic One-Liners

function isPowerOfTwo(n) {
  return n > 0 && (n & (n - 1)) === 0;
}
// WHY it works: a power of two in binary is a single 1-bit followed by zeros (e.g., 8 = 1000).
// Subtracting 1 flips that single bit to 0 and ALL bits after it to 1 (1000 - 1 = 0111).
// ANDing the two leaves ZERO bits in common — n & (n-1) === 0 ⟺ n has EXACTLY one bit set.

function countSetBits(n) {                          // Brian Kernighan's algorithm — O(number of 1 bits)
  let count = 0;
  while (n) {
    n &= (n - 1);                                      // clears the LOWEST set bit each iteration
    count++;
  }
  return count;
}
// Same n & (n-1) trick, used differently: each application removes exactly the lowest 1-bit, so the
// loop runs once per SET bit, not once per bit position — faster than checking all 32 bits individually
// when the number has few 1-bits.

Bitmasks for Subset Representation

// Represent a SUBSET of n items as an n-bit integer — bit i set means "item i is included":
function allSubsets(items) {
  const n = items.length, result = [];
  for (let mask = 0; mask < (1 << n); mask++) {         // every integer from 0 to 2ⁿ-1 IS a distinct subset
    const subset = [];
    for (let i = 0; i < n; i++) if (mask & (1 << i)) subset.push(items[i]);
    result.push(subset);
  }
  return result;
}
// This is an ALTERNATIVE to backtracking-based subset generation — often used in DP-over-subsets
// problems (like the Traveling Salesman DP) where each mask IS a dp array index.

Interview Questions

Q1: Explain why XOR-ing an entire array together finds the "single number" when every other value appears exactly twice.

XOR has three properties that combine to make this work. First, it's COMMUTATIVE and ASSOCIATIVE, meaning the order in which you XOR a sequence of values together doesn't affect the final result — you can freely reorder and regroup the XOR operations across the whole array. Second, XOR-ing any value with itself produces zero (a ^ a = 0) — this is the "self-canceling" property. Third, XOR-ing any value with zero returns that value unchanged (a ^ 0 = a) — zero acts as XOR's identity element. Because every duplicate value appears EXACTLY twice in the array, and XOR is commutative/associative (letting you conceptually group each duplicate pair together regardless of where they actually sit in the array), XOR-ing all of a duplicate pair's two occurrences together produces zero, and that zero then XORs with everything else without changing it (since a ^ 0 = a). After every duplicate pair has effectively canceled itself out this way, the only value that DOESN'T have a partner to cancel with — the single unique number — is the only thing left in the final accumulated result. This achieves the answer in a single O(n) pass with O(1) extra space, which is strictly better than the "obvious" alternative of using a hash set to track which numbers have been seen once versus twice (which would require O(n) extra space) — the bit-level cancellation trick sidesteps needing any auxiliary data structure at all.

Q2: Walk through why `n & (n - 1)` clears exactly the lowest set bit of n, and describe two different problems this fact solves.

Consider n in binary ending in some number of trailing zeros followed by its lowest set bit (a 1), like ...abc1000 for some lowest set bit at a particular position, with everything below it being zero. Subtracting 1 from n causes that lowest 1-bit to "borrow": the lowest set bit flips from 1 to 0, and every bit BELOW it (which were all 0) flips to 1 as part of the borrow chain — so n-1 becomes ...abc0111 (using the same example pattern), while every bit ABOVE the lowest set bit position remains completely unchanged from n. When you then compute n & (n-1) (bitwise AND), every bit above the lowest set bit position matches between n and n-1 and stays the same, the lowest set bit position itself is 1 in n but 0 in n-1 so the AND produces 0 there, and every bit below that position is 0 in n so the AND produces 0 there regardless of what n-1 has — the net effect is that n & (n-1) equals n with EXACTLYits lowest set bit cleared to 0, and nothing else changed. Two distinct, well-known problems exploit this single fact in different ways. First, checking whether n is a POWER OF TWO: a power of two has EXACTLY ONE set bit in its binary representation (like 8 = 1000), so clearing that one bit with n & (n-1) must produce exactly zero — if n & (n-1) !== 0, n had more than one set bit and therefore isn't a power of two (combined with an explicit n > 0 check, since the formula alone doesn't correctly handle zero or negative numbers). Second, COUNTING the number of set bits in n (Brian Kernighan's algorithm): repeatedly applying n = n & (n-1) in a loop removes exactly one set bit per iteration, so counting how many iterations it takes before n becomes zero directly gives the total count of set bits — and critically, this loop runs proportional to the NUMBER OF SET BITS specifically (not the total number of bit positions, like 32), making it faster than checking every individual bit position when the number happens to have relatively few 1-bits set.

Q3: How would you adapt the "single number" XOR trick if every OTHER number in the array appears exactly THREE times instead of twice?

Plain XOR-everything-together no longer works for this variant, because XOR's self-canceling property only handles PAIRS correctly — XOR-ing a value with itself three times (a ^ a ^ a) simplifies to just `a` (since a ^ a cancels the first two occurrences to 0, leaving 0 ^ a = a), which means values appearing three times DON'T cancel out to zero the way pairs do; they'd incorrectly contribute their own value to the final XOR result, corrupting the answer. The correct generalization requires thinking about the problem BIT POSITION BY BIT POSITION rather than XOR-ing whole numbers together. For each of the (typically 32) individual bit positions, count how many numbers in the array have a 1 in that specific position, summed across the entire array. For any bit position where the unique (singleton) number has a 0 bit, that position's total count across the whole array must be a multiple of 3 exactly (since only the triple-appearing numbers contribute to that position, and each contributes 3 copies of whatever bit they have there). For any bit position where the singleton number has a 1 bit, the total count at that position will be (a multiple of 3) + 1 — one extra 1 contributed by the singleton itself, on top of however many complete triples also happened to have a 1 there. So by computing, for each bit position independently, whether the total count of 1-bits at that position modulo 3 equals 0 or 1, you can reconstruct the singleton number bit-by-bit: if the count mod 3 is 1, that bit is set in the answer; if it's 0, that bit is clear. This still achieves O(n) time (n numbers × a constant 32 bit positions each = O(32n) = O(n)) and O(1) extra space (just 32 running counters), generalizing the core "duplicates cancel, singleton remains" idea from simple XOR pairing to modular counting per bit position for triples.

🏋️ Practice: implement singleNumber with XOR and trace through a small array to confirm the cancellation; implement isPowerOfTwo and countSetBits using the n & (n-1) trick, verifying against a naive bit-by-bit check; solve "Single Number II" (others appear three times) using the bit-counting-modulo-3 approach; implement getBit/setBit/clearBit/toggleBit and use them to solve "Number of 1 Bits"; generate all subsets of a small set using bitmask enumeration and compare against the backtracking-based Subsets solution.

🎯 Key Takeaways

  • XOR-ing an entire array cancels out every value appearing an EVEN number of times, leaving only a true singleton — O(n) time, O(1) space, no hash set needed.
  • n & (n-1) clears exactly the lowest set bit — powers-of-two detection (single set bit) with count-set-bits (Brian Kernighan's algorithm) as its two most common applications.
  • Plain XOR only handles pairs; "singleton among triples" requires counting set bits per position modulo 3 instead.
  • Bitmasks represent subsets as integers (bit i = item i included) — an alternative to backtracking for subset enumeration, especially useful as DP array indices in subset-DP problems.
  • Watch for sign-extension on right shifts of negative numbers, and always distinguish bitwise (&, |) from logical (&&, ||) operators.
