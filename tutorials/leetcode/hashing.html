Hashing Essentials Foundation

Concept: A hash map/set trades O(n) space for turning O(n) linear searches into O(1) average lookups. It's the single highest-leverage tool for converting an O(n²) brute force into O(n) — when in doubt, ask "can a hash map remember something I'd otherwise re-scan for?"

The Core Trade: Space for Time

// Brute force Two Sum — O(n²): for each element, scan the REST of the array for its complement.
function twoSumBrute(nums, target) {
  for (let i = 0; i < nums.length; i++)
    for (let j = i + 1; j < nums.length; j++)
      if (nums[i] + nums[j] === target) return [i, j];
}

// Hash map — O(n): remember what you've SEEN, check if the complement already exists.
function twoSum(nums, target) {
  const seen = new Map();               // value → index
  for (let i = 0; i < nums.length; i++) {
    const complement = target - nums[i];
    if (seen.has(complement)) return [seen.get(complement), i];
    seen.set(nums[i], i);               // remember AFTER checking — avoids using the same element twice
  }
}
// The pattern: "have I seen X before?" is exactly what a hash map answers in O(1) instead of O(n).
🎯 Two Sum is the canonical example of the space-time trade-off — know it cold. The subtlety worth internalizing: check for the complement BEFORE adding the current element to the map, so you never accidentally pair an element with itself. This exact "seen-before" check generalizes to a huge fraction of interview problems.

Hashing for Grouping

// "Group anagrams" — a SORTED string (or character-count signature) is a canonical key for all its anagrams:
function groupAnagrams(strs) {
  const groups = new Map();
  for (const s of strs) {
    const key = [...s].sort().join("");             // canonical form: same letters → same key
    if (!groups.has(key)) groups.set(key, []);
    groups.get(key).push(s);
  }
  return [...groups.values()];
}
// The general trick: find a CANONICAL REPRESENTATION that's identical for all "equivalent" items, use
// it as the hash key. Sorting, frequency-count tuples, and normalized forms are the most common canonicalizers.

Sets for O(1) Membership

// "Longest consecutive sequence" — O(n) using a SET, not sorting (which would be O(n log n)):
function longestConsecutive(nums) {
  const set = new Set(nums);
  let best = 0;
  for (const n of set) {
    if (!set.has(n - 1)) {                // only START counting from the BEGINNING of a sequence
      let length = 1;
      while (set.has(n + length)) length++;
      best = Math.max(best, length);
    }
  }
  return best;
}
// The "only start from a sequence's beginning" check is what keeps this O(n) overall — every element is
// visited by the inner while loop AT MOST once across the entire run, despite the nested-looking loops.

Interview Questions

Q1: Why does checking for the complement BEFORE adding the current element matter in Two Sum?

If you insert nums[i] into the map first and THEN check whether its complement (target - nums[i]) exists, you risk pairing an element with ITSELF: if target happens to equal 2 × nums[i], the map lookup for the complement would find the entry you just inserted for THIS SAME index, incorrectly reporting a pair using one element twice — which isn't a valid answer for Two Sum (which requires two DISTINCT indices). Checking for the complement first, before inserting the current element, guarantees that whatever match is found was placed into the map by a PREVIOUS iteration (a different, earlier index), which is always a valid, distinct pairing. This ordering detail is a small thing but is exactly the kind of edge case interviewers probe to see if a candidate deeply understands their own solution rather than having memorized it superficially.

Q2: In "Group Anagrams," why does sorting each string work as a hash key, and what's the alternative?

Two strings are anagrams of each other if and only if they contain exactly the same multiset of characters — which means that SORTING both strings' characters alphabetically produces the exact SAME resulting string for both. This makes the sorted form a CANONICAL REPRESENTATION: every string that's an anagram of "eat" (like "tea" and "ate") sorts to the identical key "aet", so grouping strings by their sorted form naturally groups all mutual anagrams together under one hash map key, while non-anagrams sort to different keys and land in different groups. Sorting each string costs O(k log k) where k is the string's length, so for n strings of average length k, this approach costs O(n·k·log k) total. The alternative is a CHARACTER-COUNT signature — instead of sorting, build a fixed-size array of 26 counts (for lowercase English letters) and use a string representation of that count array (e.g., "1,0,0,1,0,...") as the key; two anagrams produce identical count arrays without needing to sort. This costs O(k) per string instead of O(k log k), making the overall algorithm O(n·k) — asymptotically faster when strings are long, at the cost of slightly more code to build the count-based key. Both are instances of the same general idea: find a canonical form that's identical for all "equivalent" inputs, and hash on that.

Q3: Why is "Longest Consecutive Sequence" O(n) despite having what looks like a nested loop?

The algorithm puts every number into a hash SET (O(n)), then iterates over each number in the set and, for numbers that are the START of a consecutive run (meaning n-1 is NOT in the set — there's no predecessor), counts forward with a while loop checking n+1, n+2, n+3... until the streak breaks. The key insight for the complexity is that the inner while loop only ever RUNS for numbers that pass the "is this a sequence start" check — a number in the MIDDLE of a consecutive run (where n-1 IS in the set) never triggers its own inner while loop at all, it's skipped immediately by the outer iteration. Since every number belongs to EXACTLY ONE consecutive run, and only that run's starting number ever launches a while loop that walks through the run, each number in the entire input is visited by an inner while loop step AT MOST ONCE across the whole algorithm's execution — not once per outer iteration, but once total, no matter how many times the outer for-loop iterates over numbers that turn out to be mid-run and get skipped. Summing this up: the outer loop does O(1) work per element (the set.has(n-1) check) for O(n) total, and the combined work across all inner while loops is also O(n) total (each number counted at most once across every while loop combined) — giving O(n) overall, not O(n²), despite the visually nested loop structure.

🏋️ Practice: implement Two Sum with a hash map and trace through why the complement-check-before-insert ordering matters using a self-pairing example; implement Group Anagrams both with sorting and with a character-count signature, and compare for very long strings; implement Longest Consecutive Sequence and add a counter to prove the inner while loop runs O(n) times total across the whole execution, not per outer iteration; solve "contains duplicate" and "valid anagram" using sets/maps.

🎯 Key Takeaways

  • Hash maps/sets trade O(n) space for O(1) average lookup — the highest-leverage tool for turning O(n²) brute force into O(n).
  • The "have I seen X before?" pattern (Two Sum) generalizes broadly — check membership before inserting to avoid self-pairing.
  • Grouping problems (anagrams) work by finding a CANONICAL representation (sorted string, count signature) to use as the hash key.
  • Nested-looking loops can still be O(n) overall if the inner loop's total work across ALL outer iterations is bounded by n (Longest Consecutive Sequence).
  • State both time AND space complexity explicitly — a hash map solution's O(n) space is a real trade-off worth naming.
