Fast & Slow Pointers Core Pattern

Concept: Two pointers traverse the SAME structure at different speeds (typically 1x and 2x). If there's a cycle, the fast pointer eventually laps the slow one — detecting it in O(1) space, no hash set needed. Also called Floyd's Tortoise and Hare.

Recognition Cues

🔍 "Detect a cycle" in a linked list • "find the middle" of a linked list in one pass • "find the start of a cycle" • any problem where re-visiting the same state means an infinite loop/repeating sequence (e.g., "Happy Number").

Template (cycle detection)

function hasCycle(head) {
  let slow = head, fast = head;
  while (fast && fast.next) {
    slow = slow.next;              // moves 1 step
    fast = fast.next.next;         // moves 2 steps
    if (slow === fast) return true;   // they met — there's a cycle
  }
  return false;                    // fast reached the end — no cycle
}
// Why fast catches slow: think of it as fast gaining 1 step on slow EVERY iteration. If there's a cycle,
// the gap between them shrinks by 1 each step and MUST hit 0 within (cycle length) iterations —
// they can't skip past each other since fast only gains 1 net step per round.

Finding the Cycle's START (the classic hard follow-up)

function detectCycleStart(head) {
  let slow = head, fast = head;
  while (fast && fast.next) {                 // Phase 1: find the meeting point
    slow = slow.next;
    fast = fast.next.next;
    if (slow === fast) {
      let ptr = head;                          // Phase 2: reset one pointer to head, move BOTH at 1x
      while (ptr !== slow) { ptr = ptr.next; slow = slow.next; }
      return ptr;                              // they meet exactly at the cycle's start
    }
  }
  return null;
}
// The math: let the distance from head to cycle start = a, cycle start to meeting point = b,
// meeting point back to cycle start = c. When they meet, slow traveled a+b, fast traveled a+b+b+c
// (fast did an extra full loop). Since fast = 2×slow: a+b+b+c = 2(a+b) → a = c. So walking `a` steps
// from HEAD and `a` steps from the MEETING POINT (both at 1x) land on the same node: the cycle start.

Finding the Middle in One Pass

function findMiddle(head) {
  let slow = head, fast = head;
  while (fast && fast.next) {
    slow = slow.next;
    fast = fast.next.next;
  }
  return slow;         // when fast reaches the end, slow is exactly at the middle
}
// When fast has moved 2x as many steps as slow and reaches the end (length n), slow has moved n/2 —
// i.e., it's at the middle. No need to count the list length first — one pass, not two.

Interview Questions

Q1: Why does the fast pointer always eventually meet the slow pointer if a cycle exists?

Once both pointers have entered the cycle, think of the "gap" between them (measured along the cycle, in the direction of traversal) at each step. Because fast moves 2 steps per iteration and slow moves 1, fast gains exactly 1 net position on slow EVERY iteration — the gap between them strictly decreases by 1 each round, modulo the cycle's length. Since the cycle has some finite length L, the gap starts somewhere between 0 and L−1 and decreases by 1 each iteration; it's mathematically guaranteed to reach exactly 0 (meaning they occupy the same node) within at most L iterations after both pointers have entered the cycle — the gap can't "skip over" zero because it only ever decreases by exactly 1 per step, never jumping by 2 or more. If there's NO cycle, fast simply reaches the end of the list (null) first, since it's moving twice as fast, and the loop terminates via the `fast && fast.next` condition without ever finding a match. This is why the algorithm correctly and deterministically distinguishes "has a cycle" from "doesn't have a cycle" using only O(1) extra space, unlike a hash-set approach that tracks every visited node explicitly (O(n) space).

Q2: Walk through the math for why resetting one pointer to head finds the exact start of a cycle.

Label three distances: `a` = the distance from the list's head to the cycle's entry point, `b` = the distance from the cycle's entry point to wherever slow and fast first meet, and `c` = the remaining distance around the cycle from that meeting point back to the cycle's entry point (so a+b+c accounts for one full trip around from head through the meeting point back to the start of the cycle, and b+c is the cycle's total length). When slow and fast first meet, slow has traveled exactly a+b steps (it entered the cycle after `a` steps, then went `b` more steps into the cycle to reach the meeting point). Fast, moving at 2x speed the whole time, has traveled 2(a+b) steps in the same time. But fast's total distance can ALSO be expressed as a+b plus some whole number of extra full trips around the cycle (since it's faster, it likely lapped the cycle at least once) — in the simplest case, exactly one extra full loop: a+b+(b+c). Setting these equal: 2(a+b) = a+b+(b+c), which simplifies to a+b = b+c, and therefore a = c. This is the crucial result: the distance from the HEAD to the cycle's start (a) equals the distance from the MEETING POINT to the cycle's start (c, going forward around the cycle). So if you place one pointer back at the head and leave the other at the meeting point, and advance BOTH at the same speed (1 step at a time), they will cover distances a and c respectively before both landing exactly on the cycle's entry node — because a equals c, they arrive at the same node at the same time, which must be the cycle's start.

Q3: Why does the fast/slow pointer technique find the middle of a linked list in exactly one pass?

Both pointers start at the head and advance simultaneously each iteration: slow moves one node, fast moves two nodes. Consider a list of length n. By the time fast has traveled far enough to reach the end of the list (or one before it, depending on parity), it has covered roughly n steps; since fast always covers exactly twice the distance slow covers in the same number of iterations, slow will have covered roughly n/2 steps at that same moment — which is precisely the middle of the list. The loop condition `while (fast && fast.next)` stops fast just as it reaches or passes the end, and at that exact moment slow is sitting at the middle node (for an odd-length list) or the second of the two middle nodes (for an even-length list, which is the conventional definition used in most problem statements). The key efficiency insight is that this requires only ONE traversal of the list: a naive approach would first walk the entire list once just to COUNT its length n, then walk it a SECOND time to n/2 steps to reach the middle — two full passes. The fast/slow approach achieves the same result while only ever moving forward, in a single combined pass, because it discovers "where the middle is" and "arrives there" simultaneously rather than needing to know the total length in advance.

🏋️ Practice: implement cycle detection and test on both a cyclic and acyclic list; extend it to find the cycle's start node and verify against a list where you know the cycle's entry point; implement find-the-middle and use it as a building block for "reorder list" or "palindrome linked list"; solve "Happy Number" by modeling digit-square-sum sequences as an implicit linked structure detected for cycles.

🎯 Key Takeaways

  • Fast (2x speed) and slow (1x speed) pointers on the same structure detect cycles in O(1) space — the gap closes by exactly 1 each iteration once both are in the cycle.
  • Finding the cycle's exact start requires a second phase: reset one pointer to head, advance both at 1x — they meet at the cycle start because distance-from-head equals distance-from-meeting-point (a = c).
  • The same technique finds a linked list's middle in ONE pass, with no need to count the length first.
  • Always guard with `fast && fast.next` before advancing fast by two steps to avoid a null pointer error.
  • This pattern trades a hash set's O(n) space for O(1) — a common interview follow-up after a hash-set solution.
