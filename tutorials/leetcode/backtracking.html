Backtracking Advanced Pattern

Concept: Systematically try every possibility by building a partial solution incrementally, abandoning ("backtracking" from) a path the moment it can't possibly lead to a valid answer, and undoing the last choice to try the next one. It's exhaustive search made efficient via early pruning — the difference between backtracking and brute force IS the pruning.

Recognition Cues

🔍 "Generate all subsets/permutations/combinations" • "N-Queens" • "word search in a grid" • "valid parentheses combinations" • Sudoku solving • anywhere the problem says "find ALL ways to..." or "does ANY valid arrangement exist" — with constraints small enough (n ≤ ~20) that exponential search is actually tractable.

The Universal Template

function backtrack(path, choices) {
  if (isComplete(path)) {                     // BASE CASE: a full, valid solution
    results.push([...path]);                    // COPY — path will keep mutating after this
    return;
  }
  for (const choice of choices) {
    if (!isValid(choice, path)) continue;       // PRUNE — skip choices that can't lead anywhere valid
    path.push(choice);                           // CHOOSE
    backtrack(path, nextChoices(choice));        // EXPLORE
    path.pop();                                   // UN-CHOOSE — this IS the "backtrack"
  }
}
// Choose → Explore → Un-choose is the entire pattern. Every backtracking problem is this skeleton with
// different definitions of isComplete, isValid, and what "choices" means for that specific problem.
⚠️ Forgetting to COPY the path when saving a result is the #1 backtracking bug. `results.push(path)` pushes a REFERENCE to the same array object that keeps getting mutated by subsequent push/pop calls — by the time you inspect `results`, every entry has silently become the SAME (usually empty or wrong) array. Always `results.push([...path])` or `results.push(path.slice())` to save a snapshot.

Subsets — the Simplest Instance

function subsets(nums) {
  const results = [];
  function backtrack(start, path) {
    results.push([...path]);                    // EVERY partial path is a valid subset — push at every node
    for (let i = start; i < nums.length; i++) {
      path.push(nums[i]);
      backtrack(i + 1, path);                     // start from i+1 — never reuse an earlier index
      path.pop();
    }
  }
  backtrack(0, []);
  return results;
}
// Passing `start` forward (not resetting to 0) is what prevents duplicate subsets in different ORDERS
// ([1,2] and [2,1] as the "same" subset) — it enforces choosing indices in strictly increasing order.

Permutations — Every Order Matters

function permute(nums) {
  const results = [];
  function backtrack(path, used) {
    if (path.length === nums.length) { results.push([...path]); return; }
    for (let i = 0; i < nums.length; i++) {
      if (used[i]) continue;                       // PRUNE — this element is already in the current path
      used[i] = true;
      path.push(nums[i]);
      backtrack(path, used);
      path.pop();
      used[i] = false;                              // un-choose the USED marker too, not just the path
    }
  }
  backtrack([], new Array(nums.length).fill(false));
  return results;
}
// Unlike subsets, permutations DO care about order, so every unused index is tried at every position —
// no `start` index restriction, just a `used` tracker to avoid reusing the same element twice.

Handling Duplicates — a Common Hard Follow-Up

function subsetsWithDup(nums) {
  nums.sort((a, b) => a - b);                      // SORT FIRST — duplicates become ADJACENT
  const results = [];
  function backtrack(start, path) {
    results.push([...path]);
    for (let i = start; i < nums.length; i++) {
      if (i > start && nums[i] === nums[i - 1]) continue;   // skip a duplicate VALUE at the SAME tree depth
      path.push(nums[i]);
      backtrack(i + 1, path);
      path.pop();
    }
  }
  backtrack(0, []);
  return results;
}
// The `i > start` check is precise: it only skips a duplicate if it's NOT the first choice at this
// level (i === start is always allowed — that's the first time this VALUE is tried at this depth).

Interview Questions

Q1: What's the difference between backtracking and plain brute-force recursion?

Both explore a space of possibilities recursively, but backtracking's defining feature is PRUNING: the moment a partial solution can be determined to be invalid or unable to lead to any valid complete solution, backtracking abandons that branch immediately — via an early `continue` or `return` — rather than continuing to build it out fully before discovering it doesn't work. Plain brute-force recursion, by contrast, might explore every possible complete combination regardless of validity, checking correctness only once a candidate is fully formed, which wastes enormous amounts of work continuing to build branches that were already doomed early on. The practical impact of pruning can be dramatic: in N-Queens, for example, placing a queen that immediately conflicts with a previously-placed queen can be rejected instantly, avoiding the need to place N-1 more queens and THEN discover the whole arrangement was invalid — across many branches, this early rejection can cut the effective search space by orders of magnitude compared to generating every possible arrangement and filtering at the end. So "backtracking" isn't a fundamentally different algorithm from brute-force search — it's brute-force search made TRACTABLE by aggressively cutting off invalid branches as early as possible, which is exactly why the isValid() check inside the loop, evaluated BEFORE recursing further, is the single most important line in the entire pattern.

Q2: Explain the "choose, explore, un-choose" structure and why the un-choose step is necessary.

Backtracking builds a solution incrementally using a single, shared, mutable data structure (commonly called `path`) that represents the current partial solution as the recursion descends. At each decision point, the algorithm CHOOSES one option (typically by pushing it onto `path`), then EXPLORES further by recursively calling itself to build out the rest of the solution from this new, extended state. Once that recursive call returns — meaning every possibility branching from this choice has been fully explored — the algorithm must UN-CHOOSE, popping that same option back off `path`, before the surrounding loop moves on to try the NEXT alternative choice at this same decision point. This un-choose step is essential because `path` is a single shared object being mutated throughout the entire recursive traversal, not a fresh copy created for each branch — if you don't undo a choice after exploring it, the next sibling branch in the loop would incorrectly start from a `path` that still contains the previous (no-longer-relevant) choice, corrupting every subsequent branch's results with leftover state from branches that have nothing to do with it. The pop is what restores `path` to exactly the state it was in before this particular choice was tried, so the next iteration of the loop starts from a clean, correct baseline — this restoration is literally what "backtracking" means: undoing a choice to go back and try a different one from the same point.

Q3: In "Subsets with Duplicates," why does the algorithm sort the array first, and why is the skip condition specifically `i > start && nums[i] === nums[i-1]`?

When the input array can contain duplicate values, generating subsets naively (as in the basic Subsets algorithm) produces duplicate SUBSETS in the output — for example, with input [1, 2, 2], the basic algorithm would generate the subset [1, 2] twice, once by choosing the first "2" and once by choosing the second "2," even though as VALUES these produce an identical subset. To eliminate this, the algorithm needs a rule that says "don't make the same choice twice at the same decision point" — but recognizing "the same choice" requires comparing the CURRENT candidate against choices ALREADY TRIED at this exact same recursive depth (the same position in the `for` loop, i.e., the same call to `backtrack`). Sorting the array FIRST is what makes this tractable: after sorting, any duplicate VALUES become ADJACENT to each other in the array, so "has an identical value already been tried at this depth" simplifies to a cheap, local check: "is this value the same as the immediately preceding array element?" The precise condition `i > start && nums[i] === nums[i-1]` captures exactly the right case: `nums[i] === nums[i-1]` detects that this element has the same value as its predecessor in the sorted array, and `i > start` ensures this check is ONLY applied when i is NOT the very first index being considered at this recursive depth (start) — because the first time a given value is tried at a particular depth is always legitimate and must proceed normally; it's only the SECOND (and later) occurrences of an already-seen value AT THE SAME DEPTH that should be skipped as redundant. Without the `i > start` guard, the algorithm would incorrectly skip even the first, legitimate occurrence of a value whenever it happened to be preceded by an identical value earlier in the array (from an entirely different branch/depth), which would incorrectly exclude valid, non-duplicate subsets from the output.

🎯 Key Takeaways

  • Backtracking = brute-force search + early pruning — the isValid() check before recursing is what makes it tractable, not a different algorithm.
  • The universal skeleton is choose → explore → un-choose; the un-choose (pop) step restores shared mutable state for sibling branches.
  • Always snapshot results with a copy ([...path]) — pushing the mutable path object directly is the most common bug in this pattern.
  • Subsets/combinations use a `start` index to enforce increasing order (order doesn't matter); permutations use a `used` tracker (order matters, no start restriction).
  • Handling duplicates requires sorting first (to make identical values adjacent) plus an `i > start` guard to skip only repeated choices at the SAME recursive depth.
