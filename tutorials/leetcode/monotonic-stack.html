Stack / Monotonic Stack Core Pattern

Concept: Maintain a stack that's always increasing or decreasing (monotonic). When a new element would break the ordering, POP everything that violates it — each pop answers a "next greater/smaller element" question for the popped item. Every element is pushed once and popped once, giving O(n) total despite the nested-looking while loop.

Recognition Cues

🔍 "Next greater/smaller element" • "daily temperatures" (days until warmer) • "largest rectangle in histogram" • matching brackets/parentheses • "remove K digits to make the smallest number" • anywhere you need to look BACKWARD for the nearest element satisfying a comparison.

Template (next greater element)

function nextGreaterElement(nums) {
  const result = new Array(nums.length).fill(-1);
  const stack = [];                                 // stores INDICES, kept in DECREASING value order
  for (let i = 0; i < nums.length; i++) {
    while (stack.length && nums[stack[stack.length - 1]] < nums[i]) {
      const idx = stack.pop();                        // nums[i] is the "next greater" for this popped index
      result[idx] = nums[i];
    }
    stack.push(i);
  }
  return result;
}
// The stack holds indices whose "next greater element" hasn't been found YET. When nums[i] is bigger
// than the top, it's the answer for EVERYTHING on the stack that's smaller — pop them all.
🎯 Why this is O(n) despite the while loop inside a for loop: amortized analysis. It LOOKS like O(n²) — a loop inside a loop — but each element is pushed onto the stack EXACTLY ONCE and popped AT MOST ONCE across the ENTIRE algorithm's run. Summing all push/pop operations across all iterations gives at most 2n total operations, not n×n — this is the same amortized-cost reasoning as dynamic array resizing.

Balanced Parentheses — the Classic Matching Problem

function isValid(s) {
  const pairs = { ")": "(", "]": "[", "}": "{" };
  const stack = [];
  for (const ch of s) {
    if (ch === "(" || ch === "[" || ch === "{") { stack.push(ch); continue; }
    if (stack.pop() !== pairs[ch]) return false;      // mismatched, or stack was empty (pop returns undefined)
  }
  return stack.length === 0;                          // nothing left un-closed
}
// The stack naturally models "most recently opened, must be closed first" — LIFO matches the nesting
// structure of balanced brackets exactly.

Largest Rectangle in Histogram — the Hard Interview Classic

function largestRectangleArea(heights) {
  const stack = [];                          // indices, heights kept INCREASING
  let maxArea = 0;
  for (let i = 0; i <= heights.length; i++) {
    const h = i === heights.length ? 0 : heights[i];   // sentinel 0 forces the stack to fully empty at the end
    while (stack.length && heights[stack[stack.length - 1]] >= h) {
      const height = heights[stack.pop()];
      const width = stack.length ? i - stack[stack.length - 1] - 1 : i;
      maxArea = Math.max(maxArea, height * width);
    }
    stack.push(i);
  }
  return maxArea;
}
// When popping a bar, its rectangle's width spans from just after the NEW stack top (the nearest
// SHORTER bar to the left) to the current index (the nearest shorter bar to the right, exclusive) —
// the monotonic stack finds both boundaries "for free" as a side effect of maintaining the invariant.

Interview Questions

Q1: Why is the monotonic stack pattern O(n) despite having a while loop nested inside a for loop?

A naive complexity read might see "for loop containing a while loop" and assume O(n²), the same way a plain nested for-for loop would be. But the key difference is that the inner while loop's total work is BOUNDED by how many times elements are pushed and popped across the ENTIRE algorithm, not by how many times it runs per outer iteration. Each element in the input array is pushed onto the stack exactly ONCE (there's exactly one push per outer for-loop iteration) and can be popped AT MOST ONCE (once popped, an element is gone from the stack forever and can never be popped again). So across the whole execution, the total number of push operations is exactly n, and the total number of pop operations across every single while-loop execution, summed together, is at most n as well — giving at most 2n total stack operations overall, not n². This is a specific instance of AMORTIZED analysis: while any single outer iteration's while loop could theoretically pop many elements at once (looking expensive in isolation), that cost is "paid for" by all those elements having been pushed cheaply earlier and never being touched again, so the TOTAL cost summed across all iterations is linear. This is the exact same reasoning used to show dynamic array resizing is O(1) amortized per push despite occasional O(n) resize operations.

Q2: Why does the algorithm store array INDICES on the stack rather than the values themselves?

For simple "next greater element" existence checks, storing values alone would technically work — you could compare nums[i] against the value on top of the stack directly. But most real problems in this family need more than just "what is the next greater value" — they need to know things that require POSITION information: "how many days until a warmer temperature" (Daily Temperatures) needs the DISTANCE between the current index and the index being resolved, and "largest rectangle in histogram" needs the WIDTH of a rectangle, computed as the difference between indices bounding it on the left and right. Storing indices instead of values gives you everything you need for free: you can always recover the corresponding value with a single array lookup (nums[stack[top]]) when a value comparison is needed, but you cannot recover the ORIGINAL POSITION from a bare value alone (especially if the array contains duplicate values, where multiple positions share the same value and are otherwise indistinguishable). Because the vast majority of monotonic-stack problems eventually need either a distance or a width calculated from positions, storing indices is the more general, safer default — you lose nothing (values are always one lookup away) and gain the position information many variants of this pattern actually require.

Q3: In "Largest Rectangle in Histogram," why does popping a bar from the monotonic stack correctly determine both boundaries of its widest rectangle?

The stack is maintained so that the heights corresponding to the indices on it are always in INCREASING order from bottom to top. When a new bar's height is LESS than the height at the top of the stack, it means the bar at the top can no longer extend its rectangle any further to the right — its rectangle is bounded on the right by the current index i (since heights[i] is shorter, a rectangle at the popped bar's height can't include position i). Once that bar is popped, look at what's now on TOP of the stack (after the pop): because the stack was strictly increasing, whatever remains on top is the NEAREST bar to the left that's SHORTER than the bar just popped — anything taller than it would have already been popped earlier when it was blocking a bar between them. So the popped bar's rectangle spans from just after that new stack-top index (its nearest shorter neighbor on the left) to just before the current index i (its nearest shorter neighbor on the right, which is why the width calculation uses `i - stack[top] - 1`, exclusive on both ends). This means the monotonic stack invariant automatically identifies BOTH the left and right boundaries where a bar's height stops being the limiting factor — exactly the two boundaries needed to compute that bar's maximum possible rectangle — as a natural side effect of maintaining the increasing-height ordering, without any separate "find nearest smaller to the left/right" computation needed.

🎯 Key Takeaways

  • A monotonic stack maintains increasing or decreasing order — popping elements that violate it answers "next greater/smaller" for each pop.
  • Despite the nested loop appearance, it's O(n) amortized: each element is pushed once and popped at most once across the whole run.
  • Store INDICES, not values, so you can compute distances/widths — values are always one array lookup away, but positions aren't recoverable from bare values.
  • A sentinel value (0 or -Infinity) at the end forces the stack to fully drain, ensuring every remaining element gets its rectangle/answer computed.
  • The stack's monotonic invariant finds both left AND right boundaries "for free" as a side effect of maintaining ordering — no separate boundary search needed.
