Topological Sort Trees & Graphs

Concept: Order the nodes of a DIRECTED ACYCLIC GRAPH (DAG) so every edge points from earlier to later in the ordering. This models "must happen before" dependencies — course prerequisites, build steps, task scheduling — and doubles as the standard way to DETECT cycles in a directed graph.

Recognition Cues

🔍 "Course prerequisites" / "can you finish all courses" • "build order" / "task scheduling with dependencies" • "alien dictionary" (infer character order from sorted words) • anywhere a DIRECTED edge means "A must come before B" and you need a valid overall order (or to detect that no valid order exists — a cycle).

Template (Kahn's Algorithm — BFS-based)

function topologicalSort(numNodes, edges) {                // edges: [from, to] meaning from → to
  const graph = Array.from({length: numNodes}, () => []);
  const inDegree = new Array(numNodes).fill(0);
  for (const [from, to] of edges) {
    graph[from].push(to);
    inDegree[to]++;                                          // count incoming edges per node
  }

  const queue = [];
  for (let i = 0; i < numNodes; i++) if (inDegree[i] === 0) queue.push(i);   // start with NO prerequisites

  const order = [];
  while (queue.length) {
    const node = queue.shift();
    order.push(node);
    for (const next of graph[node]) {
      inDegree[next]--;                                       // this prerequisite is now satisfied
      if (inDegree[next] === 0) queue.push(next);              // ALL prerequisites satisfied → ready
    }
  }
  return order.length === numNodes ? order : [];                // fewer nodes processed than exist → CYCLE
}
// Kahn's algorithm IS cycle detection: if a cycle exists, the nodes in it NEVER reach inDegree 0
// (each depends on another in the cycle), so they never enter the queue — order.length < numNodes.
🎯 The length check at the end IS the cycle detector — no separate cycle-check needed. Every node in a cycle has at least one incoming edge from ANOTHER node in the same cycle, so its in-degree can never reach zero through normal processing (nothing outside the cycle can "unlock" it, and nodes inside the cycle are all waiting on each other). If order.length < numNodes, some nodes were permanently stuck — that's exactly and only possible if a cycle exists.

Alternative: DFS-based Topological Sort

function topologicalSortDFS(numNodes, graph) {
  const visited = new Array(numNodes).fill(0);      // 0 = unvisited, 1 = in progress, 2 = done
  const order = [];
  let hasCycle = false;

  function dfs(node) {
    if (visited[node] === 1) { hasCycle = true; return; }   // reached a node CURRENTLY on the recursion path → CYCLE
    if (visited[node] === 2) return;                          // already fully processed
    visited[node] = 1;
    for (const next of graph[node]) dfs(next);
    visited[node] = 2;
    order.push(node);                                          // push AFTER exploring all descendants
  }
  for (let i = 0; i < numNodes; i++) if (!visited[i]) dfs(i);
  return hasCycle ? [] : order.reverse();                      // REVERSE — nodes are pushed in FINISH order
}
// The THREE-COLOR (0/1/2) visited state is what distinguishes "still being explored" from "fully done" —
// re-visiting a node that's IN PROGRESS (state 1) is exactly what detects a back-edge (a cycle).

Interview Questions

Q1: Explain how Kahn's algorithm doubles as both a topological sort AND a cycle detector.

Kahn's algorithm repeatedly finds nodes with an in-degree of zero (no unsatisfied prerequisites), processes them, and "removes" their outgoing edges by decrementing the in-degree of each neighbor — when a neighbor's in-degree drops to zero, it becomes newly eligible and gets added to the processing queue. In a graph with NO cycles (a true DAG), every node is eventually reachable this way: since there's no cycle, there must always be at least one currently-unprocessed node with in-degree zero at each step (otherwise every remaining node would depend on another remaining node, forming a cycle by definition), so the algorithm can always make progress until every node has been processed. But if a CYCLE exists among some subset of nodes, every node WITHIN that cycle has at least one incoming edge from ANOTHER node inside the same cycle — meaning each cyclic node's in-degree can only be decremented to zero if all of ITS prerequisites (including ones inside the cycle) get processed first, but those prerequisites are themselves waiting on other cyclic nodes in the same mutually- dependent loop, so NONE of them can ever be the "first" to reach in-degree zero — they're all stuck waiting on each other indefinitely. This means nodes inside a cycle are NEVER added to the queue and never appear in the final `order` array, so simply checking whether order.length equals the total number of nodes at the end tells you definitively whether a cycle exists (fewer nodes processed means some were stuck in a cycle) — no separate cycle-detection pass is needed, it falls directly out of the algorithm's own termination behavior.

Q2: In the DFS-based topological sort, why must the final result be reversed?

The DFS-based approach pushes each node onto the result list only AFTER it has finished recursively exploring ALL of that node's descendants (this is a post-order-style traversal) — a node is only appended once everything reachable from it has already been fully processed and appended first. Consider a simple dependency A → B (A must come before B, meaning there's a directed edge from A to B representing "B depends on A" or "explore A's neighbor B"): DFS starting at A immediately recurses into B, B has no further unvisited neighbors so it finishes first and gets pushed onto the result list FIRST, and only afterward does the recursive call return to A, which then gets pushed SECOND. The raw result list therefore reads [B, A] — but the correct topological order requires A (the prerequisite) to come BEFORE B (the dependent) in the final answer, i.e., [A, B]. This backward relationship is completely general, not specific to this one example: because nodes are always appended only after everything reachable from them (their "descendants," which represent nodes that depend on them being processed first, if you think of edges as prerequisite relationships) has already been appended, deeper/more-dependent nodes always end up EARLIER in the raw list and higher-level/prerequisite nodes end up LATER — exactly the opposite of the desired order. Reversing the entire list at the very end corrects this, converting the "finish order" (last-finished-first, which naturally comes out as most-dependent-first) into the proper topological order (prerequisites-first).

Q3: How does the three-state (0/1/2) visited tracking in DFS-based topological sort detect a cycle?

A simple boolean visited/unvisited tracker is sufficient for basic traversal but can't distinguish two importantly different situations: a node that has been FULLY processed already (all its descendants explored, safe to skip if encountered again through a different path) versus a node that is CURRENTLY being explored — meaning it's an ANCESTOR of the current recursive call, still sitting open on the call stack somewhere above the present point in the recursion. The three-state version distinguishes these explicitly: state 0 means unvisited, state 1 means "currently in progress" (on the active recursion path — the algorithm has started exploring this node but hasn't finished all of its descendants yet), and state 2 means "fully done" (this node and everything reachable from it has been completely explored, safe to skip). The critical check happens when DFS, while exploring some node's neighbors, encounters a neighbor that's in STATE 1 — this specifically means the neighbor is an ancestor of the current node in the recursion (it's still "open," meaning there's a path from that ancestor down to the current node), and now the current node has an edge pointing BACK to that ancestor. An edge from a descendant back to one of its own ancestors is, by definition, exactly what forms a CYCLE (sometimes called a "back edge" in graph terminology) — following the graph's edges from the ancestor, down through its descendants, and back to itself via this newly-found edge traces out a closed loop. Encountering state 2 (fully done) instead is completely harmless and expected — it just means "this subtree was already explored via a different path, no need to redo it" — and does NOT indicate a cycle, which is exactly why distinguishing "in progress" from "fully done" (rather than collapsing both into a single "visited" state) is essential for correct cycle detection.

🎯 Key Takeaways

  • Topological sort orders a DAG so every edge points from earlier to later — models "must happen before" dependencies.
  • Kahn's algorithm (BFS, track in-degree, process zero-in-degree nodes) naturally detects cycles: order.length < numNodes means some nodes were permanently stuck in a cycle.
  • DFS-based topological sort appends nodes in POST-order (after all descendants finish), so the raw result must be REVERSED at the end.
  • DFS-based cycle detection requires three states (unvisited/in-progress/done) — encountering an in-progress node means a back-edge, i.e., a cycle.
  • Both approaches solve the same problem; Kahn's is often more natural for "produce a valid schedule," DFS-based is natural when already traversing for another reason.
