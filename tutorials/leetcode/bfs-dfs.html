BFS / DFS Trees & Graphs

Concept: The two fundamental ways to visit every node in a tree or graph. DFS goes DEEP first (via recursion or an explicit stack); BFS goes WIDE first, level by level (via a queue). Choosing correctly between them — not just knowing both exist — is what separates a working solution from a timeout or wrong answer.

Recognition Cues

🔍 BFS: "shortest path" in an unweighted graph • "minimum number of steps/moves" • level-order traversal • "nearest X". DFS: "explore all paths" • connected components • cycle detection • tree traversals (in/pre/post-order) • when you need to go all the way down one branch before backtracking.

Template (BFS — shortest path / level order)

function bfs(graph, start) {
  const visited = new Set([start]);
  const queue = [start];
  let level = 0;
  while (queue.length) {
    const size = queue.length;                       // snapshot: process exactly ONE level per iteration
    for (let i = 0; i < size; i++) {
      const node = queue.shift();
      // process(node) at distance = `level`
      for (const neighbor of graph[node]) {
        if (!visited.has(neighbor)) {
          visited.add(neighbor);                       // mark visited when ENQUEUEING, not when dequeuing
          queue.push(neighbor);
        }
      }
    }
    level++;
  }
}
// The `size` snapshot trick is what makes BFS naturally level-aware — without it you can still BFS,
// but you lose the ability to report "which level/distance was this node found at."
⚠️ Mark visited when ENQUEUEING, not when dequeuing — the #1 BFS bug. If you mark a node visited only when you pop it from the queue, the SAME node can be pushed onto the queue multiple times by different neighbors before any of them get processed — wasting work and, in some variants, producing wrong shortest-distance answers. Mark it visited the moment you decide to enqueue it.

Template (DFS — recursive & iterative)

// Recursive — simplest to write, uses the CALL STACK as the traversal stack:
function dfs(node, visited = new Set()) {
  if (visited.has(node)) return;
  visited.add(node);
  // process(node)
  for (const neighbor of graph[node]) dfs(neighbor, visited);
}

// Iterative — explicit stack, avoids recursion depth limits on very deep/large graphs:
function dfsIterative(start) {
  const visited = new Set();
  const stack = [start];
  while (stack.length) {
    const node = stack.pop();
    if (visited.has(node)) continue;
    visited.add(node);
    // process(node)
    for (const neighbor of graph[node]) if (!visited.has(neighbor)) stack.push(neighbor);
  }
}
// Note: iterative DFS marks visited on POP (not push) — unlike BFS. Duplicates can sit in the stack
// briefly, but since we skip already-visited nodes on pop, correctness is preserved either way here.

Number of Islands — the Canonical Grid DFS/BFS Problem

function numIslands(grid) {
  const rows = grid.length, cols = grid[0].length;
  let count = 0;
  function sink(r, c) {
    if (r < 0 || r >= rows || c < 0 || c >= cols || grid[r][c] !== "1") return;
    grid[r][c] = "0";                                 // mark visited by MUTATING the grid — no extra Set needed
    sink(r+1, c); sink(r-1, c); sink(r, c+1); sink(r, c-1);
  }
  for (let r = 0; r < rows; r++)
    for (let c = 0; c < cols; c++)
      if (grid[r][c] === "1") { count++; sink(r, c); }
  return count;
}
// "Sink" every connected land cell once you find one — DFS from any unvisited "1" flood-fills the
// entire island. Reusing the grid itself as the visited-tracker is a classic space-saving trick.

Interview Questions

Q1: Why does BFS guarantee the shortest path in an unweighted graph, but DFS doesn't?

BFS explores the graph in expanding "rings" of distance from the start node: it fully processes every node at distance 1 before touching any node at distance 2, fully processes distance 2 before distance 3, and so on — this is exactly what the queue-based, level-by-level structure enforces. Because of this strict ordering, the FIRST time BFS ever reaches a particular target node, it's guaranteed to have done so via the shortest possible number of edges — there's no way to reach that node in FEWER steps, because BFS would have already found it during an earlier, smaller-distance ring if a shorter path existed. DFS has no such guarantee: it commits to exploring one branch as deep as possible before backtracking, so it might stumble onto the target via a long, winding path long before it would have found a much shorter path through a completely different branch it hasn't explored yet — DFS finds SOME path to the target, and there's no structural reason that path has to be the shortest one. This is precisely why "shortest path" or "minimum number of steps" language in a problem statement (for an UNWEIGHTED graph — weighted graphs need Dijkstra's algorithm instead) is one of the strongest signals to reach for BFS specifically.

Q2: Why must you mark a node as visited when it's ENQUEUED in BFS, rather than when it's dequeued/processed?

Consider a node N that has two different neighbors, A and B, both currently being processed at the same BFS level. If A processes first and looks at its neighbor N, and visited-marking only happens when N is later DEQUEUED (not now, when it's merely being considered for enqueueing), then A will push N onto the queue. But then B, processed shortly after within the SAME level, ALSO looks at its neighbor N — and since N hasn't been dequeued yet (it's just sitting in the queue, not yet marked visited), B has no way of knowing N is already scheduled to be visited, so B pushes N onto the queue AGAIN. Now N appears in the queue twice, and both copies will eventually be dequeued and processed redundantly — wasted work that, at scale, can turn what should be O(V+E) into something significantly slower as duplicate entries proliferate through a densely-connected graph. In problems where BFS is being used specifically to compute shortest DISTANCES (not just visit every node), this can cause more than just inefficiency — it can lead to a node being assigned an incorrect (larger) distance if it gets processed via a later, longer path before its earlier, shorter-path duplicate entry is reached in the queue. Marking a node visited the MOMENT it's enqueued — rather than when it's later processed — closes this gap entirely: once A enqueues N and marks it visited, B's subsequent check of "has N been visited?" correctly returns true, and B never enqueues a duplicate.

Q3: When would you choose iterative DFS over recursive DFS, and what's the key difference in when nodes are marked visited?

Recursive DFS is usually the simplest to write and reason about — the call stack implicitly tracks the path back to each ancestor, and the code closely mirrors the natural recursive definition of "visit this node, then recursively visit each unvisited neighbor." The practical limitation is that very deep or very large graphs (a long chain, a deeply nested tree, or any input where the recursion depth could reach into the tens of thousands) risk a STACK OVERFLOW, since each recursive call consumes a frame on a call stack whose size is bounded by the runtime environment (this limit varies by language/platform but is a real, practical concern in interviews when input size constraints suggest very deep structures). The iterative version replaces the implicit call stack with an EXPLICIT stack data structure that lives on the heap rather than the call stack, so its only practical size limit is available memory, not a fixed call-depth ceiling — making it the safer choice for pathologically deep/large inputs. One subtle but important difference in the iterative version: because a node can be pushed onto the explicit stack multiple times by different neighbors BEFORE any of those pushes are actually processed (similar to the BFS duplicate-enqueue scenario, but here it's usually harmless), the iterative version typically marks a node visited when it's POPPED and actually processed, then skips any node it pops that's already been marked visited — rather than marking on push, the way BFS does for correctness reasons. This works correctly for DFS because, unlike BFS's shortest-path guarantee, DFS doesn't depend on strict ordering to remain correct — duplicate stack entries just result in a cheap "already visited, skip" check rather than any wrong-answer risk.

🏋️ Practice: implement both BFS and DFS on an adjacency-list graph and trace the visit order on paper first; solve "shortest path in a binary maze" with BFS and verify it finds the minimum steps; solve "number of islands" with DFS flood-fill, then re-solve it with BFS to compare; implement DFS both recursively and iteratively on a deep linked structure and observe the stack-overflow difference at scale; solve "clone graph" using either traversal with a visited map tracking original→copy node mappings.

🎯 Key Takeaways

  • BFS (queue, level-by-level) guarantees shortest path in unweighted graphs; DFS (stack/recursion, depth-first) does not.
  • Mark nodes visited on ENQUEUE for BFS (prevents duplicate/incorrect processing); DFS conventionally marks on pop/process instead.
  • Snapshot the queue size before each BFS round to process exactly one level at a time — required for level-aware BFS.
  • Recursive DFS is simplest but risks stack overflow on very deep inputs; iterative DFS with an explicit stack avoids that limit.
  • Grid flood-fill problems (Number of Islands) can reuse the grid itself as the visited-tracker, avoiding extra space.
