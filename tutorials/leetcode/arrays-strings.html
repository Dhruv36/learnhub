Arrays & Strings Essentials Foundation

Concept: Nearly every pattern in this track OPERATES on arrays and strings — so the in-place tricks, common gotchas, and building-block operations here aren't a separate pattern, they're the vocabulary every other pattern is written in.

In-Place Modification (O(1) extra space)

// Removing duplicates from a SORTED array in place, returning the new length:
function removeDuplicates(nums) {
  if (nums.length === 0) return 0;
  let write = 1;                              // where the next unique value goes
  for (let read = 1; read < nums.length; read++) {
    if (nums[read] !== nums[write - 1]) {
      nums[write] = nums[read];
      write++;
    }
  }
  return write;
}
// The "read/write pointer" pattern: READ scans everything, WRITE only advances for values you KEEP.
// This same skeleton solves "remove element", "move zeroes", "partition array" — a foundational template.

String Immutability — a Cross-Language Gotcha

// In JS, Python, Java, C#: strings are IMMUTABLE. Every "modification" allocates a NEW string.
let s = "";
for (let i = 0; i < n; i++) s += chars[i];    // O(n²) — n allocations, each copying growing content!

// Use a mutable buffer instead:
const buf = [];
for (let i = 0; i < n; i++) buf.push(chars[i]);
const s2 = buf.join("");                       // O(n) — one allocation at the end
// (Java/C#: StringBuilder. Python: "".join(list) or a list, same idea.)
⚠️ String concatenation in a loop is the single most common accidental O(n²) in interview code. Because strings are immutable across nearly every mainstream language, `result += char` inside a loop copies the ENTIRE accumulated string on every iteration. This is subtle because the code LOOKS linear. Always build with an array/buffer and join once at the end.

Prefix/Suffix Precomputation

// A common trick: precompute prefix and suffix aggregates in O(n), then answer each position in O(1).
// Example: "product of array except self" without division:
function productExceptSelf(nums) {
  const n = nums.length, res = new Array(n).fill(1);
  let prefix = 1;
  for (let i = 0; i < n; i++) { res[i] = prefix; prefix *= nums[i]; }
  let suffix = 1;
  for (let i = n - 1; i >= 0; i--) { res[i] *= suffix; suffix *= nums[i]; }
  return res;
}
// res[i] ends up as (product of everything left of i) × (product of everything right of i) — two passes,
// O(n) time, O(1) EXTRA space (the output array doesn't count).

Frequency Counting — the Universal First Move

// "Are two strings anagrams?" / "Most frequent element?" / "First unique character?" — almost always
// start with a frequency map:
const freq = new Map();
for (const ch of s) freq.set(ch, (freq.get(ch) || 0) + 1);
// For a KNOWN small alphabet (lowercase letters), an array of 26 counters beats a Map — faster, less
// overhead, and the size hints at "this is a frequency-counting problem" immediately.

Interview Questions

Q1: Why is `result += char` in a loop O(n²), and what's the fix?

Strings are immutable in essentially every mainstream language (JS, Python, Java, C#), so the += operator doesn't append in place — it allocates a brand-new string containing the old content plus the new character, then copies everything into it. On iteration k of an n-iteration loop, that copy costs O(k) (the accumulated length so far), so the total cost across all n iterations is 1+2+3+...+n ≈ O(n²), even though it's a single, innocent-looking loop. The fix is to accumulate into a MUTABLE structure — a plain array with push(), or a language's dedicated mutable-string-builder type (StringBuilder in Java/C#) — and only convert to a final string ONCE, at the end (.join("") in JS/Python). Appending to a dynamic array is O(1) amortized per operation (per the Big-O lesson's amortized-cost discussion), so the whole loop becomes O(n) total instead of O(n²).

Q2: Explain the "read/write pointer" pattern used for in-place array modification.

This pattern solves a whole family of problems — remove duplicates, remove a specific value, move zeroes to the end, partition by a condition — using two indices scanning the SAME array: a READ pointer that examines every element unconditionally, and a WRITE pointer that only advances when the current element should be KEPT in the final result. As the read pointer scans left to right, each time it finds an element that belongs in the output, that element is copied to the position the write pointer currently points at, and then the write pointer advances by one; elements that don't belong (a duplicate, a zero to be moved, a value to be removed) are simply skipped by read without ever touching write. Because write only ever moves forward and read is always at or ahead of write, you can safely overwrite arr[write] with arr[read] without ever destroying data read hasn't reached yet. This achieves the transformation using O(1) extra space (no second array) and O(n) time (a single pass), and the final value of write is exactly the length of the resulting (possibly shorter) valid portion of the array.

Q3: How does prefix/suffix precomputation turn an O(n²) problem into O(n)?

Many problems ask you to answer a question about "everything to the left/right of position i" for EVERY position i — like "product of the array except self" or "is there a valid split point where the left sum equals the right sum." A naive approach recomputes the relevant left-side (or right-side) aggregate from scratch for each position i, which costs O(n) per position and O(n²) total across all n positions. The precomputation trick instead does a single O(n) forward pass to build a PREFIX array where prefix[i] holds the aggregate (sum, product, max, etc.) of everything strictly before index i, and optionally a second O(n) backward pass for a SUFFIX array holding the same for everything after index i. Once both are built, answering the question for any individual position i becomes an O(1) lookup — just read prefix[i] and/or suffix[i] and combine them — rather than re-scanning the array. The total cost drops from O(n²) (n positions × O(n) per position) to O(n) (two linear passes to build the prefix/suffix arrays, plus O(1) per position to answer using them) — trading a small amount of extra O(n) space for a fundamentally faster algorithm.

🏋️ Practice: implement removeDuplicates and moveZeroes using the read/write pointer pattern; benchmark string concatenation vs array-join on building a 100,000-character string; implement productExceptSelf with prefix/suffix arrays, then again with O(1) extra space (reuse the output array for the prefix pass); solve "first unique character in a string" with a frequency array; solve "valid anagram" with a 26-length counter array.

🎯 Key Takeaways

  • Strings are immutable — always accumulate into an array/buffer and join once; loop concatenation is an accidental O(n²).
  • The read/write pointer pattern solves in-place array modification (dedupe, remove, partition) in O(n) time, O(1) space.
  • Prefix/suffix precomputation turns "answer this for every position" from O(n²) into O(n) by trading a small amount of space.
  • Frequency counting (Map or fixed-size array for small alphabets) is the default first move for anagram/uniqueness/frequency problems.
  • These aren't a separate "pattern" — they're building blocks every other pattern in this track assumes you already know.
