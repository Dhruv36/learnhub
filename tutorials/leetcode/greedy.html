Greedy Advanced Pattern

Concept: Make the LOCALLY best choice at each step, never reconsidering it, and trust that this produces the GLOBALLY optimal answer. Greedy is fast (usually O(n log n) from a sort, then O(n)) but only correct for problems with a specific structural property — proving that property is the real skill, not writing the loop.

Recognition Cues

🔍 "Minimum number of X to cover/achieve Y" • interval scheduling ("maximum non-overlapping intervals") • "assign X to Y to maximize/minimize Z" with a natural sorting order • Jump Game (can you reach the end) • Gas Station • problems where sorting by SOME criterion, then making one pass, intuitively "feels" like it should work.

The Exchange Argument — How to PROVE Greedy Is Correct

// Greedy is easy to CODE and dangerously easy to get WRONG — it's not obviously correct just because
// it "feels reasonable." The standard proof technique is an EXCHANGE ARGUMENT:
//   1. Assume an OPTIMAL solution exists that does NOT match the greedy choice at some step.
//   2. Show you can SWAP/EXCHANGE that optimal solution's choice for the greedy one...
//   3. ...WITHOUT making the solution any WORSE (often it becomes at least as good).
//   4. Conclude: the greedy choice is ALWAYS at least as good as any alternative — safe to commit to it.
// If you can't construct this argument, greedy is likely WRONG for the problem — this is the real skill
// interviewers are testing, more than the final code.

Interval Scheduling — a Greedy Proof Walkthrough

function eraseOverlapIntervals(intervals) {
  intervals.sort((a, b) => a[1] - b[1]);           // sort by END time — this specific choice is the crux
  let count = 0, prevEnd = -Infinity;
  for (const [start, end] of intervals) {
    if (start >= prevEnd) {                          // no overlap — keep this interval
      prevEnd = end;
    } else {
      count++;                                        // overlap — must remove ONE (this one, since it ends later)
    }
  }
  return count;                                       // minimum removals to make the rest non-overlapping
}
// WHY sort by end time (not start time): the interval that ends EARLIEST leaves the MOST room for
// future intervals to also fit without overlapping. Exchange argument: if an optimal solution kept a
// DIFFERENT interval instead of the one ending earliest, swapping in the earliest-ending one can only
// free up MORE room for subsequent choices — never less. So greedily keeping earliest-end never hurts.

Jump Game — Greedy Reachability

function canJump(nums) {
  let maxReach = 0;
  for (let i = 0; i < nums.length; i++) {
    if (i > maxReach) return false;                   // can't even REACH index i — stuck
    maxReach = Math.max(maxReach, i + nums[i]);         // greedily track the farthest reachable index
  }
  return true;
}
// The greedy insight: you don't need to track EVERY possible path — only the single number "farthest
// index reachable so far." If that number ever falls behind your current position, no path exists;
// otherwise, SOME sequence of jumps gets you to the end, and you don't need to know WHICH one.

Greedy vs Dynamic Programming — the Critical Distinction

GreedyDynamic Programming
Decision per stepONE choice, committed forever, never reconsideredExplores/remembers multiple possibilities, picks the best in hindsight
RequiresProvable "locally optimal → globally optimal" propertyOverlapping subproblems + optimal substructure (weaker requirement)
SpeedUsually O(n log n) (sort) then O(n)Usually O(n²) or worse (table of subproblems)
When greedy FAILSUse DP instead — it considers possibilities greedy discardsAlways correct if the recurrence is right (slower, but safe)
⚠️ When in doubt, DP is always safe; greedy is a targeted speed optimization that requires proof. A classic interview trap is applying greedy to a coin-change-style problem with ARBITRARY denominations (where it can genuinely fail — e.g., coins [1,3,4] making 6: greedy picks 4+1+1=3 coins, but 3+3=2 coins is better) versus a well-behaved system where greedy happens to work (standard currency denominations). Always ask "can I construct a counter-example?" before trusting greedy.

Interview Questions

Q1: What is the "exchange argument," and why is it the standard way to justify a greedy algorithm's correctness?

A greedy algorithm commits to one choice at each step and never reconsiders it — which is only correct if that locally-optimal choice can be proven to never prevent reaching a globally-optimal overall solution. The exchange argument is the standard proof technique for establishing this: you START by assuming, for contradiction, that some OPTIMAL solution exists which made a DIFFERENT choice than what the greedy algorithm would make at some particular step. You then show that this optimal solution can be MODIFIED — by "exchanging" its choice at that step for the greedy algorithm's choice instead — WITHOUT making the solution any worse (and often while keeping it equally optimal, or sometimes strictly better). If you can construct this argument successfully for every step, it proves that the greedy choice is always AT LEAST AS GOOD as whatever any hypothetical optimal solution could have chosen instead — meaning committing to the greedy choice can never cost you anything relative to optimality, which is exactly the guarantee needed to justify never reconsidering it later. This matters enormously in an interview context because simply writing a greedy-LOOKING algorithm (sort by some criterion, then scan once) is easy and often produces code that passes several test cases even when the underlying approach is actually incorrect for some inputs — the exchange argument (or explicitly failing to construct one, and instead finding a counter-example) is what actually distinguishes a genuinely correct greedy algorithm from one that merely looks plausible.

Q2: In interval scheduling (maximizing non-overlapping intervals / minimizing removals), why does sorting by END time work, while sorting by START time or by interval LENGTH does not?

The key insight is what quantity you're actually trying to preserve for future decisions: available "room" for subsequent intervals to also fit without overlapping. Among all intervals that could be chosen first, the one that ENDS earliest leaves the maximum possible remaining space on the timeline for every interval that comes after it — no other choice could leave MORE room, since any interval ending later necessarily blocks out at least as much of the timeline going forward. This is precisely what the exchange argument formalizes: if some hypothetical optimal solution chose a DIFFERENT interval at this step instead of the one with the earliest end time, you could swap it out for the earliest-ending one instead — since the earliest-ending interval finishes AT LEAST as early, it can only free up equal or MORE room for every subsequent choice, meaning the swap never makes the solution worse, and might even allow additional intervals to be fit in later that the original optimal solution's choice would have blocked. Sorting by START time, by contrast, doesn't have this property: two intervals can have very different end times despite similar start times, so choosing based on start time doesn't reliably preserve maximum "room" for the future — you could pick an interval that starts early but drags on for a very long time, unnecessarily blocking out a large portion of the timeline compared to a different early-starting-but-short interval you could have chosen instead. Sorting by interval LENGTH similarly fails, because a short interval that occurs LATE in the timeline doesn't help you make progress through the earlier portion of the schedule at all — length alone says nothing about POSITION, which is what actually determines overlap with everything that comes after. End time is specifically the quantity that directly and reliably measures "how much room does this choice leave for the future," which is exactly why it's the correct sort key here.

Q3: Give a concrete example of a coin-change scenario where greedy fails, and explain why DP is needed instead.

Consider coin denominations [1, 3, 4] and a target amount of 6. A greedy approach — always pick the LARGEST denomination that doesn't exceed the remaining amount — would proceed: remaining 6, largest fitting coin is 4, pick it (remaining becomes 2); remaining 2, largest fitting coin is now 1 (since 3 and 4 both exceed 2), pick it (remaining becomes 1); remaining 1, pick a 1 (remaining becomes 0). Total: 4 + 1 + 1 = THREE coins. But the actual OPTIMAL solution for making 6 from {1, 3, 4} is 3 + 3 = TWO coins — strictly better than what greedy found. Greedy fails here because its "always take the biggest coin that fits" heuristic has no way to "look ahead" and recognize that taking a smaller coin NOW (a 3, instead of the locally-tempting 4) sets up a much better outcome for the REMAINING amount — greedy's core weakness is exactly this inability to reconsider an earlier choice once a later choice reveals it wasn't actually optimal. This is precisely why general coin change (with ARBITRARY, unspecified denomination sets, which is how the problem is typically posed in interviews) requires dynamic programming instead: DP effectively explores the relevant space of possibilities (via the recurrence dp[amount] = 1 + min over each coin of dp[amount - coin]) and picks the genuinely optimal combination by considering every viable option at each amount, rather than committing irrevocably to whichever single choice looks best in the moment. It's worth noting that greedy DOES work correctly for "canonical" denomination systems — including standard currency systems like US coins {1, 5, 10, 25} — where a mathematical property of the denomination set happens to guarantee the greedy choice is always safe; but this correctness is a special property of THOSE SPECIFIC denomination sets, not a general guarantee, which is exactly why "does greedy actually work here, or do I need to prove/test it" is the central judgment call this entire pattern is about.

🎯 Key Takeaways

  • Greedy commits to the locally-best choice at each step, never reconsidering — fast (often O(n log n)) but only correct for problems with a provable "locally optimal → globally optimal" property.
  • The exchange argument is the standard proof technique: show any optimal solution can be modified to match the greedy choice without becoming worse.
  • Interval scheduling's correct sort key is END time specifically because it maximizes remaining "room" for future choices — start time or length don't have this property.
  • Greedy fails on general coin change with arbitrary denominations (e.g., [1,3,4] targeting 6) — DP is the safe, always-correct fallback that greedy is only a speed optimization over.
  • When you can't construct an exchange argument (or can find a counter-example), don't trust greedy — reach for DP instead.
