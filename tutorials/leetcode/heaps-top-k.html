Heaps / Top-K Advanced Pattern

Concept: A heap gives O(log n) insertion and O(1) access to the min (or max) element. For "find the top K" problems, maintaining a heap of size K turns an O(n log n) full-sort into O(n log K) — a real asymptotic win whenever K is much smaller than n.

Recognition Cues

🔍 "Kth largest/smallest element" • "top K frequent elements" • "K closest points" • merging K sorted lists • "median of a data stream" (continuously arriving data) • anywhere you need REPEATED access to the current min/max as the data set changes.

The Counter-Intuitive Core Trick: MIN-heap for TOP-K LARGEST

// To find the K LARGEST elements, maintain a MIN-heap of size K (not a max-heap!):
function findKLargest(nums, k) {
  const minHeap = new MinHeap();                    // (implementation-specific; conceptually a priority queue)
  for (const num of nums) {
    minHeap.push(num);
    if (minHeap.size() > k) minHeap.pop();            // evict the SMALLEST — it can't be in the top K
  }
  return minHeap.toArray();                           // whatever survives IS the K largest
}
// Why a MIN-heap: you need FAST access to the SMALLEST element currently in your "top K so far" set,
// because that's the one candidate to evict when a bigger number shows up. A max-heap would give you
// fast access to the LARGEST — the wrong thing to check against new incoming values.
🎯 This inversion (min-heap for top-K-largest) is the single most-tested heap insight in interviews. The heap's size stays capped at K throughout, so each push/pop is O(log K), not O(log n) — across n elements, total cost is O(n log K), meaningfully better than sorting everything (O(n log n)) when K ≪ n. Symmetrically, use a MAX-heap for finding the K SMALLEST elements.

Top K Frequent Elements — Combining Hashing + Heap

function topKFrequent(nums, k) {
  const freq = new Map();
  for (const n of nums) freq.set(n, (freq.get(n) || 0) + 1);   // Step 1: count frequencies — O(n)

  const minHeap = new MinHeap((a, b) => a.count - b.count);     // heap ordered by COUNT, not value
  for (const [num, count] of freq) {
    minHeap.push({num, count});
    if (minHeap.size() > k) minHeap.pop();                       // evict the LEAST frequent
  }
  return minHeap.toArray().map(x => x.num);                      // O(n log k) total
}
// A very common two-step composition: frequency-count with a hash map FIRST, then run the "top K"
// heap pattern on the (value, count) pairs instead of raw values.

Merge K Sorted Lists

function mergeKLists(lists) {
  const minHeap = new MinHeap((a, b) => a.val - b.val);
  for (const list of lists) if (list) minHeap.push(list);   // seed with each list's HEAD — O(k log k)

  const dummy = { next: null };
  let tail = dummy;
  while (minHeap.size()) {
    const node = minHeap.pop();                               // always the SMALLEST current head — O(log k)
    tail.next = node;
    tail = node;
    if (node.next) minHeap.push(node.next);                    // push that list's NEXT node
  }
  return dummy.next;
}
// O(N log k) total, where N is the total number of nodes across all lists and k is the number of lists —
// far better than merging lists pairwise (O(N·k)) for large k.

Common Mistakes

Forgetting to cap the heap at size KHeap grows to size n — no asymptotic benefit over full sortingAlways pop after pushing if size exceeds K

Rebuilding a heap from scratch for a data STREAMLoses the "maintain the answer incrementally" benefitPush/pop incrementally as each new element arrives — this is the pattern's whole point for streams
Not seeding a merge-K-lists heap with all list heads upfrontMisses some lists' contributions entirelyInitialize the heap with every list's head node before the main loop

Interview Questions

Q1: Why do you use a MIN-heap to find the K LARGEST elements, rather than a max-heap?

This inversion feels counter-intuitive at first, but it follows directly from asking "what operation do I actually need to perform repeatedly?" The algorithm maintains a heap capped at size K, and processes the input one element at a time: for each new element, it's added to the heap, and if the heap now has more than K elements, ONE element must be evicted to bring it back down to size K. The element that should be evicted is whichever one is LEAST likely to belong in the final "top K largest" answer — which is the SMALLEST element currently in the heap, since if the heap is maintaining candidates for "largest," the smallest among those candidates is the weakest one and the first to go if something bigger comes along. To efficiently find and remove "the smallest element currently held," you need a data structure that gives fast access to the MINIMUM — which is precisely what a MIN-heap provides (O(1) peek, O(log K) removal of the minimum). A max-heap would instead give you fast access to the maximum of the current K candidates — information that isn't useful here, since you're never trying to evict the largest element you're holding onto; you want to evict the smallest one when a new, larger candidate arrives. By symmetry, finding the K SMALLEST elements uses the opposite structure: a MAX-heap of size K, evicting the largest candidate when a smaller one arrives.

Q2: Why does bounding the heap to size K improve complexity from O(n log n) to O(n log K)?

Sorting the entire input array to find the top K elements costs O(n log n), because a general comparison-based sort has to fully order ALL n elements relative to each other, even though the problem only ultimately cares about the relative order of the top K — the remaining n-K elements' precise ordering among themselves is completely irrelevant to the final answer. The bounded-heap approach avoids doing that unnecessary work: it processes each of the n input elements ONE AT A TIME, and for each one performs a constant number of heap operations (a push, and possibly a pop if the heap exceeds size K) — critically, because the heap is capped at K elements throughout the ENTIRE algorithm, every individual push or pop operation costs O(log K), not O(log n), since a heap operation's cost is proportional to the heap's current SIZE, and that size never exceeds K. Across all n input elements, the total cost is therefore n × O(log K) = O(n log K). When K is much smaller than n (say, finding the top 10 out of a million elements), O(n log K) is dramatically faster than O(n log n) — you're paying a cost proportional to log(10) per element instead of log(1,000,000), even though you still touch every one of the n elements exactly once.

Q3: How does "Merge K Sorted Lists" use a heap to achieve O(N log k) instead of a worse complexity?

The algorithm seeds a min-heap with the HEAD node of each of the k input lists (a one-time cost of O(k log k) to build the initial heap). Then it repeatedly: pops the smallest-valued node currently in the heap (O(log k), since the heap never holds more than k elements — one representative per list at any given time), appends that node to the growing merged output list, and — critically — if the popped node had a `next` pointer (meaning its originating list has more elements), pushes that NEXT node from the SAME list into the heap to represent that list going forward. This means at every point in the algorithm, the heap holds AT MOST k elements: exactly one "current candidate" node from each list that still has remaining nodes. Since there are N total nodes across all k lists combined, and each node is pushed into the heap exactly once and popped exactly once over the course of the entire algorithm, the total number of heap operations is O(N), each costing O(log k) because the heap size never exceeds k — giving O(N log k) overall. Compare this to a naive pairwise-merge approach that merges two lists at a time and repeats: merging all k lists pairwise (in a straightforward sequential fashion, not a balanced merge tree) costs O(N·k) in the worst case, since each of the roughly k successive merge passes touches close to N nodes. The heap-based approach is asymptotically better whenever k is reasonably large, because it replaces "compare against ALL other current candidates each time" (implicit in naive pairwise merging) with "compare against a heap of size k," extracting the true global minimum in O(log k) rather than O(k) per step.

🏋️ Practice: implement "Kth Largest Element in an Array" using a size-K min-heap and compare against a full-sort baseline for large n and small k; solve "Top K Frequent Elements" combining a frequency hash map with a bounded heap; implement "Merge K Sorted Lists" with a heap and verify it against pairwise merging for correctness and (conceptually) complexity; solve "Find Median from Data Stream" using TWO heaps (a max-heap for the lower half, a min-heap for the upper half); solve "K Closest Points to Origin" with a bounded max-heap ordered by distance.

🎯 Key Takeaways

  • A heap gives O(log n) insertion and O(1) access to the min/max — bounding heap size to K turns O(n log n) sorting into O(n log K).
  • Counter-intuitive but critical: use a MIN-heap to find the K LARGEST elements (evict the smallest candidate); MAX-heap for K SMALLEST.
  • Top-K-frequent composes hashing (count frequencies) with the bounded-heap pattern applied to (value, count) pairs.
  • Merge K Sorted Lists seeds a heap with one node per list, achieving O(N log k) by never holding more than k candidates at once.
  • Heaps shine specifically for STREAMING or incrementally-arriving data where you need the current min/max maintained continuously, not just computed once.
