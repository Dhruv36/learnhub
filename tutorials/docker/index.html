Introduction: Containers vs VMs Basics

Docker packages an application and everything it needs to run — code, runtime, libraries, config — into a portable container that runs identically on any machine with Docker. It ended "works on my machine" forever. Understanding how containers differ from virtual machines — sharing the host kernel rather than emulating hardware — is the foundation everything else builds on.

The Problem Docker Solves

// "It works on my machine" — different OS, library versions, config, dependencies between
// dev/staging/prod cause bugs that appear only in some environments. Docker packages the app
// WITH its environment into an immutable image, so it runs identically everywhere:
//   dev laptop === CI runner === staging === production
// Deploy = "run this exact image". No more environment drift.

Containers vs Virtual Machines — the Core Distinction

│    Hypervisor       │                 │   Docker Engine     │
├─────────────────────┤                 ├─────────────────────┤
│    Host OS          │                 │    Host OS (kernel) │
└─────────────────────┘                 └─────────────────────┘
Virtual MachineContainer
IsolationFull OS per VM (hardware-level, hypervisor)Process-level, shares the host KERNEL
SizeGBs (includes a guest OS)MBs (just the app + its deps)
StartupSeconds to minutes (boot an OS)Milliseconds (start a process)
DensityA few per hostHundreds per host
Isolation strengthStronger (separate kernels)Weaker (shared kernel) — a kernel exploit can escape
🧩 The key insight: containers don't virtualize hardware, they isolate processes. A container is just a normal Linux process with restricted visibility — it uses the host's kernel but sees its own isolated filesystem, network, and process tree (via namespaces and cgroups — the internals lesson). That's why containers are so much lighter and faster than VMs: there's no guest OS to boot, no hardware to emulate. The trade-off is weaker isolation (shared kernel).

Your First Container

docker run hello-world                 // pull + run a test image
docker run -d -p 8080:80 nginx         // run nginx detached, map host:8080 → container:80
docker ps                              // list running containers
docker stop <id>   docker rm <id>      // stop and remove
docker images                          // list local images
// "docker run" = pull the image if needed, create a container, start it. That simple.

Interview Questions

Q1: How do containers differ from virtual machines?

A VM virtualizes hardware: a hypervisor runs multiple guest operating systems, each with its own kernel, on top of the host — strong isolation but heavy (GBs, seconds-to-boot, a few per host). A container virtualizes at the OS level: it's an isolated PROCESS that shares the host's kernel, given its own view of the filesystem, network, and process tree via Linux namespaces and resource limits via cgroups. Because there's no guest OS to boot or hardware to emulate, containers are tiny (MBs), start in milliseconds, and pack hundreds per host — at the cost of weaker isolation (a shared kernel, so a kernel vulnerability could allow escape). Containers for density/speed/portability; VMs when you need stronger isolation or different kernels.

Q2: What problem does Docker actually solve, and why is that valuable?

It eliminates environment drift — the "works on my machine" class of bugs — by packaging an application together with its exact runtime, libraries, and configuration into an immutable image that runs identically on any host with Docker (dev laptop, CI, staging, production). Deployment becomes "run this exact image" rather than "install and configure the app on this server", making builds reproducible and rollbacks trivial (run the previous image). Beyond consistency, the lightweight process-isolation model gives high density (hundreds per host → lower cost), millisecond startup (enabling autoscaling, fast CI, ephemeral test environments), and portability across clouds — and it's the packaging unit orchestrators like Kubernetes schedule.

Q3: When should you NOT use containers, or need something stronger?

When you're running untrusted or hostile code and need strong isolation: because containers share the host kernel, a kernel-level vulnerability can let a container escape to the host or other containers, so multi-tenant execution of arbitrary code warrants VM-level isolation (separate kernels) or sandboxed runtimes like gVisor or Firecracker (micro-VMs). You also don't containerize for its own sake when a workload genuinely needs a different OS kernel (Windows containers need a Windows kernel; Linux containers need Linux — Docker Desktop runs a Linux VM under the hood on Mac/Windows to provide one). And for very stateful, single-instance legacy systems the operational overhead may not pay off. The rule: containers for portable, scalable, trusted app workloads; stronger isolation for untrusted multi-tenant code.

🏋️ Practice (install Docker): run hello-world; run nginx detached with a port mapping and visit it in a browser; docker ps / stop / rm; pull and run an interactive ubuntu container (docker run -it ubuntu bash) and explore that it shares your kernel (uname -r matches the host); observe how fast a container starts vs booting a VM; list images and containers.

🎯 Key Takeaways

  • Docker packages an app + its environment into a portable, immutable image — kills "works on my machine".
  • Containers isolate PROCESSES sharing the host kernel (namespaces + cgroups); VMs virtualize hardware with separate kernels.
  • Containers: MBs, millisecond start, hundreds per host, weaker isolation. VMs: GBs, slow, stronger isolation.
  • Core concepts: image (template) → container (instance), Dockerfile (recipe), registry (storage), volume (persistence).
  • Containers are the deployment unit of modern software and the foundation for Kubernetes; use VMs/sandboxes for untrusted code.
