Volumes & Networking Basics

Containers are ephemeral — remove one and its filesystem changes vanish. Volumes provide persistent storage that outlives containers; networks let containers talk to each other and the outside world. Both are essential the moment you run anything with state or more than one container.

Why Volumes — the Ephemerality Problem

docker run -d --name db postgres        // writes data to its writable layer
docker rm -f db                          // 💀 ALL data gone (writable layer deleted)
// The container's writable layer is disposable (images-containers lesson). Anything you must
// KEEP — databases, uploads, logs — belongs OUTSIDE the container, in a volume.

Volumes vs Bind Mounts

// NAMED VOLUME — Docker manages the storage (preferred for production data):
docker volume create pgdata
docker run -d -v pgdata:/var/lib/postgresql/data postgres
// The volume persists across container removal/recreation; Docker stores it in its own area.

// BIND MOUNT — map a HOST directory into the container (great for DEV: live code editing):
docker run -v $(pwd):/app node:22 npm run dev
// Edits on the host appear instantly in the container. Tied to the host path/layout.

💾 The rule: containers are stateless; state lives in volumes. Design containers so they can be killed and recreated at any moment with no data loss — because orchestrators (Kubernetes) do exactly that. All persistent state goes to volumes (or an external managed database, which is often better than running stateful databases in containers at all).

Docker Networking — the Bridge Model

// By default, Docker creates a BRIDGE network; containers on it get private IPs and can reach
// each other. You expose containers to the host by PUBLISHING ports:
docker run -d -p 8080:80 nginx           // host:8080 → container:80
//                └ -p HOST:CONTAINER (publish). EXPOSE alone does NOT do this.
docker run -d -p 127.0.0.1:8080:80 nginx // bind only to localhost (not all interfaces)

User-Defined Networks & DNS (the key production pattern)

// Create a network; containers on it can reach each other BY NAME (built-in DNS):
docker network create appnet
docker run -d --name db  --network appnet postgres
docker run -d --name api --network appnet myapi     // connects to the DB via hostname "db"!
// Inside the api container:  DATABASE_URL=postgres://db:5432/mydb   ← "db" resolves to the DB container
// User-defined networks give automatic service DISCOVERY by container name — no hardcoded IPs.
// The default bridge does NOT provide name resolution; always use a user-defined network for
// multi-container apps (or Docker Compose, which creates one for you — compose lesson).

Interview Questions

Q1: Why do you need volumes, and named volume vs bind mount?

Because a container's writable layer is ephemeral — removing the container deletes any data written inside it — so anything that must survive container recreation (databases, uploads, logs) has to live outside the container in a volume. A named volume is storage Docker manages in its own area, referenced by name; it persists across container removal, is portable/host-agnostic, and is the right choice for persistent production data. A bind mount maps a specific host directory into the container; it's tied to the host's filesystem layout and is ideal for development (edit code on the host and see it live in the container). The guiding principle is stateless containers + state in volumes, so containers can be freely killed and recreated (as orchestrators do) without data loss — and for databases specifically, an external managed service is often preferable to running stateful containers.

Q2: How do containers communicate, and why use a user-defined network?

By default Docker puts containers on a bridge network where they get private IPs and can reach each other, while access from the host requires publishing ports (-p HOST:CONTAINER); EXPOSE alone only documents a port and doesn't publish it. The key production pattern is a user-defined network: containers attached to one can resolve each other by CONTAINER NAME via Docker's built-in DNS, so an api container can connect to a db container using the hostname "db" instead of a hardcoded IP — which is essential because container IPs change on restart. The default bridge does NOT provide this name resolution, so multi-container apps should use a user-defined network (or Docker Compose, which creates one automatically) to get stable service discovery by name.

Q3: What's the difference between EXPOSE and -p, and the host network mode?

EXPOSE in a Dockerfile is documentation only — it declares which port the app listens on but does not make it reachable from the host; publishing with -p HOST:CONTAINER at run time is what actually maps a host port to the container port (and you can bind to 127.0.0.1 to restrict access to localhost). The default bridge mode isolates the container's network and requires this publishing. Host network mode (--network host) instead shares the host's network stack directly: the container uses host ports with no mapping and no network namespace isolation — giving maximum network performance (no NAT/proxy overhead) at the cost of isolation and port-conflict safety, so it's used for specific high-throughput or host-integration cases, not as a default.

🏋️ Practice: run postgres with a named volume, write data, remove and recreate the container, and confirm the data persists; run it WITHOUT a volume and see data lost; bind-mount your source into a node container for live editing; create a user-defined network, run a db and an api on it, and connect from api to db by name; publish a port bound to 127.0.0.1 and confirm it's not reachable externally.

🎯 Key Takeaways

  • Container filesystems are ephemeral — persistent state (DBs, uploads) goes in volumes (or an external DB).
  • Named volumes = Docker-managed persistent data (production); bind mounts = host dir mapped in (dev live-editing).
  • Containers are stateless by design so they can be killed/recreated freely (as orchestrators do).
  • Publish ports with -p HOST:CONTAINER (EXPOSE only documents); bind to 127.0.0.1 to limit access.
  • Use a user-defined network for multi-container apps → DNS by container name (no hardcoded IPs); Compose does this for you.
