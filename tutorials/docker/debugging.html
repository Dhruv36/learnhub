Debugging Containers Intermediate

Containers fail in ways that feel opaque until you know the tools: a container that exits immediately, a build that won't cache, a service that can't reach another. This lesson is the practical toolkit — reading logs, shelling in, inspecting config, and the systematic approach to the most common failures.

The Core Debugging Commands

docker logs <container>          docker logs -f --tail 100 <c>    // stdout/stderr (your first look)
docker ps -a                     // ALL containers incl. exited (with exit codes + status)
docker exec -it <c> sh           // shell INTO a running container (poke around live)
docker inspect <c>               // full JSON: env, mounts, network, restart policy, exit code
docker stats                     // live CPU/memory (is it OOM-killed? throttled?)
docker events                    // real-time daemon events (starts, stops, OOM kills, deaths)
docker diff <c>                  // files changed in the container's writable layer

"My Container Exits Immediately" — the #1 Issue

docker ps -a         // shows the container as "Exited (1)" or similar
docker logs <c>       // ← the error is almost always here
// Common causes:
//  - The main process CRASHED on startup (bad config, missing env, can't reach the DB) → logs show it.
//  - The main process EXITED normally (a container lives only as long as its PID 1 process):
//      running `bash` with no TTY, a script that finishes, or a server that didn't actually start.
//  - Wrong CMD/ENTRYPOINT. A container is NOT a VM — when PID 1 exits, the container stops.
// To inspect a crashing image, override the entrypoint and poke around:
docker run -it --entrypoint sh myimage     // get a shell instead of the failing command
🔎 A container runs exactly as long as its PID 1 process. Unlike a VM, there's no background OS keeping it alive — when the main process exits (crash OR normal completion), the container stops. So "exits immediately" is almost always "the main process ended", and docker logs tells you why. This one mental model resolves most beginner confusion.

The Common Failure Playbook

SymptomLikely cause → fix
Exits immediatelyPID 1 crashed/finished → docker logs; check CMD, config, deps
Can't connect between containersNot on the same user-defined network, or using an IP → connect by service name on a shared network (volumes-networking lesson)
Port not reachable from hostEXPOSE without -p, or app bound to 127.0.0.1 inside → publish with -p; bind to 0.0.0.0 in the app
Container killed (exit 137)OOM — exceeded its memory limit → raise the limit or fix the leak; check docker stats/events
Build not using cacheAn early layer changed → reorder (dockerfile lesson); check .dockerignore
File changes not appearingEditing the image, not a bind mount → mount the source for dev
Permission denied on a volumeUID mismatch (non-root USER vs host files) → align UIDs / fix ownership

Interview Questions

Q1: A container exits immediately. Walk your diagnosis.

First, docker ps -a to confirm it exited and see the exit code/status, then docker logs <container> — the cause is almost always there. The key mental model is that a container lives exactly as long as its PID 1 (main) process: unlike a VM there's no background OS, so when that process crashes OR completes normally, the container stops. So "exits immediately" is "the main process ended" — either it crashed on startup (bad config, missing env var, unreachable dependency, wrong CMD/ENTRYPOINT — the logs show the error) or it wasn't a long-running process at all (a script that finished, or an interactive shell with no TTY). To investigate a crashing image interactively, override the entrypoint (docker run -it --entrypoint sh myimage) to get a shell and inspect the environment, files, and config before the failing command runs.

Q2: What does exit code 137 mean and how do you address it?

137 = 128 + 9, meaning the process received SIGKILL. The most common cause is the container being OOM-killed: it exceeded its memory limit (a --memory constraint or the node's available memory) and the kernel's OOM killer terminated it. Diagnose by checking docker stats (memory usage climbing to the limit), docker inspect (the OOMKilled flag), and docker events (kill events). Address it by either raising the container's memory limit if the workload legitimately needs more, or — usually better — fixing the underlying memory leak or excessive allocation in the application (the JS/Java/etc. memory lessons apply). 137 can also result from a forced docker kill. The takeaway is that a mysterious sudden death with 137 is a memory problem until proven otherwise.

Q3: Two containers can't communicate. How do you debug it?

Check that both are attached to the SAME user-defined network (docker inspect the networks, or docker network inspect the network to see connected containers) — the default bridge doesn't provide name resolution, so multi-container apps need a shared user-defined network (or Compose, which creates one). Verify the client is connecting by SERVICE/CONTAINER NAME, not a hardcoded IP (IPs change on restart), and that the name resolves (exec into the client and try nslookup/ping the other name, or curl its port). Confirm the target app is actually listening and bound to 0.0.0.0 (not 127.0.0.1, which is unreachable from other containers) on the expected port, and that no firewall/network policy blocks it. In short: same network, connect by name, target listening on 0.0.0.0 — the three checks that resolve almost all inter-container connectivity issues.

🏋️ Practice: run an image with a bad CMD, see it exit, and find the cause in docker logs; override --entrypoint sh to explore a crashing image; give a container a tiny --memory limit and a memory-hungry process to trigger exit 137; put two containers on a shared network and debug a connection by exec+curl by name; reproduce a 127.0.0.1-binding issue and fix it; inspect a container's mounts/network with docker inspect.

🎯 Key Takeaways

  • docker logs is your first and usually final stop; ps -a shows exit codes; exec/inspect/stats/events round out the toolkit.
  • A container lives only as long as its PID 1 process — "exits immediately" = the main process crashed or finished.
  • Exit 137 = OOM (raise memory or fix the leak); 143 = SIGTERM (graceful); read logs for 1/2.
  • Inter-container connectivity: same user-defined network, connect by name (not IP), target bound to 0.0.0.0.
  • Minimal images (distroless/scratch) have no shell — use docker/kubectl debug or a debug image variant.
