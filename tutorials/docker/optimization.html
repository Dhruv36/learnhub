Image Optimization Intermediate

Smaller, better-layered images pull faster, deploy faster, cost less to store, and expose a smaller attack surface. This lesson consolidates every size and build-speed lever — base image choice, layer caching, .dockerignore, and BuildKit — into one systematic approach you can apply to any Dockerfile.

Lever 1: Choose the Right Base Image

FROM node:22           // ~1GB   — full Debian, lots of tools you don't need
FROM node:22-slim      // ~250MB  — trimmed Debian
FROM node:22-alpine    // ~150MB  — musl-based, tiny (watch for musl vs glibc quirks)
FROM gcr.io/distroless // minimal — no shell/package manager (great for compiled langs)
FROM scratch           // 0 bytes — empty; only a static binary
// Rule: pick the SMALLEST base that still runs your app correctly. Alpine is a common sweet spot,
// but test — musl libc can break native modules; distroless/scratch suit static binaries best.

Lever 2: Layer Caching (build speed)

// Order instructions LEAST-changing → MOST-changing so cache survives (dockerfile lesson):
COPY package*.json ./       // changes rarely
RUN npm ci                  // ← cached until deps change (the expensive step)
COPY . .                    // source changes often — put it LATE
RUN npm run build
// If you COPY . . before installing deps, EVERY code edit busts the npm ci cache → slow builds.
// A cache bust invalidates that layer AND every layer after it.
⚡ Two goals, sometimes in tension: small final image vs fast builds. Multi-stage builds (previous track lessons) serve size; layer ordering + BuildKit cache serve speed. The best Dockerfiles do both — a well-ordered multi-stage build with a minimal runtime base.

Lever 3: Fewer, Cleaner Layers

// ❌ Each RUN is a layer; cleaning in a LATER layer doesn't shrink the earlier one:
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*      // too late — the earlier layers still carry the data
// ✅ Combine and clean in the SAME layer:
RUN apt-get update && apt-get install -y --no-install-recommends curl \
    && rm -rf /var/lib/apt/lists/*
// The cleanup only reduces size if it happens in the same RUN that added the data.

Lever 4: .dockerignore (context + cache + secrets)

// .dockerignore — keep junk OUT of the build context sent to the daemon:
node_modules
.git
*.log
.env
dist
// Benefits: smaller/faster context upload, avoids busting cache on irrelevant files, and prevents
// accidentally COPYing secrets (.env) or huge dirs (.git, node_modules) into the image.

Lever 5: BuildKit (modern builder)

FeatureBenefit
Parallel stage buildsIndependent multi-stage stages build concurrently
--mount=type=cachePersistent cache for package managers across builds (npm, pip, apt)
--mount=type=secretBuild-time secrets without leaking into layers (env-secrets lesson)
Better cache + smaller contextSkips unused stages, smarter invalidation
// Cache mount example (deps cache survives even when the lockfile changes a little):
RUN --mount=type=cache,target=/root/.npm  npm ci

Diagnosing Size

docker images                    // see image sizes
docker history <image>           // size CONTRIBUTED BY EACH LAYER — find the fat one
docker image inspect <image>     // detailed metadata
// dive (3rd-party tool)         // interactive layer explorer — find wasted space per layer

Interview Questions

Q1: How do you reduce a Docker image's size?

Attack it on several fronts. Start with the base image — pick the smallest that still runs the app (slim, then alpine, then distroless/scratch for static binaries), since the base often dominates size. Use multi-stage builds so the final image contains only the runtime artifact and production dependencies, not compilers, dev dependencies, or source. Minimize and clean layers: combine related RUN commands and remove caches (apt lists, package-manager caches) in the SAME layer that created them, because cleanup in a later layer doesn't shrink earlier ones. Add a .dockerignore to keep node_modules, .git, and build output out of the context and image. Copy only what's needed. Finally, measure with docker history (per-layer size) or dive to find and eliminate the fattest layers. The payoff is faster pulls/deploys, lower storage cost, and a smaller attack surface.

Q2: How does layer caching work and how do you exploit it for fast builds?

Each Dockerfile instruction produces a layer, and Docker caches layers: on rebuild it reuses a cached layer if that instruction and its inputs are unchanged, but the first instruction whose input changed invalidates that layer AND every layer after it. The optimization is to order instructions from least-frequently-changing to most-frequently-changing so expensive, rarely-changing steps stay cached. The canonical example: COPY the dependency manifests (package.json/lockfile) and run the install BEFORE copying the rest of the source, so editing application code (which changes often) doesn't invalidate the expensive dependency-install layer. Putting COPY . . before the install reverses this and re-installs dependencies on every code change. BuildKit adds cache mounts (--mount=type=cache) that persist a package manager's cache across builds for even faster installs.

Q3: What is .dockerignore and why does it matter?

.dockerignore lists files and directories to exclude from the build context — the set of files sent to the Docker daemon (or BuildKit) when a build starts. It matters for three reasons: performance (excluding large directories like node_modules, .git, and build output makes the context upload smaller and faster), cache stability (irrelevant file changes won't bust the build cache or get COPYed into the image), and security/correctness (it prevents accidentally copying secrets like .env or bloating the image with files that shouldn't ship). It's analogous to .gitignore but for Docker builds, and it's a simple, high-leverage optimization every project should have.

🏋️ Practice: take a node:22 image and swap to -slim then -alpine, measuring each with docker images; reorder a Dockerfile so deps install before COPY . . and observe cache reuse on a code edit; combine an apt install + cleanup into one RUN and compare sizes; add a .dockerignore and see the context shrink; run docker history to find the biggest layer; enable BuildKit and add a cache mount for npm/pip.

🎯 Key Takeaways

  • Pick the smallest base that works (slim → alpine → distroless/scratch); the base often dominates size.
  • Order layers least→most changing; copy manifests + install deps BEFORE source so edits don't bust the deps cache.
  • Combine RUNs and clean caches in the SAME layer — later cleanup doesn't shrink earlier layers.
  • Always add a .dockerignore (faster context, stable cache, no leaked secrets).
  • Use multi-stage + BuildKit (cache/secret mounts); diagnose with docker history / dive.
