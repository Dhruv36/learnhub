Container Security Advanced

Containers isolate processes but share the host kernel — so a container is NOT a security boundary as strong as a VM. Securing containers means reducing what a compromised container can do: run as non-root, drop capabilities, ship minimal images, scan for CVEs, and never trust unverified images. This is a heavily-tested senior topic.

The #1 Rule: Don't Run as Root

// By default a container's process runs as ROOT (uid 0) — and root in the container maps to root
// on the host kernel. If an attacker escapes the container, they're root on the host.
FROM node:22-alpine
WORKDIR /app
COPY --chown=node:node . .
USER node                     // ← drop to a non-root user; everything after runs unprivileged
CMD ["node", "server.js"]
// Run with least privilege. Many official images provide a non-root user (e.g., "node"); use it.
🛡️ Containers share the host KERNEL — isolation is weaker than a VM. Namespaces and cgroups limit what a container sees and uses, but a kernel vulnerability or a misconfiguration (privileged mode, mounted docker socket) can lead to host compromise. Security is defense-in-depth: assume a container CAN be breached and minimize the blast radius.

Dangerous Anti-patterns

docker run --privileged ...             // ❌ nearly full host access — almost never needed
docker run -v /var/run/docker.sock:/var/run/docker.sock ...  // ❌ container can control the daemon = host root
docker run --cap-add=ALL ...            // ❌ grants all Linux capabilities
// Running as root + writable FS + all caps + host mounts = trivial host takeover after any RCE.

Scanning & the Supply Chain

trivy image myapp:latest        // scan for OS + dependency CVEs (fail the build on HIGH/CRITICAL)
docker scout cves myapp         // Docker's built-in CVE scanner

Hardening at Runtime (compose/k8s)

services:
  api:
    image: myapp
    read_only: true
    cap_drop: [ALL]
    security_opt: ["no-new-privileges:true"]
    user: "1000:1000"
    tmpfs: ["/tmp"]
// In Kubernetes, the equivalent is a securityContext (runAsNonRoot, readOnlyRootFilesystem,
// allowPrivilegeEscalation: false, drop capabilities) — the k8s track covers Pod security.

Interview Questions

Q1: Why is running a container as root dangerous, and what do you do instead?

Because containers share the host's kernel and root inside the container maps to root on the host, so if an attacker achieves code execution and then escapes the container (via a kernel bug or misconfiguration), they gain root on the host — a full compromise. Containers are not as strong an isolation boundary as VMs, so you minimize the blast radius by running with least privilege: create and switch to a non-root USER in the Dockerfile (many official images ship one, like "node"), chown files appropriately, and enforce it at runtime (Kubernetes runAsNonRoot / a non-root user). Combine that with dropping Linux capabilities, a read-only root filesystem, no-new-privileges, and never using --privileged or mounting the docker socket, so that even a compromised process can do very little.

Q2: How is container isolation different from a VM, and what are the implications?

A VM runs its own full guest kernel on top of a hypervisor, so guests are strongly isolated from the host and each other at the hardware-virtualization level. Containers instead share the HOST kernel and are isolated only by kernel features — namespaces (separate views of processes, network, mounts, users) and cgroups (resource limits). That makes containers far lighter and faster to start, but the isolation is weaker: a kernel vulnerability or a misconfiguration (privileged mode, host mounts, the docker socket, excessive capabilities) can allow a container to affect the host or other containers. The implication is defense-in-depth — assume a container can be breached and reduce what it can do (non-root, dropped capabilities, read-only FS, seccomp/AppArmor, resource limits, minimal images) — and for hostile multi-tenant workloads consider stronger isolation like VM-based sandboxes (gVisor, Kata, Firecracker).

Q3: How do you secure the image supply chain?

Ship minimal images (distroless/alpine, multi-stage) so there's less software to carry CVEs and no shell to exploit; scan every image for known vulnerabilities with tools like Trivy, Grype, Snyk, or docker scout and gate the CI pipeline to fail on HIGH/CRITICAL findings so vulnerable images never reach production; keep no secrets in layers; and secure provenance by pinning base images to immutable digests (not mutable tags like latest), verifying image signatures (e.g., cosign/Sigstore, Docker Content Trust), and generating an SBOM to know exactly what's inside. Combined with least privilege at runtime and regular rebuilds to pick up patched base images, this closes the common supply-chain gaps — untrusted or tampered images, unpatched CVEs, and unknown contents.

🏋️ Practice: add a non-root USER to a Dockerfile and verify with whoami inside; run a container with --read-only + tmpfs and confirm writes to the FS fail; scan an image with trivy (or docker scout) and read the CVE report; try --cap-drop ALL and see which capabilities your app actually needs; demonstrate why mounting the docker socket is dangerous (list host containers from inside); pin a base image by digest.

🎯 Key Takeaways

  • Containers share the host kernel — weaker isolation than VMs; assume breach and minimize blast radius.
  • Run as non-root (USER / runAsNonRoot); root-in-container escape = host root.
  • Drop capabilities, read-only root FS, no-new-privileges, resource limits; never --privileged or mount the docker socket.
  • Ship minimal images and scan for CVEs (Trivy/Scout) in CI, failing on critical findings.
  • Secure provenance: pin digests, verify signatures, generate an SBOM (next lesson).
