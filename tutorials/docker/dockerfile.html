Writing a Dockerfile Basics

A Dockerfile is the recipe that builds an image — a sequence of instructions, each adding a layer. Writing a good one is mostly about two things: ordering instructions to maximize cache reuse and knowing the subtle differences between similar instructions (COPY vs ADD, CMD vs ENTRYPOINT, RUN forms). Get these right and builds are fast and images are correct.

A Realistic Dockerfile (annotated)

FROM node:22-alpine                    # base image (alpine = tiny)
WORKDIR /app                           # set + create the working directory

COPY package*.json ./                  # copy ONLY manifests first (see caching below)
RUN npm ci --omit=dev                  # install deps — cached unless package files change

COPY . .                               # then copy the rest of the source
RUN npm run build                      # build step

ENV NODE_ENV=production                # environment variable
EXPOSE 3000                            # document the port (doesn't publish it)
USER node                              # run as non-root (security lesson)
CMD ["node", "dist/server.js"]         # default command when the container runs

The Cache-Ordering Rule (the #1 skill)

// Docker caches each layer; on rebuild, once a layer changes, that layer AND ALL AFTER IT rebuild.
// So order instructions from LEAST-frequently-changing to MOST-frequently-changing:

// ❌ BAD — code changes bust the dependency install every build:
COPY . .            # any code edit changes this layer...
RUN npm ci          # ...so deps reinstall EVERY time (slow)

// ✅ GOOD — deps only reinstall when package files change:
COPY package*.json ./
RUN npm ci          # cached across code-only changes
COPY . .            # only THIS layer rebuilds when code changes
// Dependencies change rarely; source changes constantly → install deps first.
// This single reorder can turn a 2-minute rebuild into 5 seconds.
🥞 This is the layer-caching lesson applied. Because a changed layer invalidates everything below it, you copy the rarely-changing dependency manifests and install deps BEFORE copying your frequently-changing source. The dependency layer then stays cached across day-to-day code edits. Nearly every "why is my Docker build so slow?" question is answered by this ordering rule.

CMD vs ENTRYPOINT

// CMD: the DEFAULT command/args — easily OVERRIDDEN by `docker run image othercmd`.
CMD ["node", "server.js"]           // docker run myapp bash → runs bash instead

// ENTRYPOINT: the fixed executable — args from `docker run` are APPENDED to it.
ENTRYPOINT ["node", "server.js"]    // docker run myapp --port 4000 → node server.js --port 4000

// Common pattern: ENTRYPOINT = the program, CMD = default args (overridable):
ENTRYPOINT ["python", "app.py"]
CMD ["--port", "8000"]              // docker run myapp --port 9000 overrides just the args
// Use the EXEC form (JSON array) so signals (SIGTERM) reach your process directly — the SHELL
// form (CMD node server.js) wraps it in /bin/sh, which can swallow signals (graceful-shutdown bug).

.dockerignore & Build

// .dockerignore (like .gitignore) — exclude files from the build context sent to the daemon:
node_modules
.git
*.log
.env
// WHY: COPY . . would otherwise copy node_modules/.git into the image (bloat + slow context upload,
// and could leak secrets). Always add a .dockerignore.
docker build -t myapp:1.0 .          // build from the current dir (the "context")
docker build --no-cache -t myapp .   // ignore cache (force full rebuild)

Interview Questions

Q1: Why does Dockerfile instruction ORDER affect build speed?

Docker caches each instruction's resulting layer, and on rebuild it reuses cached layers until it hits one whose inputs changed — from that point, that layer and every layer after it must rebuild. So you order instructions from least- to most-frequently-changing. Concretely, dependencies change rarely while source code changes constantly, so you COPY the package manifests and RUN the dependency install BEFORE COPYing the application source: a code-only edit then invalidates only the final COPY (and build) layer, while the expensive dependency-install layer stays cached. Doing it the naive way (COPY . . then install) invalidates the install on every code change, turning a multi-minute rebuild into seconds — the single highest-impact Dockerfile optimization.

Q2: CMD vs ENTRYPOINT — and why the exec form?

Both define what the container runs, but differ in override behavior: CMD sets a default command/args that arguments to `docker run` REPLACE entirely; ENTRYPOINT sets a fixed executable to which `docker run` arguments are APPENDED. The idiomatic combo is ENTRYPOINT = the program and CMD = default, overridable arguments. Separately, prefer the exec form (JSON array, ["node", "server.js"]) over the shell form (node server.js): the shell form runs your process as a child of /bin/sh -c, which becomes PID 1 and may not forward signals like SIGTERM to your app, breaking graceful shutdown; the exec form makes your process PID 1 so it receives signals directly. This signal handling is why the exec form is a production best practice.

Q3: What's the purpose of .dockerignore and ARG vs ENV?

.dockerignore excludes paths from the build context that Docker sends to the daemon (and thus from COPY . .), preventing bloat and slow builds from copying node_modules/.git, and avoiding accidentally baking secrets (like .env) into the image — it works like .gitignore for builds and should always be present. ARG vs ENV: ARG defines a BUILD-TIME variable (passable via --build-arg), available only during the build and NOT present in the final image; ENV defines a variable that persists into the image and is available at runtime to the running process. Use ARG for build-time parameters (versions, flags) and ENV for runtime configuration — but never pass real secrets through either, since ARG values can appear in image history and ENV values are readable in the image; use build secrets or inject secrets at runtime instead.

🏋️ Practice: write a Dockerfile for a Node or Python app; build it, edit a source file, rebuild, and watch cached layers; reorder to copy manifests+install before source and see the speedup; add a .dockerignore and confirm node_modules isn't copied; experiment with CMD vs ENTRYPOINT overrides via docker run; convert a shell-form CMD to exec form and verify docker stop works quickly (SIGTERM handled).

🎯 Key Takeaways

  • A Dockerfile is an ordered recipe; each instruction adds a cached layer.
  • Order least- to most-frequently-changing: copy manifests + install deps BEFORE copying source (the #1 build-speed rule).
  • CMD = overridable default; ENTRYPOINT = fixed executable (args appended); use exec form so signals reach your process.
  • COPY over ADD; ENV (runtime) vs ARG (build-time); EXPOSE only documents; USER for non-root.
  • Always add .dockerignore (exclude node_modules/.git/secrets); never bake secrets into layers.
