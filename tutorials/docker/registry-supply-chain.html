Registry & Supply Chain Advanced

Images are built once and pulled everywhere — from a registry. How you tag, store, and verify those images determines whether your deployments are reproducible and whether you can trust what runs in production. This lesson covers registries, tagging discipline, and securing the software supply chain — the final, senior-level piece of the Docker story.

Registries & the Push/Pull Flow

docker build -t myapp:1.2.0 .
docker tag myapp:1.2.0 registry.example.com/team/myapp:1.2.0
docker login registry.example.com
docker push registry.example.com/team/myapp:1.2.0     // upload to the registry
// Elsewhere (CI, prod):
docker pull registry.example.com/team/myapp:1.2.0     // download by name:tag
// Registries: Docker Hub, GitHub Container Registry (ghcr.io), AWS ECR, GCP Artifact Registry,
// Azure ACR, or a self-hosted Harbor. The registry is the distribution hub for your images.

Tagging Discipline — Never Deploy :latest

myapp:latest        // ❌ MUTABLE — "latest" points to different images over time → irreproducible
myapp:1.2.0         // ✅ semantic version tag — meaningful, but a tag can still be re-pushed
myapp@sha256:abc... // ✅✅ IMMUTABLE digest — pins the EXACT image bytes, can never change
// Best practice: build with a version tag, and DEPLOY by DIGEST (or a tag you never overwrite).
// "latest" is fine for local experimentation, dangerous for production reproducibility.
📌 A tag is a movable pointer; a digest is the content itself. latest (and even a version tag) can be re-pushed to point at new bytes, so two pulls can yield different images — breaking reproducibility and enabling tampering. A sha256 digest is derived from the image content, so pinning by digest guarantees you run exactly what you tested.

Signing & Verifying (provenance)

// Sign an image so consumers can verify it came from you and wasn't altered:
cosign sign registry.example.com/team/myapp@sha256:abc...
cosign verify registry.example.com/team/myapp@sha256:abc...   // fails if tampered/unsigned
// Enforce "only signed images run" with an admission policy (Kubernetes: Sigstore policy-controller,
// Kyverno, OPA Gatekeeper) so unsigned/untrusted images are REJECTED at deploy time.

SBOM & Attestations

// An SBOM lists every package/version in the image — critical for answering "are we affected by
// CVE-XXXX?" across your fleet in minutes instead of days:
syft myapp:1.2.0 -o spdx-json > sbom.json      // generate
trivy sbom sbom.json                            // scan the SBOM for known CVEs
// SLSA provenance/attestations record HOW and WHERE the image was built (which commit, which CI run),
// cryptographically, so you can trust the artifact's origin. BuildKit can emit SBOM + provenance
// attestations attached to the image at build time.

Interview Questions

Q1: Why shouldn't you deploy the :latest tag, and what do you do instead?

Because tags are mutable pointers — "latest" (and even a version tag) can be re-pushed to reference different image bytes over time, so two pulls of the same tag can yield different images. That breaks reproducibility ("works on my machine / in staging but not prod"), makes rollbacks unreliable, and opens a tampering vector. The fix is to deploy by an immutable reference: pin to a content-addressed digest (myapp@sha256:...), which is derived from the image's exact bytes and can never change, so you run precisely what you built and tested — or at minimum use versioned tags that you never overwrite plus a retention/immutability policy on the registry. latest is fine for local experimentation, not for production.

Q2: What are the main container supply-chain risks and how do you mitigate them?

The key risks are: running a tampered or impersonated image (including typosquatted public images); shipping known CVEs from the base image or dependencies; not knowing what's actually inside an image when a new vulnerability drops; irreproducible deploys from mutable tags; and a compromised build pipeline producing malicious artifacts. Mitigations map one-to-one: sign images and verify signatures (cosign/Sigstore, Docker Content Trust) and enforce "only signed/trusted images run" via admission policies; scan images for CVEs (Trivy/Grype/Scout) in CI and gate on HIGH/CRITICAL, and rebuild regularly to absorb base-image patches; generate and store an SBOM so you can query your fleet for affected packages instantly; pin by digest for reproducibility; and record cryptographic build provenance/attestations (SLSA) from a trusted, least-privilege CI. Together these establish integrity, known contents, and trusted origin end to end.

Q3: What is an SBOM and why does it matter?

An SBOM (Software Bill of Materials) is a complete, machine-readable inventory of every component in an image — OS packages, language dependencies, and their exact versions — typically in a standard format like SPDX or CycloneDX, generated by tools such as syft or BuildKit. It matters most for vulnerability response: when a new CVE is announced (think Log4Shell), an organization with SBOMs can query across its entire fleet and answer "which images contain the affected package and version?" in minutes instead of manually inspecting each image over days. SBOMs also support license compliance, audit, and — when combined with signing and provenance attestations — a verifiable supply chain where you know exactly what's in an artifact and where it came from. Storing the SBOM alongside the image (or as an attestation) makes this continuously queryable.

🏋️ Practice: tag and push an image to ghcr.io or a local registry, then pull it by digest; observe that re-pushing :latest changes what "latest" resolves to; sign an image with cosign and verify it (then tamper and watch verification fail); generate an SBOM with syft and scan it with trivy; pin a base image FROM by @sha256 digest; set up scan-on-push or a CI scan gate.

🎯 Key Takeaways

  • A registry is the distribution hub: build → tag → push → pull (Docker Hub, ghcr, ECR, ACR, Harbor…).
  • Tags are mutable pointers; digests are immutable content. Deploy by digest for reproducibility — never :latest in prod.
  • Secure the supply chain: sign + verify (cosign/Sigstore), scan for CVEs in CI, rebuild for patches.
  • Generate SBOMs to answer "are we affected by this CVE?" across the fleet instantly; add SLSA provenance for trusted origin.
  • Registry hygiene: immutable tags, RBAC, scan-on-push, mirrors, and short-lived scoped credentials.
