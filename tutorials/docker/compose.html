Docker Compose Intermediate

Real apps have multiple containers — a web service, a database, a cache. Running each with long docker run commands is error-prone. Docker Compose declares your whole multi-container app in one YAML file and starts it with a single command. It's the standard for local development and simple deployments, and a stepping stone to Kubernetes.

A Complete Compose File

// compose.yaml
services:
  api:
    build: .                          # build from the local Dockerfile
    ports: ["3000:3000"]
    environment:
      DATABASE_URL: postgres://db:5432/app     # "db" = the service name (DNS!)
      REDIS_URL: redis://cache:6379
    depends_on: [db, cache]           # start order (not readiness — see below)
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes: ["pgdata:/var/lib/postgresql/data"]   # named volume for persistence
  cache:
    image: redis:7

volumes:
  pgdata:                              # declare the named volume

// One command brings up the whole stack:
docker compose up -d                  // build + create network + start all services
docker compose down                   // stop + remove (add -v to also remove volumes)
docker compose logs -f api            // follow one service's logs
docker compose ps                     // status

🔗 The DNS-by-service-name magic is why the api can use postgres://db:5432: Compose puts all services on one network (the volumes-networking lesson's user-defined network), so the service NAME resolves to that container. No IPs, no linking — this is the payoff that makes multi-container local dev painless.

depends_on ≠ "wait until ready" (the classic trap)

depends_on: [db]     // starts db BEFORE api — but does NOT wait for Postgres to accept connections!
// The db CONTAINER may be running while Postgres is still initializing → api's first connection fails.
// Fixes:
//  1. A healthcheck + depends_on condition:
db:
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U postgres"]
    interval: 5s
    retries: 5
api:
  depends_on:
    db: { condition: service_healthy }   // NOW api waits until db is HEALTHY
//  2. App-level retry/backoff on connect (robust regardless — the app should tolerate a not-yet-ready DB).
// The best practice is BOTH: healthcheck-gated start AND resilient connection retries in the app.

Profiles, Overrides & Env

// compose.override.yaml is auto-merged (great for dev-only settings like bind mounts):
services:
  api:
    volumes: ["./src:/app/src"]        // live code editing in dev, not in prod
// Environment from a file:
env_file: .env
// Profiles enable optional services (e.g., a debug tool) only when requested:
docker compose --profile debug up
// Multiple files: docker compose -f compose.yaml -f compose.prod.yaml up

Compose vs Kubernetes

// Compose: SINGLE host, developer-friendly, simple deployments. No auto-scaling, self-healing
//           across machines, or rolling updates at scale.
// Kubernetes (next track): MULTI-host orchestration — scheduling, scaling, self-healing, rolling
//           deploys, service discovery across a cluster. More complex; for production at scale.
// Compose is where you learn the concepts (services, networks, volumes, config) that Kubernetes
// then operationalizes across a fleet. Many teams use Compose for local dev + K8s for production.

Interview Questions

Q1: What does Docker Compose give you over multiple docker run commands?

It declares an entire multi-container application — its services, their images/builds, ports, environment, volumes, and dependencies — in one version-controlled YAML file, and manages the whole stack's lifecycle with single commands (up/down/logs). Crucially it creates a shared user-defined network automatically, so services discover each other by name via DNS (the api connects to postgres://db:5432 using the service name "db"), eliminating hardcoded IPs and manual linking. This makes the app's topology reproducible for every developer and environment, replaces fragile shell history with declarative config, and colocates persistence (named volumes) and configuration with the services. It's the standard for local multi-container development and simple single-host deployments.

Q2: Why is depends_on not enough, and how do you handle startup ordering correctly?

depends_on only controls START ORDER — it starts the dependency's container before the dependent — but it does NOT wait for the dependency's application to be READY to serve requests. So Compose can start Postgres's container and then the API while Postgres is still initializing, and the API's first connection fails. The correct approach is twofold: define a healthcheck on the dependency (e.g., pg_isready for Postgres) and gate the dependent with depends_on: { condition: service_healthy } so it waits until the dependency reports healthy; AND make the application resilient with connection retries/backoff, since dependencies can also become temporarily unavailable at runtime, not just at startup. Relying on either alone is fragile; production-grade setups do both.

Q3: When do you outgrow Compose for Kubernetes?

Compose is designed for a single host: it's excellent for local development and simple single-server deployments, but it doesn't schedule containers across multiple machines, auto-scale based on load, self-heal by rescheduling failed containers onto healthy nodes, or perform rolling updates and advanced service discovery across a cluster. You move to Kubernetes when you need production orchestration at scale — high availability across hosts, horizontal autoscaling, zero-downtime rolling deployments, and cluster-wide networking/service discovery. The concepts transfer directly (services, networks, volumes, environment config), so Compose is both a practical local-dev tool and a conceptual on-ramp; a very common pattern is Compose for local development and Kubernetes for production.

🏋️ Practice: write a compose.yaml with api + postgres + redis; docker compose up -d and confirm the api reaches the db by service name; add a named volume and prove data survives down/up; reproduce the depends_on race (api connects before db is ready) then fix with a healthcheck condition + app retry; add a compose.override.yaml with a bind mount for live dev; use --profile for an optional tool.

🎯 Key Takeaways

  • Compose declares a multi-container app in one YAML and runs it with a single command (up/down).
  • It auto-creates a network → services reach each other by NAME (DNS), plus volumes + env in one place.
  • depends_on controls start ORDER, not readiness — add healthchecks (condition: service_healthy) AND app-level retries.
  • Keep secrets out of committed YAML (.env/secrets); named volumes for persistence; down -v deletes volumes.
  • Compose = single-host dev/simple deploys; Kubernetes = multi-host orchestration at scale (concepts transfer).
