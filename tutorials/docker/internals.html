How Containers Work Advanced

A container isn't a lightweight VM — it's just a normal Linux process that the kernel has isolated using namespaces, constrained using cgroups, and given its own filesystem view via a union filesystem. Understanding these primitives demystifies everything else — why containers are fast, why isolation is weaker than a VM, and why images layer.

The Three Primitives

PrimitiveWhat it isolates / does
NamespacesWHAT a process can SEE: PID (own process tree), NET (own interfaces/IPs), MNT (own mounts), UTS (hostname), IPC, USER (uid mapping)
cgroupsHOW MUCH a process can USE: CPU, memory, I/O, PIDs — enforced limits (this is what kills OOM containers)
Union FS (overlay2)Layered, copy-on-write filesystem: read-only image layers + a writable container layer stacked into one view
🧩 A container = a process + namespaces (isolation) + cgroups (limits) + a union FS (layered rootfs). There's no guest OS and no hypervisor — that's why containers start in milliseconds and are lightweight, and also why they share the host kernel (the security caveat from the previous lesson). "Container" is a convenient name for this bundle of kernel features around a process.

Namespaces — Isolating What You See

// Each namespace gives the process an isolated view of one resource:
// PID   → the container sees its main process as PID 1, can't see host processes
// NET   → its own network stack: interfaces, IPs, routing, ports
// MNT   → its own filesystem mount points (its own root /)
// UTS   → its own hostname
// IPC   → its own shared-memory / semaphores
// USER  → maps container uids to host uids (e.g., root in container → unprivileged on host)
// Same kernel, many isolated "views" — that's the illusion of separate machines.

cgroups — Limiting What You Use

docker run --memory=512m --cpus=1.5 myapp
// cgroups (control groups) enforce resource CEILINGS. Exceed the memory limit → the kernel OOM-kills
// the process (exit 137 from the debugging lesson). CPU limits throttle. This is how one container
// can't starve the others / the host — the basis of multi-tenant density.

Union Filesystem — Why Images Layer

// overlay2 stacks layers into one view with COPY-ON-WRITE:
//   [ writable container layer ]   ← your process's changes go here (thin, disposable)
//   [ image layer: app code    ]   ┐
//   [ image layer: deps        ]   │ read-only, SHARED across all containers from this image
//   [ image layer: base OS     ]   ┘
// Reading a file: search top-down through layers. Writing a file from a lower layer: it's COPIED UP
// to the writable layer first (copy-on-write), then modified. This is why:
//   - Many containers share one image's layers on disk (space-efficient).
//   - The writable layer is ephemeral (removed with the container → volumes lesson).
//   - Deleting a file from a lower layer in a later layer doesn't reclaim its space (security/secrets!).

Interview Questions

Q1: What actually IS a container, technically?

A container is an ordinary Linux process (or process group) that the kernel isolates and constrains using three primitives: namespaces give it an isolated VIEW of system resources (its own PID tree, network stack, mount/filesystem, hostname, IPC, and user mapping), cgroups impose LIMITS on the resources it can consume (CPU, memory, I/O, process count), and a union/overlay filesystem gives it a layered, copy-on-write root filesystem (shared read-only image layers plus a thin writable layer). There is no guest operating system and no hypervisor — the container shares the host kernel — which is exactly why containers start in milliseconds and are lightweight, and also why their isolation is weaker than a virtual machine's. "Container" is just a convenient label for this bundle of kernel features wrapped around a process.

Q2: Explain namespaces vs cgroups.

They solve two different problems. Namespaces control what a process can SEE — they partition kernel resources so each container gets its own isolated view: a PID namespace so it sees its own process tree (its main process as PID 1, unable to see host processes), a network namespace with its own interfaces and ports, a mount namespace with its own filesystem/root, plus UTS (hostname), IPC, and user namespaces (which can map container root to an unprivileged host uid). cgroups (control groups) control how MUCH a process can USE — they enforce ceilings on CPU, memory, block I/O, and process counts. Namespaces provide isolation; cgroups provide resource limiting and accounting. Together they let many containers share one kernel without seeing or starving each other — and cgroup memory limits are what trigger the OOM kill (exit 137) when a container exceeds its allowance.

Q3: How does the union filesystem / copy-on-write make images efficient?

Images are built from stacked, read-only layers, and a running container adds a single thin writable layer on top; overlay2 presents this stack as one unified filesystem. Because the lower layers are read-only and immutable, they can be SHARED on disk across every container started from that image (and even across images that share a base) — so running ten containers from one image doesn't copy the image ten times. Writes use copy-on-write: modifying a file that lives in a lower layer copies it up into the writable layer first, leaving the shared layer untouched, so containers stay isolated while still sharing storage. This is why containers are space- and start-time-efficient, why the writable layer is ephemeral (removed with the container, hence volumes for persistence), and why deleting a file in a later layer doesn't reclaim the space it occupied in an earlier one (relevant to both image size and secret leakage).

🏋️ Practice: run a container and compare ps inside vs on the host (PID namespace); set --memory and trigger an OOM kill to see cgroups enforce limits; inspect overlay2 layers and see how two containers share read-only layers; modify a file from a lower layer and observe copy-on-write; trace docker run → containerd → runc conceptually; read an image's layers with docker history and map them to Dockerfile instructions.

🎯 Key Takeaways

  • A container = a host process + namespaces (isolated view) + cgroups (resource limits) + union FS (layered rootfs). No guest OS.
  • Namespaces isolate WHAT you see (PID/NET/MNT/UTS/IPC/USER); cgroups limit HOW MUCH you use (and cause OOM kills).
  • overlay2 stacks read-only image layers (shared across containers) + a thin writable layer (ephemeral), with copy-on-write.
  • Shared kernel → millisecond startup and low overhead, but weaker isolation than a VM.
  • Runtime stack: docker/containerd/runc over OCI standards; Kubernetes uses containerd via the CRI (no Docker required).
