Multi-stage Builds Intermediate

Build tools, compilers, and dev dependencies are needed to BUILD your app but not to RUN it — shipping them bloats your image and enlarges the attack surface. Multi-stage builds use one stage to build and a second, clean stage that copies only the finished artifact. This is the single most impactful image-size and security technique, and a common interview topic.

The Problem: Fat Images

// Single-stage: the final image contains the FULL toolchain, dev deps, and source — often 1GB+:
FROM node:22
WORKDIR /app
COPY . .
RUN npm ci && npm run build          // needs devDependencies, compilers, source
CMD ["node", "dist/server.js"]        // ...but running only needs dist/ + prod deps
// You ship 900MB of build machinery you'll never use at runtime — slow pulls, bigger attack surface.

The Solution: Build in One Stage, Ship From Another

// ---- STAGE 1: builder (has the full toolchain) ----
FROM node:22 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                             # ALL deps (incl. dev) for building
COPY . .
RUN npm run build                      # produces /app/dist

// ---- STAGE 2: runtime (clean, minimal) ----
FROM node:22-alpine                    # small base
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev                  # PROD deps only
COPY --from=builder /app/dist ./dist   # copy ONLY the built artifact from stage 1
USER node
CMD ["node", "dist/server.js"]
// The final image contains: a small base + prod deps + dist/. NONE of the build toolchain,
// dev deps, or source. Result: often 10× smaller (1GB → ~100MB) and a smaller attack surface.
🎯 The key line is COPY --from=builder. It reaches into a previous stage and copies just the artifact you need, leaving everything else behind. The build stage is discarded — it doesn't ship. This is why multi-stage builds simultaneously shrink images AND improve security (less software = fewer CVEs): you only ship what runtime actually needs.

Compiled Languages Benefit Even More

// Go: build a static binary in a full stage, ship it in a TINY (or empty) base:
FROM golang:1.22 AS build
WORKDIR /src
COPY . .
RUN CGO_ENABLED=0 go build -o /app ./cmd/server

FROM gcr.io/distroless/static      # or scratch (empty!) — just the binary, no OS
COPY --from=build /app /app
ENTRYPOINT ["/app"]
// A Go image can go from ~1GB (with the Go toolchain) to ~10MB (just the binary + minimal base).
// distroless/scratch have no shell/package manager → dramatically smaller attack surface.

Interview Questions

Q1: What are multi-stage builds and why use them?

A multi-stage build defines multiple FROM stages in one Dockerfile: an early "builder" stage contains the full toolchain (compilers, dev dependencies, source) and produces an artifact, and a later, clean runtime stage starts from a minimal base and copies ONLY that artifact via COPY --from=builder — the builder stage itself is discarded and never ships. The benefits are large and twofold: much smaller final images (dropping the build toolchain and dev dependencies often shrinks images 5–10×, e.g., 1GB → ~100MB, giving faster pulls and deploys) and a reduced attack surface (less installed software means fewer packages with potential CVEs, and a shell-less/minimal runtime is harder to exploit). It's the standard way to keep production images lean and secure while still having everything needed at build time.

Q2: How does COPY --from work and what should you copy?

COPY --from=<stage-or-image> copies files out of a named previous build stage (or even an external image) into the current stage, letting the final runtime stage take just the finished output while leaving the entire build environment behind. You should copy the MINIMAL runtime artifact — the compiled binary for Go/Rust, the built dist/ output plus production-only dependencies for Node, the packaged jar for Java — and nothing else: not the source, dev dependencies, build caches, or toolchain. Copying too broadly reintroduces bloat, so the discipline is to identify exactly what the process needs to run and copy only that. This selective copy is what makes the final image small and clean.

Q3: Why do compiled languages benefit even more, and what's distroless/scratch?

Compiled languages like Go or Rust produce a self-contained binary at build time that doesn't need the compiler, package manager, or even a normal OS at runtime — so the runtime stage can be extraordinarily minimal. scratch is a completely empty base image (nothing at all), and distroless images contain only a minimal set of runtime libraries with NO shell or package manager. Copying a statically-linked binary into scratch or distroless yields images of a few megabytes versus ~1GB with the full toolchain, and — critically for security — with no shell or package tooling, there's far less for an attacker to use if they get in, and almost no OS packages to accrue CVEs. The trade-off is harder debugging (no shell to exec into), which teams accept for production while using fuller images for local development.

🏋️ Practice: take a single-stage Node (or Go) Dockerfile and measure its size; convert it to multi-stage (builder + minimal runtime, copying only the artifact) and compare sizes with docker images; for Go, target scratch/distroless and see a ~10MB image; use --target to build only the builder stage; verify the runtime image has no dev deps / no compiler.

🎯 Key Takeaways

  • Build tools are needed to BUILD, not RUN — multi-stage builds ship only the finished artifact.
  • Builder stage (full toolchain) → runtime stage (minimal base) with COPY --from=builder; the builder is discarded.
  • Result: 5–10× smaller images (faster pulls/deploys) AND smaller attack surface (fewer CVEs).
  • Copy only the minimal artifact (binary / dist + prod deps); use alpine/distroless/scratch for runtime.
  • Compiled langs benefit most: a static binary in scratch/distroless = a few MB with no shell to exploit.
