Environment & Secrets Intermediate

The same image should run in dev, staging, and prod — so configuration must come from OUTSIDE the image, at runtime. Environment variables handle config; secrets (passwords, keys, tokens) need extra care because the ways people naively pass them leak into image layers, logs, and process listings. Getting secret handling right is a security-critical, interview-relevant skill.

Configuration via Environment Variables

docker run -e NODE_ENV=production -e PORT=3000 myapp        // set at runtime
docker run --env-file .env myapp                             // from a file
// In Compose:
services:
  api:
    environment:
      NODE_ENV: production
    env_file: .env
// The image stays generic; each environment injects its own config. This is 12-factor config
// (config in the environment, not the code) — the same principle as the Node/Python lessons.

The Secret-Leaking Traps (what NOT to do)

// ❌ TRAP 1: baking secrets into the image via ENV or ARG:
ENV DB_PASSWORD=supersecret            // stored in an IMAGE LAYER — visible to anyone with the image
ARG API_KEY                            // build args appear in `docker history` — leaked!
// Anyone who pulls the image can run `docker history` / inspect layers and read these.

// ❌ TRAP 2: COPYing a secrets file into the image:
COPY .env /app/.env                    // now the secret is a permanent layer (even if you later rm it —
                                       // deleting in a later layer doesn't remove it from earlier ones!)
// ❌ TRAP 3: secrets in the compose.yaml committed to git.
// ❌ TRAP 4: passing secrets as build ARGs and assuming they're gone (they persist in history).
🔐 Golden rule: secrets are INJECTED at RUNTIME, never BAKED into the image. An image is distributed and may be public; anything in a layer (ENV, ARG, COPYed files) is readable by anyone with the image, and deleting a secret in a later layer does NOT remove it from the earlier layer where it was added. So config that varies per environment → runtime env vars; actual secrets → a runtime secret mechanism (below), fetched from a secrets manager. Never in the Dockerfile.

Passing Secrets Correctly

// RUNTIME (the common case): inject from the orchestrator / secrets manager as env or mounted files:
docker run -e DB_PASSWORD="$DB_PASSWORD" myapp        // from the host env (not baked)
// Docker/Compose secrets — mounted as files at /run/secrets, not in the image or env dump:
services:
  api:
    secrets: [db_password]
secrets:
  db_password:
    file: ./db_password.txt          // (or external: true from a manager)
// The app reads /run/secrets/db_password. In Kubernetes, this is a Secret mounted as a file/env.

// BUILD-TIME secrets (e.g., a private-registry token needed only during build) — use BuildKit,
// which mounts the secret transiently WITHOUT persisting it in any layer:
RUN --mount=type=secret,id=npmtoken  npm ci   // secret available only for this RUN, not in history
docker build --secret id=npmtoken,src=./token .

Where Secrets Actually Come From (production)

Cloud instance roles (IAM)No stored secret at all — the runtime gets scoped, short-lived credentials

Env vs Files for Secrets (a subtlety)

// Environment variables are convenient but LEAKY: they show in `docker inspect`, child processes
// inherit them, and crash dumps / logs can capture them. Mounting secrets as FILES (/run/secrets)
// is generally safer for highly sensitive values. Either way, keep them out of the image and out
// of version control, and prefer short-lived credentials from a manager over long-lived static ones.

Interview Questions

Q1: Why should config come from the environment, and how do you do it?

Because the same immutable image should run unchanged across dev, staging, and production — the only thing that differs is configuration, so config must be supplied from OUTSIDE the image at runtime rather than baked in (the 12-factor "config in the environment" principle). Practically, you inject environment variables when running the container (-e / --env-file, or environment/env_file in Compose, or env/configMaps in Kubernetes), and the app reads them at startup. This keeps one reproducible image as the single build artifact promoted through environments, avoids per-environment image drift, and lets you change configuration without rebuilding. Non-sensitive config goes in plain env vars; secrets need the stricter handling below.

Q2: Why is putting a secret in ENV or ARG in a Dockerfile dangerous?

Because both become part of the image's immutable layer history, which is distributed with the image and readable by anyone who has it. An ENV value is stored in a layer and visible via docker inspect / in the running container's environment; a build ARG appears in docker history. Critically, image layers are additive and permanent — deleting a secret file or unsetting a variable in a LATER instruction does NOT remove it from the earlier layer where it was introduced, so "add then remove" still leaks. Any COPYed secrets file has the same problem. So real secrets must never be baked into the Dockerfile; they're injected at runtime (env/mounted files from a secrets manager) or, when genuinely needed during build, provided via BuildKit's --mount=type=secret, which exposes the secret transiently to a single RUN without persisting it in any layer.

Q3: How are secrets handled properly in production?

Secrets live in a dedicated secrets manager (HashiCorp Vault, AWS/GCP Secret Manager) or the orchestrator's secret system (Kubernetes Secrets), and are delivered to the container at RUNTIME — as environment variables or, preferably for sensitive values, mounted files (e.g., /run/secrets or a mounted volume) — never embedded in the image or committed to version control. Best practices layer on top: prefer short-lived, automatically-rotated, narrowly-scoped credentials over long-lived static ones (or use cloud IAM instance roles so no secret is stored at all — the runtime receives temporary scoped credentials); enable encryption-at-rest for the secret store (Kubernetes Secrets are only base64-encoded by default); restrict access with least privilege; and audit access. The through-line: keep secrets out of images and repos, inject at runtime from a managed store, and minimize their lifetime and scope.

🏋️ Practice: run the same image in two "environments" with different -e values; put a fake secret in ENV, build, and read it back with docker history/inspect (see the leak); COPY a .env in, delete it in a later layer, and confirm it's still in the earlier layer; pass a secret at runtime via env and via a mounted file; use BuildKit --mount=type=secret for a build-time token and verify it's absent from history.

🎯 Key Takeaways

  • Config comes from the environment at runtime (one image, per-env config) — the 12-factor principle.
  • NEVER bake secrets into images: ENV/ARG/COPYed files land in readable layers, and later deletion doesn't remove them.
  • Inject secrets at runtime (env or, better, mounted files like /run/secrets) from a secrets manager; use BuildKit --mount=type=secret for build-time secrets.
  • Prefer files over env for sensitive values (env leaks via inspect/child processes/logs).
  • Use short-lived, rotated, scoped credentials (or IAM roles); keep secrets out of git; encrypt the secret store at rest.
