Images & Containers Basics

Images and containers are the two nouns of Docker: an image is an immutable template, a container is a running instance of it. The concept that makes images small and builds fast — the layered filesystem — is the most important idea in this lesson and underlies image optimization, caching, and registries.

Image = Template, Container = Instance

// The class/object analogy:
docker pull node:22              // download an IMAGE (template)
docker run node:22               // create + start a CONTAINER (instance) from it
docker run node:22               // run it again → a SECOND container from the SAME image
// One image → many containers. The image is read-only; each container adds a thin writable layer.

The Layered Filesystem (the key concept)

// An image is built from stacked, READ-ONLY layers — each Dockerfile instruction adds one:
FROM node:22-alpine        → layer 1 (base OS + node)
COPY package.json .        → layer 2
RUN npm ci                 → layer 3 (node_modules)
COPY . .                   → layer 4 (your code)
// Layers are CACHED and SHARED:
//  - Rebuild: unchanged layers are reused from cache (fast builds).
//  - Multiple images sharing a base layer store it ONCE on disk (space savings).
//  - A container adds a thin WRITABLE layer on top (copy-on-write) — changes there are
//    ephemeral and lost when the container is removed.
🥞 Why layers matter everywhere: layer caching makes rebuilds fast (only changed layers and everything after them rebuild — which is why Dockerfile instruction ORDER matters, dockerfile lesson). Layer sharing makes images small on disk and fast to pull (a registry only transfers layers you don't already have). And the container's writable layer being ephemeral is exactly why you need volumes for data. Nearly every Docker best practice traces back to layers.

Image Names & Tags

registry/repository:tag
node:22-alpine                 // official image, tag 22-alpine (implicit docker.io library)
myregistry.com/team/api:1.4.2  // full path with a private registry
myapp:latest                   // 'latest' is just a DEFAULT tag — NOT "newest", NOT special!
// ⚠ Pin specific tags (node:22.3.1) or digests (@sha256:...) for reproducible builds.
// 'latest' is mutable — it can point to different images over time → non-reproducible surprises.

Working With Containers

docker run -d --name web -p 8080:80 nginx     // detached, named, port-mapped
docker ps            docker ps -a                // running / all (incl. stopped)
docker logs web      docker logs -f web           // view / follow logs
docker exec -it web sh                            // open a shell INSIDE a running container
docker stop web      docker start web             // stop/start (keeps the container)
docker rm web        docker rm -f web             // remove (force removes running)
docker stats                                       // live CPU/memory per container

Container Lifecycle

created → running → (paused) → stopped → removed
docker run       // create + start
docker stop      // graceful stop (SIGTERM, then SIGKILL after a grace period)
docker start     // restart a stopped container (state in the writable layer persists until removed)
docker rm        // delete — the writable layer (and any data in it) is GONE
// Containers are meant to be EPHEMERAL and disposable: kill and recreate freely.
// Persistent data goes in volumes; config comes from env/args — not baked into a running container.

Image & Container Management

docker images                docker rmi node:22          // list / remove images
docker pull / docker push                                 // registry transfer
docker image prune           docker container prune       // clean up dangling/stopped
docker system prune -a       // reclaim space (removes unused images/containers/networks) — careful!
docker inspect <id>          // full JSON metadata (config, mounts, network)
docker history myimage       // see the layers and their sizes

Interview Questions

Q1: Explain the difference between an image and a container.

An image is an immutable, read-only template containing an application and its dependencies, built as a stack of layers; a container is a running (or stopped) instance created from an image, with a thin writable layer added on top. The class/object analogy fits: one image can spawn many independent containers, each with its own writable layer and process, all sharing the image's read-only layers. Changes a container makes go into its writable layer and are lost when the container is removed (unless persisted to a volume). So images are what you build, version, and distribute; containers are the ephemeral runtime instances you start, stop, and throw away.

Q2: How does the layered filesystem work, and why does it matter?

An image is composed of stacked read-only layers, one per image-building instruction; a running container adds a writable layer on top using copy-on-write. This matters in three big ways. Build caching: on rebuild, unchanged layers are reused from cache and only the changed layer (and those after it) rebuild — so ordering instructions from least- to most-frequently-changing makes builds fast. Storage/transfer efficiency: layers are content-addressed and shared, so images sharing a base store it once on disk and a registry only transfers layers the target doesn't already have. Ephemerality: because the container's writable layer is discarded on removal, persistent data must live in volumes. Essentially every Docker performance and best-practice guideline derives from the layer model.

Q3: Why is relying on the `latest` tag a problem?

`latest` is not a special "newest" pointer — it's just the default tag applied when none is specified, and it's MUTABLE: the image it points to can change over time as maintainers push new builds. So a Dockerfile FROM node:latest or a deploy of myapp:latest can silently pull a different image on different days or machines, breaking reproducibility and making "it worked yesterday" bugs and inconsistent environments. For reproducible builds and deploys, pin a specific version tag (node:22.3.1-alpine) or, for absolute immutability, a content digest (@sha256:…). Reserve latest for convenience in throwaway/local contexts, never for production or CI reproducibility.

🏋️ Practice: pull an image and run two containers from it; docker ps/logs/exec into a running container; inspect docker history to see layers; build a tiny image twice, changing only the last COPY, and observe cached layers; store a file inside a container, remove it, recreate, and see the file gone; prune dangling images; compare pinning a version tag vs latest.

🎯 Key Takeaways

  • Image = immutable template (stacked read-only layers); container = running instance + a thin writable layer.
  • Layers are cached (fast rebuilds — order matters) and shared (small on disk, fast pulls); the writable layer is ephemeral.
  • Pin tags/digests for reproducibility — `latest` is a mutable default, not "newest".
  • Manage with run/ps/logs/exec/stop/rm; containers are disposable "cattle", not hand-tuned "pets".
  • Persistent data → volumes; clean up with prune; docker history/inspect to introspect images.
