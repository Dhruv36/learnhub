Syntax, Types & Variables Basics

Python favors readability: indentation IS the block structure, and "there should be one obvious way to do it". It's dynamically typed (types travel with values, not variables) but strongly typed (no silent "5" + 3). The deep idea underneath everything: variables are names bound to objects — get that and mutability, copying, and function arguments all click.

Variables Are Names, Not Boxes

x = [1, 2, 3]        # x is a NAME bound to a list object
y = x               # y binds to the SAME object (no copy!)
y.append(4)         # x is now [1, 2, 3, 4] too — one object, two names
y = [9]             # rebinds y to a NEW object; x unaffected

# id() shows object identity; 'is' compares identity, '==' compares value:
a = [1]; b = [1]
a == b              # True  (equal value)
a is b              # False (different objects)
# The name/object model explains every "why did my list change?" bug (data-structures lesson).

Strong + Dynamic Typing

"5" + 3        # ❌ TypeError — no implicit coercion (STRONG typing, unlike JS)
"5" * 3        # "555"  (str * int is defined: repetition)
int("5") + 3   # 8      — convert explicitly
n = 5; n = "hi"  # ✅ allowed — names aren't typed (DYNAMIC typing)
# Strong: errors instead of silent coercion. Dynamic: a name can rebind to any type.

Idiomatic Basics

name = "Asha"; count = 3
f"Hi {name}, {count} new items ({count * 2} total)"   # f-strings: the modern format
f"{value:.2f}"  f"{n:>10}"  f"{x=}"                     # formatting, alignment, debug

a, b = b, a                      # tuple swap, no temp
a, *rest = [1, 2, 3, 4]          # unpacking: a=1, rest=[2,3,4]
x = 10 if cond else 20           # conditional expression (Python's ternary)
big = 1_000_000                  # digit separators
PI: float = 3.14159              # optional type hint (typing lesson)

Interview Questions

Q1: Explain Python's name/object model and how it affects function arguments.

Variables are names bound to objects; assignment binds, it doesn't copy. Arguments are passed by "assignment" (a.k.a. pass-by-object-reference): the parameter becomes another name for the same object. Mutating a mutable argument in place is visible to the caller; rebinding the parameter is not. So a function can append to a passed list but can't reassign the caller's variable.

Q2: is vs == — when does the difference bite?

== compares value (via __eq__); is compares identity (same object). Use is only for singletons (None, True, False). Gotcha: small ints (−5..256) and short strings are cached/interned, so a is b may be True for them by accident — never rely on it. == None can misfire on objects with custom __eq__; is None is unambiguous.

Q3: Strong vs dynamic typing — Python is which, and what's the practical consequence?

Both strong (no implicit cross-type coercion — "5"+3 raises) and dynamic (names aren't typed; a name can rebind to any type at runtime). Consequence: type errors surface at runtime, not compile time — hence the value of type hints + mypy (typing lesson) and thorough tests. Strong typing means you convert explicitly; dynamic means you can't assume a variable's type from its name.

🏋️ Practice (python REPL / a .py file): demonstrate the y=x aliasing bug and fix it; prove a is b is False but a == b is True for two equal lists; show "5"+3 raises but int("5")+3 works; format a price with an f-string to 2 decimals and right-aligned to width 10; swap two variables without a temp.

🎯 Key Takeaways

  • Variables are names bound to objects; = binds, never copies — the root of aliasing/mutation behavior.
  • is = identity, == = value; use is only for None/True/False.
  • Strong + dynamic: no implicit coercion (convert explicitly), but names can rebind to any type.
  • Know mutable (list/dict/set) vs immutable (int/str/tuple/frozenset); immutable+hashable → keys/members.
  • f-strings for formatting; PEP 8 (snake_case, 4 spaces) enforced by ruff/black; is None, decimal for money.
