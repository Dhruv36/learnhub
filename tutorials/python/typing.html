Type Hints & Typing Advanced

Type hints add optional static typing to Python. Crucially, they don't run — Python ignores them at runtime; a separate checker (mypy/pyright) validates them, and tools (editors, Pydantic, FastAPI) read them. On any team codebase, hints + a checker in CI catch a whole class of bugs before they ship — the same value proposition as TypeScript.

The Basics

def greet(name: str, times: int = 1) -> str:
    return f"Hi {name} " * times

age: int = 30
names: list[str] = []                    # 3.9+ builtin generics (no more typing.List)
scores: dict[str, float] = {}
point: tuple[int, int] = (1, 2)

from typing import Optional
def find(id: int) -> Optional[User]:     # User | None (3.10+: `User | None` directly)
    ...
def parse(x: str | int) -> float:        # union: accepts either
    ...
⚙️ Hints are erased at runtime — def f(x: int) does NOT reject f("hi"); it runs and maybe crashes later. Validation happens either at check time (mypy in CI) or, for external data, at runtime via Pydantic. Never assume a hint enforces anything by itself — the boundary-validation rule from the TS lesson applies identically.

The Typing Toolkit

from typing import Any, Callable, Iterable, Optional, Literal, TypedDict, Protocol, TypeVar, cast

Callable[[int, str], bool]               # a function type
Literal["asc", "desc"]                   # exact allowed values (typo-proof "enum")
Iterable[int]                            # anything you can for-loop
Any                                      # opt OUT of checking (contagious — avoid)

class UserDict(TypedDict):               # a dict with a known shape
    id: int; name: str

T = TypeVar("T")                         # generics
def first(xs: list[T]) -> T | None:
    return xs[0] if xs else None

Protocols — Structural Typing (Pythonic interfaces)

from typing import Protocol
class SupportsSend(Protocol):
    def send(self, msg: str) -> None: ...

def notify(sender: SupportsSend, msg: str) -> None:   # accepts ANYTHING with a matching send()
    sender.send(msg)
# Duck typing, but CHECKED: no explicit "implements" needed — if it has send(str)->None, it fits.
# This is how you type "composition over inheritance" (OOP lesson) statically.

Static Checking & Runtime Validation

# Static (build/CI) — mypy or pyright:
mypy src/                    # reports type errors WITHOUT running the code
# strict mode catches the most (like TS strict): mypy --strict

# Runtime (boundaries) — Pydantic validates AND coerces external data:
from pydantic import BaseModel
class User(BaseModel):
    id: int; name: str; role: Literal["admin", "viewer"] = "viewer"
User.model_validate(json_from_api)       # raises a precise error on bad shape → typed object
# Static types verify your code's internal consistency; Pydantic guarantees external data
# matches at the boundary. Use both — the exact TS lesson pattern (zod ≈ Pydantic).

Interview Questions

Q1: Do Python type hints affect runtime behavior? What actually checks them?

No — the interpreter ignores annotations for execution (they're stored in __annotations__ but not enforced); def f(x: int) happily runs f("s"). Checking is external and static: mypy/pyright analyze the code without running it, editors use hints for autocomplete/errors, and libraries (Pydantic, FastAPI, dataclasses) READ them at runtime for validation/serialization. So value = static checker in CI + boundary validation for external data.

Q2: How do you guarantee data from an API actually matches its type?

Runtime validation, because hints are erased. Define a Pydantic BaseModel and Model.model_validate(payload) — it checks and coerces, raising a precise error on mismatch and returning a typed object. Static hints verify your internal code; Pydantic verifies the boundary. It's the identical zod/unknown pattern from the TypeScript lesson: never trust external shape, validate where it enters.

Q3: What is a Protocol and why is it Pythonic?

A Protocol declares a structural interface — "anything with these methods/attributes" — checked statically without requiring the class to explicitly inherit it. It types duck typing: a function annotated SupportsSend accepts any object with send(str)->None. This matches Python's composition/duck-typing culture (OOP lesson) while still giving mypy something to verify — unlike nominal ABCs that require explicit subclassing.

🏋️ Practice: annotate a small module and run mypy --strict, fixing the errors; write a Protocol and a function that accepts anything matching it; prove a wrong-typed call runs anyway (no runtime error) then catch it with mypy; validate a JSON payload with a Pydantic model and trigger a validation error; replace an Any with a precise union.

🎯 Key Takeaways

  • Hints are optional and erased at runtime — a checker (mypy/pyright) validates; libraries read them.
  • Builtin generics (list[str]), unions (X | None), Literal, TypedDict, TypeVar cover most needs.
  • Protocols type duck typing structurally — the static face of composition.
  • Static types check your code; Pydantic validates external data at the boundary — use both.
  • Avoid Any (contagious); run the checker in CI or hints rot; the ROI is the same as TypeScript's.
