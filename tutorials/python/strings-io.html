Strings, Files & I/O Basics

Text processing is Python's bread and butter. The two concepts that trip people up: strings are immutable Unicode (bytes ≠ str), and files must be handled with with so they always close. Get encoding right and a whole class of production bugs disappears.

String Essentials

s = "  Hello, World  "
s.strip()  s.lower()  s.upper()  s.title()
s.replace("l", "L")  s.split(",")  ", ".join(parts)
s.startswith("He")  s.endswith("ld")  "World" in s
s.find("o")   # index or -1     s.count("l")
"café".encode("utf-8")          # str → bytes
b"caf\xc3\xa9".decode("utf-8")  # bytes → str
# Strings are IMMUTABLE: every method returns a NEW string. Building strings in a loop with
# += is O(n²); use "".join(list_of_pieces) instead — the classic performance fix.

Formatting & Parsing

f"{name:>10}"  f"{pi:.2f}"  f"{n:,}"  f"{ratio:.1%}"  f"{x:08.2f}"   # f-string mini-language
"{} scored {}".format(name, score)        # older .format
name, score = line.split(",")             # parse CSV-ish
int("42")  float("3.14")  # convert; ValueError on bad input → validate/try
# For real CSV/JSON use the stdlib, never hand-splitting:
import json, csv
data = json.loads(text);  json.dumps(obj, indent=2)

Files — Always Use `with`

with open("data.txt", encoding="utf-8") as f:      # ALWAYS specify encoding
    for line in f:                                 # streams line-by-line — constant memory
        process(line.rstrip("\n"))
# `with` guarantees f.close() even on exception (context manager — decorators lesson).
# Modes: "r" read, "w" write (truncates!), "a" append, "x" create-exclusive,
#        "rb"/"wb" binary. Default text mode decodes with an encoding.

with open("out.json", "w", encoding="utf-8") as f:
    json.dump(obj, f, indent=2)                    # write JSON

from pathlib import Path                            # the modern path API — use over os.path
p = Path("data") / "users.csv"                      # / operator joins paths, OS-correct
p.exists()  p.read_text(encoding="utf-8")  p.suffix  p.stem  list(p.parent.glob("*.csv"))

The Encoding Trap (bytes vs str)

open("f.txt")                    # ❌ uses the platform default encoding — Windows≠Linux → mojibake/crashes
open("f.txt", encoding="utf-8")  # ✅ explicit, portable, correct
# str = Unicode TEXT (characters). bytes = raw octets. Network/files are bytes; you decode to
# str and encode back. "UnicodeDecodeError" almost always means a wrong/assumed encoding.
len("café")        # 4 characters
len("café".encode())  # 5 bytes (é is 2 bytes in UTF-8)

Interview Questions

Q1: Why is repeated string concatenation slow, and what's the fix?

Strings are immutable, so s += piece allocates a NEW string and copies the whole accumulated content each time — O(n²) over n pieces. Build a list and "".join(pieces) once (single allocation), or write to an io.StringIO buffer. Same immutability reasoning as the types lesson.

Q2: Explain str vs bytes and a real bug from confusing them.

str is a sequence of Unicode code points (text); bytes is raw octets. Files/sockets deal in bytes; you decode to str with an encoding and encode back. Bug: opening a UTF-8 file without specifying encoding uses the platform default (cp1252 on Windows), so "café" reads as "cafÃ©" or raises UnicodeDecodeError — passing tests on the dev's Mac, failing in a Windows/container prod. Always pass encoding="utf-8".

Q3: Why is `with open(...)` preferred over open()/close()?

with uses the file's context manager: __exit__ closes the handle deterministically even if the body raises — no leaked descriptors, buffers flushed. Manual close() is skipped on exceptions unless wrapped in try/finally, which is exactly what with automates. It generalizes to any resource (locks, DB connections, transactions) — the pattern is covered in the decorators/context lesson.

🏋️ Practice: read a large log line-by-line counting ERROR lines (constant memory); prove += string-building is slow vs join with timeit; write and re-read a JSON file with json.dump/load; use pathlib to glob all .csv files in a folder; trigger and fix a UnicodeDecodeError by setting encoding.

🎯 Key Takeaways

  • Strings are immutable Unicode; build with "".join(), never += in loops.
  • str = text, bytes = octets; decode/encode at boundaries; always pass encoding="utf-8".
  • Always with open(...) — deterministic close on success or exception.
  • Iterate files line-by-line for constant memory; use csv/json modules, not hand-parsing.
  • pathlib.Path over os.path; f-string mini-language for formatting.
