FastAPI & Production APIs Ultra-Advanced

FastAPI is the modern Python API framework: type hints drive validation, serialization, and auto docs. It ties together everything in this track — typing, async, Pydantic, dependency injection — into production services. (Django/DRF is the batteries-included alternative; the concepts transfer.)

Type Hints as the API Contract

from fastapi import FastAPI, HTTPException, Depends
from pydantic import BaseModel, EmailStr, Field
app = FastAPI()

class UserIn(BaseModel):                      # request model — VALIDATED automatically
    name: str = Field(min_length=1, max_length=80)
    email: EmailStr
    age: int = Field(ge=0, le=150)
class UserOut(BaseModel):                     # response model — controls what's EXPOSED
    id: int; name: str; email: EmailStr        # (note: no password field → never leaked)

@app.post("/users", response_model=UserOut, status_code=201)
async def create_user(user: UserIn):          # body parsed+validated into UserIn or 422 auto
    row = await repo.insert(user)
    return row                                 # serialized through UserOut
# The hints ARE the contract: FastAPI validates input (422 on bad data), serializes output,
# and generates OpenAPI docs at /docs — no separate schema. This is the typing lesson,
# operationalized (Pydantic = runtime validation at the boundary).

Path/Query Params & Errors

@app.get("/users/{user_id}", response_model=UserOut)
async def get_user(user_id: int, include_orders: bool = False):   # path int + query bool, typed
    user = await repo.get(user_id)
    if not user:
        raise HTTPException(status_code=404, detail="User not found")   # → clean JSON error
    return user

Dependency Injection — the Killer Feature

async def get_db():                            # a dependency (resource with teardown)
    async with pool.acquire() as conn:
        yield conn
def require_user(token: str = Depends(oauth2)) -> User:   # auth as a dependency
    user = decode(token)
    if not user: raise HTTPException(401)
    return user

@app.get("/me")
async def me(user: User = Depends(require_user), db = Depends(get_db)):
    return await repo.profile(db, user.id)
# Depends() wires shared concerns (DB sessions, auth, pagination, rate limits) declaratively
# and makes them trivially overridable in tests (app.dependency_overrides). It's constructor
# injection for endpoints — the same DI idea as the backend frameworks.

Async & Performance

@app.get("/dashboard")
async def dashboard():
    user, orders, ads = await asyncio.gather(   # concurrent I/O (concurrency lesson)
        get_user(), get_orders(), get_ads())
    return {...}
# FastAPI runs on ASGI (uvicorn) — async all the way down for high I/O concurrency.
# ⚠ a blocking call in an async endpoint stalls the loop → use async DB drivers (asyncpg),
# or def (sync) endpoints, which FastAPI runs in a threadpool automatically. Never block the loop.

Interview Questions

Q1: How do type hints give FastAPI validation, serialization, and docs "for free"?

FastAPI reads the annotations at startup: a Pydantic model parameter means "parse and VALIDATE the request body into this shape (422 on mismatch)"; response_model shapes/validates output (and hides unlisted fields); path/query types coerce and validate params. From the same hints it generates an OpenAPI schema → interactive /docs. So one source (typed models) drives input validation, output serialization, and documentation — the typing lesson's "libraries read hints at runtime" taken to its conclusion.

Q2: Why is a blocking call in an async endpoint dangerous, and what are the options?

FastAPI's async endpoints run on one event loop (ASGI); a blocking call (sync DB driver, requests, heavy CPU) never yields, freezing all concurrent requests — the same "don't block the loop" rule as Node. Options: use async-native libraries (asyncpg/httpx); declare the endpoint as plain def, which FastAPI runs in a threadpool (fine for blocking I/O); or offload CPU work to a process pool / task queue. Choose by whether the work is async-capable, blocking-I/O, or CPU.

Q3: What does Depends() give you beyond convenience?

Declarative composition of cross-cutting concerns (DB sessions with teardown, auth, pagination, rate limiting) that are resolved per-request, cached within a request, and — critically — overridable in tests via app.dependency_overrides (swap the real DB/auth for fakes without touching handlers). It's dependency injection for endpoints: testable, reusable, and explicit, versus threading globals through every function.

🏋️ Practice: build a users CRUD with Pydantic UserIn/UserOut and response_model; add a get_db and require_user dependency; trigger a 422 with bad input and a 404 via HTTPException; use asyncio.gather in an endpoint; write an httpx AsyncClient test with dependency_overrides pointing at a test DB.

🎯 Key Takeaways

  • Type hints + Pydantic drive validation (422), serialization (response_model), and auto OpenAPI docs.
  • response_model shapes output and prevents leaking internal fields; validate every input at the boundary.
  • Depends() = DI for endpoints — DB/auth/pagination composed and test-overridable.
  • ASGI/async for I/O concurrency; never block the loop (async drivers or sync-def threadpool).
  • Production concerns (auth/logging/config/deploy/observability) mirror the Node lesson — platform differs, requirements don't.
