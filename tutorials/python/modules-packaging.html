Modules, Packaging & Environments Intermediate

How Python code is organized, shared, and reproduced. The non-negotiable senior skill here is environment isolation — "works on my machine" is nearly always a dependency problem, and virtual environments plus a lockfile are the cure.

Modules & Imports

import math                      # whole module
from math import sqrt, pi        # specific names
from utils import helper as h    # alias
import package.subpackage.mod    # dotted path

# Every .py is a module; a folder with __init__.py (or a namespace) is a package.
if __name__ == "__main__":       # runs only when executed directly, not when imported
    main()                       # the standard script entry-point guard
RuleWhy
Prefer absolute imports (from myapp.utils import x)Unambiguous; relative (from .utils) only within a package
Avoid from module import *Pollutes the namespace, hides origins, breaks linters
Modules run ONCE and are cached (sys.modules)Module-level code is import-time side effect; keep it side-effect-free
Beware circular importsA imports B imports A — restructure shared code downward

Virtual Environments — Isolation Is Mandatory

# A venv is a per-project Python + packages, isolated from the system and other projects:
python -m venv .venv
source .venv/bin/activate      # Windows: .venv\Scripts\activate
pip install requests
pip freeze > requirements.txt  # pin EXACT versions (the old way)
# Without a venv, projects fight over global package versions — the #1 "works on my machine".

Modern Tooling — uv / Poetry + pyproject.toml

# pyproject.toml is the standard project manifest (deps, build, tool config):
[project]
name = "myapp"
dependencies = ["fastapi>=0.110", "pydantic>=2"]
[dependency-groups]
dev = ["pytest", "ruff", "mypy"]

# uv (2020s standard, extremely fast) manages venv + install + lockfile:
uv add fastapi          # adds to pyproject + updates uv.lock (exact tree)
uv sync                  # reproduce the environment from the lockfile
uv run pytest            # run in the project env
# uv.lock / poetry.lock pins the FULL transitive tree — commit it (npm's package-lock analog).
🔑 The reproducibility chain: a manifest (pyproject.toml) declares what you want; a lockfile (uv.lock) records exactly what was resolved; CI installs from the lockfile. This is identical to the JS modules lesson (package.json + package-lock + npm ci) — same problem, same solution, different ecosystem.

Package Structure (a shippable project)

myapp/
├── pyproject.toml
├── uv.lock
├── src/myapp/            # src-layout: prevents accidental imports of uninstalled code
│   ├── __init__.py
│   ├── api.py
│   └── services/…
└── tests/
# pip install -e .  → editable install so imports resolve during development.
# Publishing: build a wheel (uv build) → upload to PyPI (or a private index).

Interview Questions

Q1: Why are virtual environments essential, and what does a lockfile add on top?

A venv isolates a project's Python and packages so projects don't fight over global versions — the root cause of most "works on my machine" issues. But a venv + loose version ranges can still resolve differently over time; the lockfile pins the EXACT transitive dependency tree, so CI and every developer install byte-identical dependencies. Manifest declares intent, lockfile guarantees reproducibility, CI installs from the lock — the same chain as npm/package-lock.

Q2: What does `if __name__ == "__main__":` do and why use it?

When a file is run directly, its __name__ is "__main__"; when imported, __name__ is the module's name. The guard runs the entry-point code only on direct execution, so importing the module (for its functions, or during tests) doesn't trigger side effects like starting a server or running a CLI. Without it, importing a script executes its top-level actions — a classic surprise.

Q3: How would you diagnose and fix a circular import?

Symptom: ImportError or an AttributeError on a half-initialized module (A imports B at load time, B imports A before A finished). Diagnose by reading the import chain in the traceback. Fixes: extract the shared piece into a third module both import; defer the import inside the function that needs it (breaks the load-time cycle); or restructure so dependencies flow one direction. It's the same load-order issue as JS ESM cycles.

🏋️ Practice: create a venv (or uv init), add a dependency, and inspect the lockfile; split a script into a package with __init__.py and absolute imports; add the __main__ guard and prove importing doesn't run main; deliberately create a circular import and fix it by extracting shared code.

🎯 Key Takeaways

  • Every .py is a module; packages are folders; absolute imports over relative; avoid import *.
  • Virtual environments isolate per-project deps — mandatory; never install globally.
  • pyproject.toml (manifest) + uv.lock/poetry.lock (exact tree) + CI sync = reproducibility.
  • __name__ == "__main__" guards entry points so imports don't trigger side effects.
  • Modules run once and cache; keep top-level side-effect-free; fix circular imports by restructuring.
