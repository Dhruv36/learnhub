Data Structures Basics

list, dict, set, tuple — the four you'll use constantly, plus knowing their Big-O (interviews test it) and the copy semantics that cause the most bugs. Choosing the right one is half of writing fast, correct Python.

The Four + Their Complexity

TypeUse forKey ops (avg)
list [ ]Ordered sequence, index accessappend/pop-end O(1); insert/pop-front/in O(n); index O(1)
dict { k:v }Key→value lookup (ordered by insertion, 3.7+)get/set/del/in O(1)
set { }Membership, dedup, set algebraadd/in/remove O(1)
tuple ( )Fixed, immutable record; dict keysindex O(1); hashable
⚡ The senior reflex: x in my_list is O(n) — a linear scan. x in my_set/my_dict is O(1). Doing membership checks in a loop over a list is the accidental O(n²) that shows up in profiling. Convert to a set first.

Dicts — the Workhorse

d = {"a": 1, "b": 2}
d.get("c", 0)                  # 0 — safe default vs d["c"] which raises KeyError
d.setdefault("x", []).append(1)
for k, v in d.items(): ...     # NOT `for k in d` then d[k]
{**d1, **d2}                   # merge (d2 wins);  d1 | d2  (3.9+)

from collections import defaultdict, Counter, deque
groups = defaultdict(list)
for u in users: groups[u.role].append(u)     # no key-exists check needed
Counter(words).most_common(3)                 # frequency counts in one line
dq = deque(maxlen=100); dq.appendleft(x)      # O(1) both ends — use for queues, NOT list.pop(0)

Copy Semantics — Shallow vs Deep (the bug factory)

a = [[1, 2], [3, 4]]
b = a            # alias — same object (types lesson)
b = a.copy()     # or a[:] or list(a) — SHALLOW: new outer list, SAME inner lists
b[0].append(9)   # a[0] changed too! 😱
import copy
b = copy.deepcopy(a)   # fully independent
# Same story as JS spread-vs-structuredClone: shallow copies share nested objects.

Slicing & Sorting

nums[1:4]  nums[::-1]  nums[::2]  nums[-3:]    # start:stop:step (stop exclusive)
sorted(users, key=lambda u: u.score, reverse=True)   # returns NEW list
users.sort(key=lambda u: (u.role, -u.score))          # in place; tuple key = multi-sort
# key runs once per element (Schwartzian); reverse -score sorts descending within role.
import heapq
heapq.nlargest(3, users, key=lambda u: u.score)       # top-K without full sort: O(n log k)

Interview Questions

Q1: Why is `in` fast for sets/dicts but slow for lists?

Sets and dicts are hash tables: membership hashes the key and checks one bucket — O(1) average. Lists are contiguous arrays with no index on values, so `in` scans element by element — O(n). The consequence: any repeated membership test should use a set; the classic optimization is turning an O(n²) list-scan loop into O(n) by pre-building a set. (Hashability requires immutability — why list keys are illegal.)

Q2: Explain shallow vs deep copy with a concrete failure.

a.copy()/list(a)/a[:] make a new outer list whose elements are the SAME inner objects. So b = a.copy(); b[0].append(9) mutates a[0] too, because b[0] IS a[0]. copy.deepcopy recursively clones. Identical to JS spread vs structuredClone — shallow copies share nested references. Deep copy is slower; often the better design is immutability (don't mutate shared nested data).

Q3: When would you use deque, Counter, and defaultdict instead of a plain list/dict?

deque: O(1) appends/pops at BOTH ends (queues, sliding windows, BFS) — list.pop(0) is O(n). Counter: tallying frequencies and most_common in one line vs manual dict increments. defaultdict: grouping/accumulating without "if key not in d" boilerplate. Reaching for collections signals you know the stdlib and the complexity trade-offs.

🏋️ Practice: time `x in list` vs `x in set` for 100k lookups; reproduce the shallow-copy nested bug and fix with deepcopy; group a list of users by role with defaultdict; get top-3 scorers with heapq.nlargest; count word frequencies with Counter; use deque as a fixed-size sliding window.

🎯 Key Takeaways

  • list=ordered O(1) end-ops but O(n) membership/front-ops; dict/set=O(1) hash lookup; tuple=immutable/hashable.
  • Membership in a loop → use a set (avoid accidental O(n²)); deque for both-ends O(1).
  • Shallow copy shares nested objects; deepcopy for full independence — or don't mutate shared data.
  • get()/defaultdict avoid KeyError; Counter counts; heapq for top-K.
  • sort/sorted with key= (tuple key = multi-level); slicing is start:stop:step, stop exclusive.
