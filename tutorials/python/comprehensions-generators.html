Comprehensions, Iterators & Generators Intermediate

Comprehensions are Python's signature way to transform data — the map/filter of the language, but readable. Generators extend the idea to lazy, memory-constant sequences: the tool for streaming data too big to fit in RAM. This is where Python code starts to look Pythonic.

Comprehensions — Transform in One Expression

squares   = [x*x for x in range(10)]                       # list
evens     = [x for x in nums if x % 2 == 0]                # with filter
pairs     = [(x, y) for x in a for y in b]                 # nested loops
by_id     = {u.id: u for u in users}                       # dict comprehension
unique    = {w.lower() for w in words}                     # set comprehension
matrix    = [[row[i] for row in grid] for i in range(cols)]  # transpose
# Readability rule: if it needs 3+ clauses or nesting you squint at, use a loop instead.
# Comprehensions are faster than equivalent append-loops (optimized bytecode).

Generators — Lazy Sequences, Constant Memory

# Generator EXPRESSION — like a comprehension but lazy (parens, not brackets):
total = sum(x*x for x in range(1_000_000))     # no million-item list built — streams

# Generator FUNCTION — yield pauses and resumes, keeping state:
def read_large_file(path):
    with open(path, encoding="utf-8") as f:
        for line in f:
            yield line.rstrip("\n")            # one line at a time, file stays open
for line in read_large_file("10gb.log"):        # constant memory regardless of size
    if "ERROR" in line: handle(line)

def fibonacci():                                # infinite — safe because lazy
    a, b = 0, 1
    while True:
        yield a; a, b = b, a + b
from itertools import islice
list(islice(fibonacci(), 10))                    # take first 10
🧠 list comp vs generator: [...] builds the whole list in memory now; (...) yields lazily, one item at a time. For a pipeline you consume once (sum, any, a for-loop, feeding another function), the generator uses O(1) memory instead of O(n). This is the JS generators lesson in Python — same laziness, same streaming use case.

The itertools Toolkit

from itertools import chain, groupby, count, cycle, combinations, product, accumulate
chain(list_a, list_b)                 # flatten iterables lazily
groupby(sorted(rows, key=k), key=k)   # group consecutive (SORT first!)
accumulate([1,2,3,4])                 # running totals: 1,3,6,10
combinations(items, 2)  product(a, b) # combinatorics without nested loops
# itertools builds lazy pipelines — the functional, memory-efficient way to process streams.

Iterators — What for-loops Actually Do

it = iter([10, 20])       # __iter__ → iterator
next(it)  # 10            # __next__ → value, raises StopIteration when done
# A `for` loop calls iter() then next() until StopIteration. Any object with __iter__/__next__
# (or a generator) is iterable — the protocol behind comprehensions, unpacking, and *args.

Interview Questions

Q1: When would you use a generator instead of a list comprehension?

When the data is large/infinite/streamed and you consume it once: sum/any/max, a for-loop, or feeding another function. The generator holds one item at a time (O(1) memory) vs the list's O(n). Reading a 10GB file line-by-line, an infinite Fibonacci with islice, or a transform pipeline are canonical cases. Trade-off: generators are single-pass and unsized — use a list when you need random access, len, or multiple iterations.

Q2: What does yield do to a function, mechanically?

It makes the function a generator: calling it runs NO body, returning a generator object. Each next() runs until the next yield, emits that value, and SUSPENDS with all locals preserved; resuming continues after the yield. StopIteration ends it. It's a resumable stack frame — the same model as JS generators — enabling lazy, stateful sequences without manually managing an iterator class.

Q3: Why must you sort before itertools.groupby?

groupby groups only CONSECUTIVE equal keys (it's a single streaming pass, O(n), no buffering of the whole input). Unsorted data with scattered keys produces multiple tiny groups per key. Sorting by the same key first clusters them. This design keeps groupby lazy and memory-light, unlike a dict-based group that must see everything — the trade-off is you pay the sort.

🏋️ Practice: rewrite a filter+map loop as a comprehension; sum squares of a million numbers with a generator expression and confirm low memory; write a generator that streams a large file's ERROR lines; use islice on an infinite generator; group users by role with sorted+groupby; build a lazy pipeline with itertools.chain + a genexpr.

🎯 Key Takeaways

  • Comprehensions ([...], {...}, {k:v}) are the readable, fast transform idiom — keep them simple.
  • Generator expressions (...) and yield functions are LAZY: O(1) memory, single-pass, can be infinite.
  • Use generators for large/streamed/one-time data; lists for random access, len, multiple passes.
  • itertools builds lazy pipelines; groupby needs pre-sorting (groups consecutive keys).
  • for-loops = iter()+next() until StopIteration — the protocol behind everything iterable.
