OOP & Dataclasses Intermediate

Python OOP is pragmatic: no true private members (convention, not enforcement), multiple inheritance resolved by a defined algorithm, and dunder methods that hook classes into language syntax. Modern Python leans on @dataclass to kill boilerplate. Know both the mechanics and when composition beats inheritance.

Classes & the self / dunder Model

class Account:
    interest_rate = 0.02                 # CLASS attribute (shared by all instances)
    def __init__(self, owner, balance=0):
        self.owner = owner               # INSTANCE attributes (per object)
        self._balance = balance          # _single = "internal by convention"
        self.__secret = 1                # __double = name-mangled to _Account__secret

    def deposit(self, amt):              # `self` is explicit — the instance
        if amt <= 0: raise ValueError("positive only")
        self._balance += amt
        return self
    def __repr__(self):                  # dev-readable; falls back for print if no __str__
        return f"Account({self.owner!r}, {self._balance})"
    def __eq__(self, other):             # enables ==  (define __hash__ too if used in sets)
        return isinstance(other, Account) and self._balance == other._balance
    @property
    def balance(self):                   # computed attribute, read like a field
        return self._balance
# Dunders wire classes into syntax: __len__→len(), __getitem__→x[i], __iter__→for,
# __enter__/__exit__→with, __add__→+, __call__→x(). "Pythonic" = using them.

@dataclass — Boilerplate, Deleted

from dataclasses import dataclass, field
@dataclass(slots=True, frozen=True)      # slots=less memory; frozen=immutable+hashable
class Point:
    x: float
    y: float
    tags: list[str] = field(default_factory=list)   # mutable default → factory (control-flow trap!)
    def dist(self) -> float: return (self.x**2 + self.y**2) ** 0.5
# Auto-generates __init__, __repr__, __eq__ (and __hash__ if frozen). frozen mirrors the
# immutability discipline; field(default_factory=...) avoids the shared-mutable-default bug.
# For validation/serialization at API boundaries, Pydantic models (fastapi lesson) go further.

Inheritance, super() & the MRO

class Savings(Account):
    def __init__(self, owner, balance=0, rate=0.05):
        super().__init__(owner, balance)     # cooperative — call the parent
        self.rate = rate
    def deposit(self, amt):                   # override + extend
        return super().deposit(amt * (1 + self.rate))

# Multiple inheritance is resolved by the MRO (C3 linearization):
class A: ...
class B(A): ...
class C(A): ...
class D(B, C): ...
D.__mro__     # (D, B, C, A, object) — super() follows THIS order, not "the parent"
# super() means "next in the MRO", which is why cooperative multiple inheritance works.

Composition Over Inheritance

# Instead of deep hierarchies, assemble behavior:
class EmailNotifier:  def send(self, msg): ...
class Order:
    def __init__(self, notifier): self.notifier = notifier   # inject a capability
    def place(self): ...; self.notifier.send("placed")
# Same lesson as the JS classes lesson: inherit stable is-a contracts; COMPOSE capabilities.
# Python's "duck typing" makes composition natural — anything with .send() works, no interface
# needed (though typing.Protocol lets you declare the shape — typing lesson).

Class vs Static vs Instance Methods

class User:
    @classmethod
    def from_json(cls, d): return cls(d["name"])   # alternative constructor (cls = the class)
    @staticmethod
    def valid_email(s): return "@" in s            # namespaced helper, no self/cls
    def greet(self): ...                            # normal instance method

Interview Questions

Q1: What is the MRO and why does super() depend on it?

The Method Resolution Order is the linearized sequence (C3 algorithm) in which Python searches classes for attributes/methods — inspect via Class.__mro__. super() doesn't mean "the parent"; it means "the NEXT class in the MRO relative to the current one", which is what makes cooperative multiple inheritance work (each class's super() call chains through the diamond exactly once). Misunderstanding this causes "super called the wrong class" bugs in multiple inheritance.

Q2: What does @dataclass generate, and when would you NOT use it?

It generates __init__, __repr__, __eq__ (and __hash__ when frozen), plus supports defaults, slots (memory), and frozen (immutability). Use it for data-holding classes. Skip it when you need heavy custom construction/behavior (a plain class is clearer), or when you need runtime validation and serialization at boundaries — then reach for Pydantic (which validates types at runtime, unlike dataclasses which don't enforce annotations).

Q3: How does Python handle "private" members, and how do you design around no enforcement?

By convention: _name signals "internal, don't touch"; __name triggers name-mangling to _Class__name (avoids subclass collisions, not real privacy — still reachable). There's no enforced access control. Design accordingly: expose a clear public API via methods/@property, document _internal as unstable, and rely on the team convention + linters rather than the compiler. "We're all consenting adults" is the stated philosophy.

🎯 Key Takeaways

  • self is explicit; class vs instance attributes differ; dunders wire classes into syntax (len/iter/with/+).
  • @dataclass generates init/repr/eq (+hash if frozen); use field(default_factory) for mutable defaults.
  • super() = next in the MRO (C3), not "the parent" — the key to multiple inheritance.
  • Privacy is convention (_name); __name is mangling, not security.
  • Inherit stable contracts; compose capabilities (duck typing / Protocol) — deep hierarchies age badly.
