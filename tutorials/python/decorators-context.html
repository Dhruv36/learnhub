Decorators & Context Managers Intermediate

Two features that look like magic and are everywhere in real code (Flask routes, pytest fixtures, @property, with open()). Both are ordinary Python once you see the mechanism: decorators wrap functions; context managers wrap a block with setup/teardown. Being able to write both from scratch is a mid/senior checkpoint.

Decorators — Functions Wrapping Functions

import functools, time
def timed(fn):
    @functools.wraps(fn)                 # preserves fn's __name__/__doc__ (don't skip this!)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = fn(*args, **kwargs)     # call the original
        print(f"{fn.__name__} took {time.perf_counter()-start:.3f}s")
        return result
    return wrapper

@timed                                   # sugar for: slow = timed(slow)
def slow(n): ...
# A decorator is a function taking a function and returning a replacement. @timed rebinds the
# name to wrapper. The *args/**kwargs forwarding makes it work on ANY signature.

Decorators With Arguments (a factory)

def retry(times=3):                      # takes args → returns a decorator → returns a wrapper
    def decorator(fn):
        @functools.wraps(fn)
        def wrapper(*a, **k):
            for i in range(times):
                try: return fn(*a, **k)
                except Exception:
                    if i == times - 1: raise
        return wrapper
    return decorator

@retry(times=5)
def flaky(): ...
# Three nested layers = the decorator-with-arguments pattern. Read it as retry(5)(flaky).

Context Managers — Guaranteed Setup/Teardown

# The `with` statement calls __enter__ then __exit__ (even on exception):
class Timer:
    def __enter__(self): self.t = time.perf_counter(); return self
    def __exit__(self, exc_type, exc, tb):
        print(f"{time.perf_counter()-self.t:.3f}s")
        return False                     # False = don't suppress exceptions
with Timer():
    do_work()                            # timed whether it succeeds or raises

# The easy way — contextlib:
from contextlib import contextmanager
@contextmanager
def transaction(conn):
    conn.execute("BEGIN")
    try:
        yield conn                       # everything before yield = setup, after = teardown
        conn.execute("COMMIT")
    except Exception:
        conn.execute("ROLLBACK"); raise
with transaction(db) as conn:            # commit on success, rollback on error — automatically
    conn.execute(...)
🔒 Context managers guarantee cleanup — files close, locks release, transactions roll back — even when the block raises. That's why with open() is mandatory and why real DB/lock code always uses with. It's the resource-safety pattern (generators' finally, JS's AbortController) as first-class syntax.

Interview Questions

Q1: Write a decorator that times a function. What's easy to get wrong?

def timed(fn): @wraps(fn) def wrapper(*a,**k): t=perf_counter(); r=fn(*a,**k); print(perf_counter()-t); return r; return wrapper. The mistakes graders watch for: forgetting functools.wraps (wrapper masks fn.__name__/__doc__, breaking logging/introspection), not forwarding *args/**kwargs (works only on no-arg functions), and not returning the result. Bonus: a decorator-with-args needs THREE nested functions.

Q2: How does a context manager guarantee cleanup, and give two real uses.

with calls __enter__ (setup, returns the "as" value) then __exit__(exc_type, exc, tb) — which runs on normal exit AND on exception (it receives the exception info; returning True suppresses it, False propagates). So teardown is deterministic. Uses: files (close), locks (release), DB transactions (commit/rollback), temporarily changing state (cd, timing). contextlib's @contextmanager lets you write one with a try/finally around a single yield.

Q3: What does @lru_cache do and when is it dangerous?

It memoizes: caches return values keyed by (hashable) arguments, so repeat calls are O(1) — great for pure, expensive functions (recursion, parsing). Dangers: arguments must be hashable (no list/dict args); it holds references forever (unbounded @cache = memory leak — use lru_cache with maxsize); and it's wrong for impure functions (I/O, time, randomness) or when inputs mutate, returning stale results. Same caching trade-offs as every layer (JS performance lesson).

🏋️ Practice: write @timed (with wraps + arg forwarding); write @retry(times=n) as a decorator factory; memoize a slow recursive function with @lru_cache and measure; build a Timer context manager class AND a @contextmanager transaction; prove __exit__ runs when the block raises.

🎯 Key Takeaways

  • A decorator takes a function and returns a replacement; @deco is sugar for fn = deco(fn).
  • Always use @functools.wraps and forward *args/**kwargs; decorator-with-args = three nested layers.
  • @lru_cache/@cache memoize pure, hashable-arg functions — bound the size to avoid leaks.
  • Context managers (__enter__/__exit__ or @contextmanager) guarantee teardown even on exceptions.
  • with open/lock/transaction is the resource-safety idiom — never hand-roll try/finally for cleanup.
