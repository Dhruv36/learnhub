Testing with pytest Advanced

pytest is the de-facto Python test framework: plain assert, powerful fixtures, and parametrization. Testing isn't optional at mid/senior level — untested code is legacy the moment it's written. This lesson is the pytest you'll use daily plus the discipline of what to test.

The Basics — Just assert

# test_pricing.py  (pytest discovers test_*.py and test_* functions)
from pricing import apply_discount

def test_percentage_discount():
    assert apply_discount(100, 10) == 90            # plain assert — pytest introspects failures

def test_rejects_negative():
    import pytest
    with pytest.raises(ValueError, match="positive"):   # asserting exceptions
        apply_discount(-5, 10)

# Run:  pytest -q   |   pytest test_pricing.py::test_percentage_discount   |   pytest -k discount

Parametrize — One Test, Many Cases

import pytest
@pytest.mark.parametrize("price,pct,expected", [
    (100, 10, 90),
    (100, 0, 100),
    (50, 50, 25),
    (100, 100, 0),
])
def test_discount(price, pct, expected):
    assert apply_discount(price, pct) == expected
# Each tuple = a separate reported test. Table-driven tests cover edge cases without
# copy-paste — cover the BRANCHES (0%, 100%, boundaries), not just the happy path.

Fixtures — Reusable Setup/Teardown

@pytest.fixture
def db():
    conn = connect(TEST_DB)
    conn.execute("BEGIN")
    yield conn                        # everything before = setup, after = teardown
    conn.execute("ROLLBACK"); conn.close()   # runs even if the test fails (context-manager style)

def test_create_user(db):             # request the fixture by NAME as a parameter
    repo = UserRepo(db)
    user = repo.create("Asha")
    assert repo.get(user.id).name == "Asha"
# Fixtures compose (a fixture can request others) and have scopes: function (default),
# class, module, session. conftest.py shares fixtures across the whole test tree.
# The transaction-rollback fixture = fast, isolated DB tests (Node testing lesson parallel).

Mocking the Edges

from unittest.mock import patch
def test_sends_email():
    with patch("myapp.notify.send_email") as mock_send:   # patch WHERE IT'S USED, not defined
        place_order(order)
        mock_send.assert_called_once_with(order.email, "confirmed")
# Mock external boundaries (email, third-party HTTP, clock) — not your own logic.
# For HTTP, respx/responses intercept at the network layer. Inject the clock/uuid for determinism.
# The #1 mock mistake: patching the import path where the function is DEFINED, not where it's LOOKED UP.

What to Test (and Coverage Sanity)

TestSkip
Business logic branches & edge cases (0, negative, empty, boundary)Third-party library internals
Error paths (raises, invalid input)Trivial getters/pass-throughs
Integration through real DB (rollback fixture)Exact log strings
Regressions (a test per fixed bug)Coverage % for its own sake

Coverage (pytest --cov) finds UNtested code but 100% proves nothing about quality — branch coverage of the paths that matter beats line coverage of trivia.

Interview Questions

Q1: How do pytest fixtures work, and why prefer them over setUp/tearDown?

A fixture is a function decorated @pytest.fixture; tests receive it by naming it as a parameter, and pytest resolves the dependency graph. With yield, code before is setup and after is teardown (runs even on failure). They COMPOSE (fixtures request fixtures), have scopes (function→session), and are shared via conftest.py. Advantages over class-based setUp/tearDown: explicit per-test dependencies, reuse without inheritance, and fine-grained scope control. The rollback DB fixture is the fast-isolation workhorse.

Q2: You mock send_email but the real email still sends. Why?

You patched it at its definition (unittest.mock.patch("mailer.send_email")) but the code under test imported it into another module's namespace (from mailer import send_email), so it looks up its OWN reference — which you didn't patch. Patch where it's USED: patch("myapp.orders.send_email"). This "patch where it's looked up" rule is the most common mocking bug in Python.

Q3: What should you test, and is 100% coverage the goal?

Test behavior and branches: happy path, edge cases (0/empty/negative/boundary), error paths (raises), integration through the real DB, and a regression test per fixed bug. Don't test third-party internals, trivial accessors, or exact log strings. 100% coverage isn't the goal — it measures lines executed, not correctness; a test can hit a line without asserting anything meaningful. Aim for branch coverage of code that matters, and treat coverage as a gap-finder, not a target.

🏋️ Practice: write assert + pytest.raises tests for a discount function; parametrize the edge cases; build a transaction-rollback db fixture and an integration test on top; mock an email sender (patching the right path) and assert_called_once_with; add --cov and find one untested branch.

🎯 Key Takeaways

  • pytest uses plain assert with rich introspection; pytest.raises asserts exceptions.
  • @parametrize = table-driven tests covering branches/edges without duplication.
  • Fixtures (yield setup/teardown, scopes, conftest.py) compose reusable state; rollback fixture = isolated DB tests.
  • Mock external EDGES, patching where the name is USED — not your own logic.
  • Test branches and error paths; coverage finds gaps but 100% ≠ quality.
