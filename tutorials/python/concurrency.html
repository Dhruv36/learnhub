Concurrency: GIL, Threads, Async Advanced

The most misunderstood area of Python — and a guaranteed senior interview topic. One rule organizes it all: the GIL means threads don't run Python bytecode in parallel, so you pick your tool by whether work is I/O-bound (threads/async) or CPU-bound (processes). Get this decision right and you avoid the "why didn't threading speed it up?" trap.

The GIL — What It Actually Is

The Global Interpreter Lock: a mutex letting only ONE thread execute Python bytecode at a time.
Consequence: multiple threads do NOT give CPU parallelism for pure-Python work.
BUT the GIL is RELEASED during I/O (network, disk) and inside many C extensions (numpy) —
so threads DO overlap while waiting on I/O.

⇒ Threads help I/O-bound work; they do NOT help CPU-bound work. For CPU parallelism you need
  multiple PROCESSES (separate interpreters, separate GILs).
(Note: 3.13+ ships an experimental free-threaded/no-GIL build — know it exists; the mental
 model above is still the default reality.)

The Decision Table (memorize this)

WorkloadToolWhy
I/O-bound, many tasks (APIs, DB, files)asyncioOne thread, thousands of awaits — cheapest concurrency; no GIL fight (all waiting)
I/O-bound, blocking librariesThreadPoolExecutorThreads overlap during I/O; GIL released while waiting
CPU-bound (parsing, math, image)ProcessPoolExecutor / multiprocessingTrue parallelism — each process has its own GIL & core

asyncio — Cooperative Concurrency (one thread)

import asyncio, aiohttp
async def fetch(session, url):
    async with session.get(url) as r:
        return await r.json()

async def main(urls):
    async with aiohttp.ClientSession() as s:
        results = await asyncio.gather(*(fetch(s, u) for u in urls))   # ALL concurrently
    return results
asyncio.run(main(urls))
# await yields control at I/O points so one thread juggles thousands of requests.
# ⚠ ONE blocking call (requests.get, time.sleep, heavy CPU) freezes the WHOLE loop —
# use async libraries (aiohttp/httpx/asyncpg) or offload to a thread/process executor.
# This is the same event-loop model as the JS async lesson.

Threads & Processes via concurrent.futures

from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor
# I/O-bound with blocking libs → threads:
with ThreadPoolExecutor(max_workers=20) as ex:
    results = list(ex.map(download, urls))          # overlap during network waits

# CPU-bound → processes (bypass the GIL, use all cores):
with ProcessPoolExecutor() as ex:
    results = list(ex.map(cpu_heavy, chunks))       # true parallelism
# Processes cost more (separate memory, pickling args/results across the boundary) —
# worth it only when the compute dwarfs the overhead. Same trade-off as Node worker_threads.

Thread Safety (when you do share state)

import threading
lock = threading.Lock()
with lock:                    # protect shared mutable state
    counter += 1              # even with the GIL, += is read-modify-write → race without a lock
# Prefer message passing (queue.Queue) over shared state — fewer races, the same philosophy
# as the JS workers lesson. The GIL prevents memory corruption, NOT logical races.

Interview Questions

Q1: What is the GIL and how does it drive your concurrency choice?

A mutex allowing only one thread to run Python bytecode at a time, so threads give no CPU parallelism for pure-Python code. But it's released during I/O and in C extensions, so threads DO overlap I/O waits. Therefore: I/O-bound → asyncio (cheapest) or threads; CPU-bound → processes (each has its own GIL/core). The "threading didn't speed up my number-crunching" surprise is exactly the GIL — the fix is multiprocessing.

Q2: Why can a single blocking call cripple an asyncio program?

asyncio is cooperative on ONE thread: coroutines yield control at await points so the loop can run others. A blocking call (requests.get, time.sleep, a CPU loop) never yields, so the single loop thread is stuck — every other task stalls until it returns. Fixes: use async-native libraries, or push the blocking work off-loop with asyncio.to_thread / run_in_executor (threads for blocking I/O, a process pool for CPU). Same "don't block the loop" rule as Node.

Q3: threads vs processes vs asyncio — pick for: (a) 1000 HTTP calls, (b) resizing 10k images, (c) a legacy blocking DB driver.

(a) asyncio with an async HTTP client — thousands of concurrent awaits on one thread, minimal overhead. (b) ProcessPoolExecutor — CPU-bound, needs real parallelism across cores past the GIL. (c) ThreadPoolExecutor — the driver blocks and isn't async, but the GIL releases during its I/O so threads overlap the waits. Matching workload (I/O vs CPU) and library nature (async vs blocking) to the tool is the whole skill.

🏋️ Practice: time a CPU-heavy function across ThreadPool (no speedup) vs ProcessPool (scales with cores); fetch many URLs with asyncio.gather and compare to sequential; drop a time.sleep into an async coroutine and watch it stall the loop, then fix with asyncio.sleep/to_thread; add a Lock around a shared counter and prove the race without it.

🎯 Key Takeaways

  • The GIL = one thread runs Python bytecode at a time → threads give no CPU parallelism (but overlap I/O).
  • I/O-bound → asyncio (cheapest) or threads; CPU-bound → processes (own GIL/core each).
  • One blocking call freezes an asyncio loop — use async libs or offload to an executor.
  • The GIL prevents memory corruption, not logical races — lock shared state or pass messages.
  • Processes cost pickling/memory overhead — worth it only when compute dwarfs it.
