Caching Strategies Fundamentals

A memory read is ~1000× faster than a disk-backed DB query. Caching is the highest-leverage performance tool in system design — and the source of its two hardest problems: invalidation and stampedes.

The Four Write/Read Patterns

PatternHow it worksTrade-off
Cache-aside (most common)App reads cache → miss → read DB → fill cache. App writes DB → deletes cache keySimple, cache failures survivable; brief staleness window
Read-throughCache itself loads from DB on missCleaner app code; needs cache-side loader logic
Write-throughWrite goes to cache + DB synchronouslyCache always fresh; every write pays double latency
Write-backWrite to cache, flush to DB async in batchesFastest writes, absorbs bursts; data loss if cache dies before flush
⚠️ Classic interview trap: in cache-aside, delete the key on write — don't update it. Two concurrent updates can interleave and leave the cache holding the older value forever. Delete forces the next read to fetch the truth.

Eviction Policies

LRU — evict least-recently-used; the default (Redis: allkeys-lru). LFU — least-frequently-used; better when popularity is stable. TTL — every entry expires; your safety net against permanent staleness. Production rule: always set a TTL even with explicit invalidation — bugs happen.

The Three Failure Modes (senior-level material)

1. STAMPEDE / THUNDERING HERD
   Hot key expires → 10,000 requests miss simultaneously → all hit the DB → DB dies.
   Fixes: per-key mutex (one request rebuilds, rest wait) · stale-while-revalidate
          · jittered TTLs (never let 1M keys expire the same second)

2. PENETRATION
   Requests for keys that DON'T EXIST (often malicious) always miss → always hit DB.
   Fixes: cache the null result with short TTL · Bloom filter in front of the DB

3. AVALANCHE
   The whole cache tier restarts cold → 100% miss rate → DB overwhelmed.
   Fixes: cache warming before taking traffic · request rate-limiting at the DB layer
          · replicated/persistent cache (Redis AOF + replicas)

Where Caches Live (layered)

Browser cache → CDN (static) → API gateway cache → App-local (in-process)
→ Distributed cache (Redis/Memcached) → DB buffer pool

Each layer trades freshness for latency. Interview move: state which layer you're caching at and why — "cache the rendered feed in Redis for 30s, media on the CDN for a year (immutable URLs)".

💡 Staff-level insight: quote your expected hit ratio and its consequence: at 99% hits, a 300K QPS read load sends only 3K QPS to the DB — but if the cache dies, the DB takes 100×. That's why cache HA (replicas, cluster mode) is a hard requirement at scale, not an optimization.

Interview Follow-ups to Expect

• Redis vs Memcached? (Redis: data structures, persistence, replication, Lua; Memcached: simpler, multi-threaded, pure LRU cache)
• How do you invalidate across microservices? (publish invalidation events via Kafka/pub-sub, or CDC from the DB — see the Data Platforms lesson)
• What's your TTL and why? (tie it to business tolerance for staleness, not a random number)
