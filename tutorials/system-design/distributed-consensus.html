Distributed Consensus (Raft, Quorums) Senior / Staff

Several earlier lessons waved at "consensus" without opening the box: leader election (Replication & Failover), avoiding split-brain, Dynamo's tunable quorums. This lesson opens the box — HOW do a set of unreliable machines, communicating over an unreliable network, agree on ONE truth?

Why Consensus Is Hard — the Core Impossibility

// The FLP impossibility result (1985): in an ASYNCHRONOUS network (no bound on message delay) with
// even ONE faulty node, NO algorithm can GUARANTEE consensus in bounded time — you fundamentally
// can't distinguish "that node is slow" from "that node is dead" with certainty (echoing the
// Heartbeat lesson's failure-detector limit).
// Real systems sidestep this THEORETICAL impossibility with PRACTICAL assumptions: partial
// synchrony (messages usually arrive within some bound, even if not guaranteed), and accepting that
// availability may occasionally be sacrificed for correctness during truly pathological conditions.

Quorums — the Simplest Consensus-Adjacent Tool

// A quorum is simply: "more than half of N nodes must agree." With N=5, a quorum is 3.
// Why MAJORITY specifically: any two quorums (majority sets) drawn from the same N nodes are
// GUARANTEED to OVERLAP by at least one node — you can never have two DISJOINT majorities.
//   N=5, quorum=3: quorum A = {1,2,3}, quorum B = {3,4,5} → they share node 3, always
// This overlap guarantee is what PREVENTS split-brain: two DIFFERENT groups of nodes can never both
// simultaneously believe they have a valid majority, because any two majority-sized groups MUST
// share at least one common node — and that shared node can only agree with ONE of them.
🎯 This is the SAME N/W/R quorum math from the Key-Value Store case study, generalized. W+R > N guaranteeing read/write overlap and "majority quorums always overlap each other" are the identical mathematical idea — overlapping sets guarantee at least one node witnesses/enforces consistency across any two operations. Recognizing this as ONE underlying principle, reapplied, is exactly the kind of connection staff-level engineers are expected to make across seemingly different problems.

Raft — Consensus You Can Actually Explain in an Interview

// Raft (designed explicitly to be more UNDERSTANDABLE than Paxos, its predecessor) has three roles:
LEADER:    handles all client writes, replicates them to followers, decides when a write is "committed"
FOLLOWER:  passively replicates the leader's log, votes in elections
CANDIDATE: a follower that hasn't heard from a leader recently — starts an ELECTION

LEADER ELECTION:
  Follower's election timeout expires (no heartbeat from leader) → becomes CANDIDATE → requests
  votes from all other nodes → wins with a MAJORITY quorum → becomes leader, starts sending heartbeats
  // Randomized election timeouts (not identical across nodes) prevent every node from becoming a
  // candidate simultaneously and splitting the vote repeatedly.

LOG REPLICATION:
  Client write → leader appends to its own log → replicates to followers → once a MAJORITY of nodes
  have the entry, it's COMMITTED (safe — durable even if the leader now crashes) → leader applies it
  and responds to the client.
  // A write is only acknowledged once a MAJORITY has it — this is the quorum-overlap guarantee again,
  // ensuring a NEW leader (elected by a majority) is guaranteed to include every previously
  // committed entry, since their majority sets must overlap.

Where This Actually Shows Up

SystemUses
etcd, ConsulRaft — for distributed configuration/coordination (Kubernetes uses etcd as its source of truth)
ZooKeeperZAB (a Paxos-like protocol) — historically the standard for distributed coordination
CockroachDB, Spanner-like systemsRaft (per data shard) — each shard's replicas run their own independent Raft group for that shard's data

Interview Follow-ups to Expect

• Why does Raft need a MAJORITY, not just "any 2 nodes," to commit a write? (majority guarantees overlap with the NEXT majority — a minority commit could be lost if those specific minority nodes are later unavailable during a new leader election)
• What happens during a network partition that splits a 5-node Raft cluster into groups of 2 and 3? (the 3-node side can still elect a leader and make progress — it holds a majority; the 2-node side CANNOT, correctly halting rather than risking a split-brain write)
• How is Raft's quorum-based commit similar to Dynamo's W+R>N model? (both rely on the mathematical guarantee that majority/overlapping sets always intersect — the same principle, applied to leader-based vs leaderless architectures respectively)
