Cost, Capacity & SLO Engineering Senior / Staff

Mid-level system design stops at "does it work and does it scale." Staff-level design adds a third question nobody asks in a whiteboard interview but every real system answers to eventually: "does it work, scale, AND cost a reasonable amount of money for the value it delivers?"

Capacity Planning — from Estimation to a Real Number

// The Back-of-Envelope Estimation lesson taught QPS/storage math. Capacity planning adds: how much
// HEADROOM above current load do you actually provision?
Provisioned capacity = current peak load × SAFETY MARGIN (commonly 2-3x) + growth projection
// Under-provisioning: outages during traffic spikes (launches, viral moments, Black Friday).
// Over-provisioning: paying for idle capacity that delivers zero value most of the time.
// The mature answer names the SPECIFIC signal that should trigger scaling UP capacity (e.g., "when
// p99 latency crosses 200ms at 70% CPU utilization, that's the trigger to add capacity") rather than
// a vague "we'll add more servers if it gets slow."

Cost Levers — What Actually Moves the Bill

LeverSavings mechanismTrade-off
Reserved/committed capacityPay less per-unit for guaranteed long-term usageLess flexibility to scale down if load drops
Spot/preemptible instancesMuch cheaper compute, but can be reclaimed with short noticeOnly safe for fault-tolerant, interruptible workloads (batch jobs, not stateful primary databases)
Caching (revisit the Caching lesson)A cache hit is FAR cheaper than a DB query — directly reduces expensive-tier loadStaleness/invalidation complexity
Tiered storageHot data on fast/expensive storage, cold/rarely-accessed data on cheap object storageAdded complexity in the data-access layer to route correctly
NativeAOT/right-sized runtimes (the .NET track's lesson)Smaller memory footprint = more instances per node = lower infra costMigration effort, potential compatibility constraints
🎯 The senior-level framing: cost is a DESIGN CONSTRAINT, stated up front, like latency or availability. "This design assumes a budget of $X/month at Y scale" changes which architecture is "correct" just as much as a latency requirement does — a technically superior but 10x more expensive design is often the WRONG answer for a system where cost efficiency is the actual priority. Name the constraint before choosing the architecture, exactly like the Trade-off Driven Design lesson's framework.

SLIs, SLOs, and Error Budgets (revisited from Observability, applied to planning)

// The Observability lesson introduced these for MONITORING; here they drive CAPACITY DECISIONS:
// If your SLO is "99.9% of requests under 200ms," capacity planning must ensure the system hits
// that LATENCY target at PEAK load, not just average load — provisioning for average load and hoping
// peaks are "close enough" is how SLOs get silently violated during exactly the moments (high traffic)
// that matter most.
// Error budget spent on INFRASTRUCTURE issues (under-capacity) is budget NOT available for shipping
// risky features — this ties capacity planning directly to product/engineering velocity, not just ops.

The Total Cost of Ownership (TCO) Framing

// A common staff-level trap: comparing only INFRASTRUCTURE cost, ignoring OPERATIONAL cost.
// Self-hosted Kafka: cheaper infra bill, but needs a team to operate/upgrade/monitor/debug it.
// Managed Kafka (Confluent Cloud, MSK): higher infra bill, but near-zero operational burden.
// TCO = infrastructure cost + engineering time to operate it + cost of OUTAGES from operational gaps
// A "cheaper" self-hosted option can easily be MORE expensive in TCO once you account for the
// engineering hours spent operating it — especially at a company without deep infra expertise already
// on staff. This is exactly the calculus behind "buy vs build" and managed-service decisions.

Interview Follow-ups to Expect

• How would you decide between building a feature yourself vs using a managed/third-party service? (compare TOTAL cost of ownership — infra + engineering time + operational risk — not just the sticker price of either option)
• Your system is hitting its SLO 99.5% of the time against a 99.9% target — what do you do? (you're burning error budget faster than sustainable — freeze risky changes, investigate root cause, likely a capacity or architecture gap needs addressing before shipping new risk)
• How do you justify a 3x cost increase for a resilience improvement to a skeptical stakeholder? (translate it into business terms — expected outage cost/frequency avoided, revenue protected — a reliability investment is fundamentally an insurance/ROI conversation, not a technical one)
