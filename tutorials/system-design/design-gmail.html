Design Gmail Case Study

Email is deceptively hard: it's a store-and-forward system with STRICT durability requirements (an email must never be silently lost), full-text search over a user's entire history, spam filtering at massive scale, and — the most-tested part — reliable delivery across independently-operated mail servers using SMTP.

Requirements

Functional: send/receive email, organize (labels/folders), search full history, attachments, spam filtering. Non-functional: NEVER lose an email (durability is non-negotiable), fast search across potentially millions of stored emails per user, handle large attachments efficiently.

Sending Email — SMTP & Store-and-Forward

Sender's client → Sender's SMTP server → (may hop through relay servers) → Recipient's SMTP server
                                                                              → stored in recipient's mailbox
// STORE-AND-FORWARD: each hop durably STORES the message before forwarding — if the next hop is
// unreachable, the message isn't lost, it's retried from durable storage. This is why email can
// tolerate a recipient server being briefly down (unlike a synchronous request that would just fail).
// DNS MX records tell a sending server WHICH mail server handles a given domain's incoming mail.

Storage — Metadata vs Content, Separated

// An email's METADATA (from, to, subject, labels, read/unread, timestamp) is small, structured,
// frequently QUERIED (list view, search filters) — fits a relational/indexed store well.
// An email's CONTENT (body, attachments) is large, mostly WRITE-ONCE, rarely modified after receipt —
// fits BLOB/object storage (like S3) better, with the metadata store holding a REFERENCE to it.
metadata DB: { emailId, from, to, subject, labels: [...], blobRef: "s3://.../emailId" }
// This separation is a recurring systems-design theme: don't force fundamentally different access
// patterns (small/hot/structured vs large/cold/write-once) into the SAME storage system.
🎯 Attachments specifically should NEVER be inline in the metadata database. A 25MB attachment stored directly in a row bloats the database, slows down every query that touches that table (even ones not needing the attachment), and complicates replication. Store attachments in object storage (S3), keep only a REFERENCE (URL/key) in the metadata row — the exact same large-blob-out-of-the-hot-path principle that shows up across many designs (avatars, video, any large binary content).

Full-Text Search Over a Mailbox

// Reuses the Search Engine case study's core structure, scoped PER USER (or per shard of users):
// build an INVERTED INDEX over each user's emails (subject + body + sender), updated incrementally
// as new mail arrives (not a full rebuild each time — an ADD-to-index operation per new email).
// At Gmail's scale, this means a search index SHARDED by user (or user-range) — a user's search
// query only needs to touch THEIR shard, not the entire global email corpus (unlike a web search
// engine, which must search across ALL documents for ANY user's query).

Spam Filtering — a Pipeline Stage

Incoming email → SPAM CLASSIFIER (ML model: sender reputation, content signals, user reports) →
  → score above threshold → route to Spam folder (still STORED — never silently dropped, since
    false positives on legitimate mail are a serious product failure)
  → below threshold → normal inbox delivery
// Spam classification runs as an early PIPELINE STAGE before storage/indexing — similar in spirit to
// the two-stage batch/online split from the Recommendation case study: the spam MODEL is trained
// offline on historical data, but classification of each NEW email happens inline, fast, at delivery time.

Interview Follow-ups to Expect

• How do you guarantee an email is NEVER lost, even during a server crash mid-delivery? (durable store-and-forward at every hop — write to persistent storage BEFORE acknowledging receipt to the previous hop, so a crash after ack means the message is safely recoverable)
• How would you implement "undo send" (a 10-second window to cancel a sent email)? (delay the ACTUAL outbound SMTP send by up to 10 seconds — the email sits in a short-lived queue the user can cancel from, then genuinely sends after the window if not canceled)
• How do you scale search as a single user's mailbox grows to millions of emails? (incremental indexing per email, sharded by user, so an individual mailbox's search index scales independently of overall email volume)
