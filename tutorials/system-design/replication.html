Replication & Failover Fundamentals

A single database instance is a single point of failure AND a throughput ceiling. Replication keeps copies of data on multiple nodes — solving both problems at once, but introducing the central tension of every distributed system: how fresh does a REPLICA'S copy need to be?

Leader-Follower (Primary-Replica) — the Default Topology

Writes → LEADER only → replicates to → FOLLOWER 1, FOLLOWER 2, FOLLOWER 3
Reads  → can go to ANY node (leader or followers) — this is how you SCALE READS horizontally

Single writer avoids write-conflict resolution entirely — the leader linearizes all writes. Followers exist for read scaling AND for failover (promote a follower if the leader dies).

Synchronous vs Asynchronous Replication

SynchronousAsynchronous
Write acknowledged afterFollower(s) confirm receiptLeader writes locally, doesn't wait
Write latencyHigher (waits on network round-trip)Lower
Data loss on leader failureNone (follower has it)Possible — unreplicated writes are lost
Used forFinancial/critical data (often 1 sync + N async — "semi-sync")Most read replicas, analytics copies
⚠️ Classic interview trap: "replication lag." With async replication, a user writes data then immediately reads from a follower that HASN'T caught up yet — they don't see their own write. Fixes: read-your-writes consistency (route a user's reads to the leader briefly after their write, or track a version/timestamp and wait for the follower to catch up), or accept eventual consistency where the product allows it.

Failover — Promoting a Follower

1. DETECT the leader is down (missed heartbeats — see the Heartbeat/Gossip lesson)
2. ELECT a new leader (the follower with the most up-to-date data, or via consensus — Raft lesson)
3. RECONFIGURE clients/proxies to point at the new leader
4. OLD leader recovers later → must REJOIN as a follower (or risk SPLIT BRAIN — two leaders accepting writes)

Split brain is the failure mode senior interviews probe hardest: if the network partitions such that clients can't reach the old leader, but the old leader is still alive and thinks it's the leader, you can end up with TWO leaders both accepting writes — a direct correctness violation. Consensus protocols (Raft) and fencing tokens exist specifically to prevent this.

Multi-Leader & Leaderless Replication (brief survey)

TopologyTrade-off
Multi-leaderWrites accepted at multiple nodes (e.g., multi-datacenter) — needs conflict resolution (last-write-wins, CRDTs, application merge logic)
Leaderless (Dynamo-style)Any node accepts writes; quorum reads/writes (W+R > N) provide consistency guarantees — the Key-Value Store case study covers this in depth

Interview Follow-ups to Expect

• How do you scale reads further once followers aren't enough? (add more followers, or shard — replication and sharding are ORTHOGONAL and usually combined)
• What happens to in-flight writes during failover? (they may fail/retry — clients need idempotent retry logic)
• Sync or async replication for a payments system? (semi-sync — at least one synchronous replica for durability, rest async for read scaling)
