Pub/Sub vs Message Queues Fundamentals

Asynchronous messaging DECOUPLES producers from consumers — the producer doesn't need the consumer to be online, fast, or even exist yet. This one architectural shift is what lets systems absorb traffic spikes, survive partial outages, and scale producers/consumers independently.

Message Queue — One Consumer Per Message

Producer → QUEUE → [msg1, msg2, msg3] → Consumer pulls msg1 (msg1 REMOVED from the queue)
                                        → Consumer 2 pulls msg2 (different consumer, different message)
// Each message is processed by EXACTLY ONE consumer (within a consumer group). Multiple consumers
// reading the SAME queue COMPETE for messages — this is how you parallelize work across a fleet.
// Examples: SQS, RabbitMQ.

Pub/Sub — Every Subscriber Gets Every Message

Publisher → TOPIC → Subscriber A receives msg1
                   → Subscriber B ALSO receives msg1 (independently)
                   → Subscriber C ALSO receives msg1
// Broadcast, not competition. Each subscriber gets its OWN copy of every message — used when
// multiple, independent systems all need to react to the SAME event.
// Examples: SNS, Google Pub/Sub, Kafka (with consumer GROUPS providing queue-like semantics too).
🎯 Kafka blurs this distinction — know why. Kafka topics support BOTH models: multiple consumer GROUPS each get a full independent copy of every message (pub/sub), while consumers WITHIN the same group compete for partitions (queue-like load balancing). This is why Kafka is the default choice for event-streaming architectures — it's not "a queue OR a pub/sub system," it's both, controlled by how you configure consumer groups.

Delivery Guarantees — the Precise Vocabulary

GuaranteeMeaningRisk
At-most-onceDelivered 0 or 1 timesMessages can be LOST
At-least-onceDelivered 1 or more timesMessages can be DUPLICATED — consumers must be idempotent
Exactly-onceDelivered exactly 1 time (semantically)Hardest to achieve — usually "effectively-once" via idempotent consumers + at-least-once delivery

The senior-level insight: true exactly-once delivery across a network is provably very hard (related to the Two Generals' Problem). Production systems achieve "effectively exactly-once" by combining AT-LEAST-ONCE delivery with IDEMPOTENT consumers (processing the same message twice has the same effect as processing it once — e.g., using a unique message ID to detect and skip duplicates). Design for idempotency; don't chase true exactly-once at the transport layer.

Ordering Guarantees

// Most distributed queues/topics guarantee ordering ONLY WITHIN A PARTITION (or a single queue
// instance), NOT globally across the whole system:
Kafka: ordering guaranteed within a PARTITION (same key → same partition → ordered)
SQS Standard: NO ordering guarantee at all (SQS FIFO queues trade throughput for strict ordering)
// If a use case needs ordering (e.g., "process a user's events in the order they happened"), route
// all of that user's messages to the SAME partition/queue via a consistent partition key (e.g., userId).

Backpressure & Dead-Letter Queues

When consumers can't keep up with producers, the queue itself acts as a BUFFER absorbing the burst — this is the core resilience benefit over synchronous calls. But queues aren't infinite: monitor queue DEPTH as a key signal, and route messages that fail processing repeatedly to a dead-letter queue (DLQ) rather than retrying forever or silently dropping them — giving you a place to inspect and manually reprocess poison messages.

Interview Follow-ups to Expect

• When would you choose a queue over pub/sub? (work distribution across a worker fleet — each job processed once, by one worker)
• When would you choose pub/sub over a queue? (multiple independent systems need to react to the same event — e.g., "order placed" triggers inventory, email, and analytics services independently)
• How do you guarantee a user's events are processed in order at scale? (partition key = user ID, route consistently to the same partition)
