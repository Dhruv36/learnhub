Design Notification System Case Study

A notification system fans out to MULTIPLE channels (push, email, SMS) from MANY internal sources (an order shipped, a comment reply, a security alert) — the core design challenge is decoupling "something happened" from "how it gets delivered," while respecting user preferences and avoiding spam.

Requirements

Functional: send notifications via push/email/SMS/in-app; respect user preferences (channel opt-in/out, quiet hours); support both transactional (password reset — must arrive) and marketing (promotional — best-effort, throttled) notifications. Non-functional: high throughput bursts (a viral post triggers millions of notifications at once), reliable delivery for critical notifications, low latency for time-sensitive ones (OTP codes).

Architecture — Decoupling Producers from Delivery

Many internal SERVICES (Order, Comments, Security...) → publish an event →
  → NOTIFICATION SERVICE (subscribes to relevant events) →
    → looks up user PREFERENCES (which channels, opted in/out) →
    → looks up user's DEVICE TOKENS / email / phone →
    → routes to the appropriate CHANNEL WORKER (Push worker, Email worker, SMS worker) →
      → each worker calls the actual THIRD-PARTY provider (APNs/FCM, SendGrid, Twilio)
// Using a message QUEUE between "event happened" and "notification sent" (the Pub/Sub lesson) is
// what absorbs BURSTS — a viral event generating a million notification-triggers doesn't need to be
// processed instantly; the queue smooths the spike into sustained worker throughput.
🎯 Separating channel WORKERS is what makes this scalable and resilient. If the Email provider (SendGrid) has an outage, only the EMAIL worker queue backs up — push and SMS notifications keep flowing unaffected. Each worker type scales independently based on its own channel's load and can retry/back off independently, rather than one monolithic "send notification" function coupling all channels' failure modes together.

Idempotency & Deduplication

// A retry (from a queue redelivery, or an upstream service retry) must NOT double-notify the user —
// "you have 3 new messages" arriving twice is a real, visible bug users notice immediately.
// Fix: an IDEMPOTENCY KEY per notification (e.g., hash of userId + eventId + channel), checked/stored
// before sending — a duplicate key is a no-op, not a re-send. Same pattern as the REST/API-Contracts
// lesson's idempotency-key discussion, applied here to prevent duplicate delivery specifically.

Priority & Throttling

TypePriorityHandling
Transactional (OTP, password reset, payment confirmation)HIGH — must arrive, low latencyDedicated high-priority queue, minimal batching
Engagement (comment reply, like)MediumCan batch/digest (e.g., "5 new likes" instead of 5 separate pushes)
Marketing (promotions)LowRate-limited per user (avoid notification fatigue), respects quiet hours

User Preference & Quiet Hours

// A preferences service (or table) stores: channels enabled per notification TYPE, quiet-hours window
// (in the user's LOCAL timezone), frequency caps. The notification service checks this BEFORE routing
// to a channel worker — critically, quiet-hours checks must respect the user's timezone, and
// TRANSACTIONAL notifications typically BYPASS quiet hours (a security alert shouldn't wait until
// morning) while marketing notifications respect them strictly.

Interview Follow-ups to Expect

• How do you avoid spamming a user with 50 separate "X liked your post" notifications in a minute? (batch/digest engagement notifications over a short window before delivering — trade a little latency for a much better user experience)
• What happens when a push notification provider (FCM) is down? (the push worker's queue backs up and retries with backoff — other channels are unaffected; consider a fallback channel for critical notifications)
• How do you guarantee an OTP arrives within 2 seconds? (a dedicated high-priority path/queue, bypass batching entirely, monitor this path's latency with a strict SLO)
