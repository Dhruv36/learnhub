Security (TLS, Encryption-at-Rest) Deep Dive

System design interviews increasingly expect you to proactively mention security — encryption in transit and at rest, secrets management, defense in depth — as part of a complete design, not just when explicitly asked. This lesson covers the mechanisms at the level a design discussion needs.

TLS — Encryption in Transit

// TLS handshake (simplified): the client and server agree on a SHARED SECRET without ever
// transmitting it in the clear, using ASYMMETRIC crypto to bootstrap a much faster SYMMETRIC session:
1. Client Hello (supported ciphers) → Server Hello (chosen cipher + certificate)
2. Client verifies the server's CERTIFICATE (signed by a trusted CA — proves "this really is example.com")
3. Client & server derive a SHARED SYMMETRIC KEY (asymmetric crypto is slow; used only to bootstrap)
4. ALL further traffic encrypted with the fast symmetric key
// Why hybrid: asymmetric (public/private key) crypto solves the "how do two strangers agree on a
// secret over an insecure channel" problem, but is too slow for bulk data — symmetric crypto (AES) is
// fast but requires both sides to already share a key. TLS uses asymmetric ONLY to establish that
// shared key, then switches to fast symmetric encryption for the actual data.

Encryption at Rest — Protecting Stored Data

LayerProtects against
Disk/volume encryption (e.g., AWS EBS encryption)Physical theft of the storage device, or unauthorized access to the underlying disk
Database-level encryption (TDE — Transparent Data Encryption)Someone with DB file access but not the DB engine's decryption key
Application-level (field) encryptionEven DBAs/operators with full DB access can't read specific sensitive fields (SSNs, payment info)
🎯 Encryption at rest does NOT protect against a compromised application or a valid, authenticated query. A classic interview trap: "we encrypt our database, so we're secure" — but if an attacker compromises the APPLICATION (SQL injection, stolen app credentials), the app decrypts data as part of normal operation and the attacker gets plaintext through that legitimate path. Encryption at rest protects against storage-level compromise (stolen disks, DB file leaks), not application-level compromise — that needs separate defenses (least privilege, input validation, WAFs).

Secrets Management

// NEVER hardcode credentials in code/config files committed to version control. Standard pattern:
App → requests a secret at RUNTIME from a secrets manager (Vault, AWS Secrets Manager)
    → secrets manager AUTHENTICATES the app (e.g., via its cloud IAM identity — no static credential
      needed just to FETCH the secret) → returns the secret, often SHORT-LIVED and auto-ROTATED
// This is the same principle from the Docker/K8s tracks' secrets lessons, applied at the systems-design
// level: secrets are runtime-injected, centrally managed, rotated, and audited — never baked into
// artifacts or checked into git.

Defense in Depth — Layered Security

Internet → WAF (blocks common attack patterns: SQLi, XSS signatures)
         → API Gateway (rate limiting, auth validation)
         → Network segmentation (private subnets — DB not directly internet-reachable)
         → Least-privilege IAM (each service can access ONLY what it needs)
         → Encryption in transit (TLS everywhere, including SERVICE-to-service, not just client-facing)
         → Encryption at rest
         → Audit logging (who accessed what, when — for detection AND compliance)
// No single layer is assumed sufficient — a breach of ONE layer shouldn't mean total compromise.

Interview Follow-ups to Expect

• Where in your architecture would you terminate TLS? (often at the load balancer/API gateway — "TLS termination" — with an option to re-encrypt for internal hops depending on the security requirements/compliance regime)
• How do you protect PII beyond basic encryption at rest? (field-level encryption for the most sensitive fields, tokenization, strict access controls, audit logging)
• How do you rotate a database credential without downtime? (short-lived credentials from a secrets manager, with overlapping validity during rotation — apps fetch fresh credentials rather than using a long-lived static one)
