Concurrency Control (ACID, Isolation) Deep Dive

Every database interview question about "two users editing the same record" or "double-booking" comes down to ISOLATION — how much concurrent transactions are allowed to interfere with each other. Most engineers can recite the isolation level names; few can explain WHICH anomaly each one actually prevents, which is exactly what separates a memorized answer from a real one.

ACID, Precisely

PropertyGuarantee
AtomicityA transaction's operations all succeed, or all roll back — no partial application
ConsistencyA transaction moves the DB from one VALID state to another (constraints/invariants hold)
IsolationConcurrent transactions appear to run as if SERIALLY, one after another (to varying degrees — see below)
DurabilityOnce committed, a transaction survives a crash (written to durable storage/WAL)

The Three Classic Anomalies — What Isolation Levels Actually Prevent

DIRTY READ:      Transaction A reads a row Transaction B has WRITTEN but NOT YET COMMITTED.
                 If B rolls back, A read data that never "really" existed.

NON-REPEATABLE READ: Transaction A reads the SAME row twice, gets DIFFERENT values, because
                 Transaction B committed an update to it IN BETWEEN A's two reads.

PHANTOM READ:    Transaction A runs the SAME query twice (e.g., "count rows where status=pending"),
                 gets a DIFFERENT ROW COUNT, because Transaction B inserted/deleted a matching row
                 in between. (Subtly different from non-repeatable read: it's about the SET of rows,
                 not one specific row's value.)

The Four Isolation Levels — Which Anomaly Each Allows

LevelDirty ReadNon-repeatable ReadPhantom Read
Read Uncommitted✅ Possible✅ Possible✅ Possible
Read Committed❌ Prevented✅ Possible✅ Possible
Repeatable Read❌ Prevented❌ Prevented✅ Possible (in the SQL standard; MySQL InnoDB actually prevents most phantoms here via next-key locking)
Serializable❌ Prevented❌ Prevented❌ Prevented
🎯 Higher isolation = fewer anomalies, but MORE locking/blocking/aborted transactions. This is a genuine trade-off, not "always pick the strongest." Read Committed is the default in Postgres/Oracle/SQL Server because it's a reasonable balance for most workloads; Serializable is reserved for genuinely correctness-critical sections (financial transfers, inventory decrement-and-check) where the performance cost is worth paying.

Optimistic vs Pessimistic Concurrency Control

PESSIMISTIC: acquire a LOCK before touching the row, hold it until commit — other transactions
             BLOCK waiting for the lock. Good under HIGH CONTENTION (many transactions touching
             the same rows) — avoids wasted work from conflicts.
   SELECT ... FOR UPDATE   -- locks the row for this transaction

OPTIMISTIC: don't lock — read a VERSION number, do your work, then on WRITE check "has the version
            changed since I read it?" If yes, someone else modified it — ABORT and retry.
            Good under LOW CONTENTION — no locking overhead for the common case of no conflict.
   UPDATE orders SET status='shipped', version=version+1 WHERE id=5 AND version=3
   -- if 0 rows affected, someone else updated it first — retry

Two-Phase Locking (2PL) — How Serializable Is Actually Implemented

GROWING phase:   acquire all locks needed, never release any yet
SHRINKING phase: release locks, never acquire any new ones after this starts
// This strict "acquire everything, then release everything" discipline is what guarantees
// serializability — but it's also exactly why 2PL can produce DEADLOCKS (two transactions each
// holding a lock the other needs) — requiring deadlock detection (a cycle in a wait-for graph) and
// aborting one transaction to break the cycle.

Interview Follow-ups to Expect

• A bank transfer debits one account and credits another — what isolation level and why? (Serializable, or explicit row locking — the correctness cost of a race condition here is unacceptable)
• Why might you choose optimistic over pessimistic locking for a high-read, low-write system? (avoids lock contention/blocking overhead when conflicts are rare)
• How do you detect and resolve a deadlock? (a wait-for graph; the DB detects a cycle and aborts one transaction, which the application must be ready to retry)
