Data Platforms (CDC, Event Sourcing) Senior / Staff

Multiple case studies referenced "keep a search index / cache / analytics store in sync with the source of truth" without explaining HOW. This lesson answers it: Change Data Capture and Event Sourcing are the two dominant patterns for propagating data changes reliably across a system, and staff-level designs are expected to name which one and why.

Change Data Capture (CDC) — Tap the Database's Own Change Log

// Every production database ALREADY records every write internally, for its own crash-recovery
// purposes (Postgres's WAL, MySQL's binlog). CDC tools (Debezium is the standard) TAIL this log and
// turn each change into an EVENT, published to a stream (Kafka):
DB write (INSERT/UPDATE/DELETE) → written to the DB's WRITE-AHEAD LOG (already happens, for durability)
                                 → Debezium reads the WAL → emits a change event → Kafka topic
                                 → consumers: search-index updater, cache invalidator, analytics pipeline...
// The KEY property: this requires ZERO application code changes. The app just writes to the database
// normally — CDC observes the database's OWN internal log, entirely out-of-band.
🎯 CDC vs "just publish an event when you write" — know the difference. An application explicitly publishing an event alongside a DB write is simpler to understand but has a real failure mode: what if the DB write succeeds but the event-publish fails (or vice versa)? Now your search index/cache is silently out of sync with the DB, with no automatic recovery. CDC eliminates this DUAL-WRITE problem entirely — there's only ONE write (to the DB), and the event stream is DERIVED from it after the fact, so it can never diverge from what the database actually committed.

Event Sourcing — the Log IS the Source of Truth

// A more radical pattern: instead of storing CURRENT STATE and deriving history from it, store the
// SEQUENCE OF EVENTS as the primary source of truth, and derive current state BY REPLAYING them:
// Traditional: accounts table: { id: 5, balance: 150 }        ← only the CURRENT value is stored
// Event Sourcing: event log: [Deposited(100), Deposited(75), Withdrew(25)]
//                 current balance = replay all events = 100+75-25 = 150 ← DERIVED, not stored directly
//
// Benefits: a COMPLETE audit trail for free (every state change is inherently recorded); can
// reconstruct state AS OF any point in time; can add NEW read models later by replaying history
// through a new projection, without having lost any information.
// Costs: replaying a long event history to get current state is expensive — mitigated with periodic
// SNAPSHOTS (cache the computed state at a point, replay only events AFTER the snapshot).

CQRS — Often Paired with Event Sourcing

// Command Query Responsibility Segregation (also covered at the code level in the ASP.NET Core
// track): WRITES go through the event log; READS are served from a separate, denormalized
// PROJECTION optimized for querying — built by consuming the event stream.
Write side:  command → validate → append event to log
Read side:   event stream → projection builder → a fast, query-optimized READ MODEL (could be a
             search index, a cache, a reporting table — often SEVERAL different projections for
             different query needs, all derived from the SAME event log)
// This is the data-platform-scale version of the ASP.NET Core CQRS lesson's within-a-service pattern.

Batch vs Streaming Processing

BatchStreaming
LatencyMinutes to hours (process accumulated data periodically)Seconds or less (process each event as it arrives)
Throughput efficiencyHigher — amortizes overhead across large chunksLower per-event efficiency, but immediate results
Use forDaily reports, ML model retraining, large aggregationsFraud detection, live dashboards, cache invalidation
ToolsSpark, Hadoop/MapReduceKafka Streams, Flink

Many real platforms use a HYBRID ("Lambda" or "Kappa" architecture): a fast streaming path for immediate-but-approximate results, reconciled/corrected later by a more thorough batch pass — the same "fast-approximate now, precise-later" trade-off pattern that appeared in the Recommendation System case study's online/offline split.

Interview Follow-ups to Expect

• How do you keep a search index in sync with a primary database without a fragile dual-write? (CDC — tail the DB's own change log, no application-level dual-write to get out of sync)
• When would you choose event sourcing over a traditional CRUD data model? (when a full audit history/point-in-time reconstruction is a genuine requirement — e.g., financial ledgers — not as a default for every domain, since it adds real complexity)
• How do batch and streaming pipelines coexist in the same platform? (streaming for low-latency approximate results; batch for periodic, thorough, corrective reprocessing of the same data)
