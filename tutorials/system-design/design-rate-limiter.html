Design API Rate Limiter Case Study

A rate limiter protects a system from being overwhelmed — by abuse, bugs, or just success. The interview's real content is the algorithm comparison (each with a genuine trade-off) and, at scale, how to make rate-limiting DECISIONS consistent across many stateless API servers.

Requirements

Functional: limit each client to N requests per time window; reject excess requests (typically HTTP 429). Non-functional: low added latency (the limiter sits on EVERY request's hot path), accurate under distributed/concurrent access, configurable per-client/per-endpoint limits.

The Four Algorithms — Know Every Trade-off

AlgorithmHowWeakness
Fixed Window CounterCount requests in the current fixed window (e.g., 12:00:00–12:01:00); reset each minuteBOUNDARY BURST: 100 requests at 12:00:59 + 100 more at 12:01:00 = 200 requests in 1 second, despite a "100/min" limit
Sliding Window LogStore a TIMESTAMP per request; count timestamps within the last N secondsMemory-heavy — stores every individual request timestamp
Sliding Window CounterWeighted average of the current + previous FIXED window, approximating a true sliding windowAn approximation, not exact — but a good memory/accuracy trade-off
Token BucketA bucket holds up to N tokens, refilling at a fixed rate; each request consumes 1 tokenNone significant — the industry-standard default; naturally allows BURSTS up to the bucket size
🎯 Token Bucket is the default answer, and "allows bursts" is a FEATURE, not a bug. Unlike fixed-window (which has the boundary-burst FLAW), token bucket's burst tolerance is intentional and bounded — the bucket size caps the maximum burst, while the refill rate caps the SUSTAINED rate. This models real traffic well: a client that's been idle can legitimately burst briefly, but can't sustain more than the configured rate. AWS, Stripe, and most production APIs use this approach (or its close cousin, Leaky Bucket, which smooths output instead of allowing bursts).

Token Bucket, Concretely

bucket = { tokens: 10, capacity: 10, refillRate: 1/sec, lastRefill: now }

function allowRequest(bucket) {
  const elapsed = now() - bucket.lastRefill;
  bucket.tokens = Math.min(bucket.capacity, bucket.tokens + elapsed * bucket.refillRate);   // refill
  bucket.lastRefill = now();
  if (bucket.tokens >= 1) { bucket.tokens -= 1; return true; }   // allow, consume a token
  return false;                                                    // reject — 429
}

Where State Lives — the Distributed Systems Part

// A rate limiter is USELESS if each of your 50 stateless API servers keeps its OWN local counter —
// a client could get 50× the intended limit by hitting different servers. State must be SHARED:
API Server 1 ↘
API Server 2 → Redis (shared counter/bucket state, using INCR + EXPIRE, or a Lua script for atomicity)
API Server 3 ↗
// Redis's single-threaded execution + atomic operations (INCR, or a Lua script combining
// check-and-decrement into one atomic step) make it the standard choice — avoids the race condition
// of "check tokens, THEN decrement" being two separate, non-atomic network round-trips.
⚠️ The atomicity trap: check-then-decrement is a race condition. If "read current token count" and "decrement if enough tokens" are TWO separate Redis calls, two concurrent requests can both read "5 tokens left," both decide to proceed, and both decrement — over- admitting requests. The fix: a Lua script (Redis executes it atomically, as one operation) or Redis's built-in atomic primitives combine the check-and-update into a single indivisible step.

Where to Enforce It

// Enforcing at the API GATEWAY/edge (before requests reach application servers) is preferred over
// enforcing in each individual service:
//   - ONE place to configure/audit limits, not N services each implementing their own
//   - Rejects abusive traffic BEFORE it consumes application server resources
//   - Still needs the shared-state (Redis) backing regardless of WHERE enforcement happens

Interview Follow-ups to Expect

• How do you rate-limit per-user vs per-IP vs globally? (different Redis KEYS — user ID, IP address, or a global key — same underlying mechanism, different key granularity)
• What happens if Redis goes down? (fail open — allow requests through, accepting temporary reduced protection — vs fail closed — reject everything — is a real product trade-off to state explicitly)
• How would you rate-limit differently for free vs paid tiers? (different bucket capacity/refill rate per tier, looked up from the client's plan at request time)
