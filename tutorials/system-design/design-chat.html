Design Real-Time Chat Case Study

This case study directly applies the Realtime Apps Deep Dive to a concrete product (like WhatsApp or Slack), adding the specific hard problems chat introduces: message ordering, delivery guarantees (sent/delivered/read receipts), and group chat fan-out.

Requirements

Functional: 1:1 and group messaging, online/offline delivery, message history, read receipts/typing indicators. Non-functional: low latency delivery (sub-second), messages must NOT be lost, must scale to millions of concurrent connections.

Connection Layer — Reusing the Realtime Deep Dive

Client → WebSocket connection → one of many CHAT SERVERS (each holds a shard of connections)
// A CONNECTION REGISTRY (Redis) maps userId → which chat server instance holds their connection.
// Sending a message to a user whose connection lives on a DIFFERENT server: publish to a shared
// message broker (Redis Pub/Sub or Kafka) — the owning server picks it up and delivers via its
// local socket. This is EXACTLY the Realtime Apps lesson's scaling pattern, applied to chat specifically.

Message Flow & Ordering

Sender → Chat Server → 1. PERSIST the message (durable storage — never lose a message)
                      → 2. Look up recipient's connection location (Redis registry)
                      → 3. Deliver via WebSocket if online, OR queue for push notification if offline
// ORDERING within a conversation matters — assign each message a MONOTONICALLY INCREASING sequence
// number PER CONVERSATION (not globally) so clients can detect gaps/reorder if messages arrive
// out of delivery order (common with retries/multiple server hops).
🎯 Persist FIRST, deliver SECOND — never the reverse. If you attempt live delivery before durably storing the message, a crash between "sent over the socket" and "written to storage" LOSES the message with no way to recover it — the receiver saw it (maybe), but it's gone from history. Always write-ahead: durable storage is the source of truth; realtime delivery is a best-effort, retriable notification on top of it.

Delivery Guarantees — Sent / Delivered / Read

SENT:      message accepted by the server, PERSISTED
DELIVERED: message reached the recipient's DEVICE (an ACK sent back over their socket)
READ:      recipient's client marks it as read (their explicit action)
// Each state transition is itself a small message that needs its own reliable delivery — "delivered"
// and "read" receipts flow back to the SENDER the same way the original message flowed to the recipient.
// A recipient OFFLINE when a message arrives gets it queued; on reconnect, the client SYNCS: fetches
// everything since their last-seen message ID/timestamp (the "catch up on reconnect" pattern from the
// Realtime Apps lesson).

Group Chat — the Fan-Out Problem

// A message to a 500-person group needs delivery to up to 500 recipients. Two strategies:
FAN-OUT ON WRITE: sender's message is immediately pushed/queued for ALL 500 recipients at send time.
  → Fast reads (each recipient's message list is pre-built), but a write is EXPENSIVE for huge groups.
FAN-OUT ON READ: store the message ONCE, associated with the group; each recipient's client fetches
  it when they open the conversation.
  → Cheap writes regardless of group size, but reads must aggregate across the group's message log.
// Most systems use fan-out on WRITE for normal-sized groups (fast, and groups are usually <1000
// members) and fall back toward fan-out-on-read-like batching for very large broadcast channels —
// the SAME fan-out trade-off reappears in the Notification System and Instagram Feed case studies.

Interview Follow-ups to Expect

• How do you ensure messages arrive in order despite network retries and multiple server hops? (a per-conversation sequence number; clients buffer and reorder if a gap is detected)
• How would you support end-to-end encryption? (the server stores/relays CIPHERTEXT it can't read; key exchange happens client-to-client — this fundamentally changes what server-side features like search-within-messages can do)
• How do you scale to a group with 100,000 members (a broadcast channel, not a normal group)? (shift toward fan-out-on-read / a pull model — fan-out-on-write at that size is prohibitively expensive)
