Migration Strategies at Scale Senior / Staff

Nearly every case study in this track was framed as "design X from scratch." In reality, staff engineers spend far more time migrating an EXISTING system — a live database, a monolith, a legacy service handling real traffic — without a maintenance window, without data loss, and without users noticing. This is its own distinct skill.

The Strangler Fig Pattern — Migrating a Monolith Incrementally

// Named after a vine that gradually envelops and replaces a host tree. Instead of a risky "big bang"
// rewrite (build the whole new system, flip a switch, hope nothing breaks):
1. Put a PROXY/router in front of the monolith.
2. Build ONE new service that replicates ONE piece of functionality (e.g., just the "user profile" endpoint).
3. Route THAT specific traffic to the new service; everything else still hits the monolith.
4. Repeat — service by service — until the monolith handles NOTHING and can be decommissioned.
// At every step, the system is FULLY FUNCTIONAL — you're never in a broken half-migrated state that
// can't serve traffic. This directly mirrors the CI/CD track's progressive-delivery philosophy
// (ship small, verify, expand) applied to an entire ARCHITECTURE migration instead of a single feature.

Database Migrations Without Downtime — Expand/Contract, at Scale

// The CI/CD and .NET tracks introduced expand/contract for a single schema change. At the SYSTEM
// level, migrating to a NEW DATABASE ENTIRELY follows the same shape, extended:
1. EXPAND:  start DUAL-WRITING — every write goes to BOTH the old and new database
2. BACKFILL: copy existing historical data from old → new (a batch job, done once, carefully)
3. VERIFY:  run READS against BOTH databases, COMPARE results, without serving the new one's answer
            yet — this "shadow read" phase surfaces bugs/discrepancies with ZERO user-facing risk
4. CUTOVER: switch READS to the new database (writes still go to both, as a safety net)
5. CONTRACT: stop writing to the old database; decommission it once confidence is high
// Each phase is independently REVERSIBLE — if step 3 reveals discrepancies, you haven't cut over yet
// and can fix the new system without any user impact.
🎯 The "shadow read" / dual-write verification step is what separates a careful migration from a reckless one. It's tempting to skip straight from backfill to cutover — but comparing old-vs-new answers on REAL production traffic, before the new system's answer actually matters to a single user, is the single highest-leverage risk-reduction step in the entire migration. It catches subtle correctness bugs (a slightly different aggregation, a timezone handling difference) that unit tests on synthetic data would never surface.

Feature Flags as a Migration Tool

// Beyond the CI/CD track's release-toggle use case, flags are the PRIMARY control mechanism for a
// live migration's blast radius:
if (flags.isEnabled("new-recommendation-engine", user)) {
  return newEngine.recommend(user);     // gradually ramp: 1% → 10% → 50% → 100% of users
} else {
  return oldEngine.recommend(user);     // instant rollback: just flip the flag back, no redeploy
}
// This lets you migrate with a LIVE, INSTANT kill-switch at every step — if the new path shows
// elevated errors/latency at 10% rollout, you flip back to 0% in seconds, not by rolling back a deploy.

Interview Follow-ups to Expect

• How would you migrate a payments system's database with ZERO data loss and no downtime? (expand → dual-write → backfill → shadow-read verification → gradual cutover → contract, with idempotent writes throughout)
• What's your rollback plan if the new system shows problems at 50% rollout? (flip the feature flag back toward 0% instantly — no redeploy needed — this is why flags, not deploy-based rollout, control the migration)
• How long do you keep the OLD system around after a "successful" migration? (a defined grace period with monitoring — long enough to catch rare, low-frequency bugs that a short verification window would miss)
