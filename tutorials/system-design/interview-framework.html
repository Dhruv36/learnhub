Interview Framework Senior / Staff

The closing lesson of this track: a repeatable STRUCTURE for a 45-minute system design interview, reusing every earlier lesson as a labeled toolbox entry. The framework itself is simple — the depth comes from everything you've already learned to plug into each step.

The Five-Phase Structure (roughly 45 minutes)

1. REQUIREMENTS CLARIFICATION      (~5 min) — do NOT skip this
2. BACK-OF-ENVELOPE ESTIMATION      (~5 min)
3. HIGH-LEVEL DESIGN                (~10 min)
4. DEEP DIVE (interviewer-directed) (~15-20 min)
5. WRAP-UP: bottlenecks & trade-offs (~5 min)

Phase 1: Requirements — the Most-Skipped, Most-Costly Step

// Ask BEFORE designing anything:
Functional:      What EXACTLY does the system need to do? (Get 2-3 core use cases, not every feature)
Non-functional:  Read-heavy or write-heavy? Consistency needs? Latency requirements? Scale (users, QPS,
                 data volume)?
Out of scope:    Explicitly agree what you're NOT designing (e.g., "I'll assume auth is handled
                 elsewhere unless you want me to design it")
// Jumping straight to "I'll use microservices with Kafka and Redis" WITHOUT this step is the #1
// reason strong engineers do poorly in these interviews — you can't make correct trade-offs (the
// entire point of this track) without first knowing what you're trading FOR.
🎯 Every case study in this track implicitly did Phase 1 first. Look back: TinyURL started with "highly available, low-latency REDIRECTS" (read-heavy!) before any architecture. Amazon started with "strong consistency for inventory, high availability for browsing" — two DIFFERENT requirements for different parts of the SAME system. Skipping this step means you're designing against requirements you invented yourself, not the ones actually being asked for.

Phase 2: Estimation — Sets the Scale of Every Later Decision

// Reuses the Back-of-Envelope Estimation lesson directly: QPS, storage, bandwidth.
// The NUMBER matters less than what it IMPLIES: "3TB total data" fits comfortably on one well-chosen
// database. "3PB" means sharding is non-negotiable from the start. State the number, then state its
// ARCHITECTURAL CONSEQUENCE — that's the actual signal, not the arithmetic itself.

Phase 3: High-Level Design — Boxes and Arrows, Justified

// Sketch: client → load balancer → API layer → [services] → data stores → caches
// For EACH box, be ready to say WHY it's there and what it's replacing (a monolith would also
// "work" — why THIS decomposition?). Don't over-design upfront — start SIMPLE (the Microservices
// lesson's "don't default to microservices" principle) and let the deep-dive phase justify adding
// complexity where the requirements actually demand it.

Phase 4: Deep Dive — Where the Track's Content Actually Gets Used

// The interviewer will steer here — toward whatever they want to probe. This is where you reach
// into the toolbox built across this entire track:
"How do you scale the database?"        → Sharding, Consistent Hashing, Replication lessons
"How do you keep it available during
 a partition?"                           → CAP Theorem, Distributed Consensus lessons
"How do you make writes fast?"           → Caching, async processing (Pub/Sub) lessons
"What if this component fails?"          → Heartbeat/Gossip, Retry/Circuit-Breaker, Multi-Region/DR
"How do you avoid overselling/
 double-booking?"                        → Concurrency Control, atomic operations (Amazon case study)
// The DEPTH of your earlier lessons is what makes this phase strong — a memorized architecture
// diagram has nothing left to say once the interviewer asks "but why?"

Phase 5: Wrap-Up — Name What You'd Improve

// Proactively identify: "the [component] is a bottleneck at [scale] because [reason] — I'd address
// it by [specific fix]." This demonstrates you're not just satisfied with "it works" — you can see
// past your own design's edges, which is precisely the Trade-off Driven Design lesson's meta-skill,
// applied one more time as a closing move.

The Meta-Lesson of This Entire Track

Every case study — TinyURL, Rate Limiter, Chat, Amazon, Instagram — was built from the SAME small set of Fundamentals (caching, consistent hashing, replication, CAP trade-offs, pub/sub) and DEEP DIVE tools (concurrency control, auth, observability), recombined differently based on each system's specific requirements. That recombination — not memorizing 12 specific architectures — is the actual, transferable skill. A NEW case study you've never seen before is solvable with the same toolbox.

Interview Follow-ups to Expect

• "We're almost out of time — what would you do differently with another 30 minutes?" (name specific unaddressed areas: haven't discussed monitoring, haven't stress-tested the data model for a specific edge case, haven't discussed the migration path from an existing system)
• "What's the weakest part of your design?" (a strong candidate volunteers this honestly and specifically — refusing to identify a weakness reads as either dishonesty or lack of critical self-review, both worse signals than a named, understood limitation)
• "How would this change if [some requirement flips]?" (this is Phase 1's importance resurfacing — re-derive the affected trade-offs live, exactly as covered in the Trade-off Driven Design lesson)
