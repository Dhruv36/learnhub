Design Key-Value Store (Dynamo) Case Study

This is the "synthesis" case study — it pulls together Consistent Hashing, CAP theorem trade-offs, and Replication into one coherent design, following the architecture of Amazon's DynamoDB paper. If you understand this case study, you understand how the Fundamentals section's lessons compose into a real system.

Requirements

Functional: put(key, value), get(key) — simple API. Non-functional: HIGH AVAILABILITY (Amazon's original motivation: a shopping cart must accept writes even during a partition — "always writable"), horizontal scalability, tunable consistency.

Data Partitioning — Consistent Hashing

// Directly reuses the Consistent Hashing lesson: keys and nodes both hash onto a RING. A key is
// stored on the first node clockwise from its hash position. Adding/removing a node only
// remaps the keys in ITS immediate neighborhood — not the whole keyspace.
// VIRTUAL NODES (each physical node maps to many points on the ring) solve uneven load distribution —
// without them, a physical node could randomly own a disproportionate arc of the ring.

Replication — N, W, R (the Quorum Model)

N = replication factor (e.g., 3) — each key is stored on 3 nodes (the next N-1 clockwise from its
    primary position on the ring)
W = write quorum — a write is considered SUCCESSFUL once W replicas acknowledge it
R = read quorum  — a read queries R replicas and returns the most recent value (via version/timestamp)

// The tunable consistency guarantee:  W + R > N  ⟹  every read overlaps with the most recent write
// (at least one node in the read set MUST have seen the latest write) — STRONG consistency.
// W + R ≤ N ⟹ reads might miss the latest write — EVENTUAL consistency, but higher availability/lower latency.
Example: N=3, W=2, R=2 → W+R=4 > N=3 → strong consistency, tolerates 1 node being slow/down per operation
Example: N=3, W=1, R=1 → fastest, most available, but a read can return STALE data
🎯 This IS the CAP theorem, made concrete and TUNABLE per-operation. Rather than a single global CP-or-AP choice, Dynamo lets you dial the trade-off per read/write via N/W/R — a critical-path write (e.g., "add to cart") might use W=1 for low latency, while a security-sensitive read might use R=N for maximum consistency. Being able to state "we'd tune W and R differently per operation, here's why" is a strong senior-level signal.

Conflict Resolution — Vector Clocks

// With MULTIPLE writers (no single leader) and network partitions, concurrent writes to the SAME
// key on DIFFERENT replicas can genuinely CONFLICT — which one is "correct"? A simple timestamp
// (last-write-wins) can silently lose data. Dynamo uses VECTOR CLOCKS to detect true conflicts:
key: "cart123", value: [milk, eggs], vector clock: {nodeA: 1}
  → nodeA writes "+bread" → {nodeA: 2}
  → MEANWHILE nodeB (network partition) writes "+butter" from the SAME starting version → {nodeB: 1}
// Neither vector clock "descends from" the other — this is a GENUINE CONFLICT, not just a race won by
// one side. The system returns BOTH versions to the client/application, which resolves it (e.g., "merge
// cart contents" — a shopping cart naturally supports simple union-merge semantics).

Handling Node Failures — Hinted Handoff & Anti-Entropy

HINTED HANDOFF: if a replica is temporarily down during a write, another node stores the write
  as a "hint" and delivers it to the original replica once it recovers — maintains availability
  without permanently losing the write.
ANTI-ENTROPY (Merkle trees — see the Bloom/Merkle lesson): periodically compare replicas' data using
  Merkle tree root hashes to detect and repair DIVERGENCE (missed writes, corruption) — without
  comparing every key individually.

Interview Follow-ups to Expect

• How do you handle a "hot key" that far exceeds any single node's capacity? (further partition/shard that specific key's data, or cache it aggressively in front of the store)
• Why choose vector clocks over simple last-write-wins? (LWW silently discards concurrent writes; vector clocks detect true conflicts and let the application resolve them meaningfully — appropriate when data loss is unacceptable)
• How does this design embody the CAP theorem's AP choice? (during a partition, EVERY reachable node still accepts writes — availability over strict consistency — reconciled later via vector clocks + anti-entropy)
