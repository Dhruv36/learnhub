Authentication (JWT, OAuth2, OIDC) Deep Dive

At the systems-design level, authentication is a SCALABILITY question, not just a security one: sessions vs tokens is fundamentally a choice between server-side state (simple, doesn't scale horizontally without shared storage) and client-held state (stateless, scales trivially, but revocation gets hard). This lesson focuses on that architectural trade-off.

Session Cookies vs Tokens — the Core Scaling Trade-off

SESSION-BASED: server creates a session, stores it (in memory or a shared store like Redis),
  gives the client a session ID cookie. Every request: look up the session ID → fetch session data.
  → Simple, INSTANT revocation (just delete the session).
  → Requires SHARED session storage across all server instances (or sticky sessions — fragile).

TOKEN-BASED (JWT): server signs a token CONTAINING the claims (user ID, roles, expiry). Client sends
  it with every request. Server VALIDATES the signature — no lookup needed.
  → STATELESS — any server instance validates independently, no shared storage needed. Scales trivially.
  → Revocation is HARD — a signed token is valid until it EXPIRES, no matter what. (Mitigate with
    short expiries + refresh tokens, or a token blocklist — which reintroduces shared state anyway.)
🎯 The revocation problem is the most-tested systems-design nuance here. "How do you log a user out immediately with JWTs?" is a favorite follow-up specifically because the naive answer ("just don't send the token anymore") is wrong — a STOLEN token remains valid until expiry regardless of client-side logout. Real answers: keep JWT expiry SHORT (minutes) paired with a longer-lived refresh token the server CAN revoke, or maintain a (shared, defeating some statelessness) blocklist of revoked token IDs checked on each request for high-security paths.

OAuth 2.0 — Delegated Authorization

// "Let App X access my Google Calendar WITHOUT giving X my Google password."
1. App X redirects user to Google's authorization endpoint
2. User logs in AT GOOGLE, approves the requested SCOPE ("read your calendar")
3. Google redirects back to X with an AUTHORIZATION CODE
4. X exchanges the code (server-side, with its own client secret) for an ACCESS TOKEN
5. X calls Google's API with the access token — scoped to EXACTLY what was approved
// X never sees the user's Google password. The access token is scoped and revocable independently.

OIDC — OAuth2 + a Standardized Identity Layer

// OAuth2 alone proves "this app has PERMISSION to access a resource" — it does NOT reliably prove
// WHO the user is (an access token isn't standardized to carry verified identity claims).
// OpenID Connect adds an ID TOKEN (a specific JWT) alongside the access token, carrying VERIFIED
// identity claims (user ID, email, when authenticated) — this is what "Sign in with Google" uses.
// Rule: OAuth2 for delegated ACCESS; OIDC for AUTHENTICATION (proving identity).

Designing Auth for a Multi-Service Architecture

// A common systems-design ask: "design the auth layer for a microservices system."
Client → API Gateway (validates JWT ONCE, at the edge)
       → forwards request + verified claims to internal services (via a header, or a re-signed
         internal token) — internal services TRUST the gateway, don't re-validate against an
         external identity provider on every call (too slow, too much coupling)
// Centralizing validation at the edge avoids every internal service independently needing IdP
// connectivity, and keeps the (relatively expensive) signature verification to ONE hop.

Interview Follow-ups to Expect

• Sessions or JWTs for a system scaling to millions of users across many server instances? (JWTs — statelessness avoids the shared-session-store bottleneck, at the cost of harder revocation)
• How do you handle token refresh without forcing re-login? (a long-lived refresh token, stored securely, exchanged for a new short-lived access token — refresh tokens ARE revocable server-side)
• Where do you validate auth in a microservices architecture? (at the API gateway/edge, once, then propagate trusted claims internally)
