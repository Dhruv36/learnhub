How to Scale a System Fundamentals

Every system at Google/Amazon scale started as one server. The scaling journey below is the backbone of every design interview — and of every real product's growth. Learn the stages, and more importantly, learn what problem forces each stage.

Stage 0 → 1: Single Server → Split the Database

[Client] → [Web server + DB on one box]        // Stage 0: fine until traffic grows
[Client] → [Web server] → [DB server]          // Stage 1: web & data tiers scale independently

Forcing problem: CPU contention — app requests and DB queries fight for the same machine.

Stage 2: Horizontal Scaling + Load Balancer

[Client] → [Load Balancer] → [Web 1..N (stateless)] → [DB]

Forcing problem: one web server maxes out. The key prerequisite: servers must be stateless — session state moves to a shared store (Redis) so any server can handle any request. Vertical scaling (bigger machine) is the alternative, but it has a hard ceiling and no fault tolerance.

Stage 3: Database Replication

              ┌→ [Replica 1]  (reads)
[Primary DB] ─┼→ [Replica 2]  (reads)
 (writes)     └→ [Replica 3]  (reads)

Forcing problem: most workloads are read-heavy (often 100:1). Replicas absorb reads and double as failover targets. New problem introduced: replication lag — a user may not see their own write. (Fix: read-your-own-writes routing; details in the Replication lesson.)

Stage 4: Caching + CDN

[Client] → [CDN (static assets)]
         → [LB] → [Web] → [Redis cache] → [DB]   // cache hit = no DB touch

Forcing problem: replicas still recompute the same hot queries. A cache serves them from memory (~100× faster). CDN pushes static content to the network edge, near users.

Stage 5: Async Processing (Message Queues)

[Web] → [Queue (Kafka/SQS)] → [Workers] → [DB/Email/Reports]

Forcing problem: slow operations (emails, video encoding, report generation) block user requests. Queues decouple: respond to the user immediately, process in the background. Queues also absorb traffic spikes — workers drain at their own pace.

Stage 6: Database Sharding

[user_id % 4 == 0] → [Shard 0]     // horizontal partitioning:
[user_id % 4 == 1] → [Shard 1]     // each shard holds a slice of the data
...

Forcing problem: the write volume or data size exceeds any single primary. Costs you accept: cross-shard joins and transactions become hard, resharding is painful (see Consistent Hashing), and hot keys can overload one shard.

Stage 7: Multi-Region

Serve users from the nearest region; survive a whole-region outage. Brings the hardest problems: cross-region replication, data residency laws, and the consistency trade-offs of the CAP theorem.

The Complete Picture

Client → DNS → CDN
              → LB → Web/API tier (stateless, autoscaled)
                       ├── Redis (cache + sessions)
                       ├── Queue → async workers
                       └── DB: sharded primaries + replicas
              (× every region, with cross-region replication)
💡 Staff-level insight: in interviews, never jump to Stage 7. State the scale first (QPS, data size — next lesson), then apply only the stages that scale demands. Over-engineering signals mid-level; matching architecture to actual load signals senior+.

Interview Follow-ups to Expect

• How do you keep sessions working when a server dies? (external session store)
• Your cache goes down — what happens? (thundering herd → see Caching lesson)
• How do you pick a shard key, and what happens if you pick wrong? (hot partitions, resharding)
