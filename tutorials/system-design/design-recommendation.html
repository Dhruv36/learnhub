Design Recommendation System Case Study

Unlike most case studies, this one is less about databases and more about a PIPELINE architecture: an offline, batch-computed heavy-lifting stage feeding a fast, online, low-latency serving stage. This split is THE core systems-design idea to demonstrate here — not the ML algorithm details.

Requirements

Functional: given a user, return a ranked list of recommended items (products, videos, posts). Non-functional: LOW LATENCY at serve time (recommendations render inline with a page load, budget: tens of milliseconds), fresh enough to reflect recent behavior, scale to millions of users/items.

The Two-Stage Architecture — the Core Idea

OFFLINE (batch, runs periodically — hourly/daily, NOT on the request path):
  Collect user interaction data (clicks, purchases, watch time) →
  Train/update a model (collaborative filtering, embeddings) →
  PRECOMPUTE candidate recommendations per user, or per-item SIMILARITY scores →
  Write results to a fast-read STORE (the online stage reads from here)

ONLINE (request path, must be FAST — tens of milliseconds):
  User requests a page → fetch PRECOMPUTED candidates for this user (a fast key-value lookup,
  NOT live model inference) → apply lightweight RE-RANKING (recency, current context, A/B test variant)
  → return top N
🎯 Why split this way: the EXPENSIVE work can't happen on the request path. Training a model or computing similarity across millions of items is far too slow for a user-facing request budget (tens of ms). By doing the heavy computation OFFLINE and just READING precomputed results online, you decouple "how sophisticated can the model be" from "how fast must the response be" — you can run an arbitrarily complex offline job without it EVER touching request latency.

Candidate Generation vs Ranking (two distinct sub-problems)

CANDIDATE GENERATION: narrow millions of items down to a few hundred PLAUSIBLE candidates, fast.
  (e.g., collaborative filtering: "users similar to you liked these"; or content-based: "similar to
  items you've engaged with")
RANKING: take those few hundred candidates and score/order them PRECISELY, using a heavier model
  that's now only evaluating a SMALL set — feasible at request time because the set is small.
// This funnel (broad-and-cheap → narrow-and-precise) is a recurring systems-design pattern anywhere
// you need both SCALE and PRECISION — the same idea appears in the Search Engine case study's
// retrieval-then-ranking split.

The Cold-Start Problem

ScenarioFallback strategy
New USER, no historyPopular/trending items; ask onboarding preference questions; demographic-based defaults
New ITEM, no interactions yetContent-based similarity to existing items (metadata/embeddings), not purely collaborative signals

Cold start is a near-guaranteed follow-up because it's a real, unavoidable gap in any pure behavior-based approach — always have a fallback path for "we have no data yet."

Freshness — Balancing Batch and Real-Time

// Pure offline batch (daily) misses "user just watched 5 cooking videos in a row" — recommendations
// feel stale/unresponsive. Hybrid approach:
//   - Heavy model retraining: daily/hourly BATCH (offline)
//   - Lightweight signal incorporation: a STREAMING pipeline (Kafka) updates a "recent activity" feature
//     that the online re-ranking stage factors in — cheap to compute, immediately responsive
// This mirrors the Data Platforms lesson's batch-vs-streaming trade-off, applied specifically to
// keeping recommendations feeling responsive without re-running the expensive model constantly.

Interview Follow-ups to Expect

• How do you evaluate whether a recommendation model is actually GOOD? (offline metrics like precision/recall against held-out data, but ultimately A/B TESTING against a real business metric — click-through rate, conversion, time spent)
• How do you avoid a filter bubble (only ever recommending very similar content)? (deliberately inject some diversity/exploration into the candidate set, not pure exploitation of known preferences)
• How would you serve recommendations for a brand-new item with zero interaction data? (content-based fallback using item metadata/embeddings until enough interaction data accumulates)
