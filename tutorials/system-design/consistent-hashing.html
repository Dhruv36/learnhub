Consistent Hashing Fundamentals

The problem: distribute keys across N servers with hash(key) % N, and the moment N changes (a server dies, you add capacity), almost every key remaps — a full cache wipe or massive data reshuffle. Consistent hashing remaps only ~1/N of keys instead.

The Ring

Hash space arranged as a circle (0 … 2³²−1):

        server A (hash=50)
       /                  \
  key k1 (30) → A      server B (hash=160)
      |                     |
  key k2 (200) → C      key k3 (100) → B
       \                  /
        server C (hash=280)

Rule: each key belongs to the FIRST server clockwise from its hash.

Add server D at 120: only keys in (50, 120] move (from B to D). Every other key is untouched. Remove B: only B's keys move, to the next server clockwise — its successor.

Virtual Nodes (the part interviews actually probe)

With few servers, the ring is lumpy — one server may own half the circle. Fix: place each physical server at 100–200 positions ("vnodes") using hash(server_id + i).

Without vnodes: A owns 45%, B owns 40%, C owns 15%   → hot server
With 150 vnodes each: every server owns ≈ 33% ± 2%    → smooth
Bonus: a beefier machine simply gets MORE vnodes (weighted capacity).

Where You'll Meet It in Production

SystemUsage
DynamoDB / CassandraPartitioning data across the cluster ring
Memcached / Redis clientsClient-side key → cache-server mapping (Ketama)
Load balancers (Envoy, nginx)Sticky routing that survives backend changes
CDNs (Akamai — where it was invented)Mapping content to edge caches
💡 Staff-level insight: consistent hashing balances keys, not load. One celebrity user (hot key) can still melt its server. Real fixes: split the hot key (append a random suffix, fan-in on read), dedicated cache tier for hot items, or bounded-load consistent hashing (each node accepts ≤ ⌈avg × 1.25⌉, spills overflow clockwise — used by Vimeo/Google).

Interview Follow-ups to Expect

• Lookup complexity? (O(log V) binary search over sorted vnode positions)
• What happens to the data when a node joins — who sends it? (successor streams the moved range; during Dynamo-style handoff, reads use quorums to stay correct)
• Alternatives? (Rendezvous/HRW hashing — simpler, no ring; jump hash — O(1), but only append/remove-last)
