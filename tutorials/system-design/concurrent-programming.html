Concurrent Programming & Deadlocks Deep Dive

This lesson zooms in from distributed-systems concurrency (the previous ACID/isolation lesson) to WITHIN-a-single-process concurrency — the race conditions, deadlocks, and coordination primitives that show up whenever multiple threads share memory, which underlies how every server you design actually executes.

Race Conditions — the Core Problem

// Two threads both run: counter++
// counter++ is NOT atomic — it's really THREE steps: READ counter, ADD 1, WRITE counter back.
Thread A: reads counter (=5)
Thread B: reads counter (=5)          ← both read the SAME value before either writes
Thread A: writes counter = 6
Thread B: writes counter = 6           ← B's write OVERWRITES A's increment — one increment LOST
// Expected result: 7. Actual result: 6. This is a race condition — the outcome depends on
// unpredictable THREAD TIMING/INTERLEAVING, not just the code's logic.

Locks — the Basic Fix, and Their Cost

lock.acquire()
counter++              // only ONE thread can be here at a time — the "critical section"
lock.release()
// Correct, but locks serialize access — under high contention, threads spend more time WAITING
// for the lock than doing useful work. This is why lock-free/atomic primitives exist for hot paths:
atomicCounter.incrementAndGet()   // uses CPU-level atomic instructions (compare-and-swap) — no lock needed

Deadlock — the Four Necessary Conditions

Thread A: locks Resource 1, then tries to lock Resource 2 (held by B) → A WAITS
Thread B: locks Resource 2, then tries to lock Resource 1 (held by A) → B WAITS
// Neither can proceed. Deadlock requires ALL FOUR conditions simultaneously:
1. MUTUAL EXCLUSION    — resources can't be shared (only one thread holds a lock at a time)
2. HOLD AND WAIT        — a thread holds one resource while waiting for another
3. NO PREEMPTION        — a resource can't be forcibly taken from a thread holding it
4. CIRCULAR WAIT        — a cycle of threads, each waiting on the next
// Break ANY ONE condition and deadlock becomes impossible — this is the basis for every prevention
// strategy below.

Deadlock Prevention Strategies

StrategyBreaks which condition
Always acquire locks in a CONSISTENT GLOBAL ORDER (e.g., always lock Resource 1 before Resource 2, everywhere in the codebase)Circular wait — no cycle can form if everyone acquires in the same order
Lock timeouts (try-lock with a timeout, back off and retry on failure)Hold and wait — a thread gives up rather than waiting forever
Acquire ALL needed locks at once, or none (all-or-nothing)Hold and wait — never hold one while blocked on another
🎯 "Lock ordering" is the single most practical, most-expected answer in interviews. It's simple to state, simple to enforce (a code-review rule or a lint check: "always lock accounts in ID order when transferring funds between two accounts"), and directly breaks the circular-wait condition — the one condition that's usually easiest to eliminate by convention without complex runtime machinery.

Higher-Level Concurrency Primitives

PrimitiveUse
MutexExclusive access to a shared resource — one thread at a time
SemaphoreAllow up to N concurrent threads (e.g., limit concurrent DB connections)
Read-Write LockMany concurrent READERS, but writers get exclusive access — good when reads vastly outnumber writes
Compare-And-Swap (CAS)Lock-free atomic "update only if the value hasn't changed" — the hardware primitive behind lock-free data structures and optimistic concurrency control

Interview Follow-ups to Expect

• How would you prevent a deadlock in code that transfers money between two accounts? (always acquire locks on accounts in a CONSISTENT order — e.g., by account ID — regardless of transfer direction)
• When would you use a semaphore instead of a mutex? (limiting concurrency to N, not just 1 — e.g., capping concurrent outbound API calls to avoid overwhelming a downstream service)
• What's the difference between a race condition and a deadlock? (a race condition produces WRONG results from unsynchronized access; a deadlock produces NO progress at all — threads permanently blocked)
