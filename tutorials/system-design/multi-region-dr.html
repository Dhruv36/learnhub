Multi-Region & Disaster Recovery Senior / Staff

Everything so far assumed a single datacenter/region. At staff level, "what if the whole region goes down" is an expected question — AWS us-east-1 has had multi-hour outages that took down huge swaths of the internet. This lesson covers the vocabulary and architecture patterns for surviving that.

RTO and RPO — the Two Numbers That Define "How Bad"

RPO (Recovery Point Objective): how much DATA can you afford to LOSE, measured in time?
  "RPO = 5 minutes" means: after a disaster, you might lose up to the last 5 minutes of writes.
RTO (Recovery Time Objective): how long can the system be DOWN before it's back up?
  "RTO = 1 hour" means: you have up to 1 hour to restore service after a disaster.
// These are BUSINESS decisions, not technical ones — a startup's internal tool might accept RPO=1hr,
// RTO=1day; a payments system might require RPO=seconds, RTO=minutes. The ARCHITECTURE is chosen to
// hit these targets — you don't pick "multi-region active-active" because it's impressive, you pick
// it because the stated RPO/RTO REQUIRE it, and anything less would be under-engineering, and
// anything more would be wasted cost.

DR Strategies — a Spectrum of Cost vs Recovery Speed

StrategyDescriptionRTO/RPOCost
Backup & RestorePeriodic backups to another region, restored on disasterHours (RTO), hours (RPO)Lowest — nearly idle standby region
Pilot LightMinimal core infrastructure always running in the DR region (e.g., a replicated database, no compute), scaled up on disasterTens of minutesLow-medium
Warm StandbyA smaller-scale but FULLY FUNCTIONAL replica running in the DR region continuouslyMinutesMedium-high
Multi-Region Active-ActiveBOTH regions serve live production traffic simultaneously, all the timeNear-zero (seconds)Highest — full duplicate capacity, plus cross-region consistency complexity
🎯 Active-active isn't "the best" — it's the most expensive AND the most complex. Running two regions BOTH live means solving cross-region data consistency continuously (not just during a disaster) — this is the CAP theorem's trade-off playing out at planetary scale, permanently, not just during an outage. Most systems don't need active-active; the interview-mature answer picks the CHEAPEST strategy that still meets the stated RTO/RPO, not reflexively the most resilient-sounding option.

Data Replication Across Regions

// Cross-region replication has fundamentally higher LATENCY than within a datacenter (tens to
// hundreds of ms vs sub-millisecond) — this shapes the consistency model choice:
SYNCHRONOUS cross-region replication: strong consistency, but every write pays cross-region latency
  — often UNACCEPTABLE for latency-sensitive paths.
ASYNCHRONOUS cross-region replication: fast local writes, but a REGION FAILURE can lose the most
  recent (unreplicated) writes — this directly SETS your RPO.
// Global databases (Spanner, CockroachDB) use consensus (the Distributed Consensus lesson's Raft,
// per-shard) across regions to get strong consistency WITHOUT a single point of failure — at the
// cost of higher write latency (a write must reach a quorum, which may span regions).

Failover — Detecting and Redirecting Traffic

// DNS-based failover: health checks monitor the primary region; on failure, DNS records update to
// point to the DR region. SIMPLE, but DNS caching/TTLs mean failover isn't instant (clients/resolvers
// may cache the old IP for minutes).
// Global load balancer / Anycast: traffic is routed at the NETWORK layer, enabling faster failover
// than DNS propagation — but more complex/expensive infrastructure.
// The failover mechanism ITSELF needs a runbook and (ideally) automated triggers — a disaster
// recovery plan nobody has ever TESTED is not a reliable plan (game days / chaos engineering exist
// specifically to validate DR actually works before you need it for real).

Interview Follow-ups to Expect

• How would you choose between pilot light and warm standby for a specific system? (map the stated RTO/RPO requirement to the cheapest strategy that satisfies it — don't default to the most expensive option)
• What happens to in-flight requests during a region failover? (they typically fail/time out and must be retried by the client against the new region — idempotency, from the REST/API-Contracts lesson, matters enormously here)
• How do you know your DR plan actually works? (regularly TEST it — scheduled failover drills/chaos engineering — an untested DR plan is a liability, not a safety net)
