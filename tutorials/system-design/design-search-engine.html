Design Search Engine Case Study

This case study is the direct, large-scale application of the Bloom/Merkle/Inverted-Index Fundamentals lesson — the inverted index IS the search engine's core data structure. The system-design work is building the PIPELINE around it: crawling, indexing at scale, and ranking.

Requirements

Functional: crawl/discover web pages, index their content, answer keyword queries ranked by relevance. Non-functional: index BILLIONS of documents, query latency under ~200ms, freshness (new/updated pages should appear in results reasonably soon).

The Pipeline — Three Stages

1. CRAWLING: bots fetch pages, follow links, discover new pages. Must respect robots.txt, avoid
   re-crawling unchanged pages too often (use ETags/Last-Modified), and DEDUPLICATE (many URLs point
   to identical or near-identical content).

2. INDEXING: parse crawled pages → build/update the INVERTED INDEX (word → [documents containing it],
   with positions for phrase search and term frequency for ranking — the Bloom/Merkle lesson's structure).
   At this SCALE, the index itself is SHARDED across many machines (no single machine holds "the
   inverted index" — each shard holds a portion of the document space).

3. QUERY SERVING: parse the user's query → look up each term in the (sharded) inverted index →
   INTERSECT/merge results across terms → RANK → return top N.

Ranking — Beyond "Contains the Word"

// A simple inverted-index lookup tells you WHICH documents contain a term — not which are most
// RELEVANT. Ranking signals, roughly in order of classic importance:
TF-IDF: Term Frequency (how often the term appears in THIS doc) × Inverse Document Frequency
        (how RARE the term is across ALL documents — common words like "the" contribute little signal)
PageRank (or similar): a document's IMPORTANCE based on how many OTHER important documents link to it
        — a recursive "importance flows through links" definition, computed offline via iterative
        graph computation across the entire web graph.
Freshness, personalization, click-through signals: modern engines layer many more ranking factors
        on top — this is the same "many-signal ranking" theme as the Recommendation System case study.
🎯 The retrieval-then-ranking funnel reappears here — recognize the pattern. A query first RETRIEVES a broad candidate set (all documents containing the query terms — fast, via the inverted index) and THEN RANKS that candidate set with a more expensive scoring function. You never run expensive ranking against the ENTIRE corpus — only against the (much smaller) retrieved candidates. This is the exact same broad-and-cheap → narrow-and-precise funnel from the Recommendation System case study, appearing again because it's the general answer to "how do you apply an expensive scoring function at massive scale."

Sharding the Index — Term-Partitioned vs Document-Partitioned

StrategyHowTrade-off
Document-partitionedEach shard holds a SUBSET of documents, with its OWN full inverted index over just those docsA single query hits EVERY shard (scatter-gather) and merges results — simpler to add documents, but every query touches all shards
Term-partitionedEach shard holds specific TERMS' full posting lists, across ALL documentsA query might touch only a FEW shards (just the ones holding its query terms) — but a common term's posting list can become a huge, unbalanced shard

Document-partitioning is the more common real-world choice (Elasticsearch's default) — the load-balancing is more predictable, even though every query technically fans out to all shards.

Interview Follow-ups to Expect

• How do you keep the index fresh without re-crawling everything constantly? (incremental crawling using ETags/Last-Modified to skip unchanged pages; prioritize re-crawling pages that historically change often)
• How do you handle a query like "New York" vs "new" AND "york" separately? (phrase search using stored term POSITIONS in the inverted index — not just presence, but adjacency)
• Why is document-partitioning generally preferred over term-partitioning at scale? (more predictable, even load distribution — a term-partitioned shard for a very common word becomes a hot spot)
