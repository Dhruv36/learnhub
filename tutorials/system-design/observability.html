Observability Engineering Deep Dive

A system you can't observe is a system you can't operate — this closing Deep Dive is what senior/staff interviews expect you to volunteer even when not explicitly asked: "how would you know if this design is actually working correctly in production?"

The Three Pillars

PillarAnswersTooling
MetricsAggregated numbers over time — "what's the p99 latency, error rate, QPS right now"Prometheus, CloudWatch, Datadog
LogsDiscrete events — "what exactly happened at 3:42:17, for this one request"ELK stack, Splunk, CloudWatch Logs
TracesThe full journey of ONE request across multiple services — "where did the time go, which service failed"Jaeger, Zipkin, OpenTelemetry
🎯 Know WHICH pillar answers which question — this is the actual interview test. "Is the system healthy right now, in aggregate?" → metrics. "Why did THIS SPECIFIC request fail?" → logs (or better, structured logs correlated by a request/trace ID). "Which of the five microservices this request touched was slow?" → distributed tracing. Reaching for the wrong pillar (grepping logs across ten services to find a performance bottleneck a trace would show in one view) is a real, common production-debugging inefficiency.

SLIs, SLOs, and Error Budgets

SLI (Service Level Indicator):  the actual MEASUREMENT — "99.95% of requests succeeded last 30 days"
SLO (Service Level Objective):   the TARGET — "we aim for 99.9% success rate"
SLA (Service Level Agreement):   the CONTRACTUAL promise to customers (often looser than the internal SLO,
                                  with penalties for violation)

ERROR BUDGET = 1 − SLO. A 99.9% SLO allows 0.1% failures — that 0.1% is a BUDGET to spend on:
  risky deploys, planned maintenance, accepting some failures rather than over-engineering for perfection.
  Burning through the budget too fast → freeze risky changes until it recovers.
// This reframes reliability as a QUANTIFIED, spendable resource rather than a vague "be more careful" —
// it's how Google's SRE model balances reliability against shipping velocity.

Alerting — On Symptoms, Not Causes

// ❌ Alert on a specific CAUSE: "CPU > 90% on server-7" — too noisy (transient spikes are often
// harmless) and too narrow (misses problems that DON'T manifest as high CPU).
// ✅ Alert on a user-facing SYMPTOM: "error rate > 1%" or "p99 latency > 2s" — this is what users
// actually experience, and it's agnostic to WHICH underlying cause is responsible this time.
// The root-cause investigation (is it CPU? a slow query? a bad deploy?) happens AFTER the alert fires,
// using metrics/logs/traces — the alert itself should fire on USER IMPACT.

Correlation IDs — Tying It All Together

// A single request touching 5 microservices generates 5 separate sets of logs. Without a shared
// identifier, correlating them into "what happened for THIS request" is nearly impossible at scale.
Client request → Gateway generates a requestId (or reuses an incoming trace ID)
              → propagated via a header (e.g., X-Request-ID) to EVERY downstream service call
              → every log line, every span in a trace, includes this SAME ID
// This is what makes distributed tracing (and correlating logs across services) actually work —
// it's infrastructure you have to deliberately build in, not something you get for free.

Interview Follow-ups to Expect

• How would you debug a slow request that touches 6 microservices? (distributed tracing with correlation IDs — see exactly which hop introduced the latency)
• What would you alert on for a new API endpoint? (symptom-based: error rate and latency percentiles, not a specific infrastructure metric)
• How do error budgets change how a team ships? (a healthy budget = ship faster/take more risks; a nearly-exhausted budget = freeze risky changes, focus on reliability)
