Design TinyURL Case Study

The classic "warm-up" system design question — deceptively simple, but a great vehicle for demonstrating estimation, encoding schemes, and the read-heavy caching story. The interview signal isn't "can you build a URL shortener," it's "do you reach for the RIGHT tools at each step."

Requirements

Functional: shorten a long URL → short code; redirect short code → original URL (with a 301/302). Non-functional: highly available, low-latency redirects (this is the READ path — it must be fast), uniqueness of short codes, URLs don't collide.

Back-of-Envelope Estimation

Assume: 100M new URLs/month, 100:1 read:write ratio (redirects vastly outnumber shortenings)
Writes: 100M / (30×86400) ≈ 40 URLs/sec
Reads:  40 × 100 = 4,000 redirects/sec
Storage: 100M/month × 5 years × ~500 bytes/record ≈ 6TB — small enough for a single well-chosen DB,
         but the READ traffic is what shapes the architecture (caching, not storage, is the challenge)

The Core Design Decision: How to Generate Short Codes

ApproachHowTrade-off
Hash the URL (MD5/SHA, take first 7 chars)hash(longURL).substring(0,7)Collisions possible (must check/retry); same URL always → same code (dedup for free, or a problem if you WANT different codes per submission)
Base62 encode an auto-incrementing IDcounter=125 → base62 encode → "cb"NO collisions by construction; needs a globally unique counter (the Unique ID Generator case study's problem, at smaller scale)
Pre-generated key poolA background service pre-generates and stores unused short codes; the write path just claims oneFastest write path (no encoding work inline); extra infrastructure (a key-generation service + storage)
🎯 Base62 encoding of an auto-increment ID is the standard "correct" answer — know WHY. It guarantees uniqueness without a collision-check round-trip (unlike hashing), and a 7-character base62 string (62^7 ≈ 3.5 trillion combinations) comfortably covers years of URLs at this scale. The follow-up interviewers ask immediately: "how do you generate a globally unique, roughly ordered ID across multiple servers without a single bottlenecked counter?" — answered by the Unique ID Generator case study (Snowflake-style IDs).

High-Level Architecture

Write path:  Client → API → generate ID → base62 encode → write {code: longURL} to DB → return short URL
Read path:   Client → GET /{code} → CACHE (Redis) check first → HIT: redirect immediately
                                                                → MISS: read DB, POPULATE cache, redirect
// This read path is a textbook CACHE-ASIDE pattern (the Caching Strategies lesson) — and because reads
// vastly outnumber writes, a high cache hit ratio here is what actually determines the system's
// real-world scalability, not the database's raw throughput.

Data Model & Storage Choice

{ shortCode: "cb", longURL: "https://...", createdAt, expiresAt, userId }
// Simple key-value access pattern (lookup by shortCode) → a KEY-VALUE STORE (DynamoDB, Cassandra) or
// even a simple relational table with shortCode as the PRIMARY KEY works fine. No complex joins/queries
// needed — this simplicity is a deliberate interview talking point: don't over-engineer the data layer
// for a problem that's fundamentally a key→value lookup.

Handling Custom Aliases & Expiration

Custom aliases ("mycompany.co/promo") need a uniqueness CHECK before insert (a unique index on the alias column handles this atomically — reject on constraint violation rather than check-then-insert, which has a race condition). Expired links: either a background job periodically purges expired rows, or — better at scale — a TTL feature on the storage layer itself (DynamoDB TTL, Redis EXPIRE) removes the operational burden of a cleanup job entirely.

Interview Follow-ups to Expect

• How do you scale the short-code generation across multiple servers? (distributed ID generation — the Unique ID Generator case study)
• How would you add click analytics without slowing down the redirect? (log the click event ASYNCHRONOUSLY — e.g., publish to a queue — the redirect response should never wait on analytics writes)
• What if a custom alias is requested that's already taken? (a unique constraint + graceful conflict response, not a check-then-insert race)
