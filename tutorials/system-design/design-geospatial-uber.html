Design Geo-Spatial Apps / Uber Case Study

The core problem — "find nearby drivers" — is fundamentally a SPATIAL indexing problem: a normal database index (B-tree on latitude, another on longitude) can't efficiently answer "what's near this 2D point," because proximity in 2D space doesn't map onto a simple sorted 1D range.

Requirements

Functional: riders find nearby available drivers; drivers continuously report location; match a rider to a nearby driver; track a trip in progress. Non-functional: LOW-LATENCY proximity queries (sub-second), handle millions of drivers updating location every few seconds (a massive WRITE load), accurate matching under real-world density variance (Manhattan vs rural).

Why a Normal Index Fails, and the Fix: Geohashing

// Querying "WHERE lat BETWEEN x AND y AND lon BETWEEN a AND b" with separate B-tree indexes on lat
// and lon requires intersecting TWO large result sets — inefficient, and gets WORSE as density grows.
//
// GEOHASHING encodes a 2D (lat, lon) pair into a SINGLE 1D string by interleaving bits of latitude
// and longitude, recursively subdividing the world into a grid:
(37.7749, -122.4194)  →  "9q8yy"     (San Francisco)
// KEY PROPERTY: nearby locations share a common PREFIX. "9q8yy" and "9q8yz" are geographically close.
// This turns "find nearby points" into "find strings sharing a prefix" — a problem regular indexes
// (or a trie — see the LeetCode track's Trie lesson) handle efficiently.
// Precision is tunable by geohash LENGTH: shorter prefix = larger area, longer = smaller/more precise.
🎯 Geohashing's prefix property is the entire trick — and its edge-case failure mode is the classic follow-up. Two points can be geographically ADJACENT but have completely DIFFERENT geohash prefixes if they straddle a grid boundary (e.g., opposite sides of the equator or a major grid line) — a naive "search nearby prefixes" query can miss genuinely close points. The fix: query the geohash cell AND its 8 neighboring cells, not just prefix-matching alone.

Alternative: Quadtrees

// A Quadtree recursively subdivides space into 4 quadrants, going deeper where DENSITY is higher
// (dense urban areas get finer subdivision; sparse rural areas stay coarse) — adapts to real-world
// non-uniform density better than geohashing's fixed grid.
// Trade-off: geohashing is simpler (just string prefix matching, works with existing indexes/DBs);
// quadtrees need a dedicated spatial data structure but adapt better to skewed density.
// Redis GEO commands (GEOADD, GEORADIUS) use geohashing internally — a common "just use Redis" answer
// for this exact problem at moderate scale.

Driver Location Updates — the Write-Heavy Half

// Millions of drivers, each pinging location every 3-5 seconds = enormous WRITE volume.
// Location data is EPHEMERAL and short-lived (only the LATEST position matters for matching) — this
// points AWAY from a durable, indexed database and TOWARD an in-memory store:
Driver app → location update → Redis (driver's geohash-indexed position, short TTL)
// No need to durably store every historical ping in the hot path — analytics/trip-history can consume
// a separate async stream (the Pub/Sub lesson) if historical GPS trails are needed later.

Rider-Driver Matching

Rider requests a ride → query nearby drivers (geohash cell + neighbors) → filter by availability,
  rating, vehicle type → RANK by estimated pickup time (not just raw distance — traffic/routing matters)
  → offer to the top candidate → driver accepts/declines (timeout → offer to next candidate)
// This is naturally a QUEUE-based dispatch, not a synchronous "pick one driver instantly" — drivers
// can decline, timeout, or already be taken by a concurrent request, requiring retry/re-offer logic.

Interview Follow-ups to Expect

• How do you handle the edge case where nearby drivers are geographically close but in different geohash cells? (query neighboring cells too, not just the exact prefix match)
• How would you scale driver location updates to 10M drivers globally? (shard Redis by REGION — a driver's geohash naturally partitions by geography, mapping cleanly onto sharding)
• How do you prevent two riders from being matched to the SAME driver simultaneously? (an atomic "claim" operation — e.g., a distributed lock or a conditional write — on the driver's availability status before confirming a match)
