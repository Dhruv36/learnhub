Heartbeat, Gossip, Retry with Back-off Fundamentals

In a distributed system, "is that node alive?" is a surprisingly hard question — you can never be 100% certain (a slow node and a dead node look identical from the outside). This lesson covers the three foundational mechanisms every failure-detection and coordination system builds on.

Heartbeats — the Simplest Failure Detector

Node B → sends "I'm alive" ping to Node A → every N seconds
Node A → if no heartbeat received for TIMEOUT seconds → assumes B is DOWN
// The core tuning trade-off:
//   Short timeout → detect failures FAST, but risk FALSE POSITIVES (a slow/GC-paused node looks dead)
//   Long timeout  → fewer false positives, but SLOWER failure detection (longer outage before failover)
⚠️ You can never distinguish "dead" from "slow/network-partitioned" with certainty. This is a fundamental limit (related to the FLP impossibility result), not an engineering oversight — it's why failure detectors are PROBABILISTIC in practice (e.g., "phi accrual" failure detectors used by Cassandra output a suspicion LEVEL rather than a binary alive/dead, adapting to each node's normal heartbeat variance instead of a single fixed timeout).

Gossip Protocol — Decentralized State Propagation

// Instead of every node reporting to a CENTRAL coordinator (a bottleneck/SPOF), each node periodically
// picks a few RANDOM peers and exchanges state with them:
Node A: "here's what I know — B is up, C is up, D hasn't been heard from in 30s"
       → gossips this to 3 random peers, who gossip to 3 more, who gossip to 3 more...
// Information spreads EXPONENTIALLY (like a rumor/epidemic) — reaches the whole cluster in O(log n)
// rounds, with NO single point of failure and NO central coordinator to scale/protect.

Real uses: Cassandra and DynamoDB-style databases use gossip for CLUSTER MEMBERSHIP (which nodes are up, joining, or leaving) — critical for a leaderless architecture with no central coordinator. Consul uses gossip (via the SWIM protocol) for service discovery and health checking.

Retry with Exponential Backoff + Jitter

// ❌ Naive retry: retry immediately, or at a FIXED interval:
retry after 1s, 1s, 1s, 1s...
// If the failure was caused by the SERVER being overloaded, thousands of clients retrying in lockstep
// makes it WORSE — a "retry storm" that can prevent the server from ever recovering.

// ✅ Exponential backoff: DOUBLE the wait each retry, capped at a maximum:
attempt 1 fails → wait 1s  → attempt 2 fails → wait 2s → attempt 3 fails → wait 4s → ... (cap at ~30s)

// ✅✅ Add JITTER — randomize the wait slightly — to prevent retries from staying SYNCHRONIZED:
wait = min(cap, base * 2^attempt) * random(0.5, 1.5)
// Without jitter, many clients that failed at the SAME moment (e.g., during a server restart) would
// all retry at the SAME moment again — recreating the exact thundering-herd problem backoff aims to fix.
🎯 Jitter is the detail that separates a textbook answer from a senior one. Plain exponential backoff without jitter still allows synchronized retry storms if many clients failed simultaneously (a very common real scenario — a brief server blip affects everyone at once). Randomizing each client's wait time desynchronizes the retries, spreading load smoothly instead of in synchronized waves.

Circuit Breakers — When to STOP Retrying

// Retrying a call to a dependency that's COMPLETELY down just wastes resources and adds latency.
// A circuit breaker tracks failure rate and "opens" (stops trying, fails FAST) after a threshold:
CLOSED (normal) → too many failures → OPEN (fail immediately, no calls attempted)
       → after a cooldown → HALF-OPEN (try ONE request to test recovery)
       → success → CLOSED again / failure → OPEN again
// Complements retry-with-backoff: backoff handles TRANSIENT failures, circuit breakers handle
// SUSTAINED outages where retrying at all is pointless until the dependency recovers.

Interview Follow-ups to Expect

• How would you detect a failed node in a 1,000-node cluster without a central coordinator? (gossip-based membership + a probabilistic/phi-accrual failure detector)
• Why can naive retries make an outage WORSE? (retry storms / thundering herd — everyone hammering a recovering server simultaneously)
• How do backoff and circuit breakers work together? (backoff spaces out retries for transient issues; a circuit breaker stops retrying altogether once failures indicate a sustained outage)
