Design Instagram News Feed Case Study

This closing case study is the deepest treatment of the fan-out trade-off that's appeared throughout this section (Chat, Notifications) — feed generation is the CLASSIC context where fan-out-on-write vs fan-out-on-read genuinely determines whether the system works at all, because of one specific problem: celebrity accounts with 100M+ followers.

Requirements

Functional: users post photos/videos; followers see posts in a ranked feed. Non-functional: feed loads FAST (users expect instant scroll), handle users following thousands of accounts AND accounts with hundreds of millions of followers, feed feels fresh/relevant.

Fan-Out on Write (Push Model) — the Default

User posts → for EACH of their followers → insert this post into that follower's PRECOMPUTED
  feed (stored e.g. in Redis, a sorted set per user keyed by timestamp)
// Reading a feed is then TRIVIAL and fast: just fetch the user's precomputed feed list — O(1)-ish,
// no computation at read time. Great for the common case (most users have a normal follower count).

The Celebrity Problem — Why Pure Fan-Out-on-Write Breaks

// A celebrity with 100M followers posting once means inserting into 100M SEPARATE feed lists —
// a single write operation that fans out to 100M writes. This is:
//   - SLOW: the post takes a very long time to propagate to everyone
//   - EXPENSIVE: 100M write operations for ONE post, repeated every time they post
//   - WASTEFUL: many of those followers may never even open the app before the post is stale/irrelevant
// This is the textbook "hot key" / "celebrity problem" in distributed systems — a small number of
// accounts generate wildly disproportionate load.
🎯 The hybrid solution is the expected senior-level answer. Use fan-out-on-WRITE for NORMAL users (the vast majority, low follower counts — precompute their followers' feeds immediately, feed reads stay fast). For accounts ABOVE a follower-count THRESHOLD (celebrities), switch to fan-out-on-READ: don't precompute at all — when a follower opens their feed, MERGE their normal precomputed feed with a live query of "recent posts from celebrities I follow" at READ time. This merge is cheap because there are relatively FEW celebrity accounts to check against, even though each has enormous reach.

Merging at Read Time (the Hybrid's Mechanics)

function getFeed(userId) {
  const precomputed = redis.getFeed(userId);               // fast — fan-out-on-write result
  const celebritiesFollowed = getFollowedCelebrities(userId);  // typically a SMALL list
  const celebrityPosts = celebritiesFollowed.flatMap(c => getRecentPosts(c));  // a few targeted queries
  return mergeAndRank(precomputed, celebrityPosts);          // combine, sort by rank/recency
}
// The follower's read gets slightly more expensive (a few extra targeted queries) — an acceptable
// trade for avoiding 100M writes on every celebrity post.

Ranking — Not Just Reverse-Chronological

// Modern feeds rank by predicted ENGAGEMENT, not pure recency — reusing the Recommendation System
// case study's retrieval-then-ranking funnel:
RETRIEVE: candidate posts from followed accounts (the fan-out result + celebrity merge above)
RANK: score each candidate (recency, predicted engagement, relationship strength, content type mix)
  → this ranking step is exactly the kind of ML-scored re-ranking from the Recommendation lesson,
    applied to a bounded candidate SET (not millions of items — just this user's followed-accounts' posts)

Interview Follow-ups to Expect

• What follower-count threshold would you use to switch to fan-out-on-read? (a tunable number based on real write-load measurements — the point where precomputing becomes measurably too expensive, often in the thousands-to-tens-of-thousands range depending on system capacity)
• How do you keep a normal user's precomputed feed from growing unbounded? (cap it — e.g., keep only the most recent N posts per user, trim older entries — full history isn't needed for a scrolling feed)
• How is this fan-out trade-off similar to the Chat case study's group-messaging problem? (the exact same push-vs-pull trade-off, applied to "followers" instead of "group members" — recognizing this as the SAME underlying pattern across different products is a strong interview signal)
