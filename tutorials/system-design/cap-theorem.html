CAP Theorem & Consistency Models Fundamentals

When a network partition splits your cluster, you must choose: refuse requests (stay Consistent) or serve possibly-stale data (stay Available). That forced choice under Partition is the CAP theorem — and partitions are not optional in distributed systems, so the real choice is only CP vs AP.

The Scenario That Makes It Concrete

[Node A] ←──✕ network cut ✕──→ [Node B]
User 1 writes balance=500 to A.   User 2 reads from B.
CP system: B says "unavailable, try later"  → correct but down     (banking)
AP system: B serves the old balance         → up but stale         (social feed)

Where Real Systems Sit

SystemChoiceWhy
Zookeeper, etcd, SpannerCPConfiguration/coordination must never diverge
Dynamo, Cassandra, RiakAP (tunable)Shopping carts & feeds prefer availability
Traditional RDBMS (single node)CA**No partition possible — CAP doesn't apply until you distribute

PACELC — the Senior-Level Extension

If Partition: trade Availability vs Consistency. Else (normal operation): trade Latency vs Consistency. Even with zero partitions, synchronously replicating every write to 3 nodes costs latency. DynamoDB chooses PA/EL (fast, eventually consistent by default); Spanner chooses PC/EC (consistent, pays latency via TrueTime). Mentioning PACELC in an interview immediately signals depth beyond the blog-post version of CAP.

The Consistency Spectrum

ModelGuaranteeUse when
Strong / LinearizableEvery read sees the latest writeMoney, inventory, locks
Read-your-own-writesYou always see your own updatesProfile edits, settings
Monotonic readsNever see older data after newerAvoid "time going backwards" in feeds
CausalRelated events keep their orderChat (reply never before its question)
EventualReplicas converge, eventuallyLike counts, view counts, DNS
💡 Staff-level insight: consistency is chosen per operation, not per system. In one e-commerce app: checkout = linearizable, product reviews = eventual, "your orders" page = read-your-own-writes. Designing one global consistency level is the classic mid-level mistake.

Interview Follow-ups to Expect

• "Your system is AP — how do replicas reconcile after the partition heals?" (last-write-wins vs vector clocks vs CRDTs)
• "How do you give read-your-own-writes on top of async replicas?" (route the user to the primary briefly after a write, or session-sticky reads)
• "Is Cassandra AP or CP?" (trick question — tunable: QUORUM reads+writes ⇒ effectively CP per-op)
