Bloom Filters, Merkle Trees, Inverted Index Fundamentals

Three specialized data structures that repeatedly show up as the "clever trick" inside larger system designs — a Bloom filter answers "definitely not present, or maybe present" in O(1) space-efficient fashion; a Merkle tree answers "are these two large datasets identical?" without transferring all the data; an inverted index is what makes full-text search possible at all.

Bloom Filter — Probabilistic Membership Testing

// A bit array + K hash functions. To ADD an element: hash it K ways, set those K bits to 1.
// To CHECK membership: hash it the same K ways — if ALL K bits are 1, "maybe present";
// if ANY bit is 0, "DEFINITELY not present."
add("user123")     → hash1("user123")=bit 7, hash2=bit 42, hash3=bit 91 → set bits 7, 42, 91 to 1
contains("user123") → check bits 7, 42, 91 — all 1 → "maybe present" (could be a false positive
                       from OTHER elements' hashes overlapping these same bits)
contains("user999") → any of ITS 3 bits is 0 → "definitely NOT present" — no false negatives, EVER
🎯 The one-directional guarantee is the entire point. Bloom filters NEVER produce false negatives (if it says "not present," that's certain) but CAN produce false positives (if it says "maybe present," it might be wrong). This asymmetry is exactly what you want as a cheap pre-check before an expensive operation: "check the Bloom filter first — if it says no, skip the disk read entirely; if it says maybe, do the real (expensive) lookup to confirm."

Real uses: preventing cache PENETRATION (the caching lesson) — check a Bloom filter before hitting the DB for a possibly-nonexistent key. Cassandra/HBase use Bloom filters to skip disk reads for SSTables that definitely don't contain a key. Chrome uses one for its Safe Browsing malicious-URL list (a huge list, checked locally, without downloading every URL).

Merkle Tree — Efficient Data Comparison & Sync

            hash(h1+h2)              ← ROOT hash — a single value representing the ENTIRE dataset
           /            \
      hash(d1)        hash(d2)         ← hash of each half
      /      \        /      \
   h(d1a)  h(d1b)  h(d2a)  h(d2b)       ← leaf hashes of individual data blocks
// To check if two REPLICAS have identical data: compare ROOT hashes. If they match, done — O(1)
// comparison regardless of dataset size. If they DIFFER, recurse into the mismatched subtree(s) —
// you only need to transfer/inspect the parts that actually differ, not the whole dataset.

Real uses: Cassandra/Dynamo use Merkle trees for ANTI-ENTROPY (detecting and repairing replica divergence without comparing every row). Git uses a Merkle-tree-like structure (commits hash their tree, which hashes its contents) — this is WHY two Git repos can quickly determine which commits differ. Blockchains use Merkle trees to let you verify a transaction is included in a block without downloading the entire block.

Inverted Index — the Engine Behind Search

// FORWARD index (the "obvious" structure): document → words it contains.
doc1: "the quick brown fox"
doc2: "the lazy fox sleeps"
// Answering "which documents contain 'fox'?" would require scanning EVERY document — O(n).

// INVERTED index: word → documents (and positions) containing it. Built ONCE, queried FAST:
"quick": [doc1]
"fox":   [doc1, doc2]
"lazy":  [doc2]
// Now "which documents contain 'fox'?" is a single O(1) hash lookup: [doc1, doc2]. This inversion
// is THE foundational data structure behind Elasticsearch, Lucene, and Google-scale search.
💡 Real inverted indexes store more than just doc IDs. Production systems store POSITIONS too (enabling phrase search — "brown fox" as an exact phrase, not just both words present anywhere), and TERM FREQUENCY (for ranking relevance — a doc mentioning "fox" 10 times ranks higher than one mentioning it once). This extra metadata is what turns a simple lookup structure into a ranked search engine — expanded on in the Search Engine case study.

Interview Follow-ups to Expect

• How do you size a Bloom filter? (trade-off between bit-array size / number of hash functions vs false-positive rate — more bits and more hashes = lower false-positive rate, more memory)
• What happens to a Bloom filter when data is DELETED? (standard Bloom filters can't remove elements without risking false negatives — use a Counting Bloom Filter instead, which uses counters instead of bits)
• How does a Merkle tree limit repair traffic in a distributed database? (only mismatched subtrees are compared/transferred — logarithmic, not linear, in dataset size)
