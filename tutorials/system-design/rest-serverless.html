REST, API Contracts, Serverless Fundamentals

How services expose functionality to each other shapes everything downstream — coupling, versioning pain, and operational cost. This lesson surveys the API style trade-offs and the serverless compute model at a systems-design level (deeper API mechanics live in the ASP.NET Core/Node.js tracks).

REST vs RPC vs GraphQL — the Real Trade-offs

StyleModelBest forWatch out for
RESTResources + HTTP verbs (GET/POST/PUT/DELETE)Public APIs, CRUD-heavy services, broad tooling/caching supportOver/under-fetching; N+1 calls for nested data
gRPC/RPCCall a remote function directly, binary protocol (Protobuf)Internal service-to-service calls, low-latency/high-throughputLess human-readable, needs codegen, browser support historically weaker
GraphQLClient specifies EXACTLY the fields it needs, one queryComplex, nested, client-driven data needs (mobile apps with varying screens)Server-side complexity (resolver N+1s), harder to cache at the HTTP layer
🎯 The interview-ready framing: match the API style to the CALLER. Public/third-party APIs → REST (universal tooling, cacheable, well-understood). Internal microservice-to-microservice → gRPC (performance matters more than human-readability, and you control both ends so codegen is fine). Client apps with highly variable data needs (web + iOS + Android hitting the same backend differently) → GraphQL (avoids over-fetching per-client).

API Contracts & Versioning

// A BREAKING change (renaming/removing a field, changing a type) breaks every client still on the
// old contract. The fix mirrors the DATABASE migration expand/contract pattern (CI/CD track):
//   - ADD new fields as OPTIONAL, never repurpose an existing field's meaning
//   - Version the API (URL path /v2/, a header, or a field) so old AND new clients work simultaneously
//   - DEPRECATE with a timeline, don't remove abruptly
// This is a recurring systems-design theme: schemas (DB or API) evolve; the discipline for evolving
// them SAFELY while old consumers still exist is always some form of expand/contract.

Idempotency — Critical for Retries

// Networks fail. Clients RETRY. If POST /charge isn't idempotent, a retried request charges the
// customer TWICE. The fix: an IDEMPOTENCY KEY the client generates once and sends with the request
// (and any retries) — the server checks "have I already processed this key?" before acting:
POST /charge  { amount: 50, idempotencyKey: "abc-123" }
// Server: if abc-123 already processed, return the CACHED result — don't charge again.
// This single pattern is assumed infrastructure in payment systems, and increasingly expected of
// any POST endpoint that isn't naturally idempotent (unlike PUT, which is idempotent by HTTP convention).

Serverless (FaaS) — the Operational Trade-off

BenefitCost
No servers to manage/patch/scale manuallyCold starts add latency (especially for infrequently-called functions)
Scales to zero — pay only for actual invocationsExecution time limits (often 15 min max) — not for long-running processes
Automatic horizontal scaling under loadVendor lock-in risk; harder local testing/debugging
Fast to ship event-driven, bursty workloadsCost can exceed a provisioned server at sustained high volume

When it fits: event-driven glue code (S3 upload triggers a thumbnail-generation function), infrequent/bursty workloads, webhooks. When it doesn't: sustained high-throughput services (provisioned servers/containers are cheaper at scale), latency-critical paths sensitive to cold starts, long-running processes.

Interview Follow-ups to Expect

• How do you avoid breaking existing clients when evolving an API? (additive changes, versioning, deprecation timeline — the expand/contract discipline)
• Why is idempotency critical for payment APIs specifically? (network retries must never double-charge — an idempotency key makes retries safe)
• When would you NOT use serverless for a new service? (sustained high QPS where cold starts and per-invocation cost outweigh the operational savings)
