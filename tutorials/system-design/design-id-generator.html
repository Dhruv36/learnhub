Design Unique ID Generator Case Study

"Generate a globally unique ID" sounds trivial (auto-increment!) until you add the real constraint: MULTIPLE servers, generating IDs CONCURRENTLY, with no single bottleneck or coordinator. This is the canonical distributed-ID problem, and Twitter's Snowflake algorithm is the canonical answer.

Requirements

Functional: generate unique IDs across a distributed system, at high throughput. Non-functional: IDs should be roughly TIME-ORDERED (helps with DB index locality and "sort by creation time" queries), no single point of failure/bottleneck, IDs fit in 64 bits (fits a standard integer type, unlike a 128-bit UUID).

Why a Simple Auto-Increment Doesn't Work

// A single DB auto-increment column is a SINGLE POINT OF FAILURE and a THROUGHPUT BOTTLENECK —
// every ID request serializes through one database. Doesn't scale, doesn't survive that DB's outage.
// UUIDs (128-bit random) solve uniqueness WITHOUT coordination — but:
//   - Not time-ordered (random) — bad for DB index locality (inserts scatter across the index,
//     hurting write performance on B-tree-indexed databases)
//   - 128 bits — 2× the storage of a 64-bit ID, at scale (billions of rows) this is real cost
// Need something in between: unique WITHOUT coordination, but STRUCTURED/ordered like an auto-increment.

Snowflake — Structured 64-bit IDs, No Coordination Needed

| 1 bit (unused/sign) | 41 bits: TIMESTAMP (ms since epoch) | 10 bits: MACHINE/WORKER ID | 12 bits: SEQUENCE |
// TIMESTAMP (41 bits): milliseconds since a custom epoch — gives ~69 years of unique timestamps,
//   and makes IDs ROUGHLY TIME-ORDERED (larger ID = later creation) — the key property auto-increment
//   had that pure UUIDs lack.
// MACHINE ID (10 bits): up to 1,024 distinct machines/workers can generate IDs SIMULTANEOUSLY without
//   coordinating with each other — each machine is assigned a fixed ID at startup (e.g., via ZooKeeper
//   or a config value).
// SEQUENCE (12 bits): up to 4,096 IDs per millisecond, PER MACHINE, for bursts within the same millisecond
//   (increment a local counter, reset it when the millisecond ticks over).
// Uniqueness guarantee: two IDs can only collide if they have the SAME timestamp, SAME machine ID, AND
// SAME sequence number — impossible if sequence increments correctly within each machine's own clock tick.
🎯 The core insight: uniqueness through STRUCTURE, not coordination. Instead of a central authority handing out sequential numbers (a bottleneck), each machine independently generates IDs guaranteed unique by CONSTRUCTION — its own fixed machine ID segment ensures no OTHER machine can produce the same ID, even without ever communicating. This is the general pattern for eliminating coordination bottlenecks: partition the ID SPACE upfront (by machine) so no runtime coordination is needed at all.

Clock Skew — the Hard Follow-up

// What if a machine's clock moves BACKWARD (NTP correction, VM migration)? A naive implementation
// could generate an ID with an EARLIER timestamp than one it already generated — breaking the
// "roughly ordered" property, or in a bad implementation, causing an outright collision.
// Mitigations: detect clock regression and REFUSE to generate IDs until the clock catches up
// (brief unavailability, but correctness preserved) — or use a HYBRID logical clock that
// tolerates small amounts of clock skew without stalling.

Alternative Approaches (know the landscape)

ApproachTrade-off
UUID v4 (random)Zero coordination, but not ordered, 128 bits, poor index locality
Database ID range allocation (each server reserves a BLOCK of IDs, e.g., 1000 at a time)Simple; a server crash "wastes" its unused reserved block, but that's an acceptable trade for simplicity
Snowflake-styleOrdered, compact (64-bit), no coordination — the standard for high-scale systems (used by Twitter, Instagram, Discord)

Interview Follow-ups to Expect

• Why does time-ordering matter for the ID? (better DB index locality on insert — sequential-ish inserts, vs random UUIDs scattering writes across a B-tree index, which hurts write throughput and cache locality)
• How do you assign machine IDs without a manual/error-prone process? (a coordination service like ZooKeeper hands out unique machine IDs at startup, or use a cloud instance's unique metadata)
• What happens if the sequence number overflows within a millisecond (more than 4,096 IDs needed)? (wait until the next millisecond tick, or accept slight ID reordering as a rare edge case)
