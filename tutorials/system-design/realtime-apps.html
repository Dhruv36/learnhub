Realtime Apps (WebSockets, Push) Deep Dive

Chat, live notifications, collaborative editing, live dashboards — all need the SERVER to push data to clients, inverting the normal request/response model. This lesson covers the transport choices and, critically, how to scale server-push connections beyond a single machine (the hardest part).

The Transport Options

OptionDirectionBest for
Long PollingClient asks repeatedly, server holds the request open until data's readyMaximum compatibility, low realtime needs
Server-Sent Events (SSE)Server → client only, over plain HTTPOne-way live updates (stock tickers, live feeds) — simpler than WebSockets when you don't need client→server
WebSocketFull-duplex, persistent connectionTrue bidirectional realtime (chat, gaming, collaborative editing)
Push Notifications (APNs/FCM)Server → device, even when the app isn't openMobile alerts when the app isn't in the foreground

The Hard Part: Scaling WebSocket Connections

// A WebSocket connection is STATEFUL and STICKY — once established, that client is bound to ONE
// specific server instance for the connection's lifetime (unlike stateless HTTP requests, which any
// instance can handle). This breaks the usual "just add more servers behind a load balancer" scaling story:

Client A connects → Server 1 (connection lives HERE)
Client B connects → Server 2 (connection lives HERE)
// If Server 1 needs to send a message to Client B (e.g., "user A sent you a message"), Server 1
// has NO DIRECT WAY to reach Client B's connection — that socket lives on Server 2's memory.
🎯 The fix: a message broker/pub-sub layer connecting server instances. All server instances subscribe to a shared broker (Redis Pub/Sub, or Kafka). Server 1 publishes "deliver this to Client B" to the broker; Server 2 (which owns Client B's actual socket) is subscribed and receives it, then pushes it down its LOCAL socket to Client B. This decouples "which server has the business logic" from "which server holds the specific client's connection" — the exact same pattern as the ASP.NET Core track's SignalR-backplane problem, at the general systems-design level.

Connection Management at Scale

// Each open connection consumes memory + a file descriptor. A single server can typically hold
// tens of thousands of concurrent WebSocket connections (tuned OS limits, event-loop-based servers)
// — but "millions of concurrent users" needs MANY server instances, each holding a SHARD of connections.
//
// A CONNECTION REGISTRY (often in Redis) tracks "which server instance holds Client X's connection" —
// so other services know WHERE to route a message for a given user:
SET connection:user123 = server-instance-7
// On disconnect, remove the entry; on reconnect (to a possibly DIFFERENT instance), update it.

Delivery Guarantees for Realtime Systems

// Realtime ≠ guaranteed delivery. A client that's offline when a message is sent WON'T receive it
// over the live socket. Design choices:
//   - FIRE AND FORGET: fine for ephemeral data (live cursor positions, "user is typing" indicators)
//   - PERSIST + DELIVER ON RECONNECT: store the message (e.g., in the chat's message history), and
//     when the client reconnects, fetch anything missed since their last-seen timestamp/message ID
//   - The chat/notification CASE STUDIES build directly on this distinction — know which each product
//     need actually requires before defaulting to "just use WebSockets for everything."

Interview Follow-ups to Expect

• How do you send a message to a specific user when you don't know which server holds their connection? (a connection registry, e.g., in Redis, mapping user → server instance)
• What happens to in-flight messages if a server holding WebSocket connections crashes? (those clients disconnect and must reconnect — often to a different instance — messages sent during the gap need a "catch up since last seen" mechanism if they must not be lost)
• WebSocket vs SSE for a live sports-score feed? (SSE — it's one-directional, simpler, and works over plain HTTP/CDN-friendly infrastructure more easily than WebSockets)
