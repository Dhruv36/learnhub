Design Amazon / E-Commerce Case Study

An e-commerce platform's defining hard problem is INVENTORY correctness under massive concurrency — "don't oversell the last item" is a genuine distributed-systems race condition, not a business-logic detail, and it's what most interviews on this topic actually probe.

Requirements

Functional: browse/search products, manage inventory, add to cart, checkout/payment, order tracking. Non-functional: strong consistency for inventory (never oversell), high availability for browsing (product pages should basically never be down), handle massive traffic spikes (Black Friday, flash sales).

The Core Problem: Preventing Overselling

// ❌ Naive: check stock, THEN decrement — a race condition under concurrency:
SELECT stock FROM products WHERE id=5;    -- returns 1 (last item)
-- TWO customers both read "1" here, concurrently, before either writes
UPDATE products SET stock = stock - 1 WHERE id=5;   -- BOTH succeed → stock = -1, oversold

// ✅ Atomic conditional decrement — the check and the update happen as ONE indivisible operation:
UPDATE products SET stock = stock - 1 WHERE id=5 AND stock > 0;
-- If 0 rows affected → out of stock, the "read" and "write" can't be split by another transaction
🎯 This is the SAME class of bug as the Rate Limiter's check-then-decrement race — recognize the pattern. "Read a value, decide based on it, then write" is ALWAYS a race condition under concurrency unless the check-and-write happens ATOMICALLY. The fix is always some form of "make the condition part of the write itself" (a conditional UPDATE, a CAS operation, a database constraint) rather than two separate round-trips.

Handling a Flash Sale (Extreme Contention on One Item)

// 100,000 concurrent requests all hitting the atomic UPDATE for the SAME row is still a bottleneck
// EVEN WITH the correct atomic pattern — that one database ROW becomes a serialization point.
// At extreme scale: RESERVE stock in a fast in-memory store (Redis, using DECR — atomic, very high
// throughput) BEFORE touching the durable database at all:
Redis: DECR flash_sale:item5:stock   → if result >= 0, the request WON a unit; async persist to DB
                                      → if result < 0, sold out — reject immediately, no DB touched at all
// This absorbs the extreme burst in memory, protecting the database entirely, then asynchronously
// reconciles the authoritative durable record.

Checkout — a Multi-Service Saga

// Checkout touches Inventory, Payment, and Order services — a DISTRIBUTED TRANSACTION, solved with
// the Microservices lesson's SAGA pattern:
1. Reserve inventory (compensate: release)
2. Charge payment    (compensate: refund)
3. Create order       (compensate: cancel)
// If payment fails after inventory was reserved, run inventory's compensating action (release) —
// this is why "reserve, don't immediately decrement" matters: a reservation can be safely undone.

Interview Follow-ups to Expect

• How do you handle a customer's cart across multiple devices? (server-side cart tied to the account, not device-local storage — syncs automatically)
• How would you design for Black Friday-level traffic on the browse/search path specifically? (aggressive caching + CDN for product pages, since browsing is read-heavy and tolerates eventual consistency — very different treatment than the inventory-write path)
• What happens if payment succeeds but the order-creation step fails? (the saga's compensating action — refund the payment — and the system must be built to detect and trigger this reliably, e.g., via a saga orchestrator with persistent state, not just in-memory logic that a crash could lose)
