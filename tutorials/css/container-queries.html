Container Queries Advanced

Media queries answer one question: how big is the viewport? But a component doesn't live in the viewport — it lives in a slot. The same ProductCard renders in a 900px main column, a 280px sidebar, and a 500px modal on the same page at the same viewport width. Media queries cannot express that. Container queries let a component respond to its own available space — the missing piece that makes design-system components genuinely reusable, and the biggest change to responsive design since media queries themselves.

Why It Took 15 Years (the circularity problem)

Naively, "style children by parent width" is circular: children's styles can change the parent's width, which changes the children's styles, forever. The solution that finally shipped: containment. Declaring container-type: inline-size promises the engine that this element's width comes from layout above, not from its contents — breaking the cycle. That's why the container declaration is mandatory, and why a container whose width isn't constrained by its parent layout "doesn't work".

The Two Steps

/* 1. Establish a container (on the SLOT/wrapper, not the component itself): */
.card-slot { container: card / inline-size; }        /* shorthand: name / type */
/* or: container-type: inline-size; container-name: card; */

/* 2. Query it from ANY descendant: */
.product-card { display: flex; flex-direction: column; gap: .75rem; }   /* narrow default */

@container card (min-width: 420px) {
  .product-card { flex-direction: row; }              /* wide slot: image beside text */
  .product-card img { inline-size: 40%; }
}
@container card (min-width: 640px) {
  .product-card .description { display: block; }      /* roomy: show more content */
}

Unnamed queries (@container (min-width: …)) match the nearest ancestor container — fine for simple cases; names prevent surprises when containers nest (a card inside a sidebar inside a page region).

Container Units — Component-Relative Sizing

@container card (min-width: 300px) {
  .product-card h3 { font-size: clamp(1rem, 4cqi, 1.4rem); }
}
/* cqi = 1% of container inline size · cqb = block size · cqw/cqh = width/height
   cqmin/cqmax = smaller/larger of cqi|cqb
   Fluid type that scales with the CARD — the same card is proportionally correct
   in every slot. Viewport units can't express this. */

Style Queries — Conditioning on Context, Not Just Size

.hero-section { --context: promo; }
@container style(--context: promo) {
  .card { border-color: gold; background: linear-gradient(…); }
}
/* Children restyle based on a parent's CUSTOM PROPERTY value. Ships in Chromium/WebKit;
   check current support. Powerful pairing: size queries for GEOMETRY, style queries for MOOD. */

Worked Example — One Card, Three Homes

<main class="card-slot">    <product-card>  →  horizontal, big type, full description
<aside class="card-slot">   <product-card>  →  vertical, compact, description hidden
<dialog class="card-slot">  <product-card>  →  horizontal-compact

/* SAME component, SAME markup, ZERO props/modifiers/JS.
   The pre-container-query alternatives, for contrast:
   ① .product-card--sidebar modifier classes (manual, forgotten, drift)
   ② ResizeObserver + JS class toggling (layout thrash, SSR mismatch)
   ③ media queries guessing "sidebar exists above 1100px" (breaks on refactor)
   Container queries delete all three. */

Where Media Queries Still Win

Container queriesMedia queries
Component-level adaptation (cards, tables, widgets, nav items)Page-level layout (is there a sidebar at all? burger vs full nav)
Anything published in a design systemUser preferences: dark mode, reduced motion, contrast, print
Type scaling within a module (cqi)Viewport-tied concerns (dvh heroes, safe areas)

Rule of thumb that ends most debates: viewport decisions → media; slot decisions → container. A mature codebase uses media queries a handful of times (page shell, preferences) and container queries throughout the component layer.

Gotchas & Sharp Edges

GotchaExplanation / fix
"My query never fires"The container's width isn't constrained (it's shrink-wrapping content). Give the slot a real width via grid/flex/explicit sizing
Container can't query itselfBy design (circularity) — queries style DESCENDANTS. Wrap the component in a slot div if you must restyle its root
inline-size containment side effectsThe container also gets style+layout containment: floats don't escape, margins don't collapse through — occasionally visible
size vs inline-sizecontainer-type: size requires a constrained HEIGHT too (rarely what you want); inline-size is the default choice
Nested containersUnnamed @container matches the NEAREST — name your containers in any non-trivial layout
Legacy browsersGreen across the board since 2023; for long-tail support, design the no-query default (narrow layout) as the fallback — it's progressive enhancement by construction

Interview Questions

Q1: Why do container queries require container-type, mechanically?

To break the circular dependency: if children's styles could change the container's size, which gates children's styles, layout never stabilizes. inline-size containment declares the width comes from ancestor layout, so the engine can resolve queries in one pass. No containment, no guarantee, no queries.

Q2: A design-system consumer reports your card "ignores its breakpoints" inside their custom panel. Diagnose.

Their panel isn't a container (no containment ancestor) or the panel's width is content-driven. Fix on the system side: ship the slot — export a .card-slot wrapper with container declared — or document the containment contract. This exact conversation is why systems ship containers, not just components.

Q3: Compare container queries vs ResizeObserver-driven classes.

Same capability class, but CQ is declarative, synchronous with layout (no flash of wrong state), SSR-safe (no hydration flicker), and runs in the style engine (no main-thread observer churn). ResizeObserver remains for JS-behavioral responses (virtualization, canvas redraws) — not for styling.

Q4: What do cqi units give you that % and vw can't?

% resolves against the parent box per-property (and not for font-size in the useful sense); vw is viewport-blind to slots. cqi is "percent of my container's inline size" for ANY property, including font-size — proportional components that hold their design ratio in every slot.

🏋️ Practice (Playground):
1. Build the one-card-three-homes demo: main/aside grid, one .card component, container on the slots, horizontal↔vertical flip at 420px.
2. Break it deliberately: remove container-type and watch queries die; explain why before re-adding.
3. Add cqi-based heading size and drag the sidebar's track width — watch type scale with the slot.
4. Nest containers (page region + card slot), omit names, observe the mismatch, then fix with named containers.

🎯 Key Takeaways

  • Media = viewport; container = slot. Components respond to their own space — reusability media queries never allowed.
  • Two steps: container: name / inline-size on the wrapper; @container name (condition) on descendants.
  • Containment exists to break circularity — unconstrained-width containers are the #1 "not firing" cause.
  • cqi/cqb units = component-relative fluid sizing; style() queries condition on context variables.
  • Design systems ship the slot (container) as part of the component contract.
  • Baseline-supported since 2023 — production-safe, with the narrow default as natural fallback.
