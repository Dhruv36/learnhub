CSS Performance Ultra-Advanced

CSS hurts performance in two distinct ways: as a render-blocking download (loading) and as style/layout work (runtime). Different problems, different tools.

Loading: CSS Blocks First Paint

The browser will not paint anything until all stylesheets in <head> arrive and parse — otherwise users would see unstyled flashes. So every KB of CSS delays first paint.

<!-- Critical CSS pattern (used by news/e-commerce sites): -->
<style>/* inlined: header, hero, above-the-fold — the ~10KB that styles first paint */</style>
<link rel="preload" href="/css/full.css" as="style" onload="this.rel='stylesheet'">

<!-- Split by purpose — print CSS shouldn't block screen rendering: -->
<link rel="stylesheet" href="print.css" media="print">
TacticEffect
Ship less (purge unused)Real-world stylesheets are typically 60–90% unused — build tools (PurgeCSS, Tailwind JIT) strip it

Runtime: Reflow & Repaint

Layout (reflow) — geometry changed; expensive, can cascade to the whole page
Paint           — pixels changed (color, shadow); cheaper
Composite       — layers moved (transform/opacity); cheapest — GPU only

Rules that follow: animate only transform/opacity (Transitions lesson); batch DOM reads/writes in JS (interleaving them forces layout thrashing — layout recalculated per iteration).

content-visibility — Skip Rendering What's Off-Screen

.comment-section, .footer-mega {
  content-visibility: auto;
  contain-intrinsic-size: auto 800px;   /* placeholder height so scrollbar doesn't jump */
}

The browser skips style/layout/paint for off-screen sections entirely — long pages (feeds, docs, comment threads) commonly cut initial rendering cost by half or more. The companion contain: content isolates a component's layout so its changes can't force the whole page to reflow.

Selector Cost (mostly a myth, one exception)

Selector matching is highly optimized — readable selectors beat "fast" ones every time. The one real cost: broad :has() / universal descendant patterns on huge DOMs (previous lesson). Profile before believing any selector folklore.

💡 Production tip: measure in this order — Lighthouse "Eliminate render-blocking resources" and "Reduce unused CSS" for loading; DevTools Performance panel for runtime (purple layout storms = thrashing). Optimizing without these numbers is superstition.
