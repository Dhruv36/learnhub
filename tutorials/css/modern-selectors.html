Modern Selectors (:has, :is, nesting) Modern CSS

What you'll master: :has() as a genuine "parent selector" (finally solving a decades-old, previously-JS-only problem), :is()/:where() for selector-list deduplication and the real specificity difference between them, native CSS nesting and its relationship to Sass, and :not()'s multi-argument modern form. Prerequisite: Cascade Layers.

Part 1 — :has(): the "Parent Selector" CSS Never Had

.card:has(img) { border: 2px solid gold; }        /* select a .card, but ONLY IF it contains an img */
.form-group:has(input:invalid) { background: #fee; }  /* style a form-group based on ITS OWN
                                                            descendant's validity state */
a:has(> img) { border: none; }                     /* only if img is a DIRECT child (combinators
                                                        work inside :has()'s argument too) */
h2:has(+ p) { margin-bottom: 8px; }                 /* select an h2, but ONLY IF a p immediately
                                                        follows it — a genuine "look AHEAD" selector */

This is the single most significant, longest-awaited CSS feature covered in this entire section: :has() lets a selector match based on the PRESENCE of something INSIDE or AFTER it — genuinely selecting a PARENT based on its children's state, something CSS historically had NO way to express at all (every other selector/combinator only ever looks at an element itself or its ANCESTORS/PRECEDING siblings, never descendants or following content). Before :has(), "style this container differently if it happens to contain an image" or "style a form field's wrapper red if the field inside it is currently invalid" REQUIRED JavaScript — adding or removing a class based on watching the DOM for exactly this condition. :has() expresses it as pure, declarative CSS, with the browser handling the "watching" automatically and efficiently.

Part 2 — :is() and :where(): Selector-List Deduplication, With a Real Specificity Difference

/* WITHOUT :is() — repetitive, error-prone to keep all three in sync: */
header h1, header h2, header h3,
article h1, article h2, article h3,
footer h1, footer h2, footer h3 { margin-top: 0; }

/* WITH :is() — the SAME selector, expressed once: */
:is(header, article, footer) :is(h1, h2, h3) { margin-top: 0; }

:is() collapses combinatorial selector lists into one compact, DRY declaration — a real, practical readability and maintainability win, especially as the number of combined selectors grows (3×3=9 individual selectors above, collapsed to one line). The genuinely important, real difference from its sibling :where():

:is(header, article, footer) h1 { }    /* specificity = the HIGHEST-specificity selector INSIDE
                                            the :is() list — here, still just (0,0,1) since all
                                            three are simple type selectors, but a MIXED list like
                                            :is(#nav, .card) WOULD contribute (1,0,0) */
:where(header, article, footer) h1 { }  /* specificity = ZERO, always, regardless of what's inside —
                                             :where()'s argument list NEVER contributes any
                                             specificity at all */

:is() takes on the specificity of its HIGHEST-specificity internal argument (so :is(#header, .card) carries id-level specificity, because #header is the highest-specificity option inside it); :where() is specifically defined to ALWAYS contribute ZERO specificity, no matter what's inside it — genuinely useful for CSS reset/utility authors who want the CONVENIENCE of a selector list without it accidentally out-competing a consuming project's own, intentionally-low-specificity overrides later.

Part 3 — Native CSS Nesting: What Changed From Preprocessors

/* Sass (a PREPROCESSOR — compiled to plain CSS before shipping): */
.card {
  padding: 16px;
  &:hover { background: #f5f5f5; }
  .title { font-weight: 700; }
}

/* NATIVE CSS nesting (2023+, no build step required — runs directly in the browser): */
.card {
  padding: 16px;
  &:hover { background: #f5f5f5; }
  .title { font-weight: 700; }
}
/* the syntax is DELIBERATELY nearly identical — but this runs AS-IS in the browser,
   with ZERO compilation step needed */

Native CSS nesting's syntax was deliberately designed to closely resemble Sass's (easing the transition for the huge existing base of Sass-experienced developers), but the MECHANISM is fundamentally different: Sass nesting is compiled away entirely before the browser ever sees it (a build tool flattens the nested syntax into plain, non-nested CSS selectors); native CSS nesting runs directly in the browser, with no build step required at all — a genuine, real reduction in tooling dependency for projects that previously needed Sass PRIMARILY for its nesting convenience (custom properties, covered earlier, similarly reduced the need for Sass variables specifically). One real, worth-knowing syntax gotcha: a nested rule starting with a bare TYPE selector (not &, a class, or another combinator) needs an explicit & prefix or it can be ambiguously parsed — p { span { } } nesting works in most contexts, but the safest, most universally-compatible habit is prefixing nested type selectors explicitly when in doubt.

Part 4 — :not()'s Modern, Multi-Argument Form

/* OLD (CSS3) — :not() only ever accepted ONE simple selector, requiring chaining for multiple: */
li:not(.active):not(.disabled) { }

/* MODERN — :not() accepts a full, comma-separated SELECTOR LIST directly: */
li:not(.active, .disabled) { }
/* both produce the IDENTICAL result — the modern form is simply more compact and more readable,
   especially as the exclusion list grows beyond 2-3 items */

A small, genuinely welcome syntax improvement — functionally equivalent to the older chained form, but noticeably more readable once excluding more than 2 conditions, and directly consistent with :is()/:where()'s own selector-list argument style, part of a broader, deliberate modernization trend making CSS's selector syntax more internally consistent across these related pseudo-classes.

Part 5 — Worked Example: :has() Replacing Real JavaScript

/* BEFORE :has() — required JS watching form validity and toggling a class: */
form.addEventListener("input", () => {
  document.querySelectorAll(".form-group").forEach(group => {
    const invalid = group.querySelector("input:invalid");
    group.classList.toggle("has-error", !!invalid);
  });
});
.form-group.has-error { border-left: 3px solid red; }

/* AFTER :has() — the identical visual result, ZERO JavaScript: */
.form-group:has(input:invalid) { border-left: 3px solid red; }

This is a genuinely representative, real example of :has()'s practical impact: an entire, previously-necessary JS event listener plus a manual class-toggling loop collapses into ONE line of pure, declarative CSS — the browser's own native validity-state tracking (the HTML track's Constraint Validation lesson) now drives the styling directly, with no JS synchronization code, no risk of the class drifting out of sync with actual validity state, and genuinely less code to maintain. This is a real, concrete instance of the broader "check the platform before reaching for JS/a library" theme this entire multi-track curriculum has repeated since the HTML track's lesson-16 method.

Interview Questions

Q1 (beginner): What genuinely new capability does :has() give CSS that no selector/combinator had before?

The ability to select a PARENT (or an earlier sibling) based on the presence of something INSIDE it or AFTER it — every other CSS selector/combinator can only look at an element itself or its ancestors/preceding siblings, never its descendants or following content. Before :has(), styling a container differently based on its children's state (e.g., "highlight this form group if the input inside it is invalid") required JavaScript to watch and toggle a class manually.

Q2 (beginner): What's the real difference between :is() and :where(), beyond them both accepting a selector list?

:is() takes on the specificity of its HIGHEST-specificity internal argument (a mixed list like :is(#id, .class) carries id-level specificity). :where() is specifically defined to ALWAYS contribute ZERO specificity, regardless of what's inside it — useful for reset/utility authors who want selector-list convenience without accidentally out-competing a consuming project's own, intentionally low-specificity overrides.

Q3 (mid): Is native CSS nesting the same mechanism as Sass nesting, given their near-identical syntax?

No — despite deliberately similar syntax (easing the transition for Sass-experienced developers), the mechanism is fundamentally different. Sass nesting is compiled away entirely by a build tool before the browser ever sees the CSS (flattened into plain, non-nested selectors). Native CSS nesting runs directly in the browser with zero build step required — a real reduction in tooling dependency for projects that previously needed Sass primarily for nesting convenience.

Q4 (mid): Rewrite li:not(.active):not(.disabled):not(.hidden) using the modern :not() form, and explain the practical benefit.

li:not(.active, .disabled, .hidden) — functionally identical, but noticeably more compact and readable, especially as the exclusion list grows. It's also directly consistent with :is()/ :where()'s own selector-list argument syntax, part of a broader modernization making these related pseudo-classes' syntax more internally consistent.

Q5 (senior): Your team replaces a JS-based "highlight the form group containing an invalid field" implementation with .form-group:has(input:invalid) { border-left: 3px solid red; }. A code reviewer flags a concern: "won't this run on EVERY keystroke across the whole form, potentially causing performance issues on a very large form with 100+ fields?" Assess this concern.

The concern reflects a reasonable INSTINCT (CSS selectors DO need to be re-evaluated as the DOM's relevant state changes) but is largely misplaced in practice for this specific case: browsers' CSS selector-matching engines are HIGHLY optimized, native machinery — built specifically to efficiently re-evaluate selectors (including :has()) as relevant DOM/state changes occur, and this is generally comparable to or FASTER than the equivalent hand-written JS solution it replaced (which itself was running a querySelectorAll loop across the whole form on every single input event, arguably MORE expensive than letting the browser's own optimized selector engine handle it natively). That said, the underlying performance-scoping instinct IS worth taking seriously in GENERAL for :has(): a very broad, unscoped :has() query across an ENTIRE large page (rather than scoped to a specific, bounded component like .form-group) can carry real, measurable cost at sufficiently large DOM scale, and the general discipline of scoping selectors as narrowly as reasonably possible (a principle that applies to ALL CSS selectors, not uniquely to :has()) remains sound advice. Recommendation: acknowledge the reviewer's instinct as generally sound selector-performance hygiene, but note this SPECIFIC case (a scoped :has() replacing an objectively MORE expensive JS querySelectorAll loop) is very likely a net performance IMPROVEMENT, not a regression — and if genuine concern remains, the correct next step is MEASURING (via DevTools' Performance panel, recording actual keystroke-triggered style recalculation time) on the real form with its real field count, rather than assuming a regression from the selector's syntax alone.

Q6 (senior): Design a fully CSS-only (zero JavaScript) "empty state" indicator for a product list: when a filterable product grid has zero visible matching items (all filtered out via some other mechanism, e.g. hidden via display:none by a JS filter), a "No products match your filters" message should automatically appear, and disappear the instant at least one product becomes visible again.

This is a genuine, real application of :has() combined with the :not() pseudo-class, taking advantage of :has()'s "look inside for the ABSENCE of a condition" capability (via negation inside the argument): .product-grid:has(.product:not([hidden])) ~ .empty-state { display: none; } .empty-state { display: block; } — the empty-state message is block/visible by DEFAULT, and is HIDDEN specifically when the product grid contains AT LEAST ONE product that is NOT hidden (i.e., at least one visible match exists) — inverting the logic so the "hide the message" rule does the actual conditional work, since :has() can express "contains at least one match" directly, while "contains ZERO matches" needs to be expressed as its logical inverse via this hide-when- present pattern. (An equivalent, sometimes more intuitive alternative: .empty-state { display: none; } .product-grid:not(:has(.product:not([hidden]))) ~ .empty-state { display: block; } — showing the message specifically when the grid does NOT contain any visible product, using :not() wrapping :has() to express the direct, positive "zero matches" condition instead of the inverted version.) Either form achieves a genuinely zero-JavaScript empty-state indicator, automatically and instantly reacting to whatever OTHER mechanism (a JS filter script, in this scenario) is toggling individual products' hidden state — the CSS itself contains NO knowledge of filtering logic at all, purely reacting declaratively to the resulting DOM state via :has(). The senior point: recognizing that :has() combined with :not() and the general sibling combinator (~) can express a surprisingly wide range of conditional-visibility logic that used to require dedicated JS state-tracking, is precisely the kind of "check what the platform can now do before reaching for script" instinct this lesson (and this multi-track curriculum's recurring theme) is meant to instill.

🏋️ Exercises

1 (easy): :has(), your first real use. Build a card containing an optional image (some cards have one, some don't). Using :has(img), give ONLY the cards that actually contain an image a distinct border style, with zero JavaScript and zero extra classes.
Solution 
.card:has(img) { border: 2px solid gold; }

Confirm cards WITHOUT an image show no special border, while cards WITH one do — purely from the presence of the img element inside, with no class ever added to distinguish them manually.

2 (easy): :is() vs :where(), specificity compared. Build :is(#sidebar, .card) p { color: navy; } and a competing .content p { color: red; } (equal or lower specificity than the :is() version, depending on which internal argument wins). Then swap to :where(#sidebar, .card) p { color: navy; } and observe whether red now wins instead, confirming :where()'s zero-specificity behavior.
Expected result

With :is(): navy likely wins (since #sidebar contributes id-level specificity to the whole :is() selector). With :where(): red should now win, since :where() contributes ZERO specificity regardless of its #sidebar argument — a direct, hands-on, side-by-side confirmation of Part 2's real specificity difference.

3 (medium): Native nesting, refactored from flat CSS. Take a flat, non-nested set of related rules (.card, .card:hover, .card .title, .card .price) and rewrite them using native CSS nesting. Confirm the rendered result is visually identical before and after — nesting is a SYNTAX change, not a behavior change.
Solution 
.card {
  padding: 16px;
  &:hover { background: #f5f5f5; }
  .title { font-weight: 700; }
  .price { color: #059669; }
}

Rendering should be pixel-identical before and after the refactor — confirming nesting is purely an authoring-syntax convenience, producing the exact same underlying selectors/cascade behavior, not a new capability in itself.

4 (medium): The zero-JS form validator. Build Part 5's complete example: a form with several required/typed inputs, each wrapped in a .form-group, with .form-group:has(input:invalid) { border-left: 3px solid red; } as the ONLY CSS needed. Confirm groups correctly highlight red the instant their input becomes invalid (e.g., an empty required field, or a malformed email) and un-highlight the instant it becomes valid — with zero JavaScript anywhere in the page.
Verification

Typing into a required field and then clearing it should show the red border appear IMMEDIATELY (as soon as the field's :invalid state becomes true, which happens live as you type, not just on submit) with zero JS running — confirming the browser's native validity tracking (HTML track's Constraint Validation lesson) drives this CSS directly and instantly.

5 (hard): The empty-state indicator, built. Implement Q6's complete zero-JS empty-state pattern: a product grid with several .product items, a simple JS toggle button that hides/shows individual products (simulating a filter, via toggling a [hidden] attribute on one product at a time for testing), and an .empty-state message that appears ONLY when ALL products are hidden, using either of Q6's two CSS approaches. Confirm the message appears/disappears correctly as you toggle different combinations of hidden products.
Solution (positive form) 
.empty-state { display: none; }
.product-grid:not(:has(.product:not([hidden]))) ~ .empty-state { display: block; }

With at least one product visible: message stays hidden. Hide ALL products one by one: the message should appear the INSTANT the last visible product becomes hidden, and disappear again the instant you unhide even one — confirming the CSS reacts live and correctly to the resulting DOM state, with zero JavaScript involved in the actual show/hide-message LOGIC itself (only in the test harness toggling individual products' hidden attribute).

6 (hard): The performance scoping discipline, applied. Build TWO versions of a :has()-based highlight rule: one deliberately UNSCOPED (a bare *:has(input:invalid) { ... } applied globally across a large, deliberately bloated test page with many unrelated elements), and one properly SCOPED (.form-group:has(input:invalid) { ... }). Using DevTools' Performance panel, record typing rapidly into a form field on both versions and compare any measurable difference in style-recalculation time, writing a short conclusion about Q5's scoping discipline based on your OWN measurement rather than assumption.
What to look for

On a sufficiently large, bloated test page, the unscoped *:has(...) version may show a measurably longer style-recalculation phase in the Performance panel's timeline compared to the properly-scoped .form-group:has(...) version, since the browser must consider a much larger set of elements as potential matches for the universal selector's :has() condition. Whether the DIFFERENCE is actually significant enough to matter in practice depends heavily on real page size/complexity — the exercise's genuine point is developing the HABIT of measuring this directly (per Q5's recommendation) rather than assuming either "it's always fine" or "it's always a problem" without checking your own actual, specific page.

🎯 Key Takeaways

  • :has() is a genuine "parent selector" (and lookahead selector) — the first CSS mechanism able to select based on descendants/following content, eliminating a whole historical class of JS-required styling logic.
  • :is() takes on its highest-specificity internal argument's specificity; :where() always contributes zero specificity regardless of contents — the real, practical difference determining which to use for resets/utilities versus general selector-list convenience.
  • Native CSS nesting shares Sass's syntax deliberately but runs directly in the browser with zero build step — a fundamentally different mechanism (runtime vs. compile-time) despite the visual similarity.
  • :not()'s modern form accepts a comma-separated selector list directly, replacing the older chained :not(a):not(b) form for improved readability.
  • :has() combined with :not() and sibling combinators can express surprisingly sophisticated conditional-visibility logic (like empty-state indicators) entirely in CSS, with zero JavaScript.
  • Browser selector-matching engines (including :has()) are highly optimized native machinery, generally comparable to or faster than equivalent hand-written JS — general selector-scoping discipline still applies, but should be verified by measurement, not assumed as a blanket concern.
