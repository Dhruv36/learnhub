Layers, :has() & Modern Selectors Advanced

Three platform additions quietly rewired how production CSS is organized: cascade layers end specificity wars by policy, :has() is the parent selector we waited 25 years for, and native nesting retired the main reason teams needed Sass. This lesson covers each deeply — including the sharp edges.

Cascade Layers — Priority by Policy, Not by Accident

/* Declare the order ONCE, up front (first declaration wins the ordering): */
@layer reset, tokens, base, layout, components, utilities;

@layer reset      { * { margin: 0; box-sizing: border-box; } }
@layer base       { h1 { font-size: 2rem; } a { color: var(--color-action); } }
@layer components { .btn { padding: .6rem 1.2rem; background: var(--color-action); } }
@layer utilities  { .hidden { display: none !important; } }

/* Third-party CSS: import INTO a low layer — its wild specificity becomes irrelevant: */
@import url("datepicker-lib.css") layer(vendor);
RuleConsequence
Later layer beats earlier layer, regardless of specificity.hidden (utilities) beats #app .modal .content (components). Specificity now only matters WITHIN a layer
Unlayered styles beat ALL layersBoth your escape hatch and the migration hazard: legacy unlayered CSS silently dominates a new layer system
!important INVERTS layer orderimportant-reset beats important-utilities — by design (earlier layers' importants are "more protected"). Rarely relevant, famously asked
Layers can nest@layer components.buttons — dot-scoped sub-layers for big systems

What this replaces: ITCSS's file-ordering discipline, specificity-hack utilities (.u-hidden.u-hidden), and "load order roulette" with vendor CSS. The architecture lesson formalizes the standard six-layer stack.

:has() — Selecting Upward and Sideways

/* Parent by child: */
.card:has(img)                { padding: 0; }              /* media card variant */
.card:has(> .badge--sale)     { outline: 2px solid gold; } /* direct-child condition */

/* Form UX with ZERO JavaScript: */
.field:has(:user-invalid)         { border-color: #cf222e; }
.field:has(:focus-visible)        { outline: 2px solid var(--color-action); }
label:has(+ input:required)::after { content: " *"; color: #cf222e; }

/* State machines in pure CSS: */
body:has(dialog[open])            { overflow: hidden; }    /* scroll-lock while any modal open */
.form:has([type=checkbox]:checked) .submit { opacity: 1; pointer-events: auto; }
.gallery:has(.item:hover) .item:not(:hover) { opacity: .5; }  /* dim siblings of hovered */

/* Quantity queries — style by COUNT of children: */
.grid:has(> :nth-child(5))        { grid-template-columns: repeat(3, 1fr); }  /* 5+ items → 3 cols */
.tabs:has(> :last-child:nth-child(2)) .tab { flex: 1; }                        /* exactly 2 tabs */

Each line above replaces a JS event listener + class toggle + the bug where they drift apart. Combinator trick: A:has(+ B) selects A when B follows it — effectively a "previous sibling" selector, which CSS never had.

⚡ Performance model: :has() makes the engine re-validate ancestors when descendants mutate — cost scales with the anchor's breadth. .card:has(img): cheap, bounded. body:has(.anything) or *:has(…): re-checks enormous subtrees per mutation. Keep anchors narrow and conditions shallow; it's a scalpel that can be swung like a club.

:is() / :where() — Grouping With Intent

/* Combinatorial explosion, solved: */
:is(.article, .aside, .modal) :is(h2, h3) { margin-block-start: 2rem; }
/* = 6 selectors in one — and one invalid argument doesn't kill the rule (forgiving parsing),
   unlike a plain comma list where one bad selector voids EVERYTHING. */

/* :where() = same matching, ZERO specificity — the library-author tool: */
:where(button, [role="button"]) { cursor: pointer; font: inherit; }
/* any consumer .my-btn overrides this with (0,1,0) — defaults that never fight back */

Specificity recap: :is()/:not()/:has() take their most specific argument; :where() is always (0,0,0). Modern resets (open-props, tailwind preflight) are written in :where() for exactly this reason.

Native Nesting — Sass's Killer Feature, Standardized

.card {
  padding: 1rem;
  border: 1px solid var(--color-border);

  & .title { font-weight: 600; }         /* descendant */
  &:hover  { border-color: var(--color-action); }
  &.featured { background: gold; }        /* compound: .card.featured */
  .sidebar & { padding: .5rem; }          /* REVERSED: .sidebar .card — context override */

  @media (min-width: 48rem) { padding: 1.5rem; }   /* nested at-rules — no selector repetition */
  @container card (min-width: 420px) { flex-direction: row; }
}

Rules that keep nesting healthy: max 2 levels (same discipline as descendant selectors — deep nesting couples CSS to DOM structure and compounds specificity); state/media nesting is the sweet spot; if you're nesting to namespace, you wanted BEM or layers.

Worked Example — Deleting a JS File

/* BEFORE: form.js — 40 lines toggling .has-error, .is-focused, .submit-enabled classes */
/* AFTER: */
.field:has(:user-invalid)                        { border-color: #cf222e; }
.field:has(:user-invalid) .error-msg             { display: block; }
.field:has(:focus-visible)                       { border-color: var(--color-action); }
.form:has(:user-invalid) [type=submit]           { opacity: .5; pointer-events: none; }
/* The JS that remains: actual submission. State PRESENTATION is now the style engine's job —
   which also means it can never desynchronize from the actual input state. */

Interview Questions

Q1: How do cascade layers change what "specificity" means day to day?

Between layers, specificity is irrelevant — order of the @layer declaration decides. So the architecture question moves from "how specific is this selector" to "which layer does this rule belong in". Within a layer, flat-specificity discipline still applies. Unlayered beats layered — the migration trap; !important inverts the order — the trivia trap.

Q2: Why was :has() impossible for so long, and what changed?

Old engines matched selectors right-to-left, upward only — parent-conditional styling would require re-evaluating ancestors whenever any descendant changed, which naive invalidation made pathological. Modern engines built fine-grained invalidation sets that track exactly which :has() conditions depend on which subtree features, making the cost proportional and bounded — hence the "keep anchors narrow" guidance.

Q3: Show three concrete UI behaviors that moved from JS to CSS with :has().

Scroll-locking body while a dialog is open (body:has(dialog[open])); enabling submit only when terms are checked (.form:has(:checked) .submit); wrapper error styling from input validity (.field:has(:user-invalid)). Each removes a listener + class toggle + a desync bug class.

Q4: When would you still choose Sass in 2026?

Loops/mixins generating utility scales, sharing constants into media query conditions (var() can't go there), legacy codebases already on it, and very old browser targets needing compiled output. For theming, nesting, and organization, the platform now covers it — new greenfield projects increasingly skip the preprocessor entirely.

🎯 Key Takeaways

  • Layers: declare order once; later layer wins regardless of specificity; unlayered beats all (migration trap); !important inverts.
  • :has() = parent/previous-sibling/quantity selection — form UX, modals, and state presentation without JS; keep anchors narrow.
  • :is() groups (max-arg specificity, forgiving parsing); :where() ships zero-specificity defaults — the library tool.
  • Native nesting covers Sass's daily value — with the same ≤2-level discipline.
  • :focus-visible, :user-invalid, popover, @scope, ::backdrop — the modern vocabulary auditors and interviews now expect.
  • Feature-detect selectors with @supports selector() and design fallbacks as the default state.
