Backgrounds, Borders & Shadows Foundations

What you'll master: background-image sizing/positioning precisely (the background-size keywords everyone half-remembers), gradients as genuine images (not a separate "gradient" concept), multiple layered backgrounds, border-radius's full 8-value syntax for asymmetric shapes, and box-shadow's real parameter order plus the multi-shadow layering technique professional UI relies on. Closes Foundations. Prerequisite: Typography.

Part 1 — background-image, Sizing, and Positioning

.hero {
  background-image: url("hero.avif");
  background-size: cover;       /* fill the ENTIRE box, cropping overflow — the "hero image" default */
  background-position: center;  /* which part of the (possibly cropped) image is visible */
  background-repeat: no-repeat; /* without this, small images TILE by default — a classic surprise */
}
background-size valueBehaviorUse for
coverScales up until the box is FULLY covered, cropping whatever overflowsHero banners, card thumbnails — never leaves gaps, may crop edges
containScales up until the WHOLE image is visible, may leave empty space (letterboxing)Logos, product images where cropping is unacceptable
100% 100%Stretches to exactly fill the box, ignoring aspect ratioRarely correct — usually distorts the image; avoid unless genuinely intended
a fixed length (200px auto)Exact size, aspect ratio preserved if one axis is autoRepeating background patterns/textures at a specific tile size

The single most common real surprise: background-image tiles (repeats) by default — any image smaller than its container, without an explicit background-repeat: no-repeat, silently repeats to fill the remaining space, which looks like a rendering bug to anyone who hasn't hit it before but is simply the unset default.

Part 2 — Gradients Are Images, Not a Separate Concept

.box {
  background-image: linear-gradient(to right, #1a73e8, #a142f4);
  background-image: linear-gradient(135deg, red 0%, yellow 50%, green 100%);   /* multi-stop, angled */
  background-image: radial-gradient(circle at top left, white, transparent);   /* radiating from a point */
  background-image: conic-gradient(from 0deg, red, yellow, green, blue, red);  /* rotating around a point —
                                                                                   pie charts, color wheels */
}

This is a genuinely important mental model correction: a gradient IS a background-image value — not a separate CSS feature with its own property. This means gradients compose with everything real background-images do: background-size, background-position, and — most usefully — LAYERING several backgrounds together (Part 3), including a gradient stacked ON TOP of a real photo (the extremely common "dark gradient overlay for readable text on a hero photo" pattern).

Part 3 — Multiple Backgrounds: Layering (the Overlay Pattern)

.hero {
  background-image:
    linear-gradient(to bottom, rgb(0 0 0 / 0), rgb(0 0 0 / 0.7)),   /* TOP layer: dark gradient */
    url("hero-photo.avif");                                          /* BOTTOM layer: the photo */
  background-size: cover, cover;      /* one value PER layer, comma-separated, same order */
  background-position: center, center;
}

Comma-separated background-image values stack as LAYERS, first-listed ON TOP — this is precisely the mechanism behind "photo with a dark gradient fade at the bottom so white heading text stays readable," one of the single most common real-world hero-section patterns, achieved with ZERO extra markup (no separate overlay <div> needed) — every other background property (background-size, -position, -repeat) accepts the SAME comma-separated, per-layer syntax, matched up by position in the list.

Part 4 — border-radius: the Full 8-Value Syntax

.box { border-radius: 8px; }                          /* all 4 corners equal */
.box { border-radius: 8px 16px; }                     /* top-left/bottom-right, top-right/bottom-left */
.box { border-radius: 8px 16px 24px 4px; }            /* TL, TR, BR, BL — clockwise from top-left */
.box { border-radius: 50%; }                          /* a perfect circle/ellipse (on a square/rect box) */

/* Each corner ALSO accepts two values for a genuinely ELLIPTICAL (non-circular) corner curve: */
.blob {
  border-radius: 60% 40% 30% 70% / 60% 30% 70% 40%;   /* horizontal radii / vertical radii —
                                                           the "organic blob shape" technique */
}

The slash syntax (horizontal-radii / vertical-radii) is the genuinely underused, powerful part — it lets each corner curve DIFFERENTLY on its horizontal vs. vertical axis, which is exactly the mechanism behind the "organic, irregular blob shape" decorative pattern seen throughout modern illustration-heavy marketing sites — achieved with pure CSS on a single element, no SVG or image asset required.

Part 5 — box-shadow: Parameter Order and Layering

.card {
  box-shadow: 0 4px 6px rgb(0 0 0 / 0.1);
  /*           ^x  ^y  ^blur  ^color
     x-offset, y-offset, blur-radius, (optional spread-radius), color */
}
.card-inset {
  box-shadow: inset 0 2px 4px rgb(0 0 0 / 0.2);   /* INSET flag — shadow INSIDE the box, not outside */
}

Parameter order is fixed and easy to misremember: x-offset y-offset blur-radius spread-radius color (spread is optional and less commonly used — it expands/contracts the shadow's size independent of blur). The inset keyword, placed first, flips the shadow to render INSIDE the element's border edge instead of outside — useful for pressed/active button states and inner-shadow depth effects.

.card {
  box-shadow:
    0 1px 2px rgb(0 0 0 / 0.06),     /* tight, subtle shadow — close to the surface */
    0 4px 8px rgb(0 0 0 / 0.08),     /* medium shadow — mid-depth */
    0 12px 24px rgb(0 0 0 / 0.10);   /* soft, wide shadow — ambient depth */
}

Comma-separated box-shadow values LAYER exactly like backgrounds (Part 3) — and this specific multi-shadow technique (several shadows of increasing blur/offset and DEcreasing opacity, stacked) is precisely how professional design systems (Material Design's elevation system, most modern UI kits) achieve a convincingly soft, realistic depth effect that a SINGLE shadow value never quite replicates — real-world light doesn't cast one crisp shadow, it casts a soft gradient of overlapping penumbra, and layering multiple shadow values is the practical CSS approximation of that.

Interview Questions

Q1 (beginner): Why does a small background-image sometimes appear to repeat/tile across an element unexpectedly?

background-repeat defaults to repeating the image to fill any remaining space beyond its natural size — this is the unset default, not a bug. Any background-image smaller than its container needs an explicit background-repeat: no-repeat if tiling isn't the deliberate intent.

Q2 (beginner): What's the practical difference between background-size: cover and contain?

cover scales the image UP until it fully fills the box, cropping whatever overflows the box's edges — never leaves empty space, may crop image content. contain scales up until the WHOLE image is visible within the box, potentially leaving empty (letterboxed) space if the box's aspect ratio doesn't match the image's — never crops, may show gaps.

Q3 (mid): How would you create a "dark gradient overlay for readable text on a hero photo" effect without adding any extra HTML elements?

A single background-image property with TWO comma-separated values: a linear-gradient() listed first (rendering as the TOP layer) stacked over a url() of the photo (the bottom layer) — since gradients are themselves background-image values, they layer with real images using the exact same comma-separated multi-background syntax, requiring zero additional overlay markup.

Q4 (mid): Explain box-shadow's parameter order and what the inset keyword changes.

Order is x-offset, y-offset, blur-radius, an optional spread-radius, then color. inset, placed first in the value, flips the shadow to render INSIDE the element's border edge (an inner-shadow/ pressed-button effect) instead of the default outside-the-box rendering — useful for active/ pressed states and inset depth effects that shouldn't extend beyond the element's own boundary.

Q5 (senior): A design system's cards use a single box-shadow: 0 4px 8px rgba(0,0,0,0.15) for "elevation," but designers report the shadows look flat and artificial compared to reference designs from Material Design or similar systems. Diagnose and propose a fix.

Diagnosis: a single box-shadow value produces one crisp, uniform-opacity shadow shape — real-world ambient light produces a SOFT, gradually-diffusing penumbra that no single shadow value can replicate, since a single value has one fixed blur radius and one fixed opacity throughout. Professional elevation systems (Material Design explicitly documents this) achieve convincing depth by LAYERING 2-3 shadow values with increasing blur/offset and DEcreasing opacity — a tight, higher-opacity shadow close to the surface, plus a soft, lower-opacity, more-spread shadow further out, approximating how real light actually falls off. Fix: replace the single shadow with a layered set, e.g. box-shadow: 0 1px 2px rgb(0 0 0 / 0.06), 0 4px 8px rgb(0 0 0 / 0.08), 0 12px 24px rgb(0 0 0 / 0.10) — and if the design system defines multiple ELEVATION LEVELS (card, modal, dropdown, each needing progressively "higher" shadows), each level should get its OWN layered shadow set with proportionally larger offsets/blur/spread, exactly mirroring how Material Design's numbered elevation tokens work, rather than simply scaling one single shadow value's numbers up per level (which reproduces the same "flat" problem at every tier, just at different sizes).

Q6 (senior): Your team wants a reusable "frosted glass" card effect (a translucent, blurred background revealing content behind it, like iOS's control center) achieved purely in CSS. Design the approach, combining concepts from this lesson and note any browser-support caveat worth flagging.

The core technique combines a semi-transparent background-color (an rgba/rgb-with-alpha value, e.g. background: rgb(255 255 255 / 0.6)) with the backdrop-filter property (specifically backdrop-filter: blur(12px)) — the semi-transparent background lets whatever is BEHIND the element show through partially, while backdrop-filter blurs specifically that background content (not the card's own content), producing the frosted-glass effect. A layered box-shadow (per this lesson's Part 5 technique) and a subtle border (often a semi-transparent white border, border: 1px solid rgb(255 255 255 / 0.3)) typically complete the effect, adding the soft edge definition real frosted glass has. Browser-support caveat worth flagging explicitly to the team: backdrop-filter, while broadly supported in current browsers, has historically had inconsistent support/required vendor prefixes on some engines, and — more practically important — it can carry a real, measurable PAINT-stage performance cost (the HTML track's Critical Rendering Path lesson's Stage 5) on lower-end devices when applied to many elements or animated/scrolled frequently, since blurring is genuinely more compute-intensive than a flat background-color; recommend using it deliberately on a small number of prominent UI elements (a nav bar, a modal) rather than broadly across many repeated list items, and testing on a representative lower-end device before shipping at scale, exactly the same "measure on the actual target device" discipline the Performance lesson established for other CSS/rendering choices.

🏋️ Exercises

1 (easy): cover vs contain, compared. Build two identical 200×200px boxes with the same non-square image as background-image, one with background-size: cover, one with contain. Observe and describe the difference (cropping vs. letterboxing).
Expected observation

The cover box should show the image filling the entire 200×200 area with some content cropped off (likely the top/bottom or sides, depending on the image's original aspect ratio). The contain box should show the ENTIRE image, with visible empty space (the box's own background color, if any, or transparent) on two sides where the image's aspect ratio doesn't match the square container.

2 (easy): Gradient overlay, no extra markup. Build a hero section with a real background photo and a bottom-to-top dark gradient overlay ensuring white heading text stays readable, using ONE background-image property with two comma-separated values — no additional div.
Solution 
.hero {
  background-image:
    linear-gradient(to top, rgb(0 0 0 / 0.75), rgb(0 0 0 / 0));
    url("photo.avif");
  background-size: cover;
  color: white;
}

Correction — note both values need to be part of ONE comma-separated background-image declaration: background-image: linear-gradient(...), url(...); on a single line. Confirm the heading text at the bottom of the hero (where the gradient is darkest) remains clearly readable against the photo.

3 (medium): The organic blob shape. Using border-radius's slash syntax, create an irregular, organic-looking blob shape on a colored div (no image, no SVG). Try at least 3 different value combinations and pick the one that looks most convincingly "non-geometric."
Example 
.blob {
  width: 300px; height: 300px;
  background: hsl(217 83% 60%);
  border-radius: 63% 37% 54% 46% / 55% 48% 52% 45%;
}

Small tweaks to any of the 8 values should visibly shift the shape's "personality" — this hands-on experimentation is the fastest way to develop real intuition for how the slash syntax's two axes interact, faster than reasoning about it purely from the property's abstract definition.

4 (medium): Layered elevation shadows. Build 3 cards representing 3 elevation levels (e.g., "resting," "hovered," "dragging") each with its OWN layered box-shadow set (2-3 shadows each), where higher elevation levels have progressively larger offset/blur/spread values across their layered shadows. Compare against a version using a single shadow value scaled up per level, and note which looks more convincing.
Example scale 
.elevation-1 { box-shadow: 0 1px 2px rgb(0 0 0 / .06), 0 2px 4px rgb(0 0 0 / .08); }
.elevation-2 { box-shadow: 0 2px 4px rgb(0 0 0 / .08), 0 6px 12px rgb(0 0 0 / .10); }
.elevation-3 { box-shadow: 0 4px 8px rgb(0 0 0 / .10), 0 16px 32px rgb(0 0 0 / .14); }

The layered, progressively-scaled version should look noticeably more convincing as genuine "lifting off the page" depth than a single-shadow-value-scaled-up version at the same three levels — a direct, side-by-side confirmation of Q5's interview answer.

5 (hard): Frosted glass card. Build Q6's complete frosted-glass effect: a card with semi-transparent background-color, backdrop-filter: blur(), a subtle semi-transparent border, and a layered box-shadow, positioned OVER a colorful background image or gradient so the blur effect is actually visible. Test in at least two different browsers if possible and note any rendering differences.
Solution 
.frosted {
  background: rgb(255 255 255 / 0.55);
  backdrop-filter: blur(14px);
  border: 1px solid rgb(255 255 255 / 0.35);
  box-shadow: 0 8px 24px rgb(0 0 0 / 0.12), 0 2px 6px rgb(0 0 0 / 0.08);
  border-radius: 16px;
}

The card should show the background behind it visibly BLURRED specifically within the card's own bounds, while content placed directly on the card remains crisp — confirming backdrop- filter targets only what's BEHIND the element, not the element's own contents, exactly as Q6 described.

6 (hard): Full hero section, assembled. Combine everything from this lesson into one complete, realistic hero section: a background photo with a layered gradient overlay (dark at the bottom, fading to transparent at the top), a heading with a layered text-shadow or box-shadow'd card containing the heading text, and border-radius applied to any card elements. Verify readability of all text against the underlying photo at multiple scroll/viewport positions.
What "complete" verification looks like

Every text element should remain clearly readable against its specific background region — this is the actual functional test of whether the gradient overlay's darkness/positioning was tuned correctly, not just whether the individual CSS properties are syntactically present. If any text becomes hard to read at a specific viewport width (the gradient's darkest region shifting relative to where the text renders, due to responsive reflow), that's a genuine bug this exercise is designed to surface — adjust the gradient's stops or the text's positioning until it holds up across the range you test.

🎯 Key Takeaways

  • background-image tiles by default — always set background-repeat: no-repeat unless tiling is the deliberate intent; cover crops to fill, contain letterboxes to show everything.
  • Gradients ARE background-image values, not a separate feature — they compose with background-size/-position and, critically, LAYER with real images via comma-separated multi-background syntax (the zero-markup overlay pattern).
  • border-radius's full 8-value slash syntax (horizontal-radii / vertical-radii per corner) enables genuinely organic, non-circular shapes with pure CSS.
  • box-shadow's parameter order is x-offset, y-offset, blur, (spread), color — inset flips it to render inside the element.
  • Layering 2-3 box-shadow values with increasing offset/blur and decreasing opacity approximates real ambient light's soft penumbra — the actual technique behind professional "elevation" design systems, versus a single flat shadow value.
  • backdrop-filter + a semi-transparent background achieves frosted-glass effects natively, but carries a real paint-stage performance cost worth testing on representative devices before broad use.
