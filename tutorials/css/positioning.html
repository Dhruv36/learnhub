Positioning Basics

Normal flow lays boxes top-to-bottom; flex and grid arrange them; positioning is for the exceptions — elements that float above, pin to edges, or stick on scroll: badges, dropdowns, tooltips, sticky headers, modals. The concepts that make it predictable (and that interviews probe): containing blocks and stacking contexts.

The Five Values, Precisely

ValueIn flow?Offsets relative to…Canonical use
staticYes (default)— (top/left/z-index ignored)Everything normal
relativeYes — original space is KEPTIts own normal position (visual shift only)Anchor for absolute children; nudges; enabling z-index
absoluteNo — removed, space collapsesNearest positioned ancestor's padding boxBadges, dropdowns, tooltips, overlays
fixedNoThe viewport… unless an ancestor has transform/filter (see trap)Chat bubbles, cookie bars, app headers
stickyYes until thresholdNearest scrollport; pins at its offset, bounded by its PARENTTable headers, section nav, this site's sidebar

Containing Blocks — Where "Relative To" Comes From

/* The pattern you'll write 1000 times: */
.avatar-wrapper { position: relative; }           /* 1. make the parent the containing block */
.notification-dot {
  position: absolute;                              /* 2. child positions against it */
  top: -4px; right: -4px;
  width: 12px; height: 12px; border-radius: 50%;
  background: #cf222e;
}

Skip step 1 and the dot anchors to the next positioned ancestor up the tree — often none, so it lands relative to the page and appears in a corner. The #1 positioning bug in existence.

⚠️ The fixed-position trap: any ancestor with transform, filter, perspective, or certain will-change/contain values becomes the containing block even for fixed descendants. Symptom: someone adds a subtle transform animation to a page wrapper and every "fixed" element starts scrolling with the page. Diagnosis: walk up the tree in DevTools looking for transforms.

Absolute Sizing & Centering Recipes

/* Fill the containing block (overlay): */
.overlay { position: absolute; inset: 0; }                       /* inset = top/right/bottom/left */

/* Pin both edges = implicit width (no width property needed): */
.toast-region { position: fixed; inset-inline: 1rem; bottom: 1rem; }

/* Center something of unknown size: */
.pop { position: absolute; top: 50%; left: 50%; transform: translate(-50%, -50%); }
/* translate % = own size — that's the whole trick */

/* Center with margins (also works, needs inset + auto): */
.pop2 { position: absolute; inset: 0; margin: auto; width: fit-content; height: fit-content; }

Absolutely positioned boxes shrink-wrap their content by default; give width explicitly or pin opposite edges.

Sticky — the Rules That Make It "Not Work"

.toc { position: sticky; top: 1rem; }   /* needs: an offset, and freedom to travel */
Sticky requirementThe failure when violated
An offset (top/bottom/…)No threshold → behaves as relative, never pins
No overflow: hidden/auto/scroll ancestor between it and the scrollerThe overflow ancestor becomes its scrollport — it "sticks" inside a box that doesn't scroll → appears dead. The classic sticky bug
A parent TALLER than itselfSticky never leaves its parent — a full-height parent means no travel range

Sticky table headers: thead th { position: sticky; top: 0; } inside the scrolling wrapper — put backgrounds on the th (rows scroll beneath) and use border-collapse: separate to keep borders attached.

z-index & Stacking Contexts — the Senior-Level Half

/* The bug report: "modal (z-index: 9999) renders UNDER the header (z-index: 10)" */
.page-section { transform: translateZ(0); }   /* ← someone's 'performance fix' created a context */
.page-section .modal { z-index: 9999; }        /* competes only INSIDE .page-section */
.header { z-index: 10; }                        /* competes at page level → wins */

The model: a stacking context is a sealed bag. Children's z-indexes — however large — are resolved inside the bag; the bag competes with the outside world using its root's own stacking level. No value inside can escape.

Creates a stacking contextNotes
positioned element + z-index ≠ autoThe classic, intentional one
opacity < 1, transform, filter, backdrop-filter, perspective, clip-path, maskThe accidental ones — a fade-in animation can re-layer a page
will-change of those, contain: paint, mix-blend-modeThe obscure ones
isolation: isolateThe DELIBERATE one — create a context on purpose to cap a component's internals
flex/grid children with z-indexYes — even without position!

Paint order without z-index: backgrounds → non-positioned blocks → floats → inline content → positioned elements in DOM order (later on top). That's why a later sibling covers an earlier one with no z-index anywhere.

Escaping the Game Entirely: the Top Layer

<dialog>…</dialog>            dialog.showModal()   → renders in the TOP LAYER
<div popover>…</div>          + popovertarget      → same, with light-dismiss

The top layer sits above every stacking context by specification — no z-index can compete. Modern modals, popovers, and toasts use these primitives (plus ::backdrop) instead of playing the numbers game; frameworks' "portals" solve the same problem by re-parenting DOM. For tethered tooltips/dropdowns, CSS anchor positioning (anchor-name/ position-anchor) is arriving to replace measure-and-place JavaScript.

The z-index Scale (production hygiene)

:root {
  --z-dropdown: 100;  --z-sticky: 200;  --z-overlay: 300;
  --z-modal: 400;     --z-toast: 500;
}
.dropdown { z-index: var(--z-dropdown); }   /* raw numbers banned by stylelint */

Five tiers cover almost any product. The scale turns "what number beats what" from archaeology into policy — every mature codebase adopts one after its first z-index: 999999 incident.

Interview Questions

Q1: Explain stacking contexts to a junior who just wrote z-index: 999999.

z-index competes only among siblings within the same stacking context. A parent with transform/opacity/filter creates a sealed context: children are layered inside it, then the whole group competes externally at the parent's level. So 999999 inside a context under the header loses to the header's 10. Fixes: lift the element out, remove the context creator, use the top layer, or restructure with the z-scale.

Q2: relative vs absolute — what happens to the space each occupied?

relative keeps its slot in flow (neighbors don't move; the shift is purely visual — even its own original space is preserved). absolute is removed entirely; siblings reflow as if it never existed. That's why relative works for nudges and absolute for overlays.

Q3: Why did the platform add the top layer, and what used it first?

Because no z-index value can guarantee "above everything" once arbitrary stacking contexts exist — libraries fought each other forever. The top layer is spec-level "always on top": fullscreen API first, then <dialog>.showModal() and the popover attribute. It's why native modals need no z-index management at all.

Q4: Your sticky sidebar works locally but not in production. Name two likely culprits.

(1) A wrapper gained overflow-x: hidden (often added to clamp a horizontal-scroll bug) — killing sticky inside it. (2) The parent's height now equals the sidebar's (e.g., a grid change), leaving no travel range. Both are structural, not sticky syntax.

🏋️ Practice (Playground):
1. Avatar + notification dot: build it, delete the parent's position:relative, watch the dot fly away, restore it.
2. Reproduce the modal-under-header bug: header z-index 10; wrapper with transform; modal z-index 9999 inside. Fix it two ways (remove transform; use <dialog>).
3. Sticky section headings in a scrolling list (top: 0) — then add overflow: hidden to a wrapper and watch sticky die; remove it.
4. Build a centered tooltip with the translate(-50%,-50%) recipe on hover of a button.

🎯 Key Takeaways

  • absolute needs an anchored parent (relative) — the containing block decides what offsets mean.
  • transform/filter on an ancestor hijacks even fixed positioning — the classic "fixed stopped working".
  • Sticky needs an offset, a scroll path free of overflow ancestors, and parent room to travel.
  • z-index lives inside stacking contexts (created by z-index AND by opacity/transform/filter…); children can't escape; isolation:isolate creates one on purpose.
  • dialog/popover render in the top layer — above all contexts, ending the numbers game for modals.
  • Adopt a 5-tier z-index token scale; ban raw numbers.
