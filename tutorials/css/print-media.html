Print & Other Media Production

What you'll master: @media print and what genuinely needs to change for a printed page, the @page rule for margins/page-breaks, the surprisingly useful trick of exposing link URLs in printed output, and — the track's final, closing lesson — a genuine reflection on how nearly everything taught across this entire 28-lesson CSS curriculum applies directly to this one, final, different medium. This closes the entire CSS Mastery v4 track. Prerequisite: Accessibility in CSS; this is the CSS track's final lesson.

Part 1 — @media print: What Genuinely Needs to Change

@media print {
  nav, .sidebar, .ads, .video-player, button, .no-print { display: none; }
  body { font-size: 12pt; color: black; background: white; }
  a { color: black; text-decoration: none; }
  .card { box-shadow: none; border: 1px solid #ccc; }   /* shadows are meaningless on paper;
                                                             a real border reads clearly instead */
}

A genuinely useful mental exercise, directly connecting to this whole track's recurring "what does this element ACTUALLY need" discipline: printed output has NO interactivity (hover states, buttons, video players, navigation menus are all functionally meaningless on paper), often has LIMITED or no color reproduction (many printers default to grayscale), and typically wastes precious paper on anything that isn't the actual CONTENT a user wanted to print in the first place. The @media print block's job is systematically stripping away everything that only made sense in the interactive, on-screen context — navigation, ads, video embeds, buttons, decorative shadows — leaving genuinely just the CONTENT, appropriately restyled for a static, physical page.

Part 2 — @page: Margins and Page-Break Control

@page {
  margin: 2cm;              /* real, physical page margins — a genuinely different concept
                                from any on-screen margin/padding */
}
.article-section { break-inside: avoid; }    /* don't let this section's content SPLIT
                                                  awkwardly across two physical pages */
h2 { break-after: avoid; }                    /* don't leave a heading stranded ALONE at the
                                                  very bottom of a page, with its actual
                                                  content beginning on the NEXT page */
.chapter { break-before: page; }              /* always start this element on a genuinely
                                                  NEW physical page */

@page is a genuinely distinct, print-specific at-rule controlling the physical page itself (margins measured in real, physical units — cm, in — since a printed page has an actual physical size, unlike a screen's variable viewport). break-inside/break-before/break-after (the modern, standardized versions of the older page-break-* properties) solve genuinely real, physical-printing problems no on-screen layout ever has to consider: a table or image awkwardly split across two separate printed pages, or a heading stranded alone at a page's bottom with its actual content beginning on the following page — both real, jarring, and entirely avoidable printed-output annoyances.

Part 3 — The Underused, Genuinely Clever Trick: Exposing Link URLs

@media print {
  a[href^="http"]::after {
    content: " (" attr(href) ")";    /* the ATTR() function, met in the HTML track's lesson 3 —
                                          pulls the actual href value directly into printed
                                          content */
    font-size: 0.8em;
    color: #555;
  }
}

This is a genuinely clever, real, and surprisingly underused technique directly reusing the Selectors lesson's ::after generated-content mechanism, combined with the attr() function: on a printed, physical page, a hyperlink's underlying destination URL is otherwise completely INVISIBLE and unusable (there's no cursor to hover, no click to follow) — this rule makes every link's actual destination genuinely READABLE directly in the printed text itself, which is precisely the kind of small, thoughtful, medium-appropriate detail that separates a print stylesheet that was actually THOUGHT ABOUT from one that merely hid the navigation and called it done.

Part 4 — Beyond Print: Other Media Types and Features

@media screen { }    /* the IMPLICIT default for all normal, on-screen browser rendering —
                          rarely needs to be written explicitly, since it's the assumed default */
@media speech { }     /* for screen-reader/speech-synthesis rendering specifically — a real,
                          if genuinely rare-to-actually-use-in-practice, media type */
@media (scripting: none) { }   /* detects whether JAVASCRIPT is genuinely disabled/unavailable —
                                   directly enabling the HTML track's progressive-enhancement
                                   philosophy to be expressed IN CSS itself: show a "this feature
                                   needs JavaScript" message ONLY when scripting is truly absent */
@media (hover: hover) and (pointer: fine) { }   /* detects a genuine MOUSE-like input (hover
                                                    capability + precise pointing) versus a
                                                    touch-only device — lets hover-dependent
                                                    UI patterns be conditionally applied ONLY
                                                    where hover genuinely, meaningfully exists */

Two of these deserve real, specific attention beyond print: (scripting: none) is a genuinely powerful, underused feature letting CSS ITSELF detect the HTML track's own progressive-enhancement scenario (JavaScript genuinely unavailable) — showing a graceful "this feature requires JavaScript" fallback message purely through CSS, with zero script needed to detect and display it (since if scripting were available, the script itself could hide such a message; if it's NOT available, the CSS-only detection is precisely what's needed instead). (hover: hover) and (pointer: fine) solves a real, common, modern problem: touchscreen devices technically CAN trigger a CSS :hover state (via a tap), but that state then often gets confusingly "stuck" with no mouse to move away and naturally clear it — gating hover-dependent UI effects behind a genuine hover-capability check prevents this specific, real, sticky-hover-state confusion on touch devices.

Interview Questions

Q1 (beginner): What kinds of elements typically get hidden in a print stylesheet, and why?

Navigation menus, buttons, video players, ads, and other interactive/decorative chrome — everything that only makes sense in an interactive, on-screen context. Printed output has no hover/click capability and often limited color reproduction, so a good print stylesheet strips away anything that isn't the genuine, static CONTENT a user actually wanted to print, avoiding wasted paper and ink on functionally meaningless elements.

Q2 (beginner): What does the attr(href) trick in a print stylesheet accomplish?

It uses ::after generated content combined with the attr() function to pull a link's actual href URL directly into the printed text itself — since a physical page has no clickable links or hover tooltips, this makes an otherwise completely invisible, unusable destination URL genuinely readable in the printed output, a small but thoughtful, medium-appropriate detail.

Q3 (mid): Why would you use (hover: hover) and (pointer: fine) rather than just checking for touch support directly?

Touchscreens can technically trigger CSS :hover via a tap, but the hover state can then get confusingly "stuck" with no mouse to move away and naturally clear it. (hover: hover) and (pointer: fine) together detect a genuine mouse-like input capability (actual hover support plus precise pointing), letting hover-dependent UI effects apply ONLY where hover genuinely, meaningfully makes sense as an interaction model — avoiding the sticky-hover-state problem on touch-primary devices.

Q4 (mid): How does the (scripting: none) media feature relate to the HTML track's progressive-enhancement philosophy?

It lets CSS itself detect whether JavaScript is genuinely unavailable, enabling a graceful "this feature requires JavaScript" fallback message to be shown purely through CSS with zero script needed — precisely because if scripting WERE available, the script itself could handle hiding such a message, and if it's genuinely NOT available, CSS-only detection is exactly the mechanism needed instead. It's a direct, CSS-native expression of the same base-then-enhance philosophy the HTML track established for HTML/JS.

Q5 (senior): Your company's documentation site gets frequent support tickets about "the printed PDF version looks broken — tables are cut in half across pages, and code blocks are unreadable because they're printed in light gray on white." Diagnose and propose a complete print stylesheet fix.

Diagnosis, addressing two distinct, real problems: (1) tables cut across pages — missing break-inside: avoid on table rows (or the whole table, if short enough to reasonably fit on one page) allows the browser's default page-breaking algorithm to split a table wherever it happens to run out of vertical space, mid-row, producing exactly the reported "cut in half" symptom; (2) light-gray-on-white code blocks — the documentation's on-screen theme almost certainly uses a dark or colored background for code blocks (a common syntax-highlighting pattern) with light text designed for THAT background — but if no print-specific override exists, the code block's BACKGROUND likely gets stripped/ignored by many printers' default behavior (backgrounds are commonly NOT printed by default in many browsers/print drivers, specifically to save ink) while the light TEXT color remains, producing exactly the reported illegible light-on-white result. Complete fix: (1) add break-inside: avoid to table rows/short tables in @media print; (2) explicitly override code block colors for print — pre, code { background: #f5f5f5 !important; color: #1a1a1a !important; border: 1px solid #ddd; } — forcing genuinely legible, dark-text-on- light-background rendering regardless of the on-screen theme's syntax-highlighting colors (the !important here is a legitimate, narrow, print-specific use — overriding potentially very specific syntax-highlighter-generated inline styles or highly-specific selectors that would otherwise be hard to out-specificity in a simple print override). Testing methodology worth recommending: use the browser's own "Print Preview" (not just visually eyeballing the screen version) as the actual verification tool for both fixes, since print rendering has real, browser-specific quirks (like the default background-stripping behavior) that only become visible in an actual print preview, not in normal on-screen inspection.

Solution + verification
