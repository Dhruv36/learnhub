CSS Grid Layout

What you'll master: the two-dimensional mental model that resolves flexbox's uneven-row limitation from the previous lesson, the fr unit and grid-template-columns/rows syntax, named grid areas for genuinely readable layout code, auto-fill/auto-fit with minmax() as the true zero-media-query responsive grid, and explicit vs implicit grid tracks. Prerequisite: Flexbox — this lesson directly resolves the last lesson's closing "uneven last row" problem.

Part 1 — Two Dimensions, Genuinely Simultaneous

.grid { display: grid; }   /* direct children become GRID ITEMS, placed into a true 2D
                                row-AND-column structure the container defines up front */

Recall last lesson's closing diagnosis: flexbox's flex-wrap distributes space independently PER ROW, with zero coordination across rows — a structural limitation, not a bug, because flexbox is fundamentally one-dimensional. Grid exists specifically to solve the case where rows genuinely need to share consistent column boundaries: you define the column (and row) TRACKS once, on the container, and every row automatically aligns to those same tracks — a partial last row's items snap to the identical column widths as full rows above, because there's only ONE set of column definitions governing the entire grid, not one independent decision per row.

Part 2 — grid-template-columns/rows and the fr Unit

.grid {
  display: grid;
  grid-template-columns: 200px 1fr 1fr;   /* fixed 200px sidebar, then TWO equal flexible columns */
  grid-template-rows: 80px auto 60px;      /* fixed header, flexible middle, fixed footer */
  gap: 20px;
}

fr ("fraction unit") is grid's own flexible-sizing unit — conceptually similar to flexbox's flex-grow, but operating on GRID TRACKS rather than individual items: 1fr 1fr splits remaining space (after fixed-size tracks like the 200px sidebar are subtracted) into two EQUAL shares; 1fr 2fr splits it into a 1:2 ratio. Crucially, fr distributes space AFTER content's minimum required size and any fixed tracks are already accounted for — genuinely different from a percentage, which has no such content-aware floor.

.grid {
  grid-template-columns: repeat(3, 1fr);         /* shorthand for 1fr 1fr 1fr — "3 equal columns" */
  grid-template-columns: repeat(4, minmax(0, 1fr));  /* 4 equal columns that CAN shrink to 0 basis,
                                                          preventing content from forcing overflow */
}

repeat() avoids writing out repetitive track lists by hand; combined with minmax() (a track that's never smaller than its first argument, never larger than its second), this pairing becomes the standard, robust way to define equal-width columns that also correctly handle unexpectedly-long content inside any single cell without breaking the whole grid's proportions.

Part 3 — Named Grid Areas: Layout That Reads Like a Diagram

.page {
  display: grid;
  grid-template-columns: 200px 1fr;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "sidebar header"
    "sidebar main"
    "sidebar footer";
}
.sidebar { grid-area: sidebar; }
.header  { grid-area: header; }
.main    { grid-area: main; }
.footer  { grid-area: footer; }

This is grid's most genuinely readable feature: grid-template-areas lays out the ENTIRE page structure as a literal, visual ASCII diagram directly in the CSS — "sidebar header" reads exactly as it looks, a sidebar spanning the full height beside a header/main/footer stack. Any team member (or your own future self) can understand the whole page's structural layout by reading this ONE declaration, without mentally tracing numbered column/row line positions — a genuine, significant readability win over both flexbox's implicit ordering and grid's own numbered-line placement syntax (grid-column: 1 / 3, functionally equivalent but considerably less self-documenting for complex layouts).

Part 4 — auto-fill/auto-fit + minmax(): the True Zero-Media-Query Grid

.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
  gap: 20px;
}

This is grid's direct, superior answer to flexbox's flex: 1 1 250px pattern from the previous lesson — and it fixes that pattern's exact uneven-last-row flaw. Reading it precisely: "fit as many columns as possible, each at LEAST 250px wide, and let them grow (1fr) to fill any remaining space evenly." Because grid defines these tracks ONCE for the entire container (not per-row, unlike flex-wrap), EVERY row — including a partial last row — aligns to the identical column widths; a last row with 2 items snaps to exactly 2 of the SAME-width columns that exist in the fuller rows above, rather than stretching to fill the whole row's width the way flexbox's per-row redistribution did.

auto-fill vs auto-fit — the one genuinely subtle, frequently-confused distinction in this pattern: both create as many columns as currently fit; they differ ONLY when there are fewer actual items than columns that COULD fit. auto-fill keeps the extra, empty column tracks reserved (items stay their minmax-minimum size, with visible empty space to the right). auto-fit COLLAPSES those empty tracks to zero width, letting the actual existing items' 1fr stretch to fill the freed-up space instead. For a typical card grid where you want existing cards to expand and fill the row rather than leaving an awkward empty gap when there aren't enough cards to fill it, auto-fit is almost always the desired, correct choice.

Part 5 — Explicit vs Implicit Grid: What Happens With Extra Items

.grid {
  grid-template-columns: repeat(3, 1fr);   /* the EXPLICIT grid — 3 columns, deliberately defined */
  grid-auto-rows: 150px;                    /* controls the size of any IMPLICIT rows auto-created */
}
/* 10 items in a 3-column explicit grid → 4 ROWS are created automatically (the 4th only partially
   filled) — these auto-created rows are the IMPLICIT grid, and grid-auto-rows controls their sizing,
   completely independent from grid-template-rows (which only governs EXPLICITLY defined rows) */

The explicit grid is what you deliberately define via grid-template-columns/ -rows; the implicit grid is whatever ADDITIONAL rows/columns the browser must create to accommodate items beyond what the explicit definition accounts for — a real, distinct concept with its OWN sizing controls (grid-auto-rows/grid-auto-columns), separate from the explicit grid's own track sizing. This distinction explains why setting grid-template-rows: 100px 100px has NO effect on a grid's 5th, 6th, 7th... row if only 2 rows were explicitly defined — those later rows are implicit, and need grid-auto-rows to control their height instead.

Part 6 — Deep Dive: Grid Line-Based Placement (When Named Areas Aren't Enough)

.featured-item {
  grid-column: 1 / 3;    /* span from grid LINE 1 to grid LINE 3 — occupying 2 columns' worth */
  grid-row: 2 / 4;        /* span from line 2 to line 4 — 2 rows tall */
}
.featured-item {
  grid-column: span 2;    /* shorthand: "span 2 columns from wherever auto-placement puts me" */
}

Grid lines are numbered starting at 1, running along BOTH edges of every track (so a 3-column grid has 4 vertical lines: before column 1, between 1&2, between 2&3, and after column 3) — genuinely useful for items that need to SPAN multiple tracks (a featured product card taking up 2×2 cells in an otherwise uniform grid) beyond what named areas conveniently express for irregular, ad-hoc spanning needs. Named areas (Part 3) remain the more readable choice for a page's OVERALL, relatively fixed structural regions; line-based placement is the right tool for items with irregular, data-driven, or dynamically-varying spans within an otherwise regular grid.

Interview Questions

Q1 (beginner): What's the fundamental difference between flexbox and grid, in terms of dimensionality?

Flexbox is one-dimensional — it arranges items along a single row or column, with each wrapped row/column making independent space-distribution decisions. Grid is genuinely two-dimensional — you define row AND column tracks together, up front, on the container, and every item (across every row) aligns consistently to that same shared track structure, which is precisely what resolves flexbox's per-row-independent distribution limitation for true grid-shaped content.

Q2 (beginner): What does the fr unit represent, and how is it different from a percentage?

fr ("fraction unit") represents a share of the space REMAINING after fixed-size tracks and content-minimum sizes are already accounted for — 1fr 2fr splits remaining space into a 1:2 ratio. Unlike a percentage (which has no content-aware floor and is calculated against the total container size regardless of other tracks), fr specifically composes correctly alongside fixed px tracks and content minimums in the same track list.

Q3 (mid): Explain the difference between auto-fill and auto-fit in repeat(auto-fit/fill, minmax(250px, 1fr)), specifically when there are fewer items than columns that could fit.

Both create as many column tracks as currently fit at the minmax minimum width — they behave identically when there are ENOUGH items to fill every possible column. They differ only when there are FEWER items: auto-fill keeps the extra, unused column tracks reserved (existing items stay at their minimum size, with visible empty space where the unused tracks would be); auto-fit collapses those empty tracks to zero width, letting the existing items' 1fr stretch to fill the freed space instead. auto-fit is almost always the desired choice for a card grid, so existing cards expand rather than leaving an awkward gap.

Q4 (mid): A grid explicitly defines grid-template-rows: 100px 100px 100px (3 rows), but the container has 12 items in a 4-column layout, producing a 4th row. That 4th row's height doesn't match the specified 100px. Diagnose.

The 4th row is part of the IMPLICIT grid — created automatically to accommodate items beyond what the explicit 3-row definition accounts for. grid-template-rows only controls EXPLICITLY defined rows; implicit, auto-created rows are sized by the separate grid-auto-rows property (which defaults to auto/content-based sizing if not set). Fix: add grid-auto-rows: 100px alongside the explicit definition to make auto-created rows match the same height.

Q5 (senior): Your team migrated a card-grid component from flexbox (flex: 1 1 250px + flex-wrap) to grid (repeat(auto-fit, minmax(250px, 1fr))) specifically to fix the uneven-last-row bug. QA now reports a DIFFERENT issue: when there's only 1 item in the entire grid, that single card stretches to fill the ENTIRE container width, looking oversized and awkward, whereas the old flexbox version kept it at a reasonable 250px.

Diagnosis: this is auto-fit's exact defining behavior (Q3) — with only 1 item, auto-fit collapses ALL the other (empty, unused) column tracks to zero and lets the single existing item's 1fr stretch to consume the entire freed space, which is correct, intentional behavior for the MULTI-item case but produces an oversized single card when there's just one item. This is a genuine, real edge case at the boundary of what auto-fit is designed for — not a bug in the migration, but an unconsidered edge case of the chosen pattern. Fix options: (1) switch to auto-fill instead of auto-fit specifically — auto-fill would keep the OTHER empty tracks reserved at their minmax minimum, so the single existing card stays at a reasonable 250px-ish width with visible empty space beside it (matching the OLD flexbox behavior's visual result, though for a different underlying reason); (2) if visible empty space is itself undesirable, add a max-width to the individual card (e.g., max-width: 400px) as an additional constraint, capping how large ANY single stretched card can become regardless of auto-fit's space-filling behavior, while still benefiting from auto-fit's gap-avoiding behavior in the multi-item case. The senior point: this is a genuine, real tradeoff between auto-fill and auto-fit that only surfaces at low item counts — correctly diagnosing WHICH of the two behaviors (reserved empty tracks vs. stretched existing items) matches the actual desired design intent for the SPECIFIC component's expected item-count range, rather than defaulting to auto-fit everywhere out of habit, is the actual skill this scenario tests.

Q6 (senior): Design the CSS Grid layout for a dashboard with: a fixed-height header, a fixed-width sidebar, a main content area that itself needs an internal 2-column sub-layout, and a footer — where the WHOLE page should use named grid areas for readability, but the main content's internal 2-column split should use line-based placement since it's a nested, independent grid.

The correct architecture uses grid-template-areas for the OUTER, page-level structure (matching Part 3's recommendation that named areas suit overall, relatively fixed structural regions) and a SEPARATE, NESTED grid (with its own display:grid and its own track definitions, likely using simple fr-based line placement since it's a straightforward, uniform 2-column split with no complex named regions of its own) for the inner content area: .page { display: grid; grid-template-columns: 240px 1fr; grid-template-rows: 80px 1fr 60px; grid-template-areas: "sidebar header" "sidebar main" "sidebar footer"; } .main { grid-area: main; display: grid; grid-template-columns: 2fr 1fr; gap: 20px; } — the main area's grid-area:main places IT correctly within the outer page grid, and its OWN display:grid establishes an entirely independent, nested grid context for its two internal columns, with its own separate track definitions that have no relationship to the outer grid's tracks at all (grids don't share track definitions across nesting boundaries — each display:grid starts a fresh, independent grid formatting context). This composition — named areas for stable, page-level regions; a simpler nested grid with straightforward line/fr-based placement for a self-contained internal layout that doesn't need its own named-area readability treatment — is the practical, real-world pattern for handling a page with BOTH a fixed overall structure and independently-laid-out inner sections, matching each part's actual complexity to the appropriately-weighted tool rather than forcing one single grid declaration to describe the entire page's full nested structure at once (which would be considerably harder to read and reason about as one large, flat set of named areas).

5 (hard): The single-item edge case, diagnosed and fixed. Reproduce Q5's exact scenario: an auto-fit card grid with only 1 card, observing it stretch to fill the entire container width. Solution + comparison 
/* fix (a): */

/* fix (b): */

.card { max-width: 400px; }

Fix (a): the single card renders at exactly 250px (its minmax minimum), with empty space beside it. Fix (b): the single card stretches but stops at 400px (the imposed cap), with any remaining space beyond that showing as empty (since grid still tries to fill the 1fr track, but the card's own max-width constrains its actual rendered size within that track). Both are valid, genuinely different visual outcomes — worth explicitly noting which one your own design intent would actually prefer for a real single-item scenario.

🎯 Key Takeaways

  • Grid is genuinely two-dimensional — row and column tracks are defined together on the container, so every row (including a partial last row) aligns to the same tracks, directly resolving flexbox's per-row-independent uneven-last-row limitation.
  • fr distributes space AFTER fixed tracks and content minimums are accounted for — composes cleanly with px tracks in the same track list, unlike percentages.
  • grid-template-areas' ASCII-diagram syntax makes overall page structure directly readable from the CSS itself — the strongest readability win over numbered line-based placement for a page's main regions.
  • repeat(auto-fit, minmax(min, 1fr)) is grid's superior, uneven-row-free answer to flexbox's responsive-card-grid pattern — auto-fit collapses empty tracks (items expand to fill), auto-fill reserves them (items stay minimum size).
  • The explicit grid (deliberately defined tracks) and implicit grid (auto-created for overflow items) are genuinely separate concepts with separate sizing controls (grid-template-rows vs grid-auto-rows).
  • Named areas suit a page's overall, relatively fixed structure; line-based placement (or a separate nested grid) suits irregular spans or genuinely independent internal sub-layouts.
