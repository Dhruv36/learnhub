Design Systems & Architecture Ultra-Advanced

One person's CSS needs taste. Fifty people's CSS needs architecture: naming conventions, tokens, and rules that make the ten-thousandth component look like the first. This is what "senior frontend" means in practice.

Naming: BEM (still the lingua franca)

.card { }                    /* Block — the component */
.card__title { }             /* Element — a part of it (__) */
.card__title--featured { }   /* Modifier — a variant (--) */

<article class="card card--featured">
  <h3 class="card__title">…</h3>
</article>

What BEM really buys: flat specificity (every selector is one class) and grep-ability (searching card__title finds exactly one thing). Scoped-CSS tooling (CSS Modules, styled-components) automates the same isolation — the naming discipline still applies inside each component.

The Layer Cake (how production stylesheets are organized)

@layer reset, tokens, base, layout, components, utilities;

tokens      → custom properties only (colors, spacing, type scale)
base        → element defaults (h1, a, code) — no classes
layout      → page scaffolding (.container, .stack, .grid)
components  → the bulk: one file per component
utilities   → single-purpose overrides (.hidden, .sr-only) — may use !important

Spacing & Type Scales — No Magic Numbers

:root {
  --space-1: .25rem; --space-2: .5rem; --space-3: 1rem;
  --space-4: 1.5rem; --space-5: 2.5rem; --space-6: 4rem;
  --text-sm: .875rem; --text-base: 1rem; --text-lg: 1.25rem; --text-xl: 1.6rem;
}
/* Review rule: any raw px/rem spacing value in a component is a smell.
   6 spacing values cover 95% of real UI — that constraint IS the consistency. */

Component Checklist (what "done" means in a design system)

RequirementExample for a Button
All interactive stateshover, active, :focus-visible, disabled, loading
Theming via tokens onlycolors from --color-action, never hex
Density/size variants--btn-sm/md/lg from the spacing scale
Works in both themes + RTLlogical properties (padding-inline)
Documented in isolationStorybook page with every state visible

Utility-First (Tailwind) — the Other Church

<button class="px-4 py-2 rounded-lg bg-green-700 text-white hover:bg-green-800">Save</button>

Same goals, inverted mechanism: the constraint lives in the utility scale instead of component classes. Trade-off: faster to build, harder to theme deeply; markup carries the styling. Big-company reality: many run both — Tailwind for product surfaces, a token-based component library for the design system core. Knowing when each wins is the senior answer; tribal loyalty is the junior one.

💡 Staff-level insight: a design system is a product with users (your engineers), not a stylesheet. It succeeds on adoption mechanics: migration codemods, lint rules that block raw values, versioned releases with changelogs, and a deprecation policy. The CSS is the easy 20%.
