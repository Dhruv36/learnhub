Keyframe Animations Motion & Interaction

What you'll master: @keyframes syntax and when it's the correct choice over transitions, the animation shorthand's full parameter set (including the commonly-confused fill-mode), animation-iteration-count and infinite looping, orchestrating multiple animated elements with animation-delay, and JS-driven animation events for coordinating CSS animations with real application logic. Closes the Motion & Interaction section. Prerequisite: Transitions.

Part 1 — Transitions vs Animations: the Real Distinction

TRANSITIONS  — smooth a change between TWO states (start, end) — reactive, triggered by
                something ELSE changing (:hover, a class toggle)

@KEYFRAMES   — define an EXPLICIT SEQUENCE of states (as many as needed), independent of any
                triggering interaction — can run automatically on page load, loop infinitely,
                and express complex, multi-step motion a simple two-state transition can't

The previous lesson's transitions are fundamentally REACTIVE — they need some other state change to respond to. @keyframes animations are self-contained sequences that can run on their own, without needing a triggering interaction at all — a loading spinner that spins continuously from page load, a subtle background gradient that shifts colors over 10 seconds in an endless loop, or a multi-step "shake" effect (left, right, left, right, center) that a simple two- state transition genuinely cannot express, since a transition only ever has ONE start and ONE end state, never a defined MIDDLE sequence.

Part 2 — @keyframes Syntax and the animation Shorthand

@keyframes pulse {
  0%   { transform: scale(1);    opacity: 1; }
  50%  { transform: scale(1.05); opacity: 0.8; }
  100% { transform: scale(1);    opacity: 1; }
}
/* from/to are valid ALIASES for 0%/100% specifically, when there are only two keyframe steps: */
@keyframes fade-in {
  from { opacity: 0; }
  to   { opacity: 1; }
}

.badge {
  animation: pulse 2s ease-in-out infinite;
  /*         ^name  ^duration ^easing   ^iteration-count */
}

Each percentage represents a point in the animation's timeline, and every property SET at that percentage is what the element should look like at that exact moment — the browser interpolates smoothly between EVERY consecutive pair of keyframes automatically, exactly like a transition does between just two states, but now across as many defined steps as needed. The animation shorthand's full parameter list is longer than transition's, and order matters:

animation: name duration timing-function delay iteration-count direction fill-mode play-state;
/* the two you'll actually use constantly beyond the basics: iteration-count and fill-mode */

Part 3 — animation-fill-mode: the Property Everyone Gets Wrong Once

@keyframes slide-in {
  from { transform: translateX(-100%); opacity: 0; }
  to   { transform: translateX(0);     opacity: 1; }
}
.toast {
  animation: slide-in 0.4s ease-out;
  /* WITHOUT fill-mode: the element SNAPS BACK to its original CSS (transform: none, opacity
     whatever it was before) the INSTANT the animation finishes — even though the "to" keyframe
     clearly specified the FINAL desired state! */
}
.toast {
  animation: slide-in 0.4s ease-out forwards;
  /*                                 ^^^^^^^^ RETAINS the LAST keyframe's styles after finishing */
}

This is the single most common, genuinely confusing first-time keyframe-animation bug: by DEFAULT, an animation's effects are entirely temporary — the moment it completes, the element reverts to whatever its styles would be WITHOUT the animation running at all (its normal, un-animated CSS), regardless of what the final keyframe declared. animation-fill-mode: forwards is the fix, telling the browser to RETAIN the styles from the animation's final keyframe after it finishes — essential for any entrance animation (a toast sliding in, a fade-in) where the "arrived" state needs to actually persist, not just flash briefly before reverting. (backwards similarly applies the FIRST keyframe's styles during any animation-delay, before the animation actually starts; both combines forwards+backwards.)

Part 4 — Looping and Orchestration

.spinner {
  animation: rotate 1s linear infinite;    /* infinite = loop forever; linear = constant speed,
                                               correct for continuous rotation (Transitions
                                               lesson's easing discussion, applied here) */
}
@keyframes rotate {
  to { transform: rotate(360deg); }
}

/* ORCHESTRATING several elements to animate in SEQUENCE, not simultaneously: */
.item:nth-child(1) { animation-delay: 0s;    }
.item:nth-child(2) { animation-delay: 0.1s;  }
.item:nth-child(3) { animation-delay: 0.2s;  }
/* each item fades/slides in slightly AFTER the previous one — a "staggered reveal" effect */

animation-iteration-count: infinite is the standard, simple way to create continuous, looping motion — and as noted in Part 2, linear easing is specifically CORRECT here (not a compromise), since continuous rotation has no natural "arrival" or "departure" to justify acceleration/deceleration. The staggered-delay pattern (each sibling's animation-delay offset by a small, consistent increment via :nth-child) is a genuinely common, professional technique for list/grid entrance animations — items appearing to "cascade" in one after another rather than all popping in simultaneously, a small but real polish detail that separates a hand-crafted-feeling interface from a generic one.

Part 5 — Coordinating With JavaScript: animationend and Beyond

const badge = document.querySelector(".badge");
badge.addEventListener("animationend", () => {
  badge.classList.remove("is-animating");    /* clean up after a one-shot animation completes */
});
badge.addEventListener("animationiteration", () => {
  console.log("one loop of an infinite animation just completed");
});

Real applications frequently need to coordinate a CSS animation with actual JS logic — removing a "new" badge's pulsing animation after it's played a few times, triggering a follow-up action once an entrance animation finishes, or synchronizing several independently-CSS-animated elements. The animationend event (fired once, when a NON-infinite animation completes) and animationiteration (fired at the end of EACH loop, for infinite/repeating animations) are the direct hooks for this — the exact JS-CSS coordination pattern the previous lesson's transitionend event served for transitions, now for keyframe animations specifically.

Interview Questions

Q1 (beginner): When would you reach for @keyframes instead of a transition?

When the effect needs an explicit MULTI-STEP sequence (more than just a start and end state) — a shake effect, a multi-color background cycle — or when the animation needs to run AUTOMATICALLY/continuously without any triggering interaction, like a loading spinner. Transitions only ever smooth between two reactive states; @keyframes define a self-contained sequence independent of any triggering change.

Q2 (beginner): Why does an entrance animation sometimes visually "flash then disappear" instead of staying in its final state?

By default, an animation's visual effects are temporary — the moment it completes, the element reverts to whatever styles it would have WITHOUT the animation running, regardless of what the final keyframe declared. animation-fill-mode: forwards fixes this by retaining the last keyframe's styles after the animation finishes.

Q3 (mid): Why is linear easing specifically correct (not a compromise) for a continuously-looping spinner, while ease/ease-in-out is correct for a one-time entrance animation?

Continuous, cyclical motion (a spinner rotating forever) has no natural "arrival" or "departure" moment to justify acceleration/deceleration — constant speed IS the physically-natural choice for ongoing rotation, and using ease/ease-in-out on a loop can introduce a subtle, unwanted hitch at each loop boundary where deceleration meets the next cycle's acceleration. A one-time entrance animation, by contrast, genuinely has a start and end that benefit from matching real physical intuition (an object accelerating away or decelerating as it arrives), which is exactly what ease-based curves communicate.

Q4 (mid): How would you make 5 list items fade in one after another (staggered) rather than simultaneously?

Give each item an incrementing animation-delay, typically via :nth-child selectors (0s, 0.1s, 0.2s, 0.3s, 0.4s for 5 items) — each item's fade-in keyframe animation starts slightly after the previous one's, producing a cascading, staggered reveal effect rather than all 5 items appearing at the exact same instant.

Q5 (senior): A "new feature" badge uses animation: pulse 1s ease-in-out infinite to draw attention, but a UX review flags that it never stops, becoming a persistent, mildly irritating distraction for users who've already noticed and dismissed the feature. Propose a fix that preserves the attention-grabbing intent without the permanent-annoyance downside.

The core tension: infinite looping genuinely serves the initial "notice this" goal, but has no built-in mechanism to recognize when that goal has been ACHIEVED (the user noticed) and should stop. Fix: replace infinite with a bounded animation-iteration-count (e.g., 3) — the badge pulses a FEW times to draw initial attention, then settles into its final resting state via animation-fill-mode: forwards, no longer perpetually distracting once the attention-grabbing job is done. For an even more refined version: listen for animationiteration (fired at the end of each loop) or animationend (if iteration-count is finite) and additionally use JS to permanently suppress the animation on FUTURE visits (e.g., a localStorage flag set once the user has actually interacted with/clicked the badge, or simply once it's finished its bounded 3-iteration run this session) — ensuring returning users who've already seen and engaged with the feature don't get re-shown the same attention-grabbing animation indefinitely across every future visit either. The senior framing: "grab attention" and "don't permanently annoy" are BOTH legitimate, simultaneously satisfiable requirements once recognized as needing a BOUNDED, not infinite, animation plus a persistence mechanism tracking whether the attention-grabbing goal has already been met.

Q6 (senior): Design a toast-notification system where multiple toasts can appear in quick succession (e.g., 3 API calls completing within 500ms of each other, each triggering its own toast). Specify the animation architecture ensuring toasts stack visually without overlapping or jarring simultaneous pop-ins, and how you'd coordinate their eventual dismissal.

Architecture: each toast, on creation, gets its OWN slide-in @keyframes animation (animation: slide-in 0.3s ease-out forwards) — but rather than relying on animation-delay (which would require knowing in advance how many toasts are coming and their exact relative timing, brittle for a genuinely async, unpredictable arrival pattern), the STAGGERING should be handled by the actual DOM insertion timing: each new toast is appended to a flex-column container (newest at top or bottom, a real design decision) with the container's own gap providing gu consistent spacing, and CSS Grid/Flexbox's normal layout reflow (NOT animation-delay) naturally pushes existing toasts to make room as a new one is inserted — meaning the "staggering" is really just each toast's OWN entrance animation firing at its OWN real insertion moment (whenever the triggering API call actually completes), with the layout engine (not animation timing) handling the visual repositioning of already-present toasts. This is more ROBUST than pre-calculated animation-delay staggering specifically because real API completion timing is inherently unpredictable — a fixed delay schedule would assume toasts arrive at neat, regular intervals, which genuinely async operations don't guarantee. For dismissal: each toast independently sets its own auto-dismiss timer (e.g., setTimeout 4000ms after its OWN creation, not a shared global timer), triggering a REVERSE exit animation (animation: slide-out 0.3s ease-in forwards) and, critically, removing the toast from the DOM only on that exit animation's animationend event (not immediately on the timer firing) — ensuring the exit animation genuinely completes visually before the element vanishes, exactly mirroring the previous lesson's display-toggle race-condition fix, now applied to keyframe animations instead of transitions. The senior synthesis: coordinating MULTIPLE independently-timed, asynchronously-triggered animated elements is fundamentally a per-element timing and event-driven cleanup problem, not a pre-choreographed animation-delay schedule — recognizing this distinction is what separates a robust toast system from one that breaks the moment real-world async timing doesn't match a designer's neat, evenly-spaced mockup assumption.

🏋️ Exercises

1 (easy): The fill-mode bug, reproduced and fixed. Build a "slide in from left" entrance animation (Part 3's example) WITHOUT fill-mode. Trigger it (e.g., on page load) and observe the element briefly sliding in, then instantly snapping back to its original, un-animated position. Add fill-mode: forwards and confirm it now stays in its "arrived" position.
Expected before/after

Without forwards: the element visibly completes its slide-in motion, then INSTANTLY jumps back to translateX(0)'s absence — i.e., wherever it would be with NO animation applied at all (likely its default, un-transformed position, which might coincidentally look similar OR obviously different depending on the specific keyframes — the key giveaway is the sudden, jarring snap at the exact moment the animation "completes"). With forwards: the element smoothly arrives and STAYS at its final keyframe's position with no snap-back at all.

2 (easy): linear vs ease, on a loop. Build a continuously- rotating spinner (animation: rotate 1s ease-in-out infinite) and a second one using linear instead. Watch both for several full rotations and describe the difference at each loop boundary.
Expected observation

The ease-in-out version should show a subtle but noticeable "pause" or hitch at the exact moment each rotation completes and the next begins — the deceleration finishing one cycle meeting the acceleration starting the next creates a slight stutter in otherwise-continuous rotation. The linear version should rotate with genuinely constant, unbroken speed with no perceptible loop- boundary artifact at all — direct, hands-on confirmation of Q3's reasoning.

3 (medium): Staggered list reveal. Build a list of 6 items, each with a fade-in-and-slide-up @keyframes animation, staggered via incrementing animation-delay per :nth-child. Confirm they cascade in visually rather than all appearing simultaneously.
Solution 
@keyframes fade-up {
  from { opacity: 0; transform: translateY(10px); }
  to   { opacity: 1; transform: translateY(0); }
}
.item { animation: fade-up 0.4s ease-out forwards; opacity: 0; }
.item:nth-child(1) { animation-delay: 0s; }
.item:nth-child(2) { animation-delay: 0.08s; }
.item:nth-child(3) { animation-delay: 0.16s; }
/* ...continuing the 0.08s increment... */

Note the base .item { opacity: 0 } — needed so items don't briefly flash visible before their delayed animation starts (fill-mode:forwards only affects styles AFTER the animation runs, not during any preceding delay, unless backwards/both is also used — a real, easy-to-miss detail worth discovering directly in this exercise).

4 (medium): The bounded-pulse fix, implemented. Build Q5's exact "new feature" badge scenario: an infinite pulse, then refactor it to pulse exactly 3 times (animation-iteration-count: 3) and settle into a final resting state via fill-mode: forwards. Confirm via the browser that it visibly pulses 3 times then stops, rather than continuing forever.
Solution 
.badge {
  animation: pulse 1s ease-in-out 3 forwards;
  /*                              ^3 iterations, then hold the FINAL keyframe's state */
}

Watch it complete exactly 3 visible pulses and then hold still — confirming both the bounded iteration-count AND fill-mode are working together correctly, directly resolving Q5's UX complaint about permanent distraction while preserving the initial attention-grabbing effect.

5 (hard): animationend cleanup, wired. Build a "shake" error animation on a form field (triggered by adding a .shake class via JS on validation failure). Use animationend to automatically REMOVE the .shake class once the animation completes, so the SAME class can be re-added (and the animation re-triggered) on a SUBSEQUENT validation failure without needing a page reload or manual class-removal elsewhere.
Reference implementation 
@keyframes shake {
  0%, 100% { transform: translateX(0); }
  25% { transform: translateX(-8px); }
  75% { transform: translateX(8px); }
}
.shake { animation: shake 0.3s ease-in-out; }

field.addEventListener("animationend", () => field.classList.remove("shake"));

function triggerShake() {
  field.classList.remove("shake");   // remove first, in case it's already present
  void field.offsetWidth;             // force a REFLOW — necessary to let the browser "notice"
                                       // the class was removed before re-adding it, or the animation
                                       // won't re-trigger on an already-shaking element
  field.classList.add("shake");
}

The void field.offsetWidth reflow-forcing line is the exercise's real, subtle payoff — without it, rapidly re-triggering the shake (submitting invalid input twice quickly) may FAIL to re-animate at all, since the browser doesn't naturally re-run an animation when a class is removed and immediately re-added within the same rendering frame; forcing a synchronous layout read in between is the standard, real technique to guarantee re-triggering works reliably.

6 (hard): The complete toast system. Implement Q6's full architecture: a toast container that toasts get dynamically appended to (via JS, simulating 3 toasts firing within 500ms of each other from separate setTimeout calls), each with its own slide-in entrance animation, its own independent auto-dismiss timer, and a slide-out exit animation with DOM removal correctly gated on that exit animation's animationend event (not the timer firing directly). Verify no toast disappears mid-animation and that rapid-fire toasts stack/reflow correctly via normal flex layout rather than pre-calculated delays.
Reference core 
function showToast(message) {
  const toast = document.createElement("div");
  toast.className = "toast";
  toast.textContent = message;
  toastContainer.appendChild(toast);   // flex container; normal layout handles stacking/reflow

  setTimeout(() => {
    toast.classList.add("toast-exit");   // triggers the slide-out @keyframes via CSS
    toast.addEventListener("animationend", () => toast.remove(), { once: true });
  }, 4000);
}
// simulate 3 async-arriving toasts:
setTimeout(() => showToast("Saved!"), 100);
setTimeout(() => showToast("Synced"), 300);
setTimeout(() => showToast("Uploaded"), 480);

Verify visually: each toast slides in at its own real arrival moment (not a pre-choreographed schedule), existing toasts smoothly reflow to make room via the container's normal flex layout (no animation-delay math involved in the stacking itself), and each toast's removal from the DOM happens only AFTER its exit animation visibly completes — confirming the complete, race- condition-free architecture Q6's answer described.

🎯 Key Takeaways

  • @keyframes define an explicit, self-contained multi-step sequence that can run independently of any triggering interaction — transitions only ever smooth a reactive change between two existing states.
  • animation-fill-mode: forwards is essential for any entrance animation — without it, the element snaps back to its pre-animation state the instant the animation completes, regardless of the final keyframe.
  • linear easing is genuinely correct (not a compromise) for continuous, cyclical motion like spinners — ease-based curves can introduce a hitch at each loop boundary.
  • Staggered animation-delay per item (often via :nth-child) produces a cascading reveal effect — a small, professional polish detail for list/grid entrances.
  • animationend/animationiteration are the JS coordination hooks for cleanup, re-triggering, and synchronizing CSS animations with real application logic — directly mirroring transitionend's role for transitions.
  • Coordinating multiple independently, asynchronously-triggered animated elements (a toast system) is fundamentally a per-element timing and event-driven cleanup problem, not a pre-choreographed delay schedule — real async timing rarely matches a neat, evenly-spaced mockup assumption.
