Units & Sizing Foundations

What you'll master: absolute vs relative units and when each earns its place, the em-compounding trap that makes rem the safer default, viewport units and their real use cases (plus the mobile-viewport-height gotcha), the modern clamp()/min()/max() functions that replace most media queries for fluid sizing, and calc() for genuinely mixed-unit arithmetic. Prerequisite: Box Model.

Part 1 — Absolute vs Relative: the Real Distinction

ABSOLUTE (fixed, same physical size everywhere):
  px    — the only absolute unit anyone actually uses in screen CSS

RELATIVE (scale based on something else):
  em    — relative to the CURRENT element's font-size (or parent's, for font-size itself)
  rem   — relative to the ROOT (html) element's font-size, ALWAYS — never compounds
  %     — relative to the parent's corresponding dimension
  vw/vh — relative to the VIEWPORT'S width/height
  ch    — relative to the width of the "0" character in the current font (rarely needed, but exact
           for things like limiting a text column to "70 characters wide")

The entire practical argument in this lesson centers on relative units' real superpower: they respect the user's own browser font-size setting. A user who bumps their browser's default text size up (a genuine, common accessibility accommodation for low vision) sees EVERYTHING sized in rem/em scale up proportionally with it; anything sized in raw px stays exactly the same physical size regardless of that setting, silently defeating the user's own accessibility preference. This is why font-size, spacing, and most sizing in a well-built modern site uses rem — not because px is "wrong," but because rem respects a real, common accessibility need px structurally cannot.

Part 2 — The em Compounding Trap (and Why rem Exists)

html { font-size: 16px; }         /* the root */
.card { font-size: 1.2em; }       /* 1.2 × 16px = 19.2px */
.card .badge { font-size: 1.2em; }  /* 1.2 × 19.2px = 23px — COMPOUNDED, not 1.2 × 16 again! */
.card .badge .count { font-size: 1.2em; }  /* 1.2 × 23px = 27.6px — compounds AGAIN */

This is em's genuine, well-known trap: because em is relative to the CURRENT element's own computed font-size (which itself may already be an em-scaled value from an ancestor), nesting em-sized elements causes each level to multiply against the PREVIOUS level's already-scaled result — sizes compound unpredictably and grow (or shrink) exponentially with nesting depth, a genuine, real maintenance nightmare in any deeply-nested component structure.

html { font-size: 16px; }
.card { font-size: 1.2rem; }        /* 1.2 × 16px = 19.2px — relative to ROOT, always */
.card .badge { font-size: 1.2rem; } /* STILL 1.2 × 16px = 19.2px — no compounding, ever,
                                        regardless of nesting depth */

rem ("root em") always resolves against the <html> element's font-size, completely independent of nesting depth — this is precisely why rem has become the default, recommended unit for font-size and most spacing in modern CSS, while em is reserved for the genuinely rare cases where compounding relative-to-THIS- element scaling is actually the desired behavior (a component that should scale ALL its internal spacing proportionally to its OWN font-size, regardless of where it's placed — a real but comparatively uncommon need).

Part 3 — Viewport Units: the Real Use Cases and the Mobile Gotcha

.hero { height: 100vh; }    /* full viewport height — the classic "full-screen hero" pattern */
h1 { font-size: 4vw; }       /* scales continuously with viewport width — "fluid typography" */

vw/vh (1 unit = 1% of viewport width/height) are the right tool specifically for content that should genuinely scale with the VIEWPORT itself (a full-screen hero section, a continuously fluid heading size) — but 100vh on mobile carries a real, well-known gotcha: mobile browsers' address bar dynamically shows/hides as the user scrolls, changing the ACTUAL visible viewport height, and 100vh was historically calculated against the LARGEST possible viewport (address bar hidden) — causing content sized at 100vh to be taller than the actually-visible area when the address bar is showing, creating an annoying, real "why is the bottom of my hero section cut off on my phone" bug. The modern fix: 100dvh ("dynamic viewport height" — genuinely tracks the CURRENT, real visible viewport, updating live as the address bar shows/hides), now supported in all current browsers and the correct default choice over plain vh for full-screen mobile layouts.

Part 4 — clamp(): Fluid Sizing Without Media Queries

h1 {
  font-size: clamp(1.5rem, 4vw + 1rem, 3rem);
}
/*            ^MINIMUM    ^PREFERRED       ^MAXIMUM
   Never smaller than 1.5rem, never larger than 3rem, and FLUIDLY scales with
   viewport width in between — continuously, not in the discrete jumps a
   media-query-based approach would produce. */

This is the single biggest practical upgrade to responsive sizing in modern CSS: instead of writing several media-query breakpoints each hardcoding a different fixed font-size (the old approach — 3-4 separate rules, each a guess at "good enough" sizes for arbitrary breakpoint ranges, with visible SIZE JUMPS at each breakpoint boundary), clamp(min, preferred, max) computes a continuously fluid value that smoothly scales with the viewport, bounded by sensible min/max limits, in ONE line, with zero jump discontinuities at any specific width. The preferred value commonly mixes viewport units with a fixed unit (4vw + 1rem) specifically so the scaling rate AND a baseline floor both contribute — pure vw alone would hit zero at a zero-width viewport, an edge case the added rem term avoids in a real, meaningful way.

min() and max() are clamp's simpler siblings — width: min(90%, 600px) ("whichever is SMALLER" — a fluid width that never exceeds 600px, replacing the common old max-width pattern with more expressive power when combined with other functions) and max() the inverse logic. All three accept mixed units and even nested calc() expressions, making them the modern toolkit's actual replacement for a large fraction of what used to require explicit breakpoint-based media queries.

Part 5 — calc(): Mixed-Unit Arithmetic

.sidebar { width: calc(100% - 250px); }   /* fill remaining space after a fixed-width sibling */
.section { padding-top: calc(2rem + 5vh); }  /* combine a fixed base with a viewport-relative amount */
.grid-item { width: calc((100% - 40px) / 3); }  /* three equal columns with two 20px gutters between them */

calc() is the general-purpose escape hatch for exactly one recurring problem: CSS normally can't mix DIFFERENT unit types in a single value (you can't write width: 100% - 250px directly) — calc() is the explicit syntax permitting that mixed arithmetic, evaluated by the browser at layout time. It composes naturally with custom properties (Modern CSS section, later) and with clamp/min/max, and is genuinely necessary any time a dimension needs to combine a percentage/fr/viewport-relative amount with a fixed pixel amount in the same expression — the "fill remaining space after this fixed sidebar" pattern above is one of the single most common real-world calc() use cases in production layouts.

Interview Questions

Q1 (beginner): Why is rem generally preferred over px for font-size in modern CSS?

rem scales proportionally with the user's own browser font-size setting — a real, common accessibility accommodation for users with low vision who increase their default text size. Fixed px values ignore that setting entirely, rendering at the same physical size regardless of what the user configured, silently defeating their accessibility preference. rem isn't "more correct" abstractly — it specifically respects a real, common user need px structurally cannot.

Q2 (beginner): What's the practical difference between em and rem?

em is relative to the CURRENT element's own computed font-size (which may itself already be em-scaled from an ancestor, causing compounding across nested levels). rem is always relative to the ROOT (html) element's font-size, regardless of nesting depth — no compounding, ever. This makes rem the safer default for most sizing; em is reserved for the comparatively rare case where genuinely wanting a component's internal sizing to scale relative to ITS OWN font-size is the actual desired behavior.

Q3 (mid): A full-screen hero section using height: 100vh renders correctly on desktop but is visibly cut off at the bottom on mobile Safari. Diagnose.

Mobile browsers' address bar dynamically shows/hides during scroll, changing the actual visible viewport height — 100vh has historically been calculated against the LARGEST possible viewport (address bar hidden), so content sized at 100vh can exceed the actually-visible area when the address bar is showing, cutting off the bottom. Fix: 100dvh (dynamic viewport height), which tracks the current, real visible viewport and updates live as the address bar's visibility changes — supported in all current browsers and the correct modern default for full-screen mobile layouts.

Q4 (mid): Explain what clamp(1.5rem, 4vw + 1rem, 3rem) does and why it's preferred over several media-query breakpoints for responsive font-size.

It computes a font-size that's never smaller than 1.5rem, never larger than 3rem, and continuously, fluidly scales with viewport width in between via the preferred value (4vw + 1rem) — as opposed to several discrete media-query breakpoints each hardcoding a fixed size, which produces visible, discontinuous JUMPS at each breakpoint boundary and requires guessing "good enough" sizes for arbitrary width ranges. clamp() achieves smooth, continuous scaling with zero jump discontinuities in a single line.

Q5 (senior): Your team's design system defines spacing exclusively in px (8px, 16px, 24px...) for "pixel-perfect consistency with design mockups," while your accessibility audit flags that text-heavy pages become cramped and hard to read when users increase browser font-size, since spacing doesn't scale with it. Propose a resolution that satisfies both concerns.

The tension is real but resolvable: convert the design system's spacing SCALE to rem (8px→0.5rem, 16px→1rem, 24px→1.5rem, assuming the standard 16px root font-size — a straightforward, mechanical conversion with ZERO visual difference at default browser settings, since rem and the equivalent px value render identically when the root font-size is unchanged). This satisfies both concerns simultaneously: designers still get their exact pixel-equivalent spacing values at default settings (pixel-perfect fidelity to mockups is preserved), AND when a user increases their browser's font-size for accessibility, spacing NOW scales proportionally alongside text size, keeping the layout's proportions and legibility intact rather than cramping as text grows while fixed-px spacing stays static. The senior framing: "pixel-perfect" and "accessible" are not actually in tension here — they were only in apparent conflict because px was being used as if it were the only way to achieve exact values, when rem achieves IDENTICAL exact values at default settings while ALSO gracefully scaling for users who need it. The fix is a systematic, mechanical unit conversion across the design system's spacing tokens (Architecture section, later in this track, covers design tokens formally), not a compromise between the two goals.

Q6 (senior): A marketing team wants a hero heading that's "huge on desktop, reasonably sized on mobile, and never breaks to more than 2 lines regardless of screen width." Design the CSS, and explain why clamp() alone doesn't fully solve this specific requirement.

clamp() elegantly solves the FONT-SIZE fluidity (huge on desktop, reasonably sized on mobile, smooth scaling between) — e.g. font-size: clamp(2rem, 6vw + 1rem, 5rem) — but it does NOT, by itself, guarantee a specific LINE COUNT, because line-wrapping depends on the actual heading TEXT LENGTH relative to its available width, which clamp's font-size alone doesn't control. The "never more than 2 lines" requirement needs an ADDITIONAL, distinct technique layered on top: either (a) a max-width constraint on the heading tuned (through actual testing across real heading text lengths) to keep typical content within 2 lines at the clamped font-size range — an imprecise, content-dependent tuning exercise, not a pure-CSS guarantee; or (b) if truly strict 2-line enforcement regardless of content length is required, a genuinely different CSS feature entirely — line-clamp (-webkit-line-clamp: 2; combined with overflow:hidden and display: -webkit-box) which truncates text with an ellipsis after exactly 2 rendered lines, sacrificing "always show full text" for "guarantee the line count," a real product tradeoff the marketing team needs to explicitly accept (truncating a long headline vs. allowing it to occasionally wrap to a 3rd line). The senior point: clamp() is the correct tool for the SIZE-fluidity part of this compound requirement, but "never more than N lines" is a fundamentally separate CSS problem (line-wrapping/truncation) that needs its own distinct technique — recognizing where one tool's scope ends and a different, genuinely separate CSS mechanism is needed is exactly the kind of requirement-decomposition skill that distinguishes a correct solution from a plausible-sounding but incomplete one.

🏋️ Exercises

1 (easy): The em compounding trap, reproduced. Build three nested divs, each with font-size: 1.3em, and observe the actual rendered font sizes at each level (compute what you'd expect from Part 2's formula first, then verify in DevTools' Computed panel). Then replace all three with rem and confirm identical sizing at every level.
Expected math

With a 16px root: level 1 = 1.3×16 = 20.8px, level 2 = 1.3×20.8 = 27px, level 3 = 1.3×27 = 35.2px — genuinely compounding, exactly as Part 2 describes. With rem: all three levels render at 1.3×16 = 20.8px identically, regardless of nesting — direct, hands-on confirmation of why rem is the safer default for anything that shouldn't compound.

2 (easy): dvh vs vh, if testable. If you have access to a real mobile device (or accurate mobile emulation with address-bar behavior), build a 100vh full-screen section and a 100dvh version side by side. Scroll to trigger the address bar's show/hide behavior and observe whether either section's bottom edge shifts/gets cut off. If no real device is available, research and write out from the spec what dvh guarantees that vh historically didn't.
Expected finding

On a real mobile device: the 100vh section's bottom edge may become cut off or reveal a gap as the address bar shows/hides during scroll (historically calculated against the LARGEST possible viewport); the 100dvh section should track the CURRENT real visible viewport continuously, without cutoff. If testing without a device: the spec-level guarantee is that dvh dynamically recalculates as the actual visible viewport changes, while vh's behavior here has been historically inconsistent/browser-dependent specifically for this dynamic-toolbar case.

3 (medium): Fluid typography, built. Replace a heading using three separate media-query breakpoints (each with its own fixed font-size) with a single clamp() declaration achieving equivalent min/max bounds. Resize the browser continuously across the full width range and confirm the clamp() version scales smoothly with no visible jump, while noting (before removing it) that the media-query version DID show visible jumps at its breakpoints.
Solution 
/* before — jumps at each breakpoint: */
h1 { font-size: 2rem; }
@media (min-width: 600px) { h1 { font-size: 3rem; } }
@media (min-width: 1000px) { h1 { font-size: 4rem; } }

/* after — one line, continuously fluid: */
h1 { font-size: clamp(2rem, 4vw + 1rem, 4rem); }

Slowly resizing the browser window across the media-query version should show the heading SNAP to a new size at exactly 600px and 1000px; the clamp() version should show continuous, gradual growth with no snap points at all — a directly observable, visual confirmation of the qualitative difference this lesson describes.

4 (medium): calc() for a real layout. Build a page layout with a fixed 250px sidebar and a main content area that should fill ALL remaining horizontal space, side by side, using calc(). Then add a 20px gap between them and adjust your calc() expression to account for it correctly.
Solution 
.layout { display: flex; gap: 20px; }
.sidebar { width: 250px; flex-shrink: 0; }
.main { width: calc(100% - 250px - 20px); }
/* or, more robustly with flexbox already handling this: .main { flex: 1; } needs no calc at all —
   worth noticing that flexbox's own flex-grow often replaces calc() for exactly this common pattern */

The exercise's real payoff, worth noticing explicitly: while calc() correctly solves this, a flex container with flex:1 on the main content area achieves the identical result with NO calc() at all — a preview of the Layout section's flexbox lesson, and a genuine, common case where a purpose-built layout mode makes manual calc() arithmetic unnecessary.

5 (hard): The accessibility conversion, performed. Take a set of 6 spacing values defined in px (4px, 8px, 12px, 16px, 24px, 32px) and convert them to a matching rem scale (assuming a 16px root font-size). Build a small page using both the px and rem versions side by side. Using your browser's zoom/font-size accessibility setting (not page zoom — the actual "minimum font size" or default font-size browser setting), increase the base font size and confirm the rem version's spacing scales proportionally while the px version stays fixed.
Conversion + expected observation 
4px → 0.25rem   8px → 0.5rem   12px → 0.75rem
16px → 1rem     24px → 1.5rem  32px → 2rem

At the default 16px root, both versions should look pixel-identical. After increasing the browser's default font-size setting: the rem-spaced version's gaps/padding should visibly grow proportionally alongside the now-larger text, while the px-spaced version's gaps stay exactly the same absolute size, increasingly cramped relative to the now-larger text — a direct, hands-on demonstration of Q5's interview scenario.

6 (hard): Compound requirement, decomposed. Implement Q6's full scenario: a hero heading using clamp() for fluid font-size PLUS -webkit-line-clamp:2 (with the required overflow/display properties) to strictly cap it at 2 lines with ellipsis truncation for overly long text. Test with both a short heading (fits easily) and a deliberately very long one (should truncate with "...") at multiple viewport widths.
Solution 
h1 {
  font-size: clamp(2rem, 5vw + 1rem, 4rem);
  display: -webkit-box;
  -webkit-line-clamp: 2;
  -webkit-box-orient: vertical;
  overflow: hidden;
}

The short heading should render normally, well within 2 lines, completely unaffected by the line-clamp properties. The long heading should visibly truncate with an ellipsis exactly at the end of the 2nd rendered line, at every viewport width tested — confirming the two techniques are genuinely independent, composable solutions to two DIFFERENT parts of the compound requirement (fluid sizing vs. strict line-count enforcement), exactly as Q6's answer distinguished.

🎯 Key Takeaways

  • Relative units (rem especially) respect the user's own browser font-size accessibility setting — px sizing silently defeats it, which is the real, non-cosmetic argument for rem over px.
  • em compounds across nested levels (relative to the CURRENT element's own already-scaled font-size); rem always resolves against the root, never compounding — rem is the safer default for most sizing.
  • 100dvh (dynamic viewport height) replaces 100vh for full-screen mobile layouts — it tracks the actual, current visible viewport as the mobile address bar shows/hides, where 100vh historically didn't.
  • clamp(min, preferred, max) replaces most breakpoint-based responsive font-sizing with continuous, jump-free fluid scaling in one line.
  • calc() is the explicit syntax for mixing different unit types in one expression (100% - 250px) — genuinely necessary for combining percentage/viewport-relative amounts with fixed pixel amounts, though purpose-built layout modes (flexbox's flex:1) sometimes make it unnecessary.
  • "Fluid size" and "strict line-count limit" are separate CSS problems needing separate techniques (clamp() vs. line-clamp) — decomposing a compound requirement into its actual independent parts is the real skill.
