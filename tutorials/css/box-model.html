Box Model Basics

Every element the browser renders is a rectangle made of four nested layers, and layout is nothing but arranging these rectangles. When you truly internalize this one model — including its two famous traps, box-sizing and margin collapsing — about half of all CSS confusion permanently disappears.

The Four Layers

┌─────────────── margin (transparent — pushes NEIGHBORS away, not part of the box) ──────────┐
│  ┌───────────── border (has width, style, color — part of the box) ────────────────┐        │
│  │  ┌─────────── padding (space INSIDE — shows the background) ────────────┐        │        │
│  │  │                          content                                     │        │        │
│  │  └──────────────────────────────────────────────────────────────────────┘        │        │
│  └────────────────────────────────────────────────────────────────────────────────────┘      │
└──────────────────────────────────────────────────────────────────────────────────────────────┘
Padding = breathing room inside (background visible).  Margin = distance to others (transparent).
Backgrounds paint under padding+content (and border, under its gaps).

box-sizing — the First Fix Every Project Ships

/* content-box (the 1996 default): width applies to CONTENT ONLY */
.card { width: 300px; padding: 20px; border: 1px solid; }
/* rendered width = 300 + 20 + 20 + 1 + 1 = 342px 😡 — and 300px + 50% widths overflow parents */

/* border-box: width means the VISIBLE BOX (content + padding + border) */
*, *::before, *::after { box-sizing: border-box; }
/* now width: 300px renders 300px; padding eats inward. Every production stylesheet does this. */

Why the pseudo-elements in the reset? * doesn't reach ::before/::after — without them, generated content still uses content-box and yields off-by-N bugs in decorated components. Why not on html + inherit? That variant (html{box-sizing:border-box} *{box-sizing:inherit}) lets a component opt out for its whole subtree — some design systems prefer it; either is fine, pick one.

📐 Margin is NEVER inside width — in either model. A 300px border-box element with 20px margins occupies 340px of horizontal space but IS 300px wide. Total space = box + margins.

Margin Collapsing — the Behavior That Feels Like a Bug

/* 1. ADJACENT SIBLINGS: vertical margins merge to the LARGER (not the sum) */
p { margin-block: 24px; }        /* gap between two p's = 24px, not 48px */
h2 { margin-bottom: 16px; } p { margin-top: 24px; }   /* gap = 24px (max), not 40px */

/* 2. PARENT ↔ FIRST/LAST CHILD: margins collapse THROUGH the boundary */
.section { background: gray; }
.section > h2 { margin-top: 32px; }
/* Expectation: 32px gray space above the h2.
   Reality: the margin escapes and pushes .section itself down; the h2 hugs the top. */

/* Anything that 'separates' the margins stops the collapse: */
.section { padding-top: 1px; }        /* or border, or overflow:auto */
.section { display: flow-root; }      /* the modern, purpose-built fix */

/* 3. Collapsing NEVER happens: horizontally, in flex/grid containers, or across flow-root */

Why does this exist? Typographic tradition: stacked paragraphs and headings should have consistent rhythm, not doubled gaps. It predates app UIs — which is why app-style layouts (flex/grid + gap) simply switched it off.

Sizing Keywords Modern Layout Runs On

width: min-content;    /* narrowest without overflow — longest word decides */
width: max-content;    /* everything on one line — no wrapping */
width: fit-content;    /* max-content, capped by available space — 'shrink-wrap' buttons/tags */
min-width: 0;          /* opt OUT of the implicit min-content floor (flex/grid overflow fix) */
aspect-ratio: 16 / 9;  /* modern ratio boxes — replaces the padding-top:56.25% hack */
min(), max(), clamp()  /* math at parse time: width: min(90%, 1100px) = fluid with a cap */

The conflict order interviewers ask: when width, min-width, and max-width disagree, min-width beats max-width beats width.

Percentages Resolve Against WHAT? (the quiet trap)

Property with %Resolves against
width, margin-*, padding-* (all four sides!)Parent's WIDTH
heightParent's height — but only if the parent's height is defined; otherwise ignored (the height:100% classic)
transform: translateThe element's own size (that's why translate(-50%,-50%) centers)
font-size, line-heightParent's font-size

Yes — padding-top: 10% uses parent width. That quirk powered a decade of aspect-ratio hacks; today write aspect-ratio and move on. And for full-viewport heights, prefer 100dvh over percent chains.

Shorthand Fluency & Logical Properties

margin: 8px;                   /* all four */
margin: 8px 16px;              /* vertical | horizontal */
margin: 8px 16px 24px;         /* top | horizontal | bottom */
margin: 8px 16px 24px 4px;     /* top right bottom left — clockwise from noon */
margin-inline: auto;           /* logical: both text-flow sides — the modern centering */
padding-block: 2rem;           /* logical: top+bottom */
border-inline-start: 3px solid; /* the 'left border' that flips itself for RTL (Arabic/Hebrew) */

Logical properties (inline = text-flow axis, block = stacking axis) make internationalized products work without an RTL stylesheet fork. New code should default to them.

Worked Example — a Card, Fully Reasoned

.card {
  box-sizing: border-box;        /* from the global reset */
  inline-size: min(100%, 340px); /* fluid, capped — logical 'width' */
  padding: 1rem 1.25rem;         /* inner space; background visible here */
  border: 1px solid #d0d7de;
  border-radius: 8px;
}
.card > * + *  { margin-block-start: .75rem; }  /* 'owl selector': rhythm BETWEEN children only —
                                                   no first-child exception, no collapse surprises */

Interview Questions

Q1: An element has width:200px, padding:16px, border:2px, margin:10px, border-box. How wide is the box, and how much horizontal space does it occupy?

The box renders 200px wide (padding and border absorbed inward, leaving 200−32−4 = 164px for content). It occupies 200 + 10 + 10 = 220px of horizontal space. Margin is spacing, not box.

Q2: Explain margin collapsing to a junior, including when it does NOT happen.

Vertical margins of adjacent block siblings — and of a parent with its first/last child when nothing separates them — merge to the larger value, for typographic rhythm. It never happens horizontally, inside flex/grid, across padding/border/flow-root boundaries, or on floated/absolute elements. Fix-of-choice today: flow-root on the parent, gap between siblings.

Q3: Why does padding-top: 56.25% create a 16:9 box, and what replaced it?

All padding percentages resolve against parent WIDTH, so top padding = 56.25% of width = 9/16 — height proportional to width. Replaced by aspect-ratio: 16/9, which also plays well with min/max constraints.

Q4: display:none vs visibility:hidden vs opacity:0 — including accessibility and events?

none: no box, no space, gone from the accessibility tree. hidden: space kept, invisible, no events, also removed from a11y tree (visibility can be re-shown on a descendant!). opacity:0: space kept, still focusable/clickable and still announced — which makes it the dangerous one for "hiding" interactive things.

🎯 Key Takeaways

  • Four layers: content, padding (inside, painted), border, margin (outside, transparent, never in width).
  • border-box globally, including ::before/::after — width means the visible box, always.
  • Vertical margins collapse (siblings AND through parents); flex/grid/flow-root/padding stop it; gap sidesteps it.
  • Percent padding/margins use parent width; percent height needs a defined parent height; translate % uses own size.
  • min-width beats max-width beats width; min-content/fit-content/aspect-ratio/clamp are modern sizing vocabulary.
  • Space siblings with gap or the owl selector; prefer logical properties for RTL-ready code.
