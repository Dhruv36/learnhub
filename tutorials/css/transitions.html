Transitions Motion & Interaction

What you'll master: the transition shorthand's real parameter order, easing functions and what they actually communicate (not just "which one looks nice"), transitioning multiple properties with independent timing, the genuinely important "transitions need TWO states" mental model, and why some properties simply can't be transitioned at all. Prerequisite: Transforms.

Part 1 — The Core Mental Model: Transitions Animate BETWEEN Two States

.btn {
  background: steelblue;
  transition: background 0.2s ease;
}
.btn:hover {
  background: darkslateblue;    /* the SECOND state — transition animates FROM the first
                                    (whatever background currently is) TO this, over 0.2s */
}

This is the single most important conceptual point, and the thing that trips up nearly every beginner's first attempt: a transition doesn't create movement by itself — it smooths the change BETWEEN two states that already exist (typically the default state and a :hover/:focus/class-toggled state). Write the transition property, then simply CHANGE the property's value somewhere else (a pseudo-class, a JS-toggled class) — the browser interpolates every intermediate frame automatically. This is fundamentally different from keyframe animations (next lesson), which define an explicit SEQUENCE of states independent of any triggering interaction.

Part 2 — The Shorthand's Real Parameter Order

transition: background-color 0.3s ease-in-out 0.1s;
/*           ^property         ^duration ^easing    ^delay */

transition: all 0.3s ease;    /* "all" — transitions EVERY animatable property that changes —
                                  convenient but a real performance/precision tradeoff (Part 5) */

transition:
  transform 0.3s ease,
  opacity 0.5s ease 0.1s;      /* comma-separated — DIFFERENT properties, DIFFERENT timing, in ONE
                                   declaration */

Order is property, duration, easing function, delay — easing and delay are both optional (defaulting to ease and 0s respectively if omitted). The comma-separated multi- property form is genuinely useful and underused: a card that should scale up QUICKLY on hover but fade its shadow in more SLOWLY is exactly the kind of nuanced, professional-feeling detail this syntax enables in one declaration, rather than forcing every transitioning property to share identical timing.

Part 3 — Easing Functions: What They Actually Communicate

transition-timing-function: linear;         /* constant speed throughout — feels mechanical, robotic */
transition-timing-function: ease;            /* the DEFAULT — starts slow, speeds up, ends slow */
transition-timing-function: ease-in;         /* starts SLOW, accelerates — good for elements LEAVING */
transition-timing-function: ease-out;        /* starts FAST, decelerates — good for elements ARRIVING */
transition-timing-function: ease-in-out;     /* slow-fast-slow, more pronounced than plain "ease" */
transition-timing-function: cubic-bezier(0.34, 1.56, 0.64, 1);  /* CUSTOM curve — can overshoot
                                                                     past 100% before settling
                                                                     ("bounce"/"spring" feel) */

Easing isn't decoration — it maps onto genuine, real-world physical intuition, which is precisely why the wrong easing choice makes an animation feel subtly artificial even when the DURATION and property are otherwise correct: real physical objects don't move at constant speed (linear is specifically why CSS-animated loading spinners/rotations often deliberately use linear — for CONTINUOUS, cyclical motion, constant speed is the natural, expected choice; but for anything with a clear start/end, like a menu opening, our physical intuition expects acceleration/deceleration). ease-out (fast start, slow finish) is the near-universal correct choice for elements ENTERING the screen/menus opening — matching how a real object decelerates as it "arrives" and settles; ease-in for elements LEAVING — accelerating away, matching how something looks when it's being dismissed/discarded. A custom cubic-bezier with values allowing overshoot past 100% is the standard technique behind the "springy," slightly-bouncy feel common in modern polished UI — genuinely engineered, not accidental.

Part 4 — Not Everything Can Be Transitioned

.box { display: none; transition: display 0.3s; }   /* display CANNOT be smoothly interpolated —
                                                          it's a discrete keyword with no meaningful
                                                          "halfway point" between none and block */
.box { height: auto; transition: height 0.3s; }        /* height:auto genuinely can't be transitioned
                                                            EITHER — auto has no fixed numeric value
                                                            for the browser to interpolate FROM/TO */

Two genuinely important, real limitations worth knowing precisely: (1) properties with non-numeric, discrete values (display, most keyword-based properties like visibility's hidden/visible switch) have no meaningful in-between state to interpolate — the browser can only make instant jumps between such values, transition or not (a real, common real workaround: transition opacity alongside a delayed display switch, or use the newer @starting-style + transition-behavior: allow-discrete combo for proper display transitions, a 2024+ CSS feature specifically solving this exact historical gap); (2) height: auto genuinely cannot be transitioned — auto is not a concrete numeric value the browser can interpolate a smooth in-between sequence for, so transitioning "from 0 to auto" silently fails to animate at all (it just jumps). The standard, longstanding workaround: transition to/from an explicit, KNOWN pixel value (via JS reading scrollHeight) instead of auto, or (the more modern, if still-limited- support, answer) the newer calc-size() function specifically designed to solve this exact historical CSS gap.

Part 5 — Deep Dive: transition: all's Real Cost

.card { transition: all 0.3s ease; }    /* convenient, but: */
/* (1) transitions EVERY property that changes, including ones you didn't intend (a padding change
     from a RESPONSIVE media query now animates too, possibly undesirably)
   (2) the browser must WATCH every property for changes, a genuine (if usually small) ongoing cost
   (3) it can accidentally transition properties like width/height/left that force LAYOUT recalculation
       every frame (Critical Rendering Path lesson) — "all" doesn't discriminate between cheap
       (transform/opacity) and expensive (layout-triggering) properties AT ALL */

.card { transition: transform 0.3s ease, opacity 0.3s ease; }   /* explicit — only what's
                                                                     genuinely intended to animate */

The real, practical recommendation: transition: all is fine for small, simple components where the convenience genuinely outweighs the risk — but for anything performance- sensitive or where OTHER properties might change unexpectedly (a responsive breakpoint, a JS-driven class toggle affecting several properties at once), explicit property lists are the more disciplined, predictable choice, directly connecting to the previous lesson's "transform/opacity are cheap, layout-triggering properties are not" distinction — all makes it easy to accidentally transition an expensive property without noticing, simply because it changed as a SIDE EFFECT of something else.

Interview Questions

Q1 (beginner): Why doesn't adding transition: background-color 0.3s to an element make anything visibly animate by itself?

A transition smooths the change BETWEEN two states that must already exist elsewhere (a :hover state, a JS-toggled class) — it doesn't create movement on its own. Without an actual property value change triggered by some interaction or state toggle, there's nothing for the transition to interpolate between, so nothing visibly animates.

Q2 (beginner): Why can't height: auto be smoothly transitioned?

auto is not a concrete numeric value — the browser has no fixed starting or ending number to interpolate a smooth sequence of intermediate frames between. Transitioning to/from auto simply jumps instantly rather than animating. The standard workaround is transitioning to/from an explicit pixel value (often read via JS from scrollHeight) instead of relying on auto directly.

Q3 (mid): When would you choose ease-out over ease-in for a transition, and why does the choice matter?

ease-out (fast start, slow finish) suits elements ENTERING the screen or menus opening — it matches real-world physical intuition of an object decelerating as it arrives and settles into place. ease-in (slow start, fast finish) suits elements LEAVING or being dismissed — accelerating away. Using the "wrong" one for the interaction's actual direction makes the animation feel subtly artificial even though duration and the animated property are otherwise correct, because it contradicts the physical intuition viewers unconsciously expect.

Q4 (mid): What's the practical risk of using transition: all as a default habit, beyond it being "less precise"?

It transitions EVERY property that changes for ANY reason, including unintended side effects (a responsive media query changing padding now animates too, possibly undesirably) — and critically, it doesn't discriminate between cheap, composite-only properties (transform, opacity) and expensive, layout-triggering ones (width, left, margin) at all, making it easy to accidentally transition a genuinely expensive property without noticing, simply because it changed as a side effect of something unrelated.

Q5 (senior): A dropdown menu uses transition: all 0.3s ease and display: none/block toggling to show/hide. QA reports the menu "sometimes doesn't animate at all, just pops instantly," but only intermittently, and the team can't reproduce it consistently in their own testing.

Root cause: display is a discrete, non-interpolatable property (Part 4) — transitioning it always either does nothing (an instant jump) or, depending on exact timing/browser behavior, can produce inconsistent results because the element is being removed from/added to the render tree entirely at the same moment other transitioning properties (perhaps opacity, also being toggled) are trying to animate on an element that's about to stop existing in the render tree. The "intermittent, hard to reproduce" symptom specifically matches this kind of race condition between a discrete property change and a continuous one happening simultaneously — different browsers, or even different runs in the same browser under different load conditions, may resolve the exact timing slightly differently. Fix: separate the CONCERNS explicitly — use opacity (a genuinely interpolatable property) as the actual visual transition, and delay the display switch itself using a technique like transition-behavior: allow-discrete + @starting-style (the modern, 2024+ CSS-native solution built specifically for this exact problem), or the longstanding JS-assisted workaround (set display:block first, wait one animation frame via requestAnimationFrame, THEN trigger the opacity transition, and reverse the order on close, removing display only AFTER the opacity transition's transitionend event fires) — either approach cleanly separates the discrete on/off state from the continuous, genuinely-animatable opacity value, eliminating the race condition entirely rather than fighting an inherently un-transitionable property.

Q6 (senior): Your design system wants a consistent, deliberate "motion language" across the entire product — specific easing curves and durations that feel cohesive rather than every component author picking arbitrary values. Design the system.

The correct architecture treats easing curves and durations as DESIGN TOKENS (directly connecting to custom properties, covered in the Modern CSS section, and the design-tokens concept from the Architecture section ahead) rather than values re-invented per component: define a small, deliberate SET of named durations (e.g., --duration-fast: 150ms for micro-interactions like button hovers; --duration-base: 250ms for typical UI transitions like dropdowns; --duration-slow: 400ms for larger, more significant transitions like page-section reveals) and a small set of named easing curves (--ease-out: cubic-bezier(0, 0, 0.2, 1) for arriving elements; --ease-in: cubic-bezier(0.4, 0, 1, 1) for leaving ones; --ease-spring: a custom overshoot curve for playful/bouncy brand moments, used SPARINGLY and deliberately, not everywhere). Every component author then composes transitions from these shared tokens (transition: transform var(--duration-base) var(--ease-out)) rather than picking arbitrary millisecond/curve values independently — producing a genuinely cohesive, intentional "motion language" across the whole product exactly the way a shared COLOR palette produces visual cohesion, rather than every component's author making an independent, ungoverned aesthetic choice. The system should be DOCUMENTED with the semantic REASONING behind each tier (why fast/base/slow exist as distinct tiers, when to reach for spring vs standard easing) so future component authors apply the system deliberately rather than treating it as an arbitrary lookup table — directly mirroring how the earlier z-index-token interview question (Positioning lesson) argued for semantic tiers over ad-hoc numeric escalation, the identical architectural principle now applied to motion design specifically.

🏋️ Exercises

1 (easy): The "nothing animates" bug, reproduced and fixed. Build a button with transition: background-color 0.3s but with NO :hover (or any other) state change defined at all. Confirm nothing visibly happens on interaction. Add a :hover background-color change and confirm the transition now works.
Expected result

Without any state change: hovering/clicking the button produces zero visible animation, regardless of the transition property being correctly declared — direct, hands-on confirmation of Part 1's core claim that transitions smooth CHANGES, not create movement independently. Adding the :hover background-color value makes the transition immediately, visibly functional.

2 (easy): Easing, compared side by side. Build 4 identical boxes, each transitioning transform: translateX(200px) on hover with a DIFFERENT easing function (linear, ease-in, ease-out, a custom bouncy cubic-bezier). Hover all 4 in sequence and describe the felt difference in each one's "personality."
Expected observations

linear: constant, mechanical-feeling speed throughout. ease-in: a slow, hesitant start that suddenly accelerates near the end. ease-out: a quick, energetic start that gently decelerates and settles — likely the one that "feels most natural" for this specific arriving-motion case. bouncy cubic-bezier: overshoots past the 200px mark slightly before settling back, a genuinely playful, springy feel distinct from all three standard easings.

3 (medium): Multi-property, independent timing. Build a card that on hover: scales up QUICKLY (transform, 0.15s) while its box-shadow grows in more SLOWLY and with a slight delay (0.4s, 0.1s delay) for a layered, staggered feel. Use the comma-separated multi-property transition syntax in one declaration.
Solution 
.card {
  transition: transform 0.15s ease-out, box-shadow 0.4s ease-out 0.1s;
}
.card:hover {
  transform: scale(1.03);
  box-shadow: 0 20px 40px rgb(0 0 0 / 0.15);
}

The scale should complete noticeably before the shadow finishes growing in — a genuinely more polished, layered feel than both properties animating with identical timing, and a direct, practical demonstration of Part 2's underused comma-separated syntax.

4 (medium): The height:auto workaround. Build an expandable FAQ-style accordion panel. First attempt: transition height from 0 to auto directly — confirm it doesn't animate (jumps instantly). Fix it using JS to read the panel's scrollHeight and transition to that explicit pixel value instead.
Solution core 
function expand(panel) {
  panel.style.height = panel.scrollHeight + "px";   // explicit, KNOWN pixel value — genuinely animatable
}
function collapse(panel) {
  panel.style.height = panel.scrollHeight + "px";   // set explicit height first (from current auto state)
  requestAnimationFrame(() => panel.style.height = "0px");  // then transition down, next frame
}

The direct height:auto attempt should visibly jump instantly with no animation at all, confirmed by watching it side by side with the scrollHeight-based fix, which should show a genuinely smooth expand/collapse — a direct, hands-on demonstration of Part 4's real, well-known CSS limitation and its standard workaround.

5 (hard): The display-toggle race condition, reproduced and fixed. Reproduce Q5's exact bug: a dropdown using display:none/block plus transition:all, toggled rapidly (click show, then immediately click hide, repeated quickly several times) — observe inconsistent/ instant-jump behavior. Fix it using the opacity + delayed-display JS technique (or transition-behavior: allow-discrete + @starting-style if your browser supports it) and confirm consistent, reliable animation even under rapid toggling.
Reference fix (JS technique) 
function show(el) {
  el.style.display = "block";
  requestAnimationFrame(() => el.style.opacity = "1");
}
function hide(el) {
  el.style.opacity = "0";
  el.addEventListener("transitionend", () => el.style.display = "none", { once: true });
}

Under rapid repeated toggling, this version should remain visually consistent and correctly animated every time, where the naive display+transition:all version likely showed at least some instances of instant, non-animated jumps — direct, hands-on confirmation of Q5's diagnosis and fix.

6 (hard): The motion-token system, built. Implement Q6's complete design-token system: 3 duration custom properties, 3 easing custom properties, applied consistently across at least 4 different components (a button hover, a card hover, a dropdown open/ close, a modal fade-in) — with zero component using a hardcoded, un-tokenized duration/easing value anywhere. Document which tier you chose for each component and why.
What a good system demonstrates 
:root {
  --duration-fast: 150ms; --duration-base: 250ms; --duration-slow: 400ms;
  --ease-out: cubic-bezier(0, 0, 0.2, 1);
  --ease-in: cubic-bezier(0.4, 0, 1, 1);
}
.btn:hover { transition: transform var(--duration-fast) var(--ease-out); }
.card:hover { transition: transform var(--duration-base) var(--ease-out); }
.dropdown { transition: opacity var(--duration-base) var(--ease-out); }
.modal { transition: opacity var(--duration-slow) var(--ease-out); }

The justification write-up (button=fast for a micro-interaction, modal=slow for a significant, attention-grabbing transition) is the actual skill being demonstrated — showing DELIBERATE reasoning per component rather than uniformly applying one tier everywhere, which would defeat the whole point of having a differentiated tier system in the first place.

🎯 Key Takeaways

  • Transitions smooth a change BETWEEN two existing states — they don't create movement by themselves; a :hover/:focus/class toggle must exist to trigger the animated change.
  • The shorthand order is property, duration, easing, delay — comma-separated multi-property syntax allows genuinely independent timing per property in one declaration.
  • Easing communicates real physical intuition: ease-out for arriving elements, ease-in for leaving ones, linear mainly for continuous/cyclical motion — the wrong choice makes correct duration/properties still feel subtly artificial.
  • Discrete properties (display) and auto values (height: auto) genuinely cannot be smoothly transitioned — standard workarounds separate the discrete/auto concern from a genuinely animatable one (opacity, an explicit pixel height).
  • transition: all is convenient but risks animating unintended property changes and doesn't discriminate between cheap (transform/opacity) and expensive (layout-triggering) properties — explicit lists are the more disciplined choice for performance-sensitive components.
  • A deliberate, tokenized "motion language" (shared duration/easing custom properties, applied consistently) produces cohesive, intentional-feeling motion across a whole product — the same design-system principle as a shared color palette, applied to timing.
