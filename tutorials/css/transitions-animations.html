Transitions & Animations Intermediate

Motion is communication: state changed, this is pressable, look here, this is loading. The design half is taste; the engineering half is the rendering pipeline — knowing which properties animate on the GPU compositor and which force the whole page to re-layout every frame. Get that wrong and your beautiful animation stutters on every mid-range phone.

The Rendering Pipeline (why some properties are cheap)

Style → Layout → Paint → Composite
        (geometry) (pixels) (layers moved on GPU)

Animate WIDTH  → re-runs Layout + Paint + Composite EVERY FRAME, whole affected subtree, main thread
Animate TRANSFORM/OPACITY → only Composite — GPU moves pre-painted layers; runs on the
        compositor THREAD, so it stays smooth even while JavaScript is busy
Animate freelyThink twice (paint)Avoid animating (layout)
transform (translate/scale/rotate), opacitycolor, background-color, filter — repaint, usually fine for small areaswidth height top left margin padding font-size flex-basis — reflow storms
/* ❌ janky drawer */  .drawer { left: -300px; transition: left .3s; }
/* ✅ 60fps drawer  */  .drawer { transform: translateX(-100%); transition: transform .3s; }
/* Same pixels on screen. Entirely different pipeline. */

Transitions — Animating State Changes

.btn {
  background: var(--brand);
  transition: background-color .15s ease, transform .15s ease,
              box-shadow .15s ease;                 /* NAME the properties — never 'all' */
}
.btn:hover  { background: var(--brand-dark); transform: translateY(-1px); }
.btn:active { transform: translateY(0); transition-duration: .05s; }   /* press = instant */

/* Asymmetric timing — menus open deliberately, close quickly: */
.menu { opacity: 0; visibility: hidden; transition: opacity .15s, visibility 0s .15s; }
.menu.open { opacity: 1; visibility: visible; transition: opacity .25s, visibility 0s; }
/* the visibility DELAY keeps it clickable-none after fade-out — the classic fade pattern */
TimingGuidance
Hover/press feedback100–200ms; below ~80ms reads as flicker
Panels, modals, drawers200–350ms; over 500ms feels like lag
Easingease-out for entrances (fast→settle), ease-in for exits, ease default; spring-like: cubic-bezier(0.34, 1.56, 0.64, 1) (overshoots)
Enter vs exitExits slightly faster than entrances — users wait for arrivals, not departures
⚠️ Why never transition: all: it animates properties you didn't intend — including layout ones added later by someone else, and theme-swap color changes that should be instant. It's also how "the page slowly morphs when dark mode toggles" bugs are born. Name what you animate.

Keyframe Animations — Multi-Step & Looping

@keyframes pulse { 0%, 100% { opacity: 1; } 50% { opacity: .35; } }
.skeleton { animation: pulse 1.5s ease-in-out infinite; }

@keyframes slide-in {
  from { transform: translateX(100%); opacity: 0; }
  to   { transform: translateX(0);    opacity: 1; }
}
.toast { animation: slide-in .3s ease-out both; }
/* 'both' = apply from-state before start AND hold end-state after — kills the
   flash-of-final-state bug (animation-fill-mode) */

/* shorthand: name duration timing delay iteration-count direction fill-mode play-state */
animation: pulse 1.5s ease-in-out 0s infinite alternate both running;
Sub-propertyProduction note
animation-fill-mode: bothAlmost always what you want on enter/exit animations
animation-play-state: pausedPause loops when off-screen/tab hidden (IntersectionObserver toggle) — battery respect
animation-direction: alternatePing-pong loops without writing mirrored keyframes
steps(8)Discrete frames — sprite sheets, typewriter effects, blinking cursors
animationend / transitionend eventsJS hook to clean up after motion (remove toast from DOM)

Modern Motion Toolkit

/* 1. View Transitions — page/DOM changes crossfade & morph with ONE call: */
document.startViewTransition(() => updateDOM());
::view-transition-old(root) { animation-duration: .2s; }   /* customizable via pseudo-elements */

/* 2. Scroll-driven animations — progress tied to scroll, ZERO JS, off main thread: */
.progress-bar { animation: grow linear; animation-timeline: scroll(); }
.reveal       { animation: fade-up both; animation-timeline: view(); }  /* animates as it enters viewport */

/* 3. Animating to height:auto (the eternal problem) — the grid trick that works today: */
.accordion__panel { display: grid; grid-template-rows: 0fr; transition: grid-template-rows .3s; }
.accordion__panel.open { grid-template-rows: 1fr; }
.accordion__panel > div { overflow: hidden; }

/* 4. @property — make custom properties animatable (gradient angles, numeric counters): */
@property --angle { syntax: ""; initial-value: 0deg; inherits: false; }
.conic { background: conic-gradient(from var(--angle), …); transition: --angle .4s; }

Accessibility — Non-Negotiable

@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: .01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: .01ms !important;
    scroll-behavior: auto !important;
  }
}

Vestibular disorders make large motion physically nauseating. The blanket kill-switch is the baseline; a refined approach keeps opacity fades (harmless) and removes translations/scaling/parallax. Test via DevTools → Rendering → Emulate prefers-reduced-motion.

Performance Debugging Workflow

1. DevTools → Performance → record the janky interaction
2. Read the frames track: green = fine · red triangles = dropped frames
3. Purple 'Layout' blocks inside the animation = you're animating a layout property
4. Fix: move to transform/opacity; for unavoidable paint, isolate with will-change — SPARINGLY:
   .drawer:hover { will-change: transform; }   /* pre-promote just-in-time, not globally —
                                                   every promoted layer costs GPU memory */

Interview Questions

Q1: Why does transform animate smoothly while left doesn't — mechanically?

left changes geometry → the engine re-runs layout for the affected tree, repaints, recomposites — per frame, on the main thread, competing with JS. transform doesn't affect layout: the element's layer is rasterized once and the GPU re-positions it each frame on the compositor thread — immune even to main-thread jank. Same reason opacity is cheap: it's a compositing operation.

Q2: How do you animate an accordion to height auto?

Historic hacks: max-height (breaks timing on tall content) or JS measuring scrollHeight. Modern: the grid-template-rows 0fr→1fr trick (widely supported), or interpolate-size / calc-size() where available. Mention overflow:hidden on the inner wrapper — graders listen for it.

Q3: What are View Transitions and what problem do they solve?

document.startViewTransition(cb) snapshots old state, runs your DOM update, then animates old→new (crossfade by default; shared elements can morph via view-transition-name) — replacing fragile manual FLIP animation code. Works same-document everywhere modern, and cross-document for MPAs where supported.

Q4: Your loading spinner freezes during data processing. Why, and what are the two fixes?

It's animated on the main thread (or it's a GIF) and JS is blocking. Fix 1: make it a CSS transform/opacity animation — compositor-driven, survives main-thread work. Fix 2: move the processing to a Web Worker so the main thread never blocks. Both together is the production answer.

🎯 Key Takeaways

  • Pipeline: Layout → Paint → Composite. transform/opacity live in Composite — animate those; layout properties reflow per frame.
  • Compositor animations survive main-thread jank — the mechanical reason CSS beats JS for motion.
  • Name transition properties (never all); 100–200ms feedback, 200–350ms panels; ease-out in, ease-in out; fill-mode both.
  • Modern kit: View Transitions, scroll-driven timelines, grid 0fr height-auto trick, @property for animatable variables.
  • prefers-reduced-motion kill-switch ships in every stylesheet, no exceptions.
  • will-change is a scalpel, not a vitamin; the Performance panel's purple layout blocks are the diagnosis.
