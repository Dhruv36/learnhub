The Cascade & Inheritance Foundations

What you'll master: "Cascading" in Cascading Style Sheets, precisely — the full, ordered tie-breaking sequence (not just specificity), which properties inherit and why that's not arbitrary, the inherit/initial/unset/revert keywords, and the practical "why is this value coming from" debugging skill. Prerequisite: Selectors & Specificity — this lesson is specificity's missing context: what happens BEFORE and AFTER it in the full decision process.

Part 1 — The Cascade Is a Multi-Step Sort, Not Just Specificity

Last lesson's specificity algorithm answers ONE step of a larger process. The full cascade, in actual, official order, for any given property on any given element:

1. ORIGIN & IMPORTANCE   (highest to lowest priority)
   a. User-agent !important        (browser's own forced styles — vanishingly rare)
   b. User !important               (a user's own forced accessibility stylesheet — rare)
   c. Author !important             (YOUR !important rules)
   d. Author normal                 (your normal rules — everything you usually write)
   e. User normal                   (a user's own stylesheet, non-forced)
   f. User-agent normal             (the browser's default stylesheet — h1 is bold by default, etc.)

2. CASCADE LAYERS   (if used — Modern CSS section, later) — layer order beats specificity within
   the same origin/importance tier

3. SPECIFICITY   (last lesson's full algorithm) — within the same origin/importance/layer

4. SOURCE ORDER   (last lesson's tie-break) — within identical specificity

The genuinely important, under-known fact hiding in step 1: your normal author styles (1d) beat the user's own normal styles (1e) — but a user's !important override (1b) beats even YOUR !important (1c). This ordering is deliberate and accessibility-driven: it guarantees a user with a personal high-contrast or large-text stylesheet (assistive technology, often) can always force their OWN accessibility overrides to win, no matter how aggressively a site author writes CSS — the cascade's origin ordering is partly a built-in accessibility guarantee, not just a technical tie-break mechanism.

Part 2 — Inheritance: Which Properties Pass Down, and Why

<div style="color: navy; border: 1px solid red;">
  <p>This text is navy — color INHERITED.</p>
  <p>This paragraph has NO border — border did NOT inherit.</p>
</div>

Some properties automatically pass their computed value down to descendant elements if the descendant doesn't set its own value; others never do. This isn't arbitrary — it maps directly onto a genuine design principle: typography-related properties inherit (it would be absurd to re-declare font-family on every single nested span); box-model and layout properties don't (a border/margin/background on a parent visually enclosing it does NOT imply every child should also get that same border — that would make basic page layout nearly unusable).

Inherits by defaultDoes NOT inherit by default
color, font-family, font-size, font-weight, line-height, letter-spacing, text-align, visibility, cursor, list-stylemargin, padding, border, background, width, height, display, position, box-shadow, flex/grid properties

The practical payoff: set typography defaults ONCE, high up (commonly on body or html), and let them flow down through the entire document for free — body { font-family: system-ui; color: #1a1a1a; line-height: 1.6; } styles nearly every text node on the page without a single additional selector, precisely because these ARE the inheriting properties.

Part 3 — The Four Explicit Value Keywords

.child {
  color: inherit;    /* explicitly TAKE the parent's computed value, even for non-inheriting props */
  border: initial;   /* reset to the property's SPEC-DEFINED default (border: none, for example) */
  color: unset;      /* inherit() if the property naturally inherits, initial() if it doesn't — "do
                        the sensible default thing" */
  all: revert;       /* reset EVERY property back to what the USER-AGENT stylesheet would have set —
                        genuinely different from initial; useful for "undo my own CSS entirely" */
}

Concrete use cases: border: inherit forces a non-inheriting property to inherit anyway (rare, deliberate). all: unset is a common, useful reset trick for one specific element that needs to shed ALL its browser/framework default styling and start clean (a button being restyled as a plain inline link, for instance — recall the HTML track's "restyle a button to look like a link" exercise; all: unset is the one-line way to do exactly that cleanly). revert is the more surgical, less commonly needed option — genuinely useful mainly when debugging "what would this look like with NONE of our site's CSS touching it at all."

Part 4 — Worked Example: Diagnosing an Inheritance Surprise

button {
  font: inherit;    /* buttons DON'T inherit font by default — a genuine, deliberate browser exception! */
}

This is a real, famous CSS gotcha worth knowing by name: form controls (button, input, select, textarea) are a deliberate EXCEPTION to normal font inheritance in every browser's default stylesheet — they use a system UI font by default rather than inheriting your page's chosen font-family, which is why a carefully-set body { font-family: "Inter", sans-serif } mysteriously fails to reach your buttons specifically, while reaching every paragraph and heading correctly. The one-line, universally-recommended fix — button, input, select, textarea { font: inherit; } — is genuinely necessary boilerplate nearly every project needs, not a sign you did something wrong; it's compensating for a real, spec-defined default exception, not a bug in your own CSS.

Part 5 — Deep Dive: Computed vs. Used vs. Actual Value (why DevTools shows two numbers sometimes)

.box { width: 50%; }   /* SPECIFIED value: 50% */
                        /* COMPUTED value: 50% (percentages often stay as percentages until layout) */
                        /* USED value: 400px    (the ACTUAL pixel result, once the parent's real
                                                   width is known and layout has actually run) */

DevTools' Computed panel sometimes shows a percentage and sometimes shows a resolved pixel value for the same property — this isn't inconsistent tooling, it's showing you different stages of the SAME value's lifecycle: the value you SPECIFIED, the value the CASCADE resolved it to after inheritance/keywords are applied, and the final USED value only knowable after layout (the HTML track's Critical Rendering Path lesson's Stage 4) actually computes real geometry. Knowing these are three genuinely distinct stages — not tooling inconsistency — is what lets you correctly interpret "why does the Styles panel say 50% but the Computed panel says 400px" without confusion.

Interview Questions

Q1 (beginner): List the full cascade order, not just specificity.

Origin and importance first (user-agent !important > user !important > author !important > author normal > user normal > user-agent normal), then cascade layer order, then specificity (the id/class/type algorithm), then source order as the final tie-break. Specificity is one STEP within this larger, ordered process, not the whole cascade by itself.

Q2 (beginner): Why do typography properties inherit but box-model properties don't?

It reflects a genuine design principle rather than arbitrary rule-making: re-declaring font-family/color on every nested element would be absurdly repetitive, so typography inherits by default; but a parent's border/background/margin visually enclosing it does NOT imply every child should also render that same border — automatic inheritance of box-model properties would make ordinary page layout nearly unusable, so those don't inherit.

Q3 (mid): A button doesn't pick up the page's carefully-set font-family despite every paragraph correctly inheriting it. Diagnose.

Form controls (button, input, select, textarea) are a deliberate exception in every browser's default (user-agent) stylesheet — they use a system UI font rather than participating in normal font inheritance from ancestors. The standard, necessary fix is an explicit "button, input, select, textarea { font: inherit; }" rule, compensating for this real, spec- defined default rather than indicating any mistake in the page's own CSS.

Q4 (mid): Explain the difference between initial, inherit, unset, and revert with one example each.

inherit: color: inherit forces the parent's computed color even on a property that wouldn't normally inherit. initial: border: initial resets to CSS's spec-defined default for that property (none, for border). unset: color: unset acts as inherit if the property naturally inherits, or initial if it doesn't — "do the sensible default." revert: all: revert resets every property back to what the browser's OWN default stylesheet would render, as if none of the page's authored CSS had touched that element at all — genuinely different from initial for many properties (e.g. a browser default like display:list-item on li, which initial would NOT restore but revert would).

Q5 (senior): Your team ships a "high contrast mode" toggle that applies a stylesheet using !important throughout. A user reports their OWN browser extension's custom accessibility stylesheet stopped working once your toggle shipped. Diagnose using the cascade's origin ordering.

The cascade's origin order places author !important (your toggle's stylesheet) ABOVE user normal styles, but genuinely BELOW user !important — so if the user's accessibility extension writes its overrides WITHOUT !important (a normal user stylesheet), your author !important now correctly and predictably wins over it per the cascade spec, which is precisely the reported symptom. This isn't a bug in either stylesheet — it's the cascade behaving exactly as specified. The fix isn't technical (you can't and shouldn't try to detect or defer to arbitrary user stylesheets); it's a product/support conversation: advise the user that accessibility extensions/browser features needing to override site-authored !important styles should themselves use !important (many serious accessibility tools, like browser reader modes and OS-level high-contrast forcing modes, are specifically implemented at the user-agent level — origin tier (f) — or through !important precisely because they need this exact guarantee). The senior insight: the cascade's origin ordering is DESIGNED to let genuine user overrides win when they use the matching importance tier — the "bug" is actually the system correctly enforcing a hierarchy the user's specific tool didn't invoke strongly enough.

Q6 (senior): A design system wants a single "reset" utility class that can be applied to any element to strip ALL site-specific styling, making it render exactly as a totally unstyled browser default would. Compare all/initial vs all/revert vs all/unset for this specific goal, and recommend one.

all: initial resets every property to CSS's SPEC-DEFINED initial values — but many elements' sensible default appearance (an li's bullet, a button's default padding/border, an h1's bold weight and larger size) comes from the browser's USER-AGENT STYLESHEET, not from the spec's bare initial values, so all:initial would strip these too, likely producing a MORE stripped-down appearance than "default browser rendering" — an h1 would lose its bold/size entirely, rendering as plain unstyled text, not as a normal h1. all: revert resets every property back to whatever the USER-AGENT stylesheet specifies for that element — an h1 stays bold and large (its genuine browser default), a li keeps its bullet, a button keeps its native button chrome — which is precisely "render exactly as a totally unstyled browser default would," the stated goal. all: unset falls between: inheriting properties revert to inherited values, non-inheriting properties go to their initial (spec) values — useful for a DIFFERENT goal ("blend into whatever context this element sits in") but not for "look like an untouched browser default," since it still uses initial (not user-agent-default) values for non-inheriting properties. Recommendation: all: revert is the correct choice for the stated goal specifically — it's the only one of the three that resolves to the actual browser default rendering rather than the CSS spec's bare initial values, a distinction many CSS authors genuinely don't realize exists until they hit exactly this use case.

🏋️ Exercises

1 (easy): Inheritance prediction drill. Without testing first, predict which of these would inherit from a parent div to a child span with no explicit styles: color, border-radius, font-weight, background-color, text-transform, box-shadow, cursor. Then verify in a real page.
Answers

Inherit: color, font-weight, text-transform, cursor. Don't inherit: border-radius, background-color, box-shadow — all box-model/visual-decoration properties, not typography. If your prediction matched the general "typography inherits, box-model/decoration doesn't" principle even for properties not explicitly listed in Part 2's table, you've internalized the underlying RULE rather than just memorized a list — that's the actual goal of this exercise.

2 (easy): The button font-fix, applied and explained. Build a page with body { font-family: Georgia, serif } and a button with default styling. Confirm the button does NOT render in Georgia. Add the standard fix and confirm it now does. Explain in one sentence why this fix is "standard boilerplate" rather than a project-specific hack.
Explanation

Every browser's user-agent stylesheet deliberately excludes form controls from normal font inheritance (Part 4) — this isn't specific to any one project's CSS architecture, it's a universal browser default every project encounters identically, which is exactly why "button, input, select, textarea { font: inherit }" appears near-universally across CSS resets/normalize files rather than being something any individual team invented.

3 (medium): all:unset in practice. Take a real <button> with default browser styling and restyle it to look and behave EXACTLY like a plain inline text link (no border, no background, underlined, cursor:pointer, inheriting the surrounding text's font/color) using all:unset plus a minimal set of additional declarations. Confirm it's still a real, keyboard- operable button (HTML track lesson 16's point) despite looking identical to a link.
Solution 
button.link-styled {
  all: unset;
  cursor: pointer;
  text-decoration: underline;
  color: inherit;
}

all:unset strips the button's default padding, border, and background in one line; the three additions restore exactly the link-like visual cues needed. Verify with Tab+Enter/Space that it's still genuinely a functioning button — a direct, hands-on confirmation of the HTML track's "costume vs contract are independent" principle, now demonstrated via the cascade's own reset mechanism.

4 (medium): initial vs revert, observed. Build an <li> with a custom list-style: square and custom color set on it. Apply all:initial to one copy and all:revert to another (identical) copy. Compare: does either retain the bullet marker? Does either look like a "genuinely default, unstyled" li? Explain the difference you observe using Q6's reasoning.
Expected observation

all:initial's li likely loses its bullet marker entirely (list-style's initial spec value is effectively "none"-equivalent territory depending on exact spec details, and importantly the browser's actual default bullet comes from the user-agent stylesheet's list-style-type: disc, not from the property's bare initial value) — it renders MORE stripped than a genuine default li. all:revert's li retains its bullet, because it explicitly reverts to what the browser's own default stylesheet specifies for li elements. This hands-on side-by-side comparison is the clearest possible demonstration of Q6's answer — the difference is often invisible in documentation prose but immediately obvious once rendered side by side.

5 (hard): Cascade origin ordering, simulated. Since simulating a real "user stylesheet" requires browser-specific setup, instead build a conceptual demonstration: three style blocks representing (a) "user-agent-equivalent" defaults, (b) your "author" styles, and (c) an "!important override" block, loaded in a deliberately confusing ORDER in the HTML (c before a before b). Predict which wins for a shared property, then verify, proving cascade origin/importance overrides pure source order.
Solution + expected result 
<style>p { color: red !important; }</style>   <!-- loaded FIRST in source -->
<style>p { color: blue; }</style>              <!-- loaded SECOND, normally would win on source order -->

Despite the blue rule loading LATER (which would normally win a source-order tie at equal specificity/importance), red wins — because !important as an IMPORTANCE tier outranks source order entirely, regardless of load sequence. This directly demonstrates that source order is only the LAST tie-break, consulted after origin/importance and specificity have already been evaluated — exactly Part 1's ordered sequence, proven with a two-line example rather than taken on faith.

6 (hard): Computed vs used value, observed directly. Build a div with width: 50% inside a parent with a specific pixel width. In DevTools, find BOTH the Styles panel's specified value (50%) and the Computed panel's resolved pixel value. Then resize the parent (via a media query or just editing its width live in DevTools) and confirm the COMPUTED pixel value updates while the SPECIFIED 50% in your source CSS never changes at all.
What this proves

The Styles panel's 50% never changes — it's literally your source code, unaffected by anything at runtime. The Computed panel's pixel value DOES change as the parent resizes — because it's the USED value, recalculated by the layout stage (Critical Rendering Path lesson's Stage 4) every time the parent's actual size changes. This is the concrete, hands-on distinction Part 5 described: one number is what you wrote, the other is what the browser currently resolves it to given real, current layout — genuinely different things that happen to share a units-free display in casual DevTools glancing, but are conceptually and mechanically distinct stages of the same property's lifecycle.

🎯 Key Takeaways

  • The full cascade is origin/importance → cascade layers → specificity → source order — specificity is one step within this larger, ordered sequence, not the whole story.
  • User !important genuinely outranks author !important — a deliberate, accessibility-motivated design guaranteeing user overrides (assistive stylesheets) can always win when needed.
  • Typography properties inherit (color, font-family, line-height...); box-model/decoration properties don't (margin, border, background...) — reflecting a real design principle, not arbitrary rule-making.
  • Form controls are a deliberate browser-default exception to font inheritance — the "font: inherit" fix is standard, necessary boilerplate, not a project-specific hack.
  • inherit/initial/unset/revert are four genuinely distinct reset keywords — revert specifically restores browser-default rendering, which initial does NOT (many defaults live in the user-agent stylesheet, not the spec's bare initial values).
  • Specified, computed, and used values are three distinct stages of a property's lifecycle — DevTools' Styles vs Computed panels show different stages, not inconsistent information.
