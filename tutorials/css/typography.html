Typography Foundations

What you'll master: font stacks and the fallback logic behind them, line-height's actual computation rule (a genuinely common source of confusion), @font-face and the font-display tradeoff (extending the HTML track's CLS discussion into full CSS control), variable fonts as a real, modern replacement for shipping many separate font-weight files, and the text-wrap: balance/ pretty properties solving a decades-old typesetting annoyance natively. Prerequisite: Colors & Color Functions.

Part 1 — Font Stacks: Fallback Logic, Not Decoration

body {
  font-family: "Inter", -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif;
}

Each name is tried IN ORDER until one is actually available on the user's system/downloaded successfully — this isn't decorative redundancy, it's a genuine fallback CHAIN: "Inter" (your chosen web font, if it loads), then a sequence of each major OS's own default system font (-apple-system/BlinkMacSystemFont for macOS/iOS, Segoe UI for Windows, Roboto for Android/Chrome OS), then the GENERIC keyword sans-serif as the final, guaranteed-available catch-all every browser must support. Generic keywords (serif, sans-serif, monospace, cursive, fantasy, system-ui) are the only names CSS guarantees will always resolve to SOMETHING — always end a font-family list with one, as the genuine last resort if every named font fails to load.

Part 2 — line-height: the Computation Rule That Actually Matters

p { line-height: 1.5; }       /* UNITLESS — computes as 1.5 × THIS element's OWN font-size */
p { line-height: 24px; }      /* FIXED — always 24px regardless of font-size, INHERITS as 24px */
p { line-height: 150%; }      /* PERCENTAGE — computes ONCE at the parent, then INHERITS as that
                                  fixed computed pixel value, NOT recalculated per descendant! */

This is a genuinely subtle, real distinction worth understanding precisely: unitless line-height inherits as the RAW NUMBER (1.5), and each descendant RECOMPUTES it against its OWN font-size — so a nested element with a larger font-size correctly gets proportionally MORE line-height too. Percentage/fixed-unit line-height inherits as the ALREADY-COMPUTED pixel value — a child with a different font-size inherits the SAME fixed pixel line-height as its parent, which can look proportionally wrong (too cramped for larger text, too loose for smaller text) since it doesn't recompute per element. This is precisely why the overwhelming, near-universal recommendation is unitless line-height — it's the only form that correctly scales WITH each element's own font-size as it inherits down through nested elements of varying sizes, which is almost always the actually-desired behavior.

Part 3 — @font-face and font-display: Full Control

@font-face {
  font-family: "Inter";
  src: url("/fonts/inter-var.woff2") format("woff2-variations");
  font-weight: 100 900;              /* VARIABLE font — one file covers the ENTIRE weight range */
  font-display: swap;
}

The HTML track's Performance lesson covered font-display's CLS tradeoff at a survey level (swap vs. optional); here's the complete value set and when each genuinely fits:

ValueBehaviorBest for
blockInvisible text for a short window, then the custom font (default browser behavior if unset)Icon fonts where a fallback would look broken/wrong (rare to actually need)
swapFallback font shows immediately, swaps to custom font when ready — fastest content visibilityMost body text — the near-universal default choice
fallbackA very brief invisible window, THEN fallback, with only a short remaining window to swap — if the custom font isn't ready by then, it's skipped for this page view entirelyA middle ground when swap's visible layout shift is undesirable but block's long invisible wait is worse
optionalFallback shows immediately and usually never swaps (browser decides based on connection speed/cache) — zero layout shiftDecorative/non-critical text where brand-font consistency matters less than CLS

Part 4 — Variable Fonts: One File, Every Weight

/* THE OLD WAY — 4 separate font files, 4 separate HTTP requests: */
@font-face { font-family: "Inter"; src: url("inter-400.woff2"); font-weight: 400; }
@font-face { font-family: "Inter"; src: url("inter-600.woff2"); font-weight: 600; }
@font-face { font-family: "Inter"; src: url("inter-700.woff2"); font-weight: 700; }
@font-face { font-family: "Inter"; src: url("inter-900.woff2"); font-weight: 900; }

/* THE MODERN WAY — ONE file covers the entire continuous weight range: */
@font-face {
  font-family: "Inter";
  src: url("inter-variable.woff2") format("woff2-variations");
  font-weight: 100 900;      /* a RANGE, not a fixed number */
}
h1 { font-weight: 750; }    /* ANY value in the range works — not just 400/700 stops! */

A variable font packs an entire continuous design-space (weight, and sometimes width/slant/optical- size too) into ONE font file — real, direct performance payoff (one HTTP request instead of several, often a smaller total download than even 2-3 separate static weights combined) AND genuine design flexibility (font-weight: 550 is a real, valid, distinctly-rendered value, not silently rounded to the nearest static weight the way a traditional font family would). This is now the standard, recommended approach for any project using more than 2 weights of the same typeface.

Part 5 — text-wrap: balance and pretty: a Decades-Old Problem, Solved Natively

h1 { text-wrap: balance; }   /* headings: distributes text EVENLY across lines, avoiding a lonely
                                  short last line or a dramatically uneven line-length split */
p  { text-wrap: pretty; }    /* body text: avoids "orphans" (a single short word alone on the
                                  last line) using a slightly more performant algorithm suited
                                  to longer text than balance */

Before this (2023+) feature, avoiding an awkward heading wrap — three words on line one, one lonely word alone on line two — required either manual <br> tags (fragile; breaks the moment the viewport width or the heading text changes) or JavaScript libraries built specifically to solve this ONE cosmetic problem. text-wrap: balance solves it with one line of CSS, genuinely responsively, recalculating automatically at every viewport width with zero JS — a small but real, satisfying example of a long-standing manual workaround becoming unnecessary as the platform absorbs the capability natively, directly continuing the HTML track's repeated "check the platform before reaching for a library" theme.

Interview Questions

Q1 (beginner): Why does a font stack list several font names instead of just one?

It's a fallback CHAIN, tried in order until one is available/loads successfully — your chosen web font first, then each major OS's own default system font as reasonable intermediate fallbacks, ending with a generic keyword (sans-serif, etc.) as the guaranteed-available last resort. This isn't decoration — it's genuine resilience against font-loading failure or unsupported fonts.

Q2 (beginner): Why is unitless line-height (1.5) generally recommended over percentage or fixed-pixel line-height?

Unitless line-height inherits as the raw number and RECOMPUTES against each descendant's own font-size — a nested element with larger text correctly gets proportionally more line-height. Percentage/fixed-pixel line-height computes ONCE at the point it's set and inherits as that fixed pixel value, so a descendant with a different font-size inherits the SAME fixed spacing, which looks proportionally wrong (too cramped or too loose) rather than scaling correctly.

Q3 (mid): What problem do variable fonts solve, concretely, versus shipping several static font-weight files?

Performance (one HTTP request/file instead of several, often smaller total bytes than even 2-3 separate static weights) AND design flexibility (any value across the font's declared weight range, like font-weight: 550, renders distinctly and correctly, rather than being silently rounded to the nearest static weight a traditional multi-file font family would be limited to). One file packs the entire continuous design-space instead of a fixed set of discrete snapshots.

Q4 (mid): Explain what text-wrap: balance does and why it couldn't be reliably achieved before its introduction.

It distributes a heading's text evenly across its wrapped lines, avoiding an awkward lonely short last line or a dramatically uneven line-length split — recalculated automatically and responsively at any viewport width. Before this CSS feature, achieving the same effect required either manual, fragile <br> tags (which break the moment the viewport or the actual heading text changes) or a JavaScript library specifically built to measure and re-balance text — both real, non-trivial workarounds for what's now a single native CSS declaration.

Q5 (senior): A page's body text (using a percentage line-height set on the body element) looks correctly spaced for paragraph text but noticeably cramped inside a component that intentionally uses a larger font-size for callout text. Diagnose and fix.

Diagnosis: the body's percentage-based line-height (e.g., 150%) computed ONCE against the body's own base font-size, and that resulting FIXED pixel value is what inherits down to the callout component — since the callout intentionally uses a larger font-size, it's inheriting a line-height pixel value that was correctly proportioned for the SMALLER base text, now looking cramped relative to its own larger text. Fix: change the body's line-height to a unitless value (e.g., line-height: 1.5) — this is the exact, textbook case Part 2 describes, and switching to unitless resolves it globally in one line, since every element (including the callout) will now correctly recompute 1.5× against ITS OWN font-size rather than inheriting a stale fixed pixel value computed at a different, smaller size.

Q6 (senior): Your team ships a marketing site using 6 different static weight files (300, 400, 500, 600, 700, 900) of the same typeface family, totaling ~380KB, and a performance audit flags font loading as a significant contributor to slow LCP. Propose a remediation and quantify the expected improvement category.

Direct remediation: replace the 6 static files with ONE variable font file covering the 100-900 weight range — variable fonts commonly total LESS than even 2-3 static weights combined (the shared glyph outline data across weights compresses far more efficiently as interpolatable variation data than as 6 fully-separate, redundant glyph sets), so a realistic expectation is dropping from ~380KB across 6 requests to perhaps 80-120KB in a SINGLE request — both a substantial byte reduction AND, critically for LCP specifically, elimination of 5 additional font-fetch round trips that would otherwise each potentially delay text rendering if any heading/body text depends on a weight not yet loaded. Additional, complementary fix given this touches LCP specifically (direct callback to the HTML track's Performance lesson): ensure font-display: swap (or optional, if brand- font consistency matters less than CLS here) is set, and consider a rel="preload" hint (also from that lesson) for the single variable font file specifically, since it's now clearly a critical, page-wide resource worth prioritizing explicitly rather than leaving to default discovery timing. The senior framing: this is simultaneously a bytes-reduction fix (variable font consolidation) AND a request-count reduction fix (6 fetches to 1) AND a render-priority fix (preload) — three complementary levers addressing the same reported LCP symptom from three genuinely different angles, which is a stronger, more complete remediation than any single lever alone.

🏋️ Exercises

1 (easy): Font stack, built correctly. Write a complete, properly-ordered font stack for a project wanting "Poppins" as its primary web font, falling back through the major OS system fonts, ending in a generic. Test it by deliberately misspelling "Poppins" and confirming the page still renders reasonably via the fallback chain rather than breaking.
Solution 
body { font-family: "Poppins", -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif; }

With "Poppins" misspelled (e.g., "Popins"), the browser should silently fall through to the system-font fallbacks — the page should look DIFFERENT (a system font instead of Poppins) but should NOT look broken or use some jarring browser-default serif — direct, hands-on confirmation that the fallback chain is doing real, protective work, not just sitting there unused.

2 (easy): line-height inheritance, observed. Build a body with line-height: 150% containing a nested span with font-size: 2em. Separately, build the same structure with line-height: 1.5 (unitless) instead. Compare the nested span's actual line spacing in both versions using DevTools' Computed panel.
Expected difference

150% version: the span's Computed line-height shows the SAME fixed pixel value as the body's own computed line-height, despite the span's font-size being twice as large — visibly cramped relative to the larger text. Unitless 1.5 version: the span's Computed line-height shows a LARGER pixel value, correctly recalculated as 1.5× the span's OWN (larger) font-size — this is the precise, hands-on distinction Part 2 and Q5 both describe.

3 (medium): font-display, compared side by side. Load a real web font (Google Fonts or similar) four times on one test page, one instance each with font-display: block, swap, fallback, and optional, each styling a distinct, large heading. Using DevTools' Network throttling (Slow 3G), reload and observe the loading sequence/timing differences across all four.
Expected observations

block: the heading is invisible for a noticeable window, then appears already in the custom font. swap: a fallback-font heading appears almost immediately, then visibly swaps to the custom font once it arrives — the most noticeable "flash" of the four. fallback: a brief invisible window, then fallback text, with the custom font swapping in only if it arrives within a short subsequent window (on a throttled connection, it may miss this window and simply stay on the fallback for this page view). optional: fallback text appears immediately and, on a throttled connection, likely NEVER swaps to the custom font at all for this load — the browser judged the connection too slow to justify waiting.

4 (medium): Variable font weight exploration. Using a variable font (Google Fonts has several, e.g. "Inter" or "Roboto Flex"), build a slider (a real input type="range", HTML track lesson 10) that lets you drag through font-weight values from 100 to 900 on a live heading, updating in real time via a tiny bit of JS reading the slider's value. Confirm intermediate values (like 437 or 690) render as genuinely distinct weights, not rounded to a nearby "traditional" stop.
Reference core 
<input type="range" min="100" max="900" value="400" id="weight">
<h1 id="demo" style="font-family: 'Inter Variable', sans-serif;">Variable weight</h1>
<script>
weight.addEventListener("input", () => demo.style.fontWeight = weight.value);
</script>

Dragging the slider should show smooth, continuous weight changes at every value — a direct, tactile demonstration of the "entire continuous design-space in one file" claim from Part 4, genuinely felt rather than just described.

5 (hard): text-wrap: balance, before and after. Build a page with 5 different heading texts of varying lengths (some naturally wrap to an awkward 1-word-left- over second line at certain viewport widths). Screenshot or note their wrapping WITHOUT text-wrap: balance at 3 different viewport widths. Add text-wrap: balance and repeat, comparing how the line distribution changes.
Expected observation

Without balance: at least one heading, at least one tested viewport width, should show a visibly awkward split (e.g., 4 words then 1 lonely word alone on the second line). With balance added: the SAME heading at the SAME viewport width should redistribute more evenly (e.g., 3 words then 2 words, or a similarly balanced split) — and critically, this should hold true automatically across ALL 3 tested viewport widths without any manual adjustment per width, directly demonstrating the "responsive, zero-JS, recalculates automatically" claim.

6 (hard): The full remediation, applied. Simulate Q6's scenario: build a page loading 6 separate static @font-face weight declarations (fake/placeholder URLs are fine if you don't have real files) of one typeface. Note the six separate @font-face blocks and six implied HTTP requests. Rewrite it as ONE variable-font @font-face declaration with a weight range, plus font-display: swap and a rel="preload" hint for the single file. Write a two-sentence summary of the three distinct improvements this rewrite delivers (bytes, requests, render-priority).
Solution + summary 
<link rel="preload" href="/fonts/inter-variable.woff2" as="font" type="font/woff2" crossorigin>
<style>
@font-face {
  font-family: "Inter";
  src: url("/fonts/inter-variable.woff2") format("woff2-variations");
  font-weight: 100 900;
  font-display: swap;
}
</style>

Summary: this single change reduces total font bytes (one variable file vs. six redundant static files), reduces HTTP request count from six to one (removing five round-trips that could each independently delay text rendering), and the preload hint explicitly raises this now- critical single resource's fetch priority ahead of where the browser's default discovery timing would otherwise place it — three genuinely distinct performance levers, each addressing a different dimension of the original LCP complaint from Q6.

🎯 Key Takeaways

  • Font stacks are genuine fallback chains (chosen font → OS system fonts → a guaranteed generic keyword) — never omit the final generic.
  • Unitless line-height recomputes correctly against each descendant's own font-size as it inherits; percentage/fixed-unit line-height inherits as a stale, already-computed pixel value — unitless is the near-universal correct default.
  • font-display's four values (block/swap/fallback/optional) trade off invisible-text time against layout-shift risk — swap is the default choice for most body text, optional for CLS-sensitive, brand-flexible cases.
  • Variable fonts pack an entire continuous weight (and sometimes width/slant) range into one file — better performance (fewer requests, often fewer total bytes) AND genuine design flexibility versus several separate static-weight files.
  • text-wrap: balance/pretty solve heading/paragraph line-wrapping awkwardness natively and responsively, replacing fragile manual <br> tags or JS libraries built for this one purpose.
  • Typography performance issues (slow LCP from font loading) are often addressable from multiple independent angles at once — bytes, request count, and render priority — rather than a single fix.
