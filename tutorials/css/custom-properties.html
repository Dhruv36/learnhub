Custom Properties & Theming Advanced

Custom properties ("CSS variables") are not Sass variables in new clothes. Sass variables compile away into fixed values; custom properties are live values in the cascade — they inherit down the tree, can be scoped to any subtree, respond to media queries, and can be read and written from JavaScript at runtime. That difference is the entire mechanism behind dark mode, design tokens, and themable component libraries.

Mechanics You Must Actually Know

:root { --brand: #2f8d46; --space-3: 1rem; }        /* :root ≈ global scope */
.card  { --pad: 1.25rem; padding: var(--pad); }       /* scoped: this subtree only */
.btn   { background: var(--btn-bg, var(--brand, #333)); }   /* nested fallbacks */
BehaviorDetail (interview-grade)
They inheritLike color/font — set --gap on a container, every descendant's var(--gap) sees it (unless re-set closer)
Case-sensitive--Brand ≠ --brand (unlike property names)

The Three-Tier Token Architecture (how Material/Polaris/Primer work)

/* Tier 1 — RAW PALETTE. Never referenced by components. */
:root { --green-600: oklch(0.55 0.15 150); --gray-900: #0d1117; --gray-50: #f6f8fa; }

/* Tier 2 — SEMANTIC. What things MEAN. Components use ONLY these. */
:root {
  --color-action:      var(--green-600);
  --color-bg:          #ffffff;
  --color-surface:     var(--gray-50);
  --color-ink:         #1f2328;
  --color-border:      #d0d7de;
}

/* Tier 3 — COMPONENT (optional). Public knobs for one component. */
.btn { --btn-bg: var(--color-action); background: var(--btn-bg); }

Why three tiers: a rebrand edits tier 1 only. Dark mode swaps tier 2 only. A one-off CTA overrides tier 3 only. Components never change for any of these — that isolation is what "design system" means in CSS terms. The discipline that makes it stick: ban raw values in components via a stylelint rule (declaration-property-value-allowed-list or a custom plugin) — consistency stops being a code-review plea and becomes a build failure.

Dark Mode — the Complete Production Pattern

/* 1. Defaults (light) + OS preference: */
:root { --color-bg: #fff; --color-ink: #1f2328; --color-surface: #f6f8fa; color-scheme: light dark; }
@media (prefers-color-scheme: dark) {
  :root { --color-bg: #0d1117; --color-ink: #e6edf3; --color-surface: #161b22; }
}
/* 2. EXPLICIT user choice (data-theme set by JS) — must beat the OS default: */
:root[data-theme="light"] { --color-bg: #fff;     --color-ink: #1f2328; --color-surface: #f6f8fa; }
:root[data-theme="dark"]  { --color-bg: #0d1117; --color-ink: #e6edf3; --color-surface: #161b22; }

/* 3. Components mention ONLY semantics — zero theme awareness: */
body { background: var(--color-bg); color: var(--color-ink); }
<script>
/* 4. The FOUC killer: decide theme SYNCHRONOUSLY in <head>, before first paint */
document.documentElement.dataset.theme =
  localStorage.theme ?? (matchMedia("(prefers-color-scheme: dark)").matches ? "dark" : "light");
</script>

Details that separate polished from amateur: color-scheme makes scrollbars/form controls match; dark surfaces use elevation via lighter surfaces (not shadows); pure #fff text on dark halos — use #e6edf3-ish; re-check every contrast pair in dark (it fails more often).

Component APIs via Scoped Variables

/* The component exposes knobs with defaults: */
.badge { background: var(--badge-bg, var(--color-surface));
         color: var(--badge-ink, var(--color-ink)); }

/* Contexts override without touching the component: */
.header  .badge { --badge-bg: #fff3cd; }
.sidebar .badge { --badge-bg: #dafbe1; }

/* Density/size systems — one multiplier re-scales everything: */
:root { --density: 1; }
[data-density="compact"] { --density: 0.8; }
.row { padding-block: calc(0.75rem * var(--density)); }

The JS Bridge — Runtime Superpowers

getComputedStyle(el).getPropertyValue("--color-action");   // read (returns raw string, trim it)
el.style.setProperty("--accent", userPickedColor);           // write — instant re-theme, no reflow of CSS files
/* Real uses: user-configurable accent colors · charts colored from data ·
   cursor-following gradients (--x/--y from mousemove) · progress bars (--pct) */

And @property registration upgrades a variable from "string blob" to typed + animatable:

@property --pct { syntax: ""; initial-value: 0%; inherits: false; }
.meter::before { width: var(--pct); transition: --pct .4s ease; }   /* animates smoothly! */

Interview Questions

Q1: What can custom properties do that Sass variables fundamentally cannot?

Exist at runtime: inherit and re-resolve per element, change via media queries/classes/JS without recompilation, power user-controlled theming, and animate (with @property). Sass variables are gone by the time the browser sees the CSS — one compiled value, forever.

Q2: Explain "invalid at computed-value time" and why it surprises people.

var() substitution happens after parsing, so the parser can't reject color: var(--x) when --x is 12px. At computed-value time the declaration becomes invalid — and the spec says the property falls back to unset (inherit or initial), NOT to your var() fallback (which only covers a MISSING variable). Result: text silently inheriting body color instead of turning red. @property's typed syntax restores sane behavior (invalid → initial-value).

Q3: Design a theming system where a user picks any accent color. Sketch it.

Store the pick, set --accent on :root from JS at startup (synchronously) and on change. Derive the palette in CSS: hover = oklch(from var(--accent) calc(l - .07) c h), tints via color-mix, and auto-contrast foreground via lightness check (or color-contrast() where available). Components consume only --color-action, which tier-2 maps to var(--accent). One variable in, whole coherent theme out.

Q4: How do design tokens reach both CSS and JS/native without drifting?

Single source (JSON/YAML tokens) compiled by style-dictionary-style tooling into CSS custom properties, TS constants, and platform formats. The CSS variables are a build ARTIFACT — edits go to the token source, CI regenerates all targets, drift becomes impossible.

🏋️ Practice (Playground):
1. Build the full dark-mode stack: tiers, media query, data-theme override, and a toggle button — verify OS-follow AND manual override.
2. Reproduce the invalid-at-computed-value surprise: --x: 12px; color: var(--x, red) — inspect what actually renders.
3. Make a progress meter animated via a registered @property percentage.
4. Build the density system (compact/comfortable) with one --density multiplier across paddings.
5. Wire a color input to setProperty('--accent', …) and derive hover/tint shades with color-mix.

🎯 Key Takeaways

  • Custom properties are live cascade values: inheriting, scopable, media-query-swappable, JS-writable — Sass can't do any of that.
  • Three tiers: raw palette → semantic → component knobs. Components speak semantics only; lint out raw values.
  • Dark mode = tier-2 swap via prefers-color-scheme + data-theme override + synchronous head script (no flash) + color-scheme for native widgets.
  • Fallbacks cover MISSING variables, not INVALID values — @property adds types, defaults, and animatability.
  • calc() glues variables to math; var() can't live inside media query conditions — build-time tokens there.
  • Tokens compiled from one source to CSS+JS+native is how orgs keep 40 teams consistent.
