Custom Properties & Theming Modern CSS

What you'll master: custom properties as genuinely LIVE, cascade-and-inheritance- participating values (not compile-time constants like Sass variables), the real mechanics of scope and the fallback value syntax, runtime theming via a single class toggle, and @property for typed, animatable custom properties. This lesson formalizes the pattern used implicitly throughout the Colors, Positioning, and Transitions lessons already. Prerequisite: Keyframe Animations; the Cascade & Inheritance lesson's inheritance mechanics are directly assumed here.

Part 1 — The Fundamental Distinction: Live CSS, Not a Compile-Time Constant

:root {
  --brand-color: #1a73e8;
}
.btn {
  background: var(--brand-color);
}

If you've used Sass/Less variables, the single most important thing to un-learn: a CSS custom property is a genuine, live runtime value that participates fully in the cascade and inheritance — it can be read and CHANGED by JavaScript at runtime (element.style.setProperty(...)), it can be OVERRIDDEN at any level of the DOM tree (a component-scoped override, a media-query-scoped override, a JS-toggled theme class), and its current value can differ for the SAME property name depending on which element you inspect, because it genuinely inherits and cascades like any other CSS property. A Sass variable, by contrast, is resolved once, at BUILD time, into a fixed literal value baked directly into the compiled CSS — it cannot change after the CSS is generated, cannot be read/modified by JavaScript, and has no concept of "current value at this specific element" at all. This distinction is the entire reason custom properties enable genuine runtime theming (Part 3) where Sass variables structurally cannot.

Part 2 — Scope, Cascade, and Fallback Values

:root { --gap: 16px; }              /* defined globally, on the root — the common, default scope */

.card {
  --gap: 24px;                       /* OVERRIDDEN locally, for this component and its descendants only */
  padding: var(--gap);                /* resolves to 24px HERE, and 16px everywhere else on the page */
}

.tight-card {
  padding: var(--gap, 8px);           /* FALLBACK — used ONLY if --gap is genuinely undefined/unset
                                         at this point in the cascade, not merely "falsy" */
}

Custom properties follow the EXACT SAME cascade and inheritance rules the Foundations section's Cascade lesson established for every other CSS property — a --gap defined inside .card is scoped to that element and its descendants, exactly like any inherited value, and doesn't leak out to affect sibling elements outside .card. The var(--name, fallback) syntax's second argument is a genuine fallback used specifically when the custom property is entirely UNDEFINED at that point (never declared anywhere in the relevant scope) — a real, useful safety net for a component library shipped for OTHER teams to consume, where you can't guarantee every consuming project will have defined every expected custom property.

Part 3 — Runtime Theming: the Actual Payoff

:root {
  --bg: #ffffff;
  --text: #1a1a1a;
  --card-bg: #f5f5f5;
}
[data-theme="dark"] {
  --bg: #1a1a1a;
  --text: #f0f0f0;
  --card-bg: #2a2a2a;
}
body { background: var(--bg); color: var(--text); }
.card { background: var(--card-bg); }

<script>
document.documentElement.dataset.theme =
  localStorage.getItem("theme") || "light";
document.getElementById("toggle").addEventListener("click", () => {
  const next = document.documentElement.dataset.theme === "dark" ? "light" : "dark";
  document.documentElement.dataset.theme = next;
  localStorage.setItem("theme", next);
});
</script>

This is the concrete, complete payoff of Part 1's "live, not compile-time" distinction: an ENTIRE site's color scheme switches INSTANTLY, with zero page reload and zero re-fetching of any CSS file, by toggling ONE data attribute — every element referencing var(--bg), var(--text), etc. automatically re-resolves to the new values the instant the attribute changes, because those references are genuinely LIVE bindings, not baked-in literals. This exact pattern (a root-level data attribute or class toggling a set of redefined custom properties) is the real, production-standard technique behind virtually every "light/dark mode toggle" or "brand theme switcher" feature you've ever used on a real website — and it directly composes with the Colors lesson's prefers-color-scheme media query (which can set the SAME custom properties automatically based on OS preference, with the JS toggle serving as a manual override on top).

Part 4 — Composing Custom Properties: Building Complex Values From Simple Parts

:root {
  --hue: 217;
  --saturation: 83%;
}
.btn         { background: hsl(var(--hue) var(--saturation) 51%); }
.btn:hover   { background: hsl(var(--hue) var(--saturation) 45%); }
.btn-warning { --hue: 38; }    /* changing ONE variable re-derives EVERY dependent value automatically */

This directly extends the Colors lesson's HSL-palette-from-one-hue technique into a genuinely live, reusable system: because --hue is referenced (not copy-pasted) everywhere it's needed, overriding it ONCE on .btn-warning automatically cascades that new hue through EVERY property that derives from it — the base background, the hover state, anything else built on top — with zero duplication and zero risk of the derived values drifting out of sync, directly resolving the exact "hardcoded hover colors silently drift when the base changes" problem the Colors lesson's senior interview question raised, now shown as a complete, general pattern rather than one specific example.

Part 5 — @property: Typed, Animatable Custom Properties

@property --progress {
  syntax: "<percentage>";
  initial-value: 0%;
  inherits: false;
}
.bar {
  --progress: 0%;
  background: conic-gradient(steelblue var(--progress), #eee 0);
  transition: --progress 0.5s ease;    /* WITHOUT @property, this genuinely doesn't work — see below */
}
.bar:hover { --progress: 75%; }

This is a real, important, and somewhat surprising limitation worth knowing precisely: by default, a plain custom property (declared without @property) is treated as an opaque, untyped STRING as far as the browser's animation/transition engine is concerned — you genuinely cannot transition or animate a plain custom property's value, even if it looks like a number/percentage to a human reader, because the browser has no way to know it's SUPPOSED to be interpolatable rather than an arbitrary string. @property (a real, if still-recent, CSS feature) explicitly REGISTERS a custom property with a genuine TYPE (<percentage>, <color>, <length>, etc.) — once registered this way, the browser CAN correctly interpolate it smoothly across a transition or animation, unlocking effects (an animatable percentage-based conic-gradient "progress ring," for instance) that are genuinely impossible with plain, untyped custom properties.

Interview Questions

Q1 (beginner): What's the fundamental difference between a CSS custom property and a Sass/Less variable?

A custom property is a genuine, live runtime value that participates in the cascade and inheritance, can be read/modified by JavaScript at any time, and can differ in its resolved value at different points in the DOM tree. A Sass/Less variable is resolved once at BUILD time into a fixed literal value baked into the compiled CSS — it cannot change after compilation and has no runtime presence or JS accessibility at all.

Q2 (beginner): How does a single class/attribute toggle switch an entire site's theme instantly, with no CSS re-fetch?

Every themed property (background, text color, etc.) references a custom property via var(...) rather than a hardcoded value. A root-level selector (like [data-theme="dark"]) redefines those custom properties' VALUES. Because var() references are live bindings (Part 1), the instant the data-theme attribute changes, every element referencing those custom properties automatically re-resolves to the new values — no CSS file changes, no page reload, just the cascade re-evaluating with new custom property values in scope.

Q3 (mid): Why can't a plain custom property (declared without @property) be smoothly transitioned or animated, even if its value looks like a number?

By default, custom properties are treated as opaque, untyped strings by the browser's animation engine — it has no way to know a given custom property is SUPPOSED to be an interpolatable number/ percentage/color rather than an arbitrary string value, so it can't compute meaningful intermediate values for a smooth transition. @property explicitly registers the property with a genuine type (via its syntax descriptor), which is what allows the browser to correctly interpolate it across a transition or animation.

Q4 (mid): A design system defines --hue and derives a button's background, hover, and active states all from hsl(var(--hue) ...). What happens when a --hue override is applied to one specific button variant, and why is this better than three separately hardcoded color values per variant?

Overriding --hue on that specific variant automatically re-derives EVERY property that references it (background, hover, active) through the cascade, with zero additional code — changing ONE value updates every dependent value consistently. Three separately hardcoded colors per variant would require manually keeping all three in sync any time the variant's color needs to change, with a real risk of them silently drifting apart over time (exactly the problem the Colors lesson's color-mix() discussion addressed for hover states specifically, now generalized to a complete custom-property-composition pattern).

Q5 (senior): Your team's dark-mode implementation toggles a data-theme attribute and redefines ~30 custom properties per theme. A new component added by another team hardcodes several colors directly (not using var()) "because it was faster," and now looks broken/inconsistent in dark mode. Propose both an immediate fix and a systemic prevention.

Immediate fix: audit the new component's hardcoded color values and replace each with the appropriate existing themed custom property (var(--bg), var(--text), var(--card-bg), etc.) from the established theming system — a straightforward, mechanical fix once the offending hardcoded values are identified (a simple grep for hex codes / rgb()/hsl() literals in the component's CSS is often sufficient to find them). Systemic prevention: (1) a stylelint rule (declaration-property-value-disallowed-list, or a custom rule) specifically flagging hardcoded color values in component CSS, requiring var(--...) usage instead — catching this class of mistake in CI before merge rather than after a visible dark-mode bug ships; (2) clear, discoverable documentation of the full themed custom-property list (ideally auto-generated from the actual :root/[data-theme] definitions, so it can't drift out of sync with the real available properties) so new contributors know what's available rather than reaching for hardcoded values out of unfamiliarity; (3) a dark-mode visual regression test (or even a simple manual QA checklist item) specifically triggered for any new component's PR, catching exactly this kind of "looks fine in light mode, breaks in dark mode" bug before it reaches production. The senior framing: the individual bug is a quick, mechanical fix, but its recurrence (a "systemic" pattern implied by "another team" not knowing the convention) argues for tooling/process prevention (lint rule + documentation + testing checkpoint) rather than relying on every future contributor independently remembering a convention nobody enforced for them.

Q6 (senior): Design a multi-brand theming system (the same component library, white-labeled for 5 different client brands, each with their own color palette, but sharing identical component structure/behavior) using custom properties.

Architecture: define a layer of SEMANTIC custom properties (--color-primary, --color-surface, --color-text, --color-border...) that every component's CSS references exclusively — components never reference a brand's specific hue/hex value directly, only these semantic names. Each of the 5 brands then gets its OWN selector block (e.g., [data-brand="acme"], [data-brand="globex"]...) redefining ONLY these semantic custom properties to that brand's actual palette values — the component CSS itself is written ONCE and never needs to know which brand is currently active, exactly mirroring the light/dark theming pattern from Part 3, generalized from 2 themes to 5 BRANDS. For brands needing a full palette derived from one base hue (directly reusing Part 4's composition technique), each brand block might additionally define its own --brand-hue and --brand-saturation, with the semantic properties themselves derived via hsl(var(--brand-hue) ...) rather than being independently hardcoded per brand — meaning a brand's ENTIRE themed palette can be adjusted by changing just 1-2 raw values, not 10+ individually-matched semantic properties. Runtime brand switching (if genuinely needed, e.g., an internal preview tool for trying different brands) works identically to Part 3's dark-mode JS toggle — set data-brand on the root, and every component's colors re-resolve instantly with zero re-render logic needed anywhere in the actual component code, since the components were never brand-aware in the first place, only semantic- property-aware. The senior architectural insight: the KEY design decision is the semantic layer (--color-primary, not --acme-blue) — component code depending on semantic names rather than brand- specific names is what makes the SAME component library genuinely, fully brand-agnostic, with all 5 brands' actual visual identity living entirely in 5 small, isolated selector blocks rather than scattered conditionally throughout component logic.

🏋️ Exercises

1 (easy): Live vs compile-time, proven. Set --spacing: 16px on :root, use it in 3 different elements' padding. In the DevTools Console, run document.documentElement.style.setProperty('--spacing', '32px') and observe ALL THREE elements update their padding simultaneously, live, with no page reload.
Expected result

All 3 elements should visibly, instantly update their padding to 32px the moment the Console command runs — direct, hands-on proof that custom properties are genuinely live runtime bindings a script can manipulate, fundamentally different from a Sass variable that would need a full CSS recompilation and page reload to change at all.

2 (easy): Scope, demonstrated. Define --gap: 16px on :root and override it to --gap: 32px inside ONE specific .card element. Confirm via DevTools' Computed panel that elements INSIDE that card resolve --gap to 32px, while elements OUTSIDE it (siblings, or other cards without the override) still resolve to 16px.
Expected result

Inspecting an element inside the overridden .card should show --gap computing to 32px in the Computed panel; inspecting an element in a DIFFERENT card (or elsewhere on the page) should show 16px — direct, hands-on confirmation that custom properties genuinely scope and inherit following the exact same rules as any other CSS property, not some special global-only mechanism.

3 (medium): Full light/dark theme toggle, built. Implement Part 3's complete pattern: a :root theme (light) and a [data-theme="dark"] override defining at least 4 semantic custom properties (background, text, card background, border color), a toggle button, and localStorage persistence so the choice survives a page reload.
Verification checklist

Toggling should instantly switch every themed element's colors with zero flash/reload. Reloading the page after toggling to dark should IMMEDIATELY show dark mode (from the localStorage read on page load), not briefly flash light-mode-then-dark — if you observe a flash, the theme- reading script needs to run BEFORE the page's visible content renders (typically an inline script in the head, not a deferred one at the end of body) to avoid this real, common "flash of incorrect theme" bug.

4 (medium): Composed color system, built. Build Part 4's --hue/--saturation composition pattern completely: a base button style deriving background AND hover state from shared --hue/--saturation properties, plus 2 button variants (e.g., .btn-danger, .btn-success) each overriding ONLY --hue. Confirm both variants' hover states correctly, automatically derive from their overridden hue with zero additional CSS written for the hover states themselves.
Solution 
.btn { --hue: 217; --sat: 83%; background: hsl(var(--hue) var(--sat) 51%); }
.btn:hover { background: hsl(var(--hue) var(--sat) 43%); }
.btn-danger  { --hue: 4; }
.btn-success { --hue: 142; }

Both variant buttons' hover states should correctly darken THEIR OWN hue (red darkening to a darker red, green to a darker green) despite the :hover rule itself being written ONCE, generically — direct confirmation that the composition correctly cascades through to derived states without any variant-specific hover code.

5 (hard): @property, animated. Build Part 5's animatable progress-ring example completely: a --progress custom property registered via @property with syntax "<percentage>", driving a conic-gradient, transitioning smoothly on hover. First build it WITHOUT @property and confirm the transition fails (instant jump, no smooth animation). Then add @property and confirm smooth interpolation.
Expected before/after

Without @property: hovering should show the conic-gradient's angle jump INSTANTLY to the new percentage, despite the transition being declared — direct, hands-on proof of Part 5's "plain custom properties can't be animated" limitation. With @property registered: the SAME hover interaction should now show the gradient smoothly sweeping from the old percentage to the new one over the transition's duration — a genuinely striking before/after difference for identical-looking CSS otherwise.

6 (hard): The 5-brand theming system, built. Implement Q6's complete architecture: a semantic custom-property layer (--color-primary, --color-surface, --color-text) that a single shared "card" component references exclusively, plus 3 different [data-brand="..."] blocks each redefining these semantic properties to a distinct palette. Build a brand-switcher control and confirm the SAME card component instantly re-themes correctly across all 3 brands with zero changes to the card's own CSS.
Solution shape 
.card { background: var(--color-surface); color: var(--color-text); border: 1px solid var(--color-primary); }

[data-brand="acme"]   { --color-primary: #1a73e8; --color-surface: #f0f6ff; --color-text: #0d2b52; }
[data-brand="globex"] { --color-primary: #d93025; --color-surface: #fff0ef; --color-text: #5c110c; }
[data-brand="initech"]{ --color-primary: #188038; --color-surface: #eefbf1; --color-text: #0c3d1c; }

Switching data-brand on the root should instantly re-theme the SAME .card markup with zero changes to .card's own CSS rules — direct confirmation that the semantic-property layer genuinely decouples component styling from brand-specific values, exactly as Q6's architecture described.

🎯 Key Takeaways

  • Custom properties are genuinely LIVE runtime values participating in the cascade and inheritance — readable/writable by JS at any time — fundamentally unlike Sass/Less variables, which resolve once at build time into fixed literals.
  • Custom properties scope and inherit exactly like any other CSS property; var(--name, fallback)'s fallback applies only when the property is entirely undefined, not merely falsy.
  • Runtime theming (instant light/dark or brand switching, zero reload) is the direct, concrete payoff of custom properties being live — a single root-level class/attribute toggle re-resolves every referencing element instantly.
  • Composing derived values (hover/active states) from a shared base custom property eliminates the "hardcoded values silently drift" problem — override the ONE source value to correctly re-derive everything built on it.
  • Plain custom properties are untyped, opaque strings to the animation engine and genuinely cannot be transitioned — @property registers an explicit type, unlocking real interpolation.
  • A semantic custom-property layer (--color-primary, not a brand-specific name) is the architectural key to genuinely brand-agnostic, multi-theme component libraries — components depend on semantics, never on any specific theme's literal values.
