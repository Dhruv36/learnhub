Flexbox Layout

What you'll master: the one-dimensional mental model that explains every flexbox property's naming, the full axis/alignment system (main vs cross), the flex-grow/shrink/basis algorithm precisely (not just "flex: 1"), gap's clean replacement of margin-based spacing hacks, and the specific, common flexbox bugs (unwanted shrinking, wrong wrap behavior) with their actual causes. Prerequisite: Positioning — this lesson is the direct payoff of "flexbox/grid are the tools for sibling-sharing-space layouts" from that lesson's close.

Part 1 — The Core Mental Model: One Dimension at a Time

.container { display: flex; }   /* direct children become FLEX ITEMS, arranged along ONE axis */

Flexbox is fundamentally a one-dimensional layout system — it excels at arranging items along a SINGLE row or a SINGLE column, distributing space and alignment along that one axis intelligently. (Grid, the next lesson, is flexbox's two-dimensional counterpart — rows AND columns simultaneously.) Every flexbox property's confusing "main axis" vs "cross axis" naming exists precisely because flexbox doesn't assume you're always working horizontally — the axis can be row (default, horizontal) or column (vertical), and the property names deliberately describe relationships to WHICHEVER axis is currently "main," not fixed horizontal/vertical terms.

Part 2 — The Axis System: Main vs Cross

.row    { display: flex; flex-direction: row; }      /* MAIN axis = horizontal (the default) */
.column { display: flex; flex-direction: column; }   /* MAIN axis = vertical — everything below
                                                          "rotates" 90° conceptually */

/* MAIN-axis alignment (row example): */
justify-content: flex-start | flex-end | center | space-between | space-around | space-evenly;

/* CROSS-axis alignment (row example — this is VERTICAL alignment, for a row!): */
align-items: flex-start | flex-end | center | stretch | baseline;

The naming is precise, not arbitrary: justify-content ALWAYS controls distribution along the MAIN axis (however it's currently oriented); align-items ALWAYS controls alignment along the CROSS axis (perpendicular to main). Switch flex-direction from row to column and BOTH properties' real-world visual effect rotates 90° WITHOUT changing a single line of code that uses them — justify-content: center now centers VERTICALLY instead of horizontally, because it's still doing the identical job (center along the main axis), just on a now-vertical main axis. Internalizing "main/cross," not "horizontal/ vertical," is what makes flexbox's naming suddenly make complete sense rather than feeling arbitrary.

justify-content valueDistribution
flex-start / flex-endAll items packed at the start/end of the main axis
centerAll items packed together, centered
space-betweenEqual gaps BETWEEN items only — first/last items flush against the edges
space-aroundEqual space AROUND each item (so edge gaps are HALF the between-item gaps)
space-evenlyTruly equal space everywhere, including edges — the most visually "even" option

Part 3 — flex-grow, flex-shrink, flex-basis: the Real Algorithm

.item {
  flex-grow: 1;      /* how much EXTRA space this item claims, relative to siblings' grow values,
                         once all items' basis sizes are satisfied — 0 by default (claims none) */
  flex-shrink: 1;    /* how much this item SHRINKS, relative to siblings, if there's NOT enough
                         space — 1 by default (shrinks proportionally) */
  flex-basis: auto;  /* the item's STARTING size before grow/shrink are applied — auto uses its
                         natural content size, or an explicit width/height if set */
}

.item { flex: 1; }              /* SHORTHAND for flex-grow:1, flex-shrink:1, flex-basis:0 —
                                    "share available space EQUALLY, ignoring natural content size" */
.item { flex: 1 1 auto; }        /* the "flexible but content-aware" pattern — grows/shrinks, but
                                    starts from natural content size, not zero */
.item { flex: 0 0 auto; }        /* "never grow, never shrink, just be your natural size" —
                                    the pattern for a fixed-size item beside flexible ones */

The real algorithm, step by step: (1) every item starts at its flex-basis size; (2) the browser computes whether the total of all basis sizes is LESS than the container (extra space to distribute — flex-grow governs this) or MORE than the container (deficit to absorb — flex-shrink governs this); (3) extra/deficit space is distributed PROPORTIONALLY according to each item's grow/shrink value relative to the SUM of all items' grow/ shrink values — an item with flex-grow: 2 claims TWICE as much of the extra space as a sibling with flex-grow: 1, not simply "twice its own size." This is why flex: 1 (shorthand for grow:1, shrink:1, basis:0) is the specific, common pattern for "N equal-width columns" — starting every item's basis at 0 means the ENTIRE container width becomes "extra space" to distribute, and equal grow values of 1 split it perfectly evenly regardless of each item's actual content length.

Part 4 — The flex-shrink Trap (a Real, Common Bug)

.toolbar {
  display: flex;
}
.toolbar-icon {
  width: 24px;
  height: 24px;
  /* flex-shrink DEFAULTS TO 1 — meaning this icon WILL shrink below its declared 24px if the
     toolbar runs out of space, distorting/squishing it! */
}
.toolbar-icon {
  flex-shrink: 0;   /* THE FIX — "never shrink below your natural/declared size" */
}

This is a genuinely common, real bug specifically because it's counter-intuitive: setting an explicit width on a flex item does NOT guarantee it stays that size — flex-shrink's DEFAULT value of 1 means every flex item is shrinkable by default, and a crowded flex container (too many items, or items with long text, competing for limited space) WILL squeeze a fixed-width icon or button smaller than its declared width to make everything fit, unless flex-shrink: 0 explicitly opts it out. Any icon, logo, or fixed-size UI element inside a flex container that occasionally appears "squished" on narrow viewports or with more sibling content than usual is very likely missing this exact property.

Part 5 — gap: the Clean Replacement for Margin Hacks

.container {
  display: flex;
  gap: 16px;              /* space BETWEEN items only — never before the first or after the last */
  gap: 16px 24px;          /* row-gap column-gap independently, for wrapped flex layouts */
}

gap (now universally supported, and covered in the Box Model lesson's "spacing between siblings is the parent's job" architectural principle) creates space PURELY between adjacent items — no edge-case margin-collapsing concerns (Box Model lesson), no manual "every item except the last gets margin-right" selector tricks, and no risk of accidentally adding unwanted space at the container's own edges. This single property has genuinely eliminated an entire historical category of flexbox spacing workarounds (the :not(:last-child) margin-selector pattern, negative- margin container tricks) that were standard practice before broad gap support.

Part 6 — flex-wrap and the Real-World Responsive Card Grid

.card-grid {
  display: flex;
  flex-wrap: wrap;    /* items that don't fit on one row FLOW to a new row, instead of shrinking
                         indefinitely or overflowing */
  gap: 20px;
}
.card {
  flex: 1 1 250px;    /* grow/shrink, but prefer 250px — wraps to a new row once there's no
                         longer room for a 250px-ish card */
}

This is the classic, extremely common "responsive card grid with zero media queries" pattern: flex: 1 1 250px tells each card "you'd like to be 250px, but you're flexible — grow to fill extra space, shrink if needed, but once shrinking would squeeze you below a REASONABLE size, wrap to a new row instead." Combined with flex-wrap: wrap on the container, this produces a grid that automatically adjusts its own column COUNT based on available width — 4 columns on a wide desktop, 2 on a tablet, 1 on mobile — genuinely without a single explicit breakpoint, purely from the interaction of flex-basis and flex-wrap.

Interview Questions

Q1 (beginner): What's the difference between the main axis and the cross axis in flexbox?

The main axis is whatever direction flex-direction specifies (row = horizontal by default, or column = vertical) — justify-content ALWAYS distributes items along this axis. The cross axis is perpendicular to it — align-items ALWAYS aligns items along this axis. Switching flex-direction from row to column rotates both axes 90°, and both properties' real-world visual effect rotates with them, without any code change to the properties themselves — they're defined relative to the axis, not fixed horizontal/vertical directions.

Q2 (beginner): Why might a flex item with an explicit width: 24px still render narrower than 24px in a crowded container?

flex-shrink defaults to 1 — every flex item is shrinkable by default regardless of any explicit width set, and a container without enough total space will squeeze items smaller than their declared/natural size to fit everything, proportional to each item's shrink value. The fix is flex-shrink: 0 on anything (icons, fixed-size buttons) that must never shrink below its declared size.

Q3 (mid): Explain what flex: 1 1 250px means and why it produces a responsive grid with no media queries.

flex-grow:1 (claim available extra space), flex-shrink:1 (shrink proportionally if needed), flex-basis:250px (prefer starting at 250px). Combined with flex-wrap: wrap on the container, each item tries to be around 250px, growing to fill extra space or shrinking somewhat if space is tight — but once shrinking would squeeze items below a reasonable size, they wrap to a new row instead of continuing to shrink indefinitely. This naturally produces a self-adjusting column count (more columns on wide viewports, fewer on narrow ones) purely from the flex-basis/flex-wrap interaction, with zero explicit breakpoints.

Q4 (mid): What's the practical difference between flex: 1 and flex: 1 1 auto?

flex: 1 (shorthand for grow:1, shrink:1, basis:0) starts every item at ZERO size, meaning the entire container width becomes "extra space" to distribute equally — items completely ignore their own natural content size, producing perfectly equal-width columns regardless of content length. flex: 1 1 auto starts each item at its OWN natural content size (basis:auto) before growing/ shrinking — an item with more text content starts larger and may end up wider than a sibling with less content, even though both have identical grow/shrink values. The choice depends on whether equal widths or content-aware widths are the actual design intent.

Q5 (senior): A responsive navigation bar uses flex items with flex: 1 for each nav link, intending equal-width links. On a narrow viewport with 6 nav links, some link text is being truncated/wrapping awkwardly mid-word, and the design team wants long link labels to always display in full. Diagnose the tension and propose a fix.

The tension: flex: 1 (basis:0) deliberately ignores each item's natural content size to force EQUAL widths — but equal widths mean a link with a longer label (e.g., "International Shipping") gets the SAME width as one with a short label ("Home"), forcing the longer text to wrap/truncate at narrow viewports where equal-share width is smaller than the long text needs. This is a direct, inherent consequence of choosing equal-width distribution over content-aware sizing — not a bug, but the EXACT tradeoff flex:1's basis:0 makes. Fix options depend on the actual design priority: (1) if "no truncation, ever" is the true priority, switch to flex: 0 1 auto (content-aware sizing, no forced equal-width claim) and accept that link widths will vary based on their own text length — visually less uniform, but no more truncation; (2) if equal-width IS still desired as a design constraint, add flex-wrap: wrap to the nav container and accept links wrapping to a SECOND row on narrow viewports rather than staying forced into an ever-shrinking single row; (3) for a nav bar specifically, the most common real-world resolution is neither — collapse to a hamburger-menu pattern below a specific viewport width (a distinct, separate responsive-design decision, covered in the upcoming Responsive Design lesson), sidestepping the equal-width-vs-content-length tension entirely by not trying to fit all 6 links horizontally at narrow widths in the first place. The senior framing: recognizing this as an INHERENT tradeoff of the chosen flex-basis strategy (not a bug to "fix" within the current approach) is what leads to correctly identifying which of these three genuinely different architectural responses actually addresses the team's real underlying priority.

Q6 (senior): Your team's card-grid component (flex: 1 1 250px; flex-wrap: wrap; on a flex container) works correctly for grids of 3-8 cards, but at exactly certain card counts (like 5 or 7 cards in specific container widths), the LAST row shows 1-2 cards stretched noticeably wider than the cards in rows above them, creating a visually inconsistent, "unbalanced" look designers are flagging. Diagnose the root cause.

Root cause: flex-wrap's row-filling algorithm distributes EXTRA available space (via flex-grow) independently PER ROW, not globally across the entire grid — when the last row happens to contain fewer items than a "full" row (e.g., 2 cards in a grid that otherwise fits 3 per row), those 2 remaining cards' flex-grow:1 causes them to stretch and split the FULL row's available width between just the two of them, rendering noticeably wider than the 3-per-row cards above, purely because flexbox has no native concept of "the grid as a whole" — each row is an independent flex-line making its own grow/shrink distribution decision, with zero coordination across rows. This is a genuine, real, structural limitation of flexbox for GRID-shaped content specifically (not a bug in the implementation) — flexbox is fundamentally one-dimensional per row/column, and "keep every row's item widths visually consistent with every OTHER row" requires reasoning across TWO dimensions simultaneously, which is precisely outside flexbox's actual design scope. The correct fix is switching this specific component to CSS GRID (the next lesson) instead of flexbox — grid natively defines column tracks that apply CONSISTENTLY across every row simultaneously (e.g., grid-template-columns: repeat(auto-fill, minmax(250px, 1fr))), so a partial last row's cards align to the SAME column widths as full rows above, rather than each row independently redistributing its own available space. This is a genuinely common, real-world "flexbox worked until it didn't" story that correctly and directly motivates the next lesson's entire premise — recognizing WHEN flexbox's one-dimensional nature stops being sufficient, rather than continuing to patch a fundamentally one-dimensional tool for a genuinely two-dimensional grid problem, is the actual senior judgment call here.

🏋️ Exercises

1 (easy): Axis-rotation, felt directly. Build a flex container with 3 items, justify-content: center, align-items: center. Observe the centering with flex-direction: row, then switch to flex-direction: column with NO other changes and observe how the visual centering direction changes.
Expected observation

row: items center HORIZONTALLY as a group (justify-content, main axis) and each individually centers VERTICALLY within the container's cross-axis height (align-items). column: the SAME two properties now center the group VERTICALLY (justify-content, now the main axis) and each item HORIZONTALLY (align-items, now the cross axis) — zero code change to the properties themselves, confirming Part 2's "main/cross, not horizontal/vertical" mental model directly.

2 (easy): The flex-shrink bug, reproduced and fixed. Build a flex toolbar with 5 icon divs (width: 32px each) and enough OTHER content (long text buttons) to genuinely overcrowd the container at a moderate viewport width. Observe the icons shrinking below 32px. Add flex-shrink: 0 to the icons only and confirm they now stay exactly 32px while other, shrinkable content adjusts instead.
Expected before/after

Before: DevTools' box-model diagram on an icon should show an ACTUAL width less than the declared 32px, confirming shrinking occurred. After flex-shrink:0: the icon's actual rendered width matches exactly 32px regardless of container crowding, while the OTHER (still shrinkable) text buttons absorb the space pressure instead — a direct, hands-on fix of Part 4's exact bug.

3 (medium): flex:1 vs flex:1 1 auto, compared. Build 3 flex items with deliberately different text-content lengths (a short word, a medium phrase, a long sentence). Apply flex: 1 to all three and observe equal widths despite different content lengths. Switch to flex: 0 1 auto and observe widths now vary based on each item's own content.
Expected contrast

flex:1 version: all three items render at IDENTICAL widths, with the long-sentence item's text likely wrapping to multiple lines to fit its forced-equal width. flex:0 1 auto version: each item's width visibly reflects its own content length — the short word's item is narrow, the long sentence's item is wide — directly demonstrating Part 3/Q4's basis:0-vs-basis:auto distinction.

4 (medium): The responsive card grid, built with zero media queries. Build a card grid (flex-wrap: wrap, gap: 20px, cards at flex: 1 1 250px) with 8 cards. Resize the browser continuously from very wide to very narrow and count how many columns appear at 3-4 different widths, confirming the column count adjusts automatically with no explicit breakpoint written anywhere in your CSS.
Expected observation

At a wide viewport: likely 4+ columns. Narrower: 3, then 2, then 1 — the exact column count at each width depends on your container's actual size and the 250px basis, but the KEY observation is that this transition happens SMOOTHLY and automatically as you resize, with no visible "snap" at a specific media-query breakpoint, because none exists — confirming Part 6's "responsive with zero media queries" claim directly.

5 (hard): The uneven-last-row bug, reproduced. Build the exact card-grid pattern from exercise 4, but with exactly 5 cards at a container width that fits 3 per row (so the last row has only 2 cards). Observe and measure (via DevTools) whether those final 2 cards render wider than the 3 cards in the row above. Confirm this is genuinely flexbox's per-row distribution behavior, not a bug in your specific CSS.
Expected observation

The final row's 2 cards should measure NOTICEABLY wider (via DevTools box-model) than the 3 cards in the row above — each row independently distributes its own flex-grow space, so 2 items splitting a full row's width each get a meaningfully larger share than 3 items splitting that same width. This direct, measured reproduction is the concrete evidence behind Q6's "flexbox is the wrong tool for THIS specific problem" diagnosis — worth experiencing before the next lesson's grid-based fix.

6 (hard): Nav-link tension, all three resolutions built. Implement all THREE proposed fixes from Q5 for a 6-link nav bar with mixed-length labels at a narrow viewport: (a) flex: 0 1 auto (content-aware, no truncation, uneven widths), (b) flex: 1 + flex-wrap: wrap (equal widths, wraps to 2 rows), (c) a simple hamburger-menu collapse below a chosen viewport width (can be a basic version — full accessible hamburger patterns come later in the HTML/ JS tracks). Compare all three side by side and write one sentence on which you'd actually recommend for a real product nav, and why.
What a good comparison identifies

(a) preserves readability perfectly but looks visually uneven/unpolished — usually undesirable for a primary nav specifically, where uniformity reads as more "designed." (b) preserves equal widths but the 2-row wrap often looks accidental/unintentional for a nav bar rather than a deliberate design choice. (c) the hamburger collapse is overwhelmingly the real-world standard choice for exactly this scenario — it sidesteps the tension entirely rather than compromising on either readability or visual consistency, which is why it's the near-universal pattern real products actually ship for nav bars with more links than comfortably fit at narrow widths.

🎯 Key Takeaways

  • Flexbox is one-dimensional — main axis (justify-content) and cross axis (align-items) rotate together with flex-direction, which is why internalizing "main/cross" rather than "horizontal/vertical" makes the naming click.
  • flex-grow/shrink/basis is a real algorithm: basis sets the starting size, then extra/deficit space distributes proportionally by grow/shrink values relative to siblings — flex:1's basis:0 is specifically what forces equal widths regardless of content.
  • flex-shrink defaults to 1 — every flex item is shrinkable by default even with an explicit width set; flex-shrink:0 is required to guarantee a fixed size under crowding.
  • gap cleanly replaces margin-based spacing hacks between flex items, with no edge-case margin-collapsing or first/last-child selector tricks needed.
  • flex: 1 1 [basis]px + flex-wrap: wrap is the classic zero-media-query responsive grid pattern — column count adjusts automatically from the basis/wrap interaction alone.
  • Flexbox distributes space independently PER ROW when wrapped — a partial last row's items can render visibly wider than full rows above, a genuine structural limitation (not a bug) that correctly motivates reaching for CSS Grid instead for true grid-shaped, cross-row-consistent layouts.
