Cross-browser & Accessibility Ultra-Advanced

The last mile of production CSS: users who aren't on the latest Chrome, with perfect vision, on a fast Mac, with a mouse. This is where legal exposure (ADA/EAA lawsuits are routine), real-device bugs, and inclusive design converge — and where a written support policy plus a repeatable audit checklist replace heroics.

A Support Policy, Not Vibes

Example policy (write yours down, put it in the README):
· Baseline "widely available" features: use freely
· Baseline "newly available": use with @supports fallback
· Beyond that: progressive enhancement only, never core functionality
· Test matrix: latest Chrome/Firefox/Edge, Safari (macOS+iOS — the engine that differs most),
  one mid-range Android (real hardware), plus forced-colors and 200% zoom passes

Baseline (the caniuse/web.dev badge) formalizes "safe": newly available = shipping in all engines; widely available = 30+ months everywhere. It ends "can we use X?" debates with a lookup instead of a meeting. Remember the long tail: enterprise WebViews, old iPads, KaiOS — decide explicitly whether they get a degraded-but-working experience (usually yes).

Graceful Degradation with @supports

/* Baseline that works everywhere: */
.gallery { display: flex; flex-wrap: wrap; gap: 1rem; }

/* Enhancement where supported: */
@supports (grid-template-rows: subgrid) {
  .gallery { display: grid; grid-template-columns: repeat(3, 1fr); }
  .gallery .card { display: grid; grid-template-rows: subgrid; grid-row: span 3; }
}
@supports selector(:has(a)) { … }        /* selector-level detection */
@supports not (container-type: inline-size) { … }  /* inverted */

Design the fallback as the default and layer enhancements on — progressive enhancement by construction. Vendor prefixes are autoprefixer's job from browserslist config; handwritten prefixes are 2012 archaeology.

Forced Colors Mode — the Audit Most Teams Fail

/* Windows High Contrast strips your palette: backgrounds gone, colors replaced by system ones.
   UI that communicated by background-color alone VANISHES. */
@media (forced-colors: active) {
  .badge   { border: 1px solid; }               /* structure carries meaning now */
  .btn     { forced-color-adjust: none; }        /* opt OUT only when truly necessary (brand logos) */
  .selected{ outline: 2px solid; }               /* selection state must survive */
}
/* System color keywords work WITH the mode: Canvas, CanvasText, LinkText, ButtonFace, Highlight */

Test in one minute: DevTools → Rendering → Emulate forced-colors: active. If your toggle states, selected rows, and progress bars disappear — that's what those users get.

The .sr-only Utility (and when NOT to use it)

.sr-only {           /* visible to screen readers, invisible on screen */
  position: absolute; width: 1px; height: 1px;
  margin: -1px; padding: 0; border: 0;
  clip-path: inset(50%); overflow: hidden; white-space: nowrap;
}
<button><svg aria-hidden="true">…</svg><span class="sr-only">Search</span></button>

/* Why not display:none / visibility:hidden? Both remove content from the accessibility
   tree too — silent to screen readers. The clip pattern hides ONLY visually.
   When NOT to use it: content that sighted keyboard users also need (skip links) should
   become VISIBLE on focus: .skip-link:focus { position: static; width: auto; … } */

Zoom vs Font-Size Preference (precision matters in interviews)

Browser ZOOM (Ctrl +):     scales the CSS pixel — px, rem, everything grows. Layout reflows.
Font-size PREFERENCE:      changes the root em — ONLY rem/em respond. px text ignores the user.
WCAG 1.4.4 tests both paths to 200% without loss of content/function.
⇒ type & spacing in rem; test at 200% for clipped fixed-height containers; line-height unitless.

Interview Questions

Q1: How does accessibility scale to a 50-engineer org without 50 experts?

The design-system move: bake it into components once — focus states, contrast-checked tokens, motion guards, ARIA patterns, forced-colors survival — and every product team inherits compliance. Auditing 12 components beats auditing 400 screens. Add CI automation (axe on stories, contrast lint) for regressions and a manual keyboard/SR pass on new patterns only.

Q2: Zoom vs font-size preference — why do both matter and what breaks each?

Zoom scales the CSS pixel (everything grows — breaks fixed-height clipping, tested at 200%); font-size preference changes root em (only rem/em respond — px text silently ignores the user's explicit setting). Ship rem type + flexible containers and both paths pass WCAG 1.4.4.

Q3: What's your @supports strategy for a feature like subgrid or :has()?

Fallback-first: write the universally-working layout as default, wrap the enhancement in @supports (property: value) or @supports selector(). Never gate core functionality on new features; check Baseline status against the written support policy; remove the guard when the policy's floor passes the feature.

Q4: Automated tools report 0 issues. Is the page accessible?

No — automation catches ~30%: it can verify contrast math and attribute presence, but not whether alt text is meaningful, focus order is logical, announcements make sense, or the keyboard flow is usable. The manual trio — keyboard pass, screen-reader spot check, zoom/forced-colors emulation — covers what machines can't judge.

🏋️ Practice:
1. Run the full audit checklist on your LearnHub clone: keyboard pass, 200% zoom, reduced-motion, forced-colors, contrast in both themes — file an issue per failure.
2. Build the visible-on-focus skip link and verify with Tab.
3. Emulate forced-colors and fix one component whose state vanishes (add structural borders).
4. Write a two-line support policy for a hypothetical product (audience: banks vs startups) and defend the difference.

🎯 Key Takeaways

  • Write the support policy; Baseline turns "can we use it" into a lookup; @supports layers enhancements over working fallbacks.
  • Safari + one real Android in the matrix; dvh, safe-area, 16px inputs, scrollbar-gutter are the mobile recurring set.
  • The audit eight: contrast (both themes), focus-visible, keyboard pass, 200% zoom, reduced motion, touch targets, not-color-alone, SR spot check.
  • Forced-colors strips your palette — structure (borders/outlines) must carry state; test via Rendering panel.
  • .sr-only clips visually only; display:none silences AT too; skip links become visible on focus.
  • Scale a11y through the component library + CI, not through heroic final audits.
