Selectors & Specificity Basics

CSS is a rule engine: selectors decide which elements a rule targets, declarations decide what happens to them, and when multiple rules disagree, the cascade decides who wins. Nearly every "CSS is broken!" moment in a real codebase is a cascade fight you can't see. This lesson makes the whole resolution algorithm visible — after it, you debug styles by reasoning, not by adding !important until it works.

How the Browser Matches Selectors (the part nobody teaches)

When the browser builds the render tree, it doesn't ask "what does .card p match?" — it asks, for each element, "which rules match me?" And it evaluates selectors right to left: for .card p, it first finds every p, then walks each one's ancestors looking for .card. Two consequences:

1. The rightmost part (the "key selector") should be specific — .card p makes the engine check ancestors of every paragraph on the page, while .card-text checks only elements carrying that class.
2. There is no "parent selector" in the old model because matching only ever looks up the tree — which is exactly why :has() (which looks down) took 25 years and a new engine architecture to ship.

nth-child — the Mini-Language

li:nth-child(3)        /* exactly the 3rd child */
li:nth-child(2n)       /* every even child (same as 'even') */
li:nth-child(2n+1)     /* odd — zebra striping tables */
li:nth-child(n+4)      /* the 4th onward — e.g. "show first 3, style the rest" */
li:nth-child(-n+3)     /* the FIRST three */
li:nth-last-child(2)   /* second from the END */
li:nth-child(odd of .visible)   /* odd among a FILTERED subset (modern) */
⚠️ The gotcha everyone hits: :nth-child counts all siblings by position — p:nth-child(2) matches a p only if the second child happens to BE a p. To count only paragraphs, use p:nth-of-type(2).

Specificity — the Complete Scoring Algorithm

Score every selector as a 3-tuple (ID, CLASS, ELEMENT):

p                        → (0,0,1)    element
.card p                  → (0,1,1)    class + element
.card .title             → (0,2,0)
.btn:hover               → (0,2,0)    pseudo-CLASSES count at class level
.btn::before             → (0,1,1)    pseudo-ELEMENTS count at element level
[type="email"]           → (0,1,0)    attributes count at class level
#main .card              → (1,1,0)
:where(.card) p          → (0,0,1)    :where() contributes ZERO
:is(.a, #b) p            → (1,0,1)    :is()/:not()/:has() take their MOST specific argument

Compare left→right, like version numbers: (1,0,0) beats (0,99,99).
Ties → the LATER rule in source order wins (the cascade).

The Full Cascade — What Actually Beats What

Specificity is only step 4 of the resolution order. The complete priority ladder, highest first:

#LevelNotes
1Transitions/animations in flightActively animating values win temporarily
2!important — user-agent, then user, then authorNote the inversion: the USER's important styles beat YOURS — accessibility by design
3Author styles by cascade layerLater @layer beats earlier, regardless of specificity (Modern CSS lesson)
4Inline style=""Beats all selectors (but loses to !important)
5Selector specificityThe 3-tuple above
6Source orderLast one wins

Then, for properties no rule set: inheritance (text properties like color, font-*, line-height flow down from ancestors) and finally the property's initial value. Explicit controls: inherit, initial, unset (inherit if inheritable, else initial), revert (back to browser default).

Worked Example — Debugging a Real Conflict

/* Why is my button text white-on-white? */
#checkout .btn { color: white; background: white; }   /* (1,1,0) — legacy rule */
.btn-primary   { color: white; background: green; }    /* (0,1,0) — your new rule. LOSES. */

/* Three fixes, ranked: */
.btn-primary { background: green !important; }         /* ❌ starts an arms race */
#checkout .btn-primary { background: green; }           /* ⚠ matches (1,2,0) — works, deepens the mess */
/* ✅ Actually fix it: delete or de-ID the legacy rule; keep everything at (0,1,0) */

The DevTools workflow: F12 → select the element → Styles panel. Every matching rule is listed, losers struck through, winner on top, and each rule shows its file:line. Hovering a selector even shows its specificity. This panel answers "why isn't my style applying" in 10 seconds; guessing never does.

Production Strategy: Keep Specificity FLAT

/* ❌ The arms race (real code from real codebases): */
#sidebar div.widget ul li a.link { color: red; }    /* (1,3,3) */
/* six months later, to override it: */
#sidebar div.widget ul li a.link.special { … }       /* (1,4,3) 😱 */

/* ✅ One class, one job: */
.widget-link { color: red; }                          /* (0,1,0) — anyone can override */

Interview Questions

Q1: Walk me through what wins: inline style vs #id vs !important vs a later @layer.

!important beats inline; inline beats any selector specificity; among layered author rules, layer order beats specificity; within the same layer, specificity (1,0,0) > any (0,x,y); ties fall to source order. Bonus point: user-important beats author-important — the accessibility inversion.

Q2: Why does CSS match selectors right-to-left?

Per-element matching: for each element the engine tests the key (rightmost) selector first, then walks ancestors to verify the rest. Right-to-left makes the common case (no match) fail fastest. It's why over-general key selectors (.nav *) are the one selector "performance" concern with any truth to it.

Q3: What's the difference between :is() and :where(), and when do you reach for each?

Identical matching, different specificity: :is() adopts its most specific argument; :where() is always zero. Use :is() for grouping in app code; use :where() when publishing defaults that consumers must be able to override with a bare class (design systems, resets).

Q4: unset vs revert?

unset = inherit for inheritable properties, initial otherwise — "as if no author CSS". revert = roll back to the browser's default stylesheet — "as if only the UA styled it". revert is what you want when undoing your own reset for one element.

🏋️ Practice (Playground):
1. Build a list and zebra-stripe it with :nth-child(odd); then add a non-li element in the middle and watch the stripes shift — fix with :nth-of-type.
2. Write three rules targeting the same element at (0,1,0), (0,2,0), (1,0,0) and predict the color before checking DevTools.
3. Recreate the white-on-white button bug above and fix it three ways; feel why the flat fix is calmer.
4. Style every external link with a[href^="http"]:not([href*="yourdomain"]) + a ↗ via ::after.

🎯 Key Takeaways

  • Resolution order: importance → layers → inline → specificity (ID, CLASS, ELEMENT) → source order.
  • One ID outranks any number of classes — compare tuples left to right, never sum them.
  • Pseudo-classes count as classes; pseudo-elements as elements; :where() counts as nothing.
  • Engines match right-to-left, per element — keep key selectors specific; :has() is the deliberate exception.
  • Production CSS keeps everything at (0,1,0): single classes, no IDs, utilities own !important.
  • The DevTools Styles panel is the debugger — struck-through rules ARE the explanation.
