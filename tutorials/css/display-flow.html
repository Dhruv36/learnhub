Display & Normal Flow Layout

What you'll master: "normal flow" as the layout that happens with zero CSS (formalizing what the HTML track's block/inline distinction hinted at), the real display values and what each genuinely changes, inline-block as the historical bridge technology flexbox mostly replaced, and overflow's four values precisely. This opens the Layout section — the CSS track's structural core. Prerequisite: Backgrounds, Borders & Shadows; the HTML track's Text & Headings lesson's block/inline concept is the direct foundation here.

Part 1 — Normal Flow: the Layout Before Any CSS

Recall the HTML track's block-vs-inline distinction (Text & Headings lesson): block elements stack vertically, filling available width; inline elements flow horizontally within a line, like words. This IS "normal flow" — the default layout algorithm every element participates in unless explicitly removed from it (via float, position, or becoming a flex/grid item). Understanding normal flow deeply matters because every OTHER layout mode in this section (flexbox, grid, positioning) is meaningfully defined by HOW it opts elements OUT of normal flow and what replaces it — you can't fully understand what flexbox changes without first knowing precisely what it's changing FROM.

Part 2 — display: the Property That Assigns Layout Behavior

display: block;         /* new line, fills available width — div, p, h1-h6, ul, li... default to this */
display: inline;        /* flows within text, width/height IGNORED, only horizontal margin/padding
                            visually affects layout — span, a, strong... default to this */
display: inline-block;  /* flows like inline (sits within a text line) BUT respects width/height/
                            vertical margin/padding like block — the historical bridge technology */
display: none;           /* removed from the RENDER TREE entirely (HTML/Critical-Rendering-Path
                            lesson's Stage 3) — no box, no space reserved, genuinely absent */
display: flex;           /* the element becomes a FLEX CONTAINER — its direct children become flex
                            items, opting OUT of normal flow entirely (next lesson) */
display: grid;           /* same idea, for GRID (two lessons ahead) */

The critical, often-missed distinction: inline elements ignore width, height, and vertical (top/bottom) margin/padding — setting height: 100px on a <span> does genuinely nothing, because inline layout doesn't have a concept of "this element's own height" separate from its line's natural text height. This single fact explains a real, common beginner confusion: "why doesn't padding-top on my inline link push other content down" — it's visually painted (the background/border DO extend), but it doesn't affect the actual LINE-BOX height or surrounding layout, precisely because inline layout structurally doesn't reserve vertical space the way block layout does.

Part 3 — inline-block: the Bridge, and Why Flexbox Mostly Replaced It

.nav-item {
  display: inline-block;
  width: 120px;
  padding: 10px;
  vertical-align: top;    /* often needed — see the whitespace-gap issue below */
}

Before flexbox (2015-2017 real-world adoption), inline-block was the standard technique for "several boxes side by side, each with real width/height/padding control" — genuinely useful, but with two well-known, awkward quirks flexbox specifically doesn't have:

  • The whitespace gap: inline-block elements are, structurally, still INLINE — a newline/space in your HTML SOURCE between two inline-block elements renders as an actual, small visible gap between them (a whitespace text node, exactly as the HTML track's Text & Headings lesson described for any inline elements). Historical, genuinely annoying workarounds: removing all whitespace between tags in the source (harms readability), setting the parent's font-size: 0 (a hack), or HTML comments bridging the gap (<div></div><!--\n--><div></div>). Flexbox has NO such gap — flex items are never subject to inline whitespace rendering at all.
  • Vertical alignment quirks: inline-block elements align along the text vertical-align baseline by default (designed for aligning inline content with surrounding TEXT, not for aligning a row of independent boxes with each other) — frequently needing an explicit vertical-align: top override just to make several inline-block boxes of different heights actually line up as a human would expect. Flexbox's align-items is a purpose-built, unambiguous alternative with no such default- baseline surprise.

inline-block still has a genuine, narrow remaining use case: when you specifically WANT an element to flow WITHIN a line of text (wrapping naturally alongside surrounding text) while ALSO needing real width/height/padding control — a styled inline badge or tag within a paragraph, for instance. For "lay out several sibling boxes in a row," flexbox is now the correct default choice in nearly every case.

Part 4 — overflow: What Happens When Content Doesn't Fit

overflow: visible;   /* the DEFAULT — content spills outside the box, uncropped, unscrolled */
overflow: hidden;    /* excess content is CLIPPED and simply invisible — no scrollbar offered */
overflow: scroll;    /* ALWAYS shows scrollbars (even if content actually fits) — rarely the right choice */
overflow: auto;      /* shows scrollbars ONLY IF content genuinely overflows — the usual correct choice */
overflow-x: auto; overflow-y: hidden;   /* independent per-axis control */

overflow: auto is, in practice, almost always the right choice over scroll — scroll's permanently-visible scrollbar (even on content that currently fits perfectly) is a real, if minor, visual paper cut most users notice as slightly "off," while auto shows a scrollbar only when genuinely needed and disappears cleanly when it isn't. Recall the HTML track's Critical Rendering Path lesson's display: none vs visibility: hidden distinction — overflow: hidden is a THIRD, related but distinct concept: the element itself remains fully present and laid out, only its OVERFLOWING CHILD CONTENT specifically gets clipped at the parent's boundary, a genuinely different mechanism from either of those two other hiding techniques.

Part 5 — Deep Dive: Block Formatting Contexts (BFC) — the Concept Behind Several "Fixes"

A Block Formatting Context is a genuinely important, somewhat hidden CSS concept: certain properties/values (overflow: hidden/auto, display: flow-root, float, position: absolute/fixed, display: flex/grid) create a NEW, independent block formatting context — a self-contained layout "room" that contains and clips its own floats, and — directly connecting to the Box Model lesson's margin-collapsing discussion — does NOT let its children's margins collapse with its own margins. This is precisely WHY overflow: hidden and display: flow-root were listed as valid fixes for the parent+first-child margin- collapse problem in that earlier lesson — they're not special-cased margin fixes, they're a genuine, predictable SIDE EFFECT of establishing a new formatting context, which happens to also prevent margin collapsing as one of several consistent behaviors that come with it. Understanding BFCs as ONE underlying concept — rather than memorizing "these specific properties fix margin collapse" as an unconnected list of tricks — is what lets you correctly predict NEW situations (a float being correctly contained by its parent, for instance) that share the identical root cause.

Interview Questions

Q1 (beginner): Why does setting width and height on a <span> have no visible effect?

span defaults to display: inline, and inline layout structurally ignores width, height, and vertical margin/padding — it has no concept of its own dimensions separate from its line's natural text-driven height. To give it real dimensional control, it needs display: inline-block (staying within a text line) or display: block/flex (leaving the text line entirely).

Q2 (beginner): What's the practical difference between overflow: scroll and overflow: auto?

scroll always shows scrollbars, even when the content currently fits without any actual overflow — a permanent, often unnecessary visual element. auto shows scrollbars ONLY when content genuinely overflows the box, and cleanly shows none when it fits — the correct default choice in nearly every real use case.

Q3 (mid): Why does inline-block sometimes produce small, unwanted gaps between adjacent elements, and how does flexbox avoid this?

inline-block elements remain structurally INLINE, so whitespace (a newline or space) in the HTML SOURCE between two inline-block tags renders as an actual small visible gap — the same whitespace- collapses-to-a-space-character behavior the HTML track described for any inline content. Flexbox items are not subject to this inline whitespace-rendering behavior at all — there's no equivalent gap, regardless of source formatting, because flex layout doesn't process source whitespace the same way inline layout does.

Q4 (mid): What is a Block Formatting Context, and name two distinct behaviors it produces.

A self-contained layout "room" established by certain properties (overflow: hidden/auto, display: flow-root, float, position: absolute/fixed, display: flex/grid) that contains/clips its own floats independently of the rest of the page. Two distinct, predictable behaviors it produces: (1) its children's margins do NOT collapse with the BFC-establishing element's own margins (directly explaining why overflow:hidden/flow-root fix the parent+first-child margin-collapse problem); (2) it fully contains floated children within itself (the classic "clearfix" problem, historically solved by exactly these same BFC-establishing properties).

Q5 (senior): A junior developer asks "why does adding overflow: hidden to a parent div fix my margin-collapse bug, when overflow is supposed to be about scrolling?" How do you explain this without it sounding like an unrelated coincidence?

The explanation centers on Block Formatting Context as the unifying concept: overflow: hidden (along with several other properties) doesn't JUST control scroll/clip behavior — as a genuine, documented SIDE EFFECT, it establishes a new Block Formatting Context for that element, and one of a BFC's defining, consistent properties is that margin collapsing doesn't cross its boundary. So the fix isn't a coincidental quirk of the overflow property specifically — it's a predictable consequence of BFC establishment, and ANY of the several BFC-triggering properties (display: flow-root being the most semantically honest choice, since its ENTIRE purpose is establishing a BFC with no scroll/clip side effects at all) would produce the identical margin-collapse fix for the identical underlying reason. Teaching it this way — one concept explaining several seemingly separate "tricks" — is what lets the junior developer correctly predict and diagnose FUTURE situations (float containment, for instance) sharing the same root mechanism, rather than memorizing an unconnected list of "properties that happen to fix margin collapse."

Q6 (senior): Your team's legacy codebase uses overflow: hidden extensively as a general-purpose "fix weird layout bugs" hammer, applied somewhat indiscriminately. A new feature needs a dropdown menu that visually extends slightly BEYOND its parent card's boundary, but the dropdown is being invisibly clipped. Diagnose and propose both an immediate fix and a longer-term recommendation.

Immediate diagnosis: the parent card almost certainly has overflow: hidden applied (likely as one of the codebase's habitual "fix weird bugs" applications, possibly originally added to solve an unrelated margin-collapse or float-containment issue elsewhere) — and overflow: hidden's CLIPPING behavior is now clipping the dropdown, which needs to render OUTSIDE the card's bounds, exactly as designed; this is overflow:hidden doing precisely what it's specified to do, just now in conflict with a new, different requirement. Immediate fix: since the dropdown genuinely needs to escape the card's clipping boundary, options include removing overflow:hidden from the card if whatever it was originally fixing can be resolved differently (check whether it was solving a NOW-irrelevant margin-collapse issue that a scoped display:flow-root elsewhere could handle instead), or — if the overflow:hidden serves a genuine ongoing purpose elsewhere in that card (e.g., clipping an image to rounded corners) — using a portal/teleport pattern (rendering the dropdown as a sibling OUTSIDE the clipped card in the actual DOM, then position:absolute/fixed positioning it visually where the dropdown trigger is, a common pattern in component frameworks specifically for this exact recurring conflict). Longer-term recommendation: audit the codebase's overflow:hidden usages specifically to distinguish DELIBERATE clipping (image crops, card boundaries that should genuinely contain their content) from ACCIDENTAL BFC-establishment side-effect usage (added originally just to fix a margin-collapse or float issue, with no actual desire to clip anything) — for the latter category, replace with display: flow-root specifically, which establishes the same BFC and gets the same margin/float-containment benefits WITHOUT the clipping side effect, making future conflicts like this dropdown case categorically impossible rather than requiring case-by- case firefighting each time a new feature happens to need to escape a card's bounds.

🏋️ Exercises

1 (easy): Inline dimension test. Build a <span> and try setting width: 200px, height: 100px, and padding-top: 20px on it — observe that none visibly affect layout. Change display to inline-block and confirm all three now work as expected.
Expected observation

As inline: the span's box stays exactly text-sized regardless of the width/height/padding-top declarations — DevTools' box-model diagram should confirm the actual rendered dimensions ignore those values entirely. As inline-block: the same declarations now visibly and correctly apply — direct, hands-on confirmation of Part 2's core claim.

2 (easy): The whitespace-gap bug, reproduced and fixed. Build 3 inline-block boxes with visible borders, written with normal HTML formatting (newlines/indentation between them, as any reasonable developer would write). Observe the small gaps between them. Convert the parent to display: flex and confirm the gaps disappear with no other changes.
Expected result

The inline-block version should show small (roughly font-size-dependent) visible gaps between each box's border, purely from the whitespace in your source formatting. The flex version, with IDENTICAL HTML source formatting, should show the boxes flush against each other (or exactly as far apart as any explicit gap/margin you've set, with no EXTRA unintended space) — confirming flexbox structurally doesn't process inline whitespace the way inline-block does.

3 (medium): overflow, all four values compared. Build a 200×100px box with content that's deliberately taller than it (300px of text). Apply each overflow value (visible, hidden, scroll, auto) to 4 copies side by side and describe the visible difference in each.
Expected differences

visible: text spills out below the box's visible boundary, uncropped, no scrollbar. hidden: text is cleanly clipped exactly at 100px height, with no way to see or scroll to the rest. scroll: a scrollbar appears (vertical, since content overflows vertically) allowing access to the clipped content. auto: functionally identical to scroll in THIS specific case (since content genuinely does overflow) — the difference from scroll would only become visible if you then REDUCED the content to fit within 100px: auto's scrollbar would disappear, scroll's would remain visible regardless.

4 (medium): Reproduce and explain a BFC fix. Build the Box Model lesson's parent+first-child margin-collapse scenario again (a bare div containing a p with margin-top: 40px). This time, fix it using display: flow-root specifically (rather than padding or flex, used in that earlier lesson), and explain in your own words — referencing THIS lesson's BFC concept — why flow-root resolves it.
Explanation

display: flow-root establishes a new Block Formatting Context for the parent div with NO other side effects (unlike overflow:hidden, which also clips overflowing content — a side effect flow-root deliberately doesn't have, since its entire purpose IS establishing a BFC cleanly). One of a BFC's defining properties, per Part 5, is that margin collapsing doesn't cross its boundary — so the child paragraph's margin-top can no longer collapse with and "escape through" the parent's own top edge, correctly creating genuine internal spacing instead. This confirms flow-root as the most semantically precise of the several valid fixes, since it does exactly one thing (establish a BFC) with no incidental clipping/scrolling behavior riding along.

5 (hard): The dropdown-clipping conflict, reproduced and resolved. Build Q6's exact scenario: a card with overflow: hidden (simulating a legacy "fix" for something else, like rounded-corner image clipping) containing a button that reveals a dropdown menu meant to visually extend below the card's boundary. Confirm the dropdown gets invisibly clipped. Then implement BOTH proposed fixes from Q6 (removing/replacing the overflow with flow-root if nothing else needs clipping; alternatively, restructuring the dropdown to render as a DOM sibling outside the card, positioned absolutely to visually align with the trigger button) and compare.
What to verify

Original: the dropdown's lower portion should be invisibly cut off exactly at the card's bottom edge. Fix 1 (flow-root instead of overflow:hidden, assuming nothing else in the card genuinely needs clipping): the dropdown should now render fully, uncropped. Fix 2 (dropdown as a DOM sibling, absolutely positioned): also renders fully uncropped, while allowing the card to KEEP its overflow:hidden for a genuine, separate clipping need (like a rounded-corner image) that fix 1 would have had to sacrifice. Building both fixes concretely demonstrates the tradeoff Q6's answer described in the abstract — sometimes the BFC-property itself is genuinely serving two purposes and removing it entirely isn't viable, requiring the more involved sibling-positioning approach instead.

6 (hard): Normal-flow-only page, then progressively converted. Build a simple 3-section page layout (header, two-column body, footer) using ONLY normal flow (no flex, no grid, no positioning) — just block-level divs stacking vertically, and see how far you can get toward a real 2-column body layout using only inline-block for the two columns. Note every awkward workaround required (whitespace-gap hacks, vertical-align fixes, width percentage math for gutters). Then rebuild the two-column section with flexbox and count how many of those workarounds disappear.
What this exercise demonstrates

The inline-block version typically needs: the whitespace-gap fix (removing source whitespace or using font-size:0), vertical-align:top on both columns (default baseline alignment looks wrong for unequal-height columns), and manual percentage-width math to leave room for a gutter between them (since inline-block has no native gap concept). The flexbox version needs none of these — just display:flex and a gap value. Counting the concrete number of workarounds eliminated ( typically 3+ distinct hacks) is a direct, quantified demonstration of why flexbox became the default over inline-block for this exact use case, rather than an abstract preference.

🎯 Key Takeaways

  • Normal flow (block elements stack vertically, inline elements flow horizontally like text) is the layout every element participates in by default — every other layout mode is defined by how it opts elements OUT of it.
  • Inline elements ignore width/height/vertical margin/padding entirely — a real, structural limitation, not a bug; inline-block bridges to add dimensional control while staying within a text line.
  • inline-block's whitespace-gap and baseline-alignment quirks are real, historical annoyances flexbox structurally doesn't have — flexbox is the modern default for row-of-boxes layouts.
  • overflow: auto (scrollbar only when genuinely needed) is almost always preferable to overflow: scroll (permanently visible); overflow:hidden clips a fully-present element's overflowing children, distinct from display:none or visibility:hidden.
  • Block Formatting Context is the single underlying concept explaining why overflow:hidden/auto, display:flow-root, float, and position:absolute/fixed all independently "fix" margin collapsing and float containment — one mechanism, several triggering properties.
  • display: flow-root is the semantically precise way to establish a BFC with zero side effects (no clipping/scrolling) — preferred over overflow:hidden specifically when the only goal is preventing margin collapse or containing floats.
