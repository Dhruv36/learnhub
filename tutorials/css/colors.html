Colors & Color Functions Foundations

What you'll master: every color notation and when each is the right choice, HSL as the notation actually useful for programmatic color manipulation, the modern relative-color and color-mix() functions that eliminate a whole category of manual color-math, contrast ratios and WCAG's real numeric requirements, and the color-scheme/prefers-color-scheme mechanism behind dark mode. Prerequisite: Units & Sizing.

Part 1 — The Notations, and When Each Earns Its Place

#1a73e8              HEX — compact, universal, the most common in real codebases
rgb(26 115 232)       RGB — same color, explicit channels; accepts a 4th alpha value
rgb(26 115 232 / 0.5) RGB with alpha — 50% transparent
hsl(217 83% 51%)      HSL — Hue/Saturation/Lightness; the notation actually useful for MANIPULATION
hsl(217 83% 51% / .5) HSL with alpha
oklch(58% 0.19 258)   OKLCH — the modern, PERCEPTUALLY uniform notation (Part 4)

Hex and RGB describe a color as raw channel INTENSITY — genuinely useful for exact-value precision (matching a brand guideline's exact hex code) but nearly useless for REASONING about a color, since "make this more saturated" or "make this 20% lighter" has no obvious hex/RGB-channel operation behind it. HSL exists specifically to fix that: Hue (0-360°, a position on the color wheel), Saturation (0-100%, how vivid vs. gray), Lightness (0-100%, how close to black/white) map directly onto how humans actually THINK about color, making HSL the right notation whenever a color needs to be programmatically varied — a hover state that's "10% darker," a palette generated from one base hue at different lightness steps.

Part 2 — HSL in Practice: Generating a Palette From One Hue

:root {
  --brand-hue: 217;
  --brand-sat: 83%;
}
.brand-50  { background: hsl(var(--brand-hue) var(--brand-sat) 95%); }  /* near-white tint */
.brand-500 { background: hsl(var(--brand-hue) var(--brand-sat) 51%); }  /* the base brand color */
.brand-900 { background: hsl(var(--brand-hue) var(--brand-sat) 15%); }  /* near-black shade */

This is HSL's real, practical payoff: an ENTIRE tint/shade palette (the "50, 100, 200...900" scale every modern design system ships) can be generated from ONE hue and saturation value, varying only lightness — a systematic, formula-driven approach replacing the old practice of manually picking 9+ separate hex codes by eye and hoping they stay visually coherent as a family. (Custom properties, used here, get their own full lesson in the Modern CSS section — this preview shows exactly why HSL and custom properties are so often used together.)

Part 3 — Modern Color Functions: color-mix() and Relative Color Syntax

.btn:hover {
  background: color-mix(in srgb, var(--brand-color) 85%, black);
  /* "85% of the brand color, mixed with 15% black" — a hover-darken effect that AUTOMATICALLY
     tracks whatever --brand-color currently is, with zero manual color math */
}

.btn:active {
  background: oklch(from var(--brand-color) calc(l - 0.1) c h);
  /* RELATIVE COLOR SYNTAX: take the brand color's OWN oklch lightness (l) and subtract 0.1 from
     it, keeping chroma (c) and hue (h) unchanged — a precise, programmatic darken */
}

Both are recent (2023+, now broadly supported) CSS features solving the identical real problem: a hover/active/disabled STATE variant of a color that should stay correct even if the base brand color is later changed by a design update — without color-mix()/relative syntax, teams historically hand- picked a SEPARATE hardcoded hex value for each state, which then silently drifted out of sync with the base color whenever a rebrand or theme update changed it. color-mix() is the simpler, more widely-applicable tool ("blend these two colors by this ratio"); relative color syntax is more surgical (adjust ONE specific channel of a color while preserving the others exactly) and particularly powerful combined with OKLCH specifically, because OKLCH's lightness channel behaves PERCEPTUALLY uniformly (Part 4) in a way HSL's lightness genuinely doesn't.

Part 4 — Why OKLCH Is Replacing HSL for Serious Color Work

hsl(60 100% 50%)    /* pure yellow at 50% lightness */
hsl(240 100% 50%)   /* pure blue at 50% lightness — SAME lightness number, but blue looks
                        dramatically DARKER to the human eye than yellow does */

This is HSL's real, well-documented flaw: its "lightness" value is a mathematical average of RGB channels, NOT a measure of actual human-PERCEIVED brightness — different hues at the identical HSL lightness value can look meaningfully lighter or darker to an actual viewer, which quietly breaks any color-scale or accessibility-contrast reasoning built on the assumption that "same lightness number = same perceived brightness." OKLCH is a newer color space specifically engineered so equal lightness values DO look equally bright to human perception, across every hue. This matters concretely for exactly the palette-generation pattern from Part 2: an OKLCH-based tint/shade scale produces PERCEPTUALLY consistent steps (each step looks like a genuinely equal jump in brightness) where an HSL-based one can produce visibly uneven jumps depending on which hue you started from — a real, measurable quality difference design systems increasingly care about, and the direct reason major design tools and systems have been migrating their color tooling toward OKLCH.

Part 5 — Contrast: the Actual WCAG Numbers

WCAG AA (the real-world baseline target):
  Normal text:        4.5:1 minimum contrast ratio
  Large text (18pt+/14pt+bold):  3:1 minimum

WCAG AAA (stricter, not universally required):
  Normal text:  7:1
  Large text:   4.5:1

Contrast ratio is a precise, computable NUMBER (not a subjective "does this look readable" guess) — comparing the relative luminance of text color against its background, on a scale from 1:1 (identical, invisible) to 21:1 (pure black on pure white, maximum possible contrast). This directly connects to the HTML track's Accessibility lessons: insufficient contrast is a genuine, common WCAG Perceivable- pillar failure, and — unlike many accessibility checks — it's mechanically, precisely checkable: DevTools' color picker shows a live contrast ratio AND an explicit AA/AAA pass-fail indicator the moment you pick a text/background color pair, removing any guesswork. Never ship a final color decision without this check specifically — "it looks fine to me" is not equivalent to "it passes 4.5:1," and the gap between the two is exactly where real accessibility complaints originate.

Part 6 — Dark Mode: color-scheme and prefers-color-scheme

:root { color-scheme: light dark; }   /* tells the BROWSER both themes are supported —
                                            form controls, scrollbars auto-adapt for free */

@media (prefers-color-scheme: dark) {
  :root {
    --bg: #1a1a1a;
    --text: #f0f0f0;
  }
}
@media (prefers-color-scheme: light) {
  :root {
    --bg: #ffffff;
    --text: #1a1a1a;
  }
}

prefers-color-scheme reads the user's OS-level (or browser-level) theme preference — a genuine media query, functioning exactly like a viewport-width breakpoint but for a preference rather than a dimension. The often-missed companion, color-scheme: light dark on the root, is what makes BUILT-IN, un-styled browser UI (native form control chrome, scrollbars, the default page background before your CSS even loads) correctly switch to dark styling too — omitting it is a common, real cause of "my dark-mode CSS looks right except native checkboxes/scrollbars are still light-themed and jarring," since that specific mismatch isn't something your own color variables control at all.

Interview Questions

Q1 (beginner): Why is HSL often preferred over hex/RGB when a color needs to be systematically varied (e.g., generating a palette)?

HSL's three channels (Hue, Saturation, Lightness) map directly onto how humans actually think about and reason with color — "make this lighter," "make this less saturated" are direct, obvious operations on HSL's own channels, while hex/RGB describe raw channel intensities with no intuitive operation corresponding to "lighter" or "more vivid." An entire tint/shade palette can be generated from one hue+saturation pair by varying only lightness — a systematic approach hex/ RGB doesn't offer.

Q2 (beginner): What are the actual WCAG AA contrast ratio requirements for normal and large text?

4.5:1 minimum for normal text, 3:1 minimum for large text (18pt+, or 14pt+ bold). These are precise, computable numbers based on relative luminance — not a subjective "readable enough" judgment — and DevTools' color picker shows the live ratio plus an explicit pass/fail indicator for any chosen text/background pair.

Q3 (mid): Explain the specific flaw in HSL's lightness channel that OKLCH corrects.

HSL's lightness is a mathematical average of RGB channel values, not a measure of actual human- perceived brightness — different hues at the IDENTICAL HSL lightness number can look meaningfully lighter or darker to a real viewer (pure yellow vs. pure blue at the same 50% HSL lightness look very different in perceived brightness). OKLCH is engineered so equal lightness values DO look equally bright across every hue, making it the more reliable choice for generating perceptually consistent color scales or reasoning precisely about brightness.

Q4 (mid): A dark-mode implementation correctly switches all custom-styled elements' colors via prefers-color-scheme, but native checkboxes and the browser's scrollbar remain light-themed and visually jarring. Diagnose.

Missing color-scheme: light dark on the root element — prefers-color-scheme media queries only control the AUTHOR's own CSS custom properties/rules, while color-scheme is the separate property that tells the BROWSER itself both themes are supported, enabling it to automatically re-theme its own native, unstyled UI (form control chrome, scrollbars, default backgrounds) to match the active scheme. The two mechanisms are complementary and both required for a fully consistent dark-mode experience.

Q5 (senior): Your design system's color tokens are all defined in HSL, and a recent accessibility audit found that several "equally light" tint-scale steps across different brand hues (blue-100, green-100, red-100) have measurably different, inconsistent contrast ratios against the same text color, despite identical HSL lightness values. Diagnose and propose a fix.

Direct instance of HSL's perceptual-uniformity flaw (Part 4): "100" tint steps across different hues were generated using the SAME HSL lightness percentage, but because HSL lightness doesn't correspond to actual perceived brightness consistently across hues, the ACTUAL rendered brightness — and therefore actual contrast ratio against a fixed text color — genuinely differs hue to hue despite the tokens' lightness values matching numerically. Fix: migrate the color token generation to OKLCH, which is specifically engineered for this exact guarantee — equal OKLCH lightness values produce equal PERCEIVED brightness (and correspondingly, much more consistent actual contrast ratios) regardless of hue. This is a genuinely mechanical remediation (regenerate the tint/shade scale in OKLCH using the same target hue/chroma per brand color, varying only OKLCH lightness) rather than a redesign, and it directly and systematically resolves the audit's finding rather than requiring per-token manual contrast tweaking, which would need to be redone every time a new brand hue is added to the system in the future.

Q6 (senior): A product wants hover/active/disabled state colors for all 12 of its brand palette's base colors, automatically derived (not hand-picked) so a future rebrand only requires changing the 12 base values. Design the CSS architecture.

The correct architecture avoids hand-picking 36+ additional hardcoded state colors (12 base × 3 states) entirely, using color-mix() or relative color syntax to DERIVE every state color from its base at the point of use: .btn:hover { background: color-mix(in oklch, var(--base-color) 85%, black); } and .btn:disabled { background: color-mix(in oklch, var(--base-color) 50%, gray); } — using OKLCH as the mixing color space specifically (rather than the default srgb) so the DARKENING/desaturation effect itself is perceptually consistent across all 12 different base hues (a direct application of Part 4's OKLCH-uniformity property to this exact problem — mixing in srgb would produce visibly inconsistent darkening amounts across different hues, the identical flaw as HSL, since srgb interpolation shares the same non-perceptual- uniformity issue). With this architecture, a future rebrand changing any of the 12 base --brand-N custom properties automatically, correctly regenerates all 36 derived state colors with zero additional hand-maintained code — the state-color RULES are written once, generically, and apply identically to every current and future base color the design system defines, directly satisfying the stated requirement.

🏋️ Exercises

1 (easy): HSL palette generation. Pick one hue/saturation pair. Generate a 5-step tint/shade scale (very light to very dark) by varying ONLY the lightness value (e.g., 95%, 75%, 51%, 30%, 12%). Render all 5 as swatches and confirm they read as a coherent, related family.
Example 
--h: 217; --s: 83%;
.step-1 { background: hsl(var(--h) var(--s) 95%); }
.step-2 { background: hsl(var(--h) var(--s) 75%); }
.step-3 { background: hsl(var(--h) var(--s) 51%); }
.step-4 { background: hsl(var(--h) var(--s) 30%); }
.step-5 { background: hsl(var(--h) var(--s) 12%); }

All five should read clearly as "the same color family at different lightness levels" — the direct, hands-on demonstration of HSL's systematic palette-generation payoff from Part 2.

2 (easy): Contrast-check drill. Pick 4 text/background color pairs from any real site you like the look of (or your own earlier CSS-track pages). Using DevTools' color picker contrast indicator, check each against AA's 4.5:1 (normal text) threshold. Report how many pass and, for any failures, adjust ONE of the two colors minimally until it passes.
What to expect

Real-world designs, even professionally-made ones, not infrequently include at least one pairing (often light-gray text on white, used for "secondary" or "muted" text) that falls short of 4.5:1 — finding this in practice, rather than only reading about the requirement, is the exercise's real point. Adjusting a failing pair usually means darkening light text or lightening a light background slightly — small, often visually negligible changes that make a real accessibility difference.

3 (medium): color-mix() hover states. Build 3 buttons with 3 different base --brand-color values (defined as custom properties). Write ONE generic hover rule using color-mix() that correctly darkens whichever base color is in play, with no per-button hardcoded hover color. Change one button's base color and confirm its hover state updates automatically with zero additional CSS.
Solution 
.btn { background: var(--brand-color); }
.btn:hover { background: color-mix(in oklch, var(--brand-color) 85%, black); }

.btn-a { --brand-color: #1a73e8; }
.btn-b { --brand-color: #d93025; }
.btn-c { --brand-color: #188038; }

All three buttons' hover states should darken correctly and consistently despite having entirely different base hues, with only ONE hover rule written — confirming the generic, reusable derivation pattern this lesson recommends over hand-picked per-color hover values.

4 (medium): Dark mode, fully wired. Build a small page with prefers-color-scheme-based light/dark custom properties AND color-scheme: light dark on the root. Include at least one native form control (a checkbox or a select). Toggle your OS/browser's dark mode setting and confirm BOTH your custom-styled elements AND the native form control switch themes correctly. Then remove color-scheme and observe what breaks.
Expected before/after

With color-scheme set: the native checkbox/select should render with dark-appropriate chrome (darker background, lighter checkmark, etc.) automatically. After removing color-scheme: the custom-styled parts of the page should still correctly switch (they're controlled by your own prefers-color-scheme rules), but the native form control should visibly remain in its LIGHT default appearance despite the rest of the page being dark — a direct, hands-on reproduction of this lesson's most commonly-missed dark-mode detail.

5 (hard): HSL vs OKLCH, perceptually compared. Generate the identical "5 steps at fixed lightness, varying only hue" swatch set (e.g., 5 different hues all at "50% lightness") twice — once in HSL, once in OKLCH. View both side by side and assess: do all 5 HSL swatches look equally bright to your own eye? Do all 5 OKLCH swatches? Write two sentences on what you observe.
Expected observation 
/* HSL: same lightness number, hue varies */
hsl(60 100% 50%)  hsl(120 100% 50%)  hsl(240 100% 50%)  hsl(300 100% 50%)  hsl(0 100% 50%)

/* OKLCH: same lightness number, hue varies */
oklch(70% 0.15 90)  oklch(70% 0.15 145)  oklch(70% 0.15 260)  oklch(70% 0.15 320)  oklch(70% 0.15 20)

The HSL set should show visibly UNEVEN perceived brightness (yellow/green typically look noticeably lighter than blue/red at the identical 50% lightness number); the OKLCH set should look considerably more evenly bright across all 5 hues at the identical 70% lightness number — a direct, visual, side-by-side confirmation of Part 4's core claim, seen with your own eyes rather than taken on the lesson's word alone.

6 (hard): The full derived-state system. Implement Q6's complete architecture: a design system with 4 base brand colors (as custom properties), and ONE set of generic hover/active/disabled rules using color-mix() in oklch that correctly derive state colors for ALL 4 base colors with zero per-color hardcoded overrides. Verify by adding a 5th base color later and confirming its states derive correctly with no CSS changes needed at all.
Solution 
.btn:hover     { background: color-mix(in oklch, var(--brand-color) 85%, black); }
.btn:active    { background: color-mix(in oklch, var(--brand-color) 70%, black); }
.btn:disabled  { background: color-mix(in oklch, var(--brand-color) 50%, gray); opacity: 0.6; }

/* adding a 5th color needs ONLY this — zero changes to the rules above: */
.btn-e { --brand-color: #7c3aed; }

The 5th color's hover/active/disabled states should render correctly and consistently the MOMENT its --brand-color is defined — no new CSS rule was needed anywhere, directly and concretely satisfying the "future rebrand/new-color needs zero additional hand-maintained code" requirement from Q6's interview scenario.

🎯 Key Takeaways

  • Hex/RGB describe exact channel values (precise, but hard to reason about); HSL's Hue/Saturation/Lightness maps onto how humans actually think about color, making it the right tool for systematic palette generation and variation.
  • color-mix() and relative color syntax derive hover/active/state colors FROM a base color, eliminating hardcoded state colors that silently drift out of sync when the base changes.
  • OKLCH corrects HSL's real flaw — HSL's lightness is a raw RGB average, not perceived brightness, so equal HSL lightness looks unevenly bright across different hues; OKLCH's lightness IS perceptually uniform.
  • WCAG AA requires 4.5:1 contrast for normal text, 3:1 for large text — precise, computable numbers checkable directly in DevTools, never a subjective "looks readable" judgment.
  • Dark mode needs BOTH prefers-color-scheme (your own CSS's theme switching) AND color-scheme: light dark on the root (native browser UI like form controls/scrollbars re-theming) — missing the second is the most common incomplete-dark-mode bug.
  • Mixing/deriving colors in oklch (rather than the default srgb) keeps the derived effect perceptually consistent across different base hues — the same uniformity property applied to color transformations, not just static palettes.
