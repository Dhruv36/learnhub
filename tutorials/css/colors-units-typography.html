Colors, Units & Typography Basics

Three systems that decide whether an interface feels professional: how you specify color, which units you measure in, and how text is set. Each has a "works in a demo" level and a "survives accessibility audits and dark mode" level — this lesson teaches the second.

Color — Four Syntaxes and When Each Earns Its Place

color: #2f8d46;                    /* hex — compact, what design tools export; #rgba adds alpha (#2f8d46cc) */
color: rgb(47 141 70 / 0.8);       /* channels + alpha — what getComputedStyle returns */
color: hsl(140 50% 37%);           /* hue 0-360°, saturation, lightness — human-tweakable */
color: oklch(0.55 0.15 150);       /* lightness, chroma, hue — perceptually uniform (modern) */

Why HSL changed workflows: variants become arithmetic — hover = same hue, lightness −8%; a palette = one hue, stepped lightness. Why OKLCH is replacing it: HSL lies about lightness — hsl(60 100% 50%) (yellow) is visually far brighter than hsl(240 100% 50%) (blue) despite "equal" L. OKLCH's lightness is perceptually true, so a 10-step scale generated in OKLCH looks evenly spaced across every hue — which is exactly how modern design-token palettes (Tailwind v4, Radix) are generated. It also unlocks wide-gamut P3 colors newer screens can display.

/* Runtime color math — variants from ONE source token: */
--brand: oklch(0.55 0.15 150);
--brand-hover: oklch(from var(--brand) calc(l - 0.07) c h);      /* relative color syntax */
--brand-tint: color-mix(in oklch, var(--brand), white 85%);       /* backgrounds */

currentColorThe element's computed text color — svg { fill: currentColor } makes icons follow text automatically

color-scheme: light darkOn :root — native widgets (scrollbars, form controls) follow your theme

Contrast — the Non-Negotiable Number

WCAG minimums: 4.5:1 for body text, 3:1 for large text (≥24px, or ≥18.7px bold) and UI components/graphics. Check every token pair in both themes — dark mode fails more often (pure #fff on dark backgrounds also halos; slightly-dimmed #e6edf3 reads better). DevTools' color picker displays the ratio live and suggests compliant alternatives. Also: never encode meaning in color alone — pair state colors with an icon or text.

Units — the Decision Table

UnitRelative toReach for it when…
pxFixed (CSS pixel — density-independent)Hairlines, borders, shadows, radii — physical details that shouldn't scale with text
remRoot font-size (16px default)Type and spacing — the default. Respects the user's browser font-size setting
emElement's own font-size (parent's, for font-size itself)Component-internal scaling: button padding in em → size variants from one rule. ⚠ compounds when nested

vw/vhViewportHero sections; ⚠ 100vh breaks under mobile URL bars
dvh/svh/lvhDynamic/small/large viewport100dvh = actually-visible height as browser chrome collapses — use it
chWidth of "0"Reading measure: max-width: 65ch; input widths matched to expected content
lhElement's line-heightSpacing that aligns to the text rhythm (icon sizing next to text)
♿ Why rem vs px is an accessibility issue, precisely: browser zoom scales everything, and px follows it fine. But the browser's default font-size setting (Settings → Appearance — used heavily by low-vision users) only affects rem/em. Set body text in px and you've silently overridden that user's explicit preference. WCAG 1.4.4 audits test text-only scaling to 200%.

Fluid Values with clamp()

h1   { font-size: clamp(1.8rem, 1.2rem + 2.5vw, 2.6rem); }
/*                       └min┘  └── preferred ──┘  └max┘
   The rem term in 'preferred' keeps zoom working (pure-vw type ignores user scaling: WCAG fail).
   Same idea for spacing: */
.section { padding-block: clamp(2rem, 6vw, 5rem); }

Typography — the Settings That Make Text Readable

body {
  font-family: system-ui, -apple-system, "Segoe UI", Roboto, "Noto Sans", sans-serif;
  font-size: 1rem;              /* 16px baseline — never smaller for body */
  line-height: 1.6;             /* UNITLESS — recomputes per element (1.6em would inherit a frozen px value) */
  -webkit-font-smoothing: antialiased;   /* macOS: only on dark backgrounds — thins light-on-dark text */
}
h1, h2, h3 { line-height: 1.15; text-wrap: balance; }   /* tight leading + no orphan words */
p { max-width: 65ch; text-wrap: pretty; }                /* 45–75ch measure + paragraph orphan control */
PropertyProduction guidance
font-weight400 body / 600–700 headings; each extra webfont weight = another file — 2-3 weights max
letter-spacingLarge headings: −0.01 to −0.03em; ALL-CAPS labels: +0.05em; body: leave alone
font-variant-numeric: tabular-numsEqual-width digits — tables, timers, prices stop jiggling
font-feature-settings / ligaturesCode fonts with ligatures in editors, off in body text
hyphens: auto + langJustified/narrow columns need hyphenation, which needs the correct lang attribute
text-overflow: ellipsisNeeds the trio: overflow:hidden; white-space:nowrap; multi-line = -webkit-line-clamp

Webfont Loading — the Full Production Recipe

<link rel="preload" href="/fonts/inter-var.woff2" as="font" type="font/woff2" crossorigin>

@font-face {
  font-family: Inter;
  src: url(/fonts/inter-var.woff2) format("woff2");
  font-weight: 100 900;            /* variable font: every weight, ONE file */
  font-display: swap;              /* show fallback text immediately (FOUT), never invisible (FOIT) */
}
/* Kill the swap layout-jump: size the fallback to match the webfont */
@font-face {
  font-family: "Inter-fallback";
  src: local("Arial");
  size-adjust: 107%; ascent-override: 90%;   /* values from fontaine/capsize tooling */
}
body { font-family: Inter, "Inter-fallback", system-ui, sans-serif; }

Self-host fonts (third-party font CDNs no longer share caches across sites and add a connection), subset to the scripts you actually use (latin subset ≈ 25KB vs 300KB full), and remember: system-ui costs zero bytes and zero milliseconds — the correct choice until brand explicitly demands otherwise.

Interview Questions

Q1: rem vs em — precise difference and one legitimate use of em?

rem resolves against the root font-size; em against the element's own (parent's, when setting font-size itself), so em compounds when nested. Legit em use: component-internal proportions — button padding in em scales the whole control from a single font-size change.

Q2: Why is OKLCH replacing HSL for design systems?

Perceptual uniformity: OKLCH lightness matches human vision, so generated scales are visually even across hues, contrast is easier to hit, and interpolation doesn't detour through gray. Plus wide-gamut P3 support and relative-color math (oklch(from …)) for programmatic variants.

Q3: Walk through FOIT vs FOUT and how you'd eliminate both problems.

FOIT: invisible text while the webfont loads (default in some engines) — kill with font-display:swap. FOUT: the visible fallback swaps to the webfont with a layout jump — minimize by preloading the font AND metric-matching the fallback (size-adjust/ascent-override) so the swap is pixel-neutral. Variable fonts reduce total requests.

Q4: A designer hands you #777 text on white (4.48:1). Ship it?

No — body text needs 4.5:1; 4.48 fails audits. Darken a step (#767676 is the classic 4.54:1 boundary), or reserve #777 for large text (3:1 rule). Bonus: codify the pair as tokens with a stylelint contrast check so it can't regress.

🏋️ Practice (Playground):
1. Build a 5-step green scale twice — HSL lightness steps vs OKLCH lightness steps — and compare evenness by eye.
2. Set an article at max-width 65ch / line-height 1.6, then break it (30ch, 90ch, 1.2, 2.0) and feel each failure mode.
3. Make a price table jiggle-free with tabular-nums; toggle it off and watch the digits dance.
4. Implement clamp() type for h1 (min 1.8rem, max 2.6rem) and verify BOTH: resize fluidity and Ctrl+ zoom still growing text.

🎯 Key Takeaways

  • Hex for tool interchange, HSL for human tweaks, OKLCH for generated palettes and P3 — with color-mix/relative-color for variants.
  • Contrast: 4.5:1 body, 3:1 large/UI — verified per token pair, in both themes, never encoded by color alone.
  • rem for type/spacing (user preference!), px for physical details, em for internal proportions, dvh over vh, ch for measure.
  • Fluid = clamp(min, rem+vw, max) — the rem term keeps zoom legal.
  • Text: 16px+ body, unitless line-height 1.5–1.7, 45–75ch, balance/pretty wrapping, tabular-nums for data.
  • Fonts: system stack by default; else woff2 + subset + variable + preload + swap + metric-matched fallback.
