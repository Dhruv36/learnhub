SOLID & Clean Code Professional Java

What you'll master in this lesson: the five SOLID principles that guide object-oriented design — each with a concrete Java example and, crucially, what it looks like when violated; the clean-code practices (naming, small functions, DRY vs premature abstraction) that make code maintainable; and the judgment to apply these as guidelines, not dogma. SOLID and clean code are the vocabulary of software-quality discussions in every code review and senior interview — and they tie together the OOP, Interfaces, and Design Patterns lessons into a coherent design philosophy.

Part 1 — S: Single Responsibility Principle (SRP)

// VIOLATION — a class doing three unrelated things (persistence, formatting, email):
class User {
    void save()      { /* database logic */ }
    String toHtml()  { /* presentation logic */ }
    void sendEmail() { /* notification logic */ }
}   // three reasons to change → a change to email logic risks breaking persistence.

// SRP — each class has ONE reason to change:
class User { /* just the data + domain behavior */ }
class UserRepository { void save(User u) { ... } }        // persistence
class UserRenderer  { String toHtml(User u) { ... } }     // presentation
class UserNotifier  { void email(User u) { ... } }        // notification
💡 "A class should have one reason to change." When a class mixes concerns (data + persistence + presentation), a change to any one concern risks the others, testing is hard (you can't test formatting without a database), and reuse is impossible. SRP is the most important and most-violated principle — it's the "do one thing" rule from the Methods lesson at the class level. The layered structure it produces (domain, repository, service, presentation) is exactly the architecture the Spring Boot track uses.

Part 2 — O: Open/Closed Principle (OCP)

// VIOLATION — adding a shape means EDITING this method (and every method like it):
double area(Shape s) {
    if (s instanceof Circle c)    return Math.PI * c.r() * c.r();
    else if (s instanceof Square sq) return sq.side() * sq.side();
    // add a Triangle → edit here, and everywhere else that switches on shape type
}

// OCP — open for EXTENSION, closed for MODIFICATION. Use polymorphism:
interface Shape { double area(); }           // each shape implements its own area()
record Circle(double r) implements Shape { public double area() { return Math.PI * r * r; } }
record Square(double side) implements Shape { public double area() { return side * side; } }
// Adding a Triangle = a NEW class; NO existing code changes.
🎯 "Open for extension, closed for modification." You should be able to add new behavior (a new shape, a new payment method) by adding new code, not editing existing, tested code. Polymorphism (Inheritance lesson) and interfaces (Interfaces lesson) are how you achieve it — each new type provides its own implementation of a contract. (Note: sealed types + pattern matching, from the Modern Java lessons, deliberately make the opposite trade-off for closed hierarchies — OCP is a guideline weighed against exhaustiveness, not an absolute.)

Part 3 — L: Liskov Substitution Principle (LSP)

// VIOLATION — a subtype that breaks the parent's contract:
class Rectangle { void setWidth(int w); void setHeight(int h); int area(); }
class Square extends Rectangle {
    void setWidth(int w)  { this.width = this.height = w; }   // ❌ changes BOTH — violates Rectangle's contract
}
// Code that works for Rectangle (setWidth(5); setHeight(4); expect area==20) BREAKS for Square (area==16).
// A Square "is a" Rectangle mathematically but NOT behaviorally — the classic LSP counterexample.
🔍 "Subtypes must be substitutable for their base types without breaking behavior." If code written against a base type misbehaves when given a subtype, the subtype violates LSP — the inheritance is wrong even if "is-a" sounds true. The lesson: inheritance must preserve the base's behavioral contract, not just its shape. Violations (like Square/Rectangle) are a strong signal to use composition instead (Inheritance lesson) — model both as separate types, or compose, rather than forcing a broken subtype relationship.

Part 4 — I & D: Interface Segregation & Dependency Inversion

// I — INTERFACE SEGREGATION: don't force clients to depend on methods they don't use.
// VIOLATION — a "fat" interface:
interface Worker { void work(); void eat(); void sleep(); }   // a Robot must implement eat()/sleep()?!
// FIX — small, focused interfaces:
interface Workable { void work(); }
interface Feedable { void eat(); }        // a Robot implements only Workable

// D — DEPENDENCY INVERSION: depend on ABSTRACTIONS, not concretions.
// VIOLATION — high-level code hard-wired to a concrete class:
class OrderService { private final MySqlDatabase db = new MySqlDatabase(); }   // coupled to MySQL
// FIX — depend on an interface, inject the implementation (Interfaces lesson):
class OrderService {
    private final Database db;                        // an abstraction
    OrderService(Database db) { this.db = db; }       // injected — swap MySQL for a mock or Postgres
}
🎯 Interface Segregation keeps interfaces small and role-focused so implementers aren't forced to stub out methods they don't need. Dependency Inversion is "program to an interface, not an implementation" (Interfaces lesson) plus "inject the dependency" — it's the direct theoretical basis of Spring's dependency injection (Spring Boot track). Both point the same way: depend on small abstractions, wire concrete implementations in from outside. Together the five principles produce loosely-coupled, testable, extensible code.

🔍 DRY vs premature abstraction — the crucial nuance. "Don't Repeat Yourself" means don't duplicate a piece of knowledge. But two pieces of code that look similar today may represent different concepts that will evolve differently — merging them creates a wrong coupling that's painful to unpick later. The rule of thumb ("rule of three"): tolerate a little duplication until you see the pattern three times and understand the real abstraction. Over-eager DRY (abstracting on the first duplication) is as harmful as copy-paste — it's the same over-engineering instinct as pattern fever.

Part 6 — Deep Dive: Principles as Guidelines, Not Laws

  • SOLID serves goals, not the reverse. The goals are code that's decoupled, testable, and easy to change safely. When a SOLID principle would add complexity without serving those goals (a needless interface for a single implementation), skip it — dogmatic SOLID produces over-abstracted, indirection-heavy code as bad as spaghetti.
  • The principles interlock. SRP produces the small classes that OCP extends; DIP and ISP produce the small interfaces you program to; LSP keeps inheritance honest. They're facets of one idea: manage dependencies so change is local and safe.
  • Clean code is for humans. Code is read far more than written; the audience is the next developer (often you). Optimize for their comprehension — clear names, small units, obvious flow — over cleverness. "Any fool can write code a computer understands; good programmers write code humans understand" (Martin Fowler).
  • Judgment over rules. Every principle here has a "but" — OCP vs sealed exhaustiveness, DRY vs premature abstraction, interfaces vs YAGNI. Seniority is knowing the principle, knowing its exception, and choosing well for the situation.
🔍 Senior framing: SOLID and clean code are the shared language of software quality — every serious code review and design discussion draws on them — but their real content is a single aim: structure code so that change is safe, local, and cheap, because most of a system's cost is in changing it after it's written. SRP localizes change to one class; OCP lets you add behavior without editing tested code; LSP keeps polymorphism trustworthy; ISP and DIP decouple through small abstractions so pieces can vary and be tested independently. Clean-code practices make the result legible to the humans who maintain it. The mature engineer wields these as guidelines calibrated by judgment, not commandments: applying a principle when it earns its keep and consciously breaking it when dogma would add complexity (the interface with one implementation, the abstraction extracted too early, the OCP hierarchy where a sealed exhaustive switch is clearer). Interviewers ask about SOLID to hear whether you understand why each principle exists and can reason about the trade-offs — not whether you can recite five acronyms. The strongest answers pair the principle with its purpose and its exception, because that reveals someone who designs for maintainability rather than for a checklist.

Interview Questions

Q1 (beginner): What is the Single Responsibility Principle?

SRP states that a class should have one reason to change — that is, it should be responsible for a single concern. A User class that handles its data, database persistence, HTML rendering, and email sending has four reasons to change, so a change to email logic can break persistence, it's hard to test one concern in isolation, and none of the parts are reusable. Applying SRP, you split those concerns into focused classes (User, UserRepository, UserRenderer, UserNotifier), each with one responsibility and one reason to change. It's the class-level version of "a method should do one thing," and it's the foundation of the layered architectures used in real applications.

Q2 (beginner): Explain the Open/Closed Principle with an example.

OCP says software entities should be open for extension but closed for modification — you should be able to add new behavior without editing existing, tested code. The anti-example is an area(Shape) method with an if/instanceof chain over shape types: adding a new shape forces you to modify that method (and every similar one), risking regressions. Applying OCP, you define a Shape interface with an area() method and let each shape implement its own; adding a Triangle is a new class that implements the interface, with no changes to existing code. Polymorphism and interfaces are the mechanism. (A nuance: sealed types with exhaustive pattern matching intentionally trade OCP for compile-time completeness on closed hierarchies, so OCP is a guideline balanced against other concerns.)

Q3 (mid): What is the Liskov Substitution Principle and give a violation?

LSP states that objects of a subtype must be substitutable for their base type without altering the correctness of the program — code written against the base type must work when given any subtype. The classic violation is Square extends Rectangle: a Rectangle's contract lets you set width and height independently (setWidth(5); setHeight(4) should give area 20), but a Square forced to keep sides equal changes both when you set one, so code relying on the Rectangle contract computes the wrong area (16) — the substitution breaks behavior. Even though a square "is a" rectangle mathematically, it isn't behaviorally, so the inheritance is wrong. The principle teaches that inheritance must preserve the base's behavioral contract (preconditions, postconditions, invariants), and a violation is a signal to use composition or separate types instead of a broken subtype relationship.

Q4 (mid): What is Dependency Inversion and how does it relate to Spring?

The Dependency Inversion Principle says high-level modules should depend on abstractions, not on concrete low-level implementations, and that both should depend on abstractions. In practice: instead of OrderService instantiating a concrete MySqlDatabase (hard-wiring it to MySQL and making it untestable), OrderService depends on a Database interface and receives the implementation from outside — dependency injection. This decouples the high-level business logic from the specific technology, so you can swap MySQL for Postgres or inject a mock in tests by changing configuration, not code. It's "program to an interface, not an implementation" plus "inject the dependency," and it is precisely the theoretical foundation of Spring's dependency injection: Spring beans declare their collaborators as interface-typed constructor parameters, and the container supplies concrete implementations, so the concrete choice becomes a late-bound, replaceable decision. DIP is why the Spring programming model is testable and flexible.

Q5 (senior): DRY is widely taught, yet "premature abstraction" is a real problem. Reconcile them.

They're reconciled by being precise about what DRY actually forbids: duplication of knowledge — a single fact, rule, or decision expressed in more than one place, so that changing it requires editing multiple spots and risks inconsistency. That kind of duplication should be removed. What DRY does not mandate is deduplicating code that merely looks similar but represents different concepts that happen to have the same shape today. Merging those creates a false abstraction — a coupling between things that will evolve independently — and when they later diverge you're forced to add parameters, flags, and special cases to the shared code until it's more complex and fragile than the duplication would have been, and painful to pull back apart. The practical discipline is the "rule of three": tolerate a small amount of apparent duplication until you've seen the pattern about three times, because only then do you understand which parts are genuinely the same concept (the real abstraction) and which are coincidental. Abstracting on the very first duplication is guessing at an abstraction you don't yet understand — the same over-engineering instinct as pattern fever, just wearing DRY's clothes. So the senior stance is: aggressively remove duplicated knowledge; be patient about abstracting duplicated appearance; and prefer a little well-understood duplication over a premature, wrong abstraction, since duplication is cheap to fix later while a bad abstraction is expensive to remove.

Q6 (senior): Argue for treating SOLID as guidelines rather than strict rules.

Because SOLID exists to serve concrete goals — code that is decoupled, testable, and safe and cheap to change — and blindly maximizing the principles can defeat those very goals by producing over- abstraction: a proliferation of one-method interfaces, indirection layers, and speculative extension points that make simple logic hard to follow and change, which is a failure of maintainability just as much as a god class is. Each principle, applied without judgment, has a failure mode: dogmatic OCP adds an inheritance/strategy hierarchy where a sealed exhaustive switch would be clearer and safer; strict DIP wraps every collaborator in an interface even when there will only ever be one implementation (violating YAGNI and adding navigation overhead); aggressive ISP shatters cohesive interfaces into so many pieces that the design becomes hard to comprehend; and reflexive SRP can over-fragment logic that genuinely belongs together. The principles also sometimes conflict with each other and with other valuable properties (exhaustiveness, simplicity, performance), so they can't all be absolutes. The mature approach is to understand the purpose behind each principle and apply it when it demonstrably makes change safer or the code clearer for the humans who maintain it, and to consciously decline or break it when adherence would add complexity without benefit — keeping the concrete implementation until a second one actually appears, inlining an abstraction that isn't earning its keep, choosing a closed sealed hierarchy over an open one when completeness matters more than extensibility. Guidelines calibrated by judgment produce better systems than rules followed by rote; the acronyms are a means to maintainability, not the definition of it, and the ability to say when to apply each and when not to is exactly what distinguishes design maturity from checklist compliance.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Apply SRP. Given a Report class that computes data, formats it as HTML, and writes it to a file, split it into responsibilities and name each class.
Solution

ReportCalculator (computes the data), ReportFormatter (data → HTML/other formats), and ReportWriter (persists/output). Each has one reason to change, so you can change the output format without touching calculation, test formatting without a filesystem, and reuse the calculator with a different formatter. This is SRP producing a testable, layered design.

2 (easy): Fix an OCP violation. Refactor an if/instanceof area() method into a polymorphic design so adding a shape needs no edits to existing code.
Solution

Define interface Shape { double area(); } and give each shape (as a record) its own area(). The generic code just calls shape.area(). Adding a Triangle is a new record implementing Shape — zero edits to existing, tested code. Extension by addition, not modification.

3 (medium): Spot the LSP violation. Explain why Square extends Rectangle breaks code that sets width and height independently, and give a better design.
Solution

Rectangle's contract allows independent width/height, but Square forces them equal, so setWidth(5); setHeight(4); area() yields 16 instead of the expected 20 — a subtype breaking the base's behavioral contract. Better design: don't make Square a subtype of Rectangle; model them as separate implementations of a common Shape interface (each immutable with its own area()), or use composition. The "is-a" test alone is insufficient — behavior must be substitutable.

4 (medium): Apply DIP. Refactor a class that does new MySqlUserDao() internally so it depends on an abstraction and the implementation is injected. Explain the testing benefit.
Solution

Define a UserDao interface; make the class take a UserDao via its constructor instead of instantiating MySqlUserDao. Now you inject MySqlUserDao in production and a mock/fake in tests, so you can unit-test the class's logic with no database. This is Dependency Inversion enabling testability — and exactly how Spring wires beans.

5 (hard): DRY judgment. Two methods have near-identical code today but handle different business concepts (tax calc vs shipping calc). Argue whether to deduplicate now, and what would change your answer.
Solution

Don't deduplicate yet: they're the same shape but different concepts (tax and shipping rules will likely evolve independently), so merging creates a coupling that later forces flags/parameters and special cases as they diverge — a premature, wrong abstraction. Tolerate the duplication (it's cheap to fix later). What would change the answer: if they represent the same knowledge (a single rule expressed twice), deduplicate immediately; and by the "rule of three," once you see the genuinely-shared pattern a third time and understand the real abstraction, extract it then. DRY targets duplicated knowledge, not coincidental resemblance.

6 (hard): Explain-it test. Out loud: (a) each SOLID letter in one sentence with its purpose; (b) DRY vs premature abstraction; (c) why SOLID is guidelines not laws.
Self-check rubric

Pass (a) if you gave SRP (one reason to change), OCP (extend without modifying), LSP (subtypes substitutable), ISP (small focused interfaces), DIP (depend on abstractions/inject). Pass (b) if you said DRY targets duplicated knowledge, and rule-of-three guards against abstracting coincidental similarity. Pass (c) if you framed SOLID as serving decoupling/testability/safe-change and named a case to break a principle. This synthesizes OOP, Interfaces, Patterns — and grounds the Spring Boot track's DI-based architecture.

🎯 Key Takeaways

  • SRP — one reason to change (split concerns into focused classes); the class-level "do one thing."
  • OCP — open for extension, closed for modification (add types via polymorphism, don't edit tested code).
  • LSP — subtypes must honor the base's behavioral contract (else compose, don't inherit).
  • ISP (small role interfaces) + DIP (depend on abstractions, inject) — the theoretical basis of Spring's dependency injection.
  • Clean code: intent-revealing names, small functions, few parameters, comments that explain why; DRY targets duplicated knowledge, not looks (rule of three).
  • All of it is guidelines serving one goal — make change safe, local, and cheap — applied with judgment, not as dogma.
