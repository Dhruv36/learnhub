Arrays Foundations

What you'll master in this lesson: how arrays store data contiguously and why that makes indexing O(1); the fixed-size limitation that pushes you toward ArrayList; the default-value and bounds rules that cause common exceptions; multi-dimensional arrays; the essential Arrays utility methods; and the shallow-vs-deep copy distinction that causes subtle bugs. Arrays are the lowest-level collection and the foundation every other data structure is built on.

Part 1 — Declaring & Creating Arrays

int[] nums = new int[5];         // 5 ints, all initialized to 0 (the default)
int[] primes = {2, 3, 5, 7, 11}; // array literal — size inferred from the contents
String[] names = new String[3];  // 3 Strings, all initialized to NULL (reference default!)

nums[0] = 10;                    // set by index (0-based)
int first = nums[0];             // get by index
int len = nums.length;           // LENGTH is a field, not a method (no parentheses)
                                 // — contrast String.length() and List.size() which ARE methods
💡 Defaults matter. A new int[5] is all zeros; a new boolean[5] is all false; but a new String[3] (or any reference-type array) is all null. Iterating a fresh String[] and calling a method on an element gives a NullPointerException until you populate it. And note the inconsistency the language never fixed: arrays use .length (field), Strings use .length() (method), collections use .size() (method) — memorize it.

Part 2 — Why Arrays Are Fast, and Their One Big Limitation

An array is a contiguous block of memory — element i lives at base_address + i × element_size. That's why arr[i] is O(1): the CPU computes the address arithmetically, no searching. This memory layout also gives excellent cache locality — iterating an array sequentially is one of the fastest things a computer does.

// The limitation: arrays are FIXED-SIZE. You choose the length at creation and can't change it.
int[] a = new int[3];
// a[3] = 99;     ❌ ArrayIndexOutOfBoundsException — index 3 doesn't exist (valid: 0,1,2)
// there is no a.add(...) — arrays can't grow.

// To "resize", you allocate a bigger array and copy:
int[] bigger = Arrays.copyOf(a, 6);   // new length 6, old elements copied, rest default
🎯 This limitation is the whole reason ArrayList exists. ArrayList wraps an array and handles growth for you — when it fills up, it allocates a larger backing array (typically 1.5×) and copies. So "array vs ArrayList" isn't really a choice between unrelated things: ArrayList is a growable array. Use a raw array for fixed-size, performance- critical, or primitive data; use ArrayList (Collections lesson) for the common grow-as-you-go case.

Part 3 — Bounds, Iteration & the Off-by-One Recap

int[] arr = {10, 20, 30};
// Indexed loop — when you need the position:
for (int i = 0; i < arr.length; i++) {    // < length, NOT <= (indices are 0..length-1)
    System.out.println(i + ": " + arr[i]);
}
// For-each — when you just need the values (your default, no index bug possible):
for (int value : arr) {
    System.out.println(value);
}

arr[3];     // ❌ ArrayIndexOutOfBoundsException — the classic. Access is always checked at runtime.
arr[-1];    // ❌ also out of bounds (no negative indexing like Python)
🔒 Java bounds-checks every array access at runtime and throws ArrayIndexOutOfBoundsException rather than letting you read arbitrary memory (as C would). This is a core safety guarantee — no buffer overruns corrupting memory — at a tiny performance cost the JIT often eliminates when it can prove the index is safe.

Part 4 — Multi-Dimensional Arrays

int[][] grid = new int[3][4];        // 3 rows, 4 columns, all 0
grid[1][2] = 99;                     // row 1, column 2
int rows = grid.length;              // 3
int cols = grid[0].length;           // 4

// Literal form:
int[][] matrix = {
    {1, 2, 3},
    {4, 5, 6}
};

// JAGGED arrays — rows of different lengths (Java's 2D arrays are arrays OF arrays):
int[][] jagged = new int[3][];       // 3 rows, each null for now
jagged[0] = new int[]{1};            // row 0 has 1 element
jagged[1] = new int[]{1, 2, 3};      // row 1 has 3 — perfectly legal

// Iterate a 2D array:
for (int[] row : grid) {             // each element IS an int[] (a row)
    for (int cell : row) { ... }
}

⚠️ Printing an array directly is useless: System.out.println(arr) prints something like [I@1b6d3586 (type + hashcode), because arrays don't override toString(). Use Arrays.toString(arr) (or Arrays.deepToString for nested arrays). Likewise, comparing arrays with == or .equals() compares references — use Arrays.equals for content.

Part 6 — Deep Dive: Shallow vs Deep Copy

// Copying an array of PRIMITIVES is safe — the values are copied:
int[] a = {1, 2, 3};
int[] b = Arrays.copyOf(a, 3);
b[0] = 99;               // a[0] is STILL 1 — independent copies of the int values ✅

// Copying an array of OBJECTS is a SHALLOW copy — the REFERENCES are copied, objects shared:
Point[] src = { new Point(1, 1), new Point(2, 2) };
Point[] dst = Arrays.copyOf(src, 2);
dst[0].x = 99;           // src[0].x is ALSO 99! — both arrays' [0] point to the SAME Point object

// A DEEP copy clones each element too:
Point[] deep = new Point[src.length];
for (int i = 0; i < src.length; i++) deep[i] = new Point(src[i].x, src[i].y);
🔍 Senior nugget: Arrays.copyOf, clone(), and System.arraycopy all make a shallow copy — they duplicate the array's slots but not the objects those slots point to. For a primitive array that's a full independent copy; for an object array, both arrays share the same elements, so mutating an element through one is visible through the other. This shallow/deep distinction (revisited in the Immutability lesson as "defensive copying") is behind a whole category of "why did my supposedly-copied data change?" bugs.

Interview Questions

Q1 (beginner): Why is array access O(1)?

An array is a contiguous block of memory and all elements are the same size, so the address of element i is computed arithmetically as base + i × elementSize — a single calculation, no traversal. That constant-time random access, plus excellent cache locality from the contiguous layout, is arrays' defining performance advantage and the reason they underlie ArrayList, StringBuilder, and hash tables.

Q2 (beginner): What are the default values in a newly created array?

Numeric primitive arrays default to 0 (0.0 for floating point), boolean[] to false, char[] to '', and any reference-type array (String[], Object[], etc.) to null. The null default catches people out: iterating a fresh String[] and calling a method on an element throws NullPointerException until you assign real objects. Unlike local variables (which must be explicitly initialized), array elements always get these defaults.

Q3 (mid): Array vs ArrayList — how do you choose?

An array is fixed-size, can hold primitives directly (no boxing), has the lowest memory/perf overhead, and supports multi-dimensional shapes — use it for fixed-size or performance-critical numeric data. ArrayList is a growable array: it resizes automatically, offers rich methods (add, remove, contains), and integrates with the Collections framework and generics — use it for the common "I don't know the size up front" case, which is most of the time. The key trade-off: ArrayList only holds objects (so primitives are boxed, costing memory and speed in hot numeric loops), whereas an int[] stores raw ints. Default to ArrayList for flexibility; drop to an array for primitives-in-bulk or a fixed known size.

Q4 (mid): Explain shallow vs deep copy for arrays.

Copying an array (Arrays.copyOf, clone, System.arraycopy) produces a new array whose slots are copies of the originals. For a primitive array those slots hold values, so the copy is fully independent — a deep copy in effect. For an object array the slots hold references, and copying duplicates the references but not the pointed-to objects, so both arrays share the same elements — a shallow copy. Mutating an element through one array is visible through the other. A deep copy additionally clones each element (e.g. a loop constructing new objects), giving full independence. Knowing which you have prevents surprise shared mutation.

Q5 (senior): Arrays.asList(myArray) then .add(x) throws UnsupportedOperationException, and modifying an Integer[] vs an int[] passed to it behaves differently. Explain the traps.

Arrays.asList returns a fixed-size list backed by the array — it's a view, not an independent ArrayList. So structural operations (add/remove) throw UnsupportedOperationException, while set works and writes through to the underlying array. To get a mutable copy, wrap it: new ArrayList<>(Arrays.asList(a)). The second trap is varargs generics: Arrays.asList takes T.... Pass an Integer[] and you get a List<Integer> of the elements; but pass an int[] and T infers to int[] itself, yielding a List<int[]> of size 1 (the whole array as one element) — because primitives aren't objects and can't be the T elements. This is why Arrays.asList(intArray).size() surprisingly returns 1. Use Arrays.stream(intArray).boxed().toList() for an int[].

Q6 (senior): Why does Java bounds-check every array access, and what does the JIT do about the cost?

Bounds checking is a safety guarantee: an out-of-range index throws ArrayIndexOutOfBoundsException instead of reading or writing arbitrary memory, which in C is the classic buffer-overflow vulnerability. It's part of what makes the JVM memory-safe. The cost is a comparison per access, which sounds expensive in tight loops — but the JIT performs bounds-check elimination: when it can prove an index is always in range (e.g. a standard for (i = 0; i < a.length; i++) loop), it removes the redundant checks, and it can hoist checks out of loops or use range analysis. So idiomatic loops pay little or nothing, while pathological/unprovable accesses keep the check. This is a good example of the JVM trading a small guaranteed safety cost that the optimizer usually reclaims — and a reason to write loops in the canonical form the JIT recognizes.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Defaults. Create new int[3], new boolean[3], and new String[3] and print each with Arrays.toString. Predict the contents first.
Solution

[0, 0, 0], [false, false, false], [null, null, null]. The null default on the reference array is the one to remember — it's why iterating an unpopulated object array and calling methods NPEs.

2 (easy): Sum and max. Given int[] a = {4, 9, 2, 7};, compute the sum and the maximum with a for-each loop.
Solution

int sum = 0, max = a[0]; for (int n : a) { sum += n; if (n > max) max = n; }. Note initializing max to a[0] (not 0 — that would be wrong for all-negative arrays). For-each is ideal here; no index needed.

3 (medium): Content equality. Create two separate arrays with the same numbers. Compare them with ==, .equals(), and Arrays.equals(). Explain the three results.
Solution

== → false (different objects). .equals() → false too (arrays don't override equals, so it's inherited identity comparison — same as ==). Arrays.equals() → true (element-wise content comparison). Only the last does what you usually want. Same story for toString: use Arrays.toString.

4 (medium): The shallow copy trap. Make an array of two mutable Point objects, copy it with Arrays.copyOf, mutate an element through the copy, and observe the original. Then make it a deep copy so it doesn't happen.
Solution

After the shallow copy, copy[0].x = 99 also changes original[0].x — both slots reference the same Point. Deep copy: loop and construct new Point(src[i].x, src[i].y) for each slot; now mutations are isolated. This is defensive copying, previewed — critical when returning internal arrays from a class.

5 (hard): Rotate an array in place. Rotate {1,2,3,4,5} right by 2 to get {4,5,1,2,3}, using O(1) extra space (the reversal trick). Handle k > length.
Solution

Normalize k %= n (so k > n wraps). Reverse the whole array, then reverse the first k and reverse the rest: reverse([1,2,3,4,5]) → [5,4,3,2,1]; reverse first 2 → [4,5,3,2,1]; reverse last 3 → [4,5,1,2,3]. Three in-place reversals, O(n) time, O(1) space — a classic interview array problem that rewards knowing the trick over a naive shift-one-at-a-time (O(n·k)).

6 (hard): Explain-it test. Out loud: (a) why array indexing is O(1) and what limitation that layout forces; (b) the difference between shallow and deep array copies; (c) why Arrays.toString/Arrays.equals exist instead of the array's own methods.
Self-check rubric

Pass (a) if you said contiguous memory → address arithmetic → O(1), but fixed size (hence ArrayList). Pass (b) if you tied shallow to copying references (objects shared) vs deep cloning each element. Pass (c) if you noted arrays don't override toString/equals, so the defaults show identity/hashcode and you need the Arrays helpers. These feed directly into the Collections lessons.

🎯 Key Takeaways

  • Arrays are contiguous, fixed-size — indexing is O(1), but you can't grow them (hence ArrayList).
  • Defaults: 0/false for primitives, null for reference-type arrays; use .length (a field, no parens).
  • Every access is bounds-checked at runtime (ArrayIndexOutOfBoundsException); valid indices are 0..length-1.
  • Use Arrays.toString/Arrays.equals/Arrays.sort/copyOf — arrays' own toString/equals compare identity, not content.
  • Array copies are shallow — independent for primitives, shared-element for objects; deep-copy elements when you need isolation.
