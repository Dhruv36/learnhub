Interfaces & Abstract Classes Object-Oriented Java

What you'll master in this lesson: what an interface really is (a contract, the purest form of abstraction); how it differs from an abstract class and the exact decision framework for choosing between them; default, static, and private interface methods (Java 8+ powers); functional interfaces as the bridge to lambdas; how the diamond problem is resolved; and why "program to an interface, not an implementation" is the single most valuable design habit in Java. This is where OOP stops being about hierarchies and becomes about contracts.

Part 1 — An Interface Is a Contract

public interface Payable {
    void pay(double amount);         // an abstract method — the CONTRACT (implicitly public abstract)
    // no fields (except constants), no constructors, no state
}

public class Invoice implements Payable {     // "Invoice promises to be Payable"
    @Override public void pay(double amount) { /* charge the customer */ }
}
public class Salary implements Payable {      // an UNRELATED class can also be Payable
    @Override public void pay(double amount) { /* deposit to employee */ }
}

// The power: code depends on the CONTRACT, not the concrete class:
void process(Payable p) { p.pay(100); }       // works for Invoice, Salary, or anything Payable
process(new Invoice());
process(new Salary());

An interface declares what a type can do without saying how. It's a promise: "any class that implements me provides these methods." This is abstraction — the second OOP pillar — in its purest form. Crucially, a class can implement many interfaces (unlike single inheritance), so interfaces model capabilities that cut across unrelated types.

💡 A class can be Comparable, Serializable, and Runnable all at once — three capabilities from three interfaces. It could never extend three classes. This "mix in capabilities" flexibility is a core reason interfaces are preferred over inheritance for most design.

Part 2 — Default & Static Methods (Java 8+)

Before Java 8, interfaces held only abstract methods. Java 8 added default methods (with a body) and static methods, which changed interface design profoundly:

public interface Collection<E> {
    boolean add(E e);                        // abstract — implementers must provide

    default boolean isEmpty() {              // DEFAULT — a body implementers inherit for free
        return size() == 0;                  // can call other interface methods
    }
    static <E> Collection<E> empty() {        // STATIC — a factory/utility on the interface itself
        return new EmptyCollection<>();
    }
    private void helper() { }                // PRIVATE (9+) — shared logic for default methods, hidden
}
🎯 Why default methods exist: they let an interface evolve without breaking every implementer. When Java 8 added stream() and forEach() to the Collection interface, every existing implementation (yours included) got them for free as defaults — no code changes required. Without default methods, adding a method to a widely-implemented interface would break the entire ecosystem. Use them for convenience methods derivable from the core contract, not to smuggle in state (interfaces still can't have instance fields).

Part 3 — Interface vs Abstract Class: The Decision

How many can you have?Implement manyExtend one (single inheritance)
Instance state (fields)No (only public static final constants)Yes — fields with any access
ConstructorsNoneYes (called by subclasses via super)

The decision rule, sharpened:

  • Need shared mutable state or a constructor? → abstract class (interfaces can't hold fields).
  • A capability many unrelated types share? → interface (they can implement it alongside whatever else they are).
  • Both? A common pattern is an interface plus a "skeletal" abstract class that implements it (List + AbstractList): callers depend on the interface, implementers optionally extend the skeleton to avoid boilerplate.
🔍 Modern default: reach for an interface first. It keeps implementers free to extend something else, enables composition, and is what frameworks depend on. Add an abstract class only when genuine shared state/implementation justifies it. "CAN-DO → interface; IS-A-with-shared-guts → abstract class" is the one-line answer, but the instinct should lean interface.

Part 4 — Functional Interfaces & the Lambda Bridge

@FunctionalInterface                     // OPTIONAL annotation — compiler enforces "exactly one abstract method"
public interface Validator<T> {
    boolean isValid(T value);            // the SINGLE abstract method (SAM)
}

// A functional interface can be implemented by a LAMBDA — concise inline behavior:
Validator<String> notBlank = s -> s != null && !s.isBlank();
notBlank.isValid("hi");                  // true

// The JDK's built-in functional interfaces (java.util.function) cover most needs:
//   Predicate<T>   T -> boolean        Function<T,R>  T -> R
//   Consumer<T>    T -> void           Supplier<T>    () -> T

A functional interface has exactly one abstract method, which lets Java treat a lambda or method reference as an instance of it. This is the mechanism behind the entire Streams API and modern Java's functional style — covered fully in the Lambdas lesson. For now, the key insight: interfaces aren't just for classes to implement; a one-method interface is the type of a piece of behavior.

Part 5 — The Diamond Problem & default Method Conflicts

interface A { default String hello() { return "A"; } }
interface B { default String hello() { return "B"; } }

class C implements A, B {
    // ❌ COMPILE ERROR: "class C inherits unrelated defaults for hello() from A and B"
    // You MUST resolve it explicitly:
    @Override public String hello() {
        return A.super.hello();      // choose A's version (or B's, or something new)
    }
}
🔍 Java sidesteps the classic diamond problem. C++'s multiple inheritance of state creates genuine ambiguity (which copy of a field?). Java interfaces have no state, so the only conflict possible is two default methods with the same signature — and the compiler forces you to resolve it explicitly with an override (optionally delegating via A.super.method()). No silent ambiguity, no duplicated state. This is why Java allows "multiple inheritance of behavior" (via default methods) safely while forbidding multiple inheritance of state (only one superclass).

Part 6 — Deep Dive: Program to an Interface, Not an Implementation

This is the most valuable habit in the lesson. Declare variables, parameters, and return types by the most general interface that works — not the concrete class:

// ❌ Tied to a specific implementation:
ArrayList<String> names = new ArrayList<>();
public ArrayList<String> getNames() { ... }
public void process(HashMap<String,Integer> data) { ... }

// ✅ Program to the interface — flexible and honest about what you need:
List<String> names = new ArrayList<>();          // can swap to LinkedList without touching callers
public List<String> getNames() { ... }           // callers depend on List, not ArrayList
public void process(Map<String,Integer> data) { ... }   // accepts any Map

Why it pays off: (1) you can change the implementation (ArrayList → LinkedList, HashMap → TreeMap) without changing a single caller; (2) methods declare the minimum capability they truly need (a List, not specifically an ArrayList), making them more reusable; (3) testing is easier because you can pass a mock or a simpler implementation of the interface.

🔍 Senior nugget: this principle is the foundation of dependency injection and the whole Spring framework — beans depend on interfaces (PaymentGateway), and the container injects whichever implementation is configured, so you swap Stripe for a mock in tests by changing configuration, not code. "Program to interfaces + inject dependencies" is the same idea stated at the class level (composition, Inheritance lesson) and the architecture level (DI, Spring Boot track). When you catch yourself typing a concrete collection type in a signature, ask whether the interface would do — it almost always does.

Interview Questions

Q1 (beginner): What is an interface and how does it differ from a class?

An interface is a contract: a set of method signatures (plus optional default/static methods and constants) that any implementing class promises to provide. Unlike a class it can't be instantiated, can't hold instance state (only public static final constants), and has no constructor. Its purpose is abstraction — specifying what a type can do without how. The two defining advantages over class inheritance: a class can implement many interfaces (mixing capabilities), and interfaces let unrelated types share a common contract so code can depend on the contract rather than a concrete type.

Q2 (beginner): Can an interface have method bodies? Since when?

Yes, since Java 8. Interfaces can have default methods (instance methods with a body that implementers inherit and may override) and static methods (utilities/factories on the interface itself); Java 9 added private methods to share code among default methods without exposing it. They still cannot have instance fields or constructors. Default methods exist chiefly so an interface can gain new methods without breaking existing implementations — how Java 8 added stream()/forEach() to Collection without breaking every collection class in existence.

Q3 (mid): When do you choose an abstract class over an interface?

Choose an abstract class when you need shared instance state (fields), a constructor, or substantial common implementation among a family of closely related "is-a" types — the abstract class provides those and leaves the varying parts abstract. Choose an interface when you're defining a capability that possibly-unrelated types can adopt, since a class can implement many interfaces but extend only one class. Because default methods let interfaces carry behavior now, the deciding factor is mostly state: need shared mutable fields → abstract class; a stateless contract/capability → interface. A frequent best-of-both pattern is an interface plus an optional skeletal abstract implementation (e.g. List + AbstractList).

Q4 (mid): What is a functional interface and why does it matter?

A functional interface has exactly one abstract method (a "SAM" — single abstract method), optionally marked @FunctionalInterface so the compiler enforces the constraint. It matters because Java lets you supply a lambda expression or method reference wherever a functional interface is expected — the lambda is an instance of that interface. This is the foundation of modern functional Java: Runnable, Comparator, Predicate, Function, and the entire Streams API are functional interfaces driven by lambdas. So a one-method interface effectively becomes "the type of a behavior" you can pass around as a value.

Q5 (senior): How does Java handle the diamond problem, and why can it allow "multiple inheritance of behavior" safely?

The classic diamond problem is ambiguity from multiple inheritance of state and implementation — if two parents both define a field or method, which does the child get? Java forbids multiple class inheritance (one superclass only), so state can't be inherited ambiguously. Interfaces contribute no instance state, so the only possible conflict is two inherited default methods with the same signature — and Java refuses to guess: the class fails to compile until you explicitly override the method, where you can delegate to a chosen parent via InterfaceName.super.method(). So Java permits multiple inheritance of behavior (default methods) because behavior conflicts are resolvable by an explicit override with no hidden state to duplicate, while banning multiple inheritance of state which is what makes the diamond genuinely ambiguous. The resolution rule also has a precedence: a concrete method in a superclass beats an interface default, and a more specific interface's default beats a less specific one; only truly unrelated defaults force the manual override.

Q6 (senior): Explain "program to an interface, not an implementation" and connect it to dependency injection.

The principle says declare references, parameters, and return types by the most general interface that satisfies the need (List, Map, PaymentGateway) rather than a concrete class (ArrayList, HashMap, StripeGateway). Benefits: you can swap implementations without touching callers; methods advertise the minimum capability they require, maximizing reuse; and you can substitute test doubles. Dependency injection is this principle operationalized at the object-wiring level: a class declares its collaborators as interface-typed constructor parameters and lets a container (or caller) supply a concrete implementation — so OrderService depends on the PaymentGateway interface, and you inject StripeGateway in production and a mock in tests by changing configuration, not code. Both are the same idea: depend on contracts, not concretions, so the concrete choice becomes a late-bound, replaceable decision. This is the backbone of the Spring framework and testable architecture generally.

🏋️ Exercises (attempt before opening solutions)

1 (easy): A capability across unrelated types. Define a Discountable interface with double applyDiscount(double price). Implement it in two unrelated classes (Membership and Coupon). Write a method that takes any Discountable.
Solution

Both classes implements Discountable with their own logic; a double checkout(double price, Discountable d) calls d.applyDiscount(price) and works for either, plus any future implementer. This is the whole value of interfaces: unrelated types sharing a capability, and code depending on the capability, not the class.

2 (easy): Default method. Add a default method boolean givesDiscount() to Discountable returning applyDiscount(100) < 100. Confirm both implementers get it without writing it.
Solution

The default method has a body in the interface, calling the abstract applyDiscount. Both implementers inherit it for free — you didn't add it to either class. This mirrors how the JDK adds convenience methods to interfaces without breaking implementers. Note it's derivable from the core contract, which is the appropriate use of a default.

3 (medium): Interface vs abstract class decision. You're modeling Shape (needs a shared name field and a common describe()) and Drawable (a capability shapes and also buttons and images can have). Decide which is an interface and which an abstract class, and justify.
Solution

Shape → abstract class: it has shared state (name) and shared implementation (describe()), and its subtypes form one is-a family. Drawable → interface: it's a capability that unrelated types (shapes, buttons, images) share, and a class should be able to be Drawable alongside whatever else it is. A Circle can extend Shape implements Drawable — one base family, one mixed-in capability. This is the canonical split.

4 (medium): Resolve a diamond conflict. Create interfaces Swimmer and Diver each with a default move(). Make a class implement both and resolve the conflict two ways: pick one, then combine both.
Solution

Without an override the class won't compile ("inherits unrelated defaults"). Pick one: @Override public String move() { return Swimmer.super.move(); }. Combine: return Swimmer.super.move() + " then " + Diver.super.move();. The compiler forced an explicit decision — no silent ambiguity — which is exactly how Java keeps multiple inheritance of behavior safe.

5 (hard): Refactor to interfaces. Given a method void report(ArrayList<Order> orders, HashMap<String,Double> totals), widen the parameter types to interfaces and explain what flexibility you gained. Then show a caller passing a LinkedList and a TreeMap.
Solution

Change to void report(List<Order> orders, Map<String,Double> totals). Now a caller can pass new LinkedList<>() and new TreeMap<>() — the method never cared about the concrete type, only that it's a List/Map. You gained: implementation freedom (swap without touching the method), broader reusability, and easier testing (pass a simple stub). This is "program to an interface" applied to a real signature.

6 (hard): Explain-it test. Out loud: (a) interface vs abstract class and the deciding factor; (b) why default methods were added and what they enable; (c) "program to an interface" and its link to dependency injection.
Self-check rubric

Pass (a) if you keyed the decision on state (abstract class) vs capability/multiple-implements (interface). Pass (b) if you said they let interfaces evolve without breaking implementers (and enable functional interfaces/lambdas). Pass (c) if you connected depending-on-contracts to swapping implementations and injecting mocks. These directly set up the Lambdas, Collections, and entire Spring Boot track.

🎯 Key Takeaways

  • An interface is a contract (pure abstraction) — a class implements many, modeling capabilities that cut across unrelated types.
  • Since Java 8, interfaces have default/static (and 9+ private) methods — chiefly to let interfaces evolve without breaking implementers.
  • Deciding factor: need shared state/constructor → abstract class; a stateless capability → interface (lean interface first).
  • A functional interface (one abstract method) is the type of a behavior — implementable by a lambda (Lambdas lesson).
  • Java avoids the diamond problem: no inherited state, and conflicting default methods must be resolved by an explicit override (X.super.method()).
  • Program to an interface, not an implementation — the habit behind swappable code, testing, and dependency injection.
