Exceptions & Error Handling Core Libraries

What you'll master in this lesson: the exception hierarchy and the checked-vs-unchecked distinction unique to Java; how try/catch/finally and try-with-resources work (and why the latter fixed a whole class of leaks); how to design your own exceptions and choose checked vs unchecked; and the error-handling anti-patterns (swallowing, catching Exception, losing the cause) that turn small bugs into unfindable ones. Good error handling is invisible when it works and catastrophic when it's wrong — this lesson is about getting it right.

Part 1 — The Exception Hierarchy

Throwable
├── Error                     ← serious JVM problems you should NOT catch (OutOfMemoryError,
│                               StackOverflowError). Unrecoverable — let them crash.
└── Exception
    ├── (checked exceptions)  ← IOException, SQLException — the compiler FORCES you to handle
    │                           or declare them. "Expected" recoverable conditions.
    └── RuntimeException      ← (UNCHECKED) NullPointerException, IllegalArgumentException,
                                IndexOutOfBoundsException — programming errors; not compiler-enforced.
CategoryCompiler-enforced?RepresentsExamples
ErrorNoUnrecoverable JVM failureOutOfMemoryError, StackOverflowError
Checked ExceptionYes — catch or declareExpected, recoverable conditionsIOException, SQLException
RuntimeException (unchecked)NoProgramming bugsNullPointer, IllegalArgument, IllegalState
💡 Java is the only mainstream language with checked exceptions — ones the compiler forces you to handle or declare with throws. The idea was to make error conditions impossible to ignore. In practice they're controversial (they clutter signatures and tempt developers to swallow them), which is why modern Java code and frameworks lean heavily on unchecked exceptions. Understanding the distinction — and the debate — is a guaranteed interview topic.

Part 2 — try / catch / finally

try {
    int result = riskyOperation();        // code that might throw
    process(result);
} catch (IllegalArgumentException e) {     // catch MORE SPECIFIC types first
    log.warn("bad input", e);              // pass the exception 'e' — preserve context!
} catch (Exception e) {                     // more general — must come after specific ones
    log.error("unexpected", e);
} finally {
    cleanup();                              // ALWAYS runs — even if an exception is thrown or return hit
}

// Multi-catch — handle several types the same way without duplication:
try { ... }
catch (IOException | SQLException e) { log.error("io/db failure", e); }
🎯 Order matters: catch specific before general. A catch (Exception e) before catch (IOException e) won't compile — the specific one is unreachable. finally always runs (for cleanup), even on return or a thrown exception. But note: a return or throw inside finally swallows any exception from the try block — a nasty bug, so never return from finally.

Part 3 — try-with-resources (the leak fix)

// THE OLD WAY — verbose, and easy to leak if close() itself throws or you forget finally:
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("data.txt"));
    return br.readLine();
} finally {
    if (br != null) br.close();      // boilerplate, and close() can throw and mask the real error
}

// TRY-WITH-RESOURCES (Java 7+) — auto-closes anything implementing AutoCloseable:
try (var br = new BufferedReader(new FileReader("data.txt"))) {
    return br.readLine();
}   // br.close() is called AUTOMATICALLY, even on exception — no finally needed

// Multiple resources close in REVERSE order of declaration:
try (var in = new FileInputStream(src);
     var out = new FileOutputStream(dst)) {
    in.transferTo(out);
}   // out closes first, then in
🔒 Always use try-with-resources for anything closeable — files, streams, sockets, JDBC connections/statements/result sets. It guarantees close() is called even on exceptions, closes in reverse order, and correctly handles the case where both the body and close() throw (the body's exception is primary; the close exception is attached as a "suppressed" exception, retrievable via getSuppressed()). This single feature eliminated the resource-leak bugs that plagued pre-Java-7 code — the manual finally-close pattern was error-prone and often silently wrong.

Part 4 — Throwing & Designing Exceptions

// Throw with a clear, actionable message:
if (amount <= 0)
    throw new IllegalArgumentException("amount must be positive, got: " + amount);

// A custom exception — usually extend RuntimeException (unchecked) in modern code:
public class InsufficientFundsException extends RuntimeException {
    private final long shortfall;
    public InsufficientFundsException(long shortfall) {
        super("short by " + shortfall + " cents");   // message
        this.shortfall = shortfall;                  // carry structured data for the handler
    }
    public long getShortfall() { return shortfall; }
}

// WRAPPING — never lose the original cause when translating exceptions:
try { jdbc.query(...); }
catch (SQLException e) {
    throw new DataAccessException("failed to load user " + id, e);   // 'e' = the CAUSE (chained!)
}
🎯 Always chain the cause. When you catch a low-level exception and throw a higher-level one, pass the original as the cause (new MyException(msg, e)). This preserves the full stack trace ("caused by...") so you can trace the real origin. Throwing a new exception without the cause discards the trail and turns debugging into guesswork — one of the most damaging error-handling mistakes.

Part 5 — Checked vs Unchecked: How to Choose

Use…WhenBecause
Unchecked (RuntimeException)Programming errors; caller can't sensibly recoverDon't clutter every signature; fail fast
Checked (Exception)Recoverable conditions the caller must handleCompiler forces acknowledgment

The pragmatic modern guidance: default to unchecked exceptions. Reserve checked exceptions for genuinely recoverable situations where you're confident every caller should be forced to deal with the condition — and even then, many teams and frameworks (Spring's DataAccessException being the canonical example) deliberately wrap checked exceptions in unchecked ones. Validation failures, illegal states, and "this should never happen" conditions are unchecked (IllegalArgumentException, IllegalStateException).

🔍 Why the shift away from checked? Checked exceptions don't compose well with lambdas/streams (functional interfaces don't declare checked exceptions, forcing ugly wrapping), they leak implementation details up through layers, and — worst — they tempt developers to write catch (IOException e) {} just to make the compiler happy, which is worse than not catching at all. The Java community broadly regards mandatory checked exceptions as a well-intentioned experiment that didn't pan out, though opinions vary — a nuanced answer here impresses interviewers.

Part 6 — Deep Dive: The Error-Handling Anti-Patterns

  • Swallowing (empty catch). catch (Exception e) {} is the cardinal sin — it hides failures, so bugs surface far away with no clue where they came from. If you truly can't handle it, at minimum log it with the exception; usually you should rethrow or wrap.
  • Catching Exception (or Throwable) too broadly. This accidentally catches bugs you didn't anticipate (including things like NPE) and, with Throwable, even Errors you should never catch. Catch the specific types you can actually handle.
  • Losing the cause. throw new MyException("failed") without passing the caught exception destroys the stack trace trail (Part 4).
  • Using exceptions for control flow. Throwing/catching to implement normal logic (e.g. loop termination) is slow (stack-trace capture is expensive) and obscures intent — exceptions are for exceptional conditions.
  • Catching and logging and rethrowing. This "log and throw" double-handles, producing duplicate log noise. Handle it once: either log-and-recover here, or wrap-and-throw for a higher layer to log.
🔍 Senior framing: the golden rule is "handle exceptions at the level that can actually do something about them." Low-level code should let exceptions propagate (or wrap them with context); the top of a request/operation is where you translate them into a response, a retry, or a user-facing message — exactly what a Spring @RestControllerAdvice does (Spring Boot track). An exception caught too early, swallowed, or stripped of its cause is a debugging nightmare that turns a five-minute fix into a five-hour hunt. Good error handling is mostly about not interfering with the exception's journey to the place that can handle it.

Interview Questions

Q1 (beginner): What's the difference between checked and unchecked exceptions?

Checked exceptions (subclasses of Exception but not RuntimeException, e.g. IOException) are enforced by the compiler: a method that can throw one must either catch it or declare it with throws, forcing callers to acknowledge it. Unchecked exceptions (RuntimeException and its subclasses, e.g. NullPointerException, IllegalArgumentException) are not compiler-enforced and typically represent programming bugs. Error (e.g. OutOfMemoryError) is also unchecked and represents unrecoverable JVM problems you shouldn't catch. The practical guidance is to use unchecked for programming errors and unrecoverable conditions, and reserve checked for recoverable situations you want to force callers to handle.

Q2 (beginner): What does finally do, and when might it not run?

finally runs after the try block whether or not an exception was thrown and even if the try/catch executes a return — it's for cleanup that must always happen (though try-with-resources is preferred for closing resources). It does not run if the JVM exits during the try (System.exit()), if the thread is killed, or if the JVM crashes/an Error takes it down. A dangerous subtlety: a return or throw inside finally overrides/swallows any exception or return from the try block, so you should never return or throw from finally.

Q3 (mid): What is try-with-resources and what problem did it solve?

Try-with-resources (Java 7) declares one or more resources in the try (...) header; any resource implementing AutoCloseable has its close() called automatically when the block exits, whether normally or via exception, closing multiple resources in reverse order of declaration. It solved the error-prone manual pattern of closing in a finally block, which frequently leaked resources (forgotten closes, null checks, and — subtly — a close() that threw could mask the original exception). Try-with-resources also handles that masking correctly: the body's exception is primary and the close exception is attached as a "suppressed" exception. It's the standard, mandatory-in-practice way to manage files, streams, sockets, and JDBC objects.

Q4 (mid): Why is exception chaining important, and how do you do it?

Exception chaining preserves the original cause when you translate a low-level exception into a higher-level one, so the full origin is retained in the stack trace as a "Caused by:" section. You do it by passing the caught exception as the cause: throw new ServiceException("failed to load", e) (or initCause(e)). Without it — throw new ServiceException("failed to load") — you discard where the failure actually originated (say, a specific SQLException line), leaving only your generic message, which makes production debugging nearly impossible. Chaining lets you raise the abstraction level of exceptions across layers (translating SQLException to a domain DataAccessException) without losing diagnostic information.

Q5 (senior): Make the case for and against checked exceptions.

For: checked exceptions make recoverable failure conditions part of the API contract, so callers can't silently ignore them — the compiler forces a decision, which is valuable for truly recoverable, expected conditions (a file that might not exist, a network call that might fail) where the caller has a meaningful response. Against: in practice they impose real costs. They propagate up through every layer, coupling high-level code to low-level failure types (or forcing wrapping at each boundary); they don't work with lambdas and streams because standard functional interfaces don't declare checked exceptions, forcing awkward try/catch inside lambdas or wrapper utilities; and — most damaging — they pressure developers into writing empty or perfunctory catch blocks just to satisfy the compiler, which is worse than an unchecked exception that would have propagated visibly. The result is that much modern Java and most major frameworks (notably Spring, which translates checked SQLExceptions into an unchecked DataAccessException hierarchy) favor unchecked exceptions. The balanced senior position: default to unchecked; use checked only for conditions that are both genuinely recoverable and that every caller should be compelled to handle, recognizing that the language's checked-exception experiment is widely viewed as not having achieved its goals.

Q6 (senior): At what layer should exceptions be handled, and why is catching too early harmful?

Handle an exception at the level that can actually do something meaningful about it — retry, fall back, translate to a user-facing response, or make a policy decision — and let it propagate everywhere it can't. Low-level code (a repository, a parser) usually shouldn't catch and swallow; it should let the exception rise, optionally wrapping it with context (chaining the cause) as it crosses an abstraction boundary. The natural handling point is the top of a logical operation — a request handler, a job runner, a transaction boundary — where you have enough context to decide the response; in a web app that's a centralized handler like @RestControllerAdvice that maps exceptions to HTTP responses. Catching too early is harmful because the low-level code lacks the context to respond correctly, so it either swallows the error (hiding failures), logs prematurely (creating duplicate noise as it's caught again above), or makes a poor local decision (e.g. returning a default that masks a real problem). It also fragments error policy across the codebase instead of centralizing it. The principle: don't interfere with an exception's journey to the place that can handle it; add context on the way up, and handle once, high up, where the decision belongs.

🏋️ Exercises (attempt before opening solutions)

1 (easy): try-with-resources. Rewrite a manual finally-close file read as a try-with-resources. Explain what it guarantees.
Solution

try (var br = new BufferedReader(new FileReader(path))) { return br.readLine(); }. It guarantees br.close() runs whether the block completes normally or throws, with no null-check or finally boilerplate, and if both the read and the close throw, the read's exception is primary and the close's is suppressed. Cleaner and strictly more correct than the manual pattern.

2 (easy): Specific before general. Write a try/catch that handles NumberFormatException specifically and any other RuntimeException generally. Then swap their order and explain the compile error.
Solution

Correct order: catch (NumberFormatException e) then catch (RuntimeException e). Swapped, the compiler reports that the NumberFormatException catch is unreachable — since NumberFormatException is a subclass of RuntimeException, the general catch would already have handled it. Specific-before-general is a compiler-enforced rule.

3 (medium): Chain the cause. Write a repository method that catches a (simulated) SQLException and rethrows a custom unchecked DataAccessException preserving the cause. Print the stack trace and find the "Caused by".
Solution

catch (SQLException e) { throw new DataAccessException("load failed for " + id, e); } where DataAccessException extends RuntimeException with a (String, Throwable) constructor calling super(msg, cause). The printed trace shows your message then "Caused by: java.sql.SQLException: ..." with the original line — exactly the trail you'd lose if you dropped the cause.

4 (medium): The finally trap. Write a method that returns 1 from the try block but has return 2; in finally. Predict the result and explain why it's a bug.
Solution

It returns 2 — the finally's return overrides the try's. Worse, if the try had thrown an exception, the return 2 would swallow it entirely, silently hiding the failure. This is why you never put return (or throw) in finally: it discards the try block's real outcome, exception and all.

5 (hard): Fix the anti-patterns. Given try { risky(); } catch (Exception e) { }, list everything wrong and rewrite it properly.
Solution

Wrong: (1) catches Exception too broadly — swallows unanticipated bugs including NPE; (2) empty body swallows the failure entirely (no log, no rethrow), so failures vanish silently. Fix: catch the specific type(s) risky() can actually throw and that you can handle; if you can't handle it, don't catch it (let it propagate) or wrap-and-rethrow with context; if you genuinely must catch broadly at a top-level boundary, at least log.error("...", e) and rethrow or translate. Never leave the catch empty.

6 (hard): Explain-it test. Out loud: (a) checked vs unchecked and the modern preference; (b) what try-with-resources guarantees and the suppressed-exception detail; (c) the "handle at the right layer" principle and why swallowing is the cardinal sin.
Self-check rubric

Pass (a) if you tied checked to compiler-enforced/recoverable and noted the shift toward unchecked. Pass (b) if you said auto-close on any exit, reverse order, and body-exception-primary/close-suppressed. Pass (c) if you said handle where you can act, propagate/wrap otherwise, and empty catch hides bugs. These feed directly into the Spring Boot error-handling lesson.

🎯 Key Takeaways

  • Hierarchy: Error (don't catch), checked Exception (compiler-enforced, recoverable), unchecked RuntimeException (bugs).
  • Catch specific before general; finally always runs — never return/throw from it.
  • Use try-with-resources for anything closeable — auto-close on any exit, reverse order, suppressed-exception handling.
  • Always chain the cause (new X(msg, e)) when wrapping; carry structured data in custom exceptions.
  • Default to unchecked exceptions; reserve checked for genuinely recoverable, must-handle conditions.
  • Avoid the anti-patterns: swallowing, catching too broadly, losing the cause, control-flow-by-exception — and handle at the layer that can actually act.
