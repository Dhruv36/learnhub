Exceptions & Optional Intermediate

Java's exception system is explicit: checked exceptions force callers to handle or declare them. The senior skills are the checked-vs-unchecked decision, resource safety with try-with-resources, and using Optional to design away NullPointerExceptions — the "billion-dollar mistake".

try / catch / finally / try-with-resources

try (var conn = dataSource.getConnection();       // try-with-resources: auto-closes
     var stmt = conn.prepareStatement(sql)) {     // (in reverse order, even on exception)
    return stmt.executeQuery();
} catch (SQLException e) {
    throw new DataAccessException("query failed for " + id, e);   // WRAP with context (chain the cause)
}
// try-with-resources works on any AutoCloseable — replaces error-prone finally { conn.close() }.
// Multi-catch: catch (IOException | SQLException e) { ... }
// finally runs always (cleanup) — but try-with-resources is preferred for closing resources.

Custom Exceptions & Good Practice

public class OrderNotFoundException extends RuntimeException {
    public OrderNotFoundException(long id) { super("Order not found: " + id); }
}
// Rules:
// - Throw specific types (IllegalArgumentException for bad args, IllegalStateException for bad state).
// - ALWAYS chain the cause: throw new X("context", originalException) — preserves the stack trace.
// - NEVER swallow: catch (Exception e) {} hides bugs. At minimum log; usually rethrow.
// - Don't use exceptions for control flow (they're expensive and obscure logic).
// - Catch narrowly; let unexpected exceptions bubble to a boundary that logs + reports.

Optional — Designing Away null

public Optional<User> findById(long id) { ... }    // signals "maybe absent" in the TYPE
findById(id)
    .map(User::email)                              // transform if present
    .filter(e -> e.endsWith("@corp.com"))
    .orElseThrow(() -> new UserNotFoundException(id));   // or orElse(default) / orElseGet(supplier)
findById(id).ifPresent(this::notify);
// Optional makes absence EXPLICIT and composable — no more silent nulls.
// Rules: use Optional as a RETURN type; do NOT use it for fields, parameters, or collections
// (use empty collections). Never call .get() without checking — it's the new NPE.
☕ The billion-dollar mistake: null. Optional doesn't remove null from the language, but it moves "might be absent" into the type signature, so callers can't ignore it. Combined with @Nullable annotations and avoiding null returns, it eliminates most NPEs by design rather than by defensive checks everywhere.

Interview Questions

Q1: Checked vs unchecked exceptions — the design intent and the debate.

Checked (extend Exception) force callers to handle or declare — intended for recoverable, expected conditions (IO, SQL). Unchecked (extend RuntimeException) are programming errors / precondition violations the caller usually can't sensibly recover from (NPE, IllegalArgument). The debate: many modern APIs and frameworks (Spring) favor unchecked because checked exceptions cause boilerplate and leaky abstractions (throws clauses propagating everywhere) and are often just rethrown. Pragmatic rule: unchecked for programmer errors; checked sparingly, only when the caller can genuinely act on it.

Q2: How does try-with-resources work and why is it better than finally?

Any resource implementing AutoCloseable declared in the try(...) header is closed automatically when the block exits — normally OR via exception — in reverse declaration order, before catch/ finally run. It also handles "suppressed exceptions" (if close() throws while an exception is already propagating, the original is preserved and the close-exception is attached). Manual finally { r.close(); } is verbose, easy to forget, and can mask the original exception if close() throws. try-with-resources eliminates the whole class of resource-leak bugs.

Q3: How does Optional improve on returning null, and how can it be misused?

It encodes "possibly absent" in the return type, so the compiler-visible contract forces callers to address the empty case (map/orElse/orElseThrow) instead of silently dereferencing null → NPE. It composes functionally. Misuse: using it for fields or parameters (adds allocation/awkwardness — use plain null or overloads there), returning Optional of a collection (return an empty collection instead), and calling .get() without checking (just relocates the NPE to NoSuchElementException). Optional is a return-type tool for "may not exist" single values.

🏋️ Practice: convert a finally-close block to try-with-resources; write a custom exception and throw it with a chained cause; build a findById returning Optional and chain map/filter/orElseThrow; trigger and fix an NPE-by-Optional.get(); use multi-catch for two exception types; prove finally runs even when the try throws.

🎯 Key Takeaways

  • Checked (compiler-enforced, recoverable) vs unchecked (RuntimeException, programmer errors); modern code leans unchecked.
  • try-with-resources auto-closes AutoCloseables safely — always prefer it over finally for cleanup.
  • Chain the cause (new X("context", e)); never swallow; catch narrowly; don't use exceptions for control flow.
  • Optional makes absence explicit in return types — use map/orElseThrow, never bare .get().
  • Optional for return values only; empty collections over Optional<List>; nulls out of your public API.
