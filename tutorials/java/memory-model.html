The Java Memory Model Concurrency

What you'll master in this lesson: the visibility problem — why one thread's write may never be seen by another even without any race on atomicity; what volatile guarantees (and what it doesn't); the happens-before relationship that is the formal heart of the Java Memory Model (JMM); and why "it worked in testing" is worthless for concurrency correctness. This is the most conceptually demanding lesson in the section and a strong senior-interview differentiator — most developers know synchronized but not why it's needed for visibility, not just atomicity.

Part 1 — The Visibility Problem

class Worker {
    private boolean running = true;        // NOT volatile

    public void run() {
        while (running) {                  // may loop FOREVER, even after another thread sets running=false!
            doWork();
        }
    }
    public void stop() { running = false; }   // another thread calls this — but run() might never notice
}

This looks obviously correct: one thread loops while running, another sets it false to stop it. Yet the loop can run forever. There's no atomicity problem here — a boolean write is atomic. The issue is visibility: the JVM and CPU are allowed to cache running in a register or CPU cache, so the looping thread may keep reading a stale true and never observe the other thread's write.

🚨 Visibility is a separate problem from atomicity, and it's more insidious. Without explicit synchronization, there is no guarantee that a write by one thread ever becomes visible to another — the memory model permits each thread to work with its own cached view. This isn't hypothetical: the infinite-loop bug above happens in practice (often only under optimization, or on certain hardware), which is why it's a classic. The fix is to establish a memory-ordering guarantee, via volatile or synchronized.

Part 2 — volatile: Visibility Without Locking

private volatile boolean running = true;   // volatile → writes are IMMEDIATELY visible to all threads

// Now the loop is guaranteed to see running=false and stop.

volatile guarantees two things: (1) visibility — a write is flushed to main memory and reads come from main memory, so every thread sees the latest value; and (2) ordering — reads/writes of the volatile can't be reordered around it (a memory barrier), and everything the writing thread did before the volatile write is visible to a thread that reads it.

⚠️ What volatile does NOT give you: atomicity of compound operations. volatile int count; count++; is still a race — count++ is read-modify-write, and volatile only makes each individual read and write visible, not the whole sequence atomic. So volatile is correct for a simple flag (one thread writes, others read) or a value that's only ever assigned (not incremented), but wrong for counters or any check-then-act/read-modify-write. For those you need synchronized, a lock, or an Atomic class (Atomics lesson).

Part 3 — happens-before: The Formal Guarantee

The JMM defines correctness through the happens-before relationship: if action A happens-before action B, then A's effects (including all memory writes) are guaranteed visible to B. Without a happens-before edge between two threads' actions, the JVM makes no visibility or ordering promise. The rules that establish happens-before:

Happens-before ruleMeaning
Program orderWithin one thread, statements happen-before later statements
Monitor lockUnlocking a monitor happens-before the next lock of it (synchronized)
VolatileA write to a volatile happens-before every subsequent read of it
Thread startthread.start() happens-before the thread's first action
Thread joinA thread's last action happens-before another thread returning from join() on it
🔍 happens-before is the mental model that makes concurrency reasoning rigorous. "Is this write visible to that read?" reduces to "is there a happens-before chain connecting them?" — through a lock, a volatile, a thread start/join, or transitively. synchronized works for visibility precisely because releasing a lock happens-before acquiring it, so everything done under the lock is visible to the next lock holder. This formalism is why the answer to "why did synchronized fix my visibility bug?" isn't hand-waving — it's a defined guarantee.

Part 4 — Safe Publication

// "Publishing" an object = making it visible to other threads. Done UNSAFELY, a thread can see a
// half-constructed object (fields at defaults) even though the constructor "finished":
class Holder { int value; Holder() { value = 42; } }
Holder shared;                              // plain field
// Thread A: shared = new Holder();         ← another thread might see shared != null but value == 0!

// SAFE publication establishes happens-before so the fully-constructed object is visible:
volatile Holder shared;                     // volatile field — safe
// or publish via a synchronized block, a final field, a concurrent collection, or an AtomicReference.
🔍 The double-checked locking pitfall — the historical example of unsafe publication — is why the singleton field must be volatile: without it, a thread could see a non-null but partially-initialized instance. final fields have a special JMM guarantee: once a constructor completes, other threads that see the object through a safe reference are guaranteed to see correctly-initialized final fields. This is part of why immutable objects (all-final fields) are automatically thread-safe: they're safely published and can't change.

Part 5 — Deep Dive: Why Testing Can't Prove Concurrency Correctness

  • Concurrency bugs are nondeterministic. A visibility or race bug depends on timing, thread scheduling, CPU, JIT optimization level, and load — it may never manifest in testing and then appear in production under different conditions. "It passed the tests" says nothing.
  • The memory model permits surprising executions. The JMM allows reorderings and caching that don't happen on your dev machine but do on a different CPU or after the JIT optimizes. Code can be latently broken and run correctly for months.
  • Correctness must be reasoned, not observed. The only way to know concurrent code is correct is to establish that every needed visibility/ordering has a happens-before guarantee — by construction, not by running it. This is the opposite of most testing intuition.
  • So minimize the surface. The practical response: shrink the amount of shared mutable state that needs reasoning. Immutable objects, thread confinement, and battle-tested concurrent utilities reduce the code you must formally reason about to a minimum.
🔍 Senior framing: the JMM is the reason concurrency is genuinely hard — the compiler, JIT, and CPU are all allowed to reorder and cache memory operations for performance, and the memory model is the contract defining which reorderings are visible across threads. A senior engineer internalizes that absence of a happens-before relationship means no guarantee at all, that a bug's non-appearance in testing is meaningless, and therefore that the winning strategy is to design so that very little code needs this reasoning: make state immutable, confine it to single threads, and delegate the shared parts to java.util.concurrent classes written and verified by experts. Understanding volatile vs synchronized vs atomics, and being able to say "this is safe because of the happens-before edge from X to Y," is precisely what separates people who use concurrency primitives from people who reason about them.

Interview Questions

Q1 (beginner): What is the visibility problem?

It's that a write to a shared variable by one thread is not guaranteed to become visible to another thread without proper synchronization. The JVM and CPU may cache the variable in a register or CPU cache, so a reading thread can keep seeing a stale value indefinitely. The classic example is a non-volatile boolean running flag: one thread loops while (running) and another sets it false to stop the loop, but the loop can run forever because it never observes the updated value. Visibility is distinct from atomicity — even a single-atomic-write variable has a visibility problem — and it's fixed by establishing a memory ordering guarantee via volatile or synchronized.

Q2 (beginner): What does volatile guarantee, and what does it not?

It guarantees visibility — a write to the volatile is immediately visible to all threads (writes go to main memory, reads come from it, not a cached copy) — and ordering via a memory barrier, so operations aren't reordered across it and everything the writer did before the volatile write is visible to a reader after the volatile read. It does not provide atomicity for compound operations: volatile int count; count++ is still a race because the read-modify-write sequence isn't atomic — volatile only makes each individual read and write visible. So volatile is right for a simple flag or a value that's only assigned (published), and wrong for counters or check-then-act logic, which need atomics or locks.

Q3 (mid): Explain the happens-before relationship.

Happens-before is the formal ordering the Java Memory Model uses to define visibility: if action A happens-before action B, then A's memory effects are guaranteed visible to B, and A is ordered before B. If no happens-before relationship connects two actions in different threads, the JMM gives no visibility or ordering guarantee between them. Key rules that establish it: program order within a single thread; releasing a monitor happens-before the next acquisition of it; a volatile write happens-before every subsequent read of that volatile; Thread.start() happens-before the started thread's actions; and a thread's final actions happen-before another thread's return from join(). It's transitive, so you can chain edges. Reasoning about concurrent correctness reduces to checking that a happens-before chain connects each write to the reads that must see it.

Q4 (mid): Why does synchronized guarantee visibility, not just mutual exclusion?

Because of the monitor happens-before rule: releasing a lock happens-before any subsequent acquisition of the same lock. So when a thread exits a synchronized block, all the writes it made inside are flushed and guaranteed visible to the next thread that enters a block synchronized on the same monitor. Acquiring the lock acts as a read barrier (see the latest values) and releasing acts as a write barrier (publish your changes). That's why synchronized solves both problems at once: mutual exclusion prevents interleaving (atomicity), and the lock's happens-before edge ensures the changes are visible across threads (visibility). This is also why merely making increment() synchronized but reading the counter without synchronization is subtly wrong — the read needs the happens-before edge too, which is why you synchronize the getter as well.

Q5 (senior): What is safe publication, and how do final fields and immutability relate to it?

Safe publication is making a newly-constructed object visible to other threads in a way that guarantees they see it fully and correctly initialized. The hazard is that without proper publication, another thread can observe a reference to the object as non-null while the object's fields still hold their default values — a "partially constructed object" — because the JMM permits the write of the reference to be reordered before the writes of the fields. An object is safely published if the sharing establishes a happens-before edge: publishing it through a volatile field, an AtomicReference, a field guarded by synchronized, a concurrent collection, or a static initializer. final fields get a special JMM guarantee: after the constructor completes, any thread that obtains the object through a reference published after construction is guaranteed to see the correctly-initialized values of its final fields, without additional synchronization. This is a core reason immutable objects are inherently thread-safe: with all fields final and set in the constructor, they are safely published by that guarantee, can't be observed half-built, and never change afterward, so they need no locks and can be shared freely. It's also why the double-checked-locking singleton requires the instance field to be volatile — to safely publish the fully-constructed instance.

Q6 (senior): Why can't testing establish concurrency correctness, and what follows for how you design concurrent code?

Because concurrency bugs from the memory model — stale reads, missing visibility, unsafe publication, reordering — are nondeterministic and environment-dependent. Whether a broken execution actually occurs depends on thread scheduling, the specific CPU and its memory-ordering behavior, the JIT's optimization decisions (which differ once code is "hot"), and system load, none of which your test run controls. The JMM explicitly permits executions that don't happen on your development machine but are legal and will occur elsewhere — so code can be latently broken yet pass every test and run correctly for a long time before failing in production under different conditions. Passing tests therefore cannot demonstrate the absence of a memory-model bug; it can only show the bug didn't happen this time. What follows is that concurrent correctness must be reasoned — you establish that every required cross-thread visibility/ordering is backed by a happens-before guarantee, by construction — and that the practical design strategy is to minimize the code that needs such reasoning: make state immutable (no shared mutation to reason about), confine mutable state to a single thread, and delegate any genuinely shared mutable state to the expert-written, verified java.util.concurrent classes rather than hand-rolling synchronization. Stress/fuzz tools (jcstress, thread-weaving) and static analysis can raise confidence, but the foundation is design that makes the reasoning tractable, not tests that hope to trip the bug.

🏋️ Exercises (attempt before opening solutions)

1 (easy): The stuck loop. Write the non-volatile running flag with one thread looping and another stopping it (after a sleep). Then add volatile and compare behavior.
Solution

Without volatile, the looping thread may never terminate (it can cache the stale true) — especially with JIT optimization; you may see it hang. With volatile boolean running, the write is visible and the loop stops. This is the visibility problem and its simplest fix demonstrated. (Behavior can vary by JVM/hardware — the point is that the non-volatile version is allowed to hang.)

2 (easy): volatile's limit. Make a counter field volatile and increment it from two threads. Predict whether the total is correct and explain.
Solution

The total is still wrong (less than expected) — volatile makes each read and write visible but does not make count++ (read-modify-write) atomic, so the race persists. Fix with synchronized or AtomicInteger. This proves volatile ≠ atomicity.

3 (medium): Identify happens-before. For a producer thread that sets data then a volatile ready = true, and a consumer that spins until ready then reads data — explain why the consumer safely sees data.
Solution

The producer's write to data is before its volatile write to ready (program order), the volatile write happens-before the consumer's volatile read of ready, and the consumer's read of data is after that (program order). By transitivity, the producer's data write happens-before the consumer's data read, so it's guaranteed visible — even though data itself isn't volatile. This is the classic "publish via a volatile flag" pattern.

4 (medium): Safe publication. Show the unsafe-publication risk of sharing a new Holder() through a plain field, then make it safe. Name three safe-publication mechanisms.
Solution

Through a plain field, another thread might see the reference non-null but value == 0 (reordered). Safe mechanisms: a volatile field, an AtomicReference, publishing inside a synchronized block (or reading/writing through the same lock), a concurrent collection, or a static initializer; and final fields are safely visible after construction. Any of these establishes the happens-before edge that makes the fully-built object visible.

5 (hard): Reason, don't test. Given a piece of concurrent code that passes all your tests, argue why that doesn't prove it's correct, and describe how you'd actually gain confidence.
Solution

Passing tests only shows the broken interleaving/visibility didn't occur under the tested scheduling, CPU, JIT state, and load — all of which differ in production, and the JMM permits executions your machine never produced. To gain real confidence: reason that every cross-thread read has a happens-before edge to the write it must see (via locks/volatile/start/join), minimize shared mutable state (immutability, confinement, concurrent utilities), review against the memory model, and use specialized tools (jcstress for memory-model tests, thread-interleaving/fuzzing, static analyzers). Correctness is established by construction and reasoning, not by observation.

6 (hard): Explain-it test. Out loud: (a) visibility vs atomicity and the stuck-loop bug; (b) what volatile guarantees and its limit; (c) happens-before and why synchronized/immutability give visibility.
Self-check rubric

Pass (a) if you separated "seen at all" (visibility) from "not interleaved" (atomicity) with the flag example. Pass (b) if you said visibility+ordering but not compound atomicity. Pass (c) if you stated the happens-before edges (lock release→acquire, volatile write→read, final-field publication) and tied immutability to safe publication. This is the conceptual peak of the concurrency section.

🎯 Key Takeaways

  • Visibility is separate from atomicity — without synchronization, one thread's write may never be seen by another (the stuck-loop bug).
  • volatile guarantees visibility and ordering for a single variable — but NOT atomicity of compound ops like count++.
  • happens-before is the JMM's formal visibility guarantee — established by locks, volatiles, thread start/join, and program order (transitive).
  • synchronized gives visibility because lock-release happens-before lock-acquire; that's why it fixes visibility, not just races.
  • Safe publication (volatile/final/synchronized/concurrent collections) prevents seeing half-constructed objects; final fields + immutability are safely published by design.
  • Testing can't prove concurrency correctness — reason via happens-before and minimize shared mutable state.
