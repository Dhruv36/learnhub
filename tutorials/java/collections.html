Collections: List, Set, Map Core Libraries

What you'll master in this lesson: the shape of the Collections Framework and the three interfaces you'll use constantly (List, Set, Map); what each guarantees and when to choose it; the core operations and how to iterate safely; and the everyday idioms (factory methods, computeIfAbsent, getOrDefault) that make collection code clean. This lesson is about using collections fluently; the next (Internals & Choosing) dives into how they work and the Big-O trade-offs interviewers probe.

Part 2 — List: Ordered, Indexed, Duplicates Allowed

List<String> names = new ArrayList<>();     // the default List — a growable array
names.add("Ada");                          // append
names.add("Grace");
names.get(0);                              // "Ada" — index access (O(1) for ArrayList)
names.set(1, "Linus");                     // replace at index
names.remove("Ada");                       // remove by value
names.contains("Linus");                   // membership check (O(n) — scans)
names.size();                              // count (a method, unlike array.length)

for (String n : names) { ... }             // for-each iteration
names.forEach(System.out::println);        // method-reference iteration
🎯 ArrayList is your default List — fast index access and iteration. LinkedList exists but is rarely the right call (the next lesson explains why: poor cache behavior usually outweighs its theoretical insert advantage). Reach for ArrayList unless a measured reason says otherwise.

Part 3 — Set: Uniqueness

Set<String> tags = new HashSet<>();         // no duplicates, NO order guarantee
tags.add("java");
tags.add("java");                          // ignored — already present
tags.size();                               // 1

// Choosing a Set implementation is about ordering:
Set<String> hash   = new HashSet<>();       // fastest; unordered
Set<String> linked = new LinkedHashSet<>();  // preserves INSERTION order
Set<String> tree   = new TreeSet<>();        // SORTED order (natural or by Comparator)

// Set relies on equals()/hashCode() to detect duplicates — your value objects MUST override both
// (equality lesson) or "equal" objects won't dedupe.
🔑 A HashSet uses hashCode/equals to decide membership. If you store your own objects without correctly overriding both, duplicates that "should" be equal will coexist — the vanishing-object problem from the equality lesson, in Set form. This is why the equality lesson comes before collections.

Part 4 — Map: Key → Value

Map<String, Integer> scores = new HashMap<>();
scores.put("Ada", 95);                     // add/replace
scores.get("Ada");                         // 95
scores.get("Nobody");                      // null — missing key returns null
scores.getOrDefault("Nobody", 0);          // 0 — safer than get + null check
scores.containsKey("Ada");                 // true
scores.remove("Ada");

// Iterating a Map — over entries (most efficient):
for (Map.Entry<String,Integer> e : scores.entrySet()) {
    System.out.println(e.getKey() + " = " + e.getValue());
}
scores.forEach((k, v) -> System.out.println(k + " = " + v));   // cleaner

Map implementations mirror Set's ordering choices: HashMap (unordered, fastest), LinkedHashMap (insertion order — great for LRU caches), TreeMap (sorted by key). Keys must have correct equals/hashCode, and immutable keys are strongly preferred (equality lesson).

Part 5 — The Idioms That Make Map Code Clean

// getOrDefault — avoid null checks on lookup:
int count = wordCounts.getOrDefault(word, 0);

// merge / compute — update-in-place, perfect for counting:
wordCounts.merge(word, 1, Integer::sum);        // "add 1, starting from absent"

// computeIfAbsent — build a multi-map (Map of Lists) with no null-check boilerplate:
Map<String, List<String>> byCity = new HashMap<>();
byCity.computeIfAbsent("NYC", k -> new ArrayList<>()).add("Ada");
// creates the list on first use, returns the existing one after — one clean line

// putIfAbsent — set only if the key is missing:
config.putIfAbsent("timeout", "30");
🎯 merge, computeIfAbsent, and getOrDefault replace the verbose "check if present, if not create/initialize, then update" pattern that dominates naive Map code. The word-count with merge(word, 1, Integer::sum) and the multi-map with computeIfAbsent(key, k -> new ArrayList<>()) are idioms worth memorizing — they show up in interviews and code review constantly.

Part 6 — Immutable Factory Methods & Safe Iteration

// Concise immutable collections (Java 9+) — great for constants and returns:
List<String> days   = List.of("Mon", "Tue", "Wed");     // immutable, throws on modification
Set<Integer> primes = Set.of(2, 3, 5, 7);
Map<String,Integer> m = Map.of("a", 1, "b", 2);
// List.copyOf(existing) makes an independent immutable snapshot.

// The ConcurrentModificationException trap (control-flow lesson) — removing during for-each:
for (String s : list) {
    if (s.isBlank()) list.remove(s);   // ❌ throws ConcurrentModificationException
}
list.removeIf(String::isBlank);        // ✅ the clean, correct way to remove conditionally
⚠️ Two everyday safety points. (1) List.of/Set.of/Map.of return immutable collections — calling add/put throws UnsupportedOperationException. They're ideal for constants and defensive returns, but wrap in a mutable copy (new ArrayList<>(List.of(...))) if you need to modify. (2) Never structurally modify a collection while iterating it with for-each — use removeIf or the iterator's remove(), or you'll get ConcurrentModificationException (even single-threaded).

Interview Questions

Q1 (beginner): What's the difference between a List, a Set, and a Map?

A List is an ordered sequence accessed by index that allows duplicates (e.g. ArrayList). A Set is an unordered (by default) collection of unique elements — adding a duplicate is a no-op (e.g. HashSet). A Map stores key→value associations with unique keys and is not technically a Collection (it doesn't extend the interface). Choose List when order/duplicates/index matter, Set when you need uniqueness/membership tests, and Map when you look things up by a key.

Q2 (beginner): Why must objects stored in a HashSet or as HashMap keys override equals and hashCode?

Because hash-based collections use hashCode to pick a bucket and equals to confirm a match within it. If your objects don't override them, the defaults are identity-based, so two logically-equal objects are treated as different: a HashSet keeps both (no dedupe) and a HashMap can't find a key you inserted with an equal-but-not-identical key. Correctly overriding both — derived from the same fields, honoring the contract — makes equal objects share a bucket and compare equal, so uniqueness and lookups work. Immutable keys are strongly preferred so the hash can't change after insertion.

Q3 (mid): How do you count word frequencies with a Map cleanly?

Use merge: for each word, counts.merge(word, 1, Integer::sum) — this inserts 1 if the key is absent, otherwise applies the remapping function (sum the existing value with 1). It replaces the classic verbose pattern of "get the current count or 0, add 1, put it back". Alternatives: counts.put(word, counts.getOrDefault(word, 0) + 1), or with streams Collectors.groupingBy(w -> w, Collectors.counting()). merge is the idiomatic imperative answer and a common interview micro-task.

Q4 (mid): Explain computeIfAbsent and a problem it solves elegantly.

computeIfAbsent(key, mappingFunction) returns the value for key if present; otherwise it computes a value with the function, stores it, and returns it. Its killer use is building a "multi-map" — a Map<K, List<V>> — where you'd otherwise write "if the list for this key is null, create and put it, then add". Instead: map.computeIfAbsent(key, k -> new ArrayList<>()).add(value) creates the list only on first use and returns the existing one thereafter, all in one line. It removes the null-check boilerplate and the associated bugs, and (in ConcurrentHashMap) does it atomically.

Q5 (senior): You return an internal List field from a getter and a caller mutates it, corrupting your object. What are the options and their trade-offs?

Returning the live internal list leaks mutability — the caller can add/remove/clear it, breaking your object's invariants (the defensive-copying problem from the Immutability lesson, applied to collections). Options: (1) return a defensive copy (new ArrayList<>(internal)) — fully isolates callers but costs an allocation per call and callers' changes don't reflect back (usually what you want); (2) return an unmodifiable view (Collections.unmodifiableList(internal)) — cheap, callers can't modify it, but it's a live view so your later internal changes show through, and it throws if the caller tries to modify (a runtime surprise); (3) make the field an immutable collection (List.copyOf) so there's nothing to leak; (4) expose behavior instead of the collection (an addItem method, an Iterable, or a stream) so callers never hold the collection at all. The senior default is to not expose mutable internals — prefer returning an immutable copy/view or exposing operations — and to document which guarantee the getter provides.

Q6 (senior): Why can removing an element during for-each throw even in single-threaded code, and what's happening under the hood?

For-each desugars to an Iterator. Most non-concurrent collections are fail-fast: they keep an internal modCount that increments on every structural modification. When the iterator is created it snapshots the expected modCount; on each next()/hasNext() it compares, and if the collection's modCount has changed out from under it — because you called list.remove(x) directly rather than through the iterator — it throws ConcurrentModificationException to prevent undefined behavior (skipped or repeated elements from a shifted backing array). This is single-threaded here: "concurrent" in the exception name refers to modification concurrent with iteration, not multiple threads. The correct fixes are the iterator's own remove() (which updates the expected modCount in step), Collection.removeIf(predicate), collecting removals for a second pass, or iterating a copy. Note it's best-effort/fail-fast, not a guarantee — you shouldn't rely on it firing, you should avoid the pattern.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Pick the collection. For each, name the best collection and why: (a) an ordered playlist with possible repeats; (b) the set of unique visitor IPs; (c) a phone book (name → number).
Solution

(a) List (ArrayList) — order matters, duplicates allowed, index access. (b) Set (HashSet) — uniqueness is the whole point; order irrelevant. (c) Map (HashMap) — lookup a number by name. Matching the interface to the requirement (order? duplicates? key lookup?) is the core skill.

2 (easy): Dedupe with a Set. Given a List<String> with duplicates, produce a list of unique values preserving first-seen order.
Solution

new ArrayList<>(new LinkedHashSet<>(list)) — the LinkedHashSet dedupes while preserving insertion order, then wrap back into a list. (A plain HashSet would lose order.) Or a stream: list.stream().distinct().toList(), which also preserves encounter order.

3 (medium): Word count. Count word frequencies in a List<String> two ways: with merge and with getOrDefault. Confirm they match.
Solution

merge: for (String w : words) counts.merge(w, 1, Integer::sum);. getOrDefault: counts.put(w, counts.getOrDefault(w, 0) + 1);. Both produce the same map. merge is the more idiomatic and slightly more efficient (one map operation vs two). Either is far cleaner than a manual containsKey/get/put dance.

4 (medium): Multi-map with computeIfAbsent. Given a list of people with a city, build a Map<String, List<String>> of city → names, without any null checks.
Solution

for (Person p : people) byCity.computeIfAbsent(p.city(), k -> new ArrayList<>()).add(p.name());. On the first name for a city, computeIfAbsent creates the list; afterward it returns the existing one. One line, no null handling — the canonical multi-map idiom.

5 (hard): Safe conditional removal. From a List<Order>, remove all cancelled orders. Write the version that throws ConcurrentModificationException, then two correct versions.
Solution

Throws: for-each with list.remove(order) inside. Correct #1: list.removeIf(o -> o.isCancelled()); (cleanest). Correct #2: explicit iterator — for (var it = list.iterator(); it.hasNext(); ) if (it.next().isCancelled()) it.remove();. Both avoid the fail-fast exception by not structurally modifying the list outside the iterator.

🎯 Key Takeaways

  • Three core abstractions: List (ordered, indexed, duplicates), Set (unique), Map (key→value); program to the interface.
  • ArrayList and HashMap/HashSet are the defaults; Linked* preserves insertion order, Tree* keeps sorted order.
  • Objects in a Set or as Map keys need correct equals/hashCode (ideally immutable) or dedupe/lookups fail.
  • Use getOrDefault, merge, and computeIfAbsent to replace verbose check-then-update boilerplate.
  • List.of/Set.of/Map.of are immutable; never structurally modify a collection during for-each (use removeIf).
