Synchronization & Locks Concurrency

What you'll master in this lesson: how synchronized provides mutual exclusion to fix the race conditions from the last lesson; what a monitor lock is and the difference between synchronized methods and blocks; the explicit Lock API (ReentrantLock, ReadWriteLock) and when it beats synchronized; and the two great perils of locking — deadlock and contention. This is the classic "make it thread-safe" toolkit, and getting locking right (and knowing when to avoid it) is core to every concurrency interview.

Part 1 — synchronized: Mutual Exclusion

class Counter {
    private int count = 0;
    public synchronized void increment() { count++; }    // only ONE thread at a time in this method
    public synchronized int get() { return count; }
}
// Now the two-thread test from the last lesson prints exactly 200,000 — no lost updates.

The synchronized keyword enforces mutual exclusion: at most one thread can execute a synchronized method/block on a given object at a time. A thread must acquire the object's monitor lock to enter; others block (BLOCKED state) until it releases. This makes the compound count++ operation atomic — no interleaving is possible — so the race is gone.

💡 Every Java object has an intrinsic "monitor" lock. A synchronized instance method locks on this; a synchronized static method locks on the class object. Acquiring the lock also creates a memory barrier that guarantees visibility (Memory Model lesson) — changes made under the lock are visible to the next thread that acquires it. So synchronized solves both problems at once: atomicity (mutual exclusion) and visibility.

Part 2 — Synchronized Methods vs Blocks

// Whole-method synchronization — simple, but locks for the ENTIRE method:
public synchronized void process() { validate(); count++; log(); }   // holds the lock during slow log()!

// Synchronized BLOCK — lock only the critical section, on a chosen lock object:
private final Object lock = new Object();     // a dedicated private lock (best practice)
public void process() {
    validate();                                // not under the lock
    synchronized (lock) { count++; }           // lock ONLY the shared-state mutation
    log();                                      // not under the lock
}
🎯 Prefer narrow synchronized blocks over whole methods, and lock on a private final object. Holding a lock during slow work (I/O, logging) needlessly blocks other threads — keep the critical section as small as possible. And synchronizing on this (or a public object) lets external code lock on the same monitor and accidentally deadlock or interfere; a private final Object lock that only your class knows about is safer. Never synchronize on a String literal or a boxed primitive (they may be shared/interned).

Part 3 — The Explicit Lock API

import java.util.concurrent.locks.*;

private final ReentrantLock lock = new ReentrantLock();
public void process() {
    lock.lock();                     // acquire
    try {
        count++;                     // critical section
    } finally {
        lock.unlock();               // ALWAYS release in finally — or you leak the lock forever
    }
}

// ReentrantLock adds capabilities synchronized lacks:
lock.tryLock();                      // attempt without blocking (returns false if held) — avoid deadlock
lock.tryLock(1, TimeUnit.SECONDS);   // wait up to 1s, then give up
lock.lockInterruptibly();            // acquire but respond to interruption
new ReentrantLock(true);             // FAIR lock — FIFO order (avoids starvation, slower)

🔍 Default to synchronized; reach for ReentrantLock when you need its extra abilities — a timeout (tryLock) to avoid deadlock, interruptible acquisition, fairness, or multiple condition variables. The cost of Lock is that you must release it in a finally (forgetting is a lock leak that hangs every waiter), whereas synchronized releases automatically. For read-heavy data, a ReentrantReadWriteLock lets many readers share access while writers get exclusivity.

Part 4 — Deadlock: The Classic Trap

// Two threads, two locks, acquired in OPPOSITE order → deadlock:
// Thread 1:                          Thread 2:
synchronized (lockA) {                synchronized (lockB) {
    synchronized (lockB) { ... }          synchronized (lockA) { ... }
}                                     }
// T1 holds A waiting for B; T2 holds B waiting for A. Neither releases. FROZEN forever.

A deadlock occurs when threads wait on each other's locks in a cycle, so none can proceed. It needs four conditions (Coffman conditions): mutual exclusion, hold-and-wait, no preemption, and a circular wait. Break any one and deadlock is impossible — the practical lever is breaking the circular wait.

🎯 The #1 deadlock prevention: always acquire multiple locks in a consistent global order. If every thread that needs both lockA and lockB always takes them in the same order (say A then B), no cycle can form. Other tools: tryLock with a timeout (give up and retry rather than block forever), holding locks for the shortest time, and minimizing the number of locks. Deadlocks are often intermittent (timing-dependent) and hard to reproduce — prevention by lock-ordering discipline beats debugging.

Part 5 — wait/notify & Higher-Level Coordination

// The low-level primitive: threads wait for a condition and get notified. MUST hold the lock:
synchronized (lock) {
    while (!conditionMet) {          // ALWAYS wait in a WHILE loop (guard against spurious wakeups)
        lock.wait();                 // releases the lock and waits until notified
    }
    // ...proceed...
}
// Another thread:
synchronized (lock) { conditionMet = true; lock.notifyAll(); }   // wake waiters
⚠️ You rarely write wait/notify directly. It's error-prone (must hold the lock, must loop on the condition to handle spurious wakeups, easy to lose notifications). Modern code uses the higher-level java.util.concurrent tools instead: CountDownLatch (wait for N events), Semaphore (limit concurrent access), CyclicBarrier (threads rendezvous), BlockingQueue (producer/consumer without any manual locking), and CompletableFuture (async composition). These are safer, clearer, and cover almost every real coordination need — reach for them first.

Part 6 — Deep Dive: The Cost of Locks & the Alternatives

  • Contention is the enemy of scalability. When many threads compete for the same lock, they serialize — only one proceeds at a time, and the rest block, so adding cores stops helping. A hot, coarse-grained lock can make a multi-threaded program slower than a single-threaded one.
  • Reduce contention, don't just add locks. Techniques: shrink the critical section (lock less code); split one lock into many (lock striping, as ConcurrentHashMap does); use read-write locks for read-heavy data; and — best — avoid shared mutable state entirely.
  • Lock-free alternatives. Atomic classes (Atomics lesson) use CPU compare-and-swap instructions to update a single variable without locking, avoiding blocking entirely for simple cases like counters. Concurrent collections do this internally.
  • The best lock is no lock. Immutable objects (Immutability lesson) need no synchronization; thread confinement (keep data in one thread) needs none; message-passing designs avoid shared state. Locking is a tool of last resort for genuinely shared mutable state.
🔍 Senior framing: the arc of concurrency expertise is realizing that locks are correct but costly, and that the goal is to minimize shared mutable state so you need as few locks as possible. Overuse of coarse locks produces correct-but-unscalable code; overuse of fine-grained locks produces fast-but-deadlock-prone code. The masters' answer is architectural: prefer immutability, confine state to threads, use concurrent collections and atomics for the shared parts that remain, and apply explicit locks sparingly and with strict ordering discipline. "How would you make this thread-safe?" is really asking "how would you minimize and safely manage shared mutable state?" — and the best answer often removes the need for a lock rather than adding one.

Interview Questions

Q1 (beginner): What does synchronized do?

It provides mutual exclusion: at most one thread at a time can execute a given synchronized method or block on a particular object, because a thread must acquire that object's intrinsic monitor lock to enter and others block until it's released. This makes a compound operation like count++ atomic, eliminating the race conditions caused by interleaving. It also establishes a memory barrier, so writes made while holding the lock are visible to the next thread that acquires it — solving both atomicity and visibility. A synchronized instance method locks on this; a static one locks on the class.

Q2 (beginner): What's a monitor lock and what does it lock on?

Every Java object has an associated intrinsic lock called a monitor. Entering a synchronized block or method acquires that monitor, and only one thread can hold it at a time; other threads attempting to acquire it block (enter the BLOCKED state) until it's released, which happens automatically when the synchronized block/method exits (even via exception). A synchronized instance method uses the monitor of this; a synchronized static method uses the monitor of the Class object; a synchronized block uses the monitor of whatever object you name in synchronized(obj) — which is why choosing a private final lock object gives you control and prevents outside code from locking on the same monitor.

Q3 (mid): When would you use ReentrantLock instead of synchronized?

When you need capabilities synchronized lacks: a non-blocking tryLock() or a timed tryLock(time) to avoid waiting/deadlocking indefinitely; interruptible lock acquisition (lockInterruptibly) so a waiting thread can be cancelled; fairness (FIFO ordering to prevent starvation); the ability to acquire in one method and release in another; or multiple condition variables per lock. The trade-off is that ReentrantLock requires manual release in a finally block (forgetting leaks the lock and hangs waiters), whereas synchronized releases automatically and is simpler and less error-prone. So the guidance is: use synchronized by default and switch to ReentrantLock only when you specifically need its extra features. For read-heavy shared data, a ReentrantReadWriteLock allows concurrent readers with exclusive writers.

Q4 (mid): What is deadlock and how do you prevent it?

Deadlock is a state where two or more threads are each waiting for a lock held by another, forming a cycle, so none can ever proceed. It requires four simultaneous conditions (Coffman): mutual exclusion, hold-and-wait, no preemption, and circular wait. The most practical prevention is to break the circular wait by imposing a global lock-acquisition order — every thread that needs multiple locks always acquires them in the same defined order, so no cycle can form. Additional techniques: use tryLock with a timeout so a thread gives up and retries rather than blocking forever; hold locks for the shortest possible time; reduce the number of locks; and avoid calling foreign/unknown code while holding a lock (it might acquire more). Because deadlocks are often intermittent and timing-dependent, disciplined lock ordering is far more reliable than trying to debug them after the fact.

Q5 (senior): Why can adding locks make a program slower, and how do you address lock contention?

Locks enforce serialization: while one thread holds a lock, every other thread wanting it blocks, so a section guarded by a hot, coarse-grained lock effectively runs single-threaded no matter how many cores you have — and you also pay context-switching and blocking overhead. A heavily contended lock can make a parallel program slower than a sequential one (Amdahl's law: the serialized fraction caps speedup). To address contention: (1) shrink the critical section so the lock is held for as little code/time as possible, and never during slow I/O; (2) split the lock — lock striping partitions the data under many locks so unrelated operations don't contend (how ConcurrentHashMap scales); (3) use a read-write lock so many readers proceed concurrently and only writers are exclusive, for read-heavy data; (4) use lock-free atomics (compare-and-swap) for simple shared variables like counters, avoiding blocking entirely; and, most fundamentally, (5) reduce shared mutable state — immutable objects need no locks, thread confinement needs none, and concurrent collections encapsulate the synchronization efficiently. The senior instinct is that the best scaling strategy is to design so threads rarely need the same lock, not to sprinkle locks and hope.

Q6 (senior): Why prefer java.util.concurrent utilities over raw wait/notify?

Raw wait/notify is a correct but treacherous low-level primitive: you must hold the object's monitor to call them; you must wait inside a while loop re-checking the condition because wakeups can be spurious (a thread can return from wait without being notified) and because between notification and reacquiring the lock the condition may have changed; you must choose notify vs notifyAll carefully (a lost or wrong notification can hang threads permanently); and the whole protocol is easy to get subtly wrong in ways that deadlock or livelock intermittently. The java.util.concurrent utilities — CountDownLatch, Semaphore, CyclicBarrier, BlockingQueue, Phaser, CompletableFuture, and the executor framework — encapsulate these patterns correctly and expressively, so you state intent ("wait for N tasks", "allow at most K concurrent", "hand items between producer and consumer") instead of hand-coding condition signaling. They're written and tested by concurrency experts, handle the edge cases, compose better, and make code far more readable. The rule is to reach for the highest-level concurrency abstraction that fits the problem and drop to wait/notify (or raw locks) only when nothing higher-level suffices — which is rare.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Fix the race. Take the unsynchronized Counter from the Threads lesson and make it correct with synchronized. Verify it now prints exactly 200,000.
Solution

Make increment() and get() synchronized (both — reading needs the memory barrier too). Now the two-thread test reliably prints 200,000 because only one thread at a time executes count++, making it effectively atomic. (An AtomicInteger would be a lock-free alternative — Atomics lesson.)

2 (easy): Method vs block. Rewrite a whole-method synchronized that does validation, a counter update, and logging so that only the counter update is locked. Explain the benefit.
Solution

Move validation and logging outside a synchronized (lock) { count++; } block on a private final lock. Benefit: the lock is held only for the tiny critical section, not during the (potentially slow) validation and logging, so other threads contend for far less time — better throughput. Narrow critical sections are a key contention-reduction technique.

3 (medium): Create a deadlock. Write two threads that acquire two locks in opposite orders and observe the hang. Then fix it by ordering the locks consistently.
Solution

Thread 1 locks A then B; Thread 2 locks B then A — with a small sleep between acquisitions to force the interleaving, the program hangs (each holds one lock, waits for the other). Fix: make both threads acquire A then B (a global order); now no cycle can form and it completes. This is lock-ordering deadlock prevention demonstrated.

4 (medium): tryLock timeout. Use ReentrantLock.tryLock(1, SECONDS) so a thread that can't get the lock in time backs off instead of blocking forever. Note when this is useful.
Solution

if (lock.tryLock(1, TimeUnit.SECONDS)) { try { ... } finally { lock.unlock(); } } else { /* back off / retry */ }. Useful to avoid indefinite blocking and as a deadlock-avoidance strategy: if you can't get all needed locks within a timeout, release what you hold and retry, breaking potential cycles. Note the mandatory finally unlock().

5 (hard): Producer/consumer without manual locks. Implement a producer and consumer coordinating via a BlockingQueue instead of wait/notify. Explain what the queue handles for you.
Solution

A shared BlockingQueue<Item>: the producer calls queue.put(item) (blocks if full), the consumer calls queue.take() (blocks if empty). No synchronized, no wait/notify, no condition loops — the queue handles all the locking, blocking, and signaling internally and correctly. This is why you reach for java.util.concurrent utilities: they encapsulate the error-prone coordination and let you express intent.

6 (hard): Explain-it test. Out loud: (a) how synchronized fixes a race and what a monitor is; (b) the deadlock conditions and the lock-ordering fix; (c) why locks hurt scalability and the alternatives.
Self-check rubric

Pass (a) if you said mutual exclusion via the object's monitor, making compound ops atomic + visible. Pass (b) if you named circular wait and consistent global lock ordering. Pass (c) if you explained serialization/contention and named striping/read-write locks/atomics/immutability. These lead into the Memory Model, Atomics, and Concurrent Collections lessons.

🎯 Key Takeaways

  • synchronized gives mutual exclusion via an object's monitor lock — making compound operations atomic (and visible), fixing races.
  • Prefer narrow synchronized blocks on a private final lock over whole-method synchronization; never lock during slow I/O.
  • ReentrantLock adds tryLock/timeout/interruptible/fairness — but you must unlock() in finally; default to synchronized.
  • Deadlock comes from circular waits — prevent it with a consistent global lock-acquisition order (and tryLock timeouts).
  • Prefer high-level java.util.concurrent utilities (BlockingQueue, latches, semaphores) over raw wait/notify.
  • Locks are correct but costly — contention kills scalability; reduce shared mutable state (immutability, confinement, striping, atomics) rather than just adding locks.
