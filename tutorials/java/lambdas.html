Functional Interfaces & Lambdas Core Libraries

What you'll master in this lesson: what a lambda really is (an implementation of a one-method interface); the built-in functional interfaces (Predicate, Function, Consumer, Supplier) you'll use everywhere; the four kinds of method references and when each applies; variable capture and the "effectively final" rule; and how lambdas differ from anonymous classes (including the this gotcha). Lambdas are the gateway to the Streams API and modern Java — this lesson makes them click.

Part 1 — A Lambda Is a Function You Can Pass Around

// Before lambdas — an anonymous class to supply behavior (verbose ceremony):
Runnable r = new Runnable() {
    public void run() { System.out.println("running"); }
};

// A lambda — the same thing, minus the boilerplate:
Runnable r = () -> System.out.println("running");

// Sorting by a rule, then and now:
list.sort(new Comparator<String>() {
    public int compare(String a, String b) { return a.length() - b.length(); }
});
list.sort((a, b) -> a.length() - b.length());     // lambda — concise, readable

A lambda is a compact way to write an instance of a functional interface — an interface with exactly one abstract method (Interfaces lesson). The compiler matches the lambda's parameters and body to that single method. So a lambda isn't a new kind of thing floating free; it's an object implementing a one-method interface, written inline. This is why lambdas only work where a functional interface type is expected.

💡 Lambda syntax forms: () -> expr (no args), x -> expr (one arg, parens optional), (x, y) -> expr (multiple), and (x, y) -> { statements; return v; } (block body). The parameter types are inferred from the target functional interface, so you rarely write them.

Part 2 — The Built-in Functional Interfaces

You almost never need to declare your own — java.util.function provides the standard ones, and the whole JDK is built around them:

InterfaceAbstract methodShapeUse for
Predicate<T>boolean test(T)T → booleanFilters, conditions
Function<T,R>R apply(T)T → RTransformations/mapping
Consumer<T>void accept(T)T → voidSide effects (print, save)
Supplier<T>T get()() → TLazy/deferred production
BiFunction<T,U,R>R apply(T,U)(T,U) → RTwo-arg transforms
UnaryOperator<T>T apply(T)T → TSame-type transforms
Predicate<String> isLong = s -> s.length() > 5;      isLong.test("hello!");   // true
Function<String,Integer> len = s -> s.length();       len.apply("abc");        // 3
Consumer<String> print = s -> System.out.println(s);  print.accept("hi");
Supplier<Double> rnd = () -> Math.random();           rnd.get();
// Composition: predicates and functions combine —
Predicate<String> longAndUpper = isLong.and(s -> s.equals(s.toUpperCase()));
Function<String,Integer> lenPlus1 = len.andThen(n -> n + 1);
🔍 Primitive-specialized versions (IntPredicate, ToIntFunction, IntSupplier…) avoid boxing in hot code — the JDK provides them because Predicate<Integer> would box every int. Reach for the primitive variants in performance-sensitive numeric pipelines.

Part 3 — Method References (lambdas' shorthand)

When a lambda just calls an existing method, a method reference (::) says it more clearly. There are four kinds:

KindSyntaxEquivalent lambda
Static methodInteger::parseInts -> Integer.parseInt(s)
Instance method of a particular objectSystem.out::printlnx -> System.out.println(x)
Instance method of an arbitrary object of a typeString::toUpperCases -> s.toUpperCase()
ConstructorArrayList::new() -> new ArrayList<>()
list.forEach(System.out::println);                 // vs  x -> System.out.println(x)
names.stream().map(String::toUpperCase);           // vs  s -> s.toUpperCase()
strings.stream().map(Integer::parseInt);           // vs  s -> Integer.parseInt(s)
Supplier<List<String>> factory = ArrayList::new;   // vs  () -> new ArrayList<>()
🎯 Use a method reference when the lambda body is just a call to a named method with the same arguments — it's more readable and signals intent. Keep a lambda when there's any extra logic. The "arbitrary object" form (String::toUpperCase) is the one that confuses people: the stream element becomes the receiver, so String::toUpperCase means "call toUpperCase() on each String".

Part 4 — Variable Capture & "Effectively Final"

int factor = 3;                       // a local variable
Function<Integer,Integer> scale = n -> n * factor;   // lambda CAPTURES factor

// factor = 4;   ❌ COMPILE ERROR — a captured local must be "effectively final"
                 // (never reassigned after initialization)

// The workaround people reach for (and shouldn't, usually):
int[] counter = {0};
list.forEach(x -> counter[0]++);      // captures the ARRAY reference (final), mutates its contents
// This compiles but is a code smell — prefer a stream reduction / collector for accumulation.
🔍 Why the rule? A lambda may outlive the method that created it (stored, passed to another thread, run later). Java captures local variables by value — it copies them into the lambda — so allowing reassignment would create confusing "which value?" semantics and thread races. Requiring captured locals to be effectively final (assigned once) makes the captured value unambiguous. Instance and static fields are not subject to this (they're captured via the this reference and can be mutated) — which is a subtle capture/leak consideration (Enums & Nested lesson).

Part 5 — Lambda vs Anonymous Class (the this gotcha)

LambdaAnonymous class
Can implementOnly a functional interface (1 method)Any interface/class (multiple methods)
this refers toThe enclosing instanceThe anonymous class instance itself
Can have fields/stateNoYes
Compiled toinvokedynamic (no extra .class file)A separate $1.class
public class Widget {
    private String name = "widget";
    void demo() {
        Runnable lambda = () -> System.out.println(this.name);   // 'this' = the Widget ✅
        Runnable anon = new Runnable() {
            public void run() { System.out.println(this); }      // 'this' = the anonymous Runnable!
        };
    }
}
🔍 The this difference matters. Inside a lambda, this is the enclosing object (usually what you want). Inside an anonymous class, this is the anonymous instance — so to reach the outer object you'd write Widget.this. This trips people migrating anonymous classes to lambdas. Use a lambda for a one-method interface; fall back to an anonymous class only when you need multiple methods or instance state.

Part 6 — Deep Dive: Why Lambdas Changed Java

Lambdas (Java 8) weren't just syntactic sugar — they enabled a shift from imperative to declarative code, because behavior became a value you pass:

  • Behavior parameterization. A method can take a Predicate or Comparator and let the caller supply the varying logic — one filter method serves every condition. Before lambdas this required a class per behavior (the Strategy pattern with boilerplate); lambdas make it a one-liner.
  • The Streams API. list.stream().filter(x -> ...).map(...).collect(...) is only readable because each step takes a lambda. Streams (next lesson) are the payoff — lambdas are the enabling mechanism.
  • Deferred execution. A Supplier lets you pass how to compute a value without computing it yet — the basis of lazy evaluation, e.g. logger.debug(() -> expensiveMessage()) only builds the message if debug is enabled.
🔍 Senior framing: lambdas turn the Strategy, Command, and Template Method patterns from multi-class ceremonies into inline expressions. "Pass behavior, not just data" is the mental shift. Under the hood the compiler uses invokedynamic and the LambdaMetafactory to create lambda instances lazily and efficiently — they don't each generate a class file like anonymous classes do, so they're lighter. Understanding that a lambda is still just an object implementing a functional interface keeps the "magic" grounded.

Interview Questions

Q1 (beginner): What is a lambda and what type does it have?

A lambda is a concise inline implementation of a functional interface — an interface with exactly one abstract method. Its type is that functional interface: a lambda can only be assigned where such an interface is expected, and the compiler maps the lambda's parameters and body to the interface's single method. So Runnable r = () -> ...; makes the lambda a Runnable. It's not a free-floating function — it's an object implementing a one-method interface, written without the anonymous-class boilerplate.

Q2 (beginner): Name the four core functional interfaces and their shapes.

Predicate<T> takes a T and returns boolean (test) — for conditions and filters. Function<T,R> takes a T and returns an R (apply) — for transformations. Consumer<T> takes a T and returns nothing (accept) — for side effects like printing or saving. Supplier<T> takes nothing and returns a T (get) — for deferred/lazy production. These four (T→boolean, T→R, T→void, ()→T) plus their bi- and primitive-specialized variants cover almost every functional need, which is why you rarely declare your own functional interface.

Q3 (mid): What are the kinds of method references? Give an example of each.

Four kinds: (1) static method — Integer::parseInt for s -> Integer.parseInt(s); (2) instance method of a specific object — System.out::println for x -> System.out.println(x); (3) instance method of an arbitrary object of a type — String::toUpperCase for s -> s.toUpperCase(), where the argument becomes the receiver; (4) constructor — ArrayList::new for () -> new ArrayList<>(). Use a method reference when the lambda is exactly a call to a named method; it reads better and states intent. The "arbitrary object" form is the subtle one — the first parameter is the object the method is invoked on.

Q4 (mid): What does "effectively final" mean and why does capture require it?

A local variable is "effectively final" if it's never reassigned after initialization — even without the final keyword. A lambda (or anonymous/inner class) may capture local variables, and Java requires captured locals to be effectively final. The reason: a lambda can outlive its enclosing method (stored, deferred, run on another thread), and Java captures locals by value (copying them into the lambda), so permitting later reassignment would make it ambiguous which value the lambda sees and would introduce data races. Requiring effective finality makes the captured value stable and unambiguous. Note this restriction applies only to local variables — instance and static fields are captured through this and can be mutated.

Q5 (senior): How does a lambda differ from an anonymous class beyond syntax?

Several ways. Semantically, this inside a lambda refers to the enclosing instance, whereas in an anonymous class this refers to the anonymous instance itself — so migrating an anonymous class that uses this to a lambda changes meaning. A lambda can only implement a functional interface (one abstract method) and can't declare its own fields or additional methods; an anonymous class can implement any interface/abstract class, hold state, and override multiple methods. Implementation-wise, lambdas compile to an invokedynamic instruction resolved at runtime by LambdaMetafactory — no separate class file is generated, and the JVM can create/cache the instance efficiently (stateless lambdas can be a singleton), whereas each anonymous class produces its own $1.class loaded eagerly. Capture semantics for locals are the same (effectively final), but the lighter footprint and the this difference are the practically important distinctions.

Q6 (senior): Give a case where passing a Supplier<T> instead of a T materially changes behavior.

A Supplier<T> defers computation until get() is called, so it turns eager evaluation into lazy evaluation — which matters whenever producing the value is expensive or conditionally unnecessary. Canonical example: logging. log.debug("state=" + expensive()) always builds the message (concatenation and expensive() run) even when debug logging is disabled, wasting work in production; log.debug("state={}", () -> expensive()) or an API taking a Supplier only invokes expensive() if the level is enabled. Similarly, Optional.orElseGet(supplier) vs orElse(value): orElse evaluates its argument even when the Optional is present, while orElseGet only calls the supplier when the value is absent — so a costly or side-effecting default should be a supplier. Objects.requireNonNullElseGet, Map.computeIfAbsent, and exception factories (orElseThrow(IllegalStateException::new)) follow the same pattern. Passing behavior (a supplier) rather than a precomputed value gives the callee control over whether and when the work happens — the essence of lazy evaluation and a real performance and correctness lever.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Four interfaces. Write a Predicate<Integer> for "even", a Function<Integer,Integer> for "square", a Consumer<Integer> that prints, and a Supplier<Integer> returning 42. Invoke each.
Solution

Predicate<Integer> even = n -> n % 2 == 0; even.test(4);; Function<Integer,Integer> sq = n -> n * n; sq.apply(5);; Consumer<Integer> p = System.out::println; p.accept(7);; Supplier<Integer> forty2 = () -> 42; forty2.get();. Note each uses its interface's method name (test/apply/accept/get) — knowing those by heart speeds up stream code.

2 (easy): Method references. Rewrite these lambdas as method references: s -> s.trim(), x -> System.out.println(x), s -> Integer.parseInt(s), () -> new HashMap<>().
Solution

String::trim (arbitrary-object instance method), System.out::println (specific-object instance method), Integer::parseInt (static), HashMap::new (constructor). Recognizing which of the four kinds each is builds fluency for reading stream pipelines.

3 (medium): Effectively final. Write a loop that builds a list of lambdas each printing the loop variable, then run them. Show why capturing the loop variable directly is a problem and how the for-each variable fixes it.
Solution

A classic for (int i=0;...) can't capture i in a lambda — i is reassigned each iteration, so it isn't effectively final (compile error). A for-each loop (for (String s : list)) creates a fresh effectively-final variable per iteration, so x -> print(s) captures the right value. This is why for-each plays nicely with lambdas and indexed loops don't.

4 (medium): Composition. Compose a Predicate "non-null AND non-blank" and a Function "trim then uppercase" using and/andThen.
Solution

Predicate: Predicate<String> ok = s -> s != null; ok = ok.and(s -> !s.isBlank()); (order matters for null safety — check non-null first). Function: Function<String,String> f = String::trim; f = f.andThen(String::toUpperCase);. Composition lets you build complex behavior from small reusable pieces — the functional-programming payoff.

5 (hard): Lazy default. Demonstrate the difference between Optional.orElse(expensive()) and Optional.orElseGet(() -> expensive()) when the Optional is present, by making expensive() print a message.
Solution

With a present Optional, orElse(expensive()) still calls expensive() (its argument is evaluated before orElse runs) — you see the message even though the default isn't used. orElseGet(() -> expensive()) does not call it, because the supplier is only invoked when the value is absent — no message. This proves why a costly or side-effecting default belongs in a supplier: laziness via passed behavior.

6 (hard): Explain-it test. Out loud: (a) what a lambda is and why it needs a functional interface; (b) the effectively-final capture rule and its reason; (c) how this differs between a lambda and an anonymous class.
Self-check rubric

Pass (a) if you said a lambda implements a one-abstract-method interface inline. Pass (b) if you explained by-value capture + outliving the method → needs a stable value. Pass (c) if you said lambda this = enclosing instance, anonymous this = the anon instance. These set up the Streams lesson, where lambdas are everywhere.

🎯 Key Takeaways

  • A lambda is an inline implementation of a functional interface (one abstract method) — behavior you can pass as a value.
  • Learn the built-ins: Predicate (T→boolean), Function (T→R), Consumer (T→void), Supplier (()→T), and their variants.
  • Method references (::) — static, bound instance, arbitrary-object instance, constructor — are shorthand when a lambda just calls a named method.
  • Captured locals must be effectively final (captured by value, may outlive the method); fields are not restricted.
  • this in a lambda is the enclosing instance (unlike an anonymous class); lambdas need no state and compile lighter (invokedynamic).
  • Lambdas enable behavior parameterization, the Streams API, and lazy evaluation (Supplier) — the shift to declarative Java.
