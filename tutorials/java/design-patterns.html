Design Patterns in Java Professional Java

What you'll master in this lesson: the design patterns that appear constantly in Java code and interviews — Singleton, Factory, Builder, Strategy, Observer, Decorator, Adapter — with idiomatic Java implementations; how modern Java features (enums, lambdas, records) make several patterns simpler than the classic textbook versions; and — most importantly — the judgment to recognize patterns and apply them where they genuinely help rather than forcing them everywhere. Patterns are a shared vocabulary for proven solutions; knowing them well (including when not to use them) is a hallmark of a senior developer.

Part 1 — What Patterns Are (and Aren't)

A design pattern is a named, reusable solution to a recurring design problem — distilled experience, not a library. The value is threefold: proven solutions you don't have to reinvent, a shared vocabulary ("let's use a Strategy here" conveys a whole design in two words), and a lens for recognizing structure in problems. The classic "Gang of Four" patterns fall into three families: creational (how objects are made), structural (how they're composed), and behavioral (how they interact).

⚠️ Patterns are tools, not goals. The anti-pattern is "pattern fever" — forcing a Factory, a Singleton, and an Abstract Visitor into a problem that a plain method would solve. Overusing patterns adds indirection and complexity that obscures simple logic. The skill is recognizing when a problem genuinely matches a pattern (extensibility needed here, object creation is complex there) and reaching for the simplest thing otherwise. Many "patterns" also exist partly to work around older-language limitations that modern Java (lambdas, enums, records) has since removed.

Part 2 — Creational: Singleton, Factory, Builder

// SINGLETON — exactly one instance. The BEST way in Java is an enum (Enums lesson):
public enum Config {
    INSTANCE;
    public String get(String key) { ... }
}
// (Thread-safe, lazy, serialization/reflection-proof — no boilerplate.)

// FACTORY METHOD — create objects without exposing the concrete class to callers:
public interface Shape { double area(); }
public static Shape create(String type) {
    return switch (type) {
        case "circle"    -> new Circle();
        case "rectangle" -> new Rectangle();
        default -> throw new IllegalArgumentException(type);
    };
}
// Callers depend on Shape (the interface), not the concrete classes → program to an interface.

// BUILDER — construct a complex object step by step (great for many optional parameters):
Pizza p = new Pizza.Builder()
    .size("large")
    .addTopping("mushroom")
    .addTopping("olive")
    .build();
// Avoids the "telescoping constructor" anti-pattern (many overloaded constructors); readable,
// and the built object can be immutable (Immutability lesson).
🎯 The Builder pattern is the everyday hero here. When a class has more than a few parameters — especially optional ones — constructors become unreadable (new Pizza("large", true, false, null, 2) — what do those mean?). A builder gives named, optional, order-independent construction and produces an immutable result. It's one of the most useful patterns in real Java (and libraries like Lombok generate it for you). Singleton via enum and Factory for decoupling creation from concrete types round out the creational trio you'll use most.

Part 3 — Behavioral: Strategy & Observer (now lambda-powered)

// STRATEGY — encapsulate interchangeable algorithms; swap behavior at runtime.
// CLASSIC (pre-lambda): an interface + a class per strategy. MODERN: just a lambda!
public interface DiscountStrategy { double apply(double price); }

double checkout(double price, DiscountStrategy strategy) { return strategy.apply(price); }
checkout(100, p -> p * 0.9);          // 10% off — the strategy IS a lambda
checkout(100, p -> p - 5);            // $5 off — different behavior, no new class

// OBSERVER — notify dependents when state changes (pub/sub).
public interface Listener { void onEvent(Event e); }
private final List<Listener> listeners = new CopyOnWriteArrayList<>();   // thread-safe (Concurrency lesson)
public void fire(Event e) { listeners.forEach(l -> l.onEvent(e)); }
button.addListener(e -> System.out.println("clicked"));   // register a lambda observer
🔍 Lambdas turned several "patterns" into one-liners. Strategy, Command, and Template Method were originally multi-class ceremonies because Java couldn't pass behavior as a value; now a functional interface + a lambda (Lambdas lesson) expresses them directly. So "which design pattern is this?" sometimes has the answer "it used to be Strategy, but now it's just a lambda." Recognizing that modern language features subsume classic patterns is itself a senior insight — you apply the idea (interchangeable behavior, decoupled notification) with far less boilerplate.

Part 4 — Structural: Decorator & Adapter

// DECORATOR — add behavior by WRAPPING, not subclassing (composition over inheritance!).
// Java's own I/O is built on it:
Reader r = new BufferedReader(new InputStreamReader(new FileInputStream(file)));
//         └─ adds buffering └─ adds char decoding    └─ raw bytes
// Each wrapper adds a responsibility; you compose exactly the capabilities you need.

// ADAPTER — make an incompatible interface usable, by wrapping and translating:
public class LegacyPaymentAdapter implements PaymentGateway {   // the interface your code wants
    private final OldPaymentSystem legacy;                       // the incompatible thing you have
    public void pay(Money amount) {
        legacy.doPayment(amount.getCents(), amount.getCurrency().name());   // translate the call
    }
}
🔍 Decorator is "favor composition over inheritance" (Inheritance lesson) made into a pattern. Instead of a subclass explosion (BufferedFileReader, GzipBufferedFileReader…), you compose wrappers — the exact InstrumentedList idea from the Inheritance lesson. Adapter lets new code talk to old/third-party code by translating between interfaces (program to your interface, adapt the foreign one behind it). Both are everyday structural tools, and both are visible in the JDK itself, which makes them easy to recognize.

Part 6 — Deep Dive: Judgment — When to Use a Pattern

  • Recognize the problem first, then name the pattern. Don't start with "I'll use a Factory" — start with "callers shouldn't depend on which concrete class is created," which is a Factory. The problem justifies the pattern, not the other way around.
  • Prefer the simplest thing that works. A single implementation needs no Strategy interface; one product needs no Factory; a class with three parameters needs no Builder. Add the pattern when the problem it solves (multiple algorithms, complex creation, many optional params) actually exists — YAGNI.
  • Use modern features that subsume patterns. Lambdas for Strategy/Command, enums for Singleton, records for immutable value objects, sealed types + pattern matching instead of Visitor. The classic multi-class implementations are often unnecessary now.
  • Patterns are for communication as much as structure. Even when a pattern is simple, naming it (a *Factory, a *Strategy) documents intent for the next reader. But a misapplied pattern name misleads — so name accurately.
🔍 Senior framing: the mark of maturity with patterns is not knowing more of them but exercising judgment about them — recognizing the underlying problem, reaching for a pattern only when it earns its complexity, and knowing which classic patterns modern Java has made obsolete or trivial. Junior developers either don't know patterns (and reinvent them badly) or, having just learned them, over-apply them (pattern fever), wrapping simple code in factories and singletons that add indirection without benefit. The senior sits between: fluent in the vocabulary, quick to recognize when a Strategy/Decorator/Builder genuinely fits, and equally quick to say "this is just a method" or "a lambda does it." Patterns are a means to the real goals — code that's decoupled, extensible where it needs to be, and readable — not an end. Interviewers ask about patterns to test exactly this: whether you can connect a real design problem to a proven solution and justify the trade-off, not whether you can recite the Gang of Four. The best answer to "what pattern would you use?" often includes "or, more simply, none — here's why."

Interview Questions

Q1 (beginner): What is a design pattern and why use one?

A design pattern is a named, reusable solution to a recurring software-design problem — distilled best practice, not code you import. Using one gives three benefits: you apply a proven solution instead of reinventing (and re-botching) it; you gain a shared vocabulary, so saying "use a Strategy here" communicates a whole design concisely to teammates; and you get a lens for recognizing structure in problems. The classic Gang of Four patterns are grouped into creational (object creation), structural (object composition), and behavioral (object interaction). The key caveat is that patterns are tools to apply when a problem genuinely matches, not goals to force everywhere.

Q2 (beginner): What's the best way to implement a Singleton in Java, and why?

A single-element enum: enum Singleton { INSTANCE; ... }. It's the best because the JVM guarantees exactly one instance, created lazily and thread-safely with no locking code, and enum instances are immune to the two ways hand-rolled singletons break — serialization (which would otherwise create a second instance) and reflection (which can invoke a private constructor). The classic static-field-plus-private-constructor approach works but needs careful extra code (double-checked locking with a volatile field for lazy init, readResolve for serialization) to match those guarantees. The enum gives all of it in a few lines. (In frameworks like Spring, you usually don't hand-write singletons at all — beans are singletons managed by the container.)

Q3 (mid): When would you use the Builder pattern?

When constructing an object involves many parameters — especially several optional ones — so that constructors become unreadable or explode into many overloads (the "telescoping constructor" anti-pattern). A builder provides a fluent, step-by-step API with named methods (.size("large").addTopping("olive").build()), making the construction readable and order-independent, allowing optional parameters to be omitted cleanly, enabling validation in build(), and producing an immutable final object. It's ideal for configuration objects, DTOs with many fields, and any value object with more than a handful of constructor arguments. Libraries like Lombok can generate builders automatically, and records cover the simpler all-required-fields case. If a class has only two or three required parameters, a plain constructor is simpler and a builder is overkill.

Q4 (mid): How have lambdas changed the Strategy and Observer patterns?

They collapsed the boilerplate. Classically, Strategy required an interface plus a separate class per algorithm, and Observer required implementing a listener interface in a named class — because Java couldn't pass behavior as a value. With lambdas and functional interfaces, a strategy is just a lambda you pass to a method (checkout(price, p -> p * 0.9)), and an observer is a lambda you register (button.addListener(e -> ...)), with no extra classes. The ideas — interchangeable algorithms selected at runtime, and decoupled notification of dependents — remain valuable, but the implementation is a one-liner. This is a broader theme: modern Java features (lambdas for Strategy/Command/Template Method, enums for Singleton, records for value objects, sealed types + pattern matching for Visitor) subsume several classic patterns, so recognizing when a pattern has become a language feature is part of applying them well.

Q5 (senior): Explain the Decorator pattern and why it embodies "composition over inheritance."

Decorator adds responsibilities to an object by wrapping it in another object that implements the same interface, delegates to the wrapped instance, and adds behavior before/after — so you can stack capabilities dynamically. Java's java.io is the canonical example: new BufferedReader(new InputStreamReader(new FileInputStream(file))) composes raw byte reading, character decoding, and buffering, each a separate decorator you include only if needed. It embodies composition over inheritance because the alternative — expressing every combination of features via subclassing — causes a combinatorial explosion of classes (BufferedFileReader, GzipBufferedFileReader, EncryptedGzipBufferedFileReader…) and the fragile-base-class coupling that the Inheritance lesson warns about. Decorator instead assembles behavior from small, single-responsibility wrappers at runtime, so you get exactly the capabilities you want in any combination, each wrapper is independently testable and reusable, and adding a new capability is a new decorator rather than an edit to a hierarchy. It's the same principle as the InstrumentedList wrapper from the Inheritance lesson, formalized as a reusable structural pattern — flexible, open for extension, and free of the coupling deep inheritance imposes.

Q6 (senior): How do you decide whether to apply a design pattern versus keeping it simple?

By starting from the problem, not the pattern, and applying the pattern only when the specific problem it solves actually exists — and choosing the simplest thing that works otherwise. Concretely: identify the real design pressure (Do callers need to be decoupled from which concrete class is created? Are there multiple interchangeable algorithms? Is object construction genuinely complex with many optional parameters? Do dependents need to react to state changes?). If such a pressure is present, the matching pattern (Factory, Strategy, Builder, Observer) earns its indirection by making the code more decoupled, extensible, or readable. If it isn't — a single implementation, one product, three constructor arguments — the pattern adds indirection and cognitive overhead for no benefit, which is "pattern fever," a classic over-engineering smell. Two further judgments sharpen this: first, apply YAGNI — don't add a Strategy interface for a hypothetical future second algorithm that may never come; add it when the second algorithm arrives. Second, prefer modern language features that subsume patterns (a lambda instead of a Strategy class, an enum instead of a hand-rolled Singleton, a record instead of a value-object builder, sealed types + pattern matching instead of Visitor), since the classic multi-class implementations often exist only to work around older-Java limitations. Patterns are means to the real ends — decoupling, extensibility where needed, and readability — so the mature decision is to reach for one when it demonstrably serves those ends and to say "this is just a method" or "a lambda does it" when it doesn't. Communicating that trade-off, rather than reflexively pattern-matching every problem to a Gang-of-Four name, is what distinguishes senior judgment.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Enum singleton. Implement a Logger singleton as an enum with a log(String) method, and access it from two places to confirm one instance.
Solution

enum Logger { INSTANCE; public void log(String m){...} }; both Logger.INSTANCE references are identical. It's thread-safe, lazy, and immune to serialization/reflection attacks — all with no boilerplate versus the classic static-field singleton. This is the idiomatic Java singleton.

2 (easy): Strategy as a lambda. Write a sort that takes a Comparator strategy and call it with two different lambdas (by length, then reverse alphabetical).
Solution

list.sort(Comparator.comparingInt(String::length)); and list.sort(Comparator.reverseOrder());. Comparator is the Strategy pattern — an interchangeable algorithm passed in — and a lambda/method-reference expresses each strategy without any extra class. Classic Strategy made trivial.

3 (medium): Builder. Write an immutable HttpRequest with a fluent builder supporting required url and optional method/headers/body. Show why a builder beats overloaded constructors here.
Solution

A static nested Builder with chained setters returning this and a build() producing an immutable HttpRequest. It beats constructors because with several optional parameters, constructor overloads explode combinatorially and call sites like new HttpRequest(url, null, headers, null) are unreadable — the builder names each optional field and lets you omit the ones you don't need. build() can also validate (e.g. url non-null).

4 (medium): Decorator. Given a DataSource interface with a read(), write decorators that add caching and logging by wrapping, and compose them. Explain the advantage over subclassing.
Solution

Each decorator implements DataSource, holds a wrapped DataSource, and adds its behavior around delegate.read(): new CachingSource(new LoggingSource(new FileSource())). Advantage over subclassing: you compose exactly the capabilities you want in any order without a class per combination (CachingLoggingFileSource…), each decorator is single-responsibility and reusable, and adding a new capability is a new wrapper — composition over inheritance, just like java.io.

5 (hard): Recognize and simplify. You inherit code with a StrategyFactoryProvider producing single-method strategy classes for a feature that has one implementation. Critique it and simplify.
Solution

This is pattern fever: a Factory producing a Provider producing Strategy classes for a feature with a single implementation adds layers of indirection with zero benefit — no interchangeable algorithms exist, so there's nothing to abstract. Simplify to a plain method (or, if a second behavior is truly imminent, a single functional-interface parameter taking a lambda). Add the Strategy abstraction only when a real second implementation appears (YAGNI). The critique demonstrates the senior judgment that patterns must earn their complexity.

6 (hard): Explain-it test. Out loud: (a) the three pattern families and one example each; (b) how lambdas/enums/records subsume classic patterns; (c) the judgment for when to apply a pattern vs keep it simple.
Self-check rubric

Pass (a) if you named creational (Factory/Builder/Singleton), structural (Decorator/Adapter), behavioral (Strategy/Observer). Pass (b) if you cited lambda→Strategy, enum→Singleton, record→value object, sealed+pattern-matching→Visitor. Pass (c) if you said start from the problem, apply YAGNI, prefer the simplest thing, and use a pattern only when its specific pressure exists. This sets up SOLID and the whole Spring Boot track (DI, Proxy, Template).

🎯 Key Takeaways

  • Patterns are named, reusable solutions + a shared vocabulary — creational, structural, behavioral — applied when a problem matches, not forced.
  • Creational: enum Singleton (the best), Factory (decouple creation), Builder (complex/optional params → immutable object).
  • Lambdas subsume Strategy/Command/Observer; enums subsume Singleton; records subsume value-object builders; sealed+pattern-matching subsume Visitor.
  • Structural: Decorator (wrap to add behavior — composition over inheritance, e.g. java.io), Adapter (translate an incompatible interface).
  • Patterns are everywhere in the JDK and frameworks (Comparator=Strategy, BufferedReader=Decorator, Spring beans=Singleton) — recognizing them makes APIs click.
  • Judgment matters most: recognize the real problem, apply YAGNI, prefer the simplest thing, and know when "it's just a method/lambda."
