Concurrency Patterns & Pitfalls Concurrency

What you'll master in this lesson — the concurrency capstone: the failure modes that make multithreaded code hard — deadlock, livelock, and starvation — and the concrete rules that prevent them; the two disciplined ways to write correct concurrent code (confinement/immutability vs guarded shared state); the classic producer-consumer pattern with a BlockingQueue; and the senior mindset that ties the whole section together — share less, and when you must share, share safely. Prerequisites: the whole Concurrency section — Threads, Synchronization, the Memory Model, Executors, and Concurrent Collections.

Part 1 — Deadlock & the Lock-Ordering Rule

// DEADLOCK: two threads each hold a lock the other needs, and both wait forever.
void transfer(Account a, Account b, BigDecimal amt) {
    synchronized (a) {                 // Thread 1: transfer(x, y) locks x, waits for y
        synchronized (b) {             // Thread 2: transfer(y, x) locks y, waits for x  → DEADLOCK
            a.debit(amt); b.credit(amt);
        }
    }
}

// FIX: always acquire locks in a CONSISTENT GLOBAL ORDER (e.g. by account id):
void transfer(Account a, Account b, BigDecimal amt) {
    Account first  = a.id() < b.id() ? a : b;      // lower id ALWAYS locked first
    Account second = a.id() < b.id() ? b : a;
    synchronized (first) { synchronized (second) { a.debit(amt); b.credit(amt); } }
}
🎯 Deadlock happens when threads acquire multiple locks in different orders and form a cycle of "I hold this, I need that." The classic case is two accounts transferred in opposite directions concurrently: thread 1 locks A then wants B, thread 2 locks B then wants A — neither can proceed. The standard fix is a global lock ordering: define a total order on locks (by id, hash, or any stable key) and always acquire them in that order, which makes a cycle impossible. Other defenses: tryLock with a timeout (back off and retry instead of blocking forever), holding one lock at a time, or avoiding shared locks entirely (Part 3). Being able to explain the four Coffman conditions and that breaking any one prevents deadlock — lock ordering breaks "circular wait" — is a staple senior question.

Part 2 — Livelock, Starvation & Race Conditions

// The RACE CONDITION you must recognize: check-then-act is not atomic.
if (!map.containsKey(k)) {      // Thread A checks: absent
    map.put(k, compute());      // Thread B ran the same check+put in between → both put → lost update
}
// FIX: an atomic operation instead of check-then-act:
map.computeIfAbsent(k, key -> compute());   // atomic on a ConcurrentHashMap (Concurrent Collections lesson)
🔍 Most concurrency bugs are race conditions on shared mutable state, and the most common shape is "check-then-act." Testing a condition and then acting on it is two steps, and another thread can change the state in between — the check is stale by the time you act, producing lost updates, duplicate work, or corrupted invariants. The fix is to make the check-and-act a single atomic operation: an atomic method like computeIfAbsent/putIfAbsent, a compare-and-swap (Atomic*, Concurrent Collections lesson), or a lock held across both steps. Deadlock/livelock/starvation are the liveness failures (threads can't make progress); races are the safety failures (wrong results) — a correct concurrent program must avoid both.

Part 3 — Strategy 1: Don't Share (Confinement & Immutability)

// The easiest thread-safe code has NO shared mutable state. Two ways to get there:

// (a) IMMUTABILITY — an immutable object is inherently thread-safe (no writes to race on):
record Money(BigDecimal amount, Currency currency) {}   // (Immutability lesson) — freely shared

// (b) CONFINEMENT — keep mutable state on ONE thread, so no sharing occurs:
ThreadLocal<SimpleDateFormat> fmt =                     // SimpleDateFormat is NOT thread-safe...
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));   // ...so give each thread its own
// Local variables are confined by definition — they live on one thread's stack.
🎯 The best way to make code thread-safe is to eliminate shared mutable state entirely. Two disciplines achieve it. Immutability (Immutability lesson): an object that never changes after construction has nothing to race on, so immutable objects and records are inherently thread-safe and can be shared freely — this is why functional-style, immutable data is a concurrency superpower. Confinement: keep mutable state reachable from only one thread — local variables are confined automatically (each thread has its own stack), and ThreadLocal gives each thread a private copy of something that isn't thread-safe (the classic use is SimpleDateFormat). "Share less" beats "share carefully" because it removes the possibility of a race rather than managing it.

Part 4 — Strategy 2: Share Safely (Guard All Access)

// When state MUST be shared and mutable, guard EVERY access with the SAME lock, and keep
// the critical section small:
class Counter {
    private long count;                              // guarded by 'this'
    synchronized void inc()      { count++; }        // read-modify-write must be atomic
    synchronized long get()      { return count; }   // READS need the lock too — for visibility (Memory Model)
}

// Prefer higher-level tools over raw synchronized where they fit:
private final AtomicLong count = new AtomicLong();   // lock-free counter (Concurrent Collections)
count.incrementAndGet();
// ...and prefer a concurrent collection over synchronizing a plain one:
private final Map<K,V> m = new ConcurrentHashMap<>();
🔒 If you share mutable state, every access — writes AND reads — must be guarded by the same lock. A half-locked class (locking writes but not reads) still has a visibility bug: an unlocked read may see a stale value because, without the memory barrier the lock provides, there's no happens-before guarantee (Memory Model lesson). Keep critical sections small (hold the lock only for the shared-state manipulation, never across slow I/O) to reduce contention, and document which lock guards which state (@GuardedBy). Where a higher-level construct fits — an Atomic* for a counter, a ConcurrentHashMap for a map — prefer it: they're less error-prone and often faster than manual synchronized.

Part 5 — The Producer-Consumer Pattern

// A BlockingQueue decouples producers from consumers and handles all the waiting/signaling for you:
BlockingQueue<Task> queue = new LinkedBlockingQueue<>(1000);   // BOUNDED — back-pressure, not OOM

// Producer thread(s):
queue.put(task);          // BLOCKS if the queue is full → natural back-pressure on fast producers

// Consumer thread(s):
Task t = queue.take();    // BLOCKS if the queue is empty → no busy-waiting
process(t);

// In practice you rarely hand-roll this — a ThreadPoolExecutor IS a producer-consumer:
// submit() = produce onto its work queue; worker threads = consumers (Executors lesson).
🎯 Producer-consumer is the workhorse concurrency pattern, and a BlockingQueue implements it correctly for you. Producers put work, consumers take it, and the queue handles all the blocking and signaling — no manual wait()/notify() (which is notoriously error-prone). Crucially, use a bounded queue: when it fills, put blocks, applying back-pressure that slows fast producers to the rate consumers can handle — an unbounded queue instead grows until it exhausts memory under overload. You usually get this pattern for free from a ThreadPoolExecutor, which is literally producers (submit) feeding a work queue drained by consumer worker threads — recognizing that connects this pattern to the Executors lesson.

Part 6 — Deep Dive: A Discipline, Not a Bag of Tricks

  • Correctness has two halves: safety and liveness. Safety = never a wrong result (no races); liveness = always makes progress (no deadlock/livelock/starvation). You must design for both — a program that's race-free but deadlocks is as broken as one that never blocks but corrupts data.
  • Prefer "share less" to "share carefully." Immutability and confinement remove the possibility of a race; locks merely manage it. Reach for shared mutable state and locks only when the problem genuinely requires shared mutation.
  • Use the highest-level tool that fits. A BlockingQueue, an ExecutorService, a ConcurrentHashMap, an Atomic*, or a CompletableFuture encodes correct concurrency you'd otherwise have to get right by hand. Raw synchronized/wait/notify is the assembly language of concurrency — a last resort.
  • Concurrency bugs are non-deterministic — design them out, don't test them out. A race may pass a thousand test runs and fail in production. Reason about happens-before, keep critical sections small, and lean on proven constructs rather than hoping tests catch timing bugs.
🔍 Senior framing: concurrency mastery is a discipline, not a collection of API tricks, and this capstone ties the section into one coherent mental model: a correct concurrent program must satisfy both safety (no race ever produces a wrong result) and liveness (threads always make progress — no deadlock, livelock, or starvation), and the two failure families need different defenses. Races come from unsynchronized shared mutable state, most often in the check-then-act shape, and are fixed by making the operation atomic (a compound atomic method, a compare-and-swap, or a lock held across the whole check-and-act). Deadlock comes from acquiring multiple locks in inconsistent orders forming a circular wait, and the canonical fix is a global lock ordering (breaking the circular-wait Coffman condition), with tryLock-timeouts and single-lock designs as alternatives. The overarching strategy, though, is to need fewer locks in the first place: immutability makes objects inherently thread-safe (nothing to race on) and confinement (locals, ThreadLocal) keeps mutable state on a single thread — "share less" eliminates races where "share carefully" only manages them. When you genuinely must share mutable state, the rule is absolute: guard every access, reads included (an unguarded read has a visibility bug per the memory model), with the same lock, keep critical sections small to limit contention, and prefer higher-level constructs — Atomic*, ConcurrentHashMap, BlockingQueue, ExecutorService, CompletableFuture — over hand-rolled synchronized/ wait/notify, because those constructs encode correct concurrency you'd otherwise have to re-derive and get wrong. The producer-consumer pattern via a bounded BlockingQueue exemplifies the whole philosophy — it decouples components, handles waiting/signaling correctly, and provides back-pressure so overload degrades gracefully instead of exhausting memory — and a thread pool is just this pattern productized. Finally, the senior recognizes that concurrency bugs are non-deterministic: a race can pass thousands of test runs and then corrupt data in production under a timing that testing never hit, so correctness must be designed in by reasoning about happens-before and using proven constructs, not tested in after the fact. That combination — knowing the safety/liveness split, defaulting to immutability and confinement, guarding shared state completely, reaching for high-level tools, and treating concurrency as something to reason about rather than debug — is exactly the judgment senior interviews probe when they ask you to fix a deadlock, spot a race, or design a thread-safe component.

Interview Questions

Q1 (beginner): What is a deadlock and how do you prevent it?

A deadlock is when two or more threads are blocked forever, each waiting for a resource (typically a lock) that another holds, forming a cycle — thread 1 holds lock A and wants B, thread 2 holds B and wants A, so neither can proceed. The classic example is transferring between two accounts in opposite directions concurrently, where each thread locks one account and waits for the other. The standard prevention is to impose a consistent global order in which locks are always acquired (e.g. always lock the account with the lower id first), which makes the circular wait impossible. Other techniques include using tryLock with a timeout so a thread backs off and retries rather than blocking indefinitely, holding only one lock at a time, or avoiding shared locks altogether via immutability/confinement. Formally, deadlock requires four conditions (mutual exclusion, hold-and-wait, no preemption, circular wait) and breaking any one prevents it — lock ordering breaks circular wait.

Q2 (beginner): What is a race condition? Give the most common form.

A race condition is a bug where the correctness of the program depends on the relative timing of threads — the same code can produce different, sometimes wrong, results depending on how operations interleave. It arises from unsynchronized access to shared mutable state. The most common form is check-then-act: a thread checks a condition and then acts on it, but the two steps aren't atomic, so another thread can change the state in between, making the action based on a stale check. For example, "if the map doesn't contain key K, put a value" can have two threads both pass the check and both put, losing one update. Another form is read-modify-write like count++ (read, add, store — three steps that can interleave and lose increments). The fix is to make the compound operation atomic: use an atomic method (computeIfAbsent, putIfAbsent), a compare-and-swap (AtomicInteger), or hold a lock across the entire check-and-act so no other thread can interleave.

Q3 (mid): What's the difference between deadlock, livelock, and starvation?

All three are liveness failures — the program fails to make progress — but by different mechanisms. In a deadlock, threads are blocked and completely stuck in a cycle of waiting for each other's locks; nothing moves and nothing will ever move. In a livelock, threads are not blocked — they're actively running and changing state — but they keep reacting to each other in a way that produces no net progress; the textbook image is two people in a hallway each stepping aside in the same direction repeatedly. Livelock often comes from overly "polite" retry/backoff logic where threads keep yielding to each other in lockstep; the fix is usually randomized backoff so they desynchronize. Starvation is when a particular thread is perpetually denied a resource or CPU time it needs — not because of a cycle, but because others keep getting it first (e.g. an unfair lock always granting to other waiters, or thread-priority abuse); the fix is fairness (fair locks, avoiding priority games). Distinguishing them matters because the remedies differ — lock ordering for deadlock, backoff randomization for livelock, fairness for starvation — and all are distinct from race conditions, which are safety (wrong-result) failures rather than liveness ones.

Q4 (mid): You need a component to be thread-safe. Walk through your options in order of preference.

I'd prefer eliminating shared mutable state before managing it. First, immutability: if the component's data never changes after construction (a value object, a record), it's inherently thread-safe — no writes means nothing to race on — and can be shared freely; this is the simplest and most robust option. Second, confinement: if there is mutable state but it can be kept on a single thread, do that — local variables are confined automatically, and ThreadLocal gives each thread its own copy of something non-thread-safe (like SimpleDateFormat), avoiding sharing entirely. Third, higher-level concurrent constructs: if state must be shared, prefer a purpose-built thread-safe tool over manual locking — an AtomicLong for a counter, a ConcurrentHashMap for a map, a BlockingQueue for handoff, an ExecutorService for task execution — because they encode correct, often lock-free concurrency and are hard to misuse. Last, explicit locking: when nothing higher-level fits, guard every access (reads and writes) with the same lock, keep the critical section small, document the guarding lock, and use a consistent lock order if multiple locks are involved. The guiding principle is "share less, and when you must share, share safely with the highest-level tool available" — dropping to raw synchronized only when the abstraction level truly requires it.

Q5 (senior): Explain the producer-consumer pattern and why the queue should be bounded.

Producer-consumer decouples the threads that generate work (producers) from the threads that process it (consumers) by placing a queue between them: producers put items on the queue, consumers take items off, and neither needs to know about the other's rate or existence. In Java you implement it with a BlockingQueue, which handles all the synchronization internally — put blocks when the queue is full and take blocks when it's empty, so there's no busy-waiting and no error-prone manual wait()/notify(). The benefits are decoupling (you can scale producers and consumers independently), smoothing of bursty load, and clean shutdown semantics. The queue should be bounded because a bounded queue provides back-pressure: when consumers fall behind and the queue fills, put blocks the producers, throttling them to the rate the consumers can sustain. With an unbounded queue there's no such feedback — under sustained overload the queue grows without limit until the process runs out of memory and crashes, which is far worse than briefly slowing producers. So bounding turns overload into graceful degradation (slower intake) instead of a catastrophic OOM. This is exactly how a ThreadPoolExecutor works internally — submit produces tasks onto a work queue that worker threads consume — and it's why choosing the executor's queue bound and rejection policy is a real capacity-planning decision, not a detail. Recognizing that back-pressure is a first-class design concern, and that unbounded buffering is a latent outage, is a senior-level insight.

Q6 (senior): Why can't you rely on testing to catch concurrency bugs, and what do you do instead?

Because concurrency bugs are non-deterministic: whether a race manifests depends on the precise interleaving of threads, which is influenced by timing, core count, load, the JIT, and memory visibility — none of which your tests control. A racy piece of code can execute correctly in millions of runs on your machine and then fail in production under a slightly different timing, because the tests simply never happened to hit the bad interleaving; the bug was always there, just latent. Worse, adding logging or a debugger changes timing and can hide it (a "heisenbug"). So testing can increase confidence and catch some issues (stress tests, tools like jcstress, thread sanitizers, static analyzers help), but it cannot prove the absence of races the way it can for deterministic logic. The senior response is to design correctness in rather than test it in: minimize shared mutable state (immutability and confinement remove whole classes of race), reason explicitly about the happens-before relationships the memory model guarantees (so you know reads see the writes you intend), guard all shared access with the same lock and keep critical sections small, use a consistent lock order to preclude deadlock, and lean on well-tested high-level constructs (ConcurrentHashMap, BlockingQueue, ExecutorService, Atomic*) that encode correct concurrency instead of re-deriving it. In short, you make the code correct by construction and argument, and use testing/tooling as a safety net rather than the primary guarantee — treating concurrency as something to reason about, which is precisely the discipline that separates engineers who write reliable concurrent systems from those who ship timing bombs.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Cause and fix a deadlock. Write the two-account opposite-transfer deadlock, then fix it with lock ordering.
Solution

Two threads calling transfer(a,b) and transfer(b,a) lock in opposite orders and deadlock. Fix by always locking the lower-id account first, so both threads acquire in the same order and no cycle can form — breaking the circular-wait condition.

2 (easy): Fix a check-then-act race. Replace an if(!map.containsKey) map.put with an atomic operation.
Solution

Use map.computeIfAbsent(k, key -> compute()) on a ConcurrentHashMap — the check and put are one atomic operation, so two threads can't both insert. The separate check-then-act had a window for a lost update.

3 (medium): Make a class thread-safe two ways. Take a mutable counter and make it safe with (a) synchronization and (b) an atomic.
Solution

(a) synchronized on both inc() and get() (reads need the lock for visibility). (b) AtomicLong with incrementAndGet() — lock-free and simpler. Prefer the atomic for a plain counter.

4 (medium): Producer-consumer. Implement it with a bounded BlockingQueue and explain the back-pressure behavior.
Solution

Producers queue.put(task), consumers queue.take(); with a bounded queue, put blocks when full, throttling producers to the consumers' rate. An unbounded queue would grow until OOM under overload — bounding gives graceful degradation.

5 (hard): Diagnose non-progress. Given symptoms, decide whether it's deadlock, livelock, or starvation, and name the fix.
Solution

Threads blocked in a lock cycle → deadlock (lock ordering / tryLock). Threads busy but no progress, reacting to each other → livelock (randomized backoff). One thread never served while others proceed → starvation (fair locks, no priority abuse). A thread dump distinguishes blocked (deadlock) from running (livelock).

6 (hard): Explain-it test. Out loud: (a) safety vs liveness; (b) share-less (immutability/confinement) vs share-safely (guard all access); (c) why bounded producer-consumer and why you design rather than test concurrency correctness.
Self-check rubric

Pass (a) if you said safety=no wrong results (races), liveness=progress (deadlock/livelock/starvation). Pass (b) if you said immutability/confinement remove races and, when sharing, guard every access with one lock + prefer high-level tools. Pass (c) if you said bounded = back-pressure vs OOM and races are non-deterministic so must be designed out. That completes concurrency.

🎯 Key Takeaways

  • Correctness = safety (no races) + liveness (no deadlock/livelock/starvation) — design for both.
  • Deadlock comes from inconsistent lock order → impose a global lock ordering.
  • The common race is check-then-act → make it atomic (computeIfAbsent, CAS, or one lock).
  • Share less first: immutability and confinement remove races; only then share safely — guard every access (reads too) with the same lock.
  • Prefer high-level tools (BlockingQueue, ExecutorService, ConcurrentHashMap, Atomic*) over raw synchronized/wait/notify.
  • Use a bounded producer-consumer queue for back-pressure; design concurrency correctness in — you can't test it in.
