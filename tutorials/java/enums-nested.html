Enums & Nested Classes Object-Oriented Java

What you'll master in this lesson: why enums are far more than named constants — they can hold fields, methods, and per-constant behavior, and are the correct way to model a fixed set of options; why an enum is the best singleton; and the four kinds of nested classes (static nested, inner, local, anonymous) with the crucial inner-class memory-leak trap. Both features are about expressing intent precisely — using the right construct instead of loose constants or top-level clutter.

Part 1 — Enums Are Type-Safe Constants (and much more)

// The old, unsafe way — "int constants" with no type safety:
public static final int STATUS_ACTIVE = 0, STATUS_PAUSED = 1, STATUS_CLOSED = 2;
void setStatus(int s) { ... }     // setStatus(99) compiles fine — garbage accepted!

// The enum way — a real type with a fixed, exhaustive set of values:
public enum Status { ACTIVE, PAUSED, CLOSED }
void setStatus(Status s) { ... }  // setStatus(Status.ACTIVE) — nothing else compiles

Status s = Status.ACTIVE;
s.name();       // "ACTIVE"          s.ordinal();   // 0 (position — avoid depending on it)
Status.valueOf("PAUSED");           // parse from string
Status.values();                    // array of all constants — great for iterating
💡 Enums give you compile-time safety and exhaustiveness. A method taking a Status can only receive a valid status — no magic numbers, no invalid values, and a switch over an enum can be checked for completeness by the compiler. Whenever you have a fixed, known set of options (states, directions, days, roles), an enum is almost always the right choice over int/String constants.

Part 2 — Enums with Fields, Constructors & Methods

This is what surprises people: an enum is a full class. Each constant is an instance, and you can give enums state and behavior:

public enum Planet {
    MERCURY(3.303e+23, 2.4397e6),          // each constant calls the constructor
    EARTH  (5.976e+24, 6.37814e6),
    JUPITER(1.9e+27,   7.1492e7);

    private final double mass, radius;      // per-constant FIELDS
    Planet(double mass, double radius) {    // constructor (implicitly private)
        this.mass = mass; this.radius = radius;
    }
    public double gravity() {               // METHOD — behavior on the enum
        return 6.67300E-11 * mass / (radius * radius);
    }
}
Planet.EARTH.gravity();       // 9.8 — real behavior, not just a label
for (Planet p : Planet.values()) System.out.println(p + ": " + p.gravity());

Part 3 — Per-Constant Behavior (the "enum as strategy" trick)

public enum Operation {
    PLUS  { public int apply(int a, int b) { return a + b; } },   // each constant OVERRIDES
    MINUS { public int apply(int a, int b) { return a - b; } },
    TIMES { public int apply(int a, int b) { return a * b; } };

    public abstract int apply(int a, int b);   // the contract each constant must implement
}
Operation.PLUS.apply(3, 4);    // 7 — the enum dispatches to PLUS's implementation
🔍 This is a clean replacement for a switch-on-type. Instead of a switch that must be updated everywhere when you add an operation, each constant carries its own behavior, and adding DIVIDE forces you to implement apply for it (the compiler won't let you forget). This "constant-specific method" pattern turns an enum into a type-safe strategy table — a favorite of clean-code advocates and a strong interview answer to "how would you avoid a big switch statement?".

Part 4 — Enum as the Best Singleton

public enum Config {
    INSTANCE;                               // exactly one instance, guaranteed by the JVM
    private final Properties props = load();
    public String get(String key) { return props.getProperty(key); }
}
Config.INSTANCE.get("db.url");
🎯 A single-element enum is the best way to implement a singleton (Effective Java, Item 3). The JVM guarantees exactly one instance, it's thread-safe and lazy-loaded without any locking code, and — uniquely — it's safe against serialization and reflection attacks that can break hand-rolled singletons (creating extra instances). The classic private static final INSTANCE + private constructor works but requires care; enum INSTANCE is bulletproof and one line. (More on singletons in the Design Patterns lesson.)

public class Outer {
    private int x = 10;

    static class StaticNested {              // NO implicit reference to an Outer instance
        void go() { /* can't touch x — needs an Outer object explicitly */ }
    }
    class Inner {                            // has an IMPLICIT reference to the enclosing Outer
        void go() { System.out.println(x); } // can use Outer's fields directly
    }
    void method() {
        Runnable r = new Runnable() {        // ANONYMOUS class — inline one-off implementation
            public void run() { System.out.println(x); }
        };
        // Modern equivalent: Runnable r = () -> System.out.println(x);  (lambda — Lambdas lesson)
    }
}

Part 6 — Deep Dive: The Inner-Class Memory Leak

// A non-static inner class silently holds a reference to its enclosing instance.
// If the inner object outlives the outer, it KEEPS THE OUTER ALIVE — a memory leak:
class Screen {                               // imagine a large UI object
    private byte[] pixels = new byte[10_000_000];
    Runnable makeCallback() {
        return new Runnable() {              // anonymous inner class → holds 'this' (the Screen!)
            public void run() { /* ... */ }  // even if it never uses Screen's fields
        };
    }
}
// If that Runnable is stored somewhere long-lived (an event bus, a static list), the whole
// 10MB Screen can't be garbage-collected as long as the callback lives. Classic Android/UI leak.
🔍 The fix and the rule: if a nested class doesn't need the outer instance's state, make it static (a static nested class holds no reference to the outer object). This is Effective Java Item 24: "favor static member classes over non-static". The same applies to lambdas — a lambda that captures this (or an instance field) also captures the enclosing instance; a lambda that captures nothing instance-related doesn't. This leak is invisible in code review and shows up as OutOfMemoryError under load, which is why it's a favorite senior interview question.

Interview Questions

Q1 (beginner): Why use an enum instead of int or String constants?

Type safety and expressiveness. An enum defines a distinct type with a fixed set of valid values, so a method parameter typed as the enum can only receive one of those values — the compiler rejects anything else, unlike int/String constants where a wrong number or misspelled string compiles and fails at runtime. Enums also give readable names, iteration via values(), safe parsing via valueOf, compiler-checked exhaustive switches, and the ability to attach fields and methods. Whenever the set of options is fixed and known at compile time (states, roles, directions), an enum is the right model.

Q2 (beginner): Can enums have methods and fields? Give an example.

Yes — an enum is a full class whose constants are instances. You can declare private fields, a constructor (implicitly private, called once per constant with the arguments in the constant declaration), and methods. For example a Planet enum stores mass and radius per constant and has a gravity() method computing surface gravity; Planet.EARTH.gravity() returns 9.8. You can even give each constant its own behavior by overriding an abstract method in the constant's body (constant-specific methods), turning the enum into a type-safe strategy table.

Q3 (mid): What are the four kinds of nested classes and how do they differ?

Static nested: declared static inside a class; it's namespaced under the outer type but has no implicit reference to an outer instance — use it for a helper logically tied to the outer class but not to any instance. Inner (non-static): a non-static class inside a class; each instance holds an implicit reference to an enclosing instance and can access its fields — use when it genuinely needs the outer object's state. Local: declared inside a method, visible only there, capturing effectively-final locals. Anonymous: an unnamed inline class that implements an interface or extends a class in one expression — a one-shot implementation, now often replaced by lambdas for functional interfaces. The key distinction is whether they capture the enclosing instance (all but static nested do), which has memory implications.

Q4 (mid): Why is a single-element enum considered the best singleton?

Because the JVM guarantees exactly one instance of each enum constant, created lazily and in a thread-safe way with no locking code, and — critically — enum instances are immune to the two attacks that break hand-rolled singletons: serialization (a deserialized ordinary singleton would create a second instance unless you add readResolve) and reflection (reflection can invoke a private constructor to make extra instances, but the JVM forbids reflectively instantiating enums). So enum Singleton { INSTANCE; } gives a correct, concise, attack-proof singleton in one line, whereas the classic static-field approach requires careful extra code to match those guarantees. This is Effective Java Item 3.

Q5 (senior): Explain how a non-static inner class (or capturing lambda) can cause a memory leak.

A non-static inner class instance holds an implicit, hidden reference to the enclosing outer instance (that's how it accesses the outer's fields). If the inner instance is stored somewhere long-lived — an event listener registry, a static collection, a callback queue, a background task — it keeps the entire outer object reachable, so the garbage collector cannot reclaim the outer even after the rest of the program is done with it. If the outer is large (a UI screen, a cache, a big buffer), this is a significant leak that manifests as growing memory and eventually OutOfMemoryError, and it's invisible in the source because the reference is implicit. The same applies to lambdas and anonymous classes that capture this or an instance field. The remedy is to make the nested class static (no enclosing reference) and pass in only the specific data it needs, or use a lambda that captures nothing instance-related, or hold the outer via a WeakReference when a back-reference is unavoidable. This is why Effective Java advises favoring static member classes.

Q6 (senior): Why should you avoid persisting or branching on ordinal()?

ordinal() returns the enum constant's position (0-based) in its declaration order, which is an incidental property, not a stable contract. If you store ordinals in a database, serialize them, or use them as array indices tied to meaning, then any future edit to the enum — reordering constants, inserting a new one in the middle, removing one — silently shifts every subsequent ordinal and corrupts the mapping between stored numbers and constants, with no compile error. The documentation explicitly warns against relying on it (it exists mainly for EnumMap/EnumSet internals). Instead persist name() (stable unless you rename) or, better, give the enum an explicit, deliberately-assigned code field (e.g. a database id) and map by that, so the wire/storage representation is decoupled from declaration order. The principle: never let a value's identity depend on a mutable declaration ordering.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Basic enum. Define a Direction enum (NORTH/EAST/SOUTH/WEST). Iterate values() printing each with its ordinal(), and parse "SOUTH" with valueOf.
Solution

for (Direction d : Direction.values()) System.out.println(d + " " + d.ordinal()); prints each with 0..3; Direction.valueOf("SOUTH") returns the SOUTH constant (throws IllegalArgumentException for an unknown name). Note you're printing ordinal for illustration only — don't build logic on it.

2 (easy): Enum with data. Give Coin (PENNY/NICKEL/DIME/QUARTER) a cents field via a constructor and a method to sum a list of coins.
Solution

Each constant declares its value: PENNY(1), NICKEL(5), DIME(10), QUARTER(25); with a private final int cents and constructor. A static helper sums coin.getCents() across a list. This shows enums carrying real data, not just labels.

3 (medium): Constant-specific behavior. Build an Operation enum (PLUS/MINUS/TIMES/DIVIDE) where each constant implements an abstract apply(int,int). Then try adding a constant without implementing apply.
Solution

Each constant has a body overriding apply. Adding MOD without an apply implementation is a compile error — the abstract method forces you to define behavior for every constant. That compiler enforcement is the advantage over a switch statement, which would silently miss the new case.

4 (medium): Enum singleton. Implement a Logger as a single-element enum with a log(String) method. Access it from two places and confirm it's the same instance.
Solution

enum Logger { INSTANCE; public void log(String m) { ... } }. Both Logger.INSTANCE references are identical (== true) — the JVM guarantees one instance. No synchronization, no double-checked locking, immune to serialization/reflection. Compare mentally to the boilerplate of a classic singleton.

5 (hard): Fix an inner-class leak. Given a large Cache class whose createTask() returns an anonymous Runnable that never uses the cache's fields, explain the leak and refactor so the task doesn't retain the Cache.
Solution

The anonymous Runnable is a non-static inner class, so it implicitly holds the Cache instance — storing the task long-term pins the whole cache in memory. Fix: make the task a static nested class (or a top-level class, or a static-context lambda) that takes only the data it needs as parameters, so it holds no reference to the Cache. Now the cache can be GC'd when otherwise unreachable. This is the "favor static member classes" rule applied.

6 (hard): Explain-it test. Out loud: (a) three advantages of enums over int constants; (b) why a single-element enum is the best singleton; (c) how a non-static inner class leaks memory and the fix.
Self-check rubric

Pass (a) if you named type safety, attached data/behavior, and exhaustive switches/iteration. Pass (b) if you cited JVM-guaranteed single instance plus serialization/reflection safety. Pass (c) if you explained the implicit outer reference pinning the outer object, fixed by static. These prepare you for Design Patterns (singleton/strategy) and Lambdas (capture).

🎯 Key Takeaways

  • Enums are type-safe constants with names, iteration, and exhaustive switches — use them for any fixed set of options.
  • Enums are full classes: they can hold fields, constructors, and methods, even per-constant behavior (a type-safe strategy table).
  • A single-element enum is the best singleton — one guaranteed instance, thread-safe, and immune to serialization/reflection attacks.
  • Don't build logic on ordinal() — reordering constants corrupts it; persist name() or an explicit code.
  • Four nested classes: static nested (no outer ref), inner/local/anonymous (capture the outer instance).
  • Favor static nested classes — a non-static inner class (or capturing lambda) holds the outer instance and can leak memory.
