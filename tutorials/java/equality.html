equals, hashCode & Object Object-Oriented Java

What you'll master in this lesson: the Object methods every class inherits; the difference between reference equality (==) and logical equality (.equals()); the equals/hashCode contract and the exact, dramatic bug you get when you break it; how to write both correctly (and why records do it for you); and the rules for toString, Comparable, and Comparator. This is one of the most reliably-tested interview topics in all of Java — get it wrong and your objects silently vanish from collections.

Part 1 — The Object Class: What Every Object Inherits

Every class in Java implicitly extends java.lang.Object, inheriting a handful of methods you'll frequently override:

MethodDefault behaviorYou override it to…
equals(Object)Reference equality (this == other)Define logical equality (same content)
hashCode()Identity-based (roughly the memory address)Return equal hashes for equal objects
toString()ClassName@1b6d3586 (useless)A readable description for logs/debugging
getClass()The runtime class (final — can't override)—
💡 The defaults are based on object identity: two references are "equal" only if they're the same object, and the hashCode is derived from identity. That's fine for objects with no notion of value equality, but wrong for value objects — a Money, a Point, a User — where two separate objects with the same fields should be considered equal. For those, you override equals and hashCode together.

Part 2 — == vs equals (reference vs logical equality)

String a = new String("hi");
String b = new String("hi");
a == b;            // false — two DIFFERENT objects (== compares references/identity)
a.equals(b);       // true  — same CONTENT (String overrides equals to compare characters)

Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);
p1 == p2;          // false — different objects
p1.equals(p2);     // FALSE too — UNLESS Point overrides equals! (default is ==)
🎯 == asks "are these the same object?" (identity). .equals() asks "are these logically equal?" — but only if the class overrides it; otherwise .equals() falls back to ==. So your custom class's objects are equal only if you teach it what equality means. For primitives there's no .equals() — == compares values directly and is correct.

Part 3 — The equals/hashCode Contract (memorize this)

Java's collections rely on a contract between these two methods. Break it and hash-based collections (HashMap, HashSet) misbehave in baffling ways.

THE CONTRACT:
  1. If a.equals(b) is true, then a.hashCode() == b.hashCode()  ← the critical one
  2. Consistency: repeated calls return the same result (if the object doesn't change)
  3. equals is reflexive (a.equals(a)), symmetric (a.equals(b) ⟺ b.equals(a)),
     transitive (a=b, b=c ⟹ a=c), and null-safe (a.equals(null) is false)
  4. The reverse of rule 1 is NOT required: equal hashCodes don't imply equal objects
     (that's a "hash collision" — allowed and normal)
// The golden rule: OVERRIDE BOTH, OR NEITHER. Override equals but not hashCode and you break rule 1.
public final class Point {
    private final int x, y;
    public Point(int x, int y) { this.x = x; this.y = y; }

    @Override public boolean equals(Object o) {
        if (this == o) return true;                    // fast path: same object
        if (!(o instanceof Point p)) return false;     // type check + pattern bind (16+)
        return x == p.x && y == p.y;                    // compare the fields that define equality
    }
    @Override public int hashCode() {
        return Objects.hash(x, y);                      // derive from the SAME fields
    }
}

Part 4 — Deep Dive: The Vanishing-Object Bug

Here is exactly what happens when you override equals but forget hashCode — the single most instructive bug in Java:

// Point overrides equals() but NOT hashCode() (so hashCode is identity-based):
Set<Point> visited = new HashSet<>();
visited.add(new Point(1, 1));
visited.contains(new Point(1, 1));   // FALSE — the object "vanished"!

Map<Point,String> labels = new HashMap<>();
labels.put(new Point(1, 1), "home");
labels.get(new Point(1, 1));         // null — can't find the key we just put in!

Why? A HashMap stores entries in buckets chosen by hashCode(). The two Point(1,1) objects are equals(), but with identity-based hashCodes they get different hashes → they land in different buckets. get/contains computes the hash of the lookup key, goes to that bucket, and doesn't find the entry (which is sitting in a different bucket). The object is present but unreachable — it "vanished". Adding a correct hashCode (equal objects → equal hashes → same bucket) fixes it instantly.

🚨 This is the reason "override both or neither" is stated so forcefully. The bug is silent (no exception), data-dependent, and hard to trace back to the missing method. It's also why records (Modern Java lesson) — which auto-generate a correct, consistent equals/hashCode/toString from their components — are the preferred way to write value objects today.

Part 5 — toString, Comparable & Comparator

@Override public String toString() {          // for logs, debugging, error messages
    return "Point(" + x + ", " + y + ")";     // the default ClassName@hash is useless
}

// Comparable<T> — the object's NATURAL ordering (one canonical sort). Implement compareTo:
public final class Version implements Comparable<Version> {
    private final int major, minor;
    @Override public int compareTo(Version o) {
        int c = Integer.compare(major, o.major);   // negative/zero/positive
        return c != 0 ? c : Integer.compare(minor, o.minor);   // tie-break on minor
    }
}
Collections.sort(versions);        // uses compareTo — the natural order

// Comparator<T> — an EXTERNAL, swappable ordering (many orderings, defined outside the class):
list.sort(Comparator.comparing(User::getLastName)
                    .thenComparing(User::getFirstName)       // multi-key sort, readable
                    .reversed());
🔍 Comparable vs Comparator: implement Comparable when there's one obvious natural order for the type (version numbers, dates), and it lives inside the class. Use a Comparator when you need different orderings, or to sort a type you don't own, or to sort by different keys in different places — it's defined outside the class and passed in. Comparator.comparing(...).thenComparing(...) builds multi-key sorts cleanly. A subtle contract rule: your compareTo should be consistent with equals (compareTo == 0 iff equals is true), or sorted collections like TreeSet will behave surprisingly.

Part 6 — Deep Dive: Writing equals Correctly (the subtle traps)

  • Use instanceof, not getClass(), carefully. instanceof permits subclasses to be equal to base instances (can break symmetry with subclassing); getClass() requires exact same class (breaks Liskov substitutability). For most value types, prefer making the class final and using instanceof — which removes the dilemma entirely. This is a genuine senior-level subtlety.
  • Compare the right fields. Include exactly the fields that define logical equality — usually all "value" fields, excluding caches, derived values, or identity fields you don't want to matter.
  • Handle null and type mismatch. equals(null) must return false (the instanceof check handles both null and wrong-type in one line, since null instanceof X is false).
  • Keep hashCode in sync. Derive it from the same fields equals uses, via Objects.hash(...). If equals changes, update hashCode.
  • Never mutate fields used in hashCode after using the object as a key. The object's bucket was chosen by its old hash; mutating it makes it unreachable in the map — the same vanishing bug from a different angle.
🔍 The modern shortcut: for a pure value object, declare a record Point(int x, int y) {} and the compiler generates a correct, contract-honoring equals, hashCode, and toString from the components. Hand-write these methods only when you need custom equality semantics or can't use a record. Let your IDE or Objects.hash/Objects.equals do the mechanical parts — hand-rolled versions are where the contract bugs creep in.

Interview Questions

Q1 (beginner): What's the difference between == and .equals()?

== compares references — for objects it's true only if both point to the exact same instance (identity); for primitives it compares values directly. .equals() is a method meant to compare logical equality (same content), but only if the class overrides it; otherwise Object.equals just does ==. So String, wrappers, and records compare by content via .equals(), while a custom class compares by content only if you override equals. Rule of thumb: use .equals() for object value comparison, == for primitives and deliberate identity checks.

Q2 (beginner): Why must you override hashCode whenever you override equals?

Because the contract requires equal objects to have equal hash codes, and hash-based collections (HashMap, HashSet) depend on it. If you override equals so two objects are logically equal but leave the default identity-based hashCode, those equal objects get different hashes, land in different buckets, and the collection can't find them — set.contains(equalObject) returns false and map.get(equalKey) returns null even though you just inserted them. Overriding both, derived from the same fields, keeps equal objects in the same bucket so lookups work.

Q3 (mid): State the equals/hashCode contract precisely.

(1) Consistency with equality: if a.equals(b), then a.hashCode() == b.hashCode() — the load-bearing rule. (2) The converse is not required: unequal objects may share a hash code (a collision), which is normal. (3) equals must be reflexive (a.equals(a)), symmetric (a.equals(b) ⟺ b.equals(a)), transitive (a.equals(b) & b.equals(c) ⟹ a.equals(c)), consistent (same result on repeated calls if unchanged), and return false for null. (4) hashCode must return the same value across calls while the object's equality-relevant state is unchanged. Violating any of these produces subtle collection bugs.

Q4 (mid): Comparable vs Comparator — when do you use each?

Comparable<T> defines a type's single natural ordering via compareTo, implemented inside the class — use it when there's one obvious canonical order (numbers, dates, versions) and you own the class. Comparator<T> is an external, reusable ordering object — use it when you need multiple orderings, want to sort by different keys in different contexts, or must sort a type you don't control. Comparators compose: Comparator.comparing(key).thenComparing(other).reversed() builds multi-key and reversed sorts declaratively. A class can have a natural order (Comparable) and still be sorted by ad-hoc Comparators elsewhere. Note the consistency-with-equals recommendation for Comparable so sorted collections behave.

Q5 (senior): instanceof vs getClass() in equals — what's the trade-off?

The choice affects how equality interacts with inheritance. Using getClass() (require exactly the same class) makes equals reject instances of subclasses, which preserves symmetry and transitivity even across subclasses but violates the Liskov substitution principle — a subclass instance is never equal to a base instance even when it logically should be, and it breaks if frameworks create proxy subclasses. Using instanceof (accept subtypes) upholds substitutability but can break symmetry: if a subclass adds a significant field and also considers it in equals, then base.equals(sub) and sub.equals(base) can disagree, violating the contract. There's no fully general fix for adding a value component to a subclass while preserving the contract — which is exactly why Effective Java recommends favoring composition over inheritance for value types, and why the clean practical answer is to make value classes final and use instanceof: no subclasses exist, so the dilemma disappears. Records enforce this by being implicitly final.

Q6 (senior): You use a mutable object as a HashMap key and later mutate a field that participates in equals/hashCode. What happens and why?

The entry becomes effectively lost. When you put the key, the map computes its hash code and stores the entry in the corresponding bucket. Mutating an equality/hash field afterward changes what the object's hashCode now returns, but the entry is still physically in the old bucket. A subsequent get with an equal key (or the same mutated object) computes the new hash, looks in the new bucket, and finds nothing — the entry is unreachable, though it still occupies memory and appears during full iteration. This is the vanishing-object bug caused by mutation rather than a missing hashCode. The fix is to use immutable keys — which is a major reason String and boxed primitives (immutable) are ideal keys, and why records/value objects used as keys should be immutable. If you must key by a mutable object, remove it before mutating and re-insert after.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Default equals. Write a Point class with x,y and NO overrides. Create two Point(1,1) and compare with == and .equals(). Explain both results.
Solution

Both == and .equals() return false — with no override, equals is inherited from Object and just does == (identity), and these are two distinct objects. This shows the default is reference equality; you must override to get value equality.

2 (easy): Fix it. Add a correct equals and hashCode to Point (using Objects.hash). Re-run the comparison and confirm .equals() is now true.
Solution

equals: same-object fast path, instanceof Point p check, then x == p.x && y == p.y. hashCode: Objects.hash(x, y). Now .equals() is true and (crucially) the two objects have equal hash codes. == is still false — they're still different objects — which is correct.

3 (medium): Reproduce the vanishing bug. Take the version of Point with equals but NO hashCode. Put one in a HashSet, then test contains with an equal Point. Then add hashCode and retest.
Solution

Before adding hashCode: contains(new Point(1,1)) returns false — the equal object hashes to a different bucket (identity hash) than the stored one, so it's unreachable. After adding a matching hashCode: contains returns true. You've reproduced and fixed the single most important consequence of the contract.

4 (medium): Natural ordering. Make a Version(major, minor) implement Comparable so that sorting [2.0, 1.5, 1.10, 1.2] gives [1.2, 1.5, 1.10, 2.0] (numeric, not lexicographic). Verify with Collections.sort.
Solution

compareTo: compare major with Integer.compare, and if equal, compare minor. Numeric comparison makes 1.10 sort after 1.5 (minor 10 > 5), unlike string sorting where "1.10" < "1.5". This is a classic real bug (version strings sorted lexically) and why numeric compareTo matters.

5 (hard): Multi-key Comparator. Sort a List<User> by last name ascending, then first name ascending, then age descending — using Comparator.comparing().thenComparing(). Handle a possible null last name.
Solution

list.sort(Comparator.comparing(User::getLastName, Comparator.nullsLast(Comparator.naturalOrder())).thenComparing(User::getFirstName).thenComparing(User::getAge, Comparator.reverseOrder()));. The nullsLast wrapper handles null last names safely; thenComparing chains tie-breakers; reverseOrder() on the age key makes it descending. This composability is why Comparator beats hand-written multi-key comparisons.

6 (hard): Explain-it test. Out loud: (a) the equals/hashCode contract and the vanishing-object bug; (b) instanceof vs getClass() in equals and the final-class resolution; (c) why mutable objects make dangerous map keys.
Self-check rubric

Pass (a) if you stated "equal ⟹ equal hashCode" and explained different buckets → unreachable. Pass (b) if you named the substitutability vs symmetry trade-off and "make it final + instanceof". Pass (c) if you explained the bucket is chosen at insert time, so mutating the hash field strands the entry. These are staple interview answers and lead into Immutability and Collections.

🎯 Key Takeaways

  • Every class inherits Object methods; the defaults are identity-based — override equals/hashCode/toString for value objects.
  • == is reference identity; .equals() is logical equality only if overridden.
  • The contract: equal objects must have equal hash codes — override both or neither, derived from the same fields.
  • Breaking it makes objects vanish from HashSet/HashMap (different buckets) — a silent, baffling bug.
  • Comparable = one natural order inside the class; Comparator = external, composable orderings (comparing().thenComparing()).
  • Use immutable value objects (ideally records) for equality and as map keys; never mutate a key's equality fields after insertion.
