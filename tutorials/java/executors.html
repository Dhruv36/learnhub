Executors & Futures Concurrency

What you'll master in this lesson: the executor framework that replaced manual thread management — thread pools, task submission, and clean shutdown; Future and Callable for tasks that return results; how to size and choose pools; and CompletableFuture for composing asynchronous work without callback hell. This is the level you actually work at in modern concurrent Java: you don't create threads, you submit tasks to an executor. Getting pool configuration right is a common source of production incidents and a frequent interview topic.

Part 1 — The Executor Framework: Submit Tasks, Not Threads

import java.util.concurrent.*;

// Create a pool of reusable worker threads ONCE:
ExecutorService pool = Executors.newFixedThreadPool(4);   // 4 threads handle all submitted tasks

// Submit tasks — the pool queues them and runs them on its threads:
pool.submit(() -> System.out.println("task on " + Thread.currentThread().getName()));
pool.submit(() -> doWork());

// ALWAYS shut down when done — otherwise the JVM won't exit (pool threads are non-daemon):
pool.shutdown();                                  // stop accepting new tasks; finish queued ones
pool.awaitTermination(30, TimeUnit.SECONDS);      // wait for completion
// pool.shutdownNow();                            // attempt to cancel running tasks too
💡 The executor decouples "what work" from "which thread runs it". You submit Runnable/Callable tasks; the pool manages a fixed set of reusable threads, a work queue, and scheduling. This gives you bounded resource use (no unbounded thread creation), reuse (thread creation is expensive — pooling amortizes it), and lifecycle management. It's the standard model — new Thread() in application code is almost always a mistake when an executor would do.

Part 2 — Callable & Future: Tasks That Return Values

// Runnable returns nothing. Callable<T> returns a T (and can throw checked exceptions):
Callable<Integer> task = () -> { Thread.sleep(100); return 42; };

Future<Integer> future = pool.submit(task);       // submit returns a Future — a handle to the result

// ...do other work while the task runs...

Integer result = future.get();                    // BLOCKS until the task completes, returns the result
Integer r2 = future.get(1, TimeUnit.SECONDS);     // block with a timeout (throws TimeoutException)
future.cancel(true);                              // attempt to cancel
future.isDone();                                  // non-blocking status check
🎯 Future.get() blocks the calling thread until the result is ready — so submit all your tasks first, then collect results, to run them concurrently. Calling get() right after each submit() serializes them (you wait for each before starting the next), defeating the point. Also, get() re-throws a task's exception wrapped in an ExecutionException — that's how you find out a task failed (a raw thread would swallow the exception silently).

// Sizing rule of thumb:
//   CPU-BOUND work:   pool size ≈ number of cores (Runtime.getRuntime().availableProcessors())
//   I/O-BOUND work:   more threads than cores (they spend time blocked, so more can overlap)
// For production, prefer constructing a ThreadPoolExecutor directly with a BOUNDED queue and a
// rejection policy — the Executors factories often use UNBOUNDED queues that hide overload as
// growing memory instead of failing fast.
⚠️ The unbounded-queue trap. newFixedThreadPool uses an unbounded task queue: under overload, tasks pile up in memory indefinitely (eventually OutOfMemoryError) instead of applying backpressure. And newCachedThreadPool creates threads without limit, which can exhaust the machine. For production, construct a ThreadPoolExecutor with a bounded queue and a rejection policy (CallerRunsPolicy, or reject and shed load) so overload fails fast and visibly rather than silently degrading. This is a real, common outage cause.

Part 4 — CompletableFuture: Composing Async Work

// A plain Future can only be blocked on. CompletableFuture (8+) lets you CHAIN async steps:
CompletableFuture<String> cf = CompletableFuture
    .supplyAsync(() -> fetchUser(id), pool)        // run async, produce a User
    .thenApply(user -> user.getEmail())            // transform the result (like map)
    .thenCompose(email -> lookupAsync(email))      // chain another async call (like flatMap)
    .exceptionally(ex -> "default@example.com");    // handle failures in the chain

// Combine independent async tasks:
CompletableFuture<Integer> a = CompletableFuture.supplyAsync(() -> computeA());
CompletableFuture<Integer> b = CompletableFuture.supplyAsync(() -> computeB());
a.thenCombine(b, (x, y) -> x + y)                  // run A and B concurrently, combine when both done
 .thenAccept(sum -> System.out.println(sum));

// Wait for many:
CompletableFuture.allOf(a, b).join();              // wait for all to complete
🔍 CompletableFuture is Java's answer to callback hell — it lets you express "do A, then transform, then do B with the result, and handle errors" as a readable pipeline instead of nested callbacks, and it runs the steps asynchronously. thenApply maps a result, thenCompose chains a dependent async call (flatMap), thenCombine joins two independent futures, and exceptionally/handle deal with failures. It's the foundation of non-blocking service code (and Spring's async support). Provide your own executor to the *Async methods rather than relying on the default common pool for anything that blocks.

Part 5 — Structured Concurrency (a look ahead, Java 21+)

// Managing a group of related concurrent tasks with clean error/cancellation handling (preview):
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Subtask<User>  user  = scope.fork(() -> fetchUser(id));     // start subtasks
    Subtask<Order> order = scope.fork(() -> fetchOrder(id));
    scope.join();                  // wait for all
    scope.throwIfFailed();         // if any failed, cancel the rest and propagate
    return new Result(user.get(), order.get());   // both succeeded
}   // scope auto-closes; treats the group as a unit
🔍 Structured concurrency (incubating/preview in recent JDKs) treats a group of subtasks as a single unit of work with a defined scope: if one fails, the others are cancelled; the parent waits for all; and the whole thing has clear lifetime and error propagation — bringing the discipline of structured programming (blocks, try/finally) to concurrency. It pairs naturally with virtual threads (next lesson). You don't need it day-to-day yet, but it's the direction modern Java concurrency is heading, and worth knowing exists.

Part 6 — Deep Dive: Why the Executor Model Wins

  • Resource control. A pool bounds how many threads exist and (with a bounded queue) how much work can pile up, giving backpressure. Manual thread-per-task has no such limit and falls over under load.
  • Separation of concerns. Tasks (Runnable/Callable) describe what; the executor decides how and where. You can swap a fixed pool for a virtual- thread executor without touching the tasks — the same decoupling as programming to an interface.
  • Results and errors are first-class. Future/CompletableFuture give you the result, a way to wait or time out, cancellation, and — crucially — exception propagation, which raw threads swallow.
  • Correct sizing is a design decision, not an afterthought. Pool size, queue bounds, and rejection policy determine whether your system degrades gracefully or collapses under load — they belong in design and load testing, not defaults.
🔍 Senior framing: the executor framework embodies the lesson that in modern Java you work with tasks and pools, not threads. The hard part isn't the API — it's the operational reasoning: how many threads for this workload (CPU-bound → cores; I/O-bound → more, or virtual threads); what happens under overload (bounded queue + rejection policy for backpressure, not an unbounded queue that OOMs); how failures propagate (through Future.get/exceptionally, not silently lost); and how to shut down cleanly. A frequent production incident is a mis-sized or unbounded pool that hides overload until memory runs out or latency explodes. Knowing to construct a ThreadPoolExecutor deliberately — bounded queue, sensible size, explicit rejection — rather than reaching for a convenience factory is a marker of concurrency maturity, and pairs with the coming shift to virtual threads for I/O-bound workloads.

Interview Questions

Q1 (beginner): What is an ExecutorService and why use one?

An ExecutorService is a managed thread pool: you submit tasks (Runnable/Callable) and it runs them on a maintained set of reusable worker threads, handling queuing, scheduling, and lifecycle. You use one instead of creating threads directly because it bounds resource usage (a fixed number of threads rather than unbounded creation), reuses threads (amortizing the cost of thread creation), provides results and exception handling via Future, and gives clean shutdown. It decouples the work from the execution mechanism, so it's the standard way to run concurrent tasks in modern Java — you rarely write new Thread().

Q2 (beginner): What's the difference between Runnable and Callable?

Runnable's run() returns nothing and can't throw checked exceptions. Callable<T>'s call() returns a value of type T and may throw checked exceptions. You submit a Callable to an executor and get a Future<T> to retrieve its result (and any exception, re-thrown wrapped in ExecutionException). Use Runnable for fire-and-forget side-effect tasks and Callable when the task produces a result or needs to signal failure via a checked exception.

Q3 (mid): How do you size a thread pool?

It depends on whether the work is CPU-bound or I/O-bound. For CPU-bound tasks, the pool should be roughly the number of available cores (Runtime.getRuntime().availableProcessors()), possibly plus one — more threads than cores just add context-switching overhead without more parallelism, since the CPUs are the bottleneck. For I/O-bound tasks, threads spend most of their time blocked waiting, so you want more threads than cores so that while some are blocked others make progress; a common heuristic is cores × (1 + waitTime/computeTime), tuned by measurement. Beyond raw size, production pools should use a bounded queue and a rejection policy so overload produces backpressure rather than unbounded memory growth. And for high-scale I/O-bound work, virtual threads (Java 21) change the calculus by making a thread-per-task model cheap. The real answer is "measure under realistic load," but knowing the CPU-vs-I/O distinction is the starting framework.

Q4 (mid): What does CompletableFuture add over a plain Future?

A plain Future only lets you poll (isDone), cancel, or block on the result (get) — you can't compose or react to it without blocking. CompletableFuture makes async results composable and non-blocking: you can chain transformations (thenApply), chain dependent async calls (thenCompose, like flatMap), combine independent futures (thenCombine, allOf/anyOf), and handle errors within the pipeline (exceptionally, handle) — all as a readable pipeline that runs asynchronously, replacing nested callbacks ("callback hell"). It's the basis of non-blocking service composition. Best practice is to supply your own executor to the *Async methods rather than relying on the default common pool, especially for tasks that block.

Q5 (senior): Why is the unbounded queue in Executors.newFixedThreadPool dangerous, and what do you do instead?

newFixedThreadPool is backed by an unbounded LinkedBlockingQueue, so when tasks arrive faster than the fixed threads can process them, the excess accumulates in the queue without limit. Under sustained overload this means memory grows until the JVM throws OutOfMemoryError — the system doesn't push back, it silently buffers and then dies, and latency balloons as the queue lengthens (tasks wait a long time before running). There's no backpressure signal to callers that the system is saturated. The production-grade approach is to construct a ThreadPoolExecutor directly with a bounded work queue (e.g. ArrayBlockingQueue(capacity)) and an explicit rejection policy: when the queue is full, AbortPolicy throws so callers know to back off, CallerRunsPolicy makes the submitting thread run the task (naturally slowing intake — a simple backpressure mechanism), or a custom policy sheds load. You also set core/max pool sizes and a keep-alive deliberately. This way overload manifests as fast, visible failures or throttling you can monitor and respond to, rather than a slow-motion memory crash — a distinction that separates code that survives a traffic spike from code that falls over. The same reasoning applies to newCachedThreadPool, whose unbounded thread creation can exhaust the machine.

Q6 (senior): How does the executor model prepare a codebase for virtual threads, and what changes / doesn't when you adopt them?

Because tasks are already expressed as Runnable/Callable submitted to an ExecutorService, adopting virtual threads is largely a matter of swapping the executor — e.g. replacing a fixed pool with Executors.newVirtualThreadPerTaskExecutor() — without rewriting the tasks, exactly the payoff of decoupling work from execution. What changes: for I/O-bound workloads you no longer size or pool threads to conserve them, because virtual threads are cheap (you can have a million), so the thread-per-task model becomes viable and you stop tuning pool sizes and worrying about a small pool's threads all blocking on I/O; blocking calls become fine again because a blocked virtual thread cheaply unmounts from its carrier OS thread. What does not change: the memory model and synchronization rules are identical (races, visibility, and happens-before still apply — virtual threads don't make shared mutable state safe); CPU-bound work is still limited by cores, so pooling/parallelism limits still matter there; and you must avoid pinning virtual threads (holding a synchronized monitor or a native call across a blocking operation can pin the carrier thread, undermining scalability — prefer ReentrantLock in those spots). You also still want backpressure at the request admission level, since unlimited cheap threads can overwhelm a downstream database or service even if the JVM copes. So the executor abstraction makes the transition mostly mechanical, virtual threads remove the thread-scarcity constraint for I/O, but correctness, CPU limits, and system-level flow control remain your responsibility.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Pool basics. Create a fixed pool of 3 threads, submit five tasks that print their thread name, shut it down cleanly. Observe thread reuse.
Solution

var pool = Executors.newFixedThreadPool(3);, submit five Runnables, then pool.shutdown(); pool.awaitTermination(...). The five tasks run across only three thread names (e.g. pool-1-thread-1..3) — the pool reuses its threads rather than creating five. Without shutdown, the JVM wouldn't exit.

2 (easy): Callable result. Submit a Callable<Integer> that computes a value after a sleep, and retrieve it via Future.get(). Add a timeout.
Solution

Future<Integer> f = pool.submit(() -> { Thread.sleep(200); return 42; }); then f.get() blocks and returns 42, or f.get(100, TimeUnit.MILLISECONDS) throws TimeoutException because the task takes longer. Future gives you the result, timeout, and cancellation a raw thread doesn't.

3 (medium): Concurrent, not serial. Submit three slow tasks and collect their results so they run concurrently. Show the wrong version that accidentally serializes them.
Solution

Right: submit all three first (collect the Futures), then call get() on each — total time ≈ the slowest task. Wrong: get() immediately after each submit() — total time ≈ the sum, because you block on each before submitting the next, serializing them. Submit-all-then-collect is the pattern for real concurrency.

4 (medium): CompletableFuture chain. Compose: fetch a user async, map to their email, then look up something by email async, with a fallback on error. Use supplyAsync/thenApply/thenCompose/exceptionally.
Solution

CompletableFuture.supplyAsync(() -> fetchUser(id), pool).thenApply(User::getEmail).thenCompose(email -> lookupAsync(email)).exceptionally(ex -> fallback);. thenApply transforms the value, thenCompose chains the dependent async call (avoiding a nested CompletableFuture<CompletableFuture<...>>), and exceptionally recovers from any failure in the chain — a readable async pipeline, no nested callbacks.

5 (hard): Bounded pool with backpressure. Construct a ThreadPoolExecutor with a bounded queue and CallerRunsPolicy. Flood it with tasks and explain what CallerRunsPolicy does under overload.
Solution

new ThreadPoolExecutor(4, 4, 0L, MILLISECONDS, new ArrayBlockingQueue<>(100), new ThreadPoolExecutor.CallerRunsPolicy()). When both the threads and the 100-slot queue are full, new submissions run on the calling thread instead of being queued or dropped — which slows the producer down (it's now busy running a task), naturally throttling intake. That's built-in backpressure: overload slows the source rather than growing memory unbounded. Contrast the unbounded default that would OOM.

6 (hard): Explain-it test. Out loud: (a) why submit-tasks-to-a-pool beats thread-per-task; (b) Future vs CompletableFuture; (c) the unbounded-queue danger and the bounded- queue-plus-rejection-policy fix.
Self-check rubric

Pass (a) if you cited bounded resources/reuse/results/shutdown and task-vs-execution decoupling. Pass (b) if you said Future only blocks, CompletableFuture composes async steps and handles errors. Pass (c) if you explained unbounded queue → OOM under overload vs bounded queue + rejection policy → backpressure. These lead into Concurrent Collections and Virtual Threads.

🎯 Key Takeaways

  • Submit tasks to an ExecutorService pool, don't create threads — reuse, bounded resources, results, and clean shutdown (shutdown + awaitTermination).
  • Callable returns a value via a Future; submit all tasks then collect results (don't get() after each submit()).
  • Size pools by workload: CPU-bound ≈ cores, I/O-bound = more; know the CPU-vs-I/O distinction.
  • Avoid unbounded queues/pools in production — use a bounded queue + rejection policy for backpressure, not OOM.
  • CompletableFuture composes async work (thenApply/thenCompose/thenCombine/exceptionally) — no callback hell.
  • The task/pool model makes adopting virtual threads (and structured concurrency) mostly a matter of swapping the executor.
