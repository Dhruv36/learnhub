Testing with JUnit & Mockito Professional Java

What you'll master in this lesson: writing effective unit tests with JUnit 5 and AssertJ; the Arrange-Act-Assert structure and what makes a test good (fast, isolated, readable, deterministic); mocking dependencies with Mockito to test in isolation; the testing pyramid and where different test types fit; and TDD and coverage as tools rather than dogma. Testing is what lets you change code with confidence — the professional skill that makes all the design principles from earlier lessons actually pay off, and a universal interview and code-review topic.

Part 1 — A JUnit 5 Test & Arrange-Act-Assert

import org.junit.jupiter.api.*;
import static org.assertj.core.api.Assertions.*;   // AssertJ: fluent, readable assertions

class CalculatorTest {
    @Test
    void addsTwoNumbers() {
        // ARRANGE — set up the inputs and the object under test
        Calculator calc = new Calculator();
        // ACT — perform the one action being tested
        int result = calc.add(2, 3);
        // ASSERT — verify the outcome
        assertThat(result).isEqualTo(5);
    }

    @Test
    void throwsOnDivideByZero() {
        Calculator calc = new Calculator();
        assertThatThrownBy(() -> calc.divide(1, 0))     // asserting an exception
            .isInstanceOf(ArithmeticException.class);
    }
}
💡 Every test follows Arrange-Act-Assert (AAA). Set up the world, do the one thing, check the result. A test that acts multiple times or checks unrelated things is doing too much — one logical assertion per test keeps failures diagnostic (you know exactly what broke). AssertJ's fluent assertThat(x).isEqualTo(y) reads like a sentence and gives better failure messages than JUnit's built-in assertEquals. JUnit 5 is the modern standard (JUnit 4 is legacy).

Part 2 — What Makes a Good Test (FIRST)

PropertyMeaning
FastMilliseconds — you run them constantly; slow tests get skipped
IsolatedNo dependence on other tests, order, or shared state
RepeatableSame result every run (no clocks, randomness, network — deterministic)
Self-validatingPass/fail automatically; no manual inspection of output
TimelyWritten with (or before) the code, not bolted on much later
🎯 Test behavior, not implementation. A good test asserts what the code does (its observable outcome/contract), not how it does it internally. Tests coupled to implementation details break on every refactor even when behavior is unchanged — making them a liability that discourages improving the code. Name tests by the behavior they verify (throwsOnDivideByZero, not test1), so a failing test name tells you what broke. Flaky (non-deterministic) tests are worse than no test — they erode trust until people ignore failures.

Part 3 — Mockito: Testing in Isolation

import static org.mockito.Mockito.*;

class OrderServiceTest {
    @Test
    void chargesTheGatewayOnCheckout() {
        // ARRANGE — mock the collaborators so we test OrderService in ISOLATION:
        PaymentGateway gateway = mock(PaymentGateway.class);
        OrderRepository repo   = mock(OrderRepository.class);
        when(repo.findById(1L)).thenReturn(Optional.of(new Order(1L, 100)));   // stub a return value

        OrderService service = new OrderService(gateway, repo);   // constructor injection pays off (OOP lesson)

        // ACT
        service.checkout(1L);

        // ASSERT — verify the interaction with the mock:
        verify(gateway).charge(100);                 // was charge(100) called?
        verify(repo).markPaid(1L);
    }
}
🔍 Mocks replace real dependencies so you test one unit in isolation. mock() creates a fake; when(...).thenReturn(...) stubs what it returns; verify(...) checks it was called as expected. This is why constructor injection (OOP/SOLID lessons) matters so much: because OrderService takes its dependencies in the constructor, you can new it with mocks and no Spring context — fast, focused tests. Depend on interfaces (DIP), and mocking is trivial. A caution: don't over-mock. Mocking everything, including value objects and internal logic, produces tests that assert your implementation back at you and break on refactors — mock external/expensive collaborators, use real objects for simple ones.

Part 4 — The Testing Pyramid

            /\
           /  \        FEW  End-to-End tests (whole system, slow, brittle) — a thin top layer
          /────\
         /      \      SOME Integration tests (real DB, HTTP — Testcontainers)
        /────────\
       /          \    MANY Unit tests (fast, isolated, milliseconds) — the broad base
      /────────────\
LevelTestsSpeed / Count
UnitOne class/method in isolation (mocked deps)Milliseconds; the majority
IntegrationComponents together (real DB, real HTTP)Slower; a moderate number
End-to-EndThe whole system through its UI/APISlow, brittle; a few critical paths
🎯 Most tests should be fast unit tests (the broad base). Unit tests are cheap to write, run in milliseconds, and pinpoint failures precisely, so they form the bulk of a healthy suite. Integration tests (real database via Testcontainers, real HTTP) catch wiring and boundary problems the mocks hide, but are slower and fewer. End-to-end tests exercise the whole system but are slow and brittle, so reserve them for a handful of critical user journeys. An inverted pyramid (mostly slow E2E tests) is a common anti-pattern — the suite becomes slow and flaky, so people stop running it. (The Spring Boot testing lesson applies this exact pyramid with slice tests and Testcontainers.)

Part 5 — Parameterized Tests, Fixtures & Coverage

// PARAMETERIZED — one test, many inputs (don't copy-paste near-identical tests):
@ParameterizedTest
@CsvSource({ "2, 3, 5", "0, 0, 0", "-1, 1, 0" })
void adds(int a, int b, int expected) {
    assertThat(new Calculator().add(a, b)).isEqualTo(expected);
}

// LIFECYCLE hooks for shared setup/teardown:
@BeforeEach void setup() { ... }   // runs before EACH test (fresh state → isolation)
@AfterEach  void clean() { ... }
@BeforeAll static void once() { ... }   // once for the class (expensive shared setup)
🔍 Coverage is a guide, not a goal. Line/branch coverage tells you what code the tests execute, not whether the tests assert the right things — you can have 100% coverage with tests that check nothing meaningful. Chasing a coverage number (100% mandated) leads to trivial tests written to hit lines, not to verify behavior. Use coverage to find untested code (a genuinely useful signal), but judge test quality by whether it would catch real bugs, not by the percentage. Cover the important paths — business logic, edge cases, error handling — well; don't obsess over getters.

Part 6 — Deep Dive: TDD, and Why Testing Enables Everything Else

  • TDD (Test-Driven Development) — red/green/refactor: write a failing test, make it pass with the simplest code, then refactor safely. Its real benefit isn't the tests per se but the design pressure: code that's hard to test is usually poorly designed (tight coupling, hidden dependencies), so writing the test first pushes you toward small, decoupled, injectable units. TDD is a valuable discipline, not a mandate — many write tests after the code and get most of the benefit.
  • Tests are the safety net for change. Every design principle from earlier lessons — SOLID, refactoring toward patterns, performance optimization — depends on being able to change code without breaking it. A good test suite is what makes that safe; without it, developers fear touching working code, and the system ossifies. Testing is the enabler of maintainability.
  • Tests are documentation. Well-named tests describe what the code is supposed to do, in executable form that can't go stale (unlike comments). A new developer reads the tests to understand the behavior.
  • The cost model favors testing. A bug caught by a unit test costs seconds; the same bug in production costs hours, an incident, and user trust. Tests front-load a small cost to avoid a large one — the same "fail fast, cheaply" logic as compile-time type checking.
🔍 Senior framing: testing is not a chore appended to development but the practice that makes confident change possible — and confident change is the whole point of good design. The mature engineer writes tests that verify behavior (so they survive refactoring and catch real regressions), keeps them fast and isolated (so the suite is actually run), mocks external collaborators but not everything (so tests aren't just implementation echoes), structures the suite as a pyramid (many fast unit tests, fewer integration, a handful of E2E), and treats TDD and coverage as tools calibrated by judgment rather than rituals to satisfy. Crucially, testability is a design signal: code that's painful to test — requiring elaborate setup, static mocking, or a full framework context — is telling you it's too coupled, and the fix is better design (dependency injection, small units, interfaces) not more test machinery, which is why the SOLID and OOP lessons and this one reinforce each other. Interviewers probe testing to learn whether you understand it as the foundation of sustainable software — the thing that lets a team move fast without breaking things — rather than as a coverage metric to hit. The strongest signal you can give is designing for testability by default and writing tests that would genuinely catch the bugs that matter.

Interview Questions

Q1 (beginner): What is the Arrange-Act-Assert structure?

It's the standard three-part structure of a unit test. Arrange sets up the preconditions — construct the object under test and its inputs (and any mocks). Act performs the single operation being tested. Assert verifies the outcome — the return value, state change, or interaction. Keeping tests in this shape, with one logical action and one logical assertion per test, makes them readable and makes failures diagnostic: when a test fails you know precisely which behavior broke. A test that acts several times or asserts many unrelated things is doing too much and should be split.

Q2 (beginner): What are the properties of a good unit test (FIRST)?

Fast — runs in milliseconds so you can run the whole suite constantly; slow tests get skipped. Isolated — independent of other tests, execution order, and shared state, so one failure doesn't cascade and tests can run in any order/in parallel. Repeatable — deterministic, giving the same result every run regardless of environment (no reliance on the wall clock, randomness, network, or external services). Self-validating — it asserts pass/fail automatically with no manual inspection. Timely — written alongside or before the code rather than bolted on much later. Tests that violate these (especially flaky, slow, or order-dependent ones) erode trust in the suite until people ignore it, which is worse than having no tests.

Q3 (mid): What is mocking and when should (and shouldn't) you use it?

Mocking replaces a real dependency with a controllable fake so you can test a unit in isolation: with Mockito you mock() a collaborator, stub its responses with when(...).thenReturn(...), and assert interactions with verify(...). Use it for dependencies that are external, slow, non-deterministic, or hard to set up — databases, HTTP clients, message queues, third-party services — so the test stays fast and deterministic and focuses on the class under test. Constructor injection makes this easy: you instantiate the class with mocks and no framework. When not to mock: don't mock value objects, simple data, or the class's own internal logic — using real instances there keeps tests meaningful. Over-mocking (mocking everything) produces brittle tests that just assert the implementation back at you and break on any refactor, and they can pass while the real integration is broken. The balance is: mock the awkward boundaries, use real objects for the simple collaborators, and cover the real wiring with a few integration tests.

Q4 (mid): Explain the testing pyramid.

The testing pyramid describes the ideal proportion of test types by level. The broad base is unit tests — many of them, each testing one class/method in isolation with mocked dependencies, running in milliseconds and pinpointing failures precisely. The middle is integration tests — fewer, exercising components together with real infrastructure (a real database via Testcontainers, real HTTP) to catch wiring and boundary issues that unit tests with mocks miss. The thin top is end-to-end tests — a small number driving the whole system through its UI/API to validate critical user journeys; they're slow and brittle, so you keep them few. The shape matters because fast, numerous unit tests give quick, reliable feedback, while slow tests are expensive and flaky. The common anti-pattern is the inverted pyramid — mostly end-to-end tests — which yields a slow, unreliable suite that developers stop running, defeating the purpose. A healthy suite is mostly unit tests with just enough integration and E2E coverage.

Q5 (senior): Why should tests verify behavior rather than implementation, and what goes wrong otherwise?

Because the purpose of a test suite is to let you change code with confidence, and that only works if tests break when behavior breaks, not when implementation changes. A test that asserts observable outcomes — return values, state changes, the contract a caller relies on — keeps passing through any refactor that preserves behavior, so it protects you while you improve the internals and it fails only for real regressions, making it a trustworthy signal. A test coupled to implementation details — the exact sequence of internal method calls, private state, specific collaborators invoked in a specific way (often from over-mocking) — breaks every time you refactor even though nothing observable changed. That has three bad effects: it makes refactoring expensive and therefore discourages the very design improvements that keep code healthy (the test becomes an anchor, not a safety net); it produces false alarms that desensitize the team to failures; and it often gives false confidence, because such tests assert that the code does what it does rather than what it should do, so they can be green while the actual behavior is wrong. The discipline is to test through the public interface, assert outcomes a real caller would observe, mock only true external boundaries, and name tests by the behavior they verify. Tests are meant to describe and protect the contract; when they instead mirror the implementation, they invert from an enabler of change into an obstacle to it.

Q6 (senior): Is 100% code coverage a good goal? What does coverage actually tell you?

No — 100% coverage is a poor goal, though coverage is a useful diagnostic. Coverage measures which lines/branches your tests execute, not whether they assert anything meaningful about that execution: you can achieve 100% coverage with tests that call every method and verify nothing, or that assert trivialities, so a high number gives false assurance while real defects (wrong logic, unhandled edge cases, incorrect outputs) go uncaught. Mandating 100% also has perverse effects — developers write low-value tests purely to touch lines (testing getters, exercising unreachable branches, adding assertions that can't fail), which inflates the metric, slows the suite, and creates maintenance burden without improving quality; it can even discourage deleting dead code because that would "drop coverage." What coverage is good for is finding gaps: a report showing an important branch or error path with zero coverage is a genuine signal that something untested exists, and that's worth acting on. So the right use is to look at uncovered code and ask "should this be tested?", to cover business logic, edge cases, and error handling thoroughly, and to accept that some code (trivial accessors, generated code, simple glue) needn't be exhaustively tested. Quality of tests — would they actually catch the bugs that matter — is the real measure, and it isn't captured by a percentage. Teams often set a reasonable floor (say, on critical modules) to catch regressions in coverage, but treat the number as a guardrail, not a target to maximize.

🏋️ Exercises (attempt before opening solutions)

1 (easy): AAA test. Write a JUnit 5 + AssertJ test for a Calculator.add, following Arrange-Act-Assert, plus one asserting a divide-by-zero exception.
Solution

Arrange a Calculator, act with calc.add(2,3), assert assertThat(result).isEqualTo(5). For the exception: assertThatThrownBy(() -> calc.divide(1,0)).isInstanceOf(ArithmeticException.class). Clear structure, behavior-named tests, fluent assertions with good failure messages.

2 (easy): Parameterized. Convert three near-identical add tests into one @ParameterizedTest with a @CsvSource. Explain the benefit.
Solution

One @ParameterizedTest method taking (a, b, expected) with @CsvSource({"2,3,5","0,0,0","-1,1,0"}). Benefit: no copy-paste, each case runs and reports separately, and adding a case is one line — DRY applied to tests without losing per-case failure reporting.

3 (medium): Mock a collaborator. Test an OrderService.checkout that uses a PaymentGateway and OrderRepository, mocking both and verifying the gateway is charged. Note why constructor injection helps.
Solution

mock both dependencies, stub repo.findById, new OrderService(gateway, repo), call checkout, then verify(gateway).charge(...). Constructor injection lets you build the service with mocks and no Spring context — fast, isolated. This is the payoff of DIP/constructor injection from the SOLID/OOP lessons.

4 (medium): Make a flaky test deterministic. A test fails intermittently because it uses LocalDate.now() and a random value. Refactor the code/test to be repeatable.
Solution

Inject a Clock (and a seedable random or a supplier) into the code instead of calling now()/Math.random() directly; in the test, pass a fixed Clock.fixed(...) and a controlled value. Now the test is deterministic and repeatable. This is both a testing fix and a design improvement (injecting time/randomness is dependency inversion) — untestable code signaled a coupling to fix.

5 (hard): Critique a test suite. A suite is 100% covered but mostly end-to-end, slow, occasionally flaky, and mocks internal value objects. List what's wrong and how you'd restructure it.
Solution

Problems: inverted pyramid (mostly slow, brittle E2E) → slow, unreliable suite people avoid; flakiness → eroded trust; over-mocking value objects → tests echo implementation and break on refactor; 100% coverage is a false-comfort metric that may hide weak assertions. Restructure: push most tests down to fast, isolated unit tests (real value objects, mocked external deps), keep a moderate set of integration tests (Testcontainers) for wiring, and only a few E2E for critical journeys; remove determinism sources; and judge tests by whether they catch real bugs, using coverage only to find gaps.

6 (hard): Explain-it test. Out loud: (a) AAA and FIRST; (b) mocking's purpose and the over-mocking trap; (c) the pyramid and why coverage is a guide not a goal.

🎯 Key Takeaways

  • Structure tests as Arrange-Act-Assert; make them FIRST (Fast, Isolated, Repeatable, Self-validating, Timely).
  • Test behavior, not implementation — so tests survive refactoring and catch real regressions; name tests by the behavior verified.
  • Mock external/expensive collaborators (Mockito) to test in isolation — enabled by constructor injection/DIP; don't over-mock.
  • Follow the testing pyramid: many fast unit tests, some integration (Testcontainers), few E2E — avoid the inverted pyramid.
  • Use parameterized tests and lifecycle hooks for clean, isolated tests; treat coverage as a gap-finder, not a target.
  • Testing is the safety net that makes confident change possible — and testability is a design signal (hard-to-test = too coupled).
