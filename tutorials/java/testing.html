Testing with JUnit Advanced

JUnit 5 is the standard Java test framework, paired with AssertJ (fluent assertions) and Mockito (mocking). The same testing philosophy from the other tracks applies: many focused unit tests for logic, integration tests through real collaborators, mock only the edges. Untested Java is legacy the moment it merges.

JUnit 5 Basics

import org.junit.jupiter.api.*;
import static org.assertj.core.api.Assertions.*;

class PricingTest {
    @Test
    void appliesPercentageDiscount() {
        assertThat(Pricing.apply(100, 10)).isEqualTo(90);   // AssertJ: fluent, readable
    }
    @Test
    void rejectsNegative() {
        assertThatThrownBy(() -> Pricing.apply(-5, 10))     // asserting exceptions
            .isInstanceOf(IllegalArgumentException.class)
            .hasMessageContaining("positive");
    }
    @BeforeEach void setUp() { /* runs before each test */ }
    @AfterEach  void tearDown() { /* cleanup */ }
}
// AssertJ (assertThat(x).isEqualTo(...)) reads better than JUnit's assertEquals and gives
// richer failure messages; it's the de-facto assertion library.

Parameterized Tests — One Test, Many Cases

@ParameterizedTest
@CsvSource({ "100, 10, 90", "100, 0, 100", "50, 50, 25", "100, 100, 0" })
void discount(int price, int pct, int expected) {
    assertThat(Pricing.apply(price, pct)).isEqualTo(expected);
}
// Also: @ValueSource, @EnumSource, @MethodSource for complex inputs. Cover the BRANCHES
// (0%, 100%, boundaries), not just the happy path — that's where bugs hide.

Mockito — Mocking Collaborators (the edges)

@ExtendWith(MockitoExtension.class)
class OrderServiceTest {
    @Mock  PaymentGateway gateway;          // a test double
    @Mock  OrderRepository repo;
    @InjectMocks OrderService service;      // gateway + repo injected in

    @Test
    void placesOrderAndCharges() {
        when(gateway.charge(anyLong())).thenReturn(Result.ok());   // stub behavior
        service.place(order);
        verify(gateway).charge(order.total());                     // verify interaction
        verify(repo).save(any(Order.class));
    }
    @Test
    void propagatesDeclinedPayment() {
        when(gateway.charge(anyLong())).thenThrow(new PaymentDeclined());
        assertThatThrownBy(() -> service.place(order)).isInstanceOf(PaymentDeclined.class);
    }
}
// Mock EXTERNAL collaborators (gateways, repositories at the unit level) — not the class under
// test. Constructor injection (OOP lesson) makes this trivial: pass mocks in. Overusing mocks
// (mocking value objects, your own logic) tests the mocks, not the behavior.

Interview Questions

Q1: What should you mock, and what's the danger of over-mocking?

Mock external collaborators the unit talks to (gateways, repositories, third-party clients) so the unit is tested in isolation and deterministically. Don't mock the class under test, value objects/ records, or the database (mocking the DB validates your mock, not your SQL — use Testcontainers). Over-mocking couples tests to implementation details (every internal call becomes a verify()), making refactors break tests without catching real bugs — tests become change-detectors, not behavior guards. Prefer mocking at architectural boundaries only.

Q2: How does constructor injection improve testability?

It makes dependencies explicit and replaceable: the class receives its collaborators through the constructor, so a test can pass in mocks/fakes directly (or @InjectMocks does it). Field/setter injection hides dependencies and often requires reflection or a running container to substitute them. Constructor injection also enforces required dependencies (final fields, fail-fast) and documents the class's needs — it's the DI style Spring itself recommends, and the concrete payoff of "program to interfaces + compose" from the OOP lesson.

Q3: Why use Testcontainers instead of an in-memory DB (H2) or mocks?

Fidelity. Mocks don't run SQL at all. In-memory DBs like H2 have different SQL dialects, types, and behaviors than production Postgres/MySQL, so tests pass while real queries (dialect-specific functions, constraints, JSON columns, locking) fail in prod. Testcontainers runs the ACTUAL database engine in Docker, disposable per test run, giving high-fidelity integration tests at the cost of Docker + slower startup. Use it for repository/query tests; keep pure logic in fast unit tests.

🏋️ Practice: write JUnit 5 + AssertJ tests with assertThat and assertThatThrownBy; parameterize edge cases with @CsvSource; test an OrderService with @Mock collaborators and verify interactions; test both success and declined-payment paths; set up a Testcontainers Postgres integration test; prove constructor injection lets you pass mocks without a framework.

🎯 Key Takeaways

  • JUnit 5 + AssertJ (fluent asserts) + Mockito (mocks) is the standard stack; assertThatThrownBy for exceptions.
  • @ParameterizedTest covers branches without duplication; test error paths, not just happy paths.
  • Mock external collaborators only — never the DB (Testcontainers) or the class under test; over-mocking tests the mock.
  • Constructor injection makes mocking trivial and dependencies explicit.
  • Integration tests use real dependencies (Testcontainers); coverage finds gaps but 100% ≠ quality.
