Concurrency Advanced

Java has real OS threads and shared memory — powerful and dangerous. Unlike the single-threaded JS/Node model, Java concurrency is about coordinating shared mutable state safely. Modern Java pushes you toward high-level tools (Executors, CompletableFuture, concurrent collections) and now virtual threads. The senior core: the memory model and why "just add synchronized" isn't enough.

Threads & the Modern Way to Use Them

// Low-level (rarely write this directly):
Thread t = new Thread(() -> doWork());  t.start();  t.join();

// Preferred: Executors manage a POOL — you submit tasks, not manage threads:
try (var pool = Executors.newFixedThreadPool(8)) {   // AutoCloseable in 19+
    Future<Integer> f = pool.submit(() -> compute());
    int result = f.get();                             // blocks for the result
}
// Sizing: CPU-bound ≈ number of cores; I/O-bound → more (threads block on I/O).

The Root Problem: Races on Shared Mutable State

class Counter { int n; void inc() { n++; } }   // n++ is READ-MODIFY-WRITE — NOT atomic
// Two threads calling inc() can both read n=5, both write 6 → one increment LOST.
// Three fixes, in order of preference:

// 1. Don't share mutable state (immutability / confinement) — best.
// 2. Use atomic/concurrent utilities:
AtomicInteger n = new AtomicInteger();  n.incrementAndGet();     // lock-free, atomic
ConcurrentHashMap<K,V> map = new ConcurrentHashMap<>();          // thread-safe map
// 3. Synchronize (mutual exclusion) as a last resort:
synchronized (lock) { n++; }    // only one thread in the block at a time
⚠️ Why synchronized alone isn't the whole story: the Java Memory Model governs VISIBILITY — without synchronization, one thread's writes may never become visible to another (cached in registers/CPU caches). synchronized and volatile establish "happens-before" ordering so writes are visible. A race is about both atomicity AND visibility — which is why ad-hoc flags without volatile cause "impossible" bugs.

CompletableFuture — Async Composition

CompletableFuture<User> uf = CompletableFuture.supplyAsync(() -> loadUser(id), pool);
CompletableFuture<Prefs> pf = CompletableFuture.supplyAsync(() -> loadPrefs(id), pool);
uf.thenCombine(pf, (user, prefs) -> render(user, prefs))     // combine two async results
  .thenAccept(this::send)
  .exceptionally(ex -> { log(ex); return null; });           // handle failures
// Run independent calls concurrently and compose — the Java analog of Promise.all
// (JS async lesson). thenApply/thenCompose/allOf/anyOf mirror the promise combinators.

Virtual Threads (21+) — the Game-Changer

try (var pool = Executors.newVirtualThreadPerTaskExecutor()) {
    for (var req : requests) pool.submit(() -> handle(req));   // MILLIONS of cheap threads
}
// Virtual threads are lightweight (JVM-scheduled, ~KB each): a blocking call parks the virtual
// thread and frees the OS thread for others. So you write simple BLOCKING code that scales like
// async — no callback/reactive complexity. This closes the gap with Go goroutines and makes
// "thread-per-request" viable again for I/O-bound servers. A major modern-Java talking point.

Interview Questions

Q1: Why is n++ dangerous across threads, and what does the memory model add beyond atomicity?

n++ is read-modify-write: two threads can read the same value and both write value+1, losing an update — an atomicity failure. But there's a second issue: VISIBILITY. Without a happens-before relationship (via synchronized/volatile/atomics), a thread's write may stay in its CPU cache and never become visible to another thread, so a loop reading a "changed" flag can spin forever. The Java Memory Model defines when writes become visible; correct concurrency needs both atomicity AND visibility, which is why volatile (visibility only), atomics (both, lock-free), and synchronized (both, with mutual exclusion) exist.

Q2: When do you reach for synchronized vs Atomic vs ConcurrentHashMap vs immutability?

Immutability/confinement first — no shared mutable state, no races. AtomicInteger/AtomicReference for single-variable atomic updates (counters, CAS) — lock-free and fast. ConcurrentHashMap (and other java.util.concurrent collections) for shared data structures — engineered for concurrency, far better than Collections.synchronizedMap. synchronized/ReentrantLock only when you must protect a COMPOUND invariant across multiple fields atomically. Escalate from "don't share" up to "lock" as the last resort.

Q3: What are virtual threads and why do they matter?

Lightweight threads (21+) scheduled by the JVM onto a small pool of OS threads; a blocking operation parks the virtual thread and releases the carrier OS thread, so millions can exist. This lets you write straightforward blocking, thread-per-request code that scales like async/reactive code for I/O-bound workloads — eliminating the complexity of CompletableFuture chains and reactive frameworks for that use case. It brings Go-goroutine-style concurrency to Java and is reshaping how high-throughput Java servers are written.

🏋️ Practice: demonstrate a lost-update race on a shared counter, then fix it with AtomicInteger; make a visibility bug with a non-volatile stop flag and fix with volatile; run two independent tasks with CompletableFuture.supplyAsync + thenCombine; use an ExecutorService with Future.get; try the virtual-thread executor submitting many blocking tasks; create and then resolve a two-lock deadlock.

🎯 Key Takeaways

  • Java has real threads + shared memory — races come from unsynchronized shared mutable state.
  • Concurrency = atomicity AND visibility; the Java Memory Model + volatile/synchronized/atomics provide happens-before.
  • Prefer immutability/confinement → atomics/concurrent collections → synchronized as last resort.
  • Use Executors (pools), not raw Threads; CompletableFuture composes async work (Java's Promise.all).
  • Virtual threads (21+) make simple blocking thread-per-request code scale — a major shift.
