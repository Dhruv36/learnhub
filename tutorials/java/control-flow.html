Control Flow & Loops Foundations

What you'll master in this lesson: how to branch (if, modern switch) and repeat (the four loop forms) correctly; the off-by-one and fall-through bugs that plague loops and switches; when to use each loop; how break, continue, and labels really work; and why the enhanced for-each loop should be your default. Control flow is where logic bugs live — this lesson is about writing loops and branches that are correct the first time.

Part 1 — Branching with if / else

int score = 82;
char grade;
if (score >= 90) {
    grade = 'A';
} else if (score >= 80) {          // checked only if the first was false
    grade = 'B';
} else {
    grade = 'C';
}
// Braces are optional for single statements but ALWAYS use them — the "goto fail" class of
// bugs comes from brace-less ifs where a second line silently falls outside the condition.
⚠️ Always use braces, even for one line. Apple's infamous "goto fail" SSL vulnerability (2014) came from a brace-less if where an accidentally duplicated line ran unconditionally, disabling certificate validation. Braces cost nothing and prevent an entire class of bug. Also: if (x = 5) is a classic C typo (assignment, not comparison) — Java protects you because if requires a boolean, so it won't compile unless x is already boolean.

Part 2 — The Modern switch (arrow form)

The old C-style switch with case: and break is a bug magnet. Modern Java (14+) gives you the arrow form — no fall-through, and it can produce a value:

// Classic switch — note the mandatory break, or execution FALLS THROUGH to the next case:
switch (day) {
    case "SAT":
    case "SUN":
        type = "weekend";
        break;              // forget this and it falls into the next case — a real bug
    default:
        type = "weekday";
}

// Modern arrow switch (14+) — no fall-through, no break, and it's an EXPRESSION:
String type = switch (day) {
    case "SAT", "SUN" -> "weekend";      // multiple labels, comma-separated
    default           -> "weekday";
};                                        // note the semicolon — it's an expression

// For multi-line arms use a block with yield:
int len = switch (size) {
    case "S" -> 1;
    case "M" -> 2;
    case "L" -> { System.out.println("large"); yield 3; }
    default  -> 0;
};
🎯 Prefer the arrow switch. It eliminates the #1 switch bug (forgotten break causing fall-through), returns a value so you can assign directly, and the compiler checks exhaustiveness for enums and sealed types (Pattern Matching lesson). Use the old colon form only when maintaining legacy code.

Part 3 — The Four Loops

// 1. FOR — when you know the count / need the index:
for (int i = 0; i < 5; i++) {           // init; condition; step
    System.out.println(i);              // 0 1 2 3 4
}

// 2. ENHANCED FOR (for-each) — iterate a collection/array with NO index. YOUR DEFAULT:
int[] nums = {10, 20, 30};
for (int n : nums) {                    // "for each n in nums"
    System.out.println(n);
}

// 3. WHILE — repeat while a condition holds; may run zero times:
while (!queue.isEmpty()) {
    process(queue.poll());
}

// 4. DO-WHILE — runs the body AT LEAST ONCE, then checks:
do {
    input = readInput();
} while (input == null);                // reprompt until we get something
LoopUse whenRuns zero times?
forYou need the index or a known countYes (condition checked first)
for-eachIterating a collection/array (no index needed)Yes
whileLoop until a condition changes, count unknownYes
do-whileBody must run at least once (menus, prompts)No (runs once, then checks)
💡 Default to for-each. It removes the index, which removes off-by-one errors and <-vs-<= mistakes entirely. Only drop to an indexed for when you genuinely need the index (e.g. to modify by position, or iterate two arrays in lockstep). Streams (later) replace many loops entirely.

Part 4 — break, continue & Labels

for (int i = 0; i < 10; i++) {
    if (i == 5) break;        // exit the loop entirely
    if (i % 2 == 0) continue; // skip to the next iteration
    System.out.println(i);    // prints 1, 3 (stops at 5)
}

// Labeled break — escape NESTED loops in one jump (rare but the clean tool for it):
outer:
for (int r = 0; r < rows; r++) {
    for (int c = 0; c < cols; c++) {
        if (grid[r][c] == target) {
            System.out.println("found at " + r + "," + c);
            break outer;      // breaks BOTH loops, not just the inner one
        }
    }
}
🔍 A plain break exits only the innermost loop — a common bug when searching a 2D grid is expecting it to exit both. Labeled break is the clean solution; the alternatives (a flag variable, or extracting the search into a method that returns) are often even clearer. Avoid labels for anything but escaping nested loops.

Part 5 — Deep Dive: The Off-by-One Error

The most common loop bug in all of programming. It comes from the boundary conditions:

int[] arr = new int[5];       // valid indices are 0, 1, 2, 3, 4 (length is 5)

for (int i = 0; i <= arr.length; i++) {   // ❌ i reaches 5 → ArrayIndexOutOfBoundsException
    arr[i] = i;
}
for (int i = 0; i < arr.length; i++) {    // ✅ i stops at 4 — use <, not <=, with length
    arr[i] = i;
}
for (int i = 1; i < arr.length; i++) { }  // ❌ skips index 0 (starts at 1)

Three habits eliminate most off-by-one bugs: (1) start at 0 and use < length (not <= length); (2) prefer for-each, which has no index to get wrong; (3) when you do need indices, say the range out loud — "zero up to but not including length" — because the fencepost boundary is exactly where the bug hides.

Part 6 — Deep Dive: The Infinite Loop (and how to not ship one)

// Forgetting to advance the condition variable:
int i = 0;
while (i < 10) {
    System.out.println(i);    // i never changes → runs forever
}                             // ← missing i++

// Floating-point conditions never hitting the exact value:
for (double d = 0.0; d != 1.0; d += 0.1) { }  // ❌ 0.1 isn't exact in binary → may never equal 1.0
for (int n = 0; n < 10; n++)     { }          // ✅ integer counting is exact — loop 10 times
⚠️ Never loop on floating-point equality. Because 0.1 can't be represented exactly in binary, accumulating it may skip past 1.0 and loop forever. Count with integers and derive the double if needed (d = n * 0.1), or use < instead of !=. A deliberate infinite loop (while (true) with an inner break) is fine and common — an accidental one is the bug.

Interview Questions

Q1 (beginner): When would you use a do-while instead of a while?

When the body must run at least once before the condition is checked. A while checks first and may run zero times; a do-while runs the body, then checks, so it runs one or more times. The canonical case is input prompting or a menu: you want to show the prompt once and repeat only if the input is invalid. If "zero iterations" is a valid outcome, use while.

Q2 (beginner): What's fall-through in a switch and how does the arrow form fix it?

In the classic colon-style switch, once a matching case runs, execution continues into subsequent cases until it hits a break — "falling through". Forgetting break is a silent, common bug (though intentional fall-through is occasionally useful for grouping labels). The arrow form (case X -> ...) executes only the matched arm with no fall-through and no break needed, and it can be used as an expression that yields a value, so it's both safer and more concise.

Q3 (mid): Why prefer the enhanced for-each loop, and when can't you use it?

For-each removes the index entirely, eliminating off-by-one and boundary mistakes and reading as intent ("for each element") rather than mechanics. Prefer it for straightforward iteration. You can't use it when you need the index itself (modifying by position, parallel iteration of two collections), when you need to iterate in reverse, or when you must remove elements during iteration — for-each hides the iterator, so removing via the collection throws ConcurrentModificationException; you need an explicit Iterator and its remove(), or removeIf. In those cases use an indexed for or the iterator directly.

Q4 (mid): Explain a labeled break and give a case where it's the right tool.

A plain break exits only the innermost enclosing loop. A labeled break (break outer; where outer: precedes an outer loop) exits that labeled loop directly, escaping multiple nested levels in one statement. The right case is searching a 2D structure: once you find the target in an inner loop, you want to stop both loops — a plain break would only end the inner one and the outer would keep scanning. Labeled break expresses this cleanly, though extracting the search into a method and returning is often even clearer.

Q5 (senior): You iterate a list with for-each and call list.remove(x) inside. It throws ConcurrentModificationException. Why, and what are the correct fixes?

For-each compiles to an Iterator. Most collections track a modification count (modCount); structurally modifying the list directly (via list.remove) bumps that count, and on the next iterator.next() the iterator detects the mismatch and throws ConcurrentModificationException — a fail-fast mechanism protecting you from undefined iteration behavior. Correct fixes: (1) use the iterator's own iterator.remove() in an explicit while (it.hasNext()) loop, which updates the modCount consistently; (2) use list.removeIf(predicate) — the cleanest for conditional removal; (3) collect items to remove in a second pass; or (4) iterate a copy. Note this can also occur single-threaded — it's about structural modification during iteration, not just concurrency.

Q6 (senior): Contrast writing a data transformation as an explicit loop vs a stream. What are the trade-offs?

An explicit loop is imperative: you manage the accumulator, index, and mutation, which is maximally flexible (early exit, complex state, side effects) and often fastest for simple cases with no allocation overhead. A stream is declarative: filter/map/collect expresses what transformation you want, is more readable for pipelines, composes well, and parallelizes with one change — but it allocates, can be harder to debug, and is awkward for index-dependent logic, early termination with side effects, or checked exceptions. The senior judgment: use streams for readable transformations of collections; keep loops for tight numeric code, stateful iteration, or when an early break/exception flow is clearer. Neither is universally "better" — match the tool to the shape of the problem. (Streams get a full lesson later.)

🏋️ Exercises (attempt before opening solutions)

1 (easy): FizzBuzz. Print 1–20; for multiples of 3 print "Fizz", of 5 "Buzz", of both "FizzBuzz". This is the world's most common screening question — do it from memory.
Solution 
for (int i = 1; i <= 20; i++) {
    if (i % 15 == 0)      System.out.println("FizzBuzz");  // check 15 FIRST
    else if (i % 3 == 0)  System.out.println("Fizz");
    else if (i % 5 == 0)  System.out.println("Buzz");
    else                  System.out.println(i);
}

The trap: check % 15 (both) before % 3 and % 5, or 15 prints "Fizz" and stops. Order of conditions is the whole game.

2 (easy): Rewrite a classic switch as an arrow switch. Take a colon-style switch mapping a day abbreviation to "weekend"/"weekday" and convert it to the expression form.
Solution 
String type = switch (day) {
    case "SAT", "SUN" -> "weekend";
    default           -> "weekday";
};

No breaks, no fall-through, assigns directly. Note the trailing semicolon — it's an expression.

3 (medium): Off-by-one hunt. This code should sum an array but crashes: for (int i = 0; i <= a.length; i++) sum += a[i];. Fix it two ways — indexed and for-each.
Solution

Indexed: change <= to < (valid indices are 0..length−1). For-each: for (int n : a) sum += n; — no index, no boundary to get wrong. The for-each version can't have this bug at all, which is the argument for defaulting to it.

4 (medium): Safe removal. Given a List<Integer>, remove every even number. First write the version that throws ConcurrentModificationException, then fix it.
Solution

Throws: a for-each that calls list.remove(n) inside. Fix (cleanest): list.removeIf(n -> n % 2 == 0);. Or an explicit iterator: for (var it = list.iterator(); it.hasNext(); ) { if (it.next() % 2 == 0) it.remove(); }. The point: don't structurally modify a collection you're iterating with for-each.

5 (hard): 2D search with labeled break. Given an int[][] grid and a target, print the first (row, col) where it appears and stop scanning entirely. Then rewrite it as a method that returns the coordinates.
Solution

Labeled: an outer: label on the row loop and break outer; on match. Method version: return new int[]{r, c} on match (return exits everything naturally), null or {-1,-1} if not found. The method form is usually cleaner because return does the "escape all loops" job without a label — a good instinct: if a labeled break feels awkward, extract a method.

6 (hard): Explain-it test. Out loud: (a) the four loops and when each fits; (b) why the arrow switch is safer than the classic one; (c) the two most common infinite-loop causes and how to avoid them.
Self-check rubric

Pass (a) if you tied for/for-each/while/do-while to known-count / iteration / condition / run-once. Pass (b) if you named fall-through and expression-value. Pass (c) if you named "condition variable never advances" and "floating-point equality never hits" with the integer-counting fix. These reappear constantly in the Collections and Streams lessons.

🎯 Key Takeaways

  • Always brace your ifs; prefer the arrow switch (no fall-through, returns a value, checks exhaustiveness).
  • Four loops: for (known count/index), for-each (your default), while (condition), do-while (run at least once).
  • Off-by-one lives at the boundary — use i < length (not <=); for-each removes the bug entirely.
  • break/continue control an iteration; a labeled break escapes nested loops (or extract a method and return).
  • Avoid infinite loops: ensure the condition can become false, and never loop on floating-point equality.
  • Don't structurally modify a collection during for-each — use removeIf or the iterator's remove().
