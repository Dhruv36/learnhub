Control Flow & Methods Basics

Java's control flow is C-family and familiar; the modern additions (switch expressions, enhanced for, var) make it far less verbose than legacy Java. This lesson covers flow plus method mechanics — overloading, varargs, and the argument-passing behavior that trips people up.

Conditionals & Modern switch

if (score >= 90) grade = "A";
else if (score >= 80) grade = "B";
else grade = "C";

// switch EXPRESSION (modern) — returns a value, no fall-through, exhaustive:
String type = switch (day) {
    case MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY -> "weekday";
    case SATURDAY, SUNDAY -> "weekend";
};
// Pattern matching in switch (21+) — type + deconstruction:
String describe = switch (obj) {
    case Integer i when i > 0 -> "positive int " + i;
    case String s            -> "string of length " + s.length();
    case null                -> "null";
    default                  -> "other";
};
☕ The old switch STATEMENT falls through without break — the classic bug. The arrow -> switch EXPRESSION doesn't fall through, must be exhaustive (compile error if you miss a case on an enum/sealed type), and returns a value. Prefer it.

for (int i = 0; i < n; i++) { }
for (var item : items) { }                    // enhanced for (foreach) — prefer for iteration
while (cond) { }   do { } while (cond);
items.forEach(System.out::println);           // functional style (streams lesson)

Methods — Signatures, Overloading, Varargs

public int add(int a, int b) { return a + b; }
public double add(double a, double b) { return a + b; }   // OVERLOADING — same name, diff params
public int sum(int... nums) {                              // VARARGS — nums is an int[]
    int total = 0;
    for (int n : nums) total += n;
    return total;
}
sum(1, 2, 3);   sum();   // varargs accepts any count (incl. zero)
// Overload resolution is COMPILE-TIME by argument types (static dispatch) — distinct from
// OVERRIDING (runtime polymorphism, OOP lesson). Confusing the two is a classic interview miss.

Argument Passing (from the types lesson, applied)

void reset(int x)     { x = 0; }          // primitive copy — caller's value unchanged
void clear(List l)    { l.clear(); }      // mutates the SAME list — caller sees it
void swap(List l)     { l = new ArrayList(); }  // reassigns local copy — caller UNAFFECTED
// Pass-by-value of the reference: mutate the object yes, repoint the caller's variable no.

Interview Questions

Q1: Overloading vs overriding — mechanism and dispatch timing.

Overloading: same method name, different parameter lists, in the same class — resolved at COMPILE time by the static types of the arguments (static dispatch). Overriding: a subclass provides a new implementation of a superclass method with the same signature — resolved at RUNTIME by the actual object's type (dynamic dispatch, the basis of polymorphism, OOP lesson). Overloading is "which method"; overriding is "whose version".

Q2: Why prefer switch expressions over statements?

They return values (assignable), don't fall through (no missing-break bugs), require exhaustiveness (compile error if an enum/sealed case is unhandled — safer refactors), and support pattern matching with guards. The old statement's implicit fall-through was a decades-long bug source. Same readability/safety upgrade as modern language features generally.

Q3: What does static mean, and when is a static method appropriate?

static members belong to the class, not instances — one shared copy, accessed via ClassName.member without creating an object. Appropriate for stateless utilities (Math.max), factory methods, and constants (static final). Avoid static mutable state (it's global shared state — concurrency hazards, hard to test). Instance methods when behavior depends on per-object state.

🏋️ Practice: rewrite an if-else chain as a switch expression; write overloaded add(int,int)/add(double,double) and a varargs sum; demonstrate that clearing a passed list affects the caller but reassigning it doesn't; use pattern-matching switch on Object; create a static utility method and call it without an instance.

🎯 Key Takeaways

  • switch EXPRESSIONS (arrow, value-returning, exhaustive, no fall-through) over statements; pattern matching in 21+.
  • Enhanced-for / streams over index loops; labels for rare nested breaks.
  • Overloading = compile-time by argument type; overriding = runtime by object (don't conflate).
  • Pass-by-value of references: mutate the object, can't repoint the caller's variable.
  • Access modifiers scope visibility; static = class-level (great for stateless utilities, dangerous as mutable state).
