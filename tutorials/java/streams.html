Streams & Lambdas Intermediate

Streams brought functional data processing to Java: declare what transformation you want, not the loop that does it. Backed by lambdas and functional interfaces, they make the filter/map/reduce style idiomatic. The senior nuances: laziness, the terminal-vs-intermediate split, and knowing when a plain loop is clearer.

Lambdas & Functional Interfaces

Runnable r = () -> System.out.println("run");        // () -> body
Comparator<User> byScore = (a, b) -> a.score() - b.score();
Function<String, Integer> len = String::length;     // method reference
// A lambda implements a FUNCTIONAL INTERFACE (exactly one abstract method):
//   Function<T,R>  Consumer<T>  Supplier<T>  Predicate<T>  BiFunction<T,U,R>  Runnable
Predicate<User> active = u -> u.active();
// Method references: User::score (instance), Integer::parseInt (static), ArrayList::new (ctor)

The Stream Pipeline

List<String> topAdmins = users.stream()          // SOURCE
    .filter(u -> u.role() == Role.ADMIN)          // intermediate (lazy)
    .sorted(Comparator.comparingInt(User::score).reversed())
    .limit(10)
    .map(User::name)                              // intermediate (lazy)
    .toList();                                     // TERMINAL — triggers execution
// Intermediate ops (filter/map/sorted/limit/distinct) are LAZY — they build a plan and do
// nothing until a TERMINAL op (toList/collect/forEach/reduce/count/findFirst) runs.
// The pipeline processes elements ONE AT A TIME through all stages (not stage-by-stage over
// the whole list) — enabling short-circuiting (findFirst/limit stop early).

Collectors — Reducing to Results

import static java.util.stream.Collectors.*;
Map<Role, List<User>> byRole = users.stream().collect(groupingBy(User::role));
Map<Role, Long> counts     = users.stream().collect(groupingBy(User::role, counting()));
double avg   = users.stream().collect(averagingInt(User::score));
String names = users.stream().map(User::name).collect(joining(", "));
Map<Boolean, List<User>> parts = users.stream().collect(partitioningBy(u -> u.score() > 80));
// reduce for custom aggregation:
int total = users.stream().mapToInt(User::score).sum();   // primitive stream avoids boxing

Laziness & Short-Circuiting (interview favorite)

Optional<User> first = users.stream()
    .filter(u -> { System.out.println("checking " + u); return u.score() > 90; })
    .findFirst();     // stops at the FIRST match — doesn't check the rest
// peek() confirms laziness: nothing prints until a terminal op runs.
// anyMatch/allMatch/findFirst/limit short-circuit; count/collect are eager over the pipeline.

Primitive Streams & Parallel

IntStream.rangeClosed(1, 100).sum();                 // no Integer boxing
users.stream().mapToInt(User::score).average();      // OptionalDouble
long n = bigList.parallelStream().filter(...).count();  // uses the common ForkJoinPool
// ⚠ parallelStream helps only for LARGE data + CPU-bound, stateless, associative ops —
// and NEVER with shared mutable state (race conditions). Measure; it's often slower for
// small/IO-bound work. Same "parallelism has overhead" lesson as every concurrency topic.

Interview Questions

Q1: Intermediate vs terminal operations, and why does laziness matter?

Intermediate ops (filter, map, sorted, limit) return a new Stream and are LAZY — they only record the plan. Terminal ops (collect, forEach, reduce, count, findFirst) trigger execution and produce a result or side effect. Laziness enables (a) fusing operations so each element flows through the whole pipeline once (not N passes), and (b) short-circuiting — findFirst/limit/anyMatch stop as soon as the answer is known, so an infinite or huge source can be processed. Without a terminal op, nothing runs.

Q2: When is parallelStream a win, and when a trap?

Win: large datasets, CPU-bound, stateless, associative operations (sum, count, independent maps) where splitting overhead is dwarfed by work — it uses the common ForkJoinPool across cores. Trap: small data (overhead dominates), I/O-bound work (threads block), ordered/stateful operations, or ANY shared mutable state (data races → wrong results). It also competes for the shared pool with other parallel streams. Always benchmark; default to sequential.

Q3: Why must stream lambdas be side-effect-free?

Because ordering and execution count aren't guaranteed the way an imperative loop's are: laziness fuses stages, short-circuiting may skip elements, and parallel execution runs stages concurrently in nondeterministic order. A filter/map that mutates external state (a shared counter, a list) produces race conditions in parallel and confusing results even sequentially. Keep them pure; perform effects in a terminal forEach (and even then, prefer collecting to a result).

🏋️ Practice: build a pipeline: filter active users, sort by score desc, take 5, map to names, join; groupingBy role with counting; use peek to observe laziness and findFirst to observe short-circuiting; sum scores with an IntStream (no boxing); prove reusing a stream throws; write the same transformation as a plain loop and compare readability.

🎯 Key Takeaways

  • Lambdas implement functional interfaces (one abstract method); method references are shorthand.
  • Pipeline = source → lazy intermediates → terminal (which triggers execution); elements flow through once.
  • Laziness enables fusion + short-circuiting (findFirst/limit/anyMatch stop early).
  • Collectors (groupingBy, partitioningBy, joining, counting) reduce to maps/strings/stats.
  • Keep lambdas pure; use IntStream to avoid boxing; parallelStream only for large CPU-bound stateless work.
