Streams API Core Libraries

What you'll master in this lesson: the stream pipeline model (source → intermediate ops → terminal op) and the crucial lazy evaluation that makes it efficient; the everyday operations (filter, map, flatMap, reduce, collect); the Collectors that turn streams into lists, maps, and groupings; when parallel streams help and when they hurt; and the rules (stateless, non-interfering, single-use) that keep streams correct. Streams are how modern Java processes data — this lesson makes the pipeline model second nature.

Part 1 — The Pipeline: Source → Intermediate → Terminal

List<String> names = List.of("Ada", "Grace", "Linus", "Alan");

List<String> result = names.stream()          // 1. SOURCE — get a stream from the collection
    .filter(n -> n.length() > 3)               // 2. INTERMEDIATE — keep names longer than 3
    .map(String::toUpperCase)                  //    INTERMEDIATE — transform each
    .sorted()                                  //    INTERMEDIATE — order them
    .collect(Collectors.toList());             // 3. TERMINAL — produce a result (runs the pipeline)
// result = [ALAN, GRACE, LINUS]

// The imperative equivalent — more code, more mutable state, less clear intent:
List<String> out = new ArrayList<>();
for (String n : names) if (n.length() > 3) out.add(n.toUpperCase());
Collections.sort(out);

Part 2 — Laziness: Why Streams Are Efficient

// Intermediate operations do NOTHING until a terminal operation runs. Watch the order:
Stream.of("a", "bb", "ccc", "dddd")
    .filter(s -> { System.out.println("filter " + s); return s.length() > 1; })
    .map(s -> { System.out.println("map " + s); return s.toUpperCase(); })
    .findFirst();      // TERMINAL — and it stops as soon as it finds ONE result

// Output:  filter a, filter bb, map bb   → then STOPS (short-circuit!)
// It does NOT process ccc or dddd. Each element flows through the whole pipeline one at a time,
// and findFirst short-circuits. This is why streams over huge/infinite sources can be efficient.
🎯 Streams are lazy and element-at-a-time, not stage-at-a-time. A naive mental model imagines filter processing everything, then map processing everything — wrong. Each element is pulled through the entire pipeline, and short-circuiting terminals (findFirst, anyMatch, limit) stop early. This laziness means you can stream infinite sources (Stream.iterate, Stream.generate) as long as something limits them, and it avoids materializing intermediate collections. It's the key efficiency property of the API.

Part 3 — The Core Operations

// filter — keep elements matching a Predicate:
nums.stream().filter(n -> n % 2 == 0)

// map — transform each element (T → R):
names.stream().map(String::length)              // Stream<String> → Stream<Integer>

// flatMap — flatten nested structure (Stream of Lists → Stream of elements):
listOfLists.stream().flatMap(List::stream)      // [[1,2],[3]] → 1,2,3

// reduce — fold to a single value:
nums.stream().reduce(0, Integer::sum)           // sum; identity 0, accumulator +

// sorted / distinct / limit / skip:
nums.stream().distinct().sorted().limit(5)

// terminals producing answers:
long c = nums.stream().filter(n -> n > 0).count();
boolean any = nums.stream().anyMatch(n -> n < 0);
Optional<Integer> max = nums.stream().max(Integer::compare);   // Optional (may be empty)
🔍 map vs flatMap: map turns each element into exactly one element; flatMap turns each element into a stream and concatenates them, so it flattens one level of nesting. Use flatMap when each element expands into many (a list of a customer's orders → all orders), or to unwrap Stream<Optional<T>>. It's the operation people forget exists and then reinvent badly.

Part 4 — Collectors: Turning Streams into Results

import static java.util.stream.Collectors.*;

// To collections:
List<String> list = stream.collect(toList());        // or stream.toList() (16+)
Set<String>  set  = stream.collect(toSet());
Map<Long,User> byId = users.stream().collect(toMap(User::id, u -> u));

// Joining strings:
String csv = names.stream().collect(joining(", ", "[", "]"));   // "[Ada, Grace]"

// GROUPING — the powerhouse: partition elements into a Map by a key:
Map<Dept, List<Employee>> byDept =
    employees.stream().collect(groupingBy(Employee::dept));

// Grouping WITH a downstream collector (count / sum / map per group):
Map<Dept, Long> countByDept =
    employees.stream().collect(groupingBy(Employee::dept, counting()));
Map<Dept, Double> avgSalary =
    employees.stream().collect(groupingBy(Employee::dept, averagingDouble(Employee::salary)));
🎯 groupingBy (optionally with a downstream collector like counting(), summingInt(), mapping(), toList()) is the single most powerful stream tool — it replaces pages of manual multi-map building. The word-count and "group orders by customer, sum totals" tasks that dominate data code become one expression. toMap throws on duplicate keys unless you supply a merge function — a common gotcha.

Part 5 — Parallel Streams: Power and Peril

// One method turns a stream parallel, splitting work across the common ForkJoinPool:
long count = hugeList.parallelStream()
    .filter(this::expensiveCheck)
    .count();

// When it HELPS: large datasets + CPU-bound, independent, stateless work + a splittable source
//   (ArrayList/arrays split well; LinkedList doesn't).
// When it HURTS or BREAKS:
//   - small data (thread coordination overhead > the work)
//   - I/O-bound work (threads block; use async instead)
//   - shared mutable state (RACE CONDITIONS — the results are wrong, not just slow)
//   - order-dependent logic (parallel reorders; forEachOrdered costs the benefit)
⚠️ Parallel streams are a trap for the unwary. They're trivially easy to turn on (parallelStream()) and easy to misuse. The lambdas you pass MUST be stateless and non-interfering — mutating a shared ArrayList or counter from a parallel stream is a data race producing corrupt results. They share one JVM-wide ForkJoinPool.commonPool, so a blocking or long parallel stream can starve every other one. Rule of thumb: default to sequential; reach for parallel only for large, CPU-bound, independent workloads on splittable sources, and measure. In server code (already handling requests concurrently), parallel streams are usually the wrong tool.

Part 6 — Deep Dive: The Stream Contract (stateless, non-interfering, once)

  • Streams are single-use. Once a terminal operation runs, the stream is consumed — reusing it throws IllegalStateException. A stream is not a collection you can re-iterate; create a fresh one from the source each time.
  • Lambdas must be non-interfering. A stream's operations must not modify the source collection during the pipeline (ConcurrentModificationException or worse). Transform into a new result; don't mutate what you're streaming.
  • Prefer stateless, side-effect-free lambdas. map/filter functions should compute from their input only, not read/write external mutable state — this is required for correctness under parallelism and clarity always. Accumulate with reduce/collect, not by mutating a captured variable.
  • Don't overuse forEach for accumulation. stream.forEach(x -> list.add(x)) is an anti-pattern — use collect(toList()). forEach is for genuine side effects (printing), and even then forEachOrdered if order matters in parallel.
🔍 Senior framing: streams are a declarative pipeline, and their guarantees (laziness, potential parallelism, fusion of operations) only hold if your lambdas are pure — stateless and side-effect-free. The reason "don't mutate shared state in a stream" is stated so firmly is that it silently breaks under parallelism and undermines the whole model. Think of a stream as transforming data into a new result, never as a fancy loop that pokes at external variables. When you need debugging visibility, peek exists, but a stream whose correctness depends on side effects is a stream that should have been a loop.

Interview Questions

Q1 (beginner): What are the three parts of a stream pipeline?

A source, zero or more intermediate operations, and one terminal operation. The source produces the stream (collection.stream(), Stream.of). Intermediate operations (filter, map, sorted…) return a new stream and are lazy — they're recorded but not executed. The terminal operation (collect, count, forEach, reduce) triggers execution and produces a result or side effect, consuming the stream. No terminal op → nothing runs.

Q2 (beginner): What's the difference between map and filter?

filter takes a Predicate and keeps only elements that match, so it can reduce the number of elements but doesn't change their type. map takes a Function and transforms each element into another value (possibly a different type), producing exactly one output per input, so the count stays the same but the type/value changes. You combine them: filter to select, map to transform. (And flatMap when each element expands into many.)

Q3 (mid): Explain stream laziness and short-circuiting with an example.

Intermediate operations don't execute when declared — they only run when a terminal operation pulls elements through the pipeline, and each element flows through all stages before the next element starts (element-at-a-time), not stage-by-stage. Short-circuiting terminals like findFirst, anyMatch, and limit stop as soon as they have their answer. Example: stream.filter(p).map(f).findFirst() on a long list processes elements one by one and stops at the first that survives the filter — it never touches the rest. This makes streams efficient (no intermediate collections, early termination) and lets them work over infinite sources bounded by limit.

Q4 (mid): What is groupingBy and how do downstream collectors extend it?

Collectors.groupingBy(classifier) partitions stream elements into a Map<K, List<T>> keyed by the classifier function — e.g. employees grouped by department. Its power comes from the two-argument form groupingBy(classifier, downstream), where the downstream collector transforms each group's list: counting() gives a Map<K,Long> of group sizes, summingInt/averagingDouble aggregate a field per group, mapping(f, toList()) transforms elements before collecting, and you can nest groupingBy for multi-level grouping. This replaces large amounts of manual map-building and is the workhorse for analytics-style data processing.

Q5 (senior): When are parallel streams appropriate, and what are the failure modes?

Parallel streams help only when several conditions hold together: a large dataset, work that's CPU-bound and independent per element, lambdas that are stateless and non-interfering, and a splittable source (arrays and ArrayList split cheaply; LinkedList, Stream.iterate, and I/O sources don't). When those hold, splitting across cores via the fork-join framework can give near-linear speedups. Failure modes: on small data the coordination/splitting overhead exceeds the work, so it's slower; for I/O-bound work threads block and you get no benefit (and starve the pool); with shared mutable state you get data races and silently wrong results, not just slowness; and order-sensitive processing is broken unless you pay for forEachOrdered. Critically, all parallel streams share the single JVM-wide ForkJoinPool.commonPool, so one slow or blocking parallel stream degrades every other one — a real problem in servers, which are already parallel at the request level. The senior stance: sequential by default; use parallel only for big CPU-bound independent batch work, on a splittable source, and only after measuring — often a custom executor or a different concurrency model is better.

Q6 (senior): Why is mutating external state from stream operations discouraged even when it "works" sequentially?

Because it violates the contract the entire Streams model relies on — that pipeline operations are stateless and non-interfering — and that contract is what enables laziness, operation fusion, and parallelism. A pipeline that mutates a captured variable or shared collection happens to produce the right answer sequentially, but: (1) it silently breaks if someone adds .parallel(), because concurrent unsynchronized mutation is a data race giving corrupt or lost results; (2) it couples correctness to execution order and the number of times an element is processed, which the stream is free to change; (3) it defeats readability — the point of a stream is to express a transformation declaratively, and a side-effecting stream is just an obfuscated loop; and (4) it can interfere with the source (ConcurrentModificationException) if the mutation touches what you're streaming. The correct approach is to express accumulation as a reduce or a Collector (which are designed to combine partial results safely, including in parallel) and reserve genuine side effects for a deliberate forEach. "A stream transforms input into output; it should not reach outside itself" — following that keeps the code correct under every execution strategy the API is allowed to choose.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Basic pipeline. From a List<String>, produce a sorted list of the uppercase forms of names longer than 4 characters, using a stream.
Solution

names.stream().filter(n -> n.length() > 4).map(String::toUpperCase).sorted().toList();. Read it top to bottom as a description of the transformation — that readability is the point versus the equivalent imperative loop with a mutable list and explicit sort.

2 (easy): Sum and count. Given List<Integer>, compute the sum of the even numbers and how many there are, with streams.
Solution

Sum: nums.stream().filter(n -> n % 2 == 0).mapToInt(Integer::intValue).sum(); (note mapToInt → an IntStream with a direct sum()). Count: nums.stream().filter(n -> n % 2 == 0).count();. Using the primitive IntStream avoids boxing and gives numeric terminals like sum/average.

3 (medium): flatMap. Given List<List<String>>, produce a single flat, distinct, sorted list of all strings.
Solution

lists.stream().flatMap(List::stream).distinct().sorted().toList();. flatMap turns each inner list into a stream and concatenates them into one flat stream; then distinct/sorted operate on the flattened elements. Trying this with map would give you a Stream<Stream<String>> — the wrong shape.

4 (medium): groupingBy. Given a list of orders each with a customerId and amount, build a map of customerId → total amount, and a map of customerId → order count.
Solution

Total: orders.stream().collect(groupingBy(Order::customerId, summingDouble(Order::amount)));. Count: orders.stream().collect(groupingBy(Order::customerId, counting()));. The downstream collector (summingDouble/counting) aggregates each group in one expression — this replaces a manual loop building and updating a map.

5 (hard): Spot the parallel bug. This is meant to collect results but is wrong: List<R> out = new ArrayList<>(); data.parallelStream().forEach(x -> out.add(f(x)));. Explain the bug and fix it.
Solution

ArrayList isn't thread-safe, and a parallel stream calls the lambda from multiple threads simultaneously, so concurrent out.add(...) calls race — you get corrupted state, lost elements, or an exception (a nondeterministic bug). Fix: List<R> out = data.parallelStream().map(this::f).collect(toList()); — the collector is designed to combine partial results safely across threads. Never mutate shared state from a (parallel) stream; express the accumulation as a collector.

6 (hard): Explain-it test. Out loud: (a) the pipeline model and lazy element-at-a-time evaluation; (b) map vs flatMap vs reduce; (c) when parallel streams help and their main failure modes.
Self-check rubric

Pass (a) if you described source/intermediate/terminal + laziness + short-circuit. Pass (b) if you said map is 1:1, flatMap flattens 1:many, reduce folds to one value. Pass (c) if you named large CPU-bound independent splittable work as the win and races/overhead/shared-pool as the failures. These are the exact stream questions in senior interviews and set up production data code in the Spring Boot track.

🎯 Key Takeaways

  • A stream pipeline = source → intermediate (lazy) → terminal (eager); no terminal, nothing runs.
  • Streams are lazy and element-at-a-time, with short-circuiting terminals — efficient and able to handle infinite sources.
  • Core ops: filter (select), map (1:1 transform), flatMap (flatten 1:many), reduce (fold), plus collect.
  • Collectors — especially groupingBy with downstream collectors — turn streams into lists, maps, and aggregations declaratively.
  • Parallel streams help only for large, CPU-bound, independent, splittable work with pure lambdas — otherwise they're slower or corrupt; default to sequential and measure.
  • Keep stream lambdas stateless, non-interfering, side-effect-free; streams are single-use; accumulate with collect/reduce, not shared mutation.
