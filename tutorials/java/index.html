Syntax, Types & Variables Basics

Java is statically typed, compiled to bytecode, and runs on the JVM ("write once, run anywhere"). It's verbose but predictable — the type system catches errors at compile time. The concept that underlies every later gotcha: the primitive vs reference distinction, and how variables hold one or the other.

Compile → Run

// Hello.java  (public class name must match the filename)
public class Hello {
    public static void main(String[] args) {   // the JVM entry point
        System.out.println("Hello, World!");
    }
}
// javac Hello.java  → Hello.class (bytecode)   →   java Hello  (JVM runs it)
// Modern JDKs also run single files directly:  java Hello.java

Primitives vs References — the Core Distinction

int x = 42;                 // PRIMITIVE — the variable holds the VALUE directly
int y = x;                  // copies the value; y and x are independent
String s = "hi";            // REFERENCE — the variable holds a POINTER to an object
int[] a = {1, 2, 3};        // reference; a "points to" the array object
int[] b = a;                // copies the REFERENCE — a and b point to the SAME array
b[0] = 99;                  // a[0] is now 99 too!

☕ Java is always pass-by-value — but for reference types, the VALUE passed is the reference (pointer). So a method can mutate the object a parameter points to (visible to the caller) but can't make the caller's variable point elsewhere. The #1 "is Java pass by reference?" interview trap — the answer is "no, pass-by-value of a reference".

Types, Wrappers & Autoboxing

Integer boxed = 42;        // autoboxing: int → Integer (wrapper object)
int prim = boxed;          // unboxing: Integer → int
List<Integer> list = ...;  // collections hold OBJECTS, not primitives → wrappers required

// The autoboxing traps:
Integer a = 1000, b = 1000;
a == b;              // false! == compares REFERENCES for objects (a,b are different Integer objects)
a.equals(b);         // true — .equals compares VALUE. ALWAYS use .equals for objects.
Integer c = 100, d = 100;
c == d;              // true — the Integer cache (-128..127) reuses objects; NEVER rely on this
Integer n = null; int z = n;   // NullPointerException on unboxing null

var, final & Strings

var count = 10;              // local type inference (still statically typed as int)
final int MAX = 100;        // cannot be reassigned (Java's const)
String s = "Hello";
s.concat(" World");         // Strings are IMMUTABLE — this returns a NEW string, s unchanged
String r = s.concat(" World");   // capture the result
// Build strings in loops with StringBuilder, NOT += (which creates a new String each time — O(n²)):
var sb = new StringBuilder();
for (var w : words) sb.append(w);
String out = sb.toString();
"Hi %s, %d items".formatted(name, n);   // formatting;  text blocks: """ multi-line """

Interview Questions

Q1: Is Java pass-by-reference? Prove your answer.

No — always pass-by-value. For primitives, the value is copied. For objects, the value copied is the REFERENCE (pointer): the method gets its own copy of the pointer to the same object. So it can mutate the object (caller sees it) but reassigning the parameter to a new object doesn't affect the caller's variable. Demo: a method that does list.add(x) affects the caller; a method that does list = new ArrayList() does not.

Q2: Why does Integer a = 1000, b = 1000; a == b give false but 100 give true?

== compares references for objects. Autoboxing 100 uses the Integer cache (−128..127), returning the SAME cached object → true. 1000 is outside the cache, creating two distinct objects → false. The lesson: never use == for object equality; always .equals(). Relying on the cache is a latent bug that flips based on values.

Q3: Why are Strings immutable, and what are the consequences?

Immutability enables safe sharing (string pool interning), thread-safety, and use as HashMap keys (hashCode is stable). Consequences: every "modification" allocates a new String, so concatenation in loops is O(n²) — use StringBuilder; and sensitive data (passwords) should be char[] you can wipe, since Strings linger in memory. The string pool (literals are interned) is why "a"=="a" is true but new String("a")==new String("a") is false.

🏋️ Practice: demonstrate reference sharing (b=a; mutate b; observe a); prove Integer== differs at 100 vs 1000 and fix with equals; trigger an NPE by unboxing null; benchmark String += vs StringBuilder in a loop; write a method proving Java can mutate but not reassign a passed object.

🎯 Key Takeaways

  • Primitives hold values (copied by value); references hold pointers (copied by reference) — the core distinction.
  • Java is always pass-by-value; for objects, the reference is passed by value (mutate yes, reassign no).
  • == compares references for objects — use .equals(); beware the Integer cache (−128..127).
  • Strings are immutable (pool, thread-safe, hashable) — StringBuilder for loops.
  • var infers types (still static); final = const; BigDecimal for money.
