How Java Runs: JVM, JDK & Bytecode Foundations

What you'll master in this lesson: what actually happens between the Java source you type and a program running on your machine; why Java's "write once, run anywhere" promise exists and how the JVM delivers it; the difference between the JDK, JRE, and JVM (a guaranteed interview question); what bytecode is and why it matters; and how to compile and run your first program while understanding every step. No prerequisites — this is lesson one, and the mental model you build here explains everything from performance to "why is my classpath broken" later in the track.

Part 1 — The Problem Java Was Built to Solve

In 1995, shipping software meant compiling your C or C++ code separately for every target: one binary for Windows on Intel, another for Solaris on SPARC, another for a Mac. A pointer bug or a platform-specific quirk could crash the whole machine. Java's designers at Sun Microsystems made a radical bet: compile once to a portable intermediate format, then let a small program on each machine execute that format. That program is the Java Virtual Machine (JVM), and the portable format is bytecode.

The slogan was "Write Once, Run Anywhere" (WORA). It works because your code never talks to the operating system directly — it talks to the JVM, and the JVM (a different build for each platform) translates. You write to one stable target; Oracle, the OpenJDK community, Amazon, and others maintain the JVMs that make it run on everything from a laptop to a cloud server to an Android phone's ancestors.

💡 The mental model that pays off all track long: Java is compiled (source → bytecode) and interpreted/JIT-compiled (bytecode → machine code at runtime). It is not "compiled like C" and not "interpreted like Python" — it's a hybrid, and almost every Java performance characteristic (slow startup, fast steady-state, warmup) falls out of this fact.

Part 2 — The Journey of a Program, Step by Step

Write Hello.java and run it. Here is the complete journey — learn it in outline now, because the JVM, Class Loading, and Performance lessons later zoom into almost every step:

1. SOURCE          You write Hello.java — plain text, human-readable.
                   public class Hello { public static void main(String[] a){...} }

2. COMPILE         `javac Hello.java` → Hello.class
                   The compiler checks types/syntax and emits BYTECODE (not machine code!).
                   Bytecode is the JVM's instruction set — platform-independent.

3. LAUNCH          `java Hello` starts a JVM process and asks it to run class Hello.

4. CLASS LOADING   The JVM's class loader finds Hello.class (on the classpath),
                   verifies the bytecode is safe, and loads it into memory.

5. INTERPRET       The JVM starts INTERPRETING bytecode instruction by instruction.
                   Correct, but relatively slow.

6. JIT COMPILE     The JVM watches which methods run often ("hot" code) and the
                   Just-In-Time compiler translates those to native machine code,
                   optimized for THIS CPU. Now it runs at near-C speed.

7. EXECUTE + GC    Your program runs. Meanwhile the Garbage Collector reclaims
                   memory you no longer reference — no manual free()/delete.

Three facts from this journey shape the whole track:

  1. Bytecode is the real artifact. A .class (or a .jar, which is just a zip of .class files) is what you ship and what the JVM runs. You can even read it: javap -c Hello disassembles bytecode back to readable instructions.
  2. The JVM is a runtime, not just a translator. It loads classes, verifies safety, manages memory, runs the GC, and JIT-compiles hot paths. "The JVM" is where most of Java's engineering depth lives.
  3. Startup is slow, steady-state is fast. Interpretation first, then JIT warmup — why a short-lived Java CLI feels sluggish but a long-running server is blazing. (This is exactly the gap that GraalVM native images and Project CRaC close — a topic in the Spring Boot production lesson.)

Part 3 — JDK vs JRE vs JVM (the classic interview question)

These three acronyms trip up beginners constantly, and interviewers ask precisely because the answer reveals whether you understand the platform. They nest like Russian dolls:

TermStands forContainsYou need it to…
JVMJava Virtual MachineThe engine that executes bytecode (loader, interpreter, JIT, GC)Run bytecode (it's the abstract spec + concrete implementation)
JREJava Runtime EnvironmentJVM + the standard class libraries (java.lang, java.util, …)Run Java programs
JDKJava Development KitJRE + development tools (javac, javap, jar, jshell, debugger)Develop Java programs

One sentence for the interview: "The JDK is what you develop with, the JRE is what you run with, and the JVM is the engine inside the JRE that actually executes the bytecode." Since Java 11, Oracle stopped shipping a standalone JRE — you download a JDK and can create a slimmed runtime with jlink if you only need to run. So in practice today you install a JDK.

🔧 Which JDK to install? Prefer a current LTS (Long-Term Support) release — Java 21 (LTS) is the modern default, with Java 17 still common. Good free distributions: Eclipse Temurin (Adoptium), Amazon Corretto, or Azul Zulu. All are builds of the same OpenJDK source; pick one, install it, and run java -version to confirm.

Part 4 — Write, Compile, Run (follow along)

Don't just read this — do it. Install a JDK, open any editor, and follow all four steps.

Step 1: the minimum program

// Hello.java  — the filename MUST match the public class name
public class Hello {
    public static void main(String[] args) {   // the JVM's entry point
        System.out.println("Hello, World!");
    }
}

Every token here earns its place, and later lessons dissect each; for now: public class Hello must match the filename Hello.java (a rule beginners hit immediately); public static void main(String[] args) is the exact signature the JVM looks for to start — static so it runs without creating an object, String[] args for command-line arguments; System.out.println prints a line to standard output.

Step 2: compile to bytecode

javac Hello.java      # produces Hello.class (bytecode) in the same folder
ls                    # you now see Hello.java AND Hello.class

Open Hello.class in a text editor — it's binary gibberish, not readable text. That's bytecode. If javac reports errors, it caught them at compile time, before the program ever ran — one of Java's core strengths over dynamically-typed languages.

Step 3: run on the JVM

java Hello            # NOTE: no .class extension — you name the CLASS, not the file
# → Hello, World!

# Modern JDKs (11+) can also run a single source file directly, compiling in memory:
java Hello.java       # no separate javac step — great for quick experiments

Step 4: look inside the bytecode

javap -c Hello        # disassemble the .class back to readable JVM instructions

You'll see instructions like getstatic, ldc ("load constant"), and invokevirtual. You will rarely read bytecode in your career — but seeing it once makes "Java compiles to bytecode that the JVM runs" concrete instead of abstract. That's the whole point of this step.

🧪 Even faster experiments: JShell. Run jshell (ships with the JDK) and type System.out.println(2 + 2) — it evaluates immediately, no class or main needed. Use it throughout this track to test a snippet in seconds.

Part 5 — Deep Dive: Interpreted, Compiled, or Both?

"Is Java compiled or interpreted?" is a trick interview question — the correct answer is both, and explaining why demonstrates real understanding:

  • Ahead-of-time compilation happens once, from source to bytecode (javac). This catches type errors and produces the portable artifact.
  • At runtime, the JVM interprets bytecode at first — walking instructions one by one. This starts instantly but runs slower.
  • The JIT (Just-In-Time) compiler profiles the running program, identifies "hot" methods and loops, and compiles those to native machine code — with optimizations an ahead-of-time compiler couldn't do because the JIT sees real runtime behavior (which branches actually get taken, which types actually appear). This is why Java's steady-state performance rivals C++.

The consequence is warmup: a Java server is slowest in its first seconds and fastest after the JIT has optimized the hot paths. Benchmarking Java naively (timing the first run) gives wildly misleading numbers — a real trap covered in the Performance lesson.

🔍 Senior-level nugget: the JVM is a specification, and HotSpot (in OpenJDK) is the dominant implementation, but not the only one. Bytecode is also the target for other languages — Kotlin, Scala, Groovy, and Clojure all compile to JVM bytecode and interoperate with Java. "The JVM ecosystem" is bigger than the Java language, and that portability of the bytecode format is why.

Part 6 — Deep Dive: The Classpath (where the JVM finds code)

When you write java Hello, how does the JVM find Hello.class — and the hundreds of library classes a real program uses? The answer is the classpath: an ordered list of locations (folders and .jar files) the class loader searches. Misusing it is the source of the two most-feared beginner errors:

ErrorMeaningCause
ClassNotFoundException / NoClassDefFoundErrorThe JVM couldn't find a class it needsThe class/jar isn't on the classpath
Error: Could not find or load main classThe launcher can't locate your entry classWrong class name, wrong folder, or classpath missing .
# Set the classpath explicitly (folders + jars, separated by : on Unix, ; on Windows):
java -cp .:lib/gson.jar com.example.App
#      └── current dir + a library jar; App is found because its package folder is under "."

In real projects you almost never manage the classpath by hand — build tools (Maven, Gradle) and the module system do it for you (the Build Tools lesson). But understanding that "the JVM searches an ordered list of locations for .class files" turns these cryptic errors from mysteries into a checklist.

Part 7 — Java's Evolution in Six Lines (interview context)

1995   Java 1.0 at Sun — applets, WORA, automatic memory management
2004   Java 5 — generics, enums, annotations, for-each (a huge leap)
2014   Java 8 — lambdas + Streams + the new Date/Time API (the modern era begins)
2017   Java 9 — the module system (Project Jigsaw); 6-month release cadence begins
2021   Java 17 (LTS) — records, sealed types, pattern matching maturing
2023   Java 21 (LTS) — virtual threads (Loom), pattern matching for switch, the modern baseline

Two takeaways: since Java 9, a new version ships every six months, with an LTS every 2–3 years that companies actually standardize on (8, 11, 17, 21) — so "what Java version?" usually means an LTS. And Java 8 is the great dividing line: code before it is imperative and verbose; code after it (lambdas, streams, var, records) is dramatically more concise. This track teaches modern, post-8 Java throughout.

Interview Questions

Q1 (beginner): What is the JVM and why does it exist?

The Java Virtual Machine is the runtime engine that executes Java bytecode. It exists to deliver "write once, run anywhere": Java source compiles to platform-independent bytecode, and a platform-specific JVM on each machine translates that bytecode to native operations. So the same .class/.jar runs unchanged on Windows, Linux, or macOS. The JVM also provides memory management (garbage collection), bytecode verification for safety, and JIT compilation for performance — it's far more than a translator.

Q2 (beginner): Explain the difference between JDK, JRE, and JVM.

They nest. The JVM is the engine that runs bytecode. The JRE is the JVM plus the standard class libraries — everything needed to run a Java app. The JDK is the JRE plus development tools (javac compiler, debugger, javap, jar, JShell) — everything needed to develop. In short: JDK to build, JRE to run, JVM inside the JRE does the executing. Note that since Java 11 there's no separate JRE download; you install a JDK.

Q3 (mid): Is Java compiled or interpreted? Defend your answer.

Both. Java is compiled ahead of time from source to bytecode (javac), and at runtime the JVM interprets that bytecode initially, then JIT-compiles hot methods to native machine code. So it's a hybrid: the ahead-of-time step gives portability and type safety; the runtime interpret-then-JIT model gives fast startup relative to full AOT and near-native steady-state speed. This hybrid is why Java has "warmup" — the first executions are interpreted and slow, later ones run optimized native code.

Q4 (mid): What is bytecode, and what are the practical consequences of Java using it?

Bytecode is the JVM's platform-independent instruction set — the output of javac, stored in .class files. Consequences: (1) portability — one artifact runs on any JVM; (2) a shared platform — other languages (Kotlin, Scala, Groovy) compile to the same bytecode and interoperate with Java; (3) runtime optimization — because the JVM sees bytecode plus real execution behavior, the JIT can optimize better than a purely static compiler in some cases; (4) tooling — bytecode can be inspected (javap), instrumented (profilers, APM agents), and manipulated (frameworks like Spring and mocking libraries use bytecode generation).

Q5 (senior): Explain JIT warmup and why it complicates benchmarking and short-lived processes.

The JVM starts by interpreting bytecode, profiles execution, and only after a method becomes "hot" (crosses invocation/loop thresholds) does the JIT compile it to optimized native code — sometimes speculatively, deoptimizing if assumptions break. So performance rises over the first seconds/minutes (warmup). This complicates benchmarking: timing the first iterations measures the interpreter, not steady state, so you must warm up and use tools like JMH that account for it. It also penalizes short-lived processes (CLIs, serverless functions) that die before the JIT pays off — which is why AOT solutions like GraalVM native image (instant startup, no warmup, at the cost of some peak throughput and dynamic features) and checkpoint/restore (Project CRaC) exist for those workloads.

Q6 (senior): A jar runs on your machine but throws NoClassDefFoundError in production. Walk through the diagnosis.

NoClassDefFoundError means a class that was present at compile time is missing (or failed to initialize) at runtime — distinct from ClassNotFoundException (an explicit reflective lookup failed). Diagnosis: first confirm the classpath/module path in the failing environment actually includes the jar containing that class — a dependency present locally but not packaged/deployed is the classic cause (fat-jar vs thin-jar packaging, a provided-scope dependency, or a container missing a lib). Second, rule out version mismatch: a different version of a jar on the prod classpath may lack the class or method (jar hell), so check for duplicate/conflicting jars. Third, if the class exists but a static initializer threw, the first load failed and subsequent loads report NoClassDefFoundError — look earlier in the logs for the original ExceptionInInitializerError. The root fix is almost always making the build tool produce a deployment artifact whose classpath matches what ran locally.

🏋️ Exercises (do them in order — solutions inside, but attempt first)

1 (easy): Install and verify. Install an LTS JDK (Temurin 21). From a terminal, run java -version and javac -version. Confirm both print a 21.x version. Then launch jshell and evaluate "Java " + 17 + 4 — predict the output before you press Enter.
Solution & what matters

java -version and javac -version should both report 21 (if javac is missing you installed a JRE-only bundle — reinstall a JDK). JShell prints "Java 174" — not "Java 21" — because + evaluates left-to-right: "Java " + 17 is already a String, so + 4 concatenates "4". String concatenation vs numeric addition on + is a genuine beginner trap you just inoculated yourself against.

2 (easy): The full compile/run cycle by hand. Create Hello.java with the classic program, compile with javac, list the folder to see the .class, run with java Hello. Then deliberately rename the file to hello.java (lowercase) and recompile — observe the error.
What you should observe

Compiling hello.java containing public class Hello fails with "class Hello is public, should be declared in a file named Hello.java". The public class name and filename must match exactly, case included. (Trick: a class that is not public can live in a differently-named file — but stick to matching them.) You've now met the compiler enforcing a real rule at compile time, before any execution.

3 (medium): Read the bytecode. Run javap -c Hello on your compiled class. Find the instruction that loads the "Hello, World!" string and the one that calls println. You don't need to understand every line — just locate those two.
Solution

You'll see something like ldc #N // String Hello, World! (load constant — the string) followed by invokevirtual #M // Method java/io/PrintStream.println (the call). getstatic before them fetches System.out. The takeaway isn't fluency in bytecode — it's that your one-line program became a small sequence of stack-machine instructions the JVM executes. Bytecode is real and inspectable.

4 (medium): Single-file and JShell. Run your program with java Hello.java (no compile step) and confirm it works. Then in JShell, define a method int sq(int x){ return x*x; } and call sq(9). Note how neither required a class with main.
Why this matters

Single-file source launch (11+) and JShell (9+) exist to kill the friction of the classic ceremony for learning and prototyping. In real projects you'll use a build tool and full classes, but for "does this snippet work?" these two are the fastest path — use them constantly through this track.

5 (hard): Break the classpath, then fix it. Put Hello.java in a package: add package com.example; as the first line, move the file to a com/example/ folder, recompile, and try to run it. Get it working with the correct package name and classpath.
What you should find

With a package declaration, the class's full name is com.example.Hello, and its .class must live in com/example/. Compile from the project root: javac com/example/Hello.java. Run with the fully qualified name and the root on the classpath: java -cp . com.example.Hello. Running java Hello or running from inside the folder fails — the package structure and classpath must agree. This is exactly the model build tools automate, and understanding it demystifies "package/classpath" errors forever.

6 (hard): Explain-it test. Out loud, with no notes, explain: (a) the full journey from .java to running program, naming bytecode and the JIT; (b) the JDK vs JRE vs JVM distinction in one sentence each; (c) why Java has "warmup" and what that implies for a serverless function. Stumble anywhere → reread that part.
Self-check rubric

You pass (a) if you said source → javac → bytecode → class loading → interpret → JIT to native. You pass (b) if you framed them as nesting (JDK ⊃ JRE ⊃ JVM) with develop/run/execute roles. You pass (c) if you connected interpret-then-JIT to slow-first-fast-later and noted a short-lived function may die before JIT pays off (hence native-image/AOT). Teaching-back is the strongest self-test — use it at the end of every lesson in this track.

🎯 Key Takeaways

  • Java compiles source to portable bytecode (javac); the JVM runs it — this is "write once, run anywhere".
  • The JVM interprets bytecode first, then JIT-compiles hot code to native — Java is compiled and interpreted, hence warmup.
  • JDK ⊃ JRE ⊃ JVM: develop with the JDK, run with the JRE, the JVM is the engine inside.
  • Compile with javac Hello.java, run with java Hello (name the class, not the file); prototype with java Hello.java or JShell.
  • The classpath is the ordered list of locations the class loader searches — the root cause of NoClassDefFound errors.
  • Java ships every 6 months with an LTS (8, 11, 17, 21) companies standardize on; Java 8 marks the shift to modern, concise Java.
