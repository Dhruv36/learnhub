Methods & Parameters Foundations

What you'll master in this lesson: how to write methods that read well and compose; the exact meaning of a method signature and why it matters for overloading; the difference between overloading and overriding (a guaranteed interview pairing); how varargs, static vs instance methods, and recursion work; and the professional habits — small methods, clear names, no hidden side effects — that separate maintainable code from spaghetti. Methods are how you tame complexity; this lesson is about doing it well.

Part 1 — Anatomy of a Method

public static int add(int a, int b) {
//  │      │     │    │   └─ parameters (name + type)
//  │      │     │    └───── method name (verb, camelCase)
//  │      │     └────────── RETURN TYPE (void if it returns nothing)
//  │      └──────────────── static: belongs to the CLASS, not an instance
//  └─────────────────────── access modifier (public/private/protected/package)
    return a + b;             // must return an int (matches the return type)
}

The signature — the method name plus its parameter types — is how Java identifies a method. Crucially, the return type is NOT part of the signature. This one fact explains why you can't overload two methods that differ only by return type, and it's a favorite interview gotcha.

Part 2 — Parameters, Arguments & Passing

void greet(String name) {       // 'name' is a PARAMETER (the declaration)
    System.out.println("Hi, " + name);
}
greet("Ada");                   // "Ada" is an ARGUMENT (the actual value passed)

// Recall from the Types lesson: Java is ALWAYS pass-by-value.
void reset(int[] data) {
    data[0] = 0;                // ✅ mutates the shared array — caller sees it
    data = new int[]{9};        // ❌ repoints the local copy only — caller unaffected
}
🎯 A method can mutate the object a reference parameter points to (the caller sees the change) but cannot reassign the caller's variable. This is the pass-by-value-of-a-reference rule from the Types lesson, and it's the single most-tested method concept. A corollary: methods that take a collection and mutate it are a common source of "spooky action at a distance" bugs — prefer returning a new value over mutating a parameter when you can.

Part 3 — Overloading (same name, different parameters)

// Same method NAME, different parameter lists — the compiler picks by arguments:
int    max(int a, int b)        { return a > b ? a : b; }
double max(double a, double b)  { return a > b ? a : b; }
int    max(int a, int b, int c) { return max(max(a, b), c); }   // reuse via a third overload

max(3, 5);        // calls the (int,int) version
max(3.0, 5.0);    // calls the (double,double) version

// ❌ ILLEGAL — can't overload on return type alone (signature = name + params, NOT return type):
int    parse(String s) { ... }
double parse(String s) { ... }   // compile error: "method already defined"
🔍 Overload resolution is a compile-time decision based on the declared (static) types of the arguments, not their runtime types. This is why overloading differs fundamentally from overriding (next Part), which dispatches at runtime. Overloading with autoboxing and widening can get genuinely ambiguous — foo(1) when there's both foo(long) and foo(Integer) — so keep overload sets simple and unsurprising.

Part 4 — Overloading vs Overriding (the classic pairing)

OverloadingOverriding
WhatSame name, different parameters, same class (or inherited)Subclass redefines a superclass method with the same signature
ResolvedAt compile time (static type)At runtime (actual object type — dynamic dispatch)
Return typeCan differ (params must differ)Same or a covariant subtype
PurposeConvenience — variations of one operationPolymorphism — subclass-specific behavior
Marker—@Override (compiler-checked)
class Animal { String speak() { return "..."; } }
class Dog extends Animal {
    @Override String speak() { return "Woof"; }   // OVERRIDE — same signature, new behavior
}
Animal a = new Dog();
a.speak();   // "Woof" — runtime type (Dog) decides. THIS is polymorphism. (See Inheritance lesson.)

Part 5 — static Methods, Varargs & Recursion

// STATIC — no object needed; utility/factory methods. Called on the CLASS:
static int square(int x) { return x * x; }
int r = MathUtil.square(5);        // no instance created

// VARARGS — accept zero or more arguments of a type (treated as an array inside):
static int sum(int... nums) {      // the ... must be the LAST parameter
    int total = 0;
    for (int n : nums) total += n;
    return total;
}
sum();            // 0
sum(1, 2, 3);     // 6 — pass any number of ints

// RECURSION — a method that calls itself; needs a BASE CASE or it StackOverflows:
static long factorial(int n) {
    if (n <= 1) return 1;          // BASE CASE — stops the recursion
    return n * factorial(n - 1);   // recursive case, moving toward the base
}
⚠️ Every recursion needs a base case that's actually reached. Miss it (or recurse the wrong direction) and each call stacks a new frame until the JVM throws StackOverflowError. Java does not optimize tail recursion, so deep recursion on large inputs will overflow even when correct — convert to a loop for anything that could recurse thousands of levels deep. Recursion shines for naturally recursive structures (trees, divide-and-conquer); it's a liability for simple linear iteration.

Part 6 — Deep Dive: What Makes a Method "Good"

Syntax is the easy part; writing methods that a teammate (or you, in six months) can understand is the real skill. The professional standards:

  • Do one thing. A method named save that also validates, logs, and sends an email does four things — split them. The name should fully describe the behavior; if you need "and" to describe it, it's two methods.
  • Name with a verb, precisely. calculateTax, findUserByEmail, isEligible — the name should let a reader skip the body. Boolean methods read as questions (isEmpty, hasNext).
  • Keep it short. If a method doesn't fit on a screen, it's probably doing too much. Extract sub-steps into well-named private methods — the names become documentation.
  • Minimize parameters. More than three or four is a smell; bundle related ones into an object. Avoid boolean flag parameters (render(true) tells the reader nothing) — two methods or an enum is clearer.
  • Avoid hidden side effects. A method named getBalance() that also mutates state violates the reader's trust. Queries shouldn't change things; commands can.
🔍 Senior nugget — command/query separation: a method should either return a value (a query, no side effects) or change state (a command, returns void), but ideally not both. When a method both mutates and returns, callers can't reason about it safely, and it's hard to test. This principle, plus "do one thing", is most of what "clean code" means at the method level — the SOLID/Clean Code lesson goes deeper.

Interview Questions

Q1 (beginner): What is a method signature, and what does it include?

A method's signature is its name plus its ordered parameter types (e.g. max(int, int)). It's what Java uses to identify and resolve a method. Notably it does not include the return type, the parameter names, access modifiers, or thrown exceptions. This is why two methods can't differ by return type alone (same signature = conflict), and why overloading requires the parameter lists to differ.

Q2 (beginner): What's the difference between an argument and a parameter?

A parameter is the variable in the method declaration (void greet(String name) — name is the parameter). An argument is the actual value supplied at the call site (greet("Ada") — "Ada" is the argument). Parameters are the placeholders; arguments fill them. People use the terms loosely, but the distinction matters when discussing pass-by-value: the argument's value is copied into the parameter.

Q3 (mid): Explain overloading vs overriding and when each is resolved.

Overloading is multiple methods with the same name but different parameter lists in a class; the compiler picks one at compile time based on the arguments' declared types. Overriding is a subclass providing a new implementation of an inherited method with the same signature; the JVM picks the implementation at runtime based on the object's actual type (dynamic dispatch), which is the mechanism behind polymorphism. Mnemonic: overloading = same name/different params/compile-time; overriding = same signature/subclass/runtime. Marking overrides with @Override lets the compiler catch signature typos that would otherwise silently become overloads.

Q4 (mid): What are varargs, what are their constraints, and their gotcha?

Varargs (Type... name) let a method accept zero or more arguments of a type, received as an array inside the method. Constraints: a method can have only one varargs parameter and it must be last. Gotchas: (1) passing an array directly works (sum(myArray)) because the array is the varargs form; (2) overloading between a varargs method and a fixed-arity one can be ambiguous, and the compiler prefers the more specific fixed-arity match; (3) passing a single null to Object... is ambiguous (null array vs one null element) — cast to disambiguate. Varargs also allocate an array per call, a minor cost in hot loops.

Q5 (senior): Why does Java not optimize tail recursion, and what's the practical consequence for algorithm design?

Java deliberately preserves full stack frames for every call so that stack traces remain complete and accurate (each frame identifies a caller) and security/reflection features that inspect the call stack work correctly — tail-call elimination would collapse frames and break those guarantees. HotSpot may inline some calls but does not perform general tail-call optimization. The consequence: an elegant tail-recursive algorithm that would run in constant stack space in Scheme or (with some effort) Scala will StackOverflowError in Java on large inputs. So for iteration-shaped recursion (walking a long list, accumulating), convert to an explicit loop or an explicit stack/queue data structure; reserve recursion for genuinely tree-shaped or divide-and-conquer problems where depth is logarithmic. Know your maximum recursion depth relative to the ~thousands-of-frames stack limit.

Q6 (senior): Argue for command/query separation with a concrete failure it prevents.

Command/query separation says a method should either return a value without side effects (a query) or change state and return void (a command), not both. The failure it prevents: consider a Stack.pop() that both mutates (removes the top) and returns (the popped value) — a reader who logs log.debug("top=" + stack.pop()) has silently mutated the stack just by logging, and a retry or a duplicated log line corrupts state. Methods that both mutate and return are also hard to test (you can't check the return without triggering the effect) and hard to cache/retry. Enforcing CQS makes code reasoning local: seeing a query, you know it's safe to call freely; seeing a command, you know it's the state-changing step. Java's own APIs sometimes violate this for convenience (Iterator.next, Map.put returning the old value), which is exactly why those are common footguns.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Overload set. Write three describe methods: one taking an int, one a String, one a boolean, each printing a different message. Call all three and observe which runs.
Solution

The compiler dispatches by argument type at compile time: describe(5) → the int version, describe("hi") → the String version, etc. This is overloading — same name, different parameter types. Note you could not add a fourth describe(int) returning a different type; the parameter list must differ.

2 (easy): Varargs average. Write double avg(int... nums) returning the average, or 0 for no arguments. Test with 0, 1, and several arguments.
Solution

if (nums.length == 0) return 0; int sum = 0; for (int n : nums) sum += n; return (double) sum / nums.length;. Note the (double) cast to avoid integer division (Operators lesson), and the empty-args guard so avg() doesn't divide by zero.

3 (medium): Recursion vs iteration. Write factorial recursively, then iteratively. Then call the recursive one with n = 100000 and observe what happens.
Solution

Recursive: base case n <= 1 → 1, else n * factorial(n-1). Iterative: a loop multiplying 1..n. The recursive call with 100000 throws StackOverflowError — 100000 stacked frames exceed the stack, even though the logic is correct. (Also note both overflow long quickly — factorial grows enormous; real code uses BigInteger.) Lesson: recursion depth is bounded by the stack.

4 (medium): Override, not overload. Make a Shape class with double area() returning 0, and a Circle that overrides it. Deliberately misspell the override as double aera() without @Override, then add @Override and observe the compiler error.
Solution

Without @Override, aera() compiles fine — it's just a new, unrelated method, and shape.area() silently calls the base version returning 0 (a nasty bug). Adding @Override to the misspelled method triggers "method does not override a method from its superclass" — the compiler catches the typo. This is exactly why you always annotate overrides.

5 (hard): Refactor a god method. Given a 40-line processOrder that validates, computes totals, saves, and emails, refactor it into a short orchestrating method calling four well-named private methods. Explain what improved.
Solution

processOrder becomes ~4 lines: validate(order); var total = computeTotal(order); save(order, total); notify(order);. Each step is a private method with a name that documents it. Improvements: readable at a glance (the high-level flow is obvious), each piece is independently testable, and a change to (say) the email logic is isolated to notify. The extracted names are the documentation. This is "do one thing" applied.

6 (hard): Explain-it test. Out loud: (a) what a signature is and why you can't overload on return type; (b) overloading vs overriding including when each resolves; (c) why deep recursion is dangerous in Java specifically.
Self-check rubric

Pass (a) if you said signature = name + parameter types, return type excluded. Pass (b) if you tied overloading to compile-time/params and overriding to runtime/dynamic-dispatch/same-signature. Pass (c) if you noted no tail-call optimization + fixed stack → StackOverflowError, convert to iteration. These underpin the OOP, Inheritance, and Collections lessons ahead.

🎯 Key Takeaways

  • A signature = method name + parameter types (return type excluded) — so you can't overload on return type alone.
  • Methods pass arguments by value; they can mutate a shared object but not repoint the caller's reference.
  • Overloading = same name/different params, resolved at compile time; overriding = same signature in a subclass, resolved at runtime — always mark overrides @Override.
  • static methods belong to the class; varargs (T...) must be last; recursion needs a reachable base case and can StackOverflow (no tail-call optimization).
  • Good methods do one thing, are named with precise verbs, stay short, take few parameters, and avoid hidden side effects (command/query separation).
