I/O, Files & NIO Core Libraries

What you'll master in this lesson: the streams-vs-readers distinction (bytes vs characters) and why charset matters; buffering and why it makes I/O orders of magnitude faster; the modern java.nio.file API (Path, Files) that makes file work a one-liner; how to read/write text and process large files without exhausting memory; and the resource-management and encoding rules that prevent leaks and mojibake. I/O is where "works on my machine" bugs are born — this lesson is about doing it correctly and efficiently.

Part 1 — The Two Families: Streams (bytes) vs Readers (characters)

Byte streamsCharacter streams
Base classesInputStream / OutputStreamReader / Writer
UnitRaw bytesCharacters (charset-decoded)
Use forBinary data (images, zip, network)Text (files, JSON, logs)
CharsetN/A (raw)Required — bytes ⇄ chars needs an encoding
// The bridge between them (bytes → chars) is where charset MUST be specified:
InputStream in = new FileInputStream("data.txt");
Reader reader = new InputStreamReader(in, StandardCharsets.UTF_8);   // ✅ explicit charset
// new InputStreamReader(in)   ❌ uses the platform default — the mojibake bug (Strings lesson)
💡 Choose the family by the data: bytes for binary, characters for text. The moment you convert bytes to text (reading a text file, decoding a network payload), a charset is involved — and if you don't specify it, Java uses the platform default, which differs across machines and causes the "works locally, garbled in production" bug. Always pass StandardCharsets.UTF_8.

Part 2 — Buffering: The Performance Essential

// UNBUFFERED — one system call per byte/char. Reading a 1MB file = ~1 million syscalls. Glacial.
Reader r = new FileReader("big.txt");
int c;
while ((c = r.read()) != -1) { ... }        // one OS read() per character — catastrophically slow

// BUFFERED — reads a big chunk into memory, serves from it. Orders of magnitude faster:
try (BufferedReader br = new BufferedReader(new FileReader("big.txt", UTF_8))) {
    String line;
    while ((line = br.readLine()) != null) { process(line); }   // reads whole lines, from a buffer
}
🎯 Always buffer stream-based I/O. A BufferedReader/ BufferedInputStream wraps the underlying stream and reads in large blocks (default 8KB), turning a million tiny system calls into a few hundred — often a 10–100× speedup. Unbuffered byte-at-a-time I/O is one of the most common performance mistakes. (The modern Files methods in Part 3 buffer internally, so prefer them and you get this for free.)

Part 3 — The Modern Way: java.nio.file (Path & Files)

import java.nio.file.*;

Path path = Path.of("data", "users.txt");        // platform-independent path building

// Read/write whole files in ONE line (for reasonably-sized files):
String content = Files.readString(path, UTF_8);          // whole file → String
List<String> lines = Files.readAllLines(path, UTF_8);     // whole file → List<String>
Files.writeString(path, "hello", UTF_8);                  // write a String
Files.write(path, lines, UTF_8);                          // write lines

// File operations without the old java.io.File clunkiness:
Files.exists(path);   Files.createDirectories(dir);   Files.copy(src, dst);
Files.delete(path);   Files.size(path);   Files.move(a, b);
🔍 Prefer java.nio.file (Path/Files) over the legacy java.io.File. The NIO.2 API (Java 7) is cleaner, throws informative exceptions (the old File methods returned a useless boolean on failure with no reason), handles symlinks and file attributes properly, and its Files helpers buffer internally. Files.readString/writeString (Java 11) make small-file text I/O a genuine one-liner — no manual reader, buffer, or close.

Part 4 — Streaming Large Files (don't load it all)

// ❌ Files.readAllLines on a 10GB file → OutOfMemoryError (loads everything into memory)

// ✅ Files.lines returns a lazy Stream<String> — processes line by line, constant memory:
try (Stream<String> lines = Files.lines(path, UTF_8)) {   // MUST close it (it holds the file handle)
    long errorCount = lines.filter(l -> l.contains("ERROR")).count();
}   // try-with-resources closes the stream AND the underlying file

// Copy/transfer large binary data efficiently:
try (InputStream in = Files.newInputStream(src);
     OutputStream out = Files.newOutputStream(dst)) {
    in.transferTo(out);          // efficient chunked copy (Java 9+), no manual byte buffer
}
⚠️ Match the tool to the file size. readString/readAllLines are convenient but load the entire file into memory — fine for config and small data, fatal for large logs or datasets. For big files use Files.lines() (a lazy stream processed one line at a time in constant memory) and close it with try-with-resources, because unlike a collection-based read, the stream keeps the OS file handle open until closed. Forgetting to close Files.lines leaks file descriptors.

Part 5 — Resource Management & try-with-resources (recap in context)

// EVERY stream/reader/writer/channel is a resource that MUST be closed. Always try-with-resources:
try (var br = Files.newBufferedReader(path, UTF_8);
     var bw = Files.newBufferedWriter(out, UTF_8)) {
    br.lines().map(String::toUpperCase).forEach(line -> {
        try { bw.write(line); bw.newLine(); }
        catch (IOException e) { throw new UncheckedIOException(e); }   // adapt checked → unchecked in lambdas
    });
}   // both closed automatically, in reverse order, even on exception
🔒 Leaking file handles is a real production failure — each OS has a limit on open file descriptors, and a leak (a stream never closed, often on an error path) eventually throws "too many open files" and takes the service down. Try-with-resources (Exceptions lesson) is the guarantee against this. Note the checked-exception friction in lambdas: IOException can't propagate out of a stream lambda, so you wrap it in an unchecked UncheckedIOException — a concrete example of why checked exceptions and streams don't compose well (Exceptions lesson).

Part 6 — Deep Dive: Blocking I/O, and Where It Meets Concurrency

  • Classic Java I/O is blocking. read() parks the thread until data arrives. That's simple and fine when one thread handles one file/connection, but it means a server handling 10,000 connections the naive way needs 10,000 blocked threads — expensive in memory and scheduling.
  • Why NIO channels/selectors exist. java.nio added non-blocking channels and selectors so one thread can manage many connections (the basis of high-throughput servers like Netty). It's more complex, which is why frameworks abstract it.
  • Virtual threads change the calculus (Java 21). Project Loom's virtual threads make the simple "one blocking thread per connection" model scale to millions of threads, because a virtual thread that blocks on I/O cheaply yields its carrier OS thread. This is a major shift: the readable blocking style regains its scalability, reducing the need for callback-based non-blocking code. (Full treatment in the Virtual Threads lesson.)
🔍 Senior framing: the through-line from I/O to concurrency is that blocking a thread on I/O is the scalability bottleneck. The historical answers were thread-per-request (simple, doesn't scale), non-blocking NIO/reactive (scales, complex), and now virtual threads (simple and scales). Knowing that "an I/O-bound server's ceiling is how many threads can wait cheaply" connects this lesson to executors, reactive programming, and the whole design of modern Java servers — and to why parallel streams (Streams lesson) are the wrong tool for I/O-bound work (blocking threads in the shared pool). For everyday file work, though: use Files, buffer, specify the charset, close with try-with-resources, and stream large files.

Interview Questions

Q1 (beginner): What's the difference between byte streams and character streams?

Byte streams (InputStream/OutputStream) move raw bytes and are for binary data — images, compressed files, network payloads. Character streams (Reader/Writer) move characters and are for text, which means they must decode/encode bytes using a charset. The bridge classes InputStreamReader/OutputStreamWriter convert between them, and that's exactly where you must specify the charset (e.g. UTF-8) or risk platform-default-encoding bugs. Rule of thumb: bytes for binary, readers/writers for text.

Q2 (beginner): Why wrap a stream in a BufferedReader?

To avoid making an operating-system call for every tiny read. An unbuffered reader issues a syscall per character/byte, so reading a large file can mean millions of syscalls — extremely slow. A BufferedReader reads a large block (default ~8KB) into memory in one syscall and serves subsequent reads from that buffer, cutting syscalls by orders of magnitude, typically a 10–100× speedup. It also adds convenient methods like readLine(). Buffering is essential for any stream-based I/O; the modern Files helper methods buffer internally so you get it automatically.

Q3 (mid): How do you read a very large file without running out of memory?

Don't load it all at once — process it incrementally. Avoid Files.readAllLines/readString (which materialize the whole file) and instead use Files.lines(path, charset), which returns a lazy Stream<String> that reads and processes one line at a time in roughly constant memory, so you can filter/count/transform a multi-gigabyte log without OOM. Crucially, wrap it in try-with-resources because the stream holds the open file handle until closed (unlike a collection). For binary, read in chunks with a buffer or use InputStream.transferTo. The principle is streaming/incremental processing over whole-file loading whenever size is unbounded.

Q4 (mid): Why prefer java.nio.file over java.io.File?

NIO.2 (Path/Files, Java 7) is a cleaner, more capable replacement. The legacy File methods signal failure by returning a bare boolean with no reason (a failed delete() just returns false), whereas Files throws informative IOExceptions explaining what went wrong. NIO.2 handles symbolic links, file attributes, and permissions properly, supports directory-walking and file-system watching, provides atomic operations, and its helper methods (readString, writeString, lines, copy) are concise and buffer internally. Path also builds paths in a platform-independent way. There's essentially no reason to use java.io.File in new code.

Q5 (senior): Explain blocking vs non-blocking I/O and how virtual threads change the trade-off.

Traditional Java I/O is blocking: a thread calling read() is parked by the OS until data is available, holding its (expensive, ~1MB-stack, kernel-scheduled) platform thread idle the whole time. The thread-per-connection model built on this is simple and readable but caps out because thousands of concurrent connections need thousands of blocked threads — memory and scheduler overhead become the bottleneck. Non-blocking I/O (java.nio channels + selectors, and reactive frameworks on top) lets a single thread multiplex many connections via an event loop, scaling to huge connection counts, but at the cost of inverted, callback/reactive-style code that's much harder to write, read, and debug. Java 21's virtual threads (Project Loom) change the calculus: a virtual thread is a lightweight, JVM-scheduled thread, and when it blocks on I/O the JVM unmounts it from its carrier OS thread so the OS thread is freed to run others — meaning you can have millions of virtual threads each written in the simple blocking one-thread-per-request style, yet get non-blocking-level scalability. This largely dissolves the old "simple but unscalable vs scalable but complex" dilemma for I/O-bound workloads: you write straightforward blocking code and the runtime makes it scale. It doesn't help CPU-bound work (you still have limited cores) and doesn't magically fix synchronized bottlenecks, but for the common case of servers waiting on databases and downstream services, it's a major simplification.

Q6 (senior): A long-running service eventually fails with "too many open files". What happened and how do you prevent it?

The service is leaking file descriptors — every OS process has a limit on simultaneously open files/ sockets, and the code is opening streams, readers, sockets, or (commonly) not closing something like a Files.lines() stream or a JDBC connection/result set, so open handles accumulate until the limit is hit and any new open fails. The classic cause is a resource opened but not closed on an error path — a manual close() in the wrong place, or a finally that doesn't run, or a returned Stream/Cursor the caller forgot to close. Diagnose by monitoring the process's open descriptor count over time (it climbs steadily), and by auditing every place a closeable is created. Prevent it by using try-with-resources for every AutoCloseable (streams, readers, channels, connections, statements, result sets), which guarantees close() even on exceptions; being especially careful with APIs that return a closeable you must close (Files.lines, DirectoryStream, JDBC objects); using connection pools that enforce return; and raising the OS ulimit only as a stopgap, not a fix. Static-analysis tools flag unclosed resources, and load testing surfaces slow leaks before production does. The root discipline is: if it's AutoCloseable, it goes in a try-with-resources — no exceptions.

🏋️ Exercises (attempt before opening solutions)

1 (easy): One-line read/write. Write a String to a file and read it back using Files.writeString / Files.readString with an explicit UTF-8 charset.
Solution

Files.writeString(path, "héllo", StandardCharsets.UTF_8); then String s = Files.readString(path, StandardCharsets.UTF_8);. The explicit charset ensures the non-ASCII character round-trips correctly on any platform — omit it and a non-UTF-8 default could corrupt it. No manual reader/writer or close needed.

2 (easy): Buffered line read. Read a text file line by line with a BufferedReader in try-with-resources, counting the lines.
Solution

try (var br = Files.newBufferedReader(path, UTF_8)) { long n = br.lines().count(); }, or a while ((line = br.readLine()) != null) count++; loop. The try-with-resources closes the reader automatically. Buffering makes it fast; the charset makes it correct.

3 (medium): Count matches in a large file. Without loading the whole file, count how many lines contain "ERROR", using Files.lines. Explain why it's memory-safe and why closing matters.
Solution

try (Stream<String> lines = Files.lines(path, UTF_8)) { long n = lines.filter(l -> l.contains("ERROR")).count(); }. Memory-safe because Files.lines is lazy — it reads and discards one line at a time rather than materializing the file. Closing matters because the stream holds the open OS file handle until closed; without try-with-resources you'd leak a descriptor each call.

4 (medium): Charset round-trip bug. Write bytes of a UTF-8 string, then read them back with the wrong charset (e.g. ISO-8859-1) and observe the corruption. Then fix it.
Solution

Encoding "café" as UTF-8 then decoding as ISO-8859-1 turns the multi-byte "é" into two garbage characters ("cafÃ©") — mojibake. Decoding with the matching UTF-8 charset restores "café". The lesson: the read charset must match the write charset, and relying on the platform default (which may differ between the writer's and reader's machines) is exactly how this happens in production. Always specify charset on both ends.

5 (hard): Efficient file copy. Copy a large binary file from one path to another efficiently, using streams and try-with-resources, without a hand-rolled byte buffer.
Solution

try (var in = Files.newInputStream(src); var out = Files.newOutputStream(dst)) { in.transferTo(out); } — transferTo (Java 9+) copies in efficient chunks internally, no manual byte[] buffer or loop. Both streams close automatically in reverse order. (Even simpler for a plain copy: Files.copy(src, dst, StandardCopyOption.REPLACE_EXISTING).) The point: don't hand-roll buffer loops when the library does it correctly and efficiently.

Pass (a) if you tied bytes→binary, chars→text, and named the bridge/charset requirement. Pass (b) if you explained syscall-per-unit vs block reads and the 10–100× effect. 🎯 Key Takeaways

  • Byte streams (InputStream/OutputStream) for binary; character streams (Reader/Writer) for text — always specify the charset (UTF-8) at the bytes⇄chars boundary.
  • Buffer stream I/O (or use Files helpers, which buffer internally) — unbuffered byte-at-a-time reads are 10–100× slower.
  • Prefer modern java.nio.file (Path, Files): readString/writeString/lines/copy over legacy java.io.File.
  • For large files, stream with Files.lines() (constant memory) and close it — it holds the file handle.
  • Wrap every closeable in try-with-resources — leaked descriptors cause "too many open files" outages.
  • Classic I/O blocks a thread; virtual threads (Java 21) let simple blocking code scale — the link from I/O to concurrency.
