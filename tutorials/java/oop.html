Classes, Objects & Encapsulation Object-Oriented Java

What you'll master in this lesson: the class-vs-object distinction (blueprint vs instance); how constructors initialize objects and how this works; what encapsulation really protects and why "just make it public" is a trap; the meaning of each access modifier; static vs instance members; and the concept of an invariant — the promise a well-designed class keeps. This is the foundation of every class you'll ever write.

Part 1 — Class vs Object

A class is a blueprint; an object is a thing built from it. The class Account describes what every account has (a balance) and does (deposit, withdraw); each new Account(...) is a distinct object with its own balance.

public class Account {
    private double balance;                       // FIELD (instance state) — each object has its own

    public Account(double opening) {              // CONSTRUCTOR — runs when you `new` an Account
        this.balance = opening;                   // 'this' = the object being constructed
    }
    public void deposit(double amt) {             // METHOD (behavior)
        balance += amt;
    }
    public double getBalance() { return balance; }
}

Account a = new Account(100);   // ONE object
Account b = new Account(50);    // a DIFFERENT object — b.balance is independent of a.balance
a.deposit(25);                  // a.balance → 125; b.balance still 50
💡 this refers to "the current object". You need it to disambiguate a field from a parameter of the same name (this.balance = balance), to pass the current object to another method, or to call another constructor (this(...)). When there's no ambiguity it's optional, but writing this.field in constructors is a common, readable convention.

Part 2 — Constructors, Overloading & this()

public class Rectangle {
    private final int width, height;

    public Rectangle(int width, int height) {     // primary constructor
        this.width = width;
        this.height = height;
    }
    public Rectangle(int side) {                  // OVERLOADED constructor (a square)
        this(side, side);                         // delegates to the primary — DRY, one place for logic
    }
    // If you write NO constructor, Java gives you a no-arg default. Write ANY constructor
    // and the default disappears — a common surprise when `new Rectangle()` stops compiling.
}
🎯 Constructor rules that trip people up: (1) a constructor has no return type and its name matches the class exactly; (2) you get a free no-arg constructor only if you declare none — add one constructor and you must add the no-arg one back if you still want it; (3) constructors can call each other with this(args) (must be the first statement), keeping initialization logic in one place; (4) a subclass constructor implicitly calls super() first (Inheritance lesson).

Part 3 — Encapsulation: The Core OOP Principle

Encapsulation means bundling data with the methods that operate on it, and controlling access so outside code can't put the object into an invalid state. The mechanism is simple — private fields, public methods — but the reason is what matters:

// WITHOUT encapsulation — a public field anyone can corrupt:
public class BadAccount {
    public double balance;        // anyone can do account.balance = -1_000_000;
}

// WITH encapsulation — the class enforces its own rules (INVARIANTS):
public class Account {
    private double balance;       // nobody outside can touch it directly
    public void withdraw(double amt) {
        if (amt <= 0)            throw new IllegalArgumentException("amount must be positive");
        if (amt > balance)      throw new IllegalStateException("insufficient funds");
        balance -= amt;           // the ONLY way balance changes — rules always enforced
    }
    public double getBalance() { return balance; }   // read-only access
}
🔒 Encapsulation isn't about hiding for its own sake — it's about guaranteeing invariants. An invariant is a truth the object always maintains (a balance is never negative; a date's month is 1–12). Public fields make invariants impossible to guarantee because any code can violate them. Private fields + validating methods mean the class is the single gatekeeper. This is why "make it public to fix the access error" is almost always the wrong move — you're removing the guard, not the problem.

The professional default is as private as possible. Start every field private and every method at the lowest access that works; widen only when a real caller needs it. A class's public members are its contract — everything public is something you've promised not to break, so keep the public surface small.

Part 5 — static vs Instance Members

public class Counter {
    private static int totalCreated = 0;    // STATIC — ONE copy shared by the whole class
    private int id;                         // INSTANCE — each object has its own

    public Counter() {
        totalCreated++;                     // increments the shared class-level counter
        this.id = totalCreated;             // this object's own id
    }
    public static int getTotalCreated() {   // static method — called on the CLASS
        return totalCreated;                // can only touch static members (no 'this')
    }
}
Counter.getTotalCreated();   // called on the class, no object needed
new Counter(); new Counter();
Counter.getTotalCreated();   // 2
🔍 static belongs to the class, not any object. A static field is shared by every instance (a global counter, a cache, a constant); a static method runs without an object and therefore can't use this or touch instance fields. Constants are static final (public static final double PI = 3.14159;). Overusing static turns objects into glorified global variables and hurts testability — reach for it for genuinely class-level data (constants, factories, utilities), not to dodge creating objects.

Part 6 — Deep Dive: The Four Pillars in Context

OOP is traditionally summarized as four pillars. You'll meet the other three in the next lessons, but here's the map so you see where each fits:

PillarMeansWhere
EncapsulationBundle data + behavior; control access to protect invariantsThis lesson
AbstractionExpose what not how; interfaces as contractsInterfaces lesson
InheritanceA subtype reuses/specializes a supertypeInheritance lesson
PolymorphismOne reference type, many runtime behaviors (dynamic dispatch)Inheritance lesson
🔍 Senior nugget: the pillars are a teaching device, not a design goal. Modern Java practice actually de-emphasizes inheritance (it couples tightly and is easy to misuse) in favor of composition and interfaces. What genuinely matters day-to-day is encapsulation (objects that protect their own state) and abstraction (depend on contracts, not implementations). Reels of inheritance hierarchies are a code smell more often than a virtue — a theme the Interfaces and Design Patterns lessons develop.

Interview Questions

Q1 (beginner): What is the difference between a class and an object?

A class is a blueprint or template that defines the fields (state) and methods (behavior) a type of thing has. An object (or instance) is a concrete thing created from that class with new, occupying memory and holding its own copy of the instance fields. One class, many objects: Account is the class; each new Account(...) is a separate object with its own balance. Analogy: the class is the architectural plan; objects are the houses built from it.

Q2 (beginner): What is encapsulation and why does it matter?

Encapsulation is bundling data with the methods that act on it and restricting direct access to that data (private fields, public methods). It matters because it lets the class enforce its invariants — rules like "balance is never negative" — by making the methods the only way to change state, where validation lives. Without it (public fields) any code can put the object into an invalid state, and bugs become impossible to localize. It also decouples callers from internal representation, so you can change the internals without breaking users.

Q3 (mid): Explain the access modifiers and the principle guiding their use.

Four levels: private (same class), package-private/default (same package), protected (same package plus subclasses), and public (everywhere). The guiding principle is least privilege: default to private and widen only when a real collaborator requires it, because every non-private member is part of the contract you must maintain. A small public surface means fewer promises, easier refactoring, and clearer intent. protected is specifically for members that subclasses need to override or use, and it's a design commitment (you're inviting extension).

Q4 (mid): static vs instance members — when is static appropriate?

An instance member belongs to each object (its own copy of the field, methods that use this); a static member belongs to the class (one shared copy, methods callable without an object and unable to use this). Static is appropriate for genuinely class-level things: constants (static final), pure utility functions (Math.max), factory methods, and shared counters/caches (with care for thread-safety). It's inappropriate as a way to avoid creating objects or to hold mutable global state, because that produces hidden coupling and makes code hard to test (you can't easily substitute static dependencies). Prefer instance members and dependency injection for anything with behavior or state you'd want to vary or mock.

Q5 (senior): What is a class invariant, and how do constructors and encapsulation work together to preserve it?

A class invariant is a condition that must hold for every object of the class throughout its observable lifetime — e.g. "a Range's low ≤ high", "an Account's balance ≥ 0". The constructor is responsible for establishing the invariant: it validates its arguments and refuses (throws) rather than build an invalid object, so no object ever exists in a bad state. Encapsulation is responsible for preserving it: fields are private, and every method that mutates state re-checks the rules before changing anything, so there's no path to violate the invariant from outside. Together they guarantee "if it exists, it's valid" — which lets every other method assume the invariant and skip defensive checks, a huge simplification. Public fields or a constructor that skips validation break this and force every caller to re-verify state, defeating the point of the class.

Q6 (senior): Critique the "getter and setter for every field" habit.

Reflexively generating a getter and setter for every private field produces an anemic object — effectively a public data bag with extra ceremony — that has all the boilerplate of encapsulation and none of its benefits. A public setter that just assigns is barely different from a public field: it lets callers drive the object's state instead of asking the object to do something, so invariants leak out into the callers, business logic scatters, and the object stops protecting itself. Better design exposes behavior: instead of setBalance, offer deposit/withdraw that enforce rules; make fields final and set them once in the constructor where possible; add getters only for state callers legitimately need to read. (For pure data-carriers, a record — a later lesson — is the honest, concise choice.) The instinct to avoid is treating encapsulation as a mechanical getter/setter ritual rather than a design principle about who owns the object's rules.

🏋️ Exercises (attempt before opening solutions)

1 (easy): A well-encapsulated class. Write a Temperature class storing Celsius privately, with a constructor that rejects values below absolute zero (−273.15), a getCelsius(), and a getFahrenheit(). Prove the invariant holds.
Solution

Private final double celsius;; constructor throws IllegalArgumentException if celsius < -273.15; getFahrenheit() returns celsius * 9/5 + 32. Because the field is final and validated in the one constructor, no Temperature can ever be below absolute zero — the invariant is guaranteed by construction. Try new Temperature(-300) and confirm it throws.

2 (easy): Constructor delegation. Give the class from exercise 1 a second constructor taking Fahrenheit, delegating to the Celsius one via this(...). Avoid duplicating the validation.
Solution

public Temperature(double f, boolean fahrenheit) { this((f - 32) * 5/9); } — it converts and delegates to the primary constructor, which does the single validation. (A cleaner real design uses a static factory fromFahrenheit(double) to avoid the awkward boolean flag — a nice segue to the "avoid flag parameters" idea from the Methods lesson.)

3 (medium): The static counter. Write a class whose objects each get a unique sequential id, tracked with a static counter. Create three and print their ids and the total created.
Solution

A private static int seq = 0; incremented in the constructor, assigning this.id = ++seq;. Three objects get ids 1, 2, 3; a static getTotalCreated() returns seq = 3. The static field is shared across all instances — that's exactly the class-level state static is for. (Note: not thread-safe as written — a preview of the Concurrency and Atomics lessons.)

4 (medium): Break and fix an invariant. Start with a BankAccount that has a public balance. Show code that corrupts it to a negative value, then refactor to private + deposit/withdraw so it's impossible.
Solution

With public double balance, any caller does acct.balance = -5000; — invariant violated, and you can't tell where. After refactoring to a private field with validating methods, the only paths that change balance enforce "positive amount" and "sufficient funds", so a negative balance is unreachable. This is the encapsulation argument made concrete.

5 (hard): Behavior over state. Given an anemic Order with public getters/setters for items, status, and total, redesign it so callers can't set an inconsistent state (e.g. status "SHIPPED" with no items). Move the rules inside.
Solution

Remove the setters; expose behavior methods: addItem(item) (recomputes total), ship() (throws if items is empty or status isn't valid to ship). status and total become read-only, derived/controlled internally. Now an Order can't be put into an invalid state from outside — the rules live in one place (the object) instead of scattered across every caller. This is the "expose behavior, not state" principle from Q6.

6 (hard): Explain-it test. Out loud: (a) class vs object; (b) what encapsulation protects (invariants) and why public fields break it; (c) when static is and isn't appropriate.
Self-check rubric

Pass (a) if you said blueprint vs instance with independent state. Pass (b) if you defined an invariant and explained private-fields-plus-validating-methods as the single gatekeeper. Pass (c) if you allowed static for constants/utilities/class-level data but flagged it for mutable global state and testability. These underpin Inheritance, Interfaces, and Immutability next.

🎯 Key Takeaways

  • A class is a blueprint; an object is an instance with its own state — this refers to the current object.
  • Constructors establish valid state; you get a free no-arg constructor only if you declare none; delegate shared init with this(...).
  • Encapsulation = private fields + controlled methods, whose real purpose is to guarantee invariants.
  • Default to the least access (private); everything public is a promise — keep the surface small.
  • static belongs to the class (shared state, no this) — right for constants/utilities, wrong as global mutable state.
  • Expose behavior, not raw state — reflexive getters/setters produce anemic objects that don't protect themselves.
