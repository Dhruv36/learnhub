OOP: Classes & Interfaces Basics→Intermediate

Java is object-oriented to its core. The four pillars — encapsulation, inheritance, polymorphism, abstraction — are interview table stakes, but the deeper skills are: interfaces vs abstract classes, the equals/hashCode contract, and composition over inheritance. This lesson goes past definitions into the trade-offs.

Encapsulation — the Class

public class Account {
    private double balance;                 // private state — the encapsulation boundary
    public Account(double opening) { this.balance = opening; }   // constructor
    public void deposit(double amt) {       // controlled mutation with invariants
        if (amt <= 0) throw new IllegalArgumentException("positive only");
        balance += amt;
    }
    public double getBalance() { return balance; }   // read access
}
// Fields private, behavior public. Invariants (balance rules) live in methods, not scattered.

Inheritance & Polymorphism

public abstract class Shape {               // abstract: can't instantiate; may have partial impl
    abstract double area();                 // subclasses MUST implement
    void describe() { System.out.println("area=" + area()); }  // shared behavior
}
public class Circle extends Shape {
    private final double r;
    Circle(double r) { this.r = r; }
    @Override double area() { return Math.PI * r * r; }   // @Override = compiler-checked
}
Shape s = new Circle(2);   s.describe();    // POLYMORPHISM: runtime picks Circle.area() (dynamic dispatch)
// One reference type (Shape), many runtime types — the basis of extensible design.

Interfaces — Contracts (and Modern Powers)

public interface Payable {
    void pay(double amount);                 // abstract method (the contract)
    default void refund(double amount) {     // DEFAULT method — impl without breaking implementers
        pay(-amount);
    }
    static Payable noop() { return a -> {}; } // static factory
}
class Invoice implements Payable {           // a class can implement MANY interfaces
    public void pay(double amount) { ... }
}
// Interfaces = capabilities (Comparable, Iterable, Runnable). Since Java 8 they can have
// default + static methods; since 9, private methods. Functional interfaces (one abstract
// method) enable lambdas (streams lesson).

Interface vs Abstract Class — the Decision

InterfaceAbstract class
InheritanceImplement MANYExtend ONE (single inheritance)
StateNo instance fields (constants only)Can have fields/state
Use forA CAPABILITY multiple unrelated types share (Comparable)A base with shared state + partial implementation (a family)
ConstructorsNoneYes

Rule of thumb: "CAN-DO" (a capability) → interface; "IS-A with shared guts" → abstract class. Default to interfaces + composition; reach for abstract classes when there's real shared state.

The equals / hashCode Contract (guaranteed interview topic)

@Override public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Account a)) return false;   // pattern-matching instanceof (16+)
    return Double.compare(balance, a.balance) == 0;
}
@Override public int hashCode() { return Objects.hash(balance); }
// THE CONTRACT: equal objects MUST have equal hashCodes. Override BOTH or neither —
// override equals without hashCode and your objects break as HashMap keys / HashSet members
// (they land in wrong buckets, "vanishing" from the set). Records generate both correctly.

Composition Over Inheritance

// Instead of a deep hierarchy, inject collaborators:
class OrderService {
    private final PaymentGateway gateway;      // HAS-A, not IS-A
    private final Notifier notifier;
    OrderService(PaymentGateway g, Notifier n) { this.gateway = g; this.notifier = n; }
}
// "Favor composition over inheritance" (Effective Java): inheritance couples subclass to
// superclass internals (fragile base class); composition + interfaces is flexible and testable.
// Program to INTERFACES, inject dependencies — the seed of Spring's DI (spring-boot lesson).

Interview Questions

Q1: Interface vs abstract class — when each?

Interface: a capability that unrelated types can share (a class implements many), no instance state, since 8 has default/static methods — use for contracts (Comparable, Runnable) and to enable polymorphism without an inheritance relationship. Abstract class: a partial base with shared state and implementation for a family of related types (single inheritance). "CAN-DO" → interface, "IS-A with shared guts" → abstract class. Modern Java leans interfaces + composition.

Q2: Explain the equals/hashCode contract and a concrete failure.

If a.equals(b) then a.hashCode() == b.hashCode() (the reverse isn't required). Override equals without hashCode and two "equal" objects likely get different hashCodes → they hash to different buckets, so a HashSet contains duplicates and map.get(equalKey) returns null — the object seems to "vanish". Always override both consistently (Objects.hash / Objects.equals), or use a record which generates them from components.

Q3: Why "favor composition over inheritance"?

Inheritance exposes and couples to the superclass's implementation — a base-class change can break subclasses (fragile base class), and you get one parent only. Composition assembles behavior from injected collaborators behind interfaces: flexible (swap implementations), testable (inject mocks), and avoids the diamond/fragility problems. Inherit only for genuine is-a with a stable contract; otherwise compose. It's Effective Java's canonical advice and the basis of DI frameworks.

🏋️ Practice: build Shape (abstract) + Circle/Rectangle with @Override area(); demonstrate polymorphism through a Shape[] loop; implement Comparable to sort custom objects; write a class with correct equals/hashCode and prove it works as a HashSet member (then break it by removing hashCode); refactor an inheritance chain into composition with injected interfaces.

🎯 Key Takeaways

  • Four pillars: encapsulation (private state + methods), inheritance, polymorphism (dynamic dispatch), abstraction.
  • Interface = capability (implement many, no state, default/static methods); abstract class = shared base (extend one, has state).
  • equals/hashCode: override both consistently or objects break as map keys/set members.
  • @Override is compiler-checked — always use it; program to interfaces.
  • Favor composition + DI over deep inheritance — flexible, testable, avoids fragile base classes.
