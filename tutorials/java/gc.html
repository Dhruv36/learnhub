Garbage Collection JVM & Performance

What you'll master in this lesson: how the garbage collector actually reclaims memory (mark-sweep-compact and generational collection); the fundamental trade-off between throughput, latency, and footprint; the main collectors (G1, ZGC, Parallel, Serial) and when to choose each; what "stop-the-world" pauses are and why they matter; and how to read GC logs and tune the heap. Garbage collection is Java's defining feature — it removes manual memory bugs but introduces its own tuning discipline, and understanding it is essential for any latency-sensitive system and a deep interview topic.

Part 1 — How Collection Works: Mark, Sweep, Compact

// The core algorithm, in three conceptual phases:
1. MARK    Start from GC ROOTS (thread stacks, static fields) and trace every reachable object.
           Everything reached is LIVE; everything else is garbage.

2. SWEEP   Reclaim the memory of unmarked (unreachable) objects.

3. COMPACT Move surviving objects together to eliminate fragmentation, so free space is contiguous
           (fast bump-pointer allocation) — not all collectors compact, but most do for the old gen.

The key idea (from the Memory lesson): the GC reclaims unreachable objects, found by tracing references from GC roots. This reachability approach — unlike reference counting — correctly collects cycles. Compaction matters because sweeping alone leaves "holes" (fragmentation) that slow allocation and waste space; moving survivors together restores fast, contiguous allocation.

💡 You never call the GC explicitly. System.gc() is only a suggestion the JVM may ignore, and calling it is almost always wrong — it can force an expensive full collection at a bad time. The GC runs automatically based on heap pressure. Your job is not to trigger it but to feed it well (low allocation rate, objects that die young) and, when necessary, to choose and tune the right collector.

Part 2 — Generational Collection (why it's fast)

// Using the young/old heap split from the Memory lesson:
MINOR GC  — collects the YOUNG generation (Eden + survivors). Frequent, fast, short pauses.
            Most objects are dead here (die young), so little is copied → cheap.

MAJOR / FULL GC — collects the OLD generation (and often everything). Infrequent, slower,
            longer pauses. Long-lived objects accumulate here and are expensive to collect.

// Objects that survive several minor GCs are PROMOTED (tenured) to the old generation.
🎯 Generational collection is why modern GC is efficient. Because most objects die young, collecting the small young generation frequently reclaims most garbage cheaply, and the expensive whole-heap collection happens rarely. This is the single most important GC concept: the collector does far less work than scanning the entire heap every time. A common performance problem is a high promotion rate — objects surviving to the old generation that shouldn't — which triggers costly major GCs; it usually points to objects living longer than necessary or an undersized young generation.

Part 3 — The Fundamental Trade-off: Throughput vs Latency vs Footprint

GoalMeansFavored by
ThroughputMax work done; GC total time minimized (accepts longer pauses)Batch jobs, analytics
LatencyShort, predictable pauses (accepts more total GC work)Interactive services, APIs, trading
FootprintMinimal memory/CPU overheadConstrained/embedded environments

You cannot maximize all three — improving one usually costs another. A throughput collector may pause for a full second occasionally but waste little total time; a low-latency collector keeps pauses under a millisecond but does more background work (more CPU). The right choice depends on what your application needs: a nightly batch job wants throughput; a user-facing API wants low, predictable latency.

🔍 "Stop-the-world" (STW) pauses are moments when the GC halts all application threads to do its work safely. Every collector has some STW pauses; the difference is their length and frequency. For a latency-sensitive service, a long STW pause means requests stall — a 500ms full GC pause is 500ms of frozen users. Minimizing and bounding these pauses is the whole point of modern low-latency collectors.

Part 4 — The Collectors & Choosing One

CollectorOptimized forUse when
G1 (default, 9+)Balanced — predictable pauses on large heapsThe sensible default for most server apps
ZGC / ShenandoahUltra-low latency (sub-millisecond pauses)Large heaps + strict latency SLAs
ParallelMax throughput (longer pauses OK)Batch/analytics; pause-tolerant
SerialTiny footprint, single-threadedSmall heaps, containers, CLIs
🎯 Start with the default (G1) and only change it with a measured reason. G1 divides the heap into regions and collects the ones with the most garbage first ("garbage first"), aiming for a configurable pause-time target — a good balance for most server workloads. Reach for ZGC or Shenandoah when you have a large heap and hard latency requirements (they keep pauses sub-millisecond even on huge heaps by doing most work concurrently). Use Parallel for throughput-only batch jobs and Serial for tiny single-core/container workloads. Choosing a collector is a deliberate decision driven by your latency/throughput needs and heap size — not a default to fiddle with speculatively.

Part 5 — Reading GC Logs & Basic Tuning

# Enable GC logging (Java 9+ unified logging):
java -Xlog:gc*:file=gc.log -Xms4g -Xmx4g -jar app.jar

# Key heap flags:
-Xms4g -Xmx4g          # initial and MAX heap — set them EQUAL in production (avoid resize pauses)
-XX:+UseG1GC           # select the collector (G1 is default)
-XX:MaxGCPauseMillis=200   # G1 pause-time target
-XX:+HeapDumpOnOutOfMemoryError   # auto-dump on OOM for post-mortem analysis

# What to look for in logs / monitoring:
#   - Pause durations (are STW pauses within your SLA?)
#   - GC frequency (too frequent → heap too small or allocation rate too high)
#   - Post-GC live-set size (growing steadily → a leak — Memory lesson)
#   - Promotion rate (high → objects tenuring too fast → major GC pressure)
🔍 The most common tuning lever is heap size. Too small → frequent GCs and OOM risk; too large → longer pauses and wasted memory. Set -Xms equal to -Xmx in production so the heap doesn't resize (resizing causes pauses and unpredictability). Beyond that, collector choice and pause-time targets are the main knobs — but don't tune blindly: enable GC logging, measure against your actual workload and SLA, and change one thing at a time. Most GC "problems" are actually application problems (a leak, a high allocation rate, an unbounded cache) that no tuning will fix.

Part 6 — Deep Dive: The GC Mindset for Real Systems

  • Feed the generational hypothesis. The GC is optimized for objects that die young, so the best-performing code keeps most objects short-lived and confined — don't hold references longer than needed, don't promote transient data to long-lived caches. High promotion and high allocation rates are what stress the collector.
  • Latency comes from pauses, not total GC time. For an interactive service, a collector that spends more total CPU on GC but keeps pauses tiny (ZGC) is better than one that's "efficient" overall but pauses for a second. Match the collector to whether you care about tail latency or aggregate throughput.
  • Size the heap for the workload, and pin it. Enough headroom to avoid constant GC, not so much that pauses (in pausing collectors) grow; equal -Xms/-Xmx for predictability; and account for containers (set heap relative to the container memory limit — modern JVMs are container-aware).
  • Diagnose before tuning. GC logs and heap dumps tell you whether you have a leak (rising post-GC live set), allocation pressure (frequent minor GCs), promotion pressure (frequent major GCs), or genuinely need a different collector. Tuning without this data is guessing.
🔍 Senior framing: the mature view of GC is that it's a well-engineered system you mostly cooperate with rather than fight. The application's memory behavior — allocation rate, object lifetimes, promotion — matters far more than flags; a program that allocates modestly and lets objects die young will perform well under almost any collector, while a program with a leak or an unbounded cache will defeat any tuning. So the senior playbook is: write allocation-conscious code where it's hot, choose the collector deliberately for your latency-vs-throughput profile (default G1, ZGC/Shenandoah for strict latency on big heaps, Parallel for batch), size and pin the heap, enable GC logging, and diagnose real problems from data rather than reaching for magic flags. Being able to explain the throughput/latency/footprint trade-off, why generational collection is fast, what a stop-the-world pause costs a service, and how you'd read a GC log to distinguish a leak from allocation pressure is exactly the depth that distinguishes a senior answer — GC is where "Java developer" meets "systems engineer."

Interview Questions

Q1 (beginner): How does garbage collection decide what to reclaim?

By reachability. The collector starts from GC roots (live thread stack variables, static fields, etc.) and traces every object reachable through references — those are live. Anything not reached is unreachable and therefore garbage that can be reclaimed. This is why it's called a tracing collector, and why it correctly handles reference cycles (two objects pointing at each other but referenced by nothing else are both unreachable). You don't decide manually; making an object unreachable (dropping all references to it) is what makes it eligible, and the GC reclaims it at some later collection.

Q2 (beginner): What is a "stop-the-world" pause?

A stop-the-world (STW) pause is a period during garbage collection when the JVM suspends all application threads so the collector can do work that requires a stable view of memory (like moving objects or finalizing which are live). During an STW pause the application does nothing — no requests are served — so a long pause directly translates to latency: a 500 ms full-GC pause freezes users for half a second. Every collector has some STW pauses; the difference between collectors is largely how short and infrequent they keep them. Minimizing STW pause time is the central goal of low-latency collectors like ZGC and Shenandoah, which do most of their work concurrently with the application.

Q3 (mid): Explain the throughput vs latency trade-off in GC.

Throughput means maximizing the total useful work the application does, which a GC optimizes by minimizing the total time spent collecting — even if that means occasional long pauses. Latency means keeping individual GC pauses short and predictable so no single operation stalls, which typically costs more total GC work/CPU because the collector does much of its work concurrently and in smaller increments. You can't maximize both: a throughput collector (Parallel) may pause for a second occasionally but waste little aggregate time — great for a batch job where total runtime is what matters; a latency collector (ZGC/Shenandoah) keeps pauses sub-millisecond but uses more CPU overall — essential for an interactive service where a long pause means stalled requests and bad tail latency. Footprint (memory/CPU overhead) is the third axis, favored in constrained environments. The right collector follows from which of these your workload actually needs.

Q4 (mid): Which garbage collectors exist and how do you choose?

The main ones: G1 (the default since Java 9) divides the heap into regions and collects the highest-garbage regions first, targeting predictable pauses — a balanced choice good for most server applications. ZGC and Shenandoah are low-latency collectors that keep pauses sub-millisecond even on very large heaps by doing almost all work concurrently — choose them when you have big heaps and strict latency SLAs. Parallel maximizes throughput at the cost of longer pauses — good for pause-tolerant batch/analytics jobs. Serial is single-threaded with minimal overhead — suited to small heaps, single-core or memory-constrained containers, and short-lived CLIs. The selection method is to start with the default G1, identify your priority (tail latency vs total throughput vs footprint) and heap size, and switch deliberately with measurement — ZGC/Shenandoah for latency-critical large-heap services, Parallel for throughput batch, Serial for tiny footprints. You shouldn't change collectors speculatively.

Q5 (senior): How would you diagnose and address frequent long GC pauses in a latency-sensitive service?

First, gather data rather than guess: enable unified GC logging (-Xlog:gc*) and inspect pause durations, frequency, which generation is being collected, the post-GC live-set trend, and the promotion rate; correlate GC pauses with the service's latency spikes to confirm GC is the cause. Then interpret: (1) if minor GCs are frequent, the allocation rate is high or the young generation is too small — reduce allocation in hot paths (reuse buffers, primitives over boxed, avoid needless object churn) or enlarge the young gen so short-lived objects die there instead of promoting; (2) if major/full GCs are frequent or long, the old generation is filling — check the promotion rate (objects tenuring too fast, often from an undersized young gen or objects living longer than needed) and the post-GC live set (if it climbs steadily, it's a leak — take a heap dump and fix the retained references, since no GC tuning cures a leak); (3) if the live set is stable but pauses still exceed the SLA, the collector or heap size is the issue — for strict latency, switch from a pausing collector to ZGC or Shenandoah, which keep pauses sub-millisecond by working concurrently, and size the heap with enough headroom while setting -Xms equal to -Xmx to avoid resize pauses; for G1, set a realistic MaxGCPauseMillis target. Also verify container-awareness (heap sized within the container limit) and rule out application causes like System.gc() calls or huge short-lived allocations. The disciplined approach — measure, distinguish allocation pressure vs promotion pressure vs leak vs collector mismatch, then apply the specific fix and re-measure — is what separates real GC work from cargo-cult flag tweaking. Often the fix is in the application (allocation/lifetime/leak), not the flags.

Q6 (senior): Why is System.gc() almost always the wrong tool, and what should you do instead?

System.gc() is only a hint that the JVM is free to ignore, and when it isn't ignored it typically triggers a full, stop-the-world collection — the most expensive kind — at a moment chosen by your code rather than by the collector's own heuristics, which are far better informed about heap state. So it commonly does harm: injecting long pauses into a latency- sensitive service, disrupting the generational collector's tuned cadence, and masking the real problem. People reach for it to "fix" memory pressure or force cleanup, but if you're near OOM, System.gc() won't help because the objects are reachable (a leak) — the fix is to release references, not to collect harder; and if allocation pressure is the issue, the answer is to allocate less or size the heap appropriately, not to force collections. The correct approach is to let the JVM manage collection automatically and instead: diagnose with GC logs and heap dumps, fix leaks by cutting retained references, reduce allocation rate in hot paths, size and pin the heap, and choose the right collector for your latency/throughput profile. (The only legitimate uses of a GC hint are narrow — e.g. benchmarking harnesses controlling state, or right before a deliberate latency-tolerant idle period — and even then it's a hint. In production request paths it should essentially never appear; many teams disable it with -XX:+DisableExplicitGC.)

🏋️ Exercises (attempt before opening solutions)

1 (easy): Minor vs major. Explain the difference between a minor and a major GC in terms of which generation, frequency, and pause length.
Solution

A minor GC collects the young generation (Eden + survivors) — frequent, fast, short pauses, and reclaims most garbage cheaply because most objects die young. A major/full GC collects the old generation (and often the whole heap) — infrequent but slower with longer pauses, because long-lived tenured objects are expensive to collect. Objects surviving several minor GCs get promoted to the old generation.

2 (easy): Enable GC logging. State the flags to log GC and to set a fixed 2GB heap, and name three things you'd look for in the log.
Solution

-Xlog:gc*:file=gc.log -Xms2g -Xmx2g. Look for: pause durations (within SLA?), GC frequency (too frequent → heap too small / high allocation), and the post-GC live-set trend (rising → leak). Promotion rate and full-GC frequency are also key signals.

3 (medium): Choose a collector. For each, pick a collector and justify: (a) a nightly ETL batch job; (b) a low-latency trading API; (c) a tiny CLI in a constrained container.
Solution

(a) Parallel — batch cares about total throughput, not pause length, so accept longer pauses for minimal total GC time. (b) ZGC/Shenandoah — strict tail-latency needs sub-millisecond pauses even under load. (c) Serial — tiny footprint, single-threaded, minimal overhead for a small heap/short life. Matching collector to the throughput/latency/footprint priority is the skill.

4 (medium): Leak vs tuning. A service OOMs and someone suggests raising -Xmx. The post-GC live set is climbing steadily. What's really wrong and what should you do?
Solution

A steadily climbing post-GC live set means objects are being retained that should be released — a leak, not undersizing. Raising -Xmx only delays the OOM. The fix is to take a heap dump, find what's retaining memory (path to GC roots), and cut the reference — bound a cache, deregister listeners, clear ThreadLocals, etc. No GC tuning cures a leak.

5 (hard): Reduce major-GC pressure. A service has frequent, long major GCs and a high promotion rate, but no leak (stable live set). Outline application- and JVM-level approaches.
Solution

High promotion + frequent major GC without a leak means too many objects survive minor GCs and tenure. Application-level: reduce object lifetimes and allocation churn so more objects die young (avoid holding transient data, reuse buffers, don't stash short-lived objects in longer-lived structures). JVM-level: enlarge the young generation so short-lived objects are collected there before promotion, tune survivor sizing/tenuring threshold, ensure adequate total heap, and consider a low-pause collector (G1 with a pause target, or ZGC/Shenandoah) if pause length is the pain. Measure the promotion rate before and after each change.

6 (hard): Explain-it test. Out loud: (a) mark-sweep-compact and reachability; (b) generational collection and why it's fast; (c) the throughput/latency/footprint trade-off and how it drives collector choice.
Self-check rubric

Pass (a) if you described tracing from roots to mark live, sweeping garbage, compacting to avoid fragmentation. Pass (b) if you said most objects die young → collect the small young gen cheaply and often, old gen rarely. Pass (c) if you contrasted total-GC-time (throughput) vs pause-length (latency) vs overhead (footprint) and mapped G1/ZGC/Parallel/Serial to needs. This caps the JVM & Performance foundation before the Performance lesson.

🎯 Key Takeaways

  • GC reclaims unreachable objects via mark-sweep-compact (tracing from GC roots; compaction avoids fragmentation).
  • Generational collection (frequent cheap minor GCs, rare expensive major GCs) is fast because most objects die young.
  • You trade off throughput vs latency vs footprint — you can't maximize all three; stop-the-world pauses are what hurt interactive services.
  • Collectors: G1 (balanced default), ZGC/Shenandoah (low latency, big heaps), Parallel (throughput/batch), Serial (tiny footprint).
  • Never call System.gc(); set -Xms = -Xmx; enable GC logging and diagnose before tuning.
  • Most "GC problems" are application problems (leaks, high allocation/promotion) — feed the generational GC well and fix the code, not just the flags.
