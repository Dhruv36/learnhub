var, Text Blocks & Newer APIs Modern Java

What you'll master in this lesson: var local type inference and when it helps vs hurts readability; text blocks for multi-line strings; the convenient collection/stream additions (List.of, toList, Stream.toList, enhanced String methods); and a practical map of what each recent Java version added, so you know what's available on the LTS you're targeting. This lesson ties together the quality-of-life features that make post-Java-8 code dramatically more concise — and the judgment about when to use them.

Part 1 — var: Local Type Inference

// var infers the type from the initializer — but it's STILL statically typed:
var count = 10;                      // inferred int
var name = "Ada";                    // inferred String
var users = new ArrayList<User>();   // inferred ArrayList<User> — no repeated type on both sides
var entry = Map.entry("k", 42);      // inferred Map.Entry<String,Integer> — saves a verbose type

// var is NOT dynamic typing — the type is fixed at compile time:
var x = 10;
// x = "hello";   ❌ COMPILE ERROR — x is an int forever

// RESTRICTIONS: var needs an initializer and only works on LOCAL variables:
// var y;                 ❌ can't infer with no value
// var f = () -> 42;      ❌ can't infer a lambda/method-ref target type
// (var) fields, parameters, and return types are NOT allowed — locals only.
🎯 Use var when it improves readability, not reflexively. It shines when the type is obvious from the right-hand side (var users = new ArrayList<User>() — repeating ArrayList<User> adds nothing) or genuinely verbose (Map.Entry<String,List<Integer>>). It hurts readability when the type isn't obvious from the initializer (var result = process() — what's the type?). The rule: var is for reducing noise where the type is already clear, not for hiding types the reader needs to see.

Part 2 — Text Blocks (multi-line strings)

// BEFORE — escaping and concatenation soup:
String json = "{\n" +
              "  \"name\": \"Ada\",\n" +
              "  \"role\": \"engineer\"\n" +
              "}";

// AFTER (15+) — a text block: triple quotes, real newlines, no escaping:
String json = """
    {
        "name": "Ada",
        "role": "engineer"
    }""";
// The closing """ position sets the "incidental whitespace" baseline — indentation left of it is stripped,
// so you can indent the block to match your code without it appearing in the string.

String sql = """
    SELECT id, name
    FROM users
    WHERE active = true
    """;
💡 Text blocks are ideal for embedded JSON, SQL, HTML, and any multi-line literal — readable, no \n or \" clutter. Two subtleties: the position of the closing """ determines how much leading whitespace is stripped (align it under your content), and a trailing newline appears only if the closing quotes are on their own line. Use \ at line-end to suppress a newline, and \s to preserve trailing spaces.

Part 3 — Convenient Collection & Stream Additions

// Immutable factory methods (9+) — concise constants and returns:
List<String> days = List.of("Mon", "Tue", "Wed");     // immutable
Map<String,Integer> m = Map.of("a", 1, "b", 2);
Set<Integer> s = Set.of(1, 2, 3);

// Stream.toList() (16+) — shorter than collect(Collectors.toList()):
List<String> upper = names.stream().map(String::toUpperCase).toList();

// Enhanced String methods:
"  hi  ".strip();              // Unicode-aware trim (11+)
"  ".isBlank();               // true — whitespace-only (11+)
"ab".repeat(3);              // "ababab" (11+)
"a\nb\nc".lines();           // Stream<String> of lines (11+)
"Hi %s".formatted("Ada");    // instance formatting (15+)

// Records, sealed types, pattern matching (covered in their own lessons) round out modern Java.

🔍 Target an LTS. Companies standardize on Long-Term Support releases (8, 11, 17, 21), so "what Java version?" usually means one of those. Java 8 is the legacy floor still seen in older systems; 17 and 21 are today's modern baselines. Knowing which feature is available on which version matters in practice (you can't use records on Java 8) and in interviews. Features ship via 6-month releases and mature over several versions (pattern matching arrived in pieces from 16 to 21), so "preview" features become stable a version or two later.

Part 5 — Other Modern APIs Worth Knowing

  • The HTTP Client (11). java.net.http.HttpClient replaced the clunky HttpURLConnection with a modern, fluent, async-capable client — no third-party library needed for basic HTTP.
  • The module system (9, Project Jigsaw). module-info.java lets you declare explicit dependencies and encapsulation boundaries between modules. Powerful but adopted slowly; most application code still uses the classpath, while the JDK itself is modularized.
  • Sequenced collections (21). A unifying SequencedCollection interface giving getFirst()/getLast()/reversed() across lists, deques, and ordered sets — filling a long-standing gap.
  • Switch expressions, enhanced generics, and more continue to arrive — the language evolves continuously now, so "modern Java" is a moving target you keep up with via release notes and MDN-equivalent resources like the JDK documentation.

Part 6 — Deep Dive: Concise vs Clear (the judgment)

Modern features let you write dramatically less code — but "less code" and "clearer code" aren't always the same, and mature engineers exercise judgment:

  • var trades explicitness for brevity. Use it where the type is obvious (constructor calls, literals); avoid it where the reader would have to hunt for the type (a method call returning a non-obvious type). The goal is readability, not fewer characters.
  • Text blocks improve embedded literals but tempt overuse. A large SQL statement embedded in Java is readable as a text block, but a truly large query might belong in a .sql file. Match the tool to the size.
  • Records, streams, and pattern matching are usually clearer — they express intent directly. But a dense stream pipeline or nested pattern can become write-only code; break it up or fall back to a loop when clarity demands.
  • Consistency matters. Follow the codebase's conventions on var, text blocks, and functional style — a mixed style is harder to read than either extreme.
🔍 Senior framing: the arc of modern Java is from verbose-and-explicit toward concise-and-declarative, and the skill is knowing when concision aids comprehension and when it obscures. The best code uses these features to make intent obvious — a record says "this is data," a stream says "this is a transformation," a sealed switch says "these are all the cases" — not to win a golf contest. When a modern feature makes the reader work harder to understand what the code does, it's the wrong call, however fashionable. Prefer the version that a competent colleague reads correctly on the first pass. That judgment — not knowledge of the syntax — is what distinguishes fluent modern-Java engineers.

Interview Questions

Q1 (beginner): Is var dynamic typing? What are its limits?

No. var is local type inference: the compiler infers the variable's type from its initializer and fixes it permanently, so it remains fully statically typed — var x = 10 makes x an int, and assigning a String to it later won't compile. Limits: it requires an initializer (there's nothing to infer from otherwise), it works only for local variables (not fields, method parameters, or return types), and it can't be used where the target type is needed to resolve the expression (a bare lambda or method reference). Its purpose is reducing redundant type noise, not adding runtime flexibility.

Q2 (beginner): What are text blocks and when do you use them?

Text blocks (Java 15) are multi-line string literals delimited by triple quotes (""") that preserve line breaks and require no escaping of newlines or embedded quotes. They're for multi-line content — JSON, SQL, HTML, formatted messages — replacing the old concatenation-with-\n-and-\" soup with something readable that looks like the literal it produces. The position of the closing """ sets the incidental-whitespace baseline, so you can indent the block to match surrounding code without that indentation appearing in the string. Use them for embedded literals of moderate size; very large content may belong in an external resource file.

Q3 (mid): When does var help readability and when does it hurt?

It helps when the type is already obvious from the right-hand side or is verbose: var users = new ArrayList<User>() avoids repeating the type, and var entry = Map.entry(...) spares a long generic type — the reader loses nothing. It hurts when the type isn't apparent from the initializer, most commonly a method call: var result = service.process() forces the reader to go find what process() returns. The guiding principle is that var should reduce noise, not hide information the reader needs; use an explicit type whenever seeing it aids comprehension, and follow the team's convention for consistency.

Q4 (mid): Which LTS versions matter, and name a feature that requires a modern one.

The Long-Term Support releases companies standardize on: Java 8 (the legacy floor — lambdas/streams/ java.time), Java 11 (var, HTTP client, string methods), Java 17 (records stable, sealed types, text blocks), and Java 21 (virtual threads, pattern-matching switch, record patterns). Features requiring modern LTS: records need 16+/17, sealed types need 17, virtual threads and pattern-matching switch need 21 — none work on Java 8. Because new versions ship every six months and features often arrive as previews before stabilizing, knowing the version map tells you what you can actually use on a given deployment target.

Q5 (senior): How do you decide whether to adopt a concise modern feature in a given piece of code?

The deciding criterion is whether the feature makes the code's intent clearer to the next reader, not whether it's shorter or newer. Concretely: does it reduce genuine noise (repeated types, escaping, boilerplate) while keeping the meaning obvious? Then adopt it — a record announces "this is data," a stream announces "this is a transformation," a text block shows the literal as-is. Does it instead hide information the reader needs (a var masking a non-obvious return type), or pack so much into one expression that it becomes write-only (a deeply nested pattern or a dense multi-stage stream)? Then prefer the more explicit form or break it up. Two further considerations: availability — the feature must exist on the target LTS — and consistency — matching the codebase's established style is more readable than locally optimal but inconsistent choices. The senior instinct is that these features are tools for expressing intent, and the right amount of each is "as much as makes the code read correctly on the first pass, and no more"; reflexively maximizing or avoiding any of them is a mistake. Readability for the team over cleverness for the author.

Q6 (senior): Modern Java ships features every six months. How should a team keep its codebase current without churn?

Treat version and feature adoption as a deliberate, periodic decision rather than either freezing on an old release or chasing every feature. Practically: standardize on an LTS (17 or 21 today) and plan upgrades on a cadence, since LTS releases get long support and are what libraries and tooling target; skipping straight between LTS versions is a common, manageable strategy. When adopting, prioritize features that remove real pain (records for the pervasive value-object boilerplate, try-with-resources, var for verbose types, virtual threads for I/O-bound scalability) and let cosmetic ones spread organically through code review conventions rather than mass refactors — rewriting working code purely to use a new syntax is churn with risk and little benefit. Establish team conventions (a style guide entry on var, text blocks, functional style) so usage is consistent, enable the compiler's new diagnostics and static analysis to catch issues, and pilot preview features in non-critical paths before they stabilize. Keep dependencies and build tooling current alongside the JDK, because that's often where upgrade friction actually lives. The balance is "stay on a supported modern LTS, adopt high-value features intentionally, and avoid rewrite-for-fashion" — currency without thrash.

🏋️ Exercises (attempt before opening solutions)

1 (easy): var judgment. For each, decide if var is appropriate: var list = new ArrayList<String>();, var total = calculateTotal();, var i = 0;. Justify.
Solution

(1) Appropriate — the type is obvious from new ArrayList<String>(); repeating it adds nothing. (2) Questionable — calculateTotal()'s return type isn't visible, so the reader must go look; an explicit type is clearer. (3) Fine — 0 obviously infers int. The pattern: obvious-from-RHS good, hidden-behind-a-call bad.

2 (easy): Text block. Write a JSON object as a text block and confirm no \n or escaped quotes are needed. Note where the closing """ goes.
Solution

A """-delimited block with the JSON indented under it and the closing """ aligned to the content baseline. Real newlines and unescaped " inside just work. The closing quotes' column determines how much leading indentation is stripped; putting them on their own line adds a trailing newline. Far more readable than concatenated escaped strings.

3 (medium): Modern collection code. Build an immutable list of three names, stream it to uppercase, and collect with toList() (not collect(Collectors.toList())). Then try to modify the immutable source.
Solution

var names = List.of("ada","grace","linus"); var upper = names.stream().map(String::toUpperCase).toList();. names.add("x") throws UnsupportedOperationException — List.of is immutable. Stream.toList() (16+) is the concise terminal, and it also returns an unmodifiable list.

4 (medium): Version awareness. For records, sealed types, virtual threads, and var, state the minimum Java version. Which work on Java 11? On Java 17?
Solution

var: 10/11. Records: 16 (stable 17). Sealed types: 17. Virtual threads: 21. On Java 11: only var. On Java 17: var, records, sealed types — but not virtual threads. Knowing this prevents "why won't this compile on our JDK?" surprises and answers a common interview check.

5 (hard): Refactor to modern Java. Take a verbose class: a hand-written value object, a concatenated multi-line SQL string, and a collect(Collectors.toList()) pipeline. Modernize all three and note the judgment calls.
Solution

Value object → a record (removes ~40 lines, correct equals/hashCode). SQL → a text block (readable, no escaping) — or an external .sql file if it's large. Pipeline → .toList(). Judgment calls: keep the SQL inline only if it's small; use var only where types stay obvious; ensure the target JDK supports records/text blocks (16/15+). The wins are real, but each choice is weighed for clarity and availability, not applied blindly.

6 (hard): Explain-it test. Out loud: (a) why var is static inference not dynamic typing, and its readability rule; (b) what text blocks solve and the closing-quote subtlety; (c) the concise-vs-clear judgment for adopting modern features.
Self-check rubric

Pass (a) if you said compile-time-fixed type, locals-only, use where type is obvious. Pass (b) if you said multi-line literals without escaping and closing-quote position controls whitespace. Pass (c) if you framed adoption around clarity-of-intent and availability, not brevity for its own sake. This closes the Modern Java section.

🎯 Key Takeaways

  • var is compile-time local type inference (not dynamic) — use it where the type is obvious, avoid it where it hides a needed type.
  • Text blocks (""") give clean multi-line literals for JSON/SQL/HTML; the closing-quote position controls whitespace stripping.
  • Modern conveniences: List.of/Map.of (immutable), Stream.toList(), and enhanced String methods (strip, isBlank, repeat, lines).
  • Target an LTS (8/11/17/21) and know the version map — records need 16+, sealed types 17, virtual threads 21.
  • Also modern: the HTTP client (11), the module system (9), sequenced collections (21).
  • Exercise judgment: use these features to make intent clearer, not merely to write less — prefer the version a colleague reads correctly first pass.
