Records, Sealed & Modern Java Intermediate

Java since 17 (the current LTS baseline; 21 added more) is far less verbose. Records kill data-class boilerplate, sealed types enable exhaustive modeling, and pattern matching ties them together. This lesson is the modern Java a mid/senior engineer is expected to write — not the 2010 style.

Records — Immutable Data Classes

public record Point(int x, int y) { }              // that's it — the whole class
// Auto-generates: constructor, x()/y() accessors, equals(), hashCode(), toString().
var p = new Point(1, 2);
p.x();  p.equals(new Point(1, 2));  // true — value equality, correct hashCode (collections lesson!)

// Records can have validation (compact constructor), extra methods, static factories:
public record Money(long cents, String currency) {
    public Money {                                 // compact constructor — validate/normalize
        if (cents < 0) throw new IllegalArgumentException("negative");
        currency = currency.toUpperCase();
    }
    public Money plus(Money o) { return new Money(cents + o.cents, currency); }
    public static Money usd(long c) { return new Money(c, "USD"); }
}
// Records are IMMUTABLE and final — perfect for DTOs, value objects, map keys, and
// multiple return values. They solve the equals/hashCode boilerplate correctly, every time.

Sealed Types — Controlled Hierarchies

public sealed interface Shape permits Circle, Rectangle, Triangle { }
public record Circle(double r) implements Shape { }
public record Rectangle(double w, double h) implements Shape { }
public record Triangle(double base, double height) implements Shape { }
// 'sealed' = only the PERMITTED types may implement it. The compiler knows the COMPLETE set,
// enabling EXHAUSTIVE switches (no default needed — and a NEW subtype breaks compilation
// everywhere it's unhandled, so you can't forget a case).

Pattern Matching — the Payoff

// instanceof pattern (16+): test + bind in one:
if (obj instanceof String s && s.length() > 3) { use(s); }   // s is in scope, typed

// switch with patterns + record DECONSTRUCTION (21+) — exhaustive over a sealed type:
double area = switch (shape) {
    case Circle(double r)        -> Math.PI * r * r;         // destructure the record!
    case Rectangle(double w, double h) -> w * h;
    case Triangle(double b, double h)  -> 0.5 * b * h;
    // no default needed — sealed + records make this exhaustive; adding Square won't compile
};
// This is algebraic-data-type-style modeling: a closed set of shapes + exhaustive handling.
// The same discriminated-union idea as the TS/Python lessons, now type-safe in Java.

Interview Questions

Q1: What does a record give you, and when is it the wrong choice?

A record auto-generates a canonical constructor, component accessors, and correct equals/hashCode/ toString, and is immutable and final — ideal for DTOs, value objects, map keys, and multiple return values. Wrong when you need mutability, an identity distinct from field values (JPA entities, objects with lifecycle), inheritance of state (records can't extend classes), or hidden internal fields. Rule: records model VALUES (equal-by-content); classes model ENTITIES (equal-by-identity).

Q2: What do sealed types enable that a normal interface doesn't?

They close the set of implementers to a compiler-known list (permits), so switches over them can be EXHAUSTIVE without a default — and adding a new permitted subtype causes compile errors at every non-updated switch, forcing you to handle it. This gives algebraic-data-type modeling (a closed sum of variants) with compile-time completeness checking — the type-safe version of the discriminated unions from the TS/Python lessons, invaluable for state machines and expression trees.

Q3: How do records + sealed + pattern matching combine into a design style?

Model a domain as a sealed interface with record variants (a closed sum type), then handle it with an exhaustive switch that deconstructs each record's components. The compiler guarantees you cover every case and re-checks when the set changes. This replaces the visitor pattern and instanceof-cast chains with concise, safe, refactor-friendly code — e.g., an AST, a result type (Success/Failure), or UI states. It's Java catching up to functional pattern matching.

🏋️ Practice: replace a verbose data class with a record and confirm equals/hashCode work in a HashSet; add validation via a compact constructor; model Shape as a sealed interface with record variants and write an exhaustive area() switch; deconstruct records in the switch; prove that adding a new variant breaks the switch until handled; use a text block for a JSON template.

🎯 Key Takeaways

  • Records = immutable value classes with auto equals/hashCode/toString — for DTOs, value objects, keys.
  • Compact constructors validate/normalize; records can have methods and static factories.
  • Sealed types close the implementer set → exhaustive switches; new variants force you to handle them.
  • Pattern matching (instanceof binding, switch with record deconstruction) = safe, concise dispatch.
  • records + sealed + patterns = algebraic-data-type modeling (the type-safe discriminated union).
