JVM Internals & GC Advanced

Java runs on the JVM, which manages memory automatically via garbage collection and optimizes code at runtime via the JIT. You don't free memory manually — but at senior level you must understand the memory model, how GC works, and how to diagnose leaks and pauses, because production Java problems are usually memory/GC problems.

JVM Memory Layout

Thread stacks / PCExecution state per thread—

The heap is split by generation (the "weak generational hypothesis": most objects die young): Young gen (Eden + two Survivor spaces) for new objects, Old gen for long-lived survivors.

How Garbage Collection Works

Reachability: an object is GARBAGE when unreachable from GC ROOTS (stack locals, statics,
active threads). Unreachable ≠ "no references" — cycles are collected because the CYCLE is
unreachable from roots (unlike naive reference counting).

Generational collection:
  Minor GC: collects Young gen — FAST and frequent. Survivors age; long-lived ones get
            promoted to Old gen.
  Major/Full GC: collects Old gen — slower, can cause a longer "stop-the-world" PAUSE.
Goal of tuning: keep pauses short and throughput high by minimizing promotion and Full GCs.

Memory Leaks in a GC'd Language (yes, they happen)

// A "leak" = objects stay REACHABLE by accident, so GC can't free them. Classic causes:
static final Map<K,V> CACHE = new HashMap<>();   // unbounded static cache — grows forever
list.add(obj);   // never removed — a collection that only grows
// forgotten listeners/callbacks holding references; ThreadLocals not removed in pools;
// mutable keys in a HashMap. Fixes: bounded caches (Caffeine/LRU), WeakReference/WeakHashMap
// for canonicalizing maps, remove listeners, clean up ThreadLocals. Same reachability logic
// as the JS memory lesson — GC frees the unreachable; leaks are accidental reachability.
🔬 Diagnosing: rising Old-gen after each Full GC (memory not reclaimed) = a leak. Take a heap dump (jmap -dump or on OOM with -XX:+HeapDumpOnOutOfMemoryError), open in Eclipse MAT / VisualVM, and find the "dominator tree" — the objects retaining the most memory and the reference path (GC root) keeping them alive. Exactly the retainer-path workflow from the JS memory lesson.

Interview Questions

Q1: How does generational GC work and why is it efficient?

It exploits the weak generational hypothesis — most objects die young. New objects go in the Young gen (Eden); frequent, fast Minor GCs collect it, and the few survivors are aged and eventually promoted to the Old gen, which is collected less often by slower Major/Full GCs. Because the vast majority of garbage is in the cheaply-collected Young gen, overall GC cost is low. Tuning aims to minimize promotion (so Old gen fills slowly) and avoid long Full-GC stop-the-world pauses; modern collectors (G1/ZGC) do most work concurrently to keep pauses tiny.

Q2: Can a garbage-collected language leak memory? Give causes and how you'd find it.

Yes — a leak is objects that remain REACHABLE from GC roots by accident, so GC won't free them. Causes: unbounded static caches/collections that only grow, forgotten listeners/callbacks holding references, ThreadLocals not removed in pooled threads, mutable keys, and long-lived objects retaining short-lived ones. Diagnose by watching Old-gen usage after Full GCs (rising = leak), then a heap dump analyzed in MAT/VisualVM — the dominator tree shows the biggest retainers and the GC-root reference chain keeping them alive. Fix by breaking that reference (bounded caches, Weak/SoftReferences, removing listeners).

Q3: What does the JIT do, and why must Java benchmarks warm up?

The JIT profiles bytecode at runtime and compiles hot methods to optimized native code, inlining and specializing based on observed behavior — so steady-state Java runs near native speed, and escape analysis can even elide heap allocations. But early executions run interpreted or un-optimized, so a naive microbenchmark measures cold, slow code and misleads. You must warm up (run enough iterations for JIT compilation to kick in) — which is why JMH exists. It also means startup is slower than steady state, motivating AOT/native-image (GraalVM) for serverless.

🏋️ Practice: run a program with -Xlog:gc* and observe Minor vs Full GC; create a leak with an unbounded static map, watch memory climb, then bound it; trigger an OOM with -XX:+HeapDumpOnOutOfMemoryError and open the dump in VisualVM to find the retainer; set -Xms=-Xmx and observe steadier GC; write a JMH benchmark and compare warmed vs cold timing.

🎯 Key Takeaways

  • Heap holds objects (GC-managed); stack holds frames/locals; know the regions.
  • Generational GC: fast frequent Minor GC (Young), slower Full GC (Old); tuning minimizes pauses & promotion.
  • Pick the collector by goal: G1 (default/balanced), ZGC/Shenandoah (low pause), Parallel (throughput).
  • GC frees the UNREACHABLE — leaks are accidental reachability (unbounded caches, listeners, ThreadLocals); diagnose with heap dumps + dominator tree.
  • JIT compiles hot code to native (warm up benchmarks with JMH); native-image trades startup for peak throughput.
