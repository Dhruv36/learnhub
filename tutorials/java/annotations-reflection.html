Annotations & Reflection Core Libraries

What you'll master in this lesson: what annotations actually are (metadata, not behavior), how retention policies decide whether an annotation survives to runtime, how to write your own annotation and read it with reflection, what reflection costs and why frameworks lean on it, and — the senior distinction — why annotation processing (compile-time) and code generation are usually better than runtime reflection. This is the lesson that demystifies how Spring, JPA, JUnit, and Jackson work under the hood: they're all reading annotations via reflection. Prerequisites: Interfaces, Generics, and Classes & Objects.

Part 1 — Annotations Are Metadata, Not Code

@Override                       // tells the compiler "this MUST override a supertype method" (compile check)
public String toString() { ... }

@Deprecated                     // marks an API as obsolete — compiler warns callers
@SuppressWarnings("unchecked")  // silences a specific compiler warning
@FunctionalInterface            // compiler enforces exactly one abstract method (Lambdas lesson)
🎯 An annotation does nothing by itself — it's structured metadata attached to code that something else reads and acts on. @Override is read by the compiler (which errors if the method doesn't actually override anything); @Test is read by JUnit (which finds and runs the method); @Entity is read by JPA. The annotation is inert data; the behavior lives in the processor — a compiler check, a test runner, a framework's startup scan. Understanding this split (metadata here, behavior elsewhere) is the key to reasoning about every annotation-driven framework you'll use.

Part 2 — Retention: Does the Annotation Survive to Runtime?

@Retention(RetentionPolicy.SOURCE)   // discarded after compile — for tools/compiler only (e.g. @Override)
@Retention(RetentionPolicy.CLASS)    // in the .class file but NOT loaded at runtime (default; rarely used)
@Retention(RetentionPolicy.RUNTIME)  // kept and readable via reflection at runtime — what frameworks need

@Target(ElementType.METHOD)          // where it may be placed: TYPE, FIELD, METHOD, PARAMETER, ...
@Retention(RetentionPolicy.RUNTIME)  // MUST be RUNTIME for Spring/JUnit/JPA to read it reflectively
public @interface Loggable { }       // @interface DECLARES a new annotation type
RetentionLives untilExample use
SOURCECompilation only@Override, Lombok, annotation processors
CLASSIn the bytecode, not at runtimeBytecode tools; the default
RUNTIMEReadable via reflection@Test, @Entity, @Autowired
🔑 Retention is the single most important thing to understand about custom annotations. If a framework reads an annotation at runtime (Spring scanning beans, JUnit finding tests), that annotation must be @Retention(RUNTIME) — otherwise it's gone by the time the code runs and the framework sees nothing. A very common "why isn't my custom annotation working?" bug is forgetting RUNTIME retention. Conversely, annotations consumed at compile time (Lombok, annotation processors, @Override) use SOURCE because they've done their job before the program ever runs.

Part 3 — Writing and Reading Your Own Annotation

// 1. Declare it, with elements (they look like methods, can have defaults):
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Retryable {
    int attempts() default 3;                 // usage: @Retryable(attempts = 5)
    Class<? extends Throwable>[] on() default { Exception.class };
}

// 2. Use it:
public class PaymentService {
    @Retryable(attempts = 5)
    public void charge() { ... }
}

// 3. Read it via reflection and act on it:
Method m = PaymentService.class.getMethod("charge");
if (m.isAnnotationPresent(Retryable.class)) {
    Retryable r = m.getAnnotation(Retryable.class);
    int n = r.attempts();                      // 5 — now YOUR code can implement retry behavior
}
🔧 This three-step pattern — declare, annotate, reflectively read — is exactly how every annotation-driven framework works. Spring's @Transactional, JUnit's @Test, and Jackson's @JsonProperty are all custom annotations with RUNTIME retention that the framework discovers by reflecting over your classes at startup and then wires behavior around. When you write @Retryable and a bit of reflection to honor it, you've built a miniature version of what Spring AOP does. Elements (attempts(), on()) are the annotation's configuration, and they may only be constants, enums, class literals, strings, or arrays of those — not arbitrary objects.

Part 4 — Reflection: Inspecting & Invoking at Runtime

Class<?> c = obj.getClass();                     // the runtime Class object

// Inspect structure:
for (Field f  : c.getDeclaredFields())  { ... }  // fields (incl. private)
for (Method m : c.getDeclaredMethods()) { ... }  // methods

// Access private members (requires setAccessible — and is what breaks encapsulation):
Field f = c.getDeclaredField("balance");
f.setAccessible(true);                            // bypass private (can be blocked by the module system / security)
Object value = f.get(obj);

// Create instances and invoke dynamically:
Object instance = c.getDeclaredConstructor().newInstance();   // like `new`, but reflective
Method m = c.getMethod("charge");
m.invoke(instance);                               // call it by name, resolved at runtime
🔍 Reflection lets code inspect and manipulate classes it didn't know about at compile time — read fields, call methods by name, instantiate types, all decided at runtime. That's the superpower behind dependency injection (Spring instantiates your beans and injects fields it discovered), ORMs (JPA reads/writes entity fields), serializers (Jackson maps JSON to object fields), and test runners. But it comes at a cost and it breaks the compiler's guarantees: a method called via invoke isn't checked at compile time, setAccessible(true) defeats private (and the module system may forbid it), and refactoring tools can't see reflective usage. Reflection is a framework tool, not something you reach for in ordinary application code.

Part 5 — The Cost of Reflection & the Compile-Time Alternative

// Reflection is SLOWER than a direct call (metadata lookup, no inlining) and defers errors to runtime:
m.invoke(obj);        // a typo in the method name compiles fine, fails at runtime with an exception

// The modern alternative — ANNOTATION PROCESSING (compile-time code generation):
//   Lombok, MapStruct, Dagger, and Spring's AOT engine read annotations at COMPILE time and
//   GENERATE real Java code — so the runtime does direct calls, not reflection.
//   Benefits: full speed, errors caught at compile time, works with GraalVM native image.
⚡ Prefer compile-time processing over runtime reflection when you can. Reflection is flexible but slow (no JIT inlining, metadata lookups) and pushes errors to runtime. Annotation processors run during compilation, read the same annotations, and generate ordinary source code — so the result is as fast as hand-written code, type-checked at compile time, and compatible with ahead-of-time compilation (GraalVM native image, which can't do arbitrary runtime reflection). This is why modern frameworks are shifting toward AOT: MapStruct generates mappers, Dagger generates DI wiring, and Spring's AOT engine pre-computes at build time what older Spring did reflectively at startup. Knowing that reflection and annotation processing are two ways to consume the same annotations — one at runtime, one at compile time — is a strong senior signal.

Part 6 — Deep Dive: Why This Powers Every Framework You Use

  • Annotations decouple "what" from "how." You declare intent (@Transactional, @Test, @Entity) and the framework supplies the behavior — so your code stays declarative and the mechanism stays in the framework.
  • Reflection is what reads that intent at runtime. Startup scanning, DI, ORM mapping, and serialization all reflect over your annotated classes. This is why those annotations need RUNTIME retention and why frameworks have a "component scan" phase.
  • The proxy connection. Once a framework knows (via reflection) that a method is @Transactional or @Cacheable, it wraps the bean in a proxy that adds the behavior — which is exactly the AOP mechanism and the source of the self-invocation gotcha (Spring lessons). It all traces back to reading an annotation.
  • Application code rarely needs reflection directly. If you're reaching for setAccessible(true) in business logic, it's usually a design smell; reflection belongs in libraries and frameworks, and even there, compile-time generation is increasingly preferred.
🔍 Senior framing: annotations and reflection together are the mechanism that makes the entire Java framework ecosystem possible, and understanding them turns Spring, JPA, JUnit, and Jackson from magic into predictable machinery. The mental model is a clean separation: an annotation is inert metadata that declares intent on a program element, and the behavior lives in a processor that reads it — the compiler for @Override, a test runner for @Test, a DI container for @Autowired. Whether the processor can even see the annotation is governed by retention: runtime frameworks require RUNTIME so they can discover it via reflection, the runtime capability to inspect and invoke code chosen by name rather than fixed at compile time — which is precisely how a container instantiates beans it never imported, an ORM reads private entity fields, and a serializer maps JSON to objects. That power has real costs: reflection is slower (no inlining, metadata lookups), it defeats the compiler's type and access guarantees (setAccessible bypasses private, a mistyped method name fails only at runtime), and it fights ahead-of-time compilation — which is why the modern trajectory is toward compile-time annotation processing and code generation (Lombok, MapStruct, Dagger, Spring AOT) that consumes the same annotations at build time to emit fast, type-checked, native-image-friendly code. Recognizing that reflection-driven annotations lead directly to proxies (once the framework knows a method is transactional/cacheable it wraps the bean) closes the loop to the AOP and self-invocation topics, showing these aren't separate features but one story. The mature engineer therefore uses annotations to keep application code declarative, understands the reflective machinery well enough to debug "why isn't my annotation being picked up?" (almost always a retention or scanning issue), avoids reflection in ordinary business logic, and knows when compile-time generation is the better tool — which is exactly the depth an interviewer is checking when they ask "how does Spring know to make this method transactional?" or "how would you implement a custom annotation like @Test?"

Interview Questions

Q1 (beginner): What is an annotation, and does it change program behavior on its own?

An annotation is metadata — structured information attached to a program element (class, method, field, parameter) — and by itself it does nothing to program behavior. It's inert data that some other component reads and acts upon: the compiler acts on @Override (verifying the method really overrides a supertype method) and @Deprecated (warning callers); JUnit acts on @Test (discovering and running the method); JPA acts on @Entity; Spring acts on @Autowired. So the behavior always lives in a processor — a compiler check, a test runner, a framework's startup scan — not in the annotation. This separation of declarative intent (the annotation) from the mechanism (the processor) is what makes annotations powerful and is the key to understanding every annotation-driven framework.

Q2 (beginner): What does @Retention(RUNTIME) do and why does it matter?

Retention controls how long an annotation is kept. SOURCE means it's discarded after compilation (used by the compiler or source-level tools like Lombok); CLASS (the default) keeps it in the .class file but it isn't loaded at runtime; RUNTIME keeps it and makes it readable via reflection while the program runs. It matters because any framework that reads an annotation at runtime — Spring finding beans, JUnit finding tests, JPA mapping entities — can only see annotations with RUNTIME retention. If you write a custom annotation intended for a runtime framework and forget @Retention(RUNTIME), it's gone by the time the code executes and the framework sees nothing — one of the most common "my custom annotation isn't working" bugs. Compile-time-only annotations use SOURCE because they've already done their job before the program runs.

Q3 (mid): How would you implement a custom annotation like @Test or @Retryable?

Three steps. (1) Declare the annotation with @interface, giving it @Target (where it can go, e.g. METHOD) and @Retention(RUNTIME) (so it's readable at runtime), plus any elements for configuration (e.g. int attempts() default 3;). (2) Use it on the relevant methods. (3) Read and act on it via reflection: obtain the Class/Method, check isAnnotationPresent(Retryable.class), call getAnnotation(...) to read its elements, and then implement the behavior — for @Test, discover all methods carrying it and invoke each; for @Retryable, wrap the invocation in a retry loop honoring attempts(). This is exactly how JUnit and Spring work: they scan your classes reflectively, find methods carrying their runtime-retained annotations, and wire behavior around them. Building this yourself demonstrates you understand that the annotation is just metadata and the processor supplies the behavior.

Q4 (mid): What is reflection, what does it enable, and what are its downsides?

Reflection is the ability to inspect and manipulate classes, methods, and fields at runtime — obtaining a Class object, listing its members, reading/writing fields (even private ones via setAccessible), instantiating types, and invoking methods chosen by name rather than fixed at compile time. It enables the core of the framework ecosystem: dependency injection (a container instantiates and wires beans it discovered), ORMs (reading/writing entity fields), serialization (mapping JSON to object fields), and test runners (finding and invoking test methods) — all operate on classes they didn't know at compile time. The downsides are significant: it's slower than direct calls (metadata lookups, no JIT inlining), it defeats compile-time safety (a method invoked reflectively isn't type-checked, so a typo fails only at runtime; setAccessible(true) bypasses private, breaking encapsulation and possibly blocked by the module system), and it's invisible to tooling (refactoring and dead-code analysis can't follow reflective calls). Because of this, reflection belongs in libraries and frameworks, not ordinary application logic, and modern frameworks increasingly replace it with compile-time code generation.

Q5 (senior): When would you prefer compile-time annotation processing over runtime reflection, and why?

Prefer compile-time annotation processing whenever the work can be decided at build time rather than needing genuine runtime dynamism — which covers most framework wiring. An annotation processor runs during compilation, reads the same annotations, and generates ordinary Java source (or bytecode) that does the work directly; examples are Lombok (boilerplate), MapStruct (object mappers), Dagger (DI graphs), and Spring's AOT engine (pre-computed bean wiring). The advantages over runtime reflection are concrete: (1) performance — generated code is direct method calls the JIT can inline, with no per-call metadata lookup, so it's as fast as hand-written code; (2) compile-time safety — errors (a missing mapping, an unsatisfiable dependency) are caught at build time rather than surfacing as a runtime exception in production; (3) ahead-of-time compatibility — GraalVM native image and other AOT compilers use closed-world analysis and can't perform arbitrary runtime reflection, so reflection-heavy frameworks need extensive hints or break, whereas generated code just works; and (4) transparency — the generated source is real code you can read and debug, versus opaque reflective magic. The trade-offs are longer/more complex builds and less runtime flexibility (you can't adapt to types unknown at compile time). So the senior heuristic is: if the set of types and behaviors is known at build time, generate code; reserve runtime reflection for genuinely dynamic scenarios (plugins, truly late-bound types). This is exactly why the industry — and Spring itself — is moving from startup-time reflection toward build-time AOT, and being able to articulate that shift and its motivations (native image, startup speed, memory) signals real depth.

Q6 (senior): Trace how @Transactional goes from an annotation to actual transactional behavior.

It's a chain of the concepts in this lesson. @Transactional is a custom annotation declared with @Retention(RUNTIME) so it survives to runtime. During startup, Spring performs a reflective scan of your beans, and when it finds a bean with @Transactional methods, it doesn't register the raw bean — it creates a proxy that wraps it (via JDK dynamic proxies or CGLIB subclassing). That proxy's job, established because reflection revealed the annotation, is to open a transaction before the method, commit on success, and roll back on the appropriate exceptions, delegating to your real method in between. So the annotation (metadata) is read by reflection (the mechanism), which drives the framework to insert a proxy (the behavior) — the same three-part story (metadata → reflection → proxy) behind @Cacheable, @Async, and Resilience4j's annotations. This also explains the famous self-invocation gotcha: because the behavior lives in the proxy that wraps the bean, only calls arriving through the proxy from another bean get it — an in-class this.method() call bypasses the proxy and silently gets no transaction. And it explains why Spring's AOT mode pre-computes this at build time to avoid the startup reflection. Being able to narrate the full path — runtime retention, component scan, reflection, proxy creation, and the resulting self-invocation limitation — demonstrates you understand not just that Spring is "annotation-driven" but the actual machinery, which is what separates someone who uses the framework from someone who understands it.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Retention experiment. Declare a custom annotation with SOURCE retention, try to read it reflectively, and explain what happens.
Solution

isAnnotationPresent/getAnnotation return false/null because a SOURCE-retained annotation is discarded after compilation and isn't in the runtime class metadata. Change it to @Retention(RUNTIME) and reflection can now see it — demonstrating why runtime frameworks require RUNTIME retention.

2 (easy): Read an annotation's element. Write @Retryable(attempts=5), put it on a method, and reflectively print the attempts value.
Solution

Get the Method, call m.getAnnotation(Retryable.class).attempts() → 5. The element is read like a method call on the annotation instance. This is how a framework reads an annotation's configuration to drive behavior.

3 (medium): Mini test runner. Given a class with methods annotated @Test, use reflection to find and invoke all of them, reporting pass/fail.
Solution

Iterate getDeclaredMethods(), filter by isAnnotationPresent(Test.class), newInstance() the class, and invoke each test method inside a try/catch — no exception = pass, exception = fail. That's the core of what JUnit does, proving the annotation is just metadata and your runner supplies the behavior.

4 (medium): Reflection cost. Explain (or micro-measure) why calling a method via Method.invoke is slower than a direct call.
Solution

Reflective invocation involves metadata lookup, access checks, argument boxing into an Object[], and it inhibits JIT inlining/optimization that a direct, statically-known call enjoys. It's fine for one-time framework setup but not a hot loop — which is one reason frameworks cache reflective handles or move to generated code.

5 (hard): Reflection vs codegen. Argue when you'd implement a bean mapper with runtime reflection vs a compile-time processor like MapStruct.
Solution

Reflection-based mapping is flexible and quick to write but slow (per-field reflection) and errors surface at runtime; a compile-time processor (MapStruct) generates direct field-copy code that's fast, type-checked at build time, and native-image-friendly. Prefer codegen when the types are known at compile time (almost always for DTO↔entity mapping); reserve reflection for genuinely dynamic cases.

6 (hard): Explain-it test. Out loud: (a) annotation = metadata + processor = behavior; (b) why retention governs framework visibility; (c) how @Transactional becomes a proxy and why self-invocation bypasses it.
Self-check rubric

Pass (a) if you said the annotation is inert and a compiler/runner/container supplies behavior. Pass (b) if you said runtime frameworks need RUNTIME retention to reflect over it. Pass (c) if you traced metadata → reflective scan → proxy, and explained an in-class call bypasses the proxy. That's the whole machinery of annotation-driven frameworks.

🎯 Key Takeaways

  • An annotation is metadata — it does nothing itself; a processor (compiler, test runner, framework) supplies the behavior.
  • Retention decides visibility: runtime frameworks need @Retention(RUNTIME) or they can't see your annotation.
  • Custom annotations follow declare → annotate → reflectively read — exactly how @Test/@Entity/@Autowired work.
  • Reflection inspects/invokes code at runtime (powering DI, ORM, serialization) but is slower, unsafe (bypasses private), and untyped.
  • Prefer compile-time annotation processing / codegen (Lombok, MapStruct, Spring AOT) for speed, compile-time safety, and native-image support.
  • Annotation + reflection → proxy is the chain behind @Transactional — and why self-invocation bypasses it.
