Immutability & Defensive Copying Object-Oriented Java

What you'll master in this lesson: what makes an object truly immutable and the concrete recipe to build one; why immutability gives you thread-safety, safe sharing, and reliable hash keys for free; the difference between final and immutable (they're not the same); the defensive-copying technique that plugs the leaks a naive "immutable" class still has; and when immutability costs too much. Immutability is the single highest-leverage design habit for correctness — especially in concurrent code — and a recurring interview theme.

Part 1 — What "Immutable" Means

An immutable object's observable state cannot change after construction. You've already met the canonical example — String — where every "modifying" method returns a new object. An immutable class makes the same guarantee for your own types.

public final class Money {                    // final: no subclass can add mutability
    private final String currency;           // final fields: assigned once, in the constructor
    private final long amountCents;

    public Money(String currency, long amountCents) {
        this.currency = currency;
        this.amountCents = amountCents;
    }
    public String getCurrency()  { return currency; }     // getters only — NO setters
    public long   getAmount()    { return amountCents; }

    public Money plus(Money other) {         // "modification" returns a NEW object
        return new Money(currency, amountCents + other.amountCents);
    }
}
Money a = new Money("USD", 500);
Money b = a.plus(new Money("USD", 300));     // a is unchanged; b is a new Money(800)

Part 2 — The Recipe for an Immutable Class

🎯 Records do most of this for you. A record Money(String currency, long amountCents) {} is implicitly final, has final fields, no setters, a canonical constructor, and generated equals/hashCode/toString — the immutability recipe in one line. But records do not automatically defensively copy mutable components (Part 4), so the deep understanding still matters. (Records get their own lesson in Modern Java.)

Part 3 — Why Immutability Is Worth It

  • Thread-safe for free. An object that never changes can be read by any number of threads with zero synchronization — there are no writes to race on. This alone makes immutability the most powerful concurrency tool (Concurrency lessons lean on it heavily).
  • Safe to share and cache. You can hand out the same instance everywhere without fear a caller mutates it under you — no defensive copies on every access, no "who owns this object?" confusion.
  • Reliable hash keys and set members. Since the fields behind equals/hashCode can't change, an immutable object's hash is stable — it can't "vanish" from a HashMap the way a mutated key does (equality lesson).
  • Easier to reason about. A value that can't change has no temporal state to track; you never ask "what is it now?" — it's the same as when it was created. This eliminates a whole category of bugs.
🔍 The JDK's most important value types are immutable for exactly these reasons: String, all the wrapper classes (Integer, Long…), BigDecimal, BigInteger, and the entire modern date/time API (LocalDate, Instant, Duration) — the old mutable java.util.Date is a cautionary tale the new API deliberately reversed.

Part 4 — Deep Dive: Defensive Copying (the leak a naive "immutable" class has)

Making fields final is not enough if a field references a mutable object. A final reference can't be reassigned, but the object it points to can still be mutated — and if you leak that reference, your "immutable" class isn't:

public final class Period {
    private final Date start, end;           // Date is MUTABLE (a JDK design mistake)
    public Period(Date start, Date end) {
        this.start = start;                  // ❌ stores the caller's Date reference directly
        this.end = end;
    }
    public Date getStart() { return start; } // ❌ hands the internal Date back to callers
}

// TWO leaks let outside code mutate a "final" field's target:
Date d = new Date();
Period p = new Period(d, new Date());
d.setYear(0);                    // LEAK 1: caller kept a reference and mutated it → p changed!
p.getStart().setYear(0);         // LEAK 2: getter returned the internal object → mutated it!
// THE FIX — defensively COPY on the way in AND on the way out:
public final class Period {
    private final Date start, end;
    public Period(Date start, Date end) {
        this.start = new Date(start.getTime());   // copy the INPUT — sever the caller's reference
        this.end   = new Date(end.getTime());
        if (this.start.after(this.end))           // validate the COPIES (not the originals — TOCTOU)
            throw new IllegalArgumentException("start after end");
    }
    public Date getStart() { return new Date(start.getTime()); }   // copy the OUTPUT — don't leak internals
    public Date getEnd()   { return new Date(end.getTime()); }
}
🔍 Senior nugget — the two rules and a subtlety: (1) copy mutable constructor arguments before storing them, and (2) copy mutable fields before returning them. There's a security-relevant detail: validate the copy, not the original argument — if you check the original then store it, an attacker could mutate it between the check and the store (a time-of-check/time-of-use race). The best solution, though, is to avoid mutable components entirely: use immutable types (LocalDate/Instant instead of Date) as fields, and defensive copying becomes unnecessary. This whole problem is why the modern date/time API is immutable.

Part 5 — final Is Not the Same as Immutable

final List<String> names = new ArrayList<>();   // 'names' the REFERENCE is final...
names.add("Ada");         // ...but the LIST is fully mutable! final ≠ immutable
// names = new ArrayList<>();  ❌ can't REASSIGN the reference — that's all final prevents

// For a truly unmodifiable collection:
List<String> fixed = List.of("a", "b", "c");           // immutable (throws on modification)
List<String> view  = Collections.unmodifiableList(src); // an unmodifiable VIEW (src can still change!)
⚠️ Two common confusions. (1) final on a reference field only stops reassignment of the variable — the object it points to can still be mutated. True immutability requires the referenced object to be immutable too. (2) Collections.unmodifiableList(src) returns a read-only view: callers of the view can't modify it, but if someone else still holds src, changes to src show through the view. List.of(...) / List.copyOf(...) create genuinely independent immutable collections. Know which guarantee you have.

Part 6 — Deep Dive: When Immutability Costs Too Much

Immutability is the default to reach for, but it isn't free, and a senior engineer knows the trade-offs:

  • Allocation churn. Every "modification" allocates a new object. For a hot loop that updates one field a million times, that's a million short-lived objects — the exact reason StringBuilder (mutable) exists alongside String (immutable). For bulk-building, use a mutable builder, then freeze into an immutable result.
  • Large objects. Copying a big object to change one field is wasteful. Persistent data structures (structural sharing) mitigate this, but plain immutable copies of large structures can hurt.
  • The pragmatic pattern: immutable value objects for your domain (money, dates, coordinates, DTOs), mutable objects with controlled, encapsulated state where performance or the problem demands it (builders, accumulators, entities with lifecycle). Prefer immutable; make mutable a deliberate, localized exception.
🔍 Senior framing: "immutable by default, mutable by necessity." The value of immutability is that it removes questions — thread-safety, aliasing, temporal state — so you spend it where those questions are expensive (shared, concurrent, keys) and relax it where the cost is real and the object is confined to one thread/scope. The builder pattern (Design Patterns lesson) is the standard bridge: mutate freely while building, produce an immutable result.

Interview Questions

Q1 (beginner): What is an immutable object and how do you make one?

An immutable object's observable state can't change after construction. The recipe: make the class final (no subclass can add mutability), make all fields private final, provide no setters or other mutators, initialize everything in the constructor, and — if any field references a mutable object — defensively copy it on the way in and out so no external reference to the internal state escapes. Any "modification" returns a new instance instead of changing the existing one, exactly like String. A record gives you most of this automatically.

Q2 (beginner): Name three benefits of immutability.

(1) Thread-safety for free — an object that never changes can be shared across threads with no synchronization, since there are no writes to race. (2) Safe sharing and caching — you can hand out the same instance everywhere without defensive copies or worrying a caller will mutate it. (3) Stable hash keys — the equality-relevant fields can't change, so an immutable object is a reliable HashMap key and won't "vanish" from hash collections. (Bonus: easier reasoning — no temporal state to track.)

Q3 (mid): Why is final not sufficient for immutability?

final on a field prevents reassigning the reference, but not mutating the object the reference points to. A final List<String> names can't be pointed at a different list, yet names.add(...) still works because the list itself is mutable. True immutability requires that every field is either a primitive/immutable type, or a mutable type that's defensively copied and never leaked. So a class with all-final fields can still be mutable if a field holds a mutable object that's exposed via a getter or captured from a constructor argument. Immutable = final fields + immutable/copied field targets + no leaks.

Q4 (mid): What is defensive copying and when is it required?

Defensive copying is duplicating a mutable object at the boundary of a class so external code can't reach and mutate the class's internal state through a shared reference. It's required whenever an (otherwise) immutable class has a field of a mutable type: copy the argument in the constructor before storing it (so the caller's continued reference to the original can't affect you), and return a copy from getters (so callers can't mutate the field you handed back). Without both, an "immutable" Period holding a mutable Date can be changed from outside via either the retained constructor argument or the getter's return value. The best avoidance is to use immutable field types (LocalDate instead of Date) so no copying is needed.

Q5 (senior): Explain the time-of-check/time-of-use subtlety in defensive copying.

If a constructor validates a mutable argument and then stores it directly, there's a window between the check and the use where the argument can be mutated by another thread (or by the caller via a retained reference), so the stored object may violate the invariant you just verified — a TOCTOU (time-of-check-to-time-of-use) race. Concretely: you check start.before(end) on the caller's Date, it passes, but before you store it the caller mutates start to be after end; your object now holds an invalid, unchecked state. The fix is to copy first and validate the copy: since the copy is private and unreachable by anyone else, no one can mutate it between the check and the store. This ordering — copy, then validate the copy, then store the copy — is the correct, attack-resistant pattern, and it's a favorite senior interview detail because it reveals whether you understand that a check on a shared mutable object is only as good as your exclusive access to it.

Q6 (senior): Argue for "immutable by default" and identify where you'd deliberately choose mutability.

Immutable by default because immutability eliminates entire bug classes: no data races (nothing to synchronize), no aliasing surprises (sharing is always safe), no stale/temporal reasoning (the value is what it was created as), and reliable equality/hashing. In modern concurrent, multi-threaded systems this is the cheapest correctness you can buy, which is why value objects, DTOs, config, and domain values should be immutable, and why the JDK made the new date/time API and records immutable. You deliberately choose mutability where immutability's per-change allocation cost is prohibitive or the object genuinely models evolving state confined to one owner: hot-loop accumulators and buffers (StringBuilder), builders assembling a complex object before freezing it, large data structures updated in place, and stateful entities with a lifecycle managed within a boundary. The discipline is to keep mutability local and encapsulated — mutate inside a confined scope, then expose an immutable result or protect it behind synchronization — rather than letting mutable objects flow freely through the system. "Immutable by default, mutable by necessity, mutation kept local."

🏋️ Exercises (attempt before opening solutions)

1 (easy): Build an immutable value. Write an immutable Point(int x, int y) the long way (final class, final fields, no setters, a withX(int) that returns a new Point). Then rewrite it as a one-line record.
Solution

Long way: final class Point { private final int x, y; ...; Point withX(int nx){ return new Point(nx, y); } }. Record: record Point(int x, int y) {} — implicitly final, final fields, accessors, equals/hashCode/toString generated. Both are immutable; the record is the modern default for pure value objects with no mutable components.

2 (easy): final ≠ immutable. Declare final List<String> list = new ArrayList<>();, add elements, then try to reassign list. Explain what final actually prevented.
Solution

list.add("x") works — the list is mutable. list = new ArrayList<>() is a compile error — final only forbade reassigning the reference. So final froze the variable, not the object. For an unmodifiable collection use List.of(...) or List.copyOf(list).

3 (medium): Break a naive immutable class. Write a DateRange that stores two java.util.Date fields directly (no copying), then mutate one from outside via both the retained constructor argument and the getter. Show both leaks.
Solution

Leak 1: keep the Date you passed in and call setTime on it — the range's internal field changes because it's the same object. Leak 2: call range.getStart().setTime(0) — the getter returned the internal Date, now mutated. Both prove that final fields holding mutable objects aren't enough.

4 (medium): Fix it with defensive copies. Repair DateRange: copy in the constructor and copy in the getters, validating the copies. Confirm both previous leaks are now closed.
Solution

Constructor: this.start = new Date(start.getTime()); (and same for end), then validate this.start/this.end. Getters: return new Date(field.getTime()). Now the retained argument and the getter return value are independent copies, so external mutation can't touch the internals. Even better: switch the fields to Instant/LocalDate and drop the copying entirely.

5 (hard): View vs copy. Create a source list, wrap it with Collections.unmodifiableList, then modify the source and observe the view. Then use List.copyOf and repeat. Explain the difference.
Solution

With unmodifiableList(src): you can't modify the view directly, but modifying src is reflected through the view — it's a live read-only view, not a snapshot. With List.copyOf(src): the copy is fully independent; later changes to src don't appear. Lesson: "unmodifiable" (can't change through this handle) is not the same as "immutable/independent" (a genuine snapshot). Choose based on whether the backing data can change behind you.

6 (hard): Explain-it test. Out loud: (a) the immutable-class recipe and three benefits; (b) why final isn't immutability; (c) defensive copying, both directions, and the TOCTOU reason to validate the copy.
Self-check rubric

Pass (a) if you listed final class/final fields/no setters/copy-mutable-fields and named thread-safety, safe sharing, stable keys. Pass (b) if you said final freezes the reference, not the object. Pass (c) if you copied on input and output and explained validating the copy avoids a check-then-mutate race. These carry directly into Records, Concurrency, and the Spring Boot DTO practices.

🎯 Key Takeaways

  • Immutable = state can't change after construction: final class, private final fields, no setters, set in constructor.
  • Immutability gives thread-safety, safe sharing, and stable hash keys for free — the default to reach for, especially in concurrent code.
  • final freezes a reference, not the object it points to — true immutability needs immutable (or copied) field targets.
  • Defensively copy mutable fields on input and output; validate the copy (not the original) to avoid a TOCTOU race — or use immutable field types and skip it.
  • An unmodifiable view isn't a copy — List.copyOf for an independent snapshot.
  • Immutability costs allocation — use a mutable builder, then freeze, for hot paths and large objects ("immutable by default, mutable by necessity").
