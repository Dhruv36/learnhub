Memory: Stack, Heap & References JVM & Performance

What you'll master in this lesson: the JVM's memory areas — the per-thread stack vs the shared heap, and what lives where; the young/old generation structure of the heap (which the GC lesson builds on); the reference types (strong, soft, weak, phantom) and what they mean for garbage collection; and the two memory failures you'll actually hit — StackOverflowError and the OutOfMemoryError family, including memory leaks in a garbage-collected language. This grounds the primitives-vs-references distinction from lesson 2 in the actual runtime layout and sets up garbage collection.

Part 1 — Stack vs Heap

Stack (one per thread)Heap (shared by all threads)
HoldsMethod frames: local variables, parameters, referencesAll objects and arrays
LifetimeA frame lives while its method runs (LIFO)Until no longer reachable (GC decides)
ManagementAutomatic (push on call, pop on return)Garbage-collected
SpeedVery fast (just move a pointer)Slower (allocation + GC)
Thread-shared?No — each thread has its ownYes — the source of concurrency hazards
void method() {
    int x = 42;                  // x (a primitive) lives ON THE STACK, in this method's frame
    Person p = new Person();     // the Person OBJECT is on the HEAP; 'p' (the reference) is on the stack
}   // when method() returns, its frame pops: x and the reference p vanish.
    // The Person object stays on the heap until it's unreachable, then the GC reclaims it.
💡 This is the runtime picture behind the primitives-vs-references distinction (Types lesson). A primitive local lives directly in the stack frame. An object always lives on the heap; the variable holds a reference (a pointer) to it, and that reference lives on the stack (for a local) or inside another heap object (for a field). When a method returns, its stack frame — with its locals and references — is discarded instantly, but the heap objects those references pointed to survive until nothing reachable references them.

Part 2 — The Heap's Generational Structure

┌───────────────────────── HEAP ─────────────────────────┐
│  YOUNG GENERATION                    OLD GENERATION     │
│  ┌────────┬────────┬────────┐        ┌────────────────┐ │
│  │  Eden  │  S0    │  S1    │  ───►  │   Tenured      │ │
│  └────────┴────────┴────────┘        └────────────────┘ │
│  new objects here    survivor spaces   long-lived objects│
└─────────────────────────────────────────────────────────┘
// Most objects die YOUNG (the "weak generational hypothesis"). New objects go in Eden;
// survivors of a minor GC move to a survivor space, then to Old if they keep surviving.
// Metaspace (class metadata) is SEPARATE, in native memory (not the heap).
🔍 The heap is split by object age because most objects die young. The vast majority of allocations (loop temporaries, method-local objects) become garbage almost immediately, so the GC collects the small, busy young generation frequently and cheaply (a "minor GC"), and the old generation of long-lived objects rarely (a "major/full GC"). This generational design is the single biggest reason modern GC is fast — it's the foundation the GC lesson builds on. Note metaspace (class metadata, from the JVM lesson) is separate native memory, not part of the object heap.

Part 3 — Reference Types & Reachability

An object is eligible for garbage collection when it's no longer reachable from a "GC root" (a live thread's stack, a static field, etc.). Java has four reference strengths that change this:

ReferenceGC behaviorUse for
Strong (normal)Never collected while strongly reachableOrdinary references (the default)
SoftCollected only when memory is lowMemory-sensitive caches
WeakCollected at the next GC if only weakly reachableCanonicalizing maps (WeakHashMap), metadata
PhantomEnqueued after finalization; can't retrieve the objectAdvanced cleanup (replaces finalize)
Map<Key, Data> cache = new WeakHashMap<>();   // entries vanish when the KEY is no longer used elsewhere
SoftReference<byte[]> softCache = new SoftReference<>(bigArray);   // dropped under memory pressure
🔍 Reachability, not reference-counting, decides collection. Java uses reachability analysis (trace from GC roots), which correctly handles cycles — two objects referencing each other but nothing else are both collectible (a reference-counting scheme would leak them). The weaker reference types let you tell the GC "keep this only if convenient": a SoftReference for a cache you'll rebuild if reclaimed, a WeakReference/WeakHashMap for associations that shouldn't keep their keys alive. Misusing strong references where weak ones belong is a classic leak source (Part 5).

Part 4 — StackOverflowError

// The stack is bounded (~512KB-1MB). Exceed it and you get StackOverflowError:
int recurse(int n) {
    return recurse(n + 1);        // no base case → frames pile up until the stack overflows
}
// Causes: infinite/too-deep recursion (Methods lesson — no tail-call optimization),
// or accidental mutual recursion. FIX: add/reach the base case, or convert to iteration.
⚠️ Each method call pushes a frame; deep or unbounded recursion exhausts the stack → StackOverflowError. Because Java doesn't optimize tail recursion (Methods lesson), even a correct recursive algorithm overflows on large inputs — convert to a loop or an explicit stack for deep structures. This is a separate failure from heap exhaustion (next), and the two get confused: a stack overflow is too many nested calls, not too many objects.

Part 5 — OutOfMemoryError & Leaks in a GC'd Language

// "Java has garbage collection, so no memory leaks" — FALSE. You leak by keeping references alive.
// A classic leak: a static collection that grows and is never cleared:
class Cache {
    private static final Map<Key, Value> MAP = new HashMap<>();   // static → lives forever
    static void add(Key k, Value v) { MAP.put(k, v); }             // never removed → grows unbounded
}
// The objects ARE reachable (via the static map), so the GC can't collect them → OutOfMemoryError: Java heap space.
OOM variantMeaningCommon cause
Java heap spaceHeap full of reachable objectsLeak (unbounded cache/collection) or genuinely too little heap
GC overhead limit exceededGC running constantly, reclaiming littleNear-leak: heap almost full
MetaspaceClass metadata area fullClass-loader leak (JVM lesson)
unable to create new native threadOS thread limit hitThread leak (not pooling)
🚨 Garbage collection reclaims unreachable objects — a "leak" in Java is keeping objects reachable that you no longer need. The usual culprits: static/long-lived collections that grow without bound; listeners/callbacks registered and never removed; ThreadLocals on pooled threads (the value outlives the request); caches without eviction; and inner classes/lambdas holding their enclosing instance (Enums & Nested lesson). You diagnose leaks with a heap dump (e.g. via jmap or an OOM auto-dump) analyzed in a tool (Eclipse MAT, VisualVM), which shows what's retaining the memory and the reference chain to a GC root.

Part 6 — Deep Dive: Allocation, Escape Analysis & Memory-Aware Design

  • Allocation is cheap; GC pressure is the cost. Allocating an object in Eden is nearly free (bump a pointer), but every allocation is future garbage the GC must trace and reclaim. High allocation rates → frequent GC → latency. So "avoid allocation in hot loops" is really "avoid creating GC pressure."
  • Escape analysis can eliminate allocations. If the JIT proves an object never "escapes" a method (isn't stored anywhere outside), it can allocate it on the stack or scalar-replace it (use its fields directly) — so some object creation costs literally nothing. This is why you shouldn't prematurely "optimize away" objects the JIT already handles.
  • Design implications. Reuse buffers in tight loops, prefer primitives/primitive arrays over boxed collections in hot numeric code (no per-element object header), size collections to avoid resize churn, and bound caches with eviction. But measure first — most code isn't allocation-bound.
  • Off-heap and value types. For extreme cases, direct ByteBuffers and (future) Project Valhalla value types reduce object overhead and pointer chasing. Rarely needed, but part of the memory-tuning toolkit.
🔍 Senior framing: Java's automatic memory management removes manual free() bugs but replaces them with a different discipline — managing reachability. The senior mental model is: the stack is fast and automatic; the heap is where objects live and the GC works; you don't leak by forgetting to free, you leak by forgetting to release references (clear collections, remove listeners, unset ThreadLocals, bound caches). Performance-wise, the lever is usually allocation rate and object lifetime (feeding the generational GC well — most objects should die young), not micro-optimizing individual allocations that escape analysis already handles. Understanding stack vs heap, reachability, the generational layout, and the reference types is what lets you read a heap dump, size a heap, choose a cache reference strength, and reason about GC behavior — the concrete skills the GC and Performance lessons build directly on.

Interview Questions

Q1 (beginner): What's the difference between the stack and the heap?

The stack is per-thread memory holding method call frames — each frame contains that method's local variables, parameters, and object references — and it's managed automatically in LIFO order: a frame is pushed when a method is called and popped when it returns, which is very fast. The heap is a single region shared by all threads where all objects and arrays live; heap memory is reclaimed by the garbage collector when objects become unreachable, not on method return. So a primitive local and an object reference sit on the stack, but the object they may refer to sits on the heap. Being thread-private makes the stack safe; the shared heap is where concurrency hazards arise.

Q2 (beginner): When is an object eligible for garbage collection?

When it becomes unreachable — that is, when there is no chain of references from any "GC root" (a live thread's stack variables, static fields, JNI references, etc.) to the object. At that point nothing in the running program can use it, so the GC may reclaim it. Java determines this by reachability analysis rather than reference counting, which means cycles are handled correctly: two objects that reference each other but are referenced by nothing else are both unreachable and collectible. Setting a reference to null or letting it go out of scope makes an object eligible only if no other reachable reference remains.

Q3 (mid): Explain the generational structure of the heap and why it exists.

The heap is divided into a young generation (Eden plus two survivor spaces) and an old/tenured generation. New objects are allocated in Eden; when a minor GC runs, surviving objects are moved to a survivor space, and objects that survive enough collections are promoted to the old generation. This design exploits the weak generational hypothesis: the empirical observation that most objects die young (short-lived temporaries vastly outnumber long-lived ones). By collecting the small, high-garbage young generation frequently and cheaply (minor GCs) and the old generation rarely (expensive major/full GCs), the collector does far less work than scanning the whole heap every time, which is the primary reason modern GC achieves good throughput and low pauses. (Metaspace, holding class metadata, is separate native memory.)

Q4 (mid): Can you have a memory leak in Java? Give an example.

Yes. Garbage collection reclaims unreachable objects, so a leak in Java means unintentionally keeping objects reachable that you no longer need, preventing their collection and growing the heap until OutOfMemoryError. Classic examples: a static or otherwise long-lived collection (cache, list, map) that you keep adding to but never remove from or bound, so it grows without limit; registering a listener/observer/callback and never deregistering it, so the registrant keeps the listener (and its enclosing objects) alive; a ThreadLocal whose value is never removed on a pooled thread, so it outlives the logical task; a non-static inner class or lambda capturing a large enclosing instance and being stored somewhere long-lived; and unclosed resources holding native memory. You diagnose these with a heap dump analyzed in a tool like Eclipse MAT, which shows the objects dominating the heap and the reference chain from a GC root that's keeping them alive. The fix is to release the reference: remove from the collection, deregister the listener, clear the ThreadLocal, bound the cache with eviction.

Q5 (senior): What are soft, weak, and phantom references, and when would you use each?

They express degrees of "keep this object only if it's worth it," influencing when the GC may reclaim a reachable-only-through-them object. A strong reference (the normal kind) prevents collection entirely. A soft reference lets the GC keep the object as long as memory is plentiful but reclaim it under memory pressure before throwing OOM — ideal for a memory-sensitive cache you can afford to lose and rebuild, since it grows when memory allows and shrinks when it's tight. A weak reference is reclaimed at the next GC once the object is only weakly reachable — used for associations that must not keep their target alive, the canonical example being WeakHashMap, whose entries disappear when the key is no longer strongly referenced elsewhere (so it won't leak by pinning keys), and for canonicalizing/interning maps and metadata keyed by objects. A phantom reference is even weaker: you can never retrieve the referent through it, and it's enqueued after the object is finalized, so it's used to schedule post-mortem cleanup deterministically — the modern, safer replacement for the deprecated finalize() (often via Cleaner). The practical guidance: default to strong references; reach for soft for reclaimable caches, weak for "don't keep it alive on my account" associations, and phantom (or Cleaner) for precise native-resource cleanup — and remember that overusing strong references where weaker ones belong is a common leak cause, while overusing soft/ weak can hurt performance or cause surprising nulls.

Q6 (senior): An application's heap usage grows steadily under steady load until it OOMs. Walk through your diagnosis.

A monotonic climb in heap under constant load is the signature of a memory leak — objects being retained that should be released. First, confirm it's a leak versus undersizing: watch the heap after full GCs (via GC logs or a monitor); if the post-GC live set keeps rising over time rather than stabilizing, memory is being retained, not just churned. Then capture a heap dump — configure -XX:+HeapDumpOnOutOfMemoryError to get one at the failure, or take one with jmap/JFR while usage is high — and open it in an analyzer like Eclipse MAT. Use the dominator tree / "leak suspects" to find which objects retain the most heap, then follow the path to GC roots for the dominant objects to see what is keeping them alive: the reference chain names the culprit. Typical findings map to known patterns: an ever-growing static or singleton collection/cache with no eviction; listeners/callbacks accumulating because they're never deregistered; ThreadLocals on container-pooled threads holding per-request objects across requests; keys pinned by strong references where a WeakHashMap was intended; or a class-loader leak (metaspace variant) from redeploys. Cross-check by comparing two heap dumps taken some time apart to see which object populations are growing. Once the retaining structure is identified, the fix is to release references — bound and evict caches, deregister listeners, clear ThreadLocals in a finally, use appropriate reference strengths — and add a regression guard (a test or a monitored metric on the collection size / post-GC heap). Only after ruling out a leak would you consider simply raising -Xmx, since more heap merely delays a true leak's OOM. The throughline is that in a GC'd language you debug leaks by reachability: find what's reachable that shouldn't be, and cut the reference.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Stack vs heap placement. For void f() { int n = 5; int[] a = new int[3]; }, state what lives on the stack and what on the heap, and what happens when f returns.
Solution

On the stack (in f's frame): the primitive n and the reference a. On the heap: the int[3] array object. When f returns, the frame pops, so n and the reference a disappear immediately; the array object becomes unreachable and is eligible for GC (reclaimed at some later collection).

2 (easy): Trigger a StackOverflowError. Write a method that recurses with no base case and observe the error. Explain why it's not an OutOfMemoryError.
Solution

Unbounded recursion pushes frames until the fixed-size stack is exhausted → StackOverflowError. It's not OOM because you didn't run out of heap (few or no objects created) — you ran out of stack from too many nested call frames. The two failures have different causes: too-deep calls vs too-many-reachable-objects.

3 (medium): Build a leak. Create a static collection that keeps accumulating objects and never releases them; explain why the GC can't reclaim them and how to fix it.
Solution

A static final List<byte[]> you keep adding large arrays to grows the heap until OOM. The GC can't collect the arrays because they remain reachable through the static list (a GC root) — that's the definition of a Java leak. Fixes: remove entries when done, bound the collection with eviction (an LRU cache), or use weak/soft references if the entries are a reclaimable cache.

4 (medium): WeakHashMap behavior. Put an entry in a WeakHashMap keyed by an object, drop your strong reference to the key, force GC, and check the map. Explain.
Solution

After dropping the only strong reference to the key and a GC, the entry disappears from the WeakHashMap — the map holds the key only weakly, so once nothing else references the key it's collectible and the entry is removed. This is why WeakHashMap is used for associations that shouldn't keep their keys alive (canonicalizing maps, metadata), avoiding the leak a strong-keyed HashMap would cause.

5 (hard): Reason about GC pressure. Explain why creating millions of short-lived objects in a loop can hurt latency even though each allocation is cheap, and how the generational GC and escape analysis relate.
Solution

Each allocation is a cheap pointer bump in Eden, but millions of them fill Eden quickly, triggering frequent minor GCs — and while the generational GC handles short-lived garbage efficiently (most die young, so minor GCs are cheap), the rate of collection still consumes CPU and can add pauses/latency. Escape analysis can eliminate some of these allocations entirely if the JIT proves the objects don't escape the method (stack allocation / scalar replacement), so not every "new" in a loop actually allocates. The practical takeaway: high allocation rate = GC pressure; reduce it (reuse buffers, primitives over boxed) for hot paths, but measure, since the JIT and generational GC already handle much of it.

6 (hard): Explain-it test. Out loud: (a) stack vs heap and what lives where; (b) reachability and how leaks happen in a GC'd language; (c) the reference types and a use for each of soft and weak.
Self-check rubric

Pass (a) if you placed primitives/references on the stack, objects on the heap, and noted frame pop vs GC. Pass (b) if you said GC collects unreachable, so a leak is retaining reachable objects (static collections, listeners, ThreadLocals). Pass (c) if you gave soft→reclaimable cache and weak→WeakHashMap/don't-pin-keys. These are the exact concepts the GC lesson uses.

🎯 Key Takeaways

  • The stack (per-thread) holds method frames — locals and references; the heap (shared) holds all objects; frames pop on return, heap objects live until unreachable.
  • The heap is generational (young: Eden+survivors, old: tenured) because most objects die young — the basis of fast GC; metaspace is separate.
  • Collection is by reachability from GC roots (handles cycles); reference strengths — strong/soft/weak/phantom — tune what survives.
  • StackOverflowError = too many nested calls (deep recursion); the OutOfMemoryError family = too many reachable objects (or metaspace/threads).
  • Leaks exist in Java — you keep references alive (static collections, listeners, ThreadLocals, unbounded caches); diagnose with heap dumps.
  • Performance lever is allocation rate and object lifetime (feed the generational GC); escape analysis makes some allocations free — measure before optimizing.
