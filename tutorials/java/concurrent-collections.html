Concurrent Collections & Atomics Concurrency

What you'll master in this lesson: the thread-safe collections (ConcurrentHashMap, CopyOnWriteArrayList, BlockingQueue) that let multiple threads share data without hand-written locking; the atomic classes (AtomicInteger, AtomicReference) that update single variables lock-free via compare-and-swap; why ConcurrentHashMap scales where synchronized maps don't; and how these are the practical tools you reach for instead of raw locks. This lesson turns the theory of the last three into the collections and primitives you actually use.

Part 1 — Why Not Just synchronize a HashMap?

// The naive "thread-safe map" — a synchronized wrapper (or the legacy Hashtable):
Map<String,Integer> m = Collections.synchronizedMap(new HashMap<>());
// Every operation locks the WHOLE map — only one thread at a time, even for unrelated keys.
// Under contention this serializes everything → terrible scalability.

// Worse, compound operations still aren't atomic even with a synchronized map:
if (!m.containsKey(k)) m.put(k, v);   // ❌ RACE — another thread can put between the check and the put
// You'd have to lock externally around the whole compound op, re-serializing everything.

Wrapping a HashMap with a single lock makes it correct but not scalable: every access — even to different keys — contends on one lock, so threads serialize. And check-then-act sequences still race unless you lock externally. The java.util.concurrent collections solve both: fine-grained internal locking for scalability, and atomic compound operations built in.

Part 2 — ConcurrentHashMap: The Workhorse

ConcurrentHashMap<String,Integer> map = new ConcurrentHashMap<>();

// Thread-safe AND scalable — different keys can be updated concurrently (fine-grained locking):
map.put("a", 1);
map.get("a");

// ATOMIC compound operations — no external locking needed, no check-then-act race:
map.putIfAbsent("a", 0);                          // atomic "put only if absent"
map.compute("a", (k, v) -> v == null ? 1 : v + 1); // atomic read-modify-write
map.merge("a", 1, Integer::sum);                  // atomic counter update (the word-count idiom)
map.computeIfAbsent("k", key -> new ArrayList<>()); // atomic lazy init
🎯 ConcurrentHashMap is the default shared map for concurrent code. Internally it locks at the granularity of individual buckets/bins (not the whole map), so operations on different keys proceed in parallel — this is lock striping, and it's why it scales where a synchronized map doesn't. Its atomic methods (compute, merge, putIfAbsent, computeIfAbsent) let you do read-modify-write safely without any external lock. It does not allow null keys or values (unlike HashMap) — deliberately, to avoid ambiguity in concurrent gets.

Part 3 — Atomic Classes: Lock-Free Single Variables

import java.util.concurrent.atomic.*;

// The race-free counter — no synchronized, no lock:
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();          // atomic ++; returns the new value
count.addAndGet(5);
count.compareAndSet(10, 20);      // set to 20 ONLY if currently 10 (compare-and-swap)

AtomicReference<Config> config = new AtomicReference<>(initial);
config.updateAndGet(c -> c.withTimeout(30));   // atomic reference update
AtomicLong total = new AtomicLong();
LongAdder busyCounter = new LongAdder();        // faster than AtomicLong under HIGH contention
🔍 Atomics use CPU compare-and-swap (CAS), not locks. Instead of blocking, CAS does "read the value, compute the new one, and atomically write it back only if it hasn't changed since the read — otherwise retry." This is lock-free: no thread ever blocks, so there's no deadlock and less overhead for simple updates like counters. Under very high contention many retries can occur, which is why LongAdder/LongAccumulator exist — they spread the count across cells to reduce contention, at the cost of a slightly more expensive total read. For a single shared counter, an AtomicInteger beats a synchronized block.

🔍 Match the collection to the access pattern. CopyOnWriteArrayList makes reads lock-free and cheap but copies the whole backing array on every write — perfect for a rarely-changing list read often (event listeners, config) and terrible for write-heavy data. BlockingQueue is the backbone of producer/consumer: put blocks when full, take blocks when empty, so it handles all the coordination (Synchronization lesson) without any wait/notify. Choosing wrong (a copy-on-write list for frequent writes) is a real performance bug.

Part 5 — Deep Dive: Compare-and-Swap and the ABA Problem

// CAS in pseudocode — the heart of lock-free programming:
do {
    old = value.get();               // read the current value
    new = compute(old);              // compute the update
} while (!value.compareAndSet(old, new));   // write ONLY if unchanged; else retry with the new current

// The ABA problem: a value goes A → B → A between your read and your CAS. Your CAS SUCCEEDS
// (it's A again), but the intervening change (B) may matter — you missed that it was touched.
// AtomicStampedReference fixes it by pairing the value with a version "stamp" that always changes.
🔍 Senior nugget: CAS is how modern hardware supports lock-free algorithms — a single atomic instruction that conditionally updates. Atomics and concurrent collections are built on it. The ABA problem is the classic subtlety: CAS only checks that the value is what you last read, not that it was never changed — so if it was modified to B and back to A, your CAS can't tell, which breaks algorithms (like lock-free stacks) that care about intervening mutations. AtomicStampedReference (a value + a monotonically-changing stamp) or AtomicMarkableReference solve it. You rarely hit ABA in application code, but knowing it exists and why version stamps fix it is a strong senior signal.

Part 6 — Deep Dive: Choosing Your Concurrency Tool

Bringing the whole section together — given shared mutable state, here's the decision order, from cheapest/safest to most manual:

  1. Can it be immutable or thread-confined? Then you need no synchronization at all — always the first choice (Immutability, Threads lessons).
  2. Is it a single variable (counter, flag, reference)? Use an atomic — lock-free, no deadlock, minimal overhead.
  3. Is it a shared collection? Use a concurrent collection (ConcurrentHashMap, BlockingQueue) — they handle the locking efficiently and offer atomic compound operations.
  4. Is it a small critical section over multiple related fields? Then use synchronized or a Lock — but keep it narrow, and order locks consistently (Synchronization lesson).
  5. Only build custom lock-free structures (raw CAS loops, handling ABA) if profiling proves you need them — this is expert territory and easy to get wrong.
🔍 Senior framing: the entire concurrency section reduces to this ladder. The masters' instinct is to climb as little as possible: prefer no-shared-state (immutability, confinement), then lock-free primitives (atomics), then the concurrent collections that encapsulate correct, scalable locking, and only fall to explicit locks for genuine multi-field critical sections — never hand-rolling lock-free algorithms without a proven need. This ordering isn't just about performance; it's about minimizing the code you have to reason about with the memory model (Memory Model lesson), because that reasoning is where concurrency bugs hide. "Make it thread-safe" answered well is usually "restructure so almost nothing is shared and mutable, and use the right pre-built tool for the little that is."

Interview Questions

Q1 (beginner): Why use ConcurrentHashMap instead of a synchronized HashMap?

A synchronized HashMap (via Collections.synchronizedMap or the legacy Hashtable) guards every operation with a single lock over the whole map, so even threads accessing unrelated keys contend and serialize — it's correct but doesn't scale. ConcurrentHashMap uses fine-grained internal locking (lock striping at the bucket level) so operations on different keys proceed concurrently, giving much better throughput under contention. It also provides atomic compound operations (putIfAbsent, compute, merge, computeIfAbsent) so common read-modify-write patterns are safe without external locking, which a plain synchronized map can't offer (check-then-act still races there).

Q2 (beginner): What does AtomicInteger give you over a synchronized counter?

Lock-free atomic updates. incrementAndGet, addAndGet, and compareAndSet perform the read-modify-write as a single atomic hardware operation (compare-and-swap), so multiple threads can update the counter correctly without any thread ever blocking on a lock. Compared to a synchronized counter it avoids lock acquisition overhead and the possibility of blocking/contention on a monitor, and it can't deadlock. For a single shared numeric value it's the right, simplest tool; for very high contention, LongAdder scales even better by splitting the count across cells.

Q3 (mid): How does compare-and-swap (CAS) work, and how do atomics use it?

CAS is a single atomic CPU instruction that takes a memory location, an expected old value, and a new value, and writes the new value only if the location currently holds the expected old value, reporting whether it succeeded. Atomic classes use it in a retry loop: read the current value, compute the desired new value, and compareAndSet(old, new); if another thread changed the value in between, the CAS fails and you retry with the fresh current value. This achieves correct concurrent updates without locks — no blocking, no deadlock — which is why it's called lock-free. The trade-off is that under high contention threads may retry repeatedly (wasting CPU), which is what LongAdder mitigates by reducing contention on a single location.

Q4 (mid): When is CopyOnWriteArrayList appropriate, and when is it a mistake?

It's appropriate for collections that are read very frequently and written very rarely — the canonical case is a list of event listeners or observers, or rarely-changing configuration. Reads require no locking and never see a concurrent modification (iterators operate on an immutable snapshot, so no ConcurrentModificationException), which makes concurrent reads fast and safe. It's a mistake for write-heavy or large collections, because every mutating operation copies the entire backing array — O(n) per write and lots of garbage — so frequent writes make it extremely slow and memory-hungry. The rule: copy-on-write trades cheap reads for expensive writes, so use it only when reads vastly outnumber writes and the collection is small-to-moderate.

Q5 (senior): Explain the ABA problem and how to solve it.

The ABA problem is a subtle flaw in CAS-based algorithms: CAS only verifies that a location's value equals the value you last read, not that it was never changed in between. If another thread changes the value from A to B and then back to A between your read and your CAS, your compareAndSet(A, ...) succeeds as if nothing happened — but the intervening change to B may have mattered. This breaks lock-free structures like a Treiber stack, where a node could be popped (A→B), reused, and pushed back (B→A), so a CAS that "sees A" wrongly assumes the stack is unchanged and may corrupt links or resurrect a freed node. The solution is to make the value carry a version that always changes: AtomicStampedReference pairs the reference with an integer stamp you increment on every update, so CAS checks both value and stamp — an A→B→A round-trip changes the stamp, so the CAS correctly fails. AtomicMarkableReference is a lighter variant with a single boolean mark. In practice, application code rarely constructs lock-free algorithms directly (you use JDK atomics and concurrent collections that handle this internally), but recognizing ABA and knowing version-stamping fixes it demonstrates real understanding of lock-free concurrency.

Q6 (senior): Given some shared mutable state, walk through how you'd decide which concurrency mechanism to use.

I'd climb a ladder from safest/cheapest to most manual, trying to do as little synchronization as possible. First, can the state be eliminated as shared-mutable? Make it immutable (no synchronization needed at all — freely shareable) or confine it to a single thread (thread confinement — no sharing, no races); this is always the preferred answer because it removes the memory-model reasoning entirely. If it must be shared and mutable, next: is it a single variable (a counter, a flag, a swappable reference)? Use an atomic (AtomicInteger/AtomicReference, or LongAdder under heavy contention) — lock-free, no deadlock, minimal overhead. If it's a shared collection, use the appropriate concurrent collection (ConcurrentHashMap for maps with its atomic compute/merge, BlockingQueue for producer/consumer, CopyOnWriteArrayList for read-mostly lists) — they encapsulate correct, scalable locking and atomic compound operations. Only if the invariant spans multiple related fields that must change together do I use explicit synchronized/Lock, keeping the critical section narrow, locking on a private object, and ordering locks consistently to avoid deadlock. And I'd only hand-build a lock-free algorithm with raw CAS (handling ABA) if profiling proved a real bottleneck — it's expert-level and error-prone. The unifying principle is minimizing the amount of shared mutable state and therefore the amount of code that needs delicate memory-model reasoning; the right tool is usually the highest rung of that ladder that fits, not a hand-written lock.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Race-free counter, lock-free. Replace a synchronized counter with an AtomicInteger and run the two-thread increment test. Confirm it's correct without any lock.
Solution

AtomicInteger count = new AtomicInteger(); ... count.incrementAndGet();. The two-thread test reliably prints 200,000 with no synchronized — incrementAndGet is an atomic CAS operation. Simpler and non-blocking compared to the synchronized version.

2 (easy): Concurrent word count. Have multiple threads count words into a ConcurrentHashMap using merge. Explain why no external lock is needed.
Solution

map.merge(word, 1, Integer::sum) from multiple threads — merge on ConcurrentHashMap is atomic, so the read-modify-write of each count is safe without any external synchronization, and different keys update concurrently thanks to lock striping. Contrast a plain map where get+put would race.

3 (medium): synchronizedMap vs ConcurrentHashMap. Show the check-then-act race on a Collections.synchronizedMap, then fix it atomically with ConcurrentHashMap.putIfAbsent.
Solution

On a synchronized map, if (!m.containsKey(k)) m.put(k, v); races — two threads can both pass the check and both put (the second overwriting), because the map's per-call locking doesn't make the compound sequence atomic. Fix: concurrentMap.putIfAbsent(k, v), a single atomic operation that inserts only if absent, with no external lock. This is why atomic compound methods matter.

4 (medium): Producer/consumer with BlockingQueue. Wire a producer and consumer through a bounded ArrayBlockingQueue. Explain what happens when the queue is full or empty.
Solution

Producer queue.put(item) blocks when the queue is full (backpressure — it can't outrun the consumer); consumer queue.take() blocks when empty (waits for work). The bounded queue thus coordinates the two and limits memory, with no manual locking or signaling. This is the idiomatic producer/consumer, replacing wait/notify entirely.

5 (hard): CAS retry loop. Implement an atomic "update a reference by applying a function" using a manual compareAndSet loop on an AtomicReference, and explain the retry.
Solution

Config old, next; do { old = ref.get(); next = fn.apply(old); } while (!ref.compareAndSet(old, next));. If another thread updated ref between the get and the compareAndSet, the CAS fails and the loop retries with the new current value, so the update is always applied to the latest state without locking. (This is exactly what updateAndGet does for you — you'd normally use that.) Note the ABA caveat if identity of the value matters.

6 (hard): Explain-it test. Out loud: (a) why ConcurrentHashMap scales over a synchronized map; (b) how CAS makes atomics lock-free and the ABA problem; (c) the tool-selection ladder for shared mutable state.
Self-check rubric

Pass (a) if you cited lock striping (per-bucket) + atomic compound methods vs whole-map lock. Pass (b) if you described CAS retry loops as non-blocking and ABA as "A→B→A fools CAS", fixed by stamps. Pass (c) if you gave immutable/confined → atomics → concurrent collections → locks → custom lock-free. This synthesizes the whole concurrency section.

🎯 Key Takeaways

  • A synchronizedMap locks the whole map (serializes); ConcurrentHashMap uses lock striping to scale and offers atomic compute/merge/putIfAbsent.
  • Atomics (AtomicInteger, AtomicReference) update single variables lock-free via CAS — no blocking, no deadlock; LongAdder for high contention.
  • Pick the collection for the access pattern: CopyOnWriteArrayList (read-heavy), BlockingQueue (producer/consumer), ConcurrentHashMap (default shared map — no nulls).
  • CAS is the basis of lock-free code; the ABA problem (A→B→A fools CAS) is solved with version-stamped references.
  • Tool ladder: immutable/confined → atomics → concurrent collections → locks → custom lock-free — climb as little as possible to minimize memory-model reasoning.
