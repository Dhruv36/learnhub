Collections: Internals & Choosing Core Libraries

What you'll master in this lesson: how ArrayList, LinkedList, HashMap, and TreeMap actually work under the hood; the Big-O of every common operation and — crucially — why the theoretical winner often loses in practice (cache locality); how a HashMap handles collisions and resizes, and why it treeifies buckets; and a decision table for choosing the right implementation. This is the collections material interviewers dig into hardest, because it separates people who memorized the API from people who understand the data structures.

Part 1 — ArrayList vs LinkedList Internals

// ArrayList: backed by a resizable ARRAY (contiguous memory)
[ Ada ][ Grace ][ Linus ][ ... ][ ... ]     indices 0,1,2 in one block
  → get(i) is O(1): direct address arithmetic
  → add at end is amortized O(1); when full, it grows (~1.5×) and copies
  → insert/remove in the MIDDLE is O(n): everything after shifts

// LinkedList: a doubly-linked chain of NODES scattered in memory
[Ada] ⇄ [Grace] ⇄ [Linus]                   each node holds value + prev/next pointers
  → get(i) is O(n): must walk the chain from an end
  → insert/remove GIVEN a node reference is O(1) (just relink pointers)
  → but FINDING the position is O(n), so most real operations are still O(n)
🎯 The counter-intuitive truth: use ArrayList almost always, even for lots of inserts. LinkedList's O(1) insert only helps if you already hold the node (via a ListIterator); accessing by index is O(n). And ArrayList's contiguous memory has vastly better cache locality — the CPU prefetches the next elements — while LinkedList chases pointers all over the heap, causing cache misses that dwarf the algorithmic difference. In real benchmarks ArrayList beats LinkedList even for many mid-list operations. LinkedList's genuine niche is a Deque/queue (add/remove at both ends), and even there ArrayDeque is usually faster.

Part 2 — How HashMap Works (the big one)

// A HashMap is an ARRAY of "buckets". Placing a key:
1. Compute key.hashCode(), then spread the bits (internal hash() mixes high bits down)
2. bucket index = hash & (capacity - 1)      // capacity is a power of 2 → fast bit-masking
3. Put the entry in that bucket.

// COLLISIONS (two keys → same bucket) are handled by chaining:
bucket[5] → [keyA,valA] → [keyB,valB]        // a linked list within the bucket
  → get(keyB): go to bucket 5, walk the chain, use equals() to find keyB

// TREEIFICATION (Java 8+): if one bucket's chain grows past 8 entries (and capacity ≥ 64),
// that bucket converts from a linked list to a RED-BLACK TREE → O(log n) worst case instead of O(n).
// This defends against hash-collision denial-of-service attacks.
HashMap operationAverageWorst case
get / put / containsKeyO(1)O(log n) — treeified (was O(n) pre-Java 8)
IterationO(capacity + size)—
🔍 Load factor & resizing: a HashMap resizes (doubles capacity and rehashes every entry) when size > capacity × loadFactor (default 0.75). Resizing is O(n) but amortized away, keeping average operations O(1). If you know you'll insert ~1000 entries, construct new HashMap<>(1400) to avoid repeated resizes — a real performance tuning knob. The power-of-two capacity is why the index uses a fast & mask instead of %.

Part 3 — TreeMap & Sorted Structures

// TreeMap: a self-balancing RED-BLACK TREE, keys kept in SORTED order.
TreeMap<String,Integer> sorted = new TreeMap<>();
  → get/put/containsKey: O(log n)   (slower than HashMap's O(1), but ordered)
  → gives you range queries HashMap can't:
sorted.firstKey(); sorted.lastKey();
sorted.floorKey("m");        // greatest key ≤ "m"
sorted.ceilingKey("m");      // least key ≥ "m"
sorted.headMap("m");         // all entries with key < "m" — a sub-view
sorted.subMap("a", "m");     // range view
🎯 Choose TreeMap/TreeSet only when you need sorted order or range queries — otherwise pay O(1) with a HashMap, not O(log n). Between them sits LinkedHashMap: O(1) like HashMap but preserving insertion order (or access order, which makes it a one-line LRU cache via removeEldestEntry). The three map choices map to three needs: no order (Hash), insertion order (Linked), sorted (Tree).

Part 5 — The Decision Framework

Need key→value lookup?  ──► MAP
  need sorted keys / ranges?  ──► TreeMap
  need insertion/access order (LRU)?  ──► LinkedHashMap
  otherwise  ──► HashMap                (the default)

Need uniqueness?  ──► SET   (same three ordering choices: Hash / Linked / Tree)

Need an ordered sequence?  ──► LIST
  ──► ArrayList  (almost always)

Need a queue/stack (add/remove at ends)?  ──► ArrayDeque
  (as a stack, prefer ArrayDeque over the legacy Stack class; as a queue, over LinkedList)
🔍 Defaults to internalize: ArrayList for sequences, HashMap/HashSet for lookups/uniqueness, ArrayDeque for stacks/queues. Deviate only for a specific need (sorted → Tree, insertion order → Linked). Avoid the legacy synchronized classes Vector, Hashtable, and Stack — they're relics kept for compatibility; use the modern unsynchronized collections and, for concurrency, the dedicated concurrent collections (a later lesson).

Part 6 — Deep Dive: Why Cache Locality Beats Big-O

The most important performance lesson in collections isn't in the Big-O table. Modern CPUs are enormously faster than main memory, so they rely on caches, and cache behavior often dominates:

  • Contiguous beats scattered. Iterating an ArrayList reads one cache-line-friendly block; the CPU prefetches ahead and rarely stalls. Iterating a LinkedList follows pointers to nodes scattered across the heap — each hop is a potential cache miss costing ~100× a cache hit. So an O(n) array scan can crush an O(n) linked-list scan by a large constant factor.
  • Big-O hides the constant. Complexity classes describe growth, not absolute speed. Two O(n) operations can differ 10–50× because of memory access patterns. LinkedList's "O(1) insert" advantage evaporates once you account for the O(n) traversal to reach the position and the cache misses along the way.
  • The practical rule: prefer arrays/ArrayList and contiguous structures; reach for node-based structures only when their specific operation (e.g. O(1) removal via a held iterator, or a genuine tree/graph shape) is essential — and measure, don't assume.
🔍 Senior nugget: this is why ArrayList is the near-universal default and why performance work in Java (and everywhere) increasingly focuses on data layout (arrays of primitives, struct-of-arrays, minimizing pointer chasing) rather than shaving algorithmic complexity. Knowing Big-O is table stakes; knowing when the constant factors from memory access dominate is what distinguishes a senior answer. It also explains value types / project Valhalla's motivation: flattening objects into contiguous memory.

Interview Questions

Q1 (beginner): What's the Big-O of get, add, and contains for ArrayList vs LinkedList?

ArrayList: get(i) O(1) (array indexing), add at the end O(1) amortized, insert/remove in the middle O(n) (shifting), contains O(n) (linear scan). LinkedList: get(i) O(n) (walk the chain), add/remove at the ends O(1), insert/remove in the middle O(1) if you already hold the node but O(n) to find it, contains O(n). The headline: ArrayList wins random access; LinkedList only wins end operations and node-referenced edits — and even then ArrayList/ArrayDeque usually beat it in practice due to cache locality.

Q2 (beginner): How does a HashMap achieve O(1) lookup?

It stores entries in an array of buckets. To find a key it computes the key's hash code, maps it to a bucket index (via bit-masking against the power-of-two capacity), and goes directly to that bucket — a constant-time jump rather than a search. Within the bucket it uses equals to find the exact key among any collisions. As long as the hash function spreads keys well and the load factor is kept low (resizing when ~75% full), buckets stay short, so lookups are O(1) on average. The array indexing plus good hashing is what makes it constant-time.

Q3 (mid): How does HashMap handle collisions, and what changed in Java 8?

Collisions (distinct keys mapping to the same bucket) are handled by chaining: each bucket holds a list of entries, and lookup walks the list using equals. Pre-Java 8 that chain was a plain linked list, so a bucket with many collisions degraded to O(n), and adversarial inputs with colliding hashes enabled a denial-of-service attack. Java 8 added treeification: once a single bucket exceeds 8 entries (and the table capacity is at least 64), that bucket converts from a linked list into a red-black tree, bounding worst-case lookup at O(log n) instead of O(n). It reverts to a list if the bucket shrinks below 6. This makes HashMap resilient to hash flooding and improves pathological-case performance.

Q4 (mid): When would you choose a TreeMap over a HashMap?

Choose TreeMap when you need the keys in sorted order or need range/nearest queries that a hash structure can't provide: iterating entries in key order, finding the first/ last key, or floorKey/ceilingKey/headMap/subMap for "the greatest key ≤ x" or "all keys in [a, b)" — invaluable for things like time-series lookups, leaderboards, or interval matching. The cost is O(log n) operations instead of HashMap's average O(1), and it requires keys to be Comparable or a Comparator. If you only need key→value lookup with no ordering, HashMap is strictly better on speed. If you need insertion order rather than sorted order, use LinkedHashMap.

Q5 (senior): Explain why LinkedList often loses to ArrayList despite better insertion complexity.

Two reasons, both about the gap between asymptotic complexity and real hardware. First, LinkedList's O(1) insert/remove only applies when you already have a reference to the node (e.g. via a ListIterator positioned there); reaching an arbitrary index is O(n), so a typical "insert at position i" is O(n) traversal plus O(1) relink — no better than ArrayList's O(n) shift, and ArrayList's shift is a tight System.arraycopy of contiguous memory. Second and more important is cache locality: ArrayList stores elements contiguously, so iteration and scanning stream through cache lines with hardware prefetching, while LinkedList nodes are separate heap objects connected by pointers, so traversal chases references to scattered addresses, incurring frequent cache misses (each ~100× costlier than a hit) and extra memory overhead (two pointers plus an object header per node). The constant factor from memory access dominates, so in real benchmarks ArrayList beats LinkedList even for many mid-list operations. LinkedList's honest niche is a deque with heavy end operations — and even there ArrayDeque (a contiguous ring buffer) usually wins.

Q6 (senior): A HashMap-heavy service degrades under load with a specific input set. Diagnose the likely causes.

Several HashMap-specific pathologies fit "degrades with a specific input set". (1) Poor or adversarial hashCode distribution: if many keys collide into few buckets, average O(1) degrades toward O(log n) (treeified) or, with keys lacking Comparable, still-costly chains — and maliciously crafted colliding keys are a classic hash-flooding DoS (mitigated but not eliminated by treeification). Fix: a well-spreading hashCode using all identifying fields, or a keyed hash for untrusted input. (2) Missing initial sizing: inserting a large known number of entries into a default-capacity map triggers repeated resizes (each a full O(n) rehash), causing latency spikes and allocation churn under load — presize with new HashMap<>(expectedSize / 0.75f). (3) Mutating keys after insertion so entries strand in wrong buckets, turning lookups into misses and unbounded reinsertion. (4) Concurrency misuse: sharing a plain HashMap across threads without synchronization can, on resize, spin into an infinite loop (pre-Java 8) or corrupt state and lose entries — use ConcurrentHashMap. Diagnosis path: profile to confirm time is in map operations, inspect key distribution (bucket sizes / collision rate), check for resize storms in allocation profiles, and audit whether the map is shared across threads. The senior instinct is that "works for most inputs, degrades for a specific set" strongly implicates hash distribution or sizing, not the algorithm itself.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Predict the Big-O. For an ArrayList, state the complexity of: get(500), add(x) at end, add(0, x) at front, contains(x). Then do the same for LinkedList.
Solution

ArrayList: get O(1), add-end O(1) amortized, add-front O(n) (shift all), contains O(n). LinkedList: get O(n), add-end O(1), add-front O(1), contains O(n). Note add-front is the one place LinkedList clearly wins — but if you're also indexing, ArrayList still dominates overall.

2 (easy): The accidental O(n²). Explain why for (int i=0;i<list.size();i++) list.get(i); is O(n²) for a LinkedList but O(n) for an ArrayList.
Solution

For a LinkedList, each get(i) walks from an end to index i — O(i) — so the loop sums to O(n²). For an ArrayList, each get(i) is O(1), so the loop is O(n). The fix for either: use a for-each loop or iterator, which advances one node at a time — O(n) regardless of implementation. This is a real, easy-to-miss performance bug.

3 (medium): Choose the structure. For each, pick the best and justify Big-O: (a) autocomplete needing all words with a given prefix in order; (b) a 100k-entry lookup table by id; (c) a fixed-size most-recently-used cache.
Solution

(a) TreeMap/TreeSet — sorted order + subMap/range gives prefix ranges in O(log n); HashMap can't do ranges. (b) HashMap — O(1) lookup, no ordering needed; presize it. (c) LinkedHashMap with access-order and removeEldestEntry — O(1) operations and automatic eviction of the least-recently-used entry. Matching the structure to the access pattern is the whole skill.

4 (medium): Presize a HashMap. You'll insert ~10,000 entries. Explain what happens with a default HashMap and how to avoid it.
Solution

A default HashMap starts at capacity 16 and resizes (doubling + rehashing all entries, O(n) each time) whenever it passes 75% full — for 10k entries that's roughly ten resizes, each rehashing more entries. Presize with new HashMap<>(13_400) (≈ 10000 / 0.75, rounded up to allow for the load factor) so it never resizes during the inserts. Under load this removes latency spikes and allocation churn.

5 (hard): Benchmark intuition. Without running it, argue why iterating and summing a 10-million-element ArrayList<Integer> can be several times faster than the same on a LinkedList<Integer>, even though both iterations are O(n).
Solution

ArrayList stores its element references contiguously, so iteration streams through memory with hardware prefetching and few cache misses. LinkedList stores each element in a separate node object scattered across the heap, connected by pointers, so each step is a pointer dereference to a likely uncached address — frequent cache misses, each ~100× a cache hit, plus more memory traffic from node overhead. Same O(n), very different constant factor. (Both also pay Integer unboxing; a raw int[] would be faster still — no boxing, tightest layout.) This is the "cache locality beats Big-O" lesson.

6 (hard): Explain-it test. Out loud: (a) how HashMap gets O(1) and what treeification defends against; (b) the ArrayList-vs-LinkedList reality and the role of cache locality; (c) the three map choices and the need each serves.
Self-check rubric

Pass (a) if you described bucket-array + hash indexing + equals within bucket, and named hash-collision DoS as the treeification motive. Pass (b) if you explained LinkedList's index access is O(n) and contiguous memory/prefetching makes ArrayList faster despite equal Big-O. Pass (c) if you mapped Hash/Linked/Tree to no-order/insertion-order/sorted. These are exactly the follow-ups interviewers chain from "which collection would you use?".

🎯 Key Takeaways

  • ArrayList = contiguous array (O(1) index, O(n) middle-insert); LinkedList = node chain (O(n) index) — default to ArrayList.
  • HashMap = bucket array + hashing → O(1) average; collisions chain, and long chains treeify to O(log n) (DoS defense).
  • Presize large maps and give good hashCodes to keep operations O(1) and avoid resize storms.
  • TreeMap/TreeSet = red-black tree, O(log n), for sorted order and range queries; LinkedHashMap preserves insertion/access order (LRU).
  • Cache locality often beats Big-O — contiguous structures win by large constant factors; measure, and favor arrays/ArrayList.
  • Avoid legacy Vector/Hashtable/Stack; use ArrayDeque for stacks and queues.
