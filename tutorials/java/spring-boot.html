Spring Boot Ultra-Advanced

Spring Boot is the dominant Java backend framework — the thing most "Java developer" jobs actually mean. Its foundation is dependency injection (the Inversion of Control container), with auto-configuration and starters removing boilerplate. This lesson ties the whole track together into a production REST service.

Dependency Injection — the Core

@Service                                      // a Spring-managed BEAN
public class OrderService {
    private final PaymentGateway gateway;     // dependencies via CONSTRUCTOR injection
    private final OrderRepository repo;
    public OrderService(PaymentGateway g, OrderRepository r) {   // Spring provides these
        this.gateway = g; this.repo = r;
    }
}
// Inversion of Control: you don't `new` your dependencies — the container creates and WIRES
// them (singletons by default). This is "program to interfaces + compose" (OOP lesson)
// operationalized: swap implementations via config, inject mocks in tests (testing lesson).
// Prefer CONSTRUCTOR injection (final fields, testable, fail-fast) over @Autowired fields.

A REST Controller

@RestController
@RequestMapping("/api/users")
public class UserController {
    private final UserService service;
    public UserController(UserService s) { this.service = s; }

    @GetMapping("/{id}")
    public UserDto get(@PathVariable long id) {
        return service.find(id).orElseThrow(() -> new NotFoundException(id));  // Optional (exceptions lesson)
    }
    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public UserDto create(@Valid @RequestBody CreateUserRequest req) {   // @Valid → 400 on bad input
        return service.create(req);
    }
}
// @Valid + Jakarta Validation annotations (@NotBlank, @Email, @Min) validate the body → 400.
// Use DTOs/records for request/response — never expose JPA entities (leaks, coupling).

Data Access with Spring Data JPA

public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByEmail(String email);       // method name → generated query!
    List<User> findByRoleAndActiveTrue(Role role);
}
// Spring Data generates the implementation. @Transactional for multi-step writes.
// Watch the N+1 problem (databases lesson): use JOIN FETCH / @EntityGraph for relations.

Cross-Cutting Concerns Done Right

// Global error handling → clean JSON, no leaked stacks (errors/exceptions lessons):
@RestControllerAdvice
class GlobalExceptionHandler {
    @ExceptionHandler(NotFoundException.class)
    ResponseEntity<ApiError> notFound(NotFoundException e) {
        return ResponseEntity.status(404).body(new ApiError("NOT_FOUND", e.getMessage()));
    }
    @ExceptionHandler(MethodArgumentNotValidException.class)  // validation failures → 400
    ResponseEntity<ApiError> invalid(...) { ... }
}
// Config from environment (never hardcode): application.yml + @ConfigurationProperties,
// profiles (dev/prod). Spring Security for auth (JWT/OAuth2). Actuator for /health,
// /metrics (observability — Node production lesson parallels).

Interview Questions

Q1: What is dependency injection / IoC, and why does Spring use it?

Inversion of Control means the framework, not your code, creates objects and wires their dependencies: instead of `new PaymentGateway()` inside OrderService, the container instantiates beans and injects collaborators (by default as singletons). Benefits: loose coupling (depend on interfaces, swap implementations via config), testability (inject mocks — testing lesson), lifecycle management, and centralized configuration. It's "favor composition + program to interfaces" (OOP lesson) enforced by a container. Constructor injection is preferred because it yields final fields, explicit required dependencies, and framework-free testability.

Q2: Why should you not expose JPA entities directly from controllers?

Several reasons: it couples your API contract to your database schema (a column rename becomes a breaking API change); it can leak sensitive/internal fields; and serializing lazy-loaded relationships outside a transaction throws LazyInitializationException or triggers unexpected N+1 queries. Map entities to purpose-built DTOs/records for requests and responses — controlling exactly what's exposed, decoupling API evolution from persistence, and preventing the classic serialization pitfalls. Same "response_model shapes output" idea as the FastAPI lesson.

Q3: How does Spring Boot reduce boilerplate, and what's the risk?

Starters bundle curated dependencies; auto-configuration inspects the classpath and configures sensible beans (a DataSource if a JDBC driver is present, an embedded Tomcat for web, etc.); convention-over-configuration and Spring Data's query derivation remove wiring code. The risk is "magic" — developers who don't understand what's auto-configured struggle to debug, override, or reason about startup, and can pull in heavyweight defaults. Mitigate by understanding the mechanisms (conditional beans, actuator's /conditions), keeping configuration explicit where it matters, and not treating the framework as a black box.

🏋️ Practice: scaffold a Spring Boot app (start.spring.io) with Web + JPA + Validation; build a @RestController + @Service + JpaRepository for users with DTOs; add @Valid validation returning 400; add a @RestControllerAdvice for 404/400; write a Spring Data derived query (findByEmail); test the controller with MockMvc and the repo with @DataJpaTest/Testcontainers; expose Actuator /health.

🎯 Key Takeaways

  • Spring Boot's core is DI/IoC: the container creates and wires beans — prefer constructor injection.
  • Layered stereotypes: @RestController (thin) → @Service (logic) → @Repository (data); DTOs at the boundary.
  • Spring Data JPA derives queries from method names; watch N+1 (JOIN FETCH/@EntityGraph); @Transactional for writes.
  • Cross-cutting: @Valid (400), @RestControllerAdvice (typed errors), Security (authz), Actuator (observability).
  • Never expose entities — map to DTOs; externalize config; the production checklist mirrors the Node/FastAPI lessons.
