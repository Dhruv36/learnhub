Threads & the Runnable Model Concurrency

What you'll master in this lesson: what a thread is and how the JVM maps it to the OS; the two ways to define work (Runnable vs subclassing Thread) and why the first is right; the thread lifecycle and states; why start() and run() are different (a classic interview trap); and — most importantly — the fundamental problem concurrency introduces: shared mutable state and race conditions. This lesson sets up the entire concurrency section; you almost never use raw threads directly in modern code, but you must understand them to use the higher-level tools correctly.

Part 1 — What Is a Thread?

A thread is an independent path of execution within a process. Your main method runs on one thread; you can start more, and they run concurrently — on a multi-core machine, genuinely in parallel. Each thread has its own call stack (local variables, method calls) but shares the heap (objects) with every other thread in the process. That sharing is the source of both concurrency's power and all its dangers.

// Every Java program starts with the "main" thread. You can see the current one:
System.out.println(Thread.currentThread().getName());   // "main"

// A platform thread maps to an OS thread — real, but relatively HEAVY (~1MB stack, kernel-scheduled).
// This is why you don't create thousands of them casually (Executors and Virtual Threads lessons).
💡 Why concurrency at all? Two reasons: (1) responsiveness — do slow work (I/O, a network call) on a background thread so the main flow isn't blocked; (2) throughput — use multiple CPU cores to do more work per second. The cost is complexity: the moment two threads touch the same mutable data, you enter a world of race conditions, visibility bugs, and deadlocks that this section exists to navigate.

Part 2 — Defining Work: Runnable vs Thread

// WAY 1 (preferred) — implement Runnable, the "unit of work", and hand it to a thread:
Runnable task = () -> System.out.println("working on " + Thread.currentThread().getName());
Thread t = new Thread(task);
t.start();                        // starts a NEW thread that runs the task

// WAY 2 (discouraged) — subclass Thread and override run():
class MyThread extends Thread {
    public void run() { System.out.println("working"); }
}
new MyThread().start();
🎯 Prefer Runnable (or a lambda) over subclassing Thread. A Runnable separates what to do (the task) from how to run it (a thread, a pool, a virtual thread), so the same task works with an ExecutorService or any execution strategy. Subclassing Thread couples the work to one execution mechanism, burns your single inheritance slot (Inheritance lesson: favor composition), and can't be reused. In practice you rarely create Threads directly at all — you submit Runnables/Callables to an executor (next lessons).

Part 3 — start() vs run() (the classic trap)

Thread t = new Thread(() -> System.out.println(Thread.currentThread().getName()));

t.start();    // ✅ starts a NEW thread; the task runs on it (prints "Thread-0")
t.run();      // ❌ just a normal METHOD CALL — runs on the CURRENT thread (prints "main")!
🎯 Calling run() directly does NOT start a new thread — it executes the task synchronously on the calling thread, like any method. Only start() asks the JVM to create a new thread and invoke run() on it. This is one of the most common concurrency interview questions precisely because the mistake compiles and "works" (the code runs) — it just silently runs sequentially with no concurrency. Also: you can only call start() once per Thread; a second call throws IllegalThreadStateException.

Part 5 — Deep Dive: The Race Condition (the core problem)

class Counter {
    private int count = 0;
    public void increment() { count++; }    // NOT atomic! it's read-modify-write (3 steps)
    public int get() { return count; }
}

// Two threads each increment 100,000 times. Expected: 200,000. Actual: LESS (nondeterministic).
Counter c = new Counter();
Runnable job = () -> { for (int i = 0; i < 100_000; i++) c.increment(); };
Thread t1 = new Thread(job), t2 = new Thread(job);
t1.start(); t2.start(); t1.join(); t2.join();
System.out.println(c.get());     // e.g. 137,428 — LOST UPDATES!

Why? count++ is three operations: read count, add 1, write back. Two threads can interleave: both read 5, both compute 6, both write 6 — two increments produce one update. This is a race condition: the result depends on the unpredictable timing of thread interleaving. The bug is nondeterministic (different result each run), often invisible in testing, and catastrophic in production.

🚨 This is the single most important concept in concurrency. Whenever two or more threads access shared mutable state and at least one writes, you have a potential race condition unless you coordinate access. The next lessons are all about that coordination: synchronized/locks (mutual exclusion), the memory model (visibility), atomics (lock-free counters), and concurrent collections. But the deepest fix ties back to the Immutability lesson: immutable objects have no races because there's no shared mutable state to corrupt — which is why "make it immutable" is the most powerful concurrency tool.

Part 6 — Deep Dive: Why Raw Threads Are the Wrong Level

  • Threads are expensive. A platform thread costs ~1MB of stack and a kernel scheduling slot. Creating one per task doesn't scale — a server spawning a thread per request collapses under load. You pool and reuse threads (Executors lesson) instead.
  • Manual lifecycle management is error-prone. Starting, joining, handling exceptions, and shutting down threads by hand is fiddly and easy to get wrong (leaked threads, unhandled exceptions vanishing silently).
  • The right level is a task submitted to an executor. You express work as a Runnable/Callable and hand it to an ExecutorService, which manages a pool of threads, queuing, and results (Future). This is the standard modern model — you almost never write new Thread() in application code.
  • Virtual threads (Java 21) change the economics for I/O-bound work — cheap enough to create millions — but even then you submit them to an executor, not manage them by hand (Virtual Threads lesson).
🔍 Senior framing: understanding raw threads is necessary to reason about concurrency correctly, but using them directly is almost always a code smell in modern Java. The progression of this section mirrors the abstractions built on top of threads: threads (the primitive) → synchronization and the memory model (correctness) → executors and futures (managing threads) → concurrent collections and atomics (safe shared data) → virtual threads (scaling I/O). Each layer exists because the one below it is too low-level or error-prone for everyday use. Know the foundation; work at the highest safe abstraction.

Interview Questions

Q1 (beginner): What is a thread, and what does it share with other threads?

A thread is an independent path of execution within a process. Each thread has its own call stack (holding its local variables and method-call frames) and program counter, but all threads in a process share the heap — the objects. This sharing is what lets threads cooperate (and run in parallel across cores for throughput or off-load slow work for responsiveness), and it's also the root of concurrency's hazards: when multiple threads read and write the same shared, mutable objects, you get race conditions and visibility problems unless access is coordinated.

Q2 (beginner): What's the difference between start() and run()?

start() asks the JVM to create a new thread and invoke the task's run() method on that new thread, so it executes concurrently. Calling run() directly is just a normal method call — it executes the task synchronously on the current (calling) thread, with no new thread created and no concurrency. The trap is that calling run() compiles and appears to work (the code runs), but it silently runs sequentially. Also, start() may be called only once per Thread object; a second call throws IllegalThreadStateException.

Q3 (mid): Why prefer implementing Runnable over extending Thread?

Because it separates the work from the execution mechanism. A Runnable (or lambda) is a task that can be run by a raw thread, an ExecutorService pool, a scheduled executor, or a virtual thread — you choose the execution strategy independently. Extending Thread hard-couples the task to being run by that specific thread subclass, wastes the single-inheritance slot (so the class can't extend anything else), and can't be resubmitted or reused. It's a direct application of "favor composition over inheritance" and "program to an interface". In modern code you don't even create threads directly — you submit Runnable/Callable tasks to an executor.

Q4 (mid): Explain a race condition using count++.

A race condition is a bug where the result depends on the nondeterministic timing of thread interleaving. count++ looks atomic but is actually three steps: read count, add one, write the result back. If two threads run it concurrently on a shared counter, they can interleave so both read the same value (say 5), both compute 6, and both write 6 — two increments produce only one update (a "lost update"). Run two threads incrementing 100,000 times each and the total is usually less than 200,000, and different every run. The fix is to make the compound operation atomic/mutually exclusive — via synchronized, a lock, or an AtomicInteger — or to avoid shared mutable state altogether (immutability, thread confinement).

Q5 (senior): Why is thread-per-task the wrong model at scale, and what replaces it?

Platform threads are OS threads, each carrying a large (often ~1MB) stack and requiring kernel scheduling, so they're a scarce, heavyweight resource. Creating one per task means unbounded thread creation under load: memory is exhausted by stacks, the scheduler thrashes with context switches, and throughput collapses — a server that spawns a thread per request falls over as traffic rises. Manually managing thread lifecycles (start/join/exception-handling/shutdown) is also error-prone and leaks threads. The replacement is a thread pool via ExecutorService: you submit tasks (Runnable/Callable) and a bounded set of reusable worker threads executes them from a queue, giving you backpressure, bounded resource use, result handling via Future, and clean shutdown. For I/O-bound workloads, virtual threads (Java 21) change the economics — they're cheap enough to have millions, so a thread-per-request style becomes viable again — but you still submit them through an executor rather than managing them by hand. The principle is to decouple tasks from a bounded, managed execution resource rather than mapping each task onto a costly OS thread.

Q6 (senior): How does immutability relate to thread safety, and why is it the most powerful concurrency tool?

Race conditions and visibility problems arise specifically from shared mutable state — two threads contending over data that changes. An immutable object can't change after construction, so there is nothing to contend over: any number of threads can read it simultaneously with no synchronization, no locks, no visibility barriers, and no possibility of seeing a half-updated or torn state (given safe publication). This eliminates the entire class of concurrency bugs by construction rather than by careful coordination — which is why it's the most powerful tool: correct synchronization is hard to get right (too little causes races, too much causes contention and deadlocks), whereas immutability sidesteps the problem entirely. The practical strategy is to make value objects immutable (final classes/records with final fields, defensive copying of any mutable components), confine mutable state to a single thread when possible, and use synchronization only for the genuinely-shared mutable state that remains. "Immutable by default, share freely; mutable only when necessary, and then synchronize" is the backbone of a sound concurrency design, and it's why the JDK made String, the wrappers, and the new date/time API immutable.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Two threads. Start two threads (via Runnable lambdas) that each print their thread name five times. Observe the interleaved output.
Solution

Runnable r = () -> { for (int i=0;i<5;i++) System.out.println(Thread.currentThread().getName()); }; new Thread(r).start(); new Thread(r).start();. The two threads' outputs interleave unpredictably — a visible demonstration that they run concurrently and the scheduler decides ordering. Run it a few times; the interleaving differs.

2 (easy): start vs run. Call run() directly on a Thread, then start(), printing the current thread name in the task. Explain the different output.
Solution

t.run() prints "main" — it's a plain method call on the current thread, no new thread. t.start() prints "Thread-0" — the task runs on a newly created thread. This proves only start() introduces concurrency; run() is synchronous.

3 (medium): Reproduce a race. Build the unsynchronized Counter, run two threads incrementing 100,000 times each, join them, and print the total. Run it several times.
Solution

The total is consistently less than 200,000 and varies between runs — lost updates from the non-atomic count++ interleaving. This nondeterminism is the signature of a race condition. (If it ever prints exactly 200,000, run it more times or increase the count — the race is timing-dependent, which is exactly why it's dangerous: it may pass in testing.)

4 (medium): join coordination. Start a thread that computes a value (sleeps then sets a field), and have main wait for it with join() before reading the result.
Solution

Start the worker, then worker.join(); before reading — join blocks main until the worker terminates, guaranteeing the result is ready (and, per the memory model, visible — next lesson). Without join, main might read the field before the worker wrote it. This is basic thread coordination.

5 (hard): Runnable vs Thread refactor. Given a class extending Thread to do a task, refactor it to a Runnable and run it both on a raw thread and (conceptually) on an executor. Explain the flexibility gained.
Solution

Extract the run() body into a Runnable (or lambda). Now the same task runs via new Thread(task).start() or executor.submit(task) or a virtual-thread executor — the work is decoupled from how it's executed. You also freed the inheritance slot and made the task reusable/testable. This is why every concurrency API accepts Runnable/Callable, not Thread subclasses.

6 (hard): Explain-it test. Out loud: (a) what threads share and why that's dangerous; (b) start vs run; (c) what a race condition is via count++ and how immutability sidesteps it.
Self-check rubric

Pass (a) if you said own stack, shared heap → shared mutable state hazards. Pass (b) if you said start creates a new thread, run is a synchronous method call. Pass (c) if you described read-modify-write interleaving causing lost updates and noted immutable state has nothing to race on. These set up synchronization, the memory model, and the rest of the section.

🎯 Key Takeaways

  • A thread has its own stack but shares the heap — shared mutable state is the root of all concurrency hazards.
  • Prefer Runnable/lambdas over subclassing Thread — decouple the task from how it's run.
  • start() creates a new thread; run() is just a synchronous method call (the classic trap); start() is once-only.
  • Thread states: NEW → RUNNABLE → (BLOCKED/WAITING) → TERMINATED; coordinate with join/sleep/interrupt.
  • A race condition (e.g. non-atomic count++) causes nondeterministic lost updates — the core problem the rest of this section solves.
  • Raw threads are the wrong everyday level — use executors; and immutability eliminates races by removing shared mutable state.
