Strings & Text Foundations

What you'll master in this lesson: why Strings are immutable and the consequences that flow from it (the string pool, thread-safety, the O(n²) concatenation trap); the difference between == and .equals() for Strings and why new String("a") behaves differently from a literal; when to use StringBuilder; the essential String methods; text blocks and formatting; and the security/encoding issues that bite in production. Strings are the most-used type in Java — mastering their quirks pays off daily.

Part 1 — Strings Are Immutable (the defining fact)

String s = "Hello";
s.concat(" World");         // returns a NEW string — s is UNCHANGED
System.out.println(s);      // "Hello" — the return value was discarded!
s = s.concat(" World");     // capture it: now s refers to a new "Hello World" object

s.toUpperCase();            // also returns a new string; the original is untouched
String upper = s.toUpperCase();   // must assign the result

A String's characters can never change after creation. Every method that "modifies" a String — concat, toUpperCase, replace, substring, trim — returns a brand-new String and leaves the original intact. This trips up every beginner: calling s.trim() and expecting s to change does nothing unless you assign the result.

💡 Why immutable? Immutability buys three big things: (1) safe sharing — the same String object can be referenced everywhere and interned in a pool without fear it'll change under you; (2) thread-safety — immutable objects need no synchronization; (3) safe as hash keys — a String's hashCode never changes, so it's a reliable HashMap key. These benefits are exactly why String was designed this way, and why "make your value objects immutable" (a later lesson) is standard advice.

Part 2 — The String Pool & == vs equals

String a = "hello";              // a LITERAL — goes in the string POOL (interned)
String b = "hello";              // SAME pooled object — the JVM reuses it
System.out.println(a == b);      // true — both point to the one pooled "hello"

String c = new String("hello");  // NEW forces a fresh object OUTSIDE the pool
System.out.println(a == c);      // false! — different objects (== compares identity)
System.out.println(a.equals(c)); // true — .equals compares CONTENT ✅

String d = c.intern();           // intern() returns the pooled version
System.out.println(a == d);      // true
🎯 Always compare String content with .equals(), never ==. Literals are pooled so == sometimes works — which makes it a dangerous habit, because the moment a String comes from user input, a file, a network call, or new String(), == silently returns false for equal content. Use .equals() (or "literal".equals(userInput) to also dodge NPE on a null input), or equalsIgnoreCase when case shouldn't matter.

Part 3 — The Concatenation Trap: StringBuilder

// BAD — building a string in a loop with += :
String result = "";
for (String word : words) {
    result += word;        // each += creates a NEW String and copies everything so far → O(n²)!
}

// GOOD — StringBuilder mutates one buffer in place → O(n):
StringBuilder sb = new StringBuilder();
for (String word : words) {
    sb.append(word);       // no new object per iteration
}
String result = sb.toString();

// A single concatenation or a couple is fine — the compiler optimizes `"a" + b + "c"`.
// It's concatenation IN A LOOP that's the performance killer.
⚠️ The O(n²) trap. Because Strings are immutable, result += word allocates a new String and copies all prior characters every iteration. For a loop of n words averaging m characters, that's quadratic work — fine for 10 items, catastrophic for 100,000. This is one of the most common real-world Java performance bugs. StringBuilder keeps a growable char buffer and appends in place, making it linear. (Use StringBuffer only if you need thread-safety, which is rare; StringBuilder is the default.)

Part 6 — Deep Dive: Encoding, Unicode & Security

A String is a sequence of UTF-16 code units, not "bytes" and not always "one char = one visible character". This matters more than beginners expect:

  • Bytes ⇄ Strings need an explicit charset. new String(bytes) and str.getBytes() without a charset use the platform default, which differs across machines and causes "works on my laptop, mojibake in production" bugs. Always specify: new String(bytes, StandardCharsets.UTF_8).
  • Emoji and some characters are two code units (surrogate pairs), so length() can exceed the number of visible characters and charAt can land mid-character. For code-point-correct work use codePointCount / codePoints().
  • Case conversion is locale-sensitive. The famous "Turkish-I" bug: "TITLE".toLowerCase() in a Turkish locale produces a dotless ı, breaking comparisons. For protocol/identifier work use toLowerCase(Locale.ROOT).
🔒 Security nugget: never build SQL or shell commands by concatenating user input into a String ("SELECT * WHERE name='" + input + "'") — that's SQL/command injection. Use parameterized queries. And sensitive data (passwords) is often held in a char[] rather than a String, precisely because Strings are immutable and pooled: you can't reliably wipe a String from memory, but you can zero a char[] after use. Immutability is usually a virtue; here it's a liability.

Interview Questions

Q1 (beginner): Are Strings mutable in Java? What follows from the answer?

No — Strings are immutable; their characters can't change after creation. Every "modifying" method returns a new String. Consequences: (1) you must capture the return value (s = s.trim()); (2) building strings in a loop with += is O(n²) because each step allocates and copies — use StringBuilder; (3) Strings are safe to share, cache/intern in the pool, use as HashMap keys (stable hashCode), and use across threads without synchronization. Immutability is the root fact from which most String behavior follows.

Q2 (beginner): Why is == unreliable for comparing Strings?

== compares object identity (same reference), not content. String literals are interned in the pool, so two equal literals share one object and == returns true — which lulls people into using it. But a String from user input, a file, a network call, or new String("x") is a distinct object, so == returns false even when the content is equal. Always use .equals() for content comparison; prefer "constant".equals(maybeNull) ordering to also avoid a NullPointerException.

Q3 (mid): Explain the string pool and what intern() does.

The string pool is a JVM-managed cache of unique String instances. String literals are automatically interned: the first occurrence creates the object, later identical literals reuse it, so they're reference-equal. This saves memory and makes literal comparison fast. Strings created at runtime (new String, concatenation of non-constants, parsing) are not pooled by default. str.intern() returns the canonical pooled instance for that content — adding it to the pool if absent — so a == b.intern() can be made true for equal content. Interning is occasionally used to save memory on many duplicate strings, but it's easy to overuse and can pressure the pool; usually you just rely on .equals().

Q4 (mid): When would you use StringBuilder vs StringBuffer vs plain concatenation?

Use plain + for a fixed, small number of concatenations — the compiler compiles "a" + b + "c" efficiently (into StringBuilder/StringConcatFactory under the hood). Use StringBuilder whenever you build a string incrementally, especially in a loop, to avoid the O(n²) allocation of repeated +=. Use StringBuffer only when the same builder is shared and mutated across multiple threads and you want its methods synchronized — a rare case; StringBuffer is the older, synchronized version and StringBuilder is its faster unsynchronized replacement for the common single-threaded case.

Q5 (senior): Walk through the memory/allocation behavior of String result = ""; for (...) result += item; and why it's quadratic.

Each result += item is result = new StringBuilder().append(result).append(item).toString() (or equivalent). Critically, it copies the entire current result into a fresh buffer every iteration. If the final string has length N built from roughly equal pieces, iteration k copies ~k characters, so total work is 1+2+…+n ≈ N²/2 — quadratic in the output size — plus n intermediate String allocations that become garbage, pressuring the GC. Hoisting a single StringBuilder outside the loop and appending gives amortized O(1) per append (the buffer doubles on growth), so total work is linear and only one String is materialized at the end. On large inputs the difference is seconds vs milliseconds and a lot of GC churn — which is why this is a canonical performance-review finding.

Q6 (senior): A service works in dev but returns garbled non-ASCII text in production. Give the likely cause and the fix.

The most likely cause is charset mismatch from relying on the platform default encoding. Code like new String(bytes), str.getBytes(), or a FileReader/InputStreamReader without an explicit charset uses Charset.defaultCharset(), which may be UTF-8 on the dev laptop but a different encoding (e.g. a legacy Windows codepage) on the server — so multi-byte UTF-8 sequences get decoded wrong, producing mojibake. The fix is to specify the charset everywhere bytes and text meet: new String(bytes, StandardCharsets.UTF_8), str.getBytes(UTF_8), Files.readString(path, UTF_8), and set file.encoding/-Dfile.encoding=UTF-8 or (JDK 18+) rely on the new UTF-8-by-default. Also verify the HTTP Content-Type charset and the database/column encoding, since the mismatch can be at any boundary. The rule: never let text encoding be implicit.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Immutability proof. Set String s = "hi";, call s.toUpperCase(); (without assigning), and print s. Then do s = s.toUpperCase(); and print again. Explain.
Solution

First print: "hi" — the uppercase result was created and discarded because you didn't capture it. Second: "HI" — now you assigned the new String back to s. Strings never mutate in place; you must assign the return value. This single experiment is the whole immutability lesson.

2 (easy): == vs equals. Compare a literal "java" with new String("java") using both == and .equals(). Predict, then verify.
Solution

== → false (literal is pooled, new String is a separate object). .equals() → true (same content). Now compare two literals with == → true (both pooled). Conclusion: == is identity and only "works" by accident for pooled literals — use .equals().

3 (medium): The concatenation benchmark. Build a 100,000-element string two ways — += in a loop and StringBuilder — timing each with System.nanoTime(). Report the ratio.
Solution

The += version takes dramatically longer (often 100×+) because it's O(n²): each append copies the whole accumulated string. StringBuilder is O(n). This is not a micro-difference — it's the gap between milliseconds and seconds, and it grows with input size. (Caveat: warm up the JIT and run multiple times for honest numbers — see the Performance lesson.)

4 (medium): The split gotcha. Split "1.2.3" on the dot with split(".") and print the array length. Then fix it. Explain.
Solution

split(".") returns an empty array — . is a regex metacharacter matching any character, so it splits between every char and trailing empties are trimmed, leaving nothing. Fix: split("\\.") (escaped dot) → ["1","2","3"], or split(Pattern.quote(".")). Remember split/replaceAll/matches take regexes.

5 (hard): Reverse words, keep order robust. Given " the quick brown ", produce "brown quick the" — words reversed, collapsing extra whitespace. Use strip, split, and a builder.
Solution

Approach: s.strip().split("\\s+") splits on runs of whitespace (collapsing the double spaces), giving ["the","quick","brown"]; then iterate in reverse appending to a StringBuilder with single-space separators, or use String.join(" ", reversedList). The \\s+ (one-or-more whitespace) and the leading strip() are what make it robust to messy input — naive split(" ") would produce empty strings from the double spaces.

🎯 Key Takeaways

  • Strings are immutable — every "modifying" method returns a new String; you must assign the result.
  • Immutability gives safe sharing, thread-safety, the string pool, and stable hash keys.
  • Compare content with .equals(), never == (which is identity and only "works" for pooled literals).
  • Building strings in a loop with += is O(n²) — use StringBuilder.
  • split/replaceAll take regexes (escape a literal dot); text blocks and formatted/String.join clean up multi-line and templated text.
  • Always specify a charset (UTF_8) at byte⇄String boundaries; use Locale.ROOT for identifiers; never concatenate user input into SQL.
