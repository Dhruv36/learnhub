Records & Sealed Types Modern Java

What you'll master in this lesson: records — the concise, immutable data carriers (Java 16+) that generate equals/hashCode/toString for you and replace mountains of boilerplate; how to add validation and methods to them; sealed types (Java 17+) that let you say "these are the only subtypes"; and how records + sealed types + pattern matching combine into algebraic data types that make illegal states unrepresentable. These are the features that modernized Java's data modeling — and increasingly common in interviews.

Part 1 — Records: Data Classes Without the Boilerplate

// BEFORE — a simple immutable value object was ~40 lines of boilerplate:
public final class Point {
    private final int x, y;
    public Point(int x, int y) { this.x = x; this.y = y; }
    public int x() { return x; }  public int y() { return y; }
    @Override public boolean equals(Object o) { /* ...15 lines... */ }
    @Override public int hashCode() { return Objects.hash(x, y); }
    @Override public String toString() { return "Point[x=" + x + ", y=" + y + "]"; }
}

// AFTER — a record. ONE line, same behavior:
public record Point(int x, int y) { }
// The compiler generates: private final fields, a canonical constructor, accessors x()/y(),
// and correct equals(), hashCode(), toString(). All from the "components" (x, y).
💡 A record is the honest way to write a data carrier. It's implicitly final, its fields are final (so it's immutable — Immutability lesson), and the generated equals/hashCode honor the contract (equality lesson) automatically — no vanishing-object bug. Use records for DTOs, value objects, coordinates, API responses, map keys, and any "group of related values" — which is a huge fraction of the classes you write.

Part 2 — Adding Validation & Behavior

public record Money(long cents, String currency) {
    // COMPACT CONSTRUCTOR — validate/normalize; no parameter list, fields assigned automatically after:
    public Money {
        if (cents < 0) throw new IllegalArgumentException("cents must be non-negative");
        currency = currency.toUpperCase();      // normalization writes to the component
    }
    // You can add methods (records aren't just dumb data):
    public Money plus(Money other) {
        if (!currency.equals(other.currency)) throw new IllegalArgumentException("currency mismatch");
        return new Money(cents + other.cents, currency);
    }
    // ...and static factories:
    public static Money ofDollars(long dollars) { return new Money(dollars * 100, "USD"); }
}
🎯 The compact constructor (public Money { ... } — note no parameter list) is where you validate and normalize: it runs before the fields are assigned, so throwing here means an invalid Money can never exist (the invariant guarantee from the OOP lesson). Records can have methods, static factories, and additional constructors — they're real classes optimized for the data-carrier case, not a limited "struct".

Part 3 — What Records Can and Can't Do

Implement interfacesBe mutable

⚠️ Records are shallowly immutable. The record's own fields are final, but if a component is a mutable object (a List, a Date), that object can still be mutated — the same defensive-copying issue from the Immutability lesson. Records do not auto-copy mutable components, so for a record Team(String name, List<Player> players) you should defensively copy in the compact constructor (players = List.copyOf(players)) if you want true immutability.

Part 4 — Sealed Types: A Closed Set of Subtypes

// "sealed" restricts WHICH classes can extend/implement — you list the permitted subtypes:
public sealed interface Shape permits Circle, Rectangle, Triangle { }

public record Circle(double radius) implements Shape { }
public record Rectangle(double w, double h) implements Shape { }
public record Triangle(double base, double height) implements Shape { }
// No other class can implement Shape — the hierarchy is CLOSED and known to the compiler.

// A permitted subtype must be: final, sealed (with its own permits), or non-sealed (reopened).
🔍 Sealed types let you tell the compiler "these are ALL the possibilities." Normally an interface can be implemented by anyone, so the compiler can never know all implementations. Sealing enumerates them, which enables exhaustiveness checking: a switch over a sealed type that covers every permitted subtype needs no default case, and if you later add a subtype, every such switch fails to compile until you handle it. This turns "did I handle all cases?" from a runtime hope into a compile-time guarantee.

Part 5 — Deep Dive: Algebraic Data Types (records + sealed + pattern matching)

The three modern features compose into something powerful: algebraic data types — a way to model "a value is exactly one of these shapes" that the compiler fully understands.

sealed interface Result<T> permits Success, Failure { }
record Success<T>(T value) implements Result<T> { }
record Failure<T>(String error) implements Result<T> { }

// A switch over the sealed type is EXHAUSTIVE — compiler-checked, no default needed:
String describe(Result<User> r) {
    return switch (r) {                                  // pattern matching for switch (21+)
        case Success<User> s -> "got " + s.value().name();
        case Failure<User> f -> "failed: " + f.error();
        // no default! — if you add a third subtype, THIS won't compile until handled
    };
}
🔍 Senior framing: this is the functional-programming concept of "sum types" (a value is one of several variants) plus "product types" (a record groups fields), long standard in Haskell, Rust, and Scala, now first-class in Java. The payoff is making illegal states unrepresentable: instead of a nullable field, a boolean flag, and a comment, you model exactly the valid shapes as sealed subtypes, and the compiler forces every consumer to handle each one. A whole class of "I forgot to handle that case" bugs disappears. Records give you the data variants cheaply, sealing closes the set, and pattern-matching switch consumes them exhaustively — they were co-designed for this (Pattern Matching lesson goes deeper). This is the biggest shift in how modern Java models data.

Part 6 — When to Use (and Not Use) Records

  • Use a record when the class is fundamentally a transparent carrier of its data — a DTO, a value object, a tuple-like return, an event, a coordinate, a map key. If the class is "just its fields," a record is the right, honest choice.
  • Don't use a record when the class has meaningful mutable state, hides its representation (encapsulation where the internal fields aren't the API), needs inheritance, or its identity isn't defined by its data. Records deliberately expose their components via accessors, so they're the wrong tool when you want to hide the internals.
  • Records are great for local/private modeling too — a quick record inside a method to group intermediate values, or a private record for an internal data structure, costs nothing.
🔍 The mental test: "Is this class defined by its data, and is that data meant to be visible?" If yes → record. If the class is about behavior with hidden state, or it needs to be mutable or extensible, → a regular class. Don't force records where encapsulation of internals matters — their transparency is a feature for data and a mismatch for behavior-rich objects.

Interview Questions

Q1 (beginner): What is a record and what does it generate?

A record (Java 16+) is a concise declaration for an immutable data-carrier class. From its "components" (the header parameters, e.g. record Point(int x, int y)) the compiler generates: private final fields, a canonical constructor, public accessor methods (x(), y()), and correct equals, hashCode, and toString based on all components. The record is implicitly final and its fields are final, so it's immutable. It replaces the ~40 lines of error-prone boilerplate a value object used to need, and the generated equals/hashCode honor the contract automatically.

Q2 (beginner): Can a record have validation or methods?

Yes. A record can have a compact constructor (public Point { ... } with no parameter list) that runs validation/normalization before the fields are assigned — throwing there prevents an invalid instance from ever existing. It can also declare instance methods, static methods, static factories, additional (non-canonical) constructors that delegate to the canonical one, static fields, and implement interfaces. What it can't do is add extra instance fields beyond its components, be mutable, or extend another class. So records are real classes tailored for data, not restrictive structs.

Q3 (mid): What are sealed types and what do they enable?

A sealed class or interface restricts which types may extend or implement it via an explicit permits clause, so the set of direct subtypes is closed and known to the compiler; each permitted subtype must itself be final, sealed, or non-sealed. The key enablement is exhaustiveness checking: because the compiler knows all possible subtypes, a switch (with pattern matching) that covers every permitted subtype needs no default, and adding a new subtype makes every such switch fail to compile until it's handled — turning "did I cover all cases?" into a compile-time guarantee. Sealed types also document intent (this hierarchy is deliberately closed) and, with records and pattern matching, form algebraic data types.

Q4 (mid): Are records fully immutable? What's the catch?

Records are shallowly immutable: their own component fields are final and can't be reassigned, and there are no setters. But if a component is a reference to a mutable object — a List, an array, a Date — that underlying object can still be mutated, because the record stores the reference and doesn't automatically copy it. So a record Team(String name, List<Player> players) exposes and retains the caller's list, which either party can then modify. To get true immutability you defensively copy mutable components in the compact constructor (players = List.copyOf(players)) and rely on accessors returning the immutable copy. This is the same defensive-copying concern as any immutable class; records don't solve it for you.

Q5 (senior): How do records, sealed types, and pattern matching combine into algebraic data types, and why does that matter?

Together they express the two fundamental ways of composing data from functional programming. Product types: a record groups several values into one ("a Point is an x and a y"). Sum types: a sealed interface with permitted record subtypes says a value is exactly one of a closed set of variants ("a Result is a Success or a Failure"). Pattern-matching switch then consumes such a value by destructuring each variant, and because the type is sealed the compiler verifies the switch is exhaustive with no default. Why it matters: this lets you make illegal states unrepresentable. Instead of a single class with nullable fields, boolean flags, and comments describing which combinations are valid, you model exactly the valid shapes as distinct variants; the compiler then forces every consumer to handle every variant and rejects impossible combinations at compile time. A large class of defensive null checks and "I forgot that case" bugs simply cannot occur. This data-modeling style — long standard in Haskell, Rust, Scala — is now idiomatic Java, and it changes how you design domains: enumerate the real states as types rather than encoding them in flags, and let exhaustiveness checking keep every handler honest as the model evolves.

Q6 (senior): When would you deliberately NOT use a record, despite the convenience?

Records are the right tool only when a class is fundamentally a transparent carrier of its data — its identity and equality are defined by its components, and exposing those components via accessors is acceptable. You avoid a record when: (1) the class needs encapsulation of a representation that isn't its API — e.g. it stores data in an internal form and exposes different, computed operations; records force accessors for every component, leaking the internal shape; (2) it must be mutable or model evolving state with a lifecycle (an entity, an accumulator, a builder); (3) it needs inheritance — records are implicitly final and extend Record, so they can't extend a class or be extended; (4) its identity is not value-based — e.g. a JPA entity keyed by a database id where two rows with the same fields are still different entities, and value-equality would be wrong (using records as JPA entities is a known anti-pattern for exactly this reason); or (5) equality/hashCode need custom semantics that don't match "all components equal." In these cases a regular class with private fields, chosen accessors, and behavior methods is the correct design. The test is "is this thing defined by its data, and is that data meant to be visible?" — if not, encapsulate with a normal class rather than exposing it as a record.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Record vs class. Write an immutable Point(int x, int y) as a record, then list what the compiler generated. Confirm equals and toString work as expected.
Solution

record Point(int x, int y) {}. Generated: final fields, canonical constructor, x()/y() accessors, value-based equals/hashCode, and toString like Point[x=1, y=2]. Two Point(1,2) are equals and share a hashCode — no vanishing-object bug, no boilerplate.

2 (easy): Compact constructor validation. Write a record Percentage(int value) that rejects values outside 0–100 in a compact constructor.
Solution

record Percentage(int value) { public Percentage { if (value < 0 || value > 100) throw new IllegalArgumentException("0..100"); } }. The compact constructor validates before assignment, so an out-of-range Percentage can never exist — the invariant is guaranteed by construction, exactly like the encapsulation lesson but in one line.

3 (medium): Defensive copy in a record. Write record Team(String name, List<String> members) that is genuinely immutable. Show the bug without defensive copying, then fix it.
Solution

Without copying, a caller can keep and mutate the passed list, or mutate the list returned by members() — the record isn't truly immutable. Fix in the compact constructor: public Team { members = List.copyOf(members); }, which stores an immutable copy so neither the caller's original nor the accessor's return can be modified. Records don't auto-copy mutable components — you must.

4 (medium): Seal a hierarchy. Model a sealed interface Payment permitting Cash, Card, and Wallet records. Write an exhaustive switch computing a fee, with no default.
Solution

sealed interface Payment permits Cash, Card, Wallet {} with three implementing records. A switch (payment) { case Cash c -> ...; case Card c -> ...; case Wallet w -> ...; } needs no default because the compiler knows those are all the possibilities. Add a fourth payment type and this switch won't compile until you handle it — exhaustiveness as a compile guarantee.

5 (hard): A Result type. Build sealed interface Result<T> with Ok<T>(T value) and Err<T>(String message) records, and a method that pattern-matches to either return the value or a fallback. Explain how this beats returning null.
Solution

The sealed Result with two record variants forces callers to handle both success and failure explicitly via an exhaustive switch, and carries a real error message on failure. Compared to returning null for failure: there's no ambiguity (null could mean "not found" or "error" or a bug), no NPE risk, the failure reason is captured, and the compiler makes ignoring the failure case impossible. It's the algebraic-data-type way to model "success or a described failure" — illegal states (a success with an error message, or vice versa) are unrepresentable.

6 (hard): Explain-it test. Out loud: (a) what a record generates and the mutable-component caveat; (b) what sealing enables (exhaustiveness); (c) how records + sealed + pattern matching make illegal states unrepresentable.
Self-check rubric

Pass (a) if you listed final fields/accessors/equals/hashCode/toString and the shallow-immutability defensive-copy caveat. Pass (b) if you said sealing closes the subtype set enabling default-free exhaustive switches that break on new subtypes. Pass (c) if you connected product+sum types to modeling only valid shapes and forcing every case to be handled. These set up the Pattern Matching lesson directly.

🎯 Key Takeaways

  • Records generate final fields, accessors, and correct equals/hashCode/toString from their components — immutable data carriers in one line.
  • Validate/normalize in the compact constructor; records can have methods, static factories, and implement interfaces — but can't extend classes, add fields, or be mutable.
  • Records are shallowly immutable — defensively copy mutable components (List.copyOf) for true immutability.
  • Sealed types (permits) close the set of subtypes, enabling compiler-checked exhaustiveness (no default needed).
  • Records + sealed + pattern matching = algebraic data types that make illegal states unrepresentable and force every case to be handled.
  • Use records for data defined by its (visible) fields; use a regular class when you need encapsulation, mutability, inheritance, or non-value identity (e.g. JPA entities).
