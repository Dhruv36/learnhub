Syntax, Types & Variables Foundations

What you'll master in this lesson: Java's two families of types (primitives and references) and the single distinction that explains a dozen later gotchas; how variables actually store data; every primitive type and when to use it; autoboxing and its traps; and why int a = b and int[] a = b behave so differently. This is the bedrock — the "is Java pass-by-reference?" interview question, the "why did my object change?" bug, and the "== vs .equals" confusion all trace back here.

Part 1 — Java Is Statically Typed (and why you'll be glad)

Every variable in Java has a type known at compile time, and the compiler enforces it. You cannot put a String into an int variable, and you'll be told the moment you try — before the program runs. Contrast Python or JavaScript, where a type mismatch surfaces as a runtime crash, possibly in production, possibly at 3am.

int count = 42;          // count is forever an int
count = "hello";         // ❌ COMPILE ERROR: incompatible types — caught immediately
String name = "Ada";     // name is a String
name = 42;               // ❌ COMPILE ERROR
💡 The trade-off, stated honestly: static typing costs you some verbosity (you declare types) in exchange for the compiler catching a whole class of bugs for free, plus better tooling (autocomplete, refactoring) because the IDE knows every type. Java bets that on a large, long-lived codebase this trade is overwhelmingly worth it — and decades of enterprise Java say the bet paid off. var (later in this track) reduces the verbosity without giving up the static type.

Part 2 — The Central Distinction: Primitives vs References

This is the concept of the lesson. Java has exactly two kinds of types, and they store data differently:

int x = 42;                 // PRIMITIVE — the variable HOLDS THE VALUE 42 directly
int y = x;                  // copies the value; y and x are now independent
y = 99;                     // x is STILL 42 — they were never linked

String s = "hi";            // REFERENCE — the variable holds a POINTER to an object
int[] a = {1, 2, 3};        // reference — 'a' points to the array object on the heap
int[] b = a;                // copies the REFERENCE — a and b point to the SAME array
b[0] = 99;                  // a[0] is now 99 too! (one object, two pointers to it)

Picture it physically. A primitive variable is a labeled box with the number written inside. A reference variable is a labeled box with an address written inside — the actual object lives elsewhere (on the heap), and multiple boxes can hold the same address.

PRIMITIVE:   x [ 42 ]        y [ 42 ]        ← two independent boxes

REFERENCE:   a [ →──┐ ]                     ┌─────────────┐
             b [ →──┼──────────────────────→│ [99, 2, 3]  │  one array,
                    └──────────────────────→└─────────────┘  two pointers
Primitives (8 of them, lowercase)References (everything else)
What the variable holdsThe value itselfA pointer to an object
Assignment / passing copiesThe value (independent copy)The pointer (shared object)
Can be null?NoYes
Where it livesStack (fast)Object on the heap (GC-managed)

🎯 Internalize this and half of Java's "surprises" evaporate. "Why did modifying b change a?" — because they point to the same object. "Why does == fail on Strings?" — because it compares pointers, not contents. "Is Java pass-by-reference?" — no, but the reference's value (the pointer) is copied. Every one of these is this Part restated.

This has caused real production incidents (counting things that grew past 2 billion). Use long when a count could get large, or Math.addExact() which throws on overflow.

Part 4 — Wrapper Classes & Autoboxing

Every primitive has a matching wrapper class — the object version: int→Integer, double→Double, boolean→Boolean, and so on. You need them because collections and generics only hold objects, not primitives — a List<int> is illegal; it must be List<Integer>.

Integer boxed = 42;         // AUTOBOXING: int → Integer automatically
int prim = boxed;           // AUTO-UNBOXING: Integer → int automatically
List<Integer> nums = new ArrayList<>();
nums.add(5);                // the int 5 is autoboxed to Integer to go in the list

Autoboxing is convenient but hides two genuine traps that appear in interviews and bugs:

// TRAP 1 — == on wrapper objects compares REFERENCES, not values:
Integer a = 1000, b = 1000;
System.out.println(a == b);        // false! two distinct Integer objects
System.out.println(a.equals(b));   // true — .equals compares VALUE (always use this)

Integer c = 100, d = 100;
System.out.println(c == d);        // true?! — the Integer CACHE (-128..127) reuses objects
// NEVER rely on this — the same code flips from true to false based on the number's size.

// TRAP 2 — unboxing a null wrapper throws:
Integer maybe = null;
int val = maybe;                   // NullPointerException on the hidden unboxing!
🎯 Rules that save you: (1) Always use .equals() to compare wrapper values, never ==. (2) Prefer primitives (int) over wrappers (Integer) unless you specifically need nullability or a collection — primitives can't be null and don't have the cache trap. (3) Be suspicious of any Integer that could be null in arithmetic.

Part 5 — Declaring & Initializing Variables

int count;                  // DECLARED but not initialized
// System.out.println(count); ❌ compile error: "variable count might not have been initialized"
count = 10;                 // now initialized — Java forces you to before use (local vars)

final int MAX = 100;        // final = cannot be reassigned (Java's "constant")
// MAX = 200;               ❌ compile error

var total = 0;              // var — the compiler INFERS the type (int here). Still static!
var name  = "Ada";          // inferred String. var only works with an initializer.
// var x;                   ❌ can't infer a type with no value

Two important nuances beginners miss:

  • Local variables must be initialized before use — the compiler enforces "definite assignment". But instance/static fields get defaults automatically: 0 for numbers, false for boolean, null for references. This asymmetry surprises people.
  • var is not dynamic typing. var name = "Ada" makes name a String permanently — the compiler just saved you typing the type. You can't later assign an int to it. It's inference, not "any".

Part 6 — Deep Dive: Is Java Pass-by-Value or Pass-by-Reference?

This is the single most misunderstood question in Java, and getting it exactly right marks you as someone who understands the language. The answer: Java is always pass-by-value. Full stop. But you must understand what "value" means for each type family.

// For a PRIMITIVE, the value is the number — a copy is passed:
void tryToChange(int n) { n = 99; }
int x = 5;  tryToChange(x);  // x is STILL 5 — the method got a copy

// For a REFERENCE, the value passed is the POINTER (also copied):
void mutate(int[] arr) { arr[0] = 99; }      // follows the pointer, mutates the shared object
void reassign(int[] arr) { arr = new int[]{7}; }  // repoints the LOCAL copy only

int[] a = {1, 2, 3};
mutate(a);      // a[0] is now 99  — the method changed the object 'a' points to ✅
reassign(a);    // a is UNCHANGED  — the method repointed its own copy of the pointer ✅

The resolution: a method receives a copy of the reference (the pointer value). Through that copy it can reach and mutate the same object the caller sees. But if it reassigns the parameter to a new object, it only changes its local copy of the pointer — the caller's variable still points at the original. "Java can change what your object contains, but can't make your variable point somewhere else."

🔍 Senior framing: the confusion exists because "pass-by-value of a reference" looks like pass-by-reference for mutation. True pass-by-reference (as in C++ with &) would let a method reassign the caller's variable itself — Java cannot do this, and that's the precise, provable difference. Demo it with the reassign method above: if Java were pass-by-reference, a would become {7}; it doesn't.

Interview Questions

Q1 (beginner): What's the difference between a primitive and a reference type?

A primitive variable holds the actual value (e.g. the number 42) directly; there are exactly eight (boolean, byte, short, int, long, float, double, char), they can't be null, and they're copied by value. A reference variable holds a pointer to an object that lives on the heap; it can be null, and assigning/copying it copies the pointer so two variables can share one object. The practical upshot: mutating an object through one reference is visible through every reference to it, whereas primitives are always independent copies.

Q2 (beginner): Why does Integer a=1000,b=1000; a==b give false but with 100 give true?

== on objects compares references (identity), not values. Autoboxing small values uses the Integer cache for −128..127, returning the same cached object, so c==d is true at 100. 1000 is outside the cache, so autoboxing creates two distinct objects and a==b is false. The lesson: never use == for wrapper value comparison — always .equals(). Relying on the cache is a latent bug that flips based on the magnitude of the number.

Q3 (mid): Is Java pass-by-value or pass-by-reference? Prove it.

Always pass-by-value. For primitives the value copied is the number, so a method can't change the caller's variable. For objects the value copied is the reference (pointer): the method gets its own copy pointing at the same object, so it can mutate that object (visible to the caller) but cannot repoint the caller's variable. Proof: a method that does list.add(x) affects the caller (mutation through the shared object), but a method that does list = new ArrayList<>() does not (it only reassigned its local copy of the pointer). If Java were pass-by-reference, the second would change the caller's variable too — it doesn't.

Q4 (mid): When would you choose Integer over int, and what does it cost?

Choose the wrapper Integer when you need (1) nullability — e.g. a database column or API field that can be absent, where null means "no value" and 0 would be a wrong sentinel; or (2) to store values in generic collections (List<Integer>, Map<String,Integer>), which only hold objects. Costs: memory (an object header plus the value, vs a bare 4 bytes), the risk of a NullPointerException on auto-unboxing, the == identity trap, and boxing/unboxing overhead in hot loops. Default to the primitive; reach for the wrapper only when nullability or generics require it.

Q5 (senior): Explain integer overflow and how you'd guard a monetary or counting calculation against it.

Java's integer types are fixed-width two's-complement and wrap silently on overflow — Integer.MAX_VALUE + 1 becomes Integer.MIN_VALUE, no exception. This has caused real outages (counters, buffer sizes, financial sums exceeding ~2.1 billion). Guards: (1) use a wider type — long for large counts/timestamps; (2) use Math.addExact/multiplyExact, which throw ArithmeticException on overflow so you fail loudly rather than corrupt data; (3) for arbitrary precision use BigInteger; (4) for money, never use binary floating point at all — double can't represent 0.10 exactly and accumulates rounding error — use BigDecimal with explicit scale and rounding, or integer cents. The senior instinct is to distrust silent numeric limits and choose types/operations that make overflow impossible or loud.

Q6 (senior): A method receives a StringBuilder, appends to it, and the caller sees the change; another receives a String, does s += "x", and the caller sees nothing. Reconcile with "Java is pass-by-value".

Both are pass-by-value of a reference — the difference is mutability, not passing semantics. StringBuilder is mutable: append modifies the shared object the copied pointer refers to, so the caller observes it. String is immutable: s += "x" is s = s.concat("x"), which creates a new String and reassigns the method's local pointer copy to it — the caller's pointer still refers to the original, so the caller sees nothing. This is the perfect illustration that "can the caller see my change?" depends on whether you mutated the shared object (yes) or reassigned your local reference (no) — never on pass-by-reference, which Java doesn't have. Immutability (a later lesson) is precisely what makes String safe to share despite this.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Independence vs sharing. In JShell, set int p = 5; int q = p; q = 9; and print p. Then int[] a = {5}; int[] b = a; b[0] = 9; and print a[0]. Explain the different results.
Solution

p is still 5 (primitives copy the value; q is independent). a[0] is now 9 (b copied the reference; a and b point to the same array). This one experiment is the whole primitive/ reference distinction — if you can explain it, you understand the lesson.

2 (easy): The wrapper equality trap. Predict and then test: Integer a = 127, b = 127; a == b; and Integer c = 128, d = 128; c == d;. Then redo both with .equals().
Solution

a == b → true (127 is in the −128..127 cache, same object). c == d → false (128 is outside the cache, distinct objects). Both .equals() → true. Conclusion you should carry forever: use .equals() for wrapper values; == is an identity check that happens to "work" for small numbers, which makes it a dangerous bug.

3 (medium): Prove pass-by-value. Write two methods on an int[] parameter: one that sets arr[0] = 99, one that does arr = new int[]{99}. Call each on the same array and print the array after. Explain why one change is visible and the other isn't.
Solution

The first (mutating arr[0]) is visible to the caller — it followed the shared pointer and changed the object. The second (reassigning arr) is invisible — it only repointed the method's local copy of the pointer. This proves Java passes a copy of the reference: mutation of the shared object propagates; reassignment of the parameter does not. If Java were pass-by-reference, both would be visible.

4 (medium): Overflow and the fix. Compute 1_000_000 * 1_000_000 as an int and print it. Then compute it as 1_000_000L * 1_000_000. Then try Math.multiplyExact(1_000_000, 1_000_000).
Solution

The int version overflows to a wrong (garbage) value silently. The long version gives the correct 1_000_000_000_000 (note the L forces long arithmetic — without it, int*int overflows before the result is widened). Math.multiplyExact throws ArithmeticException — the "fail loud" option. Lesson: pick a type wide enough, and force it early; use the *Exact methods when silent corruption is unacceptable.

5 (hard): The null-unboxing NPE. Write a method int total(Map<String,Integer> m, String key) that returns m.get(key) * 2. Call it with a key that isn't in the map. Diagnose the crash and fix it robustly.
Solution

m.get(missingKey) returns null (an Integer), and null * 2 auto-unboxes null → NullPointerException — and the stack trace points at the multiplication, not the lookup, which is why this bug is confusing. Fix: Integer v = m.getOrDefault(key, 0); return v * 2;, or guard with an explicit null check. The deeper lesson: any place a wrapper meets arithmetic is a hidden unboxing that can NPE — treat Map.get results as nullable.

6 (hard): Explain-it test. Out loud, no notes: (a) draw the box-and-pointer picture for primitives vs references; (b) state Java's parameter passing rule and prove it with the mutate-vs-reassign example; (c) explain why == is unsafe on wrappers and what the Integer cache has to do with it.
Self-check rubric

Pass (a) if you distinguished "value in the box" from "address in the box, object elsewhere". Pass (b) if you said "always pass-by-value; for objects the copied value is the pointer" and used reassign-is-invisible as the proof. Pass (c) if you tied ==-is-identity to the cache making it deceptively work for −128..127. These three explanations underpin equality, collections, and immutability lessons ahead.

🎯 Key Takeaways

  • Java is statically typed — types are known and checked at compile time.
  • Primitives hold the value (copied independently); references hold a pointer (copied to share the same object) — the distinction behind most Java surprises.
  • Eight primitives; int and double are the defaults; whole numbers over 2.1 billion need long — overflow is silent.
  • Wrappers (Integer…) enable collections/generics and nullability, but bring the == identity trap, the Integer cache, and null-unboxing NPEs — prefer primitives, always compare with .equals().
  • Java is always pass-by-value; for objects the copied value is the reference, so methods can mutate the shared object but can't repoint the caller's variable.
  • var is compile-time inference (still static), final makes a variable non-reassignable; use BigDecimal for money.
