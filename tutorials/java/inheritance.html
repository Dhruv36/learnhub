Inheritance & Polymorphism Object-Oriented Java

What you'll master in this lesson: how a subclass extends a superclass and what it inherits; the exact mechanism of polymorphism (dynamic dispatch) and why it's the engine of extensible design; the roles of super, @Override, final, and abstract; upcasting/downcasting and the instanceof check; and — most important — when not to use inheritance, because misused inheritance is one of the most common sources of rigid, fragile code. This is a topic where the interview tests judgment, not just syntax.

Part 1 — extends: The "is-a" Relationship

public class Animal {
    protected String name;                 // protected → visible to subclasses
    public Animal(String name) { this.name = name; }
    public String speak() { return "..."; }
    public void eat() { System.out.println(name + " eats"); }
}

public class Dog extends Animal {          // Dog IS-A Animal
    public Dog(String name) {
        super(name);                       // call the superclass constructor FIRST
    }
    @Override
    public String speak() { return "Woof"; }   // specialize inherited behavior
    public void fetch() { System.out.println(name + " fetches"); }  // add new behavior
}

Dog d = new Dog("Rex");
d.eat();      // inherited from Animal — "Rex eats"
d.speak();    // overridden — "Woof"
d.fetch();    // Dog-specific

A subclass inherits the superclass's non-private members and can (a) use them as-is, (b) override methods to specialize behavior, and (c) add new members. The relationship is "is-a": a Dog is an Animal. If you can't say "X is-a Y" truthfully, inheritance is the wrong tool (Part 6).

Part 2 — super and Constructor Chaining

public class Dog extends Animal {
    private String breed;
    public Dog(String name, String breed) {
        super(name);            // MUST be the first statement — builds the Animal part first
        this.breed = breed;
    }
    @Override
    public String speak() {
        return super.speak() + " (from Dog)";   // super.method() calls the PARENT's version
    }
}
🎯 Every constructor calls its superclass constructor first. If you don't write super(...) explicitly, the compiler inserts super() (the no-arg one) — and if the superclass has no no-arg constructor, your code won't compile until you call the right super(args). This "construct from the top of the hierarchy down" rule guarantees each layer is initialized before the layer below uses it. super.method() (distinct from the constructor call) invokes the parent's implementation from within an override.

Part 3 — Polymorphism & Dynamic Dispatch (the engine)

Animal[] zoo = { new Dog("Rex"), new Cat("Milo"), new Animal("thing") };
for (Animal a : zoo) {
    System.out.println(a.speak());   // "Woof", "Meow", "..." — the RUNTIME type decides!
}

Every element is declared as Animal, but at runtime the JVM calls the actual object's overridden method — Dog.speak() for the Dog, Cat.speak() for the Cat. This is polymorphism via dynamic (runtime) dispatch, and it's the whole point of inheritance: you write code against the general type (Animal) and it automatically does the right thing for each specific type — including types that don't exist yet.

🔍 Why this is powerful: a method void feedAll(List<Animal> animals) works for any current or future Animal subclass without modification. You've decoupled the code that uses animals from the code that defines each kind. This "program to the supertype" idea is the seed of every plugin system, framework callback, and the Strategy pattern — though, as Part 6 argues, interfaces usually deliver it more cleanly than class inheritance.

Part 4 — abstract Classes

public abstract class Shape {              // abstract: CANNOT be instantiated directly
    public abstract double area();        // abstract METHOD: no body — subclasses MUST implement
    public void describe() {              // concrete method: shared behavior for all shapes
        System.out.println("area = " + area());   // calls the subclass's area() polymorphically
    }
}
public class Circle extends Shape {
    private final double r;
    public Circle(double r) { this.r = r; }
    @Override public double area() { return Math.PI * r * r; }
}
// new Shape();   ❌ compile error — abstract classes can't be instantiated
Shape s = new Circle(2);   s.describe();   // "area = 12.566..."

An abstract class is a partially-built base: it can hold state (fields), concrete methods (shared behavior), and abstract methods (a contract each subclass must fulfill). It exists to be extended, capturing what a family of related types have in common while leaving the varying parts abstract. (When there's no shared state or implementation, an interface is the better tool — next lesson.)

Part 5 — Casting, instanceof & final

Animal a = new Dog("Rex");     // UPCAST — implicit, always safe (a Dog IS an Animal)
// a.fetch();                  ❌ compile error — the Animal type doesn't know about fetch()

if (a instanceof Dog d) {      // pattern-matching instanceof (16+): checks AND binds 'd'
    d.fetch();                 // safe — we've verified it's a Dog
}

Dog d2 = (Dog) a;              // DOWNCAST — explicit; throws ClassCastException if 'a' isn't a Dog

// final PREVENTS extension/override:
public final class String { }           // no class can extend String
public final int method() { }           // no subclass can override this method
🔍 Upcasting (subtype → supertype) is implicit and always safe. Downcasting (supertype → subtype) is a runtime bet that throws ClassCastException if wrong — so guard it with instanceof (ideally the pattern-matching form that checks and binds in one step). Frequent downcasting is a design smell: it usually means you're circumventing polymorphism and should push the behavior into an overridden method instead. final on a class forbids subclassing (e.g. String); on a method it forbids overriding.

Part 6 — Deep Dive: Favor Composition Over Inheritance

This is the most important idea in the lesson and a staple of senior interviews. Inheritance is seductive for code reuse, but it has real costs, and modern design leans away from it:

  • Inheritance is the tightest coupling there is. A subclass depends on the superclass's implementation, not just its interface. A change to the superclass can silently break subclasses — the "fragile base class" problem.
  • It breaks encapsulation. To override correctly you often must know how the superclass calls its own methods internally (self-use). The classic example: extending ArrayList to count additions, overriding both add and addAll — but addAll internally calls add, so you double-count. The subclass is coupled to an internal detail.
  • You get one superclass, forever. Single inheritance means a rigid, committed hierarchy. Real requirements cut across hierarchies.
// INHERITANCE (fragile) — Stack extends Vector, so it inherits Vector's insert-anywhere methods,
// breaking the "only push/pop at the top" invariant. (java.util.Stack is a real cautionary tale.)

// COMPOSITION (robust) — HAS-A instead of IS-A:
public class InstrumentedList<E> {
    private final List<E> list;            // wraps a list instead of extending one
    private int addCount = 0;
    public InstrumentedList(List<E> list) { this.list = list; }
    public boolean add(E e)  { addCount++; return list.add(e); }   // no self-use surprise
    public int getAddCount() { return addCount; }
}
// Composition forwards to a held object. No coupling to internals, works with ANY List
// implementation, and can't be broken by a superclass change. This is the Decorator pattern.
🔍 Senior rule (Effective Java, Item 18): use inheritance only for a genuine "is-a" relationship and when the superclass was designed and documented for extension. Otherwise favor composition + interfaces: hold a collaborator, forward to it, depend on its interface. Composition is more flexible (swap implementations, inject mocks), avoids fragile-base-class bugs, and isn't limited to one parent. This principle is the foundation of dependency injection (Spring) and most design patterns. "Prefer composition" doesn't mean "never inherit" — it means inherit deliberately, not reflexively for reuse.

Interview Questions

Q1 (beginner): What does a subclass inherit, and what does super do?

A subclass inherits all non-private members (fields and methods) of its superclass — private members exist in the object but aren't directly accessible. It can use them, override methods, and add new members. super has two uses: super(args) as the first statement of a constructor calls the superclass constructor to initialize the inherited part; and super.method() inside an override calls the superclass's version of that method (useful when you want to extend rather than replace the parent's behavior).

Q2 (beginner): What is polymorphism, and how does Java implement it?

Polymorphism means a single reference of a supertype can refer to objects of different subtypes, and calling an overridden method invokes the actual object's implementation. Java implements it via dynamic dispatch: for an overridable instance method, the JVM looks up the method on the object's runtime class (through a method table), not the declared type of the reference. So Animal a = new Dog(); a.speak(); calls Dog.speak(). This lets you write code against the general type that automatically works for every subtype, including ones added later — the basis of extensible design.

Q3 (mid): Abstract class vs interface — when would you use an abstract class?

Use an abstract class when you have a family of related types that share state and/or concrete implementation as well as a contract — the abstract class provides fields, common methods, and abstract methods subclasses must implement, capturing the "template" of the family. You accept single inheritance (a class extends only one). Use an interface when you're defining a capability that possibly-unrelated types can have (a class implements many), with no shared state. Modern Java blurs this because interfaces can have default methods, but the deciding factors remain: shared mutable state and a single is-a family push toward an abstract class; a cross-cutting capability implemented by many types pushes toward an interface. (Full treatment next lesson.)

Q4 (mid): Explain upcasting and downcasting and when each is safe.

Upcasting converts a subtype reference to a supertype (Animal a = new Dog()); it's implicit and always safe because a Dog genuinely is an Animal — you just lose access to Dog-specific members through that reference. Downcasting converts a supertype reference back to a subtype (Dog d = (Dog) a); it's explicit and unsafe — a runtime ClassCastException if the object isn't actually that subtype. So downcasts must be guarded, ideally with pattern-matching instanceof (if (a instanceof Dog d)) which tests and binds in one step. Needing to downcast frequently signals a design where you should have used polymorphism (an overridden method) instead of type-checking.

Q5 (senior): Explain the fragile base class problem with a concrete example.

The fragile base class problem is that a subclass can break when its superclass changes, even if the superclass's public contract is unchanged, because the subclass depends on the superclass's internal behavior. The canonical example (Effective Java): you extend HashSet to count how many elements are ever added, overriding add to increment a counter and call super.add, and overriding addAll similarly. It over-counts, because HashSet.addAll is implemented by calling add internally — so adding three elements via addAll increments the counter three times in your addAll override plus three more via the inherited self-calls to your add. Your subclass silently depends on an undocumented implementation detail (self-use), and a future JVM that changes how addAll works would change your behavior again. The fix is composition: a wrapper that holds a Set and forwards to it counts correctly and is immune to the collection's internals. This is precisely why "favor composition over inheritance" exists.

Q6 (senior): Why is calling an overridable method from a constructor dangerous?

Because construction runs top-down (superclass constructor first), but dynamic dispatch always targets the most-derived override. So if a superclass constructor calls an overridable method, it invokes the subclass's override before the subclass constructor has run — meaning the subclass's own fields are still at their default values (null/0/false) when its override executes. The override then operates on an uninitialized object, producing NullPointerExceptions or silently wrong behavior. Classic demo: a base constructor calls an overridable init(); the subclass override reads a field the subclass constructor hasn't set yet. The rule: constructors should only call private, static, or final methods (which can't be overridden), never overridable instance methods. This is also an argument for making classes final or designing extension points very deliberately.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Basic hierarchy. Create Animal with a speak() returning "...", and Dog/Cat overriding it. Put them in an Animal[] and loop, printing each speak(). Confirm the runtime type decides.
Solution

The loop over an Animal[] containing a Dog and a Cat prints "Woof" and "Meow" — the JVM dispatches to each object's overridden method despite the declared type being Animal. That's polymorphism via dynamic dispatch, demonstrated in five lines. Add @Override to both to let the compiler verify your signatures.

2 (easy): Constructor chaining. Give Animal a name field set via its constructor, and make Dog call super(name). Try removing the super(name) call and read the compiler error.
Solution

With Animal(String name) and no no-arg constructor, omitting super(name) from Dog fails: the compiler tries to insert super(), finds no matching no-arg constructor, and errors ("there is no default constructor available in Animal"). Adding super(name) as the first statement fixes it. This shows the mandatory top-down construction rule.

3 (medium): Abstract template. Make an abstract Shape with abstract area() and concrete describe() that prints the area. Implement Circle and Rectangle. Try to instantiate Shape directly.
Solution

new Shape() is a compile error — abstract classes can't be instantiated. describe() (defined once in the base) calls area() polymorphically, so each shape describes itself correctly. This is the template-method shape: shared skeleton in the base, varying step abstract. Note describe depends only on the abstract contract, not on any concrete shape.

4 (medium): Safe downcast. Given an Animal a that might be a Dog, call fetch() only if it really is one, using pattern-matching instanceof. Then show the version that throws ClassCastException.
Solution

Safe: if (a instanceof Dog d) d.fetch(); — checks and binds d only when a is a Dog. Unsafe: ((Dog) a).fetch(); throws ClassCastException if a is actually a Cat. The pattern form is the modern, safe idiom. If you find yourself doing this a lot, reconsider — a doAction() overridden per subclass often removes the need to check types at all.

5 (hard): Inheritance vs composition. Build a "counting set" that tracks how many elements were ever added — first by extending HashSet (overriding add and addAll), then by composition (wrapping a Set). Add three elements via addAll and compare the counts.
Solution

The inheritance version over-counts (e.g. reports 6 for 3 elements) because HashSet.addAll internally calls add, triggering your overridden add in addition to your overridden addAll. The composition version (hold a Set, forward add/addAll to it, increment once) reports the correct 3 and is immune to how the underlying set is implemented. This is the fragile-base-class problem and its fix, made concrete — the single most convincing argument for composition.

6 (hard): Explain-it test. Out loud: (a) how dynamic dispatch makes polymorphism work; (b) the fragile base class problem and why composition avoids it; (c) why you shouldn't call overridable methods from constructors.
Self-check rubric

Pass (a) if you said the JVM dispatches on the object's runtime type via its method table, so the override runs regardless of the reference's declared type. Pass (b) if you gave the self-use over-count example and explained composition forwards to a held object with no internal coupling. Pass (c) if you noted top-down construction means the override runs before the subclass's fields are initialized. These are staple senior-interview answers and set up the Interfaces and Design Patterns lessons.

🎯 Key Takeaways

  • extends models an is-a relationship; a subclass uses, overrides, and adds members.
  • Constructors chain top-down — super(...) runs first; super.method() calls the parent's version.
  • Polymorphism via dynamic dispatch calls the runtime object's overridden method — the engine of extensible design.
  • Abstract classes provide shared state + partial implementation + a contract; upcasting is safe, downcasting needs an instanceof guard.
  • Favor composition over inheritance: inheritance is the tightest coupling and causes the fragile base class problem — inherit only for genuine is-a with a class designed for extension.
  • Never call overridable methods from a constructor (the override runs before the subclass is initialized).
