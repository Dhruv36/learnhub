Generics Core Libraries

What you'll master in this lesson: why generics exist (compile-time type safety without casts); how to write generic classes and methods; bounded type parameters and wildcards (? extends / ? super) with the PECS rule that every interviewer asks about; and — the concept that explains half of generics' quirks — type erasure, which is why you can't do new T[] or list instanceof List<String>. Generics look intimidating; this lesson makes the rules make sense by grounding them in erasure.

Part 1 — Why Generics Exist

// BEFORE generics (Java < 5) — collections held Object, requiring casts and risking runtime errors:
List names = new ArrayList();
names.add("Ada");
names.add(42);                       // oops — no one stopped this
String s = (String) names.get(1);    // ClassCastException at RUNTIME (42 isn't a String)

// WITH generics — the type is part of the contract, checked at COMPILE time:
List<String> names = new ArrayList<>();
names.add("Ada");
names.add(42);                       // ❌ COMPILE ERROR — caught immediately
String s = names.get(0);             // no cast needed — the compiler knows it's a String
💡 Generics move type errors from runtime to compile time and eliminate casts. The type parameter (<String>) is a promise the compiler enforces: this list holds only Strings. You get safety and cleaner code (no casting). This is the same "catch it at compile time" philosophy as static typing, applied to containers and reusable code.

Part 2 — Generic Classes & Methods

// A generic CLASS — T is a type parameter, chosen by the user of the class:
public class Box<T> {
    private T value;
    public void set(T value) { this.value = value; }
    public T get() { return value; }
}
Box<String> b = new Box<>();     // T = String for this instance
b.set("hi");  String s = b.get();  // type-safe, no cast

// A generic METHOD — the <T> before the return type declares a per-call type parameter:
public static <T> T firstOrNull(List<T> list) {
    return list.isEmpty() ? null : list.get(0);
}
String first = firstOrNull(List.of("a", "b"));   // T inferred as String

// Convention for type parameter names: T (type), E (element), K/V (key/value), R (result), N (number)

Part 3 — Bounded Type Parameters

// <T extends Number> — T can be Number or any subclass (Integer, Double...). "Upper bound."
public static <T extends Number> double sum(List<T> nums) {
    double total = 0;
    for (T n : nums) total += n.doubleValue();   // can call Number's methods — the bound guarantees them
    return total;
}
sum(List.of(1, 2, 3));        // T = Integer ✅
sum(List.of(1.5, 2.5));       // T = Double ✅
// sum(List.of("a", "b"));    ❌ String isn't a Number

// Multiple bounds: <T extends Number & Comparable<T>>
🎯 A bound (extends) both restricts which types are allowed and enables calling that supertype's methods on T. Without the bound, T is treated as Object and you could only call Object methods. Bounds are how a generic method does useful work with its type parameter while staying type-safe.

Part 4 — Wildcards & PECS (the interview favorite)

// ? extends T  — an UPPER-bounded wildcard: "some unknown subtype of T". You can READ T, not write.
double total(List<? extends Number> nums) {     // accepts List<Integer>, List<Double>, ...
    double sum = 0;
    for (Number n : nums) sum += n.doubleValue(); // reading as Number is safe
    // nums.add(1);   ❌ can't add — we don't know the exact subtype (could be List<Double>)
    return sum;
}

// ? super T  — a LOWER-bounded wildcard: "some unknown supertype of T". You can WRITE T, not read as T.
void addNumbers(List<? super Integer> dst) {     // accepts List<Integer>, List<Number>, List<Object>
    dst.add(1);  dst.add(2);                       // writing Integers is safe (they fit any supertype)
    // Integer x = dst.get(0);  ❌ can't read as Integer — the list might be List<Object>
}
🎯 PECS — "Producer Extends, Consumer Super". If a parameter produces values you read out, use ? extends T. If it consumes values you put in, use ? super T. The canonical example is Collections.copy(dest, src): the source produces (List<? extends T> src), the destination consumes (List<? super T> dest). This one mnemonic answers the most common generics interview question — memorize it and understand why (you can only safely read from a producer and write to a consumer).

Part 5 — Deep Dive: Type Erasure (why generics are "not real" at runtime)

Java generics are a compile-time feature. After the compiler checks types, it erases them — replacing type parameters with their bounds (or Object) and inserting casts. At runtime, List<String> and List<Integer> are both just List. This design (chosen for backward compatibility with pre-generics code) explains a family of otherwise-baffling restrictions:

List<String> a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
a.getClass() == b.getClass();          // TRUE — both are just ArrayList at runtime

// Things erasure forbids:
if (a instanceof List<String>) { }   // ❌ can't check a parameterized type (info is gone)
T value = new T();                     // ❌ can't instantiate a type parameter
T[] arr = new T[10];                   // ❌ can't create a generic array
class Box<T> { static T shared; }      // ❌ no generic static fields (T varies per instance)
// Overloading that differs only by generic type doesn't work — both erase to the same signature:
void f(List<String> l) {} void f(List<Integer> l) {}   // ❌ "same erasure"
🔍 Senior nugget: erasure is the unifying explanation for generics' quirks. Can't do new T[]? Because at runtime there's no T to size the array's element type. Can't instanceof List<String>? The <String> doesn't exist at runtime. Why the "unchecked cast" warnings? The compiler can't verify a cast to a generic type because the type argument is erased. Work around it with a Class<T> token passed in (clazz.newInstance() / reflection), or @SuppressWarnings("unchecked") when you've reasoned that it's safe. Contrast C#, whose generics are reified (kept at runtime) — a common comparison interviewers make.

Part 6 — Generics & Arrays Don't Mix (covariance trap)

// Arrays are COVARIANT (a String[] IS an Object[]) — and this is a runtime hole:
Object[] arr = new String[3];
arr[0] = 42;                     // compiles! but throws ArrayStoreException at RUNTIME

// Generics are INVARIANT (a List<String> is NOT a List<Object>) — safer, caught at compile time:
List<Object> list = new ArrayList<String>();   // ❌ COMPILE ERROR — invariance protects you
// This invariance is exactly WHY wildcards (? extends / ? super) exist — to regain flexibility safely.
🔍 Arrays are covariant (a design decision predating generics) and enforce element-type safety only at runtime (ArrayStoreException). Generics chose invariance, catching the equivalent mistake at compile time — strictly safer. The cost of invariance is lost flexibility (you can't pass a List<String> where a List<Object> is expected), which is precisely the gap wildcards fill. This is why generic code uses ? extends/? super so often.

Interview Questions

Q1 (beginner): What problem do generics solve?

They provide compile-time type safety for reusable code and eliminate casts. Before generics, collections held Object, so you could accidentally put the wrong type in and only find out via a ClassCastException at runtime when casting on the way out. With generics the type parameter (List<String>) is enforced by the compiler: wrong-type insertions fail to compile, and retrieval needs no cast because the type is known. So generics catch a whole class of errors earlier and make code cleaner and self-documenting about what a container or method works with.

Q2 (beginner): What does <T extends Number> mean?

It's a bounded type parameter: T may be Number or any subtype (Integer, Double, BigDecimal…), and nothing else. The bound does two things: it restricts callers to number types, and — inside the generic code — it lets you call Number's methods (like doubleValue()) on a T, which you couldn't do with an unbounded T (treated as Object). "extends" here covers both classes and interfaces, and you can have multiple bounds with &.

Q3 (mid): Explain PECS with an example.

PECS = "Producer Extends, Consumer Super". Use ? extends T for a parameter you only read from (it produces T values) and ? super T for a parameter you only write to (it consumes T values). Why: from a List<? extends Number> you can safely read each element as a Number, but you can't add — the actual list might be List<Integer> and you'd be inserting the wrong subtype; from a List<? super Integer> you can safely add Integers (they fit any supertype list) but can only read elements as Object. The textbook example is Collections.copy(List<? super T> dest, List<? extends T> src): src produces, dest consumes. PECS tells you which wildcard makes an API maximally flexible while type-safe.

Q4 (mid): What is type erasure and name three things it prevents?

Type erasure is the compiler's process of removing generic type information after type-checking: type parameters are replaced by their bound (or Object) and casts are inserted, so at runtime List<String> and List<Integer> are both just List. It was chosen for backward compatibility with pre-generics bytecode. Because the type argument doesn't exist at runtime, you cannot: (1) create a generic array (new T[]) or instantiate a type parameter (new T()); (2) use instanceof with a parameterized type (x instanceof List<String>); (3) overload methods that differ only by generic type argument (they erase to the same signature); and relatedly, you get "unchecked" warnings when casting to generic types because the compiler can't verify them. Passing a Class<T> token is the common workaround when you need the type at runtime.

Q5 (senior): Why are arrays covariant but generics invariant, and what problem does each choice trade off?

Arrays are covariant: String[] is a subtype of Object[]. This was necessary before generics so that general array-processing methods (e.g. sort(Object[])) could accept any array, but it's type-unsafe: you can assign a String[] to an Object[] variable and then store an Integer in it, which compiles but throws ArrayStoreException at runtime — the check is deferred to runtime via a per-store type check. Generics are invariant: List<String> is not a subtype of List<Object>, so the analogous mistake (List<Object> l = someStringList; l.add(42);) is a compile error — strictly safer, catching it earlier and without a runtime check. The trade-off is flexibility: invariance means you can't pass a List<String> where List<Object> is expected even when it would be safe (read-only), which is exactly why bounded wildcards (? extends/ ? super) exist — they reintroduce controlled covariance/contravariance while preserving compile-time safety. It also means generics and arrays combine poorly (you can't create new List<String>[]), because the runtime store-check arrays rely on can't work with erased generic element types.

Q6 (senior): You need a factory that creates instances of a generic type T at runtime. Erasure forbids new T(). How do you do it?

Because T is erased, the runtime has no constructor for it, so you must supply the type information explicitly — the standard idiom is to pass a type token, a Class<T>, into the factory and use reflection: T obj = clazz.getDeclaredConstructor().newInstance(); (the older clazz.newInstance() is deprecated). The Class<T> carries the concrete type at runtime that erasure stripped from T. Variations: pass a Supplier<T> (MyType::new) so callers provide construction directly — cleaner and avoids reflection's exceptions and access issues; or, for generic arrays, create (T[]) Array.newInstance(componentClass, size). For capturing generic types with their parameters (like List<String>) despite erasure, frameworks use the "super type token" trick (an abstract generic subclass whose getGenericSuperclass() retains the parameter) — that's how Jackson's TypeReference and Guava's TypeToken work. The general principle: whenever you need a type at runtime that generics erased, thread the type in as data (a Class, a factory lambda, or a super type token).

🏋️ Exercises (attempt before opening solutions)

1 (easy): Generic Box. Write a generic Pair<A,B> holding two values of different types, with accessors. Create a Pair<String,Integer>.
Solution

class Pair<A,B> { private final A a; private final B b; ...getters... } — two independent type parameters. new Pair<>("Ada", 42) gives a Pair<String,Integer> with type-safe accessors, no casts. (A record — record Pair<A,B>(A a, B b){} — does this in one line.)

2 (easy): Bounded method. Write <T extends Comparable<T>> T max(List<T> list) returning the largest element. Explain why the bound is required.
Solution

Loop tracking the max via if (item.compareTo(best) > 0) best = item;. The extends Comparable<T> bound is what lets you call compareTo — without it, T is Object, which has no compareTo, and the code won't compile. The bound both restricts to comparable types and enables the comparison.

3 (medium): PECS in practice. Write a method copy(List<? super T> dest, List<? extends T> src) that copies all elements. Then explain why swapping the wildcards would break it.
Solution

for (T item : src) dest.add(item);. src is a producer (? extends T) so you can read each element as T; dest is a consumer (? super T) so you can add T to it. Swap them and it breaks: you can't read T from a ? super T (only Object), and you can't add T to a ? extends T. PECS isn't arbitrary — each direction only permits the safe operation.

4 (medium): Erasure surprises. Predict which compile and which fail: (a) new ArrayList<String>().getClass() == new ArrayList<Integer>().getClass(); (b) x instanceof List<String>; (c) two overloads f(List<String>) and f(List<Integer>).
Solution

(a) Compiles and is true — both erase to ArrayList at runtime. (b) Compile error — you can't instanceof a parameterized type (only instanceof List). (c) Compile error — both erase to f(List), "same erasure". All three are direct consequences of erasure; recognizing that pattern is the point.

5 (hard): Type token factory. Write a generic <T> T create(Class<T> clazz) that instantiates T via its no-arg constructor. Explain why you need the Class parameter.
Solution

return clazz.getDeclaredConstructor().newInstance(); (handling the reflective exceptions). You need the Class<T> because erasure removes T at runtime — new T() is impossible, so you pass the concrete type in as a token and use reflection to construct it. A cleaner alternative is a Supplier<T> factory (create(MyType::new)) that sidesteps reflection entirely.

6 (hard): Explain-it test. Out loud: (a) what generics buy you over raw Object containers; (b) PECS and the read/write reasoning behind it; (c) type erasure and three restrictions it causes.
Self-check rubric

Pass (a) if you said compile-time safety + no casts. Pass (b) if you tied extends→producer→read and super→consumer→write with why each other operation is unsafe. Pass (c) if you named erasure and three of: no new T[]/new T(), no instanceof parameterized, no generic-only overloads, unchecked warnings. These are the exact generics questions in senior screens.

🎯 Key Takeaways

  • Generics give compile-time type safety and remove casts — errors caught early, code self-documenting.
  • Write generic classes (Box<T>) and methods (<T> T f(...)); bounds (<T extends Number>) restrict and enable method calls.
  • PECS: ? extends T to read from a producer, ? super T to write to a consumer.
  • Type erasure: generics are compile-time only — no new T[]/new T(), no instanceof List<String>, no generic-only overloads.
  • Arrays are covariant (runtime ArrayStoreException); generics are invariant (compile-time safety) — which is why wildcards exist.
  • Need a type at runtime? Pass a Class<T> token or a factory Supplier<T>.
