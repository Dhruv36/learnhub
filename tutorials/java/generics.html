Generics Intermediate

Generics give compile-time type safety to containers and algorithms — List<String> instead of a raw List of Objects. The deep, interview-favorite topics: type erasure (generics vanish at runtime) and bounded wildcards (the PECS rule). Understanding erasure explains most generic "why can't I do that?" errors.

Generic Classes & Methods

public class Box<T> {                    // T = type parameter
    private T value;
    public void set(T v) { value = v; }
    public T get() { return value; }
}
Box<String> b = new Box<>();          // diamond <> infers the type
b.set("hi"); String s = b.get();      // no cast, compiler-checked

public <T> T firstOrNull(List<T> xs) { // generic METHOD
    return xs.isEmpty() ? null : xs.get(0);
}
// Bounded type parameter — "some T that IS a Number":
public <T extends Number> double sum(List<T> xs) {
    double t = 0; for (T x : xs) t += x.doubleValue(); return t;
}

Type Erasure — the Key Concept

// Generics are a COMPILE-TIME feature. At runtime, List<String> and List<Integer> are both
// just List — the type parameter is ERASED (replaced by Object or the bound). Consequences:
List<String> a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
a.getClass() == b.getClass();        // true! same runtime class
new T();                             // ❌ can't — T is unknown at runtime
new T[10];                           // ❌ can't create generic arrays
if (x instanceof List<String>)       // ❌ can't — only `instanceof List<?>`
// Erasure is why generics interoperate with legacy raw types, and why these limits exist.
// (Contrast: C#/C++ reify generics; Java erases for backward compatibility.)

Wildcards & the PECS Rule

// ? extends T = "some subtype of T" — a PRODUCER (you read T out, can't add):
double total(List<? extends Number> xs) {   // accepts List<Integer>, List<Double>, ...
    double t = 0; for (Number n : xs) t += n.doubleValue();   // READ as Number: OK
    return t;                                                  // xs.add(...) would NOT compile
}
// ? super T = "some supertype of T" — a CONSUMER (you write T in, read as Object):
void fill(List<? super Integer> xs) {       // accepts List<Integer>, List<Number>, List<Object>
    xs.add(1); xs.add(2);                    // WRITE Integers: OK
}
// PECS: "Producer Extends, Consumer Super." Read from it → extends; write to it → super.
// This is the rule that makes Collections.copy(dest, src) signatures work.
🧩 Why the asymmetry? With ? extends Number the compiler knows every element is at least a Number (safe to read as Number) but doesn't know the exact subtype (unsafe to add — you might add a Double to a List<Integer>). With ? super Integer it's the reverse: safe to add Integers, but reads only guarantee Object. PECS falls right out of this.

Bounded Types & Generic Interfaces

public class SortedBox<T extends Comparable<T>> {   // T must be comparable to itself
    T max(T a, T b) { return a.compareTo(b) >= 0 ? a : b; }
}
// Comparable<T> and Comparator<T> are the canonical generic interfaces you'll implement:
users.sort(Comparator.comparing(User::score).reversed().thenComparing(User::name));

Interview Questions

Q1: What is type erasure and name three things it makes impossible.

The compiler removes generic type parameters after checking, replacing them with Object (or the bound) — so at runtime List<String> and List<Integer> are the same class, List. Impossible because of it: new T() and new T[] (the type isn't known at runtime), instanceof with a parameterized type (List<String>), overloading methods that differ only by generic parameter (same erased signature), and getting the actual type argument via reflection on a variable. Erasure exists for backward compatibility with pre-generics (raw) code.

Q2: Explain PECS with an example.

Producer Extends, Consumer Super. If a method READS T out of a collection (the collection produces values), use ? extends T — you can read as T but not add (exact subtype unknown). If it WRITES T in (consumes values), use ? super T — you can add T but reads only guarantee Object. Example: Collections.copy(List<? super T> dest, List<? extends T> src) — src is a producer (extends), dest is a consumer (super). It maximizes what callers can pass while staying type-safe.

Q3: Why can't you create a generic array (new T[])?

Arrays are covariant and reified (they know their element type at runtime and throw ArrayStoreException on bad writes), while generics are erased (no runtime type). A new T[] couldn't perform the runtime type check arrays require, so it's disallowed to prevent silent heap pollution. Workarounds: create Object[] and cast (with @SuppressWarnings, carefully), or use a List<T>, or pass a Class<T>/array factory. This covariance-vs-erasure mismatch is a favorite deep question.

🏋️ Practice: write a generic Box<T> and a generic method; write sum(List<? extends Number>) and prove you can't add to it; write fill(List<? super Integer>) and prove you can add; try new T[] and read the compile error; sort objects with Comparator.comparing(...).thenComparing(...).

🎯 Key Takeaways

  • Generics = compile-time type safety; always parameterize (raw types defeat the purpose).
  • Type erasure: generics vanish at runtime → no new T(), no new T[], no instanceof List<String>.
  • PECS: Producer Extends (read-only), Consumer Super (write) — the wildcard rule.
  • Bounded types (<T extends Comparable<T>>) constrain what T can be.
  • Comparable/Comparator are the generic interfaces you'll implement most.
