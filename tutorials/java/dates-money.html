Dates, Times & BigDecimal Core Libraries

What you'll master in this lesson: the modern java.time API (Java 8+) and why it replaced the broken old Date/Calendar; how to represent a date, a time, a moment (Instant), and a zoned time correctly; the timezone bugs that cause real outages; and — critically — why you must never use double for money and how BigDecimal fixes it. These are two of the most bug-prone areas in all of programming, and both have real financial consequences when done wrong.

Part 1 — Why the Old Date API Was Broken

// java.util.Date & Calendar (pre-Java 8) — a catalogue of design mistakes:
Date d = new Date(2024, 1, 15);   // ❌ year is 1900-based (→ year 3924!), month is 0-based (→ Feb!)
d.setHours(25);                    // ❌ MUTABLE — dates change under you, not thread-safe
// Calendar.MONTH is 0-based too (January = 0), a source of endless off-by-one bugs.
// SimpleDateFormat is NOT thread-safe — sharing one across threads corrupts output silently.

The original date/time classes were mutable, not thread-safe, riddled with 0-based/1900-based off-by-one traps, and mixed up the distinct concepts of "a date", "a time", and "a moment in time". Java 8's java.time package (JSR-310, inspired by Joda-Time) replaced them with a clean, immutable, well-separated API. Never use the old classes in new code.

💡 The single biggest improvement: java.time types are immutable (Immutability lesson), so they're thread-safe and free of the "spooky mutation" bugs of the old API. Every "modification" (plusDays, withMonth) returns a new object.

Part 3 — Timezones: Where the Real Bugs Live

// THE RULE: store and compute in UTC (Instant); convert to a zone only for DISPLAY.
Instant serverTime = Instant.now();                          // UTC — canonical, unambiguous
ZonedDateTime userView = serverTime.atZone(ZoneId.of("America/New_York"));  // convert for the user

// Same instant, different local times:
ZonedDateTime ny = serverTime.atZone(ZoneId.of("America/New_York"));
ZonedDateTime tokyo = serverTime.atZone(ZoneId.of("Asia/Tokyo"));   // 14 hours apart, same moment

// Use ZoneId names (region-based), NOT fixed offsets — regions handle Daylight Saving automatically:
ZoneId.of("America/New_York")    // ✅ shifts EST↔EDT correctly
ZoneId.of("-05:00")              // ❌ a fixed offset ignores DST — wrong half the year
🔍 The timezone golden rule: compute and persist in UTC, convert to local only at the edges (display/input). Storing local times without zones, or doing arithmetic across DST boundaries, causes classic bugs — duplicated or skipped hours during the DST switch, meetings off by an hour, "midnight" that isn't. Use region-based ZoneIds (which know DST rules) not fixed offsets. This "UTC everywhere internally" discipline prevents a whole category of production incidents.

Part 4 — Never Use double for Money

// THE CLASSIC BUG — binary floating point can't represent decimal fractions exactly:
System.out.println(0.1 + 0.2);        // 0.30000000000000004  — NOT 0.3!
double total = 0;
for (int i = 0; i < 10; i++) total += 0.1;
System.out.println(total);            // 0.9999999999999999  — NOT 1.0!
// Accumulate these errors over millions of transactions → real money lost, audits failed.

double and float are binary floating point: they represent numbers as sums of powers of two, and most decimal fractions (0.1, 0.2, 0.01) have no exact binary representation — just as 1/3 has no exact decimal representation. The tiny rounding errors accumulate, and for money that's unacceptable. The fix is BigDecimal, which does exact base-10 arithmetic.

Part 5 — BigDecimal Done Right

// ✅ Construct from a STRING (or int), NEVER from a double (which is already imprecise):
BigDecimal price = new BigDecimal("19.99");        // exact
BigDecimal bad   = new BigDecimal(19.99);          // ❌ captures the double's imprecision: 19.989999...
BigDecimal ok    = BigDecimal.valueOf(19.99);      // acceptable (uses Double.toString internally)

// Arithmetic returns NEW BigDecimals (immutable), and you MUST specify rounding for division:
BigDecimal sum = price.add(new BigDecimal("5.00"));
BigDecimal tax = price.multiply(new BigDecimal("0.08"));
BigDecimal share = total.divide(new BigDecimal("3"), 2, RoundingMode.HALF_UP);  // scale + rounding required

// Compare with compareTo, NOT equals (equals also compares SCALE: 1.0 != 1.00 by equals!):
new BigDecimal("1.0").equals(new BigDecimal("1.00"));       // false! (different scale)
new BigDecimal("1.0").compareTo(new BigDecimal("1.00")) == 0;  // true (value equality) ✅
🎯 Three BigDecimal rules that catch everyone: (1) construct from a String, not a double — new BigDecimal(0.1) stores the imprecise binary value, defeating the purpose. (2) divide can throw ArithmeticException for a non-terminating result (like 1/3) unless you supply a scale and RoundingMode — always specify them. (3) Use compareTo for value comparison, because equals also compares scale, so 1.0 and 1.00 are "unequal". A common alternative for money is storing integer cents/minor units in a long — exact and fast, converting to decimal only for display.

Part 6 — Deep Dive: Modeling Money Properly

  • A money amount is not just a number — it has a currency. Adding USD to EUR is nonsense; a proper Money type pairs an amount with a currency and rejects mismatched operations. Consider a dedicated type (or a library like Joda-Money / JavaMoney) rather than a bare BigDecimal.
  • Rounding is a business decision, made explicitly. HALF_UP, HALF_EVEN ("banker's rounding", which reduces bias over many operations), FLOOR — the choice affects real totals and must match the domain's rules and regulations. Never let rounding be implicit.
  • Scale (decimal places) matters. Most currencies use 2 decimal places, but some use 0 (JPY) or 3 (some dinar). Set and preserve scale deliberately.
  • The integer-cents alternative. Storing amounts as long minor units (cents) is exact, fast, and avoids BigDecimal's verbosity — at the cost of manual scaling and overflow awareness for very large sums. Many payment systems do this.
🔍 Senior framing: the two topics in this lesson share a theme — use the type that matches the domain's semantics, not the convenient primitive. A moment in time is not a long of "millis in local time"; it's an Instant. Money is not a double; it's exact decimal-or-integer arithmetic with a currency and an explicit rounding policy. The bugs from ignoring this — DST outages, financial discrepancies — are expensive and embarrassing, and interviewers probe them precisely because they reveal whether you think about correctness at the domain level. "Why not double for money?" and "how do you handle timezones?" are near-universal questions.

Interview Questions

Q1 (beginner): Why not use double for money?

Because double is binary floating point and most decimal fractions (0.1, 0.2, 0.01) can't be represented exactly in binary, so arithmetic accumulates tiny errors — famously 0.1 + 0.2 == 0.30000000000000004. Over many transactions these errors compound into real monetary discrepancies, failed reconciliations, and audit problems. Use BigDecimal, which performs exact base-10 arithmetic, or store integer minor units (cents) in a long. The rule is that money needs exact decimal arithmetic, which binary floating point fundamentally cannot provide.

Q2 (beginner): What replaced the old Date/Calendar API and why?

The java.time package (JSR-310, Java 8), inspired by Joda-Time. It replaced Date/Calendar/SimpleDateFormat because those were mutable (and thus not thread-safe), riddled with error-prone conventions (0-based months, 1900-based years), and conflated distinct concepts. java.time is immutable (thread-safe, no spooky mutation), uses natural 1-based months and full years, cleanly separates a date (LocalDate), a time (LocalTime), a moment (Instant), and a zoned time (ZonedDateTime), and provides a thread-safe DateTimeFormatter. New code should never use the legacy classes.

Q3 (mid): How do you handle timezones correctly?

Store and compute in UTC — typically as an Instant — and convert to a specific timezone only at the boundaries, for display or when parsing user input. This keeps all internal timestamps unambiguous and comparable. Use region-based zone identifiers (ZoneId.of("America/New_York")), which encode Daylight Saving rules, rather than fixed offsets (-05:00), which are wrong for half the year. Represent a zoned moment with ZonedDateTime when the zone matters, and be deliberate about DST edge cases (the switch creates a skipped hour in spring and a repeated hour in autumn). "UTC internally, local at the edges, region zones for DST" prevents most timezone bugs.

Q4 (mid): What's the difference between LocalDateTime, Instant, and ZonedDateTime?

LocalDateTime is a date-and-time with no timezone — "2024-01-15 14:30" as it would read on a wall clock somewhere, but you can't know the actual moment without a zone, so it's for zone-agnostic values (a store's opening time). Instant is a single point on the UTC timeline (essentially a precise timestamp), ideal for recording when something happened and for comparisons/durations — it's the canonical "moment". ZonedDateTime combines a date-time with a specific ZoneId, so it unambiguously identifies a moment and its local representation, including DST rules — use it when the timezone is part of the meaning (a meeting in New York). Rule of thumb: persist/compute with Instant; use LocalDate/LocalDateTime for zone-less values; use ZonedDateTime when you must present or reason about a specific zone.

Q5 (senior): What are the pitfalls of BigDecimal, and how do you use it correctly for money?

Several. (1) Constructing from a double — new BigDecimal(0.1) stores the double's imprecise binary value (0.1000000000000000055...), defeating the point; always construct from a String (or int), or use BigDecimal.valueOf. (2) Division — divide throws ArithmeticException for a non-terminating decimal (e.g. dividing by 3) unless you specify a scale and a RoundingMode; always provide them, and choose the rounding mode deliberately (HALF_EVEN/banker's rounding reduces cumulative bias). (3) Equality — equals compares scale as well as value, so 1.0 and 1.00 are unequal; use compareTo() == 0 for value comparison. (4) Scale management — set the appropriate scale for the currency (2 for most, 0 for JPY) and preserve it through operations. For robustness, model money as a dedicated type pairing a BigDecimal (or integer minor units) with a Currency so cross-currency operations are rejected, and make rounding an explicit business policy. Many systems instead store integer cents in a long for exactness and speed, converting to decimal only for display. The overarching point: money needs exact arithmetic, explicit rounding, currency awareness, and careful equality — none of which primitives give you.

Q6 (senior): A financial report's totals are off by a few cents intermittently. Walk through the likely causes.

Small, intermittent discrepancies in monetary totals point to precision and rounding issues. (1) Floating-point arithmetic somewhere in the pipeline — even one double used to sum or scale amounts introduces representation errors that accumulate differently depending on order and volume, producing off-by-a-cent drift; the fix is exact types (BigDecimal/integer cents) end to end, including in the database column type (use DECIMAL/NUMERIC, not FLOAT/DOUBLE). (2) Rounding applied inconsistently — rounding at different stages (per-line vs on the total), or with different modes, changes the sum; the classic is "round each line then sum" vs "sum then round", which legitimately differ, and the domain must specify which is correct. (3) Scale mismatches — mixing 2-, 3-, and unscaled values, or a currency with non-2 decimals, causes silent truncation. (4) Distribution/allocation remainders — splitting an amount three ways (e.g. $10.00/3) leaves a remainder cent that must be allocated deterministically (largest-remainder method), and naive rounding drops or double-counts it. (5) Equality/aggregation bugs from using BigDecimal.equals (scale sensitive) as a map key or in grouping. The diagnostic approach is to trace an affected total back through each computation, checking the type at every step (is any double involved?), the rounding points and modes, the scale, and how remainders are allocated — intermittency usually comes from data-dependent rounding boundaries. The correct architecture is exact types throughout, a single well-defined rounding policy applied at defined points, explicit remainder allocation, and value comparison via compareTo.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Prove the double bug. Print 0.1 + 0.2, then sum 0.1 ten times in a loop. Then redo both with BigDecimal constructed from Strings.
Solution

double: 0.1 + 0.2 → 0.30000000000000004; the loop → 0.9999999999999999. BigDecimal: new BigDecimal("0.1").add(new BigDecimal("0.2")) → exactly 0.3; summing new BigDecimal("0.1") ten times → exactly 1.0. String construction is essential — new BigDecimal(0.1) would reintroduce the error.

2 (easy): java.time basics. Compute the date 45 days from today, and someone's age in years given their birth date, using LocalDate and Period.
Solution

LocalDate.now().plusDays(45) (immutable — returns a new date). Period.between(birthDate, LocalDate.now()).getYears(). Note plusDays doesn't mutate now(); you use the returned value. No 0-based month or 1900 offset traps like the old API.

3 (medium): Timezone conversion. Given an Instant, display it in both "America/New_York" and "Asia/Tokyo". Confirm they're the same moment but different local times.
Solution

instant.atZone(ZoneId.of("America/New_York")) and instant.atZone(ZoneId.of("Asia/Tokyo")) — same underlying instant, local times ~13–14 hours apart. Both convert from the one canonical UTC moment. Using region ZoneIds means DST is handled automatically. This models the "store UTC, display local" rule.

4 (medium): Safe division. Split $100.00 three ways with BigDecimal. Show the naive version that throws, then fix it with scale + RoundingMode, and note the leftover cent.
Solution

new BigDecimal("100.00").divide(new BigDecimal("3")) throws ArithmeticException (non-terminating 33.333...). Fixed: .divide(new BigDecimal("3"), 2, RoundingMode.HALF_UP) → 33.33 each. But 3 × 33.33 = 99.99 — a leftover cent that a correct money split must allocate to one party (largest-remainder), not silently drop. This is the remainder-allocation subtlety from Q6.

5 (hard): Model a Money type. Sketch an immutable Money(BigDecimal amount, Currency currency) whose add rejects mismatched currencies and whose equality is value-based. Note the design choices.
Solution

A final class (or record) with a BigDecimal amount and a Currency currency; add(Money other) throws if !currency.equals(other.currency) then returns new Money(amount.add(other.amount), currency); use compareTo (not equals) for amount comparison, or normalize scale. Design choices: immutability (thread-safe, safe sharing), currency pairing (prevents nonsensical cross-currency math), explicit rounding policy on any division, and deliberate scale. This is why libraries like Joda-Money exist.

6 (hard): Explain-it test. Out loud: (a) why double fails for money and the two correct approaches; (b) the "UTC internally, local at edges" timezone rule; (c) the three BigDecimal gotchas (String construction, division rounding, equals-vs-compareTo).
Self-check rubric

Pass (a) if you said binary can't represent decimal fractions exactly → BigDecimal or integer cents. Pass (b) if you said store/compute UTC (Instant), convert to region zones for display, DST-aware. Pass (c) if you named constructing from String, requiring scale+RoundingMode on divide, and compareTo for value equality. 🎯 Key Takeaways

  • Use java.time (immutable, thread-safe); never the legacy Date/Calendar/SimpleDateFormat.
  • Pick the right type: LocalDate/LocalTime (no zone), Instant (a UTC moment), ZonedDateTime (zone-aware).
  • Timezones: store/compute in UTC, convert to local only for display; use region ZoneIds (DST-aware), not fixed offsets.
  • Never use double for money — binary can't represent decimals exactly; use BigDecimal or integer cents.
  • BigDecimal: construct from a String, always give divide a scale + RoundingMode, and compare with compareTo (not equals).
  • Model money as a type with a currency and an explicit rounding policy — match the type to the domain, not the convenient primitive.
