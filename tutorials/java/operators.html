Operators & Expressions Foundations

What you'll master in this lesson: every operator you'll actually use, the ones that cause subtle bugs (integer division, the difference between / and %, the remainder-of-negatives surprise), how type promotion silently changes your results, why && short-circuits and when that matters for correctness, and the operator precedence rules worth memorizing versus the ones worth parenthesizing around. Operators look trivial; the bugs they cause are not.

Part 1 — Arithmetic and the Integer Division Trap

int  sum  = 5 + 3;          // 8
int  diff = 5 - 3;          // 2
int  prod = 5 * 3;          // 15
int  quot = 7 / 2;          // 3  ← NOT 3.5! integer division TRUNCATES toward zero
int  rem  = 7 % 2;          // 1  ← modulo (remainder)
double d  = 7.0 / 2;        // 3.5 — one operand is double → double division
⚠️ Integer division truncates — the #1 arithmetic bug. 7 / 2 is 3, not 3.5, because both operands are int. To get 3.5, at least one operand must be floating point: 7.0 / 2 or (double) 7 / 2. The classic disaster: average = total / count where both are int silently drops the fractional part. And (a + b) / 2 for an int midpoint loses precision the same way.
// The negative-remainder surprise (interview favorite):
System.out.println(-7 % 3);    //  -1  (result takes the sign of the DIVIDEND, not the divisor)
System.out.println( 7 % -3);   //   1
// So testing "is n even?" with `n % 2 == 1` FAILS for negative odd numbers (-7 % 2 == -1).
// Use `n % 2 != 0` instead, or Math.floorMod(n, 2) which always returns a non-negative result.

Part 2 — Type Promotion (the silent conversion)

When operators combine mixed types, Java promotes the "smaller" type to the "larger" one before computing. This is usually helpful and occasionally a bug:

int    i = 5;
double x = i + 2.0;        // i promoted to double → 7.0
long   L = i * 1_000_000L; // i promoted to long → correct big result

// The trap: promotion happens AFTER an overflowing int operation, not before:
long bad  = 1_000_000 * 1_000_000;      // int*int OVERFLOWS first, THEN widens → wrong!
long good = 1_000_000L * 1_000_000;     // long arithmetic from the start → correct

// char is a number under the hood:
char c = 'A';
int code = c + 1;          // 66 — 'A' is 65; char promotes to int in arithmetic
char next = (char)(c + 1); // 'B' — cast back explicitly
ConversionAutomatic (widening)?Example
Smaller → larger (int→long→double)Yes, implicitdouble d = anInt;
Larger → smaller (double→int)No — needs an explicit cast (may lose data)int n = (int) 3.9; // 3

Part 3 — Comparison & the == vs equals Divide

int a = 5, b = 5;
a == b;                     // true — for PRIMITIVES, == compares values ✅

String s1 = new String("hi"), s2 = new String("hi");
s1 == s2;                   // false! — for OBJECTS, == compares REFERENCES
s1.equals(s2);              // true — .equals compares CONTENT ✅

// Relational operators return boolean:
5 < 3;   // false        5 >= 5;  // true        5 != 3;  // true
🎯 The rule from the Types lesson, restated: == on primitives compares values; == on objects compares identity (same object?). For value comparison of objects — Strings, wrappers, your own classes — use .equals(). This single confusion is behind a huge fraction of beginner bugs and at least one interview question in every Java screen.

Part 4 — Logical Operators & Short-Circuiting

boolean r1 = (a > 0) && (b > 0);   // AND — true only if BOTH
boolean r2 = (a > 0) || (b > 0);   // OR  — true if EITHER
boolean r3 = !(a > 0);             // NOT — inverts

// SHORT-CIRCUIT: && stops at the first false; || stops at the first true.
// This is not just an optimization — it's a correctness tool:
if (list != null && !list.isEmpty()) { ... }   // if list is null, isEmpty() is NEVER called → no NPE
if (s == null || s.isBlank())         { ... }   // if s is null, isBlank() is skipped → no NPE

// The order matters! Reversing it crashes:
if (!list.isEmpty() && list != null) { ... }   // ❌ NPE if list is null (isEmpty runs first)
🔍 Short-circuiting is a guard, not a micro-optimization. Put the null/cheap/guard check first so the expensive or unsafe check only runs when it's safe. The single-character operators & and | exist too but do not short-circuit (they're bitwise on ints, non-short-circuit boolean on booleans) — you almost always want &&/||.

Part 5 — Assignment, Increment & the Ternary

int n = 10;
n += 5;    // n = n + 5  → 15   (also -= *= /= %= and bitwise variants)
n++;       // POST-increment: use n's value THEN add 1
++n;       // PRE-increment: add 1 THEN use n's value

int i = 5;
int j = i++;   // j = 5, i = 6   (post: assigned the OLD value)
int k = ++i;   // k = 7, i = 7   (pre: assigned the NEW value)

// The ternary operator — a compact if/else that produces a VALUE:
int max = (a > b) ? a : b;                 // "if a>b then a else b"
String label = (n == 1) ? "item" : "items"; // great for choosing a value inline
💡 Use the ternary for choosing between two values; use a full if/else for choosing between two actions. Nesting ternaries more than one level deep is a readability crime — reach for if/else or a switch instead.

Part 6 — Deep Dive: Operator Precedence (and when to stop memorizing)

Java evaluates operators in a fixed precedence order. The high-value ones to know:

postfix    n++  n--
unary      ++n  --n  !  (cast)  -n
multiplicative   *  /  %
additive         +  -
relational       <  >  <=  >=  instanceof
equality         ==  !=
logical AND      &&
logical OR       ||
ternary          ? :
assignment       =  +=  -=  ...

So 2 + 3 * 4 is 14 (* before +), and a && b || c means (a && b) || c. But the professional habit is not to memorize the whole table — it's to parenthesize anything non-obvious. Code is read far more than written; (a && b) || c costs two characters and removes all doubt. Reserve precedence knowledge for reading others' code; write with parentheses.

🔍 Senior nugget — the + ambiguity: + is both numeric addition and String concatenation, resolved left-to-right by type. 1 + 2 + "x" is "3x" (numbers add, then concatenate), but "x" + 1 + 2 is "x12" (first + makes it a String, so the rest concatenates). This exact ambiguity is a favorite interview trick and a real logging bug ("count: " + a + b concatenates instead of summing — wrap the sum in parentheses).

Interview Questions

Q1 (beginner): What does 7 / 2 evaluate to and why?

3, not 3.5. Both operands are int, so Java does integer division, which truncates the fractional part toward zero. To get 3.5 you need floating point: make one operand a double (7.0 / 2 or (double) 7 / 2). The trap in real code is average = sum / count with both int, which silently drops precision.

Q2 (beginner): What's the difference between i++ and ++i?

Both increment i by 1; they differ in the value the expression yields. i++ (post-increment) evaluates to i's old value, then increments; ++i (pre-increment) increments first, then evaluates to the new value. So int j = i++ gives j the old value while int k = ++i gives the new one. When the result isn't used (a bare i++; statement or a for-loop step) they're interchangeable.

Q3 (mid): Explain short-circuit evaluation and give a case where it's required for correctness, not just speed.

&& stops evaluating at the first false and || at the first true, so the right operand may never run. This is a correctness tool for guarding: if (obj != null && obj.isValid()) — if obj is null the left side is false and isValid() is never called, avoiding a NullPointerException. Reversing the order (obj.isValid() && obj != null) crashes. The non-short-circuit operators &/| would evaluate both sides and NPE, which is why they're the wrong choice for guarded logic.

Q4 (mid): Why does long x = 1_000_000 * 1_000_000 give the wrong answer?

The multiplication is int * int, computed entirely in 32-bit int arithmetic, which overflows (the true product exceeds Integer.MAX_VALUE) and wraps to a garbage value. Only after that broken result is produced is it widened to long for the assignment — too late. The fix is to make the arithmetic happen in long from the start by making an operand long: 1_000_000L * 1_000_000. Widening applies to operands before an operation only when types are mixed within that operation; assignment-target width doesn't retroactively change how the operation was computed.

Q5 (senior): Predict System.out.println(1 + 2 + "3" + 4 + 5) and explain the rule.

"3345". + is left-associative and its meaning depends on operand types evaluated left to right: 1 + 2 are both numbers → 3; 3 + "3" has a String operand → concatenation → "33"; from then on every operand is appended as text → "333", "3334"… giving "33345". (Recompute carefully: 1+2=3, +"3"="33", +4="334", +5="3345".) The rule: once a String appears, the rest of the left-to-right chain is concatenation. This is why "sum=" + a + b is a bug for numeric sums — you must parenthesize (a + b) to force the addition first.

Q6 (senior): -7 % 3 is -1, but you need a value in [0, 3) for a hash bucket. What's wrong and what's the fix?

Java's % is a remainder, not a mathematical modulo: the result takes the sign of the dividend, so -7 % 3 == -1. Using that directly as an array index (arr[key.hashCode() % arr.length]) throws ArrayIndexOutOfBoundsException when the hashCode is negative — a real, subtle bug in hand-rolled hash tables. The fix is Math.floorMod(key.hashCode(), arr.length), which implements true floored modulo and always returns a value in [0, length) for a positive divisor. (Bit-masking hash & (length - 1) also works but only when length is a power of two — which is exactly why HashMap sizes its table to powers of two.)

🏋️ Exercises (attempt before opening solutions)

1 (easy): Average done right. Given int total = 7, count = 2;, compute the average as a double that prints 3.5. Show the wrong version too and explain the difference.
Solution

Wrong: double avg = total / count; → 3.0 (integer division happens first, then the 3 is widened). Right: double avg = (double) total / count; → 3.5 (the cast forces double division). Casting one operand — not the result — is the fix.

2 (easy): Even/odd for negatives. Write isOdd(int n) that works for negative numbers. Test it on -7, -4, 3.
Solution

return n % 2 != 0; (not == 1, which fails for -7 since -7 % 2 == -1). Math.floorMod(n, 2) == 1 also works. The takeaway: never compare % results against a specific sign-bearing value unless you've accounted for negatives.

3 (medium): Short-circuit guard. Given a String s that may be null, write a single boolean expression that is true when s is a non-empty string, without risking an NPE. Then write the buggy reversed version and explain why it crashes.
Solution

Safe: s != null && !s.isEmpty() — if s is null the left side is false and isEmpty() is never evaluated. Buggy: !s.isEmpty() && s != null — isEmpty() runs first on a null reference → NPE. Short-circuit order is the whole point.

4 (medium): Post vs pre increment. Predict the final values: int i = 0; int a = i++; int b = ++i; int c = i++ + ++i;. Then run it.
Solution

a = 0 (post: old value, then i→1). b = 2 (pre: i→2, then read). For c: i++ reads 2 (then i→3), ++i makes i→4 and reads 4, so c = 2 + 4 = 6, final i = 4. Mixing pre/post in one expression is legal but confusing — a reason not to write code like c in practice.

5 (hard): The concatenation bug. You log System.out.println("Score: " + score1 + score2); where both are ints, expecting their sum. It prints them stuck together. Explain and fix, then show a case where the numbers happen to add first.
Solution

Left-to-right: "Score: " + score1 is already a String, so + score2 concatenates. Fix: "Score: " + (score1 + score2) — parentheses force the numeric addition before it meets the String. Contrast score1 + score2 + " total", where the two ints add first (both numeric) and only then the String is appended. Position relative to the first String decides everything.

6 (hard): Explain-it test. Out loud: (a) why double d = 5/2 is 2.0 and how to fix it; (b) the short-circuit rule and a guard that depends on it; (c) the + operator's type-dependent behavior with an example.
Self-check rubric

Pass (a) if you said 5/2 computes in int (=2) before widening, fixed by casting an operand. Pass (b) if you gave the null-guard-first example and noted the RHS may not run. Pass (c) if you showed a case flipping between sum and concatenation based on a String's position. These recur in the Strings, Collections, and Streams lessons.

🎯 Key Takeaways

  • Integer division truncates (7/2==3); make an operand double for decimals; force long early to avoid overflow-before-widening.
  • % is a remainder that takes the dividend's sign — use Math.floorMod for a non-negative modulo (e.g. hash buckets).
  • == compares values for primitives but identity for objects — use .equals() for object/String value comparison.
  • &&/|| short-circuit — put null/guard checks first so unsafe checks only run when safe.
  • + means add or concatenate depending on operand types left-to-right; parenthesize sums that sit after a String.
  • Know precedence enough to read code; parenthesize anything non-obvious when writing it.
