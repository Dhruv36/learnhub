The Module System (JPMS) Modern Java

What you'll master in this lesson: what the Java Platform Module System (Java 9+) actually solved — strong encapsulation and reliable configuration beyond what public and the classpath ever gave you — how a module-info.java declares requires/exports, why "public" no longer means "accessible to everyone," and the honest senior take: most applications don't modularize, and knowing why is as important as knowing how. You'll also see how modules enable jlink custom runtimes and why the classpath's "JAR hell" motivated all of this. Prerequisites: JVM & Class Loading, Interfaces, Build Tools.

Part 1 — The Problem: The Classpath and "JAR Hell"

Classpath weaknessWhat goes wrong
No strong encapsulationAny public class is reachable — internal packages leak and get depended on
No reliable configurationA missing JAR isn't detected until a NoClassDefFoundError at runtime
Flat, unorderedTwo versions of a library → one silently shadows the other (JAR hell)
Split packagesThe same package spread across JARs causes ambiguous, order-dependent loading
🎯 JPMS exists to fix two things the classpath never could: strong encapsulation and reliable configuration. On the classpath, everything is a flat bag of classes: any public type in any JAR is reachable by anything, so a library's internal classes leak and people depend on them; and there's no declaration of what depends on what, so a missing JAR only surfaces as a NoClassDefFoundError deep into a run rather than at startup. Add duplicate/conflicting versions silently shadowing each other ("JAR hell") and split packages, and large systems become fragile. Modules introduce an explicit layer above packages that names dependencies and controls access — turning runtime surprises into declared, checkable structure.

Part 2 — module-info.java: requires & exports

// src/com.myapp/module-info.java — placed at the module's source root:
module com.myapp {
    requires com.acme.util;          // I DEPEND on this module (must be present or startup fails)
    requires java.sql;               // even the JDK is modular now (java.base is implicit)

    exports com.myapp.api;           // this package is PUBLIC to other modules
    exports com.myapp.spi to com.myapp.plugins;   // QUALIFIED export — only to named modules
    // packages NOT exported are strongly encapsulated — invisible OUTSIDE the module, even if public
}
🔑 A module declares its dependencies (requires) and its public API (exports) — and everything else is hidden, even public classes. This is the headline change: on the classpath, public meant "accessible to all"; in a module, a public class in a non-exported package is invisible to other modules. So encapsulation becomes real at the package/module boundary, not just the class. requires makes dependencies explicit and checked at startup (a missing required module fails fast, not mid-run). A qualified export (exports ... to) opens a package only to specific named modules — useful for an SPI you want your own plugins to see but not the world.

Part 3 — Access, Reflection & opens

// Reflection is BLOCKED by default across module boundaries — even on exported packages,
// deep reflection (setAccessible on private members) needs explicit opening:
module com.myapp {
    opens com.myapp.model;                    // allow RUNTIME reflection into this package (e.g. for Jackson/JPA)
    opens com.myapp.model to com.fasterxml.jackson.databind;   // qualified: only to Jackson
}
// open module com.myapp { ... }              // opens EVERY package for reflection (escape hatch)

// exports  = compile-time + runtime access to public types
// opens    = runtime reflective access (incl. private members via setAccessible)
🔍 exports and opens are different, and the distinction trips everyone. exports grants normal compile-time and runtime access to the public types of a package. opens grants reflective access at runtime — including deep reflection (setAccessible(true) on private fields) — which frameworks like Jackson, Hibernate/JPA, and Spring rely on to read your objects' fields (Annotations & Reflection lesson). Because the module system blocks deep reflection by default (that's the "strong" in strong encapsulation), a very common migration error is a framework failing with "cannot access a member of class ... with modifiers private" until you opens the relevant package (ideally a qualified open to just that framework's module). This is the concrete point where JPMS and the reflective-framework ecosystem collide.

Part 4 — The Modular JDK & jlink

// The JDK itself was split into ~70 modules (java.base, java.sql, java.xml, ...). This enables:

// jlink — build a CUSTOM runtime containing ONLY the modules your app uses:
jlink --add-modules com.myapp --output myapp-runtime
// → a self-contained ~40MB runtime instead of a full ~300MB JDK — great for containers/serverless.

// jdeps — analyze what your code actually depends on (and flags use of internal JDK APIs):
jdeps --list-deps myapp.jar
📦 Modularizing the JDK enabled jlink: a stripped-down runtime with only the modules you need. Because the platform is now a graph of modules with declared dependencies, the tooling can compute the transitive closure your app requires and assemble a minimal custom runtime image — dramatically smaller than shipping a whole JDK, which matters for container images and cold-start times. This is one of the clearest practical payoffs of JPMS even for teams that don't modularize their own code: you can still jlink against the modular JDK. jdeps helps you understand and clean up dependencies (and warns when you're reaching into encapsulated internal JDK APIs like the old sun.misc.Unsafe).

Part 5 — Should You Modularize? The Honest Answer

// Reality: the vast majority of applications run on the CLASSPATH, not the module path — and that's fine.
//   • Libraries benefit most (clear API surface, strong encapsulation for consumers).
//   • Applications often see little benefit and real migration cost (opens/exports churn, tooling).
//   • Spring Boot apps are typically NOT modularized — the fat-jar + classpath model dominates.

// The "automatic module" bridge: a plain JAR on the module path becomes an automatic module
// (name derived from the JAR / Automatic-Module-Name manifest entry) so migration can be gradual.
⚖️ Knowing when not to modularize is the senior point. JPMS is genuinely valuable for the JDK and for libraries (a clean, enforced public API and hidden internals benefit every consumer), but for most applications the cost — retrofitting module-info.java, chasing opens/exports for every reflective framework, tooling and third-party JARs that aren't modularized — often outweighs the benefit, and the ecosystem reflects this: most apps, including typical Spring Boot services, still run on the classpath with the fat-jar model, and that's a perfectly legitimate choice. Automatic modules (a plain JAR treated as a module with an inferred name) exist precisely to allow gradual, partial adoption. Understanding modules deeply and being able to say "we don't need them here" is more credible than reflexively modularizing everything.

Part 6 — Deep Dive: Encapsulation as a First-Class Concern

  • JPMS makes architecture enforceable. Before modules, "don't depend on our internals" was a convention (and a wiki page); modules make it a compile-time error. That's real architectural governance, not documentation.
  • "public" is no longer the whole story. Accessibility now depends on both the access modifier and whether the package is exported/opened — a genuine shift in the mental model of Java visibility.
  • Reflection is the friction point. The entire framework ecosystem runs on reflection, and strong encapsulation blocks it by default — so opens (qualified where possible) is the bridge, and migration pain is almost always reflective-access pain.
  • The value is asymmetric: libraries yes, most apps no. Match the tool to the case — modular JDK and modular libraries deliver clear wins; application modularization is optional and often skipped for good reasons.
🔍 Senior framing: the module system is best understood as Java elevating encapsulation and dependency structure from convention to enforced language feature, motivated by two real classpath failures — the lack of strong encapsulation (any public type anywhere was reachable, so internal APIs leaked and calcified into de-facto public contracts) and the lack of reliable configuration (missing or conflicting JARs surfaced as runtime NoClassDefFoundErrors and silent version shadowing rather than declared, checkable dependencies). A module-info.java answers both by declaring what a module requires (explicit, startup-checked dependencies) and exports (its deliberate public API), with everything else strongly encapsulated — which is why the headline conceptual shift is that public no longer means universally accessible: a public type in a non-exported package is invisible across the module boundary, so Java visibility now depends on modifier and module declaration together. The distinction between exports (compile/runtime access to public types) and opens (runtime reflective access, including deep reflection into private members) is where theory meets the real ecosystem: because frameworks like Jackson, Hibernate, and Spring depend on reflecting over your objects (Annotations & Reflection lesson) and JPMS blocks deep reflection by default, nearly all migration pain is reflective-access pain, resolved by qualified opens to exactly the frameworks that need it. Modularizing the JDK into ~70 modules delivered a concrete win independent of whether you modularize your own code — jlink can assemble a minimal custom runtime containing only the modules your app transitively needs, shrinking images and cold-starts, which matters for containers and serverless. But the mature judgment is about applicability: JPMS's benefits are asymmetric — libraries gain a clean enforced API and hidden internals that help every consumer, while most applications (including typical Spring Boot services on the fat-jar/classpath model) see modest benefit against real retrofit and tooling cost, so choosing to stay on the classpath is legitimate and common, with automatic modules available for gradual adoption where needed. The engineer who can explain what problems JPMS solves, how requires/ exports/opens and the redefinition of accessibility work, why reflective frameworks force opens, and — crucially — when modularization isn't worth it, demonstrates exactly the blend of mechanism knowledge and cost-aware judgment that distinguishes someone who understands the platform from someone who has only read the feature list.

Interview Questions

Q1 (beginner): What problems does the Java module system solve?

Two that the classpath couldn't. First, strong encapsulation: on the classpath every public type in every JAR is reachable, so a library's internal classes leak and consumers end up depending on them, making internals impossible to change safely; modules let a library expose only chosen packages and hide the rest, even if those hidden classes are public. Second, reliable configuration: the classpath has no declaration of dependencies, so a missing or wrong JAR isn't detected until a NoClassDefFoundError occurs at runtime, possibly far into execution; modules declare their dependencies explicitly with requires, which are checked at startup so problems fail fast. Modules also address JAR hell (silent version shadowing) and split packages. In short, JPMS turns implicit, runtime-discovered structure into explicit, checkable structure — better encapsulation and better configuration reliability for large systems.

Q2 (beginner): In a module, does public still mean accessible everywhere?

No — that's the key conceptual change. In the classpath world, public meant "accessible to any code anywhere." In a module, a public class is accessible to other modules only if its package is exported; a public class in a non-exported package is strongly encapsulated and invisible outside its own module. So accessibility now depends on two things together: the access modifier (public/protected/…) and whether the enclosing package is exported (or opened, for reflection). This lets a module have plenty of public classes for its own internal use that remain hidden from consumers, which is exactly the strong encapsulation the classpath lacked. It's a genuine shift in the mental model of Java visibility that surprises people migrating code.

Q3 (mid): What's the difference between exports and opens?

exports grants normal access — compile-time and runtime — to the public types of a package, so other modules can reference and call them like any API. opens grants reflective access at runtime, including deep reflection (using setAccessible(true) to read/write private members), which normal exports does not permit. The distinction matters enormously in practice because the framework ecosystem — Jackson serializing your objects, Hibernate/JPA mapping entity fields, Spring injecting and inspecting beans — relies on deep reflection over your classes, and the module system blocks that by default as part of strong encapsulation. So a package can be exported (its API is usable) yet still reject reflection until you also opens it. The common failure is a framework throwing "cannot access a member ... with modifiers private" until you add opens com.myapp.model to com.fasterxml.jackson.databind; (a qualified open, exposing it reflectively only to the framework that needs it rather than to everyone). Summarized: exports = "you may use my public API," opens = "you may reflect into my package's internals at runtime."

Q4 (mid): What is jlink and why did the modular JDK enable it?

jlink is a tool that assembles a custom, minimal Java runtime image containing only the modules your application actually needs, rather than a full JDK/JRE. It was enabled by the modularization of the JDK itself into ~70 modules (java.base, java.sql, java.xml, and so on) with declared dependencies: because the platform is now a graph of modules, the tooling can compute the transitive closure of modules your app requires and package just those into a self-contained runtime. The payoff is a much smaller image — on the order of tens of MB instead of a few hundred — which is valuable for container images (smaller layers, faster pulls), serverless/cold-start scenarios, and constrained environments, and it also means the runtime carries less unused surface. This is a concrete benefit of JPMS that even teams who don't modularize their own application can enjoy, since they can still jlink against the modular JDK (often via a plugin in their build tool). It's one of the most tangible practical wins of the module system.

Q5 (senior): Should a typical application modularize? Make the case both ways.

Usually not, and being able to justify that is the point. For modularizing: it enforces architecture at compile time — module boundaries make "don't depend on our internals" a compile error rather than a convention, which is real governance for a large codebase with many teams; it gives reliable startup-time dependency checking; and it enables clean API surfaces. Against, for most apps: the benefits accrue far more to libraries (whose consumers gain from a clean, enforced public API and hidden internals) than to an application that is the top of its own dependency graph and has no external consumers to encapsulate against; meanwhile the costs are real — you must write and maintain module-info.java, chase opens/exports for every reflective framework (Jackson, JPA, Spring, test frameworks), and cope with third-party JARs that aren't modularized (relying on automatic modules with inferred names that can be unstable). The ecosystem has voted with its feet: the overwhelming majority of applications, including typical Spring Boot services, run on the classpath with the fat-jar model, and that is a legitimate, well-supported choice. The senior position is therefore case-based: modularize libraries (and enjoy the modular JDK + jlink regardless), but for most applications stay on the classpath unless there's a concrete need — a large multi-team codebase wanting enforced boundaries, or a requirement for a minimal jlinked runtime that pushes you toward it. Reflexively modularizing every app is a classic case of adopting a feature for its own sake and paying migration cost for little return; recognizing the asymmetric value (libraries yes, most apps no) and articulating when the trade-off flips is exactly the judgment that signals seniority.

Q6 (senior): How do modules interact with the reflection-heavy framework ecosystem, and what does that imply for migration?

They interact through friction, and understanding it explains almost all real migration pain. The framework ecosystem — dependency injection, ORMs, serializers, test runners — is built on reflection, frequently deep reflection that reads and writes private fields (Annotations & Reflection lesson): Jackson populates private fields from JSON, Hibernate reads/writes entity state, Spring inspects and injects beans. The module system's core promise is strong encapsulation, so it blocks deep reflection across module boundaries by default, even into exported packages. The consequence is that when you modularize an application, code that compiled and ran fine on the classpath starts failing at runtime with access errors (InaccessibleObjectException, "cannot access a member ... with modifiers private") the moment a framework tries to reflect into your model classes. The remedy is opens — granting reflective access to the relevant packages — and the best practice is a qualified open (opens com.myapp.model to com.fasterxml.jackson.databind) that exposes internals only to the specific framework module that needs them, preserving encapsulation against everyone else; the blunt escape hatch is an open module that opens everything, sacrificing the very benefit you modularized for. The migration implications are: expect the work to be dominated by mapping which frameworks reflect into which packages and adding the right (qualified) opens; expect trouble from third-party JARs that aren't modules (automatic modules bridge this but with caveats); and weigh whether the encapsulation gain justifies the churn given that you're re-opening much of what you just closed. This tension — strong encapsulation versus a runtime built on reflection — is the defining practical reality of JPMS adoption, and it's a large part of why application modularization is often judged not worth it while the modular JDK (which app code consumes without deep-reflecting into it) is universally beneficial.

Solution

Solution

Modularize only for a concrete need (enforced multi-team boundaries, a minimal jlinked runtime). You can still jlink against the modular JDK regardless.

🎯 Key Takeaways

  • JPMS solves two classpath failures: strong encapsulation and reliable configuration.
  • module-info.java declares requires (checked dependencies) and exports (public API); everything else is hidden.
  • public ≠ accessible — a public type in a non-exported package is invisible across modules.
  • exports = compile/runtime access to public types; opens = runtime reflective access (what Jackson/JPA/Spring need).
  • The modular JDK enables jlink — minimal custom runtimes for containers/serverless.
  • Modularize libraries; most apps (incl. Spring Boot) stay on the classpath — knowing when not to is the senior call.
