Pattern Matching & Switch Modern Java

What you'll master in this lesson: pattern matching for instanceof (16+) that checks-and-binds in one step; the modern switch expression and pattern-matching switch (21+) with type patterns, guards, and exhaustiveness; record deconstruction patterns that destructure data; and how these combine with sealed types to replace fragile instanceof-chains and visitor boilerplate with clear, compiler-checked code. Pattern matching is Java's newest major feature and a fast-rising interview topic — this lesson shows why it changes how you write conditional logic.

Part 1 — Pattern Matching for instanceof

// BEFORE — the redundant "check, then cast" dance:
if (obj instanceof String) {
    String s = (String) obj;          // cast again — repetitive and error-prone
    System.out.println(s.length());
}

// AFTER (16+) — the type pattern checks AND binds in one step:
if (obj instanceof String s) {         // if obj is a String, bind it to 's'
    System.out.println(s.length());    // use 's' directly — no cast
}

// The binding is scoped intelligently (flow typing):
if (!(obj instanceof String s)) return;   // 's' is in scope AFTER this line (obj must be a String)
System.out.println(s.length());

// Combine with conditions — 's' is available in the same expression:
if (obj instanceof String s && s.length() > 5) { ... }
💡 The type pattern eliminates the cast that always followed an instanceof. Beyond less code, it removes a real bug source: the cast and the check could drift apart (check String, cast to Integer). Binding them makes that impossible. The compiler's "flow typing" also knows where the binding is valid, including after an early return — a small but genuinely useful ergonomic win.

Part 2 — The Switch Expression (recap + expansion)

// The arrow switch is an EXPRESSION that returns a value (Control Flow lesson) — no fall-through:
int numLetters = switch (day) {
    case MONDAY, FRIDAY, SUNDAY -> 6;
    case TUESDAY                -> 7;
    case THURSDAY, SATURDAY     -> 8;
    case WEDNESDAY              -> 9;
};   // exhaustive over the enum → no default needed; compiler checks all cases covered

// Block arms use yield to produce the value:
String size = switch (n) {
    case 1 -> "one";
    default -> { String r = compute(n); yield r; }
};

Part 3 — Pattern Matching for switch (Java 21)

// Switch on the TYPE of a value, binding each case — replaces long instanceof-chains:
String format(Object obj) {
    return switch (obj) {
        case Integer i -> "int: " + i;
        case Long l    -> "long: " + l;
        case String s  -> "string of length " + s.length();
        case null      -> "null";                    // switch can now handle null explicitly
        default        -> "unknown: " + obj;
    };
}

// GUARDED patterns — add a condition with 'when':
String categorize(Object obj) {
    return switch (obj) {
        case Integer i when i < 0  -> "negative";
        case Integer i when i == 0 -> "zero";
        case Integer i             -> "positive";
        default                    -> "not an int";
    };
}
🎯 Pattern-matching switch replaces the classic if/else instanceof ladder — the verbose, error-prone chain of "if it's this type do X, else if that type do Y". It's more readable, and over a sealed type it's exhaustive: cover every permitted subtype and you need no default; miss one and it won't compile. The when guard adds conditions, and switch can now match null as an explicit case rather than throwing.

Part 4 — Record Deconstruction Patterns

record Point(int x, int y) { }
record Line(Point start, Point end) { }

// Destructure a record directly in the pattern — bind its components:
static String describe(Object obj) {
    return switch (obj) {
        case Point(int x, int y)        -> "point at " + x + "," + y;   // x, y bound from the record
        case Line(Point s, Point e)     -> "line from " + s + " to " + e;
        default                          -> "other";
    };
}

// NESTED deconstruction — reach into nested records in one pattern:
case Line(Point(var x1, var y1), Point(var x2, var y2)) ->
    "line (" + x1 + "," + y1 + ") → (" + x2 + "," + y2 + ")";   // all four bound at once
🔍 Record patterns destructure data in the match itself. Instead of matching a Point and then calling p.x() and p.y(), you bind x and y directly in the pattern — and you can nest to pull apart nested records in one expression. Combined with sealed types and records, this is full algebraic-data-type pattern matching (Records & Sealed lesson): the exact style of concise, exhaustive data processing found in Rust/Scala/Haskell, now in Java.

Part 5 — Deep Dive: Replacing the Visitor Pattern

Before pattern matching, processing a closed hierarchy of types cleanly required the Visitor pattern — a notoriously verbose piece of OOP boilerplate (an accept method on every type, a visitor interface with a method per type, double dispatch). Sealed types + pattern-matching switch collapse it:

// The Visitor pattern (old way): ~one interface + accept() per type + a visitor impl.
// Dozens of lines, and adding a type touches every visitor.

// Pattern matching (new way) — the whole "operation over the hierarchy" in one method:
sealed interface Expr permits Num, Add, Mul { }
record Num(double v) implements Expr { }
record Add(Expr l, Expr r) implements Expr { }
record Mul(Expr l, Expr r) implements Expr { }

static double eval(Expr e) {
    return switch (e) {
        case Num(double v)     -> v;
        case Add(Expr l, Expr r) -> eval(l) + eval(r);   // recurse, destructuring as we go
        case Mul(Expr l, Expr r) -> eval(l) * eval(r);
    };   // exhaustive over the sealed Expr — no default, compiler-verified
}
🔍 Senior framing: the trade-off between the two styles is the classic "expression problem." The Visitor pattern makes adding a new operation easy (write one visitor) but adding a new type hard (touch every visitor). Pattern matching over a sealed type flips it: adding an operation is a new method, and adding a type makes every exhaustive switch fail to compile until updated — the compiler points you at every place that needs handling, which for a closed, stable hierarchy (an AST, a set of events, a state machine) is exactly what you want. Pattern matching isn't just terser Visitor; it moves "did I handle the new case everywhere?" from a manual audit to a compile error. This is why interpreters, compilers, and domain state machines are the poster-child use cases, and why the feature was co-designed with records and sealed types.

Interview Questions

Q1 (beginner): What does pattern matching for instanceof improve?

It combines the type check and the cast into one step with a bound variable: if (obj instanceof String s) tests whether obj is a String and, if so, binds it to s, which you use directly without a separate cast. This removes the repetitive "check then cast" boilerplate and a real bug source (the cast target drifting from the checked type). The binding is flow-scoped — available wherever the compiler can prove the check succeeded, including after an early return — and it can be combined with further conditions in the same expression.

Q2 (beginner): What is a pattern-matching switch?

A switch (Java 21) whose cases match on the type (and shape) of the selector and bind a variable, rather than only on constant values. For example case Integer i -> ..., case String s -> ... dispatch on the runtime type and bind i/s. It can also handle case null explicitly, add when guards for conditions, and deconstruct records. It replaces long if/else instanceof chains with a concise expression, and over a sealed type it's checked for exhaustiveness so no default is required.

Q3 (mid): What is a record deconstruction pattern and why is it useful?

A record pattern destructures a record in the match, binding its components directly: case Point(int x, int y) -> ... matches a Point and binds x and y in one step, instead of matching the Point and then calling p.x()/p.y(). Patterns nest, so case Line(Point(var x1, var y1), Point(var x2, var y2)) pulls apart a nested structure and binds all four coordinates at once. It's useful because it makes conditional logic over data read as a description of the data's shape, reduces accessor noise, and — with sealed types — gives concise, exhaustive processing of algebraic data types (the same style as Rust/Scala).

Q4 (mid): How do guards (when) work and why does case ordering matter?

A guard adds a boolean condition to a pattern case: case Integer i when i < 0 -> ... matches only when obj is an Integer and the guard is true. Ordering matters because switch evaluates cases top to bottom and takes the first match: a more general case (e.g. an unguarded case Integer i) placed before a more specific guarded one (case Integer i when i == 0) would shadow it, so the specific/guarded cases must come first. The compiler catches some dominance errors (a case that can never be reached), but you should order from most specific to most general deliberately. Guards let you keep type-based dispatch and value-based conditions in one clean structure instead of nesting ifs inside cases.

Q5 (senior): How do sealed types + pattern matching relate to the Visitor pattern and the expression problem?

Both solve "perform operations over a hierarchy of types," and the choice is governed by the expression problem — the tension between easily adding new types vs new operations. The Visitor pattern makes adding an operation easy (implement one new visitor with a method per type) but adding a type hard and invasive (every existing visitor must gain a method), at the cost of substantial boilerplate: an accept method on each type, a visitor interface, and double dispatch. Sealed types + pattern-matching switch flip the trade-off: adding an operation is just a new method with a switch, and adding a type is easy to introduce but causes every exhaustive switch to fail compilation until it's handled — which, for a closed and relatively stable hierarchy (an AST, a fixed set of events, a state machine), is precisely the desired behavior because the compiler enumerates every site needing an update rather than letting you silently miss one. So pattern matching isn't merely a terser Visitor; it changes the safety model, converting "did I handle the new variant everywhere?" from a manual audit into compile errors. Visitor still wins when the type set is genuinely open/extensible by third parties (you can't seal it) or when you want to add operations without recompiling the hierarchy; for a closed domain you own, pattern matching over a sealed type is clearer and safer.

Q6 (senior): Why is omitting default in an exhaustive sealed switch a feature, not a risk?

Because it turns hierarchy evolution into a compile-time checklist. When you switch over a sealed type and cover every permitted subtype, the compiler proves exhaustiveness and lets you omit default. The payoff comes later: if someone adds a new permitted subtype to the sealed interface, every exhaustive switch that lacked a default now fails to compile, pointing precisely at each place that must handle the new case. Adding a default "just in case" defeats this — it silently absorbs the new subtype (often doing the wrong thing, like a generic fallback or an ignored case), so the bug surfaces at runtime instead of build time. The discipline is to omit default for exhaustive sealed switches so the type system enforces that all consumers stay in sync with the model, and to use default only when the input is genuinely open-ended (e.g. switching over arbitrary Object) where a catch-all is legitimately needed. This is the same "make illegal/forgotten states a compile error" philosophy behind sealed types and records — the compiler becomes your exhaustiveness auditor.

🏋️ Exercises (attempt before opening solutions)

1 (easy): Type pattern. Rewrite a check-then-cast instanceof block (checking for a String and printing its length) using the type pattern.
Solution

if (obj instanceof String s) System.out.println(s.length()); — one line, no separate cast, and the check and use are bound together so they can't drift. Contrast the old if (obj instanceof String) { String s = (String) obj; ... }.

2 (easy): Type switch. Write a method that returns a description of an Object that's an Integer, String, List, or null, using a pattern-matching switch.
Solution

switch (o) { case null -> "null"; case Integer i -> "int " + i; case String s -> "str " + s.length(); case List<?> l -> "list of " + l.size(); default -> "other"; }. Note the explicit case null (otherwise a null selector throws) and the default because Object is open-ended.

3 (medium): Guards. Write a switch over an Integer that returns "negative", "zero", or "positive" using when guards. Explain the required ordering.
Solution

case Integer i when i < 0 -> "negative"; case Integer i when i == 0 -> "zero"; case Integer i -> "positive";. The guarded cases must come before the unguarded case Integer i — otherwise the unguarded one would match all integers first and shadow the guards. Most specific first.

4 (medium): Record deconstruction. Given record Point(int x,int y), write a switch that returns "origin" for (0,0), "on x-axis" when y==0, "on y-axis" when x==0, else "general", using record patterns and guards.
Solution

case Point(int x, int y) when x==0 && y==0 -> "origin"; case Point(int x, int y) when y==0 -> "on x-axis"; case Point(int x, int y) when x==0 -> "on y-axis"; case Point p -> "general";. Deconstruction binds x/y for the guards; ordering runs most-specific first. This reads as a direct description of the geometric cases.

5 (hard): Expression evaluator. Build the sealed Expr (Num/Add/Mul) from the lesson and write an exhaustive eval with record patterns and no default. Then add a Sub type and observe the compiler.
Solution

eval switches over Expr with case Num(double v), case Add(Expr l, Expr r), case Mul(Expr l, Expr r), recursing — no default because it's exhaustive over the sealed type. Adding a record Sub(Expr l, Expr r) implements Expr and its permits entry makes eval fail to compile ("not exhaustive") until you add the Sub case — the compiler pointing you at exactly what needs handling. That's the safety win over Visitor.

🎯 Key Takeaways

  • Type patterns (obj instanceof String s) check and bind in one step — no separate cast, flow-scoped binding.
  • Pattern-matching switch (21+) dispatches on type/shape, handles null, and supports when guards — replacing instanceof ladders.
  • Record patterns deconstruct data in the match (nesting supported), binding components directly.
  • Over a sealed type, pattern-matching switch is exhaustive — omit default so new subtypes break the build (a feature).
  • Order cases most-specific/guarded first; a broader case shadows narrower ones.
  • Sealed + records + pattern matching replace the Visitor pattern, turning "handle the new case everywhere" into compile errors (the expression problem).
