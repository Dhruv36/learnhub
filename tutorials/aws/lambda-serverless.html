Lambda & API Gateway Intermediate

Serverless flips the model: instead of running servers, you deploy functions that AWS runs on demand and scales automatically, and you pay only for the milliseconds they execute. Lambda is the compute; API Gateway is the front door that turns HTTP requests into function invocations. Together they build APIs and event-driven backends with no infrastructure to manage — but they come with real trade-offs (cold starts, statelessness, limits) you must design around.

Lambda Fundamentals

AspectDetail
ModelUpload a function (handler); AWS runs it per event, scaling out concurrently. No servers.
TriggersAPI Gateway, S3, DynamoDB Streams, SQS, SNS, EventBridge, schedules, and more.
ConfigMemory (128 MB–10 GB; CPU scales with it), timeout (≤15 min), runtime, env vars, IAM role.
PricingPer request + per GB-second of execution. Idle = free.
ConcurrencyEach concurrent invocation = one isolated instance; scales automatically (up to limits).
❄️ Cold starts: when Lambda spins up a new execution environment (first call or scaling up), there's added latency to initialize the runtime and your code. Mitigate with smaller packages, lighter runtimes, Provisioned Concurrency (kept warm) for latency-sensitive paths, and by doing heavy setup (DB clients) outside the handler so it's reused across warm invocations.

Statelessness & the Execution Environment

// Code OUTSIDE the handler runs once per environment and is REUSED across warm invocations:
const db = new Client();          // reused — initialize connections/SDK clients here
exports.handler = async (event) => {   // runs per invocation
  // no local state you can rely on between calls; /tmp is ephemeral scratch only
  return { statusCode: 200, body: "ok" };
};
// Persist anything durable to DynamoDB/RDS/S3. Never assume the same instance handles the next request.

🚪 Let the gateway own cross-cutting concerns. Authentication, throttling, validation, and caching belong at API Gateway, not duplicated in every function. Use a Cognito or Lambda authorizer for auth, usage plans/throttling to protect downstream, and caching to cut cost and latency — keeping the Lambda focused on business logic.

Event-Driven & Async Patterns

// Synchronous (API): client waits for the Lambda response (API Gateway → Lambda).
// Asynchronous (event): S3 upload → Lambda (thumbnail); Lambda retries on failure, use a DLQ.
// Stream/queue: SQS → Lambda (poll & batch); DynamoDB Streams → Lambda (react to changes).
// Orchestration: Step Functions coordinate multi-step workflows (retries, branching, long waits).
🧩 Design for idempotency and retries. Async and queue-triggered Lambdas can be invoked more than once (at-least-once delivery, retries), so make handlers idempotent — use an idempotency key, conditional writes, or dedup so processing the same event twice is safe. Attach a dead-letter queue (DLQ) to capture events that keep failing instead of losing them.

Interview Questions

Q1: What is a Lambda cold start and how do you mitigate it?

A cold start is the extra latency incurred when Lambda has to create a fresh execution environment to run your function — which happens on the first invocation, after periods of inactivity, and whenever it scales out to more concurrent instances. AWS must provision the sandbox, load your runtime and code, and run your initialization before the handler executes; subsequent "warm" invocations reuse that environment and skip the setup. To mitigate it: keep deployment packages small and dependencies lean, choose a fast-initializing runtime, and move expensive setup (database clients, SDK clients, config loading) to the module scope outside the handler so it runs once per environment and is reused. For latency-sensitive endpoints, Provisioned Concurrency keeps a pool of environments initialized and warm so requests never pay the cold-start cost. You accept cold starts on background/async work where the extra latency doesn't matter.

Q2: Why must Lambda functions be stateless and idempotent?

Lambda gives no guarantee about which execution environment handles a given request — it may reuse a warm one or spin up a new one, and environments are recycled unpredictably, with only an ephemeral /tmp and no durable local storage. So any state you need to keep must live in an external store (DynamoDB, RDS, S3, ElastiCache); you can't rely on in-memory state persisting between requests. Idempotency matters because many Lambda triggers deliver at-least-once and Lambda retries failed async invocations, so the same event can be processed more than once. If the handler isn't idempotent, retries cause duplicate side effects (double charges, duplicate records). You achieve idempotency with idempotency keys, conditional/atomic writes, and deduplication, and you attach a dead-letter queue so events that repeatedly fail are captured rather than silently lost. Statelessness plus idempotency is what makes serverless scale and retry safely.

Q3: What does API Gateway handle that you shouldn't put in Lambda?

API Gateway is the managed front door and should own the cross-cutting HTTP concerns: request routing and method/path mapping, authentication and authorization (via Cognito user pools, Lambda authorizers, or IAM), rate limiting and throttling (with usage plans and API keys to protect downstream services), request/response validation and transformation, TLS termination, and response caching. Centralizing these means every function gets consistent auth and protection without re-implementing it, the downstream Lambdas and databases are shielded from abusive traffic, and caching cuts both latency and cost. The Lambda should then focus purely on business logic. You choose HTTP APIs for the common, cheaper, lower-latency case and REST APIs when you need the richer feature set like API keys/usage plans, fine-grained request validation, or edge-optimized endpoints.

🏋️ Practice: Build a serverless API: an HTTP API Gateway route → Lambda → DynamoDB (create and read an item). Move the DynamoDB client outside the handler and measure the cold vs warm latency. Add a Cognito or Lambda authorizer and a throttling limit at the gateway. Then wire an async path: an S3 upload triggers a Lambda that writes a record, make the handler idempotent, and attach a DLQ. Finally, coordinate a two-step process with Step Functions.

🎯 Key Takeaways

  • Lambda runs functions on demand, scales automatically, and bills per request + GB-second — no servers, idle is free.
  • Cold starts add startup latency; mitigate with small packages, init outside the handler, and Provisioned Concurrency on hot paths.
  • Functions are stateless (ephemeral /tmp) and often invoked at-least-once — persist state externally and make handlers idempotent; use DLQs.
  • API Gateway owns routing, auth, throttling, validation, and caching (HTTP API = default; REST API = richer features) so Lambdas stay focused on logic.
  • Respect limits: 15-min timeout (long jobs → Step Functions/Fargate), concurrency vs RDS connections (RDS Proxy), payload/package size (layers, S3 pointers).
