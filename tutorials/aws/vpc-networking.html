VPC Networking Basics

A VPC (Virtual Private Cloud) is your own isolated network inside AWS — you control its IP range, subnets, routing, and firewalls. Nearly every "my instance can't reach the internet" or "the database is exposed" problem is a VPC problem. This is the connective tissue that decides what can talk to what, and the public/private subnet split is the single most important pattern to internalize.

The Core Building Blocks

ComponentRole
VPCAn isolated virtual network with a CIDR block (e.g., 10.0.0.0/16).
SubnetA slice of the VPC CIDR that lives in one AZ. Public or private by routing.
Route tableRules deciding where a subnet's traffic goes (local, internet, NAT, peering…).
Internet Gateway (IGW)Attached to the VPC; gives public subnets two-way internet access.
NAT GatewayLets private subnets make outbound internet calls without being reachable inbound.
Security GroupStateful firewall on an ENI/instance (allow rules only).
Network ACL (NACL)Stateless firewall at the subnet boundary (allow + deny rules).

Public vs Private Subnets (the essential pattern)

VPC 10.0.0.0/16, spread across AZ-a and AZ-b:
  Public subnets  (route 0.0.0.0/0 → Internet Gateway)   → ALB, NAT Gateway, bastion
  Private subnets (route 0.0.0.0/0 → NAT Gateway)         → app servers, containers
  Private (isolated, no 0.0.0.0/0 route)                  → databases (RDS), caches
// A subnet is "public" ONLY because its route table sends internet-bound traffic to an IGW.
// App tier sits private (reachable only via the ALB); DB tier is fully isolated.
🔑 What makes a subnet public is its route table, not a checkbox. A subnet is public if it has a route 0.0.0.0/0 → Internet Gateway and its instances have public IPs. Private subnets reach the internet outbound only through a NAT Gateway (for updates/API calls) but can't be reached from the internet. Put load balancers in public subnets, apps in private, databases in isolated private subnets — defense in depth.

Security Groups vs NACLs

Security GroupNetwork ACL
Attached toInstance/ENISubnet
StateStateful (return traffic auto-allowed)Stateless (must allow both directions)
RulesAllow onlyAllow and Deny
EvaluationAll rules; any match allowsNumbered rules, first match wins
Typical usePrimary control ("app SG allows 443 from ALB SG")Coarse subnet guardrails / explicit denies
🧠 Security groups can reference each other. Instead of hardcoding IPs, write "DB SG allows 5432 from the App SG" — as app instances scale in/out, the rule follows automatically. This SG-to-SG referencing is the clean, dynamic way to model tiers. Prefer security groups as your main control; reach for NACLs only for coarse subnet-level allow/deny.

Connecting Beyond the VPC

NeedTool
Reach AWS services (S3/DynamoDB) privatelyVPC Endpoints (Gateway/Interface) — no NAT, no public internet
Connect two VPCsVPC Peering (1:1) or Transit Gateway (hub for many)
Connect on-premSite-to-Site VPN or Direct Connect (dedicated line)
Expose your service to other VPCs privatelyAWS PrivateLink
🔌 VPC Endpoints keep AWS-service traffic off the internet. A Gateway Endpoint for S3/DynamoDB (free) or an Interface Endpoint (PrivateLink) lets private-subnet resources reach AWS services without a NAT Gateway or public exposure — cheaper and more secure. This is a common cost/security win over routing everything through NAT.

Interview Questions

Q1: Explain the difference between a public and a private subnet.

The difference is entirely in routing. A public subnet has a route table entry sending internet-bound traffic (0.0.0.0/0) to an Internet Gateway, and its instances have public IPs, so they can both initiate and (if their security group allows) receive traffic from the internet. A private subnet has no route to the Internet Gateway; to reach the internet outbound — for OS updates or calling external APIs — its traffic is routed to a NAT Gateway that lives in a public subnet, and nothing on the internet can initiate a connection back to it. The standard three-tier layout puts internet-facing components (load balancers, NAT, bastions) in public subnets, application servers in private subnets reachable only through the load balancer, and databases in isolated private subnets with no internet route at all — which is defense in depth by network placement.

Q2: Security groups vs network ACLs?

Security groups are stateful firewalls attached to an instance's network interface: they have allow rules only, and because they're stateful, return traffic for an allowed connection is automatically permitted (you don't write a reverse rule). They can also reference other security groups, so you can express "the database SG allows port 5432 from the app SG" and it tracks instances as they scale. Network ACLs are stateless firewalls at the subnet boundary: they support both allow and deny rules, are evaluated by rule number with first-match-wins, and being stateless you must explicitly allow return traffic in both directions. In practice security groups are the primary, fine-grained control, and NACLs are used for coarse subnet-level guardrails or to explicitly deny specific ranges. The most common bug is forgetting NACLs are stateless and blocking the ephemeral return ports.

Q3: How do resources in a private subnet reach AWS services or the internet securely?

For general outbound internet access (patching, third-party APIs), private-subnet resources route 0.0.0.0/0 to a NAT Gateway that sits in a public subnet — this allows outbound connections while preventing any inbound connections from the internet, and you deploy one NAT Gateway per AZ for availability. For AWS services specifically, the better approach is VPC Endpoints: a Gateway Endpoint for S3 and DynamoDB (free) or an Interface Endpoint (PrivateLink) for most other services keeps that traffic entirely on the AWS network — no NAT, no public internet, lower cost, and tighter security. So the pattern is: NAT Gateway for arbitrary internet egress, and VPC Endpoints to reach AWS services privately, avoiding both the cost and the exposure of sending that traffic out through NAT.

🏋️ Practice: Build a VPC (10.0.0.0/16) with a public and a private subnet in each of two AZs. Add an Internet Gateway and route the public subnets to it; add a NAT Gateway per AZ and route the private subnets to it. Launch an app instance in a private subnet and confirm it can curl the internet but isn't reachable inbound. Create an app SG and a DB SG where the DB SG only allows 5432 from the app SG. Add an S3 Gateway Endpoint and confirm S3 access no longer traverses NAT.

🎯 Key Takeaways

  • A VPC is your isolated network (CIDR + subnets + route tables + gateways); a subnet lives in one AZ.
  • Public vs private is defined by routing: public routes 0.0.0.0/0 → Internet Gateway; private routes it → NAT Gateway (outbound only); databases sit in isolated subnets.
  • Security groups = stateful, allow-only, instance-level, can reference each other (the primary control); NACLs = stateless, allow+deny, subnet-level guardrails.
  • Use VPC Endpoints to reach S3/DynamoDB/other AWS services privately (cheaper and safer than NAT); one NAT Gateway per AZ for HA.
  • Never expose databases or SSH/DB ports to 0.0.0.0/0; plan non-overlapping CIDRs; connect VPCs via peering/Transit Gateway and on-prem via VPN/Direct Connect.
