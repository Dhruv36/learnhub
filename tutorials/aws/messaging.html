SQS, SNS & EventBridge Intermediate

Decoupling is how distributed systems stay resilient: instead of services calling each other directly and failing together, they communicate through messaging. AWS gives you three complementary tools — SQS (queues, point-to-point), SNS (pub/sub fan-out), and EventBridge (an event bus with routing) — and knowing which to reach for, and how they combine, is a hallmark of event-driven design.

The Three Services at a Glance

ServicePatternOne-liner
SQSQueue (point-to-point)A buffer of messages one consumer group pulls and processes
SNSPub/Sub (fan-out)Publish once → pushed to many subscribers (queues, Lambda, HTTP, email/SMS)
EventBridgeEvent bus + routingRoute events by content rules to many targets; SaaS/AWS event integrations, schedules
📬 Pull vs push. SQS is pull: consumers poll the queue and delete messages when done — the queue absorbs load and lets you process at your own pace. SNS and EventBridge are push: they deliver events to subscribers/targets as they arrive. Queues buffer and smooth; pub/sub and buses distribute and route.

SQS — the Buffer

Producer → [ SQS queue ] → Consumer(s) poll, process, then DELETE the message
// Standard queue:  ~unlimited throughput, at-least-once delivery, best-effort ordering.
// FIFO queue:      exactly-once processing + strict ordering (lower throughput).
// Visibility timeout: a polled message is hidden while being processed; if not deleted in time,
//   it reappears for another consumer (so make processing idempotent).
// Dead-Letter Queue (DLQ): messages that fail N times move here for inspection.
🛡️ Queues make you resilient to spikes and failures. If the consumer is down or slow, messages wait safely in the queue instead of being lost or overwhelming the service. This buffering (load leveling) plus retries and a DLQ is why you put a queue between a bursty producer and a rate-limited consumer.

SNS — Fan-Out

                       ┌→ SQS queue A  → order-processing
Publisher → SNS topic ─┼→ SQS queue B  → analytics
                       ├→ Lambda        → notifications
                       └→ HTTPS / email / SMS
// One publish is delivered to ALL subscribers. Classic "fan-out": SNS → multiple SQS queues,
// so each downstream service gets its own durable copy to process independently.

EventBridge — the Router

// A serverless event bus that ROUTES events by rules matching their content:
{
  "source": ["orders"],
  "detail-type": ["OrderPlaced"],
  "detail": { "amount": [{ "numeric": [">", 100] }] }   // rule: only big orders
}  → target: a Lambda, an SQS queue, a Step Function, another bus…
// Also: native events from 100+ AWS services & SaaS partners, plus scheduled rules (cron).
🧭 EventBridge is SNS's smarter cousin for AWS-native, content-routed events. Where SNS blindly fans out to all subscribers, EventBridge filters and routes each event to specific targets based on rich content-based rules, integrates natively with AWS/SaaS event sources, supports a schema registry and archive/replay, and can run scheduled jobs. Use SNS for simple high-throughput fan-out/notifications; use EventBridge when you need routing, filtering, and ecosystem integrations.

Interview Questions

Q1: SQS vs SNS — when do you use each, and how do they combine?

SQS is a message queue for point-to-point, pull-based processing: a producer puts messages on the queue and one consumer group polls, processes, and deletes them. It buffers load, lets consumers work at their own pace, and provides retries and dead-letter queues — ideal when you want to decouple a producer from a slower/bursty consumer and guarantee work isn't lost. SNS is publish/subscribe, push-based fan-out: you publish a message once to a topic and it's delivered to all subscribers (Lambdas, HTTP endpoints, email/SMS, and importantly SQS queues) — ideal when one event must notify many consumers. The powerful combination is the fan-out pattern: SNS topic → multiple SQS queues, one per downstream service. That gives each consumer its own durable, independently-paced buffer with retries/DLQ, while a single publish reaches them all — decoupling and resilience together.

Q2: What is a visibility timeout and why must SQS consumers be idempotent?

When a consumer receives a message from an SQS queue, the message isn't deleted — it's made invisible to other consumers for the duration of the visibility timeout while that consumer processes it. If the consumer finishes and explicitly deletes the message within that window, it's gone; but if processing takes longer than the timeout or the consumer crashes without deleting, the message becomes visible again and another consumer picks it up. Combined with SQS standard queues' at-least-once delivery, this means the same message can legitimately be processed more than once, so handlers must be idempotent — using deduplication keys, conditional writes, or otherwise making repeated processing harmless — to avoid duplicate side effects. You also set the visibility timeout to comfortably exceed worst-case processing time and attach a dead-letter queue so messages that repeatedly fail are captured instead of cycling forever.

Q3: When would you choose EventBridge over SNS?

EventBridge is a serverless event bus built for content-based routing and ecosystem integration, so I choose it when I need to route events to different targets based on their content, filter which events go where with rich rules, integrate with the 100+ AWS services and SaaS partners that emit events natively, or run scheduled (cron-like) jobs — plus features like a schema registry and event archive/replay. SNS, by contrast, is optimized for simple, very high-throughput fan-out and notifications: it pushes each message to all subscribers with lower latency and cost but without content-based routing. So for straightforward "broadcast to everyone" or notification use cases (especially at extreme throughput, or to email/SMS), SNS is the leaner choice; for event-driven architectures that need routing, filtering, AWS/SaaS event sources, and scheduling, EventBridge is the better fit. They can also be used together in a larger event-driven system.

🏋️ Practice: Build the fan-out: an SNS topic subscribed by two SQS queues; publish one message and see it land in both. Add a Lambda consumer on one queue, set a visibility timeout, make the handler idempotent, and attach a DLQ with maxReceiveCount=3 (force a failure to watch a message move to the DLQ). Then create an EventBridge rule that only routes OrderPlaced events over $100 to a target, and add a scheduled rule that triggers a Lambda every 5 minutes.

🎯 Key Takeaways

  • SQS = pull-based queue that buffers work, smooths spikes, and retries (with DLQs); SNS = push-based pub/sub fan-out; EventBridge = content-routed event bus with AWS/SaaS integrations and schedules.
  • The fan-out pattern (SNS → multiple SQS queues) gives every consumer its own durable, independently-paced buffer from a single publish.
  • SQS standard = at-least-once + best-effort order; FIFO = exactly-once + strict order (lower throughput). Make consumers idempotent.
  • Visibility timeout hides a message while it's processed; set it above worst-case processing time and use long polling to cut cost.
  • Choose EventBridge over SNS when you need content-based routing/filtering, native AWS/SaaS event sources, scheduling, or replay; SNS for simple high-throughput fan-out/notifications.
