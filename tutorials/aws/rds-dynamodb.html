RDS & DynamoDB Intermediate

Data is where architecture decisions get expensive to reverse, so AWS's two flagship database services are worth understanding deeply. RDS (and Aurora) is managed relational SQL — pick it when you need joins, transactions, and a flexible query model. DynamoDB is a fully managed NoSQL key-value/document store — pick it for predictable single-digit-millisecond access at essentially unlimited scale. Knowing when to reach for which, and how each scales, is a core senior skill.

RDS / Aurora — Managed Relational

FeatureWhat it gives you
Managed enginesPostgreSQL, MySQL, MariaDB, Oracle, SQL Server; Aurora (MySQL/Postgres-compatible, cloud-native)
Multi-AZA synchronous standby in another AZ; automatic failover for availability
Read replicasAsynchronous copies to scale read traffic (and cross-region DR)
Automated backupsPoint-in-time recovery + snapshots
AWS managesPatching, backups, failover, host health — you keep schema/queries/indexes
🔁 Multi-AZ ≠ read replica. Multi-AZ is a synchronous standby you can't read from — its whole job is automatic failover for high availability. Read replicas are asynchronous copies you can read from to offload read traffic (they lag slightly and aren't for HA). Interviews love this distinction: Multi-AZ = availability, read replicas = read scalability.

Aurora Specifics

// Aurora separates compute from a distributed storage layer replicated 6 ways across 3 AZs.
//   - Up to 15 low-lag read replicas sharing that storage; fast failover.
//   - Aurora Serverless v2: auto-scales capacity to load (great for spiky/variable workloads).
//   - Global Database: cross-region replication with ~1s lag for DR / global reads.
// Choose Aurora when you want RDS compatibility with better performance, scaling, and resilience.

DynamoDB — Managed NoSQL at Scale

ConceptMeaning
Table + itemsSchemaless items (JSON-like) identified by a primary key
Partition keyHashed to distribute data across partitions — choose for even spread
Sort keyOptional; enables range queries within a partition (composite key)
GSI / LSISecondary indexes to query by other attributes
CapacityOn-Demand (pay per request) or Provisioned (set RCU/WCU + autoscaling)
ExtrasDynamoDB Streams (change feed), DAX (in-memory cache), TTL, global tables
🔑 DynamoDB rewards designing for your access patterns first. Unlike SQL, you model the table around how you'll query, not around normalized entities. A bad partition key (low cardinality or a "hot" value everyone hits) creates a hot partition that throttles. Get the key design right and DynamoDB delivers single-digit-millisecond latency at any scale with no servers to manage.

SQL (RDS) vs NoSQL (DynamoDB) — Choosing

Choose RDS/Aurora when…Choose DynamoDB when…
Complex queries, joins, ad-hoc analyticsKnown, simple access patterns (get/put by key)
Strong multi-row transactions & relational integrityMassive scale & predictable low latency matter most
Flexible/evolving queries on structured dataServerless, spiky, or huge-throughput workloads
Moderate, vertically-scalable volumeYou want zero DB ops and seamless horizontal scale
⚖️ It's about access patterns, not "SQL good / NoSQL good." Relational shines when queries are varied and relationships matter; DynamoDB shines when the access pattern is known and you need horizontal scale with flat latency. Many systems use both — RDS for the transactional core, DynamoDB for high-throughput features (sessions, feeds, carts).

Interview Questions

Q1: RDS Multi-AZ vs read replicas — what's the difference?

Multi-AZ deployment maintains a synchronous standby replica in a different Availability Zone whose sole purpose is high availability: you can't read or write to it, but if the primary fails (hardware, AZ outage, patching), RDS automatically fails over to the standby, usually within a minute or two, and DNS is updated so the app reconnects. Read replicas are asynchronous copies you can direct read traffic to, so they scale read-heavy workloads by offloading queries from the primary; because replication is async they can lag slightly and are not intended as an HA mechanism (though a replica can be promoted, and cross-region replicas help DR). The concise framing: Multi-AZ is for availability (automatic failover, one hidden standby), read replicas are for read scalability (multiple readable copies). They're complementary and often used together.

Q2: How do you choose a good DynamoDB partition key?

DynamoDB hashes the partition key to spread items across partitions, so the key must distribute both storage and request traffic evenly. A good key has high cardinality and no single value that receives a disproportionate share of reads/writes — e.g., a user ID or a well-distributed UUID rather than a status field with three values or a "today's date" that every write hammers. A skewed or low-cardinality key creates a hot partition that throttles even when the table has plenty of provisioned capacity. Techniques to fix skew include adding a sort key to form composite keys, using a GSI for alternate access patterns, and write-sharding a naturally hot key by suffixing it with a random shard number and fanning out reads. Fundamentally you design the key from your access patterns so that load is spread — that's what lets DynamoDB stay at single-digit-millisecond latency at scale.

Q3: When would you pick DynamoDB over a relational database?

I pick DynamoDB when the access patterns are known and simple (get/put/query by key), the workload needs to scale horizontally to very high, spiky, or unbounded throughput, and I want predictable single-digit- millisecond latency with essentially zero database operations — no servers, patching, or capacity planning beyond choosing on-demand or provisioned capacity. It's ideal for things like user sessions, shopping carts, activity feeds, IoT ingestion, and other high-volume features with a clear key-based access shape. I'd choose RDS/Aurora instead when I need complex or ad-hoc queries, joins across entities, multi-row ACID transactions, and a flexible relational model — the transactional core of most business apps. The deciding factor isn't "NoSQL vs SQL" in the abstract but whether the access patterns are fixed and scale-critical (DynamoDB) or varied and relationship-heavy (relational). Plenty of architectures use both.

🏋️ Practice: Launch a Multi-AZ RDS PostgreSQL instance in private subnets; connect only from an app instance via SG-to-SG rules. Add a read replica and route read-only queries to it. Then create a DynamoDB table with a composite key (partition + sort), add a GSI, and compare a Query vs a Scan. Enable a Stream and a TTL attribute. Reflect on which of your app's features fit RDS vs DynamoDB.

🎯 Key Takeaways

  • RDS/Aurora = managed relational SQL (joins, transactions, flexible queries); DynamoDB = managed NoSQL key-value/document at unlimited scale with flat low latency.
  • Multi-AZ = synchronous standby for automatic failover (availability); read replicas = async readable copies for read scaling — don't confuse them.
  • Aurora adds a distributed 6-way storage layer, up to 15 low-lag replicas, Serverless v2 autoscaling, and Global Database for cross-region DR.
  • DynamoDB: design the table around access patterns; choose a high-cardinality, evenly-accessed partition key to avoid hot partitions; Query (not Scan).
  • Choose by access pattern: varied/relational/transactional → RDS; known-pattern/high-scale/predictable-latency → DynamoDB. Keep databases in private subnets with backups + PITR.
