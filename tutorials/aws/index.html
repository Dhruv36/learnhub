Accounts, IAM & Regions Basics

Before you launch a single server, you have to understand how AWS is organized and how it decides who is allowed to do what. Almost every "why can't I…" and every security incident traces back to two things: the account/region model (where your resources physically live and how they're isolated) and IAM (who is authorized to touch them). Get this foundation right and the rest of AWS is just services layered on top.

The Global Structure: Regions & Availability Zones

ConceptWhat it is
RegionA geographic area (e.g., us-east-1, eu-west-1). Isolated from other regions; you choose one for latency, cost, and data-residency reasons.
Availability Zone (AZ)One or more discrete data centers within a region, with independent power/network. A region has 3+ AZs.
Edge LocationHundreds of PoPs used by CloudFront/Route 53 to serve content close to users.
🌍 Design across AZs, choose across Regions. High availability inside a region means spreading resources across multiple AZs — a single data-center failure shouldn't take you down. Regions are for latency to users, legal data-residency, and disaster recovery. Most services are regional (an S3 bucket, an EC2 instance live in one region); a few (IAM, Route 53, CloudFront, billing) are global.

The Account Model

Root user (the email you signed up with)  → almost unlimited power; LOCK IT AWAY.
  └─ IAM users / roles              → day-to-day identities with scoped permissions
AWS Organizations                    → many accounts under one umbrella (billing + guardrails)
  ├─ Management (payer) account
  ├─ OU: Security   (log-archive, audit accounts)
  ├─ OU: Workloads  (prod account, staging account, dev account)
  └─ Service Control Policies (SCPs) → org-wide guardrails that CAP what accounts can do
🔐 The account is the primary blast-radius boundary. Separate prod, staging, and dev into different accounts so a mistake or breach in dev can't touch prod. This is the modern best practice (a "multi-account landing zone"), managed with AWS Organizations + SCPs. Even a solo project benefits from a separate account per environment.

IAM: The Four Building Blocks

ElementRole
UserA long-lived human/service identity with credentials. Minimize these — prefer roles + federation.
GroupA collection of users; attach policies to the group, not each user.
RoleAn identity with permissions that is assumed temporarily — by EC2/Lambda, by another account, or by a federated user. No long-lived keys.
PolicyA JSON document listing allowed/denied actions on resources. Attached to users, groups, or roles.

How an IAM Policy Reads

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-bucket/*",
    "Condition": { "IpAddress": { "aws:SourceIp": "203.0.113.0/24" } }
  }]
}
// Effect (Allow/Deny) + Action (what) + Resource (on which ARN) + optional Condition (when).
// Evaluation: an explicit Deny ALWAYS wins; otherwise you need an explicit Allow; default is deny.
🧠 The IAM decision rule: start from implicit deny → if any policy has an explicit Deny for the action, DENY (nothing overrides it) → otherwise if some policy Allows it, ALLOW → otherwise DENY. This "deny by default, explicit-deny-wins" model is the single most important IAM fact to internalize.

Roles > Long-Lived Keys (the golden rule)

// BAD: bake an access key into an EC2 instance / app config.
//   → keys leak, never rotate, and grant standing access.
// GOOD: attach an IAM ROLE to the EC2 instance / Lambda / ECS task.
//   → AWS injects TEMPORARY credentials (via STS) that auto-rotate. Nothing to leak.
// For humans: use IAM Identity Center (SSO) / federation, not per-person IAM users with keys.

Interview Questions

Q1: Explain the difference between an IAM user and an IAM role.

An IAM user is a long-lived identity with its own credentials (a password and/or access keys) intended for a specific person or service; it exists until deleted. An IAM role is an identity with a permission set that no one owns permanently — it's assumed temporarily, and when you assume it, STS issues short-lived credentials that expire automatically. Roles are how EC2 instances, Lambda functions, and ECS tasks get permissions (via an attached role, so there are no static keys to leak), how one AWS account grants access to another (cross-account roles), and how federated/SSO users get access without an IAM user each. The best practice is to minimize IAM users with static keys and lean on roles and federation, because temporary auto-rotating credentials dramatically reduce the risk of leaked, never-rotated keys.

Q2: How does IAM decide whether a request is allowed?

Everything is denied by default (implicit deny). When a request is made, AWS gathers all applicable policies — identity-based policies on the user/role, resource-based policies (like an S3 bucket policy), permission boundaries, session policies, and organizational SCPs — and evaluates them. The rule is: if any applicable policy contains an explicit Deny for that action/resource, the request is denied and nothing can override it; otherwise, if at least one policy explicitly Allows it (and no boundary/SCP removes it), it's allowed; if nothing allows it, the implicit deny stands. So the mental model is "deny by default, an explicit Allow is required, and an explicit Deny always wins." SCPs and permission boundaries act as ceilings — they can only take permissions away, never grant them.

Q3: Why use multiple AWS accounts instead of one?

The AWS account is the strongest isolation boundary AWS offers — separate accounts have separate resource namespaces, separate default limits, and no implicit trust between them. Splitting workloads into multiple accounts (typically prod, staging, dev, plus dedicated security/log-archive accounts) means a compromise or a fat-fingered deletion in one environment can't reach another, blast radius is contained, and you get clean per-environment billing and quota separation. AWS Organizations ties them together for consolidated billing and centralized governance, and Service Control Policies apply org-wide guardrails (e.g., "no one, not even an account admin, may disable CloudTrail or launch in an unapproved region"). This "multi-account landing zone" is the standard enterprise pattern; tools like AWS Control Tower automate setting it up.

🏋️ Practice: Secure a fresh account: enable MFA on the root user and then stop using it. Create an admin IAM group with a scoped policy and a user in it. Write a policy that allows read-only access to a single S3 bucket and test it. Launch an EC2 instance with an instance role that can read that bucket (no keys) and confirm it works via the CLI on the box. Finally, note which of your resources are regional vs global by switching the region selector.

🎯 Key Takeaways

  • Regions are isolated geographies; spread across multiple AZs for HA within a region. Most services are regional; IAM, Route 53, CloudFront, and billing are global.
  • Lock away the root user (MFA, don't use it daily); do everything through scoped IAM identities.
  • IAM = Users, Groups, Roles, Policies. Prefer roles + federation over long-lived access keys — roles hand out temporary, auto-rotating STS credentials.
  • Policy evaluation: deny by default → explicit Allow required → explicit Deny always wins. SCPs/boundaries are ceilings that only subtract.
  • Separate accounts per environment (via Organizations + SCPs) is the primary blast-radius boundary — the multi-account landing zone pattern.
