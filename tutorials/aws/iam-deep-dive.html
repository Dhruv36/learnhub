IAM Deep Dive: Policies, Roles & STS Advanced

The basics lesson introduced IAM; this one goes to the depth interviews and real incidents demand. You'll master the full policy-evaluation logic, the different policy types and how they interact, how AssumeRole/STS actually issues temporary credentials, and cross-account access. This is the material that separates "I clicked AdministratorAccess" from someone who can design and debug real authorization.

The Policy Types (and how they combine)

Resource-basedA resource (S3 bucket, SQS, KMS key)Who (incl. other accounts) can act on this resource

The Full Evaluation Logic

1. Default = DENY (implicit).
2. Is there an explicit DENY anywhere (identity/resource/SCP/boundary)? → DENY. (always wins)
3. Do the ORG SCPs allow the action?        → if not, DENY.
4. Does a permission boundary allow it?      → if not, DENY.
5. Does an identity-based OR resource-based policy ALLOW it? → if yes, ALLOW; else DENY.
// Boundaries and SCPs can only REMOVE permissions (ceilings). An Allow is required from
// identity or resource policy. An explicit Deny overrides every Allow, everywhere.
🧠 Ceilings vs grants. The mental model: SCPs and permission boundaries are ceilings that cap the maximum — they never grant anything. Identity-based and resource-based policies are the actual grants. The effective permission is the intersection of all ceilings with the union of grants, minus any explicit Deny. This is the single most-tested IAM concept at senior level.

AssumeRole & STS — Temporary Credentials

// A role has TWO policies:
//   Trust policy       → WHO may assume it (a principal: user, service, or another account)
//   Permission policy  → WHAT the assumed session may do
// Assuming it:
sts:AssumeRole → returns { AccessKeyId, SecretAccessKey, SessionToken, Expiration }
// These are TEMPORARY (minutes to hours) and auto-expire. EC2/Lambda/ECS assume roles
// transparently; humans assume via the console/CLI or SSO. Nothing long-lived to leak.
🔑 The trust policy is the gate; the permission policy is the grant. To assume a role you must (a) be allowed by the role's trust policy (who) and (b) have sts:AssumeRole permission yourself. STS then mints short-lived credentials. This is the backbone of every "no static keys" pattern — instance roles, cross-account access, and federation all run on AssumeRole.

Cross-Account Access

Account A (dev) role wants to read a bucket in Account B (prod):
  In B: role "ReadData" with a TRUST policy allowing principal = Account A (or a specific role in A)
        and a permission policy allowing s3:GetObject on the bucket.
  In A: the identity has sts:AssumeRole on B's role ARN.
  Flow: A calls sts:AssumeRole(B::ReadData) → temp creds → reads B's bucket.
// Prefer role-assumption over resource policies granting foreign accounts directly — it's
// auditable, temporary, and centrally revocable.

Interview Questions

Q1: Walk through how IAM evaluates a request end to end.

Everything starts denied by default. AWS collects every policy relevant to the request — the identity's identity-based policies, any resource-based policy on the target, permission boundaries on the principal, organizational SCPs on the account, and any session policy from AssumeRole — and evaluates them together. First, if any of them contains an explicit Deny for the action/resource, the request is denied and nothing can override that. Otherwise the request must survive every ceiling: the SCPs must allow it, and the permission boundary (if present) must allow it — these only cap permissions, they never grant. Finally there must be an explicit Allow from an identity-based or resource-based policy. If all ceilings permit it and there's an Allow with no explicit Deny, it's allowed; if any ceiling blocks it or nothing allows it, the implicit deny stands. The concise model: explicit Deny wins; boundaries/SCPs are intersecting ceilings; an Allow from identity or resource policy is required.

Q2: How does assuming a role work, and why is it central to AWS security?

A role has two policies: a trust policy that defines who is allowed to assume it (a principal — an IAM user, an AWS service like EC2/Lambda, another account, or a federated identity), and a permission policy that defines what the resulting session can do. To assume it, the caller must be permitted by that trust policy and must itself have sts:AssumeRole permission; STS then returns temporary credentials (access key, secret, session token) with a short expiry. This is central because it eliminates long-lived credentials: EC2 instances, Lambda functions, and ECS tasks transparently assume roles to get auto-rotating temporary keys, humans assume roles via SSO/federation, cross-account access works by assuming a role in the other account, and CI systems use OIDC to assume a role instead of storing static keys. Temporary, scoped, revocable credentials issued on demand are the foundation of AWS's "no static secrets" security posture.

Q3: A user has an Allow for an action but the call is still denied. Why?

Because an Allow is necessary but not sufficient — something above it is blocking the request. The most common causes, in order: (1) an explicit Deny somewhere applicable (in the identity policy, a resource-based policy, an SCP, or a permission boundary) always overrides the Allow; (2) an organizational SCP on the account doesn't permit the action, so even an account admin is capped; (3) a permission boundary on the user caps their effective permissions below what the identity policy grants; (4) a resource-based policy (e.g., an S3 bucket policy or KMS key policy) denies or fails to allow the cross-account/principal; or (5) a condition in a policy isn't met (wrong source IP, missing MFA, wrong tag). To debug, I use the IAM Policy Simulator and check for explicit Denies and the ceilings (SCPs/boundaries) rather than just adding more Allows — the fix is usually removing a Deny or adjusting an SCP/boundary/condition, not granting more.

🏋️ Practice: Create a role with a trust policy scoped to a specific user and a permission policy allowing read on one bucket; assume it via the CLI and inspect the temporary credentials. Set up cross-account access between two of your accounts using AssumeRole. Add a permission boundary that caps a user below their identity policy and observe the effect. Finally, add an explicit Deny and use the IAM Policy Simulator to see it override an Allow.

🎯 Key Takeaways

  • Policy types: identity-based & resource-based (grants) vs permission boundaries & SCPs (ceilings that only subtract) vs session policies (narrow a session).
  • Evaluation: default deny → explicit Deny always wins → must pass every ceiling (SCP, boundary) → needs an explicit Allow from identity or resource policy.
  • A role has a trust policy (who may assume it) and a permission policy (what the session can do); STS issues temporary, auto-expiring credentials via AssumeRole.
  • Cross-account, instance roles, federation, and CI (GitHub OIDC) all run on AssumeRole — the backbone of "no long-lived keys."
  • Use IAM Identity Center/federation for humans, OIDC for CI; apply least privilege, scope trust policies and iam:PassRole, and debug with the Policy Simulator.
