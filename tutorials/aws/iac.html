Infrastructure as Code Advanced

Clicking around the AWS console doesn't scale, isn't reproducible, and leaves no audit trail. Infrastructure as Code (IaC) defines your cloud resources in version-controlled files that you apply to create/update them — so infrastructure becomes reviewable, repeatable, and automatable. The main tools on AWS are CloudFormation (native), CDK (real programming languages), and Terraform (cloud-agnostic). Understanding declarative provisioning, state, and drift is essential senior knowledge.

Why IaC

BenefitHow IaC delivers it
ReproducibilitySpin up identical dev/staging/prod (and DR) from the same code
Version control & reviewChanges are diffs in Git, peer-reviewed via PRs, revertible
AuditabilityGit history = who changed what infrastructure, when, and why
AutomationProvision in CI/CD pipelines; no manual console steps
ConsistencyEliminates config drift and "works in my environment" surprises
📦 Declarative, not imperative. You describe the desired end state ("this VPC, these subnets, this RDS") and the tool figures out the create/update/delete actions to reach it — the same reconciliation idea as Kubernetes. You don't script "run these API calls in order"; you declare what should exist and let the engine converge to it.

State & the Plan/Apply Cycle

terraform plan    # diff desired (code) vs current (state) → shows what will change
terraform apply   # make it so; update the state file
// STATE maps your code to real resources. Terraform keeps a state file (store it REMOTELY in
// S3 + DynamoDB lock, never local/committed); CloudFormation/CDK keep state inside the stack (AWS-managed).
// DRIFT = someone changed a resource by hand → code no longer matches reality. Detect & re-converge.
🗄️ State is the crux — and the footgun. The tool needs a record of which real resources its code owns. Terraform's state file must live in remote, locked, shared storage (S3 + DynamoDB lock) so a team doesn't corrupt it or leak the secrets it can contain. Never commit state to Git or keep it only on a laptop. CloudFormation/CDK avoid this by having AWS manage stack state for you — a real advantage of the native tools.

Interview Questions

Q1: What is Infrastructure as Code and why does it matter?

Infrastructure as Code means defining your cloud resources — networks, servers, databases, IAM, etc. — in declarative, version-controlled files that a tool applies to create and update the real infrastructure, instead of clicking through a console. It matters because it makes infrastructure reproducible (you can stand up identical dev, staging, prod, and DR environments from the same code), reviewable and auditable (every change is a Git diff reviewed in a PR, with history showing who changed what and why, and easy rollback), automatable (provisioning runs in CI/CD with no manual steps), and consistent (it eliminates configuration drift and environment-specific surprises). It's declarative: you describe the desired end state and the engine computes the actions to reach it, much like Kubernetes reconciliation. In short, IaC turns infrastructure into software-engineering artifacts with all the discipline — versioning, review, testing, automation — that implies.

Q2: What is state in Terraform, and how should you manage it?

State is Terraform's record mapping the resources declared in your code to the actual resources it created in the cloud, including their attributes; Terraform uses it to compute the diff between desired (code) and current (state/reality) on plan and to know what to change on apply. It must be managed carefully: store it remotely and shared — the standard is an S3 bucket with a DynamoDB table for state locking — so team members don't run concurrent applies that corrupt it, and so it's durable and encrypted. You never commit state to Git or leave it only on a laptop, both because that breaks collaboration/locking and because state can contain secrets in plaintext. You also isolate state per environment (separate state files or workspaces) to limit blast radius. CloudFormation and CDK sidestep this by having AWS manage stack state internally, which removes the remote-state responsibility entirely — a notable operational advantage of the native tools.

Q3: CloudFormation/CDK vs Terraform — how would you choose?

If the organization is all-in on AWS and wants a fully managed experience with no external state to operate, CloudFormation (or CDK, which synthesizes to CloudFormation) is compelling: state is AWS-managed inside the stack, it integrates natively with AWS features, and CDK lets developers use real programming languages (TypeScript, Python) with loops, abstractions, and unit tests while still getting CloudFormation's managed deployments. If the environment is multi-cloud, or the team wants one consistent tool and language across providers with the largest ecosystem of providers/modules, Terraform is the better fit — but you take on managing remote state and locking. So I weigh cloud breadth (AWS-only vs multi-cloud), whether the team prefers a DSL (HCL/YAML) or a general-purpose language (CDK/Pulumi), and how much they value AWS-managed state vs Terraform's portability and ecosystem. Many AWS-centric teams pick CDK for the developer experience; multi-cloud teams standardize on Terraform.

🏋️ Practice: Using Terraform (or CDK), define a small stack: a VPC with public/private subnets and an S3 bucket. Configure remote state in S3 with a DynamoDB lock. Run plan, review it, then apply. Change a resource by hand in the console and re-run plan to see the drift. Refactor the network into a reusable module/construct and instantiate it for both a dev and a prod environment with different variables.

🎯 Key Takeaways

  • IaC defines infrastructure declaratively in version control → reproducible, reviewable, auditable, automatable, drift-free environments.
  • Tools: CloudFormation (AWS-native, managed state), CDK (real languages → CloudFormation), Terraform (cloud-agnostic, you manage state), Pulumi (multi-cloud, real languages).
  • State maps code to real resources; Terraform state must be remote + locked (S3 + DynamoDB), never committed; CloudFormation/CDK have AWS manage it.
  • Always review the plan before apply; keep secrets in Secrets Manager/SSM, not templates; pin provider/module versions.
  • Structure with modules/constructs per concern and per environment to enable reuse and limit blast radius; make all changes through code to avoid drift.
