ECS / EKS / Fargate Intermediate

Once you've containerized an app (see the Docker track), AWS gives you several ways to run it in production. ECS is AWS's own container orchestrator, EKS is managed Kubernetes, and Fargate is a serverless launch type that runs containers without you managing any servers. The interview question is always: which do you pick, and why?

The Landscape

ServiceWhat it is
ECRElastic Container Registry — AWS's private Docker image registry (push/pull images).
ECSAWS-native orchestrator: define tasks/services, it schedules and scales them. Simpler, deeply integrated.
EKSManaged Kubernetes control plane. Portable, huge ecosystem, more moving parts.
FargateA serverless launch type for ECS and EKS — no EC2 nodes to manage; you pay per task's vCPU/memory.
🧱 Fargate is a launch type, not a competitor to ECS/EKS. ECS and EKS are the orchestrators (how you define and schedule containers); the launch type is where they run: EC2 (you manage the node fleet, cheaper at scale, full control) or Fargate (AWS runs the compute, no servers, pay per task). So the two real questions are separate: (1) ECS vs EKS for orchestration, and (2) EC2 vs Fargate for the compute.

ECS Concepts

Task Definition   → the blueprint: which image(s), CPU/memory, ports, env, IAM task role, logging
Task              → a running instance of a task definition (one or more containers together)
Service           → keeps N tasks running, integrates with an ALB, does rolling deploys & scaling
Cluster           → logical grouping; runs on EC2 capacity or Fargate
// Very similar mental model to Kubernetes Deployments/Pods, but AWS-native and simpler.

Team is small / not K8s-experiencedYou have K8s expertise / existing K8s tooling

🎯 Default to ECS+Fargate for simplicity; choose EKS for Kubernetes portability and ecosystem. ECS with Fargate is the lowest-operational-overhead way to run containers on AWS — no control plane to think about, no nodes to patch. Reach for EKS when you specifically want Kubernetes: multi-cloud portability, existing K8s skills/tooling, or the CNCF ecosystem. Both can run on Fargate to eliminate node management.

Interview Questions

Q1: ECS vs EKS — how do you decide?

Both run containers, but ECS is AWS's own orchestrator while EKS is managed Kubernetes, and the decision comes down to whether you specifically need Kubernetes. I choose ECS when I want the simplest path with the least operational overhead and deep AWS integration — its task/service/cluster model is straightforward, there's no control plane or K8s version churn to manage, and it's a great fit for teams that aren't Kubernetes experts and are happy being all-in on AWS. I choose EKS when I want Kubernetes for its own sake: portability and avoiding lock-in (the same manifests can run elsewhere), an existing investment in K8s skills and tooling, or the CNCF ecosystem (Helm, operators, CRDs, service meshes). The trade-off is that EKS brings more moving parts and operational complexity. A common default is ECS with Fargate for simplicity, escalating to EKS when the Kubernetes ecosystem or portability genuinely adds value.

Q2: What exactly is Fargate, and when would you not use it?

Fargate is a serverless compute launch type for containers that works under both ECS and EKS: instead of provisioning, patching, and scaling a fleet of EC2 nodes to run your tasks/pods, you just specify each task's CPU and memory and AWS runs it on managed infrastructure, billing per task's resources with no idle nodes. It removes node management entirely, which is ideal for small teams, variable or spiky workloads, and anyone who wants minimal ops. You'd choose the EC2 launch type instead when you have large, steady, high-utilization fleets where bin-packing your own instances (and using Spot/Reserved pricing) is cheaper, when you need special hardware like GPUs or specific instance types, or when you need node-level control (privileged daemons, custom kernels). So Fargate optimizes for operational simplicity and elasticity; EC2 launch type optimizes for cost control at scale and low-level control.

Q3: How should a container on ECS/EKS get AWS permissions?

Through an IAM role attached to the task, never by baking access keys into the image or environment. In ECS you assign a task role to the task definition, and the container automatically receives temporary, auto-rotating credentials scoped to exactly the permissions it needs (e.g., read one S3 bucket, write one DynamoDB table); in EKS the equivalent is IRSA (IAM Roles for Service Accounts) or EKS Pod Identity, which maps a Kubernetes service account to an IAM role. This follows least privilege and the same "roles over long-lived keys" principle as everywhere else in AWS — there are no static secrets to leak or rotate, and each workload gets only the access it requires. Baking keys into images is the classic mistake because images get shared and end up in registries and logs, leaking standing credentials.

🏋️ Practice: Push a container image to ECR (enable image scanning). Create an ECS cluster and a Fargate service (desired=2, across two AZs) behind an ALB, using a task role that grants read access to one DynamoDB table (no keys in the image). Add service auto-scaling on CPU and do a rolling deploy of a new image version. Then compare: estimate the cost of running the same steady fleet on the EC2 launch type with Spot capacity.

🎯 Key Takeaways

  • ECR = private image registry; ECS = AWS-native orchestrator; EKS = managed Kubernetes; Fargate = serverless launch type under ECS/EKS.
  • Two independent choices: ECS vs EKS (orchestrator) and EC2 vs Fargate (compute). Fargate is not a competitor to ECS/EKS.
  • Default to ECS+Fargate for simplicity and low ops; choose EKS when you specifically need Kubernetes portability, skills, or the CNCF ecosystem.
  • EC2 launch type is cheaper at large steady scale (bin-packing + Spot/Reserved) and gives hardware/node control; Fargate wins for spiky load and minimal ops.
  • Give containers permissions via the ECS task role / EKS IRSA (temporary creds, least privilege) — never bake keys into images; spread across AZs and auto-scale.
