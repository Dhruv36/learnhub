EC2 & Auto Scaling Basics

EC2 (Elastic Compute Cloud) is AWS's virtual servers — the compute foundation everything else sits on or replaces. The real skill isn't launching one instance; it's building a fleet that scales with load and survives failures: a Load Balancer in front, an Auto Scaling Group behind it across multiple AZs, and the right purchasing model to keep the bill sane. That trio is the classic production compute pattern.

Anatomy of an EC2 Instance

PieceWhat it is
AMIAmazon Machine Image — the template (OS + preinstalled software) an instance boots from.
Instance typeThe hardware profile: family + size (e.g., t3.micro, m6i.large, c7g.xlarge). Families target general/compute/memory/GPU.
EBS volumeNetwork-attached persistent block storage (the instance's "disk"). Survives stop/start.
Security GroupA stateful virtual firewall on the instance's network interface (covered in VPC lesson).
User dataA boot script that runs on first launch (bootstrap the app, join the fleet).
💾 Instance store vs EBS: some instance types have local "instance store" disks that are ephemeral — data is lost when the instance stops or terminates. EBS volumes are network-attached and persist. Put anything you care about on EBS (or better, in S3/RDS), never on instance store.

The Production Trio: ALB + ASG across AZs

Users → Application Load Balancer (spans AZ-a, AZ-b, AZ-c)
             │  health checks each target; routes only to healthy ones
             ▼
        Auto Scaling Group  (min=2, desired=3, max=10)
          ├─ EC2 in AZ-a
          ├─ EC2 in AZ-b
          └─ EC2 in AZ-c
// The ASG launches/terminates instances from a Launch Template to hit the desired count,
// replaces unhealthy ones automatically, and scales on a policy (e.g., target 50% CPU).
⚖️ Auto Scaling gives you two things at once: elasticity (match capacity to demand so you're not over-provisioned) and self-healing (a failed instance is detected via health checks and replaced). Always span multiple AZs so one data-center outage just removes a fraction of capacity instead of taking you down.

// Target tracking (the easy default): "keep average CPU at 50%" — ASG adds/removes to hold the target.
// Step scaling:   +2 instances if CPU > 70% for 5 min, +4 if > 90% …
// Scheduled:      scale up to 20 every weekday 9am (predictable business-hours load).
// Predictive:     ML forecasts load and provisions ahead of the curve.

Interview Questions

Q1: How do you make an EC2-based web app highly available and elastic?

Put the instances behind an Application Load Balancer and manage them with an Auto Scaling Group that spans at least two (ideally three) Availability Zones. The ALB health-checks each instance and routes traffic only to healthy ones; the ASG maintains a desired count, replaces any instance that fails a health check (self-healing), and scales the count up or down based on a policy — most simply target tracking, e.g., "hold average CPU at 50%." Spreading across AZs means a single data-center failure only removes a slice of capacity rather than the whole service, and the ASG will launch replacements in the healthy AZs. The final requirement is that the instances be stateless — session/state lives in ElastiCache, DynamoDB, RDS, or S3 — so any instance can be added or removed without losing data. That combination gives you both elasticity (capacity follows demand) and resilience (failures are auto-healed).

Q2: When would you use Spot instances, and what's the catch?

Spot instances use AWS's spare capacity at up to ~90% off On-Demand, so they're ideal for fault-tolerant, interruptible, stateless workloads: batch processing, CI/CD runners, big-data jobs, rendering, and stateless web/worker fleets that can lose a node gracefully. The catch is that AWS can reclaim a Spot instance whenever it needs the capacity back, giving only a two-minute interruption notice, and prices/availability vary. So you must design for interruption — checkpoint work, drain/finish gracefully on the termination notice, keep no critical local state, and diversify across multiple instance types and AZs (and often mix Spot with some On-Demand/Reserved base capacity via an ASG's mixed-instances policy). You would not put a primary database or a stateful singleton on pure Spot.

Q3: ALB vs NLB — how do you choose?

An Application Load Balancer operates at Layer 7 (HTTP/HTTPS), so it understands requests and can route by host and path, terminate TLS, support WebSockets and HTTP/2, do content-based routing to different target groups, and integrate with WAF — it's the default for web apps and microservices. A Network Load Balancer operates at Layer 4 (TCP/UDP), so it's protocol-agnostic and optimized for extreme throughput and very low latency, preserves the client source IP, can provide static/Elastic IPs, and handles non-HTTP protocols. Choose ALB when you need HTTP-aware routing and web features; choose NLB when you need raw TCP/UDP performance, millions of connections, static IPs, or to expose non-HTTP services. In practice many architectures use an ALB for the app tier and an NLB where they need L4 characteristics.

🏋️ Practice: Create a Launch Template, then an Auto Scaling Group (min 2, max 6) across two AZs behind an Application Load Balancer with a health check on /health. Add a target-tracking policy at 50% CPU and generate load to watch it scale out and back in. Terminate an instance by hand and watch the ASG replace it. Finally, add a Spot allocation to the ASG's mixed-instances policy and observe the cost difference.

🎯 Key Takeaways

  • EC2 = virtual servers: AMI (template) + instance type (hardware) + EBS (persistent disk) + security group + user-data bootstrap.
  • Production compute = ALB in front + Auto Scaling Group across multiple AZs → elasticity plus self-healing.
  • Purchasing: On-Demand for spikes, Savings Plans/Reserved for steady baseline (~up to 72% off), Spot for interruptible work (~up to 90% off, 2-min reclaim).
  • Keep instances stateless (state in RDS/S3/ElastiCache/DynamoDB) so any instance can be added or replaced freely.
  • ALB = L7 HTTP routing; NLB = L4 high-throughput/static-IP; scale via target-tracking, step, scheduled, or predictive policies.
