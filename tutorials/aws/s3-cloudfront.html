S3 & CloudFront Basics

S3 (Simple Storage Service) is object storage — effectively infinite, 11-nines-durable, and the default home for files, backups, data lakes, and static sites on AWS. CloudFront is the CDN that puts that content (and your dynamic APIs) close to users at edge locations worldwide. Together they're the cheapest, most durable way to serve content at scale — and S3 misconfiguration is also the most famous cause of data breaches, so security matters here.

Object Storage vs Block/File

TypeServiceNature
ObjectS3Whole objects by key in a flat bucket; HTTP API; infinite scale; not a filesystem
BlockEBSA raw disk attached to one instance; low-latency; you put a filesystem on it
FileEFS / FSxA shared network filesystem (NFS/SMB) mountable by many instances
🗂️ S3 is not a filesystem. It stores objects addressed by a key in a flat namespace — the "folders" you see are just key prefixes (logs/2026/01/file.gz). You can't append to or partially edit an object; you replace it. This model is what lets S3 scale essentially without limit and hit 99.999999999% (11 nines) durability by replicating across AZs.

Storage Classes (match access pattern to cost)

ClassFor
S3 StandardFrequently accessed, hot data
S3 Intelligent-TieringUnknown/changing access — auto-moves objects between tiers; safe default
Standard-IA / One Zone-IAInfrequent access (cheaper storage, retrieval fee); One Zone = less durable, cheaper
Glacier Instant / Flexible / Deep ArchiveArchival — from ms retrieval down to hours, cheapest storage
💰 Lifecycle policies do the work for you: automatically transition objects Standard → IA → Glacier as they age, and expire them after N days. This is the main lever for S3 cost. When you don't know the access pattern, Intelligent-Tiering auto-optimizes with no retrieval-fee surprises.

S3 Security: the part that makes headlines

// Layers of access control (all default to PRIVATE):
//   Block Public Access      → account/bucket switch that overrides everything; leave ON unless truly public
//   Bucket policy            → resource-based policy (who/what can access this bucket)
//   IAM policies             → identity-based (what a user/role may do)
//   ACLs                     → legacy, mostly disabled now
//   Encryption               → SSE-S3 / SSE-KMS at rest (on by default); enforce TLS in transit
//   Presigned URLs           → time-limited signed link to a private object (upload/download without public access)
🚨 The classic breach: a public bucket. Buckets are private by default and "Block Public Access" is on by default — leave it on. Serve web content through CloudFront with Origin Access Control so the bucket itself stays private; use presigned URLs for temporary user access. Never make a data bucket public to "make it work."

CloudFront: the CDN Layer

User → nearest Edge Location ──cache hit?──► serve instantly from edge
                                    │miss
                                    ▼
                          Origin (S3 bucket, ALB, or any HTTP server) → cache & serve
// Benefits: lower latency (content near users), less origin load, TLS termination at the edge,
// DDoS protection (AWS Shield), WAF integration, and edge compute (CloudFront Functions / Lambda@Edge).
⚡ Cache behavior is everything. Tune TTLs and cache keys: cache static assets aggressively (long TTL, fingerprinted filenames), and be deliberate about which headers/cookies/query strings are part of the cache key (too many = poor hit ratio). Invalidate paths on deploy, or better, version asset filenames so you never need to invalidate.

Static Website Hosting Pattern

S3 bucket (private, holds the built site)  ◄── Origin Access Control
        ▲
   CloudFront distribution (HTTPS via ACM cert, caching, custom domain)
        ▲
   Route 53 (your-domain.com → the CloudFront distribution)
// The standard way to host a SPA/static site: cheap, fast, HTTPS, globally cached, bucket stays private.

Interview Questions

Q1: How would you host a secure, fast static website / SPA on AWS?

Store the built assets in a private S3 bucket (Block Public Access on) and serve them through a CloudFront distribution using Origin Access Control, so the bucket itself is never public — CloudFront is the only thing allowed to read it. CloudFront provides HTTPS (with an ACM certificate), global edge caching for low latency, TLS termination, and DDoS/WAF protection. Route 53 maps the custom domain to the distribution. For caching, I fingerprint asset filenames (e.g., app.abc123.js) and set long TTLs so browsers and edges cache aggressively, while the HTML has a short TTL; on deploy either invalidate or rely on the changed filenames. This gives a cheap, globally fast, HTTPS site with the origin bucket kept private — the standard, secure pattern.

Q2: How do you give a user temporary access to a private S3 object?

Use a presigned URL. The application, using credentials that do have permission on the object, generates a URL that embeds a signature and an expiry; anyone with that URL can perform the specific action (GET to download, or PUT to upload) until it expires, without the bucket being public and without the user having any AWS credentials. This is how you let a user download their private file or upload directly to S3 from the browser (offloading the bytes from your servers) while keeping the bucket locked down. You scope it tightly — a single object, a short expiry, the exact method — so the temporary grant is minimal. It's far safer than loosening the bucket policy, and it leverages the same identity's permissions rather than creating public access.

Q3: Explain S3 storage classes and how you'd optimize cost.

S3 offers classes tuned to access frequency: Standard for hot data; Standard-IA/One Zone-IA for infrequently accessed data (cheaper storage, a retrieval fee, One Zone trading durability for price); and the Glacier family (Instant, Flexible, Deep Archive) for archival, trading retrieval latency (milliseconds to hours) for the lowest storage cost. To optimize, I attach lifecycle policies that automatically transition objects to colder classes as they age and expire them when no longer needed — e.g., logs go Standard → IA after 30 days → Glacier after 90 → deleted after a year. When the access pattern is unknown or changing, I use Intelligent-Tiering, which automatically moves objects between tiers based on actual access with no retrieval fees, so I don't get surprised by IA/Glacier retrieval charges. The principle is matching each object's storage class to how often it's actually read.

🏋️ Practice: Create a private bucket (confirm Block Public Access is on) and upload a file. Generate a presigned GET URL and open it in a browser. Add a lifecycle rule that moves objects to Standard-IA after 30 days and expires them after 365. Then build the static-site pattern: put an index.html in the bucket, create a CloudFront distribution with Origin Access Control, and load the site over HTTPS — verifying the bucket stays private (direct S3 URL is denied).

🎯 Key Takeaways

  • S3 = infinitely scalable, 11-nines-durable object storage; objects by key in a flat namespace (not a filesystem, no partial edits).
  • Buckets are private by default with Block Public Access on — keep it on; serve public content via CloudFront OAC, and use presigned URLs for temporary access.
  • Match storage class to access pattern (Standard → IA → Glacier); automate with lifecycle policies or Intelligent-Tiering to control cost.
  • CloudFront caches content at edge locations for low latency, offloads the origin, terminates TLS, and adds Shield/WAF; tune TTLs and cache keys for hit ratio.
  • Standard secure static-site stack: private S3 + CloudFront (OAC, ACM HTTPS) + Route 53; enable versioning on critical buckets.
