SELECT & Filtering Basics

SQL is declarative: you describe what data you want, and the database's query planner figures out how to get it. That mindset — plus understanding three-valued logic (NULL) and the logical order of execution — separates people who "know some SQL" from people who write correct, fast queries. Examples use PostgreSQL; concepts transfer to MySQL.

The Anatomy of SELECT

SELECT name, email, score              -- which COLUMNS (projection)
FROM   users                            -- which TABLE
WHERE  score >= 80 AND active = true    -- which ROWS (filter)
ORDER BY score DESC                      -- sort
LIMIT  10;                               -- cap results

-- SELECT * is fine for exploration; in application code, name columns explicitly
-- (stable across schema changes, less data over the wire, index-only scans possible).

The Logical Order of Execution (the key mental model)

You WRITE:    SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ... ORDER BY ... LIMIT
DB EVALUATES: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
                                                    ↑
-- This explains two classic confusions:
-- 1. You CAN'T use a SELECT alias in WHERE (WHERE runs before SELECT):
--      SELECT price*qty AS total FROM x WHERE total > 100   -- ❌ 'total' unknown in WHERE
--      ...WHERE price*qty > 100                              -- ✅ (ORDER BY can use the alias)
-- 2. WHERE filters ROWS (before grouping); HAVING filters GROUPS (after) — aggregates lesson.

Filtering — the WHERE Toolkit

WHERE score BETWEEN 80 AND 100          -- inclusive range
WHERE role IN ('admin', 'editor')       -- set membership
WHERE name LIKE 'A%'                     -- pattern: % = any chars, _ = one char (ILIKE = case-insensitive)
WHERE created_at >= '2026-01-01'         -- dates compare naturally
WHERE email IS NOT NULL                  -- NULL check (see below — NOT = 'value')

NULL — Three-Valued Logic (the #1 SQL trap)

-- NULL means "UNKNOWN", not zero and not empty string. Comparisons with NULL yield UNKNOWN:
WHERE email = NULL        -- ❌ never matches ANYTHING (NULL = NULL is UNKNOWN, not true)
WHERE email IS NULL       -- ✅ the correct test
WHERE email IS NOT NULL   -- ✅

-- UNKNOWN propagates and surprises:
score > 50                -- rows where score IS NULL are EXCLUDED (comparison is UNKNOWN, not true)
NOT (score > 50)          -- still excludes NULL scores!
COALESCE(discount, 0)     -- replace NULL with a default
score = 5 OR score IS NULL -- must handle NULL explicitly when you want it
🕳️ Why NULL is three-valued: a comparison can be TRUE, FALSE, or UNKNOWN. WHERE keeps only TRUE rows, so any row where the condition is UNKNOWN (because a NULL was involved) is dropped — even under NOT. This silently changes result counts. Always reason about "what happens to NULLs?" in every filter. It's the single most common source of "missing rows" bugs.

DISTINCT & Basic Expressions

SELECT DISTINCT country FROM users;                    -- unique values
SELECT name, score * 1.1 AS adjusted FROM users;       -- computed column with alias
SELECT CASE WHEN score >= 90 THEN 'A' WHEN score >= 80 THEN 'B' ELSE 'C' END AS grade FROM users;
SELECT UPPER(name), LENGTH(email), COALESCE(nickname, name) FROM users;   -- functions

Interview Questions

Q1: Explain the logical order of execution and one bug it prevents.

SQL evaluates FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT, even though you write SELECT first. Knowing this explains why a SELECT alias isn't available in WHERE (WHERE runs before SELECT computes the alias) but IS available in ORDER BY (which runs after SELECT); why WHERE can't reference aggregates (grouping hasn't happened) but HAVING can; and why filtering in WHERE (pre-aggregation) is different from HAVING (post-aggregation). It's the mental model that makes SQL predictable rather than trial-and-error.

Q2: What is NULL, and how does it break naive queries?

NULL means "unknown", producing three-valued logic: comparisons involving NULL yield UNKNOWN, not TRUE/FALSE. WHERE keeps only TRUE rows, so any condition touching a NULL is treated as not-matched — including under NOT (NOT UNKNOWN is still UNKNOWN). So WHERE score > 50 and WHERE NOT (score > 50) BOTH exclude NULL scores, and email = NULL never matches. You must use IS NULL/IS NOT NULL, and COALESCE to supply defaults, and consciously decide whether NULLs should be included in each filter — the most common cause of "rows mysteriously missing".

Q3: When is SELECT * a problem?

In application code: it couples the app to column order/count (breaks when the schema changes), ships unnecessary data (bandwidth, memory), prevents covering/index-only scans (the planner must hit the table for all columns), and can break with joins that introduce duplicate column names. It's fine for ad-hoc exploration. Production queries name exactly the columns needed — smaller, stabler, and often faster.

🏋️ Practice (any Postgres/MySQL, or sqlite): create a users table; write SELECTs with WHERE ranges, IN, LIKE; insert a row with a NULL column and prove = NULL misses it while IS NULL finds it; show that WHERE score > x drops NULL scores and fix with COALESCE; try using a SELECT alias in WHERE (see the error) then move the expression; use CASE to bucket scores into grades.

🎯 Key Takeaways

  • SQL is declarative; the planner decides HOW — you specify WHAT.
  • Logical order: FROM→WHERE→GROUP BY→HAVING→SELECT→ORDER BY→LIMIT — explains alias/aggregate rules.
  • NULL = unknown → three-valued logic; comparisons with NULL are UNKNOWN and get filtered out (even under NOT). Use IS NULL / COALESCE.
  • WHERE toolkit: BETWEEN, IN, LIKE/ILIKE, IS NULL; CASE for conditional columns.
  • Name columns in app code (not SELECT *); anchor LIKE patterns for index use.
