Subqueries & CTEs Intermediate

Subqueries nest a query inside another; CTEs (Common Table Expressions, the WITH clause) name them for readability and enable recursion. These turn complex questions into composable steps. The nuances: correlated subqueries (which can be slow) and knowing when a JOIN is better than a subquery.

Subquery Forms

-- Scalar (returns one value):
SELECT name, score, (SELECT AVG(score) FROM users) AS avg_all FROM users;

-- IN / NOT IN (a set):
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders WHERE total > 100);

-- EXISTS (existence check — often faster, NULL-safe):
SELECT * FROM users u WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- Derived table (subquery in FROM):
SELECT dept, avg_sal FROM (SELECT dept, AVG(salary) avg_sal FROM emp GROUP BY dept) t
WHERE avg_sal > 50000;
⚠️ NOT IN + NULL trap: if the subquery returns ANY NULL, NOT IN returns no rows at all (because x NOT IN (1, NULL) is UNKNOWN, never TRUE — three-valued logic again). Prefer NOT EXISTS (NULL-safe) or add WHERE col IS NOT NULL to the subquery. This is a subtle, common bug.

Correlated Subqueries (per-row, watch performance)

-- A CORRELATED subquery references the outer row → it runs once PER outer row:
SELECT u.name,
       (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS order_count  -- per-user
FROM users u;
-- Readable, but potentially O(n) subquery executions. The planner often optimizes it, but for
-- large data a JOIN + GROUP BY (set-based) is usually faster:
SELECT u.name, COUNT(o.id) FROM users u LEFT JOIN orders o ON o.user_id=u.id GROUP BY u.id, u.name;
-- EXISTS is correlated too but short-circuits at the first match — efficient for "has any".

CTEs — Name Your Steps

WITH high_spenders AS (
    SELECT user_id, SUM(total) AS spend
    FROM orders GROUP BY user_id HAVING SUM(total) > 1000
),
recent AS (
    SELECT user_id FROM orders WHERE created_at > now() - interval '30 days'
)
SELECT u.name, hs.spend
FROM users u
JOIN high_spenders hs ON hs.user_id = u.id
JOIN recent r        ON r.user_id = u.id;
-- CTEs read top-to-bottom like a pipeline — far clearer than deeply nested subqueries.
-- Note: in some engines CTEs are an "optimization fence" (materialized); modern Postgres
-- inlines them by default (or use MATERIALIZED / NOT MATERIALIZED to control it).

Recursive CTEs — Hierarchies & Graphs

WITH RECURSIVE subordinates AS (
    SELECT id, name, manager_id FROM employees WHERE id = :ceo_id   -- anchor (base case)
    UNION ALL
    SELECT e.id, e.name, e.manager_id
    FROM employees e JOIN subordinates s ON e.manager_id = s.id     -- recursive step
)
SELECT * FROM subordinates;   -- the whole org tree under the CEO
-- Recursive CTEs traverse trees/graphs (org charts, category trees, bill-of-materials,
-- shortest paths). The anchor seeds; the recursive part joins back to the CTE until no new rows.

Subquery vs JOIN — Which?

Use a subquery/EXISTS when…Use a JOIN when…
Checking existence ("has any order")You need COLUMNS from both tables
Filtering by an aggregate of another tableCombining/enriching rows
Readability of a clear step (CTE)Set-based aggregation over relations (often faster)

Interview Questions

Q1: Correlated vs non-correlated subquery, and the performance implication.

A non-correlated subquery is independent — it runs once and its result is reused. A correlated subquery references the outer query's current row, so conceptually it runs once per outer row (O(n) executions) — readable but potentially slow on large data. Planners often optimize simple cases, but for aggregations the set-based JOIN + GROUP BY is usually faster and clearer. EXISTS is correlated yet efficient because it short-circuits at the first matching row rather than counting all of them.

Q2: Why prefer NOT EXISTS over NOT IN?

NOT IN breaks with NULLs: if the subquery returns any NULL, x NOT IN (..., NULL) is UNKNOWN for every x (three-valued logic), so the whole query returns no rows — a silent, surprising bug. NOT EXISTS is NULL-safe (it checks row existence, not value equality) and often optimizes better because it can stop at the first match. Use NOT EXISTS for anti-joins, or guarantee the NOT IN subquery has no NULLs.

Q3: When would you use a recursive CTE?

For hierarchical or graph-structured data: traversing an org chart (all reports under a manager), category/menu trees, bill-of-materials explosions, threaded comments, or graph reachability/ shortest paths. Structure: an anchor query (the starting rows) UNION ALL a recursive query that joins the employees/edges table back to the CTE, repeating until no new rows are produced. For cyclic graphs, add a visited-path array or depth limit to prevent infinite recursion.

🏋️ Practice: write a scalar subquery adding the global average to each row; find users with orders via IN and EXISTS; reproduce the NOT IN + NULL zero-rows bug and fix with NOT EXISTS; convert a correlated count subquery to a JOIN + GROUP BY; build a multi-step CTE pipeline; write a recursive CTE to list an org tree under a manager.

🎯 Key Takeaways

  • Subqueries: scalar, IN/EXISTS, derived tables; EXISTS short-circuits and is NULL-safe.
  • NOT IN with a NULL in the subquery returns nothing — use NOT EXISTS.
  • Correlated subqueries run per outer row (can be slow) — prefer JOIN + GROUP BY for aggregation.
  • CTEs (WITH) name steps for readable pipelines; recursive CTEs traverse trees/graphs.
  • JOIN when you need columns from both; subquery/EXISTS for existence/aggregate filters.
