Query Optimization Advanced

Slow database queries are the #1 cause of slow applications. The optimization skill is a repeatable workflow — find the slow queries, read their plans, apply the right fix — plus a catalogue of the fixes that solve 95% of real cases. This lesson is that catalogue.

The Workflow

1. FIND the slow queries — don't guess:
   Postgres: pg_stat_statements (total/mean time per query shape), slow-query log
   APM traces (which endpoint → which query)
2. EXPLAIN ANALYZE the worst offender (indexes lesson: Seq Scans, estimate gaps, sorts)
3. Apply the matching fix (below); re-measure. One change at a time.
4. Guard: statement_timeout so a bad query can't hold the fleet hostage.

Rewrites That Unlock Plans

-- Sargable predicates (Search ARGument ABLE — can use an index):
WHERE created_at >= '2026-01-01' AND created_at < '2026-02-01'  -- ✅
WHERE DATE(created_at) = '2026-01-15'                            -- ❌ function on column
WHERE score > 90                                                 -- ✅  (not score+10 > 100)

-- EXISTS instead of IN for big correlated sets; NOT EXISTS instead of NOT IN (NULL trap):
SELECT * FROM users u WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- Pre-aggregate BEFORE joining (avoid fan-out, shrink the join input):
WITH order_totals AS (SELECT user_id, SUM(total) t FROM orders GROUP BY user_id)
SELECT u.name, ot.t FROM users u JOIN order_totals ot ON ot.user_id = u.id;

-- UNION ALL (no dedup) over UNION when duplicates are impossible/irrelevant.

Materialized Views — Precomputed Reads

CREATE MATERIALIZED VIEW daily_revenue AS
SELECT date_trunc('day', created_at) d, SUM(total) revenue FROM orders GROUP BY 1;
REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue;   -- on a schedule / after loads
-- A stored query RESULT (unlike a view, which re-runs). Perfect for dashboards/aggregates
-- that tolerate staleness — the denormalization trade (schema lesson) with built-in refresh.

Connection & Server-Side Levers

-- Connection pooling (app-side or PgBouncer): opening connections is expensive; too many
--   connections thrashes the DB. Pool sized to the DB's capacity (Node/Java lessons).
-- statement_timeout / lock_timeout: bound worst-case damage.
-- work_mem: sorts/hashes spill to disk when it's too small (EXPLAIN shows "external sort").
-- Read replicas for read-heavy scaling; caching layers before the DB (System Design track).

Interview Questions

Q1: An endpoint got slow this week. Walk your full diagnosis, DB included.

Start at the endpoint: APM/trace shows time split (app vs DB) and the queries issued — first check query COUNT per request (N+1 shows as many fast queries; fix with a join/batch). If it's one slow query: pg_stat_statements to confirm its cost, then EXPLAIN ANALYZE — hunt Seq Scans (missing/ defeated index), estimate-vs-actual gaps (run ANALYZE), sort spills (work_mem/index). Check what changed this week: data growth crossing a plan threshold, a dropped index, stale stats after a bulk load. Fix one variable, re-measure, and add a regression guard (statement_timeout + the query in a dashboard).

Q2: What makes a predicate sargable, and why does it matter?

Sargable = the predicate can use an index seek: the bare column on one side of a comparison with a constant/parameter range — email = :x, created_at >= :from, name LIKE 'a%'. Wrapping the column in a function (LOWER(email)), arithmetic (score+10 > 100), casts, or a leading wildcard forces the engine to evaluate the expression per row — full scan, index useless. Fixes: move computation to the other side (score > 90), rewrite date functions as ranges, or create an expression index matching the function. One line of predicate style is often the entire difference between 5ms and 5s.

Q3: When do you cache/materialize instead of optimizing the query?

When the query is inherently expensive (large aggregates, multi-way joins) AND the result tolerates staleness. Options by freshness need: app/Redis cache with TTL (seconds-minutes; cheapest), materialized view refreshed on schedule (dashboard aggregates), counter-cache/denormalized columns maintained on write (always-fresh reads, write cost), or moving the work off the request path entirely (async job → stored result). The discipline: define acceptable staleness first, pick the cheapest layer that meets it, and design invalidation explicitly — cache bugs come from skipping that step. This is the System Design caching lesson applied to SQL.

🏋️ Practice: enable pg_stat_statements and find your top query by total time; make a non-sargable date filter, watch the Seq Scan, rewrite as a range; reproduce N+1 in a script (log query count), fix with one JOIN; build a materialized view for a daily aggregate and REFRESH it; set statement_timeout and watch a runaway query get killed.

🎯 Key Takeaways

  • Workflow: find (pg_stat_statements/APM) → EXPLAIN ANALYZE → one fix → re-measure; timeouts as guardrails.
  • The big four fixes: missing/defeated indexes, N+1 batching, sargable rewrites, keyset pagination.
  • Pre-aggregate before joins; EXISTS/NOT EXISTS; UNION ALL when dedup isn't needed.
  • Cache or materialize expensive tolerant-of-staleness reads — with explicit invalidation.
  • Pool connections, bound timeouts, size work_mem; replicas/caches before heroic query tuning.
