Scaling: Replication, Partitioning, Sharding Ultra-Advanced

One database server eventually isn't enough. The escalation ladder — replicas for reads, partitioning for big tables, sharding for writes — is the capstone of this track and the bridge to the System Design track's scaling lessons. Each step buys capacity and costs complexity; senior judgment is knowing when each is actually needed.

Step 0 — Exhaust the Single Node First

Indexes + query optimization + caching + connection pooling + bigger hardware
carry most products further than expected — a well-tuned Postgres handles tens of
thousands of QPS. Distributed complexity is a LAST resort, not a first move.

Read Replicas — Scale Reads

            writes                    async WAL streaming
  App ────────────────▶ PRIMARY ────────────────────────▶ REPLICA(s)
        reads (SELECTs) ─────────────────────────────────▶   ↑ read-only
-- The primary streams its write-ahead log; replicas replay it. Most apps are read-heavy
-- (10:1+), so replicas absorb the bulk of traffic and double as failover targets.
⏱️ Replication LAG is the catch: async replicas trail the primary by milliseconds-to-seconds. A user who writes then immediately reads from a replica may not see their own write ("read-your-writes" violation — profile edit seems to vanish). Fixes: route reads-after-writes to the primary (sticky session for N seconds), or sync replication for critical reads (latency cost). This trade-off IS the System Design track's replication lesson in practice.

Partitioning — Split a Big Table WITHIN One Database

CREATE TABLE events (id BIGINT, created_at TIMESTAMPTZ, ...) PARTITION BY RANGE (created_at);
CREATE TABLE events_2026_07 PARTITION OF events FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');
-- One logical table, many physical pieces. Queries filtering on the partition key scan only
-- the relevant partitions (PARTITION PRUNING); old data drops instantly (DROP PARTITION vs
-- million-row DELETEs). Range (time-series) is most common; also LIST and HASH.
-- Partitioning helps BIG TABLES (manageability + pruning) — it does NOT add write capacity;
-- it's still one server.

Sharding — Split Data ACROSS Databases (the last resort)

shard = hash(user_id) % N        →  users 0-24 on DB-A, 25-49 on DB-B, ...
-- Each shard is an independent database holding a slice of the data. Writes finally scale
-- horizontally. THE COSTS (why it's last):
--   • cross-shard queries/joins are hard (fan-out + merge in the app)
--   • cross-shard TRANSACTIONS lose easy ACID (sagas/2PC complexity)
--   • resharding when N changes is painful → consistent hashing (System Design lesson)
--   • hot keys (one celebrity user) can still overload one shard
-- Shard-key choice is EVERYTHING: pick the key most queries filter by (usually tenant/user id)
-- so queries stay single-shard. Managed options (Citus, Vitess, cloud-native DBs) before DIY.

The Escalation Ladder (the interview answer)

Interview Questions

Q1: Your database is at capacity. Walk the escalation ladder with reasons.

First identify WHICH resource is exhausted. Slow queries → indexes/rewrites/caching (cheapest, usually sufficient). Read-bound → add read replicas and a cache layer (most traffic is reads), handling replication lag with read-your-writes routing. Big unwieldy tables (slow maintenance, old-data purges) → partition by time/tenant for pruning and instant drops. Only when WRITE throughput or total data genuinely exceeds one primary → shard by a key that keeps queries single-shard, accepting cross-shard complexity. At each step, prefer managed offerings. The discipline: measure the actual bottleneck; never buy distributed-systems complexity you don't need.

Q2: Partitioning vs sharding — people conflate them. Distinguish precisely.

Partitioning splits one large table into physical pieces WITHIN a single database server: one logical table, transparent to queries, gains partition pruning (scan only relevant pieces) and instant old-data drops — but the server's total write capacity is unchanged. Sharding splits data ACROSS independent database servers, each owning a slice keyed by the shard key: writes scale horizontally, but the application (or middleware) must route queries, cross-shard joins/ transactions become hard, and resharding is a project. Partitioning = manageability on one node; sharding = capacity beyond one node.

Q3: How do you choose a shard key, and what goes wrong with a bad one?

Choose the key that (a) most queries filter by — so they hit ONE shard (tenant_id/user_id in SaaS), (b) has high cardinality and even distribution — so load spreads, and (c) co-locates data that transacts together — so transactions stay single-shard. Bad keys: timestamp for write-heavy data (all today's writes hammer one shard), low-cardinality fields (country → few giant shards), or a key orthogonal to queries (every request fans out to all shards). A hot single key (celebrity user) still overloads its shard — mitigations: key-splitting with suffixes, dedicated caches. And changing N later means resharding — consistent hashing minimizes the data moved (System Design track).

🏋️ Practice: set up a Postgres streaming replica (Docker) and demonstrate replication lag with a write-then-read; create a RANGE-partitioned events table, verify pruning with EXPLAIN, and drop an old partition instantly; sketch a sharding plan for a SaaS (shard key, routing, what breaks cross-shard); design the read-your-writes routing rule for a profile-edit flow.

🎯 Key Takeaways

  • Exhaust the single node first (indexes, caching, pooling, hardware) — distribution is a last resort.
  • Replicas scale READS; async lag breaks read-your-writes — route recent writers to the primary.
  • Partitioning = one server, pruned scans + instant drops for big tables; no extra write capacity.
  • Sharding scales WRITES across servers; shard-key choice (single-shard queries, even spread, co-located transactions) is everything.
  • The ladder: tune → cache → replicas → partition → shard; managed services before DIY. (Continue in the System Design track.)
