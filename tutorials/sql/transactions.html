Constraints & Transactions Intermediate

Constraints let the database enforce data rules so bad data can't get in. Transactions group statements into an all-or-nothing unit — the foundation of correctness for money, inventory, and any multi-step change. ACID is the guarantee, and it's a guaranteed interview topic.

Constraints — Rules the DB Enforces

ConstraintGuarantees
PRIMARY KEYUnique + NOT NULL row identity
UNIQUENo duplicate values (enables upserts; needed for one-to-one)
NOT NULLValue required
FOREIGN KEYReferenced row exists (referential integrity; ON DELETE CASCADE/RESTRICT)
CHECKCustom rule: CHECK (price >= 0), CHECK (status IN (...))
DEFAULTValue when none supplied
🛡️ Enforce invariants in the DATABASE, not just the app. The database is the last line of defense — multiple services, scripts, and future code all write to it, and an app-only check can be bypassed. A UNIQUE constraint prevents duplicate emails even under a race a SELECT-then-INSERT would lose; a CHECK guarantees no negative balances ever. Constraints turn "we hope the data is valid" into "invalid data cannot exist."

Transactions & ACID

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;   -- debit
UPDATE accounts SET balance = balance + 100 WHERE id = 2;   -- credit
COMMIT;    -- both apply together; if anything fails → ROLLBACK, NEITHER applies
-- Without a transaction, a crash between the two updates loses $100 (money debited, not credited).
ACIDMeans
AtomicityAll statements commit or none do (no partial results)
ConsistencyConstraints hold before and after; the DB never violates its rules
IsolationConcurrent transactions don't corrupt each other (isolation lesson)
DurabilityOnce committed, survives crashes (written to durable storage/WAL)

Savepoints & Error Handling

BEGIN;
INSERT INTO orders ...;
SAVEPOINT after_order;
INSERT INTO order_items ...;         -- if this fails:
ROLLBACK TO after_order;             -- undo just this part, keep the order; then retry/adjust
COMMIT;
-- Application code MUST handle failures: on any error, ROLLBACK. ORMs/frameworks usually wrap
-- a request in a transaction and roll back on exception — but know what's happening underneath.

The Concurrency Primitive: Atomic Updates

-- ❌ Read-modify-write in app code is a RACE (lost update):
--   bal = SELECT balance; app computes bal-100; UPDATE SET balance = bal-100;
--   two concurrent runs both read the same balance → one debit lost.
-- ✅ Let the DB do it atomically in one statement:
UPDATE accounts SET balance = balance - 100 WHERE id = 1 AND balance >= 100;
-- Check rows-affected = 1 to confirm sufficient funds. This + transactions + isolation
-- (next lesson) is how correct money handling works.

Interview Questions

Q1: Explain ACID with the money-transfer example.

Transferring $100 from A to B is two updates (debit A, credit B). Atomicity: both happen or neither — a crash mid-way rolls back so money is never half-moved. Consistency: constraints (e.g., balance >= 0) hold throughout; the DB never ends in a rule-violating state. Isolation: if another transfer runs concurrently, they don't interleave to corrupt balances (governed by the isolation level). Durability: once COMMIT returns, the change is on durable storage (WAL/redo log) and survives a crash. Together they make multi-step changes safe — which is why financial systems use relational databases.

Q2: Why enforce rules with constraints instead of application code?

The database is shared and permanent: multiple services, background jobs, admin scripts, and future code all write to it, and any of them can bypass an app-layer check. Constraints are enforced by the DB itself, so invalid data literally cannot be inserted regardless of the writer — and they're race-safe (a UNIQUE constraint blocks duplicate emails even when two SELECT-then-INSERT requests race, which app checks miss). App validation gives fast user feedback and rich messages; DB constraints are the guarantee. Use both, but treat the database as the source of truth for invariants.

Q3: How do you correctly decrement inventory/balance under concurrency?

Don't read-modify-write in application code (SELECT balance, compute, UPDATE) — two concurrent runs read the same value and one update is lost. Instead perform the change atomically in a single statement: UPDATE accounts SET balance = balance - :amt WHERE id = :id AND balance >= :amt, then check that exactly one row was affected (0 = insufficient funds). Wrap related changes in a transaction, and for multi-row invariants use appropriate isolation/locking (next lesson). The DB's atomic update is the primitive; the app checks the affected-row count for the business rule.

🏋️ Practice: add UNIQUE/CHECK/FK/NOT NULL constraints and try to violate each; run a two-statement money transfer in a transaction and ROLLBACK to prove atomicity; reproduce a lost-update by reading-then-writing in two sessions, then fix with an atomic UPDATE + affected-row check; use a SAVEPOINT to partially roll back; observe that a UNIQUE constraint blocks a duplicate insert a race would allow.

🎯 Key Takeaways

  • Constraints (PK, UNIQUE, NOT NULL, FK, CHECK, DEFAULT) enforce invariants in the DB — the last line of defense.
  • Transactions (BEGIN/COMMIT/ROLLBACK) make multi-step changes atomic; SAVEPOINT for partial rollback.
  • ACID: Atomicity, Consistency, Isolation, Durability — the correctness guarantee (money-transfer example).
  • Avoid read-modify-write races: do atomic UPDATEs (col = col - x WHERE ...) and check affected rows.
  • Keep transactions short (no I/O inside); always ROLLBACK on error; enforce rules in the DB, not only the app.
