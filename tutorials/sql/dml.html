INSERT, UPDATE, DELETE Basics

DML (Data Manipulation Language) changes data. The critical safety concept: UPDATE and DELETE without a WHERE clause hit EVERY row — the most famous production disaster in SQL. Beyond safety, the senior topics are upserts, RETURNING, and batching.

INSERT

INSERT INTO users (name, email, score) VALUES ('Asha', 'a@b.com', 92);
INSERT INTO users (name, email) VALUES              -- multi-row (ONE statement, fast):
    ('Bo', 'bo@x.com'), ('Cy', 'cy@x.com'), ('Di', 'di@x.com');
INSERT INTO archive (id, name) SELECT id, name FROM users WHERE inactive;  -- INSERT ... SELECT
INSERT INTO users (name) VALUES ('Ed') RETURNING id, created_at;  -- get generated values back (Postgres)

UPDATE — and the WHERE Discipline

UPDATE users SET score = score + 10 WHERE id = 42;      -- ✅ scoped
UPDATE users SET active = false WHERE last_login < '2025-01-01';
UPDATE users SET tier = 'gold', updated_at = now() WHERE spend > 1000;   -- multiple columns

UPDATE users SET score = 0;      -- 💀 NO WHERE → resets EVERY user's score. Irreversible without a backup.
🛑 The #1 SQL production disaster: UPDATE/DELETE without WHERE (or with a WHERE that matches more than intended). Defenses: (1) run a SELECT with the same WHERE first to see the affected rows; (2) wrap in a transaction and check the row count before COMMIT; (3) many teams enable "safe update mode" which rejects unqualified UPDATE/DELETE. Always know how many rows you're about to change.

DELETE (and TRUNCATE)

DELETE FROM users WHERE id = 42;
DELETE FROM sessions WHERE expires_at < now();     -- cleanup
TRUNCATE TABLE staging;      -- removes ALL rows, fast, resets sequences — but not transactional-friendly
                             -- in the same way & can't have a WHERE. DELETE for row-level; TRUNCATE to empty.
-- Soft delete (often preferred): keep the row, flag it —
UPDATE users SET deleted_at = now() WHERE id = 42;  -- recoverable, audit-friendly (needs query filters everywhere)

UPSERT — Insert-or-Update

-- Postgres: ON CONFLICT
INSERT INTO counters (key, n) VALUES ('visits', 1)
ON CONFLICT (key) DO UPDATE SET n = counters.n + 1;    -- increment or create atomically
INSERT INTO users (email, name) VALUES ('a@b.com', 'Asha')
ON CONFLICT (email) DO NOTHING;                         -- ignore duplicates
-- MySQL:  INSERT ... ON DUPLICATE KEY UPDATE n = n + 1;
-- Upserts need a UNIQUE constraint on the conflict target. They're the atomic, race-free way to
-- do "create if missing, else update" — better than SELECT-then-INSERT (which has a race window).

Batching & Performance

-- Multi-row INSERT (one round trip) beats N single INSERTs by a lot.
-- For huge loads: COPY (Postgres) / LOAD DATA (MySQL) — far faster than INSERTs.
-- Deleting millions of rows: batch in chunks to avoid long locks / huge transactions:
DELETE FROM logs WHERE created_at < :cutoff LIMIT 10000;   -- repeat until 0 rows affected
-- Wrap related changes in a transaction (transactions lesson) so they commit atomically.

Interview Questions

Q1: How do you safely run a large UPDATE in production?

First run a SELECT with the exact WHERE to see which/how many rows match. Then wrap the UPDATE in a transaction (BEGIN), execute it, verify the affected row count matches expectations, and only then COMMIT (ROLLBACK if wrong). For very large updates, batch in chunks (WHERE ... LIMIT n, repeat) to avoid long locks, oversized transactions, and replication lag. Consider running during low traffic and having a rollback/backup plan. The habit that prevents disasters: never issue an unqualified UPDATE/DELETE, and always know the row count.

Q2: Why is an upsert better than SELECT-then-INSERT?

SELECT-then-INSERT has a race window: two concurrent requests both SELECT "not found", both INSERT, and one fails on the unique constraint (or you get duplicates without one). An upsert (INSERT ... ON CONFLICT DO UPDATE/NOTHING, or MySQL's ON DUPLICATE KEY UPDATE) performs the check and the write atomically at the database level using a UNIQUE constraint, so it's race-free and a single round trip. It's the correct primitive for "create if missing, else update", counters, and idempotent writes.

Q3: Soft delete vs hard delete — trade-offs?

Hard delete (DELETE) removes the row — simplest, reclaims space, but destroys history and can cascade or orphan related data. Soft delete sets a deleted_at/is_deleted flag, keeping the row for audit, recovery, and referential history — but requires every query to filter out soft-deleted rows (easy to forget → "deleted" data reappears), grows the table, and complicates unique constraints. Choose per domain: soft delete for user-facing/regulated data needing recovery/audit; hard delete (or archival + delete) for transient data. Often paired with a background purge.

🏋️ Practice: multi-row INSERT then INSERT...SELECT into an archive; run an UPDATE inside a transaction, check the row count, and ROLLBACK; write an upsert that increments a counter with ON CONFLICT; implement soft delete + a query that excludes deleted rows; chunk-delete old rows in a loop; use RETURNING to get a generated id.

🎯 Key Takeaways

  • UPDATE/DELETE without WHERE hits every row — SELECT first, use a transaction, know the row count.
  • Multi-row INSERT / COPY for bulk; RETURNING to get generated values in one round trip.
  • Upsert (ON CONFLICT / ON DUPLICATE KEY) is the atomic, race-free "create-or-update" — needs a UNIQUE constraint.
  • Batch huge deletes/updates in chunks to avoid long locks and replication lag.
  • Soft delete (flag) for audit/recovery (filter everywhere) vs hard delete; wrap related writes in transactions.
