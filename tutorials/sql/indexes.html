Indexes & Query Plans Advanced

An index is the difference between scanning 50 million rows and jumping to 12 of them. Knowing how B-tree indexes work, which queries can use them, and how to read EXPLAIN output is the single highest-leverage database skill for application engineers — and the most reliable senior SQL interview topic.

How a B-tree Index Works

CREATE INDEX idx_users_email ON users(email);
-- A B-tree: a sorted, balanced tree over the column's values, each entry pointing to its row.
-- Lookup: walk the tree in O(log n) — a few page reads instead of scanning the whole table.
-- Think: the index at the back of a book vs reading every page.
-- The PRIMARY KEY and UNIQUE constraints automatically create indexes.
-- Cost: every INSERT/UPDATE/DELETE must also update every index → writes slow down,
-- storage grows. Indexes are a READ-vs-WRITE trade — don't index everything.

What a B-tree Index Can (and Can't) Serve

-- ✅ Uses idx(email):
WHERE email = 'a@b.com'            -- equality
WHERE email LIKE 'asha%'           -- LEFT-anchored prefix
WHERE created_at > '2026-01-01'    -- ranges (on an indexed column)
ORDER BY created_at DESC LIMIT 10  -- sorted access without a sort step

-- ❌ Can't use the index:
WHERE LOWER(email) = 'a@b.com'     -- function on the column (index stores raw values)
                                   --   fix: expression index → CREATE INDEX ON users(LOWER(email))
WHERE email LIKE '%gmail.com'      -- leading wildcard (no known prefix to descend the tree)
WHERE score + 10 > 100             -- arithmetic on the column (rewrite: score > 90)
WHERE status != 'active'           -- negations usually scan

Composite Indexes — the Leftmost-Prefix Rule

CREATE INDEX idx ON orders(user_id, status, created_at);
-- The index is sorted by user_id, THEN status, THEN created_at. It serves:
WHERE user_id = 5                                     -- ✅ (leftmost column)
WHERE user_id = 5 AND status = 'paid'                 -- ✅
WHERE user_id = 5 AND status = 'paid' AND created_at > X  -- ✅ (full use)
WHERE status = 'paid'                                 -- ❌ skips the leftmost column
WHERE user_id = 5 AND created_at > X                  -- partial (user_id only; created_at filtered after)
-- Rule: columns are usable LEFT to RIGHT; an equality chain then one range.
-- Order columns: most-selective equality first, ranges last.

Reading EXPLAIN (the workflow)

EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 5 AND status = 'paid';
-- Key things to read:
Seq Scan on orders  (cost=.. rows=2000000)     -- 💀 full table scan on a big table = missing index
Index Scan using idx on orders (rows=12)       -- ✅ tree walk + row fetches
Index Only Scan                                 -- ✅✅ answered ENTIRELY from the index (covering)
Bitmap Heap Scan                                -- middle ground for medium selectivity
rows=ESTIMATE vs actual rows                    -- huge mismatch → stale stats → run ANALYZE
Buffers: shared hit/read                        -- memory vs disk I/O
-- Workflow: slow query → EXPLAIN ANALYZE → find the Seq Scan / bad estimate / sort → add or
-- fix the index (or rewrite the predicate) → re-EXPLAIN and compare.

Covering, Partial & Special Indexes

-- COVERING (index-only): include all needed columns so the table is never touched:
CREATE INDEX idx ON orders(user_id) INCLUDE (total, created_at);   -- Postgres INCLUDE
-- PARTIAL: index only the rows you query (smaller, faster):
CREATE INDEX idx_active ON users(email) WHERE deleted_at IS NULL;
-- Other types: GIN (JSONB, arrays, full-text), GiST (geo), BRIN (huge append-only tables),
-- Hash (equality only). B-tree is the default and right answer ~95% of the time.

Interview Questions

Q1: How does a B-tree index make queries fast, and what does it cost?

It maintains the column's values in a sorted, balanced tree; a lookup descends from the root in O(log n) page reads to find matching entries, each pointing to its table row — versus O(n) scanning every row. Sorted order also serves ranges and ORDER BY without sorting. Costs: every write must update every index on the table (slower INSERT/UPDATE/DELETE), extra storage, and maintenance — so indexes are a deliberate read-optimization trade, added for measured query patterns, not sprinkled everywhere.

Q2: Explain the leftmost-prefix rule with an example.

A composite index on (user_id, status, created_at) is sorted by user_id first, then status within each user, then created_at. Queries can use it only for a LEFTMOST run of columns: filtering by user_id works; user_id+status works; status alone can't use it (the tree isn't organized to find statuses without a user_id). And after a range condition, later columns can't be used for descending the tree. Hence the design rule: put equality-filtered columns first (most selective early), range columns last, and create the index to match your real WHERE clauses.

Q3: A production query is slow. Walk your exact diagnosis process.

Get the real query and run EXPLAIN ANALYZE (with BUFFERS). Look for: a Seq Scan on a large table where a filter is selective (missing/unusable index — check for functions on columns, leading wildcards, type mismatches); a big gap between estimated and actual rows (stale statistics — run ANALYZE); expensive Sorts (add an index matching ORDER BY); nested-loop joins over huge inputs (missing FK index). Fix one thing, re-EXPLAIN, and compare actual times. Also check it's not the N+1 pattern at the app layer — one slow endpoint is often many fast queries.

🏋️ Practice: create a 1M-row table (generate_series), query by an unindexed column with EXPLAIN ANALYZE (Seq Scan), add the index and compare; break the index with LOWER(email) then fix with an expression index; build a composite index and verify the leftmost rule with EXPLAIN on three query shapes; create a partial index for active rows; find an Index Only Scan by adding INCLUDE columns.

🎯 Key Takeaways

  • B-tree = sorted tree, O(log n) lookups; serves equality, ranges, prefixes, and ORDER BY.
  • Indexes cost writes + storage — add them for measured patterns; index FKs you join on.
  • Functions/wildcards/arithmetic on the column defeat the index — expression indexes or rewrites.
  • Composite indexes follow the leftmost-prefix rule: equalities first, range last.
  • EXPLAIN ANALYZE is the tool: hunt Seq Scans, estimate mismatches (ANALYZE), and sorts; covering/partial indexes for the last mile.
