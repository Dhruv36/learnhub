Isolation Levels & Locking Advanced

The I in ACID has settings. Isolation levels trade correctness against concurrency: stricter levels prevent more anomalies but allow less parallelism. Understanding the anomalies, the levels that prevent them, and pessimistic-vs-optimistic locking is core senior material — it's how you reason about "what happens when two requests touch the same row".

The Anomalies (what can go wrong)

AnomalyWhat happens
Dirty readRead another transaction's UNCOMMITTED change (which may roll back)
Non-repeatable readRe-read the same ROW in one transaction → different value (someone committed between)
Phantom readRe-run the same QUERY → new rows appear (someone inserted matching rows)
Lost updateTwo read-modify-write transactions overwrite each other's change
Write skewTwo txns each check a condition then write, jointly violating an invariant (e.g., both on-call doctors go off duty)

The Isolation Levels

LevelPreventsNotes
READ UNCOMMITTEDNothing extraDirty reads possible (Postgres treats it as Read Committed)
READ COMMITTEDDirty readsPostgres default. Each statement sees the latest committed data — values can change between statements
REPEATABLE READ+ non-repeatable reads (in PG also phantoms)MySQL/InnoDB default. The transaction sees a consistent snapshot from its start
SERIALIZABLEAll anomalies (incl. write skew)As if transactions ran one-by-one; may abort with serialization errors — you must retry
BEGIN ISOLATION LEVEL REPEATABLE READ;  ...  COMMIT;
-- MVCC (multi-version concurrency control): Postgres/MySQL keep multiple row VERSIONS, so
-- READERS DON'T BLOCK WRITERS and vice versa — each transaction reads an appropriate snapshot.
-- That's why plain reads are cheap even under writes; conflicts arise on WRITE-write.

Solving the Lost Update (the practical core)

-- Two ways to make read-then-write safe:

-- 1. PESSIMISTIC: lock the row while you work
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;   -- row locked; others' FOR UPDATE waits
-- compute in app...
UPDATE accounts SET balance = :new WHERE id = 1;
COMMIT;                                            -- lock released
-- Guaranteed, but holds locks (reduces concurrency); risk of deadlocks.

-- 2. OPTIMISTIC: version check, retry on conflict (no locks held)
UPDATE accounts SET balance = :new, version = version + 1
WHERE id = 1 AND version = :version_i_read;
-- 0 rows affected → someone changed it → re-read and retry.
-- Better under low contention; the standard pattern in web apps and ORMs (@Version).
⚖️ The decision: hot rows + high contention or must-not-retry flows → pessimistic (FOR UPDATE). Typical web traffic with rare conflicts → optimistic versioning (cheaper, no lock waits). And remember the simplest fix from the transactions lesson: when the change is expressible in one statement (SET balance = balance - x), a single atomic UPDATE needs neither.

Deadlocks

-- Txn A locks row 1, wants row 2; Txn B locks row 2, wants row 1 → DEADLOCK.
-- The DB detects it and KILLS one transaction (deadlock victim) — your app must retry.
-- Prevention: acquire locks in a CONSISTENT ORDER (e.g., always lower id first),
-- keep transactions short, and lock only what you need (FOR UPDATE with a precise WHERE).

Common Mistakes

MistakeSymptomFix
Read-then-write without locking/versioningLost updates under loadAtomic UPDATE, FOR UPDATE, or version column
Assuming the default level prevents everythingAnomalies at Read Committed surprise youKnow your DB's default; raise per-transaction when needed
SERIALIZABLE without retry logicRandom 40001 serialization errors bubble to usersRetry loop on serialization failures
Long transactions holding locksBlocked writers, deadlocks, replication lagShort transactions; no external I/O inside
Inconsistent lock orderingDeadlocksGlobal lock order; keep lock sets small
Ignoring write skewInvariants violated despite "no conflicts"SERIALIZABLE, or explicit locking of the checked rows

Interview Questions

Q1: Explain the isolation levels and what each additionally prevents.

Read Uncommitted allows dirty reads (seeing uncommitted data). Read Committed (Postgres default) prevents dirty reads — each statement sees only committed data, but two reads in one transaction can differ (non-repeatable). Repeatable Read (MySQL default) gives the transaction a consistent snapshot — re-reads return the same data (Postgres's implementation also prevents phantoms). Serializable makes concurrent transactions equivalent to some serial order, additionally preventing write skew — at the cost of serialization aborts that applications must retry. Stricter = fewer anomalies, less concurrency; choose per-transaction based on the invariants at stake.

Q2: Optimistic vs pessimistic locking — mechanics and when each wins.

Pessimistic: SELECT ... FOR UPDATE locks the rows up front; others wait until you commit — guarantees exclusive access, right for hot rows/high contention or when a retry is unacceptable, but holds locks (throughput cost, deadlock risk). Optimistic: read freely, then write with a version guard (WHERE version = :read_version, incrementing it); zero rows affected means a concurrent change — re-read and retry. No locks held, so it scales better when conflicts are rare (typical web traffic); it's what ORM @Version implements. Rule of thumb: contention high → pessimistic; contention low → optimistic; expressible in one atomic UPDATE → neither.

Q3: What is write skew, and why does only SERIALIZABLE stop it?

Two transactions each read overlapping data, verify a constraint, and write DIFFERENT rows — each write is individually fine, but together they violate the invariant. Classic: two on-call doctors each check "at least one other doctor is on call" and both go off duty. Snapshot-based levels (Repeatable Read) don't detect it because neither transaction wrote a row the other read-modified — no write-write conflict. SERIALIZABLE tracks read/write dependencies (SSI in Postgres) and aborts one transaction to preserve a serial order. Alternative: explicitly lock the rows your check depends on (FOR UPDATE) to force the conflict.

🏋️ Practice (two psql sessions): demonstrate a non-repeatable read at Read Committed, then prevent it with Repeatable Read; produce a lost update with read-then-write, then fix with FOR UPDATE and again with a version column; create a deadlock with two sessions locking in opposite order and read the error; run SERIALIZABLE transactions that conflict and implement a retry.

🎯 Key Takeaways

  • Know the anomalies (dirty, non-repeatable, phantom, lost update, write skew) and which level stops each.
  • Defaults differ: Postgres = Read Committed, MySQL = Repeatable Read; set per-transaction when invariants demand.
  • MVCC = readers don't block writers (snapshots); conflicts are about writes.
  • Lost updates: atomic UPDATE > optimistic versioning (low contention) > FOR UPDATE (high contention).
  • SERIALIZABLE needs retry logic; deadlocks need consistent lock ordering and short transactions.
