Schema Design & Normalization Intermediate

Good schema design prevents whole classes of bugs — no duplicate data to keep in sync, no impossible states. Normalization is the discipline; knowing when to denormalize for performance is the senior judgment. This is where data modeling meets the System Design track.

Keys & Relationships

CREATE TABLE users (
    id         BIGSERIAL PRIMARY KEY,          -- surrogate key (auto-increment); UUID also common
    email      TEXT UNIQUE NOT NULL,            -- natural/candidate key + constraint
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE TABLE orders (
    id       BIGSERIAL PRIMARY KEY,
    user_id  BIGINT NOT NULL REFERENCES users(id),   -- FOREIGN KEY → referential integrity
    total    NUMERIC(10,2) NOT NULL,
    status   TEXT NOT NULL DEFAULT 'pending'
);
-- PK = unique row identity. FK = a reference to another table's PK; the DB ENFORCES that the
-- referenced row exists (can't insert an order for a non-existent user; controls deletes).
RelationshipModeled by
One-to-many (user → orders)FK on the "many" side (orders.user_id)
Many-to-many (students ↔ courses)A JUNCTION table (enrollments: student_id, course_id)
One-to-oneFK + UNIQUE, or shared PK

Normalization — Eliminate Redundancy

-- ❌ Unnormalized: repeating/embedded data → update anomalies
orders(id, user_name, user_email, product1, product2, product3)
-- Change a user's email → must update every one of their order rows (miss one = inconsistent).

-- ✅ Normalized: each fact stored ONCE, referenced by key:
users(id, name, email)   orders(id, user_id)   order_items(order_id, product_id, qty)
Normal formRequires
1NFAtomic values (no lists in a column), a key — no repeating groups (product1/2/3)
2NF1NF + every non-key column depends on the WHOLE key (no partial dependency on part of a composite key)
3NF2NF + no transitive dependencies (non-key columns depend only on the key, not on other non-key columns)

Practical rule: "every non-key fact depends on the key, the whole key, and nothing but the key." 3NF is the default target — it removes update/insert/delete anomalies. Higher forms (BCNF+) matter rarely.

When to Denormalize (deliberately)

-- Normalization optimizes for WRITES/consistency; it can require many JOINs for reads.
-- At scale, controlled denormalization trades duplication for read speed:
--   • a cached order_count column on users (avoid COUNT on every profile view)
--   • storing a product_name snapshot on order_items (history + no join; price at purchase time)
--   • materialized views for expensive aggregates
-- The cost: you must keep the duplicate in sync (triggers/app logic/jobs). Denormalize with EYES
-- OPEN, driven by measured read patterns — not preemptively. (System Design: read vs write trade-offs.)

Interview Questions

Q1: What problems does normalization solve? Explain 3NF simply.

It eliminates data redundancy and the update/insert/delete ANOMALIES redundancy causes: if a fact (a user's email) is stored in many rows, changing it means updating all of them — miss one and the data is inconsistent. Normalization stores each fact once and references it by key. 3NF, plainly: every non-key column depends on the key, the whole key, and nothing but the key — no column should be derivable from another non-key column (a transitive dependency, like storing city AND its zip-derived state). Reaching 3NF removes most anomalies and is the practical default.

Q2: When and how would you denormalize?

When measured READ patterns make normalized joins/aggregates too slow — normalization favors write consistency, but read-heavy paths may need duplication. Techniques: precomputed counter/summary columns (order_count on users), snapshotting values that shouldn't change (product name/price at purchase time on order_items — which is also correct modeling of history), and materialized views for expensive aggregates. The cost is keeping duplicates in sync (triggers, application logic, or scheduled jobs), so denormalize deliberately, guided by data, with a sync strategy — not preemptively. It's the read-vs-write trade-off from the System Design track.

Q3: How do you model a many-to-many relationship, and why?

With a junction (associative) table holding foreign keys to both sides — e.g., enrollments( student_id, course_id) with a composite or surrogate PK, each row representing one student-course pairing. You can't model many-to-many with a single FK column (that's one-to-many), and stuffing IDs into a list column violates 1NF and can't be joined/indexed. The junction table also naturally holds relationship attributes (enrolled_at, grade). Foreign keys on it enforce that both referenced rows exist.

🏋️ Practice: design users/orders/order_items with PKs and FKs; take an unnormalized table with repeated user data and split it to 3NF; model a students↔courses many-to-many with a junction table (plus an enrolled_at attribute); add a denormalized order_count with a plan to keep it in sync; choose correct types for money, timestamps, and a status field.

🎯 Key Takeaways

  • PK = row identity; FK = enforced reference; junction tables model many-to-many.
  • Normalize (target 3NF): store each fact once — "depends on the key, the whole key, nothing but the key" — to kill anomalies.
  • Denormalize deliberately for measured read performance, with a sync strategy (the read/write trade-off).
  • Types matter: NUMERIC for money, TIMESTAMPTZ (UTC) for time, JSONB only for sparse/variable data.
  • Foreign keys enforce integrity; lists-in-a-column violate 1NF — use child tables.
