Sorting, Limiting & Aggregates Basics

Aggregation collapses many rows into summary values — counts, sums, averages per group. The concepts that trip people up: GROUP BY rules, WHERE vs HAVING, and how NULLs and COUNT interact. These appear in nearly every analytical query and every SQL interview.

Aggregate Functions

SELECT COUNT(*)          AS total_rows,        -- counts ALL rows (incl. NULLs)
       COUNT(email)     AS with_email,        -- counts NON-NULL emails only!
       COUNT(DISTINCT country) AS countries,
       SUM(score), AVG(score), MIN(score), MAX(score)
FROM users;
-- KEY: COUNT(*) counts rows; COUNT(column) IGNORES NULLs in that column.
-- SUM/AVG also ignore NULLs (AVG divides by the count of non-NULLs, not total rows).

GROUP BY — Aggregate Per Group

SELECT country, COUNT(*) AS users, AVG(score) AS avg_score
FROM users
GROUP BY country
ORDER BY users DESC;
-- Rule: every column in SELECT must be EITHER in GROUP BY OR inside an aggregate.
--   SELECT country, name, COUNT(*) ... GROUP BY country   -- ❌ 'name' is neither
-- (Postgres enforces this strictly; MySQL historically allowed it and returned arbitrary values —
--  a classic source of wrong results. Enable ONLY_FULL_GROUP_BY.)
SELECT country, role, COUNT(*) FROM users GROUP BY country, role;   -- multi-column groups

WHERE vs HAVING (the exam question)

SELECT country, COUNT(*) AS users
FROM users
WHERE active = true          -- WHERE filters ROWS *before* grouping (can't use aggregates)
GROUP BY country
HAVING COUNT(*) > 100        -- HAVING filters GROUPS *after* aggregation (uses aggregates)
ORDER BY users DESC;
-- From the execution-order lesson: WHERE runs before GROUP BY, HAVING after.
-- Put row conditions in WHERE (they shrink the data early = faster); group conditions in HAVING.

Sorting & Limiting (pagination)

SELECT * FROM users ORDER BY score DESC, name ASC   -- multi-key (tie-break by name)
LIMIT 20 OFFSET 40;                                  -- page 3 (rows 41–60)
-- ⚠ ORDER BY without a UNIQUE tiebreaker gives NONDETERMINISTIC order across runs —
--   pagination can skip/duplicate rows. Always include a unique column (e.g. id) in ORDER BY.
-- ⚠ Large OFFSET is SLOW (the DB scans+discards OFFSET rows). For deep pages use KEYSET
--   pagination:  WHERE (score, id) < (:last_score, :last_id) ORDER BY score DESC, id DESC LIMIT 20.
ORDER BY score DESC NULLS LAST;                       -- control where NULLs sort

Interview Questions

Q1: WHERE vs HAVING — precisely.

WHERE filters individual ROWS before grouping and cannot reference aggregates (grouping hasn't happened yet). HAVING filters GROUPS after aggregation and is where aggregate conditions (COUNT(*) > 100) go. It follows from execution order (WHERE before GROUP BY, HAVING after). Performance: put row-level conditions in WHERE so they shrink the dataset before the expensive grouping; reserve HAVING for genuinely group-level conditions. Putting a row condition in HAVING works but filters later, doing more work.

Q2: COUNT(*) vs COUNT(column) vs COUNT(DISTINCT column)?

COUNT(*) counts every row including those with NULLs. COUNT(column) counts only rows where that column is NON-NULL (aggregates ignore NULLs). COUNT(DISTINCT column) counts distinct non-NULL values. The distinction matters: COUNT(email) on a table where some emails are NULL undercounts total users. Likewise AVG(score) divides the SUM by the count of non-NULL scores, not by total rows — so NULLs quietly change averages. Always be explicit about NULL handling in aggregates.

Q3: Why is OFFSET pagination slow at scale, and what's the alternative?

OFFSET N means the database must generate and then DISCARD the first N ordered rows before returning the page — so page 10,000 scans ~200,000 rows. Cost grows linearly with the page number. Keyset (seek) pagination instead remembers the last row's sort key and does WHERE (sort_col, id) < (last_sort, last_id) ORDER BY ... LIMIT n — using an index to jump directly to the next page in roughly constant time. Trade-off: keyset can't jump to an arbitrary page number, but it's the right choice for feeds/infinite scroll.

🏋️ Practice: compute COUNT(*), COUNT(email), and AVG(score) on a table with NULLs and explain the differences; GROUP BY country with AVG and HAVING COUNT(*) > k; try selecting a non-grouped column to see the error; implement OFFSET pagination then rewrite it as keyset pagination; add a unique tiebreaker to an ORDER BY and observe stable ordering.

🎯 Key Takeaways

  • COUNT(*) counts rows; COUNT(col)/SUM/AVG ignore NULLs — be explicit.
  • GROUP BY: every SELECT column must be grouped or aggregated (enable ONLY_FULL_GROUP_BY on MySQL).
  • WHERE filters rows pre-aggregation (put row conditions here); HAVING filters groups post-aggregation.
  • ORDER BY needs a unique tiebreaker for stable pagination; NULLS FIRST/LAST controls NULL placement.
  • Deep OFFSET is slow — use keyset/seek pagination for large offsets/feeds.
