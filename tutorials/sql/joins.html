JOINs Basics→Intermediate

JOINs combine rows from multiple tables on a related column — the essence of a relational database. Mastering the four join types, understanding how NULLs and duplicates behave in joins, and knowing the N+1 problem are constant interview and production concerns.

The Four Joins (Venn-diagram intuition)

SELECT u.name, o.total
FROM users u
JOIN orders o ON o.user_id = u.id;      -- INNER: only rows with a match on BOTH sides
JoinReturnsUse for
INNER JOINOnly matching rows (both sides)"Users WITH orders"
LEFT JOINAll left rows + matches (NULLs where no right match)"All users, with their orders (or none)"
RIGHT JOINAll right rows + matchesRare — flip to LEFT for readability
FULL OUTER JOINAll rows from both, NULLs where no matchReconciliation / diffing datasets
CROSS JOINCartesian product (every combo)Generating combinations; rarely intentional

LEFT JOIN + Finding Missing Rows (the anti-join)

-- "Users who have NEVER ordered": LEFT JOIN then keep the NULLs:
SELECT u.name
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE o.id IS NULL;              -- no matching order → o.id is NULL
-- This IS-NULL-after-LEFT-JOIN pattern is the standard "find rows without a related row".
⚠️ The LEFT JOIN + WHERE trap: putting a condition on the RIGHT table in WHERE silently turns a LEFT JOIN into an INNER JOIN — because WHERE runs after the join and NULL rows fail the condition. If you want to filter the right table while keeping unmatched left rows, put the condition in the ON clause instead: LEFT JOIN orders o ON o.user_id = u.id AND o.status = 'paid'. ON vs WHERE placement for outer joins is a top interview discriminator.

Multi-Table Joins & Aggregation

SELECT u.name, COUNT(o.id) AS order_count, COALESCE(SUM(o.total), 0) AS lifetime_value
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.name              -- group by the PK to keep names correct
ORDER BY lifetime_value DESC;
-- COUNT(o.id) (not COUNT(*)) counts real orders — for users with none it's 0, because
-- COUNT ignores the NULL o.id from the unmatched LEFT JOIN. COUNT(*) would return 1 (the row).

Join Duplicates (a subtle bug)

-- If a user has 3 orders, joining users→orders produces 3 ROWS for that user.
-- SUM over another one-to-many join then DOUBLE-COUNTS ("fan-out"):
SELECT u.id, SUM(o.total), SUM(p.amount)   -- ❌ if BOTH orders and payments are 1-to-many,
FROM users u JOIN orders o... JOIN payments p...  -- the cross-product inflates both sums!
-- Fix: aggregate each side in a subquery/CTE first, then join the aggregates.

The N+1 Problem (production killer)

-- Application code doing: fetch users, then a query PER user for their orders:
users = query("SELECT * FROM users");           -- 1 query
for u in users: query("SELECT * FROM orders WHERE user_id = ?", u.id)  -- N queries!
-- 100 users = 101 round trips → slow. Fix with ONE join (or one IN query):
SELECT * FROM orders WHERE user_id IN (:ids);   -- or a JOIN
-- ORMs cause this silently (lazy loading); use eager fetch / JOIN FETCH / dataloaders.

Interview Questions

Q1: INNER vs LEFT JOIN, and how do you find "rows without a match"?

INNER returns only rows with a match on both sides; LEFT returns all left rows plus matches, with NULLs on the right where there's no match. To find left rows WITHOUT a related right row (an anti-join), LEFT JOIN and then filter WHERE right.id IS NULL — the unmatched rows are exactly those whose right columns are NULL. Classic uses: users who never ordered, products never sold, accounts without a subscription.

Q2: Why does a condition on the right table in WHERE break a LEFT JOIN?

The LEFT JOIN produces NULLs for unmatched right rows, but a WHERE condition on a right column (e.g., o.status = 'paid') runs AFTER the join and evaluates to UNKNOWN for those NULL rows, so they get filtered out — effectively converting the LEFT JOIN into an INNER JOIN. To filter the right side while preserving unmatched left rows, put the condition in the ON clause (it's applied during the join, before NULLs are generated). ON vs WHERE for outer joins is the key nuance.

Q3: What is the N+1 query problem and how do you fix it?

Fetching a list (1 query) then issuing one additional query per item to load related data (N queries) — 1 + N round trips, which is slow and hammers the DB, common with ORM lazy loading. Fix by loading the related data in a single query: a JOIN, or a batched WHERE fk IN (:ids), or the ORM's eager-fetch / JOIN FETCH / dataloader mechanisms. Detection: enable SQL logging and watch for repeated identical queries. It's one of the most common real-world performance bugs.

🏋️ Practice: join users and orders (INNER vs LEFT); find users with no orders via LEFT JOIN + IS NULL; demonstrate the WHERE-vs-ON trap by moving a right-table condition and watching rows vanish/return; compute per-user order counts with COUNT(o.id) and LEFT JOIN (0 for none); reproduce fan-out double-counting with two one-to-many joins and fix with pre-aggregation.

🎯 Key Takeaways

  • INNER = matches only; LEFT = all left + matches (NULLs otherwise); anti-join = LEFT + WHERE right IS NULL.
  • On outer joins, filter the right table in ON (keeps unmatched rows); WHERE turns LEFT into INNER.
  • COUNT(right.id) not COUNT(*) after LEFT JOIN; pre-aggregate to avoid fan-out double-counting.
  • N+1 (per-row queries) is a top performance bug — use JOINs / IN / eager fetch.
  • Index the join (FK) columns; a missing ON is an accidental CROSS JOIN.
