Window Functions Intermediate

Window functions compute across a set of rows related to the current row — without collapsing them like GROUP BY does. Running totals, rankings, per-group top-N, moving averages, row-over-row comparisons: all clean one-liners. They're a favorite advanced interview topic because they replace gnarly self-joins and subqueries.

The Core Idea: Aggregate WITHOUT Collapsing

-- GROUP BY collapses 100 users into N rows. A window function KEEPS all 100 rows and adds a
-- computed column looking across a "window" of related rows:
SELECT name, department, salary,
       AVG(salary) OVER (PARTITION BY department) AS dept_avg,   -- avg of this row's dept
       salary - AVG(salary) OVER (PARTITION BY department) AS diff_from_avg
FROM employees;
-- PARTITION BY = the group (like GROUP BY, but rows aren't merged). No PARTITION = whole table.

Ranking Functions

SELECT name, department, salary,
       ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rn,   -- 1,2,3,4 (unique)
       RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rnk,  -- 1,2,2,4 (gaps on ties)
       DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS drnk, -- 1,2,2,3 (no gaps)
       NTILE(4)     OVER (ORDER BY salary) AS quartile
FROM employees;
FunctionOn ties…
ROW_NUMBER()Always unique (arbitrary among ties) — use for dedup/pagination
RANK()Same rank, then SKIPS (1,2,2,4)
DENSE_RANK()Same rank, NO skip (1,2,2,3)

The Top-N-Per-Group Pattern (classic interview)

-- "Top 3 earners per department" — ROW_NUMBER in a subquery, filter outside:
SELECT * FROM (
    SELECT name, department, salary,
           ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rn
    FROM employees
) t
WHERE rn <= 3;
-- You can't filter on a window function in WHERE directly (it's computed in SELECT, after WHERE) —
-- so wrap it in a subquery/CTE and filter the alias. This pattern replaces messy correlated subqueries.

Running Totals, Moving Averages & Offsets

SELECT day, revenue,
       SUM(revenue) OVER (ORDER BY day) AS running_total,           -- cumulative
       AVG(revenue) OVER (ORDER BY day ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS moving_7d,
       LAG(revenue)  OVER (ORDER BY day) AS prev_day,               -- previous row's value
       LEAD(revenue) OVER (ORDER BY day) AS next_day,
       revenue - LAG(revenue) OVER (ORDER BY day) AS day_over_day   -- growth
FROM daily_sales;
-- ORDER BY inside OVER defines the running order; the ROWS/RANGE frame defines the window extent.
-- LAG/LEAD access other rows without a self-join — the row-over-row comparison workhorse.

Interview Questions

Q1: Window functions vs GROUP BY — the key difference?

GROUP BY collapses each group into a single summary row (you lose the individual rows). A window function computes an aggregate/ranking over a window of related rows (defined by PARTITION BY / ORDER BY / frame) but KEEPS every original row, adding the computed value as a new column. So you can show each employee's salary alongside their department average, or a running total per row — impossible with plain GROUP BY without a self-join. Use windows when you need both the detail rows and a computed cross-row value.

Q2: ROW_NUMBER vs RANK vs DENSE_RANK on ties?

All rank within a partition by an ORDER BY. On ties: ROW_NUMBER assigns distinct sequential numbers arbitrarily (1,2,3,4) — good for deduplication and pagination; RANK gives tied rows the same rank then skips (1,2,2,4) — like sports standings; DENSE_RANK gives the same rank with no gaps (1,2,2,3). Choose ROW_NUMBER when you need exactly one row per position (e.g., latest record per key: ROW_NUMBER()... = 1), RANK/DENSE_RANK when ties should share a position.

Q3: How do you get the top N rows per group, and why the subquery?

Compute ROW_NUMBER() OVER (PARTITION BY group ORDER BY metric DESC) in an inner query, then filter WHERE rn <= N in an outer query/CTE. The subquery is required because window functions are evaluated in the SELECT phase — AFTER WHERE — so you can't reference the window alias in the same query's WHERE. Wrapping it lets the outer WHERE filter on the already-computed row number. This pattern (also used for "latest row per key" with rn = 1) replaces slow correlated subqueries or self-joins.

🏋️ Practice: add a department-average column with AVG() OVER (PARTITION BY); rank employees three ways (ROW_NUMBER/RANK/DENSE_RANK) and compare ties; get the top-3 per department with the subquery pattern; compute a running total and a 7-row moving average over daily sales; use LAG to compute day-over-day growth; get the latest order per user with ROW_NUMBER = 1.

🎯 Key Takeaways

  • Window functions compute across related rows WITHOUT collapsing them (unlike GROUP BY).
  • OVER (PARTITION BY ... ORDER BY ... frame): partition = group, ORDER BY = running order, frame = extent.
  • ROW_NUMBER (unique), RANK (gaps on ties), DENSE_RANK (no gaps) — pick by tie behavior.
  • Top-N-per-group and latest-per-key: ROW_NUMBER in a subquery/CTE, then filter the alias.
  • Running totals, moving averages, LAG/LEAD replace self-joins for cross-row analytics.
