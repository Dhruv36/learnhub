Caching & Background Services Advanced

Two production-critical patterns for work that doesn't fit the request/response model: caching avoids recomputing/refetching expensive data, and background services run long-lived work OUTSIDE the request pipeline entirely. Both connect directly to the .NET DI lesson's Singleton lifetime and its captive-dependency rule.

In-Memory Caching

builder.Services.AddMemoryCache();

public class ProductService(IMemoryCache cache, AppDbContext db) {
    public async Task GetAsync(int id) {
        return await cache.GetOrCreateAsync($"product:{id}", async entry => {
            entry.SlidingExpiration = TimeSpan.FromMinutes(10);   // evict if unused for 10 min
            return await db.Products.AsNoTracking().FirstOrDefaultAsync(p => p.Id == id);
        });
    }
}
// IMemoryCache is registered as a SINGLETON internally — the cached data lives in THIS process's
// memory. GetOrCreateAsync is the standard "get it, or compute+store it if missing" pattern.
⚠️ In-memory cache is PER-INSTANCE — a real problem at scale. If your API runs as multiple replicas (the Kubernetes track's HPA scaling multiple Pods), each instance has its OWN separate in-memory cache — a write on instance A isn't visible to instance B, and cached data can be inconsistent across instances. For multi-instance deployments, a DISTRIBUTED cache (Redis, via IDistributedCache — the Redis track) shares one cache across all instances.

Distributed Caching (Redis)

builder.Services.AddStackExchangeRedisCache(options =>
    options.Configuration = builder.Configuration.GetConnectionString("Redis"));

public class ProductService(IDistributedCache cache, AppDbContext db) {
    public async Task GetAsync(int id) {
        var cached = await cache.GetStringAsync($"product:{id}");
        if (cached is not null) return JsonSerializer.Deserialize(cached);

        var product = await db.Products.AsNoTracking().FirstOrDefaultAsync(p => p.Id == id);
        if (product is not null)
            await cache.SetStringAsync($"product:{id}", JsonSerializer.Serialize(product),
                new DistributedCacheEntryOptions { SlidingExpiration = TimeSpan.FromMinutes(10) });
        return product;
    }
}
// Same interface pattern, but IDistributedCache stores serialized data in an EXTERNAL store (Redis) —
// every instance of the API shares the SAME cache, consistent across the whole deployment.

Output Caching — Caching Whole HTTP Responses

builder.Services.AddOutputCache();
app.UseOutputCache();

app.MapGet("/products", GetAllProducts).CacheOutput(policy => policy.Expire(TimeSpan.FromSeconds(30)));
// Caches the ENTIRE HTTP response for repeat requests to the same endpoint/parameters — even cheaper
// than app-level caching for read-heavy, rarely-changing endpoints, since the handler doesn't even run
// on a cache hit.

BackgroundService — Long-Running Work Outside Requests

public class OrderCleanupService(IServiceScopeFactory scopeFactory, ILogger logger)
    : BackgroundService {
    protected override async Task ExecuteAsync(CancellationToken stoppingToken) {
        while (!stoppingToken.IsCancellationRequested) {
            using var scope = scopeFactory.CreateScope();      // ← the captive-dependency fix, applied
            var db = scope.ServiceProvider.GetRequiredService();   // fresh SCOPED DbContext
            var stale = await db.Orders.Where(o => o.Status == "Abandoned").ToListAsync(stoppingToken);
            db.Orders.RemoveRange(stale);
            await db.SaveChangesAsync(stoppingToken);
            logger.LogInformation("Cleaned up {Count} orders", stale.Count);

            await Task.Delay(TimeSpan.FromHours(1), stoppingToken);   // run again in an hour
        }
    }
}
builder.Services.AddHostedService();
// BackgroundService is itself hosted as a SINGLETON by the framework — which is EXACTLY why it must
// create its own scope to access Scoped services like DbContext (the .NET DI lesson's captive-dependency
// rule, now in a concrete, common real-world scenario).
🔗 This is the captive-dependency pattern from the .NET DI lesson, made concrete. A BackgroundService runs for the app's ENTIRE lifetime — it IS effectively a Singleton. It can never directly inject a Scoped DbContext (that would capture one instance forever, eventually hitting a disposed context). Creating a fresh IServiceScopeFactory-derived scope on each iteration is the standard, correct pattern — not a workaround, THE pattern.

Interview Questions

Q1: What's the difference between IMemoryCache and IDistributedCache, and when does the difference actually matter?

IMemoryCache stores cached data directly in the current process's memory — it's extremely fast (no network hop) but is entirely LOCAL to that one running instance of the application. IDistributedCache is an abstraction over an EXTERNAL cache store (most commonly Redis, via AddStackExchangeRedisCache) — reading and writing incurs a network round-trip, making it slightly slower per-operation than in-memory access, but the cached data is stored OUTSIDE any single application instance and is therefore SHARED and consistent across every instance of the application that connects to it. This distinction becomes critical the moment an application is deployed as MULTIPLE REPLICAS (which is the normal case for anything running behind a load balancer or scaled horizontally in Kubernetes, per the K8s track's HPA lesson): with IMemoryCache, each replica has its own independent, disconnected cache — a write that updates instance A's cache does nothing to instance B's cache, so two requests hitting different instances could see inconsistent data, and cache invalidation logic that clears one instance's cache leaves the others stale. IDistributedCache solves this because there's exactly ONE logical cache (the Redis instance/cluster) that every application replica reads from and writes to, so a cache update or invalidation is immediately visible to all instances. The practical rule: IMemoryCache is fine for a single-instance deployment or for data where slight cross-instance inconsistency is genuinely tolerable; any horizontally-scaled, multi-replica production API needs IDistributedCache for correctness.

Q2: Why does a BackgroundService need to create its own DI scope to access a DbContext?

A class registered via AddHostedService<T>() (which BackgroundService implementations use) is hosted by the framework as effectively a SINGLETON — it's instantiated once when the application starts and its ExecuteAsync method runs for the entire lifetime of the application, often in a loop performing periodic work. DbContext, on the other hand, is registered as SCOPED (correctly, per the earlier EF Core lesson) precisely because it's meant to represent one unit-of-work's worth of state, typically scoped to a single HTTP request, and is disposed at the end of that scope. If a BackgroundService tried to inject IOrderRepository or AppDbContext DIRECTLY into its constructor, it would trigger the exact captive-dependency bug from the .NET DI lesson: because the BackgroundService is only ever constructed ONCE for the app's whole lifetime, whatever Scoped DbContext instance happened to be resolved at that single construction moment would be permanently captured — and since nothing ever creates a fresh scope around it, that DbContext would likely already be disposed (or at minimum stale) by the time the background loop's later iterations tried to use it, causing an ObjectDisposedException or worse. The correct pattern, shown in the lesson, is to inject IServiceScopeFactory (which, like IServiceProvider, is safe for a Singleton to hold) and explicitly call scopeFactory.CreateScope() at the START of each unit of work inside the loop, resolving a FRESH DbContext from that new scope each time, then letting the `using` block dispose that scope (and its DbContext) when that iteration's work is done — giving the background service correctly-scoped, non-captive access to Scoped services on each pass.

Q3: What's the difference between application-level caching (IMemoryCache/IDistributedCache) and output caching?

Application-level caching (via IMemoryCache or IDistributedCache) caches a specific PIECE OF DATA — typically the result of an expensive computation or database query — inside your service/business logic code; you explicitly decide what key to cache under and what value to store, and the caching happens INSIDE the endpoint handler, which still runs on every request (it just might skip the expensive database call if the cache has a hit). Output caching instead caches the ENTIRE HTTP RESPONSE for a given endpoint and request signature (URL, query parameters, sometimes headers) at the ASP.NET Core pipeline level, configured declaratively with .CacheOutput(...) on an endpoint (or globally via middleware policies) — on a cache hit, the request never even reaches your endpoint handler code at all; the cached response bytes are served directly from the output cache. This makes output caching even cheaper than application-level caching for endpoints where the ENTIRE response can legitimately be reused unchanged for a period of time (a public, read-heavy, rarely-changing listing endpoint is the classic case), since it skips not just the database query but the entire handler execution, model binding overhead, and serialization on every cache hit. The trade-off is that output caching is coarser-grained — you cache the whole response as a unit, keyed by the request itself, whereas application-level caching gives you fine-grained control over exactly what data is cached and under what key, which matters when a single response combines multiple independently-cacheable pieces of data with different appropriate cache lifetimes, or when caching needs to vary based on business logic more complex than the raw request URL/parameters.

🏋️ Practice: add IMemoryCache to a service with GetOrCreateAsync and confirm a second identical request skips the database; switch to IDistributedCache with a local Redis instance and confirm behavior is equivalent; add output caching to a read-heavy endpoint and verify (via logging) the handler doesn't run on cache hits; build a BackgroundService that periodically queries the database via a correctly-created scope; deliberately inject DbContext directly into a BackgroundService constructor and observe the resulting error.

🎯 Key Takeaways

  • IMemoryCache is per-instance (fast, but inconsistent across replicas); IDistributedCache (Redis) is shared across all instances — required for correctness in multi-replica deployments.
  • Output caching caches the whole HTTP response at the pipeline level — cheaper than app-level caching since the handler doesn't run at all on a hit, but coarser-grained.
  • BackgroundService (via AddHostedService) is effectively a Singleton running for the app's whole lifetime — never inject Scoped services (like DbContext) directly into it.
  • The correct pattern is injecting IServiceScopeFactory and creating a fresh scope per unit of work inside the loop — the .NET DI lesson's captive-dependency fix, applied concretely.
  • Always have a cache invalidation strategy on writes, and match cache duration to actual data volatility — stale caches are a common source of confusing bugs.
