Clean Architecture, CQRS & API Versioning Production

As an API grows past a handful of endpoints, structure starts to matter as much as correctness. This lesson covers three production-scale architecture patterns: Clean Architecture's dependency direction, CQRS's separation of reads from writes (often via MediatR), and how to evolve an API's contract without breaking existing clients.

Clean Architecture — the Dependency Rule

// Layers, from the CENTER outward:
//   Domain      — entities, business rules. Depends on NOTHING else.
//   Application — use cases/services orchestrating domain logic. Depends only on Domain.
//   Infrastructure — EF Core, external APIs, file system. Depends on Application/Domain (implements their interfaces).
//   Presentation (API) — controllers/Minimal API endpoints. Depends on Application.
//
// THE DEPENDENCY RULE: dependencies point INWARD only. Domain knows nothing about EF Core, ASP.NET
// Core, or any framework. Infrastructure depends on abstractions DEFINED in Application/Domain
// (interfaces), not the other way around — this is DEPENDENCY INVERSION applied at the architecture level.
🎯 The payoff: your business logic doesn't know EF Core exists. Domain and Application layers define interfaces (IOrderRepository) that Infrastructure implements (using EF Core). This means you could swap EF Core for Dapper, or swap SQL Server for PostgreSQL, touching ONLY the Infrastructure layer — the core business rules never change. It also makes the Application/Domain layers trivially unit-testable with no database at all, since they only depend on interfaces.

CQRS — Separating Reads from Writes

// Command Query Responsibility Segregation: model WRITES (commands) and READS (queries) as
// DIFFERENT code paths, even if they hit the same database.
public record CreateOrderCommand(int CustomerId, List Items) : IRequest;   // a WRITE
public record GetOrderByIdQuery(int Id) : IRequest;                              // a READ

public class CreateOrderHandler(AppDbContext db) : IRequestHandler {
    public async Task Handle(CreateOrderCommand cmd, CancellationToken ct) {
        var order = new Order { CustomerId = cmd.CustomerId, Items = cmd.Items };
        db.Orders.Add(order);
        await db.SaveChangesAsync(ct);
        return order.Id;
    }
}
// Controller/endpoint just sends the command/query — it doesn't know HOW it's handled:
[HttpPost]
public async Task Create(CreateOrderCommand cmd) => Ok(await mediator.Send(cmd));
// WHY separate them? Reads and writes often have genuinely different needs:
//   - Queries can use lean, denormalized DTOs (AsNoTracking, no domain rules) optimized purely for display
//   - Commands enforce full business rules/validation/invariants
//   - At extreme scale, reads and writes can even use DIFFERENT DATA STORES entirely (a read-optimized
//     cache/search index fed by events from the write side) — but that's an advanced extension, not the
//     baseline CQRS pattern most teams actually use.
// MediatR is the most common library for wiring this up — it decouples "sending a request" from "who
// handles it," similar in spirit to how DI decouples "needing a service" from "which implementation."

API Versioning — Evolving Without Breaking Clients

builder.Services.AddApiVersioning(options => {
    options.DefaultApiVersion = new ApiVersion(1, 0);
    options.AssumeDefaultVersionWhenUnspecified = true;
});

[ApiVersion("1.0")]
[Route("api/v{version:apiVersion}/orders")]
public class OrdersV1Controller : ControllerBase { ... }     // old clients keep working

[ApiVersion("2.0")]
[Route("api/v{version:apiVersion}/orders")]
public class OrdersV2Controller : ControllerBase { ... }     // new shape, new clients opt in
// Versioning strategies: URL segment (/api/v2/orders — most explicit/cacheable), query string
// (?api-version=2.0), or a custom header. URL segment is the most common for public APIs — visible,
// cacheable, and unambiguous.
⚠️ A breaking change to a live API is a production incident waiting to happen. Renaming a field, changing a type, or removing a property breaks every client still calling the old shape — exactly the "expand/contract" problem from the CI/CD deployment-strategies lesson, now applied to API contracts instead of database schemas. Versioning lets you introduce breaking changes in a NEW version while old clients keep working against the old one, on their own migration timeline.

Interview Questions

Q1: Explain Clean Architecture's dependency rule and why it matters.

Clean Architecture organizes code into concentric layers — Domain at the center (entities and core business rules), Application around it (use cases orchestrating domain logic), then Infrastructure (concrete implementations: EF Core, external HTTP clients, file I/O) and Presentation (API controllers/endpoints) on the outside. The DEPENDENCY RULE states that dependencies may only point INWARD: outer layers can depend on inner layers, but inner layers must never depend on outer ones — concretely, the Domain and Application layers contain plain C# with no reference to EF Core, ASP.NET Core, or any specific framework or infrastructure technology at all. This is achieved through DEPENDENCY INVERSION: the Application/Domain layers DEFINE interfaces representing what they need (like IOrderRepository), and the Infrastructure layer provides concrete IMPLEMENTATIONS of those interfaces using whatever technology is chosen (EF Core, Dapper, an external API client) — so even though Infrastructure "depends on" Application/Domain (implementing their interfaces), the actual RUNTIME dependency, wired up via DI, flows the other way at the code level: Application only ever references the abstraction, never the concrete implementation. This matters practically for two major reasons: it makes the Domain and Application layers — where the actual business value and rules live — fully unit-testable with zero database or framework dependencies (you can construct a fake IOrderRepository trivially), and it means swapping a technology choice (a different ORM, a different database, a different external payment provider) only requires changes within the Infrastructure layer, leaving the core business logic completely untouched.

Q2: What is CQRS, and when is it actually worth adopting?

CQRS (Command Query Responsibility Segregation) is the practice of modeling WRITES (commands, which change state and typically enforce business rules/validation/invariants) and READS (queries, which just retrieve and shape data for display) as distinct, separately-defined code paths, rather than routing both through the same general-purpose service methods. In its common, moderate implementation (often using a library like MediatR to decouple "sending a request" from "which handler processes it"), a command like CreateOrderCommand is handled by a dedicated handler that applies full business logic and persists via the normal write model, while a query like GetOrderByIdQuery is handled by a separate, much simpler handler that can use lean, denormalized DTOs and skip unnecessary overhead (like EF Core change tracking, since a read never needs to save anything back) — both still typically hit the SAME underlying database in this common form; a more extreme, advanced variant additionally uses entirely separate data stores for reads and writes (a read-optimized cache or search index kept in sync via events from the write side), but that's a significant additional complexity most teams never need. CQRS is genuinely worth adopting when reads and writes have meaningfully DIFFERENT needs — complex business rules on the write side that don't belong anywhere near simple read logic, or read patterns (reporting, dashboards, search) that would be awkward or inefficient to serve through the same model used for enforcing write-side invariants. It is explicitly NOT a default — for a straightforward CRUD application without this kind of read/write asymmetry, introducing CQRS (and especially MediatR's indirection of "send a message, something somewhere handles it") adds real ceremony and cognitive overhead without a matching payoff, which is why experienced teams treat it as a tool for GENUINE complexity, not a pattern to apply reflexively to every project.

Q3: Why is API versioning necessary, and what are the common strategies?

Once an API has real external clients depending on its contract — the shape of request/response bodies, available endpoints, expected behavior — any BREAKING change (renaming or removing a field, changing a field's type, altering an endpoint's behavior in an incompatible way) will break every client still calling the old contract, the moment that change deploys to production; unlike an internal application where you control both sides of every call, an API's clients may be entirely outside your control (third-party integrations, mobile apps that can't be force-updated, other teams' services) and can't simply be updated in lockstep with your deployment. This mirrors exactly the database migration problem from the CI/CD and .NET tracks — old and new "versions" of a contract potentially need to coexist during a transition — except applied to the API's public shape rather than a database schema. Versioning solves this by letting you introduce a NEW version of an endpoint (or the whole API) that carries the breaking change, while the OLD version continues to exist and function unchanged for clients that haven't migrated yet, giving them their own timeline to update. The common strategies for signaling which version a request wants are: URL segment versioning (/api/v2/orders — the most common approach for public APIs, since it's explicit, human-readable, cacheable by intermediary proxies/CDNs since the URL itself differs, and unambiguous), query string versioning (?api-version=2.0 — keeps the URL path stable but is less visible/cacheable), and custom header versioning (a dedicated header like X-Api-Version — keeps URLs completely clean but is the least discoverable/debuggable, since the version isn't visible in the URL at all). Whichever strategy is chosen, the broader practice also requires a deliberate DEPRECATION policy — communicating a timeline for when old versions will eventually be retired, rather than maintaining every version indefinitely, which would otherwise become an ever-growing maintenance burden.

🏋️ Practice: structure a small API into Domain/Application/Infrastructure/Presentation projects, with Domain referencing nothing else; define a repository interface in Application and implement it with EF Core in Infrastructure; add MediatR and implement one Command and one Query handler for the same resource; add URL-segment API versioning with a v1 and v2 controller for the same resource showing a genuine breaking change (e.g., a renamed field); sketch a deprecation timeline for retiring v1.

🎯 Key Takeaways

  • Clean Architecture's dependency rule: dependencies point inward only — Domain/Application know nothing about EF Core/ASP.NET Core; Infrastructure implements interfaces THEY define (dependency inversion).
  • This makes core business logic framework-agnostic, trivially unit-testable, and lets you swap infrastructure technology without touching business rules.
  • CQRS separates command (write) and query (read) handling into distinct code paths — valuable when reads/writes have genuinely different needs, not a default for simple CRUD.
  • MediatR is the common library decoupling "send a request" from "which handler processes it," commonly pairing with CQRS.
  • API versioning (URL segment is most common) lets breaking contract changes ship in a new version while old clients keep working on the old one — the same expand/contract principle as database migrations, applied to API contracts.
