Middleware Pipeline Intermediate

Every request flows through a PIPELINE of middleware components before reaching your endpoint — authentication, CORS, exception handling, logging are all middleware. Understanding the pipeline as a nested chain (not a flat list) explains ordering bugs that trip up nearly every ASP.NET Core developer at some point.

The Pipeline Is a Nested Chain, Not a Flat List

app.Use(async (context, next) => {
    Console.WriteLine("1: before");
    await next();              // calls the NEXT middleware in the chain
    Console.WriteLine("1: after");
});
app.Use(async (context, next) => {
    Console.WriteLine("2: before");
    await next();
    Console.WriteLine("2: after");
});
app.MapGet("/", () => "Hello");

// Output for a request to "/":
//   1: before
//   2: before
//   (endpoint runs, returns "Hello")
//   2: after
//   1: after
// This is why it's a PIPELINE, not a list — each middleware wraps everything after it, like nested
// function calls. Order matters enormously: middleware registered EARLIER wraps middleware registered LATER.
🥪 Think "sandwich," not "assembly line." Each app.Use() call is a layer wrapping the REST of the pipeline. Code before `await next()` runs on the way IN (request); code after `await next()` runs on the way OUT (response) — in REVERSE order of registration. This mental model resolves nearly every "why did my middleware run in this order" confusion.

The Standard Middleware Order (and why it matters)

app.UseExceptionHandler("/error");    // 1. FIRST — must wrap EVERYTHING to catch downstream exceptions
app.UseHttpsRedirection();            // 2. Redirect before anything processes the (insecure) request
app.UseStaticFiles();                 // 3. Serve static files early, short-circuiting before routing/auth
app.UseRouting();                     // 4. Determines WHICH endpoint will handle the request
app.UseCors();                        // 5. Must be AFTER UseRouting, BEFORE UseAuthorization
app.UseAuthentication();              // 6. WHO is making the request (populates User)
app.UseAuthorization();               // 7. Are they ALLOWED — needs routing (to know the endpoint's policy) + authentication
app.MapControllers();                 // 8. Actually invoke the matched endpoint
// Getting this order wrong causes real, hard-to-diagnose bugs: Authorization before Authentication means
// there's no User to check yet; ExceptionHandler registered late means it can't catch exceptions
// thrown by earlier middleware.

Short-Circuiting the Pipeline

app.Use(async (context, next) => {
    if (!context.Request.Headers.ContainsKey("X-Api-Key")) {
        context.Response.StatusCode = 401;
        await context.Response.WriteAsync("Missing API key");
        return;                    // ⚠️ NOT calling next() — the pipeline stops HERE
    }
    await next();                  // only proceeds if the check passes
});
// Middleware can choose NOT to call next(), stopping the request from reaching anything after it —
// this is exactly how authentication/authorization middleware reject unauthorized requests before
// they ever reach your endpoint code.

Writing Custom Middleware — Two Styles

// Inline (quick, good for simple cases):
app.Use(async (context, next) => {
    var sw = Stopwatch.StartNew();
    await next();
    Console.WriteLine($"{context.Request.Path} took {sw.ElapsedMilliseconds}ms");
});

// A dedicated class (better for reuse, testability, DI of dependencies):
public class RequestTimingMiddleware(RequestDelegate next, ILogger logger) {
    public async Task InvokeAsync(HttpContext context) {
        var sw = Stopwatch.StartNew();
        await next(context);
        logger.LogInformation("{Path} took {Ms}ms", context.Request.Path, sw.ElapsedMilliseconds);
    }
}
app.UseMiddleware();

Interview Questions

Q1: Explain the ASP.NET Core middleware pipeline as a nested chain, and why registration order matters.

Each middleware registered via app.Use() (or app.UseXxx() helper methods) receives a reference to the NEXT middleware in the pipeline (conventionally called `next`), and it's up to that middleware whether and when to invoke it. Rather than behaving like a flat sequential list where each item just "runs and moves on," the pipeline is structurally a set of NESTED function calls: the first-registered middleware wraps everything after it, the second wraps everything after IT, and so on — like Russian nesting dolls, or a sandwich where each layer wraps the next. Practically, this means code written BEFORE the `await next()` call inside a middleware runs on the way IN as the request travels toward the endpoint (in registration order — first-registered runs first), while code written AFTER `await next()` runs on the way OUT as the response travels back (in REVERSE registration order — the last middleware to run "before" is the first to run "after"). This explains why registration order is so consequential: an exception handler registered LAST can't catch exceptions thrown by middleware registered earlier (because it never gets to wrap them), while one registered FIRST wraps the entire rest of the pipeline and can catch anything thrown anywhere downstream — which is exactly why UseExceptionHandler is conventionally the very first middleware registered.

Q2: Why must UseAuthentication() come before UseAuthorization() in the pipeline?

These two middleware answer fundamentally different, sequentially-dependent questions. UseAuthentication() is responsible for determining WHO is making the request — it inspects the incoming request (a cookie, a bearer token in the Authorization header, etc.), validates it against the configured authentication scheme, and if valid, populates HttpContext.User with a ClaimsPrincipal representing the authenticated identity (or leaves it as an anonymous/unauthenticated principal if no valid credentials were presented). UseAuthorization() is responsible for determining whether that (now-identified, or explicitly anonymous) user is ALLOWED to access the specific endpoint being requested — checking role requirements, policy requirements, or [Authorize] attributes against HttpContext.User. If UseAuthorization() runs BEFORE UseAuthentication(), it has no populated User to evaluate at all — HttpContext.User would still be in its default, unauthenticated state regardless of whether the request actually included valid credentials, so every authorization check would effectively fail (or behave inconsistently) even for legitimately authenticated users, since the identity information it needs to make that decision was never established. This ordering dependency — you must know WHO someone is before you can decide WHAT they're allowed to do — is precisely why ASP.NET Core's documented standard middleware order always places UseAuthentication() strictly before UseAuthorization().

Q3: What does it mean for middleware to "short-circuit" the pipeline, and give a real-world example.

Short-circuiting means a middleware deliberately chooses NOT to call the `next` delegate it was given, which stops the request from ever reaching any middleware (or the eventual endpoint) registered after it in the pipeline — instead, that middleware writes a response directly and the request processing effectively ends right there, with execution then unwinding back out through the "after next()" portions of only the middleware that DID run before it. This is a deliberate, commonly-used pattern rather than a bug: authentication/authorization middleware is a textbook example — if a request lacks valid credentials or doesn't satisfy an authorization policy, that middleware writes a 401 or 403 response and simply returns without calling next(), so the actual endpoint logic (which might perform expensive database work, or return sensitive data) never executes at all for an unauthorized request. Other common short-circuiting examples include rate-limiting middleware rejecting a request that exceeds a quota with a 429 response, or a custom API-key-check middleware immediately rejecting requests missing a required header, as in the code example above. The general principle is that short-circuiting lets cross-cutting middleware act as a genuine GATE on the rest of the pipeline — cheaply rejecting requests that shouldn't proceed further, before any more expensive downstream work (routing, model binding, business logic, database access) is wasted on a request that was never going to succeed anyway.

🏋️ Practice: add three inline middleware with before/after console output and trace the exact execution order for a request; deliberately swap UseAuthentication() and UseAuthorization() and observe authorized requests start failing; write custom middleware that short-circuits requests missing a header, returning 401 without calling next(); build a dedicated middleware CLASS (not inline) with an injected ILogger; reorder UseExceptionHandler() to run late and confirm it no longer catches an exception thrown by earlier middleware.

🎯 Key Takeaways

  • The middleware pipeline is a NESTED chain, not a flat list — code before `await next()` runs on the way in (registration order); code after runs on the way out (reverse order).
  • Registration order matters critically: UseExceptionHandler must be FIRST (to wrap everything); UseAuthentication must precede UseAuthorization (need to know WHO before checking WHAT they can do).
  • Middleware can short-circuit by not calling next() — the standard mechanism for auth/rate-limiting middleware to reject requests before they reach expensive downstream logic.
  • Write middleware inline (app.Use, quick/simple) or as a dedicated class (app.UseMiddleware<T>, better for reuse/DI/testability).
  • Follow the documented standard order (ExceptionHandler → HttpsRedirection → StaticFiles → Routing → CORS → Authentication → Authorization → endpoints) — deviating causes real, often subtle bugs.
