Production Readiness Production

This closing lesson ties ASP.NET Core to the infrastructure tracks: health checks feed directly into Kubernetes probes, OpenTelemetry gives you observability across a distributed system, and containerizing the app connects back to the Docker track's multi-stage build patterns. This is where "the app works" becomes "the app is actually operable in production."

Health Checks — Feeding Kubernetes Probes Directly

builder.Services.AddHealthChecks()
    .AddDbContextCheck()                          // can we reach the database?
    .AddCheck("redis")                        // is the cache reachable?
    .AddUrlGroup(new Uri("https://payment-provider/health"), "payments");   // is a critical dependency up?

app.MapHealthChecks("/health/live",  new() { Predicate = _ => false });        // liveness: is the PROCESS alive?
app.MapHealthChecks("/health/ready", new() { Predicate = check => true });     // readiness: can it serve traffic?
🔗 This is a direct, load-bearing connection to the Kubernetes track's probes lesson. /health/live should answer "is this process wedged?" — deliberately checking almost NOTHING (no dependencies), because a database outage shouldn't cause Kubernetes to kill and restart a perfectly healthy Pod (the exact liveness anti-pattern that lesson warned about). /health/ready should check real dependencies (DB, cache, critical APIs) — if they're down, the Pod should be pulled from Service endpoints (removed from load balancing) WITHOUT being restarted, since restarting won't fix an external outage.

OpenTelemetry — Distributed Tracing Across Services

builder.Services.AddOpenTelemetry()
    .WithTracing(tracing => tracing
        .AddAspNetCoreInstrumentation()      // auto-trace incoming HTTP requests
        .AddHttpClientInstrumentation()      // auto-trace OUTGOING HTTP calls (e.g., to another microservice)
        .AddEntityFrameworkCoreInstrumentation()   // auto-trace database queries
        .AddOtlpExporter(o => o.Endpoint = new Uri("http://otel-collector:4317")))   // ship traces out
    .WithMetrics(metrics => metrics
        .AddAspNetCoreInstrumentation()
        .AddRuntimeInstrumentation());        // GC, thread pool, memory metrics (the .NET memory lesson's data, exposed)

// A single incoming request that calls two OTHER services gets ONE correlated TRACE spanning all three —
// essential for debugging latency/errors in a microservices architecture (the K8s/CI-CD tracks' world).

Structured Logging

logger.LogInformation("Order {OrderId} created for customer {CustomerId}, total {Total:C}",
    order.Id, order.CustomerId, order.Total);
// NOT: logger.LogInformation($"Order {order.Id} created...")  ← string interpolation loses structure!
// Structured logging keeps OrderId/CustomerId/Total as SEARCHABLE, FILTERABLE FIELDS in your log
// aggregator (Seq, ELK, Application Insights) — not just baked into an opaque message string. This is
// what makes "find every log for order 12345" or "show me the p95 Total across all orders today" possible.

Containerizing an ASP.NET Core App

# Multi-stage build (Docker track) — build with the full SDK, run with just the runtime:
FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build
WORKDIR /src
COPY *.csproj .
RUN dotnet restore
COPY . .
RUN dotnet publish -c Release -o /app

FROM mcr.microsoft.com/dotnet/aspnet:9.0 AS runtime   # runtime-only image — no SDK/compiler shipped
WORKDIR /app
COPY --from=build /app .
USER app                                              # non-root (the Docker security lesson)
ENTRYPOINT ["dotnet", "MyApi.dll"]
# Combined with NativeAOT (the .NET performance lesson) for even smaller, faster-starting images in
# latency-sensitive or serverless deployments.

Graceful Shutdown

// Kubernetes sends SIGTERM before killing a Pod (the K8s debugging lesson's NoExecute taint eviction).
// ASP.NET Core listens for this via IHostApplicationLifetime — finish IN-FLIGHT requests, stop accepting
// NEW ones, close connections cleanly, THEN exit:
app.Lifetime.ApplicationStopping.Register(() => {
    logger.LogInformation("Shutting down gracefully — draining in-flight requests");
});
// Without graceful shutdown handling, rolling deployments/scale-downs can abruptly kill requests
// mid-flight — a direct link to the K8s Deployment rolling-update and readiness-probe behavior.

Interview Questions

Q1: How do ASP.NET Core health checks connect to Kubernetes liveness and readiness probes, and why must they check different things?

ASP.NET Core's health check middleware exposes HTTP endpoints (conventionally /health/live and /health/ready) that Kubernetes's liveness and readiness probes (from the Kubernetes track) call directly to decide a Pod's fate. These two endpoints must be deliberately configured to check very DIFFERENT things, because liveness and readiness failures trigger fundamentally different Kubernetes actions. The liveness endpoint answers "is this process fundamentally wedged/deadlocked and needs to be RESTARTED?" — it should check almost nothing beyond the process actually being able to respond at all (often configured with a Predicate that excludes all registered checks), because if it included a database connectivity check and the database happened to go down, Kubernetes would interpret that as "this Pod is broken" and restart it — but restarting a perfectly healthy application process does absolutely nothing to fix an external database outage, and worse, if EVERY replica's liveness check fails simultaneously due to the same database outage, Kubernetes could end up restart-looping the entire fleet, amplifying an external outage into a self-inflicted one. The readiness endpoint instead answers "can this Pod currently serve traffic successfully RIGHT NOW?" and SHOULD check real dependencies (database, cache, critical external APIs) — if the database is unreachable, readiness should fail, causing Kubernetes to remove that Pod from the Service's load-balanced endpoints (no traffic routed to it) WITHOUT restarting it, which is the correct response since the Pod itself is fine and will resume serving traffic automatically once its dependency recovers and readiness passes again.

Q2: What is structured logging, and why does it matter more in production than plain string messages?

Structured logging means log calls capture the VARIABLE PARTS of a message as separate, named, typed FIELDS — logger.LogInformation("Order {OrderId} created for customer {CustomerId}", order.Id, order.CustomerId) — rather than baking those values into an opaque, already-formatted string via interpolation (logger.LogInformation($"Order {order.Id} created for customer {order.CustomerId}")). The critical difference only becomes apparent once logs leave a developer's console and flow into a production log aggregation system (Seq, the ELK stack, Application Insights, or similar): with structured logging, the log aggregator receives OrderId and CustomerId as genuinely separate, indexed, queryable fields attached to that log entry, so an operator can later run a query like "show me every log entry where OrderId = 12345" or "what's the distribution of Total across all order- creation logs from the last hour" directly against the aggregator — these are essentially database queries against structured log data. With plain string interpolation, all of that information is permanently flattened into a single opaque text blob the moment the log call executes; the log aggregator can still full-text search the resulting string, but it has no idea that "12345" in that string specifically represents an OrderId versus some other number that happened to appear in the message, making precise, reliable querying and aggregation across large volumes of production logs far harder or outright impossible. This distinction is a direct, practical payoff of the same structured-vs-unstructured-data principle that shows up throughout observability tooling, and it's why production ASP.NET Core code should consistently use the message-template placeholder syntax rather than string interpolation in logging calls.

Q3: How does OpenTelemetry help diagnose problems in a microservices architecture?

In a single-service, monolithic application, a slow or failing request can usually be diagnosed by looking at that one application's logs and profiling data. In a microservices architecture (the world the CI/CD and Kubernetes tracks describe), a single incoming user request often triggers a CHAIN of calls across multiple independently-deployed services — an API gateway calls an orders service, which calls a payments service, which calls an external provider — and if that overall request is slow or fails, the critical question "WHICH of these several services/calls actually caused the problem" is very hard to answer by looking at each service's isolated logs separately, since there's no inherent way to correlate "this slow log entry in the payments service" with "this specific slow user request that came in through the gateway 30 seconds earlier." OpenTelemetry solves this with DISTRIBUTED TRACING: it automatically instruments incoming HTTP requests, outgoing HTTP client calls, and database queries (via AddAspNetCoreInstrumentation, AddHttpClientInstrumentation, AddEntityFrameworkCoreInstrumentation), and — critically — it PROPAGATES a shared trace context (trace and span IDs) across service boundaries as part of the outgoing HTTP calls between services, so every operation involved in handling one original user request, no matter how many downstream services it touches, gets tagged as part of the SAME logical trace. When these traces are exported (via an OTLP exporter) to a tracing backend, an operator can pull up the complete trace for one slow request and see a visual timeline/waterfall of every hop it took — the gateway spent 5ms, the orders service spent 200ms mostly waiting on a database query, the payments service spent 3 seconds calling an external provider — immediately revealing exactly where the time went and which specific service or dependency was the actual bottleneck or point of failure, something that would otherwise require painstakingly cross-referencing timestamps across several separate services' logs.

🏋️ Practice: add health checks for a database and a fake dependency, exposing separate /health/live and /health/ready endpoints with different Predicates; wire them into a local Kubernetes deployment's liveness/readiness probes and simulate a dependency outage to confirm the Pod is removed from Service endpoints but NOT restarted; convert a string-interpolated log statement to structured logging and confirm the fields are queryable in your log tool; add OpenTelemetry tracing across two services calling each other and inspect the resulting correlated trace; containerize the app with a multi-stage Dockerfile running as a non-root user.

🎯 Key Takeaways

  • Liveness checks almost nothing (is the process alive?); readiness checks real dependencies (can it serve traffic?) — mismatching these causes cascading restarts during outages (the Kubernetes probes lesson, applied).
  • Use structured logging (named placeholders, not string interpolation) so production log aggregators can filter/query by field, not just full-text search.
  • OpenTelemetry auto-instruments HTTP/database calls and propagates trace context across service boundaries — essential for diagnosing latency/errors in microservices.
  • Containerize with multi-stage builds (SDK to build, runtime-only to ship) running as non-root — direct application of the Docker track's optimization and security lessons.
  • Handle graceful shutdown (IHostApplicationLifetime, SIGTERM) so rolling deployments and scale-downs don't abruptly drop in-flight requests.
