Routing & Model Binding Basics

Routing decides WHICH endpoint handles a request; model binding decides how the request's data becomes typed C# parameters/objects; model VALIDATION (next lesson) decides whether that data is acceptable. This lesson goes deeper into route constraints, route groups, and the binding sources that power both Minimal APIs and Controllers.

Route Templates & Constraints

app.MapGet("/products/{id:int}", (int id) => ...);              // :int constrains AND type-converts
app.MapGet("/products/{sku:regex(^[A-Z]{{3}}-\\d{{4}}$)}", ...);   // regex constraint
app.MapGet("/files/{*path}", (string path) => ...);              // {*path} = catch-all (matches slashes too)
app.MapGet("/orders/{id:int:min(1)}", ...);                        // constraints can be CHAINED
// A request to /products/abc with the :int constraint simply DOESN'T MATCH that route — ASP.NET Core
// tries other routes, or returns 404 if none match. Constraints are a routing-time filter, not validation.

Route Groups — Organizing Related Endpoints

var users = app.MapGroup("/users").WithTags("Users");    // a shared prefix + shared config

users.MapGet("/", GetAllUsers);
users.MapGet("/{id:int}", GetUserById);
users.MapPost("/", CreateUser);
users.MapDelete("/{id:int}", DeleteUser)
     .RequireAuthorization("AdminOnly");                  // per-endpoint config still works within a group

// Groups can also apply shared behavior to ALL endpoints within them:
var api = app.MapGroup("/api/v1").RequireAuthorization().WithOpenApi();
// Route groups reduce repetition (prefix, auth, tags) across many related endpoints — the Minimal API
// answer to what a Controller class's shared route/attributes give you.

Binding Complex Objects & Query Collections

public record ProductFilter(string? Category, decimal? MinPrice, int Page = 1);

app.MapGet("/products", (ProductFilter filter) => ...);
// GET /products?category=Books&minPrice=10&page=2 binds ALL THREE properties automatically from the
// query string onto the record — complex query-parameter binding without manual parsing.

app.MapGet("/search", ([FromQuery] string[] tags) => ...);
// GET /search?tags=red&tags=blue&tags=green binds an ARRAY from repeated query keys.
🎯 Binding source inference follows consistent, learnable rules — but attributes exist for exactly the cases where inference is ambiguous. A complex type with no route-segment match defaults to the BODY in Minimal APIs; if you actually wanted it from the query string (like ProductFilter above works automatically for simple records, but more complex nested types might not), an explicit [FromQuery] resolves the ambiguity. Know the default rules, and know the escape hatches.

Route Ordering & Ambiguity

app.MapGet("/users/me", GetCurrentUser);         // more specific — should be registered/matched first
app.MapGet("/users/{id}", GetUserById);           // less specific — would otherwise "swallow" /users/me
// ASP.NET Core's routing uses a scoring system (more specific segments win), so this specific case
// usually resolves correctly regardless of registration order — but genuinely ambiguous routes throw
// an AmbiguousMatchException at request time, a startup-adjacent bug worth knowing the shape of.

Interview Questions

Q1: What do route constraints actually do, and what DON'T they do?

A route constraint like {id:int} is a ROUTING-TIME filter: it determines whether an incoming request's URL segment is even considered a match for that route template at all, based purely on its shape (can this segment be parsed as an integer?). If the constraint fails — a request to /products/abc against a route requiring {id:int} — that route simply doesn't match, and ASP.NET Core either tries other registered routes or falls through to a 404 if nothing matches; the handler for that route is never even invoked, so there's no opportunity for it to run custom error-handling logic for that specific case. Constraints also perform automatic TYPE CONVERSION as part of matching — a successful :int match means the handler receives an already-parsed int, not a string requiring manual int.Parse(). What constraints explicitly do NOT do is enforce business/application-level VALIDATION rules: a constraint can ensure an id "looks like a valid integer shape," but it cannot check that the id actually corresponds to an existing user, that a value falls within a legal business range beyond simple numeric bounds, or any rule requiring database access or complex logic — those are the job of the model validation pipeline (the next lesson) or explicit checks inside the handler/service layer. The distinction matters because relying on constraints alone gives you clean 404s for malformed URLs, but real data validation still needs to happen separately.

Q2: How does ASP.NET Core decide where to bind a parameter's value from, without explicit attributes?

The framework applies a set of INFERENCE RULES based on the parameter's name and type, in the absence of an explicit [FromX] attribute. If the parameter's name matches a segment defined in the route template (like {id} in "/users/{id}"), it binds from the ROUTE. If the parameter is a simple type (string, int, Guid, and similar primitives/simple types) and doesn't match a route segment, it binds from the QUERY STRING. If the parameter is a COMPLEX type (a class or record with multiple properties, like a request DTO) and isn't itself a recognized special type, it's assumed to come from the REQUEST BODY, deserialized as JSON — this is the default and most common case for POST/PUT payloads. Certain framework-specific types (HttpContext, HttpRequest, CancellationToken) are injected directly regardless of client data. And separately, any parameter type registered in the DI container is treated as a SERVICE dependency rather than client-supplied data at all, resolved automatically just like constructor injection. When these inference rules would produce the wrong result for a particular case — most commonly wanting a complex type bound from the query string instead of the body, or wanting a value explicitly pulled from a header — you use the explicit binding attributes ([FromQuery], [FromBody], [FromHeader], [FromRoute], [FromForm]) to override the default inference and make the binding source unambiguous.

Q3: What's the benefit of route groups (MapGroup), and what problem do they solve?

Without route groups, endpoints that logically belong together — all the operations on a "users" resource, say — end up repeating the same route prefix ("/users/...") on every single MapGet/MapPost call, and any shared cross-cutting configuration (a common authorization policy, API versioning, OpenAPI tags/documentation grouping) has to be applied individually to each endpoint, which is both repetitive and error-prone — it's easy to add a new endpoint to the group and forget to reapply the shared authorization requirement, accidentally leaving it unprotected. MapGroup("/users") creates a grouped route builder that all endpoints registered on it inherit the prefix from, and lets you apply shared configuration ONCE at the group level (.RequireAuthorization(), .WithTags(), .WithOpenApi(), etc.) that automatically applies to every endpoint mapped within that group, while individual endpoints can still layer on their own additional, more specific configuration (like a stricter "AdminOnly" policy on just the delete endpoint). This is the Minimal API equivalent of what a Controller class's shared [Route("api/users")] base attribute and class-level filters/attributes provide in the MVC style — a way to express "these endpoints are related and share this configuration" once, rather than repeating it, reducing both boilerplate and the risk of inconsistently-configured endpoints within the same logical resource.

🏋️ Practice: add route constraints (:int, :min, a regex) to several endpoints and test both valid and invalid requests; create a MapGroup for a resource with a shared prefix and a shared RequireAuthorization, then add one endpoint with an additional stricter policy; bind a complex filter record from query-string parameters; bind a repeated-key array from the query string; deliberately create two ambiguous routes and observe the AmbiguousMatchException, then fix it.

🎯 Key Takeaways

  • Route constraints ({id:int}, :min, :regex) filter routing by shape and auto-convert types — they are NOT business validation.
  • MapGroup shares a prefix and cross-cutting config (auth, tags, OpenAPI) across related endpoints, reducing repetition and configuration drift.
  • Binding sources: route → simple type not in route → query; complex type → body (default); special types and DI services bypass client data entirely.
  • Explicit [FromX] attributes exist specifically to override binding inference when the default would be wrong or ambiguous.
  • Genuinely ambiguous routes throw AmbiguousMatchException at request time — design routes so at most one can match a given request.
