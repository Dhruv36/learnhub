Controllers & MVC Intermediate

Controllers group related endpoints into a class with shared conventions, filters, and automatic behaviors via [ApiController]. Most large, long-lived ASP.NET Core APIs still use controllers — this lesson covers the conventions Minimal APIs don't have, and where the real behavioral differences are.

A Basic API Controller

[ApiController]                              // enables automatic model validation + binding source inference
[Route("api/[controller]")]                  // [controller] = "Orders" (from the class name minus "Controller")
public class OrdersController : ControllerBase {
    private readonly IOrderService _orders;
    public OrdersController(IOrderService orders) => _orders = orders;   // constructor injection, same DI system

    [HttpGet("{id:int}")]                     // GET api/orders/5
    public async Task> Get(int id) {
        var order = await _orders.FindAsync(id);
        return order is null ? NotFound() : Ok(order);
    }

    [HttpPost]                                // POST api/orders
    public async Task> Create(CreateOrderRequest request) {
        var order = await _orders.CreateAsync(request);
        return CreatedAtAction(nameof(Get), new { id = order.Id }, order);
    }
}

What [ApiController] Actually Gives You

BehaviorWithout [ApiController]With [ApiController]
Invalid model stateYou manually check ModelState.IsValidAUTOMATIC 400 response with error details — before your action even runs
Binding source inferenceMust specify [FromBody] etc. explicitly more oftenSmarter inference (similar to Minimal APIs) — less attribute noise
Multipart/form-data & routing requirementsLooserStricter, more consistent API conventions enforced
✅ [ApiController] is why you rarely see manual ModelState.IsValid checks in modern controllers. If a bound request fails data-annotation validation (validation-errors lesson), the framework short-circuits BEFORE your action method body runs at all, returning a standardized 400 response automatically. This is a significant behavioral difference from Minimal APIs, which don't get this for free (you wire validation explicitly there).

Action Results — the Controller Equivalent of Results.X()

return Ok(data);              // 200
return NotFound();            // 404
return CreatedAtAction(nameof(Get), new { id }, data);   // 201 + Location header pointing back at Get
return BadRequest(ModelState);   // 400 (often automatic via [ApiController])
return NoContent();            // 204
// ActionResult lets a single method return EITHER a typed T (implicitly wrapped as 200 Ok) OR any
// other IActionResult (NotFound, BadRequest, etc.) — the same explicit-HTTP-semantics idea as Minimal
// API Results, expressed through the controller-base-class helper methods instead.

Filters — Cross-Cutting Logic Around Actions

public class LogActionFilter : IActionFilter {
    public void OnActionExecuting(ActionExecutingContext context) { /* before the action runs */ }
    public void OnActionExecuted(ActionExecutedContext context) { /* after the action runs */ }
}

[ServiceFilter(typeof(LogActionFilter))]
public class OrdersController : ControllerBase { ... }
// Filters run at specific points in the request pipeline (Authorization → Resource → Action → Exception
// → Result), letting you inject cross-cutting logic (logging, caching, custom auth checks) around MANY
// actions without repeating code in each one — a controller-specific complement to middleware (next lesson).

Interview Questions

Q1: What does the [ApiController] attribute actually change about a controller's behavior?

[ApiController] opts a controller into a set of API-specific conventions designed for building HTTP APIs (as opposed to traditional server-rendered MVC views). The most significant behavioral change is AUTOMATIC MODEL VALIDATION RESPONSE: if the bound request model fails data-annotation validation (from the validation-errors lesson) — a required field missing, a string exceeding a max length — the framework short-circuits BEFORE the action method's body ever executes, automatically returning a 400 Bad Request with a standardized problem-details response describing the validation errors, with zero code required in the action itself; without [ApiController], you'd need to manually check ModelState.IsValid at the top of every action and construct the error response yourself. It also improves BINDING SOURCE INFERENCE, reducing how often you need explicit [FromBody]/[FromQuery] attributes by applying smarter default rules (complex types default to the body, simple types to the query/route, similar to how Minimal APIs infer binding). Finally, it enforces some stricter, more consistent API routing/response conventions overall. In short, it's the attribute that transforms a controller from "a general MVC controller that happens to return JSON" into "a controller specifically tuned for building well-behaved HTTP APIs," and it's why idiomatic modern ASP.NET Core API controllers rarely contain manual validation-checking code.

Q2: How do filters differ from middleware, and when would you use each?

Both let you inject cross-cutting logic without repeating it in every handler, but they operate at different layers and with different granularity. Middleware (the next lesson) runs for EVERY request that passes through the pipeline, regardless of which endpoint (or even whether any endpoint) ends up handling it — it has no inherent awareness of controllers, actions, or MVC-specific concepts, and operates purely on the HttpContext at the level of "a request came in, a response will go out." Filters are an MVC/controller-specific concept that hook into much finer-grained points SPECIFICALLY within the action-invocation pipeline — Authorization filters (run first, can short-circuit before model binding even happens), Resource filters (around model binding), Action filters (immediately before/after the action method itself executes, with access to action arguments and the result), Exception filters (specifically for handling exceptions thrown during action execution), and Result filters (around the result being executed/written to the response). This finer granularity — especially having direct access to the specific action being invoked, its bound arguments, and its return value — makes filters better suited for MVC-specific cross-cutting concerns like custom model validation logic, action-specific caching keyed on arguments, or exception handling scoped to certain controllers, while middleware is the right tool for pipeline-wide concerns that apply uniformly regardless of routing outcome, like authentication, CORS, or request logging. Filters only apply to requests handled by MVC controllers; Minimal API endpoints use a related but separate mechanism (endpoint filters) for similar fine-grained hooks.

Q3: What's the practical difference between choosing Controllers and Minimal APIs for a new project?

Both ultimately run on the identical underlying ASP.NET Core pipeline, DI container, and configuration system, so the choice is about developer ergonomics and structure rather than capability. Controllers group related endpoints into a class, get automatic model validation and richer binding inference for free via [ApiController], and gain access to the fine-grained filter pipeline (Authorization/Resource/Action/Exception/Result filters) — this structure pays off significantly when an API has MANY related endpoints that share substantial cross-cutting conventions (a common route prefix, consistent validation and error-formatting behavior, shared authorization policies, action-level caching), since the class-based, attribute-driven model centralizes and enforces that consistency. Minimal APIs define endpoints as standalone functions with less inherent structure — you don't get automatic validation short-circuiting or the full filter pipeline out of the box, and you wire cross-cutting concerns more explicitly (via middleware, route groups, and endpoint filters) — but this leanness is exactly the appeal for smaller APIs, microservices, or situations where the endpoints don't share much beyond basic infrastructure, since there's less ceremony and less "magic" implicit behavior to reason about. Because both coexist on the same pipeline, teams can and do mix them within one application (e.g., Minimal APIs for a handful of simple endpoints alongside controllers for a larger, more conventional resource), and many teams choose based on existing team familiarity and the specific API's expected size and complexity growth rather than a hard rule.

🏋️ Practice: build a controller with [ApiController] exposing GET/POST/DELETE actions for a resource; send an invalid request body and observe the automatic 400 response with no manual ModelState check; use CreatedAtAction and verify the Location header in the response; implement a custom IActionFilter that logs before/after an action runs and apply it to one controller; compare the same resource implemented as a Minimal API endpoint versus a controller action and note the structural differences.

🎯 Key Takeaways

  • [ApiController] enables automatic 400 responses on invalid model state (short-circuits before the action runs) plus smarter binding inference — the biggest controller-specific behavior.
  • Controllers use constructor injection identically to Minimal APIs — same DI system, just class-based structure.
  • ActionResult<T> (Ok/NotFound/CreatedAtAction/BadRequest/NoContent) is the controller equivalent of Minimal API's Results.X().
  • Filters (Authorization/Resource/Action/Exception/Result) are finer-grained, MVC-specific hooks around action execution — complementary to pipeline-wide middleware.
  • Controllers suit large APIs with many related endpoints sharing conventions; Minimal APIs suit lean/microservice APIs — both run on the same pipeline and can coexist.
