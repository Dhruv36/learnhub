EF Core: Data Access Intermediate

The .NET track's EF Core lesson covered query performance (N+1, tracking). This lesson wires EF Core specifically into an ASP.NET Core app's DI/hosting model, and covers the repository pattern debate and the unit-of-work concept SaveChanges() actually implements — both frequent architecture-interview topics.

Registering DbContext in ASP.NET Core

builder.Services.AddDbContext(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("Default")));
// AddDbContext registers DbContext as SCOPED by default — exactly right for a web app: one DbContext
// instance per HTTP request (the DI & Hosting lesson's captive-dependency rule applies here directly —
// never register a DbContext as Singleton, and never inject it into a Singleton service directly).

Injecting DbContext into a Service

public class OrderService(AppDbContext db) : IOrderService {
    public async Task FindAsync(int id) =>
        await db.Orders.AsNoTracking().FirstOrDefaultAsync(o => o.Id == id);

    public async Task CreateAsync(CreateOrderRequest request) {
        var order = new Order { CustomerId = request.CustomerId, Total = request.Total };
        db.Orders.Add(order);
        await db.SaveChangesAsync();      // persists ALL pending changes in ONE transaction
        return order;
    }
}
// Because DbContext is Scoped and OrderService is typically also Scoped, they share the SAME DbContext
// instance for the whole request — multiple repository/service calls within one request see a
// CONSISTENT view of pending changes before SaveChangesAsync() commits them.

SaveChanges() as the Unit of Work

// A single SaveChangesAsync() call wraps ALL tracked changes — inserts, updates, deletes accumulated
// across the DbContext during the request — into ONE database transaction:
db.Orders.Add(newOrder);
db.Customers.Update(existingCustomer);       // marks it as changed (tracked entity)
db.Products.Remove(discontinuedProduct);
await db.SaveChangesAsync();                  // ALL THREE changes commit together, or NONE do (atomicity)
// This is the "Unit of Work" pattern: DbContext already IS one — you don't need to hand-roll your own
// wrapper around it. Understanding this is why some teams argue a separate Repository/UnitOfWork
// abstraction over EF Core is often unnecessary indirection (see the interview question below).

The Repository Pattern Debate

Argument FOR a repository layerArgument AGAINST (using DbContext/DbSet directly)
Abstracts away EF Core specifics for easier swapping/testingDbContext IS ALREADY an abstraction (over ADO.NET/SQL) and IS ALREADY testable via a real in-memory provider

Enforces a consistent data-access boundaryDbSet<T> + LINQ already IS a clean, well-understood boundary
Useful if you might swap ORMs entirelyRarely happens in practice; premature abstraction for a hypothetical
⚖️ Modern guidance leans toward using DbContext directly (via injected services), not wrapping it in a generic Repository<T>. EF Core's DbContext + DbSet<T> already provides testability (via EF Core's in-memory or SQLite providers for tests) and a clean data-access boundary. A thin generic repository often just re-implements a WORSE version of IQueryable<T> while adding an indirection layer. Domain-specific repositories with meaningful, business-focused methods (not generic CRUD wrappers) remain a reasonable pattern — the anti-pattern is a repository that offers nothing beyond what DbContext already gives you.

Interview Questions

Q1: Why must DbContext be registered as Scoped in an ASP.NET Core application?

DbContext is explicitly NOT thread-safe — it maintains internal state (the change tracker, an open connection to the database for the duration of its use) that isn't designed to be accessed concurrently from multiple threads. In a web application, each incoming HTTP request is typically processed on its own logical flow of execution, and Scoped is precisely the lifetime that gives "one instance per request" — every service resolved during that single request's handling that depends on DbContext gets the SAME instance, ensuring a consistent, single-threaded view of tracked entities and pending changes for that request, and the DbContext (along with its underlying connection) is properly disposed at the end of the request. If DbContext were registered as Singleton, every request across every concurrent user would share the exact same instance, and multiple simultaneous requests calling into it concurrently would cause thread-safety violations, subtle data corruption, or crashes — this connects directly to the general DI lifetime rule from the .NET track that a Singleton must never depend on (or effectively become) something request-scoped, since DbContext's very nature assumes single-request usage. AddDbContext() registers it as Scoped by default specifically to enforce this correct behavior automatically.

Q2: What is the "Unit of Work" pattern, and how does EF Core's DbContext already implement it?

The Unit of Work pattern describes tracking a set of related changes (inserts, updates, deletes) made during a single business operation and committing them all together as one atomic transaction — either everything succeeds, or nothing does, so the database never ends up in a partially-updated, inconsistent state if something fails partway through. EF Core's DbContext already implements exactly this: as you call db.Orders.Add(...), mark an existing tracked entity's properties as changed, or call db.Products.Remove(...), the DbContext accumulates all of these as PENDING changes in its internal change tracker without touching the database at all; only when you call SaveChangesAsync() does EF Core translate the ENTIRE accumulated set of changes into the appropriate SQL statements and execute them within a single database transaction, committing all of them together or rolling back entirely on failure. This is precisely why a request handler can perform several logically related mutations across multiple DbSets (adding an order, updating a customer's order count, removing a discontinued product) and call SaveChangesAsync() just once at the end to commit them all atomically as one unit of work — you don't need to hand-build a separate UnitOfWork wrapper class around DbContext, because DbContext already fills that role natively.

Q3: Should you wrap EF Core's DbContext in a generic Repository pattern? What's the modern guidance?

Historically, wrapping data access behind a generic Repository<T> interface (with methods like GetById, GetAll, Add, Update, Delete) was common advice, often justified by "abstracting away the ORM" and "making data access testable." Modern guidance in the EF Core/ASP.NET Core community increasingly pushes back on this for a specific reason: DbContext and DbSet<T> ARE ALREADY a well-designed abstraction — DbContext sits above raw ADO.NET/SQL, and DbSet<T> exposes a full, powerful IQueryable<T> LINQ surface — and it's already directly testable via EF Core's in-memory or SQLite test providers, without needing a hand-rolled interface in front of it. A generic Repository<T> wrapping this typically ends up either exposing a THINNER, LESS capable API than DbSet<T> already offers (forcing awkward workarounds when a query needs something the repository interface didn't anticipate), or it just re-implements IQueryable<T> badly, adding an indirection layer with limited real benefit while making it HARDER to see and optimize the actual queries being generated (a relevant concern given the N+1/tracking issues from the .NET EF Core lesson). The more nuanced modern position isn't "never use a repository" but rather: avoid a GENERIC, CRUD-only repository that adds nothing beyond what DbSet<T> provides; a domain-specific service or repository with meaningful, business-focused methods (like IOrderService.GetOverdueOrdersForCustomerAsync(customerId)) that happens to use DbContext internally is a perfectly reasonable and common pattern — the distinction is whether the abstraction adds real value (encapsulating a meaningful business query) or is pure indirection around operations DbContext already does well.

🏋️ Practice: register a DbContext with AddDbContext and inject it into a Scoped service; perform several related mutations across multiple DbSets and confirm they commit together in one SaveChangesAsync() call; deliberately try injecting DbContext into a Singleton and observe the captive-dependency issue (or startup validation error); write a domain-specific service method with a meaningful business query, then compare it to what a generic Repository<T> wrapper would look like for the same need; write and apply a migration, then run it as a separate deployment step rather than on app startup.

🎯 Key Takeaways

  • AddDbContext registers DbContext as Scoped by default — one instance per HTTP request, matching its non-thread-safe design; never make it Singleton.
  • Multiple mutations across different DbSets accumulate as pending changes and commit together atomically in one SaveChangesAsync() call — DbContext already IS the Unit of Work.
  • Modern guidance leans away from generic Repository<T> wrappers — DbContext/DbSet<T> is already a clean, testable abstraction; prefer domain-specific services with meaningful methods if you add a layer at all.
  • Injecting DbContext into a Singleton is the classic captive-dependency bug from the .NET DI lesson — use IServiceScopeFactory if a Singleton genuinely needs data access.
  • Run migrations as a deliberate, separate deployment step rather than automatically on every app startup in production.
