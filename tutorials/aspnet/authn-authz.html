AuthN/AuthZ (Identity, JWT, OIDC) Advanced

Authentication (who are you?) and authorization (what can you do?) are distinct concerns handled by separate middleware (the previous lesson's ordering rule). This lesson covers the standard JWT bearer token flow for APIs, the role vs policy authorization models, and how OIDC fits in for delegating login to an external identity provider.

JWT Bearer Authentication — the Standard API Auth Flow

builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options => {
        options.TokenValidationParameters = new TokenValidationParameters {
            ValidateIssuer = true,   ValidIssuer = "https://auth.example.com",
            ValidateAudience = true, ValidAudience = "my-api",
            ValidateLifetime = true,                       // reject EXPIRED tokens
            IssuerSigningKey = new SymmetricSecurityKey(key)   // verify the SIGNATURE wasn't tampered with
        };
    });

app.UseAuthentication();     // populates HttpContext.User FROM a valid token's claims
app.UseAuthorization();

[Authorize]                  // requires ANY authenticated user
[HttpGet("profile")]
public IActionResult GetProfile() => Ok(new { Name = User.Identity!.Name });
🔑 A JWT is a self-contained, signed claim — the server doesn't "look it up" anywhere. Unlike a traditional session cookie (which is a reference the server looks up in a session store), a JWT carries its claims (user ID, roles, expiry) directly, cryptographically signed. Validation is purely computational: check the signature matches, check it hasn't expired, check issuer/audience match — no database round-trip needed. This statelessness is exactly why JWTs scale so well for APIs (no shared session store required across server instances).

Role-Based vs Policy-Based Authorization

[Authorize(Roles = "Admin")]                    // simple: requires the "Admin" role claim
public IActionResult DeleteUser(int id) { ... }

// Policies express RICHER rules than a single role check:
builder.Services.AddAuthorization(options => {
    options.AddPolicy("MinimumAge18", policy =>
        policy.RequireAssertion(ctx =>
            ctx.User.HasClaim(c => c.Type == "age") &&
            int.Parse(ctx.User.FindFirst("age")!.Value) >= 18));
});

[Authorize(Policy = "MinimumAge18")]
public IActionResult BuyRestrictedItem() { ... }
// Policies can combine multiple claims, custom logic, or even injected services (via a custom
// IAuthorizationHandler) — roles are really just a convenient special case of policy-based auth.

OIDC — Delegating Login to an External Identity Provider

// OAuth 2.0 handles AUTHORIZATION (delegated access — "let this app act on my behalf").
// OpenID Connect (OIDC) is built ON TOP of OAuth 2.0 to ALSO handle AUTHENTICATION (who is this user?)
// — it standardizes an ID TOKEN (a specific JWT) that proves identity, alongside the access token.
//
// The typical flow (Authorization Code flow, the recommended one for web apps):
//   1. Your app redirects the user to the identity provider (Auth0, Azure AD, Google, Okta...)
//   2. User logs in THERE (your app never sees their password)
//   3. Provider redirects back with an authorization CODE
//   4. Your app exchanges the code for an ID token (identity) + access token (API access) — server-side
//   5. Your app validates the ID token and establishes the user's session
// This is why "Sign in with Google/Microsoft" buttons work — OIDC is the standard underneath them.
🎯 OAuth2 ≠ authentication; OIDC = OAuth2 + a standardized identity layer. A classic interview trap: OAuth 2.0 alone was designed for AUTHORIZATION (delegated, scoped access to a resource on someone's behalf), not for proving WHO someone is — using raw OAuth2 as a login mechanism was a common historical misuse. OIDC fixes this properly by adding the ID token and a standardized way to fetch user profile info, making it the correct protocol for "let users log in with their Google/Microsoft account."

ASP.NET Core Identity — Building Your Own User System

// When you own the user store yourself (not delegating to Google/Auth0), ASP.NET Core Identity provides
// user management, password hashing, and login out of the box:
builder.Services.AddIdentity()
    .AddEntityFrameworkStores();
// Handles: password hashing (never store plaintext!), account lockout after failed attempts, email
// confirmation, two-factor auth, role management — a lot of security-sensitive plumbing you don't
// want to hand-roll yourself.

Interview Questions

Q1: How does JWT bearer authentication work, and why is it well-suited to APIs?

A JSON Web Token is a self-contained, cryptographically SIGNED piece of data encoding a set of claims — typically who the user is (subject/user ID), what they're allowed to do (roles/scopes), and when the token expires. In the typical flow, a client authenticates once (via a login endpoint or an external identity provider) and receives a JWT, which it then includes in the Authorization header (as a "Bearer" token) on every subsequent API request. On the server, JWT bearer authentication middleware validates the token PURELY COMPUTATIONALLY: it verifies the cryptographic signature matches (proving the token wasn't tampered with and was genuinely issued by a trusted party holding the corresponding signing key), checks that the token hasn't expired, and validates that the issuer and intended audience match expectations — critically, none of this requires a database lookup or call to a session store, unlike traditional server-side session cookies where the server has to look up session state by a session ID on every request. This statelessness is exactly why JWTs scale so well for APIs: any server instance, without any shared session state, can independently validate any valid token, which is essential for horizontally-scaled, stateless API deployments (directly relevant to the Kubernetes track's scaling story) where you can't guarantee two consecutive requests hit the same server instance.

Q2: What's the difference between OAuth 2.0 and OpenID Connect (OIDC)?

OAuth 2.0 was designed to solve AUTHORIZATION — specifically, DELEGATED access: letting a third-party application access a specific resource on a user's behalf, with a specific scope of permissions, without that application ever seeing the user's actual password (the classic example is "let this app read my Google Calendar"). Crucially, OAuth 2.0 by itself says nothing standardized about AUTHENTICATION — proving WHO a user actually is — and historically, developers sometimes misused a successfully-obtained OAuth2 access token as a stand-in for "this proves who the user is," which is not what access tokens were designed to guarantee or standardize. OpenID Connect is built as a well-defined identity layer ON TOP of OAuth 2.0, specifically adding what OAuth2 lacked for authentication: it standardizes an ID TOKEN — itself a specific, well-defined JWT — that reliably carries verified identity claims about the authenticated user (their unique subject identifier, name, email, when they authenticated), plus a standardized way for the client application to fetch additional user profile information from the identity provider. So the practical distinction is: use OAuth2 (alone) when you need delegated, scoped ACCESS to an API/resource on a user's behalf; use OIDC when you need to actually AUTHENTICATE a user — establish and verify who they are, which is what powers "Sign in with Google/Microsoft/Auth0" login flows.

Q3: What's the difference between role-based and policy-based authorization in ASP.NET Core?

Role-based authorization, expressed with [Authorize(Roles = "Admin")], is the simplest authorization model: it checks whether the authenticated user's claims include membership in one of the specified role(s), and grants or denies access based purely on that single check. It's easy to understand and sufficient for straightforward "only admins can do this" scenarios. Policy-based authorization is a more general and flexible mechanism: you register a named POLICY (via builder.Services.AddAuthorization(options => options.AddPolicy(...))) that can express arbitrarily rich authorization logic — combining multiple claims together (requiring both a specific role AND a minimum "age" claim value), custom assertion functions evaluating the ClaimsPrincipal however you need, or even fully custom logic via a dedicated IAuthorizationHandler class that can have services INJECTED into it (for example, checking a database to see if the user owns the specific resource being accessed — resource-based authorization). You then apply the policy declaratively with [Authorize(Policy = "MinimumAge18")]. The key insight is that role-based authorization is really just a simple, built-in special case OF the more general policy-based system — roles map naturally onto a particular kind of claim check — so reaching for policies isn't "instead of" roles, it's the more powerful, extensible tool you graduate to whenever authorization logic needs to be more sophisticated than a single role membership check, all while keeping the same clean, declarative [Authorize] attribute syntax on your actions.

🏋️ Practice: configure JWT bearer authentication with full TokenValidationParameters and protect an endpoint with [Authorize]; issue a token, then deliberately tamper with it and confirm validation rejects it; add a role-based [Authorize(Roles="Admin")] endpoint and test with tokens that do/don't have the role; define a custom policy combining two claims and apply it; sketch (or implement with a test provider) the OIDC authorization code flow for "Sign in with Google"; set up ASP.NET Core Identity for a self-hosted user store and register a new user.

🎯 Key Takeaways

  • Authentication (UseAuthentication, WHO) must run before Authorization (UseAuthorization, WHAT) — the middleware pipeline lesson's ordering rule applied to real auth.
  • JWTs are self-contained, signed claims validated computationally (signature, expiry, issuer/audience) with no database lookup — this statelessness is why they scale for APIs.
  • Policy-based authorization generalizes role-based authorization — roles are a simple built-in special case of the richer policy system (custom assertions, injected services, resource-based checks).
  • OAuth 2.0 = delegated AUTHORIZATION (scoped access on someone's behalf); OIDC = OAuth2 + a standardized identity layer (ID token) for actual AUTHENTICATION — don't misuse OAuth2 access tokens as proof of identity.
  • ASP.NET Core Identity handles security-sensitive plumbing (password hashing, lockout, 2FA) for self-hosted user stores — never hand-roll password storage.
