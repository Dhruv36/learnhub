Validation & Error Handling Intermediate

An API's job isn't just to succeed correctly — it's to FAIL predictably, with a consistent, machine-readable error format clients can rely on. This lesson covers data-annotation validation, FluentValidation for complex rules, and centralizing exception handling so error responses are consistent across the whole API instead of ad hoc per endpoint.

Data Annotations — Declarative Validation

public class CreateOrderRequest {
    [Required, StringLength(100, MinimumLength = 1)]
    public string CustomerName { get; set; } = "";

    [Range(0.01, 100000)]
    public decimal Total { get; set; }

    [EmailAddress]
    public string? ContactEmail { get; set; }
}
// With [ApiController] (previous lesson), a request failing these rules is automatically rejected with
// a 400 BEFORE the action method runs — no manual checking needed. Without [ApiController] (Minimal
// APIs, or MVC without the attribute), you must check ModelState.IsValid explicitly.

Data Annotations in Minimal APIs — Not Automatic

app.MapPost("/orders", ([AsParameters] CreateOrderRequest request, IValidator validator) => {
    var result = validator.Validate(request);
    if (!result.IsValid) return Results.ValidationProblem(result.ToDictionary());
    // ... proceed
});
// Minimal APIs do NOT automatically enforce data annotations the way [ApiController] does — you must
// validate explicitly (often via a library like FluentValidation, or an endpoint filter that runs
// validation for every endpoint in a group, avoiding per-endpoint repetition).

FluentValidation — Complex, Composable Rules

public class CreateOrderValidator : AbstractValidator {
    public CreateOrderValidator() {
        RuleFor(x => x.CustomerName).NotEmpty().MaximumLength(100);
        RuleFor(x => x.Total).GreaterThan(0);
        RuleFor(x => x.Items).NotEmpty().WithMessage("An order must have at least one item");
        RuleFor(x => x).Must(HaveConsistentTotal).WithMessage("Total doesn't match item sum");   // cross-field
    }
    private bool HaveConsistentTotal(CreateOrderRequest r) => r.Total == r.Items.Sum(i => i.Price);
}
// FluentValidation separates validation logic from the model class entirely (no attributes cluttering
// the DTO), supports rules data annotations can't express (cross-field checks, conditional rules,
// injecting a service into the validator for a DB-backed rule like uniqueness).
🎯 Data annotations vs FluentValidation is a "simple vs complex rules" trade-off. Annotations are concise and colocated for simple per-property rules ([Required], [Range]); FluentValidation shines for rules spanning multiple properties, conditional logic, or rules needing injected dependencies (checking a value's uniqueness against the database). Many production APIs use both — annotations for the easy cases, FluentValidation where annotations fall short.

Centralized Exception Handling — One Place, Consistent Format

// ❌ try/catch scattered in every action — repetitive, inconsistent error formats across endpoints:
[HttpPost]
public async Task Create(CreateOrderRequest request) {
    try { return Ok(await _orders.CreateAsync(request)); }
    catch (Exception ex) { return StatusCode(500, new { error = ex.Message }); }   // leaks internals!
}

// ✅ A GLOBAL exception handler — one place, consistent shape, no leaked internals:
app.UseExceptionHandler(errApp => errApp.Run(async context => {
    var feature = context.Features.Get();
    var (status, title) = feature?.Error switch {
        ValidationException => (400, "Validation failed"),
        NotFoundException   => (404, "Resource not found"),
        _                    => (500, "An unexpected error occurred")   // never leak ex.Message for 500s
    };
    context.Response.StatusCode = status;
    await context.Response.WriteAsJsonAsync(new ProblemDetails { Status = status, Title = title });
}));

ProblemDetails — the RFC 7807 Standard Error Shape

{
  "type": "https://tools.ietf.org/html/rfc7231#section-6.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "errors": { "CustomerName": ["The CustomerName field is required."] }
}
// ASP.NET Core's automatic [ApiController] validation responses (and Results.ValidationProblem in
// Minimal APIs) already use this standardized shape — consistent, well-understood by API clients and
// tooling, rather than every team inventing its own ad hoc error JSON format.

Interview Questions

Q1: Why is a global exception handler preferable to try/catch in every action?

Scattering try/catch blocks across every controller action or Minimal API handler leads to several real problems: it's repetitive boilerplate duplicated everywhere; it's easy to forget on a new endpoint, leaving that one unprotected; and without strict discipline, different developers end up producing inconsistently-SHAPED error responses across different endpoints (one returns { error: "..." }, another returns a plain string, another leaks the raw exception message) — which makes the API harder for clients to consume reliably, since error handling can't be written generically against a consistent contract. A global exception handler, configured once via app.UseExceptionHandler (or middleware you write yourself), catches ANY unhandled exception that propagates up from ANYWHERE in the request pipeline, in ONE centralized place, and is responsible for mapping exception types to appropriate HTTP status codes and producing a consistently-shaped error response (ideally standardized as ProblemDetails/RFC 7807) for every single endpoint in the application, without any of them needing their own try/catch at all for the general case. This also creates a single, safe checkpoint to ensure sensitive internal details (stack traces, raw exception messages, connection strings) are never accidentally leaked back to a client — the handler can log the full exception detail server-side while returning only a generic, safe message externally for truly unexpected (500-level) errors.

Q2: Why don't Minimal APIs automatically validate data annotations the way [ApiController]-decorated controllers do?

[ApiController]'s automatic model-state validation is a specific, deliberate MVC-framework feature — it hooks into the controller action-invocation pipeline (specifically, an implicit Authorization/ Resource filter under the hood) to check ModelState.IsValid after binding and BEFORE the action method runs, short-circuiting with a 400 if validation fails. Minimal APIs were designed as a leaner, lower-ceremony alternative that intentionally doesn't carry over every MVC convention automatically — they don't have an equivalent implicit validation step wired in by default, so a request with an invalid CreateOrderRequest (missing a required field, an out-of-range value) will bind successfully and reach the handler's body regardless of whether it satisfies its data annotations, unless you explicitly validate it. In practice this means Minimal API projects need to deliberately wire up validation themselves — commonly via a library like FluentValidation invoked explicitly at the top of each handler, or (in more recent ASP.NET Core versions) an ENDPOINT FILTER registered once on a route group that runs validation for every endpoint within that group automatically, achieving similar centralization to what [ApiController] gives controllers, but requiring an explicit opt-in rather than coming for free. This is an important practical gotcha when migrating between the two styles or choosing between them for a new project.

Q3: When would you choose FluentValidation over simple data annotations?

Data annotations are well suited to simple, PER-PROPERTY rules that can be expressed as a single attribute directly on a model's property — [Required], [StringLength], [Range], [EmailAddress] — and their concise, colocated syntax (the rule sits right next to the property it validates) makes them easy to read for straightforward cases. FluentValidation becomes the better choice once validation logic outgrows what a single attribute can naturally express. Specifically: CROSS-FIELD rules that need to compare multiple properties together (like confirming an order's Total actually equals the sum of its line items' prices) don't map cleanly onto a single-property attribute at all. CONDITIONAL rules (a field is required only if another field has a certain value) become awkward with annotations but are natural to express fluently (When(...) clauses). Rules requiring an INJECTED DEPENDENCY — like checking that an email address isn't already registered, which requires querying the database — can't be expressed with a stateless attribute at all, since attributes have no access to DI, but a FluentValidation validator class can have services injected into its constructor just like any other class. FluentValidation also fully separates validation logic out of the model/DTO class entirely, which some teams prefer for keeping models as plain data-holders without validation-specific attributes cluttering their definition. In practice, many production APIs use BOTH — data annotations for straightforward per-property constraints, and FluentValidation specifically for the more complex, cross-cutting, or dependency-requiring rules that annotations genuinely can't express well.

🏋️ Practice: add data annotations to a request model in an [ApiController]-decorated controller and confirm invalid requests are auto-rejected with 400 before the action runs; build the equivalent Minimal API endpoint and observe validation is NOT automatic, then wire up FluentValidation explicitly; write a FluentValidation rule spanning two properties (cross-field); implement a global exception handler mapping several custom exception types to appropriate status codes, ensuring 500 responses never leak exception messages; return a ProblemDetails-shaped error and inspect its JSON structure.

🎯 Key Takeaways

  • Data annotations give concise per-property validation, auto-enforced by [ApiController] controllers (400 before the action runs) but NOT automatic in Minimal APIs — those need explicit validation wiring.
  • FluentValidation handles what annotations can't: cross-field rules, conditional logic, and rules needing injected dependencies (e.g., DB-backed uniqueness checks).
  • A single global exception handler (app.UseExceptionHandler) replaces scattered try/catch, giving consistent error shapes and a safe checkpoint against leaking internal details.
  • Never return raw exception messages/stack traces to clients for 500-level errors — log full detail server-side, return a generic safe message externally.
  • Standardize error responses on ProblemDetails (RFC 7807) — the same shape ASP.NET Core's built-in validation responses already use.
