Dependency Injection & Configuration Basics

The .NET DI & Hosting lesson covered service lifetimes in the abstract; this lesson applies them specifically to ASP.NET Core web apps, plus how configuration (appsettings.json, environment variables, secrets) layers together — and how the two connect via the Options pattern, one of the most-used production patterns in real ASP.NET Core codebases.

Configuration Sources & Precedence

// Configuration is built from MULTIPLE sources, layered in order — LATER sources OVERRIDE earlier ones:
//   1. appsettings.json                       (base config, checked into source control)
//   2. appsettings.{Environment}.json          (e.g., appsettings.Development.json — env-specific overrides)
//   3. Environment variables                   (typically how secrets/prod config are injected — CI/CD lesson)
//   4. Command-line arguments                  (highest precedence)
// This layering is exactly the Docker/K8s "config from the environment" principle applied to .NET —
// one deployable app, config supplied per-environment rather than baked in.
// appsettings.json:
{ "Smtp": { "Host": "smtp.example.com", "Port": 587, "EnableSsl": true } }

// Read a raw value:
var host = builder.Configuration["Smtp:Host"];             // colon = nested key path

// Environment variable OVERRIDE uses double-underscore for nesting:
// Smtp__Host=prod-smtp.example.com   (overrides the JSON value when set)

The Options Pattern — Strongly-Typed, Injectable Configuration

public class SmtpOptions {
    public string Host { get; set; } = "";
    public int Port { get; set; }
    public bool EnableSsl { get; set; }
}

builder.Services.Configure(builder.Configuration.GetSection("Smtp"));   // bind the section

public class EmailService(IOptions options) {   // inject strongly-typed config
    private readonly SmtpOptions _smtp = options.Value;
    public Task SendAsync(string to, string subject) {
        // use _smtp.Host, _smtp.Port, _smtp.EnableSsl — no magic strings, no manual parsing
    }
}
// IOptions is itself resolved through DI — configuration becomes just another injectable dependency,
// strongly typed and validated once at binding time instead of re-read/re-parsed from strings everywhere.
🔗 The Options pattern is where DI and configuration meet. Instead of services reaching into IConfiguration directly with magic string keys scattered across the codebase, you bind a config section to a POCO once and inject the strongly-typed result — refactor-safe (rename a property, the compiler catches every usage), testable (inject a fake IOptions<T> in tests), and centralizes what configuration a service actually depends on.

IOptions vs IOptionsSnapshot vs IOptionsMonitor

InterfaceBehaviorUse for
IOptions<T>Singleton — value computed ONCE at startup, never changesConfig that never changes at runtime
IOptionsSnapshot<T>Scoped — re-read PER REQUEST if the underlying source changedConfig that may change and you want it fresh per-request
IOptionsMonitor<T>Singleton with CHANGE NOTIFICATIONS — .OnChange() callbackLong-lived services (background workers) reacting to config changes live

Environments — Development, Staging, Production

if (app.Environment.IsDevelopment()) {
    app.UseSwagger();               // only expose API docs/detailed errors in dev
    app.UseDeveloperExceptionPage();
} else {
    app.UseExceptionHandler("/error");   // production: a safe, generic error handler (validation-errors lesson)
}
// ASPNETCORE_ENVIRONMENT env var (Development/Staging/Production) drives this — the same "one image,
// config from the environment" principle from the Docker track, now expressed in application code.

Interview Questions

Q1: How does ASP.NET Core's configuration system layer multiple sources together?

ASP.NET Core builds its configuration from an ORDERED sequence of providers, each of which can contribute or OVERRIDE keys set by earlier providers — later sources win on key conflicts. The typical default order is: appsettings.json first (the base configuration, safe to commit to source control since it shouldn't contain secrets), then appsettings.{EnvironmentName}.json (e.g., appsettings.Development.json or appsettings.Production.json, letting you override specific values per environment), then environment variables (commonly how actual secrets and environment-specific production values get injected at deploy time, using double-underscore syntax like Smtp__Host to represent nested JSON keys), and finally command-line arguments, which take the highest precedence of all. This layered, override-based model directly mirrors the "one image, configuration supplied by the environment" principle from containerized deployments (the Docker track) — the same compiled application artifact runs unchanged across dev/staging/production, with only the configuration values differing per environment, typically injected via environment variables in a container or Kubernetes context (ConfigMaps/Secrets) rather than baked into the build.

Q2: What problem does the Options pattern (IOptions<T>) solve compared to reading IConfiguration directly?

Reading configuration values directly via IConfiguration["Smtp:Host"] scattered across many services has several real drawbacks: the key is a magic string with no compile-time checking (a typo silently returns null instead of failing to build), there's no single place documenting what shape of configuration a given service actually needs, and every consumer has to handle parsing/converting string values to the right types itself. The Options pattern solves this by defining a plain POCO class matching the shape of a configuration section (like SmtpOptions with Host/Port/EnableSsl properties), binding a configuration section to it ONCE via builder.Services.Configure<SmtpOptions>(builder.Configuration.GetSection("Smtp")), and then injecting IOptions<SmtpOptions> (or a related variant) into any service that needs it, accessing the bound, already-typed values via .Value. This makes configuration dependencies EXPLICIT (a service's constructor signature literally documents what config section it depends on), refactor-safe (renaming a property is caught by the compiler everywhere it's used, unlike a string key), and testable (a unit test can construct a fake IOptions<SmtpOptions> with test values and inject it directly, with no need for a real configuration source at all) — configuration becomes just another strongly-typed dependency resolved through the same DI system as everything else.

Q3: What's the difference between IOptions<T>, IOptionsSnapshot<T>, and IOptionsMonitor<T>?

They differ in how — and whether — they respond to configuration changing at runtime (e.g., an admin editing an appsettings.json file, or an underlying config provider like Azure App Configuration pushing an update, while the application is already running). IOptions<T> is registered as a Singleton and computes its bound value ONCE, the first time it's resolved, and never re-reads it again for the lifetime of the application — appropriate for configuration that genuinely never changes without a full application restart, which is the most common case and the simplest to reason about. IOptionsSnapshot<T> is registered as Scoped, meaning it re-evaluates the configuration binding once PER SCOPE (in a web app, effectively once per HTTP request) — so if the underlying configuration source has changed since the last request, a new request sees the updated values, without needing an application restart, at the cost of re-binding on every request. IOptionsMonitor<T> is registered as a Singleton but provides an explicit CHANGE-NOTIFICATION mechanism: you can call .CurrentValue at any time to get the latest bound configuration, and you can subscribe to live updates via .OnChange(callback), which is essential for long-lived services that aren't scoped to a request at all — like a BackgroundService or a singleton cache — that still need to react to configuration changing while they're running, since they can't rely on a new Scoped instance being created per request the way a typical web request handler can.

🏋️ Practice: define an appsettings.json section and bind it to a POCO with the Options pattern, injecting IOptions<T> into a service; override a config value with an environment variable using the double-underscore syntax and confirm it takes precedence; create an appsettings.Development.json override and switch ASPNETCORE_ENVIRONMENT to see it take effect; register a service using IOptionsMonitor<T> and subscribe to .OnChange(); gate Swagger/detailed error pages behind app.Environment.IsDevelopment().

🎯 Key Takeaways

  • Configuration layers multiple sources (appsettings.json → per-environment JSON → env vars → command-line args), later sources overriding earlier ones — mirrors the "config from the environment" container principle.
  • The Options pattern binds a config section to a strongly-typed POCO once, injected via IOptions<T> — refactor-safe, testable, no scattered magic strings.
  • IOptions<T> (Singleton, read once) vs IOptionsSnapshot<T> (Scoped, re-read per request) vs IOptionsMonitor<T> (Singleton with live .OnChange() notifications for long-lived services).
  • ASPNETCORE_ENVIRONMENT drives environment-specific behavior (Swagger/detailed errors in dev only, generic error handling in production).
  • Never commit real secrets in appsettings.json — inject them via environment variables or a secrets manager, consistent with the Docker env-secrets lesson.
