SignalR (Realtime) Advanced

HTTP request/response is a poor fit for servers pushing updates to clients unprompted — chat messages, live notifications, collaborative editing. SignalR is ASP.NET Core's real-time communication library: it picks the best available transport automatically and gives you a simple RPC-style programming model over WebSockets (with graceful fallbacks).

Transport Negotiation — the Core Idea

// SignalR abstracts over THREE possible transports, negotiating the best one the client/server/network
// actually support, transparently:
//   1. WebSockets      — full-duplex, persistent connection (the ideal case, used when available)
//   2. Server-Sent Events — server-to-client streaming over HTTP (one-way, fallback)
//   3. Long Polling     — client repeatedly asks "anything new?" (works almost everywhere, least efficient)
// Your application code is IDENTICAL regardless of which transport is actually negotiated — SignalR
// picks WebSockets when possible and silently falls back for older browsers/restrictive networks/proxies.

A Hub — the Server-Side RPC Endpoint

public class ChatHub : Hub {
    public async Task SendMessage(string user, string message) {
        await Clients.All.SendAsync("ReceiveMessage", user, message);   // broadcast to EVERY connected client
    }
    public override async Task OnConnectedAsync() {
        await Groups.AddToGroupAsync(Context.ConnectionId, "lobby");     // GROUPS: broadcast to a subset
        await base.OnConnectedAsync();
    }
}
app.MapHub("/chatHub");
// A Hub method the CLIENT calls (SendMessage) can, in turn, invoke a method the CLIENT defines
// (ReceiveMessage) — RPC in BOTH directions, unlike a traditional request/response API.

The Client Side (JavaScript)

const connection = new signalR.HubConnectionBuilder().withUrl("/chatHub").build();

connection.on("ReceiveMessage", (user, message) => {      // handle messages the SERVER pushes
    appendToChatLog(`${user}: ${message}`);
});

await connection.start();
await connection.invoke("SendMessage", "Alice", "Hello!");   // call a method on the HUB

Targeting Clients — Groups & Users

TargetReaches
Clients.AllEvery connected client
Clients.CallerOnly the client that invoked this hub method
Clients.Group("lobby")All connections added to the "lobby" group
Clients.User(userId)All connections belonging to a specific authenticated user (across their tabs/devices)
Clients.OthersEveryone EXCEPT the caller

Scaling SignalR — the Backplane Problem

// A single SignalR server keeps track of which connections belong to which groups IN MEMORY. With
// MULTIPLE server instances (the Kubernetes HPA scaling story), a message sent from instance A needs
// to reach clients connected to instance B — but instance A has no idea instance B's connections exist.
//
// A BACKPLANE (Redis is the standard choice) solves this: all server instances publish/subscribe
// through Redis, so a message broadcast on ANY instance reaches clients connected to ANY instance:
builder.Services.AddSignalR().AddStackExchangeRedisBackplane(options =>
    options.Configuration = builder.Configuration.GetConnectionString("Redis"));
// Without a backplane, SignalR silently breaks the moment you scale beyond one instance — messages
// only reach clients connected to the SAME server instance that sent them.
⚠️ This is the same distributed-state problem as caching, applied to connections. Just as IMemoryCache is per-instance and needs Redis (IDistributedCache) to be consistent across replicas, SignalR's connection/group tracking is per-instance and needs a Redis BACKPLANE for the same reason. Any time you scale horizontally, ask "does this feature assume single-instance in-memory state?" — caching and SignalR are the two most common places that bites developers.

Interview Questions

Q1: How does SignalR handle different transport mechanisms, and why does that matter?

SignalR is built to abstract away the underlying communication mechanism entirely from your application code. It supports three transports, tried in order of preference during a connection negotiation handshake: WebSockets (a full-duplex, persistent TCP-based connection — the ideal, most efficient option, used whenever the client, server, and any intermediate network infrastructure support it), Server-Sent Events (a one-way, server-to-client streaming mechanism over a standard HTTP connection, used as a fallback when WebSockets aren't available but the environment still supports streaming), and Long Polling (the client repeatedly issues short-lived HTTP requests asking "is there anything new for me?" — the least efficient option but the one most universally compatible, working even through restrictive corporate proxies or very old browsers that support neither WebSockets nor SSE). Critically, your Hub and client code is IDENTICAL regardless of which transport actually gets negotiated — you write SendMessage/ReceiveMessage-style RPC calls, and SignalR transparently handles picking the best available option and falling back gracefully. This matters because it means developers don't need to write separate code paths or handle transport-specific quirks themselves; real-world deployment environments vary enormously (corporate networks that block WebSocket upgrades, older client browsers, restrictive proxies), and SignalR's negotiation ensures the application still functions — just with reduced efficiency — in environments where the ideal transport isn't available.

Q2: What problem does a SignalR backplane solve, and why is it necessary when scaling horizontally?

A single SignalR server instance keeps track, entirely in its own process memory, of which client connections are currently active and which groups each connection belongs to. This works perfectly fine with exactly one server instance: when a Hub method calls Clients.Group("lobby").SendAsync(...), that instance knows exactly which of ITS connections are in the "lobby" group and can push the message to them directly. The problem appears the moment the application is scaled to run as MULTIPLE instances behind a load balancer (the standard production pattern, directly relevant to the Kubernetes track's horizontal scaling) — different clients' WebSocket connections land on different server instances (instance A, instance B, instance C, depending on load balancing), but each instance's in-memory connection/group tracking is completely ISOLATED from the others. So if a message is broadcast from code running on instance A, it can only reach clients whose WebSocket connection happens to also be attached to instance A — clients connected to instance B or C never receive it, even though from the application's perspective it looks like the message "should" have reached everyone. A backplane (commonly Redis, via AddStackExchangeRedisBackplane) solves this by having every server instance publish outgoing messages to, and subscribe to incoming messages from, a shared Redis pub/sub channel — so a message originating on instance A is relayed through Redis to instances B and C as well, which then deliver it to their own locally-connected clients, making broadcasts and group messaging correctly reach every relevant client regardless of which server instance they happen to be connected to. This is conceptually identical to why IDistributedCache (Redis) is needed instead of IMemoryCache once an application scales beyond a single instance — any feature relying on in-memory, per-instance state breaks silently at scale without an externalized, shared coordination mechanism.

Q3: How does SignalR's RPC model differ from a traditional HTTP request/response API?

A traditional HTTP API follows a strict request/response model: the CLIENT always initiates communication by sending a request, and the SERVER responds — the server can never spontaneously push data to the client outside of responding to something the client asked for, which makes it a poor fit for scenarios where the server needs to notify clients of events that originate elsewhere (a new chat message from another user, a live stock price update, a notification triggered by a background process). SignalR instead maintains a PERSISTENT connection (ideally over WebSockets) and enables RPC (remote procedure call) style communication in BOTH directions over that single connection: a client can invoke a method defined on the server-side Hub class (connection.invoke("SendMessage", ...) calls the Hub's SendMessage method), and — the key difference — the SERVER can independently invoke a method that the CLIENT has registered a handler for (Clients.All.SendAsync("ReceiveMessage", ...) triggers every connected client's registered "ReceiveMessage" handler), entirely outside the context of any specific request the client made. This lets the server push updates the instant something relevant happens — a message sent by another user, a completed background job, a live data change — rather than requiring clients to repeatedly poll an HTTP endpoint asking "has anything changed yet?" which is both less efficient (constant polling overhead) and inherently less real-time (limited by the polling interval) than a persistent connection the server can push through immediately.

🏋️ Practice: build a simple chat Hub with SendMessage/ReceiveMessage and a matching JavaScript client, testing with two browser tabs; add group support so messages only reach clients in the same "room"; use Clients.Caller vs Clients.Others vs Clients.All and observe the difference; run two instances of the app locally (different ports) with no backplane and confirm a message from one doesn't reach clients connected to the other; add a Redis backplane and confirm cross-instance broadcasting now works.

🎯 Key Takeaways

  • SignalR negotiates the best available transport (WebSockets → SSE → Long Polling) transparently — your Hub/client code stays identical regardless.
  • A Hub is a server-side RPC endpoint: clients invoke Hub methods, and Hub methods can invoke client-registered methods — RPC in both directions over one persistent connection.
  • Target clients precisely with Clients.All/Caller/Others/Group/User rather than always broadcasting to everyone.
  • SignalR's connection/group tracking is per-instance in-memory — scaling to multiple server instances silently breaks broadcasting without a Redis backplane.
  • This mirrors the caching lesson's distributed-state problem exactly: any in-memory, per-instance state (cache, SignalR connections) needs externalizing (Redis) once you scale horizontally.
