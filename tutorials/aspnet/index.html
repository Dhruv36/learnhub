Minimal APIs: First Endpoint Basics

ASP.NET Core's Minimal APIs let you build an HTTP API with a fraction of the ceremony of traditional controllers — endpoints are just functions mapped to routes. This is now the default starting point for new ASP.NET Core APIs, and understanding its pieces (the WebApplicationBuilder, endpoint mapping, and parameter binding) is the foundation for everything else in this track. Prerequisite: the .NET (C#) track through async/await.

The Whole App in a Few Lines

var builder = WebApplication.CreateBuilder(args);   // sets up config, logging, DI container
builder.Services.AddScoped();   // register services (DI lesson)

var app = builder.Build();                          // builds the app + DI container

app.MapGet("/users/{id:int}", async (int id, IUserRepository repo) =>
    await repo.FindAsync(id) is { } user ? Results.Ok(user) : Results.NotFound());

app.MapPost("/users", async (CreateUserRequest req, IUserRepository repo) => {
    var user = await repo.CreateAsync(req.Name, req.Email);
    return Results.Created($"/users/{user.Id}", user);
});

app.Run();                                           // starts listening for requests
🎯 builder → app is the two-phase pattern you'll see everywhere. Phase 1 (before Build()): register services into the DI container, configure options. Phase 2 (after Build()): configure the middleware pipeline and map endpoints. This mirrors the .NET DI & Hosting lesson's Generic Host — Minimal APIs are built directly on top of it, not a separate framework.

Automatic Parameter Binding

app.MapGet("/search", (string q, int page = 1) => ...);        // query string: ?q=hello&page=2
app.MapGet("/users/{id}", (int id) => ...);                     // route parameter: /users/42
app.MapPost("/users", (CreateUserRequest body) => ...);         // request BODY, deserialized from JSON
app.MapGet("/me", (HttpContext ctx) => ...);                    // special types injected directly
app.MapGet("/orders", (IOrderService svc) => ...);              // SERVICES resolved from DI automatically
// The route handler's PARAMETER NAMES AND TYPES drive binding — no [FromQuery]/[FromBody] attributes
// needed for the common cases (though they exist for disambiguation when needed).

Minimal APIs vs Controllers (preview of the next lessons)

// Minimal APIs: endpoints are functions, defined inline or grouped — lean, less ceremony, great for
//   microservices/small-to-medium APIs and situations where every endpoint is meaningfully different.
// Controllers (MVC, controllers-mvc lesson): class-based, convention-heavy, more built-in structure
//   (filters, model binding attributes, [ApiController] behaviors) — often preferred for LARGE APIs
//   with many related endpoints sharing cross-cutting concerns.
// Both run on the SAME underlying ASP.NET Core pipeline — it's a style/structure choice, not a different

Interview Questions

Q1: What is the builder/app two-phase pattern in ASP.NET Core, and why does it matter?

WebApplication.CreateBuilder(args) returns a WebApplicationBuilder that represents the CONFIGURATION phase of the application: at this point you register services into the dependency injection container (builder.Services.Add...), configure options, and set up logging/configuration sources — nothing is actually running yet. Calling builder.Build() finalizes this configuration into an immutable WebApplication instance (the `app`), which enters the SECOND phase: configuring the middleware pipeline (app.Use...) and mapping endpoints (app.Map...) — this is where you define what happens when a request actually arrives. Finally, app.Run() starts the Kestrel web server and begins listening for requests. This two-phase separation matters because the DI container and configuration are finalized at Build() — you can't register new services after that point, only configure how requests are handled — and understanding this ordering explains a common early mistake (trying to register a service after Build() and getting a compile error or runtime failure). This pattern is built directly on .NET's Generic Host (from the DI & Hosting lesson), unifying DI, configuration, logging, and the web server into one coherent startup sequence — Minimal APIs don't use a separate hosting model, they're a lean, function-based way of mapping endpoints on top of the same infrastructure controllers use.

Q2: How does automatic parameter binding work in Minimal API route handlers?

ASP.NET Core inspects the route handler DELEGATE's parameters — their names and types — and infers where each value should come from, without requiring explicit [FromX] attributes for the common cases. A parameter whose name matches a ROUTE TEMPLATE segment (like {id} in "/users/{id}") is bound from the route; a simple type (string, int, and similar) not matching a route segment is bound from the QUERY STRING; a complex type (a class/record like CreateUserRequest) is bound by DESERIALIZING the REQUEST BODY as JSON; and certain special framework types — HttpContext, HttpRequest, CancellationToken — are injected directly by the framework rather than coming from client input. Additionally, any parameter type that's REGISTERED IN THE DI CONTAINER (like IOrderService) is automatically resolved as a service dependency, exactly as constructor injection works for controllers — so a Minimal API handler can request both HTTP-derived data and injected services in the same parameter list, and the framework figures out which is which based on registration and type. When binding would genuinely be ambiguous (for example, wanting a complex type to come from the query string instead of the body), you can use explicit attributes like [FromQuery], [FromBody], or [FromRoute] to disambiguate.

Q3: What's the difference between Minimal APIs and Controllers, and when would you choose each?

Both run on the exact same underlying ASP.NET Core request pipeline, DI container, and configuration system — the difference is purely in how you STRUCTURE and DEFINE endpoints, not a different framework. Minimal APIs define endpoints as simple functions mapped directly to routes (app.MapGet/MapPost/etc.), with little ceremony — no required base class, no attribute-heavy conventions — which makes them lean and quick to read for smaller APIs, microservices, or situations where each endpoint's logic is genuinely distinct and doesn't share much structure with others. Controllers (covered in the next lesson) use a class-based, convention-driven MVC-style approach: related endpoints are grouped into a controller class, with built-in support for model-binding attributes, action filters, and [ApiController] behaviors (automatic model validation responses, binding source inference) that reduce boilerplate specifically when you have MANY related endpoints sharing cross-cutting concerns (a consistent set of filters, a common base route, shared authorization policies). In practice, large enterprise APIs with dozens of related endpoints often favor controllers for that shared structure and tooling, while newer, smaller, or microservice-style APIs increasingly default to Minimal APIs for their reduced ceremony — and a single application CAN mix both styles, since they coexist on the same pipeline.

🏋️ Practice: scaffold a minimal API project and add GET/POST/DELETE endpoints for a simple resource; inject a service via DI directly into an endpoint handler; bind a route parameter, a query parameter, and a JSON body in three different endpoints; return typed Results (Ok/NotFound/Created/BadRequest) and verify the actual HTTP status codes with curl/Postman; convert a synchronous handler to async and confirm it still works correctly for I/O-bound logic.

🎯 Key Takeaways

  • WebApplication.CreateBuilder → register services (Phase 1) → Build() → configure pipeline/map endpoints (Phase 2) → Run(). All service registration happens before Build().
  • Minimal APIs are built on the same Generic Host, DI, and middleware pipeline as controllers — a structural choice, not a separate framework.
  • Parameter binding is inferred from parameter names/types: route segments, query string, JSON body, special framework types, and DI services — all in one parameter list.
  • Use typed Results (Ok/NotFound/Created/BadRequest/NoContent) for explicit, testable HTTP semantics rather than returning raw objects.
  • Minimal APIs suit lean/microservice APIs; Controllers (next lesson) suit large APIs with many related endpoints sharing cross-cutting concerns — both coexist on one pipeline.
