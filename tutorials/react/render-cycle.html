The Render Cycle Effects & Lifecycle

What you'll master: the precise sequence React runs — trigger → render → reconcile → commit → effects — what causes re-renders (and what doesn't), why "parent renders = children render" until you opt out, Strict Mode's double-invocation, and the purity rules that concurrent rendering enforces. This is the mental model every later lesson (effects, memoization, suspense) plugs into. Prerequisites: Components & JSX (elements vs fibers), State (snapshots).

Part 1 — The Five Phases, End to End

// 1. TRIGGER   — something schedules a render:
//                initial mount | setState | parent re-rendered | context value changed
// 2. RENDER    — React calls your component functions (top-down from the scheduled
//                fiber), collecting new element trees. Pure computation. NO DOM here.
// 3. RECONCILE — new elements diffed against current fibers (position/type/key rules
//                from Lists & Keys) → a minimal list of DOM mutations.
// 4. COMMIT    — mutations applied to the real DOM synchronously; refs attached;
//                useLayoutEffect runs (before the browser paints).
// 5. PASSIVE   — the browser paints; then useEffect callbacks run.
//
// Renders can happen WITHOUT commits: if reconciliation finds no changes, or a
// setState bails out (same value), the DOM isn't touched. "It re-rendered" and
// "the DOM updated" are different claims — profiling shows the first, users see the second.
Vocabulary precision wins interviews: "render" = React calling your function (cheap, frequent, pure). "Commit" = DOM mutation (batched, minimal). Most performance fear aims at renders; most real cost lives in commits and effects. Optimizing renders that produce zero DOM changes is often optimizing noise — measure first.

Part 2 — What Re-renders, Exactly

function App() {                 // state lives here
  const [n, setN] = useState(0);
  return (
    <>
      <Counter n={n} onUp={() => setN(n + 1)} />
      <Sidebar />                {/* takes NO props related to n… */}
      {expensiveStatic}
    </>
  );
}
// setN triggers App to re-render. Then:
// • Counter re-renders — props changed. Expected.
// • Sidebar ALSO re-renders — not because props changed, but because App re-ran
//   and produced a NEW  element. React re-renders children by default;
//   "props didn't change" is only checked if Sidebar is wrapped in React.memo.
// • RE-RENDER ≠ DOM CHANGE: Sidebar's output is identical, so reconciliation
//   produces zero mutations for it. Wasted CPU, invisible to the user — usually fine.

// THE exception worth knowing — children as props ("lift content up"):
function App() {
  const [n, setN] = useState(0);
  return <Layout><ExpensiveTree /></Layout>;   // if state lived in Layout instead…
}
// When Layout re-renders from ITS OWN state, props.children is the SAME element
// object App created — reference-equal — so React skips re-rendering ExpensiveTree
// entirely. Composition gives you memoization for free; this is why "lift state
// down, push content up" is the first performance tool, before memo().

setState with the same value (Object.is)Nothing (bail-out; React may render the component once more before bailing)

Part 3 — Render Purity: the Rules and Why They're Real

// During render (the function body), you must NOT:
function Bad({ items }) {
  items.sort(byPrice);                    // ❌ mutate props/state (visible outside)
  window.title = "…";                     // ❌ touch the outside world
  fetch("/api/log");                      // ❌ side effects
  const [s, setS] = useState(0);
  if (someCase) setS(1);                  // ❌ set state → infinite render loop
  return <div>{Math.random()}</div>;      // ❌ nondeterminism — output differs per call
}
// Where those things belong:
// mutations of data   → in the OWNER, via setState with copies
// document/title, subscriptions, network → useEffect (after commit)
// state responses to events → event handlers
// "set state during render" has ONE sanctioned form: derived-state-reset —
//   if (prevId !== id) { setPrevId(id); setDraft(""); }  // React restarts render
//   immediately, before touching DOM — still cheaper than an effect + double paint.
//   (But prefer key={id} remounts — Lists & Keys — when resetting everything.)

Why so strict? Because React reserves the right to call your function and discard the result: concurrent rendering can start a render at low priority, abandon it when urgent input arrives, and restart. A render that logged, mutated, or fetched has now done it twice. Purity is what makes rendering interruptible, resumable, and (in the future) parallelizable.

Part 4 — Strict Mode: the Purity Fuzzer

<StrictMode><App /></StrictMode>   // dev-only behavior; production is untouched

// In development, Strict Mode deliberately:
// 1. DOUBLE-INVOKES component functions, initializers, and updaters —
//    impure renders produce visibly wrong results (e.g. items pushed twice).
// 2. Runs every effect's mount → cleanup → mount cycle TWICE —
//    effects missing cleanup break loudly (duplicate listeners/intervals).
// 3. Warns on deprecated APIs.

// The classic confusion:
useEffect(() => { console.log("mounted"); }, []);   // logs TWICE in dev
// This is not a bug to suppress with a useRef-did-run flag — it's the tool telling
// you: "your effect must be safe to run twice." Write idempotent effects with
// cleanup and both logs become harmless.
Senior smell: code that "fixes" Strict Mode double-effects with if (didRun.current) return; guards. That silences the alarm instead of the fire — the same remount happens in production with tab restoration, fast refresh, offscreen rendering, and key resets. Effects must survive re-execution; that's the contract the next lesson is built on.

Part 5 — Commit Timing: useLayoutEffect vs useEffect vs Paint

// After React mutates the DOM (commit), two effect queues run at different times:
//
// useLayoutEffect: SYNCHRONOUS, after DOM mutation, BEFORE the browser paints.
//   → for reading layout (measure a tooltip, scroll position) and synchronously
//     re-positioning — the user never sees the intermediate frame.
// useEffect: after paint, asynchronously.
//   → for everything else: data, subscriptions, logging. Doesn't block pixels.

function Tooltip({ anchorRef, children }) {
  const tipRef = useRef(null);
  const [pos, setPos] = useState(null);
  useLayoutEffect(() => {
    const a = anchorRef.current.getBoundingClientRect();
    const t = tipRef.current.getBoundingClientRect();
    setPos({ top: a.top - t.height - 8, left: a.left });   // re-render BEFORE paint
  }, [anchorRef]);
  return (
    <div ref={tipRef} style={pos ? { top: pos.top, left: pos.left } : { visibility: "hidden" }}>
      {children}
    </div>
  );
}
// With useEffect instead: one painted frame at the wrong position, THEN the jump —
// a visible flicker. That flicker is the entire difference between the two hooks.
// Default to useEffect; reach for useLayoutEffect only for measure-then-adjust.

Interview Questions

Beginner: Walk through what happens after a button's onClick calls setState.

React schedules a re-render (batching other updates in the same event). Render phase: the component function runs with the new state snapshot, returning new elements; children are called recursively. Reconciliation diffs new elements against fibers to compute minimal DOM changes. Commit applies them synchronously, updates refs, runs layout effects; the browser paints; then passive effects (useEffect) run.

Beginner: Does a child re-render when its parent does, even with unchanged props?

Yes — by default React re-renders the whole subtree; it doesn't compare props unless the child is wrapped in React.memo. The exception: element trees passed as props (like children) that are reference-equal from a previous render are skipped. Re-rendering with identical output costs CPU but produces zero DOM changes.

Mid: Why does my useEffect(…, []) log twice in development?

Strict Mode intentionally mounts, unmounts (running cleanup), and remounts every component in dev to verify effects are resilient to re-execution — the same lifecycle produced in production by key changes, fast refresh, and future offscreen features. The fix is never to guard against the second run but to give the effect correct cleanup so running twice is harmless.

Mid: When is useLayoutEffect correct, and what does it cost?

When you must read the committed DOM (measurements, scroll positions) and adjust state/styles before the user sees the frame — tooltips, popover positioning, scroll restoration. It runs synchronously between DOM mutation and paint, so everything in it delays pixels; heavy work there is jank by construction. Everything that doesn't need pre-paint timing belongs in useEffect.

Senior: Explain the children-as-props optimization and why it works without memo.

If a stateful component receives its expensive content as children (<Layout><Expensive/></Layout>), the element for <Expensive/> is created by the parent's render. When Layout re-renders from its own state, that element object is unchanged (same reference), and React bails out of re-rendering it — reconciliation sees an identical element and reuses the subtree wholesale. It's structural memoization: no comparison function, no memo wrapper, just moving state down so content is constructed above the churn.

Senior: What does concurrent rendering change about the phases, and what stays guaranteed?

The render phase becomes interruptible: React can pause a low-priority render (e.g. from startTransition), handle urgent input with a quick high-priority render, then resume or restart the abandoned work — which is why renders may run multiple times without committing. Guarantees that hold: commit is always synchronous and un-interruptible (users never see half-applied DOM), effects run only for committed renders, and state updates keep their ordering. The purity rules are exactly the price of interruptibility.

🏋️ Graded Exercises

1 (easy): In App → Header, Body(state here) → Row×3, Footer, the Body's state updates. List what re-renders.
Solution 
// Body and its subtree: Body, Row, Row, Row. NOT App, Header, or Footer —
// renders propagate DOWN from the component that owns the changed state, never up
// or sideways. (This is the argument for keeping state as low as possible.)
2 (easy): Why does <div>{Math.random()}</div> violate render rules, and where could the random value legally come from?
Solution 
// Renders must be deterministic: same props/state → same output. Concurrent
// re-runs or Strict Mode double-invokes produce different UI each call.
// Legal homes: generate once in useState's lazy initializer —
// useState(() => Math.random()) — or in the event that creates the data.
3 (medium): This component pushes to an analytics array in its body and shows each event twice in dev. Explain and fix.
Solution 
// The push is a render side effect; Strict Mode double-invokes render, so every
// mount records twice (and any re-render records again — worse than 2×).
// Fix: move it to a post-commit effect tied to what it reports:
useEffect(() => { analytics.push({ page }); }, [page]);
// Now it runs once per committed page value (twice in dev with cleanup-less
// idempotent append — if exactly-once matters, dedupe in the analytics layer).
4 (medium): A dropdown positions itself with useEffect and users report a flicker: it appears top-left for a frame, then jumps. Fix it and explain the frame timeline before and after.
Solution 
// Before: commit paints the dropdown at its default position (frame 1 visible),
// THEN useEffect measures and sets position → second commit → frame 2 correct.
// Fix: useLayoutEffect — measurement + setState run after DOM mutation but BEFORE
// paint; React re-renders synchronously, and the first painted frame is already
// correct. Also render it visibility:hidden until positioned, as a belt-and-braces.
5 (medium): Using the children-as-props pattern, refactor so ExpensiveChart stops re-rendering when the sidebar toggles: function Page(){ const [open,setOpen]=useState(false); return <div><button onClick={()=>setOpen(o=>!o)}/>{open && <Nav/>}<ExpensiveChart data={data}/></div>; }
Solution 
// Move the toggle state into a component that receives the chart as children:
function Collapsible({ children }) {
  const [open, setOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setOpen(o => !o)} />
      {open && <Nav />}
      {children}                       {/* same element reference across toggles */}
    </div>
  );
}
function Page() {
  return <Collapsible><ExpensiveChart data={data} /></Collapsible>;
}
// Page doesn't re-render on toggle (state moved down); Collapsible re-renders but
// its children prop is reference-equal → React skips the chart. Zero memo() used.
6 (hard): A metronome component drifts: useEffect(() => { const id = setInterval(tick, 500); }, [bpm]) — after changing bpm five times there are six intervals running. Explain via the render cycle and fix.
Solution 
// Each bpm change re-runs the effect (deps changed) — but the previous interval
// was never cleared: no cleanup function. Effects don't replace each other;
// cleanup is the ONLY teardown mechanism. Strict Mode would have caught this on
// day one (double-mount → two intervals immediately).
useEffect(() => {
  const id = setInterval(tick, 60000 / bpm);
  return () => clearInterval(id);      // runs before next effect AND on unmount
}, [bpm]);
7 (hard): The Profiler shows ProductGrid rendering on every keystroke in the search box, taking 40ms each. Two teammates propose (a) React.memo(ProductGrid), (b) moving the search state down into SearchBox. Evaluate both, including what each depends on to keep working.
Solution 
// (b) is structural and robust: if only SearchBox owns the query (grid shows all
// products until submit/debounce), keystrokes re-render just the input. Nothing
// can silently regress it. Choose it when the grid doesn't need per-key updates.
// (a) works ONLY while every ProductGrid prop stays reference-stable — one inline
// handler or fresh array from the parent and the memo silently dies; it also
// still costs a props comparison per keystroke. Choose it when the grid DOES
// consume the live query (filter-as-you-type) — then pair it with stable props
// (useCallback/useMemo) and consider debouncing/deferring the query value
// (Concurrent Features) since 40ms × keystroke is above the jank line either way.

🎯 Key Takeaways

  • Five phases: trigger → render (pure calls) → reconcile (diff) → commit (DOM + layout effects, pre-paint) → paint → passive effects. Render ≠ DOM change.
  • Re-renders flow down from the state owner to the entire subtree; props comparison only exists if you add memo; reference-equal element props (children) skip for free.
  • Render bodies stay pure: no side effects, no mutation, no nondeterminism, no unconditional setState — the price of interruptible concurrent rendering.
  • Strict Mode double-invokes renders and remount-cycles effects in dev to expose impurity and missing cleanup — fix the effect, never flag-guard the symptom.
  • useLayoutEffect = measure/adjust before paint (sparingly); useEffect = everything else, after paint.
  • The Profiler's "why did this render" answers re-render mysteries with data; structural fixes (state placement, composition) come before memoization.
