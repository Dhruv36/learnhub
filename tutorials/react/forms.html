Forms: Controlled & Uncontrolled Foundations

What you'll master: the controlled-component model (React state as the single source of truth for inputs), every form control's controlled API, uncontrolled forms with refs and FormData — and the honest trade-off between them — validation UX patterns (when to show errors is a design decision), and a production-grade form built step by step. Prerequisites: State, Events, and the HTML Forms section — native validation attributes still matter.

Part 1 — Two Sources of Truth, Pick One

A DOM input holds its own value natively — type and it changes. React state is a second place a value can live. The controlled/uncontrolled question is just: who owns the value?

// UNCONTROLLED — the DOM owns it. React reads it when needed:
<input defaultValue="Ada" ref={nameRef} />        // defaultValue: initial only
// later: nameRef.current.value

// CONTROLLED — React state owns it. The DOM mirrors state:
const [name, setName] = useState("Ada");
<input value={name} onChange={e => setName(e.target.value)} />
//      ↑ every render forces the input to equal state
//                ↑ every keystroke updates state → re-render → input shows it

The controlled loop looks circular but is the point: the input can never disagree with state. Anything can read name at any time, transform it, validate it, or overwrite it — and the UI follows. The price: a render per keystroke and a bit of ceremony per field.

ControlledUncontrolled
Source of truthReact stateDOM
Live validation/formatting/filtering✅ natural❌ awkward
Other UI reacts while typing (char count, preview, disable button)✅❌
Ceremony per fieldstate + onChange each~zero (name attribute)
Performance on huge formsrender per keystroke (usually fine, measure)no renders while typing
Read valuesalways in stateat submit: FormData/refs
The pragmatic rule: controlled when the UI must react to the value as it changes; uncontrolled (+ FormData) when you only need values at submit. "Controlled everything" is dogma, not engineering — a login form gains nothing from four setters. Form libraries (React Hook Form) are essentially industrial-grade uncontrolled with subscriptions.

Part 2 — The Controlled API for Every Control

// text/textarea/number/etc. — value + onChange:
<textarea value={bio} onChange={e => setBio(e.target.value)} />

// checkbox — CHECKED, not value:
<input type="checkbox" checked={agreed} onChange={e => setAgreed(e.target.checked)} />

// radio group — one state, checked computed per option:
{["s", "m", "l"].map(size => (
  <label key={size}>
    <input type="radio" name="size" value={size}
           checked={choice === size}
           onChange={e => setChoice(e.target.value)} />
    {size.toUpperCase()}
  </label>
))}

// select — value on the SELECT (not `selected` on options, unlike HTML):
<select value={country} onChange={e => setCountry(e.target.value)}>
  <option value="">Choose…</option>
  <option value="in">India</option>
  <option value="de">Germany</option>
</select>

// multi-checkbox group — a Set in state beats parallel booleans:
const [toppings, setToppings] = useState(new Set());
const toggle = t => setToppings(s => {
  const next = new Set(s);                     // copy — Sets are mutated in place otherwise
  next.has(t) ? next.delete(t) : next.add(t);
  return next;
});

// file input — ALWAYS uncontrolled (value is read-only for security):
<input type="file" onChange={e => setFile(e.target.files[0])} />

// number caveat: e.target.value is a STRING ("", "3.", "3e2" mid-typing).
// Keep the string in state; parse at submit — parsing per keystroke fights the user.

Part 3 — Uncontrolled Done Right: FormData

function SignupForm({ onSubmit }) {
  function handleSubmit(e) {
    e.preventDefault();
    const data = Object.fromEntries(new FormData(e.currentTarget));
    // { email: "...", plan: "pro", newsletter: "on" } — keyed by name attributes
    onSubmit(data);
  }
  return (
    <form onSubmit={handleSubmit}>
      <input name="email" type="email" required />          {/* native validation still runs! */}
      <select name="plan" defaultValue="free">
        <option value="free">Free</option><option value="pro">Pro</option>
      </select>
      <label><input name="newsletter" type="checkbox" /> News</label>
      <button>Sign up</button>
    </form>
  );
}
// Zero state, zero onChange, native required/type validation for free.
// This exact shape is also what React 19's form actions build on
// (
 receives FormData) — see Server Components.

Part 4 — Validation: When Matters More Than What

Checking an email regex is trivial; the engineering is in when errors appear. The consensus UX: validate on blur, re-validate on change once touched, validate everything on submit — "reward early, punish late":

function useField(validate) {
  const [value, setValue] = useState("");
  const [touched, setTouched] = useState(false);
  const error = validate(value);                 // derived — never stored separately (State lesson)
  return {
    error: touched ? error : null,               // only SHOW errors after first blur
    inputProps: {
      value,
      onChange: e => setValue(e.target.value),   // once touched, feedback is now live
      onBlur:   () => setTouched(true),
    },
    forceTouch: () => setTouched(true),
    isValid: !error,
  };
}

function EmailField() {
  const email = useField(v =>
    !v ? "Email is required"
    : !/^\S+@\S+\.\S+$/.test(v) ? "That doesn't look like an email"
    : null);
  return (
    <div>
      <label htmlFor="email">Email</label>
      <input id="email" type="email" aria-invalid={!!email.error}
             aria-describedby="email-err" {...email.inputProps} />
      {email.error && <p id="email-err" role="alert">{email.error}</p>}
    </div>
  );
}
// error is DERIVED from value — storing setError state alongside is the classic
// redundant-state bug: they drift. touched is real state (it's a user event).
Layer, don't replace, native validation. required, type="email", minLength, pattern give you validation before JavaScript loads, and correct semantics for assistive tech. Custom validation adds better messages and cross-field rules on top. And always revalidate on the server — client validation is UX, not security (Security).

Part 5 — Build It: a Production Checkout Form

function CheckoutForm({ onComplete }) {
  const [form, setForm] = useState({ name: "", email: "", qty: "1", agree: false });
  const [touched, setTouched] = useState({});
  const [status, setStatus] = useState("idle");        // idle | submitting | error

  const set = e => {
    const { name, type, value, checked } = e.target;
    setForm(f => ({ ...f, [name]: type === "checkbox" ? checked : value }));
  };
  const blur = e => setTouched(t => ({ ...t, [e.target.name]: true }));

  // All validation DERIVED in one place — one object, easy to test in isolation:
  const errors = {
    name:  form.name.trim() ? null : "Name is required",
    email: /^\S+@\S+\.\S+$/.test(form.email) ? null : "Valid email required",
    qty:   +form.qty >= 1 && Number.isInteger(+form.qty) ? null : "Quantity must be a whole number ≥ 1",
    agree: form.agree ? null : "You must accept the terms",
  };
  const isValid = Object.values(errors).every(e => e === null);
  const show = k => touched[k] ? errors[k] : null;     // visibility policy in ONE place

  async function submit(e) {
    e.preventDefault();
    setTouched({ name: true, email: true, qty: true, agree: true });  // reveal everything
    if (!isValid) return;
    setStatus("submitting");
    try {
      await api.checkout({ ...form, qty: +form.qty }); // parse the string HERE
      onComplete();
    } catch {
      setStatus("error");                              // keep the user's input! No reset.
    }
  }

  return (
    <form onSubmit={submit} noValidate>
      <Field label="Name"  name="name"  value={form.name}  error={show("name")}
             onChange={set} onBlur={blur} />
      <Field label="Email" name="email" value={form.email} error={show("email")}
             onChange={set} onBlur={blur} type="email" />
      <Field label="Qty"   name="qty"   value={form.qty}   error={show("qty")}
             onChange={set} onBlur={blur} inputMode="numeric" />
      <label>
        <input type="checkbox" name="agree" checked={form.agree} onChange={set} onBlur={blur} />
        I accept the terms
      </label>
      {show("agree") && <p role="alert">{errors.agree}</p>}

      {status === "error" && <p role="alert">Payment failed — please try again.</p>}
      <button disabled={status === "submitting"}>
        {status === "submitting" ? "Processing…" : "Pay now"}
      </button>
    </form>
  );
}
// Production details baked in: one generic change handler; status enum (not
// isSubmitting+isError booleans); submit reveals all errors; failure preserves
// input; double-submit blocked by disabled; qty parsed once at the boundary.

Interview Questions

Beginner: Controlled vs uncontrolled — define both and name the deciding question.

Controlled: React state is the source of truth; the input renders value from state and reports changes via onChange. Uncontrolled: the DOM keeps the value; React reads it on demand (refs/FormData) and at most sets defaultValue. Deciding question: does anything need to react to the value while it's being typed? Yes → controlled. Only at submit → uncontrolled is less code.

Beginner: Why defaultValue instead of value for an uncontrolled input?

value makes React enforce that exact value every render — without onChange the input is locked. defaultValue sets the initial DOM value and then leaves ownership with the DOM, which is the entire uncontrolled contract. (Checkboxes: defaultChecked.)

Mid: What causes "A component is changing an uncontrolled input to be controlled"?

The value prop transitioned from undefined (React treats the input as uncontrolled) to a string (now controlled). Typical source: state initialized as useState()/useState(null) or a field read from a not-yet-loaded object (user?.name). Fix: initialize every field to "", or render the form only after data arrives (or key it on the record id).

Mid: Design the error-display policy for a good form UX.

Track touched per field. Before first blur: no errors (don't punish mid-typing). On blur: validate and show. Once touched: revalidate every change so fixes clear errors instantly ("reward early, punish late"). On submit: mark all touched, show everything, focus the first invalid field. Errors are derived from values, not stored; only touched-ness is state.

Senior: A 60-field enterprise form is janky on every keystroke. Diagnose and fix without a library, then say what a library would do.

Cause: one state object at the top means every keystroke re-renders all 60 fields. Fixes: (1) go uncontrolled + FormData — zero renders while typing, validate at blur/submit; (2) if fields must be controlled, isolate state per field component so a keystroke re-renders one field; (3) memoize field rows and pass stable handlers. React Hook Form does exactly this: uncontrolled inputs registered by ref, with a subscription model so only fields whose validation state changed re-render.

Senior: How do controlled inputs interact with React's batching when you reformat while typing (e.g. credit-card spacing)?

Reformatting in onChange (setValue(format(raw))) works, but naive formatting breaks the cursor: React resets the input to the formatted string and the caret jumps to the end — infuriating for mid-string edits. Production answer: compute the new caret from the delta and restore it via setSelectionRange in a layout effect, or use inputMode/masking libraries that solve caret math. Also format on a copy — never block raw keystrokes, or fast typists lose input.

2 (easy): Convert this to uncontrolled with FormData: a login form (email, password) that calls login(email, password) on submit.
Solution 
function Login() {
  function onSubmit(e) {
    e.preventDefault();
    const { email, password } = Object.fromEntries(new FormData(e.currentTarget));
    login(email, password);
  }
  return (
    <form onSubmit={onSubmit}>
      <input name="email" type="email" required />
      <input name="password" type="password" required />
      <button>Log in</button>
    </form>
  );
}
// Nothing re-renders while typing; native `required` gates the submit.

function Shipping() {
  const [method, setMethod] = useState("standard");
  const [phone, setPhone] = useState("");

5 (medium): Implement the blur-then-live policy for a single "username" field (required, ≥3 chars, lowercase only) without any library.
Solution 
function Username() {
  const [v, setV] = useState("");
  const [touched, setTouched] = useState(false);
  const error =
    !v ? "Required"
    : v.length < 3 ? "At least 3 characters"
    : v !== v.toLowerCase() ? "Lowercase only"
    : null;
  const visible = touched ? error : null;
  return (
    <>
      <input value={v}
             onChange={e => setV(e.target.value)}
             onBlur={() => setTouched(true)}
             aria-invalid={!!visible} />
      {visible && <p role="alert">{visible}</p>}
    </>
  );
}
6 (hard): Build a two-field "confirm password" pair where the mismatch error appears on the confirm field only after it's touched, clears live, and submit is blocked until both are valid.
Solution 
function Passwords({ onValid }) {
  const [pw, setPw]   = useState("");
  const [pw2, setPw2] = useState("");
  const [touched2, setTouched2] = useState(false);

  const pwError  = pw.length >= 8 ? null : "At least 8 characters";
  const matchErr = pw2 === pw ? null : "Passwords don't match";   // cross-field: derived from BOTH
  const show2 = touched2 ? matchErr : null;
  const valid = !pwError && !matchErr;

  return (
    <form onSubmit={e => { e.preventDefault(); if (valid) onValid(pw); else setTouched2(true); }}>
      <input type="password" value={pw}  onChange={e => setPw(e.target.value)}
             placeholder="Password" />
      <input type="password" value={pw2} onChange={e => setPw2(e.target.value)}
             onBlur={() => setTouched2(true)} placeholder="Confirm"
             aria-invalid={!!show2} />
      {show2 && <p role="alert">{show2}</p>}
      <button disabled={!valid && touched2}>Create account</button>
    </form>
  );
}
// Cross-field validation is just derivation over multiple values — no extra machinery.
// Note matchErr recomputes when EITHER field changes, so fixing pw clears the error too.
7 (hard): Your product has a 40-field settings page; only 3 fields need live behavior (a slug preview, a password meter, a char counter). Design the form architecture and justify it.
Solution 
// Hybrid: the honest reading of the controlled/uncontrolled trade-off.
// 1. The form is uncontrolled overall: name attributes + one onSubmit reading
//    FormData → 37 fields cost zero state and zero renders while typing.
// 2. The 3 live fields are controlled ISLANDS — each its own small component
//    owning its state, rendering its preview/meter/counter locally. Keystrokes
//    re-render only that island (a few nodes), never the page.
// 3. Validation: native attributes + one submit-time pass over FormData;
//    the islands validate live with the touched pattern.
// 4. Still one : controlled islands' inputs carry name attributes too,
//    so FormData collects everything uniformly at submit.
// This beats both extremes: fully controlled = 40 fields re-rendering per key;
// fully uncontrolled = the 3 live features become ref-poking hacks.

🎯 Key Takeaways

  • Controlled = React state owns the value (value + onChange, UI can react per keystroke); uncontrolled = DOM owns it (defaultValue, read via FormData/refs at submit). Choose per field, not per ideology.
  • Control-specific props: checked for checkbox/radio, value on <select>, file inputs always uncontrolled, number inputs keep strings in state.
  • FormData(e.currentTarget) + name attributes reads a whole uncontrolled form in one line — and native validation attributes still apply.
  • Errors are derived from values every render; touched is the only validation state — show on blur, update live after, reveal all on submit.
  • Never flip value between undefined and string (initialize to ""); never reset user input on a failed submit; parse strings once at the boundary.
  • Forms are an accessibility hotspot: real labels, aria-invalid, role="alert" errors linked by aria-describedby — and the server revalidates regardless.
