Error Boundaries & Resilience Patterns & Quality

What you'll master: why one render error white-screens an entire React app by default, error boundaries — what they catch, what they don't, and where to place them — recovery patterns (retry, reset keys), error reporting pipelines, and the full resilience checklist for production UIs. Prerequisites: The Render Cycle, Lists & Keys (reset-by-key), Routing (route errorElement).

Part 1 — Why React White-Screens

// A component throws during render:
function PriceTag({ product }) {
  return <span>{product.price.toFixed(2)}</span>;   // product.price undefined → TypeError
}
// React CANNOT safely keep the tree: the render was mid-reconciliation, state
// half-computed. Rather than show corrupt UI, React 16+ UNMOUNTS THE ENTIRE
// ROOT. One optional widget's null field = a blank page for the whole app.
//
// The default is total failure. Resilience means declaring, in the tree,
// "if THIS region dies, contain it here" — that declaration is an error boundary.

Part 2 — The Boundary Component (Yes, a Class)

// The one place classes remain required — no hook equivalent exists (yet):
class ErrorBoundary extends React.Component {
  state = { error: null };

  static getDerivedStateFromError(error) {     // phase 1: RENDER the fallback
    return { error };                          // pure — runs during render
  }

  componentDidCatch(error, info) {             // phase 2: REPORT (after commit)
    reportError(error, {                       // side effects belong here
      componentStack: info.componentStack,     // which component chain threw
      boundary: this.props.name,
    });
  }

  reset = () => this.setState({ error: null });

  render() {
    if (this.state.error) {
      return this.props.fallback?.(this.state.error, this.reset)
        ?? <DefaultErrorPanel onRetry={this.reset} />;
    }
    return this.props.children;
  }
}

// Usage — fallback as a render prop (Composition lesson) for flexibility:
<ErrorBoundary name="chart" fallback={(err, reset) => (
  <div role="alert">
    Chart failed to load. <button onClick={reset}>Try again</button>
  </div>
)}>
  <RevenueChart data={data} />
</ErrorBoundary>
// In practice most teams use the react-error-boundary package (same mechanics,
// maintained, adds useErrorBoundary + reset keys) — knowing the raw class is
// for understanding and interviews.
Boundaries CATCH errors thrown in…Boundaries DO NOT catch…
Descendants' render functionsEvent handlers (use try/catch — React isn't rendering then)
Lifecycle methods / hook bodies during renderAsync code: setTimeout, promises, effects' async work
Constructors / initializers below themErrors thrown in the boundary component itself
Lazy chunk load failures (Code Splitting)SSR errors (server rendering has its own path)
The async gap is the one that bites: a failed fetch does NOT hit a boundary — the throw happens in a promise callback, not in render. Async errors reach boundaries only if you re-throw them during render: store the error in state, then if (error) throw error; in the component body (this is what useErrorBoundary().showBoundary(err) and suspense-mode query libraries do for you). Handled-vs-thrown is a design decision: expected failures (404, validation) render status UI (Data Fetching); unexpected ones throw to the boundary.

Part 3 — Placement: the Resilience Architecture

// Boundaries are regional bulkheads. The standard three-layer layout:

<ErrorBoundary name="app" fallback={FullPageError}>          {/* 3. last resort */}
  <Shell>
    <Routes>                                                  {/* 2. per route — */}
      { path: "dashboard", element: <Dashboard />,            {/*    errorElement in */}
        errorElement: <RouteError /> }                        {/*    data routers */}
    </Routes>
  </Shell>
</ErrorBoundary>

// 1. Per independent WIDGET inside a page:
function Dashboard() {
  return (
    <div className="grid">
      <Widget name="sales"><SalesChart /></Widget>       {/* each Widget wraps an */}
      <Widget name="traffic"><TrafficMap /></Widget>     {/* ErrorBoundary + Suspense */}
      <Widget name="alerts"><AlertsFeed /></Widget>
    </div>
  );
}
// TrafficMap's crash costs ONE tile; sales and alerts keep working.
//
// Placement heuristic = the Suspense heuristic (Code Splitting): wrap regions
// that are INDEPENDENTLY USEFUL. Boundary + Suspense pair naturally at the same
// seams — one handles "not ready", the other "broke":
function Widget({ name, children }) {
  return (
    <ErrorBoundary name={name} fallback={(e, r) => <TileError onRetry={r} />}>
      <Suspense fallback={<TileSkeleton />}>{children}</Suspense>
    </ErrorBoundary>
  );
}
// Too many boundaries: errors hide in tiles nobody checks. Too few: blast
// radius grows. The shell (nav! escape routes!) must NEVER share a boundary
// with content — users must always be able to navigate away from a broken page.

Part 4 — Recovery: Reset Is a State Problem

// A boundary's reset re-renders the SAME children — if the crashing state
// persists (bad prop, poisoned store), it re-crashes instantly. Real recovery
// strategies, in order of preference:

// 1. RESET KEYS — recover automatically when inputs change (the user navigates,
//    picks a different record):
<ErrorBoundary resetKeys={[productId]} …>      {/* error state clears when the */}
  <ProductPanel id={productId} />              {/* id changes — new data, new try */}
</ErrorBoundary>
// (react-error-boundary's API; hand-rolled: compare props in getDerivedStateFromProps
//  or remount via key={productId} OUTSIDE the boundary.)

// 2. RETRY with cache invalidation — reset the boundary AND refetch the data
//    that fed the crash:
fallback={(err, reset) => <TileError onRetry={() => {
  queryClient.invalidateQueries({ queryKey: ["sales"] });   // clear poisoned data
  reset();                                                   // then re-render
}} />}

// 3. FULL REMOUNT for stateful/imperative children (the crashed-widget key bump —
//    Lists & Keys ex. 7): reset alone reuses fibers; a key change rebuilds them.

// 4. RELOAD as the honest last resort (chunk errors after deploy — Code
//    Splitting ex. 6). A reload button beats an infinite crash loop.
// Anti-pattern: auto-retrying render errors in a loop — a deterministic crash
// retried is a CPU heater. Retry on USER action or INPUT change, not timers.

Part 5 — The Reporting Pipeline (Senior Layer)

// A caught error users can dismiss is still a bug you must hear about:
function reportError(error, context) {
  // Sentry/Datadog/Rollbar — the shape every service wants:
  errorTracker.captureException(error, {
    tags: { boundary: context.boundary },
    extra: {
      componentStack: context.componentStack,   // React's chain of components
      release: BUILD_ID,                        // WHICH deploy — source maps key
      route: location.pathname,
    },
  });
}
// Also wire:
// window.addEventListener("error", …)              — uncaught sync errors
// window.addEventListener("unhandledrejection", …) — the async gap's safety net
// Production builds minify: upload SOURCE MAPS to the tracker per release or
// stacks are unreadable. Alert on error-RATE per boundary, not per event —
// one user's extension throwing ≠ a regression; every dashboard tile failing = page.
//
// componentStack vs stack: error.stack says WHERE the code threw;
// componentStack says WHICH TREE PATH rendered it — both, always.

Interview Questions

Beginner: What is an error boundary and why does React unmount everything without one?

A class component implementing getDerivedStateFromError (render a fallback) and/or componentDidCatch (report) that catches errors thrown while rendering its descendants. Without one, React can't trust a tree whose render threw mid-reconciliation — showing it could mean corrupt UI — so it unmounts the root. Boundaries declare where failure should be contained instead.

Beginner: Name three kinds of errors boundaries do NOT catch, and the alternative for each.

Event handler errors — try/catch in the handler (React isn't rendering). Async errors (promises, timers, fetch callbacks) — handle as status state, or re-throw into render via showBoundary; plus a global unhandledrejection listener as the net. Errors in the boundary component itself — keep fallbacks trivially simple and rely on the next boundary up.

Mid: How do you route an async data error INTO a boundary, and when should you?

Store it and throw during render: const [err, setErr] = useState(null); if (err) throw err; then catch(e => setErr(e)) in the async code — or use useErrorBoundary().showBoundary(e) / suspense-mode queries which throw for you. Do it for unexpected failures where the region is unusable (malformed payload, auth invariant broken); keep expected failures (404s, empty results, validation) as designed status UI — a boundary fallback for "no search results" is UX malpractice.

Mid: Why must a retry do more than reset the boundary, and what does a correct retry look like?

Reset re-renders the same children with the same inputs — a deterministic crash recurs immediately. Correct retry clears the cause alongside the error state: invalidate/refetch the query that produced the poison data, bump a key to remount stateful children, or navigate to changed inputs (resetKeys). The boundary handles containment; recovery is a state-management problem you solve with the state tools.

Senior: Design the error-handling architecture for a fintech dashboard: 12 widgets, a trade form, and strict audit requirements.

Containment: app boundary (static full-page fallback + reload), route errorElements, and a boundary+ Suspense per widget so one feed failure costs one tile — nav/shell outside all content boundaries. The trade form gets its own boundary but expected failures (rejected orders, validation, timeouts) are status-machine states (Reducers lesson), never boundary fallbacks — money UX demands explicit, recoverable states. Async gap covered by unhandledrejection + query error hooks. Reporting: every boundary tags its name; release + sourcemaps uploaded per deploy; audit requirement met by forwarding componentDidCatch events into the same immutable audit log as trade events, with correlation ids linking the error to the in-flight order. Alerting on per-boundary error rate; the trade-form boundary pages on-call at a far lower threshold than a chart tile.

Senior: A teammate proposes wrapping every component in a boundary via a HOC "so nothing can ever crash." Critique.

Failure containment at every node destroys failure visibility: errors get caught by micro-boundaries rendering tiny fallbacks nobody notices, pages silently degrade into fields of broken tiles, and integration bugs surface as vague UX complaints instead of tracked exceptions. It also breaks UX coherence — a form where one field crashed shouldn't offer the other fields as if valid — and adds a wrapper per component (tree noise, devtools sludge). Boundaries belong at independently useful seams: app, route, widget — chosen deliberately, with fallbacks designed for each region and reporting that treats every catch as a bug to fix, not a success.

1 (easy): Wrap the LearnHub-style playground's preview pane in a boundary so user-code crashes show "Your code threw: {message}" with a reset button.
Solution 
<ErrorBoundary fallback={(err, reset) => (
  <div role="alert" className="preview-error">
    Your code threw: <code>{err.message}</code>
    <button onClick={reset}>Reset preview</button>
  </div>
)}>
  <UserCodePreview source={code} />
</ErrorBoundary>
// Reset works here (unlike a data crash) because the user will EDIT the code —
// the input changes between attempts. Even better: resetKeys={[code]}.
2 (easy): Why does this crash never reach the boundary? <ErrorBoundary><button onClick={() => { throw new Error("boom"); }}>Buy</button></ErrorBoundary> Where does it go, and what's the fix?
Solution 
// The throw happens in an event handler — React isn't rendering, so no boundary
// participates. It surfaces as an uncaught error (window "error" event, console).
// Fix: try/catch in the handler and set error STATE:
onClick={() => { try { buy(); } catch (e) { setError(e); } }}
// …then either render the error inline, or `if (error) throw error;` in the
// body to route it into the boundary deliberately.
3 (medium): Implement useAsyncError() — a hook returning a function that forwards any async error into the nearest boundary — and use it in a websocket effect.
Solution 
function useAsyncError() {
  const [, setState] = useState();
  return useCallback(err => {
    setState(() => { throw err; });     // updater throws DURING render → boundary
  }, []);
}

function LiveFeed() {
  const throwAsync = useAsyncError();
  useEffect(() => {
    const ws = new WebSocket(URL);
    ws.onerror = () => throwAsync(new Error("Feed connection failed"));
    ws.onmessage = e => { /* … */ };
    return () => ws.close();
  }, [throwAsync]);
  /* … */
}
// The updater-that-throws trick converts an async-context error into a
// render-phase throw — exactly what react-error-boundary's showBoundary does.
4 (medium): Add resetKeys support to the Part 2 class boundary: when any value in the array changes, clear the error automatically.
Solution 
class ErrorBoundary extends React.Component {
  state = { error: null, prevKeys: this.props.resetKeys ?? [] };

  static getDerivedStateFromProps(props, state) {
    const keys = props.resetKeys ?? [];
    const changed = keys.length !== state.prevKeys.length ||
      keys.some((k, i) => !Object.is(k, state.prevKeys[i]));
    if (changed) return { error: null, prevKeys: keys };   // input changed → new try
    return null;
  }
  static getDerivedStateFromError(error) { return { error }; }
  /* componentDidCatch + render as before */
}
//  now recovers when the user picks a
// different product — no retry button needed for the navigation-away case.
5 (medium): The dashboard's TileError fallback reads useAuth() to show the user's support tier — and one day the auth context itself is what crashed. Trace the failure and restate the fallback design rule.
Solution 
// AuthProvider's crash unmounts down to the APP boundary. But even in the
// widget-crash case: TileError calling useAuth() means the FALLBACK depends on
// live app machinery — if that machinery is the broken part (provider missing,
// context poisoned), rendering the fallback THROWS AGAIN, and the error escapes
// to the next boundary up: your containment tier just widened because the
// fallback was fancy.
// Rule: fallbacks are the crash-safe zone — static markup, props already in
// hand (boundary name, error message), zero hooks into app state, zero data
// fetching. Want the support tier? Pass it as a plain prop captured earlier,
// or leave it out.
6 (hard): Build the full Widget wrapper for a dashboard tile: boundary + suspense + retry that invalidates the tile's query + reporting with the tile name, and make the fallback accessible.
Solution 
function Widget({ name, queryKey, children }) {
  const qc = useQueryClient();
  return (
    <ErrorBoundary
      name={name}
      onError={(err, info) => reportError(err, { boundary: name, ...info })}
      fallback={(err, reset) => (
        <div role="alert" className="tile tile-error">
          <p>This panel hit a problem.</p>
          <button onClick={() => {
            qc.invalidateQueries({ queryKey });     // clear the poisoned data FIRST
            reset();                                // then re-render → refetch → suspend
          }}>Try again</button>
        </div>
      )}>
      <Suspense fallback={<div className="tile tile-skel" aria-hidden="true" />}>
        {children}
      </Suspense>
    </ErrorBoundary>
  );
}
// Order matters: boundary OUTSIDE suspense (a crash while suspended still
// contains); invalidate BEFORE reset (or the same cached poison re-crashes);
// role="alert" announces the failure; the skeleton is aria-hidden (decoration).

🎯 Key Takeaways

  • Default React is all-or-nothing: one render throw unmounts the root. Boundaries are declared bulkheads — app, route, and independently-useful widget seams; shell/nav never inside.
  • The class API: getDerivedStateFromError renders the fallback (pure), componentDidCatch reports (effects). Boundaries pair with Suspense at the same seams.
  • They catch render-phase throws only — handlers use try/catch, async errors need re-throw-into-render (showBoundary) or status UI. Expected failures are states, not crashes.
  • Recovery = clearing the cause, not just the flag: resetKeys on input change, retry-with-invalidation, key remounts, reload for chunk errors. Never auto-retry a deterministic crash.
  • Fallbacks are crash-safe zones: static, hook-free, unable to throw.
  • Every catch feeds the tracker with boundary name, componentStack, and release (sourcemaps) — alert on rates per region, and treat each caught error as a bug with a paper trail.
