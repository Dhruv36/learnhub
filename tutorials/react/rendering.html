Lists, Keys & Conditional Rendering Basics

JSX has no if or for — you compose UI with JavaScript expressions: ternaries, &&, and map. The one genuinely deep topic here is keys — the mechanism by which React matches list items across renders, and the source of some of the sneakiest UI bugs in existence.

Conditional Rendering — the Three Forms

{isLoggedIn ? <Dashboard /> : <Login />}          // either/or
{error && <Alert message={error} />}                // render-if (nothing when falsy)
{items.length === 0 && <EmptyState />}

// Early returns for whole-component branching (cleanest for loading/error/empty):
function Profile({ user, loading, error }) {
  if (loading) return <Spinner />;
  if (error)   return <ErrorPage error={error} />;
  if (!user)   return <NotFound />;
  return <main>…</main>;                            // happy path last, unindented
}
// null/undefined/false render NOTHING — returning null is how a component says "skip me".
🐛 The && number trap: {items.length && <List />} renders a literal 0 when empty (0 is falsy but renderable!). Fix: items.length > 0 && or a ternary. This exact bug ships to production weekly, worldwide.

Lists — map + key

<ul>
  {todos.map(todo => (
    <TodoItem key={todo.id} todo={todo} onToggle={() => toggle(todo.id)} />
  ))}
</ul>
// key: STABLE (same item → same key every render), UNIQUE among siblings, on the
// OUTERMOST mapped element. Filter+map chains compose fine — it's just JavaScript.

Why Keys Exist — the Reconciliation Story

Without keys, React matches list children BY POSITION:
  render 1:  [A, B, C]        render 2:  [X, A, B, C]   (inserted at front)
  positional match: A→X, B→A, C→B, (new)→C   →  React REWRITES all four
With id keys, React matches BY IDENTITY: X is new, A/B/C untouched → one insertion.

The real damage isn't performance — it's STATE. Component state (input text, focus,
checkbox, animation) lives with the MATCHED slot:
  index-as-key + delete first row  →  every row's state shifts up one 😱
  ("I deleted Bob but Carol's checkbox cleared")
Key choiceVerdict
Stable id from data (db id, uuid assigned at CREATION)✅ Correct
Array index⚠ Acceptable ONLY for static, never-reordered, never-filtered lists with stateless items
Math.random() / uuid() during render❌ New key every render = remount everything, state wiped, focus lost

The Key Superpower — Deliberate Remounting

<ProfileForm key={userId} userId={userId} />
// Changing key tells React: this is a DIFFERENT instance — unmount, remount, fresh state.
// The idiomatic way to reset a form when switching records — no effect-that-resets-state hacks.

Interview Questions

Q1: Explain exactly what goes wrong with index keys when deleting the first item of a stateful list.

Keys 0..n-1 still exist after deletion — React matches old item[1] to new key 0's slot, etc.: every component keeps the WRONG item's state (inputs, checkboxes, focus shift up one row). React sees "same keys, updated props", not "first item gone". Stable ids make identity explicit so the deleted item unmounts and others keep their state.

Q2: How would you reset a child form when the selected user changes — without useEffect?

key={userId} on the form. Key change = new identity = unmount/remount with fresh useState initials. The effect-based alternative (watch userId, setState back to defaults) is more code, runs after paint (flicker), and misses fields. Key-as-reset is the documented idiom.

Q3: Why must keys be unique only among SIBLINGS, and where does key NOT work like a prop?

Reconciliation matches within one parent's child list — different lists can reuse ids freely. And key is consumed by React, not passed to the component: reading props.key is undefined; pass the value separately if the child needs it.

🏋️ Practice: build a todo list with a text input per row; use index keys, delete the first row, and watch state corruption live; fix with ids. Then implement the {0} bug and both fixes. Finish with key={selectedId} form-reset.

🎯 Key Takeaways

  • Conditionals are expressions: ternary, && (mind the 0 trap!), early returns; null renders nothing.
  • Lists are map(); key on the mapped root — stable, unique among siblings.
  • Keys are identity for reconciliation: index keys corrupt state on reorder/delete; random keys remount everything.
  • key change = deliberate remount — the idiomatic state-reset tool.
  • key isn't a prop; ids should be born with the data.
