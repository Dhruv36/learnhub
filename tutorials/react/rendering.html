Lists, Keys & Conditional Rendering Foundations

What you'll master: rendering collections with map, what keys actually do inside reconciliation (not "a React rule" — a correctness mechanism), why index-as-key corrupts state and how to prove it, keys as an intentional reset tool, and the reconciliation rules that decide when components keep or lose their state. Prerequisites: State — you need to care about where state lives before keys make sense — and JS Arrays.

Part 1 — Lists Are Just Arrays of Elements

const USERS = [
  { id: "u1", name: "Ada",   role: "admin" },
  { id: "u2", name: "Linus", role: "dev" },
  { id: "u3", name: "Grace", role: "dev" },
];

function UserList() {
  return (
    <ul>
      {USERS.map(u => <UserRow key={u.id} user={u} />)}   {/* array of elements — JSX renders it */}
    </ul>
  );
}

// Chain array methods for filter/sort/slice views — derived in render, never stored (State lesson):
{USERS.filter(u => u.role === "dev")
      .toSorted((a, b) => a.name.localeCompare(b.name))
      .map(u => <UserRow key={u.id} user={u} />)}

There is no list primitive in React — JSX renders arrays of elements, and everything you know from Array Methods is the list API. The one new requirement: key.

Part 2 — What Keys Are Actually For

When a list re-renders, React holds the old array of elements and the new one, and must answer: which old item corresponds to which new item? The answer decides which fibers (state, DOM nodes, effects) survive. Without keys, React matches by position. Keys replace position with identity:

// Render 1:  [A, B, C]        Render 2:  [X, A, B, C]   (X prepended)
//
// WITHOUT keys (positional matching):
//   old[0] A ↔ new[0] X   → "A changed into X": update props, KEEP A's state/DOM for X ⚠️
//   old[1] B ↔ new[1] A   → B's fiber now renders A's props ⚠️
//   old[2] C ↔ new[2] B   → C's fiber renders B ⚠️
//   (none) ↔ new[3] C     → mount a NEW C with fresh (empty) state ⚠️
//   Every row re-rendered, and every row's STATE is attached to the wrong data.
//
// WITH stable keys:
//   key "a"↔"a", "b"↔"b", "c"↔"c" matched regardless of position → untouched;
//   key "x" is new → one mount. Correct AND minimal work.
The one-sentence truth: a key tells React "this child is the same logical entity as the child with this key last render." State follows the key, not the data you passed. Get keys wrong and React's bookkeeping attaches the right state to the wrong rows — silently.

Part 3 — Why Index-as-Key Corrupts State (the Proof)

function Guests() {
  const [guests, setGuests] = useState(["Ada", "Linus"]);
  return (
    <>
      <button onClick={() => setGuests(g => ["Grace", ...g])}>Add to top</button>
      {guests.map((name, i) => (
        <label key={i}>                        {/* ❌ index as key */}
          {name}: <input placeholder="meal choice" />   {/* DOM state lives here */}
        </label>
      ))}
    </>
  );
}
// 1. Type "vegan" into Ada's input (row key=0).
// 2. Click "Add to top" → Grace becomes index 0.
// 3. React matches new key=0 (Grace) to old key=0 (Ada's fiber+DOM):
//    Grace now shows "vegan". Ada's row shows Linus's old input. Data shifted; state didn't.
//
// Same corruption hits: checkbox selections, uncontrolled inputs, row animations,
// component state (expanded/collapsed), and effect cleanup order.
Key choiceVerdictWhy
Database/entity id (u.id)✅ bestStable, unique, survives reorder/insert/delete
Stable natural key (email, slug, ISO date of the row)✅ fineSame properties if truly unique & immutable
Generated at creation time (crypto.randomUUID() stored on the item)✅ fineCreated once with the data — stable thereafter
Array index⚠️ last resortOnly for static, never-reordered, never-filtered, stateless rows
Math.random() / uuid in render❌ neverNew key every render = unmount + remount every row, every render

Part 4 — Keys as a Feature: Intentional Resets

Keys aren't only for lists. Changing a key on any element tells React "different entity — throw the old one away." That's the idiomatic way to reset state:

// 1. Reset a whole form when switching records — instead of syncing every field:
<ProfileForm key={selectedUserId} userId={selectedUserId} />
// New user id → new key → old form unmounts (all its state gone), fresh one mounts.
// This one line replaces a minefield of "sync props to state" effects.

// 2. Restart an animation / replay a video / re-trigger an intro:
<Confetti key={celebrationCount} />

// 3. Force a remount to clear an error boundary or a poisoned third-party widget:
<ChartWidget key={retryToken} data={data} />
Interviewers love this flip: junior answer — "keys silence the console warning." Senior answer — "keys are reconciliation identity: in lists they preserve state across reorders; outside lists a changed key is a deliberate demolition order." Same prop, both directions.

Part 5 — Reconciliation Rules Beyond Keys

// RULE 1 — same position + same type → same fiber, state survives, props update:
{isSpecial ? <Panel tone="gold" /> : <Panel tone="gray" />}   // ONE Panel, state kept

// RULE 2 — type changes → old subtree unmounted (state destroyed), new one mounted:
{isEditor ? <RichEditor /> : <PlainEditor />}                 // switch = full reset of both

// RULE 3 — host type changes reset EVERYTHING BELOW, even identical children:
{fancy ? <section><Form /></section> : <div><Form /></div>} // Form remounts on toggle!

// RULE 4 — conditional siblings shift positions; React is smarter than you'd fear:
{showBanner && <Banner />}      {/* false renders a "hole" — a null slot — */}
<Content />                      {/* so Content keeps its position and state either way */}

// RULE 5 — hiding vs unmounting is a real decision:
{open && <HeavyPanel />}                          // unmount: frees memory, LOSES state
<div style={{ display: open ? "" : "none" }}>     // hide: keeps state & DOM, pays memory
  <HeavyPanel />
</div>

Rule 5 is a genuine trade-off: unmount cheap things, hide things that are expensive to rebuild or hold precious state (a half-filled form, a scrolled list, a video position). For huge lists, the real answer is virtualization — rendering only visible rows — covered in the Performance section.

Part 6 — Production List Patterns

// 1. Empty / loading / error states are part of EVERY list — design them up front:
function Results({ items, status }) {
  if (status === "loading") return <SkeletonRows count={5} />;
  if (status === "error")   return <RetryPanel />;
  if (items.length === 0)   return <p>No matches — try clearing filters.</p>;
  return <ul>{items.map(i => <Row key={i.id} item={i} />)}</ul>;
}

// 2. Grouped lists — nested maps need keys at EACH level:
{Object.entries(byCategory).map(([cat, items]) => (
  <section key={cat}>
    <h3>{cat}</h3>
    {items.map(i => <Row key={i.id} item={i} />)}
  </section>
))}

// 3. Keys must be unique among SIBLINGS only — reusing ids across
//    different lists on the page is fine.

// 4. key is consumed by React — it is NOT passed as a prop. If the child
//    needs the id, pass it separately:
<Row key={item.id} id={item.id} item={item} />

Missing key entirelyPositional matching (index behavior) + console warningKey every mapped element — on the outermost element returned

Interview Questions

Beginner: Why does React need keys on list items?

On re-render React must match old children to new ones to decide what to update, move, mount, or unmount. Keys give each child a stable identity so matching follows the data rather than array position — preserving each item's component state and DOM across reorders, insertions, and deletions, and minimizing DOM operations.

Beginner: When is index-as-key acceptable?

Only when all three hold: the list never reorders or gets filtered/inserted-into (append-only at most), items have no state (no inputs, no selection, no animations), and there's genuinely no stable id. Static footer links qualify. Anything user-editable does not.

Mid: A row's input text jumps to a different row after deleting an item. Diagnose.

Index keys (or no keys). Deleting item i shifts every later item down one index; React matches by key=index, so each later fiber — including its uncontrolled input DOM — is reused for the next item's data. The state stayed at position i; the data moved. Switch to entity-id keys and each fiber follows its item.

Mid: How do you reset a child component's state from the parent without a callback or effect?

Change its key. A new key means "different entity": React unmounts the old instance (discarding all its state) and mounts a fresh one. Canonical use: <Form key={recordId}/> so switching records never leaks the previous record's half-edited fields — replacing fragile "sync props into state" effects with one attribute.

Senior: Explain React's list diffing and its complexity trade-off.

A full minimal tree-edit-distance diff is O(n³); React opts for O(n) heuristics: compare level by level, assume different types mean different subtrees (no cross-type matching), and use keys to match children within a level via a map lookup. The cost: React never moves a component across tree levels or matches across type changes — those become unmount/mount. Keys are the developer's side of the bargain: you supply identity, React supplies linear-time reconciliation.

Senior: Two lists concatenated (pinned then recent) can contain the same entity. Keys?

Raw ids would collide (duplicate keys among siblings → state bleeding, wrong-node updates). Namespace per section: key={`pin-${id}`} / key={`rec-${id}`}. Consequence worth stating: an item moving from recent to pinned changes key, so it remounts and loses row-local state — if that state must survive (e.g., playing audio), lift it out of the row or render one list with a computed section field instead of two concatenated maps.

3 (medium): Render products sorted by price, only those in stock, with a "Nothing in stock" empty state — no extra state variables.
Solution 
function InStock({ products }) {
  const visible = products.filter(p => p.stock > 0)
                          .toSorted((a, b) => a.price - b.price);   // derived, not state
  if (visible.length === 0) return <p>Nothing in stock</p>;
  return <ul>{visible.map(p => <li key={p.id}>{p.name} — ${p.price}</li>)}</ul>;
}

Solution

{tab === "write" ? <Editor doc={doc} mode="write" /> : <Editor doc={doc} mode="preview" />}

// Documents: key by document id → new doc = remount = clean draft:
<Editor key={doc.id} doc={doc} mode={tab} />

5 (medium): Predict the bug: a chat renders messages with key={i} and each row has a "⭐ favorite" toggle held in row state. New messages are prepended.
Solution 
// Prepend shifts every existing message's index by +1. Row fibers stay with
// their old index-keys, so every star you set stays at its POSITION: after one
// new message arrives, every favorite appears on the message ABOVE the one you

6 (hard): Build a TransferList: two side-by-side lists, items move between them on click. Choose keys so item state (a note input per row) survives the move — then explain why that requires ONE rendered list mentally split, not two maps.
Solution 
// Two separate maps = two sibling groups: moving an item changes its GROUP,
// and React never matches across groups/positions like that → remount, note lost.
// Render ONE keyed list; membership is data; CSS grid places the columns:
function TransferList({ items, onMove }) {          // items: [{id, side:"L"|"R", label}]
  return (
    <div className="transfer">
      {items.map(it => (
        <div key={it.id} className={`cell side-${it.side}`}
             style={{ gridColumn: it.side === "L" ? 1 : 2 }}>
          <button onClick={() => onMove(it.id)}>{it.label} ⇄</button>
          <input placeholder="note" />              {/* survives the move: same key, same fiber */}
        </div>
      ))}
    </div>
  );
}
// The fiber follows key={id} no matter which column CSS paints it in.
7 (hard): Your error tracker shows a widget crashes and stays broken until page reload. Using only concepts from this lesson, add a "Try again" that guarantees a pristine widget, and say why re-rendering alone wouldn't fix it.
Solution 
function WidgetSlot({ data }) {
  const [attempt, setAttempt] = useState(0);
  return (
    <>
      <Widget key={attempt} data={data} />
      <button onClick={() => setAttempt(a => a + 1)}>Try again</button>
    </>
  );
}
// A re-render reuses the same fiber — the widget's poisoned internal state and
// DOM survive, so it re-crashes. Bumping the key changes its identity: full
// unmount (state, DOM, effects all discarded) and a clean mount.
// (Pairs with error boundaries — see Error Boundaries & Resilience.)

🎯 Key Takeaways

  • Lists are plain arrays of elements; filter/sort/group are derived views computed in render, never stored in state.
  • Keys are reconciliation identity: React matches old↔new children by key (else by position), and component state + DOM follow the key.
  • Index keys corrupt state the moment order changes — inputs, selections, and animations attach to positions instead of items; ids created with the data are the fix.
  • Changing a key is a feature: it's React's sanctioned "reset this subtree" — record-switching forms, animation replays, crashed-widget recovery.
  • Reconciliation rules: same position+type = state survives; type change (even just the wrapper tag) = subtree remount; unmount-vs-hide is a real memory/state trade-off.
  • Keys are unique per sibling group only, live on the mapped root element, and never arrive as a prop.
