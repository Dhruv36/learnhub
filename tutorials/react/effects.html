useEffect & Data Fetching Intermediate

Render must stay pure — so where do side effects (fetches, subscriptions, timers, DOM APIs) go? useEffect: code that runs after render to synchronize your component with an external system. It is also React's most misused API — this lesson covers the correct uses AND the officially-blessed list of times you don't need it.

The Anatomy

useEffect(() => {
  const ctl = new AbortController();                       // setup
  fetchUser(id, ctl.signal).then(setUser);
  return () => ctl.abort();                                 // CLEANUP — before re-run & at unmount
}, [id]);                                                   // deps: re-run when any changes
DepsRunsMeaning
[id]After mount + whenever id changes (Object.is compare)"Stay synchronized with id"
[]After mount only (cleanup at unmount)"Connect once" — subscriptions, global listeners
noneAfter EVERY renderAlmost always a mistake

The lifecycle to internalize: render → paint → cleanup(previous) → effect(current). Cleanup runs with the PREVIOUS render's values (its closure). StrictMode dev-mounts effects twice (setup→cleanup→setup) precisely to prove your cleanup works — if that breaks you, the effect was leaky.

Data Fetching — Done Right by Hand

function useUser(id) {
  const [state, setState] = useState({ status: "loading" });   // discriminated union (TS lesson!)
  useEffect(() => {
    const ctl = new AbortController();
    setState({ status: "loading" });
    fetch(`/api/users/${id}`, { signal: ctl.signal })
      .then(r => { if (!r.ok) throw new Error(`HTTP ${r.status}`); return r.json(); })
      .then(data => setState({ status: "success", data }))
      .catch(err => err.name !== "AbortError" && setState({ status: "error", err }));
    return () => ctl.abort();          // ← kills the RACE: switching id fast can't let the
  }, [id]);                             //   slow old response overwrite the new one
  return state;
}
// Note the async pattern: the effect fn itself can't be async (it must return cleanup,
// not a promise) — use .then or an inner async fn.
🏭 Production reality: this hand-rolled version misses caching, deduping, revalidation, retries, pagination. Real apps use TanStack Query / SWR (or router loaders / RSC — later lesson): useQuery({ queryKey: ["user", id], queryFn }) gives cache + dedupe + refetch-on-focus + stale-while-revalidate. Learn the manual version to understand WHAT they solve; ship the library.

You Might NOT Need an Effect (the official greatest hits)

❌ Effect anti-pattern✅ Instead
Derive state: effect watches items+query, sets filteredCompute during render: const filtered = items.filter(…) (+ useMemo if provably expensive)
Reset child state when a prop changes (effect + setState)key={userId} remount (rendering lesson)
React to a BUTTON CLICK via state→effect chainsDo it in the event handler — effects are for "being on screen", handlers for "user did X"
Chains: effect sets A → effect on A sets B → …Compute in one place (render or handler); chains = render cascades + bugs
"Update parent when my state changes" effectLift the state; call the callback in the handler

Litmus: "Which external system does this effect synchronize with?" No answer → you don't need the effect.

Legit Effects Beyond Fetching

// Subscriptions & global listeners (with AbortController from the DOM lesson):
useEffect(() => {
  const ctl = new AbortController();
  window.addEventListener("keydown", onKey, { signal: ctl.signal });
  const ws = connectSocket(room, onMessage);
  return () => { ctl.abort(); ws.close(); };
}, [room]);

// Non-React widgets (charts, maps):
useEffect(() => {
  const chart = new Chart(canvasRef.current, config);
  return () => chart.destroy();
}, [config]);

// document.title, analytics page-views, focus management — sync with the browser.

Stale Closures & the Deps Contract

useEffect(() => {
  const id = setInterval(() => setCount(count + 1), 1000);   // ⚠ count from THIS render: logs of 0→1 forever
  return () => clearInterval(id);
}, []);                                                       // lied about deps!

// Fix A — functional update (no dep needed):
setInterval(() => setCount(c => c + 1), 1000);
// Fix B — honest deps [count] (interval re-created per tick — fine, cleanup handles it)
// Fix C — useRef for "latest value" without re-running (escape hatch, use sparingly)
// The rule: deps list EVERYTHING from component scope the effect reads —
// the exhaustive-deps lint rule is right; silencing it is how stale bugs are farmed.

Common Mistakes

Interview Questions

Q1: The interval-logs-1-forever bug — explain and give all three fixes.

The [] effect captured render #1's count (stale closure); setCount(count+1) computes 0+1 each tick. Fixes: functional update (reads current state, no dep), honest [count] deps (recreate per change — cleanup makes it correct), or a ref holding the latest. Naming it "stale closure, the deps contract exists because of closures" is the senior framing.

Q2: Why does StrictMode run effects twice in dev, and what does it catch?

It simulates unmount/remount (setup→cleanup→setup) to prove effects are resilient — catching missing cleanups (duplicate subscriptions, double intervals) and effects that can't survive re-running. Production runs once. "My effect breaks under StrictMode" = the effect is broken; fetch effects without abort/dedupe show duplicate requests, which the library solutions (or proper cleanup) absorb.

Q3: When is fetching in useEffect the wrong choice entirely?

Most production cases: TanStack Query/SWR (cache, dedupe, revalidate, retries), router loaders (fetch before render, no waterfalls), or Server Components (fetch on the server, ship HTML). Effect-fetching creates client waterfalls (parent renders → child mounts → child fetches) and re-implements caching badly. Hand-rolled effect fetching is for learning and tiny cases.

🏋️ Practice: build useUser with abort + discriminated-union state; prove the race by switching ids rapidly with throttled network (DevTools); reproduce the stale-interval bug and apply all three fixes; then delete an unnecessary derive-state effect from a filter component (compute in render instead).

🎯 Key Takeaways

  • Effects synchronize with EXTERNAL systems, after paint; every setup returns its cleanup.
  • Deps are a contract (Object.is per item): list everything read; fix designs rather than silence the lint.
  • Fetch effects need AbortController (races!) and status-union state; StrictMode double-run is a leak detector.
  • Don't effect what you can derive in render or do in the handler — "which external system?" is the litmus.
  • Production fetching = Query/SWR/loaders/RSC; know what they solve because you built the naive version.
