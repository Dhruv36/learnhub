useEffect Mastery Effects & Lifecycle

What you'll master: what effects are actually for (synchronizing with external systems — not "doing stuff after render"), the dependency array semantics precisely, cleanup as a first-class contract, the complete catalog of effect misuse (most effects you'll see in the wild shouldn't exist), and race-condition-safe async effects. Prerequisites: The Render Cycle (when effects run) and JS Closures (why deps exist at all).

Part 1 — The Real Job Description

An effect synchronizes your component with a system outside React: a server, a browser API, a subscription, a third-party widget. The mental model is not "run code after render" — it's "keep external thing X consistent with state Y":

useEffect(() => {
  // SETUP: make the external system match current props/state
  const conn = chat.connect(roomId);          // external system: chat server
  conn.on("message", onMessage);

  return () => {                              // CLEANUP: undo THIS setup completely
    conn.close();
  };
}, [roomId]);                                  // SYNC KEYS: re-sync when these change

// Timeline for a user switching rooms A → B, then leaving the page:
//   mount:        setup(A)
//   roomId → B:   cleanup(A), setup(B)        ← cleanup ALWAYS runs before re-setup
//   unmount:      cleanup(B)
// One connection alive at any moment. That invariant — setup/cleanup pairs that
// perfectly nest — is the entire discipline of this hook.
Think in synchronization, not lifecycle. "componentDidMount thinking" asks when code runs; effect thinking asks what state the external system should be in. That's why Strict Mode's remount cycle (setup → cleanup → setup) is harmless for a correct effect: the end state is identical. If double-running your effect breaks something, the effect — not Strict Mode — is wrong.

Part 2 — Dependency Semantics, Precisely

useEffect(() => { document.title = `${unread} unread`; });          // no array:
// runs after EVERY commit. Rarely what you want; usually an oversight.

useEffect(() => { analytics.pageView(); }, []);                     // []:
// setup once after mount, cleanup at unmount. Correct ONLY if the effect truly
// reads nothing that can change (the linter will tell you when you're lying).

useEffect(() => { conn = connect(roomId); … }, [roomId]);           // [deps]:
// re-sync when any dep changes (Object.is comparison, element-wise).

// THE RULE (not a style preference — a correctness rule):
// every reactive value read inside the effect belongs in the array.
// Reactive = props, state, and anything computed from them in the component body.
// The eslint-plugin-react-hooks "exhaustive-deps" rule enforces exactly this.
// Suppressing it is 95% of "works on my machine" effect bugs: the effect keeps
// running against a STALE CLOSURE — old values photographed at setup time.
// When deps feel wrong, don't delete them — restructure so they're not needed:

// A. Updater form removes state deps:
useEffect(() => {
  const id = setInterval(() => setCount(c => c + 1), 1000);   // reads no `count`
  return () => clearInterval(id);
}, []);                                                        // honest empty array

// B. Move the value INTO the effect if it's only used there:
useEffect(() => {
  const url = `/api/rooms/${roomId}`;         // derived inside — url isn't a dep
  …
}, [roomId]);

// C. Objects/functions recreated every render make deps fire every render:
const options = { serverUrl, roomId };         // ❌ new reference each render
useEffect(() => { connect(options); }, [options]);   // re-connects every render!
// Fix: depend on the primitives and build the object inside:
useEffect(() => { connect({ serverUrl, roomId }); }, [serverUrl, roomId]);

Part 3 — You Might Not Need an Effect (the Deletion Catalog)

The most common React code-review comment is deleting an effect. Each of these patterns appears in real codebases constantly, and each has a better non-effect answer:

The effect people writeWhy it's wrongReplace with
useEffect(() => setFullName(first + last), [first, last])Extra render per change; two sources of truthDerive in render: const fullName = first + last
useEffect(() => setFiltered(filter(items, q)), [items, q])Same — cascading renderDerive; useMemo if measured-expensive

"Run once" app init in a component effectRe-runs per mount; belongs to the app, not a componentModule scope, or guard in main.jsx before render
The litmus test: "should this code run because the component displayed, or because the user did something?" Displayed → effect. Did something → event handler. Almost every effect bug traces to event logic living in an effect, where it re-fires on re-syncs it never asked for.

Part 4 — Async Effects Done Right: Races and Cancellation

// The bug shipped by every React team at least once:
useEffect(() => {
  fetch(`/api/search?q=${query}`)
    .then(r => r.json())
    .then(setResults);            // ❌ what if a SLOWER response for an OLDER query
}, [query]);                      //    lands AFTER the newest one? Stale data wins.

// Fix 1 — ignore flag (the canonical pattern):
useEffect(() => {
  let ignore = false;
  fetch(`/api/search?q=${query}`)
    .then(r => r.json())
    .then(data => { if (!ignore) setResults(data); });
  return () => { ignore = true; };      // cleanup of render N runs before setup N+1 —
}, [query]);                            // so only the LATEST effect's flag stays false

// Fix 2 — AbortController: also cancels the network request itself:
useEffect(() => {
  const ctrl = new AbortController();
  fetch(`/api/search?q=${query}`, { signal: ctrl.signal })
    .then(r => r.json())
    .then(setResults)
    .catch(e => { if (e.name !== "AbortError") setError(e); });  // aborts aren't errors
  return () => ctrl.abort();
}, [query]);

// Note the shape: the effect callback is NOT async (useEffect must return a
// cleanup function, not a promise) — you call an async fn or use .then inside.

This is a real race condition, not a theoretical one: type "re" → "react", and the "re" query hits a cold cache while "react" hits a warm one — the wrong results render. The cleanup-based invalidation generalizes to every async effect: subscriptions delivering after unsubscribe, timers firing after clear, observers reporting after disconnect. Full data-fetching architecture (loading states, caching, and why libraries own this in production) is the next lesson.

Part 5 — Build It: a useWindowSize That's Actually Correct

function useWindowSize() {
  const [size, setSize] = useState(() => ({           // lazy init: read once at mount
    w: window.innerWidth, h: window.innerHeight,
  }));

  useEffect(() => {
    let frame = 0;
    function onResize() {
      cancelAnimationFrame(frame);                     // throttle to one update per frame —
      frame = requestAnimationFrame(() =>              // resize fires at high frequency
        setSize({ w: window.innerWidth, h: window.innerHeight }));
    }
    window.addEventListener("resize", onResize);
    return () => {                                     // cleanup: BOTH the listener and
      cancelAnimationFrame(frame);                     // any in-flight frame
      window.removeEventListener("resize", onResize);
    };
  }, []);                                              // window never changes: honest []

  return size;
}
// Every discipline in one hook: lazy initial read, external system (window),
// high-frequency throttling, complete cleanup, truthful deps. Packaging effects
// behind hooks like this is the subject of Custom Hooks.

Interview Questions

Beginner: When does an effect run, and when does its cleanup run?

Setup runs after the commit that mounted it or changed its dependencies (after paint, for useEffect). Cleanup runs immediately before the next setup (when deps change) and at unmount. So for deps that change n times you get: setup, (cleanup+setup)×n, cleanup — perfectly nested pairs, never two live setups.

Beginner: What's the difference between no dependency array, [], and [a, b]?

Omitted: after every commit. []: after mount only (cleanup at unmount). [a, b]: after mount and after any commit where a or b changed by Object.is. The array must list every reactive value the effect reads — it's a declaration of what the effect synchronizes with, not a performance knob.

Mid: What is a stale closure in an effect, and two structural fixes?

The effect callback captures the variables of the render that created it. If deps understate what it reads, the callback keeps running with those photographed values — an interval reading count from mount forever, for example. Fixes: use functional updates so the effect doesn't read the state at all (setCount(c => c+1)); or list the value honestly in deps so the effect re-creates with fresh values (and cleanup handles the churn); for callbacks, keep them stable with useCallback so they can be honest deps.

Mid: Why shouldn't derived state live in an effect?

Because it renders twice per change (once stale, once corrected — visible flicker under load), creates a second source of truth that can drift, and makes update order matter. Anything computable from current props/state should be computed during render — memoized only if measured-expensive. Effects are for synchronizing with things outside React, and derived values aren't outside.

Senior: Design the cleanup contract for an effect wrapping a third-party imperative widget (a map library).

Setup: instantiate the widget against a ref'd DOM node, apply current props, subscribe to widget events translating them to React callbacks. Cleanup: unsubscribe everything, call the widget's destroy() to release DOM/GL resources, null the instance ref. Prop changes that the widget can apply incrementally get their own narrow effects (map.setZoom(zoom) keyed on [zoom]) so you don't rebuild the world per change. The invariant to state: after any sequence of prop changes and Strict-Mode-style remounts, exactly one widget instance exists and reflects current props.

Senior: A fetch effect intermittently shows results for the previous search. Walk through the failure and the two standard fixes.

Effects for successive queries run concurrently at the network level; HTTP gives no ordering guarantee, so an older request can resolve after a newer one and its setResults wins — last-write-wins by arrival, not by intent. Fix 1: closure flag set in cleanup (ignore = true) so only the most recent effect may commit results. Fix 2: AbortController in cleanup — same correctness plus the browser actually cancels the socket work; catch and swallow AbortError. In production, data libraries (React Query/SWR) implement this plus caching and deduping — the pattern is still worth knowing because every async subscription has the same shape.

🏋️ Graded Exercises

1 (easy): Sync the document title to `${unread} unread messages`, restoring the original title on unmount.
Solution 
useEffect(() => {
  const original = document.title;
  document.title = `${unread} unread messages`;
  return () => { document.title = original; };
}, [unread]);
2 (easy): Find the two bugs: useEffect(async () => { const r = await fetch(url); setData(await r.json()); }, []);
Solution 
// Bug 1: async callback returns a Promise — React expects a cleanup function;
//        any cleanup is silently impossible.
// Bug 2: deps [] but the effect reads `url` — stale closure when url changes.
useEffect(() => {
  let ignore = false;
  (async () => {
    const r = await fetch(url);
    const data = await r.json();
    if (!ignore) setData(data);
  })();
  return () => { ignore = true; };
}, [url]);
3 (medium): Delete every unnecessary effect from this component and say why: it has effects computing total from cart, saving cart to the server when a checkoutClicked flag flips, and logging a pageview on mount.
Solution 
// 1. total-from-cart effect → DELETE. Derived state:
const total = cart.reduce((s, i) => s + i.price, 0);      // compute in render
// 2. checkoutClicked flag + effect → DELETE BOTH. Event logic belongs in the handler:
function handleCheckout() { api.checkout(cart); }          // no flag state at all
// 3. pageview on mount → KEEPS being an effect (synchronizing with an external
//    analytics system because the component appeared):
useEffect(() => { analytics.pageView("cart"); }, []);
// Score: 3 effects → 1. Typical.
4 (medium): Build a useDocumentVisibility() hook returning "visible"/"hidden", correctly cleaned up.
Solution 
function useDocumentVisibility() {
  const [state, setState] = useState(document.visibilityState);
  useEffect(() => {
    const onChange = () => setState(document.visibilityState);
    document.addEventListener("visibilitychange", onChange);
    return () => document.removeEventListener("visibilitychange", onChange);
  }, []);
  return state;
}
5 (medium): This chat component double-sends the join message in dev. The teammate's fix is a useRef did-join guard. Review it.
Solution 
// Reject the guard. Strict Mode's remount is REVEALING the bug: join has no
// matching leave, so the effect isn't idempotent — production remounts (key
// change, navigation back) would double-join too.
useEffect(() => {
  const conn = chat.join(roomId);          // setup
  return () => conn.leave();               // the missing symmetric cleanup
}, [roomId]);
// join/leave pairs nest perfectly now; dev's double cycle is harmless and the
// guard (plus its subtle "never rejoins after remount" bug) disappears.
6 (hard): Implement useDebouncedValue(value, ms) and explain the cleanup's role in the debounce semantics.
Solution 
function useDebouncedValue(value, ms) {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), ms);
    return () => clearTimeout(id);         // ← the debounce ITSELF:
  }, [value, ms]);                          // each new value cancels the pending
  return debounced;                         // timer before arming a new one, so only
}                                           // a value that survives `ms` unchanged lands
// Usage: const q = useDebouncedValue(input, 300);  // then fetch on [q]
// The cleanup isn't bookkeeping here — it IS the algorithm.
7 (hard): An effect subscribes to a store with store.subscribe(cb), and cb uses props (filter). A teammate wrote deps [store] and suppressed the linter because adding filter "resubscribes too much". Provide two correct designs.
Solution 
// Design A — honest deps; make resubscription cheap and correct:
useEffect(() => {
  const cb = items => setVisible(items.filter(i => i.tag === filter));
  const unsub = store.subscribe(cb);
  return () => unsub();
}, [store, filter]);          // resubscribe on filter change — usually trivially cheap

// Design B — subscribe once, read latest via ref (when resubscribe is truly costly):
const filterRef = useRef(filter);
useEffect(() => { filterRef.current = filter; });          // keep ref fresh each render
useEffect(() => {
  const cb = items => setVisible(items.filter(i => i.tag === filterRef.current));
  const unsub = store.subscribe(cb);
  return () => unsub();
}, [store]);                  // deps now honestly minimal — cb reads no reactive value directly
// (This ref pattern is what the experimental useEffectEvent formalizes.)
// Suppressing the linter is the one non-option: cb would filter with the
// mount-time value forever.

🎯 Key Takeaways

  • Effects synchronize React with external systems; the unit is a setup/cleanup pair that nests perfectly across dep changes, remounts, and unmounts.
  • Deps are a truth declaration — every reactive value the effect reads. Stale closures come from lying; fix structure (updater form, values inside, stable callbacks), never the linter.
  • Most effects in the wild should be deleted: derived state → render; event logic → handlers; resets → keys; parent notification → same handler.
  • Async effects need race protection: ignore flag or AbortController in cleanup, so only the newest request commits.
  • Object/function deps rebuilt per render make effects fire per render — depend on primitives, construct inside.
  • Strict Mode's double cycle is a test harness: idempotent effects with real cleanup pass it for free; did-run guards just hide production bugs.
