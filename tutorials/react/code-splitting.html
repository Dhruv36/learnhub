Code Splitting & Suspense Performance

What you'll master: why bundle size is the performance cost users pay before React renders anything, lazy() + Suspense mechanics, where to split (routes first, then heavy islands), preloading strategies that hide the latency you created, and Suspense as a general async-UI boundary — including data. Prerequisites: Routing, JS Modules (dynamic import()), Tooling (bundlers).

Part 1 — The Cost Nobody Profiles: the Bundle

// The Profiler measures renders. But before ANY render:
// download JS → parse → execute → hydrate/mount. On a mid-range phone over 4G,
// 1MB of JS ≈ multiple seconds of blank/frozen page — spent partly on the admin
// panel, the chart library, and the markdown editor the user may NEVER open.
//
// Code splitting = letting the bundler cut the app at import() seams:
import("./ChartPanel")        // a SEPARATE file, fetched when this line RUNS
// Static import = in the main bundle, paid by everyone at startup.
// Dynamic import() = its own chunk (plus its exclusive dependencies), paid on demand.
//
// Audit before cutting: `vite build` size report / bundle analyzer — find the
// heavy libraries and ask "who actually needs this at startup?" The usual
// suspects: charts, editors, PDF/export, maps, admin sections, locale data.

Part 2 — lazy + Suspense: the Mechanics

import { lazy, Suspense } from "react";

const ChartPanel = lazy(() => import("./ChartPanel"));
// lazy() takes a function returning import()'s promise, and returns a component.
// First render of : React calls the loader, then SUSPENDS —
// pauses that subtree, shows the nearest Suspense fallback, resumes when loaded.
// (Requires a default export; named: .then(m => ({ default: m.ChartPanel })))

function Dashboard() {
  const [showChart, setShowChart] = useState(false);
  return (
    <>
      <button onClick={() => setShowChart(true)}>Analytics</button>
      {showChart && (
        <Suspense fallback={<ChartSkeleton />}>    {/* shown while the chunk loads */}
          <ChartPanel />                            {/* mounted after; cached after that */}
        </Suspense>
      )}
    </>
  );
}
// The chunk loads ONCE per session — later mounts are instant.
// Suspense boundary placement = UX decision: the fallback replaces EVERYTHING
// inside the boundary. Wrap tight (the chart slot keeps the page interactive);
// wrapping the whole page blanks it. Multiple boundaries = independent loading
// regions, exactly like error boundaries (next section — they pair naturally).

Part 3 — Where to Split: Routes First, Then Islands

// 1. ROUTES — the highest-value cut, aligned with user intent:
const Admin    = lazy(() => import("./pages/Admin"));      // visitors skip admin code
const Settings = lazy(() => import("./pages/Settings"));
const router = createBrowserRouter([
  { path: "/", element: <Shell />, children: [
    { index: true, element: <Home /> },                    // home stays STATIC — it's
    { path: "admin/*",  element: <Admin /> },              // the critical path
    { path: "settings", element: <Settings /> },
  ]},
]);
// One  around the Shell's  gives every route a fallback slot.
// (Data-router idiom: route.lazy = () => import("./pages/Admin") loads component
//  AND loader together.)

// 2. HEAVY ISLANDS below the fold or behind interaction: modals, editors,
//    charts, exports. The trigger is the interaction that reveals them.

// 3. LIBRARY-ONLY splits — no component involved, just defer the payload:
async function exportPdf(data) {
  const { jsPDF } = await import("jspdf");        // 300KB paid at click time,
  new jsPDF().text(render(data), 10, 10).save();  // by the 2% who export
}

// ANTI-SPLITS: tiny components (a chunk has HTTP+cache overhead — don't split
// 3KB), anything on the critical first paint, and code shared by every route
// (the bundler hoists it to a shared chunk anyway).

Part 4 — Preloading: Hiding the Latency You Created

// Splitting trades startup cost for navigation cost. Preloading buys both:
// fetch the chunk BEFORE the user needs it, in idle/hover time.

const loadAdmin = () => import("./pages/Admin");     // one loader, shared:
const Admin = lazy(loadAdmin);                        // - lazy uses it
<Link to="/admin" onMouseEnter={loadAdmin}           // - hover starts the fetch
      onFocus={loadAdmin}>Admin</Link>               //   (~200ms head start = free nav)
// import() dedupes: called twice, one network fetch.

// Strategy ladder:
// on-hover/focus  → likely next navigation (as above)
// on-idle         → requestIdleCallback(loadSettings) after first paint:
//                   critical path stays lean, likely routes warm up quietly
// on-viewport     → IntersectionObserver (Refs lesson's useOnScreen) for
//                   below-fold islands
// Combined with route-loader data prefetch (Routing lesson ex. 7), a hover can
// warm BOTH the code chunk and the data cache — navigation renders instantly.

Part 5 — Suspense Beyond Code: Async UI in General

// Suspense is not a lazy()-only feature — it's React's generic "this subtree
// isn't ready" protocol. Suspense-enabled DATA sources use the same boundaries:

// React Query (suspense mode) / RSC / use():
function ProductPage({ id }) {
  const { data } = useSuspenseQuery({ queryKey: ["product", id], queryFn: … });
  return <h1>{data.name}</h1>;          // no status branches — not-ready = suspend
}
<Suspense fallback={<PageSkeleton />}>
  <ProductPage id={42} />               {/* code chunk AND data share one fallback */}
</Suspense>

// What boundaries buy over per-component status flags (Data Fetching lesson):
// • Loading UI is declared ONCE at the layout level, not per fetch.
// • Sibling suspending children are batched into one fallback — no spinner chorus.
// • SSR streaming: the server flushes the shell, streams each boundary's HTML
//   as its data resolves (Server Components lesson).
//
// TRANSITIONS interact critically: wrapping navigation in startTransition keeps
// the CURRENT screen up (no fallback flash) while the next screen suspends —
// the router does this for you; the mechanism is the next lesson's subject.
//
// Fallback design rules: skeletons that match the final layout (no reflow jump),
// delay indicators ~150ms to avoid flashes on fast loads (Data Fetching ex. 3),
// and never let a boundary wrap so wide that typing/scroll context is lost.

Part 6 — Production Details (Senior Layer)

// 1. FAILED CHUNKS: a deploy replaces hashed chunk files; a user's stale tab
//    then requests a chunk that 404s → lazy throws → ERROR BOUNDARY territory:
//    catch it, offer "New version available — reload". (Pair every route
//    Suspense with an error boundary that detects chunk-load errors.)
// 2. VENDOR CHUNKING: bundlers split node_modules into vendor chunks with
//    content hashes — your weekly deploys change app code, not React's hash,
//    so returning users hit cache. Don't hand-tune until the analyzer says so.
// 3. Measure with Lighthouse/WebPageTest on throttled mobile: the metrics that
//    move are TTI/TBT and LCP. A split that doesn't move a metric is complexity
//    for free — revert it.
// 4. Named chunks for debugging: import(/* webpackChunkName: "admin" */ "./Admin")
//    (Vite: use the analyzer; names come from filenames.)

Interview Questions

Beginner: What does code splitting change about how an app loads?

Instead of one bundle containing every feature, the bundler emits chunks at dynamic import() boundaries. Startup downloads only the critical-path chunk; feature chunks load when their code path runs (navigation, interaction). Users pay for what they use, when they use it — cutting time-to-interactive on the first visit.

Beginner: How do lazy and Suspense cooperate?

lazy(() => import(…)) wraps the chunk's promise as a component; rendering it before the chunk arrives makes the component suspend. The nearest <Suspense> ancestor catches that and renders its fallback until the promise resolves, then swaps in the real subtree. The chunk is cached for the session afterward.

Mid: Where do you place Suspense boundaries, and what's the trade-off axis?

Boundary placement = which region shows a fallback together. Too wide: one slow piece blanks unrelated, already-usable UI. Too narrow: spinner confetti — many independent flickers. Practical layout: one boundary at the route outlet (page-level skeleton), plus tighter ones around independently-slow islands (chart, comments). Siblings under one boundary batch into a single fallback, which is usually the calmer UX.

Mid: Your route-split app now shows a spinner on every first navigation. Fix the regression.

Preload on intent: share the import thunk between lazy() and the nav link's hover/focus handlers so the fetch starts ~200ms before the click; warm likely routes in requestIdleCallback after first paint; and wrap navigation in a transition (data routers do this) so the current page stays visible while the chunk+data load — the fallback only appears on genuinely slow loads. Combined, most navigations render with zero visible loading.

Senior: Explain the post-deploy "ChunkLoadError" pattern and your production handling.

Chunks are content-hashed; a deploy deletes/replaces old hashes. Tabs opened pre-deploy hold a manifest pointing at dead URLs — their next lazy navigation 404s and the lazy component throws. Handling: an error boundary at route level that detects chunk-load failures (error name/message match) and renders "A new version is available — reload," optionally auto-reloading once (guard against loops with a sessionStorage flag). Serving old chunks for a grace period (immutable storage, additive deploys) reduces the incidence; the boundary handles the rest. This failure is guaranteed at scale — treating it as an edge case is how you get white-screen reports every release.

Senior: Suspense-for-data vs status-flag rendering — when is each right?

Status flags (Data Fetching lesson) keep loading logic local and explicit — right for fine-grained in-component states (inline retries, optimistic UI, partial refresh spinners) and codebases not on suspense-enabled sources. Suspense boundaries centralize loading at layout seams — right when multiple sources should share one skeleton, for SSR streaming, and with RSC/useSuspenseQuery where not-ready is the idiom. They compose: boundaries for the page's structural loading, status flags for interaction-level feedback inside loaded regions. The mistake is dogmatic exclusivity in either direction.

Solution 
// lazy() inside the body creates a NEW component type each render — React sees

// Same disease as defining components inside components (Foundations L1).
// Fix: hoist to module scope: const Editor = lazy(() => import("./Editor"));
3 (medium): A "Download CSV" button pulls in a 200KB parsing library currently in the main bundle. Remove it from the startup cost without any component changes.
Solution 
async function downloadCsv(rows) {
  const { unparse } = await import("papaparse");        // chunk fetched at click
  const blob = new Blob([unparse(rows)], { type: "text/csv" });
  const a = Object.assign(document.createElement("a"),
    { href: URL.createObjectURL(blob), download: "export.csv" });
  a.click();
  URL.revokeObjectURL(a.href);
}
// Library-only split: no lazy/Suspense needed — import() is just an async module
// load. Add a busy state on the button for slow networks.
4 (medium): Implement hover-preloading for a lazy settings route so that clicking after a hover never shows the fallback on a warm network.
Solution 
const loadSettings = () => import("./pages/Settings");   // shared thunk
const Settings = lazy(loadSettings);

<NavLink to="/settings"
         onMouseEnter={loadSettings}
         onFocus={loadSettings}>                        {/* keyboard parity */}
  Settings
</NavLink>
// import() caches: hover fires the fetch; the click's lazy render finds the
// module already resolved → no suspension, no fallback. onFocus covers
// keyboard users — preloading is UX, so it gets the a11y treatment too.
5 (medium): Your dashboard has four widgets, each lazy. Product wants: page skeleton until the LAYOUT code is in, then each widget loading independently. Sketch the boundary structure.
Solution 
<Suspense fallback={<PageSkeleton />}>        {/* outer: the layout chunk */}
  <DashboardLayout>                            {/* lazy layout suspends here */}
    <Suspense fallback={<WidgetSkeleton />}><Sales /></Suspense>
    <Suspense fallback={<WidgetSkeleton />}><Traffic /></Suspense>
    <Suspense fallback={<WidgetSkeleton />}><Errors /></Suspense>
    <Suspense fallback={<WidgetSkeleton />}><Uptime /></Suspense>
  </DashboardLayout>
</Suspense>
// Nested boundaries: the outer one owns "nothing is ready"; once layout mounts,
// each widget's boundary isolates its own loading — one slow widget never
// blanks the other three. Boundary structure IS the loading UX spec.
6 (hard): Write the chunk-failure recovery: an error boundary that detects a failed dynamic import, auto-reloads once, and shows a manual reload prompt if it recurs. (Error boundary API: static getDerivedStateFromError — next section covers it fully.)
Solution 
class ChunkErrorBoundary extends React.Component {
  state = { failed: false };
  static getDerivedStateFromError(err) {
    const isChunkError = /Loading chunk|Failed to fetch dynamically imported/i
      .test(err?.message ?? "");
    return { failed: isChunkError ? "chunk" : "other" };
  }
  componentDidCatch(err) {
    if (this.state.failed === "chunk" &&
        !sessionStorage.getItem("chunk-reloaded")) {     // loop guard
      sessionStorage.setItem("chunk-reloaded", "1");
      location.reload();                                  // stale tab → fresh manifest
    }
  }
  render() {
    if (this.state.failed === "chunk")
      return (<div role="alert">
        A new version of the app is available.
        <button onClick={() => location.reload()}>Reload</button>
      </div>);
    if (this.state.failed) throw new Error("rethrow to outer boundary");
    return this.props.children;
  }
}
// Wrap each route's Suspense: …
// First failure: silent auto-reload fixes the stale-manifest case invisibly.
// Recurrence (reload didn't help — network issue): honest manual prompt.
// Non-chunk errors re-throw to the real error boundary — this one has ONE job.
7 (hard): The bundle analyzer shows: main 480KB (react, router, your UI kit, date-fns, a charting lib used on 2 of 9 routes, an i18n bundle with 12 locales). Design the splitting plan, predict which metric each cut moves, and name the cut you'd refuse.
Solution 
// 1. Chart lib → route-level lazy on the 2 chart routes (+ hover preload).
//    Biggest single cut; moves TTI/TBT for 7 of 9 routes' visitors.
// 2. Locales → dynamic import per locale: import(`./locales/${lang}.js`).
//    11/12ths of i18n weight gone for every user; moves TTI slightly, mostly
//    download size on slow networks.
// 3. Routes themselves → lazy each non-home route, Suspense at the outlet,

// 4. date-fns → check imports: per-function imports (tree-shaking) usually
//    shrinks it to a few KB in place — a FIX, not a split.
// REFUSE: splitting react/router/UI kit out of the critical path — every route
// needs them at startup; "splitting" them only adds a round-trip. (Letting the
// bundler put them in a cached vendor chunk is fine — that's caching strategy,
// not lazy loading.) Verify each step with the analyzer + throttled Lighthouse;
// revert any cut that doesn't move its metric.

🎯 Key Takeaways

  • Bundle size is pre-render performance: every startup byte is paid by every user — audit with the analyzer before cutting.
  • lazy(() => import(…)) at module scope + Suspense fallbacks; chunks load once per session; boundary placement is the loading-UX design.
  • Split at routes first, then interaction-gated heavy islands, then library-only import() — never tiny components or the critical path.
  • Preload what you split: shared thunk on hover/focus, idle warming, viewport observers — most navigations should never show the fallback.
  • Suspense is the general not-ready protocol: data sources, streaming SSR, and transitions all speak it; boundaries batch siblings into calm, single fallbacks.
  • Production hardening: chunk-failure error boundaries with reload recovery, layout-matched skeletons, and metric verification (TTI/LCP on throttled mobile) for every split.
