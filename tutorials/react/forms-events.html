Forms & Events Basics

React wraps DOM events in a thin, cross-browser layer (camelCase props, delegation at the root — the DOM lesson's pattern, industrialized) and gives you two ways to handle form data: controlled (React state owns the value) and uncontrolled (the DOM owns it, you read on demand). Knowing when each wins is the lesson.

Events in React

<button onClick={handleSave}>Save</button>               // reference, not handleSave()
<button onClick={() => remove(item.id)}>✕</button>       // arrow when you need args
<form onSubmit={(e) => { e.preventDefault(); submit(); }}>  // SUBMIT on the form —
                                                              // Enter key included, free
<input onChange={…} onKeyDown={e => e.key === "Escape" && close()} />
// Note: React's onChange fires per keystroke (like DOM 'input', NOT DOM 'change').
// Handlers receive a SyntheticEvent — same API as native (e.target, preventDefault…),
// pooled no more (React 18+): safe to use async.

Controlled Inputs — State Owns the Value

function Signup() {
  const [form, setForm] = useState({ email: "", plan: "free", tos: false });
  const set = (e) => {
    const { name, type, value, checked } = e.target;
    setForm(f => ({ ...f, [name]: type === "checkbox" ? checked : value }));  // one handler,
  };                                                                            // computed key
  return (
    <form onSubmit={(e) => { e.preventDefault(); api.signup(form); }}>
      <input   name="email" type="email" value={form.email} onChange={set} required />
      <select  name="plan" value={form.plan} onChange={set}>
        <option value="free">Free</option><option value="pro">Pro</option>
      </select>
      <label><input name="tos" type="checkbox" checked={form.tos} onChange={set} /> Accept</label>
      <button disabled={!form.tos}>Create account</button>    {/* UI derives from state ✅ */}
    </form>
  );
}
// value/checked bound to state + onChange updating it = controlled. The payoff:
// instant validation, conditional UI, transforms (uppercase, masks) — all just renders.

Uncontrolled — the DOM Owns It, Read on Submit

function Feedback() {
  return (
    <form onSubmit={(e) => {
      e.preventDefault();
      const data = Object.fromEntries(new FormData(e.currentTarget));   // the DOM lesson's trick
      api.send(data);
    }}>
      <textarea name="message" defaultValue="" required />   {/* defaultValue, not value */}
      <input name="rating" type="number" defaultValue={5} />
      <button>Send</button>
    </form>
  );
}
// Zero re-renders while typing. refs (useRef) give targeted reads/focus when needed.
Choose controlled when…Choose uncontrolled when…
Live validation, per-keystroke UI (search, char counters, enable/disable)Submit-only forms — read once at the end
Value transforms/masks while typingPerformance-sensitive big forms (no render per keystroke)
Value needed elsewhere in the UI as it changesIntegrating non-React libraries
Production reality: form libraries (react-hook-form) are uncontrolled-by-default with subscriptions — the best of both; know the fundamentals they wrap

Validation Pattern (controlled)

const errors = validate(form);                     // DERIVED during render — not state!
const touched = useTouched();                       // track blur per field (small state)
<input … onBlur={() => touch("email")} aria-invalid={!!(touched.email && errors.email)} />
{touched.email && errors.email && <p role="alert">{errors.email}</p>}
<button disabled={Object.keys(errors).length > 0}>Submit</button>
// Errors derive from form state (single source of truth); 'touched' keeps pristine
// fields quiet — the :user-invalid idea implemented in React. Server re-validates ALWAYS.

Interview Questions

Q1: Controlled vs uncontrolled — define both and give the decision rule.

Controlled: value={state} + onChange → React state is the single source of truth, re-render per keystroke, enables live validation/transforms/derived UI. Uncontrolled: DOM holds the value; read via FormData/refs on submit; zero typing re-renders. Rule: need the value DURING editing → controlled; need it only AT submit → uncontrolled. Mention react-hook-form as the production middle path.

Q2: Why does React warn about switching between controlled and uncontrolled?

It happens when value goes undefined → defined (or back) — usually state initialized as undefined. React can't reconcile who owns the value; behavior becomes ambiguous (user edits vs prop pushes). Fix: initialize to "" and keep value always defined, or commit to defaultValue.

Q3: Build field-level validation UX like the CSS :user-invalid behavior.

Derive errors = validate(form) in render; keep touched flags set onBlur; show error + aria-invalid only when touched && error; validate all + focus first error on submit. Derived-not-stored errors can't go stale; touched prevents scolding pristine fields; role="alert" announces to screen readers — a11y carries over from the HTML track.

🏋️ Practice: build the Signup form controlled (one object + computed-key handler) with derived validation + touched; then rebuild it uncontrolled with FormData; add a char-countdown textarea to feel why that one must be controlled.

🎯 Key Takeaways

  • camelCase handler props, references not calls, submit on the FORM; React onChange = DOM input.
  • Controlled = state owns value (live UX); uncontrolled = DOM owns it, FormData on submit (cheap).
  • One state object + name/computed-key handler scales; value must never flip defined↔undefined.
  • Errors are DERIVED, touched is state, server re-validates always.
  • Form libraries wrap these exact fundamentals — learn the fundamentals first.
