Server Components & Next.js Ultra-Advanced

React Server Components (RSC) split the component tree across the network: server components run only on the server — their code never ships to the browser — while client components hydrate and handle interactivity. This builds directly on the HTML track's SSR/hydration lesson: RSC is the industry's answer to "hydration cost scales with shipped JS". Next.js App Router is where you'll use it.

The Two Component Kinds

// SERVER component (the DEFAULT in Next.js App Router) — async, touches the backend directly:
async function ProductPage({ params }) {
  const product = await db.products.find(params.id);     // no API layer needed! runs server-side
  return (
    <article>
      <h1>{product.name}</h1>
      <Markdown source={product.description} />          {/* heavy lib stays on the server */}
      <AddToCart productId={product.id} />               {/* interactive island */}
    </article>
  );
}

// CLIENT component — opts in with a directive; this is where hooks/events live:
"use client";
function AddToCart({ productId }) {
  const [qty, setQty] = useState(1);
  return <button onClick={() => cart.add(productId, qty)}>Add</button>;
}

RSC vs SSR — the Distinction Interviews Test

SSR:  renders the FIRST PAINT's HTML on the server — then ships ALL component JS
      and hydrates everything. Faster first view; same bundle.
RSC:  server components' code NEVER ships. The bundle contains only client islands.
      They compose: Next.js SSRs the initial HTML AND uses RSC — server tree stays
      server-only, client islands hydrate. Result: bundle ∝ interactivity, not page size.

Server Actions — Mutations Without API Routes

"use server";
export async function addTodo(formData) {
  await db.todos.insert({ text: formData.get("text") });
  revalidatePath("/todos");                       // refresh the server tree
}
// Client: <form action={addTodo}> — works BEFORE hydration (progressive enhancement,
// the HTML forms lesson's ideal), typed end-to-end, no hand-rolled POST endpoint.
// Security note: actions are public endpoints — validate input server-side like any API.

Streaming & Suspense — Progressive Delivery

<Suspense fallback={<SkeletonGrid />}>

</Suspense>

// fallbacks — the HTML track's streaming-SSR row, expressed as JSX. loading.tsx is
// this pattern applied to a whole route segment.

Interview Questions

Q1: What problem do RSC solve that SSR alone doesn't?

SSR speeds up first paint but still ships and hydrates ALL component JS — cost grows with the page. RSC removes server-only components from the bundle entirely (their markdown renderers, data layers, formatting libs never download); hydration cost becomes proportional to actual interactivity. It's the islands architecture (HTML track) built into React's component model.

Q2: Where do you place "use client" boundaries in a product page, and why?

At the interactive leaves: AddToCart button, image gallery, review form. The page, product data rendering, markdown description stay server-side (zero JS). Placing the directive high (page level) drags the whole import graph client-side. Rule: server by default, client at the smallest subtree that needs state/events — and remember server children can be passed INTO client components via children to keep even wrappers thin.

Q3: How do Server Actions change the classic frontend/backend split?

The API-route layer for first-party mutations largely dissolves: a typed server function is imported and used as a form action or called from client code; React handles the transport, revalidation refreshes the server tree, and forms work pre-hydration. What DOESN'T change: they're public endpoints — auth/validation/rate-limiting still apply. Third-party consumers still need real APIs.

🏋️ Practice: scaffold create-next-app; build a products list (server component + direct data), a detail page with [id], an AddToCart client island, a Server Action form with revalidatePath, and a Suspense-streamed slow section; then check the browser bundle to verify the server components shipped no JS.

🎯 Key Takeaways

  • Server components run only server-side (async, direct data, zero bundle); client components hydrate interactivity.
  • "use client" is a boundary that drags its import subtree client-side — place it at the leaves.
  • RSC ≠ SSR: SSR speeds first paint; RSC shrinks the bundle; Next.js composes both with streaming Suspense.
  • Server Actions = typed, progressively-enhanced mutations — still public endpoints: validate and authorize.
  • Parallel fetches + Suspense per slow region; secrets and heavy libs stay server-side by construction.
