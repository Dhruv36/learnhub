Architecture for Large Apps Ultra-Advanced

Everything so far scales a component; this lesson scales a codebase — 50 engineers, 400 routes, five years. React itself is unopinionated about structure, so the architecture IS your decisions: file organization, layer boundaries, state classification, error/loading strategy, and the enforcement that keeps them true. (This is the CSS design-systems lesson's twin, for logic.)

Organize by Feature, Not by Type

❌ /components (300 files)  /hooks  /utils        ← type-sorted: every feature change
                                                     touches five distant folders
✅ src/
   features/
     checkout/      components/  hooks/  api.ts  types.ts  index.ts   ← PUBLIC API
     catalog/       …
     auth/          …
   shared/          ui/ (design system)  lib/ (utils)  api/ (client, interceptors)
   app/             routes, providers, layout shells

// The index.ts is the feature's contract: other features import ONLY from
// features/checkout — never reach into its internals. Enforce with ESLint
// (import boundaries / no-restricted-imports); a monorepo (Nx/turborepo) makes the
// boundaries physical packages. This is the modules lesson's encapsulation, at scale.

The Layer Cake — Separating Concerns

UI components        pure props → JSX; no fetching, no business rules (trivially testable)
Hooks layer          useCheckout(): wires state + queries + actions for the feature
Data layer           TanStack Query hooks / RSC loaders; ALL fetch policy lives here
API client           one wrapper: base URL, auth, typed errors, retry (the async lesson's api())
Domain logic         pure functions: pricing, validation, transforms — no React imports at all

// The test: can you swap REST→GraphQL touching only the bottom two layers?
// Can you unit-test pricing without rendering anything? If yes, the layers are real.

State Classification (the decision that shapes everything)

Server state  → TanStack Query / RSC     (cache semantics, not client state!)
URL state     → the router                (filters, tabs, pagination — shareable)
Client state  → Zustand / context+reducer (cart, UI prefs — the small remainder)
Local state   → useState                  (toggles, inputs — keep it local!)
Form state    → react-hook-form / actions
// Misclassification is THE root cause of most Redux-era bloat. Classify first;
// the global store should be embarrassingly small. (Full table: Context & State lesson.)

Error & Loading Architecture — Boundaries Everywhere

<ErrorBoundary fallback={<PageError />}>          // route level: page fails, app survives
  <Suspense fallback={<PageSkeleton />}>
    <Route />
      <ErrorBoundary fallback={<WidgetError />}>   // widget level: chart fails, page survives
        <Suspense fallback={<ChartSkeleton />}><Chart /></Suspense>
      </ErrorBoundary>
  </Suspense>
</ErrorBoundary>
// Same philosophy as the JS errors lesson: catch where you can ACT (render a fallback),
// report to telemetry (Sentry), let the rest bubble. Error boundaries are class components
// or react-error-boundary; they don't catch event-handler errors (handlers try/catch themselves).

The Provider Stack & Composition Root

// app/providers.tsx — ONE place where the app is assembled:
<QueryClientProvider client={queryClient}>
  <ThemeProvider>
    <AuthProvider>
      <RouterProvider router={router} />
    </AuthProvider>
  </ThemeProvider>
</QueryClientProvider>
// Everything injectable (api client, feature flags, analytics) enters here —
// tests and Storybook swap providers, not internals. Dependency injection, React-style.

Interview Questions

Q1: Design the frontend architecture for a 40-engineer dashboard product. Go.

Feature folders with public APIs (lint-enforced), shared/ui design system (Storybook + visual regression), layered data access (Query + one api client), state classified (server/URL/client/local), route-level code splitting with error/Suspense boundaries per level, provider composition root, TS strict, CI: types+lint+RTL+bundle budgets+E2E criticals. Team-scaling story: features ≈ ownership boundaries; monorepo packages when teams need independent velocity. The structure mirrors Conway's law deliberately.

Q2: Why must UI components not fetch, in one concrete scenario?

A ProductCard fetching its own price: renders in a 50-card grid → 50 requests (waterfall, no dedupe), untestable without network mocks everywhere, and switching to batched/GraphQL data means editing every card variant. With a data layer: one deduped query feeds props; cards are pure, tested with plain props; the transport swaps in one file. Separation = the change-cost argument, not aesthetics.

Q3: Where do error boundaries fit relative to try/catch and global handlers?

Boundaries catch RENDER-phase throws in their subtree → fallback UI + report; granularity decides blast radius (route vs widget). Event handlers and async code use try/catch (JS errors lesson) — boundaries never see those. window.onerror/unhandledrejection remain last-resort telemetry. Three layers, three jobs — naming that separation is the senior answer.

🏋️ Practice: restructure a type-sorted todo app into feature folders with index.ts contracts + an ESLint boundary rule; extract its fetching into a Query-based data layer; add route+widget error boundaries with a fake flaky component; write one RTL test per layer to prove the seams.

🎯 Key Takeaways

  • Feature folders with public-API index files, lint-enforced — encapsulation at codebase scale.
  • Layers: pure UI ← hooks ← data (Query/loaders) ← api client ← pure domain logic; swap-ability is the test.
  • Classify state (server/URL/client/local/form) — the global store should be tiny.
  • Error + Suspense boundaries at route AND widget level; handlers/async still use try/catch.
  • Architecture survives only when CI enforces it: boundaries, budgets, tests, visual regression.
