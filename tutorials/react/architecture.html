Architecture for Large Apps Patterns & Quality

What you'll master: organizing React codebases that stay changeable at 100k+ lines — feature-based structure, the dependency rules that prevent big-ball-of-mud, layering (UI / feature logic / data), design systems as internal products, and the decision records that keep teams aligned. This is the capstone of sections 1–5: every previous lesson appears here as an architectural force. Prerequisites: honestly, the whole track.

Part 1 — The Real Problem: Change Cost, Not Code Size

// Architecture questions are all one question: "when X changes, how many files
// must I touch, and how confident am I nothing else broke?"
// The forces you've already met, renamed as architecture:
// • State placement (State Architecture)  → who may know about which state
// • Contracts (Props)                     → what's public vs internal
// • Composition (Composition Patterns)    → extension without modification
// • Seams (Custom Hooks, Context DI)      → where implementations swap
// • Blast radius (Performance, Boundaries)→ failure and re-render containment
// Large-app architecture = applying those at the FOLDER and TEAM level.

Part 2 — Feature-Based Structure (and Why Type-Based Fails)

// TYPE-BASED (how tutorials start, how monoliths rot):
src/
  components/   ← 400 components from every domain, alphabetical soup
  hooks/        ← useCart next to useTooltip next to useAuth
  utils/        ← the junk drawer
// One feature's change touches every folder; nothing says what depends on what;
// deleting a feature is archaeology.

// FEATURE-BASED (scales with team count):
src/
  app/                  ← composition root: providers, router, shell (Context lesson's pyramid)
  features/
    cart/
      components/       ← CartDrawer, CartBadge — cart's OWN ui
      hooks/            ← useCart (the feature's public capability)
      api.ts            ← cart endpoints (Data Fetching's request layer)
      cartReducer.ts    ← business rules (Reducers lesson — unit-tested)
      index.ts          ← THE PUBLIC API: exports CartProvider, useCart, CartBadge
    checkout/
    search/
    auth/
  shared/
    ui/                 ← Button, Modal, Tabs — domain-FREE design system
    lib/                ← fetch client, formatters, hooks like useDebouncedValue
// The unit of ownership, review, testing, code-splitting (a feature ≈ a route
// chunk), and DELETION is the feature folder. rm -rf features/wishlist works.

Part 3 — The Dependency Rules (the Actual Architecture)

// Folders are cosmetic; the RULES make the architecture:
// 1. app → features → shared. Never backwards:
//    shared/ui/Button importing from features/cart = the design system now
//    depends on a domain — instant big-ball-of-mud seed.
// 2. Feature-to-feature imports go through index.ts ONLY:
import { useCart } from "@/features/cart";          // ✅ the public contract
import { cartReducer } from "@/features/cart/cartReducer";  // ❌ reaching into guts
//    The index is the feature's props contract (Props lesson, folder-sized):
//    internals refactor freely; the export surface is semver-serious.
// 3. Sibling features that get chatty (checkout imports 6 things from cart) are
//    telling you something: extract the shared concept (an orders feature? an
//    entities/ layer for shared domain types) or merge them.
// 4. ENFORCE mechanically — rules that live in review comments die:
//    eslint-plugin-boundaries / import/no-restricted-paths in CI, so
//    "features may not import feature internals" is a build failure, not a nag.
//
// Cross-feature COMMUNICATION without imports:
// • Compose in app/:  — the root
//   wires features together; features stay ignorant of each other.
// • Events over calls for loose coupling: checkout dispatches "order_placed";
//   analytics/inventory features subscribe (store or event emitter) — n
//   listeners without n imports.

Part 4 — Layering Inside a Feature: UI / Logic / Data

// The same three layers in every healthy feature — with everything they contain
// already built in previous lessons:

// DATA:   api.ts + query hooks — server-state lane (Data Fetching)
export function useProduct(id) {
  return useQuery({ queryKey: ["product", id], queryFn: () => api.getProduct(id) });
}
// LOGIC:  reducers, state machines, validators — PURE, exhaustively unit-tested
//         (Reducers, Forms) — no React imports in this layer where possible
// UI:     components consuming hooks from the layers above; presentational
//         pieces take plain props (testable without providers — Testing lesson)

// The payoff is SWAPPABILITY at each seam (Custom Hooks lesson's "hooks are the
// seam"): REST→GraphQL touches api.ts; a redesign touches ui/; a business-rule
// change touches one reducer + its tests. Each layer's tests live at the right
// pyramid level. When a component contains all three layers interleaved —
// fetch + rules + markup in one 400-line file — that's the refactor signal,
// and the extraction order is: rules out first (pure, easiest), then data
// hooks, then what remains is honest UI.

Part 5 — The Design System Is an Internal Product

// shared/ui is not "misc components" — it's a PRODUCT with users (your teams):
// • API discipline: enums over boolean piles, slots over config, dual-mode
//   state (Props, Composition) — because 30 teams inherit every mistake.
// • Zero domain knowledge: a Button that knows about carts is a feature
//   component wearing the wrong folder.
// • Accessibility solved ONCE: the Modal's focus trap, the Menu's keyboard nav
//   (Composition ex. 6) — headless or compound, so product teams can't ship
//   the inaccessible version.
// • Documented in Storybook (with the MSW handlers from Testing reused);
//   versioned or at least CHANGELOG'd; breaking changes ship with codemods.
// The org-level test: can a new feature be built WITHOUT adding to shared/ui?
// Mostly yes → the system covers the domain. Every feature adding "one small
// shared component" → the system is a junk drawer with a nicer name.

// DECISION RECORDS — the cheapest architecture tool:
// docs/adr/0007-server-state-via-react-query.md
//   Context: 3 features hand-rolled fetch caches, drifting…
//   Decision: all server state through React Query; stores hold client state only.
//   Consequences: delete featureX/cache.ts; new features follow Data Fetching L10.
// One page, immutable, dated. Six months later "why is it like this?" has an
// answer, and "let's rewrite it" has to argue with the original constraints.

Interview Questions

Beginner: Why feature folders over components/hooks/utils folders?

Code that changes together should live together: a feature's components, hooks, API calls, and tests form one unit of ownership, review, and deletion. Type-based folders scatter each feature across the tree, so every change fans out, nothing documents what belongs to what, and removing a feature means hunting fragments. Feature folders make the common operations — build, change, delete a feature — local.

Beginner: What belongs in shared/ and what's the one rule about it?

Domain-free code any feature may use: the design system (Button, Modal), the HTTP client, generic hooks and formatters. The rule: shared never imports from features — dependencies point one way (app → features → shared). The moment shared code knows about carts or auth, every consumer transitively couples to those domains.

Mid: How do two features interact without importing each other's internals?

Three sanctioned channels: (1) public contracts — import only from the other feature's index.ts, which is its deliberate API; (2) composition at the root — app/ wires providers and pages together so features stay mutually ignorant (<CartProvider><CheckoutPage/>); (3) events — one feature emits domain events ("order_placed") through a store/emitter, others subscribe, giving n consumers without n dependencies. Chatty sibling imports are a design signal: extract the shared concept or merge the features.

Mid: What's an ADR and why do teams that skip them regret it?

An Architecture Decision Record: a one-page dated note capturing context, the decision, and its consequences ("all server state via React Query, because three hand-rolled caches drifted"). Without them, decisions live in departed heads: new members re-litigate settled questions, "temporary" hacks lose their expiry context, and rewrites get proposed against constraints nobody remembers. ADRs make the architecture arguable — you amend a record, not an oral tradition.

Senior: You inherit a 150k-line type-based React monolith, 4 teams, weekly merge conflicts. Sequence the first quarter.

Weeks 1–2: map reality — dependency-graph tooling to find natural clusters and the tangle hotspots; pick the seams that match team ownership (Conway alignment). Weeks 3–4: erect the target skeleton (app/features/shared), move ONE low-coupling feature completely, and land the lint rules (dependency direction, no-deep-imports) as warnings. Then: strangler-pattern migration — every touched file moves to its feature; new code must follow the rules (errors now, not warnings, for new paths); each team owns migrating its features on a public checklist. Parallel: freeze utils/ (no additions), extract the design system candidates used by 3+ teams. Quarter-end goal isn't "done" — it's boundaries enforced in CI, conflicts down (teams touch their own folders), and deletion demonstrated once. Big-bang rewrites lose; converging pressure wins.

Senior: When do micro-frontends earn their cost, and what must be shared regardless?

They solve a team-topology problem — independent deploys for genuinely independent teams whose release cadences conflict — not a code-organization problem (enforced monolith boundaries solve that far cheaper). Earn-in signals: separate teams with separate on-call and deploy schedules, acquisitions with distinct stacks, an app whose CI time became the org's bottleneck. Non-negotiable shared surface: the design system (or users see the seams), routing/URL contracts, auth/session, telemetry standards, and a dependency-dedup strategy (module federation/shared deps) so users don't download React n times. Without the org problem, micro-frontends are distributed-systems tax on a UI.

2 (easy): This import appears in features/checkout/CheckoutPage.tsx: import { CartLineItem } from "@/features/cart/components/CartDrawer/LineItem"; — what's wrong, and the two legitimate resolutions?
Solution 
// Deep import into another feature's internals: cart can no longer refactor
// LineItem without breaking checkout — the contract is bypassed.
// Resolution A: cart deliberately exports it — add LineItem to features/cart/index.ts
//   (it's now public API, semver-serious).
// Resolution B: it's not really cart's — a generic line-item row belongs in
//   shared/ui with domain data passed as props.
// (And the lint rule should have made this a CI failure, not a review catch.)
3 (medium): A 380-line OrdersPage.tsx contains: two fetches, date/currency helpers, refund-eligibility rules, filter state, and the markup. Write the extraction plan with target paths, in order.
Solution 
// Order: pure first (safest), data second, UI last.
// 1. Refund rules → features/orders/refundPolicy.ts (pure fns) + unit tests.
// 2. Date/currency helpers → shared/lib/format.ts IF generic; else keep in feature.
// 3. Fetches → features/orders/api.ts + features/orders/hooks/useOrders.ts
//    (query hooks — Data Fetching shape).
// 4. Filter state → URL search params (State Architecture) via a
//    features/orders/hooks/useOrderFilters.ts.
// 5. What remains in OrdersPage.tsx: composition of hooks + presentational
//    components (OrderRow, FilterBar → features/orders/components/), each
//    taking plain props (testable without providers).
// 6. features/orders/index.ts exports OrdersPage and nothing else (yet).
// Each step ships independently green — no big-bang.
4 (medium): Write the ESLint boundary config (conceptually) enforcing: dependency direction, index-only feature imports, and a domain-free shared/ — plus one rule you'd deliberately NOT add.
Solution 
// eslint-plugin-boundaries element types:
// app: src/app/**   feature: src/features/*/**   shared: src/shared/**
// Rules:
// 1. shared  may import: shared only            (direction)
// 2. feature may import: shared + feature-INDEX of OTHER features:
//    allow "features/*/index.ts", deny "features/*/!(index).*" cross-feature
//    (own-feature internal imports stay free)
// 3. app may import: anything (it's the composition root)
// NOT added: forbidding feature→feature entirely. Tempting purity, but it
// pushes teams to copy-paste or smuggle via shared/ — the index contract is
// the enforceable middle ground. Boundaries should channel dependencies,
// not pretend they don't exist.
5 (medium): Draft the ADR for the decision this track already made: "server state lives in the query cache, client state in colocated useState/reducers, one small store for globals." Context, decision, consequences — one page.
Solution 
# ADR-0003: State placement policy
Date: 2026-07 · Status: accepted

## Context
Features were mixing server data into component state (staleness bugs
after mutations), and two teams proposed adopting Redux for everything,
while URL-restorable views (filters) were breaking refresh/share.

## Decision
- Server data: React Query, keyed per endpoint; no server data in stores.
- View state users expect to survive refresh/share: URL search params.
- Feature/UI state: colocated useState/useReducer; context per feature
  for subtree sharing.
- One Zustand store ONLY for: auth session, notifications (read by
  non-React code and 3+ features).

## Consequences
+ Staleness, dedup, invalidation handled by the cache, not hand-rolled.
+ Deleting a feature deletes its state (colocation).
- Two data APIs to learn (query hooks + store); onboarding doc updated.
- Existing features/inventory/cache.ts to be migrated (ticket ORD-412).
Revisit if: offline-first requirements arrive (would need store-backed
persistence layer).

🎯 Key Takeaways

  • Architecture = minimizing the blast radius of change; every track lesson (placement, contracts, composition, seams) reappears at folder and team scale.
  • Feature folders with index-only public contracts make features the unit of ownership, review, splitting, and deletion; type-based folders scatter every change.
  • The dependency rules are the architecture: app → features → shared, never backwards, enforced in CI — rules in review comments die.
  • Inside features, layer data / pure logic / UI so each seam swaps and tests independently; extraction order when untangling: rules, then data, then UI.
  • The design system is a domain-free internal product — API discipline, a11y solved once, changes shipped like releases. ADRs make decisions outlive their authors.
  • Right-size to your stage: enforced monolith boundaries beat micro-anything until team topology — not fashion — forces the split.
