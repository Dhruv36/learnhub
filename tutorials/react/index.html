Components & JSX Foundations

What you'll master: React's core mental model — UI as a function of state — what a component actually is (a function the framework calls, not a template), how JSX compiles to plain function calls, what an element is versus a component versus the DOM, and how to decompose a screen into a component tree the way senior engineers do it. Prerequisites: the JavaScript track through Modules — React is 90% JavaScript fluency, and every "React question" in interviews is secretly a closure, reference-identity, or array-method question.

Part 1 — The One Big Idea: UI = f(state)

Before React, UI code was imperative: "find the row, add a class, update the counter text, remember to also update the badge." Every feature multiplied the number of transitions you had to hand-code, and missing one meant a stale screen. React inverts this: you write a function that says what the UI looks like for any given data, and React figures out the DOM changes.

// IMPERATIVE (jQuery era): describe every TRANSITION — n states means n² transitions to get right
$("#count").text(count);
if (count > 9) $("#badge").addClass("hot");
else $("#badge").removeClass("hot");
if (count === 0) $("#clear-btn").hide(); else $("#clear-btn").show();

// DECLARATIVE (React): describe every STATE — one rendering of the truth
function Counter({ count }) {
  return (
    <div>
      <span className={count > 9 ? "badge hot" : "badge"}>{count}</span>
      {count > 0 && <button>Clear</button>}
    </div>
  );
}
The mental model that survives every React version: your components are pure descriptions. Data changes → React calls your functions again → compares the new description to the old one → patches only the real DOM that differs (reconciliation). You never "update the UI"; you update data and re-describe. Hold onto this — The Render Cycle makes it precise.

Part 2 — What a Component Actually Is

A component is a plain JavaScript function that (1) starts with a capital letter, (2) accepts one object argument (props), and (3) returns JSX (or null). That's the whole contract.

function UserCard({ name, role }) {          // destructure props right in the signature
  return (
    <article className="card">
      <h3>{name}</h3>
      <p>{role}</p>
    </article>
  );
}

// Used like an HTML tag — but this is NOT HTML, it's a function reference:
<UserCard name="Ada Lovelace" role="Engineer" />

Why the capital letter is not a style choice: JSX compiles differently based on case. <div> becomes the string "div" (a host/DOM element); <UserCard> becomes the identifier UserCard (your function). Name a component userCard and React renders a literal unknown HTML tag — a classic silent-failure interview trap.

// You CALL components by rendering them — never by invoking them yourself:
<UserCard name="Ada" />     // ✅ React calls UserCard when it needs to
{UserCard({ name: "Ada" })}  // ❌ works today, but breaks hooks, keys, and reconciliation —
                             //    React can't track this as a component instance

Part 3 — JSX Is Just Function Calls

JSX is syntax sugar. A compiler (Babel, SWC, esbuild — wired up in Tooling) rewrites it to plain calls before the browser ever sees it:

// You write:
const ui = (
  <section className="hero">
    <h1>Hello, {user.name}</h1>
    <UserCard name={user.name} role="admin" />
  </section>
);

// The compiler emits (modern automatic runtime):
import { jsx as _jsx, jsxs as _jsxs } from "react/jsx-runtime";
const ui = _jsxs("section", {
  className: "hero",
  children: [
    _jsx("h1", { children: ["Hello, ", user.name] }),
    _jsx(UserCard, { name: user.name, role: "admin" })
  ]
});

The result of those calls is an element: a small, plain, immutable object — { type, props, key } — describing what you want. Elements are cheap; creating thousands per render is normal. They are not DOM nodes and not component instances:

ThingWhat it isWho creates it
ComponentYour function — the recipeYou, once, at module level
Element{type, props} object — one rendered descriptionJSX, every render
Fiber (instance)React's internal bookkeeping node holding stateReact, during reconciliation
DOM nodeThe real browser object on screenReact DOM, during commit
Why this matters beyond trivia: because JSX is expressions, everything JavaScript can do to values works on UI — store elements in variables, put them in arrays, pass them as props, return them from map(). There is no template language to learn; there is only JavaScript. That's the design bet React made, and it's why JS fluency is the real React skill.

Part 4 — JSX Rules That Trip Everyone

// 1. One root — a return can only be ONE expression. Wrap siblings in a Fragment:
return (
  <>                          {/* <> = Fragment: groups children, renders NO DOM node */}
    <Header />
    <Main />
  </>
);

// 2. className, htmlFor — not class, for (those are JS reserved words; props are JS):
<label htmlFor="email" className="field-label">

// 3. Braces take EXPRESSIONS, not statements — no if/for inside JSX;
//    use ternaries, &&, or compute above the return:
<p>{isAdmin ? "Admin" : "Member"}</p>

// 4. style takes an OBJECT with camelCase keys, not a CSS string:
<div style={{ backgroundColor: "navy", paddingTop: 8 }}>   // outer {} = JSX, inner {} = object

// 5. Everything closes — , 
,  (XML rules, not lenient HTML):
<input type="text" />

// 6. {false}, {null}, {undefined} render nothing (that's what makes cond &&  work) —
//    but {0} renders "0". The classic bug:
{items.length && <List items={items} />}     // ❌ renders literal "0" when empty!
{items.length > 0 && <List items={items} />} // ✅ boolean, renders nothing

Part 5 — Build It: Decomposing a Real Screen

The senior skill isn't writing one component — it's slicing a design into the right tree. Take a product-list page and build it up:

// Step 1 — start with one lump. It works, and that's fine:
function ProductPage() {
  return (
    <div>
      <header><h1>Shop</h1><input placeholder="Search…" /></header>
      <ul>
        <li><img src="/cam.jpg" /><b>Camera</b><span>$999</span></li>
        <li><img src="/mic.jpg" /><b>Microphone</b><span>$149</span></li>
      </ul>
    </div>
  );
}
// Step 2 — extract by RESPONSIBILITY (one reason to change each), not by size:
function SearchHeader() {
  return <header><h1>Shop</h1><input placeholder="Search…" /></header>;
}

function ProductRow({ product }) {          // repeated shape → component + data
  return (
    <li>
      <img src={product.image} alt={product.name} />
      <b>{product.name}</b>
      <span>${product.price}</span>
    </li>
  );
}
// Step 3 — the page becomes structure + data, and the hardcoded rows become a map:
const PRODUCTS = [
  { id: 1, name: "Camera",     price: 999, image: "/cam.jpg" },
  { id: 2, name: "Microphone", price: 149, image: "/mic.jpg" },
];

function ProductPage() {
  return (
    <div>
      <SearchHeader />
      <ul>
        {PRODUCTS.map(p => <ProductRow key={p.id} product={p} />)}
      </ul>
    </div>
  );
}
// key={p.id} is mandatory for lists — the full why lives in Lists, Keys & Conditional Rendering.

The decomposition heuristics that scale: extract when a chunk (a) repeats, (b) has its own data requirements, (c) has its own reason to change, or (d) makes the parent unreadable. Don't extract speculatively — a 40-line component is healthier than eight 5-line ones you have to chase across files. Component boundaries also become re-render boundaries later (see The Re-render Model), so good slicing now is free performance later.

Part 6 — Under the Hood: What Happens at Startup (the Senior Layer)

// Every React app has exactly one entry like this:
import { createRoot } from "react-dom/client";
import App from "./App";

createRoot(document.getElementById("root")).render(<App />);

// What actually happens:
// 1.  creates ONE element object: { type: App, props: {} }.
// 2. render() starts reconciliation: React CALLS App(props) — your function runs.
// 3. Its returned elements are walked; every component type gets called in turn,
//    depth-first, until the tree is only host elements ("div", "h1", …).
// 4. Commit phase: React DOM creates the real nodes, in one batch, and attaches
//    ONE delegated event listener system at the root (not per-element).

Two facts from this that pay off later: your component functions must be pure (React may call them extra times — Strict Mode double-invokes on purpose to catch impurity), and rendering is not the same as touching the DOM — render is cheap function calls; commit is the expensive part React batches and minimizes.

Interview-grade phrasing: "React separates reconciliation (calling components, diffing element trees — interruptible, pure) from commit (applying DOM mutations — synchronous, batched). JSX produces element objects; components are functions from props to elements; the renderer owns the DOM." If you can say that and back each clause, you understand React's core.

Interview Questions

Beginner: What does "declarative UI" mean, concretely, versus imperative?

Imperative code specifies transitions: "when X happens, find this node and change it." Declarative code specifies states: "given this data, the UI is this." React lets you write only the state description and derives the transitions (DOM patches) itself, which eliminates the class of bugs where some transition path forgot to update some piece of the screen.

Beginner: Why must component names be capitalized?

The JSX compiler uses case to decide what to emit: lowercase tags become string types (host elements like "div" handled by the DOM renderer); capitalized tags become identifier references to your function/class. <userCard /> compiles to the string "usercard" — an unknown element — instead of a call site for your component.

Mid: What exactly does JSX compile to, and what is a React element?

Each JSX tag compiles to a call — _jsx(type, props) (or React.createElement in the classic runtime) — which returns an element: an immutable plain object like { type, props, key }. Elements are lightweight descriptions, recreated every render; they are neither DOM nodes nor component instances. React diffs new element trees against previous ones to compute minimal DOM updates.

Mid: Why does defining a child component inside a parent's function body cause bugs?

Every parent render creates a new function object for the child. React compares element types by reference for components, sees a "different" type at the same position, and unmounts the old subtree and mounts a new one — destroying the child's state, effects, and DOM (visible as inputs losing focus and state resetting). Components must be declared at module scope so their identity is stable.

Senior: Distinguish component, element, fiber/instance, and DOM node.

Component: your function, the reusable recipe. Element: the plain object one JSX expression produces on a given render — a snapshot description. Fiber: React's internal mutable node per mounted component, holding state, effect lists, and links into the tree — it persists across renders and is where hooks actually live. DOM node: the browser object the renderer creates/patches during commit. Renders create elements; reconciliation matches them to fibers; commit syncs fibers to DOM.

Senior: Why does React insist component functions be pure, and how does it enforce the habit?

Because React reserves the right to call them at times you don't control: re-renders from parents, Strict Mode's deliberate double-invocation in development, and concurrent rendering where a render may start, pause, throw away, and restart. Any side effect in the function body (mutation, subscriptions, logging counters) runs an unpredictable number of times. Side effects belong in event handlers and useEffect; render must be a pure props/state → elements mapping.

🎯 Key Takeaways

  • React's model: components are pure functions from props/state to element trees; data changes trigger re-description, and React reconciles the DOM — you never mutate the UI directly.
  • JSX compiles to _jsx(type, props) calls returning cheap immutable element objects; lowercase = host tag string, Capitalized = your function's identity.
  • Component ≠ element ≠ fiber ≠ DOM node: recipe → per-render description → React's stateful bookkeeping → the real browser object.
  • JSX rules with teeth: one root (Fragments), className/htmlFor, expressions-only in braces, style objects, self-closing tags, and 0 && renders "0".
  • Decompose by responsibility and repetition, not size; always at module scope — inline component definitions remount on every render.
  • Render (calling your functions) is cheap and must be pure; commit (DOM mutation) is batched by React — this split is the foundation for everything in the Performance section.
