Components & JSX Basics

React's one big idea: UI = f(state). You never manipulate the DOM ("add a row, toggle a class"); you describe what the UI looks like for the current data, and React reconciles the real DOM to match. Components are the functions; JSX is the description syntax. Prerequisite: the JavaScript track through Modules — React is 90% JavaScript fluency.

Your First Component

function UserCard({ name, role }) {                 // a component = a function returning JSX
  return (
    <article className="card">
      <h3>{name}</h3>                                {/* any JS expression in braces */}
      <p>{role.toUpperCase()}</p>
    </article>
  );
}
<UserCard name="Asha" role="admin" />               // used like an element — MUST be Capitalized
// lowercase = HTML tag, Capitalized = your component. That's how JSX tells them apart.

JSX — What It Actually Is

// JSX is syntax sugar the build step compiles to function calls:
<h1 className="title">Hi {name}</h1>
// ↓ compiles to ↓
jsx("h1", { className: "title", children: ["Hi ", name] })
// So: JSX values are OBJECTS ("React elements") — cheap descriptions, not DOM nodes.
// That's why you can put them in variables, arrays, props, and return them.
JSX ruleWhy
className, htmlFor, camelCase attrs (onClick, tabIndex)It compiles to JS — class/for are reserved words; props are JS object keys
One root element per return (or a Fragment <>…</>)A function returns ONE value; fragments group without adding DOM nodes
{expression} only — no statementsif/for don't produce values; use ternaries, &&, map (rendering lesson)
style={{ color: "red" }}Inner braces = a JS object; CSS properties camelCase (fontSize)
Text is auto-escaped{userInput} is XSS-safe by default; the escape hatch is literally named dangerouslySetInnerHTML
Self-close void elements: <img />, <br />JSX is stricter than HTML's forgiving parser

Composition — Components All the Way Down

function App() {
  return (
    <Layout>
      <Sidebar />
      <UserList>
        {users.map(u => <UserCard key={u.id} {...u} />)}
      </UserList>
    </Layout>
  );
}
function Layout({ children }) {                     // children: whatever was nested inside
  return <div className="layout">{children}</div>;
}
// children is just a prop — the React version of slots. Composition (nesting components,
// passing JSX as props) replaces inheritance entirely: there is no `extends` in idiomatic React.

Rendering Model — What Happens on Update

state changes → React CALLS your component functions again (a "render")
             → gets a new element tree (cheap objects)
             → DIFFS it against the previous tree (reconciliation)
             → commits the MINIMAL set of real DOM changes

Two consequences you must internalize:
1. Your component function runs MANY times — it must be PURE during render:
   no mutations of outside state, no API calls, no Math.random-into-markup surprises.
   (Side effects get their own lesson: useEffect.)
2. "Re-render" ≠ "re-create the DOM" — it's a diff; usually only a text node changes.

StrictMode (development) double-invokes renders precisely to expose impure render functions — if double-running your component breaks it, the component was already broken.

Interview Questions

Q1: What is a React element vs a component vs the DOM?

Component = the function (blueprint). Element = the object a JSX expression produces ({type, props, children}) — an immutable description. DOM = what React's renderer builds/patches to match the element tree. Rendering = calling components to get elements; committing = touching the DOM. Cheap descriptions + diffing is why declarative UI is affordable.

Q2: Why must render be pure, and how does React enforce it?

React may call your function any number of times (StrictMode ×2, concurrent rendering may discard renders) — impure renders (mutations, API calls, subscriptions) then run unpredictably often. Purity makes renders replayable/interruptible. Enforcement is by convention + StrictMode's double render exposing violations in dev.

Q3: Why doesn't React use inheritance for component reuse?

Composition covers every case with plain functions: wrap (Layout children), configure (props), inject content (JSX-as-props), share logic (custom hooks). Inheritance couples via base-class internals — the exact fragility the prototypes lesson warned about. React's docs literally say: no reason to use inheritance.

🏋️ Practice: scaffold Vite+React, build UserCard, a Layout with children, and compose an App rendering three cards from an array. Break it on purpose: lowercase the component, define a component inside App — observe both failure modes.

🎯 Key Takeaways

  • UI = f(state): describe, don't manipulate; React diffs element trees and patches the DOM minimally.
  • JSX compiles to function calls — hence className, {expressions}, one root, capitalized components.
  • Elements are cheap immutable objects; components are functions that must stay pure during render.
  • children + props-as-JSX = composition; there is no inheritance in idiomatic React.
  • Never define components inside components; StrictMode double-render is a purity test, not a bug.
