The Re-render Model & memo Performance

What you'll master: a disciplined performance workflow — measure, diagnose, apply the cheapest fix — the structural fixes that beat memoization (state placement, composition), how React.memo actually works and every way it silently breaks, and list virtualization. Prerequisites: The Render Cycle (this lesson operationalizes it), Props (reference identity).

Part 1 — The Workflow: Measure → Diagnose → Cheapest Fix

// Performance work without a profile is guessing — and React guesses badly:
// renders that look scary are often free (no commit), and the real cost hides
// in one wide component. The workflow:
//
// 1. MEASURE: React DevTools Profiler → record the janky interaction.
//    Read: which commits are slow (>16ms budget for 60fps)? Which components
//    are wide (expensive) vs frequent (rendering too often)?
// 2. DIAGNOSE with "why did this render?": parent render / props changed
//    (which one?) / hooks changed / context.
// 3. FIX, cheapest first:
//    a. Move state DOWN  (shrink the blast radius)         — structural, free
//    b. Move content UP  (children-as-props)               — structural, free
//    c. Virtualize long lists                              — structural
//    d. React.memo + stable props                          — maintenance cost
//    e. useMemo/useCallback for identity/compute            — next lesson
// Structural fixes can't rot. Memoization is a contract every future edit can
// silently void — it's the LAST tool, not the first.
The two distinct diseases: renders too often (cheap component, storming frequency — fix identity/placement) vs render too expensive (fine frequency, wide flame bar — fix the component: virtualize, split, memoize the computation). Naming which one you have is half the diagnosis; memo only treats the first.

Part 2 — Structural Fix A: Move State Down

// SICK: typing re-renders the entire page — the query lives at the top:
function Page() {
  const [query, setQuery] = useState("");
  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      <ExpensiveHeader />      {/* re-renders per keystroke */}
      <HugeProductGrid />      {/* re-renders per keystroke */}
      <Footer />               {/* re-renders per keystroke */}
    </>
  );
}

// CURED: state colocated with its only consumers — nothing else can re-render:
function Page() {
  return (
    <>
      <SearchBox />            {/* query + input + suggestions live INSIDE */}
      <ExpensiveHeader />
      <HugeProductGrid />
      <Footer />
    </>
  );
}
// Renders flow DOWN from the state owner (Render Cycle) — so the single most
// effective optimization in React is owning state as low as possible.
// This is the same colocation rule from State Architecture, now as a perf tool.

Part 3 — Structural Fix B: Move Content Up (children-as-props)

// When state MUST live high (a scroll listener, a theme), pass the expensive
// content THROUGH the stateful component instead of creating it inside:
function ScrollTracker({ children }) {
  const [y, setY] = useState(0);
  useEffect(() => { /* scroll listener → setY */ }, []);
  return <div className={y > 100 ? "scrolled" : ""}>{children}</div>;
}

<ScrollTracker>
  <HugeProductGrid />      {/* element created by the PARENT — reference-stable */}
</ScrollTracker>
// ScrollTracker re-renders on every scroll tick, but children is the SAME element
// object each time → React bails out of the entire grid subtree. Structural
// memoization, zero memo() (mechanism explained in Render Cycle Part 2).

Part 4 — React.memo: the Contract and Its Failure Modes

const ProductRow = React.memo(function ProductRow({ product, onAdd }) {
  return <li>{product.name} <button onClick={() => onAdd(product.id)}>+</button></li>;
});
// Contract: when the parent re-renders, compare new props to old with Object.is,
// key by key (shallow). All equal → SKIP re-rendering this subtree.
//
// memo is defeated — silently, no warning — by ANY fresh-per-render prop:
<ProductRow product={{ ...p }} />                 // ❌ fresh object literal
<ProductRow onAdd={id => add(id)} />              // ❌ fresh arrow function
<ProductRow style={{ margin: 4 }} />              // ❌ fresh style object
<ProductRow><Icon /></ProductRow>                 // ❌ children = fresh element
// …and by context: a consumer inside re-renders on context change regardless.
//
// To make memo REAL, every prop must be stable:
const onAdd = useCallback(id => dispatch({ type: "added", id }), []);   // stable fn
<ProductRow product={p} onAdd={onAdd} />          // p from state — stable until changed
// …or pass dispatch itself (stable forever, Reducers lesson), or primitives.
//
// Custom comparator — rare, sharp knife:
React.memo(Row, (prev, next) => prev.item.id === next.item.id
                             && prev.item.version === next.item.version);
// Return true = SKIP render. Get it wrong (too eager) and the UI goes stale —
// the worst kind of bug. Prefer fixing prop identity over custom equality.
Where memo earns its keep: list rows under a hot parent (100 rows, one changes — 99 skips), expensive leaves below frequently-updating ancestors, and design-system components used everywhere. Where it's noise: components that re-render with their props actually changing (comparison cost + code weight, zero skips), and cheap components (a memo check can cost as much as re-rendering a <li>). The React Compiler (auto-memoizing builds) is making manual memo increasingly legacy — but you'll maintain pre-compiler code for years, and the failure modes above are exactly what it automates away.

Part 5 — Build It: Fixing a Janky 5,000-Row Table

// Symptom: checkbox toggle in one row → 300ms freeze. Profile says: ALL rows
// re-render (parent state changed), each ~0.06ms × 5000 = the freeze.
//
// Step 1 — stabilize rows (memo + stable callback + primitives):
const Row = React.memo(function Row({ item, selected, onToggle }) {
  return (
    <tr className={selected ? "sel" : ""}>
      <td><input type="checkbox" checked={selected}
                 onChange={() => onToggle(item.id)} /></td>
      <td>{item.name}</td><td>{item.price}</td>
    </tr>
  );
});

function Table({ items }) {
  const [selected, setSelected] = useState(() => new Set());
  const onToggle = useCallback(id => setSelected(s => {
    const n = new Set(s); n.has(id) ? n.delete(id) : n.add(id); return n;
  }), []);                                         // stable: functional update, no deps

  return <table><tbody>
    {items.map(item => (
      <Row key={item.id} item={item}
           selected={selected.has(item.id)}        // PRIMITIVE boolean per row —
           onToggle={onToggle} />                  // not the whole Set (whose identity
    ))}                                            // changes every toggle)
  </tbody></table>;
}
// Toggle now re-renders exactly 2 rows (the one checked, and none other —
// selected flips only for it). 5000 → 2. But initial render still mounts 5000 rows…

// Step 2 — virtualize: render only what's visible:
import { useVirtualizer } from "@tanstack/react-virtual";
function VirtualTable({ items }) {
  const parentRef = useRef(null);
  const v = useVirtualizer({
    count: items.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 36,                        // row height
    overscan: 10,                                  // buffer above/below viewport
  });
  return (
    <div ref={parentRef} style={{ height: 600, overflow: "auto" }}>
      <div style={{ height: v.getTotalSize(), position: "relative" }}>
        {v.getVirtualItems().map(vi => (
          <div key={items[vi.index].id} style={{ position: "absolute",
               top: 0, transform: `translateY(${vi.start}px)`, width: "100%" }}>
            <Row item={items[vi.index]} … />
          </div>
        ))}
      </div>
    </div>
  );
}
// DOM nodes: 5000 → ~25. Mount, scroll, and update costs all collapse. Above a
// few hundred rows, virtualization IS the fix — memo just polishes it.

Interview Questions

Beginner: What does React.memo do, exactly?

Wraps a component so that when its parent re-renders, React shallow-compares the new props against the previous ones (Object.is per key) and skips re-rendering the subtree if all are equal. It doesn't block re-renders from the component's own state, its context subscriptions, or actual prop changes — it only filters parent-driven renders.

Beginner: Why is "move state down" a performance fix?

Re-renders start at the state owner and flow down its subtree. State owned by the page re-renders the page per change; the same state owned by a leaf re-renders one leaf. Colocating state shrinks the render blast radius structurally — no memoization contract to maintain, nothing to rot.

Mid: A memoized component still re-renders every time. List the causes in the order you'd check.

1) A fresh-per-render prop: inline object/array/function, or JSX children (compare with "why did this render" → which prop). 2) Context it consumes changed — memo doesn't filter subscriptions. 3) Its own state/hook changed. 4) A custom comparator returning false-negatives. 5) The parent passes a "stable" value that's actually rebuilt (a useMemo with churning deps). The fix follows the cause — usually stabilizing the named prop.

Mid: When is virtualization the right call, and what does it trade?

When a list's DOM size — not React's render count — is the cost: hundreds+ of rows, slow mounts, scroll jank, high memory. It renders only the viewport window (plus overscan) against a sized spacer. Trades: absolute positioning constraints, in-page find/ctrl-F breaks for unrendered rows, accessibility needs care (aria-rowcount), dynamic row heights add complexity, and SEO content isn't in the DOM. Below ~100 cheap rows it's not worth those costs.

Senior: Your profile shows a 45ms commit every keystroke in a form beside a chart. Walk your fix sequence.

Diagnose first: "why did this render" on the chart — parent render, caused by form state living in the shared parent. Fix 1 (structural): move form state into the form component; if the chart needs the submitted values only, keystroke churn never touches it. If the chart consumes live values, Fix 2: memo the chart with stable props and pass a debounced/deferred value (useDeferredValue — Concurrent lesson) so keystrokes stay urgent and chart updates lag pleasantly. Only if the chart's own 45ms render is irreducible do I look inside it — memoize its data transforms, or move rendering off the React path (canvas draws in an effect). Re-profile after each step; stop at the first that hits budget.

Senior: Argue for and against adopting React.memo as a team-wide default, and land a policy.

For: hot paths get protected by default; juniors can't forget it where it matters. Against: most components never need it (comparison cost, code noise); it silently dies under one inline prop — a default breeds false confidence; it hides structural problems (state placement) behind a patch; and the React Compiler is automating exactly this. Policy that survives review: structural fixes first (state placement, composition); memo only at profile-named boundaries, with stable-prop discipline enforced in review for those files; a comment on each memo naming the interaction it protects, so future editors know what they'd regress; adopt the compiler when the toolchain allows and delete the manual layer.

🏋️ Graded Exercises

1 (easy): A page's <Clock /> (ticks every second, state inside) sits beside <HeavyReport />. Does the report re-render each second? Why?
Solution 
// No. Clock owns the ticking state; renders flow down from the OWNER only.
// Siblings and ancestors are untouched. This is colocation already working —
// the page would only break if someone lifted the time to the parent.
2 (easy): Explain why this memo does nothing: const Card = React.memo(Card); <Card meta={{ tags }} />
Solution 
// {{ tags }} constructs a new object every parent render; shallow comparison
// finds meta !== prevMeta each time → never skips.
// Fix: pass tags directly () — a state-held array is
// reference-stable until it actually changes.
3 (medium): Using structural fixes only (no memo), stop <ExpensiveTree /> re-rendering when the theme button toggles: function App(){ const [dark,setDark]=useState(false); return <div className={dark?"dark":""}><button onClick={()=>setDark(d=>!d)}/><ExpensiveTree/></div>; }
Solution 
function ThemeShell({ children }) {
  const [dark, setDark] = useState(false);
  return (
    <div className={dark ? "dark" : ""}>
      <button onClick={() => setDark(d => !d)} />
      {children}
    </div>
  );
}
function App() {
  return <ThemeShell><ExpensiveTree /></ThemeShell>;
}
// State moved down into ThemeShell; the tree arrives as a reference-stable
// children element → React reuses it wholesale on every toggle.
4 (medium): The Part 5 table: a teammate replaces selected={selected.has(item.id)} with selectedSet={selected} "so rows can check membership themselves." Predict the profiler result and explain.
Solution 
// Every toggle creates a NEW Set (immutable update) → selectedSet's identity
// changes for ALL 5000 rows → every memo comparison fails → full re-render storm
// returns. The boolean derivation in the parent was the load-bearing design:
// it converts one changing reference into 4998 unchanged primitives + 2 changed
// ones. Rule: derive the NARROWEST prop each child needs at the boundary.
5 (medium): Your dashboard renders 40 StatCards; profiling a websocket price tick shows all 40 re-render, though each card shows one symbol. The parent holds prices as an object in state. Propose two fixes and rank them.
Solution 
// Cause: one prices object in parent state — every tick replaces it, parent
// re-renders, all cards re-render.
// Fix A (best, structural): per-symbol subscription — an external store
// (External Stores lesson): each StatCard does usePriceStore(s => s.prices[symbol]).
// A tick re-renders only the card whose selected value changed. Parent renders never.
// Fix B (patch): memo(StatCard) + pass price={prices[sym]} (primitive) — cards
// with unchanged prices skip. Parent still re-renders per tick and diffs 40 props.
// A removes the storm at the source and scales to 400 cards; B is a band-aid
// that keeps the hot parent. Rank: A, then B only if a store is off the table.
6 (hard): Write a useRenderCount(label) dev hook that logs how many times a component rendered and why that's a ref not state, then use it to verify the Part 5 fix (2 rows re-render, not 5000).
Solution 
function useRenderCount(label) {
  const n = useRef(0);
  n.current += 1;                              // render-time write is fine here: dev-only
  useEffect(() => {                            // log after commit, once per render
    console.log(`[render] ${label}: ${n.current}`);
  });
  return n.current;
}
// Ref, not state: counting renders WITH state would schedule another render per
// render — an infinite loop by construction. The count is bookkeeping (Refs lesson).
// Verification: add useRenderCount(`row-${item.id}`) in Row, toggle one checkbox,
// and count the log lines for that commit — exactly the toggled row (and none
// other) should log. In the Profiler the same shows as 1 colored bar, 4999 gray.
// ponytail: console-count harness; React DevTools Profiler is the real tool —
// this hook is for quick assertions in reviews and tests.
7 (hard): An infinite-scroll feed (virtualized, memoized rows) still jank-spikes every time new items append: profile shows the parent feed re-rendering and all visible rows' items prop changing. The append code is setItems([...items, ...page]). Explain why memo can't help the visible rows here, and design the fix.
Solution 
// The rows' props are item OBJECTS drawn from a rebuilt array — but spreading
// preserves the OLD item references, so item={items[i]} is stable… the actual
// breakage is elsewhere: rows also receive props derived per render, e.g.
// onVisible={() => track(item.id)} or index-based data — fresh identities for
// every visible row on every append.
// Fix design:
// 1. Keep item references stable across appends (spread already does; never map
//    to new objects on ingest without need).
// 2. Stabilize callbacks: one useCallback((id) => …) taking the id, not a
//    closure per row.
// 3. Don't pass index-derived props that shift on append (index itself!) —
//    virtualizers hand rows their own start offset; key by item.id so appends
//    don't re-key existing rows.
// 4. If the feed parent's own render is the spike (building 10k-element
//    children), let the virtualizer's getVirtualItems() keep the map small —
//    the parent maps ~25 visible items, not the full list.
// Then re-profile: append should re-render the parent (cheap, 25 rows mapped)
// and mount only the NEW rows; existing visible rows all skip via memo.

🎯 Key Takeaways

  • Workflow: profile the interaction, name the disease (too often vs too expensive), apply the cheapest durable fix, re-measure. Never optimize unprofiled.
  • Structural fixes first: state down (shrink the owner's subtree), content up (reference-stable children) — they can't rot and cost nothing to maintain.
  • React.memo = shallow prop comparison; one inline object/function/children prop kills it silently. Stabilize every prop or don't bother.
  • Derive the narrowest prop per child (booleans from Sets) — one changing reference becomes N-2 unchanged primitives.
  • Above a few hundred rows, DOM size is the cost: virtualize; memo is not a DOM strategy.
  • Profile production builds on throttled CPUs; budget 16ms/frame, ~100ms/interaction; the React Compiler is automating manual memo — write code it can verify.
