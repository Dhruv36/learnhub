Performance: memo, useMemo, useCallback Advanced

First truth: re-renders are usually fine — rendering produces cheap objects; only the DOM diff is committed. React performance work is about the exceptions: big subtrees re-rendering for unrelated changes, expensive computations re-running, and long lists. The tools are three memo APIs + structure — and the discipline is profile first.

Why Components Re-render (know the causes cold)

1. Their STATE changed
2. Their PARENT re-rendered  ← the big one: children render by default, props or not
3. Their CONTEXT value changed
// Props "changing" is NOT a cause by itself — parent rendering is. memo() is what
// makes props the deciding factor.

The Three Memo Tools

// React.memo — skip re-render if props are shallow-equal (Object.is per prop):
const UserRow = memo(function UserRow({ user, onSelect }) { … });

// useMemo — cache a VALUE between renders (recompute when deps change):
const sorted = useMemo(() => [...items].sort(byScore), [items]);

// useCallback — cache a FUNCTION identity (it's useMemo for functions):
const onSelect = useCallback((id) => setSelected(id), []);
// The trio works TOGETHER: memo is defeated by unstable props —
// inline objects ({x:1}), inline arrays, and inline functions are NEW references
// every render, so memo compares unequal and renders anyway. useMemo/useCallback
// exist largely to make memo actually work.
ToolUse whenDon't
memo()Pure display components receiving stable props, in hot paths (rows, cells)Wrap everything "just in case" — comparison itself costs
useMemoProvably expensive computes (sort/filter 10k, chart data); stabilizing object deps/propsWrap trivial expressions — overhead > savings
useCallbackFunctions passed to memo children or used in effect depsEvery handler by reflex

Structure Beats Memo — the Two Free Patterns

// 1. PUSH STATE DOWN — typing re-renders only the search box, not the page:
function Page() {                            // ❌ query state here re-renders everything
  return <><SearchBox /><HugeTable /></>;    // ✅ state lives INSIDE SearchBox
}

// 2. LIFT CONTENT UP (children as props) — children created by the PARENT don't
//    re-render when the wrapper's state changes:
function ThemeWrapper({ children }) {
  const [theme, setTheme] = useState("light");     // toggling theme re-renders wrapper…
  return <div className={theme}>{children}</div>;  // …but children (same element refs) are skipped
}
<ThemeWrapper><HugeTable /></ThemeWrapper>
// These cost nothing and can't go stale — reach for them BEFORE the memo trio.

Long Lists — Virtualize

// 10k rows = 10k DOM nodes = slow regardless of memo. Render the visible window (~30):
import { useVirtualizer } from "@tanstack/react-virtual";
// constant DOM size, absolute-positioned rows in a spacer — the JS perf lesson's
// virtualization, packaged for React. Anything > ~200 rows in a scroll container qualifies.

Concurrent Tools — Prioritizing Updates

// useTransition: mark expensive updates as interruptible/low-priority:
const [isPending, startTransition] = useTransition();
setQuery(value);                                   // urgent: the input echoes instantly
startTransition(() => setFilter(value));           // non-urgent: the 10k-row filter can lag/interrupt

// useDeferredValue: the value-flavored version:
const deferredQuery = useDeferredValue(query);
const results = useMemo(() => filter(items, deferredQuery), [items, deferredQuery]);
// Typing stays 60fps; results catch up. This is React's time-slicing surfaced as API.

The Workflow — Profile, Don't Guess

React DevTools → Profiler tab → record the slow interaction:
· flame graph = who rendered, how long, WHY ("props changed: onSelect") — the exact
  unstable-prop culprit, named
· "Highlight updates" checkbox = visual re-render storms
Then: fix structure → memo the hot path → verify with a second recording.
React Compiler (auto-memoization) is rolling out — it makes manual memo/useCallback
largely unnecessary where adopted, but the mental model above still explains WHAT it does.

Interview Questions

Q1: Why does React.memo frequently "not work", mechanically?

memo does Object.is per prop. JSX like onClick={() => …} or style={{…}} creates new references every parent render, so comparison fails and the child renders anyway. Fix: stabilize with useCallback/useMemo, restructure so props are primitives, or drop the memo. Understanding this = understanding reference semantics (the arrays lesson) applied to React.

Q2: A search box re-renders a 5k-row table per keystroke. Give the fix ladder.

① Structure: push query state into the SearchBox; table receives committed/submitted value. ② If live filtering is required: useDeferredValue/useTransition so typing stays urgent and filtering lags gracefully. ③ useMemo the filtered array; memo the Row. ④ Virtualize — 5k DOM rows is the real crime. Profile between steps; often ① alone ends it.

Q3: What do useTransition/useDeferredValue actually do that setTimeout can't?

They mark updates as low-priority INSIDE React's scheduler: concurrent rendering can start the expensive render, INTERRUPT it when urgent input arrives, and discard/restart — keeping the input responsive without manual debouncing or stale intermediate paints. setTimeout just delays; it can't interrupt an in-progress render or coordinate with React's prioritization.

🏋️ Practice: build the keystroke-renders-table demo; fix it four ways (state-down, children-as-props, memo+useCallback, useDeferredValue) profiling each with React DevTools; then virtualize a 10k list and compare DOM node counts.

🎯 Key Takeaways

  • Re-renders are usually cheap; parents render children by default — memo makes props the gate.
  • memo is defeated by unstable references; useMemo/useCallback exist to stabilize them (and cache real work).
  • Structure first (state down, children up) — free, stale-proof, usually sufficient.
  • Virtualize long lists; transitions/deferred values keep typing urgent while big renders lag.
  • Profile → fix → re-profile; the React Compiler is automating the trio, not the model.
