Props & State Basics

Two kinds of data drive every React component: props (inputs from the parent — read-only) and state (the component's own memory — changing it triggers re-render). Every React bug report ever filed traces to misunderstanding one of five rules on this page.

Props — Read-Only Inputs

function Badge({ label, tone = "info", onDismiss }) {   // destructure + defaults
  return (
    <span className={`badge badge--${tone}`}>
      {label}
      {onDismiss && <button onClick={onDismiss}>✕</button>}   {/* callbacks are props too */}
    </span>
  );
}
// Props flow DOWN; events flow UP via callback props. That one-way loop IS React's data flow.
// props are READ-ONLY: mutating them breaks the f(state) contract and purity.

State — useState and the Five Rules

function Counter() {
  const [count, setCount] = useState(0);        // [current, setter]; arg = initial value
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
RuleThe bug it prevents
1. Setting state requests a re-render; count is a snapshot — it does NOT change mid-functionsetCount(count+1); console.log(count) logs the OLD value — not a bug, the model
2. Updates are batched: three set-calls in one handler = one re-rendersetCount(count+1) ×3 = +1 total (same snapshot thrice!)
3. Use the functional form when new state depends on oldsetCount(c => c + 1) ×3 = +3 ✅ — also immune to stale closures
4. Never mutate — replace: state changes are detected by referenceitems.push(x) = invisible; setItems([...items, x]) = render
5. Expensive initial value? Pass a functionuseState(() => readLocalStorage()) runs once, not every render

Immutable Update Patterns (the arrays lesson, applied)

setUser({ ...user, name: "Asha" });                          // object field
setItems([...items, newItem]);                                // add
setItems(items.filter(i => i.id !== id));                     // remove
setItems(items.map(i => i.id === id ? { ...i, done: true } : i));   // update one
setUser({ ...user, address: { ...user.address, city } });     // nested = spread each level

Where State Lives — Lifting & Deriving

// Siblings need shared data? State lives in their CLOSEST COMMON PARENT:
function Filters() {
  const [query, setQuery] = useState("");
  return <>
    <SearchBox value={query} onChange={setQuery} />      {/* down as props */}
    <ResultList query={query} />                          {/* both see one truth */}
  </>;
}

// DON'T store what you can COMPUTE — derive during render:
const visible = items.filter(i => i.text.includes(query));    // ✅ derived
// const [visible, setVisible] = useState(...)               // ❌ second copy → sync bugs
// Litmus: "could this get out of sync with its source?" → derive it.

Interview Questions

Q1: A handler calls setCount(count+1) three times — what renders and why?

+1. All three read the same snapshot (say 0) and enqueue "set to 1"; batching renders once. Functional updates (c => c+1) enqueue transformations that compose: +3. Explaining snapshot-per-render + batching + updater queue is the complete answer.

Q2: Why does React require immutability when JavaScript doesn't?

Change detection is reference equality (Object.is) — O(1), no deep compares. Mutation keeps the reference, so React (and memo/effect deps) sees "unchanged". Immutability also preserves previous trees for concurrent rendering and time-slicing. It's the arrays-lesson reference semantics as an architectural contract.

Q3: Where should state live for a filterable product table (search box, category dropdown, table, result count)?

query + category in the closest common parent; the table receives filtered rows or the filters; result count is DERIVED (filtered.length — never stored). If distant components need it, that's the Context/State-management lesson. Phrase that scores: single source of truth, lift to the lowest common owner, derive everything derivable.

🏋️ Practice: build the Counter with a +3 button (prove functional updates); a TodoList with add/toggle/remove using the four immutable patterns; then a two-sibling search (lift state) with a derived match count.

🎯 Key Takeaways

  • Props down (read-only), events up (callback props) — the one-way loop.
  • State is a per-render snapshot; sets are batched requests; functional form when depending on previous.
  • Replace, never mutate — reference equality is the change-detection contract.
  • Lift shared state to the closest common parent; derive computed data in render; never duplicate truth.
  • Lazy initializer for expensive initial state; controlled inputs make React state the source of truth.
