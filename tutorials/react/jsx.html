JSX Deep Dive Foundations

What you'll master: everything JSX can express — children of every kind, spread props, conditional and dynamic rendering idioms, Fragments with keys, how whitespace and escaping behave, and the patterns (element-as-prop, dynamic tag types) that make component APIs flexible. Prerequisites: Components & JSX — you know JSX compiles to element objects; now we use that fact deliberately.

Part 1 — Children: the Prop With Special Syntax

Whatever sits between a tag's open and close becomes props.children. It can be text, elements, an array, an expression — any JavaScript value:

function Card({ title, children }) {          // children is just a prop
  return (
    <section className="card">
      <h3>{title}</h3>
      {children}                              {/* render whatever the caller nested */}
    </section>
  );
}

<Card title="Billing">
  <p>Next invoice: March 1.</p>            {/* ← this subtree is props.children */}
  <button>Update card</button>
</Card>

This is React's composition primitive — the equivalent of a slot. A component that renders children doesn't need to know what's inside it, which keeps layout components (cards, modals, layouts) generic. The full pattern language is in Composition Patterns.

// What children actually IS varies — robust components use React.Children or just render it:
<Card />                       // children === undefined
<Card>hi</Card>                // children === "hi"                (single string)
<Card><X /></Card>             // children === element object       (single child, no array!)
<Card><X /><Y /></Card>        // children === [element, element]   (array only when multiple)

Part 2 — Props Syntax, Fully

<Widget
  title="Plain string"                 // string literal
  count={42}                           // any expression needs braces (numbers too!)
  user={{ name: "Ada" }}               // object literal: outer braces JSX, inner braces object
  onClick={handleClick}                // pass the FUNCTION, don't call it
  disabled                             // bare prop = {true}
  data-testid="widget-1"               // data-* / aria-* pass through as-is (keep the dashes)
  aria-label="Main widget"
/>

// Spread: forward every key of an object as individual props
const inputProps = { type: "email", required: true, placeholder: "you@example.com" };
<input {...inputProps} />

// Order matters — later wins. The wrapper-component idiom:
function TextInput(props) {
  return <input type="text" {...props} className={`field ${props.className ?? ""}`} />;
  // type BEFORE the spread = a default the caller can override;
  // className AFTER = the wrapper always gets the last word (here, merging instead).
}
Spread is a boundary tool, not a default. {...props} shines in thin wrappers around host elements (design-system buttons/inputs) where forwarding every HTML attribute is the point. Between your own components, explicit props keep the data flow greppable — spreading everything everywhere is how unknown-attribute warnings and accidental prop leaks happen.

Part 3 — Conditional Rendering: the Complete Idiom Set

function Inbox({ user, messages, error, loading }) {
  // 1. Early return — best for whole-component branches (guards):
  if (error)   return <ErrorPanel error={error} />;
  if (loading) return <Spinner />;

  return (
    <div>
      {/* 2. Ternary — two branches inline: */}
      {user ? <b>{user.name}</b> : <a href="/login">Sign in</a>}

      {/* 3. && — render-or-nothing (guard against 0!): */}
      {messages.length > 0 && <Badge count={messages.length} />}

      {/* 4. Variable — when logic is too big for inline: */}
      {renderRows(messages)}
    </div>
  );
}

// 5. Lookup map — cleaner than nested ternaries for 3+ states:
const STATUS_VIEW = {
  idle:    <p>Ready.</p>,
  running: <Spinner />,
  done:    <p>✅ Complete</p>,
  failed:  <p>❌ Failed — retry?</p>,
};
function Status({ state }) { return STATUS_VIEW[state] ?? null; }

Choosing: guards for "the whole view is different", ternary for exactly two inline branches, && for optional extras, a lookup object once you'd nest ternaries. Nested ternaries in JSX are the single most-flagged readability smell in React code review.

Part 4 — Fragments, Keys on Fragments & Returning Arrays

// Long form  is required when a fragment needs a KEY —
// e.g. a definition list where each item is a dt+dd PAIR:
import { Fragment } from "react";

function Glossary({ terms }) {
  return (
    <dl>
      {terms.map(t => (
        <Fragment key={t.id}>          {/* <> can't take props; Fragment can take key */}
          <dt>{t.word}</dt>
          <dd>{t.meaning}</dd>
        </Fragment>
      ))}
    </dl>
  );
}
// Why it matters: 
 only allows dt/dd children — a wrapper 
 per pair would be
// invalid HTML. Fragments group for React without emitting DOM.

Part 5 — Dynamic Types & Elements as Values

// 1. The tag itself can be a variable — it just becomes the `type` field:
function Heading({ level = 2, children }) {
  const Tag = `h${level}`;                   // must be Capitalized VARIABLE to be dynamic
  return <Tag>{children}</Tag>;
}
<Heading level={1}>Title</Heading>           // renders 


// 2. Polymorphic "as" prop — the design-system staple:
function Button({ as: Component = "button", ...props }) {
  return <Component className="btn" {...props} />;
}
<Button>Save</Button>                        //
Element-as-prop vs component-as-prop: pass an element (left={<Logo size=20 />}) when the caller configures it; pass a component (icon={Logo}, rendered as <props.icon />) when the receiver must control the props (e.g. inject isActive). Both appear constantly in component-library APIs and interviews love the distinction.

Part 6 — Text, Whitespace & Escaping (the Gotcha Layer)

// 1. JSX collapses whitespace like HTML: newlines + indentation between tags vanish.
<b>Hello</b>
<i>world</i>          // renders "Helloworld" — add {" "} for an explicit space:
<b>Hello</b>{" "}<i>world</i>

// 2. Interpolated strings are AUTO-ESCAPED — this is React's XSS defense:
const evil = "<img src=x onerror=alert(1)>";
<p>{evil}</p>          // renders the literal text, not an image tag. Safe by default.

// 3. The escape hatch is named scary on purpose:
<div dangerouslySetInnerHTML={{ __html: sanitizedMarkdown }} />
// ONLY with sanitized content (DOMPurify) — full threat model in the JS Security lesson.

// 4. HTML entities work in text: &nbsp; &copy; &rarr; — or just paste the
//    real character; JSX files are UTF-8 JavaScript.

Interview Questions

Beginner: What is props.children and why is it useful?

It's the prop holding whatever was nested between a component's opening and closing tags — text, one element, or an array of them. It lets container components (cards, modals, layouts) render arbitrary caller-supplied content without knowing what it is: composition instead of configuration.

Beginner: Difference between onClick={fn} and onClick={fn()}?

The first passes the function for React to call on click. The second calls it immediately during render and passes its return value as the handler. If fn sets state, the second form triggers render → set state → render… an infinite loop. To pass arguments, wrap in an arrow: onClick={() => fn(id)}.

Mid: When do you need <Fragment key=…> instead of <>?

When mapping a list where each item produces multiple sibling elements that must be grouped without a DOM wrapper — each group needs a key, and the shorthand can't take props. Classic case: <dt>/<dd> pairs inside a <dl>, or table cells where an extra <div> would be invalid HTML.

Mid: How does prop spread order interact with defaults and overrides?

JSX props compile to an object built left-to-right, so later keys overwrite earlier ones. Props placed before {...rest} are defaults the caller can override; props after the spread are enforced by the component and can't be overridden. Wrappers exploit both: default type before, merged className after.

Senior: Element-as-prop vs component-as-prop vs render prop — when each?

Element-as-prop (right={<Icon size={20}/>}): caller fully configures the content; receiver just places it. Component-as-prop (icon={Icon}): receiver instantiates it, so the receiver can inject props the caller can't know (active state, sizes from context). Render prop (renderRow={(item) => …}): receiver must pass data back to the caller's markup — inversion in both directions. Choose the weakest tool that works: element → component → function.

Senior: Explain how React's JSX escaping prevents XSS, and its precise limits.

Interpolated values become text nodes (or attribute values) via the DOM API, never parsed as HTML, so <script> in data renders inert. Limits: dangerouslySetInnerHTML bypasses it entirely; URL-bearing props (href, src) accept javascript: URLs from data (modern React warns/blocks some cases, but validate protocols yourself); and escaping does nothing about markup you build server-side. Auto-escaping covers injection into content, not every sink.

🎯 Key Takeaways

  • children is a normal prop with nesting syntax: undefined, a single value, or an array — never assume the shape.
  • Prop syntax: braces for any expression, bare prop = true, data-*/aria-* pass through; spread order decides defaults (before) vs enforced values (after).
  • Conditional idioms in order of preference: early-return guards, one ternary, && with a real boolean, lookup maps — never nested ternaries.
  • Keyed <Fragment> groups multi-element list items without illegal wrapper DOM.
  • The tag is just a value: capitalized variables, as-prop polymorphism, and element/component-as-prop slots are how flexible component APIs are built.
  • Interpolation auto-escapes (XSS-safe by default); dangerouslySetInnerHTML and URL props are the exceptions you must guard.
