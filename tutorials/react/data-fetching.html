Data Fetching Effects & Lifecycle

What you'll master: the full lifecycle of server data in a client app — status modeling, race safety, a complete hand-rolled useFetch, why production teams use React Query/SWR (the actual problems those libraries solve: caching, deduping, invalidation, stale-while-revalidate), and mutation patterns including optimistic updates. Prerequisites: useEffect Mastery and JS Fetch & HTTP.

Part 1 — Server Data Is a Different Kind of State

The state lessons covered client state — values your app owns (form text, open panels). Server data is fundamentally different, and treating them the same causes most data bugs:

Client stateServer data
OwnerYour componentThe server — your copy is a cache
Can it be stale?Never — you're the sourceAlways — it aged the moment it arrived
Failure modesNoneLoading, errors, races, timeouts, retries
Shared across screens?RarelyConstantly (same user object everywhere)
Right tooluseState/useReducerA cache with lifecycle — hand-rolled or library
The reframe that organizes this whole lesson: you never "store server data in state" — you cache a snapshot of it. Caches need: a key (what request), status (fresh, loading, failed), invalidation (when to refetch), and dedup (two components, one request). Do it by hand to understand it; use a library to ship it.

Part 2 — Modeling the Lifecycle: One Status, Not Boolean Soup

// ❌ the boolean pile — allows impossible states (loading AND error AND data):
const [data, setData] = useState(null);
const [isLoading, setIsLoading] = useState(false);
const [isError, setIsError] = useState(false);

// ✅ a discriminated union — exactly one state at a time (State lesson, applied):
// { status: "idle" } | { status: "loading" }
// | { status: "success", data } | { status: "error", error }
const [state, setState] = useState({ status: "idle" });

// UI becomes a total function of status — every case handled, none impossible:
switch (state.status) {
  case "idle":    return <Prompt />;
  case "loading": return <Skeleton />;
  case "error":   return <Retry error={state.error} onRetry={refetch} />;
  case "success": return state.data.length === 0
                    ? <Empty />
                    : <List items={state.data} />;
}
// Note "success but empty" is a UI state too — designed, not an accident.

Part 3 — Build It: a Complete useFetch

function useFetch(url) {
  const [state, setState] = useState({ status: "loading" });
  const [attempt, setAttempt] = useState(0);              // bump to refetch

  useEffect(() => {
    const ctrl = new AbortController();
    setState({ status: "loading" });                      // reset on url change

    (async () => {
      try {
        const res = await fetch(url, { signal: ctrl.signal });
        if (!res.ok) throw new Error(`HTTP ${res.status}`);   // fetch doesn't throw on 4xx/5xx!
        const data = await res.json();
        setState({ status: "success", data });
      } catch (err) {
        if (err.name !== "AbortError")                    // aborted ≠ failed
          setState({ status: "error", error: err });
      }
    })();

    return () => ctrl.abort();                            // race safety + cancels network
  }, [url, attempt]);

  return { ...state, refetch: () => setAttempt(a => a + 1) };
}

// Usage:
function Profile({ userId }) {
  const user = useFetch(`/api/users/${userId}`);
  if (user.status === "loading") return <Skeleton />;
  if (user.status === "error")   return <Retry onRetry={user.refetch} />;
  return <h1>{user.data.name}</h1>;
}
// Everything from useEffect Mastery is load-bearing here: abort in cleanup kills
// races; status resets on url change; res.ok check because fetch only rejects on
// network failure; refetch = a dep bump, not a duplicated call path.

Part 4 — What the Hand-Rolled Version Can't Do (Why Libraries Exist)

// Mount  in the header AND the sidebar:
// → useFetch runs twice. Two identical requests. Two copies that can disagree.
//
// Navigate away and back:
// → loading spinner again, even though you had the data 3 seconds ago.
//
// User edits their name on another screen:
// → every cached copy elsewhere is silently wrong until its component remounts.
//
// These aren't missing features of our hook — they're a missing ARCHITECTURE:
// a shared, keyed, invalidatable cache that lives OUTSIDE the component tree.
// React Query (TanStack Query) — the de-facto standard shape:
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";

function Profile({ userId }) {
  const { data, status, error } = useQuery({
    queryKey: ["user", userId],                 // cache key — THE core concept
    queryFn: () => api.getUser(userId),
    staleTime: 30_000,                          // fresh for 30s: no refetch on remount
  });
  // …same status-based rendering as before
}
// What the key buys, automatically:
// • DEDUPING: 10 components, same key → one request, one shared copy.
// • CACHE-FIRST: remount → instant cached data, refetch in background if stale
//   ("stale-while-revalidate" — show something now, correct it quietly).
// • INVALIDATION: after a mutation, mark keys dirty → everything refetches:
const qc = useQueryClient();
const rename = useMutation({
  mutationFn: newName => api.renameUser(userId, newName),
  onSuccess: () => qc.invalidateQueries({ queryKey: ["user", userId] }),
});
// • Plus: retries with backoff, window-refocus refetch, pagination/infinite
//   scroll primitives, request cancellation — all the parts teams rebuild badly.
Interview positioning: "I can write the effect-based fetch with abort and status modeling — and in production I reach for React Query because server cache management (dedup, invalidation, staleness) is an app-level concern, not a component-level one." Knowing both layers, and why the boundary sits where it does, is the senior answer. Server Components move part of this to the server — later in the track.

Part 5 — Mutations & Optimistic Updates

// Pessimistic (default): wait for the server, then update. Simple, honest, slower UX:
async function toggleLike() {
  setBusy(true);
  try {
    const updated = await api.like(postId);
    setPost(updated);                    // server's word is final
  } finally { setBusy(false); }
}

// Optimistic: update the UI NOW, roll back if the server disagrees.
// For actions that almost never fail (likes, toggles, reorders):
async function toggleLike() {
  const before = post;
  setPost(p => ({ ...p, liked: !p.liked, likes: p.likes + (p.liked ? -1 : 1) }));
  try {
    await api.like(postId);
  } catch {
    setPost(before);                     // rollback to the snapshot
    toast("Couldn't save your like — try again.");
  }
}
// Rules of optimism:
// 1. Snapshot before you touch anything; rollback is the snapshot, not "reverse math"
//    (another update may have landed in between).
// 2. Tell the user when you roll back — silent reversion feels like a glitch.
// 3. Only for high-success, low-stakes actions. Payments are never optimistic.
// React Query packages this as onMutate (snapshot) / onError (rollback) / onSettled (refetch).

Interview Questions

Beginner: Walk through fetching data on mount, with loading and error states.

In a useEffect (empty deps if the URL is static): set status loading, fetch, check res.ok, parse JSON, set success with data, catch errors into an error status, and abort in cleanup. Render by branching on the single status value: skeleton, retry panel, empty state, or the data.

Beginner: Why doesn't fetch reject on a 500 response?

By spec, fetch rejects only on network-level failure (DNS, offline, CORS, abort). Any HTTP response — including 4xx/5xx — resolves successfully; the status lives on res.ok/res.status. Production code normalizes this once in a request layer that throws on !res.ok, so components treat HTTP errors and network errors uniformly.

Mid: What is stale-while-revalidate and why is it the default UX of modern data layers?

On re-request of a cached key, show the cached (possibly stale) data immediately and refetch in the background, swapping in fresh results when they land. Users see content in 0ms instead of a spinner, and correctness converges within a round-trip. It matches how server data actually works — always slightly stale anyway — and turns "loading" from the common case into the cold-start-only case.

Mid: How do query keys drive React Query's behavior?

The key (["user", userId]) identifies a cache entry: components with the same key share one request and one copy (dedup); the key's cache entry carries freshness (staleTime) deciding cache-vs-refetch; and invalidation targets keys (exact or by prefix — invalidate ["user"] to dirty every user) to trigger refetches after mutations. Keys are to server state what reducer actions are to client state: the addressing scheme.

Senior: Design optimistic updates for a kanban board's drag-and-drop, including failure handling.

On drop: snapshot the affected columns, apply the move locally (remove/insert with new order), fire the mutation with explicit positioning (cursor/order index, not "after visible item" — the server may see a different board). On error: restore the snapshot, toast with a retry action, and refetch the board to reconverge (another user may have moved cards meanwhile). On success: refetch or apply the server's canonical response, because concurrent moves mean your local guess and the server's truth can differ. Guard rails: disable optimistic mode when a previous mutation for the same card is in flight (queue or collapse them) to avoid interleaved rollbacks.

Senior: Your dashboard makes 14 requests on load, several serialized. Diagnose and fix at the architecture level.

Serialization usually means fetch-on-mount waterfalls: a parent awaits data before rendering children that fetch their own. Fixes in order: parallelize independent queries at a common ancestor or route loader (fire all before/at render, not in nested mounts); collapse chatty endpoints into a composed one (BFF pattern) when the client is stitching what the server could join; cache cross-screen shared resources so navigation doesn't refetch; and consider moving composition server-side (RSC/loaders) so the client receives one payload. Then verify in the network tab that the request graph is wide, not deep.

1 (easy): Render a user list from /api/users with loading, error+retry, and empty states, using the useFetch from Part 3.
Solution 
function Users() {
  const q = useFetch("/api/users");
  if (q.status === "loading") return <p>Loading…</p>;
  if (q.status === "error")   return <button onClick={q.refetch}>Failed — retry</button>;
  if (q.data.length === 0)    return <p>No users yet.</p>;
  return <ul>{q.data.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
2 (easy): Why is if (!res.ok) throw new Error(...) inside the try block essential? Trace what happens without it when the API returns 500 with an HTML error page.
Solution 
// Without it: fetch resolves fine (500 is a response!), res.json() then throws
// a SyntaxError parsing the HTML ("Unexpected token '<'") — you get a misleading
// parse error, or worse, a JSON error body flows into setState as "success" and
// components crash reading missing fields far from the fetch. The ok-check turns
// HTTP failure into a proper error state at the boundary.
3 (medium): Add a 150ms "no flash" rule to useFetch's consumers: skeleton only if loading exceeds 150ms.
Solution 
function useDelayedFlag(active, ms = 150) {
  const [show, setShow] = useState(false);
  useEffect(() => {
    if (!active) { setShow(false); return; }
    const id = setTimeout(() => setShow(true), ms);
    return () => clearTimeout(id);
  }, [active, ms]);
  return show;
}
// Usage:
const showSkeleton = useDelayedFlag(q.status === "loading");
if (q.status === "loading") return showSkeleton ? <Skeleton /> : null;
// Fast responses render nothing-then-content: calmer than a 40ms spinner blink.
4 (medium): Two sibling components each call useFetch("/api/config"). Without adding a library, share one request and one copy via context.
Solution 
const ConfigContext = createContext(null);

function ConfigProvider({ children }) {
  const config = useFetch("/api/config");     // ONE fetch, at the common ancestor
  return <ConfigContext.Provider value={config}>{children}</ConfigContext.Provider>;
}
const useConfig = () => useContext(ConfigContext);
// Siblings call useConfig() — same status object, zero duplicate requests.
// This is the primitive version of what a query cache does with keys, app-wide.
5 (medium): Implement search-as-you-type over /api/search?q= combining useDebouncedValue (from the effects lesson) with race-safe fetching. Which requests fire for the keystrokes "r","re","rea" typed within 200ms, with 300ms debounce?
Solution 
function Search() {
  const [input, setInput] = useState("");
  const q = useDebouncedValue(input, 300);
  const results = useFetch(`/api/search?q=${encodeURIComponent(q)}`);
  return (
    <>
      <input value={input} onChange={e => setInput(e.target.value)} />
      {results.status === "success" && <List items={results.data} />}
    </>
  );
}
// "r","re","rea" within 200ms: each keystroke cancels the previous debounce timer;
// only "rea" survives 300ms → exactly ONE request fires. The abort in useFetch
// additionally covers the case where a previous debounced value's request is
// still in flight when the next lands. Debounce reduces request count;
// abort guarantees ordering — you need both.
6 (hard): Build useOptimisticToggle(resource): returns [value, toggle] where toggle flips instantly, calls api.set(resource, next), rolls back on failure, and ignores rapid double-clicks while a request is in flight.
Solution 
function useOptimisticToggle(resource, initial) {
  const [value, setValue] = useState(initial);
  const inFlight = useRef(false);

  async function toggle() {
    if (inFlight.current) return;            // collapse rapid clicks
    inFlight.current = true;
    const before = value;
    const next = !value;
    setValue(next);                          // optimistic
    try {
      await api.set(resource, next);
    } catch {
      setValue(before);                      // snapshot rollback
      toast("Change didn't save.");
    } finally {
      inFlight.current = false;
    }
  }
  return [value, toggle];
}
// inFlight is a ref, not state: it gates logic without needing a render.
// Alternative to ignoring: queue the latest intent and send it on settle —
// better UX for toggles, more code; state the trade-off in review.
7 (hard): A product page fetches product → then reviews (needs product.id) → then reviewer avatars (needs review.userIds). It takes 1.4s on a 200ms-RTT connection. Explain the math and restructure to ~500ms without changing the API.
Solution 
// Waterfall: 3 serialized round-trips ≈ 3 × (200ms RTT + server time) ≈ 1.4s.
// Observation: reviews only need the product ID — which the page ALREADY HAS
// from the route (/products/42). And avatars depend on reviews, truly.
// Restructure:
//   t=0: fire product(42) AND reviews(42) in parallel   ← was serialized for no reason
//   t=~250ms: reviews land → fire avatar batch
//   t=~500ms: everything rendered
const productQ = useFetch(`/api/products/${id}`);
const reviewsQ = useFetch(`/api/products/${id}/reviews`);        // parallel — same render
const userIds  = reviewsQ.status === "success"
  ? [...new Set(reviewsQ.data.map(r => r.userId))] : null;
const avatarsQ = useFetch(userIds ? `/api/users?ids=${userIds.join(",")}` : null);
// (useFetch skips null URLs.) General rule: a fetch may depend on another fetch's
// RESPONSE — never on another fetch's component merely having mounted. Depth of
// the request graph = number of serial RTTs; keep it as shallow as data truly allows.

🎯 Key Takeaways

  • Server data is a cache of someone else's state: keyed, stale-able, shareable — model it as such, not as component state.
  • One discriminated status (idle/loading/success/error — plus designed empty) replaces boolean soup and makes impossible UI unrepresentable.
  • The correct hand-rolled fetch: effect + AbortController cleanup, res.ok check, status reset on key change, refetch via dep bump.
  • Libraries earn their place at the architecture level: dedup, cache-first with stale-while-revalidate, and key-based invalidation after mutations.
  • Optimistic updates = snapshot, apply, rollback-with-notice; reserve for high-success low-stakes actions and reconverge with the server after.
  • Fight waterfalls: parallelize at ancestors, batch chatty calls, keep the request graph shallow — serial round-trips are pure added latency.
