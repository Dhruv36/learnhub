Context & State Management Intermediate

Lifting state works — until props tunnel through five layers that don't care ("prop drilling"). Context solves distribution; reducers solve update logic; external stores solve scale and performance. The senior skill is matching the tool to the state's scope, not defaulting to the biggest hammer.

Context — Teleporting Values Down the Tree

const ThemeContext = createContext("light");

function App() {
  const [theme, setTheme] = useState("light");
  return (
    <ThemeContext.Provider value={theme}>        {/* modern alias: <ThemeContext value=…> */}
      <Page />                                     {/* no theme prop anywhere between */}
    </ThemeContext.Provider>
  );
}
function ThemedButton() {
  const theme = useContext(ThemeContext);          // nearest provider above wins
  return <button className={`btn btn--${theme}`} />;
}
⚡ The performance fact everyone learns the hard way: when a provider's value changes (Object.is), every consumer re-renders — regardless of memo. And value={{ user, setUser }} creates a NEW object every provider render → all consumers re-render every time. Fixes: useMemo the value, split contexts (state vs dispatch, or per-concern), keep contexts for low-frequency data (theme, auth, locale) — not for rapidly-changing state.

useReducer — Update Logic as Data

function cartReducer(state, action) {
  switch (action.type) {
    case "add":    return { ...state, items: [...state.items, action.item] };
    case "remove": return { ...state, items: state.items.filter(i => i.id !== action.id) };
    case "clear":  return { ...state, items: [] };
    default: throw new Error(`Unknown action ${action.type}`);
  }
}
const [cart, dispatch] = useReducer(cartReducer, { items: [] });
dispatch({ type: "add", item });
// When: multi-field state with interdependent transitions, many update sites, or
// "next state depends on previous" everywhere. Wins: logic centralized & PURE (unit-test
// the reducer alone!), components just describe WHAT happened.
// Context + useReducer = a homegrown store — fine for medium apps:
// <CartContext value={cart}><CartDispatch value={dispatch}> … (split = fewer re-renders)

External Stores — Zustand & Redux Toolkit

// Zustand — the 2026 default for client state (tiny, no providers):
const useCart = create((set) => ({
  items: [],
  add:    (item) => set(s => ({ items: [...s.items, item] })),
  remove: (id)   => set(s => ({ items: s.items.filter(i => i.id !== id) })),
}));
const items = useCart(s => s.items);        // SELECTOR: re-render only when THIS slice changes
const add   = useCart(s => s.add);           // ← the perf win context can't give you

// Redux Toolkit — same ideas, more ceremony + the best DevTools (time travel):
// createSlice({ name, initialState, reducers: { add(state, a) { state.items.push(a.payload) } } })
// (Immer inside: "mutating" syntax producing immutable updates.)

The Decision Table (the interview answer)

State scopeTool
One componentuseState
A few related componentsLift to common parent
Complex transitions, many update sitesuseReducer
Low-frequency app-wide values (theme, auth, locale)Context (+ useMemo'd value, split contexts)
High-frequency shared client state (cart, filters, UI)Zustand / Redux Toolkit (selector subscriptions)
Server data (users, products, anything fetched)TanStack Query / SWR — NOT the store. It's cache with staleness/refetch semantics, not client state
URL-shaped state (filters, tabs, pagination)The URL (router lesson) — shareable, back-button-correct

The modern insight that shrank Redux apps by 80%: most "global state" was server cache + URL state wearing a Redux costume. Classify first; the store keeps only what's left.

Interview Questions

Q1: Why is context "not a state manager", precisely?

Context is a dependency-injection/transport mechanism: it distributes a value and re-renders ALL consumers on every value change — no selectors, no subscription granularity. Stores subscribe components to SLICES (useCart(s => s.items)) so unrelated changes skip them. Context carries the store (or low-frequency values); it shouldn't BE the high-frequency store.

Q2: Design state for an e-commerce app: products, cart, theme, search filters, checkout form.

Products = server cache (Query, keyed by filters). Filters = URL params (shareable, back-button). Cart = client store (Zustand) or context+reducer if small — persisted to localStorage. Theme = context (low-frequency) + data-theme attribute (CSS lesson). Checkout form = local state/react-hook-form; server re-validates. The classification — server/URL/client/local — IS the answer they're grading.

Q3: Why did reducers win for complex state — what's the actual benefit of actions?

Transitions become named, serializable events ("what happened") processed by one pure function — unit-testable without rendering, loggable/replayable (Redux DevTools time travel), and new transitions can't be scattered inconsistently across handlers. It's the discriminated-union + exhaustive-switch pattern (TS lesson) applied to state.

🏋️ Practice: build the cart three ways — context+useReducer (split state/dispatch contexts), then Zustand with selectors, then break the naive version (inline provider object) and watch consumers re-render with React DevTools' highlight-updates. Classify a real app's state into the four buckets.

🎯 Key Takeaways

  • Context distributes (all consumers re-render on change) — useMemo values, split contexts, keep it low-frequency.
  • useReducer = pure, testable, centralized transitions; context+reducer is a legit small store.
  • Stores add selector-granular subscriptions — the perf feature context lacks; Zustand is the modern default, RTK for DevTools/ecosystem.
  • Classify first: server state → Query; URL state → router; only true client state → store; local stays local.
  • Escalate by scope (useState → lift → reducer → context → store) — never start global.
