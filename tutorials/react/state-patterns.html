State Architecture Patterns State Management

What you'll master: the placement question — where should each piece of state live — via lifting and colocation, controlled vs uncontrolled components (the form idea generalized to any component), state taxonomies that keep apps legible (UI vs client vs server vs URL state), and normalization for relational data. Prerequisites: the whole track so far — this lesson is the architecture layer over State, Reducers, and Context.

Part 1 — The Placement Algorithm

// For every piece of state, run this:
// 1. WHO reads it? WHO writes it?
// 2. Put it in the LOWEST component that contains all readers and writers
//    (their lowest common ancestor).
// 3. If that ancestor is far from the writers, pass state down + events up
//    (Props lesson); if couriers pile up, consider context (Context lesson).
//
// Two forces pull in opposite directions:
// COLOCATE (push down): state as close to usage as possible —
//   re-renders stay small (Render Cycle), components stay portable, deletion is easy.
// LIFT (pull up): the moment TWO components need the same truth —
//   one owner, no sync code, no drift. "Sync two states" is always wrong;
//   there is no correct implementation of it, only races.

// The classic lift: filter input + filtered list are siblings:
function ProductsPage() {
  const [query, setQuery] = useState("");          // lifted to the common ancestor
  return (
    <>
      <SearchBox value={query} onChange={setQuery} />
      <ProductList query={query} />
    </>
  );
}
// The classic colocate: the SearchBox's focus-ring state, its "recently used"
// dropdown open flag — nobody else cares; they live INSIDE SearchBox.
The senior habit is re-placement. State placement isn't decided once — features migrate it. A "show archived" toggle starts inside the list (colocated), then the header wants a count badge → lift to the page. The refactor is mechanical (move useState up, pass value+setter down); what matters is noticing the moment two owners would otherwise appear.

Part 2 — Controlled vs Uncontrolled Components (the General Pattern)

The forms lesson's question — who owns the value — applies to every interactive component you build, and mature libraries answer it with both:

// Uncontrolled: the component owns its state; parent gets events.
<Accordion defaultOpen={["intro"]} onOpenChange={ids => track(ids)} />
// Easy to use, works standalone — but the parent CAN'T force it (e.g. "collapse
// all" button elsewhere on the page can't reach in).

// Controlled: the parent owns it; component is pure display + events.
<Accordion open={openIds} onOpenChange={setOpenIds} />
// Parent has full power (collapse-all is trivial) at the cost of wiring.

// Library-grade components support both with the presence-of-prop convention:
function Accordion({ open, defaultOpen = [], onOpenChange, items }) {
  const [internal, setInternal] = useState(defaultOpen);
  const isControlled = open !== undefined;          // prop present = controlled mode
  const openIds = isControlled ? open : internal;

  function toggle(id) {
    const next = openIds.includes(id)
      ? openIds.filter(x => x !== id) : [...openIds, id];
    if (!isControlled) setInternal(next);           // own it only in uncontrolled mode
    onOpenChange?.(next);                           // always report
  }
  /* render from openIds */
}
// Rules: never switch modes mid-life (the forms warning generalizes);
// value + defaultValue together is an API error worth console.warn-ing.

Part 4 — Normalization: Relational Data Without Tears

// The nested shape APIs return rots as state:
const [board, setBoard] = useState({
  lists: [{ id: "l1", title: "Todo",
            cards: [{ id: "c1", text: "Ship", assignee: { id: "u1", name: "Ada" } }] }],
});
// Renaming Ada = a triple-nested map over every list's every card. Moving a card
// between lists = two deep surgeries. The same user object is copied N times.

// NORMALIZE — flat tables by id + arrays of ids for order (a client-side DB):
const initial = {
  users: { u1: { id: "u1", name: "Ada" } },
  cards: { c1: { id: "c1", text: "Ship", assigneeId: "u1" } },
  lists: { l1: { id: "l1", title: "Todo", cardIds: ["c1"] } },
  listOrder: ["l1"],
};
// Rename Ada: one spread — { ...s.users, u1: { ...s.users.u1, name } }
// Move a card: filter one cardIds, splice into another — order data, not objects.
// One user object; every card points at it — impossible to have two disagreeing Adas.

function moveCard(state, { cardId, from, to, index }) {
  return { ...state, lists: {
    ...state.lists,
    [from]: { ...state.lists[from],
              cardIds: state.lists[from].cardIds.filter(id => id !== cardId) },
    [to]:   { ...state.lists[to],
              cardIds: [...state.lists[to].cardIds.slice(0, index), cardId,
                        ...state.lists[to].cardIds.slice(index)] },
  }};
}
// Components join at render: list.cardIds.map(id => cards[id]) — cheap, and each
// row can subscribe to just its own card. Normalize when data is RELATIONAL
// (shared references, reordering, cross-links); keep small tree-shaped data nested.

Part 5 — Derived State's Final Boss: getDerivedStateFromProps Syndrome

// The recurring disease across all frameworks: copying props into state "to edit".
function EmailEditor({ contact }) {
  const [draft, setDraft] = useState(contact.email);    // ⚠️ frozen at mount
  // …then someone "fixes" prop changes with an effect:
  useEffect(() => { setDraft(contact.email); }, [contact.email]);   // ❌ now user edits
}                                                        // get clobbered by refetches!

// The three CORRECT designs, by intent:
// A. "Editing a draft OF record X" → keyed remount; draft init is honest:
<EmailEditor key={contact.id} contact={contact} />      // new record = fresh editor

// B. "Fully controlled" → no copy at all; parent owns the draft:
<EmailEditor value={draft} onChange={setDraft} />

// C. "Reset only when the LOGICAL identity changes" → compare-and-reset in render
//    (the one sanctioned render-time setState, from the Render Cycle lesson):
const [prevId, setPrevId] = useState(contact.id);
if (prevId !== contact.id) { setPrevId(contact.id); setDraft(contact.email); }
// A is right 90% of the time. The effect version is never right: it can't
// distinguish "new record" from "same record refetched" — B and C can.

Interview Questions

Beginner: What is "lifting state up" and when do you do it?

Moving state to the lowest common ancestor of everything that reads or writes it, passing the value down as props and changes up as callbacks. Do it the moment two components need one truth — shared state with two owners requires synchronization, which is a bug factory; one owner requires none.

Beginner: Why colocate state instead of keeping it all at the top?

Smaller re-render blast radius (only the owning subtree redraws — renders flow down from the state owner), better portability (the component works anywhere without wiring), and easier deletion. Top-heavy state turns every keystroke into a page render and every component into a dependent of the shell.

Mid: Design a dual-mode (controlled/uncontrolled) Tabs component's API.

value/onValueChange for controlled, defaultValue for uncontrolled; value !== undefined selects the mode. Internally: useState from defaultValue, read from the prop when controlled, write internal state only when uncontrolled, always fire onValueChange. Warn on value+defaultValue together and on mode switches mid-lifetime. This is the exact convention native inputs and Radix/Headless UI follow, so it's what consumers expect.

Mid: Which app state belongs in the URL, and what do you gain?

Anything a user expects refresh, back/forward, and shared links to preserve: route, search query, filters, sort, page, selected record, open modal for a resource. Gains: deep-linking and shareability, free persistence across remounts, back-button correctness, and server-side renderability of the same view. The URL is state management with a browser-grade undo stack built in.

Senior: When and how do you normalize client state, and what's the render-time cost story?

Normalize when data is relational — entities referenced from multiple places, reordering, many-to-many links: flat maps by id plus id arrays for order (mirroring DB tables). Updates become single-level spreads; entities can't fork. Render-time joins (ids.map(id => table[id])) are O(visible rows) and cheap; with selector-subscribed stores, each row component re-renders only when its entity changes — better than nested state where a deep update rebuilds the whole branch. Keep genuinely tree-shaped, single-owner data nested; normalization is for relations, not a default.

Senior: A dashboard's "selected row" state is needed by the table, a detail pane, a toolbar, and analytics. Walk the placement decision including the URL option.

Readers/writers span the page → lift at least to the page component. Then apply the URL test: users share "look at this record" links and expect back to return — so the selected id belongs in the route or search params (?selected=42), making the page's owner the router. The table/toolbar receive the id + an onSelect that calls setSearchParams; the detail pane fetches by id via the query cache (server state stays in its lane); analytics subscribes to URL changes it already tracks. Local useState would work until the first "send me a link" — placement decisions should anticipate that test, not retrofit it.

🏋️ Graded Exercises

1 (easy): Temperature converter: Celsius and Fahrenheit inputs that stay in sync. What's the state, and where?
Solution 
function Converter() {
  const [celsius, setCelsius] = useState("");     // ONE state — the other is derived
  const fahrenheit = celsius === "" ? "" : (celsius * 9/5 + 32).toFixed(1);
  return (
    <>
      <input value={celsius} onChange={e => setCelsius(e.target.value)} />°C
      <input value={fahrenheit}
             onChange={e => setCelsius(e.target.value === "" ? ""
               : ((e.target.value - 32) * 5/9).toFixed(1))} />°F
    </>
  );
}
// Two states would need syncing = drift. One source, one derivation, in the
// common parent of both inputs.
2 (easy): Each ProductCard has a "♥" toggle held in card-local state. The header must now show total favorites. What moves where?
Solution

const [favorites, setFavorites] = useState(new Set());
<Header count={favorites.size} />
<ProductCard fav={favorites.has(p.id)}
             onToggle={() => setFavorites(s => {
               const n = new Set(s); n.has(p.id) ? n.delete(p.id) : n.add(p.id); return n;
             })} />
// The card's press-animation state stays colocated — only the shared truth lifts.
3 (medium): Classify each and name its home: (a) which accordion section is open in a FAQ, (b) items in a wishlist synced to the account, (c) the current search results, (d) the search term itself, (e) "user prefers reduced motion".
Solution 
// a. UI state → colocated useState in the FAQ (nobody else reads it).
// b. Client app state persisted server-side → server is the truth: mutations via
//    the query cache (optimistic toggle), components subscribe by key.
// c. Server cache → query library keyed by the term.
// d. URL state → ?q= (shareable, refresh-proof, back-button correct).
// e. Neither React state nor JS first: CSS @media (prefers-reduced-motion) —
//    read it in JS (matchMedia hook) only where logic must branch.
4 (medium): Convert this to dual-mode: a Rating component currently uncontrolled-only (defaultValue, internal state). Add controlled support without breaking existing callers.
Solution 
function Rating({ value, defaultValue = 0, onChange, max = 5 }) {
  const [internal, setInternal] = useState(defaultValue);
  const isControlled = value !== undefined;
  const current = isControlled ? value : internal;

  function set(next) {
    if (!isControlled) setInternal(next);
    onChange?.(next);
  }
  return Array.from({ length: max }, (_, i) => (
    <button key={i} onClick={() => set(i + 1)}>{i < current ? "★" : "☆"}</button>
  ));
}
// Old callers (defaultValue only) hit the uncontrolled path unchanged; new
// callers pass value+onChange and own it. Presence of `value` is the switch.
5 (medium): The bug: a settings form loads user via query, initializes fields with useState(user.name), and background refetch wipes a user's typing. Fix it three ways and pick one.
Solution 
// This is Part 5's disease. The effect "fix" (setState on user change) IS the bug.
// A. key={user.id} on the form — resets on genuine record change, immune to
//    refetches of the same record (id unchanged → no remount). ✅ PICK THIS.
// B. Fully controlled draft in the parent alongside the query — parent decides
//    merge policy; heavier, right when a "you have unsaved changes vs server" UX is needed.
// C. Render-time compare-reset on user.id — same semantics as A without remount,
//    keeps focus; use when the form is expensive to remount.
// The chooser: does "same logical record, fresher data" ever need to update the
// form? No → A. Yes, with conflict UI → B.

🎯 Key Takeaways

  • Placement algorithm: lowest common ancestor of all readers/writers — colocate by default, lift at the moment a second owner would appear. Never sync two states.
  • Controlled/uncontrolled is a general component API axis: dual-mode via value-presence is the design-system convention; never switch modes mid-life.
  • Four state kinds, four homes: UI → colocated; client app → reducer/context/store; server cache → query library; shareable/refresh-safe → the URL.
  • Normalize relational data into id-keyed tables + order arrays: single-spread updates, no forked entities, render-time joins that stay cheap.
  • Props-to-state copies are the recurring disease: cure with keyed remounts, full control, or render-time compare-reset — never a syncing effect.
  • State architecture is re-placement over time; the skill is hearing the triggers (second reader, shared link, drifted copies) as they appear.
