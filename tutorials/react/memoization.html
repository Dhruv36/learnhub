useMemo & useCallback Performance

What you'll master: what these hooks actually cache, the two legitimate reasons to use them (expensive computation, reference stability for consumers that compare), the dependency-chain discipline that makes memoization hold end-to-end, and the judgment to not use them — plus where the React Compiler changes the calculus. Prerequisites: The Re-render Model & memo — these hooks exist mostly to make that lesson's memo contracts hold.

Part 1 — What They Are (and Aren't)

// useMemo: cache a VALUE between renders, recompute only when deps change:
const sorted = useMemo(() => [...items].sort(byPrice), [items]);
//    render 1: runs the sort, stores result keyed by [items]
//    render 2 (same items reference): returns the CACHED array — same reference
//    render 3 (new items): recomputes

// useCallback: cache a FUNCTION's identity. Literally useMemo for functions:
const onSelect = useCallback(id => setSel(id), []);
// useCallback(fn, deps) === useMemo(() => fn, deps)

// What they are NOT:
// • Not "make it faster" dust — each adds a deps comparison + cache slot per render.
// • Not semantic guarantees — React MAY discard memo caches (offscreen, future
//   features). Code must stay CORRECT without them: they optimize, never fix logic.
// • Not for every function/value — the default is NO memoization until a consumer
//   needs identity or a profile names the computation.
The two — and only two — legitimate triggers: (1) the computation is measurably expensive (sorting/filtering thousands, heavy transforms); (2) the result's reference identity matters downstream — it feeds a memoized child's props, a dependency array, or a context value. No trigger → no hook. "It re-renders often" alone is trigger for neither.

Part 2 — Trigger 1: Expensive Computation

function Analytics({ transactions, threshold }) {        // 50k rows
  // Without memo: every render — ANY state change in this component — re-groups 50k rows:
  const byCategory = useMemo(() => {
    const groups = {};
    for (const t of transactions) {
      if (t.amount < threshold) continue;
      (groups[t.category] ??= []).push(t);
    }
    return groups;
  }, [transactions, threshold]);        // recompute ONLY when inputs change

  const [tab, setTab] = useState("food");   // tab switches now cost ~0 —
  return <Chart data={byCategory[tab]} />;  // the grouping is cached
}
// "Expensive" is measured, not vibed: wrap it in performance.now() or check the
// Profiler's flame width. A .map over 20 items is nanoseconds — the useMemo
// bookkeeping costs more than it saves. Four-digit row counts or nested loops
// are where it starts to matter.

Part 3 — Trigger 2: Reference Stability for Comparing Consumers

// The three consumers that compare references — feed them stable ones:

// A. Memoized child props (completing the previous lesson's contract):
const Toolbar = React.memo(ToolbarImpl);
function Page({ user }) {
  const [query, setQuery] = useState("");
  const actions = useMemo(() => ({                     // stable until user changes
    save:   () => api.save(user.id),
    share:  () => api.share(user.id),
  }), [user.id]);
  return (<>
    <input value={query} onChange={e => setQuery(e.target.value)} />
    <Toolbar actions={actions} />                      {/* keystrokes: memo HOLDS */}
  </>);
}

// B. Dependency arrays — an unstable value in deps = effect churn:
const options = useMemo(() => ({ page, pageSize }), [page, pageSize]);
useEffect(() => { fetchWith(options); }, [options]);   // fires per page change, not per render
// (Often better: depend on the primitives directly — memo the object only when
//  it's passed onward as one.)

// C. Context values — the Context lesson's fix, restated:
const value = useMemo(() => ({ user, login, logout }), [user, login, logout]);

// And the function flavor — a handler passed to a memoized list:
const onToggle = useCallback(id => {
  setSelected(s => { const n = new Set(s); n.has(id) ? n.delete(id) : n.add(id); return n; });
}, []);   // ← empty deps possible BECAUSE the functional update reads no state.
// This composition — functional updates enabling stable callbacks — is the
// single most useful memoization idiom in React.

Part 4 — The Chain Discipline: Memoization Is All-or-Nothing

// Memoization holds only if the WHOLE dependency chain is stable. One weak link
// re-derives everything downstream:

function Report({ raw }) {
  const cleaned = raw.filter(r => r.valid);            // ❌ fresh array every render…
  const stats   = useMemo(() => heavyStats(cleaned), [cleaned]);   // …so this NEVER caches
  const layout  = useMemo(() => toLayout(stats), [stats]);         // …so neither does this
}

// Fixed — stabilize the head of the chain:
function Report({ raw }) {
  const cleaned = useMemo(() => raw.filter(r => r.valid), [raw]);
  const stats   = useMemo(() => heavyStats(cleaned), [cleaned]);   // ✅ caches while raw stable
  const layout  = useMemo(() => toLayout(stats), [stats]);
}
// Debugging tool: in dev, log identity — useEffect(() => console.log("stats changed"), [stats])
// — and walk UP the chain to the first thing churning. It's usually a filter/map/
// object literal at the top, or a prop the PARENT rebuilds (fix it there).
// Same discipline for useCallback: a callback is only as stable as its deps;
// [onSave] where onSave is the parent's inline arrow = stable-looking churn.

The React Compiler postscript: React's compiler (stable in 2025+ toolchains) auto-memoizes components, values, and callbacks by analyzing your code — the manual hooks in Parts 2–4 are exactly what it inserts for you. Where it's enabled, hand-written useMemo/useCallback becomes noise to delete. The concepts stay load-bearing: you still read pre-compiler code, debug identity chains, and write the rule-following code (purity, honest deps) the compiler requires to do its job.

Interview Questions

Beginner: useMemo vs useCallback in one breath?

useMemo(fn, deps) calls fn and caches its return value; useCallback(fn, deps) caches the function itself — useCallback(fn, deps) ≡ useMemo(() => fn, deps). Both return the cached reference until a dep changes by Object.is. One caches results, the other identities.

Beginner: Why doesn't wrapping every function in useCallback speed the app up?

A function's identity only matters to consumers that compare it: memoized children, dep arrays, context. A plain <button onClick={…}> compares nothing — the button re-renders with its parent either way. Meanwhile every useCallback pays a per-render deps comparison and cache slot, so blanket usage adds cost and noise with zero skipped work.

Mid: Why do stable callbacks and functional updates go together?

A callback that reads state must list it in deps — so it gets a new identity every time that state changes, defeating the stability it was wrapped for. Functional updates (setX(prev => …)) let the callback compute from React's latest value without reading the closure, so deps can be honestly empty and the identity is stable forever. It's the pattern that makes memoized lists actually work.

Mid: Is useMemo guaranteed to run its function only when deps change?

No — that's the direction it optimizes, not a semantic guarantee. React may discard memo caches (to free memory, for offscreen trees, in future features) and recompute on a later render. Consequences: never put side effects or must-run-once logic in useMemo, and never rely on the cached identity for correctness — only for performance. Once-only semantics belong to useState lazy initializers or refs.

Senior: Walk through debugging "my memoized child still re-renders" through the identity chain.

Profiler → "why did this render" names the prop. Then walk the chain upward: is the prop a useMemo'd value? Check its deps — log identity changes per dep (useEffect(() => console.log("dep X changed"), [X])) to find which churns. Typical roots: a raw .filter()/.map() above the memo, an object literal in the parent's JSX, a callback dep that's itself inline in the grandparent, or context value rebuilt per render. Fix at the root — the first unstable link — not by adding more memo downstream; then re-profile to confirm skips actually happen.

Senior: With the React Compiler enabled, what's your policy on these hooks?

New code: don't write manual useMemo/useCallback for identity or render cost — the compiler inserts finer-grained memoization than humans maintain, provided components follow the rules (purity, no render mutations) its ESLint plugin verifies. Keep explicit useMemo where the algorithm is the point (a deliberate cache boundary others reason about) and for API-level contracts (a hook returning a documented-stable function). Existing code: delete manual memoization opportunistically when files come under edit, verifying with the Profiler. And the fundamentals stay in interviews and code review — the compiler automates the mechanism, not the understanding.

🏋️ Graded Exercises

1 (easy): Should this be memoized? const upper = name.toUpperCase(); in a component re-rendering on every keystroke. Answer with the trigger test.
Solution 
// Trigger 1 (expensive)? toUpperCase on one string: nanoseconds. No.
// Trigger 2 (identity consumer)? A string is compared by value everywhere —
// identity is meaningless for primitives. No.
// → No memoization. Compute in render. (Primitives NEVER need useMemo.)
2 (easy): Fix the never-caching memo: const visible = useMemo(() => filter(items, opts), [items, { ...opts }]);
Solution 
// The dep { ...opts } is a FRESH object every render — deps "change" every time,
// so the memo recomputes unconditionally (worse than no memo: adds bookkeeping).
const visible = useMemo(() => filter(items, opts), [items, opts]);
// …and if opts is itself rebuilt by the parent, fix THAT (memoize where it's
// created, or destructure to primitive deps: [items, opts.minPrice, opts.tag]).
3 (medium): A searchable list: input + 8,000 items filtered by query + a memoized <Row> per item with an onPick callback. Write the component with correct, minimal memoization.
Solution 
function Picker({ items, onPicked }) {                 // 8k items
  const [q, setQ] = useState("");

  const visible = useMemo(                              // trigger 1: 8k-row filter
    () => items.filter(i => i.name.toLowerCase().includes(q.toLowerCase())),
    [items, q]);

  const onPick = useCallback(id => onPicked(id), [onPicked]);   // trigger 2: memo rows
  // (only as stable as onPicked — if the parent inlines it, fix the parent)

  return (
    <>
      <input value={q} onChange={e => setQ(e.target.value)} />
      <ul>{visible.map(i => <Row key={i.id} item={i} onPick={onPick} />)}</ul>
    </>
  );
}
// NOT memoized: q handling, the JSX, visible.length — no triggers. And above a
// few hundred visible rows, add virtualization (previous lesson) before more memo.
4 (medium): This effect fires on every render. Explain and give two fixes: const cfg = { url, retries: 3 }; useEffect(() => { client.configure(cfg); }, [cfg]);
Solution 
// cfg is rebuilt each render → new identity → deps "changed" → effect re-runs.
// Fix A (preferred): depend on primitives, build inside:
useEffect(() => { client.configure({ url, retries: 3 }); }, [url]);
// Fix B: stabilize the object if it's ALSO passed elsewhere:
const cfg = useMemo(() => ({ url, retries: 3 }), [url]);
useEffect(() => { client.configure(cfg); }, [cfg]);
// A is less machinery; B earns its keep only when cfg feeds multiple consumers.
5 (medium): Code-review this: const handleSubmit = useCallback(() => { api.save(form); }, []); — the form state is in the same component. What's wrong, and the two honest repairs?
Solution 
// Classic deps lie: [] but reads `form` → the callback closes over the MOUNT
// render's form forever. Every submit saves the initial (empty) form. useCallback
// didn't cause it, but stability pressure motivated the lie.
// Repair A — honest deps: useCallback(() => api.save(form), [form])
//   (identity changes when form changes — fine unless a memoized child needs it)
// Repair B — no useCallback at all: a submit handler on a 
 element isn't
//   compared by anything; plain function, always-fresh closure, zero machinery. ✅
// B unless a comparing consumer exists. Never repair a deps lie by keeping the lie.
6 (hard): Build a useStableHandlers(handlers) hook: accepts an object of possibly-inline functions and returns identity-stable wrappers that always call the latest versions. State when this is legitimate versus a smell.
Solution 
function useStableHandlers(handlers) {
  const ref = useRef(handlers);
  useEffect(() => { ref.current = handlers; });        // latest-ref, every commit

  return useMemo(() => {
    const stable = {};
    for (const key of Object.keys(ref.current)) {
      stable[key] = (...args) => ref.current[key]?.(...args);   // trampoline
    }
    return stable;
  }, []);                                              // built once; identities eternal
}
// const { onSave, onCancel } = useStableHandlers({ onSave: () => save(draft), onCancel });
// Legitimate: library/design-system boundaries where you can't control callers'
// inline props but must keep internal memoization intact (the useEventListener
// pattern from Custom Hooks, generalized).
// Smell: sprinkling it inside app code to avoid fixing parent identity or to
// paper over deps — and note the constraint it creates: handlers are only safe
// to call from events/effects (post-commit), not during render, since the ref
// updates after commit. Assumes a fixed key set (built once).
7 (hard): The Profiler shows <Chart> (memoized, expensive) re-rendering on every parent render. Its props: data={useMemo(…, [rows])}, options={CHART_OPTIONS} (module constant), onZoom={handleZoom} (useCallback with [viewport]), and viewport changes on every mousemove over the map beside it. Untangle the chain and fix it so chart re-renders happen only when rows change.
Solution 
// Chain audit: data ✅ stable on [rows]. options ✅ module constant. onZoom ❌ —
// deps [viewport] churn per mousemove → new identity per parent render → memo dies.
// Root cause: the callback READS viewport but zoom events are rare while
// viewport changes are constant — classic frequency mismatch.
// Fix 1 (latest-ref): stabilize onZoom, read viewport through a ref:
const viewportRef = useRef(viewport);
useEffect(() => { viewportRef.current = viewport; });
const onZoom = useCallback(delta => {
  applyZoom(viewportRef.current, delta);               // fresh at CALL time
}, []);                                                 // identity: eternal
// Fix 2 (structural, consider first): does viewport belong in this parent at
// all? If only the map reads it per-mousemove, colocate it in the map component —
// the parent stops re-rendering entirely, and the original onZoom deps become
// harmless. (Previous lesson: state down beats memo repair.)
// After either: Profiler should show Chart gray on mousemove, colored only on
// rows changes. Verify, don't assume.

🎯 Key Takeaways

  • useMemo caches a value, useCallback a function identity — optimizations only, never semantics; React may discard caches.
  • Exactly two triggers: measured-expensive computation, or a consumer that compares references (memo props, dep arrays, context values). Otherwise: don't.
  • Stable callbacks + functional updates is the core idiom — empty honest deps because the updater reads nothing.
  • Memoization is a chain: one unstable link (raw filter/map, inline literal, churning dep) silently re-derives everything below — fix the first link, not the last.
  • Never lie in deps for stability; that trades correctness for identity. Fix with functional updates, latest-refs, or by deleting the memoization.
  • The React Compiler automates these hooks for rule-following code — the understanding stays; the hand-written ceremony is on its way out.
