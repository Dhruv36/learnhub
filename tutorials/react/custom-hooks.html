Custom Hooks Effects & Lifecycle

What you'll master: why the Rules of Hooks exist (the array-on-the-fiber model that makes them non-negotiable), extracting stateful logic into hooks that compose, API design for hooks (arguments, return shapes, stability guarantees), and the judgment call of when a custom hook earns its existence. Prerequisites: everything in this section — custom hooks are state + effects + refs behind a function boundary. You've already written several (useDebouncedValue, useFetch, usePrevious, useOnScreen).

Part 1 — Why the Rules of Hooks Exist (the Mechanism)

// React stores hook data as a LIST on the component's fiber, matched BY CALL ORDER:
function Form() {
  const [name, setName] = useState("");     // fiber.hooks[0]
  const [email, setEmail] = useState("");   // fiber.hooks[1]
  useEffect(persistDraft, [name, email]);   // fiber.hooks[2]
}
// Render N: React walks the list — hook call #1 gets slot 0, #2 gets slot 1…
// There are no names, no keys. ORDER IS THE IDENTITY.

// Break the order and slots shift:
if (user) {
  const [name] = useState("");              // ❌ sometimes slot 0, sometimes absent
}
const [email] = useState("");               // …email reads NAME's slot when user is set
// → "Rendered more/fewer hooks than during the previous render" — or worse,
//   silent state cross-wiring.

// Hence the two rules:
// 1. Only call hooks at the TOP LEVEL — never in ifs, loops, nested functions,
//    or after early returns.
// 2. Only call hooks from React functions — components or other hooks (the use-
//    prefix is how the linter knows to check call-order discipline inside).
The interview version: hooks trade explicit identity (like a key or a name) for positional identity, which is what lets useState() work with zero configuration. The rules aren't style — they're the contract that keeps slot i meaning the same thing every render. Conditional logic goes inside hooks (a condition in the effect, an enabled flag), never around them.

Part 2 — Extraction: From Duplicated Logic to a Hook

// Two components independently track online status:
function StatusBar() {
  const [online, setOnline] = useState(navigator.onLine);
  useEffect(() => {
    const up = () => setOnline(true), down = () => setOnline(false);
    window.addEventListener("online", up);
    window.addEventListener("offline", down);
    return () => { window.removeEventListener("online", up);
                   window.removeEventListener("offline", down); };
  }, []);
  return <span>{online ? "🟢 Online" : "🔴 Offline"}</span>;
}
// SaveButton has the same 10 lines to disable itself offline…

// Extraction is mechanical: cut the hooks, paste into a function, return what's used:
function useOnlineStatus() {
  const [online, setOnline] = useState(navigator.onLine);
  useEffect(() => {
    const up = () => setOnline(true), down = () => setOnline(false);
    window.addEventListener("online", up);
    window.addEventListener("offline", down);
    return () => { window.removeEventListener("online", up);
                   window.removeEventListener("offline", down); };
  }, []);
  return online;
}

function StatusBar()  { const online = useOnlineStatus(); return <span>{online ? "🟢" : "🔴"}</span>; }
function SaveButton() { const online = useOnlineStatus(); return <button disabled={!online}>Save</button>; }

Crucial semantics: each call to a custom hook gets its own state and effects — StatusBar and SaveButton each have an independent listener pair and online flag. Custom hooks share logic, never state. (Sharing the state itself needs lifting or Context — or a shared external store.) And what you named it matters: components describe UI; hooks describe capabilities — useOnlineStatus, not useOnlineListener (implementation) or OnlineHelper (unlintable).

Part 3 — Hook API Design

// RETURN SHAPES — three idioms, choose by arity and usage:
function useToggle(init = false) {                       // pair, like useState:
  const [on, setOn] = useState(init);                    // for value + one operation
  const toggle = useCallback(() => setOn(o => !o), []);
  return [on, toggle];                                   // callers rename freely:
}                                                        // const [open, flip] = useToggle()

function useFetch(url) {                                 // object: many fields,
  /* … */                                                // self-documenting at the call site,
  return { status, data, error, refetch };               // destructure what you need
}

function useOnlineStatus() { return online; }            // bare value: read-only signals

// STABILITY is part of the contract — returned functions should be useCallback-
// wrapped so consumers can safely put them in dep arrays or pass to memoized
// children. An unstable `refetch` that changes identity every render silently
// re-triggers every effect that lists it. Document what's stable.

// ARGUMENTS: accept primitives where possible (object args re-created each render
// become the useEffect-object-deps problem INSIDE your hook). If you take
// callbacks, either require stability or route them through the latest-ref
// pattern (useEffect Mastery, ex. 7) so your hook doesn't resubscribe per render.

Part 4 — Composition: Hooks Calling Hooks

// The power move: hooks stack like functions, each layer testable and replaceable.

// Layer 1 — primitive (from useEffect Mastery):
function useDebouncedValue(value, ms) { /* …setTimeout + cleanup… */ }

// Layer 2 — capability, composed from layer 1 + useFetch:
function useSearch(query) {
  const debounced = useDebouncedValue(query, 300);
  const result = useFetch(query ? `/api/search?q=${encodeURIComponent(debounced)}` : null);
  return { ...result, isDebouncing: debounced !== query };
}

// Layer 3 — the component reads like its spec:
function SearchPage() {
  const [q, setQ] = useState("");
  const { status, data, isDebouncing } = useSearch(q);
  return (
    <>
      <input value={q} onChange={e => setQ(e.target.value)} />
      {isDebouncing && <Dots />}
      {status === "success" && <Results items={data} />}
    </>
  );
}
// Debounce timing, race handling, and status modeling each live in exactly one
// place. Swap useFetch for React Query inside useSearch and no component changes —
// hooks are the seam where implementations rotate.

Part 5 — Build It: useLocalStorage (a Production-Grade Primitive)

function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {              // lazy init: read disk once
    try {
      const raw = localStorage.getItem(key);
      return raw !== null ? JSON.parse(raw) : initialValue;
    } catch {                                              // corrupt JSON / privacy mode
      return initialValue;
    }
  });

  const set = useCallback(updater => {                     // stable, supports fn form
    setValue(prev => {
      const next = typeof updater === "function" ? updater(prev) : updater;
      try { localStorage.setItem(key, JSON.stringify(next)); } catch {}
      return next;
    });
  }, [key]);

  useEffect(() => {                                        // cross-tab sync — the detail
    const onStorage = e => {                               // that separates toy from tool
      if (e.key === key && e.newValue !== null) {
        try { setValue(JSON.parse(e.newValue)); } catch {}
      }
    };
    window.addEventListener("storage", onStorage);
    return () => window.removeEventListener("storage", onStorage);
  }, [key]);

  return [value, set];                                     // useState-shaped: drop-in
}

// const [prefs, setPrefs] = useLocalStorage("prefs", { theme: "dark" });
// Every lesson in this section shows up: lazy init, try/catch at the storage
// boundary, stable setter with functional updates, external-system sync (other
// tabs!) with cleanup, and a familiar return contract.

Part 6 — When NOT to Write a Custom Hook

SituationBetter tool
Pure computation, no hooks inside (formatPrice)Plain function — hooks impose call rules for nothing
Used once, unlikely to repeat, 5 linesLeave it inline — indirection has a reading cost
It returns JSXThat's a component — name it like one
Wrapping one hook 1:1 (useMyState = useState)Delete it — abstraction without added meaning
Sharing STATE between componentsLift state / Context / store — hooks duplicate state per caller
"Might need options later" — config params for hypothetical callersExtract for the second real caller, not the imagined tenth
The extraction bar: a custom hook is justified by either duplication (second real usage) or naming a concept that makes the component readable (useSearch hides four hooks behind the word the spec uses). "It made the file shorter" is neither. Every hook is API surface someone maintains — and its deps, stability, and cleanup contracts travel with it.

Interview Questions

Beginner: What is a custom hook, really?

A plain JavaScript function whose name starts with use and that calls other hooks. The prefix signals to React's linter to enforce the Rules of Hooks inside it. Custom hooks let you extract and reuse stateful logic — each caller gets independent state/effects; only the logic is shared.

Beginner: Why can't hooks be called conditionally?

React identifies each hook by its position in the call sequence on the component's fiber — there are no names. A conditional call shifts every subsequent hook into the wrong slot on some renders, so state lands in the wrong variables or React throws a hook-count error. Conditions belong inside hook logic, not around hook calls.

Mid: Do two components using the same custom hook share anything at runtime?

Only code. Each call site executes the hook's useState/useEffect against its own fiber, producing fully independent state, effects, and cleanup. If shared state is the goal, the hook can instead subscribe both callers to one external source (context value, store, module-level emitter) — the hook then shares the subscription pattern while the store holds the single truth.

Mid: What stability guarantees should a well-designed hook's return value carry?

Returned functions should be identity-stable (useCallback) so consumers can list them in dependency arrays and pass them to memoized children without churn; returned objects should either be memoized or documented as fresh-per-render. The hook's own effect deps must be honest, and callback arguments from the consumer shouldn't force resubscription (latest-ref internally). In short: a hook's identity behavior is part of its public API.

Senior: How would you design useEventListener(target, type, handler) so consumers can pass inline handlers without resubscribing every render?

Store the handler in a ref updated every render (useEffect(() => { ref.current = handler })), and subscribe once per [target, type] with a stable trampoline (e => ref.current(e)). The listener's identity never changes, so add/remove runs only when target/type change, yet the trampoline always invokes the freshest closure — no stale state, no churn, no linter suppression. This latest-ref pattern is the standard answer for "stable subscription, fresh callback," and is what React's experimental useEffectEvent formalizes.

Senior: Your team's useEntityForm hook has grown 14 options and 9 return fields. Critique and refactor strategy.

It's a god-hook: multiple capabilities (fetching, validation, dirty-tracking, submission, navigation guards) fused behind flags, so every caller pays for all of it, tests need the union of setups, and each new option multiplies paths. Refactor: decompose into single-capability hooks (useEntity, useValidation, useDirtyGuard, useSubmit) that the current hook composes for backward compatibility; new call sites compose directly. Options shrink because composition replaces configuration — callers include the layers they want instead of flagging away the ones they don't.

3 (medium): Extract useInterval(callback, delay) that pauses when delay is null and never suffers stale closures, then build a pausable clock with it.
Solution 
function useInterval(callback, delay) {
  const saved = useRef(callback);
  useEffect(() => { saved.current = callback; });        // latest-ref: fresh closure
  useEffect(() => {
    if (delay === null) return;                          // paused: no interval, no cleanup needed
    const id = setInterval(() => saved.current(), delay);
    return () => clearInterval(id);
  }, [delay]);                                           // resubscribe only on delay change
}

function Clock() {
  const [now, setNow] = useState(() => new Date());
  const [running, toggle] = useToggle(true);
  useInterval(() => setNow(new Date()), running ? 1000 : null);
  return <p onClick={toggle}>{now.toLocaleTimeString()} {running ? "⏸" : "▶"}</p>;
}
4 (medium): Build useMediaQuery(query) (e.g. "(max-width: 600px)") returning a live boolean.
Solution 
function useMediaQuery(query) {
  const [matches, setMatches] = useState(() => matchMedia(query).matches);
  useEffect(() => {
    const mql = matchMedia(query);
    setMatches(mql.matches);                             // re-sync if query changed
    const onChange = e => setMatches(e.matches);
    mql.addEventListener("change", onChange);
    return () => mql.removeEventListener("change", onChange);
  }, [query]);
  return matches;
}

6 (hard): Implement useUndoableState(initial) returning { value, set, undo, redo, canUndo, canRedo } — the undo-stack exercise from the State lesson, generalized into a reusable hook.
Solution 
function useUndoableState(initial) {
  const [hist, setHist] = useState({ past: [], present: initial, future: [] });

  const set = useCallback(next => setHist(h => ({
    past: [...h.past, h.present],
    present: typeof next === "function" ? next(h.present) : next,
    future: [],                                         // new edit clears redo
  })), []);
  const undo = useCallback(() => setHist(h => h.past.length === 0 ? h : ({
    past: h.past.slice(0, -1),
    present: h.past[h.past.length - 1],
    future: [h.present, ...h.future],
  })), []);
  const redo = useCallback(() => setHist(h => h.future.length === 0 ? h : ({
    past: [...h.past, h.present],
    present: h.future[0],
    future: h.future.slice(1),
  })), []);

  return { value: hist.present, set, undo, redo,
           canUndo: hist.past.length > 0, canRedo: hist.future.length > 0 };
}
// One state object because past/present/future change TOGETHER (State lesson);
// all mutators functional + stable; the return object names every capability.

🎯 Key Takeaways

  • Hooks live in an order-matched list on the fiber — the Rules (top level only, React functions only) keep slot i meaning the same thing every render.
  • Custom hooks extract stateful logic behind a use-named function; every caller gets independent state — shared state needs context/stores, not hooks alone.
  • Hook API design: useState-pair, object, or bare-value returns; stable returned functions; primitive params; latest-ref for consumer callbacks. Identity behavior is public API.
  • Hooks compose in layers — primitives → capabilities → features — giving each concern one home and making implementations swappable at the hook seam.
  • Extraction bar: real duplication or a concept worth naming; not one-liners, not JSX-returners (components), not speculative flexibility.
  • The section's patterns are the toolkit: lazy init, cleanup pairs, functional updates, refs for machinery — a custom hook is where they get packaged.
