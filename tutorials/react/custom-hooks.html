Custom Hooks Intermediate

A custom hook is just a function named useSomething that calls other hooks — and it's React's entire code-reuse story: stateful logic extracted, shared, and tested independently of any UI. This is what replaced mixins, HOCs, and render props.

Extraction — From Component to Hook

function useLocalStorage(key, initial) {
  const [value, setValue] = useState(() => {
    const raw = localStorage.getItem(key);
    return raw != null ? JSON.parse(raw) : initial;    // lazy init (props-state lesson)
  });
  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));   // sync with the external system
  }, [key, value]);
  return [value, setValue];                             // mimic useState's API — familiar shape
}
// Usage — indistinguishable from useState:
const [theme, setTheme] = useLocalStorage("theme", "light");

The crucial mental model: hooks share logic, not state. Two components calling useLocalStorage get independent state (each call = its own useState slots). Shared STATE is the previous lesson (context/stores); shared BEHAVIOR is this one.

The Rules of Hooks — and Why They Exist

1. Only call hooks at the TOP LEVEL — never in ifs, loops, or nested functions
2. Only call hooks from components or other hooks (the use* naming lets lint enforce this)

// WHY: React stores hook state in an ORDERED LIST per component. Call order IS identity:
//   render 1: useState(a) → slot 0, useState(b) → slot 1
//   render 2 (if a hook was skipped): everything after it reads the WRONG slot 💥
// Conditional logic goes INSIDE hooks (effects can early-return), not around them.

A Production Toolkit — Four Hooks Worth Memorizing

// 1. useDebouncedValue — the closures lesson meets React:
function useDebouncedValue(value, ms) {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const t = setTimeout(() => setDebounced(value), ms);
    return () => clearTimeout(t);                      // cleanup IS the debounce reset
  }, [value, ms]);
  return debounced;
}
const results = useSearch(useDebouncedValue(query, 300));

// 2. useMediaQuery — subscribe to a browser API:
function useMediaQuery(q) {
  const [matches, setMatches] = useState(() => matchMedia(q).matches);
  useEffect(() => {
    const mql = matchMedia(q);
    const onChange = (e) => setMatches(e.matches);
    mql.addEventListener("change", onChange);
    return () => mql.removeEventListener("change", onChange);
  }, [q]);
  return matches;
}

// 3. useOnClickOutside — the dropdown-closer:
function useOnClickOutside(ref, handler) {
  useEffect(() => {
    const listener = (e) => { if (ref.current && !ref.current.contains(e.target)) handler(e); };
    document.addEventListener("pointerdown", listener);
    return () => document.removeEventListener("pointerdown", listener);
  }, [ref, handler]);          // ⚠ handler identity — see the useCallback note below
}

// 4. useAsync — the effects lesson's fetch pattern, generalized (or: just use TanStack Query)

Designing Good Hook APIs

PrincipleExample
Return shapes callers expect[value, setter] for state-like; { data, status, error } objects for queries (destructure-and-rename friendly)
Accept primitives where possibleuseUser(id: number) not useUser({ id }) — object args are new references every render → effect churn
Stabilize returned functionsWrap in useCallback so consumers can safely put them in deps
One hook, one concernuseAuth + useCart, not useAppEverything — hooks compose, so keep them atomic
Name by capabilityuseDebounce, useOnScreen, usePagination — the use prefix is REQUIRED for lint to check rules

Testing Hooks

import { renderHook, act } from "@testing-library/react";
test("useLocalStorage persists", () => {
  const { result } = renderHook(() => useLocalStorage("k", 1));
  act(() => result.current[1](42));
  expect(result.current[0]).toBe(42);
  expect(JSON.parse(localStorage.getItem("k"))).toBe(42);
});
// Stateful logic tested with zero components — the payoff of extraction.

Interview Questions

Q1: Why does hook state depend on call ORDER, and what design alternative did React reject?

Hooks are stored in a per-component array; each call reads the next slot — no names, so order is identity, hence the top-level rule. The alternative (keyed/named state) was rejected for name-collision and composition problems (two copies of one hook in a component would clash). Order-based slots make composition free at the cost of the two rules.

Q2: Custom hooks vs HOCs vs render props — why did hooks win?

HOCs/render props reuse stateful logic through the TREE: wrapper pyramids ("wrapper hell"), name collisions in injected props, unclear data sources. Hooks reuse through plain FUNCTION composition: flat, explicit (const x = useX()), type-friendly, and multiple instances coexist. Same capability, no tree distortion.

Q3: Build useOnScreen(ref) — what does a correct implementation include?

useState(false) + effect creating an IntersectionObserver on ref.current, setting state from entry.isIntersecting, observer.disconnect() in cleanup, deps [ref]. Bonus points: an options param (threshold/rootMargin) that's primitive-or-memoized, SSR-safety (no window during render), and noting it composes: const visible = useOnScreen(ref); const data = useQuery({ enabled: visible }) — lazy-loading via composition.

🏋️ Practice: build useLocalStorage, useDebouncedValue, and useOnClickOutside; compose the first two into a persisted, debounced search box; write a renderHook test for one; trigger the call-order crash deliberately (hook inside if) and read the error.

🎯 Key Takeaways

  • Custom hook = useX function calling hooks — shares LOGIC; each caller gets independent state.
  • Two rules exist because state lives in call-order slots; the use prefix makes them lintable.
  • Cleanup-driven patterns (debounce via effect cleanup) unify the closures + effects lessons.
  • API design: state-like tuples or status objects, primitive params, useCallback'd returns, one concern per hook.
  • renderHook tests logic without UI — the extraction payoff; hooks replaced HOCs/render-props via flat function composition.
