Testing React Patterns & Quality

What you'll master: the testing philosophy that makes React tests survive refactors (test behavior users see, not implementation), React Testing Library end-to-end — queries, events, async — mocking the network with MSW, testing hooks and providers, and the pyramid judgment of what to test at which level. Prerequisites: the track so far; tooling assumed: Vitest (or Jest) + Testing Library + MSW — the 2026 default stack.

Part 1 — The Philosophy: Test What the User Sees

// Two tests of the same Counter:

// ❌ IMPLEMENTATION test — breaks on any refactor, catches no real bugs:
expect(counter.state.count).toBe(1);             // reaches into internals
expect(setCountSpy).toHaveBeenCalledWith(1);     // asserts HOW, not WHAT

// ✅ BEHAVIOR test — survives useState→useReducer, renaming, memoizing:
render(<Counter />);
await user.click(screen.getByRole("button", { name: /increment/i }));
expect(screen.getByText("Count: 1")).toBeInTheDocument();

// The rule (Testing Library's thesis): interact through what users can see and
// do — roles, labels, text, clicks, typing. Assert on rendered output. If a
// refactor keeps behavior identical, tests must stay green; if a user-visible
// behavior breaks, a test must fail. Every deviation from that trades
// refactor-freedom for false precision.

Part 2 — Queries: the Accessibility-First API

import { render, screen } from "@testing-library/react";

// Priority order — use the FIRST that applies (it mirrors how users find things):
screen.getByRole("button", { name: /save/i });   // 1. role+name: how AT users navigate
screen.getByLabelText(/email/i);                 // 2. form fields by their label
screen.getByPlaceholderText(/search/i);          // 3. weaker (placeholder ≠ label)
screen.getByText(/no results/i);                 // 4. non-interactive content
screen.getByTestId("chart-canvas");              // 5. LAST resort: invisible to users

// The three variants encode EXPECTATIONS:
getBy…    → must exist NOW (throws otherwise) — assert presence
queryBy…  → may be absent (null) — assert ABSENCE: expect(queryBy…).toBeNull()
findBy…   → will appear ASYNC (returns a promise, retries ~1s) — after fetches

// A hidden payoff: role queries FAIL on inaccessible markup. A clickable div
// (Events lesson's sin) has no button role — the test can't find it. Testing
// Library quietly enforces the a11y you should have written anyway.

Part 3 — Interactions & Async: userEvent and findBy

import userEvent from "@testing-library/user-event";

test("adds a todo", async () => {
  const user = userEvent.setup();                 // ALWAYS setup() first (v14 API)
  render(<TodoApp />);

  await user.type(screen.getByLabelText(/new todo/i), "Ship the release");
  await user.keyboard("{Enter}");

  expect(screen.getByRole("listitem")).toHaveTextContent("Ship the release");
});
// userEvent > fireEvent: type() fires the full real sequence (keydown, input,
// keyup per char), respects disabled, moves focus — it simulates a USER.
// fireEvent dispatches one synthetic event — a tool for edge cases only.

test("shows results after search", async () => {
  const user = userEvent.setup();
  render(<Search />);
  await user.type(screen.getByRole("searchbox"), "react");

  // findBy = getBy + waitFor: retries until the async UI lands or times out:
  expect(await screen.findByText(/33 results/i)).toBeInTheDocument();
  // Loading is expected UI too — test the states, not just the destination:
  // (assert the spinner appeared while pending, if the UX contract cares)
});
// NEVER: arbitrary sleeps (flaky), waitFor(() => {}) with empty bodies,
// or asserting immediately after an async action (races).
// act() warnings mean an update happened outside the library's awareness —
// the fix is await-ing the userEvent/findBy properly, not wrapping things in act.

Part 4 — The Network: Mock at the HTTP Seam With MSW

// Mocking fetch/axios per-test couples tests to the request library and skips
// your parsing/error code. MSW intercepts at the network level — your ENTIRE
// data layer (request fns, res.ok checks, query cache) runs for real:

import { http, HttpResponse } from "msw";
import { setupServer } from "msw/node";

const server = setupServer(
  http.get("/api/products", () =>
    HttpResponse.json([{ id: 1, name: "Camera", price: 999 }])),
);
beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());        // per-test overrides don't leak
afterAll(() => server.close());

test("renders products", async () => {
  render(<ProductList />, { wrapper: AppProviders });
  expect(await screen.findByText("Camera")).toBeInTheDocument();
});

test("shows retry on server error", async () => {
  server.use(http.get("/api/products", () =>     // override for THIS test
    HttpResponse.json({ message: "boom" }, { status: 500 })));
  render(<ProductList />, { wrapper: AppProviders });
  expect(await screen.findByRole("button", { name: /retry/i })).toBeInTheDocument();
});
// Error paths, empty payloads, slow responses (delay()) — each is one handler
// override. This is where Data Fetching's status machines earn their tests.

Part 5 — Providers, Hooks & Reducers: Testing Each Layer

// 1. A renderWithProviders util — the Context lesson's DI payoff:
function renderWithProviders(ui, { user = FAKE_USER, route = "/" } = {}) {
  return render(
    <MemoryRouter initialEntries={[route]}>          {/* router for tests */}
      <QueryClientProvider client={makeTestQueryClient()}>  {/* fresh cache per test! */}
        <AuthProvider initialUser={user}>
          {ui}
        </AuthProvider>
      </QueryClientProvider>
    </MemoryRouter>
  );
}
// makeTestQueryClient: retries OFF (error tests shouldn't wait 3 backoffs).

// 2. Custom hooks — renderHook for hooks with no natural component:
import { renderHook, act } from "@testing-library/react";
test("useUndoableState undoes", () => {
  const { result } = renderHook(() => useUndoableState(0));
  act(() => result.current.set(5));
  act(() => result.current.undo());
  expect(result.current.value).toBe(0);
});
// But prefer testing hooks THROUGH a component that uses them when one exists —
// closer to reality, one less API.

// 3. Reducers — plain function tests, no React at all (Reducers lesson ex. 4):
expect(cartReducer(state, { type: "added", product })).toEqual(expected);
// The layering: pure logic (reducers, utils) → unit tests, hundreds, instant.
// Components → behavior tests through the real providers. Full flows → a few
// E2E (Playwright) against a real browser. The classic pyramid, React-flavored.

Part 6 — What to Test (and What Not) — Senior Layer

Worth testingSkip (or delete the test)
User flows: fill form → submit → see result/errorStatic markup ("renders a div with class x")
Conditional logic: empty/loading/error/permission statesThird-party internals (the router routes, the library fetches)
Business rules in reducers/utils (exhaustively)Styles/layout (snapshot noise — visual tools do this)
Edge cases the type system can't stop (boundary values, races)Implementation details: state shape, call counts, hook order
Regressions: every production bug gets a test firstTrivial pass-through components
Accessibility contract (roles/names — often free via queries)Whatever a 5-second manual check covers better, forever
Snapshot tests deserve their own warning: a full-component snapshot asserts "nothing changed," which fails on every intentional change — so developers reflexively update it, which means it asserts nothing. Reserve tiny, deliberate snapshots for stable serialized output (a generated config, an error payload). If reviewers can't tell a bad diff from a good one, the snapshot is noise.

Interview Questions

Beginner: What does "test behavior, not implementation" mean concretely in React?

Interact only through the rendered UI (click buttons by their accessible name, type into labeled fields) and assert only on what renders (text, roles, attributes users perceive). Never reach into component state, props, hook internals, or spy on setters. Then refactors that preserve behavior keep tests green, and tests fail exactly when users would notice — the two properties that make a suite worth maintaining.

Beginner: getBy vs queryBy vs findBy?

getBy throws if absent — for asserting present-now elements. queryBy returns null — the only correct way to assert absence. findBy returns a promise that retries — for elements that appear after async work (fetches, transitions). Choosing the wrong one produces either false failures (getBy on async UI) or races (asserting without awaiting).

Mid: Why MSW over mocking your api module?

Module mocks replace your data layer with the test author's assumptions — parsing, res.ok handling, query-cache behavior, and URL construction all go untested, and the mock's shape silently drifts from the real API. MSW intercepts actual HTTP, so the entire client-side pipeline runs for real against controlled responses; error/empty/slow cases are handler overrides; and the same handlers reuse in Storybook and dev. You mock the one thing you don't own — the server — at the seam where it actually sits.

Mid: How do you test a component that needs router, query client, and auth context?

A renderWithProviders util wrapping the real providers with test-controlled values: MemoryRouter with an initial route, a fresh QueryClient per test (retries disabled), AuthProvider with an injectable user. Tests read as renderWithProviders(<Orders />, { user: adminUser, route: "/orders/7" }). This is dependency injection paying off — and if a component needs excessive provider setup, that's design feedback: split a presentational inner component and test it with plain props.

Senior: Design the test strategy for the checkout form from the Forms lesson (validation policy, submission states, server errors).

Reducer/validators as pure unit tests: every rule and boundary value (qty 0/1/1.5/"", email shapes) — exhaustive and instant. Component behavior tests for the UX contracts: error appears on blur not on first keystroke, fixes clear live, submit reveals all errors and focuses the first invalid field, button disables while submitting, failure keeps input values (each is one user-flow test). MSW for the submission: success path, 402 decline path (asserting the preserved form + error message), timeout with delay. One E2E in Playwright for the happy path through the real backend sandbox. Explicitly untested: styling, layout, the query library's retry internals. The suite maps one-to-one to the lesson's UX decisions — which is the point: tests encode the contract.

Senior: Your suite is green but production breaks weekly. Audit the likely causes.

(1) Implementation-detail tests: they verify wiring, not behavior — refactor-safe bugs sail through. (2) Mocks drifted from reality: module-mocked APIs returning shapes the real server stopped sending — move to MSW with contract-checked handlers (or contract tests against the API schema). (3) Coverage concentrated on pure logic while integration seams (provider wiring, route params, cache invalidation) are untested — add flow tests through real providers. (4) Only happy paths: no error/empty/slow/race cases — test the status machines. (5) Env gaps: jsdom can't catch layout, real-browser, or timing bugs — add a thin Playwright layer for the flows that keep breaking. Then instrument: every production incident gets a failing test before the fix, which over time reshapes the suite around what actually breaks.

🏋️ Graded Exercises

1 (easy): Test the useToggle-powered sidebar: hidden initially, appears after clicking "☰", disappears on second click.
Solution 
test("toggles the sidebar", async () => {
  const user = userEvent.setup();
  render(<Shell />);

  expect(screen.queryByRole("navigation")).toBeNull();          // absent: queryBy
  await user.click(screen.getByRole("button", { name: "☰" }));
  expect(screen.getByRole("navigation")).toBeInTheDocument();   // present: getBy
  await user.click(screen.getByRole("button", { name: "☰" }));
  expect(screen.queryByRole("navigation")).toBeNull();
});
2 (easy): This test is flaky on CI. Fix it and name each smell: render(<Users />); fireEvent.click(screen.getByTestId("load-btn")); setTimeout(() => { expect(screen.getByText("Ada")).toBeInTheDocument(); }, 500);
Solution 
test("loads users", async () => {
  const user = userEvent.setup();
  render(<Users />);
  await user.click(screen.getByRole("button", { name: /load/i }));
  expect(await screen.findByText("Ada")).toBeInTheDocument();
});
// Smells: setTimeout assertion (runs AFTER the test ends — asserts nothing,
// then flakes when it half-works); fireEvent for a click a user makes;
// getByTestId where a role+name exists; missing async handling entirely.
3 (medium): Using MSW, test the Data Fetching lesson's useFetch-driven list for all four states: loading skeleton, success, empty ("No users yet"), and 500 → retry button that works.
Solution 
test("success", async () => {
  render(<Users />);
  expect(screen.getByTestId("skeleton")).toBeInTheDocument();   // loading first
  expect(await screen.findByText("Ada")).toBeInTheDocument();   // then data
});

test("empty state", async () => {
  server.use(http.get("/api/users", () => HttpResponse.json([])));
  render(<Users />);
  expect(await screen.findByText(/no users yet/i)).toBeInTheDocument();
});

test("error then retry succeeds", async () => {
  const user = userEvent.setup();
  server.use(http.get("/api/users", () =>                       // fail ONCE:
    HttpResponse.json({}, { status: 500 }), { once: true }));
  render(<Users />);
  await user.click(await screen.findByRole("button", { name: /retry/i }));
  expect(await screen.findByText("Ada")).toBeInTheDocument();   // default handler now
});
// { once: true } makes the override self-expire — the retry hits the original
// success handler: a real recovery test, not a mock of one.
4 (medium): Test the Composition lesson's compound <Tabs>: correct panel shown initially, switches on click, and the misuse guard (Tabs.Tab outside Tabs throws).
Solution 
test("switches panels", async () => {
  const user = userEvent.setup();
  render(
    <Tabs defaultValue="a">
      <Tabs.List>
        <Tabs.Tab id="a">First</Tabs.Tab><Tabs.Tab id="b">Second</Tabs.Tab>
      </Tabs.List>
      <Tabs.Panel id="a">Alpha content</Tabs.Panel>
      <Tabs.Panel id="b">Beta content</Tabs.Panel>
    </Tabs>
  );
  expect(screen.getByText("Alpha content")).toBeInTheDocument();
  expect(screen.queryByText("Beta content")).toBeNull();
  await user.click(screen.getByRole("tab", { name: "Second" }));
  expect(screen.getByText("Beta content")).toBeInTheDocument();
  expect(screen.getByRole("tab", { name: "Second" }))
    .toHaveAttribute("aria-selected", "true");                 // a11y contract too
});

test("guards against misuse", () => {
  expect(() => render(<Tabs.Tab id="x">Lost</Tabs.Tab>))
    .toThrow(/must be used inside /);
});
// Note what ISN'T tested: TabsContext's shape, useState vs useReducer inside —
// free to refactor.
5 (medium): Write the test that would have caught this production bug: the Lists & Keys guest-list bug (index keys + prepend corrupting inputs). Behavior-only.
Solution 
test("row inputs stay with their guest when one is added on top", async () => {
  const user = userEvent.setup();
  render(<GuestList initial={["Ada", "Linus"]} />);

  await user.type(screen.getByRole("textbox", { name: /ada/i }), "vegan");
  await user.click(screen.getByRole("button", { name: /add to top/i }));

  expect(screen.getByRole("textbox", { name: /ada/i })).toHaveValue("vegan");
  expect(screen.getByRole("textbox", { name: /grace/i })).toHaveValue("");
});
// With key={i} this fails exactly as users experienced it: Grace's box shows
// "vegan". No mention of keys anywhere — the test pins the BEHAVIOR, and any
// future keying mistake re-fails it. (Requires labeled inputs — which the
// a11y-first queries pushed you toward anyway.)
6 (hard): Test the optimistic like button (Data Fetching Part 5): instant UI flip, rollback + notice on server failure, and no double-fire on rapid clicks. Use MSW delays.
Solution 
test("optimistic flip, rollback on failure", async () => {
  const user = userEvent.setup();
  server.use(http.post("/api/like", async () => {
    await delay(100);
    return HttpResponse.json({}, { status: 500 });
  }));
  render(<LikeButton postId="1" initialLiked={false} />);

  await user.click(screen.getByRole("button", { name: /like/i }));
  expect(screen.getByRole("button", { name: /unlike/i }))
    .toBeInTheDocument();                                   // flipped BEFORE response

  expect(await screen.findByText(/couldn't save/i)).toBeInTheDocument();
  expect(screen.getByRole("button", { name: /^like/i }))
    .toBeInTheDocument();                                   // rolled back
});

test("rapid clicks send one request", async () => {
  const user = userEvent.setup();
  let calls = 0;
  server.use(http.post("/api/like", async () => {
    calls++; await delay(100); return HttpResponse.json({ ok: true });
  }));
  render(<LikeButton postId="1" initialLiked={false} />);

  await user.click(screen.getByRole("button", { name: /like/i }));
  await user.click(screen.getByRole("button", { name: /unlike/i }));   // in-flight
  await waitFor(() => expect(calls).toBe(1));               // collapsed
});
// The delay() creates the in-flight window the double-click test needs; the
// accessible-name flip doubles as the state assertion. Counting handler calls
// is a BOUNDARY assertion (requests sent), not an implementation detail —
// the server contract is exactly what we're testing.
7 (hard): Your team inherits a 400-test suite: 90 snapshot tests (all updated blindly for months), 120 tests asserting on Redux action dispatches, 40 timing-based flakes, and no error-path coverage. Write the triage plan with effort/payoff reasoning.
Solution 
// 1. DELETE the 90 blind snapshots (hours, instant payoff): they assert
//    nothing (updated reflexively) and slow every refactor. Keep any whose
//    diffs reviewers actually read (likely ~5 serialized-output ones).
// 2. Flakes (40): quarantine immediately (CI trust is the asset), then fix in
//    order of flake-rate: sleeps → findBy/waitFor; shared store/cache state →
//    per-test instances. Un-quarantine as fixed; delete any nobody can explain.
// 3. Action-dispatch tests (120): don't mass-rewrite. Rule going forward: new
//    code gets behavior tests; touch a file → convert its tests. Mass-convert
//    only the ~20 covering money/security flows now — highest regression cost.
//    Many dispatch tests are really reducer tests in disguise → refactor those
//    into plain reducer unit tests (cheap, and they were the valuable part).
// 4. Error paths (missing): add MSW + provider-wrapped flow tests for the top
//    5 user journeys' failure modes first (checkout failure, auth expiry,
//    empty states) — this is where "green suite, weekly incidents" lives.
// 5. Instrument the loop: every production incident lands a failing test
//    before its fix. Six months of that reshapes coverage around reality
//    better than any up-front rewrite budget.
// Net: suite shrinks (~400 → ~300), runs faster, and starts failing for
// reasons that matter — the only metric that counts.

🎯 Key Takeaways

  • Test through the user's senses: role/label queries, userEvent interactions, rendered-output assertions — refactor-proof by construction, a11y-enforcing for free.
  • getBy asserts presence, queryBy absence, findBy async arrival; sleeps and empty waitFors are how flakes are born.
  • Mock at the HTTP seam (MSW) so your whole data layer runs for real; error/empty/slow cases are one handler override each.
  • Layer the pyramid: pure logic (reducers/validators) exhaustively as unit tests; component behavior through real providers with a renderWithProviders util (fresh cache/store per test); a thin E2E crust.
  • Blind snapshots and implementation assertions are negative-value tests — deleting them is a quality improvement.
  • The suite's job: fail when users would notice, stay green when they wouldn't — and grow a regression test from every production incident.
