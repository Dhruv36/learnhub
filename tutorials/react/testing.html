Testing React Advanced

The philosophy that reshaped frontend testing: test what the user experiences, not how the component is built. Tests that query by visible text/roles survive refactors; tests that poke internals break on every rename while catching nothing. Stack: Vitest (runner) + React Testing Library (render/query/interact) + MSW (network mocking) + Playwright (E2E).

The RTL Model

import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

test("adds a todo", async () => {
  const user = userEvent.setup();
  render(<TodoApp />);
  await user.type(screen.getByRole("textbox", { name: /new todo/i }), "Buy milk");
  await user.click(screen.getByRole("button", { name: /add/i }));
  expect(screen.getByText("Buy milk")).toBeInTheDocument();
});
// Renders into a real (js)DOM; queries find things THE WAY USERS DO (role, label, text);
// userEvent simulates real interaction sequences (focus, keydown, input…), not bare fireEvent.

Queries — the Priority Order (memorize this)

PriorityQueryWhy
1getByRole("button", { name: /save/i })How assistive tech sees the page — testing + a11y in one move (roles come from semantic HTML — the HTML track pays off here)
2getByLabelText / getByPlaceholderTextForm fields as users identify them
3getByTextNon-interactive content
last resortgetByTestIdNo accessible handle exists — often a smell that the markup lacks semantics
getBy…    exists NOW (throws if not)         queryBy…  returns null (assert ABSENCE)
findBy…   await it to APPEAR (async UI)       …AllBy    the list versions
expect(screen.queryByText("Error")).not.toBeInTheDocument();       // absence
expect(await screen.findByText("Saved!")).toBeInTheDocument();     // appears after async

Mocking the Network — MSW (the game changer)

// Mock at the NETWORK boundary, not the fetch function — components run their REAL code:
import { setupServer } from "msw/node";
import { http, HttpResponse } from "msw";

const server = setupServer(
  http.get("/api/users", () => HttpResponse.json([{ id: 1, name: "Asha" }]))
);
beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

test("shows users and handles errors", async () => {
  render(<UserList />);
  expect(await screen.findByText("Asha")).toBeInTheDocument();

  server.use(http.get("/api/users", () => HttpResponse.error()));   // per-test override
  render(<UserList />);
  expect(await screen.findByRole("alert")).toHaveTextContent(/failed/i);
});
// vs vi.mock(fetch): MSW tests the full pipeline (URL, params, parsing, error branches),
// works identically in tests, Storybook, and dev — one mock definition everywhere.

What to Test (and What Not To)

TestSkip
User flows: type → click → see resultImplementation: state values, method calls, "renders without crashing"
Both branches: success AND error/empty/loading UIThird-party internals (the router routes, Query caches)
Edge cases via props: empty list, long text, disabledStyling details (visual tools do that better)
Custom hooks with logic (renderHook — hooks lesson)Trivial pass-through components
Accessibility basics fall out FREE of role queriesSnapshot-everything (churn without meaning; snapshot small, stable output only)

The trophy 🏆 (modern testing shape): some static (TS+ESLint catch most bugs) → few unit (pure logic: reducers, utils) → MANY integration (RTL component flows — best cost/confidence) → few E2E (Playwright: the critical paths — login, checkout — in a real browser).

Interview Questions

Q1: Why does RTL deliberately make it hard to access component internals?

Because tests coupled to internals verify the implementation, not the behavior — they break on refactors (false negatives) and pass on broken UX (false positives). Querying the rendered, accessible output means tests fail exactly when a USER would notice. "The more your tests resemble the way your software is used, the more confidence they give you" — the guiding principle.

Q2: MSW vs vi.mock('axios') — argue the trade.

Function mocks stub your OWN layer: request building, params, response parsing, error mapping all go untested, and mocks drift from reality. MSW intercepts at the network boundary: components execute their real data path against declared API behavior, error/latency cases are one-line overrides, and the same handlers serve tests/dev/Storybook. Cost: slightly more setup — worth it everywhere beyond trivial.

Q3: Design the test suite for a checkout form (validation, submit, server errors).

Integration (RTL+MSW): happy path (fill valid → submit → success UI + assert request payload via MSW), each validation rule (blur → message appears, role="alert"), server rejection (MSW 400 → field errors shown), double-submit disabled state. Unit: the validate() function's edge cases (pure — cheap to exhaust). E2E (Playwright): ONE full checkout against staging. Ratio matches the trophy; role-based queries double as an a11y audit.

🏋️ Practice: set up Vitest+RTL+MSW; test the TodoApp flow (add/toggle/empty state), a UserList with success+error MSW handlers, and one custom hook via renderHook; deliberately write an internals-coupled test, refactor the component, and watch which test survives.

🎯 Key Takeaways

  • Test behavior users see; query by role/label/text (testid last) — a11y auditing falls out free.
  • userEvent + await + findBy for async; queryBy for absence; getBy for presence-now.
  • MSW mocks the network boundary — real component code runs; per-test overrides for errors.
  • The trophy: types+lint → few unit (pure logic) → many integration (RTL) → few E2E (Playwright).
  • Coverage of flows and branches beats line-percentage worship; snapshots only for small stable output.
