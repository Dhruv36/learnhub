Concurrent Features Performance

What you'll master: the urgency model behind concurrent React — some updates must be instant, others can lag — startTransition/useTransition, useDeferredValue, how transitions interact with Suspense (no-fallback navigation), and when these tools beat debouncing. Prerequisites: The Render Cycle (interruptible rendering — this is why purity mattered), Suspense.

Part 1 — The Problem: One Update, Two Urgencies

// Type into a filter box over a 20,000-row list:
function FilterList({ rows }) {
  const [query, setQuery] = useState("");
  const visible = expensiveFilter(rows, query);        // 80ms of work
  return (<>
    <input value={query} onChange={e => setQuery(e.target.value)} />
    <Rows items={visible} />
  </>);
}
// Every keystroke: ONE setState → ONE 80ms render that updates BOTH the input
// text and the list. The input can't show the letter until the list is done —
// typing feels broken. But the two parts have different deadlines:
//   input echo  → must be THIS frame (urgent)
//   list update → 100ms late is fine (deferrable)
// Debouncing delays EVERYTHING (input included if naively done, results always).
// Concurrent rendering instead runs them at two priorities: urgent renders
// synchronously; deferred renders in the background, INTERRUPTIBLE by the next
// keystroke — abandoned mid-render if stale. (Purity made this legal.)

Part 2 — useTransition: Marking Updates Deferrable

import { useTransition } from "react";

function FilterList({ rows }) {
  const [query, setQuery] = useState("");          // urgent state (the input)
  const [filterQ, setFilterQ] = useState("");      // deferred state (the list)
  const [isPending, startTransition] = useTransition();

  function onChange(e) {
    setQuery(e.target.value);                      // urgent: renders this frame
    startTransition(() => {                        // transition: low priority
      setFilterQ(e.target.value);                  // rendered in background,
    });                                            // restarted if a new keystroke lands
  }

  const visible = useMemo(() => expensiveFilter(rows, filterQ), [rows, filterQ]);

  return (<>
    <input value={query} onChange={onChange} />
    <div style={{ opacity: isPending ? 0.6 : 1 }}>  {/* stale-but-visible signal */}
      <Rows items={visible} />
    </div>
  </>);
}
// Typing "react" fast: 5 urgent renders (input instant every time), and the
// transition render RESTARTS per keystroke — only the final "react" list ever
// commits. isPending styles the stale list instead of hiding it.
//
// The API shape: startTransition(callback) — the STATE UPDATES inside the
// callback are marked low-priority. The callback itself runs synchronously.
// Transitions don't make the 80ms cheaper — they make it interruptible and
// keep it off the critical path. (If the work itself must shrink: virtualize,
// memoize — the previous lessons. These tools COMPOSE.)

Part 3 — useDeferredValue: the Same Idea, Value-Shaped

// When you don't control the setState (the value arrives as a prop), defer the VALUE:
function SearchResults({ query }) {                  // parent updates query urgently
  const deferredQuery = useDeferredValue(query);     // lags behind during load
  const results = useMemo(() => expensiveSearch(deferredQuery), [deferredQuery]);
  const isStale = deferredQuery !== query;           // the pending signal, derived

  return <div style={{ opacity: isStale ? 0.6 : 1 }}><Rows items={results} /></div>;
}
// Mechanics: urgent render happens with the OLD deferred value (cheap — memo
// hit); then a background render with the new value; interrupted if query
// changes again. useMemo on the deferred value is LOAD-BEARING: without it the
// urgent render re-runs the expensive work anyway and you've deferred nothing.
//
// Choosing:
// useTransition    — you own the setState; you get isPending; can wrap several updates.
// useDeferredValue — you receive a value (prop/context); minimal wiring.
// Both > debounce for CPU-bound work: no fixed delay (fast machines get instant
// results), auto-cancellation, pending signals. Debounce still wins for
// NETWORK-bound work — it reduces REQUESTS (Data Fetching ex. 5); deferral only
// schedules renders. Combine: debounce the fetch, defer the render.

Part 4 — Transitions + Suspense: Navigation Without Blanking

// The default Suspense behavior on NEW data: component suspends → fallback shows.
// During a TRANSITION, React instead keeps the CURRENT UI mounted while the
// next tree loads in the background — no fallback flash:

function App() {
  const [tab, setTab] = useState("home");
  const [isPending, startTransition] = useTransition();
  function switchTab(next) {
    startTransition(() => setTab(next));     // suspending? keep current tab visible
  }
  return (<>
    <TabBar active={tab} onSwitch={switchTab} pending={isPending} />
    <Suspense fallback={<PageSkeleton />}>
      {tab === "home" ? <Home /> : <Analytics />}   {/* lazy + data-suspending */}
    </Suspense>
  </>);
}
// Without the transition: click → whole area blanks to PageSkeleton → new tab.
// With it: click → old tab stays (TabBar shows a pending indicator) → new tab
// swaps in complete. The skeleton only appears on FIRST loads (nothing to keep).
// React Router's data navigation wraps route changes in transitions for exactly
// this reason — the Routing lesson's "previous page stays interactive" behavior
// is this mechanism.
// Rules of transitions: the update must be non-urgent AND allowed to be
// inconsistent with the input for a moment. Never wrap controlled-input value
// updates (the echo lags = typing feels broken — the exact bug we started with).

Part 5 — The Priority Model (Senior Layer)

// What "concurrent" concretely means in React 18/19:
// • Urgent lane: clicks, typing, sync setState — rendered synchronously, blocking.
// • Transition lane: startTransition/useDeferredValue — rendered in chunks,
//   YIELDING to the main thread between units of work (time-slicing), so input
//   events can land mid-render; a newer update in the same lane ABORTS and
//   restarts the render with fresh state.
// • Committed atomically: users never see a half-rendered tree (Render Cycle's
//   guarantee) — interruption happens between renders and commits, not mid-DOM.
//
// What it is NOT:
// • Not parallelism — one thread, cooperative scheduling.
// • Not faster renders — the same work, better scheduled. CPU cost unchanged.
// • Not automatic — legacy setState stays urgent; you opt updates into the
//   slow lane. (Frameworks/routers increasingly do the opting for you.)
//
// Effects & transitions: effects for a transition render run after ITS commit,
// normally. Renders that were abandoned never commit → their effects never run —
// another reason render purity was non-negotiable.
// Debugging: React DevTools Profiler shows transition renders separately; a
// "laggy" deferred UI usually means the expensive part isn't actually inside
// the deferred subtree (memo missing → urgent render still pays it).

Interview Questions

Beginner: What problem do transitions solve?

A single state change often drives both urgent UI (input echo, click feedback) and expensive UI (a filtered 20k-row list). Rendered together, the expensive part blocks the urgent part and interaction feels frozen. startTransition marks the expensive updates low-priority: urgent renders commit immediately; the transition renders in the background, interruptible and restartable when newer input arrives.

Beginner: useTransition vs useDeferredValue?

Same scheduling, different ergonomics. useTransition wraps state updates you make and hands you isPending. useDeferredValue takes a value you receive (prop, context) and returns a lagging copy — for when you don't own the setter. Staleness with deferred values is derived: deferred !== current.

Mid: Why is useMemo essential next to useDeferredValue?

The urgent render re-runs the component with the old deferred value. Without memoization, the expensive computation runs again right there — on the urgent path — and deferral bought nothing. With useMemo keyed on the deferred value, the urgent render is a cache hit (old value → old result), and only the background render pays for the new computation. Deferral schedules renders; memo makes the urgent one cheap.

Mid: How do transitions change Suspense behavior, and why does routing rely on it?

Outside a transition, a suspending update swaps committed UI for the boundary's fallback. Inside one, React keeps the current committed tree on screen while the suspended next tree readies in the background — fallbacks only appear for first-time mounts with nothing to keep. Routers wrap navigation in transitions so page changes never blank the app: the old page stays interactive, a pending bar shows, and the new page commits whole.

Senior: Transitions vs debouncing vs virtualization for a slow search-as-you-type — architect the full answer.

Different layers, all three: virtualize the results list so rendering N rows costs ~25 rows — shrink the work (largest fix). Debounce the network request (~300ms) so keystrokes don't multiply server load — shrink the I/O. Transition/defer the results-state update so whatever render cost remains never blocks the input — schedule the work. Debouncing alone adds fixed latency on fast machines and can't help CPU-bound filtering; transitions alone still burn full CPU per keystroke; virtualization alone still couples input to one big render. Ordering by impact: measure, virtualize, debounce I/O, defer the rest, verify input latency in the Profiler on a throttled CPU.

Senior: What made concurrent rendering possible only after years of React development, and what breaks it in userland?

Interruptible rendering requires that running a component produce no observable effects — otherwise an abandoned render leaves half-applied changes. Hence the long groundwork: render purity rules, effects moved post-commit, Strict Mode double-invocation to flush violations, and useSyncExternalStore for tear-free external reads. Userland breakage: render-time side effects and mutations (run twice or in abandoned renders), reading mutable external sources directly during render (tearing — different components see different snapshots mid-slice), and effects that assume every render commits. Code that follows the rules gets concurrency free; code that doesn't produces the subtlest bug class in modern React.

🏋️ Graded Exercises

1 (easy): A tag-filter sidebar: clicking a tag filters a heavy grid. Make the click feel instant with useTransition, with the active tag highlighting immediately.
Solution 
function TagFilter({ tags }) {
  const [active, setActive] = useState(null);        // urgent: highlight
  const [gridTag, setGridTag] = useState(null);      // deferred: the grid
  const [isPending, startTransition] = useTransition();

  function pick(tag) {
    setActive(tag);                                  // this frame
    startTransition(() => setGridTag(tag));          // background
  }
  return (<>
    {tags.map(t => <button key={t} className={t === active ? "on" : ""}
                           onClick={() => pick(t)}>{t}</button>)}
    <HeavyGrid tag={gridTag} dimmed={isPending} />
  </>);
}
2 (easy): Why does this still jank? const deferred = useDeferredValue(query); const rows = bigFilter(data, deferred);
Solution 
// bigFilter runs on EVERY render — including the urgent one that re-renders
// with the OLD deferred value. Old value or not, the filter itself is the 80ms.
// Fix: const rows = useMemo(() => bigFilter(data, deferred), [data, deferred]);
// Now the urgent render is a cache hit and only the background render computes.
3 (medium): Add a "results may be outdated" treatment to a deferred search: dim + an inline note, but only when staleness lasts > 300ms (no flicker on fast machines).
Solution 
function Results({ query }) {
  const deferred = useDeferredValue(query);
  const isStale = deferred !== query;
  const showNote = useDelayedFlag(isStale, 300);     // Data Fetching lesson's hook

  const rows = useMemo(() => search(deferred), [deferred]);
  return (
    <div style={{ opacity: isStale ? 0.6 : 1 }}>
      {showNote && <p role="status">Updating results…</p>}
      <Rows items={rows} />
    </div>
  );
}
// Dimming is instant (cheap, subtle); the textual note waits out fast updates.
// role="status" announces the update politely to screen readers.

5 (medium): A live dashboard receives websocket ticks that update a big chart, and users complain dropdown menus stutter during bursts. Apply the urgency model without touching the socket code.
Solution 
// Tick updates are deferrable (a chart 100ms behind is invisible); menu
// interaction is urgent. Route the socket's setState through a transition:
useEffect(() => {
  const unsub = socket.subscribe(batch => {
    startTransition(() => setChartData(batch));    // background lane
  });
  return unsub;
}, []);
// Bursts now yield to input events; a superseding batch aborts the stale render
// (only the freshest commits — free coalescing). If bursts exceed render
// capacity, ALSO rAF-batch upstream (External Stores lesson ex.) — scheduling
// and volume-reduction compose.
6 (hard): Build useSearchExperience(fetchFn): input value (urgent), debounced fetch (300ms, race-safe), deferred rendering of results, and a single status for the UI ("typing" | "fetching" | "stale-render" | "ready"). Compose only hooks from this track.
Solution 
function useSearchExperience(fetchFn) {
  const [input, setInput] = useState("");                 // urgent
  const debounced = useDebouncedValue(input, 300);        // Effects lesson
  const query = useFetch(debounced ? fetchFn(debounced) : null);   // Data Fetching
  const results = query.status === "success" ? query.data : [];
  const deferredResults = useDeferredValue(results);      // this lesson
  const rows = useMemo(() => deferredResults, [deferredResults]);

  const status =
    input !== debounced          ? "typing"        // debounce window open
    : query.status === "loading" ? "fetching"      // network in flight
    : deferredResults !== results ? "stale-render" // render lagging data
    : "ready";

  return { input, setInput, rows, status };
}
// Each layer handles its own latency: debounce (network volume), abort inside
// useFetch (race), deferral (render scheduling). The status union makes the
// three kinds of "waiting" distinguishable — a senior UX detail: typing shows
// nothing, fetching shows a spinner, stale-render just dims.
7 (hard): A teammate reports: "I wrapped our whole form submit in startTransition so the modal close feels faster, but now sometimes the success toast shows before the server call finishes, and under Strict Mode things double-fire." Untangle every misconception and rewrite correctly.
Solution 
// Misconceptions:
// 1. Transitions schedule RENDERS, not async work — wrapping an await-chain in
//    startTransition doesn't background the network call; code after await also
//    loses transition status (the callback's sync portion is what's marked).
// 2. The toast raced because they optimistically set success state before the
//    await — a state-machine problem (Reducers lesson), not a scheduling one.
// 3. Strict Mode double-fires point at side effects during render or missing
//    cleanup — transitions never cause double network calls; impure code does.
// Correct shape:
async function onSubmit(data) {
  dispatch({ type: "save_started" });               // urgent: button → spinner
  try {
    await api.save(data);                           // plain async — not a render concern
    dispatch({ type: "save_succeeded" });           // toast keyed to THIS event
    startTransition(() => {
      navigate("/list");                            // the actually-deferrable part:
    });                                             // suspending nav w/o blanking
  } catch (e) {
    dispatch({ type: "save_failed", error: e });
  }
}
// Transitions own one job here: the post-success navigation that may suspend.
// Sequencing lives in the handler's state machine; async lives in await.

🎯 Key Takeaways

  • The urgency model: input echo and click feedback are urgent; expensive derived UI can lag — concurrent React renders them in separate lanes.
  • startTransition/useTransition mark your own setStates deferrable (with isPending); useDeferredValue defers a value you receive — memoized derivation is mandatory or nothing is deferred.
  • Transition renders are interruptible and restartable: rapid input aborts stale background work, and only the freshest state commits — atomic, never half-rendered.
  • Transitions + Suspense = no-fallback updates: current UI stays during loads; skeletons only on first mounts. This is how modern routing feels instant.
  • Deferral schedules CPU work; debounce reduces network volume; virtualization/memo shrink the work itself — compose all three, in that diagnostic order.
  • The whole machine rests on render purity — impure renders are the bug class concurrency exposes.
