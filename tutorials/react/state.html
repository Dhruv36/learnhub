State & useState Foundations

What you'll master: what state actually is (a snapshot per render, not a live variable), the useState API completely — lazy initialization, functional updates, batching — immutable update patterns for objects and arrays, choosing what belongs in state at all, and the closure model that explains every "stale state" bug you'll ever hit. Prerequisites: Props and JS Closures — state "gotchas" are closures wearing a React costume.

Part 1 — Why State Exists, and Where It Lives

Props come from the parent; but something must remember across renders — the typed text, the open/closed flag, the fetched list. Local variables can't: every render calls your function fresh, and all its variables are recreated. useState asks React to store a value outside your function, on the component's fiber, and hand it back on every render:

import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);     // [current value, updater], initial 0
  //     ↑ this render's snapshot   ↑ the ONLY way to change it

  return <button onClick={() => setCount(count + 1)}>Clicked {count}×</button>;
}
// Click → setCount(1) → React re-renders Counter → useState returns 1 this time.
// Two Counters on a page = two independent fibers = two independent counts.
The snapshot model — internalize this sentence: count is a constant within one render. Calling setCount never changes count; it schedules a re-render in which useState will return the new value to a new count constant. State reads are photographs; setters order the next photograph taken.

Part 2 — The Snapshot Model, Proven

function Demo() {
  const [n, setN] = useState(0);

  function handleClick() {
    setN(n + 1);      // n is 0 → schedules 1
    setN(n + 1);      // n is STILL 0 in this snapshot → schedules 1 again
    setN(n + 1);      // still 0 → schedules 1
    console.log(n);   // logs 0 — this render's photograph never changes
  }
  // After the click: n === 1, not 3. All three calls computed 0 + 1.
}

// The fix — FUNCTIONAL UPDATES receive the latest pending value:
setN(prev => prev + 1);   // 0 → 1
setN(prev => prev + 1);   // 1 → 2
setN(prev => prev + 1);   // 2 → 3  ✅ updater functions queue and chain
// Same model, async flavor — the interview classic:
function Timer() {
  const [sec, setSec] = useState(0);
  useEffect(() => {
    const id = setInterval(() => setSec(sec + 1), 1000);   // ❌ closes over sec === 0
    return () => clearInterval(id);
  }, []);                                                   // runs once → sec stuck at 1
}
// The interval callback is a closure over the FIRST render's sec (0), so it sets 1 forever.
// Fix: setSec(s => s + 1) — ask React for the current value instead of trusting the closure.

Rule of thumb: if the next value is computed from the previous value, always use the functional form. It's immune to snapshots, batching, and stale closures. If the next value is independent (setOpen(true), setQuery(input)), the plain form is fine.

Part 3 — Batching, Lazy Init & the Bail-Out

// 1. BATCHING: multiple setters in one event = ONE re-render (React 18+: also in
//    timeouts, promises, native handlers — "automatic batching"):
function submit() {
  setSaving(true);
  setError(null);
  setAttempts(a => a + 1);
}                                // → a single render sees all three new values

// 2. LAZY INITIALIZATION: useState(expensive()) runs expensive() EVERY render
//    (its result ignored after the first). Pass a function to run it once:
const [data, setData] = useState(() => JSON.parse(localStorage.getItem("draft") ?? "{}"));
//                                ↑ initializer function — called only on mount

// 3. BAIL-OUT: setting state to the SAME value (Object.is) skips the re-render:
setCount(0);   // if count is already 0 → no render
// …but with objects, "same" means same REFERENCE. A fresh {} is always "different":
setUser({ ...user });   // re-renders even though contents are identical

Part 4 — Immutable Updates: Objects & Arrays

React detects change by reference (Object.is on the state value). Mutate in place and the reference survives → React bails out → no re-render. Every update must produce a new object/array at each changed level:

const [form, setForm] = useState({ name: "", address: { city: "", zip: "" } });

// ✅ update a field — copy the object, override one key:
setForm({ ...form, name: "Ada" });

// ✅ NESTED update — copy every level on the path:
setForm({ ...form, address: { ...form.address, city: "Pune" } });

// ❌ the classic silent failure:
form.address.city = "Pune";   // mutates…
setForm(form);                // …same reference → bail-out → UI never updates
Operation❌ Mutates✅ Immutable
Add itemarr.push(x)[...arr, x]
Remove itemarr.splice(i, 1)arr.filter(a => a.id !== id)
Update itemarr[i].done = truearr.map(a => a.id === id ? { ...a, done: true } : a)
Insert at iarr.splice(i, 0, x)[...arr.slice(0, i), x, ...arr.slice(i)]
Sortarr.sort(f)arr.toSorted(f) (or copy first)
Replace fieldobj.k = v{ ...obj, k: v }
Spread is shallow — and that's by design. {...form} copies one level; nested objects are shared by reference until you copy along the changed path. That's efficient (unchanged branches keep their identity, so memoized children skip re-renders) but means deep updates need a copy per level. When nesting makes this painful, that's a signal: flatten the state, or move to useReducer (optionally with Immer).

Part 5 — What Belongs in State (and What Doesn't)

// THE #1 STATE SMELL — storing what you can compute:
const [items, setItems]   = useState([]);
const [total, setTotal]   = useState(0);      // ❌ derivable from items
const [hasAny, setHasAny] = useState(false);  // ❌ derivable from items

// Every code path that updates items must remember to update total and hasAny — some won't.
// ✅ Derive during render. Renders are cheap; sync bugs are not:
const [items, setItems] = useState([]);
const total  = items.reduce((s, i) => s + i.price, 0);
const hasAny = items.length > 0;
CandidateState?Instead
User input, toggles, selection✅ yes—
Server data you fetched✅ yes (or a data library)Data Fetching
Anything computable from props/state❌ noDerive in render (useMemo only if measured-slow)
A prop you just want to rename❌ noUse the prop

Structure heuristics: group state that changes together ({x, y} for a position, not two setters you must keep in sync); split state that changes independently (form text vs. modal open — one giant object means every keystroke "changes" everything); avoid contradictions by construction (status: "idle"|"loading"|"error" instead of isLoading + isError both true).

Part 6 — Build It: an Editable Todo List (Every Pattern Together)

function TodoApp() {
  const [todos, setTodos] = useState(() =>                     // lazy init from storage
    JSON.parse(localStorage.getItem("todos") ?? "[]"));
  const [filter, setFilter] = useState("all");                 // independent → separate state

  const visible = filter === "done" ? todos.filter(t => t.done)  // derived — NOT state
                : filter === "open" ? todos.filter(t => !t.done)
                : todos;
  const openCount = todos.filter(t => !t.done).length;           // derived — NOT state

  const add    = text => setTodos(ts => [...ts, { id: crypto.randomUUID(), text, done: false }]);
  const toggle = id   => setTodos(ts => ts.map(t => t.id === id ? { ...t, done: !t.done } : t));
  const remove = id   => setTodos(ts => ts.filter(t => t.id !== id));
  // All functional updates: correct even if two updates land in one batch.

  return (
    <>
      <AddForm onAdd={add} />
      <FilterBar value={filter} onChange={setFilter} />
      <ul>
        {visible.map(t => (
          <TodoRow key={t.id} todo={t} onToggle={toggle} onRemove={remove} />
        ))}
      </ul>
      <footer>{openCount} remaining</footer>
    </>
  );
}
// State lives in ONE owner; children are stateless prop/callback components (Props lesson).
// Persisting todos back to localStorage belongs in an effect — next section of the track.

Interview Questions

Beginner: Why doesn't count update immediately after setCount(count + 1)?

State is a per-render snapshot. The setter doesn't assign a variable — it schedules a re-render; only that next render's useState call returns the new value. Within the current function body, count is a constant photograph of what state was when the render started.

Beginner: When must you use the functional update form?

Whenever the next state is computed from the previous state — increments, toggles, appending to lists — especially with multiple updates per event, async callbacks (intervals, promises), or anywhere a stale closure could hold an old snapshot. setX(prev => …) always receives the latest queued value.

Mid: What is automatic batching in React 18, and what changed from 17?

Batching collapses multiple state updates into one re-render. React 17 batched only inside React event handlers; updates in setTimeout, promise callbacks, or native listeners each triggered their own render. React 18's createRoot batches everywhere by default. Escape hatch for the rare must-read-DOM-between-updates case: flushSync.

Mid: Why does mutating state break React, mechanically?

React decides "did state change?" with Object.is on the state value, and downstream optimizations (memo, dependency arrays) compare references shallowly. In-place mutation preserves the reference, so the setter bails out without rendering, memoized children see "equal" props, and effects don't fire. The data moved but every change detector said no.

Senior: A senior reviewer sees const [fullName, setFullName] = useState(first + " " + last) alongside state for first and last. What's the review comment?

Redundant state: fullName is derivable, and it silently freezes — updating first won't refresh it since initializers run only at mount, so two sources of truth drift. Delete the state and derive const fullName = `${first} ${last}` in render; memoize only if profiling ever shows the derivation is hot (it won't be for string concat).

Senior: How would you model fetch status to make invalid states unrepresentable?

Replace independent booleans (isLoading, isError, data) — which permit contradictions like loading+error simultaneously — with a discriminated union: {status:"idle"} | {status:"loading"} | {status:"success", data} | {status:"error", error}. One field to branch on, TypeScript narrows each arm, and transitions become explicit — which is the doorway to useReducer/state machines when events multiply.

4 (medium): A profile form has firstName, lastName, email. Implement it with ONE state object and a single generic change handler.
Solution 
function Profile() {
  const [form, setForm] = useState({ firstName: "", lastName: "", email: "" });
  const onChange = e =>
    setForm(f => ({ ...f, [e.target.name]: e.target.value }));   // computed key from name=

  return (
    <form>
      <input name="firstName" value={form.firstName} onChange={onChange} />
      <input name="lastName"  value={form.lastName}  onChange={onChange} />
      <input name="email"     value={form.email}     onChange={onChange} />
    </form>
  );
}
// Grouping is right here: the fields are one cohesive record submitted together.

6 (hard): Build an undo-able counter: +/− buttons and an Undo button that steps back through history. Design the state shape first.
Solution 
function UndoCounter() {
  // ONE state: the history stack. Current value = derived (last element).
  const [history, setHistory] = useState([0]);
  const value = history[history.length - 1];

  const apply = delta => setHistory(h => [...h, h[h.length - 1] + delta]);
  const undo  = ()    => setHistory(h => h.length > 1 ? h.slice(0, -1) : h);

  return (
    <div>
      <button onClick={() => apply(-1)}>−</button>
      <b>{value}</b>
      <button onClick={() => apply(1)}>+</button>
      <button onClick={undo} disabled={history.length === 1}>Undo</button>
    </div>
  );
}
// Storing value AND history separately would be redundant state — the stack is the truth.
7 (hard): Explain precisely why this stopwatch sticks at 1, then fix it two different ways: useEffect(() => { const id = setInterval(() => setSec(sec + 1), 1000); return () => clearInterval(id); }, []);
Solution 
// The effect runs once (deps []). Its callback closes over the mount render's
// sec (0). Every tick computes 0 + 1 → sets 1 → re-renders — but the interval
// callback is never recreated, so it keeps reading the old snapshot forever.
// Fix A (best): functional update — no dependence on the closure:
setInterval(() => setSec(s => s + 1), 1000);
// Fix B: make the closure fresh — add sec to deps so the effect re-subscribes
// each second: useEffect(…, [sec]) with clearInterval in the cleanup.
// (Works, but tears down/recreates the interval every tick — A is cleaner.)

🎯 Key Takeaways

  • State is a per-render snapshot on the fiber: setters never mutate the current render's value — they schedule the next render, whose useState returns the new value.
  • setX(prev => …) whenever next depends on previous — it chains through batches and ignores stale closures; plain form only for independent values.
  • React 18 batches all updates in one pass into one render; same-value sets (Object.is) bail out — which is why in-place mutation renders nothing.
  • Immutable updates: new reference per changed level — spread objects, map/filter/toSorted arrays; deep pain is a signal to flatten or move to useReducer.
  • Minimal state: store only what can't be derived; group what changes together, split what doesn't; encode status as one enum, not boolean soup.
  • Lazy-initialize expensive starts with useState(() => …); mutable non-visual values go in refs, not state.
