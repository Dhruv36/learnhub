External Stores (Redux, Zustand) State Management

What you'll master: what an external store fundamentally is (state outside the tree + subscriptions), useSyncExternalStore — the primitive everything is built on — selectors as the re-render precision tool, Zustand and Redux Toolkit in practice, and the decision framework for context vs store vs query cache. Prerequisites: Reducers, Context (especially its performance ceiling), State Architecture.

Part 1 — The Idea: State That Outlives and Outsits the Tree

// Everything so far lived ON a fiber (useState/useReducer) and traveled through
// the tree (props/context). An external store flips it: a plain object in module
// scope, with React components SUBSCRIBING to the parts they read.
//
// Why bother?
// 1. PRECISION: subscribers pick a SLICE (selector) — only components whose
//    slice changed re-render. Context's ceiling (any change wakes all consumers) lifts.
// 2. LIFETIME: the store survives unmounts and exists before mount — non-React
//    code (websocket handlers, router guards, analytics) can read/write it.
// 3. NO PROVIDER PYRAMID: import the hook, use it — any component, any depth.
//
// The cost: state leaves React's world — devtools, purity guarantees, and scoping
// discipline now depend on the store library and your conventions.

Part 2 — The Primitive: useSyncExternalStore

// React 18's official bridge to ANY external source. Build a tiny store with it:
function createStore(initialState) {
  let state = initialState;
  const listeners = new Set();
  return {
    getState: () => state,
    setState: updater => {
      state = typeof updater === "function" ? updater(state) : updater;
      listeners.forEach(l => l());                    // notify AFTER swap
    },
    subscribe: listener => {                          // returns unsubscribe —
      listeners.add(listener);                        // the same contract as
      return () => listeners.delete(listener);        // addEventListener cleanup
    },
  };
}

const counterStore = createStore({ count: 0, step: 1 });

// The hook — three arguments, precise contracts:
function useStore(store, selector) {
  return useSyncExternalStore(
    store.subscribe,                        // how to be notified
    () => selector(store.getState()),       // how to read a SNAPSHOT (client)
    () => selector(store.getState())        // server snapshot (SSR — same here)
  );
}

function CountDisplay() {
  const count = useStore(counterStore, s => s.count);   // slice: count only
  return <b>{count}</b>;
}
function StepPicker() {
  const step = useStore(counterStore, s => s.step);     // slice: step only
  return <input value={step} onChange={e =>
    counterStore.setState(s => ({ ...s, step: +e.target.value }))} />;
}
// Incrementing count re-renders CountDisplay ONLY — StepPicker's selected value
// (compared by Object.is between notifications) didn't change. THAT precision is
// the entire pitch. Also: browser APIs are "stores" too — useSyncExternalStore
// is the right way to bind navigator.onLine, matchMedia, localStorage events.
Why not useState+useEffect to mirror a store? Tearing: during concurrent renders, different components could read the store at different moments and paint inconsistent frames. useSyncExternalStore guarantees all subscribers in one render see one snapshot, and forces synchronous re-render if the store changes mid-render. That's why it exists as a primitive instead of a recipe — and it's the interview answer for "what problem does it solve": consistency under concurrency.

Part 3 — Zustand: the Minimal Production Store

import { create } from "zustand";

const useCartStore = create((set, get) => ({
  items: [],                                             // state and actions co-live
  add: product => set(state => {
    const existing = state.items.find(i => i.id === product.id);
    return existing
      ? { items: state.items.map(i => i.id === product.id
          ? { ...i, qty: Math.min(i.qty + 1, i.stock) } : i) }
      : { items: [...state.items, { ...product, qty: 1 }] };
  }),
  remove: id => set(state => ({ items: state.items.filter(i => i.id !== id) })),
  clear: () => set({ items: [] }),                       // set merges top-level keys
  total: () => get().items.reduce((s, i) => s + i.price * i.qty, 0),
}));

// Components subscribe with selectors — same precision as Part 2, zero ceremony:
function CartBadge() {
  const count = useCartStore(s => s.items.length);       // renders only when length changes
  return <span className="badge">{count}</span>;
}
function AddButton({ product }) {
  const add = useCartStore(s => s.add);                  // actions are stable refs:
  return <button onClick={() => add(product)}>Add</button>;  // this NEVER re-renders
}
// Outside React (websocket handler, route guard):
useCartStore.getState().clear();

// Selector discipline — the one Zustand trap:
const { items, add } = useCartStore();                   // ❌ no selector = whole store,
                                                         //    re-renders on ANY change
const [items, add] = useCartStore(useShallow(s => [s.items, s.add]));  // ✅ multi-slice:
// a fresh array/object per call needs shallow comparison, or it "changes" every time.

Notice the reducer lesson paying rent: the add rules (dedupe, stock cap) moved into the store verbatim. Zustand is "reducer thinking without action ceremony" — mutations are named methods. Middleware adds the production extras: persist (localStorage), devtools (Redux DevTools), immer (draft mutation for deep updates).

Part 4 — Redux Toolkit: the Same Ideas, Institutionalized

import { createSlice, configureStore } from "@reduxjs/toolkit";

const cartSlice = createSlice({
  name: "cart",
  initialState: { items: [] },
  reducers: {
    added(state, action) {                       // Immer inside: "mutation" is safe here
      const existing = state.items.find(i => i.id === action.payload.id);
      if (existing) existing.qty = Math.min(existing.qty + 1, existing.stock);
      else state.items.push({ ...action.payload, qty: 1 });
    },
    removed(state, action) {
      state.items = state.items.filter(i => i.id !== action.payload);
    },
  },
});

export const store = configureStore({ reducer: { cart: cartSlice.reducer } });
//  at the root; then:
const items = useSelector(s => s.cart.items);            // selector, same precision idea
const dispatch = useDispatch();
dispatch(cartSlice.actions.added(product));              // generated action creators

// What Redux adds over Zustand — and what it costs:
// + Action log & time-travel devtools: every change is a serializable event —
//   auditability for complex apps and support debugging.
// + One store, enforced structure (slices), a huge middleware/ecosystem, RTK Query.
// + Serializable-state discipline → SSR hydration, persistence, replay.
// − Provider + slice + dispatch ceremony; a learning curve your juniors pay;
//   boilerplate pressure that tempts teams into SET_FIELD anti-patterns (Reducers lesson).
// RTK is the right default where the EVENT LOG matters or the org already runs
// Redux. Zustand where you want store precision with library-free ergonomics.

Interview Questions

Beginner: What is an external store, and how does a component stay in sync with it?

A plain object holding state outside the component tree, exposing getState, a way to update, and subscribe(listener). Components bind via useSyncExternalStore (directly or through a library hook): React subscribes, re-reads the snapshot on notification, and re-renders if the selected value changed. State lives outside; React renders a subscription to it.

Beginner: What is a selector and why do stores center on them?

A function from store state to the slice a component needs (s => s.items.length). After each store change, the library compares the selector's new result to the last one and re-renders only on difference. Selectors give per-component re-render precision that context fundamentally lacks — they're the reason stores scale to hot state.

Mid: What problem does useSyncExternalStore solve that useState+useEffect mirroring doesn't?

Tearing under concurrent rendering: with manual mirroring, an interrupted render could paint some components from an old store value and others from a new one — one frame, two truths. useSyncExternalStore guarantees a consistent snapshot per render pass and synchronously re-renders when the store mutates mid-render. It also handles the subscribe/unsubscribe lifecycle and SSR snapshots correctly.

Mid: Zustand vs Redux Toolkit — give a real recommendation.

Same core (single store, immutable updates — RTK via Immer, selector subscriptions). RTK adds an enforced event architecture: serializable actions, a log, time-travel, middleware, RTK Query — at the price of ceremony. Zustand is a few KB with near-zero API: methods instead of actions. Recommend Zustand for most product apps (small global surface); RTK when the action log is a requirement (audit, support-replay), the app is event-sourced by nature, or the org standardized on it.

Senior: Why is a module-level store singleton a problem for SSR, and the fix?

On the server, module scope is shared across requests: user A's dispatch mutates the same store instance user B's render reads — cross-request data leakage, the worst kind of bug. Fix: a store factory called per request/render, delivered via context (StoreProvider creating the instance in useState's initializer); client-side you may still keep one instance. Libraries document exactly this pattern (Redux's per-request store, Zustand's create-in-provider for Next.js).

Senior: Design the state layer for a live trading dashboard: 500 instruments ticking multiple times/second, a watchlist, order forms, and an activity log.

Prices: external store written by the websocket handler (outside React), normalized prices[symbol]; each row subscribes with a per-symbol selector so a tick re-renders exactly one row — context or lifted state would melt at this frequency. Batch bursts (rAF-coalesce notifications) to cap render rate. Watchlist: small persistent client state — same store (persist middleware) or user API via query cache if server-synced. Order forms: local component state (drafts are ephemeral, colocate); submission via mutations with the status-machine pattern. Activity log: append-only slice — an RTK-style event model fits, or a capped array in the store. URL keeps the selected instrument. The principle: partition by write-frequency and reader-set, then pick the narrowest tool per partition.

🏋️ Graded Exercises

1 (easy): Using the Part 2 createStore, build a theme store and a button component that toggles it, with a selector-precise subscription.
Solution 
const themeStore = createStore({ theme: "dark" });

function ThemeToggle() {
  const theme = useStore(themeStore, s => s.theme);
  return (
    <button onClick={() => themeStore.setState(s =>
      ({ theme: s.theme === "dark" ? "light" : "dark" }))}>
      {theme === "dark" ? "🌙" : "☀️"}
    </button>
  );
}
2 (easy): Why does const { items, total } = useCartStore(); in a tiny badge component re-render it when an unrelated discountCode field changes?
Solution 
// No selector → the subscription is to the ENTIRE store state object, whose
// reference changes on every set() — any field, any change, every subscriber.
// Fix: useCartStore(s => s.items.length) — subscribe to precisely what you render.
3 (medium): Bind navigator.onLine with useSyncExternalStore (replacing the useOnlineStatus from Custom Hooks) and say what improved.
Solution 
function subscribe(cb) {
  window.addEventListener("online", cb);
  window.addEventListener("offline", cb);
  return () => {
    window.removeEventListener("online", cb);
    window.removeEventListener("offline", cb);
  };
}
function useOnlineStatus() {
  return useSyncExternalStore(
    subscribe,
    () => navigator.onLine,      // client snapshot
    () => true                   // server snapshot: assume online for SSR markup
  );
}
// Improvements: no state-mirror to drift, tear-free under concurrency, an explicit
// SSR answer, and less code — the browser IS the store; this hook just binds it.
4 (medium): Write a Zustand notifications store: push(message) adds with a generated id and auto-removes after 5s; dismiss(id) removes early. Where does the timer live and why?
Solution 
const useToasts = create((set) => ({
  toasts: [],
  push: message => {
    const id = crypto.randomUUID();
    set(s => ({ toasts: [...s.toasts, { id, message }] }));
    setTimeout(() => set(s =>                       // timer lives IN the action:
      ({ toasts: s.toasts.filter(t => t.id !== id) })), 5000);
  },                                                 // the store outlives components,
  dismiss: id => set(s => ({ toasts: s.toasts.filter(t => t.id !== id) })),
}));
// A component-effect timer would die if the pushing component unmounts — the
// store's lifetime is the right home for lifecycle that must outlive the tree.
// (Actions aren't reducers: side effects like setTimeout are fine here — the
// PURE part is each set() updater.)
5 (medium): This selector re-renders its component on every store change: useStore(s => s.items.filter(i => i.done)). Explain and fix two ways.
Solution 
// filter() returns a NEW array every call — Object.is(new, old) is always false,
// so every notification looks like a change even when contents are identical.
// Fix A: select the source, derive in render:
//   const items = useStore(s => s.items);
//   const done = useMemo(() => items.filter(i => i.done), [items]);
//   (re-renders only when items' reference changes — i.e., actual updates)
// Fix B: shallow-compare the selected result:
//   useStore(useShallow(s => s.items.filter(i => i.done)))
//   (element-wise comparison; still runs filter per notification — fine when cheap)
// A for heavy derivations, B for cheap ones needing exact-slice semantics.
6 (hard): Convert the Part 3 cart store into a per-instance, SSR-safe setup for a Next.js app (store factory + context), keeping the same component-facing API.
Solution 
import { createStore as createZustandStore, useStore as useZustandStore } from "zustand";

function createCartStore() {                          // factory — one per provider
  return createZustandStore((set) => ({
    items: [],
    add: p => set(s => /* …same rules… */ s),
    remove: id => set(s => ({ items: s.items.filter(i => i.id !== id) })),
  }));
}

const CartStoreContext = createContext(null);
export function CartProvider({ children }) {
  const [store] = useState(createCartStore);          // lazy: one instance per mount,
  return (                                            // per request on the server
    <CartStoreContext.Provider value={store}>{children}</CartStoreContext.Provider>
  );
}
export function useCartStore(selector) {
  const store = useContext(CartStoreContext);
  if (!store) throw new Error("useCartStore needs ");
  return useZustandStore(store, selector);
}
// Call sites are unchanged: useCartStore(s => s.items.length).
// Module singleton → per-request instance: no cross-user leakage on the server,
// and tests/storybook get fresh isolated stores by wrapping with the provider.
7 (hard): A review finds your team's Redux store holds: fetched products, the checkout form's field values, a modalOpen flag, the auth token, and websocket-fed stock levels. Sort each into the right home and justify the two you'd leave in Redux.
Solution 
// products (server data)        → query cache: staleness/dedup/invalidation are
//                                  its whole job; Redux copies go stale silently.
// checkout form fields          → local state in the form (colocate; keystroke
//                                  churn doesn't belong in a global log).
// modalOpen                     → useState next to the modal trigger — pure UI state.
// auth token                    → KEEP (or context): genuinely global client
//                                  state, read by non-React code (api layer
//                                  via getState) — store access outside
//                                  components is the deciding feature.
// stock levels (websocket-fed)  → KEEP: hot, client-received, many subscribers —
//                                  selector precision is exactly the store's
//                                  strength; write from the socket handler
//                                  outside React, subscribe per-product in rows.
// Result: Redux shrinks to the two things that need store semantics; everything
// else returns to its lane — the four-kinds taxonomy enforced with a delete key.

🎯 Key Takeaways

  • External stores hold state outside the tree with subscription access; selectors give per-component re-render precision that context structurally can't.
  • useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot) is the tear-free primitive — for stores and for browser APIs alike.
  • Selector discipline: minimal slices, primitives over fresh objects, shallow compare for tuples, memoize heavy derivations — or you rebuild context's problem.
  • Zustand = store precision with method ergonomics; Redux Toolkit = the same plus an enforced, serializable event log — choose by whether the log is a requirement.
  • The healthy layering: URL / query cache / one small store / context injection / colocated state — server data in a client store is the red flag.
  • SSR and multi-instance needs kill module singletons: store factories injected via context, one instance per request or per widget.
