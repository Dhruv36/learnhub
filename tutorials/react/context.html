Context State Management

What you'll master: what context is actually for (dependency injection down the tree, not a state manager), the provider/consumer mechanics, the re-render behavior that makes naive context slow — and every mitigation: value memoization, state/dispatch splitting, context slicing — plus the provider-composition patterns of real apps. Prerequisites: Props (reference identity), The Render Cycle, useReducer.

Part 1 — The Problem: Prop Drilling vs the Alternative's Cost

// Drilling: theme passes through 4 components that don't use it:
<App theme={theme}>                 // owns it
  <Layout theme={theme}>            // courier
    <Sidebar theme={theme}>         // courier
      <NavItem theme={theme} />     // finally consumes it
// Annoying — but EXPLICIT: every dependency is visible in the signature, greppable,
// and testable by passing props. Drilling 2-3 levels is fine. Don't reach for
// context at the first relay; reach for it when COUPLING appears: components
// that take props solely to hand them on, signatures changing in lockstep.

const ThemeContext = createContext("light");     // arg = default when NO provider above
                                                  // (tests, Storybook — not a fallback value)
function App() {
  const [theme, setTheme] = useState("dark");
  return (
    <ThemeContext.Provider value={theme}>
      <Layout />                                 {/* no theme prop anywhere */}
    </ThemeContext.Provider>
  );
}
function NavItem() {
  const theme = useContext(ThemeContext);        // reads NEAREST provider above
  return <a className={`nav-${theme}`}>…</a>;
}
// Nesting providers = scoping: an inner 
// re-themes just its subtree — nearest-wins, like CSS custom properties.
What context is: a wire that makes a value available to any descendant without threading it. What it is not: a store. It has no state of its own — state still lives in a component (or reducer/store) and context only transports it. Judging "context vs Redux" is a category error; the real axis is where state lives, and context is just one delivery mechanism.

Part 2 — The Re-render Contract (Learn This Cold)

// RULE: when a provider's `value` changes (Object.is), EVERY component that
// calls useContext(ThatContext) re-renders — regardless of React.memo walls.
// Memo blocks prop-driven renders; context is a subscription that tunnels past it.

// COROLLARY — the classic self-inflicted wound:
function App() {
  const [user, setUser] = useState(null);
  return (
    <AuthContext.Provider value={{ user, setUser }}>   {/* ❌ fresh object EVERY render */}
      <Everything />
    </AuthContext.Provider>
  );
}
// ANY state change in App — even unrelated — rebuilds {user, setUser}, "changes"
// the value, and re-renders every auth consumer in the app.

// Fix 1 — memoize the value:
const value = useMemo(() => ({ user, setUser }), [user]);

// Fix 2 (better) — SPLIT state and updater into two contexts:
<UserContext.Provider value={user}>                    {/* changes when user changes */}
  <UserDispatchContext.Provider value={setUser}>       {/* NEVER changes (stable setter) */}
    {children}
// A LoginButton that only needs setUser subscribes to the dispatch context and
// re-renders exactly never. Readers pay only for what they read.
// Also from the Render Cycle lesson, applied here: a provider component's OWN
// re-render doesn't re-render {children} it receives as a prop — they're
// reference-equal elements. So a well-shaped provider:
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState("dark");
  const value = useMemo(() => ({ theme, setTheme }), [theme]);
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>;
}
// …re-renders itself on toggle, skips re-rendering the children WHOLESALE, and
// only useContext(ThemeContext) callers anywhere below wake up. That's the
// machinery working as designed: subscribers pay, couriers don't.

Part 3 — The Standard Packaging: Provider + Hook Module

// auth-context.jsx — the pattern you'll write in every real app:
const AuthContext = createContext(null);
const AuthDispatchContext = createContext(null);

export function AuthProvider({ children }) {
  const [state, dispatch] = useReducer(authReducer, { user: null, status: "anon" });
  return (
    <AuthContext.Provider value={state}>
      <AuthDispatchContext.Provider value={dispatch}>
        {children}
      </AuthDispatchContext.Provider>
    </AuthContext.Provider>
  );
}

export function useAuth() {
  const ctx = useContext(AuthContext);
  if (ctx === null)                                   // fail fast, at the real cause:
    throw new Error("useAuth must be used inside ");
  return ctx;
}
export function useAuthDispatch() { /* same guard */ }

// What this buys:
// • The raw context objects never leave the module — consumers can't misuse them.
// • The null-check converts "silent default leaked into prod" into a dev-time throw.
// • The hook is a seam: swap reducer for a store later; callers never change.
// • Reducer + split contexts = the "mini-Redux" from the previous lesson, packaged.

// each layer may CONSUME the ones above it (Api uses Config; Auth uses Api):
<ConfigProvider>
  <ApiProvider>
    <AuthProvider>
      <ThemeProvider>

      </ThemeProvider>
    </AuthProvider>
  </ApiProvider>

// If the pyramid annoys you, compose it once:
const withProviders = (...Providers) => ({ children }) =>
  Providers.reduceRight((kids, P) => <P>{kids}</P>, children);

Part 5 — Performance Playbook for Context (Senior Layer)

// Symptom: typing in one field re-renders 200 components.
// Diagnosis order:
// 1. Is the provider value rebuilt per render?  → useMemo it.           (Part 2)
// 2. Do writers subscribe to state they don't read? → split state/dispatch contexts.
// 3. Is ONE context carrying unrelated domains (user + cart + prefs)?
//    → SLICE into multiple contexts; consumers subscribe to their slice only.
// 4. Does a consumer re-render expensively for a field it barely uses?
//    → extract the context read into a tiny parent, pass primitives to a
//      memoized child:
function CartBadge() {                       // subscribes, cheap
  const { itemCount } = useContext(CartContext);
  return <ExpensiveBadge count={itemCount} />;   // memoized, gets a primitive
}
// 5. Still hot (high-frequency writes, hundreds of subscribers)?
//    → context is the wrong transport: external store with per-selector
//      subscriptions (useSyncExternalStore / Zustand) — next lesson.
//
// Note: React has no built-in "context selector" — useContext(X) wakes for ANY
// value change, even fields you didn't read. Slicing IS the selector mechanism.

Interview Questions

Beginner: What problem does context solve, and what's the API?

Passing data to deep descendants without threading props through every level. createContext(default) makes a context; a <Ctx.Provider value=…> supplies it to a subtree; useContext(Ctx) reads the nearest provider's value (the default only applies with no provider above). Typical cargo: theme, locale, auth, injected services.

Beginner: When is prop drilling actually preferable?

Short distances (2–3 levels), few couriers, or when explicitness matters: props are visible in signatures, greppable, and make components testable in isolation. Context hides the dependency — worth it for genuinely cross-cutting, widely-read values, not for saving two prop declarations.

Mid: Exactly which components re-render when a context value changes?

Every mounted component that calls useContext for that context under that provider — compared by Object.is on the value. React.memo on intermediate components doesn't stop it (context bypasses props), but non-consuming components in between don't re-render just because the wire passes through them — and children passed as elements to the provider are reference-equal and skipped.

Mid: Why split state and dispatch into two contexts?

Because their change frequencies differ: state changes often; dispatch/setters are identity-stable forever. Merged into one object, every state change re-renders write-only components (buttons, forms) that never read state. Split, writers subscribe to the never-changing dispatch context and re-render zero times. It's the cheapest large win in context performance.

Senior: Context vs an external store with selectors — when does context lose?

Context has no selector granularity: consumers wake on any value change, even for unread fields, so it degrades when (a) the value updates frequently, (b) subscribers are numerous, or (c) consumers need different slices. Stores subscribed via useSyncExternalStore re-render only components whose selected slice changed. Context wins for low-churn injection (theme, auth, services) and scoped wiring; stores win for hot, wide, sliceable state. Most apps correctly use both.

Senior: How do you test a component that consumes three contexts?

Render it inside the real providers with controlled initial values — a renderWithProviders test util that wraps children in AuthProvider/ThemeProvider/etc. and accepts overrides. For service contexts (api client), inject fakes through the same provider seam — that's the DI payoff: no module mocking. The throwing accessor hooks also make "forgot a provider" a clear test failure instead of undefined-behavior debugging. If tests need heavy per-test context plumbing, that's a smell the component reads too much — pass slices as props to a presentational inner component and test that directly.

1 (easy): Wire a LocaleContext: provider owning locale state and a t(key) function; a deep Greeting consumes it.
Solution 
const LocaleContext = createContext(null);

function LocaleProvider({ children }) {
  const [locale, setLocale] = useState("en");
  const value = useMemo(() => ({
    locale, setLocale,
    t: key => MESSAGES[locale][key] ?? key,
  }), [locale]);
  return <LocaleContext.Provider value={value}>{children}</LocaleContext.Provider>;
}
const useLocale = () => {
  const ctx = useContext(LocaleContext);
  if (!ctx) throw new Error("useLocale needs ");
  return ctx;
};
function Greeting() { const { t } = useLocale(); return <h1>{t("hello")}</h1>; }

Solution

const login  = useCallback(creds => …, []);

4 (medium): Build a scoped DensityContext for a data table: the table provides "compact"/"comfortable" to its cells, a toolbar button inside toggles it, and two tables on one page stay independent.
Solution 
const DensityContext = createContext(null);

function DataTable({ children }) {
  const [density, setDensity] = useState("comfortable");
  const value = useMemo(() => ({ density, toggle: () => setDensity(d =>
    d === "compact" ? "comfortable" : "compact") }), [density]);
  return (
    <DensityContext.Provider value={value}>
      <table className={`tbl-${density}`}>{children}</table>
    </DensityContext.Provider>
  );
}
// Two s = two providers = independent scopes; nearest-provider-wins
// does the isolation. Context isn't only app-global — scoped providers are the
// backbone of compound components.
5 (medium): Using the split-context auth module from Part 3, write LogoutButton so it never re-renders on user changes, and state how you'd verify that.
Solution 
function LogoutButton() {
  const dispatch = useAuthDispatch();          // dispatch context ONLY — stable value
  return <button onClick={() => dispatch({ type: "logged_out" })}>Log out</button>;
}
// It subscribes to AuthDispatchContext, whose value (the reducer's dispatch) never
// changes identity → zero context-driven re-renders.
// Verify: React DevTools Profiler → record a login/logout of ANOTHER session
// updating user state → LogoutButton should show no render (gray). Or add a
// render counter in dev and assert it stays at 1 across user changes.
6 (hard): An app has one AppContext holding { user, cart, theme, notifications } and everything is slow. Design the refactor: name the contexts you'd create, their change frequencies, and the migration path that doesn't break 80 consumers at once.
Solution 
// Slice by domain × change frequency:
// UserContext            (low churn)   + UserDispatchContext (never)
// ThemeContext           (rare)        + toggle in same value (rare enough)
// CartContext            (medium)      + CartDispatchContext (never)
// NotificationsContext   (high churn!) → consider a store w/ selectors if
//                                        many subscribers only show a badge count.
// Migration without a big bang:
// 1. Create the four sliced providers; mount them INSIDE the old AppProvider.
// 2. Reimplement AppContext's value as a useMemo composition of the slices —
//    old consumers keep working, now one layer down.
// 3. Move consumers to sliced hooks (useUser, useCart…) incrementally — each
//    move immediately stops that component waking for other domains.
// 4. When AppContext has no consumers (grep), delete it.
// Measure at steps 0 and 3 with the Profiler to prove the win (typing in cart
// no longer renders the notification bell, etc.).

🎯 Key Takeaways

  • Context transports a value to any descendant — dependency injection down the tree. State still lives in components/reducers; context is the wire, not the store.
  • The contract: value changes (Object.is) → every consumer re-renders, through memo walls. Inline value objects are the #1 self-inflicted storm; useMemo or split contexts.
  • Split state from dispatch: writers subscribe to the stable dispatch context and re-render never. Slice god-contexts by domain and churn — slicing is React's selector mechanism.
  • Package as module: Provider component + throwing accessor hooks; never export the raw context. Defaults are for provider-less rendering, not fallbacks.
  • Right cargo: theme/locale/auth/services/scoped wiring. Wrong cargo: keystrokes, mouse positions, server caches, "everything".
  • When churn × subscribers outgrows context, the answer is a store with per-selector subscriptions — the next lesson.
