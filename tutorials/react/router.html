Routing & URL State State Management

What you'll master: how client-side routing actually works (History API + render switching), React Router's core surface — routes, nested layouts with Outlet, params, navigation — search params as first-class state, loaders and the render-fetch waterfall they kill, and route protection. Prerequisites: State Architecture (the URL-state taxonomy), Data Fetching. Examples use React Router v6.4+ (the data-router APIs).

Part 1 — What a Client Router Actually Does

// An SPA has ONE html page. "Navigation" is three moves:
// 1. Intercept link clicks (preventDefault — Events lesson) so no page reload happens.
// 2. history.pushState(state, "", "/products/42")  — change the URL bar, add a
//    history entry. The browser does NOT request anything.
// 3. Render the component tree that matches the new URL. Back/forward buttons
//    fire "popstate" — the router listens and re-renders accordingly.
//
// That's the whole trick: the URL becomes a STATE INPUT (the most privileged one —
// shareable, bookmarkable, back-button-aware), and the router is a controlled
// component over it. Everything else is ergonomics on top.

Part 2 — Routes, Nesting & Layouts

import { createBrowserRouter, RouterProvider, Outlet, Link, NavLink } from "react-router-dom";

const router = createBrowserRouter([
  {
    path: "/",
    element: <Shell />,                        // layout route: header/nav chrome
    errorElement: <RouteError />,              // renders on loader/render errors below
    children: [                                // children render INTO Shell's 
      { index: true, element: <Home /> },      // "/" exactly
      { path: "products", element: <ProductsLayout />, children: [
        { index: true, element: <ProductList /> },       // /products
        { path: ":productId", element: <ProductPage /> }, // /products/42 — dynamic segment
      ]},
      { path: "login", element: <Login /> },
      { path: "*", element: <NotFound /> },    // catch-all, LAST
    ],
  },
]);

function App() { return <RouterProvider router={router} />; }

function Shell() {
  return (
    <>
      <header>
        {/* NavLink knows when it matches — free active styling: */}
        <NavLink to="/products" className={({ isActive }) => isActive ? "on" : ""}>
          Products
        </NavLink>
      </header>
      <main><Outlet /></main>                  {/* matched child renders here */}
    </>
  );
}
// The mental model: the URL is parsed into a MATCH STACK (Shell → ProductsLayout →
// ProductPage), rendered as nested components via Outlets. Navigating within
// /products swaps only the deepest match — Shell and ProductsLayout keep their
// state (same position + type: the reconciliation rules you already know).

Part 3 — Params, Navigation & Search Params

import { useParams, useNavigate, useSearchParams } from "react-router-dom";

// 1. Path params — identity ("which resource"):
function ProductPage() {
  const { productId } = useParams();                   // "/products/42" → "42" (a STRING)
  const product = useQuery(["product", productId], …); // params drive data keys
}

// 2. Programmatic navigation — for AFTER actions (form success, logout):
const navigate = useNavigate();
navigate(`/products/${created.id}`);                   // push
navigate("/login", { replace: true });                 // replace: no back-button return
navigate(-1);                                          // history back
// Links for navigation the USER initiates (real : middle-click, copy, a11y);
// navigate() for navigation your CODE initiates. Don't onClick+navigate a fake link.

// 3. Search params — view configuration ("how am I looking at it"):
function ProductList() {
  const [params, setParams] = useSearchParams();       // ?q=cam&sort=price&page=2
  const q    = params.get("q") ?? "";
  const sort = params.get("sort") ?? "relevance";
  const page = Number(params.get("page") ?? 1);

  function update(patch) {
    setParams(prev => {
      const next = new URLSearchParams(prev);
      for (const [k, v] of Object.entries(patch)) {
        if (v === null || v === "" ) next.delete(k);   // empty = absent, clean URLs
        else next.set(k, String(v));
      }
      return next;
    }, { replace: true });                             // replace for filter TWEAKS —
  }                                                    // don't spam history entries
  // update({ q: "cam", page: null })  → ?q=cam&sort=…  (new search resets page)
}
// The taxonomy from State Architecture, realized: productId (identity) is a path
// param; q/sort/page (view config) are search params; hover state stays useState.

Part 4 — Loaders: Fetch Before Render

// The fetch-on-mount pattern (Data Fetching lesson) has a built-in waterfall:
// navigate → render spinner → effect fires → fetch → render data.
// Data routers flip it: the fetch starts AT NAVIGATION, before rendering:

{
  path: ":productId",
  element: <ProductPage />,
  loader: async ({ params, request }) => {             // runs on navigation start
    const url = new URL(request.url);                  // search params available too
    const res = await fetch(`/api/products/${params.productId}`);
    if (res.status === 404)
      throw new Response("Not found", { status: 404 }); // → nearest errorElement
    return res.json();                                  // resolved BEFORE render
  },
}

function ProductPage() {
  const product = useLoaderData();                      // data is just… there
  return <h1>{product.name}</h1>;                       // no loading branch needed
}

// Sibling loaders run in PARALLEL (list + user + config = one round-trip time),
// which structurally kills the parent→child fetch waterfall from Data Fetching ex. 7.
// useNavigation() gives global pending state for a top progress bar:

{navigation.state === "loading" && <TopProgressBar />}
// Loaders + a query cache compose: loader does queryClient.ensureQueryData(key) —
// navigation-time start, cache-grade dedup/staleness. Best of both.

Part 5 — Protected Routes & Redirects

// Guard = a layout route that checks auth and either renders the Outlet or redirects:
import { Navigate, useLocation } from "react-router-dom";

function RequireAuth() {
  const { user, status } = useAuth();                   // Context lesson's hook
  const location = useLocation();

  if (status === "loading") return <FullPageSpinner />; // don't flash the login page
  if (!user)
    return <Navigate to="/login" state={{ from: location }} replace />;
  return <Outlet />;                                    // authed: render children
}

// Route table: wrap a SUBTREE, not each route:
{ element: <RequireAuth />, children: [
  { path: "account", element: <Account /> },
  { path: "orders",  element: <Orders /> },
]}

// Login bounces back to where the user was headed:
function Login() {
  const location = useLocation();
  const navigate = useNavigate();
  const from = location.state?.from?.pathname ?? "/";
  async function onSubmit(creds) {
    await login(creds);
    navigate(from, { replace: true });                  // replace: login page isn't history
  }
}
// SECURITY REALITY: this is UX, not protection. All client code ships to the
// browser; anyone can read it. The API must enforce authorization on every
// request — the guard just avoids showing pages that would 401 anyway.
// (JS track → Security for the full threat model.)

Interview Questions

Beginner: How does client-side routing differ from server routing?

Server routing: every navigation is an HTTP request returning a new document. Client routing: one document; the router intercepts link clicks, updates the URL via the History API (pushState — no request), and swaps the rendered component tree to match; back/forward fire popstate. Result: instant transitions and preserved app state, at the cost of needing an SPA fallback on the server and manual scroll/focus/title management.

Beginner: Path params vs search params — when each?

Path params (/products/:id) identify which resource — the page is about a different thing without them. Search params (?sort=price&page=2) configure how you're viewing it — optional, defaultable, combinable. Litmus: required identity → path; shareable view state → search.

Mid: What do nested routes + Outlet buy over a flat route list?

Layout sharing without repetition (shell/section chrome defined once, children render into Outlet), state preservation during sibling navigation (parents keep matching → same position/type → reconciliation reuses them, so a sidebar's scroll survives), URL-structure-as-UI-structure legibility, and per-subtree cross-cutting: one errorElement or auth-guard layout protects a whole branch.

Mid: What problem do route loaders solve versus fetch-in-useEffect?

Effect fetching serializes render→effect→fetch→re-render and compounds into waterfalls when nested components each fetch. Loaders start data at navigation intent, run all matched routes' loaders in parallel, keep the previous page interactive while loading (with global pending state), and route errors to errorElement. Components render with data already present — no per-component loading branches, no mount-time races.

Senior: Design URL state for a flight search: origin/destination/dates, 12 filters, a selected result, and a comparison tray (up to 3 flights). Justify each placement.

Origin/destination/dates: search params — the core shareable query (?from=BLR&to=NRT&d=2026-08-01). Twelve filters: search params too, but only non-defaults serialized (empty = absent) to keep URLs clean; replace-navigation for tweaks so back returns to the previous search, not 30 filter states. Selected result: path param (/flights/:offerId) — it's a distinct, deep-linkable resource page. Comparison tray: judgment call — ?compare=a,b,c if sharing a comparison is a product feature; otherwise client state (store) since it's a workspace, not a view. Everything derives from the URL on load, so refresh reproduces the session — the acceptance test for the design.

Senior: Users report: refresh on any deep page 404s in prod but works in dev, and after fixing that, back-button lands them mid-page with the wrong focus. Explain both and fix.

404s: dev servers ship an SPA fallback by default; production hosting didn't — the server literally has no /products/42 resource. Fix: rewrite all non-asset paths to index.html (host config — netlify redirects, nginx try_files) so the router boots and matches. Scroll/focus: SPAs neither reset scroll on forward-nav nor announce navigation to assistive tech. Fix: <ScrollRestoration /> (top on push, restored position on pop) and on location change move focus to the main heading (tabIndex={-1} + .focus()) so keyboard/screen-reader users land at the top of the new page instead of mid-DOM on a stale element.

3 (medium): Implement a sortable, paginated table whose state lives entirely in the URL: ?sort=name&dir=asc&page=3. Clicking a column toggles dir; changing sort resets page.
Solution 
function useTableParams() {
  const [params, setParams] = useSearchParams();
  const sort = params.get("sort") ?? "name";
  const dir  = params.get("dir") ?? "asc";
  const page = Number(params.get("page") ?? 1);

  function sortBy(col) {
    setParams(prev => {
      const n = new URLSearchParams(prev);
      if (col === sort) n.set("dir", dir === "asc" ? "desc" : "asc");
      else { n.set("sort", col); n.set("dir", "asc"); n.delete("page"); } // reset page
      return n;
    }, { replace: true });
  }
  function goPage(p) {
    setParams(prev => { const n = new URLSearchParams(prev);
      p <= 1 ? n.delete("page") : n.set("page", String(p)); return n; });  // push: pages
  }                                                                        // are locations
  return { sort, dir, page, sortBy, goPage };
}
// Refresh, share, and back all reproduce the exact table view — zero useState.

🎯 Key Takeaways

  • Client routing = intercepted clicks + History API + tree swapping; the URL is your most privileged state input — shareable, bookmarkable, back-aware.
  • Nested routes render through Outlets: layouts define chrome once, keep state across sibling navigation, and host per-branch error/auth boundaries.
  • Identity → path params (strings — parse them); view config → search params (empty = absent, replace for tweaks); Link for users, navigate() for code.
  • Loaders fetch at navigation intent, in parallel across matches, with the previous page still interactive — the structural fix for effect-fetch waterfalls; compose them with the query cache.
  • Route guards are UX; the API authorizes. Distinguish unauthenticated (redirect + return-to) from unauthorized (403 page).
  • Production checklist: SPA fallback on the host, route-level code splitting, ScrollRestoration, focus-to-heading on navigation, centralized path builders.
