Routing Intermediate

An SPA swaps components instead of loading pages — the router maps URL ↔ UI and keeps the back button, deep links, and sharing working. The underrated insight of this lesson: the URL is state — often the best home for it. Examples use React Router (the concepts transfer to TanStack Router and Next.js file-routing directly).

Route Configuration & Layouts

const router = createBrowserRouter([
  { path: "/", element: <RootLayout />, errorElement: <ErrorPage />,
    children: [                                   // NESTED: children render inside the layout
      { index: true, element: <Home /> },        // index = the "/" child
      { path: "products", element: <ProductsLayout />,
        children: [
          { index: true, element: <ProductGrid /> },
          { path: ":productId", element: <ProductDetail /> },   // dynamic segment
        ]},
      { path: "*", element: <NotFound /> },      // catch-all
    ]},
]);
<RouterProvider router={router} />

function RootLayout() {
  return <><Nav /><main><Outlet /></main><Footer /></>;   // Outlet = child slot
}
// Nested routes = nested LAYOUTS: /products/42 renders RootLayout → ProductsLayout →
// ProductDetail, each persisting while only the changed level swaps. This mapping of
// URL segments to layout hierarchy is the core idea of every modern router.

Navigation & Params

<Link to="/products/42">Details</Link>            // real <a> (SEO, cmd-click) + SPA swap
<NavLink to="/products" className={({isActive}) => isActive ? "active" : ""}>  // nav highlighting

const { productId } = useParams();                  // ":productId" → { productId: "42" } (STRING!)
const navigate = useNavigate();
navigate("/checkout");                               // programmatic (after submit)
navigate(-1);                                        // back
navigate("/login", { replace: true });               // no history entry (redirects)

URL as State — searchParams

const [params, setParams] = useSearchParams();      // ?q=shoes&sort=price&page=2
const q = params.get("q") ?? "";
setParams(p => { p.set("page", "3"); return p; });

// WHY store filters/tabs/pagination in the URL instead of useState:
//  shareable (send a colleague the filtered view) · back-button-correct ·
//  survives refresh · deep-linkable from emails/docs
// Litmus from the state lesson: "would a user want to share/bookmark this view?" → URL.

Loaders — Fetch Before Render (killing waterfalls)

{ path: ":productId", element: <ProductDetail />,
  loader: ({ params }) => fetchProduct(params.productId) }   // runs ON NAVIGATION, before render

function ProductDetail() {
  const product = useLoaderData();                            // data is ALREADY there
  …
}
// vs useEffect fetching: render spinner → mount → fetch → re-render (a client waterfall,
// worse when nested children each fetch). Loaders parallelize per route level and integrate
// pending/error UI. Same philosophy as Next.js server fetching — data before paint.

Protected Routes

function RequireAuth({ children }) {
  const { user } = useAuth();
  const location = useLocation();
  if (!user) return <Navigate to="/login" state={{ from: location }} replace />;
  return children;
}
// After login: navigate(location.state?.from ?? "/") — return the user where they were.
// ⚠ Client-side guards are UX, not security — the API must enforce auth (HTML forms lesson's
// "client validation is not a boundary", again).

Code Splitting per Route (modules lesson, applied)

const Settings = lazy(() => import("./pages/Settings"));
{ path: "settings", element: <Suspense fallback={<Spinner/>}><Settings /></Suspense> }
// Route boundaries are the natural split points — each page loads only its bundle.

Interview Questions

Q1: How does client-side routing actually work under the hood?

The History API: pushState/replaceState change the URL without a request; a popstate listener catches back/forward; the router matches the new path against its route tree and renders the matching elements. Link intercepts click → preventDefault → pushState → render. Server config must serve index.html for all paths (the SPA fallback) or deep links 404.

Q2: Loaders vs useEffect fetching vs React Query — position all three.

Effect fetching: after-mount waterfalls, DIY everything — the baseline to avoid at scale. Loaders: fetch starts AT navigation, parallel per route level, integrated pending/error UI — great for route-shaped data. React Query: cache/dedupe/revalidate across the app, component-level needs, mutations. Production apps commonly combine: loaders (or RSC) for initial route data, Query for interactive/cached data. Knowing they compose is the senior answer.

Q3: Which state belongs in the URL, and what breaks when it doesn't?

Anything defining "which view is this": filters, search, sort, pagination, selected tab/item. In useState instead: refresh resets it, back button surprises users, links can't share it, and analytics/support can't reproduce views. The URL is free, persistent, shareable state — the state-management decision table's forgotten row.

🏋️ Practice: build a product browser: nested layouts (root → products → :id), NavLink highlighting, filters in searchParams (verify share/refresh/back), a loader for the detail page, a RequireAuth wrapper, and a lazy-loaded settings route. Break it: use <a> instead of Link and watch state die.

🎯 Key Takeaways

  • Routes map URL segments to nested layouts via Outlet; index routes and catch-alls complete the tree.
  • Link/NavLink over <a>; params are strings; replace for redirects.
  • The URL is state: filters/tabs/pagination in searchParams — shareable, refresh-proof, back-correct.
  • Loaders fetch at navigation time (no waterfalls); route boundaries are the code-split points.
  • Client guards are UX; the server enforces auth. History API + SPA fallback is what's underneath.
