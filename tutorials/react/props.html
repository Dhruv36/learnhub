Props & Component Contracts Foundations

What you'll master: props as a component's public API — one-way data flow and why React enforces it, immutability and reference identity (the concept 80% of React performance questions hinge on), defaults, callback props ("data down, events up"), and how to design prop APIs that survive growth. Prerequisites: Components & JSX, JS Objects (reference vs value semantics).

Part 1 — The Mental Model: Props Are Function Arguments With a Contract

A component's props are literally its function parameter — one object per render. But treat them as more than arguments: props are the component's public API. Everything a parent can configure, and nothing else, flows through them. Two rules give React its predictability:

RuleMeaningPayoff
One-way data flowData goes parent → child, only. A child can't reach up.To find where any value comes from, walk up the tree — one direction, no cycles.
Props are read-onlyNever assign to or mutate props inside a component.Renders stay pure and repeatable; React can compare by reference to skip work.
function PriceTag({ amount, currency = "USD" }) {   // default value, plain JS syntax
  // amount = amount * 1.2;        ❌ never reassign — compute a NEW value instead:
  const withTax = amount * 1.2;   // ✅ derive locally
  return <span>{withTax.toFixed(2)} {currency}</span>;
}
"Read-only" is deeper than "don't reassign". Mutating an object inside a prop (props.user.name = "x") is worse: the parent still holds the same reference, so React sees "nothing changed", skips re-renders, and the UI silently diverges from the data. Reference identity — did the object change, or its contents? — is the axis the whole Performance section turns on.

Part 2 — Data Down, Events Up

Children can't write to props — so how does anything ever change? The parent passes a callback prop; the child calls it with event data; the parent owns the state change. This is the fundamental loop of every React app:

function Parent() {
  const [cart, setCart] = useState([]);                 // state lives here (next lesson)

  function handleAdd(productId) {                       // parent decides WHAT happens
    setCart(c => [...c, productId]);
  }

  return <ProductRow id={7} name="Camera" onAdd={handleAdd} />;   // pass behavior DOWN
}

function ProductRow({ id, name, onAdd }) {              // child reports events UP
  return (
    <li>
      {name}
      <button onClick={() => onAdd(id)}>Add to cart</button>   {/* child knows WHEN, not WHAT */}
    </li>
  );
}

Naming convention with teeth: event props are onXxx (onAdd, onSelect), handlers are handleXxx. The child fires semantic events in its own vocabulary ("a product was added"), never presuming what the parent will do — that's what keeps ProductRow reusable in a wishlist, a comparison page, anywhere.

Part 3 — Every Kind of Prop, and When to Use Each

<DataTable

  pageSize={25}                        // config: scalars with sensible defaults
  onRowClick={openDetails}             // events: child → parent notifications

/>

Prop API design heuristics that show up in senior code review:

HeuristicWhy
Prefer one object prop for cohesive data (user={user}) over five scalars (userName, userEmail…)The shape travels together; adding a field doesn't change every call site
…but prefer scalars when the component needs only 1–2 fieldsNarrow contracts re-render less and are easier to test/mock
Booleans multiply: variant="primary"|"ghost" beats isPrimary isGhostEnums make illegal combinations unrepresentable
Don't pass what a child can derive (items + itemCount)Duplicate sources of truth drift; derive in the child
Stable prop names over implementation names (onSelect, not onTrClick)The contract outlives the markup

Part 4 — Build It: Evolving a Component's Contract

// v1 — a Rating display. Minimal contract:
function Rating({ value }) {                        // value: 0–5
  return <span>{"★".repeat(value)}{"☆".repeat(5 - value)}</span>;
}

// v2 — product wants max=10 somewhere. Extend with a DEFAULT (non-breaking):
function Rating({ value, max = 5 }) {
  return <span>{"★".repeat(value)}{"☆".repeat(max - value)}</span>;
}

// v3 — make it interactive, but ONLY if the parent opts in.
//      Presence of the callback = capability switch:
function Rating({ value, max = 5, onChange }) {
  const interactive = typeof onChange === "function";
  return (
    <span role={interactive ? "slider" : undefined} aria-label="rating">
      {Array.from({ length: max }, (_, i) => (
        <button
          key={i}
          disabled={!interactive}
          onClick={() => onChange?.(i + 1)}        // report the NEW value up; parent owns state
        >{i < value ? "★" : "☆"}</button>
      ))}
    </span>
  );
}
// The same component is now a display OR a control — the parent decides by what it passes.
// Note what we never added: internal state. `value` stays parent-owned, so two Ratings
// can share one value, and the parent can validate/veto changes. This is the
// "controlled component" idea — Forms builds on it heavily.

Part 5 — Reference Identity: the Prop Fact That Runs the Framework (Senior Layer)

// Every render creates NEW object/array/function literals:
function Parent() {
  const style = { color: "red" };          // new object EVERY render
  const items = data.filter(d => d.ok);    // new array EVERY render
  const onPick = () => select(id);         // new function EVERY render
  return <Child style={style} items={items} onPick={onPick} />;
}
// Child's props are "different" by === on every render even if visually identical.
// Consequences, in increasing severity:
// 1. React.memo(Child) never skips — memoization defeated (Performance lesson).
// 2. If Child uses items in a useEffect dependency array → effect re-runs every render.
// 3. If Child stores props in state "to copy them" → stale duplicates.

// This is NOT a bug to fix everywhere — creating objects in render is normal and fine
// until a measurement says otherwise. The skill is KNOWING it's happening, so when a
// memoized child re-renders or an effect loops, you look at prop identity first.
The exam question behind half of React interviews: "why does this memoized component still re-render?" Answer: a prop is a fresh reference each render — inline object, array from .map/.filter, arrow function. Fixes (useMemo, useCallback, hoisting constants) live in useMemo & useCallback; the diagnosis lives here.

Part 6 — Validating Contracts: TypeScript Over PropTypes

// Modern codebases type props instead of runtime-checking them:
type RatingProps = {
  value: number;
  max?: number;                              // optional = has a default
  onChange?: (next: number) => void;         // optional capability
};

function Rating({ value, max = 5, onChange }: RatingProps) { /* … */ }

// What TS buys over legacy PropTypes: errors at compile time (not console warnings
// in production), autocomplete on every call site, and refactors that can't miss a caller.
// PropTypes survives only in old codebases — recognize it, don't start with it.

MistakeConsequenceFix
Mutating a prop object (props.user.tags.push(x))Parent's data changes without a re-render; UI and state divergeCopy-then-change ([...tags, x]) — in the owner, via a callback

Interview Questions

Beginner: Why are props read-only, and how does data ever change?

Read-only props keep every render a pure function of inputs, so React can re-run components freely and compare results. Change happens by "data down, events up": the parent owns the value in state, passes it down, and passes a callback the child invokes; the parent updates state, and new props flow down on the next render.

Beginner: How do you give a prop a default value?

Destructuring defaults in the signature: function Btn({ variant = "primary" }). It applies when the prop is undefined (absent or explicitly undefined) — but not for null, which is passed through as-is. Class components used defaultProps; for function components that's deprecated.

Mid: What breaks when you mutate an object prop in place instead of replacing it?

React and its optimization layers (memo, effect dependency arrays, state-change detection) compare by reference. In-place mutation keeps the reference identical, so everything concludes "unchanged": memoized children skip renders they need, effects don't re-run, and if the object also lives in parent state, the parent won't re-render either. The data changed but no one was notified — the definitive source of "UI is stale" bugs.

Mid: Why is useState(props.value) usually a bug?

useState reads its argument only on mount; subsequent renders ignore it. So the copy captures the first props.value forever, and when the parent later passes a new value the child shows the stale one. Correct designs: keep it fully controlled (render from the prop, change via callback), or make "initial only" explicit (defaultValue) and add a key to remount when the source changes.

Senior: Why does passing an inline arrow function as a prop matter — and when does it not?

A new function identity per render makes the prop unequal by === each time. That's irrelevant for a plain DOM handler on an unmemoized child (it re-renders with its parent anyway). It matters when the receiver is wrapped in React.memo, when the function lands in a dependency array, or when it's a context value — there it silently defeats every skip. The senior answer is "it depends on the consumer", not "always useCallback": premature memoization adds cost and noise.

Senior: How do you design the prop API for a reusable component used by 30 teams?

Treat it as a public API with semver discipline: minimal surface (each prop = permanent maintenance), enums over boolean flags, semantic event names decoupled from markup, slots (children/element props) over configuration flags for content, pass-through (className, ...rest to the root) for escape hatches, and types as the contract. Additive changes only; breaking changes get a codemod. The goal: callers depend on meaning, never on internal structure.

1 (easy): Write Greeting taking name with default "friend", rendering "Hello, {name}!".
Solution 
function Greeting({ name = "friend" }) {
  return <p>Hello, {name}!</p>;
}
2 (easy): Spot the bug: function Total({ items }) { items.sort((a,b) => a.price - b.price); return <b>{items[0].price}</b>; }
Solution 
// .sort() mutates IN PLACE — this reorders the parent's array through the prop.
// Fix — copy first (or use the non-mutating method):
const cheapest = [...items].sort((a, b) => a.price - b.price)[0];
// or: items.toSorted((a,b) => a.price - b.price)[0]
3 (medium): Build Counter as a fully stateless component: props value, onIncrement, onDecrement, with the minus button disabled at 0.
Solution 
function Counter({ value, onIncrement, onDecrement }) {
  return (
    <div>
      <button onClick={onDecrement} disabled={value === 0}>−</button>
      <span>{value}</span>
      <button onClick={onIncrement}>+</button>
    </div>
  );
}
// Stateless = the parent owns value; this Counter is pure display + events up.
4 (medium): Refactor this prop API and justify each change: <Modal isSmall isLarge isDanger showCloseX titleText="Hi" bodyText="…" />
Solution 
<Modal size="sm" tone="danger" dismissible title="Hi">
  <p>…</p>
</Modal>
// isSmall/isLarge → size enum (can't be both small AND large);
// isDanger → tone enum (room for "warning" later without a new boolean);
// showCloseX → dismissible (names the meaning, not the pixel);
// bodyText → children (slot: callers can pass any markup, not just text).
5 (medium): A parent renders <Child config={{ theme: "dark" }} /> and Child is wrapped in React.memo. Why does Child re-render every time, and name two fixes.
Solution 
// The object literal creates a NEW reference every parent render; memo's
// shallow prop comparison sees config !== prevConfig and re-renders.
// Fix 1 — hoist (it never changes):  const CONFIG = { theme: "dark" };  // module scope
// Fix 2 — memoize if it's derived:   const config = useMemo(() => ({ theme }), [theme]);
// Fix 3 — flatten: pass theme="dark" (scalar props compare by value).
6 (hard): Implement the v3 interactive Rating from Part 4, then add a readOnly prop that wins even if onChange is passed. Which prop should a screen reader user perceive?
Solution 
function Rating({ value, max = 5, onChange, readOnly = false }) {
  const interactive = !readOnly && typeof onChange === "function";
  return (
    <span aria-label={`Rating: ${value} of ${max}`}>
      {Array.from({ length: max }, (_, i) => (
        <button key={i} type="button" disabled={!interactive}
                aria-pressed={i < value}
                onClick={() => onChange?.(i + 1)}>
          {i < value ? "★" : "☆"}
        </button>
      ))}
    </span>
  );
}
// readOnly &&-guards the capability: explicit intent beats inferred capability.
// Screen readers get the aria-label summary and disabled state on the buttons.
7 (hard): Parent passes onSave={(data) => api.save(data)} to a memoized Form, and also passes draft (an object rebuilt by useMemo keyed on [userId]). Form still re-renders every parent render. List the possible causes in the order you'd check them.
Solution 
// 1. onSave: inline arrow = new identity per render → memo fails. (Most likely.)
//    Check: wrap in useCallback([api]) or hoist.
// 2. children: if Form receives JSX children, they're fresh elements each render —
//    memo compares children by reference too.
// 3. draft's useMemo can be discarded (React may drop memo caches) or its deps
//    ([userId]) may themselves change identity — verify userId is a primitive.
// 4. A context Form consumes changed value — memo doesn't block context updates.
// Diagnosis tool: React DevTools Profiler → "why did this render?".

🎯 Key Takeaways

  • Props are the component's public API: one-way, read-only, and the only sanctioned input besides context — design them like you'll maintain them for years.
  • "Data down, events up": parents own state and pass onXxx callbacks; children report semantic events with data, never assuming what the parent does.
  • Never mutate through a prop — in-place changes keep the reference identical, so React's entire change-detection stack concludes nothing happened.
  • Every render rebuilds inline objects/arrays/functions: harmless by default, decisive when the consumer memoizes, depends, or subscribes — diagnose identity before reaching for useCallback.
  • Contract heuristics: enums over boolean piles, pass sources not derivatives, capability via optional callbacks, TypeScript as the contract language.
  • Copying props into state freezes them at mount — stay controlled, or name it defaultX and key the component to reset.
