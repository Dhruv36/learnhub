Production Patterns Full-Stack React

What you'll master: the last mile — shipping React with monitoring you can act on, Web Vitals as budgets, feature flags and safe rollouts, accessibility and security as release gates, and the pre-launch checklist that assembles the entire track. This is the capstone: every section contributes a subsystem; here they become one operated product.

Part 1 — Observability: If You Can't See It, It's Broken

// The three signal streams a production React app emits:

// 1. ERRORS (L23's pipeline, now complete):
//    boundary catches + window.onerror + unhandledrejection → tracker, tagged
//    with release, boundary name, componentStack, route. Sourcemaps per deploy.
//    Alert on RATE per region, not per event.

// 2. WEB VITALS from REAL users (lab Lighthouse lies about your P75):
import { onLCP, onINP, onCLS } from "web-vitals";
onLCP(m => beacon("vitals", m)); onINP(m => beacon("vitals", m)); onCLS(m => beacon("vitals", m));
//    LCP < 2.5s  — largest content paint (images/fonts/streaming — L27's levers)
//    INP < 200ms — interaction latency (long renders — the L18-21 toolkit)
//    CLS < 0.1   — layout shift (sized images, matched skeletons — L20/27)
//    Track P75 per route, per device class. A regression alert on these catches
//    what no unit test can: the app got slower for real people.

// 3. PRODUCT EVENTS — an analytics module (context-injected, L14's DI) so
//    "did the feature get used" and "where do flows drop off" have answers.
//    Instrument flows (started/completed/failed), not clicks.
//
// The discipline that makes all three usable: RELEASE TAGGING. Every beacon
// carries the build id → any regression graph points at a deploy diff.

Part 2 — Web Vitals as Budgets, Not Trivia

// A budget is a number in CI, not an aspiration in a doc:
// • bundle budget: main chunk < 200KB gz — the analyzer runs in CI, PRs that
//   cross the line fail with the diff attributed (which import grew).
// • vitals budget: P75 LCP/INP/CLS per key route; nightly synthetic run +
//   RUM dashboards; regression > 10% pages the owning team.
// The causal chains you now own end-to-end:
// LCP regressions ← unsized/unoptimized images (L27), blocking fonts, slow
//   TTFB (caching mode wrong, L27), fallback-less streaming order (L26).
// INP regressions ← wide re-renders on interaction (L18: state placement),
//   unmemoized hot lists (L18-19), missing transitions on heavy updates (L21),
//   synchronous work in handlers (L6: throttle to rAF).
// CLS regressions ← skeletons that don't match final layout (L20), images
//   without dimensions, late-injected banners.
// When a budget fires, this track IS the runbook.

Part 3 — Feature Flags & Safe Rollouts

// Deploy ≠ release. Flags separate them:
const flags = useFlags();                       // context-injected (L14), fetched
{flags.newCheckout ? <CheckoutV2 /> : <CheckoutV1 />}   // at session start

// The rollout ladder for anything risky:
// internal users → 1% → 10% (watch error rate + vitals + funnel per COHORT)
// → 50% → 100% → delete the flag (a flag past its rollout is dead code with
// a live switch — inventory and reap them monthly).
//
// Flag hygiene rules:
// • Flags gate COMPOSITION (which component renders — L22's seams), never
//   sprinkle if(flag) through business logic.
// • The OFF path stays tested until the flag is deleted (both variants in CI).
// • Kill switches for external dependencies: the reviews service melting
//   shouldn't take checkout down — flag it off, the L23 boundary catches the
//   transition, the page degrades gracefully.
// • SSR/RSC: evaluate flags server-side per request (cookie/header bucketing)
//   or hydration mismatches greet every experiment (L26's purity rule).

Part 4 — Accessibility as a Release Gate

// The track built a11y in from the start — production makes it enforceable:
// ALREADY IN HAND: real buttons/forms (L6/L7), labels + aria-invalid +
// role=alert errors (L7), focus management in modals/menus (L11, L22 ex. 6),
// route-change focus + ScrollRestoration (L17), role-based test queries that
// fail on inaccessible markup (L24).
// THE PRODUCTION LAYER:
// 1. Automated floor in CI: eslint-plugin-jsx-a11y + axe checks in component
//    tests — catches ~40% of issues (missing alts, contrast, roles) for free.
axe(await render(<CheckoutForm />));           // jest-axe: fails on violations
// 2. The manual pass no tool replaces, per release, on changed flows:
//    unplug the mouse (tab through: reachable? visible focus? logical order?),
//    screen reader smoke test (does the flow NARRATE sensibly?),
//    200% zoom + reduced-motion + forced-colors render checks.
// 3. Live-region audit: async updates (toasts L22, route changes L17, cart
//    counts) announce via aria-live — silent updates are invisible updates.
// A feature that can't be completed keyboard-only is not done — that's the
// gate, in the definition of done, not a quarterly audit.

Part 5 — Security: the Front-End's Share

// The JS track's Security lesson holds the full model; React's slice of it:
// 1. XSS: JSX escaping covers interpolation (L2) — the audit points are the
//    bypasses: every dangerouslySetInnerHTML (sanitize with DOMPurify, or
//    delete), href/src built from user data (block javascript: URLs),
//    third-party embed html.
// 2. Secrets: NOTHING sensitive in client bundles — VITE_/NEXT_PUBLIC_ env
//    vars ship to everyone; grep builds for keys in CI. Server components/
//    actions hold real secrets (L26).
// 3. Auth tokens: httpOnly cookies over localStorage (XSS can't read them);
//    client route guards are UX, authorization lives at the data layer
//    (L17/L26/L27 — the three-tier rule).
// 4. Server actions/handlers: authenticate + authorize + schema-validate every
//    input INSIDE (L26's threat model) — client validation is UX only (L7).
// 5. Dependencies: lockfiles + audit in CI + minimal dependency surface —
//    the supply chain is part of your bundle.
// 6. Headers (set at the edge/server): CSP (kills inline-script XSS), no
//    third-party scripts without subresource integrity + a flag to kill them.

Interview Questions

Beginner: What are the three Core Web Vitals and what does each measure?

LCP (Largest Contentful Paint, target <2.5s): when the main content becomes visible — loading speed. INP (Interaction to Next Paint, <200ms): worst-case latency between a user interaction and the next frame — responsiveness. CLS (Cumulative Layout Shift, <0.1): how much the page jumps around — visual stability. Google uses them in ranking; users feel them as "fast," "responsive," and "calm."

Beginner: Why do feature flags matter beyond A/B testing?

They decouple deploy from release: code ships dark, releases to 1% → 10% → 100% with metrics watched per step, and rolls back by flipping a flag instead of reverting a deploy under pressure. They also provide kill switches for flaky external dependencies. The discipline: flags gate component composition, both paths stay tested, and finished flags get deleted.

Mid: Your P75 INP on the orders page regressed from 120ms to 380ms after Tuesday's deploy. Walk the diagnosis.

Release tag names the deploy → diff it. Reproduce with the Profiler on a throttled CPU on that page's interactions. Likely finds, checked in order: a state lift that widened re-render blast radius (L18 — "why did this render" on the biggest flame bar), a list that lost its memo/virtualization (L18), new synchronous work in a handler (L6 — long task in the performance panel), a heavy update that needs a transition (L21). Fix the structural cause, verify in RUM that P75 recovers, and add the interaction to the synthetic check so the same regression pages before users feel it.

Mid: Where can XSS still enter a React app despite JSX escaping, and what's the audit?

The bypasses: dangerouslySetInnerHTML (grep every instance — sanitize with DOMPurify or remove), URL-typed props built from user data (href/src accepting javascript: — validate protocols), third-party HTML (emails, CMS rich text, markdown renderers with raw-HTML options), and injected third-party scripts. Add CSP headers as the backstop so a slipped payload can't execute. Escaping covers interpolation; the audit covers everywhere you asked React not to escape.

Senior: Design the go-live plan for replacing a revenue-critical checkout with a rewrite.

Ship it dark behind a flag with both paths in CI (behavior tests + MSW for every failure mode — declines, timeouts, double-submits). Instrument the funnel identically in both versions (started → step completion → paid, plus error boundaries named per step) so comparison is apples-to-apples. Rollout: employees → 1% (watch error rate, INP, and — decisive — conversion delta per cohort) → 10% for a full week (weekly seasonality) → 50% → 100%. Rollback is the flag, rehearsed. Keep V1 deployable for a full cycle after 100%; then delete flag and V1 in one PR. The senior tells: cohort funnel metrics as the gate (not just errors), duration at 10% (traffic patterns, not vibes), and the deletion scheduled from day one.

Senior: A stakeholder asks "is the front-end production-ready?" — what's your actual assessment procedure?

Walk the subsystem gates with evidence, not vibes: containment (throw a test error per region — does it contain, report with sourcemapped stacks, and recover?), observability (are P75 vitals and error rates per route on a dashboard someone owns, tagged by release?), budgets (do bundle/vitals numbers fail CI?), state/data (kill the network mid-flow — does the UI degrade to designed states? refresh mid-checkout — does the URL restore the view?), quality (do tests fail on a behavior break? does keyboard-only complete every core flow?), security (bundle grep for secrets, sink audit, authz spot-check with a downgraded session), and change-safety (can we flag off the riskiest feature right now?). Each gate is a ten-minute check; "ready" is the list with checkmarks and named owners, and the honest answer names the unchecked boxes.

🏋️ Graded Exercises

1 (easy): Wire web-vitals reporting with route and release attribution, batching beacons with sendBeacon on page hide.
Solution 
import { onLCP, onINP, onCLS } from "web-vitals";

const queue = [];
function report(metric) {
  queue.push({ name: metric.name, value: metric.value, id: metric.id,
               route: location.pathname, release: __BUILD_ID__ });
}
onLCP(report); onINP(report); onCLS(report);

addEventListener("visibilitychange", () => {
  if (document.visibilityState === "hidden" && queue.length) {
    navigator.sendBeacon("/api/vitals", JSON.stringify(queue));
    queue.length = 0;                       // sendBeacon survives page teardown
  }
});
2 (easy): Review this flag usage and fix it: function PriceLabel({ price }) { const flags = useFlags(); const final = flags.newTax ? price * 1.18 : price * 1.15; return <span>{fmt(final)}</span>; }
Solution

// features/pricing/tax.ts: export const taxRate = flags => flags.newTax ? 0.18 : 0.15;

3 (medium): Add the CI a11y floor to the checkout form tests (jest-axe) and list three real issues it will catch and three it cannot.
Solution 
import { axe } from "jest-axe";
test("checkout form has no axe violations", async () => {
  const { container } = renderWithProviders(<CheckoutForm />);
  expect(await axe(container)).toHaveNoViolations();
});
// WILL catch: inputs without accessible labels, insufficient color contrast
// (computed styles), missing/invalid ARIA roles and attributes, buttons with
// no accessible name, duplicate ids.
// CANNOT catch: focus ORDER that jumps illogically, a focus trap that doesn't
// return focus (behavioral), whether the screen-reader narration makes SENSE,
// keyboard operability of custom widgets (needs interaction tests — L24 role
// queries + userEvent.keyboard cover part), motion/zoom rendering issues.
// Hence: automated floor + the manual keyboard/SR pass. Neither substitutes.
4 (medium): Your bundle-budget CI check fails: the main chunk grew 84KB. The diff added import { groupBy } from "lodash", a date-picker on the (rarely used) reports page, and an error-tracker SDK upgrade. Resolve each.
Solution 
// 1. lodash: full-package import pulls everything. Per-method import

// 2. Date-picker on a rare page: route-level lazy already exists? Then it's

//    the reports chunk (L20). If reports isn't split: split it now.
// 3. Tracker SDK: legit growth in critical-path code (it must load early to

//    and RAISE THE BUDGET EXPLICITLY in the same PR — budgets are managed
//    numbers, not immovable ones; the point is that growth is a decision.
5 (medium): Design the kill-switch for the third-party reviews widget on the product page: flag, boundary, and degraded UI working together.
Solution 
function ReviewsSection({ productId }) {
  const flags = useFlags();
  if (!flags.reviewsWidget)                       // kill switch: instant, no deploy
    return <ReviewsFallback productId={productId} />;   // degraded: static count + link

  return (
    <ErrorBoundary name="reviews" fallback={() => <ReviewsFallback productId={productId} />}>
      <Suspense fallback={<ReviewsSkeleton />}>
        <ThirdPartyReviews productId={productId} />   {/* lazy: its SDK stays out
                                                          of the main bundle (L20) */}
      </Suspense>
    </ErrorBoundary>
  );
}
// Three layers, three failure speeds: the boundary catches it breaking for one
// user (automatic, contained); the flag turns it off for everyone when their
// service degrades (operational, minutes); the same ReviewsFallback serves both
// so the degraded experience is designed once. Alert on the boundary's error
// rate → that's the signal to flip the flag.

🎯 Key Takeaways

  • Production readiness is subsystems with gates: containment, observability, budgets, flags, a11y, security — each an ordinary lesson applied with stakes, each with a named owner.
  • Observe real users: error rates per boundary, P75 vitals per route, funnel events — all tagged by release so every regression points at a diff.
  • Budgets live in CI and fail builds; vitals regressions have causal chains this track taught you to walk (LCP→assets/streaming, INP→render blast radius, CLS→sized layouts).
  • Flags separate deploy from release: composition-level gating, cohort-metric rollout ladders, rehearsed rollback, scheduled deletion.
  • Accessibility and security are release gates, not audits: CI floors plus the manual passes (keyboard-only, sink/secret grep, downgraded-session authz checks) tools can't do.
  • The track's real lesson: seniority is diagnosis — naming the force at play and the smallest structure that resolves it. You now have the full toolbox. Go build. 🎓
