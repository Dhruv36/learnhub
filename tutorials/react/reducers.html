useReducer & State Machines State Management

What you'll master: centralizing update logic in a reducer — events in, next state out — when it beats scattered setters, reducer design (action vocabulary, immutability, testability), lifting reducers with dispatch-down, and upgrading to explicit state machines for gnarly async flows. Prerequisites: State (immutable updates), Custom Hooks. This is also Redux's core idea — learn it here, the library version is a footnote.

Part 1 — From Setters to Events

// useState thinking: components MANIPULATE state directly, everywhere:
function Cart() {
  const [items, setItems] = useState([]);
  const add = p => setItems(is => { /* dedupe, increment qty, cap at stock… */ });
  const remove = id => setItems(is => is.filter(i => i.id !== id));
  const setQty = (id, q) => setItems(is => { /* clamp, remove-if-zero… */ });
  // …the RULES of the cart are smeared across handlers, some in children.
}

// useReducer thinking: components DESCRIBE WHAT HAPPENED; one function owns the rules:
function cartReducer(state, action) {
  switch (action.type) {
    case "added": {
      const existing = state.items.find(i => i.id === action.product.id);
      if (existing) {
        return { ...state, items: state.items.map(i =>
          i.id === action.product.id
            ? { ...i, qty: Math.min(i.qty + 1, i.stock) }   // the cap rule lives HERE, once
            : i) };
      }
      return { ...state, items: [...state.items, { ...action.product, qty: 1 }] };
    }
    case "removed":
      return { ...state, items: state.items.filter(i => i.id !== action.id) };
    case "qty_changed": {
      if (action.qty === 0)                                  // remove-at-zero rule: HERE
        return cartReducer(state, { type: "removed", id: action.id });
      return { ...state, items: state.items.map(i =>
        i.id === action.id ? { ...i, qty: Math.min(action.qty, i.stock) } : i) };
    }
    case "cleared":
      return { ...state, items: [] };
    default:
      throw new Error(`Unknown action: ${action.type}`);     // typos fail loudly
  }
}

function Cart() {
  const [state, dispatch] = useReducer(cartReducer, { items: [] });
  return <CartView items={state.items}
                   onAdd={p => dispatch({ type: "added", product: p })}
                   onRemove={id => dispatch({ type: "removed", id })} />;
}
The mental shift: actions are events in the domain's vocabulary ("added", "qty_changed"), not commands about state ("set_items"). The reducer is the single arbiter of what each event means. That inversion is why the pattern scales: new rule → edit one function; debugging → log actions and replay; testing → expect(reducer(s, a)).toEqual(s2), no React needed.

Part 2 — The Contract: Reducers Are Pure

// (state, action) => newState — and NOTHING else:
// ❌ no mutation:        state.items.push(x)         (same reference → no re-render;
//                                                     breaks time-travel/replay)
// ❌ no side effects:    fetch(), localStorage, Date.now(), Math.random()
//    (React may run reducers twice in Strict Mode — a fetch would fire twice;
//     randomness makes replay produce different results)
// ❌ no async:           reducers return VALUES, not promises.
//
// Where do effects go, then?
// 1. In the event handler, AROUND the dispatch:
async function handleSave() {
  dispatch({ type: "save_started" });
  try {
    const saved = await api.save(draft);
    dispatch({ type: "save_succeeded", saved });
  } catch (e) {
    dispatch({ type: "save_failed", error: e.message });
  }
}
// The async flow lives in the handler; each RESULT is just another event the
// reducer interprets. This handler shape is 90% of what Redux middleware does.
//
// 2. Or in useEffect, reacting to state the reducer produced (sync-to-external).
//
// Timestamps/ids: generate in the ACTION creator (handler), pass as payload —
// action = { type: "added", id: crypto.randomUUID(), at: Date.now() }.

Part 4 — Build It: Async Flow as a State Machine

The biggest payoff is async UI. Model a file-upload flow the boolean way and it leaks impossible states; model it as a machine and invalid combinations can't exist:

// States: idle → picking? no — keep it real:
// idle → uploading → success
//              ↘ error → (retry) → uploading
//        uploading → (cancel) → idle
function uploadReducer(state, action) {
  switch (state.status) {                       // ← branch on CURRENT STATE first:
    case "idle":                                //   a machine = state × event → state
      if (action.type === "upload_started")
        return { status: "uploading", file: action.file, progress: 0 };
      return state;                             // events that don't apply are IGNORED —
    case "uploading":                           // no "cancel" crash when already idle
      switch (action.type) {
        case "progressed":  return { ...state, progress: action.pct };
        case "succeeded":   return { status: "success", url: action.url };
        case "failed":      return { status: "error", file: state.file, error: action.error };
        case "cancelled":   return { status: "idle" };
        default:            return state;
      }
    case "error":
      if (action.type === "retried")
        return { status: "uploading", file: state.file, progress: 0 };
      if (action.type === "dismissed") return { status: "idle" };
      return state;
    case "success":
      if (action.type === "reset") return { status: "idle" };
      return state;
    default:
      return state;
  }
}

function Uploader() {
  const [state, dispatch] = useReducer(uploadReducer, { status: "idle" });
  const ctrlRef = useRef(null);

  async function start(file) {
    dispatch({ type: "upload_started", file });
    ctrlRef.current = new AbortController();
    try {
      const url = await api.upload(file, {
        signal: ctrlRef.current.signal,
        onProgress: pct => dispatch({ type: "progressed", pct }),
      });
      dispatch({ type: "succeeded", url });
    } catch (e) {
      if (e.name !== "AbortError")
        dispatch({ type: "failed", error: e.message });
    }
  }

  switch (state.status) {                        // UI = total function of machine state
    case "idle":      return <FilePicker onPick={start} />;
    case "uploading": return <Progress pct={state.progress}
                               onCancel={() => { ctrlRef.current.abort();
                                                 dispatch({ type: "cancelled" }); }} />;
    case "error":     return <RetryPanel error={state.error}
                               onRetry={() => start(state.file)}
                               onDismiss={() => dispatch({ type: "dismissed" })} />;
    case "success":   return <Done url={state.url}
                               onReset={() => dispatch({ type: "reset" })} />;
  }
}
// Notice what CANNOT happen: progress on an idle upload, success with an error
// banner, double-start while uploading (upload_started ignored unless idle).
// The reducer refuses at one chokepoint — no defensive ifs scattered in the UI.
// Each status also carries exactly its own data (file+progress / url / error):
// the discriminated-union modeling from the State lesson, now with transitions.

// 2. Reducer + context = "mini-Redux" for a feature (full recipe next lesson):
const TasksContext = createContext(null);
const TasksDispatchContext = createContext(null);      // two contexts — readers of
function TasksProvider({ children }) {                  // dispatch-only never re-render
  const [tasks, dispatch] = useReducer(tasksReducer, []);
  return (
    <TasksContext.Provider value={tasks}>
      <TasksDispatchContext.Provider value={dispatch}>
        {children}
      </TasksDispatchContext.Provider>
    </TasksContext.Provider>
  );
}

// 3. Splitting: one reducer per FEATURE, not one giant app reducer — compose
//    by slice if they truly share events:
function appReducer(state, action) {
  return {
    cart:   cartReducer(state.cart, action),
    upload: uploadReducer(state.upload, action),
  };
}

// 4. Immer for deep updates — mutate a draft, get immutable results:
import { produce } from "immer";
const reducer = produce((draft, action) => {
  switch (action.type) {
    case "toggled": {
      const t = draft.items.find(i => i.id === action.id);
      t.done = !t.done;                                  // "mutation" — safe on the draft
      break;
    }
  }
});
// Worth it when nesting is deep; overkill for flat state. (Redux Toolkit bakes it in.)
Where Redux fits: Redux = this exact pattern (one store, reducers, actions) as an app-wide singleton with devtools (action log, time-travel), middleware, and ecosystem. Modern Redux Toolkit is ergonomic — but reach for it only when state is genuinely global and event-sourced audit trails matter. Feature-local reducer+context covers most apps; External Stores weighs the options honestly.

Interview Questions

Beginner: What does useReducer give you over useState?

A single pure function owning all update logic: components dispatch descriptive actions, the reducer computes the next state. Benefits: related fields update atomically, rules live in one testable place, children get one stable dispatch instead of many setters, and the action log documents every way state can change.

Beginner: Why must a reducer be pure, in practical terms?

React may invoke it more than once per dispatch (Strict Mode double-invocation, interrupted renders), so side effects would duplicate; purity makes reducer(state, action) deterministic, enabling unit tests as plain assertions, action-replay debugging, and re-render detection via reference change — mutation defeats all three at once.

Mid: How do async operations work with reducers if reducers can't be async?

The async lives in the event handler (or a custom hook): dispatch a "started" event, await the work, dispatch "succeeded" or "failed" with the result as payload. The reducer only interprets those completion events — synchronously. This keeps every state transition in the reducer while the handler orchestrates timing. Middleware (thunks/sagas) is this pattern given a formal home in Redux.

Mid: Why is dispatch safe to pass to memoized children without useCallback?

React guarantees the dispatch function's identity is stable for the component's lifetime — same reference every render, like state setters. A memoized child receiving only dispatch and primitive props will never re-render from callback churn, which is exactly why "dispatch down, events up" beats drilling per-action callbacks.

Senior: When do you structure a reducer as an explicit state machine, and what does it buy?

When behavior depends on where you are, not just what happened — async flows (upload, checkout, auth), multi-step wizards, media players. Branch on current status first, events second; non-applicable events return state unchanged. Buys: impossible states are unrepresentable (each status carries only its own data), late/duplicate async events are ignored at one chokepoint instead of guarded everywhere, the transition table is documentation, and edge cases become enumerable (states × events) rather than combinatorial (2^booleans).

Senior: Critique: a codebase where every form field dispatches {type: "SET_FIELD", field, value} to a global Redux store.

Three smells. Command-actions: SET_FIELD encodes no domain meaning — the reducer is a glorified Object.assign, so the pattern's benefits (rules centralization, meaningful logs) evaporate while its ceremony remains. Wrong scope: keystroke-level state is ephemeral and local; globalizing it re-renders distant subscribers per keypress and pollutes the action history. Wrong tool: uncontrolled inputs or local component state handle typing; promote to the store only at meaningful boundaries (submitted, draft-saved) as real events. Reducers earn their keep on decisions, not dictation.

Solution 
function counterReducer(state, action) {
  switch (action.type) {
    case "incremented": return { count: state.count + 1 };
    case "decremented": return { count: Math.max(0, state.count - 1) };
    case "reset":       return { count: 0 };
    default: throw new Error(`Unknown action: ${action.type}`);
  }
}

3 (medium): Convert this useState cluster to one reducer: a filter panel with query, category, page — where changing query or category must reset page to 1.
Solution 
function filtersReducer(state, action) {
  switch (action.type) {
    case "query_changed":    return { ...state, query: action.query, page: 1 };
    case "category_changed": return { ...state, category: action.category, page: 1 };
    case "page_changed":     return { ...state, page: action.page };
    default: throw new Error(`Unknown action: ${action.type}`);
  }
}
// The page-reset rule was previously a landmine: every future query setter had
// to remember setPage(1). Now the coupling is structural — one event, one
// transition, impossible to forget.

Solution 
const p = { id: 1, name: "Cam", stock: 2 };
let s = { items: [] };

5 (medium): A "cancel" click during upload sometimes flashes the success screen anyway (response landed first). Show how the Part 4 machine already handles the reverse (response after cancel) and patch the handler for this ordering.
Solution 
// Reverse case (cancel, then late "succeeded" dispatch): status is "idle" —
// the idle branch ignores "succeeded". The machine absorbs it. ✅
// This case (success lands, THEN user's cancel click dispatches "cancelled"):
// status is already "success", whose branch ignores "cancelled". ✅ Also absorbed!
// The flash happens BEFORE the click — it's a UX race, not a state bug. Patch at
// the source: the handler aborts the request, so make abort win the race:
onCancel={() => {
  ctrlRef.current.abort();                 // kills the fetch — succeeded never dispatches
  dispatch({ type: "cancelled" });
}}
// Lesson: the machine guarantees CONSISTENCY for any event order; eliminating
// the unwanted order itself is the handler's job (abort the underlying work).
6 (hard): Model a 3-step checkout wizard (cart → shipping → payment → confirmed) as a reducer state machine: "continued" and "went_back" navigate, each step validates before advancing, "edited" from confirmation jumps to a named step, and payment failure keeps data but shows an error.
Solution 
const ORDER = ["cart", "shipping", "payment"];

function checkoutReducer(state, action) {
  const { step } = state;
  switch (action.type) {
    case "continued": {
      const error = validate(step, state.data);          // pure helper per step
      if (error) return { ...state, error };
      const i = ORDER.indexOf(step);
      return i === ORDER.length - 1
        ? { ...state, step: "confirming", error: null }
        : { ...state, step: ORDER[i + 1], error: null };
    }
    case "went_back": {
      const i = ORDER.indexOf(step);
      return i > 0 ? { ...state, step: ORDER[i - 1], error: null } : state;
    }
    case "field_changed":
      return { ...state, data: { ...state.data, [action.name]: action.value } };
    case "edited":                                        // from confirmation screen
      return step === "confirming" ? { ...state, step: action.target } : state;
    case "payment_succeeded":
      return step === "confirming" ? { step: "confirmed", data: state.data } : state;
    case "payment_failed":
      return step === "confirming"
        ? { ...state, error: action.error }               // data intact, error shown
        : state;
    default: return state;
  }
}
// Navigation, validation gating, and payment outcomes all in one auditable
// function; the UI is a switch on state.step. Adding a step = editing ORDER
// and one validate case.

🎯 Key Takeaways

  • useReducer inverts state management: components dispatch domain events; one pure function decides what they mean. Rules centralize, handlers thin out.
  • Reducers are pure — no mutation, effects, or randomness. Async orchestration lives in handlers dispatching started/succeeded/failed events.
  • Choose it when updates carry rules, fields change together, or many children mutate the same state — the stable dispatch replaces callback bundles for free.
  • State-machine style (branch on status, then event) makes impossible states unrepresentable and absorbs late/duplicate async events at one chokepoint.
  • Actions are events ("qty_changed"), never commands ("set_qty") — command-actions collapse the pattern into ceremony around a setter.
  • Reducer + split contexts = feature-scoped state management; Redux is the same idea app-wide with devtools — the reducer itself ports unchanged.
