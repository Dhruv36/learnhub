Composition Patterns Patterns & Quality

What you'll master: the escalation ladder of component APIs — slots, compound components (Tabs/Tab done properly with scoped context), render props, and headless components — plus portals for escaping layout, and how to recognize which pattern a problem actually needs. Prerequisites: JSX Deep Dive (element/component-as-prop), Context (scoped providers), Custom Hooks.

Part 1 — The Ladder: Configuration → Composition

// Every component API evolves under the same pressure. Watch a Card rot:
<Card title="Invoice" />                                     // v1: fine
<Card title="Invoice" subtitle="March" icon="doc" />          // v2: growing
<Card title="Invoice" titleSize="lg" iconPosition="right"
      showDivider headerAction="export" headerActionVariant…  // v5: config hell —
// every layout wish becomes a prop; the component reimplements JSX badly.

// The cure is always the same: stop DESCRIBING content, ACCEPT it. The ladder:
// 1. children            one content hole                    (JSX Deep Dive)
// 2. slots               several named holes                 (Part 2)
// 3. compound components structured children that coordinate  (Part 3)
// 4. render props        receiver sends DATA back to caller  (Part 4)
// 5. headless            you own the logic, caller owns ALL UI (Part 5)
// Climb only as high as the problem forces you — each rung costs API complexity.

Part 2 — Slots: Named Holes

function Card({ header, actions, footer, children }) {
  return (
    <section className="card">
      {header && <header className="card-hd">{header}</header>}
      <div className="card-body">{children}</div>
      {actions && <div className="card-actions">{actions}</div>}
      {footer && <footer className="card-ft">{footer}</footer>}
    </section>
  );
}

Part 3 — Compound Components: a Family Speaking Through Context

// When the pieces must COORDINATE (which tab is active?), slots aren't enough —
// you want  and its children to share state invisibly:

const TabsContext = createContext(null);

function Tabs({ value, defaultValue, onValueChange, children }) {
  const [internal, setInternal] = useState(defaultValue);      // dual-mode:
  const active = value !== undefined ? value : internal;       // State Architecture
  const select = id => { if (value === undefined) setInternal(id); onValueChange?.(id); };
  const ctx = useMemo(() => ({ active, select }), [active]);
  return <TabsContext.Provider value={ctx}><div className="tabs">{children}</div></TabsContext.Provider>;
}

function useTabs(part) {
  const ctx = useContext(TabsContext);
  if (!ctx) throw new Error(` must be used inside `);
  return ctx;
}

Tabs.List = function TabsList({ children }) {
  return <div role="tablist">{children}</div>;
};
Tabs.Tab = function Tab({ id, children }) {
  const { active, select } = useTabs("Tab");
  return (
    <button role="tab" aria-selected={active === id}
            onClick={() => select(id)}>{children}</button>
  );
};
Tabs.Panel = function Panel({ id, children }) {
  const { active } = useTabs("Panel");
  return active === id ? <div role="tabpanel">{children}</div> : null;
};

// The caller composes freely — order, wrappers, extra elements all allowed:
<Tabs defaultValue="overview">
  <Tabs.List>
    <Tabs.Tab id="overview">Overview</Tabs.Tab>
    <Tabs.Tab id="billing">Billing <Badge>2</Badge></Tabs.Tab>   {/* try THAT with a tabs={[...]} prop */}
  </Tabs.List>
  <Tabs.Panel id="overview"><Overview /></Tabs.Panel>
  <Tabs.Panel id="billing"><BillingHistory /></Tabs.Panel>
</Tabs>
Why context, not cloneElement: older compound components walked children injecting props via React.cloneElement — which breaks the moment a caller wraps a Tab in a div, a Fragment, or their own component. Scoped context works at any depth, survives any wrapper, and is the pattern every modern library (Radix, Headless UI, React Aria) uses. This is the Context lesson's "scoped provider" use-case as a design tool.

Part 4 — Render Props: When the Receiver Has Data the Caller Needs

// Slots pass finished UI down. But what if the CONTAINER computes something the
// caller's markup needs — like each row's data in a virtualized list?
function VirtualList({ items, rowHeight, renderRow }) {
  const { visible, offsetFor } = useVirtualWindow(items, rowHeight);   // container's knowledge
  return (
    <div className="viewport">
      {visible.map(item => (
        <div key={item.id} style={{ transform: `translateY(${offsetFor(item)}px)` }}>
          {renderRow(item)}                       {/* caller's markup × container's data */}
        </div>
      ))}
    </div>
  );
}

Part 5 — Headless Components: All Logic, Zero Markup

// The endpoint of the ladder: ship behavior as a hook; the caller owns every pixel.
function useDisclosure({ defaultOpen = false } = {}) {
  const [open, setOpen] = useState(defaultOpen);
  const buttonRef = useRef(null);

  const toggle = useCallback(() => setOpen(o => !o), []);
  const close  = useCallback(() => { setOpen(false); buttonRef.current?.focus(); }, []);

  // PROP GETTERS — the headless idiom: pre-wired prop bundles callers spread on:
  const getButtonProps = (props = {}) => ({
    ref: buttonRef, "aria-expanded": open, onClick: e => { props.onClick?.(e); toggle(); },
    ...props,
  });
  const getPanelProps = (props = {}) => ({
    hidden: !open,
    onKeyDown: e => { props.onKeyDown?.(e); if (e.key === "Escape") close(); },
    ...props,
  });
  return { open, toggle, close, getButtonProps, getPanelProps };
}

// Caller: any markup, any styling system — the behavior contract rides along:
function FAQItem({ q, a }) {
  const d = useDisclosure();
  return (
    <li className="faq">
      <button {...d.getButtonProps({ className: "faq-q" })}>{q} {d.open ? "▾" : "▸"}</button>
      <div {...d.getPanelProps({ className: "faq-a" })}>{a}</div>
    </li>
  );
}
// This is how Downshift, TanStack Table, React Aria work: the hard parts
// (keyboard nav, ARIA, focus management, state machines) are packaged; the
// markup — the part every design system wants different — is yours.
// Choose headless when: multiple visual variants of one behavior, a design
// system with its own DOM opinions, or logic complex enough to be the product.

Part 6 — Portals: Composition That Escapes the DOM

import { createPortal } from "react-dom";

function Modal({ open, onClose, children }) {
  if (!open) return null;
  return createPortal(
    <div className="overlay" onClick={onClose}>
      <div role="dialog" aria-modal="true" onClick={e => e.stopPropagation()}>
        {children}
      </div>
    </div>,
    document.body                       // render THERE, but live HERE in the React tree
  );
}
// Why: overflow:hidden ancestors, z-index stacking contexts, and transform
// containers clip/trap absolutely-positioned UI. Portals opt the DOM out while
// keeping the React relationships: context flows in (a themed modal stays
// themed), events bubble up the COMPONENT tree (Events lesson), state is normal.
// Users: modals, toasts, dropdowns/popovers, tooltips. Pair with the focus
// management from Refs ex. 7 — a portal without a focus trap is a keyboard trap.

Interview Questions

Beginner: What does "composition over configuration" mean for component APIs?

Instead of describing desired content through props (title, iconPosition, headerActionVariant — configuration), the component exposes holes (children, named slots) and callers pass real JSX. The component keeps structure and behavior; callers keep content and layout freedom. APIs stay small because every new content wish is just different JSX, not a new prop.

Beginner: What is a portal and what stays "normal" about a portaled subtree?

createPortal(jsx, domNode) renders the subtree into a different DOM container (usually document.body) while it remains a normal child in the React tree: context reaches it, state and lifecycle are ordinary, and events bubble to React ancestors. Only the DOM location changes — which is exactly what escaping overflow/z-index clipping requires.

Mid: Design the API for a Select supporting custom option rendering, groups, and a custom trigger. Which pattern and why?

Compound components: <Select> owning state via scoped context, with Select.Trigger, Select.Options, Select.Group, Select.Option value=…. Groups and custom rendering fall out of free composition (anything can wrap anything); the trigger is just another part reading context. A configuration API (options={[…]} renderOption={…} renderGroup={…} renderTrigger={…}) would need a render prop per region — compound components are that, with JSX as the syntax. Add dual-mode value control and you've described Radix's actual Select.

Mid: Why did hooks kill most render-prop usage, and where does the pattern remain correct?

Render props were how stateful logic got shared pre-hooks — but consuming three of them nested three wrapper components, creating fake hierarchy and prop-name collisions. A hook delivers the same values as plain variables. The pattern remains correct where the receiver controls placement/iteration of caller markup: per-row rendering in lists/tables, virtualizers passing position data, autocompletes weaving items into managed DOM. The test: does the component call your function with data, per item, in positions it owns? Render prop. Just sharing state? Hook.

Senior: What are prop getters in headless design and what contract must they honor?

Functions like getButtonProps(userProps) returning pre-wired prop bundles (ARIA attributes, refs, handlers) the caller spreads onto their own elements. The contract: merge, never clobber — compose the caller's handlers with yours (call theirs first, respect defaultPrevented), merge classNames/styles, let caller props win on conflicts you don't own, and keep getter output stable enough for memoized consumers. Getters are the headless pattern's answer to "how do behavior and caller markup both attach to one element without either knowing the other's details."

Senior: Your design system's Modal is used 90 ways and drowning in props. Lay out the migration.

Audit props into three buckets: structure (title, footer, closeX → become slots/compound parts), behavior (open/onClose, focus trap, Escape — stays core, possibly a headless useModal), and variants (size/tone — stay props; enums are fine for finite design tokens). Ship Modal.Header/Body/Footer compound parts over the same context; reimplement the legacy prop API as a thin wrapper composing the new parts — zero-day compatibility, one codemod per prop cluster later. Extract useModal (portal + trap + scroll lock + Escape) for the teams that need fully custom chrome. The instinct to check: every migration step must keep both APIs rendering pixel-identical — snapshot the 90 usages first.

3 (medium): Build a compound <Disclosure> (Disclosure.Button, Disclosure.Panel) with dual-mode open control and the throwing-hook guard.
Solution 
const DiscCtx = createContext(null);
const useDisc = () => {
  const ctx = useContext(DiscCtx);
  if (!ctx) throw new Error("Disclosure parts need ");
  return ctx;
};

function Disclosure({ open, defaultOpen = false, onOpenChange, children }) {
  const [internal, setInternal] = useState(defaultOpen);
  const isOpen = open !== undefined ? open : internal;
  const toggle = () => {
    if (open === undefined) setInternal(o => !o);
    onOpenChange?.(!isOpen);
  };
  const ctx = useMemo(() => ({ isOpen, toggle }), [isOpen]);
  return <DiscCtx.Provider value={ctx}>{children}</DiscCtx.Provider>;
}
Disclosure.Button = ({ children }) => {
  const { isOpen, toggle } = useDisc();
  return <button aria-expanded={isOpen} onClick={toggle}>{children}</button>;
};
Disclosure.Panel = ({ children }) => {
  const { isOpen } = useDisc();
  return isOpen ? <div>{children}</div> : null;
};

6 (hard): Extend Part 5's useDisclosure into useMenu — trigger + floating list with full keyboard support (arrows, Home/End, Escape, focus return) — exposing only prop getters. List what each getter wires.
Solution 
function useMenu(itemCount) {
  const [open, setOpen] = useState(false);
  const [active, setActive] = useState(0);
  const triggerRef = useRef(null);

  const close = useCallback(() => {
    setOpen(false);
    triggerRef.current?.focus();                      // focus contract (Refs ex. 7)
  }, []);

  const getTriggerProps = (p = {}) => ({
    ref: triggerRef, "aria-haspopup": "menu", "aria-expanded": open,
    onClick: e => { p.onClick?.(e); setOpen(o => !o); setActive(0); },
    onKeyDown: e => { p.onKeyDown?.(e);
      if (e.key === "ArrowDown") { e.preventDefault(); setOpen(true); setActive(0); } },
    ...p,
  });
  const getMenuProps = (p = {}) => ({
    role: "menu", hidden: !open,
    onKeyDown: e => {
      p.onKeyDown?.(e);
      const max = itemCount - 1;
      if (e.key === "ArrowDown") { e.preventDefault(); setActive(a => Math.min(a + 1, max)); }
      if (e.key === "ArrowUp")   { e.preventDefault(); setActive(a => Math.max(a - 1, 0)); }
      if (e.key === "Home")      { e.preventDefault(); setActive(0); }
      if (e.key === "End")       { e.preventDefault(); setActive(max); }
      if (e.key === "Escape")    close();
    },
    ...p,
  });
  const getItemProps = (index, p = {}) => ({
    role: "menuitem", tabIndex: index === active ? 0 : -1,     // roving tabindex
    "data-active": index === active || undefined,
    onPointerEnter: e => { p.onPointerEnter?.(e); setActive(index); },
    onClick: e => { p.onClick?.(e); close(); },
    ...p,
  });
  return { open, active, close, getTriggerProps, getMenuProps, getItemProps };
}
// Trigger: ARIA popup wiring + open-on-ArrowDown. Menu: roving keyboard nav.
// Items: roving tabindex + hover/click coordination. The caller renders any
// markup and spreads — every hard behavior travels in the getters.

🎯 Key Takeaways

  • Component APIs rot toward config explosions; the cure is accepting JSX — climb the ladder (children → slots → compound → render props → headless) only as far as the problem forces.
  • Slots split ownership cleanly: component keeps structure and behavior, callers keep content — most flexibility needs end here.
  • Compound components coordinate through scoped context (never cloneElement), with throwing hooks guarding misuse and dual-mode state control at the root.
  • Render props survive hooks precisely where receivers weave caller markup into owned positions with per-item data; for pure logic sharing, a hook.
  • Headless = behavior as a hook with merging prop getters — ARIA, keyboard, and focus packaged; markup fully caller-owned.
  • Portals relocate DOM without leaving the React tree — context, state, and event bubbling intact; always paired with focus management.
