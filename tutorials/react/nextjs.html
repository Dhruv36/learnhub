Part 2 — The Rendering & Caching Matrix

// Every route lands in one of three modes — decided by what your code does:

// STATIC (default): rendered at BUILD time, served from CDN. Applies when the
// route uses no request-time data. Fastest possible; content frozen at deploy.
export default async function AboutPage() {
  const team = await getTeam();          // build-time fetch → static HTML
}

// DYNAMIC: rendered per REQUEST — triggered by using request data:
// cookies(), headers(), searchParams, or opting out of caching.
export default async function AccountPage() {
  const session = await getSession(cookies());   // per-user → dynamic
}

// ISR (Incremental Static Regeneration): static + a shelf life:
export const revalidate = 3600;          // serve static; re-render at most hourly
// First request after expiry: stale page served, regeneration in background —
// stale-while-revalidate (L10's concept) applied to whole PAGES.

// On-demand revalidation — the mutation story (pairs with server actions):
revalidatePath("/products/42");          // after a mutation touching it
revalidateTag("products");               // fetch(url, { next: { tags: ["products"] }})
                                          // → invalidate every fetch tagged "products"
// fetch() in server components is cache-aware:
fetch(url)                                // cached per route mode
fetch(url, { cache: "no-store" })         // always fresh (forces dynamic)
fetch(url, { next: { revalidate: 60 } })  // per-request-cache TTL
The senior framing: Next.js collapses "server data freshness" into the question you already know from L10 — how stale is acceptable? — answered per page (revalidate), per fetch (next.revalidate), or per event (revalidateTag after mutations). Debugging "why is my page showing old data" is always answering "which cache: build output, full-route, fetch cache, or router cache?" — check them in that order.

Part 4 — Route Handlers & Middleware: the Non-UI Server

// app/api/webhooks/stripe/route.ts — for callers that AREN'T your pages
// (webhooks, mobile apps, third parties). Your own UI mostly uses server
// components (reads) + actions (writes) instead:
import { NextResponse } from "next/server";

export async function POST(request) {
  const sig = request.headers.get("stripe-signature");
  const event = stripe.webhooks.constructEvent(await request.text(), sig, SECRET);
  if (event.type === "invoice.paid") await activateSubscription(event);
  return NextResponse.json({ received: true });
}

// middleware.ts (project root) — runs BEFORE routing, on the edge, every request:
export function middleware(request) {
  const session = request.cookies.get("session");
  if (!session && request.nextUrl.pathname.startsWith("/account"))
    return NextResponse.redirect(new URL("/login", request.url));   // L17's guard,
  return NextResponse.next();                                        // server-side
}
export const config = { matcher: ["/account/:path*", "/admin/:path*"] };
// Middleware = fast, shallow checks (cookie exists? locale? redirect?). It runs
// on EVERY matched request with tight limits — real authorization still lives
// in the data access layer (actions/handlers verify the session's CLAIMS).
// The L17 lesson holds: client guards are UX, middleware is routing hygiene,
// authorization is where the data is.

Interview Questions

Beginner: What do layout.tsx, loading.tsx, and error.tsx each map to in plain React?

layout.tsx is a nested persistent wrapper — the Outlet pattern: it keeps state and doesn't remount when child pages change. loading.tsx auto-wraps the segment in a Suspense boundary with that file as fallback. error.tsx is an error boundary (hence "use client") with a reset prop. The framework turns the track's three seam components into filenames.

Beginner: Static vs dynamic rendering — what flips a route to dynamic?

Static (default): rendered at build, served from CDN, same HTML for everyone. A route becomes dynamic when it consumes request-specific data — cookies(), headers(), searchParams — or opts out of caching (no-store, dynamic = "force-dynamic"). ISR is the middle: static output with a revalidation shelf life.

Mid: Walk the "user edits a product, list still shows the old name" debugging path.

It's a cache-layer hunt, outermost first: (1) is the list page static/ISR and simply not revalidated? — the mutation must call revalidatePath("/products") or share a tag; (2) is a cached fetch feeding it? — tag it and revalidateTag; (3) client-side router cache showing a stale segment? — router.refresh() after the action settles (actions revalidating paths handle this). The fix is declaring the mutation's blast radius — L10's invalidation concept at page scale — not sprinkling no-store until it works (which silently buys dynamic rendering everywhere).

Mid: How do generateStaticParams and ISR combine for a 100k-product catalog?

generateStaticParams pre-builds the pages worth paying for at deploy (top sellers, landing-linked items) — instant CDN hits. The long tail isn't built; each page renders on first request, then persists as static with revalidate giving it a freshness window. Combined with revalidateTag on price/stock mutations, you get near-static economics over a huge catalog with bounded staleness — no build that takes hours, no fully-dynamic serving costs.

Senior: Where does middleware fit in the auth story, and what must it never be?

Middleware is the routing-hygiene tier: cheap, edge-executed checks — session cookie present? → redirect to login preserving the destination; locale/AB routing; header shaping. It must never be the authorization system: it runs before routing with limited runtime, matchers drift from real route surfaces, and non-page entry points (route handlers, server actions, direct fetches) don't care about your redirect. Authorization lives where data is accessed — every action/handler verifies the session and the specific permission (L26's action threat model). Defense in depth: middleware for UX-speed redirects, data layer for actual security, client guards (L17) for polish.

Senior: How do you keep a Next.js codebase from becoming framework-locked mud?

Apply L25 unchanged: app/ contains thin composition roots — a page imports its feature's public components/hooks and wires params; features/ owns components, actions, queries, and tests with framework imports concentrated in known files (actions, route handlers); shared/ stays domain- and framework-free. Server actions delegate to plain service functions (testable without the framework; portable to any backend). The test: could you swap the routing/rendering shell without touching business logic? If reducers, components, and services import next/* everywhere, no — and every framework migration in your future just became a rewrite.

6 (hard): Build the Stripe webhook route handler with signature verification, idempotency, and correct status codes — and explain why this can't be a server action.
Solution 
// app/api/webhooks/stripe/route.ts
export async function POST(request) {
  const payload = await request.text();                  // RAW body — verify before parse
  const sig = request.headers.get("stripe-signature");
  let event;
  try {
    event = stripe.webhooks.constructEvent(payload, sig, process.env.STRIPE_WHSEC);
  } catch {
    return new Response("Invalid signature", { status: 400 });   // reject forgeries
  }

  const seen = await db.webhookEvent.findUnique({ where: { id: event.id } });
  if (seen) return Response.json({ received: true });    // IDEMPOTENT: Stripe retries —
  await db.webhookEvent.create({ data: { id: event.id } }); // process each id once

  switch (event.type) {
    case "invoice.paid":       await activateSubscription(event.data.object); break;
    case "customer.deleted":   await cleanupCustomer(event.data.object); break;
  }
  return Response.json({ received: true });              // 2xx fast, or Stripe re-sends
}
// Why not a server action: actions are RPC endpoints for YOUR React clients —
// invoked via the framework's serialization protocol, tied to form/transition
// semantics. Stripe speaks plain HTTP POST with its own signing scheme and
// retry contract; it needs a raw-body HTTP endpoint with explicit status codes.
// Route handlers exist precisely for non-React callers.

🎯 Key Takeaways

  • The App Router files are this track's seams as conventions: layout=Outlet, loading=Suspense, error=boundary — with automatic per-segment wiring and prefetching links.
  • Three rendering modes per route — static (build), ISR (static + shelf life), dynamic (per request) — flipped by what you read; request-data reads sink to the smallest component.
  • Freshness is declared: revalidate for time, revalidateTag/Path for events after mutations; stale-data debugging = walking the cache layers outermost-in.
  • Middleware redirects, layouts render decisions, the data layer authorizes — three tiers, and only the last one is security.
  • next/image and next/font are the cheap CWV wins (LCP, CLS); generateStaticParams + ISR give static economics on huge dynamic catalogs.
  • Keep app/ thin over L25's feature folders and actions delegating to plain services — the framework hosts your architecture; it shouldn't become it.
