Pub/Sub & Streams Intermediate

Redis does messaging two ways: Pub/Sub (fire-and-forget broadcast) and Streams (a durable, replayable log with consumer groups — essentially a lightweight Kafka). Knowing the difference, and specifically why Pub/Sub loses messages, is a common interview discriminator.

Pub/Sub — Fire-and-Forget Broadcast

// Subscriber:
SUBSCRIBE news                  // (or PSUBSCRIBE news.* for patterns)
// Publisher:
PUBLISH news "breaking!"        // delivered to all CURRENTLY-connected subscribers
// Every subscriber connected AT PUBLISH TIME gets the message. That's it.
⚠️ Pub/Sub has NO persistence and NO delivery guarantee. If a subscriber is disconnected (crash, network blip, slow) when a message is published, it never sees it — messages are not stored, not replayed, not acknowledged. It's "at-most-once" fire-and-forget. Great for ephemeral real-time fan-out (live notifications, chat presence, cache-invalidation signals) where a missed message is acceptable. NOT for anything that must not be lost — use Streams.

Streams — a Durable, Replayable Log

XADD events * type "signup" userId 42       // append an entry; * = auto id (timestamp-seq)
XLEN events                                  XRANGE events - +      // read entries (persisted!)
XREAD COUNT 10 STREAMS events 0             // read from the beginning (or a specific id)
XREAD BLOCK 5000 STREAMS events $           // block, wait for NEW entries (like tail -f)
// Unlike Pub/Sub, entries are STORED in the stream — a consumer that reconnects can read what it
// missed. It's an append-only log (like Kafka), with ids for resuming exactly where you left off.

Consumer Groups — Scalable, Reliable Processing

XGROUP CREATE events workers 0              // create a consumer group
XREADGROUP GROUP workers c1 COUNT 1 STREAMS events >   // c1 reads NEW messages for the group
// ... process the message ...
XACK events workers 1526984-0              // ACKNOWLEDGE it (mark done)
XPENDING events workers                     // messages delivered but not yet ACKed
XCLAIM events workers c2 60000 1526984-0   // reassign a stuck message to another consumer
// A consumer GROUP lets multiple workers SHARE a stream: each message goes to exactly ONE
// consumer in the group (load balancing), with ACKs and pending-message tracking so a crashed
// consumer's unacked messages can be reclaimed (XCLAIM). This is at-least-once processing.

Pub/Sub vs Streams — the Decision

Pub/SubStreams
PersistenceNone (ephemeral)Stored (replayable log)
Missed while offline?Lost foreverRead later from your last id
DeliveryAt-most-once, fire-and-forgetAt-least-once (with ACKs)
Consumer scalingAll subscribers get every messageConsumer groups load-balance across workers
Use forLive notifications, presence, cache-invalidation signalsJob queues, event sourcing, reliable pipelines

Rule of thumb: if losing a message is fine → Pub/Sub (simpler, lower overhead). If messages must be processed reliably, replayed, or load-balanced → Streams. Streams are the modern default for queue-like work (superseding the list-based queue from the data-structures lesson).

Interview Questions

Q1: Why can Redis Pub/Sub lose messages, and when is that acceptable?

Pub/Sub delivers each published message only to subscribers connected at that instant; it stores nothing and offers no acknowledgment or replay. So if a subscriber is disconnected, slow, or crashed when the message is published, it never receives it — at-most-once, fire-and-forget. That's acceptable for ephemeral real-time signals where a miss is harmless or self-correcting: live notifications, typing/presence indicators, dashboards, and cache-invalidation broadcasts (a missed invalidation is bounded by the TTL). For anything that must not be lost — jobs, events, financial messages — you need durability and acknowledgments, which means Redis Streams (or a dedicated broker like Kafka/RabbitMQ).

Q2: How do Redis Streams with consumer groups provide reliable processing?

A stream is an append-only, persisted log; entries stay until trimmed, so consumers can read from any id and replay what they missed. A consumer group (XGROUP) lets multiple workers share the stream: XREADGROUP delivers each new message to exactly ONE consumer in the group (load balancing), and Redis tracks it as "pending" until the consumer XACKs it. If a consumer crashes before acking, its pending messages remain visible (XPENDING) and can be reassigned to another consumer with XCLAIM/XAUTOCLAIM — giving at-least-once delivery with progress tracking. This makes Streams a Redis-native, Kafka-lite for reliable job/event processing, unlike the fire-and-forget Pub/Sub.

Q3: Redis Streams vs Kafka — when do you outgrow Redis?

Streams are excellent for moderate-throughput, reliable messaging with the operational simplicity of already having Redis: consumer groups, acks, replay, and low latency. You reach for Kafka when you need very high sustained throughput and horizontal partitioning across many brokers, long-term retention of large event histories (Redis is memory-bound, so retaining terabytes is impractical), strong ordering/partitioning semantics at scale, or a broad ecosystem of connectors and stream processing. In short: Redis Streams for real-time, bounded, reliable pipelines where Redis is already in your stack; Kafka for the big-data-scale, durable event backbone. Many systems use both at different tiers.

🏋️ Practice: SUBSCRIBE in one session and PUBLISH in another; disconnect the subscriber, publish, reconnect, and confirm the message was lost; XADD entries to a stream and XREAD from 0 to see persistence; create a consumer group and process + XACK a message across two consumers; check XPENDING and XCLAIM a stuck message; cap a stream with MAXLEN.

🎯 Key Takeaways

  • Pub/Sub = ephemeral fire-and-forget broadcast: no persistence, at-most-once — misses if offline.
  • Streams = durable, replayable append-only log with ids for resuming — read what you missed.
  • Consumer groups (XREADGROUP/XACK/XPENDING/XCLAIM) load-balance across workers with at-least-once delivery.
  • Pub/Sub for tolerable-loss real-time signals; Streams for reliable jobs/events/pipelines.
  • Cap streams (MAXLEN); ack messages; reach for Kafka at very high throughput / long retention.
