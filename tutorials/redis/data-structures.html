Hashes, Lists, Sets & Sorted Sets Basics→Intermediate

Redis's power is its data structures — choosing the right one for your access pattern is the core skill. Each structure has O(1) or O(log n) operations that would be awkward or slow to build elsewhere. This lesson covers the four workhorses and, crucially, when to reach for each.

Hashes — Field→Value Maps (objects)

HSET user:1 name "Asha" age 30 plan "pro"
HGET user:1 name                HGETALL user:1          HMGET user:1 name age
HINCRBY user:1 loginCount 1     // atomic field increment
HDEL user:1 plan                HEXISTS user:1 age      HKEYS user:1
// Use for OBJECTS: store a user/product as one key with fields, instead of many string keys.
// More memory-efficient than separate keys, and you can update one field without rewriting all.

Lists — Ordered, Push/Pop Both Ends

LPUSH q "a"    RPUSH q "b"      // push left/right
LPOP q         RPOP q           // pop left/right (O(1))
LRANGE q 0 -1                    // all elements
LLEN q          LINDEX q 0
BLPOP q 5      BRPOP q 5        // BLOCKING pop (wait up to 5s) — the queue primitive
// Use for: QUEUES (LPUSH + BRPOP = producer/consumer), stacks, recent-items lists (capped with LTRIM).
// O(1) at the ends, O(n) in the middle — don't use as a random-access array.

Sets — Unique, Unordered Members

SADD tags "redis" "db" "cache"  SREM tags "cache"
SISMEMBER tags "redis"          // O(1) membership
SMEMBERS tags                   SCARD tags              // count
SINTER a b       SUNION a b     SDIFF a b               // set algebra!
SRANDMEMBER tags 2              SPOP tags
// Use for: unique collections (tags, unique visitors), membership checks, and relationship
// algebra — "users who follow BOTH A and B" is one SINTER. Deduplication is free.

Sorted Sets — the Crown Jewel (scored + ranked)

ZADD leaderboard 4200 "asha" 3800 "bo" 5100 "cy"
ZSCORE leaderboard "asha"       // 4200
ZINCRBY leaderboard 100 "asha"  // atomic score bump
ZRANGE leaderboard 0 9 WITHSCORES         // lowest 10 (ascending)
ZREVRANGE leaderboard 0 9 WITHSCORES      // TOP 10 (descending)
ZRANK leaderboard "asha"        // 0-based rank (ZREVRANK for descending)
ZRANGEBYSCORE leaderboard 4000 5000        // members in a score range
ZCOUNT leaderboard 4000 5000
// A set where each member has a SCORE, kept sorted. O(log n) inserts and rank queries.
// THE structure for leaderboards, priority queues, rate limiting by time, and any "top-N by X".
🏆 Why sorted sets are special: they maintain sorted order on every insert (O(log n)) and let you query by rank OR by score range in log time. A leaderboard for millions of players — "add a score, get the top 10, get my rank" — is three simple commands. Building that in a relational DB means ORDER BY + LIMIT on every read (expensive) or maintaining a rank column. Sorted sets are the single most interview-relevant Redis structure.

Interview Questions

Q1: How would you build a leaderboard for millions of players?

A sorted set. ZADD leaderboard score player adds/updates a player's score in O(log n); ZINCRBY atomically bumps it; ZREVRANGE leaderboard 0 9 WITHSCORES returns the top 10; ZREVRANK leaderboard player gives a player's rank; ZRANGEBYSCORE fetches everyone in a score band. All are O(log n) or O(log n + k) and always reflect current order because the sorted set maintains order on insert. Compared to a relational ORDER BY ... LIMIT on every read (or maintaining a rank column), it's dramatically simpler and faster — which is why "design a leaderboard" almost always expects Redis sorted sets. Add a TTL or periodic reset for time-boxed boards.

Q2: When do you use a Hash vs multiple String keys for an object?

Use a Hash to store an object's fields under ONE key (HSET user:1 name .. age ..). Benefits: it's more memory-efficient than many separate string keys (Redis has optimized small-hash encodings), lets you read/update individual fields (HGET/HSET a field) without rewriting the whole object, and keeps related data grouped and expirable together. Multiple string keys make sense only when fields need independent TTLs or you access exactly one field's value in isolation via patterns. For the common "cache a user/product record" case, a hash is the idiomatic, efficient choice.

Q3: How do you implement a reliable work queue in Redis?

The simple version: producers LPUSH jobs onto a list; consumers BRPOP (blocking right-pop) to wait for and atomically remove the next job — no polling, and single-threaded atomicity means each job goes to exactly one consumer. For reliability (a consumer crash shouldn't lose the job), use BRPOPLPUSH/LMOVE to atomically move the job to a per-consumer "processing" list, then remove it on completion (and recover orphaned jobs from processing lists). For richer needs — consumer groups, acknowledgments, replay, and at-least-once delivery — Redis Streams (pub/sub-streams lesson) are the modern, more robust choice, effectively a Redis-native Kafka-lite.

🏋️ Practice: store a user object as a hash and HINCRBY a counter field; build a queue with LPUSH + BRPOP across two sessions; use SADD + SINTER to find common followers of two users; build a leaderboard with ZADD, then get top-10 (ZREVRANGE) and a player's rank (ZREVRANK); count unique visitors with PFADD/PFCOUNT; cap a "recent items" list with LTRIM.

🎯 Key Takeaways

  • Hashes = objects (field→value, efficient, per-field updates); Lists = queues/stacks (O(1) ends, BLPOP blocks).
  • Sets = unique membership + algebra (SINTER/SUNION/SDIFF) for relationships and dedup.
  • Sorted Sets = scored + ranked (O(log n)): leaderboards, top-N, priority, time-based — the most interview-relevant structure.
  • Choose the structure for the access pattern; also know HyperLogLog (approx counts) and Bitmaps (presence).
  • Avoid O(n) replies (SMEMBERS/HGETALL) on huge collections and unbounded structures.
