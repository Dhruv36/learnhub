Replication & Sentinel Advanced

A single Redis node is a single point of failure. Replication copies data to replicas for read scaling and redundancy; Sentinel adds automatic failover so a primary crash doesn't take you down. These are the high-availability foundations, mirroring the replication lessons in the SQL and MongoDB tracks.

Replication — Primary/Replica

// On a replica:
REPLICAOF primary-host 6379      // this node becomes a read-only copy of the primary
// The replica does an initial full sync (RDB) then streams the primary's write commands to stay
// current — ASYNCHRONOUSLY. Topology:
//        writes           async replication
//   App ────────▶ PRIMARY ──────────────────▶ REPLICA (read-only)
//        reads ────────────────────────────────▶  ↑ scale reads here
// Replicas serve reads (scaling read-heavy workloads) and provide redundancy/failover targets.
⏱️ Replication is asynchronous → replicas can lag. The primary acknowledges a write before replicas have it, so a read from a replica may not reflect a just-made write (read-your-writes violation) — and a primary crash can lose the last few unreplicated writes. This is the same lag trade-off as SQL/MongoDB replicas: read from the primary for must-be-fresh data; use WAIT to require a write reach N replicas before proceeding when durability matters.

Sentinel — Automatic Failover

// Sentinel is a separate process that MONITORS the primary + replicas and handles failover:
// 1. MONITOR: Sentinels ping the primary; if it's unreachable, they agree it's down (a QUORUM
//    of Sentinels must concur — avoids one flaky Sentinel triggering a needless failover).
// 2. FAILOVER: Sentinels ELECT one replica and promote it to primary; reconfigure other replicas
//    to follow the new primary.
// 3. DISCOVERY: clients ask Sentinel "who is the current primary?" and reconnect — apps don't
//    hardcode the primary address.
// Run an ODD number of Sentinels (3+) across failure domains so a majority can decide.
sentinel monitor mymaster 127.0.0.1 6379 2   // "mymaster", quorum of 2

Why Quorum & Odd Numbers (recurring theme)

// A QUORUM (majority) of Sentinels must agree the primary is down AND authorize the failover.
// This prevents SPLIT-BRAIN (two primaries) during network partitions: only the partition with a
// majority can promote a new primary; the minority side can't. Hence odd numbers (3, 5) — a
// majority is always reachable if fewer than half fail. Same majority principle as MongoDB
// elections and distributed consensus generally.

Interview Questions

Q1: How does Redis achieve high availability?

Replication + Sentinel. Replication makes read-only replicas that asynchronously copy the primary's data — providing redundancy and read scaling. Sentinel is a separate monitoring process (run an odd number, 3+) that watches the primary and replicas; when a quorum of Sentinels agrees the primary is down, they elect a replica, promote it to primary, reconfigure the others to follow it, and let clients discover the new primary by querying Sentinel rather than a fixed address. This gives automatic failover within seconds without manual intervention. The quorum requirement prevents split-brain during partitions. Managed offerings and Redis Cluster provide this HA out of the box.

Q2: Why must Sentinel use a quorum, and why odd numbers?

A quorum (majority) of Sentinels must both agree the primary is unreachable and authorize the failover so that a single misbehaving or network-isolated Sentinel can't trigger an unnecessary or conflicting failover. Critically, during a network partition only the side holding a majority can promote a new primary — the minority side cannot — which prevents split-brain (two primaries accepting writes and diverging). Odd numbers (3, 5) guarantee that a majority is always available as long as fewer than half the Sentinels fail, and avoid tie situations. It's the same majority/quorum principle behind MongoDB elections and consensus protocols like Raft.

Q3: What happens to writes during a failover, and how do you minimize loss?

Because replication is asynchronous, writes acknowledged by the old primary but not yet replicated are lost when a replica is promoted — and during the brief promotion window, writes fail (clients should retry with backoff). To minimize loss and exposure: configure min-replicas-to-write (refuse writes unless at least N replicas are connected/current, shrinking the unreplicated window), use WAIT to require critical writes reach replicas before the app proceeds, keep replication lag low, and use Sentinel-aware clients that reconnect quickly to the new primary. You can't get zero loss with async replication without hurting latency; the config trades durability against availability to match your requirements.

🏋️ Practice (local multi-node or Docker): start a primary + replica with REPLICAOF and confirm reads work on the replica; write to the primary and observe replica lag; set up 3 Sentinels monitoring the primary, kill the primary, and watch a replica get promoted; point a Sentinel-aware client at Sentinel and confirm it follows the new primary; experiment with min-replicas-to-write.

🎯 Key Takeaways

  • Replication = read-only replicas asynchronously copying the primary — redundancy + read scaling.
  • Async lag: replicas can be stale, and a primary crash can lose unreplicated writes (read primary for freshness; WAIT for durability).
  • Sentinel monitors, elects, and promotes on primary failure, and gives clients current-primary discovery.
  • Quorum + odd number of Sentinels prevents split-brain — the majority principle again.
  • Use Sentinel-aware clients, retry during failover, min-replicas-to-write; managed Redis / Cluster provide HA turnkey.
