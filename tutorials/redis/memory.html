Memory Optimization & Eviction Ultra-Advanced

Redis lives and dies by RAM — it's the whole dataset's home and the hard limit on capacity. When memory runs out, Redis's eviction policy decides what happens, and that choice can silently break correctness if you get it wrong. This capstone covers keeping memory in check and configuring eviction deliberately.

maxmemory & Eviction Policies

maxmemory 4gb                    // cap Redis's memory usage
maxmemory-policy allkeys-lru     // what to do when the cap is hit
PolicyBehavior
noevictionReject writes with an error when full (reads still work). Safe default for a data store — nothing is silently lost
allkeys-lruEvict the least-recently-used key (any key). The classic CACHE policy
allkeys-lfuEvict the least-FREQUENTLY-used key — better when popularity is stable
volatile-lru / -lfu / -ttlEvict only keys that HAVE a TTL (by LRU / LFU / soonest-expiry)
allkeys-random / volatile-randomEvict a random key — rarely the best choice
⚠️ The eviction policy is a correctness decision, not just tuning. If Redis is a CACHE, allkeys-lru/lfu is right — evicting cold entries is fine, they reload from the DB. But if Redis holds data you can't lose (sessions, queues, rate-limit state), an allkeys eviction policy will silently DELETE that data under memory pressure — a subtle, dangerous bug. For such data use noeviction (fail writes loudly) or volatile-* with TTLs only on truly disposable keys. Match the policy to whether the data is reconstructable.

Interview Questions

Q1: What happens when Redis hits its memory limit, and why is the policy a correctness issue?

When used memory reaches maxmemory, Redis applies its maxmemory-policy: noeviction rejects further writes with an error (reads still work); the allkeys-* policies evict existing keys (LRU/LFU/random) to make room; the volatile-* policies evict only keys that have a TTL. This is a correctness decision because the allkeys policies will DELETE any key under pressure — fine for a pure cache (evicted entries reload from the source of truth), but catastrophic if Redis holds non- reconstructable data like sessions, queues, or rate-limit counters, which would silently vanish. So you match the policy to the data: allkeys-lru/lfu for caches, noeviction (or volatile-* with TTLs only on disposable keys) when losing data is unacceptable. Choosing blindly is a classic subtle production bug.

Q2: How does Redis approximate LRU/LFU, and why not exact?

Tracking exact least-recently/least-frequently-used order for every key would require extra memory per key and bookkeeping on every access — expensive for a system optimizing for speed and RAM. Instead, on eviction Redis samples a small number of keys (maxmemory-samples, ~5 by default) and evicts the best candidate from that sample; increasing the sample size approaches true LRU/LFU at more CPU cost. LFU additionally uses a compact probabilistic access counter that decays over time to estimate frequency. The result is an efficient approximation that's more than good enough for cache eviction while keeping memory and CPU overhead minimal — a deliberate accuracy-for-efficiency trade, much like HyperLogLog trades exactness for tiny memory.

Q3: How do you keep Redis memory under control and plan capacity?

Set maxmemory below physical RAM (so you hit your controlled eviction policy, not the OS OOM killer) with an appropriate policy for the data. Reduce per-item footprint: TTL everything cacheable to auto-reclaim, store objects as hashes (compact small-hash encoding) instead of many string keys, keep keys/values short, use HyperLogLog/Bitmaps for counts/flags, and cap collections (LTRIM, stream MAXLEN). Monitor via INFO memory (used_memory vs maxmemory, evicted_keys, fragmentation) and MEMORY USAGE/DOCTOR. Plan capacity as roughly (avg entry size × key count) plus headroom for replication and the RDB fork's copy-on-write spike, and when the working set genuinely exceeds one node's RAM, scale horizontally with Redis Cluster rather than pushing a single node to the edge.

🏋️ Practice: set maxmemory small + allkeys-lru, fill Redis past it, and watch keys get evicted (evicted_keys in INFO); switch to noeviction and see writes fail instead; compare MEMORY USAGE of an object stored as 5 string keys vs one hash; add TTLs and observe reclamation; count uniques with HyperLogLog vs a set and compare memory; check the fragmentation ratio and reason about headroom for an RDB fork.

🎯 Key Takeaways

  • Redis is RAM-bound; set maxmemory below physical RAM + a deliberate maxmemory-policy.
  • Eviction policy is a CORRECTNESS choice: allkeys-lru/lfu for caches; noeviction/volatile-* for data you can't lose.
  • Cut memory: TTL everything, hashes over many keys, short keys/values, HyperLogLog/Bitmaps, cap collections.
  • LRU/LFU are sampled APPROXIMATIONS (maxmemory-samples) — efficient, not exact.
  • Plan capacity with headroom for replication + RDB-fork copy-on-write; outgrow one node → Redis Cluster.
