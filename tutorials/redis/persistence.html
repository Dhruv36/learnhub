Persistence: RDB & AOF Intermediate

Redis is in-memory, but it can persist to disk so data survives restarts. There are two mechanisms — RDB snapshots and AOF (append-only file) — with different durability/performance trade-offs. Understanding them determines how much data you can lose in a crash, a decision every production Redis deployment must make consciously.

RDB — Point-in-Time Snapshots

save 900 1        // snapshot if ≥1 key changed in 900s
save 300 10       // ...or ≥10 keys in 300s   (config in redis.conf)
BGSAVE            // manually trigger a background snapshot
// RDB forks the process and writes the ENTIRE dataset to a compact binary .rdb file at intervals.
// PRO: compact, fast to load (great for restarts/backups), minimal runtime overhead.
// CON: you lose everything since the LAST snapshot if Redis crashes (could be minutes of data).
// The fork can briefly spike memory (copy-on-write) on write-heavy workloads.

AOF — Append-Only Log of Writes

appendonly yes
appendfsync everysec     // fsync policy: always | everysec | no
// AOF logs EVERY write command to a file; on restart Redis replays the log to rebuild state.
// fsync policy controls durability vs speed:
//   always   → fsync every write: safest (lose ~0), slowest
//   everysec → fsync once per second: lose ≤1s of data, good balance (the common choice)
//   no       → let the OS decide: fastest, least durable
// AOF grows over time; Redis REWRITES it periodically (BGREWRITEAOF) to a compact form.

The Recommended Setup: Both

// Production best practice is to enable BOTH:
appendonly yes           // AOF for durability (recent writes)
save 900 1               // RDB for fast restarts + compact backups
// On restart, Redis uses the AOF (more complete) to recover. You get RDB's fast backups/restarts
// AND AOF's low data-loss window. Redis can also use a hybrid RDB+AOF format (aof-use-rdb-preamble).
💾 The core question: how much data can you afford to lose? Pure cache (data reconstructable from the DB) → persistence barely matters; you might even disable it. Session store or anything where losing recent writes hurts → AOF everysec (≤1s loss). Redis as a primary store for critical data → AOF always + replication + backups. Match the durability config to the data's importance — the default assumptions are not always right for your use case.

Durability Beyond a Single Node

// Persistence protects against a SINGLE node's restart/crash. For real durability you also need:
// - REPLICATION (next lesson): a replica has a copy, survives a node loss.
// - BACKUPS: copy RDB files off-box (disaster recovery).
// - w-style acknowledgment via WAIT: WAIT 1 100 blocks until ≥1 replica has the write (100ms timeout).
// "Durability" is layered: fsync (this node) + replicas (other nodes) + off-site backups.

Interview Questions

Q1: RDB vs AOF — the trade-offs and when to use each.

RDB takes periodic point-in-time snapshots of the whole dataset to a compact binary file: low runtime overhead and fast restarts/backups, but you lose everything since the last snapshot on a crash (potentially minutes). AOF logs every write command and replays it on restart, with an fsync policy (always = ~0 loss but slow, everysec = ≤1s loss and the common balance, no = fastest/least durable); it's more durable but the file is larger and restart is slower (though periodic rewrites compact it). Use RDB alone only for pure caches where data is reconstructable; use AOF (everysec) when recent writes matter; the production recommendation is BOTH — AOF for durability and RDB for fast backups/restarts.

Q2: How much data can you lose with each persistence configuration?

It depends on the config: RDB with save-every-N — up to the snapshot interval (minutes) of writes. AOF appendfsync no — up to whatever the OS buffers (seconds to more). AOF everysec — at most about one second of writes (the usual sweet spot). AOF always — essentially zero on that node (each write is fsynced before acknowledgment), at a significant latency cost. But single-node fsync only protects against process restart on that machine; hardware loss still loses data unless you add replication (a replica holds a copy) and off-box backups, and WAIT can require a write to reach N replicas before proceeding. So "acceptable data loss" is a layered decision: fsync policy + replicas + backups, matched to the data's importance.

Q3: When is it fine to disable persistence entirely?

When Redis is a pure, disposable cache whose data can be fully and cheaply reconstructed from the source of truth (a database) — losing the entire cache on restart just causes a cold-start miss storm you handle with warming/rate-limiting (avalanche mitigation from the caching lesson), not data loss. Disabling persistence removes fork/AOF overhead and maximizes throughput. But be deliberate: sessions, rate-limit counters that must survive restarts, queues, or any Redis-as- source-of-truth data are NOT reconstructable and need persistence (and usually replication). The decision is always "what breaks if this data vanishes on restart?"

🏋️ Practice (local Redis): enable RDB, add keys, BGSAVE, kill and restart Redis, confirm recovery; enable AOF everysec, add keys, kill without a snapshot, restart, and confirm the writes survived; compare startup time with a large AOF vs an RDB; observe AOF growth then trigger BGREWRITEAOF; reason about the right config for a cache vs a session store vs a queue.

🎯 Key Takeaways

  • Redis is in-memory but can persist: RDB (periodic snapshots) and AOF (log of writes).
  • RDB: compact, fast restart/backups, but loses data since the last snapshot; AOF: durable (≤1s with everysec), larger, slower replay.
  • Production best practice: enable BOTH (AOF for durability, RDB for backups); Redis recovers from AOF.
  • Match persistence to loss tolerance: pure cache → maybe none; session store → AOF everysec; critical → AOF always + replicas + backups.
  • Durability is layered: fsync (node) + replication (other nodes) + off-box backups; WAIT for replica acknowledgment.
