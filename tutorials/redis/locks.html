Distributed Locks Advanced

A distributed lock ensures only ONE process across many servers does something at a time — running a scheduled job once, preventing duplicate processing, serializing access to a shared resource. Redis is commonly used for this, but the naive implementation has subtle, dangerous bugs. Getting the details right (and knowing the limitations) is a senior-level topic.

The Naive Lock (and why each part matters)

// Acquire: SET the lock key only if it doesn't exist, with a TTL, storing a UNIQUE token:
SET lock:job "unique-token-abc" NX PX 30000
//          key    value(owner)   NX  PX=30s TTL
// NX = only if not exists (mutual exclusion). PX = auto-expire so a crashed holder doesn't
// deadlock the lock forever. The unique token identifies THIS owner (critical for release).
if (acquired) { try { doWork(); } finally { releaseLock(); } }
🔑 Three things the naive lock MUST have: (1) NX for mutual exclusion, (2) a TTL so a crashed holder's lock auto-releases (no permanent deadlock), and (3) a unique owner token so you only release YOUR lock. Missing any one is a bug. The most dangerous omission is the token — see below.

The Release Bug — Why You Need a Lua Script

// ❌ WRONG release: DEL lock:job
// Danger: your work took longer than the TTL → your lock EXPIRED → another process acquired it →
// your DEL now deletes SOMEONE ELSE'S lock. Two processes run concurrently. Corruption.

// ✅ CORRECT release: check the token AND delete ATOMICALLY (Lua runs atomically in Redis):
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end" 1 lock:job unique-token-abc
// "Only delete the lock if it's still MINE." The check-then-delete must be atomic — doing GET then
// DEL as two commands has the same race. Single-threaded Lua guarantees atomicity.

The Deeper Problem — Locks and Time

// Even the correct single-instance lock has a fundamental issue: if your process STALLS (GC pause,
// long syscall) past the TTL, the lock expires and another process legitimately acquires it WHILE
// you think you still hold it. No Redis lock fully prevents this. Mitigations:
//   - Keep critical sections SHORT (< TTL with margin).
//   - Use a FENCING TOKEN: an ever-increasing number issued with each lock; the protected resource
//     rejects operations with a stale token (so a delayed old holder's writes are refused).
//   - Make operations IDEMPOTENT so duplicate execution is harmless.
// Locks reduce contention; correctness ultimately needs fencing/idempotency at the resource.

Redlock — Multi-Node Locking (know the debate)

// A single-Redis lock can be lost on failover (the new primary may not have the lock — async
// replication). REDLOCK is an algorithm to acquire a lock across N independent Redis primaries,
// requiring a MAJORITY to agree, for stronger guarantees.
// The debate: distributed-systems experts (Kleppmann) argue Redlock isn't safe for correctness
// under clock skew / GC pauses without fencing tokens; Redis's author (antirez) defends it for
// its intended use. The senior takeaway: for EFFICIENCY (avoid duplicate work) a simple lock is
// fine; for CORRECTNESS (must never double-execute), you need fencing tokens / idempotency, and a
// consensus system (ZooKeeper/etcd) may be more appropriate than any Redis lock.

Interview Questions

Q1: Design a correct Redis distributed lock. What are the three essentials?

Acquire with SET lockKey uniqueToken NX PX ttl: NX gives mutual exclusion (only one holder), PX sets a TTL so a crashed holder's lock auto-expires instead of deadlocking forever, and the unique token identifies the owner. Release must be an atomic check-and-delete via a Lua script: "if the lock's value equals my token, delete it, else do nothing" — because if you just DEL, a lock that expired mid-work (and was re-acquired by another process) would be deleted by you, letting two processes run concurrently. Those three — NX, TTL, and token-checked atomic release — are the non-negotiables; missing any is a real bug.

Q2: Why can't a Redis lock guarantee mutual exclusion under all conditions?

Because locks interact badly with time and failures. If the lock holder stalls (a long GC pause, slow syscall, or scheduler delay) beyond the TTL, the lock expires and another process legitimately acquires it while the first still believes it holds it — two processes in the critical section, no Redis mechanism prevents this. Additionally, on a single-Redis setup a failover can hand out the lock again because async replication may not have propagated it. The robust answer is not a "better lock" but defense at the resource: fencing tokens (a monotonically increasing number checked by the protected resource, so a stale holder's operations are rejected) and/or making the operation idempotent so duplicate execution is harmless. Locks manage contention; correctness needs fencing/idempotency.

Q3: What's the Redlock debate, and how do you decide what to use?

Redlock acquires a lock across multiple independent Redis primaries requiring a majority, to survive single-node failover. Martin Kleppmann argued it's unsafe for correctness because clock skew, GC pauses, and process stalls can still allow two holders, and that safety requires fencing tokens regardless; Redis's author defended it for its intended efficiency use cases. The practical decision framework: if the lock is for EFFICIENCY (avoiding duplicate/wasted work where an occasional double-run is merely inefficient), a simple single-instance lock (or Redlock) is fine. If it's for CORRECTNESS (double execution causes data corruption or financial error), don't rely on any timeout-based lock alone — use fencing tokens and idempotency, and consider a consensus-based coordinator like etcd or ZooKeeper designed for that guarantee.

🏋️ Practice: implement acquire with SET NX PX + a random token; implement release as an atomic Lua check-and-delete; demonstrate the DEL-without-token bug by expiring a lock mid-work and deleting another owner's lock; add a fencing token and reject stale-token operations at a mock resource; use the lock to make a scheduled job run once across two "instances".

🎯 Key Takeaways

  • Correct lock = SET key token NX PX ttl (mutual exclusion + auto-expire + owner identity).
  • Release with an atomic Lua check-and-delete ("delete only if it's mine") — never a bare DEL.
  • TTL vs work duration matters: a stall past the TTL can cause double execution — no lock fully prevents it.
  • For hard correctness, add fencing tokens + idempotency at the resource; locks alone manage contention.
  • Redlock (multi-node) exists and is debated; use etcd/ZooKeeper when correctness is critical, simple locks for efficiency.
