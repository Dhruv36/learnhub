Production Use Cases Ultra-Advanced

This lesson assembles everything into the concrete patterns Redis is famous for — rate limiting, leaderboards, sessions, queues, and more. Each is a small recipe built from the primitives you've learned, and each is a common system-design interview answer ("how would you build X?").

Rate Limiter — Atomic Counter + TTL

// Fixed window: N requests per minute per user. INCR + EXPIRE, checked atomically:
function allow(userId) {
  const key = `rate:${userId}:${currentMinute()}`;
  const count = INCR(key);           // atomic (single-threaded)
  if (count === 1) EXPIRE(key, 60);  // set TTL on first hit of the window
  return count <= 100;               // allow up to 100/min
}
// Simpler and race-free vs app-side counting (strings lesson). For smoother limiting without
// window-edge bursts, use a SLIDING WINDOW with a sorted set (below) or token-bucket via Lua.

Leaderboard — Sorted Set

ZADD leaderboard 4200 "asha"            // add/update a score
ZINCRBY leaderboard 50 "asha"           // atomic score change on an event
ZREVRANGE leaderboard 0 9 WITHSCORES    // top 10
ZREVRANK leaderboard "asha"             // this player's rank (0-based)
ZRANGEBYSCORE leaderboard 4000 5000     // players in a score band
// Time-boxed boards: use a key per period (leaderboard:2026-07) with a TTL, or reset on schedule.

Sliding-Window Rate Limiter — Sorted Set

// Store each request's timestamp as a sorted-set member scored by time; trim old, count recent:
ZADD rate:user:1  
ZREMRANGEBYSCORE rate:user:1 0    // drop entries older than 60s
ZCARD rate:user:1                               // requests in the last 60s → compare to limit
EXPIRE rate:user:1 60
// Wrap these in a Lua script for atomicity. Smoother than fixed-window (no edge bursts).

Session Store

SET session:abc123 "{userId:1, ...}" EX 1800     // session with a 30-min sliding TTL
// On each request: GET the session, and EXPIRE it again to slide the window (keep-alive).
// Fast (sub-ms), auto-expiring, and shared across all app instances (stateless web tier —
// the Node/System-Design scaling lesson). The classic reason to add Redis to a web app.

Atomicity Across Multiple Commands

// Some recipes need several commands to run atomically. Two tools:
// 1. MULTI/EXEC (transactions): queue commands, execute together (no interleaving) — but no
//    conditional logic between them. WATCH adds optimistic locking (abort if a key changed).
// 2. LUA SCRIPTS (EVAL): run arbitrary logic atomically server-side — the preferred way for
//    check-then-act patterns (rate limits, locks, conditional updates). One round trip, atomic.
// Prefer Lua for anything with conditional logic; single-threaded execution guarantees atomicity.

Interview Questions

Q1: Design a rate limiter with Redis (fixed and sliding window).

Fixed window: key it per user per time bucket (rate:user:minute), INCR on each request — atomic, so no lost counts — and set EXPIRE to the window length on the first hit; allow while the count is ≤ the limit. Simple and cheap, but it permits bursts at window edges (up to 2× the limit across a boundary). Sliding window: use a sorted set of request timestamps — ZADD the current time, ZREMRANGEBYSCORE to drop entries older than the window, ZCARD to count what remains, compare to the limit, and EXPIRE the key — ideally wrapped in a Lua script so the check-and-record is atomic. This smooths the edge-burst problem at slightly more cost. Token-bucket (tokens refilled over time, implemented in Lua) is another common choice for allowing controlled bursts.

Q2: How do you build a leaderboard, and why is Redis ideal?

A sorted set: ZADD/ZINCRBY to set or bump a player's score (O(log n)), ZREVRANGE 0 9 WITHSCORES for the top 10, ZREVRANK for a specific player's rank, and ZRANGEBYSCORE for a score band — all in logarithmic time and always reflecting current order because the structure stays sorted on insert. Redis is ideal because these ranking queries are exactly what sorted sets are built for: a relational equivalent needs an ORDER BY ... LIMIT scan on every read or a maintained rank column, both far more expensive at scale. For time-boxed boards, use a key per period with a TTL or reset on schedule. It's the canonical "design a leaderboard" answer.

Q3: When do you need Lua scripts or MULTI/EXEC, and which do you prefer?

When a recipe requires multiple commands to execute atomically — no other client interleaving — such as check-then-act logic in rate limiters, distributed-lock release, or conditional updates. MULTI/EXEC queues commands and runs them together without interleaving (WATCH adds optimistic locking to abort if a watched key changed), but you can't branch on intermediate results inside the transaction. Lua scripts (EVAL) run arbitrary logic — including conditionals and loops — atomically on the server in a single round trip, because Redis executes the whole script without interruption (single-threaded). I prefer Lua for anything with conditional logic (the common case); MULTI/EXEC+WATCH suits simple all-or-nothing groupings or optimistic-concurrency patterns. Both leverage Redis's single-threaded atomicity.

🎯 Key Takeaways

  • Rate limiter: fixed window (INCR+EXPIRE) or sliding window (sorted set + Lua) for smoothness.
  • Leaderboard: sorted set (ZADD/ZREVRANGE/ZREVRANK) — the canonical Redis use case.
  • Sessions: SET EX with sliding TTL — fast, auto-expiring, shared across a stateless web tier.
  • Also: queues (List/Streams), approx counts (HyperLogLog), presence (Bitmaps), locks, pub/sub, geo/autocomplete.
  • Use Lua scripts for atomic check-then-act logic (preferred) or MULTI/EXEC+WATCH — single-threaded guarantees atomicity.
