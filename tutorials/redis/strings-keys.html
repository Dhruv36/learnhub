Strings, Keys & TTL Basics

Strings are Redis's most basic type — but they're used as counters, flags, cached JSON, and more. The concept that makes Redis a great cache is TTL (time-to-live): keys that automatically expire. And atomic operations like INCR are the foundation of counters and rate limiters.

String Commands

SET key "value"                 GET key
SET k v EX 3600                 // set with a 1-hour TTL in one atomic command
SET k v NX                      // set only if NOT exists (returns nil if key exists)
SET k v XX                      // set only if it DOES exist
MSET a 1 b 2 c 3                MGET a b c        // multi-get/set (one round trip)
APPEND key " more"              STRLEN key
GETSET key newval               // set new, return old (atomic swap)
SET session:abc "{...json...}" EX 1800   // cached JSON with a 30-min TTL

Atomic Counters — the Killer Feature

INCR pageviews                  // atomically +1, returns the new value
INCRBY downloads 5              DECR stock              DECRBY stock 3
INCRBYFLOAT price 0.50
// Because Redis is single-threaded, INCR is atomic WITHOUT any lock — two clients incrementing
// concurrently never lose an update (unlike read-modify-write in application code, which races).
// This is the primitive behind view counters, rate limiters, and unique-id generators.
🔢 Why INCR beats app-side counting: doing x = GET; SET x+1 in your app is a read-modify-write race — two clients read the same value and both write the same +1, losing an increment. INCR does it atomically inside Redis, so concurrent increments are always correct. This atomic-counter pattern is the seed of the rate limiter and unique-ID generator in the use-cases lesson.

TTL — Automatic Expiration

EXPIRE key 60                   // expire in 60 seconds
PEXPIRE key 500                 // milliseconds
EXPIREAT key 1735689600         // at a Unix timestamp
TTL key                         // seconds left (-1 = no expiry, -2 = key doesn't exist)
PERSIST key                     // remove the TTL (make permanent)
SET key val EX 300              // set + TTL atomically (preferred over SET then EXPIRE)
// TTL is what makes Redis a natural CACHE: entries expire automatically, bounding memory and
// staleness. Sessions, cached queries, and rate-limit windows all rely on it.
// How expiration actually works (interview detail):
// - LAZY: a key is checked/removed when accessed after its TTL.
// - ACTIVE: Redis periodically samples keys with TTLs and removes expired ones.
// Together they bound memory without scanning every key. An expired key still counts toward
// memory until removed by one of these — usually negligible, but relevant at huge scale.

Key Management

EXISTS key                      DEL key1 key2           UNLINK key   // DEL but frees memory async
TYPE key                        // string | hash | list | set | zset | stream
RENAME old new                  COPY src dst
SCAN 0 MATCH cache:* COUNT 100  // cursor-based iteration — NON-BLOCKING (use over KEYS)
// ⚠ KEYS pattern scans the ENTIRE keyspace and BLOCKS the single thread — never in production.
// SCAN returns a cursor + a batch; loop until the cursor is 0. Same for HSCAN/SSCAN/ZSCAN.

Interview Questions

Q1: Why is INCR preferred over reading, incrementing, and writing back?

Read-modify-write in application code is a race: two clients GET the same value, both compute +1, and both SET the same result — one increment is lost. INCR performs the read-modify-write inside Redis as a single atomic command; because Redis executes commands one at a time (single-threaded), concurrent INCRs are serialized and every increment counts, with no application-side locking. This atomicity is why Redis counters are correct under high concurrency and why INCR (with TTL) is the building block for rate limiters, view counters, and sequence generators.

Q2: How does key expiration work, and why does it matter?

TTL makes a key auto-delete after a time — the mechanism that makes Redis a self-cleaning cache (bounding memory and staleness for sessions, cached data, rate-limit windows). Redis expires keys two ways: lazily (checking/removing an expired key when it's next accessed) and actively (a background job periodically samples keys with TTLs and removes expired ones). Together they reclaim memory without scanning the whole keyspace. Practical implications: set TTLs atomically with SET ... EX (so a crash can't leave a TTL-less key), and know that an expired-but-not-yet-removed key still occupies memory momentarily — usually negligible.

Q3: Why is KEYS dangerous and what do you use instead?

KEYS pattern scans the entire keyspace in one blocking operation; since Redis is single-threaded, it stalls every other client for the duration — on a large keyspace that can be seconds of latency spikes or effective downtime. SCAN is the safe alternative: it's cursor-based, returning a small batch of keys and a cursor each call, so you iterate incrementally without blocking (looping until the cursor returns 0). It may return keys added/removed during iteration and can return duplicates, but it never blocks. HSCAN/SSCAN/ZSCAN do the same for large hashes/sets/sorted sets. The rule: never KEYS in production, always SCAN.

🏋️ Practice: SET a key with EX and watch TTL count down; INCR a counter from two redis-cli sessions and confirm no lost updates; use SET NX to implement "run once"; store cached JSON with a TTL; run KEYS * vs SCAN on a keyspace with many keys; demonstrate the SET-then-EXPIRE crash risk vs atomic SET EX; use UNLINK vs DEL on a large key.

🎯 Key Takeaways

  • Strings hold values, counters, flags, cached JSON; MSET/MGET for batch round trips; SET NX/XX for conditional sets.
  • INCR/INCRBY are atomic (single-threaded) — correct counters without app-side locks (rate limiters, IDs).
  • TTL (EXPIRE / SET EX) makes Redis a self-cleaning cache; set it atomically to avoid TTL-less leaks.
  • Expiration is lazy + active — bounds memory without full scans.
  • Never KEYS in production — use SCAN (cursor, non-blocking); UNLINK frees big keys async.
