Caching Patterns Intermediate

Caching is Redis's most common job, and doing it correctly is more subtle than "check cache, else hit DB". This lesson covers the strategies, the invalidation problem ("one of the two hard things in computer science"), and the three failure modes — stampede, penetration, avalanche — that every senior engineer must know. It mirrors the System Design track's caching lesson, grounded in Redis commands.

Cache-Aside (Lazy Loading) — the Default

async function getUser(id) {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);            // HIT
  const user = await db.query("SELECT ... WHERE id=?", id);   // MISS → load from DB
  await redis.set(`user:${id}`, JSON.stringify(user), "EX", 3600);  // populate + TTL
  return user;
}
// The app manages the cache: read cache → on miss, read DB and populate. On WRITE, INVALIDATE:
async function updateUser(id, data) {
  await db.update(id, data);
  await redis.del(`user:${id}`);       // DELETE the key (don't update it — see note)
}
🔑 Invalidate by DELETING, not updating. On a write, delete the cache key so the next read re-fetches fresh data. Updating the cache value directly races: two concurrent writes can interleave so the cache ends up holding the older value permanently. Delete-on-write forces a clean reload. (Same lesson as the CSS/JS caching discussions — invalidation is the hard part.)

The Strategies Compared

PatternHowTrade-off
Cache-aside (lazy)App reads cache, loads DB on miss, deletes on writeSimple, resilient (cache down ≠ app down); brief staleness window; first read is a miss
Write-throughWrite to cache + DB togetherCache always fresh; every write pays both latencies
Write-behind (write-back)Write to cache, async-flush to DBFast writes, absorbs bursts; risk of data loss if cache dies before flush
Read-throughCache layer loads from DB on missCleaner app code; needs a cache-side loader

Cache-aside is the default for most web apps — resilient and simple. TTL is your safety net: set one even with explicit invalidation, so a missed delete can't cache stale data forever.

The Three Failure Modes (senior must-know)

1. CACHE STAMPEDE / THUNDERING HERD
   A hot key expires → thousands of concurrent requests all MISS → all hit the DB at once → DB dies.
   Fixes: - a lock so ONE request rebuilds while others wait/serve stale (SET NX as a mutex)
          - stale-while-revalidate: serve the old value, refresh in the background
          - jittered/randomized TTLs so 1M keys don't expire the same second

2. CACHE PENETRATION
   Requests for keys that DON'T EXIST (often malicious) always miss → always hit the DB.
   Fixes: - cache the NULL/"not found" result with a short TTL
          - a Bloom filter in front to reject known-absent keys cheaply

3. CACHE AVALANCHE
   The whole cache restarts cold (or many keys expire together) → 100% miss → DB overwhelmed.
   Fixes: - cache warming before taking traffic
          - jittered TTLs (avoid synchronized expiry)
          - request rate-limiting / a circuit breaker at the DB layer
          - HA Redis (replication) so it doesn't go fully cold

Interview Questions

Q1: Walk through cache-aside and why you delete rather than update on writes.

Cache-aside: on read, check Redis; on a hit return it; on a miss, load from the database, populate the cache with a TTL, and return. On a write, update the database and then DELETE the cache key so the next read reloads fresh data. You delete rather than update the cached value because directly writing the new value races: two concurrent writes (or a write interleaving with a slow read that's about to populate) can leave the cache holding an older value indefinitely. Deleting forces the next reader to fetch the current truth. Always pair it with a TTL as a safety net so any missed deletion self-heals within the TTL.

Q2: Explain cache stampede and how you prevent it.

A stampede (thundering herd) happens when a popular key expires and thousands of concurrent requests simultaneously miss and all hit the database to rebuild the same value — potentially overwhelming it. Prevention: (1) a rebuild lock — the first request acquires a short lock (SET key NX), rebuilds, and others wait briefly or serve the stale value, so only one DB query runs; (2) stale-while-revalidate — serve the expired value while a single background task refreshes it; (3) jittered TTLs — add randomness so a million keys don't expire in the same second. These combine well; the goal is to collapse N concurrent rebuilds into one.

Q3: What are cache penetration and avalanche, and their fixes?

Penetration: requests for keys that don't exist always miss the cache and hit the database every time — often malicious (probing random ids). Fix by caching the negative/"not found" result with a short TTL, and/or putting a Bloom filter in front to cheaply reject keys known to be absent. Avalanche: a large fraction of the cache becomes unavailable at once — a cold restart or many keys expiring simultaneously — so nearly all traffic misses and floods the database. Fix with cache warming before serving traffic, jittered TTLs to avoid synchronized expiry, database-side rate limiting / circuit breakers to protect it during recovery, and HA Redis (replication) so it rarely goes fully cold. All three modes share a theme: protect the database from correlated misses.

🏋️ Practice: implement cache-aside for a user lookup (get→miss→load→set EX; del on update); prove the update-vs-delete race conceptually; add jitter to TTLs; simulate a stampede (expire a hot key, fire concurrent reads) and add a SET NX rebuild lock; cache a "not found" result to stop penetration; sketch an event-driven invalidation with pub/sub.

🎯 Key Takeaways

  • Cache-aside is the default: read cache → miss → load DB → set with TTL; DELETE (not update) on write.
  • Always set a TTL even with invalidation — it's the self-healing safety net.
  • Stampede (hot-key expiry): rebuild lock (SET NX) / stale-while-revalidate / jittered TTLs.
  • Penetration (missing keys): cache negatives / Bloom filter. Avalanche (cold/mass-expiry): warming, jitter, DB rate-limit, HA Redis.
  • The DB is the source of truth; the cache is disposable — invalidation strategy follows your staleness tolerance.
