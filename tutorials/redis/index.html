Introduction & Data Model Basics

Redis is an in-memory data structure store — it keeps data in RAM, so operations are sub-millisecond. But calling it "a cache" undersells it: it's a cache, a message broker, a rate limiter, a leaderboard, a queue, and a session store, all built on rich data structures. The key to using it well is understanding why in-memory + single-threaded shapes everything.

Why Redis Is Fast — the Two Design Choices

1. IN-MEMORY: data lives in RAM, not on disk. RAM access is ~100,000× faster than disk seeks →
   sub-millisecond reads/writes. (Persistence to disk exists for durability — persistence lesson —
   but the working data is in memory.)
2. SINGLE-THREADED command execution: one command runs at a time, to completion. No locks, no
   race conditions between commands, and every operation is effectively ATOMIC. This is why Redis
   can do 100k+ ops/sec on one core — it never pays for locking/context-switching on the hot path.
   (Modern Redis uses threads for I/O and background tasks, but command execution is serial.)
🧵 Single-threaded is a feature, not a limitation. Because commands execute one at a time, INCR is atomic without any lock, and a Lua script or transaction runs with no other command interleaving. The flip side: a single slow command (a big KEYS *, a huge sort) blocks EVERYTHING — the Redis equivalent of "don't block the event loop" from the Node lesson. Avoid O(n) commands on large datasets in production.

The Data Model: Keys → Typed Values

// Redis is a giant dictionary: string KEYS mapping to typed VALUES.
SET user:1:name "Asha"           // String
HSET user:1 name "Asha" age 30   // Hash (field→value map)
LPUSH queue:jobs "job1"          // List (ordered)
SADD tags:post:1 "redis" "db"    // Set (unique members)
ZADD leaderboard 4200 "asha"     // Sorted Set (members scored/ranked)
// The VALUE's type determines which commands work on it. Choosing the right structure for the
// access pattern is the core Redis skill (data-structures lesson).

Key Naming Conventions

user:1000:profile          // colon-separated namespaces (a convention, not enforced)
session:abc123
cache:product:42
rate:ip:203.0.113.5
// Good key design makes data discoverable and enables patterns (SCAN cache:product:*).
// Keys are binary-safe strings; keep them short-ish (they cost memory) but readable.

What Redis Is (and Isn't) For

Great forNot for
Caching (the classic use)Primary store for data you can't lose (unless configured carefully)
Sessions, rate limiting, countersComplex queries / joins / ad-hoc analytics
Leaderboards (sorted sets)Datasets far larger than RAM (RAM is the limit)
Queues, pub/sub, streamsRelational integrity / transactions across many entities
Real-time features (sub-ms latency)Storing large blobs / files

Redis usually complements a primary database (Postgres/MongoDB) — it's the fast layer in front, not the system of record. This is the "cache + datastore" split from the System Design track.

Redis vs Memcached (the classic comparison)

Memcached: simpler, multi-threaded, pure LRU string cache. Great if you ONLY need a
           key→string cache and want to scale across cores trivially.
Redis:     rich data structures, persistence, replication, pub/sub, scripting, transactions.
           The default choice today — you rarely outgrow into "just Memcached".

Interview Questions

Q1: Why is Redis so fast, and why is single-threaded a good design?

Two reasons: it keeps data in RAM (orders of magnitude faster than disk), and it executes commands on a single thread, one at a time to completion. Single-threading eliminates locking and context-switching overhead on the hot path and makes every command atomic — INCR needs no lock, and scripts/transactions run without interleaving — enabling 100k+ ops/sec per core. The trade-off: one slow O(n) command blocks all others (the same "don't block the thread" discipline as Node's event loop), so you avoid commands like KEYS * on large datasets and use non-blocking variants (SCAN) instead. Redis scales across cores via multiple instances/sharding rather than threads.

Q2: Is Redis just a cache? What else is it good for?

No — caching is the most common use, but Redis is an in-memory data-structure server. Its typed values enable: rate limiting and counters (atomic INCR with TTL), leaderboards and rankings (sorted sets with O(log n) rank queries), sessions, queues and job processing (lists/streams), real-time messaging (pub/sub and streams), distributed locks, and geospatial queries. The unifying theme is sub-millisecond operations on rich structures. It typically complements a primary database rather than replacing it — the fast layer in front of a durable system of record.

Q3: Redis vs Memcached — when would you pick each?

Memcached is a simpler, multi-threaded, pure key→string LRU cache — pick it if you literally only need a string cache and value trivial multi-core scaling and operational simplicity. Redis offers rich data structures, persistence, replication/HA, pub/sub, streams, scripting, and transactions, making it the default modern choice because it covers far more use cases and you rarely regret the extra capability. In practice most teams reach for Redis; Memcached remains reasonable for extremely simple, huge, ephemeral string caches where its multi-threaded model is a fit.

🏋️ Practice (redis-cli or a local Redis): SET/GET a string; store a user as a hash and read a field; push to a list and pop; add members to a set and check membership; add scored members to a sorted set and read the top ranks; run KEYS * then SCAN and note why SCAN is preferred; design a key-naming scheme for cached products and sessions.

🎯 Key Takeaways

  • Redis is an in-memory data-structure store — sub-ms ops because data is in RAM.
  • Single-threaded command execution = atomic operations, no locks; but one slow command blocks everything (use SCAN, not KEYS *).
  • Data model: string keys → typed values (String/Hash/List/Set/Sorted Set); choose the structure for the access pattern.
  • Far more than a cache: rate limiting, leaderboards, queues, pub/sub, locks — usually complementing a primary DB.
  • RAM-bound; use namespaced key conventions; Redis is the default over Memcached for its richer capabilities.
