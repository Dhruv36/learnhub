Redis Cluster Advanced

Replication scales reads and adds HA, but every node still holds the ENTIRE dataset — so you're capped by one machine's RAM and write throughput. Redis Cluster shards the data across multiple primaries so both storage and writes scale horizontally. The mechanism — hash slots — and its constraints on multi-key operations are the key concepts.

Hash Slots — How Data Is Distributed

// Redis Cluster divides the keyspace into 16384 HASH SLOTS. Each key maps to a slot:
//     slot = CRC16(key) % 16384
// Each primary owns a RANGE of slots (e.g., 3 primaries → ~5461 slots each). The key's slot
// determines which node stores it. Clients cache the slot→node map and route directly.
//        Primary A: slots 0–5460       Primary B: 5461–10922      Primary C: 10923–16383
// Adding a node = MOVE some slots (and their keys) to it — online, incrementally.
🎰 Why 16384 fixed slots (not hashing directly to nodes)? A fixed slot layer decouples keys from the changing node count. When you add/remove nodes, you reassign SLOTS between nodes (and only those slots' keys move) — the key→slot mapping never changes. It's Redis's form of consistent hashing: rebalancing moves a fraction of data, not everything. (Same problem the System Design consistent-hashing lesson solves.)

The Multi-Key Constraint (the big gotcha)

// A command touching MULTIPLE keys only works if those keys are in the SAME slot (same node) —
// the cluster can't do a cross-node MGET/transaction/SINTER atomically:
MGET user:1 user:2               // ❌ CROSSSLOT error if they hash to different slots
// HASH TAGS force keys into the same slot — put {...} in the key; only that part is hashed:
SET {user:1}:profile "..."       SET {user:1}:settings "..."
// Both hash on "user:1" → same slot → same node → multi-key ops work.
MGET {user:1}:profile {user:1}:settings   // ✅ same slot
// Design implication: co-locate keys you access together using hash tags (e.g., all of one
// user's / one tenant's keys), or avoid multi-key operations across unrelated keys.

Cluster Topology & Failover

// Each PRIMARY in the cluster has its own REPLICA(s). Nodes gossip health via a cluster bus.
// If a primary fails, its replica is promoted automatically (built-in failover — no separate
// Sentinel needed). A majority of primaries must agree, so the cluster needs enough nodes to
// maintain a majority (typically ≥3 primaries).
//   Shard 1: Primary A + Replica A'      Shard 2: Primary B + Replica B'   ...
// Cluster gives BOTH horizontal scaling (sharding) AND high availability (per-shard replicas).

Interview Questions

Q1: How does Redis Cluster distribute data, and why 16384 hash slots?

Redis Cluster partitions the keyspace into 16384 fixed hash slots; each key maps to a slot via CRC16(key) % 16384, and each primary node owns a contiguous range of slots. Clients cache the slot→node map and route each command directly to the owning node. The fixed slot layer decouples keys from the (changing) number of nodes: when you add or remove a node you reassign whole slots (and only those slots' keys migrate), while the key→slot function stays constant — so rebalancing moves a fraction of the data rather than rehashing everything. It's Redis's take on consistent hashing, minimizing data movement during topology changes.

Q2: What's the multi-key limitation in a cluster and how do hash tags solve it?

Because keys are spread across nodes by slot, a command that operates on multiple keys (MGET, SINTER, a MULTI transaction, a multi-key Lua script) can only execute if all its keys live in the SAME slot (hence the same node) — otherwise the cluster returns a CROSSSLOT error, since it can't atomically coordinate across nodes. Hash tags fix this: if a key contains {...}, only the substring inside the braces is hashed, so SET {user:1}:profile and SET {user:1}:settings both hash on "user:1" and land in the same slot/node, letting you run multi-key operations on them. The design guidance is to co-locate keys you access together (e.g., all of one user's or one tenant's keys) via a shared hash tag — while avoiding over-concentrating too much on one tag, which creates a hot slot.

Q3: When do you choose Redis Cluster over a single primary with replicas?

Choose Cluster when a single node can no longer hold the dataset in RAM or handle the write throughput — i.e., you need to scale storage and writes horizontally by sharding across multiple primaries. A single primary + replicas (with Sentinel) already gives you high availability and read scaling (reads served by replicas), and every replica holds the full dataset, so it's simpler and sufficient until you hit one machine's memory/write ceiling. Cluster adds real complexity — cluster-aware clients, the same-slot constraint on multi-key operations, and more operational surface — so, like SQL/MongoDB scaling, you exhaust vertical scaling and replication first and shard only when genuinely required. Managed Cluster offerings reduce the operational burden.

🏋️ Practice (redis-cli --cluster or Docker): create a 3-primary cluster and observe slot assignment; SET keys and see which node owns each (CLUSTER KEYSLOT); trigger a CROSSSLOT error with MGET on differently-slotted keys, then fix it with hash tags; add a node and watch slot migration; kill a primary and confirm its replica is promoted; reason about co-locating a tenant's keys with a hash tag.

🎯 Key Takeaways

  • Cluster shards data across primaries via 16384 hash slots (CRC16(key) % 16384) — scales storage + writes.
  • Fixed slots decouple keys from node count — adding/removing nodes moves only some slots (consistent-hashing-like).
  • Multi-key ops require the SAME slot; hash tags {…} co-locate related keys to enable them.
  • Each shard has replicas with built-in failover (no separate Sentinel); needs ≥3 primaries for majority.
  • Shard only after vertical scaling + replicas; watch for hot slots; managed services run Cluster for you.
