Async Patterns Async Node

What you'll master: Node's async idioms in practice — promisifying legacy callbacks, concurrency control (limits, batching), timeouts and cancellation with AbortSignal, retries with backoff and jitter, and the error-handling discipline that keeps async servers debuggable. Prerequisites: the JS track's Promises and Combinators — this lesson is those tools under server load, plus Node-specific machinery.

Part 1 — From Callbacks to Promises: the Bridge

// Node's legacy convention — error-first callbacks:
someApi.getUser(id, (err, user) => {
  if (err) return handle(err);          // ALWAYS first-position error check
  // …
});
// You'll read this forever (old libs, some AWS SDKs, event APIs). Write promises.

// The bridge — util.promisify converts the convention mechanically:
import { promisify } from "node:util";
const getUser = promisify(someApi.getUser.bind(someApi));   // note the bind —
const user = await getUser(id);                              // `this` is lost otherwise

// Hand-promisify when the callback isn't error-first (events, multi-arg):
function onceListening(server) {
  return new Promise((resolve, reject) => {
    server.once("listening", resolve);
    server.once("error", reject);       // ALWAYS wire the failure path too —
  });                                    // or errors hang the await forever
}
// (events.once(emitter, "listening") does exactly this — EventEmitter lesson.)
// And the reverse trap — don't wrap what's already promised:
// new Promise(res => fetch(url).then(res))   ❌ the explicit-constructor antipattern

Part 2 — Concurrency Control: the Server-Side Difference

// The JS track taught the combinators; servers add a constraint the browser
// rarely hits: UNBOUNDED CONCURRENCY IS AN OUTAGE.
await Promise.all(userIds.map(id => enrich(id)));   // 50,000 ids →
// 50k simultaneous DB queries: pool exhausted, DB pushed over, memory spiked.
// The browser caps you at ~6 connections; Node happily fires everything.

// The tool: a concurrency limiter. Build the canonical one:
export function pLimit(max) {
  let active = 0;
  const queue = [];
  const next = () => {
    if (active >= max || queue.length === 0) return;
    active++;
    const { fn, resolve, reject } = queue.shift();
    fn().then(resolve, reject).finally(() => { active--; next(); });
  };
  return fn => new Promise((resolve, reject) => {
    queue.push({ fn, resolve, reject });
    next();
  });
}

const limit = pLimit(10);                            // 10 in flight, ever
const results = await Promise.all(
  userIds.map(id => limit(() => enrich(id))));       // same shape, bounded reality
// (npm: p-limit — same API. Build it once to trust it.)
//
// Choosing the number: pool size / downstream rate limits / memory per task —
// the limit encodes RESPECT for the weakest dependency. And remember
// allSettled vs all: partial-failure tolerance is a per-callsite decision.

Part 3 — Timeouts & Cancellation: AbortSignal Everywhere

// Rule: EVERY await on the outside world gets a deadline. A hung dependency
// without timeouts = your requests hang = your callers hang = cascade.

// AbortSignal is the standard plumbing, and Node core accepts it broadly:
const res = await fetch(url, { signal: AbortSignal.timeout(5000) });
await fs.readFile(p, { signal });
const child = spawn(cmd, args, { signal });
setTimeout(cb, ms, { signal });                       // even timers

// Compose deadlines with user-driven cancellation:
const signal = AbortSignal.any([
  AbortSignal.timeout(5000),                          // deadline
  req.signal,                                          // client disconnected (http)
]);

// Make YOUR async functions cancelable — accept a signal, forward it, check it:
async function processOrder(order, { signal } = {}) {
  const items = await db.getItems(order.id, { signal });   // forward to I/O
  for (const item of items) {
    signal?.throwIfAborted();                              // check between steps
    await reserve(item, { signal });
  }
}
// On abort you get an AbortError — distinguish it (cleanup, not alarm):
try { await processOrder(o, { signal }); }
catch (err) {
  if (err.name === "AbortError") return metrics.count("order.canceled");
  throw err;
}
// Why cancellation matters at load: a canceled request that KEEPS doing DB
// work is capacity theft — under stress, exactly when clients time out and
// retry, you do double work unless aborts propagate.

Part 4 — Retries: Backoff, Jitter, and Knowing When Not To

export async function retry(fn, {
  attempts = 3, baseMs = 200, maxMs = 5000,
  retryOn = isTransient, signal,
} = {}) {
  let lastErr;
  for (let i = 0; i < attempts; i++) {
    signal?.throwIfAborted();
    try { return await fn({ signal }); }
    catch (err) {
      lastErr = err;
      if (!retryOn(err) || i === attempts - 1) throw err;
      const backoff = Math.min(baseMs * 2 ** i, maxMs);   // exponential: 200, 400, 800…
      const jitter = backoff * (0.5 + Math.random() * 0.5); // ⭐ jitter: without it,
      await setTimeout(jitter, { signal });                 // every client retries in
    }                                                       // sync waves — thundering herd
  }
  throw lastErr;
}

function isTransient(err) {
  return err.code === "ECONNRESET" || err.code === "ETIMEDOUT" ||
         err.status === 429 || err.status >= 500;
  // NEVER retry: 400/401/403/404/422 — the request is WRONG; repeating it is
  // log spam at best, duplicate side effects at worst.
}
// The prerequisite nobody skips twice: only retry IDEMPOTENT operations.
// A timed-out POST /charge may have SUCCEEDED — retrying double-charges.
// Fix at the API level: idempotency keys (REST lesson) make retries safe.
// (import { setTimeout } from "node:timers/promises" — the awaitable timer.)

Part 5 — Async Error Discipline

// 1. The unhandled rejection: every promise chain needs an owner.
doBackgroundSync();                       // ❌ fire-and-forget: a rejection here
                                           //    hits unhandledRejection → crash (L5)
doBackgroundSync().catch(err => log.error({ err }, "bg sync failed"));  // ✅ owned

// 2. Fail fast vs fail soft — a deliberate choice per await:
const user = await getUser(id);           // fail fast: can't proceed without it
const recs = await getRecs(id).catch(() => []);   // fail soft: degraded is fine
// Mark the soft ones LOUDLY (log the swallow) — silent .catch(() => {}) is how
// outages hide for weeks.

// 3. Preserve the chain — wrap with `cause`, never rethrow bare strings:
try { await chargeCard(order); }
catch (err) {
  throw new Error(`charge failed for order ${order.id}`, { cause: err });
}   // logs show YOUR context AND the root ECONNRESET — both matter.

// 4. await in loops — sequential when order matters, and say so:
for (const migration of migrations) await run(migration);   // ✅ deliberately serial
// vs the accidental serial map (JS track's classic) — reviewed with "is this
// order-dependent?" as the question.

// 5. Top-level orchestration (entry files):
main().catch(err => { console.error(err); process.exitCode = 1; });
// — the one place a broad catch is correct: translate failure to exit code.

Part 6 — Composed Patterns (Senior Layer)

// These three compose into the standard "resilient call" wrapper:
const result = await retry(
  ({ signal }) => limit(() => fetchJson(url, { signal: AbortSignal.any([signal, AbortSignal.timeout(3000)]) })),
  { attempts: 3, signal: req.signal }
);
// deadline per attempt + bounded concurrency + jittered retries + caller abort.
// This shape (plus circuit breaking + metrics) is what resilience libraries
// package; each layer's WHY now lives in your head, not the README.
//
// Deduplication — many callers, one in-flight operation (cache stampede guard):
const inflight = new Map();
export function dedupe(key, fn) {
  if (inflight.has(key)) return inflight.get(key);        // join the existing call
  const p = fn().finally(() => inflight.delete(key));
  inflight.set(key, p);
  return p;
}
// 200 requests hit an expired cache entry → ONE upstream call, 200 awaiters.
// (The Caching lesson pairs this with TTLs; React Query does this client-side.)
//
// Queues over concurrency for SUSTAINED work: pLimit bounds a burst; a job
// queue (persistent, retryable, observable) is the tool once work outlives
// the request or must survive restarts — the Scaling lesson's territory.

Interview Questions

Beginner: What is the error-first callback convention and how do you modernize it?

Legacy Node async APIs take a final callback invoked as cb(err, result) — error in the first position, checked before touching the result. Modernize with util.promisify (binding methods to preserve this), or use the node:fs/promises-style promise APIs where provided; hand-wrap only nonstandard signatures, always wiring both resolve and reject paths.

Beginner: Why is Promise.all(items.map(fetchThing)) dangerous on a server when it's fine in a browser?

Browsers cap concurrent connections (~6 per host), implicitly throttling you. Node has no such cap: mapping 10,000 items fires 10,000 simultaneous operations — exhausting DB pools, tripping rate limits, and spiking memory (every in-flight result buffered). Server code bounds concurrency explicitly (p-limit) at a number derived from the weakest downstream resource.

Mid: Design the timeout/cancellation story for a request handler that calls three services.

One request-scoped signal: AbortSignal.any([req.signal, AbortSignal.timeout(overallBudget)]) — client disconnects and the total deadline both cancel everything. Each service call gets its own per-attempt timeout composed with the request signal, and forwards the signal into fetch/DB calls so work actually stops. Handlers distinguish AbortError (cleanup, count a cancellation metric) from real failures. Budgeting: per-call timeouts sum (or nest, for sequential calls) under the overall budget minus response-writing headroom — a 5s budget doesn't grant three 5s calls.

Mid: Why exponential backoff AND jitter — what does each solve?

Exponential backoff solves persistence: retrying every 100ms hammers a struggling service; doubling delays gives it recovery room while still eventually succeeding. Jitter solves synchronization: if 1,000 clients fail together (a blip), fixed backoff schedules 1,000 retries at the same instants — recurring thundering herds that re-kill the recovering service. Randomizing each delay spreads the wave into a trickle. Backoff is per-client politeness; jitter is fleet-level decorrelation — production retry logic needs both, plus a retry budget/cap.

Senior: A checkout intermittently double-charges customers. The client has retry logic. Walk the failure and the layered fix.

The retry fires on timeout — but a timeout means "no response," not "no effect": the original POST /charge may have completed after the client gave up, so the retry charges again. Layered fix: (1) idempotency keys — client generates a unique key per checkout intent; the server stores key→result and replays the stored response for duplicates, making retries safe by contract; (2) retry policy — only on transport-level failures and 5xx/429, never on ambiguity without the key, with capped jittered backoff; (3) reconciliation — a sweeper matching provider charges to orders catches anything that slips; (4) the provider's own idempotency support (Stripe et al.) used end-to-end. The principle: retries are only as safe as the API's idempotency design — fix the contract, not the client loop.

Senior: Explain request coalescing (dedupe) and where it sits versus caching and rate limiting.

Coalescing collapses concurrent identical operations into one in-flight promise all callers share — keyed by operation identity, cleared on settle. It solves the stampede window that caching alone leaves: when a hot cache entry expires, N requests miss simultaneously and all hit the origin; with coalescing, one refreshes while N−1 wait for it. Distinctions: caching stores results across time; coalescing shares work within a moment; rate limiting bounds throughput regardless of identity. Production caches (and React Query client-side) compose all three: serve-stale-while- revalidate + single-flight refresh + limits. Caveat: coalescing shares failures too — one bad upstream response rejects every waiter, so pair it with retry policy deliberately.

3 (medium): Import 20,000 records via an API allowing 5 concurrent requests and 100 req/s. Write the import loop respecting both limits.
Solution 
import { setTimeout as sleep } from "node:timers/promises";

const limit = pLimit(5);                          // concurrency bound
let windowStart = Date.now(), windowCount = 0;

async function rateGate() {                       // 100/s token-ish gate
  if (Date.now() - windowStart >= 1000) { windowStart = Date.now(); windowCount = 0; }
  if (++windowCount > 100) {
    await sleep(1000 - (Date.now() - windowStart));
    return rateGate();
  }
}

const results = await Promise.allSettled(
  records.map(rec => limit(async () => {
    await rateGate();
    return retry(({ signal }) => api.upsert(rec, { signal }), { attempts: 3 });
  })));

const failed = results.filter(r => r.status === "rejected");
console.log(`imported ${results.length - failed.length}, failed ${failed.length}`);
// allSettled: one bad record shouldn't abort 19,999 good ones — collect and
// report. ponytail: window-reset rate gate is approximate; a token bucket
// (Caching/Redis lesson) is the precise version.
4 (medium): Add a proper timeout story to getDashboard() which currently awaits getUser, getOrders, getRecs sequentially with no deadlines: total budget 2s, recs are optional, orders depend on user.
Solution 
async function getDashboard(userId, { signal } = {}) {
  const budget = AbortSignal.any(
    [AbortSignal.timeout(2000), ...(signal ? [signal] : [])]);

  const user = await getUser(userId, { signal: budget });        // required, serial
  const [orders, recs] = await Promise.all([
    getOrders(user.id, { signal: budget }),                      // required
    getRecs(user.id, { signal: AbortSignal.any([budget, AbortSignal.timeout(500)]) })
      .catch(() => []),                                          // optional: tight
  ]);                                                            // deadline + fail-soft
  return { user, orders, recs };
}
// Orders and recs parallelize (only user was a real dependency); recs get a
// 500ms sub-deadline so a slow recommender can't eat the budget; one shared
// signal means a client disconnect stops ALL downstream work.

6 (hard): Implement dedupe with a twist: coalesced callers each get their own AbortSignal respected — if SOME callers abort, they reject with AbortError, but the underlying operation continues for the rest; it aborts only when ALL callers have aborted.
Solution 
const inflight = new Map();   // key → { promise, controller, waiters }

export function dedupe(key, fn, { signal } = {}) {
  let entry = inflight.get(key);
  if (!entry) {
    const controller = new AbortController();
    entry = { controller, waiters: 0,
      promise: fn({ signal: controller.signal })
        .finally(() => inflight.delete(key)) };
    inflight.set(key, entry);
  }
  entry.waiters++;

  return new Promise((resolve, reject) => {
    const onAbort = () => {
      entry.waiters--;
      if (entry.waiters === 0) entry.controller.abort();   // last one out: cancel
      reject(new DOMException("Aborted", "AbortError"));    // MY caller sees abort
    };
    signal?.addEventListener("abort", onAbort, { once: true });
    entry.promise.then(
      v => { signal?.removeEventListener("abort", onAbort); resolve(v); },
      e => { signal?.removeEventListener("abort", onAbort); reject(e); });
  });
}
// Reference counting on waiters: individual cancellations detach cheaply;
// the expensive upstream call is aborted only when nobody wants it anymore.
// This is exactly how production request-coalescing layers behave.
7 (hard): A nightly sync job (fetch 500 pages from a partner API, upsert each to the DB) currently: Promise.all over all pages, no timeouts, retries everything 5× on any error, and crashed last night taking the partner's API with it. Rewrite the design, not just the code — name each pattern you apply and why.
Solution 
// Postmortem of the original: unbounded fan-out (500 concurrent fetches) hit
// the partner like a DDoS; their 429s/timeouts triggered 5× blanket retries
// WITHOUT jitter (retry waves), including retrying 401s from an expired token
// (guaranteed-failure spam); one rejection nuked the whole Promise.all so
// partial progress was lost; no timeouts meant hung sockets held the job open.
// Redesign:
async function nightlySync({ signal } = {}) {
  const limit = pLimit(5);                       // ① bounded: respects partner capacity
  const results = await Promise.allSettled(      // ② partial-failure tolerant
    pages.map(page => limit(() =>
      retry(                                     // ③ retries: transient-only,
        ({ signal: s }) => syncPage(page, {      //    exponential + JITTER, max 3
          signal: AbortSignal.any([s, AbortSignal.timeout(10_000)]),  // ④ deadline
        }),
        { attempts: 3, retryOn: isTransient, signal }
      ))));

  const failed = results
    .map((r, i) => ({ r, page: pages[i] }))
    .filter(x => x.r.status === "rejected");
  await recordSyncReport({ total: pages.length, failed });   // ⑤ observable outcome
  if (failed.length > pages.length * 0.1) process.exitCode = 1;  // ⑥ honest exit:
}                                                 //    scheduler alerts past threshold
// ⑦ syncPage's upsert is idempotent (keyed on partner ids) so retries and
//    re-runs are safe — the job can simply run again for the failed subset.
// ⑧ And the meta-fix: respect their rate-limit headers (429 + Retry-After)
//    inside isTransient/backoff — being a good client is an uptime feature.

🎯 Key Takeaways

  • Bridge legacy with promisify (+bind) and hand-wrapped promises that always wire the failure path; never wrap what's already a promise.
  • Server rule #1: bound your concurrency — pLimit at a number derived from the weakest downstream; allSettled when partial failure is tolerable.
  • Every outward await gets a deadline; AbortSignal composes deadlines with caller cancellation and must be forwarded so aborted work actually stops.
  • Retries: transient errors only, exponential backoff with jitter, capped attempts — and only on idempotent operations (idempotency keys make them so).
  • Every promise chain has an owner; fail-fast vs fail-soft is chosen per await, with soft paths logged; wrap errors with cause to keep the chain.
  • Compose: limit + timeout + retry + dedupe is the standard resilient call; sustained work graduates from patterns to queues.
