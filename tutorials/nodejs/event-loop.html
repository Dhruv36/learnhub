Event Loop & libuv Internals Advanced

The JS event-loop lesson covered the browser's one-macrotask/drain-microtasks model. Node's loop is libuv's, with distinct phases and two extra queues (nextTick, microtasks) that jump ahead of everything. This is the highest-signal Node interview topic and the model behind every "why is my server slow" answer.

The libuv Phases (one full tick)

┌─▶ timers          expired setTimeout / setInterval callbacks
│   pending         deferred I/O callbacks (some system errors)
│   poll            ★ retrieve new I/O events; RUN their callbacks; may BLOCK here waiting
│   check           setImmediate callbacks
│   close           'close' events (socket.on('close'))
└── (repeat)

BETWEEN EVERY phase (and after each callback): drain process.nextTick queue,
                                                 then the Promise microtask queue.
So priority per step:  nextTick  >  promises/await  >  the next phase's callbacks.

The Ordering Puzzles

setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
// In the MAIN module: order is NON-DETERMINISTIC (depends on loop start timing).
// INSIDE an I/O callback: 'immediate' ALWAYS wins (poll → check is next; timers is a full loop away).

const fs = require("node:fs");
fs.readFile(__filename, () => {
  setTimeout(() => console.log("timeout"), 0);
  setImmediate(() => console.log("immediate"));      // → immediate, timeout (deterministic here)
});

Promise.resolve().then(() => console.log("promise"));
process.nextTick(() => console.log("nextTick"));
console.log("sync");
// → sync, nextTick, promise   (nextTick jumps even ahead of promises)

nextTick — Power and Footgun

// Runs BEFORE the loop continues — even before promise microtasks. Recursion STARVES the loop:
function loop() { process.nextTick(loop); }  loop();   // 💥 I/O never processes again
// Legit uses are rare (emit events after the current op, API consistency).
// App code should prefer promises/queueMicrotask; know nextTick exists mainly for the interview
// and for reading library internals.

What Blocks vs What Doesn't (the server-critical part)

// The single JS thread runs YOUR code. libuv handles I/O off-thread. So:
crypto.pbkdf2Sync(...)          // ❌ BLOCKS — sync crypto on the request path freezes all users
JSON.parse(hugeString)          // ❌ BLOCKS — CPU-bound, no async version
while (heavyLoop) {}            // ❌ BLOCKS — obviously
await pool.query(...)           // ✅ non-blocking — libuv waits, thread serves others
crypto.pbkdf2(..., cb)          // ✅ async version → runs on libuv's THREAD POOL (4 threads default)

// libuv's thread pool (UV_THREADPOOL_SIZE, default 4) handles: fs, dns.lookup, async crypto,
// zlib. Saturate it (many parallel bcrypt/zlib) and even "async" work queues → tune the size
// or offload to worker_threads (scaling lesson). Network I/O uses the OS (epoll), NOT the pool.

Measuring Event-Loop Health

import { monitorEventLoopDelay } from "node:perf_hooks";
const h = monitorEventLoopDelay(); h.enable();
setInterval(() => { if (h.mean/1e6 > 50) logger.warn(`loop lag ${(h.mean/1e6)|0}ms`); h.reset(); }, 5000);
// Event-loop lag = the p99 latency killer. Rising lag under load = something is blocking.
// This metric belongs on every production dashboard (observability, errors lesson).

Interview Questions

Q1: setTimeout(0) vs setImmediate — when is the order deterministic?

In the main module they race (depends on how fast setup reaches the loop, sub-millisecond timer readiness). INSIDE an I/O callback the loop is in poll → the very next phase is check (setImmediate), while timers is a whole loop away → immediate reliably precedes timeout. The lesson: never rely on their relative order for correctness; use them for their SEMANTICS (immediate = "after this poll phase").

Q2: Explain how one thread serves 10k concurrent connections.

Network I/O is registered with the OS (epoll/kqueue/IOCP); libuv's poll phase collects ready events and runs their JS callbacks one at a time. While a request awaits the DB, the thread runs OTHER requests' callbacks — connections are cheap state, not threads. It falls apart only for CPU-bound work, which has no I/O to await and hogs the single thread — the reason workers exist.

Q3: What is the libuv thread pool, and what surprises people about it?

A pool (default 4 threads) that runs fs operations, dns.lookup, async crypto, and zlib — NOT network sockets (those use the OS directly). The surprise: firing many parallel bcrypt hashes or gzip calls saturates 4 threads, so further "async" calls queue behind them, adding latency that looks like a mystery. Fix: raise UV_THREADPOOL_SIZE or move heavy CPU crypto to worker_threads.

🏋️ Practice: reproduce the setTimeout/setImmediate ordering both in main and inside fs.readFile; write the nextTick starvation loop and watch a pending timer never fire; add the loop-lag monitor, then hit a sync-crypto endpoint under load and watch lag climb.

🎯 Key Takeaways

  • Phases: timers → pending → poll → check → close; nextTick + microtasks drain between every step and every callback.
  • Priority: nextTick > promises > next phase; setImmediate = "after this poll", timers = "after delay".
  • One thread runs your JS; libuv + OS handle I/O — CPU work blocks everyone, no exceptions.
  • The 4-thread pool runs fs/async-crypto/zlib (not sockets); saturating it queues "async" work.
  • Event-loop lag is the latency metric to watch; blocking shows up there first.
