REST API Design & Validation Basics→Intermediate

An API is a contract consumed by strangers (including future-you). Good REST design is mostly discipline: resource-shaped URLs, honest methods/status codes, validated inputs, consistent errors, and pagination from day one. This lesson is the checklist that survives code review at good companies.

Resource Modeling — URLs Are Nouns

GET    /api/v1/products                 list (filter/sort/paginate via query)
POST   /api/v1/products                 create → 201 + Location
GET    /api/v1/products/42              read one → 200 / 404
PUT    /api/v1/products/42              replace → 200
PATCH  /api/v1/products/42              partial update → 200
DELETE /api/v1/products/42              → 204
GET    /api/v1/products/42/reviews      nested resource (one level max — then link, don't nest)

❌ /getProducts, /product/delete/42, POST /products/42/update   ← verbs in URLs = RPC cosplay
✔ Actions that aren't CRUD become sub-resources: POST /orders/42/cancellation
✔ /api/v1 prefix: version from day one — breaking changes get v2, old clients keep working

Validation — zod at Every Boundary (the TS lesson, server-side)

const CreateProduct = z.object({
  name: z.string().min(1).max(200),
  price: z.number().int().positive(),            // integer cents!
  tags: z.array(z.string()).max(10).default([]),
});
const ListQuery = z.object({
  page: z.coerce.number().int().min(1).default(1),      // coerce: query strings → numbers
  limit: z.coerce.number().int().min(1).max(100).default(20),   // ALWAYS cap limit
  sort: z.enum(["price", "-price", "createdAt"]).default("createdAt"),
});

// As middleware — validated data replaces the raw input:
const validate = (schema, source = "body") => (req, res, next) => {
  const r = schema.safeParse(req[source]);
  if (!r.success) return res.status(400).json({ code: "VALIDATION", errors: r.error.flatten() });
  req[source] = r.data; next();                   // downstream sees TYPED, defaulted, capped data
};
app.post("/api/v1/products", validate(CreateProduct), createProduct);
// Validate body, query, params, headers — all four arrive as untrusted strings.

Response & Error Envelope — Pick One Shape, Everywhere

// Success: the resource (or a list envelope):
{ "data": [...], "pagination": { "page": 2, "limit": 20, "total": 4123 } }
// Errors — machine-readable code + human message + field details:
{ "code": "VALIDATION", "message": "Invalid input",
  "errors": { "price": ["must be positive"] }, "requestId": "…" }
// requestId ties the client report to your logs (middleware lesson). Inconsistent error
// shapes are the #1 complaint API consumers have — one envelope, enforced by the error boundary.

Pagination — Offset vs Cursor (a system-design crossover)

Offset:  GET /products?page=50&limit=20   → OFFSET 980
  simple, jumpable — but deep pages get slow AND items shift when rows are inserted mid-scroll
Cursor:  GET /products?after=eyJpZCI6OTgw&limit=20   → WHERE id > 980 (opaque cursor)
  stable under writes, O(1) at any depth — but no random page jumps
Rule: admin tables → offset is fine; infinite feeds/big data → cursor.

Interview Questions

Q1: Design the API for a todo SaaS (lists, todos, sharing). Sketch endpoints + key decisions.

/v1/lists, /v1/lists/:id/todos (one nesting level), /v1/todos/:id for direct access; sharing as POST /v1/lists/:id/members {email, role} → membership resource. Decisions to narrate: cursor pagination for todos, zod validation middleware, error envelope with codes, ETags on list GETs, idempotency keys on todo-creation from mobile (retry!), and authorization scoping every query (WHERE list_id IN user's memberships — auth lesson).

Q2: PUT vs PATCH — semantics and the trap in each?

PUT replaces the whole resource (missing fields = removed — clients must send everything; idempotent by definition). PATCH applies a partial change (send only what changes; make it validated so undefined ≠ null semantics are explicit: null = clear field, absent = leave alone). The trap: implementing PUT as merge (it silently becomes PATCH) — pick semantics and document them.

Q3: Why cursor pagination for feeds — the precise failure of offset?

Two failures: performance (OFFSET n scans and discards n rows — page 5000 = misery) and correctness under writes (a new row inserted at the top shifts everything: users see duplicates or miss items between pages). A keyset cursor (WHERE created_at,id < last-seen) is index-seek fast and stable because it anchors to VALUES, not positions. Trade-off: no jump-to-page-N.

🏋️ Practice: build /v1/products with the full checklist: zod-validated body+query, capped pagination (try both offset and cursor), one error envelope, ETag on GET-by-id, and a deliberate uncapped-limit endpoint you then fix. Document it by generating OpenAPI from the schemas.

🎯 Key Takeaways

  • URLs are nouns; methods are the verbs; /v1 from day one; one nesting level.
  • zod-validate body/query/params — coerce, default, CAP limits; typed data flows downstream.
  • One response/error envelope with machine codes + requestId; DTO-map, never leak rows.
  • Offset pagination for small admin tables; cursors for feeds/scale.
  • ETags, rate limits, idempotency keys, OpenAPI-from-schemas — the production baseline.
