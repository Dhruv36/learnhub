Files, Paths & Buffers Runtime Foundations

What you'll master: the modern promise-based fs API (and when sync is actually right), cross-platform path handling, Buffers — binary data and encodings, the concept browsers hid from you — atomic writes, watching, and the safety rules (TOCTOU, traversal) that make file code production-grade. Prerequisites: The Node Runtime, async/await.

Part 1 — Three fs APIs, One Rule

import fs from "node:fs/promises";          // 1. PROMISES — your default:
const text = await fs.readFile("notes.txt", "utf8");
await fs.writeFile("out.txt", text.toUpperCase());

import { readFileSync } from "node:fs";     // 2. SYNC — blocks the event loop:
const config = JSON.parse(readFileSync("config.json", "utf8"));
// LEGITIMATE at startup (before the server listens: config, certs — simpler
// code, nothing to block yet) and in CLI scripts. FORBIDDEN on request paths.

import { readFile } from "node:fs";         // 3. CALLBACK — the legacy layer
readFile("x", "utf8", (err, data) => {});   // you'll read in old code; don't write it.

// The rule, restated from the runtime lesson: any fs call per request must be
// async. One readFileSync in a hot handler serializes your whole server.
// (Under the hood, ASYNC fs runs on libuv's thread pool — files have no
// non-blocking OS API like sockets do. Event Loop lesson makes this precise.)

Part 2 — The Everyday fs Toolkit

// Reading & writing:
await fs.readFile(p, "utf8");                 // whole file as string
await fs.readFile(p);                          // no encoding → Buffer (Part 4)
await fs.writeFile(p, data);                   // create-or-REPLACE
await fs.appendFile("app.log", line + "\n");
await fs.mkdir("out/reports", { recursive: true });   // mkdir -p
await fs.rm("tmp", { recursive: true, force: true }); // rm -rf
await fs.rename(from, to);                     // move (same volume: atomic!)
await fs.copyFile(src, dest);
await fs.readdir("uploads", { withFileTypes: true }); // Dirent[]: .isDirectory()

// Metadata:
const st = await fs.stat(p);                   // size, mtime, isFile()…
// EXISTENCE — the interview trap:
if (await fs.exists(p)) {}                     // doesn't exist in fs/promises, on purpose
try { await fs.access(p); } catch {}           // exists NOW…
// …but check-then-act is a RACE (TOCTOU): the file can vanish between check
// and use. Correct pattern: just DO the operation and handle the error:
try {
  const data = await fs.readFile(p, "utf8");
} catch (err) {
  if (err.code === "ENOENT") return defaults;  // "no such file" — expected case
  throw err;                                   // everything else is a real error
}
// Error codes you'll match constantly: ENOENT (missing), EACCES (permissions),
// EEXIST (already there), EISDIR/ENOTDIR (wrong kind).

Part 3 — Paths: Never Concatenate Strings

import path from "node:path";

path.join("uploads", user, file)        // "uploads/u1/a.png" — correct separators
path.resolve("data", "db.json")         // ABSOLUTE from cwd: /app/data/db.json
path.dirname("/a/b/c.txt")              // "/a/b"
path.basename("/a/b/c.txt", ".txt")     // "c"
path.extname("photo.JPEG")              // ".JPEG" (normalize case yourself)

// WHY not `dir + "/" + file`: Windows (\ separators, C: drives), doubled or
// missing slashes, and no normalization of ".." segments.
//
// Two bases to be deliberate about:
process.cwd()          // where the USER ran node from — right for CLI inputs
import.meta.dirname    // where THIS FILE lives — right for assets shipped
                       // with your code (templates, migrations):
const tpl = path.join(import.meta.dirname, "templates", "email.html");
// Mixing these up = "works in dev, ENOENT under systemd/Docker" (cwd differs).

// THE security pattern (from the runtime lesson's server, now named):
const ROOT = path.resolve("uploads");
const target = path.resolve(ROOT, userInput);          // resolve COLLAPSES ../
if (!target.startsWith(ROOT + path.sep)) throw new Error("Traversal blocked");
// Resolve-then-verify-prefix. Never sanitize ".." with string replaces —
// encodings and platform quirks will beat you.

Part 4 — Buffers: Binary Data, Finally Explicit

// A Buffer is a fixed-size chunk of raw bytes outside V8's heap —
// Node's Uint8Array subclass. Everything I/O is bytes; strings are an
// INTERPRETATION of bytes via an encoding:

const buf = Buffer.from("héllo", "utf8");
buf.length                                   // 6 — BYTES, not characters! é = 2 bytes
"héllo".length                               // 5 — UTF-16 code units. Different axes.
buf.toString("utf8")                         // "héllo" — bytes → string, encoding named
buf.toString("hex")                          // "68c3a96c6c6f"
buf.toString("base64")                       // "aMOpbGxv"

// The encodings that matter: utf8 (text default), base64 (binary-in-JSON,
// data URLs), hex (hashes, ids), latin1 (legacy protocols).
Buffer.from(base64String, "base64")          // decode: string → bytes
Buffer.alloc(16)                             // 16 zeroed bytes (alloc, not
                                              // allocUnsafe, unless profiling says so)
Buffer.concat([a, b])                        // join chunks — the streams staple

// Where Buffers appear whether you like it or not:
// fs.readFile without encoding · network sockets · crypto digests ·
// file uploads · anything streamed (next section's whole subject).
const hash = crypto.createHash("sha256").update(buf).digest("hex");

// The classic multi-byte bug — chopping bytes mid-character:
const bytes = Buffer.from("café au lait");
bytes.subarray(0, 4).toString()              // "caf�" — é's 2 bytes split!
// Never slice byte buffers and assume valid text; decode whole buffers, or
// use string_decoder/TextDecoder(stream:true) across chunk boundaries.
The mental model: bytes are the truth; strings are a lens. Every boundary (file, socket, hash) speaks bytes, and every conversion must name its encoding. Code that "works in English and breaks in French/Hindi/emoji" always turns out to be an implicit-encoding or byte-length-vs-char-length bug — this section is the vaccine.

Part 5 — Production Patterns

// 1. ATOMIC WRITES — never let readers see a half-written file:
async function writeAtomic(file, data) {
  const tmp = `${file}.${process.pid}.${Date.now()}.tmp`;
  await fs.writeFile(tmp, data);
  await fs.rename(tmp, file);                // same-volume rename is atomic:
}                                             // readers see old OR new, never half
// (writeFile alone can be interrupted mid-write: crash = corrupt config/JSON.)

// 2. JSON files as tiny persistence (fine for CLIs/small tools):
async function loadState(file, fallback) {
  try { return JSON.parse(await fs.readFile(file, "utf8")); }
  catch (err) {
    if (err.code === "ENOENT") return fallback;
    throw err;                                // corrupt JSON should CRASH, not default —
  }                                           // silent data loss hides behind fallbacks
}
const saveState = (file, state) => writeAtomic(file, JSON.stringify(state, null, 2));

// 3. WATCHING for changes:
const watcher = fs.watch("config", { recursive: true });
for await (const event of watcher) {          // async-iterable API
  console.log(event.eventType, event.filename);
}
// Caveats that fill Stack Overflow: duplicate events per save (editors write
// twice), platform differences, renames reported oddly → debounce, and reach
// for chokidar when fidelity matters.

// 4. BIG FILES: readFile loads EVERYTHING into memory. A 2GB log = 2GB heap.
//    Line-by-line and chunked processing belong to streams:
import readline from "node:readline";
const rl = readline.createInterface({ input: createReadStream("huge.log") });
for await (const line of rl) { /* one line at a time, constant memory */ }
// The full story — pipelines, backpressure — is the Streams lesson.

Part 6 — Temp Files, Permissions & Cross-Platform (Senior Layer)

// Temp files done right:
import os from "node:os";
const dir = await fs.mkdtemp(path.join(os.tmpdir(), "report-"));
// mkdtemp = unique, race-free directory. Clean up in finally { fs.rm(dir, …) }.

// Permissions (POSIX):
await fs.writeFile("secret.pem", key, { mode: 0o600 });   // owner read/write only
// Modes matter for keys/tokens on disk; on Windows they're approximated — don't
// build security on chmod alone there.

// Cross-platform checklist (the Windows-CI-is-red starter pack):
// • path.join/sep everywhere — zero hand-built "/" paths
// • case-sensitivity: Linux prod is case-sensitive; macOS dev usually isn't —
//   "Logo.png" vs "logo.png" ships broken. Match case exactly.
// • line endings: split on /\r?\n/, write "\n" (git's autocrlf handles files)
// • reserved names (con, aux), path length limits on Windows — avoid exotic names
// • file locking doesn't exist portably — coordinate via atomic renames or a
//   real database once multiple processes write (Scaling lesson's territory).

Interview Questions

Beginner: Why does Node have three fs APIs, and when is the sync one acceptable?

History and use-cases: callback style is the original, promises (fs/promises) the modern default, sync the blocking variant. Sync is right where blocking is harmless or even desirable — startup (loading config/certs before the server accepts traffic) and CLI scripts — because the code is simpler and nothing concurrent exists yet. On any request path, sync fs serializes the event loop and every user pays for one user's disk read.

Beginner: What is a Buffer and why does buf.length !== str.length?

A Buffer is Node's raw-bytes container (a Uint8Array subclass, memory outside the V8 heap) — how binary data from files, sockets, and crypto is represented. buf.length counts bytes; str.length counts UTF-16 code units. Non-ASCII characters occupy multiple UTF-8 bytes ("é" = 2, "🎉" = 4), so the numbers diverge — and Content-Length headers, size limits, and slicing must use byte math (Buffer.byteLength).

Mid: What's TOCTOU and the correct fs idiom that avoids it?

Time-of-check-to-time-of-use: verifying a condition (file exists, is writable) and then acting on it as two steps — the world can change in between (deletion, symlink swap — a real attack vector for privileged processes). The idiom: don't check, act — attempt the read/write/mkdir and handle the specific error code (ENOENT, EEXIST) that encodes the condition. One atomic syscall replaces check+act; exclusive-create flags ({ flag: "wx" }) express "create only if absent" race-free.

Mid: How do you write a config file so a crash can never corrupt it?

Write-to-temp-then-rename: write the full content to a uniquely named temp file on the same volume, then fs.rename over the target — rename within a filesystem is atomic, so any reader (and any crash) sees either the complete old file or the complete new one, never a prefix. Add fsync on the temp file (and directory) if the durability bar is power-loss, not just process-crash. This is also why "same volume" matters: cross-device rename falls back to copy+delete and loses atomicity.

Senior: A service reads user-referenced files. Give the complete safety checklist.

(1) Path traversal: resolve against a fixed root and verify the result stays under it (prefix + separator check) — after decodeURIComponent, never via string-stripping "..". (2) Symlinks: an in-root symlink can point outside — fs.realpath before the prefix check, or open with O_NOFOLLOW semantics where the threat model requires. (3) Size limits before reading (stat first or stream with a cap) — one huge file is a DoS. (4) Type check: is it a regular file (not a directory, FIFO, device)? (5) TOCTOU discipline: act-and-handle-errors, no check-then-read. (6) Least privilege: the process user can only read the intended tree — the OS as the final boundary when the code check slips.

Senior: Your log-processing job OOMs on large days. It reads each 5GB file with readFile, splits on newlines, and maps rows. Redesign it and name the memory profile.

readFile materializes 5GB as one Buffer, then toString doubles it as a string, then split creates millions of line strings — three multiples of file size peak-resident. Redesign: stream it — createReadStream into readline (or a split transform), processing one line at a time with constant memory (a chunk buffer plus one line), aggregating results incrementally; parallelize across files, not within one. If lines are written to another store, use pipeline with backpressure so a slow sink throttles the read (Streams lesson). Memory profile goes from O(file) to O(chunk + longest line) — and byte-boundary correctness (multi-byte chars split across chunks) comes free from readline/string_decoder rather than hand-split buffers.

🏋️ Graded Exercises

1 (easy): Write lsx.js: list a directory (argv or cwd), marking directories with a trailing "/" and showing file sizes in KB.
Solution 
import fs from "node:fs/promises";
import path from "node:path";

const dir = process.argv[2] ?? ".";
for (const entry of await fs.readdir(dir, { withFileTypes: true })) {
  if (entry.isDirectory()) { console.log(entry.name + "/"); continue; }
  const { size } = await fs.stat(path.join(dir, entry.name));
  console.log(`${entry.name}  ${(size / 1024).toFixed(1)}KB`);
}
2 (easy): What's wrong here, on two axes? if (fs.existsSync(f)) { const data = await fs.readFile(f, "utf8"); }
Solution 
// Axis 1 — TOCTOU race: f can be deleted between the check and the read;
// the "guarded" read still needs error handling, so the guard buys nothing.
// Axis 2 — existsSync BLOCKS the event loop (sync call in otherwise-async code).
try {
  const data = await fs.readFile(f, "utf8");
} catch (err) {
  if (err.code !== "ENOENT") throw err;
  // handle missing-file case here
}
3 (medium): Build a JSON "database": load() with ENOENT-fallback and corrupt-file crash, save() atomic, and update(fn) composing them. State the concurrency limit honestly.
Solution 
import fs from "node:fs/promises";

export function jsonStore(file, fallback = {}) {
  async function load() {
    try { return JSON.parse(await fs.readFile(file, "utf8")); }
    catch (err) {
      if (err.code === "ENOENT") return structuredClone(fallback);
      throw err;                          // corrupt JSON = loud failure, not defaults
    }
  }
  async function save(state) {
    const tmp = `${file}.${process.pid}.tmp`;
    await fs.writeFile(tmp, JSON.stringify(state, null, 2));
    await fs.rename(tmp, file);           // atomic swap
  }
  async function update(fn) {
    const state = await load();
    const next = fn(state) ?? state;
    await save(next);
    return next;
  }
  return { load, save, update };
}
// ponytail: single-process only — two processes calling update() interleave
// load/save and lose writes. Multi-writer needs a real DB (Databases lesson).

5 (medium): Implement checksumDir(dir): recursively hash every file (sha256, hex) into { relativePath: hash }, with stable ordering — the core of a build-cache/integrity tool.
Solution 
import fs from "node:fs/promises";
import path from "node:path";
import crypto from "node:crypto";

export async function checksumDir(dir) {
  const root = path.resolve(dir);
  const out = {};
  async function walk(d) {
    const entries = await fs.readdir(d, { withFileTypes: true });
    entries.sort((a, b) => a.name.localeCompare(b.name));     // stable order
    for (const e of entries) {
      const full = path.join(d, e.name);
      if (e.isDirectory()) { await walk(full); continue; }
      const bytes = await fs.readFile(full);                   // Buffer — no encoding
      out[path.relative(root, full).replaceAll(path.sep, "/")] =  // portable keys
        crypto.createHash("sha256").update(bytes).digest("hex");
    }
  }
  await walk(root);
  return out;
}
// Hashing BYTES (not utf8 strings) keeps binaries correct; forward-slash keys
// make the manifest identical across OSes — diffable in CI.
6 (hard): Build a rotating logger: log(line) appends with a timestamp; when the file exceeds 1MB it rotates to app.log.1 (bumping .1→.2 up to .5, dropping the oldest). Handle concurrent log() calls within the process.
Solution 
import fs from "node:fs/promises";

export function rotatingLogger(file, { maxBytes = 1_048_576, keep = 5 } = {}) {
  let queue = Promise.resolve();                 // serialize ALL ops: append and
                                                  // rotate can't interleave
  async function rotate() {
    await fs.rm(`${file}.${keep}`, { force: true });          // drop oldest
    for (let i = keep - 1; i >= 1; i--) {
      try { await fs.rename(`${file}.${i}`, `${file}.${i + 1}`); }
      catch (err) { if (err.code !== "ENOENT") throw err; }   // gaps are fine
    }
    await fs.rename(file, `${file}.1`);
  }
  function log(line) {
    queue = queue.then(async () => {
      const entry = `${new Date().toISOString()} ${line}\n`;
      await fs.appendFile(file, entry);
      const { size } = await fs.stat(file);
      if (size >= maxBytes) await rotate();
    });
    return queue;
  }
  return { log };
}
// The promise-chain queue is the in-process mutex: appends and rotation are
// strictly ordered, so no append lands mid-rotation.
// ponytail: single-process again; real apps use pino + external rotation
// (Logging lesson) — this exercise is the mechanism those tools wrap.
7 (hard): An upload endpoint saves user files as uploads/<originalname>. List every attack/failure in that one line, then write the hardened version.
Solution 
// The one line's rap sheet:
// 1. Traversal: originalname = "../../etc/cron.d/evil" escapes uploads/.
// 2. Overwrite: two users upload "resume.pdf" — second silently replaces first;
//    or attacker overwrites another user's known filename.
// 3. Weird names: null bytes, reserved Windows names, 4KB-long names, "."/"..".
// 4. No size cap: one 50GB "file" fills the disk.
// 5. Trusting claimed type: "photo.jpg" containing PHP/HTML → stored-XSS or
//    RCE if ever served/executed.
// Hardened:
import crypto from "node:crypto";
import path from "node:path";

const ROOT = path.resolve("uploads");
const MAX = 10 * 1024 * 1024;
const OK_EXT = new Set([".png", ".jpg", ".pdf"]);

async function saveUpload(stream, originalName, declaredBytes) {
  if (declaredBytes > MAX) throw httpError(413);
  const ext = path.extname(originalName).toLowerCase();
  if (!OK_EXT.has(ext)) throw httpError(415);

  const id = crypto.randomUUID();                       // WE name the file —
  const target = path.join(ROOT, `${id}${ext}`);        // user input never touches
  // (belt-and-braces prefix check anyway:)              // the path
  if (!path.resolve(target).startsWith(ROOT + path.sep)) throw httpError(400);

  await pipeline(stream, byteLimiter(MAX), createWriteStream(target, { flags: "wx" }));
  //            streamed (no memory spike)  ^cap enforced   ^wx: fail if exists
  return { id, originalName };                          // original name = metadata
}                                                        // in the DB, not the path
// Serve back with Content-Disposition + a sanitized display name, from a domain
// that can't execute anything. The principle: user input becomes DATA
// (a DB column), never STRUCTURE (a filesystem path).

🎯 Key Takeaways

  • fs/promises everywhere requests flow; sync fs only at startup and in CLIs — one blocking read serializes the server.
  • Act, don't check: attempt operations and branch on err.code (ENOENT et al.) — check-then-act is a race by construction.
  • Paths are built with path.join/resolve, based deliberately (cwd for user input, import.meta.dirname for shipped assets), and user input is confined by resolve-then-prefix-verify.
  • Bytes are the truth, strings a lens: Buffers carry I/O, every conversion names its encoding, sizes are byte math, and multi-byte characters forbid naive slicing.
  • Production file-writing is atomic (temp + rename); big files are streamed, never readFile'd; watching needs debouncing.
  • Cross-platform and security are checklists, not vibes: case sensitivity, cwd drift, size caps, symlinks, and server-chosen filenames.
