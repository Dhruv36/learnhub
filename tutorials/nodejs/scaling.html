Scaling: Clusters & Workers Advanced

One Node process uses one CPU core. Scaling means using all cores and, eventually, many machines. Two distinct problems: throughput (more concurrent I/O-bound requests → run multiple processes) and CPU-bound work (don't block the loop → worker threads). Different tools; interviews test whether you know which.

Use All Cores — Cluster / Process-per-Core

// The OS load-balances connections across N processes sharing a port:
import cluster from "node:cluster";
import { availableParallelism } from "node:os";
if (cluster.isPrimary) {
  for (let i = 0; i < availableParallelism(); i++) cluster.fork();   // one worker per core
  cluster.on("exit", () => cluster.fork());                          // auto-respawn
} else { startServer(); }
// In production you usually DON'T hand-roll this — PM2 (pm2 start app.js -i max) or, better,
// a container orchestrator (Kubernetes: N pods, one process each) does it with health checks,
// rolling restarts, and autoscaling. Cluster is the mechanism; orchestration is the practice.
🔑 The prerequisite for horizontal scaling: STATELESSNESS. Multiple processes/pods can't share in-memory state — sessions, caches, rate-limit counters, and websocket maps must move to Redis/DB. An in-memory session breaks the moment a second instance exists. (This is the System Design track's "stateless web tier" rule, in Node terms.)

Worker Threads — for CPU-Bound Work

import { Worker } from "node:worker_threads";
function runHeavy(data) {
  return new Promise((resolve, reject) => {
    const w = new Worker("./heavy-worker.js", { workerData: data });
    w.on("message", resolve); w.on("error", reject);
    w.on("exit", (c) => c && reject(new Error(`exit ${c}`)));
  });
}
// heavy-worker.js: parentPort.postMessage(crunch(workerData))
// Same model as the JS Web Workers lesson: separate thread, message passing (structured clone
// or transfer for big buffers), NO shared state. For repeated jobs use a POOL (piscina library)
// sized to cores. Report generation, image resize, PDF, heavy crypto → workers, off the request path.

Cluster vs Workers — the Decision

Cluster / processesWorker threads
SolvesThroughput: many concurrent requests across coresCPU tasks blocking one event loop
IsolationFull processes (a crash kills one worker only)Threads in one process (share the process's fate)
MemorySeparate heaps (higher overhead)Shared process, cheaper; SharedArrayBuffer possible
Use forThe web tier itselfOffloading specific heavy computations

Beyond One Machine — the Async-Job Pattern

// Slow/heavy work shouldn't block the HTTP response at all — enqueue and return:
app.post("/reports", async (req, res) => {
  const job = await queue.add("generate-report", { userId: req.user.sub });   // BullMQ + Redis
  res.status(202).json({ jobId: job.id });                                     // 202 Accepted
});
// Separate WORKER processes consume the queue, do the work, store results; client polls or
// gets a webhook/websocket. This decouples request latency from work duration, absorbs spikes
// (the queue buffers), and scales workers independently — the System Design track's message-queue
// lesson, realized. Horizontal scaling = stateless web tier + shared datastore + queue + workers.

Interview Questions

Q1: Cluster vs worker_threads — when each, in one breath?

Cluster (multiple processes) scales THROUGHPUT of I/O-bound requests across cores with full isolation — it's how you run the web tier. Worker threads offload CPU-bound COMPUTATION off the event loop within a process. Cluster doesn't help a single blocking computation (each worker still has one loop); workers don't multiply request-serving capacity. Different axes: parallel requests vs parallel compute.

Q2: What must be true before you can run N instances behind a load balancer?

Statelessness: no request-affecting state in process memory. Sessions → Redis/DB, caches → shared cache, rate-limit counters → Redis, uploaded files → object storage, websocket routing → pub/sub adapter. Any instance must handle any request identically. In-memory anything that survives across requests is a horizontal-scaling bug waiting for the second pod.

Q3: A report endpoint takes 40s and times out under load. Redesign it.

Decouple: POST enqueues a job (Redis/BullMQ) and returns 202 + jobId immediately; dedicated worker processes generate the report and store it; client polls GET /reports/:jobId or receives a webhook/websocket on completion. Benefits: request latency independent of work time, spikes buffered by the queue, workers scaled separately, retries/DLQ for failures. This is the async-job pattern every heavy backend converges on.

🏋️ Practice: run the cluster snippet and hit it with a load tool (autocannon) — watch all cores light up; move a CPU-heavy endpoint into a worker and confirm other requests stay responsive during it; sketch (or build with BullMQ) a 202-enqueue + worker-consume report flow.

🎯 Key Takeaways

  • One process = one core; cluster/PM2/K8s run process-per-core for throughput.
  • Horizontal scaling requires a STATELESS web tier — externalize sessions/caches/counters.
  • Worker threads offload CPU-bound compute (pool them); cluster doesn't fix blocking.
  • Heavy/slow work → 202 + queue + separate workers, decoupling latency from duration.
  • At scale: stateless tier + shared datastore + queue + workers — the System Design pattern in Node.
