Routing & Middleware Basics

Middleware is Express's whole architecture: an ordered pipeline of functions, each seeing (req, res) and deciding to respond or pass control on via next(). Auth, logging, parsing, rate limiting, errors — all middleware. Master the pipeline and you've mastered Express (and recognized the same pattern in Fastify, Koa, ASP.NET, Django…).

The Pipeline Model

request → [logger] → [json parser] → [auth] → [router → handler] → response
              │            │             │
              └── next() ──┴── next() ───┘        any middleware may SHORT-CIRCUIT by responding

function requestLogger(req, res, next) {
  req.id = crypto.randomUUID();                       // decorate req for downstream
  const start = Date.now();
  res.on("finish", () =>                              // log AFTER response completes
    logger.info({ id: req.id, method: req.method, url: req.url,
                  status: res.statusCode, ms: Date.now() - start }));
  next();                                              // ← forget this and the request HANGS
}
app.use(requestLogger);
// ORDER = registration order. Parser before routes that read req.body;
// auth before what it protects; error handler LAST. Order bugs are middleware bugs.

The Middleware Zoo (what app.use lines mean)

app.use(express.json({ limit: "100kb" }));      // body parsing + DoS limit
app.use(helmet());                                // security headers (CSP, nosniff… — HTML security lesson)
app.use(cors({ origin: ["https://app.example.com"], credentials: true }));
app.use(rateLimit({ windowMs: 60_000, max: 100 }));   // per-IP throttle (system design: rate limiter!)
app.use("/static", express.static("public"));     // file serving
app.use("/api/admin", requireRole("admin"));      // PATH-SCOPED: only /api/admin/* pays the cost

Routers — Structure by Feature

// users.router.js — a mini-app per feature (React architecture lesson's twin):
export const usersRouter = Router();
usersRouter.get("/", listUsers);
usersRouter.get("/:id", loadUser, showUser);      // per-ROUTE middleware chain:
usersRouter.put("/:id", requireAuth, loadUser, requireOwner, updateUser);

// app.js composes:
app.use("/api/users", usersRouter);
app.use("/api/orders", ordersRouter);

// loadUser: fetch-or-404 once, decorate req — handlers stay tiny:
async function loadUser(req, res, next) {
  const user = await repo.find(Number(req.params.id));
  if (!user) return res.status(404).json({ code: "NOT_FOUND" });
  req.user = user; next();
}

Error Handling — the 4-Arg Contract

// Async handlers: rejected promises must reach next(err).
// Express 5 forwards rejections automatically; on 4.x use a wrapper:
const wrap = (fn) => (req, res, next) => fn(req, res, next).catch(next);

// THE error middleware — four args EXACTLY (that's how Express identifies it), registered LAST:
app.use((err, req, res, next) => {
  if (err instanceof ValidationError)             // the JS errors lesson's taxonomy, applied
    return res.status(400).json({ code: err.code, errors: err.meta });
  if (err instanceof NotFoundError)
    return res.status(404).json({ code: err.code });
  logger.error({ err, id: req.id });               // unknown = bug: log loudly…
  res.status(500).json({ code: "INTERNAL" });      // …tell softly (no stack to clients)
});
// One boundary maps error types → responses — handlers just throw.

Interview Questions

Q1: Design the middleware order for a production API and justify each position.

request-id + logger (observe everything, even failures) → helmet (headers on all responses) → CORS (answer preflights early) → rate limit (shed abuse before work) → body parser with limits (needed by routes; after limiter so floods don't parse) → auth (scoped to protected paths) → routers → 404 catch-all → error handler last. The reasoning IS the answer: cost and dependency order.

Q2: How does Express know a middleware is an error handler, and how do errors reach it?

Arity: functions registered with four parameters are error middleware. Sync throws are caught by Express; async rejections reach it via next(err) (auto in v5, wrapper in v4). next(err) SKIPS all normal middleware to the next error handler — the pipeline's exception channel, mirroring the JS errors lesson's catch-at-the-boundary.

Q3: Middleware vs decorators/filters in other stacks — what's the transferable idea?

Chain-of-responsibility around a request: compose cross-cutting concerns (auth, logging, validation) without touching handlers. Fastify hooks, Koa's onion model (await next() lets you run code on the way OUT — response-time measurement done right), ASP.NET middleware, Django middleware — same pattern, different syntax. Learning the model transfers to any framework interview.

🏋️ Practice: build the full stack: request-id logger, auth stub reading Bearer tokens, feature router with loadUser, wrap() for async, and the typed error handler. Break each mistake in the table deliberately (hang a request, reorder the parser) and observe the failure mode.

🎯 Key Takeaways

  • Everything is (req, res, next) in registration order; respond OR next() — never neither (hang) nor both (headers sent).
  • Scope middleware to paths/routes; decorate req (id, user) for downstream.
  • Async rejections must reach next(err) — Express 5 auto, v4 wrapper.
  • One 4-arg error boundary, last, mapping the error taxonomy to status codes.
  • The pipeline is chain-of-responsibility — the same architecture in every server framework.
