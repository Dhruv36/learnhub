Testing APIs Advanced

The same testing-trophy philosophy from React, server-side: few unit tests for pure logic, MANY integration tests hitting real routes, few E2E. The high-value layer is the HTTP integration test — it exercises routing, middleware, validation, and error mapping together, exactly as clients do.

The Pyramid, Server Edition

LayerToolTests
UnitVitest / node:testPure functions: validators, pricing, mappers — exhaustive, fast
Integration ★Vitest + supertestReal routes through the whole middleware stack against a real (test) DB
ContractPact / schema checksAPI shape matches what consumers expect (microservices)
E2E / loadPlaywright / k6Critical flows; performance under load

Integration Tests with supertest

import request from "supertest";
import { app } from "../app.js";                    // export app WITHOUT listen() — testable

test("POST /api/users validates and creates", async () => {
  await request(app).post("/api/users").send({ email: "bad" })
    .expect(400).expect(r => expect(r.body.code).toBe("VALIDATION"));

  const res = await request(app).post("/api/users")
    .send({ email: "a@b.com", name: "Asha" }).expect(201);
  expect(res.body.id).toBeDefined();
  expect(res.headers.location).toMatch(/\/api\/users\//);
});
// One test exercises: routing, json parser, zod middleware, handler, DB, error mapping,
// status + headers — the integration payoff. Structure app/server split so app has no
// side effects on import (also good production hygiene).

The Test Database Question (the real work)

// Tests must hit a REAL database (mocking the DB tests your mock, not your SQL). Options:
// 1. Testcontainers — spin up a throwaway Postgres in Docker per suite (highest fidelity)
// 2. A dedicated test DB, migrated fresh, TRUNCATE between tests (fast, common)
// 3. Each test in a transaction that ROLLS BACK at teardown (fastest isolation)
beforeEach(async () => { await db.query("BEGIN"); });
afterEach(async () => { await db.query("ROLLBACK"); });   // pristine state, no cleanup code
// Seed deterministic fixtures; never share mutable state between tests (the arrays-lesson
// mutation trap bites test suites too).

Mock the EDGE, Not Your Own Code

// Third-party HTTP (Stripe, email) → intercept at the network boundary (nock / MSW node),
// like the React testing lesson — your request-building and error handling run for real:
nock("https://api.stripe.com").post("/v1/charges").reply(402, { error: "card_declined" });
await request(app).post("/api/pay").send(order).expect(402);
// Clock/random: inject them or use fake timers (vi.useFakeTimers) so tests are deterministic.
// DON'T mock your own repository/service layer in integration tests — that's what you're testing.

Interview Questions

Q1: Why hit a real database in integration tests instead of mocking it?

Mocking the DB tests your assumptions about the DB, not the DB — real SQL bugs (constraint violations, wrong joins, N+1, transaction behavior, type coercion) slip through. A throwaway Postgres (Testcontainers) or a per-test transaction-rollback gives real fidelity with isolation. You mock the EDGES you don't own (payment providers), not the datastore your correctness depends on.

Q2: Design the test suite for a payments endpoint.

Unit: amount/currency validation edge cases. Integration (supertest + real DB + nock'd Stripe): happy path (201, charge recorded, correct request to Stripe), declined card (Stripe 402 → 402), idempotency key replay (same key → same result, one charge), authz (can't pay another user's order → 404), validation failures (400). E2E: one real sandbox checkout. The ratio and the failure-branch coverage are what's graded.

Q3: How do you keep integration tests fast AND isolated?

Transaction-per-test with rollback (no cleanup code, pristine state, milliseconds), a single migrated schema reused across tests, connection pooling, and running suites in parallel with separate schemas/databases. Testcontainers for fidelity when transaction rollback can't model the scenario (e.g., testing commit behavior itself). Determinism via injected clock/uuid so nothing flakes.

🏋️ Practice: split app/server; write supertest integration tests for your users CRUD covering 201/400/404; add the transaction-rollback beforeEach/afterEach against a test DB; nock a fake external API and test both success and its error branch; prove a BOLA test (user A can't read user B's resource).

🎯 Key Takeaways

  • Trophy shape: few unit (pure logic), many integration (supertest through the real stack), few E2E.
  • Export app without listen(); supertest drives real routes + middleware + validation + errors.
  • Use a real test DB (containers or transaction rollback); mocking the DB hides SQL bugs.
  • Mock third-party HTTP at the network edge; inject clock/random for determinism.
  • Cover failure branches — validation, authz (BOLA!), error mapping — not line percentage.
