HTTP & Express Basics Basics

Build the raw server once — then you know exactly what Express adds (routing, parsing, middleware) and what it doesn't (everything else). HTTP literacy — methods, status codes, headers — is the real content of this lesson; frameworks come and go, the protocol doesn't.

The Raw Server (know what's underneath)

import { createServer } from "node:http";
createServer((req, res) => {
  // req: method, url, headers, and a READABLE STREAM of the body
  // res: a WRITABLE STREAM you set status/headers on, then end
  if (req.method === "GET" && req.url === "/health") {
    res.writeHead(200, { "content-type": "application/json" });
    res.end(JSON.stringify({ ok: true }));
  } else { res.writeHead(404); res.end(); }
}).listen(3000);
// Everything a framework does — routing, body parsing, cookies — is sugar over this.

Express — the Minimal Framework

import express from "express";
const app = express();
app.use(express.json());                          // parse JSON bodies → req.body

app.get("/api/users/:id", async (req, res) => {
  const user = await repo.find(Number(req.params.id));   // params are STRINGS
  if (!user) return res.status(404).json({ code: "NOT_FOUND" });
  res.json(user);                                  // sets content-type, serializes, ends
});
app.post("/api/users", async (req, res) => {
  const user = await repo.create(req.body);        // validated in the REST lesson!
  res.status(201).location(`/api/users/${user.id}`).json(user);
});

const server = app.listen(process.env.PORT ?? 3000);
// req.params (:id) · req.query (?page=2 — strings!) · req.body (needs a parser) · req.headers

The 401 vs 403 interview classic: 401 = "we don't know who you are" (missing/bad credentials — retry WITH auth); 403 = "we know you, and no" (authenticated but unauthorized).

Interview Questions

Q1: Why does idempotency matter practically, and how do you make POST safe to retry?

Networks fail AFTER the server processed but BEFORE the client heard — clients retry, and non-idempotent POSTs duplicate (double charge!). Fix: idempotency keys — client sends a unique key header; server stores key→result and replays the stored response on retry. PUT/DELETE are naturally retry-safe; that's why the distinction exists.

Q2: What does CORS actually protect, and from what?

It's a BROWSER mechanism protecting USERS: it stops evil.com's JS from reading responses made with a victim's ambient credentials to your API. It does NOT protect your server (curl/Postman skip it). Preflights (OPTIONS) ask permission for non-simple requests; the server's headers grant it. Real API protection = auth, rate limiting, validation.

Q3: Walk a request from socket to res.json in Express.

libuv accepts the connection → http parses request line/headers, streams the body → Express walks its middleware stack in registration order (json parser buffers+parses body onto req.body) → first matching route handler runs → res.json sets content-type, serializes, writes, ends. Knowing it's "an ordered array of functions around a stream pair" demystifies the middleware lesson.

🏋️ Practice: build the raw-http health server, then the Express users CRUD (in-memory Map). Exercise every status: 201+Location, 204 delete, 404, 400 on bad id. Test with curl -i; break it: forget a return after a guard and read the crash.

🎯 Key Takeaways

  • Express = routing + parsing + middleware over node:http streams — nothing magical.
  • Methods have semantics: GET safe, PUT/DELETE idempotent, POST needs idempotency keys for retries.
  • Status codes are the API's contract: 2xx/4xx-whose-fault/5xx-our-fault; 401 vs 403 precisely.
  • Everything from the wire is a string — parse and validate; limit body sizes.
  • CORS protects browser users, not your API; return after early responses.
