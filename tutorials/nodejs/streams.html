Streams & Backpressure Async Node

What you'll master: processing unbounded data in constant memory — the four stream types, backpressure (the concept that separates people who use streams from people who understand them), pipeline as the only safe way to connect streams, Transform streams, and async iteration as the modern consumption idiom. Prerequisites: EventEmitter (streams are emitters), Buffers.

Part 1 — Why Streams: the Memory Argument

// Serve a 4GB file, two ways:
// ❌ buffer everything:
const data = await fs.readFile("video.mp4");   // 4GB in the heap PER REQUEST —
res.end(data);                                  // 10 concurrent viewers = OOM

// ✅ stream it:
import { createReadStream } from "node:fs";
import { pipeline } from "node:stream/promises";
await pipeline(createReadStream("video.mp4"), res);   // ~64KB chunks in flight —
                                                       // memory is CONSTANT in file size
// A stream is a sequence of chunks over time instead of one value in memory.
// Everything large or unbounded is stream-shaped: request/response bodies,
// files, sockets, compression, CSV exports, logs. You've been using them all
// along — req and res ARE streams; this lesson makes you deliberate about it.
//
// The four types:
// Readable   — a source (file read, req, process.stdout of a child)
// Writable   — a sink (file write, res, socket)
// Duplex     — both, independent directions (a socket)
// Transform  — Duplex where output = f(input) (gzip, encryption, parsers)

Part 2 — Backpressure: the Whole Point

// Fast source, slow sink — a 500MB/s disk read piped to a 5MB/s phone client.
// Where do the other 495MB/s go each second? Without flow control: into MEMORY.
//
// The mechanics (what "handling backpressure" concretely means):
writable.write(chunk)      // returns FALSE when its internal buffer passed
                           // highWaterMark (default 64KB bytes / 16 objects)
// false = "stop sending" — a REQUEST, not enforcement. Then:
writable.once("drain", resume)   // buffer flushed below the mark → resume

// Manual loop (know it; rarely write it):
function writeAll(ws, chunks) {
  for (const chunk of chunks) {
    if (!ws.write(chunk)) {
      await once(ws, "drain");        // PAUSE the producer until the sink catches up
    }
  }
}
// Ignore the false and nothing crashes today — the buffer just grows. That's
// the insidious part: it "works" in dev (fast localhost sink) and OOMs in prod
// (slow real clients). Backpressure bugs are load-shaped.
//
// pipe()/pipeline() implement this pause/resume dance for you: the readable is
// paused whenever the writable says false, resumed on drain — end to end
// through every stage. THAT propagation is what "streams compose" means.
The one-sentence definition to give in interviews: backpressure is the mechanism by which a slow consumer slows down a fast producer — signaled by write() returning false and the drain event, propagated automatically by pipeline — so throughput degrades to the slowest stage instead of memory absorbing the difference.

Part 3 — pipeline: the Only Safe Way to Connect Streams

import { pipeline } from "node:stream/promises";
import { createGzip } from "node:zlib";

// Compress a file — three stages, one call:
await pipeline(
  createReadStream("access.log"),
  createGzip(),
  createWriteStream("access.log.gz"),
);   // resolves on success, REJECTS on any stage's failure

// Why never .pipe() in production code:
readable.pipe(gzip).pipe(writable);
// 1. ERRORS DON'T PROPAGATE: an error in `readable` doesn't destroy gzip or
//    writable — and unhandled stream errors crash (Events lesson) or leak.
// 2. NO CLEANUP: a failed destination leaves the source open — fd leaks,
//    half-written files.
// pipeline() wires error forwarding + destroys ALL stages on any failure +
// gives you a promise. It also accepts async generators as stages (Part 5) —
// .pipe() is for REPL demos; pipeline is for code.
//
// HTTP file serving, done right:
app.get("/report.csv", async (req, res) => {
  res.setHeader("Content-Type", "text/csv");
  try {
    await pipeline(buildReportStream(), res);
  } catch (err) {
    if (err.code === "ERR_STREAM_PREMATURE_CLOSE") return;  // client bailed — normal
    log.error({ err }, "report failed");
    if (!res.headersSent) res.status(500).end();
  }
});
// Client-disconnect = premature close = NOT an alarm. Distinguish it (the
// AbortError of streams) or your error dashboards drown in user behavior.

Part 4 — Transform Streams: Processing in the Pipe

import { Transform } from "node:stream";

// A CSV-line filter: keep rows where status === "error" — over ANY size file:
function errorRowsOnly() {
  let carry = "";                                   // ⭐ chunks ≠ lines: a chunk
  return new Transform({                            //   ends mid-line; carry the tail
    transform(chunk, encoding, callback) {
      const lines = (carry + chunk).split("\n");
      carry = lines.pop();                          // last piece may be incomplete
      const out = lines.filter(l => l.includes(",error,")).join("\n");
      callback(null, out ? out + "\n" : "");        // push downstream (or nothing)
    },
    flush(callback) {                               // end of input: emit the carry
      callback(null, carry.includes(",error,") ? carry + "\n" : "");
    },
  });
}

await pipeline(
  createReadStream("huge.csv"),
  errorRowsOnly(),
  createWriteStream("errors.csv"),
);
// The carry pattern is THE transform skill: chunk boundaries are arbitrary
// (mid-line, mid-multibyte-char — Buffers lesson), so stateful reassembly is
// yours. Same shape powers JSON-lines parsers, protocol framers, and every
// "split by delimiter" stream. objectMode: true switches the currency from
// bytes to JS objects (rows, messages) — parse once, then object stages.

Part 5 — Async Iteration: the Modern Consumption Idiom

// Readables are async iterables — for await IS backpressure-aware consumption:
for await (const chunk of createReadStream("data.bin")) {
  await process(chunk);        // the stream PAUSES while you await — flow
}                               // control by construction, no drain bookkeeping

// Async generators as pipeline stages — transforms without the class:
async function* toUpper(source) {
  for await (const chunk of source) yield chunk.toString().toUpperCase();
}
await pipeline(process.stdin, toUpper, process.stdout);

// And generators as SOURCES — stream from anything (DB cursors, APIs):
async function* allUsers(db) {
  let page = 0, rows;
  do {
    rows = await db.users.findPage(page++, 1000);   // paginate under the hood
    yield* rows;                                     // emit row by row
  } while (rows.length > 0);
}
await pipeline(
  Readable.from(allUsers(db)),
  toCsvRow(),                     // objectMode transform
  createGzip(),
  createWriteStream("users.csv.gz"),
);
// A million-row export at constant memory, DB paced by the gzip+disk speed —
// the pattern behind every serious export/ETL endpoint. Web Streams
// (ReadableStream — fetch bodies) interop via Readable.fromWeb/toWeb when the
// two stream worlds meet.

Part 6 — Operational Realities (Senior Layer)

// 1. highWaterMark is a TUNING knob, not a limit: per-stage buffer target.
//    Bigger = more throughput (fewer pauses), more memory per stream. 64KB
//    default is right until measured otherwise; 1000 concurrent streams ×
//    1MB HWM = 1GB just in buffers.
// 2. objectMode HWM counts OBJECTS (16 default) — 16 giant rows can be more
//    memory than 64KB of bytes; size it by object weight.
// 3. Destroy on abort: wire AbortSignal — pipeline(src, dst, { signal }) —
//    so client timeouts release fds and DB cursors (the L7 cancellation rule,
//    stream-shaped).
// 4. Partial output on failure: a failed pipeline leaves a half-written
//    destination file — write to temp + rename on success (fs lesson's
//    atomicity) for anything consumers might read.
// 5. Metrics: bytes through, stage durations, premature-close rate. A rising
//    premature-close rate is a UX/latency signal (users abandoning downloads),
//    not an error signal.
// 6. Don't stream the small stuff: JSON APIs returning 2KB don't need streams —
//    res.json() is correct. Streams earn their complexity from SIZE (big),
//    CARDINALITY (many rows), or LATENCY (start sending before you have it all).

Interview Questions

Beginner: What problem do streams solve, and name the four types.

Processing data larger than memory (or arriving over time) in constant space: chunks flow through stages instead of materializing whole payloads. Types: Readable (sources), Writable (sinks), Duplex (both directions, independent — sockets), Transform (Duplex where output derives from input — gzip, parsers). HTTP req/res, files, and sockets are all streams already.

Beginner: What is backpressure?

The flow-control mechanism matching a fast producer to a slow consumer: write() returns false when the sink's buffer exceeds its highWaterMark, the producer pauses, and the drain event signals resume. pipeline/pipe propagate this pause/resume through every stage, so a slow final sink throttles the original source — without it, the speed difference accumulates as memory.

Mid: Why is pipeline() preferred over chaining .pipe()?

pipe() only forwards data; errors don't propagate across stages, so a mid-chain failure leaves other stages running — leaked file descriptors, hung responses, and unhandled 'error' events that crash the process. pipeline() forwards errors, destroys every stage on any failure, resolves/rejects as a promise (composing with async/await and try/catch), accepts an AbortSignal, and takes async generators as stages. It's the difference between plumbing and plumbing with shutoff valves.

Mid: Design a transform that parses newline-delimited JSON from a socket. What's the subtle requirement?

Chunk boundaries are arbitrary — a chunk usually ends mid-line (and can even split a multi-byte character). The transform must keep a carry buffer: append each chunk, split on newlines, hold back the final partial segment, JSON.parse the complete lines (emitting objects in objectMode), and flush the carry at end-of-stream. Error policy per line (skip-and-count vs fail-the-stream) is a stated design decision. This reassembly-across-chunks requirement is the essence of every framing/parsing stream.

Senior: A CSV-export endpoint works in staging but OOMs production pods during big exports. The code: query all rows, map to CSV strings, res.send(joined). Redesign and name each principle.

Three materializations of the full dataset (rows array, string array, joined string) × concurrent exports = memory scaling with data × users. Redesign as a pipeline: a paginated async-generator source (DB cursor — thousands of rows resident, not millions), an objectMode CSV transform, gzip, into res — with pipeline propagating backpressure so a slow client download paces the DB reads (principle: throughput degrades to the slowest stage; memory stays flat). Wire the request's abort signal so abandoned exports release the cursor (cancellation). Distinguish premature close from failure in error handling. Set Content-Disposition and start streaming immediately — time-to-first-byte drops from "after the query" to milliseconds (latency principle: send before you have it all). Staging never caught it because staging data was small — size-dependent bugs need production-scale fixtures in tests.

Senior: When do you reach for objectMode streams versus just async iteration with a for-await loop, versus arrays?

Arrays: bounded, small data — simplest, correct default. Plain for-await over an async generator: sequential processing with natural pacing — right for most row-by-row jobs, no stream machinery needed. objectMode pipelines earn their ceremony when you need stream composition: multiple reusable stages (parse → validate → enrich → serialize) shared across jobs, fan-in to byte streams (gzip, sockets, files) in the same pipeline, buffering tuned per stage (highWaterMark as a decoupling buffer between bursty producer and steady consumer), or integration with stream-speaking libraries. The gradient is ceremony versus composability — choose the least machinery that gives the composition you actually use.

3 (medium): Build a lineNumbers() transform that prefixes each line with "N: " — correct across chunk boundaries — and use it in a pipeline from stdin to stdout.
Solution 
import { Transform } from "node:stream";

function lineNumbers() {
  let carry = "", n = 0;
  return new Transform({
    transform(chunk, enc, cb) {
      const lines = (carry + chunk).split("\n");
      carry = lines.pop();
      cb(null, lines.map(l => `${++n}: ${l}\n`).join(""));
    },
    flush(cb) {
      cb(null, carry ? `${++n}: ${carry}\n` : "");
    },
  });
}
await pipeline(process.stdin, lineNumbers(), process.stdout);
// $ cat app.js | node numberer.js — verify a file larger than 64KB numbers
// correctly across the chunk seams (that's the carry doing its job).
4 (medium): Add a byte-limit guard to an upload pipeline: a transform that counts bytes and destroys the stream with a 413-worthy error past 10MB. Why must this be a stream, not an after-the-fact check?
Solution 
function byteLimit(max) {
  let seen = 0;
  return new Transform({
    transform(chunk, enc, cb) {
      seen += chunk.length;
      if (seen > max) {
        const err = new Error("Payload too large");
        err.status = 413;
        return cb(err);                     // pipeline destroys ALL stages —
      }                                     // including the incoming request
      cb(null, chunk);
    },
  });
}
try {
  await pipeline(req, byteLimit(10 * 1024 * 1024), createWriteStream(tmpPath));
} catch (err) {
  if (err.status === 413) return res.status(413).end();
  throw err;
}
// Why in-stream: an after-the-fact check has already RECEIVED (and buffered or
// written) the oversized body — the attacker's 50GB still cost you disk and
// bandwidth. Mid-stream destruction stops reading at ~10MB; the socket closes.
// (Content-Length pre-checks help but lie easily — chunked encoding has none.)
5 (medium): Using an async generator source, stream the numbers 1..1,000,000 as JSON-lines over HTTP, and demonstrate (with logging) that a paused client pauses the generator.
Solution 
import { Readable } from "node:stream";

async function* numbers(n) {
  for (let i = 1; i <= n; i++) {
    if (i % 100_000 === 0) console.log(`generated ${i}`);   // the probe
    yield JSON.stringify({ n: i }) + "\n";
  }
}

http.createServer(async (req, res) => {
  res.setHeader("Content-Type", "application/x-ndjson");
  await pipeline(Readable.from(numbers(1_000_000)), res).catch(() => {});
}).listen(3000);

// $ curl -s localhost:3000 | pv -L 100k > /dev/null     (rate-limited client)
// Server logs show "generated" lines arriving SLOWLY, paced by the 100KB/s
// client — the generator suspends at `yield` whenever res says false. That
// pause-at-yield is backpressure reaching all the way into your loop, free.
6 (hard): Build the full production export endpoint: DB-cursor generator → objectMode CSV transform (with header row) → gzip → response, with abort wiring, premature-close handling, and correct headers. Annotate each defense.
Solution 
app.get("/export/users.csv.gz", async (req, res) => {
  res.setHeader("Content-Type", "application/gzip");
  res.setHeader("Content-Disposition", 'attachment; filename="users.csv.gz"');

  const ac = new AbortController();
  req.on("close", () => ac.abort());                 // ① client gone → stop the DB

  async function* rows() {
    const cursor = db.users.cursor({ batchSize: 1000 });  // ② constant DB memory
    try {
      for await (const row of cursor) yield row;
    } finally { await cursor.close(); }               // ③ cursor released on ANY exit
  }

  function toCsv() {
    let first = true;
    return new Transform({
      objectMode: true,                               // ④ rows in, bytes out
      transform(row, _e, cb) {
        const head = first ? "id,name,email\n" : "";
        first = false;
        cb(null, head + `${row.id},${csvEscape(row.name)},${row.email}\n`);
      },
    });
  }

  try {
    await pipeline(Readable.from(rows()), toCsv(), createGzip(), res,
                   { signal: ac.signal });            // ⑤ abort tears down all stages
  } catch (err) {
    if (err.name === "AbortError" ||
        err.code === "ERR_STREAM_PREMATURE_CLOSE") {
      metrics.increment("export.abandoned");          // ⑥ user behavior ≠ error
      return;
    }
    log.error({ err }, "export failed");              // ⑦ real failures logged;
    if (!res.headersSent) res.status(500).end();      //    headers may be long sent —
  }                                                   //    mid-stream failure truncates
});                                                   //    (gzip makes truncation detectable)
// Memory: ~batchSize rows + stream buffers, whether 1K or 100M users.
// First byte: after the first batch — seconds before a buffered version.
7 (hard): A log-shipping agent tails a file and POSTs each line to an ingest API. Under log bursts it OOMs; the dev "fixed" it by dropping to one POST at a time, and now it falls behind forever. Design the correct middle using this section's tools, and state the unavoidable trade-off.
Solution 
// The two failures are the two ends of one dial:
// unbounded concurrency (OOM) ←——————→ concurrency 1 (can't keep up).
// The middle: batching + bounded concurrency + bounded buffering + shedding.
async function shipLogs({ signal }) {
  const BATCH = 500, FLUSH_MS = 2000, MAX_INFLIGHT = 4, MAX_BUFFER = 50_000;
  const limit = pLimit(MAX_INFLIGHT);              // L7: bounded fan-out
  let batch = [];

  async function flush() {
    if (batch.length === 0) return;
    const payload = batch; batch = [];
    await limit(() => retry(                        // L7: jittered retries,
      ({ signal: s }) => postLines(payload, { signal: s }),
      { attempts: 3, signal }));
  }
  const timer = setInterval(() => flush().catch(logShipError), FLUSH_MS);
  timer.unref();

  for await (const line of tailFile("app.log", { signal })) {  // stream source:
    batch.push(line);                               // backpressure vs the FILE
    if (batch.length >= BATCH) await flush();       // ← await = the agent READS
                                                    //   no faster than it ships
    if (batch.length > MAX_BUFFER)                  // dead ingest API:
      batch.splice(0, batch.length - MAX_BUFFER);   //   shed oldest, bounded memory
  }
  clearInterval(timer); await flush();              // drain on shutdown (L5)
}
// The unavoidable trade-off, stated: when the source outruns the sink
// LONG-TERM, you choose exactly one — grow memory (OOM), block the source
// (fall behind / backpressure the app), or DROP data (shedding). There is no
// fourth option; production log shippers all pick "bounded buffer + shed +
// count what you dropped" because logs are fail-soft. Payments would pick
// blocking + durable queue instead — the choice is per-domain, and making it
// explicitly is the design.

🎯 Key Takeaways

  • Streams process unbounded data in constant memory; req/res/files/sockets already are streams — Readable, Writable, Duplex, Transform.
  • Backpressure: write() → false → pause → drain → resume, propagated end-to-end by pipeline — without it, speed mismatches become memory, and only under load.
  • stream/promises pipeline is the only production way to connect stages: error propagation, universal teardown, promises, signals; .pipe() is a demo tool.
  • Transforms own chunk-boundary reassembly (the carry/flush pattern); objectMode moves the currency from bytes to rows; premature close is user behavior, not failure.
  • Async iteration and generators are streams' modern face: for-await consumes with built-in pacing, generators are sources and stages, pause-at-yield is backpressure reaching your code.
  • Streams earn complexity from big/many/early data — and long-term source>sink mismatches force an explicit choice: block, buffer-bounded-and-shed, or OOM.
