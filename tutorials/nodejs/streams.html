Streams & Backpressure Advanced

Streams process data in chunks instead of loading it all into memory — the difference between serving a 2GB file on a 512MB server and crashing. Node's HTTP req/res, files, sockets, and compression are all streams. The killer concept is backpressure: how a fast producer avoids drowning a slow consumer.

Why Streams — the Memory Argument

// ❌ Loads the WHOLE file into RAM before sending — 2GB file = 2GB spike, OOM under concurrency:
app.get("/download", async (req, res) => res.send(await readFile("huge.zip")));

// ✅ Stream it — constant memory (~64KB chunks), starts sending immediately:
import { pipeline } from "node:stream/promises";
app.get("/download", (req, res) =>
  pipeline(createReadStream("huge.zip"), res).catch(() => res.destroy()));
// pipeline = the safe connector: wires source→dest, propagates errors, cleans up on failure.

await pipeline(

Backpressure — the Core Idea

// write() returns FALSE when the destination's buffer is full:
const ok = writable.write(chunk);
if (!ok) readable.pause();                        // stop reading…
writable.once("drain", () => readable.resume());  // …resume when the buffer empties

// Manual handling is error-prone → ALWAYS use pipeline()/pipe(), which do this for you.
// WITHOUT backpressure (e.g. a for-loop of writes ignoring the return value): a fast reader +
// slow writer buffers unbounded IN MEMORY → the exact OOM streams were meant to prevent.
// Backpressure is just flow control: the slow end sets the pace.

Async Iteration — Streams as for-await (the JS generators lesson)

// Readable streams are async-iterable — clean consumption with natural backpressure:
for await (const chunk of createReadStream("app.log", { encoding: "utf8" })) {
  process(chunk);                       // awaiting here pauses the stream automatically
}
// Line-by-line for huge files without loading them:
import { createInterface } from "node:readline";
for await (const line of createInterface({ input: createReadStream("huge.log") })) {
  if (line.includes("ERROR")) count++;
}
// Web Streams (ReadableStream/TransformStream) are the cross-platform standard now also in Node —
// same concepts, used by fetch() bodies.

Interview Questions

Q1: What is backpressure and what happens without it?

Flow control: when the writable's internal buffer fills, write() returns false, signaling the readable to pause until a 'drain' event. Without it, a fast source keeps pushing into a slow sink's buffer, which grows unbounded in memory — turning a "streaming" download into the OOM crash streaming was supposed to prevent. pipeline/pipe implement it correctly; manual write loops usually don't.

Q2: Why stream a 2GB file instead of readFile + send?

readFile buffers all 2GB in RAM before the first byte ships — multiply by concurrent downloads and you OOM; TTFB is also terrible (wait for full read). Streaming sends ~64KB chunks with constant memory, starts immediately, and backpressure matches send speed to the client's download speed. It's memory, latency, and concurrency all at once.

Q3: Design a CSV import endpoint for 500MB uploads.

Stream the request body (it IS a readable) → a CSV parser Transform → batch rows → bulk-insert per batch (DB lesson), all via pipeline for backpressure/errors. Never buffer the whole file. Add: per-batch transactions, progress via chunks processed, statement timeouts, and destroy on client abort. Constant memory regardless of file size — the streaming payoff.

🏋️ Practice: build a download endpoint two ways (readFile vs stream) and watch process.memoryUsage() on a big file; build the read→gzip→write pipeline; process a huge log line-by-line with readline counting matches; break backpressure with a raw write-loop and observe memory climb.

🎯 Key Takeaways

  • Streams process data in chunks → constant memory for arbitrarily large data.
  • Four types: Readable, Writable, Transform, Duplex — composed with pipeline().
  • Backpressure (write() → false → pause → drain → resume) prevents fast-producer OOM; pipeline handles it.
  • Readable streams are async-iterable (for await) — the generators lesson, server-side.
  • Always pipeline (errors + cleanup); destroy sources on client abort; never buffer huge inputs.
