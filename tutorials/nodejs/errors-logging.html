Error Handling & Logging Intermediate

The JS errors lesson taught the taxonomy; this applies it to a server that must stay up under failure and be debuggable at 3am. Two pillars: a single error boundary that maps failures to responses, and structured logs that a machine can search.

The Error Taxonomy on the Server

class AppError extends Error {
  constructor(msg, { code, status = 500, cause, meta } = {}) {
    super(msg, { cause });
    this.name = new.target.name; this.code = code; this.status = status; this.meta = meta;
    this.isOperational = true;                      // KNOWN failure vs a bug (see below)
  }
}
class ValidationError extends AppError { constructor(m, meta){ super(m,{code:"VALIDATION",status:400,meta}); } }
class NotFoundError   extends AppError { constructor(m){ super(m,{code:"NOT_FOUND",status:404}); } }
class ConflictError   extends AppError { constructor(m){ super(m,{code:"CONFLICT",status:409}); } }
// Handlers just THROW; the boundary maps type → HTTP.

Operational vs Programmer Errors — the Distinction That Decides Crash-or-Continue

Operational (expected)Programmer (bug)
Bad input, 404, DB timeout, upstream 503, rate limitundefined is not a function, bad assumption, null deref
Handle per-request → 4xx/5xx response, log, continue servingState may be corrupt → log, alert, let the process crash + restart (supervised)

This is why isOperational exists: the top-level handler answers known failures and keeps serving; truly-unknown errors and unhandled rejections mean "we don't know what state we're in" → graceful crash beats limping on corrupt state.

The Express Error Boundary (middleware lesson, completed)

app.use((err, req, res, next) => {
  const status = err.status ?? 500;
  if (status >= 500) req.log.error({ err, reqId: req.id });   // 5xx = our fault → loud + alert
  else               req.log.warn({ code: err.code, reqId: req.id });  // 4xx = client → quiet
  res.status(status).json({ code: err.code ?? "INTERNAL",
                            message: status < 500 ? err.message : "Internal error",  // no leaks
                            reqId: req.id });                  // ties response ↔ logs
});

Structured Logging — JSON, Not console.log

import pino from "pino";                            // fast, JSON, the Node standard
const logger = pino({ level: process.env.LOG_LEVEL ?? "info" });

// A child logger per request carries context automatically:
app.use((req, res, next) => { req.id = crypto.randomUUID();
  req.log = logger.child({ reqId: req.id, path: req.path }); next(); });

req.log.info({ userId, orderId }, "order placed");
// → {"level":30,"time":…,"reqId":"…","userId":…,"orderId":…,"msg":"order placed"}
// Why JSON: log platforms (Datadog, Loki, CloudWatch) INDEX fields — you query
// reqId:abc123 or level:error AND path:/checkout. console.log strings can't be queried.
// NEVER log secrets/PII/tokens (pino redact paths strip them). Levels: trace/debug/info/warn/error/fatal.

Process-Level Safety Nets

process.on("unhandledRejection", (reason) => { logger.fatal({ reason }, "unhandled rejection"); shutdown(1); });
process.on("uncaughtException",  (err)    => { logger.fatal({ err }, "uncaught exception");   shutdown(1); });
// These are LAST RESORT: log the evidence, then exit and let the orchestrator restart a
// clean process. Continuing after an uncaught exception runs on undefined state — worse.
// The three observability pillars (system design track): LOGS (events, here) +
// METRICS (rates/latencies — Prometheus) + TRACES (request path across services).

Interview Questions

Q1: Operational vs programmer errors — and how it drives your crash policy.

Operational = anticipated runtime failures (bad input, timeouts, 404s) handled per-request with a response and a log — the server keeps serving. Programmer errors = bugs indicating broken assumptions; after logging, crash and restart because continuing runs on possibly-corrupt state (leaked connections, half-written data). "Let it crash" is deliberate, paired with a supervisor (PM2/K8s) — not negligence.

Q2: Why structured JSON logging over formatted strings?

Log platforms index fields, so you filter/aggregate: level:error AND path:/checkout in the last hour, grouped by code. String logs force fragile regex parsing. JSON also carries consistent context (reqId, userId) via child loggers, enabling request correlation and alerting on field conditions. It's the difference between grepping and querying.

Q3: A request fails intermittently in production. Trace your debugging path.

Get the reqId from the user's error response → query logs for that reqId → follow the correlated lines (which middleware, which downstream, the cause chain from { cause }). If distributed, the trace id spans services. Metrics tell you the RATE (spike? one user?), logs tell you the WHY. This is exactly why reqId-on-every-line and cause-chaining are non-negotiable.

🏋️ Practice: add pino + per-request child logger + the typed error boundary to your API; throw each error type and verify status + log level; log a fake token and then add redaction; register the process handlers and trigger an unhandledRejection to watch graceful exit.

🎯 Key Takeaways

  • Typed error taxonomy → one boundary maps type to status; handlers just throw.
  • Operational errors are handled per-request; programmer errors crash + restart (supervised).
  • pino JSON logs with levels + child loggers = queryable, correlated observability.
  • Never leak stacks to clients or secrets to logs; reqId ties response to logs.
  • Process handlers are last-resort loud-then-exit; logs+metrics+traces are the three pillars.
