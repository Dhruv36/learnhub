Auth: JWT & Sessions Intermediate

Authentication ("who are you") and authorization ("what may you do") are where API mistakes become breaches. This lesson: password storage done right, the sessions-vs-JWT decision honestly, and the authorization bug class that tops real-world vulnerability lists (BOLA/IDOR).

Passwords — Non-Negotiables

import argon2 from "argon2";                       // or bcrypt — NEVER sha256/md5/plaintext
const hash = await argon2.hash(password);           // salted, slow BY DESIGN (GPU-resistant)
const ok = await argon2.verify(user.passwordHash, password);
// Same generic error for wrong-email and wrong-password (no user enumeration).
// Rate-limit login attempts (429). Password reset: single-use, expiring, HASHED token in DB.
// Why slow hashing: a leaked DB of fast hashes cracks at billions/sec on GPUs;
// argon2/bcrypt cost factors make each guess expensive — that's the entire point.

Option A — Server Sessions (cookie-based)

login → create session row {id, userId, expiresAt} (Redis/DB)
      → Set-Cookie: sid=<random>; HttpOnly; Secure; SameSite=Lax
request → cookie arrives automatically → look up session → req.user
logout/compromise → DELETE the row: revocation is INSTANT ✅

// Cookie flags (the JS security lesson, server side):
// HttpOnly = XSS can't read it · Secure = HTTPS only · SameSite=Lax = CSRF largely dead
// Cost: a session-store lookup per request (Redis makes this ~sub-ms).

Option B — JWT (token-based)

const token = jwt.sign({ sub: user.id, role: user.role },
                       process.env.JWT_SECRET, { expiresIn: "15m" });
// Client sends: Authorization: Bearer 
const payload = jwt.verify(token, secret);          // signature + expiry checked — STATELESS ✅

// A JWT is three base64url parts: header.payload.signature
// ⚠ base64 ≠ encryption — the payload is READABLE by anyone. Never put secrets in it.
// The signature only proves it wasn't MODIFIED and was issued by you.
// The fatal flaw: you cannot revoke a JWT before it expires (it's self-contained) —
// hence SHORT lifetimes (15m) + REFRESH tokens:
//   refresh token = long-lived, stored server-side (revocable), HttpOnly cookie,
//   ROTATED on each use (reuse detection = stolen-token alarm) → issues new 15m access tokens.

Sessions vs JWT — the Honest Table

SessionsJWT + refresh
RevocationInstant (delete row)Access token: wait out 15m; refresh: revocable
Per-request costStore lookup (Redis)CPU-only signature verify — no store
Multi-serviceShared store neededAny service with the public key verifies — microservices' favorite
TransportCookies (CSRF: SameSite)Header (no CSRF) — but where does the SPA store it? (XSS risk → HttpOnly refresh cookie pattern)
VerdictMonolith/SSR → sessions. Distributed APIs/mobile → short JWT + rotated refresh. "JWT because modern" is not a reason.

Middleware — Authentication then Authorization

function requireAuth(req, res, next) {
  const token = req.headers.authorization?.replace(/^Bearer /, "");
  if (!token) return res.status(401).json({ code: "UNAUTHENTICATED" });
  try { req.user = jwt.verify(token, process.env.JWT_SECRET); next(); }
  catch { return res.status(401).json({ code: "TOKEN_INVALID" }); }   // expired/tampered
}
const requireRole = (...roles) => (req, res, next) =>
  roles.includes(req.user.role) ? next() : res.status(403).json({ code: "FORBIDDEN" });

app.get("/api/admin/stats", requireAuth, requireRole("admin"), handler);

The #1 Real-World Auth Bug: BOLA / IDOR

// Authenticated ≠ authorized for THIS object:
app.get("/api/orders/:id", requireAuth, async (req, res) => {
  const order = await orders.find(req.params.id);
  res.json(order);                                   // 💥 any logged-in user reads ANY order
});
// ✅ Ownership in the QUERY itself:
const order = await orders.findOne({ id: req.params.id, userId: req.user.sub });
if (!order) return res.status(404).json({ code: "NOT_FOUND" });   // 404, not 403 — don't
                                                                   // confirm existence
// Broken Object Level Authorization = OWASP API risk #1. Every object fetch filters by
// the caller's rights — no exceptions, enforced in code review.

Interview Questions

Q1: Walk the full refresh-token rotation flow and why rotation matters.

Login → access JWT (15m) + refresh token (opaque, hashed in DB, HttpOnly cookie). Access expires → POST /refresh → server verifies the refresh token, REVOKES it, issues a new pair. If a REVOKED refresh token is presented again → theft detected → kill the whole session family. Rotation turns a stolen refresh token from "long-term access" into "one use, then alarm".

Q2: Why can't you 'just log out' a JWT, and what are the partial fixes?

Verification is local (signature + expiry) — no server state consulted, so nothing to delete. Fixes: short expiry (bounds the damage window), a denylist of revoked token ids (jti) checked per request — which quietly reinvents sessions — or versioned claims (tokenVersion column bumped on logout-all). The honest answer: if instant revocation is a hard requirement, sessions were the right tool.

Q3: 401 vs 403 vs 404 in authorization design?

401: not authenticated (missing/expired credentials — client should re-auth). 403: authenticated, insufficient rights (role gates — safe to reveal the resource exists). 404 for ownership failures on per-object access: returning 403 confirms the id EXISTS, enabling enumeration. Security-aware APIs return 404 for "not yours".

🏋️ Practice: build register/login with argon2, 15m JWTs + refresh rotation (in-memory store), requireAuth/requireRole middleware, and an orders endpoint — first with the BOLA bug (prove it with two users via curl), then fixed with ownership queries returning 404.

🎯 Key Takeaways

  • argon2/bcrypt only; generic login errors; rate-limit; hashed single-use reset tokens.
  • Sessions: instant revocation, store lookup; JWT: stateless verify, unrevocable — short access + rotated refresh bridges it.
  • JWT payloads are public; pin algorithms; secrets belong in managers, not git.
  • AuthN then authZ middleware; 401 vs 403 vs enumeration-safe 404.
  • BOLA is the top API bug: ownership filters inside every object query.
