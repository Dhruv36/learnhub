Databases & ORMs Intermediate

How Node talks to Postgres/MySQL: drivers, pools, parameterized queries, transactions, and the ORM decision. Deep SQL lives in the SQL track — this lesson is the Node-side plumbing that production services get right (or leak connections and get injected).

Connection Pools — Why and How

import pg from "pg";
const pool = new pg.Pool({ connectionString: process.env.DATABASE_URL, max: 10 });
const { rows } = await pool.query("SELECT * FROM products WHERE id = $1", [id]);
// A TCP+TLS+auth handshake per request would dwarf the query time. The pool keeps
// N warm connections; queries borrow and return them.
// Sizing reality: Postgres suffers past a few hundred connections TOTAL —
// pool.max × instance-count must respect that (PgBouncer when it can't).
// Exhausted pool + no statement_timeout = every request queues forever → set timeouts.

SQL Injection — the Rule With Zero Exceptions

db.query(`SELECT * FROM users WHERE email = '${email}'`);        // ❌ email = ' OR 1=1 --
pool.query("SELECT * FROM users WHERE email = $1", [email]);      // ✅ parameterized:
// values travel SEPARATELY from SQL text — the driver never lets data become syntax.
// Dynamic structure (sort column) can't be parameterized → ALLOWLIST it:
const col = { price: "price", date: "created_at" }[req.query.sort] ?? "created_at";

Transactions — All or Nothing

const client = await pool.connect();               // ONE connection for the whole tx
try {
  await client.query("BEGIN");
  await client.query("UPDATE accounts SET balance = balance - $1 WHERE id = $2", [amt, from]);
  await client.query("UPDATE accounts SET balance = balance + $1 WHERE id = $2", [amt, to]);
  await client.query("COMMIT");
} catch (e) {
  await client.query("ROLLBACK");
  throw e;
} finally {
  client.release();                                  // ← forget this = pool leak = dead server
}
// Money movement, multi-row invariants, insert-parent-then-children: transaction or bust.
// (Isolation levels & locking: the SQL track's advanced lessons.)

  where: { email },
  include: { orders: { take: 10, orderBy: { createdAt: "desc" } } },

// Migrations: prisma migrate dev — schema changes become versioned SQL files in git.
// Policy that scales: ORM for the 90% CRUD; raw SQL ($queryRaw / builder) for the hot 10% —
// and ALWAYS look at the generated SQL before shipping a new query path.

The N+1 Problem (the ORM's classic trap)

const posts = await prisma.post.findMany();                   // 1 query
for (const p of posts) p.author = await prisma.user.findUnique({ where: { id: p.authorId } });
// ↑ 1 + N queries — 200 posts = 201 round trips. The ORM made the loop LOOK innocent.
// ✅ include/join (1-2 queries):
const posts = await prisma.post.findMany({ include: { author: true } });
// Detection: query logging in dev (log: ['query']) — if a list endpoint logs 50 queries, N+1.
// Same disease in every ORM in every language (Hibernate, ActiveRecord…) — interviews love it.

Interview Questions

Q1: Why exactly do parameterized queries stop injection?

The SQL text (with placeholders) and the values are sent to the database SEPARATELY — typically via prepare/execute protocol messages. The parser builds the query plan from the text alone; values are bound as pure data afterward and can never be re-parsed as syntax. String escaping tries to make data "safe inside syntax"; parameterization removes data from syntax entirely — that's why it has no bypass.

Q2: Your API p99 spikes; DB CPU is low. Pool suspects?

Pool exhaustion: all connections busy/leaked, requests queueing (low DB CPU because queries aren't RUNNING). Check pool wait metrics, look for missing release() in transaction paths, long transactions holding connections (awaiting external APIs inside a tx!), or pool.max too small for concurrency. Fixes: finally-release audits, move slow non-DB work outside transactions, right-size with pool telemetry.

Q3: ORM vs query builder vs raw — your policy and why?

Prisma/Drizzle for CRUD and relations (types from schema, migrations, speed of development); builder/raw for reporting, bulk ops, and hot paths — with query logging in dev and EXPLAIN before shipping. The failure modes to name: N+1 from lazy relation loops, and unseen generated SQL. The policy is about VISIBILITY, not ideology.

🏋️ Practice: wire pg with a pool; demonstrate injection on a template-string endpoint then fix it; write the transfer transaction with a deliberate thrown error proving rollback; create an N+1 endpoint with Prisma logging on, count the queries, fix with include.

🎯 Key Takeaways

  • One pool per process; borrow-and-release; size against the DB's total connection budget; timeouts everywhere.
  • Parameterize values, allowlist identifiers — injection has no other complete answer.
  • Transactions on one client with finally-release; multi-row invariants demand them.
  • ORM for velocity, SQL visibility for safety: log queries, watch for N+1, EXPLAIN hot paths.
  • Migrations are versioned code in the pipeline; the app's DB user is least-privilege.
