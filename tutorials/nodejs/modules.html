Modules: CommonJS & ESM Runtime Foundations

What you'll master: Node's two module systems and how each actually works — CommonJS's runtime require with caching and live wrapper functions, ESM's static import — how Node decides which system a file uses, interop between them, and resolution (how require("express") finds code). Prerequisites: the JS Modules lesson covers ESM syntax; this lesson is Node's side of the story — CJS, the mode rules, and interop.

Part 1 — CommonJS: Modules as Functions Called at Runtime

// logger.js — a CommonJS module:
const level = process.env.LOG_LEVEL ?? "info";        // module-scoped, private

function log(msg)  { console.log(`[${level}] ${msg}`); }
function error(msg){ console.error(`[ERROR] ${msg}`); }

module.exports = { log, error };                      // THE export: one object

// app.js:
const logger = require("./logger");                   // runs logger.js, gets the object
const { log } = require("./logger");                  // destructure if you like
logger.log("started");

// What Node ACTUALLY does with your file — wraps it in a function:
(function (exports, require, module, __filename, __dirname) {
  /* your code here */
});
// That's why require/module/__dirname "just exist" — they're parameters. And
// why top-level variables are private: they're locals of this wrapper function.
// module.exports vs exports: `exports` is a mere alias to module.exports.
//   exports.log = log          ✅ adds a property to the same object
//   exports = { log }          ❌ reassigns the LOCAL PARAMETER — exports nothing
// When in doubt, assign module.exports and never think about it again.
// CACHING — require runs a module ONCE, ever:
// counter.js:  let n = 0; module.exports = { inc: () => ++n, get: () => n };
const a = require("./counter");
const b = require("./counter");
a.inc(); a.inc();
console.log(b.get());        // 2 — a and b are THE SAME OBJECT (require.cache)
// Consequence 1: modules are natural singletons (db pools, config — used constantly).
// Consequence 2: top-level side effects run once, at first require, in
// dependency order — which is why "just requiring" a file can start a server.

Part 2 — ESM in Node: Static, Async, Strict

// logger.mjs (or .js with "type": "module" in package.json):
const level = process.env.LOG_LEVEL ?? "info";
export function log(msg)  { console.log(`[${level}] ${msg}`); }
export default { name: "logger" };

// app.mjs:
import def, { log } from "./logger.mjs";     // NOTE: extension REQUIRED in Node ESM
log("started");

// The deep differences from CJS (not just syntax):
// 1. STATIC: imports are parsed before execution — the whole graph is known
//    up front. Enables: named-export verification at load, tree-shaking in
//    bundlers, and circular-dependency handling via live bindings.
// 2. LIVE BINDINGS: import { n } sees the exporter's CURRENT n; CJS copies
//    the value that existed at require time (JS track → Modules, proven there).
// 3. ASYNC LOADING: which is why top-level await works ONLY in ESM:
const config = await fs.readFile("config.json", "utf8");   // legal at top level
// 4. Always strict mode; and no require/module/__dirname/__filename. Instead:
import.meta.url                              // this file's file:// URL
import.meta.dirname                          // __dirname's replacement (Node 20+)
// 5. Dynamic import — the runtime escape hatch, works in BOTH systems:
const { heavy } = await import("./heavy.js");   // returns a promise of the namespace

Part 3 — Which System Does a File Use? (the Mode Rules)

// Node decides per file, by exactly these rules:
// .mjs  → always ESM          .cjs → always CommonJS
// .js   → whatever the NEAREST package.json's "type" says:
//         "type": "module"    → ESM
//         "type": "commonjs" or absent → CJS (the legacy default)
//
// package.json:
{ "type": "module" }          // ← new projects: set this, write ESM everywhere
//
// The errors that teach you these rules the hard way:
// "require is not defined in ES module scope"  → you're in ESM, wrote CJS
// "Cannot use import statement outside a module" → you're in CJS, wrote ESM
// "__dirname is not defined"                    → ESM: use import.meta.dirname
// Both errors name the fix; read them, don't paste them into a search engine.
The 2026 stance: write ESM for new code — it's the standard, the browser-shared syntax, and where Node's features land (top-level await). CJS is legacy you must READ fluently (a decade of npm, many older codebases) and occasionally write (config files for tools that demand it — .cjs makes the intent explicit). The interop below is how the two worlds coexist in one dependency tree.

Part 4 — Interop: the Rules of the Boundary

// ESM importing CJS — works, with one nuance:
import express from "express";               // ✅ CJS module.exports → default import
import pkg from "./cjs-thing.cjs";
const { helper } = pkg;                       // named destructure OFF the default
// (Node can often detect named exports in CJS via static analysis — when it
// works, `import { Router } from "express"` runs too — but the default-import
// form is the always-safe one.)

// CJS requiring ESM — historically the hard direction:
const esm = require("./thing.mjs");          // ❌ threw ERR_REQUIRE_ESM for years
// Node 22+: require(esm) WORKS if the ESM graph has no top-level await.
// On older Node, the escape hatch is dynamic import from CJS:
(async () => {
  const { helper } = await import("./thing.mjs");   // ✅ works everywhere
})();

// Library authors publish BOTH via conditional exports (package.json):
{
  "exports": {
    ".": {
      "import": "./dist/index.mjs",          // ESM consumers get this
      "require": "./dist/index.cjs"          // CJS consumers get this
    }
  }
}
// The "exports" field ALSO locks the package's internals: consumers can only
// import declared paths — deep imports like "pkg/lib/private.js" break by
// design (the L25-React-lesson "index is the contract" idea, npm-enforced).

Part 5 — Resolution: How require("x") Finds Code

// Three shapes of specifier, three algorithms:
require("./utils")      // RELATIVE: resolve against this file's directory.
                        //   CJS tries: ./utils.js, ./utils.json, ./utils/index.js
                        //   ESM: NO extension guessing — "./utils.js" exactly.
require("node:fs")      // CORE: built-in module. (Bare "fs" works; the node:
                        //   prefix is modern practice — explicit + typo-proof.)
require("express")      // BARE: walk UP the directory tree looking in
                        //   ./node_modules, ../node_modules, ../../…, until
                        //   found or filesystem root — then in that package,
                        //   follow package.json "exports" (or legacy "main").
//
// This upward walk explains real behaviors:
// • Nested node_modules can shadow versions (npm lesson: how trees dedupe).
// • A file OUTSIDE your project can't find your deps (no node_modules above it).
// • Monorepos work: packages/*/ resolve shared deps from the root node_modules.
//
// Your own ergonomics — subpath imports (the # prefix, package.json):
{ "imports": { "#src/*": "./src/*" } }
import { db } from "#src/db.js";             // no more ../../../ ladders — built in,
                                              // no bundler alias config needed.

Part 6 — Module Design in Practice (Senior Layer)

// 1. SIDE-EFFECT-FREE modules: importing should DEFINE, not DO.
//    export function createServer() {...}    ✅ caller decides when
//    createServer().listen(3000)  at top level ❌ importing = deploying
//    (One deliberate exception: an entry file — index.js/main.js — whose JOB
//    is side effects. Keep it thin, calling functions others export.)
// 2. The singleton-by-cache pattern, used deliberately:
//    db.js: export const pool = createPool(env.DB_URL) — every importer shares
//    one pool because modules load once. Great for pools; a trap for anything
//    tests must isolate (module state persists across tests — inject instead).
// 3. Circular imports: Node doesn't crash — it hands one side a PARTIAL module
//    (CJS: whatever exports existed mid-execution; ESM: uninitialized bindings
//    that throw on access). Symptom: "X is undefined" only sometimes / only in
//    prod import order. Fix structurally: extract the shared piece both need
//    into a third module — cycles are a design smell before a runtime bug.
// 4. Entry points: "main"/"exports" for consumers, "bin" for CLIs — the npm
//    lesson's territory; the resolution rules above are what those fields feed.

Interview Questions

Beginner: CommonJS vs ESM — the practical differences?

CJS: require/module.exports, loaded synchronously at runtime, values copied at require time, Node's legacy default. ESM: import/export, parsed statically before execution, live bindings, async loading (hence top-level await), the JS standard shared with browsers. Node picks per file: .mjs/.cjs explicitly, .js by the nearest package.json "type". New code: ESM; reading CJS fluently: mandatory.

Beginner: Where do require, __dirname, and module come from, and why don't they exist in ESM?

CJS files are wrapped in a function — (function (exports, require, module, __filename, __dirname) { … }) — so those are parameters, not globals. ESM is a different loading pipeline with no wrapper; its equivalents are import.meta.url and import.meta.dirname, and importing is syntax, not a passed-in function.

Mid: Explain require's cache and both of its faces.

The first require of a path executes the module and stores module.exports in require.cache; every later require returns that same object. Good face: free singletons — one DB pool, one config object, shared everywhere by import. Bad face: module-level mutable state persists for the process lifetime, leaking across tests and making load order matter; and top-level side effects fire at first-require time, which may be far from where you expect. Design response: modules define, entry points do, and anything tests must isolate gets injected rather than cached.

Mid: How does bare-specifier resolution work, and what does package.json "exports" change?

require("express") walks up from the requiring file checking each directory's node_modules until it finds the package, then consults its package.json. Legacy: the "main" field points at one entry, and any internal file is deep-importable. "exports" replaces that with a declared public surface: a map of allowed subpaths, with conditions ("import" vs "require", "node" vs "default") selecting different files per consumer type. Anything not listed is unreachable — packages gained real encapsulation and dual-format publishing in one field.

Senior: A service intermittently crashes with "cannot read properties of undefined" from a module that's "definitely exported". The stack points at first startup. Diagnose.

Classic circular import. A requires B which requires A: Node returns A's partial exports (whatever was assigned before the cycle re-entered) to B, so B captured undefined for a function A defines later. Intermittency comes from entry-order: whichever module loads first changes who gets the partial view (tests vs prod entry differ). Confirm by tracing the require graph (node --trace-require-module flags or madge). Fix structurally: extract what B needs from A into a third module both import — not by reordering requires (fragile) or lazy-requiring inside functions (hides the design problem, though it's an acceptable tourniquet).

Senior: You maintain a widely-used library in 2026. Lay out your module/publishing strategy.

Author in ESM (or TS compiled to both). Publish dual-format via conditional "exports" — "import" → .mjs, "require" → .cjs, plus "types" per condition for TS — because half the ecosystem still requires. Lock internals with "exports" (only intended subpaths listed), which lets me refactor freely; each breaking surface change is semver-major. Avoid top-level await in the ESM build (it breaks Node 22's require(esm) bridge). No side effects on import — consumers call an init/factory; this also keeps the package tree-shakeable ("sideEffects": false for bundlers). Test the matrix in CI: ESM import, CJS require, TS both modes, oldest-supported LTS — dual-publishing bugs (the "dual package hazard": both copies loaded, instanceof breaking across them) only surface there, so instance-identity-sensitive state lives in a single shared .cjs core if needed.

🏋️ Graded Exercises

1 (easy): Write a mathx module twice — CJS (mathx.cjs) and ESM (mathx.mjs) — exporting add, mul, and a default object with both. Consume each from a matching file.
Solution 
// mathx.cjs
const add = (a, b) => a + b, mul = (a, b) => a * b;
module.exports = { add, mul };
// use-cjs.cjs
const { add } = require("./mathx.cjs");

// mathx.mjs
export const add = (a, b) => a + b;
export const mul = (a, b) => a * b;
export default { add, mul };
// use-esm.mjs
import mathx, { mul } from "./mathx.mjs";   // extension mandatory
2 (easy): Predict what b.js logs and why: a.js is let n = 1; exports.n = n; setTimeout(() => { n = 2; exports.n = 99; }, 10); and b.js does const a = require("./a"); setTimeout(() => console.log(a.n), 50);
Solution 
// 99. Two separate facts:
// • `n = 2` is invisible: exports.n copied the VALUE 1; reassigning the local
//   n never touches the exports object (CJS = value copies, not live bindings).
// • `exports.n = 99` mutates the SHARED exports object b holds — b sees it.
// CJS exports are one shared object; property mutation propagates, variable
// reassignment doesn't. (ESM's live bindings would have shown 2 for `import {n}`.)
3 (medium): A repo has "type": "module" but one tool insists its config be CommonJS. Show the three-file setup that keeps everything working, and the error you'd see without it.
Solution 
// package.json:  { "type": "module" }        → all .js files are ESM
// src/app.js:    import … (ESM, happy)
// tool.config.cjs:                            ← the .cjs extension OVERRIDES
module.exports = { preset: "x" };             //   "type" for this one file
// Without it (tool.config.js with module.exports):
//   ReferenceError: module is not defined in ES module scope
// The extension is the per-file escape hatch in both directions (.mjs in a
// CJS project, .cjs in an ESM project).
4 (medium): Demonstrate the require cache: a config.js that reads a file and logs "loading config" — prove it loads once across three requires, then show how a test could get a fresh copy.
Solution 
// config.cjs
console.log("loading config");
module.exports = JSON.parse(require("node:fs").readFileSync("config.json", "utf8"));

// main.cjs
require("./config.cjs"); require("./config.cjs"); require("./config.cjs");
// → "loading config" printed ONCE.

// Fresh copy in a test (CJS only):
delete require.cache[require.resolve("./config.cjs")];
const fresh = require("./config.cjs");        // logs again — re-executed
// The better design lesson: cache-busting in tests is a smell — export a
// loadConfig() FUNCTION instead, and the test calls it with a fixture path.
// (ESM has no supported cache-delete — one more push toward factories.)
5 (medium): Set up subpath imports so #lib/* maps to ./src/lib/*, use it, and explain what problem it solves better than bundler aliases.
Solution 
// package.json
{
  "type": "module",
  "imports": { "#lib/*": "./src/lib/*" }
}
// src/routes/users.js
import { db } from "#lib/db.js";              // instead of ../../lib/db.js
// Why better than bundler aliases: it's resolved by NODE ITSELF — works in
// plain `node`, in the built-in test runner, in ESLint's resolver, without
// teaching every tool a custom alias config. The # prefix guarantees no clash
// with npm package names. Refactors that move files change one mapping.
6 (hard): Create a deliberate circular dependency (order.js ↔ customer.js), show the failure in both CJS and ESM flavors, then fix it structurally.
Solution 
// CJS flavor:
// order.cjs:    const { fmtCustomer } = require("./customer.cjs");
//               exports.fmtOrder = o => `#${o.id} for ${fmtCustomer(o.c)}`;
// customer.cjs: const { fmtOrder } = require("./order.cjs");   // ← cycle re-enter:
//               exports.fmtCustomer = c => c.name;             //   gets PARTIAL order
//               exports.summary = c => c.orders.map(fmtOrder); // fmtOrder === undefined!
// Run: TypeError only when summary() is called — a landmine, not a startup crash.
// ESM flavor: same shape throws ReferenceError ("Cannot access before
// initialization") on access to the uninitialized binding — earlier and louder.
//
// Structural fix — extract the shared layer both need:
// format.mjs:   export const fmtCustomer = c => c.name;
// order.mjs:    import { fmtCustomer } from "./format.mjs"; export const fmtOrder = …
// customer.mjs: import { fmtOrder } from "./order.mjs";     export const summary = …
// The graph is now a DAG: format ← order ← customer. Cycles almost always mean
// one module holds two responsibilities — the extraction names the hidden one.
7 (hard): Design the package.json for an internal library consumed by (a) an ESM Node service, (b) a legacy CJS service, and (c) a bundled frontend — with locked internals and a /testing subpath of fakes. Note the one hazard your design must dodge.
Solution 
{
  "name": "@acme/billing",
  "type": "module",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",       // (a) ESM service
      "require": "./dist/index.cjs",      // (b) legacy CJS service
      "default": "./dist/index.mjs"       // (c) bundlers pick import/default
    },
    "./testing": {
      "types": "./dist/testing.d.ts",
      "import": "./dist/testing.mjs",     // fakes ship separately: prod bundles
      "require": "./dist/testing.cjs"     // never accidentally include them
    }
  },
  "sideEffects": false,                    // tree-shaking green light for (c)
  "files": ["dist"],
  "engines": { "node": ">=20" }
}
// Locked internals: ONLY "." and "./testing" resolve — dist/internal/* is
// unreachable, so refactors inside dist are invisible to consumers.
// THE HAZARD: dual-package — if the ESM service and a CJS dep both load
// @acme/billing, they get DIFFERENT module instances (.mjs and .cjs), so
// module-level state (a rate-limiter map, an instanceof check on a custom
// error class) silently splits. Dodge: keep the package stateless, or put
// shared state/classes in ONE .cjs core file that both builds import.

🎯 Key Takeaways

  • CJS = files wrapped in functions, loaded synchronously at runtime, exports copied, cached forever after first require; ESM = static graph, live bindings, async loading, top-level await.
  • Mode rules: .mjs/.cjs are explicit; .js follows the nearest package.json "type" — set "type": "module" in new projects and write ESM (with mandatory extensions).
  • Interop: ESM imports CJS as a default; CJS reaches ESM via dynamic import (or Node 22's require(esm), sans top-level await); libraries publish both through conditional "exports".
  • Resolution: relative paths resolve from the file, bare specifiers walk up node_modules, "exports" locks package internals, "#imports" kill ../../ ladders natively.
  • The cache makes modules singletons — deliberate for pools/config, a leak for test-isolated state; modules define, entry points do.
  • Circular imports hand out partial modules (undefined/TDZ landmines) — the fix is always extracting the shared piece, never reordering requires.
