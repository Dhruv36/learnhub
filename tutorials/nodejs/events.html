EventEmitter Async Node

What you'll master: the observer pattern that underlies all of Node's core — servers, sockets, streams, processes are all emitters — the full API including the special error event, listener-leak hygiene, promise/iterator bridges, and when events beat callbacks/promises as a design choice (and when they don't). Prerequisites: Async Patterns.

Part 1 — The Pattern, and Why Node Is Built on It

import { EventEmitter } from "node:events";

const uploader = new EventEmitter();
uploader.on("progress", pct => bar.update(pct));       // subscribe (many allowed)
uploader.on("done", file => console.log("saved", file));
uploader.emit("progress", 40);                          // publish → all listeners run,
uploader.emit("done", "video.mp4");                     //   SYNCHRONOUSLY, in order

// Why this shape instead of a callback? A callback answers ONE completion.
// Events model things that happen REPEATEDLY, to MANY interested parties,
// where the source shouldn't know who's listening:
// req/res, sockets, streams, process signals — everything in Node that
// "happens over time" speaks this protocol:
server.on("request", handler);
socket.on("data", onChunk);
process.on("SIGTERM", shutdown);
// So this lesson is secretly "how to read Node's core APIs".
// The full toolkit:
emitter.on("evt", fn)            // subscribe
emitter.once("evt", fn)          // auto-unsubscribe after first firing
emitter.off("evt", fn)           // unsubscribe — needs the SAME function reference!
emitter.emit("evt", a, b)        // fire with any args; returns false if 0 listeners
emitter.listenerCount("evt")
emitter.removeAllListeners("evt")  // sledgehammer — cleanup of emitters you OWN only
emitter.prependListener("evt", fn) // jump the queue (rare, smells like coupling)

// The reference identity trap:
uploader.off("progress", pct => bar.update(pct));   // ❌ different arrow — removes nothing
const onProgress = pct => bar.update(pct);
uploader.on("progress", onProgress);
uploader.off("progress", onProgress);                // ✅ same reference

Part 2 — The "error" Event: the One With Teeth

// SPECIAL RULE: emitting "error" with NO listener THROWS — and as an
// exception from emitter code, it usually escapes everything and CRASHES
// the process (L5's uncaughtException path):
const socket = net.createConnection(9999);       // nothing listening there
// → Uncaught Error: connect ECONNREFUSED  💥 process down

socket.on("error", err => {                      // one listener = handled
  log.warn({ err }, "connection failed, will retry");
});
// RULE: every emitter that CAN error (sockets, streams, servers, child
// processes — i.e., everything doing I/O) gets an error listener, immediately,
// at creation. "My Node app crashes randomly under load" is, more often than
// not, an unhandled 'error' on some socket deep in a dependency.
//
// Design symmetry when you EMIT: errors are events like any other —
this.emit("error", new Error("parse failed"));   // let owners decide policy —
// never throw from inside your emitter's async work (nobody can catch it).

Part 3 — Building an Emitter: Extend, Emit, Document

// The idiomatic shape — your class IS an emitter:
import { EventEmitter } from "node:events";

export class JobRunner extends EventEmitter {
  #queue = [];
  #running = false;

  add(job) {
    this.#queue.push(job);
    this.emit("queued", { id: job.id, depth: this.#queue.length });
    this.#drain();
  }

  async #drain() {
    if (this.#running) return;
    this.#running = true;
    while (this.#queue.length > 0) {
      const job = this.#queue.shift();
      this.emit("start", { id: job.id });
      try {
        const result = await job.fn();
        this.emit("done", { id: job.id, result });
      } catch (err) {
        this.emit("job-error", { id: job.id, err });   // note: NOT "error" —
      }                                                 // a failed job isn't an
    }                                                   // emitter failure; don't
    this.#running = false;                              // crash zero-listener owners
    this.emit("idle");
  }
}

// Consumers wire what they care about — logger, metrics, UI, none know of each other:
const runner = new JobRunner();
runner.on("done",      e => metrics.increment("jobs.done"));
runner.on("job-error", e => log.error(e, "job failed"));
runner.on("idle",      () => maybeExit());
// Document your events like an API: name, payload shape, ordering guarantees —
// they ARE your API. (TypeScript: type the event map for compile-checked payloads.)

Part 4 — Leaks & Lifecycle: the Production Hygiene

// The classic leak — subscribing per request to a long-lived emitter:
app.get("/live", (req, res) => {
  bus.on("update", send);                      // ❌ added EVERY request…
  // …and never removed: bus holds `send` (and its closure over res) FOREVER
});
// Symptom: "MaxListenersExceededWarning: 11 update listeners added" — that
// warning (default threshold 10) is a LEAK DETECTOR, not noise to silence
// with setMaxListeners(0). Memory grows; old responses are retained; every
// emit does more work.

// The discipline: every on() has a scheduled off():
app.get("/live", (req, res) => {
  const send = data => res.write(`data: ${JSON.stringify(data)}\n\n`);
  bus.on("update", send);
  res.on("close", () => bus.off("update", send));   // lifecycle-bound cleanup
});
// Same rule you learned for React effects and fs.watch — subscription and
// cleanup are one unit. AbortSignal integrates here too:
bus.on("update", send, { signal: controller.signal });  // abort = auto-off
// (Node's on/once accept { signal } — cancellation unifies, as in L7.)

Part 5 — Bridges: Promises and Async Iteration

import { once, on } from "node:events";

// once() — promisify a single occurrence (with error wiring built in!):
const server = app.listen(0);
await once(server, "listening");             // resolves on listening,
                                              // REJECTS if "error" fires first —
const [socket] = await once(net.connect(port), "connect");   // the L7 hand-wrap, free

// on() — consume an event stream as an async iterator:
for await (const [msg] of on(bus, "message", { signal })) {
  await handle(msg);                          // sequential processing with
}                                              // backpressure-ish pacing
// Caveat: on() buffers unboundedly while you await — a fast emitter + slow
// handler grows the buffer. True flow control is what STREAMS add (next
// lesson); on() suits low-rate control events, not data firehoses.
//
// Choosing the shape (design guidance, not law):
// one-shot result            → promise (or once() around a legacy emitter)
// stream of data w/ pressure → stream / async iterator
// broadcast to N unknown listeners, repeatedly → EventEmitter
// in-process module decoupling ("order_placed") → a small typed event bus —
//   the React-track's "events over imports" idea, server-side. Cross-process
//   needs a real broker (Redis pub/sub, queues — Caching/Scaling lessons).

Part 6 — Semantics That Bite (Senior Layer)

// 1. emit() is SYNCHRONOUS: all listeners run before emit returns.
//    A slow listener blocks the emitter (and the loop — L6). Listeners doing
//    real work should schedule it (queueMicrotask, setImmediate, a queue) and
//    return fast.
// 2. Async listeners' rejections are UNHANDLED by the emitter:
bus.on("update", async e => { await save(e); });   // save() rejects → nobody catches
//    → wrap listener bodies (try/catch + log) or use events.captureRejections:
const bus = new EventEmitter({ captureRejections: true });
bus[Symbol.for("nodejs.rejection")] = err => log.error({ err });
// 3. Ordering: listeners run in registration order — DON'T design protocols
//    that depend on it across modules (registration order is load order — fragile).
// 4. emit returns false when nobody listened — "fire into the void" detection:
if (!bus.emit("audit", entry)) log.warn("no audit listener attached!");
// 5. Memory: an emitter retains every listener's closure. Long-lived emitter +
//    short-lived subscribers = the leak in Part 4. Short-lived emitter (a req)
//    + listeners = fine; it all releases together.

Interview Questions

Beginner: What is an EventEmitter and where does Node use it?

Node's observer-pattern primitive: objects that maintain listener lists per event name; on subscribes, emit invokes all listeners synchronously with the payload. Nearly every core async object is one — HTTP servers ("request"), sockets ("data", "close"), streams, child processes, process itself ("SIGTERM") — so fluency in the API is fluency in reading Node code.

Beginner: What's special about the "error" event?

If "error" is emitted with no listener attached, Node throws the error instead — and since it originates inside emitter internals, it typically becomes an uncaught exception that kills the process. Hence the rule: any emitter that performs I/O gets an error listener at creation. Conversely, when designing emitters, reserve "error" for genuine emitter failure — domain-level failures get their own event names so they can't crash inattentive consumers.

Mid: What does MaxListenersExceededWarning usually indicate and how do you fix it properly?

A listener leak: some code path adds listeners to a long-lived emitter repeatedly (per request, per retry, per render) without removing them — the default threshold of 10 exists precisely to surface this. Proper fix: find the unpaired on() and bind its removal to the subscriber's lifecycle (res.on("close", …), { signal }, once()). Raising the limit is correct only when a legitimately high static fan-out exists — and then you raise it to a documented number, never 0 (unlimited = detector off).

Mid: Compare events.once() and events.on() with hand-rolled listener code.

once(emitter, "evt") returns a promise for the next occurrence and — crucially — rejects if "error" fires first, encapsulating the resolve/reject wiring people hand-write wrong (forgetting the error path, leaking the unused listener). on(emitter, "evt") adapts an event stream into an async iterator for for await, with AbortSignal support — but it buffers unboundedly while the consumer is slow, so it fits control-plane events, while data-plane flows belong to streams with real backpressure.

Senior: Design an in-process event bus for a modular monolith — API, guarantees, and the boundaries of the pattern.

A single typed emitter module: event map documented (name → payload schema, versioned), publish helper that validates payloads in dev, subscriptions via { signal } so features detach on shutdown, captureRejections wired to logging, and a "no listener" warning on emit-into-void for must-be-consumed events. Guarantees stated explicitly: synchronous, in-process, at-most-once, no replay, lost on crash — which defines the boundary: it decouples modules within one process ("order_placed" → analytics, email, inventory listeners), but anything needing durability, retries, cross-process delivery, or ordering under failure graduates to a real broker/queue. The bus's value is dependency-direction hygiene (emitters don't import consumers) — not delivery semantics it doesn't have.

Senior: A service crashes with an uncaught ECONNRESET that no application code seems to produce. Walk the diagnosis.

Signature of an unhandled "error" event on a socket owned by a dependency — DB drivers, HTTP keep-alive agents, Redis clients all hold long-lived sockets that reset when peers/LBs recycle connections. Diagnosis: the crash stack shows the emit site (deep in node:net/internal), not your code; capture it via process.on("uncaughtException") logging with the error's origin, or run with --trace-uncaught. Then find which client lacks an error handler: audit each long-lived client's setup for .on("error"), check library issues for known handler gaps, and add handlers + reconnect policy at every client's creation point. Interim mitigation: the fatal-log-and-exit handler from L5 keeps restarts clean while you hunt. The broader lesson: every dependency that holds sockets is an emitter someone must be listening to.

🏋️ Graded Exercises

1 (easy): Build a Countdown emitter: start(n) emits "tick" each second with the remaining count, then "done". Use it with both listeners.
Solution 
class Countdown extends EventEmitter {
  start(n) {
    const timer = setInterval(() => {
      n--;
      this.emit("tick", n);
      if (n === 0) { clearInterval(timer); this.emit("done"); }
    }, 1000);
  }
}
const c = new Countdown();
c.on("tick", n => console.log(n));
c.once("done", () => console.log("🚀"));
c.start(3);
2 (easy): Why does this still log after "unsubscribe", and fix it: bus.on("msg", m => console.log(m)); /* later */ bus.off("msg", m => console.log(m));
Solution 
// Two different arrow-function objects: off() removes by reference identity and
// finds no match — the original stays subscribed.
const onMsg = m => console.log(m);
bus.on("msg", onMsg);
bus.off("msg", onMsg);
// (Or subscribe with an AbortSignal and abort — no reference bookkeeping.)
3 (medium): Wire a robust TCP client: connect, greet, log replies — with the error handling that keeps every failure (refused, mid-session reset) from crashing, and a single "close" cleanup.
Solution 
import net from "node:net";

const socket = net.createConnection({ host, port });

socket.on("error", err => {                   // FIRST thing wired: refused,
  log.warn({ code: err.code }, "socket error"); // reset, timeout all land here
});                                            // instead of killing the process
socket.on("connect", () => socket.write("HELLO\n"));
socket.on("data", chunk => log.info({ reply: chunk.toString().trim() }));
socket.on("close", hadError => {               // fires after error OR clean end —
  log.info({ hadError }, "connection closed"); // ONE place for cleanup/reconnect
  scheduleReconnect();
});
// "error" then "close" is the guaranteed order on failure — policy (retry,
// alert) belongs in close's single choke point, logging in error.
4 (medium): An SSE endpoint leaks: after a day, 4,000 "price" listeners and rising. Find the bug and fix with both the manual and the signal-based pattern: app.get("/prices", (req, res) => { sse(res); market.on("price", p => res.write(fmt(p))); });
Solution 
// Each request subscribes to the app-lifetime `market` emitter and never
// unsubscribes; closed connections leave dead listeners holding res closures.
// Manual:
app.get("/prices", (req, res) => {
  sse(res);
  const onPrice = p => res.write(fmt(p));
  market.on("price", onPrice);
  res.on("close", () => market.off("price", onPrice));
});
// Signal-based (tidier — one line of cleanup wiring):
app.get("/prices", (req, res) => {
  sse(res);
  const ac = new AbortController();
  market.on("price", p => res.write(fmt(p)), { signal: ac.signal });
  res.on("close", () => ac.abort());
});
// Verify the fix: market.listenerCount("price") tracks CURRENT connections now.
// (And setMaxListeners to expected-peak-connections, documented — the warning
// stays useful.)
5 (medium): Use events.once to write listen(app, port) returning a promise of the bound port (0 = random), rejecting on EADDRINUSE — then show the equivalent hand-rolled version and what it forgets.
Solution 
import { once } from "node:events";

async function listen(app, port = 0) {
  const server = app.listen(port);
  await once(server, "listening");            // rejects automatically if "error"
  return { server, port: server.address().port };
}

// Hand-rolled — and its classic bugs:
function listenManual(app, port = 0) {
  return new Promise((resolve, reject) => {
    const server = app.listen(port);
    server.once("listening", () => {
      server.off("error", reject);            // ← forgotten in most versions:
      resolve(server);                        //   success leaves the reject
    });                                        //   listener attached forever
    server.once("error", reject);             // ← forgotten in the rest: EADDRINUSE
  });                                          //   crashes instead of rejecting
}
// once() encapsulates both cleanups — this is why the bridge functions exist.
6 (hard): Build the typed event bus from the senior question: schema-validated payloads (dev only), signal-based subscribe, captureRejections logging, and void-emit warnings for critical events. Show two features using it without importing each other.
Solution 
// bus.js
import { EventEmitter } from "node:events";

const SCHEMAS = {                              // name → validator (dev contract)
  order_placed: p => typeof p.orderId === "string" && typeof p.total === "number",
  user_deleted: p => typeof p.userId === "string",
};
const CRITICAL = new Set(["order_placed"]);    // someone MUST consume these

class Bus extends EventEmitter {
  constructor() { super({ captureRejections: true }); }
  [Symbol.for("nodejs.rejection")](err, event) {
    log.error({ err, event }, "async listener rejected");
  }
  publish(name, payload) {
    if (process.env.NODE_ENV !== "production" && !SCHEMAS[name]?.(payload))
      throw new Error(`bus: invalid payload for "${name}"`);
    const heard = this.emit(name, payload);
    if (!heard && CRITICAL.has(name))
      log.warn({ name }, "critical event had no listeners");
    return heard;
  }
  subscribe(name, fn, { signal } = {}) { this.on(name, fn, { signal }); }
}
export const bus = new Bus();

// features/checkout.js — emits, imports NO consumer:
bus.publish("order_placed", { orderId, total });

// features/email.js — consumes, imports NO producer:
bus.subscribe("order_placed", async ({ orderId }) => {
  await sendReceipt(orderId);                  // rejection → captureRejections log
}, { signal: shutdownSignal });
// Dependency arrows both point at bus.js only — the decoupling is real, and the
// stated guarantees (sync, in-process, at-most-once) are documented at the top.
7 (hard): A metrics module subscribes to 14 emitters across the app and does an HTTP POST inside each listener. Under burst load, p99 rose and one unhandled rejection crashed prod. Redesign the module citing Part 6's semantics.
Solution 
// Diagnosis against the semantics:
// • emit() is synchronous → 14 listeners each starting an HTTP POST add real
//   work to EVERY hot-path emit; under bursts that's loop pressure (L6).
// • async listeners' rejections are unowned → one failed POST = crash (L7).
// Redesign — listeners RECORD, a pump SHIPS:
const buffer = [];
function listen(emitter, event, toMetric) {
  emitter.on(event, payload => {
    buffer.push(toMetric(payload));            // synchronous, O(1), no I/O —
  });                                           // emit() returns instantly
}
// one owned background pump, batched + bounded + fail-soft:
const pump = setInterval(() => {
  if (buffer.length === 0) return;
  const batch = buffer.splice(0, 500);
  postMetrics(batch).catch(err => {            // OWNED rejection (L7 rule)
    log.warn({ err, dropped: batch.length }, "metrics ship failed");
  });                                           // drop on failure: metrics are
}, 5000);                                       // fail-soft by nature — never
pump.unref();                                   // block exit, never crash prod
// Plus a cap: if (buffer.length > 50_000) buffer.length = 25_000; — bounded
// memory under a dead metrics backend. Listeners are now pure recorders;
// all I/O lives in one supervised place.

🎯 Key Takeaways

  • EventEmitter is Node's broadcast primitive — servers, sockets, streams, and process all speak it; reading core APIs = reading event contracts.
  • "error" without a listener throws and usually kills the process: wire error handlers at creation on everything that does I/O; reserve the name for emitter failure.
  • Every on() has a lifecycle-bound off() (reference-identical, or via once/{ signal }); MaxListeners warnings are leak detectors, never noise.
  • emit() is synchronous and listeners' async rejections are unowned — listeners return fast and own their promises (or captureRejections).
  • events.once/on bridge to promises and async iteration with error wiring built in — but on() has no backpressure; data firehoses belong to streams.
  • Events shine at one-to-many broadcast and module decoupling with stated (weak) guarantees; request-response stays promises, durable delivery graduates to brokers.
