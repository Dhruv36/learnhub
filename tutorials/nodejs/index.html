Runtime, Modules & npm Basics

Node.js is JavaScript outside the browser: V8 (the engine) + libuv (the async I/O layer) + system APIs (files, network, processes). Its founding bet — one thread + non-blocking I/O beats thread-per-request for I/O-heavy servers — powers Netflix, PayPal, and most startup APIs. Prerequisite: the JavaScript track (especially async, event loop, modules).

Modules — ESM in Node

// package.json: { "type": "module" }  → .js files are ESM
import { readFile } from "node:fs/promises";     // node: prefix = built-in (explicit, safe)
import express from "express";                    // bare = node_modules
import { helper } from "./utils.js";              // relative needs the EXTENSION in ESM

// Legacy CommonJS (recognize everywhere, write rarely):
const fs = require("fs");  module.exports = { helper };
// Interop: ESM can import most CJS (default import); CJS require() of ESM is limited.
// __dirname doesn't exist in ESM: use import.meta.dirname (or fileURLToPath(import.meta.url)).

The One Mental Model That Matters

ONE thread runs your JS. Every request shares it.
I/O (db, files, network) is delegated to libuv → thread never waits → huge concurrency.
CPU work (parsing MBs, crypto loops, image resize) BLOCKS EVERY REQUEST:

app.get("/report", (req, res) => {
  const data = crunchFor3Seconds();   // ❌ all other users frozen for 3s
  res.json(data);
});
// Fixes: worker_threads (scaling lesson), queues, streaming. This single fact
// explains Node's sweet spot (APIs, I/O glue, realtime) and its wrong tool
// (CPU-heavy batch work — or at least not on the request path).

Interview Questions

Q1: Why is Node fast for I/O-heavy workloads despite one thread?

Threads waiting on I/O waste memory and context switches; Node parks the request (callback/promise) while libuv's async I/O (epoll/kqueue/IOCP) waits, so one thread multiplexes tens of thousands of in-flight requests. The trade: CPU-bound work has nowhere to hide — hence workers. It's the event-loop lesson's model with server stakes.

Q2: CommonJS vs ESM in Node — practical differences that bite?

ESM: static imports (tree-shakable, async-loaded), file extensions required, import.meta instead of __dirname, top-level await. CJS: synchronous require, dynamic paths easy, still the format of much of npm. Interop is asymmetric (ESM→CJS fine; CJS→ESM needs dynamic import()). New code: type: module; know CJS to read the ecosystem.

Q3: Where do process.nextTick and setImmediate fit vs browser microtasks?

Node's loop has phases (timers → pending → poll/I-O → check → close). process.nextTick runs before other microtasks after the current operation — starvation-capable; setImmediate runs in the check phase (after I/O). setTimeout(0) vs setImmediate ordering varies in main script but setImmediate wins inside I/O callbacks — the classic trivia. Prefer promises/microtasks in app code.

🎯 Key Takeaways

  • Node = V8 + libuv: one JS thread, non-blocking I/O — massive I/O concurrency, zero CPU forgiveness.
  • ESM with node: prefixes and extensions; import.meta replaces __dirname; CJS is read-only legacy.
  • fs/promises + path.join + node:crypto; sync calls are startup-only.
  • Config via process.env (--env-file), exact deps via lockfile + npm ci, runtime pinned via engines.
  • Never block the loop — the sentence that summarizes all Node architecture.
