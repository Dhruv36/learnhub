Production Hardening & Deployment Ultra-Advanced

The gap between "runs on my laptop" and "survives production": graceful shutdown, health checks, security headers, resource limits, and observability. This is the checklist that turns an API into a service — and the one senior engineers are expected to recite.

Graceful Shutdown — Don't Drop In-Flight Requests

const server = app.listen(PORT);
function shutdown(signal) {
  logger.info(`${signal} — draining`);
  server.close(async () => {                        // stop ACCEPTING; finish in-flight requests
    await pool.end();                                // close DB pool
    await queue.close();                             // release other resources
    process.exit(0);
  });
  setTimeout(() => process.exit(1), 10_000).unref(); // force-exit if drain hangs
}
process.on("SIGTERM", () => shutdown("SIGTERM"));    // K8s/Docker send SIGTERM on rollout
process.on("SIGINT",  () => shutdown("SIGINT"));     // Ctrl+C
// Without this, a deploy KILLS active requests mid-flight (users see errors) and leaks
// connections. Every orchestrated deploy depends on honoring SIGTERM.

Health Checks — Liveness vs Readiness

app.get("/livez",  (req, res) => res.sendStatus(200));   // "process is alive" — restart if failing
app.get("/readyz", async (req, res) => {                 // "ready for traffic" — depends are up
  try { await pool.query("SELECT 1"); res.sendStatus(200); }
  catch { res.sendStatus(503); }                         // DB down → pull me from the LB, don't kill me
});
// The distinction matters: a failing DB should make you UNREADY (stop routing traffic) but
// still ALIVE (don't restart-loop — the DB, not the process, is the problem).

The Production Dockerfile (multi-stage, non-root)

FROM node:22-slim AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci                                    # exact lockfile
COPY . .
RUN npm run build

FROM node:22-slim
WORKDIR /app
ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev && npm cache clean --force    # no devDeps in the runtime image
COPY --from=build /app/dist ./dist
USER node                                     # ← non-root (least privilege)
EXPOSE 3000
CMD ["node", "dist/server.js"]
// NODE_ENV=production: Express skips dev overhead, libraries optimize. Small base image =
// smaller attack surface + faster pulls. (Full Docker depth: the Docker track.)

Interview Questions

Q1: Walk a zero-downtime deploy of a Node service on Kubernetes.

New pods start → readyz passes only when DB/deps reachable → LB routes to them → old pods get SIGTERM → they stop accepting (server.close), finish in-flight requests, drain the pool, exit → rolling update completes with no dropped requests. Requires: honoring SIGTERM (graceful shutdown), correct readiness (don't advertise ready early), and backward-compatible DB migrations (expand/contract) so old and new run together briefly.

Q2: Liveness vs readiness — why two probes?

Liveness answers "is the process healthy enough to keep running?" — failing it RESTARTS the pod (for deadlocks/unrecoverable state). Readiness answers "should it receive traffic NOW?" — failing it removes the pod from the load balancer without restarting. A dependency outage (DB down) should make you unready (stop routing) but stay alive (restarting won't fix the DB and causes crash loops). Conflating them turns a DB blip into a restart storm.

Q3: Your service is healthy but users see intermittent 502s during deploys. Cause?

Missing/incorrect graceful shutdown: pods receive SIGTERM and exit immediately, killing in-flight requests (502 from the LB). Fix: server.close to drain, a shutdown grace period longer than the longest request, and readiness flipping to unready BEFORE draining so the LB stops sending new traffic first. Often paired with too-short terminationGracePeriodSeconds. It's a lifecycle bug, not a code bug.

🏋️ Practice: add SIGTERM graceful shutdown + livez/readyz to your API; write the multi-stage non-root Dockerfile and build it; add helmet + rate limiting + request timeout; expose a /metrics endpoint (prom-client) with request latency and event-loop lag; simulate a DB-down readiness failure.

🎯 Key Takeaways

  • Graceful shutdown on SIGTERM (drain, close resources, force-exit fallback) = zero-downtime deploys.
  • Liveness (restart) vs readiness (un-route) are different questions — implement both.
  • Harden: helmet, rate limits, body/timeout limits, secrets outside the image, non-root, NODE_ENV=production.
  • Multi-stage Docker, --omit=dev, small base, USER node.
  • Observability = logs + metrics (incl. event-loop lag) + traces, alerting on SLOs — the service-vs-script line.
