npm, package.json & Dependencies Runtime Foundations

What you'll master: package.json as the project manifest, semver and version ranges precisely, the lockfile's actual job, dependency categories, npm scripts as the project's task interface, and supply-chain hygiene — the discipline that separates professional dependency management from npm install-and-pray. Prerequisites: Modules (resolution, "exports").

Part 1 — package.json: the Manifest

// $ npm init -y  → then curate:
{
  "name": "invoice-api",
  "version": "1.4.2",
  "type": "module",                      // ESM (Modules lesson)
  "private": true,                       // apps: NEVER publishable by accident
  "engines": { "node": ">=22" },         // refuse wrong runtimes (with engine-strict)
  "scripts": {
    "dev": "node --watch src/server.js",
    "test": "node --test",
    "start": "node src/server.js"
  },
  "dependencies": {                      // needed AT RUNTIME in production
    "express": "^5.1.0",
    "pg": "^8.13.0"
  },
  "devDependencies": {                   // build/test/dev only — pruned in prod
    "eslint": "^9.20.0"
  }
}
// The categories matter operationally:
// npm install --omit=dev (production image builds) skips devDependencies —
// misfiling a runtime dep as dev = crash in prod; the reverse = fat images.
// Also exists: peerDependencies — "I work WITH X, the APP provides it"
// (a React component library peer-depends on react: one copy, the app's).

Part 2 — Semver: the Contract in Three Numbers

// version = MAJOR.MINOR.PATCH
// 1.4.2 → 1.4.3  PATCH: bug fix, no API change        — safe to auto-take
// 1.4.2 → 1.5.0  MINOR: new features, backwards-compatible
// 1.4.2 → 2.0.0  MAJOR: BREAKING changes — read the migration guide
//
// Ranges you'll write:
"^1.4.2"    // caret (npm's default): any 1.x.y >= 1.4.2 — accepts minor+patch.
"~1.4.2"    // tilde: any 1.4.x >= 1.4.2 — patch only.
"1.4.2"     // exact pin.
// ^ caveat everyone learns the hard way: for 0.x versions, ^0.4.2 accepts only
// 0.4.x — pre-1.0, MINOR is treated as breaking (semver's own escape clause).
//
// The honest footnote: semver is a PROMISE, not a proof. Maintainers ship
// breaking patches by accident. Which is why ranges alone would make builds
// nondeterministic — enter the lockfile.

Part 3 — The Lockfile: Reproducibility

// package.json says what you WANT ("^1.4.2" — a range).
// package-lock.json records what you GOT: the exact version, resolved URL, and
// integrity hash of EVERY package in the tree — yours and transitive.
//
// The rules that follow:
// 1. COMMIT the lockfile. Always. It's the build's reproducibility.
// 2. CI and prod install with:
//    $ npm ci        // installs EXACTLY the lockfile (deletes node_modules,
//                    // fails if lock and manifest disagree) — fast, deterministic
//    npm install is for DEVELOPMENT (updates the lock when you add/change deps).
// 3. "Works on my machine, fails in CI" with deps → someone bypassed the lock
//    (installed without committing, or CI used npm install and drifted).
//
// Updating deliberately, not accidentally:
$ npm outdated                    // what has newer versions, and how breaking
$ npm update                      // move WITHIN ranges (lockfile updated)
$ npm install express@6           // major bumps: explicit, one at a time,
                                  // with the changelog open and tests green
// Tooling (Renovate/Dependabot) turns updates into small reviewable PRs —
// the difference between continuous small upgrades and a yearly "upgrade hell" week.

Part 4 — npm Scripts: the Project's Task Interface

"scripts": {
  "dev": "node --watch src/server.js",
  "test": "node --test",
  "lint": "eslint src",
  "build": "tsc",
  "prebuild": "npm run lint",            // pre<name> runs automatically before
  "start": "node dist/server.js",
  "db:migrate": "node scripts/migrate.js",
  "check": "npm run lint && npm test"    // composition via shell operators
}
// $ npm run dev     ($ npm test / npm start — those three skip "run")
//
// Why scripts and not a README of commands:
// 1. node_modules/.bin is on PATH inside scripts — "eslint src" just works
//    without global installs (and npx eslint for one-offs).
// 2. They're THE interface: CI runs npm ci && npm test && npm run build —
//    identical to what devs run. One definition of "how to build this".
// 3. Convention travels: any Node dev knows to try npm run dev / npm test.
// Keep them thin — a script longer than a line becomes scripts/do-thing.js
// (a real file with error handling), and the npm script just invokes it.

Part 5 — Supply-Chain Hygiene (the Part That's Actually Security)

// Remember the runtime lesson: dependencies run with YOUR permissions —
// on your laptop at install time (lifecycle scripts) and in prod at runtime.
// A median app: ~1,500 transitive packages. Each is trusted code. Discipline:

// 1. AUDIT the tree you have:
$ npm audit                        // known CVEs in your tree
$ npm audit --omit=dev             // what actually ships (dev noise excluded)
$ npm audit fix                    // auto-bump within ranges; review the rest
// Wire it into CI at a severity threshold — advisory noise is real; triage,
// don't rubber-stamp either direction.

// 2. THINK before adding:
$ npm ls                           // what's already here (don't add a 2nd date lib)
// is-odd-style micro-packages → write the line yourself (JS track skills);
// check maintenance (last publish, issue tray), weight (bundlephobia),
// and install scripts (postinstall = code execution on npm install:
//   npm install --ignore-scripts in CI where possible).

// 3. CONTAIN blast radius:
// • Lockfile + integrity hashes: a tampered tarball fails to install.
// • npm ci in CI: no silent drift toward a hijacked new version.
// • Provenance/signatures (npm's provenance attestations) for what you publish.
// • For orgs: a proxy registry (Artifactory/Verdaccio) with allow-lists and
//   quarantine windows for brand-new versions (typosquats and hijacks are
//   caught in the first hours — waiting 72h dodges most).
//
// 4. The judgment call, stated as policy: a dependency must clear the bar of
//    "cheaper to maintain the dep relationship than to own the code."
//    express: obviously. leftpad: obviously not. The middle: discuss in review.

Part 6 — Publishing & Workspaces (Senior Layer)

// PUBLISHING a package (the Modules lesson's "exports" design, shipped):
{
  "name": "@acme/retry",
  "version": "1.0.0",
  "files": ["dist"],                 // allowlist what goes in the tarball
  "exports": { ".": { "import": "./dist/index.mjs", "require": "./dist/index.cjs" } }
}
$ npm publish --access public --provenance
// npm version patch|minor|major → bumps + git-tags; publish from CI, not laptops
// (tokens scoped + short-lived; provenance links the tarball to the exact
// commit and build — consumers can verify).

// WORKSPACES — one repo, many packages, npm-native:
{ "workspaces": ["packages/*", "apps/*"] }
// $ npm install                    // ONE root node_modules, local packages
//                                  // symlinked — apps/* import packages/* live
// $ npm run test -w packages/core  // target one workspace
// $ npm run build --workspaces     // or all
// This is the monorepo baseline (React track L25's org stage): shared code as
// real packages with real manifests, without publishing every internal lib.
// (Turborepo/Nx add task caching on top when build times demand it.)

Interview Questions

Beginner: dependencies vs devDependencies vs peerDependencies?

dependencies: required at runtime in production (express, pg). devDependencies: needed only to develop/build/test (eslint, TypeScript) — excluded by --omit=dev installs. peerDependencies: "I integrate with X but the consuming app must provide it" — how a React plugin ensures exactly one React copy. The category answers one question: who needs this code, and when?

Beginner: What does ^1.4.2 accept, and what's the 0.x exception?

Caret accepts anything compatible-per-semver from 1.4.2 up to (not including) 2.0.0 — new minors and patches. For 0.x versions, the leftmost non-zero digit is the breaking boundary: ^0.4.2 accepts only 0.4.x, because pre-1.0 packages treat minor bumps as breaking. Tilde (~1.4.2) restricts to patches only.

Mid: What problem does the lockfile solve, and how do npm install and npm ci treat it differently?

Ranges in package.json make resolution time-dependent — the same manifest installs different trees on different days. The lockfile freezes the entire resolved tree (exact versions + integrity hashes), making installs reproducible and tamper-evident. npm install respects the lock but may update it (adding deps, moving within ranges); npm ci installs exactly the lock, from a clean slate, and fails on manifest/lock mismatch — which is why CI and production use ci, and lock diffs get reviewed like code.

Mid: Walk through your dependency-update strategy for a production service.

Automated small PRs (Renovate/Dependabot) grouped sensibly: patches/minors auto-merged when CI is green and coverage is trusted; majors as individual PRs with the changelog read and migration steps in the description. npm audit in CI with a severity gate for the shipped tree (--omit=dev). A weekly cadence beats quarterly big-bangs — small diffs bisect trivially when something regresses. Emergency path: security patches jump the queue; pin-and-schedule anything that can't take the fix immediately.

Senior: Threat-model the npm supply chain for your org and the mitigations per threat.

Threats: typosquats (near-name packages), hijacked maintainer accounts publishing malicious versions, malicious postinstall scripts executing at install time, dependency-confusion (public package shadowing an internal name), and compromised transitive deps deep in the tree. Mitigations, layered: lockfile + npm ci (no ambient version drift; integrity hashes catch tarball tampering); install-script restrictions in CI; a proxy registry with allow-lists, internal-scope protection (@acme/* only from the private registry — kills confusion attacks), and a quarantine delay on brand-new versions (hijacks are usually caught in hours); provenance verification where available; minimal dependency policy to shrink the surface; and runtime egress monitoring, because install-time is not the only activation point. No single layer suffices; the lockfile alone only guarantees you reproducibly install the compromise from last Tuesday.

Senior: When do workspaces beat publishing internal packages, and what discipline keeps a workspace monorepo healthy?

Workspaces win when the packages' consumers all live in the same repo: changes land atomically across lib and app (no version-bump-publish-update dance), refactors are one PR, and local symlinks mean always-current code. Publishing wins when consumers are external or on independent release trains. Discipline for healthy workspaces: real package boundaries (each with its own manifest, "exports", and tests — the React L25 contract idea), no reaching across packages via relative paths (import the package name), dependency direction enforced (apps → libs, never sideways-into-internals), and task orchestration with caching (turborepo/nx) once --workspaces runs stop scaling. The failure mode to guard: the monorepo where everything imports everything — folders changed, coupling didn't.

🏋️ Graded Exercises

1 (easy): Which range: (a) a logging lib you want patches+features from; (b) a flaky parser that broke you on a patch once; (c) your company's internal design tokens where any change should be deliberate.
Solution 
// a. "^2.3.1"  — default caret: trust semver, take minors and patches.
// b. "~2.3.1"  — tilde: patches only… and honestly, after a breaking patch,
//                consider exact-pinning + Renovate PRs so every change is reviewed.
// c. "1.8.0"   — exact pin: token changes alter every screen; each bump should
//                be a visible, reviewed PR, not ambient drift.
2 (easy): A teammate's PR contains only a 4,000-line package-lock.json diff with no package.json change. What are the legitimate and illegitimate explanations, and what do you ask?
Solution 
// Legitimate: they ran `npm update` (moving within ranges) or `npm audit fix` —
// both rewrite the lock without touching manifest ranges. Also: first install
// after someone previously forgot to commit lock changes.
// Illegitimate/smelly: they ran npm install with a different npm major version
// (lockfileVersion churn), or the diff includes packages nobody added (check
// for new names, changed registry URLs, or altered integrity hashes — the
// supply-chain red flag).
// Ask: "which command produced this?" — and re-run it yourself; the diff should
// reproduce. Lockfile diffs are code review, not noise to scroll past.
3 (medium): Set up scripts for a service: dev with watch + NODE_ENV, test, lint, a typecheck, a combined check for CI, and a db migration runner — following the thin-script rule.
Solution 
"scripts": {
  "dev": "NODE_ENV=development node --watch --env-file=.env src/server.js",
  "start": "node src/server.js",
  "test": "node --test",
  "lint": "eslint src",
  "typecheck": "tsc --noEmit",
  "check": "npm run lint && npm run typecheck && npm test",
  "db:migrate": "node scripts/migrate.js"
}
// CI runs: npm ci && npm run check. scripts/migrate.js is a real file with
// error handling and arguments — the npm script just names it. (Windows note:
// inline VAR= needs cross-env, or use --env-file as here.)
4 (medium): Your prod Docker image is 1.1GB and contains eslint, typescript, and vitest. Diagnose and fix the install flow.
Solution 
# The image ran plain `npm ci` — devDependencies included.
# Multi-stage fix:
FROM node:22-slim AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci                       # full tree: build needs dev deps
COPY . .
RUN npm run build

FROM node:22-slim
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev            # runtime tree only
COPY --from=build /app/dist ./dist
CMD ["node", "dist/server.js"]
# And the categories must be honest for --omit=dev to be safe — a runtime dep
# misfiled as dev now crashes here, which is exactly the test you want failing
# in CI, not in prod.
5 (medium): npm audit reports 23 vulnerabilities (2 critical in a transitive dep of your test runner, 1 high in qs under express). Triage properly.
Solution 
// 1. Partition by exposure: npm audit --omit=dev
//    → the 2 criticals vanish (test-runner tree never ships). They're still
//    worth an update PR, but they are NOT a prod incident.
// 2. The qs high IS shipped (query parsing = attacker-reachable input).
//    npm ls qs → see which range express pins. npm audit fix → if a patched
//    qs fits express's range, done: lock diff + tests green.
//    If not: check for an express patch release; "overrides" in package.json
//    as a temporary force ({ "overrides": { "qs": "6.13.1" } }) with a ticket
//    to remove it when upstream catches up.
// 3. Encode the policy: CI gate on high+critical in --omit=dev tree;
//    dev-tree advisories go to the weekly update PR, not the pager.
// Triage = exposure × reachability, not the raw count.
6 (hard): Initialize a workspaces monorepo: packages/logger (shared lib) consumed by apps/api, with per-workspace tests runnable from the root, and demonstrate the cross-package import working without publishing.
Solution 
# root package.json
{ "name": "acme", "private": true,
  "workspaces": ["packages/*", "apps/*"],
  "scripts": { "test": "npm run test --workspaces --if-present" } }

# packages/logger/package.json
{ "name": "@acme/logger", "version": "0.1.0", "type": "module",
  "exports": { ".": "./src/index.js" },
  "scripts": { "test": "node --test" } }

# apps/api/package.json
{ "name": "api", "private": true, "type": "module",
  "dependencies": { "@acme/logger": "*" },      # "*" = any — resolves to the
  "scripts": { "test": "node --test" } }        #   local symlinked workspace

# apps/api/src/server.js
import { log } from "@acme/logger";             # bare specifier → root
log("api up");                                  #   node_modules symlink → packages/logger

$ npm install          # links everything
$ npm test             # runs each workspace's tests
$ npm run test -w @acme/logger    # or just one
# Edit packages/logger/src → apps/api sees it immediately: symlinks, no publish.
7 (hard): Write the dependency policy (one page, ADR-style) for a 10-dev product team — adding, updating, auditing, and the approval bar — such that it's enforceable by CI rather than vibes.
Solution 
# ADR-0009: Dependency policy
Status: accepted · Applies to: all services and apps

## Adding
- New runtime deps require in the PR description: what it replaces owning,
  weekly downloads/maintenance glance, license, and install-script check.
- Banned without exception: packages replicating <20 lines of stdlib-able
  code. CI: a lint rule flags new manifest entries lacking the PR template.
- Internal scope @acme/* resolves ONLY from the private registry (CI verifies
  registry config) — dependency-confusion guard.

## Installing & reproducing
- Lockfile committed; CI uses npm ci; a CI job fails if `npm ci` produces a
  dirty git tree (lock/manifest drift).
- CI installs with --ignore-scripts; the few packages needing build scripts
  are allow-listed explicitly.

## Updating
- Renovate: patches/minors grouped weekly, auto-merge on green CI.
- Majors: individual PRs, changelog link required, owner assigned.
- Security: high/critical in the --omit=dev tree block merge (audit gate in
  CI); fix-or-override within 48h, overrides carry a removal ticket.

## Auditing & pruning
- Quarterly: `npm ls --all` review per service; deps unused (depcheck in CI,
  warning tier) get removal PRs.

## Enforcement summary (all CI, no vibes):
audit gate · ci-clean-tree gate · ignore-scripts + allow-list ·
registry pinning · depcheck · PR template lint.
Revisit: if the org adopts a proxy registry with quarantine (raises several
gates a level).

🎯 Key Takeaways

  • package.json declares intent (ranges, categories, scripts, engines); the lockfile records the exact resolved reality — commit it, review its diffs, and npm ci everywhere automated.
  • Semver: major breaks, minor adds, patch fixes; caret trusts it within a major (0.x excepted), and the lockfile exists because the trust is imperfect.
  • Categories are operational: runtime vs dev decides what ships (--omit=dev); peers declare "the app provides it"; misfiling fails in prod, not review.
  • npm scripts are the project's single task interface — thin invokers with node_modules/.bin on PATH; CI runs exactly what devs run.
  • Dependencies are trusted code with install-time execution: audit the shipped tree, gate criticals, restrict scripts, quarantine the brand-new, and hold the "cheaper than owning it" bar.
  • Workspaces make internal packages real without publishing; publishing adds "files", conditional exports, and CI-based provenance.
