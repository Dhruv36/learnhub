Exceptions & Nullability Basics

Exceptions are C#'s mechanism for exceptional, unrecoverable-at-this-level failures — not a general control-flow tool. Combined with nullable value types and pattern-matching-based null handling, this lesson covers how to fail predictably and handle absence-of-a-value without exceptions where possible.

try/catch/finally — the Mechanics

try {
    ProcessOrder(order);
} catch (ValidationException ex) {          // catch SPECIFIC exception types, most-specific first
    logger.LogWarning(ex, "Invalid order {OrderId}", order.Id);
    throw;                                   // re-throw preserving the original stack trace
} catch (Exception ex) {                     // a broad catch-all — use sparingly, at boundaries
    logger.LogError(ex, "Unexpected failure processing order");
    throw;
} finally {
    connection.Close();                      // ALWAYS runs — success, exception, or early return
}
// `throw;` (bare) preserves the original stack trace. `throw ex;` RESETS it, hiding where the error
// actually originated — a classic mistake that makes production debugging much harder.

Custom Exceptions — Carry Meaningful Information

public class InsufficientFundsException : Exception {
    public decimal Requested { get; }
    public decimal Available { get; }
    public InsufficientFundsException(decimal requested, decimal available)
        : base($"Requested {requested:C} but only {available:C} available")
    {
        Requested = requested;
        Available = available;
    }
}
// A custom exception type lets CALLERS catch specifically this failure mode and react (e.g., prompt
// for a different payment method) rather than parsing a generic exception's message string.

Exceptions Are for EXCEPTIONAL Cases, Not Control Flow

// ❌ Using exceptions for expected, common outcomes — slow (stack unwinding) and obscures intent:
try { return dictionary[key]; } catch (KeyNotFoundException) { return defaultValue; }

// ✅ Use the API designed for the "might not be there" case:
return dictionary.TryGetValue(key, out var value) ? value : defaultValue;
// TryXxx patterns (TryParse, TryGetValue) avoid exception overhead for outcomes that are NORMAL,
// not exceptional — reserve real exceptions for things that genuinely shouldn't happen.
⚠️ Exceptions are relatively expensive and should signal the unexpected. A missing dictionary key, a failed TryParse, an empty search result — these are normal program outcomes, not exceptional failures, and .NET's TryXxx APIs exist precisely so you don't pay exception overhead (or obscure your intent) for them. Reserve exceptions for genuine failures: a corrupted file, a network outage, a violated invariant.

Nullable Value Types (T?) vs Nullable Reference Types

int? age = null;                 // Nullable — a value type that CAN be null (unlike plain int)
if (age.HasValue) { Use(age.Value); }
int actual = age ?? 0;           // null-coalescing: use age's value, or 0 if null

// Nullable reference types (previous lesson) are a COMPILE-TIME annotation on reference types.
// Nullable value types (T?, i.e., Nullable) are a REAL runtime wrapper struct around a value type.
// Different mechanisms, same goal: make "this might not have a value" explicit and checked.

Interview Questions

Q1: What's the difference between `throw;` and `throw ex;` in a catch block?

Both re-throw the caught exception, but they behave very differently with respect to the stack trace, which is often critical information for debugging. A bare `throw;` re-throws the EXACT SAME exception object, preserving its original StackTrace property intact — so if you eventually log or inspect the exception further up the call chain, you can still see exactly where it originally occurred, deep inside whatever method first threw it. `throw ex;` instead throws the exception object as if it were a NEW exception originating at that line, which RESETS the StackTrace to start from the catch block itself — you lose all information about where the failure actually happened, which is enormously unhelpful when debugging a production incident from logs. The rule of thumb: if you're re-throwing the same exception after logging or partial handling, always use bare `throw;`; only use `throw someNewException;` when you're deliberately wrapping the original in a different, more meaningful exception type (and even then, pass the original as the innerException parameter so it's not lost).

Q2: Why shouldn't exceptions be used for normal control flow, and what's the alternative?

Exceptions in .NET are relatively expensive to throw and catch — throwing one captures a stack trace and unwinds the call stack, which has real, measurable performance cost compared to a simple conditional check, and using them for outcomes that occur routinely (a dictionary key that's often absent, user input that often fails to parse as a number) turns a normal, expected code path into an expensive one. It also obscures the code's intent: catching KeyNotFoundException to implement "return a default if missing" buries a completely ordinary branch of logic inside exception-handling machinery that readers reasonably expect to signal something has gone wrong. .NET's TryXxx pattern exists specifically to avoid this: methods like Dictionary.TryGetValue, int.TryParse, and IEnumerable.TryGetNonEnumeratedCount return a bool indicating success and use an `out` parameter for the result on success, with no exception thrown at all for the "not found / failed" case — this is both faster and clearer, since the calling code reads as a normal conditional rather than exception-handling logic. The general principle is to reserve real exceptions for genuinely exceptional situations — a corrupted file, a network failure, a violated invariant that indicates a bug — not for outcomes that are simply one of several normal possibilities.

Q3: Explain the difference between the null-conditional (?.), null-coalescing (??), and null-forgiving (!) operators.

These serve three distinct purposes around handling potentially-null values. The null-conditional operator (?.) safely accesses a member or calls a method only if the left-hand operand is non-null; if it IS null, the whole expression short-circuits and evaluates to null immediately without throwing and without evaluating anything further in the chain — so `user?.Address?.City` returns null safely at whichever point in the chain is null, rather than throwing a NullReferenceException, and it chains cleanly through deep object graphs without nested if-checks. The null-coalescing operator (??) provides a fallback value when the left operand is null: `name ?? "Unknown"` evaluates to name if it's non-null, or the string "Unknown" if it is null — it's commonly combined with ?. as `user?.Address?.City ?? "N/A"` to get a safe default for a potentially-null deep property. The null-forgiving operator (!) is fundamentally different in kind: it's a purely COMPILE-TIME instruction to the nullable-reference-types analyzer telling it "trust me, this value is not actually null here, suppress the warning" — it performs NO runtime check or conversion whatsoever, so if you're wrong, you still get a genuine NullReferenceException at runtime; it should only be used when you have verified non-nullness through logic the compiler simply can't see (like an assertion earlier in the method), never as a way to silence warnings you haven't actually reasoned about.

🏋️ Practice: write a method that catches a specific exception, logs it, and re-throws with bare throw — then break it by changing to throw ex and compare the resulting stack traces; replace a try/catch around a dictionary lookup with TryGetValue; create a custom exception type carrying structured data and catch it specifically; chain ?. and ?? through a multi-level nullable object graph; deliberately misuse the ! operator and trigger a runtime NullReferenceException to see the risk firsthand.

🎯 Key Takeaways

  • Catch specific exception types before broad ones; use bare `throw;` to re-throw and preserve the original stack trace (never `throw ex;`).
  • Exceptions are for genuinely exceptional failures, not expected outcomes — use TryXxx patterns (TryGetValue, TryParse) for normal "might not be there" cases.
  • Custom exceptions carry structured, catchable information beyond a generic message string.
  • T? (Nullable<T>) wraps value types with a real runtime null state; nullable reference types are compile-time-only annotations — different mechanisms, same goal.
  • Know ?. (safe chaining, short-circuits on null), ?? (fallback value), ??= (assign-if-null), and ! (compile-time-only, no runtime safety) — and use ! sparingly and deliberately.
