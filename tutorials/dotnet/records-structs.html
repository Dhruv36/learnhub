Records, Structs & Equality Object-Oriented C#

What you'll master: the equality model (reference vs value, and the Equals/GetHashCode/== triad), records as value-semantic reference types with with-expressions, structs and when a value type is the right choice, record struct and readonly struct, and the design decision of class vs record vs struct. Prerequisites: Types (value/reference) and Classes. This lesson ties the type system together — and equality bugs are subtle, common, and exactly the kind interviewers probe.

Part 1 — The Equality Model: Reference vs Value

// REFERENCE EQUALITY (default for classes): "the same object?"
public class Point { public int X; public int Y; }
var a = new Point { X = 1, Y = 2 };
var b = new Point { X = 1, Y = 2 };
a == b            // FALSE — different objects, same contents
a.Equals(b)       // FALSE — default Equals is reference equality for classes
ReferenceEquals(a, b)  // FALSE — explicitly asks "same object"

// VALUE EQUALITY (what you usually WANT for data): "the same contents?"
// string and records give it automatically:
"hi" == "hi"      // TRUE — string overrides == for value equality
(1, 2) == (1, 2)  // TRUE — tuples have value equality

// The THREE things that must agree — the equality triad:
//   1. Equals(object)     — the semantic definition of equality
//   2. GetHashCode()      — MUST return the same hash for equal objects
//   3. == / !=            — the operators (should match Equals for value types)
// Break the contract between #1 and #2 and DICTIONARIES/HASHSETS SILENTLY LOSE DATA
// (an object stored under one hash can't be found if its hash disagrees with equality).

Equality is one of the most bug-prone corners of any type system because there are two reasonable meanings — "same object" (reference/identity) and "same contents" (value) — and the default for classes is identity, which is usually not what you want for data. The triad is the load-bearing rule: Equals, GetHashCode, and == must be consistent, and the specific contract that bites hardest is equal objects must have equal hash codes. Violate it and hash-based collections (Dictionary, HashSet) silently misbehave: you store an object, its contents match a lookup key, but its hash lands it in a different bucket, so the collection reports "not found" and you get phantom duplicates or lost data — a bug that's maddening because nothing throws. Records exist largely to make getting this triad right the default instead of a hand-written chore.

Part 2 — Records: Value Semantics for Reference Types

// One line generates the WHOLE value-equality apparatus + immutability + more:
public record Person(string Name, int Age);

var a = new Person("Ada", 36);
var b = new Person("Ada", 36);
a == b            // TRUE — records have VALUE equality out of the box
a.Equals(b)       // TRUE — and a consistent GetHashCode(), for free
Console.WriteLine(a);  // Person { Name = Ada, Age = 36 } — auto ToString() too

// WITH-EXPRESSIONS — non-destructive mutation (immutable update, functional style):
var older = a with { Age = 37 };     // NEW Person, a is unchanged (a.Age still 36)
// this is the immutable-update pattern from functional-patterns / angular-signals,
// baked into the language.

// DECONSTRUCTION — records auto-generate Deconstruct:
var (name, age) = a;                 // name = "Ada", age = 36

// What 'record' actually generates for you:
//   value Equals + GetHashCode (over ALL properties)
//   == / != operators
//   ToString() showing every property
//   a copy constructor + 'with' support
//   Deconstruct (for positional records)
//   init-only properties (immutable by default)

// Records still support inheritance, methods, computed properties, validation:
public record Money(decimal Amount, string Currency)
{
    public Money   // validate in the ctor body
    {
        get => field;
        init => field = value; // (illustrative) — real validation goes in a body block:
    }
    public string Formatted => $"{Amount:N2} {Currency}";     // computed member
}

A record is a reference type (class) that the compiler outfits with value semantics: automatic value-based Equals/GetHashCode/== computed over all its properties, a readable ToString, init-only immutability, and — the standout feature — with-expressions for non-destructive updates (a with { Age = 37 } produces a new copy, leaving the original untouched). This is the immutable-update discipline promoted to a language feature, and it makes records the ideal type for DTOs, domain value objects, messages, and any "bag of data compared by contents." They're not stripped-down classes — records support methods, computed properties, validation, and inheritance — they simply flip the defaults from "mutable, identity-compared" to "immutable, value-compared," which is the right default for the large fraction of your types that are fundamentally data.

Part 3 — Structs: Value Types You Design

// A struct is a VALUE TYPE (types.html): copied on assignment, no heap allocation,
// lives inline in its container. You use it to make your OWN value types.
public struct Point3D
{
    public double X, Y, Z;
    public double Magnitude => Math.Sqrt(X*X + Y*Y + Z*Z);
}
var p = new Point3D { X = 1, Y = 2, Z = 2 };
var q = p;                          // COPY — q is independent of p (value semantics)
q.X = 99;                           // p.X is still 1

// Structs get value equality by DEFAULT (unlike classes) — but the default is SLOW
// (reflection-based) and boxes; override Equals/GetHashCode/== for real use, or...
// use a 'record struct' (Part 4) to get it generated correctly.

// WHEN a struct is the right choice (the guidance is narrow):
//  ✓ small (roughly ≤ 16-24 bytes — a few fields)
//  ✓ logically a single VALUE (a point, a color, a money amount, a timestamp)
//  ✓ immutable (mutable structs are a footgun — Part 6)
//  ✓ short-lived / allocated in tight loops where avoiding heap pressure matters
//  ✗ large (copying cost dominates)
//  ✗ needs reference identity or is passed around and mutated
//  ✗ has an inheritance hierarchy (structs can't inherit)

// The BCL's structs model exactly these: int, double, DateTime, TimeSpan, Guid,
// decimal, and Span<T> — all small, value-like, immutable.

A struct is your tool for designing value types — types with the copy semantics and inline, allocation-free storage of int or DateTime, covered mechanically back in Types. The value proposition is performance: no heap allocation, no GC pressure, and (in arrays) contiguous cache-friendly memory — which is why the BCL's small value-like concepts (DateTime, Guid, TimeSpan) are structs. But the guidance is deliberately narrow: structs pay off only when they're small, genuinely value-like, and immutable. A large struct is worse than a class (every assignment copies all of it), a struct that needs identity or gets mutated through references is a bug magnet, and structs can't participate in inheritance. The professional default remains class (or record) unless you have a specific, measured reason to reach for a struct — and even then, prefer record struct so equality is generated correctly rather than left to the slow reflection-based default.

Part 4 — record struct & readonly struct

// record struct — a VALUE type with records' generated equality + ToString + with:
public readonly record struct Point(int X, int Y);
// 'readonly' → immutable (all fields readonly); 'record' → value equality generated
// correctly and FAST (not the slow reflection default); 'struct' → stack/inline, no alloc.
// This is the modern sweet spot for small immutable value objects.

var p = new Point(1, 2);
var q = new Point(1, 2);
p == q                              // TRUE — generated value equality, no boxing
var moved = p with { X = 5 };       // with-expression on a struct — new copy

// readonly struct (without record) — immutability guarantee for hand-written structs:
public readonly struct Temperature
{
    public double Celsius { get; }              // all members must be readonly
    public Temperature(double c) => Celsius = c;
    public double Fahrenheit => Celsius * 9/5 + 32;
}
// The compiler ENFORCES no mutation, which enables optimizations (defensive copies
// elided) and prevents the mutable-struct footgun (Part 6).

// The domain-wrapper pattern (types.html) is a readonly record struct:
public readonly record struct OrderId(int Value);
public readonly record struct CustomerId(int Value);
// zero-cost (same size as int), value equality, immutable, and MISUSE won't compile —
// the ideal shape for strongly-typed ids.

// ref struct — a special struct that can ONLY live on the stack (Span<T>, memory.html):
// can't be boxed, can't be a class field, can't be captured in a lambda/async — the
// compiler enforces stack-only lifetime for zero-allocation memory views.

record struct is where records and structs combine into the modern sweet spot for small value objects: readonly record struct Point(int X, int Y) gives you value semantics correctly generated (fast, non-boxing equality — far better than the struct default's slow reflection-based comparison), immutability, with-expressions, and inline allocation-free storage, all in one line. This is the ideal shape for the domain-wrapper pattern from Types — readonly record struct OrderId(int Value) is a zero-overhead strongly-typed id. readonly struct (without record) is the immutability guarantee for hand-written structs, and it matters beyond correctness: the compiler can elide defensive copies for readonly structs, so the immutability annotation is also a performance annotation. And ref struct (the Span<T> family) is the extreme: a struct the compiler pins to the stack for zero-allocation memory access, the subject of the Memory lesson.

Part 5 — Hand-Writing Equality (When You Must)

// Records/record structs generate this for you — but you'll SEE hand-written equality in
// existing code, and occasionally need it for a class with custom equality semantics:
public class Money : IEquatable<Money>
{
    public decimal Amount { get; }
    public string Currency { get; }
    public Money(decimal amount, string currency) => (Amount, Currency) = (amount, currency);

    // 1. The typed Equals (IEquatable<T> — avoids boxing, the preferred entry point):
    public bool Equals(Money? other) =>
        other is not null && Amount == other.Amount && Currency == other.Currency;

    // 2. The object.Equals override — MUST delegate to the typed one consistently:
    public override bool Equals(object? obj) => Equals(obj as Money);

    // 3. GetHashCode — MUST agree with Equals; HashCode.Combine does it right:
    public override int GetHashCode() => HashCode.Combine(Amount, Currency);
    //    ^ same fields as Equals uses, always. Use HashCode.Combine, never hand-roll
    //      (bad hash functions → collisions → O(n) dictionary lookups).

    // 4. Operators — delegate to Equals so all three agree:
    public static bool operator ==(Money? a, Money? b) =>
        a is null ? b is null : a.Equals(b);
    public static bool operator !=(Money? a, Money? b) => !(a == b);
}
// The RULES: Equals is reflexive/symmetric/transitive; equal objects have equal hashes;
// hash uses the SAME fields as Equals; null-safe throughout. Getting ANY of these wrong
// corrupts hash-based collections. This is exactly the tedium records eliminate.

You'll rarely write equality by hand in new code — that's precisely what records automate — but you'll read it constantly and occasionally need it for a class requiring custom equality semantics, so knowing the correct shape matters. The four parts must all agree: implement IEquatable<T> for a typed, non-boxing Equals; override object.Equals to delegate to it; override GetHashCode using the same fields via HashCode.Combine (never a hand-rolled hash — bad hash functions cause collisions that degrade dictionary lookups to O(n)); and define ==/!= to delegate to Equals, null-safely. The invariants — equal objects have equal hashes, hash uses the same fields as equality, and equality is reflexive/symmetric/transitive — are non-negotiable because hash-based collections depend on them. The sheer amount of careful, error-prone code here is the strongest argument for reaching for a record instead.

Part 6 — Deep Dive: class vs record vs struct, and the Mutable-Struct Trap

// THE DECISION FRAMEWORK:
// ┌─────────────────────────────────────────────────────────────────────────┐
// │ Is it fundamentally DATA compared by its contents (DTO, value object,    │
// │ message, event)?                                                          │
// │    YES → RECORD (class-based). Value equality, immutable, with-expressions.│
// │                                                                            │
// │ Is it a small (~≤24B), immutable, value-like thing in a hot/allocation-   │
// │ sensitive path (point, id-wrapper, coordinate)?                            │
// │    YES → READONLY RECORD STRUCT. Value semantics + no heap allocation.     │
// │                                                                            │
// │ Does it have IDENTITY, mutable state, behavior/invariants, a lifecycle,   │
// │ or an inheritance hierarchy (entities, services, controllers)?            │
// │    YES → CLASS. Reference semantics, encapsulation (oop.html).             │
// └─────────────────────────────────────────────────────────────────────────┘
// Default when unsure: class for entities/services, record for data. Structs are the
// deliberate performance exception, not a default.

// THE MUTABLE-STRUCT TRAP — why "immutable" is in every struct guideline:
public struct MutablePoint { public int X, Y; }
var list = new List<MutablePoint> { new() { X = 1 } };
list[0].X = 99;              // ✗ COMPILE ERROR — list[0] returns a COPY; mutating it is
                            //   pointless, so C# forbids it. You'd have to:
var temp = list[0]; temp.X = 99; list[0] = temp;   // copy out, mutate, copy back (!)

struct Counter { public int Count; public void Increment() => Count++; }
readonly Counter _c = new();
_c.Increment();             // silently operates on a DEFENSIVE COPY in some contexts —
                            //   the original _c is unchanged. Baffling.
// This is why the rule is ABSOLUTE: make structs readonly (immutable). Mutable value
// types interact with copying in ways that produce silent, surprising no-ops. Every
// struct footgun traces back to mutability + copy semantics colliding.

The class/record/struct decision comes down to what the type fundamentally is. If it's data compared by contents — a DTO, a domain value object, a message, an event — use a record (value equality and immutability are exactly right). If it's a small immutable value-like thing in a performance-sensitive path, use a readonly record struct (value semantics with no heap allocation). If it has identity, mutable state, behavior with invariants, a lifecycle, or an inheritance hierarchy — entities, services, aggregates — use a class (reference semantics and the encapsulation from the OOP lesson). The mutable-struct trap explains why "immutable" appears in every struct guideline: because structs are copied freely, mutating one through an indexer, a property, or a readonly field operates on a copy — the change silently evaporates, and the compiler either forbids it or lets it fail confusingly. Immutable structs sidestep the entire category, which is why the modern advice is: if you must use a struct, make it readonly.

Senior framing: equality and value semantics are a favorite interview area because the bugs are subtle and the correct model is non-obvious. The set-piece questions: "what happens if you override Equals but not GetHashCode?" (hash-based collections silently break — the object hashes to one bucket but equals-matches objects in another, so lookups fail, and this is a real production bug that never throws); "when would you use a struct over a class?" (small, immutable, value-like, allocation-sensitive — and a candidate who can't name the constraints is guessing); and "why are mutable structs discouraged?" (copy semantics make mutation silently operate on copies). The record era changed the practical answer to most of this — "use a record and stop hand-writing equality" is correct and current — but the underlying model still gets probed because understanding why records generate what they generate is what separates using the feature from understanding it. In review, watch for: hand-written GetHashCode that doesn't match Equals's fields (or hashes on mutable state — an object whose hash changes after insertion is lost in a dictionary), mutable structs, large structs copied through hot paths, and classes used as data bags where a record would eliminate boilerplate and bugs at once. The unifying principle: value semantics for data, reference semantics for identity — and modern C# gives you records and record structs to express both correctly with almost no code.

Interview Questions

Q1 (Beginner): What's the difference between reference equality and value equality, and what's the default for a class?

Reference (identity) equality asks "are these the same object in memory?"; value equality asks "do these objects have the same contents?". The default for a class is reference equality — two separately-constructed objects with identical field values are not equal (== and Equals both return false), because they're different objects. That's usually not what you want for data, which is why you either implement value equality by hand or, far more simply, use a record, which generates value equality automatically. string is the notable class that overrides to value equality, and structs default to value equality (though the default implementation is slow).

Q2 (Beginner): What does a record generate for you, and when should you use one?

A record generates value-based Equals and GetHashCode (over all properties), ==/!= operators, a readable ToString, init-only immutable properties, a copy constructor with with-expression support, and (for positional records) a Deconstruct method. Use a record for anything that's fundamentally data compared by its contents: DTOs, domain value objects, messages, events, configuration, query results. It flips the class defaults from mutable-and-identity- compared to immutable-and-value-compared, which is the right default for the large fraction of types that are essentially data — and it eliminates the error-prone equality boilerplate entirely.

Q3 (Mid): Explain the Equals/GetHashCode contract and what breaks if you violate it.

The contract: (1) if two objects are equal per Equals, they must return the same GetHashCode; (2) GetHashCode should use the same fields Equals uses; (3) equal-object hash consistency must hold for the object's lifetime (so hash over immutable fields). What breaks: hash-based collections (Dictionary, HashSet) place objects in buckets by hash code, then compare with Equals within the bucket. If two equal objects have different hashes, they land in different buckets, so a lookup for one won't find the other — the collection reports "not contained" for something it contains, allows duplicates it should reject, and generally loses data. Nothing throws, which makes it a nasty silent bug. Overriding Equals without GetHashCode (leaving the identity-based default) is the classic form; hashing over mutable state that changes after insertion is the other.

Q4 (Mid): When is a struct the right choice over a class, and what are the constraints?

A struct is right when the type is small (roughly ≤16–24 bytes — a few fields), logically a single immutable value (a point, color, money amount, id wrapper, coordinate), and lives in allocation-sensitive paths where avoiding heap pressure and GC matters (tight loops, large arrays where inline contiguous storage is cache-friendly). Constraints/anti-indications: don't use structs for large types (copying cost dominates every assignment and parameter pass), for types needing reference identity or that get mutated through references (mutation hits copies), or where inheritance is needed (structs can't inherit). Make them readonly to avoid the mutable-struct trap, and prefer record struct so equality is generated correctly. The BCL's structs (int, DateTime, Guid, decimal, Span<T>) all fit this profile; the default for your own types remains class/record unless you have a measured performance reason.

Q5 (Senior): Walk through the decision: you're modeling an OrderId, an Order, and an OrderCreatedEvent. What type for each and why?

OrderId → readonly record struct OrderId(int Value): it's a small, immutable value-like wrapper, and making it a distinct type prevents mixing it with other int ids (compile-time safety from Types). Struct because it's tiny and allocation-sensitive (ids are everywhere); record for correct value equality (two OrderId(5) are equal); readonly for immutability. Order → class: it's an entity with identity (two orders with identical contents are still different orders — identity is defined by its OrderId, not its data), mutable state over a lifecycle (Draft→Submitted→Shipped), behavior and invariants (the encapsulation from the OOP lesson), and possibly persistence tracking. Value equality would be actively wrong — you don't want two orders considered "the same" because their fields momentarily match. OrderCreatedEvent → record: it's an immutable data message compared by contents, a perfect fit for value equality, immutability, and with; it carries data (order id, timestamp, customer) with no behavior or identity of its own. The through-line: value semantics for values and messages (record/record struct), reference semantics for entities with identity and lifecycle (class) — and getting this split right is a core domain-modeling skill.

Q6 (Senior): A dictionary keyed by a custom object intermittently "loses" entries — ContainsKey returns false for keys that were added. Diagnose the likely causes and fixes.

This is the classic equality/hash contract failure, and there are two prime suspects. (1) Missing or inconsistent GetHashCode: the key type overrides Equals (so contents-equal keys should match) but not GetHashCode, so it uses the identity-based default — a lookup key with the same contents but different identity computes a different hash, lands in a different bucket, and is never compared against the stored key. Fix: override GetHashCode over the same fields as Equals (or make the key a record). (2) Mutable key: the key's hash is computed over fields that were changed after insertion, so the stored entry now hashes to a bucket different from where it lives — it's orphaned. Fix: use immutable keys (records/readonly record structs are ideal), or never mutate a key after it's in a collection. A third, subtler cause: the key implements value equality but the caller passes a different type (e.g. a derived type or a boxed struct) whose Equals/hash differs. Diagnosis approach: check whether the key type is a record (usually rules it out), then verify Equals and GetHashCode use identical, immutable fields, then audit for post-insertion mutation. The permanent fix is almost always "make the key an immutable record," which is exactly why records exist.

3 (Medium): Hand-write correct equality for a class GeoCoord { double Lat, Lng } implementing IEquatable<T>, then show the record version that replaces it.
Solution
// Hand-written — all four parts, consistent:
public class GeoCoord : IEquatable<GeoCoord>
{
    public double Lat { get; }
    public double Lng { get; }
    public GeoCoord(double lat, double lng) => (Lat, Lng) = (lat, lng);

    public bool Equals(GeoCoord? other) =>
        other is not null && Lat == other.Lat && Lng == other.Lng;
    public override bool Equals(object? obj) => Equals(obj as GeoCoord);
    public override int GetHashCode() => HashCode.Combine(Lat, Lng);   // SAME fields as Equals
    public static bool operator ==(GeoCoord? a, GeoCoord? b) =>
        a is null ? b is null : a.Equals(b);
    public static bool operator !=(GeoCoord? a, GeoCoord? b) => !(a == b);
}

// The record replacement — identical behavior, one line, impossible to get wrong:
public record GeoCoord(double Lat, double Lng);
// This is the punchline of the lesson: unless you need CUSTOM equality semantics (e.g.
// comparing coordinates within a tolerance), the record is strictly better — same value
// equality, plus ToString, with-expressions, and no chance of a hash/equals mismatch.
4 (Medium): Demonstrate the GetHashCode/Equals mismatch bug: a class that overrides Equals but not GetHashCode, used as a dictionary key, "losing" an entry. Then fix it.
Solution
// BUGGY — Equals overridden, GetHashCode is the identity default:
public class BadKey
{
    public string Value { get; init; } = "";
    public override bool Equals(object? o) => o is BadKey k && k.Value == Value;
    // ✗ NO GetHashCode override → uses reference-identity hash
}
var dict = new Dictionary<BadKey, int>();
dict[new BadKey { Value = "a" }] = 1;
Console.WriteLine(dict.ContainsKey(new BadKey { Value = "a" }));  // FALSE!
// The lookup key is Equals-equal but has a DIFFERENT hash (different object) → lands in a
// different bucket → never compared against the stored entry → "not found". Silent bug.

// FIXED — hash and equality agree:
public class GoodKey
{
    public string Value { get; init; } = "";
    public override bool Equals(object? o) => o is GoodKey k && k.Value == Value;
    public override int GetHashCode() => Value.GetHashCode();   // SAME field as Equals
}
// Or, obviously, just:  public record GoodKey(string Value);   ← does both correctly.
var good = new Dictionary<GoodKey, int>();
good[new GoodKey { Value = "a" }] = 1;
Console.WriteLine(good.ContainsKey(new GoodKey { Value = "a" }));  // TRUE ✓
5 (Hard): Show the mutable-struct trap three ways (indexer mutation, property-returned struct, readonly field), and explain why readonly record struct prevents all three.
Solution
public struct MutablePoint { public int X, Y; }

// TRAP 1 — mutation through a collection indexer:
var list = new List<MutablePoint> { new() { X = 1 } };
// list[0].X = 99;   // ✗ COMPILE ERROR: list[0] returns a COPY; C# forbids mutating it
//                   //   (it would be a silent no-op, so the compiler stops you)

// TRAP 2 — mutation through a property that returns a struct:
class Container { public MutablePoint Point { get; set; } }
var c = new Container();
// c.Point.X = 5;    // ✗ COMPILE ERROR: the getter returns a COPY — mutating it is useless
c.Point = c.Point with { }; // you must copy out, change, assign back (if it were mutable)

// TRAP 3 — mutating method on a readonly field operates on a defensive COPY:
struct Counter { public int Count; public void Inc() => Count++; }
class Holder { public readonly Counter C = new(); public void Bump() => C.Inc(); }
var h = new Holder();
h.Bump(); h.Bump();
Console.WriteLine(h.C.Count);   // 0! — Inc() ran on a defensive copy of the readonly field;
                               //   the original never changed. Baffling silent no-op.

// WHY readonly record struct prevents all three: it's IMMUTABLE — there are no mutating
// members to call and no settable fields, so none of these mutation attempts exist. You
// change a value only by producing a NEW one (with-expression), which is always assigned
// somewhere explicit — no hidden copies, no silent no-ops. The trap is fundamentally
// "mutation + value-copy semantics collide"; removing mutation removes the collision.
public readonly record struct Point(int X, int Y);   // safe by construction

// Account (aggregate) → class
//   An ENTITY with identity (defined BY its AccountId, not its balance), mutable state over
//   a lifecycle, invariants (no overdraft), and behavior (Deposit/Withdraw). Value equality
//   would be WRONG — two accounts aren't "the same" because their balances momentarily
//   match; identity ≠ contents. Equals should compare AccountId (or use default reference
//   equality and rely on the id).

// MoneyDeposited (event) → record
//   An immutable fact/message compared by contents; value equality, ToString, and with are
//   all natural. record (class-based, not struct) because events carry several fields and
//   are stored/serialized/passed around — no allocation-sensitivity that justifies a struct.
//   Subtlety: events must be immutable (an event is a historical fact — records enforce it).

// AccountSummary (read model DTO) → record
//   Pure data projection compared by contents; value equality is convenient for caching/
//   change-detection, immutability is right for a read model. Anemic-by-design (oop.html) —
//   a DTO SHOULD be a data bag, so a record is ideal.

// The pattern across all five: value semantics (record / readonly record struct) for values,
// ids, messages, and DTOs; reference semantics (class) for the one thing with identity,
// mutable state, and behavior — the aggregate. Get the entity-vs-value split right and the
// type choices fall out; get it wrong (e.g. a value-equal Account) and you get subtle bugs
// where distinct entities collapse into one.

🎯 Key Takeaways

  • Two equalities exist — reference (identity, the class default) and value (contents) — and the triad Equals/GetHashCode/== must agree: equal objects must have equal hashes, or hash collections silently break.
  • Records are reference types with generated value semantics: value equality/hash, ToString, init-only immutability, and with-expressions — the right default for DTOs, value objects, messages, and events.
  • Structs are value types you design — use them only when small, immutable, value-like, and allocation-sensitive; class/record is the default, structs the measured exception.
  • readonly record struct is the modern sweet spot for small immutable value objects (correct fast equality + no heap allocation); it's the ideal shape for strongly-typed id wrappers.
  • Hand-written equality (four consistent parts via IEquatable<T> + HashCode.Combine) is error-prone tedium — records exist to eliminate it; you'll mostly just read it.
  • Decision: value semantics (record/record struct) for data/values/messages, reference semantics (class) for entities with identity, lifecycle, and behavior — and always make structs immutable to avoid the copy-mutation trap.
