Control Flow & Operators Foundations

What you'll master: C#'s decision and loop constructs with their idiomatic modern forms, the null-handling operator family (?., ??, ??=) that pervades real code, switch statements vs switch expressions, loop mechanics (foreach's contract, break/continue), and guard-clause style — how professionals shape flow for readability. Prerequisites: Types & Variables. Nothing here is exotic — which is exactly why fluency and idiom matter: this is the code you write every hour.

Part 1 — Branching: if, else & the Guard-Clause Idiom

// The arrow-of-doom (nested happy path) vs GUARD CLAUSES (flat happy path):

// ✗ nesting grows with every rule:                // ✓ fail fast, then proceed flat:
if (order != null)                                 if (order is null) return Reject("no order");
{                                                  if (order.Items.Count == 0) return Reject("empty");
    if (order.Items.Count > 0)                     if (!customer.CanOrder) return Reject("blocked");
    {
        if (customer.CanOrder)                     return Approve(order);      // the POINT of the
        {                                                                       // method, unindented
            return Approve(order);
        }
    }
}

// Conditions are bool — NO truthiness (unlike JS/Python):
if (items.Count > 0)     // ✓
if (items.Count) { }     // ✗ compile error: int is not bool — a whole bug class refused
if (name is not null && name.Length > 2)   // && short-circuits: Length never runs on null

// Ternary for VALUE selection (not action selection):
var label = count == 1 ? "1 item" : $"{count} items";

C# takes a side on flow style: conditions must be genuine bools (no truthy/falsy — the compiler rejects if (count), killing the accidental-assignment and truthiness-surprise bugs other languages carry), and the community idiom is guard clauses — validate and bail early, keeping the method's real work at indentation zero. Short-circuit evaluation (&&/|| stop as soon as the answer is known) isn't trivia — it's what makes x is not null && x.Prop a safe pattern, and it means side effects in later conditions may never run (a review point).

Part 2 — The Null-Operator Family

string? nickname = user.Nickname;                 // may be null (exceptions-nullability.html)

// ?.  null-conditional: short-circuits to null instead of throwing:
int? len = nickname?.Length;                      // null if nickname is null — note int?
customer?.Orders?.Clear();                        // any null link stops the chain, no-op

// ??  null-coalescing: "if null, use this instead":
string display = nickname ?? user.FullName ?? "Guest";   // first non-null wins

// ??=  assign only if currently null (lazy defaults):
_cache ??= LoadCache();

// Together — the defensive-read idiom you'll type daily:
var city = order?.Customer?.Address?.City ?? "unknown";

// ?. with methods and events:
OnChanged?.Invoke(this, args);                    // the thread-safe event-raise idiom
list?.Add(item);                                  // no-op when list is null

// Careful: ?. converts "unexpected null" into SILENCE. For nulls that indicate BUGS,
// let it throw (or throw explicitly):
var customer = order.Customer
    ?? throw new InvalidOperationException("Order must have a customer");

These three operators are C#'s answer to null being both common and dangerous: ?. makes null-safe traversal one character instead of nested ifs, ?? expresses fallbacks declaratively, ??= handles lazy initialization. The senior nuance is the last block: null-conditionals absorb nulls, which is right when null is a legitimate absence (optional nickname) and wrong when null means a broken invariant — there, silence hides bugs, and ?? throw converts the mystery-NRE-later into a named failure now. The nullability lesson gives the compiler's side of this contract; these operators are the expression-level toolkit.

Part 3 — switch: Statement and Expression

// STATEMENT form — for executing side effects per case:
switch (command)
{
    case "start": Start(); break;                 // break is MANDATORY (no C-style fallthrough)
    case "stop":  Stop();  break;
    case "pause" or "hold": Pause(); break;       // or-patterns combine cases
    default: Console.WriteLine($"Unknown: {command}"); break;
}

// EXPRESSION form (C# 8+) — for COMPUTING a value; the modern default:
decimal shipping = zone switch
{
    "EU"           => 4.99m,
    "US" or "CA"   => 9.99m,
    "APAC"         => 14.99m,
    _              => 24.99m,                     // _ = default arm
};

// With guards (when) and ranges:
string band = score switch
{
    < 0 or > 100   => throw new ArgumentOutOfRangeException(nameof(score)),
    >= 90           => "A",
    >= 75           => "B",
    >= 60           => "C",
    _               => "F",
};
// Arms are checked TOP-DOWN; the compiler warns when arms are unreachable or the
// switch doesn't cover all inputs (exhaustiveness) — this is pattern matching's
// front door; the full pattern language gets its own lesson (pattern-matching.html).

The two switches divide by intent: statement for "do different things", expression for "compute one value" — and the expression form wins wherever it fits: it's exhaustiveness-checked (the compiler tells you about missed cases), it can't fall through or forget a break, each arm is a value (no mutation into a temp), and it composes with => bodies and ternaries. C# refusing silent fallthrough (every case must end with break/return/throw) removed one of C's most famous bug families while keeping or-patterns for the legitimate multiple-labels-one-body case.

Part 4 — Loops: for, while, do & foreach's Contract

for (int i = 0; i < items.Count; i++) { }     // index needed / stepping / mutation by index
while (retries < 3 && !success) { }            // condition-driven, may run zero times
do { input = Prompt(); } while (input is null); // must run at least once (input loops)

// FOREACH — the default for "each element":
foreach (var order in orders)
{
    Process(order);
    // order = other;      ✗ compile error: the iteration variable is readonly
}
// The contract: foreach works on anything with GetEnumerator() — arrays, List<T>,
// Dictionary (KeyValuePair<K,V>!), strings (chars), and every IEnumerable<T>
// including LAZY LINQ sequences (linq.html). Each iteration gets Current + MoveNext().

// The two foreach traps:
foreach (var o in orders)
    if (o.Cancelled) orders.Remove(o);         // ✗ InvalidOperationException:
                                               //   collection modified during enumeration
orders.RemoveAll(o => o.Cancelled);            // ✓ the idiomatic fix (or iterate a copy)

// break / continue / labeled escape:
foreach (var row in rows)
{
    if (row.Skip) continue;                    // next iteration
    if (row.Fatal) break;                      // leave the loop
}
// no labeled break in C# — for nested loops, extract a method and RETURN (cleaner anyway)

foreach is the workhorse, and its two rules repay knowing precisely: the loop variable is read-only (assigning it is a compile error — mutate through the collection, not the cursor), and the collection must not change during enumeration — the classic remove-while-iterating crash, fixed idiomatically with RemoveAll, iterating a snapshot (ToList()), or collecting removals first. The deeper contract — foreach consumes enumerators, and any IEnumerable<T> (including lazily-evaluated LINQ queries) qualifies — becomes central in the LINQ lesson, where "the query runs when foreach pulls it" explains a whole family of surprises.

Part 5 — Operators Worth Knowing Cold

// Comparison & equality (full story in records-structs.html):
==  !=  <  >  <=  >=
a == b        // value types: data equality. reference types: IDENTITY (same object) —
              // except string and records, which define VALUE equality. Know your type!
Equals / ReferenceEquals   // explicit versions when you need to be unambiguous

// is / as — type tests (pattern matching's simplest forms):
if (shape is Circle c) UseRadius(c.Radius);    // test + cast + name in one
var text = obj as string;                       // null if not a string (vs cast: throws)

// Arithmetic corner cases you WILL meet:
7 / 2      // 3      integer division truncates (types.html)
7 % 2      // 1      remainder — and -7 % 2 is -1 (sign follows dividend!)
x++ vs ++x // post/pre increment — in expressions they differ; as statements identical.
           // House style: use them as STATEMENTS only; clever inline ++ is review bait.

// Compound assignment: += -= *= /= %= &= |= ??=  — all short for x = x op y

// Range & index operators (arrays, strings, spans):
var last = items[^1];          // ^1 = one from the end
var firstThree = items[..3];   // ranges slice: [start..endExclusive]
var tail = items[2..];         // — full slicing story with spans in memory.html

Two entries here are genuinely load-bearing. First, =='s meaning depends on the operands: data for value types, identity for reference types, with string/records as deliberate value-equality exceptions — comparing two List<int> with == compiles and answers a question you probably didn't mean to ask. Second, is-with-a-pattern (obj is Circle c) has replaced the old test-then-cast dance entirely; as survives for "null means not-that-type" flows. The index/range operators (^, ..) are modern idiom worth adopting immediately — they read as intent and power spans later.

Part 6 — Deep Dive: Shaping Flow for Readability

// The professional refactors — same logic, different maintenance cost:

// 1. Boolean expressions BEAT boolean branching:
if (age >= 18) return true; else return false;       // ✗ ceremony
return age >= 18;                                     // ✓

// 2. Name complex conditions:
if (order.Total > 1000m && customer.Tier == Tier.Gold && !customer.OnHold) …   // ✗ inline riddle
bool qualifiesForExpressReview = order.Total > 1000m
    && customer.Tier == Tier.Gold && !customer.OnHold;
if (qualifiesForExpressReview) …                      // ✓ the condition documents itself

// 3. Data BEATS branching — the lookup refactor (strategy, dotnet edition):
var discount = tier switch                            // ✓ closed set: switch expression
{
    Tier.Gold => 0.15m, Tier.Silver => 0.05m, _ => 0m,
};
Dictionary<string, Func<Order, decimal>> handlers = new()   // ✓ open/extensible set:
{                                                            //   dictionary of behavior
    ["standard"] = o => o.Total,                             //   (delegates-events.html)
    ["rush"]     = o => o.Total * 1.25m,
};

// 4. Early return BEATS flag variables:
bool found = false;                                   // ✗ mutable flag + break dance
foreach (var u in users) { if (u.Id == id) { found = true; break; } }
return users.Any(u => u.Id == id);                    // ✓ (LINQ — the ultimate flow-shaper)

Control flow is where code review happens most, and these four refactors are the recurring verdicts: return the boolean you computed; name conditions so the business rule is legible; replace branching over closed sets with switch expressions (exhaustiveness-checked) and over open sets with dictionaries of delegates (extensible without edits — the strategy pattern again); and prefer early returns and query operators to flags-and-breaks state machines. The arc of this whole lesson bends toward the next ones: much imperative flow in C# ultimately dissolves into pattern matching and LINQ.

Senior framing: flow-control review heuristics that transfer to any codebase: cyclomatic complexity clusters where bugs cluster — a method needing three indentation levels of conditions is asking to be decomposed (guards, extracted predicates, lookup tables); else after a returning if is dead weight; loop bodies that both mutate the collection and iterate it are crash bait; and null-conditional chains (a?.b?.c?.d) longer than two hops usually mean the model is wrong (Law-of-Demeter violation) rather than the nulls being handled well. The C#-specific tells of current fluency: switch expressions over statement switches for value computation, is not null over != null (pattern syntax ignores operator overloads — it can't lie), and index/range operators over Length arithmetic. Small things — but interviewers pattern-match on them the way you're learning to.

Interview Questions

Q1 (Beginner): Why doesn't if (count) compile in C#, and what bug classes does that rule remove?

Conditions must be exactly bool — there is no truthiness coercion. This removes: accidental assignment-as-condition (if (x = 5) — a classic C bug — fails to compile because the assignment's value is int, not bool), and the "surprising falsy" family (empty string, zero, empty collections being silently false) that JS/Python developers debug regularly. The cost is typing count > 0 explicitly; the benefit is that every condition states its actual comparison.

Q2 (Beginner): Explain ?., ??, and ??= with one practical use each.

?. (null-conditional) evaluates the member only if the receiver is non-null, otherwise the whole expression is null: order?.Customer?.Name traverses safely — also the event-raise idiom OnChanged?.Invoke(...). ?? (null-coalescing) supplies a fallback: nickname ?? "Guest". ??= assigns only when the target is null — lazy initialization: _cache ??= Load(). Together they express "traverse, default, initialize" without a single explicit null check — with the caveat that they should model legitimate absence, not paper over broken invariants.

Q3 (Mid): switch statement vs switch expression — differences, and why prefer the expression where it fits?

The statement executes side effects per case (mandatory break, no value); the expression computes a value with pattern-matching arms. Prefer the expression for value computation because: it's exhaustiveness-checked (compiler warns on unhandled inputs — a changed enum surfaces every switch that missed the new member); arms can't fall through or forget breaks; there's no mutable temp being assigned in each branch; and it composes patterns (relational >= 90, or-combinations, guards) that the old statement needed if-chains for. The statement form remains right when cases genuinely perform different actions rather than produce values.

Q4 (Mid): What exactly does foreach require of the thing it iterates, and what are its two famous rules?

foreach compiles to: get an enumerator (GetEnumerator() — duck-typed or via IEnumerable<T>), loop MoveNext()/Current, dispose if disposable. Anything satisfying that works: arrays, List, Dictionary (yielding KeyValuePairs), strings, and lazy LINQ sequences — meaning iteration may execute the query as it pulls (the deferred-execution story). Rules: the iteration variable is read-only (assignment is a compile error), and the underlying collection must not be structurally modified during enumeration (versioned enumerators throw InvalidOperationException) — hence RemoveAll, snapshots, or collect-then-mutate for filtered removal.

Q5 (Senior): A code review shows if (user?.IsAdmin ?? false) gating an admin action, and elsewhere config?.Save(). Assess both.

user?.IsAdmin ?? false: defensible if "no user" is a legitimate state at this point (fail-closed to non-admin is the safe default) — but if a null user here means an auth bug, the pattern silently demotes a broken invariant into "just not admin", and the correct form is an explicit guard that throws or redirects; also note server-side authorization must exist regardless. config?.Save() is the more suspicious: silently skipping a write when config is null means data loss with no signal — persistence calls should almost never be null-conditional; if config can legitimately be absent, that deserves an explicit, logged branch. The shared principle: ?. encodes "absence is fine here" — review every use against whether absence actually is fine, especially on actions with side effects.

Q6 (Senior): Refactor review: a 60-line method with nested ifs over order type, customer tier, and region computing a shipping cost. What's your target shape and why?

Target: a pure function built from named parts. First, guards for invalid inputs (fail fast). Then decompose the decision: if the axes are independent, compute each factor separately (BaseRate(region), TierMultiplier(tier), TypeSurcharge(orderType)) — each a small switch expression over a closed set, exhaustiveness-checked so new enum members surface every site needing thought. If the axes genuinely interact, a single switch expression over a tuple pattern ((type, tier, region) switch { (Express, Gold, _) => …, … } — pattern lesson) makes the interaction table explicit and reviewable against the business's own pricing matrix. If the set is open/configurable, lift it to data (dictionary or database-driven rules). Wins: testable pieces, compiler-checked completeness, and a diff-able rule table instead of archaeology through nesting — plus the conversation with the domain expert becomes "is this table right?", which is the actual goal of readable flow.

🏋️ Exercises

1 (Easy): Rewrite with guard clauses (flat happy path): 
string Process(Order? order) {
  if (order != null) {
    if (order.Items.Count > 0) {
      if (!order.Cancelled) { return "processing"; }
      else { return "was cancelled"; }
    } else { return "empty order"; }
  } else { return "no order"; }
}
Solution
string Process(Order? order)
{
    if (order is null) return "no order";
    if (order.Items.Count == 0) return "empty order";
    if (order.Cancelled) return "was cancelled";
    return "processing";                     // the real outcome, at indent zero
}
// Each rule reads once, in order; adding a rule = adding a line, not a nesting level.
// Note `is null` — pattern syntax that can't be hijacked by an overloaded ==.
2 (Easy): Write DescribeAge(int age) as a single switch expression: negative/>150 throws, 0–12 "child", 13–17 "teen", 18–66 "adult", 67+ "senior".
Solution
string DescribeAge(int age) => age switch
{
    < 0 or > 150 => throw new ArgumentOutOfRangeException(nameof(age)),
    <= 12        => "child",
    <= 17        => "teen",
    <= 66        => "adult",
    _            => "senior",
};
// Top-down arm order lets each arm assume the previous ones failed — the ranges
// stay simple. throw is an expression in C#, so it can BE an arm.
3 (Medium): Given List<Order> orders, remove all cancelled orders three ways (RemoveAll, snapshot iteration, collect-then-remove), and state when each is the right choice.
Solution
// A. The idiom — one call, in-place, fastest:
orders.RemoveAll(o => o.Cancelled);
// RIGHT: whenever the condition is a simple predicate. Default choice.

// B. Snapshot iteration — when the loop body does MORE than remove:
foreach (var o in orders.ToList())          // iterate a COPY
{
    if (o.Cancelled) { Log(o); orders.Remove(o); }   // side effects + removal
}
// RIGHT: removal is entangled with other per-item work. COST: copies the list.

// C. Collect-then-apply — when you must act on removed items as a BATCH:
var toRemove = orders.Where(o => o.Cancelled).ToList();
foreach (var o in toRemove) orders.Remove(o);
NotifyCancellations(toRemove);
// RIGHT: the removed set is itself needed afterward.
// All three exist because the direct version — remove inside foreach over the same
// list — throws InvalidOperationException by design (versioned enumerators).
4 (Medium): Predict each line's output and explain the operator rule it demonstrates: 
Console.WriteLine(-7 % 2);
Console.WriteLine(7 / 2 * 2.0);
var a = new List<int> {1}; var b = new List<int> {1};
Console.WriteLine(a == b);
Console.WriteLine("ab" + 1 + 2);
Console.WriteLine((string?)null ?? "x");
Solution
-1      // % follows the DIVIDEND's sign — not true modulo
6       // left-to-right: 7/2 is INT division = 3 first; then 3 * 2.0 = 6.0 → "6"
False   // == on List = reference identity; equal content ≠ same object
        //   (a.SequenceEqual(b) would be True)
ab12    // + is left-assoc: "ab"+1 = "ab1" (string concat), then +2 = "ab12"
x       // ?? supplies the fallback for the null left operand
// Themes: evaluation order matters (cast/promote EARLY if you mean real division:
// 7 / 2.0), and ==' s meaning is type-dependent — the recurring C# equality lesson.
5 (Hard): Implement ParseRetryDelays(string csv): input like "1s,500ms,2s" returns List<TimeSpan>; junk entries are skipped with a warning; empty/null input yields an empty list. Use the lesson's toolkit (guards, TryParse, switch on suffix).
Solution
List<TimeSpan> ParseRetryDelays(string? csv)
{
    var result = new List<TimeSpan>();
    if (string.IsNullOrWhiteSpace(csv)) return result;          // guard: legit absence

    foreach (var raw in csv.Split(',', StringSplitOptions.TrimEntries))
    {
        var (numberPart, unit) = raw switch
        {
            [.., 'm', 's']       => (raw[..^2], "ms"),          // list patterns! ends with ms
            [.., 's']            => (raw[..^1], "s"),
            _                    => (raw, ""),
        };

        if (unit == "" || !int.TryParse(numberPart, out var n) || n < 0)
        {
            Console.WriteLine($"warning: skipping '{raw}'");    // junk skipped, not thrown
            continue;
        }

        result.Add(unit == "ms" ? TimeSpan.FromMilliseconds(n)
                                : TimeSpan.FromSeconds(n));
    }
    return result;
}
// ParseRetryDelays("1s,500ms,oops,2s") → [1s, 0.5s, 2s] + one warning.
// Guards for absence, TryParse for junk (a trust boundary), switch expression +
// range slicing for the suffix, continue for skip-and-carry-on.
6 (Hard): Refactor this pricing method using Part 6's playbook (name conditions, switch expressions per axis, guards). Then explain what the compiler now checks that it didn't before. 
decimal Price(Order o, Customer c) {
  decimal p = o.Total;
  if (c.Tier == Tier.Gold) { if (o.Express) { p = p * 1.1m; } p = p * 0.85m; }
  else if (c.Tier == Tier.Silver) { if (o.Express) { p = p * 1.2m; } p = p * 0.95m; }
  else { if (o.Express) { p = p * 1.25m; } }
  if (o.Total > 500m && c.Tier != Tier.None && !c.OnHold) { p = p - 10m; }
  return p;
}
Solution
decimal Price(Order o, Customer c)
{
    ArgumentNullException.ThrowIfNull(o);
    ArgumentNullException.ThrowIfNull(c);

    decimal expressMultiplier = c.Tier switch          // axis 1: express upcharge by tier
    {
        Tier.Gold   => 1.10m,
        Tier.Silver => 1.20m,
        Tier.None   => 1.25m,
    };                                                  // NO _ arm — deliberate! see below
    decimal tierDiscount = c.Tier switch                // axis 2: tier discount
    {
        Tier.Gold   => 0.85m,
        Tier.Silver => 0.95m,
        Tier.None   => 1.00m,
    };

    bool qualifiesForLoyaltyRebate =                    // named condition: legible rule
        o.Total > 500m && c.Tier != Tier.None && !c.OnHold;

    var price = o.Total
        * (o.Express ? expressMultiplier : 1m)
        * tierDiscount;
    if (qualifiesForLoyaltyRebate) price -= 10m;
    return price;

    // What the compiler checks NOW: omitting the `_` arm makes each switch
    // EXHAUSTIVE over Tier — add Tier.Platinum next quarter and BOTH switches emit
    // warnings until someone decides its rates. The old else-chain would have
    // silently lumped Platinum into the None branch. Also visible now: the original
    // applied the express multiplier BEFORE the gold discount but named neither —
    // here the formula reads as the business states it, and each factor unit-tests
    // in isolation.

🎯 Key Takeaways

  • Conditions are strictly bool (no truthiness), &&/|| short-circuit, and guard clauses keep the happy path flat — fail fast, return early.
  • The null family: ?. traverses absence, ?? defaults it, ??= lazily initializes — but reserve them for legitimate absence; broken invariants deserve ?? throw.
  • Switch expressions are the modern default for computing values: exhaustiveness-checked, fall-through-proof, pattern-powered; statements remain for per-case side effects.
  • foreach's contract: anything with an enumerator (including lazy LINQ); read-only loop variable; never structurally modify the collection mid-iteration (RemoveAll/snapshot/collect-then-apply).
  • Operator fine print: == is type-dependent (identity for most reference types), integer division truncates, % follows the dividend's sign, ^/.. index and slice.
  • Shape flow for readers: return booleans directly, name conditions, switch expressions over closed sets, dictionaries of delegates over open sets, queries over flag-loops.
