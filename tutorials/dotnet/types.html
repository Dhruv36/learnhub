Types & Variables: Value vs Reference Foundations

What you'll master: C#'s type system foundation — the value/reference split that explains half the language's design (and bugs), the numeric types and when each is right, decimal vs double, conversions and overflow behavior, boxing, var and type inference, and default values. Prerequisites: How .NET Runs. This distinction is THE C# interview staple — and unlike most trivia, it changes how you design types every day (structs and records build directly on it).

Part 1 — The Split: What a Variable Actually Holds

VALUE TYPES (struct, enum)            REFERENCE TYPES (class, interface, delegate,
  int, double, decimal, bool, char,     array, string, record class)
  DateTime, TimeSpan, Guid,
  custom structs, enums                 List<T>, arrays, your classes…

The variable holds THE DATA ITSELF.    The variable holds a REFERENCE to a heap object.

int a = 5;                             var list1 = new List<int> { 1, 2 };
int b = a;          // COPY of 5       var list2 = list1;        // COPY OF THE REFERENCE
b = 99;                                list2.Add(3);
Console.WriteLine(a);   // 5 — a       Console.WriteLine(list1.Count);  // 3! same object,
                        //  untouched                                    // two names

ASSIGNMENT ALWAYS COPIES — the question is only WHAT gets copied:
  value type  → the data       (independent copies)
  reference   → the pointer    (shared object, aliased)

Every "why did that change too?!" bug and every "why didn't my change stick?" bug is this one distinction. Assignment (and parameter passing) always copies the variable's contents; value types carry their data in the variable, so copies are independent — reference types carry an address, so copies alias one shared object. C# differs from Java here in a load-bearing way: you can define your own value types (struct), and the runtime treats them honestly (no mandatory heap, no pointer indirection, arrays of structs are contiguous memory) — the foundation of .NET's performance story (Memory).

Part 2 — The Numeric Types & the Money Rule

int      32-bit    ±2.1 billion        THE default integer
long     64-bit    ±9.2 quintillion    ids, timestamps, big counters
double   64-bit    binary floating     THE default for scientific/measured values
decimal  128-bit   BASE-10 floating    money and anything users type as digits
byte/short/uint/…                      interop, protocols, tight arrays — not defaults

// The money rule, demonstrated:
double d = 0.1 + 0.2;        // 0.30000000000000004  — 0.1 has NO exact binary form
decimal m = 0.1m + 0.2m;     // 0.3 exactly          — base-10 arithmetic

// Accumulated: 1000 payments of $0.10
double sumD = 0; for (int i = 0; i < 1000; i++) sumD += 0.10;      // 99.99999999999986
decimal sumM = 0; for (int i = 0; i < 1000; i++) sumM += 0.10m;    // 100.00 ✓

// OVERFLOW — silent by default (!), opt into safety:
int max = int.MaxValue;
max + 1                      // -2147483648 — wraps around silently
checked { max + 1 }          // OverflowException — or <CheckForOverflowUnderflow> project-wide

// Integer division trap:
5 / 2        // 2   (int ÷ int = int, truncated)
5 / 2.0      // 2.5 (one double promotes the operation)
5m / 2m      // 2.5

Three defaults cover most code — int for counts, double for measurements, decimal for money — and the discipline is knowing why: double is IEEE-754 binary (fast, hardware-native, inexact for base-10 fractions), decimal is base-10 (exact for anything representable in digits, ~10× slower, and the only defensible choice wherever cents are added — the same argument as Java's BigDecimal, but ergonomic enough that people actually use it). Silent integer overflow is the sleeper: production billing systems have wrapped negative — know checked, and use long before you're near limits.

Part 3 — Conversions, Casting & Parsing

// IMPLICIT — only when NO data can be lost (widening):
int i = 42;
long l = i;            // ✓ int fits in long
double d = i;          // ✓ (int → double is allowed; big longs CAN lose precision though!)

// EXPLICIT — you accept the risk with a cast:
double pi = 3.99;
int chopped = (int)pi;         // 3 — TRUNCATES toward zero (not rounds!)
int rounded = (int)Math.Round(pi);   // 4 — say what you mean
long big = 5_000_000_000;
int oops = (int)big;           // silent garbage (unchecked) — the cast is a promise you lied about

// Strings ↔ numbers — parsing is a TRUST BOUNDARY:
int.Parse("42")                        // throws on junk — for data you control
int.TryParse(input, out var n)         // false on junk — for USER/external input
decimal.Parse("1.234,56",
  new CultureInfo("de-DE"))            // culture matters! "1.5" isn't universal
int.Parse(s, CultureInfo.InvariantCulture)   // wire formats: ALWAYS invariant

// Culture is the hidden parameter of ToString too:
(1234.5).ToString()                    // "1234.5" or "1234,5" depending on OS culture!
(1234.5).ToString(CultureInfo.InvariantCulture)  // deterministic — for logs/protocols

The conversion rules encode one principle: silent only when safe, explicit when lossy — and a cast is you overriding the compiler's caution, so own its semantics (truncation, wrap-around). Parsing adds the two production concerns: junk input (TryParse at trust boundaries — exceptions are for exceptional cases, not user typos) and culture — number and date formats vary by locale, and the bug pattern "works on my machine, breaks on the German server" is a missing CultureInfo.InvariantCulture on wire-format parsing/serialization. Rule: user-facing text uses the user's culture; anything machine-to-machine uses invariant.

Part 4 — Boxing: When Value Types Visit the Heap

int n = 42;
object o = n;          // BOXING: allocate heap object, copy 42 into it
int back = (int)o;     // UNBOXING: runtime type-check, copy value back out

// Where it hides (the non-obvious sites):
object[] mixed = { 1, "two", 3.0 };      // 1 and 3.0 box
string s = string.Format("{0}", 42);      // object parameter → boxes (interpolation
                                          //   with modern handlers avoids it)
IComparable c = 42;                       // interface variable of a struct → boxes
enum e = Color.Red; object oe = e;        // enums box too

// Why it matters: a box is an allocation + indirection + later GC work.
// In a loop over millions of items, accidental boxing dominates profiles.

// What ELIMINATED most historic boxing: GENERICS (reified!):
List<int> ints = [1, 2, 3];              // stores raw ints — NO boxing
// .NET generics are real at runtime: List<int> is a distinct compiled type with
// int-sized storage. (Java contrast: erasure → List<Integer> boxes every element.)

Boxing is the bridge the unified type system needs — any value type can travel as object — priced at one heap allocation per crossing. The skill is recognizing the hidden sites (old non-generic APIs, interface-typed struct variables, object-parameter methods) and knowing why modern C# rarely boxes: reified generics store value types natively, generic constraints let methods accept structs without upcasting, and interpolated-string handlers dodge the object[] path. This is also a genuine .NET-vs-JVM differentiator worth being able to articulate: reified generics are why List<int> is memory-dense in C# and box-per-element in Java.

Part 5 — var, Inference, Literals & Defaults

var count = 42;                    // int — inferred at COMPILE time; fully static typing
var name = "Ada";                  // string
var order = new Order(1, "Ada");   // Order — var shines when the right side already says it
// var is NOT dynamic: the type is fixed at compile time; you just didn't repeat it.
Order order = new();               // target-typed new — the mirror image

// When var helps vs hurts (house style most teams converge on):
var items = GetItems();            // ✗ what is this? reader must look up GetItems
List<Order> items = GetItems();    // ✓ annotate when the right side doesn't say

// Literals:
1_000_000                          // digit separators
0xFF, 0b1010                       // hex, binary
3.14 (double)  3.14f  3.14m        // suffix picks the type
"text", 'c', $"n = {n}", """raw "json" fine"""    // raw string literals (C# 11)

// DEFAULTS — every type has one; fields get it, locals DON'T:
default(int)        // 0        default(bool)   // false
default(DateTime)   // 0001-01-01 — a classic "why is my date year 1?" bug
default(string)     // null      (reference types default to null — nullability lesson)
int x; Console.WriteLine(x);      // ✗ compile error: unassigned local (definite assignment)

var is compile-time inference, not dynamic typing — the compiler writes the type you didn't; IntelliSense and the type checker see it perfectly. The pragmatic rule: infer when the right-hand side names the type (constructors, casts, literals), annotate when it doesn't (method calls whose return type isn't obvious). Definite assignment — locals must be provably assigned before use — kills a whole uninitialized-variable bug class at compile time, while fields silently get default, which is where the year-0001 DateTime and null-string surprises come from (nullability lesson completes that story).

Part 6 — Deep Dive: Choosing and Designing Types

// The "primitive obsession" problem — compiles, and is wrong:
ProcessPayment(orderId: customerId, customerId: orderId);   // both are int. oops.

// Types as guardrails — make wrong code not compile:
public readonly record struct OrderId(int Value);       // records-structs.html for mechanics
public readonly record struct CustomerId(int Value);
ProcessPayment(OrderId cid, CustomerId oid)             // ✗ now a COMPILE ERROR

// Units, the same trick:
public readonly record struct Money(decimal Amount, string Currency);
// adding Money to Money: define the operator, ASSERT same currency — the type system
// now enforces what a comment used to beg for.

// Enums — closed sets of named values (a value type over an int):
public enum OrderStatus { Pending = 0, Paid = 1, Shipped = 2, Cancelled = 3 }
//   explicit numbers because they end up in DATABASES — renumbering is data corruption
//   [Flags] enums exist for bitmasks (Permissions.Read | Permissions.Write)
//   note: (OrderStatus)42 does NOT throw — validate at boundaries (Enum.IsDefined)

Decision guide:
  measured/scientific   double        money/user-entered digits   decimal
  counts/ids            int (long if big)   closed named set      enum
  domain concept that's "an int with meaning" → wrap it (record struct)
  time: DateTime (know its Kind!), DateTimeOffset for absolute instants,
        TimeSpan durations, DateOnly/TimeOnly for calendar values (js dates lesson's
        instant-vs-wall-clock model applies verbatim here)

The senior layer of "types" isn't memorizing ranges — it's using the type system as a correctness tool. Wrapping primitives into domain types (OrderId ≠ CustomerId) turns argument-swap bugs and unit confusion into compile errors at near-zero runtime cost (a readonly record struct wrapper is just the int, plus meaning). Enums encode closed vocabularies but hold surprises worth knowing (any int casts in unchecked; database-persisted values need stable explicit numbers). And date/time types deserve their own respect: DateTimeOffset for instants, DateOnly for birthdays — the exact instant-vs-calendar distinction from the JS dates lesson, with better types available.

Senior framing: value-vs-reference is the interview question because it's the design question: every type you author picks a side (struct vs class, coming lesson — size, copying semantics, identity vs value equality, heap pressure). The follow-up probes that sort candidates: "where does a value type live?" (wherever its container lives — locals on the stack, but a struct field inside a class lives on the heap inside that object; "stack vs heap" is about variables, not types); "what does == compare?" (value types: data; reference types: identity by default — string being the famous value-semantics exception); and "why did .NET add spans/ref structs?" (to extend value-type efficiency to slices of memory — Memory lesson). Answering with the model rather than the slogan is the difference between having read about C# and having designed in it.

Interview Questions

Q1 (Beginner): Value types vs reference types — what's the difference, and what does assignment do in each case?

A value-type variable (int, struct, enum) contains its data directly; a reference-type variable (class, array, string, delegate) contains a reference to an object on the managed heap. Assignment always copies the variable's contents — so value types yield independent copies (mutating one leaves the other untouched), while reference types yield two references to one shared object (mutation through either is visible through both). The same rule governs parameter passing, which is why the distinction explains both "change didn't stick" and "change leaked" bug families.

Q2 (Beginner): Why is decimal the money type, and what do the literal suffixes mean?

double/float are binary floating point: base-10 fractions like 0.1 have no exact binary representation, so money arithmetic accumulates drift (0.1 + 0.2 ≠ 0.3; a thousand 10-cent additions miss $100). decimal is a 128-bit base-10 type: exact for anything expressible in decimal digits, at ~10× the cost — irrelevant for business code, essential for correctness. Suffixes pick literal types: m decimal, f float, d/none double, L long — and forgetting m matters because 0.1 is already an inexact double before any conversion.

Q3 (Mid): What is boxing, where does it hide, and why do .NET's reified generics matter here?

Boxing wraps a value type into a heap object when it's treated as object (or as an interface it implements): one allocation plus a copy; unboxing type-checks and copies back. Hidden sites: object-parameter APIs (old string.Format), non-generic collections, assigning a struct to an interface variable, enums to object. It matters in hot paths — allocations × millions = GC pressure. Reified generics are the systemic fix: List<int> is a real runtime type storing raw ints contiguously — no boxing, cache-friendly — whereas erasure-based generics (Java) box every element. That's a marquee .NET performance differentiator worth stating precisely.

Q4 (Mid): "Value types live on the stack" — critique this common claim.

It's a half-truth. A value type lives wherever its container lives: a local int is (typically) in a stack frame or register; but a struct field of a class lives on the heap, inline inside that object; an element of int[] or List<int> lives on the heap inside the array's memory; a captured local (closure) moves to the heap with its display class. The accurate statement: value types are stored inline in their container without an object header or indirection — which is the actual performance point (density, no per-item allocation) — and stack-vs-heap is a property of the storage location, not the type. Bonus: ref struct types (Span) really are stack-only, enforced by the compiler — the exception that proves the rule needed enforcing.

Q5 (Senior): A report shows totals off by fractions of a cent, and a nightly job crashes with negative order counts after years of uptime. Connect both to this lesson and give the remediations.

Fraction-of-a-cent drift: money flowing through double somewhere — often a single conversion (a JSON deserializer defaulting to double, an Excel export round-trip, one double intermediate in a calculation chain). Remediate: decimal end-to-end (types, DTOs, DB columns — decimal(19,4)-style), plus banker's-rounding policy made explicit (Math.Round(x, 2, MidpointRounding.ToEven)) at defined points, and a regression test asserting exact totals. Negative counts: a 32-bit counter wrapped — int overflows silently past 2.1B. Remediate: long for anything that accumulates unboundedly (ids, counters, bytes), checked arithmetic (or the project-wide flag) so future overflow throws instead of corrupting, and an audit for other int-typed accumulators. Both are type-choice bugs invisible in code review unless reviewers carry these rules — which is the argument for analyzers enforcing them.

Q6 (Senior): Argue for domain wrapper types (OrderId vs int) — costs, benefits, and where you'd draw the line.

Benefits: argument-swap and unit-confusion bugs become compile errors; signatures self-document (Transfer(AccountId from, AccountId to, Money amount)); invariants get a home (Money asserts currency match in its operators); and refactoring is safer — change what an OrderId is in one place. Costs: ceremony (definitions, conversions at boundaries like JSON/DB — mitigated by source generators or converter registration), team convention overhead, and friction with libraries expecting primitives. As readonly record struct wrappers the runtime cost is ~zero (same size as the primitive, no allocation). Line- drawing: wrap identifiers that coexist (multiple id types in one signature), money, and unit-bearing quantities; don't wrap genuinely dimensionless locals (loop counters, sizes) or types that never cross a boundary where confusion is possible. It's the same judgment as any abstraction: pay ceremony where a real bug class dies.

🏋️ Exercises

1 (Easy): Predict the output, then explain with the copy rule: 
var a = new int[] { 1, 2, 3 };
var b = a;
b[0] = 99;
int x = 1; int y = x; y = 42;
Console.WriteLine($"{a[0]} {x}");
Solution
99 1
// Arrays are REFERENCE types: b = a copied the reference — one array, two names,
// so b[0] = 99 is visible through a. int is a VALUE type: y = x copied the data;
// y's reassignment can't touch x. One rule (assignment copies the variable's
// contents), two outcomes depending on what the contents ARE.
2 (Easy): Write a price calculator that reads a price and quantity from the user (TryParse, reject junk), computes the total with 19% tax in decimal, and prints it currency-formatted.
Solution
Console.Write("Unit price: ");
if (!decimal.TryParse(Console.ReadLine(), out var price) || price < 0)
{ Console.WriteLine("Invalid price."); return; }

Console.Write("Quantity: ");
if (!int.TryParse(Console.ReadLine(), out var qty) || qty <= 0)
{ Console.WriteLine("Invalid quantity."); return; }

var total = price * qty * 1.19m;                  // decimal end-to-end; m suffix
Console.WriteLine($"Total: {total:C}");           // :C = currency format (user's culture —
                                                  //  correct for DISPLAY)
3 (Medium): Demonstrate the three numeric traps — accumulation drift, cast truncation, silent overflow — each in two lines, with the fixed version beside it.
Solution
// 1. Drift:
double d = 0; for (int i = 0; i < 10; i++) d += 0.1;    // 0.9999999999999999
decimal m = 0; for (int i = 0; i < 10; i++) m += 0.1m;  // 1.0 ✓

// 2. Truncation:
Console.WriteLine((int)9.99);                            // 9 — casts chop toward zero
Console.WriteLine((int)Math.Round(9.99));                // 10 ✓ — intent stated

// 3. Overflow:
int n = int.MaxValue; Console.WriteLine(n + 1);          // -2147483648, silently
checked { long ok = (long)n + 1; Console.WriteLine(ok); }// 2147483648 ✓ — headroom
// (checked would THROW on int overflow instead of wrapping — either beats silence)
4 (Medium): A log line must be written machine-parseable and displayed to a German user. Produce both outputs for 1234567.89m and DateTime.UtcNow, and state the rule.
Solution
var amount = 1234567.89m;
var now = DateTime.UtcNow;

// MACHINE (logs, wire, storage): invariant + ISO 8601 — deterministic everywhere:
var log = string.Create(CultureInfo.InvariantCulture,
    $"amount={amount} at={now:O}");
// → amount=1234567.89 at=2026-07-19T14:30:00.0000000Z

// DISPLAY (German user): their culture does the honors:
var de = new CultureInfo("de-DE");
Console.WriteLine(amount.ToString("C", de));   // 1.234.567,89 €
Console.WriteLine(now.ToLocalTime().ToString("g", de));  // 19.07.2026 16:30

// Rule: culture is a parameter, not an ambient accident. Invariant for machines,
// explicit user culture for humans — never let the SERVER's OS culture decide either.
5 (Hard): Show a hidden-boxing hot loop, prove the fix, and explain how you'd have found it in a real app.
Solution
// BEFORE — interface-typed struct = box per iteration:
int total = 0;
IComparable<int> threshold = 10;              // int boxed into the interface variable
foreach (var v in values)                     // and old-style: ArrayList would box EVERY add
    if (v.CompareTo(10) > 0) total += v;      // v.CompareTo fine; the pattern to fear is
                                              // object/interface-typed storage of structs
// A sharper classic:
object sum = 0;
foreach (var v in values) sum = (int)sum + v; // box + unbox EVERY iteration

// AFTER — keep value types typed as themselves (or as generic T):
int sum2 = 0;
foreach (var v in values) sum2 += v;          // no allocations at all
// Generic version that still avoids boxing:
static T Max<T>(T a, T b) where T : IComparable<T> => a.CompareTo(b) >= 0 ? a : b;
// constrained call — the JIT calls the struct method directly, no box.

// Finding it for real: allocation profiler (dotnet-trace / VS profiler / dotMemory)
// shows System.Int32 boxes in the allocation report; or the ClrHeapAllocationAnalyzer
// flags box sites at build. Mystery GC pressure with value-type-heavy loops → suspect
// boxing first.
6 (Hard): Implement Money as a readonly record struct (amount + currency) with +/− operators that throw on currency mismatch, a ToString honoring a culture, and show the argument-swap bug it prevents.
Solution
public readonly record struct Money(decimal Amount, string Currency)
{
    public static Money operator +(Money a, Money b) => a.Currency == b.Currency
        ? a with { Amount = a.Amount + b.Amount }
        : throw new InvalidOperationException($"Cannot add {a.Currency} to {b.Currency}");
    public static Money operator -(Money a, Money b) => a + (b with { Amount = -b.Amount });

    public string ToString(CultureInfo culture) =>
        Amount.ToString("N2", culture) + " " + Currency;
}

var price = new Money(19.99m, "EUR");
var shipping = new Money(4.95m, "EUR");
Console.WriteLine((price + shipping).ToString(new CultureInfo("de-DE"))); // 24,94 EUR

var usd = new Money(5m, "USD");
// price + usd → InvalidOperationException — the unit bug is now IMPOSSIBLE to miss.

// And the swap-prevention:
void Refund(CustomerId customer, Money amount) { /* … */ }
// Refund(amount, customer)  ✗ does not compile — compare with Refund(int, decimal),
// where swapped arguments would have compiled and paid someone the customer id.
// Value semantics bonus: Money is compared by CONTENT (record struct equality),
// costs one allocation nowhere (it's a struct), and is immutable (readonly + with).

🎯 Key Takeaways

  • Assignment always copies the variable's contents: value types (int, struct, enum) copy data — independent; reference types (class, array, string) copy the reference — aliased. This one rule decodes both "didn't stick" and "changed too" bugs.
  • Numeric defaults: int counts, double measurements, decimal money (base-10 exact — the m suffix); overflow is silent unless checked; casts truncate.
  • Parsing/formatting are culture-sensitive: TryParse at trust boundaries, InvariantCulture for machine data, explicit user culture for display.
  • Boxing = value type as object/interface = allocation; reified generics store value types natively (the List<int> density story) — watch legacy object-typed APIs in hot paths.
  • var is static inference (annotate when the RHS doesn't name the type); locals require definite assignment, fields silently get default (year-0001 DateTime, null strings).
  • Use types as guardrails: enum for closed sets (explicit numbers if persisted), DateTimeOffset/DateOnly for time, and domain wrappers (readonly record struct) to make argument-swap and unit bugs uncompilable.
