Dependency Injection & Hosting Advanced

ASP.NET Core (and modern .NET generally) has dependency injection BUILT IN — no third-party container needed for the common case. This lesson covers the built-in DI container's service lifetimes, the Generic Host that wires everything together, and the #1 DI bug in real .NET codebases: lifetime mismatches.

The Three Service Lifetimes — Know Them Cold

LifetimeInstance createdUse for
TransientA NEW instance every time it's requested/injectedLightweight, stateless services
ScopedONE instance per request (web) / per scopePer-request state — most application/DB services
SingletonONE instance for the ENTIRE application lifetimeTruly shared, thread-safe state (caches, config)
builder.Services.AddTransient();   // new instance every injection
builder.Services.AddScoped();     // one per HTTP request
builder.Services.AddSingleton();  // one for the whole app's lifetime

The #1 DI Bug: Captive Dependencies

// ❌ A SINGLETON that depends on a SCOPED service — a "captive dependency":
public class ReportCache {                          // registered as Singleton
    public ReportCache(IOrderRepository repo) { }    // repo is Scoped (e.g., wraps a DbContext)
}
// The FIRST scope's OrderRepository instance gets captured into the singleton's field FOREVER.
// Every subsequent "request" that uses ReportCache is silently reusing the FIRST request's
// (likely now-disposed) DbContext — causing ObjectDisposedException or, worse, silently WRONG data
// shared across unrelated users' requests. The DI container detects and THROWS on this at startup
// in ASP.NET Core (with scope validation enabled) — but understanding WHY is the real interview test.

// ✅ Fix: inject IServiceScopeFactory / IServiceProvider into the singleton and create a NEW scope
// each time you need the scoped dependency, OR simply don't depend on scoped services from a singleton.
⚠️ Lifetime rule: a service should never depend on one with a SHORTER lifetime. Singleton → Scoped is the classic captive-dependency bug. Scoped → Transient and Singleton → Transient are fine (a new transient instance is cheap to create each time, so it never gets "captured"). Memorizing this one directional rule — longer-lived services can't safely hold references to shorter-lived ones — resolves most real-world DI lifetime bugs.

Constructor Injection — the Standard Pattern

public class OrdersController : ControllerBase {
    private readonly IOrderService _orderService;      // dependencies declared as constructor params
    private readonly ILogger _logger; // ILogger is a built-in generic service

    public OrdersController(IOrderService orderService, ILogger logger) {
        _orderService = orderService;
        _logger = logger;
    }
    // ASP.NET Core's DI container automatically resolves and injects both when creating this controller
    // for each incoming request — you never call `new OrdersController(...)` yourself.
}

The Generic Host — What Ties It All Together

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();                 // register framework services
builder.Services.AddScoped();   // register YOUR services
builder.Services.AddDbContext(opt => opt.UseSqlServer(connectionString));

var app = builder.Build();                          // builds the DI container + configures the pipeline
app.MapControllers();
app.Run();                                          // starts the host: web server + DI + config + logging
// The Generic Host unifies dependency injection, configuration (appsettings.json, env vars), logging,
// and the web server itself into one coherent startup pipeline — this is what `builder`/`app` represent.

Interview Questions

Q1: Explain the three DI service lifetimes and give a concrete example of when to use each.

Transient services get a BRAND NEW instance created every single time they're requested/injected — even if the same object graph is being built for a single HTTP request, two different classes that both depend on a Transient service each get their own separate instance. This suits lightweight, stateless services with no meaningful cost to constructing repeatedly, like a simple validator or formatter with no fields. Scoped services get ONE instance per "scope," which in ASP.NET Core concretely means one instance per HTTP request — multiple classes injected during that single request's handling all share the same Scoped instance, but the next request gets a fresh one. This is the right lifetime for the majority of application services, especially anything wrapping a database context (EF Core's DbContext is registered Scoped by default via AddDbContext), since you want one consistent unit-of-work per request but not shared state leaking between unrelated requests. Singleton services get exactly ONE instance for the entire lifetime of the application — created once, reused by every request/consumer forever. This is appropriate for genuinely shared, ideally immutable or thread-safe state: an in-memory cache, application configuration loaded once at startup, or a connection to an external service designed to be reused. The choice matters not just for performance but for correctness, since these lifetimes interact in ways that can silently break an application if mismatched.

Q2: What is a "captive dependency" and why is it dangerous?

A captive dependency occurs when a service with a LONGER lifetime holds a reference to a service with a SHORTER lifetime — the classic and most dangerous case being a Singleton that has a Scoped service injected into its constructor. Because the Singleton is only ever constructed ONCE, for the entire application's lifetime, whatever Scoped instance happens to be provided at that first construction (typically during the very first request that triggers the Singleton's creation) gets permanently "captured" inside the Singleton's field — it is never replaced with a fresh Scoped instance on subsequent requests, because the Singleton itself is never reconstructed. This is dangerous for two concrete reasons: first, if that captured Scoped service wraps something request-lifetime-bound (like an EF Core DbContext, which gets disposed at the end of each request), every subsequent use of the Singleton after the first request ends will throw an ObjectDisposedException, since it's holding onto an already-disposed object. Second, and more insidiously, if the captured service holds request- specific STATE (like the currently authenticated user, or request-specific data) rather than throwing an obvious error, every subsequent request could silently see and act on data belonging to the FIRST request that ever triggered the Singleton's creation — a serious, hard-to-detect data-leakage bug across unrelated users. ASP.NET Core's DI container can be configured to validate scopes at startup and will throw immediately if it detects a Singleton depending on a Scoped service, turning this into a fail-fast startup error rather than a silent runtime bug — but the underlying rule to internalize is that a service should never depend on one with a strictly shorter lifetime than itself.

Q3: How would you fix a Singleton that legitimately needs to use a Scoped service?

You can't simply inject the Scoped service directly into the Singleton's constructor — that's exactly the captive-dependency bug. The correct pattern is to inject IServiceScopeFactory (or the broader IServiceProvider) into the Singleton instead — both of these ARE safe to inject into a Singleton, since they're themselves effectively application-lifetime services that know how to create scopes on demand rather than being scoped instances themselves. Then, whenever the Singleton actually needs to use the Scoped dependency (for example, inside a background timer callback that periodically needs database access), it explicitly creates a NEW scope at that moment — `using var scope = _scopeFactory.CreateScope();` — and resolves the Scoped service FROM that fresh scope — `var repo = scope.ServiceProvider.GetRequiredService<IOrderRepository>();`. Because the scope (and everything resolved from it, including any DbContext) is disposed at the end of that `using` block, this gives the Singleton fresh, correctly-lifetimed access to the Scoped dependency every time it's needed, without ever capturing a single instance permanently. This exact pattern is the standard way to access Scoped/database services from long-lived background services (IHostedService/BackgroundService implementations) in ASP.NET Core, since those are effectively Singletons managed by the host.

🏋️ Practice: register a service as Transient, Scoped, and Singleton in turn and log a unique ID from its constructor to observe instance-creation frequency across multiple requests; deliberately create a Singleton depending on a Scoped service and reproduce the captive-dependency failure (with scope validation enabled); fix it using IServiceScopeFactory; register a DbContext and confirm it defaults to Scoped; write a BackgroundService that correctly creates a new scope on each timer tick to access a Scoped repository.

🎯 Key Takeaways

  • ASP.NET Core has DI built in: Transient (new every time), Scoped (one per request), Singleton (one for the app's lifetime) — know exactly when to use each.
  • Captive dependency = a longer-lived service holding a reference to a shorter-lived one; the classic case is Singleton depending on Scoped, causing stale state or ObjectDisposedException.
  • The directional rule: a service should never depend on one with a shorter lifetime — Scoped/Singleton → Transient is fine; Singleton → Scoped is not.
  • DbContext should be Scoped (AddDbContext's default) — it isn't thread-safe and shouldn't be shared across concurrent requests as a Singleton.
  • A Singleton that legitimately needs Scoped services should inject IServiceScopeFactory and create a fresh scope on demand, never capture a Scoped instance directly.
