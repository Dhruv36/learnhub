Testing (xUnit) Intermediate

xUnit is the most widely used .NET testing framework, especially in ASP.NET Core projects and open source. This lesson covers its conventions (which differ from older frameworks like MSTest/NUnit in small but important ways), mocking with dependency injection, and testing async code correctly.

The Basics — Fact vs Theory

public class CalculatorTests {
    [Fact]                          // a single test case, no parameters
    public void Add_TwoPositiveNumbers_ReturnsSum() {
        var calc = new Calculator();
        var result = calc.Add(2, 3);
        Assert.Equal(5, result);
    }

    [Theory]                        // a PARAMETERIZED test — runs once per InlineData
    [InlineData(2, 3, 5)]
    [InlineData(-1, 1, 0)]
    [InlineData(0, 0, 0)]
    public void Add_VariousInputs_ReturnsExpectedSum(int a, int b, int expected) {
        var calc = new Calculator();
        Assert.Equal(expected, calc.Add(a, b));
    }
}
// [Theory] avoids copy-pasting near-identical [Fact] tests for different input combinations —
// one test method, many data-driven executions, each reported as a separate result.
📛 xUnit has NO [SetUp]/[TearDown] attributes — it uses the CONSTRUCTOR and IDisposable instead. A new instance of the test class is created for EVERY test method, so the constructor naturally serves as setup (fresh state per test, no shared mutable state between tests by default), and implementing IDisposable.Dispose() serves as teardown. This is a deliberate design choice for test isolation and trips up developers coming from NUnit/MSTest/JUnit conventions.

Constructor as Setup, IDisposable as Teardown

public class OrderServiceTests : IDisposable {
    private readonly OrderService _service;
    private readonly TestDbContext _db;

    public OrderServiceTests() {                 // runs before EVERY test method
        _db = new TestDbContext();
        _service = new OrderService(_db);
    }

    [Fact]
    public void CreateOrder_ValidInput_PersistsOrder() {
        _service.CreateOrder(new Order { ... });
        Assert.Single(_db.Orders);
    }

    public void Dispose() => _db.Dispose();       // runs after EVERY test method
}

Arrange-Act-Assert (AAA) — the Universal Test Shape

[Fact]
public void Withdraw_InsufficientFunds_ThrowsException() {
    // Arrange — set up the object under test and its inputs
    var account = new BankAccount("Alex", 100m);

    // Act — perform the ONE action being tested
    var act = () => account.Withdraw(500m);

    // Assert — verify the outcome
    Assert.Throws(act);
}
// This structure — regardless of language/framework — keeps tests readable: anyone can scan a test
// and immediately see what's being set up, what's being exercised, and what's being verified.

Testing with DI — Mocking Dependencies (Moq)

public class OrderServiceTests {
    [Fact]
    public async Task CreateOrder_PaymentFails_DoesNotPersistOrder() {
        // Arrange — create a MOCK of the interface dependency (services-DI-style loose coupling pays off)
        var mockPayment = new Mock();
        mockPayment.Setup(p => p.ChargeAsync(It.IsAny(), It.IsAny()))
                   .ReturnsAsync(new PaymentResult { Success = false });   // configure fake behavior

        var service = new OrderService(mockPayment.Object);   // inject the mock, not a real processor

        // Act
        await service.CreateOrder(new Order { Total = 50m });

        // Assert
        mockPayment.Verify(p => p.ChargeAsync(50m, It.IsAny()), Times.Once);   // verify it was CALLED
    }
}
// Because OrderService depends on IPaymentProcessor (an interface), tests substitute a Moq-generated
// fake with zero changes to OrderService — the exact same DI benefit that makes production code testable.

Testing Async Code

[Fact]
public async Task GetUserAsync_ValidId_ReturnsUser() {      // the TEST METHOD itself is async
    var result = await userService.GetUserAsync(1);          // await the method under test directly
    Assert.NotNull(result);
}
// xUnit natively supports async Task test methods — no special ceremony needed. Just make the test
// method `async Task` (not async void — xUnit needs to know when it completes to report results).

Interview Questions

Q1: How does xUnit handle test setup and teardown, and why is that design different from other frameworks?

Unlike NUnit, MSTest, or JUnit, xUnit deliberately has no [SetUp]/[TearDown] (or [TestInitialize]/[TestCleanup]) attributes. Instead, xUnit creates a BRAND NEW INSTANCE of the test class for every single test method it runs, so the class's CONSTRUCTOR naturally runs before each test and serves as the setup step — any fields initialized there (a fresh service instance, a new in-memory database context) are guaranteed to be freshly created and not shared or polluted by a previous test's mutations. For teardown, the test class implements IDisposable, and xUnit calls Dispose() after each test method completes, giving a natural place to clean up resources (closing database connections, disposing file handles). This design is a deliberate test-isolation choice: it makes it structurally difficult to accidentally share mutable state between tests (since each test gets its own object graph from scratch), which prevents an entire class of flaky, order-dependent test bugs that can arise in frameworks relying on a single shared test-class instance with explicit setup/teardown hooks that developers might implement incorrectly or forget.

Q2: What's the difference between [Fact] and [Theory], and when do you use each?

[Fact] marks a test method that represents a single, specific test case with no parameters — it runs exactly once and either passes or fails as one unit; it's the right choice for tests verifying one particular scenario or behavior. [Theory] marks a PARAMETERIZED test method: the method itself takes parameters, and you supply one or more sets of input values via attributes like [InlineData(...)] (or, for more complex cases, [MemberData] pointing to a property/method that generates test cases, or [ClassData]). xUnit then runs the SAME test method body once for EACH data set provided, reporting each execution as an independent test result in the output — so a single [Theory] with five [InlineData] attributes shows up as five separate pass/fail results, not one. The practical benefit is avoiding the maintenance burden of copy-pasting a nearly identical [Fact] test for every input combination you want to verify (e.g., testing a validation method against several valid and invalid inputs): a [Theory] expresses "this same logic should hold for all these cases" in one place, making it easy to add new cases later by simply adding another InlineData line rather than duplicating an entire test method.

Q3: How does dependency injection make a class easier to unit test, using a mocking example?

A class that depends on concrete implementations (say, directly instantiating a real StripePaymentProcessor or opening a real database connection inside its own methods) is very hard to unit test in isolation — running its tests would require an actual Stripe account and network access, or a real database, making tests slow, flaky, and dependent on external systems being available and in a known state. If instead the class depends on an INTERFACE (IPaymentProcessor) supplied via its constructor — the dependency-injection pattern — a test can substitute a mock object implementing that same interface without the class under test knowing or caring about the difference. Using a mocking library like Moq, a test creates `new Mock<IPaymentProcessor>()`, configures its behavior with `.Setup(...)` (e.g., "when ChargeAsync is called with any amount, return a failed PaymentResult"), and passes `mockPayment.Object` (the actual fake implementation Moq generates) into the class under test's constructor in place of a real processor. The test can then exercise the class's real logic against this fully controlled, instantaneous fake dependency, and additionally use `mockPayment.Verify(...)` to assert that the class under test actually CALLED the dependency with the expected arguments the expected number of times — verifying both the class's behavior given a certain dependency response, and that it interacts with its dependencies correctly. This is precisely why depending on interfaces rather than concrete classes (covered in the OOP and Services & DI lessons) is considered a foundational practice for testable architecture.

🏋️ Practice: write a [Fact] test using Arrange-Act-Assert for a simple method; convert several near-duplicate [Fact] tests into one [Theory] with [InlineData]; implement IDisposable on a test class and confirm Dispose runs after each test; write a class depending on an interface and mock that dependency with Moq, verifying a call was made with specific arguments; write and run an async Task test method for an async service method; deliberately share static mutable state across two tests and observe order-dependent flakiness, then fix it via constructor-based setup.

🎯 Key Takeaways

  • xUnit uses the constructor for setup and IDisposable.Dispose() for teardown — no [SetUp]/[TearDown]; a fresh test-class instance is created per test for isolation.
  • [Fact] = one test case; [Theory] + [InlineData]/[MemberData] = the same test logic run against multiple data sets, avoiding copy-pasted near-duplicate tests.
  • Arrange-Act-Assert keeps tests readable — set up, perform one action, verify the outcome.
  • Depending on interfaces (not concrete classes) is what makes mocking with Moq possible — substitute a fake dependency with zero changes to the class under test.
  • Async test methods must be `async Task`, never `async void`, so the test runner can await completion and correctly report failures.
