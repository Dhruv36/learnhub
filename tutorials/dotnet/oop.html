OOP: Classes, Interfaces, Records Basics

C# is a class-based OOP language, but modern C# gives you three distinct shapes for modeling data: classes (mutable, identity-based), records (immutable, value-based equality), and interfaces (contracts). Choosing the right one — and understanding interface-based polymorphism — is central to idiomatic C# design and a constant interview topic.

Classes — Identity, Encapsulation, Inheritance

public class BankAccount {
    public string Owner { get; init; }         // settable only at construction (init accessor)
    public decimal Balance { get; private set; } // readable by all, settable only within the class

    public BankAccount(string owner, decimal initial) {
        Owner = owner;
        Balance = initial;
    }

    public void Deposit(decimal amount) {
        if (amount <= 0) throw new ArgumentException("Must be positive", nameof(amount));
        Balance += amount;
    }
}
// Two BankAccount instances with identical Owner/Balance are still DIFFERENT objects (== compares
// reference identity by default) — classes model things with an IDENTITY, not just a bag of data.

Records — Immutable, Value-Based Equality

public record Point(double X, double Y);        // a full class in one line: ctor, properties, equality, ToString

var p1 = new Point(1, 2);
var p2 = new Point(1, 2);
p1 == p2;               // TRUE — records compare by VALUE (all properties), unlike classes

var p3 = p1 with { X = 5 };   // non-destructive mutation — returns a NEW record with X changed, p1 untouched
// Records are the idiomatic choice for DATA — DTOs, API payloads, immutable domain values — where two
// instances with the same data SHOULD be considered equal, and you want immutability by default.
⚖️ Class vs record is a decision about identity vs value semantics. Use a class when instances represent distinct entities even with identical data (two BankAccounts both starting at $0 are still different accounts). Use a record when instances are interchangeable if their data matches (two Points at (1,2) ARE the same point). This single choice eliminates a whole category of "why doesn't equality work" bugs and makes intent explicit to every reader.

Interfaces — Contracts, Not Implementation

public interface IPaymentProcessor {
    Task ChargeAsync(decimal amount, string cardToken);
}

public class StripeProcessor : IPaymentProcessor {
    public async Task ChargeAsync(decimal amount, string cardToken) { /* ... */ }
}
public class PayPalProcessor : IPaymentProcessor {
    public async Task ChargeAsync(decimal amount, string cardToken) { /* ... */ }
}

// Consuming code depends on the ABSTRACTION, not a concrete implementation:
public class CheckoutService(IPaymentProcessor processor) {   // primary constructor (C# 12)
    public Task Checkout(decimal amount) => processor.ChargeAsync(amount, "tok_123");
}
// Swap StripeProcessor for PayPalProcessor (or a test fake) with ZERO changes to CheckoutService.
// This is the foundation of dependency injection (di-hosting lesson) and testable architecture.

Interview Questions

Q1: When would you use a class versus a record?

The decision hinges on whether instances have IDENTITY or represent pure VALUES. A class is appropriate when two instances with identical data should still be considered distinct — a BankAccount, a database Entity, a Person tracked by a system all have an identity independent of their current field values, so classes default to reference equality (== compares whether they're the same object) and are typically mutable over their lifetime as that identity's state changes. A record is appropriate when instances are fully defined by their data and two instances with the same data SHOULD be equal — a Point(1,2), a Money(100, "USD") DTO, an immutable API response payload. Records auto-generate value-based Equals/GetHashCode/ToString, are immutable by default (properties are typically init-only), and support non-destructive mutation via `with` expressions that produce a new instance rather than changing the original. In modern C# codebases, records have become the default choice for DTOs, API models, and domain value objects, while classes remain the right tool for entities with genuine identity and encapsulated behavior/state.

Q2: Why should you program against interfaces rather than concrete classes?

Depending on an interface (IPaymentProcessor) instead of a concrete implementation (StripeProcessor) decouples the consuming code from any particular implementation's details, which pays off in several ways. It enables swapping implementations without touching consumer code — moving from Stripe to PayPal, or from a real service to an in-memory fake for tests, requires only providing a different implementation of the same interface, with zero changes to CheckoutService's logic. It makes unit testing dramatically easier: a test can inject a lightweight fake or mock IPaymentProcessor that returns canned results instantly, rather than needing a real payment gateway available during tests. It also documents the actual contract a class needs, keeping coupling minimal — CheckoutService only needs "something that can charge a card," not the full surface area of a specific vendor SDK. This principle (depend on abstractions, not concretions) is the foundation dependency injection is built on, and it's why constructor parameters in well-designed C# code are typically interface types rather than concrete classes.

Q3: How do records achieve value-based equality and immutability, and how does `with` work?

When you declare a record — either the concise positional form `public record Point(double X, double Y);` or an expanded form with explicit properties — the compiler automatically generates an Equals override (and GetHashCode) that compares ALL the record's properties for equality, rather than comparing object references as a plain class would; this is why two separately-constructed Point(1,2) instances compare equal with ==, unlike two equivalent class instances. Records also generate a readable ToString() showing all property values, useful for debugging and logging. By default, a positional record's properties are init-only (settable only during construction/object initialization, not afterward), which gives you immutability without manually marking every property readonly. The `with` expression (`p1 with { X = 5 }`) provides non-destructive mutation: it doesn't modify p1 at all, but instead creates and returns a brand-new record instance that copies all of p1's property values except the ones explicitly overridden in the braces — this lets you express "a new version of this value with one field changed" while preserving full immutability of the original, which is especially valuable for functional-style state updates and avoiding shared-mutation bugs.

🏋️ Practice: model the same domain concept as both a class and a record, and compare their equality behavior; use a `with` expression to derive a modified copy of a record and confirm the original is untouched; define an interface with two implementations and write a consumer class that depends only on the interface; swap one implementation for a test fake with no changes to the consumer; refactor a deep inheritance hierarchy into composition using interfaces.

🎯 Key Takeaways

  • Classes model identity (mutable, reference equality); records model values (immutable by default, value equality auto-generated).
  • `with` expressions on records produce a new instance non-destructively — the original is never mutated.
  • Interfaces define "can do" contracts enabling multiple implementations, testability, and loose coupling; a class implements many, inherits from at most one.
  • Abstract classes suit closely related types sharing real implementation ("is a"); interfaces suit cross-cutting capabilities ("can do").
  • Program against interfaces, not concrete classes — the foundation of dependency injection and testable architecture.
