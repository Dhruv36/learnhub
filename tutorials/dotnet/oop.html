Classes & Encapsulation Object-Oriented C#

What you'll master: the anatomy of a class (fields, constructors, properties, methods), what encapsulation actually buys you, properties as the C# idiom (auto, computed, init-only, with access modifiers), the access-modifier ladder, static vs instance, object initializers and required, and the design principle of keeping objects in valid states. Prerequisites: Types (reference vs value) and Methods. Classes are how you model your domain; encapsulation is what keeps that model trustworthy as the codebase grows.

Part 1 — Anatomy of a Class

public class BankAccount
{
    // FIELD — private state, the account's data (convention: _camelCase):
    private decimal _balance;
    private readonly List<Transaction> _history = [];   // readonly: reference can't be reassigned

    // CONSTRUCTOR — establishes a VALID initial state (never a half-built object):
    public BankAccount(string owner, decimal openingBalance)
    {
        if (openingBalance < 0)
            throw new ArgumentException("Opening balance cannot be negative");
        Owner = owner;
        _balance = openingBalance;
    }

    // PROPERTY — controlled access to state (Part 3):
    public string Owner { get; }                        // read-only after construction
    public decimal Balance => _balance;                 // computed, exposes read-only view

    // METHODS — the behavior; the ONLY way to change state (invariants enforced here):
    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentException("Amount must be positive");
        _balance += amount;
        _history.Add(new Transaction(amount, DateTime.UtcNow));
    }
}

A class bundles state (fields) with the behavior that operates on it (methods), gated by constructors that guarantee an object never exists in an invalid state and properties that control access. The load-bearing idea is already visible: _balance is private, so the only way to change it is Deposit/Withdraw — which means every balance change is validated and recorded, with no exceptions. That's not bureaucracy; it's the difference between a balance you can trust and a public field anyone can set to -999999. Note readonly on the field: the reference to the history list can't be reassigned (though the list's contents still change) — a small guarantee that the collection you audit is the one that exists.

Part 2 — Encapsulation: Why Private Is the Default

// ✗ NO encapsulation — the "struct pretending to be a class" anti-pattern:
public class Order
{
    public decimal Total;             // anyone can set Total = -5
    public List<Item> Items;          // anyone can null it, clear it, add junk
    public string Status;             // "shpiped"? no validation, no closed set
}
// Every field is a public promise you can never take back, and every invariant
// (Total == sum of Items, Status ∈ known set) is unenforceable.

// ✓ WITH encapsulation — the object OWNS and PROTECTS its invariants:
public class Order
{
    private readonly List<Item> _items = [];
    public IReadOnlyList<Item> Items => _items;         // expose a READ-ONLY VIEW, not the list
    public decimal Total => _items.Sum(i => i.Price * i.Qty);   // DERIVED — can't drift
    public OrderStatus Status { get; private set; }    // settable only from inside

    public void AddItem(Item item)                     // the guarded mutation path
    {
        if (Status != OrderStatus.Draft)
            throw new InvalidOperationException("Cannot modify a submitted order");
        _items.Add(item);
    }
}
// Total can NEVER disagree with Items (it's derived). Items can't be tampered with
// (read-only view). Status transitions are controlled. The object is always valid.

Encapsulation is hiding the data and exposing only guarded operations, and its payoff compounds with codebase size: an object that protects its own invariants is one you can reason about in isolation — the balance is always non-negative, the total always matches the items, the status only moves through legal transitions, because there's no code path that could make it otherwise. The two techniques shown here recur constantly: expose collections as IReadOnlyList<T> views (never the mutable List — otherwise callers bypass every rule), and derive values like Total rather than store them (a stored total is a second source of truth that will eventually drift — the same derive-don't-store rule from the signals and state lessons, here in OOP form).

Part 3 — Properties: the C# Idiom for State Access

// AUTO-PROPERTY — compiler generates the backing field; the default for simple state:
public string Name { get; set; }                    // read-write
public int Id { get; }                              // read-only (set only in constructor)
public DateTime CreatedAt { get; init; }            // init-only: set during construction, then frozen

// ASYMMETRIC ACCESS — public read, restricted write (extremely common):
public decimal Balance { get; private set; }        // outside code reads; only THIS class writes

// COMPUTED PROPERTY — no backing field, calculated each access:
public string FullName => $"{First} {Last}";
public bool IsOverdue => DueDate < DateTime.UtcNow && !IsPaid;

// FULL PROPERTY — validation/logic in the accessors:
private decimal _price;
public decimal Price
{
    get => _price;
    set => _price = value >= 0 ? value           // 'value' is the implicit setter parameter
                              : throw new ArgumentException("Price cannot be negative");
}

// C# 13 'field' keyword — validation WITHOUT declaring the backing field yourself:
public decimal Price
{
    get;
    set => field = value >= 0 ? value : throw new ArgumentException();
}

Properties are how C# does state access — they look like fields at the call site (order.Total, account.Balance = x) but they're method pairs, so you get encapsulation without ceremony and can add validation later without changing any caller (a field-to-property change is source-compatible; that's why exposing public fields is discouraged — you can never upgrade them). The spectrum runs from auto-properties (the common case) through asymmetric access (public get; private set; — read-everywhere, write-here, the workhorse) to computed properties (derived, no storage) and full properties (validation in the setter). init-only properties are the modern immutability tool: settable during object construction, frozen forever after — the foundation for the object-initializer and required patterns in Part 5.

Part 4 — Access Modifiers & static

// The access ladder — from most to least restrictive:
private              // this class only (the DEFAULT for members; make it your reflex)
private protected    // this class + derived classes in the SAME assembly
protected            // this class + derived classes (anywhere)
internal             // this ASSEMBLY only (great for library internals)
protected internal   // this assembly OR derived classes
public               // everyone (a permanent promise — the API surface)

// Principle: start private, widen only when a real caller needs it.
// 'internal' is underused — it's how libraries keep implementation details hidden
// while sharing across their own files. (InternalsVisibleTo opens them to test projects.)

// STATIC — belongs to the TYPE, not an instance:
public class MathUtils
{
    public static double Tau => Math.PI * 2;         // no instance needed: MathUtils.Tau
    public static int Add(int a, int b) => a + b;    // MathUtils.Add(1, 2)
    private static int _instanceCount;               // shared across ALL instances (careful!)
}

// STATIC gotchas:
//  - static state is GLOBAL state: shared, thread-unsafe by default, a testing nightmare
//  - static classes can't be instantiated/inherited (Math, Console, File are static)
//  - prefer dependency injection (di-hosting.html) over static helpers for anything
//    with dependencies or state — statics are fine for PURE functions only

The access-modifier ladder is your encapsulation vocabulary, and the professional habit is start at private and widen only under pressure — every public member is a promise you maintain forever, so the smaller the public surface, the freer you are to change internals. internal is the underappreciated middle: it lets a library share code across its own files while hiding it from consumers — the boundary between "our implementation" and "our API." static members belong to the type rather than any instance; they're perfect for pure utility functions (Math.Sqrt) but a trap for anything stateful, because static state is global state — shared across the whole process, thread-unsafe unless you make it safe, and murder to test in isolation. The modern guidance: statics for pure functions, dependency injection for everything with state or dependencies.

Part 5 — Object Initializers, required & Construction Patterns

// OBJECT INITIALIZER — set properties after construction, readable and order-free:
var user = new User
{
    Name = "Ada",
    Email = "ada@example.com",
    Role = Role.Admin,
};

// The PROBLEM initializers created: nothing forces you to set Name/Email —
// 'new User { }' compiled fine and shipped a half-built object.

// 'required' (C# 11) — the compiler ENFORCES that these are set at construction:
public class User
{
    public required string Name { get; init; }        // MUST be set in the initializer…
    public required string Email { get; init; }       // …or it's a COMPILE ERROR
    public Role Role { get; init; } = Role.Member;    // optional with a default
}
new User { Name = "Ada" };                            // ✗ compile error: Email not set
new User { Name = "Ada", Email = "a@b.com" };         // ✓

// PRIMARY CONSTRUCTORS (C# 12) — parameters in the class header, in scope everywhere:
public class Repository(DbContext db, ILogger<Repository> log)   // captured, no field ceremony
{
    public async Task Save(Entity e)
    {
        log.LogInformation("Saving {Id}", e.Id);      // 'db' and 'log' just… available
        db.Add(e);
        await db.SaveChangesAsync();
    }
}

// Construction decision guide:
//  required data with invariants  → constructor (validate, guarantee valid state)
//  simple data bags / DTOs        → required init-only props + object initializer
//  dependencies (services)        → primary constructor (or classic ctor injection)

Object initializers made construction readable but reopened the half-built-object problem that constructors solved — required closes it back up at compile time, giving you the best of both: the fluent, order-free initializer syntax and the guarantee that mandatory properties are set. The result is that C# now has three good construction styles for three situations: a validating constructor when invariants must be enforced with logic, required init-only properties for data-carrying types (DTOs, configuration), and primary constructors for capturing dependencies without field boilerplate. Choosing the right one is a real design signal — a constructor that merely assigns properties one-to-one probably wants to be init-only required properties instead.

Part 6 — Deep Dive: Designing for Valid States & the Anemic-Model Debate

// THE ANEMIC DOMAIN MODEL — data with no behavior, logic scattered into "services":
public class Order { public List<Item> Items { get; set; } public OrderStatus Status { get; set; } }
public class OrderService
{
    public void Submit(Order o)                        // the object can't protect itself —
    {                                                  //   every service must remember the rules
        if (o.Items.Count == 0) throw new(...);
        o.Status = OrderStatus.Submitted;              // and any OTHER code can skip this and
    }                                                  //   just set o.Status directly
}
// Symptom: invariants live in services, not the object → they get bypassed and duplicated.

// THE RICH DOMAIN MODEL — the object owns its rules; illegal states are unreachable:
public class Order
{
    private readonly List<Item> _items = [];
    public IReadOnlyList<Item> Items => _items;
    public OrderStatus Status { get; private set; } = OrderStatus.Draft;

    public void Submit()
    {
        if (_items.Count == 0)
            throw new InvalidOperationException("Cannot submit an empty order");
        Status = OrderStatus.Submitted;                // the ONLY way to become Submitted
    }
    // "make illegal states unrepresentable": there is no path to Submitted-and-empty.
}

// The nuance (this is a real, ongoing debate):
//  - RICH models shine for genuine domain logic (orders, accounts, policies) — DDD's core
//  - ANEMIC models are FINE for DTOs, API contracts, config — data really is just data
//  - the mistake is applying either dogmatically. Match the model to the object's JOB.

The deepest lesson in OOP isn't syntax — it's making illegal states unrepresentable. When an object owns its invariants (private state + guarded mutations), whole categories of bug simply cannot occur: there is no submitted-but-empty order, no negative balance, no status that skipped a transition. The contrast is the anemic domain model — objects that are bare data with all the logic drained into "service" classes — which scatters and duplicates rules until they get bypassed. This is a genuine architectural debate, not a settled rule: rich models are the right call for objects with real behavior and invariants (the domain layer), while anemic data-holders are entirely appropriate for DTOs, API request/response types, and configuration where the data is the point. The senior skill is refusing to apply either dogmatically — a bank account should be rich, a JSON payload should be anemic, and knowing which is which is the judgment that matters.

Senior framing: encapsulation reviews probe for whether objects can be put into invalid states, and the tells are consistent. Public mutable collections (public List<T> Items { get; set; }) are a red flag — callers can null it, clear it, or add unvalidated items, so expose IReadOnlyList<T> and guard additions through methods. Public setters on things with invariants (a settable Balance) defeat the whole class — use private set or make it derived. Stored values that duplicate other state (a Total field alongside Items) will drift — derive them. And primitive-typed properties that could hold nonsense (string Status accepting "shpiped") want enums or value objects. The unifying question a senior asks of any class: "what states can this object be in, and are the illegal ones reachable?" — if illegal states are reachable, the encapsulation is incomplete regardless of how many private keywords are present. The interview version: "how do you prevent an order from being submitted twice?" — the answer is the object's own guard on the state transition, not a check in the caller.

Interview Questions

Q1 (Beginner): What is encapsulation, and what concrete benefit does making a field private give you?

Encapsulation is hiding an object's internal state and exposing only controlled operations that preserve its invariants. Making _balance private means the only way to change it is through methods like Deposit/Withdraw, so every change is validated (amount positive), recorded (transaction log), and legal (business rules) — with no code path that can bypass them. Concretely: the balance can never be negative, never change without an audit entry, and the object can be reasoned about in isolation because it protects itself. A public field surrenders all of that permanently.

Q2 (Beginner): Why use properties instead of public fields, given they look the same at the call site?

Properties are method pairs (get/set) that look like fields but let you (1) add validation in the setter without changing any caller, (2) make access asymmetric (public read, private write), (3) compute values with no backing storage, and (4) make state read-only or init-only. The decisive practical reason: exposing a public field is a one-way door — you cannot later turn it into a property without recompiling every consumer (it's a binary-breaking change), so you'd have to ship it un-validatable forever. Properties cost nothing extra (auto-properties are one line) and keep every future option open.

Q3 (Mid): Compare init, private set, and no setter for a property. When is each right?

No setter (get; only): the value is set in the constructor and never changes — truly immutable state established at construction. init: settable during object initialization (constructor or object initializer) then frozen — immutable after construction but allowing the fluent new T { X = ... } syntax; ideal for DTOs and records. private set: mutable, but only from inside the declaring class — the value changes over the object's life but only through the class's own guarded methods (a Balance that updates via Deposit). Rule of thumb: prefer the most restrictive that works — no-setter or init for values that shouldn't change after construction, private set only when the object genuinely mutates over time.

Q4 (Mid): What does "make illegal states unrepresentable" mean, and how do private state plus guarded methods achieve it?

It means designing types so that invalid combinations of state can't be constructed at all, moving whole bug classes from "caught at runtime (maybe)" to "impossible by construction." You achieve it by keeping state private (no external code can set it) and exposing only operations that enforce the rules: if the only way to reach Status.Submitted is a Submit() method that first checks the order is non-empty, then a submitted-and-empty order literally cannot exist — there's no code path to it. Contrast a public settable Status, where any caller can produce any state. Combined with enums (closed value sets), value objects, and non-nullable types, this pushes correctness into the type system where the compiler and the object's own guards enforce it, rather than relying on every caller remembering the rules.

Q5 (Senior): Explain the anemic vs rich domain model debate. Where does each belong, and what's the failure mode of applying one dogmatically?

An anemic model is data with no behavior — public getters/setters, all logic in separate "service" classes. A rich model puts behavior and invariant-enforcement on the object itself (private state, guarded methods). Rich models belong to the domain: objects with real business rules and invariants (orders, accounts, pricing policies) — this is the core of Domain-Driven Design, and it prevents rules from being scattered, duplicated, and bypassed across services. Anemic models are entirely appropriate for objects that are genuinely just data: DTOs, API request/response contracts, configuration, view models — where imposing behavior would be artificial. The failure modes are symmetric: dogmatic anemia scatters domain logic into a web of services where invariants get skipped (someone sets Status directly), while dogmatic richness bloats simple data-transfer types with pointless ceremony and fights serializers. The senior judgment is matching the model to the object's job — and recognizing that most systems have both kinds and should.

Q6 (Senior): Design review: a class has public decimal Total { get; set; }, public List<LineItem> Items { get; set; }, and public string Status { get; set; }. Redesign for correctness and explain each change.

Every property here allows an invalid state. Total: settable and independent of Items, so it can disagree with the actual sum — make it a computed property (public decimal Total => _items.Sum(i => i.Price * i.Quantity)) so it's derived and can never drift. Items: a public settable List can be nulled, cleared, or filled with unvalidated items, and additions bypass any rules — back it with a private _items, expose IReadOnlyList<LineItem>, and add items only through an AddItem method that validates and checks the order's state. Status: a settable string can hold "shpiped" or skip transitions — make it an OrderStatus enum (closed set) with private set, changed only through methods that enforce legal transitions (Submit(), Ship()), so submitted-empty and status-skips become impossible. The through-line: private state, derived values, read-only collection views, enums for closed sets, and mutation only via guarded methods — after which the object is always in a valid state by construction, and its invariants live in one place instead of in the memory of every caller.

🏋️ Exercises

1 (Easy): Write a Temperature class storing Celsius privately, exposing read-only Celsius and computed Fahrenheit properties, that rejects values below absolute zero (−273.15) in its constructor.
Solution
public class Temperature
{
    public double Celsius { get; }                     // read-only, set in ctor
    public double Fahrenheit => Celsius * 9 / 5 + 32;  // computed — always consistent

    public Temperature(double celsius)
    {
        if (celsius < -273.15)
            throw new ArgumentOutOfRangeException(nameof(celsius),
                "Below absolute zero");
        Celsius = celsius;
    }
}
// No way to construct an invalid temperature; Fahrenheit can never disagree with Celsius
// because it's derived, not stored.
2 (Easy): Convert this leaky class to properly encapsulated, listing what each change prevents: 
public class Playlist {
    public List<string> Songs = new();
    public int Count;
}
Solution
public class Playlist
{
    private readonly List<string> _songs = [];
    public IReadOnlyList<string> Songs => _songs;       // read-only view: no external clear/null
    public int Count => _songs.Count;                   // derived: can't disagree with Songs

    public void Add(string song)
    {
        if (string.IsNullOrWhiteSpace(song))
            throw new ArgumentException("Song title required");
        _songs.Add(song);                              // the ONLY add path — validated
    }
    public bool Remove(string song) => _songs.Remove(song);
}
// Prevented: public field 'Songs' being nulled/cleared/reassigned; 'Count' drifting out
// of sync (it was independent before!); unvalidated (empty) songs entering the list.
3 (Medium): Build a User type with required Name and Email (init-only), an optional Role defaulting to Member, and a computed IsAdmin. Show a valid and an invalid construction.
Solution
public enum Role { Member, Moderator, Admin }

public class User
{
    public required string Name { get; init; }
    public required string Email { get; init; }
    public Role Role { get; init; } = Role.Member;
    public bool IsAdmin => Role == Role.Admin;
}

var ok = new User { Name = "Ada", Email = "ada@x.com", Role = Role.Admin };
Console.WriteLine(ok.IsAdmin);                          // True

4 (Medium): Design a ShoppingCart where Total and ItemCount can never be inconsistent with the actual items, and items can only be added with quantity ≥ 1.
Solution
public record CartLine(string Sku, decimal UnitPrice, int Quantity);

public class ShoppingCart
{
    private readonly List<CartLine> _lines = [];
    public IReadOnlyList<CartLine> Lines => _lines;
    public int ItemCount => _lines.Sum(l => l.Quantity);          // derived
    public decimal Total => _lines.Sum(l => l.UnitPrice * l.Quantity);  // derived

    public void Add(string sku, decimal unitPrice, int quantity)
    {
        if (quantity < 1) throw new ArgumentException("Quantity must be at least 1");
        if (unitPrice < 0) throw new ArgumentException("Price cannot be negative");

        var existing = _lines.FindIndex(l => l.Sku == sku);
        if (existing >= 0)                                          // merge same SKU
            _lines[existing] = _lines[existing] with
                { Quantity = _lines[existing].Quantity + quantity };
        else
            _lines.Add(new CartLine(sku, unitPrice, quantity));
    }
}
// Total and ItemCount are computed from _lines every access — structurally impossible
// to have a $50 total with an empty cart. All mutation flows through the validated Add.
5 (Hard): Implement a BankAccount that makes these illegal states unrepresentable: overdrawn balance, deposits/withdrawals of ≤ 0, and operations on a closed account. Include a transaction history that callers can read but not tamper with.
Solution
public enum AccountState { Open, Closed }
public record Transaction(decimal Amount, DateTime At, string Kind);

public class BankAccount
{
    private decimal _balance;
    private readonly List<Transaction> _history = [];
    public AccountState State { get; private set; } = AccountState.Open;

    public decimal Balance => _balance;
    public IReadOnlyList<Transaction> History => _history;   // read-only view

    public BankAccount(decimal opening)
    {
        if (opening < 0) throw new ArgumentException("Opening balance cannot be negative");
        _balance = opening;
        _history.Add(new Transaction(opening, DateTime.UtcNow, "Open"));
    }

    public void Deposit(decimal amount)
    {
        EnsureOpen();
        if (amount <= 0) throw new ArgumentException("Deposit must be positive");
        _balance += amount;
        _history.Add(new Transaction(amount, DateTime.UtcNow, "Deposit"));
    }

    public void Withdraw(decimal amount)
    {
        EnsureOpen();
        if (amount <= 0) throw new ArgumentException("Withdrawal must be positive");
        if (amount > _balance)                              // the overdraft guard
            throw new InvalidOperationException("Insufficient funds");
        _balance -= amount;
        _history.Add(new Transaction(-amount, DateTime.UtcNow, "Withdraw"));
    }

    public void Close()
    {
        EnsureOpen();
        State = AccountState.Closed;
    }

    private void EnsureOpen()
    {
        if (State == AccountState.Closed)
            throw new InvalidOperationException("Account is closed");
    }
}
// Every illegal state has no reachable path: balance can't go negative (withdraw guard),
// no zero/negative operations (amount guards), no post-close mutation (EnsureOpen),
// and history is exposed only as a read-only view — auditable but untamperable.
6 (Hard): Given an anemic Order with a public settable Status and an OrderService.Submit that sets it, refactor to a rich model enforcing the transition Draft→Submitted→Shipped (no skips, no reversals). Explain what the service loses and gains.
Solution
public enum OrderStatus { Draft, Submitted, Shipped }

public class Order
{
    private readonly List<LineItem> _items = [];
    public IReadOnlyList<LineItem> Items => _items;
    public OrderStatus Status { get; private set; } = OrderStatus.Draft;

    public void AddItem(LineItem item)
    {
        if (Status != OrderStatus.Draft)
            throw new InvalidOperationException("Can only add items while Draft");
        _items.Add(item);
    }

    public void Submit()
    {
        if (Status != OrderStatus.Draft)
            throw new InvalidOperationException($"Cannot submit from {Status}");
        if (_items.Count == 0)
            throw new InvalidOperationException("Cannot submit an empty order");
        Status = OrderStatus.Submitted;
    }

    public void Ship()
    {
        if (Status != OrderStatus.Submitted)
            throw new InvalidOperationException($"Cannot ship from {Status}");
        Status = OrderStatus.Shipped;
    }
}

// The service becomes a thin orchestrator (persistence, notifications) — it no longer
// OWNS the transition rules:
public class OrderService(IOrderRepository repo)
{
    public async Task Submit(int id)
    {
        var order = await repo.Get(id);
        order.Submit();                    // the ORDER enforces the rule, not the service
        await repo.Save(order);
    }
}

// LOSES: the ability (and temptation) to set Status arbitrarily — the service can't
//        skip Draft→Shipped or reverse Shipped→Draft even if it tried; it won't compile
//        its way around private set, and Ship()/Submit() reject illegal transitions.
// GAINS: the rules live in ONE place (the Order), can't be duplicated or bypassed by a
//        second service, and are unit-testable without any service or database. The
//        state machine is now a property of the domain object, which is where it belongs.

🎯 Key Takeaways

  • A class bundles private state with guarded behavior; constructors guarantee a valid initial state, and methods are the only path to change it — so invariants hold by construction.
  • Encapsulation (private state + controlled operations) lets an object protect its own invariants; expose collections as IReadOnlyList<T> views and derive values rather than storing duplicates.
  • Properties are the C# state idiom: auto for simple cases, get; private set; for read-everywhere/write-here, computed for derived, full/field for validation — and they're upgradeable from auto without breaking callers (unlike public fields).
  • Start members private and widen only under real need; internal hides implementation across a library's own files; static is for pure functions, not shared mutable state.
  • required init-only properties give object-initializer syntax the safety of a constructor; choose constructor (invariants), required init props (data types), or primary constructor (dependencies) by the object's job.
  • "Make illegal states unrepresentable": rich models for behavior-bearing domain objects, anemic models for DTOs/config — match the model to the object's role rather than applying either dogmatically.
