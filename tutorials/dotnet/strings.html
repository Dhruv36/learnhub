Strings & Text Foundations

What you'll master: why strings are immutable and everything that follows from it (StringBuilder, interning, the == exception), the essential API surface, interpolation and format strings, comparison done right (ordinal vs culture — a genuine production-bug factory), encoding and UTF-8/UTF-16 reality, and raw string literals. Prerequisites: Types and Control Flow. Half of all code manipulates text; the difference between fluent and fumbling here is visible daily.

Part 1 — Immutability: the Design Decision Everything Follows From

string s = "hello";
s.ToUpper();                    // returns "HELLO" — s is UNCHANGED
Console.WriteLine(s);           // "hello" — the classic beginner surprise
s = s.ToUpper();                // ✓ every "mutating" method RETURNS a new string

// Why immutable? (same reasoning as records/functional patterns):
//  - safe sharing: any code can hold a reference without defensive copies
//  - hashable: dictionary keys can't change under the table
//  - thread-safe by construction
//  - enables INTERNING: identical literals share ONE object:
string a = "config", b = "config";
ReferenceEquals(a, b)           // true — compile-time literals are interned

// The == exception (from control-flow.html): string is a reference type
// with VALUE equality — == compares CONTENT, ordinally:
new string("hi".ToCharArray()) == "hi"      // true (content), even if not same object

// The cost of immutability — naive concatenation is O(n²):
var report = "";
for (int i = 0; i < 10_000; i++)
    report += lines[i];         // EVERY += allocates a NEW string and copies everything so far

Strings being immutable is the platform betting that safe sharing beats in-place edits — the same trade functional programming makes, and it's why strings can be dictionary keys, cross threads freely, and intern literals. The price is the accumulation anti-pattern: += in a loop re-copies the entire accumulated string each pass — fine for three fragments, quadratic disaster for ten thousand. Every consequence in this lesson (StringBuilder, spans, the API's return-a-new-string shape) traces back to this one decision.

Part 2 — StringBuilder & Efficient Assembly

// The fix for accumulation — a MUTABLE text buffer:
var sb = new StringBuilder(capacity: 4096);         // pre-size when you can estimate
foreach (var line in lines)
    sb.AppendLine(line);                             // amortized O(1) appends
sb.Append("total: ").Append(total);                  // fluent chaining; typed overloads
var report = sb.ToString();                          // ONE final string

// When you DON'T need it (know the alternatives):
var csv = string.Join(",", items);                   // joining a collection: Join wins
var path = $"{dir}/{name}.{ext}";                    // fixed small count: interpolation —
                                                     //   compiler optimizes it well
var padded = value.PadLeft(8, '0');                  // purpose-built APIs beat assembly

Rule of thumb:
  fixed handful of pieces        → interpolation ($"")
  collection with separator      → string.Join
  LOOP that grows a string       → StringBuilder
  hot path parsing/slicing       → Span<char> (memory.html — zero-allocation)

StringBuilder is the mutable escape hatch: one internal buffer, amortized-constant appends, one materialized string at the end — turning O(n²) into O(n). The senior nuance is knowing when it's not the answer: string.Join for the join-a-collection case (clearer and already optimal), interpolation for small fixed assemblies (modern interpolation compiles to efficient handlers, not naive concat), and spans when even one allocation per operation is too many. Reaching for StringBuilder to join three fragments is the cargo-cult version of this lesson; reaching for += in a report loop is the naive version. Match the tool to the shape.

Part 3 — The Essential API Surface

var s = "  Order #42: 3x Widget  ";

// Inspect:
s.Length                      // 25 (UTF-16 code units — see Part 5!)
s.Contains("Widget")          // true
s.StartsWith("Order")         // false! (leading spaces) — Trim first
s.IndexOf('#')                // 8  (-1 if absent)
string.IsNullOrEmpty(x)       // null or ""
string.IsNullOrWhiteSpace(x)  // null, "", or only spaces/tabs — the usual input guard

// Transform (each returns a NEW string):
s.Trim()                      // "Order #42: 3x Widget"  (TrimStart/TrimEnd exist)
s.ToUpperInvariant()          // casing for LOGIC → invariant; for DISPLAY → user culture
s.Replace("Widget", "Gadget")
s.Substring(2, 5)             // legacy slicing — prefer ranges: s[2..7]
s[^7..]                       // "Widget  " — index/range syntax works on strings

// Split & join:
"a,b,,c".Split(',')                                        // ["a","b","","c"]
"a, b, ,c".Split(',', StringSplitOptions.TrimEntries
                    | StringSplitOptions.RemoveEmptyEntries) // ["a","b","c"] — the usual combo
string.Join(" | ", parts)

// Search from the end, char vs string overloads (char is faster):
s.LastIndexOf('x')            // prefer char overloads where applicable

Fluency here is mundane and valuable: knowing IsNullOrWhiteSpace is the input guard, that Split's options flags handle the trim-and-drop-empties dance, that ranges (s[2..7], s[^6..]) supersede Substring arithmetic, and that every transform returns a new string (chain them, or capture the result). One habit worth installing now: casing methods are a fork in the road — ToUpperInvariant for logic (keys, comparisons, protocols), culture-aware ToUpper() only for text a human will read. Part 4 explains why that's not pedantry.

Part 4 — Comparison & Culture: the Production-Bug Factory

// THE TURKISH-I BUG — the canonical example:
"FILE".ToLower()              // on tr-TR culture: "fıle" (dotless ı!) — not "file"
// So this innocent check FAILS for Turkish users:
if (input.ToLower() == "file") …          // ✗ culture-sensitive casing in LOGIC

// The right tool — explicit comparison MODES:
string.Equals(a, b, StringComparison.OrdinalIgnoreCase)     // logic: byte-value compare
s.StartsWith("pre", StringComparison.Ordinal)               // protocols, headers, paths
s.Contains("q", StringComparison.CurrentCultureIgnoreCase)  // user-facing search

The decision table:
  Ordinal              identifiers, keys, protocol tokens, file paths   FAST, deterministic
  OrdinalIgnoreCase    case-insensitive versions of the above (headers, env vars)
  CurrentCulture       comparing/sorting text SHOWN to the user
  InvariantCulture     culture-independent but linguistic — rare; stable sort keys

// Sorting user-visible lists (same lesson as ../javascript/dates-intl.html):
names.Sort(StringComparer.CurrentCulture);          // humans expect é near e
ids.Sort(StringComparer.Ordinal);                    // machines expect determinism

// Dictionary keys — bake the comparison INTO the collection:
var headers = new Dictionary<string, string>(StringComparer.OrdinalIgnoreCase);
headers["Content-Type"] = "application/json";
headers["content-type"]                              // found ✓ — no ToLower() key mangling

String comparison has a hidden parameter — which alphabet rules apply — and letting it default is how "works here, breaks in Istanbul" bugs ship: culture-aware casing/comparison follows linguistic rules that vary (Turkish dotted/dotless i being the famous case), while ordinal compares raw values. The professional discipline: every comparison states its mode — ordinal(-ignore-case) for anything machine-meaningful, culture-aware only for human-facing text — and collections that key on strings get a StringComparer instead of normalizing keys by hand. Analyzers (CA1310 et al.) can enforce "no bare ToLower-compare" — turn them on.

Part 5 — Interpolation, Format Strings & Raw Literals

var name = "Ada"; var total = 1234.5m; var when = DateTime.UtcNow;

// Interpolation with FORMAT SPECIFIERS after the colon:
$"{name}: {total:C}"           // "Ada: $1,234.50" (current culture!)
$"{total:N2}"                  // "1,234.50"   number, 2 decimals
$"{0.875:P1}"                  // "87.5 %"     percent
$"{42:D5}"                     // "00042"      padded int
$"{when:yyyy-MM-dd HH:mm}"     // custom date format
$"{when:O}"                    // ISO 8601 round-trip — logs & wire
$"{name,10}" / $"{name,-10}"   // align right/left in 10 chars (tables)

// Culture, again — interpolation uses CURRENT culture by default:
FormattableString msg = $"total={total}";
string forLog = FormattableString.Invariant($"total={total}");   // deterministic logs
// (or string.Create(CultureInfo.InvariantCulture, $"…") — types.html exercise)

// RAW STRING LITERALS (C# 11) — quotes and backslashes without escaping:
var json = """
    { "name": "Ada", "role": "admin" }
    """;                        // closing quotes' indentation strips leading whitespace!
var regex = """C:\Users\(\w+)\.config""";      // no \\ doubling
var block = $$"""{ "total": {{total}} }""";    // $$ = interpolation with {{ }} braces

Interpolation is the default string-assembly syntax, and format specifiers are its power feature — one vocabulary (C, N, P, D, date patterns, alignment) that also works in ToString(...) and string.Format. The recurring culture theme applies with force: interpolated output is culture-sensitive, so log lines and wire payloads need the invariant forms shown. Raw literals ended the escaping misery for embedded JSON/regex/SQL — adopt them wherever a string contains quotes or backslashes, and note the elegant indentation rule: the closing delimiter's position defines what leading whitespace gets stripped.

Part 6 — Deep Dive: What a String Actually Is (UTF-16, Encoding, Spans)

// A .NET string = an immutable sequence of UTF-16 CODE UNITS (chars) — not "characters":
"héllo".Length         // 5
"🎸".Length            // 2!  — emoji outside the BMP needs a SURROGATE PAIR (2 chars)
"🎸"[0]                // '\ud83c' — half a guitar: meaningless alone
"é".Length             // 1 or 2! (é can be one codepoint or e + combining accent)

// Iterating USER-PERCEIVED characters (grapheme clusters):
var e = StringInfo.GetTextElementEnumerator("👨‍👩‍👧 x");
// family emoji = ONE text element, SEVERAL codepoints, MANY chars
// Rule: .Length and [i] are fine for ASCII-ish protocol text; for arbitrary user text,
// truncating/reversing/counting by char is WRONG — use StringInfo / Rune.

// ENCODING — strings are in-memory; bytes need an encoding chosen:
byte[] utf8 = Encoding.UTF8.GetBytes("héllo");       // the wire/disk default: UTF-8
string back = Encoding.UTF8.GetString(utf8);
// Mismatched encode/decode = mojibake ("hÃ©llo") — the bug where é becomes Ã©.
// Modern APIs (JSON, HttpClient, File.*Text) default to UTF-8 — keep it that way.

// SPANS teaser (full treatment memory.html) — slicing WITHOUT allocating:
ReadOnlySpan<char> span = "Order #42: 3x Widget".AsSpan();
var idPart = span[7..9];                              // a VIEW into the string — 0 allocations
int id = int.Parse(idPart);                           // parsers accept spans directly
// In hot parsing loops, span slicing + span-based Parse eliminates the per-piece
// substring allocations entirely — the platform's performance story reaching text.

The mental model correction that separates tiers: string.Length counts UTF-16 code units, not characters — emoji, many scripts, and combining marks span multiple chars, so index-and-truncate logic that "works in testing" corrupts real user text (cutting a surrogate pair in half produces invalid strings). For user text, think in grapheme clusters (StringInfo) or codepoints (Rune); reserve char-indexing for protocol-ish ASCII. Encoding is the sibling fact: strings are UTF-16 in memory, the world is UTF-8 on the wire, and every string↔bytes crossing names an encoding — implicitly (modern UTF-8 defaults) or disastrously (mojibake). Spans preview where the performance track goes: text processing without allocation, using views instead of copies.

Senior framing: text handling maturity shows up in review as a short checklist: comparisons state their StringComparison (and analyzers enforce it); collections keyed by string carry a StringComparer; logs and wire formats are invariant-culture and ISO-8601; user text is never truncated by char index (UI ellipsis is the UI's job; if you must, cut at grapheme boundaries); parsing hot paths use spans; and SQL/HTML/JSON are never assembled by concatenation — parameterized queries, encoders, and serializers exist because "just build the string" is the injection-bug factory (JS security lesson's argument, language-independent). None of these are clever; all of them are the difference between text code that survives international users and text code that generates the weirdest bugs in the backlog.

Interview Questions

Q1 (Beginner): Why are .NET strings immutable, and what are two consequences a developer must know?

Immutability makes strings safely shareable (no defensive copies), valid dictionary keys (hash can't drift), thread-safe, and internable (identical literals share one object). Consequences: (1) every "modifying" method returns a new string — s.Trim() without capturing the result is a no-op, the classic beginner bug; (2) repeated concatenation is quadratic — each += copies the whole accumulated string, so loops need StringBuilder (or string.Join). Bonus: string is the reference type with value-semantics == — content comparison, ordinally.

Q2 (Beginner): When do you use StringBuilder vs interpolation vs string.Join?

By shape: a fixed handful of pieces → interpolation ($"{a}: {b}") — clearest, and the compiler generates efficient code; a collection with a separator → string.Join(",", items) — purpose-built and already optimal; a loop that grows text incrementally → StringBuilder — one buffer, amortized O(1) appends, one final string. Using StringBuilder for three fragments is ceremony; using += in a 10k-iteration loop is quadratic. (Hot parsing paths have a fourth tool: spans, which avoid allocation entirely.)

Q3 (Mid): Explain the Turkish-i bug and the discipline that prevents the whole class.

Culture-aware casing follows the current culture's linguistic rules: in Turkish, lowercase of 'I' is dotless 'ı' and uppercase of 'i' is dotted 'İ'. So input.ToLower() == "file" fails on tr-TR machines for input "FILE" — logic that only ever tested on en-US silently breaks by deployment geography. The discipline: comparisons and casing used for logic (keys, tokens, paths, protocol values) are always ordinal — string.Equals(a, b, StringComparison.OrdinalIgnoreCase), ToUpperInvariant — while culture-aware operations are reserved for text humans read. Structurally: StringComparer.OrdinalIgnoreCase baked into dictionaries/sorts, and code analyzers that flag bare ToLower() comparisons.

Q4 (Mid): "🎸".Length returns 2. Explain, and state what this means for substring/truncate logic.

Length counts UTF-16 code units. The guitar emoji lives outside the Basic Multilingual Plane, so UTF-16 encodes it as a surrogate pair — two chars that are only meaningful together. Combining sequences go further: "é" can be two units (e + combining accent), and family emoji join multiple codepoints with zero-width joiners. Consequences: char-index truncation can split a surrogate pair (producing invalid text) or separate a base character from its accent; "max 10 characters" validated by Length rejects or miscounts real input. For user-perceived characters, use StringInfo (grapheme clusters) or Rune (codepoints); reserve char indexing for known-ASCII protocol text.

Q5 (Senior): A CSV export shows "Ã©" where "é" should be, and a log pipeline intermittently fails to parse numbers. Diagnose both.

"Ã©" is mojibake with a precise signature: text was encoded as UTF-8 ("é" → two bytes 0xC3 0xA9) but decoded as Latin-1/Windows-1252, which renders those bytes as 'Ã' + '©'. Somewhere the byte↔string crossing disagreed on encoding — typically a StreamWriter/reader given no encoding on a legacy default, an Excel import assuming ANSI, or an HTTP response missing charset. Fix: UTF-8 explicitly at every boundary (and a BOM or charset declaration where the consumer needs the hint — Excel infamously wants the BOM). The log-parse failures are the culture bug: numbers were written with culture-defaulted ToString() on servers with different locales — "1.5" on one box, "1,5" on another — so the parser fails by deployment geography. Fix: invariant culture on write and parse for all machine-consumed text, ISO-8601 (:O) for timestamps, plus a regression test running under a non-US culture (set the thread culture in the test) so the class stays dead.

Q6 (Senior): Design the string-handling standards section of a team's code guidelines — the five rules with their rationale.

(1) Every comparison states its mode: ordinal(-ignore-case) for machine text, culture for display; enforced by analyzer (kills Turkish-i and heisenbug sorts). (2) Machine text is invariant + ISO-8601: logs, wire formats, storage — deterministic regardless of server locale; user-facing formatting is explicitly cultured. (3) No concatenation across trust boundaries: parameterized SQL, encoder-mediated HTML, Uri builders — injection is a string-assembly disease. (4) Assembly by shape: interpolation for small fixed, Join for collections, StringBuilder for loops, spans in measured hot paths — with the O(n²) += pattern called out by name. (5) User text is graphemes, not chars: no char-index truncation of arbitrary input; length limits and slicing respect text elements; UTF-8 declared at every byte boundary. Each rule names the bug it prevents — guidelines without rationale rot; these five map one-to-one onto the incidents every team eventually has.

1 (Easy): Fix both bugs: 
var input = Console.ReadLine();
input.Trim();
if (input.ToLower() == "quit") Environment.Exit(0);
Solution
var input = Console.ReadLine();
input = input?.Trim() ?? "";                          // bug 1: Trim() RETURNS the result
if (string.Equals(input, "quit",                       // bug 2: culture-sensitive ToLower
        StringComparison.OrdinalIgnoreCase))           //   → ordinal comparison for logic
    Environment.Exit(0);
// Also handled: ReadLine can return null (end of input) — the ?? guard.
2 (Easy): Produce this exact table using interpolation alignment and format specifiers: 
Widget       |     19.99 |  87.5 %
Gadget Pro   |  1,204.50 |   3.2 %
Solution
var rows = new[] { ("Widget", 19.99m, 0.875), ("Gadget Pro", 1204.50m, 0.032) };
foreach (var (name, price, share) in rows)
    Console.WriteLine($"{name,-12} | {price,9:N2} | {share,6:P1}");
// {name,-12}  left-align in 12 chars;  {price,9:N2} right-align 9, thousands+2dp;
// {share,6:P1} percent with 1 decimal. Alignment before the colon, format after.
3 (Medium): Write Slugify(string title): "Höllo, Wörld! (2nd Edition)" → "hollo-world-2nd-edition". Lowercase invariantly, strip accents, keep letters/digits, hyphens between words.
Solution
string Slugify(string title)
{
    // 1. Decompose accents: é → e + combining mark (NFD normalization):
    var decomposed = title.Normalize(NormalizationForm.FormD);
    var sb = new StringBuilder(decomposed.Length);
    var lastWasHyphen = true;                          // suppress leading hyphen

    foreach (var ch in decomposed)
    {
        if (CharUnicodeInfo.GetUnicodeCategory(ch) == UnicodeCategory.NonSpacingMark)
            continue;                                  // 2. drop the combining marks
        if (char.IsAsciiLetterOrDigit(ch))
        {
            sb.Append(char.ToLowerInvariant(ch));      // 3. invariant lowercase (logic!)
            lastWasHyphen = false;
        }
        else if (!lastWasHyphen)                       // 4. runs of junk → ONE hyphen
        {
            sb.Append('-');
            lastWasHyphen = true;
        }
    }
    return sb.ToString().TrimEnd('-');
}
// Slugify("Höllo, Wörld! (2nd Edition)") → "hollo-world-2nd-edition"
// Lesson pieces: normalization for accents, StringBuilder for the loop,
// invariant casing, state machine instead of regex chains.
4 (Medium): Build a case-insensitive word-frequency counter for a text, returning the top 5 — using the right comparer (not ToLower), Split options, and ordering.
Solution
IEnumerable<(string Word, int Count)> TopWords(string text, int top = 5)
{
    var counts = new Dictionary<string, int>(StringComparer.OrdinalIgnoreCase);
    //            ^ the comparer lives IN the dictionary — no key mangling, original
    //              casing of first occurrence is preserved for display

    var words = text.Split([' ', '\t', '\n', '\r', '.', ',', '!', '?', ';', ':'],
                           StringSplitOptions.RemoveEmptyEntries);
    foreach (var w in words)
        counts[w] = counts.GetValueOrDefault(w) + 1;

    return counts.OrderByDescending(kv => kv.Value)
                 .ThenBy(kv => kv.Key, StringComparer.OrdinalIgnoreCase)  // stable ties
                 .Take(top)
                 .Select(kv => (kv.Key, kv.Value));
}
// "The the THE cat" → [("The", 3), ("cat", 1)] — one bucket for all "the" casings,
// zero ToLower() allocations, and deterministic ordering on ties.
5 (Hard): Demonstrate the mojibake round-trip: encode "café" as UTF-8, decode it wrongly as Latin-1, show the damage, then show why re-encoding the damaged string can't restore it — and the actual fix.
Solution
var original = "café";
byte[] utf8 = Encoding.UTF8.GetBytes(original);          // é → 0xC3 0xA9 (two bytes)

var latin1 = Encoding.Latin1;
string damaged = latin1.GetString(utf8);                  // "cafÃ©" — each UTF-8 byte
Console.WriteLine(damaged);                               //   read as its OWN Latin-1 char

// Why you can't always "fix it in post":
// The REVERSIBLE case (this one): Latin-1 maps bytes 1:1, so:
byte[] recovered = latin1.GetBytes(damaged);              // get the original bytes back
Console.WriteLine(Encoding.UTF8.GetString(recovered));    // "café" — lucky!
// The IRREVERSIBLE case: decode as Windows-1252 or ASCII instead — bytes that have
// no mapping become '?' or best-fit substitutions. Information is DESTROYED; no
// re-encoding recovers it. (ASCII decoding: 0xC3 → '?', gone forever.)

// The actual fix is prevention: declare encoding at EVERY byte boundary —
// new StreamReader(path, Encoding.UTF8), charset=utf-8 headers, UTF8 BOM for Excel —
// and never let a "default" encoding decide. Modern .NET defaults to UTF-8 almost
// everywhere; the bugs live at legacy files, Excel, and misconfigured HTTP.
6 (Hard): Write TruncateForDisplay(string s, int maxGraphemes) that appends "…" — correct for emoji/accents — and show where the naive Substring version corrupts "👨‍👩‍👧 family végétarienne".
Solution
string TruncateForDisplay(string s, int maxGraphemes)
{
    var enumerator = StringInfo.GetTextElementEnumerator(s);
    var count = 0;
    var byteIndex = 0;                                  // char index of the cut point
    while (enumerator.MoveNext())
    {
        count++;
        if (count > maxGraphemes)
            return s[..byteIndex] + "…";                // cut at an ELEMENT boundary
        byteIndex = enumerator.ElementIndex + ((string)enumerator.Current).Length;
    }
    return s;                                           // short enough — unchanged
}

var text = "👨‍👩‍👧 family végétarienne";
Console.WriteLine(TruncateForDisplay(text, 3));         // "👨‍👩‍👧 f…" — family emoji intact
                                                        //   as ONE grapheme

// The naive version:
// text.Substring(0, 3) + "…"
// The family emoji is ~8 UTF-16 chars (3 codepoints + 2 zero-width joiners, some
// as surrogate pairs). Substring(0,3) cuts INSIDE it → an invalid/garbage sequence
// renders as broken boxes, and 'végét…' style cuts can orphan combining accents.
// Grapheme enumeration counts what USERS call characters — the only safe unit for
// display truncation. (For wire limits, count BYTES in the target encoding instead.)

🎯 Key Takeaways

  • Strings are immutable: transforms return new strings (capture them!), sharing is safe, literals intern — and loop concatenation is O(n²): StringBuilder for loops, Join for collections, interpolation for small fixed assemblies.
  • Comparison has a mode, always state it: ordinal(-ignore-case) for logic/keys/protocols, culture for human-facing text — Turkish-i is what defaulting costs; bake StringComparers into collections.
  • Format specifiers (:C :N2 :P1 :D5 :O, alignment) are one vocabulary across interpolation and ToString; machine output is invariant-culture + ISO-8601, display output is user-cultured.
  • Raw string literals (""") end escape soup for JSON/regex/SQL; index/range syntax slices strings.
  • Length counts UTF-16 units, not characters — emoji/accents break char-index logic; user text needs grapheme awareness (StringInfo/Rune); strings↔bytes always names an encoding (UTF-8) or invites mojibake.
  • Never assemble SQL/HTML/URLs by concatenation; hot parse paths graduate to spans (memory.html).
