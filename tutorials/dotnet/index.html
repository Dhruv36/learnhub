C# & .NET: How It Runs Foundations

What you'll master: what ".NET" actually is (a runtime + libraries + SDK, and the history that explains the confusing names), the compilation pipeline — C# → IL → JIT → machine code — and what each stage buys, your first program and the anatomy of a project, the CLI workflow, and where C# sits among languages you may know. Prerequisites: general programming basics; if you know Java, this lesson deliberately maps the ecosystems — they rhyme, and the differences are the interesting part.

Part 1 — What .NET Is (and the Name Soup, Decoded)

.NET =  CLR      the runtime: executes your code, manages memory (GC), enforces safety
      + BCL      Base Class Library: collections, IO, HTTP, JSON — the standard toolkit
      + SDK      dotnet CLI, compilers (Roslyn), build system (MSBuild), project templates

The names, sorted once and for all:
  .NET Framework (4.x)   the LEGACY Windows-only original (2002–). Maintenance mode.
  .NET Core (1–3.1)      the 2016 cross-platform rewrite.
  .NET 5, 6, 7, 8, 9…    Core, renamed — ONE .NET, annual releases, even = LTS.
  C#                     the language (currently v13); F#/VB share the runtime.
  ASP.NET Core           the web framework ON TOP of .NET (its own track here).

Modern default: current LTS release, cross-platform (Linux containers in prod is the norm),
open source (github.com/dotnet), high-performance (TechEmpower top tier).

Interviewers still ask ".NET Framework vs .NET Core vs .NET?" because a decade of naming chaos left real confusion — the answer above is worth having verbatim. The platform triangle matters more: the CLR (Common Language Runtime) is the managed environment your code runs in — garbage collection, type safety, exceptions, JIT compilation; the BCL is the standard library you'll spend most of this track learning; the SDK is the toolchain. If you know Java: CLR ≈ JVM, BCL ≈ JDK class library, NuGet ≈ Maven Central, dotnet CLI ≈ maven/gradle — same architecture, different trade-offs (value types, reified generics, spans — coming lessons — are where C# diverges meaningfully).

Part 2 — The Compilation Pipeline: C# → IL → Machine Code

hello.cs ──(Roslyn compiler, at BUILD)──▶ hello.dll containing IL + metadata
                                            │
                                            ▼   (at RUN, per method, on first call)
                                     JIT compiler ──▶ optimized machine code for THIS CPU
                                            │
                                     tiered compilation: hot methods get RE-jitted
                                     with heavier optimization (tier 1) after proving hot

// See it yourself: sharplab.io shows C# → IL → asm live. IL for a simple add:
//   ldarg.0        load argument 0
//   ldarg.1        load argument 1
//   add            add them
//   ret            return

WHY the two-stage design:
  portability   one .dll runs on Windows/Linux/macOS, x64/ARM — JIT targets the host
  optimization  JIT knows the ACTUAL CPU (vector widths!) and runtime behavior —
                tiered compilation optimizes what profiling proves hot
  safety        IL is verified: type-safe, memory-safe before it ever runs
  interop       any .NET language compiles to IL — C# can call F# seamlessly

// The AOT alternative (performance-aot.html): NativeAOT compiles straight to native
// at build — instant startup, no JIT, smaller memory; trades away runtime codegen.

C# compiles to IL (Intermediate Language) — CPU-neutral bytecode plus rich metadata — and the CLR's JIT (just-in-time) compiler turns each method into native code on first call. Tiered compilation is the modern refinement: methods start with a quick cheap compile (fast startup), and the runtime re-compiles the hot ones with full optimization once call counts prove they matter — profile-guided optimization happening live. This is the same play as the JVM's; the C# twist is that the spectrum extends further both ways: ReadyToRun pre-compiles for faster cold starts, and NativeAOT eliminates the JIT entirely for CLI tools and serverless. Knowing the pipeline explains real behaviors: first-request slowness in web apps (JIT warmup), why benchmarks need warmup iterations, and what "managed code" means — code the CLR supervises.

Part 3 — Hello, Project: Anatomy of a Modern C# App

dotnet new console -n PriceCheck && cd PriceCheck

// Program.cs — top-level statements (no ceremonial Main class since C# 9):
Console.WriteLine("What's the price?");
var input = Console.ReadLine();

if (decimal.TryParse(input, out var price))         // safe parsing — no exception on junk
{
    var withTax = price * 1.19m;                    // m = decimal literal (money type!)
    Console.WriteLine($"With tax: {withTax:C}");    // string interpolation + currency format
}
else
{
    Console.WriteLine($"'{input}' is not a number.");
}

// PriceCheck.csproj — the ENTIRE project file (SDK-style, no boilerplate):
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net9.0</TargetFramework>
    <Nullable>enable</Nullable>            <!-- nullable reference types ON (a later lesson) -->
    <ImplicitUsings>enable</ImplicitUsings> <!-- common usings auto-included -->
  </PropertyGroup>
</Project>

Modern C# threw out its own ceremony: top-level statements replace the class Program { static void Main } ritual (which you'll still read in older code — the compiler generates exactly that underneath), and the SDK-style .csproj is a dozen readable lines where the legacy format was hundreds. The snippet already shows house style you'll build on: var for obvious types, TryParse over exception-throwing parses at trust boundaries, decimal for money (next lesson explains why double would be a bug), and interpolated strings with format specifiers. Two project-file lines deserve respect now: Nullable>enable and implicit usings are the modern defaults every lesson here assumes.

Part 4 — The CLI Workflow

dotnet new console -n App        # scaffold (also: classlib, xunit, web, worker…)
dotnet run                       # build + run (dev loop)
dotnet watch run                 # hot-reload dev loop
dotnet build                     # compile → bin/Debug/net9.0/
dotnet test                      # run the test project(s)
dotnet add package Serilog       # NuGet: add a dependency (→ .csproj PackageReference)
dotnet publish -c Release        # deployable output; -r linux-x64 --self-contained
                                 #   bundles the runtime; /t:PublishContainer → OCI image

// Solutions group projects (the multi-project norm):
dotnet new sln -n Shop
dotnet sln add src/Shop.Api src/Shop.Domain tests/Shop.Tests
dotnet new classlib -n Shop.Domain
dotnet add src/Shop.Api reference src/Shop.Domain     # project-to-project reference

Layout convention:      src/Shop.Domain      pure business logic (no I/O deps)
                        src/Shop.Api         the executable host
                        tests/Shop.Tests     xunit project referencing both

The dotnet CLI is the whole toolchain behind one verb set — scaffold, run, test, package, publish — and it's what CI servers run, so knowing it beats IDE-button knowledge (Visual Studio, Rider, and VS Code all drive the same engine). NuGet is the package ecosystem (dependencies are PackageReference entries — versioned, transitive, lock-file capable). The solution/project layout shown is the idiom you'll meet everywhere: domain logic in dependency-free class libraries, executables as thin hosts, tests as first-class projects — the same layering argument as every track here, expressed in project references (and enforceable by them: Domain simply has no reference to Api, so the dependency arrow can't point the wrong way).

Part 5 — Reading C#: a Guided Tour of a Real Snippet

// A taste of idiomatic modern C# — every feature here gets a full lesson later:
public record Order(int Id, string Customer, decimal Total);        // records-structs.html

public class OrderReport
{
    private readonly List<Order> _orders = [];                      // collection expression

    public void Add(Order order) => _orders.Add(order);             // expression-bodied

    public IEnumerable<string> TopCustomers(decimal minTotal) =>
        _orders.Where(o => o.Total >= minTotal)                     // LINQ (linq.html)
               .OrderByDescending(o => o.Total)
               .Select(o => $"{o.Customer}: {o.Total:C}")
               .Take(5);

    public string Describe(Order? order) => order switch            // pattern-matching.html
    {
        null                    => "no order",
        { Total: > 1000m }      => $"big fish: {order.Customer}",
        { Total: 0m }           => "freebie",
        _                       => $"regular ({order.Total:C})",
    };
}

Naming conventions (uniform across the ecosystem — internalize now):
  PascalCase   types, methods, properties, public members     OrderReport, TopCustomers
  camelCase    locals, parameters                             minTotal, order
  _camelCase   private fields                                 _orders
  I-prefix     interfaces                                     IEnumerable, IDisposable

Skim-reading real C# early beats meeting each feature cold later: notice how much is expressions (expression-bodied members, switch expressions, LINQ chains) — modern C# has absorbed a functional accent onto its OO skeleton, much like modern Java, but a few years ahead on ergonomics. The naming conventions are near-universal in the ecosystem (enforced by analyzers), which makes unfamiliar codebases readable — a deliberate culture the track will keep reinforcing.

Part 6 — Deep Dive: Managed Execution — What the CLR Does for (and to) You

What "managed" buys, concretely:
  MEMORY SAFETY    no dangling pointers/double-frees — GC owns object lifetimes (memory.html)
  TYPE SAFETY      IL verification + runtime checks: casts throw, arrays bounds-check
  EXCEPTIONS       structured unwinding with stack traces, finally guarantees
  REFLECTION       types carry metadata — inspect/instantiate at runtime (DI containers,
                   serializers, test frameworks are all built on this)
  UNIFIED TYPES    everything derives from System.Object; value types too (via boxing)

What it costs, and the escape hatches:
  GC pauses        (mostly tiny; tunable; memory.html) — spans/structs reduce allocation
  JIT warmup       (tiered compilation mitigates; ReadyToRun/NativeAOT eliminate)
  bounds checks    (JIT ELIDES them when it can prove safety — write provable loops)
  unsafe/interop   unsafe blocks, pointers, P/Invoke to C libraries exist when needed —
                   fenced behind explicit keywords, not the default

The mental model: C# is a SPECTRUM language — from Python-easy top-level scripts to
C-adjacent unsafe/SIMD code, in one language, one runtime. Teams start safe and
selectively buy performance where profiling demands (performance-aot.html).

"Managed" is a contract: the CLR supervises execution, and both safety guarantees and their costs follow. What separates C# from most managed peers is the escape-hatch spectrum — value types you control, stack allocation, spans over raw memory, SIMD intrinsics, and unsafe when justified — letting one codebase host business logic and near-native hot paths without a language boundary. That spectrum is this track's arc: the foundations lessons live at the safe end; Memory and Performance walk to the other.

Senior framing: "why .NET?" in an architecture interview has a modern answer that outdated impressions miss: genuinely cross-platform (Linux-first deployment, first-class containers), top-tier web performance (TechEmpower), one ecosystem covering web/ desktop/mobile/games (Unity)/cloud with one language, LTS releases on a predictable annual cadence with famously smooth upgrades, and a hiring pool concentrated in enterprise domains (finance, healthcare, logistics). The honest costs: Microsoft-stewarded governance (open source, but one vendor steers), a smaller OSS package ecosystem than npm's long tail, and Windows-era reputation debt that still colors perception. Knowing both lists — and that the Framework→Core split is why old criticism often no longer applies — reads as current, which for .NET specifically is a differentiator.

Interview Questions

Q1 (Beginner): What are the CLR, IL, and JIT, and how do they relate?

The C# compiler (Roslyn) compiles source to IL — CPU-neutral intermediate bytecode plus metadata — packaged in assemblies (.dll). At runtime the CLR (the managed runtime: GC, type safety, exceptions) executes assemblies by JIT-compiling each method to native machine code for the actual CPU on first call, with tiered compilation re-optimizing hot methods. The split buys portability (one binary, any supported OS/CPU), verified safety before execution, and optimizations informed by live runtime behavior.

Q2 (Beginner): .NET Framework vs .NET Core vs .NET 8/9 — untangle the names.

.NET Framework (4.x) is the original 2002 Windows-only platform — still serviced, feature-frozen, legacy. .NET Core (2016) was the cross-platform, open-source rewrite. From version 5 the "Core" was dropped: .NET 5/6/7/8/9 are one unified platform — annual releases, even numbers LTS. New work targets current .NET; "porting to .NET" means migrating off Framework. C# is the language on top; ASP.NET Core is the web framework on top of that.

Q3 (Mid): What is tiered compilation, and what practical behaviors does it explain?

The JIT initially compiles methods quickly with light optimization (tier 0) so startup is fast, counts invocations, and recompiles hot methods with full optimization (tier 1) — including dynamic PGO, where the runtime uses observed types/branches to devirtualize and inline. It explains: first-request latency in freshly deployed web apps (warmup), why microbenchmarks need warmup phases before measurement (or they measure tier-0 code or the JIT itself), and why long-running processes can outperform their own startup profile. The bookends: ReadyToRun pre-JITs for faster cold start; NativeAOT removes the JIT entirely, trading runtime adaptability for instant start and small footprint.

Q4 (Mid): Walk through what a modern .csproj controls and why SDK-style projects matter.

SDK-style projects (<Project Sdk="Microsoft.NET.Sdk">) declare intent, not mechanics: target framework(s), output type, feature switches (Nullable, ImplicitUsings, TreatWarningsAsErrors), package and project references — while globbing source files implicitly (no more listing every .cs, the historical merge-conflict factory). They matter because they're diff-able and hand-editable (code review of dependency changes works), multi-targeting is one line (TargetFrameworks), and the same file drives IDEs, CLI, and CI identically. Directory.Build.props adds tree-wide shared settings — how teams enforce nullable/analyzers/versions across 50 projects at once.

Q5 (Senior): You're porting a .NET Framework 4.8 service to modern .NET. What's the migration shape and where are the traps?

Shape: inventory dependencies first (the upgrade-assistant tool + ApiPort analysis) — NuGet packages needing modern equivalents, Framework-only tech (WCF server → CoreWCF or gRPC/REST; AppDomains, remoting, some WebForms — no path but rewrite); convert projects to SDK-style; retarget class libraries to netstandard2.0/modern TFMs bottom-up through the dependency graph; the host/entry project converts last. Traps: behavioral diffs (globalization/ICU, path handling cross-platform, culture defaults), System.Drawing and Windows-only APIs on Linux targets, config system change (web.config → appsettings/IConfiguration), and third-party packages abandoned pre-Core — wrap them behind interfaces (your DI seam) so replacements are swappable. Run old and new side by side with traffic mirroring where the risk warrants. The strategic point: it's a port, not a rewrite — most idiomatic C# compiles unchanged; the work concentrates at the platform edges.

Q6 (Senior): Make the honest technical case for and against choosing .NET for a new backend in a polyglot company.

For: top-tier throughput/latency among managed platforms (value types, spans, and a decade of performance investment show up in real TechEmpower/production numbers); batteries-included coherence (one blessed DI/config/logging/testing story — less framework-choice churn than Node); C#'s range from rapid CRUD to near-native hot paths without switching languages; LTS cadence with codemod-smooth upgrades; strong typing culture that scales teams; first-class containers/K8s and serverless support (plus NativeAOT for cold-start- sensitive functions). Against: single-vendor stewardship (open source, but Microsoft steers); ecosystem depth beyond the mainstream is thinner than npm/PyPI's long tail; hiring skews enterprise (great in fintech/health, thinner in startup hubs); and JVM shops get most of the same virtues with their existing operational muscle. The senior answer weighs the org: existing ops expertise, hiring market, and domain — not benchmark bragging rights.

1 (Easy): Scaffold a console app, and make it greet the name passed as a command-line argument (fallback "world"), shouting if --loud is present. Use only what this lesson showed.
Solution
// dotnet new console -n Greeter && cd Greeter && dotnet run -- Ada --loud
var name = args.FirstOrDefault(a => !a.StartsWith("--")) ?? "world";
var greeting = $"Hello, {name}!";
if (args.Contains("--loud"))
    greeting = greeting.ToUpperInvariant();
Console.WriteLine(greeting);
// Top-level statements give you `args` implicitly. Note `--` in dotnet run separates
// dotnet's args from YOUR program's args.
2 (Easy): Explain what happens, stage by stage, from dotnet run to your greeting appearing — naming every artifact and component.
Solution
1. MSBuild reads Greeter.csproj → invokes Roslyn on Program.cs (+ implicit usings).
2. Roslyn compiles C# → IL + metadata → bin/Debug/net9.0/Greeter.dll (an assembly).
3. `dotnet` launches the CLR (host loads the runtime), which loads Greeter.dll,
   verifies the IL, and finds the entry point (compiler-generated Main from your
   top-level statements).
4. The JIT compiles each method to native code for THIS cpu on first call
   (tier 0 fast compile; hot paths would be re-jitted tier 1 in a long-running app).
5. Machine code executes; Console.WriteLine calls into the BCL, which P/Invokes the
   OS's stdout. The GC never wakes up — this program barely allocates.
3 (Medium): Build a tiny number-guessing game: random target 1–100, loop reading guesses with TryParse (rejecting junk without crashing), higher/lower hints, count attempts.
Solution
var target = Random.Shared.Next(1, 101);      // Random.Shared: the modern static instance
var attempts = 0;

while (true)
{
    Console.Write($"Guess #{attempts + 1}: ");
    if (!int.TryParse(Console.ReadLine(), out var guess))
    {
        Console.WriteLine("That's not a number — try again.");
        continue;                              // junk input costs no attempt
    }
    attempts++;
    if (guess < target) Console.WriteLine("Higher!");
    else if (guess > target) Console.WriteLine("Lower!");
    else { Console.WriteLine($"Got it in {attempts}!"); break; }
}

5 (Hard): Predict the output, then explain the two C#-specific behaviors it demonstrates: 
Console.WriteLine(0.1 + 0.2 == 0.3);
Console.WriteLine(0.1m + 0.2m == 0.3m);
object boxed = 42;
Console.WriteLine(boxed.GetType());
Console.WriteLine((int)boxed + 1);
Solution
False        // double is binary floating point — 0.1 has no exact representation
True         // decimal is base-10: exact for money-style values (the m suffix matters)
System.Int32 // boxing: the int VALUE was copied into a heap object; metadata knows its type
43           // unboxing: (int) copies the value back out — then normal arithmetic

// Behavior 1: C# has a real base-10 decimal type — the money answer (types.html).
// Behavior 2: unified type system — even value types derive from object, via
// boxing/unboxing (a copy each way, with costs covered in types.html/memory.html).
// Java contrast: int has no methods and needs Integer wrappers; C# ints ARE objects
// on demand.
6 (Hard): Your team's service shows 800ms latency on the first request after each deploy, then 15ms. Using this lesson's pipeline knowledge: explain the cause chain and list three mitigations with their trade-offs.
Solution
// Cause chain: deploy → new process → CLR loads assemblies → FIRST request walks a
// cold path: every method on the request pipeline (framework + your code) is
// tier-0 JIT-compiled on first call; plus one-time costs pile on (DI container build,
// config load, connection pools, TLS). Subsequent requests hit compiled code → 15ms.

// Mitigations:
// 1. WARMUP: hit key endpoints after start, before joining the load balancer
//    (readiness probe that exercises the hot paths). Cheap, no build changes;
//    covers only the routes you warm.
// 2. ReadyToRun publish (<PublishReadyToRun>): pre-compiles IL→native at build;
//    much faster first call, still JIT-upgradeable to full optimization later.
//    Trade-off: bigger binaries, per-RID publishing.
// 3. NativeAOT (where the app model allows): no JIT at all — startup in ms.
//    Trade-offs: no runtime codegen (reflection-heavy libs need source-gen
//    alternatives), longer builds, per-platform artifacts. (performance-aot.html)
// Bonus: tiered PGO means even post-warmup perf improves over minutes — don't
// benchmark a service in its first five minutes of life.

🎯 Key Takeaways

  • .NET = CLR (managed runtime) + BCL (standard library) + SDK (dotnet CLI/Roslyn/MSBuild); Framework 4.x is legacy — modern .NET is one cross-platform, annually-released platform (even = LTS).
  • Pipeline: C# → IL (portable, verified) → JIT per method with tiered compilation (fast start, hot paths re-optimized) — explaining warmup behavior; ReadyToRun/NativeAOT trade along the same axis.
  • Modern projects are lean: top-level statements, SDK-style csproj, Nullable + implicit usings on — and the CLI (new/run/test/add package/publish) is the canonical toolchain.
  • Solutions layer by project references: dependency-free Domain libraries, thin executable hosts, first-class test projects — architecture the compiler enforces.
  • C# is a spectrum language: safe managed defaults with escape hatches (structs, spans, unsafe) down to near-native — buy performance where profiling demands.
  • House style from day one: PascalCase/camelCase/_field conventions, var for obvious types, TryParse at boundaries, decimal for money.
