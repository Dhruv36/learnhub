C# Syntax, Types & Control Flow Basics

C# is a statically-typed, compiled language on the .NET runtime — powering everything from ASP.NET web services to Unity games. This lesson covers the type system fundamentals that differ meaningfully from dynamically-typed languages: value vs reference types, nullable reference types, and modern pattern matching that make C# feel far more expressive than its "curly-brace enterprise language" reputation.

Value Types vs Reference Types (the foundational distinction)

// VALUE types (struct, int, double, bool, DateTime...) — stored inline, COPIED on assignment:
int a = 5;
int b = a;        // b is an independent COPY
b = 10;           // a is still 5

// REFERENCE types (class, string*, array, List...) — a variable holds a POINTER to the object:
var list1 = new List { 1, 2, 3 };
var list2 = list1;    // list2 points to the SAME object
list2.Add(4);
// list1 now has 4 elements too — they're the same object in memory!

// *string is a reference type but behaves immutably — every "mutation" creates a new string.
📦 This distinction explains a huge share of C# bugs and interview questions. Passing a struct into a method passes a copy (mutations inside don't affect the caller); passing a class passes a reference (mutations DO affect the caller, since both point to the same object). Records (below) and immutable design patterns exist largely to make reference-type sharing safer and more predictable.

Nullable Reference Types (NRT) — Compile-Time Null Safety

#nullable enable                  // enabled by default in modern project templates

string name = null;               // ⚠️ COMPILER WARNING: non-nullable type, null not allowed
string? nickname = null;          // ✅ explicitly nullable — the ? documents intent

void Greet(string? name) {
    if (name is not null) {
        Console.WriteLine(name.Length);   // compiler now trusts name is non-null in this branch
    }
}
// NRT doesn't add runtime null checks — it's a COMPILE-TIME analysis that makes null-handling INTENT
// explicit and catches a huge class of NullReferenceExceptions before the code ever runs.

Pattern Matching — Modern C#'s Signature Feature

object shape = new Circle(5);

string Describe(object shape) => shape switch {
    Circle { Radius: > 10 } => "big circle",
    Circle c => $"circle, r={c.Radius}",
    Rectangle { Width: var w, Height: var h } when w == h => "square",
    Rectangle => "rectangle",
    null => "nothing",
    _ => "unknown shape"
};
// Switch EXPRESSIONS (not just statements) with type patterns, property patterns, and `when` guards —
// far more expressive than a chain of if/else or a classic switch. This is idiomatic modern C#.

var vs Explicit Types

var count = 5;                 // inferred as int — fine, the type is obvious from the right side
var users = GetActiveUsers();  // inferred as List — LESS clear without knowing GetActiveUsers()
List users2 = GetActiveUsers();  // explicit — often preferred when the type isn't obvious
// var doesn't make C# dynamically typed — the compiler still infers and enforces a concrete type at
// compile time. It's purely a readability/verbosity trade-off, not a type-safety one.

Interview Questions

Q1: What's the difference between value types and reference types, and why does it matter?

Value types (structs, including all built-in numeric types, bool, and DateTime) store their data directly, and assigning one variable to another COPIES the data — the two variables are then fully independent. Reference types (classes, arrays, List<T>, delegates) store a reference (pointer) to an object on the managed heap; assigning one variable to another copies the REFERENCE, so both variables end up pointing to the same underlying object, and mutating it through either variable is visible through the other. This matters enormously in practice: passing a struct into a method and mutating it inside has no effect on the caller's copy, while passing a class and mutating its fields DOES affect the caller's object, since they share the same instance — a very common source of subtle bugs when developers assume "assignment always copies." It also underlies why immutable designs (readonly fields, records with with-expressions producing new instances) are popular for reference types: they make sharing predictable by avoiding mutation entirely rather than relying on programmers to remember which type category a given variable belongs to.

Q2: What do nullable reference types actually do — do they add runtime null checks?

No — nullable reference types (NRT, enabled with #nullable enable, on by default in modern project templates) are purely a COMPILE-TIME static analysis feature, not a runtime mechanism. When enabled, a plain reference type like string is treated by the compiler as non-nullable by default, so assigning null to it produces a compiler warning; to explicitly allow null you annotate the type with a ? (string?). The compiler then performs flow analysis to track whether a nullable variable has been null-checked in a given code path (e.g., inside an `if (name is not null)` block, subsequent uses of name are treated as non-null), and warns you if you dereference a possibly-null value without such a check. Critically, this is advisory at compile time — it does NOT insert runtime null checks or change program behavior, and it's still fully possible to get a NullReferenceException at runtime if warnings are ignored or suppressed (e.g., with the null-forgiving `!` operator). Its value is making null-handling intent explicit in the type system and catching a large class of null-related bugs during development rather than in production.

Q3: How do switch expressions with pattern matching improve on traditional switch statements?

A traditional switch STATEMENT matches a value against constant cases and executes statements per case, which becomes verbose and error-prone for anything beyond simple constant comparisons — testing an object's runtime type, inspecting its properties, or combining multiple conditions typically degenerates into chained if/else-if with type casts. A switch EXPRESSION (the `x switch { ... }` syntax) is instead an expression that evaluates to a value, and its patterns are far richer: type patterns match and simultaneously cast based on runtime type (`Circle c => ...`), property patterns destructure and match on an object's properties inline (`Circle { Radius: > 10 } => ...`), relational patterns match numeric ranges directly in the case, and `when` clauses add arbitrary boolean guards for conditions patterns alone can't express (like comparing two properties to each other). Because it's an expression, the compiler can also enforce exhaustiveness and the result naturally assigns to a variable or returns from a method without a separate mutable variable being set inside each case. The net effect is that logic which previously took many lines of type-checking and casting becomes a compact, declarative, and often more correct piece of code — which is why pattern matching is considered one of modern C#'s signature features.

🏋️ Practice: write a method that mutates a struct parameter and one that mutates a class parameter, and observe the caller-visible difference; enable nullable reference types on a small project and fix the resulting warnings; rewrite an if/else-if chain that checks an object's type and properties as a switch expression with pattern matching; add a `when` guard to a switch arm; experiment with the null-forgiving `!` operator and explain when it's appropriate.

🎯 Key Takeaways

  • Value types (struct/int/bool/DateTime) copy on assignment; reference types (class/array/List<T>) share the same object — this explains most "unexpected mutation" bugs.
  • Nullable reference types are compile-time-only static analysis (no runtime checks) that make null-handling intent explicit and catch bugs early.
  • var is type inference, not dynamic typing — the compiler still enforces a concrete type; use explicit types when the right-hand side isn't self-evident.
  • Switch expressions with type/property/relational patterns and `when` guards are far more expressive than chained if/else or classic switch statements.
  • Strings are reference types but behave immutably — every transformation returns a new string; discard-the-result is a classic bug.
