Methods, Parameters & Overloading Foundations

What you'll master: the parameter-passing model (the by-value default and what ref/out/in actually change), optional and named arguments, params, method overloading and how the compiler resolves it, expression-bodied and local functions, and the ref family for performance. Prerequisites: Types & Variables — the value/reference distinction is the whole foundation for how arguments travel. Method signatures are contracts; designing them well is a core professional skill this lesson builds directly.

Part 1 — The Passing Model: Everything Is By Value (Read This Carefully)

// C# passes arguments BY VALUE by default — it copies the VARIABLE'S CONTENTS.
// The subtlety: for reference types, the CONTENTS are the reference (types.html).

void Bump(int n)        { n = 99; }              // copies the int
void Clear(List<int> x) { x.Clear(); }           // copies the REFERENCE (both point to same list)
void Reassign(List<int> x) { x = new(); }        // reassigns the COPY — caller unaffected

int a = 5;              Bump(a);       Console.WriteLine(a);          // 5   — copy mutated
var list = new List<int>{1,2};  Clear(list);   Console.WriteLine(list.Count);  // 0 — same object!
var l2 = new List<int>{1,2};    Reassign(l2);  Console.WriteLine(l2.Count);     // 2 — copy reassigned

// The mental model that resolves ALL confusion:
//   pass-by-value copies what the variable HOLDS.
//   value type holds DATA          → method gets an independent copy
//   reference type holds a POINTER → method gets a copy of the pointer (same object),
//                                    but reassigning the pointer doesn't reach the caller

This is the single most misunderstood topic in C#, and it's misunderstood because people say "objects are passed by reference" — they are not. C# is pass-by-value, always. What confuses everyone is that a reference type's value is a reference, so a copy of that reference still points at the same heap object — mutating the object is visible to the caller, but reassigning the parameter is not. Hold the model from Types: "copy what the variable holds," and every case above becomes obvious rather than a memorized rule. This distinction is the classic mid-level interview filter precisely because the sloppy explanation is so widespread.

Part 2 — ref, out & in: Changing How Arguments Travel

// ref — pass the VARIABLE ITSELF (an alias); the method can reassign what the caller sees:
void Swap(ref int x, ref int y) { (x, y) = (y, x); }
int p = 1, q = 2;  Swap(ref p, ref q);            // p=2, q=1 — caller's variables changed
// ref requires: the argument is a variable (not a literal), assigned before the call,
//               and 'ref' at BOTH declaration and call site (visible at the call — good).

// out — like ref, but the method MUST assign it; the caller need NOT pre-assign.
// The canonical use is the Try-pattern (types.html, control-flow.html):
if (int.TryParse(input, out int value))           // out var declares inline
    Use(value);                                    //   value is definitely assigned here
bool TryGetUser(int id, out User? user) { … }      // returns success + result together

// in — pass by reference but READ-ONLY (a performance hint for big structs):
double Distance(in Point3D a, in Point3D b) { … }  // avoids copying a large struct,
                                                   //   compiler forbids mutating a/b

WHEN to reach for each (rarely for the first two):
  out  → returning a second value alongside success (Try-pattern). Common, idiomatic.
  ref  → genuine two-way variable exchange, or in-place struct mutation. Uncommon.
  in   → passing large readonly structs without a copy (memory.html). Perf-only.
// Default: DON'T use these. Return values (or tuples/records) beat out-heavy signatures
// for readability. They're precision tools, not everyday ones.

These three modifiers pass the variable itself rather than a copy of its contents — the method gets an alias to the caller's storage. out earns its keep in the Try-pattern (return success as the bool, the result via out — you've used it since Types), and in is a pure performance tool for avoiding large-struct copies. ref is powerful but usually a smell in ordinary code: a method that needs to mutate its caller's variables is often better expressed by returning a new value (or a tuple/record) — the immutability discipline applies here too. The one genuinely nice thing: ref/out are required at the call site, so aliasing is never hidden from the reader.

Part 3 — Optional, Named & params Arguments

// OPTIONAL parameters — default values, must come after all required ones:
decimal Total(decimal price, decimal taxRate = 0.19m, decimal discount = 0m)
    => price * (1 + taxRate) - discount;
Total(100m);                       // uses both defaults
Total(100m, 0.10m);                // overrides taxRate only

// NAMED arguments — clarity + skipping over optionals:
Total(100m, discount: 5m);         // skip taxRate (keep its default), set discount
CreateUser(name: "Ada", isAdmin: true, sendEmail: false);   // self-documenting call

// The BOOLEAN-TRAP fix — named args kill mystery calls:
Reserve(seat, false, true, false);          // ✗ what do these mean?!
Reserve(seat, refundable: false, aisle: true, mealIncluded: false);   // ✓ readable

// params — variable argument count (a syntactic sugar over an array/span):
int Sum(params int[] nums) => nums.Sum();
Sum(1, 2, 3);   Sum();   Sum(existingArray);        // all valid
// C# 13: params can be Span<T>/IEnumerable<T> too — params ReadOnlySpan<int> avoids
// the array allocation (used by string.Format, Console.WriteLine internally).

// GOTCHA: optional-parameter defaults are baked into the CALLER at COMPILE time.
// Change a library's default and RECOMPILE consumers, or they keep the old value.

Optional parameters plus named arguments together solve the "telescoping overloads" problem and the boolean-trap readability problem in one stroke: one method with sensible defaults, and calls that name their non-obvious arguments so Reserve(seat, refundable: false, ...) reads like documentation. params gives ergonomic variadic calls (and modern C# lets it be a ReadOnlySpan to skip the array allocation on hot paths). The one trap worth carrying: default values are compiled into the call site, so changing a library's default is a binary-breaking-ish change — consumers keep the old default until they recompile. For public library APIs, that's a real versioning consideration; overloads don't have it.

Part 4 — Overloading & How the Compiler Chooses

// Same NAME, different parameter lists — the compiler picks by argument types:
void Log(string message)          => Write($"INFO: {message}");
void Log(Exception ex)            => Write($"ERROR: {ex.Message}");
void Log(string message, int code) => Write($"[{code}] {message}");
// Overloads differ by parameter TYPES/COUNT/ORDER — NOT by return type alone
// (int F() and string F() do NOT overload — ambiguous to the compiler).

// Resolution (simplified): among applicable overloads, the compiler prefers the
// MOST SPECIFIC match, avoiding implicit conversions where an exact type exists:
Log("hi");            // → Log(string)
Log(42);              // → Log(string, int)? NO — one arg. → is there Log(int)? no →
                      //   int has no implicit conv to string/Exception → COMPILE ERROR
Log(new IOException()); // → Log(Exception)   (IOException IS-A Exception)

// The classic overloading TRAP — null is ambiguous:
Log(null);            // string? Exception? → ambiguous → compile error
Log((string)null!);   // cast to disambiguate

// Overload vs optional parameters — pick ONE style per API:
//   overloads:  distinct behaviors, different types, binary-version-safe defaults
//   optionals:  same behavior with tweakable knobs, fewer methods to maintain
// Mixing both on one method family creates resolution ambiguities — choose deliberately.

Overloading lets one operation name serve related type signatures, and the compiler resolves calls by finding the most specific applicable overload — preferring exact types over implicit conversions, and refusing to guess when two are equally good (the null ambiguity is the textbook case). Two design points matter more than the resolution algorithm: overloads can't differ by return type alone (the compiler dispatches on arguments, not on what you assign the result to), and you generally pick overloading or optional parameters for a given method family, not both — combining them breeds ambiguity, and library authors lean toward overloads for the binary-versioning reason from Part 3.

Part 5 — Expression-Bodied Members & Local Functions

// EXPRESSION-BODIED — for one-expression methods, properties, ctors:
public string FullName => $"{First} {Last}";                // property
public int Area() => Width * Height;                         // method
public decimal Discounted(decimal p) => p * (1 - Rate);      // one clean line
// Use when the body IS a single expression; fall back to { } blocks otherwise.

// LOCAL FUNCTIONS — methods declared INSIDE methods; scoped, named, closure-capable:
IEnumerable<int> Evens(int max)
{
    ValidateMax();                     // called before its definition — hoisted
    for (int i = 0; i <= max; i += 2)
        yield return i;

    void ValidateMax()                 // local: sees 'max', invisible outside Evens
    {
        if (max < 0) throw new ArgumentOutOfRangeException(nameof(max));
    }
}

// Why local functions beat private helpers or lambdas here:
//  - SCOPED: only Evens can call ValidateMax — no class-surface pollution
//  - captures locals directly (like a lambda) but with a NAME and no delegate allocation
//  - crucial for iterators/async: eager ARG VALIDATION runs immediately, while the
//    iterator body is deferred — see the deep dive

Expression-bodied members are pure ergonomics — when a member is one expression, => says so with less ceremony than a braces-and-return block. Local functions are the more substantive feature: they're named helpers scoped to a single method, capturing its locals like a lambda but without allocating a delegate, and (unlike lambdas) they can be recursive and are hoisted so call-before-definition reads naturally. Their signature use — argument validation for iterators and async methods — is the deep-dive payoff below, and it's a genuine correctness tool, not just style.

Part 6 — Deep Dive: The ref Family, Deferred Execution & Signature Design

// (A) LOCAL FUNCTIONS FIX A REAL DEFERRED-EXECUTION BUG:
IEnumerable<int> BadEvens(int max)
{
    if (max < 0) throw new ArgumentException();   // ✗ does NOT throw until enumeration!
    for (int i = 0; i <= max; i += 2) yield return i;
}
BadEvens(-1);                                      // no exception here — body is deferred
var q = BadEvens(-1);                              // still nothing
foreach (var x in q) { }                           // NOW it throws — far from the bad call

IEnumerable<int> GoodEvens(int max)                // ✓ split: eager check, deferred body
{
    if (max < 0) throw new ArgumentException(nameof(max));   // runs at CALL time
    return Iterator();
    IEnumerable<int> Iterator() { for (int i=0;i<=max;i+=2) yield return i; }
}

// (B) THE ref FAMILY FOR PERFORMANCE (memory.html goes deeper):
in Matrix4x4 m         // pass big struct by readonly-ref — no 64-byte copy per call
ref int slot = ref array[i];    // ref LOCAL: an alias into the array element
ref readonly Point Origin => ref _origin;    // return a reference, don't copy
static ref T Choose<T>(ref T a, ref T b, bool first) => ref (first ? ref a : ref b);
// Used in high-perf code (game engines, parsers) to eliminate struct copies entirely.

// (C) SIGNATURE DESIGN — the professional lens:
//  - few parameters (3-4 max); more → introduce a parameter OBJECT/record
//  - order: required first, then optional; most-important first
//  - avoid boolean parameters (use enums: SortDirection.Descending > bool desc)
//  - one job per method; if the name needs "And", split it
//  - return richness: tuples for 2-3 values, records for more, out only for Try-patterns
//  - nullable annotations honest (User? means "may be absent" — exceptions-nullability.html)

Three threads converge here. The deferred-execution bug (A) is subtle and real: because iterator (yield) and async method bodies don't run until enumerated/awaited, validation written at the top runs late — a local-function split moves the check to call time where the bad argument actually is. The ref family (B) is the performance dialect — aliases into memory that eliminate struct copies, foundational for the Memory lesson and the reason C# competes with C++ on hot paths. And signature design (C) is the everyday craft: methods are the API surface teammates read most, so few parameters, no boolean traps, honest nullability, and one-job-per-method are the habits that make a codebase legible.

Senior framing: method design is where "reads intent" is won or lost, and reviewers scan for a consistent set of tells. Parameter lists that grow past ~4 are a missing parameter object (a record OrderRequest(...) beats Create(a, b, c, d, e, f) — named construction, evolvable, testable). Boolean parameters are almost always a design smell — at the call site Send(msg, true, false) is unreadable, and an enum both documents and extends. out-heavy signatures usually want a return type (tuple/record) instead — out belongs to the Try-pattern and little else. And the interview-grade point: know that "pass by reference" is the wrong description of C# (it's pass-by-value with reference-typed values), know that iterator/async validation must be split to run eagerly, and know that the ref family exists for copy elimination — three things that separate someone who uses C# from someone who understands it.

Interview Questions

Q1 (Beginner): Is C# pass-by-value or pass-by-reference? Explain with a reference type.

C# is pass-by-value — it copies what the variable holds. For a reference type, what the variable holds is a reference, so the method gets a copy of that reference pointing at the same object: mutating the object (list.Clear()) is visible to the caller, but reassigning the parameter (list = new()) only changes the local copy of the reference and the caller sees nothing. Saying "objects are passed by reference" is the common wrong description — the accurate statement is "reference-typed values are references, and those values are passed by value." Genuine pass-by-reference requires the ref keyword.

Q2 (Beginner): What's the difference between out and ref, and where is each idiomatic?

Both pass the variable itself (an alias), so the method can change what the caller sees. ref requires the argument to be assigned before the call and the method may read or write it (two-way). out does not require pre-assignment but requires the method to assign it before returning (one-way out). Idiomatic use: out for the Try-pattern — return bool success plus the result via out (int.TryParse, Dictionary.TryGetValue), often with out var inline; ref for genuine two-way exchange or in-place struct mutation, which is uncommon in ordinary code (return values are usually clearer). Both must appear at the call site, so aliasing is never hidden.

Q3 (Mid): How does the compiler resolve overloads, and why does Log(null) fail with Log(string) and Log(Exception)?

The compiler collects all applicable overloads (those the arguments can be converted to), then picks the most specific — preferring exact types and the least/narrowest implicit conversions, and rejecting the call as ambiguous if two are equally good. Overloads must differ in parameter types/count/order, not return type. Log(null) is ambiguous because null is convertible to both string and Exception (both reference types), and neither is more specific than the other — so the compiler can't choose and errors out. Disambiguate with a cast: Log((string)null!).

Q4 (Mid): When would you choose optional parameters vs overloads for the same API, and what's the versioning trap?

Optional parameters when it's one behavior with tweakable knobs (fewer methods to maintain, cleaner with named arguments); overloads when the variants have genuinely different behavior or types, or when binary versioning matters. The trap: default values are compiled into the call site — a library that changes taxRate = 0.19m to 0.20m does not affect already-compiled consumers; they keep 0.19m until recompiled. For public APIs that's a silent behavioral split, which is why framework authors favor overloads (adding an overload is binary-safe; changing a default is not). Mixing overloads and optionals on one method family also risks resolution ambiguities — pick one style per family.

Q5 (Senior): A teammate's IEnumerable<T> method validates arguments at the top but the exception only appears deep in a foreach elsewhere. Diagnose and fix.

The method uses yield return, making it an iterator: the compiler turns the whole body into a state machine that runs nothing until enumeration begins. So the top-of-method validation is deferred along with everything else — Method(badArg) returns a lazy sequence without throwing, and the ArgumentException fires only when someone enumerates it, potentially far away in the code and stack. Fix: split into an eager public method that validates immediately and returns a private/local iterator function holding the yield body — validation now runs at call time where the bad argument lives, while the sequence stays lazily evaluated. The identical issue and fix apply to async methods (the body defers until awaited). This is the canonical reason local functions exist.

Q6 (Senior): Critique this signature and redesign it: Order CreateOrder(int customerId, string productSku, int qty, bool rush, bool giftWrap, decimal? discount, string? notes).

Problems: seven parameters (positional puzzle at the call site, CreateOrder(42, "SKU-1", 3, true, false, null, null) is unreadable); two adjacent booleans (the boolean trap — easy to transpose, meaning invisible); primitive-obsessed ids/SKUs (an int customerId and any other int are swappable — types.html domain-wrapper argument); and mixed required/optional in a flat list. Redesign: introduce a parameter object — record OrderRequest(CustomerId Customer, Sku Product, int Quantity) for the required core, with optional properties or a fluent/init-only set for Rush, GiftWrap, Discount, Notes; replace the booleans with intent-revealing types (an enum or named init properties: ShippingSpeed.Rush); wrap the ids/SKUs as readonly record struct so swaps don't compile. Result: CreateOrder(new OrderRequest(customer, sku, 3) { Shipping = ShippingSpeed.Rush }) — self-documenting, evolvable without breaking callers, and misuse becomes a compile error. The meta-point: method signatures are the most-read API surface, and this refactor trades a little ceremony for a large, permanent readability and safety win.

🏋️ Exercises

1 (Easy): Predict the output and explain each with the copy rule: 
void A(int[] arr) { arr[0] = 99; }
void B(int[] arr) { arr = new[] { -1 }; }
var x = new[] { 1 }; A(x); Console.WriteLine(x[0]);
var y = new[] { 1 }; B(y); Console.WriteLine(y[0]);
Solution
99
1
// A: the array reference is copied, but arr[0]=99 mutates the SAME object → visible.
// B: arr = new[]{-1} reassigns the COPY of the reference; the caller's y still points
//    at the original {1}. Pass-by-value copies the reference — mutate yes, reassign no.
2 (Easy): Write a TryParseRange("3-7") returning bool with out int start, out int end, handling malformed input without throwing.
Solution
bool TryParseRange(string? s, out int start, out int end)
{
    start = end = 0;                                    // out params MUST be assigned on all paths
    if (string.IsNullOrWhiteSpace(s)) return false;
    var parts = s.Split('-');
    if (parts.Length != 2) return false;
    return int.TryParse(parts[0], out start)
        && int.TryParse(parts[1], out end)
        && start <= end;
}
// Usage: if (TryParseRange("3-7", out var a, out var b)) Use(a, b);
// The Try-pattern: bool success + out results, no exceptions for expected bad input.
3 (Medium): Design a Notify method family with sensible defaults so these all work: Notify("hi"), Notify("hi", urgent: true), Notify("hi", channel: Channel.Sms). Then explain why you'd use optionals here, not overloads.
Solution
enum Channel { Email, Sms, Push }

void Notify(string message, Channel channel = Channel.Email, bool urgent = false)
{
    var prefix = urgent ? "[URGENT] " : "";
    Console.WriteLine($"{channel}: {prefix}{message}");
}
// Notify("hi");                          → Email: hi
// Notify("hi", urgent: true);            → Email: [URGENT] hi   (skip channel, name urgent)
// Notify("hi", channel: Channel.Sms);    → Sms: hi

// Why optionals + named args, not overloads: it's ONE behavior with knobs. Overloads
// would need 2^2 = 4 methods (message; message+channel; message+urgent; message+both) —
// a telescoping explosion. Named arguments give the call-site clarity that would
// otherwise require the overloads. Enum (not bool) for channel = readable and extensible.
4 (Medium): Fix this iterator so bad arguments throw immediately, and prove the difference. 
IEnumerable<int> Batch(int[] src, int size) {
    if (size <= 0) throw new ArgumentException("size must be positive");
    for (int i = 0; i < src.Length; i += size)
        yield return src[i];
}
Solution
IEnumerable<int> Batch(int[] src, int size)
{
    ArgumentNullException.ThrowIfNull(src);
    if (size <= 0) throw new ArgumentException("size must be positive", nameof(size));
    return Iterator();

    IEnumerable<int> Iterator()                          // deferred body isolated here
    {
        for (int i = 0; i < src.Length; i += size)
            yield return src[i];
    }
}
// Proof:
//   var q = Batch(arr, -1);   // ORIGINAL: no throw (body deferred) — bug hides
//                             // FIXED:    throws HERE, at the call
//   foreach (var x in q) {}   // ORIGINAL: throws only NOW, far from the cause
// The local function splits eager validation (runs at call) from the lazy yield body.
5 (Hard): Write overloaded Describe methods for int, string, and object, then predict which is chosen for each call and explain the resolution: 
Describe(42); Describe("hi"); Describe(3.14); Describe(null); Describe((object)"hi");
Solution
void Describe(int n)    => Console.WriteLine($"int: {n}");
void Describe(string s) => Console.WriteLine($"string: {s}");
void Describe(object o) => Console.WriteLine($"object: {o}");

Describe(42);           // → int    (exact match beats object)
Describe("hi");         // → string (exact match beats object)
Describe(3.14);         // → object (double has no exact overload; no implicit double→int;
                        //           boxes to object — the only applicable one)
Describe(null);         // → string (null fits string OR object; string is MORE SPECIFIC
                        //           than object → chosen, no ambiguity here since one
                        //           reference type is more derived than object)
Describe((object)"hi"); // → object (the STATIC type is object — resolution is compile-time,
                        //           it does NOT see the runtime string)
// Key lessons: exact type beats conversion; most-derived wins; overload resolution uses
// the COMPILE-TIME (static) type, not the runtime type. The last call is the classic
// "why didn't it pick string?" — because the compiler only knew 'object'.
6 (Hard): Refactor a 6-parameter SendEmail(string to, string from, string subject, string body, bool isHtml, bool highPriority) into a clean design, and show the before/after call sites.
Solution
// BEFORE — positional puzzle, boolean trap:
// SendEmail("a@x.com", "b@y.com", "Hi", "<p>...</p>", true, false);
//   which bool is which? are 'to'/'from' in the right order? unclear at the call.

// AFTER — parameter object with required core + optional knobs:
enum BodyFormat { PlainText, Html }
enum Priority   { Normal, High }

record EmailMessage(string To, string From, string Subject, string Body)
{
    public BodyFormat Format { get; init; } = BodyFormat.PlainText;
    public Priority Priority { get; init; } = Priority.Normal;
}

void SendEmail(EmailMessage message) { /* ... */ }

// Call site — self-documenting, order-independent for the options, evolvable:
SendEmail(new EmailMessage(
    To: "a@x.com", From: "b@y.com", Subject: "Hi", Body: "<p>...</p>")
{
    Format = BodyFormat.Html,        // no more mystery 'true'
    Priority = Priority.Normal,      // enum states intent; default could omit it
});
// Wins: named required args (can't transpose to/from silently — though wrapping them as
// EmailAddress record structs would make it a COMPILE error, types.html); enums kill the
// boolean trap; adding a field (e.g. Cc) doesn't break existing call sites; the record is
// trivially testable and immutable. Ceremony up front, permanent readability after.

🎯 Key Takeaways

  • C# is pass-by-value always: it copies what the variable holds. Reference types hold a reference, so mutation is visible but reassignment isn't — "objects are passed by reference" is the wrong model.
  • out (Try-pattern), ref (genuine two-way — rare), in (readonly large-struct perf) pass the variable itself; all are visible at the call site. Default to return values instead.
  • Optional parameters + named arguments kill telescoping overloads and the boolean trap — but defaults bake into call sites (a versioning trap; overloads are binary-safe). params for variadics (span-based to avoid allocation).
  • Overloads resolve by most-specific argument match at compile time (static type), can't differ by return type alone, and error on ambiguity (null). Pick overloads or optionals per family.
  • Expression-bodied members for one-liners; local functions for scoped, closure-capturing, allocation-free helpers — and to split eager validation from deferred iterator/async bodies.
  • Signature design is API design: ≤4 parameters (else a parameter object/record), enums over booleans, honest nullability, one job per method — the most-read surface deserves the most care.
