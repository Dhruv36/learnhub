Memory: Span<T>, GC, IDisposable Advanced

.NET's garbage collector handles most memory management automatically, but understanding it — plus unmanaged resource cleanup via IDisposable, and allocation-free slicing via Span<T> — is what separates developers who can diagnose production memory/performance issues from those who can't.

Stack vs Heap, GC Generations

// Value types (structs, locals) generally live on the STACK — fast, automatically reclaimed when the
// method returns, no GC involvement. Reference types (class instances) live on the managed HEAP,
// tracked and eventually reclaimed by the Garbage Collector.
//
// The GC uses GENERATIONS based on the observation that most objects die young:
//   Gen 0 — short-lived objects (most allocations); collected frequently and CHEAPLY
//   Gen 1 — a buffer between short and long-lived
//   Gen 2 — long-lived objects (caches, static data); collected rarely but EXPENSIVELY
// An object surviving a Gen 0 collection gets PROMOTED to Gen 1, then eventually Gen 2 if it keeps
// surviving. Minimizing Gen 2 collections (by not needlessly promoting short-lived objects) is a
// core performance lever in high-throughput services.
🗑️ "The GC handles memory" doesn't mean allocations are free. Every allocation is still CPU work, and excessive allocation (especially of objects that get promoted to Gen 2) causes GC pauses that hurt latency-sensitive applications. High-performance .NET code minimizes unnecessary allocations — this is exactly the problem Span<T> and object pooling solve.

IDisposable — Deterministic Cleanup for Unmanaged Resources

// The GC only knows about MANAGED memory. File handles, DB connections, network sockets are
// UNMANAGED resources the GC doesn't understand — IDisposable is how you clean those up deterministically:
public class ReportGenerator : IDisposable {
    private readonly FileStream _file = new("report.txt", FileMode.Create);
    public void Dispose() => _file.Dispose();     // releases the OS file handle immediately
}

using var report = new ReportGenerator();    // `using` guarantees Dispose() runs, even if an exception occurs
// (older syntax: using (var report = new ReportGenerator()) { ... } — same guarantee, block-scoped)
// Without `using`, the file handle stays open until the GC eventually finalizes the object — which
// could be seconds, minutes, or never under memory pressure that never triggers a Gen 2 collection.

Span<T> — Slicing Without Allocating

// ❌ Substring ALLOCATES a brand-new string on the heap every single call:
string full = "Hello, World!";
string slice = full.Substring(7, 5);     // allocates "World" as a NEW string object

// ✅ Span gives a VIEW into the existing memory — zero allocation:
ReadOnlySpan span = full.AsSpan(7, 5);   // "World" — just a pointer + length, no new heap object
// Span works over arrays, strings, and stack-allocated memory uniformly. In a hot loop parsing
// large amounts of text (e.g., a CSV parser or a web server's request parsing), replacing Substring
// calls with Span slicing can eliminate MILLIONS of allocations and dramatically cut GC pressure.
⚡ Span<T> is a stack-only ref struct — it CANNOT be stored on the heap. This restriction (no Span<T> fields in a class, no Span<T> in async methods across an await) is what lets the runtime guarantee memory safety without GC tracking — the compiler enforces that a Span never outlives the memory it points into. It's a niche but high-leverage tool specifically for allocation-sensitive hot paths (parsers, serializers, high-throughput servers).

The Dispose Pattern for Classes Holding Unmanaged Resources

public class ResourceHolder : IDisposable {
    private bool _disposed;
    private IntPtr _unmanagedHandle;    // e.g., a raw OS handle

    protected virtual void Dispose(bool disposing) {
        if (_disposed) return;
        if (disposing) { /* dispose MANAGED resources (other IDisposables) here */ }
        ReleaseUnmanagedHandle(_unmanagedHandle);   // always release unmanaged resources
        _disposed = true;
    }
    public void Dispose() {
        Dispose(true);
        GC.SuppressFinalize(this);      // tells the GC "no need to run my finalizer, I already cleaned up"
    }
    ~ResourceHolder() => Dispose(false);   // finalizer — a SAFETY NET if Dispose() was never called
}
// This is the "full" IDisposable pattern for classes wrapping truly unmanaged resources directly.
// Most application code just implements the simple Dispose() and doesn't need a finalizer at all.

Interview Questions

Q1: How does .NET's generational garbage collector work, and why does it matter for performance?

The GC is built around the empirical observation that most objects are short-lived — a huge fraction of allocations die almost immediately (a temporary object created and discarded within a single method call), while a smaller fraction live for the whole application's lifetime (caches, singleton services, static data). To exploit this, the managed heap is divided into generations. Gen 0 holds newly allocated objects and is collected very frequently but very CHEAPLY, since it only has to scan a small region of recently-allocated memory; most objects die in Gen 0 and are reclaimed almost for free. Any object that survives a Gen 0 collection (because something still references it) is promoted to Gen 1, a buffer generation; objects surviving further get promoted to Gen 2, which holds long-lived objects and is collected far less often but much more EXPENSIVELY, since a Gen 2 (full) collection has to examine a much larger portion of the heap and can pause the application noticeably. This matters for performance because the generational hypothesis only pays off if your allocation patterns match it: code that needlessly keeps short-lived-looking objects alive long enough to get promoted into Gen 1/Gen 2 forces more expensive collections, and code with a very high allocation RATE (even of short-lived objects) still spends real CPU time doing those Gen 0 collections repeatedly. High throughput, latency-sensitive services specifically minimize allocation rate and avoid unnecessarily retaining references to reduce both collection frequency and the cost of the collections that do happen.

Q2: Why does .NET need both the garbage collector and IDisposable — doesn't the GC handle cleanup?

The garbage collector only manages MANAGED memory — objects allocated on the .NET heap that the runtime fully understands and can track references to. Many resources a program uses are UNMANAGED from the runtime's perspective: open file handles, network sockets, database connections, GDI/OS handles — the GC has no visibility into these and doesn't know when a program is "really done" with them in the way it does for a plain managed object. If cleanup of such resources were left entirely to the GC via finalizers, it would happen NONDETERMINISTICALLY — potentially not until the next Gen 2 collection runs, which could be seconds, minutes, or effectively never under low memory pressure — meanwhile the underlying OS resource (a file handle, a database connection from a limited connection pool) stays held, which can exhaust a limited resource pool well before the GC gets around to reclaiming the wrapping .NET object. IDisposable solves this by giving classes an explicit, DETERMINISTIC cleanup method (Dispose()) that application code calls as soon as it's actually done with the resource — most idiomatically via a `using` statement/declaration, which guarantees Dispose() runs as soon as the scope ends, even if an exception is thrown partway through. The GC and IDisposable thus solve complementary problems: the GC reclaims managed memory automatically and eventually; IDisposable releases scarce unmanaged resources promptly and predictably, on the application's own schedule rather than the GC's.

Q3: What problem does Span<T> solve, and why is it restricted to the stack?

Span<T> solves the problem of unnecessary allocation when working with a CONTIGUOUS SLICE of existing memory — an array, a string, or stack-allocated memory. The traditional approach, like string.Substring(start, length), always allocates a brand-new object on the heap containing a copy of that slice's data, even if you only need to read it briefly; in code that does this repeatedly in a hot path (parsing large text, processing incoming network buffers, high-throughput serialization), these allocations add up to significant garbage-collector pressure and measurable overhead. Span<T> (and ReadOnlySpan<T>) instead represents a VIEW into existing memory — essentially a pointer plus a length — letting you slice, read, and pass around a subsection of an array or string without copying or allocating anything at all, uniformly whether the underlying memory is a heap array, a string, or genuinely stack-allocated memory. The critical design constraint is that Span<T> is a `ref struct`, which the compiler enforces cannot be stored as a field in a regular (heap-allocated) class, cannot be boxed, and cannot be used across an `await` in an async method — because doing any of those things could let a Span outlive the actual memory it points into (for example, if that memory was stack-allocated and the stack frame has since been popped), which would be a serious memory-safety violation. By restricting Span<T> to stack-only, purely local usage, the compiler can statically guarantee it never dangles, giving you the performance of raw pointer-like slicing with full memory safety — at the cost of it being usable only in narrower, more local scenarios than a regular object.

🏋️ Practice: allocate objects in a loop and observe Gen 0/1/2 collection counts with a profiler or GC.CollectionCount(); implement IDisposable on a class wrapping a FileStream and use `using` to guarantee cleanup; deliberately omit `using` and observe a file handle staying open; replace repeated Substring calls in a parsing loop with Span<T> slicing and compare allocations; try (and observe the compile error from) storing a Span<T> as a class field; implement the full Dispose pattern with a finalizer as a safety net.

🎯 Key Takeaways

  • Value types generally live on the stack (fast, no GC); reference types live on the GC-tracked heap, organized into generations (Gen 0 cheap/frequent, Gen 2 expensive/rare) based on "most objects die young."
  • The GC only manages memory — unmanaged resources (files, sockets, connections) need IDisposable + `using` for deterministic, prompt cleanup.
  • Span<T>/ReadOnlySpan<T> slice existing memory with zero allocation, replacing costly operations like Substring in hot paths.
  • Span<T> is a stack-only ref struct — can't be a class field, boxed, or held across an await — the compiler enforces it never outlives its memory.
  • Excessive/unnecessary allocation (not "the GC will handle it") is the real performance concern in high-throughput .NET services — minimize allocation rate and avoid needless promotion to Gen 2.
