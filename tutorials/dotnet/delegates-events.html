Delegates, Events & Lambdas Intermediate

A delegate is a type-safe reference to a method — C#'s mechanism for treating functions as values, which underlies LINQ, async callbacks, and the entire event system. Understanding delegates demystifies a lot of "how does this actually work under the hood" in C#.

Delegates — Type-Safe Function Pointers

public delegate int Operation(int a, int b);   // defines a TYPE: "a method taking (int,int), returning int"

int Add(int a, int b) => a + b;
Operation op = Add;              // op now REFERENCES the Add method
op(3, 4);                        // 7 — invoking the delegate calls Add

// Built-in generic delegate types cover nearly every case — you rarely define custom ones:
Func add = (a, b) => a + b;      // takes params, RETURNS a value
Action log = msg => Console.WriteLine(msg);   // takes params, returns VOID
Predicate isEven = n => n % 2 == 0;         // takes a param, returns bool (used by List.Find etc.)
🎯 Func, Action, and Predicate ARE delegates — just pre-defined generic ones. Every LINQ method you've used (Where(Func<T,bool>), Select(Func<T,TResult>)) is a method that accepts a delegate parameter. Lambdas (`x => x.IsActive`) are just concise syntax for creating a delegate instance inline instead of defining a named method elsewhere.

Multicast Delegates — One Delegate, Multiple Targets

Action logger = Console.WriteLine;
logger += msg => File.AppendAllText("log.txt", msg);   // += ADDS another method to invoke
logger("Hello");         // invokes BOTH: prints to console AND appends to the file
logger -= Console.WriteLine;   // -= REMOVES a method from the invocation list
// A delegate can wrap MULTIPLE methods — invoking it calls each in order. This is the foundation
// events are built on: subscribing to an event is literally += on a multicast delegate.

Events — a Safer, Encapsulated Delegate Pattern

public class Button {
    public event EventHandler? Clicked;   // `event` restricts external code to +=/-= only

    public void SimulateClick() {
        Clicked?.Invoke(this, new ClickEventArgs());       // ?. guards against ZERO subscribers (null)
    }
}

var button = new Button();
button.Clicked += (sender, args) => Console.WriteLine("Clicked!");   // subscribe
// External code can ONLY += or -= to `Clicked` — it CANNOT call button.Clicked(...) directly or
// reassign it wholesale (button.Clicked = null). The `event` keyword enforces this encapsulation,
// which a plain public delegate field would NOT — a key interview distinction.
🔒 event vs a plain delegate field: encapsulation. A public Action Clicked field lets ANY external code overwrite it entirely (obj.Clicked = someHandler, silently discarding all other subscribers) or invoke it directly, bypassing the publisher's intended control. The `event` keyword restricts external access to only += and -=, so only the DECLARING class can raise the event or clear all subscribers — this is why C# events, not raw delegates, are the idiomatic pattern for publish/subscribe.

Closures — Lambdas Capturing Variables

int count = 0;
Action increment = () => count++;    // the lambda CAPTURES `count` by reference (a closure)
increment(); increment();
Console.WriteLine(count);            // 2 — the outer variable was actually modified

// A classic closure trap — capturing a LOOP variable:
var actions = new List();
for (int i = 0; i < 3; i++) {
    actions.Add(() => Console.WriteLine(i));   // in modern C# (7.0+), each iteration has its OWN `i`
}
actions.ForEach(a => a());   // prints 0, 1, 2 — fixed in modern C#! (older C#/JS var would print 3,3,3)

Interview Questions

Q1: What is a delegate, and how do Func, Action, and Predicate relate to it?

A delegate is a type that represents a reference to a method with a specific signature — it's C#'s type-safe equivalent of a function pointer, letting you treat methods as values: pass them as arguments, store them in variables/fields, and invoke them indirectly without the caller needing to know which concrete method will actually run. Declaring `delegate int Operation(int a, int b);` defines a new delegate TYPE, and any method matching that signature (or a lambda with that signature) can be assigned to a variable of that type and invoked like a method call. Func<T1,...,TResult>, Action<T1,...>, and Predicate<T> are not a separate concept — they ARE delegates, just generic ones already defined in the .NET base class library so you rarely need to declare custom delegate types yourself. Func represents a delegate that takes zero or more parameters and RETURNS a value (the last type parameter is the return type); Action represents one that takes parameters and returns void; Predicate<T> is specifically a delegate taking one T and returning bool, used by APIs like List<T>.Find. Every LINQ method — Where(Func<T,bool> predicate), Select(Func<T,TResult> selector) — is simply a method accepting a delegate parameter, and lambda expressions are concise inline syntax for constructing a delegate instance without writing a separate named method.

Q2: Why does C# have a separate `event` keyword instead of just using public delegate fields for publish/subscribe?

A delegate is inherently multicast — you can chain multiple methods onto it with += and invoke them all — which makes it a natural fit for the observer/publish-subscribe pattern where multiple subscribers want to be notified of something. But if you simply expose a PUBLIC delegate field (say, `public Action Clicked;`), any external code has full, unrestricted access to that field: it can subscribe with +=, but it can also invoke the delegate directly as if it owned the event (button.Clicked?.Invoke() from OUTSIDE the class), or — much more dangerously — overwrite it entirely with a plain assignment (button.Clicked = someHandler), which silently discards every other subscriber that had previously registered, without their knowledge. The `event` keyword solves this by restricting what external code can do with the member to ONLY += and -=; it cannot invoke the event directly or reassign it wholesale from outside the declaring class — only code inside the class that declared the event can actually raise it (call Invoke) or clear it. This encapsulation is exactly why C# events, not raw public delegate fields, are the idiomatic and safe mechanism for publish/subscribe: publishers retain exclusive control over when the event fires, while still allowing any number of independent subscribers to safely register and unregister interest.

Q3: What is a closure, and what's the classic "capturing a loop variable" pitfall?

A closure is a lambda (or anonymous method) that references a variable from its enclosing scope — the lambda "closes over" that variable, capturing it by REFERENCE rather than by value at the moment the lambda is created. This means if the outer variable's value changes after the lambda is defined but before it's invoked, the lambda sees the UPDATED value, and if the lambda itself modifies the variable (like a counter incremented inside the lambda), that modification is visible to code outside the lambda too, since they share the same underlying variable, not independent copies. The classic pitfall historically was capturing a for-loop's iteration variable: in many older languages (and historically in C# before a language-level fix), a single loop variable `i` was reused across all iterations, so a list of lambdas created inside the loop (`() => Console.WriteLine(i)`) would all capture the SAME variable, and by the time any of them actually ran (after the loop finished), they'd all print the loop's final value rather than the value at the time each lambda was created. Modern C# (from C# 5 for foreach, and C# 7 semantics generally treat each for-loop iteration as introducing a fresh variable) fixed this specific case so that each lambda captures its own iteration's value as expected, but the underlying closure-by-reference behavior remains true generally — it's still important to understand that lambdas capture variables, not snapshots of their values, whenever an outer variable might change between a lambda's creation and its eventual invocation.

🏋️ Practice: define a custom delegate type, then rewrite the same logic using Func/Action instead; build a multicast Action with += that logs to both console and a file; implement a class exposing a public `event` and demonstrate that external code can only += and -=, not reassign it; trigger an event with the ?.Invoke() null-check pattern; write a closure that captures and mutates an outer counter variable; reproduce and explain the loop-variable-capture behavior in a for loop with lambdas.

🎯 Key Takeaways

  • A delegate is a type-safe reference to a method — lets you pass functions as values; Func/Action/Predicate are the built-in generic delegates covering nearly every case.
  • Delegates are multicast — += chains multiple methods, -= removes one; this is the foundation events are built on.
  • Use `event`, not a plain public delegate field, for publish/subscribe — it restricts external code to += / -= only, preventing wholesale reassignment or direct external invocation.
  • Always guard event invocation with ?.Invoke(...) to avoid a NullReferenceException when there are zero subscribers.
  • Lambdas form closures — they capture outer variables BY REFERENCE, not a value snapshot; modern C# gives each loop iteration its own captured variable.
