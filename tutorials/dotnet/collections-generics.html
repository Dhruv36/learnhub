Collections & Their Internals Core Libraries

What you'll master: the major collection types and — crucially — their internal data structures and Big-O costs, how to choose the right collection for an access pattern, the interface hierarchy (IEnumerable → IReadOnlyList → IList), returning collections from APIs, and the concurrent collections. Prerequisites: Generics (all collections are generic), Equality (hash collections depend on it). Choosing the wrong collection is one of the most common — and most invisible — performance mistakes.

Part 1 — The Core Collections & Their Data Structures

// The choice is really "which DATA STRUCTURE fits my access pattern" — the collection
// type is just its .NET name. Know the structure, and the Big-O follows.

List<T>              // a DYNAMIC ARRAY — contiguous memory, index-addressable
  [index]           O(1)   — direct offset into the array
  Add (amortized)   O(1)   — until capacity fills, then O(n) resize (doubles, so amortized O(1))
  Insert/RemoveAt   O(n)   — shifts all following elements
  Contains/IndexOf  O(n)   — linear scan

Dictionary<K,V>      // a HASH TABLE — key → bucket via GetHashCode (records-structs.html)
  [key] / TryGetValue O(1) average — hash to a bucket, compare within it
  Add / Remove       O(1) average
  (worst case O(n) with pathological hash collisions — rare with good hashes)

HashSet<T>           // a hash table of KEYS only — set membership
  Contains / Add / Remove  O(1) average
  set operations (Union, Intersect, Except)  — efficient, built-in

Queue<T>             // FIFO — Enqueue/Dequeue O(1)   (task queues, BFS)
Stack<T>             // LIFO — Push/Pop O(1)          (undo, DFS, call-stack-like)
LinkedList<T>        // doubly-linked list — O(1) insert/remove GIVEN a node; O(n) to FIND
                    //   (rarely the right choice — List's cache locality usually wins)
SortedDictionary<K,V> / SortedSet<T>   // balanced tree (red-black) — O(log n) ops, KEEPS SORTED
SortedList<K,V>      // sorted ARRAY — O(log n) lookup, O(n) insert; least memory of the sorted three

The single most useful thing to internalize about collections is that each is a named data structure, and its performance is that structure's performance — so if you know List<T> is a dynamic array and Dictionary<K,V> is a hash table, the Big-O costs aren't a table to memorize but a consequence you can derive. List<T> gives O(1) indexing and amortized-O(1) appends (it doubles capacity when full, so occasional O(n) resizes average out) but O(n) inserts/removes in the middle and O(n) search. Dictionary and HashSet give O(1) average lookup by hashing to a bucket — the payoff of the equality/hash contract from Records & Equality, and the reason a poor GetHashCode quietly degrades them to O(n). Queue/Stack model FIFO/LIFO access in O(1), and the sorted collections trade O(log n) operations for maintained order. Knowing the structure behind the name is what lets you choose correctly under the performance pressure of Part 2.

Part 2 — Choosing the Right Collection

// The decision is driven by your DOMINANT access pattern:

// "I look things up BY KEY" (the most common real need):
Dictionary<string, User> usersById;         // O(1) lookup — NOT List + FirstOrDefault (O(n)!)
// The #1 collection mistake: List<User> then repeatedly users.First(u => u.Id == id).
//   Each lookup is O(n); in a loop that's O(n²). A Dictionary makes each O(1).

// "I check membership / need uniqueness":
HashSet<string> seenIds;                     // O(1) Contains — NOT list.Contains (O(n))
// Deduplication, "have I processed this?", visited-sets in graph traversal.

// "I need order-of-insertion + index access":
List<Order> orders;                          // the default general-purpose collection

// "FIFO processing" → Queue<T>   |   "LIFO / undo" → Stack<T>

// "I need it always sorted" → SortedDictionary / SortedSet   (vs sort-on-demand: List.Sort())

// "Immutable / read-only exposure" → ImmutableArray<T>, IReadOnlyList<T> (Part 4)

// THE COST OF GETTING IT WRONG — a concrete example:
var users = GetUsers();                      // List<User> of 10,000
foreach (var order in orders)                // 5,000 orders
    var u = users.First(x => x.Id == order.UserId);   // O(n) EACH → 50,000,000 comparisons
// Fix: build a lookup ONCE, then O(1) per order:
var byId = users.ToDictionary(u => u.Id);    // O(n) once
foreach (var order in orders)
    var u = byId[order.UserId];              // O(1) each → 5,000 lookups total
// Same result, ~10,000× fewer operations. This pattern (build a dictionary to avoid
// repeated linear search) is one of the highest-leverage optimizations you'll ever apply.

Collection choice is dictated by your dominant access pattern, and the highest- impact rule is: if you look things up by a key, use a Dictionary, not a List with repeated First(x => x.Id == id). That List-plus-linear- search pattern is the most common invisible performance bug in real codebases — each lookup is O(n), and doing it inside a loop makes the whole thing O(n²), so a nested loop over 10,000 users and 5,000 orders does 50 million comparisons where a dictionary would do 5,000. The fix — build a Dictionary once with ToDictionary, then look up in O(1) — is one of the highest-leverage optimizations in all of application development, and it costs one line. The rest of the decisions follow the same "match the structure to the access pattern" logic: HashSet for membership/uniqueness, Queue/Stack for FIFO/LIFO, sorted collections when you need maintained order, and List as the sensible default when you just need an ordered, indexable sequence.

Part 3 — The Interface Hierarchy

// Collections implement a hierarchy of interfaces — depend on the LEAST powerful one that
// meets your need (accept the widest input, return the narrowest capability):

IEnumerable<T>         // "can be iterated" — the base. foreach, LINQ. NO count, NO index.
   ↑                  //   lazy-friendly: a query, a generator, an infinite sequence all qualify.
IReadOnlyCollection<T> // + Count (but still read-only, no mutation, no index)
   ↑
IReadOnlyList<T>       // + [index] (indexed, read-only)     ← great for RETURNING data
   ↑
ICollection<T>         // + Add/Remove/Clear (mutable)
   ↑
IList<T>               // + indexed mutation                 ← full mutable list contract
   ↑
List<T>               // the concrete implementation

// The GUIDANCE (Postel's law for APIs):
//   PARAMETERS: accept the WEAKEST interface that works — usually IEnumerable<T>.
void Process(IEnumerable<Order> orders)   // ✓ accepts List, array, HashSet, LINQ query, ...
void Process(List<Order> orders)          // ✗ needlessly rejects arrays and queries

//   RETURNS: return an interface that hides mutability you don't want callers to have.
public IReadOnlyList<Order> GetOrders()   // ✓ callers can index+count but NOT mutate your list
public List<Order> GetOrders()            // ✗ callers can .Clear() your internal state (oop.html!)

// The one caveat: IEnumerable<T> might be a LAZY query that re-executes on each enumeration
// (linq.html) — if a caller foreach-es it twice, the work happens twice. Materialize
// (.ToList()) at the boundary when you're returning a computed result callers will reuse.

Collections implement a ladder of interfaces from IEnumerable<T> (just "can be iterated") up through read-only and mutable list contracts, and using them well follows a version of Postel's law: accept the weakest interface that meets your need, return the narrowest capability you want to expose. For parameters, take IEnumerable<T> when you only iterate — it accepts lists, arrays, sets, and LINQ queries alike, where demanding a concrete List<T> needlessly rejects callers. For return values, prefer IReadOnlyList<T> over List<T> so callers can index and count but can't mutate your internal state — the same encapsulation principle from the OOP lesson, applied at the API boundary (handing back your live List lets any caller .Clear() it). The one subtlety is that IEnumerable<T> may be a lazy query that re-runs each time it's enumerated (the deferred-execution story of the LINQ lesson), so when you're returning a computed result callers will iterate more than once, materialize it with ToList() at the boundary.

Part 4 — Returning Collections & Immutability

// The problem: returning your internal mutable list lets callers corrupt your state.
public class Basket
{
    private readonly List<Item> _items = [];
    public List<Item> Items => _items;         // ✗ caller can basket.Items.Clear()!
}

// Options, from cheapest to strongest:
public IReadOnlyList<Item> Items => _items;     // ✓ read-only VIEW — no copy, no mutation.
    // caveat: it's a view — if _items changes, the caller's reference sees the change.
    // (and a determined caller can cast back to List — it's a convention, not a fortress.)

public IReadOnlyList<Item> GetItems() => _items.ToList();   // ✓ a defensive COPY (snapshot).
    // stronger isolation (caller's snapshot never changes) but O(n) allocation each call.

public ImmutableArray<Item> Items { get; }      // ✓✓ TRULY immutable — cannot be mutated,
    // shareable freely, thread-safe. Modifications return a NEW collection (structural sharing).

// The IMMUTABLE collections (System.Collections.Immutable):
ImmutableArray<T>     // immutable array — fast reads, "modify" = new array (cheap for small)
ImmutableList<T>      // tree-backed — O(log n) ops, "modify" reuses structure (cheap for large)
ImmutableDictionary<K,V>, ImmutableHashSet<T>
var updated = list.Add(item);   // returns a NEW list; 'list' is unchanged (records-structs vibe)

// FROZEN collections (net8+) — build once, read many, MAXIMALLY fast reads:
var lookup = users.ToFrozenDictionary(u => u.Id);   // slower to build, fastest possible lookup
    // perfect for reference data loaded at startup and read forever after.

Returning collections is where encapsulation meets the API boundary, and the options form a spectrum of cost versus safety. Exposing your internal List<T> directly is the mistake — any caller can mutate your object's state through it. The cheapest fix is an IReadOnlyList<T> view (no copy, no mutation, but it reflects later changes and can technically be cast back). A defensive ToList() copy gives the caller a stable snapshot at the cost of an allocation. And the immutable collections (ImmutableArray, ImmutableList, etc.) are genuinely unmodifiable — "changing" one returns a new collection with structural sharing, so they're freely shareable and thread-safe, the collection-level expression of the immutability discipline. The newer frozen collections (ToFrozenDictionary) optimize the specific build-once-read-forever pattern — slower to construct, fastest possible lookups — ideal for reference data loaded at startup, which is exactly the pattern where a dictionary's read performance matters most.

Part 5 — Concurrent Collections

// Standard collections are NOT thread-safe. Concurrent access without synchronization
// corrupts internal state — silent data loss, crashes, infinite loops (threading.html).
var dict = new Dictionary<int, int>();
Parallel.For(0, 1000, i => dict[i] = i);      // ✗ RACE — Dictionary isn't thread-safe. Corruption.

// The concurrent collections (System.Collections.Concurrent) — safe for multiple threads:
ConcurrentDictionary<K,V>    // thread-safe hash table; the workhorse of concurrent code
  dict.GetOrAdd(key, k => Compute(k));        // atomic get-or-compute — the key operation
  dict.AddOrUpdate(key, addValue, (k, old) => newValue);   // atomic update
  // NOTE: the FACTORY in GetOrAdd may run more than once under contention (only one result
  // wins) — keep it side-effect-free, or use Lazy<T> values for expensive/once-only work.

ConcurrentQueue<T> / ConcurrentStack<T>       // lock-free FIFO / LIFO
ConcurrentBag<T>             // unordered, optimized for same-thread add+remove
BlockingCollection<T>        // producer/consumer with blocking + bounding (Take waits for items)

// CHOOSING:
//   shared mutable dictionary across threads  → ConcurrentDictionary
//   producer/consumer pipeline                → BlockingCollection or Channel<T> (async-patterns.html)
//   read-mostly, rarely written               → ImmutableDictionary or a lock around a plain dict
//   single-threaded                           → the plain collections (never pay for concurrency
//                                                you don't need — concurrent collections have overhead)

Standard collections are deliberately not thread-safe — synchronization has a cost, and most collections are used by one thread — so concurrent access without coordination corrupts them silently (the subject of the Threading lesson). The concurrent collections provide thread-safe alternatives, and ConcurrentDictionary is the workhorse: its GetOrAdd and AddOrUpdate perform atomic get-or-compute and update operations that would otherwise require explicit locking. One sharp edge worth knowing: GetOrAdd's factory can run more than once under contention (only one result is kept), so the factory must be side-effect-free — or you wrap expensive once-only work in a Lazy<T>. For producer/consumer pipelines, BlockingCollection (or the async-friendly Channel<T> from the async patterns lesson) is the right tool. The overriding rule: only reach for concurrent collections when you genuinely share across threads — they carry synchronization overhead you shouldn't pay in single-threaded code.

Part 6 — Deep Dive: Capacity, Allocation & Performance Reality

// PRE-SIZE when you know the count — avoid repeated resize allocations:
var list = new List<int>(capacity: 10_000);   // one allocation vs ~14 doublings (16→32→...→16384)
var dict = new Dictionary<int, User>(capacity: expectedCount);
// Each List resize allocates a new backing array and copies everything — for large known
// sizes, pre-sizing eliminates that churn (and the resulting GC pressure).

// STRUCT-of-arrays / arrays for hot numeric paths — cache locality wins:
int[] scores = new int[n];       // contiguous, cache-friendly, no per-element overhead
// List<int> is nearly as good (it's a wrapped array), but a raw array or Span<int>
// (memory.html) is the floor for tight loops.

// AVOID the List<T>-in-the-middle-insert pattern in hot code:
list.Insert(0, item);            // O(n) — shifts EVERYTHING. In a loop → O(n²).
// If you're always inserting at the front, you wanted a Queue/LinkedList or to build
// backwards and reverse once.

// ENUMERATION allocations — foreach over List<T> uses a STRUCT enumerator (no allocation);
// foreach over IEnumerable<T> boxes the enumerator (an allocation per loop). In hot paths,
// iterate the concrete type or use CollectionsMarshal.AsSpan(list) for zero-overhead access.

// The MEASUREMENT discipline (performance-aot.html): these matter in HOT paths and at SCALE.
// Don't pre-size a 5-element list or agonize over enumerator boxing in a startup routine —
// profile first, optimize the collection usage the profiler actually flags. But DO know the
// Big-O of your data structures, because an O(n²) algorithm from a wrong collection choice
// is a problem no micro-optimization fixes.

Beyond choosing the right structure, a few realities govern collection performance at scale. Pre-sizing — passing a capacity to the constructor when you know the count — avoids the repeated resize-and-copy allocations that a growing List or Dictionary incurs (a list doubling from 16 to 16,384 reallocates ~14 times), which matters for large known sizes and the GC pressure they create. Arrays and Span<T> (from the Memory lesson) are the floor for tight numeric loops thanks to contiguous cache-friendly layout. And a couple of subtle allocations lurk: front-inserting into a List is O(n) (a loop of them is O(n²) — you wanted a different structure), and iterating an IEnumerable<T> boxes its enumerator where iterating a concrete List<T> uses a struct enumerator with no allocation. The framing that keeps this sane is the same as the Performance lesson's: these optimizations matter in hot paths and at scale — don't pre-size a five-element list — but you must always know the Big-O of your data structures, because a wrong collection choice creates an O(n²) algorithm that no micro-optimization can rescue.

Senior framing: collections are where algorithmic thinking meets everyday code, and the interview signal is whether you reason about data structures, not just API names. The set-piece question is "you have a List and you're calling First(x => x.Id == id) in a loop — what's wrong and how do you fix it?" — the answer being O(n²) from repeated linear search, fixed by building a Dictionary once (ToDictionary) for O(1) lookups, one of the most common and highest-impact optimizations in real systems. Follow-ups probe the internals: "why is Dictionary lookup O(1)?" (hashing to a bucket — and the corollary that a bad GetHashCode degrades it, tying back to Records & Equality), "when would you use a LinkedList?" (rarely — List's cache locality usually beats it despite LinkedList's theoretical O(1) inserts, a great test of whether you understand that Big-O isn't the whole story). The design-level tells: parameters typed as concrete List<T> instead of IEnumerable<T> (needlessly restrictive), public getters returning mutable List<T> that leak internal state (encapsulation break), and repeated linear searches that should be dictionary lookups. The mature posture: know each collection's underlying structure and Big-O cold, choose by dominant access pattern, expose the narrowest interface at boundaries, and reserve concurrent/immutable/frozen variants for the specific scenarios (threading, sharing, build-once-read-forever) that justify their trade-offs — while measuring before micro-optimizing the parts that don't change the algorithmic complexity.

Interview Questions

Q1 (Beginner): What data structures back List<T>, Dictionary<K,V>, and HashSet<T>, and what are their key operation costs?

List<T> is a dynamic array (contiguous memory): O(1) index access, amortized-O(1) Add (it doubles capacity when full, so occasional O(n) resizes average out), but O(n) insert/remove in the middle (shifts elements) and O(n) Contains/search. Dictionary<K,V> is a hash table: O(1) average lookup, add, and remove by key (hashing to a bucket, then comparing within it), degrading toward O(n) only with pathological hash collisions. HashSet<T> is a hash table of keys only: O(1) average Contains/Add/Remove, plus efficient set operations. Knowing the backing structure means you can derive the costs rather than memorize them.

Q2 (Beginner): When should you use a Dictionary instead of a List?

When your dominant access is lookup by key. A Dictionary gives O(1) average lookup, whereas finding an item in a List by a property (list.First(x => x.Id == id)) is O(n) — a linear scan. The difference is invisible for one lookup but catastrophic in a loop: N lookups over a List of M items is O(N·M), while a dictionary makes it O(N). If you ever search a list repeatedly by the same key, build a dictionary once (ToDictionary) and look up in O(1) — one of the most impactful optimizations in everyday code. Use a List when you need insertion order and index access and don't look up by key.

Q3 (Mid): Explain the collection interface hierarchy and the guidance for parameters vs return types.

From weakest to strongest: IEnumerable<T> (iterable, the base) → IReadOnlyCollection<T> (+Count) → IReadOnlyList<T> (+index, read-only) → ICollection<T> (+mutation) → IList<T> (+indexed mutation) → concrete List<T>. Guidance: accept the weakest interface that meets your need in parameters — usually IEnumerable<T>, so the method takes lists, arrays, sets, and LINQ queries alike; demanding a concrete List<T> needlessly restricts callers. Return the narrowest capability you want to expose — prefer IReadOnlyList<T> over List<T> so callers can index and count but not mutate your internal state (encapsulation). The caveat: IEnumerable<T> might be a lazy query that re-executes each enumeration, so materialize computed results (ToList) at the boundary if callers will iterate more than once.

Q4 (Mid): Why aren't standard collections thread-safe, and what are the concurrent alternatives?

Thread safety requires synchronization, which has a real cost (locks, memory barriers), and the vast majority of collections are used by a single thread — so the framework makes the fast, single-threaded case the default rather than making everyone pay for concurrency they don't use. Concurrent access to a standard collection without external synchronization races on its internal state (bucket arrays, size counters), causing silent corruption, lost writes, crashes, or infinite loops. The System.Collections.Concurrent types are the thread-safe alternatives: ConcurrentDictionary<K,V> (with atomic GetOrAdd/AddOrUpdate), ConcurrentQueue/ConcurrentStack (lock-free FIFO/LIFO), ConcurrentBag, and BlockingCollection for producer/consumer. Use them only when you genuinely share across threads; for single-threaded code the plain collections are correct and faster.

Q5 (Senior): A service does orders.Where(o => o.CustomerId == id) for each of thousands of customers, over a list of millions of orders, and it's slow. Diagnose and fix.

It's O(customers × orders) — for each customer, a full linear scan of all orders — so thousands × millions is billions of comparisons. The root cause is repeated linear search over a List keyed by a property. The fix is to invert it: build a lookup once. Since each customer has many orders, group them: var byCustomer = orders.ToLookup(o => o.CustomerId) (an ILookup — a dictionary of key → collection, built in one O(n) pass), then each customer's orders are byCustomer[id] in O(1). That converts O(customers × orders) into O(orders) to build plus O(customers) to look up — from billions of operations to millions. ToDictionary is the one-value-per-key version; ToLookup/GroupBy the many-values-per-key version, and recognizing which you need is the skill. If the orders come from a database, the deeper fix is to not load millions into memory at all — filter/group in the query (EF Core, efcore.html) so the database's indexes do the work. The general principle: repeated searches over the same collection by the same key should always become a pre-built dictionary/lookup.

Q6 (Senior): Walk through choosing collections for a system with reference data loaded at startup, a shared cache updated by many threads, and a work queue. Justify each.

Reference data (loaded once at startup, read forever) → a FrozenDictionary<K,V> (or FrozenSet): built via ToFrozenDictionary, it accepts a slower construction cost in exchange for the fastest possible read performance, which is exactly right when you build once and then do millions of lookups over the program's life; a regular Dictionary is fine too, but frozen is the purpose-built optimum. Shared cache updated by many threads → ConcurrentDictionary<K,V>: thread-safe without external locking, with atomic GetOrAdd for the classic "compute-and-cache-if-absent" pattern — noting the factory may run more than once under contention, so either keep it pure or store Lazy<T> values so the expensive computation happens exactly once. If the cache were read-mostly with rare writes, an ImmutableDictionary swapped atomically (or a reader-writer lock over a plain dictionary) could beat it. Work queue (producer/consumer) → a Channel<T> (async producer/consumer, the modern choice — async-patterns.html) or BlockingCollection<T> for the blocking-thread model: both give thread-safe handoff with bounding (backpressure) so producers don't outrun consumers; a bare ConcurrentQueue works but lacks the blocking/bounding that a real pipeline needs. Each choice is driven by the access pattern and concurrency requirement — the through-line of the whole lesson — and by not paying for capabilities (thread safety, immutability) where the scenario doesn't demand them.

foreach (var id in orderIds)
{
    var order = GetOrder(id);
    var user = usersById[order.UserId];              // O(1) — hash lookup
    result.Add(user.Name);
}
// Before: 10,000 × O(50,000) linear scans = 500,000,000 comparisons.
// After:  50,000 (build) + 10,000 (lookups) ≈ 60,000 operations. ~8,000× fewer.
// The single most valuable collection optimization: replace repeated linear search with
// a pre-built dictionary.
3 (Medium): Design a Catalog class that stores products internally but exposes them safely (callers can read/index but not mutate), plus an Add method. Show why the naive version is broken.
Solution
// ✗ NAIVE — leaks the internal list:
public class BadCatalog
{
    public List<Product> Products { get; } = [];   // caller: catalog.Products.Clear()! bypasses
}                                                  //   any rules, corrupts state (oop.html)

// ✓ ENCAPSULATED:
public class Catalog
{

    public void Add(Product product)                      // the ONLY mutation path — validated
    {
        if (_products.Any(p => p.Sku == product.Sku))
            throw new InvalidOperationException($"Duplicate SKU: {product.Sku}");
        _products.Add(product);
    }
}

4 (Medium): Show the difference between ToDictionary and ToLookup, and when each is correct, using an orders-by-customer example.
Solution
// ToDictionary — ONE value per key. Throws if keys aren't unique:
var orderById = orders.ToDictionary(o => o.Id);         // Id is unique → fine
// orders.ToDictionary(o => o.CustomerId);               // ✗ THROWS: a customer has many orders

// ToLookup — MANY values per key. Groups them; missing key returns EMPTY (never throws):
ILookup<int, Order> ordersByCustomer = orders.ToLookup(o => o.CustomerId);
var adaOrders = ordersByCustomer[42];                  // IEnumerable<Order> — all of customer 42's
var noneOrders = ordersByCustomer[999];                // EMPTY sequence, not an exception

// WHEN each:
//  ToDictionary → the key uniquely identifies ONE item (id → entity). O(1) single lookup.
//  ToLookup     → the key maps to a GROUP (customerId → their orders, category → products).
//                 Built in one O(n) pass; replaces repeated Where(x => x.Key == k) (O(n²)).
// Choosing wrong: ToDictionary on a non-unique key crashes; using neither and looping
// Where() is the O(n²) trap. Match the tool to "one-per-key" vs "many-per-key".
5 (Hard): Implement a thread-safe memoizing cache with ConcurrentDictionary that guarantees the expensive factory runs exactly once per key, even under heavy contention. Explain the subtlety.
Solution
// THE SUBTLETY: ConcurrentDictionary.GetOrAdd's factory can run MULTIPLE times under
// contention — several threads see a missing key, all invoke the factory, only one result
// is stored. For a pure/cheap factory that's just wasted work. For an EXPENSIVE or
// side-effectful factory (a DB call, a network fetch) running it N times is a real bug.

// FIX: store Lazy<T> values. GetOrAdd may create several Lazy WRAPPERS, but only ONE is
// stored, and .Value forces the factory exactly ONCE (Lazy is itself thread-safe):
public class MemoCache<TKey, TValue> where TKey : notnull
{
    private readonly ConcurrentDictionary<TKey, Lazy<TValue>> _cache = new();

    public TValue GetOrAdd(TKey key, Func<TKey, TValue> factory) =>
        _cache.GetOrAdd(key,
            k => new Lazy<TValue>(() => factory(k),
                LazyThreadSafetyMode.ExecutionAndPublication))   // exactly-once execution
        .Value;   // forcing .Value on the winning Lazy runs factory once; losers' Lazys are
                  //   discarded WITHOUT ever being forced, so factory never runs for them.
}

// Usage:
var cache = new MemoCache<int, User>();
var user = cache.GetOrAdd(42, id => LoadUserFromDatabase(id));   // DB hit happens once, ever
// Under 100 concurrent GetOrAdd(42) calls: possibly a few Lazy objects allocated (cheap),
// exactly ONE stored, LoadUserFromDatabase called EXACTLY ONCE, all callers get its result.
// The plain GetOrAdd(key, factory) overload would have called LoadUserFromDatabase up to
// 100 times. The Lazy wrapper is the standard idiom for expensive once-only concurrent init.
6 (Hard): You profile an app and find a hot method builds a List of 1M items with front-inserts, then does membership checks with Contains in a loop. Diagnose both problems and rewrite for correct complexity.
Solution
// THE BROKEN VERSION (both operations are O(n) inside loops → O(n²) total):
var items = new List<int>();
foreach (var x in source)          // 1,000,000 items
    items.Insert(0, x);            // ✗ O(n) — front insert SHIFTS all existing elements.
                                   //   1M front-inserts = ~500 billion element moves.
var seen = new List<int>();
foreach (var x in items)
    if (!seen.Contains(x))         // ✗ O(n) linear scan per check → O(n²) overall
        seen.Add(x);

// PROBLEM 1: List.Insert(0, ...) is O(n) — front insertion shifts the whole array each time.
// PROBLEM 2: List.Contains is O(n) — a linear scan; doing it per element is O(n²).

// THE FIX:
// (1) If you need REVERSED order, don't front-insert 1M times — append (amortized O(1))
//     and reverse ONCE, or build with a pre-sized list. If you need FIFO/LIFO behavior,
//     a Queue/Stack is O(1) per op.
var items = new List<int>(capacity: 1_000_000);   // pre-sized: no resize churn
foreach (var x in source) items.Add(x);           // O(1) amortized append
items.Reverse();                                  // ONE O(n) reverse, if reversal was the goal

// (2) Membership → HashSet: O(1) Contains/Add. And if you just want distinct items,
//     the whole loop IS a HashSet (or .Distinct()):
var seen = new HashSet<int>(items);               // O(n) total — each Add is O(1)
// or: var distinct = items.Distinct().ToList();

// COMPLEXITY: from O(n²) (billions of ops for n=1M) to O(n) (millions). The two lessons:
// front-insertion into an array-backed list is a trap (wrong structure for the access
// pattern), and repeated Contains on a List should be a HashSet. Neither is a
// micro-optimization — each fixes a quadratic algorithm that no amount of tuning saves.

🎯 Key Takeaways

  • Each collection is a named data structure with that structure's Big-O: List = dynamic array (O(1) index/append, O(n) search/mid-insert), Dictionary/HashSet = hash tables (O(1) average lookup), sorted collections = trees (O(log n)).
  • Choose by dominant access pattern — and the highest-impact rule is: lookup by key → Dictionary, never List + repeated First(x => x.Id == id) (which is O(n²) in a loop).
  • Accept the weakest interface (IEnumerable<T>) in parameters; return the narrowest capability (IReadOnlyList<T>) to protect internal state — materialize lazy queries at boundaries.
  • Immutable collections (ImmutableArray/List) are safely shareable and thread-safe; frozen collections optimize build-once-read-forever reference data.
  • Standard collections aren't thread-safe — use ConcurrentDictionary (mind the possibly-multiple GetOrAdd factory; wrap expensive init in Lazy<T>), Channel/BlockingCollection for producer/consumer — only when you actually share across threads.
  • Pre-size known-large collections, avoid front-inserts and repeated Contains on lists, and always know your structures' Big-O — a wrong collection makes an O(n²) algorithm no micro-optimization can fix.
