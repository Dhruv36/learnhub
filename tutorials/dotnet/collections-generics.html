Collections & Generics Basics

Choosing the right collection type is a top interview and production skill — the wrong choice causes real performance problems at scale. This lesson covers the core collection interfaces, when to reach for each concrete type, and how generics give you type-safe, reusable code without boxing overhead.

The Core Collection Interfaces (know the hierarchy)

InterfaceProvides
IEnumerable<T>Forward-only iteration — the base of LINQ (linq lesson)
ICollection<T>+ Count, Add, Remove, Contains
IList<T>+ Indexed access (list[i]), Insert/RemoveAt
IDictionary<TKey,TValue>Key-based lookup
ISet<T>Unique elements, set operations
// Accept the NARROWEST interface your method actually needs — maximizes flexibility for callers:
void PrintAll(IEnumerable items) { ... }     // caller can pass a List, array, HashSet, LINQ query...
// vs
void PrintAll(List items) { ... }            // forces callers to have (or create) a List specifically

Choosing a Concrete Collection Type

TypeBacking structureBest for
List<T>Resizable arrayOrdered data, index access, the general default
Dictionary<K,V>Hash tableO(1) average key lookup — the most-used lookup structure
HashSet<T>Hash tableUniqueness checks, set operations (O(1) Contains)
Queue<T>FIFO bufferProcess-in-order workloads (BFS, task queues)
Stack<T>LIFO bufferUndo history, DFS, backtracking
ConcurrentDictionary<K,V>Thread-safe hash tableShared state across threads without manual locking
// The most common performance bug: using List.Contains() in a hot loop instead of a HashSet.
var allowedIds = new HashSet(idsFromDb);   // O(1) average lookup
if (allowedIds.Contains(userId)) { ... }        // fast

var allowedIdsList = idsFromDb.ToList();
if (allowedIdsList.Contains(userId)) { ... }    // O(n) — scans the whole list every single call!
⚡ Big-O matters here in a very concrete way. List.Contains is O(n); Dictionary/HashSet lookups are O(1) average. In a loop calling Contains N times against a list of size N, that's an accidental O(n²) — a classic "worked fine in dev with 50 rows, fell over in prod with 500,000" bug. Reach for HashSet/Dictionary the moment you're checking membership repeatedly.

Generics — Type Safety Without Boxing

// A generic method works with ANY type, checked at compile time, with NO runtime type casting:
public static T Max(T a, T b) where T : IComparable {   // constraint: T must be comparable
    return a.CompareTo(b) > 0 ? a : b;
}
Max(3, 7);           // works for int
Max("a", "b");       // works for string
Max(3.5, 2.1);       // works for double
// Before generics, you'd use `object` parameters and cast — losing type safety AND paying a BOXING
// cost (wrapping a value type like int in a heap-allocated object) on every call. Generics avoid both.

Common Generic Constraints

where T : class            // T must be a reference type
where T : struct           // T must be a value type
where T : new()             // T must have a public parameterless constructor
where T : IComparable    // T must implement this interface
where T : BaseClass         // T must inherit from BaseClass

Common Mistakes

MistakeSymptomFix
Using List<T>.Contains() for repeated membership checksO(n) per check → O(n²) overall in a loopHashSet<T> or Dictionary<K,V> for O(1) lookups
Accepting List<T> as a parameter when IEnumerable<T> sufficesForces unnecessary allocations from callersAccept the narrowest interface actually needed
Using a plain Dictionary from multiple threadsRace conditions/corruptionConcurrentDictionary or external synchronization
Boxing value types via non-generic collections (ArrayList)Extra allocations, slower, no type safetyUse generic collections (List<T>, not ArrayList)
Mutating a collection while iterating it (foreach)InvalidOperationExceptionIterate a copy, or use a for loop with careful indexing
Ignoring generic constraints when they'd catch bugs at compile timeRuntime errors that could've been compile-time errorsConstrain T as tightly as the method's logic requires

Interview Questions

Q1: When would you use a List versus a Dictionary versus a HashSet?

The choice comes down to the access pattern you need. List<T> is a resizable array — use it when you need ordered data with positional/index access, or you're simply building up a sequence of items to iterate; it's the general-purpose default, but checking membership with .Contains() is O(n) since it scans linearly. Dictionary<TKey, TValue> is a hash table optimized for key-based lookup — use it whenever you need to retrieve a value by some identifying key with O(1) average time, which is by far the most common "fast lookup" need (looking up a user by ID, a config value by name, and so on). HashSet<T> is also a hash table but stores only unique elements with no associated value — use it specifically for uniqueness enforcement and O(1) average membership testing (`Contains`), and for set operations like union/intersection/except. A very common real-world performance bug is calling .Contains() repeatedly against a List inside a loop, which silently turns an intended O(n) operation into an accidental O(n²) one; switching that list to a HashSet (or the target of the check to a Dictionary) for repeated membership checks is one of the highest-leverage, lowest-effort performance fixes available.

Q2: What are generics and what problems do they solve compared to using `object`?

Generics let you write a class, method, or interface parameterized by a type (T), so the SAME code works correctly and efficiently for many concrete types, with the compiler enforcing type correctness at compile time. Before generics (or in languages/APIs that fall back to `object`), you'd write a method taking object parameters and cast to the real type inside — this loses compile-time type safety (you can pass in the wrong type and only find out at runtime via an InvalidCastException), and for value types like int it also incurs BOXING: wrapping the value in a heap-allocated object to fit the object parameter, which costs an allocation and adds GC pressure on every single call. A generic method like `T Max<T>(T a, T b) where T : IComparable<T>` avoids both problems — the compiler generates type-specific code paths (or, more precisely, ensures type safety with no boxing for value types), the call site is fully type-checked (Max("a", 5) is a compile error, not a runtime surprise), and there's no casting or boxing overhead. Generic constraints (`where T : IComparable<T>`, `where T : class`, `where T : new()`) let you additionally require that T support specific operations the method's body needs, keeping the API both flexible and safe.

Q3: Why should a method accept IEnumerable<T> instead of List<T> when possible?

Accepting the narrowest interface a method genuinely needs maximizes flexibility for every caller while fully preserving what the method itself requires. If a method only needs to iterate over items once (like a PrintAll or Sum helper), declaring the parameter as IEnumerable<T> lets ANY caller pass in a List, an array, a HashSet, a Dictionary's Values, or even a lazily-evaluated LINQ query — without that caller needing to materialize a concrete List first just to satisfy an overly specific parameter type. If the parameter were instead typed as List<T>, a caller holding an array or the result of a LINQ query would be forced to call .ToList() first, incurring an unnecessary allocation and copy purely to satisfy the signature, even though the method never needed List-specific capabilities like indexed access. The general principle — accept the most general/abstract type your logic actually requires, and return the most specific/concrete type reasonable — keeps APIs flexible for consumers while still being precise about what the implementation depends on; if a method genuinely needs indexed access or Insert/RemoveAt, then IList<T> or List<T> is the honest and correct signature, but that should be a deliberate choice based on real requirements, not a default.

🏋️ Practice: benchmark List.Contains vs HashSet.Contains in a loop over 100,000 lookups and observe the difference; write a generic method with an IComparable<T> constraint and call it with multiple types; refactor a method that takes List<T> to take IEnumerable<T> instead and confirm it still works with an array/HashSet; try mutating a List while foreach-iterating it and observe the exception; use a ConcurrentDictionary from multiple tasks and confirm it doesn't corrupt under concurrent writes.

🎯 Key Takeaways

  • Know the interface hierarchy (IEnumerable → ICollection → IList; IDictionary; ISet) and accept the narrowest one your method needs.
  • List<T> for ordered/indexed data; Dictionary<K,V> for O(1) key lookup; HashSet<T> for O(1) membership/uniqueness — the most common performance lever.
  • List.Contains() in a loop is O(n) per call → accidental O(n²) overall; switch to HashSet/Dictionary for repeated membership checks.
  • Generics give compile-time type safety with no boxing/casting, unlike using `object` — constraints (where T : ...) let the method require specific capabilities of T.
  • Queue<T>/Stack<T> for FIFO/LIFO workloads; ConcurrentDictionary for safe multi-threaded shared state.
