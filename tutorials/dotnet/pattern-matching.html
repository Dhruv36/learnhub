Pattern Matching & Switch Expressions Object-Oriented C#

What you'll master: the pattern vocabulary (type, property, relational, logical, list, and var patterns), switch expressions with exhaustiveness checking, tuple patterns for multi-value decisions, and the design lens — when pattern matching clarifies vs when it signals a missing polymorphic design. Prerequisites: Control Flow (switch basics), Records (deconstruction), and Inheritance (the polymorphism trade-off). This is where C#'s functional accent is strongest — and it's transformed how modern C# is written.

Part 1 — The Pattern Vocabulary

// A "pattern" tests a value's shape/type/contents and can BIND parts of it in one step.

// TYPE PATTERN — test type + cast + name, in one (replaces test-then-cast, control-flow.html):
if (shape is Circle c)                         // is it a Circle? if so, bind it as 'c'
    Console.WriteLine(c.Radius);               // c is a Circle here — no separate cast

// CONSTANT pattern — match specific values (incl. null):
if (status is null) { ... }
if (code is 404) { ... }

// RELATIONAL patterns — comparisons as patterns:
if (temp is > 30) { ... }                       // greater than 30
if (age is >= 18 and < 65) { ... }              // combined with 'and' (Part 2)

// PROPERTY pattern — match on an object's members:
if (order is { Total: > 1000m, Customer.Tier: Tier.Gold })   // nested property access!
    ApplyExpressReview(order);
if (point is { X: 0, Y: 0 })                     // origin

// LIST patterns (C# 11) — match array/list shape and elements:
if (args is [])                        { /* empty */ }
if (args is [var single])              { /* exactly one, bound to 'single' */ }
if (args is ["--verbose", .. var rest]) { /* first is flag, rest captured */ }
if (nums is [1, 2, ..])                { /* starts with 1, 2 */ }

// var pattern — always matches, binds the value (useful in switch arms for computed guards):
var result = input switch { var x when x.Length > 100 => "long", _ => "short" };

Pattern matching started as sugar for "test the type then cast" and grew into a rich sublanguage for testing a value's shape and extracting parts of it in a single expression. The type pattern (obj is Circle c) alone eliminated the old test-then-cast-then-use dance you saw in Control Flow. On top of that: constant patterns (including the null-safe is null), relational patterns (is > 30), property patterns that reach into members and even nested members ({ Customer.Tier: Tier.Gold }), and list patterns that match a collection's shape and bind slices ([first, .. rest]). Each is an expression that both tests and binds, which is what makes pattern-based code so much denser and clearer than the equivalent chains of ifs, casts, and null checks.

Part 2 — Switch Expressions & Logical Patterns

// The switch EXPRESSION (control-flow.html) becomes powerful when arms are PATTERNS:
decimal fee = account switch
{
    { Balance: < 0 }                      => 35m,        // property pattern arm
    { Type: AccountType.Premium }         => 0m,
    { Type: AccountType.Standard, Balance: >= 10_000m } => 0m,   // multiple conditions
    { Type: AccountType.Standard }        => 12m,
    null                                  => throw new ArgumentNullException(nameof(account)),
    _                                     => 25m,        // discard = default
};

// LOGICAL PATTERNS — 'and', 'or', 'not' combine patterns:
string category = temp switch
{
    < 0            => "freezing",
    >= 0 and < 15  => "cold",          // 'and'
    >= 15 and < 25 => "mild",
    >= 25         => "warm",
    _             => "unknown",       // NaN falls through
};
if (input is not null and not "")     { ... }     // 'not' negates a pattern
if (day is Saturday or Sunday)        { ... }     // 'or' — reads like English

// GUARDS with 'when' — arbitrary boolean conditions on an arm:
var shipping = order switch
{
    { Weight: > 20 } when order.IsInternational => 50m,   // pattern + extra condition
    { Weight: > 20 }                            => 25m,
    _                                           => 10m,
};

// EXHAUSTIVENESS — the compiler's superpower:
//   switch expressions WARN if you don't cover all cases. Add an enum member and every
//   switch missing it lights up — the compiler finds every place that needs updating.
//   (A statement switch or if/else chain gives you NO such safety.)

Switch expressions plus patterns are the modern C# idiom for turning decisions into values, and two features make them decisively better than the imperative alternatives. Logical patterns (and, or, not) let arms read almost like specifications — >= 15 and < 25 => "mild", is not null and not "" — and when guards attach arbitrary conditions for the cases patterns can't express alone. But the killer feature is exhaustiveness checking: the compiler warns when a switch expression over a closed set (an enum, a sealed hierarchy) doesn't cover every case, so when you add an OrderStatus.Refunded next quarter, every switch that forgot it produces a warning pointing you at the exact code that needs updating. An if/else chain or statement switch gives you none of that — the missed case just silently falls through — which is why value-producing decisions should be switch expressions.

Part 3 — Tuple Patterns: Multi-Dimensional Decisions

// When a decision depends on MULTIPLE values together, tuple patterns are the clean tool:
string Describe(int x, int y) => (x, y) switch
{
    (0, 0)           => "origin",
    (0, _)           => "on the Y axis",       // _ = don't care about this position
    (_, 0)           => "on the X axis",
    (var a, var b) when a == b => "on the diagonal",
    _                => "somewhere",
};

// The classic: a state-machine transition table as ONE readable switch:
State Next(State current, Event evt) => (current, evt) switch
{
    (State.Idle,     Event.Start)  => State.Running,
    (State.Running,  Event.Pause)  => State.Paused,
    (State.Running,  Event.Stop)   => State.Idle,
    (State.Paused,   Event.Resume) => State.Running,
    (State.Paused,   Event.Stop)   => State.Idle,
    _ => throw new InvalidOperationException($"No transition from {current} on {evt}"),
};
// The tuple switch IS the transition table — reviewable line-by-line against the spec,
// versus a nested if/else maze where a missing transition hides. (control-flow.html's
// "data beats branching" taken to its conclusion.)

// Combine with property patterns for rich rules — the pricing matrix from control-flow:
decimal Rate((OrderType type, CustomerTier tier) key) => key switch
{
    (OrderType.Express, CustomerTier.Gold) => 5m,
    (OrderType.Express, _)                 => 15m,
    (_, CustomerTier.Gold)                 => 8m,
    _                                      => 12m,
};

Tuple patterns are the answer to decisions that depend on several values at once — and they're transformative for the code that used to be nested-if mazes. Grouping the inputs into a tuple and switching over it produces a decision table you can read line by line against the specification: the state-machine (current, evt) switch literally is the transition table, so a missing or wrong transition is visible on inspection rather than buried three ifs deep. This is the ultimate expression of the "data beats branching" refactor from Control Flow: the pricing matrix, the state machine, the (type, tier, region) rule set — all become flat, reviewable, exhaustiveness- checked tables. When a domain expert can look at your switch and confirm it against their own matrix, the code has achieved the clarity that nested conditionals never can.

Part 4 — Deconstruction & Recursive Patterns

// Records deconstruct (records-structs.html), and patterns can match the deconstructed shape:
public record Point(int X, int Y);
public record Line(Point Start, Point End);

var description = line switch
{
    Line(( 0, 0), var end)              => $"from origin to {end}",   // NESTED deconstruction
    Line(var start, var end) when start == end => "zero-length",
    Line({ X: var x1 }, { X: var x2 }) when x1 == x2 => "vertical",   // property + binding
    _                                   => "general line",
};

// This is RECURSIVE pattern matching — patterns nest inside patterns, arbitrarily deep,
// binding names at any level. It lets you express complex structural conditions declaratively:
var result = shape switch
{
    Circle { Radius: 0 }                    => "degenerate circle",
    Circle { Radius: var r } when r < 0      => throw new ...,
    Rectangle { Width: var w, Height: var h } when w == h => "square",
    Rectangle r                             => $"{r.Width}x{r.Height}",
    _                                       => "unknown",
};

// Positional + property patterns combine freely:
if (order is { Items.Count: > 0, Customer: { IsVerified: true } as cust })
    Process(cust);           // bind a nested object AND assert conditions on it

// The 'is' expression version (not just switch) — great for guard clauses:
if (response is not { StatusCode: >= 200 and < 300 })
    return HandleError(response);       // reads: "if not a 2xx response, bail"

Because records deconstruct and patterns nest, C# supports recursive pattern matching — patterns inside patterns, binding names at any depth — which lets you express structural conditions that would otherwise be a thicket of null checks and property accesses. A single arm like Line((0, 0), var end) tests that a line starts at the origin and binds its endpoint; Rectangle { Width: var w, Height: var h } when w == h matches a square declaratively. This composes positional deconstruction, property patterns, type tests, and guards into one readable expression — and it works in is expressions too, not just switches, which makes it excellent for guard clauses (if (response is not { StatusCode: >= 200 and < 300 }) reads exactly like its intent). The density is the point: what would be a dozen lines of defensive checks becomes one pattern that states the shape you're looking for.

Part 5 — Pattern Matching vs Polymorphism: the Design Tension

// THE TENSION: switching on TYPE can be a code smell — it may signal a missing virtual method.

// ✗ Type-switch that SHOULD be polymorphism — every new shape edits this method:
decimal Area(Shape shape) => shape switch
{
    Circle c    => Math.PI * c.Radius * c.Radius,
    Rectangle r => r.Width * r.Height,
    Triangle t  => 0.5 * t.Base * t.Height,
    _           => throw new ...,          // add Hexagon → must edit THIS switch (Open/Closed!)
};

// ✓ Polymorphism — each shape owns its Area; new shapes need NO edits here (inheritance.html):
public abstract class Shape { public abstract decimal Area(); }
public class Circle : Shape { public override decimal Area() => ...; }
// Adding Hexagon just means writing Hexagon.Area() — nothing else changes.

// SO WHEN IS PATTERN MATCHING ON TYPE THE RIGHT CALL?
//  ✓ the types are CLOSED and you control them (a SEALED hierarchy — compiler exhaustiveness!):
public abstract record Json;
public sealed record JNull   : Json;
public sealed record JBool(bool Value)   : Json;
public sealed record JNumber(double Value) : Json;
public sealed record JString(string Value) : Json;
string Render(Json j) => j switch          // exhaustive over a SEALED hierarchy — compiler
{                                          // guarantees you handled every case
    JNull        => "null",
    JBool b      => b.Value ? "true" : "false",
    JNumber n    => n.Value.ToString(CultureInfo.InvariantCulture),
    JString s    => $"\"{s.Value}\"",
};  // no _ needed — sealed hierarchy is exhaustively covered

//  ✓ the types are OUTSIDE your control (you can't add a virtual method to them)
//  ✓ the behavior doesn't BELONG on the type (a rendering/serialization concern, not the
//    type's own responsibility — putting Render() on Json couples data to presentation)
//  ✗ you own the types AND the behavior is the type's own responsibility → polymorphism

The most important design idea in this lesson is the tension between pattern matching and polymorphism, because they solve overlapping problems from opposite directions. Switching on an object's type to choose behavior is often a smell — the classic Area(Shape) switch means every new shape forces an edit to that method (violating Open/Closed), whereas a virtual Area() lets new shapes slot in with no changes to existing code. But type-based pattern matching is genuinely right in several cases: when the type set is closed and you control it (a sealed hierarchy, where the compiler enforces exhaustiveness — add a case and every switch lights up); when the types are outside your control (you can't add a virtual method to a library type); and, crucially, when the behavior doesn't belong on the type — rendering, serialization, and mapping are concerns of a specific consumer, not responsibilities of the data itself, and forcing them onto the type as virtual methods couples the model to every consumer's needs. The rule of thumb: polymorphism when you own the types and the behavior is the type's own job; pattern matching when the hierarchy is sealed, foreign, or the behavior is a separate concern.

Part 6 — Deep Dive: Closed Hierarchies, Exhaustiveness & Functional C#

// SEALED hierarchies + exhaustive switches = discriminated unions (sum types), the
// functional-programming construct C# approximates today:
public abstract record Shape;
public sealed record Circle(double Radius) : Shape;
public sealed record Square(double Side)   : Shape;
public sealed record Rect(double W, double H) : Shape;
// 'sealed' on each leaf + 'abstract' base = a CLOSED set. The compiler knows these are
// ALL the shapes, so a switch over them is EXHAUSTIVELY checkable — miss one and it warns.

double Area(Shape s) => s switch
{
    Circle(var r)   => Math.PI * r * r,
    Square(var a)   => a * a,
    Rect(var w, var h) => w * h,
    // NO discard needed — the compiler proves this covers every Shape.
};  // add 'Triangle : Shape' → this switch (and every other over Shape) WARNS. Compiler-
    // driven refactoring: it finds every decision site the new case affects.

// This is a POWER pattern for modeling domains with a FIXED set of variants:
//   - Result/Either types (Ok | Error)
//   - AST nodes (Literal | BinaryOp | UnaryOp | ...)
//   - protocol messages, state variants, command types
// The combination — sealed hierarchy for the closed set, exhaustive switch for the
// dispatch — gives you type-safe, refactoring-friendly variant handling that neither
// pure polymorphism (scatters the logic across types) nor open type-switching (no
// exhaustiveness) provides.

// C# 13+ direction: the language is moving toward richer discriminated unions and more
// pattern features — this style is where idiomatic modern C# increasingly lives.

The deepest payoff combines two features from this track: a sealed hierarchy (from Inheritance) creates a closed set of variants the compiler knows completely, and an exhaustive switch expression dispatches over them with the compiler proving you handled every case — no discard arm needed, and adding a new variant produces warnings at every switch that now needs a new arm. This is C#'s current approximation of discriminated unions (sum types) from functional languages, and it's the ideal tool for domains with a fixed set of shapes: Result/Either types, AST nodes, protocol messages, command variants, state machines. It threads the needle that neither pure polymorphism nor open type-switching can: polymorphism scatters each operation across every type (bad when the behavior is a consumer's concern, not the type's), while open type-switching has no exhaustiveness guarantee (a missed case fails silently). Sealed-hierarchy-plus-exhaustive- switch gives type-safe, refactoring-friendly variant handling — and it's where idiomatic modern C# increasingly lives, with the language actively evolving toward first-class discriminated unions.

Senior framing: pattern matching is where "modern C#" is most visible, and interviews probe both fluency and judgment. Fluency: can you read and write property patterns, relational/logical patterns, tuple switches, and list patterns, and do you reach for switch expressions over statement switches for value-producing decisions (getting exhaustiveness for free)? Judgment — the deeper signal: do you know that switching on type can be a smell pointing at a missing virtual method, and can you articulate the exceptions (sealed hierarchies with compiler exhaustiveness, foreign types you can't extend, behavior that's a consumer's concern rather than the type's)? The set-piece question is "type-switch vs polymorphism — which and why?", and the strong answer isn't a dogmatic "always polymorphism" but the nuanced "polymorphism when I own the types and the behavior belongs on them; pattern matching over a sealed hierarchy when the behavior is a separate concern or I want compiler-enforced exhaustiveness." Recognizing sealed-hierarchy-plus-exhaustive-switch as C#'s discriminated-union pattern — and knowing when a fixed-variant domain (Result, AST, state machine) wants it — is a strong seniority tell, because it shows you understand not just the syntax but the type-system modeling it enables. In review, the tells: type-switches over hierarchies you own and could make virtual (usually refactor to polymorphism), non-exhaustive switches over closed sets (make the hierarchy sealed to unlock the compiler check), and imperative if/else mazes where a tuple switch would be a reviewable table.

Interview Questions

Q1 (Beginner): What is a type pattern, and how does if (x is Circle c) improve on the old approach?

A type pattern tests whether a value is a given type and, if so, casts and binds it to a new variable — all in one expression. if (x is Circle c) checks x's type, and inside the if, c is a Circle ready to use. The old approach needed three steps — test (if (x is Circle)), cast (var c = (Circle)x), and use — where the cast could drift from the test (someone changes the type in the check but not the cast) and it's more verbose. The pattern fuses them so they can't disagree, and it's the foundation the richer patterns build on.

Q2 (Beginner): Why prefer a switch expression over a switch statement for computing a value, and what safety does it add?

A switch expression produces a value directly (each arm is pattern => value), so there's no mutable temporary variable assigned across branches, no break to forget, and no accidental fall-through. Critically, it adds exhaustiveness checking: when switching over a closed set (an enum or sealed hierarchy), the compiler warns if you don't cover every case — so adding a new enum member surfaces a warning at every switch that missed it, turning "did I update all the decision sites?" from a manual hunt into a compiler-driven task. A statement switch or if/else chain offers none of this — the missed case just silently does the wrong thing (or hits the default). Reserve the statement form for arms that perform side effects rather than produce a value.

Q3 (Mid): Show how tuple patterns clean up a multi-condition decision, and why it's better than nested ifs.

When a decision depends on several values together, you group them into a tuple and switch over it, producing a flat decision table: (orderType, customerTier) switch { (Express, Gold) => 5m, (Express, _) => 15m, (_, Gold) => 8m, _ => 12m }. Compared to nested if/else, the advantages are concrete: the table maps directly to the business's own pricing/transition matrix, so a domain expert can verify it line by line; a missing combination is visible on inspection rather than hidden in nesting depth; each row is independent (no shared mutable state or ordering subtlety beyond top-to-bottom arm evaluation); and it's an expression, so it can be exhaustiveness- checked if the inputs are closed sets. The canonical example is a state machine, where (currentState, event) switch literally is the transition table — nested ifs bury the same logic in a maze where wrong or missing transitions hide.

Q4 (Mid): When is switching on a type a code smell, and when is it legitimately the right design?

Switching on type is a smell when you own the type hierarchy and the behavior is the type's own responsibility — e.g. an Area(Shape) method that switches over Circle/ Rectangle/Triangle. Every new shape forces an edit to that switch (violating Open/Closed), and the shape's own logic lives outside the shape; a virtual Area() is better, letting new shapes slot in with zero edits. It's legitimately right in three cases: (1) the type set is sealed and you control it, so the compiler enforces exhaustiveness and adding a variant surfaces every switch needing an arm (the discriminated-union pattern); (2) the types are foreign — you can't add virtual methods to library/third-party types; and (3) the behavior doesn't belong on the type — rendering, serialization, and mapping are concerns of a specific consumer, and forcing them onto the type as virtual methods couples the data model to every consumer. So the discriminator is ownership plus responsibility: own-the-types-and-behavior- belongs → polymorphism; sealed/foreign/separate-concern → pattern matching.

Q5 (Senior): Explain how sealed hierarchies plus exhaustive switches approximate discriminated unions, and why that's valuable.

A sealed record hierarchy (an abstract base with sealed leaves — JNull, JBool, JNumber, JString under Json) defines a closed set of variants the compiler knows completely. Switching over it exhaustively — with the compiler proving every variant is handled, so no discard arm is needed — gives you a discriminated union (sum type): a value that is exactly one of a fixed set of shapes, dispatched with compile-time completeness. The value is threefold: it's type-safe (you can't forget a case — the compiler warns), refactoring- friendly (add a variant and every switch that needs a new arm lights up, so the compiler drives the change across the codebase), and it separates data from operations (the variants are pure data; operations like render/evaluate/optimize live in switches wherever each consumer needs them, rather than being crammed onto the types as virtual methods). This threads the needle between pure polymorphism (which scatters every operation across every type and is wrong when the behavior is a consumer's concern) and open type-switching (which has no exhaustiveness guarantee). It's the ideal model for fixed-variant domains — Result/Either, AST nodes, protocol messages, state variants — and it's where modern idiomatic C# increasingly lives, with the language evolving toward first-class discriminated unions.

Q6 (Senior): You're building an expression evaluator (AST with Literal, Add, Multiply, Negate nodes). Design the types and evaluation, and justify pattern matching vs polymorphism.

Model the AST as a sealed record hierarchy: abstract record Expr; with sealed record Literal(double Value) : Expr;, sealed record Add(Expr Left, Expr Right) : Expr;, sealed record Multiply(Expr Left, Expr Right) : Expr;, sealed record Negate(Expr Operand) : Expr;. Then Evaluate is a recursive exhaustive switch: double Eval(Expr e) => e switch { Literal(var v) => v, Add(var l, var r) => Eval(l) + Eval(r), Multiply(var l, var r) => Eval(l) * Eval(r), Negate(var o) => -Eval(o) }; — the deconstruction patterns pull node children directly, and the sealed hierarchy makes it exhaustive with no discard arm. Why pattern matching over polymorphism here: an AST typically has many operations — evaluate, pretty-print, optimize, type-check, compile — and these are consumers' concerns, not the node's own responsibility. With polymorphism, each node type would carry an Evaluate, Print, Optimize, ... method, scattering each operation across every node and forcing all node classes to change whenever you add an operation (the expression problem's hard side). With sealed-hierarchy pattern matching, each operation is one self-contained function switching over the nodes, so adding an operation is one new function touching no node types — and adding a node type is one new record plus a compiler-flagged arm in each operation. This is the textbook case where pattern matching wins: closed variant set, behavior that belongs to consumers, and a strong desire for compiler-enforced exhaustiveness as the AST evolves. (If instead you had few operations that were genuinely the node's own behavior, polymorphism would be the cleaner choice — the design follows which axis changes more.)

1 (Easy): Rewrite this if/else chain as a switch expression with logical/relational patterns: 
string Grade(int score) {
  if (score < 0 || score > 100) throw new ArgumentException();
  if (score >= 90) return "A";
  else if (score >= 80) return "B";
  else if (score >= 70) return "C";
  else return "F";
}
Solution
string Grade(int score) => score switch
{
    < 0 or > 100 => throw new ArgumentException("Score out of range", nameof(score)),
    >= 90        => "A",
    >= 80        => "B",
    >= 70        => "C",
    _            => "F",
};
// Relational patterns (>= 90) and a logical pattern (< 0 or > 100) make the ranges read
// declaratively; arms are checked top-down so each assumes the earlier ones failed. throw
// is an expression, so validation is just another arm.

Solution
string Classify(Order? order) => order switch
{
    null                    => throw new ArgumentNullException(nameof(order)),
    { Items.Count: 0 }      => "empty",          // nested property pattern
    { Total: > 1000m }      => "large",
    _                       => "normal",
};
// { Items.Count: 0 } reaches into a nested member in one pattern — no order?.Items?.Count
// dance. Arm order matters: null first (so later arms can assume non-null), then the

3 (Medium): Implement a vending-machine state transition using a tuple pattern switch. States: Idle, HasMoney, Dispensing. Events: InsertCoin, SelectItem, Dispense, Cancel. Throw on illegal transitions.
Solution
enum State { Idle, HasMoney, Dispensing }
enum Event { InsertCoin, SelectItem, Dispense, Cancel }

State Next(State current, Event evt) => (current, evt) switch
{
    (State.Idle,       Event.InsertCoin) => State.HasMoney,
    (State.HasMoney,   Event.InsertCoin) => State.HasMoney,   // more coins, still HasMoney
    (State.HasMoney,   Event.SelectItem) => State.Dispensing,
    (State.HasMoney,   Event.Cancel)     => State.Idle,        // refund
    (State.Dispensing, Event.Dispense)   => State.Idle,
    _ => throw new InvalidOperationException($"Illegal: {evt} while {current}"),
};

Console.WriteLine(Next(State.Idle, Event.InsertCoin));       // HasMoney
Console.WriteLine(Next(State.HasMoney, Event.SelectItem));   // Dispensing
// Next(State.Idle, Event.Dispense);                         // throws — no such transition
// The switch IS the transition table: every legal move is one reviewable line, and the
// discard arm makes illegal transitions an explicit, loud failure rather than a silent
// wrong result. Compare to nested if (current == ...) { if (evt == ...) } mazes.
4 (Medium): Use list patterns to parse a simple command: [] → "help", ["version"] → "1.0", ["add", var a, var b] → sum them, ["echo", .. var rest] → join rest, anything else → "unknown".
Solution
string Run(string[] args) => args switch
{
    []                         => "help",
    ["version"]                => "1.0",
    ["add", var a, var b] when int.TryParse(a, out _) && int.TryParse(b, out _)
                               => $"{int.Parse(a) + int.Parse(b)}",
    ["echo", .. var rest]      => string.Join(' ', rest),   // slice pattern captures the tail
    _                          => "unknown",
};

Console.WriteLine(Run([]));                        // help
Console.WriteLine(Run(["version"]));               // 1.0
Console.WriteLine(Run(["add", "3", "4"]));         // 7
Console.WriteLine(Run(["echo", "a", "b", "c"]));   // a b c
Console.WriteLine(Run(["frobnicate"]));            // unknown
// List patterns match shape ([] empty, [x] one element, [head, .. tail] head+rest) AND
// bind pieces — turning argument parsing into a readable dispatch table. The 'when' guard
// handles the sub-condition (both operands numeric) that the pattern alone can't express.
5 (Hard): Model a JSON value as a sealed record hierarchy and write an exhaustive Render switch (no discard arm). Then add a JArray variant and describe what the compiler does.
Solution
public abstract record Json;
public sealed record JNull()             : Json;
public sealed record JBool(bool Value)   : Json;
public sealed record JNumber(double Value) : Json;
public sealed record JString(string Value) : Json;

string Render(Json json) => json switch
{
    JNull                => "null",
    JBool(var b)         => b ? "true" : "false",
    JNumber(var n)       => n.ToString(CultureInfo.InvariantCulture),
    JString(var s)       => $"\"{s}\"",
    // NO discard arm — the sealed hierarchy is CLOSED, so the compiler PROVES these four
    // cases cover every possible Json. It doesn't even warn about missing cases.
};

Console.WriteLine(Render(new JNumber(3.14)));   // 3.14
Console.WriteLine(Render(new JString("hi")));   // "hi"

// over Json anywhere in the codebase lights up too. This is compiler-DRIVEN refactoring:
// the sealed hierarchy turns "add a variant" into a guided task where the compiler finds

6 (Hard): Given a shape rendering requirement, decide between polymorphism and pattern matching for two scenarios and implement the better one: (a) computing Area for shapes you own; (b) rendering shapes to SVG for a specific export feature. Justify each choice.
Solution
// SHARED model — sealed hierarchy (you own the shapes, and they're a closed set):
public abstract record Shape;
public sealed record Circle(double Radius)       : Shape;
public sealed record Rect(double W, double H)    : Shape;
public sealed record Triangle(double B, double H) : Shape;

// (a) AREA → POLYMORPHISM. Area is the shape's OWN, intrinsic responsibility — a circle
//     inherently knows its area — and there's ONE such operation per shape. Putting it on
//     the type means new shapes are self-contained and no central code changes:
public abstract record Shape2 { public abstract double Area(); }
public sealed record Circle2(double Radius) : Shape2 { public override double Area() => Math.PI * Radius * Radius; }
public sealed record Rect2(double W, double H) : Shape2 { public override double Area() => W * H; }
// Add Hexagon2 → it defines its own Area(); nothing else touched. Behavior belongs on the type.

// (b) SVG RENDERING → PATTERN MATCHING. Rendering to SVG is an EXPORT FEATURE's concern,
//     NOT the shape's responsibility — a Shape shouldn't know about SVG (that couples the
//     domain model to a presentation format, and you'd repeat it for PNG, PDF, canvas...).
//     Keep it as a function in the export module, switching over the sealed hierarchy:
static string ToSvg(Shape shape) => shape switch
{
    Circle(var r)      => $"<circle r='{r}' />",
    Rect(var w, var h) => $"<rect width='{w}' height='{h}' />",
    Triangle(var b, var h) => $"<polygon points='0,0 {b},0 {b/2},{h}' />",
    // exhaustive over the sealed hierarchy — add a shape and this switch warns.
};

// JUSTIFICATION:
//  - Area is INTRINSIC + single-operation → polymorphism: cohesive (each shape owns its
//    area), Open/Closed (new shapes need no edits elsewhere).
//  - SVG (and future PNG/PDF) rendering is EXTRINSIC + one of MANY export operations →
//    pattern matching: keeps presentation OUT of the domain model, groups each export
//    format as its own function, and the sealed hierarchy gives compiler-checked
//    exhaustiveness so adding a shape flags every renderer. Forcing ToSvg() onto Shape
//    would couple the model to SVG and force every shape to change for every new format.
// The deciding question each time: is this behavior the type's OWN responsibility (→ poly)
// or a specific consumer's concern (→ pattern match over a sealed hierarchy)?

🎯 Key Takeaways

  • Patterns test a value's type/shape/contents and bind parts in one expression: type, constant, relational, property (incl. nested), list, and var patterns.
  • Switch expressions turn decisions into values with logical patterns (and/or/not), when guards, and — the key win — exhaustiveness checking over closed sets.
  • Tuple patterns turn multi-value decisions (state machines, pricing matrices) into flat, reviewable decision tables — the endpoint of "data beats branching."
  • Records deconstruct, so patterns nest recursively — expressing deep structural conditions declaratively, in is expressions and guard clauses too.
  • Switching on type can be a smell (missing virtual method) — but it's right for sealed hierarchies (compiler exhaustiveness), foreign types, and behavior that's a consumer's concern, not the type's.
  • Sealed hierarchy + exhaustive switch = C#'s discriminated unions: type-safe, refactoring-friendly variant handling for fixed-variant domains (Result, AST, state machines) — where modern idiomatic C# increasingly lives.
