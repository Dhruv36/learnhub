Inheritance, Polymorphism & Interfaces Object-Oriented C#

What you'll master: the mechanics of inheritance (virtual/ override/abstract/sealed), runtime polymorphism and why it's the point, interfaces as capability contracts (including default implementations), the pivotal "composition over inheritance" judgment, and abstract class vs interface. Prerequisites: Classes & Encapsulation. Inheritance is the most overused OOP tool — this lesson teaches both how it works and the discipline of reaching for it last.

Part 1 — Inheritance Mechanics

// Base class — shared state and behavior; virtual = "subclasses may override me":
public abstract class Employee                     // abstract: cannot be instantiated directly
{
    public string Name { get; }
    protected Employee(string name) => Name = name;   // protected: only subclasses call it

    public abstract decimal CalculatePay();        // abstract: NO body — subclasses MUST implement
    public virtual string Describe() => $"{Name}"; // virtual: HAS a default, subclasses MAY override
}

public class SalariedEmployee(string name, decimal annual) : Employee(name)   // : base(...)
{
    public override decimal CalculatePay() => annual / 12;      // MUST — it was abstract
    public override string Describe() => $"{base.Describe()} (salaried)";  // base.X calls up
}

public sealed class Contractor(string name, decimal rate, int hours) : Employee(name)
{
    public override decimal CalculatePay() => rate * hours;
    // sealed class: nobody can inherit from Contractor (ends the hierarchy)
}

// The keyword contract:
//   abstract method → no body, subclass MUST override
//   virtual method  → has a body, subclass MAY override
//   override        → replaces the base implementation (required keyword — no accidental override)
//   sealed          → on a class: no inheriting; on an override: no FURTHER overriding
//   new             → HIDES (doesn't override) a base member — almost always a bug, avoid

C# makes inheritance explicit and opt-in, unlike languages where every method is silently overridable: a method is only replaceable if the base marks it virtual (or abstract), and the subclass must say override — so you can never accidentally override, and the base author controls their own extensibility. abstract classes can't be instantiated and can declare bodyless abstract members that force subclasses to fill in; sealed ends a hierarchy (and is worth applying by default — most classes aren't designed to be inherited). The one keyword to avoid is new for member hiding: it makes a method that looks overridden but dispatches by the variable's compile-time type instead of the object's runtime type — a subtle bug generator, which Part 2 makes precise.

Part 2 — Polymorphism: the Whole Point

// Treat different concrete types uniformly through their shared base/interface:
List<Employee> staff =
[
    new SalariedEmployee("Ada", 120_000),
    new Contractor("Bob", 80, 160),
];

decimal payroll = staff.Sum(e => e.CalculatePay());   // each runs ITS OWN CalculatePay —
                                                      // the RUNTIME type decides, not the
                                                      // compile-time 'Employee' type
foreach (var e in staff)
    Console.WriteLine(e.Describe());                  // Ada (salaried) / Bob (base default)

// This is DYNAMIC DISPATCH: e is typed 'Employee', but the CLR calls the override on
// the actual object. It's what lets you add a new Employee subtype WITHOUT touching the
// payroll code — the Open/Closed Principle in action (solid comes up in modern-csharp).

// The 'new' (hiding) trap — why it's dangerous:
public class Base    { public          string Who() => "Base"; }
public class Derived : Base { public new string Who() => "Derived"; }   // HIDES, doesn't override
Base b = new Derived();
b.Who();                          // "Base"! — dispatched by the VARIABLE type (compile-time)
((Derived)b).Who();               // "Derived" — the same object, different declared type
// With a proper virtual/override, b.Who() would correctly say "Derived". Hiding breaks
// polymorphism silently — the object behaves differently depending on how you hold it.

Polymorphism — one interface, many runtime behaviors — is the reason inheritance exists. When you call e.CalculatePay() on an Employee-typed variable, the CLR dispatches to the actual object's override at runtime (dynamic dispatch), so payroll code written against the Employee abstraction works unchanged when you add a new subtype tomorrow. That's the payoff: code depends on the abstraction, and new concrete types slot in without edits (the Open/Closed Principle). The new-hiding trap is the anti-lesson: a hidden method dispatches by the variable's declared type, so the same object behaves differently depending on how you're holding it — a violation of the whole point of polymorphism, and the reason C# requires the explicit override keyword to prevent accidental hiding.

Part 3 — Interfaces: Capability Contracts

// An interface is a CONTRACT — "any type implementing me can do these things":
public interface IExporter
{
    string Format { get; }
    Task ExportAsync(IEnumerable<Order> orders, Stream output);
}

public class CsvExporter : IExporter { public string Format => "csv"; /* ... */ }
public class JsonExporter : IExporter { public string Format => "json"; /* ... */ }

// Code depends on the CONTRACT, not the concretion — swappable, testable, mockable:
async Task Export(IExporter exporter, IEnumerable<Order> orders) =>
    await exporter.ExportAsync(orders, File.Create($"out.{exporter.Format}"));

// A type can implement MANY interfaces (no multiple-inheritance-of-CLASSES limit):
public class Document : IPrintable, ISaveable, IDisposable { }

// The BCL is built on standard interface contracts you'll implement constantly:
//   IEnumerable<T>   — "can be foreach'd"           (linq.html)
//   IComparable<T>   — "has a natural sort order"
//   IEquatable<T>    — "defines value equality"     (records-structs.html)
//   IDisposable      — "owns resources to release"  (memory.html)

// DEFAULT INTERFACE METHODS (C# 8) — interfaces can provide implementations:
public interface ILogger
{
    void Log(string message);
    void LogError(string message) => Log($"ERROR: {message}");   // default — implementers
                                                                //   get it free, may override
}
// Use sparingly — for evolving an interface without breaking implementers, NOT as a
// substitute for base classes.

An interface declares what a type can do without any how — a pure capability contract. Its power is decoupling: code that depends on IExporter works with any implementation, present or future, and (critically) with a test double, which is the foundation of the dependency injection and testing lessons. A class can implement any number of interfaces (sidestepping the single-base-class limit), and the .NET standard library is a lattice of them — IEnumerable<T>, IComparable<T>, IDisposable — that your types plug into to gain framework behavior. Default interface methods let an interface ship a default body, useful for evolving a published interface without breaking every implementer — but they're a targeted evolution tool, not a license to build inheritance hierarchies out of interfaces.

Part 4 — Composition Over Inheritance: the Central Judgment

// THE CLASSIC INHERITANCE TRAP — modeling with "is-a" when reality is "has-a":
public class Duck : Bird { }                       // fine... until:
public class RubberDuck : Duck { }                 // a RubberDuck can't Fly() or Eat()!
                                                   // it inherited behavior it must now DISABLE
public class Penguin : Bird { }                    // Penguin.Fly() → throw? override to nothing?
// The hierarchy forces every subtype to fit the base's ASSUMPTIONS. When they don't,
// you get override-to-throw, empty overrides, and the fragile base class problem.

// COMPOSITION — build behavior from parts a type HAS, not a lineage it IS:
public class Bird
{
    private readonly IMovement _movement;          // HAS a movement strategy
    private readonly IFeeding _feeding;            // HAS a feeding strategy
    public Bird(IMovement movement, IFeeding feeding) => (_movement, _feeding) = (movement, feeding);
    public void Move() => _movement.Move();
}
new Bird(new Flying(), new Foraging());            // eagle
new Bird(new Swimming(), new Foraging());          // penguin — no Fly() to disable!
new Bird(new Stationary(), new NoFeeding());       // rubber duck — composes cleanly

// The guiding questions:
//   "is-a" AND the base's every behavior TRULY applies to the subtype?  → inheritance MAYBE ok
//   "has-a" / "can-do" / behavior varies independently?                 → COMPOSITION
//   need to swap behavior at runtime?                                   → COMPOSITION (strategy)
//   sharing implementation but NOT an identity relationship?            → COMPOSITION

This is the most important design judgment in the lesson, and the industry's hard-won default: prefer composition over inheritance. Inheritance couples a subtype to every assumption of its base — the moment a subtype doesn't fit (the penguin that can't fly, the rubber duck that can't eat), you get override-to-throw and empty overrides, symptoms of a wrong model. Composition builds behavior from interchangeable parts a type has, so variations combine freely and can even be swapped at runtime (the strategy pattern). The test is honest "is-a": inheritance is defensible only when the subtype genuinely is the base and every base behavior truly applies; the moment behavior varies independently, or you're inheriting just to reuse code, reach for composition. Deep inheritance hierarchies (three-plus levels) are a code smell precisely because each level multiplies these coupling assumptions.

Part 5 — abstract class vs interface: Choosing

ABSTRACT CLASS                              INTERFACE
  can have STATE (fields)                    no instance state (only const/static)
  can have constructors                      no constructors
  provides shared IMPLEMENTATION + state     contract (+ optional default methods)
  single base only (one abstract parent)     implement MANY
  models "IS-A a kind of thing"              models "CAN-DO a capability"
  members can be private/protected           members effectively public
  version-friendly (add non-abstract member) DIMs help, but historically brittle

// Use an ABSTRACT CLASS when subtypes share state AND implementation and form a real
// "is-a" family with a common identity:
public abstract class Shape
{
    public string Name { get; }                 // shared state
    protected Shape(string name) => Name = name;
    public abstract double Area();              // each shape computes its own
    public string Report() => $"{Name}: {Area():F2}";   // shared implementation
}

// Use an INTERFACE when you're describing a capability that unrelated types can have:
public interface ISerializable { string Serialize(); }
// a Shape, a User, and an Invoice can all be ISerializable without any shared lineage.

// COMBINE them — the common pattern: interface for the contract, optional abstract base
// for shared implementation:
public interface IRepository<T> { Task<T?> GetAsync(int id); Task SaveAsync(T entity); }
public abstract class RepositoryBase<T> : IRepository<T> { /* shared EF Core plumbing */ }

The decision reduces to two questions: does the abstraction need to carry state and shared implementation (abstract class), or is it a pure capability that unrelated types can possess (interface)? Abstract classes model an "is-a" family with a common identity and let you share fields and method bodies, but a type gets only one. Interfaces model "can-do" capabilities, a type can have many, and they don't presume any shared lineage — which is why the .NET convention leans heavily on interfaces for contracts. The frequent best-of-both pattern: define the contract as an interface, and optionally provide an abstract base class that implements the shared plumbing — consumers depend on the interface (maximally decoupled and testable), while implementers can inherit the base to avoid boilerplate. Default to interfaces for the contract; add an abstract base only when there's real shared implementation to hoist.

Part 6 — Deep Dive: Liskov, Fragile Base Classes & When Inheritance Earns Its Place

// LISKOV SUBSTITUTION PRINCIPLE — the correctness test for inheritance:
//   "a subtype must be usable ANYWHERE its base is expected, without surprises."

// LSP VIOLATION — the canonical example: Square is-a Rectangle... isn't:
public class Rectangle { public virtual int Width { get; set; } public virtual int Height { get; set; } }
public class Square : Rectangle
{
    public override int Width  { set { base.Width = base.Height = value; } }   // keeps them equal
    public override int Height { set { base.Width = base.Height = value; } }
}
void Stretch(Rectangle r) { r.Width = 5; r.Height = 4; Assert(r.Width * r.Height == 20); }
Stretch(new Square());     // ✗ FAILS: setting Height also changed Width → area is 16, not 20
// Square breaks code that works for Rectangle. The "is-a" was a lie in BEHAVIOR terms.

// THE FRAGILE BASE CLASS PROBLEM — base changes break subclasses invisibly:
//   a base method's internal refactor (e.g. Add() now calls AddRange()) can silently
//   break a subclass that overrode AddRange() expecting the old call pattern. Subclasses
//   depend on the base's INTERNAL behavior, not just its contract — coupling you can't see.

// WHEN INHERITANCE GENUINELY EARNS ITS PLACE:
//  - framework extension points designed for it (ControllerBase, Exception, EF entities)
//  - a true, stable is-a family where LSP holds and behavior doesn't vary independently
//  - the base is SEALED-by-intent or explicitly designed & documented for inheritance
//  - template method pattern: base defines the algorithm skeleton, subtypes fill steps
// Otherwise: compose. Mark classes 'sealed' by default so inheritance is a DELIBERATE
// decision, not an accident waiting to be abused.

Two principles make the composition-over-inheritance advice rigorous. The Liskov Substitution Principle is the correctness test: a subtype must be substitutable for its base in behavior, not just in shape — the Square/Rectangle example compiles perfectly yet breaks any code that assumes independent width and height, proving the "is-a" was false at the level that matters. The fragile base class problem is the maintenance cost: subclasses depend on the base's internal behavior, so a base's private refactoring can silently break subclasses that overrode the wrong seam — invisible coupling that composition (which depends only on public contracts) never creates. Inheritance still earns its place at framework extension points designed for it, in genuine stable families where LSP holds, and in template-method designs — but the default posture is sealed classes and composition, making inheritance a deliberate, justified choice rather than the reflexive first tool.

Senior framing: the inheritance conversation is where OOP maturity is most visible, because juniors reach for it and seniors reach past it. In review, deep hierarchies (3+ levels), overrides that throw NotSupportedException, empty overrides, and "extends X just to reuse a method" are all inheritance-abuse tells — each says the model should have been composition or an interface. The questions that signal seniority: "does this satisfy Liskov — can I use the subtype anywhere the base works without surprise?", "is this genuinely is-a, or is it has-a/can-do wearing is-a's clothes?", and "am I inheriting for polymorphism (sometimes right) or just to avoid duplication (almost always wrong — extract a helper or compose instead)?". The interview set-pieces are the Square/Rectangle LSP violation and the composition refactor of a misbehaving hierarchy; being able to name why the hierarchy is wrong (LSP, fragile base, independent behavioral variation) rather than just feeling it is the difference between pattern-matching and understanding. And the practical default — sealed everything, prefer interfaces for contracts, compose behavior — is what keeps a large codebase from calcifying into an unmaintainable inheritance lattice.

Interview Questions

Q1 (Beginner): Explain virtual, override, and abstract, and why C# requires the override keyword.

virtual marks a base method that has an implementation but subtypes may replace. abstract marks a method with no body that subtypes must implement (and forces the class itself to be abstract/non-instantiable). override is how a subtype provides the replacement. C# requires the explicit override keyword so overriding is always intentional — you cannot accidentally override a base method (as you can in languages where everything is virtual by default), and the compiler catches typos or signature mismatches (an override with no matching virtual/abstract base member is an error). This gives the base-class author control over their own extensibility surface.

Q2 (Beginner): What is polymorphism, and what does it let you do that you couldn't otherwise?

Polymorphism is treating objects of different concrete types uniformly through a shared base class or interface, where the actual method that runs is determined by the object's runtime type (dynamic dispatch), not the variable's compile-time type. Calling shape.Area() on a Shape-typed variable runs the specific shape's override. What it enables: writing code against the abstraction (a list of Shape, a payroll loop over Employee) that automatically works with new subtypes added later — no edits to the consuming code. That's the Open/Closed Principle: open for extension (new subtypes), closed for modification (existing code untouched).

Q3 (Mid): When would you choose an interface over an abstract class, and vice versa? Give the combined pattern.

Interface when you're describing a capability that potentially unrelated types can have (ISerializable, IComparable<T>), when you need a type to have multiple such contracts, or when maximal decoupling/testability matters — interfaces have no state, no constructors, and a type can implement many. Abstract class when there's genuine shared state and implementation among a true is-a family and you want to hoist it (protected fields, common method bodies, a base constructor) — but a type gets only one abstract base. The combined pattern, common in real code: define the contract as an interface (consumers depend on it, mock it, swap implementations) and optionally provide an abstract base class that implements it with shared plumbing (implementers inherit to skip boilerplate). Default to the interface for the contract; add the base only when there's real shared implementation.

Q4 (Mid): Explain "composition over inheritance" with the classic bird/duck example. What symptoms tell you a hierarchy is wrong?

Inheritance couples a subtype to all of its base's behavior. Model birds with inheritance and Bird.Fly() forces every subtype to fly — but a Penguin or RubberDuck can't, so you're stuck overriding Fly() to throw or do nothing (an LSP violation). Composition instead gives Bird a movement strategy it has (IMovement): an eagle gets Flying, a penguin Swimming, a rubber duck Stationary — variations combine freely, nothing is disabled, and behavior can even be swapped at runtime. The symptoms of a wrong hierarchy: overrides that throw NotSupportedException, empty overrides, subtypes that "are-a" the base only loosely, behavior that varies independently along multiple axes, and inheriting purely to reuse a method. Any of these means the relationship is has-a/can-do, not is-a — compose it.

Q5 (Senior): State the Liskov Substitution Principle and give a concrete violation that compiles but is wrong.

LSP: a subtype must be substitutable for its base type without altering the correctness of the program — anywhere the base works, the subtype must work with no surprising behavioral change. The canonical violation is Square inheriting from Rectangle: mathematically a square is-a rectangle, so it compiles, but if Square overrides its setters to keep width and height equal, then any code correct for Rectangle — say, "set width to 5, height to 4, expect area 20" — breaks for a Square (setting height also changed width, area is 16). The subtype violated a behavioral expectation the base established. Other real examples: a read-only collection subtype of a mutable collection (whose Add must throw), or a subclass that strengthens preconditions or weakens postconditions. The lesson: is-a in the domain doesn't guarantee is-a in behavior, and only the latter makes inheritance safe — which is why the Rectangle/Square pair is the standard argument for composition over inheritance.

Q6 (Senior): Describe the fragile base class problem and how composition avoids it. When is inheritance still the right tool?

The fragile base class problem: subclasses depend not just on a base's public contract but on its internal behavior — how its methods call each other. If a base refactors internally (e.g. Add now delegates to AddRange), a subclass that overrode AddRange expecting the old call graph can silently break or double-execute, even though the base's public contract is unchanged. This coupling is invisible — nothing in the subclass's code references the base's internals, yet it's tied to them. Composition avoids it because a composed object depends only on the public interface of the parts it holds; a part's internal refactoring can't reach it. Inheritance is still right when: the framework provides it as a designed extension point (deriving from ControllerBase, Exception, an EF entity base), there's a stable true is-a family where LSP holds and behavior doesn't vary independently, or you're implementing the template-method pattern (base owns the algorithm skeleton, subtypes fill designated abstract steps — a controlled, documented extension surface). The safeguard: seal classes by default so any inheritance is a deliberate, reviewed decision rather than an accident.

1 (Easy): Create an abstract Shape with an abstract Area() and a concrete Describe(), plus Circle and Rectangle subclasses. Show polymorphic use over a List<Shape>.
Solution
public abstract class Shape(string name)
{
    public string Name => name;
    public abstract double Area();
    public string Describe() => $"{Name} with area {Area():F2}";  // shared impl
}
public sealed class Circle(double radius) : Shape("Circle")
{
    public override double Area() => Math.PI * radius * radius;
}
public sealed class Rectangle(double w, double h) : Shape("Rectangle")
{
    public override double Area() => w * h;
}

List<Shape> shapes = [new Circle(2), new Rectangle(3, 4)];
foreach (var s in shapes)
    Console.WriteLine(s.Describe());        // each runs ITS Area() via dynamic dispatch
Console.WriteLine($"Total: {shapes.Sum(s => s.Area()):F2}");
// Adding a Triangle subtype needs ZERO changes here — the Open/Closed Principle.
2 (Easy): Define an IExporter interface (Format + Export) with CSV and JSON implementations, and a method that uses any exporter without knowing which. Why is this better than a base class?
Solution
public interface IExporter
{
    string Format { get; }
    string Export(IEnumerable<string> rows);
}
public class CsvExporter : IExporter
{
    public string Format => "csv";
    public string Export(IEnumerable<string> rows) => string.Join("\n", rows);
}
public class JsonExporter : IExporter
{
    public string Format => "json";
    public string Export(IEnumerable<string> rows) =>
        "[" + string.Join(",", rows.Select(r => $"\"{r}\"")) + "]";
}

void Save(IExporter exporter, IEnumerable<string> rows) =>
    File.WriteAllText($"out.{exporter.Format}", exporter.Export(rows));
// Interface (not base class) because: exporters share NO state or implementation — only
// a contract; a type could be an IExporter AND something else; and Save() can be tested
// with a fake IExporter. There's no is-a family here, just a capability.
3 (Medium): Refactor this misbehaving hierarchy using composition: 
public class Vehicle { public virtual void Refuel() {...} public virtual void Fly() {...} }
public class Car : Vehicle { public override void Fly() => throw new NotSupportedException(); }
public class ElectricCar : Car { public override void Refuel() => throw new NotSupportedException(); }
Solution
// The smells: Car can't Fly (override-to-throw), ElectricCar can't Refuel (another throw).
// Behavior varies along independent axes — that's composition, not a lineage.

public interface IPropulsion { void Move(); }
public interface IEnergySource { void Replenish(); }

public class Combustion : IEnergySource { public void Replenish() => Console.WriteLine("Refuel"); }
public class Battery    : IEnergySource { public void Replenish() => Console.WriteLine("Recharge"); }
public class RoadDrive  : IPropulsion  { public void Move() => Console.WriteLine("Drive"); }
public class AirDrive   : IPropulsion  { public void Move() => Console.WriteLine("Fly"); }

public class Vehicle(IPropulsion propulsion, IEnergySource energy)
{
    public void Move()      => propulsion.Move();
    public void Replenish() => energy.Replenish();
}

var car         = new Vehicle(new RoadDrive(), new Combustion());
var electricCar = new Vehicle(new RoadDrive(), new Battery());       // no Refuel to disable
var plane       = new Vehicle(new AirDrive(),  new Combustion());
// Every combination is valid by construction; nothing throws NotSupported; new energy
// sources or propulsion types compose without touching Vehicle. The two independent axes
// (how it moves, how it's energized) are now independent PARTS, not a fragile hierarchy.
4 (Medium): Implement IComparable<T> on a Version-like type so a List<SemVer> sorts correctly, and explain what the framework does with your implementation.
Solution
public class SemVer(int major, int minor, int patch) : IComparable<SemVer>
{
    public int Major => major;
    public int Minor => minor;
    public int Patch => patch;

    public int CompareTo(SemVer? other)          // <0 = before, 0 = equal, >0 = after
    {
        if (other is null) return 1;
        int m = Major.CompareTo(other.Major);   if (m != 0) return m;
        int n = Minor.CompareTo(other.Minor);   if (n != 0) return n;
        return Patch.CompareTo(other.Patch);
    }
    public override string ToString() => $"{Major}.{Minor}.{Patch}";
}

var versions = new List<SemVer> { new(1,2,0), new(1,0,5), new(2,0,0), new(1,2,3) };
versions.Sort();                                // uses YOUR CompareTo — no comparer needed
// → 1.0.5, 1.2.0, 1.2.3, 2.0.0
// What the framework does: List.Sort(), Array.Sort(), OrderBy(), SortedSet, and binary
// search all look for IComparable<T> to determine natural order. By implementing one
// method you plug into the entire BCL sorting/ordering machinery — the interface contract
// is the integration point.
5 (Hard): Demonstrate the Square/Rectangle LSP violation in code — write a test that passes for Rectangle and fails for Square — then show the composition-based fix that avoids the trap entirely.
Solution
// THE VIOLATION:
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
    public int Area => Width * Height;
}
public class Square : Rectangle
{
    public override int Width  { set { base.Width = base.Height = value; } get => base.Width; }
    public override int Height { set { base.Width = base.Height = value; } get => base.Height; }
}

bool StretchAndCheck(Rectangle r)
{
    r.Width = 5;
    r.Height = 4;
    return r.Area == 20;                         // any true Rectangle satisfies this
}
Console.WriteLine(StretchAndCheck(new Rectangle()));  // True
Console.WriteLine(StretchAndCheck(new Square()));     // False! Height=4 also set Width=4 → 16
// Square is NOT substitutable for Rectangle: it breaks code correct for the base. LSP fails.

// THE FIX — don't relate them by inheritance at all; model shapes by capability:
public interface IShape { int Area { get; } }
public sealed class Rect(int w, int h) : IShape { public int Area => w * h; }
public sealed class Sq(int side)       : IShape { public int Area => side * side; }
// A Square and a Rectangle are both IShapes (both HAVE an area) but neither IS the other.
// No mutable dimensions to violate, no substitutability trap — the "is-a" that was false
// in behavior was never asserted. Composition/interface modeling sidesteps LSP by not
// making the invalid inheritance claim in the first place.
6 (Hard): Design a payment-processing abstraction: a contract (interface), a shared base for common logic (retry/logging), and two concrete processors. Justify each choice of interface vs abstract class vs composition.
Solution
// 1. CONTRACT as an interface — consumers depend on this, mock it, swap providers:
public interface IPaymentProcessor
{
    Task<PaymentResult> ChargeAsync(decimal amount, string token);
}

// 2. Cross-cutting concerns (retry, logging) as COMPOSITION — a DECORATOR, not a base:
public class RetryingPaymentProcessor(IPaymentProcessor inner, ILogger log) : IPaymentProcessor
{
    public async Task<PaymentResult> ChargeAsync(decimal amount, string token)
    {
        for (int attempt = 1; attempt <= 3; attempt++)
        {
            try { return await inner.ChargeAsync(amount, token); }
            catch (TransientException) when (attempt < 3)
            { log.LogWarning("Retry {Attempt}", attempt); await Task.Delay(attempt * 200); }
        }
        throw new PaymentException("Exhausted retries");
    }
}

// 3. Optional ABSTRACT BASE only for genuine shared IMPLEMENTATION among real siblings:
public abstract class HttpPaymentProcessorBase(HttpClient http) : IPaymentProcessor
{
    protected abstract string Endpoint { get; }                 // subclass supplies
    public async Task<PaymentResult> ChargeAsync(decimal amount, string token)
    {
        var response = await http.PostAsJsonAsync(Endpoint, new { amount, token });
        return await ParseResult(response);                     // shared plumbing
    }
    protected abstract Task<PaymentResult> ParseResult(HttpResponseMessage r);  // varies
}

public sealed class StripeProcessor(HttpClient http) : HttpPaymentProcessorBase(http)
{
    protected override string Endpoint => "https://api.stripe.com/charges";
    protected override Task<PaymentResult> ParseResult(HttpResponseMessage r) => /* ... */;
}
public sealed class AdyenProcessor(HttpClient http) : HttpPaymentProcessorBase(http)
{
    protected override string Endpoint => "https://checkout.adyen.com/payments";
    protected override Task<PaymentResult> ParseResult(HttpResponseMessage r) => /* ... */;
}

// JUSTIFICATION:
//  - INTERFACE for IPaymentProcessor: it's a capability contract; consumers must not know
//    or care which provider; it's the DI + testing seam. Never a base class here.
//  - COMPOSITION (decorator) for retry/logging: these are cross-cutting behaviors that
//    should wrap ANY processor, stack independently, and vary at runtime — a base class
//    would force every processor to inherit them and couple them together. Composition
//    lets you add/remove/reorder them: new RetryingPaymentProcessor(new StripeProcessor()).
//  - ABSTRACT BASE for HTTP processors: Stripe and Adyen genuinely SHARE implementation
//    (the POST-and-parse skeleton) and form a real is-a family (both ARE http processors),
//    with only Endpoint + ParseResult varying — the template-method pattern, a legitimate
//    inheritance use. Sealed concrete classes stop the hierarchy at two levels.
// The through-line: interface for the contract, composition for cross-cutting behavior,
// abstract base ONLY where real shared implementation and a true is-a family coexist.

🎯 Key Takeaways

  • C# inheritance is explicit and opt-in: virtual/abstract mark overridable members, override is required (no accidental overrides), sealed ends a hierarchy, and new-hiding is a bug that breaks polymorphism.
  • Polymorphism (dynamic dispatch on the runtime type) is inheritance's purpose: code written against an abstraction works unchanged with new subtypes — the Open/Closed Principle.
  • Interfaces are capability contracts (no state, implement many) that decouple code from concretions — the foundation of DI, testing, and the BCL's IEnumerable/IComparable/IDisposable lattice.
  • Prefer composition over inheritance: build behavior from interchangeable parts a type has; inheritance couples a subtype to every base assumption, and override-to-throw/empty-override are the smells of a wrong hierarchy.
  • Abstract class (shared state + implementation, one base, is-a family) vs interface (pure capability, many, can-do) — commonly combined: interface for the contract, optional abstract base for shared plumbing.
  • LSP (behavioral substitutability, not just is-a) and the fragile base class problem make composition the safer default; seal by default and inherit only at designed extension points where the relationship truly holds.
