Generics Deep Dive Object-Oriented C#

What you'll master: why generics exist (type safety + performance without duplication), generic classes and methods, constraints (the vocabulary that makes generics useful), reified generics and why .NET's are fundamentally better than Java's, variance (in/out), and where generics genuinely help vs over-abstract. Prerequisites: Types (boxing), Inheritance (interfaces), Equality (IEquatable). Generics underpin the entire collections and LINQ story that follows.

Part 1 — The Problem Generics Solve

// WITHOUT generics — the two bad options before C# 2.0:

// Option A: type-specific duplication (a Stack per type — copy/paste hell):
public class IntStack  { public void Push(int x) {...}  public int Pop() {...} }
public class StringStack { public void Push(string x) {...} public string Pop() {...} }
// ...and one for every type you ever store. Insane.

// Option B: object-based (one class, but UNSAFE and SLOW):
public class ObjectStack
{
    public void Push(object x) {...}
    public object Pop() {...}
}
var s = new ObjectStack();
s.Push(42);                     // boxes the int (types.html) — heap allocation
s.Push("oops");                 // compiles! no type safety — any object goes in
int x = (int)s.Pop();           // "oops" → InvalidCastException at RUNTIME. cast every read.

// WITH generics — one definition, type-safe, no boxing:
public class Stack<T>            // T is a TYPE PARAMETER, filled in at use
{
    private T[] _items = new T[16];
    public void Push(T item) {...}
    public T Pop() {...}
}
var s = new Stack<int>();        // T = int; stores raw ints, NO boxing
s.Push(42);                     // ✓ type-safe
// s.Push("oops");              // ✗ COMPILE ERROR — caught at build, not runtime
int x = s.Pop();                // ✓ no cast — T IS int here

Generics solve a real dilemma that predates them: before C# 2.0, a reusable container had to be either duplicated per type (a maintenance nightmare) or object-based (which sacrifices both type safety and performance — every value type boxes on the way in, every read needs a cast, and putting the wrong type in fails only at runtime). Generics give you one definition parameterized by type — Stack<T> — that is type-safe (the compiler rejects Push("oops") on a Stack<int> at build time), allocation-free for value types (an int stored in a Stack<int> never boxes), and cast-free on read (Pop() returns a real int). This is why the entire modern collections library (List<T>, Dictionary<K,V>) is generic, and why the old non-generic ArrayList/Hashtable are effectively deprecated.

Part 2 — Generic Methods & Type Inference

// A generic METHOD parameterizes just the method, not the whole class:
public static T Max<T>(T a, T b) where T : IComparable<T>   // constraint — Part 3
    => a.CompareTo(b) >= 0 ? a : b;

Max<int>(3, 7);         // explicit type argument
Max(3, 7);              // INFERRED — the compiler figures out T = int from the args
Max("apple", "pear");   // T = string (string is IComparable<string>)

// Multiple type parameters — the classic mapping method:
public static TResult[] Map<TSource, TResult>(
    TSource[] source, Func<TSource, TResult> selector)      // delegate — delegates-events.html
{
    var result = new TResult[source.Length];
    for (int i = 0; i < source.Length; i++)
        result[i] = selector(source[i]);
    return result;
}
int[] lengths = Map(new[] { "a", "bb", "ccc" }, s => s.Length);   // TSource=string, TResult=int
//               inference reads BOTH type params from usage — no explicit args needed

// Inference limitation: it works from ARGUMENTS, not from the RETURN type alone:
T Parse<T>(string s) { ... }
// Parse("42");           // ✗ can't infer T — nothing in the args says what T is
int n = Parse<int>("42");  // must specify explicitly

Generic methods parameterize a single method rather than an entire type — useful for utilities (Max, Swap, Map) that should work across types without a generic class wrapper. The ergonomic win is type inference: the compiler deduces the type arguments from the actual argument types, so you write Max(3, 7) not Max<int>(3, 7), and inference reads multiple type parameters at once (Map infers both TSource and TResult from the array and the selector). The one limitation worth internalizing: inference works from arguments, never from the return type in isolation — a T Parse<T>(string) can't infer T because nothing in Parse("42") indicates whether you want an int or a DateTime, so you must specify it. This is the entire mechanism behind LINQ's fluent style, where inference makes long chains read cleanly.

Part 3 — Constraints: Making Type Parameters Useful

// Unconstrained T can do almost NOTHING — you can't call methods, compare, or construct it:
T Broken<T>(T a, T b) => a.CompareTo(b);   // ✗ compile error: T has no CompareTo

// CONSTRAINTS declare what T must support, unlocking those capabilities:
where T : IComparable<T>      // T implements the interface → can call CompareTo
where T : class               // T is a reference type → can be null, use 'as'
where T : struct              // T is a value type → no null, use in perf contexts
where T : new()               // T has a parameterless constructor → can 'new T()'
where T : Employee            // T derives from Employee → can access Employee members
where T : notnull             // T can't be null (nullable-aware, exceptions-nullability.html)
where T : Enum                // T is an enum
where T : IComparable<T>, new()   // COMBINE — T is comparable AND constructible

// Constraints in action — a generic repository, made possible by constraints:
public class Repository<T> where T : class, IEntity, new()
{
    public T Create() => new T();               // 'new()' allows this
    public int GetId(T entity) => entity.Id;    // IEntity constraint exposes .Id
}

// GENERIC MATH (C# 11, modern & powerful) — constrain to operator interfaces:
public static T Sum<T>(IEnumerable<T> values) where T : INumber<T>
{
    T total = T.Zero;                           // static abstract members on the interface!
    foreach (var v in values) total += v;       // + works because T : INumber<T>
    return total;
}
Sum(new[] { 1, 2, 3 });        // int
Sum(new[] { 1.5, 2.5 });       // double — ONE method, any numeric type

Constraints are what turn generics from "a box that holds anything" into "a box that can do things with what it holds." An unconstrained T is nearly useless — the compiler knows only that it's some type, so you can't compare it, call methods on it, or construct it. A where clause declares the capabilities T must have (IComparable<T> to compare, new() to construct, a base class to access its members, struct/class for the value/reference distinction), and the compiler then permits exactly those operations while still guaranteeing type safety at every call site. The modern showcase is generic math: the INumber<T> interface (with static abstract members like T.Zero and operator support) lets you write one Sum that works for int, double, decimal, and any numeric type — a capability that required ugly workarounds or per-type duplication before C# 11.

Part 4 — Reified Generics: Why .NET's Beat Java's

// .NET generics are REIFIED — the type argument EXISTS at runtime, not erased:
var list = new List<int>();
Console.WriteLine(list.GetType());        // System.Collections.Generic.List`1[System.Int32]
                                          // the runtime KNOWS it's a List of int
typeof(List<int>) == typeof(List<string>) // FALSE — genuinely distinct runtime types

// The CONSEQUENCES that make reified generics a real advantage:

// 1. NO BOXING of value types (types.html): List<int> stores raw ints, contiguously:
var nums = new List<int> { 1, 2, 3 };     // int-sized slots, cache-friendly, zero boxing
// (Java's List<Integer> boxes every element into a heap object — a real perf gap.)

// 2. RUNTIME TYPE INSPECTION works — reflection/DI/serialization see the real type:
void Handle<T>() { if (typeof(T) == typeof(string)) {...} }   // works — T is known at runtime
// DI containers resolve IRepository<Order> distinctly from IRepository<Customer> because
// they're distinct runtime types (di-hosting.html relies on this).

// 3. The JIT SPECIALIZES value-type instantiations: List<int> gets its own optimized
//    machine code with int-sized storage; reference-type instantiations (List<string>,
//    List<Order>) SHARE one implementation (all references are pointer-sized) — the best
//    of both: value types get specialized speed, reference types share code size.

// Java's ERASURE, by contrast:
//   List<String> and List<Integer> are the SAME type at runtime (both just 'List')
//   → the type argument is a COMPILE-TIME fiction, erased to Object
//   → value types (int) MUST box into Integer (perf cost)
//   → you can't do 'new T()' or 'T.class' or typeof(T) — no runtime type info
//   → unchecked-cast warnings, and generic arrays are effectively forbidden

This is a genuine, articulable advantage of .NET over the JVM, and a favorite interview differentiator. .NET generics are reified: the type argument is real at runtime, so List<int> is a distinct runtime type from List<string>, the runtime knows what a collection holds, and reflection, DI, and serialization all see the concrete instantiation. The payoffs are concrete: value types are stored inline with no boxing (List<int> is contiguous int-sized slots — cache-friendly and allocation-free, where Java's List<Integer> boxes every element), you can do typeof(T) and new T() (impossible under erasure), and the JIT specializes value-type instantiations for speed while sharing one implementation across all reference-type instantiations (since all references are pointer-sized) — an elegant code-size/speed balance. Java erases type arguments to Object at compile time, making them a static-analysis fiction that boxes value types and prevents runtime type reasoning. Understanding why reification matters — not just that it exists — is the mark of someone who understands the platform.

Part 5 — Variance: in and out

// The question variance answers: if Dog : Animal, is IEnumerable<Dog> usable as
// IEnumerable<Animal>? Sometimes yes, sometimes no — variance says which.

// COVARIANCE (out) — a producer of Dogs is a producer of Animals (T only comes OUT):
IEnumerable<Dog> dogs = GetDogs();
IEnumerable<Animal> animals = dogs;      // ✓ LEGAL — IEnumerable<out T> is covariant
// safe because you can only READ T out; every Dog read IS-A Animal.
// interface IEnumerable<out T>  — the 'out' declares covariance.

// CONTRAVARIANCE (in) — a consumer of Animals is a consumer of Dogs (T only goes IN):
IComparer<Animal> animalComparer = GetAnimalComparer();
IComparer<Dog> dogComparer = animalComparer;   // ✓ LEGAL — IComparer<in T> is contravariant
// safe because it only ACCEPTS T; anything that can compare any Animal can compare Dogs.
// interface IComparer<in T>  — the 'in' declares contravariance.

// INVARIANCE — the default; T goes both in AND out, so neither conversion is safe:
List<Dog> dogList = new();
// List<Animal> animals = dogList;       // ✗ ILLEGAL — List<T> is invariant
// If it were allowed: animals.Add(new Cat()) would put a Cat in a List<Dog>. Unsafe.

// The memory hook:
//   'out T'  → T only OUTPUT   → covariant   (producers: IEnumerable, IReadOnlyList, Func<out R>)
//   'in T'   → T only INPUT    → contravariant (consumers: IComparer, IEqualityComparer, Action<in T>)
//   both     → invariant       (mutable collections: List, IList, Dictionary)

Variance answers when a generic type with a more-derived argument can substitute for one with a less-derived argument — and the rule follows from data flow direction. Covariance (out) applies when T only comes out: an IEnumerable<Dog> is safely an IEnumerable<Animal> because every Dog you read is an Animal — reading is safe. Contravariance (in) applies when T only goes in: an IComparer<Animal> is safely an IComparer<Dog> because anything able to compare any two animals can certainly compare two dogs — accepting is safe. Mutable collections like List<T> are invariant (neither, and forbidden) precisely because T flows both ways: if List<Dog> were usable as List<Animal>, you could Add(new Cat()) and corrupt the list. You mostly consume variance (it's why you can pass an IEnumerable<Dog> where IEnumerable<Animal> is expected without a cast), but understanding the in/out mnemonic and why mutable collections can't be variant is a common senior-level question.

Part 6 — Deep Dive: When Generics Help vs Over-Abstract

// GENERICS EARN THEIR PLACE when the logic is GENUINELY type-independent:
public class Cache<TKey, TValue> where TKey : notnull      // caching logic is identical
{                                                          // regardless of what you cache
    private readonly Dictionary<TKey, TValue> _store = new();
    public TValue GetOrAdd(TKey key, Func<TKey, TValue> factory) { ... }
}
// ✓ real reuse: one correct, tested implementation serves every key/value combination.
// The collections (List, Dictionary), LINQ, Task<T>, and Result<T> are all like this.

// GENERICS OVER-ABSTRACT when you add type parameters for FLEXIBILITY YOU DON'T NEED:
public interface IRepository<TEntity, TKey, TContext, TSpec, TMapper>   // ✗ 5 type params!
    where TEntity : class where TKey : struct where TContext : DbContext { ... }
// nobody can read a call site; the abstraction models possibilities no caller uses.
// Speculative generality — the generic version of premature optimization.

// The judgment questions:
//   Is the logic TRULY the same across the types? (yes → generic; no → separate types)
//   Do you have 2+ ACTUAL uses, or 1 use and imagined futures? (YAGNI — wait for the 2nd)
//   Can a reader understand the type parameters at the CALL site? (if not, too many)
//   Would a non-generic version + a small adapter be clearer? (often yes)

// A GOOD test: generic code should make CALL SITES simpler and safer than the
// alternatives. If the generic signature is harder to read than the duplication it
// replaces, the abstraction is net-negative.

// The Result<T> pattern — generics done RIGHT for error handling (exceptions-nullability):
public readonly record struct Result<T>(bool IsSuccess, T? Value, string? Error)
{
    public static Result<T> Ok(T value) => new(true, value, null);
    public static Result<T> Fail(string error) => new(false, default, error);
}
// one type expresses "success-with-a-T or failure" for ANY T — genuine, reused abstraction.

The final skill is judgment: generics are powerful, and power invites overuse. They earn their place when the logic is genuinely type-independent — a cache, a collection, a Result<T>, a Task<T> — where one correct implementation serves every type parameter and the reuse is real. They over-abstract when you add type parameters for imagined flexibility: a repository interface with five type parameters models possibilities no caller uses, makes every call site unreadable, and is the generic form of premature optimization (speculative generality). The honest tests are YAGNI-flavored — do you have two actual uses or one use and a fantasy of futures? can a reader understand the type parameters at the call site? — and the decisive one is that good generic code makes call sites simpler and safer than the duplication it replaces; if the generic signature is harder to read than the code it abstracts away, the abstraction is net-negative and should be a concrete type (or a concrete type plus a small adapter) instead.

Senior framing: generics questions separate three tiers. Tier one uses List<T> and can write a generic method. Tier two understands constraints (why unconstrained T is nearly useless, how where unlocks capability) and can explain reification — that .NET's runtime knows List<int> from List<string>, why that means no boxing of value types, and why Java's erasure is a genuine limitation (this comparison is a common differentiator, and being able to state the boxing consequence precisely reads as real understanding). Tier three adds variance (the in/out data-flow rule and why mutable collections must be invariant) and, crucially, judgment — recognizing that five-type- parameter interfaces are a smell, that speculative generality is real, and that the test of a good abstraction is whether it simplifies call sites. In review, the tells of generics-gone-wrong are excessive type parameters, generic base classes that exist only to share a method (compose instead), and constraints so elaborate that only one type could ever satisfy them (why is it generic at all?). The modern additions worth knowing — generic math (INumber<T>, static abstract interface members) and the Result<T>/discriminated-union patterns — show the feature is still evolving toward more expressive, safer abstractions, and they're where current C# conversations increasingly go.

Interview Questions

Q1 (Beginner): What problem do generics solve, and what were the alternatives before them?

Generics let you write one type-safe, allocation-efficient definition parameterized by type, instead of the two bad pre-generics options: duplicating a class per type (a Stack<int>, a Stack<string>, ad infinitum — unmaintainable), or using an object-based container (one class, but it boxes value types, requires a cast on every read, and lets you store the wrong type with the error only surfacing as a runtime InvalidCastException). A Stack<T> gives compile-time type safety (wrong-type pushes don't compile), no boxing (a Stack<int> stores raw ints), and no casts on read. This is why the entire modern collections library is generic and ArrayList/Hashtable are effectively retired.

Q2 (Beginner): What are generic constraints, and why can't you do much with an unconstrained T?

An unconstrained T tells the compiler only "some type," so it permits almost nothing — you can't compare two Ts, call methods beyond object's, or write new T(), because the compiler can't prove T supports them. Constraints (where T : ...) declare required capabilities: IComparable<T> (can compare), class/struct (reference or value type), new() (constructible), a base type (access its members), notnull, INumber<T> (arithmetic). Once constrained, the compiler permits exactly those operations while still guaranteeing type safety at every call site — so constraints are what make a generic actually do something with its type parameter rather than just hold it.

Q3 (Mid): Explain reified generics and give three concrete consequences of .NET having them.

Reified means the type argument is preserved at runtime, not erased: List<int> is a genuinely distinct runtime type from List<string>, and the runtime knows what a generic instance holds. Three consequences: (1) No boxing of value types — a List<int> stores ints inline in int-sized, contiguous, cache-friendly slots, whereas Java's erasure forces List<Integer> to box every element into a heap object. (2) Runtime type reasoning works — typeof(T), new T(), and pattern-matching on generic type arguments all function, which is why DI containers can distinguish IRepository<Order> from IRepository<Customer> and serializers can inspect the real element type. (3) JIT specialization — value-type instantiations get their own optimized machine code with correctly-sized storage, while reference-type instantiations share a single implementation (all references being pointer-sized), balancing speed against code size. Together these make .NET generics both faster and more capable than erasure-based ones.

Q4 (Mid): What is covariance vs contravariance, and why is List<T> invariant?

Covariance (out T) lets a generic with a more-derived argument substitute for one with a less-derived argument when T only comes out: an IEnumerable<Dog> is usable as IEnumerable<Animal> because every dog you read is an animal — read-only is safe. Contravariance (in T) is the reverse for inputs: an IComparer<Animal> is usable as IComparer<Dog> because anything that compares any animals can compare dogs — accepting is safe. List<T> is invariant (neither substitution allowed) because T flows both in and out: if a List<Dog> were usable as a List<Animal>, you could call Add(new Cat()) through the Animal view and put a cat into a list of dogs, breaking type safety on the next read. Variance is only sound when data flows one direction, which is why covariant/contravariant interfaces are read-only or write-only respectively, and mutable collections that do both must be invariant.

Q5 (Senior): When do generics genuinely help, and when do they over-abstract? Give the tests you apply.

Generics help when the logic is genuinely type-independent — a cache, a collection, a Result<T>, a Task<T> — where one correct, tested implementation legitimately serves every type argument, and where the generic version makes call sites simpler and safer than duplication or object-based code. They over-abstract when you add type parameters for flexibility no caller actually uses — the five-type-parameter repository interface is the archetype: unreadable call sites, and modeling of possibilities that exist only in the author's imagination (speculative generality, the generic cousin of premature optimization). My tests: (1) is the logic truly the same across types, or am I forcing unrelated cases into one shape? (2) Do I have two-plus real uses, or one use and hypothetical futures (YAGNI — wait for the second)? (3) Can a reader understand the type parameters at the call site? (4) Is the generic signature actually clearer than the concrete alternative? The decisive heuristic: a good abstraction reduces the cognitive load at the point of use; if the generic is harder to read than what it replaces, make it concrete (optionally with a small adapter) and revisit only when a genuine second use appears.

Q6 (Senior): Design a generic Result<T> for error handling. What constraints, variance, and API shape, and how does it compare to exceptions?

Shape: a readonly record struct Result<T> (value type — no allocation per result; record — value equality and a clean API) holding a success flag and either a T value or an error (a string, an error enum, or a richer error type). Factory methods Result<T>.Ok(value) and Result<T>.Fail(error) keep construction intent-revealing, and a Match(onSuccess, onFailure) or pattern-matchable shape forces callers to handle both cases. Constraints: usually none needed on T (it's just carried), though notnull can prevent a null success value. Variance: as a struct it can't be a variant interface, but if you expose it via an interface you'd make it covariant in T (out T) since the value only comes out — letting a Result<Dog> be read as a Result<Animal>. Versus exceptions: Result<T> makes failure part of the type signature (the caller can't ignore it — no silent unhandled exception), is far cheaper for expected failures (validation, not-found, parse failures) since exceptions are costly to throw and unwind, and reads as explicit control flow. Exceptions remain right for genuinely exceptional, unexpected conditions (programming errors, unrecoverable I/O) where you want the stack unwound to a handler. The mature stance (exceptions-nullability lesson) is to use both deliberately — Result<T> for expected domain failures in your signatures, exceptions for the exceptional — and Result<T> is a textbook example of generics providing a genuine, reusable abstraction rather than speculative flexibility.

🏋️ Exercises

1 (Easy): Write a generic Swap<T> method and a generic Middle<T> that returns the middle element of an array. Show type inference at the call sites.
Solution
static void Swap<T>(ref T a, ref T b) => (a, b) = (b, a);   // no constraint needed — just moves
static T Middle<T>(T[] items) => items[items.Length / 2]; // also no constraint — just indexes

int x = 1, y = 2;
Swap(ref x, ref y);                    // T inferred as int
Console.WriteLine($"{x} {y}");         // 2 1

var mid = Middle(new[] { "a", "b", "c", "d", "e" });   // T inferred as string
Console.WriteLine(mid);                // "c"
// Neither needs a constraint because neither does anything with T beyond moving/indexing —
// a good reminder that constraints are only for capabilities you actually USE.
2 (Easy): Write a generic Max<T> that works for any comparable type. Explain why the constraint is required.
Solution
static T Max<T>(T a, T b) where T : IComparable<T>
    => a.CompareTo(b) >= 0 ? a : b;

Console.WriteLine(Max(3, 7));            // 7
Console.WriteLine(Max("apple", "pear")); // "pear" (string is IComparable<string>)
Console.WriteLine(Max(2.5, 1.5));        // 2.5

// Why the constraint: without 'where T : IComparable<T>', the compiler knows only that
// T is "some type" and has no idea it supports CompareTo — 'a.CompareTo(b)' would be a
// compile error. The constraint is a PROMISE ("T can compare itself") that unlocks the
// method call while the compiler still enforces it at every call site (Max(anObject, ...)
// where the type isn't IComparable won't compile).
3 (Medium): Build a generic Cache<TKey, TValue> with a GetOrAdd(key, factory) method, appropriately constrained. Show it caching two different value types.
Solution
public class Cache<TKey, TValue> where TKey : notnull    // dictionary keys can't be null
{
    private readonly Dictionary<TKey, TValue> _store = new();

    public TValue GetOrAdd(TKey key, Func<TKey, TValue> factory)
    {
        if (_store.TryGetValue(key, out var existing))   // hit — return cached
            return existing;
        var created = factory(key);                      // miss — compute, store, return
        _store[key] = created;
        return created;
    }
    public int Count => _store.Count;
}

// Same caching logic, two completely different type instantiations:
var userCache = new Cache<int, string>();
var name = userCache.GetOrAdd(42, id => $"User#{id}");           // computes once
var again = userCache.GetOrAdd(42, id => throw new("won't run")); // cached — factory skipped

var configCache = new Cache<string, Config>();
var cfg = configCache.GetOrAdd("db", key => LoadConfig(key));

// This is generics EARNING their place: the GetOrAdd logic is genuinely identical across
// every key/value combination, so one tested implementation serves all of them. The
// 'notnull' constraint is the only capability the logic actually requires of TKey.
4 (Medium): Demonstrate covariance and contravariance with a small Animal/Dog hierarchy — one legal covariant assignment, one legal contravariant assignment, and one illegal invariant one, each with a one-line reason.
Solution
public class Animal { }
public class Dog : Animal { }

// COVARIANCE (out) — producer of Dogs works as producer of Animals:
IEnumerable<Dog> dogs = new List<Dog> { new(), new() };
IEnumerable<Animal> animals = dogs;              // ✓ LEGAL
//  reason: IEnumerable<out T> only OUTPUTS T; every Dog read is an Animal → safe.

// CONTRAVARIANCE (in) — consumer of Animals works as consumer of Dogs:
IComparer<Animal> byAge = Comparer<Animal>.Create((a, b) => 0);
IComparer<Dog> dogComparer = byAge;             // ✓ LEGAL
//  reason: IComparer<in T> only ACCEPTS T; something that compares any Animals compares Dogs.

// INVARIANCE — mutable collection, neither conversion allowed:
List<Dog> dogList = new();
// List<Animal> asAnimals = dogList;            // ✗ ILLEGAL — compile error
//  reason: List<T> is invariant (T flows in AND out); allowing it would let
//          asAnimals.Add(new Cat()) corrupt a List<Dog>.
// The mnemonic in one breath: out=covariant (read-only), in=contravariant (write-only),
// both=invariant (mutable).
5 (Hard): Implement Result<T> as a value-type error-handling abstraction with Ok/Fail factories and a Match method, then use it in a Divide function that fails gracefully on divide-by-zero.
Solution
public readonly record struct Result<T>
{
    public bool IsSuccess { get; }
    private readonly T? _value;
    public string? Error { get; }

    private Result(bool ok, T? value, string? error) => (IsSuccess, _value, Error) = (ok, value, error);
    public static Result<T> Ok(T value) => new(true, value, null);
    public static Result<T> Fail(string error) => new(false, default, error);

    // Match FORCES the caller to handle both cases — no ignoring failure:
    public TOut Match<TOut>(Func<T, TOut> onSuccess, Func<string, TOut> onFailure) =>
        IsSuccess ? onSuccess(_value!) : onFailure(Error!);
}

static Result<double> Divide(double a, double b) =>
    b == 0 ? Result<double>.Fail("Division by zero")
           : Result<double>.Ok(a / b);

// The caller CANNOT forget the error case — it's in the type:
var message = Divide(10, 2).Match(
    onSuccess: result => $"= {result}",
    onFailure: error   => $"Error: {error}");
Console.WriteLine(message);                       // "= 5"
Console.WriteLine(Divide(10, 0).Match(r => $"= {r}", e => $"Error: {e}"));  // "Error: Division by zero"

// Why this is generics done right: ONE type expresses "a T or an error" for ANY T
// (double here, but equally User, Order, List<X>), the abstraction is genuinely reused,
// it's a value type (no allocation), and Match makes failure impossible to ignore —
// unlike an exception that can silently propagate unhandled. Expected failures become
// part of the signature (exceptions-nullability.html) instead of hidden control flow.
6 (Hard): Critique and redesign this over-generic interface. Explain what each type parameter costs and what the right shape is. 
public interface IProcessor<TInput, TOutput, TContext, TValidator, TLogger, TResult>
    where TInput : class where TOutput : class where TContext : DbContext
    where TValidator : IValidator<TInput> where TLogger : ILogger
{
    TResult Process(TInput input, TContext ctx);
}
Solution
// CRITIQUE — six type parameters, and most are wrong:
//  TContext   : a DbContext type parameter → the processor is coupled to persistence AND
//               forces every caller to thread a context type. Should be INJECTED, not generic.
//  TValidator : a validator TYPE parameter → you don't need the validator's type, you need
//               a validator INSTANCE. Inject IValidator<TInput>, don't parameterize on it.
//  TLogger    : same — logging is a dependency (di-hosting.html), inject ILogger, never a
//               type parameter. This one is almost always a mistake.
//  TResult    : unrelated to TInput/TOutput → the API can return anything, so it constrains
//               nothing and documents nothing. Almost certainly should be Result<TOutput>.
//  Net effect: a call site reads IProcessor<Order, Invoice, ShopDbContext, OrderValidator,
//              SerilogLogger, ProcessResult> — unreadable, and it models flexibility no
//              caller uses. Classic speculative generality.

// REDESIGN — generic ONLY over what genuinely varies (input/output), everything else injected:
public interface IProcessor<TInput, TOutput>
{
    Task<Result<TOutput>> ProcessAsync(TInput input);      // Result<T> replaces the stray TResult
}

public class OrderProcessor(                                // dependencies INJECTED, not generic:
    IValidator<Order> validator,
    ShopDbContext db,
    ILogger<OrderProcessor> log) : IProcessor<Order, Invoice>
{
    public async Task<Result<Invoice>> ProcessAsync(Order input)
    {
        var validation = validator.Validate(input);
        if (!validation.IsValid) return Result<Invoice>.Fail(validation.Error);
        log.LogInformation("Processing order {Id}", input.Id);
        // ... use db ...
        return Result<Invoice>.Ok(invoice);
    }
}

// WHAT CHANGED & WHY:
//  - 6 type params → 2 (TInput, TOutput) — the ONLY things that genuinely vary per processor.
//  - context/validator/logger became constructor dependencies (DI) — the right tool for
//    "this needs a collaborator," which is NOT what generics are for.
//  - the stray TResult became Result<TOutput> — a real, constrained, reused abstraction.
//  - call sites are now readable: IProcessor<Order, Invoice>.
// The lesson: generics express type-INDEPENDENT logic that varies by data type; dependencies
// and cross-cutting concerns belong to DI. Mixing them into type parameters is the most
// common over-abstraction in real .NET codebases.

🎯 Key Takeaways

  • Generics give one type-safe, boxing-free, cast-free definition parameterized by type — replacing per-type duplication and unsafe/slow object-based code; it's why all modern collections are generic.
  • Generic methods add type inference (from arguments, never the return type alone) — the mechanism behind LINQ's fluent chains.
  • Constraints (where T : ...) unlock capabilities on an otherwise-useless T: interfaces, class/struct, new(), base types, notnull, and generic math via INumber<T>.
  • .NET generics are reified (real at runtime) — no boxing of value types, working typeof(T)/new T(), distinct runtime types, and JIT specialization — a genuine advantage over Java's erasure.
  • Variance follows data flow: out T covariant (read-only producers), in T contravariant (write-only consumers), mutable collections invariant (both directions → unsafe otherwise).
  • Judgment matters: generics for genuinely type-independent logic that simplifies call sites; too many type parameters, dependency-as-type-parameter, and speculative flexibility are over-abstraction — prefer concrete types until a real second use appears.
